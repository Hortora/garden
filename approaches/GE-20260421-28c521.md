### Context

DSL method accepts a method reference to a field accessor that returns `DataStore<T>`, and needs to extract `T.class` at runtime without requiring the caller to pass an explicit `Class<T>` parameter. Normal reflection can't recover `T` from a lambda due to type erasure.

### Technique

If the functional interface extends `Serializable`, Java generates a `writeReplace()` method on the lambda's synthetic class. Calling it returns a `SerializedLambda` that captures the implementing class name and method name. Reflection on that method yields its full generic return type, from which the type argument can be extracted.

```java
// The functional interface must extend Serializable:
public interface Function1<A, R> extends java.util.function.Function<A, R>, Serializable { ... }

// At the call site — just a method reference, no Class<T> needed:
builder.rule("r1").from(CTX::persons)   // extracts Person.class automatically

// Implementation:
protected static <CTX, T> Class<T> extractElementType(Function1<CTX, DataSource<T>> f) {
    try {
        Method writeReplace = f.getClass().getDeclaredMethod("writeReplace");
        writeReplace.setAccessible(true);
        SerializedLambda sl = (SerializedLambda) writeReplace.invoke(f);
        Class<?> declaring = Class.forName(sl.getImplClass().replace('/', '.'));
        Method m = declaring.getMethod(sl.getImplMethodName());
        ParameterizedType returnType = (ParameterizedType) m.getGenericReturnType();
        return (Class<T>) returnType.getActualTypeArguments()[0];
    } catch (Exception e) {
        throw new RuntimeException("Requires a method reference (e.g. CTX::persons)", e);
    }
}
```

### Constraints

- Only works with **method references**, not lambdas. A lambda's `writeReplace()` points to a synthetic method, not the original field accessor, so there is no declared generic return type to reflect on.
- The functional interface must extend `Serializable`.
- The implementing class must be on the classpath (it's the record/class owning the accessor method).

### Why this is non-obvious

Type erasure normally prevents runtime access to generic type parameters. Most developers reach for an explicit `Class<T>` parameter instead. The `Serializable` → `SerializedLambda` → `getGenericReturnType()` chain is undocumented as a technique and only discoverable by knowing that serializable lambdas generate `writeReplace()`.
