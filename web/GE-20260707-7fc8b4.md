### Symptom
TypeScript compile error when adding `implements DataReceiver` to a class that already structurally satisfies the interface:
```
Type 'TypedDataSet | undefined' is not assignable to type 'unknown'.
  Type 'TypedDataSet | undefined' is not assignable to type 'unknown'.
```
The error points at the `dataSet` setter, not the getter.

### Context
A base class (`PagesElement`) has a property with a typed setter:
```typescript
set dataSet(value: TypedDataSet | undefined) { ... }
```

An interface (`DataReceiver`) declares the same property with a wider type:
```typescript
interface DataReceiver {
  dataSet: unknown;
  error: string;
}
```

The class structurally satisfies the interface — you can assign an instance to a `DataReceiver` variable without error. But adding `implements DataReceiver` to the class declaration triggers a compile error under `strict: true`.

### Root cause
Under `strict: true`, TypeScript enforces setter parameter contravariance. A property setter's parameter must accept at least as wide a type as the interface declares. `TypedDataSet | undefined` is narrower than `unknown`, so the setter rejects the wider interface contract.

This is correct type theory — a setter that only accepts `TypedDataSet` cannot satisfy a contract that promises to accept `unknown`. But it's surprising because:
1. The class already works as a `DataReceiver` via structural typing (assignment works)
2. The `implements` clause looks like a documentation annotation, not a new constraint
3. Getters are covariant (narrower return types are fine), so the asymmetry is unexpected

### Fix
Two approaches:

**Option A — Rely on structural typing (no `implements` clause):**
```typescript
// PagesElement structurally satisfies DataReceiver through VizTarget
// The event detail retyping in site.ts bridges the gap:
interface DataRequestDetail {
  readonly element: VizTarget;  // VizTarget extends DataReceiver
  readonly lookup: DataSetLookup;
}
```
The class participates in the type system without an explicit `implements` declaration. This is what casehub-pages chose.

**Option B — Widen the setter to match the interface:**
```typescript
set dataSet(value: unknown) {
  this._dataset = value as TypedDataSet | undefined;
  this.update();
}
```
This satisfies the interface but loses type safety at the setter boundary — callers can pass anything, and the cast silently accepts it.

### Why this is non-obvious
Structural typing and nominal `implements` have different variance rules in TypeScript. A class can structurally satisfy an interface (assignments compile, function parameters accept it) while failing to satisfy the same interface nominally (the `implements` clause rejects it). Most developers expect `implements` to be a zero-cost documentation annotation on a class that already works — discovering it imposes stricter variance checks is surprising.

### Alternatives considered
- **Widen the setter (Option B):** Works but trades compile-time safety for nominal compliance. Rejected because the setter's narrow type catches real bugs (passing raw JSON instead of TypedDataSet).
- **Cast at the `implements` site:** Not possible in TypeScript — there's no syntax to override variance at the declaration level.
- **Separate adapter class:** Over-engineering for a type-level concern.
