## Technique

Pre-render 4 directional canvas textures per unit type at application startup, then swap `sprite.material` each animation frame based on camera angle. No per-frame texture generation; no spritesheet pipeline needed.

```javascript
function makeDirTextures(drawFn, size = 128) {
  return [0, 1, 2, 3].map(dir => {
    const c = document.createElement('canvas');
    c.width = c.height = size;
    drawFn(c.getContext('2d'), size, dir);
    const tex = new THREE.CanvasTexture(c);
    tex.premultiplyAlpha = true;
    return new THREE.SpriteMaterial({
      map: tex, transparent: true,
      depthWrite: true, alphaTest: 0.1
    });
  });
}

// Per-frame direction update (called in animate())
function updateSpriteDirs() {
  unitSprites.forEach((sp, tag) => {
    const mats = sp.userData.mats;
    if (!mats) return;
    const facing = unitFacings.get(tag) ?? 0;
    sp.material = mats[getDir4(facing, sp.position, camera.position)];
  });
}
```

Each unit type gets one set of 4 `SpriteMaterial` objects at startup. Direction selection at runtime is a single Map lookup + array index — O(1), no allocation.

## Why this is non-obvious

The obvious approach is spritesheet animation (Aseprite → TexturePacker → `PIXI.AnimatedSprite`) or per-frame canvas drawing. Pre-generating textures from draw functions eliminates the build pipeline entirely while keeping art as readable JavaScript. The material swap pattern (`sprite.material = mats[dir]`) is not commonly documented as a direction-selection technique.

## When to use

- Canvas 2D art defined as draw functions (not pre-authored images)
- 4-direction facing is sufficient (front/right/back/left)
- WebGL renderer (Three.js Sprite auto-billboards to camera)
- Unit count in the hundreds or fewer (materials are shared across all instances of the same type)
