### Symptom

A Web Component's close/transition method sets up an `animationend` listener to perform cleanup after a CSS animation completes. In unit tests (vitest + jsdom), the listener never fires and the component hangs in an intermediate state — e.g., a modal's close animation never completes, so `dialog.close()` is never called.

### Context

Any component that uses the pattern: add CSS class → listen for `animationend` → perform state transition. Common in modals (exit animation before close), drawers, accordions, toast notifications. jsdom has no CSS engine — CSS animations don't run, so `animationend` never fires.

### Root cause

jsdom does not implement a CSS engine. CSS custom properties, computed styles, and animations are all absent or return defaults. `getComputedStyle(el).animationDuration` returns `'0s'` (or empty string) because no stylesheet is actually applied. Since no animation runs, the `animationend` event never fires.

### Fix

Before setting up the `animationend` listener, check if the animation will actually run:

```javascript
const duration = getComputedStyle(dialog).animationDuration;
if (!duration || duration === '0s') {
  // No CSS engine or animation disabled — skip directly to post-animation state
  dialog.close(returnValue);
  return;
}

// Real browser — set up listener
dialog.addEventListener('animationend', () => {
  dialog.close(returnValue);
}, { once: true });
```

This also serves as a natural `prefers-reduced-motion` fallback in real browsers when animation-duration is overridden to 0.

### Why this is non-obvious

The component works perfectly in a real browser. The test failure mode is not an error — it's a silent hang. The `animationend` listener is registered successfully (no error), the CSS class is applied (no error), but nothing happens. Developers typically debug by checking event listener registration or CSS class application, not by questioning whether the CSS engine exists.

### Alternatives considered

- Mocking `animationend` by dispatching it manually in tests — works but couples tests to implementation details and requires every test to know about the animation
- Using `setTimeout` as a fallback — fragile, arbitrary timing, doesn't address the root cause
- Disabling animations via a property — adds API surface just for testing
