## ESLint no-unnecessary-type-assertion conflicts with TSC --build for querySelector

### Symptom
ESLint auto-fix removes `as HTMLElement` from `querySelector()` calls in test files, then `tsc --build` fails with "Property 'dataSet' does not exist on type 'Element'". Re-adding the assertion makes ESLint fail again. Circular.

### Context
Monorepo with `typescript-eslint/strict-type-checked` rules AND `tsc --build` with project references. Test files use JSDOM and call `querySelector()` with type assertions to access custom element properties.

### Root cause
ESLint's `projectService` and TSC's `--build` mode resolve DOM types through different paths. ESLint resolves `querySelector()` return type as `HTMLElement | null` (matching the DOM lib directly), while TSC `--build` with project references resolves it as `Element | null`. Since `HTMLElement` extends `Element`, the assertion `as HTMLElement` is unnecessary from ESLint's perspective but required from TSC's.

### Fix
Relax `no-unnecessary-type-assertion` in the test file override block of `eslint.config.mjs`:

```js
{
  files: ["**/*.test.ts", "**/*.test.tsx"],
  rules: {
    "@typescript-eslint/no-unnecessary-type-assertion": "off",
  },
}
```

### Why this is non-obvious
Both tools use TypeScript's type system, so you'd expect them to agree on whether a type assertion changes the expression type. The disagreement is in the type resolution path, not the type-checking logic â€” a layer most developers never think about.
