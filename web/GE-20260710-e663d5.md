### Context
When passing a value through a typed interface that expects a specific type but
the runtime value is intentionally different (e.g. passing raw JSON through a
`TypedDataSet` slot), a type assertion is needed. `as any` works but silences
all downstream type checks on that expression.

### Technique
Use `as never` instead of `as any`. `never` is assignable to every type (it's
the bottom type), so it satisfies any constraint. But unlike `any`, it doesn't
widen — the expression remains visible to strict checks like
`--noUncheckedIndexedAccess`.

```typescript
// WIDE — silences all checks on this expression
sink.apply({ type: "snapshot", dataset: data as any });

// NARROW — satisfies the constraint without widening
sink.apply({ type: "snapshot", dataset: data as never });
```

### Why this is non-obvious
`as any` is the reflexive choice for "make the type checker stop complaining."
`as never` is semantically more precise — it says "I know this doesn't match,
but trust me" without opting out of the type system for everything downstream.
Most TypeScript developers don't think of `never` as a practical assertion tool.
