### Symptom
Error: "response.json is not a function" thrown as an uncaught TypeError,
even though the call appears to be wrapped with `.catch(() => null)`.

### Context
When calling `await response.json().catch(() => null)` on a mock or real
response object that doesn't have a `json` method (e.g. `{ ok: false, status: 500 }`
without `json`), the expression throws before any promise is created.

Common in test mocks for error responses where `json()` is only provided
on success mocks.

### What was tried (didn't work)
`const body = await response.json().catch(() => null)` — intended to
gracefully handle non-JSON responses by catching the parse error and
returning null. Works when `json` exists but returns a rejected promise
(e.g. malformed JSON). Fails when `json` is undefined entirely.

### Root cause
`response.json()` evaluates in two phases:
1. `response.json` — property lookup (returns `undefined` if absent)
2. `undefined()` — function call on undefined → **synchronous TypeError**

`.catch()` is chained to the return value of `response.json()`, but the
TypeError fires at step 2 before any return value exists. There is no
promise to attach `.catch()` to.

### Fix
Use try/catch to handle both synchronous (missing method) and asynchronous
(parse failure) errors:

```typescript
let body: unknown = null;
try { body = await response.json(); } catch { /* non-JSON or missing */ }
```

### Why this is non-obvious
The pattern `await expr().catch(handler)` is idiomatic for promise error
handling. It works for all cases where `expr()` returns a promise — including
a rejected one. The failure mode only surfaces when `expr` itself is undefined,
which is uncommon in production code but routine in test mocks where only
the relevant properties of the response are provided.
