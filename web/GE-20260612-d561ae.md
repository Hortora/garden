## exactOptionalPropertyTypes rejects undefined passed to optional properties

### Symptom
TypeScript error: `Type 'Date | undefined' is not assignable to type 'Date'` when passing
`{ referenceDate: options?.referenceDate }` to a function expecting `{ referenceDate?: Date }`.

### Context
When `exactOptionalPropertyTypes` is enabled, `prop?: Date` means the property is either
absent or `Date` — but NOT `undefined`. Passing `options?.referenceDate` evaluates to
`Date | undefined`, which includes the `undefined` case that the target type rejects.

### Root cause
`exactOptionalPropertyTypes` distinguishes between "property absent" and "property present
with value undefined". Most developers treat `prop?: T` as equivalent to `prop: T | undefined`,
but this flag makes them distinct types.

### Fix
Conditionally construct the object so the property is absent (not undefined) when the
source value is undefined:

```typescript
const opsOptions = options?.referenceDate !== undefined
  ? { referenceDate: options.referenceDate }
  : undefined;
applyOps(dataset, resolvedOps, opsOptions);
```

### Why this is non-obvious
The error message says `Type 'Date | undefined' is not assignable to type 'Date'` — it
does not mention `exactOptionalPropertyTypes`. A developer unfamiliar with this flag will
try type assertions or restructuring the interface, when the actual fix is changing how
the object is constructed.
