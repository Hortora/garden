## exactOptionalPropertyTypes rejects undefined passed to optional properties

### Symptom
TypeScript error: `Type 'Date | undefined' is not assignable to type 'Date'` when passing
`{ referenceDate: options?.referenceDate }` to a function expecting `{ referenceDate?: Date }`.

### Context
When `exactOptionalPropertyTypes` is enabled, `prop?: Date` means the property is either
absent or `Date` — but NOT `undefined`. Passing `options?.referenceDate` evaluates to
`Date | undefined`, which includes the `undefined` case that the target type rejects.

### Root cause
`exactOptionalPropertyTypes` distinguishes between "property absent" and "property present
with value undefined". Most developers treat `prop?: T` as equivalent to `prop: T | undefined`,
but this flag makes them distinct types.

### Fix
Conditionally construct the object so the property is absent (not undefined) when the
source value is undefined:

```typescript
const opsOptions = options?.referenceDate !== undefined
  ? { referenceDate: options.referenceDate }
  : undefined;
applyOps(dataset, resolvedOps, opsOptions);
```

### Why this is non-obvious
The error message says `Type 'Date | undefined' is not assignable to type 'Date'` — it
does not mention `exactOptionalPropertyTypes`. A developer unfamiliar with this flag will
try type assertions or restructuring the interface, when the actual fix is changing how
the object is constructed.

## Variant — Getter/setter accessor pairs with backing fields

### Symptom
TS2412 on getter return or setter assignment when a class has `private _foo?: T` backing
a `get foo(): T | undefined` / `set foo(value: T | undefined)` pair. The error fires on
the backing field assignment inside the setter: `this._foo = value`.

### Context
The `?` in `private _foo?: T` means "property may be absent" under exactOptionalPropertyTypes,
but the setter receives `T | undefined` (explicit union). Assigning `T | undefined` to a
`?`-optional field triggers TS2412 because `undefined` is not in the field's value type.

### Fix
Declare backing fields with explicit `| undefined` instead of `?`:

```typescript
// WRONG — TS2412
private _foo?: SomeType;

// RIGHT — compiles cleanly
private _foo: SomeType | undefined = undefined;
```

Alternative: use method-style setters (`setFoo(value: T | undefined)`) to avoid the
accessor pair entirely. This sidesteps the getter/setter type unification issue.

### Why this is non-obvious
The `?` and `| undefined` look semantically identical to most developers.
`exactOptionalPropertyTypes` makes them distinct, but the error message points at the
assignment site rather than the field declaration — leading developers to add casts or
restructure the setter logic when the real fix is a one-character change in the field
declaration.
