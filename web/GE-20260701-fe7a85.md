### Symptom
Buttons inside a Web Component stop responding after the component is reconfigured. No errors in the console — the buttons simply become inert. The problem only manifests after the first `configure()` call, not on initial render.

### Context
A light-DOM Web Component (using `this.innerHTML = ...` rather than Shadow DOM) exposes a `configure()` method that the parent calls to update the component's state. The parent also wires `addEventListener` to child elements inside the component after initial render. When `configure()` is called again (e.g., on a worker switch), it replaces the entire innerHTML, which silently destroys all DOM elements — including the ones the parent wired listeners to.

### What was tried (didn't work)
- Re-rendering the entire component on every configure call — seemed correct since it ensures all display state is fresh
- The component appeared to work perfectly on initial load; the bug only surfaced on reconfigure

### Root cause
`this.innerHTML = \`...\`` creates entirely new DOM nodes. The old nodes (with their event listeners) are garbage collected. The parent's references to `header.querySelector("#compose-btn")` now point to the old, detached node — not the new one in the live DOM. Since there's no error (the old node simply has no parent), the failure is completely silent.

### Fix

Split `configure()` into two code paths:

```typescript
configure(opts: { sessionName: string; sessionId: string }): void {
  this._sessionName = opts.sessionName;
  this._sessionId = opts.sessionId;

  const nameEl = this.querySelector("#session-name");
  if (nameEl) {
    // DOM already exists — update text only, preserve event listeners
    nameEl.textContent = this._sessionName;
  } else {
    // First configure — render the full DOM
    this.render();
  }
}
```

The key insight: use `querySelector` as a DOM-existence check. If the element exists, the DOM is already rendered — update only the data-bound text nodes. If not, this is the first render — create the full DOM structure.

### Why this is non-obvious
The mental model for `innerHTML` is "set the content." Developers expect it to be idempotent — calling it twice with the same template should produce the same result. And it does, visually. But event listeners are invisible state attached to specific DOM nodes, not to CSS selectors or IDs. The replacement nodes look identical but are different objects. No browser warning, no framework error, no test failure (unless you specifically test button clicks after reconfigure).

### Why this fix
The alternative — moving all event wiring into the component itself — would work but changes the ownership model. If the parent owns the orchestration (deciding what "Workers toggle" means in context), the listeners should stay on the parent side. The fix preserves this separation by ensuring the DOM nodes the parent wired to are never replaced.

### Alternatives considered
- **Move all listeners into the component:** Works but couples the component to its orchestration context. The component would need to know about workspace.toggleWorkers() — breaking reusability.
- **Re-wire listeners after every configure() call:** Works but fragile — every caller must remember to re-wire, and forgetting produces the same silent failure.
- **Use event delegation on the component root:** Would survive innerHTML replacement since the listener is on the component element itself, not on child nodes. Valid alternative but changes the event handling pattern.
