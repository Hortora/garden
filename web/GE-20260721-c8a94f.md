### Symptom

Playwright E2E tests for an SSE-driven Lit dashboard fail intermittently with "element(s) not found" timeouts. The mock server emits SSE events but the UI never renders them. Tests that pass in isolation fail when run sequentially. Even-numbered tests in a sequence fail while odd-numbered pass.

### Context

A Lit `app-shell` component conditionally renders either `demo-launcher` (overview) or `case-execution-view` (detail). Both create their own `EventSource` to the same `/api/scenarios/events` SSE endpoint in `connectedCallback()`. Navigation between views disconnects the old EventSource and connects a new one.

A Node.js mock SSE server tracks connected clients in a `Set<ServerResponse>`. Tests call `mockServer.waitForSSEClient()` after navigation to ensure the SSE connection is established before emitting events.

### Root cause

When navigating from overview to detail view:
1. `page.goto('/')` — `demo-launcher` connects EventSource (client A added to set)
2. Click nav item — Lit removes `demo-launcher`, adds `case-execution-view`
3. `demo-launcher.disconnectedCallback()` calls `eventSource.close()` (client A removal is async)
4. `case-execution-view.connectedCallback()` creates new EventSource (client B)
5. `waitForSSEClient()` checks `sseClients.size > 0` — resolves immediately on client A (stale, about to disconnect)
6. Test emits SSE event — goes to client A (disconnecting) instead of client B (the active view)

The `Set<ServerResponse>.size` check is unreliable because the old client hasn't been removed yet when the new client connects. The set briefly contains both, then drops to one — but `waitForSSEClient()` already resolved on the stale client.

### Fix

Track a monotonic connection counter instead of checking set size. The counter increments on every new SSE connection and never decrements.

```typescript
// In MockServer class
private sseConnectionCount = 0;

// In SSE endpoint handler, after adding client to set:
this.sseClients.add(res);
this.sseConnectionCount++;

// Wait for a NEW connection (not just any existing one)
waitForFreshSSEClient(timeoutMs = 5000): Promise<void> {
  const countBefore = this.sseConnectionCount;
  return new Promise((resolve, reject) => {
    const interval = setInterval(() => {
      if (this.sseConnectionCount > countBefore) {
        clearInterval(interval);
        clearTimeout(timer);
        resolve();
      }
    }, 50);
    const timer = setTimeout(() => {
      clearInterval(interval);
      reject(new Error('Timed out waiting for fresh SSE client'));
    }, timeoutMs);
  });
}
```

Use `waitForFreshSSEClient()` after any navigation that switches Lit views with their own EventSource connections. Use `waitForSSEClient()` (set size check) only for initial page load where no prior connection exists.

### Why this is non-obvious

The race is invisible in test output — Playwright reports "element not found" with no hint that the SSE event was delivered to the wrong component. The even/odd failure pattern (caused by connection timing relative to the mock server's `reset()` between tests) makes it look like a test isolation issue rather than a client tracking bug. The fix (monotonic counter) is conceptually simple but the diagnosis requires understanding the Lit component lifecycle, EventSource auto-reconnect semantics, and the async nature of TCP connection teardown.

### Alternatives considered

- **`waitForSSEClient()` with a delay** — fragile, doesn't guarantee the new connection is established
- **Disconnect SSE before navigation** — `mockServer.disconnectSSE()` + `reconnectSSE()` + `waitForSSEClient()` — works but adds unnecessary complexity and relies on EventSource auto-reconnect timing
- **Check `sseClients.size > countBefore`** — fails because set size doesn't increase when old disconnects and new connects simultaneously (1 → 1)
