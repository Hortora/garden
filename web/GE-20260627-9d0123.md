### Symptom
ECharts charts (pie, bar, scatter) render with axes, labels, and a canvas element — but no visible data (no bars, no pie segments, no scatter points). No error is reported. Existing tests that check for canvas existence pass.

### Context
A YAML dashboard defines a grouped displayer with a function on the same column as the group:

```yaml
columnGroup:
    source: department
functions:
    - source: department
    - source: department
      function: COUNT
```

The intent is column 0 = department label (key), column 1 = count (aggregate). Both columns reference the same `source`.

### What was tried (didn't work)
1. Investigated ECharts rendering — subtypes (DONUT, COLUMN_STACKED) seemed to correlate with empty charts but turned out to be coincidental
2. Checked subtype mapping (YAML → component) — all mappings correct
3. Checked for CSS overflow clipping — not the cause
4. Suspected duplicate column IDs in `createTypedRow` Map — was a real secondary issue but not the primary cause

### Root cause
The column classifier in `parseGroupEntry` checks conditions in this order:

```typescript
// Broken order
if (col.source === entry.columnGroup.source) {
    return { kind: "key", ... };     // ← fires first when source matches
}
if (col.function) {
    return { kind: "aggregate", ... }; // ← never reached for same-source functions
}
```

When `source: department, function: COUNT` is evaluated against `columnGroup.source: department`, the key check fires first. The `function: COUNT` is silently ignored. Both columns become "key" type, producing duplicate label-only data. Charts receive `[["dept", "dept"], ["Engineering", "Engineering"], ...]` instead of `[["dept", "count"], ["Engineering", 14], ...]`.

### Fix

Swap the priority — check `col.function` before the key match:

```typescript
if (col.function) {
    return { kind: "aggregate", ... }; // ← checked first now
}
if (col.source === entry.columnGroup.source) {
    return { kind: "key", ... };
}
```

### Why this is non-obvious
The symptom (empty charts) points toward the rendering layer — ECharts configuration, subtype mapping, or CSS issues. The data pipeline parser is several layers upstream and produces no error. The only diagnostic that reveals the bug is inspecting `dataSet.columns` on the web component from the browser: both columns show `type: LABEL` when one should be `NUMBER`. Without this inspection, the rendering layer appears to be the problem.

A secondary issue (`createTypedRow` using a Map that overwrites on duplicate column IDs) compounds the misdirection — even if you suspect a data issue, the Map overwrite makes `row.cell(id)` return the same cell for both columns, producing identical data in both columns.

### Alternatives considered
- Adding explicit `column: count` IDs to differentiate duplicate-ID columns — works as a workaround but doesn't fix the root cause; the aggregate is still never computed
- Fixing `datasetToSource` to use positional `row.cells[i]` instead of `row.cell(c.id)` — fixes the Map overwrite symptom but still produces wrong data types if the parser classifies the column incorrectly
