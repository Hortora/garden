### The technique

When dynamically generating HTML with inline `onclick` handlers that include data from external sources (e.g. channel names from an API), using `escapeHtml()` is not sufficient to prevent JS injection.

`escapeHtml()` converts `&`, `<`, `>`, `"`, `'` to HTML entities. It does NOT escape JS metacharacters: `;`, `)`, `\`, backtick. A value like `foo'); alert(1); //` survives `escapeHtml()` and breaks out of the JS string:

```html
<!-- Unsafe — channel name is external data -->
<div onclick="selectChannel('${escapeHtml(name)}')">
<!-- If name = "foo'); alert(1); //" → onclick executes injected JS -->
```

### Fix

Store the external value in a `data-*` attribute (HTML entity encoding is safe there), then read it from the DOM in a JS event listener:

```javascript
// In the HTML template:
`<div class="channel-item" data-channel="${escapeHtml(name)}">...</div>`

// After setting innerHTML:
container.querySelectorAll('.channel-item[data-channel]').forEach(el => {
    el.addEventListener('click', () => fn(el.dataset.channel));
});
```

`el.dataset.channel` retrieves the HTML-decoded string (browser decodes `&quot;` etc. back to characters). The JS engine never sees the value inside a JS string literal, so injection is impossible regardless of the value's content.

### Why this is non-obvious

Developers reach for `escapeHtml()` as the general XSS fix. It works correctly for HTML injection (preventing tag injection) and for event handlers with fixed string arguments. The gap only surfaces when the data comes from an external source — a server response, user input, or agent-generated content. If all your onclick values are hardcoded strings like `'channel'`, `escapeHtml()` in the template is harmless and the gap is invisible.
