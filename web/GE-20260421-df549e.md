## Problem

Directional sprite selection shows left-facing art when units face right and vice versa. No error — characters just appear to face the wrong way when the camera orbits.

## Symptom

After orbiting the camera 90°, units that should show a right-side profile show a left-side profile instead. The front/back views are correct; only left/right are swapped.

## Root cause

Three.js uses a right-handed coordinate system where +x is screen-right and +z is toward the viewer. The natural `Math.atan2(dx, dz)` gives an angle where positive `dx` maps to a positive (clockwise) rotation — but in Three.js screen-space, positive `dx` from unit to camera means the camera is to the right, which should select the unit's left-side profile (the side facing the camera). The sign is inverted relative to expectation.

## Fix

Negate `dx` in the atan2 call:

```javascript
function getDir4(facingAngle, unitPos, camPos) {
  const camAngle = Math.atan2(-(camPos.x - unitPos.x), camPos.z - unitPos.z);
  let rel = camAngle - facingAngle;
  while (rel < 0)          rel += Math.PI * 2;
  while (rel >= Math.PI*2) rel -= Math.PI * 2;
  return Math.round(rel / (Math.PI/2)) % 4; // 0=front, 1=right, 2=back, 3=left
}
```

## Why this is non-obvious

The handedness correction (`-dx`) is not documented in the Three.js API. It's only discoverable by testing the actual sprite selection at known camera angles. The symptom (wrong profile) could also be explained by wrong sprite art or wrong facing angle — making the root cause hard to isolate.
