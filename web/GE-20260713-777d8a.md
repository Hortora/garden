### Symptom

Focus trap wrapping comparisons (`document.activeElement === lastElement`) silently never match when `lastElement` is inside a shadow root. Focus appears to cycle correctly (the element receives focus visually) but the boundary check never triggers, so Tab wrapping doesn't fire.

### Context

Any Web Component using shadow DOM that compares `document.activeElement` against elements inside its shadow tree — focus traps, roving tabindex, active-element guards. The comparison returns `false` even though the inner element has focus.

### Root cause

When an element inside an open shadow root has focus, `document.activeElement` returns the shadow host element, not the focused inner element. This is correct per spec — shadow DOM encapsulates focus state. The actual focused element is only accessible via `shadowRoot.activeElement`.

For slotted content (light DOM distributed into shadow slots), `document.activeElement` returns the slotted element directly — the inconsistency between shadow-internal and slotted elements makes the bug intermittent and harder to diagnose.

### Fix

Walk the `shadowRoot.activeElement` chain to find the deeply-focused element:

```javascript
function getDeepActiveElement() {
  let active = document.activeElement;
  while (active?.shadowRoot?.activeElement) {
    active = active.shadowRoot.activeElement;
  }
  return active;
}
```

Replace all `document.activeElement === element` comparisons with `getDeepActiveElement() === element`.

### Why this is non-obvious

The `document.activeElement` API works correctly for all non-shadow-DOM code. The shadow DOM delegation behaviour is per-spec but undocumented in most Web Component tutorials. The failure is silent — no error, no warning — and the visual appearance is correct (the element does receive focus). The bug only manifests as missing boundary behaviour (wrap doesn't fire, guard doesn't trigger).

### Alternatives considered

- Checking `e.composedPath()[0]` on focus/blur events — only works event-by-event, not for point-in-time active element queries
- Using `this.shadowRoot.activeElement` directly — only finds elements in one shadow root level, doesn't handle nested shadow DOMs or slotted content
