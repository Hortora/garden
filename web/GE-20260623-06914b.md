### Symptom
Layout containers render correctly but no charts, tables, or metrics appear. No errors in console, no warnings, no failed network requests. The application appears to work — it just renders empty containers where visualizations should be.

### Context
A TypeScript monorepo publishes a package (`@casehubio/pages-viz`) that registers 20+ Web Component custom elements via `customElements.define()` calls at module scope in each component file. The package's `index.ts` re-exports the component classes. Consumers use a bare side-effect import: `import "@casehubio/pages-viz"`.

This worked with webpack (lenient tree-shaking). After switching host apps to esbuild (aggressive tree-shaking), custom elements stopped registering.

### What was tried (didn't work)
- Checked import order — irrelevant
- Verified `connectedCallback` firing — it wasn't, because elements weren't registered
- Checked for typos in tag names — correct
- Inspected the bundled output — the `customElements.define()` calls were completely absent

### Root cause
esbuild performs aggressive tree-shaking. A bare import (`import "@scope/pkg"`) with no used exported bindings can be dropped entirely if the package doesn't declare `"sideEffects": true` in its `package.json`. The `customElements.define()` calls are module-level side effects, but esbuild has no Web Component-specific analysis — it treats the import as dead code.

webpack is more conservative: it preserves bare imports by default regardless of the `sideEffects` field.

### Fix
Add `"sideEffects": true` to the package's `package.json`:

```json
{
  "name": "@scope/pages-viz",
  "sideEffects": true,
  ...
}
```

This tells esbuild (and all other bundlers) to preserve the module's side effects even when no exports are consumed.

The better architectural fix: move the bare import into the runtime package that exports the consumer API (e.g., `loadSite()`), so custom element registration becomes an internal concern:

```ts
// pages-runtime/src/index.ts
import "@casehubio/pages-viz"; // internal side-effect — registers custom elements
export { loadSite } from "./site.js";
```

Consumers then only need `import { loadSite } from "@casehubio/pages-runtime"` — no knowledge of internal package structure required.

### Why this is non-obvious
Both esbuild and Web Components are well-documented individually. The failure mode emerges from their interaction: esbuild's tree-shaking optimization silently removes code that the Web Component registration pattern depends on. The complete absence of any error or warning makes diagnosis exceptionally difficult — the app renders a valid DOM structure with correctly named custom elements, but those elements are never upgraded because `customElements.define()` was never called.

### Alternatives considered
- Importing named exports from pages-viz in every consumer file — works but couples consumers to internal package structure
- Using `import * as _viz from "@scope/pages-viz"` — works but fragile (linters flag unused imports)
- Dynamic import at runtime — unnecessary complexity for a build-time problem
