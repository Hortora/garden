### What it is

`marked.lexer()` v9 distributes trailing newlines between block token types in a non-uniform, undocumented way:

| Token type | `raw` field content | `rawLines` |
|------------|---------------------|------------|
| Heading `# Title` | `"# Title\n\n"` (heading line + following blank line) | 2 |
| Paragraph | `"paragraph text"` (no trailing `\n`) | 0 |
| Space (blank line) | `"\n\n"` (both the paragraph's trailing newline and any blank lines after) | 2 |
| Code fence | raw includes the closing ` ``` ` and its `\n` | varies |

### How to discover it

```js
const { marked } = require('marked');
const content = "# Title\n\nParagraph text.\n\n## Heading 2\n";
marked.lexer(content).forEach(t =>
  console.log(t.type, JSON.stringify(t.raw))
);
// heading  "# Title\n\n"
// paragraph "Paragraph text."       ← no trailing newline!
// space    "\n\n"
// heading  "## Heading 2\n"
```

### Why this matters

Code that uses `token.raw.split('\n').length - 1` to count how many source lines a token occupies gets `0` for every paragraph token. If this count is used to track line position (e.g. to map markdown tokens to source line numbers), paragraphs appear to occupy zero lines. Subsequent tokens (headings, code blocks) start at the wrong line number, breaking any logic that checks `c.lineStart < tokenEnd`.

### Where it's not documented

Not mentioned in the marked.js v9 API docs, the lexer documentation, or the migration guide. Discovered by inspecting token `raw` field values at runtime.

### Constraints

Behaviour observed in marked.js 9.1.6 (CDN and npm). May differ in marked.js v10+ — verify with the snippet above on the target version.

### Invalidation triggers

Check whenever upgrading marked.js. The distribution of trailing newlines across token types is an internal implementation detail that could change between minor versions.
