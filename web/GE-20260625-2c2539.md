### Symptom
Integration tests that individually pass start failing when run together. Symptoms appear as wrong data in components — e.g. a table showing 1 filtered row instead of 3, or a form displaying the wrong record after a click. The failures are order-dependent: tests after one that modifies URL hash params fail; the modifying test itself passes.

### Context
Any vitest test suite using JSDOM where the code under test calls `history.replaceState()` to sync application state to the URL hash (e.g. filter params, sort columns, pagination, text filter terms). A subsequent test's `loadSite()` or equivalent init function calls `restoreFromUrl(location.hash)` and silently restores the previous test's state.

### What was tried (didn't work)
- Checked custom element isolation (each test creates fresh elements)
- Checked ComponentViewState cleanup (`dispose()` calls `.clear()` — working correctly)
- Checked event listener cleanup (`AbortController.abort()` — working correctly)
- Stashed changes and tested at different commits to isolate the regression
- None of these addressed the root cause because the leak is in JSDOM's global `location` object

### Root cause
JSDOM maintains a single global `location` object across all test cases in a file. When `history.replaceState(null, "", "#/page/Contacts?tf=Bob")` runs in test N, the hash stays set for test N+1. Vitest does not reset `location.hash` between tests — it resets the DOM tree but not the Location API state.

### Fix
Add `location.hash = ""` to `afterEach` in any test file where code under test modifies the URL hash:

```typescript
afterEach(() => {
  site?.dispose();
  site = null;
  document.body.removeChild(target);
  location.hash = "";
});
```

### Why this is non-obvious
The symptom ("table shows wrong data after click") points at component state, event handling, or data pipeline logic — not at the URL. Debugging follows the data flow (click → event → pipeline → render) rather than the test environment. The URL hash is set as a side effect of the pipeline's `syncUrl()` call, making it invisible in the primary data flow. The fact that `dispose()` correctly clears all application state (ComponentViewState, FilterState, event listeners) makes the leak even harder to spot — everything that *should* be cleaned up *is* cleaned up.

### Why this fix
The alternative — resetting location in `beforeEach` — is less safe because it doesn't prevent hash pollution from reaching other test files if vitest reuses the JSDOM instance. Clearing in `afterEach` ensures the hash is clean before the test framework makes any file-boundary decisions.

### Alternatives considered
- Resetting in `beforeEach` instead of `afterEach` — works but leaves dirty state between tests, risking cross-file pollution
- Using `vi.stubGlobal('location', ...)` — overkill; creates maintenance burden for a one-line fix
- Adding URL reset to `dispose()` — wrong layer; `dispose()` is application code, not test infrastructure
