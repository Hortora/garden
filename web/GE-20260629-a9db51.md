### Symptom

All items in an ECharts custom series timeline/Gantt chart render as diamond milestone markers instead of duration bars — even rows with valid start AND end dates.

### Context

Building a `type: 'custom'` ECharts series with a `renderItem` callback. Data items have a 3-element value array: `[startTime, categoryIndex, endTime]`. The renderItem function checks `api.value(3) !== undefined` to detect milestones (items with no 4th element).

### What was tried (didn't work)

Checking `api.value(3) !== undefined` to distinguish bars from milestones — assumed out-of-range indices would return `undefined` like standard JavaScript array access.

### Root cause

ECharts `api.value(N)` does NOT return `undefined` for out-of-range indices. It returns a default value (typically `0`). So `api.value(3) !== undefined` evaluates to `true` for ALL data items, not just those with a 4th element.

This is different from JavaScript array behavior where `arr[outOfRange]` returns `undefined`.

### Fix

Use explicit flags on the data item instead of probing for values at arbitrary indices:

```typescript
// ❌ Wrong — always truthy because api.value(3) returns 0, not undefined
const isMilestone = api.value(3) !== undefined;

// ✅ Correct — use an explicit flag set during data preparation
const isMilestone = params.data?.isMilestone === true;
```

Set the flag during data preparation:
```typescript
const seriesData = items.map(item => ({
  value: [item.start.getTime(), categoryIndex, item.end?.getTime() ?? item.start.getTime()],
  isMilestone: item.end === null,  // explicit flag
}));
```

### Why this is non-obvious

JavaScript developers expect `value(outOfRange)` to return `undefined`, matching array semantics. ECharts `api.value()` uses a different internal lookup that returns defaults for missing indices — this is not documented in the ECharts custom series API reference.
