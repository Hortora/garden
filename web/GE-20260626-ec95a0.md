### Symptom

Consumer app runs `npm install @scope/my-package` from GitHub Packages (or any registry). Install succeeds, but the package's `package.json` contains dependencies like:

```json
"@scope/sibling-package": "workspace:*"
```

npm cannot resolve `workspace:*` — it's a Yarn-specific protocol. The consumer gets "no matching version" errors at install time.

### Context

Monorepo using Yarn 4 workspaces. Internal packages reference each other with `workspace:*`. CI workflow uses `npm publish` to push to GitHub Packages.

### What was tried (didn't work)

- `npm publish "$dir"` — publishes `package.json` verbatim including `workspace:*`
- Setting `NODE_AUTH_TOKEN` — authentication works fine; the problem is the published content, not access

### Root cause

`npm publish` reads `package.json` as-is and publishes it without any transformation. Yarn's `workspace:*` protocol is proprietary — npm doesn't know how to resolve it to a real version number.

Yarn 4 has its own publish command (`yarn npm publish`) that resolves `workspace:*` to the actual version of each sibling package before creating the tarball.

### Fix

Replace `npm publish` with `yarn npm publish` in CI:

```yaml
# Before (broken)
npm publish "$dir"
env:
  NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}

# After (correct)
cd "$dir"
yarn npm publish --access restricted
cd -
env:
  YARN_NPM_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Note the auth env var also changes: Yarn reads `YARN_NPM_AUTH_TOKEN`, not `NODE_AUTH_TOKEN`. If the CI workflow also uses `npm view` (for version-exists checks), keep both env vars:

```yaml
env:
  NODE_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  YARN_NPM_AUTH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### Why this is non-obvious

The publish "succeeds" — no error from npm or the registry. The tarball is created and pushed. The failure only surfaces when a consumer outside the workspace tries to install the package. If you only test by installing from within the same Yarn workspace, you never see the problem.

### Why this fix

`yarn npm publish` is the Yarn-native publish path that understands `workspace:*`. The alternative — manually patching `package.json` before publish — is fragile and error-prone.

### Alternatives considered

- **Pre-publish script to sed `workspace:*` → real versions**: Works but fragile — must handle every dependency, restore after publish, handle failures mid-publish.
- **`yarn pack` + `npm publish tarball.tgz`**: Yarn's `pack` resolves workspace:*, but the two-step process adds complexity.
