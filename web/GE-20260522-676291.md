### Symptom

In a split-pane markdown diff viewer, word-level changed words are highlighted with `<mark>` elements. After the user drags the panel divider to resize, the highlighted words visually double in size — sometimes repeatedly. Each resize further inflates the nesting.

### Context

A `ResizeObserver` watches the divider element and calls `updateDiffMap()` on each resize. `updateDiffMap()` calls `annotateRendered()` (which only modifies CSS classes and `data-*` attributes — does not reset `innerHTML`) and then `annotateWordDiffs()` (which inserts `<mark>` elements by splitting text nodes).

On the first load, word-diff marks are correctly inserted. On each subsequent resize:

1. `annotateRendered()` runs — existing `<mark>` elements remain in the DOM unchanged (their text nodes are still there)
2. `annotateWordDiffs()` creates a `TreeWalker` over the element
3. The walker visits text nodes **inside** the existing `<mark>` elements
4. The same character-range computation fires — unchanged — and wraps the already-marked text in a *second* `<mark>`
5. Each resize adds another nesting layer

The bug only manifests on resize (or any other call to `updateDiffMap()` that bypasses `syncPanelDOM`). The normal file-load path calls `syncPanelDOM` → `marked.parse()` → resets `innerHTML` from scratch, clearing all marks before `annotateWordDiffs` runs.

### What was tried

Checking whether `annotateRendered()` cleaned up marks — it only removes `diff-del`/`diff-ins` classes and `data-diff-chunk` attributes, not `<mark>` elements.

### Root cause

Two code paths both call `updateDiffMap()`:
- **Safe path:** `renderMarkdown()` → `syncPanelDOM()` resets `innerHTML` → marks are cleared before re-annotation
- **Unsafe path:** `ResizeObserver` → `updateDiffMap()` directly, skipping `syncPanelDOM()` → stale marks survive

`annotateWordDiffs` had no guard against pre-existing marks.

### Fix

Strip existing word-diff marks at the start of `annotateWordDiffs()` before re-inserting:

```js
function annotateWordDiffs(chunks) {
  // Strip stale marks before re-annotating (ResizeObserver skips innerHTML reset)
  document.querySelectorAll('mark.diff-word-a, mark.diff-word-b').forEach(m => {
    m.replaceWith(document.createTextNode(m.textContent));
  });
  // ... rest of function unchanged
}
```

`m.replaceWith(document.createTextNode(m.textContent))` unwraps the mark cleanly — adjacent text nodes will be auto-coalesced by the browser.

### Why this is non-obvious

The symptom (enlarging text on resize) points toward CSS, not JavaScript. The root cause (two separate call paths with different DOM preconditions) only becomes visible when you trace every caller of `updateDiffMap()`. The ResizeObserver path is easy to overlook because it fires silently in the background during normal user interaction.

### Why this fix

Stripping marks at entry rather than trying to detect "clean" vs "dirty" DOM state is both correct and cheap — the strip is O(marks), which is small. The alternative (always calling `syncPanelDOM` before `annotateWordDiffs`) would cause the full markdown re-render on every resize, which is expensive and causes visual flicker.
