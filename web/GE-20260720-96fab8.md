### Symptom
`NotSupportedError: Failed to execute 'define' on 'CustomElementRegistry': the name "pages-modal" has already been used with this registry` — app renders blank, workbench never mounts.

### Context
A TypeScript monorepo package (`@casehubio/pages-primitives`) has a barrel `index.ts` that re-exports both side-effect-free a11y mixins and a modal component that calls `customElements.define()` at module scope. Another package (`pages-table`) imports only the a11y mixins but goes through the barrel, pulling in the modal registration as an unwanted side effect.

Downstream consumers alias `@casehubio/pages-*` packages to their source directories for development (Vite `resolve.alias`, esbuild `alias`). The alias catches direct imports but Vite's pre-bundling or esbuild's module resolution can resolve the same package through two paths: once via the alias (→ `src/`) and once via `node_modules/dist/` (from a pre-bundled transitive dependency). Two file paths = two module evaluations = two `customElements.define()` calls = crash.

### What was tried (didn't work)
- Adding `@casehubio/pages-primitives` alias to Vite and esbuild configs — crash persists because the transitive path through pre-bundled `pages-table` bypasses the alias
- Guarding with `if (!customElements.get('pages-modal'))` — prevents crash but silently creates two class instances with potentially divergent state and broken `instanceof`

### Root cause
The barrel `index.ts` couples unrelated concerns: `export * from './a11y/index.js'` and `export * from './modal/index.js'`. When `pages-table` imports `RovingTabindexMixin` from the barrel, it forces evaluation of `pages-modal.ts` which calls `customElements.define()`. This is unnecessary — `pages-table` only needs a11y mixins.

In aliased bundler setups, the same logical module can resolve to two different file paths (source vs dist), creating two module instances. The barrel coupling amplifies this: every consumer of any export from the package triggers every side effect.

### Fix
1. Add `"exports"` map to `package.json` with separate sub-paths:
   ```json
   "exports": {
     ".": { "types": "./dist/index.d.ts", "default": "./dist/index.js" },
     "./a11y": { "types": "./dist/a11y/index.d.ts", "default": "./dist/a11y/index.js" },
     "./modal": { "types": "./dist/modal/index.d.ts", "default": "./dist/modal/index.js" }
   }
   ```
2. Add `"sideEffects"` array marking which modules have `customElements.define()` calls
3. Update internal consumers to import from the narrowest sub-path: `import { RovingTabindexMixin } from '@casehubio/pages-primitives/a11y'`

### Why this is non-obvious
The symptom (duplicate registration crash) looks like a bundler aliasing problem. The obvious fix (add more aliases, guard the define call) treats the symptom. The actual root cause is a module design issue: the barrel couples unrelated concerns so every consumer triggers every side effect. Breaking the barrel with sub-path exports eliminates the transitive side effect entirely.

### Alternatives considered
- **More granular aliases in consumer config** — fragile; each new transitive dependency could introduce a new unaliased path
- **Guarded `customElements.define()` calls** — prevents crash but two class instances still exist with broken `instanceof` and divergent state
