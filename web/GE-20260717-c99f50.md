### Symptom
Child component panels render empty — no data, no error. For string fields (`this.item.caseId`), `undefined` is returned silently. For chained calls (`this.item.amount.toLocaleString()`), a `TypeError: Cannot read properties of undefined (reading 'toLocaleString')` appears only in console.

### Context
blocks-ui `detail-pane` emits the selected row from a `pages-table` as its `item` property. When the table is fed by `extractDataSet` (the pages-data pipeline), rows are `TypedRow` instances — class objects with `.text()`, `.number()`, `.date()` accessor methods and a `.cells` array. Child panels written to accept plain JSON objects (`{ caseId: "abc", amount: 245000 }`) silently break because `TypedRow` doesn't expose named properties.

### What was tried (didn't work)
1. Checking fetch mock responses — data was correct
2. Checking component lifecycle — `updated()` fired correctly
3. Adding console.log to `caseData` getter — returned a TypedRow object that looked populated in console but had no direct property access

### Root cause
`TypedRow` is a class instance created by `createTypedRow()` in pages-data. It stores cell values in an internal `cells: CellValue[]` array indexed by column position, not as named properties. When you write `typedRow.caseId`, JavaScript returns `undefined` (no such property exists) — it doesn't throw, doesn't warn, and looks like missing data rather than a type mismatch.

### Fix

**For ID/key extraction (all panels):** dual-path accessor handles both TypedRow and plain objects:

```typescript
get caseId(): string {
  return this.item?.text?.('caseId') ?? this.item?.caseId ?? '';
}
```

**For full object extraction (template rendering):** convert TypedRow to plain object in a getter:

```typescript
get caseData(): InvestigationSummary | null {
  if (!this.item) return null;
  if (this.item.text) {
    return {
      caseId: this.item.text('caseId'),
      status: this.item.text('status'),
      amount: this.item.number('amount'),
      currency: this.item.text('currency'),
    } as InvestigationSummary;
  }
  return this.item;
}
```

### Why this is non-obvious
TypedRow in console.log looks like it has data — the cells array is populated and the object appears non-empty. The failure mode is silent `undefined` for property access, which JavaScript treats as normal. There's no TypeScript error (the property type is `any` from the parent), no runtime warning, and no indication that you need accessor methods. The dual-path pattern (`?.text?.() ?? ?.field`) is necessary because the same component may receive either TypedRow (from extractDataSet pipeline) or a plain object (from direct fetch).
