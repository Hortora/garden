## Symptom

In Playwright tests, `raycaster.intersectObjects([...unitSprites.values()])` returns empty even though the sprite tag is confirmed present in the Map (`unitSprites.has(tag)` returns `true`). The test waits for `unitHasTag(tag)` to be true, then calls `clickUnit(tag)`, which immediately returns `false`.

## Context

A WebSocket message arrives, `syncUnitLayer()` adds the sprite to the `unitSprites` Map and calls `scene.add(sprite)`. The Playwright test detects this with `waitForFunction("() => window.__test.unitHasTag(tag)")`. But calling `clickUnit()` immediately after — before the Three.js animation loop has had a chance to call `renderer.render()` — produces no raycaster hits.

## Root cause

Three.js `Object3D.matrixWorld` is only updated when `renderer.render(scene, camera)` is called (or when `scene.updateMatrixWorld()` is called explicitly). A sprite newly added to the scene has `matrixWorld` at identity (position 0,0,0). The raycaster uses `matrixWorld` to locate the sprite in world space — so it looks for the sprite at the origin, not at its actual position, and misses.

`unitHasTag()` checks the Map (`unitSprites.has(tag)`) — the Map is updated synchronously when the WebSocket message is processed. The raycaster uses `matrixWorld` — which is only updated on the next render frame. These two checks go out of sync.

## Fix

Call `scene.updateMatrixWorld(true)` inside `clickUnit()` and `clickBuilding()` before raycasting:

```javascript
clickUnit: async (tag, isEnemy = false) => {
  const sp = isEnemy ? enemySprites.get(tag) : unitSprites.get(tag);
  if (!sp || !camera) return false;
  scene.updateMatrixWorld(true); // ← force matrixWorld update before raycasting
  const ndc = sp.position.clone().project(camera);
  // ...
}
```

## Why this is non-obvious

`unitHasTag()` returning `true` is strong evidence that the sprite is ready — the Map and the scene are both updated synchronously during WebSocket processing. Nothing in the Three.js API suggests that a scene-added object needs an additional `updateMatrixWorld()` call before it can be raycasted. The failure looks like a tag mismatch or a race condition in the test framework, not a Three.js internal state issue.

### Why this fix

`scene.updateMatrixWorld(true)` is cheap (traverses the scene graph once) and idempotent — safe to call every time. The alternative of waiting for a render frame (e.g. `requestAnimationFrame`) would require async test helpers and is not available in the Playwright `evaluate()` context.

### Alternatives considered

- Waiting for `unitCount() >= N` instead of `unitHasTag(tag)`: The count can be satisfied by stale sprites from a previous test, meaning the specific new sprite's `matrixWorld` may still be identity. Did not fix it.
- Using `page.waitForTimeout()` after `unitHasTag()`: Fragile and timing-dependent; works sometimes but not reliably across machines.
