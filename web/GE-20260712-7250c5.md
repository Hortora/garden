### Symptom
Components using `DataSourceMixin` that fetch non-tabular API responses
(e.g. `TrustScoreResponse`, `PagedResponse<T>`, `VerificationResult`) either:
- Throw `EXTRACTION_ERROR: Unrecognized data shape` at runtime
- Silently flatten nested objects to `"[object Object]"` TEXT columns, losing all domain structure

### Context
After `fetchSource` was rewritten to use the pages-data extraction pipeline
(`extractDataSet`), ALL fetched JSON responses pass through the pipeline.
The pipeline expects tabular data (arrays of flat objects) and auto-infers
columns. Single objects become 1-row datasets with lossy column values.
Nested fields like `capabilityScores: { auth: 0.9 }` become TEXT `"[object Object]"`.

Components that previously cast `this.dataSet as DomainType` worked because
`dataSet` was `unknown`. After typing it as `TypedDataSet | undefined`, the
cast is both type-incorrect and runtime-broken.

### What was tried (didn't work)
- Declaring `columns` in DataSourceAdapter options — doesn't help for non-tabular responses
- Casting `this.dataSet` to the domain type — TypedDataSet has no domain properties

### Root cause
`fetchSource` unconditionally runs `extractDataSet()` on every JSON response.
There is no bypass path for components that need raw domain objects.

### Fix
Override `createSourceFactory()` in the component to bypass the extraction pipeline:

```typescript
override createSourceFactory(): SourceFactory {
  return (url) => {
    let abort: AbortController | undefined;
    return {
      connect: (sink) => {
        abort = new AbortController();
        const signal = abort.signal;
        globalThis.fetch(url, { signal })
          .then(r => { if (!r.ok) throw new Error(`HTTP ${r.status}`); return r.json(); })
          .then((data: DomainType) => {
            if (signal.aborted) return;
            this._rawData = data; // store domain object for non-table rendering
            // Optionally create TypedDataSet from a subset via fromRows()
            sink.apply({ type: 'snapshot', dataset: { columns: [], rows: [] } });
          })
          .catch(err => {
            if (signal.aborted || err.name === 'AbortError') return;
            sink.error({ message: err instanceof Error ? err.message : String(err), permanent: true });
          });
      },
      disconnect: () => { abort?.abort(); abort = undefined; },
    };
  };
}
```

This preserves DataSourceMixin's lifecycle (loading, error, endpoint reactivity,
configure, refresh) while giving the component access to the raw domain response.

### Why this is non-obvious
The DataSourceMixin API presents itself as a generic data-fetching lifecycle
manager. Nothing in its interface signals that it will destructively transform
the response. The `dataSet` property name suggests "the data you fetched" not
"a tabular projection of your data." Components that worked before the
TypedDataSet migration break silently — the extraction either errors or
produces a TypedDataSet that has none of the domain properties the component needs.

### Alternatives considered
- Removing DataSourceMixin entirely and doing direct fetch — works but loses
  lifecycle management (loading, error, endpoint reactivity, configure, refresh)
- Adding a `rawMode` flag to DataSourceMixin — cleaner but a foundation change
  beyond the scope of the migration task
