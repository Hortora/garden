## Yarn 4 npm whoami returns 'No authentication configured' with valid GitHub Packages auth

**ID:** GE-20260706-7d5f53
**Stack:** Yarn 4.10.3, GitHub Packages
**What it is:** `yarn npm whoami --publish` fails with "No authentication configured" even when `npmAuthToken` is correctly set and `yarn npm publish` works fine.
**How discovered:** Trial and error during first-time GitHub Packages publish setup.

### Description
The `yarn npm whoami` command does not work with GitHub's npm registry (`https://npm.pkg.github.com`). It returns `YN0033: No authentication configured for request` regardless of whether auth is correctly configured. However, `yarn npm publish` succeeds with the same auth configuration.

This appears to be because GitHub Packages does not implement the npm `whoami` endpoint (`/-/whoami`) — or implements it in a way Yarn's client doesn't handle.

### How to use it / where it appears
Do not use `yarn npm whoami` to verify GitHub Packages auth. Instead, verify by attempting an actual operation:

```bash
# This fails even with valid auth:
GH_PACKAGES_TOKEN=$(gh auth token) yarn npm whoami --publish

# This works and proves auth is configured:
GH_PACKAGES_TOKEN=$(gh auth token) yarn npm publish --access public
```

Alternatively, verify the token resolves without hitting the registry:
```bash
GH_PACKAGES_TOKEN=$(gh auth token) yarn config get npmScopes
# npmAuthToken should show as ********
```

### Why it's not obvious
`yarn npm whoami` is the standard way to verify npm auth in Yarn 4. Every npm registry tutorial suggests it as the verification step. When it fails, the natural conclusion is "my auth is broken" — leading to debugging cycles on a config that is actually correct.

### Caveats
This is specific to GitHub Packages. Other npm registries (npmjs.com, Artifactory) support `whoami` normally.
