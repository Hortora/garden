### Symptom
TypeScript compile error when defining `set dataset(value: TypedDataSet)` on a class extending `HTMLElement`:
```
Cannot set property 'dataset' — 'dataset' is defined as a property in type 'HTMLElement' and has a getter but no setter.
```

### Context
Building Web Component custom elements that need to receive structured data from a parent/runtime. The natural property name `dataset` collides with the built-in `HTMLElement.dataset` getter.

### What was tried (didn't work)
- Declaring `set dataset(value: T)` — TypeScript rejects it (readonly getter conflict)
- Using `declare` to shadow the property — still incompatible types (`DOMStringMap` vs `T`)

### Root cause
`HTMLElement.dataset` is a built-in readonly property that returns a `DOMStringMap` — the programmatic interface to `data-*` HTML attributes. It has a getter but no setter. TypeScript's DOM lib correctly types it as readonly, and the compiler rejects any attempt to override it with an incompatible setter type.

### Fix
Use `dataSet` (camelCase) as the property name instead:

```typescript
abstract class CasehubElement<P> extends HTMLElement {
  private _dataset: TypedDataSet | undefined;

  set dataSet(value: TypedDataSet) {
    this._dataset = value;
    this.update();
  }
  get dataSet(): TypedDataSet | undefined {
    return this._dataset;
  }
}
```

The naming collision only affects `dataset`. No other commonly-desired property names (`props`, `theme`, `error`, `totalRows`) conflict with `HTMLElement` built-ins.

### Why this is non-obvious
`dataset` is the natural name for a component that receives data. The collision with `HTMLElement.dataset` (a rarely-used API for `data-*` attributes) isn't something developers think about until the compiler rejects it. The error message points at the type conflict but doesn't explain *why* `dataset` is special on `HTMLElement`.
