### What

When you need to split DOM text nodes at character boundaries — to wrap portions of text in new inline elements like `<mark>`, `<span>`, or `<a>` — collect all the target text nodes first via `TreeWalker`, then process and replace them in **reverse order** (last node first).

```js
function applyHighlights(el, ranges, className) {
  const walker = document.createTreeWalker(el, 4); // NodeFilter.SHOW_TEXT
  const nodes = [];
  let node, off = 0;
  while ((node = walker.nextNode())) {
    nodes.push({ node, start: off, end: off + node.length });
    off += node.length;
  }
  // Process in reverse — modifications to later nodes don't shift earlier offsets
  for (let i = nodes.length - 1; i >= 0; i--) {
    const { node, start, end } = nodes[i];
    const overlaps = ranges.filter(([rs, re]) => re > start && rs < end);
    if (!overlaps.length) continue;
    // ... split node into segments, replace with fragment of text + mark nodes ...
    node.parentNode.replaceChild(fragment, node);
  }
}
```

### Why it works

Each text node's position in the pre-collected array is indexed by its cumulative character offset relative to `el.textContent`. When you replace a text node with a `DocumentFragment`, the surrounding DOM shifts — but only *after* that node. Processing in reverse means you always modify the last (highest-offset) node first. Nodes at lower offsets, processed afterward, are still at exactly the positions you recorded before any mutation.

If you process forward (lowest offset first), replacing node[0] shifts the DOM positions of nodes[1], nodes[2], etc. — but your offset table was built on the pre-mutation positions. The subsequent nodes end up misaligned.

### Why this is non-obvious

The forward pass *appears* to work when ranges are small and the DOM is simple — text nodes may be short enough that shifts land on the same node. It fails silently on longer text or when multiple changes are in the same node neighbourhood. The symptom (wrong words highlighted, or highlights placed mid-word) is easy to mistake for a range computation bug rather than a mutation-ordering bug.

### Technique origin

This is the same technique used by code editors and rich-text annotation systems (ProseMirror, CodeMirror decorations) that need to apply overlapping or adjacent markup to a text stream without restructuring the element hierarchy.
