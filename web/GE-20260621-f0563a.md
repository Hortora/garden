### Symptom
A dropdown component needs TWO datasets — one for its form record (the selected value) and one for its option values (the list of choices). The data pipeline supports one dataset per component via `casehub-data-request` → `element.dataSet`.

### Context
The `casehub-data-request` event carries `{ element: VizTarget, lookup: DataSetLookup }`. The pipeline resolves the lookup and calls `element.dataSet = result`. A second request from the same component would overwrite the form data with the options data.

### Fix
Create a proxy object that implements VizTarget but routes `set dataSet` to a different property on the real component:

```typescript
private requestOptionsData(options: DataSetOptions): void {
  const self = this;
  const proxy = {
    set dataSet(ds: TypedDataSet) { self.optionsDataSet = ds; },
    set totalRows(_n: number) { /* ignored */ },
    set theme(_t: string) { /* ignored */ },
    set error(msg: string) { console.error("Options dataset error:", msg); },
  };

  this.dispatchEvent(new CustomEvent("casehub-data-request", {
    bubbles: true,
    composed: true,
    detail: {
      element: proxy,
      lookup: { dataSetId: options.dataset, operations: [] },
    },
  }));
}
```

The pipeline resolves the lookup normally and calls `proxy.dataSet = ...`, which routes to `self.optionsDataSet` — a separate property that triggers a re-render of the options list.

### Why this is non-obvious
The natural impulse is to add multi-dataset support to the pipeline (new event type, registry changes, component base class changes). The proxy pattern requires zero pipeline changes — it works within the existing single-dataset contract by presenting a different "face" to the pipeline for each request.
