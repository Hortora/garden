### Symptom
`TS2345: Argument of type 'typeof MyAbstractClass' is not assignable to
parameter of type 'Constructor<LitElement>'. Cannot assign an abstract
constructor type to a non-abstract constructor type.`

Compiles fine when the base is concrete `LitElement`. Fails only when the
base is an abstract subclass like `PagesElement`.

### Context
Lit mixins use the pattern `function MyMixin<T extends Constructor>(Base: T)`.
`Constructor` is typed as `new (...args: any[]) => T` — the `new` keyword
requires a concrete constructor. Abstract classes cannot satisfy this because
TypeScript prevents instantiating them directly.

This surfaces when composing mixins onto abstract base classes in a component
hierarchy — e.g. `LiveRegionMixin(PagesElement)` where `PagesElement` declares
`abstract renderContent()`.

### What was tried (didn't work)
- Casting: `LiveRegionMixin(PagesElement as unknown as Constructor<LitElement>)` — compiles but loses all type information from PagesElement
- Making PagesElement non-abstract — breaks the design contract (subclasses must implement `renderContent`)

### Root cause
TypeScript's abstract constructor constraint is by design (microsoft/TypeScript#36392).
The `new (...args) => T` type in `Constructor<T>` explicitly excludes abstract classes.
There is no `abstract new` constraint in TypeScript.

### Fix
Inline the mixin's functionality directly in the consuming class instead of
composing via the mixin function:

```typescript
// Instead of:
const Base = LiveRegionMixin(PagesElement);
export class MyForm extends Base { ... }

// Do:
export class MyForm extends PagesElement<Props> {
  private _liveRegion: HTMLElement | null = null;

  private announce(msg: string, priority: "polite" | "assertive" = "polite"): void {
    if (!this._liveRegion) {
      this._liveRegion = document.createElement("div");
      this._liveRegion.setAttribute("aria-live", priority);
      this._liveRegion.setAttribute("aria-atomic", "true");
      this._liveRegion.setAttribute("role", "status");
      Object.assign(this._liveRegion.style, {
        position: "absolute", width: "1px", height: "1px",
        overflow: "hidden", clip: "rect(0,0,0,0)", whiteSpace: "nowrap",
      });
      document.body.appendChild(this._liveRegion);
    }
    this._liveRegion.setAttribute("aria-live", priority);
    this._liveRegion.textContent = "";
    void this._liveRegion.offsetHeight;
    this._liveRegion.textContent = msg;
  }

  override disconnectedCallback(): void {
    super.disconnectedCallback();
    this._liveRegion?.remove();
    this._liveRegion = null;
  }
}
```

### Why this is non-obvious
The mixin works perfectly with concrete `LitElement`. The error only appears
when the base class is abstract — which is the natural design for component
hierarchies where subclasses must implement a render method. The fix (inlining)
feels like a regression from the clean mixin composition, but TypeScript's type
system leaves no alternative.
