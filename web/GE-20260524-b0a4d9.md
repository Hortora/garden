### Symptom

Word-level diff highlighting appears to work (mark count > 0) but only headings are annotated. Paragraphs are never tagged with `data-diff-chunk` and never receive word-level `<mark>` elements. Any test that checks `count > 0` passes; a fixture-specific test immediately fails.

### Context

Code that calls `marked.lexer(content)` and uses `token.raw.split('\n').length - 1` to compute how many lines a token occupies. The result drives a line-range comparison: `c.aStart < tokenEnd && c.aEnd > line`.

### What was tried

Checked `annotateRendered` logic against expected chunk boundaries. Looked correct on paper. Ran the full Playwright test suite ‚Äî count-based assertions ("> 0") passed. Only adding a fixture-specific assertion ("evaluates must be inside a mark") caught the regression.

### Root cause

In marked.js v9, **paragraph tokens do not include a trailing newline in their `raw` field**. The `raw` field is exactly the paragraph text with no `\n`. So:

```js
const rawLines = token.raw.split('\n').length - 1;  // ‚Üí 0 for paragraphs
const tokenEnd = line + rawLines;                    // ‚Üí tokenEnd == line
// Check: c.aStart < tokenEnd  ‚Üí  c.aStart < line  ‚Üí  FALSE when c.aStart == line
```

Heading tokens include both the heading line and the following blank line (e.g. `"# Title\n\n"`, rawLines=2). Space tokens between blocks absorb the inter-block blank lines. Only paragraphs produce rawLines=0.

The diagnostic that exposed it:
```js
const tokens = marked.lexer(panels.a.content);
tokens.map(t => ({ type: t.type, rawLines: t.raw.split('\n').length - 1 }))
// paragraph: { rawLines: 0 }
// heading:   { rawLines: 2 }
// space:     { rawLines: 2 }
```

### Fix

Use `Math.max(tokenEnd, line + 1)` as the upper bound of the chunk range check for non-space tokens. This ensures a paragraph starting exactly at `line` (rawLines=0) still matches its chunk, while headings and code blocks (rawLines‚â•1) are unaffected:

```js
const rawLines  = token.raw.split('\n').length - 1;
const tokenEnd  = line + rawLines;
if (token.type !== 'space') {
  const endForCheck = Math.max(tokenEnd, line + 1);  // ‚Üê fix
  const ci = chunks.findIndex(c =>
    c.op !== 'eq' && c[startKey] < endForCheck && c[endKey] > line);
  // ...
}
line = tokenEnd;  // advance by actual rawLines ‚Äî not endForCheck
```

Crucially, `line` still advances by `rawLines` (not `endForCheck`) so subsequent tokens remain correctly positioned.

### Why this is non-obvious

The `raw` field semantics differ per token type and this is not documented. A natural assumption is that `raw` always reconstructs the original source exactly, which would mean paragraphs include their trailing newline. In reality, marked distributes trailing newlines between the paragraph token and the subsequent space token ‚Äî the paragraph `raw` stops at the last character of the paragraph text.

### Why this fix

`Math.max(tokenEnd, line + 1)` is local to the check and doesn't disturb the `line` tracker. The alternative ‚Äî adjusting `rawLines` to be at least 1 for paragraphs ‚Äî would corrupt subsequent line positions because `line` advances by `rawLines`. Separating the "check bound" from the "advance amount" is the minimal, correct fix.

### Constraints

Applies to code that uses `token.raw.split('\n').length - 1` to map marked.js lexer tokens to source file line numbers. Does not affect rendering code that only uses `token.text`.

### Alternatives considered

- **Treat `rawLines=0` as 1 for line advance**: incorrect ‚Äî paragraph actually occupies 1 line, but the blank line *after* it is captured by the subsequent space token. Advancing by 1 would double-count and misalign all subsequent tokens.
- **Patch marked.js to include trailing newline in paragraph raw**: fragile, breaks on marked updates.

### Invalidation triggers

Revisit if marked.js v10+ changes how paragraph `raw` fields are constructed. Check with `marked.lexer("text\n\n").find(t => t.type === 'paragraph').raw`.
