## Vitest vi.useFakeTimers() silently replaces vi.stubGlobal() browser API stubs

**ID:** GE-20260706-2ad2b1
**Stack:** Vitest 3.2.6
**Symptom:** Custom `requestAnimationFrame` stub (installed via `vi.stubGlobal()` in `beforeEach`) stops collecting callbacks when `vi.useFakeTimers()` is called in a specific test. The stub's callback array stays empty — RAF calls go to Vitest's fake timer implementation instead.
**Context:** Node environment tests for browser code that uses `requestAnimationFrame` for batching. Some tests also need fake timers for `setTimeout`-based reconnection logic.

### What was tried (didn't work)
- Calling `vi.stubGlobal("requestAnimationFrame", ...)` in `beforeEach` and `vi.useFakeTimers()` in the test — the stub was silently overridden
- Calling `vi.useRealTimers()` before flushing the RAF queue — the custom stub was already gone by then

### Root cause
`vi.useFakeTimers()` installs its own implementations for all timer-related globals, including `requestAnimationFrame`. This overwrites any previously installed `vi.stubGlobal()` stub for the same global. The replacement happens silently — no warning, no error.

### Fix
Two approaches:

1. **Don't mix** — in tests that need fake timers, don't rely on your custom RAF stub. Verify structural correctness (e.g. listener registration) instead of event dispatch.

2. **Re-install after** — call `vi.stubGlobal("requestAnimationFrame", ...)` AFTER `vi.useFakeTimers()` so your stub takes precedence. Call `vi.useRealTimers()` in cleanup.

### Why this is non-obvious
The override is completely silent. `vi.stubGlobal()` and `vi.useFakeTimers()` appear to be independent mechanisms, and their interaction is not documented. The failure mode is a test that never calls your RAF callback — which looks like a bug in the code under test, not a test infrastructure problem.
