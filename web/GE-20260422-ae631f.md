## Symptom

Setting `plane.visible = false` on 4096 fog overlay meshes eliminates their GPU draw calls,
but per-frame CPU time stays high. Toggling visibility on and off produces no meaningful
FPS improvement. Scene renders at ~47 FPS instead of expected ~60 FPS.

## Context

A 64×64 tile fog-of-war system creates 4096 `THREE.Mesh` objects (one per tile) with
`transparent: true` and `renderOrder: 5`. In mock mode (no real visibility data), all
planes are set `visible = false`. Despite this, performance degrades significantly compared
to a scene with no fog planes.

## Root Cause

Three.js r128 still traverses **all** scene objects on every frame regardless of
`object.visible`. The traversal loop visits each object to check `visible`, then skips
the render call. With 4096 objects, this is 4096 traversal iterations per frame even when
nothing is rendered. The transparent object sort (which runs before rasterisation) also
considers all objects in the `renderList`. The GPU draw call cost is eliminated, but the
CPU traversal and sort cost is not.

## Fix

Do not add objects to the scene at all when they will never be needed. For mode-conditional
geometry (e.g. fog planes only needed in emulated mode, not mock mode), gate creation:

```js
let hasRealTerrain = false;
// ... terrain fetch ...
if (r.ok) { hasRealTerrain = true; /* parse walls etc */ }

// Only create fog planes when there is actual visibility data to drive them
if (hasRealTerrain) {
  const fogMat = new THREE.MeshBasicMaterial({ … });
  for (let gz = 0; gz < GRID_H; gz++) {
    for (let gx = 0; gx < GRID_W; gx++) {
      const plane = new THREE.Mesh(fogGeo, fogMat.clone());
      scene.add(plane);
      fogPlanes.set(`${gx},${gz}`, plane);
    }
  }
}
```

This removes 4096 objects from the scene graph entirely in mock mode, roughly halving
per-frame traversal cost and restoring 60 FPS.

### Why This Is Non-Obvious

`visible = false` reads as "this object is not there." The Three.js API gives no indication
that invisible objects still participate in scene traversal. The natural assumption — and
the documented behaviour of `visible` — is that the renderer skips invisible objects
entirely. The performance cost is invisible in the GPU profiler (no draw calls) but shows
up only in CPU frame time.

### Why this fix

Removing objects from the scene is more reliable than `visible = false` for permanently
unused geometry, because it eliminates the traversal entirely rather than short-circuiting
partway through. The downside is that objects must be recreated if the mode changes; for
static mode decisions (mock vs emulated determined at startup), this cost is paid once.
