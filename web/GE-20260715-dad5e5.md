### What exists

When a `package.json` test script uses `"test": "vitest"` (watch mode) instead of `"test": "vitest run"`, the vitest process starts, runs all tests, then waits for keyboard input ("press h to show help, press q to quit"). In a TTY terminal this is expected interactive behaviour.

### Symptom

`yarn workspaces foreach -A run test` starts executing packages sequentially. After reaching a package with `"test": "vitest"`, the entire suite appears to hang indefinitely — zero output, no error, process alive. The batch runner waits for the child process to exit, which never happens.

### Why this is non-obvious

Each package passes when tested individually (`yarn --cwd components/foo vitest run` or even `yarn --cwd components/foo vitest` in a terminal). The hang only manifests during batch execution where stdin is piped/redirected. Four packages in a 20-package monorepo had this — the first three happened to run before the watch-mode package, masking the issue until the full suite was run.

### Root cause

vitest defaults to watch mode when no `--run` flag is passed. In a terminal, it detects the TTY and shows interactive controls. In a piped/batch context, it still waits for input but produces no visible prompt — it silently blocks.

### Fix

Always use `"test": "vitest run"` in package.json test scripts. The `run` flag exits after tests complete.

```json
{
  "scripts": {
    "test": "vitest run"
  }
}
```
