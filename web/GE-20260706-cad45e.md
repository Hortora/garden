## Yarn 4 project-level .yarnrc.yml npmScopes overrides home-level entirely — no property merge

**ID:** GE-20260706-cad45e
**Stack:** Yarn 4.10.3
**Symptom:** `npm publish` fails with "No authentication configured" despite `npmAuthToken` being set in `~/.yarnrc.yml` under the correct scope.
**Context:** Two-repo setup where project `.yarnrc.yml` declares `npmScopes.casehubio.npmRegistryServer` but no `npmAuthToken`. Home-level `~/.yarnrc.yml` has the matching `npmScopes.casehubio.npmAuthToken`.

### What was tried (didn't work)
- Put `npmAuthToken` in `~/.yarnrc.yml` under `npmScopes.casehubio` — Yarn silently ignored it because the project-level config also declares the `casehubio` scope
- `yarn config get npmScopes` showed the token as masked (`********`) from home-level, but publish still failed — the project scope took full precedence at resolution time

### Root cause
Yarn 4 does NOT merge individual properties within an `npmScopes.<name>` block across config layers. If the project-level `.yarnrc.yml` declares `npmScopes.casehubio` (even with just `npmRegistryServer`), the entire scope block from the home-level config is ignored — including `npmAuthToken`.

### Fix
Use environment variable interpolation in the project-level `.yarnrc.yml`:

```yaml
npmScopes:
  casehubio:
    npmAuthToken: "${GH_PACKAGES_TOKEN}"
    npmRegistryServer: "https://npm.pkg.github.com"
```

Then run commands with the env var set:
```bash
GH_PACKAGES_TOKEN=$(gh auth token) yarn npm publish
```

Do NOT put the token directly in the project-level file — it is tracked by git.

### Why this is non-obvious
`yarn config get npmScopes` misleadingly shows the home-level token as masked, suggesting it was resolved. The actual failure only surfaces at publish time. A developer would reasonably expect Yarn to merge properties from both config layers for the same scope — that is how most config cascade systems work.
