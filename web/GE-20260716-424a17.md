### Symptom
Native `<select>` dropdown popup appears far from the trigger element —
displaced vertically and horizontally, often in the middle of the page.

### Context
Occurs when a `<select>` is rendered inside multiple layers of shadow DOM
(e.g. shell component → page component → nav component) and the containing
element has `overflow: auto` with scrolled content.

### What was tried (didn't work)
Standard CSS on the `<select>` (width, padding, border) — styling is fine,
but the popup position is determined by the browser's native widget layer.

### Root cause
Browsers calculate native `<select>` popup positions relative to the viewport,
but miscalculate when the element is inside nested shadow roots combined with
a scrolled `overflow: auto` ancestor. The popup offset reflects the unscrolled
document position rather than the actual visual position.

### Fix
Replace native `<select>` with a custom dropdown: a `<button>` trigger with
an absolutely-positioned `<ul>` panel inside the same shadow root. The panel
uses `position: absolute; top: 100%` relative to a `position: relative` wrapper,
which shadow DOM resolves correctly.

Add ARIA attributes (`role="combobox"`, `aria-expanded`, `role="listbox"`,
`role="option"`, `aria-selected`) and keyboard handling (ArrowUp/Down, Enter,
Escape) to match native `<select>` accessibility. Register a document-level
click listener to close on outside click, removed in `disconnectedCallback`.

### Why this is non-obvious
The symptom (displaced popup) doesn't point to shadow DOM nesting as the cause.
A native `<select>` is the expected accessible solution for compact channel
selection. The positioning works correctly in a single shadow root — it only
breaks with multiple nested shadow roots plus a scrolled container, which
is a common pattern in Web Component shell architectures.
