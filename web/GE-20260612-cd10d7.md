### Symptom
Expressions like `value * 100` or `$uppercase(value)` return `undefined` when evaluated via JSONata — no error, no warning. The expression compiles, evaluates, and returns a result. The result is just wrong.

### Context
JSONata's `evaluate(data, bindings)` takes two parameters:
- **First arg (data):** the JSON document to query. Paths resolve without prefix: `value`, `account.name`
- **Second arg (bindings):** named variables. Accessed with `$` prefix: `$value`, `$x`

When migrating from Java's `Global.eval()` (which uses plain `value`) to JSONata, the natural instinct is to pass value as a binding: `evaluate({}, { value: "42" })`. This compiles and runs — but the expression `value * 100` looks for `value` in the data object (first arg), which is empty. Result: `undefined`. No error thrown.

### What was tried (didn't work)
1. Passing value as a binding and using `value` in expressions → undefined
2. Checking JSONata error output → no errors (the expression is syntactically valid, just queries empty data)

### Root cause
JSONata bindings (second arg) are accessed with `$` prefix (`$value`). Data paths (first arg) are accessed without prefix (`value`). The two namespaces are separate. Using the wrong parameter slot silently produces undefined instead of an error.

### Fix
Pass the value as **data** (first argument), not as a binding:

```typescript
// Wrong — expressions must use $value
const result = await compiled.evaluate({}, { value: cellValue });

// Right — expressions use plain value
const result = await compiled.evaluate({ value: cellValue });
```

With the data approach, expressions like `value * 100`, `$uppercase(value)`, and `value ? value : 'N/A'` all work without the `$` prefix — matching the Java convention.

### Why this is non-obvious
Both API forms are documented. The `$` prefix for bindings is mentioned in the JSONata docs, but nothing warns you that using bindings with a plain-name expression silently returns undefined instead of throwing. A developer familiar with JSONata's expression syntax but not its binding mechanics will assume `{ value }` in either parameter slot makes `value` accessible — it does, but only in the first.

### Alternatives considered
- Use bindings and rewrite all expressions to use `$value` — breaks backward compatibility with existing dashboard YAML
- Use `expression.assign("value", ...)` — same `$` prefix problem, also mutates the compiled expression
