### Symptom
Clicking a row in a text-filtered table selects the wrong record. The form shows "Alice" when "Bob" was clicked.

### Context
A table component emits a filter event when a row is clicked. The event carries data identifying the clicked row. After client-side text filtering, the display order differs from the dataset order.

### What was tried (didn't work)
Replaced `row: clickedRow` (full TypedRow object) with `rowIndex: rowIdx` (display-order integer) to simplify the event shape. Tests for unfiltered tables passed. Tests for filtered tables broke silently — the runtime used `ds.rows[rowIndex]` which returns the wrong row when display order differs from dataset order.

### Root cause
`rowIndex` is the index in `displayRows` (after filtering/sorting/pagination), not in `dataset.rows` (the original unfiltered data). The runtime's event handler falls back to `ds.rows[rowIndex]` when no `row` object is present — this works only when display order matches dataset order.

### Fix
Include BOTH in the event detail:
```typescript
detail: {
  columnId,
  rowIndex: rowIdx,   // display-order index (for UI consumers)
  row: clickedRow,    // actual TypedRow object (for runtime record selection)
  reset: false,
  group: props.filter?.group,
}
```
The runtime uses `row` (authoritative); UI consumers can use `rowIndex` for display purposes.

### Why this is non-obvious
Tests with unfiltered tables pass either way — the bug only manifests after text filtering changes the display order. The error is silent: the wrong record loads without any error message.
