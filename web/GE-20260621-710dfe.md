### Symptom

Setting `rootDir: "src"` in a shared base tsconfig (`packages/pages-tsconfig/tsconfig.json`) causes TS6059 "File is not under 'rootDir'" for every source file in extending packages. The error reports rootDir as `packages/pages-tsconfig/src/` — the base config's directory, not the extending package's directory.

### Context

A TypeScript monorepo with a shared config package (`@casehub/pages-tsconfig`) extended by all workspace packages via `"extends": "@casehub/pages-tsconfig/tsconfig.json"`. Adding `composite: true` to the base config required `rootDir` to be set. The natural assumption was to set `rootDir: "src"` in the same base config since all packages use `include: ["src"]`.

### What was tried (didn't work)

Setting `rootDir: "src"` in the base tsconfig. Expected it to resolve relative to each extending config's directory (like `include` does). Instead, TypeScript resolved it relative to the base config file's location.

### Root cause

TypeScript resolves `rootDir` relative to the config file that **declares** it, not the one that **extends** it. This is different from `include` and `outDir`, which resolve relative to the extending config. The distinction is not prominently documented.

### Fix

Set `rootDir: "src"` in each individual package's `tsconfig.json`, not in the shared base config:

```json
{
  "extends": "@casehub/pages-tsconfig/tsconfig.json",
  "compilerOptions": {
    "rootDir": "src"
  },
  "include": ["src"]
}
```

### Why this is non-obvious

Most TypeScript monorepo guides recommend putting all shared compiler options in a base config. `rootDir` looks like any other compiler option. The fact that it resolves relative to the declaring config (unlike `include`) is a subtle distinction that only surfaces at scale.
