### Symptom
After using the inbox for 10-30 seconds, clicking filter pills stops working — badges stay active, the list doesn't change, the UI appears "locked." No error is visible in the UI. The browser console shows: `Unhandled Promise Rejection: TypeError: undefined is not an object (evaluating 'item.item.status')`.

### Context
An inbox component fetches work items from `/workitems/inbox` (returns `WorkItemRootResponse[]` — each item wrapped as `{item: WorkItemResponse, childCount, ...}`). When SSE events fire (every 5-10 seconds), the inbox fetches the updated item by ID from `/workitems/{id}` and adds/updates it in the list.

The problem: `/workitems/{id}` returned a raw `WorkItemResponse` (no wrapper), but the inbox cast it as `WorkItemRootResponse` and accessed `.item.status`. Since the raw response has no `.item` property, `.item` is `undefined` → TypeError on `.item.status`.

### What was tried (didn't work)
- Investigating filter toggle logic — toggles worked correctly in unit tests
- Testing rapid click sequences — automated Playwright tests couldn't reproduce it
- Adding more filter state tests — all passed
- Checking SSE event handling — events fired correctly

### Root cause
Two bugs compounding:

1. **Mock/API shape mismatch:** `/workitems/{id}` returned `WorkItemResponse` but the component expected `WorkItemRootResponse` (with `{item: ...}` wrapper). The TypeScript cast (`as WorkItemRootResponse`) compiled but was wrong at runtime.

2. **Unhandled rejection kills the render pipeline:** The TypeError was thrown inside `handleItemAppears()` (an async method called from the SSE handler). The rejection was unhandled — no try/catch around the property access. Once this error fired, `getFilteredItems()` threw on every subsequent render cycle because the corrupted item (with no `.item` property) was in the items array. Every click triggered a re-render → crash → no UI update → "locked."

### Fix

Two-part fix:

```typescript
// 1. Handle both wrapped and unwrapped responses defensively
const raw = await response.json() as Record<string, unknown>;
const newItem: WorkItemRootResponse =
  raw.item && typeof (raw.item as Record<string, unknown>).id === 'string'
    ? raw as unknown as WorkItemRootResponse
    : { item: raw as unknown as WorkItemResponse,
        childCount: 0, completedCount: null,
        requiredCount: null, groupStatus: null };

// 2. Fix the API/mock to return the correct shape
// GET /workitems/{id} should return the wrapped shape
return json({ item, childCount: 0, completedCount: null,
              requiredCount: null, groupStatus: null });
```

### Why this is non-obvious
The symptom ("filters lock up after clicking") points at filter state management. The root cause (wrong response shape from a fetch triggered by SSE) is in a completely different code path. The TypeError only appears in the console — the UI shows no error. The bug is time-dependent (only fires after SSE events arrive, typically 5-10 seconds in), making it intermittent from the user's perspective. Automated tests couldn't reproduce it because they don't run real SSE event loops.

### Alternatives considered
- **Strict TypeScript runtime validation (e.g. zod)** — would catch the shape mismatch at the fetch site instead of the render site, but adds a dependency
- **Try/catch in getFilteredItems** — masks the symptom without fixing the cause; corrupted items stay in the array
- **Type guard before list insertion** — catches it but doesn't fix the API returning the wrong shape
