### Symptom
Building a data table that needs both virtual scrolling (render only visible rows)
and cell spanning (rowspan + colspan). The per-row CSS Grid model — each row is an
independent `display: grid` container — makes colspan work but rowspan impossible:
cells cannot span across independent grid containers.

### Context
The industry standard (AG Grid, Ant Design) works around this with position hacks:
AG Grid switches from `transform: translateY()` to `top` positioning
(`suppressRowTransform`) so spanned cells can overlay subsequent rows via z-index.
Ant Design uses `extraRender` to paint off-screen span origins as separate elements.
Both approaches require manual height calculation, z-index management, and special
cases for column alignment.

### Root cause
Per-row grid containers are the wrong abstraction. Each row is a self-contained
layout context — cells inside row A have no relationship to cells in row B. This
is the fundamental constraint, and no amount of positioning trickery fixes it cleanly.

### Fix
Use a **single CSS Grid** on the scroll container body:

```css
.body-content {
  display: grid;
  grid-template-columns: 200px 1fr 120px;
  grid-template-rows: repeat(10000, 48px); /* full scrollable height as metadata */
}
```

Row wrappers use `display: contents` (invisible to layout, preserved for ARIA):

```html
<div role="row" style="display: contents;" data-row="500">
  <div class="cell" style="grid-row: 501; grid-column: 1;">A</div>
  <div class="cell" style="grid-row: 501 / span 3; grid-column: 2;">MERGED</div>
  <div class="cell" style="grid-row: 501; grid-column: 3;">B</div>
</div>
```

Virtual scroll works because:
1. `repeat(N, 48px)` defines the scrollable height as track metadata — the browser
   doesn't allocate DOM for empty tracks
2. Fixed-size tracks make cell positioning O(1): position = track_index × track_size
3. Only cells for visible rows are rendered, placed at their actual grid-row
4. Spanning cells from above the viewport are included via boundary scan (check if
   `startIndex` cell is suppressed → read origin row → extend render window)
5. No `translateY` needed — the grid engine handles scroll positioning natively

### Why this is non-obvious
Developers familiar with virtual scroll tables assume the rendering model must be
per-row (one DOM element per row, positioned with translateY). The idea of creating
10,000 empty grid tracks sounds expensive, but uniform fixed-size tracks are just
metadata — `(count, size)` — and the browser doesn't enumerate them. The cost is
proportional to rendered cells, not total tracks. This makes CSS Grid's native
spanning available without any of the position hacks the industry has standardised on.

### Why this fix
The alternatives (AG Grid's z-index overlays, Ant Design's extraRender) all maintain
the per-row model and add complexity to work around it. The single-grid model removes
the constraint entirely — spanning is a CSS Grid primitive that just works. The
trade-off is that row-level CSS (`:hover`, `:focus outline`, `::part()`) doesn't work
on `display: contents` elements and must be applied per-cell via JS state, but this
is a few lines of code and is actually more correct for spanned cells (hovering a
4-row span explicitly controls which rows highlight).

### Alternatives considered
- **AG Grid legacy approach** — `suppressRowTransform` + `top` positioning + z-index
  overlays. Proven but requires manual height calculation, breaks with auto-height,
  and the z-index stacking creates fragile rendering order dependencies.
- **Absolute positioning overlay layer** — keep per-row grids for normal cells,
  render spanned cells as absolutely positioned overlays. Preserves the row model
  but decouples spanned cells from grid layout (column resize needs manual recalc).
- **Dual rendering mode** — per-row grids normally, switch to single grid when
  spanning is configured. Avoids the universal migration cost but creates two
  rendering paths that every future feature must support.
