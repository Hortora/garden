### Symptom
`Failed to resolve import "@casehubio/pages-data/dist/dataset/types.js"` when running vitest, even though the alias `'@casehubio/pages-data'` is configured. The test suite fails to collect — no tests run at all.

### Context
Monorepo with workspace packages importing from peer packages via deep paths like `@casehubio/pages-data/dist/datasource/types.js`. The vitest config has an alias for the base package name.

### What was tried (didn't work)
- Adding the base package alias as an object key: `{ '@casehubio/pages-data': '/path/to/src' }` — only matches the exact string, not subpath imports
- Checking import paths — they are correct in the source files
- The error message doesn't mention aliases at all — it looks like a missing file, not a misconfigured alias

### Root cause
Vite's `resolve.alias` object format only matches exact string keys. A key `'@casehubio/pages-data'` matches `import '@casehubio/pages-data'` but NOT `import '@casehubio/pages-data/dist/dataset/types.js'`. The deep import bypasses the alias entirely and tries to resolve against node_modules, which may not have the package installed (workspace link points elsewhere).

### Fix
Use the array format with a regex entry for deep imports:

```typescript
resolve: {
  alias: [
    { find: /^@casehubio\/pages-data\/dist\/(.*)/, replacement: path.resolve(__dirname, '../pages-data/src/$1') },
    { find: '@casehubio/pages-data', replacement: path.resolve(__dirname, '../pages-data/src') },
  ],
}
```

The regex entry catches all `dist/` subpath imports and rewrites them to source paths. The exact-string entry catches bare imports. Both are needed.

### Why this is non-obvious
The object format works for the base import and gives no warning that deep imports are silently unmatched. The error message points to a missing file, not an alias problem. Other configs in the same repo using array format work correctly, making the discrepancy hard to spot.
