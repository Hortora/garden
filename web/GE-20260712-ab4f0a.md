### Symptom
esbuild production build fails with `Could not resolve "/path/to/pages-data/src/dist/dataset/types.js"` — the resolved path contains a spurious `/dist` segment inside `/src`.

### Context
Monorepo packages aliased for bundling. The Vite dev server uses source-pointing aliases (`@pkg → pkg/src`) with array-form ordering to handle `/dist` deep imports. Attempting the same pattern in esbuild's `alias` option.

### What was tried (didn't work)
Pointing esbuild aliases to `/src` (same as Vite): `"@casehubio/pages-data": resolve(PAGES, "pages-data/src")`. Cross-package imports like `@casehubio/pages-data/dist/dataset/types.js` become `pages-data/src/dist/dataset/types.js`.

Adding separate `/dist` alias entries — esbuild's `alias` only matches the package name portion, not deep path prefixes. `@casehubio/pages-data/dist` is not recognized as a separate alias key.

### Root cause
esbuild's `alias` option matches the package name (everything before the first `/` after the scoped package name) and replaces it. For `@casehubio/pages-data/dist/dataset/types.js`, it matches `@casehubio/pages-data` and appends `/dist/dataset/types.js` to the replacement. Unlike Vite's array-form, esbuild does not support matching longer sub-path prefixes as separate alias entries.

### Fix
For production builds, point aliases to the package root (not `/src`):
```js
alias: {
  "@casehubio/pages-data": resolve(PAGES, "pages-data"),
}
```
This way `@casehubio/pages-data/dist/dataset/types.js` resolves to `pages-data/dist/dataset/types.js` — the pre-compiled JS files in `dist/` are used directly. The Vite dev server continues using source-pointing aliases for HMR and TypeScript support.

### Why this is non-obvious
The natural instinct is to use the same alias target for both Vite and esbuild. The different alias matching semantics (Vite supports sub-path prefixes via array form; esbuild only matches the package name) require different strategies for the same monorepo layout.
