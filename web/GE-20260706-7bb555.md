### Symptom
Virtual scroll renders only buffer rows (5 of 100) in the browser, but all tests pass. The test checks 'fewer than 100 rows rendered' which is true for both 5 rows (bug) and 15 rows (correct).

### Context
A Lit Web Component uses computeScrollWindow(scrollTop, containerHeight, rowHeight, rowCount, bufferSize) to determine which rows to render. containerHeight comes from body.clientHeight. In JSDOM, elements have no layout — clientHeight is always 0.

### What was tried (didn't work)
Testing that "rows.length < totalRows" — passes regardless of containerHeight because even with containerHeight=0, the buffer produces some rows. The test proves virtual scroll activates but not that it renders the correct window.

### Root cause
JSDOM has no layout engine. All dimension properties (clientHeight, scrollHeight, offsetHeight, getBoundingClientRect) return 0. computeScrollWindow(0, 0, 48, 100, 5) returns endIndex=5 (just the buffer) instead of endIndex=15 (10 visible + 5 buffer). Tests that check "fewer than N" pass for any positive number less than N.

### Fix
Three-layer measurement strategy:
1. Update _containerHeight on every scroll event: `if (target.clientHeight > 0) this._containerHeight = target.clientHeight`
2. Add ResizeObserver on the .body element in firstUpdated — fires when the element gets its flex-derived height
3. Add `requestAnimationFrame(() => this._updateContainerHeight())` in firstUpdated for deferred post-layout measurement

For tests needing specific window sizes, mock clientHeight:
```typescript
Object.defineProperty(body, 'clientHeight', { value: 400, configurable: true });
body.dispatchEvent(new Event('scroll'));
```

### Why this is non-obvious
The test looks correct — it verifies that virtual scroll activates and renders fewer rows than the dataset. The bug is that "fewer" means 5 (only buffer) instead of 15 (buffer + visible), and no JSDOM test can distinguish them without explicit clientHeight mocking.
