### What exists

In a Lit SPA showcase app with hash-based routing, each page is a custom element that renders when the hash matches. When a page overrides `globalThis.fetch` in `connectedCallback()` to provide mock data, and the user navigates to a different page, the mock stays in place because the SPA framework may not call `disconnectedCallback()` on the previous page (or it calls it after the new page has already connected).

### Symptom

After visiting a page with a mock fetch (e.g. a GDPR erasure form that mocks POST responses), all other pages in the SPA show empty data, "Loading..." forever, or error states. A full page reload fixes it.

### Why this is non-obvious

The mock page works perfectly in isolation. The bug only manifests through navigation — visiting the mock page then navigating away. The symptom appears on the *destination* page (e.g. Work Item Inbox shows no data) not the source page, so the root cause is not where you look.

### Root cause

`globalThis.fetch` is a singleton. Overriding it in `connectedCallback()` without restoring in `disconnectedCallback()` permanently replaces the real fetch for the entire page lifetime. SPA navigation doesn't trigger a page reload, so the override persists.

### Fix

Save and restore `globalThis.fetch` around the component lifecycle:

```typescript
private _originalFetch: typeof globalThis.fetch | null = null;

override connectedCallback(): void {
  super.connectedCallback();
  this._originalFetch = globalThis.fetch;
  globalThis.fetch = this._mockFetch as unknown as typeof fetch;
}

override disconnectedCallback(): void {
  super.disconnectedCallback();
  if (this._originalFetch) {
    globalThis.fetch = this._originalFetch;
    this._originalFetch = null;
  }
}
```
