### Symptom
Toggling a panel's `display: none` in a CSS Grid layout with `fr`-unit columns leaves
an empty gap where the panel was. The remaining panels do NOT expand to fill the space.
No error, no warning — the layout just has dead space.

### Context
Building a workbench with dockable panels that can be toggled visible/hidden.
The layout uses `grid-template-columns: 3fr 2fr 1fr` for proportional panel sizing.
Hiding a panel with `display: none` removes it visually but the grid track persists.

### Root cause
CSS Grid `fr` tracks are allocated based on the grid template definition, not on
the visibility of items occupying them. A `display: none` item is removed from the
formatting context but its track — defined by `grid-template-columns` — remains.
The track has no content so it renders as empty space.

Flex layout works differently: `display: none` on a flex item removes it from the
flex formatting context entirely. The remaining items' `flex` values redistribute
across the available space automatically.

### Fix
Use flex layout (`display: flex`) instead of CSS Grid for any container whose
children may be dynamically shown/hidden:

```css
/* Don't use this for toggleable children */
.container { display: grid; grid-template-columns: 3fr 2fr 1fr; }

/* Use this instead */
.container { display: flex; }
.child-a { flex: 3; }
.child-b { flex: 2; }
.child-c { flex: 1; }
/* display: none on child-b → child-a and child-c absorb the space */
```

### Why this is non-obvious
CSS Grid and flex both support proportional sizing (`fr` vs `flex`). A developer
familiar with flex would expect grid to behave the same way when hiding children.
The grid spec's track persistence is correct behaviour but surprising — there is
no visual or console indication that the track is still allocated.

### Why this fix
Flex is the minimal correct choice — it handles show/hide natively with no extra
code. Alternatives like dynamically rewriting `grid-template-columns` on every
toggle add complexity, require tracking which children are visible, and are
error-prone with nested layouts.
