## Symptom

`EventSource` handler registered with `addEventListener('mesh-update', handler)` never fires. No errors in the console. The `EventSource` connects successfully, the browser's network tab shows data arriving from the server at the expected interval, and `readyState` stays at `1` (OPEN). The handler is simply never called.

## Context

Server sends SSE responses using a framework that auto-wraps payload strings as bare `data:` fields (e.g. RESTEasy Reactive `Multi<String>` with `text/event-stream`). The client code registers a named-event listener expecting the server to tag events with a matching name.

## What was tried

Verified the EventSource URL was correct, confirmed the response headers showed `Content-Type: text/event-stream`, confirmed data was visible in the network tab. The connection looked healthy in every observable way.

## Root cause

`EventSource` has two distinct dispatch paths:

- **`source.onmessage`** — fires for SSE events that have no `event:` field, or have `event: message`
- **`source.addEventListener('name', handler)`** — fires only when the server emits `event: name\n` in the SSE frame

These do not overlap. A server that sends:
```
data: {"channels":...}

```
(no `event:` line) will trigger `onmessage` but never trigger any named-event listener. The listener is silently ignored — no error, no fallback.

## Fix

Use `source.onmessage` when the server sends unnamed SSE events:

```javascript
// ❌ Silently receives nothing if server sends unnamed events
source.addEventListener('mesh-update', e => {
    const data = JSON.parse(e.data);
    panel.update(data);
});

// ✅ Fires for unnamed events (no event: field in the SSE frame)
source.onmessage = e => {
    const data = JSON.parse(e.data);
    panel.update(data);
};
```

Only use `addEventListener('name', handler)` when you also control the server and it explicitly emits `event: name\n` in the SSE frame. In the browser SSE spec, `event: message` and no `event:` field are both handled by `onmessage`; any other value requires the matching `addEventListener`.

## Why this is non-obvious

The EventSource API looks symmetric — `addEventListener` appears to be the "proper" way to subscribe to events. The connection appears fully functional (open state, data flowing), giving no signal that the handler path is wrong. The bug is architectural: `onmessage` and named-event listeners address different event types and only one of them matches unnamed server output.
