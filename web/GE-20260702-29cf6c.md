## Summary

Java REST endpoint declared `@Consumes(MediaType.TEXT_PLAIN)`. TypeScript client sent
`Content-Type: application/json`. Both sides' unit/integration tests passed — the Java
test sent `text/plain`, the TypeScript test mocked `fetch`. At runtime: HTTP 415
Unsupported Media Type.

## Symptom

No test failure anywhere. The mismatch is invisible until a real browser hits the
real server.

## Context

Cross-stack project where the Java backend and TypeScript frontend are in the same
monorepo but have independent test suites (Quarkus @QuarkusTest + vitest).

## Root cause

Each test suite validates its own side of the contract in isolation. The Java test
sends the content type the Java endpoint expects. The TypeScript test mocks fetch
entirely. Neither test exercises the actual wire contract between them.

## Fix

Align the content types — either change the Java side or the TypeScript side. The
deeper fix is to have at least one test that exercises the real HTTP path across both
stacks, or to codify the wire contract (e.g., OpenAPI spec) and validate both sides
against it.

## Why this is non-obvious

Both test suites are green. A developer adding a REST endpoint and a fetch client
naturally writes tests that match their own code — the Java dev writes `text/plain`
because that's what the endpoint accepts, the TypeScript dev writes `application/json`
because that's what `JSON.stringify` produces. The mismatch is only visible when both
sides are compared simultaneously.
