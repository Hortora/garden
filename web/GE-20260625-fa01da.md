### Symptom

Browser Forward button stops working after using Back to return to a previous page. The back button works, but after pressing it, forward navigation is broken — pressing Forward either does nothing or navigates to a duplicate of the current page.

### Context

SPA with hash-based routing (e.g. `#/page/Sales?filter=region:North`). A `popstate` event listener handles browser back/forward by calling the same `navigate(path)` function used for programmatic navigation. The `navigate()` function internally calls `history.pushState()` to update the URL after activating the target page's DOM.

### What was tried (didn't work)

Using a single `navigate()` function for both programmatic navigation and popstate handling — seems clean and DRY, but silently corrupts the history stack.

### Root cause

During a `popstate` event, the browser has ALREADY updated `location.href` to the correct URL. The `popstate` handler calls `navigate()`, which calls `history.pushState()` with the same URL — creating a NEW history entry identical to the current one. This pushes a duplicate onto the stack, which wipes the forward history (pushState always discards forward entries).

The result: Back works (it was the user action), but Forward is broken because the duplicate entry replaced the forward stack.

### Fix

Separate DOM navigation from URL sync. Extract an internal navigation function that only activates DOM state (slot switching, page path computation) without touching the History API:

```typescript
// Internal: DOM activation + state only — no URL push
function navigateInternal(path: string): void {
  _navigating = true;
  const segments = path.split("/").filter(Boolean);
  currentPage = walkNavigate(root, segments, target, lazyPageResolutions);
  _navigating = false;
}

// Public API: DOM activation + URL push
navigate(path: string): void {
  navigateInternal(path);
  syncUrl("pushState");
}

// popstate: DOM only — URL is already correct
window.addEventListener("popstate", () => {
  const link = parseFromUrl(location.hash);
  if (link.page !== currentPage) {
    navigateInternal(link.page);  // NOT navigate()
  }
});
```

### Why this is non-obvious

`pushState` is the standard History API for SPA navigation — it's the "correct" tool. The problem isn't the API itself but the context: during popstate, the URL is already set by the browser, so pushState is redundant and destructive. The symptom (Forward button breaks) is disconnected from the cause (pushState during popstate), making debugging difficult. The code appears to work for Back navigation — only Forward reveals the corruption.

### Alternatives considered

- **Flag-based suppression** (`if (!isPopstate) pushState(...)`) — works but adds state management overhead and a boolean that's easy to forget in new code paths.
- **replaceState during popstate** — avoids the duplicate entry but is still unnecessary (the URL is already correct). Masks the design problem rather than fixing it.
- **Separate navigate function** (chosen) — cleanest separation of concerns. DOM activation is reusable; URL sync is opt-in. No flags needed.
