### Symptom
A data request event fires twice on initial connection instead of once. Tests expecting `events.length === 1` fail with `events.length === 2`.

### Context
Migrating a vanilla HTMLElement to LitElement. The original class called `requestDataIfNeeded()` in both its `connectedCallback` and its `props` setter. With Lit, the `props` setter is replaced by `@property({ attribute: false })` and the setter logic moves to `willUpdate()`. On first connection with pre-set props, BOTH `connectedCallback` AND `willUpdate` fire — each calling `requestDataIfNeeded()`.

### What was tried (didn't work)
- Removing `requestDataIfNeeded()` from `connectedCallback` — breaks reconnect (willUpdate doesn't fire on reconnect when props haven't changed)
- Adding `else if` branch in `willUpdate` for reconnect — willUpdate doesn't fire at all when no reactive properties changed
- Relying on `_dataRequested` flag — willUpdate resets it (`_dataRequested = false`) when detecting a lookup change

### Root cause
On first connection: `connectedCallback` runs → fires request → sets `_dataRequested = true`. Then Lit processes the queued update → `willUpdate` detects `newProps !== oldProps` (undefined → value) → resets `_dataRequested = false` (because lookup "changed") → fires request again.

### Fix
Guard the `_dataRequested` reset in `willUpdate` to only fire when transitioning between two defined prop objects, not from undefined to defined:
```typescript
override willUpdate(): void {
  const newProps = this.props;
  const oldProps = this._prevProps;
  if (newProps !== oldProps) {
    this._prevProps = newProps;
    if (oldProps !== undefined && newProps?.lookup !== oldProps?.lookup) {
      this._dataRequested = false;
      this.controller.dataSet = undefined;
    }
    this.requestDataIfNeeded();
  }
}
```
`connectedCallback` still calls `requestDataIfNeeded()` (with `_dataRequested = false` reset) to handle reconnect. On initial connection, `willUpdate` sees the props change but doesn't reset the flag — `requestDataIfNeeded` sees `_dataRequested = true` and skips.

### Why this is non-obvious
Lit's documentation describes `connectedCallback` and `willUpdate` as independent lifecycle hooks. The interaction between them on first connection with pre-set properties is not documented. A developer migrating from vanilla HTMLElement would naturally put side-effects in both places (matching the original pattern) without realizing both fire on the same connection.
