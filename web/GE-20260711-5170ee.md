### Symptom
A component using `fetchSource` with an endpoint that returns `[]` (empty JSON array) gets
`sink.error({ message: "Extraction produced no data (empty array)", permanent: true })` instead
of receiving an empty `TypedDataSet` with zero rows. The component shows an error state rather
than an empty-data state.

### Context
The pages-data extraction pipeline (`extractDataSet` → `tabulate`) explicitly throws
`DataSetError("EMPTY_RESULT", ...)` when the input is an empty array. This is intentional
for the YAML/inline data path (where empty data is likely a configuration error), but
surprising when an HTTP endpoint legitimately returns zero results.

In `fetchSource`, extraction errors route to `sink.error({ permanent: true })`, which sets
the component's `error` property. Consumers that want to display "No data" instead of an
error message must handle this case separately — either by not setting an endpoint until
data is expected, or by checking the error message.

### Root cause
`tabulate()` in `extraction.ts` line ~409:
```typescript
} else if (Array.isArray(data) && data.length === 0) {
  throw new DataSetError("EMPTY_RESULT", "Extraction produced no data (empty array)");
}
```

This treats empty arrays as an error rather than a valid empty result.

### Fix
Workaround: consumers that expect empty arrays should not render an error for
`EMPTY_RESULT` — treat it as equivalent to empty data.

Proper fix (upstream): change `tabulate` to return `{ dataset: { columns: [], data: [] }, inferredColumns: true }` for empty arrays, producing an empty TypedDataSet instead of throwing. This would require changes to the pages-data extraction pipeline.

### Why this is non-obvious
The extraction pipeline's error handling is designed for data ingestion (YAML, CSV, Prometheus)
where empty input signals misconfiguration. When the same pipeline processes REST API responses,
empty results are routine — the error-vs-empty distinction is context-dependent but the pipeline
treats all empty arrays as errors.
