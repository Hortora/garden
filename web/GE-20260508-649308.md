## Symptom

A 64×64 portrait canvas added to a UI panel appears blurry and fills the entire panel width. No CSS error. The canvas `width` and `height` HTML attributes are set correctly to 64. The canvas renders correctly in isolation but not on the full page.

## Context

A Three.js full-screen renderer requires that the main canvas fill the viewport. A common pattern adds a global CSS rule:

```css
canvas {
  display: block;
  width: 100vw !important;
  height: 100vh !important;
}
```

Later, a 64×64 portrait canvas (`<canvas id="up-portrait" width="64" height="64">`) is added to an inspect panel in a `<style>` block injected at runtime. The portrait canvas has `flex-shrink: 0` in its CSS but no explicit `width` or `height`. The global rule's `!important` overrides the HTML intrinsic size, stretching the canvas to `100vw × 100vh`. The canvas is then compressed by the flex container, producing a blurry upscaled image.

## Root cause

CSS `width` and `height` properties override the HTML `width`/`height` attributes when computing display size (the attributes set the *buffer resolution*, not the *display size*). `!important` on a type selector (`canvas`) beats any non-`!important` rule on an ID selector (`#up-portrait`), which is counterintuitive — normally ID specificity beats type specificity, but `!important` changes the cascade.

## Fix

Override with `!important` on the specific canvas ID:

```css
#up-portrait {
  border-radius: 3px;
  flex-shrink: 0;
  width: 64px !important;   /* beats canvas { width: 100vw !important } */
  height: 64px !important;
}
```

A cleaner long-term fix is to scope the full-screen rule to the renderer canvas specifically (requires adding a class or ID to `renderer.domElement`) rather than using the generic `canvas` type selector.

## Why this is non-obvious

ID selectors normally outrank type selectors in CSS specificity (0,1,0,0 vs 0,0,0,1). A developer adding `#up-portrait { width: 64px }` would expect this to win — and it would, without `!important`. The `!important` flag reverses the normal specificity order: a type-selector `!important` rule beats a non-`!important` ID-selector rule. The portrait renders correctly in isolation (no global `canvas` rule) but breaks on the full page, making the dependency non-obvious to diagnose.

### Why this fix

Matching `!important` on the ID selector is the minimal targeted fix. The alternative — scoping the full-screen rule to `renderer.domElement` — is cleaner but requires modifying the Three.js setup (adding a class to the renderer canvas) which is a larger change.

### Alternatives considered

- Removing `!important` from the global canvas rule: Breaks the full-screen renderer on some browsers where the canvas intrinsic size takes precedence.
- Using a non-canvas element (e.g. `<img>` or `<div>` with background) for the portrait: Avoids the collision but loses the ability to draw to the portrait using Canvas 2D API.
