### Symptom

Production build fails with `Could not resolve "@casehubio/pages-component"` from esbuild, but tests pass and Vite dev mode works fine.

### Context

A TypeScript monorepo where `vite.config.ts` has `resolve.alias` entries mapping `@casehubio/*` package names to local source paths. Tests use Vite (via vitest) and resolve correctly. But the production build uses esbuild directly (via `esbuild.config.mjs`), which has no equivalent alias configuration.

A subagent replaced a standalone helper function with a re-export from an aliased package:
```typescript
// Before (standalone — works everywhere)
export function emitChatEvent(...) { ... }

// After (re-export — works in Vite, breaks in esbuild)
export { emitPagesEvent as emitChatEvent } from '@casehubio/pages-component';
```

The re-export resolves in vitest (uses Vite's aliases) but fails in the esbuild production build (no aliases configured).

### What was tried (didn't work)

- Tests passed — no indication of the problem during development
- The function worked correctly in Vite dev mode

### Root cause

Vite's `resolve.alias` and esbuild's module resolution are separate systems. When a project uses Vite for dev/test and esbuild for production, any import that relies on Vite aliases will work in dev but fail in prod. Re-exports from aliased packages are particularly dangerous because they introduce the dependency implicitly — the re-exporting file doesn't contain the implementation, so it's not obvious that it depends on an aliased path.

### Fix

Keep trivial helper functions standalone instead of re-exporting from aliased packages. The function was 5 lines:

```typescript
export function emitChatEvent<T>(target: EventTarget, topic: string, payload: T): void {
  target.dispatchEvent(new CustomEvent('pages-event', {
    bubbles: true, composed: true, detail: { topic, payload },
  }));
}
```

No external dependency needed. The standalone version works in both Vite and esbuild.

**General rule:** In dual-build setups (Vite + esbuild), only import from packages that are real `dependencies` in `package.json` (resolvable by Node module resolution). Never import from packages that only resolve via Vite aliases — those are test/dev conveniences, not production dependencies.

### Why this is non-obvious

Tests passing creates false confidence. The alias resolution is invisible — the developer doesn't know which resolution system is active. The failure only surfaces at production build time, which may be much later than the code change (especially when subagents make the change and their output passes all tests).

### Alternatives considered

- **Add esbuild alias configuration** — possible but creates a maintenance burden of keeping two alias configs in sync
- **Add the package as a real dependency** — correct long-term, but adds a dependency for a 5-line function
