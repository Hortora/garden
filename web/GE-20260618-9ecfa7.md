### Symptom
Prometheus matrix query results show wrong values in label columns — e.g. `__name__` value "go_gc_heap_live_bytes" appears in the "instance" column position.

### Context
The prometheus preset uses a JSONata expression that builds `columns` and `values` arrays. The columns array is built from `$keys(result[0].metric)` with a filter to exclude `__name__`. The values array is built from `metric.*` — but the vector and matrix paths applied the filter inconsistently.

### Root cause
The vector path used `(metric ~> $sift($filterName)).*` which correctly excluded `__name__` from values. The matrix path used `$m := metric; values.[$[0]*1000, $[1], $m.*]` — raw `$m.*` includes `__name__`, creating a column count mismatch. Columns had `[timestamp, value, instance]` but values had `[ts, val, __name_value__, instance_value]`.

### Fix
Apply the same treatment to both paths. Either sift both or neither:

```javascript
// Before (inconsistent):
"result.[value[0]*1000, value[1], (metric ~> $sift($filterName)).*]"   // vector
"result.($m := metric; values.[$[0]*1000, $[1], $m.*])"               // matrix

// After (consistent — include all labels):
"result.[value[0]*1000, value[1], metric.*]"                           // vector
"result.($m := metric; values.[$[0]*1000, $[1], $m.*])"               // matrix
```

### Why this is non-obvious
The vector path worked correctly in unit tests. The matrix path was only exercised when range queries returned data, which required a different mock data format. The column/value alignment bug is silent — no error is thrown, values just appear in the wrong columns.
