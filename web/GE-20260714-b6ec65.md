### Symptom
Stack overflow or infinite loop when `refreshDataSet` is changed from
re-delivering cached data to actually re-fetching from the external source.
The original code works fine ‚Äî the recursion only appears when the semantics
change.

### Context
A reactive data pipeline with:
1. A cache manager (`DataSetManager`) that stores datasets and has an
   `onChanged` callback fired after every `apply()` (data insertion)
2. A pipeline method (`refreshDataSet`) that re-delivers data to subscribers
3. The `onChanged` callback calls `refreshDataSet` to push updated data
   to all bound components

This architecture works when `refreshDataSet` just calls `pushData` (reads
from cache). The cycle breaks naturally because `pushData` does not write
to the manager.

### What was tried (didn't work)
Changing `refreshDataSet` from "re-deliver cached data via `pushData`" to
"re-fetch from external source (disconnect + reconnect DataSource)" without
splitting the operation. The cycle:

```
refreshDataSet(id)
  ‚Üí source.disconnect() + source.connect(sink)
    ‚Üí sink.apply(event)
      ‚Üí manager.apply(id, event)
        ‚Üí onChanged(id, dataset)
          ‚Üí refreshDataSet(id)   ‚Üê infinite recursion
```

### Root cause
The `onChanged` callback was designed when `refreshDataSet` was a pure
read operation (cache serve). Changing it to a write operation (re-fetch
that triggers `manager.apply`) creates a cycle because `onChanged` calls
the same method that triggers it.

### Fix
Split into two distinct operations:

```typescript
// Re-fetch from external source ‚Äî called by user actions, stale-while-revalidate
refreshDataSet(dataSetId: DataSetId): void {
  // disconnect + reconnect DataSource, or re-resolve ExternalDataSetDef
  // Fresh data flows through: source ‚Üí sink ‚Üí manager.apply() ‚Üí onChanged ‚Üí deliverDataSet
}

// Re-deliver cached data to subscribers ‚Äî called by onChanged
deliverDataSet(dataSetId: DataSetId): void {
  // calls pushData() which reads from manager.lookup() ‚Äî no write, no cycle
}
```

The `onChanged` wiring changes from:
```typescript
onChanged: (id, dataset) => {
  pipeline.refreshDataSet(id);   // OLD ‚Äî causes infinite recursion
}
```
to:
```typescript
onChanged: (id, dataset) => {
  pipeline.deliverDataSet(id);   // NEW ‚Äî cache serve only, cycle breaks
}
```

The cycle breaks at `deliverDataSet`: it calls `pushData` which reads from
`manager.lookup()` (no `apply()`), so `onChanged` is never re-triggered.

### Why this is non-obvious
The original code works perfectly ‚Äî `refreshDataSet` calling `pushData` is
a pure read from the cache manager, so there is no cycle. The recursion is
latent: it only activates when someone changes the refresh semantics to
actually re-fetch, which is the natural evolution when you need fresh data
after mutations (e.g. post-action refresh, manual refresh button). The fix
(splitting one method into two with different write semantics) requires
understanding the full reactive chain.
