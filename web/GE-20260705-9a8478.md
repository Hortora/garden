### Symptom
npm install fails or resolves wrong packages when working on AML's webui.

### Context
On the `issue-91-aml-workbench-ui` branch of casehubio/aml, `app/src/main/webui/package.json`
declares dependencies using the npm `file:` protocol with hardcoded relative paths:

```json
{
  "@casehubio/pages-iframe-api": "file:../../../../../pages/packages/pages-iframe-api",
  "@casehubio/pages-runtime": "file:../../../../../pages/packages/pages-runtime",
  "@casehubio/pages-ui": "file:../../../../../pages/packages/pages-ui"
}
```

This requires `casehub-pages` to be checked out as a sibling of `aml` under the same
parent directory (e.g. both under `~/casehub/`). The path traversal (`../../../../../`)
is hardcoded and not configurable.

### Why this is non-obvious
Neither the AML repo nor the casehub-pages repo documents this layout requirement. A
developer cloning AML for the first time would get cryptic npm resolution errors with
no indication that a sibling checkout of a different repo is needed.

### Implications
- Version alignment between the repos is path-based, not version-number-based — npm
  version bumps in casehub-pages don't break or affect AML's file: references
- CI/CD must replicate the sibling checkout layout or use a different dependency strategy
- Any restructuring of either repo's directory layout will silently break the references
