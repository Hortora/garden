### Symptom
Tests for vanilla Web Component callbacks (column renderers, row key functions)
show the callback was never called — cell content renders as plain text despite
a renderer being set.

### Context
`PagesElement` subclasses (e.g. `PagesGroupedView`) are vanilla Web Components
that render synchronously when `dataSet` is set. Unlike Lit components which
batch property changes via `requestUpdate()`, the `dataSet` setter immediately
calls `render()`.

### What was tried (didn't work)
Setting callbacks after both `props` and `dataSet`:
```typescript
element.props = makeProps({ preset: "list" });
element.dataSet = makeGroupedDataset();
element.setColumnRenderers(renderers); // too late — render already ran
```

### Root cause
The `dataSet` setter triggers `render()` immediately. At that point,
`_columnRenderers` is still `undefined`. The subsequent `setColumnRenderers()`
updates the field and forwards to existing child tables, but does not
re-trigger the parent's render — the list mode DOM is already built.

### Fix
Set callbacks BEFORE setting `dataSet`:
```typescript
element.props = makeProps({ preset: "list" });
element.setColumnRenderers(renderers); // set before render trigger
element.dataSet = makeGroupedDataset(); // render() now sees the renderers
```

### Why this is non-obvious
Developers coming from Lit expect property changes to be batched — setting
three properties triggers one render cycle. Vanilla `PagesElement` renders
on each setter call. The ordering constraint is invisible until a test fails
with "renderer not called" and no error message.
