### Symptom
`TS2307: Cannot find module '@scope/higher-tier-package'` in CI, but the project
compiles locally without error.

### Context
Monorepo with Yarn workspaces where a lower-tier package (`pages-data`) imports a
type from a higher-tier package (`pages-component`) that already depends on
`pages-data`. CI builds packages in dependency order — the higher-tier package
isn't built yet when the lower-tier package compiles.

### What was tried (didn't work)
- Checking tsconfig references — all correct
- Verifying package.json dependencies — the circular dependency was not declared,
  which masked the problem
- Rebuilding locally — always succeeds because Yarn hoists all workspace packages

### Root cause
Yarn workspace hoisting resolves all workspace packages regardless of declared
dependencies. This means `import type { X } from "@scope/B"` works locally even
when package A has no dependency on B — the hoisted `node_modules` contains B's
types. In CI, packages build in topological order based on declared dependencies.
If B depends on A, B is built after A — so A's compilation cannot see B's output.

### Fix
Move shared types to the lower-tier package where they belong. Re-export from
the higher-tier package for backward compatibility:

```typescript
// Lower-tier package: define the type
export interface GroupNode { /* ... */ }

// Higher-tier package: re-export for backward compat
import type { GroupNode } from "@scope/lower-tier-package";
export type { GroupNode };
```

### Detection
Check that every `import ... from "@scope/X"` has a corresponding dependency in
`package.json`. If `X` depends on the importing package, the import direction is
wrong — the shared type needs to move down.

### Why this is non-obvious
The failure only manifests in CI. Local development gives zero signal that anything
is wrong — no red squiggles, no build failures, no test failures. The TS2307 error
message names the missing module but doesn't hint at a circular dependency as the
cause.
