### Symptom
Playwright E2E tests fail with CSS parser errors when using `>>>` shadow DOM piercing combinator, or silently fail to find elements inside shadow roots via `page.evaluate("document.getElementById(...)")`.

### Context
When migrating a monolithic HTML app to Web Components with Shadow DOM, E2E tests need to find elements inside shadow roots. Two approaches seem equivalent but behave very differently.

### What was tried (didn't work)
1. Built a `shadowLocator()` helper: `page.locator("host-element").locator(">>> selector")` — the `>>>` combinator causes Playwright's CSS parser to throw a parse error.
2. Used `page.evaluate("document.getElementById('render-a')")` for elements inside shadow roots — returns null because `getElementById()` only searches the document tree, not shadow roots.

### Root cause
Playwright has two distinct DOM access paths with different shadow DOM semantics:

- **Locator API** (`page.locator("css-selector")`, `page.waitForSelector()`): **automatically pierces open shadow DOM**. No special syntax needed. `page.locator("#render-a")` finds the element inside any open shadow root on the page.

- **Evaluate API** (`page.evaluate("document.querySelector(...)")`): runs raw JavaScript in the page context. **Does NOT pierce shadow DOM.** Standard DOM APIs like `getElementById()` and `querySelector()` only search the document tree. To access shadow DOM elements: `document.querySelector('host-element').shadowRoot.querySelector('#render-a')`.

The `>>>` combinator (shadow-piercing descendant combinator) is a deprecated CSS proposal. Playwright's CSS engine does not support it — attempting to use it throws a parse error in the CSS parser.

### Fix
Use Playwright's locator API directly — it auto-pierces:
```java
// ✅ Finds #render-a inside any open shadow root
page.locator("#render-a")

// ✅ Waits for element inside shadow root
page.locator("[data-diff-chunk]").first().waitFor()

// ✅ For page.evaluate() — explicit shadow root traversal
page.evaluate("document.querySelector('drafthouse-diff').shadowRoot.getElementById('body-a').scrollTop")
```

### Why this is non-obvious
The auto-piercing behaviour is buried in Playwright's locator documentation but not prominently flagged. A developer new to Shadow DOM testing naturally reaches for CSS shadow-piercing syntax (which appears in many blog posts) or raw JS DOM traversal (which is how you'd do it in the browser console). Both fail. The correct answer — "just use the locator normally, it already pierces" — feels wrong because it contradicts the Shadow DOM encapsulation model.
