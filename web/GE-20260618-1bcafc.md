### Symptom
Prometheus HTTP Requests dashboard shows `EXTRACTION_ERROR: Unrecognized data shape` despite having `type: prometheus` declared in `global.dataset`.

### Context
DashBuilder YAML allows declaring dataset defaults in `global.dataset`:
```yaml
global:
    dataset:
        type: prometheus
datasets:
    - uuid: http_requests
      url: ${prometheusUrl}/api/v1/query?query=prometheus_http_requests_total
```

The parser stores `global.dataset` as `datasetDefaults` in the runtime settings model (`PageSettings.datasetDefaults`). Individual datasets do not have `type: prometheus` — they rely on the global default.

### Root cause
`datasetDefaults` is stored in the model (`page-types.ts:23`) but the runtime never reads or merges it into individual dataset definitions before resolution. The `type` field on each dataset remains undefined, so the prometheus preset is never applied. The raw Prometheus API response `{status, data: {resultType, result}}` hits the tabulation logic as an unrecognised shape.

### Workaround
Auto-detect the Prometheus API response format in the extraction pipeline. When data matches `{status: "success", data: {resultType: string, result: array}}` and no explicit `type` is set, apply the prometheus preset automatically:

```typescript
if (def.type === undefined && looksLikePrometheusApi(current)) {
  const preset = presetRegistry.get("prometheus");
  if (preset) {
    const compiled = compileOrCached(preset.expression);
    current = await compiled.evaluate(current);
  }
}
```

### Why this is non-obvious
The `datasetDefaults` field exists in the TypeScript model with full type definitions. Tests verify it is parsed correctly from YAML. The field name implies it will be used. Nothing in the codebase indicates it's a stub — you discover it only when a dashboard that relies on `global.dataset.type` fails at extraction time.
