### Symptom
Click-outside detection using `event.composedPath().includes(this)` always
evaluates to true in jsdom tests, even when the click originates from an
element completely outside the Web Component. The dropdown/popup never closes
in tests but works correctly in browsers.

### Context
Standard pattern for detecting clicks outside a Web Component in Shadow DOM:
```javascript
document.addEventListener('click', (e) => {
  if (!e.composedPath().includes(this)) {
    this.closePopup();
  }
});
```
This works in real browsers because `composedPath()` correctly shows the
event's path through shadow boundaries. In jsdom, the path always includes
the host element regardless of where the click originated.

### Root cause
jsdom does not fully implement Shadow DOM event retargeting. The
`composedPath()` method returns an incomplete or incorrect path that
includes the host element even for events originating outside the
component's shadow tree.

### Fix
Use `this.contains(e.target as Node)` instead of `composedPath()`:
```typescript
private _handleClickOutside = (e: MouseEvent): void => {
  if (!this.contains(e.target as Node)) {
    this._popupOpen = false;
  }
};
```
This works in both jsdom and real browsers because:
- **Light DOM clicks** (outside the component): `e.target` is the actual
  clicked element, `this.contains()` returns false ✓
- **Shadow DOM clicks** (inside the component): `e.target` is retargeted
  to the host element, `this.contains()` returns true ✓

### Why this is non-obvious
`composedPath()` is the canonical recommendation for Shadow DOM event
inspection (MDN, Lit docs, lamplightdev.com). A developer would naturally
use it, write tests that pass in the browser, then see failures only in
CI/jsdom — with no error message pointing to the cause.

### Alternatives considered
- `event.target !== this` — fails for Shadow DOM clicks where the target
  IS the host due to retargeting
- Mocking `composedPath()` in tests — fragile, couples tests to
  implementation
