### What exists

`processWireMessage()` in `packages/pages-data/src/dataset/external/sources/push-source.ts` handles the `event` op by dispatching a `pages-event` CustomEvent on `config.eventTarget` and returning immediately. It does NOT call `updateSeq(msg.seq)` — that call only happens in the `snapshot`, `append`, `replace`, and `remove` branches.

### Consequence

The WebSocket source tracks `lastSeq` per subscription and sends `since: lastSeq` on reconnect (in the `ws.onopen` handler at `websocket-source.ts:54-66`). For event-only consumers — where all server→client data flows as `event` ops and no dataset ops are ever sent — `lastSeq` is never set. On reconnect, `since` is never included in the re-subscribe message. The server has no way to send only missed events.

### Why this is non-obvious

The pages push protocol documentation and API present `seq` as a general-purpose field on wire messages, and the reconnection handling in `websocket-source.ts` appears to work generically. A developer reading the API surface would naturally assume that setting `seq` on an `event` op and calling `subscribe()` on the source gives them incremental recovery for free. The gap is only visible by reading the `processWireMessage()` implementation — specifically that the `event` case returns before the shared `updateSeq()` call at the bottom of the function.

### Workaround

Use a full-catch-up model: on reconnect, reset client-side state (panels clear their entries) and have the server re-send the complete current state. For typical event volumes (20-100 entries per session), this is negligible bandwidth and avoids the false assumption of incremental recovery.

### Alternatives considered

- Manually tracking seq on the client side and sending `since` in a custom upstream message — rejected because the pages `PushSource` interface exposes only `subscribe()`/`unsubscribe()`/`close()` with no `send()` method. The underlying WebSocket is internal.
- Patching `processWireMessage()` to call `updateSeq()` for event ops — deferred to casehub-pages#98 (listen/unlisten protocol) which redesigns event subscriptions from scratch.
