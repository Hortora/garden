### Symptom
Component renders but shows no data. No errors in console. The inbox, detail panel, and queue board all show empty states despite mock data being available. The `endpoint` prop is set to `""` (empty string for same-origin).

### Context
Web Components accept an `endpoint` prop as a base URL for REST API calls. When deployed same-origin, the natural value is `""` (empty string) — fetch URLs become `/workitems/inbox` etc. Components guard their fetch calls with `if (this.endpoint)` or `if (!this.endpoint) return`.

### What was tried (didn't work)
- Checking mock data was loaded — it was
- Checking the fetch interceptor — it worked for other URLs
- Setting `endpoint="/workitems"` — caused URL doubling (`/workitems/workitems/inbox`)

### Root cause
In JavaScript, `""` (empty string) is falsy. `if (this.endpoint)` evaluates to `false` when endpoint is `""`. The component skips all data loading because it thinks no endpoint was configured. This is correct JavaScript but wrong semantics — empty string IS a valid base URL meaning "same origin."

### Fix

Use null/undefined checks instead of truthiness:

```typescript
// WRONG — empty string is falsy
if (!this.endpoint) return;
if (this.endpoint) { this.fetchItems(); }

// RIGHT — only skip when genuinely not configured
if (this.endpoint == null) return;
if (this.endpoint != null) { this.fetchItems(); }
```

`== null` catches both `null` and `undefined` but passes `""`.

### Why this is non-obvious
`if (this.endpoint)` is the natural guard pattern and appears in countless codebases. The bug only manifests when `endpoint=""` (same-origin), which is common in development but might not be the first value tested. The component appears to work with `endpoint="http://api.example.com"` and only fails silently with the empty-string case.
