### Symptom

Data carried in `Map` properties of `postMessage` payloads arrives as an
empty object `{}` when the message crosses a real iframe boundary
(different frame or origin). No error is thrown — the data silently
disappears.

### Context

A component API uses `Map<string, unknown>` as the property bag in its
message protocol. During development and testing, messages are posted via
`window.postMessage()` on the same window (the component and host share
a JS context). In this same-window scenario, the Map reference is
preserved in-memory — the message listener receives the original Map
object, not a structured-clone copy.

When the same code runs with a real iframe (cross-frame or cross-origin),
`postMessage` invokes the structured clone algorithm to serialise the
message. Map serialisation behaviour varies across browsers and contexts
— the Map may arrive as an empty plain object.

### What was tried (didn't work)

1. Typed the properties field as `Map<string, unknown>` and populated
   it with `.set()` calls — works in same-window testing, fails across
   frames.
2. Assumed structured clone handles Map correctly — the spec says it
   should, but browser implementations vary for postMessage specifically.

### Root cause

`window.postMessage()` on the same window does not actually clone the
data — the listener receives the same object reference. This masks any
serialisation issues. When postMessage crosses a frame boundary, the
structured clone algorithm runs for real, and Map objects may not
survive the round-trip depending on the browser.

The fundamental issue is that same-window `postMessage` is not a faithful
simulation of cross-frame `postMessage`. Code that works in development
(same window) can silently fail in production (real iframe).

### Fix

Use plain objects (`Record<string, unknown>`) instead of `Map` for any
data crossing a `postMessage` boundary. Plain objects are universally
supported by structured clone across all browsers and contexts.

```typescript
// Before — breaks across real iframe boundaries
interface ComponentMessage {
  type: string;
  properties: Map<string, unknown>;
}

// After — works everywhere
interface ComponentMessage {
  type: string;
  properties: Record<string, unknown>;
}
```

If Map semantics are needed inside the component, convert at the
boundary:

```typescript
window.addEventListener("message", (e) => {
  const msg = e.data as ComponentMessage;
  const props = new Map(Object.entries(msg.properties));
  // use props as Map internally
});
```

### Why this is non-obvious

The structured clone algorithm spec includes Map as a supported type.
A developer reading the spec would reasonably expect Map to work with
postMessage. The failure is silent — no error, no warning, just missing
data. And the most common development setup (same-window postMessage)
masks the problem entirely because no cloning occurs.

### Why this fix

Plain objects are the lowest-common-denominator wire format for
postMessage. They work in every browser, every context, every frame
relationship. The small ergonomic cost of using bracket notation instead
of `.get()/.set()` is negligible compared to silent data loss in
production.

### Alternatives considered

- **Keep Map, add a serialisation layer** — convert Map to/from plain
  objects at the postMessage boundary. Adds complexity and two conversion
  points that must stay in sync. Rejected: the Map adds no value over
  plain objects for a wire format.
- **Use structuredClone() directly to test** — would catch the issue in
  tests, but doesn't fix the underlying problem. The wire format should
  be safe by construction, not by testing discipline.
