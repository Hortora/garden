### Symptom
Component crashes with `DataSetError: Column "score" is NULL, not NUMBER` when a single row in a TypedDataSet has a NULL cell. The error propagates uncaught through the Lit render cycle.

### Context
Extracting numeric values from a `TypedDataSet` received via a `DataSource` (e.g. `inlineSource()`, `simulated()`). The dataset may contain rows with NULL cells — partial data, sensor gaps, or placeholder rows are common in time-series data.

### Root cause
`TypedRow` has two access patterns with very different safety characteristics:

- **`row.cell(columnId)`** — returns a `CellValue` discriminated union (`{ type: 'NUMBER', value: number } | { type: 'NULL' } | ...`). Never throws. The caller can type-guard safely.

- **`row.number(columnId)`** / `row.text()` / `row.date()` — convenience accessors that throw `DataSetError` if the cell is NULL or the type doesn't match. Designed for trusted contexts where the schema is guaranteed.

This is not documented in any API docs or README. The behaviour is only discoverable by reading `createTypedRow()` in `pages-data/src/dataset/conversion.ts`.

### Fix
For data extraction from untrusted or variable-quality sources, always use `row.cell()` with type guards:

```typescript
function extractTrendPoints(dataSet: TypedDataSet): TrendPoint[] {
  const tsCol = dataSet.columns.find(c => c.name === 'timestamp');
  const scoreCol = dataSet.columns.find(c => c.name === 'score');
  if (!tsCol || !scoreCol) return [];

  const points: TrendPoint[] = [];
  for (const row of dataSet.rows) {
    const tsCell = row.cell(tsCol.id);
    const scoreCell = row.cell(scoreCol.id);
    if (tsCell.type === 'NULL' || scoreCell.type === 'NULL') continue;
    if (tsCell.type !== ColumnType.NUMBER || scoreCell.type !== ColumnType.NUMBER) continue;
    points.push({ timestamp: tsCell.value, score: scoreCell.value });
  }
  return points;
}
```

A single malformed row is silently skipped rather than crashing the entire component.

### Why this is non-obvious
The `number()` / `text()` / `date()` methods look like the idiomatic API — they're shorter, return the unwrapped value directly, and match the column type names. A developer reaching for `row.number(col)` would expect it to return `undefined` or `NaN` for missing data, not throw. The throwing behaviour is undocumented and only visible in the `createTypedRow` source.
