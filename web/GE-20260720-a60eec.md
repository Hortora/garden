### Symptom
Chart tests assert `mockChart.setOption.mock.calls[0]` after `await el.updateComplete` but get `undefined` — setOption was never called. The chart div IS rendered in shadow DOM, all properties are set, and the ref() directive resolves. No error is thrown.

### Context
LitElement's `updated()` lifecycle hook calls `buildOption()` which returns a Promise. The option pipeline uses `.then(apply)` where `apply` calls `chart.setOption()`. Tests set `el.dataSet = ds`, `await el.updateComplete`, then assert on mock calls.

### What was tried (didn't work)
- Adding `await Promise.resolve()` after `updateComplete` — insufficient, still 0 calls
- Using `querySelector` instead of `ref()` for container — same result
- Double `await el.updateComplete` — same result

### Root cause
`updateComplete` resolves after Lit's `updated()` returns, but `updated()` starts the async `buildOption()` chain and returns immediately. The `.then(apply)` callback is queued as a microtask but hasn't fired by the time `updateComplete` resolves. `await Promise.resolve()` flushes one microtask level but the Promise chain from `buildOption` may have multiple levels.

### Fix
Use `setTimeout` to force a macrotask boundary, then await another update cycle:
```typescript
el.dataSet = ds;
await el.updateComplete;
await new Promise(r => { setTimeout(r, 0); });
await el.updateComplete;
const option = mockChart.setOption.mock.calls[0]![0];
```

The `setTimeout` ensures all pending microtasks (including the buildOption Promise chain) resolve before the test continues.

### Why this is non-obvious
`await el.updateComplete` is Lit's documented way to wait for rendering. A developer reading the docs would assume it's sufficient. The interaction between Lit's update lifecycle and an async method called from `updated()` is not documented — `updated()` is sync by contract, but nothing prevents it from starting async work.
