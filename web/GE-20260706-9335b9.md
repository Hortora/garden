### Symptom
Theme toggle (light/dark) works on all components except one. The workbench component renders with light-mode colours even when the shell switches to dark mode. No error, no console warning.

### Context
A Lit Web Component defined `.theme-light { --blocks-neutral-1: #fafafa; ... }` and `.theme-dark { ... }` inside its `static styles = css\`...\``. The document-level theme system sets the same `--blocks-*` tokens via `.blocks-theme-dark` on `document.documentElement`. The component's `render()` applied `.theme-light` or `.theme-dark` based on its own `theme` property — a separate, disconnected system from the document-level theme.

### Root cause
CSS custom properties DO inherit through shadow DOM boundaries — a property set on `<html>` is visible inside every shadow root. But when the shadow DOM's own stylesheet declares the same property (e.g. `--blocks-neutral-1: #fafafa` inside a `.theme-light` rule), the shadow-internal declaration wins via normal cascade rules. The document-level value is inherited, but the shadow-internal value is a direct declaration on a descendant — it overrides the inherited value.

The component had two independent theme systems: document-level (shell) and component-level (its own `theme` property + shadow CSS). The component-level always won because it declared values directly rather than inheriting them.

### What was tried (didn't work)
Assumed CSS custom properties would flow through shadow DOM without interference — they do, but only when not re-declared inside the shadow root.

### Fix
Remove all CSS custom property declarations for theme tokens from the component's shadow CSS. The component should inherit `--blocks-*` values from the document-level theme, not redeclare them. Also remove the component's own `theme`/`density` properties — the document-level theme system is the single source of truth.

```css
/* WRONG — shadow-internal declarations override inherited values */
.theme-light { --blocks-neutral-1: #fafafa; }
.theme-dark  { --blocks-neutral-1: #0a0a0a; }

/* RIGHT — no declarations; inherit from document-level theme */
.workbench { background: var(--blocks-neutral-1, #fafafa); }
```

### Why this is non-obvious
CSS custom property inheritance through shadow DOM is well-documented and widely understood. What's NOT intuitive is that a declaration inside shadow CSS overrides an inherited value from outside the shadow boundary — the same cascade rule that applies to regular properties. Developers testing in isolation (component renders fine with its own theme) will not notice the conflict until they embed it in a shell with a different theme system.

### Alternatives considered
- Passing theme as a property and keeping shadow-internal CSS: rejected — creates two independent theme systems that will drift
- Using `::part()` to style from outside: rejected — doesn't solve the root cause (shadow declarations still override)
