### Symptom
A LitElement component uses `var(--pages-neutral-1, #fff)` throughout its CSS but the theme variables have no effect — the fallback values always apply. Toggling dark mode changes the parent page but the component stays light.

### Context
The `pages-ui-tokens` theme system generates CSS rules under `.pages-theme-light` and `.pages-theme-dark` class selectors. The component is a custom element rendered inside a pages-runtime layout. The theme class is set on an ancestor `<div>`, and `injectTheme()` places the `<style>` there.

### Root cause
CSS custom properties DO inherit through shadow DOM boundaries — but only from the element where they're defined downward. The `.pages-theme-dark` class was on an outer container, and `injectTheme()` placed the `<style>` there. But the custom element's shadow root is a separate document fragment — the class+style must be on the host element itself (or an ancestor that the custom element inherits from).

In this case, the pages-runtime rendered a wrapper `<div>` between the themed container and the custom element, breaking the inheritance chain for the class selector (the class was on the container, but the CSS rules only matched elements with that class directly).

### Fix
Call `injectTheme()` targeting the custom element's host, and `applyThemeMode()` on the host:
```typescript
import { injectTheme, applyThemeMode, DEFAULT_THEME } from '@casehubio/pages-ui-tokens';

override connectedCallback() {
  super.connectedCallback();
  this.updateComplete.then(() => {
    injectTheme(DEFAULT_THEME, this);          // inject <style> as child of host
    applyThemeMode(this, 'light');             // set class on host
  });
}

private _toggleTheme() {
  this._darkMode = !this._darkMode;
  applyThemeMode(this, this._darkMode ? 'dark' : 'light');
}
```
The generated CSS rules (`.pages-theme-light { --pages-neutral-1: ... }`) match the host element directly. Custom properties then cascade into the shadow root and all child shadow DOMs.

### Why this is non-obvious
The mental model "custom properties inherit through shadow DOM" is correct but incomplete. The class selector that gates the variable definitions must also resolve on or above the host. When a framework inserts wrapper elements between the theme provider and the component, the class is present but the CSS rule doesn't match the wrapper — only elements with the class. Developers expect `injectTheme` on any ancestor to work, but it only works if the class is on that same ancestor and the CSS rules can match it.
