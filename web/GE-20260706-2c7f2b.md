### Symptom

Elements have no padding, margin, or other styling despite CSS
declarations being present. No error in DevTools, no warning, no
computed value — the property simply doesn't appear in computed styles.

### Context

Using CSS custom properties (`var(--pages-space-2-5)`) in a shorthand
declaration like `padding`. The custom property name is a typo or
doesn't exist in the token system — `--pages-space-2-5` doesn't exist,
though `--pages-space-2` and `--pages-space-3` do.

### What was tried (didn't work)

1. Checked DevTools computed styles — padding showed as 0 with no
   indication of why. The declaration appeared in the rules panel but
   was crossed out with no explanation.
2. Assumed the individual values were falling back to their defaults —
   they weren't, because the entire shorthand was invalid.

### Root cause

Per the CSS spec, when a `var()` reference resolves to nothing (the
custom property doesn't exist and no fallback is specified), the
value is "invalid at computed-value time." For **shorthand** properties,
this invalidates the **entire declaration** — all sub-properties revert
to their initial values, not just the one that used the bad `var()`.

```css
/* This entire declaration is silently discarded: */
padding: var(--pages-space-2-5) var(--pages-space-5) var(--pages-space-2-5) var(--pages-space-8);

/* All four padding values become 0, not just the two that used the bad var() */
```

For **longhand** properties, only the single property gets the initial
value — the damage is contained. The shorthand case is far worse
because one typo kills multiple properties at once.

### Fix

Always use fallback values in var() references, or verify the custom
property name exists in the token system before using it:

```css
/* Safe — fallback protects against missing property: */
padding: var(--pages-space-2-5, 10px) var(--pages-space-5, 20px);

/* Better — use a property that actually exists: */
padding: var(--pages-space-2) var(--pages-space-5);
```

### Why this is non-obvious

The failure is completely silent. No console error, no DevTools
warning, no red squiggle. The declaration appears in the rules panel
but is struck through — identical to how a lower-priority rule looks.
A developer would naturally check for specificity issues, not for a
missing custom property in one position of a shorthand.

### Alternatives considered

- **Use longhand properties instead of shorthands** — prevents the
  cascade failure but is verbose. Rejected: shorthands are idiomatic
  CSS and the real fix is validating property names.
- **Add fallback values to every var() call** — defensive but clutters
  the code. Acceptable for libraries consumed by third parties;
  unnecessary for internal code with a known token system.
