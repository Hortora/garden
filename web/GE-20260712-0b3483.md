### Symptom
Vite dev server returns 500 errors for cross-package imports like `@casehubio/pages-data/dist/dataset/types.js`. The error says the file doesn't exist, but it does exist at the expected location.

### Context
Monorepo with packages aliased from source via `resolve.alias` in object form:
```js
alias: {
  '@casehubio/pages-data': path.resolve(PAGES, 'pages-data/src'),
  '@casehubio/pages-data/dist': path.resolve(PAGES, 'pages-data/src'),
}
```
Cross-package imports use `/dist/` deep paths: `import { X } from '@casehubio/pages-data/dist/dataset/types.js'`.

### What was tried (didn't work)
Adding both bare and `/dist` aliases in object form — the `/dist` entry was never reached because the shorter bare alias matched first.

### Root cause
Vite's object-form `resolve.alias` iterates entries in insertion order (via `Object.entries()`). The first matching prefix wins. `@casehubio/pages-data` matches `@casehubio/pages-data/dist/dataset/types.js` before the more specific `@casehubio/pages-data/dist` entry is checked. The replacement produces `pages-data/src/dist/dataset/types.js` — the `/dist` segment is NOT stripped.

### Fix
Use array-form aliases with longer prefixes listed first:
```js
alias: [
  { find: '@casehubio/pages-data/dist', replacement: path.resolve(PAGES, 'pages-data/src') },
  { find: '@casehubio/pages-data', replacement: path.resolve(PAGES, 'pages-data/src') },
]
```
Array form preserves explicit ordering — the longer `/dist` prefix matches first and correctly strips the `/dist` segment.

### Why this is non-obvious
The object-form alias syntax looks declarative (like a map), which implies order shouldn't matter. There's no warning when a shorter alias shadows a longer one. The 500 error points to the resolved path but doesn't mention which alias was used, making the double-resolution invisible.

### Alternatives considered
- Using a plugin to rewrite `/dist` to `/src` in imports — works but adds complexity
- Aliasing every individual deep import path — doesn't scale across packages
