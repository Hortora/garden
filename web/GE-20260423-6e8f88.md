## Context

A Three.js visualizer serves multiple Quarkus profiles with different terrain heights: mock mode has flat tiles ~0.08 world units tall; emulated mode has full-height tiles of `TILE = 0.7` world units. Unit sprites and 3D models need to sit above the terrain surface in both profiles without any per-profile branching at spawn time.

## Technique

Declare a module-level `TERRAIN_SURFACE_Y` variable initialised to the mock default. Update it once in `loadTerrain()` when emulated terrain data is loaded. All unit Y-position calculations reference this variable:

```js
let TERRAIN_SURFACE_Y = 0.08; // updated in loadTerrain — mock=0.08, emulated=TILE

async function loadTerrain() {
  const r = await fetch('/qa/emulated/terrain');
  if (r.ok) {
    // ... parse terrain ...
    TERRAIN_SURFACE_Y = TILE; // emulated ground tiles are TILE tall
    hasRealTerrain = true;
  }
}

// At unit spawn — works correctly in both profiles:
const groundY = TERRAIN_SURFACE_Y + TILE * 0.5;
const flyingY = TERRAIN_SURFACE_Y + TILE * 1.1;
const unitY   = FLYING_UNITS.has(u.type) ? flyingY : groundY;
sp.position.set(wp.x, unitY, wp.z);
```

## Why this is non-obvious

The instinct is to check the active profile explicitly (via a JS flag or a separate fetch) and branch on it. The single-variable approach is simpler: the variable is set once during terrain load and all subsequent code is profile-agnostic. The same pattern works for any per-profile constant that affects geometry placement.

## Caveats

The variable must be read at spawn time (not at module load), since terrain loads asynchronously after the scene is initialised. Units spawned before terrain loads will use the mock default, which is usually correct since mock mode doesn't load terrain.
