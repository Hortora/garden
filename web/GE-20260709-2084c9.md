### Symptom

Blank page in browser during Vite dev mode. No console errors, no network errors visible in the default console view. The page loads HTML correctly but the JavaScript never executes.

### Context

A project using Vite for development (HMR, TypeScript serving) and esbuild for production bundling. The HTML entry point (`index.html`) references the production bundle filename (`./app.js`). The esbuild config builds `src/index.ts` → `dist/app.js` and copies the HTML to `dist/`.

In Vite dev mode, there is no `dist/app.js` — Vite serves TypeScript source files directly. The `<script type="module" src="./app.js">` tag generates a 404, but ESM module loads fail silently — no console error, no network error tab highlight.

### What was tried (didn't work)

- Checked browser console — no errors
- Checked network tab — the 404 is there but easy to miss among other requests
- Assumed the JavaScript was loading but the components weren't rendering

### Root cause

Vite dev mode serves TypeScript files natively (transpiling on the fly). It does not know about `app.js` — that filename only exists after esbuild runs a production build. ESM `<script type="module">` loads that 404 do not throw errors in the console, making the failure completely silent.

### Fix

Have the source HTML reference the TypeScript entry point:

```html
<!-- src/index.html — used by Vite dev mode -->
<script type="module" src="./index.ts"></script>
```

Update the esbuild config to rewrite the script src when copying HTML for production:

```javascript
// esbuild.config.mjs
import { readFileSync, writeFileSync, mkdirSync } from "fs";

mkdirSync("dist", { recursive: true });
const html = readFileSync("src/index.html", "utf8").replace('./index.ts', './app.js');
writeFileSync("dist/index.html", html);
```

This way:
- **Vite dev mode:** HTML loads `./index.ts` → Vite serves it with HMR
- **esbuild production:** HTML in `dist/` loads `./app.js` → the bundled output

### Why this is non-obvious

ESM module script failures are silent by design — the browser spec does not throw console errors for failed module loads. Combined with a blank page (no partial rendering to hint at what's wrong), the developer's instinct is to look at component logic, not the script tag. The actual cause (wrong filename) is trivially fixable once found, but the silent failure mode makes it hard to find.

### Alternatives considered

- **Vite as both dev and prod build tool** — would eliminate the mismatch, but the project uses esbuild for production because Quinoa (Quarkus) integrates with it natively
- **Vite alias to map app.js → index.ts** — works but hides the real problem; the HTML should reference the source, not an alias to it
