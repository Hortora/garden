### Symptom
`panelShadow().locator("channel-feed").textContent()` returns `""` even though `.message-item` elements are visibly rendered inside `channel-feed`'s shadow root. `feedMessages().first().waitFor()` succeeds (elements found), but all text assertions fail with empty string.

### Context
Three levels of nested LitElement Shadow DOM: `claudony-channel-panel` → `channel-feed` → `channel-message`. Each component renders its content inside its own shadow root. The test helper `feedText()` calls `textContent()` on the `channel-feed` element expecting to get all message text.

### What was tried (didn't work)
1. `panelShadow().locator("channel-feed").textContent()` — returns "" (standard DOM textContent doesn't traverse into child shadow roots)
2. `panelShadow().locator("channel-feed").innerText()` — same result
3. Playwright locator chaining `channel-feed >> .content` — finds individual content divs but misses badge text and metadata rendered by channel-message

### Root cause
Playwright's locator engine has an asymmetry between **selection** and **extraction**:

- **Selection** (finding elements): `locator("css-selector")` auto-pierces open shadow DOM. This is Playwright's custom engine — it traverses shadow roots transparently.
- **Extraction** (reading content): `locator.textContent()`, `locator.innerText()`, `locator.getAttribute()` call **standard DOM APIs** on the matched element. `element.textContent` only returns text from the element's own DOM subtree — it does NOT traverse into child elements' shadow roots.

With nested shadow DOMs (host → shadow → child-host → child-shadow → text), `textContent()` on the outer host sees the child host element as empty (its text is inside its own shadow root).

See also: GE-20260614-cd8e92 covers the locator-vs-evaluate distinction for element finding. This entry covers the extraction gap that persists even when using locators correctly.

### Fix
Use `evaluateAll()` on the leaf shadow hosts to collect text from each component's shadow root:

```java
private String feedText() {
    Object result = panelShadow().locator("channel-message").evaluateAll(
        "els => els.map(el => el.shadowRoot ? el.shadowRoot.textContent : el.textContent).join(' ')");
    return result != null ? result.toString() : "";
}
```

This targets the leaf-level shadow hosts (`channel-message` elements) found via Playwright's piercing locator, then uses `evaluate()` to read each element's `shadowRoot.textContent` — which includes all text rendered by that component (badges, content, metadata).

### Why this is non-obvious
GE-20260614-cd8e92 established the rule "use the locator API — it auto-pierces." This creates confidence that Playwright handles shadow DOM transparently. The extraction gap is the second trap: you find the right elements, your wait conditions pass, but all text assertions fail with empty strings. No error is thrown — the standard DOM API silently returns "" for content hidden behind shadow boundaries.

### Why this fix
`evaluateAll()` on the leaf shadow hosts is the sweet spot: it uses Playwright's piercing locator to find the right elements (handles the nesting), then uses the standard DOM API at the *correct level* (each component's own shadowRoot.textContent, which contains that component's rendered text). The recursive approach works but adds complexity for no gain when you know the leaf level.
