### What

`el.textContent` is *defined* as the concatenation of the text content of all descendant `Text` nodes in DOM tree order. This makes it a reliable shared coordinate system between:

1. An algorithm that computes character ranges on flat text (e.g. a diff, regex, or NLP tagger)
2. A DOM mutation pass that needs to apply those ranges as inline element boundaries

```js
// Step 1: compute ranges on flat text
const text = el.textContent;
const changedRanges = computeRanges(text); // [[startChar, endChar], ...]

// Step 2: map ranges back to individual text nodes
const walker = document.createTreeWalker(el, 4); // NodeFilter.SHOW_TEXT = 4
const nodes = [];
let node, off = 0;
while ((node = walker.nextNode())) {
  nodes.push({ node, start: off, end: off + node.length });
  off += node.length;   // cumulative offset matches textContent indexing
}

// Step 3: apply highlights (process in reverse — see GE-20260522-405c25)
for (let i = nodes.length - 1; i >= 0; i--) {
  const { node, start, end } = nodes[i];
  const overlaps = changedRanges.filter(([rs, re]) => re > start && rs < end);
  // ... split and wrap changed portions in <mark> ...
}
```

### Why it works

The DOM spec guarantees: `node.textContent` on an element = `"".join(text_node.data for text_node in preorder_traversal if isinstance(text_node, Text))`. The `TreeWalker` with `NodeFilter.SHOW_TEXT` visits exactly those nodes in exactly that order. Summing `node.length` as the walker advances recreates the same string with the same indices.

This holds even with deeply nested inline elements: `<p>Hello <strong>world</strong>!</p>` has `textContent = "Hello world!"` and three text nodes ("Hello ", "world", "!") at offsets 0, 6, 11. Modifying `<strong>` or adding `<em>` wrappers doesn't affect the offset calculation as long as you don't change the text nodes themselves.

### Why this is non-obvious

The natural impulse is to compute ranges on a manually built string from `innerHTML`, which includes tag markup — making offset arithmetic much harder. Or to work on the raw source text (e.g. markdown), which diverges from the rendered text at every inline element. The `textContent` approach requires knowing the DOM spec guarantee and recognising that `TreeWalker` + length accumulation recreates it precisely.

### Caveats

- Ranges computed on `el.textContent` become invalid the moment any text node is added or modified — always collect text nodes and apply ranges atomically before any further DOM mutation.
- Skip text nodes whose parent should be excluded (e.g. `<code>` inline spans): track their `length` in the offset counter but don't add them to the processing list, so surrounding ranges remain correctly aligned.
- `NodeFilter.SHOW_TEXT = 4` — use the numeric constant inside `window.evaluate()` (Playwright) as the `NodeFilter` global may not be resolvable in serialised evaluation contexts.
