### Symptom
A Lit mixin with two data sources (direct property and DataSource adapter) uses a single `_cachedPoints` field for both. When both are active, the adapter's `onChange` callback overwrites the direct data in the shared cache — violating the precedence rule that direct data should always win.

### Context
Building a mixin that accepts data from two paths: a `trendData` property (direct import for demos/examples) and a `trendSource` property (DataSource for live data). The spec requires `trendData` to always take precedence over adapter data when both are set.

### What was tried (didn't work)
Single `_cachedPoints` field with precedence logic in the getter: `return this.trendData ? extractFrom(trendData) : this._cachedPoints`. The adapter's `onChange` callback sets `this._cachedPoints = extractFrom(adapterData)` — but this overwrites points derived from `trendData` because onChange fires asynchronously regardless of which source is "active".

### Root cause
The adapter stays connected even when `trendData` is set (by design — so adapter data is immediately available when `trendData` is cleared). Its `onChange` callback fires on every data event. With a single cache field, `onChange` writes to the same field the getter reads, creating a race between direct data and adapter data.

### Fix
Use two separate cache fields — one per source:

```typescript
@state() private _adapterTrendPoints: TrendPoint[] = [];
@state() private _directTrendPoints: TrendPoint[] = [];

// Adapter onChange populates _adapterTrendPoints
readonly _trendAdapter = new DataSourceAdapter(this, {
  onChange: () => {
    const ds = this._trendAdapter.dataSet as TypedDataSet | undefined;
    if (ds) {
      this._adapterTrendPoints = extractTrendPoints(ds)
        .toSorted((a, b) => a.timestamp - b.timestamp);
    }
  },
});

// Direct data populates _directTrendPoints in willUpdate
override willUpdate(changed: PropertyValues): void {
  super.willUpdate(changed);
  if (changed.has('trendData') && this.trendData !== undefined) {
    this._directTrendPoints = [...this.trendData]
      .sort((a, b) => a.timestamp - b.timestamp);
  }
}

// Getter resolves precedence + applies window
get trendPoints(): TrendPoint[] {
  const source = this.trendData !== undefined
    ? this._directTrendPoints
    : this._adapterTrendPoints;
  return source.slice(-this.maxTrendPoints);
}
```

The `maxTrendPoints` slicing happens in the getter (cheap) rather than in the cache population (expensive), so runtime changes to `maxTrendPoints` take effect without cache invalidation.

### Why this is non-obvious
The single-cache approach looks correct on paper — the getter has the precedence check. The bug only manifests when both sources are active simultaneously, which happens in the transition period when a user sets `trendData` while an adapter is already connected. The adapter's asynchronous `onChange` fires after `willUpdate` has already set the cache, silently replacing direct data with adapter data.
