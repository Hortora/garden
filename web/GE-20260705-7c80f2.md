### Symptom
Clicking a filter pill does nothing — the pill's active/inactive CSS class never changes. The filter state (a `Set<string>`) is correctly updated (verified by logging), but the child component receiving the Set as a property never re-renders.

### Context
A Lit parent component (`work-item-inbox`) passes a `Set<string>` filter state to a child component (`inbox-filter-bar`) via property binding: `.activeStatusFilters=${this.statusFilter}`. The parent mutates the Set with `.add()` and `.delete()`, then calls `this.requestUpdate()`.

### What was tried (didn't work)
- Adding `this.requestUpdate()` after mutation — parent re-renders but child still doesn't
- Using `@property({ type: Object })` on the child — same result
- Checking if the event handler fires — it does, the Set has the right values

### Root cause
Lit's reactive system uses strict equality (`===`) to determine if a property changed. Mutating a Set in-place with `.add()` or `.delete()` keeps the same object reference. When Lit passes the Set to the child via property binding, strict equality says "same object, no change" → child skips re-render.

`this.requestUpdate()` forces the PARENT to re-render, but Lit still compares the property value passed to the child — same reference → child doesn't update.

### Fix

Create a new Set on every mutation so the reference changes:

```typescript
// WRONG — mutates in place, same reference
this.statusFilter.add(value);
this.requestUpdate();

// RIGHT — new Set, new reference, Lit detects change
const next = new Set(this.statusFilter);
next.add(value);
this.statusFilter = next;
```

Same pattern applies to Map and Array — always replace, never mutate.

### Why this is non-obvious
The code looks correct: `.add()` works, the Set contains the right values, the parent re-renders. The bug is invisible — the child silently stays stale. A developer debugging this would check the event handler (fires correctly), check the Set contents (correct), and conclude the child component is broken. The actual cause is Lit's reference-equality check, which is documented but not intuitive when working with collection types.

### Why this fix
`requestUpdate()` looks like the right tool — it forces a re-render. But it only forces the component that calls it, not its children. Creating a new collection on every mutation is the idiomatic Lit pattern for mutable state, analogous to React's immutable state updates.

### Alternatives considered
- **`requestUpdate()` on the parent** — doesn't propagate to children
- **Custom `hasChanged` function on `@property`** — works but adds complexity per-property; new-reference pattern is simpler and universal
- **Using arrays instead of Sets** — same problem; arrays mutated in-place also keep the same reference
