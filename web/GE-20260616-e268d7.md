## Symptom

A function parameter added to a dependency package's source is silently ignored by the consuming package. TypeScript compiles without error — but the new parameter is not passed at runtime. The consuming code calls `createDataProviderFactory()` with zero arguments even though the source now accepts `createDataProviderFactory(fetchFn?)`.

## Context

In a Yarn workspace monorepo, packages import from siblings via `workspace:*` resolution. TypeScript resolves types from the dependency's `dist/*.d.ts` files (the `"types"` field in `package.json`), NOT from the source `.ts` files. When a dependency's source is modified but not rebuilt, the `.d.ts` files are stale.

This is particularly dangerous with subagent-driven development where each subagent works on one package in isolation — Task 2 modifies `packages/core/src/provider-factory.ts` to add a parameter, but the subagent for Task 13 (in a different package) sees the old `dist/provider-factory.d.ts` and writes code that omits the parameter.

## What was tried (didn't work)

1. Searching for the parameter in the consuming package's TypeScript errors — no errors, because the old `.d.ts` says the function takes zero arguments
2. Checking the source file — the parameter was clearly there, but TypeScript wasn't reading the source

## Root cause

TypeScript resolves cross-package types from built `.d.ts` declaration files, not from source `.ts` files. The `"types": "dist/index.d.ts"` field in `package.json` points to the build output. When a dependency's source changes without a rebuild, the declaration files are stale and TypeScript happily compiles against the old signatures.

The `moduleResolution: "bundler"` setting does NOT change this — it affects how import specifiers are resolved, not whether TypeScript reads `.d.ts` vs `.ts` files across workspace boundaries.

## Fix

Always rebuild dependency packages before building or testing consumers. In a monorepo with ordered builds:

```bash
# The build:packages script builds in dependency order
yarn build:packages
# Order: @casehub/component → @casehub/data → @casehub/ui → @casehub/viz → @casehub/runtime
```

For subagent workflows: rebuild the modified dependency before dispatching the consuming subagent. Or add a pre-build step to each package that rebuilds its workspace dependencies.

## Why this is non-obvious

The failure is completely silent. TypeScript reports no errors — the stale `.d.ts` is a valid type definition, just outdated. The consuming code compiles and runs, but the new parameter is simply not passed. There is no warning, no type error, and no runtime exception (the parameter has a default value). The only symptom is incorrect runtime behaviour that traces back to a missing argument.

### Alternatives considered

- **TypeScript project references (`composite: true`)** — these can enable cross-package source-level type checking, but require significant tsconfig restructuring and don't work well with `moduleResolution: "bundler"`. Overkill for this problem.
- **Per-package `prebuild` script** — each package could rebuild its workspace dependencies before its own build. Adds complexity and slows builds.
