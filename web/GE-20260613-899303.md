### Symptom
Code expecting an array-of-arrays (e.g. `[[1686700000, "1"]]`) receives a flat array
(`[1686700000, "1"]`) when the JSONata expression maps over an input that produces
exactly one result. Downstream code that indexes `data[0]` gets a primitive instead
of a sub-array — no error, just wrong data.

### Context
Any JSONata expression that maps over an array and produces a structured result per
element. When the input array has 2+ elements, the output is correctly nested. When
it has exactly 1 element, JSONata auto-unwraps the outer array.

Example: Prometheus API returns one metric result. The preset expression
`result.[value[0] * 1000, value[1], metric.*]` produces `[1686700000000, "1", "localhost:9090"]`
instead of `[[1686700000000, "1", "localhost:9090"]]`.

### What was tried (didn't work)
- Assuming JSONata would preserve array nesting consistently regardless of input cardinality
- Using `.(expr)` mapping syntax (also flattens)

### Root cause
JSONata v2 automatically unwraps sequences containing a single element. This is
by design in JSONata's evaluation model — sequences are not arrays, and a
single-element sequence collapses to the element itself. The `.[...]` array
constructor syntax helps but doesn't prevent the outer-level unwrap.

### Fix
Detect the flat case at the consumer and re-wrap:

```typescript
let valueRows: readonly (readonly unknown[])[];
if (data.values.length > 0 && !Array.isArray(data.values[0])) {
  valueRows = [data.values];  // single-result auto-unwrap: re-wrap
} else {
  valueRows = data.values as readonly (readonly unknown[])[];
}
```

The check `!Array.isArray(data[0])` distinguishes between a properly nested
array-of-arrays and a flat array from auto-unwrap.

### Why this is non-obvious
JSONata's docs describe sequences but don't explicitly call out this unwrapping
behavior for mapped array results. The behavior is consistent with JSONata's
sequence semantics but surprising when coming from JavaScript/TypeScript where
`.map()` always returns an array of the same cardinality. Multi-element inputs
work correctly, so the bug only surfaces with single-element data — which is
common in development/testing (one metric, one record) but masked in production
(many records).

### Alternatives considered
- Wrapping all JSONata output in `[$expr]` to force array — this double-wraps multi-element results
- Post-processing all JSONata results with a normalize function — adds overhead to every evaluation
