### Symptom
Playwright E2E tests timeout waiting for Lit web components (`pages-component-terminal`, custom elements) to appear in the DOM. Page HTML loads correctly but `<div id="app"></div>` remains empty — no web components register. Browser console shows:

```
Error: Unsupported decorator location: field
    at ng (terminal.js:107:8116)
```

But this error is only visible via `page.onPageError()` — Playwright's default timeout error gives no hint of the JS cause.

### Context
- TypeScript project using Lit 3 with `@state()` and `@customElement()` decorators on class fields (standard Lit pattern)
- esbuild bundles the frontend with `target: "es2020"`
- `tsconfig.json` does NOT set `experimentalDecorators: true`
- Playwright 1.52 ships Chromium 138 which has native TC39 decorator support

### What was tried (didn't work)
1. Checked if npm packages were installed (they were — `package-manager-install=false`)
2. Checked if Quinoa was serving files (it was disabled, but `copy-quinoa-dist` Maven plugin copies pre-built dist to META-INF/resources — Quarkus serves them regardless)
3. Verified all JS bundles and chunks exist in target directory
4. Checked MIME types and chunk import paths
5. Added `page.onConsoleMessage()` + `page.onPageError()` — this finally revealed the decorator error

### Root cause
Without `experimentalDecorators: true` in tsconfig.json, esbuild treats decorators as TC39 standard decorators and passes them through as native syntax (no lowering). Chromium 138's native TC39 decorator support then executes them — calling Lit's decorator functions with `kind: "field"` for class field decorators.

Lit's TC39 decorator implementation (`@property()`/`@state()`) only handles `"accessor"` and `"setter"` kinds. When called with `"field"`, it throws `Error("Unsupported decorator location: " + s)`. This prevents all web components from registering.

The gotcha is version-dependent: older Chromium without native decorator support won't trigger the error. Dev mode with an older browser works fine — the bug only manifests in Playwright CI or any environment with Chromium 133+.

### Fix
Add two lines to `tsconfig.json`:

```json
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "useDefineForClassFields": false
  }
}
```

- `experimentalDecorators: true` tells esbuild to use legacy decorator lowering (the `__decorate` helper pattern), which Lit fully supports on plain class fields
- `useDefineForClassFields: false` is required by Lit when using legacy decorators — class fields need prototype assignment, not `Object.defineProperty`

No code changes needed — the same `@state() private _foo = ''` syntax works with both legacy and TC39 decorators.

### Why this is non-obvious
1. **Symptoms are completely misleading** — "timeout waiting for element" suggests the page didn't load or the component wasn't imported. The actual JS error is invisible without explicit error handlers.
2. **Works in dev, breaks in CI** — older browsers without native TC39 decorator support never trigger the error. The bug only appears with Chromium 133+.
3. **Multiple red herrings** — Quinoa "disabled by default in tests" warning, GITHUB_TOKEN/npm install concerns, missing bundle speculation — all plausible but wrong.
4. **The fix is a config flag, not a code change** — the decorators are syntactically correct in both modes; only the runtime semantics differ.

### Alternatives considered
- Using `accessor` keyword on all Lit properties (`@state() accessor _foo = ''`) — works with TC39 decorators but requires rewriting all existing Lit components
- Adding `quarkus-quinoa-testing` dependency with `@TestProfile(QuinoaTestProfiles.Enable.class)` — was a red herring; Quinoa's state was irrelevant since files were already served via static resources
