**Context:** A monorepo has a base package (`@casehub/component`) exporting a generic function constrained by a registry interface, and a downstream package (`@casehub/ui`) that extends that registry with additional entries.

**Symptom:** Re-exporting `getProps` from the downstream package doesn't widen its generic constraint to include the extended registry entries. TypeScript consumers of the downstream package can only pass base registry keys — extended keys produce type errors.

**What was tried (didn't work):**
- Re-exporting the function directly (`export { getProps } from "@casehub/component"`) — the constraint stays bound to the base registry
- Declaration merging on the base `ComponentTypeRegistry` — works for the interface itself but does NOT retroactively widen the generic constraint on the already-exported function

**Root cause:** TypeScript resolves generic constraints at the function's declaration site, not at the consumer's import site. When `getProps<T extends keyof BaseRegistry>()` is declared in package A, that constraint is baked in. A downstream package B can extend `BaseRegistry` via declaration merging, but the function's generic parameter `T` still only accepts `keyof BaseRegistry` as it was when the function was declared.

**Fix:**
```typescript
// In the downstream package:
import { getProps as baseGetProps } from "@casehub/component";

export interface ComponentTypeRegistry extends BaseRegistry {
  "bar-chart": BarChartProps;
  // ... extended entries
}

export const getProps = baseGetProps as <T extends keyof ComponentTypeRegistry>(
  component: Component,
  type: T,
) => ComponentTypeRegistry[T];
```

The type assertion widens the generic constraint at the consumer level while preserving the single implementation. The runtime behaviour is identical — only the type-level constraint changes.

### Why this is non-obvious
The natural instinct is declaration merging (extend the interface, expect the function to pick up the wider type). TypeScript's documentation extensively covers declaration merging for interfaces and namespaces but never addresses function generic constraints across module boundaries. The type assertion approach feels like a workaround but is actually the correct mechanism — it's the only way to widen a generic constraint without duplicating the function body.

### Alternatives considered
- **Re-implement the function** in the downstream package — works but duplicates the implementation (4 lines, but conceptually wrong — there should be one `getProps`)
- **Make the base function accept `string`** instead of `keyof Registry` — loses type safety entirely
