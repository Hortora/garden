### Symptom
A flex column layout inside a flex row item overflows the viewport by exactly the height of an injected sibling element (e.g., a 48px header bar makes the container 48px taller than the viewport). No scrollbar, no error — the bottom content is simply clipped.

### Context
Full-viewport app layout where a flex row (`flex-direction: row`) contains slot containers. One slot becomes a flex column (via CSS class) and gets a header bar prepended alongside the existing content element. The content element has `height: 100%`.

### What was tried (didn't work)
1. `height: 100%` on the slot — didn't resolve (parent height comes from `align-items: stretch`, not an explicit value)
2. `height: 100% !important` — same result, computed height still exceeded viewport
3. `max-height: 100%` — didn't constrain
4. `min-height: 0` — irrelevant (item is growing, not refusing to shrink)
5. `overflow: hidden` on the slot — clips rendering but doesn't constrain the element's own dimensions

### Root cause
`align-items: stretch` sets a *minimum* cross-axis size equal to the container's cross dimension, but it doesn't set a *maximum*. When children's total height exceeds the container's cross dimension, the flex item grows beyond the container. `height: 100%` on a child doesn't resolve because the parent's height is derived from stretch, not an explicit property — creating a circular reference the CSS spec leaves undefined.

### Fix
Bypass flex cross-axis sizing entirely with absolute positioning:
```css
.parent-flex-row {
  position: relative;
}
.slot-that-gets-injected-content {
  position: absolute;
  inset: 0;
  display: flex;
  flex-direction: column;
  overflow: hidden;
}
.slot-content {
  height: auto;
  flex: 1;
  min-height: 0;
}
```

### Why this is non-obvious
The intuition that `height: 100%` on a child constrains it to the parent's height is correct in block layout but fails in flex cross-axis context. Five different CSS approaches were tried before absolute positioning resolved it. The overflow is silent — no scrollbar appears because the parent typically has `overflow: hidden`, so the content is simply clipped at the viewport edge.

### Alternatives considered
- Using `dvh` / `svh` viewport units instead of percentage heights — doesn't address the flex cross-axis issue since the problem is the slot's own height, not the viewport reference
- Restructuring the DOM to avoid injecting siblings — would require changing the layout system's rendering, disproportionate to the fix
