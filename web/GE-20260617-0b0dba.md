### Symptom
Viz elements (`casehub-bar-chart`, `casehub-selector`, etc.) render in the DOM but
`vizElement.dataSet` is `undefined` and `vizElement.error` is empty — no data loads,
no error appears. Waiting 500ms+ makes no difference.

### Context
A `loadSite()` function renders a component tree via `renderComponent()`. During
rendering, custom elements fire `connectedCallback` which dispatches `casehub-data-request`
events (bubbles: true, composed: true). An ancestor listener handles these events and
triggers the data pipeline.

### What was tried (didn't work)
1. Checked if `casehub-data-request` event was dispatched — it was (test listener caught it)
2. Added logging to the handler — handler never fired for rendering events but DID fire for manually dispatched events
3. Checked `AbortController` signal — not aborted
4. Checked custom element registration — elements were properly registered

### Root cause
`renderComponent(target, root)` was called BEFORE `target.addEventListener("casehub-data-request", handler)`.
During rendering, `connectedCallback` fires synchronously in `appendChild`. The event
dispatches and bubbles — but no listener exists yet. The event is lost. Listeners
registered afterward catch only future events.

### Fix
Register event listeners BEFORE calling renderComponent:

```typescript
// Event delegation BEFORE render — connectedCallback fires during render
target.addEventListener("casehub-data-request", handler);
target.addEventListener("casehub-filter", filterHandler);
// ... all listeners

// Render AFTER listeners — connectedCallback events are caught
renderComponent(target, root, { permissions, onNode });
```

### Why this is non-obvious
The "no data AND no error" symptom points toward the async data pipeline (promises not
resolving, resolver context missing, dataset scope empty). Investigation naturally goes
deep into the pipeline before considering that the event never reached it. A manually
dispatched test event reaching the handler confirms the handler IS registered — but it
was registered AFTER the rendering events were lost.

### Why this fix
The alternative — buffering events and replaying them after listeners are registered —
adds complexity for no benefit. The ordering constraint is simple and correct: listeners
before render.

### Alternatives considered
- Event buffering/replay: rejected — adds complexity, masks the real ordering requirement
- Deferred rendering (setTimeout): rejected — breaks synchronous connectedCallback guarantees
- Moving data requests to a separate init phase after rendering: rejected — would require
  all custom elements to support a two-phase lifecycle
