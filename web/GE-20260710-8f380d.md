### Context
TypeScript mixins return a `Constructor<InterfaceType> & T` intersection.
Methods declared `protected` in the mixin class body cannot be expressed as
`protected` in the `InterfaceType` — TypeScript object type literals don't
support access modifiers.

### Behaviour
```typescript
function MyMixin<T extends Constructor<LitElement>>(Base: T) {
  class Host extends Base {
    protected resolveEndpoint(): string | undefined { return this.endpoint; }
    // ...
  }

  return Host as unknown as Constructor<{
    resolveEndpoint(): string | undefined;  // forced public — no way to say protected
  }> & T;
}
```

Consumers that try `protected override resolveEndpoint()` get:
```
TS2415: Property 'resolveEndpoint' is protected in type 'MySubclass'
but public in type '{ resolveEndpoint(): ... }'.
```

Consumers that omit `override` get:
```
TS4114: This member must have an 'override' modifier because it overrides
a member in the base class.
```

### Fix
Accept the limitation. Make the method public in both the class body and the
return type. Consumers use `override` (not `protected override`):

```typescript
// Mixin class body — no protected
resolveEndpoint(): string | undefined { return this.endpoint; }

// Consumer — override, no protected
override resolveEndpoint(): string | undefined {
  return `${this.endpoint}/items/${this.itemId}`;
}
```

### Why this is non-obvious
The TypeScript handbook documents mixins with Constructor<> patterns but doesn't
mention this access modifier limitation. The error messages point at the
subclass, not at the mixin's return type — making it look like the consumer's
fault when the constraint is in the mixin's type erasure.
