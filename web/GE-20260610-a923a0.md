### Symptom

`yarn install` fails with `Duplicate workspace name @melviz/core: /path/packages/core conflicts with /path/core` when creating a new TypeScript package in a monorepo that also contains a Java/Maven project.

### Context

In the Melviz monorepo, `core/` is a Java/Maven project that compiles Java to JavaScript via GWT. It has a `package.json` with `"name": "@melviz/core"` so that Yarn recognizes it as a workspace (needed for the build orchestration). When a new TypeScript package `packages/core/` was created with the same workspace name, Yarn correctly rejected it as a duplicate.

### What was tried (didn't work)

1. Naming the new TypeScript package `@melviz/engine` — avoids the collision but the new package IS the core, not the Java wrapper.

### Root cause

Java/Maven projects in hybrid monorepos often have a `package.json` purely for build orchestration (to let Yarn scripts invoke `mvn`). This claims an npm workspace name that a developer wouldn't expect a Java project to own.

### Fix

Rename the Java project's `package.json` name to `@melviz/core-gwt` (the legacy being replaced), freeing `@melviz/core` for the TypeScript replacement. Update any workspace references (e.g. `webapp/package.json` dependency).

### Why this is non-obvious

A developer creating a TypeScript package wouldn't think to check whether a Java/Maven directory claims the same npm workspace name. The error message itself is buried at the end of thousands of lines of minified Yarn source code — easy to miss.
