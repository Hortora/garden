## Symptom

A custom element class and its `customElements.define()` call are present in
the webpack production bundle (verified by string search), but the element is
not registered at runtime (`customElements.get('pages-legend')` returns `undefined`).

## Root cause

Webpack's `ModuleConcatenationPlugin` (enabled in production mode) concatenates
multiple modules into a single scope. When a module's only observable side effect
is `customElements.define()` at module scope, the concatenated code may place the
`define()` call inside a function scope that is never reached during initial
module evaluation — even when `sideEffects: true` is set in package.json.

The `export { PagesLegend }` re-export in the package index and a side-effect
`import "./components/PagesLegend.js"` in a custom-elements registration file
both failed to ensure execution.

## Workaround

Render the component's content inline in the activation handler instead of
creating a custom element instance. For simple display components, this avoids
the registration issue entirely:

```typescript
// Instead of:
const legend = document.createElement("pages-legend");
legend.props = component.props;

// Do:
const ul = document.createElement("ul");
ul.className = "pages-legend";
if (layout === "vertical") ul.style.flexDirection = "column";
// ... build DOM directly
```

## When it bites

Any vanilla Web Component (not Lit — Lit uses decorators that webpack understands)
registered via `customElements.define()` at module scope, consumed through a
re-export chain in a webpack production build with module concatenation.

Lit components using `@customElement('pages-table')` are unaffected because the
decorator registration is part of the class definition expression.
