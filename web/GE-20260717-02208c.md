### Symptom
CSS custom properties like `--pages-success-contrast`, `--pages-danger-contrast`,
`--pages-warning-contrast` are referenced in specs and issues but don't resolve —
no value is generated, causing transparent/invisible text on colored backgrounds.

### Context
When migrating from a custom `--blocks-*` token system to `@casehubio/pages-ui-tokens`,
the natural assumption is that semantic color scales include a `-contrast` suffix for
text on colored backgrounds (matching patterns like Radix's `*Contrast` tokens). Multiple
documents — design specs, GitHub issues, and even a design review — referenced these
tokens without anyone noticing they don't exist.

### Root cause
`generateScale()` in `colours.ts` produces exactly 12 steps per semantic colour
(neutral, accent, success, danger, warning, info), keyed `1` through `12`. The CSS
properties generated are `--pages-{name}-{1..12}`. There is no `-contrast` suffix,
no `-bg`/`-text` split, no special token for "readable text on this colour's background".

### Fix
Use step 12 for text on a step-9 background. Step 12 is the extreme end of the scale
and automatically adapts per theme mode:
- Dark mode: step 12 ≈ 93% lightness (near-white) — light text on medium background
- Light mode: step 12 ≈ 18% lightness (near-black) — dark text on medium background

Mapping: `*-bg` → `*-9`, `*-text` → `*-12`, `*-hover` → `*-10`.

### Why this is non-obvious
The `-contrast` naming pattern is established in other design systems (Radix, Open Props).
Anyone familiar with those systems will assume it exists here. The misconception propagated
through a spec, four GitHub issues, and a multi-round adversarial design review before
being caught by reading the source.
