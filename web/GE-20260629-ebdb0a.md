### Symptom

New Web Components are defined in source files with `customElements.define()` calls, registered in a type declaration file, but do not appear in the webpack bundle. The components silently fail to render — no error, no custom element registration, DOM shows the tag as an unknown element.

### Context

A TypeScript monorepo with Web Components that self-register at import time:

```typescript
// PagesAlert.ts
export class PagesAlert extends HTMLElement { ... }
customElements.define("pages-alert", PagesAlert);
```

A `custom-elements.ts` file provides TypeScript type declarations:

```typescript
import type { PagesAlert } from "./components/PagesAlert.js";
declare global {
  interface HTMLElementTagNameMap {
    "pages-alert": PagesAlert;
  }
}
```

An `index.ts` barrel file imports `custom-elements.ts` and re-exports components.

### What was tried (didn't work)

Adding the component to `custom-elements.ts` with `import type` and the `HTMLElementTagNameMap` declaration — assumed this would cause webpack to include the module.

### Root cause

`import type` is erased at compile time by TypeScript. It provides type information only — it does NOT cause the module to be loaded at runtime. Since `customElements.define()` is a side effect that runs when the module is imported, a type-only import means the registration never executes.

Webpack's tree-shaking correctly excludes modules that are only type-imported — they have no runtime effect.

### Fix

Components that self-register via `customElements.define()` must have **value exports** (not type-only imports) in the barrel `index.ts`:

```typescript
// ❌ Wrong — type import, erased at compile time, define() never runs
import type { PagesAlert } from "./components/PagesAlert.js";

// ✅ Correct — value export causes module to load, define() executes
export { PagesAlert } from "./components/PagesAlert.js";
```

The `custom-elements.ts` file with `import type` is still needed for TypeScript's `HTMLElementTagNameMap` augmentation — but it is NOT sufficient to cause runtime registration.

### Why this is non-obvious

The `custom-elements.ts` pattern (type imports + HTMLElementTagNameMap) looks complete — TypeScript resolves the types correctly, IDE autocomplete works, and there are no compilation errors. The failure is entirely silent at runtime: the custom element tag is simply not registered, and the browser renders it as an unknown element with no error.

Existing components in the same codebase worked because they had value exports in `index.ts` — the type-only import in `custom-elements.ts` was supplementary, not the registration mechanism. New components added only to `custom-elements.ts` appeared to follow the same pattern but missed the critical value export.
