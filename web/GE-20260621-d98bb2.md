### Symptom
`SCHEMA_MISMATCH: Cannot parse "",status="200",} 8934" as NUMBER in column "Value"` — the error blames the Value column but the data line looks valid.

### Context
Prometheus exposition format lines like:
```
http_server_requests_seconds_count{method="GET",uri="/api/users/{id}",status="200",} 8934
```
The URI label value `/api/users/{id}` contains a `}` that is NOT the label block's closing brace.

### What was tried (didn't work)
Assumed the data was malformed. Checked the mock data file — the line was valid Prometheus format. The parser was the bug.

### Root cause
The parser used `trimmed.indexOf("}")` to find the end of the label block. When a label value contains `}` (common in URI templates like `{id}`, `{name}`), `indexOf` finds the wrong brace — the one inside the label value, not the one closing the label block.

### Fix
```typescript
// Before (broken):
const braceClose = trimmed.indexOf("}");

// After (fixed):
const braceClose = trimmed.lastIndexOf("}");
```
`lastIndexOf` finds the final `}` on the line, which is always the label block's closing brace — the numeric value follows it with a space separator.

### Why this is non-obvious
The parser worked for all metrics without URI template labels. The SCHEMA_MISMATCH error points at the Value column, not at the label parsing logic — you'd naturally look at the data, not the parser's brace-matching.
