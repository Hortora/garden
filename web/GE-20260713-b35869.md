### Symptom
Swipe gesture tests pass when they should fail — a "snap back" test (drag
insufficient distance) unexpectedly triggers the velocity threshold and opens
the drawer.

### Context
Any code that computes velocity from `PointerEvent.timeStamp` deltas in a
jsdom/happy-dom test environment. The events are dispatched synchronously in
the same microtask, so `timeStamp` values are sub-millisecond apart.

### What was tried (didn't work)
- `dt > 0` guard — timestamps are non-zero but nearly identical, producing
  `dx / 0.001 = huge velocity`
- `Object.assign(event, { timeStamp: ... })` — `timeStamp` is a read-only
  getter on `Event`, cannot be overridden
- `vi.useFakeTimers()` with `advanceTimersByTime()` — does not affect
  `performance.now()` which backs `Event.timeStamp`

### Root cause
jsdom dispatches synthetic events synchronously. `PointerEvent.timeStamp` is
set from `performance.now()` at construction time. Multiple events constructed
in the same synchronous block have near-identical timestamps (< 1ms apart).
A velocity calculation `Math.abs(dx) / dt` where `dt ≈ 0.001` produces
velocities of 10,000+ px/ms — any reasonable threshold (e.g. 0.5 px/ms) is
exceeded.

### Fix
Require a minimum time window before computing velocity. 5ms is safe — real
60fps pointer events arrive ~16ms apart.

```typescript
private _computeVelocity(now: number): number {
  const dt = last.t - first.t;
  // dt >= 5 instead of dt > 0 — jsdom timestamps are sub-millisecond
  return dt >= 5 ? Math.abs(last.x - first.x) / dt : 0;
}
```

### Why this is non-obvious
`dt > 0` looks correct — it guards against division by zero. But the actual
failure is not zero division; it's near-zero division producing absurdly high
velocities. The tests pass (no error, no NaN), just with wrong behaviour —
the velocity threshold is met when it shouldn't be.
