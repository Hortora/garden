### Symptom
Runtime error when a Lit component's `static styles` array includes a shared styles constant:
```
Error: Value passed to 'css' function must be a 'css' function result:
.pt-object { margin: 0; ... }. Use 'unsafeCSS' to pass non-literal values.
```

### Context
Sharing CSS between components by exporting a styles constant. The constant was
defined as a regular template literal (backtick string), not a Lit `css` tagged
template. The component's `static styles = css\`...\${sharedStyles}\`\`` then
fails at runtime because Lit validates that interpolated values are `CSSResult`
instances, not raw strings.

### Root cause
Lit's `css` tagged template function performs a runtime type check on all
interpolated values. It only accepts `CSSResult` instances (from other `css`
calls) or numbers. Raw strings are rejected to prevent CSS injection.

### Fix
Export shared styles using the `css` tag, not a backtick string:

```typescript
// WRONG — raw string, fails at runtime
export const sharedStyles = `
  .my-class { color: red; }
`;

// RIGHT — CSSResult, works in css`` interpolation
import { css } from "lit";
export const sharedStyles = css`
  .my-class { color: red; }
`;
```

### Why this is non-obvious
TypeScript compiles both forms without error. The failure is purely at runtime
when Lit's `css` function validates its interpolations. If you're exporting
styles from a utility module that doesn't otherwise use Lit, reaching for a
plain backtick string feels natural.
