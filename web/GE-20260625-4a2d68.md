### Symptom

Values containing structural separator characters (`:`, `|`, `,`) become indistinguishable from the actual separators after retrieval via `URLSearchParams.get()`. Custom query-param parsing produces incorrect field boundaries — a component ID or column name containing `:` is split into multiple fields.

### Context

A URL query parameter uses a custom compact encoding with structural separators. For example: `?sort=my-table:Revenue:DESCENDING` where `:` separates component ID, column name, and sort order. Values are encoded with `encodeURIComponent()` before building the URL, which encodes `:` as `%3A`.

### What was tried (didn't work)

Encoding values with `encodeURIComponent()` and assuming `URLSearchParams.get()` would return the still-encoded value. The encoding IS applied correctly at the URL level — but `URLSearchParams` reverses it.

### Root cause

`URLSearchParams.get()` automatically decodes percent-encoded characters before returning the value. This is correct per the WHATWG URL spec — `URLSearchParams` is designed for standard `key=value` pairs where percent-encoding is transparent.

The problem: `encodeURIComponent("my:table")` produces `my%3Atable` in the URL string. When retrieved via `new URLSearchParams(queryStr).get("sort")`, the `%3A` is decoded back to `:`. The code then splits on `:` and sees four fields (`my`, `table`, `Revenue`, `DESCENDING`) instead of three (`my:table`, `Revenue`, `DESCENDING`).

```typescript
// Encoding (works correctly)
const url = `?sort=${encodeURIComponent("my:table")}:Revenue:DESCENDING`;
// URL string: ?sort=my%3Atable:Revenue:DESCENDING  ← %3A is structural, : is separator

// Decoding (breaks)
const params = new URLSearchParams(url.substring(1));
const sortStr = params.get("sort");
// sortStr === "my:table:Revenue:DESCENDING"  ← %3A decoded back to :
// sortStr.split(":") === ["my", "table", "Revenue", "DESCENDING"]  ← WRONG
```

### Fix

Avoid `URLSearchParams` for custom-encoded query parameters. Parse the raw query string manually:

```typescript
function parseCustomParams(queryStr: string): Map<string, string> {
  const params = new Map<string, string>();
  for (const pair of queryStr.split("&")) {
    const eqIdx = pair.indexOf("=");
    if (eqIdx === -1) continue;
    // Do NOT decode here — let the custom parser handle decoding per-field
    params.set(pair.substring(0, eqIdx), pair.substring(eqIdx + 1));
  }
  return params;
}

// Then split on structural separators FIRST, decode each field individually
const sortStr = parseCustomParams(queryStr).get("sort");
for (const entry of sortStr.split(",")) {
  const parts = entry.split(":");
  const id = decodeURIComponent(parts[0]);      // decode AFTER splitting
  const column = decodeURIComponent(parts[1]);
  const order = parts[2];  // enum value, no encoding needed
}
```

Alternatively, use a separator that `encodeURIComponent` does NOT encode. Characters NOT encoded by `encodeURIComponent`: `- _ . ! ~ * ' ( )`. Using `~` as a separator means encoded values never contain raw `~`, so splitting is always unambiguous regardless of how the value is retrieved.

### Why this is non-obvious

`encodeURIComponent` + `URLSearchParams` are both standard Web APIs — using them together feels correct. The encoding IS correct at the URL level (the URL string contains `%3A`, not `:`). The problem only manifests at the parsing level, after `URLSearchParams` silently decodes. The WHATWG spec documents this behaviour, but developers building custom encoding schemes rarely consider that `URLSearchParams` will undo their encoding.

### Alternatives considered

- **Use `~` as separator instead of `:`** — avoids the issue entirely since `encodeURIComponent` doesn't encode `~`. Trade-off: less readable URLs (`sort=my-table~Revenue~DESCENDING`).
- **Double-encode** (`encodeURIComponent(encodeURIComponent(value))`) — technically works but produces ugly URLs and is fragile (easy to forget one layer of encoding/decoding).
- **Manual query parsing** (chosen) — preserves the readable `:` separator format, handles encoding correctly, and is a small amount of code.
