## Symptom

A `click` event listener on a canvas element never fires, even when the user clearly clicks without dragging.

## Context

Three.js visualizer canvas had a camera drag handler that called `e.preventDefault()` on `mousedown` to prevent text selection. A separate raycasting handler listened for `click` on the same canvas element. The click handler was never invoked.

## Root cause

Calling `e.preventDefault()` on `mousedown` prevents the browser from generating the subsequent `click` event. The `click` event is only synthesised if neither `mousedown` nor `mouseup` calls `preventDefault()`. This is specified in the DOM spec but easy to miss because `preventDefault()` on `mousedown` is commonly used for unrelated reasons (preventing text selection, focus management).

## Fix

Replace `click` with `mouseup` and track drag distance manually:

```javascript
let downX = 0, downY = 0;
renderer.domElement.addEventListener('mousedown', e => {
  downX = e.clientX; downY = e.clientY;
});
renderer.domElement.addEventListener('mouseup', e => {
  const dx = e.clientX - downX, dy = e.clientY - downY;
  if (dx*dx + dy*dy > 25) return; // was a drag, not a click
  // raycasting / click logic here
});
```

`mouseup` fires regardless of `preventDefault()` on `mousedown`.

## Why this is non-obvious

The DOM event sequence (`mousedown` → `mouseup` → `click`) gives no visible indication that `click` was suppressed. No error, no warning. The `click` listener attaches without complaint and silently never fires.
