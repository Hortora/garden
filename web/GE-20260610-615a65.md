### Symptom
Adding 1 month to January 31 produces March 3 in JavaScript but February 28 in Java.
No error or warning — the result is silently different.

### Context
Porting Java GWT date arithmetic to TypeScript. Java's `Calendar.add(Calendar.MONTH, 1)`
on January 31 clamps to February 28 (or 29 in leap years). JavaScript's
`Date.setUTCMonth(month + 1)` overflows: February 31 becomes March 3.

### What was tried (didn't work)
Direct port of `setMonth(getMonth() + amount)` — the Java pattern — assuming
identical behavior.

### Root cause
`Date.setUTCMonth` sets the month field but does not clamp the day-of-month.
When the target month has fewer days than the current day-of-month, the excess
days overflow into the following month. Java's `Calendar.add` clamps instead.

### Fix
Check whether the day-of-month changed after `setUTCMonth`. If it did, the
month overflowed — set the date to 0 to clamp to the last day of the target month:

```javascript
function addMonths(date, amount) {
  const d = new Date(date.getTime());
  const targetMonth = d.getUTCMonth() + amount;
  d.setUTCMonth(targetMonth);
  // If day-of-month changed, we overflowed — clamp to last day of target month
  if (d.getUTCMonth() !== ((targetMonth % 12) + 12) % 12) {
    d.setUTCDate(0); // sets to last day of previous month (= target month)
  }
  return d;
}
```

### Alternatives considered
- Accept the overflow and document the divergence from Java (current Melviz approach)
- Use a date library like date-fns or Temporal (adds dependency for a single operation)

### Why this is non-obvious
The Java and JS APIs look identical (`setMonth` / `setUTCMonth`), and both handle
simple cases identically. The divergence only appears on month-end dates (29th, 30th,
31st) crossing into shorter months. In dashboard date filtering, this produces
subtly wrong time ranges that are hard to detect.
