### Symptom

A frontend project needs to consume `@org/package-*` components from a sibling repo that hasn't published to any npm registry. `npm install` can't resolve them, and `file:` protocol references break transitive dependency resolution through Vite's bundler.

### Context

Multi-repo TypeScript platforms where UI component packages exist in one repo (e.g. `blocks-ui/`) and are consumed by application repos (e.g. `life/life-ui/`). During development, packages aren't published — they exist only as built or source directories in the sibling repo.

### Fix

Use Vite `resolve.alias` to map package imports directly to source directories in sibling repos:

```typescript
import { defineConfig } from 'vite';
import { resolve } from 'path';

const blocksUi = resolve(__dirname, '../../blocks-ui');
const pages = resolve(__dirname, '../../pages/packages');

export default defineConfig({
  resolve: {
    alias: [
      // Point to src/ directories — Vite compiles TypeScript via esbuild
      { find: '@org/blocks-ui-core', replacement: resolve(blocksUi, 'packages/core/src') },
      { find: '@org/blocks-ui-widget', replacement: resolve(blocksUi, 'components/widget/src') },

      // Deep-path imports need regex aliases to strip dist/ prefix
      { find: /^@org\/pages-data\/dist\/(.*)/, replacement: resolve(pages, 'pages-data/src/$1') },
      { find: '@org/pages-data', replacement: resolve(pages, 'pages-data/src') },

      // Pin lit to one copy (avoids duplicate registration errors)
      { find: 'lit', replacement: resolve(blocksUi, 'node_modules/lit') },
      { find: '@lit/reactive-element', replacement: resolve(blocksUi, 'node_modules/@lit/reactive-element') },
    ],
  },
  server: {
    fs: {
      // Allow serving files from parent directories
      allow: ['..', '../../blocks-ui', '../../pages'],
    },
  },
  esbuild: {
    tsconfigRaw: {
      compilerOptions: {
        experimentalDecorators: true,
        useDefineForClassFields: false,
      },
    },
  },
});
```

Key details:

1. **Point to `src/` not `dist/`** — Vite compiles TypeScript directly via esbuild, so you get HMR and avoid stale builds.

2. **Regex aliases for deep imports** — Components that import `@org/pages-data/dist/dataset/types.js` need a regex alias to rewrite `dist/` to `src/`. Without this, the alias matches the package root but not subpath imports.

3. **Pin shared dependencies** — Lit and `@lit/reactive-element` must resolve to a single copy. Multiple copies cause duplicate custom element registration errors. Pin them to the component repo's `node_modules`.

4. **`server.fs.allow`** — Vite's dev server blocks file access outside the project root by default. Explicitly allow parent directories containing sibling repos.

5. **Skip `tsc` in build** — TypeScript can't resolve Vite aliases. Use `vite build` alone; esbuild handles compilation. Add `paths` to `tsconfig.json` later for IDE support.

### Why this is non-obvious

The natural approach — `file:` protocol in `package.json` — creates symlinks but breaks transitive dependency resolution through Vite's bundler. `npm link` is fragile and non-declarative. The regex alias trick for deep-path imports (`/dist/` → `/src/`) is particularly non-obvious — without it, only top-level imports resolve while subpath imports silently fail.
