### Symptom

`UQ_MERKLE_FRONTIER_SUBJECT_LEVEL` unique constraint violation (`org.h2.jdbc.JdbcSQLIntegrityConstraintViolationException`) when multiple `LedgerEntry` subclasses are saved for the same `subjectId` concurrently. The exception originates inside an async CDI observer, so it is logged as:

```
ERROR DefaultAsyncObserverExceptionHandler: Failure occurred while notifying an async Observer
[method=io.casehub.ledger.service.CaseLedgerEventCapture#onCaseLifecycleEvent(...)]
Caused by: JdbcSQLIntegrityConstraintViolationException: Unique index or primary key violation:
"PUBLIC.UQ_MERKLE_FRONTIER_SUBJECT_LEVEL INDEX ON PUBLIC.LEDGER_MERKLE_FRONTIER(SUBJECT_ID, LEVEL)"
```

### Context

Two or more parallel casehub-engine workers fire for the same case simultaneously (e.g. `pattern-analysis` and `osint-screening` on the same `contextChange` tick). Each worker writes a `LedgerEntry` subclass via `LedgerEntryRepository.save()` for the same `subjectId`.

### Root cause

`LedgerEntryRepository.save()` triggers `LedgerTraceListener.prePersist()` → `LedgerEnricherPipeline.enrich()` → Merkle frontier update (`LEDGER_MERKLE_FRONTIER` table). Two concurrent saves for the same `subjectId` both try to write level 0 of the frontier in the same instant. H2's constraint fires; PostgreSQL's row-level locking serialises the writes instead.

The Merkle enrichment pipeline is not optional and not skippable through `LedgerEntryRepository` — it is wired into the JPA entity listener.

### Fix

For `LedgerEntry` subclasses that should NOT participate in the Merkle audit chain (e.g. internal attestation records not relevant to an external verifier), persist directly via `EntityManager` to bypass the enricher pipeline:

```java
// BAD for concurrent saves — triggers Merkle chain update
ledgerRepo.save(entry);

// GOOD for entities outside the Merkle chain — bypasses enricher pipeline
@PersistenceContext(unitName = "qhorus")
EntityManager em;

em.persist(entry);  // no LedgerTraceListener, no Merkle frontier update
```

If the entity MUST be in the Merkle chain, there is no clean workaround for H2 concurrent writes. The entity writes must be serialised, or the test must accept that Merkle verification (`LedgerVerificationService.verify()`) will return false in tests due to a partial frontier.

### Why this is non-obvious

`save()` sounds like a straightforward persistence operation. The Merkle chain update is a side effect buried in the JPA entity listener — not visible in the `LedgerEntryRepository` API. The constraint violation error message names the Merkle frontier table, making it look like a Merkle configuration problem rather than a concurrency issue triggered by a seemingly innocent `save()` call.
