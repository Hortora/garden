### What Was Discovered

`JpaLedgerEntryRepository.save()` is `@Transactional`. The first DB write it makes is `LedgerSequenceAllocator.nextSequenceNumber()`, which executes:

```sql
INSERT INTO ledger_subject_sequence (subject_id, next_seq)
VALUES (CAST(?1 AS UUID), 1)
ON CONFLICT (subject_id) DO UPDATE SET next_seq = ledger_subject_sequence.next_seq + 1
```

In PostgreSQL, this `INSERT ON CONFLICT DO UPDATE` acquires a write lock on the `ledger_subject_sequence` row for the given `subject_id`. That lock is held for the **entire duration** of the enclosing `@Transactional` transaction — which includes the Merkle frontier read-compute-replace cycle that comes later in the same `save()` call.

**Effect:** A second concurrent `save()` for the same `subject_id` blocks on the `INSERT ON CONFLICT` until the first transaction commits. By the time the second transaction proceeds, the first has committed its frontier update. The second reads the committed frontier and computes the correct next Merkle state.

This means the Merkle chain is **correct for concurrent JPA saves on the same subject** — not because of any explicit frontier locking, but because the sequence allocation lock incidentally serializes the entire pipeline.

### Why This Is Non-Obvious

The Merkle frontier update (`findBySubjectId → append → replace`) is a read-modify-write cycle with no `SELECT FOR UPDATE`. Inspection of the code suggests a race condition: two concurrent transactions could both read the same stale frontier and produce inconsistent chains. In practice, this race does not occur on PostgreSQL because the upstream sequence lock serializes them. The connection between sequence allocation locking and Merkle frontier safety is not documented anywhere in the codebase.

### The Merkle Serialization Invariant

Three facts must simultaneously hold for the implicit serialization to remain valid:

1. **`save()` is `@Transactional`** — the sequence lock must cover the frontier update.
2. **`nextSequenceNumber()` is the first per-subject serializing DB write in `save()`** — the lock must be acquired before the frontier read. Other writes (e.g. `actor_identity` tokenization for `ActorType.HUMAN`) may precede it but are per-actor, not per-subject.
3. **`JpaLedgerMerkleFrontierRepository.replace()` participates in the same transaction** — via `@Transactional(REQUIRED)` propagation.

If any of these changes — `save()` split into two transactions, sequence allocation moved to `REQUIRES_NEW`, `replace()` made `NOT_SUPPORTED` — the Merkle chain silently corrupts without any test or constraint catching it.

### InMemory Path

`InMemoryLedgerEntryRepository` does not benefit from this implicit serialization. It must use an explicit per-subject `synchronized` block covering sequence allocation through frontier update.
