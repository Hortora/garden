## quarkus-ledger sequence_number index is not unique — race yields silent duplicate sequences

### Symptom

Two concurrent `@ObservesAsync` handlers for the same `caseId` both compute `sequenceNumber = 1` (or the same N). Both rows insert successfully. No exception is thrown, no constraint is violated — the DB silently accepts duplicate sequence numbers.

### Context

`ledger_entry` has an index named `idx_ledger_entry_subject_seq`:

```sql
CREATE INDEX idx_ledger_entry_subject_seq ON ledger_entry (subject_id, sequence_number);
```

The name implies ordered, per-subject sequencing — but it is a plain performance index, not a `UNIQUE` constraint. There is no uniqueness guard on `(subject_id, sequence_number)` anywhere in the schema.

### What was tried

Using `MAX(sequenceNumber) + 1` inside a `@Transactional` observer method to compute the next sequence number. Under concurrent writes, two threads read the same `MAX` before either commits — both derive the same next value and both succeed.

### Root cause

The V1000 schema migration creates only an index, not a unique constraint. The `@Transactional` boundary does not prevent the read-then-write race at `READ COMMITTED` isolation because both reads happen before either write commits.

### Fix

Replace the `MAX()` aggregate with `ORDER BY sequenceNumber DESC / setMaxResults(1) / findFirst()` — the same pattern as `casehub-engine`'s `findLatestByCaseId()`. This eliminates the aggregate scan and uses the index, but has the same theoretical race window. For true concurrency safety, add a `UNIQUE` constraint or use a DB-level sequence object upstream.

```java
private int nextSequenceNumber(UUID caseId) {
    return em.createQuery(
                    "SELECT e FROM CaseLedgerEntry e WHERE e.subjectId = :caseId ORDER BY e.sequenceNumber DESC",
                    CaseLedgerEntry.class)
            .setParameter("caseId", caseId)
            .setMaxResults(1)
            .getResultStream()
            .findFirst()
            .map(e -> e.sequenceNumber + 1)
            .orElse(1);
}
```

### Why this is non-obvious

The index name `idx_ledger_entry_subject_seq` strongly implies that sequence ordering is enforced. Developers familiar with SQL will assume a named index on `(subject_id, sequence_number)` implies a uniqueness guarantee — distinguishing a plain index from a `UNIQUE` constraint requires reading the schema DDL directly. The schema migration comment mentions per-subject ordering but does not flag the absence of a uniqueness guarantee. Duplicate rows are also invisible to consumers unless they explicitly check for non-monotonic sequence numbers.
