### Symptom

`@QuarkusTest` boots fail with:

```
jakarta.enterprise.inject.spi.DeploymentException: MessageLedgerEntry declares
persistent fields (channelId, commitmentId, ...) but does not override
domainContentBytes(). These fields are not hash-protected.
```

The exception names every `@Column` field on the subclass. The error appears at build-step augmentation time — not at runtime and not during plain JUnit tests.

### Context

`casehub-ledger`'s Quarkus deployment module includes a `LedgerProcessor` build step that validates every `LedgerEntry` subclass. If the subclass declares any `@Column` persistent fields beyond those inherited from `LedgerEntry`, it must override `domainContentBytes()` to include them in the Merkle leaf hash and agent signature. Failing to override makes those fields invisible to tamper-detection.

The requirement is enforced at augmentation time only — there is no Javadoc on `LedgerEntry.domainContentBytes()` hinting this contract exists. The base implementation returns `EMPTY_BYTES`, so a new subclass silently works in unit tests but explodes the moment `@QuarkusTest` augmentation runs.

### Root cause

`LedgerProcessor.validateDomainContentBytes()` reflectively inspects all `LedgerEntry` subclasses for `@Column`-annotated fields. If any are found and `domainContentBytes()` is not overridden (i.e. the method is still the base class's `EMPTY_BYTES` version), the build step throws `DeploymentException`.

### Fix

Override `domainContentBytes()` in the subclass, returning a canonical byte representation of all `@Column` fields, using the same pipe-separated format as `LedgerEntry.canonicalBytes()`:

```java
@Override
protected byte[] domainContentBytes() {
    String canonical = String.join("|",
        channelId     != null ? channelId.toString()     : "",
        messageId     != null ? messageId.toString()     : "",
        messageType   != null ? messageType              : "",
        correlationId != null ? correlationId            : ""
        // ... all other @Column fields; order must be stable
    );
    return canonical.getBytes(StandardCharsets.UTF_8);
}
```

Include every `@Column` field — nullable fields use empty string as the sentinel. Order must be stable (field declaration order is the natural choice).

### Why this is non-obvious

- `LedgerEntry.domainContentBytes()` has no Javadoc and no abstract modifier — the default `EMPTY_BYTES` return silently passes in unit tests.
- The error message names the fields but not the fix — developers reach for "add the fields to the parent class" before realising the fix is an override in the subclass.
- The validation only fires during Quarkus augmentation, so TDD with plain JUnit gives no signal until `@QuarkusTest` is run.
