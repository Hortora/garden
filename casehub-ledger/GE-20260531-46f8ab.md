### Symptom

`LedgerErasureService.erase(rawActorId)` always returns `ErasureResult(rawActorId, mappingFound=false, affectedEntryCount=0)` in `@QuarkusTest`, even when `LedgerEntry` records with that `actorId` have been written by the test.

### Context

Writing a `@QuarkusTest` to verify GDPR erasure — starting an investigation (which writes ledger entries with a known `actorId`), then calling `erase()` and asserting `mappingFound = true`.

### Root cause

`LedgerPrivacyProducer` produces `ActorIdentityProvider` based on a config flag:

```java
@Produces @DefaultBean @ApplicationScoped
public ActorIdentityProvider actorIdentityProvider() {
    return config.identity().tokenisation().enabled()
        ? new InternalActorIdentityProvider(em)   // creates ActorIdentity rows
        : new PassThroughActorIdentityProvider();  // no-op
}
```

The default is `false` — `PassThroughActorIdentityProvider` is a complete no-op. It never creates `ActorIdentity` rows, so `LedgerErasureService` finds nothing to erase. The `ActorIdentity` table is always empty without this flag.

### Fix

Add to `application.properties` (test):

```properties
casehub.ledger.identity.tokenisation.enabled=true
```

With this flag set, `InternalActorIdentityProvider` creates a pseudonymous `ActorIdentity` token row whenever a `LedgerEntry` is persisted with an `actorId`. `erase()` finds the token, pseudonymizes it, and returns the actual count of affected entries.

### Why this is non-obvious

The property name does not appear in casehub-ledger's documented configuration. `LedgerErasureService` is injected and callable without error — the service starts and responds normally. The return value of `mappingFound=false` looks like the actorId was never recorded in ledger entries, when in fact the issue is that the identity mapping layer was never activated.
