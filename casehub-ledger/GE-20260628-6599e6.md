### Symptom
After calling `GdprErasureService.erase(actorId, tenancyId, reason)`, a JPQL query
scoped to the case's actors returns zero erasure receipts — even though receipts exist.

### Context
A compliance service builds per-case GDPR evidence. The natural approach is to extract
`actorId` values from the case's ledger entries, then query:
`SELECT e.id FROM ErasureReceiptLedgerEntry e WHERE e.tenancyId = :tenancyId AND e.erasedActorId IN :actorIds`

### What was tried (didn't work)
Scoping the erasure receipt query by the case's actor IDs. The query returns empty
because the actor IDs in the ledger entries are now opaque tokens, not the original
identity strings stored in `ErasureReceiptLedgerEntry.erasedActorId`.

### Root cause
`GdprErasureService.erase()` tokenises the actor's identity in all ledger entries —
replacing the original `actorId` with an opaque token and severing the token→identity
mapping. `ErasureReceiptLedgerEntry` stores the *original* `erasedActorId` (pre-tokenisation).
Post-erasure, `entry.actorId` (token) ≠ `receipt.erasedActorId` (original). The IN clause
matches nothing.

### Fix
Query erasure receipts by tenant only — not by actor:
```sql
SELECT e.id FROM ErasureReceiptLedgerEntry e WHERE e.tenancyId = :tenancyId
```
This is the only correct approach post-tokenisation. The compliance report answers
"has any GDPR erasure been performed in this tenant?" — it cannot answer "was this
specific case's actor erased?" because that link is intentionally destroyed.

### Why this is non-obvious
The actor-scoped query is the natural first instinct — it's more precise and avoids
false positives from unrelated erasures. It compiles, runs without error, and returns
an empty list. No exception, no warning. The silent empty result is only detectable
through test flakiness when test ordering varies.
