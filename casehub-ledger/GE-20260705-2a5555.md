## Symptom

After updating imports from `io.casehub.ledger.runtime.model.LedgerAttestation` to `io.casehub.ledger.api.model.LedgerAttestation` (following the ledger API migration), attestation writes silently stop working. Tests asserting `assertEquals(1, attestations.size())` get 0. No exceptions in test output — the `writeAttestation` catch block logs at WARN but the message is generic and doesn't surface in default test output.

## Context

The casehub-ledger API migration split `LedgerAttestation` into two classes:

- `io.casehub.ledger.api.model.LedgerAttestation` — `@MappedSuperclass` (no `@PrePersist`, no JPA table)
- `io.casehub.ledger.runtime.model.LedgerAttestation` — `@Entity` extending the API class, with `@PrePersist` that sets `id = UUID.randomUUID()` and `occurredAt = Instant.now()`

Both classes have the same simple name. Import auto-completion and mechanical import migration tools pick the API class (alphabetically first, seems canonical).

## What was tried (didn't work)

1. Verified `saveAttestation()` method signatures match the interface — they do (API class is correct for signatures)
2. Checked tenancyId consistency between COMMAND and DONE dispatches — correct
3. Checked `findAttestationsByEntryId` query — correct
4. Added debug logging — no output because the exception is caught before logging

## Root cause

`new io.casehub.ledger.api.model.LedgerAttestation()` creates a `@MappedSuperclass` instance. JPA's `em.persist()` throws because a `@MappedSuperclass` is not a managed entity type. The exception is caught by `LedgerWriteService.writeAttestation()`:

```java
} catch (final Exception e) {
    LOG.warnf("Could not write attestation for entry %s — trust signal lost but pipeline unaffected",
            commandEntry.id);
}
```

The WARN log doesn't include the exception details, making the root cause invisible.

## Fix

Import the runtime `@Entity` class for any code that **creates** attestation instances:

```java
// In LedgerWriteService.java and ReactiveLedgerWriteService.java:
import io.casehub.ledger.runtime.model.LedgerAttestation;

// In QhorusLedgerEntryRepository.java (method signatures):
import io.casehub.ledger.api.model.LedgerAttestation;  // API class is correct here
```

The rule: **creation sites** use the runtime `@Entity` class. **Method signatures** that match an interface use the API `@MappedSuperclass` class. Polymorphism handles the rest — a runtime instance IS-A API class.

## Why this is non-obvious

The import change looks mechanically correct — the class exists at the new path, the code compiles, and the method signatures match. The failure is entirely at runtime and entirely silent. The `@MappedSuperclass` annotation on the API class is the only clue, but it's in a dependency JAR — not visible during a routine import migration.

### Why this fix

The alternative — making `LedgerWriteService` log the full exception in the catch block — would have surfaced the symptom faster but wouldn't prevent the wrong import from being chosen. The fix targets the root cause: use the right class for the right purpose.

### Alternatives considered

- Log the full exception stack in `writeAttestation()` catch block — would help debugging but doesn't prevent the bug
- Use a single `LedgerAttestation` class (no API/runtime split) — would eliminate the confusion but violates the API module's "no JPA entity" rule
