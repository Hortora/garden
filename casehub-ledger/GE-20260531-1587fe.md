### Symptom

`@QuarkusTest` fails with `UnsatisfiedResolutionException` for type `LedgerMerkleFrontierRepository` at CDI startup, even after adding `JpaLedgerEntryRepository` to `quarkus.arc.selected-alternatives`.

```
Unsatisfied dependency for type io.casehub.ledger.runtime.repository.LedgerMerkleFrontierRepository
and qualifiers [@Default]
```

### Context

Adding `LedgerVerificationService` to a `@QuarkusTest` application that previously only used `LedgerEntryRepository`. Documentation for casehub-ledger typically shows only `JpaLedgerEntryRepository` in the `selected-alternatives` list.

### Root cause

`LedgerVerificationService` injects both `LedgerEntryRepository` and `LedgerMerkleFrontierRepository`. Both JPA implementations are `@Alternative`:

- `JpaLedgerEntryRepository implements LedgerEntryRepository` — `@Alternative`
- `JpaLedgerMerkleFrontierRepository implements LedgerMerkleFrontierRepository` — `@Alternative`

CDI requires explicit selection for every `@Alternative` bean. Adding only `JpaLedgerEntryRepository` to `selected-alternatives` leaves `LedgerMerkleFrontierRepository` with no satisfied implementation — the error fires even though the frontier repo is only transitively needed through `LedgerVerificationService`.

### Fix

Add both JPA repository implementations to `quarkus.arc.selected-alternatives` in `application.properties` (both main and test):

```properties
quarkus.arc.selected-alternatives=\
  io.casehub.ledger.runtime.repository.jpa.JpaLedgerEntryRepository,\
  io.casehub.ledger.runtime.repository.jpa.JpaLedgerMerkleFrontierRepository
```

### Why this is non-obvious

Casehub-ledger documentation and examples consistently show only `JpaLedgerEntryRepository` in `selected-alternatives`. `LedgerMerkleFrontierRepository` is an internal Merkle chain detail not mentioned in consumer-facing docs. The dependency is indirect — `LedgerVerificationService` is the consumer, not the caller's code — so the missing alternative is not visible from the caller's perspective.
