### Symptom
`IllegalStateException: LedgerEntry must be persisted through LedgerEntryRepository, which handles sequence allocation, enrichment, hashing, and signing. Direct em.persist() bypasses the entire save pipeline.`

Thrown at runtime when code calls `em.persist(entry)` on any `LedgerEntry` subclass (`CaseLedgerEntry`, `MessageLedgerEntry`, `WorkerDecisionEntry`, etc.).

### Context
A Quarkus extension using casehub-ledger, after upgrading to a SNAPSHOT that added a `LedgerProcessor` Quarkus build step. Any existing code that directly persisted `LedgerEntry` subclasses via `EntityManager.persist()` breaks with the above error.

### Root cause
`casehub-ledger` ships a `LedgerProcessor` Quarkus deployment build step (`validateDomainContentBytes`) that also adds a JPA `@PrePersist` interceptor to every `LedgerEntry` subclass. The interceptor throws `IllegalStateException` if it detects a direct persist (i.e. the entity was not saved through the `LedgerEntryRepository` save pipeline).

The intent: `LedgerEntryRepository.save()` handles sequence number allocation (via `ledger_subject_sequence` table), leaf hash computation, Merkle chain update, and agent signature — bypassing it produces invalid ledger entries.

### Fix
Replace `em.persist(entry)` + `em.flush()` with `ledgerRepo.save(entry, tenancyId)`:

```java
// ❌ Blocked since casehub-ledger SNAPSHOT (will throw at runtime):
em.persist(entry);
em.flush();

// ✅ Correct:
@Inject LedgerEntryRepository ledgerRepo;
// ...
ledgerRepo.save(entry, tenancyId);
// save() handles sequence allocation, Merkle hash, and signing; no manual flush needed.
```

Also remove `@LedgerPersistenceUnit EntityManager em` injection if it was only used for `persist()` + `flush()` and the `nextSequenceNumber()` query (the repo handles sequencing internally).

### Why this is non-obvious
The `IllegalStateException` message is clear once seen, but:
1. The validation is added at Quarkus augmentation time via a deployment build step — it does not appear in the `LedgerEntryRepository` interface Javadoc or any user-facing docs.
2. Code that was previously working (`em.persist()`) breaks silently if the SNAPSHOT is updated but the calling code is not.
3. The `@PrePersist` interceptor is injected by the Quarkus extension's deployment artifact (`casehub-ledger-deployment`) — the mechanism is invisible when reading the entity class.

### Why this fix
`LedgerEntryRepository.save()` is the only supported path because it atomically allocates the per-subject sequence number from `ledger_subject_sequence`, computes the Merkle leaf hash, and updates the Merkle frontier — operations that must be coordinated to maintain chain integrity. Any bypass produces a ledger entry with an incorrect or missing sequence number and hash.

### Caveats
- `LedgerEntryRepository.save()` triggers a full Merkle chain update — see GE-20260531-d2ed26 for concurrent-write implications.
- The `ledger_subject_sequence` table must exist in the database. In test contexts using `hibernate.database.generation=drop-and-create`, this native SQL table is not created from JPA entities — use `quarkus.hibernate-orm.<pu-name>.sql-load-script` to create it (see also GE-20260612-*).
