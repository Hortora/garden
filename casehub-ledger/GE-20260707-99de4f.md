casehub-ledger LedgerEntry api/runtime split — consumer subclasses must extend JpaLedgerEntry, not LedgerEntry

### Symptom

Hibernate generates INSERT statements that include ALL parent columns (`actor_id`, `actor_role`, `actor_type`, `occurred_at`, `digest`, `subject_id`, `tenancy_id`, etc.) in the consumer's subclass table. The database returns `Column "ACTOR_DID" not found` (or whichever parent column was most recently added).

### Context

`casehub-ledger` split `LedgerEntry` into two classes:
- `io.casehub.ledger.api.model.LedgerEntry` — `@MappedSuperclass`, field definitions only, no JPA inheritance. Lives in `casehub-ledger-api` (no framework deps).
- `io.casehub.ledger.runtime.model.jpa.JpaLedgerEntry` — `@Entity @Inheritance(JOINED) @DiscriminatorColumn`, extends `LedgerEntry`. Lives in `casehub-ledger` runtime module.

Consumer subclass tables (e.g. `aml_case_opened_ledger_entry`) were designed for JOINED inheritance — they only contain the join column (`id` FK to `ledger_entry`) plus subclass-specific columns. Parent columns live in the `ledger_entry` base table.

### Root cause

When a consumer `@Entity` extends `LedgerEntry` (the `@MappedSuperclass`), Hibernate treats it as a standalone entity — ALL parent fields are mapped directly to the subclass table. The `@DiscriminatorValue` annotation on the subclass is silently ignored (no `@Inheritance` on the parent). Hibernate generates INSERTs with 20+ parent columns that don't exist in the subclass table.

### Fix

Change consumer subclasses from `extends LedgerEntry` to `extends JpaLedgerEntry`:

```java
// WRONG — @MappedSuperclass, duplicates all parent columns to subclass table
import io.casehub.ledger.api.model.LedgerEntry;
public class AmlCaseOpenedLedgerEntry extends LedgerEntry { ... }

// CORRECT — @Entity @Inheritance(JOINED), subclass table only needs its own columns
import io.casehub.ledger.runtime.model.jpa.JpaLedgerEntry;
public class AmlCaseOpenedLedgerEntry extends JpaLedgerEntry { ... }
```

The engine-ledger module (`CaseLedgerEntry`, `WorkerDecisionEntry`) already extends `JpaLedgerEntry` — consumer modules must follow the same pattern.

### Why this is non-obvious

The error message (`Column "ACTOR_DID" not found`) points at a missing column, suggesting a migration fix. The natural response is to add the column via Flyway — which leads to adding ALL parent columns to every subclass table, fundamentally misunderstanding the schema design. The actual fix (changing the superclass) is in the Java import, not the SQL.

### Why this fix

Adding parent columns to subclass tables "works" but creates a TABLE_PER_CLASS schema that duplicates data, breaks the shared `ledger_entry` base table design, and defeats the purpose of JOINED inheritance. Extending `JpaLedgerEntry` preserves the original schema contract.

### Alternatives considered

- **Add all parent columns to each subclass table via Flyway** — compiles and runs but creates redundant data storage, breaks FK relationships, and diverges from the engine-ledger module's pattern.
- **Add `io.casehub.ledger.api.model` to the Hibernate PU packages** — makes Hibernate see the `@MappedSuperclass` but doesn't fix the inheritance strategy; still generates wrong INSERT statements.
