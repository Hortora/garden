## Symptom

Following the exclude-types list in GE-20260428-9311f8 verbatim causes `UnsatisfiedResolutionException` for SPIs the module does not implement:

```
Unsatisfied dependency for type io.casehub.api.spi.WorkerContextProvider
Unsatisfied dependency for type io.casehub.api.spi.ReactiveWorkerProvisioner
```

## Root Cause

GE-20260428-9311f8 lists all 8 no-op beans to exclude. That list assumes all 8 SPIs have custom implementations. If a module only implements the blocking variants of WorkerProvisioner, CaseChannelProvider, and WorkerStatusListener, excluding the reactive variants and WorkerContextProvider no-ops leaves those SPIs unsatisfied — nobody provides them.

## Fix

Only exclude the no-op beans for SPIs your module actually replaces. For a module that implements only the three blocking SPIs:

```properties
quarkus.arc.exclude-types=\
  io.casehub.engine.internal.worker.NoOpWorkerProvisioner,\
  io.casehub.engine.internal.worker.NoOpCaseChannelProvider,\
  io.casehub.engine.internal.worker.NoOpWorkerStatusListener
```

Leave the reactive variants and EmptyWorkerContextProvider — the engine's own no-ops remain active for those SPIs.

## Why This Is Non-Obvious

The example list in GE-20260428-9311f8 reads as the complete list to exclude. The symptom of excluding too many (UnsatisfiedResolutionException) looks like an entirely different problem from excluding too few (AmbiguousResolutionException). A developer who fixed the ambiguity with the full list would not expect the "solved" configuration to then fail with a different CDI error.
