## Symptom

Attempting to implement a custom callable task (e.g. `call: human` in YAML) via `CallableTaskBuilder<T>` SPI. The `accept(Class<? extends TaskBase>)` method cannot differentiate between `call: human` and `call: anyOtherFunction` â€” both produce `CallFunction.class`.

## Context

Building a Quarkus-Flow integration module that intercepts `call: tarkus` or `call: human` workflow steps to create human task WorkItems.

## What Was Tried

Registering a `CallableTaskBuilder<CallFunction>` via `META-INF/services` and implementing `accept(Class)` to return `true` for `CallFunction.class`. This would conflict with the existing `CallFunctionExecutorBuilder` (also registered for `CallFunction.class`) and cannot distinguish between different user-defined callable names.

## Root Cause

The `CallableTaskBuilder` SPI routes by Java class type only:

```java
public interface CallableTaskBuilder<T extends TaskBase> extends ServicePriority {
    boolean accept(Class<? extends TaskBase> clazz);  // class, not instance
    void init(T task, WorkflowDefinition definition, WorkflowMutablePosition position);
    CallableTask build();
}
```

For `call: human` in YAML, the SDK produces a `CallFunction` instance (same class as ALL user-defined callable references). The `accept()` method only sees `CallFunction.class` â€” it never sees the `"human"` string. You'd need to intercept at the `init()` level, but by then you've already committed to handling this type exclusively.

## Fix

Use a CDI bridge pattern instead of the `CallableTaskBuilder` SPI for human-task integration:

```java
@ApplicationScoped
public class HumanTaskFlowBridge {
    @Inject WorkItemService workItemService;
    @Inject PendingWorkItemRegistry registry;

    public CompletableFuture<String> requestApproval(String title,
            String description, String assigneeId,
            WorkItemPriority priority, String payload) {
        WorkItem wi = workItemService.create(...);
        return registry.register(wi.id);  // suspends until CDI event completes it
    }
}
```

Workflow developers inject `HumanTaskFlowBridge` and call it from a `function()` task in their Java-defined flow â€” no YAML SPI needed.

## Why This Is Non-Obvious

The `CallableTaskBuilder` SPI looks like the right extension point based on its name and the fact that the SDK registers HTTP/OpenAPI callables this way. The limitation is only apparent when you read the interface signature carefully and trace how `CallFunction.class` is produced from any `call: <name>` YAML â€” the name information is lost before `accept()` is called.
