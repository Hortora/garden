### Symptom

M-of-N SubCase bindings defined in YAML are silently treated as standalone sub-cases â€” no group coordination occurs. The parent case completes after the FIRST child completes, ignoring the M-of-N threshold.

### Context

The `SubCase` Java model has full M-of-N support: `groupId`, `totalInGroup`, `requiredCount`, `onThresholdReached` (with `OnThresholdReached.KEEP` and `.CANCEL` policies). These fields are used correctly in the fluent DSL (see `SubCaseMofNIntegrationTest`). However, the YAML schema (`CaseDefinition.yaml`) only supports: `namespace`, `name`, `version`, `completionStrategy`, `waitForCompletion`, `inputMapping`, `outputMapping`.

### What was tried (didn't work)

Adding `groupId`, `totalInGroup`, `requiredCount` to the YAML subCase section â€” fields are silently ignored by `CaseDefinitionYamlMapper.convertSubCase()`.

### Root cause

`CaseDefinitionYamlMapper.convertSubCase()` does not map M-of-N fields from the YAML schema model to the `SubCase.Builder`. The YAML schema definition (`CaseDefinition.yaml`) does not include these fields either.

### Fix

Add M-of-N SubCase bindings via Java augmentation in the `YamlCaseHub.getDefinition()` override, not in the YAML:

```java
@Override
public CaseDefinition getDefinition() {
    CaseDefinition yaml = super.getDefinition();
    SubCase child = SubCase.builder()
            .namespace("life").name("family-vote").version("1.0.0")
            .groupId("family-vote")
            .totalInGroup(3)
            .requiredCount(2)
            .onThresholdReached(OnThresholdReached.KEEP)
            .inputMapping("{ proposal: .proposal }")
            .build();
    yaml.getBindings().addAll(List.of(
            Binding.builder().name("vote-a").subCase(child)
                    .on(new ContextChangeTrigger(".isHighValue == true")).build(),
            // ... vote-b, vote-c
    ));
    return yaml;
}
```

### Why this is non-obvious

The SubCase Java model fully supports M-of-N â€” `SubCase.builder().groupId().totalInGroup().requiredCount()` all exist and work. The assumption that YAML supports the same fields is natural. The YAML silently drops the fields with no error or warning.
