## Symptom

Calling `CaseHub.startCase()` in a `@QuarkusTest` (or in production with JPA-backed SPIs) throws:

```
io.quarkus.runtime.BlockingOperationNotAllowedException:
  Cannot start a JTA transaction from the IO thread.
```

Propagated through the Vert.x event bus as:
```
(RECIPIENT_FAILURE,8185) io.quarkus.runtime.BlockingOperationNotAllowedException:
  Cannot start a JTA transaction from the IO thread
```

The error surfaces as a `CompletionStage` failure in `startCase()` — not in any test or application code that is obviously on the IO thread.

## Root Cause

`CaseHub.startCase()` publishes a message to the Vert.x event bus. The engine's handler that processes this message — which activates the case binding and calls `WorkerProvisioner.provision()` — runs on the Vert.x IO thread by default (not annotated with `@Blocking`).

When `provision()` calls `WorkerContextProvider.buildContext()`, which in turn calls a JPA-backed `CaseLineageQuery.findCompletedWorkers()` annotated with `@Transactional(REQUIRED)`, the JTA infrastructure detects the IO thread and throws.

The casehub-engine's own tests (`WorkBrokerEndToEndTest`) don't hit this because they use inline `Worker.function()` lambdas — those don't call JPA.

## Fix

**Consumer-side (immediate):** Change `@Transactional(REQUIRED)` to `@Transactional(TxType.SUPPORTS)` on read-only JPA methods called from the provision/buildContext chain. `SUPPORTS` joins an existing transaction if one is active; otherwise proceeds without starting one — safe from any thread context.

```java
@Override
@Transactional(TxType.SUPPORTS)
public List<WorkerSummary> findCompletedWorkers(UUID caseId) {
    // JPA SELECT — safe without a transaction
}
```

**Upstream fix (casehub-engine):** Add `@Blocking` to the event-bus handler method that calls `WorkerProvisioner.provision()`, so it dispatches to the Quarkus worker thread pool instead of the IO event loop.

**Test workaround:** If the full `startCase()` round-trip is needed in tests, bypass it by firing `CaseLifecycleEvent` directly via CDI `Event<CaseLifecycleEvent>.fireAsync()` — this exercises the ledger capture and lineage query pipeline without going through the IO-threaded engine handler.

## Why This Is Non-Obvious

The stack trace shows the exception inside `startCase()` with the Vert.x error code `RECIPIENT_FAILURE,8185` — nothing in the trace mentions the engine's event-bus handler or `@Blocking`. A skilled developer would add `@Blocking` to their own application code, not look for its absence in a library's internal handler. The casehub-engine's own tests pass (using inline lambdas), so the bug is invisible until a consumer implements JPA-backed SPIs.
