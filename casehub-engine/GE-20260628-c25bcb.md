### Symptom
Sentinel heartbeat results posted to a case channel as `MessageType.STATUS` never appear in case context. No error, no warning, no log entry — the messages are silently consumed and discarded by the bridge.

### Context
`QhorusMessageSignalBridge.onMessage()` observes all `MessageReceivedEvent` CDI events. It evaluates each message against `isCommitmentResolving(messageType)` — a positive whitelist that accepts only RESPONSE, DONE, DECLINE, and FAILURE. All other message types (STATUS, EVENT, COMMAND, QUERY, HANDOFF) are silently returned without processing.

A provisioner design that delivers periodic monitoring results via `MessageType.STATUS` — semantically the correct type for "status update" — will find that no results ever reach the case engine. The bridge receives the message, evaluates it, and returns without calling `runtime.signal()`.

### What was tried (didn't work)
Rev 2 of the Layer 7 heartbeat spec proposed: heartbeat job posts results to a per-case sentinel channel as `MessageType.STATUS` → bridge picks up → `runtime.signal()` sets `.sentinelReport` in case context → bindings react. This would have silently failed — every heartbeat result would have been dropped by the bridge.

### Root cause
The bridge's `isCommitmentResolving()` whitelist is designed for the commitment lifecycle (RESPONSE to a COMMAND, DONE/DECLINE/FAILURE to close a commitment). STATUS is an observational message type — the bridge intentionally ignores it because status updates are not commitment-resolving events. This filtering is defensible for the commitment model but creates a silent trap for provisioner designs that use STATUS for result delivery.

### Fix
Bypass the channel/bridge stack entirely. Use `CaseHubRuntime.signal(caseId, "sentinelReport", reportMap)` — a public API in `casehub-engine-api` — to deliver results directly to case context. The heartbeat job already knows the caseId and runs inside the application, so channel indirection adds no value.

```java
// Instead of posting to a channel:
// messageService.dispatch(MessageDispatch.builder().type(MessageType.STATUS)...);

// Signal case context directly:
caseHubRuntime.signal(caseId, "sentinelReport", result.output())
        .toCompletableFuture().join();
```

### Why this is non-obvious
`MessageType.STATUS` is the semantically correct type for a monitoring status report. The bridge's filtering is internal to `casehub-engine-runtime` — not documented in the engine API or the `ReactiveWorkerProvisioner` SPI contract. A provisioner implementer naturally assumes all message types posted to case channels reach the engine. The silent discard (no log, no error) means the failure is only discoverable by tracing through engine bytecode or observing that case context never updates.

### Alternatives considered
- Use `MessageType.RESPONSE` instead of STATUS — works mechanically but corrupts speech act semantics (a monitoring report is not a response to a command)
- File an engine issue to extend the bridge to accept STATUS — architecturally cleaner but creates an engine dependency and may not be desired (the filtering has valid reasons)
