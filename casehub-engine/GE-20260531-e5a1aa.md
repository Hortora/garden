### Symptom

Milestones defined in YAML with `completionCriteria:` are silently ignored — the milestone appears to load but the criteria evaluator is null, so the milestone never transitions to COMPLETED.

### Context

The `Milestone` Java API model has a field called `completionCriteria` (an `ExpressionEvaluator`). The YAML schema defines the equivalent field as `condition`. `CaseDefinitionYamlMapper` converts the YAML `condition` string to a `JQExpressionEvaluator` and passes it to `Milestone.Builder.completionCriteria()`.

### Fix

Use `condition` in YAML milestone definitions:

```yaml
milestones:
  - name: assessment-complete
    condition: ".assessment != null"
```

NOT `completionCriteria`:

```yaml
# WRONG — silently ignored
milestones:
  - name: assessment-complete
    completionCriteria: ".assessment != null"
```

### Why this is non-obvious

The Java API model uses `completionCriteria` — `Milestone.getCompletionCriteria()`, `Milestone.Builder.completionCriteria()`. A developer writing YAML would naturally use the same field name. The YAML schema uses `condition` instead (matching the `Goal` schema which also uses `condition`). No documentation explains this mapping.
