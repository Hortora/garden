## Symptom

Tests that assert on plugin-written CaseFile keys (`NEAREST_THREAT`, `STRATEGY`, `ENEMY_ARMY_SIZE`, etc.) via `getLastTickResult().caseFile()` always fail ŌĆö the keys are absent even after `gameTick()` completes. Debug logs confirm the plugins ran and wrote the keys. No error, no exception.

## Context

Any test that calls `AgentOrchestrator.gameTick()` (or equivalent) and inspects the returned `CaseFile` for plugin output. Common pattern:

```java
orchestrator.gameTick();
CaseFile cf = orchestrator.getLastTickResult().caseFile();
assertThat(cf.contains(QuarkMindCaseFile.NEAREST_THREAT)).isTrue(); // always false
```

## Root cause

`CaseEngine.createAndSolve(String, Map<String,Object>, Duration)` returns immediately with the initial CaseFile ŌĆö the pre-solve state containing only translator-written keys (`game.*` namespace). The `Duration` parameter sets a `PropagationContext` timeout, not a blocking-await timeout. Plugins run on a background control loop thread after `createAndSolve()` returns. The reference returned is the same `CaseFile` object the plugins will write into, but the caller has it before any plugin has executed.

Plugin-written keys (`agent.*` namespace) are absent from the returned reference at the time of return. If tests poll the same reference later (after the async solve completes), the keys appear ŌĆö but this is timing-dependent and a race condition.

## Fix

Do not assert on plugin-written keys via the returned CaseFile reference. Instead, call `canActivate()` directly on the injected plugin CDI bean against the returned CaseFile (which contains translator-written keys only):

```java
orchestrator.gameTick();
CaseFile initialCf = orchestrator.getLastTickResult().caseFile();

// Assert on translator-written keys (present immediately):
assertThat(initialCf.contains(QuarkMindCaseFile.ENEMY_UNITS)).isTrue();

// Assert on gate semantics via canActivate() ŌĆö does not require plugin execution:
assertThat(tacticsTask.canActivate(initialCf)).isFalse(); // NEAREST_THREAT absent ŌåÆ gate not met
```

For the positive case (proving a gate IS met), construct a synthetic CaseFile with the required keys rather than waiting for the async solve:

```java
// synthetic CaseFile ŌĆö created out-of-band from CDI to test canActivate() in isolation
var withThreat = new InMemoryCaseFileRepository()
    .create("starcraft-game", Map.of(), PropagationContext.createRoot());
withThreat.put(QuarkMindCaseFile.READY, Boolean.TRUE);
withThreat.put(QuarkMindCaseFile.NEAREST_THREAT, new Point2d(50, 50));
withThreat.put(QuarkMindCaseFile.STRATEGY, "DEFEND");
assertThat(tacticsTask.canActivate(withThreat)).isTrue();
```

`CaseEngine.awaitCompletion(CaseFile, Duration)` exists as a blocking overload ŌĆö if the test needs post-solve state, call it after `createAndSolve()`. Be aware that `awaitCompletion` blocks the test thread for the full timeout if the solve does not complete.

## Why this is non-obvious

`createAndSolve()` sounds like a synchronous operation that solves and returns a result. The name implies the CaseFile returned IS the solved result. The method signature with a `Duration` parameter reinforces this: it looks like a timeout for the blocking solve. Nothing in the API surface indicates the method returns before the control loop runs, or that the returned `CaseFile` is the initial state rather than the post-solve state.
