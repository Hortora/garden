### Symptom

Application `@ConsumeEvent` handler for `casehub.action.gate.rejected` or `casehub.action.gate.expired` calls `CaseInstanceCache.get(event.caseId()).getPendingActionGate()` expecting to read the gate type. Returns null. The ledger write or case signal is silently skipped.

### Context

Writing a gate decision listener that only processes a specific gate type (e.g., SUSAR oversight gates). The listener checks the pending gate's `plannedAction().actionType()` to discriminate before writing a ledger entry.

### Root cause

`ActionGateRejectedHandler` and `ActionGateExpiredHandler` are both `@ConsumeEvent(blocking = true)` on the same event bus addresses as application consumers. They are competing consumers — execution order against application consumers is non-deterministic on the Vert.x worker thread pool.

Both handlers clear `pendingActionGate` as part of their processing (before publishing downstream):

```java
// ActionGateRejectedHandler
instance.setPendingActionGate(null);  // cleared synchronously during handler execution
instance.getCaseContext().set("actionGateRejected", Map.of(...));
this.eventBus.publish("casehub.context.changed", ...);
```

`ActionGateApprovedHandler` is `@ConsumeEvent("casehub.action.gate.approved")` **without** `blocking = true` (runs on event loop). The clinical listener's `@ConsumeEvent(blocking = true)` for the approved address runs on a worker thread — different threading context, non-deterministic ordering.

If the engine handler runs first, `pendingActionGate` is null by the time the application consumer reads it.

### Fix

Use a DB query for gate discrimination instead of `CaseInstanceCache`:

```java
@ConsumeEvent(value = "casehub.action.gate.rejected", blocking = true)
public void onRejected(ActionGateRejectedEvent event) {
    // DB query: returns non-null only for the gate type this listener cares about
    MyDomainEntity entity = MyEntity.findByCaseId(event.caseId());
    if (entity == null) return; // not our gate type — silently skip
    // proceed with ledger write or case signal
}
```

This approach is:
- **Race-free** — the DB write (Phase 3 of the three-phase pattern) commits before the gate WorkItem is created
- **Restart-safe** — survives JVM restart unlike `CaseInstanceCache` (which is in-memory)
- **Complete** — provides all domain context (entity fields) without secondary cache lookups

### Why this is non-obvious

The `CaseInstanceCache` is injected and readable without error. `getPendingActionGate()` returns a non-null value in most test scenarios (engine handler may run after the application consumer), making the race appear reliable. The failure only manifests in production under load, or when Vert.x happens to schedule the engine handler first.
