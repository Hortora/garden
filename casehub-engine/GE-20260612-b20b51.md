### Symptom
`NullPointerException: Cannot invoke "com.fasterxml.jackson.databind.ObjectMapper.readTree(byte[])" because "objectMapper" is null`

Thrown from `CaseDefinitionYamlMapper.load()` when calling `new SomeCaseClass().getDefinition()` in a plain JUnit test (no `@QuarkusTest`, no CDI context).

### Context
Tests that instantiate a `YamlCaseHub` subclass directly (e.g. `new ResearcherCase()`) and call `getDefinition()` to verify YAML structure. These tests deliberately avoid `@QuarkusTest` for speed.

### Root cause
A casehub-engine SNAPSHOT added CDI injection of `ObjectMapper` (via `ObjectMapperInjector` startup bean) into `CaseDefinitionYamlMapper`. The field `objectMapper` on `YamlCaseHub` is now a CDI injection point — when the class is instantiated without CDI, it is `null`.

Before this change, `CaseDefinitionYamlMapper` used a static `ObjectMapper` instance that was always available.

### Fix
Convert the plain JUnit test to `@QuarkusTest` so CDI injects `objectMapper`. Inject the case class rather than instantiating it directly:

```java
// ❌ Breaks without CDI:
var def = new ResearcherCase().getDefinition();

// ✅ With @QuarkusTest + appropriate test profile:
@Inject ResearcherCase researcherCase;
// ...
var def = researcherCase.getDefinition();
```

Alternatively, use the static `CaseDefinitionYamlMapper.load(InputStream, ObjectMapper, ExpressionEngineRegistry)` overload directly in the test, providing an `ObjectMapper` manually.

### Why this is non-obvious
`new ClassName()` in Java has always worked for simple value objects. The change that makes it fail is invisible at the call site — `YamlCaseHub` still has a default constructor; CDI injection points are set after construction. The NPE happens inside `getDefinition()`, not during construction, making the CDI dependency non-obvious. The previous static `ObjectMapper` worked without CDI, so existing tests were not updated when the injection was added.
