## Symptom

`@QuarkusTest` fails with `ClassSelector resolution failed` for the test class name, or Quarkus
cannot resolve SPI implementations (`@ApplicationScoped @Alternative` in-memory repositories not found)
even though the module is on the classpath.

## Context

A Maven module needs in-memory SPI implementations (e.g. `InMemoryLedgerEntryRepository`) for
`@QuarkusTest` integration tests. The implementations live in a sibling module. Adding that module
as a Maven dependency appears correct but CDI discovery fails.

## Root Cause

Quarkus's build-time augmentation uses Jandex to discover CDI beans. A JAR without
`META-INF/jandex.idx` is invisible to this scan regardless of Maven scope.

## Solution 1 — Copy sources into the test tree

Copy the in-memory implementation source files into `src/test/java/` of the consuming module.
Remove the sibling module dependency. Violates DRY but zero configuration overhead.

```bash
cp casehub-persistence-memory/src/main/java/io/casehub/persistence/memory/InMemory*.java \
   my-module/src/test/java/io/casehub/persistence/memory/
```

Established pattern in the `casehub-engine` module (from PR3).

## Solution 2 — Ship a Jandex index in the module JAR (preferred)

Add `jandex-maven-plugin` to the in-memory module's `pom.xml`. Add it as `compile` scope in
consumers (not `test` — test scope excludes the JAR from augmentation). DRY, works across
any number of consumers.

See **GE-20260427-c2b84f** for the full Jandex configuration and explanation.

## Why This Is Non-Obvious

Adding a Maven test dependency is the standard Java approach and works for plain JUnit. The
failure message (`ClassSelector resolution failed`) gives no indication that Quarkus's Jandex
scan didn't index the class. The Jandex fix feels unrelated — the connection is that Quarkus
augmentation reads Jandex, not the classloader.
