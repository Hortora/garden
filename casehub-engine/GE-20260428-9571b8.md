## Symptom

`LedgerAttestation.confidence` values are stored in the database but have zero effect
on the computed trust score. Setting `confidence=0.7` vs `confidence=1.0` produces
identical `ActorTrustScore` values.

## Context

`TrustScoreComputer.compute()` in quarkus-ledger iterated attestations and incremented
α/β by `recencyWeight` — a time-decay factor only. The `attestation.confidence` field
was present on `LedgerAttestation`, queryable, and stored — but never read during
the Beta update.

## Root cause

`TrustScoreComputer` computed `recencyWeight = 2^(-ageInDays / halfLifeDays)` and then:
```java
alpha += recencyWeight;  // NOT multiplied by confidence
```

The `confidence` field was planned but not wired into the computation.

## Fix

The Beta update should weight by confidence:

```java
final double weight = recencyWeight * Math.max(0.0, Math.min(1.0, attestation.confidence));
if (verdict == SOUND || verdict == ENDORSED) {
    alpha += weight;
} else if (verdict == FLAGGED || verdict == CHALLENGED) {
    beta += weight;
}
```

`confidence=0.0` → no contribution to the score. `confidence=1.0` → full recency weight
(identical to the old behaviour). Also: add `DEFAULT 1.0` to the DB column so existing
attestations written without explicit confidence still contribute full weight.

## Why this is non-obvious

- The `confidence` field is present in the schema, the entity, and the `saveAttestation`
  API — nothing indicates it is unused
- The trust score still changes with attestations (just ignores confidence weighting) —
  the system appears to work
- No tests exercised the difference between `confidence=0.5` and `confidence=1.0` because
  `TrustScoreComputer` is a pure Java class only tested for overall score direction
