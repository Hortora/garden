## Symptom

Changing `entryCriteria()` on a `TaskDefinition` implementation has no effect on runtime plugin activation. Plugins fire unconditionally every tick regardless of what keys are declared in `entryCriteria()`. No error, no warning.

## Context

Any `YamlCaseHub` or programmatic `TaskDefinition` implementation that overrides `entryCriteria()` to gate plugin activation based on CaseFile key presence. Expected: plugin skips when declared keys are absent. Actual: plugin always fires.

## Root cause

`TaskDefinition.canActivate(CaseFile)` is a `default` method in the interface. Bytecode inspection (`javap -p`) of the installed `casehub-core-1.0.0-SNAPSHOT.jar` shows:

```
public default boolean canActivate(io.casehub.core.CaseFile);
  Code:
    iconst_1
    ireturn
```

The default unconditionally returns `true`. The CasEngine calls `canActivate()` — not `entryCriteria()` — to decide whether to schedule a task. Since the default ignores `entryCriteria()`, the criteria are never evaluated by the engine.

## Fix

Override `canActivate()` explicitly on every `TaskDefinition` implementation that uses entry criteria:

```java
/**
 * Overrides the {@code TaskDefinition} default, which unconditionally returns {@code true}
 * in the installed casehub-core snapshot — ignoring {@link #entryCriteria()}.
 * Override required until the foundation corrects the default.
 */
@Override
public boolean canActivate(CaseFile caseFile) {
    return entryCriteria().stream().allMatch(caseFile::contains);
}
```

This override is the correct implementation the default _should_ have. Once the foundation fixes the default, the overrides can be removed.

## Why this is non-obvious

The `entryCriteria()` method is prominently documented as the plugin activation contract — the obvious expectation is that returning non-READY keys from `entryCriteria()` will gate activation. Nothing in the API surface or documentation indicates that `canActivate()` is the actual decision point and that the default implementation ignores `entryCriteria()` entirely. The symptom (unconditional activation) could have many other causes and does not point to the interface default.
