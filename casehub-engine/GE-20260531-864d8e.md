### Symptom

CDI observer method decorated with `@Observes WorkerDecisionEvent` is never invoked — no error, no log output, zero calls. Investigation reveals the event does fire (engine logs confirm workers executed), but the observer receives nothing.

### Context

Implementing a CDI observer that should react to each worker dispatch in a casehub-engine case. The observer class is `@ApplicationScoped` and the observer method signature is correct.

### What was tried (didn't work)

- Verifying the bean is CDI-discovered and the persistence unit wiring is correct
- Adding `@Transactional` to the observer method
- Checking for `quarkus.arc.exclude-types` accidentally excluding the observer class

### Root cause

`casehub-engine` fires `WorkerDecisionEvent` via `Event.fireAsync()`, not `Event.fire()`. CDI `@Observes` is the synchronous variant — it only responds to `Event.fire()`. For async events, `@ObservesAsync` is required.

From the engine internals (not documented in casehub-engine's API docs):
```java
// Inside the engine — fires async
event.fireAsync(new WorkerDecisionEvent(caseId, workerId, capabilityTag, traceId));
```

Synchronous observers registered with `@Observes` are simply skipped for async events with no warning or error.

### Fix

Replace `@Observes` with `@ObservesAsync` on any method observing `WorkerDecisionEvent`:

```java
// BAD — silently never called
@Transactional(TxType.REQUIRES_NEW)
public void onWorkerDecision(@Observes WorkerDecisionEvent event) { ... }

// GOOD
@Transactional(TxType.REQUIRES_NEW)
public void onWorkerDecision(@ObservesAsync WorkerDecisionEvent event) { ... }
```

Note: `@ObservesAsync` handlers run on a separate executor thread. If the handler needs database access, `@Transactional(REQUIRES_NEW)` gives it an independent transaction (decoupled from the engine worker's transaction).

### Why this is non-obvious

The observer method signature looks correct and CDI compiles without error. Nothing in the engine documentation states that `WorkerDecisionEvent` is async. The absence of invocations looks like a CDI discovery or wiring problem, not an `@Observes` vs `@ObservesAsync` mismatch. You must read the engine source to discover the `fireAsync()` call.
