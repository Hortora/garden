## What It Is

`CaseHubRuntime.signal(UUID caseId, String path, Object value)` does NOT fire a named signal
event. It directly patches the case context at the specified dot-notation path and then fires
`CONTEXT_CHANGED` — which triggers goal evaluation, binding evaluation, and case completion checks.

## How to Use It

```java
// Sets context.workers.researcher.exited = true
caseHubRuntime.signal(caseId, "workers.researcher.exited", true);

// Sets context.event = {type: "click", source: "button"}
caseHubRuntime.signal(caseId, "event", Map.of("type", "click", "source", "button"));
```

`CaseContextImpl.applyAndDiff()` splits the path on `.` and creates nested maps:
- `"workers.researcher.exited"` → `{workers: {researcher: {exited: true}}}`
- `"event"` → `{event: {type: "click", ...}}`

After the patch, the engine fires `CaseContextChangedEvent`, which evaluates all goal conditions.
This means `signal()` can be used to satisfy goals and drive case completion — not just to
notify the case that something happened.

## Why This Is Non-Obvious

The method is named `signal()` and takes a `signal` parameter — typical signal dispatch APIs
fire events by name. You'd expect it to trigger a `signalReceived` handler that routes the
payload somewhere. Instead it writes directly to context and immediately fires context evaluation.

Verified by reading `SignalReceivedEventHandler.applySignalUnderLock()` and `CaseContextImpl.applyAndDiff()` source.

### Why this fix

Enables case auto-completion driven entirely by external infrastructure: when a tmux session
exits, the exit watcher calls `signal("workers.researcher.exited", true)` which sets context and
satisfies the YAML goal `.workers.researcher.exited == true`. No changes to Claude's workflow,
no MCP tool, no file conventions needed.
