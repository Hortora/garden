### Symptom

Child cases started via `CaseHubRuntime.startCase(definition, inputData, parentCaseId, propagationContext)` are created with `parentCaseId` set on the `CaseInstance`, but completion of the child case does NOT propagate to the parent. The parent case waits indefinitely.

### Context

The `CaseHubRuntime.startCase()` overload with `parentCaseId` looks like the correct API for programmatic sub-case creation. The parameter name suggests it establishes a full parent-child relationship. In reality, it only sets a field on the `CaseInstance` — none of the sub-case lifecycle management is triggered.

### What was tried (didn't work)

Calling `CaseHubRuntime.startCase(childDefinition, childContext, parentCaseId, propagation)` directly from application code to create N dynamic sub-cases. Expected the engine to track completion and propagate it to the parent. It does not.

### Root cause

`SubCaseExecutionHandler` (the `@ConsumeEvent` handler for `SubCaseScheduleEvent`) is the ONLY code path that performs sub-case lifecycle management. It does five things that `CaseHubRuntime.startCase()` alone does NOT do:

1. Writes `SUBCASE_STARTED` EventLog entry (metadata includes childCaseId, groupId, bindingName, outputMapping)
2. Creates/updates `SubCaseGroup` via `ReactiveSubCaseGroupRepository` for M-of-N tracking
3. Marks parent's `PlanItem` as DELEGATED and indexes `childCaseId → planItemId` for completion routing
4. Sets parent state to `CaseStatus.WAITING`
5. Registers with `PendingWorkRegistry`

`SubCaseCompletionService.handleCompletion()` searches for `SUBCASE_STARTED` EventLog entries to find the parent. Without these entries, child completion fires into the void.

### Fix

Do not use `CaseHubRuntime.startCase()` with `parentCaseId` for application-level sub-case orchestration. Instead, start child cases as independent top-level cases and implement coordination in the application layer:

1. Start each child via the normal application service (e.g. `PrReviewCaseService.startReview()`)
2. Track the parent-to-child mapping in an application-level tracker (e.g. `CoordinatedChangeTracker`)
3. Observe `CaseLifecycleEvent` for child completion/fault
4. Signal the parent case context when coordination conditions are met

This approach avoids coupling to engine internals while providing the same coordination semantics.

### Alternatives considered

- **Publishing `SubCaseScheduleEvent` to the event bus** — would go through `SubCaseExecutionHandler` and get full lifecycle. Rejected: requires constructing `CaseInstance` and `SubCase` model objects, coupling application code to engine internals.
- **Engine extension (forEach binding)** — add dynamic iteration to the YAML schema. Rejected: generic feature for one use case; application-level coordination is simpler and domain-appropriate.

### Why this is non-obvious

The `parentCaseId` parameter on `CaseHubRuntime.startCase()` creates a false sense of completeness. A developer would reasonably expect that passing a parent ID establishes the full parent-child lifecycle — EventLog linkage, completion propagation, group tracking. The parameter name gives no hint that these are handled by an entirely separate code path (`SubCaseExecutionHandler`) triggered only by YAML sub-case bindings.
