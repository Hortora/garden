## Symptom

Awaitility condition polling `CaseHubRuntime.eventLog(caseId, Set.of(CaseHubEventType.WORKER_EXECUTION_COMPLETED))` never returns any worker names — the condition times out even though debug logs confirm workers are executing and completing within milliseconds.

```java
// This always returns an empty set even after workers complete
private Set<String> completedWorkerNames(UUID caseId) {
    return caseHubRuntime.eventLog(caseId, Set.of(WORKER_EXECUTION_COMPLETED))
        .toCompletableFuture().join()
        .stream()
        .filter(r -> r.metadata() != null && r.metadata().has("workerName"))
        .map(r -> r.metadata().get("workerName").asText())
        .collect(Collectors.toSet());
}
```

## Context

`CaseHubRuntime.eventLog()` returns `List<CaseEventLogRecord>` where each record has `eventType`, `streamType`, `timestamp`, `payload`, and `metadata`. Two different event types write different metadata:

- `WORKER_SCHEDULED` — written by `WorkerScheduleEventHandler` → metadata contains `"workerName"`, `"capabilityName"`, `"inputDataHash"`
- `WORKER_EXECUTION_COMPLETED` — written by `WorkflowExecutionCompletedHandler` → metadata contains `"inputDataHash"` and `"contextChanges"` **but NOT `"workerName"`**

The `EventLog.workerId` field IS set correctly on `WORKER_EXECUTION_COMPLETED` events, but `CaseEventLogRecord` doesn't expose `workerId` — only `metadata`.

## Root cause

`WorkflowExecutionCompletedHandler.buildMetadata()` builds metadata from `idempotency` (the inputDataHash) and `contextDiff` only. The `worker.getName()` is available but was not included:

```java
private JsonNode buildMetadata(String idempotency, JsonNode contextDiff) {
    ObjectNode metadata = OBJECT_MAPPER.createObjectNode();
    metadata.put("inputDataHash", idempotency);   // only this
    if (contextDiff != null) {
        metadata.set("contextChanges", contextDiff);
    }
    return metadata;
}
```

`workerName` is set on the `EventLog.workerId` field, but `CaseHubRuntimeImpl.eventLog()` maps to `CaseEventLogRecord` which drops `workerId`.

## Fix

Poll `WORKER_SCHEDULED` events instead of `WORKER_EXECUTION_COMPLETED`. `WORKER_SCHEDULED` always carries `workerName` in metadata, and a worker is only scheduled when its binding condition is satisfied (meaning all prerequisite workers have completed and written to context).

```java
private Set<String> scheduledWorkerNames(UUID caseId) {
    return caseHubRuntime.eventLog(caseId, Set.of(CaseHubEventType.WORKER_SCHEDULED))
        .toCompletableFuture().join()
        .stream()
        .filter(r -> r.metadata() != null && r.metadata().has("workerName"))
        .map(r -> r.metadata().get("workerName").asText())
        .collect(Collectors.toSet());
}
```

**Why WORKER_SCHEDULED is semantically correct for adaptive path tests:** A binding only fires when its JQ condition is satisfied. `sar-drafting` being scheduled proves that entity-resolution, pattern-analysis, and osint-screening all completed and wrote their outputs to case context. "Was the binding triggered?" (SCHEDULED) is the right question for adaptive routing tests.

## Why this is non-obvious

`CaseHubRuntime.eventLog()` returns `CaseEventLogRecord` which silently drops the `workerId` field from `EventLog`. Debug logs (`QuartzWorkerExecutionJobListener` and `WorkflowExecutionCompletedHandler`) clearly show workers completing, so the symptom leads you to believe the event log query itself is filtering incorrectly — not that the completed events simply don't carry the expected metadata key.

### Why this fix

WORKER_SCHEDULED is not just a workaround — it's the semantically correct check. Scheduling is driven by binding evaluation, which is the Layer 5 teaching point. Testing "did the binding fire" is more direct than "did the worker function return". If the engine is working correctly, scheduling implies completion for synchronous in-process workers.

### Alternatives considered

- Cast `EventLog.workerId` field directly by querying the repository (bypasses CaseHubRuntime API) — works but couples tests to the SPI implementation
- Filter by payload content to infer which worker ran — fragile and unreliable
- Use `WorkResultSubmitter` to manually complete workers (bypasses Quartz entirely) — valid for external-worker tests, wrong for testing in-process worker dispatch
