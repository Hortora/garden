### Symptom

After migrating imports from `io.casehub.api.model.FlowWorkerFunction` (pre-#543) to the same package in the new engine-api module (#543), compilation fails: `cannot find symbol: class FlowWorkerFunction`.

### Context

Engine #543 (worker-api migration) moved `FlowWorkerFunction` from `io.casehub.api.model` in the old engine-api to `io.casehub.api.model` in the new engine-api — same package, different module. Engine #567 (remove serverlessworkflow SDK) then moved it again to `io.casehub.engine.flow` in the `casehub-engine-flow` module.

### Root cause

If you migrate against the #543 API (reading the engine-api jar or the #543 issue description), you get `io.casehub.api.model.FlowWorkerFunction`. But the installed engine jars are from the #567 branch where it moved to `io.casehub.engine.flow.FlowWorkerFunction`. The class simply doesn't exist at the #543 location in the #567 jars.

### Fix

Check the actual installed jar, not the issue description:
```bash
# Find where FlowWorkerFunction actually lives
find ~/.m2/repository/io/casehub -name "*.jar" -exec sh -c 'jar tf "$1" | grep FlowWorkerFunction && echo "  ← $1"' _ {} \;
```

As of engine #567: `import io.casehub.engine.flow.FlowWorkerFunction;`

### Why this is non-obvious

The #543 issue description documents the migration target as `io.casehub.api.model`. That was correct at the time #543 shipped but became stale when #567 moved the class again. A developer following the migration guide gets the wrong import.
