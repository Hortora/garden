### Symptom
`NullPointerException` from `Worker` compact constructor when building a Worker
without calling `.capabilityName()` or `.capabilityNames()` on the builder.

### Context
After engine#591 changed the `Worker` record from `List<Capability> capabilities`
to `Set<String> capabilityNames`, any `Worker.builder().name("x").function(fn).build()`
that omits the capability names field throws NPE.

### What was tried (didn't work)
Incremental Maven build (`mvn install`) — the stale `.class` file from the previous
compile satisfies the classpath check, so the NPE only surfaces on `mvn clean install`
or when the consumer repo does a fresh build.

### Root cause
The `Worker` compact constructor calls `Objects.requireNonNull(capabilityNames)`.
The old API had the same constraint on `capabilities`, but incremental builds masked
the breakage — test stubs that constructed Workers without capabilities compiled
against the cached old `.class` and never hit the null check.

### Fix
Add `.capabilityNames(Set.of())` for test stubs that don't need capabilities:
```java
Worker worker = Worker.builder()
    .name("stub")
    .capabilityNames(Set.of())
    .function(fn)
    .build();
```

### Why this is non-obvious
The NPE appears in code that previously compiled and ran without error. The
incremental-build masking means the breakage is invisible until a clean build —
which may happen on CI, not locally.
