### Symptom

A subclass of `CaseContextImpl` (e.g. `MapCaseFile`) loses its extra methods silently after `snapshot()`. Calling subclass methods on the snapshot fails at compile time (wrong static type) or at runtime with `ClassCastException`.

### Context

Occurs in any `CaseContextImpl` subclass that does not override `snapshot()` and whose callers expect to use subclass-specific methods on the returned copy.

### What was tried (didn't work)

Assuming `snapshot()` returns the runtime type of the receiver (standard polymorphic copy behaviour). The test suite passed because `snapshot()` was tested via the `CaseContext` interface — the type loss was only visible on cast.

### Root cause

`CaseContextImpl.snapshot()` hardcodes `new CaseContextImpl(deepCopy(data), version)`. It always returns a `CaseContextImpl`, never `this.getClass()`. Subclasses do not benefit from polymorphism without an explicit override.

### Fix

Override `snapshot()` in the subclass:

```java
@Override
public CaseContext snapshot() {
    final CaseContext base = super.snapshot(); // deep-copy via CaseContextImpl
    return new MySubclass(base.getData());     // wrap in subclass type
}
```

See GE-20260512-5bcc7b for the full technique.

### Why this is non-obvious

Polymorphic `copy()`/`clone()`/`snapshot()` returning the receiver's runtime type is the expected pattern. `CaseContextImpl.snapshot()` hardcodes the return type silently. A test that only asserts `getString()` or other `CaseContext` interface methods will not catch the type loss — only a cast or `instanceof` assertion will.
