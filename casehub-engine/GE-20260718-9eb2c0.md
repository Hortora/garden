### Symptom
A humanTask binding fires correctly (engine logs `Publishing HumanTaskScheduleEvent`), but the WorkItem is never created. The case stays in RUNNING state indefinitely — no terminal transition, no error in the case EventLog, no path to resolution.

### Context
The `rollback-human-escalation` binding in coordinated-change.yaml dispatches a humanTask when rollback has a merge conflict. The binding condition evaluates correctly and the engine dispatches the HumanTaskScheduleEvent. But WorkItemSpiAdapter.create() throws `NoSuchMethodError` (due to casehub-work binary incompatibility — #165). The exception is caught by Vert.x's uncaught exception handler and logged as `Uncaught exception received by Vert.x`, but the engine does NOT transition the case to FAULTED.

### Root cause
Worker execution failures (capability bindings) are caught at the engine level and handled via OutcomePolicy — the engine transitions to FAULTED or retries. But humanTask dispatch goes through a different code path (HumanTaskScheduleEvent → WorkItemSpiAdapter) that runs asynchronously in the Vert.x event loop. Exceptions thrown in this path escape the engine's error handling and are caught only by Vert.x's global uncaught exception handler, which logs but does not transition the case.

### Fix
None known — this is a gap in the engine's humanTask error handling. The HumanTaskScheduleEvent handler should catch exceptions from WorkItem creation and transition the case to FAULTED (or fire a recovery policy). Currently tracked implicitly via devtown#165 (the binary incompatibility that triggers it).

### Why this is non-obvious
The case appears healthy — it's in RUNNING state, the binding condition was met, the dispatch log message is present. The only signal is a Vert.x-level log message at ERROR that doesn't reference the case ID or binding name. A developer looking at the case would see a stuck case with no explanation in the case's own EventLog.
