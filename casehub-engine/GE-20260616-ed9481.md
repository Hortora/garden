### Symptom

An `ActualStateAdapter` implementation needs to check whether a case definition with a given namespace/name/version is registered in the engine. No query method returns `Optional` or `null` for not-found.

### Context

`CaseDefinitionRegistry` (in `io.casehub.engine.common.spi`, module `casehub-engine-common`) has three methods:

- `Uni<CaseMetaModel> registerCaseDefinition(CaseDefinition model)` — idempotent (returns existing if already registered), but **mutating** and **reactive**. Unsuitable for read-path existence checks.
- `CaseDefinition getCaseDefinition(CaseMetaModel definition)` — goes the opposite direction (metadata → definition). Requires a `CaseMetaModel` you don't have.
- `CaseMetaModel getCaseMetaModel(CaseDefinition caseDefinition)` — throws `RuntimeException` if no metadata is found. Cannot be used for safe existence checking.

### Root cause

The SPI was designed for the engine's internal use (register definitions, retrieve them for execution) — not for external consumers asking "does this definition exist?" No existence-query method was ever needed until the desiredstate deployment module required it.

### Fix

Use an "always PRESENT" pattern: the deployment module maintains its own in-memory set of registered case types. When the module provisions a case type, it stores the key. When the actual state adapter checks, it reads from its own internal set rather than querying the engine.

This is a documented simplification — real drift detection (detecting external case definition removal) requires adding a `findByIdentity()` or `exists()` method to `CaseDefinitionRegistry`.

### Why this is non-obvious

The natural assumption is that a registry has a lookup-by-identity method. `getCaseMetaModel()` looks like it should work — its javadoc says "lookup metadata for a case definition" — but it throws on not-found instead of returning null or Optional. You only discover this by reading the `@throws` annotation or hitting the exception at runtime.
