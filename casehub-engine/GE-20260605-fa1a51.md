### Symptom

An `@ObservesAsync PlanItemCompletedEvent` handler receives events for some binding completions but not others. No error, no warning. Bindings completed via worker execution (qhorus DONE, WorkItem completion) fire the event; bindings completed via `caseHub.signal()` (context update that satisfies a binding condition) do not.

### Context

`PlanItemCompletedEvent` is a CDI event in `io.casehub.blackboard.event`, fired via `Event.fireAsync()` from `PlanItemCompletionHandler.completePlanItemByKey()`. The Javadoc says "fires on COMPLETED terminal state" — which is accurate but incomplete. It does not clarify that only the `casehub.worker.finished` Vert.x event path goes through `PlanItemCompletionHandler`.

Context signals (`caseHub.signal()`) take a different path:
```
caseHub.signal() → CaseContext updated → CaseContextChangedEvent (Vert.x EventBus)
  → engine re-evaluates bindings, goals, milestones
  → goals may be satisfied → case may complete
```
This path updates context and evaluates goals but does NOT mark PlanItems as COMPLETED via `PlanItemCompletionHandler` — so `PlanItemCompletedEvent` never fires.

### What was tried (didn't work)

1. **Observing PlanItemCompletedEvent for all binding outcomes** — works for worker completions but silently misses context-signal completions. In a `@QuarkusTest`, calling `caseHub.signal(caseId, "styleCheck", Map.of("outcome", "APPROVED"))` updates the context but the observer never fires.

2. **Waiting longer (increasing Awaitility timeout)** — the event genuinely never fires, not a timing issue.

### Root cause

Two distinct completion paths exist in the engine:
- **Worker path:** `casehub.worker.finished` Vert.x event → `PlanItemCompletionHandler.completePlanItemByKey()` → `PlanItem.markCompleted()` → `PlanItemCompletedEvent` fired
- **Signal path:** `caseHub.signal()` → `CaseContextChangedEvent` → binding/goal re-evaluation — no PlanItem completion step

The Javadoc describes the terminal state accurately but not the trigger path. A developer reading "fires on COMPLETED terminal state" would reasonably expect it to fire for all completions.

### Fix

Fire `PlanItemCompletedEvent` directly via CDI `Event.fireAsync()` in tests to simulate worker completion:

```java
@Inject Event<PlanItemCompletedEvent> planItemEvents;

// First signal the outcome into case context
caseHub.signal(caseId, "styleCheck", Map.of("outcome", "APPROVED"));

// Then fire the event manually (simulating what the worker path does)
planItemEvents.fireAsync(new PlanItemCompletedEvent(caseId, "style-check", "style-agent-1"));
```

For production: ensure all review bindings complete via the worker path (qhorus DONE → engine), not via direct context signals. The observer will then receive all events.

### Why this is non-obvious

The event name and Javadoc suggest comprehensive coverage of PlanItem completions. The two-path architecture is not documented at the CDI event level — you discover it only by tracing the Vert.x event bus routing in `PlanItemCompletionHandler` and `CaseContextChangedEventHandler`.

### Why this fix

Direct CDI event firing in tests is the correct isolation strategy — it tests the observer in isolation from the engine's internal completion routing. In production, the worker completion path is the canonical path for review bindings.

### Alternatives considered

- **Observing CaseContextChangedEvent instead** — fires for all context changes (noisy), is a Vert.x EventBus event (not CDI), and is engine-internal.
- **Observing WorkflowExecutionCompleted** — fires before PlanItem completion, engine-internal, richer data but tighter coupling.
- **Custom ScheduleHandler override** — significant implementation effort, overrides engine defaults.
