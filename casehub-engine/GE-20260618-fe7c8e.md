## The behaviour

`WorkerFunction.AgentExec(Agent)` and `WorkerProvisioner` are NOT interchangeable mechanisms for "running an AI agent on worker execution." They are completely different engine execution paths with different lifecycles, delivery mechanisms, and failure modes:

**WorkerFunction.AgentExec(Agent):**
- Triggered when a binding fires and an INLINE worker with matching capabilities is found in `CaseDefinition.getWorkers()`
- Executed by `DefaultWorkerExecutor.executeSync(agent::execute, ...)` on a virtual thread
- `Agent.execute()` is synchronous — it blocks the virtual thread until the LLM responds
- Result is applied to case context immediately via `WorkflowExecutionCompletedHandler`
- Failure surfaces immediately via exception

**WorkerProvisioner:**
- Called by `CaseContextChangedEventHandler.tryProvision()` ONLY when NO inline worker matches the capability
- Uses `ReactiveWorkerProvisioner.provision()` — async, Vert.x IO thread compatible
- `provision()` registers the external agent (e.g. OpenClaw) but does NOT dispatch work
- Work dispatch happens separately (binding the agent to a Qhorus channel, then sending a COMMAND)
- Result delivery is asynchronous — the external agent delivers results via webhook/channel message
- Binding remains ELIGIBLE until the result arrives — a retry loop fires if provisioning fails (GE-20260616-eb98d8)

**Why this matters for OpenClaw integration:**

Using `WorkerFunction.AgentExec(Agent)` with OpenClaw's OpenAI-compatible endpoint (`/v1/chat/completions`) is NOT the same as "integrating OpenClaw as a WorkerProvisioner." The former makes OpenClaw a dumb LLM server. The latter routes to OpenClaw's skill ecosystem (banking APIs, calendar, Home Assistant). They require separate design decisions and separate implementation paths.

## Why this is non-obvious

Both involve "running an agent." The API surface looks similar. The distinction between "inline AgentExec" and "provisioner-backed external agent" is only visible by reading `CaseContextChangedEventHandler` source code. No documentation distinguishes the two execution paths.
