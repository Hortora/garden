## Worker lambda receives null for context fields added to inputSchema

**ID:** GE-20260417-4a3c22
**Stack:** Quarkus 3.17.x, Mutiny, Quartz
**Symptom:** Test counter keyed by a context field (e.g. `orderId`) added to the worker `inputSchema` always returns `was: <0>` — the field appears null inside the worker lambda even though it exists in the case context.
**Context:** Quarkus-based workflow engine where workers receive their input via `evalObjectTemplate(inputSchema)`. Input is serialized into an `EventLog.payload` at scheduling time, then deserialized in `WorkerExecutionTask` via `OBJECT_MAPPER.convertValue(eventLog.getPayload(), Map.class)`.

### What was tried (didn't work)

- Added `orderId: .orderId` to the worker `inputSchema` jq expression (`{ payment: .payment, orderId: .orderId }`)
- Verified `evalObjectTemplate` path resolution works for `.payment` (other fields arrive correctly)
- Checked that the case context was initialized with the orderId value
- All tests using `runCountByOrderId.getOrDefault(orderId, new AtomicInteger()).get()` returned `was: <0>`

### Root cause

**2026-06-12 — root cause confirmed via decompilation of casehub-engine 0.2-SNAPSHOT:**

`WorkerScheduleEventHandler.onWorkerScheduleEventHandler()` reads the **LIVE** `instance.getCaseContext().asJsonNode()` to extract worker `inputData` — not the `contextSnapshot` from the `CaseContextChangedEvent`. The engine fires the first `CaseContextChangedEvent` **before** the initial context is fully applied to the live `CaseInstance`. The `contextSnapshot` in the event is the correct full context (used to evaluate binding filters), but `WorkerScheduleEventHandler` reads the live context which may still be empty at that moment.

```java
// WorkerScheduleEventHandler line 81 — reads LIVE context, not event snapshot:
Map<String, Object> inputData = this.evalJqAsMap(instance.getCaseContext().asJsonNode(), capability.getInputSchema());
```

This explains why `evalObjectTemplate` resolves fields correctly in isolation but fields arrive null in the worker: the binding filter was evaluated against the full snapshot, but `inputData` was extracted from an empty live context milliseconds later.

### Fix

### Fix 1 — Avoid relying on worker inputData for initial context fields (workaround)

Do not track per-test state through worker input fields. Use DB queries filtered by the case UUID instead — the caseId is unique per test run and cannot be contaminated by other tests:

```java
// Instead of:
assertEquals(1, SomeBean.runCountByOrderId.getOrDefault(orderId, new AtomicInteger()).get());

// Use:
assertEquals(1, findWorkerEvents(caseId, CaseHubEventType.WORKER_EXECUTION_COMPLETED, "payment-worker").size());
```

The `findWorkerEvents` query filters by `caseId` (UUID), which is unique per test invocation.

### Fix 2 — Load entity from DB in the worker function (production pattern)

For production workers that need initial context fields: load the data from the persisted entity using a reliable key (entity ID, which is a simple string/UUID that survives the serialization cycle):

```java
// In the worker function — aeId from context may be null; use WorkerExecutionContext for caseId
final String aeIdStr = (String) context.get("aeId");
if (aeIdStr == null) return noGate();
final AdverseEvent ae = AdverseEvent.findById(UUID.fromString(aeIdStr));
```

The entity was committed before `startCase()` was called, so DB reads always succeed regardless of the engine timing issue.

### Why this is non-obvious

The `evalObjectTemplate` evaluates `.orderId` correctly when tested directly. `CaseContextChangedEvent` carries a correct `contextSnapshot` for filter evaluation, creating the false impression that the same snapshot is used for worker input — it isn't. `WorkerScheduleEventHandler` reads the LIVE context via a separate code path. Only discoverable by decompiling `WorkerScheduleEventHandler`. The symptom (`was: <0>` or all-null inputData) looks like the worker never ran, not like a timing/context split.
