## Symptom

An engine case with a Java-function worker starts and the worker executes successfully (the function lambda is called, returns a map), but:

- The case goal condition (`.advisorRecommendation != null`) never becomes true
- The case never completes — it hangs in RUNNING state forever
- No error is logged
- `Awaitility.await()` for goal completion times out silently

## Context

A `YamlCaseHub` subclass registers a Java-function worker:

```java
def.getWorkers().add(Worker.builder()
    .name("protocol-amendment-advisor-worker")
    .capabilities(List.of(Capability.builder()
        .name("protocol-amendment-advisor")
        .inputSchema("{ amendmentId: .amendmentId, trialId: .trialId }")
        // ← outputSchema MISSING
        .build()))
    .function(ctx -> WorkerResult.of(Map.of("advisorRecommendation", rec.name())))
    .build());
```

The function returns `{ advisorRecommendation: "PROCEED" }` correctly, but the case context is never updated with that value. The goal condition `.advisorRecommendation != null` evaluates to false on every context change tick.

## Root cause

`Capability.builder().outputSchema(".")` is the mechanism by which `DefaultWorkerExecutor.applyOutputSchema()` merges the worker's return map into the live case context. When `outputSchema` is `null` (the default), `applyOutputSchema()` performs no merge — the worker output is discarded.

This is consistent with YAML capability bindings where `outputSchema` is an optional field; when absent, the worker's output is intentionally ignored (useful for fire-and-forget side effects). For Java-function workers that need to write results back to the context to advance the case, `outputSchema(".")` is required.

## Fix

Add `.outputSchema(".")` to the `Capability.builder()` call:

```java
Capability.builder()
    .name("protocol-amendment-advisor")
    .inputSchema("{ amendmentId: .amendmentId, trialId: .trialId, proposedChange: .proposedChange }")
    .outputSchema(".")   // ← merges { advisorRecommendation } back into case context
    .build()
```

`"."` is the JQ identity expression — it passes the entire worker output map through `applyOutputSchema()` and merges it into the case context. The goal condition can then see the newly-written key.

## Reference

`ClinicalSusarOversightCaseHub` (casehub-clinical, Layer 8) demonstrates the complete pattern:

```java
Capability.builder()
    .name("safety-monitoring")
    .inputSchema("{ aeId: .aeId }")
    .outputSchema(".")
    .build()
```

## Why this is non-obvious

The failure is completely silent — no exception, no warning log. The case stays RUNNING with a goal that will never fire. The developer sees the function called and returning a result, with nothing indicating the result is being thrown away. `outputSchema` is an optional field in YAML (the absence is legitimate and documented), so there is no obvious reason a developer would think to add it for the programmatic builder case. The symptom (case hangs) points away from the Capability builder configuration.
