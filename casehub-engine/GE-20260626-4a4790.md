### Symptom

Building `casehub-engine-api` fails with either:
- `does not override abstract method execute(Map) in WorkerFunction` â€” worker-api has `execute()`, engine removed the `@Override`
- `is not abstract and does not override abstract method execute(Map)` â€” worker-api removed `execute()`, engine still overrides it

The error points at `FlowWorkerFunction` or `AgentWorkerFunction` in engine-api, not at the worker-api dependency.

### Context

`casehub-worker-api` is a separate repo from `casehub-engine`. Two engine issues modify the `WorkerFunction` interface:
- **engine#543** (worker-api migration): moves Worker primitives to `casehub-worker-api` but keeps `execute()` on `WorkerFunction`
- **engine#567** (remove serverlessworkflow SDK): makes `WorkerFunction` a bare marker interface (removes `execute()`)

Worker repo commit `33fc532` retains `execute()` (compatible with #543). Commit `72fc1ee` removes it (compatible with #567). The two are on the same `main` branch â€” `72fc1ee` is HEAD.

### Root cause

The two issues ship independently across two repos. `mvn -U` resolves whichever SNAPSHOT was last published or locally installed. If the worker-api jar is from #567 (bare marker) but engine jars are from #543 (expects `execute()`), or vice versa, the interface contract is violated.

### Fix

Always rebuild `casehub-worker` and `casehub-engine` from the same coordinated state:
- For #543 compatibility: `git -C worker checkout 33fc532 && mvn install`
- For #567 compatibility: `git -C worker checkout main && mvn install` (HEAD = `72fc1ee`)

Then rebuild engine from the matching branch.

### Why this is non-obvious

The error message points at engine-api classes (`FlowWorkerFunction`, `AgentWorkerFunction`), not at the worker-api dependency. The natural reaction is to fix the engine code, not to check which worker-api version is in `.m2`. The cross-repo coupling between two separately-tracked issues (#543 and #567) is invisible from either issue's description.

### Why this fix

Rebuilding from a coordinated state is the only reliable fix because the two repos don't declare explicit version ranges â€” both use `0.2-SNAPSHOT`. Version pinning would prevent the issue but isn't practical during active development.

### Alternatives considered

- Checking GitHub Packages for published SNAPSHOTs â€” unreliable because the published version may not match the locally installed engine
- Reading the jar manifest for build metadata â€” manifests don't include the source commit hash
