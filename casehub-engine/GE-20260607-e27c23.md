## Symptom

CDI deployment fails when `SignalReceivedEventHandler` is active but
`DefaultWorkerExecutionRecoveryService` is excluded via `quarkus.arc.exclude-types`.
The error names `WorkerExecutionRecoveryService` as unsatisfied — not `SignalReceivedEventHandler`
— so the connection between the two is non-obvious.

## Context

When building a test profile that needs `SignalReceivedEventHandler` active (e.g. to test
case completion driven by `CaseHubRuntime.signal()`), you cannot simply remove the handler
from the exclude list. `SignalReceivedEventHandler` injects `WorkerExecutionRecoveryService`,
which is only satisfied by `DefaultWorkerExecutionRecoveryService`. Both must be active together.

## Root Cause

`SignalReceivedEventHandler.onSignalReceived()` has a recovery path:

```java
return recoveryService
    .loadOrRestoreCaseInstance(event.caseId())
    .chain(instance -> applySignal(instance, event));
```

If the case instance is not in cache, it delegates to `WorkerExecutionRecoveryService`.
The injection point is eager — CDI validates it at augmentation time regardless of whether
the recovery path is actually hit.

## Fix

In your test profile's `quarkus.arc.exclude-types`, either:

1. **Include both handlers** — remove both `SignalReceivedEventHandler` AND
   `DefaultWorkerExecutionRecoveryService` from the exclude list simultaneously.

2. **Exclude both** — if you don't need signal handling, exclude both.

Never activate one without the other.

## Why This Is Non-Obvious

The class is called `SignalReceivedEventHandler` — the name gives no indication it needs
recovery infrastructure. `DefaultWorkerExecutionRecoveryService` is primarily associated
with the worker execution recovery path, not with signal handling.
