## Behaviour

When a CasePlanModel binding with `humanTask:` target fires, the engine publishes a `HumanTaskScheduleEvent`, which `HumanTaskScheduleHandler` handles by creating a PENDING WorkItem. The case state remains `CaseStatus.RUNNING` throughout — it never transitions to `CaseStatus.WAITING`.

The WAITING state is defined in `CaseStatus` enum but is not used automatically by the engine for humanTask bindings.

## Impact

Integration tests that assert `ci.getState() == CaseStatus.WAITING` after a humanTask binding fires will time out. The correct assertion is to poll for the WorkItem directly and then check for `CaseStatus.COMPLETED` after completing it.

## Correct test pattern

```java
// DON'T assert WAITING — the case stays RUNNING
// DO wait for the WorkItem directly
var workItemRef = new AtomicReference<WorkItem>();
await().atMost(TIMEOUT).pollInterval(POLL_INTERVAL).until(() ->
        QuarkusTransaction.requiringNew().call(() -> {
            WorkItem wi = WorkItem.find("callerRef like ?1 and status = ?2",
                    "case:" + caseId + "%", WorkItemStatus.PENDING).firstResult();
            if (wi != null) { workItemRef.set(wi); return true; }
            return false;
        }));
```

## Why this is non-obvious

The CaseStatus enum has `WAITING` — it exists and looks purpose-built for "case is waiting on external input." A developer writing integration tests naturally assumes the engine uses it. No documentation states otherwise.
