## Behaviour

`CaseInstanceRepository.updateStateAndAppendEvent(CaseInstance instance, EventLog eventLog)` atomically:
1. Persists the updated `CaseInstance` state
2. Appends the `EventLog` entry

Calling `eventLogRepository.append(eventLog)` immediately before `updateStateAndAppendEvent(instance, eventLog)` writes the same EventLog entry **twice** — once standalone and once inside the atomic state update.

```java
// WRONG — double write:
return eventLogRepository.append(startedLog)
    .chain(() -> caseInstanceRepository.updateStateAndAppendEvent(parent, startedLog))
    .replaceWithVoid();

// CORRECT — single atomic write:
return caseInstanceRepository
    .updateStateAndAppendEvent(parent, startedLog)
    .replaceWithVoid();
```

## Discovery

Only discoverable by reading the `InMemoryCaseInstanceRepository` implementation:

```java
public Uni<Void> updateStateAndAppendEvent(CaseInstance instance, EventLog eventLog) {
    rwLock.writeLock().lock();
    try {
        store.put(instance.getUuid(), instance);
        return eventLogRepository.append(eventLog); // ← appends internally
    } finally {
        rwLock.writeLock().unlock();
    }
}
```

The SPI Javadoc says "atomically updates the instance state" but does not state that it also appends the EventLog as part of the same atomic operation.

## Why this matters

With in-memory repos the duplicate is harmless during tests (just an extra entry). With the JPA implementation, `updateStateAndAppendEvent` runs inside a transaction — the duplicate appears as two distinct rows in the `event_log` table. Any query on `findByTypes(SUBCASE_STARTED)` returns both, causing `findFirst()` to return either non-deterministically. This caused a silent correctness bug in `SubCaseCompletionListener` where two `SUBCASE_STARTED` entries existed for the same child case.
