## The behaviour

`ReactiveWorkerProvisioner.terminate(String workerId, String tenancyId)` is defined in the engine SPI (`casehub-engine-api`) but has **no call site anywhere in the engine runtime**. Verified via bytecode analysis of `casehub-engine-runtime` — no class invokes `reactiveWorkerProvisioner.terminate()` or `workerProvisioner.terminate()`.

The SPI method exists. The no-op default implementations exist (`NoOpWorkerProvisioner`, `NoOpReactiveWorkerProvisioner`). But the engine never invokes it — not on case completion, not on case cancellation, not on case faulting.

This means provisioned workers are never cleaned up by the engine. The provisioner implementation must handle its own termination by observing `CaseLifecycleEvent` for terminal states.

## The fix

Observe `CaseLifecycleEvent` (from `io.casehub.engine.common.spi.event`) and call your own cleanup:

```java
@ApplicationScoped
public class ProvisionerCleanupObserver {
    @Inject MyProvisioner provisioner;

    public void onCaseTerminal(@ObservesAsync CaseLifecycleEvent event) {
        if ("CaseCompleted".equals(event.eventType())
                || "CaseFaulted".equals(event.eventType())
                || "CaseCancelled".equals(event.eventType())) {
            provisioner.cleanupForCase(event.caseId());
        }
    }
}
```

`CaseLifecycleEvent` is a public record in `casehub-engine-common`. Terminal event types are "CaseCompleted", "CaseFaulted", "CaseCancelled" (verified against `CaseStatusChangedHandler.resolveEventType()` bytecode).

## Why this is non-obvious

The `terminate()` method exists on the SPI interface, which strongly implies the engine calls it as part of the provisioner lifecycle (provision → work → terminate). The SPI javadoc says "Deregisters/cleans up a provisioned worker" — implying the engine manages this lifecycle. A provisioner implementer would naturally assume the engine handles termination and omit their own cleanup. The result: provisioned resources (heartbeat jobs, registry entries, agent sessions) are never cleaned up.

The no-op implementations further reinforce the assumption — they exist because the engine "should" call them, even though it never does.
