## What Exists

`Worker` has two creation paths with fundamentally different function signatures:

**Constructor (gives full CaseContext access):**
```java
// Function receives the live CaseContext — can read AND write
public Worker(String name, List<Capability> capabilities,
              Function<CaseContext, Map<String, Object>> function)
```

**Builder (receives only input data map — NO CaseContext):**
```java
Worker.builder()
    .name("my-worker")
    .capabilities(cap)
    .function(inputData -> WorkerResult.of(Map.of("output", value)))  // Map<String,Object> input
    .build()
```

The builder's `.function()` takes `Function<Map<String,Object>, WorkerResult>` — the input is the data extracted from the context via the capability's `inputSchema` JQ expression, not the full `CaseContext`.

## Why This Is Non-Obvious

The builder pattern is the idiomatic modern API, so developers naturally reach for it. The `Worker(name, caps, Function<CaseContext, Map>)` constructor is the older form that looks less idiomatic. There is nothing in the API surface or Javadoc that signals the different input types — both accept a function, but one receives context and one receives a map.

The symptom is subtle: with the builder form, the worker receives an empty or partial map (only what the inputSchema JQ extracts) and cannot call `ctx.set()` to write outputs.

## How to Use

When the worker function must **read from AND write to CaseContext** (e.g. plugin-style workers that read game state and write agent decisions), use the constructor:

```java
new Worker(getId(), caps, ctx -> {
    // full CaseContext access — can read and write
    String strategy = ctx.getOrDefault("agent.strategy", "MACRO");
    ctx.set("agent.output", computeOutput(strategy));
    return Map.of();  // empty — output already written to ctx
})
```

When the worker only needs input data and produces a result map (stateless transformation), the builder is appropriate:

```java
Worker.builder()
    .name("transformer")
    .capabilities(cap)
    .function(data -> WorkerResult.of(transform(data)))
    .build()
```
