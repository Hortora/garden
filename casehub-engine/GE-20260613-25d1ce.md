### Symptom

Application starts with no error but the oversight case hub fails at the first `startCase()` call with:

```
IllegalArgumentException: Binding 'susar-assessment' must have capability, subCase, or humanTask
```

Or the YAML parses without errors but the capability binding silently does nothing — the worker never fires.

### Context

Writing a YAML case definition with a worker capability binding. The natural structure (following humanTask's nested pattern) is:

```yaml
bindings:
  - name: susar-assessment
    on:
      contextChange:
        filter: ".aeId != null"
    worker:
      capability: safety-monitoring
      inputSchema: "{ aeId: .aeId }"
```

### Root cause

`io.casehub.model.Binding` is the Jackson deserialization target. Its `@JsonPropertyOrder` is:

```java
@JsonPropertyOrder({"name", "on", "when", "capability", "subCase", "humanTask", "conflictResolverStrategy"})
```

There is no `worker:` field. Jackson is configured with `FAIL_ON_UNKNOWN_PROPERTIES=DISABLED` in `CaseDefinitionYamlMapper`:

```java
ObjectMapper lenient = objectMapper.copy().disable(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES);
```

So `worker: { capability: ... }` is silently dropped. `schemaBinding.getCapability()` returns null → `convertBinding()` throws:

```java
if (schemaBinding.getHumanTask() == null) {
    throw new IllegalArgumentException("Binding '" + name + "' must have capability, subCase, or humanTask");
}
```

### Fix

`capability` must be a **direct String field** on the binding (not nested under `worker:`), referencing a named capability from `spec.capabilities`:

```yaml
spec:
  capabilities:
    - name: safety-monitoring
      inputSchema: "{ aeId: .aeId }"    # JQ applied to case context to produce worker input

  bindings:
    - name: susar-assessment
      on:
        contextChange:
          filter: ".aeId != null and .susarAssessmentComplete == null"
      capability: safety-monitoring      # direct String reference — NOT nested under worker:
```

Additionally: `inputSchema` belongs on the capability definition (evaluated as JQ against the case context), not on the binding. A Java-function worker also requires programmatic registration via `getDefinition()` override — the YAML alone cannot bind a Java function. Use `Worker.builder().function(evaluator).build()` in the `CaseHub` subclass.

### Why this is non-obvious

The `humanTask:` binding uses a nested object (`humanTask: { title: ... }`), which trains the expectation that capability bindings would also be nested under `worker: { capability: ... }`. Jackson's lenient parsing makes the error silent — no parse exception, no warning, just a cryptic runtime failure when the capability system tries to build the binding object.
