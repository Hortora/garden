### Context

A `@ConsumeEvent(blocking = true)` handler subscribes to `casehub.action.gate.rejected` (or approved/expired). Multiple case hub types share the same event bus address. The handler must determine whether this gate event belongs to its case type before writing a ledger entry or signalling the case.

### Problem with the obvious approach

The obvious approach — `CaseInstanceCache.get(event.caseId()).getPendingActionGate().plannedAction().actionType()` — has a non-deterministic race condition: the engine's gate handlers also subscribe to the same addresses and clear `pendingActionGate` synchronously. If the engine handler runs before the application consumer, `getPendingActionGate()` returns null. See GE-20260613-29d3b5.

### Technique

Store the case ID on a domain entity during the case start sequence (Phase 3 of the three-phase pattern). Query that entity by case ID in the `@ConsumeEvent` handler. Non-null = this gate belongs to this consumer.

**Domain entity setup (Phase 3):**

```java
@Transactional
void persistCaseId(UUID entityId, UUID caseId) {
    MyEntity entity = MyEntity.findById(entityId);
    entity.myCaseId = caseId;   // persisted before gate WorkItem is created
}
```

**Gate discriminator in the listener:**

```java
@ConsumeEvent(value = "casehub.action.gate.rejected", blocking = true)
public void onRejected(ActionGateRejectedEvent event) {
    MyEntity entity = MyEntity.findByCaseId(event.caseId());
    if (entity == null) return; // not our gate type

    // entity.id, entity.enrollmentId, entity.grade etc. available here
    // no secondary lookups needed
    ledgerWriter.writeEntry(entity, "REJECTED", Instant.now(), event.rejectedBy());
    caseHub.signal(event.caseId(), "assessmentComplete", true);
}
```

**Why it works:**

- Phase 3 commits `myCaseId` before `startCase()` returns and before the engine creates the gate WorkItem
- By the time the gate event fires and the consumer runs, the DB row exists
- The Panache query is race-free — the DB is the source of truth, not an in-memory cache
- Survives JVM restart (unlike `CaseInstanceCache`, which is in-memory)

**Side benefit:** The entity provides all domain context fields (IDs, grade, status) without secondary cache or context lookups.

### Why this is non-obvious

Developers familiar with the engine naturally reach for `CaseInstanceCache` since it's injected and always returns the case instance. The race condition only manifests in production under load. The DB query approach feels indirect but is structurally correct.
