### Symptom
Outcome recording integration point collects only failure evidence — CBR store
or any feedback-loop consumer never accumulates successful routing decisions.
No error or warning is raised; the data store simply has a 100% failure rate
for all recorded outcomes.

### Context
Adding a listener or integration at `WorkerOutcomeResolvedEvent` to record ALL
routing outcomes for a feedback loop (e.g. CBR-enriched routing in blocks#16).
The goal is to record both successful and failed routing decisions so future
routing can learn from historical outcomes.

### Root cause
Despite the name "Resolved" suggesting all outcomes are covered,
`WorkerOutcomeResolvedEvent` fires ONLY for non-success outcomes. Its Javadoc:
"Published by `WorkflowExecutionCompletedHandler` when a worker returns a
non-success `WorkerOutcome`."

The success path in `onWorkflowExecutionCompletedHandler()` applies output,
records the success outcome in `_outcomes`, and publishes `CaseContextChangedEvent`
— but never publishes `WorkerOutcomeResolvedEvent`. Only the `handleSemanticFailure()`
path (Declined, Failed, Expired) publishes it.

### Fix
Use `WorkflowExecutionCompleted` as the integration point — it fires for ALL
outcomes: `Success`, `Declined`, `Failed`, `Expired`. The handler method
`onWorkflowExecutionCompletedHandler()` receives every completion and forks
internally based on outcome type.

```java
// Correct — fires for all outcomes
@ConsumeEvent(value = EventBusAddresses.WORKER_EXECUTION_FINISHED)
public Uni<Void> onWorkflowExecutionCompletedHandler(WorkflowExecutionCompleted event) {
    // event.outcome() is Success | Declined | Failed | Expired
}

// Wrong — fires only for Declined, Failed, Expired
@ConsumeEvent(value = EventBusAddresses.WORKER_OUTCOME_RESOLVED)
// WorkerOutcomeResolvedEvent — never sees Success outcomes
```

### Why this is non-obvious
The name "WorkerOutcomeResolvedEvent" strongly implies that all worker outcomes
are "resolved" through this event. The actual filtering to non-success-only is
buried in the Javadoc and the handler's control flow. A developer scanning the
event bus addresses would naturally choose `WORKER_OUTCOME_RESOLVED` for
"all outcomes" integration, not `WORKER_EXECUTION_FINISHED`.

### Why this fix
`WorkflowExecutionCompleted` is the canonical entry point for all worker
completions. Rather than patching `WorkerOutcomeResolvedEvent` to also fire
on success (which would change its semantics for existing consumers), the
correct approach is to integrate at the entry point that already sees
everything. This was validated during the blocks#16 adversarial design review.

### Alternatives considered
- **Patching `WorkerOutcomeResolvedEvent` to also fire on success** — rejected
  because it changes semantics for existing consumers (e.g. blackboard's
  PlanItem lifecycle) who depend on it being failure-only.
- **Adding a new `WorkerOutcomeAllEvent`** — rejected as unnecessary; the
  entry-point event already exists.
