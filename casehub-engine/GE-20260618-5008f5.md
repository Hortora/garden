## The technique

When you need to substitute a CDI bean for ALL test classes (not per-method), use a test-only `@Alternative @Priority(10) @ApplicationScoped` CDI bean registered in `quarkus.arc.selected-alternatives` in test `application.properties`. This approach:

1. Does NOT change the CDI profile between test classes (no Quarkus restart)
2. The bean is active for the entire test run — all classes share the same substitution
3. The test bean can be request-aware — inspect inputs to return different responses per test case

```java
// src/test/java/com/example/TestMyChatModelProvider.java
@Alternative
@Priority(10)
@ApplicationScoped
public class TestMyChatModelProvider extends MyChatModelProvider {

    @Override
    public ChatModel get() {
        return new ChatModel() {
            @Override
            public ChatResponse doChat(ChatRequest request) {
                // Examine the request to decide which response to return
                String text = extractUserText(request);
                if (text.contains("error-trigger")) {
                    return errorResponse();
                }
                return successResponse();
            }
        };
    }
}
```

```properties
# test/resources/application.properties
quarkus.arc.selected-alternatives=\
  ...,\
  com.example.TestMyChatModelProvider
```

## When to use

Use this when:
- Multiple test classes need the substitution (not just one)
- The bean's CDI profile change would force a Quarkus restart (e.g. beans used in `@ApplicationScoped` lazy-init paths)
- The substitution behavior needs to be request-aware (different responses for different inputs)

Use `@InjectMock` only when:
- A single test class needs a unique mock behavior
- The bean is NOT involved in any lazy-init path that bakes values at augmentation time
- No other `@QuarkusTest` class runs in the same Quarkus instance with a conflicting profile

## Why this is non-obvious

Most Quarkus documentation recommends `@InjectMock` as the standard test double mechanism. The CDI-restart consequence of mixing `@InjectMock` and non-mocked test classes in the same JVM is not documented. The `@Alternative + selected-alternatives` approach is documented as a production CDI pattern, not as a testing technique.
