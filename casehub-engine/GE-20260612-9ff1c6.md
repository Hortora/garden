## Programmatic worker binding added to existing YamlCaseHub fires on the initial empty-context event

**Symptom:** Worker binding added programmatically in `YamlCaseHub.getDefinition()` override fires immediately at case start. The worker receives all-null `inputData`. `PlanningStrategyLoopControl` marks the plan item RUNNING. Subsequent context events (with the full initial context) cannot re-dispatch the binding — the plan item is permanently RUNNING.

YAML `humanTask` bindings (e.g. safety-review, DSMB) fire correctly on the second event with the full context, because their filters require non-null fields (`.requiresSeniorMonitor == true`), which are false against the empty first event. The programmatic binding with `.susarAssessmentComplete == null` fires vacuously (null == null is true in JQ).

**Context:** `YamlCaseHub.getDefinition()` is overridden to add a worker and a `ContextChangeTrigger` binding programmatically. The case starts with `startCase(initialContext).join()`. The engine fires the first `CaseContextChangedEvent` before the initial context is applied to the live `CaseInstance`. The programmatic binding's filter evaluates to true against the empty snapshot; the worker is dispatched; `inputData` is all null.

### What was tried (didn't work)

- Added `.aeId != null` guard to the binding filter — `.aeId` is also null in the empty snapshot
- Added `.grade != null` guard — `.grade` is also null
- Added `.requiresSeniorMonitor == true` guard — does NOT fire on the empty first event (correct), but also never fires on the second event because by then `susarAssessmentComplete` has already been written to the context by the first (empty) worker run, making the binding filter permanently false
- `@InjectMock` on a policy SPI — discovered separately that this causes the mock to be activated before `@BeforeEach` stubs are applied, making the context empty for all events

### Root cause

The engine fires `CaseContextChangedEvent` before the initial context is fully applied to the live `CaseInstance`. `CaseContextChangedEventHandler` evaluates binding filters against the `contextSnapshot` from the event (which may be the correct full context on the second event), but `WorkerScheduleEventHandler` reads `instance.getCaseContext()` LIVE — which reflects the empty pre-application state at the time it processes the `WorkerScheduleEvent`.

`PlanningStrategyLoopControl.filterToDispatchable()` keeps only bindings whose plan item is `PENDING`. Once the empty-context first event dispatches the worker and marks the plan item `RUNNING`, no subsequent event can re-dispatch it. The full-context event never reaches the worker.

From decompiling `WorkerScheduleEventHandler` (casehub-engine 0.2-SNAPSHOT):
```java
// Line 81 — reads LIVE context, not event snapshot:
Map<String, Object> inputData = this.evalJqAsMap(instance.getCaseContext().asJsonNode(), capability.getInputSchema());
```

And `PlanningStrategyLoopControl.filterToDispatchable()` (casehub-engine-blackboard 0.2-SNAPSHOT):
```java
// Keeps only PENDING plan items — RUNNING items are permanently blocked:
return selected.stream().filter((b) ->
    plan.getPlanItemByBindingName(b.getName())
        .map((pi) -> pi.getStatus() == PlanItemStatus.PENDING)
        .orElse(true))
    .toList();
```

### Fix

**Do not add programmatic worker bindings to existing YamlCaseHub case definitions.** Follow the AML Layer 9 pattern: create a dedicated oversight case hub (separate YAML, separate `YamlCaseHub` subclass) for each worker that needs to fire based on initial context data.

```java
// WRONG — adds worker binding to existing case hub:
@Override
public CaseDefinition getDefinition() {
    CaseDefinition def = super.getDefinition();
    def.getBindings().add(Binding.builder()
        .name("susar-assessment")
        .on(new ContextChangeTrigger(".susarAssessmentComplete == null"))
        .capability(SAFETY_MONITORING).build());
    def.getWorkers().add(Worker.builder()...);
    return def;
}

// CORRECT — AML Layer 9 pattern: dedicated case hub with its own YAML:
@ApplicationScoped
public class ClinicalSusarOversightCaseHub extends YamlCaseHub {
    public ClinicalSusarOversightCaseHub() { super("clinical/susar-oversight.yaml"); }
    @Override public CaseDefinition getDefinition() {
        // workers added here have their own YAML context; no timing conflict
    }
}
```

The dedicated case hub is started by the parent service after evaluating whether oversight is needed, passing the entity data as `initialContext` directly (not relying on the engine to propagate it from a parent case).

**Secondary fix — if a binding must exist in the existing case:** make the worker load entity data from the DB using a stable entity ID (e.g. `aeId`), rather than relying on JQ-extracted context fields. The entity was committed before `startCase()` was called, so DB reads always succeed.

### Why this is non-obvious

YAML `humanTask` bindings work correctly in the same case because their filters require affirmative Boolean fields (`.requiresSeniorMonitor == true`), which are false against the empty snapshot. This masks the empty-context issue entirely for YAML bindings. Programmatic bindings with JQ null-equality guards (`== null`) fire vacuously. The `PlanningStrategyLoopControl` idempotency mechanism (which is what makes YAML bindings work reliably in the full suite) is precisely what permanently blocks the programmatic binding from ever getting the full context.

The fix (AML Layer 9 pattern) is in the AML reference implementation but is not documented anywhere as the canonical approach for new oversight workers. Only discoverable by reading AML source code or decompiling engine internals.
