### What exists

`ChainedReactiveActionRiskClassifier.narrower()` resolves two `GateRequired` decisions by picking the one with the **smaller `candidateGroups` list** as more restrictive. If sizes are equal, the one with the shorter `expiresIn` wins. `null` candidateGroups maps to `Integer.MAX_VALUE` (least restrictive — treated as "anyone can approve").

```java
// From ChainedReactiveActionRiskClassifier (decompiled)
private RiskDecision.GateRequired narrower(GateRequired a, GateRequired b) {
    int sizeA = a.candidateGroups() == null ? Integer.MAX_VALUE : a.candidateGroups().size();
    int sizeB = b.candidateGroups() == null ? Integer.MAX_VALUE : b.candidateGroups().size();
    if (sizeA != sizeB) {
        return sizeA < sizeB ? a : b;   // fewer groups = more restrictive
    } else if (a.expiresIn() != null && b.expiresIn() != null) {
        return a.expiresIn().compareTo(b.expiresIn()) <= 0 ? a : b;
    } else {
        return a.expiresIn() != null ? a : b;
    }
}
```

### Semantics

| candidateGroups | Interpretation | Restrictiveness |
|----------------|----------------|-----------------|
| `["household-admin"]` (1 entry) | Only admin can approve | Most restrictive |
| `["household-admin", "household-member"]` (2 entries) | Either adult can approve | Less restrictive |
| `null` | Anyone can approve (platform decides) | Least restrictive |

### Impact on classifier design

When designing `candidateGroups` for a `GateRequired` decision:
- **Urgent safety / time-sensitive gates:** use more groups (e.g. `["admin", "member"]`) → faster response because more people can approve → treated as less restrictive in the chain → does not override a tighter constraint from another classifier
- **High-consequence gates:** use fewer groups (e.g. `["admin"]`) → narrower approval pool → treated as more restrictive → wins over classifiers that return broader groups

### Discovery path

Only discoverable by decompiling `ChainedReactiveActionRiskClassifier` in the engine runtime jar. Not in any public API docs, the `GateRequired` Javadoc, or the engine#402 issue description.

### Why this is non-obvious

The field name `candidateGroups` suggests an eligibility list — more candidates sounds like more people must consent (stricter). The actual semantics are inverted: more entries in the eligibility list means more people can approve, which is *less* restrictive. Designing a gate with `["admin", "member"]` thinking it is "stronger" because two groups are involved will produce the opposite of the intended effect when chained with a classifier that returns `["admin"]` alone.
