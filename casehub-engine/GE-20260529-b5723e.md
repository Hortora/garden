## Symptom

Quarkus startup fails with 31+ deployment problems after adding `casehub-engine` as a compile dependency to a module that implements its SPIs (WorkerProvisioner, CaseChannelProvider, WorkerStatusListener):

```
Found 31 deployment problems:
[1] Unsatisfied dependency for type io.casehub.engine.common.spi.scheduler.WorkerExecutionManager
[2] Unsatisfied dependency for type io.casehub.engine.common.spi.CaseInstanceRepository
[3] Unsatisfied dependency for type io.casehub.engine.common.spi.EventLogRepository
...
```

None of the error messages mention the dependency choice — they all point at internal engine CDI wiring.

## Root Cause

`casehub-engine` is the full Quarkus runtime module — it ships `@ApplicationScoped` CDI beans that depend on persistence SPIs (`CaseInstanceRepository`, `EventLogRepository`, `JobScheduler`, `WorkerExecutionManager`, etc.). These beans have no implementations in a module that only implements the worker provisioner SPIs.

`casehub-engine-api` is the pure-Java module containing only the SPI interfaces (`WorkerProvisioner`, `CaseChannelProvider`, `WorkerStatusListener`), domain model (`Worker`, `ProvisionContext`, `CaseChannel`), and no CDI beans.

A module implementing the SPIs needs the API — not the runtime.

## Fix

In `pom.xml`, change:
```xml
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-engine</artifactId>
</dependency>
```
to:
```xml
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-engine-api</artifactId>
</dependency>
```

The app module that assembles everything should include `casehub-engine` (compile or runtime scope). The `quarkus.arc.exclude-types` list in the app module still needs to exclude the engine's no-op SPI beans for the SPIs the implementing module replaces — but only the ones it actually replaces (see GE-20260428-9311f8 for the full list and GE-20260529-b5723e for why partial exclusion is the correct pattern).

## Why This Is Non-Obvious

The 31 error messages describe missing implementations of engine-internal SPIs — `CaseInstanceRepository`, `JobScheduler` — that look like they need to be provided. A developer unfamiliar with the engine's internal structure would spend time trying to provide those implementations rather than questioning the artifact choice. The two artifacts have near-identical names and the runtime one was the natural first choice. See GE-20260428-9311f8 for the related no-op bean collision problem that typically surfaces alongside this issue.
