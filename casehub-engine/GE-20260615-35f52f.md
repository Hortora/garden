## Symptom

`@QuarkusTest` integration tests that start cases and drain to `status=completed` time out. Cases start successfully, `CONTEXT_CHANGED` events fire, `CaseContextChangedEventHandler` runs binding evaluation — but no workers are ever selected. No error, no warning. The case stays `RUNNING` indefinitely.

The symptom misdirects toward Quartz job execution, CDI observer blocking, tenancy ID mismatches, or `@DefaultBean` resolution failures. None of these are the cause.

## Context

The panels refactor (`734ed65e`, 2026-06-09) restructured `CaseContextImpl` to wrap all data under panel names. `asJsonNode()` changed from returning flat working data:
```json
{"transaction": {...}, "entityResolution": {...}}
```
to a panel-wrapped document:
```json
{"working": {"transaction": {...}}, "semantic": {...}, "episodic": {...}}
```

Every JQ evaluation point in the engine — binding filters, `when` conditions, goals, milestones, `inputSchema`, `outputSchema` — called `context.asJsonNode()`. JQ expressions like `.transaction != null` now evaluate against the panel document where `transaction` lives under `.working.transaction`. The expression evaluates to `null` → `false` → binding does not fire → no worker selected → case stays RUNNING.

The engine's own tests were updated to use `.working.*` paths as part of the panels refactor, so engine CI stayed green. Consumer apps (AML, clinical, life, devtown) were never updated — their YAML still uses unqualified paths.

### What was tried (didn't work)

1. Hypothesised `fireAsync()` awaiting blocks `CONTEXT_CHANGED` — wrote a test proving blocking observers don't block case completion. Fix was valid for a secondary issue but not the root cause.
2. Hypothesised `QhorusInboundCurrentPrincipal` CDI ambiguity causes tenancy mismatch — verified both principals return the same UUID. Red herring.
3. Hypothesised `WorkerDecisionEventCapture` observer deadlocks under H2 lock contention — plausible but not reproducible in isolation.
4. **Reproduced in AML** (`AmlLayer5ResourceTest`) — test output showed `Rules+milestones+goals processed` with zero `Agent selected` log lines. Bindings evaluated but none matched. Traced to `JQExpressionEngine.evaluate()` calling `context.asJsonNode()` which returns the panel document.

## Root cause

`JQExpressionEngine.evaluate()` and all other JQ evaluation call sites use `context.asJsonNode()` which, post-panels-refactor, returns `{"working":{...}, "semantic":{...}, "episodic":{...}}`. JQ expressions written as `.transaction != null` look for a top-level `transaction` key — which doesn't exist in the panel document. The key is at `.working.transaction`.

## Fix

All JQ evaluation points must evaluate against the working panel specifically:

```java
// Before (broken):
jqEvaluator.eval(expr, context.asJsonNode());

// After (fixed):
jqEvaluator.eval(expr, context.panel(ContextPanel.WORKING).asJsonNode());
```

**8 production files changed:**
- `JQExpressionEngine.evaluate()` — binding filters, when, goals, milestones
- `WorkerScheduleEventHandler` — inputSchema
- `CaseContextChangedEventHandler` — inputSchema, humanTask inputMapping, tryProvision, subCase inputMapping, AgentRoutingContext
- `DefaultWorkOrchestrator` — AgentRoutingContext, inputSchema
- `ListExpressionResolver` — candidateGroups/Users JQ resolution
- `SubCaseCompletionService` — child output mapping

**55 test files updated:** all `.working.field` → `.field` in JQ expressions.

`asJsonNode()` itself is NOT changed — it correctly returns the panel document for serialization, event log storage, and context diffing. The fix is at the evaluation layer, not the data model.

## Why this is non-obvious

The failure is completely silent. Bindings evaluate — `CaseContextChangedEventHandler` logs "Rules+milestones+goals processed" — but no binding condition is true. There is no "expression evaluated to false" warning. The symptom (cases stuck in RUNNING) naturally leads to investigating Quartz, CDI, tenancy, and observer chains — all of which are working correctly. The actual root cause is a JSON document structure change in a method that 8 different call sites use implicitly.

### Why this fix

Evaluating against the working panel (not the full document) is correct because YAML case definitions are about case state — which IS the working panel. The panel structure is an engine implementation detail that YAML authors should not be aware of. Changing `asJsonNode()` itself would break serialization, event log storage, and context diffing, all of which need the full panel document.

### Alternatives considered

- Change `asJsonNode()` to return working panel data — breaks serialization and recovery
- Add a `workingAsJsonNode()` convenience method — adds a new concept without fixing the evaluation contract
- Update all consumer YAML to use `.working.*` paths — makes the panel structure leak into every YAML definition; wrong abstraction level
