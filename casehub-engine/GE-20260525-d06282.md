## Symptom

`@QuarkusTest` passes initially but fails on surefire `rerunFailingTestsCount` retries with:

```
(RECIPIENT_FAILURE,8191) CaseDefinition not found for case: <uuid>
```

The first test run succeeds in starting the case (HTTP 200) but workers never complete within the Awaitility window (10 seconds). The retry starts a new Quarkus instance, re-registers the case definition, but `SchedulerService.registerScheduledTriggers()` immediately fails because `getCaseDefinition(instance.getCaseMetaModel())` returns null.

Confusingly, a standalone diagnostic test injecting `CaseDefinitionRegistry` directly passes — proving the registry IS populated at startup. The failure only surfaces in the retry, under timing pressure.

## Context

`casehub-engine-testing` provides three `@Alternative @Priority(1)` CDI beans:

```java
@Alternative @Priority(1) @ApplicationScoped
public class TestCaseMetaModelRepository extends InMemoryCaseMetaModelRepository {}

@Alternative @Priority(1) @ApplicationScoped
public class TestCaseInstanceRepository extends InMemoryCaseInstanceRepository {}

@Alternative @Priority(1) @ApplicationScoped
public class TestEventLogRepository extends InMemoryEventLogRepository {}
```

`@Alternative @Priority(1)` auto-selects these over the `@Alternative @ApplicationScoped` (no priority) versions in `casehub-engine-persistence-memory`. Without `casehub-engine-testing` indexed, only the no-priority alternatives are discovered — which Quarkus/Arc auto-activates as the sole implementations. The registry uses whichever `CaseMetaModelRepository` bean is active. Between test retries, the Quarkus instance restarts and the entire startup sequence reruns against the same in-memory store — but the timing between registration completion and the HTTP request changes.

## Root cause

`casehub-engine-testing` does not embed a Jandex index (`META-INF/jandex.idx`). Without an explicit `quarkus.index-dependency` entry, Quarkus augmentation does not scan the jar. The three `Test*Repository` beans are invisible to CDI.

The `casehub-engine` runtime jar DOES embed Jandex (it uses jandex-maven-plugin), so engine beans like `DefaultCaseDefinitionRegistry`, `JQExpressionEngine`, etc. are discovered automatically. Only the testing jar is missing the index.

## Fix

Add to `app/src/test/resources/application.properties`:

```properties
# engine-testing provides @Priority(1) auto-selected in-memory impls of all engine SPIs:
# TestCaseMetaModelRepository, TestCaseInstanceRepository, TestEventLogRepository
quarkus.index-dependency.engine-testing.group-id=io.casehub
quarkus.index-dependency.engine-testing.artifact-id=casehub-engine-testing
```

Also required alongside it (following devtown's test properties pattern):

```properties
quarkus.index-dependency.engine-common.group-id=io.casehub
quarkus.index-dependency.engine-common.artifact-id=casehub-engine-common
quarkus.index-dependency.engine-scheduler-quartz.group-id=io.casehub
quarkus.index-dependency.engine-scheduler-quartz.artifact-id=casehub-engine-scheduler-quartz
quarkus.index-dependency.engine-persistence-memory.group-id=io.casehub
quarkus.index-dependency.engine-persistence-memory.artifact-id=casehub-engine-persistence-memory
```

## Why this is non-obvious

The error message "CaseDefinition not found for case: `<uuid>`" comes from `SchedulerService.registerScheduledTriggers()` — a method that is called during `CASE_STARTED` event processing, deep inside the engine runtime. Nothing in the stack trace points at `casehub-engine-testing` or at missing CDI beans.

The first test run succeeds (case starts, workers complete in milliseconds per debug logs), which proves the registry IS populated. Only on retry does the failure surface — because the Quarkus restart changes the timing between `DefaultCaseDefinitionRegistry.onStart()` completing and the HTTP request arriving. The diagnostic test injecting `CaseDefinitionRegistry` directly always passes because it doesn't exercise the async engine startup chain.

The connection between "add an index-dependency" and "fix a CaseDefinition lookup 5 layers deep in the engine" requires knowing the full chain: missing index → wrong `@Alternative` bean active → subtle timing difference on retry → registry appears populated but CaseMetaModel lookup fails due to inconsistent in-memory state between Quartz job threads and the Vert.x event loop.
