### Symptom

Adding `casehub-engine-queue` as a dependency in a consuming project (e.g. `casehub-iot/webapp`) causes a compilation failure:

```
package io.casehub.platform.view does not exist
cannot find symbol: class SubjectViewOrchestrator
```

The error points at the consumer's code, not at the dependency chain.

### Context

`casehub-engine-queue` declares a compile-scope dependency on `casehub-platform-view` without an explicit version — it relies on the engine parent POM's `<dependencyManagement>` to provide the version. When a different project (like `casehub-iot`) consumes `casehub-engine-queue`, the engine parent's dependency management doesn't apply. The consuming project's own dependency management chain (`casehub-iot-parent` → `casehub-parent`) must also declare `casehub-platform-view` — but `casehub-parent` doesn't include it.

### Root cause

`casehub-platform-view` is not in the `casehub-parent` BOM `<dependencyManagement>` section. The engine's own build succeeds because the engine parent manages the version. But transitive resolution across repo boundaries requires the consuming project's BOM chain to also manage it.

### Fix

Add `casehub-platform-view` to the consuming project's parent POM `<dependencyManagement>`:

```xml
<dependency>
    <groupId>io.casehub</groupId>
    <artifactId>casehub-platform-view</artifactId>
    <version>${casehub.version}</version>
</dependency>
```

And add it as a direct dependency in the consuming module's `pom.xml`:

```xml
<dependency>
    <groupId>io.casehub</groupId>
    <artifactId>casehub-platform-view</artifactId>
</dependency>
```

The proper long-term fix is adding `casehub-platform-view` to `casehub-parent` BOM so all consuming projects resolve it transitively without per-project additions.

### Why this is non-obvious

The error message says "package does not exist" — pointing at the consumer's source file, not at the dependency chain. The natural reaction is to check imports, not to investigate why a transitive dependency's version isn't resolving. The engine-queue module compiles fine in its own repo, so the issue only surfaces when consumed cross-repo.
