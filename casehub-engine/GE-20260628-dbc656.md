### Symptom
Integration test deadlocks: `ConditionTimeoutException` waiting for a trust routing attestation (e.g. `sar-drafting` capability tag) that never appears. The investigation starts, all prerequisite workers complete, the target worker is dispatched and executes, but the attestation is never written.

### Context
The `AmlTrustRoutingObserver` writes attestations on `@ObservesAsync WorkerDecisionEvent`. Tests that check for attestations (e.g. polling `attestationRepo.findByInvestigationCaseId()`) before approving the MLRO oversight gate deadlock: the attestation requires the event, the event requires worker completion, and worker completion requires gate approval.

### What was tried (didn't work)
- Increasing Awaitility timeout — the attestation never arrives regardless of wait time
- Checking CDI observer wiring — observer is correctly registered
- Checking attestation repository — entries for other capabilities exist, just not the gated one

### Root cause
`WorkerDecisionEvent` is fired from `WorkflowExecutionCompletedHandler` — at worker **completion** time, not dispatch time. When a worker returns `PlannedAction` (e.g. `SAR_FILING`), the engine creates an oversight gate (WorkItem). The worker is not considered "completed" until the gate is approved. Therefore:

1. Test waits for sar-drafting attestation (written on `WorkerDecisionEvent`)
2. `WorkerDecisionEvent` fires on worker completion
3. Worker completion requires gate approval
4. Gate approval is called AFTER the attestation wait → deadlock

### Fix
Approve the gate BEFORE waiting for the attestation:

```java
// WRONG — deadlocks with PlannedAction workers
Awaitility.await().until(() -> attestationRepo.findByInvestigationCaseId(caseId)
    .stream().anyMatch(a -> "sar-drafting".equals(a.capabilityTag)));
awaitAndApproveGate(caseId);  // too late

// RIGHT — gate approval unblocks worker completion, which fires the event
awaitAndApproveGate(caseId);  // first
Awaitility.await().until(() -> attestationRepo.findByInvestigationCaseId(caseId)
    .stream().anyMatch(a -> "sar-drafting".equals(a.capabilityTag)));
```

### Why this is non-obvious
The event name `WorkerDecisionEvent` strongly suggests a routing **decision** — something that happens at dispatch time when the engine decides which worker to use. The actual firing point (worker completion) is only discoverable by reading `WorkflowExecutionCompletedHandler` source code. Tests that worked before PlannedAction was introduced break silently when a worker is converted to use oversight gates.
