### Symptom

Quarkus startup fails with:

```
jakarta.enterprise.inject.spi.DeploymentException: Found 2 deployment problems:
[1] Unsatisfied dependency for type io.casehub.engine.common.spi.SubCaseGroupRepository
    - injection target: SubCaseExecutionHandler constructor
[2] Unsatisfied dependency for type io.casehub.engine.common.spi.SubCaseGroupRepository
    - injection target: SubCaseCompletionService constructor
```

### Context

When a CaseHub harness adds `casehub-engine-blackboard` (required for SubCase lifecycle and M-of-N quorum) alongside `casehub-engine-persistence-memory`, the blackboard beans (`SubCaseExecutionHandler`, `SubCaseCompletionService`) need `SubCaseGroupRepository`. The in-memory implementation (`MemorySubCaseGroupRepository`) exists in persistence-memory but is `@Alternative @ApplicationScoped` — CDI does not activate `@Alternative` beans unless explicitly selected.

The same applies to `MemoryPlanItemStore` and `MemoryReactivePlanItemStore` from the same module.

### What was tried (didn't work)

- Adding Jandex index entries for persistence-memory — beans are discovered but not activated (they're @Alternative)
- Assuming engine-testing would provide @Priority(1) overrides — engine-testing has TestCaseMetaModelRepository, TestCaseInstanceRepository, TestEventLogRepository but NOT TestSubCaseGroupRepository or TestPlanItemStore

### Root cause

`casehub-engine-persistence-memory` ships all repository implementations as `@Alternative @ApplicationScoped` — they require explicit `quarkus.arc.selected-alternatives` to be activated. Engine-testing provides `@Priority(1)` overrides for CaseInstanceRepository and EventLogRepository but not for SubCaseGroupRepository or PlanItemStore.

### Fix

Add to `application.properties` (test or production):

```properties
quarkus.arc.selected-alternatives=\
  io.casehub.persistence.memory.MemorySubCaseGroupRepository,\
  io.casehub.persistence.memory.MemoryPlanItemStore,\
  io.casehub.persistence.memory.MemoryReactivePlanItemStore
```

Append to existing `selected-alternatives` if other beans (e.g. `JpaLedgerEntryRepository`) are already listed.

### Why this is non-obvious

The error names `SubCaseGroupRepository` — but `MemorySubCaseGroupRepository` IS on the classpath and IS Jandex-indexed. The CDI error doesn't mention that the bean exists but is an unselected `@Alternative`. You have to read the source of `MemorySubCaseGroupRepository` to discover the `@Alternative` annotation, then check whether engine-testing provides an override (it doesn't for these SPIs).

## Variant — Runtime failure from implicit activation of remaining @Alternative beans

### Symptom

`@QuarkusTest` starts cleanly (no DeploymentException) because `casehub-engine-testing` provides `@Priority(1)` overrides for `CaseInstanceRepository` and `EventLogRepository`. But the first `startCase()` call in certain test classes fails with:

```
(RECIPIENT_FAILURE,8191) CaseInstance not found or wrong tenant: <uuid>
```

The error comes from `InMemoryCaseInstanceRepository.update()` (persistence-memory) — the bean is implicitly activated by Quarkus Arc as the sole `@Alternative` without a competing non-alternative, but the activation path produces intermittent initialization-order failures.

### Context

When only `SubCaseGroupRepository` and `PlanItemStore` are in `selected-alternatives`, the remaining 4 `@Alternative` beans from persistence-memory rely on Arc's implicit fallback activation:
- `InMemoryCaseInstanceRepository`
- `InMemoryCaseMetaModelRepository`
- `InMemoryEventLogRepository`
- `MemoryReactivePlanItemStore`

This implicit activation works for most test classes but fails intermittently when the test class is among the first to trigger `startCase()` in a fresh JVM — the `CaseStartedEventHandler` calls `caseInstanceRepository.update()` and the instance isn't found.

### Fix

Add all 4 remaining persistence-memory `@Alternative` beans to `selected-alternatives`:

```properties
quarkus.arc.selected-alternatives=\
  ...,\
  io.casehub.persistence.memory.InMemoryCaseInstanceRepository,\
  io.casehub.persistence.memory.InMemoryCaseMetaModelRepository,\
  io.casehub.persistence.memory.InMemoryEventLogRepository,\
  io.casehub.persistence.memory.MemoryReactivePlanItemStore
```

Verified in casehub-aml#58: 4 of 6 failing test classes fixed by explicit selection. 2 remaining failures (`AmlLayer7ResourceTest`, `AmlTrustRoutingAttestationTest`) may have a deeper root cause — investigation ongoing.
