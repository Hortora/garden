### Symptom

Spec hash drift detection silently misses case definition content changes. Two `CaseDefinition` objects with identical namespace+name+version but completely different capabilities, workers, bindings, milestones, or goals produce the same `hashCode()` and are `equals()`.

### Context

Building a desired-state deployment module that tracks provisioned node specs via hash comparison. `CaseTypeNodeSpec` carried a `CaseDefinition definition` field. The record's auto-generated `hashCode()` delegates to `CaseDefinition.hashCode()` for that field — which only uses three identity fields.

### What was tried (didn't work)

Using `CaseDefinition` directly as a record field on `CaseTypeNodeSpec`, relying on Java record auto-generated `hashCode()` for spec hash drift detection. The assumption was that `CaseDefinition.hashCode()` would include all fields — it does not.

### Root cause

`CaseDefinition` explicitly overrides `hashCode()` and `equals()` to use only `namespace`, `name`, and `version`:

```java
public int hashCode() {
    return Objects.hash(new Object[]{this.namespace, this.name, this.version});
}

public boolean equals(Object o) {
    if (!(o instanceof CaseDefinition that)) return false;
    return Objects.equals(this.namespace, that.namespace)
        && Objects.equals(this.name, that.name)
        && Objects.equals(this.version, that.version);
}
```

This is a deliberate identity-based equality (two definitions with the same identity are "the same definition" regardless of content). But it defeats any system that uses `hashCode()` to detect content changes.

Additionally, `CaseDefinition` is a mutable JavaBean with setters (`setTitle()`, `setSummary()`) and mutable `ArrayList` fields (`getCapabilities().add(...)`). Placing it on an immutable record violates the value-type contract — external code can mutate the definition after record construction.

### Fix

Replace `CaseDefinition definition` with `Map<String, Object> definitionPayload` — the raw parsed YAML content:

```java
public record CaseTypeNodeSpec(
    String namespace, String name, String version,
    String title, String summary,
    String definitionFile,
    Map<String, Object> definitionPayload  // not CaseDefinition
) implements DeploymentNodeSpec { ... }
```

- Immutable via `Collections.unmodifiableMap(new LinkedHashMap<>(payload))`
- Correct recursive `hashCode()`/`equals()` for nested maps and lists
- Decoupled from engine-api's mutable domain class
- The handler constructs `CaseDefinition` from the payload at provision time

Use `Collections.unmodifiableMap()` instead of `Map.copyOf()` because YAML-parsed maps may contain null values.

### Why this is non-obvious

Java record auto-generated `hashCode()` delegates to each field's `hashCode()`. The assumption that `CaseDefinition.hashCode()` includes all fields is natural — you have to read the engine-api bytecode to discover the override. The class has 15+ fields; nothing in the API suggests only 3 are used for equality.

### Why this fix

Using `Map<String, Object>` instead of a typed domain object feels like a loss of type safety. But the alternative — fixing `CaseDefinition.hashCode()` to include all fields — would break every `HashMap<CaseDefinition, ...>` and `Set<CaseDefinition>` in the engine that relies on identity-based equality. The `Map<String, Object>` approach is the right boundary: the spec carries data, the handler interprets it.

### Alternatives considered

- **Fix CaseDefinition.hashCode()** — breaks all identity-based collections in the engine; the override is deliberate
- **Add a contentHash() method to CaseDefinition** — pollutes the domain class with drift-detection concerns
- **Wrap in a ContentHashable<CaseDefinition>** — unnecessary abstraction for a single use case
