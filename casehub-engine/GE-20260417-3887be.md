## Reset shared test counter after blocking startCase() to minimise async contamination

**ID:** GE-20260417-3887be
**Stack:** Quarkus 3.17.x, JUnit 5
**Context:** `@QuarkusTest` with async Quartz workers and a shared `static final AtomicInteger runCount`. `@BeforeEach` resets to 0, but a previous test's worker completing *after* the reset contaminates the counter. Switching to per-case DB queries (see GE-20260417-d67b22) isn't possible in all cases.

### The technique

Place a second `runCount.set(0)` immediately *after* the blocking `startCase()` call, not just in `@BeforeEach`:

```java
@Test
void workerRunsOnceOnDuplicateSignal() {
    UUID caseId = bean.startCase(Map.of("orderId", "order-dedup")).toCompletableFuture().join();

    // Reset AFTER the blocking join — previous tests' async workers have had time to complete
    // and increment runCount. This starts the measurement window cleanly.
    SomeBean.runCount.set(0);

    bean.signal(caseId, "payment", Map.of("amount", 100, "currency", "EUR"));
    bean.signal(caseId, "payment", Map.of("amount", 100, "currency", "EUR"));

    await().atMost(10, SECONDS)
        .untilAsserted(() -> assertEquals(1, SomeBean.runCount.get(), "Dedup must prevent double execution"));
}
```

`startCase().toCompletableFuture().join()` is a blocking call that involves DB writes and application startup logic. By the time it returns, any asynchronous work from the *previous* test has almost certainly completed. Resetting the counter immediately after `join()` — rather than in `@BeforeEach` — closes the contamination window.

### Why this is non-obvious

The `@BeforeEach` placement feels natural and correct. The insight — that a blocking `startCase()` join serves as a natural barrier for previous async worker completion — requires understanding that `join()` blocks long enough for prior async tasks to settle. This depends on the relative timing of test setup vs. worker execution, which isn't obvious from the API.

### Caveat

This is a mitigation, not a guaranteed fix. Use per-case DB queries (GE-20260417-d67b22) when possible — those are immune to contamination by design.
