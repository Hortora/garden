## Symptom

An engine integration test passes when run alone but fails when run alongside other test methods in the same `@QuarkusTest` class. The Awaitility poll times out with a generic "condition not met in 15 seconds" message. No stack trace pointing to the root cause.

## Context

Multiple test methods in the same class each start a case that progresses to a humanTask binding, creating a PENDING WorkItem. The golden path test queries `WorkItem.find("status", WorkItemStatus.PENDING).firstResult()` and completes the first PENDING WorkItem it finds. If another test's case created a WorkItem first, the golden path test completes the WRONG WorkItem. Its own case never gets its WorkItem completed, so the case never reaches COMPLETED.

## Root cause

`WorkItem.find("status", PENDING)` is a global query — it returns any PENDING WorkItem in the database, not just the one for the current test's case. In `@QuarkusTest`, all test methods share the same Quarkus instance and database. WorkItems from other tests are visible.

## Fix

Filter by `callerRef` prefix. The engine's `HumanTaskScheduleHandler` sets `callerRef` to `case:{caseId}/pi:{planItemId}`. Filter with:

```java
String callerRefPrefix = "case:" + caseId;
WorkItem wi = WorkItem.find("callerRef like ?1 and status = ?2",
        callerRefPrefix + "%", WorkItemStatus.PENDING).firstResult();
```

## Why this is non-obvious

The test passes alone because there's only one PENDING WorkItem. The failure mode (Awaitility timeout) gives no hint about cross-test interference. The developer investigates engine state transitions, binding conditions, and async event processing — none of which are the problem.
