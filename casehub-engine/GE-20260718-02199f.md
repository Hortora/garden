### Symptom
CoordinatedChangeObserver receives `CaseLifecycleEvent` with `caseStatus=COMPLETED` for a sub-case that hit a failure goal (`review-abandoned`). Observer treats it as success — signals `completedReviews` instead of `reviewFaulted`. The coordinator proceeds to merge a closed PR.

### Context
The engine's internal event chain already carries goal metadata:
- `CaseStatusChanged` record has `satisfiedGoalName` and `satisfiedGoalKind` fields
- `CaseStatusChangedHandler` writes them to EventLog metadata (line 111-114)
- `CaseOutcomeObserver` receives them via `fireOutcomeObservers()` (line 160)

But `CaseLifecycleEvent` (the CDI event fired via `Event.fireAsync()` for external observers) did NOT include these fields. The factory method `CaseLifecycleEvent.of(CaseInstance, ...)` accepted 6 args — none for goal metadata.

### Root cause
`CaseStatusChangedHandler.onCaseStatusChangedHandler()` calls `lifecycleEvents.fireAsync(CaseLifecycleEvent.of(caseInstance, commandType, eventType, null, "System", traceId))` — the 6-arg factory that predates goal-aware completions. Goal metadata was added to `CaseStatusChanged` and `CaseOutcomeObserver` but the CDI lifecycle event was never updated.

### Fix
Add `satisfiedGoalName` and `satisfiedGoalKind` as the last two fields of the `CaseLifecycleEvent` record. Add an 8-arg `of()` factory. Update `CaseStatusChangedHandler` to propagate `event.satisfiedGoalName()` and `event.satisfiedGoalKind()` through the new factory.

### Why this is non-obvious
The data existed at every internal checkpoint — EventLog metadata, CaseOutcomeObserver — but was silently dropped at the one boundary an application-level observer uses. No error, no warning. `COMPLETED` looked like success because, until failure goals existed, it always was.
