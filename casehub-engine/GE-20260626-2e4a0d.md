### Symptom

`mvn install -Dmaven.test.skip=true` on engine main fails:
```
AgentWorkerFunction is not abstract and does not override abstract method
execute(Map) in WorkerFunction
```

Running `mvn -U` to pull fresh SNAPSHOTs from GitHub Packages does not help — the published SNAPSHOTs are also from the broken state.

### Context

Engine main HEAD (`37b2eea8 refactor(#543): migrate Worker primitives to casehub-worker-api`) removed `Worker`, `Capability`, etc. from engine-api and added a dependency on `casehub-worker-api`. But `casehub-worker-api` main (`72fc1ee`) has `WorkerFunction` as a bare marker interface (no `execute()` method) — driven by engine#567, not #543. The engine-api classes (`FlowWorkerFunction`, `AgentWorkerFunction`) still override `execute()`, which no longer exists.

### Root cause

Engine main incorporated the #543 migration commits but the `casehub-worker-api` repo had already shipped the #567 change (removing `execute()` from `WorkerFunction`). The two repos are out of sync — engine main expects the #543-era worker-api, but the published worker-api is the #567-era version.

### Fix

Build engine from the `issue-570` branch (which includes both #543 and #567 changes) instead of main:
```bash
git -C engine checkout issue-570-expression-engine-output-schema
mvn -f engine/pom.xml install -Dmaven.test.skip=true
```

Or build from pre-#543 (`c04c7135`) if you need the old API:
```bash
git -C engine checkout c04c7135
mvn -f engine/pom.xml install -Dmaven.test.skip=true
```

### Why this is non-obvious

Engine main is expected to compile. `git log` shows clean commit messages. The issue (#543) is marked CLOSED. Nothing signals that main is broken until you try to build it — and the error points at engine-api code, not at the cross-repo dependency mismatch.
