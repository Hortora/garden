### Symptom
`git push upstream main` appears to succeed (shows old..new SHA range, exit 0),
but commits from the current branch never reach the remote. The remote advances to
wherever your local `main` is — not where your current branch is.

### Context
Fork-model multi-repo workflow: on branch `issue-60`, ran
`git push upstream main` expecting current branch changes to land on remote main.
Git pushed local `main` to upstream instead. No warning, no error.

### Root cause
`git push <remote> <refspec>` means "push the local ref named `<refspec>` to the
remote ref named `<refspec>`". When you are on a different branch, the named local
ref (`main`) is whatever your local main points to — not your current branch.
Git does exactly what you asked; the mental model mismatch is the bug.

### Fix
Two options:
1. Checkout main first: `git checkout main && git push upstream main`
2. Push current branch to remote main explicitly: `git push upstream HEAD:main`

### Detection
The push output SHA range reveals it: `393ca03..d500aa3 main -> main` shows
where local `main` was, not where your current branch is. Compare against
`git log --oneline -1` on your current branch to spot the discrepancy.

### Why this is non-obvious
`git push upstream main` reads as "push to upstream's main" — which is what you
want. The gotcha is that git interprets the second `main` as the LOCAL ref to push
FROM, not the destination only. Most developers only notice when they check the
remote and find their current branch's commits missing.
