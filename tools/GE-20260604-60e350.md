### Symptom

Asking an LLM to apply `Brief` or `Reference` mode to a README causes per-capability explanations and contextual sections to disappear entirely — not compact, but gone. The module structure tables survive; the reasoning for why each capability exists does not.

### Context

When using the `write-content` skill with `[B] Brief` routing for a project README, the mode constraint "no prose paragraphs where tables work" is applied globally across all sections. Sections that cannot be expressed as table entries (e.g., "why SPLADE matters for regulatory text", "what the NLI classifier does in the fact space pipeline") are stripped rather than compacted.

### What was tried (didn't work)

Applying `Brief / Reference` mode globally — correct for module tables and status sections, incorrect for per-capability explanation sections.

### Root cause

A README is a multi-mode document. Different sections serve different purposes and should be written under different mode constraints:
- Lead + status + module tables → `Reference / inventory`
- Per-capability explanations → `Explanation / discursive`

Applying a single mode globally causes the wrong mode to be used for explanation sections. The mode constraint "no prose where tables work" is correct for inventory sections but incorrectly strips explanations that have no table equivalent — the information simply disappears.

### Fix

Classify a README as a multi-mode document, not as `Brief`. Process section by section, switching mode as you go. The `write-content` skill now has a dedicated `[R] README` form with an explicit section-to-mode map.

Test before compressing any section: "Could a developer who hasn't been in the design discussions understand *why* this matters from the table entry alone?" If no — the prose is load-bearing and must stay, tightened not removed.

### Why this is non-obvious

The `Brief / Reference` mode constraint is correct and well-stated. The mistake is assuming it applies uniformly to all sections of a README. READMEs look like a single document but behave as several different document types concatenated — the mode that is right for a module inventory table is wrong for a capability explanation.

### Constraints

Applies to any multi-mode document (README, ARC42STORIES.MD) where different sections serve different rhetorical purposes. Relevant any time an LLM is instructed to apply a single content mode to a heterogeneous document.

### Invalidation triggers

Revisit if the `write-content` skill gains per-section mode switching at a framework level. The README form added 2026-06-04 addresses this.
