### Context

Fixing a logic bug in JavaScript that runs inside a browser renderer (e.g. Electron or a web app served by a backend). The fix requires multiple iterations. Each full test cycle involves launching the app (JVM warmup, Electron startup) — 2–3 minutes per attempt.

### Technique

1. Add a diagnostic `window.evaluate` to the existing Playwright test that prints the exact intermediate values the algorithm uses (token rawLines, chunk boundaries, etc.)
2. Run the full test once to get the diagnostic output
3. Extract those values into a self-contained `/tmp/verify-fix.js` script that re-implements only the broken algorithm using the real data
4. Iterate on the fix entirely in Node.js — each iteration takes 1–2 seconds instead of 2–3 minutes

```js
// /tmp/verify-fix.js — standalone reproduction using diagnostic output
'use strict';

// From diagnostic: window.evaluate(() => marked.lexer(content).map(t => ({type:t.type, rawLines: ...})))
const tokens = [
  { type: 'heading',   rawLines: 2 },
  { type: 'paragraph', rawLines: 0 },  // ← the surprising value
  { type: 'space',     rawLines: 2 },
  // ...
];

// Re-implement the algorithm with OLD logic
function annotateOld(tokens, chunks) { /* ... */ }

// Re-implement with NEW logic (the proposed fix)
function annotateNew(tokens, chunks) { /* ... */ }

// Assert
const results = annotateNew(tokens, chunks);
console.assert(results.includes('p(evaluates)'), 'paragraph must be tagged');
console.log('✅ fix verified');
```

Run with `node /tmp/verify-fix.js` — instant feedback.

### Why this is non-obvious

The natural response to a browser-side JS bug is to add `console.log` in the renderer and run the full test suite to see it. The diagnostic-then-extract approach decouples the "reproduce the state" step (one full run) from the "iterate on the fix" step (many fast runs), collapsing iteration time by 100x.

### Constraints

Requires that the algorithm is pure or can be extracted without DOM dependencies. Works best when the bug involves data transformation (token processing, diff computation, range mapping) rather than DOM manipulation side-effects.

### Alternatives considered

- **Add console.log and run tests repeatedly**: works but each iteration is 2–3 minutes
- **Unit-test the renderer code directly**: ideal but requires restructuring the code to be importable from Node.js (not always feasible for inline `<script>` blocks)
- **Playwright `page.evaluate` with the fix applied**: still requires a full app launch per test

### Invalidation triggers

Not tied to a library version — this is a general debugging workflow.
