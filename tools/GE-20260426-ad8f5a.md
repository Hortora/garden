### Symptom

The producer project's CI is green — `mvn deploy` runs successfully and specific sub-modules publish their JARs to GitHub Packages. But a downstream consumer project fails with:

```
[ERROR] Failed to execute goal on project consumer-module:
Could not collect dependencies for project com.example:consumer:jar:1.0-SNAPSHOT
[ERROR] Failed to read artifact descriptor for io.casehub:api:jar:0.2-SNAPSHOT
[ERROR]   Caused by: The following artifacts could not be resolved:
    io.casehub:parent:pom:0.2-SNAPSHOT (absent):
    Could not find artifact io.casehub:parent:pom:0.2-SNAPSHOT
    in github (https://maven.pkg.github.com/casehubio/*)
```

### Context

Multi-module Maven project with this `pom.xml` structure:

```
parent/ (root, io.casehub:parent:pom)
  ├── api/          (io.casehub:api:jar)  — maven.deploy.skip=false
  ├── engine/       (io.casehub:engine:jar) — maven.deploy.skip=false
  └── codegen/      (io.casehub:codegen:jar) — inherits skip=true
```

Root `pom.xml` has `<maven.deploy.skip>true</maven.deploy.skip>` as the default. The intent is to prevent the root aggregator POM from being published. Sub-modules that should publish override to `false`.

The producer CI is green. The consumer CI resolves `io.casehub:api` but fails trying to read its effective POM.

### What was tried

- Verified `io.casehub:api:jar:0.2-SNAPSHOT` was present in GitHub Packages — it was
- Verified consumer had correct `<repositories>` configuration — it did
- Verified GITHUB_TOKEN / PAT had `read:packages` scope — it did
- Added `-U` to force snapshot update — still failed

### Root cause

When Maven resolves `io.casehub:api:jar:0.2-SNAPSHOT`, it reads the `api` module's own POM (`api-0.2-SNAPSHOT.pom`) from the registry. That POM declares `io.casehub:parent:pom:0.2-SNAPSHOT` as its `<parent>`. Maven then needs to download the parent POM to construct the effective POM (inheriting properties, dependency management, plugin config, etc.).

The root parent POM (`io.casehub:parent`) was never deployed because of `maven.deploy.skip=true` on the root module. It only exists in the producer's local `.m2` cache — not in GitHub Packages. Any other machine trying to resolve the artifact hits a 404 on the parent POM and fails.

### Fix

Change the root `pom.xml` default from `true` to `false`:

```xml
<!-- Root pom.xml properties -->
<!-- The root POM is the parent of all published modules — it MUST be deployed
     so downstream resolvers can construct the effective POM of api, engine, etc. -->
<maven.deploy.skip>false</maven.deploy.skip>
```

Sub-modules that genuinely should not be published (codegen, test utilities, etc.) add their own override:

```xml
<!-- In codegen/pom.xml properties -->
<maven.deploy.skip>true</maven.deploy.skip>
```

Alternatively: deploy only the root POM explicitly by configuring the deploy plugin in the root's `<build><plugins>` (not `<pluginManagement>`) with `<skip>false</skip>` using `<inherited>false</inherited>` — but changing the root default is simpler.

### Why this is non-obvious

The producer CI succeeds — deploying the root POM is skipped silently, and all sub-module deployments succeed. Nothing in the producer's build indicates a problem.

The failure only appears in consumer projects on a different machine. The error message names `io.casehub:parent` (the root POM) as the missing artifact — but developers focus on `io.casehub:api` (the artifact they actually declared as a dependency). The parent POM's absence is an indirect consequence of a deploy-skip configuration that appeared correct.

The distinction between "aggregator POM" (does not need deploying) and "parent POM" (must be deployed for consumers to resolve child artifacts) is subtle. A project can be both simultaneously.
