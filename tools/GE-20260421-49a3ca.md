## Symptom

`tmux display-message -p "#{pane_activity}" -t <session>` returns an empty string for detached sessions. Any code that treats a blank result as "session not found" will incorrectly treat all detached sessions as expired/gone.

Also: `tmux display-message -p <format> -t <target>` (flag order `-p` then `-t`) fails with "too many arguments (need at most 1)" in tmux 3.6a. The `-t` flag must come before `-p`.

## Context

Occurs whenever the tmux session has no attached client (the common case for server-side session management). `#{pane_activity}` only has a value when a client is attached. `#{window_activity}` is populated regardless.

The argument order issue is silent on some tmux versions but fatal on 3.6a.

## What was tried (didn't work)

- `tmux display-message -p "#{pane_activity}" -t name` → blank string
- `tmux display-message -p "#{pane_activity} -t name"` → parse error
- `tmux display-message -p <fmt> -t <target>` → "too many arguments (need at most 1)"

## Root cause

`#{pane_activity}` is a client-side format variable — it tracks when the pane was last visible to an attached client, not when output was last produced. `#{window_activity}` is server-side and reflects actual pane output activity.

The `-t` / `-p` ordering constraint appears to be a parsing quirk in tmux 3.6a's argument handling that is not documented in the man page.

## Fix

Use `#{window_activity}` instead of `#{pane_activity}` for headless activity checks:

```bash
tmux display-message -t <session> -p "#{window_activity}"
```

Returns a Unix epoch timestamp (seconds) as a string. Parse with `Long.parseLong(raw.trim())`.

Argument order: always `-t <target>` before `-p <format>`.

## Why this is non-obvious

The tmux man page documents both formats without distinguishing client-side vs server-side scope. The blank result (not an error) makes it appear the session doesn't exist rather than that the format variable is unavailable. The argument order constraint is undocumented and version-specific.
