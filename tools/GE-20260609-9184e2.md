### Symptom
A conflict check comparing V-numbers across branches reports a conflict at V1 even though only one branch added V1 — the other branch just inherited it from main.

### Context
Scanning git branches for Flyway migration V-number conflicts using `git ls-tree -r --name-only <branch>`. A feature branch created after main had V1 committed inherits V1 in its tree — so `ls-tree` shows V1 on both main and the feature branch, producing a false conflict.

### What was tried
`git ls-tree -r --name-only <branch>` is the obvious tool for "show me all files in this branch". It works correctly for that purpose — but "all files in branch" includes everything inherited from the base, not just what the branch added.

### Root cause
`git ls-tree` shows the complete file tree at a given commit — it does not distinguish files added on this branch from files inherited from the branch point. A branch is just a pointer to a commit; its tree includes all ancestor commits.

### Fix
Use `git log --name-only --diff-filter=A <base>..<branch>` to find only files ADDED uniquely on the branch vs the base:

```python
# Wrong — shows ALL files including inherited:
rc, file_list = run_git("ls-tree", "-r", "--name-only", branch, cwd=project)

# Correct — shows only files ADDED on this branch since base:
rc, file_list = run_git(
    "log", "--name-only", "--diff-filter=A",
    "--format=", f"{base_branch}..{branch}",
    cwd=project
)
```

The `--format=` (empty format) suppresses commit headers, leaving only filenames. Filter blank lines from the output.

### Why this is non-obvious
`git ls-tree` reads like "list this branch's files" — which it does, correctly. The gotcha is that "this branch's files" means the full tree at HEAD, not the delta introduced by the branch. The distinction only matters when you need branch-unique content rather than the full snapshot.
