## Symptom

A GitHub Actions job with multiple `continue-on-error: true` steps reports a green (success) result in the UI even when one or more steps have actually failed.

## Context

Occurs in full-stack or multi-module build workflows where `continue-on-error: true` is used to ensure all modules build and report their status rather than stopping at the first failure.

## Root Cause

`continue-on-error: true` on a step means the job continues when that step fails, but crucially: **the job's final conclusion is determined by the job-level outcome, not individual step outcomes**. If no step after the failed one causes the job to exit non-zero, the job reports success.

The step's `outcome` field is still `failure` (accessible as `${{ steps.X.outcome }}`), but this is only useful if something reads it and acts on it.

## Fix

In the summary/reporting step (which must use `if: always()`), count failed step outcomes and exit with code 1:

```yaml
- name: Build summary
  if: always()
  env:
    OUTCOME_A: ${{ steps.moduleA.outcome }}
    OUTCOME_B: ${{ steps.moduleB.outcome }}
  run: |
    FAIL_COUNT=0
    for outcome in "$OUTCOME_A" "$OUTCOME_B"; do
      case "$outcome" in
        failure|skipped) FAIL_COUNT=$((FAIL_COUNT + 1)) ;;
      esac
    done
    if [ "$FAIL_COUNT" -gt 0 ]; then
      echo "::error::$FAIL_COUNT module(s) failed."
      exit 1
    fi
```

Note: `skipped` must also count as failure — if a step is skipped because a prior step failed (without `continue-on-error`), the skipped outcome masks the failure.

## Why This Is Non-Obvious

The name `continue-on-error` implies the error is still tracked. Developers expect the job to reflect the worst outcome of its steps. In practice, `continue-on-error` delegates failure tracking entirely to the workflow author — there is no automatic roll-up.
