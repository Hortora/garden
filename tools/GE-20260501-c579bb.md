## Technique

When repos have a dependency graph (A depends on B depends on C), wire their CI pipelines so that a successful publish in an upstream repo automatically triggers a rebuild in each downstream repo.

**Upstream repo publish workflow** — add a trigger step after successful publish:

```yaml
- name: Trigger downstream CI
  if: github.event_name != 'pull_request' && success()
  run: |
    for repo in work qhorus; do
      gh api repos/casehubio/$repo/dispatches \
        -f event_type=upstream-published \
        -f client_payload[source]="${GITHUB_REPOSITORY}" \
        2>/dev/null && echo "  ✅ $repo triggered" || echo "  ⚠️  $repo failed"
    done
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Downstream repo publish workflow** — add `repository_dispatch` to the `on:` triggers:

```yaml
on:
  repository_dispatch:
    types: [upstream-published]
  push:
    branches: [main]
  workflow_dispatch:
```

## Why this is non-obvious

The natural approach is either: (a) a monorepo, (b) a single orchestrator workflow that builds everything in sequence, or (c) manual re-runs. `repository_dispatch` creates a lightweight event-driven chain that respects the dependency graph without a central orchestrator or a monorepo.

The chain self-heals: if upstream A changes and publishes, it automatically kicks off B and C in order. If only C changes, only C's own push trigger fires — no unnecessary rebuilds upstream.

## Dependency graph example

```
parent → ledger, connectors
ledger → work, qhorus
connectors → work
work → engine
qhorus → engine, claudony
engine → claudony
```

Each arrow = one `repository_dispatch` trigger step in the upstream's publish workflow, one `repository_dispatch: types: [upstream-published]` listener in the downstream's `on:` block.

## Constraints

- Requires `GITHUB_TOKEN` to have permission to dispatch to other repos in the same org (it does by default within the same org)
- The `repository_dispatch` event always runs on the default branch — not on feature branches
- Does not carry branch context — downstream always rebuilds from main
