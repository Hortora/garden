## Symptom

A bash script uses `grep -c` piped with `|| echo 0` to get a line count that defaults to zero when there are no matches. A subsequent non-empty check (`[ "$VAR" -gt 0 ]` or similar) fires incorrectly on a clean working tree.

```bash
DIRTY=$(git status --short | grep -c . || echo 0)
# Intended: DIRTY=0 when clean, DIRTY=N when dirty
# Actual on clean repo: DIRTY="0\n0" (two lines)
```

## Root cause

`grep -c` exits with code 1 when there are no matches, but it still outputs `0` to stdout before exiting. The `|| echo 0` is meant to handle the no-match case — but since `grep -c` already wrote `0` to stdout, the pipeline now has two items: the `0` from `grep -c` and the `0` from `echo`. The variable receives `"0\n0"` — a two-line string that is non-empty and therefore truthy for any `[ -n "$VAR" ]` or `[ "$VAR" -gt 0 ]` check.

## Fix

Use `wc -l` instead of `grep -c`:

```bash
DIRTY=$(git status --short | wc -l | tr -d ' ')
# Returns 0 (single line, no ambiguity) on clean repo
# Returns N on dirty repo
```

`wc -l` always exits 0, always outputs a single integer. No `||` needed.

## Why this is non-obvious

`grep -c` looks like a natural word-count substitute — it counts matching lines and its exit code signals "found" vs "not found". The documentation never mentions that combining it with `|| echo 0` produces double output. The symptom (stash prompt fires on a clean repo) doesn't point at the grep line — it looks like a logic error in the downstream condition.
