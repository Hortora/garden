### Context

`mcp__intellij-index__ide_move_file` is used to relocate a Java source file and
update all references across the project. For a Java interface moved to a new
package, the expectation is that all dependent files get their imports updated.

### Symptom

After `ide_move_file` completes and reports success, some dependent files have
their field type or method return type updated to the fully-qualified class name
(FQN) rather than a simple name with a new import:

```java
// Expected after move:
import io.casehub.ledger.api.spi.ActorIdentityProvider;
// ...
ActorIdentityProvider actorIdentityProvider;

// Actual in some files:
io.casehub.ledger.api.spi.ActorIdentityProvider actorIdentityProvider;
```

The code compiles correctly (FQN is valid Java), but the style is inconsistent
and the FQN usage is flagged by IntelliJ inspections.

### Root Cause

IntelliJ's MCP file-move refactoring updates references but may choose FQN over
import-plus-simple-name depending on the file's existing import state and the
refactoring heuristic used. Files that previously used the class via an in-package
import (no explicit import needed) are the most likely to receive FQN treatment
after the class moves to a different package.

### Fix

After every `ide_move_file` call, compile the affected module and look for FQN
usages in the modified files:

```bash
mvn compile -pl <module> -q
# then inspect the reported "modified files" from ide_move_file result
```

For each file using FQN: add the correct import and replace the FQN with the
simple name. This is a quick mechanical cleanup, but it must be done before
committing — reviewers will flag FQN usages in non-Javadoc code.

### Why this is non-obvious

The `ide_move_file` result says "success" and lists the affected files — nothing
signals that some files were updated with FQN rather than imports. The code
compiles without errors. The issue is only visible when inspecting the diff.
