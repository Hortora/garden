## Symptom

`mvn test -pl engine` fails with `cannot find symbol: class NewClass` even though
`NewClass` exists in a sibling `api/` module and the project compiles fine from root.

## Context

Multi-module Maven project where `engine` depends on `api`. New classes were added to
`api/` in the current working tree but `mvn install` has not been run since. The `-pl`
flag builds only the specified module and resolves dependencies from the local Maven
repository (`~/.m2/repository`), which still has the old version of the `api` jar.

## What was tried

Running `mvn test -pl engine` directly after adding new classes to `api/`.

## Root cause

`-pl engine` does not recompile sibling modules. It uses whatever jar is installed in
the local Maven repo. If `api/` has been modified since the last `mvn install`, the
local repo jar is stale and the new classes are invisible to the engine module.

## Fix

Run `mvn install -DskipTests` from the project root first to update all local repo jars,
then run the module-specific test:

```bash
mvn install -DskipTests -q   # updates local repo jars for all modules
mvn test -pl engine          # now resolves updated api/ jar
```

## Why non-obvious

The error message (`cannot find symbol`) looks like a compilation error in `engine/`,
not a dependency resolution problem. The fix — running install on a different module —
is non-obvious. Developers familiar with Maven often assume `-pl` is safe to use after
any local edit.
