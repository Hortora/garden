## GitHub Pages `configure-pages` action fails if Pages not enabled via API first

**ID:** GE-20260413-83dedd
**Stack:** GitHub Pages, GitHub Actions (`actions/configure-pages@v5`), `gh` CLI
**Symptom:** The `Deploy site to GitHub Pages` workflow fails at the "Setup Pages" step with: `Get Pages site failed. Please verify that the repository has Pages enabled and configured to build using GitHub Actions. Error: Not Found`.
**Context:** Enabling GitHub Pages for a newly-public repository via the `gh` API and immediately triggering the Actions deployment workflow.

### What was tried (didn't work)
- Triggered the workflow immediately after making the repo public — fails
- Used `gh api repos/{owner}/{repo}/pages -X POST -f source='{"branch":"main","path":"/docs"}'` — fails with "Your current plan does not support GitHub Pages for this repository" (private repo)

### Root cause
The `configure-pages` action reads the Pages configuration from the GitHub API before doing anything. If Pages has not yet been enabled with `build_type=workflow`, the API returns 404 and the action aborts. The action does not enable Pages itself.

Two-step sequence required:
1. Enable Pages via API: `gh api repos/{owner}/{repo}/pages -X POST --field 'build_type=workflow'`
2. Only then trigger the workflow (push to main or `gh workflow run`)

### Fix
```bash
# Step 1: Enable Pages with Actions build type
gh api repos/OWNER/REPO/pages -X POST --field 'build_type=workflow'

# Step 2: Push or manually trigger — workflow will now succeed
git push origin main
```

Note: `--field` (not `-f`) is required for the `build_type` parameter; using `-f 'build_type=workflow'` results in an "invalid key" error from the gh CLI.

### Why this is non-obvious
The action's error message says "verify Pages is enabled" — implying you need to check settings — not "enable Pages first". Developers assume the action will configure Pages itself since it's called `configure-pages`. The `--field` vs `-f` distinction in the gh CLI also has no obvious error message.
