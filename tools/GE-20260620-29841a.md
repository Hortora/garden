## Symptom

GitHub Actions CI fails with 401 Unauthorized when Maven resolves dependencies
from GitHub Packages, even though `setup-java` is configured with `server-id`,
`server-username`, and `server-password`, and the main Build step has
`GITHUB_TOKEN` set.

## Context

A workflow has multiple steps that invoke Maven:
1. `setup-java` with `server-password: GITHUB_TOKEN`
2. A Playwright browser install step: `mvn -pl runtime exec:java ...`
3. The main Build step: `mvn install` with `env: GITHUB_TOKEN`

Step 2 runs Maven, which resolves dependencies including GitHub Packages
artifacts. But step 2 has no `env: GITHUB_TOKEN` — only step 3 does.

## Root cause

`setup-java`'s `server-password: GITHUB_TOKEN` does NOT inject the token. It
writes a Maven settings.xml that references an **environment variable named**
`GITHUB_TOKEN`. The actual value must be present as an env var when Maven runs.
Each workflow step has its own env scope — setting `GITHUB_TOKEN` on one step
does not make it available to other steps.

setup-java itself succeeds regardless (it only writes settings.xml). The 401
only surfaces later, when a step actually runs Maven against a GitHub Packages
URL without the env var set.

## Fix

Add `env: GITHUB_TOKEN` to **every** workflow step that invokes Maven, not just
the main build step:

```yaml
- name: Install Playwright browsers
  run: cd server && mvn -pl runtime exec:java ...
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}  # required for GitHub Packages auth

- name: Build
  run: cd server && mvn install
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## Why this is non-obvious

The `server-password: GITHUB_TOKEN` syntax in setup-java looks like it's
receiving the token value — but it's receiving a variable **name**. GitHub's
docs focus on the authentication mechanism without clarifying this indirection.
The 401 error message gives no hint that the env var is missing — it looks
identical to a permissions or token-scope problem.
