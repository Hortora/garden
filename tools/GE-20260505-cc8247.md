## Symptom

```bash
git merge-base --is-ancestor feat/my-branch upstream/main
# exits 1 (false) — branch appears unmerged
```

But the PR was merged weeks ago.

## Context

Checking whether a feature branch had been merged into upstream by testing
SHA ancestry. The branch had been merged via GitHub PR using a **merge commit**
(not a rebase merge). The original branch commits have different SHAs from
what landed in `upstream/main`.

## Root cause

`--is-ancestor` checks whether the exact commit SHA appears as an ancestor in
the target. With a merge commit, GitHub creates a new merge commit whose
parents include the branch tip — but the individual commit SHAs from the branch
are not directly rewritten into the linear history. They're reachable but not
ancestors in the linear sense `--is-ancestor` tests.

With a rebase merge, the original SHAs are replayed onto main and would pass.

## Fix

Use PR status as the authoritative source:

```bash
gh pr view 177 --repo casehubio/engine --json state -q .state
# → MERGED
```

Or search for the merge commit by PR number in the log:

```bash
git log upstream/main --oneline | grep "(#177)"
# → 9a86860 feat: cancelCase... (#177)
```

## Why this is non-obvious

`--is-ancestor` feels like the right tool for "is this branch merged?" but it
tests SHA ancestry, not content. GitHub's merge commit strategy means original
commit SHAs are preserved in the branch but are not directly present in the
linear ancestor chain of the merge target. The PR status API is the only
reliable source of truth.
