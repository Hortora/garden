## `gh issue close` only accepts one issue number — bulk close requires a loop

**ID:** GE-20260414-c08ba3
**Stack:** GitHub CLI (`gh`)
**Labels:** `#github` `#automation` `#cli`

### Symptom

```
$ gh issue close 28 29 30 --repo owner/repo
Error: accepts 1 arg(s), received 3
```

Passing multiple issue numbers to `gh issue close` fails immediately. No partial success.

### Context

Bulk-closing stale issues after completing a sprint or phase. Feels natural to pass multiple IDs — `gh issue list` accepts `--limit N` and other subcommands accept multiple targets.

### What was tried

Passing space-separated issue numbers directly: `gh issue close 28 29 30 31`. Failed with arity error.

### Root cause

`gh issue close` is implemented to accept exactly one positional argument. Unlike `gh issue list` (which accepts filters) and some other `gh` subcommands, the `close` action has no bulk mode.

### Fix

Loop explicitly:

```bash
for issue in 28 29 30 31 33 34 35; do
  gh issue close $issue --repo owner/repo --comment "Work complete."
done
```

Or as a one-liner:

```bash
for i in 28 29 30; do gh issue close $i --repo owner/repo; done
```

### Why this is non-obvious

`gh issue list`, `gh issue view`, and similar subcommands accept filters and bulk targets, making multi-arg close feel like it should work. The inconsistency across `gh` subcommands is not documented prominently.
