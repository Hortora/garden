## GOAP wrong goal key produces empty plan — units idle with no error

**ID:** GE-20260417-c6e3db
**Stack:** GOAP / A* planning, Java
**Labels:** `#goap` `#planning` `#silent-failure` `#debugging`

### Symptom
Units assigned to a GOAP group receive no intent and stand idle. No exception is thrown. The only trace is a debug log line showing `plan=[]` for the group.

### Context
Occurs when the goal condition key passed to `planner.plan(worldState, goalCondition, actions)` names a condition that no action in the library can set to `true` given the starting WorldState. Common when a group is assigned a goal shared with a different group but the starting state makes that goal unreachable via the available action chain.

### What was tried
The Drools classification rules, TacticsRuleUnit population, and dispatch switch were all inspected first — all appeared correct. The planner and action templates were not initially suspected because the planner returned cleanly (empty list, no exception).

### Root cause
The kiting group was assigned goal `"unitSafe"` via `goalConditionKey()`. The WorldState for kiting has `lowHealth: false`. The only action that sets `unitSafe: true` is RETREAT, which requires `lowHealth: true` as a precondition. RETREAT is therefore inapplicable. KITE sets `onCooldown: false` but not `unitSafe`. The planner exhausts the search space and returns `List.of()`. The `if (!plan.isEmpty())` guard in the dispatch loop silently skips the group.

### Fix
Assign the correct goal for the group. For kiting (which should fire KITE then ATTACK), the goal is `"enemyEliminated"` — reachable as `KITE → ATTACK`. The goal key must be one that at least one action chain can satisfy from the group's WorldState.

```java
// goalConditionKey() — wrong
case "KITING" -> "unitSafe";  // unreachable: RETREAT requires lowHealth=true

// goalConditionKey() — correct
case "KITING" -> "enemyEliminated";  // reachable: KITE clears onCooldown, ATTACK sets enemyEliminated
```

### Why this is non-obvious
The planner returns an empty list for two distinct reasons: (1) goal already satisfied (no-op), and (2) goal unreachable. Both are valid, non-exceptional outcomes. The `if (!plan.isEmpty())` guard is correct code — it just silently absorbs the unreachable-goal case. Without a WARN log when a non-empty group gets an empty plan, the failure is invisible at INFO level. The symptom (units idle) looks identical to "no enemies nearby" or "wrong strategy mode".
