## Symptom

`git branch --merged main` returns nothing (or `merged=0`) for branches whose
content is definitively on main. The branch appears unmerged even though every
file change it introduced is present in the main branch.

## Context

After a squash or interactive rebase that changes commit SHAs, the branch's
tip commit is no longer an ancestor of main — even if all the content was
preserved. `--merged` performs pure SHA ancestry checks; it has no awareness
of content equivalence.

```bash
# This lies after squash-rebase
git branch --merged main | grep feat/my-feature   # → nothing

# This tells the truth
git log main --oneline --grep="feat: my feature title"   # → SHA match found
```

## Root cause

`git branch --merged <branch>` checks whether the branch tip is reachable by
walking the commit graph from `<branch>`. After a squash-rebase, the new main
commit containing the squashed content has a completely different SHA than the
original branch commits — they share no graph path.

## Fix

Verify content presence by commit subject, not SHA ancestry:

```bash
# For each branch you want to verify:
git log main --oneline --grep="$(git log -1 --format="%s" <branch>)"
# Non-empty result → content is on main
# Empty result → branch is genuinely unmerged
```

For bulk branch cleanup, check a representative commit from each branch:

```bash
for b in branch1 branch2 branch3; do
  subject=$(git log -1 --format="%s" "$b")
  found=$(git log main --oneline --grep="$subject" | wc -l | tr -d ' ')
  echo "$b: content_on_main=$found"
done
```

## Why this is non-obvious

`git branch --merged` is the standard, documented tool for identifying merged
branches. Nothing in its output or man page warns that squash-rebased content
is invisible to it. A developer who squashed a feature branch into main and
then runs `git branch --merged` to clean up will see those branches as
unmerged and skip them — accumulating stale branches indefinitely, or worse,
deleting them thinking they're not yet merged when they are.
