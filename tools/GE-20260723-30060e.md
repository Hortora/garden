IntelliJ MCP `ide_edit_member`, `ide_replace_member`, and `ide_replace_text_in_file` report success but don't always flush edits to disk when called from parallel subagents, score: 11/15

### Symptom
MCP tool calls return `{"success": true}`. `ide_diagnostics` shows zero errors (IDE buffer is correct). But `mvn compile` fails ŌĆö the file on disk is unchanged. The edit exists only in IntelliJ's in-memory document model.

### Context
Occurs when multiple Claude Code subagents call IntelliJ MCP edit tools concurrently against the same project. Single-agent sequential edits persist reliably. The issue is specific to concurrent access ŌĆö the MCP server processes edits but IntelliJ's VFS doesn't flush all of them to disk.

### What was tried (didn't work)
- `ide_reformat_code` after edits ŌĆö sometimes forces a flush, inconsistent
- `ide_optimize_imports` after edits ŌĆö same inconsistency
- `ide_sync_files` ŌĆö syncs FROM disk TO IDE (wrong direction), would overwrite in-memory changes

### Root cause
IntelliJ's Virtual File System (VFS) batches disk writes. When multiple MCP edit requests arrive concurrently, the VFS may coalesce or drop some flush operations. The MCP plugin reports success based on the document model update, not the disk write. There's no explicit `saveAllDocuments()` call after each edit operation.

### Fix

**Workaround 1 ŌĆö Single agent at a time (most reliable):**
Dispatch one subagent for code edits, not multiple in parallel. Sequential edits through a single agent persist reliably.

**Workaround 2 ŌĆö Edit tool with replace_all: true:**
The Claude Code `Edit` tool with `replace_all: true` writes directly to disk, bypassing IntelliJ entirely. Use for import fixes and mechanical text replacements after `ide_edit_member` handles structural changes.

**Workaround 3 ŌĆö Verify and fix up:**
After agent completion, run `mvn compile` to detect unflushed edits. Fix remaining issues with `Edit` (replace_all: true) which always writes to disk.

### Why this is non-obvious
The MCP tool returns success. The IDE shows the correct code. `ide_diagnostics` confirms zero errors. Only an external build tool (Maven) reveals that the disk file is stale. The failure is completely silent from the MCP client's perspective.

### Alternatives considered
- Running `ide_build_project` to force a save-all before build ŌĆö IntelliJ build also has classpath issues (worker-api dependency) making it unreliable as a verification step
- Using `ide_sync_files` after edits ŌĆö this syncs diskŌåÆIDE, not IDEŌåÆdisk, so it would overwrite the in-memory changes
