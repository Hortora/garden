## Symptom

All commits land without `Refs #N` or `Closes #N` footers. No GitHub issues are created before or during implementation. The CLAUDE.md `## Work Tracking` section says issues are required before implementation begins — but subagents ignore this entirely.

## Context

Using `superpowers:subagent-driven-development` to implement a multi-task feature. CLAUDE.md has Work Tracking enabled with automatic behaviours: "Before implementation begins — check if an active issue or epic exists. If not, run issue-workflow Phase 1." All 10 implementation commits are merged to main without any issue references.

## Root Cause

Each subagent starts with a fresh context. CLAUDE.md automatic behaviours are instructions to the *controller Claude* (the parent session), not to subagents. The subagent prompt provided by the controller did not include issue-workflow requirements. The subagent had no knowledge of the work-tracking convention and followed only the explicit task instructions it was given.

## Fix

Explicitly include issue-workflow requirements in each implementer subagent prompt. Add to the "Before You Begin" section:

```
Before writing any code:
1. Check for an existing GitHub issue: gh issue list --state open --repo mdproctor/quarkmind
2. If none exists for this task, create one and use its number in your commit footer.
3. All commits must include Refs #N or Closes #N.
```

Or, handle issue creation at the controller level before dispatching the implementer, then pass the issue number as part of the task context.

## Why Non-Obvious

CLAUDE.md automatic behaviours feel like global configuration. The mental model is "Claude will always do X". But subagents are independent Claude instances with isolated context — they receive only what the controller explicitly provides in their prompt. The parent's CLAUDE.md is not loaded into subagent context.
