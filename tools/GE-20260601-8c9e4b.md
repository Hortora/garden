### Context
You need to make a commit on a branch (e.g. an empty stamp commit) but the
branch is checked out in a linked worktree — `git checkout <branch>` fails
because git prevents two worktrees from sharing the same branch. `git commit`
requires the target branch to be the current branch.

### Technique
Use `git commit-tree` to create the commit object directly, then
`git update-ref` to advance the branch ref — no checkout required:

```bash
BRANCH_TIP=$(git rev-parse <branch>)
TREE=$(git rev-parse "<branch>^{tree}")
NEW=$(git commit-tree -m "chore: branch closed" -p "$BRANCH_TIP" "$TREE")
git update-ref "refs/heads/<branch>" "$NEW"
```

This creates an empty commit (reusing the branch tip's tree) and moves the
branch pointer to it. Works on any branch regardless of worktree lock state.

For a non-empty commit, replace `"<branch>^{tree}"` with the tree SHA of
the content you want to commit.

### Why this is non-obvious
The standard path is `git checkout <branch> && git commit`. When that fails
due to a worktree lock, most developers either remove the worktree first or
give up. The plumbing commands `commit-tree` and `update-ref` are not
commonly known outside git internals work — but they compose cleanly and
require no working tree state at all.
