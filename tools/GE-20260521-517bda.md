### Symptom

Running `git rebase -i backup/pre-squash-main-YYYYMMDD` immediately produces file conflicts
on the very first commit — even though the working branch was created from `main` and
the backup branch is just an older snapshot of `main`.

### Context

A squash session creates a backup branch (`backup/pre-squash-main-YYYYMMDD`) before
rewriting history. The working branch is cut from current `main`. The natural instinct
is to rebase the working branch onto the backup branch to preserve the squash window.

### Root cause

The backup branch and the working branch are **siblings** — they both descend from the
same merge-base (the commit where the previous squash landed). The backup is NOT an
ancestor of the working branch's oldest commit. `git rebase` attempts to re-apply the
oldest commits onto the backup tip, which already has those changes in a different
(squashed) form — instant conflict.

```
           backup/pre-squash (squashed)
          /
merge-base
          \
           commit-A → commit-B → ... (working branch range)
```

Using `backup` as the base tells git "replay A, B, ... on top of backup" — but backup
already contains A+B+... squashed, so every commit conflicts.

### Fix

Use the **direct parent of the oldest commit in the squash range** as the base, not
the backup branch:

```bash
# Find the parent of the oldest commit in range
OLDEST=$(git log --format="%H" backup/pre-squash..HEAD | tail -1)
BASE=$(git log -1 --format="%P" "$OLDEST")

# Rebase using the true ancestor
GIT_SEQUENCE_EDITOR="cp /tmp/rebase_todo.txt" git rebase -i "$BASE"
```

Or use `HEAD~N` where N is the exact count of commits in the squash range — always
a direct ancestor.

### Why this is non-obvious

The backup branch *looks* like the right base — it's what `main` looked like before
the squash began. But "what main looked like before" is a sibling, not a parent.
The error messages say "conflict" not "wrong base", sending you down the wrong
debugging path.

### Constraints

Applies when the backup branch was created by a *prior* squash session (i.e., the
backup itself contains the squashed result of the range you're trying to squash again).
Does not apply if the backup is the literal parent commit of the range.
