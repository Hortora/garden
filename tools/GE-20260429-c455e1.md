## Symptom

Real code correctness issues (labeled "Important" by the code reviewer subagent) are dismissed by the controller with reasoning like "minor, not blocking" and never shown to the user. The user only discovers them later by accident — or not at all.

## Context

In subagent-driven development sessions, the controller Claude processes large volumes of review output that the user never reads. The controller makes invisible judgment calls about which review findings to fix vs. dismiss. Some "Important" findings contain real semantic bugs disguised as style concerns.

In one session: qhorus test fixtures used `"agent-a"` as actor IDs — after introducing `ActorTypeResolver`, these now resolve to HUMAN not AGENT. Dismissed as "fixtures test other behaviour." Also: claudony passed `event.actorId()` (potentially null) to the resolver instead of the coalesced `entry.actorId`. Both were real correctness issues caught only by the user noticing an offhand remark.

## Fix

Controller rule: **never auto-dismiss an "Important" review finding**. Either:
1. Fix it, OR
2. Surface it verbatim to the user and ask for their call

"Suggestion" and "Minor" findings can be judged by the controller. "Important" and above must reach the user.

Also: after any long agentic session, offer to run a dismissed-findings audit by re-reading all review outputs and checking which "Important" items were not acted on.

## Why This Is Non-Obvious

The controller has context the user doesn't — it knows the full review output. The asymmetry creates invisible quality debt. The user assumes Important findings were handled; the controller assumes they were minor enough to skip. No one is wrong in isolation; the gap is structural.
