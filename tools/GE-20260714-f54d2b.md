### Symptom
Vitest tests pass locally (sibling repo present) but fail in CI with
`Failed to resolve import "@casehubio/pages-primitives"`. The vitest config
hardcodes aliases like `path.resolve(__dirname, '../../../pages/packages/...')`.

### Context
Multi-repo setup: `blocks-ui` consumes `pages` packages. Locally, both repos are
siblings — vitest aliases point to pages source for hot-reload. In CI, only blocks-ui
is checked out — the sibling path doesn't exist, and there's no error at config parse
time.

### Fix
Wrap sibling repo aliases in `existsSync` guards. When the sibling exists, use the
source alias; otherwise fall back to node_modules resolution:

```typescript
import { existsSync } from 'fs';
import path from 'path';

export default defineConfig({
  resolve: {
    alias: [
      ...(existsSync(path.resolve(__dirname, '../../../pages/packages/pages-primitives/src'))
        ? [{ find: '@casehubio/pages-primitives',
             replacement: path.resolve(__dirname, '../../../pages/packages/pages-primitives/src') }]
        : []),
      // non-sibling aliases are unconditional
      { find: '@casehubio/blocks-ui-core',
        replacement: path.resolve(__dirname, '../../packages/blocks-ui-core/src') },
    ],
  },
});
```

### Why this is non-obvious
The natural approach is either "always alias" (breaks CI) or "never alias" (loses local
source resolution). The `existsSync` guard is a runtime conditional in build config — not
a pattern most developers reach for. The failure mode is also delayed: vitest configs with
missing alias targets don't error at parse time, only when a test imports the package.
