## WorldState.get() returns false for absent keys — open-world semantics

**ID:** GE-20260417-988839
**Stack:** GOAP / A* planning, Java (custom WorldState record)
**Labels:** `#goap` `#worldstate` `#open-world` `#preconditions`

### What it is
In a typical GOAP `WorldState` implementation backed by `Map<String, Boolean>`, calling `get(key)` for a key not present in the map returns `false` — not null, not an exception. This makes `WorldState` an open-world model: conditions that haven't been explicitly set are implicitly false.

### How it's implemented
```java
public boolean get(String key) {
    return Boolean.TRUE.equals(conditions.get(key));
}
```

`Map.get()` returns `null` for absent keys. `Boolean.TRUE.equals(null)` returns `false`. The result: any condition not set in the WorldState is treated as `false`.

### Why it matters for action applicability
`GoapAction.isApplicable()` calls `state.get(key)` for each precondition. If a precondition requires `onCooldown: false` and the WorldState doesn't contain the `onCooldown` key at all, `state.get("onCooldown")` returns `false` — satisfying the precondition. This means:

- An action requiring `someCondition: false` is applicable to any WorldState that either explicitly sets `someCondition: false` **or omits it entirely**.
- An action requiring `someCondition: true` is **not** applicable to a WorldState that omits the key.

### Practical consequence
WorldStates for groups that don't explicitly set `onCooldown` implicitly satisfy `onCooldown: false`. This is usually correct — but it means the distinction between "condition is false" and "condition was never set" is invisible. For defensive correctness, set all relevant boolean conditions explicitly in every WorldState, even when the value is `false`:

```java
// Fragile — onCooldown defaults to false via absent-key semantics
new WorldState(Map.of("inRange", true, "enemyVisible", true))

// Explicit — intent is clear, immune to semantics changes
new WorldState(Map.of("inRange", true, "enemyVisible", true, "onCooldown", false))
```

### Why this is undocumented
This behaviour is a direct consequence of `Boolean.TRUE.equals(null) == false` and the standard `Map.get()` null-for-absent contract. It is not stated in any GOAP specification or tutorial. You discover it by reading the `get()` implementation or by encountering a case where an action fires on a WorldState that doesn't explicitly set a condition it checks.
