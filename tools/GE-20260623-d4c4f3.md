## Symptom

Running `lsof -p $PID -d cwd -Fn` to get a process's working directory produces thousands of lines of output instead of a single path.

## Context

Trying to map a running process PID to its current working directory on macOS, typically to associate a process with a project or config file.

## What was tried

```bash
lsof -p $PID -d cwd -Fn 2>/dev/null | grep '^n' | sed 's/^n//'
```

Expected: one line (the CWD path). Got: hundreds or thousands of lines — all file descriptors in name format.

## Root cause

The `-Fn` flag switches lsof to field output mode, outputting ALL fields for ALL file descriptors in the format `f<type>\nn<name>`. The `-d cwd` filter is supposed to restrict to the cwd file descriptor, but on macOS the combination results in all descriptors being output anyway. The `Fn` output mode overrides the `-d` selector behaviour.

## Fix

Use standard column output and filter with awk:

```bash
CWD=$(lsof -p "$PID" 2>/dev/null | awk '$4 == "cwd" {print $NF; exit}')
```

The standard (non-field) output has column 4 as the file descriptor type (`cwd`, `txt`, `mem`, etc.) and the last column as the path. The `exit` ensures only the first match is returned.

## Why this is non-obvious

The `lsof` man page lists `-d cwd` and `-F` as independent options. The interaction between field-output mode and the descriptor filter is not documented — the natural assumption is that `-d cwd -Fn` would give you the CWD in field format (one path), not bypass the filter entirely.
