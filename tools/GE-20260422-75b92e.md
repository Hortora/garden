### Symptom
`git checkout feat/foo` or `git rebase ... feat/foo` from the main worktree
fails with:

```
fatal: 'feat/foo' is already used by worktree at '/path/.worktrees/feat/foo'
```

### Context
A feature branch is checked out in a git worktree (e.g. created via
`git worktree add .worktrees/feat/foo feat/foo`). Attempting to operate on that
branch from the main worktree — checkout, rebase, or reset — fails because git
enforces that a branch can only be checked out in one worktree at a time.

### What was tried (didn't work)
- `git checkout feat/foo` from main worktree — blocked
- `git rebase --onto upstream/main <cut> feat/foo` from main worktree — blocked

### Root cause
Git's worktree safety mechanism prevents the same branch from being checked out
in two places simultaneously. The restriction applies to all operations that
would move the branch HEAD, not just `git checkout`.

### Fix
Run the operation from within the worktree directory using `git -C`:

```bash
# Rebase the branch from within its worktree
git -C .worktrees/feat/foo rebase --onto upstream/main <cut-point> feat/foo

# Or cd into the worktree
cd .worktrees/feat/foo
git rebase upstream/main
```

`git -C <path>` runs the command as if executed from that directory — the
branch is already checked out there, so the safety check passes.

### Why this is non-obvious
The error message names the worktree path, making it obvious why it fails. But
the fix isn't mentioned anywhere in the error — you have to know about `git -C`
or think to cd into the worktree. Most rebase workflows assume you're operating
from the repo root.
