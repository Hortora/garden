## Use `terminal.paste()` not `ws.send()` to inject text into an xterm.js terminal from JavaScript

**ID:** GE-0127
**Stack:** xterm.js 5.x, @xterm/addon-attach
**Labels:** `#pattern` `#testing`
**What it achieves:** Reliably sends text through the xterm.js AttachAddon input pipeline, including correct bracketed paste mode handling, without depending on a WebSocket reference that may be stale after reconnects.
**Context:** When using `@xterm/addon-attach`, the natural approach to inject text from JavaScript is `ws.send(text)` — the same socket the AttachAddon uses. This breaks in two ways: the WebSocket reference goes stale on reconnect, and `ws.send()` bypasses bracketed paste mode.

### The technique

```javascript
// WRONG — bypasses AttachAddon, breaks on reconnect, no bracketed paste
ws.send(text);

// RIGHT — routes through AttachAddon's onData handler, handles bracketed paste
terminal.paste(text);
```

`terminal.paste(data)` fires the xterm.js `onData` event. AttachAddon listens to `onData` and forwards it to the WebSocket. This path is always current regardless of reconnects. xterm.js automatically wraps the text in `\x1b[200~...\x1b[201~` if the connected application has enabled bracketed paste mode.

### Why this is non-obvious
`ws.send()` works correctly for single keystrokes, making it seem like the right approach for longer text too. The failure mode is subtle: `ws.send()` succeeds without error but the text either doesn't arrive or arrives incorrectly in applications that have enabled bracketed paste mode. `terminal.paste()` is documented in the xterm.js API but its role as the correct input injection method for addon-based setups is not explained.

### When to use it
Any xterm.js + AttachAddon setup where JavaScript needs to inject text programmatically — key bars, macro buttons, test harnesses, paste-from-clipboard handlers.
