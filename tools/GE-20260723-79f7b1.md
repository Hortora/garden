### Symptom
`ide_refactor_rename` reports success but renames the wrong symbol.
Line and column target a TypeScript interface name, but the tool renames
an adjacent property or variable declaration instead. No error is returned.

### Context
Observed when renaming TypeScript interfaces (`TableProps`) and class
declarations (`PagesTable`) using exact line/column positions from
`ide_find_class`. The PSI element resolved at the cursor position was a
nearby property (`pageSize`) or constant (`FALSY`) rather than the intended
declaration name.

### What was tried (didn't work)
1. Using `ide_find_class` to get exact line/column — correct coordinates,
   wrong PSI resolution
2. Using `relatedRenamingStrategy: "none"` to limit scope — same issue,
   wrong target selected

### Root cause
The IntelliJ MCP plugin's PSI element resolution at a given position in
TypeScript files doesn't always land on the declaration name. Interface
declarations and class declarations appear to have PSI boundaries that
don't align with the visible token positions — the cursor hits a child
element (property, field) rather than the containing declaration name.

### Fix
Use `ide_replace_text_in_file` for text-based find-and-replace, or
`Edit` with `replace_all: true` for TypeScript symbol renames. Only use
`ide_refactor_rename` for Java/Kotlin where PSI resolution is reliable.

### Why this is non-obvious
The tool reports success with the correct change count, and the
line/column positions match `ide_find_class` output exactly. The wrong
rename is only discovered when compilation fails or tests break — by
which point you've moved on and the damage is scattered.

### Alternatives considered
- Syncing files before rename (`ide_sync_files`) — no effect, PSI was
  already current
- Using different column offsets — unreliable, no pattern to which
  offset would hit the right element
