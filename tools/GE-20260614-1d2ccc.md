## Node.js stdout block-buffers in detached Docker containers — logs invisible without TTY

**ID:** GE-20260614-1d2ccc
**Stack:** Docker/Podman, Node.js 18+, Testcontainers

### Symptom

A Node.js container started with `-d` (detached) produces no visible log output.
`docker logs <container>` returns empty. Testcontainers log-based wait strategies
(`Wait.forLogMessage(...)`) hang until timeout. The container is running; the application
is executing; output simply never appears.

### Context

Any containerised Node.js application started in detached mode without `-t` (TTY
allocation). Common when using Testcontainers `GenericContainer` without
`withCreateContainerCmdModifier(cmd -> cmd.withTty(true))`.

### What was tried (didn't work)

- Verified the container was running (`podman ps` shows Up)
- Checked container logs immediately after start — empty
- Added startup sleep before reading logs — still empty
- Assumed the container had a bug and wasn't producing output

### Root cause

Node.js detects whether stdout is a TTY using `process.stdout.isTTY`. When a TTY is
present, stdout is line-buffered (each `console.log` flushes immediately). Without a TTY
(which is the case for `docker run -d`), Node.js switches to block-buffered mode and only
flushes when the buffer fills (~64KB) or the process exits. In practice, a long-running
Node.js server never fills the buffer, so no logs ever appear in `docker logs`.

### Fix

Allocate a TTY at container creation:

**With Testcontainers:**
```java
container.withCreateContainerCmdModifier(cmd -> cmd.withTty(true));
```

**With Docker CLI:**
```bash
docker run -d -t ...   # -t allocates a PTY even in detached mode
```

**With docker-compose:**
```yaml
services:
  app:
    tty: true
```

The TTY flag forces Node.js back into line-buffered mode, making every `console.log`
visible immediately.

### Why this is non-obvious

The Docker documentation describes `-t`/`--tty` as "Allocate a pseudo-TTY" with no
mention of stdout buffering. Nothing in `docker logs` output explains why it's empty.
Node.js's buffering rules are documented in the Node.js stream docs but are not
cross-referenced from any container tooling. The standard mental model is that `-d`
just detaches the terminal — not that it changes the buffering behaviour of the process.
