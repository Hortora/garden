### Symptom

Running `gh repo edit <repo> --visibility public` (or `private`) exits with a
long error block reproducing the full `--help` output and the message:

```
use of --visibility flag requires --accept-visibility-change-consequences flag
```

No changes are made. The flag name is not mentioned in the short help (`gh repo edit --help` only lists the flags, not the dependency between them).

### Context

Any attempt to change a repository's visibility — public ↔ private ↔ internal —
via `gh repo edit`. Affects scripts and one-off CLI commands alike.

### What was tried

`gh repo edit <repo> --visibility public` — fails immediately with the above error
even when the rest of the command is valid.

### Root cause

GitHub CLI requires explicit acknowledgement that changing visibility has
consequences (e.g. making a private repo public exposes all its history and forks).
The acknowledgement flag is mandatory but is not printed in the short help output —
only the full error text reveals it.

### Fix

Always append `--accept-visibility-change-consequences` when changing visibility:

```bash
gh repo edit <repo> --visibility public --accept-visibility-change-consequences
gh repo edit <repo> --visibility private --accept-visibility-change-consequences
```

For bulk changes (multiple repos, no loops):
```bash
gh repo edit org/repo1 --visibility public --accept-visibility-change-consequences
gh repo edit org/repo2 --visibility public --accept-visibility-change-consequences
```

### Why this is non-obvious

The short `--help` output lists `--visibility` with no mention of its dependency
on `--accept-visibility-change-consequences`. The error message is buried at the
end of a full help block dump, making it easy to miss. A developer scripting
multiple visibility changes would not anticipate this guard flag.
