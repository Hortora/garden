### Symptom
Commenting `/retest` on a PR triggers a new workflow run, but it immediately
shows as `completed skipped`. CI does not re-run. The failed checks remain.

### Context
Repos with a `/retest` GitHub Actions workflow (triggered on `issue_comment`
events) use it to re-run CI on a PR branch. The workflow checks conditions
before deciding whether to rerun — and often concludes there is nothing to do,
producing a `skipped` conclusion rather than re-triggering the build.

### What was tried (didn't work)
- Commenting `/retest` on the PR — workflow ran but produced `completed skipped`
- Commenting `/retest` a second time — same result

### Root cause
`/retest` workflows typically check whether the PR's current HEAD SHA already
has a passing or in-progress check run. If conditions aren't met (e.g. the
workflow logic gates on specific failure states it doesn't see), the job skips
rather than triggers a full build.

### Fix
Use `gh run rerun` directly to re-run only the failed jobs in a specific run:

```bash
# Find the failed run ID
gh run list --repo <owner/repo> --limit 10 --json databaseId,conclusion,displayTitle \
  | jq '.[] | select(.conclusion == "failure") | {id: .databaseId, title: .displayTitle}'

# Rerun only the failed jobs
gh run rerun <run-id> --repo <owner/repo> --failed
```

The `--failed` flag re-runs only the failed jobs, not the entire workflow.
The result updates the same run ID in-place — check the same ID again to see
the new conclusion.

### Why this is non-obvious
The `/retest` comment appears to work — a new workflow run appears immediately.
The `completed skipped` status looks like the retest concluded there was nothing
to do, so it's tempting to wait for something to happen. Nothing will. The
`gh run rerun --failed` approach bypasses the bot logic entirely and directly
re-executes the failed jobs.
