### Symptom

```
failed to fork: HTTP 403: The repository exists, but it contains no Git content.
Empty repositories cannot be forked.
```

Returned immediately when running `gh repo fork casehubio/<name> --clone=false` on a newly-created repository.

### Context

When bootstrapping a new GitHub repository in a fork-based workflow (casehubio org → mdproctor fork → rewire remotes), the typical sequence is: create repo → fork → clone. This order fails if the repository has no commits yet.

### What was tried (didn't work)

Running `gh repo fork` immediately after `gh repo create` on an empty repository.

### Root cause

GitHub's API rejects fork requests for repositories with no Git content. An empty repository (no commits, no default branch) is considered unforkable via the API, even though it exists and is accessible.

### Fix

Push at least one commit to the upstream repository before attempting to fork:

```bash
# 1. Create the repo
gh repo create casehubio/<name> --public

# 2. Clone it
git clone https://github.com/casehubio/<name>.git /path/to/<name>

# 3. Add content, commit, push to upstream
git -C /path/to/<name> add .
git -C /path/to/<name> commit -m "init: scaffold"
git -C /path/to/<name> push upstream main   # or origin main before remotes are rewired

# 4. Now fork
gh repo fork casehubio/<name> --clone=false
```

### Why this is non-obvious

The error says "empty repositories cannot be forked" — but the repository clearly exists. The GitHub UI shows an empty repository and allows manual fork; the API restriction does not appear in GitHub's fork documentation. The natural bootstrap sequence (create → fork → clone → first commit) fails silently unless the developer knows to reverse the order.

### Invalidation triggers

Revisit if GitHub changes its API to allow forking empty repositories.
