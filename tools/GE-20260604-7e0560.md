## Symptom

`git rebase -i` aborts mid-way with:

```
error: The following untracked working tree files would be overwritten by merge:
        path/to/file.md
Please move or remove them before you merge.
Aborting
```

This happens even though the file is listed in `.gitignore` on the current HEAD (after a gitignore commit) and the file was "cleaned up" in earlier commits.

## Context

A file was accidentally committed via `git add -A` in commit A. It was then deleted in commit B, and added to `.gitignore` in commit C. During rebase, git replays the commits in order. When it reaches commit A (which creates the file), the untracked version of the file still exists in the working tree — it was never removed from the filesystem, only from git tracking. Git refuses to overwrite it.

```
Sequence on disk:
  A — git add -A created docs/spec.md in history
  B — removed docs/spec.md from history
  C — added docs/superpowers/specs/ to .gitignore

Working tree at rebase time:
  docs/spec.md exists as UNTRACKED (it was never rm'd, just ignored)

During rebase replay of A:
  git tries to create docs/spec.md → finds existing untracked file → ABORTS
```

## Root cause

`git rebase` works by replaying commits onto a new base. When replaying commit A, git examines what files it would create and checks for conflicts with the current working tree. An untracked file at the same path counts as a conflict — git won't silently overwrite it even if the file would be deleted by a later commit in the same rebase sequence. Git does not look ahead.

The gitignore commit (C) is irrelevant: it hasn't been replayed yet at the point where A is being applied.

## Fix

Before rebasing, remove or move the untracked file:

```bash
# Option 1: move it aside
mv docs/spec.md /tmp/spec-backup.md
git rebase -i origin/main
# file is safe to restore or ignore after rebase

# Option 2: delete it (if it's a disposable workspace artifact)
rm docs/spec.md
git rebase -i origin/main
```

**Prevention:** After any cleanup commit that removes an accidentally-committed file, also delete the untracked original from the filesystem before doing rebase operations:

```bash
rm path/to/accidentally-committed-file.md
# or add to .gitignore AND remove from filesystem
```

## Why this is non-obvious

The error looks like a regular "checkout would overwrite files" error typically seen with `git checkout` or `git pull`. In a rebase context, developers expect conflicts to be about tracked files with divergent content — not untracked files that "don't exist" in git's view. The connection between "a commit in the replay range creates this file" and "untracked file in working tree" is non-obvious, especially when the file has already been cleaned up in a later commit on the same branch.

Compounding factor: the developer may have added the file to `.gitignore` already, creating the false impression that git is ignoring it. `.gitignore` prevents future tracking but does not remove the file from the filesystem — and it does nothing to prevent a replay commit from trying to create it.
