## What Most Developers Would Do

The `subagent-driven-development` skill dispatches one task at a time — implement, review spec, review quality, mark complete, repeat. Sequential dispatch is safe but slow when unblocked tasks are independent.

## The Technique

When two or more tasks touch fully disjoint file sets with no shared state or sequencing dependency, dispatch both implementer subagents in the **same message** (parallel tool calls). Mark both as `in_progress` in `TaskUpdate` simultaneously before dispatching. Collect both results before running reviews.

```
// Both dispatched in one message:
Agent({ description: "Task 3: ReactiveJpaChannelStore + ReactiveJpaMessageStore", ... })
Agent({ description: "Task 5: InMemoryReactive*Store + unit tests", ... })
```

After both return, run their spec reviews in parallel too. Run quality reviews in parallel if they're independent. This cuts elapsed time roughly in half for two parallel tasks.

## Safety Conditions (all must hold)

1. **Disjoint files** — no file is written by both tasks
2. **No data dependency** — task B does not consume output of task A
3. **No CDI/sequencing concern** — both tasks can compile and test independently
4. **Different test scopes** — if both write tests, they use different test classes / directories

If any condition fails, revert to sequential.

## Why This Is Non-Obvious

The skill documentation implies strict sequential execution (one subagent at a time). The parallel dispatch pattern requires the controller to actively identify independence and batch the Agent tool calls — this is not a default behaviour and requires judgment about file-level disjointness.

## Example

Task 3 (JPA implementations: `store/jpa/ReactiveJpa*.java`) and Task 5 (InMemory wrappers: `testing/src/main/java/*.java` + unit tests) share no files and have no data dependency. Dispatched in parallel, both completed and were reviewed sequentially, saving one full implementation round-trip.
