### Symptom
Replacing `const gridStyle = \`grid-row: ...\`` in one method (`_renderCell`) also replaced the identical pattern in `_renderCheckbox` and `_renderExpandCell`, introducing `spanStyle is not defined` ReferenceError because the replacement text referenced a variable only available in `_renderCell`.

### Context
During a large refactoring of a 2500-line Lit component, multiple methods contained the same `const gridStyle = ...` assignment. The intent was to replace it in `_renderCell` only, but `ide_replace_text_in_file` replaced all three occurrences.

### Root cause
`ide_replace_text_in_file` performs a global find-and-replace across the entire file. When the `searchText` matches multiple locations, ALL are replaced — there is no parameter to limit to a specific occurrence or range.

### Fix
When replacing text that may appear in multiple locations:

1. **Make the search text unique** — include enough surrounding context (preceding/following lines) that only the target location matches
2. **Use a Python script** for method-level precision — find the method by its signature, compute start/end offsets, replace only within that range:

```python
start_idx = content.find('  private _renderCell(')
end_marker = content.find('\n\n  private _nextMethod(', start_idx)
# Replace only within content[start_idx:end_marker]
```

3. **Verify after replacement** — grep for the search text to confirm it was replaced in the right number of locations

### Why this is non-obvious
The tool name `replace_text_in_file` suggests a single targeted replacement. Most text editors default to "replace next" with an option for "replace all". The MCP tool has no such distinction — it is always "replace all". The 1-occurrence confirmation in the tool result (`Replaced 1 occurrence(s)`) vs `Replaced 3 occurrence(s)` is the only signal, and it's easy to miss when focused on the next step.
