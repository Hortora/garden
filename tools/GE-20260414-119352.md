## Symptom

Running a multi-file Java class rename on macOS with `sed -i '' 's/\bOldName\b/NewName/g'`
produces no error, no output, and no substitutions. The files are untouched. A full
compile cycle is needed to discover the rename silently did nothing.

## Context

macOS ships BSD sed, which does not support `\b` word-boundary assertions. GNU sed
(Linux, most CI) supports them. The command succeeds — exit code 0 — but the pattern
never matches.

## What was tried

```bash
find . -name "*.java" | grep -v target | xargs sed -i '' 's/\bDispatchRule\b/Binding/g'
```

## Root cause

BSD sed's ERE and BRE dialects do not include `\b`. The pattern is accepted syntactically
but matches nothing, so the `-i` flag writes back the original file unchanged.

## Fix

Use `perl -pi` instead — it supports `\b` on all platforms:

```bash
find . -name "*.java" | grep -v target | \
  xargs perl -pi -e 's/\bOldName\b/NewName/g'
```

For Java class renames where import lines also need updating, combine both in one pass:

```bash
find . -name "*.java" | grep -v target | xargs perl -pi -e '
  s/import io\.example\.OldName;/import io.example.NewName;/g;
  s/\bOldName\b/NewName/g;
'
```

## Why non-obvious

`sed` exits 0 and writes the files — there is no error. A developer naturally assumes the
rename worked and discovers the problem only at the next compile. On Linux the same
command works perfectly, so the failure is macOS-specific and won't appear in CI if CI
runs Linux.
