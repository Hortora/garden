### Symptom
`gh repo fork casehubio/<name> --clone=false` returns:
`failed to fork: HTTP 403: The repository exists, but it contains no Git content. Empty repositories cannot be forked.`

### Context
Creating a new repo with `gh repo create`, then immediately attempting to fork it before pushing any content.

### What was tried (didn't work)
Attempted fork immediately after `gh repo create` — assumed the repo was ready for forking since it existed and was accessible.

### Root cause
GitHub's fork API requires at least one commit in the source repo. An empty repo (no branches, no commits) returns HTTP 403 — the same status code used for actual permission failures.

### Fix
Push initial content before forking. The correct sequence:
1. `gh repo create casehubio/<name> --public`
2. `git clone` → create files → `git commit` → `git push`
3. **Then** `gh repo fork casehubio/<name> --clone=false`
4. Rewire remotes: `git remote rename origin upstream && git remote add origin <fork-url>`

### Why this is non-obvious
HTTP 403 = "Forbidden" — the natural debugging path is to check PAT scopes, org permissions, and repo visibility settings. The actual cause (empty repo) is buried in the error message body text, and the status code actively misleads toward an auth investigation.
