## Symptom
Writing a comprehensive multi-issue implementation plan in a single Claude response triggers:
> `API Error: Claude's response exceeded the 32000`
The session is cut off mid-response. Nothing is written; context is lost.

## Context
Invoking `superpowers:writing-plans` for a large epic (8 issues × 5 domains with full
concrete code per task) causes the response to exceed ~32 000 output characters/tokens.
The model has capacity in its context window — the limit is on *emitted output*, not
held context.

## What Was Tried
Writing the full plan — all tasks with complete Java code blocks — as one uninterrupted
response to a single tool-call sequence.

## Root Cause
Claude Code enforces a per-response output size limit (~32 000 chars), independently of
the context window. The error message says "exceeded the 32000" but does not say
"context" — it is an *output* limit, not an input limit. The two are easily confused.

## Fix
Break large plan generation into chunks committed separately:

1. **Write the plan file structure + header only** — commit.
2. **Write Tasks 1–N in one response** — commit.
3. **Continue with Tasks N+1–M** — repeat until complete.

Or: generate one plan file per epic issue rather than one monolithic file.
The `writing-plans` skill should be told the scope explicitly:
> "Write the plan for Issue #74 only (5 tasks max per response)."

## Why This Is Non-Obvious
The error message uses "32000" which looks identical to context-window terminology.
A developer's first instinct is to reduce context (delete files, shorten prompts)
rather than split the *output*. The context window was nowhere near full when the
error fired.

### Why this fix
Chunking output is the only lever — context reduction does not help because
the problem is on the emission side.

### Alternatives considered
- Reducing code verbosity in the plan: insufficient — the sheer number of tasks
  (40+) exceeds the limit regardless of per-task detail level.
- Using a subagent: subagents have the same per-response limit.
