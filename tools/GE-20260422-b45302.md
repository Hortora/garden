## Wrapper script eliminates shell expansion prompts in Claude Code agents without disabling security

**Stack:** Claude Code, Bash
**What it does:** When a Claude Code agent needs to call a tool whose path depends on an environment variable (e.g. `${SOREDIUM_PATH:-~/path/to/soredium}`), using a tiny wrapper script in the working directory eliminates the "Contains expansion" permission prompt without requiring `--dangerously-skip-permissions`.

### The technique

Create a wrapper script (e.g. `run-scanner.sh`) that handles path resolution internally:

```bash
#!/usr/bin/env bash
# run-scanner.sh — resolves SOREDIUM_PATH so the agent doesn't have to
SOREDIUM="${SOREDIUM_PATH:-$HOME/claude/hortora/soredium}"
exec python3 "$SOREDIUM/scripts/dedupe_scanner.py" "$@"
```

Add the wrapper to the agent's `settings.json` allowlist:
```json
"Bash(bash run-scanner.sh *)"
```

The agent calls `bash run-scanner.sh . --top 50` — a plain string with no `${...}` syntax. The wrapper runs the expansion; Claude never sees it in the command string.

### Why this works

Shell expansion syntax in a **script file** is evaluated by the shell at runtime — Claude Code never inspects script file contents for expansions. The "Contains expansion" check only fires on the **command string** Claude constructs and passes to the Bash tool. By moving the expansion into a pre-existing script, the agent's command string stays expansion-free.

### Why developers wouldn't naturally reach for this

The obvious response to expansion prompts is to either hardcode the path (fragile) or use `--dangerously-skip-permissions` (disables security). The wrapper pattern keeps the permission model intact while eliminating the prompt entirely.
