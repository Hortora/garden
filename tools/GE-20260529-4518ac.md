### Symptom
A `sequenceDiagram` block fails to render — usually with a generic parse error or
blank output. No helpful message identifies which character caused the failure.

### Context
Any of these characters appear inside `Note over` text or arrow label text
in a Mermaid `sequenceDiagram` block:
- Semicolon `;` in a Note (e.g. `Note over Engine: step 1; step 2`)
- Em dash `—` or Unicode arrow `→` in Note or arrow label text
- Curly braces `{}` in arrow labels (e.g. `A->>B: writeContext({key: "val"})`)

### Root cause
Three separate lexer issues, all producing the same silent failure:

1. **Semicolons** — Mermaid treats `;` as a statement terminator everywhere,
   including inside Note text. `Note over Engine: step 1; step 2` is parsed as
   two statements: a valid Note and then the invalid fragment `step 2`.

2. **Em dash and Unicode arrows** — `—` (U+2014) and `→` (U+2192) are not in
   Mermaid's lexer character set for Note text in common bundled versions
   (including Typora's embedded Mermaid). They cause a lexer error with no
   position information.

3. **Curly braces in arrow labels** — `{` and `}` in arrow label text are
   interpreted as Mermaid's object/map syntax, breaking the label parse.

### Fix
Replace the offending characters with ASCII equivalents:

| Avoid | Use instead |
|-------|-------------|
| `;` in Note text | `,` or reword |
| `—` (em dash) | `-` or `,` |
| `→` (Unicode arrow) | `->` or `to` or `then` |
| `{}` in arrow labels | plain text description |
| `> N` in Note text | `gt N` (angle bracket also triggers issues) |

### Why this is non-obvious
All three issues produce the same generic render failure with no indication of
which character or line caused it. Developers naturally write Note text as prose
(with semicolons for lists, em dashes for asides) and copy arrow labels from code
comments. The Mermaid documentation does not list reserved characters for Note
or arrow label contexts.
