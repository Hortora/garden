### Context

When rewording commits in a non-interactive git rebase using `exec git commit --amend`, the natural approach is to pass the message inline with `-m "..."`. This fails or mangles content when messages contain Unicode characters (→, –, «, etc.) due to shell quoting and encoding interactions.

### The technique

Write the intended message to a temp file before starting the rebase, then reference it in the exec line with `-F`:

```bash
# Before the rebase — write message files
cat > /tmp/msg_fix.txt << 'EOF'
refactor(api): rename DispatchRule → Binding, CaseHubDefinition → CaseDefinition (Closes #37)
EOF

# In the GIT_SEQUENCE_EDITOR Python script — inject exec after target commit
for l in lines:
    out.append(l)
    if len(parts) >= 2 and parts[1].startswith("256f94c"):
        out.append('exec git commit --amend -F /tmp/msg_fix.txt\n')

# Run the rebase
GIT_SEQUENCE_EDITOR="python3 /tmp/editor.py" git rebase -i <base>
```

The `-F` flag reads the message from the file, bypassing all shell string handling. The heredoc `<< 'EOF'` (quoted sentinel) prevents shell expansion inside the message.

### Why this is non-obvious

Most documentation and examples for `git commit --amend` show the `-m "..."` form. The `-F` flag is documented but rarely demonstrated for rebase exec use. The failure mode with `-m` and Unicode is silent on some systems (message is mangled) or an error on others — neither gives a clear pointer toward `-F`.

### Alternatives considered

- `-m "..."`: fails silently with Unicode → characters on macOS zsh; shell quoting complexity grows with message length
- `EDITOR=true git commit --amend`: bypasses editing but relies on `EDITOR=true` being available; less portable
- Python's `subprocess` to write the message: works but more complex than necessary
