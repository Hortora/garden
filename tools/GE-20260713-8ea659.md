### Symptom

Tests fail after landing a cross-repo branch. Error messages show wrapper exception text ("All 3 attempts failed") instead of root cause messages ("bad state", "permanent", class names for null-message exceptions).

### Context

Session A (engine) creates a commit on a worker repo branch to parameterise `WorkerFunction<T>`. The commit also refactors adjacent catch blocks in `DefaultWorkerExecutor`, removing root cause unwrapping, null message fallback, and `InterruptedPolicyException` propagation — all deliberately added by a previous session's issue #8 work. The refactoring looks like cleanup in the context of the larger diff.

Session B (worker) reviews the branch for landing. The parameterisation is correct. The exception handling regression is buried in whitespace and indentation changes across the same file.

### What was tried (didn't work)

1. Reviewing the diff visually — the catch block changes blended into the larger parameterisation refactoring and indentation realignment
2. The cross-repo session ran its own tests on the branch — but against a different test configuration that didn't exercise the same failure paths

### Root cause

Cross-repo commits have no guardrails against regressing prior work in the target repo. The committing session has no context about *why* the existing code was written that way — it sees code to simplify, not a deliberate design decision from a previous session. The three specific regressions:
- Root cause unwrapping (`e.getCause()`) removed — wrapper message leaked to callers
- Null message fallback (`e.getClass().getName()`) removed — NPE returned null
- `InterruptedPolicyException` catch block removed — infrastructure interrupts swallowed as `Failed` instead of propagating

### Fix

Run the target repo's full test suite on the branch before landing cross-repo work. In this case, `mvn test` caught all three regressions immediately — 3 test failures with clear expected-vs-actual messages pointing at the exception handler.

### Alternatives considered

- Relying on CI after push — delays feedback by minutes and uses CI as a test runner
- Reviewing the diff more carefully — fragile; indentation changes mask semantic changes in adjacent code

### Why this is non-obvious

The commit message and diff title accurately describe the intended change (parameterisation). The exception handling changes look like incidental cleanup — shorter code, fewer catch blocks, simpler flow. A reviewer focused on the parameterisation correctness (which IS correct) naturally skims the "cleanup." The regression is semantic, not syntactic — the code compiles and runs, it just returns different error messages and silently swallows interrupts.
