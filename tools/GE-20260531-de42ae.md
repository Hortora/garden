## Context

Building an auto-commit system for AI agents: automatically opening a CaseHub commitment at the start of each agent turn (`before_tool_call` hook) and closing it at turn end (`agent_end` hook) — without the LLM needing to manage commitment state.

## The Problem with a Stack

The natural first design: maintain a stack of open commitment IDs per agent, push on `before_tool_call`, pop and close all at `agent_end`. This produces:

1. **Wrong granularity** — an agent calling 5 tools in one turn creates 5 commitments. The user sees one instruction; the ledger shows 5 tracked obligations. Watchdogs fire 5 times.
2. **LIFO close complexity** — a `Stack<commitmentId>` with LIFO close logic at `agent_end`.
3. **Recursive auto-commit** — when the LLM explicitly calls `casehub_done` (to close a manual commitment), `before_tool_call` fires and opens an *auto*-commitment for the `casehub_done` call itself. Auto-commit on commitment tools creates nested commitments.
4. **Exclusion list grows unpredictably** — read-only tools (like `casehub_status`) obviously shouldn't trigger auto-commit, but the set of tools that shouldn't commit is broader and changes over time.

## The Technique: Per-Turn Flag

Use a single optional commitment ID per agent (not a stack). On `before_tool_call`:

```typescript
if (this.turnCommitments.has(event.agentId)) return; // already open this turn
if (AUTO_COMMIT_EXCLUDED_TOOLS.has(event.toolName)) return;
// ... open commitment, store single ID
```

On `agent_end`:

```typescript
const commitment = this.turnCommitments.get(event.agentId);
if (!commitment) return; // no open commitment
this.turnCommitments.delete(event.agentId); // clear before done call
// ... close single commitment
```

The exclusion set covers ALL commitment lifecycle tools (commit, done, reject, checkpoint, escalate) plus read-only tools. This prevents recursive auto-commit when the LLM explicitly manages its own commitments.

## Why This Works

One user instruction = one agent turn = one commitment. The Watchdog fires once per turn, not once per tool. The ledger shows one entry per user action. The stack collapses to a `Map<agentId, CommitmentEntry>` — a flag, not a LIFO structure.

The exclusion set is stable: everything that IS the commitment system is excluded. New tools that aren't commitment tools don't need special handling.

## Why This is Non-Obvious

Developers reaching for auto-commit naturally think "each tool call is a unit of work" — leading to the per-tool-call stack. The insight that the right granularity is the *agent turn* (not the tool call) comes from the user's perspective: they sent one message and expect one accountability record. The recursion issue with commitment tools is easily missed until the first test where the LLM calls `casehub_done` and the log shows a new commitment being opened for the `casehub_done` call itself.
