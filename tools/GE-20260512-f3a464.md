## Behaviour

`ide_refactor_rename` positioned on a Java package declaration can rename within the same hierarchy level, but cannot perform a "Move Package" operation that changes the package depth or root.

Renaming `dev.claudony.server` → `dev.casehub.server` (same depth, different name) works.

Renaming `dev.claudony` → `io.casehub.claudony` (different root, different depth) does **not** work via `ide_refactor_rename`. The tool either renames only the leaf segment or fails to propagate across all subpackages.

## Context

The distinction in IntelliJ is between **Rename** (same package parent, new leaf name) and **Move** (change parent package, keep or change name). The MCP only exposes Rename semantics. Move Package is a separate IntelliJ refactoring action with no MCP equivalent.

## Fix

For cross-root package moves, use IntelliJ's UI directly:

1. Project view → expand to the package directory
2. Right-click the package → **Refactor → Move...**
3. Enter destination package (e.g. `io.casehub`)
4. Keep the package name (e.g. `claudony`)
5. Apply — IntelliJ updates all imports across all modules

Then fix any corruption from the Move Package bug (see GE-20260512-8c282a).

## Why undocumented

The `ide_refactor_rename` MCP tool description says "Safe rename — updates all references" without specifying that it only covers same-level renames. The distinction between Rename and Move is not surfaced in any MCP documentation. A developer needing a cross-root package move would naturally try `ide_refactor_rename` first and find it insufficient only after testing.
