### What
When smoothing A* waypoints with a line-of-sight check, use sub-tile sampling instead of
Bresenham. Step along the direction vector at intervals strictly less than the unit's movement
speed, checking `isWalkable()` at each sample.

### How

```java
private static boolean hasLos(TerrainGrid grid, Point2d from, Point2d to) {
    double dx   = to.x() - from.x();
    double dy   = to.y() - from.y();
    double dist = Math.sqrt(dx * dx + dy * dy);
    if (dist < 0.001) return grid.isWalkable((int) from.x(), (int) from.y());
    double nx = dx / dist;
    double ny = dy / dist;
    double step = movementSpeed * 0.8; // strictly less than speed
    for (double d = 0; d <= dist + step; d += step) {
        double t = Math.min(d, dist);
        if (!grid.isWalkable((int)(from.x() + nx * t), (int)(from.y() + ny * t))) return false;
    }
    return true;
}
```

The `+step` in the loop bound ensures the endpoint itself is always sampled.

### Why this is non-obvious
The standard LOS algorithm taught for grid pathfinding is Bresenham, which checks whether the
theoretical centre-to-centre line crosses wall tile boundaries. This is correct for ray-casting
but wrong for discrete-step movement: a unit moving at speed `s` per tick may land in a tile
that the Bresenham line passes through only at its boundary (sub-tile), never at its centre.
The result is a unit that "should" be able to move in a straight line but hits walls and gets
stuck in path-invalidation loops — with no error, because the wall enforcement correctly fires
on the fractional tile position.

The sampling step must be `< movement_speed`. At `0.8×` there is always at least one sample
inside any tile the unit could land on. Coarser than `1.0×` misses tiles; Bresenham misses
sub-tile crossings entirely.

### Performance note
Sub-tile sampling costs more than Bresenham (more iterations). For typical RTS map sizes
(64×64) and A* paths of 20–30 waypoints, the smoothing pass over the whole path runs
in <1 ms. Profile before optimising.
