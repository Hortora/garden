### Symptom
`ide_replace_text_in_file` runs indefinitely when the replacement text is large (~2KB+). No error, no timeout — the tool simply never returns. The MCP connection stays open but the tool must be stopped manually via `TaskStop`.

### Context
Refactoring a 2500-line Lit Web Component (`pages-table.ts`). Replacing a 50-line method body with a 60-line replacement. The search text matched correctly (verified by a subsequent smaller replacement on the same file). The replacement text was the only variable — reducing it to a few lines made the tool work normally.

### What was tried (didn't work)
1. Waiting — tool ran for 2+ minutes with no progress
2. Stopping and retrying — same result on retry
3. `ide_sync_files` after stopping — also hung, suggesting the MCP server was in a bad state

### Root cause
The IntelliJ MCP plugin's `ide_replace_text_in_file` implementation appears to have a performance cliff or blocking operation when processing large replacement strings. The exact threshold is approximately 2KB of replacement text. The tool does not have a built-in timeout.

### Fix
Use a Python script to perform the replacement directly on the filesystem:

```python
filepath = '/path/to/file.ts'
with open(filepath, 'r') as f:
    content = f.read()

# Find method boundaries by signature
start_idx = content.find('  private methodName(')
end_marker = content.find('\n\n  private nextMethod(', start_idx)

new_method = '''  private methodName(...) {
    // new implementation
  }'''

content = content[:start_idx] + new_method + content[end_marker:]
with open(filepath, 'w') as f:
    f.write(content)
```

After the Python replacement, call `ide_sync_files` to update IntelliJ's index.

### Why this is non-obvious
The tool works perfectly for small replacements. There is no error message, warning, or timeout — it just hangs silently. The natural assumption is that the tool is processing a complex diff, not that it has hit a size limit. The distinction between "search text too large" and "replacement text too large" is also non-obvious.

### Alternatives considered
- Breaking the replacement into multiple smaller `ide_replace_text_in_file` calls — fragile when the intermediate states are syntactically invalid
- Using `Edit` tool — blocked by project hooks enforcing IntelliJ-first editing
- Writing the entire file with `Write` — works but loses the surgical nature of the change
