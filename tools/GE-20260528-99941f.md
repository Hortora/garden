### Symptom

Documentation or methodology using "vertical slice" to mean "a delivery increment
that cuts through horizontal architecture layers" confuses readers who know VSA —
Vertical Slice Architecture — a code organisation pattern where all code for one
feature co-locates across layers in a single directory or file. Same metaphor,
entirely different domain.

### Context

When designing architecture documentation methodology (e.g. delivery planning
frameworks built on top of arc42), it is natural to describe a delivery increment
as a "vertical slice" — it literally cuts vertically through horizontal layers.
The problem: Vertical Slice Architecture (Jimmy Bogard, widely used in .NET and
growing in Java) already owns this term in the software architecture space.

### Root cause

"Vertical slice" has an established prior meaning — Vertical Slice Architecture —
that predates delivery planning usage of the term. A reader familiar with VSA reads
"vertical slice" and expects a code organisation pattern (feature co-location),
not a delivery planning concept.

VSA is primarily a .NET community pattern but is actively spreading into Java
codebases (2023–2025). Any team building methodology documentation may not encounter
the collision until a developer familiar with VSA reads their docs.

### Fix

Use a domain-neutral term instead of "slice" for delivery planning concepts.
"Chapter" (from Arc42Stories) works well — it carries no prior architecture pattern
meaning and fits naturally into a "living story" metaphor. Other alternatives:
"increment", "capability milestone", "delivery increment".

Reserve "slice" language explicitly for VSA if your documentation discusses both
code organisation and delivery planning.

### Why this is non-obvious

The metaphor is genuinely correct in both cases — a vertical cut through horizontal
layers — so the naming feels right until a VSA-familiar reader flags the ambiguity.
VSA is well-known in .NET but less universal in Java, so the collision is invisible
to many teams until it causes real confusion.
