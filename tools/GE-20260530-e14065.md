### What happened
A field validator banning BiDi control characters to prevent prompt injection implemented a standard check: reject codepoints 0–31 (C0), 127–159 (C1), U+200E/200F (LRM/RLM), U+202A–202E (LRE/RLE/PDF/LRO/RLO), and U+2066–2069 (LRI/RLI/FSI/PDI). Code review caught that U+061C was missing. The validator would have passed strings containing the Arabic Letter Mark.

### Root cause
U+061C (Arabic Letter Mark, ALM) was added in Unicode 6.3 as a BiDi control character. It is in the Arabic block (U+0600–U+06FF), not in any standard control character range (C0: 0x00–0x1F, C1: 0x80–0x9F). It has no visible glyph and controls BiDi text direction.

Standard documentation and implementations of "ban control characters" focus on C0/C1 ranges. The dedicated BiDi control ranges (LRM/RLM/LRE etc.) are more widely known. U+061C predates none of these — it was simply added later and sits in an unexpected block.

### Fix
Add explicit check for U+061C alongside other BiDi controls:

```java
private static boolean isBanned(final int cp) {
    if (cp <= 0x001F) return true;                      // C0 control chars
    if (cp >= 0x007F && cp <= 0x009F) return true;     // DEL + C1 control chars
    if (cp == 0x061C) return true;                      // ALM — Arabic Letter Mark (BiDi, Unicode 6.3)
    if (cp == 0x200E || cp == 0x200F) return true;     // LRM, RLM
    if (cp >= 0x202A && cp <= 0x202E) return true;     // LRE, RLE, PDF, LRO, RLO
    if (cp >= 0x2066 && cp <= 0x2069) return true;     // LRI, RLI, FSI, PDI
    if (cp == 0x200B || cp == 0xFEFF) return true;     // ZWSP, BOM/ZWNBSP
    if (cp == 0x200C || cp == 0x200D) return true;     // ZWNJ, ZWJ
    if (cp == 0x2028 || cp == 0x2029) return true;     // LINE SEP, PARA SEP
    return false;
}
```

### Why this is non-obvious
The Unicode BiDi control characters everyone knows (LRM, RLM, LRE, RLE, PDF, LRO, RLO, LRI, RLI, FSI, PDI) are all in the General Punctuation block (U+2000–U+206F). U+061C is in the Arabic block. No standard "control character" resource bundles it with the others. The Unicode BiDi Algorithm spec mentions it, but security-focused implementations typically enumerate the well-known ranges without reading the full BiDi spec for outliers.
