## `git cherry-pick --continue` applies all remaining queued commits, not just the conflicted one

**ID:** GE-20260416-a5e990
**Stack:** git 2.x
**Symptom:** Running `git cherry-pick A B C` stops on a conflict in A. After resolving and running `git cherry-pick --continue`, all three commits are applied (A, B, and C). Then running `git cherry-pick B C` again causes `CONFLICT (add/add)` because B and C were already applied by `--continue`.
**Context:** Multi-commit cherry-pick where the first commit in the sequence has a conflict.

### What was tried (didn't work)
- After `git cherry-pick --continue` successfully applied A, B, and C (all shown in output), the developer ran `git cherry-pick B C` thinking `--continue` only finished A → produces add/add conflicts on files already created by B and C

### Root cause
`git cherry-pick A B C` queues all three commits. Stopping on A's conflict does not cancel B and C — they remain queued. `git cherry-pick --continue` resumes the *entire* sequence from where it stopped, applying all remaining queued commits in order. The output log after `--continue` shows all applied commits (not just A), but this can be missed when reading quickly.

### Fix
After `git cherry-pick --continue`, check what was actually applied before cherry-picking more commits:

```bash
# After --continue, verify what's now on the branch
git log --oneline <branch> ^<base>

# Only cherry-pick commits that are NOT already in the log
```

Or use `--no-commit` per commit to stay in control:
```bash
git cherry-pick --no-commit A   # resolve conflict, then:
git cherry-pick --continue      # applies only A with --no-commit
git cherry-pick --no-commit B
git cherry-pick --no-commit C
git commit -m "..."             # one combined commit, or commit each
```

### Why this is non-obvious
`cherry-pick --continue` feels analogous to `rebase --continue` where you're advancing one step. But cherry-pick queues all commits upfront and `--continue` resumes the full queue — not just the one that conflicted. The terminal output after `--continue` lists all applied commits, but this is easy to miss when you're focused on the conflict resolution.
