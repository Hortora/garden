## Jekyll `permalink: pretty` silently breaks all relative image paths

**ID:** GE-20260413-66dbe0
**Stack:** Jekyll (all versions), GitHub Pages
**Symptom:** Images in Markdown docs load fine when pages use `.html` URLs but silently fail (broken image icon, no error) after switching to `permalink: pretty`. No build error, no 404 in CI — images simply don't appear in the browser.
**Context:** Any Jekyll site where Markdown pages use relative image references like `images/file.png` and the config sets `permalink: pretty` (or any directory-style permalink).

### What was tried (didn't work)
- Verified images existed on disk — they did
- Ran `jekyll build` — no warnings or errors
- Checked browser network tab — images requested at wrong URL e.g. `/user-guide/01-installation/images/file.png` instead of `/user-guide/images/file.png`

### Root cause
`permalink: pretty` outputs each page as `<path>/index.html` inside a subdirectory. The page is served at `/user-guide/01-installation/` (trailing slash). Relative `images/file.png` resolves relative to that directory — `/user-guide/01-installation/images/file.png` — which doesn't exist. The images live at `/user-guide/images/file.png` (one level up).

### Fix
Chain Liquid `replace` filters in the layout template to prepend `../` to relative paths — no source file changes needed:

```liquid
{{ content
  | replace: 'src="images/', 'src="../images/'
  | replace: 'href="01', 'href="../01'
  | replace: 'href="02', 'href="../02'
  ...
  | replace: '.md"', '/"'
}}
```

This runs at build time and corrects all relative references in one pass. Alternative: set an explicit per-page `permalink: /section/:basename` in `_config.yml` defaults to keep pages at the same depth as their assets.

### Why this is non-obvious
`permalink: pretty` sounds like a cosmetic change. Nothing in Jekyll's build output warns that relative paths are now wrong. The symptom (broken images) is identical to "image file missing" — the path mismatch is only visible in the browser's network tab. Developers typically fix the images by hunting for missing files rather than reconsidering permalink style.
