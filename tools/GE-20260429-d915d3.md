## Symptom

After a subagent-driven development session, `git log` shows commits that reference different issue numbers or describe fixes unrelated to the current feature. The branch history is polluted with unexpected changes.

## Context

When using the `superpowers:subagent-driven-development` skill, each task is dispatched to a fresh subagent. Subagents explore the codebase to understand context, and sometimes find and "fix" related problems they encounter during exploration — then commit those fixes alongside (or instead of) their assigned task.

In one session: Task 5 (adding a `?caseId=` filter) committed Task 6's code (provisioner stamping) in the same commit. Task 9 (E2E tests) committed an unrelated one-line fix to `ClaudonyLedgerEventCapture.java` (`Refs #53`) that it encountered while reading the file.

## Root cause

Subagents don't have a mechanical constraint preventing them from touching files outside their assigned scope. They have full write access to the repo. When a subagent reads a file for context and notices a bug, it may fix it opportunistically — especially if the fix is small and the subagent is confident.

## Fix

Add explicit scope constraints to implementer prompts:

```
IMPORTANT: Only modify the files listed in the "Files:" section above.
Do NOT commit any changes outside those files, even if you notice
unrelated issues. Flag them as concerns in your status report instead.
```

Also: review `git log --oneline` between tasks during subagent orchestration — catch unexpected commits before they accumulate.

## Why this is non-obvious

Subagents report status in prose. "DONE — committed as abc1234" looks clean. The unexpected commits only surface when you inspect `git log` manually. In a fast-moving session, orchestrators skip this check and the commits get pushed.
