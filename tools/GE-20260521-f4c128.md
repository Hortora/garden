### Symptom
`git push` to a newly-created GitHub repo fails with `Permission denied (publickey)` /
`fatal: Could not read from remote repository` — even though `gh repo create` just
succeeded moments before.

### Context
`gh repo create` creates the GitHub repo and returns the HTTPS URL, but if a SSH remote
URL (`git@github.com:owner/repo.git`) is added manually or via a script, the push fails
on any machine where SSH keys aren't configured for GitHub.

### What was tried
Push via SSH remote: failed. Checked `gh repo view` — repo existed correctly.

### Root cause
SSH key authentication is a per-machine setup step that `gh` doesn't validate before the
push. The remote URL format (`git@...` vs `https://...`) determines which auth mechanism
is used — there's no automatic fallback.

### Fix
Switch the remote to HTTPS before pushing:
```bash
git remote set-url origin https://github.com/owner/repo.git
git push -u origin main
```
HTTPS uses GitHub CLI's stored credentials (set up once via `gh auth login`) and works
on any machine where `gh` is authenticated, regardless of SSH key configuration.

### Why this is non-obvious
`gh repo create` succeeds and the repo exists — so the failure appears to be a push
problem, not an auth problem. The SSH URL vs HTTPS URL distinction isn't surfaced during
repo creation, and the error message names SSH specifically but doesn't suggest HTTPS as
a fallback.
