### Symptom

Compilation fails with `cannot find symbol: class X`. GitHub Packages API reports the
SNAPSHOT version's `updated_at` as months ago. Conclusion: the artifact isn't being
published. Wrong fix applied (excluding module from CI).

### Context

Multi-repo Maven project using `0.2-SNAPSHOT` dependencies published to GitHub Packages.
Upstream repo refactored package structure (moved classes from `runtime.model` to
`api.model`).

### What was tried (didn't work)

1. Checked `gh api "/orgs/.../packages/maven/.../versions" --jq '.[0].updated_at'` —
   returned a date months in the past
2. Concluded the artifact was stale and not being published
3. Excluded the module from CI — wrong fix, masked the real problem

### Root cause

Two compounding misdiagnoses:

1. **API metadata is misleading:** The version-level `updated_at` reflects when the
   version entry (`0.2-SNAPSHOT`) was *created*, not when the latest timestamped JAR
   was uploaded. New SNAPSHOTs upload daily as timestamped files under the same version
   entry — the metadata never changes.

2. **Error message misread:** `cannot find symbol: class LedgerEntry` means Maven
   *found the JAR* but the class isn't at the expected package path. This is a package
   migration, not a missing artifact. The real signal:
   `cannot find symbol` ≠ `could not resolve artifact`.

### Fix

When diagnosing SNAPSHOT issues:

1. Check the actual artifact files under the version, not the version-level metadata
2. Read the compilation error carefully:
   - `cannot find symbol` → class moved/renamed — inspect with `javap -p -cp <jar> <fqcn>`
   - `could not resolve artifact` → genuinely missing from the registry
3. Use `javap` or `jar tf` to verify what's actually in the downloaded JAR before
   concluding the artifact is stale

### Why this is non-obvious

The `updated_at` field name strongly implies recency. The natural conclusion — "last
updated 2 months ago, so the artifact is stale" — is reasonable but wrong. Combined
with a compilation error that *looks* like a missing dependency, the misdiagnosis is
self-reinforcing: stale metadata confirms the theory, so the actual error message
(which points at a package migration) is never examined closely.

### Alternatives considered

- Checking the CI publish logs of the upstream repo — would have shown recent successful
  publishes and immediately disproved the "not publishing" theory
- Running `mvn dependency:resolve -U` with debug output — shows the actual timestamped
  JAR being downloaded, proving the artifact exists
