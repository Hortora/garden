## What it is

Use `actions/cache/restore@v4` and `actions/cache/save@v4` as separate steps rather than the combined `actions/cache@v4`. This gives explicit control over *when* the cache is written and *what* is written — independent of whether the job succeeded or failed.

## When to use it

Any workflow where the cache contents should depend on per-step outcomes rather than overall job success. The canonical case: an incremental build that tracks which modules built successfully. Failed modules should NOT have their SHA recorded (so the next run retries them); succeeded modules should.

## The pattern

```yaml
# Restore at the start — gets the most recent matching cache
- uses: actions/cache/restore@v4
  with:
    path: .incremental-state
    key: state-${{ github.ref_name }}-${{ github.run_id }}
    restore-keys: |
      state-${{ github.ref_name }}-

# ... build steps run, some succeed, some fail ...

# Save at the end with if: always() — runs even when steps failed
# Write only the SHA for modules that succeeded; keep previous SHA for failures
- name: Save state
  if: always()
  run: |
    record() {
      local name=$1 cur=$2 outcome=$3 prev=$4
      [ "$outcome" = "success" ] && echo "${name}=${cur}" || echo "${name}=${prev}"
    }
    { record ledger "$SHA_L" "$OUTCOME_L" "$PREV_L"; ... } > .incremental-state/shas.txt

- uses: actions/cache/save@v4
  if: always()
  with:
    path: .incremental-state
    key: state-${{ github.ref_name }}-${{ github.run_id }}
```

## Why the combined action can't do this

`actions/cache@v4` auto-saves in a post-step that runs after the job completes. You cannot control what is written to disk before that post-step runs based on individual step outcomes — by then, all steps have finished. The separate save action runs as an explicit step where you have full access to `steps.<id>.outcome` and can decide exactly what state to persist.

## Why non-obvious

The GitHub Actions docs present `actions/cache` as the standard pattern. The separate restore/save actions are documented but presented as advanced usage. The motivation for splitting them — outcome-dependent state — is not mentioned in the docs; it's only obvious once you've hit the limitation of the combined action in an incremental build scenario.
