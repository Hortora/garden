### Symptom

Running `ls design/` shows only `EPIC-CLOSED.md` and `JOURNAL.md` — the directory appears to have no `.meta` file. But the `.meta` file IS there and contains critical workflow state.

### Context

The cc-praxis work lifecycle stores epic state in `design/.meta` (a hidden dotfile). The `work-end` skill checks for this file to determine branch state. When inspecting the `design/` directory during debugging or handover review, bare `ls` silently skips it.

### What was tried (didn't work)

`ls design/` — returned only visible files. Concluded `.meta` was absent, causing confusion about whether the branch had been properly initialised or whether state was orphaned.

### Root cause

POSIX `ls` omits filenames starting with `.` by default. This is correct Unix behaviour, but `.meta` and `.pause-stack` are hidden by convention in the cc-praxis workflow, making them systematically invisible to the most natural inspection command.

### Fix

```bash
ls -la design/
```

Or specifically for the state files:

```bash
ls -la design/.meta design/.pause-stack 2>/dev/null
```

### Why this is non-obvious

`ls` hiding dotfiles is basic Unix, so a skilled developer knows the rule. The non-obvious part is that the cc-praxis workflow deliberately uses dotfiles for state — and the natural "check the directory" command gives a false picture of what's there. The mismatch between "I'm checking the directory" and "I'm only seeing half of it" is easy to miss when you're focused on the workflow, not the shell conventions.
