## Context

Testing that elements (minerals, geysers, creep, buildings) are genuinely visible to a human in a Three.js WebGL scene. Standard approaches — checking scene graph counts, asserting NDC projection, or using `@QuarkusTest` Playwright tests — all pass even when the element is 2 pixels wide, buried in terrain geometry, or using the wrong colour.

The only test that proves human visibility: sample the actual rendered pixel and assert its colour.

## Technique

Three components working together:

### 1. Renderer setup

```javascript
renderer = new THREE.WebGLRenderer({ antialias: true, preserveDrawingBuffer: true });
```

`preserveDrawingBuffer: true` prevents the GPU from clearing the drawing buffer between frames, making pixel readback possible.

### 2. `window.__test` helpers

```javascript
// Aims camera at the first element of a given type and returns its screen position
focusOnFirstMineral: () => {
  const sp = mineralMeshes.values().next().value;
  if (!sp) return null;
  camTarget.set(sp.position.x, 0, sp.position.z);
  updateCamera();
  const v = sp.position.clone().project(camera);
  const sz = renderer.getSize(new THREE.Vector2());
  return { x: Math.round((v.x+1)/2*sz.width), y: Math.round((-v.y+1)/2*sz.height) };
},

// Samples the actual rendered WebGL pixel
samplePixel: (x, y) => {
  const c = document.createElement('canvas');
  c.width = 1; c.height = 1;
  const ctx = c.getContext('2d');
  ctx.drawImage(renderer.domElement, Math.round(x), Math.round(y), 1, 1, 0, 0, 1, 1);
  const d = ctx.getImageData(0, 0, 1, 1).data;
  return { r: d[0], g: d[1], b: d[2], a: d[3] };
},
```

One `focusOnFirstX()` function per element type (mineral, geyser, creep, building). Each one sets `camTarget` to the element's world position and returns its screen coordinates.

### 3. Integration test against the real jar

```java
class ReplayVisualizerIT {
    static Process serverProcess;
    static Browser browser;

    @BeforeAll
    static void start() throws Exception {
        serverProcess = new ProcessBuilder(
            "java", "-Dquarkus.http.port=8082",
            "-jar", "target/quarkus-app/quarkus-run.jar"
        ).directory(new File(".")).start();
        // poll /q/health/ready until 200
        waitForServer(30);
        browser = Playwright.create().chromium()
            .launch(new BrowserType.LaunchOptions().setHeadless(true));
    }

    @Test
    void mineralPixelIsCyanInRealReplay() throws Exception {
        var page = browser.newPage();
        page.navigate("http://localhost:8082/visualizer.html");
        page.waitForFunction("() => window.__test?.wsConnected?.() === true");
        page.waitForFunction("() => window.__test.mineralCount() > 0");
        page.waitForTimeout(200); // one render frame

        var pos = (Map<?,?>) page.evaluate("() => window.__test.focusOnFirstMineral()");
        page.waitForTimeout(200); // camera and re-render settle

        var pixel = (Map<?,?>) page.evaluate(
            "() => window.__test.samplePixel(" + pos.get("x") + ", " + pos.get("y") + ")");
        int r = ((Number)pixel.get("r")).intValue();
        int b = ((Number)pixel.get("b")).intValue();

        assertThat(b).as("mineral pixel must be cyan (B>R), not sandy terrain (R>G>B)")
            .isGreaterThan(r);
    }
}
```

**Colour assertions by element type:**
- Minerals (`0x44ccff`): `B > R`
- Geysers (`0x00dd66`): `G > R`
- Creep (`0x9030c0`): `B > G`
- Enemy buildings: check against the tint colour used (`0xff4422` → `R > G`)

## Why this is non-obvious

The standard approach is `waitForFunction("() => window.__test.mineralCount() > 0")` — a scene graph check. This passes even when the element is 2px wide, buried in terrain, or the wrong profile is being tested. Pixel sampling against the real jar is the only way to prove the element is visible to a human. Most developers would not think to start the production jar as a subprocess and sample WebGL pixels.

### Why this fix

`@QuarkusTest` Playwright tests always run in mock profile (different map size, camera distance, CDI beans). They cannot test the real replay path. Starting the real jar is the only way to test what the user actually sees.
