## Symptom

After merging an epic branch to main without running `/epic` close (e.g. via `finishing-a-development-branch`), `design/.meta` ends up on main. Running `/epic` detects "Not on epic branch, `.meta` exists" — but there is no route to Workflow B (close the epic). Instead it falls into the orphaned-cleanup path inside Step A0, which is part of the *start* workflow, not the close workflow.

The issue is only visible if you want to run the full epic close (artifact promotion, journal merge, issue closure) after the code is already on main.

## Context

The epic skill's detection table has three states:

| State | Routes to |
|-------|-----------|
| Not on epic branch, no `.meta` | Offer to start new epic |
| On epic branch, `.meta` exists | Ask: close or start? |
| On epic branch, no `.meta` | Warn: incomplete setup |

"Not on epic branch, `.meta` exists" is a fourth state not in the table. The skill defaults to the start workflow's orphaned-cleanup check rather than offering close.

## Root cause

The skill detection only routes to Workflow B when currently on the epic branch. If `git merge epic-foo` was run directly on main (bypassing the epic close), the branch is gone and the skill has no recovery path.

## Fix

Manual workaround: temporarily recreate the epic branch from the last commit before merge, switch to it, then run `/epic`. Alternatively, run the close steps manually:
1. Close the GitHub issue: `gh issue close <N>`
2. Archive plans: `mv plans/<plan>.md plans/attic/<epic-name>/`
3. Remove `.meta`: `git rm design/.meta && git commit`
4. Run `java-update-design` to sync DESIGN.md with journal content

## Why this is non-obvious

The merge-then-close sequence looks natural for projects using `finishing-a-development-branch`, which merges code but does not close the epic. The two are distinct operations with no guard preventing you from running one without the other. The resulting state (code on main, `.meta` orphaned, issue open) is valid but unrecoverable via the skill.
