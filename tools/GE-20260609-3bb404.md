### Symptom
A regex pattern meant to match one markdown table row instead returns the separator row merged with the next data row. The parser reports wrong keys and values with no error.

### Context
Writing a Python markdown table parser using `re.finditer()` with a pattern like:

```python
re.finditer(r"^\|\s*(.+?)\s*\|\s*(.+?)\s*\|", text, re.MULTILINE)
```

The separator row `|----------+-------------|` does not match on its own — but with `\s*` between groups, the regex spans the trailing newline and merges it with the next row.

### What was tried
Pattern seemed correct by inspection. Only visible when running interactively and printing `row.groups()` — the merged result `('----------+-------------|', 'adr')` reveals the cross-line capture.

### Root cause
`\s*` matches `\n` in Python. In MULTILINE mode, `^` and `$` anchor to line boundaries but character classes like `\s` still cross them. The separator's trailing `|` triggers `\s*` to consume the newline, then `\|` matches the `|` that opens the next row.

### Fix
Replace `.+?` and `\s*` between cell boundaries with `[^|\n]+` to prevent crossing line boundaries:

```python
# Before (crosses newlines):
re.finditer(r"^\|\s*(.+?)\s*\|\s*(.+?)\s*\|", text, re.MULTILINE)

# After (stays on one line):
re.finditer(r"^\|\s*([^|\n]+?)\s*\|\s*([^|\n]+?)\s*\|", text, re.MULTILINE)
```

### Why this is non-obvious
`\s*` between structural markers reads as "optional whitespace" — not "optional whitespace including newlines that cross into the next row." MULTILINE mode makes `^` and `$` line-aware, which gives a false sense that the whole match is line-scoped.
