## Symptom

`git stash pop` returns exit 1 with:

```
error: Your local changes to the following files would be overwritten by merge:
    <file>
Please commit your changes or stash them before you merge.
Aborting.
On branch main
...
The stash entry is kept in case you need it again.
```

## Context

Ran `git stash` before a rebase to clear unstaged changes. The rebase modified a file that was also in the stash. After the rebase, `git stash pop` conflicts and aborts.

## Root Cause

The rebase rewrote the working tree for a file the stash also touched. `stash pop` treats this as an unresolvable conflict and aborts rather than attempting a three-way merge — the stash entry remains intact.

## Fix

Drop the stash:

```bash
git stash drop
```

The rebase already incorporated the change for that file. The stash content for it is superseded. No work is lost.

## Why This Is Non-Obvious

Exit 1 + "Aborting" looks like data loss. The reassuring message "The stash entry is kept in case you need it again" appears at the end of a multi-line error block and is easy to miss. Most developers will retry, panic, or attempt a force-pop before reading it carefully.
