## Technique

When you have 2–5 independent implementation tasks in different repositories, dispatch them as parallel Claude Code subagents with `isolation: "worktree"`. All tasks complete in the time of the slowest one rather than the sum of all.

## When to use

- Tasks are in different repos (no shared file state)
- Each task is fully self-contained (no output from one feeds into another)
- Each task is non-trivial enough to benefit from isolation (>15 min each)

Do NOT use when tasks share files, when one task's output is another task's input, or when tasks need to coordinate.

## How it works

```
Agent(
  description: "Task A in repo X",
  prompt: "... fully self-contained prompt with all context ...",
  isolation: "worktree"
)
Agent(
  description: "Task B in repo Y", 
  prompt: "... fully self-contained prompt with all context ...",
  isolation: "worktree"
)
Agent(
  description: "Task C in repo Z",
  prompt: "... fully self-contained prompt with all context ...",
  isolation: "worktree"
)
# All three dispatched in a single message — run concurrently
```

## Demonstrated result

Three implementation tasks across three repos (quarkus-qhorus, casehub-engine, quarkus-ledger):
- Each estimated 20–30 min sequentially
- Dispatched in parallel with worktree isolation
- All completed in ~8 minutes total

The tasks were: writing `LedgerAttestation` on terminal commitment outcomes, aligning `PropagationContext.traceId` with OTel span, and creating `ActorTypeResolver` utility. All fully independent, all in different repos.

## Critical prompt requirements

Each agent prompt must be **fully self-contained** — the agent has no context from the parent session:
- Include the exact file paths to read/modify
- Include the build commands (don't assume the agent knows the project)
- State what "done" looks like (commit message, test requirement)
- Include the issue number to reference in the commit

Vague prompts produce vague results. The quality of the briefing determines the quality of the output.

## Why this is non-obvious

The natural approach is sequential: implement one, review, implement the next. Parallel dispatch feels risky because you can't supervise each agent. The worktree isolation removes the risk — each agent works in its own branch copy, changes don't interfere, and you review the results after all complete. The `isolation: "worktree"` parameter is the key — without it, agents writing to the same files would conflict.
