### Symptom

`<retryFailedDeploymentCount>10</retryFailedDeploymentCount>` is configured. All 10 retries complete within ~5 seconds and all fail with the same error. The retry configuration appears to have no effect.

### Context

Deploying a SNAPSHOT artifact to GitHub Packages for the first time. The repository namespace may not be fully indexed yet, causing the immediate verification GET to fail. Or the `distributionManagement` URL is wrong (see related gotcha).

### What was tried

Setting `retryFailedDeploymentCount` to 10 expecting it to retry with backoff after transient failures.

### Root cause / Undocumented behaviour

`retryFailedDeploymentCount` retries the complete deploy cycle (upload + verify) with **zero delay** between attempts. This is not documented anywhere in the Maven deploy plugin documentation. The plugin source code shows no sleep or backoff between retry iterations.

For GitHub Packages first-upload scenarios (where the registry needs time to index the artifact before it becomes queryable), rapid-fire retries all fail within the same indexing window and provide no benefit.

### Fix

There is no built-in delay option in `maven-deploy-plugin`. Workarounds:

1. **Add a sleep step before deploy in CI** — gives GitHub Packages time to initialize the namespace after a prior step that created it.

2. **Run deploy twice in the workflow** — first run initializes the namespace (even if it fails), second run succeeds because the registry now exists.

3. **Fix the root cause** — ensure the `distributionManagement` URL matches the actual repo name exactly (the most common cause of this pattern).

### Why this is non-obvious

The `retryFailedDeploymentCount` parameter name implies retry-with-recovery behaviour. The absence of any delay is not documented. Developers naturally assume it provides exponential backoff like most retry mechanisms.
