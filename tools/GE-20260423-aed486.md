## Symptom

After running `git filter-repo` (even on a run that fails partway through), subsequent `git push` commands fail with:

```
fatal: 'origin' does not appear to be a git repository
```

The remote has silently disappeared.

## Context

`git filter-repo` removes all configured remotes as a safety measure to prevent accidental pushes of rewritten history to the wrong remote. This happens on every run — including runs that abort with an error. It is documented in the manual under "Why is my origin removed?" but easy to overlook when working under time pressure.

## Root cause

By design. `git filter-repo` treats any configured remote as a risk vector for accidentally pushing bad history, so it aggressively removes them before rewriting.

## Fix

After every `git filter-repo` run, re-add the remote before pushing:

```bash
git remote add origin https://github.com/<owner>/<repo>.git
# then push
git push --force origin <branch>
```

## Why this is non-obvious

The error message (`'origin' does not appear to be a git repository`) sounds like a URL or authentication problem, not a "your remote was deliberately deleted" problem. The connection between running filter-repo and losing the remote is not obvious, especially on a failed run where you didn't expect any state changes.

### Why this fix

Re-adding the remote is the only option — filter-repo provides no flag to preserve remotes. The design is intentional.
