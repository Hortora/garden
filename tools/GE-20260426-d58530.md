### Symptom

`mvn deploy` uploads artifacts successfully (HTTP 200/201 on every PUT) but then immediately fails with:

```
[WARNING] Encountered issue during deployment: Failed to deploy artifacts:
Could not find artifact io.casehub:api:pom:0.2-20260426.164523-1 in github
(https://maven.pkg.github.com/casehubio/casehub-engine)
[INFO] Retrying deployment attempt 2 of 10
```

All 10 retries fail. Maven exits with BUILD FAILURE. The CI run is red.

### Context

`distributionManagement` in `pom.xml` pointed to a GitHub repo name that did not exist:

```xml
<url>https://maven.pkg.github.com/casehubio/casehub-engine</url>
```

The actual GitHub repository was named `casehubio/engine`, not `casehubio/casehub-engine`.

### What was tried

- Verified GITHUB_TOKEN had `packages: write` permission — it did
- Checked `server-id` in `actions/setup-java` matched `<id>github</id>` — it did
- Increased `retryFailedDeploymentCount` — all retries failed immediately
- Tried running `mvn deploy` locally with credentials — same failure

### Root cause

GitHub Packages accepts PUT requests to any URL path under `maven.pkg.github.com/ORG/` regardless of whether the referenced repository exists. It returns HTTP 200/201, making the upload appear successful. However, the package is created in a "phantom" namespace that is not queryable — it is associated with a non-existent repository and cannot be retrieved via GET.

Maven's deploy plugin verifies the upload by immediately GETting the artifact. This GET fails (404 or equivalent), causing the misleading "Could not find artifact" error even though the PUT succeeded.

### Fix

Correct the `distributionManagement` URL to match the actual GitHub repository name exactly:

```xml
<distributionManagement>
  <repository>
    <id>github</id>
    <url>https://maven.pkg.github.com/OWNER/ACTUAL-REPO-NAME</url>
  </repository>
  <snapshotRepository>
    <id>github</id>
    <url>https://maven.pkg.github.com/OWNER/ACTUAL-REPO-NAME</url>
  </snapshotRepository>
</distributionManagement>
```

The URL must match the repository name exactly as it appears on GitHub, including case.

### Why this is non-obvious

The upload succeeds — HTTP 200/201 is returned — so the natural assumption is that the artifact reached GitHub Packages. The failure looks like a transient retrieval issue, a permissions problem, or a GitHub Packages eventual-consistency lag. Nothing in the error or the logs indicates the repository name is wrong. The real cause is only discoverable by cross-referencing the URL in `distributionManagement` against the actual GitHub repository name.
