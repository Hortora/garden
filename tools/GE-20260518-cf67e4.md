## Symptom

Feature enforcement code (e.g. field validation, REST surface changes, mapper updates) is silently absent after merging via `git cherry-pick -X ours`. Service-layer code in non-conflicting files works correctly. REST endpoints and response types behave as if the feature was never implemented. No git warning, no compile error, tests pass until the specific feature path is exercised.

## Context

Two concurrent epic branches each modified overlapping files (e.g. `WorkItemMapper.java`, `WorkItemResponse.java`, `WorkItemTemplateResource.java`). When cherry-picking the second branch onto main after the first had already merged, conflicts arose. Using `git cherry-pick -X ours` to resolve conflicts kept the first branch's version for all conflicting files — silently discarding the second branch's additions to those files.

The second branch's service-layer changes (in non-conflicting files like `WorkItemService.java`) survived. The REST surface changes (in conflicting files) were lost. The result: business logic enforced data correctly at the service layer but the data never reached the response — `excludedUsers` was validated on write but always returned `null`.

## What was tried (didn't work)

- Reading IntelliJ's file view (showed cached pre-fix state — see separate entry)
- Checking that service-layer enforcement code was present (it was — wrong layer)
- Running unit tests for the service layer (passed — tests don't cover the response mapping)

## Root cause

`-X ours` is a **conflict resolution strategy**, not a merge strategy. It tells git: "when a hunk conflicts, always take our side." For any file with a single conflicting hunk, the entire file's conflict resolution favours the existing branch — including all the additions from the cherry-picked commit in that file. The cherry-picked commit's changes to those files are silently discarded.

## Fix

Never use `-X ours` or `-X theirs` when merging feature branches where both branches genuinely added new capabilities to the same files. Instead:

1. Accept conflicts and resolve them manually, keeping **both** sets of changes.
2. After the cherry-pick, immediately run a full test suite for the feature being merged — not just unit tests for the service layer.
3. For concurrent epics, consider sequential merge ordering (merge one, then rebase the other on top) rather than cherry-picking.

**Detection:** After any merge that used `-X ours`, run `git log main..<branch> --oneline` on the project repo to verify the commit list. Then inspect key files in the merged commit with `git show <commit> -- <file>` to confirm all additions are present.

## Why this is non-obvious

`-X ours` sounds like "use our version for conflicts" — developers expect it to apply to the *conflict markers*, not to silently drop entire feature additions from the cherry-picked commit. The strategy operates at the hunk level: if any part of a file conflicts, the entire conflicting region takes "our" side, which can include discarding new code from the other branch that happened to be near the conflict. No warning is emitted. The commit appears in `git log`, the file appears modified, but the feature code is gone.

### Alternatives considered

- Manual conflict resolution per file: correct but tedious at scale (40+ files)
- `git merge` instead of `git cherry-pick`: would have shown all conflicts explicitly but requires branch reorganisation
