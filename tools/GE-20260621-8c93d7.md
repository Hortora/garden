### Symptom
`git stash pop` after switching from an epic branch to main produces:
```
CONFLICT (content): Merge conflict in HANDOFF.md
```
The stash pop succeeds for all other files; only HANDOFF.md conflicts.

### Context
Workspace repo using a handover methodology where `HANDOFF.md` is written and
committed on both the epic branch (session-end snapshot) and main (session-start
update). When you stash uncommitted workspace changes on the epic branch, switch
to main, and pop — git tries to apply the epic branch's stash against main's
different `HANDOFF.md`, producing a 3-way merge conflict.

### What was tried (didn't work)
`git stash` → `git checkout main` → `git stash pop` — conflict on HANDOFF.md.

### Root cause
`HANDOFF.md` exists in diverged form on both branches. Stash records the delta
from the epic branch's HEAD. When popped on main, git treats main's `HANDOFF.md`
as the base and the stash's HANDOFF.md as a patch — they've diverged, so it
conflicts.

### Fix
Cherry-pick specific commits to workspace main instead of stash/pop:
```bash
git -C <workspace> checkout main
git -C <workspace> cherry-pick <commit-sha>   # the specific change to land on main
```
If you're already in the conflict state, resolve with `--theirs` (take the stash
version) then drop the stash:
```bash
git checkout --theirs HANDOFF.md
git add HANDOFF.md
git stash drop
```

### Why this is non-obvious
`git stash` appears to be a neutral save/restore — the conflict source is invisible.
The setup requires knowing that `HANDOFF.md` is written independently on both branches,
which is a methodology detail not visible from the git history.
