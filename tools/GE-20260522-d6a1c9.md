### Symptom

After physically moving a project repo to a new path (e.g. `~/claude/quarkmind` → `~/claude/casehub/quarkmind`), IntelliJ either cannot find the project in Recent Projects, shows it greyed out, or opens to an error because the project cannot be located — even after updating what seems like the obvious place.

### Context

Occurs any time a project directory is relocated on disk: repo reorganisation, moving into a monorepo or subdirectory, renaming a parent folder. IntelliJ stores project location in three separate files, none of which update automatically.

### What was tried (didn't work)

Updating only `recentProjects.xml` — the project appeared in Recent Projects but still failed to open (trust error). Updating only `trusted-paths.xml` — the project still didn't appear in the recent list.

### Root cause

IntelliJ records the project path in three independent files:

1. `~/Library/Application Support/JetBrains/IntelliJIdea*/options/recentProjects.xml` — the Recent Projects list entry (key attribute of the `<entry>` element)
2. `~/Library/Application Support/JetBrains/IntelliJIdea*/options/trusted-paths.xml` — the trust whitelist entry (key attribute of the `<entry>` element in `TRUSTED_PROJECT_PATHS`)
3. `.idea/workspace.xml` inside the project — `last_opened_file_path` in the `PropertiesComponent` JSON

All three must be updated together. None of them update automatically when the directory moves.

### Fix

```bash
# Find the IntelliJ options directory (pick the active version)
ls ~/Library/Application\ Support/JetBrains/

# 1. recentProjects.xml — change the entry key
#    Find: <entry key="$USER_HOME$/old/path">
#    Replace: <entry key="$USER_HOME$/new/path">

# 2. trusted-paths.xml — change the entry key
#    Find: <entry key="$USER_HOME$/old/path" value="true" />
#    Replace: <entry key="$USER_HOME$/new/path" value="true" />

# 3. .idea/workspace.xml inside the project — change last_opened_file_path
#    Find: "last_opened_file_path": "/old/path/some-file.java"
#    Replace: "last_opened_file_path": "/new/path/some-file.java"

# Then restart IntelliJ for changes to take effect.
```

`$USER_HOME$` is IntelliJ's token for the home directory — keep it as-is, don't expand to the literal path.

**Note:** If IntelliJ is running when you make these edits, it may overwrite them on shutdown if it writes its own state for the open project. Make the edits while the affected project is closed in IntelliJ.

### Why this is non-obvious

There is no single "project location" setting. The three files serve different concerns (recently-opened list, security trust, per-project state) and are maintained independently. The trust check (`trusted-paths.xml`) is what produces the security dialog on open — so updating only `recentProjects.xml` gets the project into the list but opening it triggers a trust error at the new path. The `.idea/workspace.xml` entry is a cosmetic last-opened-file hint that doesn't affect opening but causes a missing-file warning if left pointing to the old path.
