### Symptom

Used `git diff --name-only main..$b -- blog/` to find blog entries on closed branches that hadn't reached main. Got 18 "missing" entries on an old epic branch — all of which were already on main.

### Context

Auditing closed workspace branches for blog entries written during epics that never made it to the main workspace branch. Each branch had diverged from main weeks earlier; main had accumulated many new blog entries since.

### What was tried (didn't work)

`git diff --name-only main..$b -- blog/` — intended to show files on the branch not on main. Instead returned main's newer files as if they were missing from the branch.

### Root cause

`git diff A..B` produces the diff needed to go *from A to B*. When branch `$b` is old and missing files that exist on `main`, those files appear as *deletions* in the diff (present in A/main, absent in B/branch). `--name-only` shows file names for all changed files without indicating direction — so main's newer files flood the output alongside any genuine branch-only files.

### Fix

Use commit-graph traversal instead of diff:

```bash
# Commits on branch not reachable from main, filtered to a path
git log main..$b -- blog/
```

This correctly isolates commits *written on the branch* after it diverged — regardless of how many files main has accumulated since. An empty result means no branch-only content; a non-empty result lists only genuine branch commits.

### Why this is non-obvious

`git diff main..branch` reads naturally as "what's on the branch that main doesn't have" — but diff is bidirectional and `--name-only` strips the add/delete direction. The commit-range query (`git log`) is the right tool for "what content exists on this branch that hasn't reached main", not diff.
