## Symptom

A Maven build has a known, stable set of failures. You make source changes to your project (unrelated to the failing tests). The next build reveals new `AbstractMethodError` or `NoSuchMethodError` — an implementation class doesn't implement an interface method that exists in the API JAR. The error looks like your changes broke a contract you didn't touch.

## Root Cause

IntelliJ runs background builds for all open projects in the workspace. When a sibling project's build completes, `mvn install` runs and installs a fresh SNAPSHOT JAR into `~/.m2/repository`. Maven resolves SNAPSHOT dependencies to the "latest" locally-installed JAR on the next invocation — silently picking up the new version. If the sibling project's API and implementation JARs are built at different times (API updated first, implementation not yet rebuilt against the new API), the local `.m2` cache holds an inconsistent pair. Your project picks up the API JAR from one build and the implementation JAR from an earlier build (or vice versa).

The trigger for your project resolving the new JAR is a Quarkus augmentation cache invalidation — any source file change causes Quarkus to re-augment, which in turn re-resolves all SNAPSHOT dependencies from the local Maven cache.

## Diagnosis

Check JAR timestamps in `~/.m2/repository` for the affected artifact:

```bash
ls -la ~/.m2/repository/io/example/my-lib/1.0-SNAPSHOT/*.jar
```

A timestamp newer than your last known-good build confirms a background install is the cause.

## Fix

**Immediate:** Pin the SNAPSHOT dependency to a specific timestamped version to isolate the problem:

```xml
<version>0.2-20260529.181231-131</version>
```

**Long-term:** Ensure sibling projects publish API and implementation JARs as a consistent pair before any SNAPSHOT version is installed locally. Coordinate multi-module SNAPSHOT releases so the API JAR and its implementation are always installed together.

## Why This Is Non-Obvious

The source of the JAR change is invisible — Maven doesn't log which JAR version it resolved or that it changed since the last build. The `AbstractMethodError` points at the incompatible call site in your code, not at the JAR version mismatch. The connection to IntelliJ's background compilation requires knowing that IntelliJ runs `mvn install` on project build completion for all open workspace projects.
