## Symptom

After `git stash pop` completes (conflict reported), compilation fails with:

```
[ERROR] cannot find symbol
  symbol:   class SomeClass
  location: class io.example.SamePackage
```

The missing class exists in the same package and is unchanged. No import is needed. It compiled fine before the stash operation.

## Context

A `git stash pop` creates a conflict inside a Java file — specifically, conflict markers land between two methods at the bottom of the class:

```java
    void existingMethod() { ... }

<<<<<<< Updated upstream
=======

    void newMethod() {
        // stashed content
    }

>>>>>>> Stashed changes
}
```

The file appears complete (the class closing brace is present), but the conflict markers make it syntactically invalid. The Java compiler reports a confusing error about unrelated symbols rather than a parse error.

## What was tried

Checking imports, checking that the referenced class file exists, running `mvn clean` — none fixed the issue. The root cause wasn't visible in the error message.

## Root cause

Java's parser encounters the `<<<<<<<` marker and fails to parse the rest of the file. Classes defined after the conflict marker in the same compilation unit become undefined, and other compilation units in the same package lose the ability to see any symbols from this file. The error message names a type that was successfully compiled before, making it look like a classpath issue.

The stash pop conflict may not be obvious if: the file appeared to finish normally (closing brace at end), the conflict markers land in a location that doesn't produce a visible syntax error in the IDE, or the working directory switched branches mid-session without notice.

## Fix

Search for conflict markers across all tracked Java files:

```bash
grep -rn "<<<<<<\|>>>>>>>\|=======" src/
```

Resolve each conflict. For a simple case where one side is empty (upstream had nothing, stash added content), keep the stashed content and remove all three marker lines.

To detect silently: `git status --short` shows `UU` (both modified) or `DU` (deleted/updated) for files with unresolved conflicts.

## Why this is non-obvious

`cannot find symbol` for a class in the same package strongly implies a classpath, missing source root, or incremental build problem. The actual cause — conflict markers corrupting the same file that defines those symbols — is invisible in the error output. The mismatch between "file exists on disk" and "compiler can't see it" leads to a long investigation of wrong areas.
