### Symptom

After squashing a feature branch and merging PR A to main, you rebase the next feature branch (PR B) onto the updated main. The rebase stalls mid-run:

```
The previous cherry-pick is now empty, possibly due to conflict resolution.
If you wish to commit it anyway, use:
    git commit --allow-empty
Otherwise, please use 'git rebase --skip'
```

The stalled commit is a legitimate fix commit that you expected to be included in PR B.

### Context

PR A's branch was squash-merged — 3 commits combined into 1. PR B's branch contained the same 3 commits individually (cherry-picked or shared history). When git rebases PR B onto the updated main, it detects 2 of the 3 commits as already applied (by patch-id) and skips them automatically. The 3rd commit can't be detected as already applied because the squash produced a different patch-id (the squash commit contains all 3 changes merged; the individual commit only contains 1).

Git applies the 3rd commit on top of main (which already contains its content), producing a no-change commit — which it calls "empty".

### Root cause

`git rebase` uses patch-id comparison to detect already-applied commits. Squashing N commits into 1 produces a patch that is the union of all N — a different patch-id than any individual commit. So squash-merged content is only auto-skipped if the original commit was merged as-is (no squash).

### Why this is non-obvious

The rebase auto-skips 2 of 3 duplicate commits with only a `warning: skipped previously applied commit` warning (easily missed). The 3rd is not recognised as a duplicate and is applied, but it produces no change, triggering the stall. Nothing in the error output says "this commit's content is already on the target branch."

### Fix

Run `git rebase --skip` to drop the empty commit. Repeat for each empty commit the rebase produces. After the rebase, verify the final diff against origin:

```bash
git diff origin/<branch> HEAD | wc -l   # should be 0 if content is identical
```

If you need to understand which commits were dropped vs retained:
```bash
git log --oneline upstream/main..HEAD   # after rebase
```

### Avoiding the problem

Before rebasing a feature branch onto a main that had squash-merges of shared commits, explicitly list the commits you expect to be in the PR and drop the duplicates in the rebase todo:

```
drop <sha>  fix: update stubs    ← already merged via squash
pick <sha>  feat: new feature
```

Or use `git rebase --onto upstream/main <last-shared-commit>` to rebase only the branch-unique commits.
