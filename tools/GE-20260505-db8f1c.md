## Context

A feature branch contains an unrelated commit below the intended commits. It
needs to be removed without touching the commits above it, and without
opening an interactive editor.

## Technique

```bash
git rebase --onto upstream/main <sha-of-unwanted-commit> feat/my-feature
```

This rebases every commit after `<sha>` onto `upstream/main`, effectively
dropping `<sha>` from the branch. The unwanted commit is excluded; everything
above it lands cleanly on top of upstream.

Then force-push:

```bash
git push origin feat/my-feature --force
```

## What developers normally do instead

`git rebase -i HEAD~N` — mark the unwanted commit as `drop`, then manually
resolve the interactive editor. Riskier with many commits; requires counting
back to find N; opens an editor that can be error-prone in scripts or
constrained environments.

## Why this is non-obvious

The three-argument form of `git rebase --onto` is rarely taught. Most developers
know `--onto` for branch-transplanting but not for commit-dropping. The mental
model — "rebase everything after this SHA onto that ref" — is not obvious from
the man page's terse syntax description.

## Example from context

Two PR branches had `40bbc76` (an unrelated feature commit) as their base
because they were cut from local `main` instead of `upstream/main`:

```bash
git rebase --onto upstream/main 40bbc76 feat/engine-221-message-type
git rebase --onto upstream/main 40bbc76 feat/engine-229-provision-context-trigger
```

Both branches dropped the commit cleanly in a single command each.
