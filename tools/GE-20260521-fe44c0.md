### Symptom
Running `work-end` (or `work-start`) from a Claude session whose CWD is workspace A,
to close a branch that belongs to workspace B, silently operates on workspace A's repos.
No error is thrown — the wrong `.meta` is read, the wrong branches are switched, and
the wrong HANDOFF.md may be committed.

### Context
Multi-workspace setups where multiple casehub repos each have their own workspace.
Triggered when you attempt to close an orphaned branch on workspace B from a session
that is open on workspace A (e.g. closing a parent workspace branch from an engine session).

### Root cause
The skill resolves paths via `git rev-parse --show-toplevel` (WORKSPACE) and
`readlink -f proj` (PROJECT), both of which are relative to the session's current
working directory. If the session CWD is workspace A, these resolve to workspace A's
repos regardless of which branch you intend to close.

### Fix
Bypass the skill entirely. Use explicit `-C <path>` for every git operation and
hard-coded absolute paths for all file reads and writes:

```bash
git -C /path/to/target-workspace checkout <branch>
git -C /path/to/target-project checkout <branch>
cat /path/to/target-workspace/design/.meta
# ... all subsequent steps with explicit paths
```

### Why this is non-obvious
The skill's path resolution feels automatic — the `proj/` symlink pattern implies
"the skill knows where the project is." In practice it resolves against the session
CWD, which is workspace A. The skill runs without errors on the wrong repos.
