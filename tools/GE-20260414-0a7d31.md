## Symptom

Running `git commit --amend -m "$(git log -1 --format='%B')Closes #17"` produces a commit message where the footer is concatenated directly to the last line of the original message — no newline separator:

```
feat: add EnemyStrategy domain records with Jackson serdeCloses #17
```

instead of the expected:

```
feat: add EnemyStrategy domain records with Jackson serde

Closes #17
```

## Context

When rewriting git history to add issue-tracking footers (e.g. `Refs #N`, `Closes #N`) using a cherry-pick loop. The intent is to append footers to each existing commit message. `$(git log -1 --format='%B')` captures the current commit's full message body.

## Root Cause

POSIX specifies that command substitution (`$(...)`) strips all trailing newline characters from the output. `git log --format='%B'` outputs the commit message followed by a trailing newline (sometimes two). All of those newlines are stripped by bash before the result is assigned to the variable or interpolated inline. The footer then concatenates directly to the last non-newline character.

## Fix

Write the full amended message explicitly with a literal blank line in the heredoc or multi-line string:

```bash
git commit --amend -m "feat: add EnemyStrategy domain records with Jackson serde

Closes #17"
```

Or, when automating across many commits, use `printf` with explicit `\n`:

```bash
ORIG=$(git log -1 --format='%s')
BODY=$(git log -1 --format='%b')
FOOTER="Closes #17"
if [ -n "$BODY" ]; then
    git commit --amend -m "$ORIG

$BODY

$FOOTER"
else
    git commit --amend -m "$ORIG

$FOOTER"
fi
```

## Why Non-Obvious

`$(git log -1 --format='%B')` appears to capture the full message. The stripping is invisible — no warning, and the resulting string looks correct when printed with `echo "$MSG"` (which re-adds a newline). The bug only manifests in the committed message.
