## Symptom

A module appears in a summary table loop with an empty or broken link (e.g. `https://github.com/casehubio/` with no repo name), even though the module built successfully. No error fires anywhere in the workflow.

## Context

Multi-module build workflows that iterate over a `MODULES` array and look up values in a separate bash associative map (e.g. `GH_REPO`, `DECISIONS`, `OUTCOMES`). If a module is added to the iteration array but its corresponding entry is omitted from the lookup map, the substitution silently returns empty string.

```bash
MODULES=(platform ledger eidos connectors)
declare -A GH_REPO=(
  [platform]="platform"
  [ledger]="ledger"
  # eidos accidentally omitted
  [connectors]="connectors"
)

for module in "${MODULES[@]}"; do
  echo "| [$module](https://github.com/casehubio/${GH_REPO[$module]}) |"
  # eidos row prints: | [eidos](https://github.com/casehubio/) |
done
```

## Root cause

Bash associative arrays return `""` for missing keys — no error, no warning. Crucially, `set -u` (nounset) does **not** trap associative array subscript access; only bare variable references trigger it. So `${GH_REPO[$module]}` silently expands to empty even with `set -euo pipefail` active.

## Fix

Keep parallel arrays and maps explicitly in sync. Optionally add a pre-loop validation guard:

```bash
for module in "${MODULES[@]}"; do
  [ -z "${GH_REPO[$module]:-}" ] && echo "::error::GH_REPO missing entry for: $module" && exit 1
done
```

Or use a single source of truth — derive display names from the module name directly rather than maintaining a parallel map when the values are predictable transformations.

## Why this is non-obvious

Developers familiar with `set -u` expect unbound variable access to error. Associative array subscript access looks identical to regular variable access syntactically, but bash treats them differently: `$UNDEFINED` triggers nounset; `${MAP[$key]}` where `$key` has no entry does not. The silent empty return is invisible until you inspect the rendered output — and even then it looks like a rendering issue rather than a missing map entry.
