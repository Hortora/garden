## Symptom

`page.waitForRequest()` for a resize event times out. The xterm.js terminal renders in
the page but `terminal.onResize` never fires, even though the page loads and `fitAddon`
appears to run.

## Context

Writing Playwright E2E tests for a page that uses xterm.js + FitAddon. The page calls
`fitAddon.fit()` on load and in a `ResizeObserver`. `terminal.onResize` is wired to call
a REST endpoint with the new dimensions.

## What was tried (didn't work)

`page.waitForRequest(request -> request.url().contains("/resize"), () -> { page.navigate(url); })`
— times out because the resize request is never made.

## Root cause

In headless Chromium, the xterm.js container element has no visible dimensions — its
computed size is 0×0 or the same as the terminal's built-in default (80×24). `fitAddon.fit()`
computes the terminal size from the DOM element geometry. When the container has no
real rendered size, it computes the same 80×24 default the terminal already has and
`terminal.onResize` does not fire (no-op — same size).

## Fix

**Step 1** — Add a conditional test hook to the page JavaScript:

```javascript
terminal.open(document.getElementById('terminal-container'));
fitAddon.fit();
if (window.__CLAUDONY_TEST_MODE__) {
    window._xtermTerminal = terminal;  // expose for Playwright to drive
}
```

**Step 2** — In the Playwright test, set the flag via `addInitScript` before navigation,
intercept the resize request via `page.route()`, then force `onResize` via `page.evaluate()`:

```java
page.addInitScript("window.__CLAUDONY_TEST_MODE__ = true;");

var capturedUrl = new java.util.concurrent.atomic.AtomicReference<String>();
page.route("**/resize**", route -> {
    capturedUrl.set(route.request().url());
    route.fulfill(new Route.FulfillOptions().setStatus(202));
});

page.navigate(BASE_URL + "/app/session.html?id=fake&name=test&proxyPeer=peer-id");

// Force onResize to fire — fitAddon.fit() won't in headless
for (int i = 0; i < 30 && capturedUrl.get() == null; i++) {
    page.evaluate("window._xtermTerminal && window._xtermTerminal.resize(100, 30)");
    page.waitForTimeout(100);
}

assertThat(capturedUrl.get()).contains("/resize").contains("cols=100").contains("rows=30");
```

## Why this is non-obvious

`fitAddon.fit()` silently succeeds — no error, no warning — it just computes the same
terminal size as the default and emits nothing. The terminal renders visually in the
Playwright screenshot. The obvious fix (`waitForRequest`) fails silently with a timeout.
Nothing in the xterm.js or FitAddon documentation mentions headless browser behaviour.
