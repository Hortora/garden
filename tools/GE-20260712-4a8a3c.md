### Symptom
After reordering components in a Java record, all existing construction sites compile without error but silently pass values into wrong fields. At runtime, handlers receive garbage data — e.g., workerId contains the tenancyId value, idempotency contains the workerId.

### Context
Java record constructors are positional. When reordering components, all construction sites must be updated. If consecutive components share the same type (`String`), the compiler cannot detect the mismatch.

Example: `ActionGateWorkerFaultedEvent(UUID caseId, String workerId, String idempotency, String tenancyId)` reordered to `(UUID caseId, String tenancyId, String workerId, String idempotency)`. Construction sites with `(uuid, workerId, idempotency, tenancyId)` now silently assign workerId→tenancyId, idempotency→workerId, tenancyId→idempotency.

### What was tried (didn't work)
1. IntelliJ `ide_build_project` — reported SUCCESS (types match positionally).
2. `ide_diagnostics` — no errors (same reason).
3. Unit tests — passed (didn't exercise the affected event path end-to-end).
4. Only integration tests caught it — cases timed out because handlers received wrong field values and couldn't find PlanItems.

### Root cause
Java record constructors are purely positional with no named-parameter support. When multiple consecutive components share the same type, reordering is invisible to the compiler. The bug is a semantic error, not a syntactic one.

### Fix
When reordering record components:
1. Use `ide_find_references` on the record class to find ALL construction sites before reordering.
2. After reordering, verify every construction site argument order matches the new component order — do not rely on compilation.
3. For records with 3+ same-type consecutive components, consider adding a compact constructor with semantic validation (e.g., assert tenancyId starts with a known prefix).

### Why this is non-obvious
Developers trust the compiler to catch argument order bugs. With distinct types it does. With same-type records, the compiler is blind and the bug surfaces only as mysterious runtime behaviour — often far from the construction site (e.g., a downstream handler timing out because it looked up a PlanItem by the wrong ID).
