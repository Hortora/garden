## Symptom

`mvn install -f aggregator.xml -pl quarkus-ledger` completes in ~4 seconds with BUILD SUCCESS. No tests run. No JARs appear in `.m2` beyond the parent POM. CI shows green but the published artifact contains only the parent POM, not the actual extension code.

## Context

Outer aggregator (`aggregator.xml`) lists sibling repos as modules:
```xml
<modules>
  <module>quarkus-ledger</module>   <!-- points to ../quarkus-ledger/pom.xml -->
  <module>quarkus-work</module>
</modules>
```

`quarkus-ledger/pom.xml` is itself a multi-module parent with submodules (`runtime/`, `deployment/`, etc.).

Running `mvn install -f aggregator.xml -pl quarkus-ledger` builds successfully but only installs `quarkus-ledger-parent` — not `runtime`, `deployment`, or any actual code.

## Root cause

When Maven builds a module via `-pl` from an outer aggregator, it treats the selected entry as a **single node** in the outer reactor. The outer reactor does not include the inner project's submodules. Maven only recurses into submodules when building the inner project as a standalone reactor (i.e., from within the inner project's directory, or when the outer aggregator listed all submodules explicitly).

## Fix

Build from within the inner project's directory instead:

```bash
# Instead of:
mvn install -f aggregator.xml -pl quarkus-ledger

# Use:
cd quarkus-ledger && mvn install --batch-mode $FLAGS
```

Or in CI, `cd` to each repo's directory before invoking `mvn install`. The full-stack aggregator is still useful for ordering (via `aggregator.xml`), but per-module builds should happen from the repo root.

## Why this is non-obvious

The build completes with BUILD SUCCESS. No error is reported. The output log shows the parent POM being installed, which looks plausible. The missing submodule builds are invisible unless you check what was actually installed in `.m2`. CI passes but publishes an empty package.

### Why this fix

Running `cd project && mvn install` treats that project's `pom.xml` as the reactor root, which correctly includes all declared `<module>` entries. The aggregator approach works for full-stack builds (`mvn install -f aggregator.xml` without `-pl`) but not for selective single-project builds within it.
