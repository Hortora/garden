## Symptom

You map out which repos should dispatch to which others by reading the dependency graph
(A depends on B, so B should trigger A). Three of your dispatch entries are wrong — B
actually triggers C and D, not A — and you only discover this by running the full-stack
build and seeing A never rebuild after B publishes.

## Context

Multi-repo Maven/Gradle CI ecosystem where each repo's publish workflow dispatches
`repository_dispatch` events to its downstream consumers. The dispatch targets are
manually maintained in each repo's `publish.yml`. They do not automatically mirror the
Maven dependency graph.

## What was tried (didn't work)

Building the dispatch map from the pom.xml dependency graph. In one case the map showed:

- `platform → work, qhorus` (wrong — platform triggers ledger, connectors)
- `connectors → (nothing)` (wrong — connectors triggers work)
- `engine → claudony, openclaw` (wrong — engine triggers flow, not claudony; claudony
  is triggered by qhorus)

All three were plausible from the dep graph. None were correct.

## Root cause

The dispatch chain is a separate concern from the dependency graph and is maintained
independently in each repo's CI workflow. The dep graph tells you *what* a repo needs
to build; the dispatch chain tells you *who to notify* when it publishes. These are
related but not identical:

- A repo may dispatch to consumers that aren't direct dependents (transitive skip)
- A repo may NOT dispatch to a direct dependent if another repo in the chain handles it
  (e.g. qhorus triggers both engine AND claudony, so engine does not need to re-trigger
  claudony)
- A repo may dispatch to an adjacent non-dependent (e.g. engine → flow is a sibling
  dispatch, not a consumer chain)

## Fix

Verify the actual dispatch targets by reading each upstream repo's publish workflow file:

```bash
grep -A 10 "Trigger downstream" /path/to/repo/.github/workflows/publish.yml
```

Build the map from the workflow files, not from pom.xml. When adding a new repo:

1. Identify which upstream repos it depends on
2. Read each upstream's publish.yml to see what it currently dispatches to
3. Determine whether any upstream needs updating to add the new repo
4. File an issue on the upstream repo (cannot commit to peer repos from a different session)

## Why this is non-obvious

The dependency graph is visible, parseable, and accurate. The dispatch chain is invisible
until you run a build and notice a repo didn't rebuild. You naturally assume "if A depends
on B, B must trigger A" — but the dispatch chain is a manually maintained separate concern
that has its own topology shaped by avoiding redundant builds.
