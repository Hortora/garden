## Symptom

`ctx.drawImage(webglCanvas, x, y, 1, 1, 0, 0, 1, 1)` followed by `ctx.getImageData(0, 0, 1, 1)` returns RGBA `[0,0,0,0]` (transparent black) regardless of what is rendered on screen. No error is thrown. The pixel sampling silently fails.

## Context

Implementing visual regression tests with Playwright against a Three.js WebGL scene. The test aims to sample the actual rendered pixel at a known world position (e.g. a mineral sprite) and assert its colour is cyan rather than sandy terrain. Copying the WebGL canvas to a 2D canvas for pixel access is the standard approach.

## What was tried

- Added `samplePixel(x, y)` to `window.__test` using canvas 2D `drawImage` from the renderer's domElement
- Test returned `{r:0, g:0, b:0, a:0}` consistently
- Confirmed the target element was rendering correctly (visible on screen)
- Tried different pixel coordinates — all returned transparent black

## Root cause

Three.js `WebGLRenderer` defaults to `preserveDrawingBuffer: false`. In this mode, the WebGL drawing buffer is cleared at the end of each frame by the browser. When `drawImage` is called on the WebGL canvas between frames (e.g. from a `page.evaluate()` call), the buffer has already been cleared — the canvas is empty.

## Fix

Pass `preserveDrawingBuffer: true` when constructing the renderer:

```javascript
renderer = new THREE.WebGLRenderer({ antialias: true, preserveDrawingBuffer: true });
```

Then `samplePixel` works reliably:

```javascript
samplePixel: (x, y) => {
  const c = document.createElement('canvas');
  c.width = 1; c.height = 1;
  const ctx = c.getContext('2d');
  ctx.drawImage(renderer.domElement, Math.round(x), Math.round(y), 1, 1, 0, 0, 1, 1);
  const d = ctx.getImageData(0, 0, 1, 1).data;
  return { r: d[0], g: d[1], b: d[2], a: d[3] };
},
```

Note: `preserveDrawingBuffer: true` has a slight performance cost (prevents certain GPU optimisations). Use it only when pixel readback is required (e.g. in test builds).

## Why this is non-obvious

The default `false` value is a performance optimisation — the docs mention it but don't emphasise that pixel sampling will silently fail. The symptom (transparent black pixels with no error) looks exactly like a coordinate error or a timing issue, sending debugging effort in the wrong direction.
