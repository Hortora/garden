### Symptom
Local compilation succeeds against a peer repo's SNAPSHOT. CI fails with:
```
[ERROR] method fromWire in class WorkItemLifecycleEvent cannot be applied to given types;
  required: String, String, String, UUID, WorkItemStatus, Instant, String, String, String, String, String
  found:    String, String, String, UUID, WorkItemStatus, Instant, String, <nulltype>, <nulltype>, <nulltype>, <nulltype>, <nulltype>
  reason: actual and formal argument lists differ in length
```
The method exists with the correct signature in the local source — but CI resolves a different API from GitHub Packages.

### Context
Multi-repo Maven project where each repo publishes a SNAPSHOT to GitHub Packages on CI success. A peer repo (`casehub-work`) added a new parameter to an API method and installed locally (`mvn install`) but had not yet pushed. The consumer (`casehub-aml`) wrote code against the new local API and compiled fine. CI for the consumer repo pulled from GitHub Packages and got the old API.

The peer repo's CI was also failing (separate issue) — so even after pushing, no new SNAPSHOT was published until the peer CI was fixed.

### Root cause
GitHub Packages SNAPSHOT is only updated when the publishing repo's CI runs successfully. Between a local `mvn install` in a peer repo and the peer CI going green and publishing, the locally-installed artifact and the published artifact have diverged. Any consumer CI that resolves from GitHub Packages will see the old API.

Timeline of divergence:
1. Developer: adds new parameter to `WorkItemLifecycleEvent.fromWire()` in `casehub-work`
2. Developer: `mvn install` in `casehub-work` — local cache has 12-param version
3. Developer: writes consumer code using 12-param API in `casehub-aml` — local compile passes
4. `casehub-work` CI: failing on unrelated test — no publish
5. `casehub-aml` CI: resolves from GitHub Packages — gets 11-param version — compile error

### Fix
1. Fix and push the peer repo CI first.
2. Wait for GitHub Packages to publish the new SNAPSHOT.
3. Re-trigger consumer repo CI.

To verify whether CI would pass before triggering it, check the peer repo's latest CI run:
```bash
gh run list --repo casehubio/work --limit 3 --json status,conclusion,createdAt \
  --jq '.[] | "\(.conclusion // .status) \(.createdAt)"'
```

### Why this is non-obvious
`mvn install` in a peer repo and `mvn test` in the consumer give a false green signal. The divergence is invisible locally because both use the local `.m2` cache. The only observable difference is CI vs local — which makes it look like a CI environment problem rather than a dependency version problem.
