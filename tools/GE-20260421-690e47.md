### What this is

A pattern for writing interactive CLI tools (prompt-response loops) so that
every code path is testable without mocking stdin, subprocess, or the terminal.

### The pattern

Instead of reading from stdin directly inside the logic function, accept a
`decide_fn` callback parameter. The logic function calls the callback with
the current item; the callback returns the decision. The CLI `__main__` block
wires it to stdin. Tests inject deterministic lambdas.

```python
# Library function — fully testable
def validate_candidates(
    report: CandidateReport,
    registry_projects: list[dict],
    rejections_path: Path,
    out_dir: Path,
    decide_fn: Callable[[dict], tuple[str, str | None]],
) -> SessionSummary:
    for candidate in report.cluster_candidates:
        decision, reason = decide_fn(candidate)
        if decision == Decision.ACCEPT:
            write_skeleton(candidate, registry_projects, out_dir)
        elif decision == Decision.REJECT:
            rejection_reg.add(candidate['centroid'], candidate['projects'], reason or '')
    return SessionSummary(...)

# CLI wiring — only in __main__
if __name__ == '__main__':
    def _stdin_decide(candidate):
        choice = input('[a]ccept / [r]eject / [s]kip: ').strip().lower()
        if choice == 'a':
            return Decision.ACCEPT, None
        elif choice == 'r':
            return Decision.REJECT, input('Reason: ').strip()
        return Decision.SKIP, None

    validate_candidates(report, projects, rejections_path, out_dir, _stdin_decide)
```

### Test usage

```python
# Accept all
summary = validate_candidates(
    report, projects, rejections_path, out_dir,
    decide_fn=lambda c: (Decision.ACCEPT, None),
)

# Accept first, reject second
decisions = iter([(Decision.ACCEPT, None), (Decision.REJECT, 'noise')])
summary = validate_candidates(
    report, projects, rejections_path, out_dir,
    decide_fn=lambda c: next(decisions),
)

# Spy on what was presented
seen = []
validate_candidates(..., decide_fn=lambda c: (seen.append(c), Decision.SKIP)[1])
```

### Why non-obvious

The obvious approach is to write `input()` calls inside the logic function and
mock `builtins.input` in tests. This is fragile, couples the test to implementation
details, and makes it impossible to test sequences of decisions without complex
mock setup. The callback pattern separates I/O from logic completely — the logic
function never knows whether it's in a test or a terminal session.
