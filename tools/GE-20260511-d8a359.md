### Context

When diagnosing whether one git branch is ahead, behind, or diverged from another, the instinct is to run `git log A ^B | wc -l` — commits on A not on B. This is ambiguous: a count of 0 means A has no commits B doesn't have, but says nothing about whether B has commits A doesn't have (i.e. A could be behind, not equal).

### The technique

Always check both directions:

```bash
# Commits on upstream that are NOT in local backup
ahead=$(git log --oneline upstream/main ^backup/pre-squash-main | wc -l | tr -d ' ')

# Commits on local backup that are NOT in upstream
behind=$(git log --oneline backup/pre-squash-main ^upstream/main | wc -l | tr -d ' ')

echo "upstream ahead of backup: $ahead"
echo "backup ahead of upstream: $behind"
```

Interpretation:
- `ahead=0, behind=0` → identical (or one is an ancestor of the other at the same point)
- `ahead=N, behind=0` → backup is a strict ancestor; upstream is N commits ahead
- `ahead=0, behind=N` → upstream is a strict ancestor; backup is N commits ahead
- `ahead=N, behind=M` → diverged — neither is an ancestor of the other

Can also use `git merge-base --is-ancestor` for a boolean check:

```bash
git merge-base --is-ancestor backup upstream && echo "backup is ancestor of upstream"
git merge-base --is-ancestor upstream backup && echo "upstream is ancestor of backup"
```

### Why this is non-obvious

Single-direction `git log A ^B = 0` is commonly read as "A and B are the same" — but it only means "B contains everything A has." In this session, a backup branch appeared to be "behind" casehubio when it was actually 41 commits *ahead*, because only one direction was initially checked. The correct reading was only found after running both directions.

### Why this fix

Two counts give a complete picture of the relationship in O(log n) time. `git merge-base --is-ancestor` is faster for a boolean but doesn't give counts. Use counts when you need to know *how far* diverged; use `merge-base` when you only need a binary yes/no.
