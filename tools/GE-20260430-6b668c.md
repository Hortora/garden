## Symptom

CI fails with `cannot find symbol: method foo(String)` on a method that demonstrably exists in the local source file. `git diff` reveals the method was added but never committed. The published artifact lacks the method.

## Context

A subagent was tasked with migrating code in repo B to use `TrustGateService.findScore()`. It:
1. Verified `findScore()` existed in repo A's local source ✓
2. Updated repo B to call `findScore()` ✓  
3. Tested locally ✓ (resolved from local `.m2`)
4. Reported DONE ✓

A *different* subagent had added `findScore()` to repo A's source in an earlier task — but also didn't commit it. The method was in the working tree but not in git. Repo A's CI published the artifact *without* `findScore()`. Repo B's CI then failed.

## Root Cause

Local `.m2` cache resolves from the working tree build (has the method). CI pulls from GitHub Packages (does not have it). The discrepancy is invisible during local testing. `git status` would have caught it, but was not checked between subagent tasks.

## Fix

After any subagent task that modifies source files, check for uncommitted changes before considering the task complete:

```bash
git status --short   # any M or ?? lines are uncommitted changes
git diff --stat      # shows what changed
```

In subagent prompts, explicitly instruct: "Verify `git status` is clean before reporting DONE. If any files are modified but not committed, commit them."

## Why This Is Non-Obvious

Subagents test against the local Maven repo (which reflects working tree builds), not GitHub Packages. The passing local test gives false confidence. The failure only surfaces in CI, which uses the published artifact. The symptom (missing method) looks like a version mismatch, not an uncommitted change.
