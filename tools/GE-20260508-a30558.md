### Symptom

After running `git rm -r <directory>`, untracked files in that directory silently remain on the filesystem. No warning or error is emitted — git exits 0 regardless.

### Context

Most common during workspace or project migrations where a directory contains a mix of tracked and untracked files — e.g. migrating brainstorm outputs, generated files, or draft plans where some were committed and others were never added to git. You assume the directory is empty after `git rm -r`; it isn't.

### What was tried

Running `git rm -r .superpowers/brainstorm/` removed all committed brainstorm directories but left 8 untracked directories in place. Similarly, `git rm -r docs/superpowers/plans/` left 3 untracked plan files. The migration appeared complete until an explicit `ls` check revealed the residue.

### Root cause

`git rm` only operates on files known to git's index (tracked files). Files that have never been committed — and were never `git add`'d — are invisible to the index and are therefore ignored by `git rm` entirely. The command exits with success regardless, so there is no signal that files remain.

### Fix

Always verify the directory is actually empty after `git rm -r`:

```bash
git rm -r <directory>/
ls <directory>/   # confirm empty or gone

# If files remain — they are untracked; remove with:
rm -rf <directory>/
```

Or, if you intend to remove all files (tracked and untracked), use both:

```bash
git rm -r <directory>/
rm -rf <directory>/
```

### Why this is non-obvious

`git rm -r <dir>` reads as "remove this directory" but it only removes the directory's tracked content. The directory may continue to exist with untracked files, and git reports success either way. A developer expecting "directory removed" behaviour will not notice until they check the filesystem explicitly.
