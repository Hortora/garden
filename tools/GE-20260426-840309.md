## Symptom

`mvn --batch-mode deploy` fails with:

```
[INFO] Retrying deployment attempt 2 of 10
[INFO] Retrying deployment attempt 3 of 10
...
[ERROR] Failed to deploy artifacts: Could not find artifact io.example:my-artifact:pom:0.2-20260426.141142-1 in github (https://maven.pkg.github.com/org/repo)
```

After 10 retries the deploy fails. Tests pass fine — only the deploy step fails.

## Context

Occurs on the **first-ever deployment** of an artifact to GitHub Packages. Subsequent deploys (once the package namespace exists with prior metadata) succeed normally.

The root module (parent POM) hits this first; if you skip parent POM deployment, the first deployable submodule hits it instead.

## Root Cause

Maven deploy plugin 3.x, before uploading a SNAPSHOT artifact, attempts to GET the existing `maven-metadata.xml` to determine the next build number. GitHub Packages returns a non-404 response for non-existent artifacts (rather than a clean 404), which Maven interprets as an error rather than "not found yet". Maven retries 10 times and gives up.

Note: `<uniqueVersion>false</uniqueVersion>` in `distributionManagement` **does not help** — Maven 3 silently ignores this setting (it was a Maven 2 feature) and always uses timestamped SNAPSHOTs for remote repos.

## Fix

**Short-term:** Switch to `mvn verify` instead of `mvn deploy` until the package namespace is initialized:

```yaml
- name: Build and test
  run: mvn --batch-mode verify
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

**Structural fix:** Skip deploying the root/parent POM (which is not a consumable artifact) and only deploy modules that produce useful JARs. Set `maven.deploy.skip=true` as the project default, then override per-module:

```xml
<!-- Root pom.xml -->
<properties>
  <maven.deploy.skip>true</maven.deploy.skip>
</properties>

<!-- api/pom.xml, engine/pom.xml etc -->
<properties>
  <maven.deploy.skip>false</maven.deploy.skip>
</properties>
```

## Why This Is Non-Obvious

The error message says "Could not find artifact" which sounds like a dependency resolution problem, not a deployment problem. The actual cause (GitHub Packages returning a non-404 for non-existent artifacts) is invisible. Developers spend time checking authentication, permissions, and pom configuration before discovering it's a first-deploy timing issue.
