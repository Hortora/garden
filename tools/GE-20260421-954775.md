## Context

When implementing iterative numerical algorithms, the standard approach is to write the code, run it, and see if the output looks right. This misses silent convergence failures — the algorithm terminates at MAX_ITERATIONS and returns a plausible-looking but wrong result. Tests with "nice" inputs pass; production inputs with degenerate structure silently produce garbage.

## Technique

Before writing any implementation code, trace the algorithm by hand for 3–4 iterations using a minimal worked example (3–5 actors/elements). Choose a **degenerate input** that stresses the design — not the happy path:

- Trust algorithms: a cyclic graph, a disconnected actor, a row of all-zero weights
- Numerical methods: near-singular matrices, extreme value ratios, zero-weight rows

For each iteration:
1. Compute the matrix product by hand (or on paper)
2. Apply the update rule exactly as the algorithm specifies
3. Check whether values are trending toward a fixed point, oscillating, or diverging

If convergence fails in the worked example, the **design has a flaw** — fix the design before writing code.

## Why this is non-obvious

Most developers write code first, then add tests. Tests use "nice" inputs where convergence is guaranteed. The hand-trace catches design flaws before a single line is written, at near-zero cost. The worked example is directly reusable as a test case.

For EigenTrust specifically: tracing a 3-actor cyclic graph (induced by the paper's dangling-node recommendation) revealed period-3 oscillation that no standard test would have caught. The fix (uniform fallback for dangling nodes) was identified before any implementation existed.
