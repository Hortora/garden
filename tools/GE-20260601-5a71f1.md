## Symptom

CLAUDE.md slowly re-accumulates content (gotchas, Flyway notes, conventions)
over sessions, undoing a previous manual extraction to `docs/GOTCHAS.md`,
`docs/FLYWAY.md`, or `scripts/README.md`. No error is raised.

## Context

Project has a modular CLAUDE.md structure where content for certain categories
has been extracted to sibling docs. `update-claude-md` is invoked automatically
via `java-git-commit` after every commit.

## Root Cause

`SKILL.md` Step 1a calls `python scripts/document_discovery.py CLAUDE.md` to
detect modular structure. This script does not exist in projects (it was never
created). The invocation silently fails and the skill falls through to
single-file mode, bypassing all modular routing logic. Every subsequent commit
adds new content directly to CLAUDE.md rather than routing it to the correct
module file.

## Fix

Replace `document_discovery.py` with bash file-presence checks for known module files:

```bash
[ -f "docs/GOTCHAS.md" ] && echo "gotchas: yes" || echo "gotchas: no"
[ -f "docs/FLYWAY.md" ]  && echo "flyway: yes"  || echo "flyway: no"
[ -f "scripts/README.md" ] && echo "scripts-readme: yes" || echo "scripts-readme: no"
```

Add an explicit content routing table (Step 4e in the skill) mapping content
categories to target files. This removes the Python dependency entirely.

## Why This Is Non-Obvious

The skill appears to work correctly in every observable way — it reads CLAUDE.md,
proposes changes, applies them, and commits. Nothing in the output indicates that
modular routing was bypassed. The symptom (file re-growth) only becomes visible
across multiple sessions and looks like normal documentation drift rather than a
broken detection step.
