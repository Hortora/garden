## git diff main..<branch> shows files as 'Added' when main deleted them but the branch predates the deletion

### Symptom

`git diff main..<branch> --name-status | grep "^A"` shows files as Added that you know should not exist — they were deleted in a previous commit on main. Specifically, files that a parallel epic deleted appear as "Added" on a branch that branched before those deletions landed.

### Root cause

`git diff main..<branch>` shows what is in the branch but not in main. If main deleted a file after the branch point, that file is present on the branch but absent on main — so it appears as "Added" from main's perspective. This is technically correct, but misleading when scanning a branch's unique contributions.

### Fix

Use `--diff-filter` to exclude files that are new-to-main only because main deleted them. Compare against the common ancestor:

```bash
# What the branch uniquely adds (excluding files restored from deletion)
git diff $(git merge-base main <branch>)..<branch> --name-status | grep "^A"

# Or: filter by whether the file actually exists on main
git diff main..<branch> --name-status | grep "^A" | while read status file; do
  git show main:"$file" &>/dev/null || echo "GENUINELY NEW: $file"
done
```

### Why this is non-obvious

The output is accurate by git's definition but misleading for the common task of "what does this branch introduce?". Deleted-then-restored files look identical to genuinely new files in the name-status output.
