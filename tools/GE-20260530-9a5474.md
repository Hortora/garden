### Symptom
`HTTP 401 Unauthorized` from Maven when resolving SNAPSHOT dependencies from GitHub Packages.
No diagnostic about which scope is missing — just "Unauthorized".

### Context
`~/.m2/settings.xml` configures the `github` server with
`<password>${env.GITHUB_TOKEN}</password>`. The shell session has no
`GITHUB_TOKEN` set. Developer runs `GITHUB_TOKEN=$(gh auth token) mvn ...` or
sets the env var via subprocess — intuitive since `gh` handles all other GitHub
auth transparently.

### Root cause
`gh auth token` returns a short-lived OAuth token (`gho_` prefix), not a classic PAT.
GitHub Packages Maven registry requires a classic PAT with `read:packages` scope. The
`gho_` token may have `repo` scope (for git operations) but not `read:packages`
(for package registry reads). The `gh` CLI itself works fine for push/pull/PR creation
because git uses a different auth path — making the failure non-obvious.

### Fix
Create a classic PAT at `github.com/settings/tokens` with `read:packages` scope.
Export as `GITHUB_TOKEN=<classic-pat>` before running Maven.

**Workaround when dependencies are already built locally:**
If peer repos have `target/` jars from a previous successful build, use
`mvn install:install-file` to populate the local `.m2` cache without hitting
the registry:
```bash
mvn install:install-file -Dfile=../peer/target/artifact-0.2-SNAPSHOT.jar \
    -DpomFile=../peer/pom.xml -Dversion=0.2-SNAPSHOT -Dpackaging=jar -q
```
Then build offline: `mvn install -DskipTests -o`.

### Why this is non-obvious
The `gh` CLI token works perfectly for all other GitHub operations (push, pull, PR, API
calls) because git and the GitHub REST API use the token differently from the Maven
Package Registry. Maven's 401 response body contains only "Unauthorized" with no
indication that the token type or scope is wrong. A developer would naturally assume
the token from `gh auth token` is sufficient since `gh` itself authenticated fine.
