## Symptom

Wall tiles render one tile too high. Using the standard game-to-canvas conversion function `tile(x, y)` to position a tile rectangle places the rectangle shifted by one tile upward relative to where units at the same coordinates appear.

## Context

In a Y-flipped coordinate system (game Y=0 at bottom, canvas Y=0 at top), there are two different formulas depending on what you're positioning:

| What | Formula | Reason |
|------|---------|--------|
| A **point** (unit/building centre) | `canvasY = (VIEWPORT_H - y) * SCALE` | Point at game y maps to canvas y |
| A **tile rectangle** top-left | `canvasY = (VIEWPORT_H - y - 1) * SCALE` | Tile at game y occupies rows y to y+1; its top-left in canvas is one row higher |

## Root cause

The tile at game row y spans y to y+1. Its bottom-left corner in canvas space is at `(VIEWPORT_H - y) * SCALE`. Its top-left (upper edge in canvas) is at `(VIEWPORT_H - (y+1)) * SCALE = (VIEWPORT_H - y - 1) * SCALE`. The standard `tile()` helper returns the bottom-left, not the top-left needed for `Graphics.rect()`.

## Fix

For tile rectangle rendering:

```javascript
// WRONG — same as tile(), gives bottom-left not top-left of rectangle
const canvasY = (VIEWPORT_H - wy) * SCALE;

// CORRECT — top-left of tile rectangle
const canvasY = (VIEWPORT_H - wy - 1) * SCALE;
```

Full wall-tile draw:

```javascript
data.walls.forEach(([wx, wy]) => {
    const canvasX = wx * SCALE;
    const canvasY = (VIEWPORT_H - wy - 1) * SCALE;  // ← -1 is essential
    g.rect(canvasX, canvasY, SCALE, SCALE).fill({ color: 0x333333, alpha: 0.85 });
});
```

## Why this is non-obvious

Both `tile()` and the rectangle formula look identical except for the `-1`. Developers copy the `tile()` function and the tiles render almost correctly — just shifted by one tile, which is easy to miss on a grid. The distinction between "point at y" and "top-left of tile at y" is conceptually obvious once stated but easy to overlook in implementation.
