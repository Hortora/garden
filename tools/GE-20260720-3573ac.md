### Symptom

TypeScript error `Module '"@scope/pkg"' has no exported member 'NewExport'` in a git worktree, even though the export was added to the source file in the worktree. The export exists in the worktree's source — but TypeScript doesn't see it.

### Context

Setup: yarn workspace monorepo with `workspace:*` dependencies. A git worktree is created for feature work. The worktree has no `node_modules`, so a symlink is created pointing to the main repo's `node_modules`.

Resolution chain (4 levels deep):
```
worktree/node_modules          → symlink to main-repo/node_modules
main-repo/node_modules/@scope/pkg  → symlink to ../../components/pkg  (yarn workspace link)
main-repo/components/pkg/dist/     → compiled output from main repo's source
```

The final resolution lands in the **main repo's** `dist/` directory — which may be on a different branch, have stale exports, or never have been built with the worktree's changes.

### What was tried (didn't work)

1. Rebuilding the package in the worktree — `tsc` runs from the worktree but the `dist/` output goes to the worktree's `components/pkg/dist/`. TypeScript resolution still follows the symlink chain to the **main repo's** `dist/`.
2. Adding tsconfig `paths` — works for direct imports but doesn't help with transitive workspace deps.

### Root cause

Yarn workspace links use **relative paths** from `node_modules/`. When `node_modules` is symlinked from another location, the relative paths resolve from the symlink target (main repo), not the symlink source (worktree). Git worktrees share `.git` but have independent working trees — the main repo's source files may be on a completely different branch.

### Fix

Rebuild the dependency in the **main repo** (where the symlinks resolve to):

```bash
# Apply the source change to the main repo's copy
# (the main repo's branch doesn't matter — dist/ is gitignored)

# Rebuild dist/ in the main repo
npx tsc -p /path/to/main-repo/components/pkg/tsconfig.build.json \
  --outDir /path/to/main-repo/components/pkg/dist

# Now the worktree's TypeScript resolution finds the updated dist/
npx tsc --noEmit -p /path/to/worktree/components/consumer/tsconfig.json
```

For vitest: use `resolve.alias` in vitest.config.ts to point directly at sibling component source directories (bypassing `node_modules` entirely). This is why vitest tests pass while `tsc --noEmit` fails — vitest aliases resolve within the worktree, tsc follows the symlink chain.

### Why this is non-obvious

The resolution chain is 4 levels deep and crosses working tree boundaries invisibly. The error message names the missing export accurately — but the cause isn't "export doesn't exist" (it does, in the worktree source), it's "TypeScript is reading a different copy of the package." The vitest/tsc split is especially confusing: tests pass (aliases bypass symlinks) while typecheck fails (follows symlinks to stale dist).

### Why this fix

Rebuilding in the main repo works because it updates the `dist/` files at the end of the symlink chain — exactly where TypeScript looks. The alternative (adding tsconfig `paths` for every workspace dependency) doesn't scale in a large monorepo and breaks when transitive deps also use workspace links.

### Alternatives considered

- **Yarn PnP mode** — eliminates `node_modules` symlinks entirely, but requires ecosystem-wide compatibility and isn't always an option
- **tsconfig project references** — would work but requires adding references for every cross-component dependency, not just `blocks-ui-core`
- **Running `yarn install` in the worktree** — requires `GH_PACKAGES_TOKEN` and creates a separate `node_modules`, duplicating disk usage
