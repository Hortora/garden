## What happened

Since OpenClaw 2026.4.23, a typed-hook policy gate was introduced for non-bundled (path-loaded)
plugins. Registering `before_prompt_build` via `api.on()` in `register()` appears to succeed —
the plugin loads, the `register()` function runs, `api.on("before_prompt_build", handler)` returns
without error — but the handler is **never called**. No error. No warning. No log line. The agent
runs without the hook firing.

## Why it's non-obvious

The failure mode is a silent success. `api.on()` returns without throwing. The plugin's banner log
("Registered before_prompt_build hook") still prints because the registration call itself runs. But
the runtime drops the registration internally and never wires the handler into the typed-hook registry.

Nothing in the plugin's output changes. Nothing in the agent's output changes. There is no observable
difference between "hook registered and firing" and "hook silently dropped" at the plugin level.

## The fix

The user (not the plugin developer) must add `hooks.allowConversationAccess: true` to their
`openclaw.json` plugin entry:

```json
{
  "plugins": {
    "entries": {
      "my-plugin": {
        "enabled": true,
        "hooks": { "allowConversationAccess": true }
      }
    }
  }
}
```

Plugin READMEs must call this out prominently — it is a user-side config requirement, not a plugin
code change. Without it, the plugin silently does nothing on every agent turn.

## Scope

Affects ALL conversation lifecycle hooks for non-bundled plugins: `before_model_resolve`,
`before_agent_reply`, `llm_input`, `llm_output`, `before_agent_finalize`, `agent_end`,
`before_agent_run`. The policy gate was added in 2026.4.23.
