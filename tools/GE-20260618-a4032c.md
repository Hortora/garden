### Symptom
CI (building main) reports compilation errors for code that local file inspection confirms is correct. The error references a class, method, or import path you can see has already been fixed.

### Context
A fix was applied to resolve a breaking API change (e.g. a moved class). The fix was committed to a feature branch. Local files reflect the fix because the session is checked out on the feature branch. CI runs against main, which never received the commit.

### What was tried
Inspecting local files to confirm the fix was applied — both files showed the correct import. CI continued to fail with the old import path.

### Root cause
The fix commit exists on the feature branch only. `git branch --all --contains <sha>` reveals this immediately: only the feature branch is listed, not main. Local file state reflects the checked-out branch, not main — so local inspection gives false confidence while CI builds a different commit.

### Fix
```bash
# Verify where a fix commit actually lives
git branch --all --contains <sha>

# If only on the feature branch — merge to main and push
git checkout main
git merge --ff-only <feature-branch>
git push origin main
```

### Why this is non-obvious
Looking at local files confirms the fix is applied. The natural assumption is that if the code is correct locally, CI should agree. The disconnect — that local file state reflects the current branch, not main — is easy to overlook when debugging CI failures. The investigator confirms the fix is present, concludes CI is wrong or stale, and loses time before realising the commit was never merged.
