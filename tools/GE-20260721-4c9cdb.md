### Symptom
Branch triage reported thousands of "missing" lines across dozens of files, making it
appear that entire features were stranded on closed branches. Manual inspection showed
the flagged content was actually main's later evolution, not lost branch work.

### Context
Auditing 47 branches with false "landed on main" stamps across 36 repos. The initial
triage script used `git diff main..branch` to classify files as `+` (branch-only),
`-` (main-only), or `~` (both differ), then dismissed `~` files as "stale branch noise."
This missed genuinely stranded content inside shared files.

### What was tried (didn't work)
`git diff --name-only main branch` — classifies files by presence but cannot distinguish
"branch changed this file and main has a different version" from "main added content the
branch predates." The two-dot diff (`main..branch`) includes main's full evolution since
the merge-base, inflating results by orders of magnitude.

### Root cause
`git diff main..branch` (two-dot) computes the symmetric difference — it shows ALL
differences between the two tips, including commits on main that have nothing to do with
the branch. For stale branches, main may have hundreds of commits the branch doesn't have,
making every shared file appear to differ.

### Fix
Use the merge-base to isolate what the branch actually introduced:

```bash
MERGE_BASE=$(git merge-base main branch)
git diff "$MERGE_BASE..branch"           # only what the branch changed
```

This shows the branch's delta from the common ancestor — the correct question for
"what did this branch contribute?" rather than "how do these two tips differ?"

### Why this is non-obvious
The two-dot syntax is the default `git diff` behaviour and reads naturally as "diff between
main and branch." The mental model ("show me what's different") is correct but the question
is wrong — "what's different between tips" ≠ "what did the branch introduce." The merge-base
approach requires understanding git's DAG model, which many developers skip.

### Alternatives considered
- `git diff --name-only` with file-presence classification — misses modifications to shared files entirely
- `git log main..branch` to find branch-specific commits, then diff each — correct but expensive and complex
- Three-way merge-tree (`git merge-tree`) — shows conflicts but not clean content differences
