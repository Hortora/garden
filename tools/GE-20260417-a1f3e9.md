## What it does

When using `@Permute(inline=false, keepTemplate=true)` with the Permuplate Maven plugin, the **template class itself is written as a separate file** to `target/generated-sources/permuplate/`, alongside the generated permuted classes.

This only matters for templates in `src/main/permuplate/` (the plugin's template directory, which is **not** on the compile path). Without this behaviour, the template class would be uncompiled and inaccessible to other code in the same module.

## When it matters

When the template class is **directly referenced by other generated code**. For example:

- `NegationScope` (template) + `ExistenceScope` (generated from it)
- `JoinBuilder` references both — `NegationScope` must be compiled

Without `keepTemplate=true`, only `ExistenceScope.java` would appear in generated sources; `NegationScope` would be absent and compilation would fail with "cannot find symbol: class NegationScope".

## APT contrast

In APT mode (`src/main/java/`), the template class is already on the compile path and gets compiled naturally. `keepTemplate=true` has no effect in APT mode — it's a no-op documented with a warning. The Maven plugin's behaviour diverges here: the template is in a non-compiled directory, so explicit output is required.

## Trigger pattern

```java
@Permute(varName = "T", values = {"Existence"}, className = "${T}Scope",
         inline = false, keepTemplate = true)
public class NegationScope<OUTER, DS> { ... }
```

Outputs to `target/generated-sources/permuplate/`:
- `NegationScope.java` (kept template, annotations stripped)
- `ExistenceScope.java` (generated)

## Why undocumented

The feature was added specifically to handle the case where a top-level non-inline template class is itself referenced by sibling generated code. There is no documentation for this distinction — the APT behaviour (no-op) and Maven plugin behaviour (explicit output) differ silently.
