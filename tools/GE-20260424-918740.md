## Symptom

Maven fails to parse `pom.xml` with an opaque error during a normal build:

```
[FATAL] Non-parseable POM /path/to/pom.xml: in comment after two dashes (--)
next character must be > not a (position: END_TAG seen ...some text --a... @34:15)
```

The build was working before. No XML structure was changed — only a comment was added.

## Context

Adding descriptive text inside an XML comment (`<!-- ... -->`) that happens to
contain two consecutive hyphens (`--`). Common when copy-pasting error messages,
command-line flags, or markdown content into comments.

Example triggering content:
```xml
<!-- KNOWN ISSUE: bootstrap fails with "Unsupported value type: [ALL-UNNAMED]"
     --add-opens args cause Json.appendValue() to fail -->
```

## Root Cause

The XML specification forbids `--` anywhere inside an XML comment. The string `-->`
ends a comment, but `--` alone (not followed by `>`) is also illegal and causes a
parse error. Maven's XML parser correctly rejects it; the error message is
misleading because it points to the character *after* the `--`, not the `--` itself.

## Fix

Replace `--` with an alternative that preserves readability:

```xml
<!-- KNOWN ISSUE: bootstrap fails with "Unsupported value type: [ALL-UNNAMED]".
     The add-opens args in the Jlama extension cause Json.appendValue() to fail. -->
```

Or use a Unicode dash (—) if the `--` is cosmetic, or break the comment to
avoid the double-hyphen entirely.

## Why Non-Obvious

XML comments look like HTML comments where `--` is harmless. Most developers
assume the only special sequence inside a comment is `-->`. The error message
reports the character *after* the `--`, making it appear the problem is
elsewhere in the file. The actual offending `--` may be many lines away from
the reported position.
