## Symptom

`actions/checkout@v6` with `ref: <7-or-40-char-SHA>` and default `fetch-depth: 1` fails after 3 retries with `Process completed with exit code 1`. No error message identifies the cause — the failure looks like a transient network issue.

## Context

Occurs when pinning a dependency checkout to a specific commit SHA for reproducibility, e.g.:

```yaml
- uses: actions/checkout@v6
  with:
    repository: org/repo
    ref: 9707bc5
    path: my-repo
```

## Root Cause

With `--depth=1` (shallow clone), the action tries to fetch:
- `refs/heads/9707bc5*`
- `refs/tags/9707bc5*`

Neither refspec pattern matches a raw commit SHA. Branch and tag names are valid; commit SHAs are not. The fetch silently fails because no matching refs exist.

## Fix

Use a branch name, a tag, or omit `ref` entirely to use the default branch:

```yaml
# Works — branch name
- uses: actions/checkout@v6
  with:
    repository: org/repo
    ref: main

# Works — tag
- uses: actions/checkout@v6
  with:
    repository: org/repo
    ref: v1.2.3

# Works — omit ref, checks out default branch
- uses: actions/checkout@v6
  with:
    repository: org/repo
```

If reproducibility is the goal, use a tag rather than a SHA. Tags are immutable by convention and are fetchable with shallow clones.

## Why This Is Non-Obvious

The error message (`exit code 1`) gives no indication that the `ref` format is the problem. Developers assume commit SHAs are a valid `ref` value — they are, for full clones — but shallow clone refspec matching only works on named refs.
