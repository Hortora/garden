## Symptom

Need to verify the actual API of a class in a SNAPSHOT dependency jar — whether it has specific annotations (`@Alternative`, `@Priority`), what its method signatures are, or whether a field became a record accessor. Reading the source repo is unreliable (source may not match the published SNAPSHOT).

## Context

During SNAPSHOT dependency drift resolution, the consumer needs to verify what the published jar actually contains, not what the source repo currently shows. The jar is in `~/.m2/repository/` but extracting and running `javap` requires multiple steps.

## Fix

Use `ide_read_file` with the `qualifiedName` parameter:

```
ide_read_file(qualifiedName="io.casehub.platform.oidc.SecurityIdentityCurrentPrincipal",
              project_path="/path/to/consumer/project")
```

IntelliJ decompiles the `.class` file from the dependency jar and returns readable Java source with all annotations, method signatures, and field types visible. Works for any class on the project's classpath — including transitive dependencies.

Combine with `ide_find_class(scope="project_and_libraries")` to discover the fully qualified name first when the package has changed.

## Why this is non-obvious

The natural approach is to download the jar, extract with `jar xf`, and run `javap -p`. This requires multiple shell commands, a temp directory, and doesn't produce readable Java. `ide_read_file` with `qualifiedName` does it in one call with full decompilation.
