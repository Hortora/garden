## Technique

In tests that assert numeric values derived from game/simulation mechanics, use the same data source as the implementation — not hardcoded literals. When a mechanic changes, the test updates at a single source of truth.

## Context

When building a physics/simulation engine with configurable stats (damage, armour, cooldowns), tests often assert exact output values. When a mechanic is later added or corrected, all tests that hardcoded the old value break simultaneously with no indication of which mechanic changed.

Example: Probe damage to Zealot shields was 5. After adding Zealot armour (1), effective damage became 4. Two tests asserted `shields == 50 - 5 = 45` and broke immediately.

## The pattern

```java
// Brittle — breaks when armour is introduced:
assertThat(zealot.shields()).isEqualTo(50 - 5); // hardcoded damage

// Robust — survives mechanic changes:
int effective = SC2Data.damagePerAttack(UnitType.PROBE) - SC2Data.armour(UnitType.ZEALOT);
assertThat(zealot.shields()).isEqualTo(SC2Data.maxShields(UnitType.ZEALOT) - effective);
```

The assertion now documents the formula (`damage - armour = effective`) rather than a magic number, and automatically stays correct when stats are updated.

## Why this is non-obvious

The instinct is to compute the expected value mentally, write it as a literal, and move on. The brittle assertion still passes all tests until the mechanic is introduced — at which point the failure looks like a test regression, not a doc assertion that needs updating. Seeing many simultaneous failures makes the root cause harder to identify.

## Where to apply

Any test where the expected value is derived from a configurable stat, formula, or constant in the production code. Use the production constant/method in the test assertion.
