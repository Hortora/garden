## Resolve cherry-pick conflict by checking out the mature-branch version of the whole file

When cherry-picking a commit onto an older base (e.g. upstream/main) where the
file has since evolved significantly on your fork, git produces multiple conflict
markers across the file — one per diverged hunk. Resolving each manually is tedious
and error-prone when the final correct state is already known.

### Fix

```bash
# After cherry-pick reports a conflict:
git checkout <mature-branch> -- <file>
git add <file>
git cherry-pick --continue --no-edit
```

Takes the fully-evolved state of the file from your fork as the resolution,
bypasses all conflict markers, and continues the cherry-pick.

### Why this is non-obvious

The instinct is to open the file and resolve each `<<<<<<<` / `=======` / `>>>>>>>`
block individually. When a file has been significantly refactored on one branch
(e.g. constructor → builder, event-bus → direct invocation), this produces 5–10
separate conflict regions that are interdependent. The `git checkout -- <file>`
approach treats the conflict as a whole-file decision, which is semantically correct
when you know the mature-branch state is right.

### Context

Applies when cherry-picking individual commits from a fork's main branch onto
an upstream base that is several commits behind. The cherry-picked commit touches
a file that was refactored in intermediate commits not being cherry-picked.
