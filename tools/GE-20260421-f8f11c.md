### The problem

A PR branch accumulated many commits over days. Main moved forward with
competing changes to the same shared index file (e.g. GARDEN.md). Rebasing
commit-by-commit would create one merge conflict per commit — 27 commits =
27 conflict resolutions, each needing manual edits.

### The technique

Instead of rebasing, squash the entire branch onto the latest main in one
commit:

```bash
# 1. Get the list of files the branch adds (no GARDEN.md yet)
git diff origin/main...<branch-tip> --name-only | grep -v GARDEN.md

# 2. Reset the PR branch to current main
git reset --hard origin/main

# 3. Restore all new entry files from the branch tip in one shot
git checkout <branch-tip> -- <file1> <file2> ...

# 4. Restore main's GARDEN.md (not the branch version)
git checkout origin/main -- GARDEN.md

# 5. Manually add only the new index entries to GARDEN.md
#    (from git diff output — you already know what's new)

# 6. Commit everything as a single squash commit
git add .
git commit -m "feat: batch consolidation — N entries"

# 7. Force-push to the PR branch
git push --force-with-lease origin <branch>
```

### Why it works

The 27 individual conflicts arise because each commit modified GARDEN.md, and
rebasing replays each modification against an incompatible base. By discarding
the commit history and building the final state directly, there's only one
operation touching GARDEN.md — so only one conflict to resolve (if any).

### When to use

- PR has many commits, all adding new files + updating a shared index file
- Main has moved significantly since the branch was created
- The commits are logically a single batch (no partial-state value)

### Why non-obvious

The instinct is to rebase and resolve each conflict in sequence. Squashing
feels like "losing history" but for a sweep/batch PR the commit-by-commit
history has no value — the entries themselves are the history. The final state
is all that matters.
