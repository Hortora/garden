## Symptom

After running `sed -i '' 's/1.0.0-SNAPSHOT/2.0.0-SNAPSHOT/g' deployment/pom.xml` to update a dependency version, the deployment module fails to build: "Could not find artifact parent:2.0.0-SNAPSHOT". The parent version was also changed, not just the dependency.

## Context

Occurs in multi-module Maven projects where the parent pom version and a dependency version happen to share the same value (e.g. both are `1.0.0-SNAPSHOT`). A global sed replace changes ALL occurrences — including the `<parent><version>` block.

## Root Cause

`sed -i '' 's/OLD/NEW/g'` replaces every occurrence of the string in the file with no awareness of XML context. A pom.xml typically has at minimum two `1.0.0-SNAPSHOT` occurrences: the `<parent><version>` and any `<dependency><version>` blocks for sibling modules.

## Fix

Target the specific block with context-aware sed or use XMLStarlet:

```bash
# Option 1: use XMLStarlet to update only the dependency version
xmlstarlet ed -u "//dependencies/dependency[artifactId='quarkus-ledger']/version" \
  -v "0.2-SNAPSHOT" pom.xml

# Option 2: scope the sed to lines after the dependency groupId appears
grep -n "quarkus-ledger" pom.xml  # find the line, then manually edit
```

## Why This Is Non-Obvious

The parent version declaration comes first in the file and looks visually identical to a dependency version. The build error ("Could not find artifact") points to the parent resolution failure, which suggests the parent was deleted or the repo is broken — not that a sed command changed its version.
