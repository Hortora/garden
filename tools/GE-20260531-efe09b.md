## What It Is

The `before_tool_call` plugin hook in OpenClaw fires before every tool invocation. The event payload is not fully enumerated in the official docs but confirmed via source code and GitHub issues:

```typescript
interface ToolCallEvent {
  agentId: string;      // the agent making the tool call
  sessionKey: string;   // current session identifier
  toolName: string;     // exact registered name of the tool (e.g. "casehub_escalate")
  toolCallId: string;   // unique ID for this invocation
  params: Record<string, unknown>; // the tool's input arguments
}
```

The `toolName` field matches the tool's registered name exactly — making it suitable for exact-name interception (e.g. catching a specific tool before it executes).

## Discovery

Found via GitHub issue #5943 (internal wiring) which shows:

```js
runBeforeToolCall(toolCall.name, toolCall.id, validatedArgs)
```

And confirmed via issue #48503 (`toolKind` and `toolInputKind` are additional discriminators the host can provide, separate from `toolName`).

The official hooks documentation at `docs.openclaw.ai/plugins/hooks` lists `before_tool_call` without enumerating the event payload fields.

## Usage

```typescript
api.on("before_tool_call", async (event: ToolCallEvent) => {
  if (event.toolName === "casehub_escalate") {
    // intercept escalation before it runs
    clearTurnCommitment(event.agentId);
  }
});
```

This is stable: `toolName` is set from the tool's registered name and is not affected by `toolKind`/`toolInputKind` discriminators (those are additional host-level metadata).

## Caveats

`agentId` and `sessionKey` were added in a later patch (GitHub issue #19381) — absent in very early 2.x builds. The `params` field contains the tool's validated input arguments, not raw input.

The `PluginApprovalResolution` return type (allow-once / allow-always / deny / timeout / cancelled) is typed in the SDK — use it rather than string comparisons.
