## Context

A project repo has accumulated workspace-only files in its history (session
handovers, blog entries, personal methodology docs) that have no place in a
shared codebase. You want to remove these files from all commits without losing
any code changes — including commits that mixed code and workspace files in
the same commit.

## Technique

Use `git filter-repo` with `--invert-paths` to strip specific files from every
commit in history, then `--prune-empty always` to drop commits that become
empty after the filtering:

```bash
git filter-repo \
  --path HANDOFF.md --invert-paths \
  --path-glob 'docs/_posts/*.md' --invert-paths \
  --path-glob 'blog/*.md' --invert-paths \
  --prune-empty always
```

This handles mixed commits correctly — if a commit touched both `HANDOFF.md`
and real Java files, the Java changes survive and only `HANDOFF.md` is removed.
The commit is preserved (not dropped) because it still has content after filtering.

A commit that touched *only* workspace files becomes empty after filtering and
is dropped automatically by `--prune-empty always` — no need to classify or
manually identify "drop-only" commits.

## Why this is non-obvious

Most developers reach for trying to classify which commits to drop vs keep, then
cherry-pick the keepers. That approach breaks on mixed commits. `filter-repo`
operates at the file level across all commits simultaneously, making the
classification problem disappear entirely.

`git filter-branch` is the older alternative but is orders of magnitude slower
and deprecated. `git filter-repo` requires a separate install but is the
correct modern tool.
