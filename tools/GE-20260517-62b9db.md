### Symptom
A HTML file with an empty `<article>` element (whitespace only) is parsed
with BeautifulSoup using `html.parser`, processed, then written back with
`str(soup)`. The output contains the article's surrounding structure
(e.g. archive header text) repeated many times across new `<p>` elements
that weren't in the original HTML.

### Context
Occurred in Sparge's enrichment pipeline. Source HTML had the structure:
```html
<body>
  <header class="archive-header"><p>Archived from ...</p></header>
  <article class="blog-post">  </article>
</body>
```
After enrichment (`str(soup)` written to enriched path), the output had
dozens of `<p>At the request of others I've setup a twitter account.</p>`
lines — the archive header text repeated — with no `<article>` wrapper.

### Root cause
When `html.parser` serialises a document with an empty `<article>` and
sibling block elements, it re-parses the structure according to HTML5
implicit closing rules. Block-level siblings of the empty element get
re-homed, and certain content duplicates across what become implicit
anonymous paragraphs. The lxml parser handles the same HTML cleanly.

### Fix
**Option 1 — Guard against empty articles before processing:**
```python
article = soup.find('article')
if article and not article.text.strip():
    article = None  # fall back to body
article = article or soup.find('body') or soup
```

**Option 2 — Use lxml parser instead of html.parser:**
```python
soup = BeautifulSoup(html, 'lxml')  # not 'html.parser'
```
lxml doesn't exhibit the re-serialisation duplication. However, lxml
normalises some HTML differently — test for your specific content.

**Option 3 — Flag empty articles as an error before processing:**
Add a scan check: if `<article>` exists but `article.children` is empty
and `article.own_text` is blank, raise an error rather than proceeding.

### Why this is non-obvious
The symptom (repeated text in output) has no obvious connection to an
empty input element. The same HTML processed with lxml produces correct
output, which makes it look like a content problem rather than a parser
problem. The error only appears after the full enrichment pipeline runs
and the file is written — not at parse time.
