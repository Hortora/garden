### Symptom
After using `ide_move_file` to move multiple classes from one package to another,
compilation fails with `package X does not exist` errors — even though all classes
are now in the same new package.

### Context
When extracting a group of related classes (e.g. AlphaDataSource, TypeNode,
FilterNode, FanOutProcessor) from `io.casehub.platform.datasource.memory` to
`io.casehub.platform.datasource.alpha`, each class retains import statements
pointing to the old package for its co-moved siblings:

```java
// In AlphaDataSource.java (now in datasource.alpha package):
import io.casehub.platform.datasource.memory.TypeNode;    // STALE — TypeNode is now in same package
import io.casehub.platform.datasource.memory.FanOutProcessor; // STALE
```

### Root cause
`ide_move_file` processes each file independently. When it moves AlphaDataSource,
it updates all *external* references (files NOT being moved) to point to the new
package. But it does not clean up imports between files that are part of the same
bulk move — it doesn't know they'll end up in the same package.

Since the co-moved classes are now in the same package, the imports should be
removed entirely (same-package classes don't need imports in Java).

### Fix
After a bulk move, run `ide_optimize_imports` on each moved file:

```
ide_optimize_imports(file="new/package/AlphaDataSource.java")
ide_optimize_imports(file="new/package/TypeNode.java")
ide_optimize_imports(file="new/package/FilterNode.java")
```

If `ide_optimize_imports` also fails (it can when the new module isn't yet in the
local Maven repo), use `ide_replace_text_in_file` to remove the stale import lines
directly:

```
ide_replace_text_in_file(
    file="new/package/AlphaDataSource.java",
    searchText="import old.package.TypeNode;\nimport old.package.FanOutProcessor;\n",
    replaceText="")
```

### Why this is non-obvious
`ide_move_file` correctly updates all external references — the only failures are
between the moved files themselves. The error messages point to the old package,
which no longer contains any classes, making it look like the move failed entirely
rather than partially.

### Alternatives considered
- Moving classes one at a time with `ide_optimize_imports` between each move — works
  but tedious and risks intermediate compile failures.
- Using `ide_refactor_rename` on the package instead of moving individual files —
  not supported by the MCP tool for package-level operations.
