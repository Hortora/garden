## What Exists But Is Undocumented

The `com.github.ocraft.s2client.protocol.data.Abilities` enum in ocraft 0.4.21 is missing constants for at least three building actions:

| Action | Expected constant | Status |
|--------|-------------------|--------|
| Build Templar Archives | `Abilities.BUILD_TEMPLAR_ARCHIVES` | Missing |
| Lurker Den research (morph trigger) | `Abilities.RESEARCH_LURKER_LEVEL` | Missing |
| Build Nydus Canal (worm exit) | `Abilities.BUILD_NYDUS_CANAL` | Missing |

These buildings exist in the SC2 game and in the `Units` enum (e.g. `Units.ZERG_NYDUS_CANAL`, `Units.ZERG_LURKER_DEN_MP`) but their corresponding `Abilities` build/morph constants are absent from the enum.

## How Discovered

Expanding `ActionTranslator.mapBuildAbility(BuildingType)` to cover all 48 BuildingType values caused compilation failure:

```
cannot find symbol
  symbol:   variable BUILD_TEMPLAR_ARCHIVES
  location: class com.github.ocraft.s2client.protocol.data.Abilities
```

The missing constants are not mentioned anywhere in the ocraft documentation, the sc2client-proto README, or the SC2 API docs. Only a compilation attempt reveals the gap.

## Workaround

Map the affected building types to `null` in the switch statement. These actions are only dispatched when the agent actively issues a build command — in a Protoss/Terran-only implementation they will never be reached:

```java
case TEMPLAR_ARCHIVES -> null;   // no Abilities constant in ocraft 0.4.21
case LURKER_DEN       -> null;   // ditto
case NYDUS_CANAL      -> null;   // ditto
```

Guard the call site against null before dispatching:

```java
Abilities ability = ActionTranslator.mapBuildAbility(intent.buildingType());
if (ability != null) {
    actionInterface.unitCommand(unit, ability, point, false);
}
```

## Why This Is Non-Obvious

The `Units` enum contains the corresponding unit type constants for all three buildings, making it reasonable to assume the `Abilities` enum is complete. There is no deprecation notice, no warning, and no stub constant — the symbol simply does not exist. The only signal is a compilation error.
