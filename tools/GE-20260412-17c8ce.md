## Enable GitHub Pages Actions source and fix branch protection via gh API — no browser

**ID:** GE-20260412-17c8ce
**Stack:** GitHub CLI (gh 2.x+), GitHub Pages API
**Labels:** `#ci-cd` `#github-pages` `#github-cli`
**What it achieves:** Enables GitHub Pages with the GitHub Actions build source and
whitelists a branch for deployment — fully scriptable, no browser required.
**Context:** Setting up GitHub Pages for a repo that requires Jekyll 4+ (incompatible
with "Deploy from branch"), or automating Pages setup in a script. Two separate API
calls are needed: one to enable Pages, one to fix the branch protection that blocks deploy.

### The technique

```bash
# 1. Enable Pages with GitHub Actions source
gh api repos/OWNER/REPO/pages --method POST --field build_type=workflow

# 2. When deploy job fails: "Branch not allowed to deploy to github-pages"
gh api repos/OWNER/REPO/environments/github-pages/deployment-branch-policies \
  --method POST --field name=main
```

The first call enables Pages and returns the current config:
```json
{
  "build_type": "workflow",
  "html_url": "https://owner.github.io/repo/"
}
```

The second call adds `main` to the allowed deployment branches for the auto-created
`github-pages` environment. This environment is created automatically when you first
enable Pages with Actions source, and it defaults to `custom_branch_policies: true`
with no branches whitelisted — causing the first deploy to always fail.

### Why this is non-obvious

The deploy failure message ("Branch 'main' is not allowed to deploy to github-pages
due to environment protection rules") gives no indication that the fix is an API call
to add a branch policy. The logical place to look — repo Settings → Pages — shows no
branch protection controls for the auto-created `github-pages` environment. Those live
in Settings → Environments, a separate section. Most developers spend time in Pages
settings looking for the fix before discovering the Environments section, or resort to
deleting and re-creating the Pages configuration.

### When to use it

- Scripted repo setup (post-create hooks, terraform, etc.)
- Switching an existing repo from "Deploy from branch" to Actions-based Pages
- CI that provisions new repos and needs Pages configured without manual steps
- Any time the deploy job fails with the branch protection error after enabling Actions-based Pages
