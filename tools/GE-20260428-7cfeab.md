## Symptom

A transparent `PlaneGeometry` mesh exists in the Three.js scene (confirmed via scene graph). `raycaster.intersectObjects()` hits it. NDC projection checks show it is within the camera frustum. Yet the rendered pixel at that position shows the opaque background colour — not the transparent plane's colour.

## Context

Creep tiles (Zerg ground overlay) were implemented as `THREE.PlaneGeometry` with `MeshBasicMaterial({ color: 0x9030c0, transparent: true, opacity: 0.6 })` placed at `y=0.02`. A large opaque ground plane (`MeshLambertMaterial`) was placed at `y=0.04` — 0.02 world units above the creep tiles. This meant creep was below the ground plane, completely occluded by depth testing.

## What was tried

- Confirmed `creepTileCount() > 0` in scene graph ✓
- Confirmed NDC projection within frustum ✓
- Moved creep to `y=0.06` (above the ground plane at `y=0.04`) — still not visible
- Added brighter colour — still not visible

## Root cause

`depthTest: true` (the Three.js default) causes WebGL to discard fragments that fail the depth test — i.e. fragments that are further from the camera than what's already in the depth buffer. The opaque ground plane at `y=0.04` writes to the depth buffer. The creep plane at `y=0.02` (below it) fails the depth test and is discarded. Even after moving creep to `y=0.06`, `depthWrite: false` on the creep material combined with the draw order caused the ground plane to win.

## Fix

Set `depthTest: false` on the transparent plane's material and assign a high `renderOrder` so it renders after all opaque geometry:

```javascript
const mesh = new THREE.Mesh(
  new THREE.PlaneGeometry(TILE * MARKER_SCALE, TILE * MARKER_SCALE),
  new THREE.MeshBasicMaterial({
    color: 0x9030c0, transparent: true, opacity: 0.7,
    depthWrite: false,
    depthTest: false,   // never occluded by other geometry
    side: THREE.DoubleSide
  })
);
mesh.renderOrder = 10;  // render after opaque geometry
mesh.position.set(wp.x, 0.08, wp.z);
```

## Why this is non-obvious

`depthTest: false` sounds like it would cause z-fighting or incorrect layering. In practice, for a ground-level overlay that should always be visible above the terrain, disabling depth testing is correct and intentional. The symptom (plane exists but renders as background colour) is identical to a positioning error, so debugging focuses on positions first — which is the wrong axis.
