## What it is

Java's `Instant.EPOCH` serialized via Jackson can produce either of:
- `"1970-01-01T00:00:00Z"` (no milliseconds)
- `"1970-01-01T00:00:00.000Z"` (with milliseconds)

The format depends on the Jackson version or the serialization configuration in use. Checking
`value === "1970-01-01T00:00:00Z"` breaks silently when the consuming service upgrades Jackson
or enables millisecond precision.

**Better:** `Date.parse(value) === 0` — returns `0` for epoch regardless of millisecond precision.

## How to use it

```typescript
function isEpoch(iso8601: string): boolean {
  return Date.parse(iso8601) === 0;
}

// In practice:
function formatIdle(lastChannelActivity: string): string {
  if (Date.parse(lastChannelActivity) === 0) {
    return "No activity recorded yet.";
  }
  const elapsedMin = Math.floor((Date.now() - Date.parse(lastChannelActivity)) / 60_000);
  return `No activity in the last ${elapsedMin} minute(s).`;
}
```

## Why it's non-obvious

The string format looks stable — `Instant.EPOCH` is a constant, why would its serialized form
change? In practice, Jackson's `JavaTimeModule` has changed its defaults between major versions,
and some configurations explicitly enable/disable millisecond serialization. The string
comparison works perfectly in testing (where the Jackson version is pinned) and breaks in
production after a routine dependency bump.

`Date.parse()` returning `0` for epoch is guaranteed by the ECMAScript specification — it
parses both variants correctly and the epoch is the only instant that returns `0`.

## Gotcha variant

This also applies when the epoch sentinel is used server-side as a "null" signal (e.g. "no
activity ever recorded"). String comparison against a hard-coded `"1970-01-01T00:00:00Z"` is
fragile; `Date.parse() === 0` (TypeScript) or
`value.equals(Instant.EPOCH)` after deserializing (Java) are stable alternatives.
