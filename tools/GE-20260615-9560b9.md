### What

After any rebase or squash, run a scope scan before pushing:

```bash
git diff origin/main..HEAD --name-only
```

Review every file in the output. Any file outside the expected directories is an unintended inclusion — catch it here, not in code review.

### Why this is non-obvious

Developers routinely do `git diff` to review content, but not a systematic scope check for *which files* are included. During a rebase or squash, commits from unrelated branches can silently merge into the diff — wrong files, wrong directories, regressions to previous PRs. The content diff is long; the stray file hides in it. The `--name-only` scan takes five seconds and makes the file list impossible to miss.

### Root cause of the problem

Rebase replays commits in sequence. If any commit in the chain touched a file that shouldn't be in this PR (e.g. a CI workflow change from a previous merge, a spec file, an unrelated module), it lands silently in the squashed result. Reviewers are the first to see it — expensive.

### The rule

`git diff origin/main..HEAD --name-only` is mandatory after every rebase or squash, before every push. Flag anything outside expected scope immediately. Do not skip it to save time.

### Enforcement — global pre-push hook

Discipline fails under pressure. Automate it instead with a global `pre-push` hook that blocks the push when always-suspicious files (`.github/`, `.gitignore`, `LICENSE`, `CODEOWNERS`, `.editorconfig`, `.mvn/`) appear in the diff:

```sh
# ~/.config/git/hooks/pre-push  (global via git config --global core.hooksPath)
#!/bin/sh
SUSPICIOUS=$(git diff origin/main..HEAD --name-only 2>/dev/null | grep -E \
  "^\.github/|^\.gitignore$|^\.gitattributes$|^LICENSE|^CODEOWNERS$|^\.editorconfig$|^\.mvn/")

if [ -n "$SUSPICIOUS" ]; then
  echo "⚠ Out-of-scope files detected:"
  echo "$SUSPICIOUS"
  exit 1
fi
```

Register globally:
```bash
mkdir -p ~/.config/git/hooks
chmod +x ~/.config/git/hooks/pre-push
git config --global core.hooksPath ~/.config/git/hooks
```

Use `--no-verify` only when the out-of-scope file is genuinely intentional (e.g. a PR that actually updates CI).
