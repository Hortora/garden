## git cherry-pick -X ours brings in new files cleanly but silently discards field additions in conflicted files

### Symptom

After `git cherry-pick -X ours <SHA>`, new files from the commit appear correctly on the target branch. However, field additions, method additions, or other modifications to existing files that conflicted are silently absent — the target branch's version of those files was kept intact.

### Root cause

`-X ours` resolves every conflict by preferring the current branch's side. For files that don't exist on the current branch (new files), there is no conflict — they land cleanly. For files that exist on both sides and conflict, the current branch's version wins entirely, discarding all changes from the cherry-picked commit to those files.

### Fix

After a `-X ours` cherry-pick, audit the diff to find which existing files were modified by the cherry-picked commit and manually apply the missing changes:

```bash
# See what the original commit changed (only modified/deleted files)
git diff <target-base>..<commit-sha> --name-status | grep "^M"

# Apply the missing additions manually to each modified file
git show <commit-sha> -- <conflicted-file>.java | patch -p1
```

### Why this is non-obvious

New files succeed visibly; modified-file changes fail silently. The branch compiles cleanly because the target branch's version of the file is valid — only the cherry-picked additions are missing. There is no error or warning. The missing fields/methods are only discoverable by reading the original commit's diff against the conflicted files.
