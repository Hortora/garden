### Symptom

After a migration was declared complete (protocol files moved from `casehubio/parent/docs/protocols/` to `casehubio/garden`), sessions kept writing new protocol entries to the old location. No error was produced. The duplicate/wrong-location writes continued for weeks before being noticed.

### Context

The migration was announced in a commit message ("fully migrated to casehubio/garden") but the old directory was never deleted. New AI sessions read PLATFORM.md, which still said "add it to `docs/protocols/`", found the directory present and writable, and wrote there successfully.

### Root cause

Two conditions together cause silent continued writes:
1. The authoritative documentation (`PLATFORM.md`) still referenced the old location
2. The old destination (`docs/protocols/`) was never deleted — it kept accepting writes without error

Neither condition alone is fatal. Together, they create an invisible split — writes go to both locations over time, and the "migrated" state is only nominal.

### Fix

A migration is only complete when:
1. The old destination **no longer exists** (or is explicitly read-only)
2. All documentation references to the old destination are updated to the new one
3. Any tooling/skill that creates artifacts in the old location is updated

Deleting the old directory is the critical step — it converts a silent mismatch into an obvious error on next write.

### Why this is non-obvious

"Migration complete" conventionally means content has been moved. It does not conventionally mean the old container has been destroyed. Developers checking "is everything in the new location?" will answer yes — without noticing that new writes still go to the old one.
