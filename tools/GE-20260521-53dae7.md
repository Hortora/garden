## Symptom

A script records `stash@{0}` after calling `git stash`, intending to pop that specific entry later. On a clean working tree, it pops the wrong stash — the one that was already on the stack before the call.

```bash
[ "$DIRTY" -gt 0 ] && git stash && STASH_REF="stash@{0}"
# Later:
git stash pop "$STASH_REF"   # Pops wrong entry if repo was clean
```

## Root cause

`git stash` exits 0 even when there is nothing to stash. It prints "No local changes to save" to stderr and exits cleanly. The `&&` in `A && git stash && STASH_REF="stash@{0}"` fires the third clause regardless — `stash@{0}` is recorded as if a new stash was created, but it actually points at whatever was already at position 0 on the stack.

## Fix

Check the output of `git stash`, not just its exit code:

```bash
stash_out=$(git stash)
echo "$stash_out" | grep -q "Saved working" && STASH_REF="stash@{0}" || STASH_REF=none
```

Then guard the pop:
```bash
[ "$STASH_REF" != "none" ] && git stash pop "$STASH_REF"
```

## Why this is non-obvious

`git stash` failing silently on a clean tree is expected. What's unexpected is that the exit code gives no way to distinguish "stashed successfully" from "nothing to stash" — both return 0. The only reliable signal is the output string, which requires reading stderr output to detect.
