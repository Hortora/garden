### Symptom

A `squash` in a non-interactive git rebase produces a commit whose subject line is the *first* commit's message and whose body contains both messages concatenated. No editor opens, no error is raised.

### Context

Non-interactive rebase session (CI, headless automation, Claude sessions without a terminal editor). `GIT_SEQUENCE_EDITOR` is set to control the todo file, but `GIT_EDITOR` is not configured. A `squash` entry appears in the todo — intending to merge two commits.

### What was observed

```
Rebasing (2/3)
[detached HEAD 9391a42] fix: move WorkflowExecutor back to engine module   ← first commit's subject
```

The resulting commit message body contained both original messages appended:

```
fix: move WorkflowExecutor back to engine module — [first message body]

fix: add serverlessworkflow deps — [second message body]
```

### Root cause

`squash` normally opens an editor to let the user write a combined message. With no editor, git cannot prompt — it falls back to concatenating both messages with the first commit's subject as the result subject. This behaviour is not documented in `git rebase` man pages or `git help rebase`.

### Fix

Use `fixup` instead of `squash` when you want to discard the squashed commit's message entirely (keeps only the first message):

```
pick abc1234  the commit you want to keep the message of
fixup def5678  the commit whose message you want to discard
```

Or use `exec git commit --amend -F /tmp/msg.txt` after a `pick` if you want a fully controlled combined message:

```
pick abc1234  first commit
pick def5678  second commit  
exec git commit --amend -F /tmp/combined_msg.txt
```

### Why this is non-obvious

The git documentation for `squash` only describes the interactive flow where an editor opens. The non-interactive fallback (concatenate + use first subject) is undiscoverable from any official documentation — found only through observing the actual output in a non-interactive session.
