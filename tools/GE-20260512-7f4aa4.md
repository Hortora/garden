### Context

When integrating with a library whose source isn't available locally, you need to know: Is this bean `@Alternative`? What `@Inheritance` strategy does this JPA entity use? What CDI scope is it in? `javap -verbose` reads the compiled bytecode and gives definitive answers faster than searching docs or remote source.

### Technique

```bash
# Check CDI annotations (scope, @Alternative, qualifiers)
javap -verbose -cp path/to.jar io.example.SomeBean \
  | grep -A2 "RuntimeAnnotation\|Alternative\|ApplicationScoped\|Singleton"

# Check JPA inheritance strategy and table mapping
javap -verbose -cp path/to.jar io.example.BaseEntity \
  | grep -A2 "Inheritance\|DiscriminatorColumn\|Table"

# List all annotation constants on a class (broad scan)
javap -verbose -cp path/to.jar io.example.SomeClass \
  | grep "Utf8.*L[a-z].*;"

# Inspect multiple classes at once (colon-separated classpath)
javap -verbose -cp jar1.jar:jar2.jar io.example.ClassA io.example.ClassB

# Locate the right JAR in Maven local repo first
find ~/.m2/repository/io/example -name "*.jar" ! -name "*sources*" ! -name "*javadoc*"

# List all classes in a JAR to find the right package
jar tf path/to.jar | grep "\.class" | sort
```

### Why this is non-obvious

Most developers search the dependency's GitHub repo, read Javadoc, or look for documentation. `javap -verbose` reveals what's **actually compiled** into the artifact — which matters when:

- Annotation processors add annotations at compile time (not visible in source)
- The library uses a private repo or is closed-source
- The docs don't mention CDI/JPA details
- You need the exact runtime annotation rather than the source-level one

The `-verbose` flag is the key: plain `javap` shows method signatures only. `-verbose` exposes the full constant pool including annotation type descriptors, making it possible to `grep` for specific annotation names.
