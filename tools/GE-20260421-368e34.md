## Technique

When a page has a multi-stage async init (renderer ready → data loaded), expose a stage-specific boolean flag in `window.__test` and use `waitForFunction` on that flag rather than a coarser "ready" signal.

```javascript
// In visualizer.js
let terrainLoaded = false;

window.__test = {
  threeReady:   () => !!renderer,    // fires before terrain exists
  terrainReady: () => terrainLoaded, // fires after loadTerrain() completes
};

async function loadTerrain() {
  // ... build 8192 tile meshes ...
  terrainLoaded = true; // ← set at the very end
}
```

```java
// In Playwright test
page.waitForFunction("() => window.__test?.terrainReady?.() === true",
    null, new Page.WaitForFunctionOptions().setTimeout(10000));
// Now safe to assert scene.children.length > 10
```

## Why this is non-obvious

`threeReady` fires as soon as `renderer` is assigned — which happens before `await loadTerrain()` completes. Tests that wait for `threeReady` then immediately assert tile count will see 0 tiles. The bug is intermittent under load (terrain may load fast enough to pass) and fails reliably only under CI slowness or on the first page load.

## Pattern

Expose named stage flags in `window.__test` for each meaningful init milestone. Use `waitForFunction` with the most specific flag for each test assertion — not a coarser "page ready" signal.
