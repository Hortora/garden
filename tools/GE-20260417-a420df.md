## PR base becomes stale when upstream merges your content as a different PR

**ID:** GE-20260417-a420df
**Stack:** Git, GitHub
**Symptom:** CI starts failing on a PR that was previously clean. `gh pr checks` shows old run IDs. New pushes don't trigger the expected CI behaviour. The PR base branch no longer reflects the actual upstream state.
**Context:** Open-source fork workflow. You open a PR (`your-fork:feature-branch → upstream:base-branch`). While your PR is in review, the upstream maintainer takes your content, applies their own fixes on a new branch, and merges it directly to `main` as a new PR. Your original PR still shows as open.

### What was tried (didn't work)

- Pushing more fixes to the feature branch — CI ran against the stale base
- Checking `gh pr checks` — showed old completed runs, not new ones
- Nothing signalled that the base had been superseded

### Root cause

When the upstream maintainer merges the PR's content from a different branch (e.g. `hibernate_reworked`) directly to `main`, the original PR's base branch (`upstream:feat/persistence/hibernate`) becomes an orphaned ref. It no longer reflects `upstream:main`. Any CI for your original PR tests against this stale base. The GitHub UI doesn't warn you — the PR still shows as open with its original base.

### Fix

1. Confirm the content is already in upstream main: `git log upstream/main --oneline`
2. Identify which of your commits are NOT yet upstream: `git log upstream/main..your-branch --oneline`
3. Create a new clean branch from upstream main and cherry-pick only your unique commits
4. Close the original PR
5. Open a new PR from the clean branch targeting the correct base (`main`)

### Why this is non-obvious

The original PR still appears open with no indication that its base is stale. CI continues to run (against the wrong base). The symptom — CI failures that don't match your local build — looks like an environment issue, not a branch topology problem.
