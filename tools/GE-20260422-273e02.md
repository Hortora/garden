## zsh variable-as-command shorthand fails when the command string contains arguments

**Stack:** zsh
**Symptom:** A variable assigned a command prefix string fails with "no such file or directory" when used as a command:
```bash
S="python3 /path/to/script.py /path/to/garden --record"
$S "GE-X × GE-Y" distinct "note"
# zsh: no such file or directory: python3 /path/to/script.py /path/to/garden --record
```

**Context:** Trying to avoid repetition when calling the same script many times with fixed prefix arguments, by assigning the prefix to a variable and using it as a command.

**Root cause:** In zsh, `$S` expands to the entire string as a single token when unquoted in certain contexts — the shell treats the whole value as the command name rather than splitting on spaces. The leading `python3` and the path and arguments all become part of the "command name" lookup.

**Fix:** Use an array or a function instead:
```bash
# Array (bash and zsh)
S=(python3 /path/to/script.py /path/to/garden --record)
"${S[@]}" "GE-X × GE-Y" distinct "note"

# Or repeat the full command in a loop
pairs=("pair1" "pair2")
for pair in "${pairs[@]}"; do
    python3 /path/to/script.py /path/to/garden --record "$pair" distinct "note"
done
```

**Why non-obvious:** The pattern `CMD="command arg1 arg2"; $CMD more-args` is a common beginner shorthand that happens to work in some shells and contexts, leading to the assumption it's universally valid. The error message names the entire variable value as the "command" which is cryptic if you don't already know about word-splitting rules.
