## Problem

Using `replace_all=True` on an Edit call where the search pattern matches both a variable definition line and call sites silently corrupts the definition. The tool reports success; the file appears to have changed correctly; but the definition is now self-referential and raises `UnboundLocalError` at runtime.

**Example:** replacing `Path(garden_root)` → `garden_path` across a function that also contained `garden_path = Path(garden_root) if garden_root else None` rewrites the definition to `garden_path = garden_path if garden_root else None`. No warning is emitted.

## Symptom

Runtime `UnboundLocalError: cannot access local variable 'garden_path' where it is not associated with a value` in tests that exercise the affected function — not an edit error, not a syntax error.

## Root Cause

`replace_all` is purely textual — it matches every occurrence of the pattern in the file, including the right-hand side of the very assignment that introduces the variable.

## Fix

After any `replace_all` that introduces a new name, verify the definition line explicitly. If corrupted, follow immediately with a targeted Edit on that specific line.

Alternatively: perform the definition-line edit separately (without `replace_all`) and use `replace_all` only for the call sites.

### Why this is non-obvious

The Edit tool reports success and the diff looks correct at a glance — the corrupted line (`garden_path = garden_path ...`) is syntactically valid Python, so no parse error surfaces. The failure only appears at runtime when the variable is accessed before assignment in a branch that skips the `if` clause.
