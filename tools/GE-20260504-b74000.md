## Symptom

A row containing PII is deleted from a Doltgres database. The delete succeeds — the row is gone from the current HEAD. But GDPR Art.17 erasure is not satisfied: the PII still exists in all prior commits in the git history.

## Context

Doltgres is a PostgreSQL-compatible git-for-SQL database. Like Git, every change is stored as a commit. Deleting a row is itself a commit — the row vanishes from HEAD but remains in every prior commit that referenced it. This is the intended behaviour for auditability; it is the wrong behaviour for GDPR erasure.

## What was tried

Application-layer tombstoning (replace PII with a hash/sentinel) satisfies the requirement in standard PostgreSQL because PostgreSQL has no historical row store. In Doltgres, the tombstone commit itself is new state in HEAD — but the original data row still appears in `dolt_history_<table>` and in any prior commit.

## Root cause

Doltgres's git semantics make complete historical erasure require explicit history rewriting via `CALL DOLT_FILTER_BRANCH(...)` — the SQL equivalent of `git filter-branch`. This is a destructive, non-trivial operation. Worse: if Merkle Mountain Range checkpoints were signed and published for the commits that contained the PII row, rewriting history invalidates those proofs, breaking the cryptographic audit trail.

## Fix

Two options, neither ideal:

**Option 1 — Use PostgreSQL backend for regulated data.** Keep Doltgres for operational/development data; use standard PostgreSQL for any data subject to GDPR erasure. Merkle proofs remain intact. Erasure is clean.

**Option 2 — Purge git history.** Run `CALL DOLT_FILTER_BRANCH('DELETE FROM table WHERE id = ?', ...)` to rewrite history across all commits. Acknowledge that any externally-published Merkle checkpoints covering those commits are now invalidated. Republish new checkpoints from the rewritten history.

## Why non-obvious

Row deletion is semantically "the data is gone" in every other database. In Doltgres, deletion is append-only at the commit level — the DELETE is a new commit, not a retraction of the old one. The distinction is invisible until a compliance audit surfaces it.

**How to apply:** Any system using Doltgres with personal data must evaluate GDPR Art.17 implications before deployment. The Doltgres backend is suitable for regulated deployments only where GDPR erasure is not required, or where erasure can tolerate Merkle proof invalidation.
