## Symptom

Specs from brainstorming sessions land in `docs/superpowers/specs/` in the project repo. CLAUDE.md routing table declares `specs → project, lands in docs/specs/` but the brainstorming skill ignores it.

## Cause

The superpowers brainstorming skill hardcodes the spec path in two places:

```
# skills/brainstorming/SKILL.md line 29:
save to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`

# skills/brainstorming/SKILL.md line 111:
Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
(User preferences for spec location override this default)
```

The `(User preferences for spec location override this default)` note implies override is possible, but no mechanism routes the CLAUDE.md routing table into the brainstorming skill's path selection. The skill reads its hardcoded path; the routing table is for a different workflow (work-end artifact promotion).

## Fix

Add an explicit override instruction to the workspace CLAUDE.md that the brainstorming skill will read at session start:

```markdown
**Spec path override:** The brainstorming skill defaults to `docs/superpowers/specs/` in the project repo. This project overrides that: specs promote to `docs/specs/` (not `docs/superpowers/specs/`). When writing a spec, use `docs/specs/YYYY-MM-DD-<topic>-design.md`.
```

Place this in the Artifact Locations section so it's visible at context load time. The brainstorming skill does respect explicit CLAUDE.md instructions when they name the override directly.
