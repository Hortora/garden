## Structure architecture docs as property claims with Guarantees and Graceful Degradation

**ID:** GE-20260414-3d73c3
**Stack:** Technical documentation (cross-cutting)
**Labels:** `#strategy` `#documentation` `#architecture`
**What it achieves:** Architecture documentation that maps directly to how platform architects and enterprise evaluators think — against a checklist of properties, not a tour of components.
**Context:** Writing technical documentation for platform architects, DevOps engineers, or enterprise decision-makers evaluating whether a system is suitable for deployment.

### The technique

Structure each section as:
1. **A property claim as the heading** — not a component name or topic label. "Concurrent sessions cannot corrupt garden state" not "Storage Architecture".
2. **Problem statement** — 1–2 sentences: what goes wrong without this property.
3. **How the system achieves it** — the mechanism, not a component tour.
4. **Guarantees callout** — explicit, concrete promises the system makes. "X is never Y", "Z always happens before W".
5. **Graceful degradation callout** — what works at reduced capacity vs. what requires intervention. Distinguishes between automatic recovery and manual action.

Example heading pairs:
- ❌ "Storage layer" → ✅ "Concurrent sessions cannot corrupt garden state"
- ❌ "Validation pipeline" → ✅ "No entry enters the garden without passing validation"
- ❌ "Deduplication" → ✅ "Duplicate knowledge is eliminated in layers, not all at once"

### Why this is non-obvious
Most developers default to component tours (here's service A, here's service B), tutorial docs (here's how to use it), or architecture diagrams with boxes and arrows. None of these map to how architects evaluate systems.

Architects work from a mental checklist of properties: "Is state consistent under concurrency? What are the failure modes? What degrades gracefully vs. fails hard?" The property-claims structure meets them at their evaluation model rather than forcing them to translate from component documentation.

The "Graceful degradation" section is the most non-obvious addition — it explicitly separates what recovers automatically from what requires operator intervention. This is the question architects ask but documentation almost never answers directly.

### When to use it
When the audience is evaluating whether to deploy or integrate a system — platform architects, enterprise decision-makers, security reviewers. Not for user tutorials or API reference docs.
