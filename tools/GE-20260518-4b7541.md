## Apply journal merges to main before merging two epics that both modify DESIGN.md

### Technique

When closing multiple epics that both modified the same document (e.g. DESIGN.md), apply all journal narrative updates to main first — before merging either epic's code — then merge code epics individually.

```bash
# 1. Apply journal narratives directly on main
git checkout main
# Edit DESIGN.md to incorporate §Section narratives from each journal
git commit -m "docs: apply design journals for epics #N and #M"

# 2. Merge code epics individually (DESIGN.md conflict is now resolved by the pre-applied narrative)
git merge --no-ff epic-foo
git merge --no-ff epic-bar
```

If two epic branches both add lines to the same DESIGN.md section, git will produce a three-way conflict on merge. By committing the combined narrative on main first, each subsequent epic merge sees the pre-merged DESIGN.md as the common ancestor and can auto-resolve.

### Why this is non-obvious

The natural instinct is to merge code first and handle doc conflicts during the merge. But DESIGN.md conflicts from two epics are structural (both adding rows to the same table, both extending the same section) rather than semantic — pre-applying them avoids conflicts entirely at the cost of one extra commit.
