## What happened

`gh run view <run-id> --log-failed` output was capped at ~10,000 lines. A Quarkus test
with Hibernate SQL trace logging (`TRACE` level) emitted hundreds of SQL parameter binding
lines per test, consuming the entire log budget before the Maven Reactor Summary (BUILD FAILURE
section) could appear. The actual error was invisible.

## Why this is non-obvious

`--log-failed` sounds like it should show only the failure — it actually shows the full log
output of failed **jobs** (which includes all passing test output plus the failure). When any
component emits verbose trace logging, the failure details scroll off the top of the 10k cap.
There is no truncation warning.

## Root cause

`gh` CLI truncates job log output at ~10,000 lines when streaming to a terminal or piping.
Verbose framework logging (SQL trace, request/response dumps) fills the budget before the
build failure section appears.

## Fix

Download the Maven Surefire XML reports artifact — they contain structured test results
without the noise:

```bash
gh run download <run-id> --repo <owner/repo> --name surefire-reports --dir /tmp/surefire
# Find failures:
find /tmp/surefire -name "TEST-*.xml" | xargs python3 -c "
import sys, xml.etree.ElementTree as ET
for f in sys.argv[1:]:
    try:
        root = ET.parse(f).getroot()
        e = int(root.get('errors', 0))
        fl = int(root.get('failures', 0))
        if e > 0 or fl > 0:
            print(f'FAIL e={e} f={fl}: {f}')
    except: pass
"
# For JVM crash / dump files (CDI startup failures):
find /tmp/surefire -name "*.dumpstream" | head -5
```

The XML reports cap at one file per test class and contain the full exception message
without any surrounding log noise.

## Constraints

Applies to CI systems where log collection has a line budget. Surefire must be configured
to write XML reports (default in Maven, but can be disabled with `surefire.useFile=false`).
Dump files (`.dump`, `.dumpstream`) appear when the JVM forks fail before any tests run
(e.g. CDI deployment failure at startup).

## Alternatives considered

- Reduce logging verbosity in CI: correct long-term fix, but requires changing `application.properties`
  to not use TRACE in CI; adds configuration surface.
- Use `gh run view --log-failed 2>/dev/null | tail -200`: helps if failure is at the end,
  but not when the buffer exhaustion pushes the error to a different position.
