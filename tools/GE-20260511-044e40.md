### Symptom

`reword` commands in a git rebase todo file are silently treated as `pick` — the commit is applied unchanged with its original message. No error, no warning. The reword appears to have been processed (rebase completes successfully) but the commit message is unchanged.

### Context

Occurs in non-interactive rebase sessions where `GIT_EDITOR` is not set or resolves to a no-op. Common in CI pipelines, headless Claude sessions, and automated scripts that use `GIT_SEQUENCE_EDITOR` to write the todo file but don't configure an editor for the subsequent reword prompt.

### What was tried

Using `reword` in the todo file via `GIT_SEQUENCE_EDITOR` python script. Rebase completed without error. Commit messages remained unchanged — only detected by running the verification grep afterwards.

### Root cause

`reword` requires an editor to open so the user can type the new message. With no functional `GIT_EDITOR`, git falls back silently to the commit's existing message, effectively treating `reword` as `pick`. Git emits no warning or error code.

### Fix

Use `exec git commit --amend -F /tmp/msg.txt` immediately after the `pick` line instead of `reword`. Store the intended message in a temp file before the rebase:

```python
# In GIT_SEQUENCE_EDITOR script — inject exec after target commit
for l in lines:
    out.append(l)
    if sha.startswith("abc1234"):
        out.append('exec git commit --amend -F /tmp/intended_msg.txt\n')
```

```bash
# Write the message file before starting the rebase
cat > /tmp/intended_msg.txt << 'EOF'
fix: correct subject line here

Body text here.
EOF
GIT_SEQUENCE_EDITOR="python3 /tmp/editor.py" git rebase -i <base>
```

### Variant — GIT_EDITOR=true causes a stall instead of a silent skip

If `GIT_EDITOR=true` is explicitly set (a common pattern to auto-accept prompts during scripted rebases), `reword` behaves differently: the rebase **halts mid-run** in a detached-HEAD state rather than silently skipping. After each `reword` step, git opens `true` as the editor, which exits immediately without modifying the file. Git interprets this as "user opened editor but made no changes" and re-prompts — leaving the rebase stuck with `git status` showing "interactive rebase in progress". The rebase must be aborted with `git rebase --abort`. No error is emitted; the stall is the only signal.

The fix is identical: replace `reword` with `exec bash /tmp/amend_N.sh` where the script calls `git commit --amend -F /tmp/msg_N.txt`.

### Why this is non-obvious

Git's `reword` documentation only mentions the interactive flow where an editor opens. There is no documented behaviour for what happens when no editor is available — the silent fallback to `pick` is undocumented and leaves no trace in the output.

### Why this fix

`exec git commit --amend -F file` is self-contained: the message is written before the rebase starts, no editor is needed, and the `-F` flag bypasses all editor lookup. The `-m "..."` alternative works for ASCII but fails silently or mangles Unicode characters (→, –, etc.) depending on shell encoding.
