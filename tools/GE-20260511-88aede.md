### Context

Maven compilation fails with errors pointing at main source files that look completely
correct — e.g. "interface expected here" in a file that clearly declares an interface,
or "cannot find symbol" for a constant that visibly exists in source. Multiple Maven
resume attempts (`-rf :module`) reproduce the same errors.

### Technique

Run `mcp__intellij__build_project` on the failing IntelliJ project. It compiles both
main and test sources using IntelliJ's own compiler and reports the actual error
locations — including test-source files that Maven's output obscures.

```
# Maven reports (misleading):
[ERROR] ServerlessWorkflowExecutor.java:[31] interface expected here
[ERROR] CaseContextChangedEventHandler.java:[339] cannot find symbol: SUBCASE_SCHEDULE

# IntelliJ build_project reports (accurate):
{"file": "engine/src/test/java/.../NoOpLedgerEntryRepository.java",
 "line": 38, "message": "Class must implement findBySubjectIdAndTimeRange(...)"}
```

The real error was in a test-only `NoOpLedgerEntryRepository` — 4 of them across
different submodules — none of which appeared in Maven's output at all.

### Why this is non-obvious

Maven's error output points at the downstream symptom (wrong compilation result from
stale jar + correct source = impossible error), not the cause. IntelliJ's build
compiles the full project including test sources from actual source, bypassing the
`.m2` stale jar problem entirely. Most developers reach for more Maven flags or re-read
the source; using the IDE's build as a diagnostic tool is not the obvious move.

### When to use

When Maven compilation errors seem implausible given the source — run
`build_project` on the IntelliJ project before spending time on Maven flags.
