## What happened

OpenClaw plugins have two lifecycle phases: `register(api)` (synchronous, called at plugin load)
and `start()` (async, called after the gateway is ready). If hook registration is moved into
`start()` because the handler needs async-initialised resources (e.g. a telemetry runtime):

```typescript
async start(): Promise<void> {
  const telemetry = await initTelemetry();
  this.api.on("before_prompt_build", (ctx) => /* uses telemetry */);
}
```

The `api.on()` call runs without error, but the handler is **never fired**. The gateway snapshots
all typed hook registrations approximately 30 seconds after `register()` returns — before `start()`
is ever called.

## Why it's non-obvious

There is no error from `api.on()`. The plugin loads and reports healthy. The `start()` method
completes normally. The hook registration log line prints. But the snapshot has already been taken.

The symptom matches a dozen other possible causes (wrong hook name, policy gate missing, handler
throwing silently) which leads developers to debug the handler logic rather than the registration
timing.

## The fix

Register hooks synchronously in `register()`. Use a lazy getter for resources that aren't
available at registration time:

```typescript
class MyPlugin {
  private telemetry?: TelemetryRuntime;

  register(api: OpenClawPluginApi): void {
    // Hooks registered synchronously — snapshot taken immediately after register() returns
    api.on("before_prompt_build", async (ctx) => {
      // Lazy: telemetry resolved later in start(), available by the time any agent turn fires
      return this.telemetry?.record(ctx) ?? {};
    });
  }

  async start(): Promise<void> {
    this.telemetry = await initTelemetry();
  }
}
```

The handler closure captures `this`, so async resources initialised in `start()` are accessible
by the time any agent turn fires (start() completes before agent traffic begins).
