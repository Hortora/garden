## Symptom

```
unknown flag: --yes
```

Running `gh repo transfer owner/repo new-org --yes` fails. Unlike `gh repo delete --yes` or `gh repo archive --yes`, the transfer subcommand does not accept a `--yes` flag for non-interactive confirmation.

## Context

Occurs when scripting a repository transfer without user input — e.g. in a handover workflow moving repos to an organisation.

## Root cause

`gh repo transfer` is not implemented with the `--yes` bypass that other destructive `gh repo` subcommands support. The CLI prompts for confirmation interactively.

## Fix

Use the GitHub REST API directly via `gh api`:

```bash
gh api \
  --method POST \
  -H "Accept: application/vnd.github+json" \
  /repos/{owner}/{repo}/transfer \
  -f new_owner={org}
```

No confirmation prompt. Returns the updated repo object including `full_name` to confirm the transfer.

## Why this is non-obvious

`gh repo delete --yes` and other transfer-adjacent commands accept `--yes`. It's natural to assume the pattern extends to `gh repo transfer`.
