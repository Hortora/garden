### Symptom

After a successful `git rebase` squash, `git push --force-with-lease origin main`
fails with the pre-push hook exiting 1. The hook fires, sees commits being pushed,
runs its squash-candidate check, and blocks the push — even though the squash just
completed.

### Context

Occurs when a pre-push hook enforces "run /git-squash before pushing" (e.g. the
`chore: add git hooks` commit in casehub-devtown). The hook is correct for normal
development pushes but is triggered by the squash delivery push itself, creating a
circular dependency: you cannot deliver the squash without bypassing the guard that
exists to ensure squashing was done.

The same applies to any delivery push at the end of a squash workflow —
`work-end`, `git-squash`, or any manual rebase that produces a force-push.

### Fix

Bypass the pre-push hook for squash delivery pushes:

```bash
git push --force-with-lease --no-verify origin main
```

`--no-verify` skips all hooks for this push only. It is safe here because
the squash has already been reviewed and approved — the hook's purpose is
already served.

### Why this is non-obvious

The hook is installed as part of the same squash infrastructure it is blocking.
There is no error message indicating a circular dependency — the hook exit 1
looks identical to a legitimate "squash needed" failure. The fix (`--no-verify`)
bypasses a hook that exists for good reason, which feels wrong; the rationale
(the hook's purpose is already satisfied at this point) is not obvious.
