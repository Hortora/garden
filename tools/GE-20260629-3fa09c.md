### Symptom

1. `git push` → rejected (non-fast-forward, remote is ahead)
2. `git pull --rebase` → "Already up to date"
3. `git push` → "Everything up-to-date"
4. Your commit is gone from the branch — `git log` no longer shows it

No error, no warning, no conflict marker. The commit silently disappears.

### Context

Occurs when the remote main advanced with changes that touch the same files as your local commit. If another session or CI pushed changes that overlap with yours (even if not identical line-for-line), the rebase considers your patch "already applied" and skips it. This is particularly dangerous in multi-session workflows where multiple Claude instances or developers push to the same branch.

### Root cause

`git rebase` uses patch-id matching to detect already-applied commits. When the remote contains changes that produce the same patch effect as your commit (even via a different commit), git considers your commit redundant and drops it. The "Already up to date" message is misleading — it means "no new remote commits to fetch", not "your commits are safe".

### Fix

**Prevention:** Always verify your commit survives after `pull --rebase`:

```bash
# Before pull --rebase, note your commit hash
git log --oneline -1
# e.g. abc1234 chore: slim dependencyManagement

git pull --rebase origin main

# After rebase, verify your commit is still present
git log --oneline | grep "slim dependencyManagement"
# If missing: your commit was dropped
```

**Recovery:** Cherry-pick from the reflog:

```bash
# Find the dropped commit
git reflog | grep "slim dependencyManagement"
# e.g. abc1234 HEAD@{3}: commit: chore: slim dependencyManagement

git cherry-pick abc1234
```

**Alternative:** Use `merge` instead of `rebase` when working on shared branches with overlapping changes — merge preserves both sides explicitly.

### Why this is non-obvious

Git prints no warning when dropping a commit during rebase. The "Already up to date" + "Everything up-to-date" sequence reads as success. A developer who doesn't manually verify `git log` after every `pull --rebase` will not notice the data loss until much later — potentially after the reflog entry has expired.

### Why this fix

Verification after rebase is the only reliable defence. `--reapply-cherry-picks` flag (Git 2.36+) partially addresses this but only for cherry-picks, not for content-equivalent commits from independent authors.
