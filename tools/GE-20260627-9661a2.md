### Symptom
`gh api --method PATCH 'orgs/<org>/packages/maven/<package>' -f visibility=public` returns 404, even with `write:packages` scope.

### Context
GitHub Packages supports changing visibility for container-type packages via the REST API. The same endpoint returns 404 for Maven packages — the API simply doesn't exist for this registry type.

### Root cause
Maven packages on GitHub inherit visibility from the repository. There is no independent visibility control and no REST API endpoint to change it. This is a deliberate limitation, not a bug — tracked as `github/roadmap#578` (Maven granular permissions).

Container packages (`ghcr.io`) support independent visibility via `PATCH /orgs/{org}/packages/{type}/{name}` with `visibility` field. Maven packages do not.

### How to verify
```bash
# Works for containers
gh api --method PATCH 'orgs/<org>/packages/container/<name>' -f visibility=public

# Returns 404 for Maven — even with write:packages
gh api --method PATCH 'orgs/<org>/packages/maven/<name>' -f visibility=public
```

### Workaround
Change the repository visibility instead: `gh repo edit <org>/<repo> --visibility public --accept-visibility-change-consequences`. All Maven packages published from that repo inherit the new visibility.

### Why this is non-obvious
The GitHub Packages docs describe visibility controls generically without calling out that Maven is excluded. The REST API reference lists the PATCH endpoint for packages but doesn't document which registry types support the `visibility` field. You discover the limitation only when the API returns 404.
