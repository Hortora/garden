## git mv fails with 'No such file or directory' when target directory doesn't exist — partially stages the deletion, data loss on follow-up git rm

### Symptom

`git mv source/file.md target/file.md` produces `fatal: renaming 'source/file.md' failed: No such file or directory` — even though `source/file.md` exists. Running `git rm source/file.md` afterward to "clean up" permanently deletes the file from the git index with no recovery path on disk.

### What was tried (didn't work)

- Shell `mv source/file.md target/file.md` — also fails because the target directory doesn't exist
- `git rm source/file.md` after the failed `git mv` — appears to clean up, but permanently removes the file from the index

### Root cause

`git mv` internally does three things: OS `mv`, `git rm <source>`, `git add <target>`. When the target directory doesn't exist, the OS `mv` fails — but git mv's internal state may have already partially staged the operation. A subsequent `git rm` on the source path then deletes it from the git index entirely, leaving no file on disk and no staged version.

### Fix

Always `mkdir -p <target_dir>` before `git mv`:

```bash
mkdir -p target/
git mv source/GE-XXXX.md target/GE-XXXX.md
```

If data loss has already occurred, recover from git history:

```bash
git show <last-good-commit>:source/GE-XXXX.md > target/GE-XXXX.md
git add target/GE-XXXX.md
```

### Why this is non-obvious

`git mv` looks like a high-level command that should handle directory creation. The OS-level failure message (`No such file or directory`) sounds like the source is missing, not the target directory. The follow-up `git rm` feels like cleanup but is actually destructive in this partial-staging state.
