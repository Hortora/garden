## Substring occurrence count in tests breaks when a new line contains the same substring

**Stack:** Python, pytest, unittest
**Symptom:** An existing idempotency test fails after a seemingly unrelated change — it asserted a value appeared exactly once, but now counts 2.

**Context:** Testing that a bash installer writes `garden-agent.log` to a `.gitignore` file exactly once (idempotency check). The test used:
```python
self.assertEqual(content.count('garden-agent.log'), 1)
```

When a second line `garden-agent.log.*` was added to the gitignore block, the count became 2 — because `garden-agent.log` is a substring of `garden-agent.log.*`.

**Root cause:** `str.count(substring)` counts overlapping substrings, not whole-line occurrences. Adding a line that *contains* the substring inflates the count even though the idempotency guarantee is still intact.

**Fix:** Count exact line occurrences instead:
```python
# Before (fragile — counts substrings)
self.assertEqual(content.count('garden-agent.log'), 1)

# After (counts exact lines)
self.assertEqual(content.count('garden-agent.log\n'), 1)
self.assertEqual(content.count('garden-agent.log.*\n'), 1)
```

**Why non-obvious:** The test passes before the change and the failure message says "2 != 1" with no indication of *why* — it looks like the idempotency guard broke rather than the test assertion being too broad.
