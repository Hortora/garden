### Symptom

Running `playwright test ... | head -60` appears to hang after printing 60 lines. No more output, no exit. The process sits alive indefinitely even though `head` has exited.

### Context

Trying to limit test output to the first N lines for quick diagnosis of test failures. Piping through `head` is the natural first choice.

### Root cause

`head -N` exits immediately after reading N lines. When it exits, the write end of the pipe becomes broken. Playwright's Node.js process receives `SIGPIPE` on the next write. Node.js may ignore `SIGPIPE` by default (it does on some platforms) and continue running — the test runner keeps executing, but all stdout writes silently fail. The process does not self-terminate.

### Fix

Use `tail -N` instead — `tail` reads from the end and does not close the pipe early:
```bash
playwright test ... 2>&1 | tail -60
```

Or write output to a file and inspect it:
```bash
playwright test ... 2>&1 > /tmp/pw-out.txt; tail -60 /tmp/pw-out.txt
```

Or cap at the OS level with `script` / `unbuffer` if you need the first N lines:
```bash
unbuffer playwright test ... | head -60
```

### Why this is non-obvious

`head` is the idiomatic "first N lines" tool and works correctly with most processes. The issue is specific to processes that (a) ignore SIGPIPE and (b) do not check write return values, leaving them alive but effectively silent. Playwright's Node.js runtime falls into this category.

### Constraints

Manifests on macOS. Linux behaviour may differ (Node.js SIGPIPE handling can differ by version and OS). The hang only occurs when piping to `head`; `tail` is safe because it doesn't close the pipe early.

### Invalidation triggers

Revisit if Node.js changes its default SIGPIPE handling (check `process.on('SIGPIPE')` defaults in Node.js release notes).
