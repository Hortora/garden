## What Developers Normally Do

Write sed scripts or use IDE bulk find-replace to rename project-wide symbols.
For renames with exclusion rules (e.g. "rename our package prefix but not
third-party deps that share the same prefix"), sed scripts require complex
negative lookaheads and still miss contextual distinctions. IDE replace-all
has no concept of "ignore this pattern when it appears in a dependency
declaration."

## The Technique

Split files by domain concern and dispatch one parallel agent per domain,
each given an explicit substitution table and explicit "DO NOT change" rules:

```
Agent 1: Java source files
  — substitution table: quarkus-ledger → casehub-ledger, etc.
  — exclusion rule: do NOT change io.quarkiverse.X where X is not our extension

Agent 2: Core documentation (DESIGN.md, integration-guide.md, etc.)
  — same substitution table

Agent 3: Historical plan/spec docs, root files, HANDOFF
  — same substitution table

Agent 4: Blog entries (frontmatter), examples, settings files
  — same substitution table
```

Use `isolation: "worktree"` on each agent. Changes auto-commit and merge back
into main. Run a final grep verification pass to catch any stragglers.

```
Agent({ isolation: "worktree", prompt: """
  Fix these exact substitutions in Java files only:
    quarkus-ledger → casehub-ledger
    io.quarkiverse.ledger → io.casehub.ledger (ONLY when it's our package, NOT third-party)
  Exclusion: do NOT change io.quarkiverse.opentelemetry or any other third-party quarkiverse lib.
  Final verification: grep to confirm zero remaining matches.
"""})
```

## Why This Is Non-Obvious

The instinct is to write sed scripts or use IDE bulk replace. Three things
make the agent approach superior for renames with exclusion rules:

1. **Contextual exclusion** — agents read the surrounding code and can
   distinguish `io.quarkiverse.ledger` (our old package) from
   `io.quarkiverse.opentelemetry` (a third-party dep). sed requires brittle
   negative lookaheads that break on unforeseen patterns.

2. **Domain splitting** — splitting by file type (Java / docs / plans / blog)
   rather than by feature lets each agent operate on a homogeneous set with
   a consistent grammar, reducing false positives.

3. **Implicit validation** — asking each agent to run a final grep
   verification at the end catches its own misses without a separate pass.

The parallel dispatch (single message, four Agent tool calls) keeps the total
elapsed time similar to a single sed pass, while producing more reliable
results on a 60+ file rename with non-trivial exclusion rules.
