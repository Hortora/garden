## Clean branch + cherry-pick when old branch has accumulated merged content

**ID:** GE-20260417-fdb17f
**Stack:** Git
**Context:** Feature branch that started from an older version of a base branch now contains N commits that have since been merged upstream via different PRs. `git rebase upstream/main` fails with 20+ conflicts because git tries to re-apply already-merged content.

### The technique

Instead of rebasing the messy branch:

```bash
# Step 1: Find which commits are ONLY on your branch (not yet upstream)
git log upstream/main..your-messy-branch --oneline
# → shows only the commits you actually added; merged content is absent

# Step 2: Create a clean branch from the correct base
git checkout -b feature-branch-clean upstream/main

# Step 3: Cherry-pick only YOUR new commits (in order, oldest first)
git cherry-pick <oldest-unique-commit> <next> <next> ...

# Step 4: Push and open new PR; close old one
git push -u origin feature-branch-clean
```

The `git log upstream/main..your-branch --oneline` output shows only commits not yet in upstream — these are the ones to cherry-pick. Commits that were merged (even via different PRs) are absent from this list.

### Why this is non-obvious

The instinct is to `git rebase upstream/main` and resolve conflicts. When your branch has accumulated multiple already-merged PRs (because the base branch evolved), rebasing re-applies each merged commit and creates conflicts with itself. The cherry-pick approach bypasses those conflicts entirely by only replaying genuinely new work.

### Why this fix

`git rebase` is correct when your commits are all new. Cherry-pick is correct when you need to surgically transplant only the unmerged commits to a clean base, skipping commits already present in the target.
