### Symptom
After extracting what appeared to be an identical private method from two large files into a shared utility class, one file's behaviour changed — a test that previously passed now fails with no obvious explanation.

### Context
Two large files (e.g. a Maven plugin's `InlineGenerator.java` at 3000 lines and an APT processor's `PermuteProcessor.java` at 2500 lines) contain methods with the same name and similar signatures. Visual inspection of the method bodies looks identical. After extracting to a shared utility class based on one file's implementation, tests in the other path break.

### Root cause
Over time, one copy of the "duplicate" method received a bug fix or feature addition that the other did not. The implementations diverged silently — the files were large enough that cross-referencing them during normal development was impractical. The copy used as the source for extraction was the incomplete one.

In this specific case, `expandMethodTypesForJ` in InlineGenerator was missing a type-parameter-addition block that existed in PermuteProcessor's version. The APT-mode `@PermuteMethod` type expansion test failed immediately.

### Fix
Before extracting duplicate methods, read both implementations side by side and diff them explicitly:

```bash
# Extract the method body from each file for comparison
grep -A 30 "private static void expandMethodTypesForJ" file1.java > /tmp/m1.txt
grep -A 30 "private static void expandMethodTypesForJ" file2.java > /tmp/m2.txt
diff /tmp/m1.txt /tmp/m2.txt
```

For each divergence: determine which version is correct (or whether both behaviours are needed), then write the canonical version into the shared utility.

### Why this is non-obvious
The files are too large to hold in working memory simultaneously. "Duplicate" means "same name and same intent" — not necessarily "same implementation". Silent divergence happens gradually: one copy gets a fix, the other doesn't, and nobody notices because both still compile. The symptom (post-extraction test failure) is indistinguishable from "wrong extraction" vs "one copy was already broken".
