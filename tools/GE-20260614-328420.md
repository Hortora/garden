## OpenClaw /v1/chat/completions requires model='openclaw', not the upstream provider model ID

**ID:** GE-20260614-328420
**Stack:** OpenClaw v2026.6.x

### Symptom

Calling OpenClaw's OpenAI-compatible chat completions endpoint with a provider model
name returns HTTP 400:

```json
{"error":{"message":"Invalid `model`. Use `openclaw` or `openclaw/<agentId>`.",
 "type":"invalid_request_error"}}
```

### Context

When `gateway.http.endpoints.chatCompletions.enabled: true` is set in `openclaw.json`,
OpenClaw exposes `POST /v1/chat/completions`. The natural assumption is to use the
configured upstream model ID (e.g. `"mock-llm/mock-model"`, `"anthropic/claude-opus-4-6"`).

### Root cause / Undocumented behaviour

The `model` field in the request does not name the upstream LLM. It names the OpenClaw
agent. OpenClaw IS the model from the caller's perspective — it receives the request,
routes it through its agent pipeline, and calls the configured upstream provider
internally. The upstream provider model is determined by the agent's config, not the
request.

Valid model values:
- `"openclaw"` — uses the default agent
- `"openclaw/<agentId>"` — routes to a specific named agent

```bash
curl -X POST http://localhost:18789/v1/chat/completions \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"model":"openclaw","messages":[{"role":"user","content":"ping"}]}'
```

OpenClaw then calls its configured upstream provider (e.g. `mock-llm/mock-model` via
WireMock, or `anthropic/claude-opus-4-6`) transparently.

### Why this is non-obvious

The OpenClaw docs describe the chatCompletions endpoint as OpenAI-compatible, which
implies `model` should identify the LLM. The error message does clarify the correct
values, but the architectural reason (OpenClaw = the model, upstream = internal) is
not explained anywhere in the docs. The distinction only becomes clear after reading
the error response.
