## Symptom

After running `workspace-init`, every new Claude Code session in the project
directory shows:

> ℹ️ No workspace configured for this project.
> Run /workspace-init to create ~/claude/private/<project>/...

Even though the workspace is fully set up.

## Context

Occurs when `workspace-init` is run with the "keep CLAUDE.md in project repo"
option (Option B). The workspace is real and complete — `wksp` symlink exists,
all artifact directories present.

## Root Cause

`check_project_setup.sh` detects a configured workspace by looking for
`## Session Start` in the **project's** `CLAUDE.md`. But that section lives
in the **workspace** `CLAUDE.md` (`~/claude/public/<project>/CLAUDE.md`),
not the project one. When Claude opens the project directory, it loads the
project CLAUDE.md — so the section is never found and the hook always fires.

## Fix

Check for the `wksp` symlink instead — `workspace-init` always creates it
(Step 7b) regardless of which CLAUDE.md option was chosen:

```bash
if [ -L "wksp" ] && [ -d "wksp" ]; then
  : # workspace configured — wksp symlink present and valid
elif grep -q "## Session Start" CLAUDE.md 2>/dev/null; then
  : # legacy detection fallback
elif grep -q "## Project Type" CLAUDE.md 2>/dev/null; then
  echo "ℹ️  No workspace configured for this project."
  ...
fi
```

`[ -L "wksp" ] && [ -d "wksp" ]` — symlink exists AND resolves to a valid
directory. Works for both `public` and `private` workspace locations.

## Why This Is Non-Obvious

The hook and the workspace CLAUDE.md both mention `## Session Start` — the
section name is consistent across the skill's documentation. The problem is
that there are *two* CLAUDE.md files involved (project + workspace) and the
hook reads the wrong one. The symptom gives no hint about which file is being
checked.
