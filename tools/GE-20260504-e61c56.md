## Symptom

Running `gh repo fork org/repo` creates `myuser/repo-1` instead of `myuser/repo`
with no error or warning. The fork appears to succeed, but is named differently
than expected.

## Context

Occurs when `myuser/repo` already exists as a separate repository (not a fork
of the same upstream). GitHub appends `-1` (then `-2`, etc.) to avoid a naming
conflict.

## Root cause

GitHub's fork API automatically deduplicates fork names by appending a numeric
suffix when the target name is taken. `gh` passes through this behaviour without
surfacing a warning that the name was changed.

## Fix

Check the returned URL after forking:
```bash
gh repo fork casehubio/claudony --clone=false
# → https://github.com/mdproctor/claudony-1  ← check this
```

If the name is wrong, either rename the conflicting repo first, or configure the
fork's remote manually to point to the `-1` variant:
```bash
git remote set-url origin https://github.com/mdproctor/claudony-1.git
```

Delete unnecessary forks with `delete_repo` scope:
```bash
gh auth refresh -h github.com -s delete_repo
gh repo delete mdproctor/repo-1 --yes
```

## Why this is non-obvious

The command prints only the fork URL, which a human may not read carefully. Scripts
that capture the fork name from `gh repo fork` output and use it verbatim will
silently operate on the wrong repo.
