## Symptom

Running `git rebase -i` with a todo file containing an `exec` command whose argument spans multiple lines fails with misleading errors like:

```
error: invalid command 'Closes'
error: invalid line 12: Closes #14, Closes #15
error: invalid command 'Co-Authored-By:'
```

The word "Closes" (from the commit message body) is parsed as a rebase command.

## Context

When building a non-interactive rebase todo file (e.g. via `GIT_SEQUENCE_EDITOR="cp /tmp/plan"`) and including an `exec` line with a multi-line commit message argument:

```
exec git commit --amend -m "feat: some feature

Closes #14

Co-Authored-By: Claude <noreply@anthropic.com>"
```

Every line after the first is treated as a separate rebase todo command, not as continuation of the exec argument.

## What was tried

Writing the `-m` argument with newlines directly in the todo file. The closing `"` on a later line doesn't help — the parser never sees it as part of the exec line.

## Root cause

The git rebase todo parser is strictly line-by-line. `exec` captures exactly one line — everything after the `exec ` keyword to end-of-line. There is no multi-line exec syntax.

## Fix

Write commit messages to temp files and reference them with `-F`:

```bash
cat > /tmp/msg1.txt << 'EOF'
feat: some feature

Closes #14

Co-Authored-By: Claude <noreply@anthropic.com>
EOF
```

Then in the rebase todo:
```
exec git commit --amend -F /tmp/msg1.txt
```

This keeps the exec line single-line regardless of message length.

## Why this is non-obvious

The error message `error: invalid command 'Closes'` gives no hint that the cause is a multi-line exec argument. A developer would investigate the rebase command itself, their GIT_SEQUENCE_EDITOR setup, or encoding — not the structure of the todo file line.

The git documentation for `exec` says "runs the given command" but does not explicitly state that it cannot span lines.
