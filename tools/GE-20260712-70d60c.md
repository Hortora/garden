### Symptom
Calling `ide_edit_member` with `member` set to the class name (e.g. `member: "SituationEvaluator"`)
to replace a constructor replaces the **entire class declaration** — all fields, methods, inner
classes, and the class body are destroyed. Only the replacement content (the constructor) survives.

### Context
Every Java class has a constructor with the same name as the class. When `ide_edit_member`
resolves `member`, it finds two matches: the class declaration and the constructor. It
resolves to the class declaration, which is the outer scope — replacing it removes everything.

### Root cause
`ide_edit_member` matches `member` against all named elements in the file. The class
declaration (`class SituationEvaluator`) is a valid match for `member: "SituationEvaluator"`,
and it takes precedence over the constructor with the same name.

### Fix
Use `ide_replace_text_in_file` for constructor changes — match the exact constructor
signature text and replace it. Alternatively, use `anchorParameterCount` to disambiguate:
```
ide_edit_member(file, class="SituationEvaluator", member="SituationEvaluator",
    parameterCount=7, content="...")
```
This forces the tool to match the constructor overload with the specified parameter count
rather than the class declaration.

### Why this is non-obvious
The `member` parameter naturally reads as "the member named X inside the class" — a
constructor is the most common member to share the class name. The destructive outcome
(entire class replaced) is disproportionate and irreversible without git reset.
