## Symptom

After renaming Maven module directories (e.g. `casehub-work-api/` → `api/`) and updating the parent `pom.xml` `<module>` entries, calling `build_project` via the IntelliJ MCP returns errors like:

```
java: package io.casehub.work.api does not exist
cannot find symbol class WorkItem
```

The errors affect modules that depend on the renamed module. The build appears broken.

## Context

IntelliJ tracks module source roots internally. After directory renames — even when done via `git mv` and reflected in `pom.xml` — IntelliJ's module structure still references the old paths until Maven is explicitly reimported. `build_project` uses IntelliJ's incremental compiler, which sees stale module wiring.

## What was tried

Running `build_project` multiple times. Syncing files via `ide_sync_files`. Neither clears the stale module structure.

## Root cause

IntelliJ's project model caches module-to-directory mappings. `git mv` changes the filesystem; `ide_sync_files` refreshes the VFS. But neither triggers a Maven reimport, which is what rebuilds the module wiring from `pom.xml`.

## Fix

After any Maven module directory rename:

1. Trigger Maven reimport in IntelliJ UI (right-click `pom.xml` → Maven → Reload Project, or use the Maven tool window)
2. Only then call `build_project` for verification

Use `mvn --batch-mode validate` via `execute_terminal_command` in the IntelliJ terminal as an alternative verification that bypasses IntelliJ's stale cache entirely.

## Why this is non-obvious

`ide_sync_files` exists precisely to refresh IntelliJ after external filesystem changes — it's the obvious tool to reach for. But VFS sync and Maven project reimport are separate operations. Sync refreshes file visibility; reimport rebuilds module wiring. The distinction isn't surfaced anywhere in the MCP documentation.
