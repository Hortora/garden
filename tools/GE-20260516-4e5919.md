## Context

You need to map integer constants from a binary protocol (ability IDs, opcode values, field tags) to your domain model. The constants aren't documented anywhere — only discoverable by reading source code or observing real data. You write a static lookup table and move on.

## The technique

Instead of a one-off script, write the discovery as a **permanent JUnit test** that:

1. Parses real data files (replays, captures, logs)
2. Collects all observed `(constant, variant, count)` tuples
3. Prints them sorted by frequency to stdout
4. Has **no assertions** — it always passes

```java
@ParameterizedTest(name = "{0}")
@MethodSource("dataFiles")
void discoverAbilityIds(Path dataFile) {
    Map<String, Long> counts = new TreeMap<>();
    // parse dataFile, collect (abilLink, abilCmdIndex, context) tuples
    counts.forEach((k, v) -> System.out.printf("  count=%5d  %s%n", v, k));
}
```

The test stays in the normal test suite permanently.

## Why this beats a one-off script

**Initial mapping:** Run the test once, paste its output into a comment in the mapping class. The sorted-by-frequency output naturally surfaces the important constants first.

**Regression detection:** When new data files are added later (new replays, updated protocol versions), re-running the test shows if new constants appeared that aren't in your mapping. A sudden `count=1  abilLink=999` entry tells you your mapping is incomplete.

**Living documentation:** The test output is reproducible. Any developer can re-run it to verify what constants actually appear in the data, without needing the original discovery context.

**Coverage proof:** If the test shows `count=0` for all unmapped constants, the mapping is complete for the data corpus. If not, the gap is visible.

## Why this is non-obvious

Most developers write a one-off script for protocol discovery, run it once, and delete it. Making it a permanent test that runs in CI turns discovery into an ongoing coverage instrument. The insight is that "no assertions" is a valid test design — the value is in the printed output and the fact that it keeps running, not in pass/fail.

## Example application

SC2 replay parsing: `AbilityDiscoveryTest` scans `.SC2Replay` files and prints all observed `(abilLink, abilCmdIndex)` pairs. The mapping table is built from the output. When new replay files are added, the test reveals whether new ability IDs appear that need mapping.
