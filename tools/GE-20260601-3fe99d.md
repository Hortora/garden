## Symptom

CI fails with `cannot find symbol: variable tenancyId / location: variable entry of type io.casehub.ledger.model.CaseLedgerEntry`. The error names the class FQCN and the missing symbol, but not the Maven artifact that should provide them.

## Context

Any Java project where two Maven artifacts share a package namespace prefix. The compilation error is accurate but insufficient for diagnosis when multiple artifacts use the same package root.

## Root cause

`javac` reports the symbol and its declaring type, but has no awareness of which Maven artifact that type came from. When two artifacts share a namespace prefix (e.g., `io.casehub.ledger.model` from `casehub-engine-ledger` vs `io.casehub.ledger.runtime.model` from standalone `casehub-ledger`), the error message leads to the wrong artifact.

In the case above: `CaseLedgerEntry` is in `casehub-engine-ledger` (package `io.casehub.ledger.model`), not in the standalone `casehub-ledger` library (package `io.casehub.ledger.runtime.model`). The namespace prefix `io.casehub.ledger` is shared, so the error sends diagnostics — and cross-team communication — to the wrong place.

## Fix

When `cannot find symbol` points to a class in a shared namespace, identify the correct artifact by:

1. Checking which artifact declares that specific sub-package:
   ```bash
   find ~/.m2 -name "*.jar" | xargs -I{} jar tf {} 2>/dev/null | grep "casehub/ledger/model/CaseLedgerEntry"
   ```
2. Or grepping the source repo:
   ```bash
   find /path/to/repos -name "CaseLedgerEntry.java" 2>/dev/null
   ```
3. Then checking whether that artifact's published SNAPSHOT includes the missing symbol.

### Alternatives considered

- Reading the error message's package prefix and assuming it maps to the artifact of the same name — fails when namespaces overlap
- Checking `mvn dependency:tree` — shows artifact versions but not package-to-artifact mapping at the class level

## Why this is non-obvious

Developers assume the package prefix uniquely identifies the artifact (`io.casehub.ledger` → `casehub-ledger`). In a multi-artifact project where one standalone library and one framework-internal module share a root namespace, this assumption breaks. The error is accurate; the inference from package to artifact is wrong.
