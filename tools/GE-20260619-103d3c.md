## GitHub Packages Maven wildcard URL resolves across all org repos — one repository entry covers the entire org

**ID:** GE-20260619-103d3c
**Stack:** GitHub Packages, Maven 3.x
**Symptom:** Needing separate `<repository>` entries in `pom.xml` for each GitHub repo in an org that publishes Maven artifacts — or artifacts published by org repos fail to resolve in CI with 404.
**Context:** Any Maven project consuming SNAPSHOT or release artifacts published to GitHub Packages by multiple repos within the same GitHub org.

### What was tried (didn't work)

- Adding a separate `<repository>` block for each dependency repo (casehubio/engine, casehubio/ledger, casehubio/qhorus, etc.) — works but verbose and must be updated whenever a new dep repo is added
- Leaving out `<repositories>` entirely — Maven only checks Maven Central; GitHub Packages artifacts are not found

### Root cause

GitHub Packages supports a `*` wildcard in Maven repository URLs at the org level. The URL `https://maven.pkg.github.com/<org>/*` resolves artifact lookups across all repositories in that org. This is not documented prominently in the GitHub Packages Maven docs — it appears only in community examples and cross-project pom.xml patterns.

### Fix

Add a single `<repositories>` entry using the org-level wildcard:

```xml
<repositories>
    <repository>
        <id>github</id>
        <name>GitHub Packages — <org></name>
        <url>https://maven.pkg.github.com/<org>/*</url>
        <snapshots><enabled>true</enabled></snapshots>
    </repository>
</repositories>
```

The CI workflow needs `packages: read` permission and the `GITHUB_TOKEN` passed as the server password (matched by `server-id: github` in `setup-java`):

```yaml
- uses: actions/setup-java@v4
  with:
    java-version: '21'
    distribution: 'temurin'
    server-id: github
    server-username: GITHUB_ACTOR
    server-password: GITHUB_TOKEN
```

### Why this is non-obvious

The GitHub Packages Maven documentation shows per-repo URLs (`https://maven.pkg.github.com/OWNER/REPOSITORY`). Nothing in the official docs mentions the `*` wildcard form. It is discoverable only by reading pom.xml files from projects that already use it, or by trial and error.
