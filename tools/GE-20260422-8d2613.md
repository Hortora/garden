## Sentinel-guarded heredoc append for idempotent bash installer blocks

**Stack:** Bash
**What it does:** Installs a block of content into an existing file (e.g., a git hook, a config file) idempotently — appends the block on first run, skips on subsequent runs, without replacing or duplicating existing content.

### The technique

```bash
HOOK="$GARDEN/.git/hooks/post-commit"
SENTINEL="# my-tool: auto-installed"

if [[ -f "$HOOK" ]] && grep -q "$SENTINEL" "$HOOK"; then
    echo "already present — skipping"
else
    cat >> "$HOOK" << 'BLOCK_EOF'
# my-tool: auto-installed
# ... block content ...
BLOCK_EOF
    chmod +x "$HOOK"
    echo "installed"
fi
```

**Key rules:**
- Place the sentinel as the **first line of the block** — so the grep check and the installed content are in sync
- Use `>>` (append) not `>` (replace) — preserves existing hook logic
- Use a **quoted heredoc delimiter** (`<< 'BLOCK_EOF'`) if the block contains variables that should be literal in the installed file
- `chmod +x` after append (hooks must be executable regardless of pre-existing content)

### Why developers wouldn't naturally reach for this

The obvious approach is to check if the entire target file exists before writing (`if [[ ! -f "$FILE" ]]`). That fails when the file already exists with other content (e.g., a git hook with pre-existing logic). The sentinel pattern handles the append-to-existing case cleanly without parsing or replacing file content.
