### Symptom
A tree type (AST, filter expression, config schema) goes through stages — parsed, resolved, evaluated — where the leaf content changes but the tree structure (AND/OR/NOT, if/else, combinators) stays the same. The naive approach defines separate full types for each stage, duplicating the combinator definitions. Or worse, uses a single type with a runtime guard that throws if an unresolved leaf reaches evaluation.

### Context
Common in expression evaluators, query parsers, and config systems where data is parsed from a loosely-typed source (YAML, JSON, user input), then resolved against a schema (column types, variable types), then evaluated.

### Root cause
The tree structure is invariant across stages — only the leaves change. Duplicating the full tree type per stage violates DRY and means combinator changes must be applied in multiple places. A single type with all leaf variants means the compiler can't distinguish stages — you need runtime guards.

### Fix
Parameterize the tree type by its leaf type:

```typescript
type ExprTree<Leaf> =
  | Leaf
  | { type: "and"; children: readonly ExprTree<Leaf>[] }
  | { type: "or"; children: readonly ExprTree<Leaf>[] }
  | { type: "not"; child: ExprTree<Leaf> };

// Stage 1: parsed (leaves unresolved)
type UnresolvedLeaf = { type: "unresolved"; name: string; args: string[] };
type ResolvedLeaf = { type: "numeric"; value: number } | { type: "string"; value: string };

type ParsedExpr = ExprTree<ResolvedLeaf | UnresolvedLeaf>;
type ResolvedExpr = ExprTree<ResolvedLeaf>;

// Resolution narrows the type
function resolve(expr: ParsedExpr, schema: Schema): ResolvedExpr;

// Evaluation only accepts resolved trees — compiler enforces this
function evaluate(expr: ResolvedExpr): Result;
```

The combinator structure is defined once. Each stage's input/output is a distinct type. Passing a `ParsedExpr` to `evaluate()` is a compile error — no runtime guard needed.

### Why this is non-obvious
Most developers reach for one of two patterns: (1) a single union type with all variants plus a runtime guard, or (2) two fully separate type definitions with duplicated combinators. The parameterized approach sits between these — it's a standard generic technique but not one most developers naturally apply to discriminated union trees. The recursive generic (`ExprTree<Leaf>` where children are `ExprTree<Leaf>[]`) is the key insight that makes it work.

### Alternatives considered
- Single type + runtime guard — works but errors are caught at runtime, not compile time. A developer building a new code path can accidentally skip resolution.
- Duplicate full types per stage — correct at compile time but DRY violation. Adding a new combinator (e.g. XOR) requires changing every stage's type definition.
