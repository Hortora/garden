## Maven relative paths resolve to wrong worktree when shell cwd changes — use absolute paths

### Symptom

`mvn test -pl module` builds the wrong module, or `-f pom.xml` targets the main worktree instead of the feature branch worktree. Build results from the wrong branch — no error, just wrong output.

### What was tried (didn't work)

- Using `mvn -f pom.xml` — still resolves relative to current cwd, which may have changed
- `cd`-ing to the worktree before running Maven — works until the next tool invocation resets cwd

### Root cause

When working in a git worktree, any tool or shell context switch can reset the working directory to a different location (e.g. the main worktree or the original checkout). Relative paths then resolve to the wrong worktree silently — Maven builds succeed but against the wrong source tree.

### Fix

Always use the absolute path to the worktree's `pom.xml`:

```bash
# Wrong — breaks if cwd changes
mvn test -pl runtime

# Right — always targets the correct worktree
mvn test \
  -f /absolute/path/to/worktree/pom.xml \
  -pl runtime
```

### Why this is non-obvious

Relative paths work correctly in interactive shell sessions where the developer controls cwd. They fail silently in automated or tool-assisted contexts where cwd is not stable. The build succeeds — just against the wrong source tree, producing misleading results.
