Skills installed from Hortora (`forage`, `protocol`, and any other skill distributed via
`~/claude/hortora/soredium/`) have their source in `~/claude/hortora/soredium/<skill>/SKILL.md`.
Editing the installed copy at `~/.claude/skills/<skill>/SKILL.md` directly works for the current
session, but the next `python3 scripts/claude-skill sync-local --all -y` from the soredium
directory silently overwrites the edit with the upstream source.

### Symptom

A skill edit works in the current session but is gone after any sync — either a manual
`sync-local --all`, or any automated sync that runs on restart. No error is produced.

### Root cause

`sync-local` copies from the soredium source directory to `~/.claude/skills/` unconditionally.
It does not diff, merge, or warn about local modifications.

### Fix

Always edit the source first:
```bash
# Edit the source
vim ~/claude/hortora/soredium/<skill>/SKILL.md

# Commit and push soredium
git -C ~/claude/hortora/soredium add <skill>/SKILL.md
git -C ~/claude/hortora/soredium commit -m "feat(<skill>): ..."
git -C ~/claude/hortora/soredium push

# Sync to installed
cd ~/claude/hortora/soredium && python3 scripts/claude-skill sync-local --all -y
```

For cc-praxis skills (`work-start`, `work-end`, `git-commit`, etc.), the source is
`~/claude/cc-praxis/<skill>/SKILL.md` and the sync command is
`python3 scripts/claude-skill sync-local --all -y` from that directory instead.

### Why this is non-obvious

The installed path `~/.claude/skills/<skill>/` feels authoritative — it is what Claude
loads. Editing it seems both correct and sufficient. The source-of-truth indirection
(soredium → installed) is not visible from the file itself, and sync does not warn
about local changes.
