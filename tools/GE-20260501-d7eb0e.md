## Symptom

`git add <files>` followed by `git commit -m "message"` reports "nothing to commit, working tree
clean". The staged files have vanished — no error, no warning.

## Context

Repos with a pre-commit hook (Spotless, Black, lint-staged, etc.) that both reformats files
and stages them. If another commit operation is simultaneously building in the background
(e.g. an IDE auto-save + format triggers a commit), the hook may absorb your staged files into
that concurrent commit under a different message.

## What was tried (didn't work)

Re-running `git status` — showed clean. Re-running `git add` — still nothing to commit. The
files appeared to be committed, but under the wrong commit message.

## Root cause

The pre-commit hook staged and committed the files (correctly, from its perspective) as part
of a background commit triggered by an IDE action. By the time the user's `git commit` ran,
there was nothing left to commit. The files landed in git history under the wrong message.

## Fix

Diagnose with `git log --oneline -5` and `git show --stat HEAD` to confirm your files are in
the wrong commit. Then:

```bash
git reset HEAD~1          # un-commit, return files to working tree
git add <your files>      # re-stage only what you want
git commit -m "correct message"
# if already pushed:
git push --force-with-lease origin <branch>
```

## Why this is non-obvious

`git status` shows "nothing to commit" — the normal signal that all is well. There's no
indication that your files were committed elsewhere. You have to check `git log` and
`git show --stat` to discover what actually happened.
