**Context:** Any time-series event store — analytics pipelines, audit logs, trend tracking, metric collectors, garden entry adoption tracking.

### What the technique is

Every event record carries two timestamps:

- `observed_at` — when the thing actually happened in the world (the git commit date, the measurement time, the incident time)
- `indexed_at` — when your system processed and stored the record

Most systems only store `indexed_at` (typically auto-set as a `created_at` column on insert). This conflates processing time with event time. It works perfectly until you need to backfill historical data — at which point every historical record has the wrong timestamp and a migration is required.

Separating the two fields means retroactive ingestion is free: insert historical events with their real `observed_at`. The analytics layer queries by `observed_at` range and doesn't care when you processed the data.

### Fix

```python
event = {
    "observed_at": commit_date,    # when it happened — the fact
    "indexed_at": datetime.now(),  # when we processed it — operational metadata
    "payload": ...,
}
```

Analytics queries use `observed_at`:

```sql
SELECT * FROM events
WHERE observed_at BETWEEN '2022-01-01' AND '2022-12-31'
ORDER BY observed_at;
```

Backfill is then a batch job inserting historical events with their real `observed_at` — no migration, no schema change, no data correction.

### Why this is non-obvious

The default database pattern (auto-set `created_at` on insert) feels sufficient because the system works correctly for all forward-looking queries. The gap only surfaces when backfill is needed, which may be months or years after the schema was designed. At that point, `created_at` is permanently wrong for every historical record. The separation costs one extra field at schema design time — trivial to add, easy to overlook, expensive to retrofit.

### Alternatives considered

- Using only `indexed_at` — works until backfill needed, then requires a migration and leaves historical records with incorrect timestamps
- Storing only `observed_at` — loses the ability to audit when data was processed, which matters for debugging pipeline delays
