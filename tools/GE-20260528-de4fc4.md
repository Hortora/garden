### Symptom
`gh pr view --json mergeable,mergeStateStatus` returns `{"mergeStateStatus":"DIRTY","mergeable":"CONFLICTING"}`. GitHub's PR UI also shows the branch as unmergeable. `gh pr merge` fails with "Pull request is not mergeable: the merge commit cannot be cleanly created."

### Context
Feature branch has a clean diff against base — `git diff base...branch --name-only` lists only files from the feature branch with no overlapping changes. The base branch was pushed to origin in the same session, minutes before the feature branch was pushed and the PR was created.

### Root Cause
GitHub computes merge test state asynchronously. When a base branch and feature branch are pushed in close succession, GitHub may not have finished recomputing the merge state before the PR is queried. The `DIRTY/CONFLICTING` state is transient — not a real git conflict.

### What Was Tried (Didn't Work)
- Waiting and retrying `gh pr merge`
- Rebasing the feature branch onto `origin/main` (rebase was a no-op — branch was already current)
- `gh pr merge --auto` — fails (auto-merge not enabled on repo)
- `gh pr merge --rebase` — same CONFLICTING error

### Fix
Merge locally and close the PR manually:
```bash
git checkout main
git merge --squash <feature-branch>
git commit -m "squash merge message"
git push origin main
gh pr close <N> --repo <owner/repo> --comment "Merged locally — DIRTY state was transient."
```

### Why This Is Non-Obvious
`CONFLICTING` strongly implies a real merge conflict requiring resolution. `gh pr view --json mergeable,mergeStateStatus` returns a definitive JSON response, not a "pending" state. Nothing in the output hints that the state is still being computed. The fix (local merge + manual PR close) is not visible from any error message.

### Why This Fix
Merging locally is reliable regardless of GitHub's computation state. The alternative (waiting for GitHub to recompute) has no reliable signal for when it's complete — `mergeStateStatus: BLOCKED` or `UNKNOWN` would indicate pending computation, but `CONFLICTING` is indistinguishable from a real conflict in the API response.
