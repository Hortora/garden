## PixiJS Graphics loop over full grid size draws invisible off-screen tiles silently

**ID:** GE-20260416-41a9cb
**Stack:** PixiJS 8.x
**Symptom:** No error, no visual artifact — everything looks correct. Performance degrades silently as the fog/grid overlay scales up. 50% or more of `PIXI.Graphics.rect().fill()` calls produce negative canvas Y coordinates and are clipped by the renderer before reaching the GPU, but still consume batch budget.
**Context:** A grid-based game where the logical grid (e.g. 64×64 tiles) is larger than the visible viewport (e.g. 32×32 tiles). A fog-of-war or tilemap overlay loops over the full logical grid size instead of only the visible portion.

### What was tried (didn't work)

- Iterating `y` from 0 to `GRID_SIZE` (64) — draws 2048 invisible tiles per frame with no complaint from PixiJS
- Checking visual output — the scene looks correct because off-screen geometry is clipped before display

### Root cause

PixiJS 8's `Graphics` batch-draw submits all geometry through its internal batcher regardless of whether rects land within the visible canvas bounds. Off-canvas geometry is clipped by the renderer *after* being processed. Iterating `y < GRID_SIZE` when the viewport only shows `y < VIEWPORT_H` rows causes the loop to produce rects at negative canvas Y coordinates (e.g. y=32 → `canvasY = (32 - 32 - 1) * 20 = -20px`), all of which are clipped but still counted in the draw batch.

### Fix

Replace the loop bound with the viewport height constant, not the grid size:

```js
// Before — iterates full 64 rows, 2048 off-screen rects per frame:
for (let y = 0; y < GRID_SIZE; y++) { ... }

// After — iterates only the 32 visible rows:
for (let y = 0; y < VIEWPORT_H; y++) { ... }
```

The visibility string is still encoded for the full grid (64×64) — just don't try to render rows that can't appear on screen.

### Why this is non-obvious

The outer loop naturally mirrors the data structure (64×64 grid → loop 64 times). The viewport height is an unrelated rendering constant that doesn't appear until you think about canvas coordinates. Since PixiJS produces no warning and the scene renders correctly, there is no feedback that half the draw work is wasted. The bug only surfaces under profiling or code review.
