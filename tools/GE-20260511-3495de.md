### Symptom

`ide_find_class` returns `"kind": "CLASS"` for a Java interface. Code that assumes
`kind: INTERFACE` to detect interfaces will incorrectly classify them as classes.

### Context

When using the `mcp__intellij-index` MCP server's `ide_find_class` tool to locate
Java types and check whether something is an interface vs a concrete class.

### Root cause

The `ide_find_class` tool does not distinguish between classes and interfaces in its
`kind` field — both are returned as `"CLASS"`. The tool's vocabulary for `kind` is
limited and does not map to Java's full type hierarchy.

### Fix

Use `get_symbol_info` (from `mcp__intellij`) at the symbol's position to get the
full declaration string, which will show `public interface Foo` vs `public class Foo`:

```
# ide_find_class result — unreliable for interface detection
{"name": "WorkflowExecutor", "kind": "CLASS", ...}

# get_symbol_info at line/column — reliable
{"documentation": "public interface WorkflowExecutor ..."}
```

### Why this is non-obvious

`ide_find_class` is the natural tool for finding class metadata. Its `kind` field
looks authoritative but silently collapses interfaces and classes into the same value.
The documentation does not mention this limitation.
