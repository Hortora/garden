### Symptom
Issue remains open after a commit containing `Closes #N` (or `Fixes #N`, `Resolves #N`)
in the message is pushed directly to the default branch (`main`/`master`).

### Context
When commits are pushed directly to `main` (not via a merged PR), GitHub's keyword-based
auto-close does not always fire. Observed in a repository where `Closes #26` appeared in
a commit message on `main` — the issue was still open the following session.

### Root cause
GitHub's auto-close keyword processing may require the commit to arrive via a **merged
Pull Request** targeting the default branch, not a direct push. The documentation states
direct pushes to the default branch are supported, but behaviour is inconsistent in
practice — particularly when commits are not the HEAD of a PR merge.

### Fix
Always verify issue state after a session that relied on commit-message closes:
```bash
gh issue list --repo owner/repo --state open
```
Close manually if still open:
```bash
gh issue close N --repo owner/repo
```
Or adopt the habit of closing issues explicitly via `gh` at the end of each session
rather than relying on commit message parsing.

### Why this is non-obvious
GitHub documentation states that push to the default branch triggers auto-close.
The failure is silent — no error, no warning, the issue just stays open. Developers
only discover it when reviewing open issues in a later session.
