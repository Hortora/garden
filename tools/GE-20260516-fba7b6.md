## Context

Running git operations across multiple `Bash` tool calls in a Claude Code session where two git repositories are active (e.g. a project repo and a workspace repo).

## What most developers would do

Use `cd /path && git add file && git commit` — or run a `cd` in one Bash call and expect it to persist for subsequent calls.

## The non-obvious behaviour

Each `Bash` tool call in Claude Code executes in an **independent subshell**. `cd` state does not persist between tool calls. A `cd /workspace` in call N reverts to the original CWD in call N+1.

This means:
- `cd /workspace && git add HANDOFF.md` in one call works
- `git commit -m "..."` in the next call silently commits to the *wrong repo* (reverts to original CWD)
- No error is raised — git finds the nearest `.git` directory above CWD

## The technique

Use `git -C <absolute-path>` for every git operation. It targets a specific repo without changing CWD, is self-contained per command, and works correctly regardless of which call it appears in:

```bash
# Always correct — targets workspace regardless of CWD
git -C /Users/mdproctor/claude/public/quarkmind add HANDOFF.md
git -C /Users/mdproctor/claude/public/quarkmind commit -m "docs: session handover"

# Always correct — targets project regardless of CWD
git -C /Users/mdproctor/claude/quarkmind add docs/adr/0001-foo.md
git -C /Users/mdproctor/claude/quarkmind commit -m "adr: 0001 foo"
```

Read the workspace and project paths from `**Workspace:**` and `**Project repo:**` properties in CLAUDE.md to get the absolute paths.

## Why this is non-obvious

Developers expect shell state to accumulate within a session. The independent-subshell behaviour is not prominent in Claude Code documentation. The failure mode is also silent — git finds a nearby `.git` and commits there without complaint, so staged files appear to commit successfully while landing in the wrong repository.
