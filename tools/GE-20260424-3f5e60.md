## Symptom

`gh api -X POST repos/owner/repo/transfer -f new_owner=target_org` returns HTTP 200
with the repository's current details (still showing the original owner). Immediately
checking for the repo at the new location fails:

```bash
gh repo view casehubio/quarkus-qhorus
# → GraphQL: Could not resolve to a Repository with the name 'casehubio/quarkus-qhorus'
```

## Context

Transferring a GitHub repository to a different organisation using the REST API
or `gh` CLI. The API call succeeds (200) but the transfer is not yet complete.
On a moderately-sized repository, the transfer takes 15–60 seconds.

## Root Cause

GitHub's repository transfer is an asynchronous background operation. The API
acknowledges the request immediately and returns the repository's current state,
but the actual transfer — migrating issues, PRs, webhooks, and updating DNS —
happens in the background. The response body still shows the original owner
because the transfer has not completed at response time.

## Fix

Poll until the new location resolves:

```bash
until gh repo view casehubio/quarkus-qhorus --json name 2>/dev/null; do
  echo "still transferring..."
  sleep 5
done
echo "Transfer complete"
```

Or use the API to check:
```bash
gh api repos/casehubio/quarkus-qhorus --jq '.full_name' 2>/dev/null
```

## Why Non-Obvious

The 200 response implies completion. The response body contains repository data,
reinforcing the impression that the transfer is done. The new location returning
"not found" looks like a permissions or naming error rather than a timing issue.
GitHub's API documentation mentions asynchronous processing but it is easy to miss.
