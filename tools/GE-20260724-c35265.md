### Symptom
Deleting an inner class `ReactiveConfig` at line 160 from `LedgerConfig.java` instead deleted `MerklePublishConfig` — a completely different inner class at a different (post-shift) line.

### Context
Two sequential `ide_refactor_safe_delete` calls targeted the same file. The first deleted the `reactive()` method, which shifted all subsequent line numbers upward. The second call used the pre-shift line number for `ReactiveConfig`, which now pointed at `MerklePublishConfig`.

### Root cause
`ide_refactor_safe_delete` with `line` parameter uses the line number at call time. After a structural edit (deletion, insertion) in the same file, line numbers shift but the caller's cached line numbers are stale.

### Fix
Always re-run `ide_file_structure` between sequential deletions targeting the same file by line number. Use the fresh line numbers from the updated structure.

```
// WRONG — second deletion uses stale line from first ide_file_structure call
ide_refactor_safe_delete(file, line=53, ...)  // deletes reactive()
ide_refactor_safe_delete(file, line=160, ...) // targets ReactiveConfig but hits MerklePublishConfig

// RIGHT — re-check structure between deletions
ide_refactor_safe_delete(file, line=53, ...)  // deletes reactive()
ide_file_structure(file)                       // get updated line numbers
ide_refactor_safe_delete(file, line=143, ...) // correct line for ReactiveConfig
```

### Why this is non-obvious
The first deletion succeeds silently. The second deletion also succeeds — it deletes a valid member, just the wrong one. No error is raised. The mistake is only discovered at build time when the accidentally-deleted member is missing.
