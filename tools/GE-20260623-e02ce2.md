### Symptom / Context
When verifying §12 issue references in ARC42STORIES.MD (or any architecture doc),
reading the issue body is unreliable. Issue bodies are written before the fix —
they describe the problem, root cause, and proposed upstream change. After the issue
is closed as COMPLETED, the body still reads as if the fix is pending.

Found during the claudony #145 quality gate: issue #93 body described a pending
upstream fix ("requires casehub-engine to add caseId() to WorkResult — one-line
upstream change"). stateReason=COMPLETED and git log --grep="#93" revealed commit
db61484 ("feat(casehub): use precise caseId lookup in onWorkerCompleted — fixes #93").

### What most developers would do instead
Read the issue body to understand status, or just check `state: OPEN/CLOSED` without
inspecting `stateReason`.

### The technique
Use `gh issue view N --repo owner/repo --json state,stateReason` to get both fields.
Map stateReason to meaning:
- `COMPLETED` → issue was implemented; verify with `git log --grep="#N"`
- `NOT_PLANNED` → deferred or won't-fix; still open technical debt
- `DUPLICATE` → closed as duplicate; check the canonical issue instead
- `REOPENED` → was closed, now active again

Then cross-check with git history to find the implementing commit:
```bash
git log --grep="#N" --oneline
```

The issue body may be stale — git history is authoritative.

### Why this is non-obvious
`state: CLOSED` + a body describing a pending fix looks like deferred technical debt.
Without `stateReason`, a doc author writes the item into §12 as outstanding when the
code already has the fix. The body is the only prose description; it's natural to
read it as the current status.
