## Symptom

A job with `continue-on-error: true` steps appears green in the GitHub UI and API (`conclusion: "success"`), but the build summary reports failures. Downstream logic using `steps.<id>.outcome` sees failures that the API hides.

## Context

Full-stack build workflow where each module is a separate step with `continue-on-error: true` so all modules build regardless of individual failures. A summary step reads `steps.<id>.outcome` to count failures and exit 1 if any. The GitHub API's job status shows `conclusion: "success"` for failed steps, making the job appear green externally while the summary step correctly exits 1.

## Root cause

GitHub Actions has two distinct fields:
- `steps.<id>.conclusion` — the reported result **after** `continue-on-error` is applied. A failed step with `continue-on-error: true` gets `conclusion: "success"`.
- `steps.<id>.outcome` — the **actual** result before `continue-on-error` masking. A failed step always has `outcome: "failure"`.

The GitHub API's `/actions/runs/{id}/jobs` endpoint returns `conclusion` (masked), not `outcome`. So the API looks green while the build is actually broken.

## Fix

Use `steps.<id>.outcome` (not `conclusion`) in summary/gate logic:

```yaml
env:
  OUTCOME_MODULE: ${{ steps.module.outcome }}   # "failure" even with continue-on-error
```

To see real per-step status via API, check the env vars passed to the summary step, not the job-level conclusion.

## Why non-obvious

The GitHub docs mention both fields but the distinction is easy to miss. The UI and API consistently show the masked `conclusion`, so it looks like the job passed. Only logic that explicitly reads `outcome` sees the truth.

**How to apply:** Any build summary or gate step that needs to know whether a `continue-on-error` step actually passed must use `steps.<id>.outcome`.
