## Symptom

A "delete all SNAPSHOT packages" workflow targeting `casehubio` org returns 226 packages when only ~49 are owned by the org. External packages (e.g. `io.quarkiverse.langchain4j.*`) appear in the results and would be deleted if not filtered.

## Context

`GET /orgs/{org}/packages?package_type=maven` returns every Maven package stored in the org's GitHub Packages registry — regardless of groupId or who published it. External library SNAPSHOTs published to the org registry during development (e.g. pre-release Quarkus extensions) are indistinguishable from org-owned packages by the API alone.

## Root Cause

The GitHub Packages API has no `groupId` filter parameter. The only way to scope the results is to filter client-side on the package name prefix (GitHub Packages uses `<groupId>.<artifactId>` as the package name).

## Fix

Always filter by `group_filter` before deleting. In a shell script:

```bash
while IFS= read -r pkg_name; do
  # Skip packages not matching the org's groupId prefix
  if [ -n "$GROUP_FILTER" ] && [[ "$pkg_name" != "$GROUP_FILTER"* ]]; then
    continue
  fi
  # ... delete logic
done <<< "$packages"
```

Default the filter to the org's own groupId (e.g. `io.casehub`) and require it to be explicitly cleared to delete externals.

## Why This Is Non-Obvious

The API endpoint name implies org ownership — "org packages" — but it reflects the registry's contents, not what the org created. A dry-run showing 226 packages where you expected ~50 is the first signal. Without a dry-run, the external packages would be silently deleted.
