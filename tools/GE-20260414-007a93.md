## Use a module-level configurable dict for scoring/bonus rules so strategy is data not code

**ID:** GE-20260414-007a93
**Stack:** Python (any version)
**Labels:** `#strategy` `#pattern` `#configuration`
**What it achieves:** Bonus/scoring rules become tunable at runtime by editing one dict, with no changes to detection logic. Downstream consumers can import the same dict to stay in sync.
**Context:** Any system with configurable scoring, classification, or bonus rules that need to be adjusted without code changes — validators, linters, scorecards.

### The technique

Define rules as a module-level dict, not as hardcoded logic inside functions:

```python
# validate_pr.py — rules as data
BONUS_RULES = {
    'constraints':             1,   # +1 if constraints field present
    'alternatives_considered': 1,   # +1 if ### Alternatives considered has ≥1 list item
    'invalidation_triggers':   1,   # +1 if invalidation_triggers field present
}

def bonus_points(bonus_results: dict) -> int:
    """Sum bonus points from detection results using BONUS_RULES."""
    return sum(BONUS_RULES.get(k, 0) for k, present in bonus_results.items() if present)
```

Downstream consumers import the same dict — scoring never diverges:

```python
# contributors.py — imports rules, doesn't redefine them
from validate_pr import BONUS_RULES, compute_bonus, bonus_points
```

To add a new bonus field: edit `BONUS_RULES` only. Detection logic reads from it automatically.
To change a bonus value: edit one number. No logic changes needed.

### Why this is non-obvious

The instinct is to hardcode the bonus values inside the detection function:
```python
def bonus_points(results):
    points = 0
    if results.get('constraints'): points += 1
    if results.get('alternatives'): points += 1
    return points
```

This works but makes the rules invisible — they're buried in control flow. When a downstream consumer (scoreboard, reporter) needs the same rules, they either duplicate them (drift risk) or import the function (but can't inspect the rules without parsing source). The dict pattern makes rules a first-class artifact: importable, inspectable, patchable at test time without monkey-patching functions.

### When to use it
Any time scoring/classification/bonus rules may need tuning without code changes, or when multiple modules need to share the same rule set. Especially valuable when rules are likely to evolve (adding new dimensions, adjusting weights).
