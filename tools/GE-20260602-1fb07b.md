## What happens

A documentation section is restructured: prose converts to bullet lists, references move to header metadata, body text is tightened. The rewrite looks correct. Later, a specific class name, a scope boundary ("not closed here"), or a cross-reference turns out to be missing. It was dropped during restructuring — not intentionally, just lost in the transformation.

## The technique

Before touching any content, enumerate every atomic fact in the source section:

1. File paths and class names
2. Configuration values, flags, environment variables
3. Issue numbers, commit hashes, blog links
4. Explicit consequences ("without this, X fails with Y")
5. Scope boundaries ("not closed here", "not applicable to Y")
6. References to external documents

Record the list. Rewrite the section. After rewriting, verify every fact appears in the output — in the body or in header metadata fields (Design refs, Navigation, Key protocols, etc.).

**Loss check:** A fact not present in either location is a loss, not a simplification. References for depth move to header fields; actionable content stays in the body.

**Strip test as complement:** After verifying fact coverage, strip all labels and read only those in sequence. Any fact visible only in prose body text is a future loss waiting to happen.

## Why non-obvious

The natural approach is to rewrite and then read the result to see if it "feels complete." Feeling complete is not the same as being complete — a well-written section with a missing scope boundary or dropped cross-reference reads fine and fails functionally. Enumeration makes loss detectable before it causes a problem.
