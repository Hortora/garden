### Symptom
`mvn deploy` returns HTTP 422 (Unprocessable Entity) when deploying a Maven SNAPSHOT artifact to GitHub Packages. The artifact version has a `-2` timestamp suffix (e.g. `0.2-20260717.131744-2`), suggesting Maven thinks a `-1` version already exists.

### Context
When a first Maven SNAPSHOT deploy to GitHub Packages partially succeeds (some artifacts uploaded) then fails (e.g. wrong `distributionManagement` URL, 403 on a different module), the already-uploaded artifacts create a ghost package in the registry. Subsequent deploy attempts see the cached `-1` metadata and try to upload `-2`, which GitHub Packages rejects with 422.

The ghost package is invisible in `gh api orgs/<org>/packages?package_type=maven` listings but is detectable via direct Maven metadata URL fetch:
```bash
curl -H "Authorization: Bearer $(gh auth token)" \
  "https://maven.pkg.github.com/<org>/<repo>/io/casehub/<artifact>/0.2-SNAPSHOT/maven-metadata.xml"
```

### What was tried (didn't work)
- Re-running the deploy — same 422
- Changing the deploy command to use `-pl` (skip reactor POM) — 422 on the next module that had a ghost
- Using `-DaltDeploymentRepository` to force the URL — still 422 (ghost is at the target URL)
- Clearing the Maven cache key in CI — doesn't help; the ghost is server-side

### Root cause
GitHub Packages treats each uploaded artifact as immutable. A failed deploy that uploaded some files but not all leaves the registry in an inconsistent state. Maven's deploy plugin sees the server-side metadata showing `-1` exists and increments to `-2`, which the server rejects because the `-1` upload was incomplete.

### Fix
Delete the ghost package via the GitHub API, then re-run the deploy:
```bash
# Find the ghost package
gh api "orgs/<org>/packages/maven/io.casehub.<artifact>" --jq '.id'

# Delete it (if it's the only version, delete the whole package)
gh api --method DELETE "orgs/<org>/packages/maven/io.casehub.<artifact>"

# Re-run deploy
gh workflow run <workflow>.yml --repo <org>/<repo>
```

Check each module that might have a ghost — a multi-module reactor can leave ghosts for any module that uploaded before the failure.

### Why this is non-obvious
The 422 error gives no indication that a ghost package exists. The package doesn't appear in API listings (`gh api orgs/.../packages`). The `-2` suffix in the artifact version is the only clue, and it's easy to misread as a Maven cache issue rather than a server-side state problem.
