## Python regex alternation matches leftmost option — longer pattern must come before any pattern that is a prefix of it

**ID:** GE-20260412-e4773d
**Stack:** Python `re` module (all versions)
**Symptom:** A regex alternation `(?:SHORT|LONG)` where SHORT is a prefix of LONG silently matches SHORT even when LONG would fully match. No error raised.
**Context:** Any Python regex using `|` where one pattern can match a prefix of what the other pattern would fully match. Surfaced with `(?:GE-\d{4}|GE-\d{8}-[0-9a-f]{6})` applied to `GE-20260410-5fd0c3` — matched `GE-2026` (first 4 digits of the date), not the full new-format ID.

### What was tried (didn't work)

```python
GE_ID_LEGACY = r'GE-\d{4}'
GE_ID_NEW    = r'GE-\d{8}-[0-9a-f]{6}'
GE_ID_ANY    = rf'(?:{GE_ID_LEGACY}|{GE_ID_NEW})'
# Applied to "GE-20260410-5fd0c3" → matched "GE-2026"
```

Both patterns are individually correct and tested. Bug only appears when applied to strings that both patterns can partially/fully match.

### Root cause

Python's `re` engine implements alternation as **ordered choice, left-to-right**: the engine tries the first alternative at the current position; if it matches, it returns immediately without trying remaining alternatives. `GE-\d{4}` matches `GE-2026` (the first 4 digits of `20260410`) and stops — the engine never tries `GE-\d{8}-[0-9a-f]{6}`, which would have matched the full string.

This is **not backtracking** — Python `re` does not look ahead to find a longer/better match once a shorter one succeeds at the current position.

### Fix

Put the longer/more-specific pattern first in the alternation:

```python
# WRONG — GE-\d{4} wins for any string starting with "GE-" + 4 digits
GE_ID_ANY = rf'(?:{GE_ID_LEGACY}|{GE_ID_NEW})'

# CORRECT — new format tried first; falls through to legacy only if it doesn't match
GE_ID_ANY = rf'(?:{GE_ID_NEW}|{GE_ID_LEGACY})'
```

General rule: in any Python regex alternation where pattern A is a prefix of pattern B, B must appear before A.

### Why non-obvious

Both patterns are individually correct. The bug only surfaces when one pattern is a prefix of what the other would match — then the shorter one silently "wins" for strings the longer one would fully handle. Developers familiar with other regex engines (PCRE, Java) expect the engine to prefer the longest match; Python `re` does not. The result is wrong data with no error or warning. The fix is trivial once you understand the cause, but diagnosing it requires knowing that Python alternation is ordered choice, not longest-match.
