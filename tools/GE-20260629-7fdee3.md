### Symptom

`gh repo fork <org>/<repo> --clone=false` returns:

```
failed to fork: HTTP 403: The repository exists, but it contains no Git content.
Empty repositories cannot be forked. (https://api.github.com/repos/<org>/<repo>/forks)
```

### Context

When creating a new GitHub repo (`gh repo create`) and immediately attempting to fork it before any commits have been pushed. Common in automated repo setup scripts that create → fork → rewire remotes as sequential steps.

### Root cause

GitHub's fork API requires at least one commit (a non-empty default branch) before the repository can be forked. The API returns 403, not a more descriptive error code.

### Fix

Push at least one commit to the org repo before forking:

```bash
# 1. Create and clone
gh repo create casehubio/new-repo --public
git clone https://github.com/casehubio/new-repo.git

# 2. Push initial commit FIRST
git add . && git commit -m "chore: initial scaffold"
git push origin main

# 3. NOW fork succeeds
gh repo fork casehubio/new-repo --clone=false

# 4. Rewire remotes
git remote rename origin upstream
git remote add origin https://github.com/$USER/new-repo.git
```

### Why this is non-obvious

The `gh repo create` docs don't mention that empty repos can't be forked. The error message says "Empty repositories cannot be forked" but this constraint isn't surfaced during repo creation or in the fork command's help text. A developer following a create-then-fork workflow hits this on every new repo.
