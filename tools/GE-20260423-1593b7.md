## Context

A Playwright test asserts that unit sprites in a Three.js scene are positioned above the terrain surface. The terrain surface height differs by Quarkus profile (mock: 0.08, emulated: TILE=0.7). Hardcoding the threshold makes the assertion fragile across profiles.

## Technique

Read the live JavaScript variable directly from the running page instead of hardcoding the threshold:

```java
// Instead of hardcoding:
// assertThat(y).isGreaterThan(0.08); // breaks in emulated mode

// Read the actual value from the JS runtime:
double terrainSurfaceY = ((Number) page.evaluate("() => TERRAIN_SURFACE_Y")).doubleValue();

List<Double> ys = ((List<?>) page.evaluate("() => window.__test.allEnemyWorldY()"))
    .stream().map(v -> ((Number) v).doubleValue()).toList();

for (double y : ys) {
    assertThat(y)
        .as("sprite Y %.3f must be above terrain surface %.3f".formatted(y, terrainSurfaceY))
        .isGreaterThan(terrainSurfaceY);
}
```

`TERRAIN_SURFACE_Y` is a module-level JS `let` variable updated during `loadTerrain()`. `page.evaluate()` reads its current value from the live browser runtime.

## Why this is non-obvious

The instinct is to replicate the profile-detection logic in Java (check the active profile, pick a constant). Reading the value from the JS runtime is simpler and always stays in sync — if the JS variable changes, the test threshold changes automatically with no Java edits. This works for any module-level JS variable that reflects runtime state.

## Caveats

The variable must be read after the page is fully initialised (after `waitForFunction("() => window.__test?.threeReady?.() === true")`). Reading it too early may return `undefined` if the terrain hasn't loaded yet.
