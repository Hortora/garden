## Resolve knowledge scope at capture time to prevent audit debt

**Stack:** Knowledge management systems, AI-assisted capture workflows

### What it achieves

Knowledge captured without a scope decision accumulates misclassification silently. Dedicated audits then exist to reclassify content retroactively — growing unboundedly as the system grows. Resolving scope at the moment of capture eliminates this class of audit entirely.

### The technique

At capture time, before writing the entry, apply three questions in order:

**1. Reformulation test:** Can this knowledge be stated in universal terms without losing meaning?
Strip all project-specific names and references. If the reformulated version is still true and useful to someone on a different project — write the universal version and route to the universal store.

**2. Three-tier scope check** (only if reformulation fails):
- Relevant only to those building this exact project → project-scoped location
- Relevant to those building this type of platform → shared/platform-scoped location
- Relevant to anyone depending on or using the platform → universal store

**3. Search before writing:** Does an entry already exist covering this knowledge?
If yes — enrich the existing entry (REVISE) rather than creating a duplicate. If the intent is to mandate rather than document, route to the protocol index instead.

### Why this is non-obvious

The natural instinct is "capture first, sort later" — adding a classification decision to the capture moment feels like friction. In practice, retroactive classification is far more expensive: the original context is gone, the author is unavailable, and the evidence for the scope decision no longer exists. Audits that should handle drift and staleness instead spend most of their time on structural misclassification.

The reformulation step is the key shift in thinking: instead of asking "is this project-specific?" (which biases toward keeping things local), ask "can this be made universal?" (which biases toward broader value).

### Alternatives considered

- **Capture everything to the universal store, prune later** — creates noise in universal indexes and degrades search quality for all users
- **Dedicated classification audits on a schedule** — works initially but audit size grows with corpus; classification decisions made without original context are lower quality
