### Symptom

A test designed to verify CRLF (`\r\n`) line ending handling passes on CI and local — but the CRLF fix is never actually exercised. Removing the fix doesn't break the test.

### Context

When testing parsers that split on `\n` (YAML frontmatter, config files, etc.), you need fixture files with `\r\n` line endings to verify the parser handles Windows-created content. The natural approach: create a `.md` fixture file with `\r\n` endings and check it into the repo.

### Root cause

Git's `core.autocrlf` setting or `.gitattributes` rules can silently normalize `\r\n` to `\n` on checkout. The fixture file in the working tree has `\n` endings regardless of what was committed. The test reads `\n`-only content, the parser handles it fine (it always could), and the test passes — giving false confidence that the CRLF fix works.

### Fix

Construct CRLF content programmatically in the test. Never use fixture files for line-ending tests.

```java
@Test
void crlfLineEndingsParseCorrectly() {
    byte[] content = "---\r\ntitle: \"CRLF test\"\r\ndomain: jvm\r\n---\r\nBody with CRLF."
            .getBytes(StandardCharsets.UTF_8);

    ExtractionResult result = extractor.extract("test/crlf.md", content);

    assertThat(result.body()).isEqualTo("CRLF test\n\nBody with CRLF.");
}
```

For parsers that read from files (e.g. `Files.readString(path)`), write the programmatic content to a `@TempDir` file — the temp directory is outside git's control:

```java
@Test
void crlfSchemaParses(@TempDir Path tempDir) throws IOException {
    String content = "---\r\nid: test\r\nid-prefix: T\r\n---\r\n";
    Path crlfFile = tempDir.resolve("schema.yaml");
    Files.writeString(crlfFile, content);
    // parse crlfFile...
}
```

### Alternatives considered

- **Mark CRLF fixtures as binary in `.gitattributes`** (`*.crlf.md binary`) — works but adds `.gitattributes` maintenance overhead and is fragile (someone removes the rule, tests silently break)
- **Use `git add --no-convert`** — only affects the add, not the checkout; doesn't solve the problem

### Why this is non-obvious

The test appears correct — it uses a fixture file with the right content, asserts the right behavior, and passes. The false positive is invisible unless you specifically verify that removing the fix breaks the test. Most developers trust a green test.
