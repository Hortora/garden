## Symptom
Running `git filter-repo --path <file> --invert-paths --refs refs/heads/<branch>` on an
existing (non-fresh) clone exits immediately with:

```
Aborting: Refusing to destructively overwrite repo history since
this does not look like a fresh clone.
  (expected freshly packed repo)
Please operate on a fresh clone instead.  If you want to proceed
anyway, use --force.
```

## Context
Using `--refs refs/heads/<branch>` to scope a filter to one working branch in an
existing repository. The expectation is that scoping to a single branch makes the
operation safe enough to run without a fresh clone.

## What was tried (didn't work)
Running with `--refs refs/heads/$WORK_BRANCH` alone — exits with the error above
regardless of how narrow the `--refs` scope is.

## Root cause
The `--refs` flag controls *which refs are rewritten*, but the fresh-clone safety
check is a completely separate guard. It inspects repo pack state, not the scope
of the requested operation. The two are independent — `--refs` does not satisfy
or bypass the freshness check.

## Fix
Add `--force`:

```bash
git filter-repo \
  --path droolsvol2/HANDOFF.md \
  --invert-paths \
  --prune-empty always \
  --refs "refs/heads/$WORK_BRANCH" \
  --force
```

## Why this is non-obvious
The `--refs` flag description implies the operation is scoped and contained to
one branch, which sounds like a safe, reversible operation. The fresh-clone
check feels like a global safety gate that a narrow scope should bypass. The
docs don't call out that `--refs` and the fresh-clone guard are orthogonal.
