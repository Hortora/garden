### Technique

When writing a specification document that an LLM reads before generating content, advisory
references to external constraint files are silently skipped. Only an explicit file-list gate
with a "do not generate until loaded" instruction actually enforces the loading.

### Advisory (fails silently)

```markdown
## Writing Style

Constraint sets for each mode live in the `write-content` skill (`modes/` directory).
When using the skill, it loads the correct mode file automatically. When writing
without the skill, apply the structural rules below directly.
```

The LLM reads this, notes where the files live, and proceeds without loading them. The phrase
"apply the structural rules below directly" gives it permission to skip loading entirely.

### Gate (enforces loading)

```markdown
### Generator pre-conditions

Before generating any section of this document, load the following files from the
`write-content` skill. Do not generate any content until all are loaded.

- `voice/anti-slop.md` — universal banned words and banned patterns
- `voice/mandatory-rules.md` — voice and content policy
- `forms/technical-documentation.md` — dual-audience rules
- `modes/reference.md` — constraint set for lookup sections
- `modes/explanations.md` — constraint set for comparative sections
- `modes/how-to.md` — constraint set for diagnostic sections
- `modes/tutorial.md` — constraint set for pattern-to-replicate sections
- `modes/argumentation.md` — constraint set for decisions and ADRs
```

The enumerated file list + "do not generate until loaded" leaves no interpretation gap.

### Why this is non-obvious

A skilled developer writing a spec document naturally reaches for advisory prose — "these
constraints live in X, apply them when writing Y." This is the correct way to document a
system for human readers. Human readers can load and apply constraints mentally; an LLM
treats a pointer the same as permission to skip. The distinction is: advisories tell the LLM
*where* to find constraints; gates tell it *when* to load them (before generating anything).

The failure is invisible without side-by-side comparison: generated content that skips mode
loading looks plausible but exhibits systematic wrong-mode patterns (prose where tables should
be, hedging where direct statements are required). It does not produce an error.

### Pattern

Structure the gate section at the top of the "Writing Style" or equivalent section:

1. Name the section explicitly: `### Generator pre-conditions`
2. State the timing: "Before generating any section of this document..."
3. Enumerate specific file paths (not directories): one file per bullet with purpose annotation
4. State the gate: "Do not generate any content until all are loaded."

The gate works for any multi-mode document where different sections require different constraint
files: technical documentation, API references, specification documents, style guides.

### Alternatives considered

- **Inline constraint summaries** — copy the constraint content into the spec directly. This
  works but duplicates content that evolves independently; the spec becomes stale. The gate
  pattern stays thin and always loads the current version.
- **Single-mode documents** — avoid the problem by keeping all sections in one mode. Not always
  possible for documents like arc42stories that are inherently multi-mode by design.
