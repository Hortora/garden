### Symptom

Documentation generated by a Sonnet subagent lists plausible but completely wrong
variant names for Java sealed interfaces. The code the same model just implemented
is correct.

### Context

SDD (Subagent-Driven Development) workflow where a Sonnet subagent implements code
in one task, then a different Sonnet subagent documents the types in a later task.
The documentation subagent does not read the source files — it generates variant
names from its training data.

### What was tried (didn't work)

Trusting the subagent's documentation output without cross-referencing the actual
source code. The documentation read plausibly — variant names like "Explicit, Role,
Capability" sounded reasonable for an `AgentRef` sealed interface — but were entirely
fabricated.

### Root cause

The subagent lacks the implementation context from the earlier task. When asked to
document sealed type hierarchies, it confabulates variant names that sound reasonable
(e.g., "Explicit, Role, Capability" for AgentRef) rather than reading the actual
permits clause (WorkerAgent, ChannelAgent, HumanAgent, ExternalAgent, ComposedAgent).

### Fix

Always verify generated documentation against actual source code for sealed type
variants. In SDD workflows, include explicit variant names in the documentation
task's brief — don't rely on the subagent to look them up.

### Why this is non-obvious

The implementation code was correct in every case. The hallucination only appeared
in documentation — a separate task with a fresh subagent that had no context from
the implementation task. A reviewer checking "does the code work?" would pass; only
checking "does the documentation match the code?" catches it.
