### Symptom
`yarn install` in CI returns 403 Forbidden when resolving `@casehubio/pages-primitives@^0.2.2`
from `npm.pkg.github.com`. The `GH_PACKAGES_TOKEN` is set (shows `***` in logs). Error looks
identical to an auth failure.

### Context
Cross-repo npm consumption: `casehubio/blocks-ui` consumes packages published by
`casehubio/casehub-pages`. Both repos in the same GitHub org. CI uses `GITHUB_TOKEN`
(repo-scoped) for npm auth.

### What was tried (didn't work)
- Verified token was set and masked in CI logs — confirmed present
- Changed `GH_PACKAGES_TOKEN` from `secrets.GITHUB_TOKEN` to `secrets.GH_PACKAGES_TOKEN || secrets.GITHUB_TOKEN` — still 403
- Assumed it was a cross-org token scope issue

### Root cause
GitHub Packages npm packages default to **private** visibility. A private package cannot be
read by `GITHUB_TOKEN` from another repository, even within the same org. The 403 is not an
auth failure — it's a visibility restriction.

### Fix
Check package visibility:
```bash
gh api '/orgs/<org>/packages?package_type=npm' --jq '.[] | "\(.name): \(.visibility)"'
```

Two options:
1. **Make packages public** via GitHub UI (Settings page per package) — no API exists
2. **Use a classic PAT** with `read:packages` scope, stored as a repo secret

### Why this is non-obvious
The 403 response is indistinguishable from an authentication failure. Nothing in the error
message, response headers, or CI logs indicates visibility is the cause. The GitHub docs
mention visibility settings but don't connect them to the cross-repo 403 symptom. You can
spend hours checking token scopes, registry URLs, and workflow permissions before discovering
the packages are simply private.

### Why this fix
Making packages public is simplest for open-source repos. The PAT approach works for private
repos but adds secret management overhead. There is no API to change visibility — it must be
done per-package in the GitHub UI.

### Alternatives considered
- Using `GITHUB_TOKEN` with elevated permissions — doesn't work; repo-scoped tokens cannot
  read private packages from other repos regardless of permissions
- Organization-level secrets — helps with secret distribution but still needs a PAT value
