## Symptom

A unit-arrival check using `distance >= 0.5` is intermittently correct. Sometimes the unit arrives at its target, sometimes it doesn't register as arrived despite being at the target position. The check works in most tests but fails on boundary cases where the distance is computed to be exactly 0.5.

## Context

In a tick-based movement system with `stepToward(current, target, speed=0.5)`: when `dist <= speed`, the unit snaps exactly to `target` (distance = 0.0). The tick BEFORE snapping, the unit is exactly `speed` tiles away (distance ≈ 0.5). A "not arrived" check of `distance >= 0.5` sits at the exact floating-point boundary — depending on floating-point arithmetic results, the intermediate position can be computed as 0.4999... or 0.5001... making the check non-deterministic.

## Root cause

Floating-point arithmetic near exact-value boundaries. `Math.sqrt(0.354² + 0.354²)` can evaluate to either side of 0.5 depending on rounding in intermediate operations.

## Fix

Use a threshold safely below the snap distance. With `speed = 0.5` tiles/tick:

```java
// Not arrived if distance ≥ 0.1 — safely below 0.5 snap threshold
if (distance(u.position(), target) >= 0.1) return false;
// distance is 0.0 (after snap) or ≥ 0.5 (not yet snapped) — never in-between
```

After snapping, distance = 0.0 (exact). Before snapping, distance > 0.5 (never in (0, 0.5)). Using `< 0.1` cleanly separates these two states.

## Why this is non-obvious

The problematic state (distance exactly = speed) only exists at the tick boundary just before snapping. Any normal distance computation at this state feels like a safe comparison of 0.5 with 0.5 — the floating-point fragility isn't apparent until a test fails intermittently.
