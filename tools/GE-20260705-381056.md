### Symptom
Skills work correctly in production (loaded from `~/.claude/skills/`) but the
source repo has no record of the changes. `git log` and `git diff` in the
source repo show nothing — the work appears to not exist.

### Context
`sync-local` copies skill files from the soredium source repo to `~/.claude/skills/`.
The copy is a deployment artifact. If the source repo has uncommitted changes,
sync-local happily deploys them — but they exist only in the working tree and
the installed copy. No git history, no recovery path.

### What was tried (didn't work)
Nothing — the problem was discovered incidentally when starting a new session.
`git status` in soredium showed ~2264 lines of modified/untracked files that
had been deployed and in daily use for days.

### Root cause
sync-local has no pre-flight check for uncommitted changes. It copies whatever
is in the working tree, committed or not. The developer assumed "if it's
deployed, it's committed" — but sync-local doesn't enforce that invariant.

### Fix
Before running sync-local, verify the source repo is clean:
```bash
git -C ~/claude/hortora/soredium status --short
```
If dirty, commit first. At session start, check for uncommitted soredium
changes and flag them before doing new work.

### Why this is non-obvious
sync-local succeeds silently. The skills load and work correctly. There is no
warning, error, or degraded behaviour. The only signal is `git status` in the
source repo — which you have no reason to check if everything is working.

### Why this fix
A pre-flight check in sync-local itself would be the structural fix, but until
that ships, the manual check at session start catches the gap. The memory
system (`feedback_commit-before-sync.md`) ensures future sessions flag it.

### Alternatives considered
- Adding a git hook to sync-local: would work but requires modifying the tool
- Committing automatically before sync: dangerous — auto-commits bypass review
- Manual discipline only: unreliable, as this incident proved
