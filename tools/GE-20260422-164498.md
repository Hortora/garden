## What It Is

A single Playwright assertion that traverses the entire Three.js scene graph, computes
world positions for all meshes and sprites, and fails if any object is outside the expected
map bounds. Catches positioning bugs (tile coordinate overflow, wrong unit placement, stale
mesh not cleaned up) without needing to look at the UI.

## When To Use

Any time geometry position correctness is an invariant — game maps, data visualisations,
UI layouts with absolute world positions. Particularly valuable when the symptom of a
positioning bug (objects appearing far away or floating) is hard to notice during development
and easy to miss in text-only test output.

## The Technique

```java
@Test
@Tag("browser")
void allSceneObjectsAreWithinMapBounds() throws Exception {
    Page page = openPage();
    // run enough ticks that the AI/game has a chance to produce off-map geometry
    for (int i = 0; i < 10; i++) orchestrator.gameTick();
    engine.observe();
    page.waitForFunction("() => window.__test.buildingCount() >= 1", null,
        new Page.WaitForFunctionOptions().setTimeout(5_000));

    @SuppressWarnings("unchecked")
    List<Map<?,?>> outliers = (List<Map<?,?>>) page.evaluate("""
        () => {
          const out = [];
          const MAX_XZ = 23, MAX_Y = 5;   // map spans ±22.4 world units
          window._three.scene.traverse(obj => {
            if (!obj.isMesh && !obj.isSprite) return;
            const p = obj.getWorldPosition(new THREE.Vector3());
            if (Math.abs(p.x) > MAX_XZ || Math.abs(p.z) > MAX_XZ || p.y > MAX_Y) {
              out.push({
                type: obj.type,
                x: p.x.toFixed(1), y: p.y.toFixed(1), z: p.z.toFixed(1),
                color: obj.material?.color?.getHexString() ?? 'none'
              });
            }
          });
          return out;
        }
    """);

    assertThat(outliers)
        .as("Scene objects outside map bounds — indicates tile position overflow " +
            "or stale mesh not cleaned up: " + outliers)
        .isEmpty();
}
```

The JS side uses `scene.traverse()` with `obj.getWorldPosition()` (which resolves the full
parent-chain transform, unlike `obj.position` which is local-only). The Java side gets back
a plain list of serialisable maps that AssertJ can format in the failure message.

## Why Non-Obvious

The obvious approach for testing 3D object positions is either pixel-level screenshot
comparison (fragile, camera-dependent) or manually checking specific known objects by tag.
`scene.traverse()` is a Three.js internal utility rarely used in tests; `getWorldPosition()`
correctly resolves parent transforms where `position` does not. The combination gives a
single, stable, camera-independent assertion over the entire scene.

In the session that surfaced this technique, a Pylon position overflow (tile y=1752 → world
z=1204) was producing purple rectangles floating 1200 world units off-map. The visual
debugging took hours; this test would have caught it in milliseconds during the first run
after the bug was introduced.
