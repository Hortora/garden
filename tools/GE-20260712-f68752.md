### Symptom
After inserting a new field or method via `ide_insert_member` that references a type not
yet imported (e.g. `new ArrayList<>()`), calling `ide_optimize_imports` or
`ide_reformat_code` does not add the missing `java.util.ArrayList` import. The file
continues to fail compilation with "cannot find symbol".

### Context
Occurs when a new member is inserted via the MCP plugin that introduces a reference to a
type not already in the file's import list. Even opening the file in the editor first
(`ide_open_file`) and then running `ide_optimize_imports` does not add the missing import.

### What was tried (didn't work)
1. `ide_optimize_imports` — only organizes existing imports, does not add missing ones
2. `ide_reformat_code` with `optimizeImports=true` — same behaviour, only removes unused
3. `ide_open_file` then `ide_optimize_imports` — opening the file first does not help
4. `ide_sync_files` then `ide_optimize_imports` — syncing first does not help

### Root cause
IntelliJ's "Optimize Imports" action (which the MCP tool wraps) only removes unused
imports and re-sorts existing ones. Adding missing imports is a separate action
("Auto Import" / "Add Import") that requires the IDE's on-the-fly code analysis to
detect unresolved references and offer quick-fixes. The MCP plugin does not expose
this action.

### Solution 1 — Explicit import addition via ide_replace_text_in_file (preferred)
**Approach:** After `ide_insert_member` introduces a new type, add the import explicitly.
**Pros:** Clean code from the start, no FQN pollution, works whether or not the file is open.
**Cons:** Requires knowing which import line to target as an anchor.

```
ide_replace_text_in_file({
  file: "path/to/File.java",
  searchText: "import java.util.Map;",
  replaceText: "import java.time.Instant;\nimport java.util.Map;"
})
```

Pattern: find an existing import in the same package group, replace it with the new
import + the original. Reliable and idempotent.

### Solution 2 — Fully-qualified inline references
**Approach:** Use FQNs in the inserted code, then let optimize imports shorten them.
**Pros:** No separate import step needed.
**Cons:** Only shortens FQNs if the file is open in the editor. Produces ugly intermediate code.

```java
private final List<InfraNodeSpec> receivedSpecs = new java.util.ArrayList<>();
```
The `ide_optimize_imports` action will then shorten the FQN to an import + simple name
on subsequent calls — but only if the file is open in the editor.

## Variant — SNAPSHOT jars cause active import removal

In multi-module Maven projects with SNAPSHOT dependencies, `ide_optimize_imports`
can actively *remove* valid existing imports — not just fail to add missing ones.
When types come from a SNAPSHOT-versioned sibling module (e.g. `BatchBranchOutcome`
from `devtown-domain` used in `devtown-app`), the IDE index may not fully resolve
the type. `ide_optimize_imports` treats the unresolved import as "unused" and
silently deletes it. The compilation error says `package BatchBranchOutcome does
not exist` — and the import that was present before the optimize call is gone.

**Mitigation:** Run `ide_sync_files` on the sibling module's source directory before
calling `ide_optimize_imports`. If that doesn't help, manage imports explicitly via
`ide_replace_text_in_file` and avoid `ide_optimize_imports` entirely on files that
reference types from SNAPSHOT siblings.

### Why this is non-obvious
"Optimize imports" sounds like it should handle all import concerns. A developer would
reasonably expect that adding a field referencing `ArrayList` and then running "optimize
imports" would add the `java.util.ArrayList` import — that is exactly what the IDE does
interactively via its quick-fix system. The MCP tool only wraps the narrower "organize
existing imports" action.
