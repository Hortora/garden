## Context
Multi-layer tutorial series where each layer is gated on milestones in an external
dependency repo. Gate labels are recorded as free text in the docs
(e.g. `blocked: engine P1.3`).

## Symptom
Layer 5 (adaptive paths) was marked `blocked: engine P1.3`. Work was deferred
because P1.3 appeared incomplete. P0 — which actually gates Layer 5 — was already
closed.

## Root cause
P1.3 gates trust-weighted routing, a Layer 6 concern. The label was set when the
milestone-to-layer mapping wasn't fully traced, and no one verified it against the
actual milestone content.

## Fix
Before treating a layer as blocked, read the dependency repo's milestone issues
directly and confirm which specific capability they gate — not just which feature
area they relate to.

## Why this is non-obvious
Milestone labels sound related to an entire feature area (P1.3 = "trust
infrastructure"). Without reading the issues, it's not obvious that P1.3 gates
trust-weighted *routing* (Layer 6) rather than the adaptive *path execution*
(Layer 5) that sounds equally trust-adjacent.
