## Symptom

At epic close, the journal merge step reports "(skipped — nothing to do)" even though `design/JOURNAL.md` contains several paragraphs of design narrative. No warning, no prompt — the step silently passes. The DESIGN.md is left unchanged.

## Context

The epic skill's journal merge (Step B3/B7) only processes journal entries that have `§Section` anchor headers:

```markdown
### 2026-05-15 · §ComponentStructure

Prose describing the design change...
```

If `design/JOURNAL.md` contains plain prose — written directly, or via `write-blog/handover` rather than `java-update-design` during commits — none of it is parsed as merge candidates. The extraction step finds zero `§Section` lines and marks the whole journal merge as skipped.

## Root cause

The `§Section` anchor format (`### YYYY-MM-DD · §SectionName`) is the merge map: it tells the epic close tool which DESIGN.md section to update and with what narrative. Without it, the tool cannot know where prose belongs in DESIGN.md.

`java-update-design` generates these anchors automatically when called during `java-git-commit`. But if commits go through standalone agents, manual commits, or `finishing-a-development-branch` (which doesn't invoke `java-git-commit`), the anchors are never written.

## Fix

Before running epic close, verify anchors exist:
```bash
grep "§" design/JOURNAL.md
```

If the output is empty, the journal merge will be skipped. Either:
1. Manually add `§Section` anchors to existing journal entries before running close, or
2. Run `java-update-design` manually for each changed section — it writes anchored entries to JOURNAL.md

## Why this is non-obvious

The journal file looks populated and the word "JOURNAL.md" appears in the epic close plan. There is no indication that the content won't be processed until the merge step runs and silently skips. The §Section anchor format is only documented inside `java-update-design`'s skill file — nothing in the epic workflow warns that plain prose is ineligible for merge.
