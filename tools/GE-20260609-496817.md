## Symptom

A Java string constant visually displays as `"DHMETA:"` in IDE and in the Claude Code Read tool
output, but sentinel matching code silently fails to recognise it in channel messages. Debugging
with string equality checks confirms the constant and the runtime value are "identical" by visual
inspection, yet `content.startsWith(META_SENTINEL)` returns false.

## Context

A string constant contains a non-printable SOH byte (U+0001) as a prefix for collision resistance:

```java
public static final String META_SENTINEL = "DHMETA:";
```

When this file is read via the Claude Code `Read` tool, the output shows `"DHMETA:"` — 7 characters.
The SOH byte is silently swallowed by the display layer. This creates a discrepancy between what
you see and what the code contains.

## Root Cause

Claude Code's Read tool (and most text editors) strip or ignore SOH (ASCII 0x01), a control
character, when displaying file content. The byte exists in the file and in the runtime string,
but is invisible in all text-based views.

## Fix

When debugging sentinel matching failures or verifying a sentinel constant, use binary inspection
rather than the Read tool:

```python
with open('/path/to/DebateProtocol.java', 'rb') as f:
    content = f.read()
idx = content.find(b'META_SENTINEL =')
print(repr(content[idx:idx+60]))
# b'META_SENTINEL = "\\x01DHMETA:";\n'  ← SOH byte visible as \\x01
```

Or from the shell:

```bash
python3 -c "
with open('DebateProtocol.java', 'rb') as f:
    print(repr(f.read(500)))
" | grep -A1 META_SENTINEL
```

## Why this is non-obvious

The SOH byte is invisible to every text display tool. There is no visual indication that a
non-printable byte is present. The Read tool shows a 7-character string; the actual constant is
8 characters. The mismatch is only discoverable via binary inspection.

## Constraints

Applies to any non-printable ASCII control character (0x00–0x1F) used as sentinels. SOH is a
common choice for sentinel prefixes specifically because it is invisible to LLMs, which is the
design intent — but that same invisibility makes debugging harder for humans.
