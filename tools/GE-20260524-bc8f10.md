### Symptom

Ran `git commit --amend -m "new message"` on an empty commit (originally created with `--allow-empty`). Command printed `No changes` and exited 0. Appeared successful — but `git log -1 --format="%s"` showed the original message unchanged across all 15 branches that had been stamped.

### Context

Stamping closed branches with a closure marker: `git commit --allow-empty -m "chore: branch closed — squash-merged to main"`, then later amending all 15 to fix the message with `git commit --amend -m "chore: branch closed"`. The amend loop iterated silently, each reporting `No changes` and `✓ <branch>` — all appearing to succeed.

### What was tried (didn't work)

`git commit --amend -m "new message"` on an empty commit — exits 0, prints `No changes`, makes no change.

### Root cause

`git commit --amend` without `--allow-empty` refuses to amend an empty commit. Git sees the result would be an empty commit, treats it as "nothing to commit", and exits silently with success (exit code 0). There is no warning or error.

### Fix

```bash
git commit --allow-empty --amend -m "new message"
```

The `--allow-empty` flag is required on both the original creation *and* any subsequent amend.

### Why this is non-obvious

Exit code 0 and the `No changes` message are indistinguishable from a successful no-op on a non-empty commit with no staged changes. Nothing signals that the amend was skipped — the command succeeds in the shell sense while failing to do anything.
