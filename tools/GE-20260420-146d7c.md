## Symptom

Running `sed -i '' '/<artifactId>foo<\/artifactId>/,/<\/dependency>/d'` to remove a Maven dependency block from `pom.xml` produces a corrupted file — the next `<dependency>` block loses its opening `<dependency>` tag, leaving orphaned child elements.

## Context

Occurs when removing a multi-line XML element from a pom.xml using sed range deletion. The end pattern (`</dependency>`) appears multiple times in the file. sed's range matching stops at the **first** occurrence of the end pattern after the start match — which may be in the middle of the target block or the very next unrelated element's closing tag.

## Root Cause

sed's address range `/start/,/end/` is line-based and greedy from left to right — it deletes from the first line matching `start` to the first subsequent line matching `end`. In a Maven pom.xml, `</dependency>` appears after every dependency. If the target block spans multiple lines and the actual closing tag isn't the immediately next `</dependency>`, the range can terminate early (deleting too little) or in some arrangements delete into the next block (deleting too much).

## Fix

Use Python or a purpose-built XML tool instead of sed for structural XML modifications:

```bash
# Python one-liner: remove the block containing a specific artifactId
python3 -c "
import re, sys
content = open('pom.xml').read()
# Remove the full <dependency>...</dependency> block containing the artifactId
content = re.sub(
    r'\s*<dependency>\s*(?:<[^/][^>]*>[^<]*</[^>]+>\s*)*<artifactId>foo</artifactId>(?:\s*<[^/][^>]*>[^<]*</[^>]+>)*\s*</dependency>',
    '', content, flags=re.DOTALL)
open('pom.xml', 'w').write(content)
"
```

Or simply edit the file directly with an editor or the `Edit` tool — any structural XML change is safer via direct edit than sed range deletion.

## Why This Is Non-Obvious

sed range deletion looks straightforward for line-based patterns. The failure mode is non-obvious because the file after deletion is syntactically valid XML (no parse error) — Maven just fails to read the pom with a cryptic "Non-parseable POM: Unrecognised tag" pointing at a different location entirely. The corruption isn't where the deleted block was.
