### Symptom

GitHub Pages deployment workflow fails immediately with:

```
Branch "main" is not allowed to deploy to github-pages due to environment protection rules.
The deployment was rejected or didn't satisfy other protection rules.
```

The workflow has been deploying from `main` for some time but suddenly stops working. No one changed the workflow file. The error is the same whether triggered by push or `workflow_dispatch`.

### Context

The repository had a GitHub Pages environment with `custom_branch_policies: true`. A previous Claude session (or manual config change) added a specific branch entry (e.g. `claude/some-feature-branch`) to the allowed deployers list. `main` was never explicitly added as an allowed branch.

When custom branch policies are enabled, GitHub Pages defaults to deny-all — only branches explicitly listed are permitted. `main` is not automatically included.

### What was tried

- Re-running the failed workflow — same error
- Checking the workflow file for branch restrictions — none present
- Checking repository Settings → Branches → Branch protection rules — nothing relevant

### Root cause

GitHub Pages environment protection has two modes:
1. **Protected branches only** — all branches with protection rules can deploy
2. **Custom branch policies** — only explicitly listed branches can deploy (deny-all by default)

When mode 2 is active, deleting or not adding `main` means it cannot deploy even if it previously could. The stale branch entry (`claude/feature-branch`) was the only allowed deployer. `main` was absent from the list.

The error message does not indicate which branches are currently allowed, making the cause invisible without checking the environment settings directly.

### Fix

Add `main` to the deployment branch policies via the GitHub API:

```bash
# Add main to allowed deployers
gh api --method POST \
  "/repos/OWNER/REPO/environments/github-pages/deployment-branch-policies" \
  --field name="main"

# Check current policies
gh api "/repos/OWNER/REPO/environments/github-pages/deployment-branch-policies" \
  | jq '.branch_policies[] | .name'

# Remove stale entries by ID
gh api --method DELETE \
  "/repos/OWNER/REPO/environments/github-pages/deployment-branch-policies/POLICY_ID"
```

Or via Settings → Environments → github-pages → Deployment branches → Add deployment branch rule.

### Why this is non-obvious

The error message names the branch restriction but gives no indication of what the allowed list contains or how to fix it. The environment protection settings are several levels deep in GitHub UI (Settings → Environments → github-pages). The stale allowed branch is invisible without explicitly querying the API. This commonly appears after an automated session adds a feature branch to the allowed list and then the branch is deleted — `main` was never explicitly allowed to begin with.
