### Symptom

A CI run triggered by `repository_dispatch` shows a `headSha` you cannot find locally. `git cat-file -t <sha>` returns "not found" even after `git fetch origin`. The SHA does not appear in `git log --all`.

### Context

Project has two remotes: `origin` (personal fork at `mdproctor/<repo>`) and `upstream` (canonical org repo at `casehubio/<repo>`). You push commits to `origin` but not `upstream`. A `repository_dispatch` event (e.g. "upstream dependency published") triggers a CI run on the org repo. The `headSha` in `gh run view` is the HEAD of the org repo's `main` at trigger time, not the fork's HEAD.

### Root cause

GitHub Actions `repository_dispatch` events fire against the **default branch of the repository that received the dispatch** — the org repo. The `headSha` it records is the current HEAD of that repo's default branch, which may have commits you have never fetched (they were pushed by collaborators or automation to the org repo, not to your fork). Your fork's `origin/main` is independent and can be behind, ahead, or diverged.

### Fix

Fetch from `upstream` (the org repo) to find the SHA:

```bash
git fetch upstream
git log upstream/main | head -5   # headSha will appear here
```

To avoid this confusion permanently, always push directly to `upstream` for changes that should trigger org-repo CI:

```bash
git push upstream main   # not: git push origin main
```

Also verify which remote `gh run list` is querying — pass `--repo casehubio/<repo>` explicitly if there is any ambiguity.

### Why this is non-obvious

When you have a fork setup, `git fetch origin` is instinctive. The SHA "not found" error after a fetch looks like corruption or a race condition. Nothing in the GitHub Actions UI flags that the `headSha` belongs to the org repo rather than your fork. The `repository_dispatch` trigger is typically used for inter-repo automation (not personal pushes), so the headSha source being the org repo rather than the actor's fork is easy to miss.
