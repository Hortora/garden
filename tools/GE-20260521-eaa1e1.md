### Symptom

Running `git checkout main && git rebase $BRANCH` to "apply branch work onto
main" produces unexpected results — main ends up at the branch tip with
diverged history replayed on top, not branch commits cleanly applied to main.

### Root cause

`git rebase $BRANCH` while on main means: "replay the commits on main that are
NOT on $BRANCH, on top of $BRANCH's tip." If main has no diverged commits,
this is a fast-forward. If main has diverged commits, those commits are
replayed on top of the branch — which is the opposite of the intended direction.

### Fix

The correct pattern to get branch work onto main cleanly:

```bash
# Step 1: rebase the branch onto current main
git checkout $BRANCH
git rebase main       # replays branch commits on top of main tip

# Step 2: fast-forward main to the rebased branch
git checkout main
git merge --ff-only $BRANCH
```

This ensures branch commits land on main in linear history with no merge commit.

### Why this is non-obvious

The mental model "rebase X onto Y means X's commits go on top of Y" leads
directly to `git checkout main && git rebase $BRANCH` — which does the
reverse. The command reads as "rebase using $BRANCH as base" but git's
direction is: the checked-out branch's unique commits are replayed onto the
argument. The correct approach requires temporarily being on the branch, not
main, to perform the rebase.
