## Symptom

Objects placed at game-tile z positions like z=-2, z=-4 either do not appear in the browser or trigger a bounds-check assertion:

```
AssertionError: no showcase objects outside map bounds
[{type=Sprite, x=-18.9, y=0.4, z=-25.9}, ...]
```

No console error is raised by Three.js itself — the sprites are added to the scene and render at the mapped world position.

## Context

Occurs in a tile-based Three.js visualizer where game tile coordinates are mapped to world coordinates with:

```javascript
function gw(gx, gz) {
  return { x: gx * TILE - HALF_W, z: gz * TILE - HALF_H };
}
```

Where `HALF_W = HALF_H = (64 * 0.7) / 2 = 22.4` for a 64×64 grid with `TILE = 0.7`.

The map has valid world bounds of ±23 units. Tile coordinates that appear "near zero" feel safe, but the world-space formula means:

| Game tile z | World z | Within ±23? |
|-------------|---------|-------------|
| 22          | -7.0    | ✓           |
| 0           | -22.4   | ✓ (barely)  |
| -1          | -23.1   | ✗           |
| -2          | -23.8   | ✗           |
| -10         | -29.4   | ✗           |

The coordinate origin is at tile (32, 32), not tile (0, 0). Any negative tile z coordinate lands outside the valid world range.

## What Was Tried

Placed additional showcase building rows at game tile z = -2, -4, -6, -10, -12 expecting them to appear "below" the unit grid. The test `allSceneObjectsAreWithinMapBounds` caught the overflow — but the coordinates reported (worldZ = -25.9, -29.4 etc.) looked like large negatives, not an off-by-one.

## Root Cause

The tile origin is at the centre of the 64×64 grid (tile 32, 32 = world 0, 0). Negative tile z values map to world z < -22.4, which exceeds the ±23 bounds. The constraint is invisible during scene construction — Three.js will happily place a sprite at worldZ = -35 without warning.

## Fix

Use only positive tile z values ≥ 0. For adding rows "south" of existing content (tile z = 2..20), continue with larger positive z values (z = 22, 24, 26, …) which map to world z approaching +23:

```
tile z = 34 → world z = (34 × 0.7) − 22.4 = +1.4   ✓
tile z = 63 → world z = (63 × 0.7) − 22.4 = +21.7   ✓ (near limit)
```

## Why This Is Non-Obvious

The tile coordinate system looks like a standard 2D grid where negative z "goes the other way". The actual constraint (origin at tile 32,32; valid range approx 0–63) is nowhere explicit in the visualizer code — it only becomes apparent by deriving the HALF_W/HALF_H constants from the grid dimensions. The test failure reports world coordinates, not tile coordinates, so the reported values (-25.9) don't immediately connect back to the tile positions (-2, -4) that caused them.
