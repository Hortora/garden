## Technique

When writing a publish/deploy step in GitHub Actions that should run on pushes to main but NOT on pull requests, use the negative guard:

```yaml
- name: Publish to GitHub Packages
  if: github.event_name != 'pull_request'
  run: mvn --batch-mode deploy -DskipTests
```

## Why this is non-obvious

The obvious approach is an allowlist:

```yaml
if: github.event_name == 'push' && github.ref == 'refs/heads/main'
```

But this silently skips publishing when the workflow is triggered via `workflow_dispatch` or `repository_dispatch` — which are common for manual re-runs and CI chaining. The allowlist requires explicit enumeration of every trigger type you want to publish on.

The negative guard (`!= 'pull_request'`) publishes on all trigger types except PRs — automatically covering `push`, `workflow_dispatch`, and `repository_dispatch` without needing to maintain an allowlist. PRs are the only case where you genuinely don't want to publish (to avoid polluting the package registry with unreviewed code).

## When to use

Whenever you have a workflow with multiple trigger types (`push`, `workflow_dispatch`, `repository_dispatch`) and need publishing to work from all of them. The ref guard (`github.ref == 'refs/heads/main'`) is still useful if you have branch workflows and only want main to publish.
