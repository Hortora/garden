## Symptom

A script uses `A && B || C` as an if/else pattern. When A is false, C executes unexpectedly — even though it was intended as an error handler for B.

```bash
[ "$STASH_REF" != "none" ] && \
  git stash pop "$STASH_REF" \
  || echo "⚠️ Stash pop failed — resolve manually"
# Intended: warn only if pop fails
# Actual: also warns when STASH_REF == "none"
```

## Root cause

Shell operator precedence evaluates `A && B || C` as `(A && B) || C`. When A is false, `A && B` is false, so the shell evaluates C — the "error handler" — even though B never ran. C is not an else branch; it fires whenever `A && B` evaluates to false, which includes both "B failed" and "A was false in the first place."

## Fix

Use explicit `if/fi`:

```bash
if [ "$STASH_REF" != "none" ]; then
  if ! git stash pop "$STASH_REF"; then
    echo "⚠️ Stash pop failed — resolve manually"
  fi
fi
```

Or, if the single-line form is genuinely needed, add a redundant guard inside C:

```bash
[ "$STASH_REF" != "none" ] && git stash pop "$STASH_REF" \
  || { [ "$STASH_REF" != "none" ] && echo "⚠️ Stash pop failed"; }
```

## Why this is non-obvious

The `A && B || C` pattern is frequently described as "shell if/else" in tutorials. It works correctly when A is always true — the only way C fires is if B fails. The failure mode is subtle and only appears when A can be false.
