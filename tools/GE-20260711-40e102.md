### Symptom

Adding a new parameter to a heavily-used method (e.g. `sendMessage` gaining a `topic` parameter) requires updating 700+ call sites across 40+ test files. A naive find-and-replace or comma-counting script produces syntax errors.

### Context

When a record or method gains a new field/parameter at the end, all existing callers need the additional argument (typically `null` for the new optional field). The call sites vary:
- Single-line calls: `tools.sendMessage("ch", "a", "cmd", "content", null, null, null, null, null, null, null);`
- Multi-line calls spanning 2-4 lines
- Calls wrapped in `assertThrows(() -> tools.sendMessage(...))`
- Calls containing nested expressions like `List.of(a, b)` whose inner commas must not be counted

### What was tried (didn't work)

1. **Simple comma counting** — counts commas in the entire line including nested `List.of(a, b)` expressions, giving incorrect arg counts (12 commas instead of 10)
2. **Line-ending replacement** — replacing `, null);` with `, null, null);` works for single-line calls but corrupts multi-line calls and `assertThrows`-wrapped calls where the line ends with `));` not `);`
3. **`endswith(");")` matching** — misses calls ending with `));` (lambda wrappers) and multi-line calls where the closing `)` is on a different line

### Fix

A Python script that finds each call by string search, then uses **paren-depth tracking** to count only top-level commas and find the correct closing paren:

```python
def count_top_level_commas(call_text):
    depth = 0
    commas = 0
    for ch in call_text:
        if ch == '(': depth += 1
        elif ch == ')': depth -= 1
        elif ch == ',' and depth == 1: commas += 1
    return commas

# For each file:
i = 0
while i < len(content):
    idx = content.find("tools.sendMessage(", i)
    if idx == -1: break
    # Find matching closing paren
    paren_start = content.index("(", idx + len("tools.sendMessage"))
    depth = 1
    j = paren_start + 1
    while j < len(content) and depth > 0:
        if content[j] == '(': depth += 1
        elif content[j] == ')': depth -= 1
        j += 1
    call = content[idx:j]
    if count_top_level_commas(call) == OLD_ARG_COUNT - 1:
        close_idx = call.rfind(")")
        call = call[:close_idx] + ", null" + call[close_idx:]
    # ... write back
```

This correctly handles single-line, multi-line, nested-expression, and lambda-wrapped calls.

**Execution order matters:** Run the single-line script first, then a second pass for multi-line calls and any remaining edge cases (lambda wrappers like `assertThrows`/`assertDoesNotThrow`).

### Why this is non-obvious

The instinct is to use sed, IDE refactoring, or a simple grep-and-replace. But Java method calls with nested expressions, lambda wrappers, and multi-line formatting create a combinatorial explosion that text-based tools can't handle reliably. The paren-depth approach is essentially a minimal parser — not complex, but not what most developers reach for when facing a "just add null to each call" task.

### Alternatives considered

- **IntelliJ Change Signature refactoring** — works within a single module but the qhorus project has test files across 6+ modules that compile separately. Change Signature doesn't reach cross-module test code that imports via Maven dependency.
- **sed/awk one-liners** — too fragile for nested expressions and multi-line calls.
