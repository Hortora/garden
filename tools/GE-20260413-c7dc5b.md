## `python-build-standalone` Windows archive is `.tar.gz` not `.zip` — and has no shared/static discriminator

**ID:** GE-20260413-c7dc5b
**Stack:** python-build-standalone (astral-sh), GitHub Actions Windows runners
**Symptom:** HTTP 404 when downloading the Windows Python archive. The filename constructed matches the macOS/Linux pattern but doesn't exist for Windows.
**Context:** Bundling a self-contained CPython runtime into an Electron app using `python-build-standalone`. Code that correctly downloads `.tar.gz` on Mac/Linux needs adjustment for Windows.

### What was tried (didn't work)
- Used `.zip` extension (Windows convention) — 404
- Added `-shared` discriminator (e.g. `x86_64-pc-windows-msvc-shared-install_only_stripped.zip`) — 404
- Added `-static` discriminator — 404

### Root cause
Windows `python-build-standalone` releases use `.tar.gz` format (not `.zip`), and the filename has no `shared`/`static` discriminator — unlike some older releases. The correct filename pattern is:
```
cpython-{VERSION}+{TAG}-x86_64-pc-windows-msvc-install_only_stripped.tar.gz
```
`tar` is available on Windows GitHub Actions runners (via Git for Windows), so extraction works identically to Mac/Linux with no PowerShell fallback needed.

### Fix
```javascript
const PLATFORM_MAP = {
  'win32-x64': { dir: 'win-x64', arch: 'x86_64-pc-windows-msvc', ext: 'tar.gz' },
  // macOS and Linux use ext: 'tar.gz' too — extraction is uniform
};
// Extract on all platforms:
execSync(`tar -xzf "${archivePath}" -C "${destDir}" --strip-components=1`);
```

### Why this is non-obvious
macOS and Linux also use `.tar.gz` so the extension looks consistent — but Windows is known for `.zip`. The `shared`/`static` discriminator appeared in some historical releases, making it natural to try. The 404 gives no hint about format; you have to inspect the actual GitHub release assets to discover the correct filename.
