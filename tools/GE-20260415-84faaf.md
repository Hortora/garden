## Git branch naming conflict: can't create a branch whose name is a prefix of an existing branch path

### Stack
Git (any version)

### Symptom
Creating a branch with a `/`-separated path fails with a cryptic error:

```
fatal: cannot lock ref 'refs/heads/feat/foo/bar':
'refs/heads/feat/foo' exists; cannot create 'refs/heads/feat/foo/bar'
```

### Context
Working with `/`-scoped branch naming conventions (e.g. `feat/module/feature`).
An existing branch `feat/foo` blocks creation of `feat/foo/bar` because Git stores
branch refs as files in a directory tree — `feat/foo` is stored as a *file* at
`refs/heads/feat/foo`, so `refs/heads/feat/foo/` cannot also exist as a *directory*.

### Root cause
Git uses the filesystem path structure for ref storage. A branch name containing `/`
creates a directory hierarchy. If `feat/foo` exists as a file at
`refs/heads/feat/foo`, no branch whose name begins with `feat/foo/` can be created —
you'd need a directory where the file sits.

### Fix
Rename the blocking branch before creating the sub-path branch:

```bash
# Rename the existing branch to free up the path
git branch -m feat/foo feat/foo/base

# Now the sub-path branch can be created
git checkout -b feat/foo/feature
```

Or design your naming convention to avoid the conflict upfront:
- `feat/foo-base` and `feat/foo-feature` (hyphen, not slash)
- `feat/foo/v1` and `feat/foo/v2` (consistent depth from the start)

### Why this is non-obvious
The error message references the internal `.git/refs/heads/` path structure, which most
developers never think about. The conflict between "branch as file" and "branch as
directory prefix" is a Git implementation detail, not a documented constraint on branch
names. The fix (renaming the existing branch) is not suggested by the error message.
