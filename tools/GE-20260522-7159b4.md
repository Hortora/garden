### Symptom

A pre-push hook that greps `git log --oneline` output for commit message patterns — e.g. `^docs(claude):` or `^chore:` — never fires. Every push passes the check silently, regardless of what was committed.

### Context

Hook scripts that scan commit subjects for squash candidates, release-note requirements, or policy violations commonly use:

```bash
git log --oneline "$range" | grep -E "^docs\(claude\):|^chore:"
```

The hook appears to work — no errors, no warnings. It simply never blocks anything.

### Root cause

`git log --oneline` prepends an abbreviated SHA and a space before the commit subject:

```
3adbd62 docs: WorkItemCreateRequest.toString() log-safety comment
```

The `^` anchor in the grep pattern matches the **start of the line** — which is the SHA, not the subject. `^docs(claude):` will never match because no line starts with `docs(claude):`. Every `grep -cE "$SQUASH_PATTERNS"` returns 0.

The hook compiled and ran correctly on every push. It just never caught anything.

### Fix

Use `--format=%s` to output only the commit subject, with no SHA prefix:

```bash
# Before (broken)
git log --oneline "$range" | grep -cE "$SQUASH_PATTERNS"

# After (correct)
git log --format=%s "$range" | grep -cE "$SQUASH_PATTERNS"
```

Apply the same fix to any `wc -l` count or `grep` that operates on commit subjects from `git log --oneline`.

### Why this is non-obvious

`git log --oneline` looks like the natural way to get "just the commit line." The output is human-readable and the subject is right there — but the SHA prefix is invisible when scanning by eye. The hook exit code is always 0 (grep returning 0 matches is not an error), so no failure signal surfaces. The bug can persist indefinitely in a live hook.
