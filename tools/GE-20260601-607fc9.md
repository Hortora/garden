## Symptom

`git push` blocked with "Pre-push: N commit(s) about to push — Run /git-squash to review history before pushing, or use --no-verify to skip." Even a single clean `ci:` commit with no squash candidates gets blocked.

## Context

A pre-push hook designed to force AI-assisted squash review before every push. The hook exits 1 regardless of whether the commits contain squash candidates. This is intentionally different from hooks that only fire when they detect noise commits.

## Root cause

The hook is designed as a hard gate, not a smart detector. Its comment says: "Pattern-based candidate detection is insufficient — run full AI analysis every time." Every push attempt triggers the block until the developer explicitly bypasses with `--no-verify`.

```bash
# The hook's logic — simplified:
total=$(git log --oneline "$range" | wc -l)
[ "$total" -eq 0 ] && continue
echo "Pre-push: $total commit(s) about to push..."
exit 1   # ← always, regardless of commit content
```

## Fix

After completing a `/git-squash` review (even if the conclusion is "all clean, no squash needed"), push with:

```bash
git push --no-verify
```

This is the intended workflow: hook fires → AI review runs → if clean or squashed → push with `--no-verify`.

## Why this is non-obvious

Most pre-push hooks analyse commit content and only block when they find squash candidates (fixup commits, revert chains, wording-only docs). A developer familiar with standard hooks expects a single clean commit to pass silently. Finding that a one-line CI fix is blocked by the same hook that would block a 40-commit feature branch is unexpected — and the error message doesn't explain that this is by design.
