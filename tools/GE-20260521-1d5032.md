### Symptom

`git rebase -i` aborts with:
```
error: 'pick' does not take a merge commit. If you wanted to replay the merge,
use 'merge -C' on the commit.
error: invalid line N: pick <sha> Merge pull request #1 from ...
```

### Context

Writing a non-interactive rebase todo file programmatically (e.g. via
`GIT_SEQUENCE_EDITOR="cp /tmp/todo.txt"`) that includes a merge commit as a plain `pick`.

### Root cause

`pick` only replays single-parent commits. Merge commits have two parents and require
special handling to preserve the merge structure.

### Fix

Two options depending on intent:

**Option 1 — Replay the merge (preserves branch topology):**
```
merge -C <sha> <sha>   # re-use the original merge commit message
merge -c <sha> <sha>   # same but opens editor to amend the message
```

**Option 2 — Drop the merge commit (common in squash sessions where pre-merge work is already squashed into one commit):**
```
drop <sha> Merge pull request #1 ...
```

When squashing the commits that *were* merged into the merge commit, the merge commit
itself becomes redundant — use `drop`.

### Why this is non-obvious

When building a rebase todo programmatically, merge commits are indistinguishable from
regular commits by subject alone. The error only surfaces at execution time, not during
todo construction. The fix (`merge -C` vs `drop`) depends on intent, which requires
understanding whether the squash absorbed the pre-merge commits.
