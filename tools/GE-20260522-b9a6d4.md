### Symptom

After creating a GitHub fork and adding it as a remote, `git push --force-with-lease`
to the fork remote is rejected even though the push to the upstream succeeds:

```
To https://github.com/upstream/repo.git
 + abc1234...def5678 main -> main (forced update)    ← succeeds

To https://github.com/fork/repo.git
 ! [rejected]        main -> main (stale info)        ← fails
error: failed to push some refs to 'https://github.com/fork/repo.git'
```

### Context

Occurs on the first push to a freshly-created fork where:
1. The fork was created via `gh repo fork --clone=false` (no local fetch)
2. The remote was added via `git remote add <fork> <url>`
3. A force-push (e.g. post-squash) is attempted without fetching from the fork first

### Root cause

`--force-with-lease` protects against overwriting upstream commits you haven't seen.
It does this by comparing the remote-tracking ref it holds locally against the actual
remote HEAD. When the fork is created without fetching, git has no remote-tracking ref
for the fork remote — or has one pointing to the pre-squash SHA. The lease check
considers this stale and rejects the push even though there is nothing to protect.

### Fix

Use `--force` (not `--force-with-lease`) for the first push to a newly-created fork:

```bash
git push --force <fork-remote> main
```

After this push establishes the remote-tracking ref, all subsequent pushes can use
`--force-with-lease` safely.

Alternatively, fetch first to establish the ref before pushing:

```bash
git fetch <fork-remote>
git push --force-with-lease <fork-remote> main
```

### Why this is non-obvious

`--force-with-lease` is the recommended safe alternative to `--force` — developers
reach for it automatically. The rejection message "stale info" doesn't distinguish
between "you're about to overwrite someone else's commits" (the intended warning) and
"you've never fetched this remote" (the actual situation). On a brand-new fork there
is nothing to protect, making the rejection purely misleading.
