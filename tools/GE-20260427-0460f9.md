## Symptom

ASCII art diagrams in Markdown code blocks look perfectly aligned in a local editor but have ragged right edges on GitHub. Box corners and vertical bars don't line up:

```
╔══════════════════════════════╗   ← looks aligned locally
║  LAYER 4 — SOCIAL COMMITMENT ║   ← right edge shifts on GitHub
╚══════════════════════════════╝
```

## Root cause

Unicode box-drawing characters (`╔`, `═`, `║`, `—`, `►`, `▼`) are "ambiguous width" in Unicode — they display as 1 column in Western monospace fonts but 2 columns in East Asian fonts. GitHub's code block font renders some of these at fractional or different widths than regular ASCII. The result: lines with em dashes or special arrows are a different display width than lines without, breaking alignment.

Additionally, `awk` counts bytes, not display columns. UTF-8 box-drawing chars are 3 bytes each, so `awk '{print length}'` returns inflated counts that don't reflect actual display width.

## Fix

Use pure ASCII in code blocks. Every character is exactly 1 byte and 1 display column in any monospace font:

| Unicode | ASCII replacement |
|---|---|
| `═`, `─` | `-` |
| `║` | `\|` |
| `╔`, `╗`, `╚`, `╝`, `┌`, `┐`, `└`, `┘` | `+` |
| `—` (em dash) | `--` |
| `►` | `>` |
| `▼` | `v` or `\|` |

Use Python to verify alignment — `len(line.rstrip())` counts characters, not bytes:

```python
with open('file.md') as f:
    lines = f.readlines()
in_block = False
for i, line in enumerate(lines, 1):
    s = line.rstrip('\n')
    if s.strip() == '```':
        in_block = not in_block
        continue
    if in_block and s and s[0] in '|+':
        print(f"{i}: {len(s)} chars")
```

## Why this is non-obvious

The diagrams look correct in VS Code, Typora, and most terminals. The misalignment only appears on GitHub. Tools like `awk` that measure line length count bytes, making it appear that lines are the same length when their display widths differ.
