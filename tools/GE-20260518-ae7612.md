## Bash tool CWD resets between tool calls — jar xf then javap in separate calls uses wrong directory

**ID:** GE-20260518-ae7612
**Stack:** Claude Code Bash tool

**Symptom:** You extract a `.class` file from a JAR in one Bash call (`cd /tmp && jar xf ...`), then run `javap` on it in the next Bash call (`javap io/some/ClassName.class`). javap either fails silently, returns output for a stale file from a previous extraction, or reports a different class than expected.

**Context:** Any workflow that chains `jar xf` (to extract a class file) with `javap` (to inspect its signature) across two separate Bash tool invocations.

### Root cause

The Claude Code Bash tool shell **resets its working directory to the session baseline between separate tool calls**. A `cd /tmp` inside Bash call N does not persist to Bash call N+1 — each call begins in the original project/workspace directory.

When `jar xf /path/to.jar io/pkg/Class.class` runs in call N with CWD `/tmp`, it extracts to `/tmp/io/pkg/Class.class`. When `javap io/pkg/Class.class` runs in call N+1, the CWD is back at the session baseline (e.g. `/Users/.../project`), so javap looks for `project/io/pkg/Class.class` — which does not exist, or finds a stale file from an earlier extraction to that path.

The symptom looks like "javap found a different version" or "javap failed silently" — not "wrong CWD". This is especially confusing when the stale file exists and produces plausible-looking but incorrect output (e.g. showing 23 constructor params when the current artifact has 22).

Note: `cd` within a **single** Bash call does persist for the duration of that call. Only across calls does the CWD reset.

This is distinct from GE-0121 (CWD invalidated by `mv`) — that entry covers the project directory being physically moved; this entry covers the expected-but-wrong assumption that `cd` state persists between calls.

### Fix

Chain all steps that depend on a changed CWD into a **single** Bash tool call:

```bash
# Wrong (across two calls):
# Call 1:
cd /tmp && jar xf ~/.m2/.../artifact.jar io/pkg/ClassName.class
# Call 2:
javap io/pkg/ClassName.class   # ← CWD has reset; wrong directory

# Right (single call):
cd /tmp && jar xf ~/.m2/.../artifact.jar io/pkg/ClassName.class && javap io/pkg/ClassName.class
```

Alternatively, use absolute paths throughout:
```bash
jar xf ~/.m2/.../artifact.jar -C /tmp io/pkg/ClassName.class
javap /tmp/io/pkg/ClassName.class
```

### Why this is non-obvious

Shell state persistence is the expected mental model. In a terminal session, `cd /tmp` persists. Developers and agents naturally assume the same applies to the Bash tool. Nothing in the tool's output signals that the CWD was reset — the next call simply starts in a different directory silently. The failure mode (stale or wrong javap output) looks like a dependency version mismatch, not a working directory issue.
