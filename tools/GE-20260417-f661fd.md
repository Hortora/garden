## GOAP action correctness: verify reachability and goal-advancement independently

**ID:** GE-20260417-f661fd
**Stack:** GOAP / A* planning, any language
**Labels:** `#goap` `#planning` `#correctness` `#debugging`

**What it achieves:** A two-step mental model for verifying GOAP action definitions that catches silent planner failures before they reach production.

**Context:** Any system using Goal-Oriented Action Planning where actions are defined as (preconditions, effects, cost) triples and a planner finds the cheapest path from a starting WorldState to a goal condition. The check applies regardless of language or planner implementation.

### What most developers would do
Verify that an action fires: set up a WorldState, call the planner, assert that the action appears in the plan. If the test passes, assume the action is correct. This catches reachability failures but misses goal-advancement failures.

### The technique
When adding or changing a GOAP action, apply two independent checks — not just "does a plan exist":

**Check 1 — Reachability:** Can the planner reach this action from the starting WorldState?
- List the action's preconditions
- Verify each precondition key is set to the required value in the starting WorldState (or can be set by a preceding action)
- If any precondition is not satisfiable from the start state, the action is unreachable — the planner will never fire it

**Check 2 — Goal advancement:** Does applying this action move the WorldState closer to the goal?
- Apply the action's effects to the WorldState
- Verify the resulting state satisfies the goal condition directly, or enables a subsequent action chain that does
- If the effects don't advance toward the goal, the planner may reach the action but still return an empty plan

These two checks are orthogonal. An action can be reachable but not advance the goal (plan loops back or returns empty). An action can advance the goal but be unreachable (plan can never fire it).

```
Example — kiting action in a tactical AI:

KITE action:
  Preconditions: {inRange: true, onCooldown: true, enemyVisible: true}
  Effects:       {onCooldown: false}
  Goal:          enemyEliminated

Check 1 (Reachability):
  Starting kiting WorldState has inRange=true, onCooldown=true, enemyVisible=true
  → KITE preconditions satisfied ✓

Check 2 (Goal advancement):
  After KITE: onCooldown=false
  → Goal is enemyEliminated, not unitSafe
  → Is enemyEliminated reachable after KITE?
  → ATTACK requires inRange=true, onCooldown=false, enemyVisible=true
  → All satisfied after KITE → ATTACK sets enemyEliminated=true ✓
  → KITE → ATTACK chain reaches the goal ✓
```

### Why this is non-obvious
The planner returns an empty list for two distinct reasons: (a) goal already satisfied, and (b) goal unreachable. Both are silent — no exception, no warning. Without distinguishing reachability from goal-advancement in your mental model, diagnosing "why is the planner returning empty?" requires tracing through the A* search manually. The two-check model makes the failure mode explicit before you run the planner.

### Why this fix
A single end-to-end test ("does a plan exist?") doesn't tell you which of the two failure modes is active when the plan is empty. The two-check model forces you to reason about each failure mode independently, which surfaces bugs at design time rather than requiring a debugger.
