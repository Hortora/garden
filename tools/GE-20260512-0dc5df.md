## macOS sed -i '' silently empties a file when the working tree file is already empty

### Symptom

`sed -i '' 's/old/new/g' file.md` exits 0, prints any success echo you added, and reports no error. The file is now empty.

### Context

Occurs when the target file is already empty in the working tree before sed runs — typically because another uncommitted session left it in an empty state, or a prior operation truncated it. `sed` processes empty input and writes empty output. Git then stages and commits the empty file without complaint.

### Root cause

`sed` reads from the file, applies substitutions, and writes back. If the input is empty, the output is empty regardless of the substitution pattern. Exit code is still 0. There is no "file was empty, nothing to do" warning. The `-i ''` flag writes the (empty) result back in place.

This bit us twice with the same file (`docs/PLATFORM.md`) across separate sessions — each time a different `sed` operation was blamed, but the real cause was the file already being empty when sed ran.

### Fix

Check that the file has content before running `sed`:

```bash
[ -s file.md ] && sed -i '' 's/old/new/g' file.md || echo "WARNING: file.md is empty, skipping"
```

Or use Python, which reads first and only writes if content changed:

```python
import pathlib
f = pathlib.Path("file.md")
content = f.read_text()
updated = content.replace("old", "new")
if updated != content:
    f.write_text(updated)
```

### Why this is non-obvious

`sed` exits 0. Any success echo you wrapped around it fires. The file appears in `git diff` as a deletion of all lines — which looks like content was removed, not that the input was empty. The symptom points to the `sed` pattern, not to the file state before `sed` ran.
