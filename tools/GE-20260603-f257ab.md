## Symptom

Fork (origin) and upstream (blessed repo) end up with different commit histories after a squash+push workflow. No error is reported — both pushes succeed, but `git log` on the fork shows more commits than `git log` on upstream for the same range.

## Context

Two-remote git workflows: `origin` is a personal fork, `upstream` is the blessed repository. The workflow squashes noise commits before delivering to upstream. The bug occurs when push order is: push-to-fork → squash → push-to-upstream.

## Root cause

Squash rewrites commit SHAs via rebase. If the fork received the pre-squash commits and upstream receives the post-squash commits, they represent different histories — the same semantic work, but different git objects. No merge or rebase can reconcile them without a force push.

## Fix

Squash **before** either push. The correct order:

```
1. Squash (git-squash on the full upstream/main..HEAD range)
2. Push to fork (origin) — squashed history
3. Push to upstream — same squashed history
```

Both destinations receive identical commits. The fork is always updated first; upstream delivery is the optional final step. The upstream can never be ahead of the fork.

## Why this is non-obvious

The squash step is conceptually "preparation for upstream delivery", so it naturally feels like it belongs between the fork push and the upstream push. The fork push feels like a safe intermediate checkpoint. In practice, both destinations must receive the same rewritten history or they permanently diverge.
