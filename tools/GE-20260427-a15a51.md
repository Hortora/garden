## Symptom

Playwright `page.mouse().click(sx, sy)` computed from `worldToScreen(wx, wz)` doesn't trigger the Three.js raycaster hit on a sprite, even though the sprite is visually on screen and the coordinates appear correct.

## Context

Unit sprites in a Three.js scene are positioned at `TERRAIN_SURFACE_Y + TILE * 0.5 ≈ 0.43` world units above y=0. The `worldToScreen(wx, wz)` utility projected `new THREE.Vector3(wx, 0, wz)` — hardcoding `y=0`. In an isometric camera at ~14 world-unit distance with FOV 45°, 1 world unit ≈ 62 screen pixels. A Y offset of 0.43 shifts the projection by ~27 pixels. A sprite appearing ~60px wide on screen requires the click to land within ~30px of centre. Projecting at y=0 when the sprite is at y=0.43 produces a click 27px below the sprite's actual centre — just enough to miss in many camera configurations.

## What was tried

- Confirmed `worldToScreen(wx, wz)` gave on-screen coordinates (within viewport bounds)
- Added `TERRAIN_SURFACE_Y + TILE * 0.5` as `wy` parameter — helped but physical click still unreliable
- Switched to `unitScreenPos(tag)` which projects the sprite's actual Three.js position — still unreliable for physical clicks due to UI element occlusion
- Resolved by using programmatic raycasting via `window.__test.clickUnit(tag)` instead of physical mouse events

## Root cause

`worldToScreen` hardcoded `y=0`. Sprites at non-zero Y positions project to a different screen location than the y=0 equivalent, and the raycaster fires a ray through the y=0 projection — missing the sprite's actual 3D bounding box.

## Fix

Add an optional `wy` parameter to `worldToScreen`:

```javascript
// Before
worldToScreen: (wx, wz) => {
  const v = new THREE.Vector3(wx, 0, wz).project(camera);
  // ...
},

// After
worldToScreen: (wx, wz, wy = 0) => {
  const v = new THREE.Vector3(wx, wy, wz).project(camera);
  // ...
},
```

Use sprite height when projecting for click targets:
```javascript
// In test:
"() => window.__test.worldToScreen(" + wx + ", " + wz + ", TERRAIN_SURFACE_Y + TILE * 0.5)"
```

Better still: use `unitScreenPos(tag)` which projects from the sprite's actual Three.js world position:
```javascript
unitScreenPos: tag => {
  const sp = unitSprites.get(tag) ?? enemySprites.get(tag);
  if (!sp || !camera || !renderer) return { x: -1, y: -1 };
  const v = sp.position.clone().project(camera);
  const sz = renderer.getSize(new THREE.Vector2());
  return { x: Math.round((v.x+1)/2*sz.width), y: Math.round((-v.y+1)/2*sz.height) };
},
```

## Why this is non-obvious

The 0.43 world-unit Y offset sounds tiny, but in an isometric camera at 14-unit distance the angular error translates to a significant screen offset (~27px). The click coordinates appear valid (they're on-screen and near the sprite's visual centre), but the Three.js raycaster fires through the y=0 projection point — a subtly different 3D ray. The symptom (click doesn't trigger raycaster) has no error message and is identical to the click landing in empty space.
