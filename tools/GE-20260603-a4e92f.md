### Symptom
`mcp__intellij-index__ide_find_file` returns paths in the form `jar:///Users/.../.m2/repository/group/artifact/version/artifact-sources.jar!/com/example/Foo.java`. Passing that exact path to `mcp__intellij-index__ide_read_file` returns "File not found".

### Context
Trying to read source files from Maven dependencies (e.g. a beta library's `-sources.jar`) by using the path returned by `ide_find_file`. The path looks valid — it is the format IntelliJ uses internally — but `ide_read_file` does not accept it.

### What was tried
Passing the `jar:///...` path verbatim from `ide_find_file` output back to `ide_read_file`. Tried with both triple-slash (`jar:///`) and double-slash (`jar://`) variants — both fail.

### Root cause
`ide_read_file` has a stricter path resolver that does not handle the `jar://` URL scheme for Maven repository JARs, even though other tools produce and consume that format internally.

### Fix
Use `ide_find_definition` instead — it resolves and decompiles on demand:
```json
{
  "tool": "mcp__intellij-index__ide_find_definition",
  "language": "Java",
  "symbol": "dev.langchain4j.agentic.declarative.DeclarativeUtil"
}
```

Or use `ide_find_class` with `scope: "project_and_libraries"` to locate the class, then navigate with `ide_find_definition`. IntelliJ decompiles `.class` files on demand even when no `-sources.jar` is attached.

### Why this is non-obvious
The path returned by one MCP tool in the same server (`ide_find_file`) is not accepted by another tool in the same server (`ide_read_file`). The natural assumption is that tool output is interoperable as tool input. The error message does not explain why the path is rejected or suggest the correct approach.
