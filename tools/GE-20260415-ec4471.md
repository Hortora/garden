## Stacked PRs from a fork: push the base branch to upstream first

### Stack
Git, GitHub

### Symptom
Creating a stacked PR via `gh pr create --base <branch>` fails with:

```
pull request create failed: GraphQL: Head sha can't be blank, Base sha can't be blank,
No commits between <upstream>:<base-branch> and <fork>:<feature-branch>
```

Or GitHub cannot resolve the diff because it cannot find the base branch in the upstream
repository.

### Context
Working on a fork (`user/repo` forked from `org/repo`). A base branch
(`feat/feature-a`) only exists in the fork from a previous PR that hasn't merged yet.
Trying to create `feat/feature-b` as a stacked PR targeting `feat/feature-a` in the
upstream fails because the upstream doesn't have `feat/feature-a`.

### Root cause
GitHub PR diff calculation requires BOTH the head and base branch to exist in the same
repository (or at least for the upstream to have the base commit). When the base branch
only lives in your fork, the upstream cannot compute the diff.

### Fix
Push the base branch to the upstream repository first, then create the stacked PR:

```bash
# Push all branches in the stack to upstream
git push upstream feat/feature-a feat/feature-b feat/feature-c

# Now create stacked PRs — base branches exist in upstream
gh pr create \
  --title "PR 1: Feature A" \
  --base main \
  --head feat/feature-a \
  --repo org/repo

gh pr create \
  --title "PR 2: Feature B (stacks on A)" \
  --base feat/feature-a \
  --head feat/feature-b \
  --repo org/repo
```

The result: each PR shows only its own diff (the commits between it and its base), making
reviews manageable. When the bottom PR merges, subsequent PRs automatically rebase their
diff view.

### Why this is non-obvious
Most developers working on forks assume they only push to `origin` (their fork). The
idea that you'd also push *branches* (not just tags or PRs) to `upstream` feels unusual.
The GitHub error message ("No commits between...") doesn't hint at the missing upstream
branch as the cause.

### Alternatives considered
- **Use fork-to-fork PRs** (PR B targets fork's branch A): GitHub supports intra-fork
  PRs but `gh pr create` against the upstream repo doesn't accept a fork branch as the
  `--base`. Adds confusion and means reviewers work in two repos.
- **Rebase all onto main**: Loses the incremental diff benefit — each PR shows all
  accumulated changes rather than just its own contribution.
