### Symptom
`git rebase -i <base>` fails with:
```
error: invalid line N: pick <sha> <merge commit message>
hint: replay the merge, use 'merge -C' on the commit.
```

### Context
Running `git rebase -i` on a range that contains real two-parent merge commits (commits with two parents — not squash merges). Common when doing full-history compaction on a repo that used `git merge` for epic landings.

### Root cause
`git rebase -i` without `--rebase-merges` cannot process two-parent commits using the `pick` command. The hint in the error mentions `merge -C` but doesn't explain the full picture.

### Fix
Use `--rebase-merges` flag:
```bash
git rebase -i --rebase-merges <base>
```
This generates a different todo format that uses `label`, `reset`, and `merge -C <sha> <label>` lines to replay merge commits. The squash/pick commands then operate on the non-merge commits only.

### Why this is non-obvious
The error `invalid line N: pick <sha>` gives no indication that the problem is a merge commit. The symptom looks like a malformed todo file. Developers familiar with interactive rebase naturally try `pick` for all commits.
