## Technique

Pipe `gh api` output through `tr -d` to strip Unicode control characters before `jq` parsing when processing data that may contain emoji or special characters in user-generated content (commit messages, PR titles, workflow names):

```bash
gh api "/repos/org/repo/actions/runs?branch=main&per_page=1" \
  2>/dev/null \
  | tr -d '\000-\010\013-\037\177' \
  | jq -r '.workflow_runs[0] // empty'
```

The `tr -d` pattern strips:
- `\000-\010` — U+0000 to U+0008 (null, control chars except tab)
- `\013-\037` — U+000B to U+001F (control chars except newline)  
- `\177` — U+007F (DEL)

This preserves tab (`\011` = U+0009) and newline (`\012` = U+000A) which are valid in JSON strings and multi-line values.

## Why This Is Non-Obvious

`jq` is strict about JSON control characters per RFC 7159: characters U+0000–U+001F must be escaped in strings. GitHub's API sometimes returns unescaped control characters in user-generated content (e.g. emoji in commit messages that contain underlying control sequences). The error `jq: parse error: Invalid string: control characters from U+0000 through U+001F must be escaped` appears to come from the data rather than the jq query.

Most developers add `--raw-input` or `--null-input` to jq flags trying to fix it, neither of which helps. The actual fix is upstream sanitisation before jq sees the data.

## Context

Reliably occurs when processing GitHub Actions run metadata for repositories where contributors use emoji or special characters in commit messages or PR titles.
