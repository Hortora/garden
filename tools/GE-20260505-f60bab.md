## MCP StdioServerParameters command='python3' spawns wrong interpreter in pyenv/venv — McpError: Connection closed

**ID:** GE-20260505-f60bab
**Stack:** Python MCP SDK, pyenv / venv environments

**Symptom:** MCP protocol integration tests all fail with `McpError: Connection
closed`. The error appears to be a protocol-level failure, with no indication
of what went wrong.

**Context:** Running pytest via a pyenv or virtual-environment Python that has
`mcp` installed. The test uses `StdioServerParameters` to spawn the MCP server
as a subprocess and communicate via stdio.

### What was tried (didn't work)

- Assumed the MCP session initialisation itself was broken
- Checked the server script for errors — it looked fine
- Read the exception trace — `Connection closed` gave no real signal

### Root cause

`StdioServerParameters(command='python3', ...)` launches the server subprocess
using the shell's `python3`, which in a pyenv environment resolves to the
system Python — not the pyenv interpreter running pytest. The system Python
has no `mcp` package, so the server crashes immediately on import. The
crash appears in stderr (`ModuleNotFoundError: No module named 'mcp'`) but
the test surface is `McpError: Connection closed`, which looks like a
protocol failure, not a missing dependency.

### Fix

```python
import sys

StdioServerParameters(
    command=sys.executable,   # uses the exact interpreter running the test
    args=[str(SERVER)],
    env={**os.environ, 'HORTORA_GARDEN': str(garden)},
)
```

`sys.executable` is always the interpreter that launched the current process,
so it matches the venv/pyenv Python that has the required packages installed.

### Why this is non-obvious

The error (`McpError: Connection closed`) surfaces at the MCP protocol layer,
not at the subprocess layer. Nothing in the exception trace mentions a wrong
Python version or a missing package. The real cause (`ModuleNotFoundError`)
is only visible in captured stderr, which is easily missed.
