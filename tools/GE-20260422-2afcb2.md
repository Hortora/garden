## Quoted heredoc delimiter prevents variable expansion in bash installer embedded scripts

**Stack:** Bash
**What it does:** When a bash installer writes a script file using a heredoc, quoting the delimiter (`<< 'EOF'`) ensures that `$VARIABLE` and `$(command)` in the heredoc body are written literally — they expand when the *installed script* runs, not when the *installer* runs.

### The technique

```bash
# WRONG — expands variables at install time
# $GARDEN_ROOT becomes whatever it is NOW, not at runtime
cat > "$AGENT_SH" << AGENT_EOF
GARDEN_ROOT="${HORTORA_GARDEN:-$HOME/.hortora/garden}"
LOG="$GARDEN_ROOT/garden-agent.log"
AGENT_EOF

# RIGHT — single-quoted delimiter prevents expansion
# $GARDEN_ROOT and $(pwd) are written literally, expand at agent runtime
cat > "$AGENT_SH" << 'AGENT_EOF'
GARDEN_ROOT="${HORTORA_GARDEN:-$HOME/.hortora/garden}"
LOG="$GARDEN_ROOT/garden-agent.log"
AGENT_EOF
```

The installed script correctly reads `HORTORA_GARDEN` from the environment when it actually runs, not hardcoded to the value present during installation.

### Why developers wouldn't naturally reach for this

Unquoted heredoc delimiters are the default in most examples and tutorials. The bug is silent — the installed script appears to work on the machine where it was installed (because `$HOME` happens to be correct), but breaks on any other machine or user account. The only way to catch it is to read the installed file and notice the hardcoded path.
