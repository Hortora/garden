### Symptom
`git push` fails with `fatal: could not read Username for 'https://github.com': Device not configured`
even though GitHub credentials are stored in the macOS Keychain (verifiable via
`security find-internet-password -s github.com`). One repo pushes fine while
another fails — no obvious difference between them.

### Context
macOS stores credentials from browser logins, `gh auth login`, or prior git sessions
in the system Keychain. But git only consults the Keychain if `credential.helper`
is explicitly configured. Without it, git never asks the Keychain and fails
immediately if no in-process credentials are available.

The inconsistency between repos arises because one repo's pushes were going via
`gh` CLI (which has its own auth path) while the other went via bare git.

### What was tried
- Verified credentials exist: `security find-internet-password -s github.com` returned the account
- Compared `.git/config` of working vs failing repo — identical
- Checked `git config --global credential.helper` — not set

### Root cause
`git config --global credential.helper` was not set. Git has no default
credential helper on macOS — it must be explicitly configured to use the
Keychain even though the Keychain already has the credentials.

### Fix
```bash
git config --global credential.helper osxkeychain
```

This tells git to use the macOS Keychain for all HTTPS credential lookups.
One-time global fix; persists across repos and sessions.

### Why this is non-obvious
macOS does not configure `credential.helper` by default even on systems
where git and the Keychain are both present. The error message "Device not
configured" gives no hint that credentials exist but aren't being consulted.
The partial working state (one repo succeeds via `gh`, another fails via git)
makes it look like a repo-specific problem rather than a global config gap.
