## `status` is a read-only variable in bash — silent failure when used as a function parameter

**ID:** GE-20260429-7d31f6
**Stack:** bash (all versions)

**Symptom:**
```
function_name:1: read-only variable: status
```
The function errors immediately on the first use of the parameter. No other indication of why.

**Context:** Defining a shell function with a parameter named `status`:
```bash
set_item() {
  local item_id=$1 status=$2 track=$3
  # → bash: status: readonly variable
}
```

**Root cause:** `status` is a reserved read-only variable in bash (it holds the exit status of the last foreground pipeline). Attempting to assign it via `local status=$2` fails because the read-only attribute cannot be overridden, even in a local scope.

**Fix:** Rename the parameter to anything other than `status`:
```bash
set_item() {
  local item_id=$1 stat=$2 track=$3
  # works fine
}
```

**Why non-obvious:** The name `status` seems entirely reasonable as a parameter name for a field value. The error message (`read-only variable: status`) points to the variable name but does not explain that `status` is a bash builtin. There is no obvious connection between "exit status" and a user-defined function parameter named `status`.

**Other reserved bash variables to avoid as parameter names:** `LINENO`, `RANDOM`, `SECONDS`, `BASH_SUBSHELL`, `IFS` (technically not read-only but dangerous to shadow).
