### Symptom

Re-running `git filter-repo` after a previous run (even days later) fails immediately:

```
The previous run is older than a day (.git/filter-repo/already_ran already exists).
See "Already Ran" section in the manual for more information.
Treat this run as a continuation of filtering in the previous run (Y/N)?
Traceback (most recent call last):
  ...
EOFError: EOF when reading a line
```

The process exits non-zero. No filter-repo work is performed.

### Context

Occurs when running `git filter-repo` in any non-interactive context — scripts,
CI, or `git -C <path>` invocations from another process — where stdin is not a
real terminal. The EOFError happens because filter-repo attempts to read a `Y/N`
prompt from stdin, gets EOF, and crashes.

### Root cause

`git filter-repo` writes `.git/filter-repo/already_ran` after its first run. On
any subsequent invocation without `--force`, it prompts interactively to confirm
whether to continue from the previous run. In a non-interactive context there is
no stdin to read from, so the prompt immediately raises `EOFError`.

### Fix

Delete the sentinel file before re-running:

```bash
rm -f .git/filter-repo/already_ran
git filter-repo <your-options>
```

Or pass `--force` to bypass the check entirely (skips the prompt unconditionally):

```bash
git filter-repo --force <your-options>
```

Use `--force` when you know the re-run is intentional and not a continuation;
use the `rm` approach when you want to be explicit about which check you're bypassing.

### Why this is non-obvious

The error message ("EOF when reading a line") is a Python traceback with no
reference to the `.git/filter-repo/already_ran` file or to the `--force` flag.
The actual cause — a non-interactive stdin combined with a Y/N prompt — is buried
in the stack trace and requires knowing that filter-repo writes a sentinel file
after each run.
