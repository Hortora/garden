## Context

A downstream repo failed to rebuild after an upstream published a new version because it was missing from the upstream's `repository_dispatch` list. Or a dispatch chain has a new repo that hasn't been triggered yet. You need a CI rebuild without making a content change to the repo.

## Technique

```bash
gh workflow run ci.yml --repo casehubio/<repo-name>
```

This triggers the repo's CI workflow via `workflow_dispatch` without touching any code. The workflow must have `workflow_dispatch:` in its `on:` triggers (most do).

To trigger the equivalent of the upstream-published dispatch event:

```bash
gh api repos/casehubio/<repo>/dispatches \
  -f event_type=upstream-published \
  -f client_payload[source]="casehubio/engine"
```

## Why this is non-obvious

The typical workaround is to make a dummy commit and push, which pollutes git history. `gh workflow run` achieves the same result cleanly. Most developers know `gh` can interact with issues and PRs; fewer know it can trigger workflow runs directly.

The distinction between `workflow_dispatch` (triggers the named workflow) and `repository_dispatch` (fires a custom event the workflow listens to) matters: use `workflow_dispatch` for the generic CI workflow, `repository_dispatch` with `event_type=upstream-published` when the workflow specifically listens for that event type.
