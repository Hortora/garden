### Symptom
Need to verify whether a branch's specific code changes actually reached main, when the
branch was squash-merged (different SHAs) and main has evolved significantly since.
File-presence checks (`+` files only) miss stranded content inside shared files.

### Context
Auditing 47 branches with false stamps across 36 repos. Initial triage checked only for
files that exist on the branch but not on main. This missed 8 files of stranded
CloudEventType + MVEL content on platform — the files existed on both sides but the
branch's specific modifications had been dropped during squash.

### The technique

For each branch, use the merge-base to extract what the branch actually added, then
verify each added line exists in main's current version:

```python
merge_base = git("merge-base", main, branch)
diff = git("diff", f"{merge_base}..{branch}", "--", source_file)

# Extract non-trivial added lines (skip blanks, braces, imports)
added_lines = [normalize(l[1:]) for l in diff.split("\n")
               if l.startswith("+") and not l.startswith("+++")
               and not is_trivial(l[1:])]

# Check against main's version
main_content = set(normalize(l) for l in git("show", f"{main}:{file}").split("\n"))
stranded = [l for l in added_lines if l not in main_content]
```

**Key design decisions:**
- **Merge-base, not two-dot diff** — isolates the branch's delta, not the full tip difference
- **Line-level, not file-level** — catches modifications to shared files, not just new files
- **Trivial-line filtering** — skips blank lines, lone braces, imports, `@Override` to reduce
  false positives from reformatting
- **Normalized comparison** — strips whitespace before matching to handle indentation changes

### Limitations
- **API evolution noise:** constructor argument reordering, type migrations, and renamed
  parameters produce false positives. The branch added `new Foo(a, b)` but main has
  `new Foo(b, a)` — the line doesn't match even though the functionality landed.
- **Files-only mode** is faster and noise-free for detecting entirely missing modules.
  Use line-level mode when file-presence passes but content loss is suspected.

### Why this is non-obvious
The natural approach is `git diff main branch --stat` — but this conflates main's evolution
with branch content. The merge-base isolation + line-in-file verification is a two-step
process that most developers wouldn't reach for. It's effectively a manual three-way merge
check without actually merging.

### Alternatives considered
- `git diff main branch --name-only` with `+`/`-`/`~` classification — misses shared-file content entirely
- `git merge-tree` for three-way comparison — shows conflicts but not clean content differences
- Per-commit `git log` + `git show` — correct but O(commits × files), much slower
