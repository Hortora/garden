### Symptom
After calling `ide_replace_member` on a Java method, the file contains a duplicate method signature — the original signature appears above the replacement, followed by the replacement content which also contains the signature. The result is a syntax error (`';' expected`, `<identifier> expected`).

### Context
Occurs when the `content` parameter passed to `ide_replace_member` includes the method's `@annotations` and/or method signature (return type, name, parameters). The tool is designed to replace only the body (between `{` and `}`), but silently accepts content that includes the signature — producing a duplicate.

### Root cause
`ide_replace_member` replaces the method **body** — the code between the opening `{` and closing `}` braces. The method signature (annotations, modifiers, return type, name, parameters) is preserved from the original. When the `content` parameter includes annotations or signature elements, they are inserted as body content above the original signature, creating a syntactically invalid duplicate.

`ide_edit_member` is the correct tool when the entire method declaration (including signature, annotations, modifiers) needs replacing.

### Fix
For body-only replacement: pass only the code that goes inside `{ }` — no annotations, no method signature, no braces.

For full method replacement (including signature changes): use `ide_edit_member` instead of `ide_replace_member`.

| Tool | Replaces | Content should include |
|------|----------|----------------------|
| `ide_replace_member` | Body only (between braces) | Body code only — no annotations, no signature |
| `ide_edit_member` | Entire declaration | Full declaration — annotations, modifiers, signature, body |

### Why this is non-obvious
The tool does not validate or reject content that includes signature elements. There is no error or warning — the duplicate is silently written. The resulting compilation errors (`;' expected`, `<identifier> expected`) don't obviously point to a duplicated method signature.
