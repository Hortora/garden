## What it is

The OpenClaw `/hooks/agent` endpoint accepts a `deliver` field. When set to `"webhook"`,
OpenClaw POSTs the finished agent result payload to the URL specified in the `to` field —
any arbitrary HTTP endpoint, not just OpenClaw's own messaging channels.

```bash
POST /hooks/agent
Authorization: Bearer SECRET

{
  "message": "Check boiler pressure and report current reading",
  "agentId": "home-agent",
  "deliver": "webhook",
  "to": "https://casehub.internal/openclaw/delivery/channel/abc123",
  "timeoutSeconds": 30
}
```

OpenClaw executes the skill, generates its LLM output, and POSTs the finished result to
the specified URL. The caller does not need to poll or maintain a connection.

## Context

The `/hooks/agent` docs lead with channel-based delivery (`channel: "telegram"`,
`channel: "whatsapp"`) and `deliver: "announce"`. The webhook delivery mode is listed
but not emphasised, and its integration potential as an arbitrary HTTP callback is not
explained in any guide.

## Why this matters

This is the key primitive for bidirectional Qhorus ↔ OpenClaw wiring. Without it:

- CaseHub can send tasks to OpenClaw (via `/hooks/agent`)
- But OpenClaw results have no path back to a Qhorus channel

With `deliver: "webhook"` pointing to a Qhorus channel delivery endpoint:

1. CaseHub fires `POST /hooks/agent` with `deliver: webhook`, `to: qhorus-endpoint`
2. OpenClaw executes the skill
3. OpenClaw POSTs the result to the Qhorus endpoint
4. Qhorus receives it as a typed message via `MessageService.dispatch()`
5. Commitment fulfilled, Watchdog disarmed, ledger records

The complete Qhorus ↔ OpenClaw bidirectional loop requires no polling and no persistent
connection — just two HTTP calls.

## Delivery modes summary

| `deliver` value | Behaviour |
|---|---|
| `"announce"` | Fallback deliver final text to chat channel |
| `"webhook"` | POST finished result to arbitrary `to` URL |
| `"none"` | No delivery — caller must poll or use streaming |
| omitted | Default (usually announce to last channel) |

## Why this is non-obvious

The docs frame `/hooks/agent` as a way to trigger an agent and receive the result in a
messaging channel (Telegram, WhatsApp, Slack). The webhook delivery mode for sending
results to arbitrary HTTP endpoints — which is what makes it useful for system integration
rather than user notification — is not highlighted in any integration guide or example.
