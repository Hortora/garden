## Context

Maintaining a fork of an upstream project (e.g. `quarkus-langchain4j`) where you need to publish to your own registry (e.g. GitHub Packages under your org) without modifying the fork's `pom.xml`. The goal: keep the pom 100% upstream-compatible so fix commits can be cherry-picked and submitted as PRs without any casehub-specific changes to exclude.

## Technique

Pass `-DaltDeploymentRepository` to `mvn deploy` in the CI workflow. Maven uses this URL for deployment instead of reading `<distributionManagement>` from the pom:

```yaml
- name: Publish fork to GitHub Packages
  run: |
    mvn --batch-mode deploy -DskipTests \
      -DaltDeploymentRepository=github::https://maven.pkg.github.com/myorg/myrepo
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

The format is `id::url` where `id` matches a server entry in settings.xml. No `<distributionManagement>` block needed in `pom.xml`.

The fork's pom stays identical to upstream. When cherry-picking fixes to submit upstream, the only casehub-specific file to exclude from the PR is the CI workflow file (`casehub-publish.yml`) — not any pom changes.

## Why this is non-obvious

The standard Maven deploy pattern adds `<distributionManagement>` to pom.xml. Most developers wouldn't reach for `-DaltDeploymentRepository` because it's a command-line override flag that isn't prominently featured in Maven getting-started docs. The flag exists specifically for the "publish somewhere other than what the pom says" use case, which is precisely what fork publishing requires.
