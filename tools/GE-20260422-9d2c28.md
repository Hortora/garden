### Context
You suspect that a set of private utility methods is duplicated across multiple large files and want to extract them to a shared module. Before doing so, you need to know which methods appear in which files without reading every file in full.

### Technique
Use `grep -l` (files-with-matches mode) and count results to identify which method names appear in multiple files:

```bash
# Check a list of candidate method names across two large source files
for method in prefixBeforeFirstDigit firstEmbeddedNumber replaceFirstEmbeddedNumber \
              isTNumberVar classNameSuffix stripNumericSuffix buildExpandedTypeArgs \
              expandTypeStringForJ expandMethodTypesForJ parseReturnTypeInfo buildReturnTypeStr; do
  count=$(grep -l "$method" InlineGenerator.java PermuteProcessor.java 2>/dev/null | wc -l)
  [ "$count" -gt 1 ] && echo "BOTH: $method"
done
```

Output (`BOTH: methodName`) immediately shows which methods are candidates for extraction. Methods appearing in only one file are local to that file and not candidates.

This is faster than reading both files in full, and more reliable than grepping for a specific pattern that might match comments or imports rather than declarations.

### Why this is non-obvious
The natural approach is to open both files and manually scan for matching methods — tedious and error-prone at scale. `grep -l` is typically used for "find files containing X", not for cross-file membership checks. The `wc -l` count on the result converts a file-list into a count, enabling the `> 1` threshold check. The combination is not obvious from either command's documentation.
