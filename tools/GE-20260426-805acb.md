## Behaviour

After adding `workflow_dispatch:` to a workflow's `on:` triggers and pushing to the default branch, the GitHub Actions API and `gh workflow run` still report the trigger as unavailable:

```
could not create workflow dispatch event: HTTP 422:
Workflow does not have 'workflow_dispatch' trigger
```

This persists for several minutes to tens of minutes after the push.

## Context

Affects any workflow file where `workflow_dispatch` was absent and is being added. The trigger becomes available at the next scheduled run of the workflow, or after the cache naturally expires.

## Root Cause (Inferred)

GitHub Actions caches compiled workflow definitions. After a push, the cached definition is not immediately invalidated — the API still sees the pre-push definition until the cache is refreshed. This refresh appears to happen on the next scheduled trigger or after a timeout.

This behaviour is **not documented** in the GitHub Actions documentation. The only mention of workflow definition caching is in the context of reusable workflows.

## Workaround

Wait for the next scheduled run (if the workflow has `schedule:` triggers) or wait a few minutes and retry. Do not interpret the 422 error as a permanent configuration problem.

Alternatively: trigger the workflow via the GitHub web UI (Actions tab → select workflow → "Run workflow") which sometimes bypasses the cache.
