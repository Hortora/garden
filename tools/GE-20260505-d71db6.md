### What
`com.github.ocraft.s2client.protocol.observation.spatial.ImageData.getData()` returns
`byte[]` directly — not a `ByteString` or `InputStream`. The internal field is
`ByteString data` (protobuf), but the public getter already calls `.toByteArray()` before
returning.

### Usage

```java
// To extract the pathing grid at game start:
observation().getGameInfo().getStartRaw().ifPresent(raw -> {
    ImageData pg = raw.getPathingGrid();
    byte[] data   = pg.getData();          // byte[] — no .toByteArray() needed
    int width     = pg.getSize().getX();
    int height    = pg.getSize().getY();
    TerrainGrid terrain = TerrainGrid.fromPathingGrid(data, width, height);
});
```

The bit encoding for 1-bit pathing grids: `walkable = (data[index/8] >> (7 - index%8)) & 1`
where `index = x + y * width`.

### How to discover
Only discoverable via IntelliJ "Go to Source" on `ImageData` in the
`ocraft-s2client-protocol-0.4.21-sources.jar`. The ocraft Javadoc does not document
the return type for `getData()`, and the protobuf source (from which ocraft wraps) exposes
`ByteString`. The `@JsonIgnore` annotation on `getData()` is the only hint something
interesting is happening.

### Why this matters
Developers familiar with protobuf assume `getData()` returns `ByteString` and call
`.toByteArray()` on it — this compiles and runs but is redundant (calling toByteArray on
an already-converted array would fail). Knowing the return is already `byte[]` simplifies
the integration code.
