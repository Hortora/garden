## GOAP cheaper plan bypasses intended action when precondition complement is missing

**ID:** GE-20260417-a7f7fc
**Stack:** GOAP / A* planning, Java
**Labels:** `#goap` `#planning` `#preconditions` `#cost-optimisation`

### Symptom
A unit classified into a kiting group (in-range + on weapon cooldown) receives an AttackIntent instead of the expected kiting MoveIntent. The GOAP planner is working correctly — it found a cheaper valid plan. The action that should be gated by cooldown state fires anyway.

### Context
Occurs when an action (e.g. ATTACK) lacks an explicit precondition for a condition that should gate it (e.g. `onCooldown: false`). The WorldState for a kiting unit has `onCooldown: true`, but if ATTACK doesn't declare `onCooldown: false` as a precondition, `GoapAction.isApplicable()` sees all ATTACK preconditions satisfied (it only checks keys it declares). ATTACK (cost 2) beats KITE+ATTACK (cost 3), so the planner correctly chooses ATTACK — the wrong outcome.

### What was tried
The kiting Drools rules and WorldState were verified to be correct. The dispatch switch was inspected. The bug was only found by writing an integration test asserting that an on-cooldown unit emits a MoveIntent rather than an AttackIntent.

### Root cause
`GoapAction.isApplicable()` checks only the keys declared in the action's precondition map. Keys absent from preconditions are ignored — the action is applicable regardless of their value in the WorldState. Adding `"onCooldown", false` to ATTACK's preconditions makes ATTACK inapplicable from the kiting WorldState (`onCooldown: true`), forcing the planner through KITE (which resets onCooldown to false) before ATTACK can fire.

### Fix
Explicitly add `"onCooldown", false` to ATTACK's precondition map:

```java
// Before — ATTACK applicable from any WorldState that has inRange=true, enemyVisible=true
"ATTACK", new GoapAction("ATTACK",
    Map.of("inRange", true, "enemyVisible", true),
    Map.of("enemyEliminated", true), 2)

// After — ATTACK only applicable when weapon is ready
"ATTACK", new GoapAction("ATTACK",
    Map.of("inRange", true, "enemyVisible", true, "onCooldown", false),
    Map.of("enemyEliminated", true), 2)
```

For groups where `onCooldown` is not set in their WorldState (e.g. in-range group), `WorldState.get("onCooldown")` returns `false` — satisfying the precondition correctly.

### Why this is non-obvious
The planner produces the optimal plan given the action library. A missing precondition doesn't cause an error; it silently makes an action over-applicable. The planner did exactly the right thing: it found the cheapest path. The bug is in the action definition, not the planner. Without an integration test specifically checking what action type is emitted for a cooldown scenario, the fault is invisible.
