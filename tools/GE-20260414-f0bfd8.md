## Context
Writing tests for 5–10 existing validators or scripts in a single session, where each has non-obvious internal behaviour that wouldn't be discoverable without reading the source.

## Technique
Dispatch parallel TDD subagents with an explicit two-phase instruction: (1) read the validator source in full, then (2) write tests based on what the code actually does — not what you assume it does.

The critical phrase: **"Read `scripts/validation/validate_X.py` first. Understand EVERY function."**

Without this instruction, agents write tests based on plausible assumptions (e.g. "link validators use HEAD requests") that may be wrong. With it, they discover the actual behaviour (e.g. `requests.get`, stderr output, template-skip patterns) and write assertions that would catch real regressions.

## Example prompt structure
```
Step 1: Read validate_links.py in full. Note what HTTP method it uses,
        which status codes it treats as OK, which domains it skips,
        what exit code it returns for each scenario.

Step 2: Write tests based on what the code actually does.
        Do NOT assume HEAD requests. Do NOT assume CRITICAL exit codes.
```

## Results
Applied to 5 push-tier validators in one session: produced 228 tests across 5 files, all passing immediately, with several non-obvious findings captured (GET not HEAD, stderr not stdout, template-skip pattern). Without the "read first" instruction, at least 3 of those findings would have produced wrong mock targets and silently-passing tests.

## Why non-obvious
The natural instruction for TDD subagents is "write tests for X". Adding "read X first" seems obvious in retrospect but is rarely explicit in prompts — agents default to plausible assumptions about APIs they haven't read, producing tests that pass against the wrong behaviour.
