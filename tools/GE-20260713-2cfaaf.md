### Symptom

`ide_create_file`, `ide_insert_member`, `ide_edit_member`, or `ide_replace_text_in_file` returns a timeout error after 300 seconds:

```
MCP server "intellij-index" tool "ide_insert_member" sent no response or progress for 300s; aborting.
```

Retrying the same operation produces duplicated content — e.g. interface members appear twice, method bodies are doubled.

### Context

Occurs during sessions with heavy IntelliJ MCP usage on large projects. Write operations take longer than the 300s idle timeout, but IntelliJ's document write action completes independently of the MCP response channel. Read operations (`ide_read_file`, `ide_find_references`, `ide_index_status`) are unaffected — they respond within seconds even when writes are timing out.

### What was tried (didn't work)

- Retrying the timed-out operation → caused duplicate content (the original write had silently succeeded)
- Using `ide_sync_files` to refresh → did not help; IntelliJ already had the correct state, the problem was the duplicate retry

### Root cause

The MCP tool timeout (300s default, configurable via `CLAUDE_CODE_MCP_TOOL_IDLE_TIMEOUT`) fires before IntelliJ finishes its response handshake, but IntelliJ's internal document write action completes independently. The timeout is a client-side abort of the response wait — not a cancellation of the server-side operation.

### Fix

After any MCP write tool timeout:

1. **Check the file state before retrying** — use `ide_read_file` (which doesn't time out) or check line count with `wc -l`
2. If the content is already present, do not retry
3. If the content is not present, retry once
4. If write tools are consistently timing out, fall back to `Edit` with `replace_all: true` (bypasses the MCP write path) for modifications, and `Write` for new files

### Why this is non-obvious

The timeout error strongly implies the operation failed. The natural response — retrying — is exactly wrong. There is no indication in the error message that the write may have succeeded. The asymmetry between reads (always fast) and writes (intermittently slow) further misleads: if reads work, you'd assume a write timeout means write failure, not write success with slow acknowledgement.

### Why this fix

Checking file state after timeout is defensive but cheap (`ide_read_file` responds in <1s even when writes are timing out). The alternative — increasing `CLAUDE_CODE_MCP_TOOL_IDLE_TIMEOUT` globally — masks the symptom and slows down legitimate timeout detection for other MCP servers.

### Alternatives considered

- **Increase timeout globally** — rejected; penalises all MCP servers for one server's latency
- **Set per-server timeout in MCP config** — viable but requires knowing the right value; doesn't prevent the duplicate-on-retry problem if the higher timeout is still exceeded
- **Always use Edit/Write instead of MCP** — rejected; loses structural editing benefits (auto-reformat, import optimization, reference-aware operations)
