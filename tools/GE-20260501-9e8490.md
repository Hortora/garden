## Playwright 1.52: `<option>` elements inside `<select>` are never "visible" — `waitFor()` with default state times out even when option is in the DOM

### Stack
Playwright Java 1.52.0 (com.microsoft.playwright:playwright)

### Symptom
`page.locator("#my-select option[value='foo']").waitFor(new Locator.WaitForOptions().setTimeout(5000))` times out with:
```
TimeoutError: Timeout 5000ms exceeded.
- waiting for locator("#my-select option[value='foo']") to be visible
- 15 × locator resolved to hidden <option value="foo">foo</option>
```
The option IS in the DOM — Playwright finds it on every poll — but reports it as "hidden". Tests that passed in older Playwright versions now fail.

### Context
Playwright 1.52 considers `<option>` elements inside a `<select>` as having zero rendered dimensions (because the options are only visible when the dropdown is open in a platform-native UI). The default `waitFor()` state is `visible`, which requires non-zero bounding box. Options never satisfy this condition regardless of whether the parent `<select>` is visible.

This affects any test that waits for a dynamically-added `<option>` to appear in a dropdown — a common pattern when channel lists, session lists, or filter dropdowns are populated via async fetch.

### What was tried
- Verified the option WAS being added to the DOM by `loadChannels()` — the `15 × locator resolved to hidden` message confirmed it
- Checked that the parent `<select>` was visible in the panel — it was
- Verified the channel was being created in `@BeforeEach` — confirmed

### Root cause
Playwright's "visible" check requires the element to have a non-zero bounding box AND not be CSS-hidden. `<option>` elements inside a closed `<select>` have zero rendered dimensions — the native dropdown is closed. Playwright correctly identifies them as having no render box and therefore "hidden".

### Fix

Use `WaitForSelectorState.ATTACHED` instead of the default (`VISIBLE`):

```java
import com.microsoft.playwright.options.WaitForSelectorState;

page.locator("#my-select option[value='foo']").waitFor(
    new Locator.WaitForOptions()
        .setState(WaitForSelectorState.ATTACHED)
        .setTimeout(5000));
```

`ATTACHED` checks only that the element exists in the DOM — it does not require visibility. Use this whenever waiting for `<option>` elements to be added to a `<select>`.

### Why this is non-obvious
The error message says "resolved to hidden" which implies a CSS `display:none` or `visibility:hidden` issue. Developers will look at the parent `<select>`, confirm it is visible, and be confused. The actual issue is that Playwright's bounding-box-based visibility check does not apply cleanly to native select elements. This was not a problem in older Playwright versions where option visibility was handled differently.
