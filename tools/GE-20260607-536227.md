## Problem

When writing a `git rebase -i` todo file manually (via `GIT_SEQUENCE_EDITOR="cp /tmp/plan.txt"`) with full 40-character SHAs, a single wrong character produces:

```
error: could not parse 'cae0e09941b363f56d38f3a1b51ea221af4289a28'
error: invalid line 13: pick cae0e09941b363f56d38f3a1b51ea221af4289a28 <subject>
```

The error gives no indication of *which* character is wrong, what the correct SHA is, or where to look. The rebase exits 1 without running.

## Symptom

`error: could not parse '<SHA>'` / `error: invalid line N: pick <SHA> <message>` when `GIT_SEQUENCE_EDITOR` feeds a pre-written todo file to `git rebase -i`. The rebase enters a detached HEAD state or aborts entirely.

## Root Cause

The full SHA in the todo file was wrong by one character (extra digit: `cae0e09**9**41b...` vs actual `cae0e0941b...`). Git validates every SHA in the todo file before executing any of it. If any SHA fails to resolve to a commit object, git rejects the entire file with a parse error naming the invalid SHA — but provides no diff against what it expected.

## Fix

Always extract full SHAs from `git log` rather than typing them:

```bash
git log --format="%H %s" main..HEAD
```

Copy-paste from that output into the todo file. Never type or truncate full SHAs by hand — even one wrong character is undetectable until the rebase fails.

If the error fires: compare the failing SHA against `git log --format="%H"` output character by character, or simply regenerate the todo file from fresh `git log` output.

### Why this is non-obvious

The error message looks like a parse problem (wrong format, unexpected token) rather than a lookup failure — "could not parse" suggests the SHA string is malformed (wrong length, invalid hex) rather than "this SHA doesn't match any commit object". A developer's first instinct is to check the format (spaces, case, length) rather than to do a character-by-character SHA comparison. The failing SHA is syntactically valid 40-character lowercase hex — it just doesn't exist in the repo.
