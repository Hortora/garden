**Symptom:** PyYAML raises "while scanning a quoted scalar — found unexpected end of stream" at a column in the middle of a string value, not at a `---` boundary.

**Context:** Parsing Markdown files with YAML frontmatter (delimited by `---`) when a title or other value contains the literal string `---` — e.g. an entry whose title describes Jekyll/Kramdown Setext heading detection: *"Kramdown parses paragraph immediately followed by --- as Setext h2 heading"*.

**Root cause:** `content.split('---', 2)` is a naive string split that does not respect YAML quoting. When a quoted value contains `---`, the split fires inside the value, feeding PyYAML a truncated YAML string. The column number in the error points to where the truncated string ends — the middle of the title — not to the `---` that caused the split.

**Fix:**
```python
# Bad — breaks when '---' appears in any quoted YAML value
parts = content.split('---', 2)
yaml.safe_load(parts[1])

# Good — splits on line-boundary document markers only
import re
m = re.match(r'^---\n(.*?)\n---\n?(.*)', content, re.DOTALL)
if m:
    fm = yaml.safe_load(m.group(1))
    body = m.group(2).strip()
```

**Why this is non-obvious:** The YAML value is syntactically valid — it's properly quoted. The mental model is "PyYAML handles quoted strings, so `---` inside quotes is fine". The error message ("unexpected end of stream at column 59") points to the middle of the title string with no indication that a split operation is the cause.
