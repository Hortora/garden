### Symptom
After using `ide_edit_member` with `member` set to the class/interface name (e.g., `member: "PlanItemStore"`), the file contains only the declaration line — all methods, fields, and the closing `}` are lost.

### Context
`ide_edit_member` is designed to replace individual members (methods, fields). When `member` equals the class name, it replaces the class/interface declaration itself. The tool interprets this as "replace the declaration line" and strips everything after it, including the body.

### What was tried (didn't work)
1. Used `ide_edit_member` with `member=PlanItemStore` to update the Javadoc and class declaration of a Java interface — the entire body (8 methods) was removed.
2. IntelliJ JPS `ide_build_project` reported SUCCESS with 0 errors — the cached `.class` files masked the corruption.
3. `ide_diagnostics` on the file also showed 0 errors (stale analysis).

### Root cause
`ide_edit_member` when targeting the class declaration replaces only the declaration, discarding the body. The JPS incremental build does not recompile unchanged-looking files (the declaration line was valid Java), so the old `.class` file persists. Only a full Maven build (via Spotless or `mvn compile`) detects the parse error.

### Fix
Never use `ide_edit_member` with `member=className` when the goal is to modify the class declaration or Javadoc. Instead:
- Use `ide_replace_text_in_file` to target the specific text (Javadoc block, declaration line) without affecting the body.
- After any `ide_edit_member` call that targets the class declaration, always verify the file with `ide_diagnostics` AND a Maven build — do not trust `ide_build_project` alone.

### Why this is non-obvious
The tool name suggests it edits a "member" — but a class IS a member of its compilation unit. The silent success from JPS build means you don't discover the corruption until much later (Spotless, Maven compile, or test failures from missing methods).

### Alternatives considered
- Using `Write` tool to rewrite the entire file — blocked by the `intellij-first-edit.sh` hook which requires IntelliJ MCP tools for `.java` files.
- Using `ide_replace_text_in_file` with regex to match and replace the declaration — this works but regex escaping of Javadoc `{@code}` patterns is fragile.
