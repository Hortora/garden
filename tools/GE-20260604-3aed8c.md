### Symptom

`git push` appears to succeed but logs:

```
remote: This repository moved. Please use the new location:
remote:   https://github.com/<new-org>/<repo>.git
To https://github.com/<old-user>/<repo>.git
   abc1234..def5678  main -> main
```

Push exits 0 — no error, no failed refs. Easy to miss in CI output or scan past in the terminal.

### Context

After a GitHub repository is transferred from a personal account (or org) to
another org, the old URL is kept as a redirect for backwards compatibility.
Any remote still pointing at the old URL continues to work — GitHub silently
proxies the push — but logs the "repository moved" notice on every push.

### Root cause

GitHub's transfer mechanism preserves the old name as a permanent redirect.
Git follows the redirect transparently, so the push succeeds. The notice is
informational only, but the old URL may eventually stop redirecting if the
old account/namespace is reclaimed or the redirect is removed.

### Fix

Update the remote URL to point directly at the new location:

```bash
git remote set-url origin https://github.com/<new-org>/<repo>.git
```

Verify with `git remote -v`.

### Why this is non-obvious

The push exits successfully — there's no broken ref, no error code, no
failed push. The "repository moved" line reads like a GitHub info banner
rather than an action required. Developers familiar with broken remotes
expect either a clean push or a clear error; a redirect-with-success sits
in an unexpected third state that's easy to ignore.
