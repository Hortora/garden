### Symptom

After using `ide_refactor_rename` to rename a Java interface (e.g. `CaseInstanceRepository` → `ReactiveCaseInstanceRepository`), implementation classes that contain the old name as a *substring* get the rename applied inside their name too — producing awkward double-prefix names like `InMemoryReactiveCaseInstanceRepositoryReactive`.

### Context

Happens when:
- Using `mcp__intellij-index__ide_refactor_rename` with `relatedRenamingStrategy: "all"` (default)
- The target is an interface or class whose name appears as a substring in other class names
- The other classes were named by convention (e.g. `InMemory` + interface name + `Reactive` suffix)

### What was tried (didn't work)

1. Single rename pass — the propagation silently renames all classes containing the substring, leaving double-prefix names that compile fine but are misleading
2. Not checking after rename — the awkward names only surface when you inspect the class names or test files that reference them

### Root cause

`relatedRenamingStrategy: "all"` instructs IntelliJ to rename not just the target symbol but also related symbols — subclasses, test classes, and any class whose name contains the renamed identifier as a substring. This is IntelliJ's standard behaviour for its Rename refactoring (Shift+F6), but when invoked via MCP the effect is invisible — you don't see the "also rename these?" dialog.

### Fix

Two approaches:

1. **Use `relatedRenamingStrategy: "none"`** and explicitly rename each related class with a separate `ide_refactor_rename` call. More work but fully controlled.

2. **Use `relatedRenamingStrategy: "all"`** (default) and immediately do a second rename pass on any classes that got awkward names. Check class names with `ide_find_class` after the rename.

### Why this is non-obvious

In the IDE GUI, the rename dialog shows a preview of all affected classes before applying. Via MCP, the rename is applied immediately with no preview — you only discover the propagation by inspecting the result. The `relatedRenamingStrategy` parameter name doesn't hint that "related" includes substring matches in class names, not just type-hierarchy relationships.
