## What it is

SC2 `GAME_EVENTS` (player command events) encode unit tags as a single `Integer` that packs two fields: `tagIndex` (upper 14 bits) and `tagRecycle` (lower 18 bits).

```
tagIndex   = rawTag >> 18
tagRecycle = rawTag & 0x3FFFF
```

The tracker events API (scelight `IBaseUnitEvent`) exposes these as separate `getUnitTagIndex()` / `getUnitTagRecycle()` fields. GAME_EVENTS `TargetUnit.getTag()` returns the packed integer — no helper to split it.

## Context

Tracker events assign unit tags in `"r-{tagIndex}-{tagRecycle}"` string format (when using scelight's `ReplaySimulatedGame` pattern). To match movement orders from GAME_EVENTS (`CmdEvent` / `SelectionDeltaEvent`) to units tracked from tracker events, the packed tag must be decoded to the same format.

## Discovered via

Binary analysis: correlating `CmdEvent.getTargetUnit().getTag()` raw values against known tracker event unit tags for the same unit, then reverse-engineering the bit layout.

## Usage

```java
static String decodeTag(int rawTag) {
    return "r-" + (rawTag >> 18) + "-" + (rawTag & 0x3FFFF);
}
```

`SelectionDeltaEvent.getDelta().getAddUnitTags()` returns `Integer[]` using the same encoding.

## Why this is non-obvious

The scelight API provides no `decodeTag` helper. The packed format is not documented in scelight, s2protocol documentation, or any SC2 modding reference I could find. It must be reverse-engineered from bit patterns.
