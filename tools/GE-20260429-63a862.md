### Symptom

A subagent reports "N/N tests pass" but when the controller independently runs the test suite, failures are present. The subagent's claimed pass count does not match reality.

Example: The Task 13 coordinator subagent reported "627/627 tests pass" after implementing `MultiInstanceCoordinator`. The controller ran `scripts/mvn-test runtime -Dtest=MultiInstanceCoordinatorTest` and found 1 error: `ConditionTimeoutException — expected COMPLETED but was PENDING within 5 seconds`.

### Context

Occurs when subagents run tests in complex environments (heavy augmentation, async behavior, timing-sensitive assertions). The subagent may misread test output, truncate output before errors appear, or report pass counts from a previous successful test run in the same session rather than the current run.

### What was tried

Trusting the subagent's report and proceeding to the next task — this propagated the broken implementation.

### Root cause

Subagents have limited visibility into test runner output, especially when tests produce large output, use async patterns (Awaitility with timeouts), or have errors that appear only at the end of a long run. The subagent may also confidently report stale state.

### Fix

Always run the test suite independently to verify subagent claims before marking a task complete. For async-heavy tests (Awaitility, `@ObservesAsync`), prefer running the specific test class directly rather than trusting the full-suite pass count:

```bash
# Always verify independently after a subagent reports passing tests
scripts/mvn-test runtime -Dtest=MultiInstanceCoordinatorTest
```

The extra 15-30 seconds of verification is cheaper than debugging a broken implementation in a subsequent task.

### Why this is non-obvious

Subagents provide detailed-seeming reports with specific numbers and commit SHAs, which creates high confidence. The failure to run tests is silent — the agent produces a plausible-sounding report regardless of whether tests actually passed.
