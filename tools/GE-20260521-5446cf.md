### What it does

When auditing local branches against an upstream repo, if every branch shows
the same number of commits behind upstream — even branches with wildly different
content — that uniformity is a signal that upstream absorbed all the work via
squash or rebase merges, producing different commit SHAs.

### Context

Running a branch audit across a fork with ~60 local feature branches. Every
branch showed exactly 214 commits behind `upstream/main`, regardless of its
size or age. A developer would investigate each branch individually assuming
the count was coincidental.

### Root cause

When upstream merges PRs via squash or rebase, it creates entirely new commit
SHAs. Git sees the branch commits as "not in upstream" because the SHAs don't
match, even though the work is there. All branches that forked from the same
local main end up behind upstream by the same count.

### Fix

Verify by checking if a key file from a branch exists in upstream:

```bash
git show upstream/main:path/to/key/file.java
```

Or search upstream log for the branch feature by keyword:

```bash
git log --oneline upstream/main | grep -i "feature-keyword"
```

If the file exists and the upstream log has a matching commit, the branch is
stale — its work was squash-merged under a different SHA.

### Why this is non-obvious

The natural assumption is that "N commits behind upstream" is unique per branch
and indicates genuinely missing work. The uniformity of the number is not
obviously a signal — a developer would treat each branch as an independent
case rather than reading the pattern across all of them.
