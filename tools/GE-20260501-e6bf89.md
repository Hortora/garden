## Symptom

A GitHub Actions workflow step exits with code 1 immediately after a `printf` output line, with no error message printed. The step contains an `if/else` error handler that never executes.

## Context

Script attempts to capture the output and exit code of a failing API call:

```bash
printf "   [del] %s ... " "$ver_name"
resp=$(gh api -X DELETE "/orgs/$ORG/packages/maven/$pkg/versions/$ver_id" 2>&1)
rc=$?
if [ $rc -eq 0 ]; then
  echo "ok"
else
  echo "FAILED: $resp"   # ← never reached
fi
```

## Root cause

GitHub Actions bash runs with `set -e` by default. Under `set -e`, any command substitution (`var=$(command)`) that returns a non-zero exit code causes the entire script to exit **immediately at the assignment line** — before `rc=$?` or any `if/else` branch runs. The `printf` line runs (output is flushed), then the script dies silently.

## Fix

Move the command directly into the `if` condition. Commands in `if` test positions are **exempt from `set -e`**:

```bash
printf "   [del] %s ... " "$ver_name"
if resp=$(gh api -X DELETE "/orgs/$ORG/packages/maven/$pkg/versions/$ver_id" 2>&1); then
  echo "ok"
else
  echo "FAILED: $resp"   # ← now reached correctly
fi
```

## Why this is non-obvious

The obvious approach — capture output in a variable, then check `$?` — works fine in an interactive shell or a script without `set -e`. GitHub Actions silently enables `set -e`, which changes the semantics. The symptom (script exits after `printf`, no error message) strongly implies a completely different problem.

### Why this fix

`if condition` syntax is the POSIX-specified way to exempt a command from `set -e`. The assignment approach requires either `|| true` appended (which swallows the error code) or explicit `set +e` / `set -e` bracketing.
