### Symptom
`assert "word" not in skill_text` unexpectedly matches (or fails to match) —
not because the routing logic is wrong, but because:

1. The SKILL.md YAML frontmatter `description:` field contains the same vocabulary as the routing table being tested.
2. Markdown bold markers (`**word**`) make `"word automatically"` fail to match `"**word** automatically"` in the raw string.

### Context
Writing pytest tests that assert routing rules in `work/SKILL.md` — e.g. checking that no routing table row auto-resumes on stack depth 1. The assertion `assert "automatically" not in text` passed the routing table check but still failed because `description: "... routes to the correct work lifecycle skill automatically."` contains the word.

### What was tried
```python
assert "automatically" not in work_text  # fails — matches description field
assert "work-resume automatically" not in work_text  # passes when it should fail — bold markers
```

### Root cause
`SKILL.md` files read as a single string include YAML frontmatter. The `description:` field often shares vocabulary with the routing body. Markdown bold markers (`**text**`) are present in the raw string but invisible when rendered, so `"X automatically"` never matches `"**X** automatically"`.

### Fix
Scope assertions to the specific section being tested:

```python
# Extract only routing table rows (pipe-delimited lines)
table_rows = [line for line in skill_text.splitlines() if "|" in line]
table_text = "\n".join(table_rows)
assert "stack has 1 entry" not in table_text

# Or extract a named step section
idx = skill_text.find("Step 7")
step7_section = skill_text[idx:]
assert "main" in step7_section
```

### Why this is non-obvious
`Path.read_text()` returns the full file including frontmatter — it looks like plain prose but contains YAML fields with similar vocabulary. Markdown bold is invisible in rendered output but present as `**...**` in the raw string, silently breaking substring equality.
