## Symptom

Running a scoped test in a multi-module project:

```bash
mvn test -pl app -am -Dtest=MyServiceTest
```

Fails with `BUILD FAILURE` on an upstream module (e.g. `api/`) that compiles fine but has no tests matching `MyServiceTest`. The error looks like the upstream module broke, not that it simply has no matching tests.

```
[ERROR] Tests run: 0, Failures: 0, Errors: 0, Skipped: 0
[INFO] BUILD FAILURE
[ERROR] No tests were executed!
```

## Context

`-am` (also make) adds all upstream modules to the reactor. Surefire's `-Dtest=` filter applies to **every module** in the reactor. Upstream modules that have no classes matching the filter fail with "no tests found" rather than being skipped silently. This is the default `failIfNoSpecifiedTests=true` behaviour.

## Root Cause

`maven-surefire-plugin` has a `failIfNoSpecifiedTests` property (default `true`) that causes the build to fail when a `-Dtest=` filter matches zero tests in a module. Since `-am` includes upstream modules that genuinely have no matching tests, those modules fail.

## Fix

Add `-Dsurefire.failIfNoSpecifiedTests=false` to suppress the failure on modules with no matching tests:

```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn --batch-mode test \
  -pl app -am \
  -Dtest=MyServiceTest \
  -Dsurefire.failIfNoSpecifiedTests=false
```

## Why This Is Non-Obvious

The error message says "No tests were executed!" on the upstream module, which looks like a test runner misconfiguration or a compile failure — not a filter mismatch. The natural response is to check the upstream module's test setup, not to suppress a surefire flag. The `-am` flag is also commonly used without thinking through its interaction with `-Dtest=`.
