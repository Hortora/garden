**Symptom:** A commit lands on `main` instead of the feature branch. `git branch --show-current` returns `main` even though `git checkout -b <branch>` earlier in the same session printed "Switched to a new branch 'issue-NNN-slug'" and was confirmed successful.

**Context:** Multi-repo workspace workflows where a project repo and a workspace repo are managed concurrently across multiple shell commands and subagent invocations. The branch was created in the project repo with `git -C "$PROJECT" checkout -b $BRANCH`, confirmed with "project branch ok", scaffold committed to the workspace repo — then later in the session the project repo was found on main.

**Root cause:** Unknown — no explicit `git checkout main` was observed. Possible triggers include: tmux pane context reset, shell session reuse, a git hook that fires on workspace operations, or concurrent subagent git operations that implicitly affect the same repo. The switch happens silently with no output or error.

**Detection:** Run `git -C <repo> branch --show-current` before any commit. `git -C <repo> status` also shows the current branch at the top of its output.

**Fix:**
```bash
# Verify before any commit
CURRENT=$(git -C "$PROJECT" branch --show-current)
if [ "$CURRENT" != "$BRANCH" ]; then
  git -C "$PROJECT" checkout "$BRANCH"
fi
```

For work-start / automated workflows, add a branch verification step immediately before the first project commit rather than relying on the confirmation from `checkout -b`.

**Recovery:** If the commit already landed on main:
```bash
git -C "$PROJECT" checkout "$BRANCH"
git -C "$PROJECT" rebase main  # brings main's stray commit into the branch
```
Then optionally reset main to remove the unintended commit (requires push --force if already pushed).

**Why this is non-obvious:** `git checkout -b` prints explicit confirmation ("Switched to a new branch") which creates a strong assumption the state persists. There's no subsequent warning when the branch reverts. The symptom (commit on wrong branch) is only detected after the fact by checking `git log` or noticing the commit is missing from the feature branch.
