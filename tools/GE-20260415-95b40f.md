## git textconv driver makes binary file diffs human-readable — SQLite databases show as SQL dumps in git diff

**ID:** GE-20260415-95b40f
**Stack:** git (all versions with textconv support)
**Labels:** `#strategy` `#git`

**What it achieves:** Binary files tracked in git (SQLite databases, PDFs, compiled assets) show meaningful text diffs in `git diff`, `git show`, and `git log -p` without changing the binary storage format.

**Context:** A SQLite database (`garden.db`) is committed to a git repository. By default `git diff` shows binary noise for every change. With textconv configured, `git diff HEAD garden.db` shows a human-readable SQL dump of exactly which rows changed.

### The technique

**Step 1 — declare the driver in `.gitattributes`** (in the repo root):
```
garden.db diff=sqlite3
```

**Step 2 — configure the driver** (local `.gitconfig` or `~/.gitconfig`):
```ini
[diff "sqlite3"]
    textconv = sqlite3
    cachetextconv = true
```

`textconv` names the command git runs on the file before diffing. `sqlite3` called with a single filename argument outputs a `.dump` — a plain-text SQL representation of all rows. `cachetextconv = true` caches the conversion so repeated diffs are fast.

**Result:**
```
-INSERT INTO checked_pairs VALUES('GE-0001 × GE-0002','distinct','2026-04-15','');
+INSERT INTO checked_pairs VALUES('GE-0003 × GE-0004','related','2026-04-15','cross-ref added');
```

The same approach works for any binary format where a command-line tool can produce text output: PDFs (`pdftotext`), Word documents (`docx2txt`), images (exiftool for metadata), compiled protobufs, etc.

### Alternatives considered
- **Export to a companion text file on every write** — two sources of truth; they diverge. The textconv approach has a single binary source with on-demand text conversion.
- **Don't commit the binary; use a migration file instead** — loses the convenience of git as the sync and audit mechanism.
- **Use a text-based format instead** — not always possible (SQLite is the right tool for the job).

### Why this is non-obvious
`textconv` is documented in `gitattributes(5)` but under "Generating diff text" — a section most developers never read. The common knowledge is "binary files show as binary in git diff; that's a limitation of git." It isn't — it's just an unconfigured option. The `cachetextconv` sub-option is further buried and easy to miss, making the first use slow enough to seem broken.

### When to use it
- Any binary file tracked in git where audit trail matters (databases, compiled configs)
- Team workflows where reviewers need to understand what changed in a binary artifact
- CI pipelines that run `git diff` to detect changes and need to inspect content
