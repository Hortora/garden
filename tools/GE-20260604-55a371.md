## Symptom

After running `git filter-repo` on an epic branch to strip workspace artifact paths, `git rebase main epic-branch` (or `git checkout main && git rebase epic-branch`) produces hundreds of warnings:

```
warning: skipped previously applied commit 2066d7b
warning: skipped previously applied commit 10dc505
...
warning: skipped previously applied commit b6081f6
hint: use --reapply-cherry-picks to include skipped commits
```

Then hits a conflict on an unrelated file (e.g. CLAUDE.md), even though that file was not touched by the epic branch.

## Context

Running `work-end` on an epic branch after `git filter-repo --invert-paths --prune-empty` was used to strip workspace artifact files (HANDOFF.md, design/.meta, plans/). The branch had been on top of `origin/main` which was ahead of `upstream/main` by 119 commits.

## What was tried

`git checkout main && git rebase issue-246-arc42stories-migration` — produced the symptom above with 119+ skipped warnings and a CLAUDE.md conflict.

## Root cause

`git filter-repo --prune-empty` removed a commit that became empty after stripping (e.g. a commit that only touched HANDOFF.md). When this commit is at the base of the epic branch, the branch's ancestry shifts back to the commit *before* the pruned one. The epic branch now diverges from `origin/main` at an earlier ancestor than expected.

When git tries to rebase from that earlier ancestor, it sees all 119 commits that were already on `origin/main` (between the new divergence point and the old one) as needing to be replayed — which it then correctly identifies as "skipped previously applied" and tries to apply anyway, causing conflicts.

## Fix

Do not use `git rebase` after filter-repo has pruned a base commit. Instead, cherry-pick the specific commits by SHA range:

```bash
# Get the oldest and newest SHA in the compacted epic branch
FIRST_SHA=$(git log --format="%H" HEAD~N..HEAD | tail -1)  # N = count of epic commits
LAST_SHA=$(git rev-parse HEAD)

git checkout main
git cherry-pick "${FIRST_SHA}..${LAST_SHA}"
# Also cherry-pick FIRST_SHA itself if it was the spec/first commit:
git cherry-pick "$FIRST_SHA"
```

Cherry-pick applies each commit independently of ancestry — it does not scan the shared history between the two branches, so it only applies exactly the commits you name.

## Why this is non-obvious

The rebase error messages ("skipped previously applied commit") suggest the commits already exist on the target branch — which is accurate — but the root cause is not the commits themselves. It is the filter-repo operation shifting the branch's divergence point by removing one ancestor commit. The fix (cherry-pick) is the correct tool for "apply these specific commits onto this branch" without caring about the ancestry relationship.
