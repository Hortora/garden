## Symptom

`gh api graphql -f query='mutation($p:ID!,$i:ID!){addProjectV2ItemById(input:{projectId:$p,contentId:$i}){item{id}}}' -f p="..." -f i="..."` returns a full GitHub HTML "unicorn" 500 error page instead of JSON.

The identical mutation with proper whitespace/newlines succeeds and returns JSON.

## Context

Attempting to compact a working GraphQL mutation into a single line for use inside a shell loop or one-liner.

## What was tried (didn't work)

Removing all whitespace from the query body to make it a one-liner — this triggers the HTML error response.

## Root cause

GitHub's GraphQL endpoint (or `gh`'s request serialisation) does not handle queries with zero internal whitespace correctly. The exact failure mode is not documented; it appears to be a parsing or routing issue on GitHub's side triggered by the compact format.

## Fix

Always include whitespace in GraphQL queries passed to `gh api graphql`. The minimal working format:

```bash
gh api graphql -f query='
mutation($project: ID!, $item: ID!) {
  addProjectV2ItemById(input: {projectId: $project, contentId: $item}) {
    item { id }
  }
}' -f project="$PROJECT_ID" -f item="$node_id"
```

## Why this is non-obvious

The same query content works when formatted and fails when compacted — whitespace should be semantically irrelevant in GraphQL. The failure returns an HTML page (not a JSON error), which looks like a network or auth failure rather than a query formatting issue.
