## GitHub Packages Maven registry does not follow repo name redirects

**ID:** GE-20260619-31e6e4
**Stack:** GitHub Actions, Maven deploy plugin 3.1.4, GitHub Packages
**Symptom:** `mvn deploy` fails with `Could not find artifact io.example:my-artifact:pom:0.2-SNAPSHOT in github (https://maven.pkg.github.com/org/old-repo-name)` after a successful `mvn verify`. The artifact URL in the error contains the old (pre-rename) repo name.
**Context:** Occurs when a GitHub repo has been renamed (e.g. `casehub-workers` → `workers`) and `<distributionManagement>` in pom.xml still references the old name. GitHub's web UI, REST API, and `git clone` all follow the redirect transparently — the Maven registry does not.

### What was tried (didn't work)
- Assumed the redirect would work since `gh repo view org/old-name` resolves correctly
- Checked GitHub Actions permissions — `packages: write` was correctly set
- Verified `GITHUB_TOKEN` had the right scopes — auth was fine

### Root cause
GitHub Packages Maven registry resolves repository URLs literally. When `maven-deploy-plugin` PUTs an artifact to `https://maven.pkg.github.com/org/old-name`, GitHub returns a success (the redirect catches the write). But when the plugin immediately GETs the artifact back to verify the upload, the redirect does not produce a valid Maven response — the artifact appears missing.

### Fix
Use the canonical repo name in all `<distributionManagement>` URLs. Find the canonical name with:
```bash
gh repo view org/repo-name --json name --jq '.name'
```
Then update pom.xml:
```xml
<distributionManagement>
    <repository>
        <id>github</id>
        <url>https://maven.pkg.github.com/org/canonical-name</url>
    </repository>
</distributionManagement>
```

### Why this is non-obvious
Every other GitHub surface — web UI, API, git operations — follows repo name redirects silently. A developer who renames a repo has no reason to expect that Maven deploy would break, especially since the error message ("Could not find artifact") suggests a permissions or configuration problem rather than a naming problem. The redirect alias works for `git push`, `gh issue`, and even `mvn verify` (which only reads) — only `mvn deploy` (which writes then reads back) fails.
