## Technique

When you have a large single commit that needs to be split into stacked PRs for review,
`git diff` + `git apply` is cleaner than interactive rebase for file-level splits.

## When to Use

- You have one large commit (or want to ignore intermediate commits) and need to split
  by file group rather than by chronology
- Interactive rebase would require resolving conflicts across many commits
- You want a clean single-commit PR per logical group

## Steps

```bash
# 1. Create fresh branch from the target base
git checkout upstream/main -b feat/pr-1-async-change

# 2. Generate a patch containing only the files for this PR
git diff upstream/main feat/big-branch -- \
  api/src/main/java/LoopControl.java \
  engine/src/main/java/ChoreographyLoopControl.java \
  > /tmp/pr1.patch

# 3. Apply the patch
git apply /tmp/pr1.patch

# 4. Commit and push
git add -A && git commit -m "feat: async LoopControl"
git push origin feat/pr-1-async-change

# 5. Repeat for PR 2 (stacked on PR 1's branch or on main)
git checkout upstream/main -b feat/pr-2-data-model
git diff upstream/main feat/big-branch -- casehub-blackboard/src/main/java/ > /tmp/pr2.patch
git apply /tmp/pr2.patch
```

## Why This Is Non-Obvious

Most developers reach for `git rebase -i` to split commits, which works chronologically but
becomes painful when the large commit contains files that belong to multiple logical groups.
`git diff <base> <branch> -- <files>` extracts a precise file-scoped patch regardless of
commit history, and `git apply` cleanly applies it to any target branch. No conflict resolution
needed if the target branch is clean.

## Constraints

- Works best when the split is by file group (not by line-level hunks within the same file)
- If two PRs modify the same file, extract one first and the other will get a smaller diff
- The patch includes deletions — useful for removing old files as part of a redesign PR
