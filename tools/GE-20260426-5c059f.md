### What and why

When iterating over a JSON array in bash and needing multiple fields per element, the common approach is to `@base64`-encode each object and decode it in the loop. This breaks silently for large objects because `jq`'s `@base64` wraps at 76 chars and `read -r` only reads one line.

`@tsv` is a robust alternative: extract only the fields you need as tab-separated values, then split on `$'\t'` with `read -r`.

### The technique

```bash
# Extract specific fields as TSV — one object per line, fields tab-separated
ITEMS=$(gh api "/repos/ORG/REPO/pulls?state=open&per_page=100" \
  --jq '.[] | [.number, .title, .user.login, .head.sha, .html_url] | @tsv' \
  2>/dev/null | tr -d '\000-\010\013-\037\177' || true)

[ -z "$ITEMS" ] && continue   # skip if no results

while IFS=$'\t' read -r pr_number pr_title pr_author pr_sha pr_url; do
  [ -z "$pr_number" ] && continue
  # use variables directly — no base64 decode, no subshell jq calls
  echo "$pr_number: $pr_title by $pr_author"
done <<< "$ITEMS"
```

### Key properties of @tsv

- Field delimiter is a **literal tab** — safe as a record separator
- Tabs **within** field values are escaped as `\t` (two-char sequence), not literal tabs — the field split on `$'\t'` is safe
- Newlines within values are escaped as `\n`
- Handles numbers, booleans, and strings natively — no `| tostring` needed for numbers
- Never wraps output — no line-length issues regardless of field length

### Why a skilled developer wouldn't naturally reach for this

The dominant pattern in shell jq tutorials is `@base64` for whole-object iteration. The `@tsv` approach requires knowing upfront which fields you need, which feels like a constraint — but in practice you always know which fields you need in the loop body. The tradeoff (declare fields upfront, avoid base64/decode overhead and line-wrap bugs) is almost always favourable.

### Caveats

- Add `| tr -d '\000-\010\013-\037\177'` after the jq call to strip control characters from API responses — emoji and special chars in titles can break JSON parsing in `gh api`
- Always add `|| true` at the end to prevent `set -e` from exiting on empty results
- `[ -z "$pr_number" ] && continue` guards against empty trailing lines in the heredoc
