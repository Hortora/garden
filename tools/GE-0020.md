## Use `typora-root-url` in YAML front matter to make Jekyll root-relative image paths work in Typora

**ID:** GE-0020
**Stack:** Typora (any version), Jekyll / GitHub Pages
**Labels:** `#blogging` `#tooling`
**What it achieves:** Image paths using Jekyll's root-relative convention (`/assets/blog/image.jpg`) render correctly in both Typora's local preview and on GitHub Pages, with no path duplication or conditional logic.
**Context:** Writing Jekyll documentation or blog posts in Typora. Jekyll resolves `/assets/...` from the site root; Typora resolves paths from the file's directory — causing a mismatch when the file is in a subdirectory like `docs/blog/`.

### The technique

Add `typora-root-url` to the YAML front matter of the markdown file, pointing to the repository root relative to the file's location:

```yaml
---
typora-root-url: ../..
---
```

Then use root-relative paths for all images in the file:

```markdown
![caption](/assets/blog/image.jpg)
```

Typora reads `typora-root-url: ../..`, resolves `/assets/blog/image.jpg` as `{file_dir}/../../assets/blog/image.jpg` (i.e., the repo root), and renders the image correctly in its local preview.

Jekyll ignores `typora-root-url` (it's not a known front matter key) and resolves `/assets/blog/image.jpg` from the site root as normal.

The same path works in both environments.

### Why this is non-obvious

The typical workaround is to use relative paths (`../../assets/blog/image.jpg`) which work in Typora but may break in Jekyll depending on where the page is rendered in the URL structure. The root-relative path (`/assets/...`) is the correct Jekyll convention, but breaks in Typora without the `typora-root-url` hint.

Most developers either accept broken previews in Typora, use relative paths and hope Jekyll renders correctly, or maintain two copies of the path. The `typora-root-url` front matter key solves it cleanly — one path, both tools.

### When to use it
- Any Jekyll project where you want to preview markdown in Typora with images rendering correctly
- The `typora-root-url` path should point to the repository root: count the directory depth of the file from root and add `../` accordingly. A file in `docs/blog/` needs `../..`; a file in `docs/` needs `..`

### Caveats
- Jekyll ignores unknown front matter keys, so `typora-root-url` has no effect on the rendered site
- Click-to-open links (href) on images are not resolved via `typora-root-url`
- Documented on the Typora support site but not widely known
