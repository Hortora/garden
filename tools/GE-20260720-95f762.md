### Symptom
A commit (`StringExpressionEvaluator`) was present on a branch but missing from main.
The branch appeared complete — stamped with `chore: branch closed — landed as <SHA> on main`.

### Context
Squash-to-main workflow: branch work is squashed into a single commit on main, then the
branch is stamped with an empty commit recording the landing SHA. The stamp signals
"this branch's work reached main."

### What was tried (didn't work)
Nothing — the orphaned commit was invisible until a cross-repo review flagged the missing
interface months later.

### Root cause
The commit was added to the branch *after* the squash had already landed on main. The
branch-closed stamp was then applied on top, making the branch look complete. The stamp
references the landing SHA, but nothing validates that all branch commits are ancestors
of that SHA.

### Fix
Never commit to a branch after its work has been squashed to main. If late work surfaces:
- Cherry-pick it directly to main, or
- Open a new branch for it

If you suspect orphaned work, compare the branch tip against the landing SHA:
```bash
git log <landing-SHA>...<branch> --oneline
```
Any commits listed are orphaned — they exist only on the branch.

### Why this is non-obvious
The branch-closed stamp creates a false sense of completeness. The stamp message says
"landed as <SHA>" which reads as "everything here reached main." There is no validation
that this is actually true — and the stamp itself is an empty commit that obscures the
orphaned commit in `git log`.
