## Symptom

After rewriting history with `git filter-repo` and swapping branch labels, `git push --force-with-lease origin main` fails:

```
! [rejected] main -> main (stale info)
error: failed to push some refs to '...'
```

Even though you're the only person who has touched this branch.

## Context

`--force-with-lease` is designed to protect against overwriting someone else's pushed commits. It checks the remote-tracking ref (`origin/main`) to confirm it matches what you last fetched. After a local history rewrite (via filter-repo or interactive rebase), your local remote-tracking ref still points to the pre-rewrite tip — the new local `main` has a completely different SHA chain. `force-with-lease` reads this as "someone else pushed to origin" and rejects.

## Root cause

`--force-with-lease` compares the local remote-tracking ref (`refs/remotes/origin/main`) against the actual remote. After a local rewrite, your remote-tracking ref is stale but you haven't fetched, so the guard fires falsely.

## Fix

Use `--force` instead of `--force-with-lease` after a local history rewrite:

```bash
git push --force origin main
```

`--force` is appropriate here because the rewrite is intentional and you know the remote history is being replaced.

## Why this is non-obvious

`--force-with-lease` is widely recommended as the "safe force push" alternative. Developers use it by habit even in situations where a straight `--force` is actually the correct tool (deliberate local history rewrite with no concurrent collaborators).
