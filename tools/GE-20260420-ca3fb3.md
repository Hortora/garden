**Symptom:** Test passes or asserts on the wrong state; no error raised.

**Stack:** Python, pytest

**Context:** A test corrupts an entry by replacing a known valid value with an invalid one. The valid value is retrieved as `list[0]` from a registry (e.g. `GARDEN_TYPES[garden]['valid_types'][0]`), then used in `str.replace()`.

### Root cause

`list[0]` assumes the first element of the list matches what actually appears in the fixture string. If list order ever changes (after a refactor, sort, or config update), `str.replace()` silently returns the original string unchanged — no exception, no warning, no indication anything went wrong. The test continues with an uncorrupted fixture.

### Fix

Scan the list to find which element actually appears in the string:

```python
original_type = next(
    t for t in GARDEN_TYPES[garden]['valid_types']
    if f'type: {t}' in fixture
)
entry = fixture.replace(f'type: {original_type}', f'type: {wrong_type}')
```

### Why this is non-obvious

`str.replace()` never raises when the substring isn't found — it returns the original string silently. The bug only surfaces when list order changes, which may be a future refactor entirely unrelated to this test, making the cause temporally distant from the symptom. Automated test runs continue to pass throughout, giving no signal that the test stopped doing anything meaningful.

### Alternatives considered

- Hardcoding the expected string directly — works but breaks if the fixture changes
- Asserting the replace actually changed the string before proceeding — defensive but adds noise
