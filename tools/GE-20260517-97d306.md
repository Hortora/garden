## Symptom

Files are moved using shell `mv` and the new location is staged with `git add db/migration/qhorus/`. The commit succeeds. But the original files are still tracked — they appear as ` D` (unstaged deletions) in `git status`. The next reader of the repo (or the CI build) still sees both copies in the artifact.

## Context

Moving multiple files into a new subdirectory using shell `mv`, then staging only the destination with `git add <newdir>/`. Common when relocating resource files (SQL migrations, config files) into a newly-created subdirectory.

## What was tried

`git add db/migration/qhorus/` after `mkdir -p db/migration/qhorus && mv db/migration/V*.sql db/migration/qhorus/`. The staging area shows the new files as added, so the commit looks complete. It isn't.

## Root cause

`git add <newdir>/` stages new content at `<newdir>/` but does not stage the removal of the original paths. Git tracks the original files separately — they are not automatically unstaged by moving them at the filesystem level. Until `git rm <original-files>` is run (or `git add -A` stages all changes), the old paths remain tracked at their original locations.

## Fix

Either use `git mv` for each file (atomic rename), or follow up the shell `mv` with explicit deletion staging:

```bash
# Option 1 — git mv (atomic, for individual files)
git mv runtime/src/main/resources/db/migration/V1__schema.sql \
       runtime/src/main/resources/db/migration/qhorus/V1__schema.sql

# Option 2 — shell mv, then stage all changes
mkdir -p db/migration/qhorus
mv db/migration/V*.sql db/migration/qhorus/
git add -A   # stages both the additions AND the deletions

# Option 3 — shell mv, then explicit git rm
mv db/migration/V*.sql db/migration/qhorus/
git add db/migration/qhorus/
git rm db/migration/V1__schema.sql db/migration/V2__schema.sql ...
```

Verify with:
```bash
git ls-files db/migration/ | grep -v qhorus   # should produce no output
```

## Why this is non-obvious

`git add <dir>/` looks like it should handle "everything in this directory" — and it does, but it only sees the additions, not the missing originals. The original paths are absent from `<dir>/` so `git add <dir>/` has nothing to delete. The staging area shows only new additions, giving a false sense that the move is complete.

### Alternatives considered

- `git add -A` works but may accidentally stage unrelated changes in a large working tree
- `git mv` requires one invocation per file — verbose for many files, but unambiguous
