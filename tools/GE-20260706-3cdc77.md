### Symptom
Called `ide_refactor_rename` targeting column 51 on a constructor declaration line
(`public ReadOnlyLayerException(String panelName)`) to rename the parameter
`panelName` → `layerName`. Instead, the class `ReadOnlyLayerException` was renamed
to `layerName` — breaking compilation across 5 files.

### Context
Renaming constructor parameters as part of a large terminology migration
(panel → layer). The rename was one of many IntelliJ refactoring calls in the
session — all others worked correctly because they targeted unambiguous symbols
(class declarations, method names, fields).

### What was tried (didn't work)
- `ide_refactor_rename` with `targetType: symbol`, `line: 19`, `column: 51` — the
  column was well past the class name, squarely on the parameter token. Expected the
  parameter to be renamed; instead the class was renamed.

### Root cause
Constructor declarations repeat the class name (`public ClassName(String param)`),
so the same line contains both the class identifier and parameter identifiers.
IntelliJ's PSI resolution picked the class name token rather than the parameter
token at the given column position.

### Fix
Use the Edit tool for parameter renames when the parameter is on a constructor
declaration line. Reserve `ide_refactor_rename` for class/method/field renames
where the symbol is unambiguous at the target position.

```java
// Instead of ide_refactor_rename targeting the parameter column:
Edit({
  file_path: "ReadOnlyLayerException.java",
  old_string: "public ReadOnlyLayerException(String panelName) {",
  new_string: "public ReadOnlyLayerException(String layerName) {"
})
```

### Why this is non-obvious
The API accepts line+column and the column was well past the class name — you'd
expect it to resolve the parameter token, not backtrack to the class. No error or
warning is given; the rename silently succeeds on the wrong symbol. The result
message says "Successfully renamed 'ReadOnlyLayerException' to 'layerName'" which
is only visible after the damage is done.

### Alternatives considered
- Reading the file first and computing the exact column offset for the parameter
  token — fragile and error-prone when whitespace varies
- Using `ide_find_symbol` to get the exact position — adds a round-trip but still
  subject to the same PSI resolution ambiguity
