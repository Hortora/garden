## Symptom

GitHub Actions CI fails with HTTP 401 or silent authentication errors when Maven tries to pull artifacts from GitHub Packages, even though `GITHUB_TOKEN` is set and passed correctly.

## Context

A Maven pom has two repository-related declarations with different `<id>` values:
- `<distributionManagement>` with `<id>github</id>` — for publishing
- `<repositories>` with `<id>github-casehubio</id>` — for pulling a shared BOM

The `setup-java` action is configured with `server-id: github`, which creates a single Maven settings.xml server entry under the id `github`. Maven uses the `<id>` value to look up credentials in settings.xml — if the repository id in the pom doesn't match a server entry, Maven uses anonymous access and GitHub Packages returns 401.

## Root cause

`setup-java`'s `server-id` parameter creates exactly one server entry in `~/.m2/settings.xml`. Any `<repository>` or `<distributionManagement>` entry in the pom with a *different* id gets no credentials at all. Maven does not propagate credentials across server ids.

## Fix

Align all repository ids in the pom with the `server-id` value that `setup-java` creates. If `server-id: github`, rename every `<id>` in `<repositories>` and `<distributionManagement>` to `github`:

```xml
<!-- In pom.xml — all ids must match setup-java's server-id -->
<repositories>
  <repository>
    <id>github</id>  <!-- was: github-casehubio -->
    <url>https://maven.pkg.github.com/myorg/*</url>
    <snapshots><enabled>true</enabled></snapshots>
  </repository>
</repositories>

<distributionManagement>
  <repository>
    <id>github</id>
    <url>https://maven.pkg.github.com/myorg/myrepo</url>
  </repository>
</distributionManagement>
```

In the CI workflow, the existing single `server-id: github` then covers both pulling and publishing.

## Why this is non-obvious

GitHub's documentation for GitHub Packages + Maven focuses on the authentication mechanism (PAT / GITHUB_TOKEN) but does not clearly explain that the server `<id>` in settings.xml must exactly match the repository `<id>` in the pom. The 401 error looks like a token permission problem rather than an id mismatch problem.

### Why this fix

Renaming the repository id is a one-line change per pom section and requires no changes to the CI workflow. The alternative — writing a full custom settings.xml with multiple server entries — is more verbose and harder to maintain, and bypasses setup-java's built-in caching and Java-setup integration.
