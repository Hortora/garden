### Symptom
Path smoothing using Bresenham line-of-sight declares a segment clear, but the unit moving along
that segment at a fractional speed (e.g. 0.5 tiles/tick) lands in a wall tile and triggers
wall enforcement — causing an infinite path-invalidation and recompute loop. The unit appears
frozen. No error is thrown.

### Context
Triggered when:
- A* produces waypoints at tile centres (e.g. `(12.5, 18.5)`)
- Smoothing jumps from an above-wall waypoint (e.g. tile 14,19) directly to a below-wall
  waypoint (e.g. tile 8,8) because the Bresenham line does not cross any WALL tile centres
- But stepToward at speed=0.5 per tick reaches tile (14,18) — a WALL — before it can reach
  tile (13,18) that the Bresenham line "crosses"

The Bresenham algorithm checks whether the **centre-to-centre line** crosses wall tiles.
Continuous movement at discrete time steps does NOT follow this exact line — at each tick
the position is `current + normalized_direction * speed`. This creates a different set of
intermediate tiles than Bresenham checks.

### Root cause
Bresenham LOS is tile-granular: it checks at integer tile-boundary crossings along the
theoretical line. Movement is sub-tile: each step of `speed` units may land in a tile that
the centre line skips over entirely, especially on near-horizontal or near-vertical diagonal
movements where the step overshoot can be up to `speed - epsilon` tiles wide.

### Fix
Replace Bresenham LOS with **sub-tile sampling**: step along the normalized direction vector
at intervals ≤ `movement_speed`, checking `isWalkable()` at each sample point.

```java
private static boolean hasLos(TerrainGrid grid, Point2d from, Point2d to) {
    double dx   = to.x() - from.x();
    double dy   = to.y() - from.y();
    double dist = Math.sqrt(dx * dx + dy * dy);
    if (dist < 0.001) return grid.isWalkable((int) from.x(), (int) from.y());
    double nx = dx / dist;
    double ny = dy / dist;
    double step = 0.4; // below movement speed (0.5) — catches all wall crossings
    for (double d = 0; d <= dist + step; d += step) {
        double t = Math.min(d, dist);
        if (!grid.isWalkable((int)(from.x() + nx * t), (int)(from.y() + ny * t))) return false;
    }
    return true;
}
```

Use `step = 0.4` when movement speed is `0.5`. The step must be strictly less than the
movement speed so that no wall tile can be skipped between consecutive sample points.

### Why this is non-obvious
Bresenham LOS is the standard algorithm taught for grid pathfinding, and it correctly
determines whether a straight line in an integer grid crosses any obstacle. Most developers
assume "LOS clear" implies "movement is safe". The mismatch is invisible until you have
both: (1) smoothing that makes large jumps and (2) wall enforcement that catches fractional
tile positions.
