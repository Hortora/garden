### Symptom

All `mcp__intellij-index__*` tools (`find_class`, `find_symbol`, `search_text`, `find_file`) return zero results — empty arrays, no errors. `ide_index_status` reports `isDumbMode: false, isIndexing: false`. The project appears open and indexed, but the index contains nothing.

Both `mcp__intellij-index__*` and `mcp__intellij__*` servers are affected — both return empty results for the same queries.

### Context

DraftHouse has a `.idea` directory at `/Users/mdproctor/claude/casehub/drafthouse/` (the workspace root containing HTML/CSS/JS). The Maven multi-module project root is at `server/` (`/Users/mdproctor/claude/casehub/drafthouse/server/pom.xml`). When `ide_open_project` opens the parent directory, IntelliJ loads a project with no Maven structure — the Java indexer has nothing to index.

### What was tried (didn't work)

1. `ide_sync_files` on the project — returned success but indexes stayed empty
2. Waiting for indexing to complete — `isDumbMode: false` already, nothing to wait for
3. Using `mcp__intellij__*` (built-in MCP) as fallback — also returned empty results
4. Multiple queries across different search types (find_class, find_file, search_text) — all empty

### Root cause

IntelliJ opened the directory as a "workspace" project (HTML/CSS/JS files) rather than a Maven project. The Java indexer had no source roots to scan. The `.idea` directory at the parent level made the project openable — but openable ≠ indexable for Java.

### Fix

Use `server/` as the `project_path`, not the parent directory:

```java
// ❌ Empty index — .idea exists but no Maven structure
mcp__intellij-index__ide_open_project(path: "/path/to/drafthouse")
mcp__intellij-index__ide_find_class(query: "DebateSession", project_path: "/path/to/drafthouse")
// → returns zero results

// ✅ Full Java index — Maven reactor pom.xml lives here
mcp__intellij-index__ide_open_project(path: "/path/to/drafthouse/server")
mcp__intellij-index__ide_find_class(query: "DebateSession", project_path: "/path/to/drafthouse/server")
// → returns all matching classes
```

### Why this is non-obvious

The project opens without error. `ide_index_status` reports healthy state. There is no indication that the index is empty — the tools just return empty arrays, which looks like "nothing matches" rather than "nothing was indexed." The `.idea` directory at the parent level is a convincing signal that the project path is correct. The failure is silent.

### Alternatives considered

- Using bash grep/find as fallback — works but loses all semantic intelligence (references, implementations, type hierarchy)
- Opening both parent and server/ projects — wasteful, server/ alone is sufficient
