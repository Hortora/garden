## Hortora garden pre-commit hook blocks all commits when untracked GE-*.md files exist

**Stack:** Hortora garden agent, git
**Symptom:** `git commit` fails with "Commit blocked: untracked garden entries must be committed first" even when the commit being attempted has nothing to do with garden entries — e.g. a GARDEN.md index update or a domain field fix.
**Context:** Any machine with the Hortora garden agent installed via `garden-agent-install.sh`. The pre-commit hook scans the entire working tree for untracked GE-*.md files and blocks ALL commits until they are cleared.

### What was tried (didn't work)
- Staging only the intended files (`git add GARDEN.md`) and committing — hook fires regardless of what's staged

### Root cause

`garden-agent-install.sh` installs a pre-commit hook that runs before every commit in the garden repo. It scans for untracked files matching the GE-YYYYMMDD pattern and exits 1 if any are found — intent is to prevent entry files from being left uncommitted and lost by a branch switch. But it blocks ALL commits, not just entry submissions, until the untracked entries are gone.

### Fix

Find what the hook is blocking on and commit those entries first:

```bash
GARDEN="${HORTORA_GARDEN:-$HOME/.hortora/garden}"
git -C "$GARDEN" ls-files --others --exclude-standard | grep -E "GE-[0-9]"
# Stage and commit the listed files, then retry your original commit
```

### Why this is non-obvious

A pre-commit hook is expected to inspect staged changes, not the entire working tree. The hook names the blocking files in its error message — which helps — but the connection between those untracked entries and an unrelated commit (e.g. a GARDEN.md index update) isn't obvious. Developers naturally assume staged-only changes are sufficient.
