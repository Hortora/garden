## Symptom

Running `git rm -r <directory>` or `git rm --force <directory>` fails with:

```
fatal: could not lookup name for submodule '<directory>'
```

The directory exists on disk and appears in `git ls-files` output, but there is no `.gitmodules` file in the repository.

## Context

Occurs when a directory was previously added as a git submodule, then the `.gitmodules` file was deleted (or never committed) but the gitlink entry (`mode 160000`) remains in the index. Git treats the entry as a submodule and requires `.gitmodules` metadata to process `git rm`.

## Diagnosis

Confirm with:
```bash
git ls-files --stage <directory>
# Output: 160000 <sha> 0	<directory>
```

Mode `160000` = gitlink (submodule reference). Mode `100644` or `100755` = regular file.

## Fix

Remove the orphaned gitlink directly from the index, then delete from the filesystem:

```bash
git rm --cached <directory>
rm -rf <directory>
git commit -m "chore: remove orphaned submodule entry"
```

`--cached` removes from the index only, bypassing the submodule name lookup that `git rm` requires.

## Why this is non-obvious

`git rm --force` is documented as bypassing safety checks. It still fails for orphaned submodules because the name lookup happens before the force flag is evaluated. The error message says "could not lookup name" which sounds like a permissions or config problem, not an index-only fix.
