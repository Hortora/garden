When `mcp__intellij-index__ide_diagnostics` is called on a file that is not currently open in an IntelliJ editor tab, it runs in degraded batch analysis mode. The result includes 100+ phantom errors including "Cannot resolve symbol 'java'", "Cannot resolve symbol 'String'", "Diamond types are not supported at language level '5'" — errors that don't exist in the actual code. The `analysisMessage` field reads "Intentions are unavailable because the file is not open in an editor." The same file with zero real errors shows `problemCount: 100` in batch mode.

### Symptom

`ide_diagnostics` returns dozens of "Cannot resolve symbol" errors for basic Java types (`java`, `jakarta`, `String`, `UUID`) that are definitely on the classpath.

### Root cause

IntelliJ uses its full semantic analysis engine when a file is open in an editor tab (with access to the module's resolved classpath and JDK). When closed, it falls back to a lighter batch analysis mode that lacks full classpath resolution.

### Fix

Always call `mcp__intellij__open_file_in_editor` on the target file before calling `ide_diagnostics`:

```python
mcp__intellij__open_file_in_editor(filePath="path/to/File.java", projectPath="/path/to/project")
mcp__intellij-index__ide_diagnostics(file="path/to/File.java", project_path="/path/to/project", severity="errors")
```

With the file open, `analysisMessage` is `null` and only real errors appear.

### Why this is non-obvious

The degraded mode produces a huge volume of plausible-sounding errors ("cannot resolve symbol") that look like real compilation failures. Nothing in the tool response indicates the root cause is the file not being open — the `analysisMessage` field exists but is easy to miss. A developer would spend significant time believing the code is broken.
