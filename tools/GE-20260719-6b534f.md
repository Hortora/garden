### Symptom
Git sync commands fail or push to the wrong remote. Fork/main diverges from
blessed/main despite work-start running sync logic. Force-push required at
work-end.

### Context
Project CLAUDE.md declared `origin = casehubio/casehub-pages (blessed repo)`
and `fork = mdproctor/casehub-pages`. Actual `git remote -v` showed origin
pointing to the fork and `upstream` pointing to the blessed repo — standard
GitHub CLI convention.

### What was tried (didn't work)
Trusted the documented labels. Built sync commands using `origin` as the
blessed remote. Push to `fork` remote name (which was a redundant alias for
origin). The sync appeared to succeed but accomplished nothing — pushed from
fork to fork.

### Root cause
Two common git remote-naming conventions exist and project docs can silently
adopt one while the actual config uses the other:
- **GitHub CLI default:** origin=personal fork, upstream=blessed
- **Alternative:** origin=blessed, fork=personal

No git command validates remote labels against documentation. No error
surfaces when a command targets the wrong remote by name — the push succeeds
to whichever URL the name actually resolves to.

### Fix
Always verify remote URLs before trusting documented labels:
```bash
git remote -v
```

In automation (skills, scripts), detect the convention dynamically:
```bash
git remote get-url upstream 2>/dev/null && echo "has-upstream"
git remote get-url fork 2>/dev/null && echo "has-fork"
```

Then branch on which remote exists rather than assuming a naming convention.

### Why this is non-obvious
Remote names are arbitrary strings — git enforces no convention. Documentation
that maps names to repos looks authoritative but has no verification mechanism.
The failure is silent: commands succeed, just against the wrong target. The
divergence only surfaces later as unexpected force-push requirements.
