### Symptom
A branch whose entire content is on main shows in `git branch --no-merged main` — i.e. git says it is not merged — even though every file change has been applied.

### Context
Branch was integrated into main via `git cherry-pick` rather than `git merge`. This is common in concurrent-epic workflows where two branches modify the same files and cherry-pick with conflict resolution (`-X ours`) is used to compose them.

### Root cause
`git branch --merged` detects merge by ancestry: a branch is "merged" only if its tip commit (or one of its ancestors) appears in the target's commit history. Cherry-pick replays commits as new commits with different SHAs — the original branch commits are never in main's ancestry, so git sees the branch as unmerged regardless of content equivalence.

### Fix
To verify a cherry-picked branch is truly complete, diff content rather than checking ancestry:

```bash
# Are there any code changes on the branch that are not on main?
git diff <branch> main -- '*.java' '*.sql'

# Is main's version of a key file newer than the branch's?
git diff <branch> main -- path/to/KeyFile.java
# If the diff shows main has MORE (additions only), the branch is behind, not ahead
```

If `git diff branch main` shows main has evolved past the branch (main has additions the branch lacks), the branch content is already on main in an evolved form.

### Why this is non-obvious
`git branch --merged` is a natural first check when cleaning up old branches. A result of "not merged" is normally a hard stop — but for cherry-picked integrations it is a false alarm. The instinct to "re-merge" would create conflicts and duplicate commits.
