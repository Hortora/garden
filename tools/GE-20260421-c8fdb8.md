### Symptom

Running `sed -i '' 's/OldPattern/NewPattern/'` on a Java file containing angle brackets (generics) silently truncates the file to 1 line. No error is printed. The original content is lost.

### Context

macOS `sed` (BSD sed) handles certain special characters in the replacement differently from GNU sed. When the file contains `<`, `>`, or generic type parameters, a substitution touching nearby lines can silently truncate the output. Affects any Java file with generics.

### What was tried

Standard `sed -i '' 's/OldClass/NewClass/g'` — appeared to succeed, no error, exit code 0. File reduced to 1 line.

### Root cause

BSD sed on macOS processes certain character sequences as control characters or metacharacters in ways that differ from GNU sed, causing early termination of output. The failure is silent because sed still exits 0.

### Fix

Use `perl -pi -e` for all in-place substitutions on Java files:

```bash
# Instead of:
sed -i '' 's/OldClass/NewClass/g' File.java

# Use:
perl -pi -e 's/OldClass/NewClass/g' File.java
```

Perl's `-pi -e` is portable across macOS and Linux, handles all Java generics safely, and has the same regex syntax.

### Why this is non-obvious

`sed -i` is ubiquitous and expected to be safe. The failure is completely silent — no error, no warning, exit code 0. You only notice when you look at the file and find it reduced to 1 line. On Linux with GNU sed the same command works fine.
