### Symptom

A script copies a `commit-msg` hook to `.githooks/`, sets `git config core.hooksPath .githooks`, then commits the hook file. The commit is blocked by the hook that was just installed — in this case "Commit blocked — no issue reference found."

### Context

Bulk hook installation across multiple repos. The script sets `core.hooksPath` to activate the hooks directory, stages the hook file, then runs `git commit`. The newly active hook fires on that commit.

### Root cause

git evaluates `core.hooksPath` at commit time. Once the path is set and the hook file exists there, git uses it for all subsequent operations in that repo — including the commit that adds the file itself.

### Fix

Two options:

**Option A — commit before activating:**
```bash
cp hook-src .githooks/commit-msg
git add .githooks/commit-msg
git commit -m "chore: add commit-msg hook"   # hook not yet active
git config core.hooksPath .githooks           # activate after commit
```

**Option B — include the bypass token in the installation commit:**
```bash
git config core.hooksPath .githooks
cp hook-src .githooks/commit-msg
git add .githooks/commit-msg
git commit -m "chore: add git hooks  no-issue: infrastructure"
```
(Works when the hook allows an explicit bypass token like `no-issue`.)

### Why this is non-obvious

The mental model is "I'm installing a hook for future commits." The reality is git activates `core.hooksPath` immediately — the hook applies to the very commit that stages it.
