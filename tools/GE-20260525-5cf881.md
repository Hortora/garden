## What it is

If an OpenClaw plugin declares `kind: "context-engine"` in its plugin metadata and is
selected via `plugins.slots.contextEngine` in config, OpenClaw hands off ALL context
assembly to that plugin — not just the `before_prompt_build` hook.

This includes: context construction, conversation history management, `/compact`
lifecycle hooks, and subagent context. The plugin controls the full context window.

## Context

OpenClaw's default context pipeline processes bootstrap files, session history, and tool
data to build each agent turn's context window. The `before_prompt_build` hook lets you
inject content into this pipeline at one point. The context engine replaces the entire
pipeline.

## How to activate

```json
// openclaw config
{
  "plugins": {
    "slots": {
      "contextEngine": "my-context-engine-plugin"
    }
  }
}
```

The plugin must expose a `kind: "context-engine"` declaration and implement the context
lifecycle hooks OpenClaw calls into: context build, compaction, subagent context prep.

## Why this matters

For a CaseHub × OpenClaw integration (`casehub-openclaw`), the context engine is the
right integration point if you need to inject Qhorus channel history, CaseMemoryStore
facts, and WorkerContextProvider lineage as a unified context package. More powerful than
`before_prompt_build` alone, which only injects at one point in the pipeline.

## Why this is non-obvious

The official docs describe `before_prompt_build`, `before_model_resolve`, and other
individual hooks. The pluggable context engine is mentioned briefly in config documentation
under `plugins.slots.contextEngine` but not in any of the hook or integration guides.
You would only find it by reading the plugin system config reference or the source.
