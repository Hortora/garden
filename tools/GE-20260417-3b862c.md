## Symptom

Running `jar xf some.jar io/example/Foo.java` to inspect a single source file from a JAR silently extracts the file — and the full directory tree — into the current working directory. If the CWD is a git repository, `git add -A` on a subsequent commit will include all of the extracted library source files.

## Context

Any use of `jar xf` without `-C <dir>` or an explicit `--dir` flag. Common when inspecting library sources pulled from a Maven sources JAR to read the implementation.

## What was tried (didn't work)

Assumed the intent was to extract to `/tmp` because the command started with `cd /tmp &&`. The shell reset CWD between commands, so the `jar xf` ran in the repo root instead.

## Root cause

`jar xf` has no default temp-dir behaviour. It extracts relative to wherever the JVM's working directory is at invocation time. If the shell resets CWD between tool calls (as Claude Code's Bash tool does), a preceding `cd /tmp` has no effect.

## Fix

Always specify the output directory explicitly:
```bash
# Correct — extracts into /tmp
mkdir -p /tmp/jar-inspect
cd /tmp/jar-inspect && jar xf /path/to/sources.jar io/example/Foo.java

# Or use --dir (JDK 9+)
jar xf --dir=/tmp/jar-inspect /path/to/sources.jar
```

If you already committed the extracted files accidentally:
```bash
git rm -r --cached io/          # unstage the library tree
git commit -m "chore: remove accidentally committed library sources"
```

## Why this is non-obvious

The `jar` command gives no indication of where it is extracting to. The output appears in the current directory silently. When running inside an AI assistant that resets CWD between Bash calls, the confusion between intent ("extract to /tmp") and effect ("extracted to repo root") is especially easy to miss.
