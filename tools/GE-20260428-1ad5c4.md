## Symptom

Fork CI passes all tests then fails at the deploy step with a 404 or 403 when trying to publish to the upstream organisation's GitHub Packages registry. The same workflow succeeds on the upstream repo.

```
Failed to deploy artifacts: Could not find artifact io.casehub:api:pom:0.2-SNAPSHOT
  in github (https://maven.pkg.github.com/upstream-org/repo)
```

## Context

A Maven project uses `mvn deploy` unconditionally on push in `maven.yml`. The `distributionManagement` URL points to the upstream org's registry. Forks run the same workflow — but `GITHUB_TOKEN` on a fork only has write access to the fork's own packages registry, not the upstream org's.

## Root cause

`GITHUB_TOKEN` is scoped to the repository it's running in. A fork's token can read from other registries but cannot write to `https://maven.pkg.github.com/upstream-org/`. The workflow is identical on both the fork and the upstream — but the token capabilities differ by repository.

## Fix

Add a `github.repository` guard to deploy only from the canonical upstream:

```yaml
- name: Build, test and publish (upstream main)
  if: github.event_name == 'push' && github.repository == 'upstream-org/repo'
  run: mvn --batch-mode deploy
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

- name: Build and test (PR or fork push)
  if: github.event_name == 'pull_request' || (github.event_name == 'push' && github.repository != 'upstream-org/repo')
  run: mvn --batch-mode verify
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Forks run `verify` (build + test); only the upstream repo runs `deploy`.

## Why this is non-obvious

The workflow file is identical on fork and upstream — the failure is purely about runtime token capabilities. The error looks like a missing artifact or URL misconfiguration, not a permissions issue. Developers typically investigate `distributionManagement` URLs or token scopes rather than the repository-conditional deploy pattern.
