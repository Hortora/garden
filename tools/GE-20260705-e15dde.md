### Symptom

After running `ide_refactor_rename` across a 26-repo IntelliJ workspace, ~15 files
have `@Inject` annotations merged with the class name: `@InjectEventBus` instead of
`@Inject EventBus`. The import for the renamed class is also dropped. Compilation
fails with "cannot find symbol" but the error message points at the merged token,
not at a missing import — misleading the diagnosis.

### Context

Occurs when the original field declaration uses whitespace alignment padding:

```java
@Inject
       OldClassName      fieldName;    // alignment padding
@InjectOtherClassName    otherField;   // next field gets merged
```

The rename tool collapses the whitespace and merges the annotation token with the
class name on the next line. Compact single-line declarations (`@Inject OldClassName field;`)
are unaffected.

### What was tried (didn't work)

1. Running `ide_optimize_imports` after the rename — adds some missing imports but
   not all (ambiguous or cross-module classes are silently skipped)
2. Trusting the rename tool's success message ("Successfully renamed ... also renamed
   N related element(s)") — the message reports success even when annotations are corrupted

### Root cause

The IntelliJ MCP rename refactoring does not preserve whitespace structure around
annotations when the field declaration spans multiple lines with alignment padding.
The token boundary between the annotation and the type name is lost during the
whitespace collapse.

### Fix

After any cross-workspace rename, run this detection grep:

```bash
grep -rn "@Inject[A-Z]\|@Mock[A-Z]" project/*/src/ 2>/dev/null | \
  grep -v "import\|@Injectable\|@InjectMock\|@InjectSpy"
```

Each hit is a corrupted annotation. Fix pattern: `@InjectEventBus eventBus;` →
`@Inject EventBus eventBus;`. Then add the missing import manually — `ide_optimize_imports`
may not resolve it for cross-module classes.

### Why this is non-obvious

The rename tool reports success for all affected files. The compilation error message
shows "cannot find symbol" at the field declaration line, which looks like a missing
import — but the actual problem is the merged annotation token. A developer would
naturally add the import, which doesn't fix the merged `@InjectClassName` token.
The annotation corruption is invisible unless you read the source line character by
character.

### Alternatives considered

- Running `ide_reformat_code` after rename — does not fix annotation merging (Spotless
  catches it as a lint error, but only if Spotless is configured)
- Doing renames one at a time — same corruption occurs; batch vs single makes no difference
