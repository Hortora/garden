## Context

When a repo is moved to a new location (e.g. `~/old/path/repo` → `~/new/path/repo`), any `git worktree` registrations that pointed to subdirectories of the old location become ghost entries — the path doesn't exist but git still tracks them as locked worktrees. Standard `git worktree prune` skips locked worktrees.

## Technique

Pass the **old (now non-existent) path** directly to `git worktree unlock`, then prune:

```bash
# Unlock using the old path (works even if the directory is gone)
git worktree unlock "/old/path/repo/.claude/worktrees/agent-abc123"

# Now prune removes it (it's no longer locked and the path doesn't exist)
git worktree prune --expire=now --verbose
# Output: "Removing worktrees/agent-abc123: gitdir file points to non-existent location"

# Clean up the orphan branch too
git branch -D worktree-agent-abc123
```

To find registered worktrees (including ghosts):
```bash
git worktree list  # shows all registered worktrees with their paths and lock status
```

## Why This Is Non-Obvious

The intuitive assumption is that `git worktree unlock <path>` requires the path to exist. It doesn't — git operates on its internal worktree registry, which records the path as a string. The unlock succeeds regardless of whether the filesystem path is valid. Without this, the only option appears to be `git worktree remove --force`, which also requires the path to exist.
