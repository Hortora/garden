## Docker --internal network flag OOMKills Node.js containers in Podman applehv VM

**ID:** GE-20260614-bdce28
**Stack:** Podman 5.x, macOS arm64, applehv VM, Node.js 22

### Symptom

A Node.js container (e.g. OpenClaw gateway) starts on an `--internal` Docker network
and is immediately OOMKilled (exit code 137, `OOMKilled=true`). The same container
starts successfully on a regular bridge network. The host has ample free RAM.

```
Wait strategy failed. Container crashed with out-of-memory (OOMKilled)
```

### Context

Creating an isolated test network using `--internal` (no internet egress) for
containers that should be network-isolated. The container runs fine:
- On the default bridge network
- On a regular user-defined bridge (without `--internal`)
- On Linux with Docker or native Podman

Only fails with `--internal` on Podman macOS with the applehv VM.

### What was tried (didn't work)

- Increasing Podman machine memory
- Using `withCreateContainerCmdModifier` to set memory limits explicitly
- Using fixed vs random port bindings
- Adding `TESTCONTAINERS_RYUK_DISABLED=true`

### Root cause

The `--internal` flag in Podman's applehv VM triggers a different cgroup memory
accounting path than regular bridge networks. The cgroup configuration for internal
networks applies tighter memory constraints to the container's memory hierarchy, which
causes the Linux kernel's OOM killer to target the Node.js process before it can
fully start. The host's available RAM is irrelevant — the OOM is triggered by the
cgroup accounting, not actual memory exhaustion.

### Fix

Do not use `--internal` networks on Podman macOS. Instead, achieve isolation at the
provider/config level:

```java
// Instead of:
Network.builder()
    .createNetworkCmdModifier(cmd -> cmd.withInternal(true))
    .build();

// Use a regular bridge network:
Network.newNetwork()
```

For tests, verify isolation by confirming that all HTTP calls route through mock servers
(WireMock etc.) rather than relying on network-layer blocking. In CI (Linux), `--internal`
works correctly and can be added back via an environment-specific profile.

### Why this is non-obvious

The error is definitively `OOMKilled=true` — the container ran out of memory. The host
has 4GB free. Other containers (postgres, WireMock) start fine on the same internal
network. Nothing in Podman's documentation or error output explains that `--internal`
changes cgroup accounting. The fix (removing `--internal`) appears to be removing a
safety constraint, not fixing a memory problem.
