## Truncated hash as SQL primary key with INSERT OR IGNORE silently discards rows on collision

**ID:** GE-20260415-7ca64f
**Stack:** Python sqlite3 (stdlib), any SQL database

**Symptom:** Rows are silently not inserted — no error, no warning, no return value indicating failure. The count of records is lower than expected, but only detectable by auditing the data, not from any exception or log output.

**Context:** Using a truncated hash (e.g. `sha256(value)[:8]`, 32-bit output) as a `TEXT PRIMARY KEY` to get a short surrogate key, combined with `INSERT OR IGNORE` for idempotency. At ~1,000 rows the birthday collision probability reaches ~1%; at 2,000 rows ~4%.

### What was tried (didn't work)
- `hashlib.sha256(pair.encode()).hexdigest()[:8]` as PK — seemed safe; SHA-256 is cryptographically strong
- `INSERT OR IGNORE` for idempotency — correctly suppresses duplicate-key errors for true duplicates

### Root cause
`INSERT OR IGNORE` treats a hash collision identically to a genuine duplicate: it silently discards the row. The 8-hex-char (32-bit) output of SHA-256 has a birthday collision probability of `1 - e^(-n²/2³²)`. At n=1,000 rows this is ~11.6%. The data appears to be correctly written; the missing row is only detectable by auditing row counts against the source.

The issue is using a *content-addressing* hash (designed for integrity checking) as a *surrogate key* (designed for uniqueness). Content-addressing works when the hash space is huge (full 256-bit SHA-256). A truncated 32-bit version has the birthday problem at modest scale.

### Fix
Use the natural key — the actual value being stored — as the primary key directly:

```python
# Instead of:
pair_hash = hashlib.sha256(pair.encode()).hexdigest()[:8]
conn.execute("INSERT OR IGNORE INTO checked_pairs (pair_hash, pair, result) VALUES (?, ?, ?)",
             (pair_hash, pair, result))

# Use the value itself as PK:
conn.execute("INSERT OR IGNORE INTO checked_pairs (pair, result) VALUES (?, ?)",
             (pair, result))
```

For bounded-length natural keys (GE-IDs, UUIDs, short strings), the natural key is always preferable: zero collision risk, semantically meaningful, readable in SQL queries.

### Why this fix
The pair string (e.g. `GE-20260414-aa0001 × GE-20260414-bb0002`) is bounded at ~40 characters — negligible as a TEXT primary key. Using it directly eliminates the collision class entirely, makes the schema self-documenting, and makes SQL queries readable (`WHERE pair = 'GE-X × GE-Y'` vs `WHERE pair_hash = 'a1b2c3d4'`). A longer hash would still carry theoretical risk and obscures what the key means.

### Why this is non-obvious
SHA-256 is marketed as collision-resistant, which it is — at full 256-bit length. The truncation fundamentally changes the collision property: 32 bits is only collision-resistant to ~65,000 values. Most developers associate SHA-256 with safety and don't think through the birthday problem at the truncated length. `INSERT OR IGNORE` compounds the invisibility by design — it is meant to suppress errors, and it does so for both legitimate duplicates and accidental hash collisions indistinguishably.
