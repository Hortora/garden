### Symptom
After resolving a merge conflict, committing, and force-pushing to a PR branch,
`gh pr view --json mergeable` still returns `"mergeable": "CONFLICTING"`.
`gh pr merge` returns HTTP 405 with "Pull Request is not mergeable". The branch
is verifiably clean — `git log origin/main..branch` shows exactly 1 commit ahead
with no conflicts.

### Context
The issue appears after any force push to a PR branch, including conflict
resolution pushes. GitHub calculates mergeability asynchronously on its servers
and caches the result. A force push does not synchronously invalidate the cache.
The PR may be perfectly mergeable in git terms but still blocked via the API.

### What was tried (didn't work)
- Waiting 5–15 seconds then retrying `gh pr merge` — still 405
- Using `gh api repos/.../pulls/N/merge --method PUT` directly — still 405
- `gh pr merge --auto` — rejected (branch protection not configured)

### Root cause
GitHub's mergeability calculation is a background job triggered by push events,
not a synchronous response. After a force push, the previous cached result
(CONFLICTING) persists until the background job completes. This takes 30–120
seconds depending on repo size and GitHub load.

### Fix
Push a fresh non-empty or empty commit to the PR branch to re-trigger the
background job:

```bash
git commit --allow-empty -m "chore: trigger mergeability re-check"
git push origin <branch>
# Wait ~20 seconds, then:
gh pr view --json mergeable  # should show MERGEABLE
gh pr merge --merge
```

Alternatively, wait 1–2 minutes after the force push before attempting to merge.

### Why this is non-obvious
The branch is demonstrably clean (`git merge-base` confirms it's a direct
ancestor of main, `git diff` shows no conflicts). Nothing in the `gh pr merge`
error message hints at a caching delay — "not mergeable" looks like a genuine
conflict. The fix (push another commit) seems counterintuitive when the goal is
to make the branch mergeable, not add more commits.

### Alternative — Upstream ref is stale (GitHub may actually be right)

Before concluding the status is a caching artefact, verify your local upstream
ref is current:

```bash
git fetch upstream
git log --oneline upstream/main -3   # did the tip change?
git merge-base --is-ancestor upstream/main <branch> && echo "still clean"
```

If `git fetch` reveals new commits on upstream/main, GitHub's CONFLICTING status
is correct — another PR merged upstream between your last fetch and now, genuinely
making the branch conflicting. The branch needs rebasing, not waiting.

The trap: `git merge-base --is-ancestor upstream/main <branch>` passes against
your **local** `upstream/main` ref, which may be minutes behind the real remote.
The check looks authoritative but is only as fresh as your last fetch.
