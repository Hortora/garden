### Symptom
`git cherry-pick --no-commit <sha1> <sha2>` fails with merge conflicts even though both commits apply cleanly on their own. The conflict appears in a file that both commits touched.

### Context
Squashing intra-branch commits: two or more commits on the same branch both modified the same file. Trying to cherry-pick them sequentially with `--no-commit` onto a clean base to produce a single squashed commit.

### Root cause
`cherry-pick` applies each commit as a delta **from its parent commit**, not from the current working tree state. When the second commit is applied, the working tree already contains the first commit's changes — so the delta from the second commit's parent conflicts with the already-applied first commit's changes. The conflict is structural: cherry-pick is not designed for intra-branch sequential application of related commits.

### Fix
Use `git reset --soft <base>` to collapse all commits into staged changes in one step, then commit once:

```bash
# Collapse all commits since the base into a single staged change-set
git reset --soft origin/main

# Commit the squashed result
git commit -m "feat: squashed commit message"
```

This bypasses cherry-pick entirely. All commits' changes are collapsed into the working tree simultaneously — no delta conflicts possible.

### Why this is non-obvious
`cherry-pick --no-commit` reads as "apply this commit without committing, so I can batch them." A developer naturally reaches for this when building a squash from existing commits. The conflict is surprising because each commit independently applies cleanly — the failure only emerges when they're applied sequentially to a working tree that already has the prior commit's changes. `reset --soft` is rarely the first tool reached for in a squash workflow.
