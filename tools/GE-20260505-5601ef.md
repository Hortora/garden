## Context

A follow-on commit (docs update, minor fix) needs to be absorbed into the
preceding commit, keeping the preceding commit's message exactly as-is.

## Technique

```bash
git reset --soft HEAD~1    # uncommit the top commit; keep changes staged
git commit --amend --no-edit  # absorb staged changes into previous commit, no message change
```

Result: one commit containing both sets of changes, with the original message
preserved verbatim.

Force-push if the branch is already pushed:

```bash
git push origin branch-name --force
```

## What developers normally do instead

`git rebase -i HEAD~2` — change `pick` to `squash` or `fixup` for the second
commit, then edit the combined message in the editor. More steps, opens an
editor, and `squash` prompts for a new message even when you want the original.

## Why this is non-obvious

`--no-edit` on `git commit --amend` is underused. Most developers associate
`--amend` with changing the message, not with adding content to the previous
commit silently. The `reset --soft` + `amend --no-edit` combination is not
taught as a squash pattern, but it's the fastest path when the goal is
specifically "absorb top commit, keep message."

## When to use

- Squashing a docs or cleanup follow-on into the feature commit it follows
- Applying the compaction policy to a PR before merging (drop noise, keep signal)
- Any case where exactly two commits need combining and the first commit's
  message is already correct
