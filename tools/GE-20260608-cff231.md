### Symptom

A Java source file contains the intended character (e.g. the SOH sentinel byte U+0001) but
editors, git diff, grep, git show, and IntelliJ all display only an empty gap or strip
the byte silently. A later code reviewer reports the constant is wrong. Text-based Edit and
replace_text_in_file tools report "string not found" when trying to fix it.

### Context

A Java constant was intended to hold a SOH (U+0001) prefix followed by visible ASCII text —
e.g. `"\u0001DHMETA:"`. The constant was written via the Write tool. The JSON parameter
contained `"\u0001DHMETA:"`. The JSON parser converted `\u0001` to the actual byte 0x01,
producing an invisible character in the file rather than the six-character escape sequence.

### What was tried (didn't work)

- Edit tool with old_string containing the constant line: "string not found" (invisible byte not matchable as text)
- mcp__intellij__replace_text_in_file with regex: "no occurrences found"
- Writing `"\u0001DHMETA:"` again in the Write tool content: same result (still produces the SOH byte)

### Root cause

JSON string parsing converts `\uXXXX` sequences to their Unicode code points. Writing
`"\u0001DHMETA:"` in any JSON-parameter tool produces a string containing the actual
byte 0x01 (SOH), not the six visible ASCII characters `\u0001`. All subsequent text-based
tools cannot match a string containing the invisible byte.

This affects any control character in U+0000-U+001F range.

### Fix

**To write the visible Java Unicode escape `\u0001` in source:** use `\\u0001` in the
JSON parameter. The JSON parser converts `\\u` to `\u` (backslash + u), leaving the
remaining `0001` as literal text. The file receives the 6-character sequence `\u0001`.

```python
# If the invisible byte is already in a file, fix it with Python binary replacement:
with open(filepath, 'rb') as f:
    raw = f.read()
fixed = raw.replace(b'\x01', b'\\u0001')   # produces visible \u0001 in file
with open(filepath, 'wb') as f:
    f.write(fixed)
```

For other control characters: `\\uXXXX` in the JSON parameter produces `\uXXXX` in the
file (visible escape text), while `\uXXXX` produces the actual byte.

### Why this is non-obvious

The Write and Edit tool parameters look like ordinary strings. There is no indication that
they undergo JSON parsing before the content reaches the filesystem. The invisible byte passes
all compilation checks (Java compiles both the literal byte and `\u0001` identically) and
the code appears to work — the bug only surfaces when another developer reads the source file
and sees 7 visible characters where the constant should have 8.

The same JSON-parameter tool call that wrote the invisible byte cannot fix it, because
the old_string in Edit also becomes an invisible byte and fails to match.

### Alternatives considered

- `\\u0001` in JSON parameter: produces `\u0001` in file (6 visible chars) — correct
- `\u0001` in JSON parameter: produces actual SOH byte 0x01 in file (invisible) — wrong
- IntelliJ replace_text_in_file with regex: also cannot match the invisible byte — wrong
