## `git worktree remove` fails after merge when worktree was used as subagent CWD

**ID:** GE-20260414-db76e9
**Stack:** git, Claude Code subagents

**Symptom:** After merging a worktree branch into main, `git worktree remove .worktrees/feat-x` fails:
```
fatal: '.worktrees/feat-x' contains modified or untracked files, use --force to delete it
```
The worktree looks clean — the merge succeeded, there are no uncommitted changes visible.

**Context:** Occurs when the worktree was used as the working directory for a Claude Code subagent. The subagent's session leaves stale filesystem state in the worktree even after its work is committed and merged.

**Root cause:** Git's worktree remove safety check detects state inconsistencies left behind when a process (the subagent) was abruptly terminated or changed directory mid-operation. The worktree's internal tracking files don't match the merged state cleanly enough for git to consider it safe to remove without force.

**Fix:**
```bash
git worktree remove --force .worktrees/feat-x
git branch -d feat/x
```

Always use `--force` when removing a worktree that was used as a subagent working directory. The force flag bypasses the safety check while the branch deletion remains safe — the code is already on main.

**Why non-obvious:** The error message suggests there are modified or untracked files, but `git status` in the worktree directory shows nothing. The cause is internal worktree tracking state, not visible file changes.
