## The Technique

Write `@ToolArg` (or equivalent tool parameter) descriptions as compact
classification guides rather than type documentation. When an LLM calls the
tool, it reads the description to determine which enum value to choose — making
the description double as a zero-configuration classifier.

```java
@ToolArg(description = """
    The message type. Choose:
    QUERY — asking for information (no action, no side effects)
    COMMAND — asking for action to be taken (side effects expected)
    RESPONSE — answering a QUERY (carries correlationId)
    STATUS — reporting progress on a COMMAND (extends deadline)
    DECLINE — refusing a QUERY or COMMAND (content must explain why)
    HANDOFF — transferring obligation to another agent (target required)
    DONE — signalling successful completion of a COMMAND
    FAILURE — signalling unsuccessful termination (content must explain why)
    EVENT — telemetry only, not delivered to agents
    """)
String type
```

## Context

Any MCP tool parameter that accepts a typed enumeration. Particularly effective
when the enum values have distinct semantic intent that an LLM should infer from
context, rather than being set mechanically.

## Why Non-Obvious

The default approach is to document what the type *is* (e.g. "one of: QUERY,
COMMAND, RESPONSE…"). The classification guide approach instead tells the LLM
*when to use each value* — making the description an embedded few-shot prompt.
The LLM reads the description at call time and classifies its own situation
against the options without any additional prompt engineering or classifier model.
A developer would naturally write documentation; this technique writes a decision
tree instead.

## Validation Method

Classification accuracy is measurable: run the LLM against a set of natural
language scenarios and check whether it selects the expected enum value. This
also validates taxonomy granularity — if two values are consistently confused,
they should be merged; if one value is consistently split, it should be divided.
