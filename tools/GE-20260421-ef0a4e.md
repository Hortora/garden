## Refactoring tests to extend an abstract base class silently removes implementation-specific test methods

### Symptom

Test count drops unexpectedly (e.g. 92 → 62) after refactoring standalone test classes to extend an abstract contract base. No test failures — just fewer tests than before.

### Context

When refactoring `InMemoryChannelStoreTest` (standalone) into a class that extends `ChannelStoreContractTest` (abstract base), the implementer replaced the entire file with factory method implementations only. The implementation-specific tests that tested store-specific behaviour beyond the contract (e.g. `scan_bySemantic`, `countByChannel`, `clear_removesAll`) were simply omitted — there was no error, just fewer tests.

### What was tried

Replacing each test class body with only the abstract method implementations, assuming the abstract base's `@Test` methods would be inherited and provide equivalent coverage.

### Root cause

The abstract base only defines the minimum shared contract. Any test methods that covered behaviour specific to the implementation (methods not in the abstract interface, edge cases, extra query filters) must be explicitly carried over to the concrete class as supplemental `@Test` methods alongside the inherited ones.

### Fix

When refactoring to an abstract base pattern, the concrete class must:
1. Implement all abstract factory methods (delegation to the store)
2. Keep or migrate ALL existing test methods that are not covered by the abstract base
3. Verify the test count before and after — it should be ≥ the original count

The concrete class should look like:
```java
class InMemoryChannelStoreTest extends ChannelStoreContractTest {
    // Factory methods (required)
    @Override protected Channel put(Channel c) { return store.put(c); }
    // ... other abstract methods ...

    // Supplemental tests (must NOT be removed)
    @Test void scan_bySemantic_returnsMatching() { ... }
    @Test void clear_removesAll() { ... }
}
```

### Why this is non-obvious

The refactoring "succeeds" silently — the build passes, all inherited tests pass, and there are no compilation errors. The only signal is a reduced test count, which is easy to miss if you're not watching the numbers. The instinct is to "simplify" the concrete class to just the delegation code.
