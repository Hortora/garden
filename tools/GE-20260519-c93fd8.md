### Symptom

After running `git checkout <branch> -- <file1> <file2>` to pull specific files from
another branch, a subsequent `git add -A` stages untracked files that were sitting
in the working directory — silently including them in the commit.

### Context

Occurs in any workflow that does a targeted branch checkout followed by `git add -A`.
Common in epic-close artifact promotion: checking out plan files from an epic branch
onto main, then staging everything to commit. Any untracked files in the working tree
(e.g. `.claude/settings.local.json`, editor temp files, local config) get swept in.

### What was tried

Used `git checkout epic-branch -- plans/file1.md plans/file2.md` to bring specific
files to the current branch, then `git add -A` to stage them. A `.claude/settings.local.json`
file that was untracked in the working tree was silently committed alongside the plan files.

### Root cause

`git add -A` is unconditional — it stages all tracked modifications AND all untracked
files in the working tree. It has no awareness of which files were just checked out.
The targeted `git checkout <branch> -- <files>` brings those files into the working
tree as modified/new, but `git add -A` then stages everything else too.

### Fix

After a targeted checkout, stage only the specific files you intend to commit:

```bash
# Instead of:
git add -A

# Use:
git add plans/file1.md plans/file2.md
# or stage by directory:
git add plans/attic/epic-name/
```

If `git add -A` is needed for other reasons, run `git status` first and verify
there are no unexpected untracked files before committing.

### Why this is non-obvious

The mental model of `git checkout <branch> -- <files>` is "get these specific files
from that branch" — it feels like a scoped operation. The follow-up `git add -A`
feels like "stage what I just checked out." But `git add -A` has no memory of the
checkout — it stages the entire working tree state regardless.
