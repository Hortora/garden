## Symptom

After running `sed -i '' 's/old/new/' file.md` on macOS, the file exists but contains 0 bytes. No error was reported.

## Context

Attempting to make a targeted in-place replacement in a text file from a Claude `Bash` tool call. The replacement pattern did not match any content in the file.

## What was tried

Used `sed -i '' 's/pattern/replacement/' file` directly — standard approach that works on Linux. File was wiped to empty with no warning or non-zero exit code.

## Root cause

BSD sed (macOS) and GNU sed (Linux) differ in their `-i` behaviour when the replacement pattern has no match. GNU sed leaves the file unchanged. BSD sed (macOS) writes an empty file when it finds nothing to replace, effectively wiping all content.

## Fix

**Never use `sed` for targeted in-place replacements in Claude sessions.** Use the `Edit` tool instead — it matches exact strings and fails explicitly if the string is not found, rather than silently destroying the file.

```bash
# WRONG on macOS — wipes file if pattern doesn't match
sed -i '' 's/old text/new text/' file.md

# RIGHT — use Claude's Edit tool for targeted string replacement
# Edit tool: old_string="old text", new_string="new text"
```

If `sed` is genuinely needed (e.g. bulk transformations), validate the pattern matches first:
```bash
grep -q 'pattern' file.md && sed -i '' 's/pattern/replacement/' file.md
```

## Why this is non-obvious

The behaviour difference between BSD sed and GNU sed on no-match is undocumented in the macOS man page. On Linux CI environments, the same command works correctly. The file is not deleted — it exists with 0 bytes — which looks like a write succeeded. Exit code is 0 in both cases.
