## Use 'git show <commit>:path > target' to recover file content from any point in git history without checkout

### What it achieves

Restores any file from any commit directly to a new filesystem path — without switching branches, stashing work, or creating a detached HEAD. Works even when the file no longer exists in the current working tree or index.

### Context

Useful when: a file is accidentally deleted from the git index (via `git rm`), a `git mv` partially stages and then loses a file, or you need a specific historical version of a file at a different path.

### The technique

```bash
# Recover a file deleted from staging, placing it at a new path
git show <last-good-commit>:original/path/file.md > new/path/file.md

# Then stage it at the new location
git add new/path/file.md

# Examples:
git show HEAD~3:quarkus/GE-20260414-23982b.md > casehub-qhorus/GE-20260414-23982b.md
git show abc1234:src/main/OldName.java > src/main/NewName.java
```

`git show <commit>:<path>` outputs the file content to stdout. Redirect with `>` to write it anywhere on disk. The commit can be a hash, tag, branch name, or relative reference like `HEAD~2`.

### Why this is non-obvious

Most developers reach for `git checkout <commit> -- <path>` to restore files, which restores to the original path and stages it. `git show` + redirect is more flexible — it lets you place the content anywhere, including a completely different path, without touching the index until you explicitly `git add`. It also works cleanly when there is no working tree at the target path yet.
