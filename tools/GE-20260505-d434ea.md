## Symptom

A doc subagent dispatched to update `DESIGN.md` and `CLAUDE.md` committed the changes to the fork's `main` branch instead of the feature branch. The subagent also confidently reported that `ProvisionContext` changes were "on main via the stash" — which was entirely false. The code changes were on a feature branch and had not touched main at all.

## Context

Working in a fork model (`origin` → your fork, `upstream` → upstream repo). Two feature branches had code changes (PRs open against upstream). A doc subagent was dispatched to update documentation to reflect both changes and told to "work on main". The agent committed docs to the fork's `main` (correct intent) but also made a fabricated claim about the state of code on main — confidently asserting that code from a feature branch had landed there via a stash.

## What was tried (didn't work)

Trusting the subagent's branch state report. The false claim only surfaced when checking `grep -c "triggerChannelId" api/src/main/java/...` on main, which returned 0.

## Root cause

Claude Code subagents don't reliably track which branch they're on in a fork model. When a subagent reads files, it reads from the working tree — which may reflect a different branch than the one the agent believes it's on. Agents can make confident-sounding false claims about what git state exists because they infer branch state from file content rather than from `git status` or `git branch`.

## Fix

After any doc subagent runs in a fork model:
1. Verify the actual git state independently: `git status`, `git log --oneline -3`, `grep <expected-symbol> <file>`
2. Verify the agent committed to the correct branch: `git log --oneline --all | head -10`
3. If the agent committed to the wrong branch: `git revert <sha>` on the wrong branch, then apply docs to the correct feature branch.

The correct pattern: doc changes belong on the feature branch that introduces the code, not on `main`. When the PR merges, docs arrive with the code.

### Why this is non-obvious

Subagents don't inherit the parent session's branch context. Even if you tell a subagent "you are on branch X", it may read the working tree (which reflects branch Y if git has auto-switched) and produce confident-sounding but fabricated assertions about what exists on each branch. The false report had no warning signs — it was stated as fact.

### Alternatives considered

- Dispatching the agent with explicit `git checkout <branch>` instructions — still doesn't prevent false state reporting, as the agent may read files without checking current HEAD
- Adding branch verification to the subagent prompt — helps, but agents can still conflate working tree content with branch state
