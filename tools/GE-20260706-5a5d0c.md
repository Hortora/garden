### Symptom
`gh api 'orgs/{org}/packages?package_type=maven'` returns a subset of packages.
Packages known to exist and be published do not appear in the results. No error,
no warning, no indication that results are truncated.

### Context
Auditing whether all Maven modules in a multi-repo build are published to GitHub
Packages. The org has 200+ packages. Default page size is 30.

### What was tried (didn't work)
- Single API call to list all packages — returned only ~30, missing entire repos
- Assumed packages were not published and began investigating CI workflows

### Root cause
The GitHub REST API `/orgs/{org}/packages` endpoint paginates at 30 items by
default. Unlike most GitHub API endpoints, the `gh` CLI does not auto-paginate
this one. There is no `total_count` field or obvious truncation marker in the
response — the caller gets a valid JSON array that simply stops early.

### Fix
Use `per_page=100` and iterate pages until an empty page is returned:
```bash
# Page through all packages
gh api 'orgs/{org}/packages?package_type=maven&per_page=100&page=1' --jq '.[].name'
gh api 'orgs/{org}/packages?package_type=maven&per_page=100&page=2' --jq '.[].name'
# ... until empty result
```

Or check specific packages by name (O(1), no pagination):
```bash
gh api 'orgs/{org}/packages/maven/{groupId}.{artifactId}/versions' \
  --jq '.[0].name' 2>/dev/null || echo "NOT FOUND"
```

### Why this is non-obvious
Most GitHub API endpoints either return all results or include clear pagination
indicators (Link headers, total_count). This endpoint silently returns a partial
list. The natural debugging path is to investigate CI/CD publishing, not API
pagination — especially when the first page contains packages from other repos
in the org, making the response look complete.

### Why this fix
Checking specific packages by name avoids pagination entirely and is O(1) per
package. Use the paginated listing only when you need the full inventory.
