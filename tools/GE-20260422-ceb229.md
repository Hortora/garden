### Behaviour

When `git rebase --onto <new-base> <upstream> <branch>` encounters a commit
whose patch is already present in `<new-base>`'s history — even under a
different SHA (e.g. after a squash merge) — git silently drops it and prints:

```
dropping <sha> <commit-message> -- patch contents already upstream
```

The rebase continues with the remaining commits. No conflict, no error, no
manual intervention needed.

### Context

This happens reliably when rebasing a PR branch onto an upstream that has
squash-merged the same content. The squash commit has a different SHA but
contains the same file changes, so git's patch-id matching identifies them
as equivalent and skips re-application.

Example: branch A was developed with commits X, Y, Z. Upstream squash-merged
these as a single commit S. When rebasing B (which includes X, Y, Z as
ancestors) onto the new upstream, git drops X, Y, Z automatically, leaving
only B's unique commits.

### Why this is undocumented

The `git rebase` man page describes `--onto` semantics and conflict handling
but does not mention the automatic patch deduplication behaviour or the
"patch contents already upstream" message. It is only discoverable by
observing the rebase output or reading git source code. Most developers
expect they need to manually identify and skip already-merged commits.

### Practical value

This makes `git rebase --onto` reliable for PR chain cleanup after squash
merges without needing interactive rebase (`-i`) to manually drop commits.
The deduplication is conservative — git only drops commits when the patch
content is identical, not just the commit message.
