### Symptom
After a bulk `sed 's/OldName/NewName/g'` rename across a markdown file,
a frontmatter-style header line becomes garbled — stray parenthesis
left behind, closing `**` missing. E.g.:

Before: `**Working name:** OldName (permanent note TBD)`
After sed: `**Name: NewName)` ← stray `)`, missing `**`

### Context
Occurs when sed is used to rename a project name that appears inside a
markdown bold marker + colon + value + parenthetical phrase, and a second
sed pass targets part of that same line. The two passes interact unexpectedly
when one pattern leaves partial content the other doesn't account for.

### What was tried (didn't work)
Single chained sed with multiple substitutions — each substitution looks
correct in isolation but the output of one feeds the next unexpectedly.

### Root cause
Markdown bold markers (`**`) and parenthetical values in headers create
strings where a greedy `.*` pattern can match more or less than intended,
leaving surrounding punctuation behind after substitution.

### Fix
After any bulk sed rename on markdown files, immediately check header and
frontmatter lines:
```bash
grep -n "^\*\*\|^---" file.md | head -20
```
For structured header lines, use targeted Edit rather than sed —
find the exact old_string and replace precisely. Reserve sed for
simple word substitution in prose body content.

### Why this is non-obvious
Sed substitutions on plain prose look safe and work correctly. The failure
mode only appears when the matched text sits inside markdown formatting
characters that sed treats as plain text — so the formatting characters
survive but in mangled positions.
