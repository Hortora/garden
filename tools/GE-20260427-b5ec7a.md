### Symptom
A new Maven/Gradle project's initial commit contains hundreds of binary `.class` and `.jar` files under `target/`. `git log --stat` shows the initial commit touching thousands of files. Subsequent `git add -A` continues to track changes in `target/`.

### Context
Creating a new project from scratch: scaffold the directory, write source files, run `mvn install`, then `git add -A && git commit`. The build produced `target/` before the `.gitignore` existed.

### Root cause
`git add -A` stages everything in the working tree that isn't already gitignored. If `target/` exists when `git add -A` first runs and `.gitignore` doesn't yet exclude it, git tracks those files. Adding `.gitignore` afterward prevents *new* files from being tracked but doesn't remove the *already-tracked* ones — they stay in git's index until explicitly removed.

### Fix
**Before** the first `git add`:
```bash
echo "target/\n*.class\n*.jar" > .gitignore
git add .gitignore
git add -A
git commit -m "initial commit"
```

**After** the mistake (to clean up):
```bash
git rm -r --cached target/
git commit -m "chore: remove target/ from tracking"
```
Then verify with `git ls-files | grep target` — should return nothing.

### Why this is non-obvious
Developers with existing projects always have `.gitignore` from the start (cloned or scaffolded). New project creation from scratch is less common and the omission isn't caught until `git log --stat` reveals the damage — often after the repo is already pushed.
