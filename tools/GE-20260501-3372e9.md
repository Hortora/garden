## Use `javap` + `jar/strings` on installed JARs and jandex.idx to diagnose API signature changes and tool registration at the bytecode level

### Stack
Java / Maven / Quarkus / jandex

### The technique

When a SNAPSHOT dependency changes its API and you're seeing confusing runtime errors (`NoSuchMethod`, tools missing, CDI failures), you can inspect the **installed JAR** directly to understand what the current API surface actually is — before touching any source code.

**Step 1 — Find the JAR in ~/.m2**
```bash
find ~/.m2/repository/io/your-org -name "your-module*.jar" 2>/dev/null | grep -v "sources\|javadoc"
```

**Step 2 — Inspect class signatures with javap**
```bash
cd /tmp && mkdir inspect && cd inspect
jar -xf ~/.m2/repository/io/your-org/your-module/version/your-module-version.jar
javap -p io/your/org/TargetClass.class
```

Output shows which methods are public, which are package-private, and their exact signatures. This immediately reveals if a method was renamed, made package-private, or has different parameters than what your code expects.

**Step 3 — Inspect jandex.idx for tool/CDI registration**

For Quarkus MCP or CDI, the jandex index controls what gets registered. Inspect it:
```bash
find /tmp/inspect -name "jandex.idx" | head -3
cat /tmp/inspect/META-INF/jandex.idx | strings | grep -oE "^[a-z][a-z_]{3,}" | sort | uniq | head -60
```

This shows what names/methods are actually indexed — for MCP, it reveals which tool names are registered in that JAR, letting you count them and compare against what the runtime returns.

**Step 4 — Compare expected vs actual tool registrations**

For MCP tool count diagnosis: generate the expected alphabetical list of all `@Tool` names across all dependencies, sort them, and find the cutoff point:
```bash
# Count @Tool annotated public methods in a JAR
grep -rn "@Tool(name" /path/to/qhorus/runtime/src/main/java/ | sed 's/.*name = "\([^"]*\)".*/\1/' | sort | uniq | wc -l
```

Cross-reference with the actual `tools/list` response to find which tools are missing.

### Context / when to use

Use this when:
- A SNAPSHOT dependency was rebuilt and now you have mysterious runtime failures
- `NoSuchMethod` or `IncompatibleClassChangeError` at runtime but not at compile time
- Tools registered at build time are mysteriously absent from the runtime response
- You need to know the exact method signature of a class without reading source code

### Why this is non-obvious

Most developers would start by reading source code or error messages, not by inspecting bytecode. `javap` is a standard JDK tool but rarely used for debugging dependency issues. The combination of `javap -p` (shows package-private methods) + `jar -xf` + `cat | strings | grep` on jandex.idx provides a complete picture of what a JAR actually exports — faster than cloning and reading the source, and guaranteed to match the installed version.

### Why this fix
This is a diagnosis technique, not a fix. It converts confusing runtime symptoms into concrete API surface facts, which then lead to the actual fix (updating call sites, rebuilding modules, adjusting configuration). The technique is especially valuable for SNAPSHOT dependency churn where source and installed JAR can diverge between sessions.

### Alternatives considered
- Reading the source code of the dependency — requires cloning the repo and finding the right version; the installed JAR is the single source of truth for what was actually built
- Reading error messages alone — `NoSuchMethod` messages include the old signature, not the new one; javap reveals both
