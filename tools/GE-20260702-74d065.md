### The technique
Call `ide_open_workspace` with the common parent directory of all repos to open 
them in a single IntelliJ window with full cross-project code intelligence:

```
ide_open_workspace(path="/Users/dev/casehub")
```

This scans the directory for immediate subdirectories containing `pom.xml`, generates 
a temporary Maven aggregator project, and opens it in IntelliJ. The result is a 
single workspace where `ide_find_class`, `ide_search_text`, `ide_find_references`, 
and all other semantic tools work across every module simultaneously.

### When to use it
- Scanning for duplicated patterns across many repos (same class name, same SPI 
  implemented independently)
- Finding all consumers of an API across the organisation
- Cross-repo rename or move impact analysis
- Consolidation planning (which repos implement the same pattern)

### How it was used
Opened 26 CaseHub repos in one window to scan for shared patterns worth extracting 
to a building-blocks library. Fan-out agents searched the workspace in parallel — 
routing patterns, gate/oversight patterns, agent/worker patterns, and config/utility 
patterns — using the workspace `project_path` for all IntelliJ tool calls.

### Caveat
The workspace index can contain stale entries — always verify IntelliJ search results 
against the filesystem before acting on them (see GE-20260702-7c79fe).

### Why this is non-obvious
Most developers open one project at a time or use grep/find for cross-repo searches. 
The MCP plugin's `ide_open_workspace` is not in the standard IntelliJ workflow — it 
generates a synthetic aggregator POM that doesn't appear in the IDE's File > Open 
Recent. The semantic search across 26 repos catches patterns that text search misses 
(subclasses, interface implementations, type hierarchy matches).
