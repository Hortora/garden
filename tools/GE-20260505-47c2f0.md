## Symptom

A `SUCCESS` and `FAILED` counter inside a `while IFS= read -r x; do ... ((SUCCESS++)); done <<< "$VAR"` loop always reads as 0 in the parent shell after the loop completes, even though the loop body definitely executed (side effects like API calls happened).

## Context

Tracking success/failure counts from a bulk API loop driven by a here-string (`<<<`).

## What was tried (didn't work)

- Incrementing `SUCCESS` and `FAILED` inside the loop body
- Checking the values after the loop — always 0
- Adding `echo` inside the loop — confirmed the loop was executing

## Root cause

In zsh (and bash with some configurations), `while ... done <<< "$VAR"` runs the loop body in a **subshell**. Variables modified inside a subshell are not propagated back to the parent shell. The API calls execute (subshell has full access to external commands), but integer variable mutations die with the subshell.

## Fix

Write the multi-line content to a temp file and redirect from that file instead:

```bash
echo "$VAR" > /tmp/items.txt
SUCCESS=0; FAILED=0
while IFS= read -r x; do
  # ... do work ...
  ((SUCCESS++))
done < /tmp/items.txt
echo "Success: $SUCCESS"
```

`while ... done < file` does NOT create a subshell in zsh — the loop runs in the current shell and variable mutations persist.

## Why this is non-obvious

Here-string (`<<<`) and file redirection (`< file`) look like equivalent input mechanisms, but they have different process scoping semantics. The subshell is invisible — no error, no warning, and side effects (API calls) still happen, so the loop appears to work.
