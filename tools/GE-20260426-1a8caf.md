### Context
You encounter a build error after making a change and aren't sure whether
your change caused it or whether it was already broken.

### Technique
```bash
git stash && <build-command> && git stash pop
```

Stashes your changes, runs the build against the clean pre-change state,
and restores your changes. If the build fails with and without your change,
it's pre-existing. If it passes without your change, your change is the cause.

### Why this is non-obvious
Developers often spend significant time investigating build errors that were
already present before their change — tracing down call stacks, reading
compiler errors, second-guessing logic — before realising the error is
pre-existing. The one-liner produces a definitive answer in the time it
takes to build once. Most developers know `git stash` for branch-switching;
using it specifically as a diagnostic isolation tool is a distinct
application of the same command.
