### The problem

A chain of dependent PRs (A→B→C→D, each branching from the previous) needs
rebasing onto a new base after A merges upstream (possibly as a squash commit).
Naively rebasing each branch onto the new upstream/main picks up conflicts from
shared history and re-applies already-merged commits.

### The technique

**Step 1 — Capture all old branch tips before touching anything:**

```bash
TIP_B=$(git rev-parse feat/branch-b)
TIP_C=$(git rev-parse feat/branch-c)
TIP_D=$(git rev-parse feat/branch-d)
```

**Step 2 — Rebase B onto the new base, cutting at the old A tip:**

```bash
# OLD_A_TIP = the tip of branch-a before it was merged/rebased
git checkout feat/branch-b
git rebase --onto upstream/main $OLD_A_TIP feat/branch-b
```

This replays only the commits unique to B (those after `$OLD_A_TIP`) onto the
new base, dropping the now-merged A commits automatically.

**Step 3 — Cascade: rebase each subsequent branch onto the previous:**

```bash
git checkout feat/branch-c
git rebase --onto feat/branch-b $TIP_B feat/branch-c

git checkout feat/branch-d
git rebase --onto feat/branch-c $TIP_C feat/branch-d
```

Each branch gets only its own unique commits replayed. The captured old tips
act as the "cut point" — everything after that tip is the branch-specific work.

**Step 4 — Force-push all branches:**

```bash
for branch in feat/branch-b feat/branch-c feat/branch-d; do
  git push --force-with-lease origin $branch
done
```

### Why this is non-obvious

The instinct is to rebase each branch directly onto upstream/main, but that
fails: the branch still carries all the A commits (now in upstream as a squash),
causing conflicts. The `--onto` flag is the key — it sets a new base while the
positional `<upstream>` argument controls what to exclude. Using the old branch
tips as cut-points isolates exactly the unique commits per branch.

Capturing the old tips first is essential: after rebasing B, `$TIP_B` changes
— you need the pre-rebase value to correctly cut C.

### Constraints
Works cleanly when each branch has a linear history and the merged content
lands upstream as a squash or identical patch. Parallel branches with conflicting
changes still require manual resolution per branch.
