## Maven SNAPSHOT jar from sibling branch persists in .m2 — tests fail with constructor mismatch

### Symptom

Tests fail with constructor argument count mismatch (e.g. `required 7 args, found 3`) after switching branches. The test code looks correct; the class definition in the source tree looks correct. The failure makes no sense.

### Root cause

A sibling branch installed a structurally different version of the same SNAPSHOT artifact (e.g. added a record field, changing a 3-arg constructor to 7-arg). Maven SNAPSHOTs are mutable — the installed jar in `~/.m2` reflects whichever branch last ran `mvn install`. Switching branches changes the source but not the cached jar; the compiler uses the source while the test runner uses the installed jar, producing a mismatch.

### Fix

After switching to any branch that shares a SNAPSHOT artifact with another branch, reinstall the module from the current branch's source before running tests:

```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn install -pl <api-module> \
  -f /path/to/root/pom.xml --no-transfer-progress -q
```

### Why this is non-obvious

The symptom points to the test or the source code — not the jar cache. The compiler and the test runner operate against different artifacts (source vs installed jar), which is invisible unless you know to check `~/.m2`. The branch switch is the cause, but it happens at a different time from the failure.
