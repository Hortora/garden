## What It Is

To test whether a token has delete permission on a resource without actually deleting anything, send a DELETE request to a known-nonexistent resource ID. The response reveals the permission check result:

- **404 Not Found** — the API checked authorisation first, passed, then looked up the resource and couldn't find it. **Permission confirmed.**
- **403 Forbidden** — the API rejected the request at the authorisation layer before even looking up the resource. **Permission denied.**

## Context

Useful when writing diagnostic scripts that need to verify delete scope on a token without side effects, and when no safe "list my permissions" endpoint exists. Works for any REST API that performs authorisation before existence checks (GitHub, most well-designed APIs).

## Example — GitHub Packages

```bash
http_code=$(curl -s -o /dev/null -w "%{http_code}" \
  -X DELETE \
  -H "Authorization: Bearer $GH_TOKEN" \
  -H "Accept: application/vnd.github+json" \
  "https://api.github.com/orgs/$ORG/packages/maven/__probe_nonexistent__/versions/0")

if [ "$http_code" = "404" ]; then
  echo "✓ delete:packages — permission confirmed"
elif [ "$http_code" = "403" ] || [ "$http_code" = "401" ]; then
  echo "✗ delete:packages — permission denied"
fi
```

## Why This Is Non-Obvious

Most developers either (a) check permissions by trying a real delete and accepting the risk, (b) look for a permissions endpoint that often doesn't exist, or (c) inspect the token's OAuth scopes via response headers. This approach requires knowing that well-designed APIs check auth before existence — and that 404 therefore implies authorisation was granted.
