### Symptom

Commits that were intended for main land on a feature branch. After discovering this, the branch history is contaminated with unrelated commits.

### Context

Working across multiple branches in a long session. `git stash` is used to temporarily set aside changes (e.g. to run validation on the clean working tree). After `git stash pop`, work continues — but on the feature branch, not main.

### What was tried (didn't work)

The natural assumption is that stash is branch-independent — it saves and restores changes to the working tree regardless of which branch you end up on. So after `git stash pop`, continuing to commit feels safe.

### Root cause

`git stash save` records the current branch name. `git stash pop` restores the working tree but does NOT switch branches — you remain on whatever branch you were on when the stash was applied. If that was a feature branch, you stay on it.

In a long session, the branch context is easy to lose track of — especially when stash/pop was used as a quick "clean state" check mid-session.

### Fix

Cherry-pick the accidentally-landed commits to the correct branch:

```bash
# Identify the commit(s) that landed on the wrong branch
git log --oneline <feature-branch> ^main | head -5

# Switch to the target branch
git checkout main

# Cherry-pick each commit
git cherry-pick <sha>

# Remove the commit from the feature branch
git checkout <feature-branch>
git reset --hard HEAD~1  # if it was the last commit
```

Prevention: after any stash pop, check `git branch --show-current` before committing.

### Why this is non-obvious

Stash feels like a clipboard — branch-agnostic save/restore. The connection between stash and current branch is not surfaced anywhere in the normal stash workflow. The pop confirmation message doesn't mention the current branch.
