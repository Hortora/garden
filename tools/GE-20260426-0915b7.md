## What it is

When wrapping a backend service in an Electron app, poll a domain-specific ready signal rather than just checking for HTTP 200 on the health endpoint. The service may respond to HTTP before it has finished initialising its application state.

## Context

A Quarkus replay service starts in ~1.7s (JVM boot) but takes an additional ~2s to parse replay files and load 22,000+ GAME_EVENTS into memory. Polling `/q/health` (HTTP 200) fires at 1.7s — the Electron window loads the visualiser before the game loop has run its first tick, producing a black screen because the WebSocket delivers no data yet.

## The technique

Poll a business-level endpoint that only returns a valid response once the application is genuinely ready for user interaction:

```javascript
function isReplayReady(cb) {
  const req = http.get('http://localhost:8080/qa/replay/status', res => {
    if (res.statusCode !== 200) { cb(false); return; }
    let body = '';
    res.on('data', d => body += d);
    res.on('end', () => {
      try { cb(JSON.parse(body).totalLoops > 0); }
      catch { cb(false); }
    });
  });
  req.on('error', () => cb(false));
  req.setTimeout(1000, () => { req.destroy(); cb(false); });
}
```

`totalLoops > 0` means the replay file is fully parsed and at least one scheduler tick has completed — the WebSocket will immediately deliver a valid game frame.

## Why this is non-obvious

Health endpoints (`/q/health`, `/actuator/health`) are designed to signal that the process is alive, not that application-level initialisation is complete. The two can be seconds apart. Most examples show polling health — the gap is only visible when the app loads UI before the data is ready.
