### The approach

When a CI build fails because a SNAPSHOT jar changed its API ("method X cannot be applied to given types"), use Python's `zipfile` + `javap` to read the current method signatures directly from the cached jar — no source access required:

```python
import zipfile, subprocess, os

jar = '/Users/you/.m2/repository/io/example/mylib/0.2-SNAPSHOT/mylib-0.2-SNAPSHOT.jar'

# Find the class of interest
with zipfile.ZipFile(jar) as z:
    names = [n for n in z.namelist() if 'TargetClass' in n and '.class' in n]
    print(names)
    if names:
        data = z.read(names[0])
        with open('/tmp/TargetClass.class', 'wb') as f:
            f.write(data)

result = subprocess.run(
    ['javap', '-classpath', '/tmp', '-p', 'TargetClass'],
    capture_output=True, text=True
)
print(result.stdout)
```

Output shows the full public API with exact signatures:

```
public interface io.example.TargetRepository {
  public abstract java.util.Optional findById(java.util.UUID, java.lang.String);
  public abstract io.example.Entry save(io.example.Entry, java.lang.String);
}
```

Compare to what your code calls to identify what changed.

### Why this is non-obvious

The natural instinct is to open the jar as a ZIP manually or try to add `-sources` to the dependency. `zipfile` + `javap` is faster: extract one class to `/tmp`, decompile. Works for any `.class` in any `.jar` including SNAPSHOTs where the source jar may not be published.

For interfaces, `javap -p` shows all abstract methods. For classes, it shows public and private (`-p`) members including the ones injected by CDI frameworks.

### Use cases

- **SNAPSHOT API drift**: downstream breaks — find what method signatures changed
- **CDI injection mismatch**: "unsatisfied dependency" errors — check if a library bean has new required injections
- **Undocumented library behavior**: check what `@ApplicationScoped` beans are exposed by a SNAPSHOT extension you didn't write
