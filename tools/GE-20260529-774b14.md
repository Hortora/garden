## Symptom

A subagent dispatched via the `Agent` tool without `isolation: "worktree"` commits its work to a branch named `worktree-agent-<id>` instead of the expected project branch. The main branch has no trace of the work.

## Context

In subagent-driven development, when an Agent is dispatched to implement a task, it may invoke `EnterWorktree` internally (e.g. if it decides to isolate its work). The worktree is a copy of the repo at an older state — before recent commits on the main branch landed. Commits in the worktree land on `worktree-agent-<id>`, not the task's working branch.

## Root Cause

The `Agent` tool's `isolation: "worktree"` parameter controls whether the *dispatcher* creates a worktree. It does not prevent the *agent itself* from calling `EnterWorktree`. If the agent independently decides to isolate, it will create its own worktree regardless of how it was dispatched.

## Fix

Explicitly instruct the agent in the prompt: "Do NOT create or use any git worktrees. All edits and commits go directly to `<project-path>` on branch `<branch-name>`." When the agent commits to a worktree branch instead of the expected branch, copy the new files manually and commit to the correct branch.

## Why This Is Non-Obvious

The distinction between "dispatched without isolation" (the controller didn't create a worktree) and "agent cannot create a worktree" (the agent is prohibited from doing so) is not obvious. Dispatching without `isolation: "worktree"` looks like it means "no worktree" — but it only controls the dispatch mechanism, not the agent's own tool calls.
