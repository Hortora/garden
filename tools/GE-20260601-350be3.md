### Symptom
`gh pr merge --rebase` fails: "Pull request is not mergeable: the merge commit cannot be cleanly created."
GitHub UI shows the PR as `CONFLICTING` / `DIRTY`. No file-level conflicts exist — the diff looks clean.

### Context
Fork model: `origin` = personal fork, `upstream` = org repo. Fork's `main` accumulated
many commits that were never merged back to upstream. A feature branch was created from
fork `main` — so its history includes commits upstream has never seen. The PR targets
upstream `main`.

### Root cause
GitHub cannot rebase the branch onto upstream `main` because the branch's base commit
does not exist in upstream's history. It's not a content conflict — it's a history
divergence. The error message implies file conflicts; there are none.

### Fix
Cherry-pick the branch's meaningful commits onto a fresh branch based from upstream `main`:

```bash
git checkout -b clean-branch upstream/main
git cherry-pick <sha1> <sha2> <sha3>    # the commits to deliver
git push origin clean-branch
```

Close the original PR, raise a new one from `clean-branch`. The new PR targets upstream `main` cleanly.

### Why this is non-obvious
The error "merge commit cannot be cleanly created" strongly implies file-level conflicts.
Standard conflict resolution (checking `git diff`, resolving files) finds nothing.
The actual cause — the branch is based on fork commits invisible to upstream — is not
surfaced by GitHub, `gh pr view`, or the merge error message.
