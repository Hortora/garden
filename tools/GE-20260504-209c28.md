## Symptom

`git push --all` completes with exit code 0 and appears to succeed, but some
branches were not pushed — they show `! [rejected]` in the output due to
non-fast-forward conflicts.

## Root cause

`git push --all` treats partial success as success for exit code purposes. Individual
branch rejections are reported in stderr but do not cause the command to exit non-zero
when at least some branches pushed successfully.

## Fix

Always check the output for `! [rejected]` lines after `git push --all`:

```bash
git push --all 2>&1 | grep "rejected\|error"
```

For a hard failure on any rejection, push branches individually or pipe through a
check:
```bash
git push --all 2>&1 | tee /tmp/push.log
grep -q "rejected" /tmp/push.log && echo "SOME BRANCHES FAILED"
```

For non-fast-forward branches where you intend to overwrite (e.g. archiving a fork):
```bash
git push --all --force-with-lease
```

## Why this is non-obvious

The exit code 0 masks the failure for any script or CI step checking `$?`. The
rejected branches appear only in the terminal output, which is easy to miss in
a long push operation with many branches.
