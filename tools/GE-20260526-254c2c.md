When Claude Code evaluates a bash for loop containing `git` commands, the shell inside the eval context cannot find the `git` binary even though it is available on the PATH in normal shell usage. The error is `command not found: git` appearing after each loop iteration, with the echo outputs still printing. The commands silently fail without aborting.

Example of what fails:
```bash
for repo in work devtown aml; do
  git -C /path/$repo branch --show-current
  echo "$repo done"
done
```

This produces: `(eval):3: command not found: git` for each iteration.

### Root cause

Claude Code's shell evaluation for for loops runs in a restricted environment where the PATH is not fully inherited for complex multi-statement blocks. The security sandbox that processes commands for auto-approval also affects how the shell environment is initialised inside loops.

### Fix

Replace for loops with explicit `&&`-chained individual commands on separate lines, or write a Python script to `/tmp/` and execute it:

```bash
git -C /path/work branch --show-current && git -C /path/devtown branch --show-current && git -C /path/aml branch --show-current
```

Or for complex iteration, write to `/tmp/script.sh` and run `bash /tmp/script.sh`.

### Why this is non-obvious

The `git` binary works fine in single-command calls. The failure only occurs inside for/while loops in the eval context, and the error message ("command not found: git") is misleading — git IS installed and available. The same loop would work in a regular terminal.
