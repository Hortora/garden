### Symptom

Browser shows unexpected HTML content (wrong page title, wrong component tags) when navigating to the expected dev server URL. The dev server appears to be running and responding on the correct port, but the content is from a completely different application.

### Context

A Quarkus dev server configured on port 8090. When started, it appeared to work — `curl` returned HTML and a 200 status. But the HTML contained `<aml-app>` instead of the expected `<div id="app">`. The page title was wrong. The JavaScript didn't load.

Three stale processes (a previous Java instance, a Node process, and a Python SimpleHTTPServer) were all listening on port 8090. The first process to bind wins the port; the Quarkus server either failed to bind (silently in background mode) or bound on a different interface.

### What was tried (didn't work)

- Assumed the server was running correctly because curl returned 200
- Debugged the JavaScript entry point and HTML files
- Checked Vite configuration and build output

### Root cause

Multiple processes from previous dev sessions were left running on the same port. `curl` received a response from Python's SimpleHTTPServer (which was serving files from a different directory), not from Quarkus. The `Server: SimpleHTTP/0.6 Python/3.14.5` response header revealed this immediately — but wasn't checked.

### Fix

Before debugging app content, always check the Server response header:

```bash
curl -sI http://localhost:8090/ | grep -i "server"
# Expected: vert.x (Quarkus), or similar
# Red flag: SimpleHTTP/0.6 Python, or any unexpected server
```

If the wrong server is responding, find and kill stale processes:

```bash
lsof -i :8090 | head -5
# Kill all listeners
kill <pid1> <pid2> <pid3>
```

### Why this is non-obvious

The symptom (wrong HTML content) points toward build configuration, entry points, or JavaScript bugs — not toward "a different server is responding." Developers instinctively debug their own code, not the environment. The 200 OK response reinforces the assumption that the right server is running. Checking the Server header is a 2-second diagnostic that most developers skip.
