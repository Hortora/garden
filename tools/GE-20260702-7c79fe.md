### Symptom
`ide_find_class` returns a fully-formed result (qualified name, file path, line number, 
column) for a class that does not exist on disk or in git history. The result is 
indistinguishable from a legitimate hit.

### Context
Multi-repo workspace opened via `ide_open_workspace` with 26 CaseHub Maven repos. 
Searching for `DefaultTrustRoutingPolicyProvider` returned a hit in `casehub-ledger` 
with `line: 28, column: 14` — looked completely real.

### What was tried (didn't work)
- Trusted the IntelliJ result and included ledger in affected-repos documentation
- Attempted to read the file via `ide_read_file` — file not found
- Attempted `ide_find_definition` at the reported position — no element found

### Root cause
IntelliJ's class index caches entries from previous indexing runs. When a class is 
deleted or was never committed, the index may retain a stale entry. In a large 
workspace with 26 modules, the index is rebuilt less aggressively, increasing the 
window for phantom entries.

### Fix
Always verify IntelliJ search results against the filesystem before acting on them:
```bash
find /path/to/repo -name "ClassName.java" -type f
git -C /path/to/repo log --all --oneline -- '**/ClassName.java'
```

If both return empty, the IntelliJ result is a phantom. `ide_sync_files` on the 
affected path may clear the stale entry, but verification before action is the 
reliable defence.

### Why this is non-obvious
The result includes every field a real result would have — qualified name, file path, 
line, column, language, kind. There is no "stale" or "unverified" flag. A developer 
trusting the IDE's code intelligence (which is the entire point of using it) would 
naturally act on the result without cross-checking the filesystem.
