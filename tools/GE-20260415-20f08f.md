## Symptom

REST API calls made via `page.request().post(url, ...)` return 401 even though
`page.setExtraHTTPHeaders(Map.of("X-Api-Key", key))` was called before navigation.
The page itself (HTML, CSS, fetch() calls in page scripts) authenticates correctly.

## Context

Writing Playwright Java E2E tests against a Quarkus server that uses `X-Api-Key` header
authentication. The base class sets the auth header via `page.setExtraHTTPHeaders()`.
Tests use `page.request()` to create test data via REST before asserting on the UI.

## What was tried (didn't work)

Setting headers via `page.setExtraHTTPHeaders()` and then calling
`page.request().post(url, RequestOptions.create().setData("..."))` — the API key
header is absent from the request, returning 401.

## Root cause

`page.setExtraHTTPHeaders()` adds headers to HTTP requests made *by the page*: page
navigation requests, and `fetch()` / `XMLHttpRequest` calls from page scripts. It does
**not** affect `page.request()`, which is bound to the `BrowserContext` and operates
as a separate HTTP client independent of the page's header configuration.

## Fix

Add the auth header explicitly to every `page.request()` call:

```java
// Does NOT work — headers set on page don't flow to page.request()
page.setExtraHTTPHeaders(Map.of("X-Api-Key", apiKey));
page.request().post(url, RequestOptions.create().setData(body));  // → 401

// Works — explicit header on the request
page.request().post(url, RequestOptions.create()
        .setHeader("X-Api-Key", apiKey)
        .setData(body));  // → 200
```

Alternatively, create a dedicated `APIRequestContext` with headers set at the context level:

```java
try (var apiContext = playwright.request().newContext(
        new APIRequest.NewContextOptions()
                .setBaseURL(BASE_URL)
                .setExtraHTTPHeaders(Map.of("X-Api-Key", apiKey)))) {
    apiContext.post("/api/sessions", RequestOptions.create().setData(body));
}
```

## Why this is non-obvious

The Playwright docs describe `setExtraHTTPHeaders()` as applying to "all requests made
by the page" — which sounds like it should include `page.request()`. The distinction
between "requests made by page scripts" and "requests made via the page's API request
context" is buried in the internals. Nothing in the error indicates the header was
missing — just a 401 from the server.
