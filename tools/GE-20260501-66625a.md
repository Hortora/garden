## Symptom

Running `git add <specific-file> && git commit -m "..."` in a repo commits far more files than expected — other files that were already staged (by another session or user) are included in the commit.

## Context

Multi-agent workflow where several Claude sessions work on different files in the same git repo. One session staged (but not yet committed) feature work. Another session ran `git add .github/workflows/maven.yml && git commit` to apply a 1-line CI fix — and the commit included 15 staged feature files from the other session, all under the wrong commit message.

## Root cause

`git commit` without `-a` commits **everything in the staging area (index)**, not just what the current `git add` call staged. If another session had previously run `git add` without committing, those staged files sit in the shared index and will be swept into the next `git commit` by anyone.

## Fix

Before committing in a shared repo, always check what's staged:

```bash
git diff --cached --name-only   # see all staged files before committing
```

If unexpected files appear, either:
- `git restore --staged <file>` to unstage them before your commit, or
- Coordinate with the other session to commit their work first

## Why non-obvious

`git add <specific-file>` feels like it should scope the commit to that file. It doesn't — it only adds to the index. The index is shared state. Any prior `git add` from any session/user accumulates there until committed or unstaged.

**Why:** Lost a clean commit boundary — CI fix bundled with unrelated feature work under wrong message. The other session's staged changes were committed without their knowledge.

**How to apply:** Always run `git diff --cached --name-only` before committing in any repo where other agents or users may be working concurrently.
