Three independent checks in the `project-init` fast-path can return wrong values even when
the workspace is correctly set up.

---

### Failure mode 1 — Issue tracking grep misses bold-markdown format

**Symptom:** `project-init` fast-path returns `ISSUES_OK=no` even when issue tracking is correctly
configured in `CLAUDE.md` — triggering unnecessary setup prompts every session.

**Context:**
Occurs when `CLAUDE.md` uses standard markdown bold formatting:
```
**Issue tracking:** enabled
```
The fast-path script checks:
```bash
grep -q "Issue tracking: enabled" CLAUDE.md
```
This search fails because `**Issue tracking:**` wraps the colon inside bold markers.

**Root Cause:** Grep targets `"Issue tracking: enabled"` but the file contains
`"**Issue tracking:** enabled"`. The markdown delimiters are visually invisible when rendered
but break substring matching in raw text.

**Fix:**
```bash
grep -qi "Issue tracking.*enabled" CLAUDE.md
```

**Why non-obvious:** The bold asterisks render invisibly in a markdown viewer — the CLAUDE.md
appears to contain exactly what grep is searching for.

---

### Failure mode 2 — Workspace check looks for `wksp/` in the wrong repo

**Symptom:** `project-init` fast-path returns `WORKSPACE_OK=no` even when the workspace is
correctly configured — triggering workspace setup prompts every session.

**Context:**
Some workspace setups place `wksp/` in the **project repo** (pointing back to the workspace)
rather than in the **workspace** (pointing to the project repo). The fast-path check:
```bash
{ [ -L "wksp" ] && [ -d "wksp" ]; }
```
runs in the workspace CWD. If the layout is `workspace/proj → project` and
`project/wksp → workspace`, the CWD check finds no `wksp` symlink and returns false.

**Root Cause:** The skill assumes `wksp/` lives in the workspace directory. The casehub workspace
convention is the reverse: `proj/` in the workspace, `wksp/` back-link in the project repo.

**Fix:**
```bash
{ [ -L "proj" ] && [ -d "proj" ]; } || { [ -L "wksp" ] && [ -d "wksp" ]; }
```

**Why non-obvious:** Both layouts are valid. The skill documents `wksp/` but not that it expects
the symlink in the workspace specifically. The failure is a silent `no`, not an error.

**See also:** [GE-20260511-4e76ab](GE-20260511-4e76ab.md) — cc-praxis workspace hook false-negative — checks wrong CLAUDE.md

### Failure mode 3 — ctx.py regex captures closing `**` instead of repo name

**Symptom:** `OWNER_REPO=**` (two asterisks, not the actual repo name) in ctx.py output,
alongside `ISSUES_OK=no`. The `OWNER_REPO=**` value is the more diagnostic symptom.

**Context:**
Occurs when `CLAUDE.md` uses bold format for the GitHub repo field:
```
**GitHub repo:** casehubio/parent
```

ctx.py extracts the repo with:
```python
m = re.search(r"GitHub repo:\s*(\S+)", claude_text)
owner_repo = m.group(1) if m else ""
```

The regex finds `GitHub repo:` correctly, but `\s*(\S+)` then captures the first
non-whitespace after the colon — which is `**` (the closing bold marker, placed
immediately after the colon with no space). The actual repo name sits after a space
the regex has already consumed past.

**Root Cause:** `**GitHub repo:**` places the closing `**` immediately after `:`,
before any whitespace. The greedy `\S+` captures the asterisks before reaching
the value.

**Fix:** Remove bold markers from `## Work Tracking` in CLAUDE.md:
```
Issue tracking: enabled
GitHub repo: casehubio/parent
```

**Why non-obvious:** `OWNER_REPO=**` reads like an unresolved glob or shell variable,
not a markdown formatting artifact. The co-symptom `ISSUES_OK=no` redirects investigation
toward issue tracking configuration, away from the markdown formatting root cause.
The markdown renders invisibly in any viewer — the CLAUDE.md appears correct on screen.
