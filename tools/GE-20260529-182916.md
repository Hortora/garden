### Symptom
`project-init` fast-path check returns `ISSUES_OK=no` even when issue tracking is correctly
configured in `CLAUDE.md` — triggering unnecessary setup prompts every session.

### Context
Occurs when `CLAUDE.md` uses standard markdown bold formatting for the issue tracking declaration:
```
**Issue tracking:** enabled
```
The fast-path script checks:
```bash
grep -q "Issue tracking: enabled" CLAUDE.md
```
This substring search fails because `**Issue tracking:**` wraps the colon inside bold markers —
the exact substring `"Issue tracking: enabled"` never appears in the file.

### Root Cause
The grep pattern targets `"Issue tracking: enabled"` (colon, space, enabled) but the file
contains `"**Issue tracking:** enabled"` (asterisks, colon, asterisks, space, enabled). The
markdown delimiters are visually invisible when rendered but break substring matching in raw text.

### Fix
Match the bold-markdown format in the fast-path check:
```bash
grep -qE "\*\*Issue tracking:\*\*\s*enabled" CLAUDE.md
```
Or use a looser pattern tolerant of either format:
```bash
grep -qi "Issue tracking.*enabled" CLAUDE.md
```

### Why this is non-obvious
The bold asterisks `**...**` render invisibly in a markdown viewer. Reading the CLAUDE.md
visually shows "Issue tracking: enabled" — indistinguishable from the string the grep
searches for. The mismatch is only visible by inspecting the raw file, and the failure
is silent (no error — just a wrong boolean result).
