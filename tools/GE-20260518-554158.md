## Symptom

After `git add -A` or `git add .`, `git status` shows a new entry in mode `160000`:

```
new file:   platform (mode 160000)
```

The commit message says something like `create mode 160000 platform`. On `git push`, the remote sees a gitlink entry pointing to a commit hash that doesn't exist in the outer repo's history. Other clones cannot resolve the reference.

## Context

A directory named `platform/` (or similar) exists in the working tree and happens to be a separate git repository (contains a `.git` folder). It was not registered as a submodule. `git add -A` silently stages it as a gitlink — a submodule reference — without warning.

Git does print a hint:
```
warning: adding embedded git repository: platform
hint: You've added another git repository inside your current repository.
```
But this hint is easy to miss when it appears between other output lines.

## Root cause

Git's `add` command detects embedded repos via the presence of a `.git` directory. When the nested repo is not registered in `.gitmodules`, Git stages it as a gitlink anyway. The `advice.addEmbeddedRepo` hint is printed but not treated as an error.

## Fix

**Immediate:** Unstage and ignore:
```bash
git rm --cached platform
echo "platform/" >> .gitignore
git add .gitignore
git commit -m "chore: ignore platform embedded repo"
```

**If already committed:** Same as above — `git rm --cached` removes the gitlink from the index while leaving the directory on disk.

**Prevention:** Add the directory to `.gitignore` before it gets staged. Once `platform/` is in `.gitignore`, `git add -A` will not stage it even if it contains a `.git` folder.

## Why this is non-obvious

The hint is printed but not prominently flagged. The resulting commit looks correct in `git log`. The problem only manifests when other developers clone or when CI tries to checkout — they get a broken submodule reference pointing to an object that doesn't exist in the outer repo.
