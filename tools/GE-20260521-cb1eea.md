### Symptom
`git diff main...epic-branch --stat` shows 1000+ insertions across 60+ files, suggesting the branch has substantial content not on main — but main already has all the feature code.

### Context
Branch was integrated via cherry-pick before main evolved further. The branch diverged early; main has since accumulated many more commits on the same files.

### Root cause
Three-dot diff (`A...B`) computes: `diff <merge-base(A,B)>..<tip-of-B>`. It shows what the branch contributed *since it split from main* — not what's currently absent from main. If main has evolved past the branch (added more to the same files), those main-side additions don't appear in the three-dot diff at all. The large diff is the branch's historical changes versus the old common ancestor, most of which main already has in a more evolved form.

### Fix
To check what is genuinely on the branch but absent from main, use two-dot diff in the branch→main direction:

```bash
# What does main have that the branch lacks? (if large: branch is behind)
git diff epic-branch main -- path/to/KeyFile.java

# Is there any code on the branch NOT on main?
git diff epic-branch main -- '*.java' | head -50
```

If `git diff branch main` shows main has additions the branch lacks (main is ahead), the branch is behind main — not ahead of it. The three-dot diff was misleading.

### Why this is non-obvious
Three-dot diff is the recommended way to see "what a branch contributed" — which is correct for active branches. But for cherry-picked branches on old divergence points, the common ancestor is far back, making the diff enormous and apparently alarming. The fix direction (branch→main, two-dot) is counterintuitive because you are used to reading diffs as main→branch.
