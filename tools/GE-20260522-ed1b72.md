### Symptom

A protocol file exists on disk in a repo's `docs/protocols/` directory but never appears in `git log` — it was written by a Claude session in a different project context that skipped committing. The file is visible via `ls` but invisible to git history.

In a concrete case: `jq-evaluation-canonical.md` was untracked in **both** the source repo and the shared garden simultaneously — two separate sessions had written it independently, and neither committed it.

### Context

Occurs in multi-repo ecosystems where multiple Claude Code sessions write shared artifacts (protocol files, garden entries) to a central store. Any session that writes a file but closes without committing — due to interruption, branch discard, or skipped wrap step — leaves the file orphaned on disk.

### Root Cause

Untracked files survive on disk across branch switches and session restarts. `git status` shows them; `git log` does not. In multi-repo setups the same file can be written independently by sessions operating in different project directories, making the duplication invisible until both repos are checked explicitly.

### Fix

Two mitigations:

1. **Session wrap check:** run `git ls-files --others --exclude-standard | grep docs/protocols/` in each relevant repo before ending a session. Flag any untracked protocol files for commit or discard.

2. **Periodic cross-repo audit:** at the start of any maintenance session, run `git status --short` across all repos in the ecosystem to surface untracked artifacts before they are overwritten or lost.

### Why This Is Non-Obvious

The normal mental model for multi-session work assumes each session commits what it writes. Untracked files are invisible to the next session unless it explicitly checks `git status` — they don't appear in diffs, logs, or branch comparisons. In a central store (garden, protocol directory) with many contributors, the accumulation is gradual and silent.
