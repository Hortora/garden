## Context

After manually deleting tracked files (with `rm`, not `git rm`) and placing replacement files in different locations, you need to stage the deletions and have git recognise them as renames rather than separate delete+add pairs — preserving file history.

## Technique

```bash
git add -u <dir>
```

`git add -u` (update index) stages all modifications and deletions to already-tracked files without staging any new untracked files. When combined with `git add <new-files>`, git's rename-detection algorithm matches content and records them as renames.

```bash
# After: rm old-files, place new-files in new locations
git add -u docs/adr/      # stage deletions of tracked files
git add docs/specs/        # stage the new files
git status                 # shows: renamed: old -> new (not deleted + added)
```

## Why This Is Non-Obvious

Most developers reach for `git rm` to stage deletions of tracked files. `git add -u` is less known but handles bulk deletions cleanly and, crucially, triggers git's content-similarity rename detection — correctly recording history as a rename rather than breaking file history with a delete+add pair.
