### Symptom

A `git rebase -i` run with a pre-written todo file that includes `drop <merge-commit-sha>` prints `BUG: sequencer.c:2725: unexpected todo_command` to stderr and exits with code **0**. The branch is unchanged. No rebase happened.

### Context

Automating `git rebase -i` non-interactively via `GIT_SEQUENCE_EDITOR="cp $todo"` to squash a commit range that contains merge commits (2-parent commits). The todo includes `drop` lines for the merge commits and `pick`/`fixup` for the regular commits.

### What was tried

Using `drop <sha>` for merge commits in the rebase todo, same as for regular commits. Exit code 0 was interpreted as success. The branch was inspected and found unchanged — the rebase had silently done nothing.

Also tried `fixup <merge-sha>` — this fails with an explicit error: `cannot squash merge commit into another commit`. That failure is at least visible.

### Root cause

Git's interactive rebase sequencer does not support `drop` for merge commits in standard (non-`--rebase-merges`) mode. The `drop` command is valid for regular commits but is classified as an "unexpected" command when applied to a merge commit, triggering the BUG path. The sequencer exits cleanly (code 0) without processing the todo.

### Fix

Use `--no-rebase-merges` flag, which linearises history by **implicitly skipping all merge commits** — they do not appear in the generated todo at all:

```bash
GIT_SEQUENCE_EDITOR="cp $todo" git rebase -i --no-rebase-merges <base-sha>
```

The todo file should only list the non-merge commits (regular 1-parent commits). Merge commits are neither picked nor dropped — they are simply absent from the linearised result. The combined content of each merge commit (what its diff introduced relative to its first parent) is preserved via the commits already in the todo.

### Why this is non-obvious

Exit code 0 strongly implies success. `BUG:` in stderr looks alarming but is easy to miss when capturing output. There is no "rebase did nothing" signal. The branch state must be inspected explicitly to detect the no-op. `fixup` on a merge commit at least fails with a clear error; `drop` fails silently.
