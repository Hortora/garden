## Context

When a type or enum is needed in repo A that currently lives in repo B, the instinct is to immediately add a dependency from A to B. Before doing this, explore the full dependency graph and ownership question first — the answer often reveals a better home for the type.

## The technique

Before adding any cross-repo dependency, answer these questions in order:

1. **Does A already depend on B?** Check all module `pom.xml` files, not just the root. A dependency that exists in one module but not in another is easy to miss.

2. **Does B depend on A?** If yes, adding A → B creates a cycle. A third home is required.

3. **Who else would need this type?** If only A and B need it today, a peer dependency (A → B) may be fine. If C, D, and E will need it, a shared artifact (`*-protocol`, `*-common`) or a dedicated repo is warranted.

4. **Is the type conceptually owned by B, or is it a shared protocol?** If the type encodes a shared vocabulary (e.g. speech-act types used by a communication mesh), it may belong in a neutral location that neither A nor B "owns".

5. **What's the cost of getting it wrong?** Adding a dependency is easy; removing one later (after downstream repos depend on your artifact) is hard. Investing 30 minutes now to explore the right home avoids a future refactor.

## Why this is non-obvious

The natural move when you need a type is to find where it lives and add a dependency. The exploration step — checking both directions of the dependency graph, considering future consumers, and evaluating conceptual ownership — is rarely done explicitly. Skipping it leads to sideways coupling between peer systems (e.g. engine depending on qhorus-api for a speech-act vocabulary that belongs to neither), which is hard to unwind once downstream consumers exist.

## Outcome in practice

In one session, this technique produced three distinct tracked issues before any code was written:
- A pragmatic short-term decision (A → B for now)
- A future extraction issue (move type to dedicated protocol repo when a third consumer appears)
- A threading follow-on (populate the new field at a higher level rather than passing null everywhere)

All three decisions were explicit, reasoned, and tracked — rather than implicit in a one-line `pom.xml` change.

### Alternatives considered

- Just add the dependency and refactor later — works but creates unplanned cross-repo breaking changes
- Define the type in both repos and translate at the bridge — duplication; translation logic leaks everywhere
- Extract immediately to a third repo — correct end state but overkill when only two consumers exist today (YAGNI violation)
