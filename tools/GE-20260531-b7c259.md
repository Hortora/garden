## Symptom

After calling `focusOnFirstGeyser()` (or any function that repositions the camera via `updateCamera()`), `samplePixel()` returns the colour from the *previous* camera position — not from the newly focused sprite. The pixel value is visually wrong but consistent (same wrong colour every run), making it look like a coordinate error rather than a timing issue.

## Context

Writing Playwright pixel-colour assertions against a Three.js scene. `window.__test.focusOnFirstGeyser()` calls `updateCamera()` and returns screen coords. `window.__test.samplePixel(x, y)` immediately reads the WebGL canvas buffer at those coords. `preserveDrawingBuffer: true` is set on the renderer.

## What was tried

- Confirmed `preserveDrawingBuffer: true` (so the buffer isn't cleared between frames)
- Verified screen coords returned by `focusOnFirstGeyser()` were correct
- Checked the Three.js scene contained the expected sprite
- Sampled multiple pixels around the expected location — all returned terrain/background colour

## Root cause

`updateCamera()` repositions the camera in JS memory. The WebGL pixel buffer reflects the *last rendered frame*, not the current camera state. With `preserveDrawingBuffer: true`, the buffer persists across frames — but only until the next `renderer.render()` call. Until that call fires (which happens asynchronously in the `requestAnimationFrame` loop), `samplePixel()` reads pixels rendered under the previous camera transform.

In a `page.evaluate()` call, no animation frames fire — so the camera move is never rendered before the pixel read.

## Fix

Call `renderer.render(scene, camera)` synchronously inside the focus helper before computing and returning screen coordinates:

```js
focusOnFirstGeyser: () => {
  const sp = geyserMeshes.values().next().value;
  if (!sp) return null;
  camTarget.set(sp.position.x, 0, sp.position.z);
  tDist = camDist;
  updateCamera();
  renderer.render(scene, camera);   // ← flush the new camera state to the pixel buffer
  const v = sp.position.clone().project(camera);
  const sz = renderer.getSize(new THREE.Vector2());
  return { x: Math.round((v.x+1)/2*sz.width), y: Math.round((-v.y+1)/2*sz.height) };
},
```

This is safe because `preserveDrawingBuffer: true` means the synchronous render holds in the buffer until the next rAF cycle overwrites it. The rAF cycle renders identical content (no game state change between the two calls), so no visual artifact.

## Why this is non-obvious

With `preserveDrawingBuffer: true` already set, developers expect the buffer to be "current". The distinction between "camera repositioned" and "frame rendered with new camera" is invisible until you understand that the animate loop drives rendering — and `page.evaluate()` doesn't yield to it.

### Why this fix

A synchronous `renderer.render()` is the minimal intervention — it guarantees the pixel buffer matches the current camera state without any async coordination. Alternatives like waiting for the next rAF cycle would require `page.waitForFunction()` polling and a sentinel flag, adding complexity and indirection.
