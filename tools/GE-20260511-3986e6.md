### Symptom

A multi-step verification script using `&&`-chained grep commands appears to produce no output for later checks — those checks seem to pass silently. The script exits 0 overall but intermediate results were never printed.

### Context

Verification scripts that chain multiple `grep` checks with `&&` and `echo` labels, e.g.:

```bash
echo "=== Old headers (must be empty) ===" && git log --oneline | grep "pattern1" \
  && echo "=== Replacements (must show 3 lines) ===" && git log --oneline | grep "pattern2"
```

When `grep "pattern1"` finds nothing (correct behaviour — "must be empty"), it returns exit code 1. The `&&` chain stops there. The "=== Replacements ===" section never runs and never prints, making it appear the check was skipped or passed silently.

### Root cause

`grep` returns exit code 1 when no lines match — this is correct POSIX behaviour, not an error. But `&&` interprets any non-zero exit code as failure and halts the chain. A grep that correctly finds nothing (desired outcome for "must be empty" checks) prevents subsequent checks from running.

### Fix

Run verification greps as independent commands, or append `|| true` to allow the chain to continue regardless of match:

```bash
# Option 1: independent commands (clearest)
echo "=== Old headers (must be empty) ==="
git log --oneline | grep "pattern1" || echo "(none — correct)"

echo "=== Replacements (must show 3 lines) ==="
git log --oneline | grep "pattern2"

# Option 2: || true to keep && chain alive
echo "=== Check ===" && (git log --oneline | grep "pattern1" || true) && echo "=== Next ==="
```

### Why this is non-obvious

The failure is invisible: the script completes, the later checks produce no output (which looks like "nothing to report"), and there's no error. Only noticed when re-running the later check in isolation and seeing it would have produced output. The "must be empty" check succeeding silently stops the "must show N lines" check from running at all.
