### Symptom
`git push origin main` with the OLD remote URL (pre-transfer) completes successfully, printing:

```
remote: This repository moved. Please use the new location:
remote:   https://github.com/<new-org>/<repo>.git
To https://github.com/<old-owner>/<repo>.git
   abc1234..def5678  main -> main
```

The push exit code is 0. Code lands in the **new** org/repo.

### Context
After a GitHub repo transfer (e.g. `mdproctor/claudony` → `casehubio/claudony`), the old
URL is kept alive as a redirect. Git push honours the redirect transparently — the "moved"
message is informational, not a failure.

### Root cause
GitHub maintains HTTP 301/307 redirects for transferred repos indefinitely. Git's HTTP
transport follows them on push, so the push delivers to the new location without requiring
a `git remote set-url` update.

### Fix / Implication
No immediate action required post-transfer. The team can continue pushing to the old origin
URL until a convenient time to run `git remote set-url origin <new-url>`. The push output
is the only signal that something changed — easy to miss if output is not read carefully.

Check the delivered destination via `git fetch upstream && git log upstream/main | head -1`
to confirm the commit landed where expected.

### Why this is non-obvious
Developers expect a push to a moved repo to fail with a 404 or a redirect error. The silent
success with an informational-looking banner is easy to miss, and it is not documented in
GitHub's repo transfer docs or git's HTTP transport docs.
