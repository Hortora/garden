## Symptom

`gh project item-add <num> --owner <org> --url "<issue-url>"` returns exit code 0 with no stdout output — identical behaviour whether the item was added successfully or silently failed to add.

## Context

Populating a GitHub Projects V2 board with issues via CLI. Running `gh project item-add` in a loop over issue URLs produces no output and zero failures, but the project remains empty.

## What was tried (didn't work)

- Checking `$?` exit code — always 0
- Looking for error output on stderr — none
- Assuming empty output = success and continuing — project stays empty

## Root cause

`gh project item-add` produces no output on success by design (`--format json` is not the default and may not surface all states). When the underlying API call fails or silently no-ops (e.g. due to a shell variable expansion issue, cross-org permission boundary, or URL format mismatch), the CLI swallows the failure — same exit 0, same empty output.

## Fix

Verify with a follow-up query after any bulk operation:

```bash
gh project item-list <num> --owner <org> --format json | jq '.items | length'
```

Or bypass `gh project item-add` entirely and use the GraphQL `addProjectV2ItemById` mutation (see GE-20260505-6e554c), which returns the created item ID — an explicit success signal.

## Why this is non-obvious

Exit code 0 with no output is the universal POSIX convention for "success". There is no documented way to distinguish "added successfully" from "silently did nothing" using `gh project item-add` alone.
