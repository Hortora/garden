## Symptom

A fine-grained PAT (URL like `https://github.com/settings/tokens/4298051396`) with `delete:packages` and `read:packages` apparently checked can successfully list org packages but every deletion attempt fails silently or with a 403/404. The `gh api -X DELETE` command returns non-zero regardless of the token.

## Root cause

Fine-grained PATs have a different permission model from classic PATs. For org-level package deletion:
- **Classic PAT** with `delete:packages`: works org-wide if the token owner is an org member/admin
- **Fine-grained PAT**: package permissions are scoped to a **resource owner**. If the token's resource owner is your personal account (`mdproctor`), not the org (`casehubio`), the `delete:packages` permission only applies to your personal packages — org packages are out of scope

The `delete:packages` checkbox appearing in the fine-grained PAT UI for personal tokens does NOT grant org-level package deletion. The org-level fine-grained PAT permissions for packages appear to not exist in the UI at all (no "Packages" entry in org permissions dropdown).

## Fix

Use a **classic PAT** with `delete:packages` scope:
1. Go to `https://github.com/settings/tokens` → "Generate new token (classic)"
2. Check `read:packages` and `delete:packages`
3. Store as the `GH_PAT` secret in your repository

## Why this is non-obvious

The fine-grained PAT UI shows `delete:packages` as a selectable scope, implying it should work. The failure mode is silent (API returns errors that look like permission denied or not found rather than "wrong token type"). There is no UI warning that org-level packages require classic tokens.

### Why this fix

Fine-grained PATs are the recommended modern approach for GitHub auth, so developers naturally reach for them. The documentation does not clearly state that org-level Maven package deletion requires classic tokens.
