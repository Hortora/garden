### What it does
Replaces a mid-history merge commit with a linear sequence without touching any file content, producing a history that is tree-identical to the original.

### When to use it
A merge commit exists in a pushed branch that blocks `git rebase -i` squash operations (git rejects `pick` on merge commits). The merge was a fast-forward merge — the feature branch tip (`parent 2`) is a direct ancestor of `parent 1`, so `tree(merge-commit) == tree(feature-branch-tip)`.

### How to apply
```bash
# Identify the merge commit's parents
git show --format="%P" <merge-sha>    # → <parent1> <parent2>

# Confirm tree equality — must return empty
git diff <parent2> <merge-sha>

# Create working branch at the feature tip (parent 2)
git checkout -b linear-work <parent2>

# Cherry-pick all commits after the merge
git cherry-pick <merge-sha>..<original-branch-tip>

# Verify tree is identical to original
git diff <original-branch-tip> linear-work    # must be empty
```

Since `tree(feature-tip) == tree(merge-commit)`, every cherry-picked commit applies to the exact same starting tree it was originally built on. No conflicts are possible.

### Why this is non-obvious
The instinct is to use `git rebase --rebase-merges` (complex, can introduce conflicts) or `git filter-repo` (rewrites entire repo history). The cherry-pick approach exploits the tree-identity invariant of fast-forward merges — a property that is easy to verify but not obvious to reach for.
