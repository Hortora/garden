### Symptom
`git rebase -i --rebase-merges` with squash operations fails mid-run with a merge conflict on a file (e.g. CLAUDE.md) that the squash plan doesn't directly target for the failing merge commit.

### Context
Full-history squash on a repo with real merge commits (e.g. epic branches landed via `git merge`). Squash operations absorb docs/chore commits into nearby feature commits. The conflict occurs when the `merge -C <sha>` replay runs.

### Root cause
The original merge commit merged two branches where both had the shared file in specific states. After earlier squash operations modified that file (absorbed docs commits into feature commits), the merge commit's expected base state no longer matches. The rebase machinery detects a conflict during the merge replay even though no squash operation targeted the merge commit directly.

### Fix
Only squash commits in the fully linear section of the history — the commits that sit above all merge commits in topological order. Leave the merge commit branches completely untouched. Identify the boundary:
```bash
# Find all true merge commits (two parents) in the range
git log --merges --format="%H %s" <base>..HEAD

# Find the most recent merge commit, squash only above it
git rebase -i <most-recent-merge-sha>   # no --rebase-merges needed for linear section
```

### Why this is non-obvious
The conflict looks unrelated to the squash plan — the file being conflicted was modified by squashes targeting different commits. The failure is only visible during the merge replay, not during the earlier squash operations that caused it. Developers expect squash conflicts to occur at the squash site, not at a downstream merge.
