## Symptom

After a `git rebase -i` squash that reorders commits, intermediate commits in
the compacted history show non-zero diffs against their counterparts in the
original branch. This looks alarming — it suggests content was lost or
corrupted — but the final HEAD is identical to the original.

## Context

When squashing, a commit was moved from position 5 to position 10 in the
rebase todo (to place it with a semantically related group). All commits
in positions 5–9 now lack that commit's change in their tree. Comparing
any of those squashed commits against their originals shows divergence.

```bash
git diff <squashed-group-5> <original-group-5>   # → 13 lines diff ← alarming!
git diff HEAD <original-HEAD>                    # → 0 lines diff  ← correct
```

## Root cause

`git rebase -i` replays each commit as a diff from its parent. When a commit
is moved later in the todo, all commits between the original and new positions
are now built on a tree that lacks that commit's change. Their trees diverge
from the originals — but the total effect at HEAD is identical because the
moved commit is eventually applied.

## Fix

Always verify the final tree, not intermediate commits:

```bash
# The authoritative check after any squash:
git diff HEAD <original-HEAD> -- ':!.claude/'   # must be 0 lines
```

Do not stop at an alarming intermediate diff. Trace through to HEAD before
concluding content was lost.

## Why this is non-obvious

When verifying a squash, the natural instinct is to compare squashed commits
against their original counterparts one by one. Non-zero intermediate diffs
look like content loss or corruption. But the rebase is not a content operation
at each step — it's a sequence of patches, and reordering changes which tree
each patch is applied to. The only meaningful verification point is the final
commit.
