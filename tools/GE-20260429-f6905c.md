## GitHub Projects v2 `updateProjectV2Field` silently returns null when updating single-select options on an existing field

**ID:** GE-20260429-f6905c
**Stack:** GitHub Projects v2 GraphQL API

**Symptom:** The mutation completes without error, returns `null` for the updated field, and the options on the existing field are unchanged.

**Context:** Attempting to add or change single-select options on an existing Projects v2 field using `updateProjectV2Field`:
```graphql
mutation {
  updateProjectV2Field(input: {
    projectId: "PVT_kwDO..."
    fieldId: "PVTSSF_lADO..."
    singleSelectOptions: [
      {name: "New Option", color: BLUE, description: "..."}
    ]
  }) {
    projectV2Field { ... on ProjectV2SingleSelectField { options { id name } } }
  }
}
```
Returns: `{"data": {"updateProjectV2Field": {"projectV2Field": null}}}`

**Root cause:** `updateProjectV2Field` does not support mutating the options of an existing single-select field — the `singleSelectOptions` argument is silently ignored for existing fields. The API returns null without any error or warning.

**Fix:** Use `createProjectV2Field` to create a new field with the desired options instead:
```graphql
mutation {
  createProjectV2Field(input: {
    projectId: "PVT_kwDO..."
    dataType: SINGLE_SELECT
    name: "My Field"
    singleSelectOptions: [
      {name: "Option A", color: BLUE, description: "..."},
      {name: "Option B", color: GREEN, description: "..."}
    ]
  }) {
    projectV2Field { ... on ProjectV2SingleSelectField { id options { id name } } }
  }
}
```

**Why non-obvious:** The mutation succeeds (no error), the return type includes the field, but `projectV2Field` is null — indistinguishable from a permissions error or a transient failure. There is no documentation indicating that single-select options on existing fields cannot be updated via this mutation. The natural assumption is that `updateProjectV2Field` with `singleSelectOptions` updates the options.
