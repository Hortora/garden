## GitHub Pages 'Deploy from branch' ignores your Gemfile and builds with Jekyll 3.9

**ID:** GE-20260412-48e9e2
**Stack:** GitHub Pages, Jekyll 4.x
**Symptom:** Jekyll build step in the Actions log completes successfully. Deploy either
fails silently with wrong output, or succeeds but the published site uses Jekyll 3.9.x
features/rendering rather than your pinned version. No warning about the version mismatch.
**Context:** Repo has a `Gemfile` pinning `gem "jekyll", "~> 4.x"`. GitHub Pages
configured to "Source: Deploy from a branch → main → /docs". Any Jekyll 4-only feature
(e.g. `{% link %}` changes, new Liquid filters, updated kramdown behaviour) silently
falls back to 3.9 behaviour.

### What was tried (didn't work)
- Pinned `gem "jekyll", "~> 4.3"` in `Gemfile` — build passed but deploy used Jekyll 3.9
- Assumed the build step's `bundle install` output meant our `Gemfile` was being used

### Root cause

GitHub Pages "Deploy from branch" uses GitHub's own managed build servers running
the `github-pages` gem, which pins Jekyll to 3.9.x. The `bundle install` visible in
the Actions log is GitHub's own pipeline step — it ignores your `Gemfile` entirely and
installs its own fixed dependency set. Your pinned version is never consulted.

### Fix

Switch GitHub Pages source from "Deploy from branch" to **GitHub Actions**, then add
a workflow that uses `ruby/setup-ruby` so your `Gemfile` controls the build:

```yaml
# .github/workflows/jekyll.yml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: ruby/setup-ruby@v1
        with:
          ruby-version: '3.3'
          bundler-cache: true
          working-directory: docs       # wherever your Gemfile lives
      - uses: actions/configure-pages@v5
        id: pages
      - run: bundle exec jekyll build --baseurl "${{ steps.pages.outputs.base_path }}"
        working-directory: docs
        env:
          JEKYLL_ENV: production
      - uses: actions/upload-pages-artifact@v3
        with:
          path: docs/_site
```

Enable via repo Settings → Pages → Source → **GitHub Actions**.

### Why this is non-obvious

The build step succeeds and produces no error. The `bundle install` output in the
Actions log looks like your `Gemfile` is being used — it isn't. The version mismatch
only surfaces as subtle rendering differences or missing features, not as a build
failure. Developers naturally assume the `Gemfile` controls the build when they see
bundler in the logs.
