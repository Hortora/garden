### Symptom

Running `gh api "/orgs/{org}/packages/maven/{package}/versions" --jq '.[0].updated_at'` returns a date months in the past (e.g. `2026-05-01T11:06:09Z`). This suggests the SNAPSHOT artifact hasn't been published recently, leading to a diagnosis of "stale publishing" or "CI not deploying."

Meanwhile, compilation fails with `cannot find symbol: class LedgerEntry` — interpreted as confirming the dependency is stale or missing.

### Context

Occurs when using the GitHub Packages REST API to check SNAPSHOT freshness, combined with a compilation error from a dependency whose classes moved packages.

### What was tried (didn't work)

1. Checked `updated_at` via `gh api` — concluded artifact was stale (months old)
2. Built a narrative around stale publishing — investigated CI config, considered excluding the module
3. Treated "cannot find symbol" as confirming the missing-dependency theory

### Root cause

Two independent issues conflated into one wrong diagnosis:

1. **`updated_at` is misleading.** GitHub Packages stores SNAPSHOT uploads as timestamped files under a single version entry (e.g. `0.2-SNAPSHOT`). The `updated_at` on the version object reflects when that version entry was *created*, not when the latest timestamped JAR was uploaded. New SNAPSHOTs (e.g. `casehub-ledger-0.2-20260706.110749-242.jar`) are added daily — the version metadata just doesn't update.

2. **"Cannot find symbol" ≠ "cannot find artifact."** The compilation error said `cannot find symbol: class LedgerEntry` — not `could not resolve artifact`. Maven resolved the JAR successfully. The class simply wasn't at the expected package path because it had been migrated (`io.casehub.ledger.runtime.model` → `io.casehub.ledger.api.model`). That's a package migration, not a missing dependency.

### Fix

**To verify SNAPSHOT freshness:** Check the actual artifact files under the version entry, or check the source repo's CI logs for recent publish runs. Don't trust version-level `updated_at`.

**To diagnose "cannot find symbol":** Read the error carefully. "Cannot find symbol" means Maven found the JAR but the class isn't where expected — likely a package migration or rename. "Could not resolve artifact" means the dependency itself is missing. These are fundamentally different problems with different fixes.

### Alternatives considered

- Checking the Maven metadata XML in `~/.m2/repository/` for lastUpdated timestamps — works but requires local cache to be current
- Using `gh api` to list individual artifact files under the version — correct but verbose

### Why this is non-obvious

The `updated_at` field name strongly implies it tracks the most recent modification. Combined with a compilation error that superficially looks like a missing dependency, the two signals reinforce each other into a coherent but wrong narrative. The actual cause (package migration) requires noticing the specific wording of the compiler error ("cannot find symbol" vs "could not resolve artifact") — a distinction that's easy to overlook under time pressure.
