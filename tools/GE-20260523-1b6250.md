## Context

Multi-repo platforms often track cross-repo dependencies in two places:

1. **Build order comment** — ensures correct publish ordering (e.g. `casehub-engine (depends on ... optionally casehub-eidos-api)`)
2. **Cross-repo dependency table** — the impact analysis tool, used to find every consuming repo when an artifact changes (rename, removal, SPI break)

These serve different consumers and are maintained independently. An optional dependency expressed only in the build order comment is invisible to the dependency table.

## The technique

When adding an optional cross-repo dependency, update **both**:

- The build order comment (with `optionally`)
- A row in the cross-repo dependency table, with `Nature: optional <purpose>`

Without the table row, propagation scans (`grep`, impact analysis tooling) won't find the consuming repo when the artifact changes.

## Why this is non-obvious

"Optionally" in the build order comment reads as complete documentation of an optional dependency. The two tracking mechanisms look like they cover the same ground — in fact they serve different consumers: CI publish ordering vs propagation impact analysis. A missing row is invisible until a breaking change to the optional artifact is missed by the consuming repo.

### Alternatives considered

- Document in build order only — leaves the consuming repo invisible to impact analysis; the dep is functionally undocumented for propagation purposes
- Add a prose note in the dependency description instead of a table row — not picked up by tooling or grep scans on the artifact name
