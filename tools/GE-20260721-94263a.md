### Symptom
Branch audit found 47 branches across 12 repos where `chore: branch closed — landed as <SHA>
on main` stamps referenced SHAs not on main. `git cat-file -e <SHA>` passed (object exists)
but `git branch --contains <SHA>` returned nothing — the SHA was unreachable from any branch.

### Context
Work-end workflow: branch content is rebased onto main, squashed into fewer commits, pushed,
then the branch is stamped with an empty commit recording the landing SHA. The stamp is
written pointing to the pre-squash SHA (the tip of main before squash rewrites history).
After squash, the pre-squash SHA becomes a dangling object reachable only from reflog.

Related but distinct from GE-20260720-95f762 (commits added *after* the stamp). This entry
covers stamps that point to SHAs that were valid *at stamp time* but become orphaned when
the squash creates new commits with different SHAs.

### What was tried (didn't work)
`git branch --contains <SHA>` — returns nothing because the SHA is dangling (not on any
branch after squash). `git cat-file -e <SHA>` — passes because the object still exists in
the object store, giving a false sense that the SHA is valid. The stamp appears correct
until you check whether the SHA is actually reachable from the base branch.

### Root cause
Squash-merge creates new commit objects with different SHAs. The pre-squash commits become
unreachable from any branch — they survive only in reflog and will be garbage-collected.
The stamp, written before or during the squash, records the pre-squash SHA which is now
a ghost reference.

### Fix
Write the stamp AFTER the squash completes, using the post-squash SHA:

```bash
# After squash and push:
LANDED_SHA=$(git rev-parse HEAD)  # HEAD is now the post-squash tip of main
git checkout "$BRANCH_NAME"
git commit --allow-empty -m "chore: branch closed — landed as $LANDED_SHA on main"
```

Additionally, add a content verification gate before stamping — diff source files between
the branch and the base branch to confirm content actually landed:

```bash
python3 verify_stamp.py "$PROJECT" "$BRANCH" "$BASE_BRANCH"
# Returns VERIFIED=yes or VERIFIED=no with list of missing files
```

### Why this is non-obvious
The stamp is written at a moment when the SHA is valid — it's the tip of main. The
invalidity happens later, silently, when the squash rewrites the commit. There's no error,
no warning, and no audit trail. The stamp looks correct until someone runs `branch --contains`
months later. The gap between "SHA exists" (`cat-file -e`) and "SHA is reachable"
(`branch --contains`) is not intuitive.
