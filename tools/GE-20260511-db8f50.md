### Symptom

A git-squash session produces a compacted `main` that is hundreds of commits behind the canonical remote. The squash completes without error. Only discovered when comparing the squashed branch to upstream and finding a large content divergence.

### Context

Repos with a fork+upstream remote layout (`origin` = personal fork, `upstream` = org canonical). Work is committed locally and pushed directly to `upstream` (bypassing the fork). When a new squash session starts — possibly days or weeks later — it resets/bases off `origin/main`, which was never synced with the upstream pushes. The fork is stale by the entire period of direct-upstream work.

### What was tried

Two separate squash sessions on the same repo. Session 1 operated from the correct `upstream`-synced state and produced a valid backup. Session 2 silently based itself on `origin/main` (stale fork), produced a v1 backup 279–346 commits behind upstream, and squashed that stale state.

### Root cause

The squash session fetched from `origin` instead of `upstream`. Because work had been pushed directly to upstream (not via the fork), `origin/main` had not received those commits. The session had no way to detect this — both remotes exist, both have a `main` branch, and `origin/main` is a perfectly valid (if stale) base.

### Fix

Before any squash session on a fork+upstream repo, verify which remote is authoritative and explicitly check for staleness:

```bash
git fetch upstream
git fetch origin
git log --oneline upstream/main ^origin/main | wc -l  # commits upstream has that fork doesn't
git log --oneline origin/main ^upstream/main | wc -l  # commits fork has that upstream doesn't
```

If upstream is ahead of origin, sync the fork before squashing:

```bash
git push origin upstream/main:refs/heads/main  # bring fork up to date
```

Always explicitly confirm the squash base comes from `upstream/main`, not `origin/main`.

### Why this is non-obvious

In a typical fork workflow, `origin` and `upstream` stay in sync via PRs. When work bypasses the fork and goes directly to upstream, the fork silently drifts. The squash session sees a valid, named remote branch and has no signal that it is stale — the only detectable symptom is a content diff discovered after the fact.
