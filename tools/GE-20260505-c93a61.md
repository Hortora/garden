## Symptom

An IntelliJ MCP tool call (e.g. `get_file_text_by_path`, `search_in_files_by_text`) that worked earlier in the same session returns HTTP 404 with "Session not found". No warning is given before the failure.

## Context

During long working sessions with Claude Code using IntelliJ MCP tools. The session connection can time out or be dropped by IntelliJ, invalidating the session token. Subsequent MCP calls fail with 404 until the connection is re-established.

## What Was Tried

Retrying the same tool call immediately — still 404. The session is gone, not transiently unavailable.

## Root Cause

IntelliJ MCP uses session-based authentication. Sessions have a timeout and can be dropped when IntelliJ is idle or the MCP server restarts. The session ID embedded in tool calls becomes invalid, and the server returns 404 rather than an auth error or a reconnect signal.

## Fix

Re-verify MCP availability using the index MCP (which has a separate connection):

```
mcp__intellij-index__ide_index_status  ← use this to check connectivity
mcp__intellij__get_file_text_by_path   ← re-attempt with any lightweight call
```

If 404 persists, the user must restart the IntelliJ MCP connection from the IDE side. Always check both MCPs independently — one can be up while the other is dropped.

## Why This Is Non-Obvious

HTTP 404 reads as "resource not found" (path error), not "session expired". Developers look for typos in the file path rather than a connection problem. The failure is silent — no prior warning that the session is about to expire.
