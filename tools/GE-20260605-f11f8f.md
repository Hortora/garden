## git rm silently succeeds on empty directories — rm -rf required

**ID:** GE-20260605-f11f8f
**Stack:** Git

**Symptom:** After running `git rm -r <dir>/` on a set of directories, `git status --short` shows nothing changed. The directories appear to still exist. No error is reported.

**Context:** Directories that previously contained tracked files had all their contents deleted (leaving them empty). Attempting to remove the directories via `git rm` produces exit code 0 with no output — but nothing is staged for deletion.

**Root cause:** Git does not track directories — only files. An empty directory has no git representation. `git rm` operates on tracked paths; since the empty directory has no tracked content, there is nothing for git to remove. The command succeeds vacuously.

**Fix:**

```bash
# This does nothing on an empty directory (no error, no change):
git rm -r some-skill/commands/

# Correct approach — remove from filesystem directly:
rm -rf some-skill/commands/

# git status will show nothing (untracked empty dirs are ignored by git)
# The directory is gone from disk; git's state is already clean
```

**Why non-obvious:** `git rm` is the instinctive tool for removing paths from a git repo. It reports success and produces no error. The natural follow-up (`git status`) also shows nothing — which looks like the operation worked, not like it silently no-oped. The distinction between "tracked file deleted" and "untracked empty directory removed" is invisible until you check the filesystem directly.
