## Symptom

A module gets a `BUILD` decision on every incremental CI run even when its source hasn't changed. Builds complete correctly and pass — just unnecessarily. The per-module duration in the summary shows it rebuilding every time.

## Context

Incremental build workflows that store per-module SHAs in a state file between runs (e.g. using `actions/cache`). The pattern has three steps: Clone repos → Collect SHAs → Build → Save build state. When a new module is added, all three locations must be updated. If the Save build state step is missed, the module's SHA is never written to the state file.

```bash
# Save build state step — platform and eidos omitted:
{
  echo "parent=${SHA_PARENT}"
  record ledger     "$SHA_LEDGER"     "${PREV_LEDGER:-none}"     "$OUTCOME_LEDGER"
  record connectors "$SHA_CONNECTORS" "${PREV_CONNECTORS:-none}" "$OUTCOME_CONNECTORS"
  # platform and eidos missing — never saved
} > .incremental-state/shas.txt
```

## Root cause

The Load previous SHAs step reads the state file and emits `sha_platform=none` for any module absent from the file (the fallback default). The decision script then compares current SHA against `none` and always returns `BUILD` — current SHA will never equal `none`. Every run is treated as first-time for that module.

## Fix

When adding a module to an incremental build workflow, update three locations atomically:

1. **Clone step** — add the `git clone` line
2. **Collect SHAs step** — capture the module's current SHA into `$GITHUB_OUTPUT`
3. **Save build state step** — add `SHA_MODULE` env var and a `record module ...` call

Missing any one of the three causes silent rebuild inflation. A checklist comment in the workflow helps:

```bash
# When adding a module: update (1) Clone, (2) Collect SHAs, (3) Save build state
```

## Why this is non-obvious

The build produces correct artefacts and the workflow passes. There is no error, no warning, no failed step — just wasted CI time. The only way to detect it is to inspect `.incremental-state/shas.txt` after a run and notice the module's name is absent, then correlate that with the decision script's "none" comparison. Without actively checking the state file contents, silent rebuild inflation can persist indefinitely.
