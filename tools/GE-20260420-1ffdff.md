## Symptom

`git rebase upstream/main` starts replaying 40+ commits, hits conflicts on commit 1/48, and
each resolution creates new conflicts in later commits. The rebase never completes cleanly.

## Context

Working on a long-running fork branch. Multiple upstream PRs have merged touching many of the
same files. The branch has accumulated commits that predate those upstream merges.

## Root Cause

Git rebase replays every commit individually against the new base. If the upstream merged
changes to the same files at multiple points, each replay commit may conflict — even if the
final net diff would apply cleanly. The number of conflict resolutions scales with the number
of intermediate commits, not just the final diff.

## Fix

Abandon the rebase. Create a fresh branch from upstream and apply only the net diff:

```bash
# 1. Abort if mid-rebase
git rebase --abort

# 2. Create clean branch from upstream
git checkout upstream/main -b feat/clean-branch

# 3. Extract net diff of all changes you actually want
git diff upstream/main feat/old-branch -- <file-or-dir-list> > /tmp/net-changes.patch

# 4. Apply cleanly
git apply /tmp/net-changes.patch

# 5. Single commit, clean history
git add -A && git commit -m "feat: [description]"
```

This produces a single clean commit with the same net effect as the entire branch, applied
on top of the latest upstream — with zero conflict resolution.

## Why This Is Non-Obvious

`git rebase` is the standard answer for "update my branch to upstream". The 48-commit cascade
is not predictable until you're already in it. The fresh-branch + diff approach feels like
"giving up" on history, but for PRs that need to land on the current upstream it produces
a cleaner result faster.
