### The convention

When a project depends on multiple foundation repos, maintain a `ROADMAP.md` in the workspace root that:
1. Phases work so each phase delivers demo-able capability (backend + UI together)
2. Maps foundation issues to application features they unblock, per phase
3. Ends with a **Foundation Priority Summary** — one table per repo telling that repo's sessions what to prioritise

### The alternative

Track cross-repo dependencies in GitHub Projects alone, or inline in CLAUDE.md foundation gates, or in HANDOFF.md cross-module sections. Each captures part of the picture but none gives a single phased view with per-repo priority signals.

### Why this convention

HANDOFF.md captures session-to-session state. CLAUDE.md captures permanent conventions. Neither is designed for multi-phase strategic planning across repos. A GitHub Project board shows live status but not critical paths or dependency rationale. ROADMAP.md fills the gap: a human-readable strategic document that foundation repo sessions can check at session start to know what matters.

### Structure

- Phase sections with demo stories (what you can show after this phase)
- Per-phase tables: devtown issues (with scale/complexity/type) + foundation prerequisites (with "blocks" column)
- Critical path diagrams (ASCII) per phase
- Cross-cutting UI pipeline section (shared dependency chain)
- Foundation Priority Summary: one table per repo with P0/P1/P2/P3/Deferred priorities

### Why this is non-obvious

The natural instinct is to use GitHub Projects or a Gantt tool for cross-repo planning. But those show status, not strategy — they don't capture why a foundation issue matters to the application, or what demo capability each phase unlocks. The key insight is combining per-repo priority signals (so foundation sessions know what to work on) with phased demo stories (so the application team knows what each phase delivers).

### When to use

Multi-repo platforms where one application repo depends on 3+ foundation repos for feature delivery. Not useful for single-repo projects or projects without phased delivery goals.
