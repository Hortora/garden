### Symptom
`ide_edit_member` with `member=RecordName` (e.g. `member=AgentWorkerFunction`) returns
`"member_not_found"` for a top-level Java record. The tool description says
`member=className` works for class declarations, but records are not found this way.

### Context
Occurs when trying to change a record's `implements` clause, type parameters, or add
methods to the record body via `ide_edit_member`. Regular classes work with
`member=className` alone. Records require the `class` parameter as well.

### What was tried (didn't work)
1. `ide_edit_member(file, member=AgentWorkerFunction, content=...)` — returned "member_not_found"
2. `ide_edit_member(file, class=AgentWorkerFunction, member=AgentWorkerFunction, content=...)` — **corrupted the file** by nesting a new record declaration inside the existing one, requiring `git checkout` to recover

### Root cause
The MCP plugin's member lookup doesn't treat record class names the same as regular class
names. After reconnecting the MCP server (`/mcp` command), the `class=` + `member=`
combination started working correctly — the initial failure was likely an indexing or
caching issue in the plugin.

### Fix
1. **Always pass both `class=RecordName` AND `member=RecordName`** when editing record declarations
2. **Reconnect the MCP server first** (`/mcp`) if the tool returns unexpected results
3. **Verify the file after editing** — run `git diff` to confirm no nesting corruption occurred
4. If corruption does occur, recover with `git checkout -- <file>`

### Why this is non-obvious
The tool documentation says `member=className` works for "class declarations" — a developer
would reasonably expect this to cover records (which are a kind of class in Java). The
nesting corruption is especially dangerous because it produces valid-looking Java that
compiles as a nested record, making the corruption hard to spot without a diff.
