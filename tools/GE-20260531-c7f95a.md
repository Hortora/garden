### Symptom
Commits made inside a `for` loop land on the wrong branch. No error is shown.

### Context
Automating `git checkout <branch>` for multiple branches in a shell loop,
suppressing output with `2>/dev/null`. One branch is checked out in a linked
worktree and cannot be checked out in the main worktree. The checkout fails —
but silently. The loop continues; subsequent `git commit` calls land on
whatever branch was current before the failed checkout.

### What was tried
Stamping a series of branches with an empty commit in a `for` loop using
`git checkout "$b" 2>/dev/null && git commit ...` — the branch locked in
a worktree caused the checkout to fail silently, and the next commit landed
on the previously-checked-out branch.

### Root cause
`2>/dev/null` suppresses stderr from the failed checkout. The shell `&&`
chains the checkout with the commit, but the for loop variable advances
regardless. If the checkout was not chained with `&&`, the commit runs
unconditionally on the wrong branch.

### Fix
Three options:

1. **Drop `2>/dev/null`** — let the checkout error surface and break the loop.

2. **Verify after checkout:**
   ```bash
   git checkout "$b"
   ACTUAL=$(git branch --show-current)
   [ "$ACTUAL" = "$b" ] || { echo "⚠️ Still on $ACTUAL — skipping"; continue; }
   git commit --allow-empty -m "chore: branch closed"
   ```

3. **Handle locked worktrees separately** — detect via `git worktree list` and
   stamp using `git commit-tree` + `git update-ref` instead (see technique entry
   for stamping branches locked in worktrees).

### Why this is non-obvious
`2>/dev/null` is a common pattern for "I don't care about the error message" —
but it also hides the signal that the checkout failed. Most developers expect
the loop to stop or skip gracefully; instead it continues on the stale branch.
The resulting commits are hard to trace without checking the branch log carefully.
