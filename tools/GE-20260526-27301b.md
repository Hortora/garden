## Context
When integrating a system with OpenClaw's WhatsApp delivery and comparing it
against a direct Meta Cloud API implementation, an apparent overlap exists.
They appear to solve the same problem — sending WhatsApp messages.

## What's actually happening
OpenClaw's WhatsApp channel uses **Baileys** — a library that reverse-engineers
the WhatsApp Web WebSocket protocol. This is personal-account tier, not the
official WhatsApp Business API.

| | OpenClaw (Baileys) | Direct (Meta Cloud API) |
|---|---|---|
| Account type | Personal WhatsApp | WhatsApp Business |
| Auth | WhatsApp account credentials | API token + Phone Number ID |
| Direction | Bidirectional (receive + send) | Outbound only |
| Status | Unofficial / reverse-engineered | Official / stable |
| Cost | Free | Per-message after threshold |

## Why this matters architecturally
The two implementations are not interchangeable and serve different use cases.
Baileys reaches personal numbers; Meta Cloud API requires a Business account.
There is no credential deduplication opportunity between a system using the
official API and OpenClaw — they operate at different tiers entirely.

## Why this is non-obvious
OpenClaw's top-level documentation describes "WhatsApp support" without
distinguishing API tier. Only reading their channel adapter source or searching
their dependency list reveals Baileys as the underlying library.
