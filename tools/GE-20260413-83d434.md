## ImageMagick `convert` silently shadowed by macOS built-in — use `magick` instead

**ID:** GE-20260413-83d434
**Stack:** ImageMagick 7.x (Homebrew), macOS (all versions)
**Symptom:** `convert input.png -crop ...` either errors cryptically or does the wrong thing despite ImageMagick being installed via Homebrew. `which convert` points to `/usr/bin/convert` — the macOS built-in — not Homebrew's ImageMagick.
**Context:** Any macOS shell session where ImageMagick is installed via Homebrew but `/usr/bin` appears before `/opt/homebrew/bin` in `$PATH`, or where the shell is invoked non-interactively (e.g. from Claude Code's Bash tool, scripts).

### What was tried (didn't work)
- `brew install imagemagick` — succeeds, but `convert` still points to the wrong binary
- `convert input.png -crop 502x470+0+0 +repage output.png` — "command not found" in some shells, wrong behaviour in others

### Root cause
macOS ships `/usr/bin/convert` (an unrelated tool). Homebrew installs ImageMagick 7's main binary as `magick` at `/opt/homebrew/bin/magick`, with a `convert` shim also at `/opt/homebrew/bin/convert`. In non-interactive shells or when `/usr/bin` precedes `/opt/homebrew/bin` in `$PATH`, the system `convert` wins.

### Fix
Use `magick` directly — it's the canonical ImageMagick 7 binary and is never shadowed:
```bash
magick input.png -crop 502x470+0+0 +repage output.png
magick mogrify -resize 50% *.png
```
Or explicitly prefix the path: `/opt/homebrew/bin/convert`.

### Why this is non-obvious
`brew install imagemagick` succeeds with no warning about the shadowing. The macOS built-in `convert` accepts some arguments without erroring, so the failure mode varies — sometimes silent wrong output, sometimes a cryptic error. Developers familiar with ImageMagick on Linux expect `convert` to work.
