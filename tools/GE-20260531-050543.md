## Symptom

A `sed -i` substitution with backreference captures produces the literal strings `\1` and `\2` in the output file instead of the captured group content.

```bash
sed -i 's/new Foo(\(.*\), "\(.*\)")/new Foo(\1, "\2", false)/g' file.java
```

Result in file: `new Foo(\1, "\2", false)` — the backslash-number literals appear verbatim.

## Context

Using GNU-style backreference syntax (`\(.*\)` for capture, `\1` for substitution) on macOS. The command succeeds with exit code 0 and no error output. The file is modified, but with literal `\1` rather than the captured content. The same command works correctly on Linux (GNU sed).

## Root cause

macOS ships BSD sed, not GNU sed. BSD sed supports basic regex (`\(`, `\)`, `\1`), but its behaviour diverges from GNU sed when the captured groups contain special characters (double-quotes, parentheses, `+`) — particularly when the overall pattern is complex. The substitution of `\1` in the replacement string silently fails for some patterns, emitting the literal.

The `-g` flag (global replace) and multi-group captures are especially prone to this.

## Fix

Use `perl -pi -e` instead of `sed -i` for reliable backreference substitution on macOS:

```bash
perl -pi -e 's/new Foo\((.*?), "(.*?)"\)/new Foo($1, "$2", false)/g' file.java
```

Or use Python for multi-file edits:

```python
import re, pathlib
p = pathlib.Path("file.java")
p.write_text(re.sub(
    r'new Foo\((.*?), "(.*?)"\)',
    r'new Foo(\1, "\2", false)',
    p.read_text()
))
```

For single-occurrence replacements where you know the exact text, use a literal replacement (no capture groups) to avoid the issue entirely.

## Why this is non-obvious

The command succeeds (exit code 0), produces no error output, and clearly modifies the file — so it appears to work. The failure only becomes visible on file inspection. GNU-sed syntax is so common that developers don't check which sed variant is installed. The failure mode (literal `\1`) looks like an escaping issue, leading to many unsuccessful attempts to escape differently.
