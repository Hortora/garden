## Symptom

`git rebase -i` exits immediately with:

```
BUG: sequencer.c:2725: unexpected todo_command
```

No rebase is started. The error gives no indication of which commit caused it.

## Context

Running `git rebase -i` with a custom todo file injected via `GIT_SEQUENCE_EDITOR="cp $PLAN_FILE"`. The range contains a local merge commit. The plan file includes a `drop <merge-commit-sha>` line for that commit.

## Root cause

`git rebase -i` without `--rebase-merges` silently excludes merge commits from the generated todo. The sequencer does not know how to process `drop` for a merge commit in this mode — it is not a recognised action for that commit type. Including it manually in a custom todo triggers an internal bug assertion rather than a user-facing error.

## Fix

**Step 1:** Inspect the default todo before overriding it:

```bash
GIT_SEQUENCE_EDITOR=cat git rebase -i <base-sha> 2>&1 | head -60
```

The default todo never includes merge commits when run without `--rebase-merges` — git flattens them out automatically. Mirror this behaviour in any custom todo: simply omit merge commit lines entirely. The merge commit's content is already captured by the individual commits on either side of it.

**Step 2:** If you need to handle merge commits explicitly, use `--rebase-merges`:

```bash
git rebase -i --rebase-merges <base-sha>
```

This changes the todo format significantly — merge commits get `merge` labels rather than `pick`.

## Why this is non-obvious

The error message (`BUG: sequencer.c`) looks like a git internal crash, not a user error. There is no mention of which line in the todo caused it, or that `drop` is invalid for a merge commit in this mode. The natural assumption is that `drop` should work for any commit.
