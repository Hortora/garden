## Symptom

After using IntelliJ's Refactor → Move Package to rename a Java package (e.g. `dev.claudony` → `io.casehub.claudony`), some `@Inject` field declarations in test files are silently corrupted:

```java
// Before (correct)
@Inject SessionRegistry registry;
@Inject CaseEventBroadcaster broadcaster;

// After refactoring (corrupted)
@Inject
       SessionRegistry      registry;
@InjectCaseEventBroadcaster broadcaster;
```

The second `@Inject` annotation merges with the next field's class name, producing a fake annotation (`@InjectCaseEventBroadcaster`) that doesn't exist. The build fails with `<identifier> expected` at the column where the merged annotation appears.

## Context

Occurs specifically when a `@Inject ClassName field;` declaration is on a single line and the class being moved is the one in that field position. Affects test files more often than production code. The corruption is silent during the refactoring — no warning is shown.

## What was tried

The Move Package dialog completed without errors or conflicts. The corruption only appeared at build time.

## Root cause

IntelliJ's Move Package refactoring appears to incorrectly handle single-line `@Inject ClassName field;` declarations when the class being moved is the type of the field. The import is removed (correct) but the annotation and type declaration on the same line get mangled during the text rewrite.

## Fix

After any Move Package refactoring, search for `@Inject[A-Z]` (no space between `@Inject` and the next class name) across the affected modules:

```bash
grep -r "@Inject[A-Z]" src/
```

For each corrupted line, split manually:
```java
// Replace this:
@InjectCaseEventBroadcaster broadcaster;

// With this:
@Inject CaseEventBroadcaster broadcaster;
```

Also add the missing import if it wasn't carried over.

## Why this is non-obvious

The refactoring reports success and shows a preview with no conflicts. The corruption produces a syntactically plausible-looking token (`@InjectClassName`) that looks like a custom CDI qualifier annotation, not a merge artifact. The real error only surfaces at compile time on lines unrelated to the class that was moved.
