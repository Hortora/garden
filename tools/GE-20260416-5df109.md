## `podman machine rm` permanently destroys the VM — it is not a stop command

**ID:** GE-20260416-5df109
**Stack:** Podman 5.x, macOS
**Symptom:** Running `podman machine rm --force podman-machine-default` to force-stop a stuck machine instead permanently deletes the VM. All containers, images cached inside the VM, and machine configuration are lost. `podman machine list` shows an empty list afterward.
**Context:** macOS Podman machine is in a broken state (SSH handshake fails, `podman ps` shows connection error). Attempting to force-stop it.

### What was tried (didn't work)
- `podman machine stop` — succeeds but machine still reports errors on restart
- `podman machine rm --force ...` — thought `--force` meant "force stop"; actually means "skip confirmation prompt and destroy immediately"

### Root cause
`podman machine rm` is analogous to `docker container rm` — it *removes/destroys* the resource, not stops it. The `--force` flag skips the interactive confirmation prompt, it does not change the operation from remove to stop. There is no `--force` flag on `podman machine stop`.

### Fix
To stop a stuck Podman machine:
```bash
podman machine stop podman-machine-default   # graceful stop
# If stuck, kill the VM process directly from Activity Monitor or:
pkill -f "podman-machine-default"
```

To recreate after accidental removal:
```bash
podman machine init --cpus 5 --memory 3800 --disk-size 31
podman machine start
# New machine auto-creates /var/run/docker.sock symlink on macOS
```

### Why this is non-obvious
`rm --force` in Docker/Podman container context means "force remove a running container without stopping it first". Applied to `machine`, the mental model shifts but the command name stays the same. `podman machine stop --help` shows no `--force` flag, making it tempting to try `rm --force` as a workaround for a stuck machine.
