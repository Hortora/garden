## Symptom

Two retrieval configurations appear to use identical precision methodology ("scored entries only"), yet one reports significantly higher precision. The difference looks like a real quality gap — decisions are made based on it (model choice, parameter tuning, issue prioritisation).

## Context

A retrieval benchmark compares two configurations (e.g. three-leg RRF vs BGE-M3 four-signal). Each returns N entries per query. Precision is computed as `relevant / total_scored` — entries without a score in the baseline are excluded from the denominator.

Configuration A returns entries that are mostly in the scored baseline. Configuration B returns entries that are a mix of scored and unscored. The unscored entries are excluded from precision calculation for both — but B has more of them.

## Root cause

The unscored entries are disproportionately noise (72% scored 0 when eventually scored). By excluding them, the precision denominator shrinks, and the remaining scored entries have a higher relevant-to-total ratio. The noisier method gets a free precision boost because its noise is invisible.

The bias is asymmetric: the method that finds more novel entries (which haven't been scored yet) is penalised if those entries are relevant, and rewarded if they are noise — but you can't tell which until you score them.

## Fix

Score all entries across all configurations before computing precision. When new entries appear in any configuration's results, score them immediately — do not defer scoring as "future work."

In practice: after running a new retrieval configuration, collect all (ge_id, scenario_id) pairs that lack scores, score them, then recompute precision for ALL configurations with the updated baseline.

### Why this fix

The alternative — noting "N entries unscored" as a caveat — creates a false sense of methodological rigour. The caveat says "precision may adjust" but doesn't quantify the direction or magnitude. In the case that triggered this entry, the adjustment was +4pp for the baseline (90% → 86%), completely reversing the apparent winner.

## Why this is non-obvious

The methodology *looks* correct and consistent — both configurations use "scored entries only" with the same scoring rubric. The bias is a second-order effect of uneven score coverage across configurations, not a first-order methodological error. A reviewer checking the methodology would approve it.

### Alternatives considered

- **Report scored-only precision with an uncertainty band** — rejected because the uncertainty is asymmetric and hard to bound without actually scoring the entries
- **Use recall instead of precision** — doesn't help; recall also depends on knowing which entries are relevant
