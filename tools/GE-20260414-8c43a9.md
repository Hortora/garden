## Context

A workflow engine or orchestrator needs to suspend execution at a point where a human or external system must act, then resume with a result — without polling, without messaging infrastructure, and without blocking a thread.

## Technique

Three-component pattern:

**1. Registry** — maps correlation ID to pending future:
```java
@ApplicationScoped
public class PendingRegistry {
    private final ConcurrentMap<UUID, CompletableFuture<String>> pending
        = new ConcurrentHashMap<>();

    public CompletableFuture<String> register(UUID id) {
        CompletableFuture<String> future = new CompletableFuture<>();
        pending.put(id, future);
        return future;
    }

    public void complete(UUID id, String result) {
        CompletableFuture<String> f = pending.remove(id);
        if (f != null) f.complete(result);
    }

    public void fail(UUID id, String reason) {
        CompletableFuture<String> f = pending.remove(id);
        if (f != null) f.completeExceptionally(new ResolutionException(id, reason));
    }
}
```

**2. Initiator** — creates the work item and returns the pending future:
```java
public CompletableFuture<String> requestApproval(String title, ...) {
    WorkItem wi = workItemService.create(...);
    return registry.register(wi.id);   // caller awaits this
}
```

**3. Resolver** — CDI event listener that completes futures when external events arrive:
```java
@ApplicationScoped
public class EventListener {
    @Inject PendingRegistry registry;

    void onEvent(@Observes WorkItemLifecycleEvent event) {
        switch (event.type()) {
            case "io.example.workitem.completed" ->
                registry.complete(event.workItemId(), event.detail());
            case "io.example.workitem.rejected" ->
                registry.fail(event.workItemId(), event.detail());
        }
    }
}
```

The workflow function returns the `CompletableFuture` — Quarkus Flow (or any async framework) suspends execution until it resolves.

## Why This Is Non-Obvious

Most developers reach for a message broker (Kafka, AMQP) or a polling loop when facing this problem. The CDI event + CompletableFuture registry achieves the same result with zero infrastructure and zero thread blocking — the future sits in a map until the human acts, then resolves synchronously via the CDI event delivery path. The missing piece that makes this work is that CDI `@Observes` delivers events synchronously in the same thread, so the registry write and the future completion happen atomically from the caller's perspective.
