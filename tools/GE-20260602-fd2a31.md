## Symptom

Running `git rebase -i <backup-branch>..HEAD` where `<backup-branch>` is a
`backup/pre-squash-*` branch (created by git-squash on a previous run) reports
`"The previous cherry-pick is now empty, possibly due to conflict resolution"`
for every commit and aborts with nothing applied.

## Context

git-squash creates backup branches by renaming `main` → `backup/pre-squash-main-<date>`
before swapping in the squashed working branch. The backup tip therefore contains the
**squashed equivalent** of every commit in the range — git's cherry-pick engine detects
them as already applied when you try to rebase on it.

## What was tried (didn't work)

Using `backup/pre-squash-main-<date>` as the base SHA for an interactive rebase of
the commits between that backup and HEAD. Every commit became empty and the rebase
aborted.

## Root cause

The backup branch IS the squash of all those commits. Its diff is equivalent to the
full range's diff. Rebasing on it means replaying commits whose content is already in
the base — git correctly sees them as no-ops.

## Fix

Use the parent of the oldest commit in the range as the base:

```bash
BASE=$(git rev-parse <oldest-sha-in-range>^)
git rebase -i "$BASE"
```

Do NOT use the backup branch tip as the base — it contains the wrong content for
this purpose even though it was derived from the same commits.

## Why this is non-obvious

The backup branch was created from the commits you want to squash, so it feels like
the right starting point. The confusion is that it represents the *result* of squashing
those commits, not their common ancestor. The symptom ("cherry-pick is now empty") points
at conflict resolution, not at the base-SHA problem, which misdirects debugging.
