### Symptom
`ide_search_text` with query `as any` returns `totalCount: 0` across a TypeScript
codebase where `grep -rn "as any"` finds 250+ matches.

### Context
Searching for TypeScript `as any` casts across a monorepo using the IntelliJ MCP
`ide_search_text` tool. The query is a two-word pattern where both words are
common individually.

### Root cause
`ide_search_text` uses IntelliJ's pre-built word index for exact (non-regex) queries.
The index tokenizes source files into individual words. A query like `as any` is
looked up as a single token — which never exists in the index because `as` and `any`
are indexed separately.

### Fix
Use bash `grep` for multi-word code pattern searches:
```bash
grep -rn "as any" /path/to/project --include="*.ts"
```

Reserve `ide_search_text` for single-word or single-token searches where the word
index is fast and accurate (e.g. class names, function names, identifiers).

For regex patterns, `ide_search_text` with `regex: true` uses IntelliJ's Find in
Files path instead of the word index — but simple grep is faster for literal
multi-word strings.

### Why this is non-obvious
The tool is named "search text" and accepts arbitrary string queries without
warning. Nothing in the API response indicates the query was incompatible with
the word index — it simply returns zero results, indistinguishable from "no
matches exist." A developer trusting the tool output would conclude the codebase
has no `as any` casts.
