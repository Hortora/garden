## Symptom

`gh issue create --label "platform,architecture"` exits with:

```
could not add label: 'platform' not found
```

The issue is NOT created. The error is clear once you see it, but easy to miss if the command is run non-interactively or in a script where you're focused on other output.

## Context

Occurs whenever you pass `--label` values that don't exist in the target repository. Labels are repo-specific — a label that exists in one repo doesn't automatically exist in another, even within the same org.

## What was tried

Using `--label "platform,architecture"` directly in `gh issue create`, expecting GitHub CLI to either create the labels or ignore missing ones gracefully. Neither happens — the entire issue creation fails.

## Root cause

`gh issue create --label` requires the label to already exist in the repository. There is no `--create-label` flag or auto-create behaviour. The error message is accurate but the failure mode is surprising because the issue body, title, and all other fields are valid.

## Fix

Two options:

**Option 1 — Drop `--label` entirely** (simplest): omit the flag if label management isn't critical. Labels can be added manually or via `gh issue edit` after creation.

**Option 2 — Create labels first**:
```bash
gh label create "platform" --repo casehubio/casehub-parent --color 0075ca
gh label create "architecture" --repo casehubio/casehub-parent --color e4e669
gh issue create --repo casehubio/casehub-parent --label "platform,architecture" ...
```

In scripts, check label existence before using:
```bash
gh label list --repo OWNER/REPO | grep -q "^platform" || gh label create "platform" ...
```

## Why this is non-obvious

The natural assumption is that `--label` behaves like `--title` — a field you supply at creation time. GitHub's web UI lets you type label names and creates them if they don't exist. The CLI does not share this behaviour. The fix (drop the flag) is trivial but you have to know that's the issue.
