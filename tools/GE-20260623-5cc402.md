## Symptom

Activity Monitor shows `com.apple.Virtualization.VirtualMachine` or `vfkit` using 14GB of memory. System feels like it's under memory pressure but actual application performance is fine.

## Context

Podman Desktop (or similar container tools using Apple's Virtualization.framework) is running a Linux VM via `vfkit`. The VM is used for running Docker-compatible containers (e.g. PostgreSQL for tests).

## Root cause

Activity Monitor and `ps aux` report **VSZ (virtual size)** — the full virtual address space the process has mapped, including memory the guest OS *may* use. The Virtualization.framework allocates a large contiguous virtual address range for the VM's maximum RAM limit upfront, but only maps physical pages on demand.

The physical RAM actually consumed is visible as **RSS (resident set size)**. For a Podman VM configured with 4GB and running only PostgreSQL, RSS is typically ~1.2GB.

## Fix

Check actual physical memory use:
```bash
# Configured max (not current use):
podman machine ls

# Actual physical RAM (RSS):
ps aux | awk '$11 ~ /vfkit/ {printf "RSS: %.0fMB\n", $6/1024}'
```

The RSS figure is what matters for memory pressure. The VSZ figure in Activity Monitor is misleading.

## Why this is non-obvious

Activity Monitor defaults to showing VSZ in the "Memory" column for processes using the Virtualization framework. Most developers assume "Memory" = physical RAM. The distinction only becomes clear when you notice RSS is an order of magnitude smaller than what Activity Monitor reports.
