## Symptom

IntelliJ MCP `get_file_text_by_path` or `read_file` returns content that looks correct, but the Java compiler rejects the file with a different error — citing a different number of arguments, a wrong type, or a missing symbol that the MCP clearly shows as present. The file "looks right" through IntelliJ MCP but `mvn compile` fails on the actual on-disk content.

## Context

After a series of Python file-fixing scripts wrote to disk, IntelliJ's MCP showed file contents from before the scripts ran. Reading `FinanceApprovalScenario.java` via `mcp__intellij__read_file` returned 9 trailing nulls; `python3 -c "with open(path) as f: print(repr(f.readlines()[131]))"` returned 15 trailing nulls (16 total args). The compiler agreed with the Python read, not the MCP read. This caused several rounds of incorrect diagnosis: "the file has the right content, so why does the compiler reject it?"

## What was tried (didn't work)

- Re-reading via MCP after each fix — still returned stale content
- Assuming compiler errors reflected the MCP file view — led to wrong fixes applied to wrong call sites

## Root cause

IntelliJ MCP's file reading tools read from IntelliJ's **internal document model** (the PSI/VFS cache), not directly from the filesystem. When files are modified externally (via Python scripts, `sed`, or other tools that bypass IntelliJ's editor), IntelliJ's VFS cache may not refresh immediately. The MCP returns the cached in-memory state, which can be seconds to minutes behind the on-disk state.

## Fix

When files are modified by external scripts (not through IntelliJ's editor):

1. **Use Python/bash reads to verify actual disk content**, not IntelliJ MCP reads:
   ```python
   with open(path) as f: content = f.read()
   ```
   or `sed -n '131p' file.java`

2. **Use `mcp__intellij-index__ide_sync_files`** to force IntelliJ to refresh its VFS cache after external modifications:
   ```
   ide_sync_files(paths=["src/main/java/..."])
   ```

3. **Trust the Java compiler over the MCP** when they disagree — `mvn test-compile` reads actual disk files.

4. **Rule of thumb:** MCP reads are authoritative for files edited through IntelliJ's editor. After external script modifications, use filesystem reads for ground truth.

## Why this is non-obvious

The MCP appears to be a direct file-reading tool — developers expect it to show what's on disk. The divergence only surfaces when external processes modify files faster than IntelliJ's VFS refreshes, which is uncommon in normal development (IntelliJ usually detects and re-reads changed files quickly). Under script-driven bulk editing, the lag becomes significant. The symptom looks like a compilation bug or a script bug, not a cache staleness issue — leading to multiple rounds of incorrect diagnosis.
