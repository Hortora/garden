## Symptom

After `git rebase -i BASE` with `GIT_SEQUENCE_EDITOR="cp $PLAN"`, `git log --oneline BASE..HEAD` shows far fewer commits than expected — with no error or warning.

## Context

Attempting to rename a single squash commit mid-branch using a `reword` plan containing only that one line. All commits after the reworded commit vanished from the branch.

```bash
# Plan file contained only:
reword abc1234 old commit message
# Commits def5678 ghi9012 (chronologically after abc1234) were silently dropped
```

## Root cause

`git rebase -i BASE` replays **only** the commits listed in the plan, in the order they appear. Any commit in `BASE..HEAD` not included in the plan is silently omitted — it becomes unreachable from the rebased branch. No warning is emitted.

The partial plan produced by `cp $PLAN` (with `GIT_SEQUENCE_EDITOR`) is treated as a complete statement of intent, not a targeted amendment.

## Fix

After detecting the truncation, cherry-pick the missing commits from the backup branch:

```bash
# cherry-pick the dropped commits in chronological order (oldest first)
git cherry-pick <sha1> <sha2> <sha3>
```

**Prevention:** Always list ALL commits in `BASE..HEAD` in the rebase plan. For a targeted message change on a non-HEAD commit, safer alternatives:

1. **HEAD commit:** `git commit --amend`
2. **Non-HEAD commit:** write the full plan with `pick` lines for every commit you want to keep — do not omit any

Always create a backup branch before rebase to enable recovery:
```bash
git branch backup/pre-rebase-$(date +%Y%m%d) HEAD
```

## Why this is non-obvious

Developers expect `reword` to act like a targeted operation — "change this message, leave everything else alone." The implicit contract ("every commit you want to survive must appear in the plan") is never surfaced by a warning or error. The symptom (fewer commits in `git log`) looks like git lost data, not like the developer's plan told it to.

Compounding factor: using `GIT_SEQUENCE_EDITOR="cp $PLAN"` with a machine-generated plan file makes it easy to produce a partial plan by mistake — the `cp` succeeds silently regardless of how many commits the plan covers.
