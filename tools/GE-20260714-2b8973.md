### Symptom
`HAS_META=no` from ctx.py even though `.meta` exists. work-end pre-conditions
fail ("`.meta` must exist") despite the scaffold having been created successfully
by work-start in the same session.

### Context
Two-repo casehub projects have a source code repo and a companion methodology
(workspace) repo. CLAUDE.md names them:
- **Project repo:** `/path/to/casehub/<name>` (source code)
- **Workspace:** `/path/to/public/casehub/<name>` (methodology — .meta, blog, plans)

ctx.py outputs:
- `WORKSPACE` = git root of CWD
- `PROJECT` = follows the `proj/` symlink to the companion

When CWD is the source code repo, ctx.py's naming inverts relative to CLAUDE.md:
- ctx.py `WORKSPACE` = CLAUDE.md "Project repo" (source code)
- ctx.py `PROJECT` = CLAUDE.md "Workspace" (methodology)

Skills that check `$WORKSPACE/design/.meta` look in the source code repo (wrong)
instead of the methodology repo (where .meta actually lives).

### Root cause
ctx.py defines WORKSPACE as the git root of CWD, not by semantic role. The
`proj/` symlink determines PROJECT. In two-repo setups where CWD is the source
repo, these names collide with CLAUDE.md's opposite naming convention.

### Fix
When ctx.py reports `HAS_META=no` in a two-repo project, check
`$PROJECT/design/.meta` (the companion repo) instead of `$WORKSPACE/design/.meta`.
Read .meta values directly from the correct path rather than relying on ctx.py's
parsed fields (BRANCH_NAME, PROJECT_SHA, ISSUE_N, COVERS will all be empty).

### Why this is non-obvious
The skill documentation uses `$WORKSPACE` to mean "the workspace repo" (methodology),
matching CLAUDE.md. But ctx.py's `$WORKSPACE` output means "git root of CWD" —
a different definition. The naming collision is silent; ctx.py doesn't warn when
.meta isn't found, it just sets `HAS_META=no`.
