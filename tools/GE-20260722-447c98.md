### Symptom
`ide_create_file` calls time out (>120s, eventually aborted at 300s) when targeting
a git worktree that was just opened as a new IntelliJ project via `ide_open_project`.
Multiple parallel calls all fail — the worktree project appears open but file creation
hangs indefinitely.

### Context
Git worktrees don't have `.idea` directories. When `ide_index_status` is called with
a worktree path, the error message says "Path exists but has no .idea directory — not
an IntelliJ project" — which sounds terminal. However, `ide_open_project` successfully
opens the worktree as a new project when `project_path` is set to any already-open
project (routing context).

The problem: immediately after `ide_open_project` returns "Project is open and ready",
the project is still indexing in the background. `ide_create_file` blocks on index
availability and times out.

### What was tried (didn't work)
- 5 parallel `ide_create_file` calls — all timed out at 120s, aborted at 300s
- Single sequential `ide_create_file` — same timeout
- `ide_edit_member` and `ide_replace_text_in_file` — also timed out on the freshly-opened project

### Root cause
`ide_open_project` returns before indexing completes for the new project. Subsequent
semantic operations (`ide_create_file`, `ide_edit_member`) require index availability
and block until indexing finishes — but the MCP timeout fires first.

The misleading `ide_index_status` error ("no .idea directory") is a red herring — 
IntelliJ CAN open directories without `.idea` as projects. The error comes from the
status check, not from an inability to open.

### Fix
1. Use `ide_open_project` with `project_path` set to any existing open project to open the worktree
2. For new file creation, fall back to the `Write` tool (edit hooks typically allow Write for new files)
3. For edits to existing files, use `Edit` with `replace_all: true` (typically allowed by hooks as find-and-replace)
4. Wait for indexing to settle before attempting `ide_create_file` or `ide_edit_member`

### Why this is non-obvious
The `ide_open_project` response says "Project is open and ready" — implying all
operations should work. The timeout happens silently with no indication that indexing
is the blocker. The `ide_index_status` error message discourages even trying to open
the worktree.
