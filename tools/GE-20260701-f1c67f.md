### Symptom
`ide_search_text`, `ide_find_symbol`, `ide_find_file` all return 0 results for any query in a TypeScript Yarn workspace monorepo, despite `ide_index_status` reporting `isDumbMode: false` and `ide_project_status` showing the project as open and active.

### Context
Discovered while implementing `@casehubio/pages-component-terminal` in the casehub-pages monorepo (Yarn 4.10.3, TypeScript 5, 9 workspace packages). Searched for `theme`, `PagesElement`, `activation.ts` — all returned empty. The project was confirmed open via `ide_project_status` (mode: active) and `ide_index_status` (isDumbMode: false, isIndexing: false).

### What was tried (didn't work)
1. `ide_search_text` with various queries (`theme`, `getComputedStyle`, `getPropertyValue`) — all 0 results
2. `ide_find_symbol` with known symbol names (`theme`) — 0 results
3. `ide_find_file` with known filenames (`activation.ts`, `PagesElement`) — 0 results
4. `ide_sync_files` to force re-sync — still 0 results after sync

### Root cause
The IntelliJ MCP Index Plugin relies on IntelliJ's Java/Kotlin indexing infrastructure. TypeScript files in Yarn workspace monorepos are not indexed by this path — the plugin's word index and symbol index don't cover `.ts`/`.tsx` files. Project lifecycle operations (`ide_project_status`, `ide_index_status`, `ide_open_file`) work correctly — it is specifically the search/navigation tools that return nothing.

### Fix
Fall back to the `Read` tool on known file paths for targeted reads. For broad text search, use bash `grep -r` despite project conventions discouraging it for class searches. `ide_project_status` and `ide_index_status` still work for project lifecycle management.

```bash
# Instead of: ide_search_text query="theme" project_path=/path/to/ts-project
# Use:
grep -r "theme" /path/to/ts-project/packages/ --include="*.ts" --include="*.tsx"
```

### Why this is non-obvious
- `ide_index_status` reports `isDumbMode: false` and `isIndexing: false` — the project appears fully indexed
- The same MCP plugin works correctly for Java/Kotlin projects in the same IntelliJ instance
- No error is returned — the tools succeed with 0 results, making it look like the query simply has no matches
- Project lifecycle tools (`ide_project_status`, `ide_open_file`) work fine, creating an impression that everything is operational
