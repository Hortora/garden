### Symptom
Subagent dispatched to work in a different repo produces correct-looking commits, but they land on the wrong branch — an unrelated feature branch left checked out by another session.

### Context
Multi-repo workflow (e.g. casehub-ops dispatching work to casehub-desiredstate). The target repo's working tree was on `issue-47-cte-pending-approval` from a paused session. The subagent committed to that branch without warning.

### What was tried (didn't work)
Nothing — the subagent succeeded. The problem was only discovered by running `git log` and `git branch --show-current` after the fact.

### Root cause
Subagents inherit the target repo's working tree state. No branch verification happens before the subagent starts. If another session left the repo on a feature branch, the subagent silently commits there.

### Fix
Before dispatching any cross-repo subagent, verify and switch:
```bash
git -C <target-repo> branch --show-current  # verify
git -C <target-repo> checkout main           # switch if needed
```

### Recovery
Cherry-pick commits to the correct branch, reset the contaminated branch:
```bash
git -C <repo> checkout main
git -C <repo> cherry-pick <sha1> <sha2> <sha3>
git -C <repo> branch -f <contaminated-branch> <original-sha>
```

### Why this is non-obvious
The subagent produces valid, passing commits. No error, no warning. The branch name doesn't appear in any subagent output. You only discover it by explicitly checking `git branch --show-current` — which you wouldn't normally do if the work "succeeded."

### Alternatives considered
- Worktree isolation per subagent: would prevent the issue but adds overhead and complexity for cross-repo work
- Checking branch in the subagent prompt: fragile — depends on the subagent following the instruction
- Checking branch in the controller before dispatch: reliable and cheap — the recommended fix
