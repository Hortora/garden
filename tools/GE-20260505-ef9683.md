## Symptom

A PR diff on GitHub looks correct — only the intended feature changes. But
`git log upstream/main..feature-branch` shows an extra unrelated commit riding
along below the feature commit.

## Context

Fork model: `origin` is your fork, `upstream` is the canonical repo. Local
`main` was one commit ahead of `upstream/main` (an unrelated feature not yet
PRed). Feature branches were cut from local `main` rather than `upstream/main`,
so that extra commit was silently included in every branch.

## What was tried

Reading the GitHub PR diff — it appears clean because GitHub diffs from the
merge-base, which happens to exclude the unrelated commit visually in some views.
Running `git log main..branch` — also misleading if local `main` includes the
extra commit.

## Root cause

`git log upstream/main..branch` is the authoritative check: it shows every
commit in the branch not yet in upstream. Local `main` is not upstream — if it
has diverged, branches cut from it carry that divergence into PRs.

## Fix

Always cut feature branches from `upstream/main` directly:

```bash
git checkout -b feat/my-feature upstream/main
```

Verify before pushing:

```bash
git log upstream/main..feat/my-feature
```

If a branch was already cut from local main with an extra commit, drop it with
`git rebase --onto`:

```bash
git rebase --onto upstream/main <sha-of-extra-commit> feat/my-feature
```

## Why this is non-obvious

The GitHub PR diff view doesn't always surface the extra commit clearly — it
shows changes from the merge-base, which can look identical to the intended
diff. The problem is only visible in `git log upstream/main..branch`.
