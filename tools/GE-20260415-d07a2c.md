## FastMCP serializes list tool returns as one TextContent per element — empty list produces zero content items

**ID:** GE-20260415-d07a2c
**Stack:** mcp Python SDK 1.27.0, FastMCP

**Symptom:** A `@mcp.tool()` function returning a Python list produces a `CallToolResult` where `result.content` has *N* items — one `TextContent` per list element — not a single item containing a JSON array. An empty list return produces `result.content = []` (zero items, not a TextContent containing `"[]"`).

**Context:** Writing tests for a FastMCP server. The tool returns `list[dict]` from a search function. Parsing with `json.loads(result.content[0].text)` returns the first dict when you expected the full list.

### What was tried (didn't work)
- `_parse(result)` using `json.loads(result.content[0].text)` — returned a single dict (first element) when the list had multiple items; returned `None` when the list was empty (content was empty, so the guard `if not result.content: return None` fired)
- Assumed FastMCP would wrap the entire list as a JSON array in a single TextContent, matching how dict returns work

### Root cause
FastMCP serializes each element of a returned list as a separate `TextContent` item. This is distinct from dict returns, which produce a single `TextContent` with the full JSON. An empty list → zero content items. A 3-element list → 3 content items.

### Fix
Use separate parse helpers based on the tool's declared return type:

```python
def _parse_dict(result) -> dict:
    """For tools returning dict (status, capture)."""
    if not result.content:
        return {}
    return json.loads(result.content[0].text)

def _parse_list(result) -> list:
    """For tools returning list (search). FastMCP: one TextContent per element."""
    if not result.content:
        return []  # empty list → zero content items
    return [json.loads(c.text) for c in result.content]
```

### Why this is non-obvious
FastMCP's dict return and list return look identical from the outside — both `result.isError = False`, both have `.content`. The difference is invisible until you have a multi-element list or an empty list. The MCP spec itself does not specify how SDK implementations should serialize Python collections, so the behaviour is implementation-defined and undocumented.
