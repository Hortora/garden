### Symptom

`git log issue-227 ^main` reports 2 commits. `git merge-base --is-ancestor issue-227 main` returns "IS ancestor." Both commands run on the same repo at the same time — contradictory results.

### Context

Occurs after a rebase-merge to main. The feature branch pointer still points to the original pre-rebase SHAs. Those SHAs are genuinely absent from main's object graph (which holds the replayed commits under new SHAs).

### Root cause

`git log A ^B` compares object IDs, not content. The old branch-tip SHA is not reachable from main, so git reports it as "not on main" — even though equivalent content landed there via rebase. The branch pointer was never advanced or deleted after the merge.

### Fix

Use `git merge-base --is-ancestor` as the definitive reachability check:

```bash
git merge-base --is-ancestor <branch> main && echo "IS ancestor" || echo "NOT ancestor"
```

This correctly answers "does main contain everything this branch has" even when SHAs diverged via rebase. Use `git log A ^B` only when you specifically need to compare object identity, not content reachability.

### Why this is non-obvious

`git log branch ^main` feels like the natural "is this merged?" check. The deception is that rebase produces new SHAs for identical content — so the old SHAs are genuinely absent from main's history. The command returns an accurate object-level answer to a question you didn't mean to ask.
