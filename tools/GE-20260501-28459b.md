## Rebuilding only the Qhorus runtime Maven module leaves the testing module stale — E2E tests fail silently with timeouts, no compile error

### Stack
Maven multi-module build with SNAPSHOT dependencies; Quarkus + Playwright E2E tests

### Symptom
After rebuilding a library's runtime module to pick up API changes, Playwright E2E tests time out waiting for elements that should appear (e.g. channel dropdown options). No compile error, no test error message indicating an API mismatch — just a timeout. The channel IS being created in `@BeforeEach` but doesn't appear in the fetched list.

### Context
A library ships two separate Maven modules: `runtime` (the main implementation) and `testing` (in-memory test doubles, e.g. `InMemoryChannelStore`). When the `runtime` API changes (e.g. `createChannel` gains new parameters), you rebuild `runtime`. But `testing` depends on internal interfaces from `runtime` — if `testing` is not also rebuilt, the `InMemoryChannelStore` may implement a stale version of the `ChannelStore` interface that doesn't handle the new parameters, causing channel creation to silently fail at runtime. The test sees no exception; the channel simply isn't in the store when the browser fetches it.

### What was tried
- Verified `@BeforeEach` was calling `createChannel` correctly (no compile error)
- Verified the REST endpoint worked in unit tests (`MeshResourceInterjectionTest` passed)
- Verified the channel was supposed to be in the store
- Rebuilt just the runtime module → no improvement
- Eventually rebuilt the `testing` module separately → tests passed

### Root cause
`mvn install -pl runtime --also-make` rebuilds the runtime module and its Maven dependencies, but NOT modules that depend on runtime (e.g. `testing`). The `testing` JAR in `~/.m2` remains at the old compiled state. When the test JVM loads both JARs, the `InMemoryChannelStore` implements an outdated interface version. The mismatch may be a no-op at the Java level (no `NoSuchMethod`) but causes incorrect runtime behaviour.

### Fix

When any Qhorus (or similar multi-module library) module changes, rebuild all consumer-visible modules:

```bash
# Rebuild runtime + testing + deployment in one pass
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn install -DskipTests -q \
  -pl runtime,testing,deployment --also-make \
  -f /path/to/qhorus/pom.xml
```

Or rebuild the entire library in one command:
```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn install -DskipTests -q \
  -f /path/to/qhorus/pom.xml
```

Avoid targeting individual submodules unless you are certain of the dependency graph.

### Why this is non-obvious
The failure symptom (E2E timeout, channel not found) looks like a timing issue or a bug in the channel creation code — not a stale JAR. The unit tests pass because `MeshResourceInterjectionTest` calls `tools.createChannel()` directly through the full Quarkus context, which uses the real runtime store, not the `InMemoryChannelStore`. Only E2E tests via the browser hit the stale testing-module path.
