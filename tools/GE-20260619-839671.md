## Inline vendored source as CI fix for local-only forks — avoids GitHub repo + publish pipeline setup

**ID:** GE-20260619-839671
**Stack:** Maven, GitHub Actions, any build system
**Symptom:** CI fails to resolve a SNAPSHOT dependency because it is a local-only fork — the modified branch was never pushed to any GitHub remote, so GitHub Packages has nothing to serve. `Could not find artifact X:Y:jar:1.0.0-SNAPSHOT`.
**Context:** A third-party library was forked locally to extract or modify a subset of its functionality. The fork was built and installed to `~/.m2` manually but never set up as a published package.

### What was tried (didn't work)

- Setting up `distributionManagement` and a publish workflow on the fork — requires creating a new GitHub repo, configuring the workflow, triggering the first publish run, and adding the repo as a `<repository>` in the consumer. High setup cost for a library you already own.
- Committing the pre-built JAR to the consuming repo — works but binary blobs in git are opaque, hard to audit, and block future source-level changes.

### Root cause

CI has no way to reach a branch that only exists on a local machine. The fix is to make the source visible to CI — either by publishing it or by making CI part of the same source tree.

### Fix

Copy the library's `src/main/java/**` directly into the consuming project's `src/main/java/`. Remove the external SNAPSHOT dependency from `pom.xml`. CI now compiles it as first-party code.

```bash
cp -r /path/to/fork/module/src/main/java/ /path/to/consumer/src/main/java/
```

Remove from `pom.xml`:
```xml
<!-- delete these -->
<dependency>
    <groupId>com.example</groupId>
    <artifactId>forked-lib</artifactId>
    <version>1.0.0-SNAPSHOT</version>
</dependency>
```

If the library includes non-Java resources alongside `.java` files (e.g. `.dat`, `.properties`), add a Maven resources declaration so they are bundled (see GE-0067):
```xml
<build>
    <resources>
        <resource>
            <directory>src/main/java</directory>
            <excludes><exclude>**/*.java</exclude></excludes>
        </resource>
        <resource>
            <directory>src/main/resources</directory>
        </resource>
    </resources>
</build>
```

### Why this is non-obvious

The reflex when a library is missing from CI is to publish it — set up GitHub Packages, add a publish workflow, configure repository URLs. Inlining the source is faster and produces less ongoing maintenance, but it feels like "wrong" direction because it grows the consuming repo. In practice, for a small, stable, owned fork, inlining is cleaner: no separate repo to keep alive, no publish timing races, no token permissions to manage.

### Alternatives considered

- **Publish to GitHub Packages**: correct long-term if the library will be consumed by multiple projects. Overhead: create repo, add publish workflow, trigger first run, configure consumer's `<repositories>`. Justified when the library has multiple consumers or changes frequently.
- **Commit pre-built JAR to the consuming repo**: avoids source duplication but creates binary blobs in git — hard to audit, rebuild, or patch.
