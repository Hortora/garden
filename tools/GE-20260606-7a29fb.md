## REST Assured GPath `$.findAll {}` returns null (not []) when no elements match

**Stack:** REST Assured 5.x, Groovy GPath / JsonPath

In REST Assured, `$.findAll { condition }` on a JSON array returns `null` when no
elements satisfy the condition — not an empty list `[]`. This means `.body("...", hasSize(0))`
throws a NullPointerException or assertion failure even though the filter is working
correctly: there simply are no matching elements.

### Symptom

```java
.body("$.findAll { it.domain != 'jvm' }", hasSize(0))
```

Fails with a Hamcrest assertion error (`was null`) even when all returned entries
correctly have `domain == 'jvm'` — the domain filter is working, but GPath returns
`null` for an empty filtered list.

### Root cause

REST Assured's JsonPath uses Groovy GPath semantics, where `findAll` on an empty
result returns `null` rather than `[]`. This differs from Java stream `.filter()`
and standard Groovy `List.findAll()` on an already-materialised list (which returns
`[]`). The `null` return is specific to GPath expression evaluation in JsonPath's
implementation.

### Fix

```java
// REST Assured GPath returns null (not []) when findAll yields no results
.body("$.findAll { it.domain != 'jvm' }", anyOf(nullValue(), hasSize(0)));
```

`anyOf(nullValue(), hasSize(0))` handles both cases: null (no matches) and an
empty list (if the implementation ever changes to return `[]`).

### Why this is non-obvious

Standard Java list filtering (`stream().filter(...)`) always returns an empty list,
never null. Even Groovy's `list.findAll { }` on a materialised list returns `[]`.
Only the GPath expression evaluation in REST Assured's JsonPath returns `null`,
and there is no documentation for this behaviour — it surfaces only as a confusing
test failure where a passing assertion appears to report "was null".
