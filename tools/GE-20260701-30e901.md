### Symptom
`ide_move_file` moves a Java file to the correct package but the wrong Maven module within a multi-module workspace.

### Context
Using `ide_open_workspace` to open 26 Maven repos as a single IntelliJ workspace. Moving `CaseMemoryStore.java` from `platform-api` to a destination of `src/main/java/io/casehub/memory` тАФ intended for the `memory-api` module (Tier 1 SPI).

### What was tried (didn't work)
Specified destination `src/main/java/io/casehub/memory` via `ide_move_file`. Expected the file to land in `memory-api/src/main/java/io/casehub/memory/` (the SPI module).

### Root cause
IntelliJ resolves the destination by finding the module whose existing source tree most closely matches the target package. The `memory/` module already had `io.casehub.memory.cbr.runtime` тАФ making `io.casehub.memory` a parent package of an existing package in that module. `memory-api/` had `io.casehub.memory.cbr` but the directory `io/casehub/memory/` didn't exist yet. IntelliJ picked `memory/` as the nearest match.

### Fix
After `ide_move_file`, verify the physical file location тАФ not just the package declaration. If the file landed in the wrong module, use `mv` to relocate within the repo. The import updates across the workspace are still correct (the FQCNs match).

### Why this is non-obvious
The package declaration and all cross-repo import updates are correct тАФ the move appears successful. The error is invisible unless you check which Maven module owns the file. In a workspace with 26 repos and dozens of modules, this is easy to miss.

### Alternatives considered
- Pre-creating the target directory in the intended module before calling `ide_move_file` тАФ not tested, may work
- Using explicit module-qualified paths in the destination тАФ `ide_move_file` accepts project-relative paths, not module-qualified ones
