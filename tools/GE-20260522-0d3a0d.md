### Symptom

After creating a squash working branch from main and running `git rebase -i`,
`git diff main <work-branch> --stat` showed a non-zero tree diff — files added
and deleted that shouldn't differ from main. A pure history rewrite cannot change
the tree state, so the diff revealed that something was wrong before the swap.

### Context

Multi-session workflow: one session creates a squash working branch from main
and begins classifying commits, while a second session concurrently runs
`work-end` on a feature branch, rebasing and merging commits onto local main.

### Root cause

A working branch created from main inherits main's `@{u}` tracking reference
(origin/main). The `git log @{u}..HEAD` range is computed at query time using
the local cached ref, not the live state of main. If another session modifies
local main between branch creation and squash execution, the working branch's
commit range no longer matches what's actually on main. The squash proceeds
against the inherited (stale) range — producing a working branch whose tree
diverges from current main.

### Fix

Before swapping the working branch onto main, run a two-dot diff:

```bash
git diff main <squash-work-branch> --stat
```

If the output is non-empty, main changed concurrently — abort. Delete the
working branch and re-run the squash from the current state of main:

```bash
git branch -D <squash-work-branch>
git log --oneline @{u}..HEAD   # verify the new range
# re-run squash from here
```

A three-dot diff (`git diff main...<work-branch>`) will NOT catch this — it
shows what the working branch added on top of the common ancestor, not the
actual tree difference at the tips.

### Why this is non-obvious

The working branch looks correct throughout: `git log @{u}..HEAD` shows a
plausible set of commits, the rebase succeeds cleanly, and the commit count
matches expectations. There is no error or warning. The only signal that
something is wrong is the post-rebase two-dot tree diff, which most squash
workflows don't include.
