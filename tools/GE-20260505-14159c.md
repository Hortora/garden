## init_garden.py writes unbolded drift counter — validate_garden --dedupe-check regex silently reads 0

**ID:** GE-20260505-14159c
**Stack:** Hortora soredium scripts

**Symptom:** Freshly initialised gardens never trigger the dedupe recommendation,
regardless of how many entries have been merged since the last sweep.

**Context:** `init_garden.py` creates `GARDEN.md`; `validate_garden.py
--dedupe-check` reads the drift counter from it. Both exist and are tested.

### What was tried (didn't work)

- Unit tests for `create_garden_md` pass — they use substring matching
- `validate_garden.py` runs without error
- The dedupe threshold check produces no warning even when drift exceeds the threshold

### Root cause

`create_garden_md` wrote:
```
Entries merged since last sweep: 0
```
(plain text)

But `validate_garden.py --dedupe-check` parses with:
```python
re.search(r'\*\*Entries merged since last sweep:\*\*\s*(\d+)', content)
```
(expects Markdown bold markers)

The regex never matches, so the counter is treated as `None` and drift
checking is silently skipped. Tests for `create_garden_md` asserted
`assertIn('Entries merged since last sweep: 0', content)` — a substring
match that passes regardless of whether bold markers are present.

### Fix

Write the bolded form in `create_garden_md`:
```python
'**Entries merged since last sweep:** 0\n'
```

And tighten the test assertion to match the exact format the parser expects:
```python
self.assertIn('**Entries merged since last sweep:** 0', content)
```

### Why this is non-obvious

Two separate files — the writer and the parser — used different formatting for
the same field. Both were independently tested and passing. The mismatch was
only visible by comparing the regex in `validate_garden.py` against the
template string in `init_garden.py`. No runtime error is produced — the
counter silently defaults to 0.
