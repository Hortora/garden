## Symptom

`gh api repos/other-org/other-repo/dispatches` returns 403 "Resource not accessible by integration" when called from a GitHub Actions workflow using `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}`.

## Context

Multi-repo CI/CD chain where repo A publishes and then dispatches a `repository_dispatch` event to repo B to trigger its build. Using `GITHUB_TOKEN` is the obvious choice since it's always available.

## What was tried

Used `GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}` in the "Trigger downstream CI" step across all publish.yml files. Dispatch silently appeared to succeed in some cases but always returned 403. The error message "Resource not accessible by integration" gives no indication that scope is the issue.

## Root cause

`GITHUB_TOKEN` is scoped to the repository where the workflow runs. It cannot make API calls that affect other repositories, including `repository_dispatch`. This is by design — GITHUB_TOKEN has no cross-repo permissions.

## Fix

Use a classic PAT (not fine-grained) with `repo` scope stored as an org or repo secret (e.g. `GH_PAT`):

```yaml
- name: Trigger downstream CI
  env:
    GH_TOKEN: ${{ secrets.GH_PAT }}   # NOT secrets.GITHUB_TOKEN
  run: |
    gh api repos/other-org/other-repo/dispatches \
      -f event_type=upstream-published
```

## Why non-obvious

`GITHUB_TOKEN` works for everything within the same repo (PRs, releases, packages, issues). The 403 error message mentions "integration" not "scope" or "cross-repo", making the cause non-obvious. Fine-grained PATs also don't work for org-level operations (see related gotcha).

**Why:** Classic PAT with `repo` scope is the only token type that works for cross-repo dispatch. Fine-grained PATs lack support for org-level `repository_dispatch`.

**How to apply:** Any workflow step that calls `gh api repos/OTHER-ORG/OTHER-REPO/...` must use a PAT, not GITHUB_TOKEN.
