### Symptom
`work-start` hard-stops at the next session: "Orphaned `.meta` on main — invoke
`work-end` to complete or discard the abandoned branch." No epic branch was started
this session; the previous session thought the epic was closed.

### Context
Using a workflow where epic branches carry branch-local markers committed to the
branch itself (`design/.meta` for branch state, `design/EPIC-CLOSED.md` for
close marker). The previous session merged the epic branch into main via
`git merge --ff-only <branch>` rather than the prescribed `work-end` path.

### Root Cause
`merge --ff-only` is perceived as a "clean" operation — it only moves the branch
pointer. But every commit on the epic branch lands verbatim on main, including
commits that were intended as branch-local artifacts. `.meta` committed in the
`init(<branch>): scaffold workspace branch` commit arrives on main and is
indistinguishable from an orphaned leftover.

### Fix
After any `merge --ff-only` of an epic branch into main, immediately check for and
remove branch-only artifacts before the next session begins:

```bash
git rm design/.meta design/EPIC-CLOSED.md 2>/dev/null
git commit -m "chore: remove branch-only markers after ff-only merge"
git push
```

Prevention: use `work-end` rather than manual `merge --ff-only` — the skill's
Step 9 marks the branch closed on the **epic branch** only (not main) and Step 10
returns to main cleanly. Manual fast-forward bypasses this and inherits all
branch-committed files.

### Why This Is Non-obvious
`--ff-only` looks like it "just moves a pointer." Developers don't think of it as
bringing files along — that's what `merge --no-ff` and squash merges feel like.
The file-location semantics of workflow markers only matter if a later tool checks
for their presence on main, which the work-start skill does.
