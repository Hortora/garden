### Symptom
`ide_move_file` hangs for 300s then returns a timeout error when moving a file
from one IntelliJ project to another (e.g. quarkmind → blocks). No error
message — just silence until the MCP idle timeout fires.

### Context
Cross-repo code extraction where the source and destination share the same
Java package name (`io.casehub.blocks.summarisation`). Both projects are open
in IntelliJ. The move destination is an absolute path in a different project.

### What was tried (didn't work)
1. `ide_move_file` with absolute destination path in the other project — 300s timeout
2. Dispatching 7 parallel `ide_read_file` calls immediately after the timeout —
   all 7 timed out, MCP became completely unresponsive for several minutes

### Root cause
`ide_move_file` operates within a single IntelliJ project context. When the
destination is outside the project's content roots, the operation has no
handler and hangs indefinitely until the MCP timeout fires.

The cascading failure occurs because the MCP server queues all tool calls
behind the stuck operation. After a timeout, the queue is not flushed —
subsequent calls execute against a server still processing the backlog.

### Fix
For cross-project extraction where package names are identical:
1. Read each file via `ide_read_file` (one at a time, not parallel)
2. Write to destination via the Write tool
3. `ide_sync_files` on both projects so IntelliJ sees the changes
4. `ide_refactor_safe_delete` on the source files (catches missed references)

After any MCP timeout, wait for recovery before issuing new calls — do not
blast parallel requests.

### Why this is non-obvious
The `ide_move_file` API accepts any path as destination — nothing in the
interface suggests it's limited to intra-project moves. The failure mode is
a silent hang, not an error message explaining the limitation.
