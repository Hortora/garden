### Symptom

A bash script using `jq`'s `@base64` to encode JSON objects, then decoding them with `base64 --decode` inside a `while IFS= read -r` loop, fails with:

```
base64: invalid input
```

The error is cryptic and suggests corrupted data or an encoding mismatch.

### Context

Pattern used in GitHub Actions workflow scripts to iterate over arrays of JSON objects in bash:

```bash
ITEMS=$(gh api "/some/endpoint" --jq '.[] | @base64')
while IFS= read -r b64; do
  DECODED=$(echo "$b64" | base64 --decode)
  field=$(echo "$DECODED" | jq -r '.field')
done <<< "$ITEMS"
```

### What was tried

- Verified the API was returning valid JSON
- Checked base64 encoding/decoding manually on small objects — worked fine
- Added `2>/dev/null` to suppress errors — script silently produced wrong output

### Root cause

`jq`'s `@base64` filter wraps its output at 76 characters (standard PEM line length). For JSON objects larger than ~57 bytes (which base64-encodes to 76+ chars), the output spans multiple lines.

`while IFS= read -r b64` reads **one line at a time**. It reads only the first 76-character chunk of the base64 string, leaving the rest in the buffer. `base64 --decode` then gets an incomplete, invalid base64 string.

The symptom — `base64: invalid input` — points to the base64 command, not to jq, completely misleading the diagnosis.

### Fix

Replace `@base64` encoding with `@tsv` to extract specific fields as tab-separated values:

```bash
# Before (broken for large objects):
ITEMS=$(gh api "/endpoint" --jq '.[] | @base64')
while IFS= read -r b64; do
  DECODED=$(echo "$b64" | base64 --decode)
  field1=$(echo "$DECODED" | jq -r '.field1')
  field2=$(echo "$DECODED" | jq -r '.field2')
done <<< "$ITEMS"

# After (robust):
ITEMS=$(gh api "/endpoint" \
  --jq '.[] | [.field1, .field2, .field3] | @tsv')
while IFS=$'\t' read -r field1 field2 field3; do
  [ -z "$field1" ] && continue
  # use $field1, $field2, $field3 directly
done <<< "$ITEMS"
```

`@tsv` uses literal tab as the field delimiter (safe from jq line wrapping) and escapes tabs/newlines within field values as `\t`/`\n` escape sequences.

### Why this is non-obvious

The error implicates `base64 --decode`, not `jq`. The obvious diagnosis is corrupted input or an encoding mismatch. The actual cause — jq line-wrapping — is not mentioned in the jq documentation for `@base64`. Small JSON objects never trigger the problem (they encode to <76 chars), so the bug only appears with larger objects.
