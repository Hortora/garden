### Symptom

`git push` fails for a `chore: branch closed` empty commit with:
```
Pre-push: 1 commit(s) about to push
Run /git-squash to review history before pushing, or use --no-verify to skip.
error: failed to push some refs to '...'
```

The exact same empty commit on a different branch pushes silently without triggering the hook.

### Context

Occurs when using a project-local pre-push hook (via `core.hooksPath = .githooks`) that
unconditionally blocks any push containing commits. The hook does not inspect commit
content — it only counts commits in a computed range.

The critical variable: whether the branch already has an upstream on the remote.

### Root cause

The hook computes the commit range differently based on remote topology:

**Branch with existing upstream** — `remote_sha..local_sha`:
```bash
range="$remote_sha..$local_sha"
```
A single new empty commit lands in this range → `total=1` → hook exits 1.

**Branch new to remote** (`remote_sha = 0000...0000`) — `origin/main..local_sha`:
```bash
base=$(git rev-parse --verify origin/main)
range="$base..$local_sha"
```
If the branch was already rebase-merged into main, all its commits are reachable from
`origin/main` → range is empty → `total=0` → hook skips silently.

### Fix

Use `--no-verify` when pushing `chore: branch closed` empty commits to branches that
already have an upstream. This is the only commit type that is definitionally safe to
skip squash review for — there is no content to squash.

```bash
git commit --allow-empty -m "chore: branch closed"
git push --no-verify --set-upstream origin <branch>
```

### Why this is non-obvious

The hook's error message implies a uniform policy ("all pushes require squash review").
Nothing in the message hints that the same empty commit will behave differently based on
whether the branch was ever pushed before. The asymmetry is invisible until you read the
hook source and trace the `remote_sha` branching logic.
