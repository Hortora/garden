## The technique

To determine whether a test failure was introduced by your changes or pre-existed them — without losing current work:

```bash
# 1. Stash your current changes
git stash

# 2. Restore the specific files you want to test from an older commit
git checkout <known-good-commit> -- path/to/file1 path/to/file2

# 3. Run the tests
npm test / mvn test / pytest

# 4. Compare results with your current state:
#    same or worse → pre-existing; better → your changes introduced it

# 5. Restore your current work
git checkout HEAD -- path/to/file1 path/to/file2
git stash pop
```

The key insight: `git checkout <commit> -- files` only restores those specific files, leaving everything else (including compiled classes, test configs, other source files) in their current state. You're testing the old version of the suspected files against the current test infrastructure.

## Why this is non-obvious

Most developers either: (a) `git stash` and `git checkout <old-commit>` — putting the entire repo in detached HEAD and losing current work; or (b) manually edit files to remove recent changes — fragile and easy to miss something. The targeted `git checkout <hash> -- files` pattern is surgical, reversible, and takes 30 seconds.

## When to use

E2E or integration test failures that look like regressions but you're not sure. Especially useful when:
- The test infrastructure is complex and might have its own bugs
- Multiple changes landed in the same session
- You want to confirm before spending time debugging

## Example

In Claudony, `ChannelPanelE2ETest` showed 4 failures after session.html and terminal.js changes. Suspected regression. Targeted checkout of those files from the handover commit revealed the same (actually worse) failures — confirming pre-existing flakiness, not a regression. Total investigation time: ~2 minutes.
