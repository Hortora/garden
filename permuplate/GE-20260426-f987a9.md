### Symptom
`method X(...) is already defined in class JoinNGate` compilation errors appear for ALL generated arities, not just one. The method body appears twice in the generated file with slightly different content.

### Context
Adding a new method to a Permuplate template class (`Join0Gate`) using `@PermuteDeclr(name="not", type="Join${i}Gate<END, DS, ${alphaList}>")` on the METHOD declaration (to set the return type) instead of on a parameter:

```java
@PermuteDeclr(name = "not", type = "Join${i}Gate<END, DS, ${alphaList}>")
public Join0Gate<Void, DS, A> not(Consumer<Join0Gate<Void, DS, A>> scopeFn) { ... }
```

### Root cause
`@PermuteDeclr` on a **method** is processed by `PermuteDeclrTransformer` as a rename+retype operation. This transformer runs in a post-pass over all methods — it finds the method by name and generates a modified clone, but does not remove the original. The result is two copies of each method in every generated class.

This is distinct from `@PermuteDeclr` on a **parameter** (which is consumed in-place and does not create a duplicate).

### Fix
Do not use `@PermuteDeclr(name=..., type=...)` to set the return type of a method in a Permuplate template. Instead, use `@PermuteReturn` for the return type and `@PermuteSelf` for self-referential returns:

```java
// Correct: @PermuteReturn for the return type
@PermuteReturn(className = "Join${i}Gate", typeArgs = "'END, DS, ' + alphaList", alwaysEmit = true)
public Object not(
        @PermuteDeclr(type = "java.util.function.Consumer<Join${i}Gate<Void, DS, ${alphaList}>>")
        Object scopeFn) { ... }
```

`@PermuteDeclr` on a parameter is safe and works correctly.

### Why this is non-obvious
The `@PermuteDeclr` documentation says "renames the method name and (optionally) return type." This reads as "modifies in place," not "generates a duplicate." The symptom appears for ALL arities simultaneously, which looks like a code generation ordering issue rather than a transformer duplication bug. The fix — using `@PermuteReturn` instead — is non-obvious because `@PermuteDeclr` appears to offer the same capability.
