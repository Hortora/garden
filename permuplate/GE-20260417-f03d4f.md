## The technique

When a template class needs one fixed type parameter plus an expanding set, annotate only the **second** (expanding sentinel) type parameter with `@PermuteTypeParam` rather than the first. This gives a "fixed `A` + growing tail" pattern.

## How it works

```java
@Permute(varName = "i", from = "3", to = "10", className = "Join${i}")
public class Join2<A,
        @PermuteTypeParam(varName = "k", from = "2", to = "${i}",
                          name = "${alpha(k)}") B> {
```

- `A` — fixed; present in all generated classes unchanged
- `B` — sentinel; `@PermuteTypeParam` replaces it with `B, C` (for i=3), `B, C, D` (for i=4), ..., `B, C, D, E, F, G, H, I, J` (for i=10)

Generated results:
- `Join3<A, B, C>` — A fixed, two expanding params
- `Join4<A, B, C, D>` — A fixed, three expanding params
- `Join10<A, B, C, D, E, F, G, H, I, J>` — A fixed, nine expanding params

The `from="2"` in `@PermuteTypeParam` starts the alpha sequence at `B` (index 2), so the expansion continues naturally from where the fixed `A` left off.

## Why this is non-obvious

The natural instinct is to annotate the **first** type parameter as the sentinel. But annotating the first param with `@PermuteTypeParam` from=1 to=${i} would replace `A` with the full sequence `A, B, C` — giving `Join3<A, B, C>` correctly but treating `A` as part of the expanding set rather than a fixed anchor. Placing the annotation on the second param with from=2 distinguishes "A is fixed" from "B onwards expands", enabling hybrid templates where the first type param has a stable identity across all arities.
