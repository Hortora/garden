## Symptom

A method with both `@PermuteMethod(varName="m", ...)` and `@PermuteParam(to="${m}", ...)`
produces clones where the parameter list is not expanded — only the sentinel parameter
`v1` appears, not `v1, v2, v3`. Call sites in the method body are also not expanded.
No error is emitted.

## Context

Occurs when `@PermuteParam(to="${m}")` references the inner `@PermuteMethod` variable `m`
for the expansion bound. The outer `PermuteParamTransformer.transform()` runs before
`@PermuteMethod` cloning — at that point, `m` is not bound in the `EvaluationContext`,
so JEXL throws an undefined-variable exception that is silently swallowed, and the
parameter expansion is skipped.

## What was tried

- Verified `@PermuteParam` syntax is correct (works on standalone methods)
- Added `@PermuteBody` as a workaround: manually coded `rd.addVariableFilter(${typeArgList(1,m,'v')}, predicate)`
- Traced the evaluation failure to `m` being undefined in the outer context

## Root cause

`PermuteParamTransformer.transform()` runs on the entire class in the outer pipeline with
the outer `EvaluationContext` (which has `i` but not `m`). Methods with `@PermuteMethod`
need their `@PermuteParam` processed **inside** `applyPermuteMethod`, after each clone is
created with the inner context where `m` (or `j`, `n`, etc.) is bound.

## Fix

Two-part fix:

**1. Skip methods with `@PermuteMethod` in the outer transform:**
```java
// In PermuteParamTransformer.transform(), skip methods that carry @PermuteMethod:
if (hasPermuteMethod(method)) return; // deferred to clone-level processing
```

**2. Run PermuteParamTransformer on each clone inside applyPermuteMethod:**
```java
// In InlineGenerator.applyPermuteMethodClone(), before @PermuteBody:
PermuteParamTransformer.transform(tmpClass, innerCtx, null);
// Now @PermuteParam(to="${m}") evaluates correctly with m in scope
// and expandAnchorAtCallSites fires naturally on the method body
```

After the fix, `@PermuteBody` workarounds for call-site expansion are no longer needed.

## Why this is non-obvious

`PermuteParamTransformer` correctly expands call sites in method bodies when it runs —
the `expandAnchorAtCallSites` code path exists and works. The failure is purely a scoping
issue: the outer pipeline runs the transformer too early. A developer would not expect
the transformer order to matter for method-scoped variables, and the silent exception
swallowing offers no diagnostic.
