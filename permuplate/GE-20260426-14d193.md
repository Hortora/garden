### Summary
When a Permuplate template method needs to return a generated class that includes the method's own type parameter `T`, reference `T` directly in the `@PermuteReturn(typeArgs=...)` string rather than using `@PermuteTypeParam` to rename `T`.

### Context
Building a scope builder family (`ScopeGateN<CTX, B, C, ...>`) where each `join()` method adds a new fact type and returns the next scope gate in the chain:

```java
// Goal: ScopeGate1<CTX, B>.join(source) ‚Üí ScopeGate2<CTX, B, T>
// where T is the method's generic type param (inferred from the source)
```

The natural Permuplate approach is `@PermuteTypeParam` to rename `T` to the appropriate alpha letter:

```java
@PermuteReturn(className = "ScopeGate${n+1}", typeArgs = "'CTX, ' + typeArgList(2, n+2, 'alpha')", when = "n < 10")
@PermuteTypeParam(varName = "k", from = "${n+1}", to = "${n+1}", name = "${alpha(k+1)}")
public <T> Object join(Function1<CTX, DataSource<T>> source) { ... }
```

This fails with a JavaParser parse error: `"Encountered unexpected token: ','"` during generation.

### Technique
Reference `T` directly in the `typeArgs` expression ‚Äî no `@PermuteTypeParam` needed:

```java
@PermuteReturn(className = "ScopeGate${n+1}",
               typeArgs = "'CTX, ' + typeArgList(2, n+1, 'alpha') + ', T'",
               when = "n < 10")
public <T> Object join(Function1<CTX, DataSource<T>> source) {
    descriptor.sources.add((Function1<CTX, DataSource<?>>) (Object) source);
    descriptor.filters.add(null);
    return cast(new ScopeGate1<>(descriptor)); // coherence renames to ScopeGate${n+1}
}
```

For `n=1`: typeArgs = `"CTX, B, T"` ‚Üí return type `ScopeGate2<CTX, B, T>` where `T` is the method's own type param, inferred from the DataSource argument. Java type inference resolves `T=Person` when the caller passes `CTX::persons`. ‚úì

### Why this is non-obvious
`@PermuteTypeParam` is the documented approach for adding method type parameters in Permuplate templates. Developers would naturally reach for it. However, `@PermuteTypeParam` on standalone (non-`@PermuteMethod`) methods processes the annotation differently and can cause parse errors when the `from`/`to` expressions reference outer Permute variables. The direct `T` approach avoids the annotation machinery entirely and is simpler ‚Äî but requires knowing that `T` from the template method signature is preserved in generated code and can be used directly in `typeArgs`.

### Caveats
- `T` is NOT renamed in the generated classes (it stays as `T`). This is fine when the caller's type inference resolves it correctly.
- If you need `T` renamed to a specific alpha letter in the class's type parameter list, this technique doesn't help ‚Äî use `@PermuteMethod` with `@PermuteTypeParam` inside the inner loop instead.
