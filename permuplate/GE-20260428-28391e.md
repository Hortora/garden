## Symptom

Runtime NullPointerException — a method on the template class (e.g., `Path3.path(Function1, Predicate1)`) returns null and triggers NPE downstream. The generated classes (Path4..6) work correctly; only the template class itself silently returns null.

## Context

A Permuplate template class with `keepTemplate=true` has new methods added using `@PermuteBody` to define their bodies. The `return null;` placeholder is used in the template class body (standard pattern when `@PermuteBody` handles everything).

## Root Cause

`@PermuteBody` replaces method bodies only in the **generated** classes (those created from the template iteration). When `keepTemplate=true`, the template class itself is emitted to the output with its **original body** — which is `return null;` when written as a placeholder.

## Fix

Write actual delegation bodies in the template class alongside `@PermuteBody`:

```java
// @PermuteBody handles generated Path4..6
@PermuteBody(body = "{ return delegateTo(fn); }")
// Actual body handles the template class Path3 itself
public Path2<END, T, B, C> pathVariant(Function1<A, B> fn) {
    return delegateTo(fn);  // real body, not return null
}
```

The `@PermuteBody` handles generated classes; the actual body handles the template class.

## Why This Is Non-Obvious

The canonical counterpart (`keepTemplate=false`) never emits the template class — so `return null;` never reaches compiled output. When porting a template from `keepTemplate=false` to `keepTemplate=true`, this creates an invisible asymmetry: `@PermuteBody` works for generated classes but is silently ignored for the template itself. No compile error, no warning — only a runtime NPE.
