## Symptom

A template class using `@PermuteVar` for cross-product generation (e.g., `i × k` combinations) fails to compile after adding two `@PermuteTypeParam` annotations — one for each axis. The generated class has duplicate type parameters (`L1, L2, L3, L1, L2` instead of separate sets).

## Context

A cross-product template like `BiCallable1x1` uses `@Permute` with `extraVars = {@PermuteVar(varName="k", ...)}`. Attempting to generify with two independent type param expansions — one for the `i` axis, one for the `k` axis:

```java
@Permute(varName="i", from="2", to="4", className="BiCallable${i}x${k}",
         extraVars = {@PermuteVar(varName="k", from="2", to="4")})
public interface BiCallable1x1<
        @PermuteTypeParam(varName="j", from="1", to="${i}", name="L${j}") A,
        @PermuteTypeParam(varName="m", from="1", to="${k}", name="R${m}") B> {
```

## Root cause

`@PermuteTypeParam` expansion iterates through combinations generated by `buildAllCombinations()`. For cross-product templates, each combination includes both `i` and `k`. When two `@PermuteTypeParam` annotations are present with different `varName`s, the transformer expands both within the same combination pass, producing overlapping/redundant type parameter sets rather than independent groups.

## Workaround

Currently none — cross-product templates (`@PermuteVar` / `extraVars`) cannot be genericised with two independent type parameter groups via `@PermuteTypeParam`. Keep `Object` types for the affected parameter groups.

Single-axis templates are unaffected — one `@PermuteTypeParam` works correctly.

## Why this is non-obvious

Cross-product generation otherwise works perfectly. The natural intuition is "two axes = two `@PermuteTypeParam` annotations with different `varName`s." The failure mode (duplicate type params in generated output rather than a clear error) takes time to diagnose.
