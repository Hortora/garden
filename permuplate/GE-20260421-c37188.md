## Symptom

A `@PermuteMacros` or `@Permute(macros=)` entry like `"prev=${i-1}"` is defined and used
in `@PermuteTypeParam(to="${prev}")`. The annotation compiles, but the generated class has
far fewer type parameters than expected — silently truncated at 1 or 0.

## Context

Occurs when a macro evaluates to a numeric value and the result is later consumed by
`EvaluationContext.evaluateInt()` (called by `@PermuteTypeParam` `from`/`to` bounds,
`@PermuteParam` bounds, etc.). The macro system stores the evaluated result via
`EvaluationContext.evaluate()`, which always returns a `String` regardless of the
JEXL expression's type.

## What was tried

- Verified macro value syntax is correct (`${i-1}` is valid JEXL)
- Checked that `prev` was in scope in the JEXL context — it was
- Traced the evaluation and found the value was `"2"` (String), not `2` (Integer)

## Root cause

`EvaluationContext.evaluate()` uses JEXL `evaluate()` which always coerces to String via
interpolation. When a macro like `prev=${i-1}` stores result `"2"` into the context map,
and `evaluateInt("${prev}")` later reads it back, the `instanceof Number` check fails
for `String`, causing a silent fallback to 0 or throwing an exception that is swallowed.

## Fix

Add `EvaluationContext.evaluateRaw()` that returns the native JEXL type for
single-interpolation templates. Distinguish via: starts with `${`, ends with `}`,
no second `${` inside. For matching templates, call `JEXL_ENGINE.createExpression(inner).evaluate(ctx)`
directly (returns `Integer`, `Long`, etc.) instead of the string-interpolation path.

Use `evaluateRaw()` when storing macro results so numeric values are preserved as
integers in the context map.

```java
// Single-interpolation: "${i-1}" → Integer 2 (not String "2")
public Object evaluateRaw(String template) {
    if (template.startsWith("${") && template.endsWith("}")
            && template.indexOf("${", 2) < 0) {
        String inner = template.substring(2, template.length() - 1);
        return JEXL_ENGINE.createExpression(inner).evaluate(jexlCtx);
    }
    return evaluate(template); // multi-segment: String always fine
}
```

## Why this is non-obvious

The JEXL expression `${i-1}` evaluates to `Integer` when called via `evaluate()` in
isolation. But `EvaluationContext.evaluate()` wraps in an interpolation template — the
result is always coerced to String. A developer would reasonably expect the integer to
flow through, especially since `evaluateInt()` succeeds when given `${i-1}` directly.
The failure only surfaces when macros act as an indirection layer.
