## Context

Using `@PermuteCase` to expand a switch statement via an inner loop. The
`body` annotation attribute is a JEXL-evaluated template for the statements
to insert in each new case.

## Symptom

The generated switch has the correct number of cases inserted (e.g. `case 2:`
`case 3:`) but each case has **no statements** — they silently fall through to
`default`. No compile error, no processor warning, no exception surfaced.

## Root cause

`PermuteCaseTransformer.buildSwitchEntry()` wraps the evaluated body string in
`{...}` and calls `StaticJavaParser.parseBlock("{" + bodyStr + "}")`. If the
body contains a Java string literal (`"arg"`, `"prefix"`, etc.), the
`StaticJavaParser` parser throws a `ParseProblemException`. The catch block
swallows it silently and returns `new BlockStmt()` — an empty block:

```java
try {
    block = StaticJavaParser.parseBlock("{" + bodyStr + "}");
} catch (Exception e) {
    block = new BlockStmt();  // silent empty fallback
}
```

Java string literals inside the JEXL-substituted body confuse the parser
context. The case is inserted with no body statements.

## Fix

Avoid Java string literals inside `body`. Use primitive expressions or
method calls instead:

```java
// ❌ Fails silently — string literal in body
@PermuteCase(varName = "k", from = "2", to = "${i}",
             index = "${k}", body = "return \"prefix\" + ${k};")

// ✅ Works — primitive expression
@PermuteCase(varName = "k", from = "2", to = "${i}",
             index = "${k}", body = "return ${k};")

// ✅ Works — method call producing a string
@PermuteCase(varName = "k", from = "2", to = "${i}",
             index = "${k}", body = "return String.valueOf(${k});")
```

## Why this is non-obvious

The symptom (cases with no body, falling through to default) appears to be a
logic error in the template, not a parse failure. There is no error message,
no warning, and no indication the body evaluation failed. A developer would
naturally suspect the wrong `from`/`to` range or annotation placement before
discovering the root cause is a silent parse exception.

### Why this fix

Avoiding string literals in body is simpler and less fragile than patching
the catch block. Method calls like `String.valueOf()` or `Integer.toString()`
are idiomatic Java and compose well with JEXL expressions.
