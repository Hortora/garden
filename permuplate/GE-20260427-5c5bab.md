## Symptom

Adding `@PermuteFilter("i != 2")` to a plain method in a `@Permute` template class has no visible effect — the method is generated at every arity regardless. No error, no warning. The condition is silently ignored.

## Context

When adding a new overload to a `@Permute` template class and needing to suppress it at a specific arity (e.g. to avoid a 2-param lambda ambiguity), `@PermuteFilter` on a regular (non-`@PermuteMethod`) method does nothing.

Concrete trigger: adding `filter(NCtxPredicate2<A,B>)` (no-ctx 2-arg) alongside `filter(Predicate2<DS,B>)` (ctx-first single-fact) at i=2. Both are 2-param functional interfaces — Java cannot resolve which overload a 2-param lambda targets, giving "reference to filter is ambiguous."

## What was tried

`@PermuteFilter("i != 2")` placed directly on the method — silently no-ops.

## Root cause

`@PermuteFilter` works on two scopes only:
1. **Class-level** — filters entire generated class instances out of the cross-product
2. **`@PermuteMethod` clone sets** — filters individual method clones generated by `@PermuteMethod`

A regular method that appears once per generated class is not part of any filterable clone set, so `@PermuteFilter` has nothing to act on.

## Fix

Wrap the method with a single-clone `@PermuteMethod` whose range evaluates to exactly one value per class generation, then add `@PermuteFilter`:

```java
// Before (filter silently ignored):
@PermuteFilter("i != 2")
public Object filter(
        @PermuteDeclr(type = "NCtxPredicate${i}<${alphaList}>")
        Object pred) { ... }

// After (filter effective — one clone per class, suppressed at i=2):
@PermuteFilter("i != 2")
@PermuteMethod(varName = "x", from = "${i}", to = "${i}", name = "filter")
@PermuteBody(body = "{ ... }")
public Object filterAllFactsNoCtx(
        @PermuteDeclr(type = "NCtxPredicate${i}<${alphaList}>")
        Object pred) { return null; }
```

The `@PermuteMethod` with `from="${i}", to="${i}"` creates exactly one clone per generated class (at `x=i`), making it part of a clone set that `@PermuteFilter` can act on. `name="filter"` renames the generated method. This is the same idiom used by `filterLatest` in JoinBuilder.

## Why this is non-obvious

The Permuplate docs describe `@PermuteFilter` as filtering permutations, which reads naturally as "filter out this method at certain arities." There is no indication that the scope distinction (class-level vs. clone-set-level) determines whether the annotation has any effect. The silent no-op is the worst possible failure mode — no error, wrong generated output.
