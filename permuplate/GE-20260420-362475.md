## Context

Permuplate's annotation processor stores per-method data in a map keyed by
`MethodDeclaration` AST nodes, across multiple transformer passes. Transformers
(e.g. `PermuteTypeParamTransformer`) mutate method nodes — renaming parameters,
replacing type references, etc.

## Problem

Using a standard `HashMap<MethodDeclaration, String>` fails when transformers
mutate the key nodes between Phase 1 (data collection) and Phase 2 (data use).
JavaParser AST nodes override `hashCode()` and `equals()` based on structural
content. Mutating a node changes its hash, so the key is effectively lost — the
map lookup silently returns `null` instead of the previously stored value.

## Solution

Use `IdentityHashMap<MethodDeclaration, String>` which keys by object identity
(`==`) rather than structural equality. Transformer mutations do not change
object identity, so the map remains valid across transformation passes.

```java
// ❌ Fails silently after transformation — hashCode changes
Map<MethodDeclaration, String> data = new HashMap<>();

// ✅ Stable across transformations — identity never changes
Map<MethodDeclaration, String> data = new IdentityHashMap<>();
```

## When to apply

Any time you collect data about AST nodes in one pass and retrieve it in a
later pass that runs transformers in between. Specifically needed for
multi-phase inference (e.g. alpha growing-tip: collect alpha letter in Phase 1
before `PermuteTypeParamTransformer` runs, retrieve it in Phase 2 after).

## Why this is non-obvious

JavaParser nodes have structural `equals()` and `hashCode()` by design — this
is what makes tree comparison and deduplication work. The side-effect that maps
break under mutation is only discoverable by observing the `null` lookups.
`IdentityHashMap` is often associated with reference semantics for edge cases,
not as a necessary tool for transformation pipelines.
