## Context

You need to generate two parallel arity-scaled functional interface families (e.g.
Consumer2..7 and Predicate2..7) that are structurally identical except for: class
name prefix, SAM method name, and return type. The naive approach is two separate
template files with duplicated `@Permute` + `@PermuteTypeParam` + `@PermuteParam`
annotations.

## Technique

Use `@PermuteVar` to cross-product a string-set axis with the integer arity axis.
Drive method name and return type via JEXL ternary macros on `@Permute`:

```java
@Permute(varName = "i", from = "2", to = "7", className = "${F}${i}",
         inline = false, keepTemplate = false,
         macros = {"method=${F == 'Consumer' ? 'accept' : 'test'}",
                   "ret=${F == 'Consumer' ? 'void' : 'boolean'}"})
@PermuteVar(varName = "F", values = {"Consumer", "Predicate"})
public interface FunctionalTemplate1<A> {

    @PermuteTypeParam(varName = "j", from = "1", to = "${i-1}", name = "${alpha(j)}")
    @PermuteDeclr(type = "${ret}", name = "${method}")
    @PermuteParam(varName = "j", from = "1", to = "${i-1}",
                  type = "${alpha(j)}", name = "${lower(j)}")
    void accept(A a);
}
```

(Consumer2..Consumer7, Predicate2..Predicate7) from a single template file.

`@PermuteDeclr(type="${ret}", name="${method}")` on the method renames both the return
type and method name per combination. The macros resolve to `"void"/"accept"` for
Consumer and `"boolean"/"test"` for Predicate.

## Why this is non-obvious

Most developers would make two separate template files â€” one per family. The `@PermuteVar`
cross-product is not an obvious fit for "two structurally similar but differently-named
families." The JEXL ternary in `macros=` to select between two literals (not produce a
computed value) is a pattern outside normal macro usage.

Note: if the two families need `@FunctionalInterface`, add
`@PermuteAnnotation(type="FunctionalInterface")` to the template â€” but ensure
`PermuteAnnotationTransformer` runs in your pipeline (non-inline pipeline required a
fix for this; see GE-20260421-e86212).

## Constraints

- Requires `@PermuteDeclr` support for methods (method rename by name+type) â€” present in Permuplate since early versions
- `@PermuteVar` cross-product with integer `@Permute` loop: both axes must use different `varName` values
- Arity-1 interfaces that serve as "kept templates" need separate hand-written files (one per family) since `keepTemplate=true` only keeps one class name
