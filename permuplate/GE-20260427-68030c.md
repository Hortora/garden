## Context

You want to add a no-ctx variant of an existing overloaded method family, but the new overload has the same Java type erasure as an existing one:

- Existing ctx-first: `fn(Consumer2<DS, A>)` — erases to `fn(Consumer2)`
- Desired ctx-free: `fn(Consumer2<A, DS>)` — also erases to `fn(Consumer2)` → **compile error**

The same applies to predicate overloads: `filter(Predicate2<DS, B>)` vs `filter(Predicate2<A, B>)`.

## Technique

Generate an entirely new family of functional interfaces with a distinct class name. Different class name = different erasure = no conflict, and both overloads can coexist.

```java
// New no-ctx functional interfaces — generated by FunctionalTemplate2
@FunctionalInterface interface NCtxConsumer1<A>    { void accept(A a); }
@FunctionalInterface interface NCtxConsumer2<A, B> { void accept(A a, B b); }
// ... NCtxConsumer3..6

@FunctionalInterface interface NCtxPredicate1<A>    { boolean test(A a); }
@FunctionalInterface interface NCtxPredicate2<A, B> { boolean test(A a, B b); }
// ... NCtxPredicate3..6
```

```java
// In JoinBuilder — both overloads coexist without conflict:
public RuleResult<DS> fn(Consumer2<DS, A> action) { ... }       // ctx-first, erases to fn(Consumer2)
public RuleResult<DS> fn(NCtxConsumer1<A>  action) { ... }       // no-ctx,    erases to fn(NCtxConsumer1)
```

The calling code naturally resolves by lambda parameter count:
- `fn((ctx, a) -> ...)` → 2-param lambda → `Consumer2<DS, A>` ✓
- `fn(a -> ...)`        → 1-param lambda → `NCtxConsumer1<A>` ✓

## Why this is non-obvious

The intuitive approach is to change the type argument order (`Consumer2<A, DS>` instead of `Consumer2<DS, A>`) or add a wrapper type. Both fail: argument-order swapping still erases to the same raw type, and wrapper types add indirection without solving erasure. The key insight is that Java erasure is per-*class*, not per-*type-argument* — so the only way to distinguish two overloads with the same arity is to use distinct interface classes.

In a generated code context (Permuplate), this is easily done by adding a second template that produces the `NCtx*` family alongside the existing one, with zero runtime overhead since both are `@FunctionalInterface`.

## Constraints

Callers must import the new interface or use explicit casts when the lambda is ambiguous (e.g. at the arity where two different interfaces happen to have the same parameter count). In that case, suppress one overload at the conflicting arity via `@PermuteFilter` on a single-clone `@PermuteMethod`.
