## Symptom

When implementing `@PermuteSource` template composition in Permuplate, the derived class ends up with doubled type parameters: `<A, B, B>` instead of `<A, B, C>`, or `<A, B>` instead of `<A, B>` (correct) but with a phantom extra param.

## Context

Permuplate's InlineGenerator applies a pipeline of transformers to each generated class. `applySourceTypeParams()` reads type parameters from the source class (already in `parentCu`) and copies them to the derived class. `PermuteTypeParamTransformer.transform()` handles implicit expansion of `@PermuteTypeParam` annotations.

## What was tried

- Checked the source class's type params — they were correct `[A, B, C]`
- Checked the derived template's `@PermuteTypeParam` annotations — none present (by design, `@PermuteSource` was supposed to eliminate the need)
- Added logging to `applySourceTypeParams` — it was copying the right params
- Conclusion was that `PermuteTypeParamTransformer` was buggy

## Root cause

`applySourceTypeParams()` was called BEFORE `PermuteTypeParamTransformer.transform()`. When TypeParamTransformer ran after the source params were already set, it saw the injected params `[A, B]` and treated them as sentinels for implicit expansion — detecting them as unreferenced type params matching the `T${j}` naming convention. It then doubled them: `[A, B]` → `[A, B, B]`.

## Fix

Run `PermuteTypeParamTransformer.transform()` FIRST in the pipeline, then call `applySourceTypeParams()` after it. TypeParamTransformer has no sentinels to act on (the template has no `@PermuteTypeParam`), so it's a no-op. Then source inference populates the type params correctly.

```java
// WRONG order:
applySourceTypeParams(generated, templateClassDecl, parentCu, ctx);
PermuteTypeParamTransformer.transform(generated, ctx, messager, element);

// CORRECT order:
PermuteTypeParamTransformer.transform(generated, ctx, messager, element);
applySourceTypeParams(generated, templateClassDecl, parentCu, ctx);
```

## Why this is non-obvious

The symptom (doubled type params) pointed directly at `applySourceTypeParams` or the source class — both were working correctly. The actual cause was a transformer order issue two steps away. The implicit expansion logic in `PermuteTypeParamTransformer` is normally harmless because templates with `@PermuteTypeParam` never have pre-set type params. With `@PermuteSource` injecting params before the transformer runs, a previously-safe code path became destructive.
