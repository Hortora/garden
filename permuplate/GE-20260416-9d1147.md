## The technique

When implementing template composition in Permuplate (`@PermuteSource` — where Template B derives from Template A's output), the key insight is that PermuteMojo **already** chains `generate()` calls sequentially within a file: Template A's output `CompilationUnit` becomes the `parentCu` input for Template B. No new file I/O or separate ordering infrastructure is needed.

## How it works

```
PermuteMojo processing order for one parent file:
1. Generate Template A → output CU₁ (contains Callable2, Callable3, ...)
2. Generate Template B using CU₁ as parentCu → output CU₂
   └── InlineGenerator reads Callable3 from CU₁ directly in memory
```

Template B's `@PermuteSource("Callable${i}")` evaluates to `"Callable3"` for `i=3`, then finds that class in `parentCu` (which IS CU₁). The generated class Callable3 is already there — no file system reads needed.

## Implementation

```java
// In InlineGenerator.applySourceTypeParams():
String sourceName = ctx.evaluate(patterns.get(0)); // → "Callable3"

// parentCu already contains Callable3 from Template A's generate() call
Optional<TypeDeclaration<?>> sourceType =
    parentCu.findFirst(ClassOrInterfaceDeclaration.class,
        c -> c.getNameAsString().equals(sourceName))
    .<TypeDeclaration<?>>map(c -> c)
    .or(() -> parentCu.findFirst(RecordDeclaration.class,
        r -> r.getNameAsString().equals(sourceName)));
// → Callable3 found immediately, in-memory
```

The only new infrastructure needed:
1. `readSourceNamePatterns()` — reads `@PermuteSource` value from template AST
2. `sortBySourceDependency()` in PermuteMojo — puts source templates first in the chain

## Why this is non-obvious

Template composition sounds like it requires separate "phases" — generate all sources first, write them to disk, then process derived templates. In reality, PermuteMojo's sequential chaining (Template A's CompilationUnit is directly passed as Template B's parentCu) means the generated classes are already in-memory when the derived template runs. The chain IS the ordering mechanism. This makes the implementation dramatically simpler than the design appears to require.
