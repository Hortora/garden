### Symptom
`@PermuteReturn(when="false")` on a template method has no effect — the method still appears in all generated classes. Equivalently, `when="i == 2"` doesn't suppress the method for i≠2.

### Context
A Permuplate template method with a concrete declared return type (not the `Object` sentinel) is annotated with `@PermuteReturn(when="false")` to suppress it from generated classes while keeping it in the kept template (keepTemplate=true):

```java
@PermuteReturn(className = "void", when = "false")
public Join2First<END, CTX, B, C> path5(Function2<...> fn2, ...) { ... }
```

### Root cause
`InlineGenerator.applyPermuteReturn()` checks whether the method's declared return type is the `Object` sentinel before processing `@PermuteReturn`. If the declared return type is anything other than `Object` (e.g. `Join2First<...>`), the method is **skipped entirely** — the `when` condition is never evaluated, and the method remains in all generated classes.

### Fix
Two options:

**Option A** — Use a method with `Object` declared return type so `applyPermuteReturn` processes it:
```java
@PermuteReturn(when = "i == 2")  // now evaluated, suppresses for i≠2
public Object path5(Function2<...> fn2, ...) { return ...; }
```

**Option B** — For methods that must exist only on a specific arity: change `when="false"` to `when="i == N"` where N is the desired arity. This requires declaring the method with `Object` return type AND ensuring `applyPermuteReturn` can process it (no `className` needed if the correct return type is declared in the template and kept verbatim via keepTemplate=true).

### Why this is non-obvious
The `@PermuteReturn` documentation says `when` controls whether the method is emitted. There is no indication that `when` is only honoured for methods with `Object` return type. The method silently stays in all generated classes with no error or warning. The bug manifests as unexpected duplicate method definitions or erasure conflicts that appear unrelated to the `when` condition.
