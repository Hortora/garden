### Symptom

`mvn compile` (or `javac`) fails with:

```
error: the switch expression does not cover all possible input values
```

in files you did not touch — immediately after adding a new type to a sealed interface's `permits` clause in a *different* file.

### Context

Occurs when an implementation plan splits "create new sealed subtype" (Task N) from "wire it into existing switch expressions" (Task N+1). The moment Task N lands, every `switch` over the sealed interface across the whole codebase becomes non-exhaustive.

In practice: added `BlinkIntent` to `Intent`'s permits clause in one commit; `EmulatedGame.applyIntent()` and `ActionTranslator` — neither touched by that commit — immediately failed to compile.

### Root cause

Java sealed interface exhaustiveness is a **compile-time constraint across the entire codebase**. Adding a type to a `permits` clause makes every existing `switch` over that interface non-exhaustive, regardless of which file or module the switch lives in. There is no grace period or runtime fallback.

### Fix

In the **same commit** that adds a type to a `permits` clause, add placeholder cases to all existing switch expressions over that interface:

```java
// In every switch over the sealed interface:
case BlinkIntent b -> {} // placeholder — implement in follow-up task
```

Find all switches to update:
```bash
grep -rn "switch (intent)" src/   # or whatever the variable name is
```

### Why this is non-obvious

Splitting "create sealed type" from "wire it in" is natural plan decomposition — they are genuinely separate concerns logically. But Java's exhaustiveness check is a *compile-time* constraint, not a runtime one. The build breaks the moment the new type appears in `permits`, not when the downstream code fails to handle it at runtime. Developers accustomed to non-sealed interfaces (which have no exhaustiveness requirement) are caught off guard.

### Constraints

Applies only to Java sealed interfaces (introduced in Java 17) with `switch` expressions or statements that must be exhaustive. Non-sealed interfaces and `instanceof` checks are unaffected.

### Alternatives considered

- **Merge both tasks into one commit** — eliminates the build gap but loses the clean separation of concerns and makes commits harder to review.
- **Use a default case** — `default -> throw new UnsupportedOperationException()` keeps the build green but loses the compile-time safety guarantee that is the whole point of sealed interfaces.

### Invalidation triggers

Would change if Java adds a "partial exhaustiveness" mode or a compiler flag to defer sealed interface checking. Unlikely — exhaustiveness is a core design goal of sealed interfaces.
