### Symptom

Writing a new Java class to a file path that already exists silently replaces the existing class. Compilation then fails with errors in unrelated files that depended on the original class's API (e.g. type parameter mismatch, missing methods).

### Context

A project has an existing `BetaMemory<C>` class (vol1-era, takes a type parameter). A new `BetaMemory` class is created without the type parameter. The file write overwrites the original. `AccumulateNode` which uses `BetaMemory<?>` then fails to compile with a confusing "type does not take parameters" error.

### What was tried

Created `BetaMemory.java` via shell redirect. No warning was issued. Only noticed when AccumulateNode failed to compile with "type does not take parameters".

### Root cause

File creation has no guard against overwriting. The error manifests in a completely different class (AccumulateNode), not at the point of the overwrite.

### Fix

Before creating any new class, verify the name doesn't already exist in the package:

```bash
find . -name "BetaMemory.java" | grep -v target/
```

Or check with the IDE's new class dialog which will warn about duplicates. Restore the original from git if already overwritten:

```bash
git checkout droolsvol2/src/main/java/org/drools/core/BetaMemory.java
```

### Why this is non-obvious

The compilation error appears in a file that imports the class, not at the site of the overwrite. "type does not take parameters" gives no indication that a file was replaced.
