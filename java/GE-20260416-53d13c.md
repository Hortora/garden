### Symptom
Existing combat tests fail after adding stop-to-fight behaviour to an enemy movement loop.
Assertions on a specific unit (e.g. probe-0) taking damage fail — shields are unchanged
even though an enemy is within attack range.

### Context
A physics simulation where enemies move toward a target each tick, then combat resolves.
Tests place an enemy at a fixed position and assert that a named unit takes damage.

### What was tried (didn't work)
Tests were passing before the stop-to-fight change. No combat logic was modified —
only the movement phase. The failure looked like a combat bug.

### Root cause
Before stop-to-fight: enemy moves each tick, so its resting position when combat resolves
determines the nearest target. Tests were implicitly relying on the enemy having moved
to a position where the intended unit was nearest.

After stop-to-fight: enemy stands still at its spawn position. The nearest target is
determined by distance from the spawn point, not the post-movement position. In the
failing tests, the enemy was spawned at (9.3, 9) — probe-1 at (9.5, 9) is 0.2 tiles
away while probe-0 at (9.0, 9) is 0.3 tiles away. Combat fires on probe-1, not probe-0.

The tests never explicitly tested "probe-0 is the nearest target" — they relied on
movement placing the enemy closer to probe-0 post-tick.

### Fix
Spawn the enemy at a position where the intended target is unambiguously nearest and
within range, regardless of movement phase. For probe-0 at (9.0, 9.0):

```
(9.0, 9.3)  →  probe-0 distance 0.3 (in 0.5 melee range)
              probe-1 distance 0.58 (outside 0.5 melee range)
```

Use (9.0, 9.3) instead of (9.3, 9.0) — directly above the target, not between two targets.

### Why this is non-obvious
The failure looks like a combat regression, not a test positioning problem.
Movement-phase changes have no obvious connection to which unit takes damage in the
combat phase. The implicit dependency (movement determines nearest target) is invisible
until the movement behaviour changes.
