## Symptom

When using `@PermuteDeclr` (or any annotation) in TYPE_USE position on a qualified constructor type like:

```java
new @Ann com.example.Outer.Inner<>()
```

`newExpr.getType().getAnnotations()` returns an empty list even though the annotation is clearly present in source.

## Context

Occurs with any JavaParser `ObjectCreationExpr` where the constructor type is a qualified name (`A.B`, `Outer.Inner`, `JoinBuilder.Join1First`). Simple unqualified names (`new @Ann Foo<>()`) work correctly.

## What was tried

- Checked `newExpr.getType().getAnnotations()` â€” empty
- Walked the AST looking for `AnnotationExpr` nodes â€” found them, but couldn't correlate back to the expression
- Moved the processing call to different pipeline stages â€” no change

## Root cause

JavaParser parses `new @Ann A.B<>()` by placing `@Ann` on the **scope type** (`A`), not on the full type (`A.B`). The `ClassOrInterfaceType` returned by `newExpr.getType()` has a `scope` field (another `ClassOrInterfaceType` for `A`) that carries the annotation. `getAnnotations()` on the leaf type is empty; `getScope().get().getAnnotations()` contains `@Ann`.

## Fix

Check both the type's own annotations and the scope's annotations:

```java
ClassOrInterfaceType type = newExpr.getType();

AnnotationExpr ann;
if (hasPermuteDeclr(type.getAnnotations())) {
    ann = getPermuteDeclr(type.getAnnotations());
} else if (type.getScope().filter(s -> hasPermuteDeclr(s.getAnnotations())).isPresent()) {
    // Qualified name: annotation lives on scope, not leaf
    ClassOrInterfaceType scope = type.getScope().get();
    ann = getPermuteDeclr(scope.getAnnotations());
    scope.getAnnotations().remove(ann); // clean up scope before replacing full type
} else {
    return;
}

// evaluate ann.type, then:
newExpr.setType(newParsedType); // replaces the whole node, annotation gone
```

## Why this is non-obvious

The annotation is *syntactically* before the full type `A.B`, so it seems like it should be on `A.B`. The Java language spec places type annotations on the rightmost component of a qualified name, but JavaParser's internal representation puts it on the leftmost (scope) component for `new` expressions. The distinction only emerges when the constructor type has a qualifier â€” single-segment names always put the annotation on the type itself.
