## Symptom

Code like this compiles without error or warning, which looks like it should
fail due to generic type invariance:

```java
Map<String, Integer> enrichStats = enricher.enrich(...);
stateStore.markEnriched(slug, new java.util.HashMap<>(enrichStats));
// markEnriched(String, Map<String, Object>) — Integer is not Object??
```

## Context

Passing the result of one method (returning `Map<String, Integer>`) as an
argument to a method expecting `Map<String, Object>`. `Map<String, Integer>`
is NOT a subtype of `Map<String, Object>` under Java generics invariance —
which makes the compilation look impossible.

## What was tried

Expecting a compile error; attempting explicit casts; assuming the code was wrong.

## Root cause

Java's diamond operator `<>` uses **target-type inference** — when the
constructor call appears in a position with a known target type (here, the
method parameter `Map<String, Object>`), the compiler resolves the type
parameters from that target. So `new HashMap<>(enrichStats)` is inferred as
`new HashMap<String, Object>(enrichStats)`.

The copy constructor `HashMap(Map<? extends K, ? extends V>)` accepts
`Map<String, Integer>` as input for a `HashMap<String, Object>` because
`Integer extends Object`. The widening is applied at construction, not via
assignment. This is well-specified behaviour but genuinely surprising.

## Fix

The code is correct as-is. Add a comment to prevent future confusion:

```java
// Diamond infers HashMap<String, Object> from the target parameter type;
// copy constructor accepts Map<? extends K, ? extends V>, widening Integer → Object
stateStore.markEnriched(slug, new java.util.HashMap<>(enrichStats));
```

If you want to be explicit:

```java
Map<String, Object> statsAsObjects = new HashMap<>(enrichStats);
stateStore.markEnriched(slug, statsAsObjects);
```

Both compile identically. The comment form documents why the implicit version works.

## Why this is non-obvious

Generic invariance (`Map<Integer>` is not a `Map<Object>`) is a well-known Java
rule. The exception — target-type inference via diamond applying widening through
a copy constructor — is not. Most developers would write an explicit cast or
change the method signature rather than trusting that the code compiles correctly.
