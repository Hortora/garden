## Context

OOPath-style DSL with a private internal traversal function `fn2: Function2<Ctx, A, Iterable<B>>`. Need to add support for array-returning (`B[]`) and single-value-returning (`B`) traversal methods without modifying the runtime traversal loop.

## Technique

New public methods (`pathArray`, `pathSingle`) wrap their input function at the DSL boundary before delegating to the private core. The runtime never sees the difference:

```java
// Array — wrap B[] to Iterable<B> at the boundary
public END pathArray(Function1<A, B[]> fn) {
    return path((ctx, a) -> Arrays.asList(fn.apply(a)), (ctx, b) -> true);
}

// Single — wrap B to Iterable<B> with null guard at the boundary
public END pathSingle(Function1<A, B> fn) {
    return path((ctx, a) -> {
        B v = fn.apply(a);
        return v != null ? List.of(v) : List.of();
    }, (ctx, b) -> true);
}

// Private core — unchanged
private END path(Function2<Ctx, A, Iterable<B>> fn2, Predicate2<Ctx, B> flt2) { ... }
```

Null safety for `pathSingle` is automatic: null return → empty Iterable → path branch is pruned with no special runtime handling.

## Why This Is Non-Obvious

The natural instinct is to modify the runtime traversal loop (e.g., `ListPathNode.primeNext()`) to handle each new type via `instanceof` or an `AccessType` enum dispatch. The wrapping approach keeps all variant logic at the DSL boundary, requires zero runtime changes, and keeps the internal model clean. Method references to array-returning or single-value methods work directly since the wrapping is transparent.
