## Symptom

Adding a new value to a Java enum causes a compile error in a seemingly unrelated file: `the switch expression does not cover all possible input values`. The error appears in a file not considered during change planning.

## Context

Java switch *expressions* (arrow syntax, `->`) without a `default` arm are exhaustiveness-checked: every enum value must be covered. A pre-change analysis that checks "all our switches have `default` cases" will miss switch expressions that achieve exhaustiveness by listing all enum values explicitly — which is the preferred pattern for compile-time safety.

When a new enum value is added, the error lands in the exhaustive-switch file, not near the enum definition.

## What was tried (didn't work)

- Grepping for `switch.*EnumType` and spot-checking files for `default` — found switches that had `default` but missed the exhaustive switch that deliberately omits it

## Root cause

The compiler enforces enum exhaustiveness on switch expressions. A switch covering all N original values is valid for N values; adding value N+1 breaks it. The fix is either adding the new value to the correct arm or adding a `default`.

## Fix

When adding enum values, search for ALL switch expressions (not just switch statements) on that enum — including those without `default`:

```bash
# Find all switch expressions on UnitType — including exhaustive ones
grep -rn "case PROBE\|case STALKER\|case MARINE" src/ --include="*.java"
# Review each hit: does the switch have a default? If not, it will break.
```

Add the new value to the appropriate existing arm:

```java
// Before (exhaustive, covers all original values):
return switch (type) {
    case PROBE, ZEALOT -> mapProtossAbility(type);
    case MARINE        -> null;  // enemy type
};

// After adding SIEGE_TANK — add to the null arm:
return switch (type) {
    case PROBE, ZEALOT          -> mapProtossAbility(type);
    case MARINE, SIEGE_TANK     -> null;
};
```

## Why this is non-obvious

Exhaustive switch expressions (without `default`) are considered best practice in Java — they give stronger compile-time safety than switches with a `default`. A developer checking "all our switches have defaults" will correctly find the non-exhaustive switches but will not look at the exhaustive ones — which are syntactically different and require the same attention when the enum grows.
