## Symptom

A Maven project has two declarations for the same `groupId:artifactId` in its POM — one
with `scope=compile` (or no scope, which defaults to compile) and another with `scope=test`.
Compilation fails with "package does not exist" even though the dependency appears to be
declared in compile scope.

Maven emits only a WARNING (buried in output, easy to miss):

```
[WARNING] 'dependencies.dependency.(groupId:artifactId:type:classifier)' must be unique:
dev.tamboui:tamboui-tui:jar -> duplicate declaration of version 0.2.0-SNAPSHOT
```

## Root cause

When the same `groupId:artifactId:type:classifier` coordinate appears twice in a POM, Maven
retains **one** entry. In practice the test-scoped entry wins, overriding the compile-scoped
one. The artifact is then on the test classpath only, making it unavailable during main-source
compilation.

## Fix

Remove the duplicate. Keep only the compile-scoped entry; the test classpath inherits from it:

```xml
<!-- WRONG: two declarations for the same artifact -->
<dependency>
  <groupId>com.example</groupId>
  <artifactId>foo</artifactId>
  <version>1.0</version>
  <!-- compile scope (default) -->
</dependency>
<dependency>
  <groupId>com.example</groupId>
  <artifactId>foo</artifactId>
  <version>1.0</version>
  <scope>test</scope>   <!-- this one wins and breaks compilation -->
</dependency>

<!-- CORRECT: single compile-scope entry -->
<dependency>
  <groupId>com.example</groupId>
  <artifactId>foo</artifactId>
  <version>1.0</version>
  <!-- compile scope — available to both main and test -->
</dependency>
```

If you genuinely need the dependency only in test scope, remove the compile-scope entry and
live with test-only availability.

## Why this is non-obvious

The compile error points to the package not existing, not to a duplicate declaration.
The WARNING mentioning the duplicate is buried in Maven's verbose output and does not
indicate which scope wins. A developer adding a compile dependency to fix a "package does
not exist" error — when a test-scoped declaration for the same artifact already exists —
will find the error persists despite the apparent fix.
