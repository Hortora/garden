### Symptom
`POST /api/endpoint?flag=1` silently behaves as if `flag=false`. No error, no warning, no 400 response. `?flag=true` works correctly.

### Context
Migrating a Python HTTP server endpoint to JAX-RS. The Python server uses the convention `params.get('flag') == '1'`. The JAX-RS endpoint uses `@QueryParam("flag") @DefaultValue("false") boolean flag`.

### Root cause
JAX-RS boolean query parameter parsing only accepts the string literals `"true"` and `"false"` (case-insensitive). Any other value — `"1"`, `"yes"`, `"on"` — is silently treated as `false`. There is no warning, no 400, no log message. The JAX-RS spec does not define parsing for numeric booleans.

### Fix
Receive the parameter as `String` and parse manually to accept both conventions:

```java
@POST
@Path("{slug}/generate-md")
public Response generateMd(
    @PathParam("slug") String slug,
    @QueryParam("dry") @DefaultValue("") String dryParam
) {
    // Accept ?dry=1 (Python/Unix convention) and ?dry=true (JAX-RS convention)
    boolean dry = "1".equals(dryParam) || "true".equalsIgnoreCase(dryParam);
    return BridgeResponse.of(bridge.call("bridge.post_generate_md", slug, dry));
}
```

### Why this is non-obvious
`@DefaultValue("false") boolean flag` looks like it should "just work" — the annotation spells out the valid values. Passing `?flag=1` looks like it should be truthy by any reasonable definition. The silent `false` with no error response makes this look like a logic bug in the handler, not a parameter parsing issue.

This is particularly invisible when the endpoint has a default of `false` anyway — the "wrong" behaviour (treating `1` as false) is identical to the default, so the parameter appears to be silently ignored rather than misunderstood.
