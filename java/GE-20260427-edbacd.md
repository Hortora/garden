## Symptom

A `@QuarkusTest` fails with nonsensical results — state from test N appears in test N+1 even though `@BeforeEach` resets the game/domain object. In this case, a Hatchery from `creepRendersAroundHatchery` persisted into `noCreepForNonZergBuilding`, generating 317 creep tiles (π×10²≈314, a radius-10 circle) for a Barracks that should produce zero.

## Context

A `SimulatedGame` CDI bean (singleton) had two new collections added: `enemyBuildings` and `mineralPatches`. The class had two clearing methods:

- `clearAll()` — called by the replay subclass's `reset()`
- `reset()` — called directly by `@BeforeEach` via `orchestrator.startGame()`

Both new collections were added to `clearAll()` but NOT to `reset()`.

## What was tried

- Checked CDI scope (`@ApplicationScoped` singleton — correct)
- Verified `@BeforeEach` fires before each test — correct
- Added logging to confirm `reset()` was called — confirmed
- Spent time tracing why a Barracks generates creep (it doesn't)

## Root cause

`SimulatedGame.reset()` and `clearAll()` are two separate methods. `reset()` explicitly lists each collection to clear (not delegating to `clearAll()`). When a new collection is added, it must be added to BOTH methods. Forgetting one leaves stale state visible to subsequent tests.

```java
// BAD — clearAll() has enemyBuildings.clear() but reset() doesn't
public synchronized void reset() {
    myUnits.clear();
    myBuildings.clear();
    enemyUnits.clear();
    // enemyBuildings.clear() ← missing!
    geysers.clear();
}

protected void clearAll() {
    myUnits.clear();
    myBuildings.clear();
    enemyUnits.clear();
    enemyBuildings.clear();  // ← present here, but not in reset()
    geysers.clear();
}
```

## Fix

Add every new collection to BOTH clearing methods:

```java
public synchronized void reset() {
    myUnits.clear();
    myBuildings.clear();
    enemyUnits.clear();
    enemyBuildings.clear();  // ← added
    geysers.clear();
    mineralPatches.clear();  // ← added
}
```

## Why this is non-obvious

The symptom — "a Barracks generates creep" — has nothing to do with the root cause. The failure looks like a logic error in the creep generation code, not a state management bug. The 317-tile count (π×10²≈314) is a specific signal: a full radius-10 circle appeared, meaning a Hatchery from a PREVIOUS test was still in the collection. The `@BeforeEach` reset appeared to be working (it was — just incompletely).
