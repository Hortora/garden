## Maven incremental build passes but `NoClassDefFoundError` at runtime ‚Äî stale `.class` files

**ID:** GE-0144
**Stack:** Maven (all versions)
**Symptom:** `NoClassDefFoundError: org/example/MyClass` at test runtime, even though `mvn compile` passed with no errors. The class clearly exists in source.
**Context:** After changing a class (e.g. adding a method override), Maven's incremental build recompiles that source file successfully. A dependent class has a cached `.class` from before the change that becomes structurally inconsistent.

### What was tried (didn't work)
- Checked pom.xml for missing dependencies ‚Äî nothing missing
- Checked classpath for duplicate jars ‚Äî none
- Re-ran `mvn compile` ‚Äî passed clean, error persisted

### Root cause
Maven's incremental compilation only recompiles files whose source timestamps have changed. If a cached `.class` becomes structurally inconsistent with a newly compiled dependency (e.g. a method signature changed, a new abstract method was added), Maven doesn't detect or flag it. The old `.class` remains on disk and is loaded at runtime.

### Fix
```bash
mvn clean test
```

Forces full recompilation. Use whenever tests throw `NoClassDefFoundError`, `ClassCastException`, or `AbstractMethodError` on classes that clearly exist in source.

### Why non-obvious
`NoClassDefFoundError` is universally diagnosed as a classpath/dependency problem ‚Äî missing jar, wrong scope, shading conflict. The symptom gives no indication that the class exists on disk but is stale. Developers inspect `pom.xml` and dependency trees while the fix is `mvn clean`.

## Variant ‚Äî Deleted source file leaves orphaned `.class`, throws `NoSuchMethodError`

**Symptom:** `NoSuchMethodError` in a test for a class that no longer exists in the source tree. The class name appears in the test report but there is no corresponding `.java` file anywhere under `src/`.

**Context:** A `.java` source file was deleted (e.g. a debug test class removed as part of cleanup), but `mvn clean` was not run. The compiled `.class` survives in `target/test-classes/` and surefire discovers and runs it. The test body calls a production method that has since been removed or renamed, producing `NoSuchMethodError`.

**Why non-obvious:** The error looks like a dependency incompatibility. Searching the source tree for the class name returns nothing ‚Äî the class appears to not exist ‚Äî yet the test runner is somehow executing it. The disconnect between "class not in source" and "class being executed" is deeply confusing.

**Fix:** same ‚Äî `mvn clean test` removes the orphaned `.class` along with all other stale compiled output.

## Variant ‚Äî IntelliJ move-file refactor leaves stale `.class` for old package location

**Symptom:** `NoClassDefFoundError` during Maven test discovery after moving a Java class to a new package using IntelliJ's IDE move-file refactor. Compilation (`mvn compile`) passes cleanly ‚Äî no error or warning.

**Context:** IntelliJ's move refactor correctly relocates the source file, updates the package declaration, and updates all import sites. It does NOT delete the compiled `.class` for the old package from `target/`. Both the old (stale, broken) and new (correct) `.class` coexist on the classpath until `mvn clean` is run.

**Why non-obvious:** Compilation succeeds with zero errors, so the stale file is completely invisible until test discovery fails. The error appears to be a classpath problem, but the classpath is correct ‚Äî the issue is a ghost `.class` from the pre-move location that the IDE's refactor didn't clean up.

**Fix:** same ‚Äî `mvn clean` before the test run.
