## Symptom

`-Dquarkus.profile=replay` (or any `-D` flag) has no effect when the app starts from a jar. The system property is absent at runtime.

## Context

Running a Quarkus fat jar from an Electron main process:

```javascript
spawn('java', ['-jar', JAR, '-Dquarkus.profile=replay'])
```

The profile is silently ignored — the app boots in default profile.

## Root cause

The `java` command syntax is `java [JVM-options] -jar <jar> [app-args]`. Everything after the jar path is passed to the application's `main(String[] args)`, not to the JVM. `-Dquarkus.profile=replay` placed after the jar becomes `args[0]`, not a system property.

## Fix

Place `-D` flags before `-jar`:

```javascript
// Wrong
spawn('java', ['-jar', JAR, '-Dquarkus.profile=replay'])

// Correct
spawn('java', ['-Dquarkus.profile=replay', '-jar', JAR])
```

## Why this is non-obvious

No error or warning — the flag is silently treated as an application argument. The JVM starts successfully; the application launches; only runtime behaviour reveals the property is absent. The ordering rule is documented but easy to forget when building argument arrays programmatically.
