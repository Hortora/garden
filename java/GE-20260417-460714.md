## Symptom

Running `mvn compile -q 2>&1 | grep "error:"` produces no output and exit code 0 — even when there are real compilation errors. A subsequent `mvn test` then fails with the same errors.

## Context

Using `-q` (quiet) in Maven to suppress verbose output during scripted builds or in subagent prompts where you want clean output. You check for errors by grepping the output and see nothing — so you conclude compilation succeeded.

## Root cause

The `-q` flag suppresses `[INFO]` lines but in some Maven/compiler-plugin version combinations also suppresses `[ERROR]` compiler output to stdout. The compilation still fails internally, but the error lines don't reach the grep. Exit code handling is also inconsistent — the overall build may return 0 if the Surefire plugin is not involved.

## Fix

Never use `-q` when the purpose is to check for compile errors. Use:

```bash
# Instead of: mvn compile -q 2>&1 | grep "error:"
mvn compile 2>&1 | grep "error:"

# Or to get a clean exit code only (not for error inspection):
mvn compile -q; echo "EXIT: $?"
```

For scripted checks, capture full output and grep it:

```bash
OUTPUT=$(mvn compile 2>&1)
echo "$OUTPUT" | grep "error:" | head -20
```

## Why this is non-obvious

`-q` is commonly used to reduce noise in CI and scripted contexts. The assumption is that errors still surface — quiet mode means "less output, same correctness signals". That assumption is wrong for compiler errors in some configurations.
