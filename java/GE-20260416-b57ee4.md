## Behaviour

`NodeList<T>.clone()` in JavaParser returns `Object`, not `NodeList<T>`. Calling `.clone()` on a typed `NodeList` compiles only if you cast the result, and the unchecked cast triggers a warning.

```java
NodeList<TypeParameter> srcTps = sourceClass.getTypeParameters();

// FAILS to compile or requires explicit cast:
NodeList<TypeParameter> copy = srcTps.clone(); // error: incompatible types

// Compiles but needs @SuppressWarnings:
@SuppressWarnings("unchecked")
NodeList<TypeParameter> copy = (NodeList<TypeParameter>) srcTps.clone();
```

## Root cause

`NodeList<T>` extends `java.lang.Object` and overrides `clone()` returning `Object` due to Java's covariant return type rules and how JavaParser implemented the cloning. The generic type is erased at the clone boundary.

## Workaround

Build a fresh `NodeList` by cloning each element individually:

```java
NodeList<TypeParameter> srcTps = sourceClass.getTypeParameters();

NodeList<TypeParameter> copy = new NodeList<>();
for (TypeParameter tp : srcTps) {
    copy.add(tp.clone());
}
// copy is a fresh NodeList<TypeParameter> — no cast, no warning
```

## Where to find it

Only discoverable by noticing the compile error or inspecting the JavaParser source. The JavaParser docs and README make no mention of this `clone()` limitation. The JavaParser `NodeList` source at `com.github.javaparser.ast.NodeList` shows the raw `clone()` signature.

## Why this is non-obvious

`NodeList<T>` is used everywhere in JavaParser and behaves like a generic collection in all other respects. The `clone()` method being typed `Object` is a surprising exception. Any code that copies NodeLists for AST transformation (e.g. copying type parameters from one class to another) will hit this silently if the author assumed `clone()` returns the typed list.
