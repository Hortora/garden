## Symptom

```
name clash: path(Predicate2<B,PathContext<T>>) and path(Predicate2<PathContext<T>,B>)
have the same erasure
```

Compile error when defining two overloads that take the same generic class instantiated with flipped type arguments.

## Context

Designing a DSL API that needs both ctx-first (`Predicate2<Ctx, B>`) and ctx-at-end (`Predicate2<B, Ctx>`) predicate overloads on the same method name. Each overload is written and compiles separately; the clash only surfaces when both exist on the same class.

## Root Cause

Java erases generic type parameters at compile time. `Predicate2<B, PathContext<T>>` and `Predicate2<PathContext<T>, B>` both erase to the raw type `Predicate2`. Java cannot distinguish them as distinct method overloads — the bytecode signatures are identical.

The same applies to any two instantiations of the same generic class: `Function2<A, B, R>` and `Function2<B, A, R>` have the same erasure `Function2`.

## Fix

Use a **distinct interface class** for each semantic role. Different class names produce different erasures:

```java
// Instead of Predicate2<B, PathContext<T>> (clashes):
@FunctionalInterface
public interface CtxLastPredicate1<A, DS> {
    boolean test(A a, DS ds);
}

// Now path(fn, Predicate2<PathContext<T>, B>) and
// path(fn, CtxLastPredicate1<B, PathContext<T>>) have different erasures — no clash
```

## Why This Is Non-Obvious

Looks like it should work — the generic arguments are visibly different. The failure is invisible while writing each overload individually; the compile error only appears when both exist simultaneously. Developers familiar with generics know about erasure abstractly but don't immediately recognise it as the cause when the type arguments are different orderings of the same types.
