## Context

Testing a class that makes `HttpClient` calls without making real HTTP
requests. The class downloads thumbnails or calls an API as part of its
logic.

## Technique

Extract the raw HTTP calls into **package-private methods** on the class.
In tests, define a `MockXxx extends Xxx` inner class that overrides those
methods. No Mockito, no framework configuration, no generic type gymnastics.

### In the production class (`Enricher.java`)

```java
public class Enricher {
    private final HttpClient http;

    public Enricher() { this.http = HttpClient.newHttpClient(); }

    // Package-private: overrideable in tests
    Enricher(HttpClient http) { this.http = http; }

    byte[] fetchUrl(String url) {
        HttpRequest req = HttpRequest.newBuilder(URI.create(url)).build();
        HttpResponse<byte[]> r = http.send(req, BodyHandlers.ofByteArray());
        return r.statusCode() == 200 && r.body().length > 0 ? r.body() : null;
    }

    String fetchJson(String url, String token) {
        // ... build request with optional auth header ...
        HttpResponse<String> r = http.send(req, BodyHandlers.ofString());
        String body = r.body();
        return r.statusCode() == 200 && !body.isEmpty() ? body : null;
    }

    // All enrichment methods call fetchUrl/fetchJson — no direct HttpClient use
    int replaceYoutubeEmbeds(Element article, Path assetsDir) throws Exception {
        // ...
        byte[] bytes = fetchUrl("https://img.youtube.com/vi/" + id + "/maxresdefault.jpg");
        // ...
    }
}
```

### In the test class (`EnricherTest.java`, same package)

```java
static class MockEnricher extends Enricher {
    final Map<String, byte[]> urlBytes = new HashMap<>();
    final Map<String, String> urlJson  = new HashMap<>();
    int fetchCallCount = 0;

    void mockBytes(String url, byte[] bytes) { urlBytes.put(url, bytes); }
    void mockJson(String url, String json)   { urlJson.put(url, json); }

    @Override byte[] fetchUrl(String url) {
        fetchCallCount++;           // lets you assert cache-hit (count stays 0)
        return urlBytes.get(url);   // returns null if URL not mocked = simulates failure
    }

    @Override String fetchJson(String url, String token) {
        return urlJson.get(url);
    }
}
```

### Test usage

```java
@Test
void youtubeEmbedReplacedWithFigure(@TempDir Path tempDir) throws Exception {
    MockEnricher e = new MockEnricher();
    e.mockBytes("https://img.youtube.com/vi/abc/maxresdefault.jpg", new byte[]{1, 2, 3});
    Path assets = Files.createDirectories(tempDir.resolve("assets"));

    Element a = article("<iframe src=\"https://www.youtube.com/embed/abc\"></iframe>");
    e.replaceYoutubeEmbeds(a, assets);

    assertEquals("yt_abc.jpg", a.selectFirst("img").attr("src"));
}

@Test
void existingThumbnailNotReDownloaded(@TempDir Path tempDir) throws Exception {
    MockEnricher e = new MockEnricher();
    Files.write(tempDir.resolve("yt_cached.jpg"), new byte[]{9});

    e.replaceYoutubeEmbeds(article("<iframe src=\".../embed/cached\"></iframe>"), tempDir);

    assertEquals(0, e.fetchCallCount, "cache hit — no HTTP call made");
}
```

## Why this is non-obvious

Most developers reach for Mockito when they need to stub HTTP calls.
Mocking `HttpClient` requires navigating `HttpResponse<T>` generics,
`BodyHandler` setup, and at least 10 lines of stub configuration. The
subclass approach replaces this with a 10-line inner class and zero
framework imports. Tests run faster (no proxy overhead) and are more
readable (stub values are just map entries).

The key insight: if you control the class, you don't need a mocking
framework — you just need a seam. Package-private methods are the seam.

## Constraints

Requires that tests live in the same package as the class under test
(standard Java convention for unit tests). Does not work for `final`
methods or `final` classes. If the class is used through an interface,
mock the interface instead.

## Alternatives considered

- **Mockito** — works but requires `@ExtendWith(MockitoExtension.class)`,
  `@Mock HttpClient`, and multiple stubbed `HttpResponse<T>` objects.
  Verbose and fragile when `HttpClient` internals change.
- **WireMock** — spins up a real HTTP server. Correct for integration tests;
  overkill for unit tests where you control the production code.
- **Dependency-inject a `Function<String, byte[]>`** — works but changes
  the public API and leaks the test concern into production signatures.
