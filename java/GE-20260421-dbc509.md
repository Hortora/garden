### Symptom
Family-extraction logic using `name.replaceAll("\\d+$", "")` returns
`"Join2First"` unchanged (digits not at the end), while correctly stripping
`"Join1Second"` → `"JoinSecond"`. Any comparison based on the extracted
"family" silently fails to match for embedded-arity names with a letter suffix.

### Context
Code-generation or annotation-processing systems that use numeric suffixes to
encode arity (e.g. `Join1First`, `Join2First`, `Tuple3`, `RuleExtendsPoint4`)
and need to group a family of generated classes by their structural root.

### Root cause
`\\d+$` only matches digit sequences at the very end of the string. A name like
`Join2First` ends in `"First"`, so no digits are matched and the string is
returned unmodified. Stripping ALL digit sequences is required.

### Fix
Use `replaceAll("\\d+", "")` to strip every numeric segment:

```java
// Wrong — only strips trailing digits
String family = name.replaceAll("\\d+$", ""); // "Join2First" → "Join2First" ❌

// Correct — strips all digit sequences
String family = name.replaceAll("\\d+", "");  // "Join2First" → "JoinFirst"         ✓
                                               // "Tuple3"     → "Tuple"             ✓
                                               // "RuleExtendsPoint4" → "RuleExtendsPoint" ✓
```

### Why this is non-obvious
When thinking "strip the arity number from a generated class name", developers
reach for `\\d+$` because they picture the number at the end. Many arity-named
classes do end in a digit (`Tuple3`, `Join4`). The failure only surfaces for the
`<prefix><digit><suffix>` pattern (e.g. `Join2First`), which is a common
convention for hierarchical class families with a semantic suffix.
