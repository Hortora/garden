## The Technique

When production code uses `random.nextDouble() < threshold` as a probability gate, inject a `Random` subclass with a fixed return value to force the always-miss or always-hit path — no mocking framework required.

```java
// Always-miss: nextDouble() returns 0.0 → 0.0 < 0.25 → true → miss every time
game.setRandomForTesting(new Random() {
    @Override public double nextDouble() { return 0.0; }
});

// Always-hit: nextDouble() returns 1.0 → 1.0 < 0.25 → false → hit every time
game.setRandomForTesting(new Random() {
    @Override public double nextDouble() { return 1.0; }
});
```

The production class holds `private Random random = new Random(42L)` and exposes a package-private `setRandomForTesting(Random r)` setter. The test injects the anonymous subclass before the assertion tick.

## Why This Is Non-Obvious

Most developers reach for Mockito (`@Mock Random random; when(random.nextDouble()).thenReturn(0.0)`) or hunt for a seed that produces the desired sequence. Both are heavier. The anonymous subclass approach works because `Random.nextDouble()` is **not** `final` — it can be overridden. The override is minimal, expressive, and introduces no test-framework dependency. It also directly tests the exact boundary condition (`< 0.25`) rather than relying on seed behaviour that might change across JVM versions.

## Applicability

Any code that holds a `Random` field and uses `nextDouble()` (or `nextInt()`, `nextBoolean()`) for probabilistic branching. The only requirement is a way to inject the `Random` instance (constructor, setter, or package-private field).
