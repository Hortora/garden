## In-memory service error messages should use human-readable names not internal UUIDs — TDD test assertions catch the difference

**ID:** GE-20260414-c4ddc9
**Stack:** Java, JUnit 5
**Labels:** `#testing` `#error-messages` `#tdd` `#api-design` `#uuid`

### Symptom

Test asserts that the exception message names the channel:

```java
assertTrue(ex.getMessage().contains("my-channel"),
    "error should name the channel");
```

But the actual message contains the UUID:

```
Rate limit exceeded for channel 'f96dd562-7627-4e63-a41b-b0baba1024b1':
max 2 messages per minute across all senders.
```

### Context

An `@ApplicationScoped` service bean was keyed by UUID (the entity's internal primary key) for its in-memory data structures. Error messages were built from the same UUID. The human-readable name was available at the call site but not threaded through.

### Root cause

The service method signature used only the UUID:

```java
public String check(UUID channelId, String sender, Integer limitPerChannel, ...) {
    // ... error formatted with channelId (UUID), not channelName
    return "Rate limit exceeded for channel '" + channelId + "'...";
}
```

The caller had `channelName` but passed only `channelId`. The service had no access to the name.

### Fix

Thread the human-readable name into the service method alongside the UUID. UUID for map lookups; name for error messages:

```java
public String check(UUID channelId, String channelName, String sender,
        Integer limitPerChannel, Integer limitPerInstance) {
    // ...
    return "Rate limit exceeded for channel '" + channelName + "'...";
}
```

### Why this is non-obvious

Using the entity ID as the lookup key is correct and natural — it's stable, indexed, and collision-free. The mistake is inadvertently using it in the error message too. Manual testing rarely checks the exact wording of error messages; TDD test assertions on `getMessage()` content surface this immediately. Write the assertion on human-readable identifiers first — it forces the right API design.
