### Symptom

Two distinct objects with the same `equals()`/`hashCode()` (e.g. two `RuleImpl("r1")` instances with the same name) map to the same `WeakHashMap` entry, causing state to bleed unexpectedly between them.

### Context

When building a registry that maps object instances to associated state (e.g. build-time metadata keyed per rule object), `WeakHashMap` seems natural for its auto-cleanup. But if the key type overrides `equals()` by name/value, distinct instances share the same slot.

### What was tried

`WeakHashMap<Rule, Object[]>` to store per-rule build state. Two separate `new RuleImpl("r1")` objects in different tests shared the same map entry and corrupted each other's state.

### Root cause

`WeakHashMap` uses `equals()` and `hashCode()` for key comparison, just like `HashMap`. The "weak" only refers to the reference strength (allowing GC), not to using identity comparison.

### Fix

Use `IdentityHashMap` when you need reference-equality (object identity, not logical equality) as the key semantics:

```java
// Wrong — uses equals()/hashCode():
Map<Rule, Object[]> state = new WeakHashMap<>();

// Correct — uses == for key comparison:
Map<Rule, Object[]> state = new IdentityHashMap<>();
```

`IdentityHashMap` uses `System.identityHashCode()` and `==`, making distinct instances always distinct keys regardless of `equals()`.

### Why this is non-obvious

`WeakHashMap` is often chosen for caches that need GC cleanup of unused entries. Developers commonly assume "weak" implies identity-based keying, but the API does not change key comparison semantics.
