## Symptom

A test asserting `assertThat(unit.shields()).isEqualTo(maxShields - SC2Data.damagePerAttack(attackerType))` fails with the actual value being `armour` points higher than expected.

Example: Probe (5 damage) attacks Zealot (1 armour, 50 shields). Expected 45, actual 46.

## Context

Any test in `EmulatedGameTest` (or similar) that asserts exact shield values after one combat tick, where the target unit has nonzero armour from `SC2Data.armour()`.

## Root Cause

`DamageCalculator.computeEffective(attacker, target)` applies armour as a flat reduction to the total damage before it is split between shields and HP. Shields are not armour-exempt.

Effective damage = `max(0, rawDamage - SC2Data.armour(targetType))`

This matches real SC2 behaviour but is not documented in `SC2Data.java`.

## Fix

Use the armour-adjusted value in assertions:

```java
int effective = Math.max(0,
    SC2Data.damagePerAttack(UnitType.PROBE) - SC2Data.armour(UnitType.ZEALOT));
assertThat(zealot.shields()).isEqualTo(zealotMaxShields - effective);
```

## Why This Is Non-Obvious

The natural assumption is that shields absorb hits at face value — armour "feels" like it should only reduce HP damage. Both SC2 and this emulator apply armour uniformly to the total damage regardless of which buffer (shield or HP) absorbs it.
