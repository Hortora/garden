### Symptom
When passing `Map<String, Object>` to a `Function<AmlTransaction, WorkerResult>`
(obtained via raw-type cast to erase the generic), the call to `apply()` succeeds
at the call site. The `ClassCastException` occurs *inside* the lambda body — at
the point where the lambda parameter is first used as the declared type, not where
`apply()` was called.

Stack trace points to the lambda, not the caller. Debugging leads to the lambda
implementation rather than the incorrect call site that passed the wrong type.

### Context
Java generic functional interfaces like `Function<T, R>` have their type parameters
erased at runtime. A common assumption is that raw-type casting
`((Function) sync.fn()).apply(mapInput)` is safe because erasure means the function
receives `Object` at the bytecode level.

This assumption is **wrong** for lambdas. Unlike anonymous inner classes, lambda
expressions compiled by javac generate bridge methods that include a `checkcast`
instruction for the concrete type parameter.

### Root cause
When javac compiles a lambda like:

```java
Function<AmlTransaction, WorkerResult> fn = (txn) -> WorkerResult.of(Map.of("risk", txn.riskScore()));
```

The generated bridge method (visible via `javap -c`) contains:

```
aload_1
checkcast AmlTransaction  // ← bridge inserts this
invokevirtual AmlTransaction.riskScore
```

The `checkcast` fires when `apply()` is called with a `Map` instead of
`AmlTransaction`. The CCE happens at the `checkcast` instruction inside
the bridge method — not at the `apply()` call site.

This differs from anonymous inner classes, where the erased `apply(Object)`
method directly invokes the implementation without a bridge `checkcast`.

### Fix
When an API accepts `Function<T, R>` with a declared `Class<T> inputType()`,
validate the input type **at the call site** before invoking the function:

```java
if (!sync.inputType().isInstance(input)) {
    throw new IllegalArgumentException(
        "Input type mismatch: expected " + sync.inputType().getName()
        + ", got " + (input == null ? "null" : input.getClass().getName()));
}
// Now safe to call with raw type — input IS the right type
((Function) sync.fn()).apply(input);
```

Do not add a "Map escape hatch" like:
```java
// WRONG — this lets Map through, but the lambda's bridge checkcast will CCE
if (!sync.inputType().isInstance(input) && !(input instanceof Map)) { ... }
```

The escape hatch silently passes the type check but the lambda still throws
CCE. Strict `isInstance()` validation produces a clear `IllegalArgumentException`
with both expected and actual types — superior diagnostics.

### Why this is non-obvious
The widespread mental model of Java erasure suggests that raw-type function
invocation is "safe" (just unchecked, not wrong). Most developers expect
`((Function) fn).apply(wrongType)` to either work (erasure) or fail at the
`apply()` call (type mismatch). The failure occurring inside the lambda body
via a compiler-generated `checkcast` is surprising and misdirects debugging
toward the lambda implementation instead of the call site.
