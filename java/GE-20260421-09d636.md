## Symptom

Power iteration does not converge within MAX_ITERATIONS when the trust graph contains a 3-cycle via the dangling-node fallback row. The algorithm silently returns the last computed vector — plausible-looking but far from the true eigenvector.

## Context

Occurs when implementing EigenTrust from the original Stanford paper (Kamvar et al. 2003). The paper recommends: when an actor has no positive attestations, use the pre-trusted distribution `p` as their matrix row. With pre-trusted set `{A}` and chain `A→B→C` where C has no attestations, C's row becomes `[1,0,0]` — all weight back to A — creating a 3-cycle: trust flows A→B→C→A→...

## Root cause

The 3-cycle causes power iteration to oscillate with period 3 rather than converge. With dampening α=0.15 this oscillation decays very slowly — requiring far more than the typical 50–100 max iterations to resolve. Manual trace of 3-actor example confirmed oscillation persists across 8+ iterations with no convergence trend.

## Fix

Use **uniform distribution** (`1/n`) as the fallback row for actors with no positive attestations. Reserve the pre-trusted distribution exclusively for the dampening step:

```
C[i][j] = 1/n  (for all j)  when actor i has no positive attestations

t = (1-α) * Cᵀ * t + α * p
```

This breaks the 3-cycle: C's row no longer concentrates all weight on A. The matrix becomes aperiodic and converges. Verified by tracing a 4-actor chain (A pre-trusted, A→B→C, D unknown): converges in ~6 iterations to stable ordering A>B>C>D with C>D demonstrating transitivity.

## Why this is non-obvious

The paper explicitly recommends using the pre-trusted distribution as the dangling-node fallback to "bootstrap" trust. This seems correct conceptually — actors with no history should defer to known-trustworthy peers. But it creates degenerate 3-cycles in small graphs that the paper never addresses (the paper targets large P2P networks where exact 3-cycles are exceedingly rare). No widely-cited EigenTrust implementation guide documents this failure mode.
