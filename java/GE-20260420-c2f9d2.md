## Symptom

Elements from a `PriorityBlockingQueue` are returned in the wrong order. No exception is thrown.
The queue appears to work correctly until the priority field is mutated on a queued element.

## Context

A `Comparable` element type has a mutable priority field. Elements are inserted with one priority,
then a strategy/scheduler mutates the priority field to re-rank them. The expectation is that
`poll()` now returns the highest-priority element by the new rank.

## Root Cause

`PriorityBlockingQueue` (and `PriorityQueue`) heapify **at insertion time only**. Mutating an
element's sort key after insertion does not trigger re-heapification. The internal heap array
retains the old ordering, so `poll()` returns elements in the original insertion order relative
to their bucket, not the new priority.

Java's documentation notes this but only in a warning about `Comparator` consistency —
it is easy to miss when working with `Comparable`.

## Fix

Make sort key fields `final`. If re-prioritisation is required, the owning collection should
expose a `reprioritise(element, newPriority)` method that removes the element, updates the
field, and re-inserts:

```java
// In the owning model class:
public void reprioritise(String planItemId, int newPriority) {
    PlanItem item = itemsById.get(planItemId);
    if (item == null) return;
    agenda.remove(item);          // O(n) but safe
    item.setPriority(newPriority); // now mutate
    agenda.add(item);             // re-heapify at insertion
}
```

Or, if priority is only ever set once at creation time, make it `final` and remove the setter
entirely — the bug cannot occur.

## Why This Is Non-Obvious

The symptom (wrong order) is identical to a comparator bug. There is no exception, no warning.
Mutating a queued element's fields is normal Java — nothing signals that `PriorityBlockingQueue`
caches the ordering at insertion time. Discovered only through code review, not test failures.
