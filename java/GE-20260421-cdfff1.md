## Hibernate L1 cache returns stale entity after bulk JPQL DELETE — em.clear() required

**ID:** GE-20260421-cdfff1
**Stack:** Hibernate ORM 6.x, JPA, Quarkus 3.32.2
**Symptom:** `em.find()` returns a non-null entity immediately after a bulk JPQL `DELETE` executed via `executeUpdate()`. The row is gone from the database but the L1 (first-level) persistence context cache still holds the object. Test assertion `assertThat(result).isEmpty()` fails even though the DB row was deleted.
**Context:** Occurs whenever `em.createNamedQuery("DELETE FROM ...").executeUpdate()` or similar bulk JPQL DML is used, then `em.find()` or a JPQL SELECT is run in the same persistence context (i.e. within the same `@Transactional` method).

### What was tried (didn't work)
- Relying on the bulk DELETE to automatically invalidate the cache — it does not. The JPA spec explicitly states that bulk DML operations "leave the persistence context in an indeterminate state".

### Root cause
Bulk JPQL DML (`executeUpdate()`) sends SQL directly to the database, bypassing the persistence context entirely. The Hibernate L1 cache has no knowledge of what rows were deleted. Subsequent `em.find()` calls check the cache first and return the stale object without hitting the database.

### Solution 1 — em.clear() after bulk DML
**Approach:** Clear the persistence context immediately after `executeUpdate()`.
**Pros:** Works everywhere; no changes to read queries.
**Cons/trade-offs:** Detaches ALL managed entities in the context, not just the deleted one — can cause lazy-loading issues if other entities in scope still need their associations loaded.

```java
em.createNamedQuery("ActorIdentity.deleteByActorId")
        .setParameter("actorId", rawActorId)
        .executeUpdate();
em.clear(); // flush the L1 cache — bulk DML leaves it stale
```

### Solution 2 — Replace em.find() with JPQL SELECT in the read method
**Approach:** Change the read method from `em.find()` to a JPQL SELECT query. JPQL queries always hit the database rather than the L1 cache.
**Pros:** Surgical — only the specific read bypasses the cache; other managed entities in the context are unaffected.
**Cons/trade-offs:** Requires changing the read method (not always possible if it's shared); JPQL SELECT has slightly more overhead than a PK lookup.

```java
// Change findByChannelId from:
return Optional.ofNullable(em.find(SlackBotBinding.class, channelId));
// To:
return em.createQuery("FROM SlackBotBinding b WHERE b.channelId = :id", SlackBotBinding.class)
        .setParameter("id", channelId)
        .getResultStream().findFirst();
```

Note: even JPQL SELECT may return cached state in some configurations — `em.clear()` (Solution 1) is the more reliable guarantee.

### Why this is non-obvious
The symptom (find returns non-null after delete) looks like the delete didn't work — not like a cache issue. Checking the database directly shows the row is gone, which makes the return value of `em.find()` completely inexplicable without knowing about bulk DML's cache bypass semantics. The JPA spec documents this behaviour but in a section developers rarely read.
