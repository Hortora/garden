## Maven `-pl <submodule>` resolves dependencies from .m2 cache — source changes in sibling modules invisible without `install`

**ID:** GE-20260504-059d1f
**Stack:** Maven 3.x (all versions)
**Symptom:** Running `mvn test -pl runtime` after editing the `api` module produces "cannot find symbol" compilation errors for the new fields/methods — even though the source changes are correct. The runtime module compiles cleanly when built from the parent.
**Context:** Multi-module Maven build where `runtime` depends on `api`. You edit `api` and run an isolated `mvn test -pl runtime` to iterate fast. Maven resolves `api` from `~/.m2/repository`, not from the source tree.

### What was tried (didn't work)
- Assumed `-pl runtime` would resolve sibling `api` from the local source tree — incorrect
- Assumed the "cannot find symbol" errors were code bugs in `runtime` — they were stale-artifact errors

### Root cause
`-pl <submodule>` without `-am` tells Maven to build only that module, resolving all dependencies from the local repository. If the sibling dependency hasn't been installed since your last edit, Maven uses the old jar. The error message ("cannot find symbol") points at the call site, not the missing install.

### Fix
Install the changed dependency module first:
```bash
mvn install -pl api -q
mvn test -pl runtime -q
```
Or rebuild everything from the parent: `mvn test`. Or use `-pl runtime -am` to auto-build upstream modules (adds build time).

### Why this is non-obvious
The error says "cannot find symbol: method trustDimension" — it looks like a code bug in `runtime`, not a stale artifact problem in `api`. The fix is in a different module from the failure. Developers familiar with Gradle's source-set resolution assume Maven has the same capability.
