## Technique

When designing a strategy or plugin interface with optional lifecycle callbacks, declare the optional methods as `default` no-ops in the interface. Implementors override only the methods they need.

## Context

A `MovementStrategy` interface needs three methods: `advance()` (always needed), `clearUnit()` (needed by stateful implementations), and `reset()` (needed by stateful implementations). A stateless `DirectMovement` implementation needs none of the lifecycle methods.

```java
public interface MovementStrategy {
    Point2d advance(String unitTag, Point2d current, Point2d target, double speed);

    // Optional lifecycle — no-op unless the implementation has per-unit state
    default void clearUnit(String unitTag) {}
    default void reset() {}
}

// Stateless — inherits no-op defaults, zero boilerplate
public class DirectMovement implements MovementStrategy {
    @Override
    public Point2d advance(String unitTag, Point2d current, Point2d target, double speed) {
        return EmulatedGame.stepToward(current, target, speed);
    }
}

// Stateful — overrides clearUnit and reset to manage its own state
public class PathfindingMovement implements MovementStrategy {
    // ... waypoints, lastTargets fields ...
    @Override public Point2d advance(...) { ... }
    @Override public void clearUnit(String unitTag) { waypoints.remove(unitTag); ... }
    @Override public void reset() { waypoints.clear(); ... }
}
```

## Why this is non-obvious

Many developers create an abstract base class with no-op implementations instead, which forces a class hierarchy. Or they put all methods in the interface without defaults, forcing every implementor to write empty method bodies. Default no-ops in the interface is the cleanest option: no class hierarchy, no boilerplate, the interface documents that these methods are optional.
