## Symptom

Code that ports Python's `urlparse()` to `java.net.URI` crashes with
`IllegalArgumentException: Illegal character in path` (or similar) on URLs
that Python handled silently.

## Context

When migrating Python URL-parsing code to Java. Python's `urlparse` accepts
any string and never raises — it returns empty fields for invalid parts.
Java's `URI.create()` validates the input against RFC 2396 and throws if it
is invalid. In production web scraping code, malformed attribute values (e.g.
Gist `src` attributes, embed URLs) are common.

## What was tried

Wrapping only the parsing logic, not `URI.create()` itself — the exception
escapes before you can handle a null result.

## Root cause

`URI.create()` is a convenience wrapper for `new URI(str)` that converts
the checked `URISyntaxException` into an unchecked `IllegalArgumentException`.
The Python equivalent truly never throws.

## Fix

Wrap the entire `URI.create()` call in a try-catch and return null (or your
sentinel) on failure:

```java
private static String youtubeVideoId(String url) {
    if (url == null || url.isEmpty()) return null;
    try {
        URI uri = URI.create(url);
        String host = uri.getHost() != null
                ? uri.getHost().toLowerCase().replace("www.", "") : "";
        // ... rest of parsing
    } catch (Exception ignored) {}
    return null;
}
```

The try-catch is not paranoid defensive coding — it is genuinely required
for any URL that comes from user-supplied content or scraped HTML attributes.

## Why this is non-obvious

Python code never guards `urlparse()` calls because it never throws.
Developers porting that code to Java apply the same no-guard pattern and are
surprised by an exception in production. The Java docs do document the
exception, but the contrast with Python's permissiveness isn't highlighted.
