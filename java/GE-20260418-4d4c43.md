## Symptom

A physics E2E test comparing "kiting unit" vs "standing still unit" passes with one cooldown value but fails after a cooldown change. The standing unit appears to significantly outperform the kiting unit despite receiving the same number of attacks.

## Context

In a stateful turn-based combat simulation where attack commands persist across ticks until explicitly cleared by a movement command. Occurs when a test uses `attack(enemyPosition)` as the "stand still" proxy for the non-kiting baseline.

## What was tried

Adjusting the HP threshold assertions when the cooldown changed from 3 ticks to 1 tick. This failed because the root cause wasn't the threshold — it was the baseline itself.

## Root cause

`AttackIntent(enemyPosition)` silently does two things most developers don't expect:
1. **Moves the unit toward the target** — the unit interprets the attack target as a move destination if not yet in range
2. **Persists the attack state across ticks** — the unit stays in `attackingUnits` and fires every tick with no cooldown gap

This gives the "standing still" baseline effectively double DPS (no cooldown gaps) while the kiting unit loses one attack per cooldown tick by retreating. The test's premise was completely invalid — it was comparing kiting vs "charge + infinite DPS", not kiting vs standing.

## Fix

Issue `MoveIntent(unit.currentPosition)` on cooldown ticks for the standing-still baseline. This:
- Keeps the unit stationary (moving to current position = no movement)
- Clears the unit from `attackingUnits`, restoring cooldown-gated firing

Both scenarios now fire at the same rate, isolating retreat movement as the only variable.

```java
// WRONG — silently moves unit toward enemy AND gives double DPS
intentQueue.add(new AttackIntent(tag, enemy.position()));

// RIGHT — stays still, respects cooldown
if (onCooldown) {
    intentQueue.add(new MoveIntent(tag, unit.position())); // clears attack state
} else {
    intentQueue.add(new AttackIntent(tag, enemy.position()));
}
```

## Why this is non-obvious

The test read as "unit A kites; unit B stands still and attacks" — but `AttackIntent` is not a "stand still and attack" command in a stateful simulation. It's a "move to and attack" command that persists until cleared. The difference only becomes visible when comparing two scenarios where attack frequency matters — and the bug can be masked entirely if both scenarios have the same DPS (they cancel out).
