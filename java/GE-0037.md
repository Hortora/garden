## Inject a Supplier<Instant> to test time-dependent logic without sleeping or mocking frameworks

**ID:** GE-0037
**Stack:** Java (all versions with lambdas), JUnit 5
**Labels:** `#testing` `#lambda`
**What it achieves:** Makes any class that reads the current time fully testable — including time-window expiry, TTL logic, and scheduled behaviour — by injecting a controllable clock, with zero production overhead.
**Context:** Any class that calls `Instant.now()`, `LocalDateTime.now()`, `System.currentTimeMillis()`, etc. inline. Common in rate limiters, caches, token expiry, session management.

### The technique

**1. Replace `Instant.now()` with a `Supplier<Instant>` field:**

```java
@ApplicationScoped
public class AuthRateLimiter {

    private Supplier<Instant> clock = Instant::now;  // production default — zero overhead

    // package-private — test infrastructure only
    void setClockForTest(Supplier<Instant> clock) {
        this.clock = clock;
    }

    boolean isRateLimited(String ip) {
        var now = clock.get();   // was: Instant.now()
        // ... sliding window logic using `now`
    }
}
```

**2. In tests, freeze time using a single-element array as a mutable holder:**

```java
@Test
void windowExpiryUnblocksIPAfterWindowPasses() {
    Instant[] time = {Instant.EPOCH};             // single-element array — mutable holder
    rateLimiter.setClockForTest(() -> time[0]);   // lambda closes over array reference

    // exhaust the limit at epoch
    for (int i = 0; i <= MAX_ATTEMPTS; i++) {
        rateLimiter.isRateLimited("10.0.0.1");
    }
    assertTrue(rateLimiter.isRateLimited("10.0.0.1"));

    // advance clock past the window — mutate the array element
    time[0] = Instant.EPOCH.plus(WINDOW).plusSeconds(1);

    // old attempts now expire; IP is unblocked
    assertFalse(rateLimiter.isRateLimited("10.0.0.1"));
}
```

The lambda `() -> time[0]` closes over the **array reference** (which is effectively final), not the element value. Mutating `time[0]` changes what the lambda returns on subsequent calls.

### Why this is non-obvious

The instinct for testing time-dependent logic is usually one of:
- **Sleep** — `Thread.sleep(WINDOW.toMillis())` — makes tests take minutes
- **Mockito `mockStatic`** — mocks `Instant.now()` statically — heavyweight, error-prone, requires `MockedStatic` cleanup
- **`java.time.Clock` injection** — correct but requires constructor/CDI changes and `Clock.fixed()` setup

The `Supplier<Instant>` approach is lighter than Clock injection (one field, one setter), avoids static mocking entirely, and costs nothing in production — `Instant::now` is a method reference, not an allocation per call.

The single-element array trick (`Instant[] time = {Instant.EPOCH}`) solves the Java closure restriction that captured variables must be effectively final: the array reference is final, but the element is mutable.

### When to use it
- Any class that embeds `Instant.now()` / `System.currentTimeMillis()` in logic paths you want to test
- Particularly effective for sliding-window algorithms, TTL checks, token expiry
- Keep `setClockForTest()` package-private — it's test infrastructure, not API
- If the class is a CDI bean, combine with a `resetForTest()` method for full test isolation (see GE-0036 for why CDI beans need explicit reset)
