## Context

An `@ApplicationScoped` service class (e.g. a REST resource or business service) contains both injected CDI dependencies and parsing/transformation logic. The parsing logic needs unit tests, but the class can't be instantiated with `new` because CDI proxies it.

## The technique

Extract the parsing/transformation methods as `static` with package-private visibility. Tests instantiate via `new` (CDI fields are null, but the static methods don't use them) and call the static methods directly:

```java
@ApplicationScoped
public class IngestService {

    @Inject StateStore stateStore; // CDI injection — null in unit tests

    // Static package-private — callable without CDI
    static List<String> parseSitemapUrls(String xml, String base) { ... }
    static List<String> parseFeedLinks(String rss) { ... }
    static String makeSlug(String date, String url) { ... }
    static boolean isPostUrl(String url) { ... }
}
```

Unit test:

```java
class IngestServiceTest {
    @Test
    void parseSitemapUrls_filtersNonPostUrls() {
        List<String> urls = IngestService.parseSitemapUrls(xml, "https://blog.com");
        assertEquals(2, urls.size());
    }
}
```

No Mockito, no `@QuarkusTest`, no port binding. Tests run in milliseconds.

## Why this is non-obvious

The instinct is to mock injected dependencies or use `@QuarkusTest` with `@InjectMock`. Both add significant overhead (Mockito bytecode, full Quarkus boot). The key insight: CDI-injected fields and static methods are orthogonal — `static` methods are resolved at compile time and work even when CDI fields are null, as long as the static method body doesn't touch `this`.
