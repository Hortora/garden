## Maven targets main project instead of worktree when shell CWD resets between tool calls

**ID:** GE-20260420-81d143
**Stack:** Maven (all versions), git worktree
**Symptom:** Running `mvn test -pl runtime` appears to succeed or fail based on
the main project's state, not the worktree's changes. Test output references
the main project path even though you believe you're working in a worktree.

**Context:** In agentic coding environments (Claude Code, Copilot CLI), the shell
CWD resets to the project root between Bash tool invocations — it is not
persisted from a previous `cd` call. When working in a git worktree at
`.worktrees/my-feature/`, any `mvn` command without an explicit `-f` flag reads
`pom.xml` from the main project directory, silently ignoring all worktree changes.

### What was tried (didn't work)
- `cd .worktrees/my-feature && mvn test -pl runtime` — `cd` does not persist;
  next invocation starts from the main project root
- Checking worktree file contents directly — files look correct because the
  worktree source is right; the build reads the wrong pom.xml

### Root cause
Maven resolves `pom.xml` relative to the process working directory. Git worktrees
are a separate directory tree, not a mount or alias. In agentic environments where
each Bash tool call gets a fresh shell, `cd` state is lost between calls, so Maven
consistently operates on the main project.

### Fix

Always use the `-f` flag with an absolute path when working in a worktree:

```bash
# Instead of:
mvn test -pl runtime

# Use:
JAVA_HOME=$(/usr/libexec/java_home -v 26) \
  mvn -f /absolute/path/to/.worktrees/my-feature/pom.xml test -pl runtime
```

For all Maven commands in a worktree session, prefix with `-f /absolute/worktree/path/pom.xml`.

### Why non-obvious
`mvn test -pl runtime` produces no error — it successfully runs tests, just against
the wrong project. The output looks normal. The symptom (tests passing or failing
unexpectedly) is attributed to code changes rather than to the wrong project being
built. The connection to shell CWD reset in agentic tools is invisible unless you
know that `cd` state does not persist between tool calls.

### Alternatives considered
- Using a `.mvn/` config file in the worktree to override defaults — does not
  affect which `pom.xml` Maven reads
- Relying on `-pl` module selection — `-pl` is relative to the reactor root (CWD),
  not the worktree
