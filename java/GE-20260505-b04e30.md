## Symptom

Compilation error when writing an assertion like:
```java
assertThat(() -> strategy.onLifecycleEvent("no-such-case")).doesNotThrowAnyException();
```
The compiler cannot infer a type for the lambda and rejects it.

## Context

Writing unit tests for `EventsOnlyStrategy.onLifecycleEvent()`, which returns `void`. The intention was to verify it doesn't throw for an unknown case ID.

## Root Cause

`assertThat()` is overloaded for many types. When passed a lambda, AssertJ tries to infer whether it's a `Callable`, `ThrowingCallable`, `AbstractAssert`, etc. A void-returning lambda doesn't match any of these unambiguously, so the compiler fails to resolve the overload.

## Fix

Use `assertThatCode()` instead — it's specifically designed for void lambdas:

```java
// WRONG — compile error
assertThat(() -> strategy.onLifecycleEvent("no-such-case")).doesNotThrowAnyException();

// CORRECT
assertThatCode(() -> strategy.onLifecycleEvent("no-such-case")).doesNotThrowAnyException();
```

## Why This Is Non-Obvious

`assertThat()` works for most things, including non-void method calls. It's natural to reach for `assertThat()` first. The compile error message (`cannot infer type arguments`) doesn't hint at the solution. The fix (`assertThatCode`) must be known specifically.
