## Symptom

Configuring an extremely long timeout (effectively "never") using `Long.MAX_VALUE` with a large `TimeUnit` produces a large negative value instead. Behavior that should be "reconnect never" becomes "reconnect immediately."

## Context

```java
// Trying to disable SSE auto-reconnect:
SseEventSource.target(target)
    .reconnectingEvery(Long.MAX_VALUE, TimeUnit.DAYS)
    .build();
```

Internally, `TimeUnit.DAYS.toMillis(Long.MAX_VALUE)` computes:
```
Long.MAX_VALUE × 86,400,000 = overflow → large negative long
```

A negative reconnect delay may be interpreted as 0 or "reconnect immediately" by the scheduling logic.

## Root cause

`TimeUnit.toMillis()` is a simple multiplication. `Long.MAX_VALUE × 86_400_000` exceeds `Long.MAX_VALUE`, silently wrapping to a negative number under Java's two's complement integer arithmetic. No exception is thrown; no warning is logged.

```java
TimeUnit.DAYS.toMillis(Long.MAX_VALUE)       // → -768614336404564992 (negative!)
TimeUnit.MILLISECONDS.toMillis(Long.MAX_VALUE) // → 9223372036854775807 (correct)
```

## Fix

Use `TimeUnit.MILLISECONDS` when the value is already in milliseconds (no conversion needed):

```java
SseEventSource.target(target)
    .reconnectingEvery(Long.MAX_VALUE, TimeUnit.MILLISECONDS)
    .build();
```

`Long.MAX_VALUE` milliseconds ≈ 292 million years — effectively "never reconnect."

## Why this is non-obvious

`Long.MAX_VALUE` feels like a safe "infinity" sentinel. `TimeUnit.DAYS` feels like the most human-readable unit for a very long duration. The combination looks natural and sensible — the failure is invisible (no exception) and the observable behavior (reconnecting immediately) seems completely unrelated to the TimeUnit choice.
