## Context

In a multi-module Maven project with this dependency order:
```
claudony-core → claudony-casehub → claudony-app
```

`ClaudonyWorkerStatusListener` in `claudony-casehub` needed to notify `CaseEventBroadcaster` in `claudony-app` of worker lifecycle events. `claudony-casehub` cannot import `claudony-app` — that would be circular.

## Technique

Define a CDI event record in the shared module (`claudony-core`) that both sides can reference:

```java
// claudony-core — accessible to both modules
public record WorkerCaseLifecycleEvent(String caseId) {}
```

The producer (lower module) fires the event using CDI's `Event<Object>`:
```java
// claudony-casehub — fires the event, no knowledge of the observer
@Inject Event<Object> events;

events.fire(new WorkerCaseLifecycleEvent(caseId));
```

The observer (upper module) observes the concrete type:
```java
// claudony-app — observes without the casehub module knowing
@ApplicationScoped
public class CaseEventBroadcaster {
    void onWorkerLifecycleEvent(@Observes WorkerCaseLifecycleEvent event) {
        emit(event.caseId());
    }
}
```

CDI resolves observer methods at build time and wires them without either module having a compile-time dependency on the other's observer/producer class.

## Why This Is Non-Obvious

The usual response to "A needs to notify B but A can't import B" is to restructure the modules, create an intermediate interface, or introduce a message broker. CDI events solve it elegantly within a single JVM — the event type in the shared module acts as the contract, and Quarkus's CDI implementation wires up observers across module boundaries transparently. The key insight is that `Event<Object>` in the lower module can fire any object — the observer in the upper module registers on the concrete type. No intermediate interface or callback registration is needed.
