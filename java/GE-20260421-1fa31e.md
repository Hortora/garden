### Symptom

After `mvn clean test`, compilation fails with `NoSuchFieldError: Class ParserConfiguration$LanguageLevel does not have member field 'JAVA_21'`. The error looks like a missing class or incompatible dependency, but the previous `mvn test` (without clean) passed fine.

### Context

Project uses an annotation processor (APT) that generates additional source files during compilation. The pom.xml pins a lower version of a transitive library (e.g. JavaParser 3.25.9) while the locally-installed APT tool was built against a newer version (3.28.0 which added `JAVA_21`).

### What was tried

`mvn clean test` — fails. `mvn test` (without clean) — passes because previously generated classes are still in target/.

### Root cause

`mvn clean` deletes target/, including APT-generated sources. On the next build, the APT processor runs and attempts to use `JAVA_21` from JavaParser. But the version in `annotationProcessorPaths` is the older pom-pinned version that lacks `JAVA_21`. Without clean, the stale generated classes were used instead, masking the mismatch.

### Fix

Upgrade the library version in pom.xml to match what the APT tool was built against:

```xml
<!-- Before -->
<javaparser.version>3.25.9</javaparser.version>

<!-- After — matches the annotation processor's built-against version -->
<javaparser.version>3.28.0</javaparser.version>
```

### Why this is non-obvious

The error message points to a missing enum constant in a library, not to a version mismatch. The root cause chain — clean removes generated files → APT runs fresh → APT uses pom-pinned library → version mismatch — requires knowing that annotationProcessorPaths uses the pom version, not the tool's bundled version. The symptom only appears after `mvn clean`.
