### Context
An annotation-processing or code-generation pipeline has a `generate(config, sourceClass)`
method built for template classes — it reads a config declaring iteration bounds,
class-name patterns, and generation parameters. You want to run a non-template class
(one that keeps its own name but receives injected/generated methods) through the same
pipeline without duplicating the generation logic.

### Technique
Synthesize a minimal `from=1, to=1` config with `className = <actualClassName>`.
The pipeline runs one iteration, produces one output class with the same name, and
handles all method expansion, annotation stripping, and file writing through the
standard path:

```java
// "RuleBuilder" is not a template — but needs generated methods injected.
// Instead of a special code path, synthesize a dummy config:
PermuteConfig syntheticConfig = new PermuteConfig(
        "i",             // loop variable (unused — range is 1..1)
        "1", "1",        // from=1, to=1 — single iteration
        className,       // className = actual class name, no ${i} template
        new String[0],
        new VarConfig[0],
        true,            // inline=true
        false);          // keepTemplate=false (replace, don't keep original)

List<Map<String, Object>> combos = buildAllCombinations(syntheticConfig);
// combos = [{i=1}] — exactly one combination, i=1 never referenced

injectMixinMethods(nonTemplateClass, allSourceCus); // inject before generate
CompilationUnit output = generate(sourceCu, nonTemplateClass, syntheticConfig, combos);
writeGeneratedFile(qualifiedName, output.toString());
```

The pipeline removes the input class from the CU and re-adds it under the same name —
the original class is replaced by an augmented version with injected methods expanded
and all processor annotations stripped.

### Why this is non-obvious
The natural response to "process a non-template class through a template pipeline" is
to add a conditional branch or a separate method. The insight is that `from=1, to=1,
className=<exact name>` is semantically equivalent to "run the pipeline once, keep the
name unchanged" — the class acts as its own template. Most developers treat configs as
something you read from annotations, not something you construct programmatically to
route processing through an existing path.
