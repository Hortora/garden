### Symptom
Code filtering on `instanceof ClassOrInterfaceDeclaration` silently processes
interfaces alongside classes. Guards intended to apply to classes only (e.g.
"inject methods into concrete classes") fire on interfaces too, causing wrong
output with no error.

### Context
Any annotation processor or AST transformer that iterates TypeDeclarations and
gates on `instanceof ClassOrInterfaceDeclaration` to target concrete classes.

### Root cause
JavaParser uses a single `ClassOrInterfaceDeclaration` node for both class and
interface declarations. The type name is misleading — there is no separate
`ClassDeclaration` node. `coid.isInterface()` is the only way to distinguish.

### Fix
Add `|| coid.isInterface()` to any exclusion guard:

```java
if (!(td instanceof ClassOrInterfaceDeclaration coid) || coid.isInterface())
    continue; // skip interfaces and non-COID types
```

### Why this is non-obvious
The class name `ClassOrInterfaceDeclaration` implies "class OR interface" is the
type, but developers naturally read it as "a class or an interface declaration
node (unified)". The intuitive expectation is that filtering on it gives only
concrete classes, similar to how `RecordDeclaration` is a distinct node for
records. It isn't. There is no `ClassDeclaration` node in JavaParser's AST.
