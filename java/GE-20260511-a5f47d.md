### Symptom
`isKnown("security-review")` returns `false` for a capability that `capabilities()` clearly includes, after deploying a custom `CapabilityRegistry` subclass that overrides `capabilities()`.

### Context
Any Java SPI where `isKnown` is implemented by calling the backing static constant directly (`ALL_CAPABILITIES.contains(capability)`) rather than delegating through the public virtual method (`capabilities().contains(capability)`). Tests with the default implementation pass because `ALL_CAPABILITIES` and `capabilities()` return the same set. The bug surfaces only when a subclass overrides `capabilities()` to add or filter entries.

### Root cause
`isKnown` bypasses virtual dispatch. The static field `ALL_CAPABILITIES` is fixed at class-load time and does not reflect runtime overrides of `capabilities()`.

### Fix

```java
// Wrong — bypasses override
public boolean isKnown(String capability) {
    return ALL_CAPABILITIES.contains(capability);
}

// Correct — delegates through virtual dispatch
public boolean isKnown(String capability) {
    return capabilities().contains(capability);
}
```

Better still: make `isKnown` a `default` method on the SPI interface itself. Subclasses only need to implement `capabilities()` and get a correct `isKnown` for free:

```java
public interface CapabilityRegistry {
    Set<String> capabilities();

    default boolean isKnown(String capability) {
        Objects.requireNonNull(capability, "capability must not be null");
        return capabilities().contains(capability);
    }
}
```

### Why this is non-obvious
The default implementation's tests pass. The bug only surfaces when a subclass overrides `capabilities()` — which is the primary extension point. By then, `isKnown` and `capabilities()` silently disagree, and diagnosing which layer returns the wrong answer takes time.
