### Symptom
Units walk straight through wall tiles despite pathfinding being wired. No error or warning.

### Context
A* pathfinder on a bounded grid (e.g. 64×64). A movement target is supplied with coordinates
outside the grid bounds (e.g. from a scouting plugin using real SC2 map coordinates on a
smaller emulated map).

### What was tried (didn't work)
- Verified pathfinding was wired in the engine (it was)
- Added diagnostic logging to confirm PathfindingMovement.advance() was called (it was)
- Assumed wall passthrough was the enemy unit — spent significant time debugging the wrong unit

### Root cause
`nearestWalkable(grid, x, y)` uses a spiral search capped at
`Math.max(grid.width(), grid.height())` — 64 for a 64×64 grid. A target at (224, 224)
requires search radius 161 to reach the nearest in-bounds tile. The cap is never reached,
the function returns the original out-of-bounds point unchanged, A* finds no path
(goal unreachable), and PathfindingMovement falls back silently to `stepToward()` —
which is DirectMovement through walls.

### Fix
Clamp coordinates to grid bounds before the spiral search:
```java
x = Math.max(0, Math.min(grid.width()  - 1, x));
y = Math.max(0, Math.min(grid.height() - 1, y));
if (grid.isWalkable(x, y)) return new int[]{x, y};
// then spiral search from clamped position
```
`nearestWalkable(224, 224)` on a 64×64 grid becomes `nearestWalkable(63, 63)` — immediately walkable, no search needed.

### Why this is non-obvious
The symptom (wall passthrough) points to pathfinding logic, not to target validation.
The fallback to DirectMovement is silent — no log, no exception.
A unit moving diagonally through a wall looks identical whether caused by DirectMovement
or a failed A* with DirectMovement fallback.
