## Behaviour

RestAssured's Groovy GPath supports finding a specific `int[]` element inside a `List<int[]>` using `find { it == [x, y] }` — even though Java `int[].equals()` is reference equality.

```java
// Endpoint returns: {"walls": [[0,18],[1,18],[20,18],...]}
// List<int[]> serialised by Jackson

// Check that [20, 18] IS in the list:
.body("walls.find { it == [20, 18] }", notNullValue())

// Check that [12, 18] is NOT in the list:
.body("walls.find { it == [12, 18] }", nullValue())
```

Both assertions work correctly.

## Why it works

RestAssured evaluates GPath expressions using Groovy, not Java. Groovy's `==` operator calls `equals()` for objects but performs deep equality for arrays and lists. `[20, 18]` in a Groovy GPath expression is a `List<Integer>`, and Groovy's `==` compares element-by-element against the deserialised JSON array.

## Why this is undocumented

The RestAssured docs show `find` for object properties (`find { it.name == 'foo' }`) but do not document array/list equality matching. The fact that Java `int[].equals()` is reference-based makes developers assume this cannot work — they typically convert to a string or iterate manually.

## Alternative (without this trick)

```java
// Check count instead of specific tile:
.body("walls.size()", equalTo(61))

// Or manually verify via deserialization:
List<List<Integer>> walls = given().get("/terrain").jsonPath().getList("walls");
assertThat(walls).contains(Arrays.asList(20, 18));
```
