## Symptom
A Python script that discovers files using `Path('.').glob()` or `find_root()` via `Path.cwd()` silently finds 0 files and writes empty/default output. No error is raised. The script reports success with the correct file count ("updated (48 skills)") but every entry it writes contains empty data.

## Context
The script is run from within a git worktree directory (e.g. `.worktrees/feature-branch/`). The worktree has `CLAUDE.md` and `README.md` (copied from the main repo at worktree creation), so any heuristic that walks up from `cwd()` looking for those files finds the worktree root — not the main repo root where the actual source files live.

## Root cause
`Path.cwd()` returns the current working directory, which is the worktree path when Claude or a subprocess runs inside the worktree. `find_skills_root()` (and similar helpers) walk upward from `cwd()` stopping at the first directory that contains `CLAUDE.md`. Worktrees inherit `CLAUDE.md`, so the walk stops early. `*.glob('*/SKILL.md')` then scans the worktree root, which has no skill directories, returning 0 files.

## What was tried
Running `generate_web_app_data.py` repeatedly from the worktree — each run reported "CHAINING_TRUTH: updated (48 skills)" and "CHAIN JS: updated (48 skills)", but inspection of the written file showed every `builds_on`, `chains_to`, and `invoked_by` list was `[]`. Rerunning with explicit debug output showed `skills_root = /path/.worktrees/test-coverage` and `found skill files: 0`.

## Fix
Always run scripts that use `Path.cwd()` for file discovery from the main repo directory, not from a worktree:
```bash
cd /path/to/main-repo
python3 scripts/generate_web_app_data.py
```

Or pass the root explicitly and use it instead of `Path.cwd()`:
```python
root = Path(__file__).parent.parent  # script's own location, not cwd
```

## Why non-obvious
The symptom — empty data written to files — gives no indication that the problem is a wrong working directory. The script reads and writes files that DO exist in the worktree (the test file and the HTML file), so no FileNotFoundError is raised. The "updated (48 skills)" message fires because the regex replacement succeeds and the new content differs from the old, but the new content is all-empty dicts. The debugging trail (subprocess output, file inspection) points toward the write logic, not the discovery logic.

### Why this fix
Using the script's own `__file__` location as the root anchor is more robust than `cwd()` — it works correctly regardless of where the caller invokes the script from.
