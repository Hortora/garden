## Symptom
Tests that assert `issue.severity == Severity.CRITICAL` always pass, even when they should fail. Or they always return False, even when the issue genuinely is CRITICAL. No error is raised.

## Context
A test file imports `Severity` from `utils.common` after doing `sys.path.insert(0, 'scripts')`. The validator under test also does `sys.path.insert(0, str(Path(__file__).parent))` internally, which maps to the same directory. Python's import machinery treats these as two separate module registrations and loads `utils.common` twice — once as `utils.common` and once as `scripts.utils.common` (or vice versa). The two `Severity` enums are distinct objects in memory.

## Root cause
Python enum identity comparison (`==`) checks object identity of the enum class, not just the value. When the same source file is loaded under two different module paths, two separate class objects are created. `Severity.CRITICAL` from one load is not `==` to `Severity.CRITICAL` from the other, even though they have the same name and value.

## What was tried
Wrote tests asserting `is_critical(i)` (a helper that checks `i.severity == Severity.CRITICAL`). Tests always returned False for issues that were demonstrably CRITICAL, leading to extensive debugging of the validator's logic before discovering the module identity issue.

## Fix
Use string comparison instead of enum identity:
```python
# Instead of:
assert issue.severity == Severity.CRITICAL

# Use:
assert issue.severity.name == "CRITICAL"

# Or check the exit code directly:
assert result.returncode == 1
```

Alternatively, ensure only one `sys.path.insert` path is used by standardising how both the test and the validator locate `utils.common`.

## Why non-obvious
The test runs without error. `print(issue.severity)` shows `Severity.CRITICAL`. `issue.severity.name` returns `"CRITICAL"`. Only `issue.severity == Severity.CRITICAL` silently returns False. The root cause requires knowing that Python's import system uses module path strings as cache keys, not file paths — so the same `.py` file loaded under two different module paths creates two distinct class objects.

### Why this fix
String comparison on `.name` bypasses enum class identity entirely and works regardless of how many times the module is loaded.
