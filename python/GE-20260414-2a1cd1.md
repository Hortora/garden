## Regex-validated date strings still crash date.fromisoformat() on invalid calendar values

**ID:** GE-20260414-2a1cd1
**Stack:** Python 3.7+, datetime.date
**Symptom:** `ValueError: day is out of range for month` raised by `date.fromisoformat()` despite the input passing a format-validation regex.
**Context:** Any code that validates date format with regex then passes the string to `fromisoformat` without a try/except, assuming the regex was sufficient.

### What was tried (didn't work)
- Validated format with `\d{4}-\d{2}-\d{2}` regex before calling `fromisoformat` — assumed this was sufficient
- The pattern ensures shape (4 digits, dash, 2 digits, dash, 2 digits) but not calendar validity

### Root cause
`\d{4}-\d{2}-\d{2}` matches `2026-13-01` (month 13) and `2026-02-31` (Feb 31). Both pass the regex. Both raise `ValueError` in `date.fromisoformat()`. The regex validates format; the standard library validates calendar validity. One bad entry crashes the entire processing loop.

### Fix
Wrap `fromisoformat` calls in try/except and continue:
```python
try:
    submitted = date.fromisoformat(s.group(1))
    last_reviewed = date.fromisoformat(r.group(1)) if r else None
except ValueError:
    continue  # skip malformed entry, don't crash the whole scan
```

### Why this is non-obvious
The regex approach feels complete — you checked the format. The assumption that format validation implies value validity is natural but wrong for calendar dates. The crash is loud (ValueError with a calendar message) but the root cause (regex not being sufficient) is non-obvious when you believe format validation was done.
