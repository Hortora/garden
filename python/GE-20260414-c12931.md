## YAML frontmatter regex silently skips files with CRLF line endings

**ID:** GE-20260414-c12931
**Stack:** Python (all versions), re module
**Symptom:** Files with YAML frontmatter are not processed — no error, no warning, just silently absent from results.
**Context:** Any Python code using regex to detect YAML frontmatter delimiters (`---`) in markdown files, particularly when files may have been created or edited on Windows.

### What was tried (didn't work)
- Verified the files existed and were readable — they were
- Checked the regex pattern against known-good files — matched fine
- Assumed the issue was in content parsing — it was upstream of that

### Root cause
The regex `r'^---\n(.*?)\n---'` anchors on `\n` (LF). Files with Windows CRLF line endings (`\r\n`) produce `---\r\n` at the delimiter line, which does not match `---\n`. The `re.DOTALL` flag makes `.` match newlines but does not normalise line endings. The file is silently skipped — no exception is raised.

### Fix
Normalise line endings before matching:
```python
content_normalized = content.replace('\r\n', '\n')
fm_match = re.match(r'^---\n(.*?)\n---', content_normalized, re.DOTALL)
```

### Why this is non-obvious
The regex looks correct. The file exists and is readable. There is no error — the file is simply skipped. The symptom (missing results) points away from a regex issue and toward content or logic problems. CRLF is invisible in most editors and terminal output.
