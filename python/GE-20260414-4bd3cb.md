## What
`validate_links.py` in the cc-praxis validation suite uses `requests.get()` to check external URLs, not `requests.head()` — despite HEAD being the conventional choice for link checkers (avoids downloading response body).

## Discovery
Found by reading the source during TDD. The mock target `requests.head` produced silently-passing tests (no HTTP call was made, no issue was reported), while patching `validation.validate_links.requests.get` produced correctly-behaving tests.

## Impact on tests
Mocking `requests.head` when the validator uses `requests.get`:
- All link checks silently succeed (no mock is hit)
- Tests asserting "404 produces a WARNING" always pass (because no check runs)
- Tests asserting "200 produces no issue" also always pass (same reason)
- Nothing fails, but the tests are testing nothing

## Correct mock target
```python
with patch('scripts.validation.validate_links.requests.get') as mock_get:
    mock_get.return_value.status_code = 404
    # now the validator will hit the mock
```

## Additional behaviour
- Broken links produce `Severity.WARNING` (exit 2), never `Severity.CRITICAL` (exit 1)
- Bot-blocking codes 403, 429, 503 are treated as OK (no issue)
- `allow_redirects=True` and `stream=True` are set on every call
- Response is explicitly closed with `response.close()` after reading status

## Why undocumented
The choice of GET over HEAD is not documented in comments or README. No other link-checker in the codebase uses this pattern. The only way to discover it is to read the source.
