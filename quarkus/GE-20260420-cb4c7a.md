## Symptom

A Gauge registered as `workitems.total` is missing from `GET /q/metrics` output. A tagged gauge like `workitems.by.status` (same registration code path, same bean) appears correctly. No error is thrown.

## Context

In Prometheus exposition format, metric names ending in `_total` are conventionally Counters. Micrometer converts dots to underscores, so `workitems.total` becomes `workitems_total`. Some Prometheus clients or Micrometer-Prometheus integrations treat `_total` as a reserved suffix and either strip, rename, or suppress the gauge silently.

## Root cause

The Prometheus data model convention is that `_total` is a required suffix for Counters and must not be used for other metric types. When a Gauge exposes `workitems_total`, tooling becomes confused: the metric looks like a Counter total but has gauge semantics. The `# HELP` and `# TYPE` lines may still appear, but the sample line with the value is absent or the metric is excluded from scrape output.

Tagged variants like `workitems_by_status{status="PENDING"}` do not end in `_total` and are unaffected.

## Fix

Name gauges to avoid the `_total` suffix:

```java
// ❌ Becomes workitems_total — conflicts with Prometheus Counter convention
Gauge.builder("workitems.total", ...).register(registry);

// ✅ Unambiguous
Gauge.builder("workitems.active", ...).register(registry);
Gauge.builder("workitems.count", ...).register(registry);
```

General rule: avoid naming any Gauge with a name that ends in `.total` (which becomes `_total` in Prometheus format).

## Why this is non-obvious

The gauge IS registered — `registry.getMeters()` includes it. The issue only manifests in the Prometheus exposition format. No exception, no warning. A tagged gauge in the same bean works fine, so the bean initialisation is not the issue. The connection between the missing metric and the `_total` naming convention is not documented in Micrometer or Prometheus SDK docs.
