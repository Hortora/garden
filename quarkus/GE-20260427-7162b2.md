### Symptom
`@QuarkusTest` starts on a random port (configured via `quarkus.http.test-port=0`), but a REST client configured via property substitution connects to the default URL (e.g. `localhost:7777`) and fails with `Connection refused`. Tests that call the service through itself all fail.

### Context
Occurs when a Quarkus app in agent mode calls back to its own server-mode REST API, or any `@RegisterRestClient` references a server URL that is configured in `application.properties` without a `%test` override. The `test-port=0` only randomises `quarkus.http.port` — it does not propagate the random port to property values elsewhere.

### What was tried (didn't work)
- Assumed `@QuarkusTest` would automatically use the test port for all internal REST calls
- Setting the URL to `http://localhost:0` — does not resolve to the actual port

### Root cause
`quarkus.http.test-port=0` assigns a random port to `quarkus.http.port` at test startup, but any other config property that references the default URL (e.g. `claudony.server.url=http://localhost:7777`) is never updated. The REST client resolves its URL at startup from the un-overridden property value.

### Fix
Add to `src/test/resources/application.properties`:
```properties
%test.your.server.url=http://localhost:${quarkus.http.port}
```
Quarkus resolves `${quarkus.http.port}` to the actual assigned random port at test startup. The `%test` prefix ensures this only applies in the test profile.

### Why this is non-obvious
There is no warning when the REST client connects to a non-test port — it silently fails with `Connection refused`. The fix requires knowing that Quarkus exposes the actual assigned port as `${quarkus.http.port}` (not `${quarkus.http.test-port}`, which stays as `0`).
