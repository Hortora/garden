### Symptom

Build-time `IllegalStateException: Panache entities do not support being attached to several persistence units: io.example.MyEntity is attached to: <default>,qhorus`

### Context

Quarkus project with two named persistence units configured via `quarkus.hibernate-orm.packages` and `quarkus.hibernate-orm.qhorus.packages`. Occurs when the same Java package appears in both lists — even if you intended only a subset of entities in that package to be managed by the second PU.

### What was tried

Added a `LedgerEntry` subclass to `io.casehub.clinical.entity` (alongside regular Panache entities), then listed `io.casehub.clinical.entity` in both `hibernate-orm.packages` and `hibernate-orm.qhorus.packages` so both entity types would be visible to their respective PUs.

### Root cause

Quarkus scans the entire package for all entity classes when a package is listed in a PU's `packages` config. Panache entities found in both scans are flagged as "belonging to two PUs" at build time — Panache's proxy mechanism cannot support this.

### Fix

Put JPA entities managed by the second PU in a **separate package** not listed under the default PU:

```properties
# Default PU — Panache entities only
quarkus.hibernate-orm.packages=io.casehub.clinical.entity

# Second PU — separate package for non-Panache JPA entities
quarkus.hibernate-orm.qhorus.packages=io.casehub.ledger.runtime.model,io.casehub.clinical.ledger
```

Create the new package (`io.casehub.clinical.ledger` or similar) and place the non-Panache JPA entities there.

### Why this is non-obvious

The natural instinct is to put a new entity subclass alongside related entities. The error appears at build time (not runtime), but the message doesn't explain that the fix requires a separate package — it just says the entity is "attached to" two PUs.
