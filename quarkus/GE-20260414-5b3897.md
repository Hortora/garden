## quarkus-flow uses the CNCF Serverless Workflow SDK directly — not Kogito

**ID:** GE-20260414-5b3897
**Stack:** quarkus-flow, Quarkus 3.x, CNCF Serverless Workflow SDK 7.17.0.Final
**Symptom:** When building integrations with quarkus-flow, assumptions about the API surface, SPI hooks, and extension mechanisms based on Kogito documentation are wrong. Kogito's `TaskService`, `ProcessRuntime`, `KieSession`, and related classes are absent.
**Context:** Any developer coming to quarkus-flow with a Quarkus background who tries to find workflow functionality through the Kogito extension, or looks for Kogito classes on the classpath, or reads Kogito SPI documentation expecting it to apply.

### What was tried (didn't work)
- Searching for Kogito classes (`org.kie.*`, `org.jbpm.*`) in the quarkus-flow classpath — none present
- Looking for Kogito-based extension mechanisms (`KogitoWorkItemHandler`, `ProcessEventListener`) — absent
- Assuming the `quarkus-kogito` extension would provide task execution hooks — wrong extension entirely

### Root cause

quarkus-flow is not built on Kogito. It uses the **CNCF Serverless Workflow Java SDK** (`io.serverlessworkflow:serverlessworkflow-bom:7.17.0.Final`) as its workflow engine. This is a completely separate project from Kogito/jBPM, developed by the CNCF Serverless Workflow community.

Quarkus does have deep Kogito integration (quarkus-kogito, quarkus-kogito-rules, quarkus-drools), which is why a Quarkus developer would naturally assume any workflow extension is Kogito-based. quarkus-flow is the exception.

Key imports in quarkus-flow:
```xml
<dependency>
  <groupId>io.serverlessworkflow</groupId>
  <artifactId>serverlessworkflow-bom</artifactId>
  <version>7.17.0.Final</version>
  <type>pom</type>
  <scope>import</scope>
</dependency>
```

The SPI extension points are in `io.serverlessworkflow.impl.executors.*`, not in any Kogito package.

### Fix

When integrating with quarkus-flow:
- Read the **CNCF Serverless Workflow spec** (not BPMN/Kogito docs)
- Look for SPIs in `io.serverlessworkflow.impl.executors.*`
- The key extension interface is `TaskExecutorFactory` (see GE-20260414-14d244)
- The workflow definition format is JSON/YAML per CNCF Serverless Workflow spec, not BPMN XML

### Why non-obvious

Quarkus is synonymous with Kogito in the workflow space — Kogito was developed as part of the Quarkus ecosystem. quarkus-flow's use of the CNCF SDK instead is completely counter to that expectation. The extension name "quarkus-flow" gives no indication of which underlying spec or SDK it uses.
