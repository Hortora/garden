## `<packaging>quarkus</packaging>` in pom.xml is non-standard and causes tooling issues — Quarkus doesn't generate it

**ID:** GE-0031
**Stack:** Quarkus 3.x, Maven 3.x (all versions)
**What it is:** The `<packaging>quarkus</packaging>` element in a Quarkus Maven project's `pom.xml` is not generated by `quarkus-maven-plugin:create` and is not needed. Adding it manually introduces compatibility issues with Maven wrapper and IDE import tooling.
**How discovered:** Code quality review noticed the element was present after manual editing; the Quarkus scaffold does not emit it; removing it had no effect on the build.

### Description

When `quarkus-maven-plugin` is declared with `<extensions>true</extensions>`, it registers a custom build lifecycle that handles everything the `quarkus` packaging type would provide. The `<packaging>` element is therefore redundant at best and a source of tooling confusion at worst.

Problems caused by `<packaging>quarkus</packaging>`:
- Maven wrapper (`./mvnw`) may mishandle lifecycle binding in some versions
- IDE project import (IntelliJ, Eclipse) can fail to recognise the packaging type and may warn or use incorrect defaults
- Some Maven plugins (surefire, failsafe) may behave unexpectedly when packaging is non-`jar`

The element does NOT appear in:
- The output of `mvn io.quarkus.platform:quarkus-maven-plugin:create`
- The official Quarkus getting-started guide
- Any Quarkus quickstart project on GitHub

### Correct pom.xml structure

```xml
<project>
    <groupId>org.acme</groupId>
    <artifactId>my-app</artifactId>
    <version>1.0.0-SNAPSHOT</version>
    <!-- No <packaging> element — defaults to jar, which is correct -->

    <build>
        <plugins>
            <plugin>
                <groupId>io.quarkus.platform</groupId>
                <artifactId>quarkus-maven-plugin</artifactId>
                <version>${quarkus.platform.version}</version>
                <extensions>true</extensions>  <!-- This handles the custom lifecycle -->
                <executions>
                    <execution>
                        <goals>
                            <goal>build</goal>
                            <goal>generate-code</goal>
                            <goal>generate-code-tests</goal>
                        </goals>
                    </execution>
                </executions>
            </plugin>
        </plugins>
    </build>
</project>
```

If you see `<packaging>quarkus</packaging>` in an existing project, remove it — the build is unaffected and tooling compatibility improves.

### Why it's not obvious

The element looks intentional — Quarkus has a custom build lifecycle, so a custom packaging type makes semantic sense. Developers unfamiliar with how Maven plugin extensions work assume it's needed. The Quarkus documentation does not explicitly say "do NOT add `<packaging>quarkus</packaging>`" — its absence from the generated scaffold is the only signal, and that requires knowing what the scaffold generates.
