### Symptom
`ConfigValidationException: SRCFG00050: <property> does not map to any root` at Quarkus test startup. App starts fine when the properties are absent from `application.properties`.

### Context
`@ConfigMapping(prefix = "foo")` interface lives in a library JAR (not the main Quarkus app module). `quarkus.index-dependency.*` configured, Jandex plugin generates `META-INF/jandex.idx`. Properties with prefix `foo.*` exist in `application.properties`.

### What was tried (didn't work)
- Adding `quarkus.index-dependency.*` alone — CDI beans discovered, config validation still fails
- Adding Jandex Maven plugin to the library module — same result
- Both together — same result

### Root cause
SmallRye Config's strict validation fires during config source loading, BEFORE the library JAR's `@ConfigMapping` is registered as a config root. Properties with no registered root trigger SRCFG00050 immediately. The mapping IS registered eventually (Jandex + index-dependency do work for CDI bean discovery), but strict validation outruns registration.

### Fix
Remove the config properties from `application.properties` until the integration is fully confirmed. The `@ConfigMapping` CDI injection works fine when no properties exist to validate — beans inject with empty/default values. Add the properties back only once the full wiring is tested.

Alternatively: put the properties under a profile excluded from tests (`%dev.foo.bar=x`), or check if `quarkus.config.mapping.validate-unknown=false` is supported in your Quarkus version.

### Why this is non-obvious
Every other Quarkus CDI mechanism in library JARs (beans, producers, observers) works cleanly with `quarkus.index-dependency`. Config mapping looks identical but has a different timing dependency on strict validation. The error "does not map to any root" implies the mapping is missing entirely — not that it's registered too late.
