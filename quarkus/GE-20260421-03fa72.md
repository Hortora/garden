## Symptom

An optional module (`quarkus-workitems-filter-registry`) defines `@ApplicationScoped` CDI beans
and REST resources. Its own `@QuarkusTest` suite passes — all beans are found and injected.

A consuming module (`quarkus-workitems-ai`) that depends on filter-registry via Maven compile dep
fails at startup with CDI `UnsatisfiedResolutionException` — the beans from filter-registry
are not discovered. No error indicates the index is missing.

## Context

The filter-registry module's tests run against the **current module's compiled classes** directly,
which Quarkus augmentation scans without an index. When the module is packaged as a JAR and
consumed by another module, Quarkus augmentation reads the JAR and needs a Jandex index
(`META-INF/jandex.idx`) inside it to discover CDI beans. Without the index, Quarkus skips
the JAR silently.

## What was tried

- Checked the consuming module's pom.xml for missing deps — deps were correct
- Tried `@QuarkusTest` with explicit bean declarations — same failure
- Added `quarkus.index-dependency.<artifactId>.*` config — works but per-consumer boilerplate

## Root cause

Quarkus only auto-indexes the current application's classes. Library JARs on the classpath
are only indexed if they contain a pre-built `META-INF/jandex.idx`, unless the consuming
app explicitly forces indexing via `quarkus.index-dependency`. The filter-registry module
had no `jandex-maven-plugin` configured, so it produced no index in its JAR.

## Fix

Add `jandex-maven-plugin` to the optional module's `pom.xml`:

```xml
<plugin>
  <groupId>io.smallrye</groupId>
  <artifactId>jandex-maven-plugin</artifactId>
  <version>3.3.1</version>
  <executions>
    <execution>
      <id>make-index</id>
      <goals>
        <goal>jandex</goal>
      </goals>
    </execution>
  </executions>
</plugin>
```

This writes `META-INF/jandex.idx` into the JAR during `package`. Any consuming Quarkus
application will then discover the module's CDI beans automatically.

## Why this is non-obvious

The module's own tests pass, giving false confidence. The failure only manifests when the
module is consumed as a compiled JAR by a different module — a deployment-time scenario
not exercised by unit or integration tests within the module itself. Quarkus docs mention
Jandex for extensions but don't clearly document it as a requirement for plain library modules.
