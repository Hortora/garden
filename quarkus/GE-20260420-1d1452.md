## What

Declare JPQL queries as `@NamedQuery` annotations on the entity class. Hibernate validates all named queries at application startup — any syntax error, unknown entity name, or non-existent field fails the boot, not the first query execution.

```java
@Entity
@Table(name = "ledger_attestation")
@NamedQuery(
    name = "LedgerAttestation.findByEntryId",
    query = "SELECT a FROM LedgerAttestation a WHERE a.ledgerEntryId = :entryId ORDER BY a.occurredAt ASC")
@NamedQuery(
    name = "LedgerAttestation.findByEntryIds",
    query = "SELECT a FROM LedgerAttestation a WHERE a.ledgerEntryId IN :entryIds")
public class LedgerAttestation { ... }
```

Repository usage:
```java
return em.createNamedQuery("LedgerAttestation.findByEntryId", LedgerAttestation.class)
        .setParameter("entryId", ledgerEntryId)
        .getResultList();
```

## Why This Is Non-Obvious

Most developers write JPQL as inline strings in repository methods. With inline strings, a typo compiles fine, passes all tests that don't exercise that path, and only fails at runtime when the specific query is first called. `@NamedQuery` moves the validation to the startup phase — any error fails the application before it accepts requests. This is standard JPA but almost universally skipped in favour of Panache or Spring Data abstractions, so practitioners rarely encounter it.

The secondary benefit: named queries are defined once on the entity and referenced by string key from any number of call sites — no string duplication, and a single source of truth for the query.

## When to use

Particularly valuable in extension libraries or modules that use plain `EntityManager` without Panache or Spring Data — where the query strings would otherwise be scattered across repository methods with no startup validation. Less valuable in applications using Panache or Spring Data, since those abstractions already type-check method names at build time.
