## Symptom

Quarkus startup fails with:

```
FlywayMigrateException: Failed to execute script V5__my_subclass_table.sql
Message: Table "LEDGER_ENTRY" not found
```

The subclass join table's `FOREIGN KEY ... REFERENCES ledger_entry (id)` runs before
the parent table exists.

## Context

When your domain subclasses a base entity provided by a Quarkus extension (e.g.
extending `LedgerEntry` from `quarkus-ledger`), you write your own Flyway migration
to create the subclass join table. Flyway picks up migrations from **all** JARs on
the classpath — your app's `db/migration/` and the extension's `db/migration/`. It
sorts them all by version number globally before running anything.

If the extension reserves V1000–V1001 for its base schema and you number your
subclass migration V5, Flyway runs: V1, V5 (FK fails), V1000... The parent table
doesn't exist when V5 tries to create the FK.

## What was tried

Naming the subclass migration `V5__order_ledger_entry.sql` — natural choice since
the domain table is V1 and V5 feels like "after the domain setup". Fails silently
until Quarkus starts up.

## Root cause

Flyway merges migration scripts from all classpath locations into a single list and
sorts by version number. There is no concept of "this JAR's migrations run first" —
version number is the only ordering mechanism. Extension migrations numbered V1000+
run *after* any application migration numbered V1–V999.

## Fix

Number the subclass join table migration **above the extension's highest base migration**.
For `quarkus-ledger` (V1000 = base schema, V1001 = actor_trust_score):

```sql
-- V1002__order_ledger_entry.sql   ← NOT V5__...
CREATE TABLE order_ledger_entry (
    id       UUID NOT NULL,
    order_id UUID NOT NULL,
    -- ... domain-specific columns
    CONSTRAINT pk_order_ledger_entry PRIMARY KEY (id),
    CONSTRAINT fk_order_ledger_entry FOREIGN KEY (id) REFERENCES ledger_entry (id)
);
```

Keep domain-only tables (no FK to extension tables) in V1–V999. Only subclass join
tables — those with a FK into the extension's schema — need to be numbered above
the extension's range.

## Why this is non-obvious

Flyway's documentation describes classpath scanning but doesn't highlight that
version ordering is global across all classpath sources. A developer familiar with
Flyway knows migrations merge from multiple locations but naturally numbers their
own scripts sequentially from V1 without considering the extension's reserved range.
The error message `Table "LEDGER_ENTRY" not found` looks like the extension isn't
on the classpath at all, not like a migration ordering problem.
