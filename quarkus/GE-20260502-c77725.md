## Symptom

`OptimisticLockException` on `work_item` in a multi-instance test, specifically when
completing all instanceCount children in a loop:

```
OptimisticLockException: Unexpected row count (expected row count 1 but was 0)
[update work_item set ... where id=? and version=?]
for entity [work_item with id 'xxxx']
```

The failing entity is the surplus child — the one beyond `requiredCount`.

## Context

Test creates a multi-instance group with `instanceCount=3, requiredCount=2` and
iterates all 3 children through claim/start/complete. Works fine on some runs; fails
intermittently (especially under CI load with async coordinator threads active).

## What was tried

- Added parentId filtering and `Awaitility.during()` stability windows — reduced
  failures but did not eliminate them
- Suspected inter-test contamination, missing `em.detach()`, or Hibernate session sharing
- None of these were the cause

## Root cause

`MultiInstanceSpawnService.createGroup()` sets:

```java
group.onThresholdReached = template.onThresholdReached != null
        ? template.onThresholdReached
        : OnThresholdReached.CANCEL.name();  // ← defaults to CANCEL
```

When `requiredCount` children complete and hit the threshold, `MultiInstanceGroupPolicy.resolve()`
calls `cancelRemainingChildren()`. This cancels child[2] asynchronously on the coordinator
thread. The test's `inTx(() -> workItemService.complete(child[2], "a", "ok"))` then:

1. Loads child[2] at version V (IN_PROGRESS)
2. Coordinator commits: child[2] → CANCELLED, version V+1
3. Test's `persistAndFlush()` tries to update with version V → OCC

## Fix

Only complete `requiredCount` children in the test — not all `instanceCount`:

```java
// Complete only requiredCount (2), not all instanceCount (3)
for (final UUID c : children.subList(0, 2)) {
    inTx(() -> workItemService.claim(c, "a"));
    inTx(() -> workItemService.start(c, "a"));
    inTx(() -> workItemService.complete(c, "a", "ok"));
}
```

The threshold fires on requiredCount completions. Attempting to complete the surplus
child races with the coordinator's CANCEL behaviour.

## Why this is non-obvious

The OCC error looks like a generic concurrency bug — two threads fighting over the same
WorkItem. The `work_item` entity identified in the stack trace could be any child. The
actual cause is a business-logic default (`onThresholdReached = CANCEL`) that triggers
asynchronous cancellation of the very entity the test is trying to complete. Nothing in
the error message hints at the CANCEL default.
