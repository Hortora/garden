### What it does
`quarkus.arc.exclude-types` in `application.properties` prevents specific classes from being registered as CDI beans, even if they are `@ApplicationScoped` and on the classpath from a library JAR. The classes remain on the classpath (compilable, injectable by type if you handle it manually) — only their CDI bean registration is suppressed.

```properties
# Suppress two conflicting CDI beans from a library JAR
# while keeping their JPA entities usable for JPQL queries
quarkus.arc.exclude-types=io.casehub.ledger.repository.CaseLedgerEntryRepository,\
                           io.casehub.ledger.service.CaseLedgerEventCapture
```

### When to use it
When two libraries on the classpath both expose `@ApplicationScoped` implementations of the same CDI interface (e.g. both provide `LedgerEntryRepository`), causing an `AmbiguousResolutionException` at startup. Excluding the conflicting bean from one library removes the ambiguity without forking or patching the library.

Also useful when a library ships a CDI bean that injects something unavailable in your app (e.g. a default `EntityManager` qualifier when your app only has named PUs) — exclude the bean, use only the library's entity classes directly.

### Why it's undocumented
The property exists in Quarkus ArC docs but is described as a build-time optimisation (removing unused beans to reduce binary size). Its use as a **conflict resolution mechanism** for multi-library classpath clashes is not documented anywhere. The CDI entity-vs-bean distinction (excluding the bean but keeping the entity class for JPQL) is not mentioned at all.

### Caveats
- Excluded beans are permanently gone from the CDI container — any injected field of that type will cause `UnsatisfiedResolutionException`
- Does not affect JPA entity scanning — a class annotated `@Entity` that is also `@ApplicationScoped` can have its CDI registration suppressed while remaining managed by Hibernate
- Use `quarkus.arc.exclude-types` (plural) — singular form does not exist
