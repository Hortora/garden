## Uni.join().all(list).andFailFast() — ordered parallel reactive collection in Mutiny

### Context

`ReactiveQhorusMcpTools.listChannels()` needed to fetch a message count per channel reactively. The naive approach would be sequential — one count per channel one after another. The reactive approach needs to be parallel but maintain the original channel list order for the output.

### Technique

```java
List<Uni<ChannelDetail>> unis = channels.stream()
        .map(ch -> messageStore.countByChannel(ch.id)
                .map(count -> toChannelDetail(ch, count.longValue())))
        .toList();
return Uni.join().all(unis).andFailFast();
```

`Uni.join().all(list).andFailFast()` subscribes to all Unis in the list concurrently and returns a `Uni<List<T>>` where:
- Results are in the **same order as the input list** (not arrival order)
- If any `Uni` fails, the combined `Uni` fails immediately (`andFailFast`)
- All subscriptions fire concurrently — parallel execution

Compare to `Multi.createFrom().iterable(...).onItem().transformToUniAndMerge(...)` which also runs in parallel but does NOT preserve order.

For sequential (order-preserving but one-at-a-time): use `transformToUniAndConcatenate`.

### Why this is non-obvious

The Mutiny docs describe `transformToUniAndMerge` (parallel, unordered) and `transformToUniAndConcatenate` (sequential, ordered) but `Uni.join().all().andFailFast()` for ordered-parallel results from a pre-built list of Unis is not prominently documented. Most developers reach for `Multi` when they need parallel operations, missing the simpler `Uni.join()` pattern.
