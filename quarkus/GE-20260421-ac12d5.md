## Uni.join().all(list).andFailFast() — ordered parallel reactive collection in Mutiny

### Context

`ReactiveQhorusMcpTools.listChannels()` needed to fetch a message count per channel reactively. The naive approach would be sequential — one count per channel one after another. The reactive approach needs to be parallel but maintain the original channel list order for the output.

### Technique

```java
List<Uni<ChannelDetail>> unis = channels.stream()
        .map(ch -> messageStore.countByChannel(ch.id)
                .map(count -> toChannelDetail(ch, count.longValue())))
        .toList();
return Uni.join().all(unis).andFailFast();
```

`Uni.join().all(list).andFailFast()` subscribes to all Unis in the list concurrently and returns a `Uni<List<T>>` where:
- Results are in the **same order as the input list** (not arrival order)
- If any `Uni` fails, the combined `Uni` fails immediately (`andFailFast`)
- All subscriptions fire concurrently — parallel execution

Compare to `Multi.createFrom().iterable(...).onItem().transformToUniAndMerge(...)` which also runs in parallel but does NOT preserve order.

For sequential (order-preserving but one-at-a-time): use `transformToUniAndConcatenate`.

### Why this is non-obvious

The Mutiny docs describe `transformToUniAndMerge` (parallel, unordered) and `transformToUniAndConcatenate` (sequential, ordered) but `Uni.join().all().andFailFast()` for ordered-parallel results from a pre-built list of Unis is not prominently documented. Most developers reach for `Multi` when they need parallel operations, missing the simpler `Uni.join()` pattern.

## Variant — null-slot pattern for partial-failure collection

When you need ordered parallel execution AND want to collect per-item failures instead of aborting on the first failure, catch exceptions inside each Uni lambda and return `null` for failures. `andFailFast()` never fires (no Uni fails), and the result list preserves input order — nulls mark failed slots.

```java
// Each Uni catches its own exception and records the failure externally.
// andFailFast() is harmless: nothing fails because exceptions are caught.
final var failures = new CopyOnWriteArrayList<StoreFailure>();
final List<Uni<String>> unis = new ArrayList<>();
for (int i = 0; i < inputs.size(); i++) {
    final int idx = i;
    final MemoryInput input = inputs.get(i);
    unis.add(Uni.createFrom().<String>item(() -> {
        try { return backend.store(input); }
        catch (RuntimeException e) {
            failures.add(new StoreFailure(idx, input, e));
            return null;          // null slot = failure at position idx
        }
    }).runSubscriptionOn(Infrastructure.getDefaultWorkerPool()));
}
// Result is ordered (matches input positions); nulls are failures.
List<String> rawIds = Uni.join().all(unis).andFailFast().await().indefinitely();
List<String> stored  = rawIds.stream().filter(id -> id != null).toList();
// failures list carries inputIndex for retry correlation
```

**When to use:** REST/queue backends where partial writes are inherent and callers need to identify which inputs succeeded without retrying successful ones. Not appropriate for transactional backends (JPA/SQLite) where all-or-nothing semantics are enforced at the transaction level.

**CopyOnWriteArrayList is required** because multiple worker threads concurrently append to `failures`. Regular ArrayList would corrupt under concurrent writes.
