## Symptom

A scheduled job starts without error in JVM mode but at native image startup throws `DateTimeParseException` or the interval is not resolved — the scheduler uses the literal string `{property}` as the interval value.

## Context

Config-driven `@Scheduled` intervals, such as making the poll frequency configurable.

## What Was Tried

```java
@Scheduled(every = "{quarkus.myext.cleanup.check-seconds}s")
void checkExpired() { ... }
```

## Root Cause

Quarkus scheduler resolves config property references using MicroProfile Config expression syntax (`${...}`), not bare brace syntax (`{...}`). Brace-only expressions are silently not resolved at augmentation time — they may pass as a literal string or throw at startup.

## Fix

```java
@Scheduled(every = "${quarkus.myext.cleanup.check-seconds}s")
void checkExpired() { ... }
```

## Why This Is Non-Obvious

The Quarkus config system uses `${...}` extensively, but some parts of the documentation show examples with bare `{...}` notation. The bare form is silently ignored rather than causing an obvious compile-time or boot-time error in JVM mode, making the bug easy to miss until native testing.
