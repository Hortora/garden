## Symptom

`Entity.find("ORDER BY createdAt ASC").list()` returns an empty list even though rows exist. No exception is thrown.

## Context

Panache's `find(String query)` interprets the string as a **WHERE clause fragment**, not a full JPQL query. Passing `"ORDER BY field ASC"` with no WHERE condition results in a malformed query like `WHERE ORDER BY field ASC` — which returns no results and does not throw.

## Root cause

Panache's `find()` prepends `WHERE` to whatever string you pass. A pure `ORDER BY` clause becomes syntactically invalid in that position, but the H2 (and PostgreSQL) query parser silently returns zero rows rather than raising an error.

## Fix

Use `findAll(Sort)` for ordering without a WHERE clause:

```java
// WRONG — treated as WHERE clause, returns empty
Entity.find("ORDER BY expiresAt ASC").list()

// RIGHT — explicit sort API
Entity.findAll(Sort.ascending("expiresAt")).list()

// Also right — include a real WHERE clause
Entity.find("1=1 ORDER BY expiresAt ASC").list()
```

## Why this is non-obvious

The Panache API makes `find(query)` look like a JPQL query method. Most Panache examples show WHERE fragments as the argument. It's natural to assume you can pass any valid JPQL fragment — but the silent empty result (no exception) hides the real issue entirely.
