## `check_messages` excludes EVENT messages by design — tests expecting EVENTs via check_messages always get fewer results than sent

**ID:** GE-20260414-23982b
**Stack:** quarkus-qhorus
**Labels:** `#messaging` `#events` `#testing` `#tdd` `#quarkus-qhorus`

### Symptom

Test sends 5 messages to an APPEND channel (3 agent messages + 2 EVENTs), then asserts:

```java
QhorusMcpTools.CheckResult result = tools.checkMessages("channel", 0L, 20, null);
assertEquals(5, result.messages().size()); // FAILS — gets 3
```

### Context

Writing E2E tests for an observer feature. Agents produce messages; observers consume EVENTs. Test verified "all messages are stored" by counting via `check_messages`, expecting EVENTs to be included.

### Root cause

`MessageService.pollAfter()` filters out `MessageType.EVENT` by design — the invariant is "agent context is never polluted with telemetry". All `check_messages` semantic paths (APPEND, COLLECT, BARRIER, EPHEMERAL, LAST_WRITE) route through this filter. EVENTs are stored in the database but excluded from every `check_messages` code path.

EVENTs are only accessible via the dedicated observer read path (`read_observer_events`) or by querying the database directly in tests.

### Fix

Use `read_observer_events` to assert EVENT delivery; use non-EVENT counts for agent-visible message assertions:

```java
// Assert agent-visible messages (3, not 5)
QhorusMcpTools.CheckResult agentResult = tools.checkMessages("channel", 0L, 20, null);
assertEquals(3, agentResult.messages().size());

// Assert EVENT delivery (observer path)
tools.registerObserver("test-obs", List.of("channel"));
QhorusMcpTools.CheckResult obsResult = tools.readObserverEvents("test-obs", "channel", 0L, 20);
assertEquals(2, obsResult.messages().size());
```

### Why this is non-obvious

`check_messages` looks like a general-purpose message reader. The EVENT exclusion is an architectural invariant ("telemetry never pollutes agent context") that is obvious once you know it but not apparent from the method signature. Tests that mix agent and observer scenarios are the most likely to hit this — the count mismatch is the symptom.
