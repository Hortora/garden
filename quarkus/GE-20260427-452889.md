### Symptom

`"Channel not found: X"` (or equivalent entity-not-found error) thrown from inside the **test method body**, even though `@BeforeEach` successfully created the entity. The `@BeforeEach` appears to complete without error.

```
java.lang.IllegalArgumentException: Channel not found: my-channel
    at QhorusMcpTools.lambda$getTelemetrySummary$0(QhorusMcpTools.java:1507)
```

### Context

`@QuarkusTest @TestTransaction` test class. `@BeforeEach` calls `@Transactional(REQUIRED)` service methods (e.g. `createChannel`, `sendMessage`). One of those service methods internally triggers a `@Transactional(REQUIRED_NEW)` sub-transaction (e.g. `LedgerWriteService.record()` which uses `REQUIRES_NEW` to ensure ledger durability regardless of outer rollback). The test method then calls another service that looks up the same entity created in `@BeforeEach`.

### What was tried

Standard JUnit 5 `@BeforeEach` pattern to share scenario setup across multiple `@Test` methods. The pattern looked correct — `@TestTransaction` is documented to start a transaction before each test and roll it back after. The channel creation seemed to work fine in isolation.

### Root cause

`@TestTransaction` wraps both `@BeforeEach` **and** the test method body in a single outer JTA transaction. When a `REQUIRES_NEW` method is called from within `@BeforeEach`, the JTA transaction manager **suspends** the outer `@TestTransaction` transaction, commits the REQUIRES_NEW work, then **resumes** the outer transaction.

The JPA `EntityManager` bound to the outer `@TestTransaction` is not reattached cleanly after the suspend/resume cycle. Entities that were persisted in the outer transaction (written to the first-level cache but not flushed to the DB) become **invisible** to subsequent JPQL queries — the EntityManager's first-level cache is either invalidated or the flush-before-query logic is disrupted. As a result, the entity lookup in the test method body returns empty even though the entity exists in the transaction's write-behind queue.

### Fix

**Move all setup (entity creation + message sending) inside the `@Test` method body**, not in `@BeforeEach`.

```java
// ❌ WRONG — REQUIRES_NEW in sendMessage corrupts outer @TestTransaction
@BeforeEach
void driveScenario() {
    tools.createChannel("my-channel", ...);
    tools.sendMessage("my-channel", ...); // triggers REQUIRES_NEW internally
}

@Test
void someTest() {
    tools.getTelemetrySummary("my-channel"); // "Channel not found: my-channel"
}

// ✅ CORRECT — all setup inside @Test, single contiguous transaction
@Test
void someTest() {
    tools.createChannel("my-channel", ...);
    tools.sendMessage("my-channel", ...);
    tools.getTelemetrySummary("my-channel"); // works
}
```

For tests that share setup across many test methods, extract a `private void setup(...)` helper and call it from each `@Test` method body.

### Why this is non-obvious

The error message ("Channel not found") points to the entity lookup in the test method — nothing suggests the problem is in `@BeforeEach`. The `@BeforeEach` completes without exception. `@TestTransaction` is documented as "starts a transaction before each test" with no mention of `REQUIRES_NEW` interactions. The natural instinct to share setup in `@BeforeEach` is standard JUnit practice. The actual cause — JTA suspend/resume disrupting JPA first-level cache visibility — requires knowing Quarkus internals to diagnose.

### Why this fix

All operations stay within a single contiguous JTA transaction context. No suspend/resume cycle occurs during setup, so the EntityManager's first-level cache remains coherent for subsequent entity lookups. The `REQUIRES_NEW` sub-transactions still commit independently (ledger durability is preserved), but the outer transaction's EntityManager is never suspended between setup and assertion.

### Alternatives considered

- `@BeforeAll` (static): not viable because `@TestTransaction` is per-method and static methods can't join a JTA transaction
- Injecting `UserTransaction` and managing it manually in `@BeforeEach`: works but requires explicit begin/commit/rollback boilerplate
- Using H2 in-memory DB with `drop-and-create` between tests: too slow for a large test class; schema recreation adds 2–3s per test
