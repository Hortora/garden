## Symptom

Flaky test failures on `assertThat(list).hasSize(1)` — intermittently reports `hasSize(2)`
or more. The test uses a shared `@ApplicationScoped` event-capture bean that is
`clear()`ed in `@BeforeEach`.

## Context

`WorkItemGroupLifecycleEventTest` used an `@ApplicationScoped EventCapture` bean
with a `clear()` method called in `@BeforeEach`. One test (`completedEventFiresExactlyOnce`)
generates multiple `IN_PROGRESS` events before the `COMPLETED` event. Async delivery
of those events sometimes happened after the next test's `@BeforeEach clear()` ran.

## What was tried

Increasing Awaitility timeout — no effect. Adding a `Thread.sleep()` before assertions — reduced
frequency but didn't eliminate. Moving `clear()` to `@AfterEach` — same problem in reverse.

## Root cause

`@ObservesAsync` delivers events on a CDI-managed thread pool. Delivery timing is
non-deterministic relative to JUnit's `@BeforeEach` lifecycle. When test N generates
multiple async events, some may still be in-flight when test N+1's `@BeforeEach` runs.
`clear()` executes, then the in-flight events from test N arrive, polluting the capture
list for test N+1.

## Fix

Filter the capture by entity ID (each test creates a unique entity):

```java
// Before: shared list cleared in @BeforeEach — race with in-flight deliveries
boolean hasStatus(GroupStatus s) {
    synchronized (events) {
        return events.stream().anyMatch(e -> e.groupStatus() == s);
    }
}

// After: filter by parentId — test N+1's capture only sees its own entity's events
boolean hasStatus(UUID parentId, GroupStatus s) {
    synchronized (events) {
        return events.stream()
                .anyMatch(e -> e.groupStatus() == s && parentId.equals(e.parentId()));
    }
}

// In the test:
UUID parentId = createEntity(); // unique per test
Awaitility.await().until(() -> capture.hasStatus(parentId, GroupStatus.COMPLETED));
assertThat(capture.byStatus(parentId, GroupStatus.COMPLETED)).hasSize(1);
```

Keep `clear()` in `@BeforeEach` as a secondary memory bound, but don't rely on it for
test isolation.

## Why this is non-obvious

`@BeforeEach` feels like the right place to reset state. The `clear()` method appears to
work — it clears the list. The race only shows up when the thread pool delivers events
from the previous test slower than usual, making it intermittent and hard to reproduce
on demand. Filtering by entity ID is a stronger isolation guarantee than timing-based resets.
