## Symptom

A CDI event observer (`@Observes WorkItemLifecycleEvent`) triggers an action (e.g. adding a label
via `WorkItemService.addLabel()`) which fires a new `WorkItemLifecycleEvent`. The observer
re-enters, processes the new event, and if that action also fires an event, you get infinite
recursion. In practice: stack overflow or silent loop until transaction timeout.

## Context

Occurred in `FilterRegistryEngine` — an observer that evaluates filter rules and applies
actions (labels, candidateGroups overrides, priority changes) when WorkItems are created/updated.
Actions that call `WorkItemService` methods trigger new lifecycle events, which re-enter
the engine.

## What was tried

- Checking event type in the observer to skip "derived" events — fragile: labelling uses
  a different event name but the engine has no clean way to distinguish "primary" from "derived"
- Using `@Priority` annotations — doesn't prevent re-entry, just controls ordering

## Root cause

Jakarta CDI synchronous `@Observes` is reentrant by design — calling `Event.fire()` from
within an observer fires synchronously and re-enters all observers of that event type.
There is no built-in reentrancy guard.

## Fix

Use a `ThreadLocal<Boolean>` guard in the observer:

```java
@ApplicationScoped
public class FilterRegistryEngine {

    private static final ThreadLocal<Boolean> RUNNING =
        ThreadLocal.withInitial(() -> false);

    void onLifecycleEvent(@Observes final WorkItemLifecycleEvent event) {
        if (Boolean.TRUE.equals(RUNNING.get())) {
            return;  // prevent recursive firing from action-triggered events
        }
        RUNNING.set(true);
        try {
            // evaluate filters and apply actions
        } finally {
            RUNNING.remove();  // remove (not set(false)) to avoid thread-local leak
        }
    }
}
```

Use `RUNNING.remove()` in finally (not `set(false)`) to avoid memory leaks in pooled-thread
environments like Quarkus worker threads.

## Why this is non-obvious

CDI's event model is fire-and-forget for async observers but synchronous and reentrant
for `@Observes`. Developers expect that firing an event "schedules" it, not that it
recursively calls the current observer immediately. The Jakarta CDI spec documents
synchronous delivery but the re-entry implication is easy to miss.
