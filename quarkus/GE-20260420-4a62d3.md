## Symptom

`@QuarkusTest` fails with `ClassSelector resolution failed` for the test class name, or Quarkus
cannot resolve SPI implementations (`@ApplicationScoped` in-memory repositories not found) even
though the module is on the test classpath.

## Context

A new Maven module needs in-memory SPI implementations (e.g. `InMemoryEventLogRepository`) for
`@QuarkusTest` integration tests. The implementations live in a sibling module
(`casehub-persistence-memory`). Adding that module as a `<scope>test</scope>` Maven dependency
appears correct but doesn't work.

## Root Cause

Quarkus's test class loader uses a special isolated class loading mechanism that doesn't honour
transitive Maven test dependencies the same way standard JUnit does. Classes that need to
participate in CDI discovery (including `@Alternative` beans and SPI implementations) must be
present in the module's own source tree, not in a dependency jar, for Quarkus's build-time
Jandex scan to pick them up.

## Fix

Copy the source files directly into the test module's `src/test/java/` tree under the same
package:

```bash
cp casehub-persistence-memory/src/main/java/io/casehub/persistence/memory/InMemory*.java \
   my-module/src/test/java/io/casehub/persistence/memory/
```

This is the same pattern the `engine` module uses (from PR3). Remove the Maven test dependency
— it's no longer needed.

## Why This Is Non-Obvious

Adding a Maven test dependency is the standard Java approach and works for plain JUnit. The
failure message (`ClassSelector resolution failed`) gives no indication that the class is found
but Quarkus's Jandex scan didn't index it. The correct fix (copying sources) feels wrong — it
violates DRY — but it's the established pattern in this codebase.
