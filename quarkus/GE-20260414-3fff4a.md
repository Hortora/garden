## Call Entity.getEntityManager().clear() after JPQL bulk updates in @QuarkusTest to see DB state

**ID:** GE-20260414-3fff4a
**Stack:** Quarkus 3.32.2, Hibernate ORM 6.x, Panache active record, JUnit 5

### What the technique does

After a JPQL bulk `UPDATE` or `DELETE` statement within a `@QuarkusTest @TestTransaction` test, call `PanacheEntityBase.getEntityManager().clear()` to evict all entities from Hibernate's first-level cache. The next `Entity.find()` will then query the database and return the post-update state.

### The problem it solves

JPQL bulk updates (`Entity.update(...)`) bypass Hibernate's first-level cache — they execute directly as SQL but don't update in-memory entity objects. After a bulk update, any entities already loaded in the session still hold their old values. A subsequent `Entity.find("instanceId", id)` returns the cached entity, not the updated one, making the test assert on stale data.

Most developers diagnose this as "the update isn't working" or "the transaction isn't committing" and spend time on those false trails.

### How to apply it

```java
@Test
@TestTransaction
void markStaleChangesStatusForOldInstances() throws InterruptedException {
    instanceService.register("stale-agent", "Will go stale", List.of());
    Thread.sleep(1100);

    // Panache JPQL bulk update — runs SQL directly, bypasses first-level cache
    instanceService.markStaleOlderThan(1);

    // Without this, find() returns the cached entity with status="online"
    Instance.getEntityManager().clear();

    Instance inst = instanceService.findByInstanceId("stale-agent").orElseThrow();
    assertEquals("stale", inst.status);  // now reads from the database
}
```

`PanacheEntityBase.getEntityManager()` is a static method available on all Panache active-record entities. It returns the current transaction's EntityManager. Calling `.clear()` on it evicts the entire first-level cache (all entity types), so any subsequent queries go to the database.

### Why developers wouldn't naturally reach for this

- Panache hides the EntityManager behind a clean API; most developers never interact with it directly
- The Quarkus/Panache documentation doesn't document `getEntityManager()` as a testing utility
- The failure mode is silent: the test passes with the wrong assertion, not fails with an error
- The obvious diagnostic (checking whether the service method works) confirms it does — the DB has the right value; only the test's in-memory view is wrong
