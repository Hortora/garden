## Symptom

Three `@TestProfile`-annotated test classes (`A2AGetTaskTest`, `WatchdogEnabledTest`, `SmokeTest`) fail with Quarkus boot failure after migrating to a named datasource:

```
Unable to find datasource 'qhorus' for persistence unit 'qhorus':
No pool has been defined for persistence unit qhorus
```

All other `@QuarkusTest` classes pass. The named datasource is correctly configured in `src/test/resources/application.properties`.

## Context

Occurs when:
- A Quarkus project uses a named persistence unit (e.g. `quarkus.hibernate-orm.qhorus.*`)
- The named datasource config lives in `src/test/resources/application.properties`
- Some test classes use `@TestProfile` causing Quarkus to restart the application context

## What was tried (didn't work)

Confirming `quarkus.datasource.qhorus.*` was present in test `application.properties` ã it was. Tests without `@TestProfile` passed. Only `@TestProfile` tests failed.

## Root cause

When `@TestProfile` causes Quarkus to restart the application context within the same test run, the restarted instance does NOT re-read `src/test/resources/application.properties` from the classpath in the same way as the initial boot. Only config provided via `QuarkusTestProfile.getConfigOverrides()` is guaranteed to be applied to the restarted context.

## Fix

Every `@TestProfile` implementation that causes a Quarkus restart must include the full named datasource block in `getConfigOverrides()`:

```java
@Override
public Map<String, String> getConfigOverrides() {
    Map<String, String> config = new HashMap<>();
    config.put("quarkus.qhorus.a2a.enabled", "true");          // module flag
    config.put("quarkus.datasource.qhorus.db-kind", "h2");
    config.put("quarkus.datasource.qhorus.jdbc.url",
        "jdbc:h2:mem:test;DB_CLOSE_DELAY=-1");
    config.put("quarkus.datasource.qhorus.username", "sa");
    config.put("quarkus.datasource.qhorus.password", "");
    config.put("quarkus.datasource.qhorus.reactive=false", "false");
    config.put("quarkus.hibernate-orm.qhorus.database.generation", "drop-and-create");
    return config;
}
```

## Why this is non-obvious

Quarkus documentation says test `application.properties` is read for `@QuarkusTest`. It doesn't document that `@TestProfile` restarts apply config overrides on top of the *base compiled config*, not on top of the *test properties file*. The natural assumption is that both are always read; that assumption is wrong for restart scenarios.
