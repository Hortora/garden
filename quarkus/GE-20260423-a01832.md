## Symptom

`@Alternative @Priority(1)` beans from a `test`-scoped dependency are not selected by CDI in `@QuarkusTest` runs. The default (non-alternative) beans are used instead. No error is thrown — the wrong beans silently win.

## Context

Occurs when a library ships a `-testing` artifact (e.g. `quarkus-mylib-testing`) containing `@Alternative` in-memory implementations intended for use in consumer test suites. Adding the artifact as a `<scope>test</scope>` dependency is not sufficient.

## Root Cause

Quarkus CDI only scans jars that are explicitly indexed. Third-party jars without a Jandex index (or without explicit `quarkus.index-dependency` config) are not scanned for CDI beans, even if the jar is on the classpath and the beans are annotated correctly.

## Fix

Add an index-dependency entry in `src/test/resources/application.properties`:

```properties
quarkus.index-dependency.mylib-testing.group-id=io.example
quarkus.index-dependency.mylib-testing.artifact-id=quarkus-mylib-testing
```

Adjust `group-id` and `artifact-id` to match the testing jar's Maven coordinates.

## Why This Is Non-Obvious

The dependency is on the classpath. The beans have the correct `@Alternative @Priority(1)` annotations. No exception is thrown. The library's documentation typically says "add the testing dependency" without mentioning the index requirement. The failure mode — wrong beans used silently — is identical to not having the dependency at all, which misleads diagnosis toward annotation or scope issues.
