## The technique

When building REST endpoints that expose the same data as MCP tools, inject the MCP tool bean directly and call its methods rather than writing new persistence queries.

```java
@Path("/api/mesh")
@Authenticated
public class MeshResource {

    @Inject
    QhorusMcpTools qhorusMcpTools;   // ← inject the @Tool bean directly

    @GET
    @Path("/channels")
    public List<QhorusMcpTools.ChannelDetail> channels() {
        return qhorusMcpTools.listChannels();   // reuses N+1-safe batch query
    }

    @GET
    @Path("/instances")
    public List<QhorusMcpTools.InstanceInfo> instances() {
        return qhorusMcpTools.listInstances(null);
    }
}
```

The `@Tool` methods return well-typed records (`ChannelDetail`, `InstanceInfo`) that are already Jackson-serialisable. No DTO mapping needed.

## Why this is non-obvious

The instinct is to inject the Panache repository or entity class directly and write a fresh query. The `@Tool` bean feels like "MCP infrastructure" rather than a callable service. But because it is `@ApplicationScoped` and CDI-managed, it is fully injectable and its methods work identically whether called from an MCP request or a REST handler.

The N+1 optimisations inside `@Tool` methods (batch GROUP BY queries, eager joins) are reused for free. Writing equivalent queries from scratch in a REST resource duplicates logic and is easy to get wrong.

## Trade-off

`@Tool` methods that return `List<Map<String,Object>>` (e.g. `getChannelTimeline`, `listEvents`) expose the MCP tool's serialisation format to the REST client. If the MCP format changes, the REST response changes. Acceptable for v1; introduce typed DTOs when the REST contract needs to be stable independently of the MCP tool format.
