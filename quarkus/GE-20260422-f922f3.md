## Symptom

`mvn test` hangs indefinitely during Quarkus augmentation when `quarkus-langchain4j-core` is a compile-scope dependency of a non-extension library module and no AI model provider is configured. No error message, no timeout — the build just blocks.

## Context

Occurs when a library module (Jandex-indexed, not a full Quarkus extension) adds `io.quarkiverse.langchain4j:quarkus-langchain4j-core` as a compile dependency to use the `EmbeddingModel` interface. During `@QuarkusTest` augmentation, the Quarkus extension activates and waits for a model provider that never arrives.

## What was tried

Adding `quarkus-langchain4j-core` as a `provided` scope — still stalls because the extension's deployment artifact is resolved transitively during augmentation.

## Root cause

`quarkus-langchain4j-core` is a Quarkus extension (has both runtime and deployment jars). When it appears on the augmentation classpath, Quarkus activates it and expects at least one model provider (OpenAI, Ollama, etc.) to be configured. Without one, augmentation stalls waiting for the CDI bean to resolve.

## Fix

Use `dev.langchain4j:langchain4j-core` (the plain Java library, **not** the Quarkus extension) as the compile dependency. This provides the `EmbeddingModel` interface for compilation without triggering Quarkus extension activation:

```xml
<dependency>
  <groupId>dev.langchain4j</groupId>
  <artifactId>langchain4j-core</artifactId>
  <version>1.13.0</version>
</dependency>
```

For CDI injection, use `Instance<EmbeddingModel>` instead of `@Inject EmbeddingModel`:

```java
@Inject
public MyBean(final Instance<EmbeddingModel> modelInstance) {
    this.model = modelInstance.isResolvable() ? modelInstance.get() : null;
}
```

When no provider is configured, `instance.isResolvable()` returns `false` gracefully. The consuming app brings `quarkus-langchain4j-openai` (or any other provider extension), which satisfies the CDI bean at their own augmentation time.

## Why this is non-obvious

The symptom is a hang with no error — not a compilation failure or a clear exception. The natural instinct is to add the Quarkus extension for its CDI integration, but the correct separation is: library module uses the plain Java API, consuming application brings the Quarkus extension.

### Why this fix

The plain Java `langchain4j-core` provides the same `EmbeddingModel` interface without triggering extension activation. `Instance<EmbeddingModel>` makes the dependency optional at CDI resolution time, so the library works both with and without a configured provider.

### Alternatives considered

- `provided` scope: still triggers augmentation because the extension's deployment artifact is resolved transitively
- Mocking `EmbeddingModel` via `@InjectMock` in every `@QuarkusTest`: works but requires all existing tests to be updated
