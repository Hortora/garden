## What it is

`quarkus.arc.selected-alternatives` in `application.properties` (or test `application.properties`) is the correct way to activate `@Alternative` CDI beans in a Quarkus application — including in `@QuarkusTest` contexts.

```properties
# In src/test/resources/application.properties or src/main/resources/application.properties
quarkus.arc.selected-alternatives=com.example.MyAlternativeRepository
```

Multiple alternatives can be comma-separated.

## Why it's undocumented / non-obvious

The standard CDI mechanism for activating alternatives is `beans.xml`:
```xml
<beans>
  <alternatives>
    <class>com.example.MyAlternativeRepository</class>
  </alternatives>
</beans>
```

In Quarkus with Arc (the CDI implementation), **`beans.xml` alternative activation is silently ignored at build time**. Arc processes CDI at build time and does not honour the `beans.xml` `<alternatives>` section. There is no warning, no error — the `@Alternative` bean is simply not activated, and the original bean (or no bean) is used instead. This causes `UnsatisfiedResolutionException` at startup if nothing else provides the bean.

The `quarkus.arc.selected-alternatives` property is the Quarkus-specific replacement but is not prominently documented alongside the `@Alternative` annotation documentation.

## Context

Common scenario: a Quarkus extension provides an `@Alternative @ApplicationScoped` repository bean (e.g. `JpaLedgerEntryRepository`) that yields to consumer-provided implementations in production. In `@QuarkusTest` integration tests, you want to activate the base implementation since no consumer repo exists in the test module.

## Fix

Add to `src/test/resources/application.properties`:

```properties
quarkus.arc.selected-alternatives=io.example.JpaLedgerEntryRepository
```

Do NOT add `<alternatives>` to `beans.xml` — it will have no effect in Quarkus Arc.

## Why this is non-obvious

The CDI spec says `beans.xml` alternatives work. Quarkus Arc silently diverges from this for build-time optimization reasons. The property name `selected-alternatives` is not listed in the CDI or Arc extension documentation alongside `@Alternative` — it requires reading the Arc configuration reference which is separate from the CDI programming model docs.

## Update — test-local classes in src/test/java need no quarkus.index-dependency

When the `@Alternative` class lives in a **external jar** (a separate Maven module), you must tell Quarkus to index it:

```properties
quarkus.index-dependency.mymodule.group-id=io.example
quarkus.index-dependency.mymodule.artifact-id=my-alternatives-module
```

However, if the `@Alternative` class lives directly in `src/test/java/` of the same module, **no `quarkus.index-dependency` is needed**. Quarkus automatically indexes the test source root for `@QuarkusTest`. Only classes in external jars require the explicit index-dependency config.

```properties
# This is sufficient when @Alternative classes are in src/test/java/:
quarkus.arc.selected-alternatives=io.example.InMemoryRepository

# quarkus.index-dependency is NOT needed — test sources are auto-indexed
```

This is useful when avoiding Maven module cycles: copy the alternative implementations into the module's own `src/test/java/` and the `selected-alternatives` property works without any additional indexing configuration.

## Update — @Alternative @IfBuildProperty beans also need selected-alternatives (not just @QuarkusTest)

When `@Alternative` is combined with `@IfBuildProperty`, the bean is registered at build time when the property matches, but **registration alone does not make it the active CDI alternative**. The `@Alternative` selection still requires `quarkus.arc.selected-alternatives`, even in production `application.properties`.

This surfaces when activating a Quarkus extension's reactive stack via a build property. For example, Qhorus's reactive services are `@Alternative @IfBuildProperty(name = "quarkus.datasource.qhorus.reactive", stringValue = "true") @ApplicationScoped`. Setting the property in `application.properties` registers the beans at augmentation but they still do not win CDI resolution without explicit selection:

```properties
# application.properties — both properties are required to activate reactive Qhorus beans
quarkus.datasource.qhorus.reactive=true
quarkus.arc.selected-alternatives=\
  io.casehub.qhorus.runtime.channel.ReactiveChannelService,\
  io.casehub.qhorus.runtime.message.ReactiveMessageService,\
  ...
```

**The @Priority workaround:** If the bean is annotated `@Alternative @Priority(N)`, Arc selects it automatically without `selected-alternatives`. The `@Priority` annotation is the preferred mechanism for extension-provided alternatives that should activate without consumer configuration. For consumer-configured beans, `selected-alternatives` remains the explicit control point.

## Update — selected-alternatives is build-time: %test. profile prefix has no effect

`quarkus.arc.selected-alternatives` is evaluated during Quarkus augmentation (build time), not at runtime. Profile prefixes like `%test.` are resolved at runtime and therefore **have no effect** on this property.

This means you must write:
```properties
# CORRECT — no profile prefix; the property applies globally at augmentation
quarkus.arc.selected-alternatives=io.casehub.persistence.memory.MemorySubCaseGroupRepository
```

Not:
```properties
# WRONG — %test. is silently ignored for build-time properties; the alternative is not activated
%test.quarkus.arc.selected-alternatives=io.casehub.persistence.memory.MemorySubCaseGroupRepository
```

The practical consequence: if you add the prefix, your `@Alternative` bean is never selected, and the CDI context boots with the default (or fails with `UnsatisfiedResolutionException`), with no indication of why. All surrounding datasource config in the same file uses `%test.` — the omission looks like a mistake but is required.

## Update — selected-alternatives DOES NOT WORK for library beans in a Quarkus extension's own @QuarkusTest (2026-06-09)

When the project under test is itself a **Quarkus extension** (has `quarkus-extension.yaml`, deployment module, etc.) rather than a Quarkus application, `quarkus.arc.selected-alternatives` silently fails to activate `@Alternative` beans from library JARs in `@QuarkusTest` augmentation — even when:
- The library JAR has an embedded Jandex index (`META-INF/jandex.idx`)
- `quarkus.index-dependency` is explicitly configured for that artifact

The symptom is `UnsatisfiedResolutionException` for the `@Alternative` bean's interface at CDI startup. The extension's augmentation does NOT pick up the selected-alternatives property, despite the property appearing in both the main and test `application.properties`.

**Root cause (confirmed via trial):** Extension-mode `@QuarkusTest` augmentation may resolve `quarkus.arc.selected-alternatives` differently from application-mode augmentation. The exact mechanism is unknown, but the property is simply not effective in this context.

**Workaround — non-`@Alternative` subclass pattern:**

Since CDI inheritance does NOT propagate `@Alternative`, a thin `@ApplicationScoped` subclass of the library's `@Alternative` class is a non-alternative DEFAULT bean:

```java
// In the extension's own runtime module:
@ApplicationScoped
class MyLedgerEntryRepository extends JpaLedgerEntryRepository {
    // No new methods — all behaviour inherited from library's @Alternative class.
    // @ApplicationScoped WITHOUT @Alternative → becomes the DEFAULT CDI bean.
    // CDI injects all @Inject fields from the parent class on this instance.
}
```

This works because:
- The subclass is `@ApplicationScoped` but NOT `@Alternative` → it satisfies injection points as the DEFAULT bean
- CDI discovers it by scanning the extension's own classes (always indexed)
- Parent class's `@Inject` fields are injected normally on the subclass instance
- No `quarkus.arc.selected-alternatives` config required

**Limitation:** The parent class must have a no-arg constructor (or `@Inject` constructor) to allow subclassing and CDI proxying. Package-private classes cannot be subclassed from a different package.

**When to use this pattern vs selected-alternatives:** Use the subclass pattern when the consuming project is an extension and `selected-alternatives` has proven unreliable. Use `selected-alternatives` for application-mode projects where it reliably works. Refs qhorus#255.
