## What it is

`quarkus.arc.selected-alternatives` in `application.properties` (or test `application.properties`) is the correct way to activate `@Alternative` CDI beans in a Quarkus application — including in `@QuarkusTest` contexts.

```properties
# In src/test/resources/application.properties or src/main/resources/application.properties
quarkus.arc.selected-alternatives=com.example.MyAlternativeRepository
```

Multiple alternatives can be comma-separated.

## Why it's undocumented / non-obvious

The standard CDI mechanism for activating alternatives is `beans.xml`:
```xml
<beans>
  <alternatives>
    <class>com.example.MyAlternativeRepository</class>
  </alternatives>
</beans>
```

In Quarkus with Arc (the CDI implementation), **`beans.xml` alternative activation is silently ignored at build time**. Arc processes CDI at build time and does not honour the `beans.xml` `<alternatives>` section. There is no warning, no error — the `@Alternative` bean is simply not activated, and the original bean (or no bean) is used instead. This causes `UnsatisfiedResolutionException` at startup if nothing else provides the bean.

The `quarkus.arc.selected-alternatives` property is the Quarkus-specific replacement but is not prominently documented alongside the `@Alternative` annotation documentation.

## Context

Common scenario: a Quarkus extension provides an `@Alternative @ApplicationScoped` repository bean (e.g. `JpaLedgerEntryRepository`) that yields to consumer-provided implementations in production. In `@QuarkusTest` integration tests, you want to activate the base implementation since no consumer repo exists in the test module.

## Fix

Add to `src/test/resources/application.properties`:

```properties
quarkus.arc.selected-alternatives=io.example.JpaLedgerEntryRepository
```

Do NOT add `<alternatives>` to `beans.xml` — it will have no effect in Quarkus Arc.

## Why this is non-obvious

The CDI spec says `beans.xml` alternatives work. Quarkus Arc silently diverges from this for build-time optimization reasons. The property name `selected-alternatives` is not listed in the CDI or Arc extension documentation alongside `@Alternative` — it requires reading the Arc configuration reference which is separate from the CDI programming model docs.

## Update — test-local classes in src/test/java need no quarkus.index-dependency

When the `@Alternative` class lives in a **external jar** (a separate Maven module), you must tell Quarkus to index it:

```properties
quarkus.index-dependency.mymodule.group-id=io.example
quarkus.index-dependency.mymodule.artifact-id=my-alternatives-module
```

However, if the `@Alternative` class lives directly in `src/test/java/` of the same module, **no `quarkus.index-dependency` is needed**. Quarkus automatically indexes the test source root for `@QuarkusTest`. Only classes in external jars require the explicit index-dependency config.

```properties
# This is sufficient when @Alternative classes are in src/test/java/:
quarkus.arc.selected-alternatives=io.example.InMemoryRepository

# quarkus.index-dependency is NOT needed — test sources are auto-indexed
```

This is useful when avoiding Maven module cycles: copy the alternative implementations into the module's own `src/test/java/` and the `selected-alternatives` property works without any additional indexing configuration.
