## @TestTransaction swallows unique-constraint violations — Hibernate never flushes within the test body

**ID:** GE-20260414-7ce32b
**Stack:** Quarkus 3.32.2, Hibernate ORM 6.x, Panache active record, JUnit 5
**Symptom:** `assertThrows(PersistenceException.class, () -> entity.persist())` or `assertThrows(..., () -> service.create(...))` does not throw — the assertion fails with "expected exception was not thrown". The DB unique constraint is correctly defined, and the same operation fails as expected against a committed database.
**Context:** Testing that a unique constraint (e.g. unique channel name) is enforced by writing two entities with the same key inside a `@TestTransaction`-annotated test method.

### What was tried (didn't work)
- Verifying the constraint exists in the Flyway migration — it did
- Adding `Entity.flush()` inside the lambda passed to `assertThrows` — partially worked but marked the transaction rollback-only, making subsequent test code unreliable
- Using `@TestTransaction` on the class — same behaviour

### Root cause

Within a single `@TestTransaction`, Hibernate queues all pending INSERTs and only flushes them to the database at the end of the transaction (commit time). When two entities with the same unique-constrained value are created inside the test body, both INSERTs are queued. The unique constraint violation only fires when Hibernate flushes — which happens **after** `assertThrows` has already exited without catching anything.

Calling `Entity.flush()` inside the lambda forces the flush, which does trigger the exception. However, once a `PersistenceException` is thrown during a `@TestTransaction` test, Hibernate marks the persistence context invalid and the transaction rollback-only. Any code after the `assertThrows` block may see unexpected exceptions or stale state.

### Fix

Use `QuarkusTransaction.requiringNew()` so each operation runs in its own committed transaction. The first commits successfully; the second fails at commit with the constraint violation; cleanup runs in a third transaction:

```java
@Test
void duplicateNameThrowsException() {
    String name = "dup-test-" + System.nanoTime();

    // First create — commits cleanly
    QuarkusTransaction.requiringNew().run(() ->
            channelService.create(name, "First", ChannelSemantic.APPEND, null));

    // Second create — fails at commit with constraint violation
    assertThrows(Exception.class, () ->
            QuarkusTransaction.requiringNew().run(() ->
                    channelService.create(name, "Second", ChannelSemantic.APPEND, null)));

    // Cleanup the committed first record
    QuarkusTransaction.requiringNew().run(() -> Channel.delete("name", name));
}
```

Note: no `@TestTransaction` on this test — each `requiringNew()` lambda is its own transaction.

### Why non-obvious

`@TestTransaction` is the recommended isolation mechanism for Quarkus data tests, and it works well for happy paths. The constraint-test failure looks like the constraint is missing, not like a flush-timing issue — there's no exception, just a missing one. The `QuarkusTransaction` API is not prominently featured in Quarkus testing docs; most examples use `@TestTransaction` or `@Transactional`.
