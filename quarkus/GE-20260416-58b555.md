### Symptom

`activeProject.getConfig()` (or any `@ApplicationScoped` CDI singleton populated by an HTTP endpoint) returns `null` in `@QuarkusTest` even though the Quarkus server starts cleanly and logs show successful startup. Tests fail with NullPointerException or fall back to stale state.

### Context

A CDI singleton (`ActiveProject`) is populated only when `POST /api/projects/{id}/activate` is called via HTTP — that endpoint calls `activeProject.set(id, config, projectDir)`. The Quarkus server's `@PostConstruct` methods run at startup (Python bridge initialises, first project activates in Python), but the Java `ActiveProject` singleton was never populated because no HTTP call was made.

`@QuarkusTest` starts the full Quarkus application in-process (including CDI container, JVM, network). All `@PostConstruct` and `@ApplicationScoped` init methods fire. But the test framework never calls any HTTP endpoints during startup — it only provides the test with a running server. Any state that only exists after an HTTP call must be set up by the test itself.

### What was tried

Assumed that since `PythonBridge.init()` (`@PostConstruct`) ran successfully and Python activated the first project, the Java `ActiveProject` singleton would also be populated. The bridge and the CDI singleton are separate — Python state does not flow into Java CDI beans automatically.

### Root cause

`@QuarkusTest` starts the Quarkus application but does not replay any HTTP calls. CDI beans populated by HTTP endpoint handlers (`@POST`, `@GET`, etc.) are only populated when those endpoints are actually called. The CDI lifecycle (construction, injection, `@PostConstruct`) is distinct from application-level business logic that runs in response to HTTP requests.

### Fix

Add `@BeforeEach` (or `@BeforeAll` with `@TestInstance(PER_CLASS)`) to the `@QuarkusTest` class that calls the setup endpoint via RestAssured before tests run:

```java
@QuarkusTest
class ScanEndpointTest {

    @BeforeEach
    void activateProject() {
        if (!kieArchivePresent()) return;
        // Fetch first project and activate it so ActiveProject CDI bean is populated
        given().when().get("/api/projects")
                .then().statusCode(200)
                .extract().jsonPath()
                .<List<Map<String,Object>>>getList("$")
                .stream().findFirst().ifPresent(project -> {
                    String id = (String) project.get("id");
                    given().contentType("application/json")
                           .when().post("/api/projects/" + id + "/activate")
                           .then().statusCode(200);
                });
    }
}
```

### Why this is non-obvious

`@PostConstruct` runs. The CDI container is up. The logs show no errors. The server is healthy — `GET /api/config` returns 200. Everything looks like setup succeeded. The null singleton is surprising because nothing in the Quarkus `@QuarkusTest` documentation distinguishes "application started" from "application-level state initialised via HTTP". The natural assumption is that a running server has been through its full startup sequence, including any activation calls. It hasn't — it's started the JVM, not the application workflow.

### Alternatives considered

- `@PostConstruct` on the test class — doesn't work because CDI injection into the test happens during `@QuarkusTest` setup, not before it
- Static `@BeforeAll` without `@TestInstance(PER_CLASS)` — requires the activation call to use a static RestAssured call; works but `@BeforeEach` is simpler and ensures fresh state
