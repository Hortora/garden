## What It Is

Quarkus Vert.x `EventBus` has two publication methods with very different semantics:

- **`eventBus.publish(address, message)`** — fan-out: ALL registered `@ConsumeEvent` handlers
  on that address receive the message
- **`eventBus.send(address, message)`** — point-to-point: only ONE handler receives the message
  (round-robin if multiple consumers registered)

## How It Matters

If you add a second `@ConsumeEvent` handler to an address that was previously point-to-point
(`send()`), only one handler fires per event — silently. There is no error, no warning, and
no indication which consumer received it.

```java
// Publisher — must use publish() for fan-out
eventBus.publish(EventBusAddresses.WORKER_EXECUTION_FINISHED, event); // ✅ all consumers receive
eventBus.send(EventBusAddresses.WORKER_EXECUTION_FINISHED, event);    // ❌ only one consumer receives

// Consumers — both annotated the same way regardless
@ConsumeEvent(EventBusAddresses.WORKER_EXECUTION_FINISHED)
public Uni<Void> engineHandler(WorkflowExecutionCompleted event) { ... }

@ConsumeEvent(EventBusAddresses.WORKER_EXECUTION_FINISHED)
public Uni<Void> blackboardHandler(WorkflowExecutionCompleted event) { ... }
```

## Why Not Documented

The Quarkus `@ConsumeEvent` documentation focuses on the consumer side and does not state
that the publisher must use `publish()` for fan-out. The Vert.x core documentation covers
this, but the Quarkus abstraction layer hides it. The failure mode (one handler silently
skipped) is not detectable from `@ConsumeEvent` alone.

## Notes

- `eventBus.publish()` is the correct choice whenever multiple modules may independently
  subscribe to the same event type (e.g. engine handler + blackboard handler on
  `WORKER_EXECUTION_FINISHED`)
- Verify the publisher implementation when adding a second `@ConsumeEvent` for an existing
  address — search for `eventBus.send(` vs `eventBus.publish(` at the publisher site
