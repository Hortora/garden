## Quarkus Vert.x @ConsumeEvent on a request() address silently starves the primary consumer

### Symptom

Adding a second `@ConsumeEvent("some.address")` bean on an address that another part of the engine uses with `eventBus.request()` causes the primary handler to intermittently miss events. Cases stop completing or transitioning correctly, with no error logged. The second consumer receives some requests and the primary misses them.

### Context

Quarkus Vert.x event bus has two sending modes:
- `eventBus.publish(address, msg)` — fan-out: all registered consumers receive the message
- `eventBus.request(address, msg)` — point-to-point: exactly one consumer receives the message (round-robin across registered consumers)

When using `request()`, registering a second `@ConsumeEvent` bean on the same address makes two consumers compete for each message in round-robin. The "extra" consumer wins ~50% of events.

### What was tried (didn't work)

Adding `@ConsumeEvent(EventBusAddresses.CASE_STARTED)` to `CaseTimeoutEnforcer` to record case start times. The enforcer received events, but the primary `CaseStartedEventHandler` stopped receiving ~50% of `CASE_STARTED` events, causing cases to silently stall in RUNNING state with no error.

### Root cause

Vert.x `request()` is inherently point-to-point. The Quarkus `@ConsumeEvent` annotation does not distinguish between `publish` and `request` addresses — it registers a consumer on the Vert.x event bus regardless. If the address is used with `request()`, multiple `@ConsumeEvent` beans create multiple competing consumers, and Vert.x distributes requests round-robin. There is no warning in logs, no compile-time error, and no runtime exception — just missing events.

### Fix

Do not add `@ConsumeEvent` to an address if the producing code uses `request()`. Determine the sending mode by reading the producer code. If you need a side-effect on the same event, use one of:

1. **Lazy observation in the scan loop** — instead of consuming the event, inspect the state at scan time:
   ```java
   // In @Scheduled scanForTimeouts(): record start time on first observation of RUNNING
   runningStartTimes.computeIfAbsent(caseId, id -> now);
   ```

2. **Fan-out via publish()** — if the producer can switch from `request()` to `publish()`, all `@ConsumeEvent` beans receive it safely. Only viable if the producer does not need a reply.

3. **Enrich the existing consumer** — add the side-effect logic directly to the primary `@ConsumeEvent` handler method.

### Why this is non-obvious

The Quarkus Vert.x documentation for `@ConsumeEvent` does not mention point-to-point vs fan-out semantics, or the interaction between `request()` and multiple consumers. The failure is a silent missing-event race — no exception, no warning, just intermittent incorrect behaviour that looks like a timing or ordering bug.
