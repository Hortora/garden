## Symptom

`mvn verify` produces `Tests run: 0, Failures: 0, Errors: 0, Skipped: 0` followed by
`BUILD FAILURE` with no useful error. The `@QuarkusTest` classes never execute. Sometimes
`ClassSelector resolution failed` appears for the test class name.

## Context

A new Maven module contains `@QuarkusTest` integration test classes named with the `*IT.java`
suffix (e.g. `BlackboardIntegrationIT.java`). The build uses `maven-failsafe-plugin` alongside
`maven-surefire-plugin`.

## Root Cause

Maven's test runner naming convention is:

- **Surefire** runs classes matching `**/*Test.java`, `**/Test*.java`, `**/*Tests.java` — **excludes `*IT.java` by default**
- **Failsafe** runs classes matching `**/*IT.java`, `**/IT*.java`, `**/*ITCase.java`

`@QuarkusTest` requires Surefire's class loading to initialise the Quarkus test runtime. When
Failsafe picks up a `*IT.java` class it starts a separate, isolated JVM that cannot load Quarkus
internals, producing silent `0 tests` or cryptic `ClassSelector` errors.

## Fix

Rename test classes from `*IT.java` to `*Test.java`. Surefire picks them up and `@QuarkusTest`
initialises correctly.

```bash
mv src/test/java/com/example/BlackboardIntegrationIT.java \
   src/test/java/com/example/BlackboardIntegrationTest.java
# Update class name inside the file to match
sed -i 's/class BlackboardIntegrationIT/class BlackboardIntegrationTest/' \
   src/test/java/com/example/BlackboardIntegrationTest.java
```

If you have a `maven-failsafe-plugin` in your POM that's no longer needed, remove it too.

## Why This Is Non-Obvious

`Tests run: 0` looks like a classpath or configuration problem. Nothing in the output reveals
that the tests ran in the wrong runner. The Quarkus documentation says to annotate with
`@QuarkusTest` but says nothing about naming conventions. The convention is a Maven standard,
not a Quarkus one — and `IT` (Integration Test) is a semantically appropriate suffix for
`@QuarkusTest` tests, making the trap easy to fall into.
