## What happened

`MeshResource.timeline()` injected `QhorusMcpTools` and called `getChannelTimeline(name, null, limit)`. When the channel doesn't exist, `QhorusMcpTools` throws `IllegalArgumentException("Channel not found: …")` at line 1540.

`MeshResource` caught `IllegalArgumentException` — but the endpoint returned HTTP 500 instead of the expected empty list.

## Symptom

`timeline()` catch block appeared to never trigger; exceptions propagated past it as `ToolCallException` wrapping the original `IllegalArgumentException`.

## Root cause

`QhorusMcpTools` is annotated with:

```java
@WrapBusinessError({ IllegalArgumentException.class, IllegalStateException.class })
```

This activates `WrapBusinessErrorInterceptor`, a CDI interceptor that intercepts every method call on the injected proxy and wraps the listed exception types into `ToolCallException` *before they exit the proxy*.

When `MeshResource` receives the thrown exception from the CDI proxy, it is already a `ToolCallException` — the original `IllegalArgumentException` is the cause. The `catch (IllegalArgumentException e)` block never sees it.

## Fix

Catch both types:

```java
} catch (IllegalArgumentException | ToolCallException e) {
    // QhorusMcpTools @WrapBusinessError wraps IllegalArgumentException
    // (unknown channel) and IllegalStateException (paused/ACL-blocked)
    // into ToolCallException before they exit the CDI proxy.
    return List.of();
}
```

## Why this is non-obvious

The exception type visible at the `catch` site differs from the exception type thrown inside the callee. This only happens through CDI proxies; calling the method directly on a real instance would throw `IllegalArgumentException` as expected. Nothing in the stack trace suggests that interception is occurring — you see `ToolCallException` but the catch says `IllegalArgumentException`, and the disconnect is invisible.

### Why this fix

Catching `ToolCallException` explicitly documents the proxy behaviour and prevents the gotcha from recurring silently. Catching `Exception` would also work but is too broad — `ToolCallException` is the precise type produced by `@WrapBusinessError`.
