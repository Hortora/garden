## Symptom

`@QuarkusTest` fails at boot with `AmbiguousResolutionException: Ambiguous dependencies for type X`. The production code compiles and the beans look correct.

## Context

Defining multiple `@ApplicationScoped` implementations of the same SPI interface (e.g. three `SkillProfileProvider` implementations) so users can choose which to activate. All three are unqualified, so CDI cannot resolve which one to inject at a single injection point.

## What was tried

Adding all three as `@ApplicationScoped` beans, assuming CDI would pick the first one or throw a clear build-time error.

## Root cause

CDI requires exactly one unambiguous bean per injection type when no qualifier is used. Multiple unqualified `@ApplicationScoped` beans implementing the same interface all compete — CDI cannot resolve the injection point and throws at container startup, not at compile time.

## Fix

Designate exactly one implementation as the default (leave it unqualified `@ApplicationScoped`). Mark all others with `@Alternative`:

```java
// Default — CDI injects this when no override is present
@ApplicationScoped
public class WorkerProfileSkillProfileProvider implements SkillProfileProvider { ... }

// Not active by default — user activates via @Alternative @Priority(1)
@ApplicationScoped
@Alternative
public class CapabilitiesSkillProfileProvider implements SkillProfileProvider { ... }

@ApplicationScoped
@Alternative
public class ResolutionHistorySkillProfileProvider implements SkillProfileProvider { ... }
```

Users override the default at their own application level:

```java
@ApplicationScoped
@Alternative
@Priority(1)
public class MyCustomProvider implements SkillProfileProvider { ... }
```

## Why this is non-obvious

The error only appears at `@QuarkusTest` boot — compile succeeds silently. The fix (marking non-default implementations `@Alternative`) is easy once known, but the CDI resolution rules around `@Alternative` vs unqualified beans are non-obvious. The natural instinct is to let the user pick via config, not via CDI qualifiers.

## Update — Variant: two @DefaultBean impls for the same type

`@DefaultBean` in Quarkus Arc means "use this if no non-default bean for this type exists."
When **two** `@DefaultBean @ApplicationScoped` beans implement the same interface, Arc cannot
resolve the injection point — both are default, neither wins — and throws
`AmbiguousResolutionException` as soon as any production bean injects that type. The symptom
only surfaces once an `@Inject` for the interface is added; before that the ambiguity exists
silently in the CDI container.

**Scenario:** Two library modules each provide a `@DefaultBean CurrentPrincipal` for
"uncontested" environments (`MockCurrentPrincipal` from casehub-platform, `QhorusInboundCurrentPrincipal`
from casehub-qhorus). Adding `@Inject CurrentPrincipal` to any production bean in the app
module triggers the error — the injection point forces CDI to resolve and it finds two equals.

**Fix — exclude one at application level:**

```properties
# application.properties — exclude the stub; the runtime principal wins
quarkus.arc.exclude-types=io.casehub.platform.mock.MockCurrentPrincipal
```

**Test fix — use quarkus.arc.selected-alternatives to force a third, definitive impl:**

```properties
# src/test/resources/application.properties
quarkus.arc.selected-alternatives=io.casehub.platform.testing.FixedCurrentPrincipal
```

**Why @DefaultBean differs from @Alternative:** `@DefaultBean` is not in the CDI spec — it is
a Quarkus Arc extension that says "this bean loses to any explicit non-default bean." But when
two `@DefaultBean` beans compete, there is no winner — Arc throws rather than picking arbitrarily.
The `@Alternative` pattern (one winner, rest marked alternative) is the correct SPI design for
exactly-one-at-a-time. `@DefaultBean` is for extension-provided stubs that yield to consumer
overrides, not for multiple peers. Two `@DefaultBean` peers should be refactored so that only one
module provides the default; the other moves to `@Alternative`.
