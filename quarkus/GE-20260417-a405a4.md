## Symptom

`mvn install` on a Quarkus extension runtime module fails with:

```
Build failure: Build failed due to errors
[error]: Build step HibernateOrmProcessor#configurationDescriptorBuilding threw an exception:
  ConfigurationException: Persistence unit '<default>' defines entities [...] but its datasource
  '<default>' cannot be found: Datasource '<default>' is not configured.
```

No datasource is configured because the runtime module is an extension — not a Quarkus application.

## Context

A `quarkus-maven-plugin` block with `<goal>build</goal>` was added to the extension runtime module's `pom.xml` (typically to enable `@QuarkusTest` support). Extension modules do not build Quarkus applications — they provide build-time and runtime code that other applications consume.

## What was tried (didn't work)

Adding `quarkus.hibernate-orm.enabled=false` to test properties — doesn't help because the `build` goal runs outside the test lifecycle.

## Root cause

The `quarkus-maven-plugin:build` goal triggers a full Quarkus application build, which includes bootstrapping the full extension processor chain including Hibernate ORM. Extension runtime modules have JPA entities (for the extension's own model classes) but no datasource — they rely on the consuming application to provide one. The build goal sees the entities, tries to configure a persistence unit, and fails.

## Fix

Remove the `quarkus-maven-plugin` `<build>` execution block from the extension runtime module's `pom.xml`. The `@QuarkusTest` infrastructure works without it — Quarkus's test framework bootstraps the application through the test lifecycle, not through the `build` goal.

```xml
<!-- Remove this entire block from extension runtime pom.xml -->
<plugin>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-maven-plugin</artifactId>
  <version>${quarkus.version}</version>
  <executions>
    <execution>
      <goals>
        <goal>build</goal>  <!-- WRONG in an extension module -->
      </goals>
    </execution>
  </executions>
</plugin>
```

Keep the `quarkus-extension-maven-plugin` (which generates extension descriptors) — only remove the `quarkus-maven-plugin` with the `build` goal.

## Why this is non-obvious

The `quarkus-maven-plugin` with `build` goal is standard in Quarkus *application* poms. An AI assistant or junior developer adding `@QuarkusTest` support to an extension module may cargo-cult this pattern from application examples. The error message points to a missing datasource, not to the plugin configuration — misdirecting debugging.

## Variant — Library modules (no extension descriptor)

The same failure mode occurs in plain library modules (not Quarkus extensions) that have `quarkus-maven-plugin:build` in their pom — for example, modules that define domain objects or SPIs but have no SPI implementations of their own. The error in this case is a CDI validation failure at build time:

```
[ERROR] Build step io.quarkus.arc.deployment.ArcProcessor#validate threw an exception:
  jakarta.enterprise.inject.spi.DeploymentException: No bean found for required type [...]
```

The module has CDI injection points (from domain classes or interfaces) but no beans to satisfy them — because the implementations live in other modules. CDI validation fires during the `build` goal's full application bootstrap and fails.

**Fix:** Same as above — remove `quarkus-maven-plugin:build` from library modules. These modules are not Quarkus applications; they don't need the build goal. `@QuarkusTest` in downstream modules handles bootstrapping through the test lifecycle.
