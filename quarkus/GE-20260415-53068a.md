## RestAssured deserialises JSON floats as Float not Double — (Double) cast throws ClassCastException

### Symptom

Test fails at runtime with:

```
java.lang.ClassCastException: class java.lang.Float cannot be cast to class java.lang.Double
```

on a line like:

```java
assertThat((Double) response.jsonPath().getMap("trustScore").get("score")).isBetween(0.0, 1.0);
```

The JSON response is valid and the field exists with a float value.

### Context

Occurs when:
- Using `response.jsonPath().get()`, `getMap()`, or `getList()` to extract values from a REST Assured response
- The JSON value is a floating-point number (e.g. `0.87`)
- You cast the returned `Object` to `Double`

### Root cause

REST Assured uses Jackson or Groovy's JSON parser to deserialise responses. When extracting values via `jsonPath()` into raw `Object` types, floating-point JSON values are returned as `Float`, not `Double`. The Java `(Double)` cast fails because `Float` is not a subtype of `Double`.

### Fix

Use `((Number) ...).doubleValue()` instead of `(Double)`:

```java
// Wrong — throws ClassCastException
assertThat((Double) response.jsonPath().getMap("result").get("score")).isBetween(0.0, 1.0);

// Correct
assertThat(((Number) response.jsonPath().getMap("result").get("score")).doubleValue())
        .isBetween(0.0, 1.0);
```

Or extract the full response as a typed POJO to avoid raw casting entirely:

```java
MyResponse result = response.as(MyResponse.class);
assertThat(result.score()).isBetween(0.0, 1.0);
```

### Why this is non-obvious

The JSON spec doesn't distinguish Float from Double — both are "number". A developer expects `0.87` in JSON to deserialise as a Java `double` (the natural wide type), not `float`. REST Assured's `jsonPath()` raw extraction silently chooses `Float`, with no warning and no documentation calling this out.
