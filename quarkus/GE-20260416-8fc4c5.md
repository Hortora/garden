## Symptom

Disk fills up. `du`, Finder, and DaisyDisk show nothing unusual. `df` shows the disk as full or nearly full. The log file at `/tmp/quarkus-*.log` is absent from the filesystem. Up to 161GB of disk space consumed invisibly.

## Context

Quarkus dev mode (`mvn quarkus:dev`) with file logging enabled writes to `/tmp/quarkus-*.log`. The log file is deleted while the JVM is running — by a cleanup script, a `rm` command, DaisyDisk, or similar. On macOS (and Linux), deleting a file that a process holds open removes the directory entry but keeps the inode alive. Disk space is not freed until the last file descriptor referencing the inode is closed.

## Diagnosis

```bash
lsof -c java | grep deleted
```

Shows the open-but-deleted file with its size. Example output:
```
java  12345  user  47u  REG  1,5  161073741824  /tmp/quarkus-app.log (deleted)
```

The `(deleted)` suffix is the tell. The size column shows the actual consumed space.

## Root Cause

POSIX file semantics: `unlink()` removes the directory entry but does not free the inode until all file descriptors referencing it are closed. The JVM's logging framework holds a file descriptor open for the duration of the process. As long as the JVM runs, the inode lives and the disk space is consumed — invisibly from any filesystem traversal tool.

## Fix

**Immediate:** Kill the JVM to free the space.

```bash
pkill -f 'quarkus:dev'
# Disk space freed when the process exits
```

**Permanent — always kill Java before deleting log files:**

```bash
pkill -f 'quarkus:dev' && sleep 2 && rm -f /tmp/quarkus-*.log
```

**Preventive — add log rotation** so the file never grows large enough to be worth deleting:

```properties
# application.properties
%emulated.quarkus.log.file.enable=true
%emulated.quarkus.log.file.path=/tmp/quarkmind-emulated.log
%emulated.quarkus.log.file.rotation.max-file-size=20M
%emulated.quarkus.log.file.rotation.max-backup-index=3
%emulated.quarkus.log.file.rotation.rotate-on-boot=true
```

Also set a graceful shutdown timeout so file handles are released cleanly on Ctrl+C:

```properties
quarkus.shutdown.timeout=10S
```

## Why This Is Non-Obvious

The file is gone from the directory listing — `ls`, `du`, Finder, and DaisyDisk all agree it does not exist. There is no error, no warning, no indication that disk space is still consumed. The only tool that reveals it is `lsof` with the `(deleted)` grep. Most developers reach for disk analysis tools first, find nothing, and are baffled. The JVM must be killed to reclaim the space — restarting Quarkus without killing the process is insufficient because the old JVM still holds the descriptor.
