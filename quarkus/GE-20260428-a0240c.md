## Symptom

After adding `@io.smallrye.common.annotation.Blocking` to a `@QuarkusTest` `@Test` method, Surefire reports:

```
[ERROR] ClassSelector [className = 'dev.example.MyTest', classLoader = null] resolution failed
Cause: Could not load class with name: dev.example.MyTest
Tests run: 0, Failures: 0, Errors: 0, Skipped: 0
BUILD FAILURE
```

The entire test class is unloaded — not just the annotated method. All other tests in the same class also disappear. No compilation error. The symptom is identical to a Quarkus CDI startup failure, which is an unrelated root cause.

## Root Cause

`@Blocking` is a SmallRye/Vert.x annotation that triggers Quarkus interceptors to dispatch the method to a worker thread pool. The Quarkus ArC CDI processor tries to process the annotation during test class bytecode transformation. JUnit's `@Test` methods are not CDI bean methods and the annotation cannot be applied to them in this context — the result is a class-transformation failure that prevents the test classloader from loading the class at all.

## Fix

Do not annotate `@Test` methods with `@Blocking`. If the issue is JTA/JPA calls on the Vert.x IO thread, the alternatives are:

1. Submit the blocking work to a separate thread before calling the reactive API:
   ```java
   ExecutorService worker = Executors.newSingleThreadExecutor();
   worker.submit(() -> myBean.doBlockingWork());
   ```

2. Change the method under test to use `@Transactional(TxType.SUPPORTS)` instead of `REQUIRED`, so it doesn't start a new transaction when called without one.

3. Bypass the Vert.x-threaded code path in tests by calling the CDI event or service directly on the test thread.

## Why This Is Non-Obvious

`@Blocking` works correctly on CDI bean methods and Quarkus REST endpoints — adding it to those never causes class loading failures. The obvious assumption is that adding it to a test method would similarly dispatch the test to a worker thread. The failure mode (classLoader=null for an unrelated class) is the same symptom as a CDI deployment failure, so developers hunt for CDI ambiguities or missing beans instead of looking at the test method annotation.
