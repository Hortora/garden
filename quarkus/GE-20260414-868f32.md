## Quarkiverse extension-descriptor goal validates ALL transitive deployment JARs — not just direct ones

**ID:** GE-20260414-868f32
**Stack:** Quarkus 3.32.2, quarkus-extension-maven-plugin 3.32.2, Quarkiverse extension layout
**Symptom:** `MojoExecutionException` from `extension-descriptor` goal listing 20–30 missing deployment artifacts from the deployment module's configuration. The error names every transitive deployment JAR, not just direct ones.
**Context:** Building a Quarkiverse extension with the standard parent/runtime/deployment module layout. The deployment module has `quarkus-core-deployment` and a few others declared, but the runtime module has many extension dependencies (REST, Panache, MCP server, health, etc.).

### What was tried (didn't work)
- Adding only the directly-used deployment JARs (`quarkus-core-deployment`, `quarkus-hibernate-orm-deployment`, `quarkus-rest-deployment`, `quarkus-flyway-deployment`)
- Running `mvn test -pl runtime` directly without installing the deployment module first

### Root cause

The `quarkus-extension-maven-plugin:extension-descriptor` goal reads the deployment module's POM **from the local Maven repository** (not the build reactor), then validates that for every deployment-variant JAR in the runtime module's transitive dependency tree, there is a corresponding explicit declaration in the deployment module's `<dependencies>`. The check is transitive: if the runtime depends on `quarkus-mcp-server-http`, and that depends on `quarkus-vertx-http`, then `quarkus-vertx-http-deployment` must be in the deployment POM.

Two compounding issues:
1. The deployment module must be `mvn install`ed before the runtime module tests will work — the plugin reads from local repo, not the reactor.
2. The list of required deployment JARs is the full transitive closure, which can be 25+ entries for a moderately complex extension.

### Fix

In the deployment module's `pom.xml`, declare every deployment-variant JAR that corresponds to any dependency in the runtime module's transitive tree. Run `mvn install -DskipTests` first to put the deployment artifact in the local repo, then `mvn test -pl runtime`.

The full list to declare can be read directly from the error output — the plugin prints every missing JAR. For a runtime module with Panache, REST, Flyway, MCP server, and health, expect to declare ~25 deployment JARs:

```xml
<!-- MCP server -->
<dependency>
  <groupId>io.quarkiverse.mcp</groupId>
  <artifactId>quarkus-mcp-server-http-deployment</artifactId>
  <version>${quarkus-mcp-server.version}</version>
</dependency>
<dependency>
  <groupId>io.quarkiverse.mcp</groupId>
  <artifactId>quarkus-mcp-server-core-deployment</artifactId>
  <version>${quarkus-mcp-server.version}</version>
</dependency>
<!-- Quarkus core — all from BOM, no version needed -->
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-arc-deployment</artifactId>
</dependency>
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-rest-deployment</artifactId>
</dependency>
<!-- ... and ~20 more -->
```

### Why non-obvious

The Quarkiverse contribution guide shows a minimal deployment POM with only `quarkus-core-deployment`. It does not mention that the full transitive closure must be declared. The error message lists the missing JARs but doesn't explain why they're required or where to add them. The "install deployment first" dependency is also undocumented — running `mvn test -pl runtime` in a fresh checkout fails even after fixing the POM, until a separate install step is run.
