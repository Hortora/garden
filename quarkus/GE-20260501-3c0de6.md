## Symptom

Schema creation fails with:
```
ERROR: syntax error at or near ")"
create table ledger_entry (..., dtype varchar(31) not null check ((dtype in ())), ...)
```
Both PostgreSQL and H2 reject the empty `IN` clause. The error appears during `@QuarkusTest` startup, not at runtime.

## Context

Occurs when a JPA entity uses `@Inheritance` (JOINED or SINGLE_TABLE strategy) and not all subclasses of the hierarchy are on the test classpath. Hibernate 6.x generates a discriminator CHECK constraint that lists known subclasses — if none are found, the list is empty, producing invalid SQL.

Typical scenario: a base entity lives in an API/library module; a concrete subclass lives in a separate module that is NOT included as a test dependency of the module being tested.

## What Was Tried

- Adding `beans.xml` and Jandex to the base entity's module — no effect (subclass module still absent)
- Adding `quarkus.index-dependency` for the base module — no effect
- Disabling Flyway — no effect (schema generation is Hibernate DDL, not Flyway)

## Root Cause

Hibernate 6.x generates `check ((dtype in (...)))` by scanning indexed subclasses at augmentation time. If the concrete subclass JAR (`casehub-engine-ledger` containing `CaseLedgerEntry extends LedgerEntry`) is not on the classpath, Hibernate sees zero subclasses and generates `dtype in ()` — which is illegal SQL.

## Fix

Add the subclass module as a **test scope** dependency to every module whose tests bring the base entity onto the classpath:

```xml
<dependency>
    <groupId>io.casehub</groupId>
    <artifactId>casehub-engine-ledger</artifactId>
    <version>${project.version}</version>
    <scope>test</scope>
</dependency>
```

Also ensure the subclass module has `jandex-maven-plugin` configured so Quarkus can index it.

## Why This Is Non-Obvious

The error message (`syntax error at or near ")"`) points to invalid SQL, not a missing dependency. The natural investigation path is SQL syntax, Hibernate config, or schema setup — not classpath completeness. The base entity module may be correctly indexed and discoverable; only the subclass is missing.
