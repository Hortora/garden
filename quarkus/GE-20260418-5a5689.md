## Symptom

A CDI dependency injected into a plain Java object comes back `null` at runtime. No error, no warning — the field is simply never populated.

## Context

A CDI `@ApplicationScoped` bean owns and instantiates a plain Java object (no CDI scope annotation). A developer adds `@Inject` to a field on the owned plain object expecting CDI to satisfy it.

```java
@ApplicationScoped
public class OwnerBean {
    private final PlainObject owned = new PlainObject(); // instantiated by owner
}

public class PlainObject {
    @Inject
    SomeService service; // NEVER populated — CDI doesn't know about this object
}
```

## Root cause

CDI only manages the lifecycle of beans it instantiates. A plain Java object created via `new` is invisible to the CDI container — it doesn't appear in the bean registry, so `@Inject` annotations on its fields are never processed. The CDI proxy mechanism only fires for objects the container creates itself.

## Fix

Place the `@Inject` on the owning CDI bean and pass the dependency to the plain object via constructor or setter:

```java
@ApplicationScoped
public class OwnerBean {
    @Inject SomeService service;

    private final PlainObject owned;

    @PostConstruct
    void init() {
        this.owned = new PlainObject(service); // inject via constructor
    }
}
```

Or make the plain object a CDI bean itself if it needs multiple dependencies:

```java
@ApplicationScoped // now CDI-managed
public class PlainObject {
    @Inject SomeService service; // now works
}
```

## Why this is non-obvious

The annotation `@Inject` compiles and runs without error on a non-CDI object. There is no `UnsatisfiedResolutionException` — the field silently stays `null`. A developer expecting CDI to "just work" because they wrote `@Inject` gets a NullPointerException deep in business logic, not at startup. The pattern is especially deceptive when the owning bean IS CDI-managed (so all its own injections work fine) and only the owned helper object's injections fail.
