## Symptom

Calling `gateway.post(channelId, message)` inside a `@Transactional` MCP tool method, where `post()` internally calls `agentBackend.post()` → `messageService.send()` (also `@Transactional`), causes a double-persist: the tool method calls `messageService.send()` first for the return value, then `gateway.post()` calls it again through the backend. The message is written twice to the DB.

## Context

Any architecture where a gateway wraps an existing persistence service. The design doc said "route through gateway.post()"; the implementation discovered the double-persist problem when the tool method still needed the `Message` entity returned by `messageService.send()` to build its response.

## Root Cause

The MCP tool method (`@Transactional`) calls `messageService.send()` to get back the persisted entity, then the design required also calling `gateway.post()` which internally calls `messageService.send()` again via `QhorusChannelBackend.post()`. Two writes in the same transaction — same data, same channel.

## Fix

Split the concerns:

1. Keep `messageService.send()` in the tool method — it returns the entity, handles rate limiting, commitment lifecycle, ledger write. This is the source-of-truth write.
2. Add a `gateway.fanOut(UUID channelId, OutboundMessage)` that dispatches **only to external backends** (not the agent backend). Call this after `messageService.send()` returns.
3. Make `gateway.post()` package-private (for unit tests only); document it as a test-only path.

```java
// In sendMessage() @Tool:
Message message = messageService.send(...);              // persistence + ledger
channelGateway.fanOut(channel.id, outboundMessage);      // external backends only
```

## Why This Is Non-Obvious

The initial design correctly identified that the gateway should be the single entry point. The transactional trap is invisible until the entity return value forces a direct `messageService.send()` call — at which point the gateway's internal call to the same service becomes redundant. The fix (fanOut vs post) is not obvious because it requires splitting one conceptual operation into two methods with different scope.
