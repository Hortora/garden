## Symptom

Deleting a `Message` entity that has replies throws:
```
org.h2.jdbc.JdbcSQLIntegrityConstraintViolationException: Referential integrity constraint violation:
"FK_MESSAGE_REPLY: PUBLIC.MESSAGE FOREIGN KEY(IN_REPLY_TO) REFERENCES PUBLIC.MESSAGE(ID)"
```

## Context

The `message` table has `in_reply_to BIGINT REFERENCES message(id)`. Deleting a parent message leaves child rows with a dangling FK — the DB rejects the delete.

The intent is to delete only the parent and leave replies intact (orphaned), not cascade-delete replies.

## What was tried

`msg.delete()` directly → FK violation.
`Message.delete("id", id)` → same error.

## Root cause

Hibernate does not automatically null-out referencing rows before deletion unless explicitly configured with `@OnDelete(action = OnDeleteAction.SET_NULL)` on the FK column, which requires schema-level support and migration.

## Fix

Manually null out the FK on child rows before deleting the parent:

```java
// Null out replies' in_reply_to before deleting the parent
Message.update("inReplyTo = null WHERE inReplyTo = ?1", messageId);
// Now safe to delete the parent
Message message = Message.findById(messageId);
message.delete();
```

This leaves replies intact but with a null `inReplyTo` — acceptable when the goal is surgical message deletion without cascading.

## Why this is non-obvious

The error message correctly identifies the FK but gives no hint about the fix. The natural assumption is to add `cascade = CascadeType.REMOVE` — but that would delete the replies, which is the opposite of the intent. The correct fix (pre-nulling the FK) is not surfaced by any Panache documentation for self-referencing tables.
