## @Produces @DefaultBean @ApplicationScoped on producer methods enables consumer-replaceable CDI defaults

**ID:** GE-20260421-1cfae6
**Stack:** Quarkus Arc 3.32.2

### The pattern

Annotate a producer method with `@Produces @DefaultBean @ApplicationScoped` to provide a built-in default that consumers can silently replace by simply declaring their own CDI bean of the same type — no `@Alternative`, no `beans.xml`, no config required on the consumer side.

```java
@ApplicationScoped
public class LedgerPrivacyProducer {

    @Inject LedgerConfig config;
    @Inject EntityManager em;  // CDI proxy — resolves to current transaction's session

    @Produces
    @DefaultBean
    @ApplicationScoped
    public ActorIdentityProvider actorIdentityProvider() {
        if (config.identity().tokenisation().enabled()) {
            return new InternalActorIdentityProvider(em);  // plain Java object
        }
        return new PassThroughActorIdentityProvider();
    }

    @Produces
    @DefaultBean
    @ApplicationScoped
    public DecisionContextSanitiser decisionContextSanitiser() {
        return new PassThroughDecisionContextSanitiser();
    }
}
```

A consumer overrides by declaring their own bean — the producer's output is ignored:

```java
@ApplicationScoped
public class MyCustomActorIdentityProvider implements ActorIdentityProvider {
    // replaces the producer's default without any configuration
}
```

### Key property: EntityManager CDI proxy in a non-CDI object

The `em` field in `LedgerPrivacyProducer` is a CDI proxy. Passing it to a plain Java constructor (`new InternalActorIdentityProvider(em)`) is safe because the proxy is thread-safe and resolves to the current persistence context when any method is called. As long as callers of `InternalActorIdentityProvider` operate within a `@Transactional` boundary, the EntityManager works correctly without the object itself being a CDI bean.

This avoids the problem of making `InternalActorIdentityProvider` a CDI bean while still giving it transactional access to the database.

### Why this is non-obvious

Quarkus documentation and examples consistently show `@DefaultBean` on class-level `@ApplicationScoped` beans:

```java
@ApplicationScoped
@DefaultBean
public class DefaultActorIdentityProvider implements ActorIdentityProvider { ... }
```

Applying `@DefaultBean` to a `@Produces` method is not documented. The producer method form is more powerful because:
1. It can return different implementations based on runtime config
2. It can construct plain Java objects that receive CDI proxies via the producer's own injected fields
3. It avoids the ambiguity of having multiple `@ApplicationScoped` implementations of the same type registered simultaneously

Most developers who want runtime-switchable CDI defaults reach for `@Alternative` + `@Priority` or config-conditional `@IfBuildProperty` (build-time only). This pattern achieves the same result at runtime with zero consumer-side changes.
