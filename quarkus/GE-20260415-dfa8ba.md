## Symptom

Outbound REST calls from a programmatic `RestClientBuilder.newBuilder()` client silently omit
the `X-Api-Key` header despite `@RegisterProvider(FleetKeyClientFilter.class)` being declared
on the `@RegisterRestClient` interface. The receiving endpoint returns 401. In a fleet/circuit-
breaker context, this triggers `recordFailure()` on every health check, opening the circuit and
marking all peers as OPEN â€” no explicit error, just degraded connectivity.

## Context

Fleet peer-to-peer calls built with `RestClientBuilder.newBuilder()`:

```java
var client = RestClientBuilder.newBuilder()
    .baseUri(URI.create(peerUrl))
    .build(PeerClient.class); // @RegisterProvider ignored here
```

The `@RegisterProvider` annotation on `PeerClient` is honoured when the client is injected via
`@RestClient`, but not reliably when built programmatically.

## What was tried (didn't work)

Assuming `@RegisterProvider` on the interface applies universally â€” as documented for injected
clients â€” to programmatic builder instances. The MicroProfile spec is silent on programmatic
builder behaviour.

## Root cause

`@RegisterProvider` is specified to apply to CDI-injected clients. The spec does not mandate
that `RestClientBuilder.newBuilder().build()` honour annotations on the interface. Quarkus's
RESTEasy Reactive implementation may or may not pick them up, making it implementation-defined
behaviour rather than a spec guarantee.

## Fix

Explicitly register the filter on the builder:

```java
var client = RestClientBuilder.newBuilder()
    .baseUri(URI.create(peerUrl))
    .connectTimeout(3, TimeUnit.SECONDS)
    .readTimeout(2, TimeUnit.SECONDS)
    .register(FleetKeyClientFilter.class)  // explicit â€” don't rely on @RegisterProvider
    .build(PeerClient.class);
```

If the filter uses CDI injection (`@Inject`), passing the class token (not an instance) lets
CDI resolve the dependency: `.register(FleetKeyClientFilter.class)`. This is supported in
Quarkus's RESTEasy Reactive programmatic builder.

## Why this is non-obvious

The `@RegisterProvider` annotation on the interface gives a strong visual signal that the filter
"just works" for all usages. The Quarkus docs show `@RegisterProvider` as the standard pattern
and don't call out the programmatic-builder exception. The failure is completely silent â€” no
warning, no exception, just missing headers â€” and the 401 error appears to come from the peer
(a network problem) rather than from the local filter not running.
