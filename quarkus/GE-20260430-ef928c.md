## Quarkus SRCFG00050 at test startup presents as ClassLoader failure, not config error

### Symptom

Tests fail at discovery — no test output, just:
```
TestEngine with ID 'junit-jupiter' encountered a critical issue during test discovery:
(1) ClassSelector [className = 'dev.claudony.SmokeTest'] resolution failed
    Cause: Could not load class with name: dev.claudony.SmokeTest
```

Looks like a classpath problem, compile error, or missing dependency. None of those apply.

### Context

Quarkus 3.32.2 validates all property keys from every config source — including `application.properties` files bundled inside dependency JARs — against registered `@ConfigMapping` roots at augmentation time. If a property key appears in any source but has no corresponding mapping, augmentation fails with `SRCFG00050`. This failure happens before any test class loads, so JUnit reports a classloading failure rather than a config error.

The underlying `ConfigValidationException` appears deep in the stack trace:
```
Caused by: io.smallrye.config.ConfigValidationException: Configuration validation failed:
    SRCFG00050: casehub.qhorus.a2a.enabled ... does not map to any root
    SRCFG00050: casehub.qhorus.cleanup.pending-reply-check-seconds ... does not map to any root
```

### Root cause

A dependency JAR bundles its own `application.properties` with default values for a property prefix. Quarkus loads all bundled `application.properties` files into the config pipeline. If the extension's deployment artifact doesn't register the `@ConfigMapping` for that prefix during augmentation (misconfigured extension descriptor), the defaults are present in the config pipeline but have no mapping root — triggering SRCFG00050.

### Fix

Add to `src/test/resources/application.properties`:

```properties
# Suppress SRCFG00050 for extension JARs that bundle unmapped defaults.
# Remove once the upstream extension properly registers its @ConfigMapping.
smallrye.config.mapping.validate-unknown=false
```

Do not add this to the main `application.properties` — it suppresses legitimate validation in production builds.

### Alternatives considered

- **Remove bundled defaults from upstream JAR** — requires upstream fix in the extension; correct long-term solution.
- **Add @ConfigMapping for the extension's prefix in the consuming app** — duplicates the extension's own config definition; breaks on extension updates.
- **Remove the offending property keys from `application.properties`** — only eliminates the consuming-app-side keys; the JAR-bundled defaults still fail.

### Why this is non-obvious

The error headline (`Could not load class`) points unambiguously at a classloading or compilation problem. The actual cause — config validation during Quarkus augmentation — requires reading 20+ lines into the stack trace to find `ConfigValidationException`. The property that suppresses it (`smallrye.config.mapping.validate-unknown`) is a SmallRye Config internal, not a documented Quarkus config property, and only appears in SmallRye source code.
