## Behaviour

A CDI observer declared on an abstract base class receives all events whose runtime type is a subtype of that base class:

```java
// Observer declared on abstract base — catches ALL subtypes
void onAnyWorkEvent(@Observes WorkLifecycleEvent event) {
    // fires for WorkItemLifecycleEvent, CaseTaskLifecycleEvent, or any other subtype
}

// Narrower observer — only catches events fired as WorkItemLifecycleEvent specifically  
void onWorkItemEvent(@Observes WorkItemLifecycleEvent event) {
    // does NOT fire for CaseTaskLifecycleEvent
}
```

## Context

When migrating a CDI event observer from a concrete type (`WorkItemLifecycleEvent`) to an abstract base type (`WorkLifecycleEvent`) so that multiple domain event subtypes can share the same observer (e.g. a generic filter engine).

## Why this is non-obvious

The CDI specification states that observer method type matching uses *assignability* (not exact type equality), but this behaviour is typically described in terms of qualifiers and parameterized types — rarely in terms of class hierarchy. Most developers assume `@Observes Foo` only catches events explicitly fired as `Foo`, not subtypes.

The practical implication: moving an observer from `@Observes WorkItemLifecycleEvent` to `@Observes WorkLifecycleEvent` makes it a generic listener for all present and future subtypes, without changing any event firing sites.

## Verified behaviour

```java
// Fires WorkItemLifecycleEvent (a subclass of WorkLifecycleEvent)
event.fire(WorkItemLifecycleEvent.of("CREATED", workItem, actor, detail));

// Both observers are notified:
// 1. @Observes WorkLifecycleEvent  ← catches it (subtype match)
// 2. @Observes WorkItemLifecycleEvent  ← catches it (exact match)
```
