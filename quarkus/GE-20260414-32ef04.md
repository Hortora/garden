## Symptom

A test annotated `@TestTransaction` creates a channel via `tools.createChannel(...)`, then calls `given().when().post("/some-endpoint")`. The HTTP handler returns 400 or 404 claiming the channel doesn't exist, even though the test just created it.

## Context

`@TestTransaction` wraps the entire test method in a transaction that is rolled back at the end. Injected CDI bean calls that are themselves `@Transactional` JOIN this ambient test transaction ‚Äî so the data is created but not yet committed. When RestAssured fires an HTTP request, it goes through the full HTTP stack in a separate thread with its own transaction context. That transaction cannot see the uncommitted data from the test transaction.

## What was tried

Verified the channel was created (it was ‚Äî within the test transaction). Confirmed the endpoint's error handling was correct. Added logging ‚Äî the HTTP handler genuinely could not find the channel.

## Root cause

Two separate transaction contexts: the test's ambient transaction (uncommitted) and the HTTP handler's transaction (separate, reads committed data only under READ_COMMITTED isolation).

## Fix

Do NOT use `@TestTransaction` on test methods that combine injected tool calls with RestAssured HTTP calls.

Instead:
- Rely on each `@Transactional` method committing its own transaction naturally
- Use unique channel/data names per test to avoid cross-test pollution

```java
// WRONG ‚Äî @TestTransaction makes channel invisible to HTTP call
@Test
@TestTransaction
void myTest() {
    tools.createChannel("my-channel", ...);  // uncommitted
    given().post("/endpoint").then().statusCode(200); // can't see the channel
}

// RIGHT ‚Äî no @TestTransaction; each @Transactional method commits
@Test
void myTest() {
    tools.createChannel("my-ch-" + UUID.randomUUID(), ...);  // commits immediately
    given().post("/endpoint").then().statusCode(200); // sees committed data
}
```

## Why this is non-obvious

`@TestTransaction` looks like it provides test isolation, which it does ‚Äî but at the cost of making all injected writes invisible to anything running in a separate transaction context (i.e., HTTP handlers). The symptom (404/400) gives no indication that the data exists but is uncommitted.
