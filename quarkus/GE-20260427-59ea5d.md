### Symptom
`@QuarkusTest` augmentation fails with:
```
org.flywaydb.core.api.FlywayException: Found more than one migration with version 1
-> /path/to/optional-module/target/classes/db/migration/V1__schema.sql (SQL)
-> /path/to/runtime/target/classes/db/migration/V1__initial_schema.sql (SQL)
```
No indication at compile time. Passes compilation, fails only at test startup.

### Context
An optional Quarkus module (e.g. `quarkus-work-notifications`) depends on the runtime module (`quarkus-work`) and defines its own Flyway migration. In tests, both modules' `db/migration/` directories land on the same classpath. Flyway treats all `db/migration/` directories from all classpath roots as a single source — so two `V1__*.sql` files from different modules collide.

### Root cause
Flyway's classpath scanning finds migration files across all JARs and directories on the classpath. When `quarkus.flyway.locations=db/migration` (the default), it scans every `db/migration/` it finds. Optional modules that introduce their own `V1__*.sql` collide with the main module's `V1__initial_schema.sql`.

### Fix
Use a high version namespace for optional module migrations. Convention in this codebase:
- Runtime module: `V1` through `V19` (core schema)
- Queues module: `V2000`, `V2001`, `V2002` (separate high namespace)
- Notifications module: `V3000` (another high namespace)
- Ledger module: `V1000`, `V2001` (ledger-specific range)

Rename your optional module's migration from `V1__schema.sql` to `V3000__schema.sql` (or the next available namespace for your module family).

Also: after renaming, run `mvn clean` — the old compiled `target/classes/db/migration/V1__*.sql` stays on disk and causes the same error even after the source rename, because the old .class-equivalent migration was already compiled. A clean build removes the stale file.

### Why this is non-obvious
The error message names both files but doesn't indicate they came from different Maven modules. The natural assumption is a duplicate file within the same module. The stale-target variant (rename in source, error persists) compounds the confusion — the source file is gone, but Maven's incremental build left the old compiled output.
