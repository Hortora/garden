### Summary
Scope file logging to a specific Quarkus profile via property prefixes so diagnostic
output is greppable from a file without requiring copy-paste from a remote terminal.

### Context
Debugging a Quarkus application running in a terminal the developer cannot easily
read from (e.g. a long-running dev session, remote pair programming, or a background
process). Console output is interleaved and too verbose to search manually.

### Technique
Add profile-prefixed file logging properties to `application.properties`:

```properties
%emulated.quarkus.log.file.enable=true
%emulated.quarkus.log.file.path=/tmp/quarkmind-emulated.log
%emulated.quarkus.log.file.level=INFO
%emulated.quarkus.log.file.rotation.max-file-size=10M
```

The `%emulated.` prefix means these only activate in the `emulated` profile —
no noise in other profiles, no permanent config to revert.

Then grep for specific diagnostic markers:
```bash
grep "PATHFINDING\|WALL-CROSS" /tmp/quarkmind-emulated.log
```

Combined with a targeted diagnostic log line added to the suspicious code path,
this immediately confirms whether a code path was reached and what state it saw —
without the developer needing to copy-paste hundreds of lines from a live terminal.

### Why this is non-obvious
Most developers reach for console output or a remote debugger. Profile-prefixed
file logging is a zero-friction middle ground: it survives process restarts, is
greppable, and is automatically scoped so it doesn't pollute other runs.
The `%profile.` property prefix works for all Quarkus runtime config — most
developers only know to use it for application-specific properties, not for
built-in Quarkus logging config.
