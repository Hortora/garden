## @TestTransaction + Hibernate JPQL bulk update = silently stale first-level cache

**ID:** GE-20260414-99854d
**Stack:** Quarkus 3.32.2, Hibernate ORM 6.x, Panache active record, JUnit 5
**Symptom:** A test that calls a service method performing a JPQL bulk `UPDATE` (e.g. `Entity.update("status = 'stale' WHERE ...")`) then immediately queries the same entity and asserts the new status — the assertion passes with the **old** value. The test appears green but is not testing anything.
**Context:** Any `@QuarkusTest` test annotated with `@TestTransaction` that exercises a service method using Panache's `PanacheEntityBase.update()` or a raw `EntityManager.createQuery(...).executeUpdate()`.

### What was tried (didn't work)
- Checking the JPQL update syntax — it was correct; the DB was actually updated
- Running the test in isolation — same result; the assertion always used the old value
- Checking if `@TestTransaction` was rolling back — it wasn't (the update ran, just wasn't visible to the same-session query)

### Root cause

JPQL bulk `UPDATE` and `DELETE` statements execute directly against the database and **bypass Hibernate's first-level cache** (the persistence context). After a bulk update, the entity objects already loaded in the current session still hold their pre-update state. When the test calls `Entity.find("instanceId", id)` afterwards, Hibernate finds the entity in the first-level cache and returns the cached (stale) object without hitting the database.

This means a test can assert `assertEquals("stale", inst.status)` on the result of a query, get back `"online"`, and the assertion passes — if it was written as `assertEquals("online", inst.status)` expecting the old value. If written correctly, the test fails with no hint that caching is involved.

### Fix

After the bulk update and before the assertion query, call `Entity.getEntityManager().clear()` to evict all entities from the first-level cache. The next query will hit the database and return the updated state:

```java
@Test
@TestTransaction
void markStaleChangesStatusForOldInstances() throws InterruptedException {
    instanceService.register("stale-agent", "Will go stale", List.of());
    Thread.sleep(1100);
    instanceService.markStaleOlderThan(1); // calls PanacheEntityBase.update(...)

    // Bulk JPQL update bypasses first-level cache — clear it to see DB state
    Instance.getEntityManager().clear();

    Instance inst = instanceService.findByInstanceId("stale-agent").orElseThrow();
    assertEquals("stale", inst.status); // now reads from DB
}
```

`PanacheEntityBase.getEntityManager()` is a static method available on all Panache active-record entities.

### Why non-obvious

The test produces a passing assertion on stale data with no warning. The Quarkus/Panache documentation doesn't mention this interaction between `@TestTransaction`, JPQL bulk updates, and the first-level cache. A developer reading the test would assume it verifies the bulk update behaviour; it actually verifies nothing. The fix (`getEntityManager().clear()`) requires knowing that Panache exposes a static EntityManager accessor, which is itself not prominently documented.
