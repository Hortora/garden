## Evolve @Tool MCP method signatures without breaking test callers using non-@Tool convenience overloads

**ID:** GE-20260414-75d145
**Stack:** quarkus-mcp-server 1.x, Quarkus 3.x
**Labels:** `#mcp` `#api-evolution` `#testing` `#quarkus` `#backward-compat`

### The problem

Adding parameters to a `@Tool`-annotated method in quarkus-mcp-server breaks all existing test callers — tests call the method directly by its Java signature, not via MCP. You can't have two `@Tool` annotations with the same `name`, and renaming breaks the MCP surface.

### The technique

Add non-`@Tool` convenience overloads that delegate to the full signature with `null` defaults. The `@Tool` annotation stays on the full-parameter method. Old signatures become thin convenience overloads used by existing tests and internal callers — zero test churn.

```java
// 4-arg convenience overload — no @Tool, used by tests and internal callers
public ChannelDetail createChannel(String name, String desc, String semantic, String barrierContributors) {
    return createChannel(name, desc, semantic, barrierContributors, null, null, null, null);
}

// 6-arg overload for intermediate callers
public ChannelDetail createChannel(String name, String desc, String semantic, String barrierContributors,
        String allowedWriters, String adminInstances) {
    return createChannel(name, desc, semantic, barrierContributors, allowedWriters, adminInstances, null, null);
}

// @Tool is only on the full signature — this is what MCP exposes
@Tool(name = "create_channel", description = "...")
@Transactional
public ChannelDetail createChannel(
        @ToolArg(name = "name", ...) String name,
        @ToolArg(name = "description", ...) String description,
        @ToolArg(name = "semantic", ..., required = false) String semantic,
        @ToolArg(name = "barrier_contributors", ..., required = false) String barrierContributors,
        @ToolArg(name = "allowed_writers", ..., required = false) String allowedWriters,
        @ToolArg(name = "admin_instances", ..., required = false) String adminInstances,
        @ToolArg(name = "rate_limit_per_channel", ..., required = false) Integer rateLimitPerChannel,
        @ToolArg(name = "rate_limit_per_instance", ..., required = false) Integer rateLimitPerInstance) {
    // implementation
}
```

Applied across four successive feature additions (4→5→6→8 args), zero test files changed.

### Why this is non-obvious

Most developers would update all test call sites (churn) or introduce a builder/request-object (complexity). The overload chain is simpler and keeps both the existing test corpus and the MCP surface clean. The key insight is that quarkus-mcp-server only registers the `@Tool`-annotated method — bare overloads are invisible to the MCP framework and act as ordinary Java methods.
