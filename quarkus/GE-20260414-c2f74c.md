## Symptom

Calling a reactive Panache method (e.g. `entity.persist()`, `Entity.find(...)`) from a
`@QuarkusTest` test method and calling `.await().atMost()` on the resulting `Uni` throws:

```
java.lang.IllegalStateException: No current Mutiny.Session found
```

Or, when a CDI method annotated with `@WithSession("datasource-name")` is called from
a plain JUnit thread, the interceptor fires and throws:

```
java.lang.IllegalStateException: No current Vertx context found
```

## Context

Quarkus's Hibernate Reactive requires operations to run on a Vert.x event loop thread
with an active Mutiny session. Test methods run on a regular JUnit thread, not a Vert.x
thread, so there is no session context.

Calls that fail before reaching any Panache/Hibernate operation (e.g. validation
that throws early) work fine — the exception is a sign the code reached the DB layer.

The `@WithSession` variant fires immediately when the annotated CDI method is invoked
(interceptor runs synchronously at method call time), so the error manifests before any
reactive subscription rather than at the `.await()` call.

## What was tried

Calling `someService.doReactiveWork().await().atMost(Duration.ofSeconds(5))` directly
from a `@Test` method in a `@QuarkusTest` class.

## Root cause

The Mutiny session is bound to the Vert.x duplicated context. Direct `.await()` from a
non-Vert.x thread has no context and no session.

## Solution 1 — @RunOnVertxContext + UniAsserter (Quarkus-native, preferred)

Annotate the test method with `@RunOnVertxContext` from `quarkus-test-vertx` and accept
a `UniAsserter` parameter instead of calling `.await()`. Do NOT call `.await()` inside
`@RunOnVertxContext` — it would deadlock the event loop.

```java
@Test
@RunOnVertxContext
void myReactiveTest(UniAsserter asserter) {
    asserter.assertThat(
        () -> someService.doReactiveWork(),
        result -> assertThat(result).isNotNull());
}
```

Dependency (if not already present):
```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-test-vertx</artifactId>
    <scope>test</scope>
</dependency>
```

## Solution 2 — Manual Vert.x context dispatch

For Quarkus projects without `quarkus-test-vertx`, or when multi-step control flow
is needed beyond what `UniAsserter` supports:

```java
import io.quarkus.vertx.core.runtime.context.VertxContextSafetyToggle;
import io.smallrye.common.vertx.VertxContext;

Uni<T> result = Uni.createFrom()
    .<T>deferred(() -> someService.doReactiveWork())
    .runSubscriptionOn(command -> {
        Context dc = VertxContext.getOrCreateDuplicatedContext(vertx);
        VertxContextSafetyToggle.setContextSafe(dc, true);
        dc.runOnContext(v -> command.run());
    });

result.await().atMost(Duration.ofSeconds(5));
```

## Why non-obvious

The error message mentions "Mutiny.Session" or "Vertx context" but the real issue is
thread dispatch — developers search for session/transaction configuration fixes when
the actual fix is moving the test onto the event loop. The symptom only appears in
tests, not in normal Quarkus request handling (which already runs on the event loop).

For the `@WithSession` variant, the error fires before any visible Panache code because
the CDI interceptor runs at method invocation time, not at subscription time — making the
stack trace misleading about where the actual session requirement originates.
