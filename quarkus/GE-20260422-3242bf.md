## Context

A library module needs to use a CDI bean (e.g. `EmbeddingModel`) that may or may not be present in the consuming application's classpath. Direct `@Inject EmbeddingModel` would fail with `UnsatisfiedResolutionException` at CDI startup when no provider is configured.

## Technique

Inject `Instance<T>` instead of `T` directly. `Instance<T>` is always satisfiable â€” CDI never fails to inject it. Call `instance.isResolvable()` at use time to check whether a bean actually exists:

```java
@Inject
public EmbeddingSkillMatcher(final Instance<EmbeddingModel> modelInstance) {
    this.model = modelInstance.isResolvable() ? modelInstance.get() : null;
}
```

At call time, check and degrade gracefully:

```java
public double score(SkillProfile profile, SelectionContext ctx) {
    if (model == null) {
        LOG.warn("No EmbeddingModel available â€” returning -1.0");
        return -1.0;
    }
    // ... call model
}
```

For unit tests, inject a concrete `EmbeddingModel` directly via a package-private test constructor â€” no `Instance<>` wrapper needed:

```java
// Test constructor â€” bypasses CDI
EmbeddingSkillMatcher(final EmbeddingModel model) {
    this.model = model;
}
```

## Why this is non-obvious

Most developers either:
1. Use direct `@Inject T` (fails at boot when bean is absent)
2. Use `@Inject @Default T` with a no-op default bean (pollutes CDI context)
3. Wrap everything in `@IfBuildProperty` conditions (ties optional behaviour to build time)

`Instance<T>` solves this purely at runtime, with no build-time conditions and no extra beans. The pattern is documented in the CDI spec but rarely seen in practice because the common case is non-optional injection.

### Why this fix

`Instance<T>` is the CDI-native solution for optional dependencies. Alternatives (no-op default beans, build-time conditions) add complexity and make testing harder. `Instance<T>` keeps the code clean and the library usable in both AI-enabled and AI-disabled contexts.
