## Flyway migrations written against H2 silently fail on PostgreSQL — H2 accepts non-standard SQL types

### Symptom

```
FlywayMigrateException: Failed to execute script V13__some_migration.sql
FlywaySqlScriptException: Failed to execute script V13__some_migration.sql
PSQLException: ERROR: type "double" does not exist
```

Flyway migrations run perfectly in CI (H2) but fail immediately on first PostgreSQL deployment.

### Context

A Quarkus project uses H2 in-memory for all `@QuarkusTest` runs and Flyway for schema
management. Migrations were written and tested only against H2. On first run against a real
PostgreSQL instance (production, staging, or a PostgreSQL-backed test), Flyway fails.

### Root cause

H2 accepts many non-standard SQL type aliases that PostgreSQL rejects:

| H2 accepts | PostgreSQL requires |
|-----------|-------------------|
| `DOUBLE` | `DOUBLE PRECISION` or `FLOAT8` |
| `TINYINT` | `SMALLINT` |
| `DATETIME` | `TIMESTAMP` |
| `MEDIUMTEXT` | `TEXT` |
| `AUTO_INCREMENT` | `GENERATED ALWAYS AS IDENTITY` or `SERIAL` |

H2 in PostgreSQL compatibility mode catches some of these, but not all. Migrations pass
`mvn test` for months with no indication of the incompatibility.

### Fix

Use standard SQL types in all Flyway migration scripts:

```sql
-- Instead of:
confidence_score DOUBLE,

-- Write:
confidence_score DOUBLE PRECISION,
```

Run at least one integration test against a real PostgreSQL instance (Testcontainers) as
part of CI to catch these before production. H2 PostgreSQL compatibility mode helps but
is not a full substitute.

### Why this is non-obvious

H2's permissiveness is intentional — it aims to be compatible with multiple databases for
testing convenience. The migration passes every test run and there is no linter or Flyway
validator that warns about non-standard types. The failure only surfaces on first contact
with a real PostgreSQL instance.
