## Symptom

Adding `quarkus-hibernate-reactive-panache` with `<optional>true</optional>` to `pom.xml` causes `@QuarkusTest` tests to fail with:

```
ConfigurationException: Unable to find datasource '<default>' for persistence unit '<default>':
No pool has been defined for persistence unit <default>
```

Tests that were passing before now fail to start the Quarkus application.

## Context

Occurs when:
1. A library/extension module adds `quarkus-hibernate-reactive-panache` as an optional dep (to support reactive consumers without forcing the dep on everyone)
2. The same module has `@QuarkusTest` integration tests using H2 JDBC

## Root Cause

Maven `<optional>true</optional>` prevents the dep from being transitively propagated to downstream consumers — but it does **not** remove the dep from the current module's own test classpath. The dep is still fully present during `mvn test`.

With the reactive Panache extension on the classpath, Quarkus's build-time bootstrap activates the Hibernate Reactive extension. That extension requires a Vert.x reactive datasource — not a JDBC pool. H2 has no Vert.x reactive driver, so no pool can be created and Quarkus fails to start.

`quarkus.arc.exclude-types` does NOT fix this — it only affects CDI bean scan, not Quarkus extension activation.

## Fix

Remove `quarkus-hibernate-reactive-panache` from the module's `pom.xml` entirely. Provide only the SPI interface (which only needs `io.smallrye.mutiny.Uni`, already on the Quarkus classpath). Consumers who want reactive add the dep to their own module.

```xml
<!-- DON'T add this to a module with @QuarkusTest using H2 -->
<!-- <dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-hibernate-reactive-panache</artifactId>
  <optional>true</optional>
</dependency> -->
```

If a pre-built reactive implementation is needed, put it in a **separate Maven module** (e.g. `quarkus-mylib-reactive`) so the reactive dep never lands on the blocking test classpath.

## Why This Is Non-Obvious

The Maven `optional` flag is commonly understood to mean "this dep won't bother consumers" — developers assume it also means "won't cause problems in my own tests." The Quarkus extension activation is a build-time concern triggered by classpath presence, not by CDI bean wiring or config. The error message (`No pool has been defined for persistence unit <default>`) doesn't mention reactive or optional dependencies at all.
