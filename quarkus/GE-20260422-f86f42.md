## What exists but isn't documented

When migrating a Quarkus project to a **named persistence unit** (e.g. `quarkus.hibernate-orm.qhorus.*`) while depending on a library whose CDI beans inject `@Default EntityManager` (e.g. `quarkus-ledger`), the default persistence unit must **remain active** even though no application entities use it.

If you only configure `quarkus.datasource.qhorus.*` and leave the default datasource unconfigured, Quarkus boots but the library's `@Default EntityManager` injection fails at runtime with:

```
Unsatisfied dependency for type EntityManager and qualifiers [@Default]
```

## Required setup

**Test `application.properties`** — both datasources must exist:

```properties
# Default datasource — for library beans that inject @Default EntityManager
quarkus.datasource.db-kind=h2
quarkus.datasource.jdbc.url=jdbc:h2:mem:test;DB_CLOSE_DELAY=-1
quarkus.datasource.username=sa
quarkus.datasource.password=

# Named 'qhorus' datasource — for application entities
quarkus.datasource.qhorus.db-kind=h2
quarkus.datasource.qhorus.jdbc.url=jdbc:h2:mem:test;DB_CLOSE_DELAY=-1
quarkus.datasource.qhorus.username=sa
quarkus.datasource.qhorus.password=

# Default PU needs a packages config to avoid scanning all entities
# Point it at a config-only package with no @Entity classes
quarkus.hibernate-orm.packages=io.your.project.config

# Both PUs need schema generation in tests
quarkus.hibernate-orm.database.generation=drop-and-create
quarkus.hibernate-orm.qhorus.database.generation=drop-and-create
```

The key insight: `quarkus.hibernate-orm.packages=<config-only-package>` assigns the default PU a package that contains no `@Entity` classes. This satisfies Quarkus's requirement that every active PU declares at least one package, without accidentally routing application entities to the default PU.

## Context

Triggered by any Quarkus library (e.g. `quarkus-ledger`) that ships CDI beans with `@Inject EntityManager em` using the default qualifier. These beans cannot be updated (library code). The consuming application must keep the default PU alive to satisfy the injection point.

## Why non-obvious

Quarkus's named datasource documentation assumes the consuming app controls all its persistence. When a library dependency has `@Default EntityManager` injection, it creates a hidden constraint: the default PU must remain even after migrating all app entities to a named PU. There is no warning, error, or documentation covering this scenario. Discovery only possible by reading the library's source or hitting the boot failure.
