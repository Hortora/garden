## Quarkus `quarkus:dev` silently skips and returns BUILD SUCCESS if the `build` goal is missing from plugin executions

**ID:** GE-0065
**Stack:** Quarkus 3.x with Maven (all versions)
**Symptom:** Running `mvn quarkus:dev` completes immediately with BUILD SUCCESS and a WARNING — the application never starts, the server never binds to a port. The warning reads: "Skipping quarkus:dev as this is assumed to be a support library."
**Context:** Any Maven project where the quarkus-maven-plugin is declared but its `<executions>` block is missing or doesn't include the `build` goal.

### What was tried (didn't work)

- `mvn quarkus:dev -Dquarkus.profile=sc2` → BUILD SUCCESS, no server started, no error
- Checking application.properties for mistakes → fine
- Checking the port → nothing listening

### Root cause

Quarkus's Maven plugin uses the presence of the `build` goal in `<executions>` as a signal that the module is a runnable application (not a library). Without it, the plugin assumes the module is a support/library module and skips `quarkus:dev` entirely. The WARNING is easy to miss because it's buried in Maven INFO output and the build still returns exit code 0.

### Fix

Add the `<executions>` block to the quarkus-maven-plugin declaration:

```xml
<plugin>
    <groupId>io.quarkus.platform</groupId>
    <artifactId>quarkus-maven-plugin</artifactId>
    <version>${quarkus.platform.version}</version>
    <extensions>true</extensions>
    <executions>
        <execution>
            <goals>
                <goal>build</goal>
                <goal>generate-code</goal>
                <goal>generate-code-tests</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

Note: `quarkus-maven-plugin:create` generates this block correctly. The issue occurs when the plugin is added manually or when an old pom.xml is updated without including executions.

### Why this is non-obvious

BUILD SUCCESS with exit code 0 is never expected to mean "I did nothing". The WARNING is the only signal and it's easy to miss. Developers typically assume a port binding issue, an application startup failure, or a wrong profile — not that the dev mode was never attempted at all.
