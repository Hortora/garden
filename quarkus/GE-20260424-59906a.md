## Behaviour

In a Quarkus multi-module Maven project, if a plain library module (no Quarkus Maven plugin, `packaging=jar`) contains `@ApplicationScoped` CDI beans and is added as a `<dependency>` of the Quarkus application module, those beans are NOT discovered by CDI. Injection of these beans at runtime fails with "Unsatisfied dependency for type X".

## Context

Occurs when splitting a single-module Quarkus app into a library module + Quarkus app module. The library beans compile fine and the JAR is on the classpath, but Quarkus CDI ignores them.

## Root Cause

Quarkus CDI bean discovery only indexes classes in JARs that are either: (a) the application itself, (b) have a Jandex index (`META-INF/jandex.idx`), or (c) are explicitly listed in `quarkus.index-dependency`. Plain JARs without Jandex are not scanned.

## Fix

Add to `claudony-app/src/main/resources/application.properties`:

```properties
quarkus.index-dependency.mylib.group-id=com.example
quarkus.index-dependency.mylib.artifact-id=mylib-core
```

Or add the Jandex Maven plugin to the library module's `pom.xml`:

```xml
<plugin>
    <groupId>io.smallrye</groupId>
    <artifactId>jandex-maven-plugin</artifactId>
    <version>3.1.2</version>
    <executions>
        <execution>
            <goals><goal>jandex</goal></goals>
        </execution>
    </executions>
</plugin>
```

## Why Undocumented

The Quarkus docs for multi-module projects mention the Jandex plugin as an option but do not clearly state that without it (or `index-dependency`), CDI beans in library JARs are silently ignored. The error message "Unsatisfied dependency" doesn't hint at the indexing problem — it looks like the bean simply doesn't exist.
