### What happened

Adding a dependency that pulls in `quarkus-hibernate-reactive-panache` (e.g. embedding `quarkus-qhorus`) caused the entire `@QuarkusTest` context to fail to start. Symptom: "Failed to start quarkus" / HibernateReactive boot failure cascade — all tests fail with a startup error, not a test failure.

### Root cause

H2 does not have a native reactive (Vert.x) database driver. When `hibernate-reactive-panache` is on the classpath and `quarkus.datasource.reactive` is not explicitly disabled, Quarkus tries to create a reactive connection pool, can't find a compatible H2 reactive driver, and throws at startup.

This affects all tests in the run, not just those that use Hibernate Reactive — the boot failure is total.

### Fix

Add to `src/main/resources/application.properties`:

```properties
%test.quarkus.datasource.reactive=false
```

This tells Quarkus to skip reactive datasource initialisation in the test profile. Hibernate Reactive falls back to JDBC mode (Hibernate ORM) for H2 tests. Hibernate ORM and Hibernate Reactive entities coexist fine in this mode.

Pattern sourced from the `quarkus-qhorus` project's own test properties — it is not documented in any Quarkus guide.

### Why this is non-obvious

The fix is required even when your own code never uses Hibernate Reactive directly. Any transitive dependency on `quarkus-hibernate-reactive-panache` is enough to trigger the startup failure. The error message ("HibernateReactive boot failure") doesn't mention H2 or the reactive driver — you need to trace the cause chain.
