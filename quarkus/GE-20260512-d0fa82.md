### Symptom

`java.sql.SQLException: Failed to enlist. Check if a connection from another datasource is already enlisted to the same transaction` — accompanied by `Enlisted connection used without active transaction` Narayana warnings. Tests fail; production (PostgreSQL) works fine.

### Context

A single `@Transactional` service method writes to two different Quarkus datasources (e.g. default for domain entities, a named `qhorus` datasource for ledger entities). Production uses PostgreSQL which supports XA natively. Tests use two separate H2 in-memory databases.

### What was tried

- Pointed both test datasources at the same H2 URL — Agroal still sees two distinct pool instances and rejects the second enlistment.
- `@Transactional(TxType.REQUIRES_NEW)` on the second write — avoids the XA error but breaks atomicity.
- Removing `@Transactional` from test methods — the service's own `@Transactional` still spans two datasources; same error.

### Root cause

Agroal's default local-transaction mode raises a guard when a second datasource tries to join an existing JTA transaction — local (non-XA) connections cannot participate in a distributed JTA transaction. H2 supports XA but must be explicitly configured; Quarkus doesn't enable XA mode for H2 by default.

### Fix

Add to test `application.properties`:

```properties
quarkus.datasource.jdbc.transactions=xa
quarkus.datasource.<named-ds>.jdbc.transactions=xa
```

Apply to every datasource that participates in multi-datasource transactions. Quarkus's H2 support wires up H2's XA datasource class automatically when `transactions=xa` is set. Run `mvn clean test` the first time to clear any stale compiled state.

### Why this is non-obvious

"Failed to enlist" sounds like a connection pool or concurrency problem, not a transaction mode configuration issue. Production works transparently. The Agroal error message gives no hint that `transactions=xa` is the fix — it reads as a runtime state problem, not a misconfiguration.
