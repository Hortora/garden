## Context

When a JPA-backed method is called from a Vert.x IO thread (e.g. by a library that invokes a CDI bean from an event-bus handler without `@Blocking`), the default `@Transactional(REQUIRED)` annotation causes:

```
io.quarkus.runtime.BlockingOperationNotAllowedException:
  Cannot start a JTA transaction from the IO thread.
```

## Technique

Change read-only JPA methods from `@Transactional` (which defaults to `REQUIRED`) to `@Transactional(TxType.SUPPORTS)`:

```java
import jakarta.transaction.Transactional;
import jakarta.transaction.Transactional.TxType;

@Override
@Transactional(TxType.SUPPORTS)
public List<WorkerSummary> findCompletedWorkers(UUID caseId) {
    // SELECT query — safe to run without an active transaction.
    return em.createQuery(
            "SELECT e FROM CaseLedgerEntry e WHERE e.caseId = :caseId ...",
            CaseLedgerEntry.class)
        .setParameter("caseId", caseId)
        .getResultList();
}
```

`SUPPORTS` semantics:
- **With active transaction:** joins it (participates, sees uncommitted data in the same transaction)
- **Without active transaction:** proceeds without one — no JTA begin/end, no IO-thread check triggered

This is correct and safe for read-only queries: SELECTs work without a transaction in READ_COMMITTED isolation (each statement sees the latest committed state). If the same method is also called inside a `@TestTransaction`, it correctly joins that transaction.

## Why This Is Non-Obvious

Most developers annotate JPA methods with `@Transactional` (REQUIRED) as a blanket rule, or not at all (relying on the persistence context). The distinction between `REQUIRED` and `SUPPORTS` for read-only methods is easy to overlook, and the failure mode — `BlockingOperationNotAllowedException` from inside a library call chain — doesn't obviously point to the `@Transactional` annotation on your own method.
