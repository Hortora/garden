## Run PostgreSQL Surefire execution first to force correct Quarkus augmentation in a dual-database test module

### Technique

To run PostgreSQL dialect validation tests alongside H2 tests in the same Maven module,
declare a dedicated Surefire execution for the PostgreSQL tests **before** the H2 execution
in the POM. Quarkus augments once per JVM — the first execution wins. The second execution
detects the mismatch and re-augments.

```xml
<plugin>
  <artifactId>maven-surefire-plugin</artifactId>
  <executions>
    <!-- Disable default execution — control ordering explicitly -->
    <execution>
      <id>default-test</id>
      <phase>none</phase>
    </execution>

    <!-- PostgreSQL FIRST: fresh JVM, db-kind as system property (pre-augmentation) -->
    <execution>
      <id>postgres-dialect-test</id>
      <phase>test</phase>
      <goals><goal>test</goal></goals>
      <configuration>
        <includes>
          <include>**/PostgresDialectValidationTest.java</include>
        </includes>
        <reuseForks>false</reuseForks>
        <systemPropertyVariables>
          <quarkus.datasource.db-kind>postgresql</quarkus.datasource.db-kind>
          <quarkus.datasource.devservices.enabled>false</quarkus.datasource.devservices.enabled>
        </systemPropertyVariables>
      </configuration>
    </execution>

    <!-- H2 SECOND: Quarkus detects mismatch, re-augments for H2 -->
    <execution>
      <id>h2-tests</id>
      <phase>test</phase>
      <goals><goal>test</goal></goals>
      <configuration>
        <excludes>
          <exclude>**/PostgresDialectValidationTest.java</exclude>
        </excludes>
      </configuration>
    </execution>
  </executions>
</plugin>
```

The PostgreSQL test class uses `@QuarkusTestResource` to start the container and inject the
JDBC URL before Quarkus starts. Flyway is bypassed (migrations may use H2-only SQL) and
Hibernate schema generation is used instead:

```java
@QuarkusTest
@QuarkusTestResource(value = PostgresTestResource.class, restrictToAnnotatedClass = true)
class PostgresDialectValidationTest { ... }

public class PostgresTestResource implements QuarkusTestResourceLifecycleManager {
    static final PostgreSQLContainer<?> PG = new PostgreSQLContainer<>("postgres:16-alpine");

    @Override
    public Map<String, String> start() {
        PG.start();
        return Map.of(
            "quarkus.datasource.jdbc.url", PG.getJdbcUrl(),
            "quarkus.datasource.username", PG.getUsername(),
            "quarkus.datasource.password", PG.getPassword(),
            "quarkus.flyway.migrate-at-start", "false",
            "quarkus.hibernate-orm.database.generation", "drop-and-create");
    }

    @Override
    public void stop() { if (PG.isRunning()) PG.stop(); }
}
```

### Why this is non-obvious

Most developers assume you can switch databases per test class using `@TestProfile` — this
works for runtime config but not for build-time config like `db-kind`. The ordering trick
(PostgreSQL first, H2 second) exploits the Quarkus augmentation cache mismatch detection to
force re-augmentation for H2 after the PostgreSQL augmentation has already run. The
`db-kind` must be a Surefire system property, not a test resource override, because test
resource config is applied after the augmentation cache decision.
