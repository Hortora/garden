## Reactive PanacheRepository<E> takes one type arg; use PanacheRepositoryBase<E,Id> for non-Long primary keys

**ID:** GE-20260420-58520c
**Stack:** Quarkus 3.32.2, quarkus-hibernate-reactive-panache

**Symptom:** `implements PanacheRepository<MyEntity, UUID>` fails to compile with: `wrong number of type arguments; required 1`. The blocking Panache equivalent `PanacheRepository<MyEntity, UUID>` compiles fine (blocking takes two args).

### Root cause
In `quarkus-hibernate-reactive-panache`, the two interfaces are:
- `PanacheRepository<Entity>` — one type arg, assumes `Long` primary key
- `PanacheRepositoryBase<Entity, Id>` — two type args, any primary key type

In blocking Panache (`quarkus-hibernate-orm-panache`), both interfaces also exist but the naming maps to the same arity. The asymmetry between blocking and reactive Panache APIs causes confusion when porting code.

### Fix
```java
// ❌ Wrong — reactive PanacheRepository takes only one type arg
@ApplicationScoped
class MyRepo implements PanacheRepository<MyEntity, UUID> { }

// ✅ Correct — use PanacheRepositoryBase for UUID (or any non-Long) PKs
@ApplicationScoped
class MyRepo implements PanacheRepositoryBase<MyEntity, UUID> { }
```

### Why non-obvious
The blocking Panache `PanacheRepository<E, Id>` takes two args and works with UUID — so developers naturally write the same for reactive. The compiler error ("wrong number of type arguments; required 1") doesn't hint at the `PanacheRepositoryBase` alternative.
