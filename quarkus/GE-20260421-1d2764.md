## Symptom

Running `mvn test` a second time (or running a single @QuarkusTest class after a full suite) fails with:

```
java.lang.RuntimeException: Failed to start quarkus
Caused by: io.quarkus.runtime.QuarkusBindException: Port already bound: 8081: Address already in use
Tests run: 5, Failures: 0, Errors: 1, Skipped: 4
```

Zero tests are run, BUILD FAILURE. The error appears on the *first* @QuarkusTest class Surefire encounters.

## Context

Occurs when a previous `mvn test` run was interrupted (Ctrl-C, timeout, or the Quarkus server process wasn't fully torn down). The Quarkus embedded server stays bound to 8081 as an orphan process.

The failure diagnostic is misleading: the error count is 1 (not the number of tests), and 4 tests show as Skipped rather than Failed — making it look like only one test actually ran.

## What was tried

- Re-running `mvn test` immediately — same failure
- Assuming it was a test bug — wasted time on wrong diagnosis

## Root cause

`@QuarkusTest` starts an embedded Quarkus server. When the JVM is killed mid-run (e.g. by a timeout), the server process inherits from the JVM and may stay alive. The next Surefire fork finds port 8081 already bound.

## Fix

Before running tests:

```bash
lsof -ti :8081 | xargs kill -9 2>/dev/null
```

Or use an alternate port to avoid the conflict entirely:

```bash
mvn test -Dquarkus.http.test-port=9999
```

The alternate-port approach is more reliable for parallel sessions (e.g. two terminal tabs both running `mvn test`).

## Why this is non-obvious

The error message points to a startup failure, not a port conflict. The symptom of "1 error, N skipped" is different from a normal test failure. Nothing in Maven output says "another process is using the port" — you must manually diagnose via `lsof`.
