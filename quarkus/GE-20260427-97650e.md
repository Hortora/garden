## Symptom

Adding a second `@ApplicationScoped` class implementing `LedgerEntryRepository` causes Quarkus boot failure:

```
Found 4 deployment problems:
[1] Ambiguous dependencies for type LedgerEntryRepository and qualifiers [@Default]
    injection target: LedgerRetentionJob#ledgerRepo
    available beans: AgentMessageLedgerEntryRepository, MessageLedgerEntryRepository
```

The library (`quarkus-ledger`) injects `LedgerEntryRepository` by interface type, and CDI sees two candidates.

## Root cause

CDI requires exactly one active bean per injection point. `@ApplicationScoped` on both implementations makes both active. The quarkus-ledger library's internal beans (`LedgerRetentionJob`, `LedgerVerificationService`, `TrustScoreJob`) inject `LedgerEntryRepository` by interface — they cannot be changed.

## Fix

Mark the **new** implementation `@Alternative` during the transitional period while both coexist. The old implementation remains the active primary; the new one is injected only when explicitly referenced by concrete type.

```java
@Alternative
@ApplicationScoped
public class MessageLedgerEntryRepository implements LedgerEntryRepository {
    // ...
}
```

Once the old implementation is deleted, remove `@Alternative` — the new one becomes the sole active bean automatically.

## Why this is non-obvious

Tests that inject the concrete type (`MessageLedgerEntryRepository`) still compile and pass — CDI resolves concrete-type injections without ambiguity. The failure only manifests in the library's interface-typed injection points, which you don't control. The error message names the library beans, not your new class, so the root cause is not immediately obvious.
