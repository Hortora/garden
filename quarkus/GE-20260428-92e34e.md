## Context

Testing async CDI event pipelines where an `@ObservesAsync` handler does work (e.g. writes to a database) and you need to verify that work completed before asserting. The typical approach is `Awaitility.await().atMost(...).untilAsserted(...)` with polling — which adds latency and can be flaky.

## Technique

`Event.fireAsync()` returns a `CompletionStage<T>` that completes only after **all** `@ObservesAsync` handlers have finished — including their `@Transactional` commits. Calling `.toCompletableFuture().join()` gives deterministic, zero-polling synchronization:

```java
@Inject Event<MyEvent> events;
@Inject MyRepository repo;

@Test
@TestTransaction
void eventWritesToDatabase() {
    UUID id = UUID.randomUUID();

    // Fires the event and blocks until all @ObservesAsync handlers have committed.
    // By the time join() returns, the handler's @Transactional method has committed.
    events.fireAsync(new MyEvent(id, "WorkerExecutionCompleted"))
          .toCompletableFuture()
          .join();

    // Now safe to assert — no polling needed.
    List<MyRecord> results = repo.findByCase(id);
    assertThat(results).hasSize(1);
}
```

The `@ObservesAsync` handler runs in a separate thread with its own transaction context:
```java
@ApplicationScoped
public class MyEventCapture {
    @Transactional
    void onEvent(@ObservesAsync MyEvent event) {
        // This transaction commits before fireAsync()'s CompletionStage completes.
        em.persist(new MyRecord(event.id(), event.type()));
    }
}
```

## Why This Is Non-Obvious

Most developers reach for `Awaitility` + polling when testing async code. The CDI spec wording ("The returned CompletionStage ... is completed with the event") is ambiguous about whether "completed" means "dispatched" or "all observers finished". It means all observers finished — including their commits. This makes it possible to write deterministic async CDI event tests with no polling, no sleep, and no Awaitility dependency.
