## Context

Building a `casehub-testing` module to provide `@Alternative @Priority(1)` in-memory repository implementations that downstream `@QuarkusTest` classes can consume by just adding the module as a test dependency — no `quarkus.arc.selected-alternatives` config needed.

## Symptom

```
jakarta.enterprise.inject.UnsatisfiedResolutionException: Unsatisfied dependency
  for type io.casehub.engine.spi.CaseInstanceRepository and qualifiers [@Default]
```

Downstream `@QuarkusTest` fails to start even though `casehub-testing` is on the classpath and its `TestCaseInstanceRepository` carries `@Alternative @Priority(1) @ApplicationScoped`.

## Root Cause

Quarkus CDI bean discovery requires a Jandex index to scan beans from classpath JARs. If the testing helper JAR does not build a Jandex index (i.e., the `jandex-maven-plugin` or equivalent is not configured), Quarkus cannot see any of its beans — including `@Alternative @Priority(1)` classes. The annotation is present in the bytecode but the JAR is effectively invisible to CDI.

The `casehub-testing` pom.xml had no Jandex configuration, so its beans were never indexed.

## Fix

**Option A (recommended for reusable testing jars):** Add Jandex indexing to the helper module's pom.xml:

```xml
<plugin>
    <groupId>io.smallrye</groupId>
    <artifactId>jandex-maven-plugin</artifactId>
    <executions>
        <execution>
            <id>make-index</id>
            <goals><goal>jandex</goal></goals>
        </execution>
    </executions>
</plugin>
```

**Option B (proven fallback):** Keep in-memory implementations in `casehub-persistence-memory` without `@Priority`, and activate them via `quarkus.arc.selected-alternatives` in `src/test/resources/application.properties`:

```properties
quarkus.arc.selected-alternatives=\
  io.casehub.persistence.memory.InMemoryCaseInstanceRepository,\
  io.casehub.persistence.memory.InMemoryCaseMetaModelRepository,\
  io.casehub.persistence.memory.InMemoryEventLogRepository
```

Option B is what every other module in the project already uses and is the safe, proven path.

## Why This Is Non-Obvious

The `@Alternative @Priority(1)` pattern is supposed to be self-activating — you expect it to "just work" when the JAR is on the classpath. The failure message says "unsatisfied dependency" which points to a missing implementation, not to a missing index. Nothing in the error or Quarkus startup log mentions Jandex.
