## Symptom

`cancelWait(corrId)` is called before `waitForReply(corrId, ...)`. The `waitForReply` call then times out instead of returning `status="cancelled"` immediately.

## Context

`waitForReply` calls `registerPendingReply(corrId, ...)` as its **first action** — this (re-)inserts the PendingReply row. So even if `cancelWait` deleted the row before `waitForReply` started, `waitForReply` creates a fresh row and the cancellation detection never fires.

The poll loop checks `pendingReplyExists(corrId)` on each iteration — cancellation is only detected if the row is deleted WHILE the loop is running.

## Root cause

The cancellation mechanism relies on row absence as its signal. But the blocking call re-inserts the row at startup, so pre-cancellation is indistinguishable from "first call" to the implementation.

## Fix

For tests: start `waitForReply` FIRST (on a background thread), then cancel from the main thread:

```java
// RIGHT — cancel WHILE wait is in its poll loop
Future<WaitResult> future = executor.submit(
    () -> tools.waitForReply("ch", corrId, 30, null));

Thread.sleep(400); // give it time to register and enter poll loop

tools.cancelWait(corrId); // fires while loop is running

WaitResult result = future.get(5, TimeUnit.SECONDS);
assertEquals("cancelled", result.status()); // ✓
```

For production: `cancelWait` is always called from a separate agent/human while the waiting agent is blocked — this works correctly.

## Why this is non-obvious

Pre-cancellation seems equivalent to concurrent cancellation. The difference is invisible until you trace through the implementation: the blocking method's own startup re-creates the exact row whose absence is the cancellation signal.
