## Symptom

`@Observes(during = TransactionPhase.AFTER_SUCCESS)` observer method is never called
in integration tests. No error, no warning — the event fires, the observer is skipped.

## Context

Writing a `@QuarkusTest` integration test for a PostgreSQL broadcaster that uses
`AFTER_SUCCESS` to publish `pg_notify()` only after the transaction commits (preventing
spurious events from rolled-back transactions). Tests fired the CDI event directly
without a transaction and the observer never ran.

## What was tried

Firing the CDI event with `Event.fire()` directly in the test method — observer not called.

## Root cause

CDI 4.0 spec: transactional observers (`AFTER_SUCCESS`, `AFTER_COMPLETION`, etc.)
are only invoked in the context of an active JTA transaction. If there is no active
transaction when the event fires, `AFTER_SUCCESS` observers are silently skipped.
This is correct CDI behaviour but easy to miss — you see the event fire in code,
but the observer never gets a chance to run.

## Fix

In `@QuarkusTest` integration tests, inject `UserTransaction` and wrap event firing:

```java
@Inject
UserTransaction tx;

void fireEvent(WorkItemQueueEvent event) throws Exception {
    tx.begin();
    queueEvents.fire(event);
    tx.commit(); // AFTER_SUCCESS observer fires here, after commit
}

// For rollback testing:
void fireAndRollback() throws Exception {
    tx.begin();
    queueEvents.fire(event);
    tx.rollback(); // AFTER_SUCCESS observer does NOT fire — correct
}
```

`@TestTransaction` can't be used here because it wraps the entire test method and
rolls back at the end — fine for DB cleanup but means you can't test commit vs rollback
behaviour within a single test.

## Why this is non-obvious

The CDI spec documents this behaviour but it's easy to miss. When debugging, you see
`Event.fire()` called in your code and naturally assume all observers will be notified.
The AFTER_SUCCESS observer appearing to not work looks like a CDI bean wiring problem
rather than a missing transaction.
