## Symptom

`@QuarkusTest` with `quarkus.datasource.reactive=true` (set via `@TestProfile`) and `vertx-jdbc-client` on the classpath fails at context startup:

```
No reactive datasource configured
```

or the reactive Hibernate session is never initialised and all `@RunOnVertxContext` tests are skipped/fail. Adding `quarkus.datasource.reactive.url=h2:mem:...` doesn't help.

## Root Cause

Quarkus's reactive pool factory is only registered by **native reactive client Quarkus extensions** — `quarkus-reactive-pg-client`, `quarkus-reactive-mysql-client`, etc. The raw `vertx-jdbc-client` jar is a JDBC bridge, but Quarkus does not wire it as a managed reactive datasource. H2 has no native async driver; there is no Quarkus reactive H2 extension.

## Fix

For reactive Hibernate Panache integration tests, use PostgreSQL with Docker:

1. Add `quarkus-reactive-pg-client` as a test dependency
2. Enable Dev Services (requires Docker): Quarkus auto-starts a PostgreSQL container
3. Set `quarkus.datasource.reactive=true` and `quarkus.datasource.db-kind=postgresql` in the test profile

If Docker is unavailable, write the reactive JPA tests as `@Disabled` with an explanatory comment. Unit tests via `InMemoryReactive*Store` (direct instantiation, `.await().indefinitely()`) still work without any datasource.

## Why This Is Non-Obvious

`vertx-jdbc-client` is described as a JDBC bridge that enables reactive-style access over blocking JDBC drivers. It's reasonable to assume adding it to the classpath would enable reactive H2. The Quarkus docs don't document this limitation explicitly — the reactive datasource section discusses PostgreSQL, MySQL, and MariaDB but doesn't mention that other drivers simply won't work regardless of the JDBC bridge.
