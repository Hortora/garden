## Symptom

A JAX-RS resource has both a bare `@GET` (no `@Path`) to handle root requests, and a wildcard `@GET @Path("{path:.*}")` to handle all other paths. The bare `@GET` is never invoked — root `GET /` is routed to the wildcard handler with `path = ""`.

```java
@GET
public Response root() { /* redirect to /ui/projects.html */ }

@GET
@Path("{path:.*}")
public Response serveStatic(@PathParam("path") String path) {
    // path is "" for GET / — root() never called
}
```

## Context

Occurs in static file serving resources that use `{path:.*}` to capture arbitrary paths. RESTEasy Reactive's path matching favours the template method over the no-path method for empty paths.

## What was tried

Assuming bare `@GET` would be preferred for exact root match — it is not, in RESTEasy Reactive.

## Root cause

RESTEasy Reactive's routing engine does not treat a bare `@GET` as more specific than `@GET @Path("{path:.*}")` when the request path is `/`. The wildcard template matches the empty segment.

## Fix

Add an early guard in the wildcard handler:

```java
@GET
@Path("{path:.*}")
public Response serveStatic(@PathParam("path") String path) {
    if (path == null || path.isEmpty()) return root(); // delegate to root handler
    // ... rest of static serving logic
}
```

## Why this is non-obvious

JAX-RS spec says more specific paths win — a bare `@GET` (path = "") should be more specific than a wildcard. RESTEasy Reactive's behaviour differs from the spec expectation and from classic RESTEasy.
