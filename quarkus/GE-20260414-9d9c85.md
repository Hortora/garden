## Context

Integration tests sometimes need to run a blocking operation (e.g. `waitForReply`) on a background thread while the main thread performs a concurrent action (e.g. `cancelWait`). The naive approach uses `Executors.newSingleThreadExecutor()`.

## Problem with raw ExecutorService

Raw JDK executor threads have no Quarkus CDI context. This means:
- `@Transactional` interceptors do not fire — beans behave as if non-transactional
- Database operations may silently fail or commit in unexpected ways
- `@RequestScoped` beans are unavailable

Symptom: the background thread's `@Transactional` method appears to work (no exception) but the data is not committed — the main thread cannot see it.

## Technique

Inject `ManagedExecutor` from MicroProfile Context Propagation instead of creating a raw executor:

```java
@Inject
ManagedExecutor executor;  // propagates CDI context to spawned threads

@Test
void testCancellationDuringWait() throws Exception {
    tools.createChannel("test-ch", "Test", null, null);
    String corrId = UUID.randomUUID().toString();

    // Background thread: CDI context propagated, @Transactional works
    Future<WaitResult> future = executor.submit(
        () -> tools.waitForReply("test-ch", corrId, 30, null));

    Thread.sleep(400); // give waitForReply time to register its PendingReply

    // Main thread: cancel while background thread is in poll loop
    tools.cancelWait(corrId);

    WaitResult result = future.get(5, TimeUnit.SECONDS);
    assertEquals("cancelled", result.status());
}
```

## Why this is non-obvious

`@Inject ManagedExecutor` looks like just another executor. The critical difference is that it propagates the current CDI context (including transaction management interceptors) to threads it creates. Most Java developers reach for `Executors.newSingleThreadExecutor()` first — there's no compile-time warning that `@Transactional` will be silently bypassed on raw threads.
