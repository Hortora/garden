## Symptom

`mvn test` fails with:

```
java.lang.IllegalArgumentException: Type mismatch between int and CompletionStage
```

No stack trace pointing at the `@Scheduled` method. The error message references `CompletionStage` even though the method has nothing to do with async code.

## Context

Quarkus Scheduler (backed by Quartz) requires `@Scheduled` methods to return `void` or `Uni<Void>`. Returning any other type — including `int` — triggers this error at augmentation time.

The error message is generated deep in Quarkus's scheduler extension processor and mentions `CompletionStage` because the processor checks for allowed return types (`void`, `Uni`, `CompletionStage`) and reports a mismatch using the first non-void type it knows about.

## Root cause

Quarkus Scheduler validates method signatures at build time. The list of allowed return types is `void`, `io.smallrye.mutiny.Uni<Void>`, and `java.util.concurrent.CompletionStage<Void>`. Anything else is rejected with this unhelpful message.

## Fix

Split into two methods:

```java
// @Scheduled wrapper — must return void
@Scheduled(identity = "my-job", every = "60s")
public void scheduledTrigger() {
    doWork(); // delegate to the int-returning method
}

// int-returning method — callable directly from tests
public int doWork() {
    // logic here
    return created;
}
```

Tests inject the service and call `doWork()` directly. The scheduler calls `scheduledTrigger()`.

## Why this is non-obvious

The error message "Type mismatch between int and CompletionStage" gives no indication that the issue is the method return type. A developer seeing this error would investigate async/reactive code, not the simple `int` return type on a scheduled method. The Quarkus docs show `void` in examples but do not explicitly state it is the only valid return type.
