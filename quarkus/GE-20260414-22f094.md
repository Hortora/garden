## H2 rejects 'key' as a column name — syntax error with no hint it's a reserved word

**ID:** GE-20260414-22f094
**Stack:** H2 2.4.x, Flyway, Quarkus 3.32.2 (H2 as test database)
**Symptom:** Flyway migration fails with `JdbcSQLSyntaxErrorException: Syntax error in SQL statement ... [*]key ... expected "identifier"`. The error marker (`[*]`) points at the column name `key` with no mention of reserved words.
**Context:** Creating a `shared_data` table with a column named `key` for a key-value store. The migration runs fine against PostgreSQL (production) but fails against H2 (test database).

### What was tried (didn't work)
- Checking for typos in the SQL — none found
- Checking Flyway version compatibility — not the issue
- Reading the H2 error message — only says "expected identifier", which looks like a syntax error elsewhere

### Root cause

`KEY` is a reserved word in H2 SQL grammar. H2 uses it in index and constraint syntax (`PRIMARY KEY`, `FOREIGN KEY`). PostgreSQL also reserves `KEY` but as a non-reserved keyword — it accepts `key` as a column name without quoting. H2 strictly rejects it.

The error message does not say "reserved word" — it says "expected identifier", pointing at the column name itself, which looks like an unrelated syntax issue.

### Fix

Rename the column to avoid the reserved word. The entity mapping must also be updated:

```sql
-- Migration: use data_key instead of key
CREATE TABLE shared_data (
    id       UUID         NOT NULL,
    data_key VARCHAR(255) NOT NULL,  -- not 'key'
    ...
    CONSTRAINT uq_shared_data_key UNIQUE (data_key)
);
```

```java
// Entity: map to the renamed column
@Column(name = "data_key", nullable = false)
public String key;  // Java field name can remain 'key'
```

Also update the `@Table` unique constraint annotation:
```java
@Table(uniqueConstraints = @UniqueConstraint(name = "uq_shared_data_key", columnNames = "data_key"))
```

### Why non-obvious

PostgreSQL allows `key` as a column name, so a migration written for PostgreSQL and tested locally will appear correct. H2 mode=PostgreSQL does not help — `KEY` remains reserved regardless of compatibility mode. The error message gives no hint that `key` is the problem word; it just says "expected identifier" with the cursor on the column name.
