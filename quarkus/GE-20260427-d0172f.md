### The technique
Annotate a `@QuarkusTest` method with `@io.quarkus.test.TestTransaction` to run the entire test inside a transaction that is automatically rolled back when the method exits. No `@AfterEach` cleanup, no manual `em.remove()`, no `DELETE FROM` queries needed.

```java
@QuarkusTest
class MyIntegrationTest {

    @Inject
    @MyQualifier
    EntityManager em;

    @Test
    @TestTransaction
    void insertsAndQueriesRealEntities() {
        MyEntity e = new MyEntity();
        e.field = "value";
        em.persist(e);
        em.flush();  // force SQL — still within the rollback-bound transaction

        var results = em.createQuery("SELECT e FROM MyEntity e", MyEntity.class).getResultList();
        assertThat(results).hasSize(1);
    }  // ← transaction rolled back here — no trace left in DB
}
```

### Why most developers don't reach for this
The reflex for JPA integration test cleanup is `@AfterEach` with `em.remove()` or `DELETE` queries. `@TestTransaction` is less well-known than `@Transactional` (which commits), and requires knowing that Quarkus provides this specific test annotation.

### Constraints
- Requires `@QuarkusTest` (full Quarkus context) — does not work in plain JUnit without the container
- `@Transactional` beans called within the test join the same transaction (REQUIRED semantics) — the rollback covers them too
- Does not work for tests that intentionally verify committed state (e.g. multi-session scenarios)
- Named datasources work correctly — specify the PU via the appropriate qualifier on the EntityManager injection
