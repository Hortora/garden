### The technique

Separate `@ObservesAsync` (coordinator) from `@Transactional` (policy) into two distinct `@ApplicationScoped` beans. The coordinator catches all transient failures and retries once; the policy bean gets a fresh transaction per call.

```java
// Bean 1: Coordinator — @ObservesAsync, NOT @Transactional
@ApplicationScoped
public class MultiInstanceCoordinator {

    @Inject MultiInstanceGroupPolicy policy;
    @Inject Event<WorkItemGroupLifecycleEvent> groupEvent;

    void onChildTerminal(@ObservesAsync WorkItemLifecycleEvent event) {
        final WorkItem child = (WorkItem) event.source();
        if (child.parentId == null || !child.status.isTerminal()) return;

        for (int attempt = 0; attempt < 2; attempt++) {
            try {
                WorkItemGroupLifecycleEvent evt = policy.process(child.id, child.status);
                if (evt != null) groupEvent.fireAsync(evt); // AFTER transaction commits
                return;
            } catch (Exception e) {
                // OCC or transient JTA failure — retry once
                // On retry: if policyTriggered=true (another thread won), process() returns null
            }
        }
    }
}

// Bean 2: Policy — @Transactional, NOT @ObservesAsync
@ApplicationScoped
public class MultiInstanceGroupPolicy {

    @Transactional
    public WorkItemGroupLifecycleEvent process(UUID childId, WorkItemStatus childStatus) {
        // Load fresh entities from DB — no detached-entity issues
        WorkItem child = WorkItem.findById(childId);
        WorkItemSpawnGroup group = WorkItemSpawnGroup.findByParentId(child.parentId);
        if (group == null || group.policyTriggered) return null;

        // Increment counter on managed entity — auto-flushed at commit
        if (childStatus == COMPLETED) group.completedCount++;
        else                           group.rejectedCount++;

        // Evaluate threshold... transition parent... return event to fire
        if (group.completedCount >= group.requiredCount) {
            group.policyTriggered = true;
            workItemService.completeFromSystem(group.parentId, "system", "threshold-met");
            return WorkItemGroupLifecycleEvent.of(..., GroupStatus.COMPLETED, ...);
        }
        return WorkItemGroupLifecycleEvent.of(..., GroupStatus.IN_PROGRESS, ...);
    }
}
```

### Why this is non-obvious

The obvious approach is to annotate the `@ObservesAsync` method itself with `@Transactional`. This fails because:
1. Self-invocation of `@Transactional` methods within the same bean bypasses CDI proxy interception
2. OCC exceptions during JTA commit propagate as wrapped exception types (not raw `OptimisticLockException`), making catch-and-retry unreliable from within the same transaction boundary
3. `fireAsync()` inside a `@Transactional` fires immediately — not after commit — so a rolled-back transaction still delivers the event

The two-bean split gives clean retry semantics: the non-transactional coordinator's `for` loop catches any exception from the policy, which always starts a fresh transaction (`REQUIRED` from a non-transactional context). The event fires only from the coordinator, after the transaction has successfully committed.

### Pairs with

- `policyTriggered` boolean on the group entity for exactly-once guarantee (OCC `@Version` prevents concurrent commits; `policyTriggered` makes retry idempotent)
- `completeFromSystem()`/`rejectFromSystem()` for system-initiated lifecycle transitions that bypass human-actor status guards
