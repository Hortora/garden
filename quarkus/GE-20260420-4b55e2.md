## Symptom

Three separate failures that appear when registering Micrometer Gauges that query JPA in a Quarkus `@ApplicationScoped` bean:

1. `GET /q/metrics` returns no workitem metrics at all (gauges not registered)
2. Gauges appear in `# HELP` and `# TYPE` lines but all values are `NaN`
3. `workitems_total` gauge not found in Prometheus output even though `workitems_by_status` is found

## Context

Attempting to register gauges in `@PostConstruct` of an `@ApplicationScoped` bean that also observes CDI events. The gauges query the JPA `WorkItemStore` at scrape time.

## Root cause

Three separate issues interact:

**Issue 1 — Lazy CDI init:** `@ApplicationScoped` beans are lazy. `@PostConstruct` only runs when the bean is first accessed. If nothing injects the metrics bean, it never initialises and gauges are never registered. A `/q/metrics` scrape hits an empty registry.

**Issue 2 — JPA context missing at scrape time:** Gauge lambdas execute in the Prometheus scrape thread (an HTTP request with request scope, but no JPA transaction). `entityManager.find()` or Panache queries throw `TransactionRequiredException`, making the gauge return `NaN`. Prometheus omits `NaN` gauge values from the output.

**Issue 3 — `_total` Prometheus naming collision:** A Gauge named `workitems.total` exports to Prometheus as `workitems_total`. Prometheus reserves the `_total` suffix for Counters. Some Micrometer-Prometheus integrations or tooling strip or rename these gauges, making them absent from the output even though `# HELP` lines appear for tagged gauges (which survive because they have a different effective name).

## Fix

All three fixes must be applied together:

```java
@ApplicationScoped
@Startup  // Fix 1: force eager init so @PostConstruct runs before first scrape
public class WorkItemMetrics {

    @Inject MeterRegistry registry;
    @Inject WorkItemStore workItemStore;

    @PostConstruct
    void registerMeters() {
        // Fix 3: use "workitems.active" not "workitems.total" — avoids _total suffix
        Gauge.builder("workitems.active", this, WorkItemMetrics::activeCount)
             .description("Non-terminal WorkItems")
             .register(registry);
    }

    // Fix 2: public + @Transactional so CDI proxy wraps the scrape call in a JPA TX
    @Transactional
    public double activeCount() {
        return workItemStore.scanAll().stream()
            .filter(wi -> wi.status != null && !wi.status.isTerminal())
            .count();
    }
}
```

For unit-testable aggregation logic, extract it as a pure static method that takes a `List<WorkItem>` — call the static from the `@Transactional` instance method.

## Why this is non-obvious

Each failure looks like a different problem. The missing gauges look like a registration bug. The NaN values look like a JPA transaction scope issue (which it is, but `@Transactional` on a private method doesn't work — it must be public for CDI interception). The `workitems_total` absence is explained only by knowing the Prometheus naming convention for counters. All three must be fixed simultaneously.
