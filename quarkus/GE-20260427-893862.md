### Context
A CDI observer needs to fire side effects (e.g. send an HTTP notification, write to an audit system) when a database mutation succeeds, but must NOT fire if the originating transaction is rolled back. The default `@Observes` fires synchronously within the transaction — if the transaction rolls back, the observer has already run.

### Technique

```java
@ApplicationScoped
public class NotificationDispatcher {

    @Transactional(Transactional.TxType.NOT_SUPPORTED)
    void onWorkItemEvent(
            @Observes(during = TransactionPhase.AFTER_SUCCESS)
            final WorkItemLifecycleEvent event) {
        // Fires only after the originating transaction commits successfully.
        // TxType.NOT_SUPPORTED: loads data (e.g. notification rules) outside
        // any transaction — avoids joining the now-committed parent tx context.
        loadRulesAndDispatch(event);
    }
}
```

Key combination:
1. `during = TransactionPhase.AFTER_SUCCESS` — CDI delivers the event only after the originating transaction commits. A rollback produces no delivery.
2. `@Transactional(TxType.NOT_SUPPORTED)` — the observer runs outside any transaction, so DB reads (loading rules) open fresh connections rather than trying to join a now-stale context.

For async dispatch (e.g. HTTP webhooks that must not block the calling thread):

```java
CompletableFuture.runAsync(() -> channel.send(payload), virtualThreadExecutor);
```

### Why this is non-obvious
`@Observes` without `during` fires inside the transaction — side effects happen even if the transaction rolls back. Developers who know `AFTER_SUCCESS` exists often pair it with `@Transactional(REQUIRED)` by habit, which tries to join a transaction that no longer exists after commit. `NOT_SUPPORTED` is the correct pairing but not obvious.

### Constraints
JTA transactions must be active on the originating call (standard for `@Transactional` Quarkus services). Does not apply to code that runs without JTA.
