Two corrections to the original entry:

**1. Interface signature correction:** The first parameter is `WorkflowMutablePosition`, not `WorkflowContext`:

```java
// Correct signature (7.13.4.Final)
public interface TaskExecutorFactory extends ServicePriority {
    TaskExecutorBuilder<? extends TaskBase> getTaskExecutor(
        WorkflowMutablePosition position,  // NOT WorkflowContext
        Task task,
        WorkflowDefinition definition);
}
```

**2. Better extension point for custom `call:` types:** If your goal is to handle a specific `call: my-custom-fn` YAML step, `CallableTaskBuilder<CallFunction>` is the correct, targeted extension point â€” not `TaskExecutorFactory`.

`TaskExecutorFactory` replaces the *entire* task factory chain â€” including all built-in types (`call:`, `do:`, `switch:`, `for:`, etc.). Implementing it means you must delegate to `DefaultTaskExecutorFactory` for everything you don't handle, or you break the built-in task types.

`CallableTaskBuilder<CallFunction>` is a narrower SPI that `DefaultTaskExecutorFactory` discovers via `ServiceLoader.load(CallableTaskBuilder.class)` when routing `call:` tasks. You only need to handle `CallFunction` instances where `task.getCall()` equals your custom name; all others can fail fast in `init()`. Registered via: `META-INF/services/io.serverlessworkflow.impl.executors.CallableTaskBuilder`.

```java
// Targeted extension for call: casehub:dispatch
public class CasehubCallableTaskBuilder implements CallableTaskBuilder<CallFunction> {
    @Override
    public boolean accept(Class<? extends TaskBase> clazz) {
        return CallFunction.class.isAssignableFrom(clazz);
    }

    @Override
    public void init(CallFunction task, WorkflowDefinition def, WorkflowMutablePosition pos) {
        if (!"casehub:dispatch".equals(task.getCall())) {
            throw new UnsupportedOperationException("only handles casehub:dispatch");
        }
        // extract args from task.getWith().getAdditionalProperties()
    }

    @Override
    public CallableTask build() {
        return (workflowCtx, taskCtx, input) -> { /* async dispatch */ };
    }
}
```

Note: `CallFunction` and `FunctionArguments` are in `io.serverlessworkflow.api.types` (not the `.func` experimental subpackage). See GE-20260604-098217.

**`TaskExecutorFactory` remains valid** when you need to handle an entirely new structural task type (not a `call:` variant) â€” for example, a new top-level task keyword like `humanTask:` or `delegate:`.

### What it is (original, corrected above)

`io.serverlessworkflow.impl.executors.TaskExecutorFactory` is a Java SPI interface that allows external implementations to handle custom task types in quarkus-flow workflows. It is registered via `META-INF/services/io.serverlessworkflow.impl.executors.TaskExecutorFactory` and discovered at runtime through Java's `ServiceLoader` mechanism.

Not mentioned in any quarkus-flow or CNCF Serverless Workflow official documentation.

### How to register a custom implementation

```java
// 1. Implement the interface (use WorkflowMutablePosition, not WorkflowContext)
public class HumanTaskExecutorFactory implements TaskExecutorFactory {
    @Override
    public TaskExecutorBuilder<? extends TaskBase> getTaskExecutor(
            WorkflowMutablePosition position, Task task, WorkflowDefinition def) {
        if (/* task matches your custom structural type */) {
            return new HumanTaskExecutorBuilder(task, def);
        }
        return null;
    }
}

// 2. Register in META-INF/services/io.serverlessworkflow.impl.executors.TaskExecutorFactory
```

### Why it matters

This SPI is the integration point for projects that need to add custom *structural* task types to quarkus-flow workflows. For custom `call:` variants, prefer `CallableTaskBuilder<CallFunction>` (see correction above).
