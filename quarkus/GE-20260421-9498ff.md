## Symptom

A `WorkerSelectionStrategy` that counts active WorkItems per candidate (`countActive()`)
returns 0 for the delegating actor even though they have active items. This causes
least-loaded routing to incorrectly include the delegator as a candidate with low load.

## Context

In `WorkItemService.delegate()`, the flow is:
1. Retrieve WorkItem from store (Hibernate first-level cache)
2. Set `item.assigneeId = toAssigneeId` and `item.status = PENDING`
3. Call `assignmentService.assign(item, DELEGATED)` ← workload counted here
4. Call `workItemStore.put(item)` ← Hibernate flush

The strategy's `countActive()` queries the DB for `WHERE assigneeId = :delegator AND status IN (ASSIGNED, IN_PROGRESS, SUSPENDED)`.

## What was tried

Placing the strategy call AFTER `workItemStore.put()` — Hibernate flushes on `put()`,
updating `assigneeId` to null/toAssigneeId in the DB before the query runs. The delegator
appears to have zero active items.

## Root cause

Hibernate flushes dirty entities when `persistAndFlush()` is called (inside `put()`).
If `item.assigneeId` was already cleared before the strategy reads workload counts,
the DB reflects the new state — the delegator's item no longer appears as "assigned to them".
The strategy sees an inaccurate snapshot.

## Fix

Call the assignment strategy BEFORE mutating `item.assigneeId` or `item.status`:

```java
public WorkItem delegate(UUID id, String actorId, String toAssigneeId) {
    final WorkItem item = requireWorkItem(id);
    // ... validate state ...

    // CORRECT: strategy fires first, with delegator still showing their full workload
    assignmentService.assign(item, AssignmentTrigger.DELEGATED);

    // Then mutate state (strategy may have already set assigneeId)
    if (item.assigneeId == null || item.assigneeId.equals(actorId)) {
        item.assigneeId = toAssigneeId;
        item.status = WorkItemStatus.PENDING;
    }
    item.delegationChain = ...;
    item.delegationState = DelegationState.PENDING;

    final WorkItem saved = workItemStore.put(item);
    // Audit uses saved.assigneeId (actual result, not toAssigneeId param)
    audit(saved.id, "DELEGATED", actorId, "to:" + saved.assigneeId);
    ...
}
```

Also: use `saved.assigneeId` (not `toAssigneeId`) in the audit record — when the
strategy overrides the requested target, the audit must reflect who actually got the item.

## Why this is non-obvious

The bug is invisible unless the delegating actor has active items AND the test checks
routing outcomes after delegation. Standard Hibernate flush-on-put semantics are
well understood, but the interaction with a query-in-the-same-transaction strategy
call is subtle. The test infrastructure creates items and delegates them in one transaction,
making the timing hard to observe without specifically testing "does the delegator's
active count affect routing?".
