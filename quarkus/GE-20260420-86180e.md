## Symptom

```
jakarta.persistence.OptimisticLockException: Unexpected row count
(expected row count 1 but was 0) [delete from subclass_table where id=?]
```

Deletion succeeds for the base table row but fails for the subclass table row — or vice versa.

## Context

Occurs when:
1. Entities are loaded by one `EntityManager` (e.g., via `repo.listAll()` which uses the repo's CDI-injected EM)
2. Another `EntityManager` (e.g., the service's own `@Inject EntityManager`) calls `em.merge(detachedEntity)` then `em.remove(mergedEntity)`
3. The entity uses `@Inheritance(strategy = InheritanceType.JOINED)` with a base table and one or more subclass tables

## Root Cause

For `JOINED` inheritance, Hibernate must delete rows from **both** the base table and the subclass table, in the correct order. When `em.merge()` re-attaches a detached entity from a different EM context, Hibernate may reconstruct the entity as the base type only — losing the concrete subclass identity. `em.remove()` then issues the DELETE in the wrong order or against the wrong table, resulting in row count 0 for the subclass DELETE.

The error message ("Unexpected row count") suggests optimistic locking or concurrent modification — neither of which is the actual cause. The real issue is polymorphic type resolution on merge.

## Fix

Use `em.find()` in the **current** EntityManager context to obtain a properly typed, managed entity before calling `em.remove()`:

```java
// WRONG — entity loaded by a different EM; merge loses subclass identity
em.remove(em.contains(entry) ? entry : em.merge(entry));

// RIGHT — find in THIS EM context; Hibernate resolves the concrete subclass type
LedgerEntry managed = em.find(LedgerEntry.class, entry.id);
if (managed != null) {
    em.remove(managed);
}
```

`em.find()` issues a polymorphic `SELECT` that returns the concrete subclass type. `em.remove()` on that managed entity correctly deletes the subclass row first, then the base row.

## Why This Is Non-Obvious

The error message says "Unexpected row count" with a hint about optimistic locking — nothing in the message suggests cross-EM context or polymorphic type resolution. The pattern `em.contains(x) ? x : em.merge(x)` is commonly seen as the "safe" re-attachment idiom and works correctly for single-table entities. For JOINED inheritance it silently loses the subclass type.

### Why this fix

`em.find()` is guaranteed to return the correct polymorphic type because Hibernate uses the discriminator value from the base table row to select the right subclass. This is the same mechanism used when loading entities via JPQL. `em.merge()` does not make this guarantee for entities from a different persistence context.
