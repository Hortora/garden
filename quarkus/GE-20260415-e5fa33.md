## Quarkiverse extension-descriptor rejects deployment pom that depends on X-deployment unless runtime pom depends on X

### Symptom

`mvn install` fails with:

```
[ERROR] Failed to execute goal io.quarkus:quarkus-extension-maven-plugin:3.32.2:extension-descriptor
  (default) on project quarkus-tarkus: The deployment artifact
  io.quarkiverse.tarkus:quarkus-tarkus-deployment::jar depends on the following
  Quarkus extension deployment artifacts whose corresponding runtime artifacts were
  not found among the dependencies of io.quarkiverse.tarkus:quarkus-tarkus:jar:
  io.quarkiverse.ledger:quarkus-ledger-deployment::jar
```

The error message blames the runtime module but the actual cause is in the deployment pom.

### Context

Occurs when:
- You have a Quarkus extension with separate `runtime/` and `deployment/` Maven modules
- The `deployment/pom.xml` declares `X-deployment` as a dependency (e.g. `quarkus-ledger-deployment`)
- The corresponding `runtime/pom.xml` does **not** declare `X` (e.g. `quarkus-ledger`) as a runtime dependency
- The `extension-descriptor` Maven goal validates this pairing and fails

### Root cause

The `quarkus-extension-maven-plugin` `extension-descriptor` goal enforces a structural rule: for every `X-deployment` artifact in the deployment pom's dependency list, the corresponding `X` runtime artifact must appear in the runtime pom's dependencies. This ensures that any deployment-time extension capability has a runtime counterpart on the classpath.

If you add a deployment dependency "just in case" or because the IDE suggested it, but the `TarkusProcessor` (or equivalent `@BuildStep` class) never actually imports or uses anything from it, the dependency is spurious — and the validator catches it.

### Fix

Remove the spurious `X-deployment` dependency from `deployment/pom.xml`. If `TarkusProcessor` doesn't import any class from `X-deployment`, it doesn't belong there.

```xml
<!-- Remove this if TarkusProcessor doesn't use it -->
<dependency>
  <groupId>io.quarkiverse.ledger</groupId>
  <artifactId>quarkus-ledger-deployment</artifactId>
  <version>1.0.0-SNAPSHOT</version>
</dependency>
```

Check your `@BuildStep` classes for actual imports from the dependency before removing — if they exist, you also need to add the runtime counterpart to `runtime/pom.xml`.

### Why this is non-obvious

The error message points to the runtime module ("not found among the dependencies of `quarkus-tarkus:jar`") but the fix is in the deployment pom. Nothing in the error output mentions the deployment pom or the spurious dependency as the cause. A developer will naturally look at the runtime module first and find nothing wrong there.

### Why this fix

Removing the unused deployment dependency is cleaner than adding a runtime dependency the extension doesn't need. Adding an unnecessary runtime dependency would bloat the extension's runtime classpath and couple the core extension to an optional module.
