### Symptom
A WebSocket integration test sends `printf 'MARK-A\n\nMARK-B\n'` to a tmux pane and then polls the history buffer waiting for both markers. The poll loop breaks immediately on the first history message, but the subsequent blank-line assertion fails — the "blank line between MARK-A and MARK-B" is never found.

### Context
Claudony-style tests that stream tmux pane history over WebSocket. The history burst on reconnect contains the shell echo of the command (`printf 'MARK-A\n\nMARK-B\n'`), which itself contains both markers on a single line with literal `\n` characters. The test's break condition — `hist.contains("MARK-A") && hist.contains("MARK-B")` — fires on the echo, before the actual command output (with real newlines between markers) arrives in the stream.

### What was tried (didn't work)
- `lastIndexOf("MARK-A")` / `lastIndexOf("MARK-B")` — still finds the echo occurrence when pane scrolling puts the actual output in scrollback before the echo
- Increasing timeouts and adding `Thread.sleep()` stabilisation
- Checking both markers present before breaking — same problem; markers appear in the echo

### Root cause
The shell echoes the command as a single line (`printf 'MARK-A\n\nMARK-B\n'`). This line contains MARK-A and MARK-B separated by literal `\n` characters (backslash + n, not actual newlines). The actual command output has MARK-A and MARK-B on separate lines with a real blank line between them. `String.contains()` and `lastIndexOf()` cannot distinguish between these two occurrences.

### Fix
Break the history polling loop only when the regex pattern matches — the pattern requires actual newline characters between the markers, which cannot occur in the single-line echo:
```java
var blankLinePattern = Pattern.compile("MARK-A[^\n]*\n[^\n]*\n[^\n]*MARK-B");

while (System.currentTimeMillis() < deadline) {
    var chunk = queue.poll(100, TimeUnit.MILLISECONDS);
    if (chunk != null) {
        buffer.append(chunk);
        if (blankLinePattern.matcher(buffer.toString()).find()) break;
    }
}
// Then assert the same pattern for the final check
assertTrue(blankLinePattern.matcher(hist).find(), "blank line preserved between markers");
```

### Why this is non-obvious
The echo appears identical to the actual output to the naked eye. `lastIndexOf()` is the natural tool for "find the output occurrence, not the echo" but fails when pane scrolling changes the order of content in the capture buffer. The regex approach works because the echo has literal `\n` chars (two-char sequences) while actual output has real newline chars.
