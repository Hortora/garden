## Symptom

`QueryTypeMismatch: Incorrect query result type: query produces 'com.example.MyEnum'
but type 'java.lang.String' was given` at runtime, even though the column is stored
as a VARCHAR via `@Enumerated(EnumType.STRING)`.

## Context

Querying a field mapped with `@Enumerated(EnumType.STRING)` using HQL `cast()` to
extract the raw string value:

```java
session.createSelectionQuery(
    "select cast(ci.state as string) from case_instance ci where ci.uuid = :uuid",
    String.class)
```

## Root cause

Hibernate 6 resolves `cast(field as string)` against the entity's type mapping, not
the underlying column type. The `state` field is mapped as `CaseStatus` (an enum),
so `cast(... as string)` returns `CaseStatus`, not `String`, regardless of how the
column is physically stored.

## Fix

Use a native SQL query to get the raw VARCHAR value directly:

```java
// ❌ HQL cast — returns enum type, not String
session.createSelectionQuery(
    "select cast(ci.state as string) from case_instance ci where ci.uuid = :uuid",
    String.class)

// ✅ Native SQL — reads the raw column value
session.createNativeQuery(
    "SELECT state FROM case_instance WHERE uuid = :uuid",
    String.class)
    .setParameter("uuid", caseId)
    .getSingleResult()
```

## Why this is non-obvious

The HQL syntax `cast(x as string)` looks like it should coerce the enum to its
string representation, which is also how the value is physically stored. The error
message names the Java type (`CaseStatus`) rather than describing the HQL/native
mismatch, making it hard to diagnose. This pattern typically surfaces when writing
tests that verify a renamed enum value is actually stored correctly in the DB.
