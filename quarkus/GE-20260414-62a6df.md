## COLLECT and EPHEMERAL channel semantics: SELECT-then-DELETE is not atomic under READ_COMMITTED isolation

**ID:** GE-20260414-62a6df
**Stack:** Quarkus 3.32.2, Hibernate ORM 6.x, Panache, H2 2.4.x / PostgreSQL
**Symptom:** Under concurrent load, two consumers calling `check_messages` on a COLLECT or EPHEMERAL channel simultaneously both receive the same messages — double delivery. The `@Transactional` method does not prevent this.
**Context:** Any implementation of "deliver-and-clear" semantics using a SELECT followed by a DELETE in the same `@Transactional` method, under the default `READ_COMMITTED` isolation level.

### What was tried (didn't work)

- Wrapping SELECT and DELETE in a single `@Transactional` method — insufficient under READ_COMMITTED
- Using Panache's `.list()` followed by `.delete()` — same isolation problem

### Root cause

Under `READ_COMMITTED` isolation (the default for H2 and PostgreSQL), two transactions that both start before either commits can both execute the SELECT and see the same rows. Both then execute the DELETE. The first DELETE commits; the second DELETE either deletes zero rows (the rows are already gone) or succeeds on its own snapshot. But both transactions have already returned the rows to their callers — double delivery has occurred.

`@Transactional` in Quarkus defaults to `READ_COMMITTED`. This isolation level provides per-statement consistency, not per-transaction snapshot consistency.

```java
// This pattern is NOT safe for exactly-once under concurrent reads:
@Transactional
public List<Message> collectAndClear(UUID channelId) {
    List<Message> messages = Message.find("channelId = ?1", channelId).list();  // snapshot A
    Message.delete("channelId = ?1", channelId);                                // deletes snapshot A's rows
    return messages;  // but another transaction already saw the same rows
}
```

### Sequential delivery (the common case)

If only one reader calls `check_messages` at a time, the deliver-and-clear pattern is correct and safe. The issue only surfaces under true concurrent reads from multiple threads/processes simultaneously.

### Fix — two options

**Option 1: `SELECT FOR UPDATE` (locks rows before reading)**

```java
@Transactional
public List<Message> collectAndClear(UUID channelId) {
    // Lock rows first — other transactions must wait
    List<Message> messages = Message.getEntityManager()
        .createNativeQuery("SELECT * FROM message WHERE channel_id = ? FOR UPDATE", Message.class)
        .setParameter(1, channelId)
        .getResultList();
    Message.delete("channelId = ?1", channelId);
    return messages;
}
```

**Option 2: Higher isolation (`REPEATABLE_READ` or `SERIALIZABLE`)**

In Quarkus, configure the datasource isolation level:
```properties
quarkus.datasource.jdbc.transaction-isolation-level=repeatable-read
```

Or per-transaction via the JTA API. `REPEATABLE_READ` ensures both the SELECT and DELETE see the same consistent snapshot. Suitable for PostgreSQL; H2 supports it.

### Mitigation for most production cases

In practice, multi-agent systems rarely have two agents calling `check_messages` on the same COLLECT/EPHEMERAL channel simultaneously. The sequential-delivery guarantee is correct for the realistic case. The concurrent double-delivery is worth documenting but typically requires explicit concurrent architecture to trigger.

### Why non-obvious

`@Transactional` sounds like it should guarantee atomicity across all operations within the method. It does guarantee atomicity (the DELETE will roll back if anything fails) and isolation from other transactions that MODIFY rows. But `READ_COMMITTED` does NOT prevent another read-only transaction from seeing the same rows before the DELETE commits. The gap between "transactional" and "exactly-once" under concurrent reads is subtle.
