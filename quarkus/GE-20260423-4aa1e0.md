## Symptom

Migrating to named Hibernate persistence units fails with an error about missing entities or a persistence unit resolution error, even though the entities are correctly annotated. The issue only appears when entities from two different libraries share a `JOINED` inheritance hierarchy.

## Context

Occurs when: (1) migrating from the default Hibernate PU to a named PU, (2) entity `B extends A` with `@Inheritance(strategy = InheritanceType.JOINED)`, and (3) `A` and `B` live in different Maven modules or libraries.

## Root Cause

JPA requires all entities in an inheritance hierarchy to share a single persistence unit. If `A` is in `io.lib.model` and `B` is in `io.other.entity`, the named PU `packages` config must include both packages — even if you only want the `io.other` entities in that PU. Specifying only `io.other` causes Hibernate to fail because it can't resolve the full hierarchy.

## Fix

Include both packages in the named PU configuration:

```properties
quarkus.hibernate-orm.mypu.packages=io.other.entity,io.lib.model
```

This forces the parent entity's library into the same persistence unit, which may have schema implications (both libraries' tables are created/migrated by the same Flyway run).

## Why This Is Non-Obvious

The constraint is a JPA specification requirement, but the error message from Quarkus/Hibernate does not say "hierarchy entities must share a PU." It typically presents as a mapping exception or missing entity error that points to the child entity, not to the inheritance relationship. The fix — including an external library's package in your PU config — feels wrong and is not mentioned in Quarkus named datasource documentation.
