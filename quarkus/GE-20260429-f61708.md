### Symptom

A `@ObservesAsync` event observer is triggered even when the firing transaction later rolls back. In tests, this manifests as duplicate event deliveries: a test expecting exactly one `COMPLETED` event receives two — one from a transaction that committed and one from a transaction that rolled back (the rolled-back transaction's event was already dispatched before the rollback occurred).

Example: `WorkItemGroupLifecycleEventTest.completedEventFiresExactlyOnceAtThreshold` received 2 `COMPLETED` group events instead of 1 when `fireAsync()` was called inside the `@Transactional` policy method.

### Context

When two concurrent child completions race for the OCC lock on a group entity, one wins and one rolls back. Both called `groupEvent.fireAsync()` inside their `@Transactional` methods before the rollback determination. The loser's event was already dispatched to the thread pool despite the transaction rolling back.

### What was tried

Calling `groupEvent.fireAsync()` inside the `@Transactional` policy method — assumed "async" meant it would fire only on successful commit, analogous to how `@TransactionPhase.AFTER_SUCCESS` works for synchronous observers.

### Root cause

`Event.fireAsync()` dispatches to a thread pool immediately when called — it does not participate in the transaction lifecycle. The CDI spec's transactional observer semantics (`@Observes(during=AFTER_SUCCESS)`) only apply to synchronous `fire()` with transactional observer qualifiers, not to `fireAsync()`. There is no built-in "fire async only on commit" mechanism in CDI 4.x.

### Fix

Move `fireAsync()` calls outside the `@Transactional` boundary. If using a two-bean pattern (coordinator + policy), have the `@Transactional` policy method return the event data, and have the non-transactional coordinator call `fireAsync()` after the policy method returns successfully:

```java
// MultiInstanceCoordinator (@ObservesAsync, NOT @Transactional)
void onChildTerminal(@ObservesAsync WorkItemLifecycleEvent event) {
    try {
        WorkItemGroupLifecycleEvent groupEvt = policy.process(childId, childStatus); // @Transactional
        if (groupEvt != null) {
            groupEvent.fireAsync(groupEvt);  // fired AFTER transaction commits
        }
    } catch (Exception e) {
        // retry...
    }
}
```

The policy method returns `null` when no event should fire (e.g. `policyTriggered` guard hit), or the event object to fire on success.

### Why this is non-obvious

`fireAsync()` intuitively suggests deferred delivery. The word "async" implies separation from the current call — it's natural to assume this includes separation from the current transaction, especially since CDI does support transactional synchronous observers. The distinction between synchronous transactional observers and async event delivery is not prominent in CDI documentation.
