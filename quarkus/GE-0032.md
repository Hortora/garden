## Quarkus `@Scheduled` allows overlapping executions by default — no warning, just a race

**ID:** GE-0032
**Stack:** Quarkus 3.x (all versions)
**Symptom:** A `@Scheduled` method with a stateful side effect (e.g. advancing a game simulation, writing to a shared queue) runs concurrently with itself when one invocation takes longer than the configured interval. No warning, no error, no log — the second invocation simply starts while the first is still running.
**Context:** Any `@Scheduled` bean where the method body takes variable time or where the system is under load. Especially dangerous when the method writes to shared mutable state.

### What was tried (didn't work)
- Assumed `@Scheduled` would wait for the previous invocation to complete before starting the next (as Spring's `@Scheduled` with `fixedDelay` does by default) — incorrect
- Assumed a warning would appear in logs if overlap occurred — no warning is emitted

### Root cause
Quarkus's scheduler uses a thread pool and does not enforce single-invocation semantics by default. Each interval fires a new invocation regardless of whether the previous one has completed. The Quarkus docs mention `concurrentExecution` but it is not presented prominently and the default is `PROCEED` (allow concurrency).

### Fix

```java
@Scheduled(
    every = "${starcraft.tick.interval:500ms}",
    concurrentExecution = Scheduled.ConcurrentExecution.SKIP
)
void gameTick() {
    // ... stateful logic
}
```

`SKIP` silently skips the new invocation if the previous one is still running.
`PROCEED` (the default) allows both to run concurrently.

For cases where you want to know when a skip occurs, log at the start of the method:
```java
@Scheduled(every = "500ms", concurrentExecution = Scheduled.ConcurrentExecution.SKIP)
void gameTick() {
    log.debugf("Tick started — frame=%d", frameCounter.get());
    // ...
}
```
A missing log line indicates a skipped invocation.

### Why this is non-obvious
The default behaviour looks correct in unit tests and light load (tests disable the scheduler; under normal operation invocations rarely overlap). The race only manifests under load or with slow external dependencies — exactly when you least want to debug it. Spring's `@Scheduled` with `fixedDelay` queues the next invocation *after* the previous completes, so Spring developers expect single-threaded behaviour by default and are surprised by Quarkus's PROCEED default.
