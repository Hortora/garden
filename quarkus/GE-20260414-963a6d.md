## Hibernate @PreUpdate fires at flush time, not at persist() — denormalized fields are stale in the returned object

**ID:** GE-20260414-963a6d
**Stack:** Quarkus 3.32.2, Hibernate ORM 6.x, Panache active record
**Symptom:** A service method appends content to a managed entity, calls `entity.persist()`, and returns the entity. The caller checks a denormalized field (e.g. `sizeBytes`) — it shows the value from the **previous** persist, not the current state. The DB is updated correctly later; only the in-memory return value is wrong.
**Context:** Chunked upload pattern: first call creates an entity (`@PrePersist` runs, sets `sizeBytes`), subsequent calls append content to the managed entity and call `persist()` (a no-op for managed entities). `@PreUpdate` is expected to recompute `sizeBytes`, but the method returns the entity before the transaction commits.

### What was tried (didn't work)
- Verified `@PreUpdate` was defined correctly and updating `sizeBytes` from `content.length()`
- Checked that the DB had the correct `sizeBytes` after the transaction committed — it did
- Added logging inside `@PreUpdate` — it was firing, but after the method returned

### Root cause

`@PreUpdate` is a Hibernate lifecycle callback that fires **immediately before Hibernate writes the entity to the database**. For managed entities, Hibernate defers the actual SQL UPDATE until a flush occurs (either explicitly or at transaction commit). Calling `entity.persist()` on an already-managed entity is a no-op — Hibernate's dirty-check mechanism will emit the UPDATE at flush time, and only then does `@PreUpdate` fire.

The service method returns the entity object to the caller before the transaction flushes. At that point, `sizeBytes` still holds the value set during the last actual database write (`@PrePersist` or the previous `@PreUpdate`), not the value that will be computed when `@PreUpdate` finally fires.

In a `@TestTransaction` test that checks the returned value directly, this produces wrong assertions without any error.

### Fix

Set the denormalized field explicitly in the service method before returning, and use `@PreUpdate` only as a DB-level safety net for ORM-driven changes:

```java
@Transactional
public SharedData store(String key, String content, boolean append, boolean lastChunk) {
    SharedData data = findOrCreate(key);
    data.content = append ? data.content + content : content;
    data.complete = lastChunk;
    // Set explicitly — @PreUpdate won't fire until flush (after we return)
    data.sizeBytes = data.content != null ? data.content.length() : 0;
    data.persist();
    return data;  // sizeBytes is now correct in the returned object
}

// @PreUpdate remains as a safety net for ORM-driven changes
@PreUpdate
void preUpdate() {
    sizeBytes = content != null ? content.length() : 0;
}
```

### Why non-obvious

The Hibernate docs describe `@PreUpdate` as "called before the update operation is performed" — this sounds like "called when you ask for an update", not "called when Hibernate decides to flush". The distinction matters when the method needs to return a consistent object before the transaction ends. `@PrePersist` behaves differently: it fires synchronously when `persist()` is called on a new (transient) entity, so developers who rely on `@PrePersist` for the initial value expect `@PreUpdate` to work the same way for updates.
