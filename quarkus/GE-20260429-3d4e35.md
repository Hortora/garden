## Test @ObservesAsync CDI observers in @QuarkusTest with @TestTransaction + fireAsync().join()

### What this is

A reliable, deterministic pattern for testing `@ObservesAsync` CDI event observers that write to a database in their own `@Transactional` context.

### How it works

An `@ObservesAsync` observer runs on a Quarkus managed executor thread with its own `@Transactional` context — separate from the test's transaction. `fireAsync()` returns a `CompletionStage` that resolves only after **all async observers have completed, including their transaction commits**. Calling `.toCompletableFuture().join()` blocks the test thread until this point, giving a precise synchronization barrier.

The test method is annotated with `@TestTransaction`, which gives the injected `EntityManager` a transaction context for reading the data the observer just committed.

```java
@QuarkusTest
class MyObserverTest {

    @Inject Event<MyEvent> events;

    @Inject
    @MyPersistenceUnit
    EntityManager em;

    @Test
    @TestTransaction
    void observer_writesRow_andIsReadable() {
        UUID id = UUID.randomUUID();

        // join() blocks until observer has committed its own transaction
        events.fireAsync(new MyEvent(id, "payload"))
              .toCompletableFuture().join();

        List<MyEntity> rows = em.createQuery(
                "SELECT e FROM MyEntity e WHERE e.refId = :id", MyEntity.class)
                .setParameter("id", id)
                .getResultList();

        assertThat(rows).hasSize(1);
        assertThat(rows.get(0).payload).isEqualTo("payload");
    }
}
```

### Why this is non-obvious

Most developers reach for `Awaitility.await().atMost(5, SECONDS).untilAsserted(...)` to test async observers — which works but is indeterminate and adds unnecessary wait time. Few realise that `fireAsync()` returns a `CompletionStage` that resolves precisely when all observers finish (including their DB commits), making `join()` an exact synchronization point with no polling.

The `@TestTransaction` annotation on the test method is also non-obvious: without it, the directly-injected `EntityManager` may not be in a transaction context, causing `javax.persistence.TransactionRequiredException` or reading from an auto-commit connection that sees a different isolation snapshot.
