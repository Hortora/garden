## Symptom

A `@QuarkusTest` class that queries all rows (e.g., `listAll()`, `findAll()`) returns more rows than expected — rows inserted by a *different* test class that ran earlier in the same test suite execution.

Tests pass in isolation (`-Dtest=MyTest`) but fail when the full suite runs.

## Context

Occurs when:
- Multiple `@QuarkusTest` classes use the same H2 in-memory URL (e.g., `jdbc:h2:mem:mydb;DB_CLOSE_DELAY=-1`)
- One test class inserts rows without cleaning up
- A later test class runs a `listAll()`-style query that returns all rows

H2's `mem:` databases persist for the lifetime of the JVM when `DB_CLOSE_DELAY=-1` is set. All `@QuarkusTest` classes that share the same URL see the same database instance — including each other's leftover data.

## Root Cause

`DB_CLOSE_DELAY=-1` keeps the H2 in-memory database alive as long as the JVM runs. Quarkus reuses the same application context across `@QuarkusTest` classes in the same surefire fork (unless a test requires a new context via `@QuarkusTestProfile`). Both conditions together mean test data accumulates.

Quarkus's `@TestTransaction` annotation rolls back transactions per test method but applies only to the annotated test class. Rows committed by other test classes without `@TestTransaction` persist across all subsequent tests.

## Fix

Give the test class that needs isolation its own H2 database URL via a profile property:

```properties
# In src/test/resources/application.properties — only affects the named profile
%retention-test.quarkus.datasource.jdbc.url=jdbc:h2:mem:retentiontestdb;DB_CLOSE_DELAY=-1
```

Activate the profile on the test class:
```java
@QuarkusTest
@TestProfile(RetentionTestProfile.class)
class LedgerRetentionJobIT { ... }

public class RetentionTestProfile implements QuarkusTestProfile {
    @Override public String getConfigProfile() { return "retention-test"; }
}
```

Or use `@TestTransaction` on the inserting test class to ensure rollback after each test method.

## Why This Is Non-Obvious

The standard advice for H2 in-memory databases is to use `DB_CLOSE_DELAY=-1` — without realising this keeps the database alive across the entire test suite. The failure is intermittent (depends on test execution order) and only manifests when running the full suite. The fix requires knowing that Quarkus profile properties can override the datasource URL per test class.
