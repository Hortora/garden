## @WithDefault("") on a String method in @ConfigMapping causes ConfigValidationException at Quarkus startup

### Symptom

Quarkus fails to start with `ConfigValidationException` or `NoSuchElementException` when a `@ConfigMapping` interface method has `@WithDefault("") String`. The property appears optional but the framework refuses to return an empty string.

### Context

When designing `@ConfigMapping` interface methods that should be optional strings (e.g. a deployment identifier, an optional URL), the reflex is:

```java
@WithDefault("")
String deploymentId();
```

This compiles but throws at startup — SmallRye Config treats an empty string default as "no value provided" for `String` return types and escalates to a validation error.

### Fix

Use `Optional<String>` with no `@WithDefault`:

```java
Optional<String> deploymentId();
```

Call sites use `.orElse("")` or `.orElse(null)` as needed.

This is the established SmallRye Config pattern for truly optional string config — the same pattern used by `quarkus.datasource` (`Optional<String> datasource()`), Quarkus Merkle publisher URL, and other core extension config.

### Why this is non-obvious

The `@WithDefault("")` annotation reads as "empty string default" which is a valid Java string. Nothing in the annotation documentation warns that SmallRye Config distinguishes between "no value" and "empty string" for validation purposes. The error message at startup doesn't clearly point to the `@WithDefault("")` as the cause.

The fix (`Optional<String>`) is not visible in the `@ConfigMapping` Javadoc — it requires reading SmallRye Config's type conversion rules or observing the pattern in existing Quarkus extension source code.
