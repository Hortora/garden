## Quarkus augmentation cache is disk-based — reuseForks=false does NOT clear it

### Symptom

Setting `<reuseForks>false</reuseForks>` in a Surefire execution to force a fresh JVM for a
test class with a different datasource config. The test still fails with the augmentation
mismatch warning and uses the wrong datasource driver — as if the fresh JVM made no difference.

### Context

A Maven module runs two Surefire executions: one for H2 tests, one for PostgreSQL tests.
The PostgreSQL execution uses `reuseForks=false` (separate JVM) and sets `db-kind=postgresql`
as a system property. The H2 execution runs first and augments successfully. The PostgreSQL
execution then starts a fresh JVM but still gets H2 driver behaviour.

### Root cause

The Quarkus augmented artifact is cached on disk at `target/quarkus-app/`. This cache
persists across JVM boundaries within the same Maven build. When Surefire starts a new JVM
with `reuseForks=false`, the new JVM's Quarkus test framework reads the existing
`target/quarkus-app/` cache from disk. If the cache was built for a different `db-kind`,
Quarkus compares the build-time config hash — but **only against system properties and
config files visible at JVM startup**, not against test resource overrides.

If the system property for `db-kind` was not set in the Surefire execution that created the
cache (i.e. the H2 execution ran first), the cache has H2 baked in. The fresh JVM's
PostgreSQL system property triggers a warning but the cached artifact is reused.

### Fix

Run the execution that needs the different `db-kind` **first** in the POM, before any other
execution augments the cache:

```xml
<!-- PostgreSQL FIRST — augments fresh with db-kind=postgresql -->
<execution>
  <id>postgres-tests</id>
  <phase>test</phase>
  <goals><goal>test</goal></goals>
  <configuration>
    <reuseForks>false</reuseForks>
    <systemPropertyVariables>
      <quarkus.datasource.db-kind>postgresql</quarkus.datasource.db-kind>
    </systemPropertyVariables>
    <includes><include>**/PostgresTest*.java</include></includes>
  </configuration>
</execution>
<!-- H2 SECOND — detects db-kind mismatch, re-augments for H2 -->
<execution>
  <id>h2-tests</id>
  <phase>test</phase>
  <goals><goal>test</goal></goals>
  <configuration>
    <excludes><exclude>**/PostgresTest*.java</exclude></excludes>
  </configuration>
</execution>
```

Also disable the default Surefire execution to prevent it running all tests outside this ordering:

```xml
<execution>
  <id>default-test</id>
  <phase>none</phase>
</execution>
```

Alternatively, delete `target/quarkus-app/` before the PostgreSQL execution using an
antrun step.

### Why this is non-obvious

`reuseForks=false` is documented as "each test class gets its own JVM" — developers
reasonably assume a fresh JVM means a fresh Quarkus augmentation. The disk-based nature of
the cache is not documented and the warning message ("build time fixed to 'h2'") does not
mention that deleting the cache or reordering executions would fix it.
