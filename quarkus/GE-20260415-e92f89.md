## Symptom

A `@QuarkusTest @TestTransaction` test method:
1. Directly calls a `@Transactional` CDI bean (e.g. a job) that writes data to the DB
2. Then makes a REST Assured HTTP call to verify the data

The REST call returns 404 (or missing data) even though the write happened moments before. No error is thrown — the data simply isn't there.

## Context

Any `@QuarkusTest @TestTransaction` test that mixes:
- A directly-injected `@Transactional` method call (which **joins** the test transaction)
- A REST Assured HTTP assertion (which runs in its own separate HTTP request context)

Common pattern: testing a scheduled job's output via REST.

## What Was Tried

Kept `@TestTransaction` on the test class and called the job via `@Inject` then checked the result via REST Assured `GET`. The GET returned 404 consistently.

## Root Cause

`@TestTransaction` wraps the entire test method in a transaction that rolls back at the end. When a `@Transactional` CDI bean is called from within that context, it **joins** the outer test transaction (default `REQUIRED` propagation). The data is written to the database but **not committed**.

The REST Assured HTTP call goes through a separate HTTP request handler — a completely different transaction context. It cannot see the uncommitted rows.

## Fix

Remove `@TestTransaction` from test classes that mix direct `@Transactional` calls with HTTP assertions:

```java
// WRONG — @TestTransaction + REST assertion
@QuarkusTest
@TestTransaction  // ← remove this
class MyJobRestTest {
    @Inject MyJob job;

    @Test
    void jobResult_appearsViaRest() {
        job.run();  // @Transactional joins test tx, NOT committed
        given().when().get("/results").then().statusCode(200);  // sees nothing
    }
}

// CORRECT — no @TestTransaction, job commits its own tx
@QuarkusTest
class MyJobRestTest {
    @Inject MyJob job;

    @Test
    void jobResult_appearsViaRest() {
        job.run();  // @Transactional starts+commits its own tx
        given().when().get("/results").then().statusCode(200);  // sees committed data
    }
}
```

Use unique identifiers per test method to prevent cross-test contamination when not using `@TestTransaction`.

## Why This Is Non-Obvious

`@TestTransaction` works correctly when you test via injected CDI beans directly — all reads and writes happen in the same transaction and everything is visible. The failure only appears when you introduce an HTTP call. The mental model of "test transaction = isolated sandbox" breaks at the HTTP boundary, where Quarkus starts a new request-scoped transaction entirely.

### Why this fix

Removing `@TestTransaction` lets the `@Transactional` method commit its own transaction, making data visible to subsequent HTTP requests. The rollback safety net is replaced by using unique test data (e.g. UUID-based actor IDs) to prevent cross-test contamination.
