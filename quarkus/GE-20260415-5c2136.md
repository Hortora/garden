## Symptom

One `@QuarkusTest` class fails with `QuarkusBindException: Port already bound: 8081: Address already in use`. All remaining tests in the same class are then skipped with "Failed to start quarkus". Looks like a test logic bug — the port error message is buried in the stack trace.

## Context

Occurs when: (a) a previous test run did not release port 8081 before the next run began, or (b) two `@QuarkusTest` class instances attempt to start Quarkus simultaneously. Intermittent in CI, consistent on a loaded dev machine. Typically surfaces as 1 ERROR + N SKIPPED in one test class only.

## What was tried (didn't work)

- Investigating the test class itself for port-related assertions (none existed)
- Increasing test timeouts
- Assuming the skipped tests were broken

## Root cause

`@QuarkusTest` defaults to `quarkus.http.test-port=8081`. Every test run that doesn't cleanly release the port before the next run starts causes a bind conflict. Maven Surefire's default JVM forking doesn't guarantee port release between test class executions.

## Fix

Add to `src/main/resources/application.properties`:

```properties
%test.quarkus.http.test-port=0
```

Port `0` instructs the OS to assign a random available ephemeral port for each `@QuarkusTest` startup. Zero configuration overhead — Quarkus, RestAssured, and `@TestHTTPResource` all pick up the assigned port automatically.

## Why this is non-obvious

The failure message ("Port already bound") suggests a test is actively holding a port. Nothing in the Quarkus test docs prominently mentions that hardcoded port 8081 is a cross-run hazard. Developers investigate their test code rather than the application.properties port config.
