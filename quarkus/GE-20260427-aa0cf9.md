## Symptom

Dynamic JPQL `IN` clause with a positional parameter compiles and passes tests on H2 but fails on PostgreSQL:

```java
jpql.append(" AND e.messageType IN ?").append(params.size() + 1);
params.add(messageTypes); // Set<String>
```

H2 silently accepts `IN ?2` and returns correct results. PostgreSQL rejects it with a query syntax error or returns no results.

## Root cause

The JPA/JPQL spec requires the `IN` predicate with a positional parameter to use parentheses: `IN (?N)`. Without them, the query is technically malformed. H2's parser is lenient and accepts both forms; PostgreSQL's parser is strict and requires the correct form.

## Fix

Wrap the positional parameter number in parentheses:

```java
// Before (breaks on PostgreSQL):
jpql.append(" AND e.messageType IN ?").append(params.size() + 1);

// After (correct):
jpql.append(" AND e.messageType IN (?").append(params.size() + 1).append(")");
```

For named parameters, the syntax is the same: `IN (:paramName)` — no change needed since the colon+name form is naturally parenthesised.

## Why this is non-obvious

H2 is the standard Quarkus test datasource. If your CI runs H2 and production runs PostgreSQL, this fails silently in all tests and only surfaces at deployment. The spec reviewer catches it; the test suite does not. The JPA spec language is also ambiguous on this point — many developers write `IN ?1` having seen examples that work in their local environment.

### Alternatives considered

- Named parameters (`:paramName`) instead of positional — correct syntax either way, avoids this class of problem entirely but requires building a `Map<String, Object>` alongside the JPQL `StringBuilder`
