## A @Transactional JAX-RS method that calls @Transactional CDI beans sees their writes immediately — no flush needed

### The technique

Mark a JAX-RS endpoint `@Transactional`. Any `@Transactional` CDI beans it calls join the same transaction (default `REQUIRED` propagation). Writes those beans make — including writes from CDI observers like `LedgerEventCapture` that fire during `WorkItemService` calls — are immediately visible to repository queries made later in the same method. No explicit `flush()`, `entityManager.refresh()`, or transaction split is needed.

### Example

```java
@POST
@Path("/run")
@Transactional  // starts the transaction
public ScenarioResponse run() {

    // WorkItemService.create() fires a CDI lifecycle event.
    // LedgerEventCapture observes it and writes a LedgerEntry — within this same TX.
    WorkItem wi = workItemService.create(request);

    // Immediately visible — no flush needed:
    WorkItemLedgerEntry entry = ledgerRepo.findLatestByWorkItemId(wi.id).orElseThrow();

    // Mutate the managed entity directly — Hibernate dirty-checking flushes at TX end:
    entry.sourceEntityId = "LOAN-8821";
    // No explicit save() or persist() needed — it's already managed.

    // Also visible immediately:
    WorkItemLedgerEntry laterEntry = ledgerRepo.findLatestByWorkItemId(wi.id).orElseThrow();
    laterEntry.causedByEntryId = entry.id;

    return buildResponse(wi.id);
}
```

### Why this is non-obvious

Developers often assume they need to split transactions or call `entityManager.flush()` to make one bean's writes visible to another bean's reads within the same request. In fact, JTA `REQUIRED` propagation means all participants share one persistence context. Hibernate's first-level cache and dirty-checking handle everything — writes are visible immediately (via the cache) and flushed atomically at commit.

The same applies to CDI observer methods (`@Observes`) called during a `@Transactional` service method: if the observer is itself `@Transactional(REQUIRED)`, it joins the enclosing transaction and its writes are part of the same unit of work.

### Alternatives considered

- **Explicit `entityManager.flush()`** — works but unnecessary and adds coupling to JPA internals
- **Splitting into two separate `@Transactional` methods** — works but forces two transactions, two commits, and more complex error handling
- **Using `REQUIRES_NEW`** — creates a nested transaction; the outer TX cannot see its writes until it commits
