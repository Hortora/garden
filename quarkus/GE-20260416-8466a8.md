## Use a no-profile-guard @ApplicationScoped bridge bean to flow profile-specific data to general-purpose consumers

**ID:** GE-20260416-8466a8
**Stack:** Quarkus 3.x, CDI
**Labels:** `#quarkus` `#cdi` `#architecture`
**What it achieves:** Shares data produced by a `@IfBuildProfile` bean with a general-purpose consumer without adding profile conditions to the consumer or polluting a shared interface.
**Context:** You have a profile-specific implementation (e.g. an emulated game engine active only in `%emulated` profile) that produces data a general CDI bean (e.g. a broadcaster active in all non-prod profiles) needs. You don't want to expand the shared interface just to carry this data.

### The technique

Create a thin `@ApplicationScoped` bean with **no profile guard** — it's always present. The profile-specific producer updates it; the general consumer reads it, receiving `null` when the producer isn't active.

```java
// Bridge bean — always registered, no @IfBuildProfile
@ApplicationScoped
public class VisibilityHolder {
    private volatile VisibilityGrid current;  // null in non-emulated modes
    public void set(VisibilityGrid g) { current = g; }
    public VisibilityGrid get() { return current; }
}

// Profile-specific producer — only active in emulated mode
@IfBuildProfile("emulated")
@ApplicationScoped
public class EmulatedEngine implements SC2Engine {
    @Inject VisibilityHolder visibilityHolder;

    @Override
    public void tick() {
        game.tick();
        visibilityHolder.set(game.observeVisibility());  // update the bridge
    }
}

// General consumer — active in all non-prod profiles
@UnlessBuildProfile("prod")
@ApplicationScoped
public class GameStateBroadcaster {
    @Inject VisibilityHolder visibilityHolder;

    void broadcast(GameState state) {
        VisibilityGrid vg = visibilityHolder.get();
        String vis = vg != null ? vg.encode() : null;
        send(new GameStateBroadcast(state, vis));  // vis is null in non-emulated modes
    }
}
```

The bridge bean is a typed null-safe slot. The consumer checks `get() != null` rather than depending on profile conditions.

### Alternatives considered

- **Adding data to the shared interface** (`SC2Engine.observeVisibility()`) — rejected because the concept doesn't belong in the interface; real SC2 handles visibility natively and would never implement this method meaningfully.
- **Adding `@IfBuildProfile` to the consumer** — rejected because the consumer (broadcaster) legitimately runs in all non-prod profiles; restricting it would break mock/replay mode visualisation.
- **CDI events** — workable but heavier: requires an event type, `@Observes`, and careful ordering. The bridge bean is synchronous and zero-overhead.

### Why this is non-obvious

The instinct is to either expand the shared interface or add profile guards to the consumer. The bridge bean pattern inverts this: the profile-specific bean reaches out to the bridge, and the consumer simply asks the bridge — with `null` as the natural "not active" signal. Most CDI documentation focuses on injection and events; the typed mutable holder as a cross-profile coordination point isn't documented anywhere.

### When to use it

- Profile-specific engine/adapter writes data that a cross-profile consumer needs
- The data doesn't belong in the shared interface
- The consumer must remain active in all profiles (not just the producing one)
- The `null` case is a meaningful "feature not active" signal (not an error)
