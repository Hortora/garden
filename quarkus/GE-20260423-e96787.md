## Symptom

After calling `em.merge(entity)`, subsequent reads from the original `entity` reference return stale data. No exception is thrown. Updates made inside the merge transaction appear lost when reading back through the original reference.

## Context

Occurs in any JPA store implementation that calls `EntityManager.merge()` on a detached entity — including Panache active record pattern where `PanacheEntityBase.getEntityManager().merge(pr)` is called.

## Root Cause

`EntityManager.merge()` returns a **new managed entity instance**. The passed-in detached object is not modified and remains detached. Any subsequent field access on the original reference reads the stale detached state, not the managed state.

The JPA spec documents this behaviour, but the method signature — returning the same type — creates a strong intuition that the passed-in object is modified in place.

## Fix

Always capture the return value:

```java
// Wrong — pr stays detached
em.merge(pr);
return pr;

// Correct — re-assign to get the managed instance
pr = em.merge(pr);
return pr;
```

In a Panache store:

```java
@Transactional
public MyEntity save(MyEntity entity) {
    if (entity.id == null) {
        entity.persist();
    } else {
        entity = MyEntity.getEntityManager().merge(entity);
    }
    return entity;
}
```

## Why This Is Non-Obvious

The method signature `T merge(T entity)` suggests an in-place operation. The method is named `merge`, not `copy` or `attach`. Java developers accustomed to mutable operations (e.g. `List.add()`) do not expect the input object to remain unchanged. No warning or exception is raised when the return value is discarded — the bug only surfaces when stale data is read back later.
