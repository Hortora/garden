## Symptom

After adding `quarkus.arc.exclude-types=io.example.NewBean` to `src/test/resources/application.properties`, the CDI beans that were previously excluded via the same property in `src/main/resources/application.properties` reappear and cause `AmbiguousResolutionException` or unexpected injection behaviour.

No warning is emitted. The previously configured exclusions are silently dropped.

## Root Cause

SmallRye Config applies standard config source priority: `src/test/resources/application.properties` has higher priority than `src/main/resources/application.properties`. When the same key exists in both files, the higher-priority source wins — the lower-priority value is completely ignored, not merged.

`quarkus.arc.exclude-types` takes a comma-separated string. There is no list-merge semantics; it behaves identically to any other string property. Setting it in the test config replaces the main config's value entirely.

## Fix

Re-declare the full combined list in `src/test/resources/application.properties`:

```properties
# Must include ALL exclusions from main application.properties here,
# because test config overrides (not appends) main config for this key.
quarkus.arc.exclude-types=\
  io.example.OriginallyExcludedBean,\
  io.example.AnotherExcludedBean,\
  io.example.NewTestOnlyBean
```

The `\` line-continuation syntax works in Quarkus application.properties.

## Why This Is Non-Obvious

Developers expect config list properties to merge (like Maven's `<includes>` which additive-extends parent POMs). SmallRye Config has no merge semantics — it's pure property override. The failure mode is also indirect: the original CDI exclusion silently stops working, causing a completely different set of errors (ambiguous dependencies, unexpected beans) that appear unrelated to the config change.
