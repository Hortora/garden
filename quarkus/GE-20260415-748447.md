## Quarkiverse extension depending on a sibling quarkus-ledger library requires manual mvn install of that library first

### What

When a Quarkiverse extension declares a `1.0.0-SNAPSHOT` dependency on a sibling library (`quarkus-ledger`) that lives in a separate local repository, running `mvn install` on the extension fails with "Could not find artifact" — even after installing the extension's own upstream modules. The sibling library must be installed into the local Maven repo (`~/.m2/`) separately before any module that depends on it will build.

### Context

```xml
<!-- In quarkus-tarkus-ledger/pom.xml -->
<dependency>
  <groupId>io.quarkiverse.ledger</groupId>
  <artifactId>quarkus-ledger</artifactId>
  <version>1.0.0-SNAPSHOT</version>
</dependency>
```

`quarkus-ledger` is a separate project at `~/claude/quarkus-ledger/` — not a module in the `quarkus-tarkus` reactor. Maven cannot resolve it from the local filesystem without a prior install.

### Fix

Install the sibling library first:

```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn install -DskipTests \
  -f ~/claude/quarkus-ledger/pom.xml
```

Then build the dependent extension normally. This step is not encoded in any `pom.xml` and not mentioned in any README — you only discover it when the build fails.

### Why this is non-obvious (undocumented)

Maven multi-module reactors handle intra-reactor dependencies automatically. Cross-repo SNAPSHOT dependencies look identical in `pom.xml` but require a completely different bootstrap step that Maven provides no guidance for. The error message ("Could not find artifact") looks like a network or repository configuration issue, not a missing install step. There is no standard Maven mechanism to declare "this dependency lives at sibling path X — install it first."
