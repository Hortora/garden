## Symptom

`given().when().post("/a2a/message:send").then().statusCode(501)` receives a 404 instead. The endpoint exists and starts correctly. Adding logging shows the request URL is `http://localhost:8081/a2a/message%3Asend`.

## Context

RestAssured's `.post(path)` and `.get(path)` methods apply percent-encoding to the path string by default. A literal `:` (colon) in a path segment is encoded to `%3A`. JAX-RS route matching is case-sensitive and literal — `@Path("/a2a/message:send")` does not match the inbound URI `/a2a/message%3Asend`.

This affects any A2A-style API path following Google's `collection:method` convention, RPC-over-REST patterns, or any path with a colon.

## What was tried

Quoting the colon, double-encoding, `given().baseUri(...)` — all still encode.

## Root cause

RFC 3986 §3.3 permits `:` (colon) in path segments after the first segment — it is not a reserved character in that position. RestAssured misapplies percent-encoding to it anyway.

## Fix

```java
given()
    .urlEncodingEnabled(false)   // ← suppress percent-encoding
    .contentType("application/json")
    .body(requestBody)
    .when().post("/a2a/message:send")
    .then().statusCode(200);
```

Apply `urlEncodingEnabled(false)` to every RestAssured call that hits a path containing a literal colon.

## Why this is non-obvious

The default assumption is that RestAssured handles URL construction correctly per RFC. The RFC explicitly permits the colon here, so the encoding is a RestAssured over-approximation. The symptom (404) gives no indication the URL is being rewritten — you only see it if you inspect RestAssured's request log.

### Why this fix

`urlEncodingEnabled(false)` disables all percent-encoding globally for that request. The alternative of pre-encoding the colon as `%3A` in the path string would cause double-encoding (`%253A`). The flag is the only clean option.
