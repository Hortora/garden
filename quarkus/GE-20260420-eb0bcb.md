## quarkus-mcp-server @Tool methods support `Uni<T>`, `CompletionStage`, and `@NonBlocking`

**ID:** GE-20260420-eb0bcb
**Stack:** quarkus-mcp-server 1.11.1, Quarkus 3.x
**Context:** Building reactive MCP tool implementations in a Quarkus extension.
The quarkus-mcp-server README and Quarkiverse extension docs only show blocking
return types for `@Tool` methods (`String`, POJO, `List<T>`). There is no
mention of reactive alternatives in the extension's own documentation.

### What is undocumented

`@Tool` methods can return reactive types for non-blocking execution:

```java
// Uni<T> — fully reactive, integrates with Mutiny pipelines
@Tool
public Uni<ChannelDetail> createChannel(
        @ToolArg String name,
        @ToolArg String semantic) {
    return channelService.create(name, semantic)
        .map(ch -> toDetail(ch));
}

// CompletionStage<T> — interoperates with Java async APIs
@Tool
public CompletionStage<String> asyncTool(@ToolArg String input) {
    return CompletableFuture.supplyAsync(() -> process(input));
}

// @NonBlocking — keeps blocking code off the event loop via worker thread
@Tool
@NonBlocking
public String blockingTool(@ToolArg String input) {
    return expensiveBlockingOperation(input);
}
```

All three forms are handled transparently by the MCP server runtime — the
caller receives the result synchronously over the MCP protocol regardless of
how the server computed it internally.

### Where it is documented (partially)

The general Quarkus MCP guide at `https://quarkus.io/guides/dev-mcp` mentions:
> "The method can return primitives, String, JsonObject, JsonArray, or any POJO
> that can be serialised to JSON. Asynchronous methods (Uni, CompletionStage or
> methods annotated with @NonBlocking) are also supported."

This is absent from:
- The quarkus-mcp-server Quarkiverse extension README
- The extension Javadoc
- Any quarkus-mcp-server release notes

### Why this matters

Without knowing reactive return types are supported, MCP extension authors either
write blocking tools that block the Vert.x event loop (correctness issue for
long-running tools like `wait_for_reply`), or build their own thread-offloading
wrappers. The `@NonBlocking` annotation alone is sufficient for simple cases; full
`Uni<T>` integration enables reactive pipelines through services and stores.
