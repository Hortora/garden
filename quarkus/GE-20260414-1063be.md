## Symptom

A Panache active-record `find()` or `list()` call with a hand-written WHERE clause returns 0 results or throws a JPQL exception, despite the data being present. No compile-time error.

## Context

Building dynamic JPQL WHERE clauses for Panache active-record style entities, using an entity alias (e.g. `wi.assigneeId`) as you would in standard JPQL.

## What Was Tried

```java
// Dynamic WHERE clause built like this:
assignment.append(" OR wi.assigneeId = :assignee OR wi.candidateUsers LIKE :like");
// ...
return WorkItem.find(query, params).list();
```

## Root Cause

Panache's short-form `find(String query, ...)` automatically wraps the condition in a full JPQL query with its own internal entity alias. The internal alias is NOT `wi` — it is generated internally by Panache. Using `wi.field` references a non-existent alias, which may silently return no results or fail.

## Fix

Use bare field names without any alias prefix:

```java
// Correct — no alias prefix:
assignment.append(" OR assigneeId = :assignee OR candidateUsers LIKE :like");
// ...
return WorkItem.find(query, params).list();
```

Alternatively, use the full JPQL form with an explicit `FROM` clause where you control the alias:
```java
WorkItem.find("FROM WorkItem wi WHERE wi.assigneeId = :a", params).list();
```

## Why This Is Non-Obvious

Standard JPQL always uses an alias (e.g. `SELECT e FROM Entity e WHERE e.field = :x`). Panache's short-form hides this, creating the expectation that field references work the same way. The error manifests as wrong results (empty list) not an exception, making it appear to be a data or parameter issue rather than a query syntax issue.
