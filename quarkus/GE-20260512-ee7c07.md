### Symptom

`UnsatisfiedResolutionException: Unsatisfied dependency for type <InterfaceType>` at application startup, even though:
- The `@Alternative` bean exists and is annotated correctly
- A `beans.xml` with `<alternatives><class>...</class></alternatives>` is in `src/main/resources/META-INF/`
- The application compiles and packages without error

### Context

Occurs when selecting an `@Alternative` bean from a dependency JAR (e.g. a library that ships a JPA implementation as `@Alternative` to a default). Standard CDI spec says `<alternatives>` in `beans.xml` selects alternatives within the bean archive. In Quarkus ArC, this mechanism is not supported.

### What was tried

Placed `beans.xml` with `<alternatives><class>io.example.JpaImpl</class></alternatives>` in `src/main/resources/META-INF/` — deployed without error, CDI failure at runtime with no hint about why.

Verified the class existed and was annotated `@Alternative @ApplicationScoped` via `javap -verbose`.

### Root cause

Quarkus ArC's CDI implementation intentionally does not honour `beans.xml` `<alternatives>` — a known divergence from the CDI spec. The `beans.xml` is present and parsed but has no effect on alternative selection.

### Fix

Use the Quarkus-specific config property:

```properties
quarkus.arc.selected-alternatives=io.example.JpaImpl
```

For multiple alternatives, comma-separate. Add to both `application.properties` and `src/test/resources/application.properties`.

### Why this is non-obvious

Standard CDI spec explicitly supports `beans.xml` alternative selection, and every CDI tutorial references this mechanism. Quarkus silently ignores it — no warning, no log message, no error that points toward the config property fix.
