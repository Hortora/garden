## Symptom

Running a Maven module with multiple `@QuarkusTest` classes intermittently fails with:

```
java.lang.RuntimeException: Unable to start HTTP server
```

One test class shows `1 Error, N Skipped`. On re-run, it may pass. The failure is non-deterministic — it passes in isolation but fails in the full suite.

## Context

A module with 4+ `@QuarkusTest` classes run by Maven Surefire in a single JVM. Quarkus restarts between classes (due to different injections or app state). The default HTTP test port is 8081.

## What Was Tried

Running the failing class in isolation — always passes. Running all classes together — intermittent failure in the second or third class to start. The Surefire summary shows the first class ran fine, then subsequent classes hit the port error.

## Root Cause

When Quarkus stops its HTTP server between test classes, the OS socket enters `TIME_WAIT` state for up to 60 seconds. The next Quarkus restart immediately tries port 8081 again, fails if the previous instance's socket hasn't cleared.

## Fix

Set `quarkus.http.test-port=0` in the module's `src/test/resources/application.properties`:

```properties
# Random port prevents TIME_WAIT conflicts when Quarkus restarts between test classes
quarkus.http.test-port=0
```

Port 0 tells the OS to assign any available ephemeral port. Each Quarkus restart gets a fresh port that isn't in TIME_WAIT. REST Assured automatically uses the correct port via Quarkus test infrastructure — no changes needed in test code.

## Why This Is Non-Obvious

The failure is intermittent — it passes when the OS socket clears quickly and fails when it doesn't. Running in isolation always passes. The error message mentions the HTTP server but not the port conflict, making it look like a Quarkus bootstrap failure. The fix (`test-port=0`) is a one-liner not mentioned in Quarkus testing documentation.
