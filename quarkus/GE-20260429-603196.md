### Symptom

OCC retry loop catches `jakarta.persistence.OptimisticLockException` but never fires — the retry never executes, and the operation silently fails. Awaitility-based tests time out (e.g. 5 seconds), with the `OptimisticLockException` visible only as a suppressed exception on `ConditionTimeoutException`, not as the directly thrown exception.

### Context

Triggered when a `@Transactional` method modifies a `@Version`-annotated JPA entity and two concurrent threads race to commit. The Hibernate flush happens during the JTA `beforeCompletion()` synchronization callback — after the method body returns but before the transaction coordinator calls `commit()`. At this point, the `OptimisticLockException` is caught by Narayana, the transaction is marked for rollback, and the exception propagates back through the JTA coordinator.

Typical retry pattern that does NOT work:

```java
try {
    policy.process();  // @Transactional
} catch (jakarta.persistence.OptimisticLockException e) {
    // Never reached — wrong exception type propagates
    retry();
}
```

### What was tried

Catching `jakarta.persistence.OptimisticLockException` directly — the catch block is never reached because the exception is wrapped or re-thrown as a different type by the Narayana JTA interceptor (`TransactionalInterceptorBase.invokeInOurTx`).

### Root cause

In Quarkus with Narayana JTA, the `OptimisticLockException` thrown during `beforeCompletion()` is handled internally by Narayana, which marks the transaction for rollback. The exception that propagates out of the `@Transactional` interceptor may be a `jakarta.transaction.RollbackException` wrapping the original, not the raw `OptimisticLockException`. The catch clause type-checks against the wrong exception class and silently skips the retry.

### Fix

Catch `Exception` broadly in the retry loop. On retry, the next `@Transactional` call loads a fresh entity with the updated version — `@Version` OCC ensures idempotency:

```java
for (int attempt = 0; attempt < 2; attempt++) {
    try {
        policy.process();  // @Transactional — starts a new transaction each call
        return;
    } catch (Exception e) {
        // OCC manifests as various JTA exception types in Quarkus/Narayana.
        // On retry, policyTriggered=true (if another thread won) → returns immediately.
    }
}
```

Pair with a `policyTriggered` boolean flag on the entity so the retry is idempotent: if another thread already resolved the state, the retry loads `policyTriggered=true` and returns without double-firing.

### Why this is non-obvious

The Quarkus `@Transactional` CDI interceptor appears to propagate exceptions transparently, so catching `OptimisticLockException` looks correct. The wrapping only occurs because the OCC fails during JTA synchronization (not inside the method body), which is invisible to the developer.
