## Context

Testing that exactly one `COMPLETED` group lifecycle event fires when an M-of-N threshold
is reached. Multiple concurrent child completions could theoretically both trigger the
coordinator — the `policyTriggered` guard prevents double-fire, but the timing between
event delivery and the assertion is not guaranteed.

## Problem with naive Awaitility

```java
// Passes as soon as the first COMPLETED event arrives
Awaitility.await().atMost(5, SECONDS)
        .until(() -> capture.hasStatus(GroupStatus.COMPLETED));

// Asserts hasSize(1) — but a concurrent duplicate event could arrive 10ms later
assertThat(capture.byStatus(GroupStatus.COMPLETED)).hasSize(1); // intermittent: hasSize(2)
```

The assert runs the moment `hasStatus(COMPLETED)` returns true — before any in-flight
duplicate has a chance to arrive.

## Technique: during() stability window

```java
// Asserts that count is EXACTLY 1 and STAYS at 1 for 300ms
// If a second event arrives within the window, the condition becomes false and Awaitility retries
Awaitility.await()
        .atMost(Duration.ofSeconds(5))
        .during(Duration.ofMillis(300))
        .until(() -> capture.byStatus(parentId, GroupStatus.COMPLETED).size() == 1);
```

`during(T)` means: the condition must be continuously true for at least T milliseconds.
Any event that pushes the count to 2 breaks the condition, Awaitility retries, and the
test fails cleanly rather than passing by race.

## Why this is non-obvious

Most developers reach for `await().until(condition)` — fire and forget. The `during()`
clause is less well-known and not in the basic Awaitility tutorial. It's specifically
valuable for "exactly N" count assertions where the count could increase due to races.
Without `during()`, such tests are subtly flaky — they fail when a duplicate arrives
before the assert, but pass when the assert runs first.
