## quarkus-hibernate-reactive-panache in an extension forces Hibernate Reactive to boot for all consumers ‚Äî @Alternative does not prevent it

**ID:** GE-20260420-dcec35
**Stack:** Quarkus 3.32.2, quarkus-hibernate-reactive-panache

**Symptom:** Adding `quarkus-hibernate-reactive-panache` to a Quarkus extension's runtime `pom.xml` causes `FastBootHibernateReactivePersistenceProvider` to try to initialise on every consumer app startup ‚Äî even those with no reactive datasource configured. Error: `Unable to find datasource '<default>' for persistence unit '<default>': No pool has been defined`.

Three test classes fail that previously passed: the two using `@TestProfile` (which restart Quarkus) and the one with `@TestTransaction` that triggers JTA coordination with all registered persistence units.

### Root cause
The Hibernate Reactive Panache extension scans for `PanacheRepositoryBase` implementations at **build time** (AOT). Any class implementing the interface ‚Äî even an `@Alternative` bean ‚Äî causes the extension to register a reactive persistence unit for the associated entity. `@Alternative` suppresses CDI bean activation but does NOT suppress the build-step registration. At runtime, Hibernate Reactive tries to find a reactive (Vert.x) pool for the registered persistence unit and fails if none is configured.

### Fix
Add to the consumer's test `application.properties`:

```properties
quarkus.datasource.reactive=false
```

This prevents Hibernate Reactive from claiming a reactive pool at runtime, even though the persistence unit was registered at build time.

### Why non-obvious
The error message points to a missing datasource pool ‚Äî suggesting the fix is to configure a datasource, not to disable reactive. Marking the `PanacheRepositoryBase` bean as `@Alternative` is the obvious CDI workaround and feels correct, but `@Alternative` only controls CDI injection ‚Äî it has no effect on Quarkus build steps. The fix (`quarkus.datasource.reactive=false`) suppresses the runtime pool lookup entirely, not via CDI.

### Alternatives considered
- `@Alternative` on reactive beans ‚Äî does not prevent build-step registration; persistence unit still created
- `quarkus.hibernate-reactive.active=false` ‚Äî property not recognised in Quarkus 3.32.2
- Testcontainers + PostgreSQL ‚Äî works but adds heavy test infrastructure; unnecessary when H2 suffices
