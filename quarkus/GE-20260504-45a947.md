## Quarkus test profile enabling a `@Scheduled` feature must also disable the scheduler — or it races direct `runComputation()` calls

**ID:** GE-20260504-45a947
**Stack:** Quarkus 3.x (all versions)
**Symptom:** Integration tests that call `job.runComputation()` directly occasionally produce assertion failures — wrong row counts, missing rows, or constraint violations — that don't reproduce when tests run in isolation or slowly.
**Context:** A `@QuarkusTestProfile` enables a feature (`casehub.ledger.trust-score.enabled=true`) to test a nightly batch job. Tests disable the scheduler implicitly by not setting a short interval — but the Quarkus scheduler still fires on the configured `@Scheduled(every=...)` trigger if any time passes.

### What was tried (didn't work)
- Tests pass in isolation and on fast machines — intermittent failures are assumed to be flaky Quarkus H2 behaviour
- Assumed enabling the feature flag without a short interval was sufficient to prevent concurrent fires

### Root cause
`@QuarkusTestProfile` settings that enable a feature do not automatically disable the Quarkus scheduler. If the test calls `runComputation()` directly AND the scheduler fires its own invocation concurrently, both write to the same H2 database tables in the same transaction window. The 24h interval makes this rare — but any test harness that runs long or advances the clock can trigger it.

### Fix
Add `quarkus.scheduler.enabled=false` to the test profile in `application.properties`:

```properties
%trust-score-test.casehub.ledger.trust-score.enabled=true
%trust-score-test.quarkus.scheduler.enabled=false   # ← required
```

The pattern is already used in Quarkus's own routing-test and routing-disabled-test profiles in this codebase — easy to miss when adding a new feature test profile.

### Why this is non-obvious
The feature profile and the scheduler are independent concerns. Enabling `trust-score.enabled=true` implies the feature works — it doesn't imply the scheduler is running. Developers set the interval to `24h` thinking "it won't fire during tests" without realising the scheduler is still active and can race direct calls. Existing profiles in the same repo had the guard, making the gap invisible until a late code review.
