## Use per-case DB query to isolate async worker assertions

**ID:** GE-20260417-d67b22
**Stack:** Quarkus 3.17.x, JUnit 5, Mutiny, Quartz
**Context:** `@QuarkusTest` integration tests with async workers (Quartz). Multiple test methods in the same class share a `static final AtomicInteger runCount`. `@BeforeEach` resets to 0, but a previous test's worker that completes *after* the `@BeforeEach` reset contaminates the counter for the next test. Result: `expected: <1> but was: <2>`.

### The technique

Query the database for events tied to *this test's* case UUID, rather than checking a global counter:

```java
// Instead of: (global counter, contaminated by other tests' async workers)
assertEquals(1, SomeBean.runCount.get());

// Use: (case-specific DB query, immune to contamination)
assertEquals(1, findWorkerEvents(caseId, CaseHubEventType.WORKER_EXECUTION_COMPLETED, "payment-worker").size());

// Where findWorkerEvents queries by case UUID:
private List<EventLog> findWorkerEvents(UUID caseId, CaseHubEventType type, String workerId) {
    return VertxContextSupport.subscribeAndAwait(() ->
        sessionFactory.withSession(session ->
            session.createSelectionQuery(
                "from EventLog where caseId = :caseId and eventType = :type and workerId = :workerId",
                EventLog.class)
            .setParameter("caseId", caseId)
            .setParameter("type", type)
            .setParameter("workerId", workerId)
            .getResultList()));
}
```

Each test creates a new case with a unique UUID (generated by `startCase()`). The DB query filters by that UUID — other tests' workers write events under different UUIDs, so they can't appear in this query.

### Why this is non-obvious

A `static final AtomicInteger` counter on the bean seems like the natural, lightweight way to count worker executions. The timing issue — that a previous test's async worker can complete after `@BeforeEach` reset but before the current test's assertion — only manifests occasionally (more likely on slower/more loaded CI machines). The DB query approach seems heavier but is actually more correct and deterministic.

### Why this fix

`@BeforeEach` reset timing is fundamentally fragile for async operations. The DB query approach uses the natural isolation boundary (case UUID) that already exists in the data model, eliminating the contamination window entirely rather than narrowing it.
