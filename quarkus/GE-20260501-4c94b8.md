## Symptom

```
java.lang.ClassCastException: Cannot cast io.vertx.mutiny.sqlclient.SqlConnection
    to io.vertx.mutiny.pgclient.PgConnection
```

Occurs when trying to use `pool.getConnection()` to get a `PgConnection` for calling
`notificationHandler()` (needed for PostgreSQL LISTEN/NOTIFY).

## Context

Needed a dedicated `PgConnection` held open for LISTEN/NOTIFY (PostgreSQL pub/sub for
distributed SSE fan-out). `PgPool.getConnection()` is the natural way to get a connection
from an injected pool.

## What was tried

```java
// Fails at runtime with ClassCastException
pool.getConnection()
    .onItem().castTo(PgConnection.class) // ClassCastException
    .subscribe()...
```

Also tried: direct cast `(PgConnection) conn` — same result.

## Root cause

Vert.x Mutiny codegen generates separate wrapper classes for each Vert.x type. 
`io.vertx.mutiny.sqlclient.SqlConnection` and `io.vertx.mutiny.pgclient.PgConnection`
are separate classes even though their underlying delegates share the `PgConnection`
delegate hierarchy. The Mutiny wrappers do NOT mirror the delegate inheritance chain.

`PgPool.getConnection()` is inherited from `io.vertx.mutiny.sqlclient.Pool` and returns
`Uni<SqlConnection>` — not `Uni<PgConnection>`. The actual delegate IS a
`io.vertx.pgclient.PgConnection`, but the Mutiny wrapper is `SqlConnection`.

## Fix

```java
// PgPool always yields PgConnection delegates — unwrap and re-wrap.
pool.getConnection().subscribe().with(conn -> {
    final io.vertx.pgclient.PgConnection pgDelegate =
            (io.vertx.pgclient.PgConnection) conn.getDelegate();
    final PgConnection pgConn = PgConnection.newInstance(pgDelegate);
    pgConn.notificationHandler(n -> handleNotification(n.getPayload()));
    pgConn.query("LISTEN " + CHANNEL).execute().subscribe().with(...);
});
```

## Why this is non-obvious

The Mutiny class hierarchy docs don't mention that wrapper types are generated flat
(no inheritance mirroring). A developer would reasonably assume that since `PgPool`
extends `Pool` and yields `PgConnection` instances at the delegate level, the Mutiny
wrappers would behave similarly. They don't. `.getDelegate()` + `.newInstance()` is
the only way to cross wrapper type boundaries.
