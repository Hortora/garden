## Context

You have a default implementation of an interface (a naive baseline, a no-op, a stub) that should be active unless a richer implementation is present. The richer implementation may be added in a future layer, by a downstream module, or conditionally.

## The technique

Mark the default implementation `@ApplicationScoped @DefaultBean`. Mark the richer implementation `@ApplicationScoped` only (no qualifier, no `@Alternative`, no `@Priority`).

```java
// Layer 1 — active only when no other bean of this type exists
@ApplicationScoped
@DefaultBean
public class NaiveAmlInvestigationService implements AmlInvestigationApplicationService { ... }

// Layer 2 — silently wins; NaiveAmlInvestigationService becomes dormant
@ApplicationScoped
public class WorkItemAmlInvestigationService implements AmlInvestigationApplicationService { ... }
```

CDI resolves to `WorkItemAmlInvestigationService` when both are present. `NaiveAmlInvestigationService` is still a valid CDI bean — injecting it by concrete type works fine — but it is no longer the resolution target for the interface.

## Why this is non-obvious

Developers reaching for "replaceable default" typically use `@Alternative @Priority(N)` (verbose, requires `@Priority` on the override) or profile-based activation (`@IfBuildProperty`, evaluated at build time only). `@DefaultBean` is simpler: the mere presence of any other bean of the same type suppresses it. No annotation on the override, no configuration, no profile.

## Constraints

- Both beans must be in the same Quarkus application (same augmentation pass).
- If you need to inject the `@DefaultBean` implementation directly by concrete type alongside the override, that works — CDI resolves by the concrete class, not by the interface.
- `@DefaultBean` is a Quarkus ARC extension, not CDI standard — import from `io.quarkus.arc.DefaultBean`.
