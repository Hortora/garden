## @Blocking @Tool + private blockingXxx helper — pattern for mixing blocking and reactive in quarkus-mcp-server

### Context

`ReactiveQhorusMcpTools` needed to return `Uni<T>` from all 39 `@Tool` methods (quarkus-mcp-server 1.11.1 supports this natively). However, 19 of those tools require blocking operations: Panache entity statics (`Message.find(...)`, `PendingReply.delete(...)`), long-poll loops with `Thread.sleep`, and complex multi-step transactional flows not yet expressible as pure reactive chains.

### Technique

Annotate the `@Tool` method with `@Blocking` and delegate to a private `blockingXxx` helper that contains the actual blocking logic:

```java
@Tool(name = "send_message", description = "...")
@Transactional
@Blocking
public Uni<MessageResult> sendMessage(/* @ToolArg params */) {
    return Uni.createFrom().item(() -> blockingSendMessage(/* params */));
}

private MessageResult blockingSendMessage(/* params */) {
    // Exact blocking logic — Panache statics, blocking service calls, etc.
    Channel ch = blockingChannelService.findByName(channelName).orElseThrow(...);
    // ... full implementation ...
    return new MessageResult(...);
}
```

**Why this works:**
- `@Blocking` tells Quarkus to execute this method on a worker thread (not the Vert.x event loop)
- `Uni.createFrom().item(supplier)` wraps the blocking result in a `Uni<T>`; the supplier executes synchronously when subscribed
- quarkus-mcp-server subscribes to the returned `Uni` and handles the result correctly
- `@Transactional` on the `@Tool` method provides transaction context on the worker thread (JTA works on worker threads)

**Key benefit:** `@Tool` methods that call each other internally can share the private `blockingXxx` helpers directly (bypassing the CDI proxy), which is critical for `requestApproval` → `blockingSendMessage` → `blockingWaitForReply` call chains that must NOT hold a transaction across a long-poll loop.

### Why this is non-obvious

The natural approach is to either: (a) make the entire class blocking, losing the reactive API; or (b) try to convert all tool methods to pure reactive chains, which fails for blocking-only operations like `Thread.sleep`. The private-helper delegation pattern cleanly separates the reactive interface from the blocking implementation.
