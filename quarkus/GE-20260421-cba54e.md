## Context

Migrating a Python server to Quarkus. The Python server calls an initialisation function at module load time (e.g. `bridge_init()`) that reads a config file, activates the first project, and populates shared singleton state. The Java equivalent (`ActiveProject` CDI bean) is only populated on an explicit HTTP POST request — so tests and the first API call fail with "no active project" because nothing initialised the state at startup.

## The technique

Create a `@ApplicationScoped` CDI bean with a `@Observes StartupEvent` method that replicates the Python startup behaviour:

```java
@ApplicationScoped
public class StartupActivation {

    @Inject ProjectsStore store;
    @Inject ActiveProject activeProject;

    void onStart(@Observes StartupEvent event) {
        try {
            List<ObjectNode> projects = store.load();
            if (projects.isEmpty()) return;
            String id         = projects.get(0).path("id").asText();
            Path   projectDir = store.getProjectDir(id);
            Path   configPath = projectDir.resolve("config.json");
            if (Files.exists(configPath)) {
                activeProject.set(id, SpargeConfig.load(configPath, projectDir), projectDir);
            }
        } catch (Exception e) {
            // Non-fatal — server works without an active project
        }
    }
}
```

`StartupEvent` fires after all `@PostConstruct` methods complete, so beans are fully initialised before this runs.

## Why this is non-obvious

Developers migrating from dynamic languages expect module-load-time initialisation to have a direct Java equivalent. The instinct is to put the logic in a `@PostConstruct` on the resource bean — but resource beans are not eagerly instantiated. The correct hook is a standalone `@ApplicationScoped` bean observing `StartupEvent`, which *is* eagerly instantiated.
