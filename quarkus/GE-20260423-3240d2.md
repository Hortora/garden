## The Technique

`PanacheEntityBase.delete(query, params)` returns a `long` count of the rows deleted. This return value is commonly discarded, but it eliminates the need for a separate count query before deleting.

```java
// Common pattern — two round-trips, TOCTOU window between find and delete
List<MyEntity> expired = MyEntity.find("expiresAt < ?1", cutoff).list();
if (!expired.isEmpty()) {
    MyEntity.delete("expiresAt < ?1", cutoff);
    log.infof("Deleted %d expired entries", expired.size());
}

// Better — one round-trip, accurate count, no TOCTOU
long deleted = MyEntity.delete("expiresAt < ?1", cutoff);
if (deleted > 0) {
    log.infof("Deleted %d expired entries", deleted);
}
```

## Why This Is Non-Obvious

Developers reaching for "how many rows were deleted" instinctively do a count query first or a find before the delete. The `delete()` return value is not highlighted in the Panache documentation — it appears in the method signature but is easy to overlook. The find-then-delete pattern also introduces a TOCTOU window: rows can expire between the find and the delete, making the logged count inaccurate.

## When to Use

Any cleanup job, expiry scheduler, or batch delete where the caller needs to know how many rows were affected — without paying for a separate SELECT round-trip.
