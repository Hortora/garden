## Maven module order causes 'missing table' schema validation failure

**ID:** GE-20260417-bbaa4b
**Stack:** Quarkus 3.17.x, Hibernate ORM, Flyway, Maven multi-module
**Symptom:** `SchemaManagementException: Schema validation: missing table [case_instance]` in a module whose tests run early in the Maven build. The error only appears in CI, not when running that module's tests alone.
**Context:** Multi-module Maven project. Flyway migrations were moved from the `engine` module to a separate `persistence-hibernate` module. The parent pom lists `casehub-blackboard` before `casehub-persistence-hibernate`. Maven runs tests in declaration order.

### What was tried (didn't work)

- Checked the failing module's own `application.properties` — Flyway config looked correct
- Ran the failing module's tests in isolation — they passed (because the `persistence-hibernate` module had been installed first by a previous `mvn install`)
- CI failure only reproduced because CI runs a clean build with all modules sequentially

### Root cause

Maven builds and tests modules in the order they're declared in the parent pom's `<modules>` list. When `casehub-blackboard` is listed before `casehub-persistence-hibernate`, its `@QuarkusTest` tests start a fresh PostgreSQL container, Flyway has never run (it lives in `casehub-persistence-hibernate`), so Hibernate's `validate` strategy finds no tables.

The failure only appears in a full multi-module build — running any single module in isolation works because `mvn install -DskipTests` was run first, putting jars on `.m2`.

### Fix

Two options:

**Option A — Reorder modules in parent pom** so `casehub-persistence-hibernate` appears before all modules that need its schema:
```xml
<modules>
    <module>casehub-persistence-hibernate</module>  <!-- must come first -->
    <module>casehub-blackboard</module>
</modules>
```

**Option B — Use Hibernate create strategy in test properties** for the upstream module, bypassing Flyway entirely:
```properties
# In casehub-blackboard/src/test/resources/application.properties
%test.quarkus.hibernate-orm.schema-management.strategy=create
```

Option B is lower coupling — the module doesn't need to know about the schema module's order.

### Why this is non-obvious

The failing module's error message (`missing table [case_instance]`) gives no indication that the cause is in a different module running later in the build. It looks exactly like a Flyway configuration or classpath problem local to the failing module. The isolation test (running the module alone) passes, which misleads the diagnosis toward "it's only a CI environment issue."
