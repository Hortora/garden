## Hibernate Reactive in Quarkus 3.x: `runOnContext()` alone throws context safety error — use `VertxContextSupport.subscribeAndAwait()`

**ID:** GE-20260416-f02f95
**Stack:** Quarkus 3.31.1, Hibernate Reactive, Panache, Mutiny
**Symptom:** `@QuarkusTest` methods that call `Panache.withSession()` or `Panache.withTransaction()` directly throw `IllegalStateException: No current Vertx context found`. After fixing that by scheduling on the event loop via `vertx.getDelegate().runOnContext()`, calls instead throw `IllegalStateException: Can't get the context safety flag: the current context is not a duplicated context`.
**Context:** Direct repository calls from JUnit test methods in a Quarkus 3.x reactive project. The existing engine tests don't hit this because they never call Panache directly from test methods — they trigger the event-driven system and wait for side effects.

### What was tried (didn't work)
- `uni.await().indefinitely()` — throws `No current Vertx context found` immediately
- `@Inject Vertx vertx` then `vertx.getDelegate().runOnContext(v -> supplier.get().subscribe()...)` with a `CompletableFuture` — gets past the first error but throws `Can't get the context safety flag: the current context is not a duplicated context` because Quarkus 3.x requires a *duplicated, safety-marked* context, not just any event loop context

### Root cause
Quarkus 3.x introduced a context safety model: Hibernate Reactive operations require a Vert.x context that has been explicitly *duplicated* and marked as safe via `VertxContextSafetyToggle`. A plain `runOnContext()` handler is on the event loop but is NOT a duplicated context — it fails the safety check. The `@RunOnVertxContext` test annotation (documented approach) handles this automatically but requires `io.quarkus:quarkus-test-vertx` which is not in the standard `quarkus-junit5` dependency.

### Fix
Use `VertxContextSupport.subscribeAndAwait()` from `io.quarkus:quarkus-vertx`, which is already on the classpath (no extra dependency needed). It creates a properly duplicated, safety-marked context, runs the `Uni` supplier on it, and blocks the test thread until completion:

```java
import io.quarkus.vertx.VertxContextSupport;
import io.smallrye.mutiny.Uni;
import java.util.function.Supplier;

// Helper in test class — no extra @Inject needed
private <T> T run(Supplier<Uni<T>> supplier) {
    try {
        return VertxContextSupport.subscribeAndAwait(supplier);
    } catch (RuntimeException e) {
        throw e;
    } catch (Throwable e) {
        throw new RuntimeException(e);
    }
}

// Usage in test:
CaseMetaModel saved = run(() -> repository.save(metaModel));
assertThat(saved.getId()).isNotNull();
```

`VertxContextSupport` is in `io.quarkus:quarkus-vertx` which is already on the classpath via `quarkus-hibernate-reactive-panache` → `quarkus-vertx`. No additional dependency needed.

### Why this is non-obvious
The error message (`No current Vertx context found`) correctly diagnoses the first problem but the obvious fix (`runOnContext`) produces a different, more cryptic error. The second error (`not a duplicated context`) is a Quarkus 3.x concept with no analogue in earlier versions — the difference between "any event loop context" and "a safe duplicated context" is not documented in the testing guide. `VertxContextSupport` is in `quarkus-vertx` but is not mentioned anywhere in the Quarkus testing documentation as a test utility — the docs exclusively point to `@RunOnVertxContext` (requires an extra artifact not in the BOM).

### Why this fix
`VertxContextSupport.subscribeAndAwait()` handles all three requirements in one call: schedules on the event loop, duplicates the context, marks it as safe. `@RunOnVertxContext` does the same but changes the test method signature to accept `UniAsserter` and requires an additional artifact. The `VertxContextSupport` approach keeps tests using standard AssertJ assertions and needs zero extra dependencies.
