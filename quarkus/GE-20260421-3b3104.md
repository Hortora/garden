## Symptom

A utility method that builds a `Map<String, Object>` from a WorkItem entity using
`workItem.getClass().getFields()` + `Field.get(workItem)` returns `null` or `0` for
**every field**, even though the entity is fully populated in the database and accessible
via normal field access (`workItem.confidenceScore`, `workItem.category`, etc.).

Downstream: all JEXL filter conditions evaluate to `false` even when they should match.
No error is thrown — the map is populated with null/zero values silently.

## Context

Occurred when building a JEXL context map for a `FilterRegistryEngine` that needed
to expose all `WorkItem` public fields as JEXL variables. The reflective approach
seemed natural for forward-compatibility (new fields on `WorkItem` would auto-appear).

## What was tried

- `Field.get(workItem)` with `field.setAccessible(true)` — still returns nulls
- Checking `Field.getModifiers()` for static/transient — correct fields selected, still nulls
- Verifying the entity is loaded (it is — direct field access like `workItem.category` works)

## Root cause

Hibernate's bytecode enhancement (used automatically in Quarkus Panache) replaces the entity's
field storage with its own mechanism in a generated subclass. When you call
`field.get(workItem)` via reflection, Java reads the **original field slot in the parent class**,
which is never written to — Hibernate stores values elsewhere in the enhanced subclass.
This is a documented limitation of Hibernate bytecode enhancement but not surfaced as an error.

## Fix

Build the JEXL context map by direct field access, not reflection:

```java
static Map<String, Object> toMap(final WorkItem wi) {
    final Map<String, Object> m = new HashMap<>();
    m.put("id",                wi.id);
    m.put("category",          wi.category);
    m.put("priority",          wi.priority);
    m.put("confidenceScore",   wi.confidenceScore);
    m.put("assigneeId",        wi.assigneeId);
    m.put("candidateGroups",   wi.candidateGroups);
    m.put("candidateUsers",    wi.candidateUsers);
    m.put("status",            wi.status);
    // ... all other public fields
    return m;
}
```

Add a drift-protection test that counts public non-static fields via reflection and
asserts the map size matches — so new fields on `WorkItem` fail the test rather than
silently disappearing from the JEXL context:

```java
@Test
void toMap_containsAllPublicNonStaticWorkItemFields() {
    final long expected = Arrays.stream(WorkItem.class.getFields())
        .filter(f -> !Modifier.isStatic(f.getModifiers()))
        .count();
    assertThat(JexlConditionEvaluator.toMap(new WorkItem()))
        .hasSize((int) expected);
}
```

## Why this is non-obvious

The entity works perfectly via normal field access and JPA queries. There is no error
or warning — `Field.get()` returns null silently. The Hibernate docs mention bytecode
enhancement as a performance feature without warning about this reflective-access side effect.
A developer familiar with Java reflection would have no reason to suspect it.
