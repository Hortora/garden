## quarkus-junit (wrong) vs quarkus-junit5 (correct) — compiles silently, fails at test runtime

### Symptom

Tests compile cleanly but `@QuarkusTest` classes either fail to load the Quarkus test framework or produce cryptic class-not-found errors at runtime. No compile-time warning.

### Context

Occurs when `pom.xml` declares:

```xml
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-junit</artifactId>  <!-- wrong -->
  <scope>test</scope>
</dependency>
```

instead of:

```xml
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-junit5</artifactId>  <!-- correct -->
  <scope>test</scope>
</dependency>
```

Particularly likely when an AI assistant, code generator, or copy-paste from an older example supplies the dependency, since `quarkus-junit` is a plausible-sounding artifact ID.

### Root cause

`quarkus-junit` is a legacy or stub artifact that resolves in Maven Central (or local repo) and compiles without error. It does not contain `@QuarkusTest`, `QuarkusTestExtension`, or any of the test infrastructure. `quarkus-junit5` is the artifact that actually provides the Quarkus JUnit 5 integration.

### Fix

Replace `quarkus-junit` with `quarkus-junit5` in `pom.xml`:

```xml
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-junit5</artifactId>
  <scope>test</scope>
</dependency>
```

### Why this is non-obvious

`quarkus-junit` resolves and compiles. There is no Maven warning, no missing-class error at compile time, and no IDE red underline. The failure only surfaces at test runtime when the JUnit 5 extension cannot be found.
