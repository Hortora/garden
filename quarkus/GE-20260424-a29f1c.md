## Symptom

`@PersistenceUnit("qhorus") EntityManager em` is committed to git but the working tree reverts to `@Inject EntityManager em` after IntelliJ opens the file. The `@PersistenceUnit` annotation and its import disappear silently — no warning, no formatting notification.

## Context

Occurs when IntelliJ's "Optimize Imports" or "Reformat Code" runs on Java files containing `@io.quarkus.hibernate.orm.PersistenceUnit`. IntelliJ treats the annotation as an unused import if it doesn't recognise the Quarkus-specific CDI producer that makes the qualifier meaningful.

## Root Cause

IntelliJ's formatter analyses annotations for usage. `@PersistenceUnit` from `io.quarkus.hibernate.orm` is a CDI qualifier resolved at runtime — the formatter doesn't see it used at the call site (it's a field annotation) and removes the import, which also strips the annotation from the field.

## Fix

Always restore from git HEAD before building:
```bash
git checkout -- path/to/affected/file.java
mvn install -DskipTests  # build immediately before IDE reformats again
```

Long-term: configure IntelliJ to not auto-reformat these files (Settings → Editor → Code Style → Java → Imports → set "Optimize imports on the fly" off for the module, or add a `.editorconfig` that disables import optimization per directory).

## Why This Is Non-Obvious

The annotation is committed — `git log` and `git show HEAD:file` both show it present. The file on disk matches HEAD until IntelliJ's background formatter runs. `mvn compile` then picks up the working tree (not HEAD), compiling the stripped version. The CDI failure only appears at test startup, not at compile time, making the root cause hard to trace.
