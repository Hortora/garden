## Symptom

Playwright E2E tests fail with `ERR_CONNECTION_REFUSED` or wrong-port errors. The failure looks like it was caused by your recent changes — but reverting those changes produces the same failures.

## Context

`PlaywrightBase` (the abstract base class for Playwright E2E tests) has a static field:

```java
protected static final String BASE_URL = "http://localhost:8081";
```

With `quarkus.http.test-port=0`, Quarkus binds to a random port on each test run. If the random port happens not to be 8081 (which is typical when 8081 is already in use), all page navigation fails. The `quarkus.http.test-port=0` setting was added specifically to prevent "Port already bound" errors — but the hardcoded URL undermined it.

## What was tried

- Reverting recent changes — same failures
- Running `git checkout <old-commit> -- terminal.js session.html` and retesting — still same failures (2 failures + 6 errors vs current 1 + 2)
- Concluded the failures were pre-existing and unrelated to the PR under review

## Root cause

`static final` fields are initialised at class-load time. `PlaywrightBase.BASE_URL` was set before the Quarkus test server started and before the real port was known. Even with `quarkus.http.test-port=0` in `application.properties`, the static final field always held `"http://localhost:8081"`.

## Fix

Make `BASE_URL` a non-final static, initialise it in `@BeforeAll` via MicroProfile Config:

```java
protected static String BASE_URL;  // not final

@BeforeAll
static void launchBrowser() {
    BASE_URL = ConfigProvider.getConfig().getValue("test.url", String.class);
    playwright = Playwright.create();
    // ...
}
```

Quarkus sets the `test.url` MicroProfile Config property to the actual bound URL (e.g. `http://localhost:54321`) after the test server starts, and `@BeforeAll` runs after Quarkus startup — so this always resolves correctly.

## Why this is non-obvious

The symptom (E2E failures with port errors) looks exactly like a regression introduced by frontend changes to `session.html` or `terminal.js`. You spend time on a regression investigation before noticing the base class has a hardcoded port. The hardcoded value was always wrong for random-port configurations but only surfaced intermittently when 8081 happened to be free.

### Why this fix

`ConfigProvider.getConfig().getValue("test.url")` is the idiomatic Quarkus test pattern — the framework sets `test.url` automatically after server startup. Using `System.getProperty("quarkus.http.port")` is an alternative but is less reliable (the system property may not be set in all Quarkus versions; `test.url` is always set in `@QuarkusTest` context).
