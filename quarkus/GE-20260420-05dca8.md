## Symptom

A `@QuarkusTest` calling `given().get("/events")` on an SSE endpoint hangs for 30+ seconds then throws:

```
java.net.SocketTimeoutException: Read timed out
```

The test never completes. Even adding `.then().statusCode(200)` before reading the body hangs.

## Context

Server-Sent Events use a persistent HTTP connection that the server never closes. REST Assured's response handling tries to read the full response body before returning control — for SSE this means it blocks forever waiting for the stream to end.

## Root cause

REST Assured buffers the entire response body into memory before any assertions can be made. SSE endpoints send `Transfer-Encoding: chunked` and never send a final empty chunk (they stay open). REST Assured's `RestAssuredResponseOptionsGroovyImpl.charsetToString()` reads until EOF, which never arrives.

## Fix

Use `java.net.http.HttpClient` (Java 11+, no extra dependencies) for SSE testing:

```java
// Integration test: check content-type only (no body read)
private HttpResponse<InputStream> connectSse(String path) throws Exception {
    HttpClient client = HttpClient.newHttpClient();
    HttpRequest request = HttpRequest.newBuilder()
        .uri(baseUri.resolve(path))
        .header("Accept", "text/event-stream")
        .build();
    return client.send(request, HttpResponse.BodyHandlers.ofInputStream());
}

@Test void sseEndpoint_returns200_withSseContentType() throws Exception {
    HttpResponse<InputStream> response = connectSse("/workitems/events");
    try (InputStream ignored = response.body()) {
        assertThat(response.statusCode()).isEqualTo(200);
        assertThat(response.headers().firstValue("content-type").orElse(""))
            .contains("text/event-stream");
    }
    // Close immediately after checking headers — don't read the body
}

// E2E test: receive an actual event
@Test void happyPath_createItem_appearsInStream() throws Exception {
    CountDownLatch latch = new CountDownLatch(1);
    List<String> lines = new CopyOnWriteArrayList<>();

    Thread.ofVirtual().start(() -> {
        try {
            HttpClient client = HttpClient.newHttpClient();
            HttpRequest req = HttpRequest.newBuilder()
                .uri(baseUri.resolve("/events"))
                .header("Accept", "text/event-stream").build();
            client.send(req, HttpResponse.BodyHandlers.ofLines())
                .body()
                .filter(l -> l.startsWith("data:"))
                .peek(lines::add)
                .findFirst()
                .ifPresent(l -> latch.countDown());
        } catch (Exception ignored) {}
    });

    Thread.sleep(300); // let connection establish
    triggerEvent();    // POST something that fires the event

    assertThat(latch.await(4, TimeUnit.SECONDS)).isTrue();
}
```

Inject `@TestHTTPResource("/") URI baseUri` to get the test server base URL.

## Why this is non-obvious

The REST Assured API looks like it should work — `given().header("Accept","text/event-stream").get("/events")` is syntactically valid. There's no documentation warning about SSE incompatibility. The SocketTimeoutException points at network I/O, not at REST Assured's body buffering.
