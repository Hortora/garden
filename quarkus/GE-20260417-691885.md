## The technique

Add `@WrapBusinessError` at the class level on an `@ApplicationScoped` MCP tools bean to convert business exceptions (`IllegalArgumentException`, `IllegalStateException`) into `isError: true` MCP tool responses — without changing any method signatures or return types.

```java
@WrapBusinessError({ IllegalArgumentException.class, IllegalStateException.class })
@ApplicationScoped
public class MyMcpTools {

    @Tool(name = "do_thing", description = "...")
    public MyResult doThing(@ToolArg(name = "id") String id) {
        // throws IllegalArgumentException if id is invalid
        // → caller receives {"result": {"isError": true, "content": [{"type":"text","text":"..."}]}}
        // NOT a JSON-RPC -32603 error
    }
}
```

## Why this is non-obvious

The naive approach for MCP tool error handling is either:
- Change all return types to `String` and return "Error: ..." text (works but loses structured returns)
- Change all return types to `ToolResponse` (invasive, breaks API)
- Add per-method try/catch (repetitive, 39 methods in Qhorus)

`@WrapBusinessError` achieves the same result with a single annotation. The interceptor checks `context.getMethod().isAnnotationPresent(Tool.class)` before wrapping — so only `@Tool`-annotated methods are wrapped, and the happy-path return type (`ChannelDetail`, `List<MessageSummary>`, etc.) is unchanged.

## Context

Applied to `QhorusMcpTools` which has 39 `@Tool` methods returning structured types. Previously, business exceptions escaped as JSON-RPC protocol errors (-32603), which Claude Code received as unreadable error objects. After adding `@WrapBusinessError`, all 39 tools produce `isError: true` content that Claude reads like normal tool output. Zero return type changes required.

## Caveat

See gotcha GE-20260417-45f47f — any other CDI bean that injects and calls the tools bean's `@Tool` methods also receives `ToolCallException` instead of the original exception. Those call sites must handle both exception types.
