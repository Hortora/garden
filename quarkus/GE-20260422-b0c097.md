## Symptom

Browser opens the visualizer and shows "Connecting..." permanently. The WebSocket handshake
completes successfully (network tab shows 101), but no game state message ever arrives.
Appears to be a WebSocket connectivity bug.

## Context

A showcase/demo endpoint seeds game state, calls `engine.observe()` to push to currently
connected browsers, then calls `engine.leaveGame()` to stop the AI scheduler from mutating
the state. Browsers that open **after** `leaveGame()` has been called receive no state.

## Root Cause

`engine.leaveGame()` sets `connected = false` in the game engine. The game loop checks
this flag and returns early — meaning no further `observe()` calls happen. `addSession()`
in `GameStateBroadcaster` adds the client to the session set but does **not** immediately
push the current game state. The new client sits in the session set waiting for a frame
event that never fires.

## Fix

Do not call `engine.leaveGame()` when browsers may connect afterward. Instead, leave the
engine running. If you need a static snapshot (AI should stop mutating), use
`simulatedGame.reset()` directly without firing `GameStarted` — this keeps `connected =
false` without stopping the ability to call `engine.observe()` manually:

```java
// Showcase endpoint — static state, no AI, browsers can still connect
simulatedGame.reset();           // resets state without starting AI scheduler
simulatedGame.spawnEnemyUnit(…); // seed showcase units
engine.observe();                // push to currently connected browsers
// Do NOT call engine.leaveGame() — browsers connecting later still need state
```

If you need `addSession()` to immediately push state, add `engine.observe()` there — but
note this breaks WebSocket tests that use flow-controlled queues (`ws.request(1)`) because
the extra observe message arrives before the test's own explicit observe call.

## Why This Is Non-Obvious

The WebSocket connection itself succeeds. Browser DevTools show an open WebSocket connection
with no errors. The symptom (stuck "Connecting...") looks exactly like a network or auth
failure. The root cause is entirely on the server side — the engine's `connected` flag
gates whether any state is ever broadcast to new clients.
