### Symptom
`CaseInstance not found or wrong tenant` (or similar state-not-found error) despite the instance being saved moments earlier. The save and lookup use the same `CaseInstanceRepository` interface, but they resolve to different `@ApplicationScoped` bean instances with separate `ConcurrentHashMap` stores.

### Context
`casehub-engine-testing` provides `TestCaseInstanceRepository extends InMemoryCaseInstanceRepository` with `@Alternative @Priority(1) @ApplicationScoped`. This is the standard test harness — it auto-activates without `selected-alternatives`.

A consumer project adds `InMemoryCaseInstanceRepository` to `quarkus.arc.selected-alternatives` in test `application.properties`, thinking it's redundant but harmless.

### What was tried (didn't work)
- Checking event bus threading and Vert.x serialization — object identity was intact
- Verifying `currentPrincipal.tenancyId()` consistency — MockCurrentPrincipal returns a constant
- Adding more beans to `selected-alternatives` — made it worse

### Root cause
Quarkus Arc assigns a **default priority of 1** to alternatives listed in `quarkus.arc.selected-alternatives` that don't have `@Priority`. This creates a priority tie:

- `InMemoryCaseInstanceRepository` — priority 1 (from `selected-alternatives` default)
- `TestCaseInstanceRepository` — priority 1 (from `@Priority(1)`)

Both are `@ApplicationScoped`, so CDI creates **two separate bean instances**. Each has its own `ConcurrentHashMap<UUID, CaseInstance> store`. If CDI resolves different injection points to different beans (or resolves non-deterministically), saves go to one store and lookups to another.

### Fix
Do not add parent classes to `selected-alternatives` when the engine-testing module already provides `@Priority(1)` subclasses:

```properties
# WRONG — creates priority tie with TestCaseInstanceRepository
quarkus.arc.selected-alternatives=...,io.casehub.persistence.memory.InMemoryCaseInstanceRepository

# RIGHT — let engine-testing's @Priority(1) beans handle it
quarkus.arc.selected-alternatives=...
# (omit InMemoryCaseInstanceRepository, InMemoryCaseMetaModelRepository, InMemoryEventLogRepository)
```

The engine-testing module exists specifically to provide auto-activating alternatives for test contexts. Adding the parent classes to `selected-alternatives` defeats the purpose.

### Why this is non-obvious
Adding a class to `selected-alternatives` feels like an explicit, high-confidence activation — surely it would "win" over a `@Priority(1)` subclass. But the default priority is also 1, creating an exact tie. The Quarkus docs mention the default priority but don't warn about ties with existing `@Priority(1)` alternatives on the classpath. The symptom (state not found) appears as a threading or serialization bug, not a CDI resolution issue.
