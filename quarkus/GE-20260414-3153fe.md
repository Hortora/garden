## Symptom

Adding `quarkus-maven-plugin` with the `build` goal to a Quarkus extension runtime module to enable `@QuarkusIntegrationTest` fails at the package phase:

```
ConfigurationException: Datasource '<default>' is not configured.
To solve this, configure datasource '<default>'.
```

The build succeeds in the `test` phase (normal `@QuarkusTest` works fine) but fails in `package` when the plugin tries to produce a deployable artifact.

## Context

Quarkus extensions intentionally omit datasource configuration — consuming applications provide their own. The `quarkus-maven-plugin build` goal performs Quarkus augmentation, which validates that all required configuration is present. Since the extension's `application.properties` has no datasource, augmentation fails.

## What Was Tried

Adding `quarkus-maven-plugin` with build/generate-code/generate-code-tests goals directly to the extension runtime module's `pom.xml`.

## Root Cause

Quarkus augmentation (the build phase) requires a fully configured application. An extension module is not an application — it's a library. The augmentation validation for Hibernate ORM requires a datasource to be present.

## Fix

Create a separate `integration-tests` Maven module that:
1. Depends on the extension runtime (`quarkus-tarkus` etc.)
2. Provides its own `src/main/resources/application.properties` with datasource config (H2 for tests)
3. Contains the `quarkus-maven-plugin` build goal and `maven-failsafe-plugin`
4. Contains `*IT.java` tests annotated `@QuarkusIntegrationTest`

```xml
<!-- integration-tests/pom.xml -->
<dependencies>
  <dependency>
    <groupId>io.quarkiverse.myext</groupId>
    <artifactId>quarkus-myext</artifactId>
    <version>${project.version}</version>
  </dependency>
  <dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-jdbc-h2</artifactId>
  </dependency>
</dependencies>
```

```properties
# integration-tests/src/main/resources/application.properties
quarkus.datasource.db-kind=h2
quarkus.datasource.jdbc.url=jdbc:h2:mem:testdb;DB_CLOSE_DELAY=-1
quarkus.flyway.migrate-at-start=true
```

## Why This Is Non-Obvious

The `@QuarkusTest` tests in the extension runtime module work perfectly — they start a full Quarkus app including the extension in the same JVM without requiring a packaged artifact. So it seems like `@QuarkusIntegrationTest` should "just work" the same way by adding the plugin. The distinction between "test-phase app boot" and "package-phase augmentation" is not surfaced clearly in the error message.

### Why this fix

The separation keeps the extension module pure (no app-specific config) while enabling native image testing in the integration-tests module where full app configuration is appropriate.
