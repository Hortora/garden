## Symptom

Calling `WorkItemStore.scan()` with an `assigneeId` filter to count active WorkItems for a
specific actor returns more items than expected — items that are in the open pool (PENDING,
`assigneeId = null`) but list the actor in `candidateUsers` are incorrectly included.

This caused least-loaded routing (`LeastLoadedStrategy`) to bias against actors who appeared
in any pool's `candidateUsers` field, making them appear busier than they were.

## Context

When building `countActive(actorId)` using `WorkItemStore.scan(query)` with `.assigneeId(actorId)`,
the JPA store generates a WHERE clause with OR semantics:

```sql
WHERE assigneeId = :assigneeId
   OR candidateUsers LIKE :assigneeIdLike
```

The `LIKE` branch exists to support inbox queries ("show me everything I can claim"),
but it also fires when you intend exact-match-only assignment semantics.

## What was tried

Using `workItemStore.scan(WorkItemQuery.builder().assigneeId(actorId).statusIn(ACTIVE_STATUSES).build())`
appeared correct from the query API but produced inflated counts.

## Root cause

`WorkItemStore.scan()` is designed for inbox queries, which are broader: an item appears
in your inbox if you're the assignee OR if you're eligible to claim it via candidateGroups/candidateUsers.
There is no `exactAssigneeId` method that generates only `assigneeId = :x` without the LIKE branch.

## Fix

Post-filter the scan result in memory:

```java
private int countActive(final String actorId) {
    return workItemStore.scan(WorkItemQuery.builder()
            .assigneeId(actorId)
            .statusIn(List.of(WorkItemStatus.ASSIGNED,
                              WorkItemStatus.IN_PROGRESS,
                              WorkItemStatus.SUSPENDED))
            .build())
            .stream()
            .filter(wi -> actorId.equals(wi.assigneeId))  // exact match only
            .count();
}
```

## Why this is non-obvious

The method is named `assigneeId(actorId)` which strongly implies exact-match semantics.
The OR/LIKE behaviour is an implementation detail of the inbox-query model not surfaced
in the API. The bug only appears in routing scenarios where workload counts matter;
in normal inbox use the broader semantics are correct.

### Why this fix

The post-filter is cheap (in-memory, no extra DB round-trip) and makes intent explicit.
A `WorkItemQuery.Builder.exactAssigneeId()` method would be cleaner long-term but
requires changing the query API; the post-filter is the minimal correct fix.

### Alternatives considered

- Adding `exactAssigneeId(String)` to `WorkItemQuery.Builder` to emit only `assigneeId = :x` — cleaner but requires API change
- Querying Panache directly (`WorkItem.count("assigneeId = ?1 AND status IN ?2", ...)`) — bypasses the abstraction layer
