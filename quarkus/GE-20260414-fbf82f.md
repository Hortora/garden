## Context

Testing condition-based logic driven by a Quarkus `@Scheduled` job without timing dependencies. The naive approach is to set a very short `every` interval and use `Thread.sleep()` — this is fragile and slow.

## Technique

Extract the evaluation logic into a separate `@ApplicationScoped` service. The `@Scheduled` method becomes a thin delegator. Tests inject the service directly and call the evaluation method, bypassing the scheduler entirely.

```java
// Production scheduler — thin delegator
@ApplicationScoped
public class WatchdogScheduler {
    @Inject WatchdogEvaluationService evaluationService;

    @Scheduled(every = "${quarkus.qhorus.watchdog.check-interval-seconds:60}s")
    public void evaluate() {
        if (!config.watchdog().enabled()) return;
        evaluationService.evaluateAll();
    }
}

// Service with real logic — fully testable
@ApplicationScoped
public class WatchdogEvaluationService {
    @Transactional
    public void evaluateAll() { /* condition evaluation */ }
}

// Test — deterministic, no sleep
@QuarkusTest
class WatchdogTest {
    @Inject WatchdogEvaluationService watchdogService;

    @Test
    void conditionFiresAlert() {
        // set up condition...
        watchdogService.evaluateAll(); // call directly — no scheduler involved
        // verify alert...
    }
}
```

## Why this is non-obvious

Most developers either mock the scheduler (complex) or test with very short intervals (flaky). The split — scheduler as delegator, logic in a separate service — keeps both production and test code clean and avoids all timing dependencies. The `@Scheduled` method becomes so thin it doesn't need testing.

### Why this fix

Mocking the scheduler requires Quarkus-specific test extensions and still doesn't give you direct control over execution timing. The extraction pattern is simpler, faster, and makes the evaluation logic independently testable without any scheduler machinery.
