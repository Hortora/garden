## QuarkusTransaction.requiringNew() pattern for testing DB constraints that need independent commits

**ID:** GE-20260414-278875
**Stack:** Quarkus 3.32.2, Narayana JTA, Panache active record, JUnit 5

### What the technique does

Use `QuarkusTransaction.requiringNew().run(() -> ...)` lambdas to give each database operation its own committed transaction. This is the correct approach for testing unique-constraint violations, foreign-key constraints, and any behaviour that only manifests when a record is actually committed to the database.

### The problem it solves

`@TestTransaction` wraps the entire test in a single transaction that rolls back at the end. Within that transaction, Hibernate queues all INSERTs and only flushes when the transaction commits — which is after the test body. `assertThrows(PersistenceException.class, () -> entity.persist())` catches nothing because the constraint violation hasn't occurred yet.

The usual workaround — adding `Entity.flush()` inside the lambda — partially works but marks the persistence context invalid, causing the transaction to become rollback-only. Any code after `assertThrows` then fails unexpectedly.

### How to apply it

```java
@Test
// No @TestTransaction — each requiringNew() lambda manages its own
void duplicateChannelNameThrowsException() {
    String name = "dup-test-" + System.nanoTime(); // unique per test run

    // Step 1: First create — gets its own transaction, commits successfully
    QuarkusTransaction.requiringNew().run(() ->
            channelService.create(name, "First", ChannelSemantic.APPEND, null));

    // Step 2: Second create — its own transaction, fails at commit with constraint violation
    assertThrows(Exception.class, () ->
            QuarkusTransaction.requiringNew().run(() ->
                    channelService.create(name, "Second", ChannelSemantic.APPEND, null)));

    // Step 3: Cleanup the committed first record
    QuarkusTransaction.requiringNew().run(() -> Channel.delete("name", name));
}
```

Key details:
- Use `System.nanoTime()` or `UUID.randomUUID()` to avoid conflicts between test runs
- Wrap cleanup in `try/finally` if there's any risk the `assertThrows` itself could throw unexpectedly
- The cleanup step is needed because step 1 committed a real record; `@TestTransaction` rollback won't clean it up

### Why developers wouldn't naturally reach for this

- `QuarkusTransaction` is not prominently featured in Quarkus testing documentation; most examples only show `@TestTransaction` or `@Transactional`
- The failure mode from using `@TestTransaction` is "expected exception not thrown" — which looks like a missing constraint, not a flush-timing issue
- The `requiringNew()` API is a JTA concept that Java developers outside enterprise contexts rarely encounter
