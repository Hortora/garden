## Symptom

Application fails to start with:

```
SRCFG00014: The config property emulated.enemy.strategy-file is required but it could not be found in any config source
```

This happens even though `defaultValue = ""` is set on the `@ConfigProperty` annotation, which appears to provide a fallback.

## Context

When adding an optional file-path property to a Quarkus `@ApplicationScoped` bean — a property users may or may not set, with no meaningful default value:

```java
@ConfigProperty(name = "emulated.enemy.strategy-file", defaultValue = "")
String strategyFile;
```

The intent is: if the property is absent, use an empty string and skip the file-loading logic.

## Root Cause

MicroProfile Config treats an empty `defaultValue` string (`""`) as "no default provided" — it is the sentinel value meaning "required, no fallback". Quarkus inherits this behaviour. When the property is absent from all config sources and `defaultValue = ""`, the runtime treats it as a required property without a default and throws SRCFG00014.

## Fix

Use `Optional<String>` with no `defaultValue`:

```java
@ConfigProperty(name = "emulated.enemy.strategy-file")
Optional<String> strategyFile;
```

Then check presence in `@PostConstruct`:

```java
strategyFile.filter(f -> !f.isBlank()).ifPresent(path -> {
    // load from file
});
```

This is the idiomatic MicroProfile pattern for truly optional properties. `Optional.empty()` is returned when the property is absent; `Optional.of(value)` when present.

## Why Non-Obvious

`defaultValue = ""` looks like it provides a fallback. The MicroProfile Config spec's use of `""` as the "absent" sentinel is not prominently documented. The error message (`required but it could not be found`) gives no hint that the defaultValue is the problem — it reads as if no defaultValue was provided at all.
