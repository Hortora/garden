## Symptom

After a dependency updates and moves an inner class (e.g. `Outer$Inner` → `NewOuter$Inner`), `mvn clean test` in the consumer project still throws:

```
java.lang.ClassNotFoundException: com.example.Outer$Inner
```

`mvn clean` was expected to fix this — it does not.

## Context

Occurs when a Quarkus application depends on a SNAPSHOT library, the library moves or renames an inner class, and the library is rebuilt and installed locally. The consuming Quarkus project's `target/` directory is cleaned, but the test still fails.

## Root Cause

Quarkus generates a deployment descriptor (`ApplicationImpl`) at build time that encodes fully-qualified type signatures — including inner class names — for REST endpoints and other reflection-heavy features. This generated class is compiled into `target/` and references the old inner class name. `mvn clean` removes `target/` but the descriptor is regenerated from the stale compiled dependency bytecode cached in the local Maven repo.

The fix requires two steps: rebuild the provider artifact (`mvn install` in the provider project) AND clean-build the consumer.

## Fix

```bash
# In the provider project (the dependency that changed)
mvn install -DskipTests

# In the consumer project
mvn clean test
```

## Why This Is Non-Obvious

`mvn clean` is expected to eliminate all stale state. The Quarkus deployment descriptor is not obviously a cached artifact — it looks like generated source that should be regenerated on every build. The error message (`ClassNotFoundException` during `ApplicationImpl` static initializer) does not point to the build descriptor as the culprit.
