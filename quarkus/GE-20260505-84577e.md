### Symptom
Setting a field on an `@ApplicationScoped` CDI bean reference in a `@QuarkusTest` has no
effect on the bean's runtime state. The test code compiles and runs without error but the
field change is invisible to any method called on the proxy.

```java
// Looks correct — actually writes to the proxy object, not the bean
@Inject TestReconciliationSource reconciliationSource;
reconciliationSource.active = true;   // sets proxy.active, not bean.active
reconciliationSource.isActive();      // returns false — reads from bean.active
```

### Context
`@ApplicationScoped` beans in Quarkus Arc are subclass proxies. Method calls are delegated
to the underlying bean instance, but **field accesses bypass delegation** — they read/write
the field on the proxy class itself, which is a separate object from the actual bean instance.

### What was tried
Injecting the bean and setting fields directly as if the reference were the concrete instance.

### Root cause
CDI subclass proxies intercept method calls (via `@Override`) but cannot intercept field
accesses — Java does not support field access interception without bytecode instrumentation.
The proxy and the bean each hold their own copy of instance fields.

### Fix
Use `static` fields on inner CDI bean classes for any state that needs to be visible across
the CDI proxy boundary. `static` fields bypass the instance-level split entirely.

```java
@ApplicationScoped
static class TestReconciliationSource implements LedgerReconciliationSource {
    static boolean active = false;   // static — shared across proxy/bean split

    @Override
    public boolean isActive() { return active; }
}

// In test:
TestReconciliationSource.active = true;   // sets the static field — works
```

Alternatively, expose a setter method — method calls ARE properly delegated.

### Why this is non-obvious
CDI proxies look exactly like the bean to the compiler. Setting `bean.field = value` compiles
without warning and the field appears to be "set" — but you're writing to the proxy's
own field, which is never read by any delegated method call. The underlying bean instance
remains at its initialised default.

The same `@ApplicationScoped` bean works perfectly for method calls — only field writes are
broken. This makes the failure mode extremely hard to spot: methods that read `this.field`
return the bean's value; direct field access from outside returns the proxy's value.
