## Add a non-@Tool overload to evolve an MCP @Tool method signature without breaking test call sites

**ID:** GE-20260414-937013
**Stack:** quarkus-mcp-server 1.11.1, Quarkus 3.32.2

### What the technique does

When adding a new optional parameter to an existing `@Tool`-annotated method, keep the `@Tool` annotation on the new N+1-arg version and add a non-annotated N-arg overload that delegates with a null value for the new parameter. MCP clients see only the annotated version via the tool schema; all existing Java test call sites compile and run unchanged.

### The problem it solves

In `@QuarkusTest` tests, tools are called directly as Java methods (not via the MCP protocol). When you add a parameter to a `@Tool` method, every test that calls it with the old signature fails to compile — potentially dozens of call sites across many test files.

Fixing all call sites mechanically is correct but noisy. Adding the overload is a one-line change that keeps the diff clean.

### How to apply it

```java
// Before: 6-arg @Tool
@Tool(name = "send_message", description = "...")
public MessageResult sendMessage(
    @ToolArg(name = "channel_name") String channelName,
    @ToolArg(name = "sender") String sender,
    @ToolArg(name = "type") String type,
    @ToolArg(name = "content") String content,
    @ToolArg(name = "correlation_id", required = false) String correlationId,
    @ToolArg(name = "in_reply_to", required = false) Long inReplyTo) {
    // ...
}

// After: add new parameter, keep @Tool on the 7-arg version only
/** Backwards-compatible overload — null artefactRefs. Used by tests and internal callers. */
public MessageResult sendMessage(String channelName, String sender, String type,
        String content, String correlationId, Long inReplyTo) {
    return sendMessage(channelName, sender, type, content, correlationId, inReplyTo, null);
}

@Tool(name = "send_message", description = "...")
@Transactional
public MessageResult sendMessage(
    @ToolArg(name = "channel_name") String channelName,
    @ToolArg(name = "sender") String sender,
    @ToolArg(name = "type") String type,
    @ToolArg(name = "content") String content,
    @ToolArg(name = "correlation_id", required = false) String correlationId,
    @ToolArg(name = "in_reply_to", required = false) Long inReplyTo,
    @ToolArg(name = "artefact_refs", required = false) List<String> artefactRefs) {
    // full implementation
}
```

**Key rules:**
- The `@Tool` annotation goes on the NEW (longer) version — this is the MCP tool
- The old (shorter) overload has no `@Tool` — invisible to the MCP schema
- The overload passes `null` for the new parameter (relying on null handling in the implementation)
- Javadoc the overload so it's clear it's a backwards-compat shim

### Why developers wouldn't naturally reach for this

The obvious fix is to update all call sites to pass `null` as the new parameter. This is correct but mechanical — especially with 30+ call sites across multiple test files. The overload pattern is idiomatic Java but developers don't usually think of it as a tool-schema evolution strategy. In the MCP context, the key insight is that MCP clients and Java tests are separate consumers with different calling conventions.
