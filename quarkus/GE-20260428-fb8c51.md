## CAST(date_trunc('day', field) AS LocalDate) in Hibernate 6 HQL forces a portable LocalDate return type

### Technique

When using `date_trunc` in Hibernate 6 HQL for GROUP BY aggregation, wrap it with
`CAST(... AS LocalDate)` to explicitly force the return type to `java.time.LocalDate`.
Without the cast, the return type is dialect-dependent — H2 and PostgreSQL can return
different Java types, causing `ClassCastException` in the application layer.

```java
// Without cast — return type is dialect-dependent (LocalDateTime, Instant, OffsetDateTime)
"SELECT date_trunc('day', w.createdAt), COUNT(w) FROM WorkItem w " +
"WHERE w.createdAt >= :from AND w.createdAt <= :to " +
"GROUP BY date_trunc('day', w.createdAt) ORDER BY date_trunc('day', w.createdAt)"

// With cast — always returns LocalDate regardless of dialect
"SELECT CAST(date_trunc('day', w.createdAt) AS LocalDate), COUNT(w) FROM WorkItem w " +
"WHERE w.createdAt >= :from AND w.createdAt <= :to " +
"GROUP BY CAST(date_trunc('day', w.createdAt) AS LocalDate) " +
"ORDER BY CAST(date_trunc('day', w.createdAt) AS LocalDate)"
```

The CAST targets Hibernate's Java type system (not SQL CAST) — `LocalDate` refers to
`java.time.LocalDate`. Hibernate 6 handles the dialect-specific DDL transparently.
This enables portable time-bucketing for throughput reports across H2 (tests) and
PostgreSQL (production) without any dialect-specific code.

### Why this is non-obvious

`date_trunc` is documented in Hibernate 6 as a supported HQL function, but the return
type behaviour is not documented. Developers naturally write the GROUP BY without the CAST
and only discover the ambiguity when the dialect changes (e.g., switching from H2 to
PostgreSQL in a Testcontainers test). The `CAST AS LocalDate` syntax targeting Java types
rather than SQL types is a Hibernate 6 HQL feature that has no equivalent in JPA JPQL.
