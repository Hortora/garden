## Symptom

Adding `casehub-testing` (which transitively brings `casehub-engine`) to a Quarkus test application's `quarkus.index-dependency` causes ALL `@QuarkusTest` classes in the module to fail with:

```
[ERROR] ClassSelector [className = 'dev.claudony.SmokeTest', classLoader = null] resolution failed
Cause: Could not load class with name: dev.claudony.SmokeTest
```

No compilation errors. Surefire reports 0 tests across the entire app module. The error message points to `SmokeTest` (a completely unrelated class), not to the actual CDI conflict.

## Root Cause

`casehub-engine` ships 8 no-op fallback implementations of its own SPI interfaces — one sync and one reactive variant per SPI:

- `NoOpWorkerProvisioner` / `NoOpReactiveWorkerProvisioner` → `WorkerProvisioner`
- `EmptyWorkerContextProvider` / `EmptyReactiveWorkerContextProvider` → `WorkerContextProvider`
- `NoOpCaseChannelProvider` / `NoOpReactiveCaseChannelProvider` → `CaseChannelProvider`
- `NoOpWorkerStatusListener` / `NoOpReactiveWorkerStatusListener` → `WorkerStatusListener`

All 8 are annotated `@ApplicationScoped` — the same priority as a consumer's own implementations. When both sets are on the CDI index, ArC cannot resolve which bean to inject and throws `AmbiguousResolutionException` for all 9 injection points (some SPIs are injected multiple times). Quarkus deployment failure causes the test JVM to never initialise the Quarkus classloader, which is why every test class fails to load with `classLoader = null`.

Correct design: engine no-ops should be `@DefaultBean` (like Claudony's own `EmptyCaseLineageQuery`), which CDI automatically overrides with any `@ApplicationScoped` alternative.

## Diagnosis

Enable CDI debug logging to surface the real error:

```bash
mvn test -Dquarkus.log.category."io.quarkus.arc".level=DEBUG \
  2>&1 | grep "Ambiguous\|Unsatisfied\|DeploymentException"
```

Output will show exactly which SPI interfaces have ambiguous beans.

## Fix

Consumer-side: exclude the engine no-ops from CDI scanning using `quarkus.arc.exclude-types`. Must list each class individually (wildcard `io.casehub.engine.internal.worker.*` excludes too much and breaks internal engine beans like `WorkerExecutionManager`):

```properties
quarkus.arc.exclude-types=\
  io.casehub.engine.internal.worker.NoOpWorkerStatusListener,\
  io.casehub.engine.internal.worker.NoOpReactiveWorkerStatusListener,\
  io.casehub.engine.internal.worker.EmptyWorkerContextProvider,\
  io.casehub.engine.internal.worker.EmptyReactiveWorkerContextProvider,\
  io.casehub.engine.internal.worker.NoOpWorkerProvisioner,\
  io.casehub.engine.internal.worker.NoOpReactiveWorkerProvisioner,\
  io.casehub.engine.internal.worker.NoOpCaseChannelProvider,\
  io.casehub.engine.internal.worker.NoOpReactiveCaseChannelProvider
```

Upstream fix (casehub-engine): change all 8 no-op beans from `@ApplicationScoped` to `@DefaultBean`.

## Why This Is Non-Obvious

The symptom (`SmokeTest classLoader=null`) points nowhere near the cause. There is no "CDI deployment failed" message in the standard test output — the entire Quarkus test infrastructure silently fails to initialise. The actual `DeploymentException` is only visible with CDI debug logging. The wildcard exclusion (`io.casehub.engine.internal.worker.*`) appears to fix the ambiguity but breaks the engine's own internal beans, producing a new wave of `UnsatisfiedDependencyException` errors.

### Why this fix
Surgical per-class exclusion (not wildcard) is required because the engine's internal beans (`WorkerExecutionManager`, `WorkloadProvider`) are in the same package and must remain discoverable.

### Alternatives considered
- Wildcard exclusion `io.casehub.engine.internal.worker.*` — removed too much, broke `WorkerExecutionManager` and `WorkloadProvider`
- `@TestProfile` with `quarkus.index-dependency.casehub-testing` in `getConfigOverrides()` — Quarkus CDI augmentation happens once per app instance, not per profile restart, so the ambiguity still occurs before any profile is applied
- Not indexing casehub-testing at all — means `TestCaseInstanceRepository` and `WorkResultSubmitter` CDI beans aren't discovered; test can't inject them
