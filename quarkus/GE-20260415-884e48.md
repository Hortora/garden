## @Alternative @Priority(n) in CDI 4.0/Quarkus globally activates the alternative — causes AmbiguousResolutionException

### Stack
Quarkus 3.31.1, CDI 4.0

### Symptom
Multiple beans implement the same SPI interface. Each alternative is annotated with
`@Alternative @Priority(10)` as well as `@ApplicationScoped`. On Quarkus startup, the
application fails with `AmbiguousResolutionException` — all implementations are active
simultaneously.

```
AmbiguousResolutionException: Ambiguous dependencies for type ContextDiffStrategy
and qualifiers [@Default]: [TopLevelContextDiffStrategy, JsonPatchContextDiffStrategy, 
NoOpContextDiffStrategy]
```

### Context
SPI pattern where one default bean and two `@Alternative` beans implement an interface.
The intent was to activate one alternative at a time via
`quarkus.arc.selected-alternatives` in `application.properties`.

### What was tried
Adding `@Priority(10)` alongside `@Alternative` — expected this to be a ranking hint
that would help CDI resolve ambiguity. Instead, `@Priority` on an `@Alternative` is what
globally activates the bean in CDI 4.0, making all three implementations simultaneously
active and causing the ambiguous resolution error.

### Root cause
In CDI 4.0 (Jakarta EE 10), `@Alternative @Priority(n)` is a shorthand for globally
activating an alternative across the entire application — no `beans.xml` or
`selected-alternatives` config needed. When ALL alternatives carry this annotation,
they are ALL active, which breaks injection points expecting a single bean.

### Solution 1 — Remove @Priority, use selected-alternatives config
**Approach:** Strip `@Priority` from all alternatives; activate one via config.
**Pros:** Per-profile/per-module control without code changes; the intended Quarkus mechanism for SPI switching.
**Cons:** Requires a config entry per activation; all alternatives default to inactive.

Remove `@Priority` from all `@Alternative` beans. Activate a specific alternative only
via `quarkus.arc.selected-alternatives` in `application.properties`:

```java
// WRONG — @Priority globally activates the alternative in CDI 4.0
@Alternative
@Priority(10)
@ApplicationScoped
public class MyAlternativeImpl implements MyInterface { ... }

// CORRECT — @Alternative only; activation controlled by config
@Alternative
@ApplicationScoped
public class MyAlternativeImpl implements MyInterface { ... }
```

```properties
# application.properties — activate the desired alternative
quarkus.arc.selected-alternatives=com.example.MyAlternativeImpl
```

### Solution 2 — Use different @Priority values to establish precedence
**Approach:** Keep `@Alternative @Priority` but assign distinct priority values; CDI resolves to the highest.
**Pros:** No config entry needed; the default (lowest priority) is always active, the override (highest priority) wins when present on the classpath. Works naturally for optional-module patterns where a higher-capability implementation should auto-replace the baseline.
**Cons:** All alternatives are always globally active (the core CDI 4.0 behaviour). The "default" must have the lowest priority; adding a third implementation requires careful priority assignment.

```java
// Default — always active; lowest priority loses when a higher one is present
@Alternative
@Priority(0)
@ApplicationScoped
public class ClaimFirstStrategy implements WorkerSelectionStrategy { ... }

// Override — higher priority wins; active only when on the classpath
@Alternative
@Priority(1)
@ApplicationScoped
public class SemanticWorkerSelectionStrategy implements WorkerSelectionStrategy { ... }
```

CDI injects the `@Priority(1)` bean when both are present; falls back to `@Priority(0)` when only
the default module is on the classpath. No `beans.xml` or `selected-alternatives` required.

**When to choose Solution 2 over Solution 1:** When the override is an optional module (not always
on the classpath) and the default must always be present. Config-based activation (Solution 1)
requires an explicit entry even for the default case; priority-based resolution (Solution 2) makes
the default zero-configuration.

### Why this is non-obvious
The `@Priority` annotation is commonly used alongside `@Alternative` in CDI tutorials
and examples from the CDI 2.x era, where `@Priority` on an alternative was a *ranking*
hint. In CDI 4.0 the semantics changed — `@Priority` on an `@Alternative` is now an
*activation* mechanism (equivalent to listing it in `beans.xml` with a priority). The
Quarkus docs for `quarkus.arc.selected-alternatives` don't warn about this interaction.

### Why this fix
`quarkus.arc.selected-alternatives` gives per-profile/per-module control via config,
which is the intended Quarkus mechanism for SPI switching (same as how `PanningStrategyLoopControl`
and other engine SPIs are activated). It doesn't require code changes to switch backends.

### Update — Test-refactoring trap: two classes declaring the same @Alternative

During a test refactoring, if two `@QuarkusTest` classes *simultaneously* declare an
`@Alternative @Priority(2)` inner class implementing the same type, Quarkus boot fails
with `AmbiguousResolutionException` even though `mvn test-compile` passes cleanly. The
compile succeeds because both classes are syntactically valid; the CDI conflict only
surfaces at application startup.

**Scenario:** Moving a test double from `ClassA` (where it was global via
`selected-alternatives`) to `ClassB` (where it should be scoped via
`QuarkusTestProfile.getEnabledAlternatives()`). During the intermediate state where
both classes exist, both `@Priority(2)` alternatives are active, causing boot failure.

**Fix:** Complete the move atomically — remove from `ClassA` and add to `ClassB` in the
same commit, or keep only one class active at any given time.

**Test-scoped activation:** When a test double should activate only for one `@QuarkusTest`
class, use `QuarkusTestProfile.getEnabledAlternatives()` instead of global
`selected-alternatives`. This avoids the two-class trap entirely. See technique entry
[GE-20260519 QuarkusTestProfile.getEnabledAlternatives() for per-class CDI scoping].

### Update — @Priority defeats QuarkusTestProfile.getEnabledAlternatives() scoping

When using `@QuarkusTestProfile.getEnabledAlternatives()` to scope an `@Alternative`
to a single `@QuarkusTest` class, `@Priority` on the bean causes it to be globally
activated regardless of the profile — every other `@QuarkusTest` class in the same
test run picks it up. The symptom is NOT `AmbiguousResolutionException` but rather
the **wrong bean silently returning in an unrelated test class**, making an assertion
fail with a valid-looking but unexpected value.

**Scenario:** `IrbCommitteePolicySpiTest` declared `@Alternative @ApplicationScoped @Priority(1) TestIrbCommitteeAssignmentPolicy` and activated it via
`getEnabledAlternatives()`. `IrbGateLifecycleTest` (no profile, expects the `@DefaultBean`
to return `"irb-committee"`) instead saw `"test-irb-committee-xyz"` — the test alternative.

**Why:** `@Priority(1)` globally activates the bean in CDI 4.0 regardless of which profile
is active. `getEnabledAlternatives()` is only additive — it adds to the global set, it
does NOT limit scope to that profile only.

**Fix:** Remove `@Priority` from the test alternative. `@Alternative @ApplicationScoped`
alone is sufficient; `getEnabledAlternatives()` handles the selection:

```java
// WRONG — @Priority makes it global, defeating profile scoping
@Alternative
@ApplicationScoped
@Priority(1)
static class TestPolicy implements MyPolicy { ... }

// CORRECT — profile selection handles activation; @Priority not needed
@Alternative
@ApplicationScoped
static class TestPolicy implements MyPolicy { ... }

public static class TestProfile implements QuarkusTestProfile {
    @Override
    public Set<Class<?>> getEnabledAlternatives() {
        return Set.of(TestPolicy.class);
    }
}
```
