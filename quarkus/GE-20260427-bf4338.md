## Context

Implementing an adapter that bridges quarkus-work `WorkItemLifecycleEvent`s to CaseHub `PlanItem` transitions. The issue specification lists EXPIRED as a terminal status (should map to FAULTED). A natural guard is `if (!event.status().isTerminal()) return;` to skip non-terminal transitions.

## Symptom

The adapter silently ignores `EXPIRED` events — `PlanItem` never transitions to `FAULTED` when a WorkItem expires, and tests time out waiting for the status change.

## Root Cause

`WorkItemStatus.isTerminal()` uses a tableswitch on ordinal values:

| Ordinal | Status | isTerminal() |
|---------|--------|-------------|
| 3 | COMPLETED | true |
| 4 | REJECTED | true |
| 5 | DELEGATED | **false** |
| 6 | SUSPENDED | **false** |
| 7 | CANCELLED | true |
| 8 | EXPIRED | **false** |
| 9 | ESCALATED | true |

EXPIRED (ordinal 8) falls into the `false` branch. Only discoverable by decompiling with `javap -p -c`.

## Fix

Do not rely on `isTerminal()` when you need to handle EXPIRED. Instead, enumerate the specific statuses you care about:

```java
WorkItemStatus status = event.status();
if (status != WorkItemStatus.COMPLETED
    && status != WorkItemStatus.REJECTED
    && status != WorkItemStatus.CANCELLED
    && status != WorkItemStatus.EXPIRED
    && status != WorkItemStatus.ESCALATED) return;
```

## Why This Is Non-Obvious

The method is named `isTerminal()` and EXPIRED is semantically a terminal state for any workflow. There is no documentation listing which statuses `isTerminal()` includes or excludes. The only way to discover the exclusion is bytecode inspection.
