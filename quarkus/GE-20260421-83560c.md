## @WithTransaction on Quarkus Hibernate Reactive store methods uses REQUIRED propagation — joins outer Panache.withTransaction(), does not create nested

### Symptom

Code reviewer flags a "critical double-transaction" issue: a reactive service calls `Panache.withTransaction(() -> store.put(entity))` where `store.put()` is annotated `@WithTransaction`. The concern is that two nested transactions will be created, causing undefined behaviour.

### Behaviour

`@WithTransaction` uses REQUIRED propagation semantics — if a transaction is already active on the current Vert.x context, the annotated method **joins** the existing transaction rather than starting a new one. Calling a `@WithTransaction` method from inside `Panache.withTransaction()` is safe and correct.

### Context

This was discovered when implementing `ReactiveJpaChannelStore.put()` with `@WithTransaction` and calling it from `ReactiveChannelService.create()` which uses `Panache.withTransaction()`. A senior code reviewer flagged this as "known footgun" based on experience with nested reactive transactions in other frameworks.

### Root cause

The reviewer's assumption was based on other reactive transaction frameworks where nested calls always create savepoints or nested transactions. In Quarkus Hibernate Reactive, `@WithTransaction` defaults to REQUIRED: check for existing session context, join if present, create new if absent.

### Why this is non-obvious / undocumented

The Quarkus docs describe `@WithTransaction` as "equivalent to `Panache.withTransaction()` but as an interceptor" without explicitly stating the propagation behaviour when called from inside an existing `Panache.withTransaction()` block. The REQUIRED semantics must be inferred from the Vert.x/Mutiny context propagation model. There is no clear warning that this is safe (which would prevent false alarm code reviews) or that it is unsafe (which would be an accurate warning).

The dual-stack design (service layer using `Panache.withTransaction()` + store layer using `@WithTransaction`) is specifically designed to work correctly: the store's `@WithTransaction` is a fallback for when the store is called directly outside a service transaction; it joins and becomes a no-op when called from inside a service-level transaction.
