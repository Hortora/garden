## Symptom

An HTTP endpoint that should respond in <50ms takes 1.5–3 seconds in the browser. Rapid successive calls are fast (50ms). Calls with a 5+ second gap between them are consistently slow again.

## Context

A visualizer panel fetched `/qa/unit/{tag}` from the server on every mouse click to display unit stats. In the browser, clicking once per second felt like a 1.5s delay before the panel appeared. Running `curl` in a loop confirmed: first call ~1.3s, second call ~165ms, third ~57ms, fourth ~52ms. With a 5-second gap between calls: back to ~1.5s.

## Root cause

Quarkus dev-mode runs a filesystem scanner before processing each HTTP request to detect source file changes and trigger live-reload. After a period of inactivity, the scanner re-scans the entire source tree on the next request. The result is cached briefly, so rapid successive calls skip the scan. This is fundamental to dev-mode's live-reload experience — not a bug, but it creates misleading "slow endpoint" symptoms.

## Fix

Move data that's already available client-side out of the HTTP round-trip. In this case, the unit data was already present in the browser via the WebSocket game-state broadcast (`lastState`). Reading from `lastState` locally makes the panel open instantly:

```javascript
function showUnitPanel(tag, isEnemy) {
  if (lastState) {
    const units = isEnemy ? (lastState.enemyUnits || []) : (lastState.myUnits || []);
    const u = units.find(u => u.tag === tag);
    if (u) { _populateUnitPanel(u, isEnemy); panel.classList.add('visible'); return; }
  }
  // Fallback: HTTP fetch (cold start or stale state)
  fetch(`/qa/unit/${encodeURIComponent(tag)}`).then(...);
}
```

If the HTTP endpoint is genuinely necessary (not redundant with a WebSocket cache), the scan overhead can be partially mitigated by keeping requests warm, but the fundamental delay can't be eliminated in dev mode.

## Why this is non-obvious

The latency is inconsistent — fast when requests arrive in quick succession, slow after pauses. This looks exactly like network jitter, connection pooling, or JIT warmup rather than a dev-mode scanner. Profiling shows the delay is fully inside Quarkus's request handling, before any application code runs. The fix is architectural (avoid the round-trip) rather than a configuration tweak.
