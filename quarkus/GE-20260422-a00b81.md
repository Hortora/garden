## Symptom

After editing a Java source file, `curl` to the endpoint still returns the old response.
The Maven process is running. `curl http://localhost:8080/q/health/ready` returns `{"status":"UP"}`.
`touch` on the file produces no effect. Multiple edits, waits of 10–30 seconds — still old code.

## Context

Occurs during long `mvn quarkus:dev` sessions, particularly after a series of rapid edits
or after a background Maven task was started with `run_in_background: true` and the tool
reported the task as "completed (exit code 0)" — which actually means the Maven launcher
exited after handing off to the Quarkus dev server, not that the server died.

## Root Cause

The Quarkus file watcher (internal to the dev server JVM) appears to stop detecting changes
in the working directory. The root cause is not fully diagnosed — may relate to inotify/
FSEvents limits on macOS, or to the watcher thread silently dying after certain error
conditions. The health endpoint reports UP because the HTTP server is still serving; it does
not reflect whether hot-reload is functioning.

## Fix

Kill and restart the Maven process:

```bash
pkill -f "quarkus:dev" && sleep 2
mvn quarkus:dev -Dquarkus.profile=<profile> -q &
# wait ~20s for startup
curl http://localhost:8080/q/health/ready
```

Verify the fix actually loaded new code by checking a response field that was changed in
the edit — do not rely on the health endpoint alone.

## Why This Is Non-Obvious

The health endpoint returns UP throughout, giving a false signal of a healthy server.
There are no log messages indicating the file watcher has stopped. The symptom (code
changes not taking effect) is identical to "I forgot to save the file" or "I edited the
wrong file". The only reliable diagnostic is checking that the endpoint response actually
reflects your edit.
