## quarkus-mcp-server silently caps tools/list at 50 — tools beyond alphabetical position 50 simply don't appear

### Stack
quarkus-mcp-server 1.11.1 (Quarkiverse)

### Symptom
A `@Tool`-annotated method exists in the codebase, the app starts without error, but the tool never appears in the MCP `tools/list` response. The method is annotated correctly, the CDI bean is discovered, and compilation succeeds. No warning in logs. Clients (e.g. Claude) cannot invoke the tool as if it doesn't exist.

### Context
When multiple CDI beans contribute `@Tool` methods, `quarkus-mcp-server` paginates `tools/list` at a default page size of 50. Tools are sorted alphabetically by name before pagination. Any tool at alphabetical position 51+ is silently dropped from the first (and only) page returned when no cursor is provided.

In this case: 49 Qhorus tools + 8 Claudony tools = 57 total. `send_input` sits at alphabetical position 51, and 6 other tools beyond it were also silently absent. The total count in the response was exactly 50 — matching the cap perfectly, but offering no signal that truncation occurred.

### What was tried
- Verified `@Tool` annotation was present and correctly formatted
- Checked CDI bean scope — bean was correctly `@ApplicationScoped`
- Verified Jandex index included the method
- Checked for naming conflicts with other beans
- Decompiled `McpServerProcessor.class` looking for a hardcoded 50 limit in the deployment processor — not found there
- Inspected Qhorus jandex.idx with `jar -xf` + `cat | strings | grep` to count @Tool names — found 49 Qhorus tools, confirmed total would be 57
- Cross-referenced actual 50-tool response list against alphabetically sorted expected list — tools 51-57 (starting with `send_*`) confirmed absent

### Root cause
`ToolMessageHandler` in quarkus-mcp-server reads `serverConfig.tools().pageSize()` which defaults to 50 via `@WithDefault("50")`. The `tools/list` handler calls `manager.fetchPage(...)` and returns only the first page. Clients that don't provide a cursor receive at most 50 tools with no indication that more exist.

### Fix

In `application.properties`:
```properties
# Disable tools/list pagination — returns all tools in one response.
# Default cap is 50; remove if total tool count exceeds client context limits.
quarkus.mcp.server.tools.page-size=0
```

Setting page-size to 0 or negative disables pagination entirely. All tools are returned in a single `tools/list` response.

### Why this is non-obvious
There is no log warning when tools are truncated. The error looks like a code problem (annotation missing, CDI not discovering the bean) when it is actually a configuration problem with a non-obvious default. The tool count in the response matches the page size exactly — a developer might not notice the discrepancy unless they independently count expected vs actual tools.

### Why this fix
Setting `page-size=0` is preferable to increasing to a larger number because: (a) the number of tools will grow over time and any fixed number will eventually be exceeded again, (b) pagination support in MCP clients is inconsistent — many clients fetch only the first page. The correct long-term fix is separate MCP endpoints per library so each stays well within limits.

### Alternatives considered
- Increase page-size to a large number (e.g. 500) — brittle, will break again as tool count grows
- Implement cursor-based pagination on the client — requires all MCP clients to support it; Claude does not handle cursors in `tools/list` reliably
- Split tools across separate MCP endpoints (separate Claudony and Qhorus endpoints) — correct long-term architecture, deferred
