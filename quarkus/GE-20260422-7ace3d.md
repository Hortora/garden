## Stale Maven JAR after branch merge causes NoSuchMethodError in @QuarkusTest

### Symptom

After merging a feature branch that added methods to an `api/` module into local `main`, `@QuarkusTest` integration tests throw `java.lang.NoSuchMethodError` for the newly added method (e.g. `Binding.getConflictResolverStrategy()`). The error appears inside Quarkus CDI proxy generated code, making it look like a code generation or reflection problem.

### Context

Occurs in multi-module Maven projects where:
1. A feature branch adds methods to a shared `api/` module
2. The branch is merged into local `main` via fast-forward
3. Tests are run directly with `mvn test -pl <module>` without rebuilding `api/`

The local `~/.m2` repository still contains the JAR compiled from before the merge. Quarkus's classloader picks up that stale JAR rather than recompiling from source.

### What was tried (didn't work)

Running `mvn test -pl casehub-resilience -Ppersistence-memory` directly after merging. The test failure stacktrace pointed to Quarkus CDI proxy code, suggesting a CDI or reflection issue — not a stale artifact.

### Root cause

Maven resolves `api` as a dependency from the local repository (`~/.m2`), not from the source tree, unless explicitly told to build it first. After a branch merge that modified `api/`, the source is updated but the installed JAR is not. Quarkus loads the old JAR, which lacks the new method.

### Fix

After merging a feature branch into a module-under-test's transitive dependency chain, rebuild and install the affected upstream modules before running tests:

```bash
mvn install -DskipTests -pl api,engine,casehub-resilience -am -q
```

The `-am` flag builds all modules that the listed modules depend on. Then run tests normally.

### Why this is non-obvious

The error message (`NoSuchMethodError` in a CDI proxy class) looks exactly like a Quarkus code generation bug or a reflection/GraalVM issue — not a classpath staleness problem. The fix is at the Maven layer, not the Quarkus layer. A developer unfamiliar with Maven's local repository resolution model will spend time inspecting CDI configuration rather than checking JAR freshness.
