## Quarkus @TestProfile and QuarkusTestResource config overrides are NOT visible to the augmentation cache decision

### What exists

`@TestProfile.getConfigOverrides()` and `QuarkusTestResourceLifecycleManager.start()` both
allow returning a `Map<String, String>` of config overrides. The documentation says these
overrides are "applied before Quarkus starts". This is true for the **runtime phase** — but
not for the **augmentation cache check**.

### The undocumented behaviour

Before starting the Quarkus application for a test class, the test framework checks whether
the existing augmented artifact in `target/quarkus-app/` is still valid for the current
configuration. This check compares a hash of build-time properties.

**The hash is computed from:**
- JVM system properties
- `application.properties` files on the classpath
- Other standard MicroProfile Config sources available at JVM startup

**The hash is NOT computed from:**
- Config returned by `@TestProfile.getConfigOverrides()`
- Config returned by `QuarkusTestResourceLifecycleManager.start()`

These sources are loaded **after** the augmentation cache decision has been made.
As a result, any build-time property (e.g. `quarkus.datasource.db-kind`,
`quarkus.native.enabled`) set only via test profile or test resource will be silently
ignored for the purposes of augmentation, and the cached artifact — built for the original
config — will be reused.

### Practical implication

To trigger re-augmentation for a different build-time config in a test, the property must
be a JVM system property, set before the test framework runs — e.g. via Surefire
`<systemPropertyVariables>`:

```xml
<systemPropertyVariables>
  <quarkus.datasource.db-kind>postgresql</quarkus.datasource.db-kind>
</systemPropertyVariables>
```

This is not documented in the `@TestProfile`, `@QuarkusTestResource`, or augmentation
documentation. The only hint is the warning message "build time fixed to 'X'" that appears
when an incompatible override is detected at runtime.

### Why this is undocumented

The Quarkus test documentation describes `@TestProfile` and `QuarkusTestResource` as
"applying config before Quarkus starts" without distinguishing between the two phases
(cache check vs application start). The augmentation cache mechanism itself is an
implementation detail with no public documentation.
