## Symptom

After adding `@WrapBusinessError({IllegalArgumentException.class, IllegalStateException.class})` to an `@ApplicationScoped` MCP tools bean, other CDI beans that inject and call the tools bean's `@Tool`-annotated methods also receive `ToolCallException` instead of the original exception type.

```
catch (IllegalArgumentException e) {  // ← this block is never reached
    return error400(e.getMessage());   //    ToolCallException arrives instead
}
```

## Context

A REST resource (`A2AResource`) injected `QhorusMcpTools` and called `tools.sendMessage(...)`, expecting `IllegalArgumentException` when the channel doesn't exist. After adding `@WrapBusinessError` to the class, the REST resource started receiving `ToolCallException` wrapping the original exception, causing HTTP 500 responses instead of the intended HTTP 400.

## Root cause

`@WrapBusinessError` is a CDI interceptor binding. It fires on every CDI proxy invocation — regardless of whether the caller is the MCP server or any other CDI bean. The interceptor checks `context.getMethod().isAnnotationPresent(Tool.class)` before wrapping (see Undocumented entry GE-20260417-01b5d5), so only `@Tool`-annotated methods are wrapped — but *all* callers of those methods receive `ToolCallException`.

## Fix

Any CDI bean that calls a `@Tool`-annotated method on a `@WrapBusinessError`-annotated class must handle `ToolCallException` alongside the original exception:

```java
try {
    tools.sendMessage(channelName, sender, type, content, ...);
} catch (IllegalArgumentException | ToolCallException e) {
    // ToolCallException wraps IAE when coming through the CDI proxy
    String msg = e instanceof ToolCallException && e.getCause() != null
        ? e.getCause().getMessage()
        : e.getMessage();
    return error400(msg);
}
```

## Why this is non-obvious

The `@WrapBusinessError` documentation focuses on the MCP server use case. There is no mention that CDI callers outside the MCP server are also affected. The symptom (HTTP 500 instead of 400) looks like a generic server error and gives no hint about the interceptor involvement.
