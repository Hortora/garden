### Symptom

`NullPointerException` from inside the base JPA repository class during a test that was passing before a new method overload was added:

```
java.lang.NullPointerException: Cannot invoke "jakarta.persistence.EntityManager.createQuery(...)"
    because "this.em" is null
    at MessageLedgerEntryRepository.listEntries(MessageLedgerEntryRepository.java:73)
    at CapturingRepo.listEntries(...)  ← not shown — because there is no override
```

The test previously passed. The only change was adding a new overloaded variant of an existing method to the base class.

### Context

The "CapturingRepo" pattern: a test class extends a JPA `@Repository` class and overrides specific methods with in-memory implementations. The JPA base class has an injected `EntityManager em` (null in pure unit tests — no CDI/Quarkus runtime). The test stub overrides the methods it uses and works because those overrides never reach `em`.

When a new overloaded method is added to the base class (e.g. `listEntries` gains a new signature with two extra parameters), the existing test stub does **not** override the new variant. Java's method resolution dispatches the call to the base class, which hits `em` → NPE.

### What was tried

The test stub already overrode the 6-argument `listEntries`. The test compiled and ran. Only after adding a new 8-argument `listEntries` overload to the production class did tests start NPE-ing — and only on the tests that called the new 8-arg variant. The 6-arg tests continued to pass, which masked the fact that a whole class of calls was hitting the base impl.

### Root cause

Java method overloading is resolved at compile time by argument count and type. The test stub's `@Override` for the 6-arg variant does not cover the 8-arg variant — they are different methods. The compiler and IDE give no warning because the 8-arg variant *is* a valid method inherited from the superclass; the stub simply didn't add an explicit override for it.

### Fix

When adding a new overloaded method to a JPA repository class that is extended by a test stub, **immediately add the corresponding override to the stub**:

```java
// In MessageLedgerEntryRepository.java — new overload added:
public List<MessageLedgerEntry> listEntries(UUID channelId, Set<String> types,
        Long afterSeq, String agentId, Instant since,
        String correlationId, boolean sortDesc, int limit) {
    // JPA impl — uses em
}

// In CapturingRepo (test stub) — must also add override:
@Override
public List<MessageLedgerEntry> listEntries(UUID channelId, Set<String> types,
        Long afterSeq, String agentId, Instant since,
        String correlationId, boolean sortDesc, int limit) {
    // in-memory impl
}
```

Alternatively, make the base method delegate to the new overload, so the stub's existing override still covers the old call path:

```java
// Old 6-arg delegates to new 8-arg:
public List<...> listEntries(UUID channelId, Set<String> types,
        Long afterSeq, String agentId, Instant since, int limit) {
    return listEntries(channelId, types, afterSeq, agentId, since, null, false, limit);
}
```

With delegation, the stub's override of the 6-arg variant intercepts all calls routed through it — the 8-arg base impl is only reached when the test explicitly calls the 8-arg variant (which must then also be overridden).

### Why this is non-obvious

The failure looks like a missing CDI injection ("em is null") rather than a missing override. The error stacktrace doesn't show the stub class at all — it shows the base class method as if the stub didn't exist. The original test stub was correct and all previous tests pass; only tests exercising the new overload hit the issue. Nothing in the compiler output flags the missing override because the method *is* legitimately inherited.
