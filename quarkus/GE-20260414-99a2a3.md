## Context

A service uses constructor injection for its main dependencies (repositories, config) — making it unit-testable via `new MyService(repo, config)`. You want to add CDI event emission to the service without forcing unit tests to provide a mock `Event<T>`.

## Technique

Field-inject the `Event<T>` separately from constructor-injected deps, and null-guard every fire call:

```java
@ApplicationScoped
public class WorkItemService {

    private final WorkItemRepository repo;
    private final TarkusConfig config;

    @Inject                          // constructor injection — required deps
    public WorkItemService(WorkItemRepository repo, TarkusConfig config) {
        this.repo = repo;
        this.config = config;
    }

    @Inject                          // field injection — optional CDI dep
    Event<WorkItemLifecycleEvent> lifecycleEvent;

    @Transactional
    public WorkItem complete(UUID id, String actor, String resolution) {
        // ... business logic ...
        workItemRepo.save(item);
        if (lifecycleEvent != null) {          // null in new MyService(...) context
            lifecycleEvent.fire(WorkItemLifecycleEvent.of("COMPLETED", ...));
        }
        return item;
    }
}
```

Unit tests continue to use `new WorkItemService(repo, config)` — `lifecycleEvent` is `null`, the guard prevents NPE, and 58 unit tests run without a CDI container.

`@QuarkusTest` integration tests use CDI injection — `lifecycleEvent` is populated, events fire normally.

## Why This Is Non-Obvious

The standard advice is "use constructor injection for everything testable". Adding a new dependency to the constructor forces all unit tests to be updated. CDI provides no built-in `Optional<Event<T>>` injection. The field + null-guard pattern bridges both worlds with minimal ceremony — but requires knowing that CDI allows mixed injection (constructor + field) on the same bean.
