## What It Does

Quarkus 3.32.2 with `quarkus-rest-jackson` automatically handles native image reflection registration for:
- Java records used as JAX-RS request/response bodies (Jackson serialization/deserialization)
- Enums stored via `@Enumerated(EnumType.STRING)` in JPA entities
- `@Provider` ExceptionMapper implementations
- CDI event types fired via `Event<T>`

No `@RegisterForReflection` annotations are needed on any of these types.

## Evidence

Static analysis of a Quarkus Tarkus project identified 17-18 types as requiring `@RegisterForReflection` based on standard guidance. The native build (`mvn verify -Pnative`) succeeded without adding a single annotation. All 19 `@QuarkusIntegrationTest` tests passed against the native binary on the first attempt.

## Context

This contradicts the widely-cited guidance to annotate REST DTO records and JPA enums with `@RegisterForReflection` when targeting native image. The official Quarkus docs and community tutorials still recommend it as a standard step.

## Why This Is Undocumented

The build-time analysis improvements in recent Quarkus versions are not accompanied by explicit documentation stating "you no longer need @RegisterForReflection for X". The guidance is preserved from older versions where it was necessary, creating unnecessary boilerplate in current projects.

## Caveat

This applies to types used within JAX-RS resource methods and CDI-managed beans that Quarkus can trace at build time. Types loaded dynamically (e.g. via `Class.forName()`, reflection from configuration strings, or loaded by third-party libraries outside Quarkus's build-time analysis) still need explicit registration.
