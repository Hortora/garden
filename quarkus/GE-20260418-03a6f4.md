## What it is

Select a CDI bean by name at startup from a config property using `Instance<T>` and `NamedLiteral`. Zero framework code beyond `@Named` on each implementation.

## When to use it

When you have multiple implementations of an interface, want to select one via config (`application.properties`), and want the selection to happen once at startup with fast-fail if the name is wrong.

## The pattern

```java
// Interface
public interface KiteStrategy {
    Point2d retreatTarget(Unit unit, List<Unit> enemies);
}

// Implementations
@ApplicationScoped @Named("direct")
public class DirectKiteStrategy implements KiteStrategy { ... }

@ApplicationScoped @Named("terrain-aware")
public class TerrainAwareKiteStrategy implements KiteStrategy { ... }

// Selector bean
@ApplicationScoped
public class TacticsController {

    @Inject
    @ConfigProperty(name = "quarkmind.tactics.kite.strategy", defaultValue = "direct")
    String kiteStrategyName;

    @Inject Instance<KiteStrategy> kiteStrategies;

    private KiteStrategy kiteStrategy;

    @PostConstruct
    void init() {
        kiteStrategy = kiteStrategies.select(NamedLiteral.of(kiteStrategyName)).get();
    }
}
```

Config:
```properties
quarkmind.tactics.kite.strategy=terrain-aware
```

## Why this is non-obvious

Most developers reach for a factory class, a switch statement, or `@Alternative` + `@Priority`. The `Instance<T>.select(NamedLiteral.of(name))` idiom is in the CDI spec but rarely shown in examples. Key benefits:
- **Fast fail at startup** — `get()` throws `UnsatisfiedResolutionException` immediately if the name doesn't match any `@Named` bean, not silently at call time
- **Zero factory boilerplate** — CDI discovers all implementations automatically; adding a new strategy is just a new `@Named` class
- **Future-extensible** — a `@Named("lua") LuaStrategy` implementation can be added without touching the selector

`NamedLiteral.of(name)` is from `jakarta.enterprise.inject.literal.NamedLiteral` (CDI 4.x / Quarkus 3.x).
