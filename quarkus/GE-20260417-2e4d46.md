## Symptom

`mvn test` passes in the current JVM session but fails on the next run (e.g. in CI or after opening a new terminal) with:

```
FlywayException: Validate failed: Migrations have failed validation
  Migration checksum mismatch for migration version 1000
  -> Applied to database : -1234567890
  -> Resolved locally    : 987654321
```

Or H2 silently reuses the old schema and tests fail with `Table "FOO" not found`.

## Context

Occurs when you edit a Flyway SQL migration file that has already been applied to the H2 in-memory database in the current JVM session.

H2 configured with `DB_CLOSE_DELAY=-1` keeps the database alive within the JVM process. Flyway's schema history records the old checksum on first apply. When you edit the SQL file and run `mvn test` again *in the same JVM*, the H2 database is still alive with the old schema history — tests pass because the old schema is intact. On the *next* fresh JVM start, Flyway detects the checksum mismatch against the now-changed file.

Common trigger: editing V1000 or V1002 in-place on a pre-release project (no deployed data, no need for ALTER TABLE migrations).

## What was tried (didn't work)

- Running `mvn test` again — same JVM, same cached H2 database, still passes
- Assuming the SQL edit was wrong — the SQL was correct; the problem was Flyway's checksum cache

## Root cause

Flyway stores the checksum of each migration in `flyway_schema_history`. Editing the SQL file after it has been applied changes the file checksum but not the stored one. H2 `mem:` databases with `DB_CLOSE_DELAY=-1` live for the lifetime of the JVM process — Flyway never re-validates against the file for an already-applied migration within the same process.

## Fix

Always use `mvn clean test` (not `mvn test`) after editing any Flyway SQL source file:

```bash
# Wrong — H2 may reuse cached schema from this JVM session
mvn test

# Correct — forces fresh JVM, fresh H2, Flyway re-applies all migrations
mvn clean test
```

`mvn clean` wipes `target/` which kills the JVM process, destroying the H2 in-memory database and forcing a complete fresh migration run.

## Why this is non-obvious

Tests appear to pass immediately after the edit (same JVM session, cached schema). The failure only surfaces on the *next* fresh run — which may be CI, a new terminal, or the next day. The symptom looks like a Flyway configuration problem rather than a test execution order issue.
