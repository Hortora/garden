### Context
A library module provides a default CDI bean (e.g. a `HolidayCalendar` implementation). Application developers should be able to override it with their own implementation without fighting with `@Alternative @Priority(1)` boilerplate.

### Technique
Use `@Produces @DefaultBean` on a producer method in the library:

```java
@ApplicationScoped
public class HolidayCalendarProducer {

    @Inject WorkItemsConfig config;

    @Produces
    @DefaultBean          // ← yields to any @ApplicationScoped HolidayCalendar bean
    @ApplicationScoped
    public HolidayCalendar produce() {
        return config.businessHours().holidayIcalUrl()
                .filter(url -> !url.isBlank())
                .map(ICalHolidayCalendar::new)
                .map(HolidayCalendar.class::cast)
                .orElseGet(() -> new ConfigHolidayCalendar(config));
    }
}
```

Application override (no annotations needed beyond the scope):

```java
@ApplicationScoped
public class CompanyHolidayCalendar implements HolidayCalendar {
    @Override
    public boolean isHoliday(LocalDate date, ZoneId zone) {
        return /* query HR system */ false;
    }
}
```

The application bean wins automatically — `@DefaultBean` makes the producer act as a fallback only when no other bean satisfies the injection point.

### Why this is non-obvious
Most developers reach for `@Alternative @Priority(1)` on the overriding bean, which requires the application to know about ArC's priority system. `@DefaultBean` inverts this: the library marks its own bean as "low priority" and the application doesn't need any special annotations. It's a Quarkus ArC feature (`io.quarkus.arc.DefaultBean`), not in the CDI spec — easy to miss.

### Constraints
Quarkus only (ArC extension). `@DefaultBean` is not available in standard CDI containers.
