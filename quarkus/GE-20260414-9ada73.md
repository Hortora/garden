## Symptom

A case or entity state changes correctly in-memory (e.g. cache shows FAULTED)
but no EventLog entry is written and no downstream event is published. No
exception is thrown. The Panache `withTransaction()` block appears to complete
silently.

## Context

An event handler retrieves an entity from an in-memory cache (not from a Hibernate
session). The entity was persisted earlier in a different Vert.x handler and is now
**detached** — it has an assigned `id` but no active session context.

Calling `entity.persist()` on a detached Panache entity triggers an INSERT (not
an UPDATE), which immediately fails with a duplicate-key constraint violation on
the primary key or a unique column. The reactive pipeline's error handling swallows
this exception, silently rolling back the entire transaction.

## What was tried (didn't work)

Calling `caseInstance.persist()` inside a `Panache.withTransaction()` block —
assumed to behave as upsert. It does not: `PanacheEntity.persist()` always
attempts INSERT on entities with an assigned id.

## Root cause

`PanacheEntity.persist()` calls Hibernate's `session.persist()`, which schedules
an INSERT for the entity. When the entity already exists in the database
(persisted earlier in a different session/transaction), the INSERT fails with a
constraint violation. Because this happens inside a reactive `Uni` chain with
`.replaceWithVoid()`, the exception is absorbed and the outer Uni completes
normally.

The state mutation (e.g. `entity.setState(FAULTED)`) happens before the transaction,
so the in-memory representation is correct even after the DB transaction fails.

## Fix

Use `session.merge(entity)` instead of `entity.persist()`. `merge()` issues an
UPDATE for detached entities with an existing identity:

```java
// ❌ Silent rollback for detached entities
return Panache.withTransaction(() -> caseInstance.persist().chain(eventLog::persist));

// ✅ Correct — merge re-attaches and updates
return Panache.withTransaction(
    () -> Panache.getSession()
        .chain(session -> session.merge(caseInstance))
        .chain(merged -> eventLog.persist()));
```

## Why this is non-obvious

The bug is invisible: the in-memory state is correct (set before the transaction),
no exception surfaces, and no downstream effect reveals the failure immediately.
The entity appears persisted from the caller's perspective. Only a missing EventLog
entry — or a missing downstream event — reveals the problem, and only if you're
looking for it. The bug is also untriggerable by unit tests and only appears when
a specific execution path is first exercised end-to-end.
