## What It Does

A Quarkus extension's runtime JAR includes two metadata files:
- `META-INF/quarkus-extension.yaml` — human-readable metadata (name, description, keywords, categories)
- `META-INF/quarkus-extension.properties` — machine-read at augmentation time; contains the `deployment-artifact` reference

The `deployment-artifact` entry in `.properties` is what Quarkus actually reads at augmentation time to locate the deployment processor (`@BuildStep` classes). The YAML file does NOT contain this reference.

## Evidence

```bash
unzip -p quarkus-myext.jar META-INF/quarkus-extension.properties
# → deployment-artifact=io.quarkiverse.myext\:quarkus-myext-deployment\:1.0.0-SNAPSHOT
```

```bash
unzip -p quarkus-myext.jar META-INF/quarkus-extension.yaml
# → No deployment-artifact field — only name, description, metadata
```

## Why This Is Undocumented

The Quarkus extension guide and Quarkiverse documentation describe the YAML file as the extension descriptor without clearly distinguishing which file serves which purpose at runtime. The `.properties` file is written by `quarkus-extension-maven-plugin` but its role in extension activation is not documented separately from the YAML.

## Practical Use

When debugging why an extension's `@BuildStep` processors are not running, check `META-INF/quarkus-extension.properties` in the runtime JAR first — if `deployment-artifact` is missing or incorrect, the extension will appear to load (features list won't include it) but no build-time processing will occur.
