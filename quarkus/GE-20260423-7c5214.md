## Symptom

`mvn quarkus:dev` starts but immediately fails. The dev server is "running" (port responds) but returns an HTML error page for every request:

```
ClassTooLargeException: Class too large: io/quarkus/runner/recorded/LogStreamProcessor$handlerNNNNNNNN
```

No code was broken — it compiled fine with `mvn compile`.

## Context

Occurs after an application accumulates a large number of enum values, switch statement cases, or string literals across CDI-managed classes. The Quarkus build-time augmentation generates handler classes that eventually exceed the JVM's 64KB constant pool limit per class.

Triggered in practice by adding 20–30 new enum values with corresponding large switch statements (e.g. mapping unit types to draw functions across multiple files).

## What was tried

- Restarting `mvn quarkus:dev` — same error
- Checking compile errors — none, `mvn compile` was clean
- Checking test failures — tests passed fine with `mvn test`

## Root cause

Quarkus generates a startup handler class (`LogStreamProcessor$handlerNNNNNNNN`) at augmentation time. When the application grows large, this generated class accumulates too many constant pool entries (string literals, method references) and hits the JVM 64KB class file limit.

The stale augmentation cache compounds this: old generated files don't get regenerated until explicitly cleaned.

## Fix

```bash
mvn clean && mvn quarkus:dev
```

The `mvn clean` removes the Quarkus augmentation cache (`target/quarkus-app/`). The next dev mode start regenerates the handler classes from scratch, producing a smaller constant pool.

## Why this is non-obvious

- `mvn compile` succeeds — the error is not a compile-time issue
- The failing class is a Quarkus-generated file, not any class you wrote
- The error message (`LogStreamProcessor$handler...`) looks unrelated to enum/switch growth
- Restarting `mvn quarkus:dev` alone doesn't help — the stale cache persists
