### Symptom
`UnsatisfiedResolutionException: Unsatisfied dependency for type <RepositoryClass>` at Quarkus augmentation when a CDI bean injects a JPA repository class from a library.

### Context
Library ships a JPA repository class (extends an abstract JPA base class) with package-private `@Inject EntityManager` fields. No `@ApplicationScoped` on the class. JAR is indexed via `quarkus.index-dependency`.

### Root cause
Quarkus ARC only treats a class as a CDI bean if it carries a scope annotation (`@ApplicationScoped`, `@RequestScoped`, etc.) or is produced by `@Produces`. A plain class with injected fields is never a CDI bean regardless of indexing. Library authors designing for annotation-based DI frameworks often omit scope annotations, assuming the container handles it.

### Fix
Introduce an interface + `@DefaultBean` no-op:

```java
public interface LineageQuery {
    List<Result> findCompleted(UUID id);
}

@ApplicationScoped
@DefaultBean
public class EmptyLineageQuery implements LineageQuery {
    public List<Result> findCompleted(UUID id) { return List.of(); }
}
```

Inject `LineageQuery` instead of the library class directly. The CDI graph is always satisfied. Wire the real JPA-backed implementation later via `@Alternative` + `@Priority`, or a `@Produces` method, once the datasource is configured.

### Why this is non-obvious
The repository class looks exactly like a CDI-injectable component — it uses `@Inject`, has dependencies, and has a service role. The error "UnsatisfiedResolutionException" doesn't say "not a CDI bean" — it reads like an indexing problem. Adding `quarkus.index-dependency` doesn't help because the class genuinely has no CDI scope.
