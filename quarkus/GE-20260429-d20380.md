## JPA JOINED inheritance: filtering by subclass column vs inherited column for same UUID — only inherited uses the index

### Symptom

A JPQL query using `WHERE e.caseId = :id` runs full scans on the subclass join despite an index existing for the same logical filter. Switching to `WHERE e.subjectId = :id` — where `subjectId` is set to the same UUID — makes the query use the index.

### Context

In JPA `JOINED` inheritance, the entity hierarchy maps to multiple tables joined on `id`. Consider:

```java
@Entity @Table(name = "ledger_entry")
public abstract class LedgerEntry {
    @Column(name = "subject_id") public UUID subjectId;  // indexed
    @Column(name = "sequence_number") public int sequenceNumber;
}

@Entity @Table(name = "case_ledger_entry")
public class CaseLedgerEntry extends LedgerEntry {
    @Column(name = "case_id") public UUID caseId;  // same value, NOT indexed
}
```

DB index: `CREATE INDEX idx ON ledger_entry (subject_id, sequence_number)` — on the **parent** table only.

At runtime, `caseId` and `subjectId` are always set to the same UUID. But the JPQL predicate determines which table column the WHERE clause targets:

```sql
-- WHERE e.caseId = :id → filters on case_ledger_entry.case_id (no index)
SELECT ... FROM ledger_entry l JOIN case_ledger_entry c ON l.id = c.id WHERE c.case_id = ?

-- WHERE e.subjectId = :id → filters on ledger_entry.subject_id (indexed)
SELECT ... FROM ledger_entry l JOIN case_ledger_entry c ON l.id = c.id WHERE l.subject_id = ?
```

### Fix

When the same UUID is stored in both the subclass and inherited columns, always query via the inherited field to use the index:

```java
// Correct — uses idx_ledger_entry_subject_seq:
em.createQuery(
    "SELECT e FROM CaseLedgerEntry e WHERE e.subjectId = :id ORDER BY e.sequenceNumber DESC",
    CaseLedgerEntry.class)
  .setParameter("id", caseId)
  .setMaxResults(1)
  .getResultStream().findFirst();

// Avoid — same result, no index benefit:
em.createQuery(
    "SELECT e FROM CaseLedgerEntry e WHERE e.caseId = :id ORDER BY e.sequenceNumber DESC",
    CaseLedgerEntry.class)
```

### Why this is non-obvious

Hibernate makes both fields appear semantically equivalent in the Java model — same type, same runtime value, same query results. There is no warning from Hibernate's query plan, no IDE hint, and no mention of this in the JPA or Hibernate ORM documentation for JOINED inheritance. The only way to discover it is to inspect the schema DDL and trace which table each mapped column lives on.
