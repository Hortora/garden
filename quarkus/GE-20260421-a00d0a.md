## Problem

A UI panel intended to show only in `%emulated` profile still appears during `@QuarkusTest` runs because the endpoint returns HTTP 200 in both profiles.

## Symptom

`configPanelHiddenInMockProfile` Playwright test fails — the panel is visible when it shouldn't be. The endpoint `/qa/emulated/config` returns 200 in `%test` profile because the `EmulatedConfigResource` bean is active (it lacks a profile guard).

## Root cause

Quarkus CDI beans annotated with `@UnlessBuildProfile("prod")` are active in ALL non-prod profiles including `%test`. The HTTP status of an endpoint is therefore not a reliable signal for "am I running in emulated mode?" — it only tells you whether the bean exists, not whether the full emulated physics engine is running.

## Fix

Add an `active` boolean field to the config bean, controlled by a profile-specific config property:

```java
// EmulatedConfig.java
@ConfigProperty(name = "emulated.active", defaultValue = "false")
boolean active;
```

```properties
# application.properties
%emulated.emulated.active=true
```

The JS client gates the panel on `cfg.active`:

```javascript
fetch('/qa/emulated/config')
  .then(r => { if (!r.ok) return null; return r.json(); })
  .then(cfg => { if (!cfg || !cfg.active) return; panel.style.display = 'block'; });
```

## Why this is non-obvious

Developers assume "HTTP 200 from an emulated endpoint = emulated mode". In Quarkus, `@UnlessBuildProfile("prod")` is the only profile guard on most QA beans, so everything is active in `%test`. The correct guard is a config flag, not the HTTP response code.
