### Symptom

When designing a desiredstate domain module for CaseHub's own topology (agents, channels, case types, trust policies), the natural assumption — inherited from the infra module — is that there are provisioning-order dependencies between entity types (e.g., "agents depend on channels" or "case types depend on agents").

### Context

Verified against the actual creation APIs:

- **AgentDescriptor** (eidos): no channel references, no case references. Capabilities and disposition are self-contained.
- **ChannelCreateRequest** (qhorus): no agent references. Channels are standalone communication primitives.
- **CaseDefinition** (engine): workers listed but not pre-resolved to agents. Agent selection happens at case instantiation via routing policies.
- **TrustRoutingPolicy** (engine): per-capability thresholds via SPI. Not per-agent.

All four entity types are independently creatable with no foreign-key-style references to each other. Binding between them happens entirely at runtime through routing policies, capability matching, and channel dispatch.

### Root cause

The CaseHub platform was designed for loose coupling. Agents discover channels by semantic (work/observe/oversight), not by ID. Cases discover agents by capability matching, not by name. Trust policies apply per capability, not per agent. This is a deliberate architectural choice — not an accident.

### Fix

Design the desiredstate graph as **flat by default** — no inherent dependency edges between node types. All four types provision in parallel. Dependencies only exist if the user explicitly declares `dependsOn` relationships in the YAML goal declaration.

This is fundamentally different from the infra module, where dependencies are natural (a `k8s_deployment` depends on a `k8s_namespace`). The deployment domain's value comes from drift detection and self-healing, not from provisioning-order orchestration.

### Why this is non-obvious

The infra module (the first desiredstate domain implementation) has rich dependency graphs. A developer building the second domain module naturally assumes the same structure. Discovering that the deployment domain is flat requires investigating each foundation API's creation parameters independently — there's no documentation that states "CaseHub entities are independently creatable."

### Alternatives considered

- Assumed channels depend on agents (agents "own" channels) — wrong; channels are standalone
- Assumed case types depend on agents (case definitions reference agents) — wrong; binding is runtime
- Assumed trust policies depend on agents (policies are per-agent) — wrong; policies are per-capability
