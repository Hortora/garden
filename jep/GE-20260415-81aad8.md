### Symptom
After starting a JEP-backed JVM process that requires `PYTHONHOME=/path/to/python3.12`, running `python3 -c "..."` in the **same shell session** fails with:

```
Fatal Python error: Failed to import encodings module
ModuleNotFoundError: No module named 'encodings'
```

Even though `python3` is the system Python and works fine in a fresh shell.

### Context
Running integration tests against a Quarkus server backed by JEP. The test setup exports `PYTHONHOME` to point at a bundled Python 3.12 installation. Test scripts then pipe curl output through `python3 -c "import json, sys; ..."` to parse responses. These `python3` calls fail.

### Root cause
`export PYTHONHOME=...` in the parent shell makes `PYTHONHOME` available to **all subprocesses** in that shell session, including `python3`. The system `python3` (e.g. 3.13) uses `PYTHONHOME` to find its stdlib. If `PYTHONHOME` points at a Python 3.12 installation, the 3.13 interpreter loads the wrong stdlib path and fails on `encodings`.

### Fix
**Option A — Unset before running system Python in the same shell:**
```bash
unset PYTHONHOME
python3 -c "import json, sys; ..."
```

**Option B — Set PYTHONHOME only in the JVM spawn environment, not the parent shell:**
```javascript
// Node.js / Electron — pass env to child_process.spawn, don't export to parent
const env = { ...process.env, PYTHONHOME: bundledPythonRoot };
spawn('java', args, { env });
// PYTHONHOME is NOT set in the parent Node process
```

**Option C — Use the bundled Python explicitly for parsing:**
```bash
/path/to/bundled/python3 -c "import json, sys; ..."
```

### Why this is non-obvious
The failure happens in a `python3` command that has nothing to do with JEP or Java. The error message (`Failed to import encodings module`) is identical to the JEP init error, so it looks like a JEP problem. The actual cause — env var bleed from a previous shell export — is not obvious unless you know to check `echo $PYTHONHOME`.
