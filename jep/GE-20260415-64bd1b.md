### What and why
When passing a Java string (e.g. a file system path) into Python via `interp.exec()`, the naive approach is string interpolation into a Python literal. This breaks on single quotes in the value (paths like `/Users/user's docs/`) and requires custom escaping that is easy to get wrong.

`interp.set(name, value)` passes a Java `String` directly as a Python variable — no string literal, no escaping, no injection risk.

### The pattern

**Broken (single quotes in path cause Python SyntaxError):**
```java
// Breaks if repoRootPath contains a single quote
interp.exec("sys.path.insert(0, '" + repoRootPath + "')");
```

**Correct:**
```java
// set() passes the Java String directly — no escaping needed
interp.set("_sparge_root", repoRootPath);
interp.exec("sys.path.insert(0, _sparge_root)");
```

The same applies to any value injection:
```java
// Passing a config value
interp.set("_github_token", token);
interp.exec("bridge.set_token(_github_token)");
```

### Why this is non-obvious
`interp.exec()` is the obvious way to run any Python statement, and string interpolation is the instinct for parameterising it. The JEP API has `set()` specifically for this purpose, but it's not prominently featured in examples — most JEP tutorials show `exec()` with hardcoded strings. Escaping functions exist but are easy to get wrong, and single-quote-in-path is a real edge case (e.g. macOS usernames with apostrophes).

`interp.invoke()` already accepts Java args directly, so the issue only arises when using `exec()` for setup code.
