### What and why
When embedding a thread-affinite interpreter (JNI/FFM-backed, single-owner) in a multi-threaded JVM HTTP server, the correct architecture is a **dedicated daemon thread** that owns the interpreter for its entire lifetime, with all callers submitting tasks via a message queue.

The instinct is to use `synchronized` — it prevents concurrent calls. But thread-affinite interpreters (JEP, some JNI libraries) require calls to originate from the **exact thread that created the resource**, not merely one thread at a time. `synchronized` satisfies concurrency but not ownership. Calling from the wrong thread typically deadlocks silently.

### The pattern

```java
@ApplicationScoped
public class PythonBridge {

    private record Task(String function, Object[] args, CompletableFuture<String> result) {}

    private final LinkedBlockingQueue<Task> queue = new LinkedBlockingQueue<>();

    @PostConstruct
    void init() {
        CompletableFuture<String> initDone = new CompletableFuture<>();
        Thread t = new Thread(() -> runLoop(initDone), "python-bridge");
        t.setDaemon(true);   // JVM exit doesn't wait for this thread
        t.start();
        try {
            initDone.get(60, TimeUnit.SECONDS); // fail fast if Python doesn't start
        } catch (TimeoutException e) {
            throw new RuntimeException("Python interpreter did not start within 60s");
        }
    }

    private void runLoop(CompletableFuture<String> initDone) {
        SharedInterpreter interp;
        try {
            interp = new SharedInterpreter();
            // ... Python initialisation ...
            initDone.complete(initResult);
        } catch (Exception e) {
            initDone.completeExceptionally(e);
            return;
        }

        while (!Thread.currentThread().isInterrupted()) {
            Task task;
            try {
                task = queue.poll(5, TimeUnit.SECONDS);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
            if (task == null) continue;

            try {
                String result = (String) interp.invoke(task.function(), task.args());
                task.result().complete(result == null ? "{\"status\":200,\"body\":null}" : result);
            } catch (JepException e) {
                task.result().complete(errorJson(e.getMessage()));
            }
        }
        try { interp.close(); } catch (Exception ignored) {}
    }

    /** Called from any HTTP thread. Blocks up to 30s for the Python thread's result. */
    public String call(String function, Object... args) {
        CompletableFuture<String> future = new CompletableFuture<>();
        queue.offer(new Task(function, args, future));
        try {
            return future.get(30, TimeUnit.SECONDS);
        } catch (TimeoutException e) {
            return errorJson("Python call timed out after 30s: " + function);
        } catch (Exception e) {
            return errorJson(e.getMessage());
        }
    }
}
```

### Key design decisions

- **`CompletableFuture<String>` per task** — decouples submission from completion; the HTTP thread blocks naturally.
- **30-second `call()` timeout** — ensures test suites never hang indefinitely if Python gets stuck.
- **60-second `init()` timeout** — surfaces startup failures immediately rather than letting the server appear healthy.
- **`queue.poll(5, TimeUnit.SECONDS)`** — allows graceful shutdown when interrupted; doesn't spin.
- **Daemon thread** — JVM exits cleanly without waiting for the Python thread.

### Why this is non-obvious
`synchronized` looks like the right tool — it's the standard Java solution for "only one thread at a time." The distinction between concurrency control and thread ownership is subtle and only matters for JNI/FFM resources with affinity requirements. Most Java developers never encounter it because standard Java objects have no thread affinity.

### Alternatives considered
- `synchronized` on `call()` — prevents concurrency, not ownership; deadlocks silently
- Thread-local `SubInterpreter` per HTTP thread — avoids affinity but multiple GILs interact unpredictably; state not shared across requests
- `CompletableFuture.runAsync()` on a single-thread executor — works but loses the explicit daemon/interrupt semantics; harder to add graceful shutdown
