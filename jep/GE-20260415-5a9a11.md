### Symptom
HTTP requests to a Quarkus server hang indefinitely — no error, no exception, no timeout. The server log shows the request was received but never responds. A JVM thread dump shows HTTP worker threads blocked inside JEP native code.

### Context
Using JEP `SharedInterpreter` in a `@ApplicationScoped` Quarkus bean. The interpreter is created in `@PostConstruct` (on a Quarkus startup thread). HTTP requests arrive on Vert.x worker threads. All `interp.invoke()` calls are guarded with `synchronized` to prevent concurrent access.

### What was tried
`synchronized` on the bridge method — prevents concurrent calls but does **not** fix the underlying issue. The first HTTP thread to acquire the lock hangs forever.

### Root cause
JEP's `SharedInterpreter` has **thread affinity**: it must be used exclusively from the thread that created it. When a different thread (an HTTP worker) calls `interp.invoke()`, JEP attempts to acquire the GIL from the wrong thread. The owner thread (startup thread, long since idle) effectively holds the GIL state; the caller waits forever. This is a **deadlock, not an exception** — JEP does not throw; it simply never returns.

`synchronized` prevents two HTTP threads from calling JEP simultaneously, but it does not transfer thread ownership. Any single HTTP thread calling JEP is already the wrong thread.

### Fix
Run a dedicated daemon thread that owns the `SharedInterpreter` for its entire lifetime. All callers submit tasks via a `LinkedBlockingQueue` and wait on a `CompletableFuture` with a timeout:

```java
private record Task(String function, Object[] args, CompletableFuture<String> result) {}
private final LinkedBlockingQueue<Task> queue = new LinkedBlockingQueue<>();

@PostConstruct
void init() {
    CompletableFuture<String> initDone = new CompletableFuture<>();
    Thread t = new Thread(() -> runLoop(initDone), "python-bridge");
    t.setDaemon(true);
    t.start();
    initDone.get(60, TimeUnit.SECONDS); // blocks until Python is ready
}

private void runLoop(CompletableFuture<String> initDone) {
    SharedInterpreter interp = new SharedInterpreter();
    // ... Python initialisation ...
    initDone.complete(initResult);

    while (!Thread.currentThread().isInterrupted()) {
        Task task = queue.poll(5, TimeUnit.SECONDS);
        if (task == null) continue;
        try {
            task.result().complete((String) interp.invoke(task.function(), task.args()));
        } catch (JepException e) {
            task.result().complete(errorJson(e.getMessage()));
        }
    }
}

public String call(String function, Object... args) {
    CompletableFuture<String> future = new CompletableFuture<>();
    queue.offer(new Task(function, args, future));
    return future.get(30, TimeUnit.SECONDS); // timeout prevents runaway hangs
}
```

The 30-second timeout on `call()` also ensures test suites don't hang forever if Python gets stuck.

### Why this is non-obvious
`synchronized` is the standard Java solution for "only one thread at a time." It correctly serialises calls. But JEP's thread affinity is not a *concurrency* constraint — it is an *ownership* constraint. The fix requires a different architecture (dedicated thread + message passing), not a stronger lock.

The JEP README says `SharedInterpreter` "should only be used from the thread it was created on," which reads as a soft style recommendation. In practice, calling from the wrong thread does not raise an exception — it deadlocks silently, making diagnosis very difficult.

### Why this fix
A single dedicated thread serialises naturally and satisfies thread affinity with no locking overhead. The task-queue pattern also centralises error handling and makes the timeout boundary explicit.

### Alternatives considered
- `synchronized` — prevents concurrent calls but doesn't fix thread ownership; still hangs
- Thread-local `SubInterpreter` — each HTTP thread gets its own interpreter; avoids affinity but multiple Python GILs interact unpredictably and state is not shared across requests
- `Py4J` sidecar — separate Python process over sockets; solves threading but adds 100–200ms per call and process management complexity
