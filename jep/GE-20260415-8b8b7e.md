### What and why
When using JEP to call Python from Java, the instinct is to use JEP's built-in type marshaling (Python `dict` ŌåÆ Java `Map`, Python `list` ŌåÆ Java `List`, etc.). This works for simple types but breaks on nested structures, custom objects, None/null handling, and type ambiguity. The alternative: **all Python bridge functions return a JSON string; Java parses it with Jackson**.

### The pattern

**Python bridge functions all return a JSON string:**

```python
# bridge.py ŌĆö all public functions return one of:
#   {"status": int, "body": Any}            ŌåÆ JSON endpoint
#   {"status": int, "content_type": "text/plain", "body": str}  ŌåÆ text endpoint

def _ok(body) -> str:
    return json.dumps({"status": 200, "body": body}, ensure_ascii=False)

def _err(status: int, msg: str) -> str:
    return json.dumps({"status": status, "body": {"error": msg}}, ensure_ascii=False)

def projects_list() -> str:
    projects = _load_projects()
    return _ok(projects)    # list of dicts ŌåÆ serialised to JSON string

def post_html(slug: str) -> str:
    # text/plain endpoint
    content = html_path.read_text(encoding="utf-8")
    return json.dumps({"status": 200, "content_type": "text/plain", "body": content},
                      ensure_ascii=False)
```

**Java parses the JSON string, never touches JEP's type system:**

```java
public static Response of(String bridgeJson) {
    try {
        JsonNode node = MAPPER.readTree(bridgeJson);
        int    status      = node.get("status").asInt();
        String contentType = node.has("content_type")
            ? node.get("content_type").asText() + "; charset=utf-8"
            : "application/json; charset=utf-8";
        JsonNode bodyNode = node.get("body");
        String   body     = bodyNode == null    ? "null"
                          : bodyNode.isTextual() ? bodyNode.asText()   // text/plain or text/html
                          :                        bodyNode.toString(); // JSON
        return Response.status(status)
            .header("Content-Type", contentType)
            .entity(body)
            .build();
    } catch (Exception e) {
        return Response.serverError().entity("{\"error\":\"bridge parse error\"}").build();
    }
}
```

### Why this is non-obvious
JEP's built-in type marshaling exists specifically for this ŌĆö it seems like the right tool. Problems emerge with:
- Nested `dict` ŌåÆ the Java side gets `Map<String, Object>` but needs to re-serialize to JSON anyway
- `None` ŌåÆ sometimes maps to `null`, sometimes to `"None"` string depending on context
- Custom Python objects ŌåÆ no automatic conversion
- `int` vs `float` ambiguity ŌåÆ `3` in Python, `3.0` in Java, `"3"` in JSON ŌĆö each context differs

The JSON-string protocol avoids all of these: Python's `json.dumps()` handles all Python types correctly; Jackson handles all Java types correctly. The boundary is a single well-defined contract (a JSON string), not a type system impedance problem.

### Alternatives considered
- JEP type marshaling directly ŌĆö works for flat structures but breaks on nested dicts, None, and custom types
- Protocol Buffers over JEP ŌĆö heavy setup, schema maintenance overhead, overkill for a local bridge
- `subprocess` with JSON stdio ŌĆö clean protocol but 100ŌĆō200ms per call is unacceptable for chained operations
