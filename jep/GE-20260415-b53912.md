### Symptom
JEP initialises without error, but the first Python call fails immediately:

```
Fatal Python error: Failed to import encodings module
Python runtime state: core initialized
ModuleNotFoundError: No module named 'encodings'
```

### Context
Embedding Python in a JVM application via JEP, using a bundled Python installation (not the system Python). `DYLD_LIBRARY_PATH` (macOS) or `LD_LIBRARY_PATH` (Linux) is set to the directory containing `libpython3.12.dylib` so JEP can find and load it.

### Root cause
`DYLD_LIBRARY_PATH` satisfies only the **dynamic linker** — it lets `libjep.jnilib` find and load `libpython3.12.dylib`. Once CPython is loaded, it needs to find its **standard library** (`encodings`, `codecs`, `io`, etc.). CPython locates its stdlib using `PYTHONHOME`. Without `PYTHONHOME`, CPython searches relative to the Python executable — but there is no Python executable when embedded in a JVM. Initialisation proceeds far enough to load Python, then fails when the first stdlib import runs.

### Fix
Set `PYTHONHOME` to the root of the bundled Python installation **before the JVM starts** (or in the environment passed to the JVM process):

```bash
export PYTHONHOME=/path/to/python/mac-arm64   # contains bin/, lib/, etc.
export DYLD_LIBRARY_PATH=/path/to/python/mac-arm64/lib:$DYLD_LIBRARY_PATH
```

Or in a Node.js Electron spawn:

```javascript
const env = {
  ...process.env,
  PYTHONHOME: path.join(pythonBase, 'mac-arm64'),
  DYLD_LIBRARY_PATH: `${path.join(pythonBase, 'mac-arm64', 'lib')}${path.delimiter}${process.env.DYLD_LIBRARY_PATH || ''}`,
};
spawn('java', jvmArgs, { env });
```

### Why this is non-obvious
`DYLD_LIBRARY_PATH` and `PYTHONHOME` look like they solve the same problem — "tell Python where things are." They are actually orthogonal:

- `DYLD_LIBRARY_PATH` → used by the **OS dynamic linker** to find shared libraries (`.dylib`/`.so`). Relevant before Python starts.
- `PYTHONHOME` → used by **CPython at runtime** to find its own stdlib. Relevant after Python starts.

Setting one does not imply the other. The error message ("No module named 'encodings'") sounds like a missing package rather than a missing environment variable, which misdirects debugging.
