### Symptom
Java compilation fails with `illegal escape character` after using `ide_replace_text_in_file`
to update a string containing `\\?` (e.g. `urlMatching("/path\\?query")`).

### Context
When `ide_replace_text_in_file` processes replacement text containing `\\` (Java escaped
backslash), it reduces the double-backslash to a single backslash in the written output.
This silently corrupts Java string literals that use `\\` for regex escaping.

### Root cause
The MCP tool's text replacement pipeline interprets `\\` as an escape sequence during
string processing, reducing it to `\` before writing to the file. The resulting `\?`
is not a valid Java escape character.

### Fix
Use the `Edit` tool with `replace_all: true` instead of `ide_replace_text_in_file` for
any replacement text containing double-backslash sequences.

### Why this is non-obvious
The replacement appears to succeed (no error from the tool), and the file looks correct
at a glance. The failure only surfaces at compilation time, and the error message
("illegal escape character") points to the affected line but not to the tool that
corrupted it. The natural assumption is a typo in the replacement text, not
silent modification by the tool.
