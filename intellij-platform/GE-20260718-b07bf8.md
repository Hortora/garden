### Symptom

After using `ide_replace_text_in_file` to add references to a new type (e.g. replacing `"BARRIER_STUCK"` with `WatchdogConditionType.BARRIER_STUCK`), calling `ide_optimize_imports` on the same file does not add the missing import for `WatchdogConditionType`. The file compiles with "cannot find symbol" errors.

### Context

Occurs during bulk refactoring — renaming a type or migrating from String to enum across many files. Each `ide_replace_text_in_file` call adds unqualified references to a type not yet imported. The expectation is that `ide_optimize_imports` will resolve the missing import automatically, as it would in the IDE's interactive editor.

### What was tried (didn't work)

1. `ide_optimize_imports` immediately after `ide_replace_text_in_file` — no effect, import not added
2. `ide_sync_files` on the target file followed by `ide_optimize_imports` — inconsistent results
3. Multiple `ide_optimize_imports` calls — same result

### Root cause

The IntelliJ VFS and PSI index do not refresh synchronously between MCP tool calls within a single Claude session. `ide_optimize_imports` operates on the file state at the last index refresh, not the state after `ide_replace_text_in_file` modified it. The text replacement writes to the Document API (immediate VFS), but the PSI layer used by import optimisation may not have re-parsed yet.

### Fix

Add imports explicitly using `ide_replace_text_in_file` on the import block:

```
ide_replace_text_in_file(
  file: "src/MyClass.java",
  searchText: "import com.example.ExistingType;",
  replaceText: "import com.example.ExistingType;\nimport com.example.NewType;"
)
```

This is reliable because `ide_replace_text_in_file` modifies text content (no PSI required), and the import block is always deterministic text.

### Why this is non-obvious

`ide_optimize_imports` works perfectly for imports that were stale before the session started — it just fails for symbols introduced during the same session. The failure is silent (no error returned, just "no changes made"), so it looks like the import simply isn't needed until `mvn compile` surfaces the error later.
