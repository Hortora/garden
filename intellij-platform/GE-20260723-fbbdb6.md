### Symptom
`ide_refactor_safe_delete`, `ide_replace_text_in_file`, and other MCP edit operations
modify files in the main repo checkout instead of the git worktree the session is
working in. Changes land on the wrong branch silently — the operation reports success.

### Context
When a git worktree is at `/path/worktrees/30/platform` (on branch `issue-384`)
and the main repo is at `/path/platform` (on branch `issue-193`), IntelliJ has
the main repo open as its project. MCP tool calls without an explicit
`project_path` pointing to the worktree operate on the main repo's files.

Calling `ide_open_project` with the worktree path may return
"Project 'platform' is open and ready" — misleadingly confirming the existing
main repo project rather than opening the worktree as a separate project.

### What was tried (didn't work)
1. Calling `ide_refactor_safe_delete` with relative paths — operated on main repo
2. Calling `ide_replace_text_in_file` with relative paths — edited main repo files
3. Calling `ide_open_project` with worktree path — returned success but didn't switch

### Root cause
The IntelliJ Index MCP plugin resolves project paths by matching open project names.
When the worktree project name matches the main repo project name ("platform"),
the plugin routes to the already-open main repo rather than opening the worktree
as a distinct project.

### Fix
1. Call `ide_open_project` with the worktree's absolute path to open it as a
   separate project window
2. Verify with `ide_project_status` — check that module paths show the worktree
   path (e.g. `/worktrees/30/platform/platform-api`), not the main repo path
3. Pass `project_path` pointing to the worktree root in every subsequent MCP call
4. After edit operations, verify files changed on disk in the worktree
   (not just MCP success response)

### Why this is non-obvious
The MCP tool calls report success — there is no error or warning that the operation
targeted a different checkout. The `ide_open_project` response "is open and ready"
reads as confirmation. The damage (wrong-branch edits) is only discovered when
`git status` in the worktree shows no changes.
