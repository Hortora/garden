### Symptom
After inserting a new `@Nested` test class via `ide_insert_member`, all existing
test methods in the file fail to compile with "cannot find symbol: method assertThat(...)".

### Context
File had `import static org.assertj.core.api.Assertions.assertThat;` at the top.
Multiple existing `@Nested` classes used `assertThat` throughout. The new member
was inserted with `reformat=true` (the default).

### What was tried (didn't work)
- Assumed the insertion itself was malformed — re-checked the inserted code
- Assumed the nested class shadowed the import — it didn't

### Root cause
`ide_insert_member` with `reformat=true` runs IntelliJ's import optimizer after
insertion. The optimizer incorrectly determined the static import was unused and
removed it — even though multiple existing methods in the file depend on it.

### Fix
After any `ide_insert_member` call, verify static imports are still present.
If removed, re-add them via `ide_replace_text_in_file`.

Alternative: pass `reformat=false` to `ide_insert_member`, then run
`ide_optimize_imports` separately if needed.

### Why this is non-obvious
The method name is `insert_member` — you expect it to add code, not remove
unrelated imports. The `reformat` parameter suggests whitespace/indentation
changes, not import removal. The compilation error points to the existing
methods, not to anything related to the insertion.
