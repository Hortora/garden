## `PsiParameter` does not extend `PsiMember` — use `getDeclarationScope()` to get the containing method

**ID:** GE-0163
**Stack:** IntelliJ Platform SDK (all versions)

**Symptom:** Code that works for `PsiMethod` and `PsiField` via `PsiMember.getContainingClass()` fails to compile or work for `PsiParameter` — `PsiParameter` does not implement `PsiMember`.

**Context:** When writing plugin code that needs to find the containing class of an arbitrary PSI element (method, field, or parameter), a natural attempt is to cast to `PsiMember` and call `getContainingClass()`. This works for methods and fields but not parameters.

### What was tried

Treating `PsiParameter` as a `PsiMember` to get its containing class.

### Root cause

`PsiParameter` extends `PsiVariable` and `PsiModifierListOwner`, but does NOT extend `PsiMember`. The containing method is accessed via `getDeclarationScope()`, not via `getContainingClass()`.

### Fix

To find the containing class of a `PsiParameter`:

```java
@Nullable
private static PsiClass getContainingClass(@NotNull PsiElement element) {
    if (element instanceof PsiClass cls) return cls;
    if (element instanceof PsiMember member) return member.getContainingClass(); // PsiMethod, PsiField
    if (element instanceof PsiParameter param) {
        // PsiParameter does NOT extend PsiMember — must go via declaration scope
        PsiElement scope = param.getDeclarationScope();
        if (scope instanceof PsiMethod method) return method.getContainingClass();
    }
    return null;
}
```

### Why non-obvious

`PsiParameter` appears to be a "member" of a method in the conceptual sense, so developers naturally expect it to implement `PsiMember`. The distinction is not documented in the IntelliJ Platform SDK docs — it's only discoverable by reading the PSI type hierarchy in source code.
