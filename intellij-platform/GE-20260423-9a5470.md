### Context
Verifying that an `Annotator` for an injected language (e.g. JEXL injected into Java string literals) produces the expected warnings in an IntelliJ plugin test.

### Technique
Use `myFixture.doHighlighting()` and filter the returned `List<HighlightInfo>` by severity and description:

```java
myFixture.configureByText("Foo.java",
    "@Permute(varName=\"i\", className=\"Join${badVar}\") public class J {}");
List<HighlightInfo> infos = myFixture.doHighlighting();
boolean hasWarning = infos.stream().anyMatch(h ->
    h.getSeverity() == HighlightSeverity.WARNING
    && h.getDescription() != null
    && h.getDescription().contains("badVar"));
assertTrue(hasWarning);
```

### Why This Is Non-Obvious
`myFixture.checkHighlighting()` with inline `<warning>`/`<error>` markers is the standard approach — but it fails for injected-language annotations because the markers must sit inside a Java string literal (inside the host), and the fixture doesn't resolve marker positions relative to injected fragment boundaries. `doHighlighting()` runs the full pipeline including injected annotators and returns all `HighlightInfo` instances, which can be filtered programmatically.
