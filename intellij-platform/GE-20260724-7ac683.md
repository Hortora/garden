IntelliJ MCP `ide_replace_text_in_file` (and other write operations like `ide_insert_member`, `ide_edit_member`) target whichever project is specified by the `project_path` parameter. When working with git worktrees, two projects may be open from the same repo — e.g. `/Users/me/repo` (main) and `/Users/me/worktrees/34/repo` (worktree). Both projects have the same name in IntelliJ ("repo").

### Symptom
Edits made via `ide_replace_text_in_file` succeed (replacement count > 0 returned) but `git diff` in the worktree shows no changes. The working tree appears clean despite confirmed edits.

### Context
Occurs when using Claude Code with IntelliJ MCP in a git worktree setup. The `project_path` parameter determines which project receives the edit. If you pass the main repo path instead of the worktree path, edits land in the main repo silently.

### Root cause
The MCP plugin routes operations to the project matching `project_path`. With worktrees, two projects share the same file structure but different `.git` references. The plugin does not warn when the target project differs from the CWD or the active session's working directory.

### Fix
Always pass the worktree-specific path as `project_path`:

```typescript
// Wrong — edits land in the main repo
ide_replace_text_in_file({
  file: "src/types.ts",
  project_path: "/Users/me/repo",  // main repo
  ...
})

// Right — edits land in the worktree
ide_replace_text_in_file({
  file: "src/types.ts",
  project_path: "/Users/me/worktrees/34/repo",  // worktree
  ...
})
```

After discovering the mistake, copy files from the main repo to the worktree and revert the main repo with `git checkout -- <files>`.

### Why this is non-obvious
The operation reports success (correct replacement count). No warning, no error. The only signal is that `git diff` in the worktree is empty — which could be misinterpreted as "the file was already up to date" rather than "edits went to the wrong project."

### Why this fix
The MCP plugin's project routing is by-design — it must support multi-project setups. The fix is purely on the caller side: always verify that `project_path` matches your intended working directory, especially in worktree setups.

### Alternatives considered
- Using `ide_sync_files` after edits — does not help; sync refreshes IntelliJ's VFS from disk, but the edits already landed in the wrong project's files on disk
- Checking `git status` before and after each edit — effective but expensive; the real fix is getting `project_path` right
