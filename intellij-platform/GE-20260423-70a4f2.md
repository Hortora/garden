### Symptom
After `myFixture.configureByText("Foo.java", "... ${<caret>i} ...")`, `myFixture.getFile()` returns the injected language file (e.g. `JexlFile`), not the Java host file. Subsequent `getFile().findElementAt(getCaretOffset())` returns null or an element in the wrong file.

### Context
`BasePlatformTestCase.configureInner(caresAboutInjection=true)`. When the caret lands inside an injected language range, the fixture detects this and switches `myFile` to the injected `PsiFile`.

### Root Cause
The fixture simulates the editor context: the active language at the caret determines the current file. When caret lands inside an injection, `myFile` is updated to the injected language file — correct for editor UX, wrong when the test needs the Java host.

### Fix
Strip `<caret>` before `configureByText`, compute target offset from the stripped string, then resolve the injected element explicitly:

```java
String src = javaSource.replace("<caret>", "");
int targetOffset = javaSource.indexOf("<caret>");
PsiFile hostFile = myFixture.configureByText("Foo.java", src);
PsiElement injected = InjectedLanguageManager.getInstance(getProject())
    .findInjectedElementAt(hostFile, targetOffset);
```

### Why This Is Non-Obvious
The `<caret>` marker is a well-known IntelliJ test convention and almost always works transparently. The injection-switching behaviour is only documented in comments inside `LightPlatformCodeInsightFixture` internals — not in `BasePlatformTestCase`'s API. The failure mode (null from `findElementAt`) looks like an injection problem rather than a file-switching problem.
