### Symptom
`ide_refactor_rename` reports success (`changesCount: 1`) but:
1. The `.java` file is not renamed — class `DeliveryCursorEntity` inside `DeliveryCursor.java`
2. Same-package references (no import statement) still use the old name
3. JPQL string literals containing the entity name are incorrectly renamed (e.g. `"FROM Message"` → `"FROM MessageEntity"`)

### Context
Renaming 10 JPA entity classes from `Channel` → `ChannelEntity` etc. via `ide_refactor_rename` with `relatedRenamingStrategy: "none"`. Most entities renamed correctly (100+ files updated). Two entities (DeliveryCursor, ChannelConnectorBinding) showed `changesCount: 1` — only the class declaration changed, not the file name or any references. Three JPQL string literals in JpaMessageStore/JpaCrossTenantMessageStore were incorrectly renamed.

### Root cause
The MCP plugin's rename operation has inconsistent handling of:
- File rename (class name → file name sync) — fails silently for some entities
- Same-package references (types used without `import`) — not scanned
- String literal analysis — over-aggressively renames entity names in JPQL queries without checking `@Entity(name=...)` annotation

### Fix
1. After every `ide_refactor_rename`, verify the file was actually renamed: `ls <expected-new-path>`
2. If the file wasn't renamed, use `ide_refactor_rename` with `targetType: "file"` to rename the file separately, then fix the class declaration manually
3. Add `@Entity(name = "OldName")` to ALL renamed JPA entities before renaming — this preserves JPQL entity names regardless of what IntelliJ does to string literals
4. After rename, search for the old class name in string literals: `ide_search_text(query="OldName", context="strings")` — fix any incorrectly renamed JPQL
5. Check same-package references: `ide_search_text(query="OldName", context="code")` — fix any that were missed

### Why this is non-obvious
The rename reports success and updates most references correctly. The partial failures are silent — no warning, no error. You only discover them at compile time, and the error messages ("class X should be in file named X.java") don't point back to the rename as the cause.

### Why this fix
Defensive verification after every rename is cheaper than debugging compilation failures across 200+ files. The `@Entity(name=...)` annotation should be added BEFORE renaming — it's a one-time cost that prevents all JPQL string literal issues.
