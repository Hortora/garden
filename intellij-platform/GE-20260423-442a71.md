### Symptom
Annotator runs O(N×tokens) times instead of once per injected file. No error, no exception, no user-visible symptom — IntelliJ deduplicates `HighlightInfo` by range+severity, so duplicate warnings are silently merged. The only observable sign is a performance hit on long expressions.

### Context
Any `Annotator` implementation for a language using a flat parser — a `PsiParser` that wraps all tokens in a single marker (`mark.done(root)`) with no composite AST nodes. Common when implementing custom language support for simple expression languages injected via `MultiHostInjector`.

### Root Cause
In a flat PSI tree every token is a direct child of the `PsiFile` root. The guard `element.getParent() != file` is always `false` (every token's parent IS the file), so the condition never triggers and the annotator body runs for every token — N times per highlight pass instead of once.

### Fix
```java
// Wrong — guard never fires in a flat PSI tree
if (element.getParent() != file) return;

// Correct — fires only for the file root itself
if (!(element instanceof MyLanguageFile)) return;
```

### Why This Is Non-Obvious
The guard looks semantically correct: "skip elements that aren't direct children of the file." But the logic is inverted — `parentOfToken != file` is false for every token, so all tokens pass through. The `instanceof` form identifies the single file-root node precisely, regardless of tree depth.

### Why This Fix
`instanceof MyLanguageFile` is true for exactly one element in the tree — the root — which is also the right entry point for running a full-file visitor (`file.getText()`, `file.accept(...)`). The `getParent()` approach can never work for flat trees; it requires composite intermediate nodes that simply don't exist.
