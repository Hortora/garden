### Problem

When writing an implementation plan for a method signature change (e.g. dropping a parameter),
caller enumeration done by manually reading files or searching by name misses callers in
distant modules. In casehub-openclaw, changing `ChannelContextWindowService.query()` from
3-arg to 2-arg missed `CasehubMcpResources.java` — a caller in the same `app/` module that
wasn't close to the resource being updated — causing an unplanned compile break discovered
mid-task.

### Technique

Before writing an implementation plan for any method signature change, call
`mcp__intellij-index__ide_find_references` on the method at its definition site:

```
ide_find_references(
  file="path/to/Service.java",
  line=<line of the method>,
  column=<column of the method name>
)
```

Use the returned file list as the authoritative caller inventory. Every file listed must
appear in the plan's "Files Modified" section.

### Why this is non-obvious

Manual enumeration (reading files, searching by class name) finds the callers you know
about. `ide_find_references` finds callers across ALL modules, including ones in distant
packages that happen to depend on the same service. In multi-module Maven projects this
is especially easy to miss — the compiler reports the break only when that specific
module builds, after plan steps have already been executed.

The non-obvious part for agentic workflows: `ide_find_references` must be invoked at
**plan-writing time**, not just at implementation time. By implementation time, a missed
caller causes a compile break mid-task that requires an unplanned fix.
