## Symptom

When testing an IntelliJ plugin's `LocalInspectionTool` via the MCP `ide_diagnostics` tool, the tool returns 0 problems even when a deliberate annotation mismatch has been introduced into the source file — exactly the scenario the inspection is designed to catch.

## Context

Any IntelliJ plugin test using the MCP `ide_diagnostics` tool to verify that a registered inspection fires on a problematic file. The file is synced (`ide_sync_files`) and `ide_diagnostics` is called — it returns `problemCount: 0` and an empty `problems` array.

## Root cause

`ide_diagnostics` runs the IDE's built-in code analysis (compiler errors, built-in warnings, run configurations). It does **not** invoke inspections registered via `localInspection` extension points in `plugin.xml`. Custom `LocalInspectionTool` implementations are part of the plugin inspection framework, which is a separate pipeline.

## Fix

Use `get_file_problems` instead. This tool invokes IntelliJ's full inspection framework including custom `LocalInspectionTool` plugins:

```python
# Wrong — doesn't run plugin inspections:
ide_diagnostics(file="src/MyTemplate.java")

# Correct — runs the full inspection framework including plugin inspections:
get_file_problems(filePath="src/MyTemplate.java", errorsOnly=False)
```

`get_file_problems` returns problems with `severity`, `description`, `line`, and `column` — including those from custom inspections.

## Why this is non-obvious

Both tools sound like they find code problems. `ide_diagnostics` is documented as getting "code diagnostics from multiple sources: file analysis (errors, warnings, intentions)". The distinction between "built-in analysis" and "plugin inspection framework" is not surfaced in the tool description. The silent 0-result is the only indication something is wrong, and it looks like the inspection is simply not firing rather than the wrong tool being used.
