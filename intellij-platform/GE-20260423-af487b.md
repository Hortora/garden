### Symptom
A string attribute containing `"A${i}x${i}"` produces a single injected file containing `"ii"` instead of two separate files each containing `"i"`. Annotators produce false-positive warnings for the concatenated token (`ii` is unknown; `i` would be known).

### Context
Any `MultiHostInjector` that calls `addPlace()` multiple times within one `startInjecting()`/`doneInjecting()` session for independent expressions in the same host literal.

### Root Cause
`MultiHostRegistrar.addPlace()` is designed to build one compound injection from multiple disjoint ranges — e.g. a string template with holes. When ranges represent independent expressions rather than parts of one compound expression, their contents are still concatenated into a single `PsiFile`.

### Fix
Move `startInjecting()`/`doneInjecting()` inside the range loop so each expression gets its own injection session:

```java
// Wrong — concatenates all expression contents into one PsiFile
registrar.startInjecting(MyLanguage.INSTANCE);
for (TextRange range : ranges) {
    registrar.addPlace("", "", host, range);
}
registrar.doneInjecting();

// Correct — each expression gets its own PsiFile
for (TextRange range : ranges) {
    registrar.startInjecting(MyLanguage.INSTANCE);
    registrar.addPlace("", "", host, range);
    registrar.doneInjecting();
}
```

### Why This Is Non-Obvious
The `addPlace()` API is documented as composable for compound expressions across ranges — correct for SQL with hole-substitution. The distinction between "independent expressions" and "one compound expression split across ranges" is not documented. The concatenation is invisible in single-expression cases.
