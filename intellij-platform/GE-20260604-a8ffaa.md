## Symptom

`ide_find_class("FlowWorkerExecutor")` returns `{"classes":[],"totalCount":0}` — class exists in the project but is not found.

## Context

Module was recently added to the project (e.g. `engine/flow/` added in the same or a recent session). The IDE shows the files in the project tree and text search via `ide_search_text` works immediately, but `ide_find_class` returns empty.

## Root Cause

`ide_find_class` uses the PSI/semantic index, which is built lazily and may lag for newly-added or recently-modified modules. The word index (used by `ide_search_text`) is updated faster and picks up new files sooner.

No error is returned — just an empty result set with `totalCount: 0`.

## Fix

Use `ide_search_text(query="ClassName", context="code")` as the first fallback when `ide_find_class` returns empty. If that also fails, fall back to `bash find` to locate the file, then read it directly.

```
# Fallback order:
1. ide_find_class("ClassName")       -- empty? try:
2. ide_search_text("ClassName", context="code")  -- still empty? try:
3. find /project/root -name "ClassName.java"
```

If the project is known to have recently-added modules, prefer `ide_search_text` from the start and skip `ide_find_class` entirely.

## Why This Is Non-Obvious

An empty result from `ide_find_class` is indistinguishable from "the class does not exist in the project." There is no "indexing in progress" signal, no partial result, and no error. A developer will waste time verifying the class is in the source tree, checking the module is included, and trying different name variations — before realising the semantic index simply hasn't caught up yet.
