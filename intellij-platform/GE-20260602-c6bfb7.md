## IntelliJ Index MCP plugin port 29170 connection-refused while IDEA is running — transient JVM pause

**ID:** GE-20260602-c6bfb7
**Stack:** JetBrains IntelliJ IDEA, Index MCP plugin (port 29170)
**Symptom:** `mcp__intellij-index__ide_index_status` (or any `mcp__intellij-index__*` tool) returns "Unable to connect. Is the computer able to access the url?" — `curl` to `http://127.0.0.1:29170/index-mcp/streamable-http` returns "Connection refused (7)". IDEA process is running and responsive.
**Context:** Occurs transiently at session start, after IDEA wakes from sleep, or during a GC pause in the plugin's embedded HTTP server. Usually self-resolves within 5–30 seconds.

### What was tried (didn't work)
- Checked `~/.claude/settings.json` for missing MCP server configuration — settings were correct
- Checked project-level settings — no MCP config there either
- Verified IDEA process was running via `pgrep idea` — confirmed running
- Attempted `curl` to port 29170 — "Connection refused", confirming server was not accepting

### Root cause

The Index MCP plugin runs an embedded HTTP server on port 29170. The server can be briefly unavailable during:
- JVM garbage collection pauses in the plugin's Ktor/Netty HTTP server
- Plugin restart triggered by a project open or IDE indexing event
- Cold start: the plugin takes several seconds to bind the port after IDEA launches

The symptom is indistinguishable from "plugin not installed" — connection refused looks the same whether the server never started or is temporarily unavailable.

### Fix

Retry once after a short pause. The work-start skill specifies: retry once, then stop and tell the user if still unavailable. Manual retry (re-invoking `ide_index_status`) is sufficient — no IDEA restart needed.

```bash
# Confirm the plugin is actually down (not misconfigured)
curl -s --max-time 5 http://127.0.0.1:29170/index-mcp/streamable-http \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list","params":{}}'
# "Connection refused" → wait 10s, retry
# Response → plugin is up
```

### Why this is non-obvious

"Connection refused" on a local port when the process is running almost always means the server didn't start, not that it's temporarily down. Developers debugging this investigate configuration (settings.json, MCP server declarations) before considering that the server is just paused. The fix (wait and retry) is counter-intuitive when the symptom looks like misconfiguration.
