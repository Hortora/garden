### Symptom
`ide_replace_text_in_file` returns `"replacements": 0` and `"No matches found"` when the search text appears to match the file content exactly. No error is raised — the operation silently does nothing.

### Context
Occurs when using multi-line `searchText` strings that include leading whitespace (indentation). The tool performs exact byte-level matching. Whitespace differences invisible in the Read tool output (tabs vs spaces, trailing spaces, different indentation levels) cause the match to fail silently.

Especially common when:
- Copying text from Read tool output (which prepends line numbers)
- Editing Java records with multi-line parameter lists
- Working with files that were recently reformatted by IntelliJ

### What was tried (didn't work)
- Using the exact text visible in Read tool output — fails due to invisible whitespace differences
- `ide_sync_files` before replacement — sometimes helps but not reliably
- Regex mode — different escaping rules, own set of issues

### Fix
1. **Use single-line unique strings** — find a short unique substring within the target and replace just that. Avoid multi-line searchText entirely when possible.
2. **Use `ide_replace_member` / `ide_edit_member`** for structural changes — these operate on AST members, not text, and are immune to whitespace issues.
3. **After silent failure, re-read the file** and compare the exact bytes at the target location before retrying with a corrected search string.

### Why this is non-obvious
The tool returns a success response with 0 replacements — there is no error, no warning, no diff showing what didn't match. A developer naturally assumes the text simply isn't in the file, when in reality it IS there but with invisible whitespace differences. The Read tool's line-number prefix formatting makes it easy to accidentally include or exclude whitespace when composing the search string.
