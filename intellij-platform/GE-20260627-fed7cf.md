### Symptom

`mcp__intellij-index__ide_find_class` with query `"QuantizationType"` returns zero results, despite `io.qdrant.client.grpc.Collections.QuantizationType` existing in the Qdrant client 1.18.1 JAR. The enum has values `UnknownQuantization`, `Int8`, `UNRECOGNIZED`.

### Context

The Qdrant Java client generates protobuf types as nested inner classes inside a single ~20,000-line outer class (`io.qdrant.client.grpc.Collections`). `ide_find_class` appears to miss some of these nested types — possibly due to indexing depth limits or the sheer size of the outer class.

This is distinct from GE-20260604-a8ffaa (PSI indexing delay for newly-added modules). In this case, the module was fully indexed (`ide_index_status` confirmed `isDumbMode: false, isIndexing: false`), but the nested enum was still invisible to `find_class`.

### Root cause

`ide_find_class` likely searches the PSI class index, which may not fully enumerate nested types inside very large protobuf-generated classes. `ide_find_symbol` uses a broader symbol index that includes nested enums, constants, and methods.

### Fix

When `ide_find_class` returns empty for a type you expect to exist in a dependency JAR, retry with `ide_find_symbol`:

```
mcp__intellij-index__ide_find_symbol(query="QuantizationType", scope="project_and_libraries")
```

`find_symbol` found the enum immediately with full qualified name, file location, and line number.

### Why this is non-obvious

`ide_find_class` is the natural tool for "does this class exist?" questions. It works reliably for top-level classes and most nested classes. The failure is silent — zero results, no error — so the caller concludes the type doesn't exist rather than suspecting an indexing gap.

### Alternatives considered

- `ide_search_text` with the enum name — also returned zero results for `"UnknownQuantization"` in this case
- Reading the decompiled JAR directly — works but pulls in massive context (1.3MB for the Collections class)
