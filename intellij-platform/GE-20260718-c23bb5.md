### Symptom

Compilation errors after `ide_replace_member`: `Modifier 'private' not allowed here`,
`';' expected`, `Variable already defined in scope`. The method appears syntactically
valid in the IDE but contains a nested duplicate declaration.

### Context

Occurs when using `ide_replace_member` to rewrite a method body. The tool inserts the
full replacement content (including method signature, modifiers, and braces) inside the
existing method declaration rather than replacing the entire body.

### What was tried (didn't work)

- `ide_edit_member` to fix the malformed method — fails because the class is unparseable
  (the duplicate declaration breaks PSI resolution, so member lookup returns "member not found")
- `ide_reformat_code` — cannot fix structural errors, only whitespace

### Root cause

`ide_replace_member` is documented as replacing the method body (between `{` and `}`), but
in practice it sometimes wraps the replacement content inside the existing declaration
instead of substituting. The result is a nested method:

```java
private void createSchema() {
    private void createSchema() {
        // ... replacement content ...
    }
}
```

The extra closing brace at the end (`}}`) also breaks subsequent method declarations.

### Fix

Use `ide_replace_text_in_file` to search for the duplicate inner declaration and remove it:

```
searchText: "    private void createSchema() {\n        private void createSchema () {"
replaceText: "    private void createSchema() {"
```

Also fix the trailing double-brace:

```
searchText: "        }}"
replaceText: "    }"
```

Then run `ide_reformat_code` on the affected range to normalize indentation.

**Prevention:** For method rewrites, prefer `ide_edit_member` (replaces the full declaration
including signature) over `ide_replace_member` (body-only replacement). When the class is
already in an error state from a prior failed edit, `ide_edit_member` will also fail —
`ide_replace_text_in_file` is the only recovery path.

### Why this is non-obvious

The symptom (`Modifier 'private' not allowed here`) suggests a stray modifier keyword,
not a duplicated method declaration. The nested structure is hard to spot because the
inner declaration has slightly different whitespace, making diff inspection misleading.
The IDE's own error highlighting may not show the full picture when the PSI tree is broken.
