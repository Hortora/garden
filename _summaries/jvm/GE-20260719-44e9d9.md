GE-20260719-44e9d9: [gotcha, 12/15] Setting OutboundMessage.target on engine COMMANDs triggers ObligorTrustPolicy — sender exemption needed | casehub-qhorus, trust-gate

When `MessageDispatch.target` is set on a COMMAND, `MessageService.dispatch()` checks `ObligorTrustPolicy.permits()`. Previously engine-dispatched COMMANDs had `target=null` and bypassed this. Adding `target=worker.name()` for parallel routing (openclaw#70) caused a regression: the trust gate rejected engine COMMANDs when `min-obligor-trust > 0`.

**Root cause:** The trust check condition `type==COMMAND && target!=null && !target.contains(":")` doesn't distinguish peer-agent COMMANDs (should be gated) from engine-orchestrated COMMANDs (should not).

**Fix:** Add `&& !dispatch.sender().contains(":")` — system senders (e.g. `casehub-engine:orchestrator`) bypass. Applied to both `MessageService` and `ReactiveMessageService`.

**How to recognise:** Any time you add or populate a field on `MessageDispatch` that is used in an enforcement gate condition, check whether engine-dispatched messages (system senders) would newly trigger that gate.
