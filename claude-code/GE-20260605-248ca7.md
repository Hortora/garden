## Design parallel agents as write-only — pass all deletions back to the parent session

**ID:** GE-20260605-248ca7
**Stack:** Claude Code, parallel agents

**Symptom:** A parallel agent tasked with deleting directories or files returns a summary saying "please run the following commands manually" instead of executing the deletions. The agent completed its creation/editing tasks but couldn't remove anything.

**Context:** Claude Code's permission model blocks destructive Bash operations (`rm -rf`) in agent tasks regardless of parent session allow-lists. Agents can freely create files and directories (Write, Edit tools) but cannot delete them via Bash — the permission gate fires and the agent stops or returns instructions.

**Technique:** Design agent tasks to be **write-only**. Agents create content files, write new skill directories, update existing files. They explicitly return a list of paths to delete in their result summary. The parent session then runs the deletions:

```
# Agent task prompt pattern:
"Create X at path Y. Do NOT attempt to delete old directories — 
list any paths that need deletion in your result for the parent session to handle."

# Agent result pattern:
"Created: skill/new-router.md, skill/java.md, skill/python.md
Needs deletion (run in parent): rm -rf old-skill/ other-old-skill/"

# Parent session:
rm -rf old-skill/ other-old-skill/
```

**Why non-obvious:** The permission model isn't documented, and the blocking appears inconsistent — agents can run validators and commit commands but not `rm -rf`. The natural instinct is to include deletions in the agent task (it's part of the same logical operation). Discovering that you need to split create/delete across agent boundary vs. parent boundary requires hitting the block first.

**Related:** See GE-20260414-b12a99 — the "half-done state" gotcha when agents block mid-task on Bash. This technique avoids the problem by never assigning Bash deletions to agents in the first place.
