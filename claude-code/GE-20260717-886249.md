### Symptom
Files created via `ide_create_file` are immediately indexed by IntelliJ — `ide_find_class`, `ide_find_references`, and `ide_search_text` all see them. But `ls`, `cat`, and Maven compilation report the files as non-existent. Maven says "Nothing to compile — all classes are up to date" even though new source files were just created.

### Context
When using the IntelliJ MCP plugin (`mcp__intellij-index__ide_create_file`) to create new `.java` files during Claude Code sessions. The tool's documentation says "The file is created through IntelliJ's VFS, so it is instantly available for ide_find_references" — this is true for IntelliJ's index, but the VFS write does not immediately flush to the physical filesystem.

### Root cause
`ide_create_file` writes to IntelliJ's Virtual File System (VFS) layer, not directly to the filesystem. The VFS is an in-memory abstraction that IntelliJ uses for all file operations. VFS-to-disk synchronization happens asynchronously and may not complete before external tools (Maven, git, shell commands) attempt to read the files.

### Fix
Use the `Write` tool (Claude Code's native file writer) for creating new `.java` files on disk, then call `ide_sync_files` to make IntelliJ aware of the new files:

```
Write tool → creates file on physical filesystem
ide_sync_files(paths: ["path/to/NewFile.java"]) → IntelliJ indexes it
```

For editing existing files, `ide_edit_member`, `ide_replace_member`, and `ide_replace_text_in_file` work correctly — they operate on files already tracked by both the VFS and the filesystem.

### Why this is non-obvious
The `ide_create_file` documentation explicitly says files are "immediately indexed" and suggests using it "instead of the Write tool for creating .java files." The IntelliJ tools confirm the file exists (find_class returns it, search_text finds content in it). Everything looks correct from IntelliJ's perspective — the failure only surfaces when Maven or git tries to access the file on disk.
