## Symptom

A Bash command that is fully covered by an allowlist rule still triggers a permission prompt labelled **"Contains expansion"** — even though the same command without variable expansion would run silently.

Example: the rule `"Bash(git -C * add *)"` is in `~/.claude/settings.json` `permissions.allow`. The command:
```bash
GARDEN=${HORTORA_GARDEN:-~/.hortora/garden}
git -C $GARDEN add domain/GE-id.md
git -C $GARDEN commit -m "..."
git -C $GARDEN push origin main
```
still prompts with "Contains expansion / Do you want to proceed?" despite the allowlist.

## Context

Claude Code has two independent permission gates:

1. **The allowlist gate** — checks if the command matches a `permissions.allow` rule. If it matches, the normal permission prompt is skipped.
2. **The expansion safety check** — fires when the Bash command contains shell variable expansion (`$VAR`, `${VAR}`, `${VAR:-default}`, etc.). This check is **separate** from the allowlist and fires regardless of whether the command is allowlisted.

The allowlist bypasses gate 1. It does not bypass gate 2.

## Root cause

The expansion check is a distinct safety mechanism that runs after allowlist evaluation. It is not documented in the permissions documentation. The only way to discover it is to observe that allowlisted commands with `$VAR` still prompt while identical commands with concrete values do not.

## Fix

Use the **concrete resolved path** in Bash commands instead of inline shell variable expansion. Resolve the variable beforehand (programmatically, not in a Bash block) and emit the git command with the literal path:

```bash
# Prompts even with allowlist:
GARDEN=${HORTORA_GARDEN:-~/.hortora/garden}
git -C $GARDEN add domain/GE-id.md

# Does not prompt (concrete path, no expansion):
git -C /Users/mdproctor/.hortora/garden add domain/GE-id.md
```

For skills and automated workflows, resolve environment variables in the skill instructions step before constructing the Bash command, then emit only the resolved form.

## Why this is non-obvious

The allowlist documentation implies that matching commands are auto-approved. Nothing in the documentation mentions a separate expansion check that is not suppressible via the allowlist. A developer adding `"Bash(git -C * add *)"` to the allowlist and still seeing prompts has no obvious next step — the natural assumption is that the allowlist rule is wrong, not that a separate undocumented mechanism is firing.
