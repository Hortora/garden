## Symptom

Downstream subagents (or spec reviewers) cannot see commits made in a parent subagent's worktree. The worktree branch advances, but the session branch does not. If a downstream task checks out the session branch, all prior worktree changes are invisible.

## Context

Discovered during eidos#24 implementation using subagent-driven-development. Task 4 made commits to `worktree-agent-a965856c045071794`. Task 5 checked out the session branch and found the code missing, then re-added it — creating a duplicate fix with a different commit message.

## Root Cause

The worktree is a separate git working tree with its own detached branch. Commits made inside the worktree are recorded on `worktree-agent-<id>`, not on the session's working branch. The harness does not automatically merge or cherry-pick worktree commits back to the session branch.

## Workaround

After each subagent task in subagent-driven-development, verify the commit landed on the session branch:

```bash
git -C <project> log --oneline origin/main..HEAD
```

The commit count should increase by the expected number. If it doesn't — or if you see the same commits listed as before — the subagent ran in a worktree and the changes are stranded.

To recover: manually cherry-pick from the worktree branch:

```bash
git -C <project> cherry-pick <worktree-branch-sha>
```

## Why This Is Non-Obvious

1. **Worktree creation is automatic** — the Agent tool creates and destroys worktrees invisibly for certain task types.
2. **No error or warning** — commits succeed normally and report success to stdout.
3. **Silent isolation** — the commits exist in git; they just exist on a different branch.
4. **Task boundary obscuration** — when downstream work re-discovers and re-implements the change, the two commits appear unrelated.

## Prevention

Use the verification command after every subagent task in a subagent-driven-development session. This catches the stranded-commit problem before it cascades to downstream tasks.
