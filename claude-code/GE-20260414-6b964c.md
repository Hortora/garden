## Two separate subagents for spec compliance then code quality catch different issues than one combined reviewer

**ID:** GE-20260414-6b964c
**Stack:** Claude Code, subagent-driven development

**Technique:** When reviewing implementation with subagents, dispatch a spec-compliance reviewer first, then a separate code-quality reviewer as a second pass. Keep them independent — the quality reviewer gets no awareness of the spec.

**Why it works:** The two reviewers optimise for different things:
- **Spec reviewer:** looks for requirement coverage gaps and extra scope ("is everything from the spec present? is anything extra present?")
- **Quality reviewer:** looks for internal consistency, structural issues, unclear flows — independent of what was specified

A combined reviewer anchors on the spec and misses structural problems. Two independent reviewers surface both categories.

**In practice:** During implementation of a dual-mode design skill (workspace mode + direct mode), the spec reviewer confirmed all requirements were met. The quality reviewer — starting fresh — caught that Steps 5-6 of the direct-mode workflow were not excluded when workspace mode was active, creating a silent contradiction with the Success Criteria section. The spec reviewer had no reason to look for this because the spec didn't describe the internal step routing.

**Pattern:**
```
Implementer → Spec reviewer (requirement coverage) → Quality reviewer (fresh context)
                    ↓ issues found?                         ↓ issues found?
              Implementer fixes → Spec reviewer re-checks   Implementer fixes → Quality re-checks
```

**Why non-obvious:** The instinct is to combine reviews to save invocations. But the two reviews have different goals — combining them produces a reviewer that does neither well, anchored to the spec throughout.
