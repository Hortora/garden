## --dangerously-skip-permissions bypasses the entire Claude Code permission system, including the deny list

**Stack:** Claude Code
**Symptom:** You add a `.claude/settings.json` with a narrow `permissions.allow` list and an explicit `permissions.deny` list to contain a background agent. You invoke `claude --print --dangerously-skip-permissions "task"` expecting that the allowlist and deny list still apply, and only the confirmation prompts are skipped. In practice, the agent is unconstrained — it can use any tool and run any command.

**Context:** Setting up an autonomous background agent (e.g. a garden dedup agent fired by a git post-commit hook) and trying to use `--dangerously-skip-permissions` to avoid confirmation prompts while keeping the `settings.json` allowlist as a security boundary.

**Root cause:** `--dangerously-skip-permissions` is not a "skip prompts" flag — it bypasses the entire permission system, including `settings.json` allow/deny lists. The `.claude/settings.json` permissions model only operates within the normal interactive permission flow. When that flow is disabled entirely, the settings file has no effect on what the agent can do.

**Fix:** Do not use `--dangerously-skip-permissions` as a security boundary mechanism. Instead, eliminate the need for it by ensuring the agent never generates commands that trigger permission prompts in the first place — specifically, avoid shell expansion syntax (`${VAR:-default}`) in bash commands. Use wrapper scripts or pre-resolved concrete paths.

**Why non-obvious:** The flag name strongly implies "skip the confirmation dialog" rather than "disable all permission enforcement." A developer reading the docs would reasonably assume that `settings.json` deny lists remain active. The name gives no indication that it overrides the entire containment model.
