## Claude Code Agent tool sub-agents do not inherit MCP server connections from the parent session

**Stack:** Claude Code

Calling an MCP tool (e.g. `mcp__intellij-index__ide_index_status`) inside a sub-agent
dispatched via the `Agent` tool returns "tool not found" or "tool unavailable" — even
when the parent session has the MCP server fully connected and the same tool works when
called directly.

### Symptom

Sub-agent returns: "The tool `mcp__intellij-index__ide_index_status` is not available in
this session." Parent session calls the identical tool immediately after and succeeds.

### Context

Occurs when delegating any MCP-dependent operation to a sub-agent via the `Agent` tool,
regardless of the MCP server type (IntelliJ, GitHub, etc.).

### Root cause

Sub-agents spawned via the `Agent` tool run with their own isolated tool registry.
MCP server connections established in the parent session are not inherited. The sub-agent
only has access to built-in tools (Bash, Read, Write, Edit, etc.) and any tools
explicitly provided in its invocation context.

### Fix

Call MCP tools directly from the main session. Never delegate MCP-dependent operations
to sub-agents via the `Agent` tool. If you need to verify an MCP connection, call it
inline — do not dispatch a sub-agent to check.

### Why this is non-obvious

The `Agent` tool description lists `Tools: *` as the sub-agent's capability, which
implies access to all available tools. In practice, `*` refers to built-in tools only —
dynamically-connected MCP servers are a session-level concept that sub-agents don't
participate in.
