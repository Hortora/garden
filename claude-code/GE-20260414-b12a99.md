## Background subagent blocks on Bash leaves file edits uncommitted in worktree

**ID:** GE-20260414-b12a99
**Stack:** Claude Code, background subagents, git worktrees

**Symptom:** A background subagent (`run_in_background=true`) returns BLOCKED after completing its file edit steps. The work appears not done. But the worktree has uncommitted file changes sitting on disk.

**Context:** Subagent was instructed to edit files, run validators, and commit. It completed the edits but hit a Bash permission denial on the validate/commit steps and returned BLOCKED. File edits are already written to the worktree filesystem — uncommitted.

**Root cause:** Background subagents execute file edits (Write, Edit tools) before reaching Bash commands. If Bash is denied, the agent stops but doesn't roll back the file writes. The work is half-done: files changed, nothing committed, git history empty.

**Fix:** After receiving a BLOCKED notification from a background subagent:
```bash
# Check the worktree for uncommitted changes
cd .worktrees/<branch-name>
git diff --stat

# If changes present: validate and commit manually
python3 scripts/validate_all.py --tier commit
git add <files>
git commit -m "<message>"
```

**Why non-obvious:** The BLOCKED status sounds like the task failed. The natural response is to re-dispatch. But the file edits succeeded — re-dispatching without checking would duplicate changes or conflict. Always inspect the worktree before deciding whether to retry.
