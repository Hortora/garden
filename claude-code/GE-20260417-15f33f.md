## AI session handover can describe committed work that was never committed

**ID:** GE-20260417-15f33f
**Stack:** Claude Code, git
**Labels:** `#handover` `#git` `#verification` `#session-continuity`

### Symptom
A HANDOFF.md (or equivalent session handover) describes a feature or fix as complete and committed. `git log` shows no such commit. The code changes exist only in the working tree.

### Context
Occurs when an AI session writes a handover from conversation memory rather than from verified git state. The session implemented the changes, the handover was written while they were still in the working tree, and the commit was never made (e.g. session ended, user closed the terminal, or the commit step was skipped). The handover accurately reflects what was done in the session — but confuses "changes exist" with "changes are committed".

### What was tried
The handover text was taken at face value at the start of the next session. The feature was assumed done. Only a `git log | grep <feature>` check revealed no commit existed.

### Root cause
Session handovers are written from conversation memory, not from git state. A handover that says "implemented X, committed" may reflect the intention or the partial action (implement without commit) rather than the verified outcome.

### Fix
At the start of any session resuming from a handover, run:

```bash
git log --oneline -10  # does the claimed work appear?
git status --short     # is it sitting in the working tree?
```

If the handover claims a feature is committed but `git log` doesn't show it, check `git status` — the work may be in the working tree and just needs to be committed. Do not assume it's lost.

```bash
# If changes are present uncommitted — verify they're correct, then:
git add <files>
git commit -m "feat: <description> (recovered from working tree)"
```

### Why this is non-obvious
The handover language ("implemented and committed X") is natural and reads as confirmed. There is no obvious signal that "committed" means "intended to commit" rather than "git commit was executed". The working tree preserves the code, so the feature isn't lost — but the next session may believe it's safe to build on, not knowing the work needs committing before any destructive operation (reset, checkout) would erase it.
