## Claude Code "Contains expansion" prompt fires for ${VAR:-default} syntax before allowlist matching

**Stack:** Claude Code
**Symptom:** Claude constructs a bash command like:
```
python3 ${SOREDIUM_PATH:-~/claude/hortora/soredium}/scripts/dedupe_scanner.py . --record "pair" distinct
```
Claude Code shows a "Contains expansion / Do you want to proceed? Yes / No" prompt, even though the command matches a pattern in `settings.json` `permissions.allow`. The allowlist doesn't suppress the prompt.

**Context:** Autonomous background agent with pre-approved commands in `.claude/settings.json`. The CLAUDE.md instructs the agent to use `${SOREDIUM_PATH:-...}` syntax to resolve an optional environment variable.

**Root cause:** Claude Code runs an expansion safety check independently of, and before, the allowlist check. Any bash command containing `${VAR:-default}`, `$(command)`, backticks, or similar shell expansion syntax triggers the prompt regardless of what the allowlist says. The two systems do not compose — allowlist approval does not suppress expansion warnings.

**Fix:** Ensure commands passed to the Bash tool never contain shell expansion syntax. Resolve variable paths in a wrapper script (where expansions are fine because the script file — not Claude — runs the expansion), then have the agent call the wrapper with concrete arguments. Simple `$VAR` references (not `${VAR:-default}`) may not trigger the check.

**Why non-obvious:** A developer adding `"Bash(python3 */dedupe_scanner.py *)"` to the allowlist would expect it to cover that command in all forms. The fact that the expansion check is an entirely separate system, consulted first, and that allowlist coverage does not override it, is nowhere documented.
