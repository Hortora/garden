### What

For each piece of content in a CLAUDE.md (or equivalent always-loaded AI context
file), apply this test: **"Is this needed before the agent knows what task it's
doing, or only when it encounters a specific situation?"**

If the latter — extract it to a separate file and leave a one-line trigger in
CLAUDE.md:

```
Before writing any Java code: read `docs/GOTCHAS.md`.
When adding Flyway migrations: read `docs/FLYWAY.md`.
```

The agent that knows where to fetch content is as capable as one that has it
pre-loaded, with far less context overhead.

### Why this is non-obvious

Most developers think about content in terms of *importance* — "is this important
enough to keep?" Almost everything in a well-maintained CLAUDE.md is important.
The useful distinction is *timing*: content needed at session start (before the
task is known) vs content that only matters when a specific trigger fires.

A 60-bullet gotchas list is important — but 55 of those bullets are irrelevant
to any given session. Loading them every time burns context window for nothing.

### Categories that reliably pass the triggered test

- Gotchas / non-obvious behaviours — activated by: "about to implement X"
- Per-file module listings — activated by: "which file is X in?" (IDE MCP answers faster anyway)
- Migration version conventions — activated by: "about to add a migration"
- Build script detail / expected test times — activated by: "running a build"
- Cross-repo / dependency prerequisites — activated by: "working with module Y"

### Categories that belong in CLAUDE.md (always-needed)

- Session start instructions (which command to run, which repo to use)
- Artifact routing (where to write handovers, specs, ADRs)
- Git discipline (two-repo setup, fork model)
- Project type, stack, naming conventions
- Module-level structure (which module owns which domain)
- Active epics and commit behaviour rules
- Ecosystem conventions (version pinning, GitHub Packages, git workflow)
