### Symptom
Setting `"PATH": "/opt/homebrew/bin:${PATH}"` in `~/.claude/settings.json` env section
strips most of the shell PATH. Tools like jenv, sdkman, Google Cloud SDK, and
`~/.local/bin` disappear from the subprocess environment. The `/doctor` command
also starts reporting `~/.local/bin is not in your PATH` after the setting is
added — even when it was fine before.

### Context
Any Claude Code session where `env.PATH` is set in `~/.claude/settings.json` or a
project-level `.claude/settings.json`. The problem affects Bash tool subprocesses
and hook subprocesses. Appears when trying to fix a hook `dirname: command not
found` error by setting PATH explicitly.

### What was tried (didn't work)
- Setting `"PATH": "/opt/homebrew/bin:${PATH}"` — `${PATH}` does not expand;
  the literal string `${PATH}` becomes part of the value
- Setting an explicit PATH list without `~/.local/bin` — strips the native Claude
  Code binary location, causing `/doctor` warnings
- Setting PATH in project-level `.claude/settings.json` to override the global —
  project-level `env.PATH` also replaces rather than merges
- Expecting `/doctor` to clear after adding `~/.local/bin` to settings.json PATH —
  `/doctor` reads the Claude Code **process** PATH set at launch by the shell, not
  the settings.json PATH used for subprocesses

### Root cause
`env` entries in settings.json SET environment variables for subprocesses — they
do not prepend to or merge with the inherited shell PATH. The value is used as-is.
`${PATH}` is not a shell expression in this context; it is a literal string.

The `/doctor` `~/.local/bin` check reads `process.env.PATH` of the Claude Code
process itself, which is set by the shell that launched Claude Code — not by
settings.json. No settings.json change can fix this check; only the shell profile
(`.zshrc`) can.

### Fix
**Remove PATH from settings.json entirely.** Claude Code inherits the shell PATH
when no `env.PATH` is set. This is the correct default.

```json
{
  "env": {
    "MY_VAR": "value"
  }
}
```

For the `/doctor` `~/.local/bin` warning: fix in the shell profile, not settings.json:
```bash
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc
```

If PATH genuinely must be set in settings.json (e.g. to add a tool unavailable
via the shell profile), list every required directory explicitly — there is no
way to reference the inherited PATH value.

### Why this is non-obvious
Every shell profile mechanism (`export PATH="$new:$PATH"`) works by prepending to
an existing value. The settings.json `env` section looks identical in shape to a
shell profile, so the natural assumption is that it prepends. It doesn't — it
replaces. The `/doctor` check looking at a different PATH than the one set by
settings.json compounds the confusion: no amount of settings.json changes will
clear the warning.
