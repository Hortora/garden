## Claude Code settings.json rejects unknown top-level fields despite additionalProperties schema

**ID:** GE-0001
**Stack:** Claude Code CLI, `~/.claude/settings.json`, any version
**Symptom:** Adding a custom top-level key to `settings.json` produces:
`Settings validation failed: Unrecognized field: <your-key>`
No error hints at a workaround. The validator blocks the change.
**Context:** Happens when you try to store custom app config in `settings.json` alongside Claude's own settings.

### What was tried (didn't work)
- Added `"personalWritingStylesPath": "~/path"` at top level → validation error
- Schema inspection showed `"additionalProperties": {}` which normally means extra fields are allowed → still rejected

### Root cause
Claude Code runs its own programmatic validation layer on top of the JSON schema. Despite `additionalProperties: {}` in the published schema (which technically permits extra fields), the runtime validator explicitly rejects any key not in its known list.

### Fix
Use the `env` block instead — it accepts arbitrary string key-value pairs and surfaces them as environment variables in Claude sessions:

```json
{
  "env": {
    "MY_CUSTOM_VAR": "~/my/path"
  }
}
```

Access in skills/hooks via `$MY_CUSTOM_VAR` or `os.environ['MY_CUSTOM_VAR']`.

### Why this is non-obvious
The JSON schema's `additionalProperties: {}` strongly implies custom top-level keys are permitted — it's the standard way to signal extensibility. A developer reading the schema would reasonably conclude their custom key is safe. The runtime rejection with no workaround hint in the error message makes this a dead end without knowing about the `env` escape hatch.
