### Symptom

Nonsense entries like `Bash(MapCaseContext)`, `Bash(evalObjectTemplate)`,
`Bash(CaseContext)` appear in `.claude/settings.local.json` after running
`gh issue close --comment "..."` with backtick-quoted class names in the body.

### Context

Occurs when the `--comment` argument to `gh issue close` (or `gh issue create`)
contains backtick-delimited text, e.g.:

```bash
gh issue close 37 --repo ... --comment "MapCaseContext has no `evalObjectTemplate` override"
```

### Root cause

The shell evaluates backtick content as command substitution before passing
the argument to `gh`. Claude Code records whatever commands execute (even
failed ones) as allowed bash permission patterns. The class names are tried as
shell commands, fail silently, and land in the allowlist.

The `gh` call succeeds and the comment renders correctly on GitHub — the only
trace is the polluted settings file.

### Fix

Use `$()` instead of backticks in the shell (backticks in double-quoted strings
are still evaluated). To pass literal backtick-formatted markdown in a `gh`
comment, use single quotes or a heredoc:

```bash
# Safe — single quotes prevent shell expansion
gh issue close 37 --repo ... --comment 'MapCaseContext has no `evalObjectTemplate` override'

# Or use --body-file with a temp file
echo 'MapCaseContext has no `evalObjectTemplate` override' > /tmp/comment.txt
gh issue close 37 --repo ... --body-file /tmp/comment.txt
```

### Why this is non-obvious

The `gh` command succeeds and GitHub renders the comment with correct
backtick formatting. Nothing fails visibly. The spurious permissions appear
in `settings.local.json` and look like legitimate entries at a glance —
they follow the `Bash(<identifier>)` format used for real allowed commands.
The connection between the class names in the comment and the allowlist
entries is non-obvious without knowing that Claude Code captures every bash
execution during a session.
