### Symptom
After dispatching a subagent to fix test compilation errors, 9 deleted Java source files reappeared in the working tree with modified content. The migration that deleted them was silently undone.

### Context
During a large type migration (engine#543), 9 type files were deleted from `engine-api` and replaced with types from a new dependency. A subagent was then dispatched to fix downstream test compilation errors. The subagent used the Read tool on the deleted files (to understand what needed fixing), which returned the HEAD content. It then used Edit to "fix" those files, recreating them on disk.

### What was tried (didn't work)
- Dispatching a subagent with explicit instructions to "only fix test files, not production source" — the subagent still read and edited the deleted production files because it had no way to know they were supposed to be absent.

### Root cause
The Read tool resolves files from git HEAD when the file doesn't exist on disk but is tracked. A subagent has no awareness of working-tree state changes made by the parent session. When it reads a "deleted" file, it gets the pre-deletion content. When it then edits that content, the file is recreated on disk.

### Fix
Commit deletions before dispatching any subagent that might touch files in the same module. The subagent will then see the committed state (files absent) and cannot recreate them.

```bash
# Delete files
rm old-types/*.java

# Commit the deletion BEFORE dispatching subagents
git add -u && git commit -m "delete: remove migrated types"

# NOW dispatch subagents — they see the committed state
```

### Why this is non-obvious
The failure is completely silent. No error, no warning. The subagent appears to succeed — it "fixes" the files and reports clean compilation. The parent session only discovers the problem when a subsequent build fails with duplicate class errors or reverted imports.
