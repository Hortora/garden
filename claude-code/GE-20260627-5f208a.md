### Symptom
A fresh Sonnet subagent dispatched for Task 2 of devtown#11 reported BLOCKED with 11 compilation errors — `PlannedAction`, `Capability`, `Worker`, `WorkerResult` classes not found. The errors did not exist in the current codebase.

### Context
`.superpowers/sdd/` is git-ignored scratch that survives across sessions. The `task-brief` script creates fresh brief files per task, but report files (`task-N-report.md`) are not cleared between sessions. A previous session (devtown#13, trust feedback loop) had written `task-2-report.md` with a BLOCKED status describing compilation failures from an engine API migration on a different branch.

### What was tried (didn't work)
The first dispatch (Sonnet) read the stale report, accepted it as its own findings, and stopped without writing any code. Only the partial `QueuedPr.java` and `PriorityLane.java` files were created before the agent terminated.

### Root cause
The subagent's prompt said "Write your report to task-2-report.md" — but the file already contained a detailed report from a previous session. The agent read it, found a BLOCKED status with specific errors, and concluded its task was blocked.

### Fix
Clear or overwrite report files before dispatching each task's implementer:
```bash
echo "" > .superpowers/sdd/task-N-report.md
```
Do this in the controller's dispatch flow, after generating the task brief and before launching the subagent.

### Why this is non-obvious
The `.superpowers/sdd/` directory is git-ignored — it doesn't appear in `git status` and isn't cleaned by `git checkout`. The stale file is invisible to the controller. The subagent's report looks legitimate (specific error messages, file paths, a plausible root cause) — nothing signals that the content is from a different session. The only clue is that the errors reference classes and issues (`devtown#13`) unrelated to the current task.
