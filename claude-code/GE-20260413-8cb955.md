## Claude Code hook silently backgrounds all Bash calls regardless of `run_in_background` flag

**ID:** GE-20260413-8cb955
**Stack:** Claude Code (all versions with hooks configured)
**Symptom:** Every Bash tool call runs in the background — output files are 0 bytes, commands that should block don't, and setting `run_in_background: false` has no effect. Long-running tests spawn multiple concurrent instances because Claude doesn't wait for completion before retrying.
**Context:** When a `PreToolUse` or similar hook is configured in `settings.json` that intercepts Bash tool calls. The hook itself may background processes regardless of the Bash tool's `run_in_background` parameter.

### What was tried (didn't work)
- Set `run_in_background: false` explicitly on Bash calls — ignored
- Used `timeout` parameter — commands still background immediately
- Read output files — all 0 bytes immediately after command "completes"
- Ran `pytest` — spawned 4 concurrent instances, all sharing state files, producing inconsistent results (62 failed vs 117 failed vs 98 failed across runs)

### Root cause
A configured hook intercepted the Bash tool and forced background execution unconditionally. The hook's implementation overrides the tool's own `run_in_background` parameter. The tool "completes" from Claude's perspective immediately, writing a background task ID — but the actual process runs asynchronously. Output file paths are returned but files are empty until the process finishes.

### Fix
Inspect `settings.json` hooks. If a `PreToolUse` hook is backgrounding commands, either:
- Remove the hook if it's not needed for the current workflow
- Modify the hook to respect the tool's `run_in_background` parameter
- Work with background execution: use the task notification system and read output files only after the notification fires

When hooks are active, treat all Bash calls as potentially backgrounded — never retry on empty output; wait for the task-completion notification instead.

### Why this is non-obvious
The `run_in_background: false` parameter reads as an authoritative instruction. There is no warning when a hook overrides it. The output file path is returned immediately, making it look like the command completed. The symptom (empty output) is identical to "command produced no output". Multiple retries exacerbate the problem by spawning additional background instances.
