### Symptom

Edit tool returns:
```
Refusing to write through symlink: /path/to/file.md.
Resolve the symlink and pass the real target path explicitly.
```

### Context

When any file targeted by Edit/Write is a symlink — as in the casehubio workspace/project repo pattern where `~/claude/public/casehub/<repo>/CLAUDE.md` symlinks to `~/claude/casehub/<repo>/CLAUDE.md` — the Edit tool refuses to write to the symlink path.

### What was tried

Using the symlink path directly in the Edit tool call. The error message says to resolve the symlink but does not say how.

### Root cause

Claude Code's Edit tool deliberately rejects writes through symlinks to prevent accidental writes to unintended locations. The check is applied to the path as provided — if it resolves to a different inode, the write is rejected.

### Fix

Resolve the real target path first with `readlink -f`, then use that path in all subsequent Edit/Write calls:

```bash
readlink -f /path/to/symlink/file.md
# → /real/path/to/file.md
```

Use the resolved path for every Edit/Write call in the session. The symlink path will continue to fail.

### Why this is non-obvious

The error message says "resolve the symlink" but does not mention `readlink -f` or that this must be a separate shell step. Users typically look for an alternative write mechanism rather than resolving first. The pattern of workspace files being symlinks to project repo files is not documented, so the failure is unexpected.
