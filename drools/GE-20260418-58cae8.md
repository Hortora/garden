### Symptom

A fact (unit, entity) gets classified into two mutually-exclusive groups in the same rule session. Both a high-salience rule (e.g., "Group: blinking" at salience 205) and a lower-salience rule (e.g., "Group: kiting" at salience 200) fire for the same fact, producing two conflicting actions dispatched in the same tick.

No error or warning — both rules fire silently.

### Context

Occurs in Drools Rule Units when using a salience-based classification pattern where groups are *intended* to be mutually exclusive (higher salience wins) but no explicit mutual-exclusion guards are written.

In practice: a Stalker unit with critically low shields AND on weapon cooldown matched both the "blinking" rule (shields-low + blink-ready) and the "kiting" rule (in-range + on-cooldown). Both fired. The unit received both a `BlinkIntent` and a `MoveIntent`, with the last-written value winning non-deterministically.

### Root cause

Drools salience determines the **order** in which matching rules fire within a single session — it does not prevent lower-salience rules from firing once a higher-salience rule has already matched the same fact. All rules whose `when` conditions are satisfied fire, ordered by salience.

"Higher salience takes priority" means "fires first this session," not "blocks other rules from firing."

### Fix

Add explicit `not` guards to lower-salience rules to prevent them firing when the higher-priority condition is already true:

```drl
rule "Group: kiting"
    salience 200
when
    eval(strategyGoal.equals("ATTACK"))
    $u: /army[ (double) this.health() / this.maxHealth() >= 0.3 ]
    /inRangeTags[ this == $u.tag() ]
    /onCooldownTags[ this == $u.tag() ]
    not /shieldsLowTags[ this == $u.tag() ]  // ← guard: don't fire if blinking takes priority
then
    groupDecisions.add("kiting:KITING:" + $u.tag());
    activeGroups.add("kiting");
end
```

Guard on the condition that defines the higher-priority group (e.g., `shieldsLowTags`), not just on the higher-priority group's activation (which would require reading `activeGroups` mid-session).

### Why this is non-obvious

The mental model of "higher salience = takes priority" naturally implies mutual exclusion — this is how priority works in most rule systems. In Drools, salience is purely an ordering mechanism. If no `not` guard prevents a lower rule from matching, it will fire after the higher-priority rule, in the same session. The bug produces no error and the second intent quietly overwrites the first.

### Alternatives considered

- **Use a single rule with a complex `when` clause** — avoids the problem but becomes hard to read and maintain as group conditions grow.
- **Read `activeGroups` in the lower rule's `when`** — would work conceptually, but `activeGroups` is a DataStore written in Phase 1 `then` blocks; Drools reactive evaluation within a single `fire()` call may not see these writes reliably in the `when` of a same-phase rule.
- **Restructure into a single-match rule with accumulate** — overkill for classification patterns.

### Invalidation triggers

Would change if Drools adds a native "once-matched" or "exclusive salience group" mechanism. Check Drools release notes when upgrading major versions.
