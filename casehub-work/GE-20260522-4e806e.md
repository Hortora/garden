## Symptom

WorkItems past `expiresAt` are never marked EXPIRED, no audit entry is written, no `SlaBreachEvent` fires, and the item reappears in every subsequent `ExpiryCleanupJob` tick. No exception appears in logs because the `@Transactional` rollback swallows it.

## Context

`ExpiryLifecycleService.checkExpired()` is `@Transactional` and loops over all expired WorkItems. `executeBreachDecision()` throws a private `BreachExecutionFailed` for `EscalateTo(∅)`. The `Chained` branch catches it — but only `Chained` does:

```java
// checkExpired() — BreachExecutionFailed is NOT caught here:
final BreachDecision leaf = executeBreachDecision(item, slaBreachPolicy.onBreach(ctx), ctx, now);
```

If a `SlaBreachPolicy` implementation returns `EscalateTo.to()` (no args) without wrapping in `Chained`, `BreachExecutionFailed` propagates out of the loop, rolls back the entire `@Transactional` method, and every WorkItem processed in that tick is un-done. On the next scheduler tick, the same items are scanned again — infinite loop.

## Root cause

`BreachExecutionFailed` is only caught by the `Chained` case inside `executeBreachDecision`. The top-level dispatch in `checkExpired()` and `checkClaimDeadlines()` has no catch for it. A policy returning bare `EscalateTo.to()` (zero groups) is a policy bug, but the runtime consequence — transaction rollback + infinite retry — is disproportionate and silent.

## Fix

Two defences, both should be applied:

**Fix A — fail fast at factory (prevents the bug at authoring time):**
```java
public static EscalateTo to(String... groups) {
    if (groups.length == 0)
        throw new IllegalArgumentException("EscalateTo requires at least one group");
    return new EscalateTo(Set.of(groups), null);
}
```

**Fix B — safe fallback at executor (prevents transaction rollback even if constructor bypass occurs):**
```java
case BreachDecision.EscalateTo escalate -> {
    if (escalate.groups().isEmpty()) {
        LOG.errorf("Policy returned EscalateTo with empty groups for %s — applying Fail", item.id);
        yield executeFail(item, new BreachDecision.Fail("escalation-misconfigured"), now);
    }
    yield executeEscalateTo(item, escalate, ctx, now);
}
```

## Why this is non-obvious

The failure is completely silent: no exception log (the transaction rolls back), no status change on the WorkItem, and the item reappears every tick. The only observable symptom is that items stay past `expiresAt` indefinitely and the scheduler job runs without visible effect.
