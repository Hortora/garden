### The technique

When a WorkItem lifecycle service has status guards designed for human-actor transitions (e.g. `complete()` requires `IN_PROGRESS`), add dedicated `completeFromSystem()` / `rejectFromSystem()` methods that accept any non-terminal status. These methods go through the same audit and event-firing path, making them transparent to observers.

```java
/**
 * Complete a WorkItem from a system actor, accepting any non-terminal status.
 * Used when group policy triggers parent completion regardless of human workflow state.
 */
@Transactional
public WorkItem completeFromSystem(final UUID id, final String actorId, final String resolution) {
    final WorkItem item = requireWorkItem(id);
    if (item.status.isTerminal()) return item; // idempotent — already done
    item.status = WorkItemStatus.COMPLETED;
    item.completedAt = Instant.now();
    item.resolution = resolution;
    final WorkItem saved = workItemStore.put(item);
    audit(saved.id, "COMPLETED", actorId, null);
    lifecycleEvent.fire(WorkItemLifecycleEvent.of("COMPLETED", saved, actorId, resolution));
    return saved;
}
```

### Why this is non-obvious

The standard reflex when system code needs to bypass a guard is to add a boolean flag (`force = true`) to the existing method, or to directly set the status on the entity without going through the service. Both approaches are worse:
- `force` flag pollutes the existing method's contract and leaks through the API
- Direct entity mutation bypasses audit logging and lifecycle event firing, breaking observers

Separate system methods keep the human-actor path clean, are self-documenting about intent, and route through the full audit/event pipeline. The `isTerminal()` guard makes them idempotent — safe to call even if a concurrent transition already completed.

### When to use

- A system coordinator (e.g. multi-instance group policy) needs to transition a parent WorkItem that may be `PENDING` (never claimed by a human) rather than `IN_PROGRESS`
- A scheduled job or background process needs to close WorkItems regardless of their current non-terminal state
- Any case where "system actor" semantics differ from "human actor" semantics but the audit trail should be identical
