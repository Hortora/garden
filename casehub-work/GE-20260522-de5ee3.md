## Technique

When designing a sealed decision type with optional chaining (primary decision, fallback if primary fails), model the chaining as a separate `Chained` record rather than embedding `thenOnBreach` on each decision variant.

**With embedded chaining (harder to understand, forces null on every record):**
```java
public sealed interface BreachDecision {
    record Fail(String reason) implements BreachDecision {}
    record EscalateTo(Set<String> groups, Duration deadline,
                      BreachDecision thenOnBreach) implements BreachDecision {} // null when no chain
    record Extend(Duration by, BreachDecision thenOnBreach) implements BreachDecision {}
}
```

**With Chained as a separate type (each record is pure, composition is explicit):**
```java
public sealed interface BreachDecision {
    record Fail(String reason) implements BreachDecision {}
    record EscalateTo(Set<String> groups, @Nullable Duration deadline) implements BreachDecision {}
    record Extend(Duration by) implements BreachDecision {}
    record Chained(BreachDecision primary, BreachDecision fallback) implements BreachDecision {}

    default BreachDecision thenOnBreach(BreachDecision fallback) {
        return new Chained(this, fallback);
    }
}
```

The executor handles `Chained` in the switch:
```java
case BreachDecision.Chained chained -> {
    try {
        yield executeBreachDecision(item, chained.primary(), ctx, now);
    } catch (BreachExecutionFailed e) {
        yield executeBreachDecision(item, chained.fallback(), ctx, now);
    }
}
```

## Why this is non-obvious

The intuitive design embeds `thenOnBreach` on `EscalateTo` and `Extend` because "what happens next" seems like a natural property of those decisions. But this makes every decision record carry a nullable continuation, forces the executor to check for chaining on each case, and makes the API harder to pattern-match exhaustively.

`Chained` as a separate record makes composition explicit (`EscalateTo.to("x").thenOnBreach(Fail("y"))`), keeps individual records pure (no nullable continuations), and makes the executor's switch arms clean — chaining is handled in exactly one place.

## When to use

- Sealed decision hierarchies where fallback execution is an orthogonal concern
- Any SPI where callers compose decisions rather than embedding fallback logic in decision types
- Cases where the executor needs to distinguish "this decision ran" from "this decision with chain ran"
