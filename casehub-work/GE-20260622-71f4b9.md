## Context

When `SlaBreachPolicy.onBreach()` returns `Exhausted("IND reporting deadline exhausted")`, the runtime transitions the WorkItem to `WorkItemStatus.ESCALATED` (terminal). An `@ObservesAsync WorkItemLifecycleEvent` observer fires and a `@ConsumeEvent("ESCALATED")` handler fires for this transition.

## Symptom

`event.detail()` returns `null` on the ESCALATED `WorkItemLifecycleEvent`. The `Exhausted(reason)` string is not accessible from the event.

## Root Cause

`ExpiryLifecycleService.executeExhausted()` writes the reason to the audit log via `writeAudit(item, "ESCALATED", reason, now)` but calls `fireLifecycleEvent("ESCALATED", item)` separately. The `fireLifecycleEvent` helper always passes `null` as the detail arg:

```java
private void fireLifecycleEvent(final String event, final WorkItem item) {
    if (lifecycleEvent != null) {
        lifecycleEvent.fire(WorkItemLifecycleEvent.of(event, item, "system", null));
        //                                                                   ^^^^ hardcoded null
    }
}
```

The reason is written to `AuditEntry.detail` in `writeAudit()` — it exists in the audit trail but not in the CDI event.

## Fix

Do NOT attempt to read `event.detail()` for the breach reason in `WorkItemLifecycleEvent` observers. Options:

**1. Use a fixed string** (simplest, most common case): the observer knows which WorkItems it handles and can use a fixed reason string in ledger entries:

```java
entry.breachReason = "IND reporting deadline exhausted without submission";
```

**2. Query the audit log** if the exact reason matters (adds a DB read):

```java
// AuditEntry.find("workItemId = ?1 and event = ?2", workItemId, "ESCALATED")
```

**3. Pass metadata in WorkItem.resolution** via a custom `SlaBreachPolicy` that sets `item.resolution` before `executeExhausted()` is called — but this requires overriding internal behaviour.

## Why this is non-obvious

The `Exhausted` record has a `reason()` accessor and the Javadoc says "records <reason> as the resolution" — implying the reason is persisted on the WorkItem. It is written to the audit log, not to `WorkItem.resolution` (unlike `executeFail()` which does `item.resolution = fail.reason()`). The lifecycle event is the natural place to read runtime metadata, and `event.detail()` looks like the right field.
