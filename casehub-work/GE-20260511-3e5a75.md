**Deprecated (EscalationPolicy):** `EscalationPolicy` is `@Deprecated` as of casehub-work#212 (2026-05-22). The interface is replaced by `SlaBreachPolicy` which returns a `BreachDecision` instead of void. See the **Alternative — SlaBreachPolicy pattern** section below for the migration path. `EscalationPolicy` will be removed in work#215.

### The technique
When implementing SLA-backed workflows with casehub-work, the domain service sets exactly two fields on the WorkItem that encode the compliance policy:

1. `claimDeadline` — the absolute instant the SLA expires (e.g. `reportedAt + grade.sla()`)
2. `candidateGroups` — the logical routing pool (e.g. `"dsmb,safety-officers"`)

Everything that happens when the SLA is **missed** (escalation target, notification channel, second WorkItem spawning) is `EscalationPolicy` SPI configuration — not domain code.

### Why developers do it differently
The `WorkItemLifecycleEvent` CDI event API makes it easy to observe EXPIRED status and react directly in the domain service. Nothing in the API surface guides you toward the `EscalationPolicy` SPI pattern. The natural impulse is to wire escalation in the same service that creates the WorkItem.

### Why this approach is better
- The same domain code works across deployments: a clinical trial might escalate to a DSMB Slack channel; the same trial on another platform might page an on-call SMS. Domain code changes nothing.
- Different WorkItem categories can carry different escalation configs with zero code changes — purely deployment configuration.
- Avoids casehub-connectors dependency in domain services.
- Makes escalation behavior testable and swappable independently of domain logic.

### Minimal example (EscalationPolicy — deprecated)
```java
workItemService.create(new WorkItemCreateRequest(
    "Adverse Event — " + ae.grade.label(),
    description,
    "adverse-event",        // category — EscalationPolicy SPI uses this to find the right policy
    "adverse-event-review", // formKey
    WorkItemPriority.URGENT,
    null,
    "dsmb,safety-officers", // candidateGroups — logical routing pool; deployer maps to concrete escalation
    null, null, "system",
    payload,
    ae.slaDeadline,         // claimDeadline — the SLA deadline; EscalationPolicy fires when this passes
    null, null, null, null, null, null, null
));
// No WorkItemLifecycleEvent observer needed. No connectors wired. EscalationPolicy SPI handles breach.
```

### Alternative — SlaBreachPolicy pattern (current, replaces EscalationPolicy)

`SlaBreachPolicy.onBreach(SlaBreachContext) → BreachDecision` separates the decision from
its execution. The runtime executes the returned decision (Fail / EscalateTo / Extend / Chained)
and fires a `SlaBreachEvent` CDI event for side-effect observers.

**Stateless two-tier escalation — the key insight:** detect the current tier via `candidateGroups`,
not serialized state. When the escalated WorkItem expires, the policy is called again; the presence
of the escalation group in `candidateGroups` tells the policy it is already at the second tier.

```java
@ApplicationScoped
public class AdverseEventSlaBreachPolicy implements SlaBreachPolicy {

    @Inject PreferenceProvider prefs;

    @Override
    public BreachDecision onBreach(SlaBreachContext ctx) {
        String escalationGroup = prefs.resolve(SettingsScope.of(ctx.scope()))
                .getOrDefault(SlaKeys.ESCALATION_GROUP).value();

        // Already at escalation tier → terminal
        if (ctx.task().candidateGroups().contains(escalationGroup)) {
            return new Fail("adverse-event-sla-exhausted");
        }

        // First breach → escalate with deadline from preferences
        Duration deadline = Duration.ofHours(prefs.resolve(SettingsScope.of(ctx.scope()))
                .getOrDefault(SlaKeys.ESCALATION_HOURS).value());
        return EscalateTo.to(escalationGroup).withDeadline(deadline);
    }
}
```

**Why stateless works:** `EscalateTo` updates `candidateGroups` on the WorkItem. When the item
expires again, `ctx.task().candidateGroups()` reflects the new groups — no serialized decision
tree, no database lookup.

**Migration from EscalationPolicy:** replace the `@ApplicationScoped EscalationPolicy` bean with
an `@ApplicationScoped SlaBreachPolicy` bean. Remove the `@ExpiryEscalation` / `@ClaimEscalation`
qualifiers — `SlaBreachPolicy` uses no qualifiers.
