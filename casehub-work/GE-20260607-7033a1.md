When a Panache query only needs entity IDs (or any single column), avoid loading full entity rows by using `SomeEntity.getEntityManager().createQuery(..., UUID.class)`. This bypasses Panache's entity materialisation and returns a typed `List<UUID>` directly.

### Context

Panache's `find(...).list()` always materialises full entities. For bulk ID queries (e.g. a scheduler fetching eligible IDs to process), this loads all columns for every row — wasteful when only the ID is needed.

### Technique

```java
// Before — loads full entities, maps to IDs in memory
@Transactional
List<UUID> findEligibleIds(Instant now, int limit) {
    return SponsorNotification
            .<SponsorNotification>find(
                    "status in (?1, ?2) and (nextRetryAfter is null or nextRetryAfter <= ?3)",
                    PENDING, FAILED, now)
            .page(0, limit)
            .list()
            .stream()
            .map(n -> n.id)
            .collect(Collectors.toList());
}

// After — JPQL projection, fetches only the id column
@Transactional
List<UUID> findEligibleIds(Instant now, int limit) {
    return SponsorNotification.getEntityManager()
            .createQuery(
                    "SELECT n.id FROM SponsorNotification n"
                    + " WHERE n.status IN (:s1, :s2)"
                    + " AND (n.nextRetryAfter IS NULL OR n.nextRetryAfter <= :now)",
                    UUID.class)
            .setParameter("s1", PENDING)
            .setParameter("s2", FAILED)
            .setParameter("now", now)
            .setMaxResults(limit)
            .getResultList();
}
```

### Why this is non-obvious

Panache's fluent API (`find`, `list`, `stream`) are the natural entry points. `getEntityManager()` is available as a static method on every Active Record entity but is not visible from Panache's fluent API surface. Most developers reach for `.list()` + stream mapping without knowing the JPA escape hatch exists.
