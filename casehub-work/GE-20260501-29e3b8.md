### Symptom
Intermittent `VerificationException: Expected at most 0 calls but received 1` or
`Expected exactly 1 call but received 2` in WireMock assertions. Failures are
non-deterministic and depend on test execution order.

### Context
`@QuarkusTest` reuses a single Quarkus instance across all tests in a class.
Notification rules created via REST API in one test persist in the H2 in-memory
database for the lifetime of the test run. WireMock is restarted `@BeforeEach`
with `dynamicPort()`.

### Root cause
Two compounding issues:

1. **Rule accumulation**: each test creates notification rules via `POST
   /workitem-notification-rules` without cleanup. Rules from test N are still
   active when test N+1 runs.

2. **Port reuse**: WireMock's dynamic port selection draws from the ephemeral
   range. In a fast test run, the OS may reassign a recently-freed port. A
   leftover rule targeting `http://localhost:PORT/hook` from a previous test
   suddenly has a live WireMock listening on that exact port in the current test.
   Deliveries intended for "no one" arrive at the current test's WireMock,
   inflating the verified call count.

`Thread.sleep()` as a synchronization mechanism compounds both issues: if the
sleep is shorter than async delivery, calls from old rules arrive after the
assertion window.

### Fix

**Cleanup**: add `@AfterEach @Transactional` to delete all persisted rules (and
any entities with FK dependencies first):

```java
@AfterEach
@Transactional
void cleanup() {
    wireMock.stop();
    WorkItemNotificationRule.deleteAll();
    AuditEntry.deleteAll();   // FK → work_item
    WorkItem.deleteAll();
}
```

**Synchronization**: replace `Thread.sleep(N)` with Awaitility:

```java
// Instead of: Thread.sleep(500);
await().atMost(5, SECONDS).untilAsserted(() ->
    wireMock.verify(postRequestedFor(urlEqualTo("/hook"))...));
```

Note: `@TestTransaction` cannot be used in test classes that mix direct service
calls with REST Assured assertions — the REST call runs in its own transaction
and cannot see uncommitted test data.

### Why this is non-obvious
The failure looks like a timing issue (delivery arrived after assertion) or a
logic bug in the notification filter. The port-reuse mechanism is invisible — the
old rule in the database has a URL that happens to be live again, with no
indication in any log. Without cleaning up rules, adding more tests makes
existing tests more likely to fail in a way that appears random. The interaction
between H2 persistence across test boundaries and WireMock's dynamic port
selection is not documented and requires knowing both independently.
