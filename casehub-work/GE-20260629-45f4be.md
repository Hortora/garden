### Symptom
A polling loop that checks WorkItem status via `WorkItemCreator.findByCallerRef()` enters an infinite cycle when the WorkItem reaches REJECTED status. Every poll returns the same REJECTED WorkItem, the caller produces a fault event every cycle, and the provisioner is never called again.

### Context
Any workflow that uses callerRef-based WorkItem lookup in a reconciliation or polling loop — e.g., PendingApprovalHandler.check() in casehub-desiredstate. The pattern: create a WorkItem with a callerRef, poll for its completion, act on the result. The expectation: after acting on a terminal result, the callerRef is "free" for future use.

### What was tried (didn't work)
Returning the rejection as a fault event and continuing to the next cycle. The REJECTED WorkItem persists at the callerRef, so `findByCallerRef()` returns it again, producing another fault event — an infinite loop.

### Root cause
`findByCallerRef()` returns the most recent WorkItem matching the callerRef regardless of status. Terminal states (REJECTED, EXPIRED, CANCELLED) persist indefinitely. There is no automatic cleanup or TTL on terminal WorkItems. The callerRef acts as a permanent reservation once a WorkItem is created.

### Fix
After detecting a REJECTED WorkItem and producing the fault event, explicitly transition the WorkItem to OBSOLETE via `WorkItemCreator.obsoleteByCallerRef(callerRef)`. OBSOLETE is a post-terminal state that signals "superseded/no longer relevant." On the next poll cycle, `findByCallerRef()` returns the OBSOLETE WorkItem, which the caller maps to "no pending approval" (same as no WorkItem found) — the callerRef is effectively freed.

### Why this is non-obvious
Terminal WorkItem states (COMPLETED, REJECTED, EXPIRED, CANCELLED) feel like "done" — you'd expect them to not interfere with future workflows. But `findByCallerRef()` returns ANY WorkItem with that callerRef, including terminal ones. The distinction between "terminal" and "invisible" is not documented. OBSOLETE is the only status that semantically means "ignore me."

### Alternatives considered
- **Encoding a cycle counter in the callerRef** (e.g., `desiredstate-approval:tenant:node:provision:2`) — works but makes callerRef unpredictable and breaks the idempotency pattern.
- **Using findActiveByCallerRef() only** — misses terminal states entirely, so you never detect approval or rejection.
