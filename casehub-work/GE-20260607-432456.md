In a `@ExtendWith(MockitoExtension.class)` test class, a `when(mock.method()).thenReturn(x)` stub in `@BeforeEach` causes `UnnecessaryStubbingException` on the specific test(s) that override it with their own stub. The error appears on the `@BeforeEach` stub, not the overriding stub — making it look like the shared setup is the problem.

### Symptom

```
org.mockito.exceptions.misusing.UnnecessaryStubbingException:
Please remove unnecessary stubbings or use 'lenient' strictness.
```

Fails only on tests that re-stub the same method. Tests that use the `@BeforeEach` stub without overriding it pass.

### Context

Common pattern: a `@BeforeEach` stubs a method to a "happy path" default (e.g. `findEligibleIds → list of IDs`). One test wants a different return value (e.g. empty list) and re-stubs it. Mockito strict mode sees the `@BeforeEach` stub as unused (because the test's own stub takes precedence) and flags it as unnecessary.

### Root cause

Mockito strict stubbing validates that every stub is exercised by the test. When a test re-stubs a method, the `@BeforeEach` stub for that method is never invoked in that test, so it's flagged as unnecessary. The test that provides the override causes the error on the shared stub, not on itself.

### Fix

Move the `@BeforeEach` stub into each test that needs it:

```java
// Bad — @BeforeEach stub fails for the test that re-stubs
@BeforeEach
void setUp() {
    when(store.findEligibleIds(any(), anyInt())).thenReturn(List.of(ID_1, ID_2, ID_3));
}

@Test
void does_nothing_when_empty() {
    when(store.findEligibleIds(any(), anyInt())).thenReturn(List.of()); // causes setUp stub to be flagged
    ...
}

// Good — stub lives in each test that needs it
@Test
void calls_all_eligible() {
    when(store.findEligibleIds(any(), anyInt())).thenReturn(List.of(ID_1, ID_2, ID_3));
    ...
}

@Test
void does_nothing_when_empty() {
    when(store.findEligibleIds(any(), anyInt())).thenReturn(List.of());
    ...
}
```

### Why this is non-obvious

The error points to the `@BeforeEach` stub (the shared one), not the overriding stub (the "culprit"). Developers look for the problem in `setUp()`, not in the individual test that triggered it. The fix — moving the stub into each test — feels like more code, not a correction.
