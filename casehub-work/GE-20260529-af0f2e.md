## Context

When implementing safety officer notification for serious adverse events, the natural approach is to observe `WorkItemLifecycleEvent` with `status = PENDING` and `category = "adverse-event"` — triggering when the WorkItem is created. The casehub-work issue documentation describes this pattern.

## Symptom

For Grade 4/5 AEs, the notification fires twice per AE instead of once.

## Root Cause

The AE escalation engine case (`ae-escalation.yaml`) defines two `humanTask` bindings for Grade 4/5:
1. `safety-review` — creates a WorkItem for `candidateGroups: [senior-safety-monitors]`
2. `dsmb-escalation` — creates a WorkItem for `candidateGroups: [dsmb]`

Both fire `WorkItemLifecycleEvent` with `status = PENDING` when created. If a notification observer filters on `category = "adverse-event"` (or any other common field), it fires for both WorkItems.

Grade 3 cases only create one WorkItem (safety-review only), so the issue is not visible during initial development if only Grade 3 is tested.

## Fix

Observe `AdverseEventReportedEvent` (a domain CDI event fired by `AdverseEventService`) instead of `WorkItemLifecycleEvent`. It fires exactly once per AE at report time, before any engine WorkItems are created. For Grade 3+ it always fires exactly once; Grade 1/2 AEs do not fire it (they use the direct WorkItem path).

```java
// ❌ Wrong — fires twice for Grade 4/5
public void onWorkItem(@ObservesAsync WorkItemLifecycleEvent event) {
    if (!"adverse-event".equals(event.category())) return;
    // notify... ← fires for both senior-safety-monitor AND dsmb WorkItems
}

// ✅ Correct — fires once per AE, Grade 3+ only
public void onAeReported(@ObservesAsync AdverseEventReportedEvent event) {
    // notify... ← fires exactly once per AE at report time
}
```

## Why This Is Non-Obvious

The issue documentation says to observe `WorkItemLifecycleEvent` for `status = PENDING`. This is correct for single-WorkItem cases. The multi-WorkItem nature of Grade 4/5 engine cases is only visible by reading the YAML case definition — there is no API surface that advertises "this domain event will create N WorkItems". The bug is also grade-gated: Grade 3 testing passes cleanly, but Grade 4/5 would send duplicate notifications to safety officers.
