## Symptom

Test boot fails with a Hibernate "entity not found" or schema error referencing `FilterRule` after adding `casehub-work` as a Quarkus runtime dependency.

## Context

Occurs when consuming `casehub-work` in a Quarkus app that uses explicit `quarkus.hibernate-orm.packages` configuration. If you add `io.casehub.work.runtime.model` (where `WorkItem` lives) but omit `io.casehub.work.runtime.filter`, the `FilterRule` entity is not scanned and Hibernate fails.

## What was tried

Added `io.casehub.work.runtime.model` to `quarkus.hibernate-orm.packages` — covers `WorkItem`, `WorkItemLabel`, etc. Tests failed with `FilterRule entity not found`.

## Root cause

`casehub-work` ships `FilterRule` in `io.casehub.work.runtime.filter` — a separate package from `io.casehub.work.runtime.model`. Both packages contain JPA entities. With explicit package scanning, Quarkus only scans the listed packages; `runtime.filter` is not documented as required.

## Fix

Add `io.casehub.work.runtime.filter` to both `application.properties` and `src/test/resources/application.properties`:

```properties
quarkus.hibernate-orm.packages=io.casehub.work.runtime.model,io.casehub.work.runtime.filter,io.casehub.aml.domain
```

## Why this is non-obvious

The error message names `FilterRule` but gives no hint that a second package needs to be scanned. Developers naturally add only `io.casehub.work.runtime.model` (the obvious home for WorkItem entities) and miss `runtime.filter` entirely. The casehub-work documentation and integration guide don't mention the `filter` package as a required Hibernate scan target.
