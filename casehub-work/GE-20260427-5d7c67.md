## Context

Adding `quarkus-work` (the full runtime module, not just `quarkus-work-api` or `quarkus-work-core`) as a dependency to a Quarkus module that already has its own `WorkloadProvider` implementation (e.g., `CasehubWorkloadProvider` from the casehub-engine module).

## Symptom

```
jakarta.enterprise.inject.AmbiguousResolutionException: Ambiguous dependencies
  for type io.quarkiverse.work.api.WorkloadProvider and qualifiers [@Default]
```

Application fails to start. Both beans are `@ApplicationScoped` without `@Alternative`, so CDI cannot resolve which to inject.

## Root Cause

`quarkus-work` (the full JAR) includes `JpaWorkloadProvider implements WorkloadProvider` as an `@ApplicationScoped` bean. It is not `@Alternative`. Any module that also provides an `@ApplicationScoped WorkloadProvider` will produce an ambiguous dependency when both JARs are on the classpath.

`quarkus-work-api` and `quarkus-work-core` do NOT include `JpaWorkloadProvider` — only the full `quarkus-work` artifact does. The ambiguity only appears when you need the full module (e.g., to get `WorkItemLifecycleEvent`).

## Solution 1 — @Alternative @Priority(1) (when another WorkloadProvider bean exists but clashes)

**Approach:** Add a higher-priority stub that wins over both `JpaWorkloadProvider` and the existing production bean.

**Pros:** Works when there IS a production WorkloadProvider present (e.g., `CasehubWorkloadProvider`) — overrides both.

**Cons/trade-offs:** Per-test-class setup; `@Alternative @Priority(1)` requires explicit CDI priority ordering.

```java
@QuarkusTest
class MyAdapterTest {

    @Alternative
    @Priority(1)
    @ApplicationScoped
    static class StubWorkloadProvider implements WorkloadProvider {
        @Override
        public int getActiveWorkCount(String workerId) { return 0; }
    }
    // ...
}
```

For production deployments combining the two, the owning module should declare its `WorkloadProvider` as `@Alternative @Priority` and configure selection explicitly.

## Solution 2 — @DefaultBean (when the engine bridge is absent — post engine#378)

**Approach:** Add a `@DefaultBean @ApplicationScoped` stub in `src/test/java` that fills the gap when no other bean exists.

**Pros:** Shared across all test classes in the module; `@DefaultBean` is semantically correct when filling a missing bean, not overriding a competing one; cleaner than a per-test static inner class.

**Cons/trade-offs:** Only works when no other active `WorkloadProvider` bean is present. If a production bean is also indexed, CDI ambiguity returns — use Solution 1 in that case.

```java
package your.app.support;

import io.casehub.work.api.WorkloadProvider;
import io.quarkus.arc.DefaultBean;
import jakarta.enterprise.context.ApplicationScoped;

@ApplicationScoped
@DefaultBean
public class StubWorkloadProvider implements WorkloadProvider {
    @Override
    public int getActiveWorkCount(String workerId) { return 0; }
}
```

Applicable when the engine no longer provides a `WorkloadProvider` implementation (engine#378 deleted `CasehubWorkloadProvider`) and `JpaWorkloadProvider` is excluded via `quarkus.arc.exclude-types`. Without a stub, CDI startup fails with `UnsatisfiedResolutionException` — not `AmbiguousResolutionException` — because there are now zero candidates, not two.

## Why This Is Non-Obvious

The error appears in a module that has worked fine without `quarkus-work` on the classpath. Adding the dependency for an unrelated reason (needing `WorkItemLifecycleEvent`) unexpectedly breaks CDI startup. There is no documentation warning that `quarkus-work` ships a non-`@Alternative` `WorkloadProvider` that will compete with any custom implementation.

## Update — 2026-06-14

**Diagnostic note:** When the "unsatisfied dependency" failure mode appears (zero WorkloadProvider candidates after excluding `JpaWorkloadProvider`), the injection that fails is `WorkItemAssignmentService`'s constructor: `WorkItemAssignmentService(WorkItemsConfig, Instance<WorkerSelectionStrategy>, WorkerRegistry, WorkloadProvider workloadProvider, ...)`. The `workloadProvider` parameter is direct injection (not `Instance<WorkloadProvider>`), so CDI cannot defer resolution — boot fails immediately.

**Test static inner class correction:** Solution 2 uses `@DefaultBean`, which is correct for standalone test source files. For `@Alternative @Priority(1)` **static inner classes** (nested inside a `@QuarkusTest` class), Quarkus auto-discovers and activates them from the current bean archive — no `quarkus.arc.selected-alternatives` entry needed. This is different from external JARs (`InMemoryWorkItemStore`), which DO require explicit `selected-alternatives`.
