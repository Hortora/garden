## Technique

When a `SlaBreachPolicy` implementation needs multi-tier escalation (first breach → escalate to group A; if group A's task also expires → fail terminally), the policy does NOT need to serialise the decision chain or store tier state externally. The WorkItem's own `candidateGroups` field carries the tier information.

`ExpiryLifecycleService` calls `onBreach()` again when the escalated WorkItem expires. At that point, `SlaBreachContext.task().candidateGroups()` contains the groups set during the first escalation. The policy reads this to determine the tier:

```java
public BreachDecision onBreach(SlaBreachContext ctx) {
    Preferences p = ctx.preferences();
    String escalationGroup = p.getOrDefault(ESCALATION_GROUP).value();

    // Second breach — escalated WorkItem also expired
    if (ctx.task().candidateGroups().contains(escalationGroup)) {
        return new BreachDecision.Fail(
            p.getOrDefault(BREACH_TERMINAL_REASON).value());
    }

    // First breach — escalate to senior group
    return EscalateTo.to(escalationGroup)
        .withDeadline(Duration.ofHours(p.getOrDefault(ESCALATION_HOURS).value()));
}
```

The same `onBreach()` method handles both tiers. No state is stored between calls; no `Chained` wrapper is needed for the normal two-tier case.

## Why this is non-obvious

The intuitive approach is to serialize the decision tree into the WorkItem (payload, label, or case context key), then deserialize it on the next breach to know what to execute. This requires a `Chained` wrapper or complex state management. The elegant solution is to recognize that `candidateGroups` already encodes the tier: if the escalation group is already assigned, the item has been through one breach cycle.

## When to use

- Two-tier SLA escalation (first breach → escalate to group B; second breach → terminal action)
- Any policy where the WorkItem's current assignment state encodes which tier has been reached
- Preference-driven policies where tier thresholds come from `PreferenceProvider`

## Critical caveat — candidateGroups is REPLACED after escalation

`ExpiryLifecycleService.executeEscalateTo()` does `item.candidateGroups = String.join(",", escalate.groups())` — it **replaces** the entire field with only the escalation group. The original groups are gone after the first breach.

Consequence: if the policy's first-tier check tests for the **original** group only, it will return `Fail` on the second breach instead of reaching the `Exhausted` branch — silently making `Exhausted` unreachable. The guard must cover both groups:

```java
// WRONG — "regulatory-affairs" gone after first escalation
boolean isRegulatory = ctx.task().candidateGroups().contains("regulatory-affairs");

// CORRECT — test both tiers
boolean isRegulatory = ctx.task().candidateGroups().contains("regulatory-affairs")
                    || ctx.task().candidateGroups().contains("regulatory-leadership");
if (!isRegulatory) return new Fail("no-policy");
if (ctx.task().candidateGroups().contains("regulatory-leadership")) return new Exhausted("...");
return EscalateTo.to("regulatory-leadership").withDeadline(Duration.ofHours(48));
```

Verified in `ExpiryLifecycleService.executeEscalateTo()` source: `item.candidateGroups = String.join(",", escalate.groups())` — line runs before `workItemStore.put(item)`.

## When not to use

- Three or more tiers with overlapping group assignments (candidateGroups may be ambiguous)
- Non-group-based tiers (e.g. tier determined by breach count — use a case context signal instead)
