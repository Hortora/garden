### Symptom
Flyway startup fails with "Found more than one migration with version 1" (or "version 1.0.0") when a Quarkus application embeds casehub-work and/or casehub-qhorus and/or casehub-engine-persistence-hibernate. These libraries ship migrations at `classpath:db/migration` that collide with each other and with consumer domain migrations.

**Variant — engine jar:** When `casehub-engine-persistence-hibernate` is on the classpath alongside `casehub-work`, the error message says `"Found more than one migration with version 1.0.0"` (not "version 1"). This is harder to diagnose — `V1.0.0` doesn't appear in the consumer's migrations or in casehub-work's migrations. It comes from `V1.0.0__Create_Quartz_Tables.sql` inside the engine persistence jar. Flyway treats `V1.0.0` and `V1` as the same version (both resolve to version 1).

### Context
Occurs when adding `io.casehub:casehub-work` and/or `io.casehub:casehub-qhorus` and/or `io.casehub:casehub-engine-persistence-hibernate` as dependencies to any Quarkus application that already has its own Flyway migrations in the V1-V21 range. When multiple libraries are present, they collide with each other even if the consumer has already renumbered to V100+.

Known collision pairs:
- casehub-work V1–V21+ vs casehub-qhorus V1–V9 (both at `classpath:db/migration`)
- casehub-work V1 vs casehub-engine-persistence-hibernate `V1.0.0` (Flyway treats both as version 1)

### What was tried
Adding casehub-work dependency and running `@QuarkusTest` suite → Flyway startup failure on V1 duplicate. Configuring explicit `quarkus.flyway.locations` is not needed because Quarkus automatically scans all classpath locations.

### Root cause
Quarkus Flyway's `classpath:` location prefix uses `ClassLoader.getResources()` which finds matching paths from **all JARs** on the classpath, not just the application's own resources directory. casehub-work ships V1–V21+, casehub-qhorus ships V1–V9, and casehub-engine-persistence-hibernate ships `V1.0.0__Create_Quartz_Tables.sql` and `V1.1.0__Create_Application_Tables.sql` — all at `classpath:db/migration`. They collide with each other and with any consumer migrations in the same version range. Flyway's version number format `V1.0.0` is treated as equivalent to `V1`, so the engine jar triggers the "duplicate version" error even though its file is named differently.

### Solution 1 — V100+ renumbering (casehub-work only, no casehub-qhorus or engine-persistence-hibernate)

Renumber the consumer application's domain migrations to start from V100 or higher:

```
# Before
db/migration/V1__my_domain_table.sql

# After
db/migration/V100__my_domain_table.sql
```

**Pros:** Simple — just rename files.
**Cons/trade-offs:** Does NOT work when casehub-qhorus or casehub-engine-persistence-hibernate is also on the classpath. Those libraries collide at V1-V9 / V1.0.0 regardless of what the consumer does. Also requires renaming all existing migration files.

### Solution 2 — Datasource-scoped subdirectories (required when casehub-qhorus or engine-persistence-hibernate is present)

Move consumer migrations into datasource-scoped subdirectories and configure explicit Flyway locations per datasource:

```
# Directory structure
db/migration/default/     ← consumer domain migrations (V100+)
db/migration/qhorus/      ← consumer ledger join tables (V1005+)
```

```properties
# Default datasource — consumer domain only (excludes all foundation JAR migrations)
quarkus.flyway.locations=classpath:db/migration/default

# qhorus named datasource — qhorus jar migrations + consumer ledger joins
quarkus.flyway.qhorus.locations=classpath:db/migration,classpath:db/migration/qhorus
```

For tests (H2 in-memory): disable Flyway and use drop-and-create — the classpath collision cannot be avoided in tests without excluding JARs from the scan:
```properties
quarkus.flyway.migrate-at-start=false
quarkus.flyway.qhorus.migrate-at-start=false
quarkus.hibernate-orm.database.generation=drop-and-create
quarkus.hibernate-orm.qhorus.database.generation=drop-and-create
```

**Pros:** Complete fix — eliminates all version collisions between casehub-work, casehub-qhorus, casehub-engine-persistence-hibernate, and consumer migrations. Explicit locations prevent future collisions if any library adds more migrations.
**Cons/trade-offs:** Requires moving all migration files into the new subdirectory structure. Requires a named datasource (`qhorus`) for the qhorus/ledger PU — which casehub-qhorus already mandates.

### Why this is non-obvious
Flyway documentation implies `classpath:db/migration` scans the application's own resources. The transitive JAR scanning behaviour is undocumented. There is no warning in any of the casehub library READMEs, CLAUDE.md files, or extension code. When the engine jar is the collision source, the error message says "version 1.0.0" — which doesn't match any migration filename in the consumer's code or in casehub-work, making the source of the conflict impossible to identify without inspecting JAR contents.
