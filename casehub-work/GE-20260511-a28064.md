### Symptom
Flyway startup fails with "Found more than one migration with version 1" when a Quarkus application embeds casehub-work. The consumer's domain migrations (e.g. `V1__clinical_trial.sql`) conflict with casehub-work's own migrations (`V1__initial_schema.sql`) — both at `classpath:db/migration`.

### Context
Occurs when adding `io.casehub:casehub-work` as a dependency to any Quarkus application that already has its own Flyway migrations using V1-V21 numbering.

### What was tried
Adding casehub-work dependency and running `@QuarkusTest` suite → Flyway startup failure on V1 duplicate. Configuring explicit `quarkus.flyway.locations` is not needed because Quarkus automatically scans all classpath locations.

### Root cause
Quarkus Flyway's `classpath:` location prefix uses `ClassLoader.getResources()` which finds matching paths from **all JARs** on the classpath, not just the application's own resources directory. casehub-work's Flyway migrations (V1–V21+ as of 0.2-SNAPSHOT) live at `classpath:db/migration` in the casehub-work JAR. The casehub-work Quarkus extension deployment processor (`WorkItemsProcessor.java`) has a TODO comment: "Flyway migration resource registration for native builds" — the team knows the approach is incomplete but there is no consumer warning.

### Fix
Renumber the consumer application's domain migrations to start from V100 or higher:

```
# Before
db/migration/V1__my_domain_table.sql
db/migration/V2__other_table.sql

# After
db/migration/V100__my_domain_table.sql
db/migration/V101__other_table.sql
```

For H2 in-memory test databases (no persisted schema history), renaming is safe. Production databases must plan a migration path.

Keep V1004+ reserved for casehub-ledger subclass join tables (platform convention).

### Why this is non-obvious
Flyway documentation implies `classpath:db/migration` scans the application's own resources. The transitive JAR scanning behaviour is undocumented. There is no warning in casehub-work's README, CLAUDE.md, or extension code. The first symptom (duplicate version error) names both conflicting scripts — which makes the root cause findable, but only after debugging the classpath.
