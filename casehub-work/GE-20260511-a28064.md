### Symptom
Flyway startup fails with "Found more than one migration with version 1" when a Quarkus application embeds casehub-work and/or casehub-qhorus. Both ship migrations at `classpath:db/migration` — casehub-work V1–V21+, casehub-qhorus V1–V9. These collide with each other and with consumer domain migrations.

### Context
Occurs when adding `io.casehub:casehub-work` and/or `io.casehub:casehub-qhorus` as dependencies to any Quarkus application that already has its own Flyway migrations in the V1-V21 range. When both libraries are present, they collide with each other even if the consumer has already renumbered to V100+.

### What was tried
Adding casehub-work dependency and running `@QuarkusTest` suite → Flyway startup failure on V1 duplicate. Configuring explicit `quarkus.flyway.locations` is not needed because Quarkus automatically scans all classpath locations.

### Root cause
Quarkus Flyway's `classpath:` location prefix uses `ClassLoader.getResources()` which finds matching paths from **all JARs** on the classpath, not just the application's own resources directory. casehub-work ships V1–V21+ and casehub-qhorus ships V1–V9, both at `classpath:db/migration`. They collide with each other and with any consumer migrations in the same version range.

### Solution 1 — V100+ renumbering (casehub-work only, no casehub-qhorus)

Renumber the consumer application's domain migrations to start from V100 or higher:

```
# Before
db/migration/V1__my_domain_table.sql

# After
db/migration/V100__my_domain_table.sql
```

**Pros:** Simple — just rename files.
**Cons/trade-offs:** Does NOT work when casehub-qhorus is also on the classpath. casehub-qhorus and casehub-work collide at V1-V9 regardless of what the consumer does. Also requires renaming all existing migration files.

### Solution 2 — Datasource-scoped subdirectories (required when both casehub-work and casehub-qhorus are present)

Move consumer migrations into datasource-scoped subdirectories and configure explicit Flyway locations per datasource:

```
# Directory structure
db/migration/default/     ← consumer domain migrations (V100+)
db/migration/qhorus/      ← consumer ledger join tables (V1005+)
```

```properties
# Default datasource — consumer domain only (excludes casehub-work and casehub-qhorus JARs)
quarkus.flyway.locations=classpath:db/migration/default

# qhorus named datasource — qhorus jar migrations + consumer ledger joins
quarkus.flyway.qhorus.locations=classpath:db/migration,classpath:db/migration/qhorus
```

For tests (H2 in-memory): disable Flyway and use drop-and-create — the classpath collision cannot be avoided in tests without excluding JARs from the scan:
```properties
quarkus.flyway.migrate-at-start=false
quarkus.flyway.qhorus.migrate-at-start=false
quarkus.hibernate-orm.database.generation=drop-and-create
quarkus.hibernate-orm.qhorus.database.generation=drop-and-create
```

**Pros:** Complete fix — eliminates all version collisions between casehub-work, casehub-qhorus, and consumer migrations. Explicit locations prevent future collisions if either library adds more migrations.
**Cons/trade-offs:** Requires moving all migration files into the new subdirectory structure. Requires a named datasource (`qhorus`) for the qhorus/ledger PU — which casehub-qhorus already mandates.

### Why this is non-obvious
Flyway documentation implies `classpath:db/migration` scans the application's own resources. The transitive JAR scanning behaviour is undocumented. There is no warning in casehub-work's or casehub-qhorus's README, CLAUDE.md, or extension code. When both libraries are present, the collision is between the libraries themselves — renumbering the consumer's migrations doesn't help.
