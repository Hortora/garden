### Symptom
After calling `WorkItemService.reject(id, actorId, reason)`, reading `WorkItemRef.resolution()` returns whatever was set before rejection — not the rejection reason passed to `reject()`.

### Context
Building a rejection workflow that needs to surface why a human rejected something — e.g., PendingApprovalHandler.check() constructing `ApprovalCheckResult.Rejected(planReference, reason)`.

### Root cause
`WorkItemService.reject()` writes the `reason` parameter to `WorkItemLifecycleEvent` (audit trail) but does not update `WorkItem.resolution`. The resolution field is set by `WorkItemService.complete()`, not `reject()`. The rejection reason lives only in the lifecycle event stream.

### Fix
Use `Objects.requireNonNullElse(ref.resolution(), "rejected")` as the reason string. Accept that detailed rejection reasons are in casehub-work's audit trail, not on the WorkItemRef. Design fault policy routing around `FaultType.APPROVAL_REJECTED` rather than the reason string content.

### Why this is non-obvious
The `reject(id, actorId, reason)` method signature strongly implies `reason` is stored on the WorkItem. The parameter name is `reason`, not `auditDetail` or `eventContext`. A developer building rejection handling would naturally expect `ref.resolution()` to carry the rejection reason.
