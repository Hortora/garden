## Symptom

`ExpiryLifecycleService` fails to compile with "cannot find symbol: method root()" after the SlaBreachPolicy wiring is added. The design document stated `Path.root()` was "already committed to platform main" — but scanning all platform branches confirms it exists in none of them.

## Context

`buildBreachContext()` in the new `ExpiryLifecycleService` uses `Path.root()` as the fallback when `WorkItem.scope` is null:

```java
Path scope = item.scope != null ? Path.parse(item.scope) : Path.root();
```

`Path.of()` (no-arg) throws `IllegalArgumentException("Path must have at least one segment")`. There is no zero-arg path factory on `platform-api` main.

## Root cause

`Path.root()` was planned but not implemented. The design doc said "already committed to platform main (your call on publishing)" — this was aspirational, not factual. The method must be added to `platform-api` and published before `casehub-work` can compile.

## Fix

1. Add `Path.root()` to `casehub-platform-api`:
```java
/** Sentinel path representing no specific scope — used when WorkItem has no assigned scope. */
public static Path root() {
    return new Path("", List.of());
}
```
Verify `JpaPreferenceProvider.ancestors()` handles empty-value root path correctly (rows with `scope = ""` or no scope rows are returned first in the resolution walk).

2. Publish new `casehub-platform-api` artifact.
3. Bump `casehub-work-api` dependency on `casehub-platform-api`.
4. Do not mark work#212 as ready until this ships.

## Why this is non-obvious

Design documents in multi-session, multi-repo projects can describe aspirational state as fact. The "already committed" language was unambiguous but wrong. The only reliable check is inspecting the actual repo branches — `git -C <platform> grep "root" HEAD -- Path.java`.
