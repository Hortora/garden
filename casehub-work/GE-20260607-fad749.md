When `doThrow(exception).when(mock).method(SPECIFIC_ID)` stubs a specific argument value, Mockito strict mode throws `PotentialStubbingProblem` when the stubbed method is called with ANY other argument in the same test. The error fires inside the production code's try-catch, making the root cause non-obvious.

### Symptom

Test passes for `method(SPECIFIC_ID)` (which throws as expected) but Mockito itself throws `PotentialStubbingProblem` when `method(OTHER_ID)` is called in the same loop or test body. The exception is caught by an outer try-catch in production code, silently swallowing it.

### Context

Occurs in `@QuarkusTest` or pure `@ExtendWith(MockitoExtension.class)` tests when:
1. A loop or batch-processing method calls `mock.method(id)` for multiple IDs
2. The test stubs `doThrow(x).when(mock).method(SPECIFIC_ID)` to simulate failure on one specific ID
3. Mockito sees calls with non-matching IDs and detects a "potential stubbing problem"

### Root cause

Mockito strict stubbing checks that stubbed invocations are actually used. When an invocation occurs with a non-matching argument, it doesn't know whether this was a mistake (wrong argument in the stub) or intentional (loop calls with multiple IDs). It throws `PotentialStubbingProblem` as a safety check.

### Fix

Wrap the specific stub with `lenient()`:

```java
// Bad — throws PotentialStubbingProblem for calls with ID_1 and ID_3
doThrow(new RuntimeException("fail")).when(delivery).attemptDelivery(ID_2);

// Good — ID_2-specific stub, lenient for other arguments
lenient().doThrow(new RuntimeException("fail")).when(delivery).attemptDelivery(ID_2);
```

`lenient()` scopes the leniency to this one stub only — strict mode still applies to all other stubs.

### Why this is non-obvious

The error fires inside the production code being tested (caught by the outer try-catch), not at the stub registration or test assertion. The stack trace points to the production code, not the stub. The fix (`lenient()`) is not discoverable from the error message alone.
