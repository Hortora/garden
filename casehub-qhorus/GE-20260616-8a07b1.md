### Symptom

A channel declared with `allowedTypes: []` (meaning "nothing permitted") becomes indistinguishable from a channel with no type restriction (meaning "all types permitted") after storage.

### Context

`ChannelCreateRequest` uses `Set<MessageType>` where:
- `null` = open (all types permitted)
- `Set.of()` = nothing permitted (explicit empty restriction)

This distinction is semantically important and documented in the record's javadoc.

### Root cause

`ChannelService.setTypeConstraints()` normalizes null before processing:

```java
final Set<MessageType> allowed = allowedTypes != null ? allowedTypes : Set.of();
```

Then stores via:

```java
ch.allowedTypes = MessageType.serializeTypes(allowed);
```

And `MessageType.serializeTypes(Set.of())` returns `null`. So the chain is:

```
null → Set.of() → serializeTypes(Set.of()) → null  (original null preserved)
Set.of() → Set.of() → serializeTypes(Set.of()) → null  (empty set lost)
```

Both paths produce `null` in the database. The "nothing permitted" semantic disappears.

### Fix

None known. This is a platform limitation in qhorus. Workaround: treat both `null` and `[]` as "open" in YAML schemas and drift detection. Document the limitation explicitly.

A proper fix would require `MessageType.serializeTypes()` to distinguish empty from null (e.g., storing an empty string for "nothing permitted" vs null for "open"), plus matching changes in `parseTypes()`.

### Why this is non-obvious

The `ChannelCreateRequest` javadoc explicitly distinguishes null from empty Set. The `setTypeConstraints()` method silently collapses this distinction. You only discover the lossy round-trip by tracing through three method calls across two classes.
