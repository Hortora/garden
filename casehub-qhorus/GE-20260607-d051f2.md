### Symptom

`MessageObserver.onMessage(MessageReceivedEvent event)` is called as expected, but `event.content()` is always `null` even though the sender called `MessageDispatch.builder().content(payload).messageType(MessageType.EVENT).build()` and passed a non-null JSON string. The dispatch succeeds without error. The content simply never arrives.

### Context

Occurs whenever `MessageType.EVENT` is used with a content payload and the receiver implements `MessageObserver`. Affects all casehub-qhorus consumers that use the MessageObserver SPI for observe-channel broadcasts.

### Root cause

`MessageObserverDispatcher` (lines 64–65) explicitly nullifies the content field for EVENT messages before constructing `MessageReceivedEvent`:

```java
final String content = message.messageType == MessageType.EVENT ? null : message.content;
```

`MessageReceivedEvent` also enforces this at construction time. This is intentional behaviour per internal protocol PP-20260508-90428f — EVENT is modelled as a pure notification signal with no payload. The API gives no indication of this constraint: `MessageDispatch.builder()` accepts `.content()` for all message types without warning, and the dispatch call succeeds silently.

### Fix

Use `MessageType.STATUS` for content-carrying observe-channel broadcasts.

STATUS:
- Carries content (not nullified by the dispatcher)
- Creates no commitment obligation
- Requires no `inReplyTo` or `correlationId`
- Is allowed on OBSERVE channels (`allowedTypes` includes `EVENT`, `QUERY`, `STATUS`)
- The `MessageDispatch.build()` implementation puts STATUS in the default case — no extra builder fields needed

```java
messageService.dispatch(MessageDispatch.builder()
    .channelId(channelId)
    .sender(getId())
    .actorType(ActorType.AGENT)
    .messageType(MessageType.STATUS)   // ← not EVENT
    .content(jsonPayload)
    .build());
```

Receivers in `onMessage()` will receive the content string populated.

### Why this is non-obvious

`MessageDispatch.builder()` accepts `.content()` for any message type without validation or warning. The dispatch call succeeds with no error. Only reading `MessageObserverDispatcher.java` source reveals the nullification. The protocol reference (PP-20260508-90428f) is internal to casehub-qhorus and not surfaced in the API documentation. A developer following PLATFORM.md's guidance to use EVENT for "passive monitoring and state sharing" on OBSERVE channels will build and test dispatch, see no errors, and only discover the data loss when the observer inspects `event.content()`.

### Constraints

Applies when using `MessageObserver` SPI. Does not affect `ChannelBackend.post()` — that SPI receives the full `Message` entity and is not subject to this nullification.

### Alternatives considered

- `MessageType.COMMAND` — carries content but creates a Commitment obligation requiring a response (DONE/DECLINE). Wrong semantics for broadcast intel.
- `MessageType.QUERY` — requires `correlationId` per `MessageDispatch.build()` validation. Adds unnecessary overhead for fire-and-forget broadcasts.
- `MessageType.STATUS` — correct: content-carrying, obligation-free, allowed on OBSERVE channels.

### Invalidation triggers

Revisit if casehub-qhorus removes the content nullification from `MessageObserverDispatcher` for EVENT type, or if PP-20260508-90428f is revised to allow EVENT payloads.
