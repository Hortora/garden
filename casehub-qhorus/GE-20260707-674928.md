### Symptom

`channelService.delete(channelId, true)` throws a `ConstraintViolationException` when the channel contains messages with `in_reply_to` references to other messages in the same channel. The `force=true` flag does not override the FK constraint.

### Context

When multiple `@QuarkusTest` classes share the same Quarkus instance (and therefore the same H2 database), channels created by one test class persist for subsequent test classes. If a channel has messages with `in_reply_to` (e.g. from `MessageDispatch.builder().inReplyTo(raiseMessageId)`), deleting the channel to recreate it fails on `FK_MESSAGE_REPLY`.

This surfaces when two test classes create channels with deterministic names (e.g. `drafthouse/debate/replay-workspace-replay`) — the second class finds the channel already exists and attempts to delete and recreate it.

### Root cause

H2's `FK_MESSAGE_REPLY` constraint on `MESSAGE.IN_REPLY_TO → MESSAGE.ID` prevents message deletion when other messages reference them. `ChannelService.delete()` deletes messages in insertion order, not reverse dependency order, so parent messages are deleted before their replies — triggering the FK violation.

### Fix

Do not delete and recreate the channel. Instead, reuse the existing channel:

```java
var existingChannel = channelService.findByName(channelName);
if (existingChannel.isPresent()) {
    var existingSession = registry.find(existingChannel.get().id());
    if (existingSession.isPresent()) {
        return existingSession.get(); // already loaded
    }
    // Orphaned channel — reuse it, don't delete
    channel = existingChannel.get();
} else {
    channel = channelService.create(ChannelCreateRequest.builder(channelName)...build());
}
```

### Why this is non-obvious

The `force=true` parameter on `channelService.delete()` suggests it will handle cleanup unconditionally. The FK violation only surfaces when messages have `inReplyTo` linkage — channels with simple messages (no replies) delete fine, so the problem is intermittent depending on test data shape.
