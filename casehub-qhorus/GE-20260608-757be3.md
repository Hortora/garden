### Symptom

`MessageService.dispatch()` returns normally but no message is stored in the channel.
No exception is thrown at the Java call boundary. The Qhorus transaction rolls back silently.

### Context

Using `MessageDispatch.Builder.artefactRefs(String)` to pass application-level metadata as
a sidecar alongside message content — e.g. `"entryType=raise|agent=REV|round=1|priority=P1"`.
The Builder accepts any String at compile time.

### Root cause

`ArtefactRefParser.parse()` is called inside `MessageService.dispatch()`. It splits the
`artefactRefs` value on commas and calls `UUID::fromString` on each token. Any non-UUID
token fails parsing and triggers a transaction rollback. The exception is caught internally —
no signal reaches the caller.

### Fix

Do not use `artefactRefs` for application-level metadata. Options:

1. **Encode metadata in `content`** — use a structured prefix (e.g. `META:key=value|...\n\n<body>`)
   and parse it in the projection handler. The `DebateChannelProjection` in casehub-drafthouse
   uses this approach (drafthouse#27).

2. **Use Qhorus `SharedData`** — store metadata in a `SharedData` artefact and pass its UUID
   as `artefactRefs`. Correct semantics but requires a two-step write.

### Why this is non-obvious

`MessageDispatch.Builder.artefactRefs(String)` has no compile-time constraint — it accepts
any String. There is no documentation stating the UUID-only runtime requirement. The dispatch
call returns normally (no Java exception is propagated to the caller), so there is no signal
that the message was rejected. The only observable symptom is the absence of stored messages.
