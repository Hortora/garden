### Symptom

`tools.sendMessage(channel, sender, "query", content, corrId, ...)` throws `ToolCallException`
wrapping `MessageTypeViolationException` when the channel has `allowedTypes` set that excludes
QUERY. But `tools.sendMessage(channel, sender, "response", content, corrId, ...)` on the same
channel succeeds (dispatched with an advisory in `DispatchResult.advisories()`).

In benchmark tests where `WorkerAgent` responds with QUERY to a typed COMMAND channel, the
`sendMessage(type=query)` recording call throws and the entire test iteration fails.

### Context

Occurs when recording a WorkerAgent's response to a Qhorus channel that has typed constraints.
The channel in the normative benchmark sets `allowedTypes = COMMAND,STATUS,FAILURE,DECLINE,DONE`.
When the 1B model responds with QUERY type (asking for clarification before acting), the harness
attempts to record that QUERY to the channel — which hard-blocks.

### What was tried (didn't work)

Expected both QUERY and RESPONSE to produce advisories (soft enforcement) since neither is in
the ALLOWED_TYPES list. Only QUERY threw; RESPONSE was dispatched silently.

### Root cause

`MessageTypePolicy.validate()` hard-enforces violations for **COMMAND and QUERY only**. All
other type violations produce advisories via `advisory()`. From CLAUDE.md:

> "COMMAND/QUERY violations are hard-enforced (throw); all other violations produce advisories
> in DispatchResult.advisories(); denial-first within COMMAND/QUERY."

The rationale: COMMAND and QUERY are obligation-creating — they open commitments in
`CommitmentService`. Hard-blocking them prevents orphan commitments. RESPONSE, STATUS, etc.
do not open commitments, so soft enforcement is safe.

### Fix

Wrap `sendMessage()` in a try-catch for non-`{DONE,FAILURE,DECLINE,STATUS}` response types in
test harnesses that record LLM responses to typed channels:

```java
try {
    // QUERY hard-throws (MessageTypeViolationException via MessageTypePolicy).
    // RESPONSE, HANDOFF, etc. produce advisories and are dispatched.
    tools.sendMessage(channel, "worker", response.messageType().toLowerCase(),
            response.content(), corrId, cmdResult.messageId(), null, null, null, null, null);
} catch (final Exception ignored) {
    // Hard-violation type — not in channel allowedTypes.
    // Response is still valid for benchmark classification.
}
```

Alternatively, only attempt to record types that are in `allowedTypes`:

```java
final Set<String> safeTypes = Set.of("done", "failure", "decline", "status");
if (safeTypes.contains(response.messageType().toLowerCase())) {
    tools.sendMessage(channel, "worker", response.messageType().toLowerCase(), ...);
}
```

### Why this is non-obvious

Developers expect type enforcement to be symmetric: either all out-of-allowedTypes types
block, or all produce advisories. The split — hard-block for COMMAND/QUERY, advisory for
everything else — is documented in CLAUDE.md but easy to miss. The error message
(`ToolCallException` wrapping `MessageTypeViolationException`) correctly identifies QUERY
as the problem, but the asymmetry with RESPONSE is only understood by reading the source.
