### Symptom
`commitmentStore.findOpenByObligor(sender, channelId)` returns empty even immediately after `messageService.send(channelId, sender, MessageType.COMMAND, ...)` with a non-null correlationId. The assertion fails in a confusing direction — `assertFalse(list.isEmpty())` fails with "expected: <false> but was: <true>".

### Context
Writing an integration test that sends a COMMAND to a channel and then asserts a commitment exists — typically to test channel deletion or commitment lifecycle. The natural instinct is to query "what did this sender commit to?"

### Root cause
For COMMAND (and QUERY) messages, `MessageService.send()` calls `commitmentService.open()` with `sender → requester` and `target → obligor`. A COMMAND without a target produces a commitment with `obligor = null`. `findOpenByObligor(sender, channelId)` filters by `c.obligor.equals(sender)` — which never matches because the sender is the requester, not the obligor.

The speech-act semantics: a COMMAND creates an obligation on the *receiver* (obligor) to fulfil, not on the sender. The sender is the requester (the one who issued the command).

### Fix
To test whether a channel has *any* open commitments — regardless of role — use `findByState(CommitmentState.OPEN, channelId)` instead:

```java
assertFalse(commitmentStore.findByState(CommitmentState.OPEN, chId).isEmpty());
```

Use `findOpenByObligor` only when you specifically need to find commitments where a given party is the *receiver* (i.e. what they owe), not when you want to find what they created.

### Why this is non-obvious
The API name `findOpenByObligor` doesn't signal speech-act role semantics. "The agent that sent the COMMAND" feels like the natural query argument, but in deontic logic the obligor is the party who must fulfil the commitment — the receiver, not the requester. There's no compile-time or runtime error; the result is simply empty.
