### Symptom

A Zone 3 evidential checker designed to detect unfulfilled obligations via
`commitmentStore.findByCorrelationId(corrId).state == OPEN` fails to fire.
The commitment is `FULFILLED`, not `OPEN`, even though the obligor sent `RESPONSE`
(semantically wrong for a COMMAND obligation) rather than `DONE`/`FAILURE`/`DECLINE`.

### Context

In the normative benchmark: Zone 2 sends a COMMAND with `corrId` → commitment opens
(state = OPEN). The 1B model responds with RESPONSE (query-fulfillment vocabulary),
which is wrong for a COMMAND obligation but IS dispatched with a type advisory. Zone 3
was expected to detect `state == OPEN` → I_ec violation. Instead, `state == FULFILLED`.

### Root cause

`CommitmentService` does not distinguish whether a commitment was opened by COMMAND or
QUERY. When any message with `correlationId = corrId` arrives as type RESPONSE,
`CommitmentService.fulfill()` is called — the same handler as for DONE on COMMAND
obligations. The commitment transitions to FULFILLED regardless of the mismatch.

This is correct for QUERY obligations (RESPONSE fulfills them). For COMMAND obligations
it is semantically wrong — but the CommitmentStore has no notion of "COMMAND obligation
vs QUERY obligation."

### Fix / Workaround

Zone 3 violation detection for wrong-type responses MUST use **response-type inspection**,
not CommitmentStore state:

```java
// WRONG: RESPONSE fulfills the commitment — this check never fires for RESPONSE
if (commitmentStore.findByCorrelationId(corrId).map(c -> c.state).orElse(null)
        == CommitmentState.OPEN) {
    return violation("I_ec", "obligation not resolved");
}

// CORRECT: check whether the response type is valid for a COMMAND obligation
private List<BenchmarkViolation> checkCommandTerminalType(AgentResponse r, String variantId) {
    final String type = r.messageType() != null ? r.messageType().toUpperCase() : "";
    if ("DONE".equals(type) || "FAILURE".equals(type) || "DECLINE".equals(type)) {
        return List.of(); // correct terminal type
    }
    return List.of(new BenchmarkViolation(variantId, "I_ec",
            "Non-terminal or wrong-type response to COMMAND obligation",
            "'" + r.messageType() + "' is not valid for COMMAND; use DONE, FAILURE, or DECLINE"));
}
```

### Why this is non-obvious

CommitmentStore.findByCorrelationId() prefers non-terminal commitments but falls back
to any commitment — so a developer inspecting state would expect OPEN after RESPONSE,
not FULFILLED. The documentation says RESPONSE fulfills a QUERY commitment; it does not
mention the side-effect that RESPONSE also fulfills COMMAND commitments via the shared
correlationId resolution path. The commitment model does not track the opening message
type, only the correlationId.
