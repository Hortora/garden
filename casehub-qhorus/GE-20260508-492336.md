## Symptom

Quarkus fails at startup with an error about a reactive Vert.x pool or reactive datasource not being configured, even though the application only uses JDBC H2 and `casehub.qhorus.reactive.enabled=false` is set.

```
Failed to start quarkus: Reactive datasource 'qhorus' requires a reactive driver
io.quarkus.runtime.QuarkusBindException: ...Vert.x pool...
```

## Context

Any Quarkus application that adds `casehub-qhorus` as a Maven dependency will trigger this failure if it does not provide a reactive datasource. The `casehub.qhorus.reactive.enabled=false` build-time flag exists in qhorus but does **not** suppress the Quarkus hibernate-reactive extension from activating — it only controls qhorus's own reactive messaging layer.

## Root Cause

`casehub-qhorus` has a compile-scope dependency on `quarkus-hibernate-reactive-panache`. When the deployment JAR of that extension is on the classpath, Quarkus activates the reactive persistence layer unconditionally at augmentation time — before any `@IfBuildProperty` guards in qhorus's own processors can suppress it. The result is that Quarkus tries to create a reactive Vert.x pool for every configured datasource, including the default H2 JDBC datasource, which has no reactive driver.

## Fix

Add these properties to `application.properties` (or `src/test/resources/application.properties` for tests):

```properties
# Suppress reactive datasource activation — casehub-qhorus pulls in
# quarkus-hibernate-reactive-panache unconditionally; this prevents Quarkus
# from trying to wire a reactive Vert.x pool to JDBC datasources.
quarkus.datasource.reactive=false
quarkus.datasource.qhorus.reactive=false
```

**Do NOT add `casehub.qhorus.reactive.enabled=false`** — this key was removed from the qhorus config model in a later 0.2-SNAPSHOT build. Adding it now causes `ConfigValidationException` at startup, a hard failure with a misleading error message. The two standard Quarkus properties above are sufficient.

## ⚠️ Deprecated property (added 2026-05-17)

`casehub.qhorus.reactive.enabled=false` was a valid workaround in earlier qhorus 0.2-SNAPSHOT builds but was **removed from the qhorus config model** by qhorus 0.2-SNAPSHOT (circa 2026-05). Adding it to `application.properties` now causes:

```
io.smallrye.config.ConfigValidationException: SRCFG00014: The config property casehub.qhorus.reactive.enabled
with the config value 'false' was converted to null from the converter
```

This is a hard startup failure with a misleading error — it looks like a config parsing problem, not a stale property name. Remove this key from any application.properties file where it was previously added.

## Why This Is Non-Obvious

The error message says "reactive datasource not configured" — pointing at the datasource config, not at qhorus. A developer troubleshooting this naturally adds reactive datasource config rather than suppressing reactive entirely. The qhorus build-time flag (`casehub.qhorus.reactive.enabled`) sounds like it should solve the problem, but it operates at the application layer, not the Quarkus augmentation layer.

### Why this fix

Setting `quarkus.datasource.reactive=false` is the correct Quarkus mechanism to suppress reactive pool creation per datasource. The two-property combination is required because qhorus uses a named datasource (`qhorus`) in addition to the default — both must be suppressed independently.

## Structural Fix

Protocol PP-20260514-f41258 prescribes the correct extension-level fix:
mark `quarkus-hibernate-reactive-panache` as `<optional>true</optional>` in the
runtime pom; in the deployment processor replace the config flag with
`Capabilities.isPresent(Capability.HIBERNATE_REACTIVE)` as the BooleanSupplier;
produce `ExcludedTypeBuildItem` for reactive classes when capability is absent and
for blocking counterparts when it is present. Classpath presence becomes the single
source of truth — no consumer-side suppression needed.
