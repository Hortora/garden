### What this is

`Delta.getRemoveMask()` retrieves the removeMask directly from its struct via `get("removeMask")` and returns it as `Pair<String,Object>`. The Pair must be stored literally in the struct — the variant name as `value1`, the payload as `value2`.

```java
// Correct struct layout for synthetic Delta:
deltaMap.put("removeMask", new Pair<>("ZeroIndices", new Integer[]{0, 2}));
deltaMap.put("addUnitTags", new Integer[]{...});
```

There is no "one-entry-Map-to-Pair" transform. If you store a `Map<String,Object>` instead of a Pair, Java's unchecked generic cast makes `getRemoveMask()` return the Map silently typed as Pair. Accessing `removeMask.value1` then reads from the wrong memory offset, producing null — making all variants behave as carry-forward with no error.

### Where to find it

`Delta.java` in the `feature/standalone-modules` fork of Scelight s2protocol:
```java
public Pair<String, Object> getRemoveMask() {
    return get(F_REMOVE_MASK); // direct get, no transform
}
```

This is confirmed by bytecode inspection: `getRemoveMask()` is a single `invokevirtual get + checkcast Pair` — no Map iteration.

### Why this is non-obvious

The s2protocol tagged union format is typically decoded as a Map with one entry (variant name → payload). The natural assumption is that Scelight stores it the same way. The actual storage as a pre-built Pair is only visible by reading the source or inspecting bytecode. There is no documentation on the expected struct layout for synthetic event construction.
