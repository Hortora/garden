## What it does

SC2EGSet pre-processed JSON replay files store supply/food values in `PlayerStatsEvent.stats` as **raw integers**:
- `scoreValueFoodUsed: 12` → 12 supply used (e.g. 12 probes at game start)
- `scoreValueFoodMade: 15` → 15 supply cap (e.g. 1 Nexus = 15 supply)

Scelight's binary parser (`RepParserEngine`) returns the same named fields as **fixed-point ×4096**:
- `getFoodUsed() == 49152` → 49152 / 4096 = 12 supply used

## How to verify

```python
for e in sample['trackerEvents']:
    if e.get('evtTypeName') == 'PlayerStats':
        print(e['stats']['scoreValueFoodUsed'])   # → 12 (raw)
        print(e['stats']['scoreValueFoodMade'])    # → 15 (raw)
        break
# Compare to Scelight: IPlayerStatsEvent.getFoodUsed() → 49152 (÷ 4096 = 12)
```

## Why it matters

Porting parsing code from Scelight binary to SC2EGSet JSON and keeping the `/ 4096` division silently corrupts all supply values (result is 0 for values < 4096). Both formats use identically named fields, so the discrepancy is invisible without inspecting actual values.

## Why it's undocumented

The SC2EGSet README says "pre-processed" but does not document encoding differences from the raw binary format. The ×4096 fixed-point encoding is an internal s2protocol detail; Scelight decodes it transparently. SC2EGSet's preprocessing pipeline also decodes these values, but this is not stated anywhere in user-facing documentation.
