### Symptom
`ClassCastException: BitArray cannot be cast to Number` at runtime when implementing
`SelectionDeltaEvent` removeMask handling. Or silently wrong building tags when extracting
Cmd events from binary SC2 replays — the selection state drifts and no unit-train intent
is emitted even though the player issued commands.

### Context
`SelectionDeltaEvent.getDelta().getRemoveMask()` returns `Pair<String, Object>` where
`value1` is the variant name and `value2` is the payload. The payload type **varies by
variant** — there is no common supertype, no documentation, and the IEM10/SC2EGSet JSON
format uses completely different variant names, making cross-format assumptions fatal.

Occurs when implementing binary replay command extraction (`AbilityMapping.onSelection`)
after having previously implemented the same logic for SC2EGSet JSON
(`IEM10CommandExtractor.applySelectionDelta`).

### What was tried (didn't work)
- Casting `value2` to `Number` for the `Mask` variant: `((Number) removeMask.value2).intValue()` — compiles, runs silently in development, then throws `ClassCastException` against human-played replays (bot replays don't emit `Mask` events, masking the bug).
- Treating `ZeroIndices` as "no removal" (equivalent to `None`) — it is actually a **retain** operation: payload is `Integer[]` of positions to keep, not to remove. Empty array means clear all selection.

### Root cause
`Pair<String, Object>` is a heterogeneous container. Payload types per variant:

| Variant | Payload type | Semantics |
|---------|-------------|-----------|
| `ZeroIndices` | `Integer[]` | Indices to **retain** (empty = clear selection) |
| `Mask` | `BitArray` (`hu.belicza.andras.util.type.BitArray`) | Bit i set → remove index i |
| `None` | unknown/null | No removal — carry forward existing selection |
| `OneIndices` | `Integer[]` | Indices to **remove** (iterate descending to avoid shift) |
| `SweepToEnd` | — | **Does not exist in binary protocol** (only in SC2EGSet JSON) |

`ZeroIndices` is also the dominant variant: 56K–109K events per game. Bot replays use only
`ZeroIndices`; human replays add `Mask` (~1,800/game) and `None` (~1,600/game). This means
the bug is invisible in bot-only test suites.

### Fix
Use `instanceof` pattern-matching for every cast; treat `null` variant (absent removeMask)
as carry-forward (same as `None`):

```java
var removeMask = delta.getRemoveMask();
String variant = removeMask != null ? removeMask.value1 : null;
boolean fullReplacement = variant == null || "ZeroIndices".equals(variant);

List<String> result = fullReplacement ? new ArrayList<>() : new ArrayList<>(currentSelection);

if ("ZeroIndices".equals(variant)) {
    // Retain-index semantics: empty array = clear all
    if (removeMask.value2 instanceof Integer[] indices && indices.length > 0) {
        result = new ArrayList<>(indices.length);
        for (int i : indices) {
            if (i < currentSelection.size()) result.add(currentSelection.get(i));
        }
    }
} else if ("Mask".equals(variant)) {
    if (removeMask.value2 instanceof hu.belicza.andras.util.type.BitArray bitArray) {
        for (int i = Math.min(result.size() - 1, bitArray.getCount() - 1); i >= 0; i--) {
            if (bitArray.getBit(i)) result.remove(i);
        }
    } else {
        log.debugf("[SELECTION] Mask variant has unexpected payload type: %s", removeMask.value2);
    }
} else if ("OneIndices".equals(variant)) {
    if (removeMask.value2 instanceof Integer[] indices) {
        for (int i = indices.length - 1; i >= 0; i--) {
            int idx = indices[i];
            if (idx < result.size()) result.remove(idx);
        }
    }
}
// "None" / null variant: result already initialised as copy of currentSelection — no removal
```

### Why this is non-obvious
`Pair<String, Object>` gives no compile-time indication that `value2` type varies per
variant. The IEM10 SC2EGSet JSON format uses `Mask`/`SweepToEnd`/`OneIndice`/`None` as
variant names — none of these match the binary protocol's `Mask`/`ZeroIndices`/`OneIndices`/`None`.
Developers familiar with SC2EGSet JSON will assume the same semantics and write
`((Number) value2).intValue()` for `Mask` — which compiles and fails only against human
replays (not bot replays, which use only `ZeroIndices`). The bug produces silent selection-state
drift rather than an exception, causing zero training intents to be extracted rather than
incorrect ones — a hard symptom to trace back to removeMask handling.
