## Symptom

`mvn install` reports "Nothing to compile — all classes are up to date" and exits with BUILD SUCCESS. At test runtime, a `NoSuchMethodError` is thrown with a constructor or method signature that matches the *old* version of a dependency, not the current source.

## Context

A locally-developed SNAPSHOT dependency (e.g. `casehub-work 0.2-SNAPSHOT`) had a record or class changed — e.g. a new field added to a Java record, changing its constructor arity. `mvn install -DskipTests` was run on the dependency to update `~/.m2`. Maven reports the classes as up to date (because class file timestamps haven't changed, or the source was marked clean) and skips recompilation. The old JAR remains in `~/.m2`.

The consuming project compiles fine (either against the correct source or a cached compilation), but at test runtime Quarkus loads the stale JAR from `~/.m2`, not the recompiled classes. The `NoSuchMethodError` appears as a *suppressed* exception inside an `awaitility` condition timeout — the top-level failure is a test timeout, which hides the real cause.

## Root cause

Maven's incremental compilation check compares source and class file timestamps. When sources haven't been *touched* (e.g. a field was added via a git operation that preserved mtime, or another session compiled the classes at the same time), Maven considers classes current and skips `javac`. The installed JAR in `~/.m2` is whatever was last explicitly installed — it may predate the source change.

## Fix

```bash
mvn clean install -DskipTests -f path/to/dependency/pom.xml
```

`clean` deletes `target/classes`, forcing a full recompile regardless of timestamps. The resulting JAR in `~/.m2` then matches the source.

## Why this is non-obvious

`mvn install` without `clean` is the natural command. "Nothing to compile" reads as success, not as "I checked and skipped." The actual failure (wrong constructor arity) surfaces as a `NoSuchMethodError` at runtime, not a compilation error — and in test suites using `awaitility`, it appears as a timeout with the real cause in a `Suppressed:` block that's easy to overlook.
