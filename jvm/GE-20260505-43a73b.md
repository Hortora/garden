## Symptom

Unit test for a CDI service that iterates `Instance<SomeProvider>` passes in isolation but fails intermittently when the service method calls `providerFor()` or `availableTypes()` more than once. The second call returns no matching providers despite the mock being set up correctly.

## Context

Mocking `Instance<T>` for unit tests of services that use `for (T t : instance)` or `instance.stream()` internally. Occurs whenever a single test exercises a code path that calls `providerFor()` twice (e.g. `onLifecycleEvent` with multiple links, each needing a provider lookup).

## What Was Tried

```java
// BROKEN — returns the same stream instance every time
when(instances.stream()).thenReturn(List.of(provider).stream());
when(instances.iterator()).thenReturn(List.of(provider).iterator());
```

The first call to `providerFor()` succeeds. The second call within the same test invocation gets an exhausted stream (0 elements) or an iterator that has already been consumed.

## Root Cause

`List.of(provider).stream()` creates a **single stream instance**. `thenReturn(x)` stores that same instance and returns it on every call. Java streams are single-use — once consumed (by `filter()`, `findFirst()`, etc.), the same stream object returns no further elements. Iterators have the same problem.

## Fix

Use `thenAnswer` to return a fresh stream/iterator on each invocation:

```java
@SuppressWarnings("unchecked")
final Instance<IssueTrackerProvider> instances = mock(Instance.class);

when(instances.iterator())
    .thenAnswer(inv -> List.<IssueTrackerProvider>of(provider).iterator());
when(instances.stream())
    .thenAnswer(inv -> List.<IssueTrackerProvider>of(provider).stream());
```

`thenAnswer` executes the lambda on every call, producing a fresh stream/iterator each time.

## Why This Is Non-Obvious

The mock setup looks correct — the provider is in the list. The symptom ("no provider found") looks like a configuration issue, not a stream lifecycle issue. Most developers reach for `thenReturn` first; `thenAnswer` is typically used for dynamic return values, not for fixing single-use-object semantics. The bug only surfaces when `providerFor()` is called twice in one test, which doesn't happen in simple happy-path tests.
