## Quarkus @Entity subclass validation needs ValidationPhaseBuildItem

### Symptom

A `@BuildStep` that checks the JPA entity hierarchy (e.g. verifying that
`@Entity` subclasses with persistent fields override a specific method) finds
zero subclasses when producing an `AdditionalBeanBuildItem`. The entity hierarchy
appears flat — only the base class is visible.

### Context

Quarkus augmentation runs in phases. The `AdditionalBeanBuildItem` phase runs
early — before Hibernate ORM has fully resolved JPA `@Entity` inheritance
(`JOINED`, `SINGLE_TABLE`, `TABLE_PER_CLASS`). At this phase, the Jandex index
knows the classes exist, but Hibernate's entity hierarchy metadata is not yet
available.

### Root cause

JPA entity inheritance resolution happens in a later build phase. A `@BuildStep`
that produces or consumes `AdditionalBeanBuildItem` runs before entity subclass
relationships are fully materialized. The Jandex index can find annotated classes,
but determining parent-child relationships across `@Inheritance` boundaries
requires the entity hierarchy to be resolved.

### Fix

Use `ValidationPhaseBuildItem` instead. This phase runs after Hibernate ORM has
resolved the full entity hierarchy. The `@BuildStep` can then use the Jandex
index to walk `@Entity` subclasses reliably:

```java
@BuildStep
void validateEntitySubclasses(ValidationPhaseBuildItem validation,
                               CombinedIndexBuildItem index) {
    // At this phase, Jandex reliably reflects the full entity hierarchy
    ClassInfo baseClass = index.getIndex()
        .getClassByName(DotName.createSimple("com.example.LedgerEntry"));
    Collection<ClassInfo> subclasses = index.getIndex()
        .getAllKnownSubclasses(baseClass.name());
    // subclasses now includes all @Entity subclasses
}
```

### Why this is non-obvious

The Jandex index itself is available at all build phases — `getAllKnownSubclasses()`
returns results at any phase. The issue is that the results are incomplete during
early phases for entity classes using `@Inheritance`. A developer writing an
extension-level validation naturally reaches for the earliest phase that has
Jandex access, and gets no error — just silently incomplete results.
