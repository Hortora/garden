### Symptom
`NoClassDefFoundError: io/quarkus/vertx/http/runtime/HttpBuildTimeConfig` during
Quarkus augmentation (`quarkus:build` or `quarkus:dev`). The stack trace points at
SmallRye `ConfigMappingInterface.createConfigurationInterface` — looks like a core
Quarkus classpath problem.

### Context
Project uses Quarkus 3.32.2 with `quarkus-quinoa:2.4.7` (a Quarkiverse extension
for serving SPAs). The project compiles cleanly — the error only appears during
Quarkus augmentation when the build step processes extension config mappings.

### What was tried (didn't work)
1. Checked `dependency:tree` — all `quarkus-vertx-http` resolved to 3.32.2 ✓
2. Scanned all casehub jars for `HttpBuildTimeConfig` in bytecode — not found
3. Reinstalled upstream deps — no change
4. Checked Quarkus version alignment across all upstream modules — all 3.32.2

### Root cause
`HttpBuildTimeConfig` was renamed to `VertxHttpBuildTimeConfig` between Quarkus 3.12
and 3.32. The BOM correctly manages the `quarkus-vertx-http` runtime jar to 3.32.2,
but Quarkiverse extensions package their own `META-INF/quarkus-config-roots.list`
inside the deployment jar. This file lists config mapping classes by fully qualified
name. When quinoa's deployment jar (built against 3.12.0) registers
`io.quarkus.vertx.http.runtime.HttpBuildTimeConfig`, SmallRye config loads it
reflectively during augmentation — triggering `ClassNotFoundException` for the
renamed class.

### Fix
Bump the Quarkiverse extension to a version built against the project's Quarkus
version. Manage the version at the BOM level so all apps in the organisation align:

```xml
<!-- Parent BOM properties -->
<quarkus-quinoa.version>2.8.3</quarkus-quinoa.version>

<!-- Parent BOM dependencyManagement -->
<dependency>
    <groupId>io.quarkiverse.quinoa</groupId>
    <artifactId>quarkus-quinoa</artifactId>
    <version>${quarkus-quinoa.version}</version>
</dependency>
```

### Diagnosis technique
1. Check the Quarkus jar for the missing class:
   `jar tf quarkus-vertx-http-3.32.2.jar | grep HttpBuildTimeConfig`
   → reveals `VertxHttpBuildTimeConfig` (renamed)

2. Scan non-class files in all jars for the old name:
   Search `META-INF/quarkus-config-roots.list` and `quarkus-javadoc.properties`
   across all dependency jars for the missing class name string

3. Check `dependency:tree -Dverbose` for `version managed from` entries:
   `(version managed from 3.12.0; omitted for duplicate)` reveals which
   extension was built against the old version

### Why this is non-obvious
The BOM does its job — the runtime jar is the correct version. The failure is in
the deployment jar's embedded resource file, which is opaque to dependency
management. The error message points at a core Quarkus class, not at the
Quarkiverse extension. `dependency:tree` shows everything resolved to 3.32.2,
giving false confidence that versions are aligned.

### Why this fix
Bumping the Quarkiverse extension version is the only fix — you can't patch the
deployment jar's `quarkus-config-roots.list` from outside. Managing at the BOM
level prevents the same mismatch in other apps using the same extension.

### Alternatives considered
- Excluding the old deployment jar and providing a patched one — too fragile,
  breaks on any extension update
- Pinning `quarkus-vertx-http-deployment` explicitly — doesn't help, the
  config-roots.list is in the Quarkiverse extension's deployment jar, not
  Quarkus's own deployment jar
