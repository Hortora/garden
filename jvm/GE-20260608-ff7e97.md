### What it is

`@OutputGuardrails(maxRetries = 2)` results in 2 total validation attempts, not "1 initial + 2 retries = 3 total." The parameter name `maxRetries` and the upstream Javadoc ("The maximum number of retries to perform") strongly suggest retries-after-first semantics, but the actual behavior is total-attempts.

### Why this matters

Test assertions expecting `maxRetries + 1` validations will fail. A developer setting `maxRetries = 1` expects one retry after the first failure (2 total) but gets only 1 total attempt — effectively disabling retries entirely. This off-by-one confusion affects both test code and production retry budget calculations.

### How to discover this

Only by writing a test with an always-rejecting `OutputGuardrail`, counting invocations with `AtomicInteger`, and comparing the count against `maxRetries`. The upstream `OutputGuardrailExecutor` decrements a counter starting from `maxRetries`, not `maxRetries + 1`. There is no documentation clarifying the semantics.

```java
@OutputGuardrails(value = AlwaysRejectGuardrail.class, maxRetries = 2)
String process(String input);

// AlwaysRejectGuardrail.validate() returns retry("rejected") every time
// Expected (intuitive): guardrail.validationCount() == 3  (1 initial + 2 retries)
// Actual: guardrail.validationCount() == 2  (2 total attempts)
```

### Why this is non-obvious

The parameter name `maxRetries` universally means "maximum number of retries after the initial attempt" in Java conventions (e.g., Spring Retry's `maxAttempts` vs `maxRetries`, Resilience4j's retry count). The upstream implementation contradicts this convention without documenting the deviation. The Quarkus wrapper `QuarkusOutputGuardrailsConfig` defaults to 3 (its own `MAX_RETRIES_DEFAULT`), adding further confusion about which default applies.
