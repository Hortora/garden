## Symptom

Tests pass in isolation but fail in the full suite with `AbstractMethodError`:

```
java.lang.AbstractMethodError: Receiver class io.example.testing.InMemoryFooStore
    does not define or inherit an implementation of the resolved method
    'abstract java.util.Map findAll()' of interface io.example.runtime.store.FooStore.
```

The module compiles cleanly. The error appears only at runtime, and only when the new interface
method (`findAll()`) is actually invoked.

## Context

Occurs when `mvn -U` (force-update SNAPSHOT metadata) pulls a newer build of a SNAPSHOT runtime
artifact that added a new interface method, while the testing artifact (`*-testing`) was compiled
against the old interface and was not rebuilt.

Maven fetches runtime and testing jars independently — if the SNAPSHOT version string is the same
(`0.2-SNAPSHOT`) for both, but different builds were published at different times, they can be
out of sync. `mvn -U` updates whichever metadata it resolves first; the other artifact may remain
stale from the local cache.

Hit in practice: `casehub-qhorus` runtime added `findAll()` to `ChannelBindingStore`; the
`casehub-qhorus-testing` jar (`InMemoryChannelBindingStore`) did not implement it. After
`mvn -U` downloaded the new runtime SNAPSHOT, tests using the testing artifact failed at runtime
with `AbstractMethodError`.

## What Was Tried (Didn't Work)

- `mvn -U` alone — downloads fresh runtime metadata but can serve stale testing artifact
- Running tests individually — passes because the new method is not invoked in isolation
- Checking compile output — no errors; the testing jar's compiled bytecode doesn't reference
  the new method

## Root Cause

SNAPSHOT versions are mutable — the same `0.2-SNAPSHOT` coordinate can resolve to different
bytecode at different times. When a new interface method is added to the runtime artifact and a
new build is published, the testing artifact compiled against the old interface becomes
structurally incompatible at runtime, even though the version string is identical.

`AbstractMethodError` is thrown by the JVM when it tries to dispatch an interface method call and
the implementing class doesn't have that method — it's a bytecode mismatch, not a missing class.
The error message names the missing method and the interface, which helps, but the root cause
(SNAPSHOT version skew) is not mentioned.

## Fix

Rebuild the affected repo from source and install locally:

```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn install -DskipTests -q \
  -f /path/to/affected-repo/pom.xml
```

This builds both the runtime and testing jars from the same source, guaranteeing interface
consistency. Both artifacts will share the same bytecode snapshot of the interface.

## Why This Is Non-Obvious

`AbstractMethodError` looks like a classpath conflict or a structurally broken class, not a
SNAPSHOT skew problem. The compile succeeds because the compiler saw the OLD interface (without
the new method) when building the testing artifact — no error surfaces until the JVM tries to
dispatch the call at runtime.

The `mvn -U` flag is supposed to "force update" — it is natural to assume it resolves all
related artifacts consistently. It does not: metadata update order is non-deterministic for
multi-artifact SNAPSHOTs, and local cache can serve one artifact while downloading another.

Distinguishing from GE-20260424-275fdc (stale SNAPSHOT from version rename): in that case the
version string itself is wrong (`1.0.0-SNAPSHOT` vs `0.2-SNAPSHOT`). Here the version string is
identical on both artifacts — the skew is between builds of the same version.
