In casehub-qhorus, `ChannelService.create()` persists the channel to the database but does NOT register it in `ChannelGateway.registry`. When `MessageService.dispatch()` subsequently calls `channelGateway.fanOut(ch.id, ch.name, message)`, the implementation does `registry.getOrDefault(channelId, List.of())` — returning an empty list for any channel that was created dynamically. No backends receive the fan-out. No error is thrown. `fanOut()` completes successfully with zero backend calls.

The `ChannelGateway.registry` is only populated by:
1. `onStart(@Observes StartupEvent)` — restores all channels that existed at boot time
2. `initChannel()` — called explicitly by the `create_channel` MCP tool
3. `registerBackend()` — registers an individual backend on an already-known channel

Channels created via `ChannelService.create()` after startup bypass all of these.

### Symptom

`registerBackend()` returns without error. `fanOut()` is called with the correct `channelId`. `ChannelBackend.post()` is never invoked. No exception or warning.

### Root cause

`ChannelService` is the persistence layer. `ChannelGateway` is the runtime dispatch layer. They are decoupled intentionally — but this means dynamic channel creation requires an explicit `initChannel()` call to wire the gateway.

### Fix

Call `channelGateway.initChannel(channel.id, new ChannelRef(channel.id, channel.name))` immediately after `channelService.create()` when fan-out to registered backends is required. Or use the `create_channel` MCP tool which calls `initChannel()` as part of its implementation.

### Why this is non-obvious

`ChannelService` and `ChannelGateway` have overlapping names and purposes. The startup recovery (`onStart`) creates the impression that the gateway always mirrors the DB. The absence of any error or warning from `fanOut()` on an unregistered channel makes the failure invisible. The symptom (backend.post() never called) appears to be a CDI scoping issue or backend registration problem rather than a missing gateway initialization.
