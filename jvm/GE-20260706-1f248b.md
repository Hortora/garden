## Symptom

Running `mvn dependency:resolve -U` to fix a compilation error from one upstream repo pulls new SNAPSHOT versions of ALL upstream repos. The fix for repo A's break introduces repo B's break, which introduces repo C's break. Each resolution cycle makes the situation worse.

## Context

A multi-repo platform where repos A (engine), B (qhorus), C (ledger) all publish `0.2-SNAPSHOT` and are being refactored in parallel (package renames, reactive-to-blocking migration, class-to-record conversion). The consumer app (devtown) depends on all three.

## What was tried (didn't work)

- Fix each compilation error as it appears — each `-U` resolution pulls the latest from ALL repos, introducing new breaks from repos that weren't the original problem.
- Selective dependency update — Maven's `-U` flag updates ALL SNAPSHOT metadata, not just the dependency you want.

## Root cause

Maven's SNAPSHOT resolution is all-or-nothing per repository. `-U` refreshes metadata for every SNAPSHOT artifact, not just the one causing the error. When multiple upstream repos are mid-refactor, the latest SNAPSHOT of each may be mutually incompatible or incompatible with the consumer's current code.

## Fix

Pin SNAPSHOT versions to the last-known-working set before starting the migration. Update one upstream repo at a time, fixing all breaks from that repo before pulling the next. Never run `-U` mid-fix session — it resets progress.

Alternatively, use a BOM (Bill of Materials) parent that publishes coordinated SNAPSHOT versions across all repos, ensuring mutual compatibility.

## Why this is non-obvious

The instinct when a dependency breaks is to pull the latest version. In a multi-repo SNAPSHOT environment, "latest" is a moving target that may be internally inconsistent. The fix (don't pull latest) feels counterintuitive.
