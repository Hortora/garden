## Symptom

A resolution method fix is applied to two of three rendering paths that call it. The third path silently continues using the old (broken) resolution — no compile error, no test failure if the affected code path isn't specifically tested.

## Context

In `EidosRenderPipeline`, `vocabUriForSlot()` was added to fix a `domainVocabulary` fallback gap. It was applied to:
- `buildDescriptorPayload()` — the LLM payload path (actively being modified this session)
- `assembleA2aCard()` — the feature under development (where the change originated)
- ❌ missed: `assembleMarkdownStructural()` — the structural renderer (not in focus)

An agent configured with only `domainVocabulary` got correct slot vocabulary context in two render formats but raw slot string in structural markdown. No test covered this combination.

## Root cause

Developers naturally fix the rendering paths they are actively looking at. The mental model is "I fixed the resolution method and its callers" — but in practice "callers" means "the callers I examined during this change", not "all callers of the old access pattern".

The structural renderer was in a separate method not part of the feature being built, so it was not in the developer's attention window during the fix.

## Fix

Before completing any fix to a shared resolution method, find all callers of the old access pattern — not just the method you changed:

```
IntelliJ: Find Usages on the old access pattern
          (e.g. the field access `descriptor.slotVocabulary()` that the new method replaces)
```

In this case: searching for `slotVocabulary()` across `EidosRenderPipeline` revealed `assembleMarkdownStructural()` as the third caller. All callers need the updated resolution.

The code reviewer who catches this will frame it as "the spec said fix both X and Y, but Z was also there." The implementer will not have noticed Z because Z was not their focus.

## Why this is non-obvious

The fix feels complete because it works on the paths you tested. Multi-renderer systems have a structural property that makes this systematically dangerous: each renderer reads the same model but lives in a separate method. There is no compile-time signal that a third renderer was missed — the old code still compiles, and it still runs correctly for agents that have the specific vocabulary field set explicitly. Only agents using the fallback path expose the gap, and only if structural markdown is the tested format.
