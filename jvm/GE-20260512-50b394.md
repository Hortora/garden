## Use @TestTransaction + unique identifiers to prevent @Scheduled interference in Quarkus tests

### What it achieves

Prevents `@Scheduled` jobs from picking up test data and firing spurious side effects (alerts, expiry, event emission) that corrupt test assertions.

### Context

`@Scheduled` methods run in their own separate Quarkus thread with their own transaction. They **cannot** see data inside a `@TestTransaction` (uncommitted), but they **can** see committed data — including entities created in `@BeforeEach` via transactional service calls. A scheduler racing against a test that created committed setup data will fire on that data and corrupt subsequent assertions.

### The technique

**Strategy 1 — @TestTransaction on test methods:** Uncommitted data is invisible to the scheduler's separate transaction. Use this when the test doesn't need to verify HTTP responses.

```java
@Test
@TestTransaction  // scheduler cannot see uncommitted data
void myTest() {
    String name = "test-" + UUID.randomUUID();  // unique per test
    // set up, assert — scheduler sees nothing
}
```

**Strategy 2 — Unique identifiers per test:** When setup must commit (e.g. REST-verified tests), use UUIDs or random suffixes so the scheduler won't match entities from other tests:

```java
String entityName = "threshold-test-" + UUID.randomUUID();
```

**Strategy 3 — Don't call the scheduled service directly:** If the test invokes the same service the scheduler drives, both will fire and double-trigger side effects.

### Why this is non-obvious

The scheduler runs on a separate thread and its transaction is completely independent. Developers writing `@BeforeEach` setup don't expect a background thread to pick up that data mid-test. The interference is timing-dependent and produces flaky, hard-to-reproduce failures.
