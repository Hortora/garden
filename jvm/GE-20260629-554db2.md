### Symptom
998 tests fail with `NoClassDefFoundError: org/assertj/core/api/Assertions` after slimming
a child POM's `<dependencyManagement>`. Other sibling modules resolve assertj fine from the
same parent.

### Context
`casehub-parent` imports the Quarkus BOM and explicitly manages `assertj-core:3.27.7` in
`<dependencyManagement>`. All child modules declare `<dependency>assertj-core</dependency>`
without a version — parent manages it.

One child (`quarkmind`) also re-imported the Quarkus BOM in its own `<dependencyManagement>`
(alongside a Drools BOM). Maven processes `<dependencyManagement>` imports in declaration order —
the child's Quarkus BOM import overrides the parent's management entries. Since the Quarkus BOM
does not manage assertj, the version pin is lost. The child's `<dependency>assertj-core</dependency>`
has no version to resolve and fails silently at compile time.

### What was tried (didn't work)
- Adding assertj-core back to child `<dependencies>` with explicit `<scope>test</scope>` — works
  as a workaround but masks the root cause
- Filing an issue to move assertj from parent's `<dependencyManagement>` to `<dependencies>` —
  wrong fix; parent setup is correct and works for all other modules

### Root cause
Maven's BOM import in `<dependencyManagement>` is not additive with inherited parent management —
a child-level BOM import can shadow parent-level version pins for any dependency not managed by
the re-imported BOM.

### Fix
Remove the redundant BOM import from the child's `<dependencyManagement>`. If the parent already
imports a BOM, the child inherits it — re-importing in the child overrides the parent's management.
Keep only BOMs that the parent does NOT import (e.g. Drools BOM, which is quarkmind-specific).

### Why this is non-obvious
The symptom (`NoClassDefFoundError`) points directly at a missing dependency — the natural fix is
to add it back. The actual cause (BOM import ordering) is invisible in the error output and requires
understanding Maven's `<dependencyManagement>` merge semantics across parent-child hierarchies.

### Alternatives considered
- Moving assertj to parent `<dependencies>` (auto-include for all children) — wrong; forces the
  dependency on modules that may not want it, and the parent's `<dependencyManagement>` pattern
  is correct and works for all other modules
