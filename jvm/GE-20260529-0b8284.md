## Symptom

Wrapping a streaming callback API into `Uni<T>` using `Uni.createFrom().emitter()` appears to work — but the Uni has no built-in timeout, the risk of calling `emitter.complete()` in both `onCompleteResponse` and `onError` exists, and `emitter()` semantics are for push streams, not one-shot results.

## Context

When bridging a callback-based async API (e.g. `StreamingChatModel.chat(request, handler)` from LangChain4j) into a Mutiny `Uni<T>`, `emitter()` looks like the natural choice. But it has three problems for one-shot use: (1) no built-in timeout — a hung provider hangs the Uni indefinitely, (2) it is designed for push streams where `emit()` may be called many times, (3) both `onCompleteResponse` and `onError` callbacks could call `emitter.complete()` by mistake, causing IllegalStateException.

## Root Cause

`Uni.createFrom().emitter()` is a push-stream primitive. For a one-shot async result (exactly one completion or error), the correct primitive is `CompletableFuture` — which has timeout support via `orTimeout()`, clear single-completion semantics, and maps cleanly to Mutiny via `Uni.createFrom().completionStage()`.

## Fix

```java
CompletableFuture<Optional<T>> future = new CompletableFuture<>();
llm.chat(request, new StreamingChatResponseHandler() {
    @Override
    public void onCompleteResponse(ChatResponse response) {
        future.complete(parseOrEmpty(response.aiMessage().text()));
    }
    @Override
    public void onError(Throwable error) {
        future.completeExceptionally(error);  // not future.complete(empty) — let onFailure handle it
    }
});
return Uni.createFrom().completionStage(
        future.orTimeout(TIMEOUT_SECONDS, TimeUnit.SECONDS))
    .onFailure().recoverWithItem(e -> {
        log.warn("Streaming LLM failed (" + e.getMessage() + "), falling back");
        return Optional.empty();
    });
```

`onError` calls `completeExceptionally` — the `onFailure().recoverWithItem()` in Mutiny handles both timeout and LLM errors uniformly, and the fallback is explicit at the Mutiny level.

## Why This Is Non-Obvious

`emitter()` is the most prominent `Uni.createFrom()` variant for async work and the name sounds right for a callback-based API. The problems (no timeout, push-stream semantics, double-complete risk) only surface under failure conditions — in the happy path, `emitter()` and `completionStage()` behave identically.
