### What happened

`casehubio/aml` CI failed with `403 Forbidden` from Maven Central when downloading the Quarkus platform BOM:

```
Could not transfer artifact io.quarkus.platform:quarkus-bom:pom:3.32.2 from/to central
(https://repo.maven.apache.org/maven2): status code: 403
```

The artifact exists on Maven Central (200 from curl on any residential IP). Other repos in the same org (engine, ledger) were green simultaneously.

### Root cause

The AML repo had no `<dependencyManagement>` in its own `pom.xml` — it relied entirely on the parent POM (in GitHub Packages) for the Quarkus BOM import. Maven's resolution order was:

1. Download `casehub-parent` from GitHub Packages (Bearer token sent)
2. Parent POM imports `quarkus-bom` — Maven tries Maven Central **while the GitHub auth session is live**

Peer repos (engine, ledger) defined their own `<dependencyManagement>` with a local version property, so Maven resolved `quarkus-bom` from Central **before** contacting GitHub Packages — no auth header contamination.

The hypothesis: Maven's HTTP connection pool or CloudFlare's bot detection caused the Central request to receive the GitHub Bearer token, resulting in a 403. Evidence: 100ms failure vs 150ms success, identical workflows, simultaneous green on other repos.

### Fix

Add the BOM import directly to the consuming module's root `pom.xml` with a **locally-defined** version property:

```xml
<properties>
  <!-- Pinned locally so Maven resolves quarkus-bom from central BEFORE
       contacting GitHub Packages for casehub-parent. Without this, Maven
       downloads GitHub Packages first (with Bearer token), then hits Central
       while the auth session is live — causing 403. -->
  <quarkus.platform.version>3.32.2</quarkus.platform.version>
</properties>

<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>io.quarkus.platform</groupId>
      <artifactId>quarkus-bom</artifactId>
      <version>${quarkus.platform.version}</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>
```

Using a **local** property (not inherited from parent) means Maven can resolve the BOM version without downloading the parent first, so the Central download happens cold — before any GitHub Packages connection.

### Why this is non-obvious

The symptom looks like a Maven Central outage or IP ban. Other repos with identical workflows pass. The fix requires understanding Maven's resolution ordering with parent POM inheritance vs local property resolution — a distinction that rarely matters in single-repo projects.

### Constraints

Applies when: repo uses GitHub Packages wildcard URL (`https://maven.pkg.github.com/org/*`) requiring Bearer auth, AND parent POM (in GitHub Packages) contains the BOM import, AND child module has no local BOM import. The local property creates a maintenance burden: must stay in sync with the parent's version when bumping Quarkus.
