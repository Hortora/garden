## Java record compact constructor as single invariant enforcement gate

When a record has cross-field constraints (e.g. two sets must not intersect, or one field is only valid when another is set), place the validation in the compact constructor. This makes invalid state unrepresentable: every construction path — whether called from production code, tests, factories, or MCP tools — hits the same validation with no escape hatch.

### Example

```java
public record ChannelCreateRequest(
        String allowedTypes,
        String deniedTypes,
        // ... other fields
) {
    public ChannelCreateRequest {
        // Validates type names and enforces allowedTypes ∩ deniedTypes = ∅
        Set<MessageType> allowed = MessageType.parseTypes(allowedTypes); // throws on invalid name
        Set<MessageType> denied  = MessageType.parseTypes(deniedTypes);  // throws on invalid name
        if (!allowed.isEmpty() && !denied.isEmpty()) {
            Set<MessageType> overlap = new HashSet<>(allowed);
            overlap.retainAll(denied);
            if (!overlap.isEmpty()) {
                throw new IllegalArgumentException(
                        "allowedTypes and deniedTypes must not intersect. Overlap: " + overlap);
            }
        }
    }
}
```

Any call to `new ChannelCreateRequest(...)` with conflicting types throws `IllegalArgumentException` immediately, before any database interaction.

### Why this is non-obvious

Most developers reach for service-layer validation (a `validate()` method in a service class) or a static factory method. Both approaches have escape hatches: callers can bypass the factory and invoke the constructor directly; service validation only fires if callers remember to call the service. The compact constructor has no bypass — it is the constructor. This distinction matters in large codebases where the record is constructed in many places (MCP tools, internal services, tests, auto-creation policies) — the invariant holds in all of them without any coordination.

### Trade-offs

- **Good:** zero escape hatches; invariant is colocated with the data; no dependency on CDI or external state
- **Good:** eliminates an entire class of "works in code path A but not B" bugs
- **Avoid for:** invariants that require I/O, CDI injection, or external lookups — compact constructors must be pure

### Alternatives considered

- **Service-layer `validate()` call:** requires every caller to remember to call it; callers can bypass it
- **Static factory method:** callers can still invoke the raw constructor directly; requires @VisibleForTesting hacks or package-private visibility tricks
- **`@Validated` / Bean Validation:** requires Quarkus/Spring context; not available in plain unit tests or static utility code
