### Symptom
`KieMemoryCompilerException: not a statement` when evaluating a block expression like `var threshold = 10; x + threshold;`.

### Context
Any MVEL3 block expression (detected by `.block()` builder method or `expression.indexOf(';') > 0` heuristic) where the last statement is a value-producing expression rather than an assignment or method call.

### What was tried (didn't work)
- `"var threshold = 10; x + threshold"` — parse error: missing `;` at `}`
- `"var threshold = 10; x + threshold;"` — javac error: `not a statement`

### Root cause
MVEL3 transpiles block expressions to Java method bodies. The generated Java wraps the block content in `{ ... }`. An arithmetic expression like `x + threshold;` is not a valid Java statement — javac rejects it. MVEL3 does not auto-wrap the last expression in a `return` statement.

MVEL3's own `executeExpression()` method (MVEL.java line 222) uses `expr.indexOf(';') > 0` as the canonical block detection heuristic.

### Fix
Use explicit `return` for the last expression in any block:

```java
// Wrong — "not a statement"
"var threshold = 10; x + threshold;"

// Correct
"var threshold = 10; return x + threshold;"

// Control flow also needs explicit return
"var r = \"unknown\"; if (age > 18) { r = \"adult\"; } else { r = \"minor\"; } return r;"
```

### Why this is non-obvious
MVEL2 (interpreter-based) implicitly returned the last evaluated expression. Developers migrating from MVEL2 or coming from expression languages like Groovy/Kotlin expect the last expression to be the return value. MVEL3's transpiler generates Java where this is not valid — the error message (`not a statement`) does not hint at the need for `return`.
