### Symptom
Models configured with `maxSequenceLength > 512` (e.g. BGE-M3 at 8192) silently
truncate input at ~400 words. Only signal is a WARN log line:
`maxLength is greater then modelMaxLength, change to: 512`

### Context
`HuggingFaceTokenizer.newInstance()` accepts a `maxLength` option, but internally
DJL clamps it to `modelMaxLength`. When `modelMaxLength` is not explicitly set,
it defaults to 512 — regardless of what the underlying model actually supports.

### What was tried (didn't work)
Setting only `maxLength` to the desired sequence length (e.g. 8192). DJL accepts
the value without error but silently reduces it.

### Root cause
DJL's internal logic: `if (maxLength > modelMaxLength) maxLength = modelMaxLength`.
The `modelMaxLength` field defaults to 512 when not provided in the options map,
even though many models support much longer sequences.

### Fix
Set `modelMaxLength` alongside `maxLength` in the tokenizer options:
```java
Map.of("maxLength", String.valueOf(config.maxSequenceLength()),
       "modelMaxLength", String.valueOf(config.maxSequenceLength()),
       "truncation", "true",
       "padding", "false")
```

### Why this is non-obvious
The parameter name `maxLength` strongly implies it controls the maximum sequence
length. There is no error or exception — only a WARN log that is easily missed
in noisy startup output. The existence of a separate `modelMaxLength` parameter
that silently overrides `maxLength` is not documented in DJL's public API docs.

### Alternatives considered
- Modifying `tokenizer.json` to set `model_max_length` — unnecessary, the options map is sufficient
- Patching DJL — upstream fix would be better but this is a one-line workaround
