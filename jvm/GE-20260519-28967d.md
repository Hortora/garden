## ChannelGateway.receiveHumanMessage() rejects messages whose type isn't in the channel's allowedTypes

**Stack:** casehub-qhorus, Quarkus 3.32

**Symptom:** `io.casehub.qhorus.api.message.MessageTypeViolationException: Channel 'clinical/deviation/.../pi-oversight' does not permit DONE. Allowed: QUERY,COMMAND`

The exception fires inside `ChannelGateway.receiveHumanMessage()` when the `InboundNormaliser` maps the raw message to a type (e.g. DONE) that is not in the channel's `allowedTypes`.

**Context:** qhorus PI oversight channels are created with `allowedTypes = QUERY,COMMAND` — the intent is that clinical sends COMMANDs to the PI, and the PI can QUERY. But when the PI sends a structured JSON response (`{"decision":"APPROVED"}`), the `ClinicalInboundNormaliser` maps it to `MessageType.DONE`. `receiveHumanMessage()` then validates the normalised type against the channel's `allowedTypes` and rejects it.

### Root cause

`receiveHumanMessage()` validates the post-normalisation message type against the channel's declared `allowedTypes`. The assumption is that `allowedTypes` restricts only outbound sends — but it actually constrains all message types, including inbound human responses normalised by the `InboundNormaliser`.

### Fix

Include all response types (DONE, DECLINE) in the channel's `allowedTypes` at creation time:

```java
static final String CHANNEL_ALLOWED_TYPES = "QUERY,COMMAND,DONE,DECLINE";
```

### Why this is non-obvious

The name `allowedTypes` sounds like it governs what the *channel owner* can send. Human-response channels (oversight, approval flows) feel like one-way: the owner sends COMMANDs, humans respond. The idea that DONE/DECLINE must be explicitly allowed for the response path contradicts the mental model of a one-directional command channel.
