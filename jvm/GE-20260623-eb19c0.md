## mvn -U with GitHub Packages 401 silently corrupts local SNAPSHOT jars

### Symptom

Unit tests pass (structural checks like `assertInstanceOf`) but integration tests
fail with `NoClassDefFoundError` for classes that were present before running
`mvn -U dependency:resolve`. The missing classes were in a SNAPSHOT jar that worked
moments earlier.

### Context

Any Maven project using GitHub Packages as a SNAPSHOT repository. Occurs when the
GitHub token is expired, revoked, or not configured — GitHub Packages returns 401
on the download attempt.

### What was tried (didn't work)

Re-running `mvn -U` (same 401, same corruption). Checking the local `.m2` jar —
it exists but is smaller than expected (partial/empty artifact).

### Root cause

`mvn -U` attempts to download the latest SNAPSHOT from the remote repository.
When GitHub Packages returns 401, Maven does not preserve the existing local jar.
Instead it overwrites the `.m2` artifact with a partial or empty file. The old
working jar is gone — there is no rollback.

The unit test passing despite the broken jar is a secondary trap: structural tests
(`assertInstanceOf(WorkerFunction.Flow.class, ...)`) may resolve the type from a
different module on the test classpath, while integration tests that actually
instantiate and execute the classes fail with `NoClassDefFoundError`.

### Fix

Build the dependency locally from source:

```bash
# 1. Check out the dependency repo at the correct commit/branch
git -C /path/to/engine checkout main

# 2. Clean untracked files from other branches (critical — uncommitted files
#    from feature branches cause compilation failures)
git -C /path/to/engine clean -fd -- api/

# 3. Build and install to local .m2
mvn -f /path/to/engine/pom.xml install -pl api,common,runtime,flow -am -DskipTests -q
```

The `git clean -fd` step is essential: if the repo has been used for feature work,
untracked source files from other branches (e.g. new classes not yet committed)
will be picked up by the compiler and cause failures unrelated to the dependency
you are trying to build.

### Why non-obvious

`-U` semantics imply "check for updates" — a developer expects either a newer jar
(success) or the existing jar unchanged (failure). Silent replacement of a working
jar with a broken one is not the expected failure mode. The two-layer confusion
(unit tests pass, integration tests fail) delays diagnosis further.
