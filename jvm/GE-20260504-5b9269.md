### Context

You need to group a flat list of items by two keys simultaneously — e.g. group attestations
by `(capabilityTag, entryId)` to produce `Map<String, Map<UUID, List<T>>>`. The naive approach
groups by the first key (O(M)), then iterates the result and re-streams the full source
collection to filter by each second key — O(N×M) total.

### What developers usually do

```java
// O(N×M): group by key1, then re-stream source per key1 group
Map<String, List<T>> byKey1 = items.stream()
        .collect(Collectors.groupingBy(i -> i.key1));

for (String key1 : byKey1.keySet()) {
    Map<UUID, List<T>> byKey2 = sourceList.stream()
            .filter(i -> key1.equals(i.key1))
            .collect(Collectors.groupingBy(i -> i.key2));
    // use byKey2...
}
```

### The technique

Use nested `Collectors.groupingBy` in a single terminal operation:

```java
// O(M): single pass produces Map<String, Map<UUID, List<T>>>
Map<String, Map<UUID, List<LedgerAttestation>>> byCapabilityAndEntry =
        actorAttestations.stream()
                .filter(a -> !CapabilityTag.GLOBAL.equals(a.capabilityTag))
                .collect(Collectors.groupingBy(
                        a -> a.capabilityTag,
                        Collectors.groupingBy(a -> a.ledgerEntryId)));

// Consume directly — no secondary filtering needed
for (var capEntry : byCapabilityAndEntry.entrySet()) {
    String capabilityTag = capEntry.getKey();
    Map<UUID, List<LedgerAttestation>> capByEntry = capEntry.getValue();
    // capByEntry is already scoped to this capabilityTag — no re-filter
}
```

### Why this is non-obvious

The nested downstream collector `Collectors.groupingBy(a -> a.key2)` is documented but
the performance implication (one pass vs N passes) is not emphasised. Developers reaching
for "group by two keys" naturally think "group by key1, then for each group, filter and
group by key2" — which re-streams the source collection. The nested form produces both
groupings simultaneously in a single O(M) pass with no intermediate filtering.

The result type `Map<K1, Map<K2, List<T>>>` is also directly usable without reassembly —
each inner map is already the correctly-filtered `Map<K2, List<T>>` needed for the next step.

### Constraints

Works for any two-level grouping where the inner map values are lists. For three-level
nesting, add a third `Collectors.groupingBy` as the downstream of the second.
