### Symptom
Adding `casehub-openclaw-casehub` as a Maven dependency to `casehub-life` causes integration tests to fail with:
```
IllegalArgumentException: Unknown channel purpose 'coordination' for case <uuid>
```
No code changes — just the dependency addition.

### Context
`casehub-openclaw-casehub` contains `OpenClawCaseChannelProvider @ApplicationScoped` which implements `CaseChannelProvider`. The engine provides `NoOpCaseChannelProvider @DefaultBean`. When the openclaw jar lands on the classpath, Quarkus CDI discovers the `@ApplicationScoped` bean and activates it — overriding the `@DefaultBean` alternative that life was using.

`OpenClawCaseChannelProvider` validates channel purposes against `NormativeChannelLayout` (work/observe/oversight only). The engine passes "coordination" as a purpose — accepted by the NoOp, rejected by the OpenClaw provider.

Additionally, `OpenClawWorkerStatusListener`, `OpenClawAgentRegistry`, `OversightGateService`, and other heartbeat-mode beans activate unnecessarily. These are designed for OpenClaw's provisioner/heartbeat integration mode — not needed for the direct-call bridge.

### What was tried (didn't work)
1. Assuming only `DirectCallBridge` and `DirectCallDeliveryResource` would activate — wrong; all `@ApplicationScoped` beans in the jar activate
2. Excluding only the obviously conflicting `OpenClawCaseChannelProvider` — insufficient; other beans (WorkerStatusListener, AgentRegistry) also caused interference

### Root cause
Quarkus CDI discovers ALL beans in all JARs on the classpath. An `@ApplicationScoped` bean always takes precedence over a `@DefaultBean` for the same type. There is no mechanism to selectively activate beans from a library JAR — it's all or nothing.

### Fix
Exclude all heartbeat/provisioner-mode beans from `casehub-openclaw-casehub` that life doesn't need. Keep only `DirectCallBridge` and `DirectCallDeliveryResource`:

```properties
quarkus.arc.exclude-types=\
  io.casehub.openclaw.casehub.OpenClawCaseChannelProvider,\
  io.casehub.openclaw.casehub.ReactiveOpenClawCaseChannelProvider,\
  io.casehub.openclaw.casehub.OpenClawWorkerProvisioner,\
  io.casehub.openclaw.casehub.ReactiveOpenClawWorkerProvisioner,\
  io.casehub.openclaw.casehub.OpenClawChannelBackend,\
  io.casehub.openclaw.casehub.ChannelContextWindowObserver,\
  io.casehub.openclaw.casehub.OpenClawAgentRegistry,\
  io.casehub.openclaw.casehub.OpenClawCasehubConfig,\
  io.casehub.openclaw.casehub.OpenClawWorkerStatusListener,\
  io.casehub.openclaw.casehub.OversightGateDispatcher,\
  io.casehub.openclaw.casehub.OversightGateService
```

Must be applied to BOTH `application.properties` and `test/resources/application.properties`.

### Why this is non-obvious
The dependency was added for `DirectCallBridge` and `DirectCallDeliveryResource` only. The expectation was that unused beans would remain dormant. In fact, CDI activates every discovered bean — and `@ApplicationScoped` beats `@DefaultBean` unconditionally. The error message ("Unknown channel purpose") gives no indication that the root cause is a CDI bean activation from a transitive dependency.
