### What it is

When a Mutiny pipeline dispatches expensive work to a worker pool via `emitOn(Infrastructure.getDefaultWorkerPool())`, any guard condition or pre-computation that might short-circuit the expensive work must be placed BEFORE the `emitOn()` call, not inside the lambda.

### How it works

A guard inside the lambda runs on the worker pool thread — paying the thread-dispatch overhead and potentially entering the expensive computation path before the guard fires.

```java
// ❌ Wrong: guard inside lambda — pays emitOn overhead before checking guard
return Uni.createFrom().voidItem()
    .emitOn(Infrastructure.getDefaultWorkerPool())
    .map(ignored -> {
        if (shouldEscalate()) {           // too late — already on worker thread
            return AgentAssignment.escalate(...);
        }
        float[] embedding = embed(text);  // expensive — pays this even if guard fires
        return score(embedding);
    });

// ✅ Correct: guard before emitOn — short-circuits without entering worker pool
if (shouldEscalate()) {
    return Uni.createFrom().item(AgentAssignment.escalate(...));
}

// Eligible computed here — lambda captures it, not the full list
List<Item> eligible = computeEligible();

return Uni.createFrom().voidItem()
    .emitOn(Infrastructure.getDefaultWorkerPool())
    .map(ignored -> {
        float[] embedding = embed(queryText);  // only paid when needed
        return score(eligible, embedding);     // works on pre-filtered list
    });
```

### Why this is non-obvious

The natural structure is to put all logic inside the lambda so it runs on the appropriate thread. The fix requires splitting logic across two "phases" — synchronous pre-screening on the calling thread (safe because it's cheap), and expensive computation on the worker pool thread. The split is non-obvious because:

1. Devs often think of the lambda as "the place where decisions get made"
2. The cost of entering the worker pool before an early-return guard seems minor but compounds when many requests trigger the guard
3. Pre-computed values captured by the lambda (e.g. `eligible` list) require understanding Mutiny's capture semantics — the lambda captures the value at the time it's defined, not when it executes

### Context where it applies

Any Mutiny pipeline that:
1. Dispatches to a worker pool for expensive I/O-bound or compute-bound work
2. Has a guard condition that can short-circuit before the expensive work
3. May pre-filter or transform the input before the expensive computation

Common examples: embedding-based routing (compute query + agent vectors), LLM inference with early reject, RAG retrieval with pre-screening.

### Constraints

The pre-screen must be thread-safe — it runs on the calling thread (often the Vert.x event loop). Keep pre-screen logic free of blocking calls. The eligible computation must also be non-blocking.
