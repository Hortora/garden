## Symptom

New methods added to repo A source compile and test fine within repo A. Writing tests
in repo B that call those new methods immediately fails with "cannot find symbol" or
"no suitable method found" — as if the implementation is missing or never written.

## Root Cause

Repo B compiles against the installed `.jar` in `~/.m2/repository`, not repo A's
source tree. Maven resolves SNAPSHOT deps from the local repository, not from source
directories of sibling projects. Until `mvn install` runs in repo A, the local repo
holds a stale JAR without the new methods.

## Fix

Run `mvn install -DskipTests` on repo A after implementing new methods, before writing
or running tests in repo B.

The correct cross-repo TDD cycle:

```
1. Write failing tests in repo A       → compile error: method doesn't exist yet
2. Implement in repo A                 → tests in A pass
3. mvn install -DskipTests on repo A  → local repo updated with new JAR
4. Write failing tests in repo B       → compile error: correct — feature not wired yet
5. Implement in repo B                 → tests in B pass
```

Skipping step 3 makes step 4 fail with a misleading error ("method not found") that
looks like the implementation is wrong, not like the install is missing.

## Why This Is Non-Obvious

The compile error ("cannot find symbol", "method not applicable") points at the
implementation as the problem. The natural response is to check the source of the
method in repo A, find it correct, and be confused. The actual cause — a stale JAR
in a completely separate location — is invisible from the error message and from
the source code.

This only affects inter-repo boundaries. Within a single multi-module Maven project,
the reactor resolves modules from source and the problem never occurs.

## Why This Fix

`mvn install -DskipTests` is the minimal operation that updates the local repo without
running tests (which may themselves depend on other infrastructure). The alternative —
pointing repo B's dependency at a file path instead of a Maven coordinate — breaks
the build for everyone else and is not viable for shared codebases.
