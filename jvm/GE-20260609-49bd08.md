### Symptom

A multi-tenant query returns data from all tenants instead of just the caller's tenant. The store-level filtering works correctly in most places, but specific code paths return unscoped results.

### Context

Application-level multi-tenancy implemented via CDI store interfaces that inject `CurrentPrincipal` and add `tenancyId` to WHERE clauses. Hibernate Panache entities also have static convenience methods (`Entity.list()`, `Entity.find()`, `Entity.findById()`).

### What was tried (didn't work)

Audited the store implementations — all correctly filter by tenancyId. The leak was in code that bypassed the store entirely.

### Root cause

Panache static methods like `WorkItem.list("parentId", parentId)` execute directly against the entity manager without going through the injected store. They have no access to the request-scoped `CurrentPrincipal` and no tenant filtering. Any code using static Panache calls in a multi-tenant app is executing unscoped queries.

### Fix

Replace every static Panache call with an equivalent store method that includes tenant filtering:

```java
// WRONG — bypasses tenant filtering
List<WorkItem> children = WorkItem.list("parentId", parentId);

// RIGHT — store method includes tenancyId in WHERE clause
List<WorkItem> children = workItemStore.findByParentId(parentId);
```

Add the missing store methods:

```java
// Store interface — default method for non-JPA backends
default List<WorkItem> findByParentId(UUID parentId) {
    return scanAll().stream()
        .filter(w -> parentId.equals(w.getParentId()))
        .toList();
}

// JPA implementation — tenant-scoped query
@Override
public List<WorkItem> findByParentId(UUID parentId) {
    return list("parentId = ?1 and tenancyId = ?2",
        parentId, currentPrincipal.tenancyId());
}
```

### Why this is non-obvious

Panache's static methods are the documented, idiomatic way to query entities in Quarkus. They appear in tutorials, quickstarts, and migration guides. Nothing warns that they become an unscoped data leak when combined with application-level multi-tenancy. The static API looks identical to the instance API but has fundamentally different scoping behaviour.

### Alternatives considered

- **Hibernate @Filter with @FilterDef**: adds tenant filtering at the entity level, but requires session-level filter activation which has its own lifecycle issues
- **PostgreSQL Row-Level Security**: defence-in-depth layer planned separately (#257) but doesn't replace application-level filtering for non-Postgres backends
- **Ban Panache static imports via linting**: catches new violations but doesn't fix existing ones; chosen as a complementary measure
