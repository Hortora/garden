## Symptom

A `@Transactional` method writes to both a JPA store and an in-memory `@Alternative` store. A downstream call in the same transaction throws an exception. After rollback:

- The JPA store is clean (the row was rolled back) ✅
- The in-memory store still shows the write ❌

Tests that assert on the in-memory store state after a simulated failure see stale data and pass or fail incorrectly.

## Context

`@Alternative @ApplicationScoped` in-memory stores (e.g. `MemoryPlanItemStore`) are used in `@QuarkusTest` to avoid requiring a real database. A `@Transactional` handler calls the in-memory store's `updateStatus()` before creating a JPA entity. The creation fails (simulated by a `FailingWorkItemStore`). The JTA transaction rolls back the JPA work, but the in-memory map write was already done and has no JTA enlistment — it stays.

## Root cause

JTA rollback only undoes work that enlisted in the transaction via `XAResource` or a `Synchronization`. A plain `ConcurrentHashMap` write does not enlist. There is no undo mechanism.

## Fix

**Design fix (preferred):** Reorder operations so the non-JTA store write happens *after* the JTA-participating work, and *only* if no exception occurred:

```java
// JPA work first (participates in @Transactional)
workItemService.create(request);
// In-memory write after — only reached if JPA work succeeded
planItemStore.save(caseId, planItemId, bindingName, RUNNING, createdAt);
item.markRunning();
```

If JPA work throws, the method exits before the in-memory write. If the in-memory write is inside the transaction but before JPA work and JPA work throws, the in-memory write is NOT rolled back.

**Test-only fix:** Call `store.clear()` in `@BeforeEach` to reset state between tests regardless of what happened in the previous test.

## Why this is non-obvious

`@Transactional` feels like a safety net — everything in the method gets rolled back. In reality, only JTA participants are rolled back. In-memory stores that substitute for JPA stores in tests are specifically designed to *not* require JTA, which means they also don't participate in rollback. The discrepancy only surfaces when testing failure paths.
