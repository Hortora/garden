## Symptom

`@QuarkusTest` fails at startup with `UnsatisfiedResolutionException` for types like `ActorDIDProvider`, `DIDResolver`, or `AgentCredentialValidator` ŌĆö even though the feature using these beans is disabled via `casehub.ledger.identity.tokenisation.enabled=false`. The test application.properties correctly sets the feature flag, but the app never starts.

## Context

A new SNAPSHOT of `casehub-ledger-runtime` added identity enricher beans (`IdentityCacheInvalidator`, `ActorDIDEnricher`, `ActorIdentityValidationEnricher`, `AgentIdentityVerificationService`, `ReactiveAgentIdentityVerificationService`, `LedgerIdentityEnforcementListener`). These beans inject `ActorDIDProvider`, `DIDResolver`, and `AgentCredentialValidator`. The test classpath for `casehub-engine-ledger` includes `casehub-ledger-runtime` but not the platform identity implementations (`casehub-platform-identity`) that provide no-op defaults.

## Root cause

Quarkus ARC performs full bean dependency validation at **augmentation time** (build time / test startup) ŌĆö not at runtime. Config flags like `casehub.ledger.identity.tokenisation.enabled=false` are runtime properties; Quarkus ARC cannot evaluate them during augmentation. The dep chain is always validated regardless of whether the feature will be active at runtime.

## Fix

Add **all** beans in the broken dependency chain to `quarkus.arc.exclude-types` in the test `application.properties`:

```properties
quarkus.arc.exclude-types=\
  io.casehub.ledger.runtime.service.identity.IdentityCacheInvalidator,\
  io.casehub.ledger.runtime.service.identity.ActorDIDEnricher,\
  io.casehub.ledger.runtime.service.identity.ActorIdentityValidationEnricher,\
  io.casehub.ledger.runtime.service.identity.ActorIdentityBindingObserver,\
  io.casehub.ledger.runtime.service.identity.AgentIdentityVerificationService,\
  io.casehub.ledger.runtime.service.identity.ReactiveAgentIdentityVerificationService,\
  io.casehub.ledger.runtime.service.identity.LedgerIdentityEnforcementListener
```

**Critical:** excluding one bean in a dependency chain causes its dependents to surface as the next unsatisfied dep. You must exclude the **entire chain** in one pass. The order does not matter, but completeness does ŌĆö adding beans one-at-a-time produces repeated failures.

## Why this is non-obvious

The runtime config flag disables the feature correctly in production. The test starts and CDI appears configured. The failure message (`UnsatisfiedResolutionException for ActorDIDProvider`) doesn't mention the ledger identity feature flag ŌĆö it looks like a missing CDI bean, not a disabled-feature artifact. The natural fix is to add the missing implementation, but the right fix is to exclude the beans that shouldn't be active in the test context at all.

### Alternatives considered

- Adding no-op `ActorDIDProvider` / `DIDResolver` stubs to the test sources: works but clutters the module with identity mocks that belong in `casehub-platform-identity`.
- Using `@IfBuildProfile` to conditionally disable beans: requires changing the library JAR, not the consumer test config.
