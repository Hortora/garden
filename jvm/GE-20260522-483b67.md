### Context
When a MongoDB document has a natural composite identity (e.g. scope + namespace + name),
the default approach is to use an `ObjectId` for `_id` and create a separate compound
unique index to enforce uniqueness. This requires index creation (out-of-band or in a
startup bean) and complicates upserts — `findOneAndUpdate` or `replaceOne` with a filter
must be used instead of simple `persist()`.

### Technique
Encode the composite key into a single string and use it as `@BsonId`. Choose a delimiter
that does not appear in any field value (e.g. `|` when fields use `/`, `.`, and alphanumeric
characters only):

```java
import org.bson.codecs.pojo.annotations.BsonId;
import io.quarkus.mongodb.panache.PanacheMongoEntityBase;
import io.quarkus.mongodb.panache.common.MongoEntity;

@MongoEntity(collection = "platform_preferences")
public class PreferenceDocument extends PanacheMongoEntityBase {

    @BsonId
    public String id;  // "{scope}|{namespace}|{name}|{subKey}"

    public String scope;
    public String namespace;
    public String name;
    public String subKey = "";
    public String value;

    public static String compoundId(String scope, String namespace,
                                    String name, String subKey) {
        return scope + "|" + namespace + "|" + name + "|" + subKey;
    }
}
```

**Benefits:**
- `_id` uniqueness is enforced at the BSON level — no separate unique index needed
- `persistOrUpdate()` becomes an upsert-by-natural-key — no `filter`/`findOne` required
- `_id` is human-readable and self-describing in the shell or Compass
- Removes a class of "duplicate key" vs "update missed" race conditions

**Upsert pattern:**
```java
PreferenceDocument doc = new PreferenceDocument();
doc.id = PreferenceDocument.compoundId(scope, namespace, name, subKey);
doc.scope = scope;
// ... set other fields ...
doc.persistOrUpdate();  // idempotent upsert by _id — correct by construction
```

### Why this is non-obvious
Most developers default to ObjectId for `_id` and reach for a separate unique index for
business-key uniqueness. The compound-key-as-string approach requires knowing that
`@BsonId` accepts any BSON-serialisable type (including `String`), and that
`persistOrUpdate()` in Panache MongoDB performs a `replaceOne` with `upsert: true`
keyed on `_id` — making the operation correct-by-construction when the `_id` is the
natural identity.
