### Context

Writing a reactive wrapper that delegates to a synchronous implementation — e.g.
`InMemoryReactiveAgentRegistry` delegating to `InMemoryAgentRegistry`.

### Technique

Use `Uni.createFrom().item(x).invoke(delegate::method).replaceWithVoid()` instead of
calling `delegate.method()` synchronously and then returning `Uni.createFrom().voidItem()`.

```java
// ❌ Synchronous-then-wrap (bad template):
@Override
public Uni<Void> register(AgentDescriptor descriptor) {
    delegate.register(descriptor);          // synchronous call outside any Uni
    return Uni.createFrom().voidItem();
}

// ✅ Idiomatic Mutiny pipeline:
@Override
public Uni<Void> register(AgentDescriptor descriptor) {
    return Uni.createFrom().item(descriptor)
        .invoke(delegate::register)         // synchronous work inside the pipeline
        .replaceWithVoid();
}
```

### Why this is non-obvious

Both approaches are **functionally identical** for in-memory implementations with no I/O —
the synchronous call completes immediately in either case. The difference is invisible at
runtime for this use case.

The problem is the pattern it sets. When a future implementer adds real I/O to the delegate
and follows the `delegate.call(); return Uni.voidItem()` template, they silently place
blocking work **outside the reactive subscription**. This can happen on the event-loop thread
and block all other requests.

The `invoke()` pipeline keeps the work inside the Mutiny subscription chain where thread
dispatch (`@Blocking`, `.runSubscriptionOn(executor)`) can safely govern it.

**Rule of thumb:** in reactive interfaces, never call blocking-capable methods outside a
Uni/Multi pipeline. `invoke()` is the right seam for synchronous side-effects inside a
pipeline.
