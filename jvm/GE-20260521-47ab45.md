## Symptom

A plain-Java (non-Quarkus) test that boots Flyway directly to verify a canonical
migration path fails with an unexpected count: `expected 8 but was 9`. The
production `src/main/resources` directory has exactly 8 migration files.

## Context

Writing a `FlywayLocationContractTest` that scans `classpath:db/ledger/migration`
to pin the canonical Flyway path for an extension. The project also has a
test-only SQL migration at `src/test/resources/db/ledger/migration/V1999__test_entry.sql`.

## Root cause

Maven Surefire puts both `src/main/resources` and `src/test/resources` on the test
classpath. When Flyway scans `classpath:db/ledger/migration`, it finds files from
**both** directories — production files from `src/main/resources/db/ledger/migration/`
and test-only files from `src/test/resources/db/ledger/migration/`.

The contract test intends to assert only production migrations, but test resources
bleed through the shared classpath path.

## Fix

Put test-only migrations under a **separate path** that does not overlap with
the production migration path:

```
src/test/resources/db/ledger/test-migration/V1999__test_entry.sql
```

Then configure the Quarkus test `application.properties` to scan both:
```properties
quarkus.flyway.locations=classpath:db/ledger/migration,classpath:db/ledger/test-migration
```

The contract test only scans `classpath:db/ledger/migration` and now sees
exactly the production set.

## Why this is non-obvious

When writing a unit test that boots Flyway directly, the Maven classpath
distinction between `main` and `test` resources disappears — both end up under
the same `classpath:` prefix. You expect the production path to contain only
production files, not test fixtures.
