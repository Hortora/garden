## Symptom

`EpistemicallyWeak` is never returned by `CapabilityHealth.probe()`, even when agents have declared `epistemicDomains` entries with low confidence scores for the relevant domain.

## Context

`CapabilityHealth.probe(AgentDescriptor descriptor, String capabilityTag, ProbeContext context)` has two semantically distinct `String` parameters:

- `capabilityTag` ā€” the name of the capability being checked (e.g. `"code-review"`)
- `ProbeContext.taskDomain` ā€” the subject domain of the task within that capability (e.g. `"rust"`)

Both are `String`. The names are different but easy to conflate at the call site, especially since both relate to the capability being dispatched.

## Root cause

Passing `capabilityTag` as `ProbeContext.taskDomain` causes `DefaultCapabilityHealth` to look up `epistemicDomains.get("code-review")`. The map keys are subject domains (`"java"`, `"rust"`) ā€” not capability names. The lookup returns null, no epistemic weakness is detected, and the agent silently passes the probe.

## Fix

Pass the actual subject domain of the task as `ProbeContext.taskDomain`, not the capability name:

```java
// Wrong ā€” conflates capability tag with task domain
ProbeContext ctx = ProbeContext.of(capabilityName);

// Correct ā€” taskDomain is the subject the agent needs confidence in
ProbeContext ctx = ProbeContext.of(taskSubjectDomain); // e.g. "rust"

// When subject domain isn't known at the call site, acknowledge the limitation:
ProbeContext ctx = ProbeContext.of(ProbeContext.UNKNOWN_DOMAIN); // EpistemicallyWeak won't trigger
```

If the calling layer doesn't yet have subject domain information, accept that `EpistemicallyWeak` requires richer task context to fire ā€” don't substitute the capability tag as a proxy.

## Why this is non-obvious

Both parameters are `String` and both relate to the capability being dispatched. The distinction ā€” "which capability" vs "what domain within that capability" ā€” is only visible in the `AgentCapability.epistemicDomains` map structure, which maps subject domains to confidence scores rather than capability names. There is no type-level enforcement of the distinction; nothing prevents passing the wrong value and no error is thrown when the lookup returns null.
