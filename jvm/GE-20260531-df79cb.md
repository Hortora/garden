## Context

Adding full-text search to a SQLite table using FTS5. The content is stored in a regular
table that also carries non-text columns (ids, timestamps, foreign keys). The goal is
relevance-ranked queries using a natural-language question, falling back to chronological
ordering when no question is provided.

## Technique

Use a **FTS5 content table** — a virtual table backed by the real table, kept in sync by
three triggers. Query with `MATCH` and order by the built-in `rank` column (BM25 score).

**Schema:**
```sql
-- Real table (owns the data)
CREATE TABLE IF NOT EXISTS memory_entry (
    memory_id  TEXT NOT NULL,
    text       TEXT NOT NULL,
    ...
    PRIMARY KEY (memory_id)
);

-- FTS5 virtual table backed by memory_entry
CREATE VIRTUAL TABLE IF NOT EXISTS memory_fts
    USING fts5(text, content='memory_entry', content_rowid='rowid');

-- Keep FTS in sync with the real table
CREATE TRIGGER IF NOT EXISTS memory_fts_ai AFTER INSERT ON memory_entry BEGIN
    INSERT INTO memory_fts(rowid, text) VALUES (new.rowid, new.text);
END;

CREATE TRIGGER IF NOT EXISTS memory_fts_ad AFTER DELETE ON memory_entry BEGIN
    INSERT INTO memory_fts(memory_fts, rowid, text) VALUES('delete', old.rowid, old.text);
END;

-- Defensive: include even if the main table is append-only
CREATE TRIGGER IF NOT EXISTS memory_fts_au AFTER UPDATE ON memory_entry BEGIN
    INSERT INTO memory_fts(memory_fts, rowid, text) VALUES('delete', old.rowid, old.text);
    INSERT INTO memory_fts(rowid, text) VALUES (new.rowid, new.text);
END;
```

**Relevance query (JDBC):**
```sql
SELECT m.*
FROM memory_entry m
JOIN memory_fts ON memory_fts.rowid = m.rowid
WHERE m.tenant_id = ?
  AND m.entity_id IN (?, ...)
  AND memory_fts MATCH ?
ORDER BY rank           -- built-in BM25 column; negative; ASC = most relevant first
LIMIT ?
```

**Question sanitisation before MATCH binding:**
```java
private static final String FTS_OPERATOR_CHARS = "[\"*^:()+\\-]";

private String sanitiseForFts(String question) {
    return question.replaceAll(FTS_OPERATOR_CHARS, " ").trim().replaceAll("\\s+", " ");
}
```

Strip operator characters (`"`, `*`, `^`, `:`, `(`, `)`, `+`, `-`) to prevent FTS5 syntax
errors. Do NOT phrase-wrap the sanitised string — wrapping the entire question in double
quotes makes FTS5 treat it as an exact phrase query, which is wrong for semantic memory
searches where you want AND-of-terms matching.

**Chronological fallback:**
```sql
SELECT * FROM memory_entry
WHERE ...
ORDER BY created_at DESC, rowid DESC
LIMIT ?
```

Use `rowid DESC` as a tiebreaker for same-millisecond timestamps to guarantee stable
ordering under concurrent inserts.

## Why This Is Non-Obvious

The FTS5 content-table pattern and the trigger maintenance requirement are documented in
the SQLite FTS5 spec, but assembling a working Java/JDBC implementation requires combining
several pieces that are never shown together: the virtual table DDL, the three trigger
templates, the JOIN query pattern, the operator-stripping sanitisation, and the `rank`
column semantics.

The `rank` column is built into FTS5 — it returns the BM25 score as a **negative** value.
Most relevance ranking systems use positive scores with DESC ordering; FTS5 uses negative
scores with ASC ordering. Getting this backwards returns the LEAST relevant results first
with no error.

The strip-not-phrase-wrap approach for operator sanitisation is also non-obvious: wrapping
the entire question in quotes changes AND-of-terms to exact-phrase matching, which breaks
multi-concept questions.

## Constraints

FTS5 is bundled in the xerial `sqlite-jdbc` jar — no separate extension needed. The content
table pattern requires `content_rowid='rowid'` on a table with an implicit SQLite rowid
(not a `WITHOUT ROWID` table). The three-trigger maintenance adds a small write overhead
per INSERT/UPDATE/DELETE — acceptable for most workloads, but for very high insert rates
consider async FTS rebuild instead.
