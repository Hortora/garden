## SQLite json_extract interprets dots in key names as JSON path separators — use quoted key syntax

### Symptom

A `json_extract` call with a dotted key returns NULL even though the JSON contains the key:

```sql
-- JSON: {"cbr.caseType": "aml", "name": "test"}
SELECT json_extract('{"cbr.caseType": "aml"}', '$.cbr.caseType');
-- Returns: NULL (not "aml")
```

No error is raised — `json_extract` silently returns NULL because it interprets `$.cbr.caseType` as "root → object `cbr` → key `caseType`", which doesn't exist.

### Context

Arises when storing Java `Map<String, String>` as JSON in a TEXT column and querying by keys that use dot-separated namespaces (e.g. `cbr.caseType`, `cbr.features`, `actor.role`). Common in attribute maps where keys are namespaced to avoid collisions across domains.

### Root cause

SQLite's JSON path syntax follows RFC 6901 conventions where `.` is the path separator. There is no configuration to change this. The key `cbr.caseType` is a single top-level key in the JSON object, but `$.cbr.caseType` navigates two levels deep.

### Fix

Quote the key name in the JSON path using double quotes inside the path string:

```sql
SELECT json_extract('{"cbr.caseType": "aml"}', '$."cbr.caseType"');
-- Returns: "aml"
```

In JDBC with PreparedStatement, concatenate the quoting in SQL (not in Java) so the key is bound as a parameter:

```sql
AND json_extract(attributes, '$.\"' || ? || '\"') = ?
```

Or construct the full path in Java and bind as a single parameter (equally safe — PreparedStatement binding is always injection-safe):

```java
String jsonPath = "$.\"" + attributeKey + "\"";
ps.setString(idx++, jsonPath);
```

### Why this is non-obvious

The SQLite JSON1 extension documentation does not mention quoted key syntax. The only way to discover `$."key.with.dots"` is through experimentation or reading the SQLite source code. PostgreSQL's `->>` operator does not have this problem because it performs top-level key lookup, not path navigation.
