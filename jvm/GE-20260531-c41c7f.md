### Symptom

`jakarta.activation.UnsupportedDataTypeException` (or `UnsupportedDataTypeException` from
`com.sun.activation`) when calling `Part.getInputStream().readAllBytes()` on a `MimeBodyPart`
constructed in-memory via `setContent(byte[], "application/pdf")` or
`setContent(byte[], "image/png")`.

### Context

Writing Jakarta Mail / angus-mail tests that build multipart MIME messages in-memory with
binary attachments. The same code works correctly against real IMAP messages fetched from
a server.

```java
// FAILS in tests — UnsupportedDataTypeException
MimeBodyPart att = new MimeBodyPart();
att.setContent(new byte[]{1, 2, 3}, "application/pdf");
byte[] bytes = att.getInputStream().readAllBytes(); // throws

// WORKS on real IMAP messages — angus-mail pre-decodes before DCH is involved
Message[] msgs = folder.getMessages();
byte[] bytes = msgs[0].getInputStream().readAllBytes(); // fine
```

### Root Cause

`Part.getInputStream()` routes through Jakarta Activation's DataContentHandler (DCH) chain.
In a plain `jakarta.mail` / `angus-mail` environment without extra activation jars,
no DCH is registered for binary MIME types (`application/pdf`, `image/png`,
`text/calendar`, etc.). The DCH chain has no handler and throws.

For real IMAP messages this doesn't matter: angus-mail decodes the transfer-encoding
(base64) directly before the DataHandler sees the stream, so `getInputStream()` works.
But for in-memory parts created via `setContent(byte[], mimeType)`, the DCH is on
the critical path and has no handler for unknown binary types.

### Fix

Call `Part.getContent()` first — it returns the raw Java object the part stores without
invoking the DCH stream path. Dispatch on the returned type:

```java
private static byte[] readPartBytes(Part part) throws Exception {
    Object raw = part.getContent();
    if (raw instanceof byte[] directBytes) {
        return directBytes;                           // in-memory binary setContent()
    } else if (raw instanceof InputStream is) {
        return is.readAllBytes();                     // real IMAP transfer-decoded stream
    } else if (raw instanceof String s) {
        // text/calendar, text/csv etc. stored as String with charset
        String charset = extractCharset(part.getContentType());
        return s.getBytes(Charset.forName(charset));
    } else {
        return part.getInputStream().readAllBytes();  // DCH-registered fallback
    }
}
```

### Why This Is Non-Obvious

`Part.getInputStream()` is the standard documented API for reading raw part bytes — the
JavaDoc makes no mention of DCH dependency or MIME-type limitations. The failure only
appears in tests (in-memory parts), not in production (real IMAP with pre-decoded streams),
making it look like a test-infrastructure problem rather than a Jakarta Activation issue.
