### Technique

When writing a `@QuarkusTest` that needs to spy on a real bean's method — allowing the first invocation to execute normally and the second to throw — the setup dispatch, `clearInvocations`, and stub configuration must happen in this exact order:

```java
// Step 1: Real setup dispatch — NO stub configured yet
//         If stub is configured before this, the setup call hits the stub and throws
messageService.dispatch(setupCommand);

// Step 2: Reset invocation count — BEFORE configuring the stub
//         clearInvocations() resets not only counts but also stub call-ordering.
//         If the stub is configured before clearInvocations(), the post-clear first
//         call hits doThrow() rather than doCallRealMethod() — the wrong invocation throws.
clearInvocations(messageService);

// Step 3: Configure stub AFTER clearInvocations — establishes call ordering from zero
doCallRealMethod()
    .doThrow(new RuntimeException("simulated failure"))
    .when(messageService).dispatch(any());

// Step 4: Trigger the code under test
serviceUnderTest.fulfill(gateId, "approved");

// Step 5: Verify — times(2) counts only the two post-clear dispatches (not the setup call)
verify(messageService, times(2)).dispatch(any());
```

### Why this is non-obvious

`clearInvocations(spy)` is documented as resetting invocation history (the count that `verify(times(N))` reads). It is not documented that it also resets **stub ordering state**. `doCallRealMethod().doThrow(...)` is an ordered chain: first call → real, second call → throw. If you configure this chain and then call `clearInvocations`, the call-order counter resets to zero. The next call after the clear is the "first" call in the chain — `doCallRealMethod()` — as expected. But if you configure the stub *before* `clearInvocations`, `clearInvocations` resets the ordering, and your `doThrow()` becomes what fires on the first post-setup call.

The correct mental model: `clearInvocations` is a full state reset of the spy for that session, not just an invocation counter reset. Configure stubs after clearing.

### Context

Applicable when a `@QuarkusTest` needs:
1. A real setup call to the spy (to populate InMemoryStore or create side effects)  
2. A `verify(times(N))` that counts only production-path calls, not setup calls
3. A mixed real/throw stub to simulate partial failure mid-method
