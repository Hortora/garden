GE-20260718-aeeb7b: [technique, 9/15] Observable @Alternative stubs for @QuarkusTest — Queue for programmable outcomes + synchronized List for call recording replaces Mockito verify in CDI integration tests | quarkus, testing, cdi, alternative

### The pattern

When a `@QuarkusTest` needs to verify that a CDI bean was called with specific arguments (what Mockito `verify()` does), but the bean must be an `@Alternative` (per SPI testing protocols), combine two data structures:

```java
@Alternative
@Priority(1)
@ApplicationScoped
public static class TestMyClient implements MyClient {
    private final Queue<MyOutcome> outcomes = new LinkedList<>();
    private final List<MyCall> calls = Collections.synchronizedList(new ArrayList<>());

    public record MyCall(String arg1, int arg2) {}

    @Override
    public MyOutcome doWork(String arg1, int arg2) {
        calls.add(new MyCall(arg1, arg2));
        MyOutcome outcome = outcomes.poll();
        if (outcome == null) throw new IllegalStateException(
            "TestMyClient: no outcome programmed for " + arg1);
        return outcome;
    }

    public void enqueue(MyOutcome... results) {
        Collections.addAll(outcomes, results);
    }

    public List<MyCall> calls() { return List.copyOf(calls); }

    public void reset() { outcomes.clear(); calls.clear(); }
}
```

### Why the synchronized list

CDI `@ObservesAsync` events are delivered on different threads. If the stub is invoked from an async event handler, the call list must be thread-safe. `Collections.synchronizedList` is sufficient — no need for `CopyOnWriteArrayList` since reads happen after all writes complete (in the assertion phase).

### Why throw on empty queue

An empty queue means the test forgot to program an outcome — a test authoring bug, not a production bug. Throwing immediately surfaces the problem instead of returning null and producing a confusing NPE downstream.

### Usage in tests

```java
@Inject TestMyClient testClient;

@BeforeEach
void setUp() { testClient.reset(); }

@Test
void myTest() {
    testClient.enqueue(new MyOutcome.Success("result"));
    // ... drive the system ...
    assertThat(testClient.calls()).hasSize(1);
    assertThat(testClient.calls().get(0).arg1()).isEqualTo("expected");
}
```

### Why this is non-obvious

Most developers reach for Mockito `verify()` to check call arguments. But `@Alternative` beans in Quarkus `@QuarkusTest` are real CDI beans — Mockito's proxy model conflicts with CDI's proxy model. The `Queue + List` pattern provides the same verification capability within CDI's lifecycle, without Mockito.
