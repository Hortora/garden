## Symptom

After a successful WebSocket handshake (101 received), the first WebSocket frame fails to parse — either the proto parse throws `InvalidProtocolBufferException` ("input ended unexpectedly") or the frame reader hangs waiting for bytes that will never arrive.

## Context

A test server uses `BufferedReader` to parse the HTTP upgrade headers:

```java
var in = new BufferedReader(new InputStreamReader(s.getInputStream()));
String key = null;
String line;
while (!(line = in.readLine()).isEmpty()) {
    if (line.startsWith("Sec-WebSocket-Key:"))
        key = line.substring("Sec-WebSocket-Key:".length()).trim();
}
// 101 sent...
// Now reads WebSocket frames directly from s.getInputStream()
```

## Root Cause

`BufferedReader` maintains an internal 8KB buffer. `readLine()` reads ahead past the `\r\n\r\n` header terminator to fill the buffer. The bytes immediately after the headers — which are the start of the first WebSocket frame — are consumed into the `BufferedReader`'s buffer. When the WebSocket frame reader subsequently reads from `s.getInputStream()` directly, those bytes are missing.

## Fix

Read HTTP headers byte-by-byte until `\r\n\r\n` is detected:

```java
private void doHandshake(Socket s) throws Exception {
    InputStream rawIn = s.getInputStream();
    StringBuilder sb = new StringBuilder();
    int b;
    while ((b = rawIn.read()) >= 0) {
        sb.append((char) b);
        int len = sb.length();
        if (len >= 4
                && sb.charAt(len - 4) == '\r' && sb.charAt(len - 3) == '\n'
                && sb.charAt(len - 2) == '\r' && sb.charAt(len - 1) == '\n') break;
    }
    // Extract key, send 101...
    // rawIn now positioned exactly at first WebSocket frame byte
}
```

The subsequent `serveFrames()` reads from the same `rawIn` reference and gets the frame bytes intact.

## Why Non-Obvious

`BufferedReader` is the standard class for reading line-by-line. Its buffering behaviour is expected and documented — but the consequence for mixed-protocol streams (HTTP headers then binary frames) is not. The failure is intermittent if the frame arrives before the buffer fills, making it look like a timing issue rather than a buffering one.
