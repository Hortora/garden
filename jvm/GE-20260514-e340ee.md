## Context

`CaseContext.evalObjectTemplate(String template)` evaluates a `{ key: .path }` template against **the live case context**. When you need to evaluate the same template against an **external map** (e.g. a WorkItem resolution JSON), the API doesn't provide a direct way to specify the source data.

## Technique

Construct a temporary `CaseContextImpl` from the external map, then call `evalObjectTemplate()` on it:

```java
// Parse the external JSON into a Map
@SuppressWarnings("unchecked")
Map<String, Object> resolutionData = MAPPER.readValue(workItem.resolution, Map.class);

// Create a temporary context backed by the external data
CaseContextImpl resolutionCtx = new CaseContextImpl(resolutionData);

// Evaluate the outputMapping expression against the resolution, not the case context
Map<String, Object> updates = resolutionCtx.evalObjectTemplate(jq.expression());

// Apply the result to the real context
instance.getCaseContext().setAll(updates);
```

This pattern appears in `WorkItemLifecycleAdapter.applyOutputMapping()` for evaluating `HumanTaskTarget.outputMapping()` against the WorkItem resolution before merging into the case context.

## Why This Is Non-Obvious

Most developers would look for a two-argument overload `evalObjectTemplate(template, sourceMap)` or a static `CaseContext.evaluate(template, data)` helper — neither exists. The insight is that `CaseContextImpl` can be used as a disposable evaluation context backed by any map, not just the live case state. The object is lightweight (just wraps a `Map` with read locks) and safe to construct and discard.

**The alternative** (building a scratch map and evaluating path expressions manually) requires re-implementing `evalObjectTemplate`'s mini-DSL parser, which handles nested paths, string literals, and numeric literals. Using a temporary `CaseContextImpl` reuses the existing parser for free.
