### Symptom

@QuarkusTest bootstrap hangs or fails with cascading UnsatisfiedResolutionExceptions when embedding casehub-engine and casehub-desiredstate in an app module that has unsatisfied SPI injection points.

### Context

Building a Quarkus application that embeds casehub-engine and casehub-desiredstate runtime as dependencies. The desiredstate runtime requires five SPI implementations (GoalCompiler, ActualStateAdapter, NodeProvisioner, FaultPolicy, EventSource) and the engine has its own SPIs. Without implementations, CDI resolution fails.

### What was tried (didn't work)

Adding broad `quarkus.arc.exclude-types` for the runtime packages:
```properties
quarkus.arc.exclude-types=\
  io.casehub.engine.internal.**,\
  io.casehub.desiredstate.runtime.**,\
  io.casehub.blackboard.**,\
  io.casehub.workadapter.**,\
  io.casehub.work.runtime.**
```

This creates worse failures — excluded beans leave their own consumers unsatisfied, creating a cascade of UnsatisfiedResolutionExceptions deeper in the CDI graph.

### Root cause

The runtime beans (ReconciliationLoop, TransitionPlanner, etc.) are consumers of the SPIs, not the SPIs themselves. Excluding them removes working code that other beans depend on. The actual problem is the missing SPI implementations, not the runtime beans.

### Fix

Provide `@DefaultBean @ApplicationScoped` stub implementations for each required SPI:

```java
@DefaultBean
@ApplicationScoped
public class StubActualStateAdapter implements ActualStateAdapter {
    @Override
    public ActualState readActual(DesiredStateGraph desired, String tenancyId) {
        return new ActualState(desired.nodes().keySet().stream()
            .collect(Collectors.toMap(id -> id, id -> NodeStatus.ABSENT)));
    }
}
```

Also add all reactive persistence alternatives to `quarkus.arc.selected-alternatives` — the engine injects reactive variants (InMemoryReactiveEventLogRepository, InMemoryReactiveCaseInstanceRepository) that aren't covered by the blocking-only alternatives listed in most examples.

### Why this is non-obvious

The intuition when facing UnsatisfiedResolutionException is to remove the bean that requires the missing dependency. But in CDI, removing a bean can create new unsatisfied dependencies for other beans that injected the removed one. The correct fix is additive (provide stubs) not subtractive (exclude consumers).

### Alternatives considered

- Excluding individual beans one by one — works initially but becomes whack-a-mole as more consumers surface
- Adding all runtime beans' dependencies as explicit stubs — correct approach but requires knowing the full CDI graph
