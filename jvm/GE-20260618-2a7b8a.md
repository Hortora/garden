## Symptom

WireMock tests in modules that don't use Camel start failing with `FatalStartupException` after `quarkus-camel-bom` is added to the root pom's `<dependencyManagement>`. The error occurs at WireMock startup, not in any Camel-related code.

```
com.github.tomakehurst.wiremock.common.FatalStartupException: 
java.lang.NoSuchMethodError: 'void org.eclipse.jetty.server.HttpChannel.recycle()'
```

## Context

A multi-module Maven project where only one module uses Camel. The Camel BOM was added to the root pom's `<dependencyManagement>` as a `<scope>import</scope>` entry to version-manage `camel-quarkus-core` and `camel-quarkus-mock` for the single Camel module.

## Root Cause

`quarkus-camel-bom` manages Jetty artifacts at version 12.x. When imported at the root pom level, it overrides the Jetty version for the entire reactor build — including modules that have nothing to do with Camel. WireMock 3.x (`org.wiremock:wiremock:3.13.0`) was compiled against and requires Jetty 11.x. The version conflict breaks WireMock's HTTP server at startup.

The BOM override is silent — Maven does not warn that a BOM import is overriding a previously-set version.

## Fix

Move the `quarkus-camel-bom` import from the root pom's `<dependencyManagement>` into the local `<dependencyManagement>` of the specific module that needs Camel:

```xml
<!-- In streams-camel/pom.xml (the only module using Camel) -->
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>io.quarkus.platform</groupId>
            <artifactId>quarkus-camel-bom</artifactId>
            <version>${quarkus.platform.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

Module-scoped `<dependencyManagement>` limits the BOM's version overrides to that module and its descendants only.

## Why This Is Non-Obvious

The error appears in WireMock code in unrelated modules, not in any Camel code. The failing module doesn't declare any Camel dependencies — the root pom is the only connection. The `FatalStartupException` stack trace points to Jetty internals, not to any BOM or dependency management configuration. Nothing in the Camel BOM documentation warns about the transitive Jetty version override.
