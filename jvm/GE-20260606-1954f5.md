### Symptom
Tests in `runtime/` compile and pass when run via `mvn test -pl api,runtime` (reactor mode) but fail with unexpected compilation errors or wrong-type errors when run as `mvn test -pl runtime` alone after changing `api/`.

### Context
Multi-module Maven project where `runtime` depends on `api`. You changed a class in `api/` (e.g. added a field to a record), compiled the whole reactor successfully, and then ran tests against only `runtime/`. Tests fail because the compiler sees the old `api` jar from `~/.m2/repository`.

### Root cause
`mvn test -pl runtime` resolves peer-module dependencies from the local Maven repository (`~/.m2/repository`), not from the in-memory reactor build graph. Only when modules are specified together in the `-pl` list does Maven use reactor output. Running a single module in isolation always uses whatever version is installed.

### Fix
Install the changed peer module first:
```bash
mvn install -pl api  # installs api-<version>-SNAPSHOT.jar to ~/.m2
mvn test -pl runtime  # now picks up the new api jar
```

Or run in reactor mode so Maven never consults the local repo for peer modules:
```bash
mvn test -pl api,runtime
```

### Why this is non-obvious
`mvn compile -pl api,runtime` succeeds because compilation is reactor-aware. The success creates the false impression that `api` changes are visible everywhere. Switching to `mvn test -pl runtime` silently drops to the installed jar, reproducing a stale classpath that only shows up as compilation errors in test code that uses the new API.

### Constraints
- Applies to SNAPSHOT dependencies between sibling modules in the same project.
- Release versions always come from the repo — this only affects SNAPSHOT resolution.
