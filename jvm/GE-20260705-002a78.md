### Symptom

Tenant-B receives notifications for events that only tenant-A should see. No error, no warning — the events silently fan out to the wrong subscriber.

### Context

A multi-tenant notification subscription system uses the DataSource alpha network (`AlphaDataSource` → `TypeNode` → `FilterNode` → `FanOutProcessor`) to evaluate subscriptions. Each subscription is wired as `dataSource.subscribe(objectType, filterExpression, processor)`. Subscriptions from different tenants can have identical user constraints (e.g., both filter on `priority == HIGH`).

### What was tried (didn't work)

Adding a tenant isolation check inside the `DataProcessor` callback (the fan-out target). This prevents acting on wrong-tenant events but does not prevent the shared `FilterNode` from evaluating a single predicate for all subscribers — when the predicate passes for tenant-A's event, tenant-B's processor still fires.

### Root cause

`TypeNode.filtersMatch()` (in `AlphaDataSource`) compares `FilterExpression.type()` + `FilterExpression.expression()` to decide whether two subscriptions can share a `FilterNode`. Two subscriptions from different tenants with identical user constraints produce identical `type()` and `expression()` values — so they share a single `FilterNode`.

`FilterNode` evaluates ONE predicate and calls `FanOutProcessor.add()` for ALL subscribers under it. When tenant-A's event passes the shared predicate, tenant-B's `DataProcessor` receives it too. The reverse also happens: when tenant-A's event fails the predicate (e.g., `priority != HIGH`), tenant-B's legitimate match is also suppressed — silent data loss.

### Fix

Embed the tenant ID in the `FilterExpression.expression()` string so `filtersMatch()` returns `false` across tenants:

```java
String expression = "tenant=" + tenancyId + ":" + mvelExpression;
FilterExpression<Object> filter = new FilterExpression<>("mvel", expression, compiledPredicate);
```

The compiled predicate includes a native Java tenancy check via `MethodHandle` as defense-in-depth — the expression string prevents node sharing, the predicate prevents evaluation leaks:

```java
Predicate<Object> tenantCheck = pojo -> {
    String pojoTenant = extractTenancyId(pojo); // MethodHandle on tenancyId()
    return tenancyId.equals(pojoTenant);
};
Predicate<Object> combined = tenantCheck.and(userConstraintPredicate);
```

This gives correct behaviour: subscriptions with the same tenant ID and same user constraints share a `FilterNode` (the optimization works within a tenant). Cross-tenant subscriptions always get separate `FilterNode` instances.

### Why this is non-obvious

The `FilterNode` sharing is an internal optimization of the alpha network — invisible to the subscription author. The API contract of `DataSource.subscribe(objectType, filterExpression, processor)` does not document that two subscribers with identical `FilterExpression` metadata will share evaluation. A developer building a multi-tenant subscription system would naturally add tenant isolation to the predicate (which is correct but insufficient) without realising that the expression string controls node sharing.

### Why this fix

The tenant ID in the expression string is the structural fix — it prevents sharing at the alpha network level. The native Java tenancy check in the predicate is defense-in-depth — it prevents leaks even if the expression string is somehow bypassed or the sharing logic changes. Both are necessary: the expression string alone would fail if `filtersMatch()` logic changed; the predicate alone would fail because the shared `FilterNode` still fans out to wrong-tenant processors.

### Alternatives considered

- **Tenant-aware `TypeNode.filtersMatch()`** — rejected because it would require changing the alpha network's core API to thread tenant context through type discrimination. The expression string approach is non-invasive and works with the existing API.
- **Per-tenant `DataSource` instances** — rejected because it fragments the alpha network. A single DataSource with tenant-prefixed expressions gives the same isolation while preserving intra-tenant sharing.
