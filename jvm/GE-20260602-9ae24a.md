### Symptom
`SomeServiceTest.some_ledger_assertion` fails intermittently with "Expected size: 1 but was: 2" â€” finds both the expected entry and an unexpected entry from a different `LedgerEntry` subclass, both with the same `subjectId`.

### Context
A `@ObservesAsync` CDI listener had a new `@Transactional(REQUIRES_NEW)` write added to it (e.g. an audit skip entry). The test under failure calls a service that fires the async event, then immediately queries `findBySubjectId(entityId)` expecting only one entry type.

### What was tried
- Assumed the test was broken in isolation â€” re-ran it and it passed (intermittent)
- Checked for cross-test correlationId reuse â€” unrelated

### Root cause
`@ObservesAsync` delivers events to the listener asynchronously on a separate thread. When a listener adds a `REQUIRES_NEW` nested transaction, that transaction commits independently â€” and immediately â€” even if no outer transaction holds it. Because the event carries the same `subjectId` as the entity the test asserts on, `findBySubjectId` returns both the expected entry and the listener's newly committed entry. The flake arises because `@ObservesAsync` may fire before or after the assertion in the test thread.

### Fix
In tests asserting ledger entry counts for a specific entity, filter by subclass type rather than asserting total count:

```java
// Before â€” breaks when any @ObservesAsync listener writes to the same subjectId
var entries = ledgerRepo.findBySubjectId(ae.id);
assertThat(entries).hasSize(1);

// After â€” asserts only on the type this test cares about
var entries = ledgerRepo.findBySubjectId(ae.id).stream()
    .filter(e -> e instanceof AdverseEventLedgerEntry)
    .toList();
assertThat(entries).hasSize(1);
AdverseEventLedgerEntry entry = (AdverseEventLedgerEntry) entries.get(0);
```

### Why this is non-obvious
Adding a `REQUIRES_NEW` write to one listener breaks an assertion in a completely different test class. The contamination is intermittent because async delivery timing determines whether the listener fires before or after the assertion â€” so the test appears flaky rather than consistently broken. The listener that was changed and the test that breaks are not obviously related.

**Relation to GE-20260522-f63c9f:** That entry covers REQUIRES_NEW contamination within `@TestTransaction` rollback. This entry covers a different mechanism: async delivery racing the test assertion thread, with no rollback involved at all.
