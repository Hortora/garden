### Symptom
Expression evaluation silently fails when the expression context Map contains
`OffsetDateTime` or `Instant` values. The JQ adapter's `ObjectMapper.valueToTree(map)`
throws `InvalidDefinitionException`, caught by the error handler, causing fallback to
default behavior. No visible error unless logging is configured.

### Context
Building a `Map<String, Object>` expression context from CloudEvent fields. The context
includes `event.getTime()` (returns `OffsetDateTime`) and `context.firstSignal()` /
`context.lastSignal()` (returns `Instant`). The Map is passed to a JQ adapter that
converts it to `JsonNode` via `ObjectMapper.valueToTree()`.

### What was tried (didn't work)
Putting `OffsetDateTime` and `Instant` values directly in the context Map — Jackson's
default `ObjectMapper` has no serializer for `java.time` types without `JavaTimeModule`.

### Root cause
Jackson's default `ObjectMapper` does not know how to serialize `java.time.OffsetDateTime`
or `java.time.Instant`. The `jackson-datatype-jsr310` module must be registered explicitly
via `mapper.registerModule(new JavaTimeModule())`. Without it, `valueToTree()` throws
`InvalidDefinitionException`.

### Fix
Convert temporal types to ISO-8601 strings in the context builder before they enter the
expression adapter:

```java
ctx.put("time", event.getTime() != null ? event.getTime().toString() : null);
ctx.put("firstSignal", context.firstSignal() != null ? context.firstSignal().toString() : null);
```

This avoids depending on `JavaTimeModule` and produces string values that JQ and MVEL
can process naturally (string comparison, parsing).

### Why this is non-obvious
The failure is silent in the hot path — the expression error handler catches the
serialization exception and degrades to default behavior (default correlation key,
filter pass-through). Tests without timestamps pass; tests with timestamps silently
fall back. The symptom ("wrong correlation key used") doesn't point at Jackson
serialization.
