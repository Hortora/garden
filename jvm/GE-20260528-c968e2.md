## The technique

casehub-work scheduler jobs (`ExpiryCleanupJob`, `ClaimDeadlineJob`) should be excluded from CDI in tests to prevent background interference. But `ExpiryLifecycleService` (the service they delegate to) can be kept in CDI and injected directly — allowing precise SLA breach testing without timing dependencies.

```properties
# In test application.properties:
# Keep ExpiryLifecycleService in CDI — injected and called directly in SLA tests.
# Exclude only the scheduler jobs that trigger it on a timer.
quarkus.arc.exclude-types=\
  io.casehub.work.runtime.service.ExpiryCleanupJob,\
  io.casehub.work.runtime.service.ClaimDeadlineJob,\
  io.casehub.work.runtime.strategy.RoutingCursorCleanupJob
  # NOT: io.casehub.work.runtime.service.ExpiryLifecycleService
```

```java
@QuarkusTest
class SlaTest {
    @Inject
    ExpiryLifecycleService expiryLifecycleService;

    @Test
    void slaBreachFires_onCheckExpired() {
        // Create WorkItem with past deadline
        given()
            .body("""{"templateRef":"household-task","title":"Past","deadline":"%s"}"""
                .formatted(Instant.now().minus(1, ChronoUnit.HOURS)))
            .when().post("/life-tasks")
            .then().statusCode(201);

        // Call the service directly — no scheduler required
        expiryLifecycleService.checkExpired();

        // Verify breach effects (WorkItem status changed, policy fired, etc.)
    }
}
```

## Why developers do it differently

The natural approach is to either: (1) exclude `ExpiryLifecycleService` from CDI entirely (no SLA breach tests possible), or (2) enable the scheduler jobs in tests with `Awaitility` timeouts (slow, flaky, depends on timing). The distinction between the scheduler jobs and the service they call enables precise, deterministic testing.

## Why this approach is better

- Deterministic — no timing or polling
- Fast — breach fires synchronously in the test thread
- Isolated — scheduler jobs don't fire in the background and contaminate test state
- Verifiable — you control exactly when the breach check runs and can assert immediately after
