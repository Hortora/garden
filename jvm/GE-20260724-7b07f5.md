## Replace event bus request-reply with direct injection for single-consumer blocking handlers

**ID:** GE-20260724-7b07f5
**Stack:** Quarkus 3.32.x, Vert.x Mutiny, CDI
**Labels:** `#quarkus` `#virtual-threads` `#architecture`
**What it achieves:** Eliminates the Vert.x event bus request-reply overhead for synchronous handler invocation, avoiding the @RunOnVirtualThread reply mechanism gap (see GE-20260724-dc9748) and reducing architectural ceremony.
**Context:** When migrating a Quarkus application from reactive to blocking/virtual threads. The event bus was used in the reactive codebase for request-reply to sequence asynchronous operations. In a blocking world, direct CDI injection gives the same ordering guarantee with less overhead.

### The technique

Identify event bus addresses that are single-producer/single-consumer pairs:

```bash
# Search for address constants
ide_search_text "CASE_STARTED" *.java
# Check: exactly one @ConsumeEvent and one send/request/publish per address
```

For each 1:1 pair, replace the event bus indirection with direct injection:

```java
// Before — event bus ceremony for a 1:1 call
@ApplicationScoped
class Reactor {
    @Inject EventBus eventBus;
    
    void startCase(CaseStartedEvent event) {
        eventBus.requestAndAwait("casehub.case.started", event);
    }
}

@ApplicationScoped
class Handler {
    @ConsumeEvent("casehub.case.started")
    @RunOnVirtualThread
    void onStarted(CaseStartedEvent event) { /* ... */ }
}

// After — direct injection
@ApplicationScoped
class Reactor {
    @Inject Handler handler;
    
    void startCase(CaseStartedEvent event) {
        handler.onStarted(event);
    }
}

@ApplicationScoped
class Handler {
    public void onStarted(CaseStartedEvent event) { /* ... */ }
}
```

Keep `eventBus.publish()` (fire-and-forget) for addresses with multiple consumers or where you genuinely want decoupled notification.

### Alternatives considered
- Keeping `requestAndAwait` and fixing the handler reply — `@RunOnVirtualThread` breaks the auto-reply and no annotation combination fixes it (see GE-20260724-dc9748)
- Using `eventBus.send()` + `CountDownLatch` — maintains decoupling but adds synchronization complexity for no benefit in a 1:1 scenario

### Why this is non-obvious
In a reactive architecture, the event bus is the natural coordination mechanism — `request().chain()` sequences operations without blocking the event loop. Developers migrating to blocking/virtual threads naturally keep the event bus patterns. The insight is that blocking code doesn't need the event bus for sequencing — a direct method call IS the sequencing. The event bus adds value only for fan-out (publish to multiple consumers) or true decoupling (consumer may not exist). For 1:1 synchronous calls, it's pure ceremony.

### When to use it
- Migrating from reactive to blocking/virtual threads in Quarkus
- Event bus address has exactly one `@ConsumeEvent` handler
- Caller needs synchronous completion (would use `requestAndAwait`)
- Do NOT use when multiple consumers subscribe to the same address
- Do NOT use when the consumer may be absent (optional observer pattern)
