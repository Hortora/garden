### Symptom

After adding a custom `HttpAuthenticationMechanism` with `getCredentialTransport()` returning `HttpCredentialTransport(Type.AUTHORIZATION, "bearer", "plugin-token")`, all `@TestSecurity` tests return 500 (OIDC server connection refused) — including tests for endpoints not handled by the custom mechanism.

### Context

Application has `quarkus-oidc` on the classpath (for JWT validation on some endpoints) and a custom mechanism for static bearer token validation on other endpoints. Both declare `Type.AUTHORIZATION` with `"bearer"` as the transport type.

### What was tried (didn't work)

1. Using the 3-arg constructor `HttpCredentialTransport(Type.AUTHORIZATION, "bearer", "plugin-token")` — `typeTarget="bearer"` matches OIDC's transport type, causing ambiguity
2. Combining `auth-mechanism=plugin-token` config with the credential transport — breaks @TestSecurity (see GE-20260628-04a38c)

### Root cause

Quarkus uses `getCredentialTransport()` for mechanism deconfliction. When two mechanisms declare the same `typeTarget` (`"bearer"`), Quarkus cannot distinguish between them for challenge sending and mechanism selection. The OIDC mechanism's challenge flow then fires for requests it shouldn't handle, triggering OIDC discovery against an absent server.

The Javadoc explicitly warns: "Note that using multiple HTTP authentication mechanisms to use the same credential transport type can lead to unexpected authentication failures."

### Fix

Return `Uni.createFrom().nullItem()` from `getCredentialTransport()`:

```java
@Override
public Uni<HttpCredentialTransport> getCredentialTransport(RoutingContext context) {
    return Uni.createFrom().nullItem();
}
```

Per Javadoc: null means "this mechanism cannot interfere with other mechanisms." The mechanism still works — Quarkus calls `authenticate()` via proactive auth. The path guard in `authenticate()` provides isolation.

### Why this is non-obvious

The 3-arg constructor's Javadoc suggests `authenticationScheme` can be different from `typeTarget` for routing purposes. The expectation is that a distinct scheme ("plugin-token" vs "bearer") would prevent transport-type conflicts. In practice, the `typeTarget` comparison is what causes ambiguity, not the scheme.
