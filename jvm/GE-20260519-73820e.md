## Symptom

Plain-Java unit test (no Quarkus CDI context) uses an anonymous `InstanceHandle<T>` implementation — only overrides `get()`. When the dispatcher calls `handle.close()`, the test throws:

```
java.lang.IllegalStateException: ArC container not initialized:
the container is not started, already shut down, or a wrong class
loader was used to load the io.quarkus.arc.Arc class
```

## Context

`io.quarkus.arc.InstanceHandle<T>` has exactly **one abstract method**: `T get()`. All other methods have defaults — including `close()`, which looks like it should be a no-op cleanup. The interface extends `AutoCloseable`, so `close()` appears to simply satisfy that contract.

Triggered any time you create an anonymous `InstanceHandle<T>` in a plain-Java test (JUnit 5, no `@QuarkusTest`) and the code under test calls `handle.close()`.

## Root Cause

`InstanceHandle.close()` default implementation calls `Arc.requireContainer().destroy(…)` to properly destroy `@Dependent`-scoped beans. This requires a live Arc container. In plain-Java tests without an Arc container, the call throws immediately.

```java
// What Arc's default close() does (simplified):
default void close() {
    Arc.requireContainer().destroy(this);  // ← blows up without a container
}
```

The interface declares only `get()` as abstract, so the developer reasonably assumes implementing `get()` is sufficient. Nothing warns that `close()` is not safe to inherit.

## Fix

Always override `close()` explicitly in test anonymous classes, even when you believe the default is innocuous:

```java
private static <T> InstanceHandle<T> handle(final T instance) {
    return new InstanceHandle<>() {
        @Override public T get() { return instance; }
        @Override public void close() { /* no-op: no Arc container in plain-Java tests */ }
    };
}
```

The comment is important — it documents why `close()` is a no-op (not an omission but a deliberate choice for the non-CDI test context).

## Why This Is Non-Obvious

`InstanceHandle` has exactly one abstract method — `get()`. Java developers familiar with functional interfaces and CDI patterns assume that implementing `get()` is sufficient. The `close()` method on `AutoCloseable` routinely does nothing when no resource needs releasing. Nothing in the interface signature, Javadoc, or compiler errors indicates that the default `close()` has a hard runtime dependency on a live Arc container.

The error message ("ArC container not initialized") is real but does not mention `InstanceHandle.close()` — the stack trace points into Arc internals, making the connection non-obvious.
