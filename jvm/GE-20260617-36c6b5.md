### Context
A JAX-RS handler method contains a synchronous polling loop (`Thread.sleep()` + repeated JPA reads) that runs for an extended period (seconds to minutes). The handler is in an application using `quarkus-rest` (RESTEasy Reactive), not `quarkus-resteasy` (Classic).

### The non-obvious part
In `quarkus-rest`, JAX-RS handler methods run on the **Vert.x event loop thread** by default. A blocking call (`Thread.sleep`, synchronous JPA, synchronous HTTP) on the event loop blocks **all** other incoming HTTP requests — they queue behind it. A polling handler that runs for 3–5 minutes effectively takes the server offline for all other requests during that time.

In our case: `POST /example/.../start` (which contains the polling loop) was called, and then `POST /openclaw/delivery/oversight/{gateId}` (the approval endpoint that unblocks the polling loop) could never be processed — the event loop was occupied by the first request, deadlocking the demo.

### Fix
Annotate the handler method with `@Blocking`:
```java
import io.smallrye.common.annotation.Blocking;

@POST
@Path("/{exampleId}/start")
@Blocking   // moves execution to Quarkus worker thread pool
public Response start(@PathParam("exampleId") String exampleId) {
    // Thread.sleep() calls and JPA reads are safe here
}
```
`@Blocking` offloads the handler to the Quarkus worker thread pool, leaving the event loop free for all concurrent requests.

### Why this is non-obvious
`quarkus-rest` and `quarkus-resteasy` have different default execution models. In the classic stack, JAX-RS methods block by default. In the reactive stack, they do not. If you use a polling loop that worked fine with `quarkus-resteasy`, it silently breaks under `quarkus-rest`. The failure mode is not an error — it is a deadlock that only surfaces when a second concurrent request is needed to unblock the first.

Note: an existing garden entry (GE-20260428-a67806) covers `@Blocking` for Vert.x event-bus JPA failures — the issue there is JPA from an IO thread. This entry is specifically about **JAX-RS handlers with polling loops** creating a deadlock by starving concurrent HTTP requests.
