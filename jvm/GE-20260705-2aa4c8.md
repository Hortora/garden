### Symptom
Entity fields of type `java.time.Instant` (mapped to `TIMESTAMP` columns) return `null` when fetched via Hibernate Reactive Panache over the H2 reactive emulated driver, even though the persisted value was non-null. Tests fail with `NullPointerException` on accessing `createdAt`, `readAt`, or other temporal fields.

### Context
When testing Hibernate Reactive Panache modules against H2 (for speed), the `quarkiverse/quarkus-reactive-h2-client:0.6.0` artifact provides a Vert.x JDBC bridge that emulates a reactive connection over H2's JDBC driver. This works for `String`, `int`, `boolean`, and `@Enumerated` fields ŌĆö but silently returns `null` for `Instant`/`TIMESTAMP` columns.

The claudony/app module uses this artifact successfully because its entities don't have `Instant` fields in the reactive-tested paths.

### What was tried (didn't work)
1. Configured both JDBC and reactive H2 URLs per the spec:
   ```properties
   quarkus.datasource.jdbc.url=jdbc:h2:mem:test;MODE=PostgreSQL;DB_CLOSE_DELAY=-1
   quarkus.datasource.reactive.url=h2:mem:test
   ```
2. Verified the Flyway migration ran correctly (schema was created)
3. Verified entities persisted without error ŌĆö only fetch returned null temporals

### Root cause
The Vert.x JDBC bridge in `quarkus-reactive-h2-client` does not correctly map H2's `TIMESTAMP` result-set values back to `java.time.Instant` when the result passes through Hibernate Reactive Panache's entity hydration. The JDBC ResultSet returns the value correctly, but the reactive bridge's type conversion layer drops it.

### Fix
Use PostgreSQL DevServices (Testcontainers) for Hibernate Reactive Panache module tests instead of H2 reactive emulation. This is the established pattern in `casehub-platform` (acl-jpa, notifications-jpa both use it).

```properties
# src/test/resources/application.properties
quarkus.datasource.db-kind=postgresql
quarkus.datasource.devservices.enabled=true
quarkus.flyway.migrate-at-start=true
quarkus.flyway.clean-at-start=true
quarkus.hibernate-orm.schema-management.strategy=none
```

Trade-off: 5-10s container startup per test run vs sub-second H2 startup. Accuracy wins ŌĆö tests run against real PostgreSQL.

### Why this is non-obvious
The artifact exists, is published on Quarkiverse, and works for simple entity types. Another module in the same workspace (claudony/app) uses it successfully. The failure is type-specific (only temporal fields) and silent (no error on persist, null on fetch) ŌĆö the obvious debugging path is "my entity mapping is wrong," not "the reactive driver drops temporal values."

### Alternatives considered
- H2 in JDBC-only mode with `quarkus.datasource.reactive=false` ŌĆö works for blocking SPI tests but defeats the purpose of testing the reactive SPI natively
- Separate test profiles (H2 for blocking, DevServices for reactive) ŌĆö unnecessary complexity when DevServices works for both
