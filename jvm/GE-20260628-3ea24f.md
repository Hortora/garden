### Symptom
Reactive service intermittently returns errors instead of the expected result (e.g. `VerificationResult.SUSPECT`) when an `@ObservesAsync` observer throws.

### Context
A reactive Mutiny pipeline uses `Uni.createFrom().completionStage(() -> event.fireAsync(...)).replaceWith(result)` to fire a CDI event and return a result. The intent is fire-and-forget, but the implementation awaits the `CompletionStage` returned by `fireAsync()`.

### Root cause
`fireAsync()` returns a `CompletionStage` that completes when all `@ObservesAsync` observers have finished. `Uni.createFrom().completionStage()` subscribes to that stage — if any async observer throws, the stage fails, the Uni fails, and the pipeline error replaces the intended result.

The coupling is invisible during normal operation because observers rarely throw. It manifests as an intermittent, hard-to-reproduce failure.

### Fix
Use `.invoke()` for fire-and-forget dispatch. The event is a side effect — the pipeline result must not depend on observer delivery:

```java
// ❌ BAD: awaits observer delivery — pipeline fails if observer throws
return Uni.createFrom().completionStage(
        () -> event.fireAsync(new SuspectEvent(...)))
        .replaceWith(VerificationResult.SUSPECT);

// ✅ GOOD: fire-and-forget — result is independent of observer delivery
return Uni.createFrom().item(VerificationResult.SUSPECT)
        .invoke(() -> {
            event.fire(payload);
            event.fireAsync(payload)
                    .exceptionally(ex -> { log.debugf(ex, "async observer failed"); return null; });
        });
```

### Why this is non-obvious
`Uni.createFrom().completionStage()` is the standard Mutiny pattern for bridging `CompletionStage`. It looks correct and works in all tests (test observers don't throw). The failure only surfaces under specific runtime conditions when a real observer fails — making it extremely hard to diagnose.

See also: GE-20260529-e43076 (the *technique* for when awaiting fireAsync IS correct — e.g. `@ConsumeEvent` chains where delivery IS the pipeline's purpose).

### Alternatives considered
- Keeping the await-style pattern with `.onFailure().recoverWithItem()` — rejected: hides the design flaw rather than fixing it. The result should structurally not depend on the event.
