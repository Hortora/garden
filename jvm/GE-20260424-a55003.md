### Symptom
After adding a new field to a Java record used as a service request object, the build fails across multiple modules with `constructor WorkItemCreateRequest in record ... cannot be applied to given types; reason: actual and formal argument lists differ in length`. The failures are spread across many files and modules — often more than expected.

### Context
`WorkItemCreateRequest` was a 16-field record used as the primary input to `WorkItemService.create()`. Adding `callerRef` as field 17 broke 16+ constructor call sites across `work-flow`, `queues-examples`, and `queues-dashboard` — modules that were not touched during the feature work.

### Root cause
Java records generate a canonical constructor that takes every field positionally. Unlike a builder pattern or named-parameter approach, every existing call site must be updated when a field is added, regardless of whether the new field is relevant to that caller.

### Fix
Add `null` (or the appropriate default) as the final argument at every call site:

```bash
# Find all call sites fast
grep -rn "new WorkItemCreateRequest(" --include="*.java" src/
```

Then update each one. For `null`-terminated patterns across many files, sed helps:
```bash
# If the old call ended with: null, null, null));
# Replace with:              null, null, null, null));
find . -name "*.java" | xargs sed -i '' \
  's/null, null, null, null, null));/null, null, null, null, null, null));/g'
```

Be careful: the trailing null pattern varies per call site. Verify each file after.

### Why this is non-obvious
The failures appear in modules that weren't involved in the feature work, making the root cause seem environmental. The error message (`actual and formal argument lists differ in length`) is clear once you see it, but the blast radius across unrelated modules is surprising. Projects that use records as DTOs with many fields across many modules should prefer a builder pattern or named-parameter wrapper to avoid this.
