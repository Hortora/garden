### Context

When building a Quarkus extension or library that optionally integrates with another library (e.g. OpenTelemetry), you want to use the library's API when it's on the classpath but gracefully do nothing when it's absent. The dependency is declared `<optional>true</optional>` in the POM.

### What developers normally do

Direct `@Inject` on the optional type:

```java
@Inject io.opentelemetry.api.trace.Tracer tracer;
```

This throws `UnsatisfiedResolutionException` at CDI bootstrap when no OTel SDK is on the consumer's classpath — the bean type isn't available, and there's no `@DefaultBean` producer to satisfy it.

The next attempt is usually a `@DefaultBean` no-op producer in the extension:

```java
@DefaultBean @ApplicationScoped
public class NoOpTracerProducer {
    @Produces Tracer tracer() { return Tracer.noop(); }
}
```

This works but forces the extension to depend on the API jar at compile time (not truly optional) and adds a bean that exists even when the real implementation is present.

### Fix

Use CDI's `Instance<T>` wrapper with `isResolvable()`:

```java
@Inject
jakarta.enterprise.inject.Instance<io.opentelemetry.api.trace.Tracer> tracerInstance;

void doWork() {
    if (tracerInstance.isResolvable()) {
        Tracer tracer = tracerInstance.get();
        Span span = tracer.spanBuilder("operation").startSpan();
        // ...
    }
}
```

`Instance<T>` never throws at bootstrap — it defers resolution to the `get()` call. `isResolvable()` returns `false` when no bean satisfies the type, without throwing. The extension compiles against the API jar (optional), and consumers who don't bring the API jar simply get `isResolvable() == false`.

### Why this is non-obvious

1. `@Inject Instance<T>` is a standard CDI feature but rarely shown in Quarkus documentation for optional library integration — most examples use `@DefaultBean` or build-time conditions
2. The `<optional>true</optional>` Maven scope means the class literally doesn't exist on some consumers' classpaths — `@Inject` fails at a level below CDI (class loading), while `Instance<T>` defers past that point
3. Unlike `@IfBuildProperty` (build-time gating), this is a runtime check that works without augmentation changes — simpler for library authors who don't control the consumer's build
