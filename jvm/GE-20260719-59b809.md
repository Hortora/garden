### Symptom
Per-document feedback distributions are incomplete or empty when computing
retrieval analysis over a time window. Documents that received feedback
show zero feedback entries in their `feedbackDistribution`.

### Context
`RetrievalTracker` provides two time-windowed query methods:
- `findRecords(corpus, since, until)` — filters on `retrieval_records.timestamp`
- `findFeedback(corpus, since, until)` — filters on `retrieval_feedback.timestamp`

Both accept `(CorpusRef corpus, Instant since, Instant until)`. The natural
assumption is that both filter on the retrieval event's timestamp. They don't.

### Root cause
`findFeedback` filters on the *feedback submission timestamp*, not the
retrieval's timestamp. This is by design — the SPI spec (§SPI Interfaces)
explicitly states: "`findFeedback()` filters on `retrieval_feedback.timestamp`
(when feedback was submitted, not when the retrieval occurred)."

Feedback is submitted asynchronously, often much later than the retrieval.
Calling `findFeedback(corpus, since, until)` with the same `until` as
`findRecords` silently excludes feedback submitted after `until` for
retrievals that occurred before `until`.

### What was tried (didn't work)
Calling `findFeedback(corpus, since, until)` with the same window as
`findRecords` and joining by `(retrievalId, sourceDocumentId)`. This
produces incomplete feedback distributions because late feedback is
excluded by the timestamp filter.

### Fix
Widen `findFeedback` upper bound to `Instant.MAX` and post-filter by
`retrievalId` against in-window retrieval records:

```java
List<RetrievalRecord> records = tracker.findRecords(corpus, since, until);
Set<String> inWindowIds = records.stream()
        .map(RetrievalRecord::retrievalId)
        .collect(Collectors.toSet());

// Widen upper bound — capture ALL feedback submitted after 'since'
List<RetrievalFeedback> allFeedback = tracker.findFeedback(corpus, since, Instant.MAX);

// Post-filter: only feedback for retrievals in-window
for (RetrievalFeedback fb : allFeedback) {
    if (inWindowIds.contains(fb.retrievalId())) {
        // This feedback belongs to an in-window retrieval
    }
}
```

### Why this is non-obvious
The method signatures are symmetric — both take `(CorpusRef, Instant, Instant)`.
Nothing in the type system or method name hints that `since`/`until` refer to
different timestamps in the two methods. The asymmetry is only documented in the
SPI spec prose, not enforced by the API.

### Alternatives considered
- Adding a `findFeedbackByRetrievalWindow()` method to the SPI — rejected as
  premature; the post-filter approach works and avoids SPI expansion.
- Using `Instant.MIN` to `Instant.MAX` for feedback — wasteful; using `since`
  as the lower bound is safe because feedback is always submitted after retrieval.
