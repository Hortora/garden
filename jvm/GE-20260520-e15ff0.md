### Context
`@QuarkusTest` starts one Quarkus instance per test class (unless classes share the same profile). When you need different `application.properties` values for one test class (e.g. a different list of config files), you can't just set system properties — they're evaluated too late for `@PostConstruct` beans.

### Technique
Implement `QuarkusTestProfile` and annotate the test class with `@TestProfile`:

```java
@QuarkusTest
@TestProfile(ChainingTest.TwoFileProfile.class)
class ChainingTest {

    public static class TwoFileProfile implements QuarkusTestProfile {
        @Override
        public Map<String, String> getConfigOverrides() {
            return Map.of(
                "casehub.platform.config.files",
                "classpath:prefs-a.yaml,classpath:prefs-b.yaml"
            );
        }
    }

    @Inject PreferenceProvider provider;

    @Test
    void later_file_wins() { ... }
}
```

Quarkus restarts the application for a new profile — the overrides are in place before any `@PostConstruct` or `@Startup` beans run.

### Why this is non-obvious
Most developers try:
1. `System.setProperty()` in `@BeforeAll` — too late; Quarkus already initialised
2. `@ConfigProperty defaultValue` — hardcodes it, can't vary per class
3. Separate `application.properties` files — Quarkus doesn't swap config files per test class without a profile

`getConfigOverrides()` is the correct solution but requires knowing `QuarkusTestProfile` exists and that a new profile causes a Quarkus restart.
