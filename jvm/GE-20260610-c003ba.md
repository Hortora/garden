### Symptom

```
[ERROR] TestEngine with ID 'junit-jupiter' encountered a critical issue during test discovery:
(1) [ERROR] ClassSelector [className = 'io.example.MyQuarkusTest', classLoader = null] resolution failed
    Cause: org.junit.platform.commons.PreconditionViolationException: Could not load class with name: io.example.MyQuarkusTest
      at io.example.MyQuarkusTest.<no-method>(SourceFile:0)
```

The error appears in the **second** Surefire fork (`jvmRun1` under the first failed run), not the first. Compilation succeeded. The class exists and compiles cleanly.

### Context

When `rerunFailingTestsCount=N` is set in Surefire:

1. **First JVM fork**: Quarkus augmentation runs. If CDI boot fails (e.g. `UnsatisfiedResolutionException`, `DeploymentException`), the Quarkus class loader throws during `@QuarkusTest` class loading. JUnit Platform marks the test as failed.
2. **Surefire captures the class name** from the first failure and stores it for the retry.
3. **Second JVM fork (retry)**: Surefire uses `ClassSelector` with the system class loader (not the Quarkus-augmented class loader) to select the test by name. The system class loader can't find the Quarkus augmented class → "Could not load class with name."

### Root cause

The "ClassSelector null classLoader" error is from the **retry mechanism** failing to locate a test that failed in the **first run** due to a CDI boot error. The real error is completely hidden by the retry error.

### Why this is non-obvious

1. The error says "Could not load class" — looks like a classpath issue, not a CDI problem.
2. The stack trace shows `<no-method>(SourceFile:0)` — looks like a compile artifact problem.
3. The retry run produces the error; the first run's actual error is buried earlier in the log.
4. Nothing points to the `.dump` file in `target/surefire-reports/` as the source of truth.

### Fix

**Find the actual error:** look at the surefire dump file from the FIRST run:

```bash
ls -t target/surefire-reports/*.dump | head -2
cat target/surefire-reports/<timestamp>-jvmRun1.dump | head -100
```

The dump contains the full `Caused by:` chain with the real CDI `DeploymentException`.

**Common root causes surfaced this way:**
- `UnsatisfiedResolutionException` — a required CDI bean missing from test classpath
- `DeploymentException` — field shadowing validation, incompatible bean types
- `LedgerProcessor validation failure` — JPA entity validation errors from Quarkus extensions

**Fix the CDI error** (not the ClassSelector error). The ClassSelector error disappears once the first-run boot succeeds.

### Why this fix

The retry's ClassSelector error is a symptom of the first run's failure — fixing the symptom (e.g. adding the class to the classpath) would not help. The dump is the only way to see the actual `Caused by:` chain because the main Surefire output only shows the second run's error.
