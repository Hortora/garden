### Symptom

`NoSuchFieldError` at runtime when an `@Entity` subclass tries to access a field
declared `@Transient` (or Java `transient`) on its `@MappedSuperclass` parent.
The field exists in the source code and compiles fine. The error appears only at
runtime after Hibernate bytecode enhancement has run.

### Context

Occurs when:
1. A `@MappedSuperclass` declares a field marked `@Transient` or `transient`
2. Quarkus/Hibernate bytecode enhancement is active (default in Quarkus)
3. A subclass `@Entity` inherits and accesses that field

### What was tried (didn't work)

- `@Transient` annotation on the field — stripped by enhancer
- Java `transient` keyword — also stripped by enhancer
- Both together — still stripped

### Root cause

Hibernate's bytecode enhancer runs at build time on `@MappedSuperclass` classes.
It strips **all non-persistent fields** from the enhanced bytecode — including
those explicitly marked `@Transient`. The enhancer treats `@Transient` as "not
part of the persistence model" and removes the field entirely, rather than
preserving it as a normal Java field that happens to not be persisted.

The Java `transient` keyword doesn't help — the enhancer treats both identically.

### Fix

**Option A — Remove the field** if it's write-only or unused by subclasses.
This was the fix in casehub-ledger #168: the `ledgerEntry` back-reference on
`LedgerSupplement` (`@MappedSuperclass`) was never read, so removing it was clean.

**Option B — Move the field to the `@Entity` subclass.** If the field IS needed
at runtime (e.g. a transient cache, a computed value), declare it on the `@Entity`
class instead of the `@MappedSuperclass`. The enhancer does not strip fields from
`@Entity` classes in the same way.

### Why this is non-obvious

`@Transient` is documented as "this field is not persisted" — the universal
expectation is that it still exists in memory as a normal Java field. Nothing in
Hibernate's or JPA's documentation says "bytecode enhancement physically removes
the field from the class." The compile succeeds; the error is runtime-only with
no warning.

### Why this fix

Removing the field (Option A) is preferred over moving it to the subclass when
the field exists only for API convenience (e.g. a back-reference that `attach()`
sets but nothing reads). Moving it to the subclass (Option B) adds a field to
every `@Entity` that may not need it. Evaluate per case — the deciding factor is
whether any code path actually reads the field.

### Alternatives considered

- Wrapping access in a `Map<UUID, LedgerEntry>` lookup instead of a direct field
  reference — rejected as over-engineering for a field that was never read.
