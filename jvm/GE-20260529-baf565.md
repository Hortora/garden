## Context

`@ObservesAsync` CDI observers are never called when the event is fired synchronously with `Event.fire()`. Only `Event.fireAsync()` delivers to `@ObservesAsync` observers. `Event.fire()` delivers only to `@Observes` (synchronous) observers.

## Symptom

`@ObservesAsync void onEvent(MyEvent event)` is never invoked. No error is thrown. No warning is logged. The `fire()` call returns successfully. The observer simply does nothing.

## Context

Occurs when:
- The event source calls `event.fire(myEvent)` (synchronous)
- One or more observers use `@ObservesAsync` expecting background delivery

Classic case: a CDI event is fired inside a reactive pipeline (`Uni.map()` callback) where blocking is undesirable. Developer adds `@ObservesAsync` to the observer so the heavy work runs on a different thread. The event fires, returns immediately — and the observer is never notified.

## Root cause

CDI specification (§10.6.2) is unambiguous: `Event.fire()` notifies only `@Observes` (synchronous) observers. `Event.fireAsync()` notifies both `@Observes` and `@ObservesAsync` observers. There is no crossover. This is not a bug in Quarkus — it is spec-compliant behaviour.

## Fix

**Option 1 (if you control the source):** Change `event.fire(myEvent)` to `event.fireAsync(myEvent)`. This delivers to `@ObservesAsync` observers on the CDI managed executor. Be aware that `fireAsync()` returns a `CompletionStage` — in a reactive context, convert it:

```java
event.fireAsync(myEvent)      // returns CompletionStage<MyEvent>
     .toCompletableFuture()   // optional: wait for completion
```

**Option 2 (if the source must stay synchronous):** Keep `@Observes` (synchronous) on the observer and spawn a virtual thread inside it for async fan-out:

```java
void onEvent(@Observes CaseChannelCreatedEvent event) {
    for (Peer peer : peers) {
        Thread.ofVirtual().name("sync-" + peer.id()).start(() -> callPeer(peer, event));
    }
    // returns immediately; peers called in background
}
```

This gives the caller's synchronous `fire()` a quick return, while the I/O work happens off-thread.

## Why this is non-obvious

The mental model — "async observer = runs on a different thread" — implies `fire()` still delivers the event, just asynchronously. The CDI specification has no such concept: `fire()` means synchronous delivery, period. Developers familiar with frameworks like Spring (where `@Async` on a method makes it run asynchronously regardless of how the caller invokes it) are especially likely to assume the same here.

The failure mode offers no diagnostic: no exception, no warning, no log message. The call succeeds and the observer simply never runs.

### Why this fix

Option 2 (`@Observes` + `Thread.ofVirtual()`) is preferable when the event source is in a reactive pipeline (`Uni.map()` callback on the Vert.x event loop). `fireAsync()` returns a `CompletionStage`, which must be composed with the `Uni` to avoid blocking — adding complexity. Option 2 keeps the reactive chain clean and the fan-out transparent to the caller.
