ConcurrentHashMap<String, ReentrantLock> as a drop-in replacement for Vert.x shared data locks when migrating to virtual threads, score: 8/15

### Context
Vert.x `vertx.sharedData().getLocalLock(key)` returns `Uni<Lock>` — it's designed for reactive pipelines on the event loop. When migrating `@ConsumeEvent` handlers from `Uni<Void>` to `@RunOnVirtualThread` + `void`, the Vert.x lock API no longer fits (requires Mutiny, returns Uni).

### Fix
Replace with a per-key `ReentrantLock` via `ConcurrentHashMap`:

```java
private final ConcurrentHashMap<String, ReentrantLock> locks = new ConcurrentHashMap<>();

void handleEvent(Event event) {
    String lockKey = "signal:" + event.caseId() + ":" + event.path();
    ReentrantLock lock = locks.computeIfAbsent(lockKey, k -> new ReentrantLock());
    lock.lock();
    try {
        // critical section — same semantics as Vert.x local lock
    } finally {
        lock.unlock();
    }
}
```

### Why this is non-obvious
`ReentrantLock` is virtual-thread-friendly — it uses `LockSupport.park()` internally, which virtual threads handle correctly (the virtual thread yields its carrier without pinning). In contrast, `synchronized` blocks pin the carrier thread, defeating virtual thread benefits. Developers migrating from reactive often reach for `synchronized` first, which is the wrong choice on virtual threads.

The `ConcurrentHashMap.computeIfAbsent` call does use `synchronized` internally for new-key creation, but only for the brief computation — this is acceptable at realistic contention levels.
