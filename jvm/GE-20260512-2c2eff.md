## Non-ANSI SQL types in Flyway migrations pass H2 tests silently but fail on PostgreSQL at deployment

### Symptom

Flyway migration passes all tests. First PostgreSQL deployment fails with `PSQLException: type "double" does not exist` or similar. No H2 test failure indicated the problem.

### What was tried (didn't work)

- Checking H2 test output — all green, migrations applied successfully
- Running H2 without `MODE=PostgreSQL` — also passes (H2 accepts even more type aliases without mode)

### Root cause

H2 accepts many vendor-specific SQL type aliases that PostgreSQL rejects. Common traps:

| Use this | Never use | Reason |
|---|---|---|
| `DOUBLE PRECISION` | `DOUBLE` | PostgreSQL has no bare `DOUBLE` type |
| `SMALLINT` | `TINYINT` | PostgreSQL has no `TINYINT` |
| `TIMESTAMP` | `DATETIME` | `DATETIME` is MySQL/H2-specific |
| `TEXT` | `CLOB`, `MEDIUMTEXT` | PostgreSQL uses `TEXT` |
| `BYTEA` | `BINARY`, `BLOB` | PostgreSQL uses `BYTEA` |

Even in `MODE=PostgreSQL`, H2 accepts some of these aliases silently.

### Fix

Use only ANSI SQL types in all Flyway migration scripts. Add `MODE=PostgreSQL` to H2 test URLs to catch as many issues as possible at test time — but treat PostgreSQL dialect test failures as the authoritative signal:

```properties
quarkus.datasource.jdbc.url=jdbc:h2:mem:testdb;MODE=PostgreSQL;DB_CLOSE_DELAY=-1
```

### Why this is non-obvious

H2 is designed for developer convenience and accepts many non-standard aliases. A migration that passes H2 with zero errors can still fail on PostgreSQL at deployment. The type incompatibility is completely silent in tests — no warning, no deprecation, just a `PSQLException` in production.
