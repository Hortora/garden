### Symptom
Compilation error: `Outer.Builder has protected access in Outer` — appears on a static interface method in a different package that uses `<B extends Outer.Builder<?, B>>` as a generic bound.

### Context
Designing an SPI interface (`DeviceTypeHandler`) that needs a shared utility method operating on a builder hierarchy. The base builder (`DeviceEntity.Builder<T, B>`) is a `protected abstract static class` — standard for builder hierarchies where the builder is constructed via concrete subclasses.

The static interface method in a different package uses the builder as a generic bound:
```java
public interface DeviceTypeHandler {
    static <B extends DeviceEntity.Builder<?, B>> B applyCommonFields(
            B builder, JsonNode node, ...) { ... }
}
```

### What was tried (didn't work)
The error message says "Builder has protected access" which suggests the issue is accessing Builder's members. But all Builder setter methods are already `public`. The error is not about calling methods on the builder — it's about the generic bound itself referencing a type that isn't visible.

### Root cause
Java's access rules apply to **type references in generic bounds**, not just method calls or field access. A `protected` nested class can only appear in a generic bound (`<B extends Protected.Class>`) from code within the same package or a subclass of the outer class. A static interface method in a different package cannot use a protected class as a generic bound — even though the concrete subclass builders it will actually receive are all public.

### Fix
Make the base builder `public`:
```java
// Before
protected abstract static class Builder<T extends DeviceEntity, B extends Builder<T, B>> { ... }

// After
public abstract static class Builder<T extends DeviceEntity, B extends Builder<T, B>> { ... }
```

### Why this is non-obvious
The error message "Builder has protected access in Outer" reads as a field/method access issue. A developer's first instinct is to check whether the builder's methods are public (they are) or to restructure the interface method to avoid naming the builder type directly. The actual fix — widening the nested class from `protected` to `public` — is simple but requires understanding that the access check applies to the type reference in the generic bound, not to any method call.

### Alternatives considered
- Moving the utility method into the same package as the builder — works but couples the SPI to the domain model's package structure
- Using `Object` as the bound and casting — loses type safety, defeats the purpose
- Making it a method on an abstract base class instead of a static interface method — adds unnecessary hierarchy
