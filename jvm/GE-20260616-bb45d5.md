### Symptom

Testcontainers fails with `Could not find a valid Docker environment` even though Podman is running and `docker ps` works. Setting `docker.host=unix:///path/to/socket` in `~/.testcontainers.properties` has no effect.

### Context

Podman installed via Homebrew on macOS. The Podman machine is running. `docker` CLI works (it's aliased to `podman`). But Testcontainers 2.x `UnixSocketClientProviderStrategy` hardcodes `/var/run/docker.sock` and ignores the `docker.host` property from `.testcontainers.properties`.

### What was tried (didn't work)

- Setting `docker.host` in `~/.testcontainers.properties` — ignored by the strategy
- Setting `docker.client.strategy=org.testcontainers.dockerclient.UnixSocketClientProviderStrategy` — loaded but still checks `/var/run/docker.sock`
- Setting `TESTCONTAINERS_RYUK_DISABLED=true` — unrelated to socket discovery

### Root cause

Testcontainers 2.x `UnixSocketClientProviderStrategy` does not read the `docker.host` property for socket path resolution. It only checks the hardcoded `/var/run/docker.sock`. The `DOCKER_HOST` environment variable IS respected by the Docker client library underneath, but only when set as an OS-level env var.

The Podman socket lives at a dynamic path under `/var/folders/` which changes on each Podman machine restart. Find it with:
```bash
find /var/folders -name "podman-machine-default-api.sock" -type s 2>/dev/null
```

### Fix

Set `DOCKER_HOST` as an environment variable (e.g. in `~/.zshenv`), dynamically resolved:

```bash
export DOCKER_HOST="unix://$(podman machine inspect --format '{{.ConnectionInfo.PodmanSocket.Path}}' 2>/dev/null)"
```

Strip `.testcontainers.properties` down to just:
```properties
testcontainers.reuse.enable=true
```

Do NOT hardcode the socket path — it lives under `/var/folders/` and changes on Podman machine restart.

### Why this is non-obvious

The Testcontainers documentation implies `docker.host` in `.testcontainers.properties` controls socket discovery. In practice, the 2.x `UnixSocketClientProviderStrategy` ignores it and only responds to the `DOCKER_HOST` environment variable. The error message ("Could not find a valid Docker environment") gives no hint that the socket path configuration is being ignored.
