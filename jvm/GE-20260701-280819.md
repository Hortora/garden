### Symptom
`Channel.memberCount()` always returns `null` despite parsing code correctly checking for `num_members` in the JSON response.

### Context
Building a cross-platform chat abstraction that surfaces channel member count. The Slack HTTP client calls `conversations.list` and parses `num_members` from the response JSON. WireMock-stubbed tests pass because the stub includes `num_members` in the response body.

### What was tried (didn't work)
Web search for "Slack conversations.list num_members" returns results stating the field is returned by default. The official docs show `num_members` in example responses without explicitly noting it requires a parameter.

### Root cause
The Slack API does **not** return `num_members` in `conversations.list` responses unless the query parameter `include_num_members=true` is explicitly passed. The same applies to `conversations.info`. Without this parameter, the field is absent from the JSON (not zero — absent entirely).

### Fix
Add `&include_num_members=true` to the API query string:
```java
private static final String LIST_BASE_QUERY =
    "?types=public_channel,private_channel&limit=200&include_num_members=true";
```

### Why this is non-obvious
Multiple web sources (including AI-generated documentation summaries) state that `conversations.list` returns `num_members` by default. The Slack docs show it in example responses. WireMock tests that manually include the field in stubs pass — creating a false sense of correctness. The failure is completely silent: no error, no warning, just `null` where an integer was expected.
