### Symptom

Config override for `@SupervisorAgent(maxAgentsInvocations=10)` does not work even though `AgenticServices.setWorkflowAgentsBuilder()` is correctly set with a config-aware wrapper.

### Context

`WorkflowAgentsBuilder` covers loop, sequence, parallel, conditional, and parallelMapper builders. But `SupervisorAgentService` is NOT created through it.

### Root cause

`buildSupervisorAgent()` in `AgenticServices` creates the builder directly:
```java
var builder = new SupervisorAgentServiceImpl<>(agentServiceClass, agentMethod, chatModel)
    .maxAgentsInvocations(supervisorAgent.maxAgentsInvocations())
```

This bypasses the `WorkflowAgentsBuilder` SPI entirely. There is no interception point.

### Fix

None available in current upstream. Config for `maxAgentsInvocations` can be declared in the config interface but cannot be wired until upstream provides either a supervisor builder SPI or the workflow-level `AgentConfigurator` (langchain4j/langchain4j#5399).

### Why this is non-obvious

All other workflow builders (loop, sequence, parallel, conditional) go through `WorkflowAgentsBuilder`. The supervisor is the exception — and there's no documentation or API design hint that it's different.
