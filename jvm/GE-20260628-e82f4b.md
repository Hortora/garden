### Symptom

A session (or cached object) is permanently recorded at the new generation but created
from the old data. Subsequent checks (`storedGen < currentGen`) evaluate to `false` —
the stale object is never invalidated.

### Context

Two volatile fields used for cross-thread publication: a "data" field (e.g. `volatile
KieBase kieBase`) and a "flag" field (e.g. `volatile long reloadGeneration`). The writer
updates data then increments the flag. The reader captures both at method entry. The
reader uses the flag to detect staleness and the data to create new objects.

### What was tried (didn't work)

Reading data first, then flag:

```java
// Writer (reload):
this.kieBase = newBase;        // volatile write: DATA
this.reloadGeneration++;       // volatile write: FLAG

// Reader (detect):
KieBase currentBase = this.kieBase;       // volatile read: DATA first
long currentGen = this.reloadGeneration;  // volatile read: FLAG second
```

On x86 (Total Store Order), this works accidentally — loads are never reordered relative
to other loads. On ARM/POWER (weaker memory models), the JVM inserts barriers, but the
JMM specification permits a valid synchronization order where the reader sees old data
and new flag.

### Root cause

The JMM (JLS §17.4.5) establishes happens-before only for volatile write → volatile
read **of the same variable**. The chain is:

    write(data) hb write(flag)         [program order in writer]
    write(flag) hb read(flag)          [volatile sync, same variable]
    read(data) hb read(flag)           [program order in reader]

Two things that both happen-before `read(flag)` — `write(data)` and `read(data)` — do
NOT have a happens-before relationship with each other. A valid synchronization order:

    read(data, B)          ← reader reads OLD data
      < write(data, A)     ← writer writes new data (but reader already read)
      < write(flag, A)     ← writer writes new flag
      < read(flag, B)      ← reader sees NEW flag

Result: reader gets `(oldData, newFlag)`. If it creates an object from `oldData` and
tags it with `newFlag`, the object is permanently stuck on stale data — the staleness
check compares generations and sees a match.

### Fix

Read the flag FIRST, then the data (standard acquire-release pattern):

```java
// Reader (detect):
long currentGen = this.reloadGeneration;  // volatile read: FLAG first (acquire)
KieBase currentBase = this.kieBase;       // volatile read: DATA second

// Writer (reload) — unchanged:
this.kieBase = newBase;        // volatile write: DATA first
this.reloadGeneration++;       // volatile write: FLAG second (release)
```

The happens-before chain now works:

    write(data) hb write(flag)     [program order in writer]
                hb read(flag)      [volatile sync, same variable]
                hb read(data)      [program order in reader]

By transitivity: `write(data)` hb `read(data)` — reader is guaranteed to see the
data written before flag N.

### Why this is non-obvious

The reverse order (data first, flag second) appears safe: "I read both volatile fields
at method entry, so I get a consistent snapshot." This intuition holds on x86 hardware
(TSO guarantees load-load ordering) but is not guaranteed by the JMM specification. The
bug is invisible on x86 and only manifests on weaker memory models — or when a
JIT compiler reorders the reads (permitted by the JMM).

The fix is a two-line swap with zero performance cost. The pattern is textbook
acquire-release, but the specific failure mode (permanent state corruption from a
one-time read reordering, not a transient inconsistency) makes it particularly
dangerous — it doesn't self-correct on retry.
