## Symptom

After adding a compile dependency to the runtime module that transitively includes new Quarkus extensions (e.g. `quarkus-config-yaml`, `quarkus-jackson`), updating `deployment/pom.xml` with the corresponding `*-deployment` artifacts, and running `mvn test -pl runtime`, the build fails with:

```
Quarkus Extension Dependency Verification Error
Deployment artifact io.example:my-extension-deployment::jar is missing the following
dependencies from its configuration:
-     io.quarkus:quarkus-config-yaml-deployment::jar
-     io.quarkus:quarkus-jackson-deployment::jar
```

The `-` prefix means "required but absent." The pom.xml was already edited to include them. A secondary symptom can appear as `TestEngine with ID 'archunit' failed to discover tests` — ArchUnit attempting class loading during a failed Quarkus augmentation.

## Context

Occurs when:
- The runtime module is tested without first installing the deployment module via `mvn install`
- The previous deployment JAR in `~/.m2` was built before the new deployment deps were added
- `mvn test -pl runtime` is used (does not trigger deployment reinstall)

## Root Cause

`quarkus-extension-maven-plugin` validates the extension descriptor by reading the **installed deployment JAR from the local `.m2` cache** — not from the deployment module's source `pom.xml`. The descriptor inside the cached JAR is stale (from before the new dependencies were declared). Editing `deployment/pom.xml` changes the source but the installed artifact remains unchanged until explicitly rebuilt and installed.

## Fix

Install the deployment module first to push the updated descriptor into `.m2`, then run the runtime tests:

```bash
mvn install -pl deployment -DskipTests
mvn test -pl runtime
```

Or simply build the full project from root the first time after any deployment pom change:

```bash
mvn install -DskipTests
```

For CI: ensure deployment always builds and installs before runtime tests.

## Why This Is Non-Obvious

The error message says the deployment artifact is "missing dependencies" and names the deployment module — so the natural assumption is to check `deployment/pom.xml`, find the missing entries, add them, and re-run. The file is correct; the installed cache is not. Nothing in the error message mentions the cache or that the source and installed state diverged. The `archunit` red-herring symptom further obscures the real cause.
