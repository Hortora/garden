# Share ExecutorService at @ApplicationScoped bean level with @PreDestroy — per-call Executors.newSingleThreadExecutor() leaks thread pools

### Symptom
Thread pool growth observed in `@ApplicationScoped` beans that enforce timeouts.
Each invocation creates an executor that is never shut down — the bean holds no reference after
the call returns, so the executor and its threads cannot be garbage collected.

### Context
An `@ApplicationScoped PolicyEnforcer` needs to submit work with a timeout (via
`Future.get(timeoutMs, TimeUnit.MILLISECONDS)`). The naive implementation creates a new
executor per call:

```java
// ❌ BAD — leaks one thread pool per call
private <T> T executeWithTimeout(Integer timeoutMs, Supplier<T> action) {
    ExecutorService executor = Executors.newSingleThreadExecutor(); // leaked
    Future<T> future = executor.submit(action::get);
    return future.get(timeoutMs, TimeUnit.MILLISECONDS);
}
```

Each call to `executeWithTimeout` creates a pool that persists for the lifetime of the JVM
because nothing shuts it down. Under load, this exhausts system threads.

### Fix
Share a single executor at the bean level; shut it down in `@PreDestroy`:

```java
@ApplicationScoped
public class DefaultPolicyEnforcer implements PolicyEnforcer {

    private final ExecutorService timeoutExecutor = Executors.newVirtualThreadPerTaskExecutor();

    @PreDestroy
    void shutdown() {
        timeoutExecutor.shutdownNow();
    }

    private <T> T executeWithTimeout(Integer timeoutMs, Supplier<T> action) {
        Future<T> future = timeoutExecutor.submit(action::get);
        try {
            return future.get(timeoutMs, TimeUnit.MILLISECONDS);
        } catch (TimeoutException e) {
            future.cancel(true);
            throw new PolicyEnforcementException("Action timed out after " + timeoutMs + "ms");
        } ...
    }
}
```

Using `newVirtualThreadPerTaskExecutor()` is preferred over a fixed pool for blocking I/O
or timeout-wrapped work — virtual threads are cheap, and the executor doesn't hold OS threads
idle between calls.

### Why this is non-obvious
The leak is invisible in development (low call rate, JVM typically restarted before exhaustion)
and non-fatal until load is applied. The symptom looks like a generic thread leak, not a
per-call executor creation issue. The `@PreDestroy` hook on the shared executor is easy to
forget when the executor is declared inline.
