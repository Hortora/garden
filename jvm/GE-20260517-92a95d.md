### Symptom
Compile error:

```
constructor WorkItemCreateRequest in record WorkItemCreateRequest cannot be applied to given types;
  required: String,String,String,String,WorkItemPriority,String,String,String,String,String,String,
            Instant,Instant,Instant,List<WorkItemLabelResponse>,Double,String,Integer,Integer,UUID,List<String>
  found:    [19 args]
  reason: actual and formal argument lists differ in length
```

### Context
Any code calling `new WorkItemCreateRequest(...)` with 19 positional arguments that was written against an earlier 0.2-SNAPSHOT build.

### Root cause
casehub-work 0.2-SNAPSHOT added two fields to the `WorkItemCreateRequest` Java record between builds (around 2026-05):
- Position 20: `UUID parentId` — parent WorkItem for sub-task hierarchies
- Position 21: `List<String> labels` — string labels for filtering/tagging

Java records use positional constructors — adding fields is a binary-incompatible change. Any code compiled against the old record signature fails to compile against the new one.

### Fix
Add two trailing `null` arguments to existing calls:

```java
workItemService.create(new WorkItemCreateRequest(
    title, description, category, type,
    priority, null, candidateGroups, null, null,
    sender, payload,
    slaDeadline, null, null,
    null, null, null, null, null,
    null,   // parentId (UUID) — added in 0.2-SNAPSHOT
    null    // labels (List<String>) — added in 0.2-SNAPSHOT
));
```

### Why this is non-obvious
`casehub-work` documentation does not publish a changelog for SNAPSHOT builds. The field addition is a silent binary-incompatible change. Tests compiled against an older version pass; tests compiled fresh against the new version fail. The error message is verbose and points at the call site, not the library.
