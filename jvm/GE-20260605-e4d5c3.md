### Symptom
`ContentRetriever.retrieve()` is never called after invoking a standalone agentic
agent, even though the retriever is correctly wired to the `AgentBuilder` via
`builder.contentRetriever(bean)`. The `AtomicBoolean called` flag on a recording
retriever stays `false`.

### Context
Occurs when testing CDI auto-wiring of `ContentRetriever` into agents. The agent
is invoked directly (`agent.ask("hello")`) rather than as a sub-agent inside a
`@SequenceAgent`, `@SupervisorAgent`, or other workflow composition. langchain4j
logs: "Improper invocation of a standalone agent outside of an agentic system,
consider using AiServices instead."

### What was tried (didn't work)
1. Verified the CDI bean was resolved (`assertThat(retriever).isNotNull()`) — passed
2. Verified the injection point existed (app booted without `UnsatisfiedResolutionException`) — passed
3. Added `assertThat(retriever.called.get()).isTrue()` after `agent.ask()` — failed
4. Checked `isResolvable()` on the `Instance<ContentRetriever>` — returned `true`
5. Confirmed `agentBuilder.contentRetriever(bean)` was called in the consumer — it was

### Root cause
langchain4j-agentic's `AgentInvocationHandler` only executes the full agent
lifecycle (including RAG augmentation via `RetrievalAugmentor`/`ContentRetriever`)
when the agent is invoked within an agentic system scope. Standalone invocation
bypasses the RAG pipeline — the `ContentRetriever` is wired to the builder but
never consulted during the invocation.

### Fix
Test CDI auto-wiring by verifying the injection point resolves (app boots) and the
agent is invocable, rather than asserting that `retrieve()` was called. To test
actual RAG invocation, wrap the agent in a `@SequenceAgent` or `@SupervisorAgent`
workflow.

### Why this is non-obvious
The `AgentBuilder.contentRetriever()` API suggests the retriever will be used on
every invocation. The wiring IS correct — the retriever IS set on the builder. The
disconnect is that the invocation handler skips RAG for standalone agents. There is
no error, no warning about the skipped retrieval — only the logged "improper
invocation" message hints at the limitation.
