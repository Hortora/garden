## The Technique

When a Quarkus CDI bean calls Panache Active Record statics (`Entity.findByIdOptional(id)`, `Entity.findByName(...)`, etc.), wrap each static call in a `protected` instance method. Unit tests override that method via an anonymous subclass, returning controlled test data without starting the Quarkus container.

```java
// Production CDI bean
@ApplicationScoped
public class HealthDomainLedgerHandler implements DomainLedgerHandler {

    @Inject LedgerEntryRepository ledgerRepository;
    @Inject LifeOutcomeAttestationWriter attestationWriter;

    // Package-visible test constructor
    HealthDomainLedgerHandler(LedgerEntryRepository repo, LifeOutcomeAttestationWriter writer) {
        this.ledgerRepository = repo;
        this.attestationWriter = writer;
    }

    HealthDomainLedgerHandler() {} // CDI no-arg

    @Override
    public void writeEntry(LifeDecisionEventType event, UUID workItemId, WorkItem workItem) {
        Optional<LifeTaskContext> ctxOpt = findContext(workItemId); // ← hook
        if (ctxOpt.isEmpty()) { LOG.warn("..."); return; }
        // ... business logic using ctx
    }

    // Hook — delegates to Panache static; overridable in tests
    protected Optional<LifeTaskContext> findContext(UUID workItemId) {
        return LifeTaskContext.findByIdOptional(workItemId);
    }
}

// Unit test — no @QuarkusTest, no database, runs in milliseconds
@ExtendWith(MockitoExtension.class)
class HealthDomainLedgerHandlerTest {

    @Mock LedgerEntryRepository repo;
    @Mock LifeOutcomeAttestationWriter writer;

    @Test void writeEntry_nullContext_doesNotWrite() {
        var handler = new HealthDomainLedgerHandler(repo, writer) {
            @Override protected Optional<LifeTaskContext> findContext(UUID id) {
                return Optional.empty(); // controlled test data
            }
        };
        handler.writeEntry(LifeDecisionEventType.COMPLETED, UUID.randomUUID(), new WorkItem());
        verify(repo, never()).save(any());
    }
}
```

The same pattern applies to any Active Record call inside a CDI bean:
- `Entity.findByIdOptional(id)` → `protected Optional<Entity> findById(UUID id)`
- `Entity.findByName(name)` → `protected Optional<Entity> findByName(String name)`
- `Entity.findByWorkItemId(id).ifPresent(...)` → `protected Optional<Entity> findRecord(UUID id)`

For SlaBreachPolicy (no CDI injection needed, just a Panache lookup):
```java
@ApplicationScoped
public class LifeSlaBreachPolicy implements SlaBreachPolicy {
    @Override
    public BreachDecision onBreach(SlaBreachContext ctx) {
        LifeDomain domain = resolveDomain(ctx); // ← hook
        ...
    }
    protected LifeDomain resolveDomain(SlaBreachContext ctx) {
        return LifeTaskContext.<LifeTaskContext>findByIdOptional(ctx.task().taskId())
                .map(tc -> tc.domain)
                .orElse(LifeDomain.HOUSEHOLD);
    }
}
```

## Why This Is Non-Obvious

The obvious alternatives are:
1. **Use `@QuarkusTest`** — starts the full Quarkus container, H2 database, CDI beans. Test takes 10–30s to start. Overkill for testing the handler's business logic (entity mapping, conditional fields, attestation calls).
2. **Use Panache Repository pattern** — injectable, mockable. But requires converting Active Record entities to Repository pattern, which is a large refactor of JPA entities.
3. **Use PowerMock/Mockito static mocking** — mocks the static call. Requires `mockStatic()`, adds `MockedStatic<T>` to every test method, and is incompatible with some CDI test setups.

The protected hook method is lighter than all three: zero test framework additions, zero entity refactoring, and the test runs in under 100ms. The tradeoff is that the `protected` scope slightly loosens access control — acceptable for non-public CDI beans where the test lives in the same package.

## Applicability

Any CDI `@ApplicationScoped` bean that calls Panache Active Record statics and needs unit-level (non-container) tests of its business logic. Most useful when:
- The bean has complex conditional field-population logic worth testing in isolation
- The bean injects other CDI services (mockable with Mockito) alongside the Panache static call
- Container startup cost is a concern (CI run time, test iteration speed)

Not needed when the entire test can be a pure computation with no entity lookups.
