### Symptom
Tests produce zero spans. No error, no warning. `InMemorySpanExporter.getFinishedSpanItems()` returns an empty list. All assertions that check for span names fail with "expected [reconcile] but was []".

### Context
Library instrumentation — a Quarkus library module (not an application) that calls `GlobalOpenTelemetry.getTracer()`. The test registers an `OpenTelemetrySdk` in `@BeforeEach`, but the instrumented class caches the tracer in a `static final` field.

### What was tried (didn't work)
Caching the tracer as a static field — the standard application instrumentation pattern:
```java
private static final Tracer TRACER = GlobalOpenTelemetry.getTracer("io.casehub.desiredstate");
```

### Root cause
`static final` fields initialize at class-load time. `GlobalOpenTelemetry.getTracer()` at that point returns a no-op tracer because no SDK has been registered yet (the test's `@BeforeEach` hasn't run). The no-op tracer is captured permanently — subsequent SDK registration does not retroactively update it.

### Fix
Call `GlobalOpenTelemetry.getTracer()` at method invocation time. The API caches internally, so repeated calls are cheap:
```java
private static final String INSTRUMENTATION_NAME = "io.casehub.desiredstate";

private void reconcile() {
    Tracer tracer = GlobalOpenTelemetry.getTracer(INSTRUMENTATION_NAME);
    Span span = tracer.spanBuilder("reconcile").startSpan();
    // ...
}
```

### Why this is non-obvious
The natural instinct is to cache the tracer — it's a common pattern in application instrumentation where the SDK is registered at startup before any class loads. Library instrumentation is different: the library is loaded by the consuming application, which may register the SDK at any point. The OTel docs mention "library instrumentation" vs "application instrumentation" but the static-field trap is not called out explicitly.

### Why this fix
`GlobalOpenTelemetry.getTracer()` is documented as safe to call repeatedly — the API maintains an internal cache keyed by instrumentation name. The per-invocation call pattern is explicitly recommended for library instrumentation in the OTel Java manual.

### Alternatives considered
- Injecting `Tracer` via CDI — works but couples the library to a CDI container, breaking non-CDI usage
- Lazy initialization with `volatile` + double-checked locking — unnecessary complexity since `getTracer()` already caches
