## Context

Integration tests use `withCrossTenantTransaction()` which executes `SET LOCAL ROLE casehub_crosstenancy`. The role is created by `RlsPolicyApplicator` when `casehub.rls.enabled=true` — but enabling RLS in tests has side effects (requires superuser to be unblocked). The role must exist BEFORE any test that uses the cross-tenant path, without necessarily enabling the full RLS stack.

## Technique

```properties
# src/test/resources/application.properties
quarkus.datasource.devservices.init-script-path=db/init-crosstenancy-role.sql
```

```sql
-- src/test/resources/db/init-crosstenancy-role.sql
DO $$
BEGIN
    IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname = 'casehub_crosstenancy') THEN
        CREATE ROLE casehub_crosstenancy BYPASSRLS;
    END IF;
END $$;
GRANT casehub_crosstenancy TO current_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO casehub_crosstenancy;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
    GRANT USAGE, SELECT ON SEQUENCES TO casehub_crosstenancy;
```

The init script runs once when Dev Services starts the container, before Hibernate schema creation. `ALTER DEFAULT PRIVILEGES` ensures tables created by `drop-and-create` automatically grant access to the role without needing to re-grant after each schema recreation.

## Why this is non-obvious

Developers typically reach for Flyway migrations or `@BeforeAll` setup to create roles. Flyway migrations don't run before schema creation; `@BeforeAll` runs after Quarkus starts (after the connection is already made). `devservices.init-script-path` is the only hook that runs before any application code — it's scoped to the Testcontainers startup phase.

The `ALTER DEFAULT PRIVILEGES` addition is also non-obvious: without it, tables created by Hibernate `drop-and-create` require explicit `GRANT` after each restart. `ALTER DEFAULT PRIVILEGES` applies to all future tables automatically.
