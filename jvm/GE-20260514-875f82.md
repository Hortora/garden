### Symptom

Attempting to use `InMemoryFooStore` from the extension's `testing` module in a unit test inside `runtime/src/test/` causes a Maven compile error: circular dependency detected.

### Context

Occurs in Quarkus extension projects structured as:
- `runtime/` — main runtime module with SPI interfaces
- `testing/` — utility module providing InMemory implementations of runtime SPIs for consumer unit tests

The testing module is designed for projects that *embed* the extension, not for the extension's own internal runtime tests.

### Root cause

The Maven dependency graph creates a cycle:
```
testing → runtime   (testing implements runtime SPIs — must depend on runtime)
runtime/src/test → testing   ← CYCLE
```

Maven detects this at compile time and rejects the build. The InMemory stores live in `testing` which already depends on `runtime`.

### Fix

For unit tests of runtime classes that need SPI implementations:

1. **Mockito** — mock the SPI directly and stub only the methods you need:
   ```java
   InstanceService svc = Mockito.mock(InstanceService.class);
   when(svc.findByInstanceId("worker-1")).thenReturn(Optional.of(instance));
   ```

2. **Inline stub** — create an anonymous class or static inner class inside the test:
   ```java
   InstanceStore store = new InstanceStore() {
       @Override public Optional<Instance> findByInstanceId(String id) {
           return id.equals("worker-1") ? Optional.of(instance) : Optional.empty();
       }
       // ...other methods
   };
   ```

Never import from the `testing` module in `runtime/src/test/`. The InMemory stores from `testing` are for consumer integration tests only.

### Why this is non-obvious

The testing module's InMemory stores look exactly like what you'd want for isolated unit tests — they're named `InMemory*Store`, they implement the right interfaces, and they're clearly intended for testing. Nothing in the naming or documentation signals that they can't be used from within the extension itself. The circular dependency only becomes apparent when Maven rejects the compile — there's no IDE warning before the build runs.
