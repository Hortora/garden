## Context

Quarkus library modules (no `quarkus:build` goal) that expose REST client DTOs containing
`java.time.Instant` fields.

## Symptom

The consuming REST service returns 422 (Unprocessable Entity) or silently misparses timestamps.
No error at startup or compile time. The failure only surfaces when the REST call fires at runtime.

## Root cause

Quarkus registers `JavaTimeModule` automatically, but `WRITE_DATES_AS_TIMESTAMPS` defaults to
`true` in many configurations. This causes `Instant` to serialize as an epoch-millisecond number
(e.g. `1749390600000`) rather than an ISO-8601 string. Library modules cannot rely on the
consuming application's global Jackson config to override this — the module ships independently
and must work regardless of how the app configures its `ObjectMapper`.

## Fix

Annotate the specific `Instant` field with `@JsonFormat` to force ISO-8601 output regardless
of global config:

```java
@JsonInclude(JsonInclude.Include.NON_NULL)
public record AddMessage(
        // ...
        @JsonFormat(shape = JsonFormat.Shape.STRING)
        @JsonProperty("timestamp")
        Instant timestamp
) {}
```

No changes to `application.properties` or `ObjectMapper` configuration are needed or safe.

## Why this is non-obvious

The natural fix is to add `quarkus.jackson.write-dates-as-timestamps=false` to the consuming
app's `application.properties` — but a library module cannot impose configuration on its
consumers. Even if the module's own test `application.properties` has the property set, the
consuming app might not. The per-field `@JsonFormat` annotation is the only approach that is
robust across all deployment contexts.

This belongs to the same failure class as `@Provider` vs `@RegisterProvider` on REST client
filters: silent at startup, only fails on the first real HTTP call in production.
