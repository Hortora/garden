### Symptom
`MismatchedInputException: Cannot deserialize value of type 'boolean' from null value (token 'JsonToken.VALUE_NULL')` at YAML deserialization time. The error mentions the field type (`boolean`) not the absent key, so developers first check whether the YAML structure is malformed rather than whether the key is present.

### Context
A Java record has a primitive `boolean` field (e.g. `AgentDisposition.delegation`). A YAML profile file omits that key entirely, assuming Jackson will default it to `false` as it would for a `String` or `Integer` field (which it sets to null). The record is loaded via `new ObjectMapper(new YAMLFactory()).findAndRegisterModules()`.

### Root cause
Jackson treats an absent YAML key as `null`. It can assign `null` to reference types (`String`, `Integer`, `Boolean`) — these just become null. It cannot assign `null` to a primitive `boolean` because primitives cannot hold null in Java. The error fires at the value assignment step, not at the key-lookup step, so the message names the type (`boolean`) rather than the absent key.

### Fix
Always include the primitive boolean field explicitly in every YAML file, even when the value is the default:
```yaml
disposition:
  delegation: false   # required — cannot be omitted; Jackson cannot default a primitive boolean to false
```
Document this requirement in the YAML schema notes for any record with primitive boolean fields.

### Why this is non-obvious
Jackson silently handles absent keys for reference types by leaving them null — this trains developers to assume absent keys are safe. The error message `Cannot deserialize value of type 'boolean' from null value` points at the field type, not the absent key. Developers first check: is the YAML key spelled correctly? Is the value malformed? — both wrong directions. The actual check needed is: is the key present at all?
