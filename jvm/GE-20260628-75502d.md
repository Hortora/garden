### Symptom
Test fails with `CaseInstance not found or wrong tenant` (or similar state-corruption error). The stack trace is clean, the error message is specific, and the failure is reproducible on retries. Debugging this error leads nowhere.

### Context
Maven Surefire with retry count > 1. The test starts an async investigation via the casehub-engine (Quartz workers, Vert.x event bus handlers). Run 1 of the test hits a different error (e.g. `ConditionTimeoutException` from Awaitility) and fails. Surefire restarts the Quarkus application for Run 2.

### What was tried (didn't work)
- Traced the `CaseInstance not found` error through `InMemoryCaseInstanceRepository.update()` — the save/update flow was correct
- Checked CDI bean resolution for `CaseInstanceRepository` — no ambiguity
- Added/removed `selected-alternatives` entries — no effect on Run 2's error
- Hours spent on the wrong error

### Root cause
Run 1's incomplete investigation leaves residual state: pending Quartz jobs, in-flight event bus messages, half-written ledger entries. When Surefire restarts the Quarkus app for Run 2, these residual artifacts interact with the fresh app state. The Run 2 error (`CaseInstance not found`) is a side effect of Run 1's incomplete cleanup — not the original failure.

The actual Run 1 error was a test deadlock: the test waited for an attestation that required gate approval, but gate approval came after the attestation wait. This was a `ConditionTimeoutException`, not `CaseInstance not found`.

### Fix
**Always check Run 1's error first.** In Surefire output, errors are listed per-run:
```
[ERROR] TestClass.testMethod
[ERROR]   Run 1: ConditionTimeoutException ...   ← THIS is the real error
[ERROR]   Run 2: CaseInstance not found ...       ← artifact of retry
[ERROR]   Run 3: CaseInstance not found ...       ← artifact of retry
```

Run 1's error is the root cause. Runs 2+ inherit corrupted state and show misleading errors.

### Why this is non-obvious
The retry error looks MORE authoritative than Run 1's error — it has a specific error message (`CaseInstance not found or wrong tenant: <UUID>`), a clean stack trace with line numbers, and it reproduces consistently on every retry. A `ConditionTimeoutException` looks like a flaky timeout. The natural instinct is to debug the specific, reproducible error — which is the wrong one.
