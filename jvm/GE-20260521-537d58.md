### Context
Plain-Java Flyway schema tests (no Quarkus, no CDI) for a module that depends on
another module's tables via JPA JOINED inheritance. The standard workaround is to
manually create the dependency table in `@BeforeAll` before running Flyway.

### Technique
Specify all migration locations directly — the same set as the production config.
Flyway sorts migrations by version number across all locations, so dependency tables
created by the upstream module's migrations (e.g. V1000) are guaranteed to exist
before the downstream subclass join migration (e.g. V2000) runs.

```java
@BeforeAll
static void migrate() {
    Flyway.configure()
            .dataSource(JDBC_URL, "sa", "")
            .locations("classpath:db/qhorus/migration", "classpath:db/ledger/migration")
            .load()
            .migrate();
}
```

No manual `CREATE TABLE` preamble needed. No `baselineOnMigrate` hack needed.

### Why this is non-obvious
The manual bootstrap approach works and is the obvious fix when Flyway complains about
a missing dependency table. Specifying upstream locations directly is not documented
as the idiomatic solution — it requires knowing that Flyway sorts across locations and
that the dependency module's JAR is on the test classpath (which it is, as a Maven
dependency). The multi-location approach also catches real version conflicts between
locations, which the manual bootstrap silently masks.
