### Symptom
Fetch URLs in browser-side JavaScript (inside an `html()` component template literal) return 404. The URL contains `${Mr}` (a minified variable name) literally in the path instead of the expected UUID value. No error at build time.

### Context
TypeScript constant `TRIAL_ID = "316e3846-..."` interpolated inside an `html()` template literal that contains a `<script>` block. The `${TRIAL_ID}` appears inside single-quoted JavaScript strings within the script block. esbuild bundles the file.

```typescript
const TRIAL_ID = "316e3846-4ea7-3b18-a6f7-e01ce6582a69";
html(`<script>
  fetch('/trials/${TRIAL_ID}/adverse-events')
</script>`)
```

### What was tried (didn't work)
1. Assumed TypeScript template literal interpolation would produce the literal UUID in the output string — it does, but esbuild then replaces the UUID with a minified variable reference `Mr`
2. The single-quoted `'/trials/${Mr}/...'` in the browser JS is literal text, not template interpolation — browsers treat `${Mr}` as characters, not a variable

### Root cause
esbuild processes the outer backtick template literal, replacing `${TRIAL_ID}` with its value. But during minification, esbuild detects the repeated UUID string and extracts it to a shared variable (`var Mr = "316e3846-..."`). The template output becomes `'/trials/${Mr}/adverse-events/'` — but this is inside single quotes in the HTML string content, where `${Mr}` is literal text in the browser.

### Fix
Declare the constants as `var` inside the `<script>` IIFE. TypeScript template interpolation places the UUID value in a JavaScript `var` declaration. Then use string concatenation for fetch URLs:

```typescript
html(`<script>
  (function() {
    var TRIAL_ID = "${TRIAL_ID}";  // Interpolated by TypeScript → "316e3846-..."
    fetch('/trials/' + TRIAL_ID + '/adverse-events')  // String concatenation, not template
  })();
</script>`)
```

### Why this is non-obvious
The TypeScript interpolation works correctly — `${TRIAL_ID}` IS replaced with the UUID. But esbuild's minification pass then extracts the UUID into a variable, producing output where the variable reference appears inside a single-quoted string. The failure is completely silent — no build error, no runtime error, just a 404 on the fetch that's hard to trace to a minification artefact.
