### Symptom
`ledgerEntryRepository.save(entry)` with `entry.notifiedAt = null` returned without error.
The test then failed at a later `assertThat(entry).isNotNull()` — appearing to suggest the
listener was propagating an exception, not that the write silently failed.

### Context
An `@ObservesAsync @Transactional(REQUIRED)` observer catches an exception and calls
a fallback method annotated `@Transactional(REQUIRES_NEW)`. Inside the fallback,
`entity.notifiedAt = null` is set and `ledgerEntryRepository.save(entity)` is called.
The entity's `@Column(name = "notified_at", nullable = false)` maps to a NOT NULL column.

### Root cause
Hibernate ORM queues inserts in an action queue — `save()` / `persist()` does not execute
the INSERT immediately. The INSERT executes during the flush, which happens at transaction
commit time. When the `REQUIRES_NEW` transaction commits, Hibernate flushes, the NOT NULL
constraint fires, Narayana throws `RollbackException` wrapped in `ArcUndeclaredThrowableException`.

An inner `catch (Exception e)` around the fallback call correctly swallowed the exception —
the outer observer did not throw. But the fallback write never persisted.

### Fix
Do not set NOT NULL columns to null. If the semantic intent is "unknown/not applicable",
use a separate nullable column as the discriminator. For example, `connectorId=null` can
distinguish "error occurred before the notifier was reached" from `connectorId` set +
`delivered=false` ("connector delivery failed") — no schema change required.

Alternatively, make the column nullable via a Flyway migration before attempting null writes.

### Why this is non-obvious
`save()` / `persist()` completes without exception in both success and failure cases.
The constraint violation is asynchronous from the application's perspective — it surfaces
only at TX commit, wrapped in JTA-specific exception types. The inner catch correctly
swallows the wrapped exception, so no exception propagates, but the write still failed.
The REQUIRES_NEW boundary gives the impression the inner TX is isolated — it is, but
the isolation includes the rollback, which is invisible to the caller.

### Why this fix
Setting null in a NOT NULL column and expecting it to persist (or to fail visibly at
save() time) is the natural assumption. The flush-at-commit model means the failure
arrives far from the call site, wrapped in JTA machinery.
