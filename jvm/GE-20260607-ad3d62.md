### Symptom

A previously-passing test suite starts failing with:

```
org.hibernate.exception.SQLGrammarException:
  Could not prepare statement [Table "LEDGER_SUBJECT_SEQUENCE" not found; SQL statement:
  MERGE INTO ledger_subject_sequence AS t USING ...]
  at io.casehub.ledger.runtime.repository.jpa.LedgerSequenceAllocator.nextSequenceNumber
```

All tests that perform any ledger write (via `JpaLedgerEntryRepository.save()`) fail.
Tests that mock `LedgerEntryRepository` via `@InjectMock` are unaffected and continue passing.

### Context

A casehub-ledger SNAPSHOT introduced `LedgerSequenceAllocator` — a class that uses a native SQL
`MERGE INTO ledger_subject_sequence` statement to allocate sequence numbers per subject. The table
is not a Hibernate `@Entity`; it is created by a Flyway migration in `classpath:db/ledger/migration`.

In `@QuarkusTest` with `quarkus.hibernate-orm.database.generation=drop-and-create`, Hibernate
drops and recreates only the tables it knows about through its `@Entity` mappings. `ledger_subject_sequence`
is invisible to Hibernate. The Flyway migration that creates it is at `classpath:db/ledger/migration`,
which is typically not in the test Flyway scan path. With `quarkus.flyway.migrate-at-start=false` (common
in multi-datasource setups to avoid V-number collisions), Flyway never runs. The table does not exist.

### Root Cause

Hibernate `drop-and-create` operates on the JPA entity model only. Native SQL tables created by
Flyway migrations are invisible to it. If Flyway is disabled in tests (a common defensive measure
when casehub-work and casehub-qhorus migrations share version numbers with application migrations),
there is no path to create `ledger_subject_sequence`.

### Fix

Switch `JpaLedgerEntryRepository` to `InMemoryLedgerEntryRepository` in test `selected-alternatives`.
The in-memory implementation holds all state in a `ConcurrentHashMap` — no database table required.

In test `application.properties`:
```properties
# Replace JpaLedgerEntryRepository with in-memory alternative (avoids ledger_subject_sequence table)
quarkus.arc.selected-alternatives=io.casehub.ledger.memory.InMemoryLedgerEntryRepository,...

# Index the in-memory module (no embedded Jandex)
quarkus.index-dependency.ledger-memory.group-id=io.casehub
quarkus.index-dependency.ledger-memory.artifact-id=casehub-ledger-memory
```

In `pom.xml` (test scope):
```xml
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-ledger-memory</artifactId>
  <scope>test</scope>
</dependency>
```

`InMemoryLedgerEntryRepository` is `@Alternative @Priority(1)` — it wins over `JpaLedgerEntryRepository`
when selected. Tests that assert on `ledgerRepo.findBySubjectId()` continue to work correctly because
the in-memory implementation persists entries within the same Quarkus application context.

### Why this is non-obvious

The failure surfaces as a SQL error deep in the ledger internals — the stack trace does not mention
`JpaLedgerEntryRepository` or `LedgerSequenceAllocator` in application code. The `drop-and-create`
log shows all `@Entity` tables being created successfully, which gives the impression the schema is
complete. The missing table is a native SQL artefact that Hibernate has no knowledge of — the connection
between "switch to in-memory repository" and "fix a SQL table not found" is not obvious.

The engine's own tests always use `InMemoryLedgerEntryRepository` in `selected-alternatives`. Downstream
consumers (clinical, AML, life) that originally used `JpaLedgerEntryRepository` for richer test coverage
hit this failure when the ledger snapshot changed without warning.

### Alternative Fix — SQL init script (keeps real JPA semantics) (2026-06-09)

**When to use this over the InMemoryLedgerEntryRepository fix:** When tests depend on real JPA semantics — specifically, when `MessageLedgerEntryRepository.findByMessageId(Long)` or similar queries must find entries written by `LedgerEntryRepository.save()`. If `save()` goes to an in-memory store and `findByMessageId()` queries H2 via JPQL, they share no backing store — timeline tests that dispatch EVENTs and then query the ledger to render telemetry will see empty results.

**Fix:** Create a SQL init script that Hibernate runs after `drop-and-create`:

`src/test/resources/import-qhorus-test.sql`:
```sql
CREATE TABLE IF NOT EXISTS ledger_subject_sequence (
    subject_id UUID        PRIMARY KEY,
    next_seq   BIGINT      NOT NULL
);
```

In test `application.properties`:
```properties
quarkus.hibernate-orm.qhorus.sql-load-script=import-qhorus-test.sql
```

Hibernate's `sql-load-script` runs after `drop-and-create` schema generation, before tests start. `CREATE TABLE IF NOT EXISTS` is idempotent — safe when the Quarkus context restarts mid-test-suite.

**Known caveat:** `ledger_subject_sequence` rows are NOT reset by `drop-and-create` (the table is invisible to Hibernate). With `DB_CLOSE_DELAY=-1`, the H2 database persists for the JVM lifetime; rows survive context restarts triggered by `@TestProfile`. Tests must use fresh random UUIDs per run as subjectIds to avoid cross-context sequence pollution.

The same script must be added to every test module that enables the ledger (`casehub.ledger.enabled=true`) and disables Flyway — including example modules. Refs qhorus#256.
