## What it does

When a JAX-RS endpoint is annotated `@Produces("text/event-stream")` and returns `Multi<String>`, RESTEasy Reactive automatically wraps each emitted string as a bare SSE event frame:

```
data: <emitted-string>\n\n
```

The client's `EventSource.onmessage` receives the string in `event.data`. No annotation or configuration is needed — this wrapping is implicit.

## Why it matters

If you want to emit **named SSE events** (with an `event:` field) or include SSE **id fields** (for `Last-Event-ID` reconnect), returning `Multi<String>` is insufficient — there is no way to add these fields to the frame. You must change the return type to `Multi<OutboundSseEvent>` (from `jakarta.ws.rs.sse`) and construct events explicitly:

```java
// ❌ String emission — only produces bare data: frames, no id: or event: possible
@GET @Path("/events") @Produces("text/event-stream")
public Multi<String> events() {
    return Multi.createFrom().ticks().every(Duration.ofSeconds(1))
        .map(tick -> "{\"status\":\"ok\"}");
}

// ✅ OutboundSseEvent — full SSE control (id, event type, data, comment)
// Requires injecting @Context Sse sse; or using SseEventBuilder
@GET @Path("/events") @Produces("text/event-stream")
public Multi<OutboundSseEvent> events(@Context Sse sse) {
    return Multi.createFrom().ticks().every(Duration.ofSeconds(1))
        .map(tick -> sse.newEventBuilder()
            .id(String.valueOf(tick))
            .name("mesh-update")
            .data("{\"status\":\"ok\"}")
            .build());
}
```

## Where to find it

This behaviour is not documented in the RESTEasy Reactive SSE docs or the Quarkus reactive messaging guide. It can be confirmed by reading `ServerSentEventMessageBodyWriter` in the RESTEasy Reactive source, or by observing the raw SSE frames in the browser network tab.

## Practical implication

Designing an SSE endpoint that starts with `Multi<String>` and later needs named events or reconnect cursors requires changing the return type — a non-trivial refactor if the endpoint is already integrated with client code. Decide upfront whether named events or id fields are needed.
