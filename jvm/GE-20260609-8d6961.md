### What a skilled developer would do instead

Place the CDI qualifier only on the injection point (`@Inject @CrossTenant CrossTenantChannelStore`). This is the canonical CDI usage. When an `@Alternative @Priority(1)` bean is added to the same interface, developers either add `@Any` to the injection point or use `@Produces` to disambiguate — both requiring changes to consumer code.

### The technique

Place the CDI qualifier on the **bean class itself** (not just on injection points). The qualifier becomes part of the bean's type set — the alternative only satisfies injection points bearing that qualifier, leaving plain `@Default` injection points unresolved (correctly):

```java
// Bean class carries the qualifier — restricts what injection points it satisfies
@Alternative
@Priority(1)       // globally enabled in @QuarkusTest contexts
@ApplicationScoped
@CrossTenant       // ← qualifier on the CLASS, not just an injection point
public class InMemoryCrossTenantChannelStore implements CrossTenantChannelStore {
    // ...
}
```

Without `@CrossTenant` on the class, this `@Alternative @Priority(1)` bean would satisfy ALL injection points of type `CrossTenantChannelStore` — including unqualified ones — causing ambiguity or incorrect wiring.

With it, only `@Inject @CrossTenant CrossTenantChannelStore` resolves to this bean. Plain `CrossTenantChannelStore` injection points remain unresolved (as intended if no default exists) or resolve to a different bean.

### Why this is non-obvious

CDI qualifiers are almost always discussed in the context of injection points, not bean declarations. The CDI spec does allow qualifiers on bean classes, but the typical mental model is "qualifier narrows *where* a bean is injected" not "qualifier restricts *which* injection points the bean can satisfy." The distinction matters when:

- You have two beans implementing the same interface — one tenant-filtered, one cross-tenant
- You want the cross-tenant version to satisfy ONLY explicitly-qualified injection points
- `@Alternative @Priority(1)` would otherwise satisfy all injection points of the type, overriding the tenant-filtered bean everywhere

### When to use

Use when:
- You have a producer method or `@Alternative` that should only wire to explicitly-qualified injection points
- The same interface has both a "default scoped" and a "privilege-scoped" implementation (e.g. tenant-filtered vs cross-tenant, or authenticated vs system-actor)
- Multiple injection points exist and you want exactly one category to use the alternative

### Verified on

Quarkus 3.32.2, CDI 4.0 (Jakarta EE 10).
