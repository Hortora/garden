### Context

An event-driven consumer (e.g. a delivery pump polling a `LinkedBlockingDeque`) needs to react to a committed database write. The producer writes a row in a JTA transaction, then signals the consumer to wake up and process the new data.

### Technique

Signal the consumer via `TransactionSynchronizationRegistry.registerInterposedSynchronization()` with `afterCompletion(STATUS_COMMITTED)`, not inline after the write:

```java
boolean hasWork = doBusinessLogic(); // writes to DB within @Transactional
if (hasWork) {
    final UUID key = entityId;
    tsr.registerInterposedSynchronization(new Synchronization() {
        @Override public void beforeCompletion() {}
        @Override public void afterCompletion(int status) {
            if (status == STATUS_COMMITTED) {
                signalQueue.signal(key); // consumer wakes AFTER commit
            }
        }
    });
}
```

### Root Cause (why inline signaling fails)

Under PostgreSQL READ COMMITTED isolation (the Quarkus default), an inline signal — `dbWrite(); signalQueue.signal(key);` — wakes the consumer while the producer's transaction is still open. The consumer's query runs in a separate transaction and cannot see uncommitted rows. Result: the consumer wakes, queries, finds nothing, goes back to sleep. The data is committed moments later but the consumer never retries.

The race window is small (microseconds between signal and commit) but deterministic under load — the consumer's query is faster than the producer's commit.

### Why this is non-obvious

The inline approach works in testing (H2 in-memory, single-thread, auto-commit) and in low-load development. The race only manifests under concurrent load with a real PostgreSQL database, making it a production-only failure.

### Alternatives considered

- **Polling with interval** — consumer polls every N seconds regardless of signals. Works but adds latency equal to the polling interval. The TSR approach gives near-zero latency in the happy path.
- **CDI `@Observes(during = AFTER_SUCCESS)`** — fires after commit, but requires a CDI event type. The TSR approach works with any signaling mechanism (queues, semaphores, latches).
