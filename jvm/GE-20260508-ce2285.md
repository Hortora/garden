## Problem

`@QuarkusTest` with `jdbc:h2:mem:dbname;DB_CLOSE_DELAY=-1` shares a single in-memory H2 database across all test classes in the same JVM run. `@TestTransaction` rolls back — but only for direct persistence calls. REST Assured HTTP calls commit in their own transactions and are not rolled back. Hardcoded business keys (e.g. `protocolId = "ONCOL-001"`) accumulate across test classes and fail on unique constraints when tests run in sequence, producing non-deterministic failures that depend on test ordering.

## Technique

Suffix every test-data business key with `UUID.randomUUID()`:

```java
// Instead of:
.body("{\"protocolId\": \"ONCOL-001\", ...}")

// Use:
.body("{\"protocolId\": \"ONCOL-" + UUID.randomUUID() + "\", ...}")
```

Or in a helper:

```java
private String uniqueProtocol() {
    return "ONCOL-" + UUID.randomUUID();
}
```

This costs nothing, eliminates state conflicts entirely, and requires no `@TestTransaction` (which breaks REST Assured assertions anyway — see related GE-0036).

## Why This Is Non-Obvious

`@TestTransaction` is the documented Quarkus mechanism for test isolation. Developers apply it expecting it to roll back all state, then find REST Assured tests still failing due to stale data. The disconnect — that HTTP calls commit outside `@TestTransaction`'s scope — is not surfaced in the Quarkus testing docs. The UUID-suffix pattern is simple once known but easy to miss when you're debugging apparently-random test ordering failures.

## Constraints

Applies when: tests use `DB_CLOSE_DELAY=-1` (shared H2), data is inserted via HTTP endpoints (not direct EntityManager), and there is a unique constraint on the business key column.
