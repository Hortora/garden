### Symptom

```
org.hibernate.QueryParameterException: No argument for named parameter ':tenancyId'
  at ReactiveSqmQueryImpl.executeReactiveUpdate
```

All message dispatches fail with HTTP 500 when `casehub.ledger.hash-chain.enabled=true`.

### Context

`LedgerMerkleFrontier.deleteBySubjectAndLevel` is a `@NamedQuery` defined in `casehub-ledger`.
The ledger repo added a `:tenancyId` predicate to the query. `ReactiveLedgerEntryJpaRepository.save()`
in `casehub-qhorus` (a sibling repo) calls this named query but was never updated to bind the new parameter.
The call site compiled and deployed without error — the unbound parameter only surfaces at runtime
when the query executes.

### Root cause

JPA named queries are resolved by name at runtime, not by signature at compile time. When the
query definition in one repo gains a new parameter, every call site in every sibling repo that
uses the same named query must be manually updated. There is no compile-time or build-time
enforcement — neither the Java compiler nor the Maven build reports the mismatch.

### Fix

Bind the new parameter at every call site that uses the named query:

```java
session.createNamedQuery("LedgerMerkleFrontier.deleteBySubjectAndLevel")
    .setParameter("subjectId", e.subjectId)
    .setParameter("level", node.level)
    .setParameter("tenancyId", e.tenancyId)  // ← add when query gains this param
    .executeUpdate()
```

### Why this is non-obvious

The build succeeds. The CDI wiring is correct. The error only appears on a specific code
path (`hash-chain.enabled=true`) that may not be covered by standard CI tests. The exception
message names the parameter (`':tenancyId'`) but gives no indication of which call site is
missing the binding — a classpath-wide search is required to find the stale call site.
