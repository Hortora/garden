### Symptom

`TrustGateService.currentScore()` returns the pre-incident trust score (0.917) after `TrustScoreJob.runComputation()` writes the correct updated score (0.884) to the `actor_trust_score` table. A direct JPQL query against the same EntityManager (`SELECT a FROM ActorTrustScore a WHERE a.actorId = :actorId`) confirms the DB row has 0.884. But `currentScore()` returns 0.917.

The mismatch persists across multiple `runComputation()` calls. The computation is correct — the read is stale.

### Context

In a `@QuarkusTest` with `@TestProfile(TrustScoringTestProfile.class)`:

1. Test seeds `WorkerDecisionEntry` + `LedgerAttestation(SOUND)` rows via `QuarkusTransaction.requiringNew()`
2. Calls `trustScoreJob.runComputation()` inside `QuarkusTransaction.requiringNew()` — this computes and persists `ActorTrustScore` rows
3. Reads the score via `trustGateService.currentScore(agentId, capability)` — returns the correct value (0.917)
4. Test writes a `LedgerAttestation(FLAGGED)` via `IncidentFeedbackService.recordFeedback()`
5. Calls `trustScoreJob.runComputation()` again — correctly recomputes to 0.884
6. Reads the score via `trustGateService.currentScore()` — **returns 0.917 (stale)**

Step 3's read loaded the `ActorTrustScore` entity into the test's ambient persistence context. Step 5's write happened in a `REQUIRES_NEW` transaction (separate EntityManager). Step 6's read finds the entity in the L1 identity map and returns it without a DB roundtrip.

### What was tried (didn't work)

- Running `trustScoreJob.runComputation()` multiple times — score stays at 0.917
- Using `TrustScoreJob` batch computation instead of `IncrementalTrustUpdateObserver` — same result
- Direct `TrustScoreComputer.compute()` with the same attestation data — correctly returns 0.513, proving the computation is correct
- Filing a bug against the trust computation library (casehub-ledger#157) — the library was correct; the bug was in the test's read path

### Root cause

JPA's first-level cache (identity map) is scoped to the `EntityManager` instance. In `@QuarkusTest`, the test method runs in a CDI request scope with one ambient `EntityManager`. CDI beans injected into the test (`TrustGateService`, `MaterializedTrustScoreSource`) share this `EntityManager`.

When `TrustScoreJob.runComputation()` runs inside `QuarkusTransaction.requiringNew()`, it gets a **different** `EntityManager` for the duration of that transaction. It correctly reads attestations, computes scores, and persists `ActorTrustScore` rows — all committed to the DB.

But the ambient `EntityManager` (from the test's request scope) still has the OLD `ActorTrustScore` entity in its L1 cache from step 3's read. When `TrustGateService.currentScore()` executes in the ambient scope, `MaterializedTrustScoreSource` queries via the ambient `EntityManager` — Hibernate finds the entity by primary key in the identity map and returns it without a DB query. The stale value is served.

### Fix

Wrap every score read that follows a `requiringNew()` write in its own `QuarkusTransaction.requiringNew()`:

```java
// WRONG — reads from ambient L1 cache (stale)
OptionalDouble score = trustGateService.currentScore(agentId, capability);

// CORRECT — fresh EntityManager, fresh DB read
OptionalDouble score = QuarkusTransaction.requiringNew().call(
    () -> trustGateService.currentScore(agentId, capability));
```

This creates a new persistence context with an empty identity map, forcing a DB roundtrip.

### Why this is non-obvious

The `requiringNew()` pattern is well-known for WRITE isolation — ensuring writes commit independently. What's surprising is that the STALE READ comes through a completely different CDI bean (`TrustGateService`) that was never involved in the original `requiringNew()` transaction. The mental model "each service gets its own EntityManager" is wrong in `@QuarkusTest` — all CDI beans in the same request scope share one ambient `EntityManager`.

The mismatch between direct JPQL (`SELECT a FROM ActorTrustScore a`) showing the correct value and `TrustGateService.currentScore()` returning the stale value is especially misleading — it suggests a bug in the service layer or the trust computation, when neither is wrong.

### Why this fix

`em.clear()` on the ambient EntityManager would also work but is a blunt instrument — it evicts ALL cached entities, not just the stale one. `requiringNew()` on the read side is surgical: it creates a throwaway persistence context for exactly one query, leaving the ambient cache intact for other operations that may still need it.

### Alternatives considered

- **`em.clear()` before reading** — works but evicts all cached entities, potentially causing `LazyInitializationException` on other entities loaded in the same scope
- **`@TestTransaction` on the test method** — changes the entire test's transaction semantics, which is not always desirable (the test may need multiple independent transactions)
- **`em.refresh(entity)` on the specific entity** — requires access to the entity instance, not the service method that returns a primitive `OptionalDouble`
