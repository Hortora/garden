### Symptom
Compile error: `package io.vertx.mutiny.ext.web.client does not exist` after adding `quarkus-vertx` as a dependency.

### Context
Adding `quarkus-vertx` to a Quarkus project that needs to make HTTP calls via Vert.x WebClient. The developer expects `quarkus-vertx` to provide the full Vert.x API surface.

### What was tried (didn't work)
Adding only `quarkus-vertx` — provides `io.vertx.core.Vertx`, `io.vertx.mutiny.core.Vertx`, `EventBus`, and timer APIs, but NOT `WebClient`.

### Root cause
`quarkus-vertx` provides the Vert.x core runtime only: `Vertx` instance, `EventBus`, timers. The WebClient is a separate Vert.x module (`vertx-web-client`) with its own Mutiny wrapper (`smallrye-mutiny-vertx-web-client`). Both are managed by the Quarkus BOM — no version needed — but they are distinct artifacts.

### Fix
Add `smallrye-mutiny-vertx-web-client` alongside `quarkus-vertx`:

```xml
<dependency>
    <groupId>io.smallrye.reactive</groupId>
    <artifactId>smallrye-mutiny-vertx-web-client</artifactId>
</dependency>
```

This gives you `io.vertx.mutiny.ext.web.client.WebClient` which returns `Uni<HttpResponse<Buffer>>` — reactive-native, no thread pool hop needed.

### Why this is non-obvious
The name `quarkus-vertx` strongly implies "Quarkus's Vert.x integration" — developers expect it to include the WebClient because WebClient is a standard Vert.x module. The split between core runtime and web client is a Vert.x modular packaging decision that isn't surfaced in the Quarkus dependency name.
