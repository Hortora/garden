## Context

A `@QuarkusTest` class extends an abstract contract test class (from the api test-jar, see GE-20260529-a4dadf). The contract test declares `@Test` methods that perform JPA operations via the SUT. Each test needs a clean database state — the standard approach is per-method `@TestTransaction` so each test runs in a rolled-back transaction.

## Technique

Place `@TestTransaction` at the class level rather than on each inherited method. Quarkus respects class-level `@TestTransaction` on all test methods in the class, including those inherited from abstract superclasses:

```java
@QuarkusTest
@TestTransaction          // ← class-level: applied to all test methods including inherited ones
class JpaAgentStateStoreTest extends AgentStateStoreContractTest {

    @Inject AgentStateStore store;

    @Override
    protected AgentStateStore store() { return store; }
    // All 7 inherited contract test methods automatically run in rolled-back transactions
    // No per-method annotation needed
}
```

Each `@Test` method — including those defined in `AgentStateStoreContractTest` — runs in its own transaction that is rolled back after the method completes, giving clean JPA state for every test with zero boilerplate in the concrete class.

## Why This Is Non-Obvious

The per-method annotation approach is what the Quarkus docs show. Whether `@TestTransaction` applies to inherited methods when placed at the class level is not documented. Developers who know they can't annotate the abstract class's methods (to keep it framework-agnostic) typically either: duplicate all tests in the concrete class with `@TestTransaction` added, or add a `@BeforeEach @Transactional void cleanup()` that manually deletes rows. Class-level `@TestTransaction` eliminates both workarounds.

## Constraints

Does not apply to reactive `@QuarkusTest` — `@RunOnVertxContext` + `UniAsserter` tests cannot use `@TestTransaction` (different execution model). In-memory implementations don't need `@TestTransaction` since they use a fresh instance per test via `@BeforeEach`.
