### Symptom
Same keyword query against the same engine returns completely different top-8 results after minor corpus growth. NL queries on the same corpus remain stable.

### Context
Benchmark comparing dense-only retrieval (nomic-embed-text, 768-dim, cosine) across the same 14 scenarios at two points in time. Corpus grew from 1,960 to 1,984 entries (24 new, 1.2% growth). Queries are pipe-separated Java class names and CDI annotations, e.g. `DefaultBean|AmbiguousResolutionException|GroupMembershipProvider|ambiguous`.

### What was tried
Re-ran all 28 queries (14 scenarios × KW + NL) expecting identical results to the #27 baseline. NL queries matched 92% (expected drift from 24 new entries). KW queries matched only 24% — near-complete result set replacement.

### Root cause
Pipe-separated keyword strings produce embeddings that sit near the decision boundary in the vector space. The embedding model (nomic-embed-text) treats the entire string — including pipe characters — as a single sentence. Java class names are tokenized into generic subwords (`AmbiguousResolutionException` → `["am", "##bi", "##guous", "##reso", "##lution", "##exception"]`), producing a low-information embedding that is equidistant from many candidate entries. Small corpus perturbations (24 new entries) shift which entries are nearest neighbors, replacing most of the top-8.

NL queries are stable because natural language sentences produce high-information embeddings with clear semantic direction — adding 24 entries doesn't shift the nearest-neighbor boundary.

### Fix
Do not use embedding search for keyword-style queries. Use BM25/full-text index for exact token matching (Qdrant supports payload text indexes). Reserve embedding search for natural language queries where it is both high-quality and stable.

See GE-20260627-4712de for the complementary finding: keyword embedding is not just unstable, it also produces low-quality results (vocabulary gap).

### Why this is non-obvious
Embedding search is presented as a general-purpose retrieval method. The instability failure mode is invisible in small benchmarks — you only discover it when re-running the same queries after a corpus update. A developer would assume that adding 24 entries to a 1,960-entry corpus has negligible effect on existing query results.
