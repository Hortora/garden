### Symptom

A `@DefaultBean` reactive bridge (`BlockingToReactiveCbrBridge`) that injects a blocking SPI interface (`CbrCaseMemoryStore`) silently resolves to the no-op `@DefaultBean` implementation (`NoOpCbrCaseMemoryStore`) instead of the `@Alternative @Priority(2)` in-memory implementation (`InMemoryCbrCaseMemoryStore`), even when:

- The alternative is listed in `quarkus.arc.selected-alternatives`
- Both JARs are indexed via `quarkus.index-dependency`
- Direct injection of the blocking interface in a test class correctly resolves to the alternative

The retrieval returns empty with no error or warning — completely silent.

### Context

This occurs when three beans implement the same interface across two external JARs:

- **JAR A** (`casehub-neocortex-memory`): `NoOpCbrCaseMemoryStore` (`@DefaultBean @ApplicationScoped`) and `BlockingToReactiveCbrBridge` (`@DefaultBean @ApplicationScoped`, injects `CbrCaseMemoryStore`)
- **JAR B** (`casehub-neocortex-memory-cbr-inmem`): `InMemoryCbrCaseMemoryStore` (`@Alternative @Priority(2) @ApplicationScoped`)

The bridge in JAR A is supposed to delegate to whichever `CbrCaseMemoryStore` is active. When `InMemoryCbrCaseMemoryStore` is activated via `selected-alternatives`, direct injection works — but the bridge's own injection point does not.

### What was tried (didn't work)

1. Adding `quarkus.index-dependency` for both JARs — no effect on resolution
2. Adding `InMemoryCbrCaseMemoryStore` to `quarkus.arc.selected-alternatives` in the memory profile — direct injection works, bridge delegation does not
3. Adding `quarkus.arc.exclude-types` for `NoOpCbrCaseMemoryStore` — breaks CDI wiring entirely (the bridge has no delegate)

### Root cause

Quarkus ARC resolves `@DefaultBean` beans' injection points at build time. When the bridge (`@DefaultBean`) is being wired, ARC resolves its `@Inject CbrCaseMemoryStore` using the standard `@DefaultBean` resolution rules — which picks the other `@DefaultBean` (`NoOpCbrCaseMemoryStore`) in the same JAR. The `@Alternative` from the external JAR is not considered for the bridge's injection point even though it would be selected for direct injection sites.

Confirmed via `System.identityHashCode()` tracing: both the test's `@Inject CbrCaseMemoryStore` and the bridge's `@Inject CbrCaseMemoryStore delegate` resolve to the same `NoOpCbrCaseMemoryStore_ClientProxy` instance.

### Solution 1 — Bypass the reactive bridge

**Approach:** Inject the blocking interface directly and offload to a worker thread.

**Pros:** No CDI resolution ambiguity. Works with any test profile.

**Cons:** Caller must handle thread offloading explicitly.

```java
@ApplicationScoped
public class MyService {
    @Inject CbrCaseMemoryStore cbrStore; // blocking, not reactive

    public Uni<List<Result>> retrieve(Query query) {
        return Uni.createFrom()
            .item(() -> cbrStore.retrieveSimilar(query, MyCase.class))
            .runSubscriptionOn(Infrastructure.getDefaultWorkerPool())
            .map(this::mapResults);
    }
}
```

### Solution 2 — Use QuarkusTestProfile.getEnabledAlternatives()

**Approach:** Override alternatives programmatically in the test profile.

**Pros:** Properly activates @Alternative beans for all injection points including bridge delegates.

**Cons:** `getEnabledAlternatives()` REPLACES (not appends to) `quarkus.arc.selected-alternatives` — ALL alternatives from the base profile must be re-declared. Missing one breaks other tests sharing the profile.

```java
public static class MemoryProfile implements QuarkusTestProfile {
    @Override
    public String getConfigProfile() { return "memory"; }

    @Override
    public Set<Class<?>> getEnabledAlternatives() {
        return Set.of(
            InMemoryCaseMetaModelRepository.class,
            InMemoryReactiveCaseMetaModelRepository.class,
            // ... ALL other memory profile alternatives ...
            InMemoryCbrCaseMemoryStore.class);
    }
}
```

### Why this is non-obvious

The bridge pattern (`@DefaultBean` reactive wrapper delegating to `@DefaultBean` blocking implementation, replaced by `@Alternative` at test time) is a standard Quarkus CDI pattern. Nothing in Quarkus documentation warns that `@DefaultBean` injection points resolve before `@Alternative` activation. The failure is completely silent — no ambiguity error, no warning, just empty results from the no-op. Only `System.identityHashCode()` tracing reveals which implementation was actually injected.

### Why this fix

Solution 1 (bypass the bridge) is preferred because it eliminates the CDI resolution issue entirely and has no test infrastructure side effects. Solution 2 works but creates a maintenance burden — every new alternative added to the memory profile must also be added to every `getEnabledAlternatives()` override.

### Alternatives considered

- `quarkus.arc.exclude-types` for the no-op: breaks CDI wiring when the bridge needs a delegate and no alternative is available
- Using `Instance<CbrCaseMemoryStore>` with programmatic lookup: adds complexity without addressing the root cause
- Filing a Quarkus ARC bug: may be working as designed (build-time CDI optimisation)
