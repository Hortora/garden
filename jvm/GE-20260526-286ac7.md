## Context

Testing a `@RegisterRestClient` interface against a WireMock server in a `@QuarkusTest`.
The REST client URL must be configured before Quarkus starts — the test fails with
`ConnectionRefusedException` if WireMock starts after the REST client is configured.

## The problem with alternatives

**Fixed port (`WireMockServer(9090)`):** Works locally, fails in CI when port 9090 is
already bound (other services, parallel builds, Kubernetes sidecars).

**`@TestProfile` with fixed port:** Same problem — the port is baked into the profile
at compile time.

**`@BeforeAll` / `@BeforeEach`:** WireMock starts after Quarkus, so the REST client
is configured with a stale URL.

## Technique

Implement `QuarkusTestResourceLifecycleManager`. Its `start()` method is guaranteed
to run **before Quarkus starts**, so config overrides returned from `start()` are
visible to the REST client URL resolution:

```java
public class WireMockResource implements QuarkusTestResourceLifecycleManager {

    static volatile WireMockServer INSTANCE;  // volatile: start() and test threads may differ
    private WireMockServer server;

    @Override
    public Map<String, String> start() {
        server = new WireMockServer(WireMockConfiguration.options().dynamicPort());
        server.start();
        INSTANCE = server;
        return Map.of(
            "quarkus.rest-client.my-client.url",
            "http://localhost:" + server.port()
        );
    }

    @Override
    public void stop() {
        if (server != null) server.stop();
    }
}
```

In the test class:

```java
@QuarkusTest
@QuarkusTestResource(WireMockResource.class)
class MyClientIT {

    @BeforeEach
    void reset() {
        WireMockResource.INSTANCE.resetAll();
    }

    @Inject
    MyService service;  // injects the @RegisterRestClient-backed client transitively

    @Test
    void callsCorrectEndpoint() {
        WireMockResource.INSTANCE.stubFor(
            post(urlEqualTo("/api/endpoint")).willReturn(aResponse().withStatus(200)));
        service.doSomething();
        WireMockResource.INSTANCE.verify(postRequestedFor(urlEqualTo("/api/endpoint")));
    }
}
```

## Why this is non-obvious

`@QuarkusTest` lifecycle ordering is not prominent in the documentation. Developers
naturally reach for `@BeforeAll` (WireMock starts after Quarkus — wrong order) or
fixed ports (fragile in CI). The insight — that `QuarkusTestResourceLifecycleManager`
is the hook designed specifically for "start something before Quarkus starts" — is
buried in the Quarkus testing documentation. Using `dynamicPort()` and returning the
URL from `start()` as a config override is the clean solution, but the connection
between "I need to inject a URL into REST client config" and "I should use
`QuarkusTestResourceLifecycleManager`" is not obvious.

**Single-class constraint:** if two `@QuarkusTest` classes register the same resource,
`start()` is called again, overwriting `INSTANCE` and invalidating stubs from the first.
Design one resource class per test class when using `static volatile INSTANCE`.
