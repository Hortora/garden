## Symptom

After a TCP probe succeeds, the WebSocket connection hangs indefinitely. The test server never responds to the HTTP upgrade request. No error is produced on either side.

## Context

A `tcpProbe()` method checks whether a port is open by creating a `Socket`, calling `connect()`, then closing it (try-with-resources). This is correct for a production SC2 process. In tests, the FakeSC2Server calls `ServerSocket.accept()` once and waits.

```java
// tcpProbe in transport
try (Socket probe = new Socket()) {
    probe.connect(new InetSocketAddress("127.0.0.1", port), 200);
}
// falls through: port is open
```

```java
// FakeSC2Server — the bug
void acceptAsync() {
    Thread.ofVirtual().start(() -> {
        Socket c = server.accept();  // fires on the PROBE connection
        doHandshake(c);              // probe is already closed — EOF
        serveFrames(c);
    });
}
```

## Root Cause

The TCP probe performs a full TCP handshake (SYN/SYN-ACK/ACK) which triggers `server.accept()` to return. The probe then closes the connection (FIN). The `doHandshake()` reads EOF. The test server thread exits. When the real WebSocket connection arrives, there is no thread left to `accept()` it — it sits in the OS backlog forever.

## Fix

The FakeSC2Server must loop on `accept()`. Probe connections fail during `doHandshake()` (IOException on EOF) and are silently ignored; the loop immediately calls `accept()` again for the real connection:

```java
void acceptAsync() {
    Thread.ofVirtual().start(() -> {
        try {
            while (!server.isClosed()) {
                Socket c = server.accept();
                Thread.ofVirtual().start(() -> {
                    try {
                        doHandshake(c);
                        client = c;
                        serveFrames(c);
                    } catch (Exception ignored) {
                        try { c.close(); } catch (Exception ignored2) {}
                    }
                });
            }
        } catch (Exception ignored) {}
    });
}
```

## Why Non-Obvious

The TCP probe pattern is standard for checking port availability before connecting. It never causes issues against a production server (which loops on `accept()` by design). The failure only manifests in tests with a minimal single-accept server — and produces no error, just an indefinite hang that looks like a WebSocket handshake timeout.
