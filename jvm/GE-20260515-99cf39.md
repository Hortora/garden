### The technique

When a module ships multiple implementations of an interface (e.g. `NoOpStrategy`, `TopLevelStrategy`, `JsonPatchStrategy`) and needs to select one via config while still allowing consumers to override entirely, use a config-driven `@Produces @DefaultBean` producer instead of CDI annotation games (`@Alternative`, `@Priority`, `@DefaultBean` on individual classes).

### Pattern

```java
@ApplicationScoped
class StrategyProducer {

    @ConfigProperty(name = "mymodule.strategy", defaultValue = "none")
    String strategy;   // package-private for direct test access

    @Produces
    @DefaultBean
    @ApplicationScoped
    MyStrategy produce() {
        return switch (strategy) {
            case "none"       -> new NoOpStrategy();
            case "top-level"  -> new TopLevelStrategy();
            case "json-patch" -> new JsonPatchStrategy();
            default -> throw new IllegalStateException(
                "Unknown mymodule.strategy: '" + strategy +
                "'. Valid values: none, top-level, json-patch");
        };
    }
}
```

The three strategy classes carry **no CDI annotations** â€” they are plain POJOs instantiated by the producer.

### Key properties

- **Consumer override:** because the produced bean is `@DefaultBean`, any consumer-provided `@ApplicationScoped MyStrategy` wins automatically with no configuration.
- **Config selection:** `@ConfigProperty` with a `defaultValue` means the producer is always safe to instantiate even when the consumer has overridden the bean (the `produce()` method is never called in that case).
- **Fail-fast:** unknown config values throw `IllegalStateException` at boot â€” include the config key name and bad value in the message.
- **Testability:** the `strategy` field is package-private, allowing plain JUnit tests to set it directly without Quarkus or Mockito.

### When to use

Use this pattern when: a library or platform module ships several bundled implementations of an interface; only one can be active at a time; the choice belongs to config (not classpath composition); and external consumers should be able to replace the whole thing with their own implementation.

Do **not** use this for SPIs that consumers are expected to extend â€” use `@DefaultBean` directly on the no-op class for those (the standard consumer-replaceable SPI pattern).

### Why non-obvious

The instinct is to use `@Alternative` / `selected-alternatives` or `@Priority` on the individual strategy classes. Those approaches either require explicit config per alternative, create CDI ambiguity between bundled implementations, or make all implementations always-on. The producer pattern cleanly separates "which one" (config) from "how CDI wires it" (producer), and the strategy classes stay as plain testable POJOs.
