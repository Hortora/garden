## Symptom

Running `mvn test -f submodule/pom.xml` throws `NoSuchMethodError` that looks like a
runtime SNAPSHOT API mismatch:

```
java.lang.NoSuchMethodError: 'void io.example.SomeService.doThing(java.util.UUID, java.lang.String)'
```

The error message shows the JVM expected a `void` return type, but the current API
returns `CompletionStage<Void>`. Investigation focuses on the calling code and the
runtime jar — both look correct. Hours pass.

## Context

Another sibling module provides the API (e.g. `casehub-claudony-casehub`). A recent
change updated a method's return type in that sibling module. Running tests on only
the downstream app module (`-f app/pom.xml` or `-pl app`) doesn't trigger recompilation
of the sibling — Maven uses whatever `.class` files are in the sibling's `target/`.

## What was tried

- Checking the runtime jar's API — confirms the NEW return type
- Inspecting the calling code — call sites look correct for the new API
- Comparing error message direction — "JVM looking for void" implies runtime has
  CompletionStage (WRONG read — actually compiled code has void)

## Root cause

The stale `.class` files in the sibling module still reference the OLD method descriptor
(`void doThing(...)`). At runtime, the JVM loads the sibling's classes from
`target/classes/` directly (not from the runtime jar). The compiled downstream code
references `CompletionStage<Void> doThing(...)` but the sibling's stale bytecode only
has `void doThing()`. The `NoSuchMethodError` message shows what the CALLER expected,
not what the runtime library provides — inverting the natural mental model.

## Fix

Recompile ALL modules from the root pom before running sub-module tests:

```bash
mvn clean compile -f root/pom.xml   # or just: mvn compile -f root/pom.xml
mvn test -f submodule/pom.xml
```

To confirm stale bytecode is the issue:

```bash
javap -p -verbose target/classes/io/example/CallerClass.class | grep "doThing"
# Look for: MethodDescriptor shows old return type in the constant pool
```

## Why non-obvious

The `NoSuchMethodError` message shows the descriptor the CALLER expected (`void`), not
what the runtime provides. Developers read this as "runtime has the wrong API" and search
the runtime jar — which actually has the correct new API. The real problem is in the
sibling module's compiled output, which no one thinks to check after running a
sub-module-scoped test command.

Additionally, the Maven sub-module invocation (`-f submodule/pom.xml`) is widely used
for fast feedback loops during development — it works correctly until a sibling API
changes, at which point it silently produces incorrect results.
