## Symptom

`mvn test` passes. `mvn install -DskipTests` fails during Quarkus augmentation:

```
DeploymentException: UnsatisfiedResolutionException: Unsatisfied dependency for type
io.casehub.ledger.repository.CaseLedgerEntryRepository and qualifiers [@Default]
  - injection target: io.casehub.ledger.service.WorkerDecisionEventCapture#ledgerRepo
```

The failing bean is not in the production codebase — it comes from a transitive dependency. Tests pass because the test profile excludes it.

## Context

A bean was added to `%test.quarkus.arc.exclude-types` in `src/test/resources/application.properties` to prevent CDI deployment failures in `@QuarkusTest`. The assumption: if it's excluded in tests, the test build is fine, and production won't need it.

But `mvn install` (or any non-test Quarkus build goal) uses the production `application.properties`, not the test profile. The `%test.*` prefix is only active during `@QuarkusTest` runs. During `mvn install -DskipTests`, Quarkus augmentation sees the full CDI graph — including the un-excluded bean — and fails.

Hit in practice: `WorkerDecisionEventCapture` (from `casehub-ledger`) injects `CaseLedgerEntryRepository`, which isn't available in Claudony's CDI context. It was in `%test.quarkus.arc.exclude-types` but not in `quarkus.arc.exclude-types`.

## Root Cause

`quarkus.arc.exclude-types` (in `src/main/resources/application.properties`) and
`%test.quarkus.arc.exclude-types` (in `src/test/resources/application.properties`) are
completely independent. The test-profile variant applies only during test augmentation.
The production variant applies during all non-test build phases.

A bean that only needs to be excluded in tests (e.g. because its dependency is not
available in the test context) must still be excluded in production if its dependency
is also absent in production.

## Fix

Add the bean to BOTH exclusion lists:

```properties
# src/main/resources/application.properties
quarkus.arc.exclude-types=...,io.casehub.ledger.service.WorkerDecisionEventCapture

# src/test/resources/application.properties
%test.quarkus.arc.exclude-types=...,io.casehub.ledger.service.WorkerDecisionEventCapture
```

If the bean is only absent in the test context (not in production), only the test exclusion is needed. If it's absent in both, both are needed.

## Why This Is Non-Obvious

The test-profile naming convention (`%test.*`) suggests the setting applies to the test lifecycle. Most Quarkus config properties work this way — `%test.quarkus.http.port` is only active in tests. `quarkus.arc.exclude-types` follows the same pattern.

The surprise: augmentation runs at build time for both test and production phases, but each phase reads different config. A developer who adds a bean to `%test.quarkus.arc.exclude-types` and sees tests pass will not expect a failure when running `mvn install`. The `mvn install` failure also occurs long after the test phase — the CDI augmentation error appears during the `package` or `install` phase's Quarkus build step, which can be easy to overlook if tests are the focus.
