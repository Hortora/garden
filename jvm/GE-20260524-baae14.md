## Maven BOM scope inheritance: omitting scope in child pom does NOT override parent BOM's test scope

### Symptom

Child module declares a dependency without a `<scope>` element (expecting to default to compile). Compilation fails with "package does not exist" — the dependency is on the classpath only in test scope, not compile scope.

### Context

Multi-module Maven project with a parent `pom.xml` containing a `<dependencyManagement>` entry that sets `<scope>test</scope>` for the dependency. Child module wants compile-scope access to the same artifact.

### What was tried (didn't work)

Removing the `<scope>test</scope>` from the child module's dependency declaration:

```xml
<!-- Child pom — scope removed, expecting compile default -->
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-engine-persistence-memory</artifactId>
  <!-- no scope element here -->
</dependency>
```

The parent BOM still wins. The dependency remains test-scoped.

### Root cause

In Maven, `<dependencyManagement>` sets the DEFAULT scope for a dependency. When the child declares the dep without an explicit scope, it INHERITS the managed scope (test). Omitting `<scope>` in the child does not reset it to compile — it uses whatever the BOM specifies.

### Fix

Explicitly override the scope in the child module's dependency declaration:

```xml
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-engine-persistence-memory</artifactId>
  <scope>compile</scope>  <!-- explicit override of BOM's test scope -->
</dependency>
```

### Why this is non-obvious

The intuition is that omitting `<scope>` = "use the default" = compile scope. In the presence of a BOM, the actual behaviour is "use the managed scope", which may be test. The override requires an explicit `<scope>compile</scope>` even though compile is normally the default.
