### Symptom

A subscriber added to a `CopyOnWriteArrayList` value in a `ConcurrentHashMap` is silently orphaned: it never receives events despite being registered. The map entry is absent even though the subscriber was added after the last removal.

### Context

A concurrent subscriber tracking map of the form `ConcurrentHashMap<K, CopyOnWriteArrayList<V>>`. On subscriber removal, the code prunes the map entry when the list becomes empty:

```java
list.remove(emitter);
if (list.isEmpty()) map.remove(key, list);
```

This is a common pattern for maintaining a per-key subscriber list with automatic cleanup.

### Root Cause

`ConcurrentHashMap.remove(key, value)` checks `value.equals(currentValue)` — not reference equality. For `CopyOnWriteArrayList`, `list.equals(list)` is always `true` (reflexive equality), regardless of the list's contents at that moment.

Between `list.isEmpty()` returning `true` and `map.remove(key, list)` executing, a concurrent `subscribe()` call can add to the **same list object**. The `remove()` call still succeeds because `list.equals(list)` is reflexive — even though the list now contains the new subscriber. The map entry is evicted, orphaning the new subscriber.

The race window is small but real on any system with concurrent SSE connections to the same channel.

### Fix

Use `computeIfPresent` returning `null` to atomically collapse the check and the removal under the map's per-bucket lock:

```java
// WRONG — TOCTOU between isEmpty() and remove()
list.remove(emitter);
if (list.isEmpty()) map.remove(key, list);

// CORRECT — atomic under ConcurrentHashMap segment lock
map.computeIfPresent(key, (k, list) -> {
    list.remove(emitter);
    return list.isEmpty() ? null : list;
});
```

The `computeIfPresent` lambda runs atomically. Returning `null` removes the map entry in the same operation. No window exists for a concurrent add between the empty-check and the removal.

### Why This Is Non-Obvious

`ConcurrentHashMap.remove(key, value)` looks like a CAS-safe conditional remove. The Javadoc says "removes the entry for the specified key only if it is currently mapped to the specified value" — which implies the check and the removal are atomic. They are, but the check uses `equals()`, not reference equality. For mutable collections, `equals()` on the same object is always reflexive-true — so the "if it is currently mapped to the specified value" condition is always satisfied regardless of the collection's current state.
