## Symptom

Downstream repo CI fails with `Could not resolve X:Y:0.2-SNAPSHOT` even though the upstream repo's CI run shows `conclusion: success` seconds earlier. Re-triggering the failed downstream CI job manually (with no code changes) fixes it.

## Context

In a multi-repo Maven ecosystem where each repo has its own GitHub Actions CI workflow that publishes to GitHub Packages on push to `main`, all repos can trigger CI simultaneously when several repos are pushed at once. The upstream CI shows as "completed / success" in the GitHub API before it has actually finished publishing the artifacts.

## Root Cause

`conclusion: success` in the GitHub Actions API (and `gh run list`) reflects the workflow run status at creation/queuing time, not artifact publish completion. The downstream CI starts seconds later, before Maven artifacts are available in GitHub Packages. The Maven resolver sees a 404 and fails.

## Fix

**Option 1 — Re-trigger** the failed downstream CI job after a few minutes. No code change needed. The upstream artifacts will be available by then.

**Option 2 — Empty commit trigger** to force a re-run:
```bash
git commit --allow-empty -m "ci: retrigger after upstream publish"
git push
```

**Option 3 — Stagger pushes** by at least 5–10 minutes when pushing multiple repos simultaneously, in dependency order: upstream first, downstream last.

## Why This Is Non-Obvious

`conclusion: success` in the GitHub API looks like "CI passed and published", but it means "the workflow run completed without error" — the publish step may still be running or recently completed with no propagation delay guarantee. The symptom (missing artifact) looks like a version mismatch or configuration error, not a timing race.

### Why this fix

Re-triggering is always safe because Maven will find the artifact once it's published. The empty commit approach is cleaner than a dummy file touch when CI is the only goal.

### Alternatives considered

- Adding explicit `sleep` between repo pushes — fragile and hard to calibrate
- Adding cross-repo workflow triggers — complex, requires PAT with `workflow` scope, coupling
- Pinning SNAPSHOT timestamps — defeats the purpose of SNAPSHOT resolution
