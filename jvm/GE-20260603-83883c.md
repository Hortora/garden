## Symptom

You install a log handler in `@QuarkusTestResource.start()` to capture log output from a
startup observer (`@Observes StartupEvent`). Both approaches are tried:

```java
java.util.logging.Logger.getLogger(MyBean.class.getName()).addHandler(handler);
org.jboss.logmanager.Logger.getLogger(MyBean.class.getName()).addHandler(handler);
```

The handler's capture list is empty after tests run — the warning was emitted (visible
in console output) but not captured by the handler.

## Context

Occurs when trying to assert that a CDI `@ApplicationScoped` bean logged a WARNING at
startup (during `StartupEvent` observation). The test uses `@QuarkusTest` with
`@QuarkusTestResource` and `restrictToAnnotatedClass = true`.

## Root cause

JBoss LogManager's logger registry is re-initialised during Quarkus augmentation and
test bootstrap *after* `QuarkusTestResourceLifecycleManager.start()` returns. Handlers
installed pre-bootstrap are attached to stale logger instances that are discarded when
the LogManager rebuilds its tree for the Quarkus application. By the time the
`StartupEvent` observer fires, the handler chain on the live logger is empty.

No error or warning is raised — the handler silently receives nothing.

## Fix / Workaround

Do not try to capture startup log output via handler installation. Instead, decompose
the verification into three parts that work reliably:

1. **CDI wiring** — inject the `@ApplicationScoped` bean and assert it is non-null.
2. **Config resolution** — inject `LedgerConfig` (or equivalent) and assert the values
   that would trigger the warning are in the expected state.
3. **Logic composition** — compose the injected config values with the warning-condition
   method (unit-tested separately via a pure static call) to prove the warning would fire.

```java
@QuarkusTest
@TestProfile(MyIT.Profile.class)
class MyIT {

    @Inject MyStartupValidator validator;
    @Inject MyConfig config;

    @Test
    void validator_isInCdiGraph() {
        assertThat(validator).isNotNull();
    }

    @Test
    void config_stateTriggersWarn() {
        // shouldWarn(true, 0) == true proven in unit test; this proves config resolves correctly
        assertThat(config.feature().enabled()).isTrue();
        assertThat(config.feature().actors().orElse(List.of())).isEmpty();
    }
}
```

The unit test for the condition logic covers `shouldWarn()` directly without CDI.
The IT covers CDI wiring and config resolution.

## Why this is non-obvious

Both the JUL API and the JBoss LogManager API *appear* to work — handlers are installed
without error and the logger instance is returned. There is no indication that the logger
instance will be discarded. The warning IS emitted (you can see it in console output),
so the bean is clearly active. Only the handler capture is silently broken.

The natural intuition is "install the handler before the app starts → catch startup logs".
This is how it works in plain JUnit with `java.util.logging`. Quarkus's bootstrap lifecycle
breaks this assumption silently.
