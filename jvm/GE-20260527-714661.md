## Symptom

A `@QuarkusTest` integration test for a PostgreSQL LISTEN/NOTIFY broadcaster
fails intermittently with `ConditionTimeoutException` — the awaitility condition
times out even though the test logic is correct and passing in most CI runs. The
same commit SHA shows both pass and fail on successive runs.

## Context

`PostgresWorkItemEventBroadcaster.startListening()` is annotated `@PostConstruct`.
It issues a `LISTEN <channel>` command via the Vert.x reactive PG client — an async
call that returns immediately; the LISTEN is established on the Vert.x event loop
after control returns to the caller. Quarkus marks the application ready after all
`@PostConstruct` calls return, but does not wait for their async work to complete.

Tests that subscribe to a filtered stream and immediately fire events may do so
before the LISTEN is registered with PostgreSQL. Events published via `pg_notify`
reach the subscriber connection only after LISTEN is active — events fired before
that are silently dropped.

## What was tried (didn't work)

Increasing the awaitility timeout from 5 seconds to 10 seconds: reduced flake
frequency but did not eliminate it (the root cause is the LISTEN not being active,
not the notify→receive path being slow).

## Fix — Warm-up probe

Before subscribing to the filtered stream, subscribe to the **unfiltered** stream and
wait for a known event to arrive. This confirms the LISTEN channel is established and
the full async path (pg_notify → subscriber connection → BroadcastProcessor) is working:

```java
// 1. Subscribe to unfiltered stream BEFORE creating any items
final List<WorkItemLifecycleEvent> warmup = new CopyOnWriteArrayList<>();
broadcaster.stream(null, null).subscribe().with(warmup::add);

// 2. Create the target item and wait for its CREATED event to arrive
final WorkItem target = createWorkItem("Target item");
Awaitility.await().atMost(Duration.ofSeconds(10))
        .until(() -> warmup.stream().anyMatch(e -> target.id.equals(e.workItemId())));

// 3. LISTEN channel confirmed active — now subscribe to the filtered stream
final List<WorkItemLifecycleEvent> received = new CopyOnWriteArrayList<>();
broadcaster.stream(target.id, null).subscribe().with(received::add);

// 4. Proceed with the test (noise items, claims, assertions)
```

The `BroadcastProcessor` is a hot stream — events emitted before subscription are
lost. Subscribing before creating any items ensures the warm-up probe captures the
CREATED event and the subsequent filtered subscription captures all later events.

## Why this is non-obvious

The application appears fully ready (HTTP is serving, CDI is wired) when the tests
start. The `@PostConstruct` has returned. There is no visible gap — but the LISTEN
command is in flight on the Vert.x event loop and may not have been acknowledged by
PostgreSQL yet. The failure only appears when the first `pg_notify` fires before that
acknowledgement arrives, which depends on container startup speed and event loop
scheduling in CI.

Bumping the awaitility timeout is the instinctive fix but addresses the wrong problem.
The test doesn't need more time to wait for the result — it needs to confirm the
infrastructure is ready before generating the result.
