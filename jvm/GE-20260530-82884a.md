### What it is

When using an abstract caching base class (with a `loadContext(key)` template method), the
template method can only receive a cache key — not additional context needed for computation
(e.g. a `LedgerEntry` that holds the public key, actor DID, and other fields). The standard
fix of extending the class and implementing `loadContext` breaks down when the computation
requires the full object.

### The technique

1. Add a `put(key, value)` method to the abstract base (exposes the underlying `ConcurrentHashMap`
   put, respecting TTL semantics).
2. Instead of extending the base and implementing `loadContext`, **compose** it as a field.
3. Override `loadContext` in the anonymous subclass to return `Optional.empty()` (never called
   externally — a no-op sentinel).
4. In the owning class, check the cache via `cache.get(key)`. On miss, compute externally
   using the full context, then call `cache.put(key, result)`.

```java
// In the abstract base:
public final void put(String key, Optional<C> value) {
    cache.put(key, new CacheEntry<>(value, now().plus(ttl)));
}

// In the owning class:
private final AbstractCachingIdentityProvider<IdentityBindingStatus> cache =
    new AbstractCachingIdentityProvider<>(Duration.ofMinutes(5)) {
        @Override
        protected Optional<IdentityBindingStatus> loadContext(String key) {
            return Optional.empty(); // never called — enrich() drives puts
        }
    };

void enrich(LedgerEntry entry) {
    Optional<IdentityBindingStatus> cached = cache.get(entry.actorId);
    if (cached.isEmpty()) {
        IdentityBindingStatus status = computeStatus(entry); // needs full entry
        cache.put(entry.actorId, Optional.of(status));
    }
}
```

### Why a skilled developer wouldn't reach for this

The natural instinct is to subclass the abstract class and implement `loadContext`, then
restructure the method signature to pass extra context. Or to abandon the abstraction and
inline the caching logic. Composition-with-no-op-sentinel lets you preserve the full
TTL/eviction/invalidation logic from the base while externalizing the computation.

### When to use

When: (a) you have a well-tested caching base with TTL and eviction, (b) the computation
requires context beyond what `loadContext(String key)` can receive, and (c) you don't want
to change the base class interface.
