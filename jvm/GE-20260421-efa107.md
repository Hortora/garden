## Symptom

`-Dexcludes="**/SomeQuarkusTest.java"` is passed to Maven but the test still produces a class-loader failure and BUILD FAILURE. The exclusion appears to have no effect.

## Context

A `@QuarkusTest` class requires Testcontainers (Docker). Docker is not available locally. The developer adds `-Dexcludes="**/SignalDedupExtendedTest.java"` hoping to skip the test. The build still fails.

## Root Cause

Surefire's `excludes` filter runs after JUnit's test discovery phase. When a `@QuarkusTest` class fails to class-load (because Quarkus cannot start without Docker), the failure occurs during JUnit's `ClassSelector` resolution — before Surefire has applied its exclusions. The exclusion filter never sees the class.

## Fix

Use a Maven profile that controls which classes are compiled into the test classpath, not which tests Surefire skips at runtime:

```bash
# Profile that activates in-memory persistence (no Docker needed)
TESTCONTAINERS_RYUK_DISABLED=true mvn test -pl engine -Ppersistence-memory \
  -Dexcludes="**/SignalTest.java"   # only exclude the one with true class-loader issue
```

The Maven profile excludes the Docker-dependent test classes from the build entirely, so they never reach JUnit's discovery phase.

## Why This Is Non-Obvious

`-Dexcludes` works correctly for tests that compile and load but should not execute. It does not work for tests that fail before they can be discovered. The error message (`ClassSelector resolution failed`) looks like a Quarkus bug, not a Surefire scoping issue.
