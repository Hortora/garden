### Symptom

`CachedTrustScoreSource` using colon (`:`) as a key separator between actorId and capabilityKey produces false matches in `startsWith()` prefix scans. The actorId format `{model-family}:{persona}@{major}` (e.g., `claude:tarkus-reviewer@v1`) contains colons.

### Context

Building an in-memory cache with `ConcurrentHashMap<String, CachedCapabilityScore>` keyed by `actorId:capabilityKey`. Batch methods like `allCapabilityScores(actorId)` scan with `startsWith(actorId + ":")` to find all capability entries for one actor.

### Root cause

An actorId like `claude:tarkus-reviewer@v1` with separator `:` produces key `claude:tarkus-reviewer@v1:review`. But `allCapabilityScores("claude")` scans with prefix `claude:` — which matches entries for ALL claude agents, not just the one with actorId `"claude"`. The colon in the actorId format and the colon separator are indistinguishable.

### Fix

Use null char (`\0`) as the separator — cannot appear in actorId or capabilityTag values:

```java
// Index
capabilityScores.put(s.actorId + "\0" + s.capabilityKey,
        new CachedCapabilityScore(s.trustScore, s.decisionCount));

// Lookup
capabilityScores.get(actorId + "\0" + capabilityTag);

// Prefix scan
String prefix = actorId + "\0";
capabilityScores.entrySet().stream()
    .filter(e -> e.getKey().startsWith(prefix))
    .collect(...);
```

### Why this is non-obvious

Colon is the natural separator choice — it's readable in debuggers and logs. The actorId format convention (`{model-family}:{persona}@{major}`) is documented in CLAUDE.md but not in the context where cache keys are being designed. The false match is silent — no error, just wrong scores returned for some actors. Only triggered when multiple agents share the same model family prefix.
