## Symptom

Adding a unit test for a service to `runtime/src/test/java/` fails to compile:

```
[ERROR] COMPILATION ERROR:
package io.quarkiverse.qhorus.testing does not exist
```

The test tries to import `InMemoryCommitmentStore` from the `testing/` module, but that
import is unavailable even though `testing` is listed as a test dependency.

## Context

Multi-module Maven project where:
- `runtime/` — the main module
- `testing/` — `@Alternative @Priority(1)` in-memory stores for consumer unit tests
- `testing/pom.xml` has `<dependency>runtime</dependency>` (testing depends on runtime)

A service unit test in `runtime/src/test/` needs to inject `InMemoryCommitmentStore`
from `testing/` to avoid CDI/database setup. Adding `testing` as a `test` scope
dependency to `runtime/pom.xml` creates a circular dependency:
`runtime → testing → runtime`.

## Root Cause

Maven circular dependency: `testing` already depends on `runtime` for the store
interfaces it implements. Adding `testing` as a test dependency of `runtime`
creates a cycle that Maven refuses to build.

## Fix

Place service unit tests in `testing/src/test/java/` instead of `runtime/src/test/java/`.
The `testing` module has both `runtime` (for the interfaces) and the in-memory
implementations on its classpath. Wire the store directly via field access:

```java
// In testing/src/test/java/.../CommitmentServiceTest.java
class CommitmentServiceTest {
    private final InMemoryCommitmentStore store = new InMemoryCommitmentStore();
    private final CommitmentService service = new CommitmentService();

    @BeforeEach
    void setup() {
        service.store = store;  // package-private field injection — no CDI needed
        store.clear();
    }
}
```

The service's `@Inject CommitmentStore store` field must be package-private (not private)
for this to work. `@Inject` on a package-private field is valid in CDI.

## Why Non-Obvious

The natural location for service unit tests is the module that contains the service.
The circular dependency only becomes apparent when you try to add the testing module
as a dependency — Maven's error message says "cycle" but doesn't suggest the workaround
of placing the test in the dependent module instead.
