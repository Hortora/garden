## Uni.createFrom().runnable() does not exist in Mutiny — use Uni.createFrom().<Void>item(Supplier) instead

**ID:** GE-20260609-630b35
**Stack:** Mutiny 2.x, Quarkus 3.x

### Symptom

Compilation error:

```
cannot find symbol
  method runnable(()->{ ... })
  location: class io.smallrye.mutiny.groups.UniCreate
```

Occurs when trying to wrap a `Runnable` lambda in a `Uni<Void>` using `Uni.createFrom().runnable(...)`.

### Context

Implementing a reactive `Uni<Void> terminate(String workerId)` method in a reactive SPI
implementation, following examples or documentation that reference `runnable()` on `UniCreate`.

Reactor (Project Reactor), Mutiny's reactive counterpart, does have similar patterns, and some
Mutiny examples in blog posts or older documentation describe `runnable()`. The method does not
exist on `io.smallrye.mutiny.groups.UniCreate`.

### What was tried

```java
// Does NOT compile — runnable() does not exist
return Uni.createFrom().runnable(() -> {
    registry.deregister(workerId);
    log.info("...");
});
```

### Fix

Use `Uni.createFrom().<Void>item(Supplier)` returning `null`:

```java
return Uni.createFrom().<Void>item(() -> {
    registry.deregister(workerId);
    log.info("...");
    return null;
});
```

Or, if the operation is truly safe to call on the subscribing thread (e.g. in-memory
`ConcurrentHashMap` operations that don't block), call it synchronously and return `voidItem()`:

```java
registry.deregister(workerId);
log.info("...");
return Uni.createFrom().voidItem();
```

The `Supplier` form is preferred when you want to defer execution until subscription and
propagate any thrown exceptions as `Uni` failures.

### Why this is non-obvious

`runnable()` is a natural name for "execute this Runnable and emit void." It appears in some
Mutiny documentation and blog posts, and is a familiar pattern from Reactor. `UniCreate` has
`item()`, `item(Supplier)`, `voidItem()`, `nullItem()`, `failure()`, `emitter()`, and others —
but not `runnable()`. The `<Void>` type witness on `item(Supplier)` is required because Java
cannot infer `Uni<Void>` from a lambda returning `null` without it.
