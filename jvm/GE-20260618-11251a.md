## Context

Implementing a JAX-RS endpoint that accepts structured CloudEvents (`Content-Type: application/cloudevents+json`) and auto-binds the body to a `CloudEvent` parameter.

## What Developers Normally Try

Declare a `CloudEvent event` parameter in the JAX-RS method and expect Quarkus REST's Jackson integration to handle deserialization:

```java
@POST
@Consumes("application/cloudevents+json")
public Response receive(CloudEvent event) { ... }  // BROKEN
```

## Why That Fails

Two compounding problems:

1. `io.cloudevents.CloudEvent` is an interface. Jackson cannot deserialize to an interface without a registered `@JsonDeserialize` annotation or a custom deserializer registered on the `ObjectMapper`.
2. Even with `cloudevents-json-jackson` on the classpath (which registers a Jackson `JsonDeserializer<CloudEvent>` via Jackson's `Module` SPI), Quarkus REST's Jackson `MessageBodyReader` is registered for `application/json`, not `application/cloudevents+json`. Quarkus REST will not pick it up for the `application/cloudevents+json` content type — the media type matching fails before deserialization is even attempted.

## The Correct Approach

Accept `byte[]` (a basic JAX-RS type that works with any content type) and deserialize manually using the CloudEvents SDK format API:

```java
@Startup
@ApplicationScoped
@Path("/webhook")
public class WebhookResource {

    private EventFormat eventFormat;

    @PostConstruct
    void init() {
        // cloudevents-json-jackson registers JsonFormat via ServiceLoader
        // (META-INF/services/io.cloudevents.core.format.EventFormat)
        eventFormat = EventFormatProvider.getInstance()
            .resolveFormat(JsonFormat.CONTENT_TYPE);
        if (eventFormat == null) throw new IllegalStateException(
            "cloudevents-json-jackson missing from classpath");
    }

    @POST
    @Consumes("application/cloudevents+json")  // 415 for wrong Content-Type
    public Response receive(byte[] body) {
        CloudEvent event;
        try {
            event = eventFormat.deserialize(body);
        } catch (RuntimeException e) {
            return Response.status(400).entity(e.getMessage()).build();
        }
        // enrich and fire to CDI
        return Response.accepted().build();
    }
}
```

Required dependencies:
```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-rest-jackson</artifactId>
</dependency>
<dependency>
    <groupId>io.cloudevents</groupId>
    <artifactId>cloudevents-json-jackson</artifactId>  <!-- registers JsonFormat via ServiceLoader -->
</dependency>
```

`@Consumes("application/cloudevents+json")` still works correctly for content-type enforcement (returns 415 for wrong type) — only auto-binding to a `CloudEvent` parameter fails.

`@Startup @ApplicationScoped` is required so `@PostConstruct` runs at startup (not on first request), ensuring `eventFormat` is ready and the null check fails fast on classpath misconfiguration.

## Why This Is Non-Obvious

The pattern of accepting `byte[]` and deserializing manually contradicts the normal JAX-RS expectation that the framework handles deserialization based on content type and Java type. The Jackson module distinction (Jackson `JsonDeserializer` vs JAX-RS `MessageBodyReader`) is not visible to application developers. The CloudEvents SDK's `EventFormatProvider` + ServiceLoader pattern is not mentioned in the Quarkus REST documentation.
