### Symptom

Under concurrent writes, a list value in a thread-safe store ends up with fewer entries than expected — some updates are silently lost. The store itself is thread-safe but the caller's update is not.

### Context

Any pattern where a caller (a) reads a mutable value from a thread-safe store, (b) modifies it outside the store, then (c) writes it back. Common with lists that are stored as values in a `ConcurrentHashMap` or similar.

### What was tried (didn't work)

```java
// RACE: two threads both read the same list, modify it, write back — last write wins
public static void recordWorkerCompletion(CaseContextImpl ctx, String workerName) {
    WritablePanelImpl episodic = ctx.writablePanel(ContextPanel.EPISODIC);
    List<Map<String, Object>> workers = episodic.getList("workers", Map.class); // read (lock released)
    // ← another thread reads the same list here ←
    workers = new ArrayList<>(workers);
    // ... modify workers list ...
    episodic.set("workers", workers); // write (lock acquired) — may overwrite concurrent write
}
```

`getList()` acquires and releases the read lock. The modification runs unguarded. `set()` acquires the write lock. Between the read and write, another thread calling the same method reads the same stale list and its write overwrites the first thread's increment.

### Root cause

Thread-safety of individual operations on a store does not make sequences of those operations atomic. The entire read-modify-write sequence must execute under a single lock.

### Fix

Hold the write lock for the full read-modify-write sequence using an atomic update method:

```java
// Add to the store class:
public void engineUpdate(String key, UnaryOperator<Object> updater) {
    lock.writeLock().lock();
    try {
        Object current = data.get(key);
        Object updated = updater.apply(current);
        if (updated != null) data.put(key, updated);
        else data.remove(key);
    } finally {
        lock.writeLock().unlock();
    }
}

// Caller uses the atomic update:
episodic.engineUpdate("workers", currentValue -> {
    List<Map<String, Object>> workers = currentValue instanceof List
        ? new ArrayList<>((List<Map<String, Object>>) currentValue)
        : new ArrayList<>();
    // ... modify workers ...
    return workers;
});
```

### Why this is non-obvious

Developers typically understand that individual operations on `ConcurrentHashMap` are thread-safe and assume this extends to "I read it safely, I write it safely, so the sequence is safe." The race window is invisible in unit tests because tests run sequentially by default. The bug only appears under load — Quartz worker completion callbacks, for example, can arrive concurrently for the same case and trigger simultaneous list updates.
