## Two-phase signing: prepareKey → sign

### What this solves

A signing operation that must run AFTER content hashing (the signature covers
the hash), but enrichers that run BEFORE hashing need the signing key material
(public key, key reference) to validate identity bindings.

Concrete case: `ActorIdentityValidationEnricher` (Priority 50) checks whether
the agent's public key appears in the DID document's verification methods. The
public key is set by the signing step. But signing must happen after hashing.
Circular dependency.

### The technique

Split the signer into two phases:

```java
@ApplicationScoped
public class AgentEntrySigner {

    private final AgentSigner signer;

    /** Phase 1: populate key material for enrichers. No signature yet. */
    public void prepareKey(final LedgerEntry entry) {
        if (entry.actorId == null || entry.agentPublicKey != null) return;
        signer.sign(entry.actorId, new byte[0])
                .ifPresent(sig -> {
                    entry.agentPublicKey = sig.publicKey();
                    entry.agentKeyRef = sig.keyRef();
                });
    }

    /** Phase 2: cryptographic seal over finalised content. */
    public void sign(final LedgerEntry entry) {
        if (entry.actorId == null || entry.agentSignature != null) return;
        signer.sign(entry.actorId, entry.canonicalBytes())
                .ifPresent(sig -> {
                    entry.agentSignature = sig.signature();
                    entry.agentPublicKey = sig.publicKey();
                    entry.agentKeyRef = sig.keyRef();
                });
    }
}
```

Pipeline ordering:
```
1. prepareKey(entry)         — key material available for enrichers
2. enricherPipeline.enrich() — identity enrichers can read agentPublicKey
3. digest = leafHash(entry)  — hash covers enriched content
4. sign(entry)               — signature covers the digest's canonical bytes
```

### Why this is non-obvious

The natural instinct is to keep signing as a single atomic operation. Splitting
it into "prepare key material" and "seal" is counterintuitive — signing is
conceptually one thing. The split only becomes necessary when there's a
dependency chain between enrichers and the signer that creates a cycle in the
pipeline ordering.

The `prepareKey()` implementation signs empty data to extract key material from
the `AgentSigner` SPI — slightly wasteful (one extra Ed25519 operation per save)
but avoids adding a key-only query method to the SPI.
