### Symptom

`NullPointerException` in a `.equals()` comparison where one side is null. The bug report
or issue description says "X.equals(Y) causes NPE when Y is null."

### Context

Defensive null-safety code review on an `InMemoryAgentRegistry.find()` filter:

```java
// Issue description claimed this NPEs when d.slot() is null:
.filter(d -> query.slot() == null || query.slot().equals(d.slot()))
```

The description had the sides reversed. The actual NPE risk was on the **left side**:

```java
// This DOES NPE when c.name() returns null:
.filter(d -> d.capabilities().stream().anyMatch(c -> c.name().equals(query.capabilityName())))
```

### Root cause

Java's `String.equals(Object o)` checks `anObject instanceof String` first. `null` is not
an instance of anything, so the method returns `false` — no NPE. The receiver (the object
`equals()` is called *on*) must be non-null; the argument does not.

```java
"foo".equals(null)   // → false (safe)
null.equals("foo")   // → NullPointerException
```

Bug reports and issue descriptions frequently identify the wrong side — they see a null in
the expression and assume it triggers the NPE, without checking which object is the receiver.

### Fix

Use `Objects.equals(a, b)` — symmetric and null-safe on both sides:

```java
// Before (partially safe, partially risky):
.filter(d -> query.slot() == null || query.slot().equals(d.slot()))
.filter(d -> query.capabilityName() == null
    || d.capabilities().stream().anyMatch(c -> c.name().equals(query.capabilityName())))

// After (fully null-safe, intent explicit):
.filter(d -> query.slot() == null || Objects.equals(d.slot(), query.slot()))
.filter(d -> query.capabilityName() == null
    || d.capabilities().stream().anyMatch(c -> Objects.equals(c.name(), query.capabilityName())))
```

### Why this is non-obvious

Many developers assume null on *either* side of `.equals()` causes NPE — a reasonable
assumption if you haven't read the javadoc carefully. Bug reports frequently identify the
wrong side, sending the reader looking for the wrong fix. The actual NPE only fires when
`.equals()` is **called on** a null reference, not when null is **passed to** it.

### Alternatives considered

- Null guard before the call: `d.slot() != null && d.slot().equals(query.slot())` — verbose
  and requires repeating the left-hand expression
- `Objects.equals()` — canonical fix: symmetric, null-safe, idiomatic Java
