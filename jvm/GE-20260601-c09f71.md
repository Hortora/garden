## Symptom

ARC42STORIES.MD §12 Risks and Technical Debt lists issues as active risks, but a spot check reveals they are closed:
```bash
gh issue view 330 --repo casehubio/engine | head -2
# title: ...
# state: CLOSED
```

The document was generated from `LAYER-LOG.md` entries that were accurate when written but have since been resolved.

## Context

When migrating a CaseHub application's `LAYER-LOG.md` to `ARC42STORIES.MD`, layer entries reference open issues as blockers or tracked risks. The LAYER-LOG is written at layer close time, which may be weeks or months before the ARC42STORIES.MD migration. Issues get resolved in the interim.

In the devtown migration, 6 issues appeared as Active Risks or Technical Debt but were all closed: `engine#330`, `work#211`, `parent#26` (all Technical Debt), `platform#24`, `work#212`, `devtown#31` (Active Risks). The LAYER-LOG appeared authoritative — none of these were obviously stale without checking GitHub.

## What was tried

The issues were initially transcribed directly from LAYER-LOG into §12 without verification. Only after a systematic `gh issue view` sweep were the stale entries discovered.

## Root cause

LAYER-LOG entries are append-only records written at a point in time. There is no mechanism to update them when referenced issues are closed. The migration step that reads LAYER-LOG to populate §12 does not trigger issue-state verification. GitHub is the only authoritative source for issue state.

## Fix

Before finalising ARC42STORIES.MD §12, run for every issue reference:
```bash
gh issue view N --repo casehubio/REPO 2>&1 | head -3
```

Apply based on state:
- **CLOSED in Active Risks:** remove the row — it is no longer an active risk
- **CLOSED in Technical Debt:** update the row — note "✅ resolved" with the closure date; keep for historical context or remove if no longer relevant
- **OPEN:** keep as-is

Also run for cross-repo references:
```bash
gh issue view N --repo casehubio/engine 2>&1 | head -3
gh issue view N --repo casehubio/platform 2>&1 | head -3
```

## Why this is non-obvious

LAYER-LOG is the authoritative source for the migration — reading it feels complete. Issues referenced there were open when written; the LAYER-LOG gives no indication they may since have been resolved. The only way to detect stale references is to query GitHub for each one.
