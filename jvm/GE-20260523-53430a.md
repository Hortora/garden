### Symptom
Code review flags a CDI cycle between two `@ApplicationScoped` beans as Critical — "application will not start." The cycle exists in the injection graph: `A → B (constructor injection) → A`. But the full build and all `@QuarkusTest` suites pass without any startup error.

### Context
`MessageService` (`@ApplicationScoped`) was given a field-injected `ChannelGateway`. `ChannelGateway` (`@ApplicationScoped`) already constructor-injected `MessageService` for its inbound dispatch paths (`receiveHumanMessage`, `receiveObserverSignal`). The cycle: `MessageService → ChannelGateway → MessageService`.

### What was tried
Code reviewer flagged the cycle as Critical, citing CDI startup failure. Using `Instance<ChannelGateway>` (lazy injection) to break it was considered as a fix.

### Root cause
The concern is correct for **pseudo-scoped** beans (`@Dependent`) which cannot be proxied — CDI cannot create a client proxy for a dependent-scoped bean, so a cycle involving one causes startup failure.

For **normal-scoped** beans (`@ApplicationScoped`, `@RequestScoped`, `@SessionScoped`), CDI mandates that the container MUST be able to create a client proxy. Quarkus Arc honors this: when `ChannelGateway`'s constructor is called, it receives a client proxy for `MessageService` — not the actual instance. The proxy is valid immediately; the actual `MessageService` instance is only resolved on the first method call. Both beans initialize without needing the other to exist first.

### Fix
Ensure all beans in the cycle are normal-scoped. Add a comment explaining the deliberate cycle so reviewers don't flag it:

```java
// Deliberate CDI cycle: MessageService → ChannelGateway → MessageService
// (via ChannelGateway.receiveHumanMessage / receiveObserverSignal).
// Both are @ApplicationScoped (normal scope). Arc resolves via client proxies —
// verified by full build and @QuarkusTest suite.
@Inject
ChannelGateway channelGateway;
```

### Why this is non-obvious
Most developers associate "CDI cycle" with startup failure. The distinction — pseudo-scoped vs normal-scoped — is in the spec but rarely surfaced in error messages or tutorials. The symptom that triggered the concern (a code reviewer calling it Critical) is exactly what skilled developers would say if they didn't know about proxy-based cycle resolution.

### Why this fix
A client proxy is the CDI-correct solution. Using `Instance<T>` (lazy injection) also works but adds indirection and a `.get()` call at every use site. Proxy resolution is transparent — callers don't change. Only use `Instance<T>` when a pseudo-scoped bean is in the cycle and proxy creation isn't possible.
