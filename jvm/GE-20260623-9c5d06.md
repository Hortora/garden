## Context

A Java functional interface is evolved to add a context parameter. The abstract method changes from 2-arg to 3-arg, and a 2-arg default is added for backward compat. Test files that used the old 2-arg lambda form are updated to `(type, actorId, ctx) -> ...` to match the new abstract method. The concrete type of `ctx` (e.g. `CommitmentContext`) is not written anywhere in the file body — it is inferred by the compiler from the interface method signature.

## Symptom

CI fails with `impsort` or `formatter` Maven plugin error:
```
Unused import: io.example.CommitmentContext
```

The type was imported because the developer added it when updating the lambda, expecting to need it. But since `ctx` is never referenced by name (it's just a parameter in an inferred-type lambda), the import is unused.

## Root Cause

Java's lambda type inference works through the target functional interface. When you write:
```java
MyInterface p = (type, actorId, ctx) -> { ... };
```

The compiler infers `ctx` as `CommitmentContext` (the 3rd parameter type from `MyInterface.abstractMethod`) without the caller needing to name that type. The import of `CommitmentContext` is therefore never used as a named type in the source file — the compiler doesn't need it.

`impsort-maven-plugin` performs static analysis and correctly flags the import as unused. The `maven-formatter-plugin` may also remove it on format.

## Fix

Do not import the type that appears only as an inferred lambda parameter. If you need to reference it explicitly (e.g. in `assertNull(ctx)` where you cast), the import is needed. If you only need it as a parameter name in the lambda, omit the import entirely.

```java
// ❌ BAD — import not needed, impsort will reject it
import io.example.CommitmentContext;

CommitmentAttestationPolicy p = (type, actorId, ctx) -> Optional.empty();

// ✅ GOOD — no import needed for inferred lambda param
CommitmentAttestationPolicy p = (type, actorId, ctx) -> Optional.empty();
```

If you DO need to name the type (e.g. for casting or explicit type annotation):
```java
// ✅ ALSO GOOD — import needed because CommitmentContext appears as a named type
import io.example.CommitmentContext;

CommitmentAttestationPolicy p = (MessageType type, String actorId, CommitmentContext ctx) -> Optional.empty();
```

## Why This Is Non-Obvious

Developers habitually add imports when updating code that involves a new type. When changing from `(type, actorId)` to `(type, actorId, ctx)`, it's natural to import the new type — you're "using" it. The compiler's type inference that makes the import unnecessary is invisible at write time; you only discover it when CI fails.

The `impsort` error message (`Unused import: io.example.CommitmentContext`) points to the import line, not to the lambda, so the connection between "lambda parameter type inference" and "unused import" isn't obvious without understanding both.
