### Symptom

`@Inject AgentHealthCheck healthCheck` in a `@QuarkusTest` throws `UnsatisfiedResolutionException` — the bean exists and is registered via `HealthBuildItem`, but CDI reports no matching bean for the injection point.

### Context

Writing a Quarkus extension that registers a `@Readiness HealthCheck` bean via `HealthBuildItem`. The bean class has `@Readiness` on the class declaration. A test tries to inject it directly to call `healthCheck.call()` and assert on the response.

### What was tried (didn't work)

```java
@Inject
AgentHealthCheck healthCheck;
```

CDI fails at deployment time with `UnsatisfiedResolutionException` because the injection point has the implicit `@Default` qualifier, but the bean has `@Readiness` as its qualifier.

### Root cause

`@Readiness` is annotated with `@Qualifier` in the MicroProfile Health API. This means it serves double duty: it classifies the health check as a readiness probe for the health subsystem, AND it acts as a CDI qualifier that restricts how the bean can be injected. Plain `@Inject` without `@Readiness` on the injection point will not match.

### Fix

Use `Arc.container().select()` with the `Readiness.Literal.INSTANCE` qualifier:

```java
AgentHealthCheck healthCheck = (AgentHealthCheck) Arc.container()
        .select(HealthCheck.class, Readiness.Literal.INSTANCE).get();
HealthCheckResponse response = healthCheck.call();
```

Or add the qualifier to the injection point:

```java
@Inject
@Readiness
HealthCheck healthCheck;
```

### Why this is non-obvious

`@Readiness` reads as a health check *type classifier* ("this is a readiness check, not a liveness check"), not as a CDI *qualifier* that restricts injection resolution. The dual role is not mentioned prominently in MicroProfile Health docs. Other extensions in the same codebase (e.g. MCP's `McpClientHealthCheck`) use `@Readiness` but never inject the bean directly in tests, so the qualifier effect is invisible until someone tries direct `@Inject`.
