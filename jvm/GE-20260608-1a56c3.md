### Symptom
A second worker session is provisioned immediately after the first exits. The case then has two live workers when only one was intended, and the ledger receives a spurious second `WorkerStarted` event before the case transitions to `COMPLETED`.

### Context
A CaseHub case definition has a single worker binding on `contextChange` with no filter (fires unconditionally). The goal condition is `".workers.researcher.exited == true"`. The case flow:

1. `startCase()` → `CaseStartedEventHandler` fires `CONTEXT_CHANGED` → binding fires → worker provisioned ✅
2. Worker exits → `ClaudonyWorkerExecutionManager` calls `signal("workers.researcher.exited", true)` → `CaseHubRuntime.signal()` patches context → fires another `CONTEXT_CHANGED`
3. `ChoreographyLoopControl.select()` receives the binding again — case is still `RUNNING` at this point (goal/completion transitions are async, next Vert.x iteration) → returns the binding as eligible → `tryProvision()` → **second worker provisioned** ❌
4. Goal transition finally fires → case reaches `COMPLETED` — but second session is already running

### Root cause
`ChoreographyLoopControl.select()` checks only `caseStatus == RUNNING` before returning all eligible bindings. Its own Javadoc states "no dedup mechanism exists in this path." There is no engine-level guard that prevents a binding from firing more than once. The engine's evaluation model is purely reactive: every `CONTEXT_CHANGED` event re-evaluates all bindings from scratch.

### Fix
Add a `when:` guard expression to the binding that excludes the re-fire condition:

```yaml
bindings:
  - name: start-session-on-init
    capability: worker
    on:
      contextChange: {}
    when: ".workers.researcher.exited != true"
```

`CaseContextChangedEventHandler.rules()` evaluates `binding.getWhen()` after the trigger filter. On exit signal context patch: `.workers.researcher.exited` is `true` → `true != true` → `false` → binding skipped. Initial context (empty): `.workers.researcher.exited` is `null` → `null != true` → `true` → binding fires.

In Java (fluent DSL):
```java
Binding.builder()
    .name("start-session-on-init")
    .capability(cap)
    .on(new ContextChangeTrigger((ExpressionEvaluator) null))
    .when(".workers.researcher.exited != true")
    .build();
```

### Why this is non-obvious
The re-fire only happens when `signal()` is called while the case is still `RUNNING` — which is the normal case, because goal/completion transitions are async (fired in the next Vert.x event loop iteration). The window is small and the failure is silent in tests if the second worker creation is mocked. The Javadoc warning exists but is not visible at the YAML authoring level. The `when:` guard is the intended mechanism but is not documented as required for single-fire bindings.
