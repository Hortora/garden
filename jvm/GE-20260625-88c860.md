### Symptom
`NullPointerException` from `Map.copyOf()` when normalizing device capabilities for drift comparison. No useful error message — just the NPE from the JDK's internal `Map.copyOf` implementation.

### Context
Building a capability normalizer that reads `DeviceEntity.capabilities()` and produces an immutable normalized map for comparison. The normalizer calls `Map.copyOf(result)` on the output. Works for SwitchDevice (all fields non-null), fails for LightDevice (nullable `brightness`, `colorTemp`) and other device types with Optional fields.

### What was tried (didn't work)
Direct `Map.copyOf()` on the capabilities map — compiles and passes for simple device types but fails at runtime for devices with nullable capabilities.

### Root cause
`DeviceEntity.capabilities()` returns a `LinkedHashMap` built by each device subclass's `capabilities()` override. Subclasses put null values for Optional fields:
- `LightDevice`: `caps.put("brightness", this.brightness)` where `brightness` is `Integer` (nullable)
- `LightDevice`: `caps.put("colorTemp", this.colorTemp)` where `colorTemp` is `Integer` (nullable)

`Map.copyOf()` throws `NullPointerException` on any null value — it enforces non-null keys AND values.

### Fix
Filter null values before `Map.copyOf()`:

```java
static Map<String, Object> normalize(Map<String, Object> capabilities) {
    var result = new LinkedHashMap<String, Object>();
    for (var entry : capabilities.entrySet()) {
        if (entry.getValue() != null) {
            result.put(entry.getKey(), normalizeValue(entry.getValue()));
        }
    }
    return Map.copyOf(result);
}
```

### Why this is non-obvious
The `capabilities()` method signature returns `Map<String, Object>` — nothing signals that values can be null. A developer would reasonably assume a populated map returned by a getter has non-null values. The NPE only surfaces for specific device types (those with Optional fields), so tests using SwitchDevice pass while LightDevice tests fail.

### Alternatives considered
- Wrapping in `Collections.unmodifiableMap()` instead of `Map.copyOf()` — works but doesn't enforce the non-null contract that `Map.copyOf()` provides. Filtering nulls is the correct fix because absent capabilities should not appear in the normalized map at all.
