## SmallRye Config 'Unrecognized configuration key... will be ignored' warning is cosmetic — @ConfigProperty injection still succeeds

**ID:** GE-20260612-c3b611
**Stack:** Quarkus 3.x, SmallRye Config 3.x
**Symptom:** During test startup (or application startup), the logs show:
```
WARN  [io.quarkus.config] (main) Unrecognized configuration key "my.custom.property" was provided; it will be ignored
```
Despite the warning saying "will be ignored", the `@ConfigProperty`-annotated field on the bean IS populated correctly. Tests that depend on the property value pass.
**Context:** A `@ConfigProperty(name = "my.custom.property", defaultValue = "")` field on an `@ApplicationScoped` bean. The property is NOT declared in any `@ConfigRoot` or `@ConfigMapping`. Quarkus's deployment-time config validation does not know about this key, so it logs the warning. Runtime injection happens independently and succeeds.

### What was tried (didn't work)
- Taking the warning at face value and assuming the property was not being injected
- Debugging why the configured value was being "ignored" — it wasn't

### Root cause
Quarkus has two separate config systems operating at different phases:

1. **Deployment-time config validation** (`ConfigurationValidator`): validates all properties in `application.properties` against declared `@ConfigRoot`/`@ConfigMapping` entries. Any key not matching a declared mapping triggers the "unrecognized" warning. This is a completeness check, not an injection gate.

2. **Runtime `@ConfigProperty` injection**: processes `@ConfigProperty` annotations on CDI beans at augmentation/startup time. This runs independently of the validation and honours the property value regardless of whether it was "recognized" by the validator.

The two systems are decoupled. A property can be unrecognized by the validator and still injected correctly.

### Fix
The warning is safe to ignore for `@ConfigProperty` fields. To suppress it: declare the config property in a `@ConfigMapping` or `@ConfigRoot` class. This is good practice for production code but unnecessary for test-only or optional config.

```java
// To suppress the warning:
@ConfigMapping(prefix = "my.custom")
public interface MyConfig {
    String property();
}
```

### Why this is non-obvious
The warning message says "will be ignored" — this is misleading. It means the key is not in any mapping declaration (and will be ignored by the validator), not that the injected value will be discarded. The distinction between Quarkus's config validation pass and CDI injection is not documented clearly, and the warning text implies the opposite of what actually happens.
