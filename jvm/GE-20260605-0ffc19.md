## Symptom

Flyway migration fails at startup with SQL error 42001 (syntax error) when the migration contains:

```sql
ALTER TABLE channel
    ADD CONSTRAINT chk_name
    CHECK (name SIMILAR TO '^[a-z]...$');
```

The migration runs fine against PostgreSQL but fails against H2 `MODE=PostgreSQL`. Error message gives no indication that `SIMILAR TO` is the problem.

## Root cause

H2's PostgreSQL compatibility mode supports `SIMILAR TO` in `SELECT`/`WHERE` queries but does NOT support it inside `ALTER TABLE ADD CONSTRAINT CHECK` expressions. The two code paths use different SQL parsers; the constraint CHECK parser does not implement `SIMILAR TO`.

## Fix

Use `REGEXP_LIKE()` — supported in both H2 2.x and PostgreSQL 15+:

```sql
ALTER TABLE channel
    ADD CONSTRAINT chk_name
    CHECK (REGEXP_LIKE(name, '^[a-z][a-z0-9]*(-[a-z0-9]+)*$')
           AND LENGTH(name) <= 200);
```

`REGEXP_LIKE(string, pattern)` is an ANSI SQL function. PostgreSQL added it in version 15. H2 2.x supports it natively. If targeting PostgreSQL 14, use the `~` operator instead (POSIX regex match), but `~` is not supported in H2.

## Why this is non-obvious

H2 `MODE=PostgreSQL` is advertised as a PostgreSQL compatibility layer. Developers reasonably expect SQL valid in PostgreSQL to also work in H2 compatibility mode. The failure is non-obvious because `SIMILAR TO` works in H2 `WHERE` clauses — only the constraint context fails. The 42001 syntax error gives no indication that `SIMILAR TO` is the problem, and the migration runs correctly in production (PostgreSQL) but fails only in test (H2).
