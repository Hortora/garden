## Symptom

Calling `TransactionSynchronizationRegistry.registerInterposedSynchronization()` from inside a `@Transactional` method throws:

```
java.lang.IllegalStateException: Syncs are not allowed to be registered
when the transaction is in state 1
```

This happens silently in `@QuarkusTest` / `@TestTransaction` integration tests â€” wrapped by `@WrapBusinessError` into a `ToolCallException` â€” and is hard to trace because "state 1" is not a JTA spec constant.

## Root Cause

Narayana's internal state machine uses "state 1" to mean **ROLLBACK_ONLY** (not `STATUS_ACTIVE = 0`). The spec (`Status.STATUS_ACTIVE = 0`) and Narayana's internal state numbering are different namespaces.

The trigger: when a `@Transactional` method throws an unchecked exception inside a `@TestTransaction`-wrapped test, JTA marks the outer transaction as ROLLBACK_ONLY. Any subsequent `@Transactional` call joins that transaction and, if it also tries to call `registerInterposedSynchronization()`, Narayana rejects the registration because the TX is no longer STATUS_ACTIVE.

Concrete scenario:
1. `@TestTransaction` wraps the full test
2. Test calls `tools.sendMessage(...)` â†’ channel is paused â†’ `IllegalStateException` propagates through `@Transactional` boundary â†’ TX marked ROLLBACK_ONLY
3. `assertThrows()` catches the `ToolCallException` wrapper â€” test continues
4. Test calls `tools.sendMessage(...)` again â†’ `MessageService.dispatch()` tries `tsr.registerInterposedSynchronization()` â†’ Narayana rejects with "state 1"

## Fix

Check `tsr.getTransactionStatus() == Status.STATUS_ACTIVE` before registering:

```java
@Inject
TransactionSynchronizationRegistry tsr;

// In dispatch logic:
if (tsr == null || tsr.getTransactionStatus() != Status.STATUS_ACTIVE) {
    // Fallback: dispatch synchronously â€” TX is rollback-only or absent
    dispatchSynchronously(event, observers);
    return;
}
tsr.registerInterposedSynchronization(synchronization);
```

`tsr.getTransactionStatus()` returns an `int` matching the `jakarta.transaction.Status` constants. Mockito's default mock returns `0` = `STATUS_ACTIVE`, so deferred-dispatch unit tests work without explicit stubbing.

## Why this is non-obvious

- "state 1" has no correspondence to the JTA spec constant `STATUS_ACTIVE = 0`; Narayana's internal state numbering is not documented
- The exception message doesn't mention "ROLLBACK_ONLY" or "STATUS_MARKED_ROLLBACK"
- `@WrapBusinessError` wraps the `IllegalStateException` â†’ `ToolCallException` before it surfaces, hiding Narayana's raw error
- The failure only manifests when a previous `@Transactional` call in the SAME `@TestTransaction` outer TX has already thrown â€” making it look like a second unrelated failure rather than TX state contamination
