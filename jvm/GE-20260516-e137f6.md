### What happened

A `@QuarkusTestProfile` needed to selectively re-include beans that were excluded in the
default test profile via `%test.quarkus.arc.exclude-types`. Returning the same key
without the `%test.` prefix from `getConfigOverrides()` replaced the default exclusion
list entirely for that profile — allowing a shorter list that omitted the beans needed
for the enabled-profile context.

### The technique

`QuarkusTestProfile.getConfigOverrides()` returns a `Map<String, String>` of
config key/value pairs. These overrides are applied with **profile priority** — they
supersede `%test.`-prefixed entries in `application.properties`.

Use this to selectively re-include CDI beans for a specific test profile:

```java
public static class CasehubEnabledProfile implements QuarkusTestProfile {
    @Override
    public Map<String, String> getConfigOverrides() {
        return Map.of(
            "claudony.casehub.enabled", "true",
            // Override the default %test.quarkus.arc.exclude-types.
            // The unprefixed key takes profile precedence — this replaces
            // the default exclusion list, allowing TestResearcherCase to be active.
            "quarkus.arc.exclude-types",
                "io.casehub.ledger.repository.CaseLedgerEntryRepository,"
                + "io.casehub.ledger.service.CaseLedgerEventCapture,"
                // ... other exclusions, but NOT TestResearcherCase
        );
    }
}
```

The key `quarkus.arc.exclude-types` (without `%test.`) wins over
`%test.quarkus.arc.exclude-types` when the profile is active, giving the profile
complete control over which beans are excluded — including the ability to include
beans that the default profile excludes.

### Why this is non-obvious

The natural assumption is that `getConfigOverrides()` adds properties on top of
existing config. In fact, when you return the same config key from `getConfigOverrides()`,
it **replaces** the profiled version entirely. The non-prefixed key in the override
takes precedence over the `%test.`-prefixed key in `application.properties`.

This means: to add one bean back to a 10-item exclusion list, you must return all
9 remaining exclusions in `getConfigOverrides()` — not just the one you want to remove.
The override is a complete replacement, not a delta.

### Alternatives considered

- Using `%<profile-name>.quarkus.arc.exclude-types` in `application.properties` — works
  but requires knowing the profile name at properties-file time and couples the properties
  file to test code.
- `@Disabled` on tests that fail without the excluded bean — masks the real problem.
