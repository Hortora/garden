### Symptom
REST endpoint returns 400 with "claimantId is required" even when the request body contains `{"claimantId":"alice"}`. Changing the body field name from `assigneeId` to `claimantId` has no effect — error persists.

### Context
JAX-RS endpoint uses `@QueryParam("claimant")` to read the caller identity — no `@RequestBody` or `@Body` annotation. A RestAssured test sends the value as a JSON body instead of a query parameter.

### What was tried
Changed the JSON body field name from `"assigneeId"` to `"claimantId"` — still failed with the same error.

### Root cause
`@QueryParam` reads exclusively from the URL query string. Any JSON body is silently discarded with no warning. The service receives `null` for the query param, triggering the "is required" validation — making the error message point to the field name rather than the mechanism.

### Fix

```java
// Wrong — body is ignored regardless of field name:
given().contentType(ContentType.JSON)
    .body("{\"claimantId\":\"alice\"}")
    .put("/workitems/" + id + "/claim")
    .then().statusCode(200);

// Correct:
given().queryParam("claimant", "alice")
    .put("/workitems/" + id + "/claim")
    .then().statusCode(200);
```

### Why this is non-obvious
The error message "claimantId is required" implicates the field name — not the transport mechanism. JAX-RS gives no indication that a body was sent and ignored. The natural fix (correcting the field name) fails silently, making the root cause opaque.
