## Symptom

Two `@DefaultBean @ApplicationScoped` beans from different JARs implement the same interface. One is from a JAR with `META-INF/jandex.idx`. Adding the fully qualified class name (or a glob pattern like `io.example.mock.*`) to `quarkus.arc.exclude-types` in both main and test `application.properties` has no effect — the bean remains registered, the CDI ambiguity persists, and the build fails with `AmbiguousResolutionException`.

## Root Cause

`quarkus.arc.exclude-types` applies to beans discovered via Quarkus's own scanning mechanism. Beans from JARs that ship `META-INF/jandex.idx` are processed directly via Jandex at a different augmentation phase that the `exclude-types` property does not hook into. The exclusion is silently applied to the wrong scope — no error, no warning, simply ignored.

## Fix

Don't use `exclude-types` for this scenario. Instead, introduce a concrete `@ApplicationScoped` (NOT `@DefaultBean`) bean in your project that implements the conflicting interface:

```java
@ApplicationScoped
public class MyNoOpImpl implements ConflictingInterface {
    // minimal implementation
}
```

CDI picks the most-specific non-default bean, which suppresses ALL `@DefaultBean` candidates regardless of which JAR they come from. If this is test-only, place the class in `src/test/java`.

## Why This Is Non-Obvious

The same `exclude-types` property works correctly for beans discovered from application source code and from JARs without Jandex indices. The failure mode is JAR-specific and produces no diagnostic output — the bean simply isn't excluded and the AmbiguousResolutionException looks unrelated to the exclusion attempt. Every retry with different class name formats (exact, glob, package wildcard) produces the same silent non-effect.
