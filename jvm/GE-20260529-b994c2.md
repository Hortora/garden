## Context

A reactive SPI bridge wraps a blocking delegate using `Uni.createFrom().item(supplier)`. The goal is to offload the blocking call off the Vert.x IO thread. `.emitOn(Infrastructure.getDefaultWorkerPool())` is chained expecting it to shift execution.

## Symptom

Vert.x blocked-thread warning fires in production. The supplier still executes on the event loop thread despite the `emitOn` chain.

## Root Cause

`emitOn(executor)` controls where **downstream signals** (item, failure, completion) are dispatched to subscribers — not where the upstream computation runs. For `Uni.createFrom().item(supplier)`, the supplier is called during subscription, on the subscription thread. If the subscription originates on the IO thread, `emitOn` has no effect on when the supplier runs — by the time the signal reaches `emitOn`, the supplier has already executed and blocked.

The correct method is `runSubscriptionOn(executor)`, which routes the subscription request itself to the given executor — meaning the supplier in `item()` evaluates on that executor.

## Fix

```java
// ❌ Wrong — emitOn does not offload the item() supplier
return Uni.createFrom()
    .item(() -> delegate.render(descriptor, context))
    .emitOn(Infrastructure.getDefaultWorkerPool());

// ✅ Correct — runSubscriptionOn routes the subscription (and the supplier) to the worker pool
return Uni.createFrom()
    .item(() -> delegate.render(descriptor, context))
    .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());
```

The `runSubscriptionOn` fix applies to any `Uni.createFrom().item(supplier)` wrapping a blocking call. `emitOn` is correct when the blocking work is in a downstream operator that follows it (e.g. `.voidItem().emitOn(pool).map(ignored -> blockingWork())`).

## Why This Is Non-Obvious

`emitOn` and `runSubscriptionOn` look symmetrical — both redirect to an executor — but they operate at different points in the Mutiny pipeline. The Vert.x blocked-thread warning is the only signal something is wrong; there is no compile-time error. The existing technique GE-20260529-ff186e shows `emitOn` working correctly for a different pattern; this gotcha explains when it silently fails.

### Why this fix

`runSubscriptionOn` is the right semantic: "evaluate this on the worker pool", which is exactly what's needed for a bridge that wraps a blocking operation in a supplier. `emitOn` is the right semantic for "deliver results to consumers on this executor" — a different concern.

## Constraints

Applies to `Uni.createFrom().item(supplier)` and `Uni.createFrom().item(value)`. For `Uni.createFrom().emitter(...)` or operator chains, `emitOn` placed before the blocking operator is correct.
