## Symptom

A method annotated `@Transactional(REQUIRES_NEW)` calls a JPA repository method to read
an entity that was updated in the outer transaction. The query always returns the
**pre-update** state — as if the outer transaction's changes never happened.

In-memory test stores (e.g. `InMemoryCommitmentStore`) mask this completely — they ignore
transaction boundaries and return the latest in-memory value. Tests pass; production silently
reads stale data.

## Context

`MessageService.send()` runs in the outer `REQUIRED` transaction. It calls
`CommitmentService.fulfill()` which updates a `Commitment` entity (state: OPEN → FULFILLED).
Then it calls `LedgerWriteService.record()` which is `@Transactional(REQUIRES_NEW)`.

Inside `REQUIRES_NEW`, calling `commitmentStore.findByCorrelationId()` returns the
**pre-update** state (OPEN) because:

1. `REQUIRES_NEW` **suspends** the outer transaction — the outer tx's connection is parked
2. A **new database connection** is obtained with its own view of committed data
3. The outer tx's `FULFILLED` update is still in the first-level cache, not yet committed
4. The new connection sees the last **committed** state: OPEN

## What was tried (didn't work)

Querying `CommitmentStore` inside the `REQUIRES_NEW` method to drive behaviour
(e.g. writing an attestation only when the commitment is terminal). Always returned OPEN.

## Root cause

`REQUIRES_NEW` semantics: the outer transaction is **suspended**, not joined.
Any JPA read inside the `REQUIRES_NEW` method goes to a fresh connection and
sees only committed database state. The outer tx's pending writes are invisible.

## Fix

Do not read entities that were modified in the outer transaction from inside a
`REQUIRES_NEW` method. Derive the needed information from the **method parameters**
or from **already-committed** data (e.g. prior ledger entries, committed at an
earlier `REQUIRES_NEW` call).

In the concrete case: derive attestation verdict from `MessageType` (a method parameter)
rather than querying `CommitmentStore`. The prior COMMAND's ledger entry (committed in a
previous `REQUIRES_NEW` call) is safely readable from the new connection.

```java
// WRONG — reads stale OPEN state from inner REQUIRES_NEW
Optional<Commitment> c = commitmentStore.findByCorrelationId(corrId); // always OPEN

// RIGHT — derive from MessageType (already known, no DB query needed)
attestationPolicy.attestationFor(message.messageType, resolvedActorId)
    .ifPresent(outcome -> writeAttestation(prior, outcome));
```

## Why this is non-obvious

- InMemory stores used in unit/integration tests ignore transaction boundaries —
  tests pass with the wrong pattern, masking the production bug entirely
- The outer transaction *appears* active (code runs within it), so developers
  assume JPA reads will see its writes
- The bug only manifests in production with a real JPA provider and connection pooling
