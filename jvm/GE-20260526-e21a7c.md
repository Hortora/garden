### What was discovered

When a JPA entity uses `@EntityListeners` with a `@PrePersist` enricher pipeline, pre-setting a field on the entity **before** calling `save()` will be respected — if the enricher guards on `if (field != null) return`. The enricher then skips that field entirely, preserving the pre-set value.

This pattern enables caller-controlled overrides of enricher logic without modifying the enricher or the entity:

```java
// Enricher (casehub-ledger TraceIdEnricher — internal implementation):
@Override
public void enrich(final LedgerEntry entry) {
    if (entry.traceId != null) {   // ← null-guard
        return;
    }
    traceIdProvider.currentTraceId().ifPresent(id -> entry.traceId = id);
}

// Caller — pre-populate before save(), bypassing enricher for this field:
entry.traceId = capturedTraceIdFromCallerThread;  // set before save()
ledgerRepo.save(entry);  // @PrePersist enricher runs but skips traceId because != null
```

### Context

Encountered in casehub-ledger's `TraceIdEnricher`, which auto-populates `LedgerEntry.traceId` from the active OTel span at `@PrePersist` time. Because `@ObservesAsync` CDI observers run on a managed executor thread where OTel's `ThreadLocal` context is empty, the enricher would always write `null`.

The fix was to capture `traceId` synchronously on the original thread, pass it through the event, and pre-populate `entry.traceId` before `save()`. The null-guard in `TraceIdEnricher` ensures the pre-set value is not overwritten.

### Why this is non-obvious

The `@PrePersist` lifecycle is typically treated as "the entity listener owns this field — don't touch it before persist". There's no JPA documentation that explicitly describes the null-guard pattern for making enrichers overridable. The behavior is emergent from standard Java null semantics, but the design choice to use a null-guard rather than unconditional assignment is not obvious from the API.

A developer looking at this would expect: "I can't set `traceId` before `save()` because the listener will overwrite it." The null-guard is only visible by reading the enricher source code — it's an undocumented contract between the enricher and its callers.

### Applicability

This pattern works when:
1. The enricher explicitly guards: `if (field != null) return`
2. The field has a meaningful distinction between "not set" (null) and "set to a value" (including empty string or 0)
3. The caller has access to the entity object before `save()` is called

It does **not** work when the enricher unconditionally overwrites: `entry.field = computedValue`. Check the enricher source before relying on this.
