## OCC + policyTriggered flag for M-of-N threshold completion â€” prevents duplicate trigger under READ COMMITTED

### What it achieves

Correct M-of-N completion threshold detection under concurrent transactions at READ COMMITTED isolation, without pessimistic locking.

### Context

Naive recount inside a completion transaction is broken at READ COMMITTED (PostgreSQL default): two concurrent child completions each see the other's write as uncommitted, both count M-1, neither triggers â€” group is permanently stuck. Pessimistic locking (`SELECT FOR UPDATE`) fixes correctness but serialises all completions through a single row lock.

### The technique

Add to the group entity:

```java
@Column(nullable = false) public int completedCount = 0;
@Column(nullable = false) public int rejectedCount = 0;
@Column(nullable = false) public boolean policyTriggered = false;
@Version public Long version = 0L;
```

When a child reaches a terminal status, in a single transaction:

1. Load the group entity â€” `@Version` is captured
2. Increment `completedCount` or `rejectedCount`
3. If threshold met **and** `policyTriggered == false`: set `policyTriggered = true`, trigger parent completion
4. Commit â€” if another transaction concurrently modified the group, `OptimisticLockException` fires
5. On `OptimisticLockException`: retry once. On retry, `policyTriggered` will be `true` (another transaction won the race â€” do nothing) or the count genuinely differs (re-evaluate)

### Why this is non-obvious

The naive recount approach looks correct â€” count completed children, compare to threshold. The READ COMMITTED visibility gap (uncommitted concurrent writes are invisible) is the non-obvious trap. The `policyTriggered` flag is the key: without it, the retrying transaction after an OCC conflict would fire the threshold action a second time even though the winner already fired it.

### Why this fix

Pessimistic locking is the obvious fix but serialises all child completions through a single row. OCC with a flag separates correctness (the flag prevents double-fire) from throughput (concurrent transactions proceed in parallel and only contend on the rare collision).

### Alternative â€” conditional UPDATE (no `@Version`, no retry)

**Approach:** Issue a single `UPDATE ... WHERE policyTriggered = false` and check the affected row count. No version field, no retry loop.

**In JPA (Panache reactive):**

```java
// Returns true if this call won â€” false means another transaction already triggered
Uni<Boolean> won = SubCaseGroupEntity.update(
    "policyTriggered = true WHERE parentCaseId = ?1 AND groupId = ?2 AND policyTriggered = false",
    parentCaseId, groupId)
  .map(count -> count > 0);
```

Caller checks `won` before proceeding with threshold actions â€” if `false`, return immediately.

**In-memory (no JPA):** Use a `synchronized` check-and-set:

```java
public synchronized boolean trySetPolicyTriggered() {
    if (policyTriggered) return false;
    policyTriggered = true;
    return true;
}
```

**Pros:** Simpler â€” no `@Version` field, no `OptimisticLockException` handling, no retry infrastructure. The `Uni<Boolean>` return pairs naturally with a reactive SPI.

**Cons/trade-offs:** Requires the database to honour the WHERE clause atomically (PostgreSQL READ COMMITTED guarantees this for single-row UPDATE). Does not compose easily if additional atomic checks are needed at trigger time. The `@Version` + retry approach is more general and survives multi-condition scenarios.
