## Quartz job store: RAM store for stateless scheduling

**Topic:** Quartz store type selection
**Alternatives exist:** yes — JDBC store for durable, clustered job persistence

### The convention

Configure Quartz with RAM store:

```properties
quarkus.quartz.store-type=ram
```

Never add Quartz tables to Flyway migrations.

### Why this style

RAM store requires zero schema setup, no datasource dependency for Quartz itself, and no stale job state across restarts. Jobs are defined by application logic and re-registered on startup — durability isn't needed. JDBC store adds schema tables, complicates Flyway migrations, and provides persistence the use case doesn't require.

### Trade-offs

RAM store: jobs lost on restart. JDBC store: durable across restarts, supports true clustering — necessary if jobs must survive application crashes or if multiple instances must not double-fire the same job.

### When not to use it

When job persistence across restarts is required, or when true clustering with leader election is needed. Use JDBC store and the appropriate Flyway migration namespace.
