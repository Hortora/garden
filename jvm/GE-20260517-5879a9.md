### Symptom
A qhorus Commitment remains OPEN after a human participant sends a RESPONSE or DONE message via `HumanParticipatingChannelBackend`. The `commitmentService.fulfill()` auto-transition in `MessageService.send()` never fires. The Commitment stays OPEN indefinitely despite the human having responded.

### Context
Occurs when using qhorus with `HumanParticipatingChannelBackend` for human governance (PI oversight, IRB approval, etc.). The COMMAND message correctly opens the Commitment. The human's response closes nothing.

### Root cause
`ChannelGateway.receiveHumanMessage()` calls `messageService.send()` with hardcoded `correlationId=null`:

```java
public void receiveHumanMessage(ChannelRef channel, InboundHumanMessage raw) {
    NormalisedMessage normalised = normaliser.normalise(channel, raw);
    messageService.send(channel.id(), normalised.senderInstanceId(),
            normalised.type(), normalised.content(),
            null,   // ← correlationId always null
            null, null, null, ActorType.HUMAN);
}
```

`MessageService.send()` only calls `commitmentService.fulfill(correlationId)` when `message.correlationId != null`. With null correlationId the commitment switch is skipped entirely. `InboundHumanMessage` has no correlationId field — it is structurally impossible for the human path to carry a correlationId. Tracked as casehubio/qhorus#154.

### Fix
In the application's response listener, call `commitmentService.fulfill()` or `commitmentService.decline()` explicitly using the known correlationId (typically the entity ID used when issuing the original COMMAND):

```java
// In PiResponseListener.process() or equivalent:
boolean approved = messageType == MessageType.DONE;
if (approved) {
    commitmentService.fulfill(entityId.toString());  // explicit close
} else {
    commitmentService.decline(entityId.toString());
}
```

Both methods are idempotent for already-terminal or non-existent commitments — safe to call even if the commitment was auto-closed by some other path.

### Why this is non-obvious
The qhorus COMMAND flow creates a Commitment automatically via `MessageService.send()` (COMMAND type + non-null correlationId). A developer naturally expects the RESPONSE/DONE flow to close it symmetrically — same mechanism, opposite direction. It doesn't. The inbound path is structurally different and the missing correlationId is invisible at call sites. The Commitment silently stays OPEN with no error.
