## Symptom

A `@ConsumeEvent` handler that calls `someCompletionStage.toCompletableFuture().join()` (or any other blocking wait) hangs indefinitely. No exception is thrown, no timeout fires, the handler simply never completes. Other events in the system may appear to stop processing too.

## Context

Triggered when spawning a child case inside a Vert.x event-bus handler:

```java
@ConsumeEvent(value = "casehub.subcase.schedule")
public Uni<Void> onSubCaseSchedule(SubCaseScheduleEvent event) {
    UUID childId = caseHubRuntime.startCase(def, ctx)
        .toCompletableFuture()
        .join(); // ← DEADLOCK
    ...
}
```

`startCase()` internally fires a `CaseLifecycleEvent` (async CDI). That event cannot propagate because the same Vert.x event-loop thread is blocked by `.join()`. The system is waiting for a result that can only be produced by the blocked thread itself.

## What was tried

Adding logging before and after `.join()` confirmed the handler entered but never returned. Checking for exceptions produced nothing. The hang only appeared when `startCase()` was the blocking call.

## Root Cause

By default, `@ConsumeEvent` handlers run on the Vert.x event-loop thread. Any blocking call on the event-loop blocks ALL event delivery on that loop, creating a deadlock when the blocking call itself depends on an event being delivered.

## Fix

Add `blocking = true` to the annotation:

```java
@ConsumeEvent(value = "casehub.subcase.schedule", blocking = true)
public Uni<Void> onSubCaseSchedule(SubCaseScheduleEvent event) {
    UUID childId = caseHubRuntime.startCase(def, ctx)
        .toCompletableFuture()
        .join(); // safe — runs on worker thread
    ...
}
```

`blocking = true` dispatches the handler to a Vert.x worker thread pool thread, freeing the event loop to continue delivering other events while `.join()` blocks.

## Why this is non-obvious

The annotation compiles and runs without error in non-blocking usage. There is no compile-time warning for blocking operations inside `@ConsumeEvent` handlers. The deadlock is silent — no exception, no log line, just an infinite hang. The failure is also timing-dependent: simple handlers that happen to complete before any re-entrant event is needed will appear to work fine, masking the bug until a handler that depends on a downstream event is added.
