### Symptom
All rule unit tests fail with `UnsupportedOperationException: You're trying to
dynamically define a class, please add the module org.drools:drools-wiring-dynamic
to your classpath` when using `RuleUnitProvider.get().createRuleUnitInstance(data)`
in plain JUnit.

### Context
GE-0063 establishes that `DataSource.createStore()` works in plain JUnit with the
`drools-quarkus` dependency. This creates the expectation that rule units are also
testable without Quarkus boot. They are not — rule unit *compilation* (generating
the executable rule session from DRL) requires the Quarkus Drools extension's
code generation, not just the runtime API.

### Root cause
`RuleUnitProvider.get()` in plain JUnit resolves to the default provider which
attempts dynamic class definition. The Quarkus Drools extension pre-compiles
rule units at build time via augmentation — the compiled rule unit is only
available when CDI injects `RuleUnit<T>` in a `@QuarkusTest` context.

### Fix
Use `@QuarkusTest` with CDI injection:

```java
@QuarkusTest
class MyRuleUnitTest {
    @Inject RuleUnit<MyRuleUnit> ruleUnit;

    @Test
    void myTest() {
        var data = new MyRuleUnit();
        try (RuleUnitInstance<MyRuleUnit> instance =
                 ruleUnit.createInstance(data)) {
            instance.fire();
        }
    }
}
```

Do NOT use `RuleUnitProvider.get().createRuleUnitInstance(data)` — this bypasses
the Quarkus augmentation and fails.

### Why this is non-obvious
GE-0063 proves `DataSource.createStore()` works without Quarkus, suggesting the
Drools API is usable in plain JUnit. The distinction is that DataStore is a
runtime data structure (no compilation needed), while RuleUnit instantiation
requires compiled DRL — which only the Quarkus extension produces.
