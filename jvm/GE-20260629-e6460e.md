### Symptom
`%dev.quarkus.arc.selected-alternatives` is set in `application.properties` alongside a non-profiled `quarkus.arc.selected-alternatives`. In dev mode, the `@Alternative` beans listed in the `%dev.` value are never activated — ArC throws `UnsatisfiedResolutionException` for the interfaces they implement, even though the beans exist on the classpath and have Jandex indices.

### Context
Multi-profile Quarkus application needing different CDI alternatives per environment:
- Production: reactive JPA stores (`engine-persistence-hibernate`)
- Dev mode: in-memory blocking stores (`engine-persistence-memory`)
- Tests: separate `application.properties` with its own `selected-alternatives`

Both `%dev.` and non-profiled values present in the same `application.properties`.

### What was tried (didn't work)
1. `%dev.quarkus.arc.selected-alternatives=MemoryPlanItemStore,...` alongside non-profiled `quarkus.arc.selected-alternatives=JpaPlanItemStore,...` — dev alternatives silently ignored
2. Removing the non-profiled value and using only `%dev.` and `%prod.` — still ignored
3. Verified `%dev.quarkus.arc.exclude-types` DOES work (OidcCurrentPrincipal successfully excluded in dev) — confirms profile overrides work for other ArC properties

### Root cause
`quarkus.arc.selected-alternatives` is a build-time fixed property that does not honour `%profile.` prefixes. The property is resolved once during Quarkus augmentation and profile-specific values are silently discarded. This is not documented — the Quarkus configuration reference does not distinguish which ArC properties support profile overrides.

### Fix
Select ALL alternatives (both JPA and memory) in the non-profiled `selected-alternatives`. Use profile-specific `exclude-types` to control which survive in each environment:

```properties
# Non-profiled: select everything — both JPA and memory alternatives
quarkus.arc.selected-alternatives=\
  io.casehub.persistence.jpa.JpaPlanItemStore,\
  io.casehub.persistence.memory.MemoryPlanItemStore,\
  io.casehub.persistence.memory.InMemoryCaseMetaModelRepository

# Dev: exclude reactive JPA stores → memory stores win
%dev.quarkus.arc.exclude-types=io.casehub.persistence.jpa.JpaPlanItemStore,...

# Prod: exclude memory stores → JPA stores win
%prod.quarkus.arc.exclude-types=io.casehub.persistence.memory.MemoryPlanItemStore,...
```

ArC processes `exclude-types` first (removing beans entirely), then resolves `selected-alternatives` among surviving beans. In each profile, only one alternative per interface survives — no ambiguity.

### Why this is non-obvious
Profile-specific overrides (`%dev.`, `%prod.`) work for most Quarkus properties, including `quarkus.arc.exclude-types`. The natural assumption is that `selected-alternatives` follows the same pattern. There is no error, warning, or log message when the profile override is ignored — the non-profiled value is silently used. The failure manifests as `UnsatisfiedResolutionException` with no hint about profile resolution.

### Why this fix
`exclude-types` is a coarser mechanism (removes the bean entirely vs just not selecting it), but it's the only ArC property that supports profile overrides. The exclude-types-first-then-resolve pattern is architecturally sound: each profile starts from the same selected set and carves out what doesn't belong.

### Alternatives considered
- **Maven profiles** — add/remove JARs per profile. Rejected: adds build complexity and doesn't work with `quarkus:dev` which uses the default Maven profile.
- **`@IfBuildProfile("dev")` wrapper beans** — create dev-only `@ApplicationScoped` beans that delegate to memory stores. Rejected: code duplication across 5+ SPIs, maintenance burden.
- **Separate `application-dev.properties`** — Quarkus doesn't support this pattern (it's a Spring Boot convention).
