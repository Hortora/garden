## Symptom

Quarkus startup fails in a module that uses in-memory persistence (no JDBC datasource):

```
ConfigurationException: Datasource '<default>' is not configured.
Persistence unit '<default>' defines entities [io.quarkiverse.work.core.filter.FilterRule],
but its datasource '<default>' cannot be found.
```

## Context

Occurs when `quarkus-work-core` is a transitive dependency (e.g. via `engine` module)
in a Quarkus test module that does not configure a datasource. `FilterRule` is a
`@Entity` (Hibernate ORM Panache) in quarkus-work-core and is discovered by Quarkus's
CDI/Hibernate scanning across all modules on the classpath.

## Root cause

`quarkus-work-core` bundles a `FilterRule` JPA entity and declares Hibernate ORM Panache
as a runtime dependency. Any module that transitively depends on `quarkus-work-core`
inherits the entity scanning, requiring a configured datasource even when the module
has no persistence of its own.

## Fix

Add to the affected module's test `application.properties`:

```properties
# Disable Hibernate ORM so quarkus-work-core's FilterRule entity
# does not require a datasource in modules using in-memory persistence.
quarkus.hibernate-orm.enabled=false
```

## Why this is non-obvious

The module didn't directly depend on `quarkus-work-core` — it arrived as a transitive
dependency through `engine`. The error message names `FilterRule` but gives no indication
of which JAR introduced it or why a datasource is needed in a module that never configured one.
