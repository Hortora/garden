### Symptom

NullPointerException inside production code immediately after adding a new parameter to a mocked method, while the test itself compiles without error:

```
java.lang.NullPointerException: Cannot invoke "Channel.id" because "channel" is null
    at OpenClawCaseChannelProvider.openChannel(...)  ← production code
```

The `when(channelService.create(...))` stub is visibly present in the test. The stub was not the suspected culprit.

### Context

A production method had a 9-parameter `create()` overload. All test stubs used 9 `anyString()`/`any()` matchers. The method was upgraded to a 10-parameter overload (adding `deniedTypes`). The stubs were not updated. All stubs continued to compile — Java sees them as matching a valid (now non-called) 9-param overload variant. Mockito finds no match for the 10-param call and returns `null` by default. Production code then dereferences `channel.id` and NPEs.

### What was tried

- Inspected the production code change (`channelService.create(...)` call) — looked correct
- Checked method parameters — passed correctly
- Suspected a CDI wiring issue — no

The stub was the last thing examined because it compiled without error and Mockito gave no warning.

### Root cause

Mockito's `when(service.method(any(), any()))` stubs are overload-specific: the N `any()` matchers match an N-parameter variant. When production code upgrades to N+1 parameters, the stub silently matches nothing. Mockito's default (lenient) mode returns `null` for unmatched calls without logging or throwing. The NPE appears at the first dereference of the null return value inside production code, which is often several lines away from the stub.

### Fix

Count the parameters in the upgraded overload and add one `any()` matcher per added parameter to **every** `when()` and `verify(..., never()).method()` call in the test. IntelliJ's "unresolved method" highlight will not help — Mockito `when(mock.method(matchers))` calls the real mock (which has no real body), so overload resolution succeeds even with the wrong count.

```java
// Before (9-param, now broken):
when(channelService.create(anyString(), anyString(), any(), any(), any(), any(), any(), any(), any()))
    .thenReturn(channel(channelId, name));

// After (10-param, matches upgraded overload):
when(channelService.create(anyString(), anyString(), any(), any(), any(), any(), any(), any(), any(), any()))
    .thenReturn(channel(channelId, name));
```

Also update all `verify(service, never()).method(...)` calls — they must also match the new arity.

### Why this is non-obvious

Mockito operates on Java method calls, not source-code signatures. The stub appears syntactically correct — the N-param `when(service.create(N matchers))` compiles because the N-param overload still exists. No compile error, no Mockito warning in default mode. The NPE lands inside production code, not in the test setup, making the stub the last suspected location.

### Why this fix

Strict stubbing mode (`@ExtendWith(MockitoExtension.class)` with `Strictness.STRICT_STUBS`) would surface this as an "unused stub" error instead. For projects using lenient mocks, the only defence is the explicit fix above. Prefer strict mode in new test classes to prevent this class of silent failure.
