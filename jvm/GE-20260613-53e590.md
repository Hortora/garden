## Symptom

An LLM renderer returns stale enriched prompts after a `RESPONSE_FORMAT` property description is updated. The cached prompt was generated under the old schema description and continues to be served. The issue is silent — no error, no warning, no observable difference until the output is inspected closely.

## Context

A renderer uses:
```java
private static final String TEMPLATE_HASH = fingerprint(PROMPT_TEMPLATE).substring(0, 8);
```

This hash is included in the cache key:
```
descriptorHash + ":" + contextHash + ":" + format.name() + ":" + TEMPLATE_HASH
```

`RESPONSE_FORMAT` is a `static final ResponseFormat` object holding JSON schema descriptions for each output field. Changing a property description (e.g. updating `capabilityNarrative` from `"including domain confidence"` to `"including input and output types when present"`) does not change `TEMPLATE_HASH`, so the cache key is identical before and after the change.

The issue is safe only as long as every `RESPONSE_FORMAT` description change coincides with a `PROMPT_TEMPLATE` edit (which does change `TEMPLATE_HASH`). This coupling is implicit — nothing enforces it.

## Root cause

`TEMPLATE_HASH` is named as though it covers the full LLM contract, but it only covers the prompt string. The structured output schema descriptions — which directly influence how the LLM formats its response — are a second, untracked component of that contract.

## Fix

Include `RESPONSE_FORMAT` schema description strings in the hash computation alongside `PROMPT_TEMPLATE`. Extract the description strings into constants so they can be included deterministically:

```java
private static final String TEMPLATE_HASH = fingerprint(
    PROMPT_TEMPLATE +
    CAPABILITY_NARRATIVE_DESC +   // extracted from ResponseFormat
    DISPOSITION_NARRATIVE_DESC
).substring(0, 8);
```

Until this is fixed: treat any `RESPONSE_FORMAT` description change as requiring a concurrent, arbitrary `PROMPT_TEMPLATE` edit to force cache invalidation.

## Why this is non-obvious

`TEMPLATE_HASH` sounds complete — "the template is hashed." The `RESPONSE_FORMAT` object is constructed separately and its descriptions are Java string literals in a builder chain, not part of the template string. There is no structural connection between the two; nothing at the call site suggests they must be kept in sync. The coupling is implicit and discovered only when a description-only change fails to produce fresh output.
