## WorkItemLifecycleEvent has no workItem() method — use event.source() cast to WorkItem

### Symptom

Code calling `event.workItem()` on a `WorkItemLifecycleEvent` fails to compile: method not found.

### Context

`WorkItemLifecycleEvent` carries the `WorkItem` entity snapshot. The obvious accessor name would be `workItem()`, but the class does not define this method. Occurs when writing CDI observers for `@ObservesAsync WorkItemLifecycleEvent`.

### Root Cause

`WorkItemLifecycleEvent` extends `WorkLifecycleEvent`, which defines a `source()` method returning `Object`. The `WorkItem` entity is stored in the private `workItem` field but is exposed via the inherited `source()` method, not via a dedicated accessor.

```java
// The actual API:
WorkItem wi = (WorkItem) event.source();
```

Note: `event.source()` returns null for events created via `WorkItemLifecycleEvent.fromWire(...)`. These are wire-deserialized events where `workItem = null` by design (the snapshot is not available on remote nodes). Always null-check before casting.

### Additional Gotcha: fromWire() for null-WorkItem tests

`WorkItemLifecycleEvent.of(String, WorkItem, String, String)` NPEs immediately if `workItem` is null — it accesses `workItem.id` inside the factory method. To construct a test event with no WorkItem snapshot, use:

```java
// Test-only: create event without a WorkItem snapshot
WorkItemLifecycleEvent event = WorkItemLifecycleEvent.fromWire(
    "COMPLETED", "/workitems/xxx", "xxx-id", taskId,
    WorkItemStatus.COMPLETED, Instant.now(), "actor", null, null, null, null);
// event.source() returns null — null check in observer fires
```

### Why Non-Obvious

No `workItem()` method and no Javadoc on `WorkItemLifecycleEvent` explaining the `source()` indirection. The pattern is inherited from `WorkLifecycleEvent` and not obvious from the subclass API.
