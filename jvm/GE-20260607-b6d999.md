### The technique

When multiple independent writers need to write `LedgerEntry` subclasses for the same logical entity (e.g., an investigation `caseId`) but cannot coordinate sequence number assignment, give each concern its own **dedicated, namespaced ledger subject** derived deterministically from the shared entity ID.

```java
// Instead of: entry.subjectId = caseId  (shared with all writers)
// Use a namespaced derivation:
static UUID attestationSubjectFor(UUID caseId) {
    return UUID.nameUUIDFromBytes(
        ("aml-trust-routing-attestation:" + caseId).getBytes(StandardCharsets.UTF_8));
}
```

Each concern gets its own independent sequence timeline. Entries can still reference the canonical entity via a separate field:

```java
entry.subjectId = attestationSubjectFor(event.caseId());  // ledger ordering
entry.investigationCaseId = event.caseId();               // domain query field
```

Application queries use `investigationCaseId` for business lookup. Ledger chains use `subjectId` for ordering and Merkle proofs.

### Why this is non-obvious

The instinct is to use the shared entity ID (`caseId`) as the ledger subject everywhere — it creates a single unified audit chain queryable by one key. This breaks when multiple writers use independent sequence-assignment mechanisms (different repositories, different sequence tables). The constraint `IDX_LEDGER_ENTRY_SUBJECT_SEQ` enforces uniqueness globally across all subtypes for a subject, but sequence assignment is often scoped to a single repository. Using dedicated subjects eliminates the coordination requirement entirely.

The derived UUID (via `UUID.nameUUIDFromBytes`) is deterministic, stable across restarts, and collision-free for any realistic number of concerns per entity.

### When to apply

Apply when:
- Two or more CDI beans write `LedgerEntry` subtypes for the same `subjectId`
- They run in independent transactions (e.g., REQUIRES_NEW vs. outer REQUIRED)
- A single global sequence allocator (e.g., `LedgerSequenceAllocator`) is not shared between them

Do **not** apply when a single `LedgerEntryRepository.save()` is the only writer — the allocator handles ordering correctly without namespace isolation.
