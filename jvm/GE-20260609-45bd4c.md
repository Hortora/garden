## @ActivateRequestContext required on methods called from Quartz worker threads

### Symptom

A method works correctly when called from a JAX-RS endpoint but throws `jakarta.enterprise.context.ContextNotActiveException: CDI context associated with @RequestScoped is not active` when called from inside a Quartz worker lambda (engine job execution).

### Context

CasehubEngine executes worker lambdas on Quartz threads via `QuartzWorkerExecutionJob`. These threads are not JAX-RS request threads — no CDI request context is active by default. Any method that (transitively) uses `@RequestScoped` beans will fail.

Common example: a method calls `WorkItemService.create()`, which uses JPA persistence context beans that are `@RequestScoped`.

### Root Cause

Quartz job threads in Quarkus are scheduler-managed threads. The CDI request scope lifecycle (activate → transaction → deactivate) is tied to JAX-RS request threads or Vert.x request handlers — not to Quartz job threads. Without explicit activation, any `@RequestScoped` injection point is inaccessible.

### Fix

Annotate the method that will be called from Quartz threads with `@ActivateRequestContext`:

```java
import io.quarkus.arc.ActivateRequestContext;

@ApplicationScoped
public class MyService {

    @ActivateRequestContext
    public String doWork(UUID caseId) {
        // workItemService.create() is safe here — request context is active
        return workItemService.create(request).id.toString();
    }
}
```

CDI activates a request context for the duration of the annotated method and deactivates it on return. If a request context is already active (e.g. called from a JAX-RS endpoint), the annotation is a no-op — safe to add unconditionally.

### Why Non-Obvious

The method works in integration tests that invoke it from a test thread (which may have a request context from `@QuarkusTest` infrastructure) or from JAX-RS endpoints. The failure only surfaces when the same method is called from a Quartz job thread. The error message mentions CDI context but does not identify which caller path lacks it.
