## Problem

Tests that need precise control over IMAP message headers (e.g. absent `Message-ID`, absent `From:`, absent `To:`) cannot use Greenmail's SMTP server — the SMTP pipeline synthesises and injects headers, changing the message before it reaches the mailbox.

## Technique

Use `MailFolder.appendMessage()` to inject a `MimeMessage` directly into a Greenmail IMAP mailbox, bypassing the SMTP pipeline entirely:

```java
// greenMail is a GreenMail instance (not GreenMailExtension)
GreenMailUser user = greenMail.getUserManager().getUser("inbox@example.com");
MailFolder inbox = greenMail.getManagers().getImapHostManager().getInbox(user);

// Build any MimeMessage — only the headers you set will appear
MimeMessage msg = new MimeMessage(Session.getInstance(new Properties()));
msg.setFrom(new InternetAddress("sender@example.com"));
msg.setSubject("No Message-ID");
msg.setText("Body");
msg.setSentDate(Date.from(Instant.now()));
// No msg.setHeader("Message-ID", ...) — it stays absent

inbox.appendMessage(msg, new Flags(), new Date());
// Message is now in the IMAP inbox with no Message-ID header
```

The message appears as UNSEEN in the mailbox, ready for IMAP polling.

## When to Use

- Testing parser behaviour when headers are absent (absent Message-ID, absent From:, BCC-only To:)
- Any case where SMTP header synthesis would contaminate the test

For tests where header precision doesn't matter, use `Transport.send()` via SMTP — it's simpler and more representative of real delivery.

## Why This Is Non-Obvious

The `GreenMailExtension` JUnit 5 wrapper does not expose `deliver()` or `getGreenMail()` — see GE-20260529-aa8445. This technique only works when you hold a `GreenMail` (bare class) instance directly, which is the case in `@QuarkusTestResource` implementations where you control the lifecycle. The `getManagers()` / `getImapHostManager()` / `getInbox()` call chain is not documented; it requires reading the Greenmail source or the `GreenMail` class javap output.
