### Context
A JAX-RS endpoint needs to call an async engine method that returns `CompletionStage<T>` (e.g. `CaseHub.startCase()` returning `CompletionStage<UUID>`), transform the result, and return an HTTP response — without blocking a Vert.x event thread.

### What developers normally do
Annotate the method with `@Blocking` and call `.toCompletableFuture().get()` — or use `Uni<Response>` and unwrap manually. Both either block a thread or require Mutiny familiarity.

### Technique
Return `CompletionStage<Response>` directly from the JAX-RS method. RESTEasy Reactive natively suspends the HTTP request on the event loop and resumes when the stage completes — no `@Blocking`, no `.get()`, no thread held:

```java
@POST
@Path("/cases/researcher")
public CompletionStage<Response> startResearcher() {
    if (engine.isUnsatisfied()) {
        return CompletableFuture.completedFuture(
            Response.status(Response.Status.SERVICE_UNAVAILABLE)
                .entity(Map.of("error", "engine not available"))
                .build());
    }
    return engine.get()
        .startCase()
        .thenApply(caseId -> Response.accepted(new CaseStartedResponse(caseId)).build());
}
```

For the absent-engine 503 path, `CompletableFuture.completedFuture(...)` returns an already-resolved stage — RESTEasy Reactive handles it the same way, completing synchronously.

### Why this is non-obvious
`CompletionStage<Response>` as a JAX-RS return type is not prominent in Quarkus docs — examples focus on `Uni<Response>` (Mutiny) or blocking endpoints. The pattern works because RESTEasy Reactive checks the return type at augmentation time and generates a non-blocking handler for `CompletionStage`. No annotation, no framework import — just the JDK type.
