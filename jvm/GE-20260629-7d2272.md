## Quarkus EntityManager returns stale cached entities with TxType.SUPPORTS and no ambient transaction

### Symptom
In an OCC retry loop: save() commits successfully (incrementing @Version to N+1),
but the next find() returns the entity at version N. The retry re-applies the same
stale context and hits the same conflict — infinite retry loop.

### Context
A JPA repository method annotated @Transactional(TxType.SUPPORTS) is called by an
@ApplicationScoped bean with no @Transactional annotation. The save() method uses
TxType.REQUIRED (creates its own transaction). After save() commits, the next find()
call with SUPPORTS runs without a transaction.

### Root cause
With TxType.SUPPORTS and no ambient transaction, the Quarkus CDI EntityManager proxy
does not create a new persistence context. The first-level cache from a previous operation
in the same thread may still contain the old entity state. The SELECT may return the
cached version instead of re-reading from the database.

With TxType.REQUIRED, each call gets its own JTA transaction and a fresh persistence
context. The SELECT always hits the database and sees the latest committed state.

### Fix
Change find() from @Transactional(TxType.SUPPORTS) to @Transactional(TxType.REQUIRED):

```java
@Override
@Transactional(TxType.REQUIRED)  // not SUPPORTS — ensures fresh persistence context
public Uni<Optional<SituationContext>> find(String situationId, String correlationKey,
                                             String tenancyId) {
    // SELECT always sees latest committed state
}
```

The overhead is trivial — a read-only transaction that commits immediately after the SELECT.

### Why this is non-obvious
SUPPORTS is the textbook choice for read-only methods — "join a transaction if one exists,
run without one otherwise." The implicit assumption is that running without a transaction
still gives you a fresh database read. The EntityManager first-level cache violates this
assumption silently — no error, no warning, just stale data.
