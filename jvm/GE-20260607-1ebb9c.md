### Symptom

A `@PostConstruct` method calls an injected service method annotated with `@Transactional`. At runtime:
- No error or exception is thrown
- The transaction is silently never started
- Database operations inside the service method fail with "no EntityManager associated" or run outside a transaction, silently committing or rolling back contrary to expectations
- Annotating `@PostConstruct` itself with `@Transactional` also has no effect

### Context

Occurs in any Quarkus `@ApplicationScoped` (or other normal-scoped) CDI bean whose `@PostConstruct` method invokes a service that requires a JTA transaction. Classic example: bootstrapping a channel or configuration record in a qhorus `ChannelService.create()` call during bean initialisation.

### Root Cause

Quarkus CDI (`Arc`) implements `@Transactional` via a CDI proxy subclass. When `@PostConstruct` executes, the proxy has not yet been created — the bean is being initialised. The call is made directly on the concrete instance, bypassing the proxy entirely. The `@Transactional` interceptor never fires.

This is not a bug in Arc: the CDI specification requires `@PostConstruct` to execute on the bean instance, not through the proxy. `@Transactional` only works through proxy dispatch.

### Fix

Wrap the transactional work in `QuarkusTransaction.requiringNew()`:

```java
import io.quarkus.narayana.jta.QuarkusTransaction;

@PostConstruct
void init() {
    QuarkusTransaction.requiringNew().call(() -> {
        channelService.findByName(CHANNEL_NAME)
            .map(c -> c.id)
            .orElseGet(() -> channelService.create(...).id);
        return null;
    });
    // Other non-transactional init...
    activeTypes = computeActiveTypes(consumers);
}
```

`QuarkusTransaction.requiringNew()` starts a real JTA transaction programmatically on the current thread. The transaction commits (or rolls back on exception) when the lambda returns. This does not require a proxy.

### Why this is non-obvious

Developers naturally annotate `@PostConstruct` with `@Transactional` (it compiles fine), or rely on injected services being `@Transactional` to "just work" when called from `@PostConstruct`. Neither approach produces any error — the annotation is present, the service is injected, the method call executes — but the transaction silently never starts. The symptom only surfaces as unexpected database behaviour (missing data, constraint violations, or EntityManager-not-bound errors) rather than a clear "no transaction" message.

### Why this fix

`QuarkusTransaction.requiringNew()` is the programmatic API specifically designed for contexts where CDI interception is unavailable — tests, listeners, and lifecycle methods. It is unambiguous about starting a new transaction and does not depend on proxy dispatch.

### Alternatives considered

- Annotating `@PostConstruct` with `@Transactional` — no effect; proxy not created yet
- Self-injecting the bean via `@Inject Instance<Self>` and calling through the proxy — works but awkward and misleading in an init method
- Moving init logic to a `@Observes StartupEvent` method — works and the proxy IS active at that point; valid alternative but changes when init runs (after full CDI startup vs during bean creation)
