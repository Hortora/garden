### Symptom
WorkItem is created successfully (handler logs confirm creation) but `workItemStore.scanAll()` returns an empty list. No error, no warning — the WorkItem silently vanishes from query results.

### Context
Occurs in `@QuarkusTest` suites using `InMemoryWorkItemStore` (the `@Alternative @Priority(1)` test implementation). The test creates a WorkItem via `WorkItemCreator.create(request)` with `tenancyId` set explicitly on the `WorkItemCreateRequest`, then queries with `workItemStore.scanAll()`.

### What was tried (didn't work)
- Verified the WorkItem was created (handler log confirmed)
- Checked CDI wiring (WorkItemSpiAdapter correctly resolved)
- Verified InMemoryWorkItemStore was the active alternative

### Root cause
`InMemoryWorkItemStore` has an asymmetry between `put()` and `scan()`:

- `put()` stamps `tenancyId` from `currentPrincipal.tenancyId()` **only when the WorkItem's tenancyId is null**
- `scan()` (and therefore `scanAll()`) filters results by `currentPrincipal.tenancyId()` **always**

When the `WorkItemCreateRequest` sets `tenancyId` explicitly (e.g. `"test-tenant"`), the value flows through `WorkItemService.create()` to the entity. `put()` sees a non-null tenancyId and does not override it. But `scan()` filters by whatever `currentPrincipal.tenancyId()` returns — which may be a different value (e.g. the platform default tenant).

Result: the WorkItem is in the store's `ConcurrentHashMap` but filtered out by every query.

### Fix
Do not set `tenancyId` on the `WorkItemCreateRequest` in test contexts. Let `InMemoryWorkItemStore.put()` stamp it from `currentPrincipal`. This ensures `put()` and `scan()` use the same tenant value.

```java
// BAD — tenancyId may not match currentPrincipal
WorkItemCreateRequest.builder()
    .title("Review")
    .tenancyId(event.tenancyId())  // explicit — bypasses put() stamp
    .build();

// GOOD — let the store stamp tenancyId from currentPrincipal
WorkItemCreateRequest.builder()
    .title("Review")
    // no .tenancyId() — put() stamps from currentPrincipal
    .build();
```

### Why this is non-obvious
The old code path (direct entity mutation + `workItem.persist()`) never hit this because `persist()` goes to JPA/Panache, not `InMemoryWorkItemStore.put()`. The tenancy filtering only matters when both creation and querying go through the in-memory store — which happens when migrating from runtime service calls to SPI calls (`WorkItemCreator`).

### Alternatives considered
- Setting `currentPrincipal.tenancyId()` to match the explicit value — possible but requires test infrastructure changes and couples tests to tenant identity
- Removing the tenancy filter from `InMemoryWorkItemStore.scan()` — would break multi-tenant test isolation
