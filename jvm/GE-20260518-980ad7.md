## Maven SNAPSHOT rebuilt by subagent mid-session causes NoSuchMethodError at test runtime despite clean compile

**ID:** GE-20260518-980ad7
**Stack:** Maven 3.x, Java 21, Quarkus 3.x

**Symptom:** Tests fail at runtime with `NoSuchMethodError: 'void SomeClass.<init>(arg1, arg2, ..., argN)'` — the exact constructor you call in source code. The compile completed successfully. The method/constructor clearly exists (or javap said it did).

**Context:** A Claude subagent or a parallel session rebuilt a SNAPSHOT dependency and installed it to the shared local Maven repo (`~/.m2`) mid-session. The parent session had already compiled against the old SNAPSHOT artifact; the new artifact has a different constructor signature (a record gained or lost fields). The Quarkus test classloader loads the new artifact at test runtime; the compiled bytecode references the old signature.

### Root cause

Maven SNAPSHOT artifacts in the local repo are mutable. When a subagent runs `mvn install` on a dependency (e.g. `casehub-work`), it overwrites `~/.m2/repository/<group>/<artifact>/<version>/artifact.jar` in place. The **compiler** used a cached `.class` output from before the rebuild — or compiled successfully against one artifact, but the **Quarkus test classloader** loads the jar at the start of the test JVM, picking up the newly installed version.

The symptom is indistinguishable from "NoSuchMethodError due to classpath mismatch" — except no classpath change was intentional.

Compounding: `javap` on the jar in `~/.m2` shows 23 constructor params, but the compiler says "required: 22" — because they ran at different points in time against different versions of the jar. The diagnostics are inconsistent.

### Fix

1. **Check if a dependency was rebuilt mid-session** — look for a subagent commit that says "rebuild" or `mvn install` on a dependency.

2. **Force recompile** to pick up the new artifact:
   ```bash
   mvn compile -pl runtime --batch-mode
   ```
   If the compile now fails with "argument lists differ in length", your source code references the old signature — update the call to match the new artifact.

3. **Count constructor params from the fresh artifact**:
   ```bash
   cd /tmp && jar xf ~/.m2/repository/<group>/<artifact>/<version>/artifact.jar path/to/Class.class && javap path/to/Class.class | grep "public <ClassName>"
   ```
   Run both `jar xf` and `javap` in a single Bash command (see gotcha: Bash CWD resets between calls). Count commas in the constructor signature carefully — the compile error "required: N" is authoritative over javap output if they disagree.

4. **Update the call site** to match the current artifact's constructor arity.

### Why this is non-obvious

Compile-time success is a strong signal that the dependency contract is satisfied. Developers expect compile + test to use the same artifact version. The SNAPSHOT mechanism makes the artifact mutable — a "successful" compile against version X can produce bytecode that fails against version X (same version string, different content) installed 30 seconds later by another process. The error message looks like a classpath or dependency order problem, not a mid-session artifact replacement.
