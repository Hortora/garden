## Symptom

A module has two named Surefire executions — one targeting only PostgreSQL/Testcontainers
tests (`<includes>**/PostgresDialect*.java</includes>` with `db-kind=postgresql`), another
for H2 tests (`<excludes>**/PostgresDialect*.java</excludes>` with `db-kind=h2`).

Running `mvn -pl <module> test -Dtest=JpaMemoryStoreTest` causes `JpaMemoryStoreTest` to
run under BOTH executions. The postgres-dialect-test execution receives it with the PostgreSQL
JDBC driver but no Testcontainers resource, producing:

```
Driver does not support the provided URL: jdbc:h2:mem:memorytest;DB_CLOSE_DELAY=-1;MODE=PostgreSQL
```

or startup failure when Docker is unavailable. `BUILD FAILURE` is reported for the
postgres-dialect-test phase; the correct h2-tests execution may never run.

## Root Cause

`-Dtest=` is a **global Surefire filter** applied across all plugin executions in the Maven
lifecycle. It does not AND with an execution's `<includes>`/`<excludes>` XML — it replaces
the effective test set for every execution. An execution that would never see `JpaMemoryStoreTest`
under normal lifecycle runs will receive it when `-Dtest=JpaMemoryStoreTest` is given.

## Fix

**Option 1 — Target a specific named execution by ID:**

```bash
mvn -pl <module> surefire:test@h2-tests
```

The `surefire:test@<execution-id>` mojo invocation runs exactly one named execution, ignoring
all others.

**Option 2 — Let the lifecycle run all executions with fail-at-end:**

```bash
mvn -pl <module> test -fae
```

Each execution uses its own `<includes>`/`<excludes>` correctly. The postgres-dialect-test
execution only picks up `PostgresDialect*.java`; JpaMemoryStoreTest runs only in h2-tests.
Use `-fae` so a Docker-unavailable failure in one execution does not prevent the H2 results
from being collected.

## Why This Is Non-Obvious

`-Dtest=` reads as "run only this class." Nothing in the Surefire documentation warns
that it overrides execution-level include/exclude patterns or injects the class into
executions whose configuration would normally exclude it. The symptom — a wrong-driver
`SQLException` — gives no indication that a test-filter flag was responsible.
