### Symptom

A `switch` statement with at least one type-pattern arm (`case MyType x ->`) and a `default ->` arm throws `NullPointerException` at runtime when the switch expression evaluates to `null`. No compile-time error or warning is produced.

```java
// Crashes with NPE when binding.target() returns null
switch (binding.target()) {
    case CapabilityTarget ct -> { /* ... */ }
    default -> { continue; }  // does NOT match null
}
```

### Context

Triggered whenever:
1. The switch uses at least one type-pattern arm (Java 21 pattern-matching switch mode)
2. The expression being switched on can be `null`
3. There is no explicit `case null` arm

In test code, this often appears when mocking dependencies that return `null` for a field, then passing the mocked object through production code that switches on that field. The code compiles and runs correctly for all non-null values.

### Root Cause

Java 21 changed the null-handling semantics of switch when pattern-matching arms are present. Traditional switch (no type patterns) throws NPE for null at the switch statement itself, but never reaches the arms. Pattern-matching switch moves null rejection into a new rule: **`null` is not matched by `default`** unless `case null` is also declared. The JLS section on switch expressions/statements specifies this explicitly, but the distinction is easy to miss when migrating from traditional switch to pattern switch.

### Fix

Add an explicit `case null` arm. The `case null, default` form combines both:

```java
switch (binding.target()) {
    case CapabilityTarget ct -> { /* proceed with capability logic */ }
    case null, default -> { continue; }  // matches both null and all other types
}
```

Or use separate arms if different handling is needed:

```java
switch (binding.target()) {
    case CapabilityTarget ct -> { /* ... */ }
    case null -> { /* handle null target explicitly */ }
    default -> { /* handle non-null non-capability targets */ }
}
```

### Why this is non-obvious

Traditional `switch` catches are deeply muscle-memoried: `default` means "everything else". Pattern-matching switch looks syntactically identical when mixed with type patterns — there is no keyword change, no new syntax to signal the different semantics. The compiler gives no warning. The failure appears at runtime with a plain `NullPointerException` at the switch statement, not inside any arm, making the diagnosis non-obvious. Developers familiar only with traditional switch will assume the `default` arm caught it.
