### Context
When implementing subprocess spawning for WorkItems, children needed to be scoped to their specific spawn group for two operations: idempotent lookup (return existing children when replaying a spawn call) and cascade cancellation (cancel only THIS group's children, not all children of the parent).

The obvious solution — adding a `spawn_group_id` FK column to the `work_item_relation` table — requires a Flyway migration and a schema change.

### Technique
Encode the group UUID in an existing string field (`createdBy`) using a structured prefix:

```java
rel.createdBy = "system:spawn:" + group.id;
```

Then filter by exact match when scoping operations to the group:

```java
final String marker = "system:spawn:" + groupId;
WorkItemRelation.findByTargetAndType(parentId, PART_OF)
    .stream()
    .filter(r -> marker.equals(r.createdBy))
    ...
```

This gives deterministic child-to-group membership without any schema change. The prefix `"system:spawn:"` makes the intent readable in the data; the UUID suffix makes it exact and collision-free.

### Why this is non-obvious
The instinct is to add a proper FK column — that's the "correct" relational design. But when the owning entity (`WorkItemSpawnGroup`) is new and the relation table (`work_item_relation`) already exists, avoiding a schema change by reusing an existing string field is both simpler and fully functional. The approach also works for cascade scoping: you're not guessing by timestamp or parent — you're filtering by an exact structured key.

### Constraints
- Only works cleanly when the string field has no other strong semantic meaning (or when the structured prefix distinguishes it clearly)
- The prefix convention must be documented and stable — consumers that filter by `createdBy.startsWith("system:")` will pick this up
- Not suitable when the field is indexed and the encoding would defeat the index

### When to prefer the FK column instead
- When the membership relation is many-to-many and needs to be queried in both directions
- When you need foreign key constraints for referential integrity
- When the group membership itself needs to be paginated or aggregated efficiently
