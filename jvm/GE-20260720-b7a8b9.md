### Symptom
Second CBR case stored for the same entity silently wipes the first domain's case. `ClinicalCaseOutcomeObserverIntegrationTest` fails with "Expecting actual not to be empty" on retrieval — the `clinical-ae` case vanishes after the `clinical-ae-trajectory` case is stored.

### Context
`ClinicalCaseOutcomeObserver.handleAeCase()` stores the same AE in two CBR domains:
1. `clinical-ae` — point-in-time features (grade, eventType, etc.)
2. `clinical-ae-trajectory` — TimeSeries trajectory features

Both calls use `ClinicalCbrService.storeIdempotent()` with the same `entityId` (aeId).

### Root cause
`storeIdempotent()` calls `store.eraseEntity(entityId, tenantId)` before `store.store()`. `eraseEntity` takes only `(entityId, tenantId)` — no domain parameter. It erases ALL cases for that entity across every domain. The second store call for `clinical-ae-trajectory` erases the `clinical-ae` case just stored.

### Fix
Use a suffixed entityId for each domain to avoid cross-domain erasure collision:
```java
// First domain — point-in-time features
cbrService.storeIdempotent(cbrCase, "clinical-ae", aeId.toString(),
    ClinicalCbrDomains.AE, ae.tenantId, caseId);

// Second domain — trajectory features (suffixed entityId)
cbrService.storeIdempotent(trajCase, "clinical-ae-trajectory", aeId + "-trajectory",
    ClinicalCbrDomains.AE_TRAJECTORY, ae.tenantId, caseId);
```

### Why this is non-obvious
The method name `eraseEntity` and its parameters `(entityId, tenantId)` strongly suggest entity-scoped erasure. Nothing in the API signature hints that it crosses domain boundaries. The failure mode is silent — no error, no warning — the first domain's case simply vanishes.
