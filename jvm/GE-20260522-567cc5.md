## Symptom

`gateway.listBackends(channelUuid)` returns an empty list in a test, even though
the backend was registered for the channel seeded in `@BeforeEach`. The channel
is present in `InMemoryChannelStore`, and the backend registration call succeeded.

## Context

`@QuarkusTest` with `@Inject InMemoryChannelStore channelStore`. In `@BeforeEach`:

```java
UUID channelUuid = UUID.randomUUID();
Channel ch = new Channel();
ch.name = "case-abc/work";
ch.semantic = ChannelSemantic.APPEND;
channelStore.put(ch);  // <-- ch.id is null
```

Later in the test:
```java
gateway.listBackends(channelUuid);  // returns empty — never matches
```

## Root cause

`InMemoryChannelStore.put(ch)` auto-assigns a new random UUID to `ch.id` if it is
`null`. The `channelUuid` variable in the test therefore does not match the UUID
that was assigned to the stored channel. The mismatch is silent — `listBackends()`
returns an empty list rather than throwing.

## Fix

Set `ch.id` explicitly before calling `put()`:

```java
UUID channelUuid = UUID.randomUUID();
Channel ch = new Channel();
ch.id = channelUuid;          // <-- set explicitly
ch.name = "case-abc/work";
ch.semantic = ChannelSemantic.APPEND;
channelStore.put(ch);
```

## Why this is non-obvious

`InMemoryChannelStore` is a test double, not production code. Its auto-assign
behaviour is an implementation convenience (mirrors what the real persistence
layer does). The behaviour is not documented — you discover it by tracing through
the test store source code. The failure mode (empty listBackends result) gives
no indication that the channel UUID doesn't match.
