## Write correlation cache BEFORE dispatching to message handlers — prevents response-arrives-before-cache race

**ID:** GE-20260618-af5596
**Stack:** Java, any async event-driven system
**Symptom:** Responses arrive as new top-level messages instead of replies in the expected thread/context — or equivalent: a handler cannot find correlation state it expects to exist, causing a new root record to be created when a reply was intended.
**Context:** Any system where: (1) an inbound message generates a new correlation ID, (2) that correlation ID is written to a cache/store, (3) the message is then dispatched to a handler, and (4) the handler may respond before the cache write commits.

### What was tried (didn't work)
Writing the cache entry AFTER calling the dispatch/gateway method:

```java
// ❌ WRONG — response from handler may arrive before cache is populated
gateway.receiveHumanMessage(channelRef, new InboundHumanMessage(..., corrId, null));
threadCache.put(channelId, corrId, slackTs);  // too late
```

This works in integration tests (sequential) but fails in production under low latency or when the handler runs on a different thread pool.

### Root cause
The dispatch call crosses a thread boundary (CDI event, virtual thread, executor). The response handler is dispatched immediately with the correlation ID, looks up the cache to find the thread context, finds nothing, and creates a new root record (new Slack thread instead of a reply).

### Fix
Write the cache entry — both in-memory and to the persistent store — BEFORE calling the dispatch method:

```java
// ✅ CORRECT — cache populated before handler can respond
if (slackTs != null) {
    threadCacheStore.save(channelRef.id(), corrId, slackTs);           // DB write first
    threadCache.computeIfAbsent(channelRef.id(), k -> new ConcurrentHashMap<>())
               .put(corrId, slackTs);                                   // in-memory write second
}
gateway.receiveHumanMessage(channelRef,                                 // dispatch LAST
    new InboundHumanMessage(senderId, content, receivedAt, metadata, corrId.toString(), null));
```

The DB write must be committed (not in an open transaction) before dispatch if the handler runs on a separate transaction context. If the dispatch is within the same transaction, an in-memory write suffices.

### Why this is non-obvious
The natural implementation is write-on-response (when the response arrives, populate the cache). The pre-populate approach requires thinking about what the handler needs to find in the cache before it even starts processing, which requires tracing the response path before writing the request path. The race is invisible in tests because test dispatch is synchronous.
