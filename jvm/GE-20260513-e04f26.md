## Store configuration on the runtime entity to eliminate a parallel callback registry

**ID:** GE-20260513-e04f26
**Stack:** Quarkus CDI, Java 21
**Labels:** `#design` `#event-driven` `#state-management`
**What it achieves:** Eliminates a CDI singleton registry bean by storing callback configuration directly on the existing domain entity, making it restart-safe and lifecycle-correct automatically.
**Context:** An event handler needs to look up configuration at callback/completion time ŌĆö e.g. a `WorkItemLifecycleAdapter` needs the `outputMapping` expression when a WorkItem completes. The configuration was set at schedule time and must be retrieved at completion time.

### The technique

Store the configuration on the existing domain entity rather than in a parallel CDI registry.

**Before (parallel registry approach):**
```java
// CDI singleton ŌĆö separate parallel state
@ApplicationScoped
class HumanTaskRegistry {
    private final Map<String, HumanTaskTarget> registry = new ConcurrentHashMap<>();

    public void register(UUID caseId, String planItemId, HumanTaskTarget target) {
        registry.put(key(caseId, planItemId), target);
    }

    public HumanTaskTarget get(UUID caseId, String planItemId) {
        return registry.get(key(caseId, planItemId));
    }
}
```

**After (entity-stored approach):**
```java
// Configuration lives on the entity ŌĆö accessible wherever the entity is
public class PlanItem {
    private final BindingTarget target;  // stored at creation time

    public BindingTarget target() { return target; }
}

// At callback time ŌĆö no registry needed
void onWorkItemLifecycle(WorkItemLifecycleEvent event) {
    CallerRef ref = CallerRef.parse(workItem.callerRef);
    CasePlanModel plan = registry.get(ref.caseId()).orElse(null);
    PlanItem planItem = plan.getPlanItem(ref.planItemId()).orElse(null);

    if (planItem.target() instanceof HumanTaskTarget t && t.outputMapping() != null) {
        Map<String, Object> output = t.outputMapping().evaluate(workItem.resolution);
        instance.getCaseContext().putAll(output);
    }
}
```

### Alternatives considered
- CDI singleton `ConcurrentHashMap<K, Config>` ŌĆö explicit and familiar, but introduces parallel state management, restart durability concerns (state lost on restart), and memory leaks if cleanup is missed
- Storing config in the WorkItem metadata/labels ŌĆö durable, but leaks engine-internal configuration into the casehub-work domain model (wrong layer)

### Why this is non-obvious
When a callback handler needs configuration from registration time, the instinct is a CDI registry bean. This is the standard pattern for correlation in event-driven systems and works correctly. The insight is that if a stable domain entity already tracks the same lifecycle, the entity IS the registry ŌĆö storing config on it eliminates an entire CDI bean, makes the state restart-safe automatically, and makes the configuration available for all entity instances without explicit lifecycle management.

The pattern also generalises: in this case, storing `BindingTarget` on `PlanItem` (instead of just `planItemId` and `workerName`) made `outputMapping` available for ALL binding types, not just `HumanTaskTarget` ŌĆö a capability improvement that a separate registry could not have provided without additional per-type registries.

### When to use it
- A callback handler needs configuration that was set at registration/schedule time
- A stable domain entity tracks the same lifecycle as the configuration (both are created together and live as long)
- The entity is accessible at callback time via an existing lookup path
- The configuration is inherently per-entity-instance (not shared across instances)

Do not use when the entity may be evicted or GC'd before the callback fires, or when the configuration spans multiple unrelated entity types.
