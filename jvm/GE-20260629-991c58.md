## Bifurcated claim path — claim-before-save for existing vs save-before-claim for new

### What it achieves

Exactly-once trigger semantics with correct expiry behaviour, regardless of whether the entity exists at claim time.

### Context

A conditional JPQL UPDATE (`SET policyTriggered = true WHERE policyTriggered = false`) provides exactly-once claim semantics but requires the entity row to exist. Two naive approaches fail:

1. **Always save-before-claim:** Creates the row for the claim, but `save()` updates `lastSignal` via the domain model (`SituationContext.withDetection()` → `SituationMapper.updateEntity()`). For post-trigger events, this refreshes `lastSignal` on every event, preventing expiry — the entity never becomes stale enough for `removeExpired()` or `isExpired()` to clean it up.

2. **Always claim-before-save:** The conditional UPDATE returns 0 rows when the entity doesn't exist, which is indistinguishable from "already claimed" (see GE-20260629-6928ab).

### The technique

Branch on `storeVersion` — the domain model carries an `OptionalLong` that is empty for newly created contexts and present for contexts loaded from the store:

```java
if (context.storeVersion().isPresent()) {
    // Existing entity — claim first, save after (avoids lastSignal refresh on claim failure)
    boolean claimed = store.tryClaimTrigger(situationId, correlationKey, tenancyId).await().indefinitely();
    if (!claimed) return true;
    store.save(context).await().indefinitely();
} else {
    // New entity — save first to create the row, then claim
    store.save(context).await().indefinitely();
    boolean claimed = store.tryClaimTrigger(situationId, correlationKey, tenancyId).await().indefinitely();
    if (!claimed) return true;
}
```

For **existing entities**: claim fails → return immediately (no save, no `lastSignal` refresh). The entity retains its original `lastSignal` and expires on schedule.

For **new entities**: save creates the row → claim operates on the newly created row. The `lastSignal` refresh is the first event's timestamp — correct since there's no prior timestamp to preserve.

### Why this is non-obvious

The interaction between the save operation's side effects (updating `lastSignal`) and the expiry mechanism (`removeExpired(cutoff)` checking `lastSignal`) is invisible until continuous post-trigger events prevent cleanup. The fix requires recognizing that the claim and save serve different purposes at different entity lifecycle stages, and ordering them differently based on entity existence.
