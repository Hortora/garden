### Symptom
`@Retry(maxRetries=3)` never retries. `@Fallback` fires immediately after the first attempt.
No exception is logged even when the connect method has `catch (Exception e) { log(...); throw e; }`.
The log shows only the fallback message, never the exception type.

### Context
Using ocraft on Mac inside a Quarkus CDI bean with MicroProfile Fault Tolerance annotations.
`S2Coordinator.setup().loadSettings(new String[]{})` fails (due to the Mac path separator bug —
see related entry), and `tryDiscoverDefaultSettings()` calls `System.exit(0)`.

### Root Cause
```java
// S2Coordinator.java in ocraft-s2client-api 0.4.21
private Map<String, Object> tryDiscoverDefaultSettings(String[] args) {
    try {
        exeSettings = ExecutableParser.loadSettings(null, null, null);
    } catch (Exception e) {
        if (args.length == 0) {
            System.err.println("Please run StarCraft II before running this API");
            System.exit(0);   // ← this, not a throw
        }
    }
}
```

Quarkus dev mode intercepts `System.exit()` and throws a special marker class — not a
subclass of `Exception` — to prevent the JVM from actually exiting. This marker:
- Bypasses `catch (Exception e)` (only catches Exception, not all Throwable)
- Is not retried by MicroProfile `@Retry` (which by default retries `Exception`, not `Error`)
- Does trigger `@Fallback` (which catches all Throwable)

### Fix
Two parts:
1. Fix the root cause (Mac path separator bug) so `loadSettings()` doesn't fail
2. Use `catch (Throwable e)` in your wrapper to at least log the real type:
```java
} catch (Throwable e) {
    log.errorf("[SC2] Connection failed: %s (type=%s)", e.getMessage(), e.getClass().getName());
    throw new RuntimeException("wrapped", e);
}
```

Do NOT call `loadSettings(new String[]{})` — use `setProcessPath()` and the system property
approach instead (set `ocraft.game.executable.baseBuild` before ocraft init).

### Why This Is Non-Obvious
- `System.exit()` in a library is unexpected — it's not in ocraft's documented failure modes
- The Quarkus dev-mode interception makes it look like a clean exit, not an error
- `catch (Exception e)` is the standard Java pattern; catching `Throwable` for library code
  is unusual and shouldn't be necessary
- The `@Retry`/`@Fallback` interaction gives the impression of one attempt being made when
  in reality the method completed (via the exit interceptor) rather than throwing
