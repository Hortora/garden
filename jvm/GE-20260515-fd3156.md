### What you observe

Placing `@DefaultBean` on the producer class rather than the `@Produces` method makes the *producer bean itself* a default bean — but the *produced bean* (the type returned by `@Produces`) remains a regular non-default `@ApplicationScoped` bean. A consumer-provided `@ApplicationScoped` bean of the same type then conflicts, causing CDI ambiguity at boot.

### Context

Any Quarkus ARC application using a CDI producer (`@Produces`) to supply a bean that should yield automatically to consumer-provided overrides.

### What was tried

```java
// WRONG — @DefaultBean on the class makes the PRODUCER BEAN default,
// not the produced ContextDiffStrategy bean
@DefaultBean
@ApplicationScoped
class ContextDiffStrategyProducer {
    @Produces
    @ApplicationScoped
    ContextDiffStrategy produce() { ... }
}
```

With this, a consumer's `@ApplicationScoped ContextDiffStrategy` causes CDI ambiguity — the produced bean is not a default bean and does not yield.

### Root cause

`@DefaultBean` applies to the bean it directly annotates. On a producer *method*, it applies to the produced bean. On the producer *class*, it applies to the producer class bean itself — not to what the producer produces.

### Fix

Place `@DefaultBean` on the `@Produces` method, not the class:

```java
@ApplicationScoped
class ContextDiffStrategyProducer {

    @Produces
    @DefaultBean         // ← on the method, not the class
    @ApplicationScoped
    ContextDiffStrategy produce() { ... }
}
```

The produced `ContextDiffStrategy` is now a default bean. Any consumer `@ApplicationScoped ContextDiffStrategy` wins automatically without configuration.

### Why this is non-obvious

The `@DefaultBean` annotation is Quarkus ARC-specific (`io.quarkus.arc.DefaultBean`) and its producer-method vs producer-class placement is not explained in the Quarkus guides. The difference only surfaces as a CDI ambiguity error when a consumer tries to override the produced bean — the error doesn't point to the placement mistake.
