## BroadcastProcessor.onNext() throws BackPressureFailure when no subscribers are registered

### Symptom

`BroadcastProcessor.onNext(payload)` throws `io.smallrye.mutiny.infrastructure.BackPressureFailure` (unchecked exception) propagating up to the calling thread and crashing the caller. No return value or null — an exception.

### Root cause

`BroadcastProcessor` is designed for push-based streaming. When there are no active subscribers, it has nowhere to deliver the item and throws `BackPressureFailure` rather than silently discarding or buffering. This is by design — the exception signals that the stream has no consumers.

### Fix

Wrap `onNext()` calls in a try-catch that silently discards `BackPressureFailure`:

```java
try {
    processor.onNext(payload);
} catch (Exception e) {
    // BackPressureFailure — no subscribers connected, discard silently
    Log.debugf("No subscribers for %s broadcast", payload.getClass().getSimpleName());
}
```

### Why this is non-obvious

The API looks like a simple publish method — most publish/emit APIs silently drop items when there are no listeners. The unchecked exception with a Mutiny-internal type name is unexpected and the fix (catch and discard) feels wrong but is correct.
