## Symptom

CI cannot resolve a dependency at the expected version — artifact downloads fail or resolve to a stale SNAPSHOT. The obvious diagnosis is that the publishing pipeline hasn't caught up.

## Context

The Maven `settings.xml` or project `pom.xml` declares two repository entries pointing to GitHub Packages: a primary (`github`) with correct authentication, and a secondary (e.g. `github-casehubio`) with a misconfigured or missing token. Maven logs show `status code: 401, reason phrase: Unauthorized (401)` on the secondary entry — but these appear as WARNINGs rather than ERRORs. The build then fails on a dependency that lives in the secondary repo.

Observed in casehub-ops: the `github-casehubio` server entry lacked credentials while the `github` entry was correctly authenticated. All `casehub-iot-api`, `casehub-ledger-api`, `casehub-platform-api`, and other SNAPSHOTs that the secondary entry was supposed to serve returned 401, causing resolution to fall back to a stale cached version.

## Root Cause

Maven resolves SNAPSHOT metadata from all configured repositories and caches failures. If a secondary repo returns 401 on a SNAPSHOT metadata request, Maven marks that resolution as failed for the cache interval and uses the last-known version. The build error names the dependency and a version number — which looks exactly like a publish timing issue. The actual failure is authentication configuration on a repository entry the developer assumed was read-only or advisory.

## Fix

Audit all Maven repository server entries in `settings.xml` and the project's `pom.xml`:

```xml
<!-- settings.xml — ensure both server IDs have credentials -->
<servers>
  <server>
    <id>github</id>
    <username>${env.GITHUB_ACTOR}</username>
    <password>${env.GITHUB_TOKEN}</password>
  </server>
  <server>
    <id>github-casehubio</id>  <!-- ← this entry must also have credentials -->
    <username>${env.GITHUB_ACTOR}</username>
    <password>${env.GITHUB_TOKEN}</password>
  </server>
</servers>
```

Alternatively, remove the duplicate repository entry if both point to the same GitHub Packages namespace — one authenticated entry is sufficient.

## Why This Is Non-Obvious

The `401 Unauthorized` messages appear as Maven WARNINGs in the CI log, buried between INFO lines about successful downloads from the primary repo. The terminal build error names the dependency and version number, which is the exact symptom of a stale publish. Developers immediately check the publish pipeline, find that it ran successfully, and are confused. The auth failure on the secondary repo is never shown as the root cause — it requires scrolling back through the WARNING lines and cross-referencing server IDs.
