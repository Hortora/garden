### Symptom
CI compilation fails with `cannot find symbol` for classes that exist locally
and compile fine on the developer machine. Error points to specific imports
(e.g. `io.casehub.platform.governance.TimeoutPolicyException`).

### Context
Multi-repo Maven project where module A (worker) depends on module B
(platform-governance) via `0.2-SNAPSHOT`. Developer builds module B locally,
adds new classes, then builds module A against the locally-installed SNAPSHOT.
Everything compiles. Developer pushes module A. CI pulls the latest *published*
SNAPSHOT of module B from GitHub Packages — which is an older build that
predates the new classes.

### What was tried (didn't work)
- Checking imports and class names — they're correct
- Local `mvn clean install` — passes because local `.m2` has the newer build
- Checking if the dependency version is correct — it is (`0.2-SNAPSHOT`)

### Root cause
Maven SNAPSHOT resolution is registry-specific. Local `.m2/repository` has
a newer build of the dependency than the remote registry. CI always resolves
from the remote registry, so it gets the stale version. The new classes
literally don't exist in the jar CI downloads.

### Fix
Push and deploy the upstream dependency first. Wait for its CI to complete
`mvn deploy` before pushing the downstream consumer.

In the observed case: platform-governance CI auto-deploys on push to main
and even triggers downstream CI via `repository_dispatch`. The gap was
purely ordering — worker was pushed before platform.

### Why this is non-obvious
The symptom — `cannot find symbol` — points toward a typo, wrong import,
or missing dependency. Nothing in the error suggests a *version* problem,
because the version (`0.2-SNAPSHOT`) is technically correct. The local build
passing reinforces the wrong diagnosis. You have to know that SNAPSHOT
resolution differs between local `.m2` and the remote registry to even
consider this.

### Alternatives considered
- Pre-push SNAPSHOT freshness check (`mvn dependency:resolve -U`): catches
  the problem but adds 5-10s latency per push and blocks intentional
  work-ahead scenarios. Decided against for now.
