### Symptom
`UnsatisfiedResolutionException` or `IllegalStateException` ("Quartz scheduler is either explicitly disabled...") for `org.quartz.Scheduler` injection in modules that depend on a module containing `WorkItemTimerService` or any bean that programmatically schedules Quartz jobs.

### Context
A Quarkus extension's runtime module injects `org.quartz.Scheduler` to programmatically schedule jobs (e.g. expiry timers). Dependent modules (ledger, notifications, queues, ai) set `quarkus.scheduler.enabled=false` in test `application.properties` to prevent `@Scheduled` methods from auto-firing. When the dependent module's `@QuarkusTest` starts, Quarkus augmentation fails because the `Scheduler` CDI bean doesn't exist.

### What was tried (didn't work)
- `quarkus.scheduler.enabled=false` — the obvious choice, but it's too aggressive
- Excluding the timer service class — breaks other CDI wiring

### Root cause
`quarkus.scheduler.enabled=false` doesn't mean "don't run scheduled methods" — it means "don't create the scheduler infrastructure at all." The `Scheduler` CDI bean is never registered, so any `@Inject Scheduler` fails at augmentation time. The config name is misleading.

### Fix
Use `quarkus.scheduler.start-mode=halted` instead:

```properties
# Wrong — removes the Scheduler bean entirely
quarkus.scheduler.enabled=false

# Right — keeps the bean available, prevents @Scheduled methods from auto-firing
quarkus.scheduler.start-mode=halted
```

`start-mode=halted` registers the `Scheduler` bean but doesn't start the Quartz scheduler. Programmatic `scheduler.scheduleJob()` calls work; `@Scheduled` methods don't auto-fire. Tests that need to trigger scheduled methods can start the scheduler explicitly.

### Why this is non-obvious
The property name `enabled=false` reads as "disable the scheduling feature" not "remove the scheduling infrastructure." A developer reasonably expects the bean to exist but be inactive. The actual behaviour — removing the bean entirely — causes failures in transitive dependencies that the developer may not even know inject `Scheduler`.
