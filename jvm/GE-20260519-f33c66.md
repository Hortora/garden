### What is undocumented

A service class annotated `@Alternative` and named with "Reactive" may internally use a `blockingService` field (injected blocking CDI service) for write operations and `Entity.find()` (a Panache static) for read operations. CDI `@Alternative @Priority(1)` in-memory stores only intercept at CDI injection points — Panache statics go directly to the underlying database regardless of what alternatives are selected.

### Context

In Quarkus Qhorus (casehubio/qhorus), `ReactiveQhorusMcpTools`:
- `sendMessage()` delegates internally to `blockingMessageService` → `MessageService.send()` → `messageStore.put()` (uses the `MessageStore` CDI alternative, i.e. `InMemoryMessageStore` in tests)
- `getChannelTimeline()` delegates to `blockingGetChannelTimeline()` → `Message.<Message>find(...)` (Panache static — queries H2 directly, bypasses `MessageStore` entirely)

Result: messages written by `sendMessage()` land in `InMemoryMessageStore` (in-memory). Messages read by `getChannelTimeline()` come from H2 (empty in tests). The two paths never share data.

### Discovery

Only visible by reading the service implementation source. The class name and method signatures suggest a uniform reactive abstraction. Not mentioned in any documentation.

### Impact on tests

Tests that call `sendMessage()` then verify the timeline via `getChannelTimeline()` always see an empty timeline in test setups using `@Alternative` in-memory stores, even though the message was "successfully" posted.

### Fix / Workaround

Identify the actual data path for each operation you need to test:
1. Check which CDI service field the MCP tool method delegates to
2. Check whether that service uses CDI store injection or Panache statics
3. For end-to-end write→read tests: use the real H2 database (don't rely on in-memory stores for both sides)

Alternatively, file an upstream issue to make the read path use the same abstraction as the write path.
