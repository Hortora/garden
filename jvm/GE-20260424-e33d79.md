## Symptom

A subagent fixed `"Unsatisfied dependency for type EntityManager and qualifiers [@Default]"` in a consuming application (Claudony) by adding `@PersistenceUnit("qhorus")` to all `EntityManager` injection points in a generic extension's runtime beans. The error disappeared in Claudony. Every other consumer of the extension then failed with the mirror error: `"Unsatisfied dependency for type EntityManager and qualifiers [@PersistenceUnit("qhorus")]"`.

## Context

Occurs when a Quarkus application configures only a named datasource/persistence unit with no default. CDI then provides only a `@PersistenceUnit("name")` qualified `EntityManager` — the `@Default` qualified one is absent. An extension injecting bare `EntityManager` fails. The tempting fix is to add the named qualifier to the extension's beans.

## Root cause

The fix confuses the symptom (missing `@Default` EntityManager) with the root cause (the extension has no mechanism to select which persistence unit to use — it always assumed the default). Adding a consumer-specific qualifier to a reusable extension is an architectural error: it hard-couples the extension to one consumer's datasource configuration.

## Fix

Add a configurable persistence unit name to the extension's config (e.g. `quarkus.ledger.datasource`, empty = default), a custom CDI qualifier (`@LedgerPersistenceUnit`), and a producer that selects the right `EntityManager` at startup:

```java
@ApplicationScoped
public class LedgerEntityManagerProducer {
    @Inject LedgerConfig config;

    @Produces @LedgerPersistenceUnit
    public EntityManager produce(@Any Instance<EntityManager> instance) {
        String pu = config.datasource().orElse("").trim();
        if (pu.isBlank()) {
            return instance.select(Default.Literal.INSTANCE).get();
        }
        return instance.select(new PersistenceUnitLiteral(pu)).get();
    }
}
```

All extension beans inject `@LedgerPersistenceUnit EntityManager` instead of bare `EntityManager`. Consumers with only a named datasource set `quarkus.ledger.datasource=mydb`; all others need no config change.

## Why this is non-obvious

The error message names the qualifier (`[@Default]`) and the natural fix is to supply that qualifier — or supply the consumer's qualifier. It takes a step back to recognise that the qualifier itself is not the problem; the problem is the extension assuming a default datasource exists. The right fix adds a level of indirection (configurable producer) rather than patching the symptom.

### Why this fix

The qualifier + producer approach is IntelliJ-stable (real CDI annotation, not stripped as an unused import), portable (works with any datasource name), and testable (the producer is a plain CDI bean).

### Alternatives considered

- Annotating all 8 injection points with `@PersistenceUnit("qhorus")`: makes the extension unusable with casehub-engine, quarkus-workitems, or any non-Qhorus consumer
- Documenting "must have a default datasource": too restrictive — named-only datasource setups are valid and common in multi-tenant apps
