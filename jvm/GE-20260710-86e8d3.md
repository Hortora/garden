# H2MVStore get() succeeds on a closed store but put() throws — read/write asymmetry after close()

## Pain (3)

Tests designed to verify "storage read failure" behavior silently pass — `MVMap.get()` returns
cached data even after the store is closed. The test appears to exercise the read-failure path
but actually exercises the normal success path. Discovered only when the expected exception type
doesn't match.

## Non-obvious (3)

`StorageManager.close()` intuitively means "nothing works anymore." H2's MVStore page cache
retains read data after close — `get()` serves from cache, `put()` hits `beforeWrite()` which
checks `checkNotClosed()` and throws `MVStoreException`.

## Reuse (3)

Affects anyone writing failure-mode tests for drools-reliability or direct H2MVStore users who
simulate storage unavailability by closing the store.

## Problem

After calling `StorageManager.close()`, `MVMap.get(key)` returns the cached value without error.
`MVMap.put(key, value)` throws `MVStoreException: This store is closed`. Tests that close the
store to simulate "storage unavailable" only trigger write failures — read operations continue
working from the page cache.

## Root Cause

H2's `MVMap.get()` reads from in-memory pages (the B-tree page cache) and never calls
`checkNotClosed()`. `MVMap.put()` calls `beforeWrite()` → `checkNotClosed()` which throws
if the store is closed. The asymmetry is an implementation detail of H2's write-path validation.

## Fix

Do not use `StorageManager.close()` to test storage read failures. For testing read failure
paths (e.g. `DroolsSessionStoreException` wrapping around `sessionIds.get()`), use a mock
or custom `Storage` implementation that throws on `get()`. The `close()` approach only tests
write-failure paths.

For write-failure testing, `close()` works correctly — `put()`, `remove()`, and other
write operations all throw `MVStoreException` as expected.

## Why this is non-obvious

The natural test approach is "close the store → call the method → assert it throws." This
works for writes but silently passes for reads, creating a false sense of test coverage.
The `get()` returning cached data looks like a successful read, not a degraded one.
