## ChannelGateway.fanOut() Fires backend.post() Before the DB Transaction Commits

### Symptom

A client receives an SSE event (or any push notification) indicating a task is "completed",
polls `GET /resource/{id}` immediately, and gets the previous (non-terminal) state — as if
the event was premature. The actual DB state catches up within milliseconds, but the window
is real and reproducible under load.

### Context

```java
@Transactional
public DispatchResult dispatch(MessageDispatch d) {
    // ... enforcement, persistence ...
    messageStore.put(message);        // persisted inside the transaction
    channelGateway.fanOut(ch.id, ...); // fires post() on a virtual thread
    return result;
}  // transaction commits here, AFTER fanOut() returns
```

`ChannelGateway.fanOut()` implementation:

```java
public void fanOut(UUID channelId, String channelName, OutboundMessage message) {
    for (BackendEntry entry : registry) {
        Thread.ofVirtual().start(() -> {  // ← starts NOW, inside the transaction
            backend.post(ref, message);   // SSE push happens here, pre-commit
        });
    }
}
```

The virtual thread is started immediately when `fanOut()` is called. The JTA transaction
has NOT committed yet — `messageStore.put()` has been called but the DB write is pending.
The virtual thread executes `backend.post()` (which sends SSE events, calls webhooks, etc.)
before the parent thread's `@Transactional` proxy commits.

### Root cause

Java virtual threads are not structured concurrency participants. `Thread.ofVirtual().start()`
starts the thread immediately regardless of any enclosing JTA transaction. The JTA spec has
no mechanism to delay thread start until after commit.

This affects all backends, not just SSE — Kafka producers, HTTP callbacks, and any other
notification dispatched via `Thread.ofVirtual()` inside a `@Transactional` method share
this race.

### Fix (application-side mitigation)

**Option 1 — Client-side retry:** Document the behavior and advise clients to treat push
events as triggers, not guarantees. The client retries the poll with a short backoff after
receiving an event:

```java
// Client: on SSE event received, poll with retry
retryWithBackoff(3, 50ms, () -> pollUntilTerminal(taskId));
```

**Option 2 — Post-commit dispatch:** Use JTA `TransactionSynchronizationRegistry` to delay
`fanOut()` until after commit (already used in `MessageObserverDispatcher`):

```java
tsr.registerInterposedSynchronization(new Synchronization() {
    public void afterCompletion(int status) {
        if (status == Status.STATUS_COMMITTED) {
            channelGateway.fanOut(ch.id, channelName, outbound);
        }
    }
    public void beforeCompletion() {}
});
```

This is the correct fix for any backend where post-commit ordering is required.

### Why this is non-obvious

The natural reading of `Thread.ofVirtual().start(...)` inside a service method is "this
runs concurrently with the rest of the method." Developers rarely think about whether the
JTA transaction has committed when the thread body executes. Virtual threads are further
confusing because they park on blocking I/O and appear to run inline — making it even less
obvious they escape the JTA context. The `@Transactional` annotation on the service method
gives a false sense that the entire method body (including any work it spawns) is
transactionally isolated.
