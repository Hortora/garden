## @Transactional on a void JAX-RS SSE Method Commits When the Method Body Returns

### What It Does

A `void` JAX-RS SSE method annotated with `@Transactional` runs its entire method body
inside a JTA transaction, then commits when `return` is reached:

```java
@GET
@Path("/tasks/{id}/stream")
@Produces("text/event-stream")
@Transactional
public void streamTask(
        @PathParam("id") String taskId,
        @Context SseEventSink sink,
        @Context Sse sse) {

    // All DB reads here are transactional
    List<Message> messages = messageService.findAllByCorrelationId(taskId);
    Commitment c = commitmentStore.findByCorrelationId(taskId).orElse(null);

    // ... logic ...

    registry.registerStream(corrId, consumer);
    // Transaction commits HERE when return executes.
    // The SseEventSink remains open AFTER commit for subsequent events.
}
```

After the method returns, the `SseEventSink` connection stays open and events can be pushed
to the client without any active transaction. The initial reads are transactionally atomic;
subsequent SSE pushes are transactionless.

### Why It's Useful

This pattern enables a clean separation:

1. **Initial check** — transactionally read the current state (commitment, message history)
2. **Register consumer** — register an event consumer on a registry
3. **Method returns / transaction commits** — the DB reads committed atomically
4. **Events flow** — the SSE sink delivers messages asynchronously, no transaction needed

This is correct and desirable. The alternative (keeping a transaction open for the duration
of the SSE stream) would hold a DB connection until the client disconnects — potentially
minutes or hours.

### Where It's Not Documented

Neither the JAX-RS 3.x spec, RESTEasy Reactive docs, nor Quarkus SSE docs mention this
behavior. The JAX-RS spec describes SSE void methods as "server-sent events endpoint" with
the server managing the connection lifecycle, but says nothing about `@Transactional`
interaction.

Discoverable only by: reading Quarkus source (`SseEventSinkImpl`), experimentation, or
knowing that `@Transactional` proxy wraps the entire method invocation and commits at
`return` regardless of the response lifecycle.

### Caveats

- If the SSE method sends an immediate event and closes the sink (immediate-close paths),
  the client may receive the event before the transaction commits (see the related gotcha
  on `fanOut()` pre-commit timing).
- This only applies to `void` return types. Methods returning `Multi<T>` or reactive
  publishers have different transaction semantics.
