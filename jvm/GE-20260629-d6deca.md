## Symptom

After migrating an API from accepting an in-memory entity object to accepting an ID and doing a store lookup internally, existing tests fail with wrong-value assertions — not compilation errors. The test appears to set up the entity correctly, but the asserted fields have default/null values instead of the test-specific ones.

Example: 5 tests in `WorkItemTemplateServiceResolutionTest` failed after migrating from `templateService.instantiate(template, ...)` to `templateService.createFromTemplate(request)`:

```
assertThat(workItem.excludedUsers).contains("alice");
// Expected: "alice"
// Actual:   null
```

## Context

The test pattern that breaks:

```java
@Transactional
WorkItemTemplate persist(String name) {
    WorkItemTemplate t = new WorkItemTemplate();
    t.name = name;
    t.createdBy = "test";
    WorkItemTemplate.persist(t);
    return t;
}

@Test
void test() {
    WorkItemTemplate t = persist("Review");
    t.excludedGroups = "legal-team";  // set AFTER persist returns
    // Old API: templateService.instantiate(t, ...) — uses the in-memory object directly
    // New API: templateService.createFromTemplate(request) — looks up by request.templateId
}
```

The old API received the template object directly, so the in-memory modification (`t.excludedGroups = "legal-team"`) was visible. The new API does `templateStore.get(request.templateId)` which fetches a fresh entity from the database — the post-persist modification was never flushed and is invisible to the store lookup.

## Root Cause

The `persist()` helper is `@Transactional`. When the test method is NOT `@Transactional`, `persist()` commits in its own transaction boundary. The returned entity is detached. Modifications to the detached entity are in-memory only — they never reach the database. The old API masked this because it used the caller's in-memory object; the new API exposes it by doing a fresh store lookup.

## Fix

Set entity properties before persisting, not after:

```java
@Transactional
WorkItemTemplate persistWith(String name, Consumer<WorkItemTemplate> customizer) {
    WorkItemTemplate t = new WorkItemTemplate();
    t.name = name;
    t.createdBy = "test";
    customizer.accept(t);  // set properties BEFORE persist
    WorkItemTemplate.persist(t);
    return t;
}

@Test
void test() {
    WorkItemTemplate t = persistWith("Review", tmpl -> {
        tmpl.excludedGroups = "legal-team";
    });
    // Now createFromTemplate() fetches the correct values from the store
}
```

## Why This Is Non-Obvious

The test looks correct — the entity has the right properties in memory, the API call uses the entity's ID, and there are no compilation errors. The failure mode is a wrong-value assertion, not an exception. The root cause (detached entity vs fresh store lookup) requires understanding both the JPA persistence context lifecycle AND the difference between the old and new API's data source.
