## Symptom

Files edited during a `git rebase -i` session are silently reverted to their
pre-rebase state. No error, no notification. The terminal shows the correct
content (`cat` confirms the edit exists), IntelliJ's editor shows the old
version — and eventually IntelliJ wins, overwriting disk.

In this session: `validateFlywayMigrationLocation` was written to
`LedgerProcessor.java` before the squash rebase ran. The rebase completed
(`BUILD SUCCESS`, 457 tests). Then `git diff HEAD` showed the method was
gone — IntelliJ had rewritten the file with its cached pre-rebase state.

## Context

Triggered during a `git-squash` workflow that used `GIT_SEQUENCE_EDITOR` to
run `git rebase -i` non-interactively. IntelliJ detects the repository state
change when the rebase completes and fires a VFS (Virtual File System) refresh.
This refresh pulls from IntelliJ's internal file cache — the state it had
**before** the rebase started — and overwrites the on-disk working tree.

Any file modified between the moment IntelliJ captured its baseline and the
moment the VFS refresh fires is at risk. The rebase itself is a git operation
on the branch history; IntelliJ's cache is a snapshot of the checked-out files
before that operation.

## Root cause

IntelliJ's VFS refresh is triggered by external git operations. When a rebase
rewrites the branch, IntelliJ treats it as "files may have changed externally"
and reloads from its cache rather than from disk. The cache represents the last
state IntelliJ synced — which predates any edits made during the rebase session.

## Fix

After any `git rebase -i` completes, explicitly verify key files before running
tests or committing:

```bash
git diff HEAD   # compare working tree against the last commit
```

If IntelliJ has reverted a file, the diff will show the expected changes as
missing. Re-edit the affected file and rebuild.

**Preventive option:** Suspend IntelliJ's VFS watcher before running a
non-interactive rebase, then re-enable after. Or edit all target files
**after** the rebase completes rather than during.

## Why this is non-obvious

The build passes. The tests pass (because the rebase itself is correct — it's
the *post-rebase* working-tree edit that's lost, not the rebase output). The
file appears correct in the terminal. Only `git diff HEAD` reveals the
discrepancy. A developer who trusts the successful build output has no reason
to suspect their last edit was silently discarded.
