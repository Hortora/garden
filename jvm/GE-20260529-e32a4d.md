## Symptom

```
java.lang.IllegalArgumentException: actorType is required
    at io.casehub.qhorus.api.message.MessageDispatch$Builder.build(MessageDispatch.java:67)
```

Appears at runtime when `MessageService.dispatch()` is called. The stack trace points at `build()` inside qhorus, not at the caller, making it appear to be a qhorus internal error.

## Context

`MessageDispatch.Builder` has many optional fields (`correlationId`, `target`, `deadline`, `inReplyTo`, etc.). `actorType` looks optional — it has the same setter pattern as the others. But `build()` validates it as a required field and throws with no indication from the API (no `@NotNull`, no Javadoc, no documentation in PLATFORM.md).

`ActorType` enum values: `HUMAN`, `AGENT`, `SYSTEM`.

## Fix

Always call `.actorType()` on the builder:

```java
messageService.dispatch(new MessageDispatch.Builder()
    .channelId(channelUuid)
    .sender("life-system")
    .actorType(ActorType.SYSTEM)          // ← required; not optional despite API shape
    .type(MessageType.COMMAND)
    .content("Approval required: " + title)
    .correlationId(correlationId)
    .deadline(deadline)
    .build());
```

Use `ActorType.SYSTEM` for automated life-system dispatches. Use `ActorType.HUMAN` when the sender is a household principal responding to a gate. Use `ActorType.AGENT` for AI agent senders.

## Why this is non-obvious

The builder API treats `actorType` identically to optional fields — it has a standard setter with no annotation or naming convention indicating it is required. Other required fields (`COMMAND` needing `correlationId`, `RESPONSE` needing `inReplyTo`) are documented in the message-dispatch-builder-validation protocol (PP-20260522-3dca14). `actorType` as a global required field is not in that protocol, not in PLATFORM.md, and not surfaced as a compile-time or early-runtime error — only at `build()` call time with a generic message.
