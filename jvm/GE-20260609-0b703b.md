### Symptom
`io.netty.handler.codec.http.websocketx.WebSocketHandshakeException: Connection closed while handshake in process`
on every WebSocket attempt after the first. SC2 is running and port is open (telnet connects),
but WebSocket handshake always fails. All 20 retries fail identically.

### Context
ocraft's `S2ClientVerticle` uses `retry(RETRY)` (no delay, immediate retry) on the WebSocket
connection observable. SC2 accepts one WebSocket client at a time.

### Root Cause
When the first WebSocket handshake fails (e.g. SC2 not yet ready), the TCP connection is
half-closed from SC2's side but Vert.x's `HttpClient` may not immediately reclaim the
connection-pool slot. SC2 sees the previous connection as still held. Subsequent rapid
retries are rejected with an empty close — SC2 won't accept a new connection while an
existing one is still in its connection table, even a half-closed one.

Separately: if a **stale SC2 process** from a previous session is holding the API port,
`processIsAlive()` in ocraft returns false the moment the newly-launched SC2 exits (due
to port conflict), causing an immediate throw before retries can run.

### Fix
Replace `retry(N)` with a delayed retry in `S2ClientVerticle.connect()`:
```java
// Instead of:
.retry(RETRY)

// Use:
.retryWhen(errors -> errors.take(RETRY).flatMap(e ->
    Observable.timer(3, TimeUnit.SECONDS)))
```
The 3-second gap gives SC2 time to release the previous connection slot.

Also: always kill stale SC2 processes before launching a new one (`pkill -9 SC2`).
SC2 in `-listen` mode does not self-terminate on disconnect.

### Why This Is Non-Obvious
- SC2 accepting TCP connections does not mean it's accepting WebSocket — it can accept
  the TCP SYN during API startup before the WebSocket handler is ready
- `python3 -c "import socket; s=socket.create_connection(('127.0.0.1',8168)); s.send(b'GET /sc2api HTTP/1.1\r\n...'); print(s.recv(4096))"` returns 0 bytes when SC2 has a stale connection, but also when it's not ready — the two failure modes look identical
- The telnet port-open check passes in both cases
- ocraft's default `retry(20)` with no delay compounds the problem — 20 immediate attempts
  all hit the same stale connection in rapid succession
