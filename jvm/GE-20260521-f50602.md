# quarkus-oidc: discovery-disabled requires jwks-path or introspection-path

### Symptom
Application startup fails with:
`ConfigurationException: Either 'jwks-path' or 'introspection-path' properties must be set when the discovery is disabled.`

### Context
When adding `quarkus-oidc` to a module and configuring `quarkus.oidc.discovery-enabled=false`
in test properties (to avoid connecting to a real OIDC server), startup fails unless one of
the two additional properties is also present.

### Root cause
Quarkus OIDC validates at startup that if discovery is disabled, it has at least one way to
obtain public keys (`jwks-path`) or validate tokens via introspection (`introspection-path`).
Without either, the extension cannot function and refuses to start.

### Fix
Add a `jwks-path` value to the test `application.properties`:

```properties
quarkus.oidc.auth-server-url=http://localhost:8180/realms/test
quarkus.oidc.discovery-enabled=false
quarkus.oidc.jwks-path=protocol/openid-connect/certs
```

The `jwks-path` is resolved relative to `auth-server-url`, so the full URL would be
`http://localhost:8180/realms/test/protocol/openid-connect/certs`. Crucially, **this URL
is never fetched at startup** — JWKS loading is lazy (happens on first token validation).
When using `@InjectMock` to replace `SecurityIdentity` and `JsonWebToken` in tests, token
validation never runs, so a dummy path causes no connection failures.

### Why this is non-obvious
`discovery-enabled=false` sounds like a complete opt-out of server connectivity. The
additional requirement for either `jwks-path` or `introspection-path` is not prominent in
the main OIDC configuration docs — it appears as a validation error at runtime rather than
being listed as a required companion property in the `discovery-enabled` documentation.
