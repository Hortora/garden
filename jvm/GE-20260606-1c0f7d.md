## Symptom

Code compiles, tests with name inputs pass. When a UUID is passed as a channel identifier,
the result object contains the UUID string in a field that should hold the channel name.
No exception is thrown.

## Context

A Uni chain resolves a channel entity, then uses it in a `.flatMap()` whose inner Uni
has a different type. A terminal `.map()` chained outside the flatMap needs the channel
entity from the outer scope — but instead uses the raw method parameter (the UUID string).

```java
// WRONG — terminal .map() outside the flatMap, uses raw 'channel' param (may be UUID)
return resolveChannelAsync(channel)
    .invoke(ch -> checkAdminAccess(ch, callerId, "delete_channel"))
    .flatMap(ch -> channelService.delete(ch.name, force)   // inner Uni<Boolean>
        .invoke(ignored -> channelGateway.closeChannel(ch.id, ...)))
    .map(deleted -> new DeleteChannelResult(channel, deleted, "deleted"));
//                                          ^ 'channel' is the UUID string, not ch.name
```

After the `.flatMap(ch -> ...)` that returns `Uni<Boolean>`, the stream item type is
`Boolean`. A `.map()` here has `deleted` as its stream item. The `ch` variable from the
outer `.flatMap(ch -> ...)` closure is NOT in scope — `ch` is captured inside the
lambda boundary. The code therefore uses `channel` (the raw input parameter), which
may be a UUID string rather than the resolved channel name.

## Root Cause

Java lambda capture: `ch` is bound as the lambda parameter of `.flatMap(ch -> ...)`.
It is only in scope within that lambda body. A `.map()` chained after the flatMap has
no access to `ch` — the `.flatMap()` has already consumed the `Uni<Channel>` and
emitted the inner `Uni<Boolean>`. The outer scope sees only the method parameters
(`channel`, `callerId`, etc.), not the intermediate resolved entity.

## Fix

Nest the terminal `.map()` inside the `.flatMap()` closure so `ch` stays in scope:

```java
// CORRECT — terminal .map() nested inside the .flatMap() closure
return resolveChannelAsync(channel)
    .invoke(ch -> checkAdminAccess(ch, callerId, "delete_channel"))
    .flatMap(ch -> channelService.delete(ch.name, force)
        .invoke(ignored -> channelGateway.closeChannel(ch.id, ...))
        .map(deleted -> new DeleteChannelResult(ch.name, deleted, "deleted")));
//          ^ 'ch' is in scope here — we're still inside the flatMap lambda
```

**General rule:** when a Uni chain passes through a `.flatMap()` that changes the stream
type, and a downstream terminal step needs a variable from the outer flatMap scope, nest
the terminal step inside the `.flatMap()` closure — not after it.

## Why This Is Non-Obvious

The bug is invisible for name inputs because `channel == ch.name` (the raw string IS
the name). It only surfaces when a UUID is passed, causing `channel != ch.name`. The
code compiles with no warnings. Tests using channel names pass. The symptom is a wrong
field value in the result object, not an exception — making it easy to miss without
explicitly testing UUID inputs. The `.flatMap()` and `.map()` calls look structurally
identical whether `ch` is in scope or not.
