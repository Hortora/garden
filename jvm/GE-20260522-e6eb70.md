## Symptom

`mvn test-compile` fails with:

```
cannot access io.example.api.DeletedClass
  class file for io.example.api.DeletedClass not found
```

The consuming module's test source never imports `DeletedClass` directly.

## Context

A peer repo is mid-migration: it deleted a class from its API module (e.g., `MessageResult` → `DispatchResult`) but the runtime module still doesn't compile — so the runtime jar in the local `.m2` cache is the old version. That old jar has methods returning `Uni<DeletedClass>`. The API jar was rebuilt (or pulled from GitHub Packages) and no longer contains `DeletedClass`.

Consumer test source calls one of those methods: `tools.sendMessage(...)`. The Java compiler resolves all types in the method's bytecode signature, including the return type. It finds `Uni<DeletedClass>` in the bytecode, looks up `DeletedClass` on the classpath — finds it in neither the new API jar nor anywhere else — and fails.

## Root Cause

The Java compiler resolves every type referenced in bytecode signatures of called methods, including return type generic parameters. An inconsistency between two jars from the same peer repo (API jar rebuilt/new, runtime jar stale/old) creates a "phantom" type reference that the consumer never wrote but still must resolve.

Surefire's `<excludes>` in pom.xml prevents execution of those tests. It does not prevent test-compile of those sources.

## Fix

Add an empty stub class at the deleted class's exact package path in the consuming repo's test sources:

```java
// src/test/java/io/example/api/DeletedClass.java
package io.example.api;

// Transitive compile-time stub — peer repo deleted this class mid-migration.
// Remove once the peer repo runtime jar is rebuilt with the new return type.
public class DeletedClass {
}
```

This satisfies the transitive compile-time lookup without touching the peer repo. The stub comment anchors the removal trigger.

## Why This Is Non-Obvious

The compile error points to `DeletedClass` — a class the consuming module never imports. The natural investigation path (checking imports, checking dependencies) finds nothing. The symptom looks like a classpath configuration problem, not a bytecode return-type resolution problem.

The fix — adding a stub in the *consumer* for a class that belongs to a *peer repo* — feels wrong at first: why should the consumer define something from another package? The answer is that the stub is purely satisfying a compile-time lookup, not providing runtime behaviour; the actual jar will (eventually) be rebuilt and the stub disappears.

Also tried: `-Dmaven.compiler.testExcludes=io/example/e2e/**` — this system property is not honoured by the Maven compiler plugin for test source exclusion from the command line; only `<excludes>` in the pom.xml configuration works.
