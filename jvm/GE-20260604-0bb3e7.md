## casehub-qhorus ChannelService.delete() auto-closes all registered backends via ChannelGateway.cleanupForDeletion()

**ID:** GE-20260604-0bb3e7
**Stack:** casehub-qhorus 0.2-SNAPSHOT
**Labels:** `#casehub-qhorus` `#channelgateway` `#channelservice` `#cleanup`

### Symptom

When designing the `end_review` lifecycle for a channel-backed session, the question arises: do registered `ChannelBackend` implementations need to be explicitly deregistered before calling `channelService.delete()`? The documentation is silent on this. Without explicit deregistration, it's unclear whether the backend would leak in the in-memory `ChannelGateway` registry.

### Context

A session lifecycle tool (`DraftHouseMcpTools`) calls `channelService.delete(channelName, force=true)` when `end_review` is called with `deleteChannel=true`. A `ReviewerChannelBackend` was previously registered on this channel via `gateway.registerBackend()`.

### Root cause

The documentation makes no mention of backend cleanup on channel deletion. The behavior was discovered by reading `ChannelGateway` source.

### What actually happens

`ChannelService.delete()` internally calls `ChannelGateway.cleanupForDeletion(channelId)` before removing the channel from persistence. `cleanupForDeletion()` calls `registry.remove(channelId)` (removes the in-memory entry) and calls `backend.close(channelRef)` on every registered backend for that channel.

```java
// ChannelGateway.cleanupForDeletion() — called internally by ChannelService.delete()
/** Called by delete_channel to clean up all backends. */
public void cleanupForDeletion(UUID channelId) {
    List<BackendEntry> entries = registry.remove(channelId);
    if (entries != null) {
        for (BackendEntry entry : entries) {
            try {
                entry.backend().close(channelRef);
            } catch (Exception ex) {
                LOG.errorf(ex, "Error closing backend %s on channel %s",
                        entry.backend().backendId(), channelId);
            }
        }
    }
}
```

Result: callers of `channelService.delete()` do NOT need to separately call `gateway.deregisterBackend()` or `backend.close()`. The delete operation is fully self-contained.

### Why this is non-obvious

There are three separate concerns — persistence (`ChannelService`), runtime dispatch (`ChannelGateway`), and backend registration — and the documentation treats them as independent. Nothing in the `ChannelService.delete()` javadoc or the qhorus developer docs mentions that gateway cleanup is included. Developers designing teardown flows naturally assume they must mirror the setup flow (create → initChannel → registerBackend → cleanupForDeletion? → delete) and may add a manual deregistration step that is both redundant and harmless but misleading.
