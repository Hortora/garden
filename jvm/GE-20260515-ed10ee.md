### Symptom

Test passes reliably in isolation but could spuriously pass under load when
a duplicate async event arrives after the assertion poll.

```java
// Looks correct — doesn't guarantee stability
Awaitility.await()
    .atMost(Duration.ofSeconds(5))
    .untilAsserted(() -> assertThat(events).hasSize(1));
```

### Context

Testing exact counts of async CDI events (`@ObservesAsync`), Vert.x event
bus messages, or any observer pattern where you need "exactly N events fired."

### Root Cause

`untilAsserted` succeeds and **exits** on the first poll where the assertion
passes. It does not continue checking. A second event arriving 50 ms later
would not cause a failure — the test has already finished. The count assertion
is a snapshot, not a stability guarantee.

### Fix

Use a `during()` stability window for the count, then a separate assertion
for the value:

```java
// Count stays at exactly 1 for 300ms — guards against late duplicates
Awaitility.await()
    .atMost(Duration.ofSeconds(5))
    .during(Duration.ofMillis(300))
    .until(() -> events.size() == 1);

// Plain assertion after stability resolves — .get(0) is safe here
assertThat(events.get(0).callerRef()).isEqualTo(expected);
```

`untilAsserted` is correct for "eventually becomes true and I don't care if it
stays true." `during` is correct for "stays true continuously for this window."

### Why This Is Non-Obvious

`untilAsserted` reads as "keep asserting until it passes," which sounds like
it would catch a count that fluctuates. In reality it exits the moment the
assertion first passes, with no further polling. The `during()` window is
the actual mechanism for stability checking and is much less discovered
than `untilAsserted`.

### Variant — Negative assertion ("this should never happen")

When asserting that an async event is **never** published, `Thread.sleep + isEmpty()` only
checks at one point in time — it misses events that fired and were consumed before the sleep
ended, and it flakes under CI load. The `during()` variant asserts the condition holds
**continuously** throughout the window:

```java
// Asserts events list stays empty for the full 300ms window
// Any event arriving within the window fails the next poll cycle
Awaitility.await()
    .during(Duration.ofMillis(300))
    .atMost(Duration.ofMillis(500))
    .untilAsserted(() -> assertThat(events).isEmpty());
```

`atMost()` sets the outer deadline; `during()` sets the minimum stable window. If the list
becomes non-empty at any point during the 300ms, `untilAsserted()` catches it on the next
poll. Non-obvious: without `during()`, `await().atMost(N).untilAsserted(isEmpty)` exits
immediately (the list is empty from the start) — the wait never actually runs.
