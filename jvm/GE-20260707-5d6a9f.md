### Symptom

Bean excluded via `quarkus.arc.exclude-types` in production `application.properties` is still discovered during `@QuarkusTest` — CDI ambiguity or unproxyable errors that were "fixed" in production config still fire in tests.

### Context

Both `src/main/resources/application.properties` and `src/test/resources/application.properties` define `quarkus.arc.exclude-types`. The test file has its own exclusions (mock beans, scheduled jobs) but does not include the production exclusions.

### Root cause

Quarkus test properties REPLACE same-key production properties entirely — they are not merged or appended. When the test config defines `quarkus.arc.exclude-types`, the production value for that key is completely ignored. Each exclusion must appear in BOTH files.

### Fix

Maintain the test `quarkus.arc.exclude-types` as a superset of the production value. When adding an exclusion to production config, add it to test config too:

```properties
# src/test/resources/application.properties
quarkus.arc.exclude-types=\
  io.casehub.platform.mock.MockGroupMembershipProvider,\
  io.casehub.platform.mock.MockCurrentPrincipal,\
  io.casehub.work.runtime.service.ExpiryCleanupJob,\
  io.casehub.desiredstate.runtime.CdiNodeProvisionerRouter
```

### Why this is non-obvious

Most developers assume test properties only override what they explicitly set, and that unrelated keys from production config still apply. For simple properties (strings, numbers) this mental model works. For multi-valued properties like `exclude-types`, the full-replacement semantics cause silent regressions.
