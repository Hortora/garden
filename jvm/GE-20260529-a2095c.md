### Symptom

In a `@QuarkusTest @WithPlaywright` class with 3 or more test methods that each call
`context.newPage()`, the third test (and beyond) hangs for 30 seconds then fails with a
Playwright `waitForSelector` timeout. Tests 1 and 2 pass consistently. Every test passes in
isolation; the failure only appears in sequence.

### Context

Occurs when using the `quarkus-playwright` quarkiverse extension with a shared
`BrowserContext` field (class-level `@InjectPlaywright`) and multiple test methods that each
open a new page to load the application.

The application must have live SSE endpoints (e.g. an `EventSource` connection for file
watching) that the browser connects to on page load.

### Root Cause

Each unclosed `Page` keeps a live SSE `EventSource` connection open in the background. The
`@WithPlaywright` annotation closes the `BrowserContext` at class teardown — but that happens
after all test methods have run, too late to prevent accumulation.

By test 3, the Quarkus server's HTTP connection pool (default size: typically 4–8 slots) is
fully consumed by the open SSE connections from tests 1 and 2. `page.navigate()` in test 3
blocks waiting for a free slot. Playwright's `waitForSelector` timeout (30 seconds default)
fires first, producing a generic "element not found" error with no mention of connection
limits.

### Fix

Close each page explicitly after use. The cleanest approach is `@BeforeEach`/`@AfterEach`:

```java
@QuarkusTest
@WithPlaywright
class MyE2ETest {

    @InjectPlaywright
    BrowserContext context;

    private Page page;

    @BeforeEach
    void openPage() {
        page = context.newPage();
    }

    @AfterEach
    void closePage() {
        if (page != null) page.close();
    }

    @Test
    void myTest() {
        page.navigate(...);
        // assert
    }
}
```

Alternative — `try/finally` per test method (works but verbose):

```java
@Test
void myTest() {
    Page page = context.newPage();
    try {
        page.navigate(...);
        // assert
    } finally {
        page.close();
    }
}
```

### Why This Is Non-Obvious

1. **The lifecycle annotation implies cleanup.** `@WithPlaywright` suggests the framework
   handles teardown. Reasonable to assume closing the `BrowserContext` at class level is
   sufficient — it is, but only after all tests have run.

2. **Positional failure hides the cause.** Tests 1 and 2 pass; test 3 always fails. This
   looks like a test ordering or state pollution problem, not a resource exhaustion problem.

3. **The error message is misleading.** Playwright reports a `waitForSelector` timeout
   ("element not found after 30s") — nothing in the output mentions HTTP connection limits or
   SSE connections.

4. **Tests pass in isolation.** Running `mvn test -Dtest=MyTest#myThirdTest` passes every
   time, because there are no accumulated connections.

### Alternatives Considered

- **`@InjectPlaywright Page page` injection** — the extension may support per-test-method
  `Page` injection, which could provide fresh pages automatically. This was not verified at
  the time of this entry; the `@AfterEach` pattern is the confirmed fix.
- **try/finally per test** — functionally correct but adds noise to every test method;
  `@BeforeEach`/`@AfterEach` is cleaner and less error-prone.
