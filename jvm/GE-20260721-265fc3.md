### Symptom
Flyway migration containing `ALTER TABLE t DROP CONSTRAINT t_col_fkey` succeeds
on PostgreSQL but fails on H2 with:
```
Constraint "T_COL_FKEY" not found; SQL statement: ALTER TABLE t DROP CONSTRAINT t_col_fkey [90057-240]
```

### Context
When a foreign key is defined inline in `CREATE TABLE`:
```sql
CREATE TABLE engagement_event (
    attempt_id VARCHAR(36) NOT NULL REFERENCES delivery_attempt(id) ON DELETE CASCADE
);
```
PostgreSQL auto-generates a constraint named `engagement_event_attempt_id_fkey`
(convention: `tablename_columnname_fkey`). H2 auto-generates `CONSTRAINT_<N>`
(sequential integer). A later migration referencing the PostgreSQL convention
name cannot find the constraint on H2.

### What was tried (didn't work)
- `ALTER TABLE t DROP CONSTRAINT engagement_event_attempt_id_fkey` — fails on H2
- `ALTER TABLE t DROP CONSTRAINT IF EXISTS engagement_event_attempt_id_fkey` — silently
  succeeds on H2 (constraint doesn't exist under that name) but leaves the old
  CASCADE FK in place, creating a second FK when the new SET NULL FK is added

### Root cause
PostgreSQL and H2 use different auto-naming conventions for inline foreign key
constraints. There is no cross-database portable way to reference an auto-named
constraint.

### Fix
Name FK constraints explicitly in the initial CREATE TABLE:
```sql
CREATE TABLE engagement_event (
    attempt_id VARCHAR(36) NOT NULL,
    CONSTRAINT engagement_event_attempt_id_fk
        FOREIGN KEY (attempt_id) REFERENCES delivery_attempt(id) ON DELETE CASCADE
);
```
Or split FK manipulation into a separate migration that uses `IF EXISTS`:
```sql
ALTER TABLE engagement_event DROP CONSTRAINT IF EXISTS engagement_event_attempt_id_fkey;
ALTER TABLE engagement_event ADD CONSTRAINT engagement_event_attempt_id_fkey
    FOREIGN KEY (attempt_id) REFERENCES delivery_attempt(id) ON DELETE SET NULL;
```
The `IF EXISTS` approach works when you only care about the PostgreSQL production
database and accept that H2 tests won't exercise the FK behaviour change.

### Why this is non-obvious
The error message says "constraint not found" which suggests a typo or missing
migration. The actual cause — auto-naming convention differences between databases —
is invisible unless you inspect H2's information_schema to see what name it assigned.

### Alternatives considered
- Querying INFORMATION_SCHEMA.TABLE_CONSTRAINTS to find the auto-generated name at runtime — works but adds complexity to what should be a simple DDL migration
- Using H2's `MODE=PostgreSQL` — does not affect constraint auto-naming behaviour
