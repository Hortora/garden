## Symptom

Runtime SQL syntax error — e.g. `SELECT count(*) FROM Message WHERE WHERE 1=1 AND channelId = ?1`. Tests using InMemory store alternatives pass; only the JPA integration test catches it.

## Context

Adding a `count(MessageQuery)` method to a Quarkus Panache JPA store. The store already has a `scan()` method that builds full JPQL: `"FROM Message WHERE " + predicateString`. By analogy, the developer passes `"WHERE " + predicateString` to `Entity.count(...)`.

## What was tried

```java
MessageQueryJpql mq = MessageQueryJpql.from(q);
return Message.count("WHERE " + mq.where(), mq.params()); // Wrong
```

## Root cause

Panache's `count(String, Object...)` treats its argument as a WHERE predicate fragment — it internally prepends `FROM Entity WHERE`. Passing `"WHERE 1=1"` produces `FROM Message WHERE WHERE 1=1` — a SQL syntax error.

Panache's `find()` / `list()` take full JPQL (`"FROM Entity WHERE ..."`), which conditions developers to include the WHERE keyword explicitly. `count()` uses a different convention — predicate-only — and this distinction is not prominent in the surface docs.

## Fix

Pass just the predicate fragment — no `WHERE` keyword:

```java
MessageQueryJpql mq = MessageQueryJpql.from(q);
return Message.count(mq.where(), mq.params()); // Correct — "1=1 AND channelId = ?1"
```

## Why this is non-obvious

When a store has both `scan()` (full JPQL) and `count()` alongside each other, the natural assumption is that both take the same format. They don't. `count()` is predicate-only. The discrepancy is invisible until the JPA execution path is exercised — InMemory store tests pass regardless, because they never call Panache.
