### Symptom
Capability trust scores remain unchanged (e.g. 11/12 = 0.917) regardless of how many FLAGGED attestations are written against an actor's decisions. Investigation naturally targets the query layer, computation logic, or score persistence — all of which are correct.

### Context
Bayesian Beta trust scoring with a four-pass pipeline (capability → dimension → cap×dim → global). Attestations are aggregated per (entryId, capabilityTag) group before scoring. An incident feedback system writes a FLAGGED attestation against an existing entry that already holds a SOUND attestation from the original review.

### What was tried (didn't work)
- Checked query layer (findAllEvents, findAttestationsForEntries) — fetches all verdicts correctly
- Checked TrustScoreComputer.compute() — handles FLAGGED by incrementing beta correctly
- Checked PerActorTrustComputer delegation — passes data through unchanged
- Checked JPA upsert — updates existing score rows correctly
- Checked TrustScoreSource/TrustGateService — reads directly from repository
- All unit tests with FLAGGED on separate entries pass

### Root cause
`TrustScoreCalculator.computeAll()` runs `buildEffectiveAttestations()` which groups attestations by (entryId, capabilityTag) and aggregates via `AttestationAggregator.weightedMajority()`. When an entry has SOUND(confidence=1.0) + FLAGGED(confidence=0.9):
- positive = 1.0, negative = 0.9
- `positive >= negative` → verdict = SOUND
- confidence = |1.0 - 0.9| / 1.9 ≈ 0.053
- Weight in compute() = decayWeight × 0.053 ≈ 0 → neither alpha nor beta increment meaningfully

The FLAGGED attestation is not ignored — it is aggregated into a near-zero-weight SOUND, which is worse: it cancels the original SOUND's contribution too.

### Fix
Use raw attestations (not aggregated) for the capability pass. Each individual attestation contributes to alpha/beta independently — SOUND increments alpha, FLAGGED increments beta, both on the same entry. The dimension pass already used raw attestations; this brings the capability pass into alignment.

```java
// Before (broken): used effectiveAttestations (pre-aggregated)
final Map<String, Map<UUID, List<LedgerAttestation>>> byCapabilityAndEntry =
        effectiveAttestations.stream()
                .filter(a -> !CapabilityTag.GLOBAL.equals(a.capabilityTag))
                ...

// After (fixed): uses rawAttestations
final Map<String, Map<UUID, List<LedgerAttestation>>> byCapabilityAndEntry =
        rawAttestations.stream()
                .filter(a -> a.capabilityTag != null
                        && !CapabilityTag.GLOBAL.equals(a.capabilityTag))
                ...
```

Note the added null guard on `capabilityTag` — raw attestations don't pass through `buildEffectiveAttestations` which normalizes nulls to GLOBAL.

### Why this is non-obvious
The aggregation step sits between data fetch and computation. Both endpoints work correctly in isolation. The symptom (score unchanged) points at the computation or query layer. The aggregation is designed to reduce noise from multiple attestors — it silently converts the sequential-evidence case (SOUND then FLAGGED over time) into the concurrent-vote case (competing opinions) without any log message or signal.

### Why this fix
Raw attestations are the right model for capability scoring because attestations on the same entry arrive at different times as sequential evidence (review outcome, then incident feedback), not as concurrent competing votes. The aggregation model was designed for the N-attestors-same-time case. Trade-off: N attestors on the same entry now contribute N× weight instead of 1× consensus weight — an idea has been logged to explore a hybrid approach.

### Alternatives considered
- Changing the WEIGHTED_MAJORITY tie-break from `>=` to `>` — still produces SOUND for the 1.0 vs 0.9 case; doesn't fix the core issue
- Adding a temporal-awareness flag to the aggregator — overengineered for the current use case; the raw-attestation approach is simpler and matches the dimension pass
- Processing attestations individually with no aggregation anywhere — would affect the global pass which intentionally uses aggregation via GlobalScoreStrategy SPI
