### Context

`MERGE INTO table USING (...) WHEN NOT MATCHED THEN INSERT` is used as an
atomic upsert — e.g. for sequence allocation tables that need "insert if absent,
update if present."

### Symptom

Under concurrent writes from multiple threads targeting the same new primary key,
one or more transactions throw:

```
Unique index or primary key violation: "PUBLIC.PK_<table> INDEX ..."
SQL statement: MERGE INTO <table> AS t USING ... WHEN NOT MATCHED THEN INSERT ...
```

The failure only occurs on first-insert for a new key. Subsequent writes (where
the row already exists) hit `WHEN MATCHED THEN UPDATE` and succeed. Serial tests
pass; concurrent tests fail.

### Root Cause

H2 2.0 (July 2021) switched the default storage engine from **page-store**
(table-level locking) to **MVStore** (MVCC / row-level). Under MVStore, `MERGE INTO`
evaluates `WHEN NOT MATCHED` non-atomically:

1. Transaction A reads — row absent → plans INSERT
2. Transaction B reads — row absent → plans INSERT
3. Transaction A inserts → succeeds
4. Transaction B inserts → unique constraint violation (row now exists)

The conflict is not suppressed — it propagates as a `ConstraintViolationException`,
poisoning Transaction B's `EntityManager` and requiring a full transaction rollback.

Documentation and Javadoc written for H2 1.x stated "H2 uses table-level locking."
This was true for page-store. It is false for MVStore.

### Fix

Replace the single-statement `MERGE INTO` with two statements:

```java
// INSERT ON CONFLICT DO NOTHING — concurrent first-inserts are safe.
// H2 2.2+ in MODE=PostgreSQL supports the no-column-target form only:
// ON CONFLICT DO NOTHING (not ON CONFLICT (col1, col2) DO NOTHING).
em.createNativeQuery(
    "INSERT INTO sequence_table (subject_id, tenancy_id, next_seq) " +
    "VALUES (CAST(?1 AS UUID), ?2, 1) " +
    "ON CONFLICT DO NOTHING")
    .setParameter(1, subjectId).setParameter(2, tenancyId).executeUpdate();

em.createNativeQuery(
    "UPDATE sequence_table SET next_seq = next_seq + 1 " +
    "WHERE subject_id = CAST(?1 AS UUID) AND tenancy_id = ?2")
    .setParameter(1, subjectId).setParameter(2, tenancyId).executeUpdate();
```

Both PostgreSQL and H2 2.2+ (in `MODE=PostgreSQL`) accept this SQL. The `UPDATE`
acquires a row lock held for the transaction duration, serialising concurrent writes.

### Why this is non-obvious

H2 2.0 silently changed the default storage engine — existing `MERGE INTO` code
that worked correctly for years under H2 1.x breaks only under concurrent load.
Serial unit tests always pass. The failure surfaces only in integration test suites
that exercise concurrent writes (e.g. Quartz workers, concurrent HTTP handlers).

The Javadoc claiming "H2 uses table-level locking" remained in the codebase long
after H2 2.x shipped, reinforcing the incorrect assumption.

### Why this fix

`MERGE INTO` evaluated in a single statement appears atomic but is not under MVCC.
The two-statement approach makes the sequence explicit: one statement that safely
handles concurrent first-inserts (DO NOTHING on conflict), and one that serialises
via row lock. This is identical to PostgreSQL's `INSERT ON CONFLICT DO NOTHING +
UPDATE` pattern and can be unified across both dialects with the same SQL.

### Alternatives considered

- **`LOCK_MODE=1` on H2 URL** — does not restore table-level locking for MVStore tables;
  the flag only affects the discontinued page-store engine.
- **`REQUIRES_NEW` inner transaction for first-insert** — adds a class and transaction
  boundary just for the initial-insert edge case; rejected as unnecessary complexity.
- **`SELECT FOR UPDATE` / pessimistic lock** — cannot lock a row that doesn't exist yet;
  fails silently on first-insert (same race condition).
