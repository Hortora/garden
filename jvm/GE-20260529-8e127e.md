## Context

When writing `@QuarkusTest` tests that interact with the database, the natural impulse
is to annotate test methods with `@jakarta.transaction.Transactional` to wrap each test
in a transaction for isolation. It compiles, individual tests pass.

## Symptom

Tests pass individually but fail when run as a suite. Data written in one test is visible
in subsequent tests, causing unexpected row counts or assertion failures. The failure is
non-deterministic and can depend on test execution order or parallelism.

## Root Cause

`@jakarta.transaction.Transactional` on a `@QuarkusTest` method **commits** the transaction
at method end â€” standard JTA semantics. Data written in the test body is durably committed.
The next test inherits that state.

`@io.quarkus.test.TestTransaction` is the Quarkus-specific annotation for test isolation:
it wraps the entire test method in a JTA transaction and **always rolls back** after the
test completes, regardless of outcome. This is the correct annotation for `@QuarkusTest`
database isolation.

## Fix

Replace `@Transactional` with `@TestTransaction` on test methods that need database isolation:

```java
// Wrong â€” commits data, leaks to subsequent tests
@Test
@jakarta.transaction.Transactional
void my_test() { ... }

// Correct â€” always rolls back, clean state for next test
@Test
@io.quarkus.test.TestTransaction
void my_test() { ... }
```

Import: `io.quarkus.test.TestTransaction` (provided by `quarkus-junit` dependency).

Note: `@TestTransaction` has its own caveat â€” `em.flush()` is never called within the
test body, so unique-constraint violations are invisible until rollback. See GE-20260414-7ce32b.

## Why This Is Non-Obvious

Both annotations share "Transaction" in their name. The rollback difference is only apparent
from Quarkus's own `@TestTransaction` docs, which are easy to miss if coming from Spring
(where `@Transactional` on a test DOES roll back) or from general JEE knowledge (where
`@Transactional` always commits). The failure is especially subtle because a single isolated
test never sees its own leakage â€” only subsequent tests do.
