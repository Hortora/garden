### Symptom
SDD implementer subagent reports BLOCKED on a task that should be mechanical — `mvn test` fails with compilation errors in files the subagent did not touch.

### Context
Upstream SNAPSHOT dependencies (casehub-engine, casehub-ledger) changed API signatures between sessions: `ReadablePanel` renamed to `ReadableLayer`, `panel()` to `layer()`, `LedgerEntry` moved packages, `ActorTrustScore.ScoreType` extracted to top-level. The project's main branch compiles against the old API — `mvn compile` fails before any subagent work begins.

The SDD controller dispatches implementer subagents without verifying the baseline compiles. Each subagent hits the same compilation failure, reports BLOCKED, and wastes ~25 minutes of wall-clock time before the controller diagnoses the root cause.

### Root cause
SDD assumes the branch starts from a compilable baseline. With SNAPSHOT dependencies, that assumption can break silently between sessions — no CI runs on main for every upstream SNAPSHOT publish.

### Fix
Before dispatching Task 1, the SDD controller runs `mvn compile` on the project. If it fails, fix the compilation errors (typically import renames) before any subagent dispatch. The subagent then sees a green baseline and can focus on its task.

```bash
# Pre-flight check before SDD dispatch
mvn compile -q -f /path/to/project/pom.xml
# If exit code != 0: fix import drift, commit, then dispatch
```

### Why this is non-obvious
The SDD controller assumes the branch's starting point compiles — a reasonable assumption for pinned dependencies. With SNAPSHOTs, upstream API changes are invisible until `mvn compile` runs. The subagent's BLOCKED report looks like a task problem, not a baseline problem, because the compilation errors are in files unrelated to the task.

### Alternatives considered
- Pin dependency versions instead of SNAPSHOTs — eliminates the problem but blocks rapid iteration across the CaseHub family
- Have each subagent fix compilation errors it encounters — violates the subagent's scope (it doesn't have cross-project context to know what the right fix is)
