### What happened

`AmlInvestigationResourceTest` and `AmlLedgerChainTest` failed with:

```
Unique index or primary key violation: "IDX_LEDGER_ENTRY_SUBJECT_SEQ ON
LEDGER_ENTRY(SUBJECT_ID, SEQUENCE_NUMBER) VALUES (UUID '...', 1)"
insert into ledger_entry (..., dtype, id) values (..., 'QHORUS_MESSAGE', ?)
```

The qhorus COMMAND dispatch was trying to write `sequenceNumber=1` for a subject (`caseId`) that already had an `AmlCaseOpenedLedgerEntry` at `sequenceNumber=1`. Investigation showed that qhorus's `LedgerWriteService.record()` computed the next sequence number via:

```java
final int sequenceNumber = repository.findLatestBySubjectId(resolvedSubjectId)
        .map(e -> e.sequenceNumber + 1).orElse(1);
```

### Root cause

`repository` was `MessageLedgerEntryRepository`, which implemented `LedgerEntryRepository` with a **subclass-scoped** JPQL query:

```java
// WRONG — only sees QHORUS_MESSAGE dtype rows
"SELECT e FROM MessageLedgerEntry e WHERE e.subjectId = :sid ORDER BY e.sequenceNumber DESC"
```

With JOINED inheritance, `FROM MessageLedgerEntry` only returns rows with `dtype='QHORUS_MESSAGE'`. The `AmlCaseOpenedLedgerEntry` committed earlier (dtype=`AML_CASE_OPENED`) was invisible to this query — so `findLatestBySubjectId` returned `Optional.empty()` and assigned `sequenceNumber=1`, colliding with the domain entry.

The `IDX_LEDGER_ENTRY_SUBJECT_SEQ` constraint is on the base `LEDGER_ENTRY` table — it enforces global uniqueness across ALL subtypes for a given subject. The sequence query must be equally global.

### Fix

Changed to base-class JPQL so all subtypes are visible:

```java
// CORRECT — sees all LedgerEntry subtypes for the subject
"SELECT e FROM LedgerEntry e WHERE e.subjectId = :sid ORDER BY e.sequenceNumber DESC"
```

The proper re-architecture (casehub-qhorus#253) introduced a dedicated `LedgerEntryJpaRepository` using `FROM LedgerEntry` throughout, and `MessageLedgerEntryRepository` no longer implements `LedgerEntryRepository` — it retains only qhorus-scoped queries.

### Why this is non-obvious

qhorus's own tests never mix non-qhorus `LedgerEntry` subtypes in the same subject, so the scoped query always looked correct in qhorus CI. The bug only surfaces in a consuming application that writes both domain entries (`AmlCaseOpenedLedgerEntry`) and dispatches qhorus messages to the same subject (`caseId`). The symptom (unique constraint on sequence number) looks like a race condition, but the actual cause is a dtype-scope bug in the sequence-assignment query.

### Constraints

Applies when: (1) multiple teams write different `LedgerEntry` subclasses to the same `subjectId`, AND (2) the sequence-assignment query is scoped to one subclass. Any `LedgerEntryRepository` implementation that narrows JPQL to a subclass is potentially broken for consumers that share subjects across dtypes.

### Why this fix

A dedicated `LedgerEntryJpaRepository` (separate from `MessageLedgerEntryRepository`) enforces at the type level that cross-dtype operations use the correct base-class query. Patching the subclass JPQL directly would not prevent future regressions when new repositories are added.
