### What it is

`AgenticProcessor.implyAiService()` produces `AnnotationsImpliesAiServiceBuildItem` for all agent annotations. This causes `AiServicesProcessor.determinedImpliedRegisterAiService()` to create synthetic `@RegisterAiServices` annotation instances for agent interfaces, which are then processed through the full AiService pipeline — including `DeclarativeAiServiceBuildItem` production, `gatherMethodMetadata()` (which reads `@InputGuardrails`/`@OutputGuardrails`), build-time bean validation via `validateGuardrails()`, and unremovable bean marking.

At runtime, the guardrail metadata stored in `AiServicesRecorder.getMetadata()` is picked up by `GuardrailServiceBuilder` via `QuarkusClassMetadataProviderFactory` (SPI), guardrail beans are resolved via `QuarkusClassInstanceFactory` (CDI), and `maxRetries` config is applied via `QuarkusOutputGuardrailsConfigBuilderFactory`. The entire chain fires without any agent-specific code.

### Why this matters

This pattern eliminated all planned C5 (Guarded Agents) runtime code. The original design called for a dedicated `AgentGuardrailListener` CDI bean, new agent-specific guardrail interfaces and annotations, upstream PRs to unseal `GuardrailResult`, and a static guardrail registry. None of this was needed — the existing SPI chain handles everything because agents are processed as AiServices at build time.

### How to discover this

Trace the build-time chain: `AnnotationsImpliesAiServiceBuildItem` → `determinedImpliedRegisterAiService()` (AiServicesProcessor.java:618) → `DeclarativeAiServiceBuildItem` → `gatherMethodMetadata()` → `gatherInputGuardrails()` / `gatherOutputGuardrails()` → `AiServicesRecorder.setMetadata()`. Then trace the runtime chain: `AgentBuilder.build()` → `AiServices.builder()` → `GuardrailService.builder().build()` → `QuarkusClassMetadataProviderFactory.getNonStaticMethodsOnClass()` → reads from `AiServicesRecorder.getMetadata()`.

The key non-obvious hop is that `GuardrailServiceBuilder.computeOutputGuardrailsForAiServiceMethod()` checks `outputGuardrailsAndConfigSetOnBuilder()` first — since nothing is set on the builder by `QuarkusAgenticContextConsumer`, it falls through to the SPI metadata path. Only ONE source of guardrails is active, preventing double execution.

### Why this is non-obvious

The natural assumption is that agents need their own guardrail wiring because they go through `AgenticServices.createAgenticSystem()` rather than `AiServiceMethodImplementationSupport`. The insight is that the inner `AiServices` proxy created by `AgentBuilder.build()` uses the same SPI hooks as any other AiService — and the build-time metadata was already registered because `AnnotationsImpliesAiServiceBuildItem` causes the agent interface to be processed by `AiServicesProcessor`.

### Alternatives considered

- **Explicit instance passing via `AgentBuilder.inputGuardrails()`** — rejected because it risks double execution when the SPI metadata path is also active
- **AgentGuardrailListener (AgentListener-based)** — rejected because `InputGuardrail.validate(UserMessage)` operates on LLM-level types, not the `AgentRequest(agenticScope, agent, inputs)` types that `AgentListener` receives
- **New agent-specific guardrail interfaces** — rejected because the existing `InputGuardrail`/`OutputGuardrail` interfaces work correctly on agents via the SPI chain
