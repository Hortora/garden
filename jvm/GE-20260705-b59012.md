### Symptom

Attempting to implement Convex Combination (weighted score-based fusion) server-side using Qdrant's `QueryFactory.formula(Formula)` as the outer query over prefetch legs. The `Formula` type suggests arbitrary score expressions, but CC requires referencing the per-leg scores from individual prefetch sub-queries (dense, sparse, BM25) to apply configurable weights: `score = α₁·dense + α₂·sparse + α₃·bm25`.

### Context

Qdrant 1.18 `QueryFactory` supports several outer query types over prefetch legs:
- `rrf(Rrf)` — rank-based fusion with configurable K
- `fusion(Fusion.RRF)` — same via the Fusion enum
- `fusion(Fusion.DBSF)` — Distribution-Based Score Fusion (normalizes per-leg scores, combines equally)
- `formula(Formula)` — custom score formula referencing vectors and payloads

The `Fusion` enum (`io.qdrant.client.grpc.Points.Fusion`) has only two values: `RRF` (0) and `DBSF` (1). No Convex Combination variant.

### Root cause

`Formula` can reference:
- Named vector similarity scores (`$score`)
- Payload field values
- Constants and arithmetic operations

It **cannot** reference individual prefetch leg scores. The prefetch architecture produces a merged candidate set — by the time the outer query (Formula or otherwise) runs, the per-leg provenance and per-leg scores are gone. There is no `$prefetch_0_score` or `$leg["dense"]` variable in the Formula DSL.

This is not documented as a limitation — the Formula API's generality suggests it should be able to express any scoring function, but the prefetch merge boundary makes per-leg score references impossible.

### Fix

Implement CC client-side with separate queries per retrieval leg:

```java
// Instead of: prefetch legs → Formula outer query (doesn't work)
// Do: separate SearchPoints per leg → client-side weighted combination

// 1. Run each leg as a standalone query
List<ScoredPoint> denseResults = client.queryAsync(denseQuery).get();
List<ScoredPoint> sparseResults = client.queryAsync(sparseQuery).get();
List<ScoredPoint> bm25Results = client.queryAsync(bm25Query).get();

// 2. Min-max normalize scores within each leg
// 3. Apply configurable weights: score = w_dense * norm_dense + w_sparse * norm_sparse + w_bm25 * norm_bm25
// 4. Merge, deduplicate, sort by weighted score
```

This trades 1 server round trip (prefetch+fusion) for 3 separate round trips, but gives full control over per-leg weights.

**Server-side alternatives that DO work:**
- `Fusion.RRF` — rank-based, no per-leg weights needed
- `Fusion.DBSF` — score-based with distribution normalization, equal weights

### Why this is non-obvious

The `Formula` query type is presented as a general-purpose scoring mechanism. Its ability to reference named vectors and payload values suggests it operates at the individual-point level with full context. The absence of prefetch-leg score references is only discoverable by inspecting the `Formula` protobuf definition and finding no such variables — the Qdrant docs don't state this limitation.

### Why this fix

Client-side CC with separate queries is the only way to get configurable per-leg weights. DBSF is the closest server-side alternative but uses equal weights after normalization — it cannot express `α=0.7 dense + 0.2 sparse + 0.1 bm25`.

### Alternatives considered

- **Qdrant `Formula` with prefetch** — investigated first; rejected because Formula cannot reference per-leg scores (this entry's finding)
- **DBSF as CC approximation** — works server-side but applies equal weights; inadequate when benchmarking specific weight configurations (e.g. Hortora/engine#33 wants α=0.5 dense vs RRF k=60)
- **Custom Qdrant plugin** — would require server-side extension development; disproportionate effort
