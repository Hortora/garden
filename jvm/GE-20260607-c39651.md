### Symptom

A `@QuarkusTest` (or any CDI container) fails to resolve a bean that definitely exists. The failure may be an `UnsatisfiedResolutionException` at startup, an ambiguous bean error, or the wrong bean being injected.

### Context

The bean is annotated with a custom `@Qualifier` (e.g. `@RiskClassifier`, defined as `@Qualifier @Retention(RUNTIME) @Target({TYPE, METHOD, FIELD, PARAMETER}) public @interface RiskClassifier {}`). The injection point uses the concrete type but omits the qualifier.

### Root cause

CDI resolves injection points by type AND qualifier set. A bean annotated with a custom `@Qualifier` has that qualifier in its qualifier set. An injection point with no qualifier carries only the implicit `@Default`. Since the bean's set is `{@RiskClassifier}` and the injection point's set is `{@Default}`, they do not match.

### Fix

```java
// Bean — has @RiskClassifier in its qualifier set
@ApplicationScoped
@RiskClassifier
public class LifeActionRiskClassifier implements ActionRiskClassifier { ... }

// Wrong — injection point has @Default; bean has @RiskClassifier → no match
@Inject
LifeActionRiskClassifier classifier;

// Correct — qualifier on injection point matches the bean
@Inject
@RiskClassifier
LifeActionRiskClassifier classifier;
```

### Why this is non-obvious

When a framework defines a qualifier to enable collection injection (`@Inject @RiskClassifier Instance<ActionRiskClassifier>`), developers naturally annotate their bean implementation with that qualifier — but think of it as "tagging the bean for the framework," not as something that affects ALL injection points for that type. The qualifier feels like it is "for the framework's use" rather than a CDI-level contract that propagates to every site that injects the bean. Direct type-safe injection (`@Inject LifeActionRiskClassifier`) silently breaks because it no longer satisfies the qualifier requirement.
