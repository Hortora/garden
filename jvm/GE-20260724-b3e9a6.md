### What went wrong

`WorkerFunction.Sync` constructor changed from `(Class<T> inputType, Function<T, WorkerResult> fn)` to `(Class<T> inputType, Class<R> outputType, BiFunction<T, WorkerScope, WorkerResult<R>> fn)`. Simultaneously, `WorkerExecutionContext` was removed entirely from `casehub-engine-api` — the class file is absent from the jar.

Call sites fail with two distinct errors:
1. "cannot infer type-variable(s)" on the `new WorkerFunction.Sync<>(...)` constructor
2. "cannot find symbol: class WorkerExecutionContext" on any `WorkerExecutionContext.current()` call

### What was expected

Either a migration path (deprecation of the old constructor) or documentation of the change.

### Root cause

Two changes landed together: (1) `WorkerFunction.Sync` was parameterised with `<T, R>` (input AND output types) instead of just `<T>`, requiring `outputType` as the second constructor arg, and (2) `WorkerScope` replaced `WorkerExecutionContext` as the mechanism for accessing case context — passed as a lambda parameter instead of a thread-local.

### Fix

1. Add `outputType` as the second constructor argument (typically `Map.class`)
2. Change the function from `Function<T, WorkerResult>` to `BiFunction<T, WorkerScope, WorkerResult<R>>`
3. Replace `WorkerExecutionContext.current().caseId()` with `scope.caseId()`

```java
// Before
.function(new WorkerFunction.Sync<>(Map.class,
        MyDescriptor::myMethod))

// After — method reference becomes lambda to accept scope
.function(new WorkerFunction.Sync<>(Map.class, Map.class,
        (input, scope) -> myMethod(input)))

// Before — accessing case context
UUID caseId = WorkerExecutionContext.current().caseId();

// After — scope is a lambda parameter
static WorkerResult myMethod(Map<String, Object> input, WorkerScope scope) {
    UUID caseId = scope.caseId();
    // ...
}
```

### Why this is non-obvious

The two changes are interdependent but produce different error messages. Fixing just the constructor arity still fails because the function type changed from `Function` to `BiFunction`. Fixing just the `WorkerExecutionContext` import still fails because the class was removed, not relocated. The `WorkerScope` replacement is only discoverable by reading the new `WorkerFunction.Sync` record definition in the jar.
