## @ApplicationScoped bean state persists across @QuarkusTest classes — tests pass in isolation but fail in suite

### Symptom

Tests pass when run individually (`mvn test -Dtest=MyTestClass`) but fail with wrong counts, unexpected state, or assertion errors when the full test suite runs. The failure is non-deterministic — depends on test execution order.

### What was tried (didn't work)

- Running `@BeforeEach` cleanup in one test class — doesn't help other test classes that share the same bean
- `@QuarkusTest` `@TestProfile` isolation — still shares the same app instance within a profile

### Root cause

All `@QuarkusTest` classes in a module share a single Quarkus application instance. State accumulated in `@ApplicationScoped` beans (rate limiters, registries, caches, counters) persists between test classes. There is no automatic reset between classes.

### Fix

Three complementary strategies:

```java
// 1. Expose package-private reset hooks on stateful beans
@ApplicationScoped
public class AuthRateLimiter {
    void resetForTest() { this.state.clear(); }  // package-private
}

// 2. Call reset in @AfterEach in every test class that uses the bean
@AfterEach void cleanup() { rateLimiter.resetForTest(); }

// 3. Use unique identifiers per test to avoid cross-test interference
String name = "test-" + UUID.randomUUID();
```

### Why this is non-obvious

Developers accustomed to Spring's per-test context reset or JUnit's per-test instance lifecycle expect a clean slate for each test class. Quarkus's shared-instance model is correct for production fidelity but requires explicit cleanup discipline. The flakiness is order-dependent and hard to reproduce reliably.
