### Symptom

CI fails with a test error deep inside a record constructor — `NullPointerException` from `Objects.requireNonNull` in a compact constructor. The build compiles successfully. All tests pass locally. The failing test is in the downstream repo (engine), but the root cause is a missing commit in the upstream repo (neocortex).

### Context

Multi-repo Maven project where repo A (engine) depends on repo B (neocortex) via SNAPSHOT artifacts published to GitHub Packages. A change in repo A requires a companion change in repo B (making a record field nullable by removing `Objects.requireNonNull`).

The developer commits the repo B change to whatever branch repo B happens to be checked out on (an unrelated feature branch), runs `mvn install` locally to update the local maven cache, and proceeds with repo A. Locally everything works — local maven resolves the updated B artifact.

CI resolves repo B from GitHub Packages, which still has the OLD artifact (the companion commit is on an unrelated branch, never pushed to B's main, never published). The old artifact still has `Objects.requireNonNull` on the field. CI compiles fine because the record parameter types haven't changed — it's still `String`. The NPE surfaces only at test runtime when the downstream code constructs the record with `null`.

### What was tried (didn't work)

1. Re-running CI — same failure (not flaky)
2. Checking the engine test code — test logic was correct
3. Looking at compilation errors — none (the type signature is unchanged)

### Root cause

The companion commit was committed to repo B's feature branch, not repo B's main branch. GitHub Packages only publishes from main. CI resolves the old published artifact where the compact constructor still validates non-null.

### Fix

1. Cherry-pick the companion commit from the feature branch to main in the upstream repo
2. Push main to upstream so CI publishes the updated artifact
3. Remove the commit from the unrelated feature branch (git reset --hard HEAD~1) so it doesn't create noise when that branch closes
4. Re-run downstream CI

Prevention: always check which branch the upstream repo is on before committing. If it's on a feature branch, switch to main first, commit, push, then switch back.

### Why this is non-obvious

The build compiles successfully in CI — the record parameter type is `String` in both old and new versions. The nullability change is in the compact constructor validation (`Objects.requireNonNull`), which is bytecode-level, not signature-level. A developer seeing "build passed, tests failed with NPE in a record constructor" would naturally look at the test code or the record itself, not at which branch the upstream repo was on when the commit was made.

### Alternatives considered

- Pushing the feature branch to upstream and publishing from there — rejected because it would publish unrelated feature work
- Adding a null check in the downstream code instead of making the upstream field nullable — rejected because it's the wrong design (the field genuinely should be nullable)
