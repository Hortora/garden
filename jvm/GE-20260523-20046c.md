## Context

You need a JAR indexed for CDI bean discovery in production but do NOT want those beans active in `@QuarkusTest` — either because they break the test scheduler, introduce unsatisfied dependencies in the test context, or interfere with test-specific alternatives.

## Technique

Use the `%prod.` profile prefix for `quarkus.index-dependency` entries in `src/main/resources/application.properties`:

```properties
# src/main/resources/application.properties

# Only active during quarkus:build (prod profile)
%prod.quarkus.index-dependency.engine.group-id=io.casehub
%prod.quarkus.index-dependency.engine.artifact-id=casehub-engine
%prod.quarkus.index-dependency.engine-common.group-id=io.casehub
%prod.quarkus.index-dependency.engine-common.artifact-id=casehub-engine-common
```

`@QuarkusTest` runs under the `test` profile. Quarkus skips `%prod.` entries during test augmentation, so those JARs are not indexed in the test context.

If the JARs also need indexing in tests (but for different beans, or with different constraints), add a separate entry to `src/test/resources/application.properties` without the profile prefix:

```properties
# src/test/resources/application.properties

# Active in @QuarkusTest (indexes engine-common for JQEvaluator @ApplicationScoped)
quarkus.index-dependency.engine-common.group-id=io.casehub
quarkus.index-dependency.engine-common.artifact-id=casehub-engine-common
```

## Why this is non-obvious

The Quarkus documentation warns that some build-time properties ignore profile prefixes — for example, the `quarkus.arc.selected-alternatives` property notes in its own test file: *"quarkus.arc.selected-alternatives is a build-time property — %test. prefix has no effect."* This creates uncertainty about whether `%prod.` works for `quarkus.index-dependency`. It does — because `index-dependency` is resolved after profile application, unlike `selected-alternatives` which is locked earlier in augmentation. The distinction is not documented.
