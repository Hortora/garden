### Context
Using `JexlBuilder().silent(true).strict(false).create()` to evaluate user-supplied or template-supplied expressions against a `Map<String,Object>` context. A condition like `workItem.priority.name() == 'URGENT'` is evaluated and unexpectedly returns false.

### Behaviour
With `silent(true)`, any JEXL exception — including `NullPointerException` from navigating a null field, `ClassCastException`, or a syntax error — is swallowed and the expression returns `null` (evaluated as false by the caller's `Boolean.TRUE.equals(result)` check).

This means:
1. A condition that references a null field (`workItem.priority` is null) silently returns false
2. A condition with a typo (`workItem.prioritee.name()`) silently returns false  
3. Both are indistinguishable from a condition that correctly evaluates to false

There is no log output, no exception, and no way for the caller to tell which case occurred.

### Root cause
`silent(true)` is documented as suppressing exceptions, but the docs don't make clear that this applies to runtime evaluation failures, syntax errors, and null navigations equally — all three produce the same silent-false result.

### Fix / Mitigation
Document context variable types explicitly so expression authors know to guard against null fields:

```java
// The context variable 'workItem.priority' is a WorkItemPriority enum constant, NOT null-safe.
// Use: workItem.priority != null && workItem.priority.name() == 'URGENT'
// NOT: workItem.priority.name() == 'URGENT'   <- silently false when priority is null
```

For the case where the condition is template-author-supplied (trusted input), add validation at template creation time using `jexl.createExpression(condition)` in a try-catch to catch syntax errors early before they silently block outcomes at runtime.

### Why this is non-obvious
JEXL's `silent(true)` mode is commonly used to make filter rule evaluation non-fatal. Developers set it once and expect expressions to "fail safely." They don't realise that "fail safely" means "silently return false for any error, including null traversal and broken syntax" — making debugging impossible without adding instrumentation.

### Constraints
- Applies to JEXL 3.x with `silent(true)`. With `silent(false)`, exceptions are thrown and distinguishable.
- With `strict(false)`, JEXL permits undefined variables (returns null) — combined with `silent(true)`, this doubles the silent-false paths.
