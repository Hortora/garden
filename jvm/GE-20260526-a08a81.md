## Symptom

`response.getStatus() / 100 != 2` check never executes on the 5xx path ŌĆö the branch is dead code. Integration tests against a real HTTP server reveal failures that unit tests with mocked Response objects masked completely.

## Context

A `@RegisterRestClient` interface method declares `jakarta.ws.rs.core.Response` as its return type. A real (or WireMock) server returns a 4xx/5xx response. The call site has:

```java
Response response = gatewayClient.invokeAgent(request);
if (response.getStatus() / 100 != 2) {   // ŌåÉ never reached
    throw new MyException("HTTP " + response.getStatus());
}
```

The `if` block is never reached on non-2xx ŌĆö a `WebApplicationException` is thrown before the check runs.

## Root cause

The MicroProfile REST Client spec mandates that the client implementation throw `WebApplicationException` (or a subclass) for non-2xx responses when the return type is `Response`. Quarkus uses `ClientWebApplicationException`. `Response.getStatus()` never has the chance to return the error code.

## Fix

Catch `WebApplicationException` explicitly and null-guard `getResponse()` ŌĆö `WebApplicationException` does not guarantee a non-null response:

```java
Response response = gatewayClient.invokeAgent(request);
try {
    if (response.getStatus() / 100 != 2) {
        throw new MyException("HTTP " + response.getStatus());
    }
} catch (WebApplicationException e) {
    int status = e.getResponse() != null ? e.getResponse().getStatus() : -1;
    throw new MyException("HTTP " + status);
} finally {
    response.close();   // always close ŌĆö see companion gotcha on Response.close()
}
```

## Why this is non-obvious

`Response` as a return type is the explicit mechanism for status-code-aware handling ŌĆö the `Response` object exists precisely so callers can inspect any status. Throwing rather than returning feels contrary to that design. Compounding the confusion: unit tests using a mocked gateway client that returns `mock(Response.class)` with `getStatus() = 503` pass correctly ŌĆö the mock bypasses `ClientWebApplicationException` entirely. The bug only surfaces in integration tests against a real HTTP server (e.g. WireMock). The unit test suite gives false confidence.

### Why this fix

Catching `WebApplicationException` preserves the application exception boundary while correctly handling what Quarkus actually throws. Alternatives like `ResponseExceptionMapper` (MicroProfile REST Client's provider-based error handling) are more complex and add a class per error type ŌĆö the direct catch is simpler and self-contained when the REST client is wrapped in a service layer that owns the exception translation anyway.
