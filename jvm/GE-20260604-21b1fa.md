## Symptom

Multi-entity semantic search: fan out across N entities with separate `POST /search`
calls, collect results with their scores, sort merged list by score descending.
Entity A returns results scoring 0.4–0.9. Entity B returns results scoring 0.2–0.9.
The merged sort looks plausible but produces silently wrong relevance ordering.

## Context

Building a multi-entity `MemoryOrder.RELEVANCE` query fan-out. Each entity's search
is a separate POST /search call. The plan was to interleave results across entities by
their `score` fields to produce a globally relevance-ranked list.

## Root cause

The `score_and_rank()` function in `mem0/utils/scoring.py` normalises by `max_possible`:

```python
max_possible = 1.0
if has_bm25:   max_possible += 1.0          # → 2.0
if has_entity: max_possible += 0.5          # → 2.5
combined = min(raw_combined / max_possible, 1.0)
```

`max_possible` is **call-specific** — it depends on whether BM25 results exist for
that entity's memories. Entity A with many text-matched memories may normalise by 2.0;
entity B with no BM25 matches normalises by 1.0. A score of 0.9 in call A and 0.9
in call B do not represent the same underlying relevance — the denominators differ.

Additionally, each call applies its own threshold gate before returning results, so
the result sets are already pre-filtered independently.

## Fix (workaround)

Do not sort merged multi-entity results by score across calls. Instead:
- Preserve per-entity score ordering (each entity's results are internally valid)
- Concatenate in entity fan-out order (entity-order concat, not global score sort)
- Document this as a known limitation: multi-entity RELEVANCE is approximate

The only way to get globally comparable scores is to issue a single POST /search
call without filtering by individual `user_id` (search across all entities at once),
but this breaks entity isolation guarantees.

## Why this is non-obvious

The API returns a `score: float` field. The field name implies it is a globally
comparable similarity score (cosine similarity lives in [-1, 1] and IS comparable
across calls). The BM25 normalisation and per-call `max_possible` variability are
implementation details not exposed in any API documentation — discoverable only via
`mem0/utils/scoring.py`. Developers building multi-entity fans naturally sort by score
and produce subtly wrong results with no error.
