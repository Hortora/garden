### Symptom
Annotation attribute resolves as `null` (treated as "not set") even though the annotation declares a non-void default. Build-time logic silently takes the wrong path â€” no error, no warning.

### Context
Writing a Quarkus extension `@BuildStep` that reads `@RegisterAiService` annotation attributes. The `chatMemoryProvider` attribute defaults to `ChatMemoryProvider.class` (auto-discover), but `instance.value("chatMemoryProvider")` returned `null` when the user didn't set it explicitly.

### What was tried (didn't work)
- `instance.value("chatMemoryProvider")` â€” returns `null` when the attribute isn't explicitly set
- Treating `null` as SKIP (disabled) â€” caused "Tool usage requires chat memory" errors for services that expected the default auto-discover behavior

### Root cause
Jandex's `AnnotationInstance.value(String)` returns only *explicitly set* attribute values. If the user writes `@RegisterAiService` without setting `chatMemoryProvider`, `value("chatMemoryProvider")` is `null` â€” even though the annotation source declares `default ChatMemoryProvider.class`. The annotation default is metadata on the annotation *type*, not on the *instance*.

### Fix
Use `valueWithDefault(IndexView, String)` which reads the annotation type's default from the Jandex index:

```java
// WRONG â€” returns null when user doesn't set the attribute
AnnotationValue val = instance.value("chatMemoryProvider");

// RIGHT â€” returns the declared default (ChatMemoryProvider.class)
AnnotationValue val = instance.valueWithDefault(index, "chatMemoryProvider");
```

The `IndexView` parameter is needed because the default lives on the annotation class definition, which must be looked up in the index.

### Why this is non-obvious
The method name `value()` strongly implies "get the value of this attribute." Nothing in the API name suggests it only returns explicitly-set values. The `valueWithDefault` alternative exists but isn't the obvious first choice. The failure is silent â€” `null` is a valid return, so no exception or warning is thrown. The build step appears to work correctly for services that DO set the attribute explicitly, making the bug intermittent and hard to trace.

### Why this fix
`valueWithDefault(index, name)` is the only Jandex API that resolves annotation defaults. There is no configuration option or alternative â€” you must pass the `IndexView` to let Jandex look up the annotation type's declared defaults.

### Alternatives considered
- Treating `null` as "use default behavior" in the calling code â€” works but duplicates the default logic and diverges if the annotation default changes
- Reading the annotation class directly via reflection â€” breaks the Quarkus build-time model (no runtime classes available during augmentation)
