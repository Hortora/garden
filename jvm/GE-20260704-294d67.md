### Symptom

After process restart, K8s Jobs (or any async worker dispatch) that were scheduled before the crash are never re-dispatched. The engine's recovery service finds the orphaned `WORKER_SCHEDULED` event logs, calls `CompositeWorkerExecutionManager.schedulePersistedEvent()`, but the composite routes to no backend — the capability is silently unrecognised. No error is logged. Cases remain stuck indefinitely.

### Context

Two CDI startup observers run at different priorities:

- `WorkerRecoveryCoordinator.onStart()` at `@Priority(22)` — subscribes to an async Uni that queries the database for orphaned `WORKER_SCHEDULED` events and calls `schedulePersistedEvent()` for each
- `WorkerLifecycleOrchestrator.onStart()` at `@Priority(APPLICATION + 10)` = 2010 — calls `runtime.initialize().await().indefinitely()` on each `WorkerRuntime` bean, which populates the resolver's capability map

Lower numeric priority fires first. Recovery subscribes at 22, orchestrator initializes at 2010.

### Root cause

The capability resolver (e.g. `JobDefinitionResolver`) loads job definitions from Quarkus `Config` in an `initialize()` method. This method is only called by `WorkerRuntime.initialize()`, which is triggered by the orchestrator at priority 2010. Before that call, the resolver's internal `definitions` map is `Map.of()` — empty.

When the recovery Uni completes (database query returns), `CompositeWorkerExecutionManager.schedulePersistedEvent()` iterates backends and calls `backend.supports(capabilityName, tenancyId)`. Each backend delegates to its resolver's `canResolve()` → `capabilities()`. Since the resolver hasn't been initialized yet, `capabilities()` returns empty. No backend matches. The composite logs a warning and returns voidItem. The recovery event is permanently lost.

The race is non-deterministic but heavily biased toward failure: the database query (which involves connection pool initialization on first use) typically completes in hundreds of milliseconds, well before priority 2010 fires. In practice, recovery almost always fails.

### Fix

Add `@PostConstruct` to the resolver so it eagerly initializes from `Config` at bean creation time — before any startup observer fires:

```java
@ApplicationScoped
public class JobDefinitionResolver {

    @Inject Config config;

    private volatile Map<String, JobDefinition> definitions = Map.of();

    @PostConstruct
    void init() {
        initialize(); // reads from Config, populates definitions map
    }

    void initialize() {
        Map<String, Map<String, String>> configJobs = loadConfigJobs();
        // ... parse and populate definitions
        definitions = Map.copyOf(parsed);
    }
}
```

CDI creates the bean and calls `@PostConstruct` before making it available for injection. Since `Config` is injected by CDI (available at bean creation time) and the initialization involves no I/O or external dependencies, this is safe.

After this fix:
- `capabilities()` returns correct results from the moment the bean is created
- `supports()` works before any startup observer fires
- The race between recovery (priority 22) and worker initialization (priority 2010) is eliminated

### Why this is non-obvious

The two priorities are in different modules (`casehub-engine` and `casehub-workers`), so the ordering is invisible unless you read both codebases. The failure is silent — no exception, no error log, just a WARN about "no backend supports capability" that looks like a configuration issue, not a race condition. The resolver works perfectly in all non-recovery paths because normal dispatches happen after startup is complete.

### Alternatives considered

- Moving the orchestrator to a lower priority (e.g. `@Priority(10)`) — rejected because it would block all startup on K8s cluster connectivity checks
- Making recovery wait for orchestrator completion — rejected because it couples two independent modules via a shared latch
