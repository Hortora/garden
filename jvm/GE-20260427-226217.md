### Summary
When adding ctx-optional overloads to a reactive DSL builder, wrap the user's lambda to the internal ctx-first form at store time. The execution hot path (JoinLeftInlet, cross-product evaluation) needs zero changes and has zero overhead.

### Context
A reactive rule DSL has two execution paths both expecting ctx-first consumers: `JoinLeftInlet` calls `consumer.accept(ctx, leftFact, rightFact)` and `evaluateAllCombinations` passes `args = [ctx, fact1, fact2, ...]` to reflection invocation.

Adding ctx-optional overloads (`ifn(Consumer1<B> fn)` alongside `ifn(Consumer2<Context<CTX>, B> fn)`) without this technique would require branching in both execution paths to detect which form was stored.

### Technique
Wrap the no-ctx (or ctx-last) form to ctx-first at the point of storing, before the value reaches the execution layer:

```java
// No-ctx overload — wraps to ctx-first Consumer2 before storing
public From1First<END, CTX, B> ifn(Consumer1<B> fn) {
    storeHead((Consumer2<Context<CTX>, B>)(ctx, b) -> fn.accept(b));
    return this;
}

// Ctx-at-end overload — flips arg order before storing
public From1First<END, CTX, B> ifn(Consumer2<B, Context<CTX>> fn) {
    storeHead((Consumer2<Context<CTX>, B>)(ctx, b) -> fn.accept(b, ctx));
    return this;
}
```

The stored value is always `Consumer2<Context<CTX>, B>` regardless of which user overload was called. `JoinLeftInlet.fire(ctx, left, right)` and the reflection-based `evaluateAllCombinations` need no changes.

### Caveats
- **Scope predicates stored directly** (not wrapped) still need `invokePredicate` to detect arity — wrapping at store time eliminates the need for arity detection in the hot path, but only for predicates that go through wrapped store methods. Raw Object storage (e.g. `ChainScope.filter(Object)`) bypasses wrapping.
- Works well for Permuplate templates via `@PermuteBody`: the body string can include the wrapper lambda and cast, substituting arity-specific types.

### Why this is non-obvious
The obvious approach — store the user's lambda directly and branch at fire time — is simpler to implement initially but couples every execution site to knowledge of the storage format. Wrapping at store time is an inversion: push complexity to the cold path (registration) and keep the hot path (firing) unconditional.
