## @QuarkusIntegrationTest in the runtime module causes class loading failures — separate integration-tests/ module required

### Symptom

`ClassCastException` or `CDIException` with no clear cause when `@QuarkusIntegrationTest` classes are in the same Maven module as `@QuarkusTest` classes. Errors reference types that are clearly on the classpath.

### What was tried (didn't work)

- Moving `@QuarkusIntegrationTest` to a different package within the runtime module — same error
- Checking for duplicate dependencies — none found

### Root cause

`@QuarkusIntegrationTest` tests the **packaged** application, not the running CDI container. When placed in the runtime module alongside `@QuarkusTest` classes, the augmented artifact is loaded twice — once for the CDI container and once for the packaged app test. This double-loading causes `ClassCastException` and `CDIException` as the same class appears as two different class loader instances.

### Fix

Place all `@QuarkusIntegrationTest` classes in a dedicated `integration-tests/` submodule:

```
casehub-work/
├── runtime/           ← @QuarkusTest classes here
├── deployment/
└── integration-tests/ ← @QuarkusIntegrationTest classes here
```

### Why this is non-obvious

The error message (`ClassCastException`, `CDIException`) points to type problems, not to module structure. The fix — moving a test class to a different Maven module — is not an obvious response to a class loading error.
