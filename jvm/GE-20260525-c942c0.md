### Symptom

`Unable to acquire JDBC Connection [Exception in association of connection to existing transaction]` thrown during a `@QuarkusTest` that calls a `@Transactional` coordinator method. The same test passes without the annotation.

### Context

A coordinator CDI bean has `@Transactional` on a method that delegates to services backed by two separate Quarkus datasources:
- Default datasource (`WorkItemService.create()`)
- Named datasource — e.g. `qhorus` (`LedgerWriteService`, `MessageService`)

### Root cause

A `@Transactional` method automatically enlists every datasource connection touched within it into a single JTA transaction. When two separate Hibernate persistence units are involved, this becomes a distributed (XA) transaction. H2 in the standard Quarkus dual-datasource test configuration does not provide XA-capable connections. The JTA transaction manager attempts to associate the second datasource's connection with the already-open transaction and fails.

### Why this is non-obvious

The error message — `Exception in association of connection to existing transaction` — reads like connection pool exhaustion or a thread-safety bug. Nothing in the message mentions XA, distributed transactions, or multiple datasources. Without knowing that `@Transactional` triggers XA enlistment when multiple datasources are touched, the annotation itself is obvious to blame but the mechanism is not.

The test suite runs with H2 (`drop-and-create`), which masks the configuration entirely until `@Transactional` is introduced.

### Fix

Do not use `@Transactional` at the coordinator level when the method touches multiple datasources. Each downstream service should own its own transaction boundary:

```java
// Wrong — creates XA transaction across default + qhorus datasources
@Transactional
public Result investigate(Transaction tx) {
    ledgerService.write(tx);       // qhorus datasource
    workItemService.create(...);   // default datasource
}

// Correct — per-service transaction boundaries
public Result investigate(Transaction tx) {
    ledgerService.write(tx);       // @Transactional on LedgerWriteService
    workItemService.create(...);   // @Transactional on WorkItemService
}
```

If true cross-datasource atomicity is required (e.g. rollback both on failure), use a saga/outbox pattern — not a coordinator-level annotation.
