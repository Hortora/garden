## QuarkusTestProfile.getEnabledAlternatives() scopes a CDI @Alternative to a single @QuarkusTest class

### Stack
Quarkus 3.32.2, JUnit 5

### Symptom
A test double (e.g. `FailingWorkItemStore`) needs to be the active CDI bean for ONE
test class but must not interfere with other `@QuarkusTest` classes in the same module.
Adding it to global `quarkus.arc.selected-alternatives` in `application.properties` leaks
it into all test classes — even those that don't need it.

### Context
`@QuarkusTest` classes share a single Quarkus application context (restarted only when
the `@TestProfile` changes). Test doubles listed in global `selected-alternatives` become
active for every class, creating hidden coupling and pollution risk.

### Fix

1. Define an inner `Profile` class implementing `QuarkusTestProfile`:

```java
@QuarkusTest
@TestProfile(MyTest.Profile.class)
class MyTest {

    public static class Profile implements QuarkusTestProfile {
        @Override
        public Set<Class<?>> getEnabledAlternatives() {
            return Set.of(MyTestDouble.class);
        }
    }

    @Alternative @Priority(2) @ApplicationScoped
    public static class MyTestDouble implements TheInterface {
        // test-only implementation
    }

    // tests...
}
```

2. Remove the test double from global `selected-alternatives` in `application.properties`.

`getEnabledAlternatives()` activates the listed classes *in addition to* global
`selected-alternatives` — only for this profile. Other test classes run without it.
Quarkus restarts the application once when switching to/from this profile.

### Why this is non-obvious
Most Quarkus CDI testing examples use `quarkus.arc.selected-alternatives` in
`application.properties`, which is global and bleeds across all test classes. The
`QuarkusTestProfile.getEnabledAlternatives()` API exists specifically for per-class
scoping but is not prominently documented alongside CDI testing patterns. The key insight:
`@TestProfile` triggers a Quarkus restart, making it the cleanest isolation boundary
available short of a separate module.

### Why this fix
Global `selected-alternatives` is the obvious approach but couples all test classes to
the test double. `getEnabledAlternatives()` activates the alternative only when the
profile is active — zero leakage. The one-restart cost per profile boundary is acceptable
in test suites.
