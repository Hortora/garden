### Symptom
A method M annotated `@Transactional` (REQUIRED) has a `catch (Exception e)` block that
swallows all exceptions to implement a fail-open contract. When an inner CDI bean
(also `@Transactional` REQUIRED) throws a RuntimeException, M's catch block swallows it
and M returns normally. But the caller of M receives an unexpected `TransactionalException`
â€” the fail-open contract silently breaks.

### Context
Occurs when:
1. Outer method M is `@Transactional` (REQUIRED â€” the default)
2. M contains a `try { } catch (Exception e)` block that swallows exceptions
3. An inner `@Transactional(REQUIRED)` CDI call throws a RuntimeException

### Root cause
When the inner `@Transactional(REQUIRED)` bean throws a RuntimeException, ArC's interceptor
marks the **outer** transaction as rollback-only and re-throws the exception into M.
M's catch block swallows the exception. When M returns normally, ArC's interceptor on M
attempts to commit â€” finds the transaction rollback-only â€” and throws
`jakarta.transaction.TransactionalException` (unchecked, wrapping `RollbackException`)
**to M's caller**. The catch block inside M never sees this exception because it fires
from the CDI interceptor AFTER M's method body returns, not from inside it.

### Why this is non-obvious
`catch (Exception e)` is assumed to be a complete exception barrier. It is â€” for code
inside the method body. But CDI `@Transactional` commit/rollback is handled by an
interceptor that wraps the entire method invocation at the proxy level. The interceptor
runs AFTER `return` â€” so any failure at commit time bypasses all in-method catch blocks.
Most developers expect the transaction to commit silently if the method returned normally.

### Fix
Do not put `@Transactional` on the fail-open method. Instead, extract the write pair to
a separate CDI bean with `@Transactional`, and call that bean from the fail-open method.
The fail-open method's catch block then correctly intercepts `TransactionalException` from
the inner bean because the inner bean's interceptor fires before `gateDispatcher.dispatch()`
returns to the caller â€” not after the caller returns. See the companion technique entry
GE-20260603-dfcecc for the pattern.
