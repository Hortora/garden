## Symptom

Tests using `quarkus-oidc-client` take 60–90 seconds to start (or OOMKill) because
a full Keycloak container is being launched — even though every test path uses a
static bearer token and the OIDC client is never called.

## Context

A module exposes two auth paths: a static bearer token (`casehub.platform.scim.token`)
and an OIDC client credentials flow (`quarkus.oidc-client.scim.*`). Tests configure
the static token only. Keycloak DevServices fires anyway because Quarkus sees
`quarkus-oidc-client` on the classpath and starts DevServices unconditionally —
it does not check whether any code path will actually acquire a token at runtime.

## Root Cause

Quarkus DevServices extension activation is classpath-driven, not call-path-driven.
The presence of `quarkus-oidc-client` is sufficient to start Keycloak DevServices
regardless of whether `quarkus.oidc-client.*.auth-server-url` is configured or any
OIDC token acquisition will occur.

## Fix

Disable Keycloak DevServices explicitly in the test `application.properties`:

```properties
# Tests use static token — OIDC client is never called; Keycloak DevServices
# would start a heavy container for nothing.
quarkus.keycloak.devservices.enabled=false
```

**Required condition:** the OIDC client bean must be injected as `Instance<OidcClient>`
(lazy CDI resolution) rather than `OidcClient` directly. With `Instance<>`, Quarkus
starts successfully even though `quarkus.oidc-client.*.auth-server-url` is absent —
the bean is only resolved if the code actually calls `.get()`. Direct injection fails
at startup with a deployment error when the OIDC client is not configured.

## Why This Is Non-Obvious

The connection between extension presence and DevServices activation is not documented
at the `quarkus-oidc-client` level. The symptom (slow/failing container startup) points
at Keycloak internals rather than at the decision to start Keycloak at all. The fix
requires knowing both that DevServices is classpath-triggered AND that `Instance<>` is
needed to survive startup without an `auth-server-url`.
