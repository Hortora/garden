## Symptom

A test delivers a `MimeMessage` with no `Message-ID` header set, polls the IMAP inbox, and asserts:

```java
assertThat(captured.get(0).metadata()).doesNotContainKey("message-id");
```

The assertion fails — `message-id` is present with a value like `<1346354118.0.1780039906512@localhost>`.

## Context

Testing an IMAP inbound connector that skips `message-id` in metadata when the header is absent. The test sets up a `MimeMessage` without calling `setHeader("Message-ID", ...)`, sends it via Greenmail's SMTP, then polls via IMAP.

## Root Cause

Greenmail's embedded SMTP server (and the Jakarta Mail `Transport` layer) automatically generates and injects a `Message-ID` header during the `sendMessage()` call when none is present. This is correct RFC 5321 SMTP behaviour — the submission agent is expected to supply `Message-ID` if missing. By the time the message appears in the IMAP folder, `Message-ID` is always present.

## Fix

Use `MailFolder.appendMessage()` to inject messages directly into the IMAP mailbox, bypassing the SMTP pipeline entirely:

```java
GreenMailUser user = greenMail.getUserManager().getUser("inbox@example.com");
MailFolder inbox = greenMail.getManagers().getImapHostManager().getInbox(user);
inbox.appendMessage(msg, new Flags(), new Date());
// Message is now in IMAP with exactly the headers you set — no SMTP synthesis
```

See companion entry GE-20260529-4691e8 for the full technique.

## Why This Is Non-Obvious

The SMTP spec says user agents "should" set `Message-ID` and submission agents "may" add one if missing. In practice Greenmail's embedded server always adds one. There is no Greenmail configuration to disable this. The only escape is bypassing SMTP entirely.
