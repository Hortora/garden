### Symptom

State machine backed by `ConcurrentHashMap` allows invalid transitions â€” two concurrent `approve()` calls both succeed, or `approve()` and `reject()` both transition from PENDING.

### Context

An `@ApplicationScoped` CDI bean uses `ConcurrentHashMap<String, PendingEntry>` to track approval state. Each entry transitions PENDINGâ†’APPROVED or PENDINGâ†’REJECTED. The map itself is thread-safe, so the code *looks* correct.

### What was tried (didn't work)

```java
public boolean approve(NodeId nodeId, StepAction action, String tenancyId, String approvedBy) {
    String k = key(nodeId, action, tenancyId);
    var entry = entries.get(k);                    // atomic read
    if (entry == null || entry.status() != Status.PENDING) {
        return false;                               // guard looks correct
    }
    var approval = new PlanApproval(entry.planReference(), approvedBy, Instant.now());
    entries.put(k, new PendingEntry(..., Status.APPROVED, approval, null));  // atomic write
    return true;
}
```

Each individual operation (`get`, `put`) is atomic, but the compound getâ†’checkâ†’put sequence is not. Thread A reads PENDING, thread B reads PENDING, both pass the guard, both write â€” last write wins silently.

### Root cause

`ConcurrentHashMap` guarantees atomicity of individual operations, not of sequences of operations. The get-check-put pattern has a check-then-act race identical to the classic `if (!map.containsKey(k)) map.put(k, v)` anti-pattern, but harder to spot because the check is on a *value property* (status field) rather than key presence.

### Fix

Use `ConcurrentHashMap.compute()` for atomic read-check-write:

```java
public boolean approve(NodeId nodeId, StepAction action, String tenancyId, String approvedBy) {
    String k = key(nodeId, action, tenancyId);
    boolean[] transitioned = {false};
    entries.compute(k, (key, entry) -> {
        if (entry == null || entry.status() != Status.PENDING) {
            return entry;  // no change â€” returns false
        }
        transitioned[0] = true;
        return new PendingEntry(entry.planReference(), Status.APPROVED,
                new PlanApproval(entry.planReference(), approvedBy, Instant.now()), null);
    });
    return transitioned[0];
}
```

The `compute()` lambda runs under the segment lock â€” the read, check, and write are a single atomic operation. The `boolean[]` pattern captures the transition result out of the lambda (lambdas can't assign to local variables).

### Why this is non-obvious

The code uses `ConcurrentHashMap` â€” "thread-safe" â€” so the developer reasonably assumes correctness. The race only manifests under concurrent approval attempts for the same node, which is rare in testing but possible in production (two humans approve the same WorkItem simultaneously, or a retry races with the original).
