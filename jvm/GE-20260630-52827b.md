### Symptom
Maven fails to resolve quarkiverse or other third-party dependencies with 401 errors, even though they exist on Maven Central. Adding or fixing GitHub Packages authentication does not help for dependencies that were never published to GitHub Packages.

### Context
When a Maven POM declares a GitHub Packages repository (for casehub SNAPSHOTs), Maven tries that repository for ALL dependencies — including third-party ones like quarkiverse extensions. GitHub Packages returns 401 for artifacts it doesn't host, and Maven treats 401 as a terminal failure — it does not fall back to Central.

### Root cause
Maven's repository resolution tries repositories in declaration order. A 401 response (as opposed to 404) is treated as an authentication failure, not "artifact not found." Maven does not continue to the next repository because it interprets 401 as "you're not authorized to access this artifact" rather than "this artifact doesn't exist here."

### Fix
Declare Maven Central explicitly BEFORE the GitHub Packages repository in the POM:

```xml
<repositories>
    <repository>
        <id>central</id>
        <url>https://repo.maven.apache.org/maven2</url>
    </repository>
    <repository>
        <id>github</id>
        <url>https://maven.pkg.github.com/OWNER/*</url>
    </repository>
</repositories>
```

This ensures Central is tried first for all artifacts. GitHub Packages is only consulted for artifacts not found on Central (i.e., your private SNAPSHOTs).

### Why this is non-obvious
Maven Central is the implicit default repository. Most developers expect it to always be consulted, regardless of other repository declarations. The 401-vs-404 distinction in Maven's fallback behaviour is not documented prominently. Related: GE-20260530-b68c00 covers the `.lastUpdated` caching dimension of this same problem.
