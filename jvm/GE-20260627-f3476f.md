## Symptom

An `@ApplicationScoped` bean needs to delegate to a `@RequestScoped` bean (e.g.
`CurrentPrincipal.tenancyId()`) but is called from mixed contexts: HTTP request
threads (request scope active), virtual threads, `@Scheduled` methods, `@ObservesAsync`
handlers, and startup observers (no request scope).

Using `try { principal.get().tenancyId(); } catch (ContextNotActiveException e) { ... }`
works but is noisy in logs, has exception overhead on every non-request call, and uses
exception-as-control-flow.

## Context

Occurs in any Quarkus application where an `@ApplicationScoped` service wraps or
delegates to `@RequestScoped` identity/context beans and is called from both
request-scoped and system-scoped contexts. Common in multi-tenancy implementations
where a TenantContext must resolve the current tenant from the authenticated principal
in HTTP handlers but fall back to a default in background operations.

## Fix

Check whether the CDI request context is active before accessing the `@RequestScoped`
bean. `Arc.container().requestContext().isActive()` returns `false` in system contexts
without throwing.

```java
@ApplicationScoped
public class DefaultTenantContext implements TenantContext {

    @Inject Instance<CurrentPrincipal> principal;

    @Override
    public String currentTenantId() {
        if (principal.isResolvable()
                && Arc.container() != null
                && Arc.container().requestContext().isActive()) {
            return principal.get().tenancyId();
        }
        return TenancyConstants.DEFAULT_TENANT_ID;
    }
}
```

Three guards in sequence:
1. `principal.isResolvable()` — bean exists and is unambiguous
2. `Arc.container() != null` — CDI container is running (fails in plain JUnit)
3. `Arc.container().requestContext().isActive()` — request scope is live on this thread

The `Instance<T>` injection (not direct `@Inject T`) is critical — it defers resolution
and prevents CDI from creating a proxy that would throw on the first method call.

## Why this is non-obvious

The obvious approach is `try/catch ContextNotActiveException`, which is what
`QhorusInboundCurrentPrincipal` uses internally. But that approach fires the exception
on every background-thread call — hundreds of times per minute in a system with
scheduled tasks and virtual thread watchers. The scope-active check is a clean boolean
conditional with no exception overhead. `Arc.container().requestContext()` is not
prominently documented for this use case — most Quarkus examples show the try/catch
pattern or avoid the problem by keeping everything `@RequestScoped`.

### Alternatives considered

- **try/catch ContextNotActiveException** — works but noisy; exception object allocation
  on every background call; log-polluting if any interceptor logs exceptions
- **Separate @DefaultBean for background contexts** — requires two beans, caller must
  know which to inject; breaks when a bean is called from both contexts
- **ThreadLocal or manual scope activation** — over-engineered; doesn't compose with
  Quarkus's own scope management
