### What went wrong

`SettingsScope.root()` was changed to `SettingsScope.root(String tenancyId)` in casehub-platform-api. The zero-arg `root()` factory was removed entirely. Call sites compile-fail with "actual and formal argument lists differ in length" — but only when the dependency snapshot updates, with no deprecation warning in the prior version.

### What was expected

A deprecation cycle or at least a compiler warning before removing the zero-arg factory.

### Root cause

Platform-api is pre-release (`0.2-SNAPSHOT`), so the breaking change was made directly. The `SettingsScope` record was updated to require `tenancyId` in all construction paths — `root()`, `of()`, and the canonical constructor.

### Fix

For methods that already have `tenancyId` in scope (e.g. `onFault(String tenancyId, ...)`, `recompile(String tenancyId, ...)`):
- Thread `tenancyId` through to `resolveConfiguration(tenancyId)` and call `SettingsScope.root(tenancyId)`

For global/platform-scoped lookups without a tenant context (e.g. `resyncIntervalFor(NodeType type)`):
- Use `SettingsScope.root(TenancyConstants.PLATFORM_TENANT_ID)` — the platform sentinel for tenant-agnostic config

```java
// Before
Preferences prefs = preferenceProvider.resolve(SettingsScope.root());

// After — with tenancyId available
Preferences prefs = preferenceProvider.resolve(SettingsScope.root(tenancyId));

// After — global config, no tenant context
import io.casehub.platform.api.identity.TenancyConstants;
Preferences prefs = preferenceProvider.resolve(SettingsScope.root(TenancyConstants.PLATFORM_TENANT_ID));
```

### Why this is non-obvious

The error message ("actual and formal argument lists differ in length") points at the call site but doesn't name the missing parameter. Without reading the updated `SettingsScope` source, you'd search for overloads or API changes rather than realising `root()` now requires a tenant ID. The fix for global lookups (`PLATFORM_TENANT_ID`) is not documented anywhere.
