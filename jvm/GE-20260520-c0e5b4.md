## Podman on macOS requires explicit DOCKER_HOST for Testcontainers

**ID:** GE-20260520-c0e5b4
**Stack:** Podman 5.x, macOS arm64, Testcontainers 2.x

### Symptom

Testcontainers (and any tool relying on a Docker socket) reports "no valid Docker
environment" and fails immediately, even though Podman is installed and running.

```
ERROR Could not find a valid Docker environment.
UnixSocketClientProviderStrategy: failed with exception InvalidConfigurationException
(Could not find unix domain socket). Root cause NoSuchFileException (/var/run/docker.sock)
```

### Context

Fresh Podman installation on macOS without running the `podman-mac-helper` installer.
Podman's default machine configuration does not create `/var/run/docker.sock`. The
system helper service must be explicitly installed to create that socket, but the
installer step is optional and easy to skip.

### What was tried (didn't work)

- Verifying `podman machine list` — machine shows as available
- Running `podman info` — Podman reports healthy
- Checking `/var/run/docker.sock` — does not exist

### Root cause

Podman on macOS forwards the Docker API through a machine-specific socket at a path
inside `/var/folders/.../T/podman/`. This path is printed by `podman machine start`
but is not the well-known `/var/run/docker.sock`. Testcontainers only knows about
the standard path unless told otherwise.

### Fix

**Step 1 — Start the Podman machine:**
```bash
podman machine start
```
The output prints the API socket path:
```
API forwarding listening on: /var/folders/8m/.../T/podman/podman-machine-default-api.sock
...
export DOCKER_HOST='unix:///var/folders/8m/.../T/podman/podman-machine-default-api.sock'
```

**Step 2 — Export the socket path before running tests:**
```bash
export DOCKER_HOST='unix:///var/folders/<path>/podman/podman-machine-default-api.sock'
```

**Step 3 — Run tests with the env var set:**
```bash
DOCKER_HOST=$DOCKER_HOST TESTCONTAINERS_RYUK_DISABLED=true mvn test -pl <module>
```

Alternatively, install `podman-mac-helper` to create `/var/run/docker.sock` permanently:
```bash
sudo /opt/homebrew/Cellar/podman/5.8.2/bin/podman-mac-helper install
podman machine stop && podman machine start
```

### Why this is non-obvious

The error message mentions Docker and `/var/run/docker.sock` — making it look like a
Docker installation problem rather than a Podman socket routing issue. `podman machine start`
does print the correct `export DOCKER_HOST=...` command, but this is easy to miss in the
startup output, and Testcontainers provides no hint that `DOCKER_HOST` is the fix.

### Why this fix

Setting `DOCKER_HOST` is the zero-install path — useful in CI or when root access is
unavailable for the helper installer. The `podman-mac-helper` approach is the permanent
solution for local development if root access is available.

## Update — discovering the Podman context when `docker info` fails

**Scenario:** `docker info` (or any Docker-API tool) fails with "cannot connect to daemon".
Docker Desktop is not installed. The machine may be stopped, not just missing a socket.

**Discovery checklist:**

```bash
# 1. Does the Docker CLI know about Podman?
docker context ls
# Look for a podman-machine-default context with a unix:// URI

# 2. Is the Podman machine running?
podman machine list
# 'Last UP' shows when it last ran — if it was days ago, the machine is stopped

# 3. If stopped, start it:
podman machine start
# Output prints the DOCKER_HOST value to export
```

**Key insight:** `docker context ls` reveals the Podman socket configuration even when the
machine is stopped — the context entry persists regardless of machine state. This is the
fastest way to confirm Podman is the Docker provider and get the correct socket path,
without having to know Podman was installed at all.

## Variant — Podman machine restart loses the API socket (gvproxy timing)

**When it occurs:** After `podman machine stop && podman machine start`, the
`-api.sock` file listed in `~/.testcontainers.properties` or `DOCKER_HOST` no longer
exists. Tests fail even though the machine is running. `podman ps` works fine (it uses
the native Podman socket, not the Docker-compatible API socket).

**Root cause:** The Docker-compatible API socket (`podman-machine-default-api.sock`) is
created by `gvproxy` — a separate process that starts after the VM. There is a ~5 second
delay between `podman machine start` returning and gvproxy creating the API socket. If
tests start immediately, the socket file is absent. Also, if the machine was restarted
with a port conflict, gvproxy may fail silently without creating the socket at all.

**Diagnosis:** Check whether the socket file exists:
```bash
ls /var/folders/.../T/podman/podman-machine-default-api.sock
# If missing, gvproxy hasn't finished yet (or failed)
ps aux | grep gvproxy  # confirm gvproxy is running
```

**Fix:** Add a short wait after `podman machine start`, then confirm the socket exists:
```bash
podman machine stop && podman machine start && sleep 5
ls /var/folders/.../T/podman/podman-machine-default-api.sock  # must exist
```

If the socket path in `~/.testcontainers.properties` became stale (e.g. after a VM port
reassignment), update it to the current path printed by `podman machine start`:
```
export DOCKER_HOST='unix:///var/folders/<new-path>/podman-machine-default-api.sock'
```
Write this path to `~/.testcontainers.properties`:
```properties
docker.host=unix:///var/folders/<new-path>/podman-machine-default-api.sock
```

**Verify:** Test the socket with curl before running Maven:
```bash
curl --unix-socket /var/folders/<path>/podman-machine-default-api.sock \
  http://localhost/v1.41/version 2>/dev/null | python3 -c "import json,sys; print(json.load(sys.stdin)['ApiVersion'])"
# Should print: 1.44
```

**Permanent fix:** Install `podman-mac-helper` — creates `/var/run/docker.sock`
permanently and survives restarts without path changes:
```bash
sudo /opt/homebrew/Cellar/podman/5.8.2/bin/podman-mac-helper install
podman machine stop && podman machine start
```

## Alternative — SSH tunnel to Podman VM socket

**When to use:** The `-api.sock` path is a broken symlink (target in `/var/folders/T/podman/`
doesn't exist), or the machine has been restarted and gvproxy hasn't recreated it yet.
An SSH tunnel creates a stable socket at a predictable path independently of gvproxy.

**How:** The Podman VM exposes an SSH daemon. Tunnel from a local Unix socket on macOS
to the VM's internal Podman socket:

```bash
# Find the SSH port from the connection URI
podman system connection ls
# e.g. ssh://core@127.0.0.1:51192/run/user/501/podman/podman.sock

ssh -nNT \
  -L /tmp/podman-tc.sock:/run/user/501/podman/podman.sock \
  -i ~/.local/share/containers/podman/machine/machine \
  -o StrictHostKeyChecking=no \
  core@127.0.0.1 -p 51192 &

export DOCKER_HOST=unix:///tmp/podman-tc.sock
export TESTCONTAINERS_RYUK_DISABLED=true
mvn test
```

**Trade-offs vs `-api.sock` path:**
- Pros: stable `/tmp/podman-tc.sock` path (not machine-specific); works when gvproxy
  socket is absent; no `podman-mac-helper` needed
- Cons: SSH tunnel process must be running before tests start; kills if shell exits;
  requires knowing the SSH port (from `podman system connection ls`)

**Verify:**
```bash
curl --unix-socket /tmp/podman-tc.sock http://localhost/_ping
# Should return: OK
```
