### Symptom

Logging `bean.getClass().getSimpleName()` on an injected CDI bean prints the ARC proxy class name (e.g. `EndpointRegistry$_ClientProxy_1234`) rather than the implementation class name.

### Context

Any `@ApplicationScoped` or `@RequestScoped` CDI bean accessed through injection. Commonly surfaces in startup logs where you want to show which implementation was selected by CDI.

### Root cause

Quarkus ARC wraps `@ApplicationScoped` and `@RequestScoped` beans in a client proxy. `getClass()` on an injected bean returns the proxy class, not the implementation. The proxy class name includes a generated suffix.

### Fix

`bean.getClass().getSuperclass().getSimpleName()` — the proxy's direct superclass is always the actual implementation. Reliable because Quarkus ARC never generates multi-level proxy chains.

```java
LOG.infof("Loaded %d endpoints into %s",
    count, registry.getClass().getSuperclass().getSimpleName());
```

### Why this is non-obvious

`getClass()` always returns the runtime type. Developers expect that to be the concrete implementation class; they don't know that for `@ApplicationScoped` beans the runtime type is a generated proxy subclass. The bug is silent — the log line looks reasonable, it just prints an ugly generated name instead of the implementation name.

### Constraints

Relies on ARC generating single-level proxy chains. Quarkus ARC documents this as an implementation property — multi-level proxies are not generated. Would need revisiting if ARC changed this, but there is no indication it will.
