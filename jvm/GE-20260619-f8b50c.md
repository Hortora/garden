## Install engine module from specific git commit to match published SNAPSHOT

**ID:** GE-20260619-f8b50c
**Stack:** Maven, GitHub Packages, Git
**Labels:** `#snapshot` `#ci-divergence` `#dependency-alignment`
**What it achieves:** Lets you verify your code compiles against the exact SNAPSHOT version that CI will use, when your local `.m2` has a newer (unpublished) version.
**Context:** When an upstream repo (e.g. engine) has commits on main that haven't been published to GitHub Packages yet (because its own CI is red), your local `mvn install` of that repo installs the new API. Your downstream code compiles locally but fails on CI because CI pulls the older published SNAPSHOT.

### The technique
1. Find the commit that introduced the API change:
```bash
git -C /path/to/engine log --oneline -- path/to/ChangedFile.java
```

2. Checkout the file from the parent commit, install just that module, then restore:
```bash
git -C /path/to/engine checkout <parent-sha> -- module/src/main/java/path/to/File.java
mvn install -DskipTests -pl module -f /path/to/engine/pom.xml
git -C /path/to/engine checkout main -- module/src/main/java/path/to/File.java
```

3. Now run your downstream build — it compiles against the same API that CI will see.

### Alternatives considered
- Pull the published SNAPSHOT from GitHub Packages directly — requires a valid PAT with `read:packages` scope, which may not be configured for local dev
- Just push and let CI tell you — wastes a CI cycle (5+ minutes) per attempt

### Why this is non-obvious
The natural instinct when local passes but CI fails is to check environment differences (JDK, OS, test infra). The actual cause — SNAPSHOT version divergence between local `.m2` and GitHub Packages — is invisible because `mvn verify` doesn't report which SNAPSHOT timestamp it resolved. The technique of selectively installing a single file from git history to simulate the published state is not a standard Maven workflow.

### When to use it
- Upstream CI is red (so new API not published)
- Your code uses the new API but CI will get the old one
- You want to verify locally before pushing rather than using CI as a test runner
