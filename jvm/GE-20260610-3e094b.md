### Context

You have a class that starts mutable (must be built up before use) and needs to become effectively read-only after a specific initialization point. Adding a separate read-only wrapper type forces callers to manage two types. Adding an `Unmodifiable*` wrapper loses access to the concrete type's internal methods.

### What developers typically do

Create a separate immutable type or an `Collections.unmodifiableMap()` wrapper — both require a type change at the boundary and lose the concrete type's methods.

### The technique

Add a `volatile boolean frozen` flag and a private `checkWritable()` guard:

```java
public class WritablePanelImpl implements WritablePanel {
    private volatile boolean frozen = false;

    public WritablePanelImpl freeze() {
        this.frozen = true;
        return this; // chainable
    }

    @Override
    public boolean isReadOnly() { return frozen; }

    private void checkWritable() {
        if (frozen) throw new ReadOnlyPanelException(panelName);
    }

    @Override
    public WritablePanel set(String key, Object value) {
        checkWritable();
        // ... normal implementation ...
    }

    // All write methods call checkWritable() as the first line
}
```

The same class serves as both the mutable builder and the frozen read-only value — callers receive it typed as `ReadablePanel` (a read-only interface) after freeze, so `set()` is not even visible to them.

### Why this is non-obvious

The obvious alternatives (wrapper class, builder/value split, `Collections.unmodifiable*`) all require managing two representations. The freeze flag keeps one class while providing a clear lifecycle contract. The `volatile` keyword is essential — without it, the frozen state may not be visible to threads that read the object after freezing.

### Constraints

Works best when the class is internal and the read-only interface (`ReadablePanel`) is returned at API boundaries. If the writable type leaks to external callers, they can still cast and call `set()` — the check is runtime, not compile-time.
