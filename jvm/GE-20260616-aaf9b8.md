## H2 MERGE USING subquery: column aliases matching PostgreSQL type names are parsed as type casts

**ID:** GE-20260616-aaf9b8
**Stack:** H2 2.4.x, MODE=PostgreSQL

### Symptom

```
org.h2.jdbc.JdbcSQLNonTransientException: Unknown data type: "TID"; SQL statement:
MERGE INTO t USING (SELECT CAST(? AS UUID) AS sid, ? AS tid) AS s ON ...
```

### Context

Adding a composite primary key to a table that was previously keyed by a single UUID.
The MERGE USING subquery adds a second column alias (`tid`) for the new VARCHAR key.
The column alias is treated as a type name by H2, causing an `Unknown data type` error.

### What was tried

Using `? AS tid` as a column alias in the USING subselect — standard SQL alias syntax,
and accepted by PostgreSQL. H2 in `MODE=PostgreSQL` rejects it.

### Root cause

H2 in `MODE=PostgreSQL` intercepts column alias names in `USING (SELECT ...)` subqueries
and attempts to resolve them as SQL type names. `tid` is a valid PostgreSQL data type
(tuple identifier), so H2 interprets `? AS tid` as `CAST(? AS TID)` — a type cast to an
unknown H2 type. Standard SQL's `AS alias` syntax is violated.

### Fix

Use an explicit `CAST(? AS VARCHAR)` and a non-type-name alias:

```sql
-- Before (breaks in H2 MODE=PostgreSQL):
MERGE INTO t USING (SELECT CAST(? AS UUID) AS sid, ? AS tid) AS s
ON t.subject_id = s.sid AND t.tenancy_id = s.tid ...

-- After (works in both H2 and PostgreSQL):
MERGE INTO t USING (SELECT CAST(? AS UUID) AS sid, CAST(? AS VARCHAR) AS tval) AS s
ON t.subject_id = s.sid AND t.tenancy_id = s.tval ...
```

The explicit `CAST(? AS VARCHAR)` resolves the ambiguity — H2 sees a type cast, not an alias.
The alias `tval` does not match any known PostgreSQL type name.

### Why this is non-obvious

Column aliases are standard SQL — `SELECT expr AS name` is universally documented as aliasing.
No SQL standard or H2 documentation warns that aliases in `USING` subqueries may be treated
as type names in compatibility modes. PostgreSQL accepts the original statement without error,
so the problem only surfaces when running H2 tests against a schema designed for PostgreSQL.
The error message `Unknown data type: "TID"` points to a type interpretation rather than a
syntax error, but this direction is completely unexpected for someone writing alias syntax.
