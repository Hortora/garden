### Symptom
A reflection test checking that all SPI methods return `Uni<T>` passes even when no
methods match, because the `isAssignableFrom` call has the receiver and argument
swapped:
```java
.filter(m -> m.getReturnType().isAssignableFrom(Uni.class))  // always false
```

### Context
Occurs in `@QuarkusTest`-free structural tests that verify a reactive SPI mirrors
its blocking counterpart. The test compiles and runs green — zero failures — but the
assertion is vacuous and catches nothing.

### Root cause
`A.isAssignableFrom(B)` means "can B be assigned to A?" — i.e. B is A or a subtype
of A. When checking "does the return type implement Uni?", the question is
"can Uni hold the return type?", which is `Uni.class.isAssignableFrom(m.getReturnType())`,
not the reverse. The inverted form always returns false because `Uni` is not a
supertype of arbitrary return types.

### Fix
```java
// Wrong — always false, test is vacuous
.filter(m -> m.getReturnType().isAssignableFrom(Uni.class))

// Correct — true when return type is Uni or a subtype
.filter(m -> Uni.class.isAssignableFrom(m.getReturnType()))
```

### Why this is non-obvious
Both forms compile. The test runs green either way. There is no warning, no assertion
error, and no stack trace — the filter simply matches nothing and the count assertion
happens to pass against zero. Caught only in code review when a reviewer noticed the
count was zero before the assertion.
