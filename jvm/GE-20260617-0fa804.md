## Context

A dependency changes a method's return type across SNAPSHOT builds — e.g. `void signal()`
becomes `CompletionStage<Void> signal()`. Code compiled against one version throws
`NoSuchMethodError` at runtime against the other. `catch (Exception e)` doesn't help
because `NoSuchMethodError` extends `Error`, not `Exception`. Changing the call site to
match the new API fixes local builds but breaks CI if CI pulls an older SNAPSHOT.

## Technique

Use `Class.getMethod()` to resolve the method by name and parameter types, ignoring the
return type. `Method.invoke()` works regardless of whether the method returns `void` or
`CompletionStage<Void>`. Handle both return types by checking `instanceof`:

```java
static void signalCompat(SomeService service, UUID id, String key, Object value) {
    try {
        Method m = SomeService.class.getMethod("signal", UUID.class, String.class, Object.class);
        Object result = m.invoke(service, id, key, value);
        if (result instanceof CompletionStage<?> cs) {
            cs.toCompletableFuture().join();
        }
        // void return: result is null — nothing to join
    } catch (Throwable t) {
        LOG.warnf(t, "signal() failed for %s — continuing", id);
    }
}
```

The `catch (Throwable t)` is essential — `Method.invoke()` wraps checked exceptions in
`InvocationTargetException`, and `NoSuchMethodError` from any remaining descriptor
mismatch extends `Error`.

## Why non-obvious

Most developers either update the call site (fixing one version, breaking the other) or
pin the dependency version (which may not be possible during an active SNAPSHOT window).
Reflection-by-name naturally abstracts over return type — developers rarely reach for
reflection for API version compatibility because it feels heavyweight for what looks like
a simple API change.

The technique is especially useful in multi-environment scenarios where CI uses a different
SNAPSHOT build than local development, and both must work simultaneously without a release.
