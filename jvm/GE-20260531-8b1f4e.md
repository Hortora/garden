### What happened

A new SPI interface was placed in `runtime/internal/recovery/spi/` because it was intended only for recovery services in the `runtime` module. But `persistence-hibernate` (a lower-level module) needed to implement that interface — and `persistence-hibernate` does not and cannot depend on `runtime` (that would invert the dependency graph: `runtime` depends on `persistence-hibernate`, not the reverse).

The result: `JpaEventLogRepository implements CrossTenantEventLogRepository` failed to compile because `CrossTenantEventLogRepository` was not on `persistence-hibernate`'s classpath.

### Why this is non-obvious

The placement logic was sound: "this is internal to recovery, put it in the recovery package." The error is that package-level privacy doesn't help when the _implementing_ class is in a lower module tier. Interface placement must follow the module dependency graph, not the intended caller's package.

**The rule:** An interface must live in a module that is a _dependency_ of (or equal to) every module that implements it. If module A implements an interface, the interface must be in a module that A already depends on.

### Fix

Move the interface to the lowest-level module that all implementors depend on. In a typical layered architecture:

```
common (SPI interfaces) ← persistence-hibernate (implementations)
                        ← persistence-memory (implementations)
                        ← runtime (callers)
```

Cross-tenant SPI interfaces that are implemented by persistence modules must live in `common`, not in `runtime`. The "internal use only" intent is enforced by naming convention and Javadoc, not by module placement.

```java
// ✗ Wrong: persistence-hibernate can't see this
package io.casehub.engine.internal.recovery.spi;  // in runtime module

// ✓ Correct: common is a dep of persistence-hibernate
package io.casehub.engine.common.spi;  // in common module

/**
 * Cross-tenant access for startup recovery services only.
 * Not for general use — inject only in recovery service classes.
 */
public interface CrossTenantEventLogRepository { ... }
```

### Context

Occurred during multi-tenancy migration that introduced `CrossTenantEventLogRepository` and `CrossTenantCaseInstanceRepository`. Initial placement in `runtime/internal/recovery/spi/` was logical from the caller's perspective but violated the compile-time dependency direction. Both interfaces were moved to `common/spi/` to fix the build.
