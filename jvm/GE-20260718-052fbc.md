### Symptom
`NoClassDefFoundError: Could not initialize class io.smallrye.faulttolerance.api.SpiAccess$Holder` when calling `Guard.create()` in a plain JUnit 5 test (no `@QuarkusTest`).

### Context
Using SmallRye Fault Tolerance's programmatic `Guard` API in a CDI bean that is also unit-tested by direct construction (`new MyBean(deps...)`). The code compiles fine — the error only appears at runtime.

### What was tried (didn't work)
- Assumed the `quarkus-smallrye-fault-tolerance` dependency was sufficient for all usage contexts.

### Root cause
`Guard.create()` uses Java SPI (`ServiceLoader`) to find the SmallRye FT implementation. In `@QuarkusTest`, the Quarkus extension provides this via CDI. In plain JUnit tests, no SPI provider is on the classpath.

### Fix
Add `smallrye-fault-tolerance-standalone` as a test-scope dependency:

```xml
<dependency>
    <groupId>io.smallrye</groupId>
    <artifactId>smallrye-fault-tolerance-standalone</artifactId>
    <scope>test</scope>
</dependency>
```

This provides a standalone SPI implementation that works without CDI.

### Why this is non-obvious
The error message (`NoClassDefFoundError` on an internal `SpiAccess$Holder` class) gives no indication that a separate artifact is needed. The API artifact (`smallrye-fault-tolerance-api`) compiles fine — the failure is purely runtime. Most developers would search for classloader issues rather than a missing SPI provider.
