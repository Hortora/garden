### Symptom

JPA entity with a `JSONB` column stores JSON correctly but deserialization fails on read:

```
MismatchedInputException: Cannot construct instance of `java.util.LinkedHashMap`
(although at least one Creator exists): no String-argument constructor/factory
method to deserialize from String value ('{"key":"value","other":42}')
```

The JSON content is valid — the error is about how it's delivered to Jackson, not about
the JSON itself.

### Context

Occurs when:
- JPA entity declares `@Column(columnDefinition = "JSONB")` or the Flyway migration uses `JSONB`
- Tests run against H2 with `MODE=PostgreSQL` (`jdbc:h2:mem:test;MODE=PostgreSQL`)
- Application code calls `objectMapper.readValue(entity.jsonColumn, new TypeReference<Map<String, Object>>() {})`

The same code works perfectly against real PostgreSQL.

### Root cause

H2's PostgreSQL compatibility mode accepts `JSONB` as a column type but stores values
differently than PostgreSQL. When Hibernate reads the column back, H2 delivers the JSON
wrapped in an extra layer of string quoting. Jackson receives a Java `String` object
containing the JSON text, but `readValue` expects to parse raw JSON input — not a
pre-wrapped String. The mismatch triggers the "no String-argument constructor" error
because Jackson tries to construct a `LinkedHashMap` from a single String argument.

PostgreSQL's JDBC driver returns JSONB column values as properly parsed JSON text that
Jackson can deserialize directly. H2's driver wraps it as a quoted string value.

### What was tried (didn't work)

- Custom Jackson deserializer for the Map type — adds complexity for a test-only issue
- `@Convert` JPA converter with explicit String unwrapping — fragile, differs between H2 and PostgreSQL
- H2's `CAST(column AS VARCHAR)` in queries — doesn't fix the Hibernate column read path

### Fix

Declare JSON-serialized columns as `TEXT` instead of `JSONB`:

**Flyway migration:**
```sql
-- Instead of:  features JSONB NOT NULL DEFAULT '{}'
-- Use:
features TEXT NOT NULL DEFAULT '{}'
```

**JPA entity:**
```java
@Column(name = "features", nullable = false, columnDefinition = "TEXT")
public String features;
```

PostgreSQL treats `TEXT` JSON identically for read/write via `objectMapper.readValue()`.
The only loss is PostgreSQL's server-side JSON validation and GIN indexing on JSONB —
neither matters for applications that deserialize to Java objects for processing
(e.g. filter-then-score CBR retrieval).

### Why this is non-obvious

The error message points at Jackson ("no String-argument constructor") which makes
developers look for Jackson configuration issues or missing type adapters. The actual
problem is in the H2 → Hibernate → Jackson pipeline: H2 delivers the value in a
format Jackson doesn't expect. The fix has nothing to do with Jackson — it's changing
the column type in the DDL.

### Why this fix

Using TEXT over JSONB is a one-line change that eliminates the H2/PostgreSQL divergence
entirely. The alternative approaches (custom deserializers, JPA converters, H2-specific
SQL) all add complexity to handle a test-database quirk. The existing `neocortex-memory-jpa`
module — a production JPA memory store — uses TEXT for its JSON `attributes` column,
validating this as an established pattern.

### Caveats

If you need PostgreSQL GIN indexing on JSON fields (e.g. `WHERE features @> '{"key": "value"}'`
queries), you need JSONB. In that case, use Testcontainers PostgreSQL for tests instead of H2.
The `neocortex-memory-jpa` module provides a reference for this setup with split Surefire
executions (PostgreSQL Testcontainers first, H2 second).
