### Symptom

`@WithSession("qhorus")` (or `@WithSession`) is applied to a CDI bean method. The method is called — but not directly from JAX-RS or a Vert.x event loop handler. It is called after `runSubscriptionOn(Infrastructure.getDefaultWorkerPool())` in a Mutiny chain. The call fails at runtime:

> Hibernate Reactive Panache requires a safe (isolated) Vert.x sub-context, but the current context hasn't been flagged as such.

The caller IS on a Vert.x worker thread (`Vertx.currentContext()` returns non-null), but the error still fires.

### Root cause

`SessionOperations.vertxContext()` in `quarkus-hibernate-reactive-panache-common` calls:

```java
VertxContextSafetyToggle.validateContextIfExists(ERROR_MSG, ERROR_MSG);
```

This check requires a **safe (isolated) Vert.x sub-context** — specifically a duplicated context that has been flagged as safe. Normal Vert.x worker pool threads carry a Vert.x context but it is NOT flagged as safe. Only Vert.x event loop threads (with their duplicated context) and contexts explicitly created with `Vertx.getOrCreateContext().duplicate()` pass the check.

`runSubscriptionOn(Infrastructure.getDefaultWorkerPool())` moves execution to a non-event-loop thread. Any reactive Panache call chained after it will fail with this error.

### Fix

Call the `@WithSession`-annotated method BEFORE the `runSubscriptionOn()` step in the Uni chain, from the event loop thread. Use `Uni.combine()` if you need to run both concurrently:

```java
// WRONG: resolve() called after worker pool switch → FAILS
Uni<Void> setup = Uni.createFrom()
    .<Void>item(() -> { setupSession(); return null; })
    .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());

return setup.flatMap(ignored -> resolver.resolve(channelId, correlationId));  // ← FAILS

// CORRECT: build both Unis on the event loop, then combine
Uni<Void> setup = Uni.createFrom()
    .<Void>item(() -> { setupSession(); return null; })
    .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());

Uni<Optional<UUID>> causedBy = resolver.resolve(channelId, correlationId);  // ← called on event loop

return Uni.combine().all().unis(setup, causedBy).asTuple()
    .map(tuple -> buildResult(tuple));
```

The CDI proxy intercepts `resolver.resolve()` on the event loop, `@WithSession` fires with the correct safe sub-context, and the session is bound. When `Uni.combine()` later subscribes to the causedBy Uni, the reactive Panache query runs within that session — regardless of which thread subscribes.

### Why this is non-obvious

`Vertx.currentContext()` returning non-null is not sufficient. The distinction between a safe (isolated) sub-context and a plain Vert.x context is internal to Quarkus's safety toggle mechanism and is not documented in `@WithSession` Javadoc or the Hibernate Reactive Panache docs. The error message mentions "safe (isolated) Vert.x sub-context" without explaining which thread types qualify. JAX-RS method threads pass (they get a duplicated context), worker pool threads don't.
