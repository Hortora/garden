## Context
When writing an ArchUnit structural rule (e.g. a parity check, a naming convention),
you want to verify the rule actually catches violations — not just that it passes on
valid code. The standard `rule.check(classes)` throws `AssertionError` on any violation,
making it unusable for asserting that a violation *should* be detected.

## Technique
Use `rule.evaluate(classes)` instead of `rule.check(classes)`. This returns an
`EvaluationResult` without throwing, which exposes `hasViolation()`:

```java
// Synthetic inner classes with deliberate violations
static class SyntheticService {
    public KeyRotationEntry recordSomething() { return null; }
}
static class ReactiveSyntheticService {
    public Uni<Void> recordSomethingAsync() { return null; } // wrong: should be Uni<KeyRotationEntry>
}

@Test
void rule_catchesUniTypeArgMismatch() {
    JavaClasses syntheticClasses = new ClassFileImporter()
            .importClasses(SyntheticService.class, ReactiveSyntheticService.class);

    EvaluationResult result = classes()
            .that().haveSimpleName("ReactiveSyntheticService")
            .should(myParityCondition(syntheticClasses))
            .evaluate(syntheticClasses);

    assertThat(result.hasViolation())
            .as("Uni<Void> where Uni<KeyRotationEntry> expected should be caught")
            .isTrue();
}
```

The synthetic classes are static inner classes of the test file, compiled alongside it
and importable by `ClassFileImporter.importClasses()`. Naming them to satisfy the rule's
`that()` predicate is essential — here the reactive class name starts with `Reactive` and
ends with `Service` to be matched by the parity condition.

## Why This Is Non-Obvious
The natural TDD instinct for "verify this rule catches violations" is to introduce a real
violation in production code and run the test suite — destructive and not repeatable.
`rule.evaluate()` is documented but rarely highlighted; most ArchUnit examples only show
`rule.check()` (the assertion path). The synthetic-inner-class approach for controlled
violations is also undocumented — it relies on the fact that `ClassFileImporter` can import
any `.class` file, including nested classes compiled from test sources.
