### Symptom

A `@BuildStep` that produces a `MultiBuildItem` via `BuildProducer<MyBuildItem>` is silently not executed. No error, no warning. The build item is never produced.

### Context

When a `@BuildStep` method produces a `MultiBuildItem` type that has no consumer (no other `@BuildStep` consumes `List<MyBuildItem>`), Quarkus's build-time dead-code elimination optimizes the step away entirely.

### Fix

Add `@Produce(ServiceStartBuildItem.class)` to the `@BuildStep` method. This forces execution by declaring a side-effect dependency on the service start lifecycle:

```java
@BuildStep
@Produce(ServiceStartBuildItem.class)  // forces execution even without consumers
void extractAgentConfigKeys(
        List<DetectedAiAgentBuildItem> agents,
        BuildProducer<AgentConfigKeyBuildItem> configKeyProducer) {
    // ...
}
```

Remove the `@Produce` annotation once a real consumer is added (another `@BuildStep` that takes `List<AgentConfigKeyBuildItem>`).

### Why this is non-obvious

The dead-code elimination is undocumented in the Quarkus extension guide. The `@BuildStep` is syntactically correct, the `BuildProducer` parameter is valid, and no diagnostic is emitted. The step is simply never called. Discovering this requires build-time debugging or prior knowledge of Quarkus's build graph optimization.
