### Symptom
`@QuarkusTest` integration tests pass with no errors or warnings. Application boots and serves requests correctly. Under production load, throughput degrades and latency spikes — or Quarkus throws `BlockingNotAllowedException` if the reactive thread-check is enabled.

No compiler warning. No test failure. No startup error. The problem is only visible under load or with explicit reactive thread checking enabled.

### Context
Occurs in any Quarkus application using `quarkus-rest` (RESTEasy Reactive) where REST resource methods call JDBC Panache (blocking ORM). `quarkus-rest` runs request handling on the Vert.x I/O thread by default. JDBC operations block the thread. Without `@Blocking`, the I/O thread is blocked for the duration of each database call.

`@QuarkusTest` does not enforce reactive thread discipline — tests pass regardless of whether `@Blocking` is declared. This gives false confidence that the configuration is correct.

### What was tried
Writing `@QuarkusTest` integration tests for REST resources with Panache calls → all tests pass. Deploying to production without `@Blocking` → event loop I/O thread blocked on each JDBC call, degrading throughput.

### Root cause
`quarkus-rest` (RESTEasy Reactive) uses the Vert.x event loop by default. `quarkus-hibernate-orm-panache` uses JDBC, which is inherently blocking. Without `@Blocking` on the resource class or method, Panache calls execute on the Vert.x I/O thread. Vert.x's design principle is that the I/O thread must never block — blocking it prevents it from handling other requests simultaneously.

`@QuarkusTest` does not reproduce the reactive thread model faithfully — it runs tests in a way that tolerates blocking calls on the I/O thread, so the violation is invisible during testing.

### Fix

Add `@Blocking` at the class level to all REST resource classes that call JDBC Panache:

```java
import io.smallrye.common.annotation.Blocking;

@Blocking                    // dispatches all methods to a worker thread
@ApplicationScoped
@Path("/household-tasks")
public class HouseholdTaskResource {

    @Inject HouseholdTaskService service;

    @GET
    public List<HouseholdTask> list() {
        return service.list();  // safe — runs on worker thread
    }
}
```

`@Blocking` at the class level applies to all methods. Individual methods can override with `@NonBlocking` if they don't need I/O.

**Alternative — virtual threads (Java 21+):**

```java
import io.smallrye.common.annotation.RunOnVirtualThread;

@RunOnVirtualThread          // runs each request on a virtual thread (non-blocking carrier)
@ApplicationScoped
@Path("/household-tasks")
public class HouseholdTaskResource { ... }
```

Virtual threads avoid blocking the carrier thread and scale better than traditional worker threads for I/O-bound workloads.

### Why this is non-obvious
`quarkus-resteasy` (classic) and `quarkus-resteasy-reactive` are easy to confuse. Classic runs on worker threads by default — `@Blocking` is not needed. Reactive (`quarkus-rest`) runs on I/O threads — `@Blocking` is required. The artifact name `quarkus-rest` does not signal "reactive", so developers familiar with classic Quarkus miss this.

Tests pass because `@QuarkusTest` does not enforce reactive thread discipline. The failure is only reproducible under concurrent production load or with Vert.x blocking check explicitly enabled. A developer can build and test a full application, ship it to production, and discover the issue only when load increases.
