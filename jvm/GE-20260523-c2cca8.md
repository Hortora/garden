## Symptom

`@QuarkusTest` (or production startup) fails with:

```
java.lang.IllegalStateException: Unable to create Scheduler
Caused by: java.lang.IllegalArgumentException: Cron expression contains 5 parts but we expect one of [6, 7]
  at io.quarkus.quartz.runtime.QuartzSchedulerImpl.<init>(...)
```

No cron expression was recently added or changed. The failing `@Scheduled` bean was already on the classpath.

## Context

A `@Scheduled(cron = "{some.property}")` bean existed in a dependency but was inactive because one of its own dependencies had an unsatisfied CDI injection point. Adding a `@DefaultBean` implementation for that missing type (e.g. a `MockPreferenceProvider`) satisfies the chain — the `@Scheduled` bean becomes active and Quartz tries to parse its cron expression for the first time.

The property resolves to a 5-part unix-style cron (e.g. `"0 2 * * ?"`) instead of the 6-or-7-part Quartz format (e.g. `"0 0 2 * * ?"`).

## Root cause

Two problems compound:

1. **Silent dormancy:** A CDI bean with `@Scheduled` was "protected" from activation by an upstream unsatisfied dependency. Satisfying that upstream dep activates the scheduled bean as a side-effect, with no indication this will happen.

2. **5-part cron default:** The cron property's default value used unix format rather than Quartz format. Quartz requires seconds as the first field; unix cron omits it. `"0 2 * * ?"` is 5 parts (unix); `"0 0 2 * * ?"` is 6 parts (Quartz).

## Fix

**Immediate:** update the cron default from 5-part to 6-part format (add seconds field):

```java
// Before — unix cron, invalid for Quartz
@Scheduled(cron = "${casehub.work.routing.cursor.cleanup-cron:0 2 * * ?}")

// After — Quartz cron with seconds field
@Scheduled(cron = "${casehub.work.routing.cursor.cleanup-cron:0 0 2 * * ?}")
```

Or disable by default:

```java
@Scheduled(cron = "${casehub.work.routing.cursor.cleanup-cron:disabled}")
```

**Diagnostic:** when `Unable to create Scheduler` appears, look for any `@Scheduled` bean that recently became active — especially in dependencies you just added or updated. Search for `@Scheduled(cron` in transitive deps and verify the cron format.

## Why this is non-obvious

The error names the Quartz scheduler, not the `@Scheduled` bean or the recently-added dependency. The failing cron expression was always there but never reached — so there's no recent change in the scheduled job itself to blame. The actual change (satisfying an unrelated injection point via a new `@DefaultBean`) is far removed from the symptom.

### Why this fix

Setting a 6-part default in the property declaration is the permanent fix. The `disabled` sentinel is correct for jobs that should be opt-in. Either prevents the error at startup for new deployments that don't configure the property.
