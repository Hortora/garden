### Symptom
A record gains a new component (e.g. `AgentDescriptor` goes from 18 to 19 fields). A comparator, mapper, or serializer that enumerates the record's fields continues to compile and pass all tests — but silently ignores the new field. The gap is only discovered when the missing field causes a production bug.

### Context
Any utility that must track a Java record's fields — field-by-field comparators, manual mappers (record A → record B), serializers that enumerate fields explicitly. The utility and the record may be in different modules, so adding a field to the record produces no compile error in the utility.

### Fix
Declare a constant in the utility matching the number of fields it handles, and add a reflection-based test that asserts the constant equals the record's actual component count (minus any intentionally skipped fields):

```java
public final class AgentDescriptorComparator {
    static final int COMPARED_FIELD_COUNT = 16;              // 18 components - agentId - tenancyId
    static final int COMPARED_CAPABILITY_FIELD_COUNT = 8;    // 9 components - name (match key)
    static final int COMPARED_DISPOSITION_FIELD_COUNT = 6;   // all components
    // ...
}
```

```java
@Test
void comparatorCoversAllDescriptorComponents() {
    int total = AgentDescriptor.class.getRecordComponents().length;
    int skipped = 2; // agentId, tenancyId — identity keys, not content
    assertEquals(total - skipped, AgentDescriptorComparator.COMPARED_FIELD_COUNT);
}

@Test
void comparatorCoversAllCapabilityComponents() {
    int total = AgentCapability.class.getRecordComponents().length;
    int matchKey = 1; // name — used as match key, not compared
    assertEquals(total - matchKey, AgentDescriptorComparator.COMPARED_CAPABILITY_FIELD_COUNT);
}

@Test
void comparatorCoversAllDispositionComponents() {
    assertEquals(
        AgentDisposition.class.getRecordComponents().length,
        AgentDescriptorComparator.COMPARED_DISPOSITION_FIELD_COUNT);
}
```

When someone adds a 19th component to `AgentDescriptor`, the reflection test fails immediately — the developer must update both the constant and the comparison logic.

### Why this is non-obvious
The natural guard against field drift is the compiler — but Java records don't produce compile errors in code that enumerates their fields by name. A comparator that handles fields `a` through `r` compiles fine when field `s` is added. The gap is invisible to static analysis. Using `RecordComponent` reflection to count fields at test time bridges the gap between "compiles" and "covers all fields."

### Alternatives considered
- Annotation processor that generates comparators from record definitions — heavy infrastructure for a single utility class
- Record `equals()` — covers all fields but includes identity keys (agentId, tenancyId) that must be excluded from content comparison
- Manual code review when adding fields — unreliable; the comparator may be in a different module and the reviewer may not know it exists
