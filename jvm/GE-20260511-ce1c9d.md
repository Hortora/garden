### Symptom
`cannot find symbol` compile errors after moving a class to a different package, even though the class exists. Two separate errors surface from one move operation.

### Context
Moving a Java class from `com.example.domain` to `com.example.plugin.flow` using bash/sed (or manual file move rather than IDE refactor). Two callers break in non-obvious ways.

### What was tried
Moved the file, updated the package declaration, updated the one explicit `import com.example.domain.Foo` that was visible. Build still failed.

### Root cause
Two distinct import mechanisms break independently:

1. **Wildcard imports** — test files using `import com.example.domain.*` no longer resolve `Foo` because it moved out of `domain/`. The error `cannot find symbol` gives no hint that it was previously satisfied by a wildcard.

2. **Same-package implicit imports** — the moved class itself referenced sibling classes (`Bar`, `Baz`) from `domain/` without explicit imports, since they were in the same package. After the move, those implicit imports no longer exist. The error also says `cannot find symbol` with no indication the symbols were ever implicitly available.

Both errors look identical. They are caused by two completely different mechanisms and require two different fixes.

### Fix
After moving the class:
1. Add explicit imports for every wildcard-covered caller: `import com.example.plugin.flow.Foo;`
2. Add explicit imports in the moved class itself for every type it previously accessed implicitly: `import com.example.domain.Bar; import com.example.domain.Baz;`

Use IDE refactor (Move Class) when available — it handles both cases automatically.

### Why this is non-obvious
`cannot find symbol` is the standard "class doesn't exist" error. Both failures produce the same error message despite having different causes. A developer who updates the one visible explicit import reasonably expects the build to pass — the wildcard and same-package failures are invisible until the build runs, and they look like the same problem.
