## The Convention

CDI qualifier annotations (e.g. `@YamlMapper`, `@CrossTenant`, `@EngineSystem`) should live in the module where the **injection point** is declared, not the module where the **producer** is declared.

**Wrong placement:**
```
api/src/main/java/io/casehub/api/engine/YamlCaseHub.java
    @Inject @YamlMapper ObjectMapper objectMapper;  // injection point in api

runtime/src/main/java/io/casehub/engine/internal/marshaller/YamlMapper.java  // ← qualifier buried in runtime
runtime/src/main/java/io/casehub/engine/internal/marshaller/CaseHubObjectMapperProducer.java  // producer
```

`api` does not depend on `runtime`, so `api` cannot see `@YamlMapper` — the injection point and its qualifier are in different modules with no shared dependency.

**Correct placement:**
```
api/src/main/java/io/casehub/api/marshaller/YamlMapper.java  // ← qualifier in api (consumer's module)

runtime/src/main/java/io/casehub/engine/internal/marshaller/CaseHubObjectMapperProducer.java
    import io.casehub.api.marshaller.YamlMapper;  // runtime imports the qualifier from api
```

`runtime` depends on `api` — the producer imports the qualifier from the consumer's module. This is the correct dependency direction.

## Why

A CDI qualifier annotation has zero runtime implementation — it is a pure marker type. Its only dependency is `jakarta.inject.Qualifier`. There is no reason to co-locate it with its producer. The injection point is the stable anchor; the producer implementation may be replaced or in a different module.

## Alternative (less common)

Place the qualifier in a shared `common` module that both `api` and `runtime` depend on. This is correct if the qualifier needs to be visible from multiple peer modules. For qualifiers specific to one interface (e.g. `@YamlMapper` is only for one `ObjectMapper` injection point), the consumer's module is the cleaner home.
