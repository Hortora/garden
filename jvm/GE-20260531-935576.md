### What happened

A Python `re.sub()` script using `[^)]+` to match a single method argument silently mangled Java source files when the argument itself contained a method call.

The pattern `r'(someMethod\()([^,)]+)(\))'` is meant to insert a new parameter after the first argument. But `[^)]+` stops at the **first** `)` it encounters — even if that `)` belongs to a nested method call rather than the outer call's closing paren.

**Before:**
```java
repository.findByUuid(instance.getUuid())
```

**Script intent:** Add `, tenancyId` as a second argument → `findByUuid(instance.getUuid(), tenancyId)`

**Actual result after re.sub:**
```java
repository.findByUuid(instance.getUuid(, tenancyId))
```

The regex matched `instance.getUuid(` as the argument (stopping at the `)` inside `getUuid()`), inserted `, tenancyId` there, and left the outer `)` intact — producing syntactically invalid code.

### Why this is non-obvious

`[^)]+` reads as "match any character except `)` — so this matches the complete argument." The edge case is that the argument *contains* a `)` from a nested call. The regex doesn't understand bracket depth; it treats all `)` characters equally regardless of nesting context.

The fix looks obvious in retrospect, but the original pattern works correctly on simple arguments like `findByUuid(uuid)` or `findByUuid(caseId)`, so it passes superficial testing and only fails on the nested-call variants.

### Fix

Use `str.replace()` with the exact known string rather than a regex pattern when the call site arguments are known ahead of time:

```python
# BAD — breaks on nested calls
content = re.sub(
    r'(repository\.findByUuid\()([^,)]+)(\))',
    lambda m: m.group(1) + m.group(2) + ', tenancyId' + m.group(3),
    content
)

# GOOD — explicit replacement for known patterns
content = content.replace(
    'repository.findByUuid(instance.getUuid())',
    'repository.findByUuid(instance.getUuid(), tenancyId)'
)
content = content.replace(
    'repository.findByUuid(uuid)',
    'repository.findByUuid(uuid, tenancyId)'
)
```

If the call sites genuinely have unknown, variable arguments, use a proper parser (e.g. `javalang` for Java AST) rather than regex.

### Context

Triggered during a large-scale Java SPI migration that added `String tenancyId` to all repository methods. The script iterated over 40+ test files adding the new argument to existing call sites. All single-level arguments migrated correctly; only call sites with `instance.getUuid()`, `UUID.randomUUID()`, etc. as the argument produced mangled output.

The broken files compiled to a parse error (`illegal start of expression`), which at least made the failure visible — no silent data correctness bug.
