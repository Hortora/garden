## CDI @Produces @ApplicationScoped fails for Java records — records are final and cannot be proxied

**Stack:** Quarkus 3.36.x, CDI (ArC)

**Symptom:** `DeploymentException: Producer method for a normal scoped bean must not have a type that is a record, because records are always final`. Application fails to start.

**Context:** Using `@Produces @ApplicationScoped` on a method that returns a Java record type. Common when using records as immutable config objects produced by a parser bean.

### Root cause

CDI normal scopes (`@ApplicationScoped`, `@RequestScoped`, `@SessionScoped`) require the container to create a **client proxy** — a subclass that lazily delegates to the actual instance. Java records are implicitly `final` and cannot be subclassed. ArC detects this at build time and fails deployment.

### Fix

Use `@Singleton` instead of `@ApplicationScoped`:

```java
@Produces
@Singleton  // not @ApplicationScoped — records can't be proxied
FederationConfig produce() {
    return config;
}
```

`@Singleton` is a **pseudo-scope** — no proxy is created. The container injects the actual instance directly. For produced config records that are set once at startup and never change, `@Singleton` is semantically identical to `@ApplicationScoped`.

### Why this is non-obvious

The error message is clear once you see it, but the constraint isn't mentioned in the Quarkus CDI guide or the Java records documentation. A developer writing `@Produces @ApplicationScoped` on a record follows the standard CDI pattern — the mismatch between "records are great for immutable data" and "CDI needs mutable proxies for normal scopes" isn't apparent until deployment fails.
