### Symptom
Code that attempts to build a `Map<String, Object>` response inside a `@Tool` method (then call `map.put(...)` to add fields) fails to compile — or is simply wrong — because the method already returns the domain object directly.

### Context
When adding a new field (`advisories`) to a `DispatchResult` record returned by `QhorusMcpTools.sendMessage()`, the natural impulse is to populate the response map explicitly with `map.put("advisories", result.advisories())`. The Quarkus MCP server serializes `@Tool` return values via Jackson — `@JsonInclude` annotations on the record fields govern what appears in the JSON response.

### Root cause
`@Tool` methods in quarkus-mcp-server return domain objects (records, POJOs) directly. The MCP framework serializes them using Jackson. There is no intermediate Map or response builder — the return type IS the response. `@JsonInclude(NON_EMPTY)` on a field means the field is omitted from the JSON when empty, and present when non-empty. No `map.put()` step exists or is needed.

### Fix
Add `@JsonInclude(JsonInclude.Include.NON_EMPTY) List<String> advisories` to the record. That's all — the field appears automatically in the MCP response when non-empty, absent when empty. No change to the `@Tool` method body beyond returning the record.

```java
// In the record:
@JsonInclude(JsonInclude.Include.NON_EMPTY) List<String> advisories

// @Tool method — no map.put() needed:
public DispatchResult sendMessage(...) {
    // ...
    return messageService.dispatch(dispatch);  // advisories serialized automatically
}
```

### Why this is non-obvious
Many web/REST frameworks separate "return the domain object" from "construct the response DTO/map". Developers familiar with those frameworks assume they need to explicitly add new fields to a response builder or map. Quarkus MCP server elides that step entirely — Jackson owns the serialization of the return value, including field-level `@JsonInclude` annotations.
