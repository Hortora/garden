### Context

Projects running H2 in `MODE=PostgreSQL` share SQL with PostgreSQL to avoid
dialect branching. PostgreSQL's upsert syntax uses `ON CONFLICT (col) DO NOTHING`
with an explicit conflict target (column list or constraint name).

### Symptom

SQL that works on PostgreSQL fails on H2 with a parse error pointing exactly at
the opening `(` after `CONFLICT`:

```
Syntax error in SQL statement
"INSERT INTO t (a, b) VALUES (?, ?) [*]ON CONFLICT (a, b) DO NOTHING"
```

H2 parses successfully up to `ON CONFLICT`, then fails when it sees `(` instead
of `DO`.

### Root Cause

H2 2.2.220 added `INSERT ... ON CONFLICT DO NOTHING` support (gated behind
`MODE=PostgreSQL`). The implementation reads exactly four tokens:

```java
// from org/h2/command/Parser.java
if (readIfCompat(ON, "CONFLICT", "DO", "NOTHING")) {
    command.setIgnore(true);
}
```

There is no code path to parse an explicit conflict target (column list or
constraint name). H2 only supports the no-target form — `ON CONFLICT DO NOTHING` —
which in PostgreSQL means "suppress any conflict regardless of which constraint fired."

This detail is absent from H2's documentation. It can only be confirmed by reading
the H2 parser source.

### Fix

Use `ON CONFLICT DO NOTHING` without specifying the conflict target:

```sql
-- Works on both H2 (MODE=PostgreSQL) and PostgreSQL:
INSERT INTO t (a, b) VALUES (CAST(?1 AS UUID), ?2)
ON CONFLICT DO NOTHING

-- Works on PostgreSQL only:
INSERT INTO t (a, b) VALUES (CAST(?1 AS UUID), ?2)
ON CONFLICT (a, b) DO NOTHING
```

The no-target form is valid PostgreSQL — it suppresses all unique constraint
violations regardless of which constraint triggered. For tables with a single
unique constraint (e.g. a single-column or composite PK), the two forms are
semantically equivalent.

### Why this is non-obvious

PostgreSQL documentation and tutorials always show the explicit column-target form
(`ON CONFLICT (col) DO NOTHING`). `MODE=PostgreSQL` implies PostgreSQL
compatibility. Nothing in H2's documentation mentions that the column-target variant
is unsupported. The only way to discover the limitation is to read H2's `Parser.java`
source code directly from the Maven sources JAR.

### Technique: reading H2 grammar from the Maven sources JAR

```python
import zipfile
sources_jar = "~/.m2/repository/com/h2database/h2/<version>/h2-<version>-sources.jar"
with zipfile.ZipFile(sources_jar) as z:
    src = z.read("org/h2/command/Parser.java").decode("utf-8")
# search for "CONFLICT" to find the grammar
```

### Alternatives considered

- **PostgreSQL column-target form** — fails on H2; cannot be used for shared SQL.
- **Separate SQL per dialect** — reintroduces the branching that `MODE=PostgreSQL`
  was meant to eliminate; rejected.
