## Class shadowing in Quarkus extensions is unreliable — use BytecodeTransformerBuildItem with ASM visitorFunction instead

### Symptom
`NoSuchMethodError` at runtime for a method that exists on your shadowed class but not on the upstream JAR class. The error appears during Quarkus `@RuntimeInit` recorder execution — the JVM loaded the upstream version of the class instead of your shadow.

### Context
A Quarkus extension runtime module creates a class with the same fully-qualified name as one in a dependency JAR (e.g. `dev.langchain4j.internal.DefaultExecutorProvider`) to add a static setter method. The shadow compiles fine — Maven resolves the source before the JAR. At runtime, the Quarkus classloader loads the upstream JAR version instead.

### What was tried (didn't work)
1. **Class shadowing** — creating `dev/langchain4j/internal/DefaultExecutorProvider.java` in the extension runtime module with the added setter method. Compiled successfully, but at runtime the upstream JAR class was loaded.
2. **`BytecodeTransformerBuildItem` with `inputTransformer`** — attempted to read our shadow class bytes via `getResourceAsStream` and swap them in. Failed because `getResourceAsStream` in a split-package scenario returns whichever class the classloader finds first — may be the upstream version.

### Root cause
The Quarkus classloader does not guarantee that extension runtime module classes are loaded before library JAR classes. In a split package (`dev.langchain4j.internal` existing in both the extension JAR and the dependency JAR), classloader ordering is non-deterministic. This is consistent with JPMS split-package prohibitions — Quarkus merely tolerates split packages with a warning, it doesn't resolve them in the extension's favour.

### Fix
Use `BytecodeTransformerBuildItem` with a `visitorFunction` (ASM `ClassVisitor`) to transform the upstream class in-place at build time. The visitor adds fields, methods, or modifies method bodies on whatever class the Quarkus build pipeline resolved — no split-package ambiguity.

```java
@BuildStep
BytecodeTransformerBuildItem addOverrideToUpstreamClass() {
    return new BytecodeTransformerBuildItem(
            "dev.langchain4j.internal.DefaultExecutorProvider",
            (name, visitor) -> new MyTransformerVisitor(visitor));
}
```

The recorder then calls the added method via reflection (since it's added at build time, the compiler doesn't know about it):

```java
Method setter = UpstreamClass.class.getMethod("myAddedMethod", MyType.class);
setter.invoke(null, myValue);
```

### Why this is non-obvious
Class shadowing is the intuitive first approach — it's how developers normally override library behaviour in Java. It compiles, tests sometimes pass (depending on classloader ordering), and the split-package warning in the Quarkus build log is easy to dismiss as cosmetic. The failure is intermittent: it may work in one test class but fail in another depending on augmentation classpath ordering, making it appear to be a test infrastructure issue rather than a fundamental classloading problem.

### Why this fix
`BytecodeTransformerBuildItem` is the Quarkus-standard mechanism for modifying third-party classes. Unlike shadowing or `inputTransformer`, the `visitorFunction` operates on the actual resolved class bytes after Quarkus has decided which version to load — eliminating the ambiguity entirely.

### Alternatives considered
- **`inputTransformer` on `BytecodeTransformerBuildItem`** — reads replacement bytes via `getResourceAsStream`, but in a split-package scenario the resource lookup has the same ambiguity as class loading. Rejected.
- **`sun.misc.Unsafe` to overwrite `static final` fields** — works but is non-portable, fragile, and relies on JVM internals that may be removed in future JDK versions. Rejected.
- **`RemovedResourceBuildItem`** — removes resources from JARs at packaging time but does not affect augmentation-time classloading. Does not solve the problem.
