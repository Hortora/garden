## Symptom

`ExceptionMapper<JsonMappingException>` (or `@ServerExceptionMapper`) registered for a JAX-RS resource never fires when Jackson throws during request body deserialization. Response is 400 with no body and no `Content-Type` header ‚Äî even though the mapper would return a structured JSON error.

## Root Cause

JAX-RS spec ¬ß4.2.4 states: if a message body reader throws `IOException`, the runtime MUST convert it to a 400 response **directly**, without invoking the exception mapper chain. `JsonMappingException` extends `IOException`, so it is caught at the JAX-RS framework layer and never reaches `ExceptionMapper<T>` or `@ServerExceptionMapper`.

Both `@Provider ExceptionMapper<JsonMappingException>` and Quarkus's `@ServerExceptionMapper` are bypassed. The fix must happen **before** the message body reader throws ‚Äî by ensuring the exception is only thrown from within the resource method body, where the exception mapper chain is active.

## Fix

Change the resource method parameter from the validated domain type to a plain DTO that does not validate in its constructor. Construct the domain object inside the method body:

```java
// BEFORE ‚Äî validation in compact constructor fires during deserialization ‚Üí IAE wrapped as JsonMappingException ‚Üí 400 no body
@POST
public Response recordOutcome(@PathParam("id") UUID id, final SarOutcome outcome) {
    feedbackService.recordOutcome(id, outcome);
    return Response.noContent().build();
}

// AFTER ‚Äî construct inside the method so IllegalArgumentException fires after deserialization
record SarOutcomeRequest(SarVerdict verdict, String reason, double score) {}

@POST
public Response recordOutcome(@PathParam("id") UUID id, final SarOutcomeRequest request) {
    final SarOutcome outcome = new SarOutcome(request.verdict(), request.reason(), request.score());
    feedbackService.recordOutcome(id, outcome);
    return Response.noContent().build();
}
```

`IllegalArgumentExceptionMapper` (which maps service-layer exceptions) then fires correctly and produces the structured 400 response.

## Why this is non-obvious

The `ExceptionMapper` is registered, the mapper logic works (unit-testable directly), and Jackson does throw the exception. There is no framework error. The only signal is a 400 with empty body. Nothing in the Quarkus docs explicitly warns that deserialization errors skip exception mappers ‚Äî the spec reference (¬ß4.2.4) is in the JAX-RS spec itself, not in Quarkus documentation.

`@ServerExceptionMapper` (RESTEasy Reactive's native handler) also does not fire ‚Äî a natural follow-up attempt that still fails for the same reason.

### Why this fix

Moving validation into the resource method body is the only approach that reliably intercepts the error. Alternatives fail: `@ServerExceptionMapper` does not fire for MBR IOExceptions; `ReaderInterceptor.aroundReadFrom()` can rethrow as `WebApplicationException` but the plumbing is fragile and couples the interceptor to Jackson internals.

### Alternatives considered

- **`@Provider ExceptionMapper<JsonMappingException>`** ‚Äî does not fire for MBR exceptions per spec ¬ß4.2.4
- **`@ServerExceptionMapper`** ‚Äî RESTEasy Reactive native, but also does not fire for MBR exceptions
- **`ReaderInterceptor`** ‚Äî can catch and rethrow as `WebApplicationException`, but JAX-RS's handling of `WebApplicationException` from interceptors is implementation-specific and harder to test
- **Bean Validation (`@Valid` + `@DecimalMin/@DecimalMax`)** ‚Äî works but requires `quarkus-hibernate-validator` dependency for what is a simple range check
