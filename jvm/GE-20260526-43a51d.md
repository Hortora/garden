### What happened

Added a new component (`traceId`) to a Java `record` in a shared library module (`casehub-engine-common`). Ran `mvn install -pl common -DskipTests` to publish the new jar to the local `.m2` repository. Then ran `mvn test-compile -pl runtime` on a dependent module — Maven reported **BUILD SUCCESS** with no errors, even though all the existing `new CaseLifecycleEvent(...)` call sites were now passing the wrong number of arguments (6 instead of 7).

The arity mismatch was only caught when running `test-compile` after modifying one of the test files in the module — which forced recompilation of the changed test file, which then failed with the 7-arg constructor error.

### Root cause

Maven incremental compilation uses timestamps and `.class` file checksums to decide which source files need recompiling. When a dependency jar changes but the **source files in the dependent module have not changed**, Maven's incremental compiler skips them — it treats the existing `.class` files as still valid. The compiler never re-checks whether the API used in those source files is still compatible with the updated dependency.

For Java records specifically, the compiler-generated canonical constructor has an exact arity — adding one component changes the constructor signature. But incremental compile doesn't catch this because it only recompiles files it considers dirty.

### Fix

Run a clean compile when changing a library API that affects call sites:

```bash
mvn clean test-compile -pl <module>
# or install fresh and force recompile with -C (no-snapshot update):
mvn install -pl <lib> -DskipTests && mvn clean compile -pl <dependent>
```

Alternatively, run the full module chain with `-am` to force dependency-aware compilation:

```bash
mvn test-compile -pl <dependent> -am
```

`-am` (also-make) rebuilds all transitive dependencies from source, so the updated library is recompiled and the dependent module's stale `.class` files are invalidated.

### Why this is non-obvious

Maven incremental compile is supposed to detect changes. The expected behavior is: changed jar → dependent source files recompile → arity errors surface. The actual behavior skips recompilation of unchanged source files, so the arity error is silently hidden. **BUILD SUCCESS** is a false green — the existing compiled artifacts are wrong but still present.

The gotcha is especially tricky with Java `record` components because adding a component looks like an internal detail but changes the canonical constructor's parameter count, which is a source-incompatible API break that compile-time should catch.
