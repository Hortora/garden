## What it is

`WorkflowApplication`'s `DefaultExecutorServiceFactory` (the default executor used to run workflow tasks and task executor `apply()` calls) uses `Executors.newCachedThreadPool()`, not the Vert.x event loop. Blocking operations — including `await().indefinitely()`, `.join()`, and synchronous JDBC calls — are safe inside workflow task executors.

## Where to find it

`serverlessworkflow-impl-core` sources, `DefaultExecutorServiceFactory.java`:
```java
public ExecutorService get() {
    if (service == null) {
        service = Executors.newCachedThreadPool();
    }
    return service;
}
```

## Why it matters

Quarkus runs most reactive code on Vert.x IO threads where blocking causes `MutinyException: You called await() from a Vert.x I/O thread`. The quarkus-flow SDK uses its own independent cached thread pool, so workflow task executors (including `CallableTask.apply()` implementations and FuncDSL step lambdas) are NOT subject to this constraint. You can safely call `.join()` on a `CompletableFuture` or use blocking reactive subscriptions inside a workflow step.

## Why this is non-obvious

The quarkus-flow documentation does not mention the thread model for task executors. The conventional Quarkus assumption is that everything reactive runs on the Vert.x event loop. Without reading the SDK source, it's easy to add an unnecessary `@Blocking` annotation or avoid blocking calls that are actually safe.
