### The behaviour

When `DefaultClaudeAsyncClient.close()` is called (e.g. by a wall-clock timeout), `cleanup()` runs:

```java
// DefaultClaudeAsyncClient.cleanup() — source-verified, not documented
Sinks.Many<Message> turnSink = currentTurnSink.getAndSet(null);
if (turnSink != null) {
    turnSink.tryEmitComplete();  // ← COMPLETE, not tryEmitError()
}
```

The active turn's `Multi<AgentEvent>` **completes** (subscriber sees `onCompletion`) rather than failing with an exception. An `onFailure` handler for `AgentTimeoutException` does NOT fire in this path.

### When it surfaces

Any code that:
1. Schedules a wall-clock timeout that calls `sdkClient.close().subscribe()`
2. Tries to detect the timeout in `onFailure().transform(e -> new AgentTimeoutException(...))`

The transform fires only if a transport-level `IOException` races ahead of `cleanup()`'s `tryEmitComplete()`. This is a non-deterministic race — the `onFailure` path may or may not fire depending on thread scheduling.

### Discovery

Found by reading `DefaultClaudeAsyncClient` source in the SDK JAR (not in any README, Javadoc, or changelog). `StreamingTransport.close()` also calls `inboundSink.tryEmitComplete()` before destroying the subprocess, confirming the completion-not-failure design.

### Fix

To guarantee `AgentTimeoutException` reaches the subscriber regardless of the race, wrap the stream in a completion-to-failure converter that checks a per-turn `timedOut` flag:

```java
AtomicBoolean timedOut = new AtomicBoolean(false);

// Timeout task:
timeoutScheduler.schedule(() -> {
    if (timedOut.compareAndSet(false, true)) sdkClient.close().subscribe();
}, timeout.toMillis(), TimeUnit.MILLISECONDS);

// In the Multi pipeline, BEFORE termination handlers:
.onCompletion().call(() -> {
    if (timedOut.get()) {
        return Uni.createFrom().failure(new AgentTimeoutException(timeout));
    }
    return Uni.createFrom().voidItem();
})
// Now onFailure().transform() reliably receives AgentTimeoutException:
.onFailure().transform(e -> {
    if (e instanceof AgentTimeoutException) return e;
    if (timedOut.get()) return new AgentTimeoutException(timeout);
    return new AgentProcessException(Objects.toString(e.getMessage(), e.getClass().getSimpleName()), e);
})
```

### Why non-obvious

The SDK's cleanup design completes sinks rather than erroring them — a deliberate choice to avoid propagating teardown errors as application-level failures. This is reasonable design for the SDK, but invisible to callers who assume timeout → exception. The only way to discover this is to read the SDK source code.
