### Symptom
CDI deployment failure at `@QuarkusTest` augmentation time ‚Äî multiple `Unsatisfied dependency for type EntityManager and qualifiers [@PersistenceUnit("qhorus")]` errors (or similar bean errors), with no indication of a JAR version problem. Configuring datasources and excluding bean types doesn't help.

### Context
A SNAPSHOT sibling project (`quarkus-ledger`) changed its Maven version from `1.0.0-SNAPSHOT` to `0.2-SNAPSHOT` in source. The dependent project's `pom.xml` still declared `1.0.0-SNAPSHOT`. Maven found the old `1.0.0-SNAPSHOT` jar in `~/.m2` and used it silently ‚Äî that jar had been compiled from an older source that contained `@PersistenceUnit("qhorus")` annotations on several CDI beans. The current source had already removed those annotations.

`mvn install` on the sibling project installed to `0.2-SNAPSHOT/` but left `1.0.0-SNAPSHOT/` untouched in `~/.m2`. The dependent project kept resolving `1.0.0-SNAPSHOT` from cache.

### What was tried (didn't work)
- Configuring a second named datasource (`quarkus.datasource."qhorus".*`) ‚Äî fixed the datasource problem, created a new Hibernate ORM entity-mapping problem
- Adding `quarkus.hibernate-orm."qhorus".packages=...` ‚Äî entity ownership conflict between two PUs
- `quarkus.arc.exclude-types=...` to exclude the offending beans ‚Äî created cascading unsatisfied dependency errors from their consumers
- `mvn install -DskipTests` on the sibling ‚Äî reinstalled, but to the new version path; the old `1.0.0-SNAPSHOT` jar remained in `~/.m2`

### Root cause
`mvn install` on a SNAPSHOT project installs to `~/.m2/repository/<group>/<artifact>/<version>/`. Changing the version creates a **new** directory ‚Äî the old version directory is never cleaned up. Dependents that still reference the old version number find the stale jar with its outdated bytecode, including any annotations that have since been removed.

### Fix
Two steps, both required:

1. **Delete the stale jar** from `~/.m2`:
```bash
rm -rf ~/.m2/repository/io/quarkiverse/ledger/quarkus-ledger/1.0.0-SNAPSHOT/
```

2. **Update the dependent pom** to the new version:
```xml
<!-- Before -->
<version>1.0.0-SNAPSHOT</version>

<!-- After ‚Äî matches what the sibling now actually builds -->
<version>0.2-SNAPSHOT</version>
```

Then reinstall the sibling and rebuild the dependent.

### Why this is non-obvious
The symptom (CDI bean injection failures) points to a configuration problem, not a stale dependency. Nothing in the Maven output indicates that a different version of the jar is being used than what the sibling currently builds. `mvn dependency:tree` shows the old version without flagging it as stale. The natural instinct is to fix CDI configuration, not to audit the `~/.m2` cache.
