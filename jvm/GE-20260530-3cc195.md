## Symptom

`String.format("%.4f", 0.87)` produces `"0,8700"` (comma decimal separator) on JVMs running with a German, French, or other decimal-comma locale. `Double.parseDouble("0,8700")` then throws `NumberFormatException`. A format/parse round-trip that passes on English-locale CI silently breaks on non-English JVMs.

## Context

Any time a formatted decimal string is stored (in a database, attribute map, file, or API response) and later parsed back. The failure is invisible in English-locale CI environments but surfaces in production on non-English JVMs or when running tests with `-Duser.language=de`.

## Root cause

`String.format(String, Object...)` uses `Locale.getDefault()`. The decimal separator character is locale-controlled. English locales use `.`; German, French, and many others use `,`. `Double.parseDouble` always expects `.` — it does not use the JVM locale.

## Fix

Always pass `Locale.ROOT` as the first argument to `String.format` when producing canonical decimal strings:

```java
// Wrong — locale-sensitive
return String.format("%.4f", v);

// Correct — always produces "0.8700"
return String.format(Locale.ROOT, "%.4f", v);
```

Add `import java.util.Locale;`.

## Why this is non-obvious

`String.format` is used so frequently that developers rarely think about the locale parameter. The failure is silent on English-locale CI — all tests pass — and only surfaces in production on non-English systems or when explicitly testing with a non-default locale. The symptom (`NumberFormatException` on `parseDouble`) does not obviously point to `String.format` as the cause.

### Why this fix

`Locale.ROOT` is the "neutral" locale — it produces the invariant canonical form independent of JVM locale settings. `Locale.ENGLISH` would also work but is slightly misleading (implies English-language content rather than locale-neutral formatting).

## Constraints

Applies wherever `String.format` is used to produce decimal strings that will be stored and parsed back. Does not apply when the formatted output is intended for human display (where locale-appropriate formatting is correct).
