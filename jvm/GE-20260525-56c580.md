### Symptom

In a target-following game loop, setting a unit's movement target to its current
position produces inconsistent results across entity types: some entities stay frozen
while others stop being frozen after one tick.

### Context

Writing geometry-sensitive combat tests in `EmulatedGame` (QuarkMind) that require
stationary enemies. Default `spawnEnemyForTesting` sets the unit target to
`EnemyBehavior.NEXUS_POS`, causing enemies to drift to (8,8) each tick and
invalidating distance-based assertions.

### Root Cause

Two movement methods can co-exist in a simulation:

```java
// moveFriendlyUnits — removes the target when the unit arrives
Point2d newPos = strategy.advance(u.tag(), u.position(), target, speed);
if (distance(newPos, target) < 0.2) friendly.unitTargets.remove(u.tag()); // ← removes

// moveEnemyUnits — no removal; target persists indefinitely
Point2d newPos = strategy.advance(u.tag(), u.position(), target, speed);
// no removal — target stays in map
```

`stepToward(from=pos, to=pos, speed)` returns `pos` immediately (`distance=0 ≤ speed`).
For enemies: the target entry persists in `unitTargets` → freeze holds every tick.
For friendlies: the target entry is removed after the first `advance` call → the unit
falls back to a default target on the next tick, breaking the freeze.

### Fix

```java
// Freeze enemy units by setting their target to their current position.
// Works because moveEnemyUnits does NOT remove targets on arrival.
game.enemy.unitTargets.put(zealotATag, new Point2d(7, 10)); // unit is at (7,10)
game.enemy.unitTargets.put(zealotBTag, new Point2d(20, 10));
```

For any target-following system: before relying on the own-position freeze, verify
whether your movement method removes the target map entry on arrival. If it does,
the freeze breaks after one tick.

### Why This Is Non-Obvious

The asymmetry between entity-type movement methods is not documented. A developer
encountering two movement handlers would reasonably assume symmetric behavior (both
remove on arrival, or neither does). The removal logic is an implementation detail
visible only from source — it exists to prevent "arrived" units from re-processing
stale targets on subsequent ticks, which is a correctness concern for some entity
types but not others (e.g. enemy targets are overwritten by AI behavior each tick
anyway, so removal is redundant and was omitted).
