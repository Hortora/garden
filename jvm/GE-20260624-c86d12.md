### Symptom
Quarkus application fails to start with:
```
Caused by: java.lang.IllegalArgumentException: maxAttempts must be >= 1, got 0
  at io.casehub.platform.api.governance.RetryPolicy.<init>(RetryPolicy.java:7)
```
All `@QuarkusTest` classes in the module fail with this error, making it appear as a massive regression.

### Context
The `casehub-platform-api` `RetryPolicy` record validates `maxAttempts >= 1` in its compact constructor. The old engine `RetryPolicy` had no validation — `new RetryPolicy(0, null, null)` was the idiom for "no retry".

Any `CaseHub` bean registered via CDI that constructs a Worker with `RetryPolicy(0, ...)` crashes the Quarkus container at startup — cascading to ALL tests in the module, not just the tests for that bean.

### What was tried (didn't work)
`new RetryPolicy(0, null, null)` — the platform-api version rejects this at construction time.

### Root cause
Platform-api governance types enforce domain invariants that the old engine types did not. `maxAttempts=0` is semantically invalid — if you don't want retries, use the dedicated `noRetry()` factory.

### Fix
```java
// Old — crashes
RetryPolicy noRetry = new RetryPolicy(0, null, null);
worker.setExecutionPolicy(new ExecutionPolicy(timeout, noRetry));

// New — correct
ExecutionPolicy noRetryPolicy = ExecutionPolicy.noRetry();
// or with custom timeout:
new ExecutionPolicy(100, new RetryPolicy(1, 0, null))
```

### Why this is non-obvious
The error manifests as a Quarkus startup failure that cascades to every `@QuarkusTest` in the module. The stack trace points to a CDI bean's `getDefinition()` method — not to the specific `RetryPolicy(0, ...)` call. A developer seeing 30+ test failures would investigate CDI wiring, not a constructor validation change in a dependency.
