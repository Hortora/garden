# @Startup @ApplicationScoped prevents blocking @PostConstruct from running on Vert.x IO thread

## The technique

Add `@Startup` to any `@ApplicationScoped` bean whose `@PostConstruct` makes a blocking call:

```java
@Startup           // forces eager init on main thread during boot
@ApplicationScoped
public class MyClient {

    @PostConstruct
    void validate() {
        // blocking: process probe, network check, file I/O, etc.
        new ProcessBuilder("claude", "--version").start().waitFor(10, TimeUnit.SECONDS);
    }
}
```

Without `@Startup`, Quarkus ARC initializes `@ApplicationScoped` beans **lazily** — on first injection. If the first injection happens from a reactive handler (inside a Vert.x event-loop callback, a REST endpoint, a CDI observer), the `@PostConstruct` runs on the Vert.x IO thread. Any blocking call there violates the non-blocking contract and either deadlocks under load or triggers a `BlockingOperationNotAllowedException`.

`@Startup` forces initialization during application boot on the main thread, before any reactive handlers are active.

## Why this is non-obvious

`@Startup` is documented as "run initialization at startup" but the docs frame it in terms of *when* (boot vs. first use), not *which thread*. The connection to IO-thread safety is not mentioned. Developers who know about `@Startup` typically use it for "I want this to run at boot time" — not as a reactive-safety mechanism.

The failure mode is intermittent: `@PostConstruct` is safe when the first injection happens from a non-reactive context (tests, CLI, HTTP request on a worker thread). It only fails when the first injection happens under reactive load — making it a production-only bug in many setups.

## Codebase pattern

Same pattern used by `PathParserConfigurator` and `MongoPreferenceIndexes` in casehub-platform. Canonical for any bean that initialises eagerly and blocks during `@PostConstruct`.

## Why non-obvious

The standard mental model for `@ApplicationScoped` is "singleton, available everywhere, initialized once." The lazy-init detail — and its thread-safety implications in a reactive runtime — is not in the CDI spec, only in Quarkus ARC internals. A developer migrating from Spring (`@Bean` methods always run on the startup thread) would have no reason to anticipate the problem.
