## Symptom

Adding `quarkus.arc.exclude-types=com.example.SomeBean` causes *new* CDI failures to appear — unsatisfied dependencies, scheduler errors, or ambiguity errors for beans that were never a problem before.

## Context

You have a production dependency (e.g. `casehub-engine`) whose JAR has no Jandex index and no `beans.xml`. Adding an entry to `quarkus.arc.exclude-types` that names a class from that JAR causes Quarkus to scan the entire JAR to locate the target class.

## Root cause

Quarkus ARC must physically find the class to exclude it. For JARs without a Jandex index, ARC falls back to scanning the whole JAR. During that scan it discovers every `@ApplicationScoped`, `@Singleton`, etc. class — registering all of them as CDI beans. The net effect is the opposite of what the exclusion intended: the JAR gains full CDI discovery while one bean is removed.

## Fix

**Option 1 — Use a production-profile prefix** so the exclusion only applies during `quarkus:build`, not during `@QuarkusTest`:

```properties
%prod.quarkus.arc.exclude-types=com.example.SomeBean
```

**Option 2 — Add `quarkus.index-dependency`** explicitly for the JAR so ARC uses the Jandex index instead of scanning:

```properties
quarkus.index-dependency.mylib.group-id=com.example
quarkus.index-dependency.mylib.artifact-id=mylib
```

Then the exclude-types works as documented.

**Option 3 — Fix the upstream library** to ship a Jandex index (`io.smallrye:jandex-maven-plugin`) so ARC never needs to scan.

## Why this is non-obvious

Exclusion is supposed to reduce beans — not add them. The documentation for `quarkus.arc.exclude-types` says nothing about triggering a full JAR scan. The symptom (new unsatisfied dependencies or scheduler failures) points away from the exclusion entry as the cause.

### Why this fix

Using `%prod.` is the lightest fix — it scopes the exclusion to the context where the ambiguity actually exists, and keeps the test ARC context clean.
