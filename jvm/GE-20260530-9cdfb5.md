### What it is

`MapPreferences.get(PreferenceKey<T> key)` returns `null` when the key is absent from the underlying map. It does **not** fall back to `PreferenceKey.defaultValue`. The `defaultValue` field on `PreferenceKey` is a non-null constructor sentinel only — its functional purpose is satisfying `Objects.requireNonNull` in the `PreferenceKey` constructor.

### How it works

```java
// MapPreferences.get() implementation:
@Override
public <T extends SingleValuePreference> T get(PreferenceKey<T> key) {
    Object value = values.get(key.qualifiedName());
    if (value instanceof SingleValuePreference) return (T) value;
    if (value instanceof String s) return key.parse(s);
    return null;  // ← key absent: returns null, does NOT use key.defaultValue
}
```

A caller expecting `key.defaultValue()` to serve as a safe default gets `null`:

```java
// WRONG — appears safe but silently gets null when key absent from YAML
DoublePreference blendFactor = prefs.get(TrustRoutingPolicyKeys.BLEND_FACTOR);
double value = blendFactor.value(); // NullPointerException

// CORRECT — null-check and provide the fallback explicitly
DoublePreference blendFactor = prefs.get(TrustRoutingPolicyKeys.BLEND_FACTOR);
double value = blendFactor != null ? blendFactor.value() : TrustRoutingPolicy.DEFAULT.blendFactor();
```

### Why this is non-obvious

`PreferenceKey` has a `defaultValue` field with a non-null constraint, which strongly implies it acts as a fallback — especially since `@ConfigProperty(defaultValue = "...")` in MicroProfile Config does exactly that. The `PreferenceKey` API looks like it should behave the same way. It does not. The `defaultValue` is used as a sentinel to satisfy the constructor but is never consulted by `MapPreferences.get()`. The null is completely silent — no log, no exception, no warning.
