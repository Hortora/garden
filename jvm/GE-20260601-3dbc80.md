## Context

Two Jandex-indexed framework JARs both provide `@DefaultBean` implementations for the same SPI type. Both are on the test classpath. Quarkus CDI augmentation reports ambiguous dependencies:

```
Ambiguous dependencies for type io.example.SomeProvider and qualifiers [@Default]
  - CLASS bean [...DefaultImpl...] (jar-a)
  - CLASS bean [...MockImpl...] (jar-b)
```

Changing scope or modifying either JAR is not an option. The less-capable no-op implementation should lose.

## Technique

Add the less-capable bean to `quarkus.arc.exclude-types` in the test `application.properties`. Quarkus removes it from CDI scanning entirely, leaving the more-capable implementation as the sole `@DefaultBean`:

```properties
# src/test/resources/application.properties
quarkus.arc.exclude-types=\
  io.example.NoOpSomeProvider,\
  io.example.AnotherNoOpBean
```

Multiple types are comma-separated with backslash continuation. Each entry is a fully-qualified class name. Excluded types are invisible to CDI — the remaining `@DefaultBean` wins without ambiguity.

## Why this is non-obvious

The natural responses to a CDI ambiguity are: change scope, add `@Priority`, or use `beans.xml` alternatives. `quarkus.arc.exclude-types` is a Quarkus-specific property that completely removes a class from CDI scanning — it doesn't just lower priority, it makes the class invisible. Most developers don't reach for it because the Quarkus docs present it as a workaround for scheduler-incompatible beans (a different use case), not as a general CDI conflict resolver.
