## ConcurrentHashMap.computeIfAbsent does not cache null â€” mapping function re-invoked on every call when absent

**ID:** GE-20260612-f6362e
**Stack:** Java (all versions)
**Symptom:** Code using `computeIfAbsent` to cache an expensive lookup result keeps calling the mapping function on every get, even after the first call. No exception, no warning â€” the function just runs repeatedly.
**Context:** The mapping function returns `null` when the target resource is not found (e.g. a channel UUID that doesn't exist yet). The developer expects null to be cached so subsequent calls skip the lookup. It isn't.

### What was tried (didn't work)
- Trusting that `computeIfAbsent` would cache the null return value the same way it caches a non-null value

### Root cause
`ConcurrentHashMap.computeIfAbsent` is specified by Java to NOT store null mappings. From the Javadoc:

> "If the specified key is not already associated with a value, attempts to compute its value using the given mapping function and enters it into this map unless null."

The phrase "unless null" is easy to miss. If the function returns null, no entry is created and the map remains unchanged. The next call to `computeIfAbsent` for the same key invokes the mapping function again.

This is by design â€” `ConcurrentHashMap` does not permit null values at all (unlike `HashMap`).

### Fix
Two options depending on the desired semantics:

**Option 1 â€” Sentinel value (recommended when null means "not found"):**
```java
private static final UUID ABSENT = new UUID(0, 0); // sentinel

UUID channelId = channelIdCache.computeIfAbsent(tenancyId, tid ->
    channelService.findByName(name).map(ch -> ch.id).orElse(ABSENT));

if (ABSENT.equals(channelId)) {
    // channel not found â€” treat as absent
    channelIdCache.remove(tenancyId); // optionally invalidate to allow retry
    return;
}
```

**Option 2 â€” Re-invoke on miss (when re-querying until available is correct):**
```java
// computeIfAbsent with null return = always re-query on miss
// Use this when "not found now" is transient and the next call should retry
UUID channelId = channelIdCache.computeIfAbsent(tenancyId, tid ->
    channelService.findByName(name).map(ch -> ch.id).orElse(null));
// null â†’ not cached â†’ next call retries the lookup automatically
```

### Why this is non-obvious
`HashMap.computeIfAbsent` has the same behaviour, but `HashMap` allows null values elsewhere, so the distinction is less visible. Developers coming from `HashMap` usage patterns assume null is a valid cached value. The Javadoc mentions it but the "unless null" clause is easy to overlook when reading quickly.
