# langchain4j-document-parser-apache-tika artifact doesn't exist for LangChain4j 1.14.1

## Symptom

Maven dependency resolution fails for `dev.langchain4j:langchain4j-document-parser-apache-tika:1.14.1` with "Could not find artifact" error.

## Context

Adding Tika document parsing to a LangChain4j RAG pipeline. Examples and tutorials reference the `langchain4j-document-parser-apache-tika` module as the standard way to parse PDFs, Word docs, etc. into LangChain4j `Document` objects.

## What was tried

- Verified LangChain4j core version (1.14.1) is available in Maven Central — it is
- Searched Maven Central for the artifact — `langchain4j-document-parser-apache-tika` exists for some versions but not 1.14.1
- Checked LangChain4j release notes — no mention of the document parser modules being removed

## Root cause

The document parser modules were restructured in LangChain4j — they use a different versioning scheme and may not be published for all LangChain4j core versions. The artifact name in older examples may refer to a module that existed in earlier releases but was deprecated, renamed, or not built for newer versions.

As of LangChain4j 1.14.1, the Tika parser module either:
- Was not published to Maven Central for this version
- Uses a different artifact ID
- Has been replaced by direct Tika integration

## Fix

Use Apache Tika directly and integrate manually with LangChain4j's `Document` and `DocumentSplitter` APIs:

```xml
<dependency>
    <groupId>org.apache.tika</groupId>
    <artifactId>tika-core</artifactId>
    <version>2.9.2</version>
</dependency>
<dependency>
    <groupId>org.apache.tika</groupId>
    <artifactId>tika-parsers-standard-package</artifactId>
    <version>2.9.2</version>
</dependency>
```

Parse documents and wrap results:

```java
Tika tika = new Tika();
String content = tika.parseToString(inputStream);
Document doc = Document.from(content, metadata);
```

## Why non-obvious

LangChain4j documentation and tutorials reference `langchain4j-document-parser-apache-tika` as if it's always available alongside the core library. The version mismatch is only discovered at Maven resolution time — there's no compile-time or runtime indication that the module doesn't exist until the build fails.

The LangChain4j GitHub repo may have the source for these modules, but they're not being published to Maven Central for all core releases.
