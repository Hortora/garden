### Symptom

`JdbcSQLSyntaxErrorException: Syntax error in SQL statement "INSERT INTO ... [*]ON CONFLICT ..."` when executing a native SQL upsert query in `@QuarkusTest`. The same statement succeeds on production PostgreSQL.

```
Caused by: org.h2.jdbc.JdbcSQLSyntaxErrorException: Syntax error in SQL statement
"INSERT INTO attestation_ref(...) VALUES(?, ...) [*]ON CONFLICT (hash, tenancy_id) DO NOTHING"
SQL statement: ... [42000-240]
```

### Context

Tests use H2 in-memory in `MODE=PostgreSQL`:
```properties
quarkus.datasource.jdbc.url=jdbc:h2:mem:testdb;MODE=PostgreSQL
```

A `@Transactional` method in a JPA `@ApplicationScoped` bean calls `em.createNativeQuery(...)` with PostgreSQL's upsert syntax to achieve idempotency on a table with a UNIQUE constraint:

```java
em.createNativeQuery(
    "INSERT INTO my_table(id, hash, tenancy_id, ...) VALUES(?, ?, ?, ...) " +
    "ON CONFLICT (hash, tenancy_id) DO NOTHING")
  .setParameter(1, uuid).setParameter(2, hash)...
  .executeUpdate();
```

### Root cause

H2's `MODE=PostgreSQL` provides partial PostgreSQL syntax compatibility — enough for DDL types (`TIMESTAMP WITH TIME ZONE`, `BIGINT GENERATED ALWAYS AS IDENTITY`, etc.) and basic DML, but not for PostgreSQL-specific upsert extensions. `INSERT ... ON CONFLICT ... DO NOTHING` is PostgreSQL-proprietary syntax that H2 2.x does not implement even in compatibility mode.

Code review and documentation suggest H2 MODE=PostgreSQL implies broad syntax compatibility; in practice, PostgreSQL DML extensions are not covered.

### Fix

Replace native SQL upsert with a JPQL existence check + conditional `em.persist()` within the same `@Transactional` boundary:

```java
@Transactional
public void insertIfAbsent(final String hash, final String tenancyId, ...) {
    // Idempotent: skip if a row with this (hash, tenancyId) already exists.
    // JPQL check-then-insert is safe within @Transactional; avoids ON CONFLICT
    // syntax that H2 in MODE=PostgreSQL does not support.
    boolean exists = !em.createQuery(
            "SELECT e FROM MyEntity e WHERE e.hash = :h AND e.tenancyId = :t")
        .setParameter("h", hash).setParameter("t", tenancyId)
        .setMaxResults(1).getResultList().isEmpty();
    if (exists) return;
    em.persist(buildEntity(hash, tenancyId, ...));
}
```

For high-concurrency scenarios where two threads may race the check:

```java
try {
    em.persist(buildEntity(hash, tenancyId, ...));
} catch (final PersistenceException e) {
    // Unique constraint violation from concurrent insert — idempotent, ignore
    if (!isCausedByConstraintViolation(e)) throw e;
}
```

### Why this is non-obvious

`MODE=PostgreSQL` implies broad compatibility. Developers writing `ON CONFLICT DO NOTHING` for idempotency — a standard PostgreSQL pattern documented in every upsert guide — would have no reason to expect H2 to reject it. The error is clear, but the correct portable alternative (JPQL check-then-insert) is not obvious and is not mentioned in H2 or Quarkus documentation.

H2's own changelog does not list `ON CONFLICT` as a missing feature; it is simply absent.

### Alternatives considered

- **H2's `MERGE INTO ... KEY(...)`** — H2's native upsert syntax. Works for H2 but fails on production PostgreSQL. Swaps one portability problem for another.
- **Testcontainers PostgreSQL** — would allow native `ON CONFLICT` in tests. Adds container startup overhead and external dependency to `@QuarkusTest`. Correct approach for dialect validation; JPQL is correct for unit-speed test isolation.
