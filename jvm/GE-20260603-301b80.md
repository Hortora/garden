### What it is
`DeclarativeUtil` in `langchain4j-agentic` exposes a single static list `chatSupplierParameterResolvers` with one registration method `addChatSupplierParameterResolver(ChatSupplierParameterResolver)`. This resolver hook fires only during `@ChatModelSupplier` and `@StreamingChatModelSupplier` processing. All other supplier annotations (`@ToolsSupplier`, `@ChatMemorySupplier`, `@ChatMemoryProviderSupplier`, `@ContentRetrieverSupplier`, `@RetrievalAugmentorSupplier`, `@AgentListenerSupplier`, `@ToolProviderSupplier`) call `checkArguments(method)` which enforces **zero parameters** before invoking the static method directly — no resolver hook exists for these paths.

### Why this matters for framework integrators
Any DI framework (Quarkus, Spring, Guice) wanting to inject CDI/Spring beans into supplier methods is blocked for all supplier types except chat model suppliers. The `@CdiBean` workaround in `quarkus-langchain4j` only works for `@ChatModelSupplier` as a result. Users who need a CDI-managed `ContentRetriever` or `ChatMemory` must use `Arc.container()` or `ApplicationContext.getBean()` inside the static method body — bypassing injection, scope management, and testability.

### How to discover this
Only by decompiling `langchain4j-agentic-1.15.1-beta25.jar` and reading `DeclarativeUtil.class`. The library has no published Javadoc for internal classes and no documentation of the SPI surface area.

```bash
# Locate the JAR
find ~/.m2/repository/dev/langchain4j/langchain4j-agentic -name "*.jar" | grep -v sources | head -1

# The class structure reveals: addChatSupplierParameterResolver() is the only hook
# All other configureAgent() branches call checkArguments() then invokeStatic()
```

### Workaround
For Quarkus: use `SyntheticCreationalContext.getInjectedReference()` in the synthetic bean creator (`AgenticRecorder`) to inject CDI beans at agent construction time, bypassing the static supplier method entirely. This requires detecting at build time that no static supplier method exists for a given type, then adding a CDI injection point for that type.

### Upstream path
An upstream PR to `langchain4j-agentic` should propose a general `SupplierParameterResolver` SPI (not chat-model-specific) and add resolver hooks to all supplier type branches in `configureAgent()`. Frame as a DI-framework-agnostic improvement — Spring users would benefit from the same hook.

### Why this is non-obvious
The library name suggests platform independence (`langchain4j` not `langchain4j-spring`), so the assumption is that all supplier types are equally extensible. The asymmetry between `@ChatModelSupplier` (has a resolver hook) and all other supplier annotations (do not) is not mentioned in any README, Javadoc, or issue tracker. The class is package-private in the library with no stable API guarantee.
