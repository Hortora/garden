## Symptom

Quarkus fails to start with:

```
SRCFG00014: Unrecognized configuration key "foo.bar.new-property" was provided;
it does not map to any root
```

even though the key is declared as `@ConfigProperty(name = "foo.bar.new-property")` in a
CDI bean.

## Context

Another bean or extension in the same module declares `@ConfigMapping(prefix = "foo.bar")`.
SmallRye strict validation treats a `@ConfigMapping` prefix as exclusively owned by the
mapping interface â€” any config key under that prefix that is NOT declared as an interface
method is rejected at startup, regardless of which bean owns the `@ConfigProperty`.

This fires in `@QuarkusTest` contexts where SmallRye strict validation is active. Can also
fire in production startup.

## Root cause

SmallRye Config's `@ConfigMapping` with a prefix creates an exclusive namespace. The
validator sees `foo.bar.new-property`, finds no matching method under `foo.bar.*` in the
mapping interface, and raises SRCFG00014. It does not check whether the key might be
consumed by a `@ConfigProperty` â€” the mapping interface wins.

## Fix

Add the property to the existing `@ConfigMapping` interface as a method, and inject the
config object rather than using `@ConfigProperty` directly:

**Before (broken):**
```java
// In some @ApplicationScoped bean:
@ConfigProperty(name = "foo.bar.new-property")
Optional<List<String>> newProperty;
```

**After (correct):**
```java
// In the @ConfigMapping interface:
@ConfigMapping(prefix = "foo.bar")
public interface FooBarConfig {
    // existing methods...
    @WithName("new-property")
    Optional<List<String>> newProperty();
}

// In the CDI bean â€” inject the config object:
@Inject FooBarConfig config;
// Use: config.newProperty()
```

SmallRye Config correctly parses comma-separated strings into `Optional<List<String>>`
when accessed via a `@ConfigMapping` interface method.

## Why this is non-obvious

The error message `SRCFG00014: it does not map to any root` suggests the key is
unrecognised globally â€” not that it conflicts with a mapping in the same module. The
`@ConfigProperty` annotation is the documented way to inject a single config value and
works everywhere except when a `@ConfigMapping` already claims the prefix. The fix
(using the mapping interface instead) is non-obvious because `@ConfigProperty` and
`@ConfigMapping` are presented as independent alternatives with the same scope.

### Why this fix

Adding to the `@ConfigMapping` interface is the correct ownership model: SmallRye
expects one authority per prefix, and the mapping interface is that authority. Using
`@ConfigProperty` to "add" a property under an owned prefix violates the model.
