### Symptom

`jakarta.enterprise.inject.UnsatisfiedResolutionException` at augmentation time when using a Quarkus library module that defines a `@DefaultBean @ApplicationScoped` SPI implementation. The exception says no bean is available for the injection type — but the bean class is clearly on the classpath and CDI-annotated.

No warning is emitted that the bean was eliminated. The error looks identical to a missing implementation.

### Context

A library is split across two Maven modules:

- `core` defines a SPI interface (`Foo`) and a `@DefaultBean @ApplicationScoped NoOpFoo implements Foo` default
- `mcp` depends on `core` and contains the only `@Inject Foo` injection point in the consuming app

An application adds both `core` and `mcp` as dependencies. Augmentation fails.

Concrete case: `casehub-connectors-core` defines `ConnectorMeshBridge` SPI and `NoOpConnectorMeshBridge @DefaultBean`. The injection point `@Inject ConnectorMeshBridge` lives only in `casehub-connectors-mcp` tool classes. Without `@Unremovable`, augmentation of an app using both modules throws `UnsatisfiedResolutionException`.

### Root cause

Quarkus ARC performs unused-bean elimination at build time. It scans for injection points within the module being augmented, but `@DefaultBean` implementations that have no injection point visible within the defining module itself are treated as unused and eliminated. ARC does not traverse sibling modules to discover injection points — it only sees what is compiled together in the augmentation classpath at that moment.

The result: the `@DefaultBean` bean is removed before the consuming app can inject it.

### Fix

Add `@Unremovable` to the `@DefaultBean` class:

```java
@DefaultBean
@Unremovable   // ← required: injection point is in a different Maven module (mcp)
@ApplicationScoped
public class NoOpConnectorMeshBridge implements ConnectorMeshBridge {
    @Override
    public void notifyDelivered(String connectorId, String destination, String content) {}
}
```

`@Unremovable` instructs ARC to retain the bean in the CDI graph regardless of whether a visible injection point is found.

### Why this is non-obvious

The `UnsatisfiedResolutionException` error message is identical whether the bean is absent from the classpath or has been eliminated by ARC. Nothing in the error output mentions bean removal. Developers familiar with `@Unremovable` associate it with extension-published beans (where the Quarkus docs describe it explicitly) — not with plain `@DefaultBean` library beans whose injection point happens to be in a sibling module.

The protocol `quarkus-extension-unremovable-consumer-beans.md` covers the extension case; this gotcha surfaces the same issue in non-extension library modules where the split is a Maven module boundary rather than an extension deployment boundary.
