## Symptom

Embedding-based search (Qdrant + nomic-embed-text) returns plausible-looking but
semantically wrong results when queried with Java class names, CDI annotations,
or framework-specific identifiers. No error — the results just aren't relevant.

Examples from a 14-scenario benchmark:
- `@DefaultBean` → matched WireMock and Arabic Letter Mark entries
- `ChatModel` → matched HTTP caching and Maven caching entries
- `shadowing` → matched Shadow DOM entries instead of JPA field shadowing
- `stream` → matched Java streams, SSE, PostgreSQL notifications instead of CloudEvents

## Context

A garden corpus (~1,960 entries) indexed with nomic-embed-text 768-dim dense
embeddings in Qdrant. Three search methods compared across 14 real-world scenarios
(6 GitHub issues + 8 spec domains):

1. `git grep` with keywords (substring matching)
2. gardenSearch with same keywords (embedding similarity)
3. gardenSearch with natural language description (embedding similarity)

## What was tried (didn't work)

Keyword-based embedding search — passing the same keyword strings used for grep
to the vector search engine. Expected: at least comparable results since the
embedding should understand the terms. Actual: catastrophic failure in 12/14
scenarios. Average precision 32% vs grep's 65%.

## Root cause

General-purpose embedding models like nomic-embed-text are trained on natural
language corpora. Java identifiers (`ChatModel`, `ConcurrentHashMap`), CDI
annotations (`@DefaultBean`, `@Alternative`), and framework config properties
(`quarkus.datasource.active`) have no pre-trained semantic weight. The model
treats them as generic English tokens:

- `ChatModel` → "chat" + "model" → matches anything about conversations or models
- `@DefaultBean` → "default" + "bean" → matches anything with "default" in it
- `shadowing` → matches the English word, not the JPA concept

This is silent — relevance scores (0.53–0.70) look normal. The results are
well-ordered noise.

## Fix

Use natural language descriptions instead of keywords when querying embedding-based
search. The same engine with NL queries ("CDI ambiguous dependency resolution when
@Default bean conflicts with @DefaultBean in multi-module Quarkus application")
recovered to 62% average precision — competitive with grep's 65%.

The keyword catastrophe is a query formulation problem, not a retrieval problem.
NL queries give the embedding model enough semantic context to locate relevant
entries even when those entries use different vocabulary.

### Why this fix

Keywords work for substring matching because they match character sequences. They
fail for embedding search because the embedding model needs semantic context to
produce meaningful vectors. A keyword like `@DefaultBean` is a pointer to a
concept — substring matching follows the pointer; embedding search needs the
concept spelled out.

## Why this is non-obvious

Embedding models are marketed as understanding meaning, not just matching strings.
The expectation is that they should handle domain vocabulary at least as well as
grep. The failure is counterintuitive: a more sophisticated retrieval mechanism
produces dramatically worse results than a simple substring match — but only for
domain-specific identifiers, not for natural language descriptions of the same
concepts.

### Alternatives considered

- **Domain-tuned embedding model:** would likely fix the vocabulary gap but
  requires significant investment in fine-tuning data and training pipeline.
- **SPLADE hybrid search:** adds learned sparse token matching alongside dense
  embeddings — may recover keyword precision without domain tuning. Testing
  planned (Hortora/engine#28).
- **BM25/full-text index:** classic keyword retrieval that scales. Would handle
  exact identifiers but adds infrastructure complexity.
