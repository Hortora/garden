### Symptom
`@QuarkusTest` with `@TestTransaction` passes — no exceptions, no assertion failures on the return value. But a subsequent query within the same test method returns empty results for records that should have been written:

```java
@Test @TestTransaction
void investigate_caseOpenedEntryExistsInLedger() {
    AmlInvestigationResult result = service.investigate(tx);
    // ledgerRepo.findBySubjectId() returns empty — entry not found
    var entry = ledgerRepo.findEntryById(result.ledgerCaseEntryId());
    assertTrue(entry.isPresent()); // FAILS
}
```

The service call succeeds, returns a non-null `ledgerCaseEntryId`, but the entry cannot be found by the repository query.

### Context
The test calls a service that writes to an audit ledger (using `@Transactional(REQUIRED)` — standard transaction propagation). The test is annotated with `@TestTransaction`, which wraps the entire test in an outer transaction and rolls it back at the end.

### Root cause
`@TestTransaction` creates an outer transaction for the test. The service's `@Transactional(REQUIRED)` joins the outer transaction — there is no commit. The ledger writes are queued in the Hibernate session but never flushed and committed to the database at the SQL level.

When the repository query runs later in the same test, it executes a SQL query. Depending on whether Hibernate has auto-flushed before the query, the results may or may not include the pending writes. In practice, with `generation=none` + H2, the pending writes from one JPA `EntityManager` are not visible to a second `EntityManager` (which may be used by the repository), so `findEntryById()` returns empty.

The writes are NOT committed — they exist only in the current session's pending state. The rolled-back outer transaction at test end confirms nothing was persisted.

This is commonly misdiagnosed as a `REQUIRES_NEW` issue, but the inner writes use `REQUIRED` (the default). The problem is purely the outer rolled-back test transaction hiding writes from subsequent queries.

### What was tried (didn't work)
Assumed the issue was `LedgerWriteService` using `REQUIRES_NEW` (it doesn't — it uses `REQUIRED`). Increasing the H2 connection pool size (irrelevant). Both failed to diagnose the real cause.

### Fix
Remove `@TestTransaction`. Use unique data identifiers per test (e.g. unique transaction IDs) to ensure tests don't interfere with each other's data. The committed data from each test remains in the H2 in-memory database for the duration of the test suite run, but since each test writes to distinct subjects, isolation is maintained without rollback.

```java
// Before — hidden writes:
@Test @TestTransaction
void investigate_caseOpenedEntryExistsInLedger() { ... }

// After — writes committed, queries see them:
@Test
void investigate_caseOpenedEntryExistsInLedger() { ... }
```

Alternatively, if rollback isolation is required, use `@Transactional(REQUIRES_NEW)` on the service layer to commit writes independently of the test transaction — but this changes the production service behaviour and is rarely the right fix.

### Why this is non-obvious
`@TestTransaction` is described as providing "test isolation" — developers assume it makes tests cleaner and safer. The expectation is that writes during the test are visible within the same test. In practice, pending writes in the outer rolled-back transaction may not be visible to subsequent queries, especially when multiple `EntityManager` instances are involved (which happens with named persistence units in Quarkus). The symptom — a service succeeds but the written data is "missing" — looks exactly like the service silently failed to write, not like a transaction visibility issue.
