# ReliableKieSession.dispose() removes persisted H2MVStore/Infinispan data — not just in-memory state

## Pain (4)

Silently destroys data you expect to survive session disposal. After disposing a reliable KieSession and "restarting", session recovery finds no persisted data — the session is empty. No error is thrown, so the data loss is discovered only when recovery produces an unexpected empty session.

## Non-obvious (3)

`dispose()` on a standard `KieSession` means "release resources" — developers naturally expect the same from a `ReliableKieSession`. The Javadoc doesn't mention that `dispose()` on a reliable session deletes persistent data.

## Reuse (3)

Affects all drools-reliability users, especially anyone writing tests that simulate restart by disposing and recovering sessions.

## Problem

After disposing a reliable KieSession and "restarting", session recovery via `PersistedSessionOption.fromSession(savedId)` finds no persisted data — the session is empty. This occurs when writing tests that simulate JVM restart by disposing the session and then recovering it.

## Root Cause

`ReliableStatefulKnowledgeSessionImpl.dispose()` calls `removeStoragesBySessionId()` internally, deleting all H2MVStore/Infinispan entries for that session. `dispose()` on a reliable session means "I'm done with this session permanently", not "release the in-memory resources".

## Solution

For test restart simulation, clear the in-memory session reference WITHOUT calling `dispose()`. Drools' own test infrastructure uses a `failover()` pattern that nulls the reference without disposing:

```java
// Don't do this:
kieSession.dispose();
// Do this instead:
kieSession = null; // or just let it go out of scope
```

For production code, only call `dispose()` when you truly want to delete the session and its persisted state permanently.
