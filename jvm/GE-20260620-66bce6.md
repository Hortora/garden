## Context

Upstream multi-repo Maven project publishes a SNAPSHOT containing a Java record. A new field is
added at a mid-position (not appended at the tail). Downstream project has tests that construct
the record positionally. Locally, Maven has cached `.class` files compiled against the old
snapshot. CI fetches the latest SNAPSHOT on every run (clean cache).

## Symptom

CI fails with:
```
NoSuchMethodError: 'void io.casehub.api.model.ProvisionContext.<init>(java.util.UUID,
  java.lang.String, io.casehub.api.model.WorkerContext, ...)'
```

Or, on strict CI configurations, a compile error:
```
constructor ProvisionContext in record ProvisionContext cannot be applied to given types;
  required: UUID, String, String, WorkerContext, PropagationContext, String, String
  found: UUID, String, WorkerContext, PropagationContext, String, String
```

Local build passes — stale cached `.class` files mask the breakage.

## Root Cause

Java record constructors are positional. Inserting a field at position N shifts all
subsequent fields by one. Every positional call site (test helpers, fixture builders) is
silently wrong — the types of the shifted args may still satisfy the compiler if adjacent
types are compatible (e.g. two adjacent `String` fields), or fail with a count mismatch.

`NoSuchMethodError` is the runtime signature of this: the JVM looks for the old N-arg
constructor, finds only the new (N+1)-arg one, and throws. The error message names the
OLD signature — which makes it look like the method was removed, obscuring that a field
was added.

Local clean builds catch it immediately. Stale `.class` caches let it through and it
only surfaces on CI (clean checkout per run).

## Fix

Add `null` (or the appropriate zero value) at the inserted field's position in each call
site. Search across the entire dependent codebase — multiple test helpers can contain the
same positional constructor call:

```java
// Before — record had 6 fields (caseId, tenancyId, workerContext, ...)
private ProvisionContext ctx(UUID caseId) {
    return new ProvisionContext(caseId, "finance", null, null, null, null);
}

// After — record gained taskType at position 3
private ProvisionContext ctx(UUID caseId) {
    return new ProvisionContext(caseId, "finance", null, null, null, null, null);
}
```

Use `ide_search_text` (or IDE find-usages on the record class) to locate ALL call sites —
not just the one that failed. Multiple test helpers in different modules can contain the
same pattern.

## Why This Is Non-Obvious

- **The error blames the caller, not the record.** `NoSuchMethodError` names the old
  constructor signature — it looks like the upstream removed a method, not added a field.
- **Local cache masks it.** Stale `.class` files compiled against the old snapshot pass
  locally; CI (clean checkout) hits it immediately. The failure is CI-only for as long as
  the local cache is warm.
- **Type-compatible insertions skip the compile check.** If the inserted field type
  matches an adjacent field's type (e.g. two consecutive `String` fields), the compiler
  may accept the shifted call silently — only the runtime throws.
- **Mid-position insertion is the key.** Tail-appended fields always cause an arg-count
  mismatch that the compiler catches on next local compile. Mid-position fields shift
  existing args into new slots without changing the total count when types align.

## Constraints

Applies to Java records with positional constructors. Named/builder patterns are immune.
Most likely in multi-repo Maven SNAPSHOT setups where the consumer's local cache is
warmer than CI's.
