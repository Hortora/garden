### Symptom
`MemoryQuery` with `.withQuestion("review history for app in repo1")` returns empty results
even though matching entities with the correct entityId, domain, and tenantId are stored.
No error — silent empty list. The test passes all other assertions but `codeAreaHistory`
is empty.

### Context
Writing `@QuarkusTest` or unit tests for `CaseMemoryRecaller` (or any CaseMemoryStore consumer)
with `InMemoryMemoryStore` on the classpath. The recaller builds a natural-language question
string from PR metadata and passes it via `MemoryQuery.withQuestion()`. The stored facts have
descriptive text that doesn't contain the question as a literal substring.

### What was tried (didn't work)
Stored a fact with text `"Security review of the app module found no issues"` and queried
with question `"review history for app in repo1"`. Expected the query to match (both mention
"app" and "review"). Got empty results.

### Root cause
`InMemoryMemoryStore.query()` applies the question filter as:
```java
.filter(m -> query.question() == null
    || m.text().toLowerCase().contains(query.question().toLowerCase()))
```
This is a full-substring match — the stored text must contain the **entire** question string
as a continuous substring. `"security review of the app module..."` does not contain
`"review history for app in repo1"` as a substring.

Real adapters handle `question` differently:
- **JPA** (`memory-jpa`): `websearch_to_tsquery` — tokenized FTS, matches individual words
- **SQLite** (`memory-sqlite`): `MATCH` with FTS5 — tokenized, ranked
- **Mem0** (`memory-mem0`): vector semantic search via `POST /search`

The in-memory adapter's naive implementation creates a false test failure — the production
path would have returned results.

### Fix
In tests using `InMemoryMemoryStore`, ensure stored text includes the question string as a
substring:
```java
// Question will be: "review history for app in repo1"
store.store(new MemoryInput(
    "module:repo1/app", domain, tenantId, caseId,
    "review history for app in repo1 — no critical findings",  // contains question
    attrs));
```

Alternatively, for unit tests with a custom `SimpleMemoryStore`, implement the same naive
filter to match `InMemoryMemoryStore` behaviour, or skip the question filter entirely if
the test only validates query construction (entityIds, limits, ordering).

### Why this is non-obvious
The `question` field on `MemoryQuery` is documented as a semantic hint for relevance ranking.
A developer would expect it to behave like a search query (tokenized, fuzzy, or semantic) —
not a literal substring filter. The in-memory adapter silently filters out results rather
than ignoring the question field, so the failure looks like a store or query construction
bug rather than a test-adapter mismatch.

### Alternatives considered
- Omitting `.withQuestion()` from the query — works but doesn't exercise the full recall path
- Building a test adapter with tokenized matching — unnecessary complexity for a test helper
