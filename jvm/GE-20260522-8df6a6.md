### Symptom
Code using `list("field in ?1", collection)` in a Panache MongoDB entity compiles cleanly
and passes unit tests but may fail at runtime or return incorrect results because the
MongoDB Panache query string parser is not the same as the Hibernate ORM JPQL parser.

### Context
Panache ORM (`PanacheEntityBase`) and Panache MongoDB (`PanacheMongoEntityBase`) expose
the same surface-level API — both have `list(String query, Object... params)`. In Panache
ORM this is JPQL; in Panache MongoDB it is a MongoDB-specific query language. The two
look identical at the call site but the backend parsers differ. JPQL-style expressions
such as `"scope in ?1"` are not guaranteed to be parsed correctly by the MongoDB backend.

### Fix
Use the unambiguous BSON `Filters` API instead of query strings for any non-trivial filter:

```java
import com.mongodb.client.model.Filters;

// ❌ Unvalidated — looks correct but relies on undocumented parsing
return list("scope in ?1", scopes);

// ✅ Unambiguous — uses the MongoDB driver's own filter builder
return list(Filters.in("scope", scopes));
```

`Filters` is part of the MongoDB Java driver, which is a transitive dependency of
`quarkus-mongodb-panache`. The overload `list(Bson filter)` is always available.

### Why this is non-obvious
The two Panache APIs share identical method names. Developers copy patterns from Panache
ORM code and assume the same query string syntax works in MongoDB mode. There is no
compile-time or startup-time error — the failure only surfaces when the specific query
path is exercised. The `Filters.*` API is the correct path but requires knowing to
import from `com.mongodb.client.model`, which isn't advertised in the Panache MongoDB
getting-started docs.
