### Symptom

After a `mvn install` picks up a new neocortex SNAPSHOT, existing code that calls `CbrQuery.of()` (5 args) or `CbrCaseMemoryStore.store()` (6 args) fails to compile with "actual and formal argument lists differ in length" or "incompatible types: java.lang.String cannot be converted to io.casehub.platform.api.path.Path". The errors only surface on a full rebuild — incremental compilation may not catch them.

### Context

`CbrQuery.of()` changed from 5 parameters `(tenantId, domain, caseType, features, topK)` to 6 parameters `(tenantId, domain, scope, caseType, features, topK)` where `scope` is `io.casehub.platform.api.path.Path`. Similarly, `CbrCaseMemoryStore.store()` gained a 7th `Path scope` parameter. No compile-time deprecation — the old overloads were removed outright.

### What was tried (didn't work)

Adding `Path.root()` as the scope argument. This compiles in test files that don't use JAX-RS, but in REST resource classes that already import `jakarta.ws.rs.Path`, the unqualified `Path.root()` resolves to the JAX-RS `@Path` annotation — producing "cannot find symbol: method root()" with no hint about which `Path` is being resolved.

### Root cause

Two issues compound:

1. **SNAPSHOT API break without overload preservation.** The old signatures were removed, not deprecated. All consumers break simultaneously on next SNAPSHOT resolution.

2. **`Path` name collision.** `io.casehub.platform.api.path.Path` (the scope type) and `jakarta.ws.rs.Path` (the JAX-RS annotation) share the simple name `Path`. In any file with `import jakarta.ws.rs.Path`, the compiler resolves unqualified `Path` to the annotation. Import optimisers (IntelliJ, IDE auto-import) will never add the platform `Path` import because the name is already resolved.

### Fix

Add `io.casehub.platform.api.path.Path.root()` (fully qualified) as the scope argument in all call sites that import `jakarta.ws.rs.Path`:

```java
// In REST resources (import jakarta.ws.rs.Path already present):
CbrQuery.of(tenantId, domain,
    io.casehub.platform.api.path.Path.root(), "clinical-ae",
    features, 10);

// In test files without JAX-RS (no collision):
CbrQuery.of(tenantId, domain,
    Path.root(), "clinical-ae",
    features, 10);
```

For `CbrCaseMemoryStore.store()`, append `io.casehub.platform.api.path.Path.root()` as the 7th argument.

### Why this is non-obvious

The compiler error says "String cannot be converted to Path" — pointing at the `caseType` argument that shifted position. The developer fixes the argument count, adds `Path.root()`, and gets a *different* error ("cannot find symbol: method root()") because `Path` now resolves to `jakarta.ws.rs.Path`. Two separate issues masquerade as one, and the second error gives no indication that a different `Path` class is needed.
