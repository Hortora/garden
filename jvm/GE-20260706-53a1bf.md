### Symptom

Test module depends on a Quarkus extension (e.g. `casehub-ledger`) whose runtime JAR contains `@Entity` classes. The test classpath includes an in-memory persistence module with `@Alternative @Priority(1)` implementations that bypass JPA entirely. Despite this, Quarkus startup fails:

```
ConfigurationException: Persistence unit '<default>' defines entities
[io.casehub.ledger.runtime.model.ActorIdentity, ...] but its datasource
'<default>' cannot be found: Datasource '<default>' is not configured.
```

### Context

The test module's `pom.xml` depends on the Quarkus extension runtime and an in-memory persistence module. No datasource is configured because the in-memory implementations handle all reads/writes via `ConcurrentHashMap` — JPA is never used at runtime.

### What was tried (didn't work)

Running tests with only the in-memory persistence module on the classpath and no datasource configuration. Expected the `@Alternative @Priority(1)` in-memory implementations to eliminate the need for any JPA/datasource setup.

### Root cause

Quarkus augmentation discovers `@Entity` classes via Jandex index scanning of extension runtime JARs. `HibernateOrmProcessor#configurationDescriptorBuilding` validates that every persistence unit with discovered entities has a configured datasource. This validation runs at **build time** — before CDI bean resolution. `@Alternative @Priority(1)` annotations are CDI concepts resolved at **runtime**. Augmentation has no visibility into which CDI alternatives will replace which beans.

The mental model "in-memory alternatives replace JPA" is correct at runtime but wrong at augmentation time.

### Fix

Add `quarkus-jdbc-h2` as a test dependency and configure a minimal H2 datasource in `application.properties`:

```xml
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-jdbc-h2</artifactId>
  <scope>test</scope>
</dependency>
```

```properties
quarkus.datasource.db-kind=h2
quarkus.datasource.jdbc.url=jdbc:h2:mem:testdb;MODE=PostgreSQL;DB_CLOSE_DELAY=-1
quarkus.datasource.username=sa
quarkus.datasource.password=

quarkus.hibernate-orm.database.generation=none
quarkus.flyway.migrate-at-start=true
quarkus.flyway.locations=classpath:db/ledger/migration
```

The datasource satisfies augmentation validation. The in-memory `@Alternative @Priority(1)` implementations still handle all actual persistence at runtime — H2 is never queried.

### Why this is non-obvious

The `@Alternative @Priority(1)` pattern is specifically designed to replace default implementations. A developer correctly reasons: "I'm replacing all JPA repositories with in-memory alternatives, so I don't need a datasource." This reasoning is valid at the CDI layer but fails because Quarkus augmentation validates the JPA layer independently, before CDI alternatives are resolved.

### Why this fix

The H2 datasource is a no-op cost — it exists only to satisfy augmentation. The alternative (`quarkus.hibernate-orm.enabled=false`) would disable the entire Hibernate subsystem, which may break other extension features that rely on entity metadata even when in-memory persistence handles the actual data.

### Alternatives considered

- `quarkus.hibernate-orm.enabled=false` — disables Hibernate entirely; rejected because extension deployment processors may rely on entity metadata for build-time validation or code generation
- Excluding entity classes from Jandex — not feasible; the entities are in the extension JAR which is already indexed
