## Technique

Test Flyway migrations with a plain-Java JUnit 5 test (no Quarkus, no Spring) that runs all migrations against H2 in-memory, then asserts schema correctness via JDBC metadata. This bypasses the `drop-and-create` path used by framework test runners and proves that your migration files actually produce the right schema.

```java
@BeforeAll
static void migrate() throws Exception {
    // Seed cross-module tables that your migrations FK to (simulate deployment order)
    try (Connection conn = DriverManager.getConnection(JDBC_URL, "sa", "")) {
        conn.createStatement().execute("""
                CREATE TABLE IF NOT EXISTS ledger_entry (
                    id UUID NOT NULL,
                    CONSTRAINT pk_ledger_entry PRIMARY KEY (id)
                )
                """);
    }
    Flyway.configure()
            .dataSource(JDBC_URL, "sa", "")
            .locations("classpath:db/migration/mymodule")
            .baselineOnMigrate(true)
            .baselineVersion("0") // Must be < V1 so V1 runs — see GE-20260519-c1ce15
            .load()
            .migrate();
}

@Test
void deadTableIsAbsent() throws Exception {
    try (Connection conn = DriverManager.getConnection(JDBC_URL, "sa", "");
         var rs = conn.getMetaData().getTables(null, null, "OLD_TABLE", new String[]{"TABLE"})) {
        assertFalse(rs.next(), "OLD_TABLE must not exist in the schema");
    }
}

@Test
void newTableExists() throws Exception {
    try (Connection conn = DriverManager.getConnection(JDBC_URL, "sa", "");
         var rs = conn.getMetaData().getTables(null, null, "NEW_TABLE", new String[]{"TABLE"})) {
        assertTrue(rs.next(), "NEW_TABLE must exist after migration");
    }
}
```

**Key details:**
- H2 with `MODE=PostgreSQL` for compatibility: `jdbc:h2:mem:test_N;MODE=PostgreSQL;DB_CLOSE_DELAY=-1`
- Use a unique DB name per test run (e.g. `System.nanoTime()` suffix) to prevent cross-run state sharing when Surefire reuses the JVM
- `getMetaData().getTables()` uses UPPERCASE table names in H2 (`"MY_TABLE"`, not `"my_table"`)
- Close the `ResultSet` explicitly in try-with-resources — it is NOT closed when the `Connection` closes
- For cross-module FK dependencies, seed the minimal table before running Flyway (just the PK column)

## Why This Is Non-Obvious

Framework test runners (Quarkus `@QuarkusTest`, Spring `@SpringBootTest`) use `drop-and-create` or schema-validation modes that bypass Flyway entirely. Tests pass even when migration files have syntax errors or wrong table names. This technique bypasses the framework to actually run Flyway — the only way to prove the `.sql` files are correct. Most developers trust `drop-and-create` and never discover the migration files have drifted from the entity definitions until they try to deploy.
