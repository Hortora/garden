### Symptom

REST endpoint returns HTTP 200 with `items.size()=0` and `totalCount=0` — no error, no warning. The endpoint works perfectly in production but returns empty results in `@QuarkusTest` integration tests, even when data is correctly seeded in `@BeforeEach @Transactional`.

### Context

Any `@QuarkusTest` that:
1. Uses `@TestSecurity(user = "admin", roles = {"household-admin"})` for endpoint access
2. Has a service layer that calls `CurrentPrincipal.groups()` to filter results by role (e.g., `LifeCaseVisibilityPolicy`, `LifeTaskVisibilityPolicy`)
3. Uses `FixedCurrentPrincipal` (`@Alternative @Priority(200)`) as the `CurrentPrincipal` implementation in tests

### What was tried (didn't work)

- Verified data was seeded correctly (it was — `@BeforeEach @Transactional` commits)
- Checked query logic (correct — `LifeCaseTracker.find()` returns results)
- Assumed `@TestSecurity` roles would flow through to `CurrentPrincipal.groups()` (they don't)

### Root cause

`@TestSecurity` and `CurrentPrincipal` are completely separate CDI mechanisms:

- `@TestSecurity` sets the Quarkus `SecurityIdentity` — this controls `@RolesAllowed` checks on REST endpoints
- `CurrentPrincipal` is a separate CDI bean (`FixedCurrentPrincipal` in tests) with its own `groups()` method that defaults to `new HashSet<>()` (empty)

When a visibility policy calls `groups.contains(HouseholdGroups.ADMIN)`, it gets `false` because `FixedCurrentPrincipal.groups()` is empty — regardless of what `@TestSecurity` says. The endpoint passes `@RolesAllowed` (SecurityIdentity has the role) but the service layer drops all results (CurrentPrincipal has no groups).

### Fix

Inject `FixedCurrentPrincipal` in the test class and explicitly set groups:

```java
@Inject
FixedCurrentPrincipal currentPrincipal;

@BeforeEach
@Transactional
void setup() {
    currentPrincipal.setGroups(Set.of(HouseholdGroups.ADMIN));
    // ... seed data
}

@AfterEach
@Transactional
void cleanup() {
    currentPrincipal.reset();
    // ... clean data
}
```

For tests with method-level `@TestSecurity` overrides (e.g., testing junior visibility), also override `CurrentPrincipal` in the test method body:

```java
@Test
@TestSecurity(user = "junior", roles = {HouseholdGroups.JUNIOR})
void juniorSeesEmptyList() {
    currentPrincipal.setGroups(Set.of(HouseholdGroups.JUNIOR));
    // ... assertions
}
```

### Why this is non-obvious

The natural assumption is that `@TestSecurity(roles = {"household-admin"})` makes the test principal an admin everywhere — both for endpoint access control and for business logic. The two mechanisms look like they should be connected, but they are completely independent CDI paths. The failure mode is silent (HTTP 200, valid JSON, zero items) with no log output indicating the filter dropped results.

### Alternatives considered

- Using `@InjectMock` on `CurrentPrincipal` — rejected because `FixedCurrentPrincipal` already exists as the test `@Alternative` and provides `setGroups()`/`reset()` lifecycle
- Adding groups to `@TestSecurity` annotation — not possible; `@TestSecurity` only controls `SecurityIdentity`, not arbitrary CDI beans
