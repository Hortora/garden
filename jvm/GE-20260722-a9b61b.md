### Symptom

`IllegalArgumentException: ensembleConfidence must be in [0,1]` thrown at runtime when constructing `EnsemblePlan` from retrieval results with negative cosine similarity scores.

### Context

`ScoredCbrCase.score()` range is `[-1, 1]` (cosine similarity — negative values are valid and represent dissimilar cases). `EnsemblePlan.ensembleConfidence` range is `[0, 1]` (a quality/confidence metric). Passing the raw score as confidence works silently for all positive scores — the bug only surfaces when a negative cosine similarity reaches the EnsemblePlan constructor.

### What was tried (didn't work)

Passing `scoredCases.get(bestIdx).score()` directly as `ensembleConfidence` — compiles, works for common positive-score cases, fails at runtime only when retrieval returns negative cosine similarity.

### Root cause

Domain mismatch: cosine similarity and confidence occupy overlapping but different numeric ranges. The overlapping positive region masks the bug during development and testing with typical (positive) scores.

### Fix

Clamp the score to the confidence range:

```java
Math.max(0.0, scoredCases.get(bestIdx).score())
```

### Why this is non-obvious

The code compiles cleanly and passes all tests with typical positive retrieval scores. The `[-1, 1]` range of `ScoredCbrCase.score` and the `[0, 1]` range of `EnsemblePlan.ensembleConfidence` are documented in separate record constructors — the mismatch is only visible when reading both validation blocks side by side. A skilled developer writing a bridge between retrieval and ensemble would naturally pass the score through, and only discover the issue when a sufficiently dissimilar case triggers a negative score.
