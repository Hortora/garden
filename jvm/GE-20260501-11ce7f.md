## MessageLedgerEntry.content is null for EVENT entries — LIKE content search silently returns nothing

### Symptom

A JPQL query with `e.content LIKE :pattern` against `MessageLedgerEntry` returns 0 results for
EVENT-type entries, even when the EVENT was posted with JSON content containing the search string.
No error is raised.

### Context

`MessageLedgerEntry` extends `LedgerEntry` (casehub-ledger JPA hierarchy). The `content` column
stores the message body for normative types (COMMAND, DONE, DECLINE, etc.). For EVENT entries,
`LedgerWriteService.record()` extracts telemetry into dedicated fields (`toolName`, `durationMs`,
`tokenCount`) and sets `content = null`. The Javadoc on the field says exactly this, but it is easy
to miss when building cross-channel queries.

### What was tried

Implemented cross-channel correlation via a JPQL LIKE query:
```java
"WHERE e.messageType = 'EVENT' AND e.content LIKE :pattern"
// pattern = "%\"correlation_id\":\"<uuid>\"%"
```
Tests asserting EVENT entries appeared in cross-channel results consistently returned 0 matches.
The LIKE pattern was syntactically correct. The content that was posted matched the pattern. The
column was simply null.

### Root cause

`LedgerWriteService` does not copy the original message content into `MessageLedgerEntry.content`
for EVENT messages. It extracts known telemetry fields and discards the rest. The field is null,
so any LIKE query on it will never match.

### Fix

Do not use content-based LIKE matching for cross-channel correlation of EVENT entries.
Instead, require agents to set an explicit `correlationId` when sending EVENT messages related to
an obligation:

```java
// Agent sends EVENT with explicit correlationId
tools.sendMessage("case-abc/observe", "agent", "EVENT",
    "{\"tool\":\"read_file\"}", corrId, null, null, null, null);

// Repository query — simple, reliable, no content search
"SELECT e FROM MessageLedgerEntry e WHERE e.correlationId = :corrId ORDER BY e.messageId ASC"
```

`correlationId` IS stored on EVENT ledger entries (it comes from the message, not from content extraction).

### Why this is non-obvious

The content field exists on the entity and is non-null for every other message type. The LIKE
pattern itself is syntactically correct. There is no error — the query succeeds and returns an
empty list. The only signal is in the Javadoc comment on the `content` field, which is easy to
overlook when writing a new cross-entity query.
