### The technique
When building GraalVM native images with JNI libraries that have no upstream reachability metadata, the tracing agent in JVM mode is the only viable discovery path for the complete config. No manual inspection of source code or documentation can reliably produce the full set of JNI, reflection, and resource entries.

### How to apply

Run the application in JVM mode with the tracing agent:

```bash
java -agentlib:native-image-agent=config-output-dir=target/agent-config \
  -jar target/quarkus-app/quarkus-run.jar \
  <application-args>
```

Exercise all code paths that use JNI. The agent generates `reachability-metadata.json` capturing every reflection, JNI, and resource access. Copy the output to `src/main/resources/META-INF/native-image/<group>/<artifact>/`.

### Why this is non-obvious
The intuitive approach is to inspect library source for `native` method declarations, `System.loadLibrary()` calls, and reflection usage — then manually register each one. This fails because:

1. **Rust→Java JNI callbacks are invisible** — DJL's tokenizer has Rust code that calls `JNIEnv::NewObject` to create Java objects. These callbacks don't appear in Java source at all.
2. **JNA bridge classes are implicit** — DJL uses JNA, which has its own JNI surface (`com.sun.jna.*`) that must be registered. Dozens of JNA classes need JNI access.
3. **Transitive resource dependencies** — ONNX Runtime bundles multiple `.dylib` files (core, JNI bridge, providers, WebGPU) that all need resource registration.
4. **No upstream metadata exists** — as of June 2026, ONNX Runtime, DJL, LangChain4j, quarkiverse/quarkus-langchain4j, and the GraalVM reachability metadata repo all lack native-image config for these libraries.

### What developers would normally do instead
Manually inspect source, write JNI/reflection config by hand, and iterate on build failures. This works for simple libraries but misses Rust→Java callbacks and implicit JNA requirements, leading to hours of debugging opaque runtime failures (Rust panics, UnsatisfiedLinkError) with no clear diagnostic.
