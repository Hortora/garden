### Symptom

Orphaned `ScheduledFuture` tasks fire indefinitely after timer reconfiguration — consuming CPU, triggering unexpected reconciliation cycles, and accumulating over time with no visible error.

### Context

Any code that stores `ScheduledFuture` references in a `Map` (or similar collection) and dynamically replaces them — e.g., recomputing interval-grouped timers when configuration changes.

### Root cause

The cancel-all → clear-map → schedule-new sequence is not atomic. When two threads execute it concurrently:

1. Thread A cancels all futures, clears the map
2. Thread B cancels all futures (already cancelled — no-op), clears the map (already empty)
3. Thread A schedules new futures, puts them in the map
4. Thread B schedules new futures, **overwrites** Thread A's map entries

Thread A's `ScheduledFuture` instances remain alive in the executor pool but their references are gone — they can never be cancelled. Each concurrent race leaks one full set of timers permanently.

### What was tried (didn't work)

Relying on `ScheduledFuture.cancel()` alone — cancelling the old futures before scheduling new ones is necessary but insufficient without atomicity. The race window is between cancel and map insertion.

### Fix

Synchronize the entire cancel-clear-schedule block:

```java
void updateTimers(Map<Duration, Set<NodeType>> newGroups) {
    synchronized (this) {
        for (ScheduledFuture<?> future : resyncFutures.values()) {
            future.cancel(false);
        }
        resyncFutures.clear();
        scheduleIntervalGroups(newGroups);
    }
}
```

The same `synchronized(this)` pattern should be used on any other method that reads or writes the futures map.

### Why this is non-obvious

The race requires two concurrent calls to the same reconfiguration method on the same object — a narrow window that's unlikely in testing but real under production load. The leaked futures produce no errors — they silently execute their tasks, which may appear as duplicate or stale processing. The symptom (extra reconciliation cycles, slightly higher CPU) doesn't obviously point to a timer leak.

Discovered in `ReconciliationLoop.TenantLoop.updateDesired()` where concurrent goal recompilation could leak interval-grouped reconciliation timers.
