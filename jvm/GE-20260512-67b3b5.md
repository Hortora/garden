## Panache find() alias-prefixed field names return empty results silently — bare field names required

### Symptom

`Entity.find("alias.fieldName = :param", ...)` returns an empty list with no exception. The same query with the same data returns results through JPQL or native SQL.

### What was tried (didn't work)

- Verifying the parameter value is correct — it is
- Checking the entity field name — it matches
- Wrapping in explicit JPQL — works correctly

### Root cause

Panache's `find()`, `list()`, and `count()` methods use a **simplified HQL dialect**, not full JPQL. This dialect does not support alias-prefixed field names (`wi.assigneeId`). Instead of throwing a parse error, it silently produces a query that returns zero results.

### Fix

Use bare entity field names in all Panache queries:

```java
// Wrong — returns empty list silently
WorkItem.find("wi.assigneeId = :id", Map.of("id", userId));

// Right
WorkItem.find("assigneeId = :id", Map.of("id", userId));
```

### Why this is non-obvious

JPQL and SQL both support alias-prefixed column references — developers carry that expectation into Panache. The silent empty result (no exception, no warning) is identical to a legitimate zero-results query, making this extremely hard to diagnose without knowing the Panache simplified dialect restriction.
