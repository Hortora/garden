### Problem
Two database writes must commit atomically, but the calling method also has a fail-open
contract: catch any exception and return normally (e.g. so an HTTP endpoint always returns
200). Putting `@Transactional` on the calling method breaks fail-open (see GE-20260603-f32ff2).
Not using `@Transactional` breaks atomicity.

### Technique
Extract the atomic write pair to a package-private `@ApplicationScoped` CDI bean with
`@Transactional` on its method. The calling method is NOT `@Transactional`.

```java
@ApplicationScoped
class GateDispatcher {                   // package-private — caller controls the API

    @Inject MessageService messageService;

    @Transactional                        // owns the atomic boundary
    void dispatch(boolean approved, UUID oversightChannelId, UUID workChannelId,
                  long commandMessageId, UUID gateId, String rawOutput) {
        messageService.dispatch(/* RESPONSE or DECLINE */);
        messageService.dispatch(/* STATUS */);
    }
}
```

The caller:
```java
public void fulfill(UUID gateId, String rawOutput) {
    try {
        // ... guard reads ...
        gateDispatcher.dispatch(approved, oversightChannel.id, workChannel.id, ...);
        // catch block WORKS here — fulfill() is not @Transactional
    } catch (Exception e) {
        log.errorf("fulfill() failed: %s", e.getMessage());
    }
}
```

If the dispatcher throws (e.g. second write fails), `TransactionalException` propagates
into `fulfill()`'s catch block. `fulfill()` is not `@Transactional`, so there's no outer
transaction to mark rollback-only. The catch block swallows the exception. The dispatcher's
transaction already rolled back. Fail-open is preserved AND both writes are atomic on the
success path.

### Why this is non-obvious
The obvious fix is `@Transactional` on the calling method — but that breaks fail-open
(the CDI interceptor fires after the method returns, past the catch block). The second
obvious fix is a try/catch in the resource layer — but that spreads the fail-open concern
away from the service. Extracting one CDI bean solves both problems without changing any
caller-visible API.

### Constraint
Pass only value types (UUID, boolean, String, long) — not JPA entities — to the
dispatcher. Entities detach when crossing a `@Transactional` boundary and accessing their
lazy associations throws `LazyInitializationException`.
