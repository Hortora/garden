### Symptom
Two threads racing to set up the same Qdrant collection — one succeeds, the other
gets an error from `createCollectionAsync` because the collection already exists.
No such error from `createPayloadIndexAsync` on the same race.

### Context
Any concurrent code path that lazily creates Qdrant collections on first use —
e.g. `ensureCollection()` called from parallel `ingest()` invocations. The blocking
`QdrantEmbeddingIngestor` uses `ConcurrentHashMap.newKeySet()` as a cache with a
check-then-act race; two threads can both see "not cached" and both attempt creation.

### Root cause
`createCollectionAsync` returns an error when the collection already exists — it is
NOT idempotent. `createPayloadIndexAsync` silently succeeds for existing indexes of
the same type — it IS idempotent. The Qdrant API has asymmetric idempotency between
these two operations.

### Fix
The race is benign via retry recovery: the losing thread's `createCollectionAsync`
fails, the exception propagates before the collection is cached, and the next call
retries via the existing-collection path (which validates and proceeds). No locking
needed — just ensure the error handling doesn't cache a failed creation.

For payload indexes, no special handling needed — concurrent `createPayloadIndexAsync`
calls are safe.

### Why this is non-obvious
Most database "create" operations are idempotent (CREATE TABLE IF NOT EXISTS, etc.).
The natural assumption is that `createCollectionAsync` follows the same pattern.
The Qdrant Java client docs don't highlight this asymmetry, and the gRPC error
message doesn't suggest "use collectionExistsAsync first."
