## waitForSession() barrier pattern — eliminate @OnOpen registration races in WebSocket integration tests

**ID:** GE-20260620-04450c
**Stack:** Quarkus WebSockets Next, @QuarkusTest, any WebSocket server with async session registration
**Context:** WebSocket integration tests that: (1) connect a client, (2) trigger a server-side broadcast, (3) poll for the message. The race: the client handshake completes before the server-side `@OnOpen` registers the session, so the broadcast fires into an empty session set.

### Technique

Add a `waitForSession()` method to the session registry / broadcaster that spins until at least one session is registered:

```java
// In the broadcaster (production class — safe, no test dependency):
public void waitForSession(long maxWaitMs) throws InterruptedException {
    long deadline = System.currentTimeMillis() + maxWaitMs;
    while (sessions.isEmpty() && System.currentTimeMillis() < deadline) {
        Thread.sleep(10);
    }
}
```

Call it in the test's `connect()` helper immediately after the client handshake completes:

```java
// In the test:
@Inject GameStateBroadcaster broadcaster;

private WebSocket connect(LinkedBlockingQueue<String> queue) throws InterruptedException {
    URI ws = URI.create(wsUri.toString().replace("http://", "ws://"));
    WebSocket result = HttpClient.newHttpClient()
        .newWebSocketBuilder()
        .buildAsync(ws, new MessageCollector(queue))
        .join();
    broadcaster.waitForSession(2000);
    return result;
}
```

The session is typically registered within a few milliseconds. The cap (2s) handles slow CI runners.

### Why this is non-obvious

The reflex is to increase the poll timeout. But if the broadcast was dropped because `sessions.isEmpty()`, no amount of poll timeout helps — the message was never sent. The fix is not to wait longer for the message; it is to ensure the session is registered before the broadcast fires.

The second reflex is `Thread.sleep(N)` in `connect()`. That works but requires tuning `N` across different environments. The `waitForSession()` pattern spins until the condition is met — no arbitrary delay, no over-waiting.
