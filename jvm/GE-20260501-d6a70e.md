## Symptom

`NoSuchElement: No value present` at `.orElseThrow()` on a Panache static query. The entity was created via `tools.createChannel()` earlier in the same test. No error during creation. The channel definitely exists.

```
[ERROR] LedgerCaptureExampleTest.allNineMessageTypes_produceLedgerEntries:67 » NoSuchElement No value present
```

## Context

Test environment uses InMemory stores selected as CDI alternatives:
```properties
quarkus.arc.selected-alternatives=\
  io.example.InMemoryChannelStore,\
  io.example.InMemoryMessageStore
```

Test creates a channel through the MCP tools layer, then queries it via Panache:
```java
tools.createChannel("my-channel", "APPEND", null, null);  // → InMemoryChannelStore ✓
Channel ch = Channel.<Channel>find("name", "my-channel")  // → JPA entity query ✗
    .firstResultOptional()
    .orElseThrow();  // NoSuchElement — table is empty
```

## Root Cause

`Channel.<Channel>find()` is a Panache static method. Panache static methods are bytecode-enhanced to use an injected `EntityManager` directly — they do **not** go through CDI dispatch. `quarkus.arc.selected-alternatives` selects which `ChannelStore` bean is active, but Panache's static `find()` bypasses the store SPI entirely and queries the JPA entity table.

With InMemory stores active, `tools.createChannel()` routes to `InMemoryChannelStore` which stores the channel in a `ConcurrentHashMap`. Nothing is written to the H2 entity table. Panache's `find()` queries the empty table and returns `Optional.empty()`.

## Fix

Use the store/service layer to look up the entity — it correctly routes through whichever store implementation is active:

```java
// Bad — bypasses CDI, always queries JPA:
Channel ch = Channel.<Channel>find("name", "my-channel").firstResultOptional().orElseThrow();

// Good — routes through active ChannelStore (InMemory or JPA):
QhorusMcpToolsBase.ChannelDetail ch = tools.listChannels().stream()
    .filter(c -> "my-channel".equals(c.name()))
    .findFirst()
    .orElseThrow();
UUID channelId = ch.channelId();
```

Or inject and use `ChannelStore` or `ChannelService` directly if the test has CDI injection available.

## Why This Is Non-Obvious

The CDI `@Alternative` mechanism feels like it should intercept *all* access to a domain concept. Developers expect `quarkus.arc.selected-alternatives=InMemoryChannelStore` to mean "all channel operations go to InMemory." But Panache static methods are not CDI-dispatched — they go directly to JPA via bytecode manipulation. The store SPI pattern and Panache's static API coexist in the same codebase, and mixing them looks correct. The failure is silent during the `createChannel()` call (succeeds in InMemory) and only surfaces on the read (empty JPA table).

### Why this fix

Using `tools.listChannels()` or the store interface is the right fix because it keeps the test isolated from the storage implementation — the same test code works regardless of which store is active (InMemory for fast tests, JPA for integration tests). Accessing Panache statics directly couples the test to JPA.

### Alternatives considered

- Switch the example to JPA stores — defeats the purpose of the InMemory store pattern for fast CI tests
- Add a second write path that also persists to JPA — creates a dual-write bug surface
