### Context
An audit ledger entry records both successful and failed delivery of a notification.
Two distinct failure modes exist:
- **Observer-level failure**: error occurred before the notifier was ever called —
  connector config was never loaded, entity lookup threw, etc.
- **Connector delivery failure**: notifier was called, connector threw.

The initial design was to use `notifiedAt = null` to mean "no notification was attempted"
and `notifiedAt = clock.instant()` to mean "notification was attempted (successfully or not)".
But the entity has `@Column(name = "notified_at", nullable = false)` — a schema migration
would be needed to make this work.

### Technique
Instead of migrating the schema to make `notifiedAt` nullable, use a column that is
**already nullable** and has different natural values per failure mode:

- `connectorId = null` → observer-level failure (never reached the notifier)
- `connectorId = "slack"` + `delivered = false` → connector delivery failure (notifier ran)

The nullable column (`connectorId`) encodes the semantic distinction. No schema change,
no migration. An FDA auditor querying `WHERE connector_id IS NULL AND delivered = FALSE`
sees exactly the observer-level failures.

### Why this is non-obvious
The first instinct is to use the semantically "correct" field (`notifiedAt`) for the
distinction, then discover it's NOT NULL, and then reach for a schema migration. The
alternative — exploiting an existing nullable field with natural semantics — requires
stepping back from the schema to ask "what columns are already nullable, and do any
of them happen to encode the right distinction?" Often they do.
