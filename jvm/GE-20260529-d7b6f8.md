### Symptom
Implementing `TrustBootstrapSource` and configuring it as `@ApplicationScoped @Alternative @Priority(1)` produces no effect — `ActorTrustScore` rows are never seeded and `TrustScoreCache` loads empty on startup. No error is thrown.

### Context
Any `casehub-ledger` consumer that needs initial trust scores for agents before the first nightly `TrustScoreJob` run (e.g. tutorial deployments, fresh installs, test suites).

### Root cause
`TrustScoreJob.runComputation()` calls `bootstrapService.bootstrapIfNew(newActors)` where `newActors = byActor.keySet()`. `byActor` is built by grouping all `LedgerEntry` records by `actorId`. On a fresh deployment with zero ledger entries, `byActor` is empty, `newActors` is empty, and `bootstrapIfNew()` is never called. `TrustBootstrapSource` is designed for **cross-deployment federation** ("actor has history in another deployment") — not for seeding known values before any history exists.

### Fix
Use direct `ActorTrustScoreRepository.upsert()` in an `@ApplicationScoped` `StartupEvent` observer instead:

```java
@ApplicationScoped
public class AmlTrustScoreSeeder {
    @Inject ActorTrustScoreRepository trustRepo;
    @Inject TrustScoreCache trustScoreCache;

    @Transactional
    void onStart(@Observes @Priority(20) StartupEvent ev) {
        // Only seed if no CAPABILITY score exists yet (idempotent across restarts)
        if (trustRepo.findCapabilityScore("sar-drafting-agent-senior", "sar-drafting").isEmpty()) {
            trustRepo.upsert("sar-drafting-agent-senior", ScoreType.CAPABILITY,
                "sar-drafting", null, ActorType.SYSTEM,
                0.90, 10, 0, 9.0, 1.0, 9, 1, Instant.now());
        }
        // Force cache reload — seeder may run before or after TrustScoreCache @Startup
        trustScoreCache.hydrate();
    }
}
```

Use `@Priority(20)` so the seeder runs AFTER `DefaultCaseDefinitionRegistry` (priority=10) to avoid Vert.x initialization ordering conflicts.

### Why this is non-obvious
The SPI name (`TrustBootstrapSource`) and its Javadoc ("fetch prior trust data for an actor") strongly imply it handles initial seeding. The actual invocation path — buried inside `TrustScoreJob.runComputation()` conditional on existing ledger entries — is invisible to SPI implementors.
