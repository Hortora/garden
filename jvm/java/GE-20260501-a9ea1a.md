## SNAPSHOT dependency upgrade silently breaks interface contracts — compile error points at your class, not the dep

When a SNAPSHOT dependency adds abstract methods to an interface you implement,
the compile error appears on YOUR implementing class — not on the dependency or
the code you were actually changing. The fix you're working on compiles fine in
isolation; the break is invisible until a full compile runs.

### Symptom

```
[ERROR] Foo.java: FooRepository is not abstract and does not override abstract
method findByAttestorIdAndCapabilityTag(String, String) in LedgerEntryRepository
```

The error names your class and the new method — it says nothing about which
dependency added that method or when.

### Context

Occurs when all of the following are true:
1. You depend on a `*-SNAPSHOT` artifact
2. That artifact's maintainer adds abstract methods to an interface you implement
3. You build after pulling the new SNAPSHOT (e.g. after `mvn -U` or a fresh CI run)
4. You haven't changed your implementing class recently — so the error feels unrelated to your current work

### Root cause

SNAPSHOT versions are mutable. A new push to the SNAPSHOT can add interface
methods without a version bump. Maven pulls the latest SNAPSHOT transparently.
The compile error correctly identifies the implementer as incomplete — but gives
no indication that the interface changed upstream.

### Fix

Implement the new abstract methods. To diagnose which dep caused it:

```bash
# See what SNAPSHOT deps are in play
mvn dependency:list | grep SNAPSHOT

# Inspect the interface in the new JAR
jar tf ~/.m2/repository/<group>/<artifact>/<version>/<jar> | grep MyInterface
jar xf <jar> <path/to/MyInterface.class> && javap -p MyInterface.class
```

### Why this is non-obvious

The compile error looks like your code has a bug. There's no message saying
"the interface you implement gained a new method". Developers naturally look at
what they just changed — not at unchanged implementing classes that were fine
yesterday. The SNAPSHOT pull is silent.
