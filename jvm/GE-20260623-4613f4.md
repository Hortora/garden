### Symptom
Security backstop appears configured but has zero effect. Endpoints without `@RolesAllowed` remain publicly accessible despite the config property being set.

### Context
When wiring OIDC and `@RolesAllowed` into a Quarkus application, a catch-all deny policy is needed so that endpoints added without security annotations fail closed. The property `quarkus.security.jaxrs.deny-unannotated-endpoints=true` looks plausible because the `quarkus.security.jaxrs` namespace exists (it contains `deny-jax-rs` from `JaxRsSecurityConfig` in `quarkus-rest-common-deployment`).

### What was tried (didn't work)
```properties
# This property does not exist — Quarkus silently ignores it
quarkus.security.jaxrs.deny-unannotated-endpoints=true
```
Quarkus does not warn about unknown config properties at startup. The line is a complete no-op.

### Root cause
The actual property is `quarkus.security.deny-unannotated-members` — defined in `io.quarkus.security.deployment.SecurityConfig.denyUnannotatedMembers()` with `@ConfigMapping(prefix = "quarkus.security")`. There is no `jaxrs` sub-namespace for this property. The `quarkus.security.jaxrs` namespace exists but maps to `JaxRsSecurityConfig` in a different deployment module (`quarkus-rest-common-deployment`) and contains only `deny-jax-rs` and `default-roles-allowed`.

### Fix
```properties
quarkus.security.deny-unannotated-members=true
```
When this is `true`, `SecurityProcessor` uses `DenyUnannotatedPredicate` at build time to add `@DenyAll` at the class level to any CDI bean class where at least one method has a security annotation but the class itself does not. This operates inside the JAX-RS container after authentication — `@RolesAllowed` annotations remain fully functional.

### Why this is non-obvious
The `quarkus.security.jaxrs` namespace is real — it just contains different properties. A developer searching for "deny unannotated endpoints quarkus" will construct the plausible-but-wrong property name. Quarkus's silent ignoring of unknown properties means no error surfaces to correct the mistake.

### Why this fix
The property was verified by decompiling `SecurityConfig.class` from `quarkus-security-deployment-3.32.2.jar` and confirming the `@ConfigMapping` prefix and method name. No alternative exists — this is the only Quarkus mechanism for denying unannotated CDI bean methods.

### Alternatives considered
- `quarkus.http.auth.permission.*.policy=deny` — operates at the wrong layer (Vert.x HTTP filter, before JAX-RS), blocks ALL requests including authenticated ones with valid `@RolesAllowed`. See GE-20260623-941ade.
