### Symptom

JPA queries return empty results or `em.find()` returns null for entities that exist in the database. No error, no warning — Hibernate simply doesn't see the entity class.

### Context

Adding a JPA entity to an application that shares a foundation module's named persistence unit (e.g. Qhorus's `qhorus` PU). The entity is in a different package than the foundation module's entities.

### Root cause

Two independent configuration points must both be set:

1. `@PersistenceUnit("qhorus")` on the EntityManager injection — without it, the default PU is used, which has no datasource configured
2. `quarkus.hibernate-orm.qhorus.packages` must include the new entity's package — Hibernate only scans packages listed in this config for the named PU

Missing either one causes silent failure. The compound nature is what makes this non-obvious — setting only the annotation or only the packages config both appear correct in isolation but produce empty results.

### Fix

Both changes are required:

```java
@Inject
@PersistenceUnit("qhorus")
EntityManager em;
```

```properties
# application.properties — extend the existing packages list
quarkus.hibernate-orm.qhorus.packages=io.casehub.qhorus.runtime,io.casehub.ledger.runtime,io.casehub.drafthouse
```

### Why this is non-obvious

Each configuration point has its own diagnostic blind spot:
- Missing `@PersistenceUnit`: the default PU has no datasource, so Quarkus may fail at startup with an opaque CDI error OR silently use a different (empty) schema
- Missing packages config: the entity class exists on the classpath and the PU is correct, but Hibernate's entity scan doesn't find it — no warning, just empty results
- Both missing: you get the compound failure of both above
