## Context

A Quarkus extension needs to inject a named CDI resource (EntityManager, DataSource, etc.) where the name comes from extension config, not hardcoded. The consuming app may have only a named persistence unit, only the default, or both.

## Technique

1. **Define a custom CDI qualifier** for the extension's internal injection points:

```java
@Qualifier
@Retention(RetentionPolicy.RUNTIME)
@Target({ElementType.FIELD, ElementType.METHOD, ElementType.PARAMETER, ElementType.TYPE})
public @interface LedgerPersistenceUnit {}
```

2. **Write a producer** that reads config and selects the right bean via `Instance<T>` + `AnnotationLiteral`:

```java
@ApplicationScoped
public class LedgerEntityManagerProducer {

    @Inject LedgerConfig config;

    @Produces @LedgerPersistenceUnit
    public EntityManager produce(@Any Instance<EntityManager> instance) {
        String pu = config.datasource().orElse("").trim();
        if (pu.isBlank()) {
            return instance.select(Default.Literal.INSTANCE).get();
        }
        return instance.select(new PersistenceUnitLiteral(pu)).get();
    }

    // AnnotationLiteral for @PersistenceUnit — creates a runtime qualifier instance
    @SuppressWarnings("ClassExplicitlyAnnotation")
    private static final class PersistenceUnitLiteral
            extends AnnotationLiteral<PersistenceUnit> implements PersistenceUnit {
        private final String value;
        PersistenceUnitLiteral(String value) { this.value = value; }
        @Override public String value() { return value; }
    }
}
```

3. **Update all extension injection points** to use the custom qualifier:

```java
@Inject @LedgerPersistenceUnit EntityManager em;
```

4. **Consumers configure via properties** — no code changes needed:

```properties
# Only needed if the app has no default persistence unit
quarkus.ledger.datasource=mydb
```

## Why this is non-obvious

The instinctive fix is to add the consumer's qualifier directly to the extension's beans, which couples the extension to one consumer. The qualifier + producer indirection is the standard CDI pattern for this problem, but it's easy to overlook because `AnnotationLiteral` is rarely used outside framework code.

The custom qualifier also makes the injection IntelliJ-stable — IDEs treat standard `@Inject` + `@YourQualifier` correctly and never strip the import, unlike `@PersistenceUnit("hardcoded-name")` which IDE import optimizers sometimes remove as unused.

### Why this fix

The qualifier is owned by the extension and has no external name — consumers never see or configure it. All configurability flows through the extension's own config key. This keeps the extension's CDI wiring self-contained.
