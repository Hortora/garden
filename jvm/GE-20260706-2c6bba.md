### Symptom

Test code attempts to instantiate `JpaLedgerEntry` directly:

```java
var entry = new JpaLedgerEntry();  // compilation error: abstract; cannot be instantiated
```

### Context

`casehub-ledger` uses a two-tier JPA inheritance model:
- `LedgerEntry` (`@MappedSuperclass`) — defines all persistent fields
- `JpaLedgerEntry` (`@Entity`, abstract) — JOINED inheritance root
- Concrete subclasses: `PlainLedgerEntry`, `KeyRotationEntry`, `ActorIdentityBindingEntry`, `ErasureReceiptLedgerEntry`

Test code that creates generic audit entries needs the concrete `PlainLedgerEntry`, not the abstract base.

### Root cause

`JpaLedgerEntry` is the JPA inheritance root marked `abstract` — it exists to carry `@Entity`, `@Inheritance(JOINED)`, `@NamedQuery`, and `@EntityListeners` annotations. Domain-specific subclasses extend it for join-table fields. `PlainLedgerEntry` is the concrete subclass for domain-agnostic event writes (introduced with `OutcomeRecorder` in V1009).

### Fix

```java
import io.casehub.ledger.runtime.model.PlainLedgerEntry;

var entry = new PlainLedgerEntry();
entry.subjectId = UUID.randomUUID();
entry.entryType = LedgerEntryType.EVENT;
entry.actorId = "test-actor";
entry.actorType = ActorType.AGENT;
```

### Why this is non-obvious

The class hierarchy has four levels (`LedgerEntry` → `JpaLedgerEntry` → `PlainLedgerEntry`), and `JpaLedgerEntry` appears to be the "runtime" entity that tests should use. The name `PlainLedgerEntry` doesn't immediately suggest "this is what you instantiate in tests" — it sounds like a simplified variant rather than the default concrete type.
