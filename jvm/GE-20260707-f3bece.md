### Symptom
Consumer `@QuarkusTest` integration tests fail with `CaseInstance not found or wrong tenant` — 34 of 228 tests. The reactive repository and blocking repository maintain separate in-memory stores despite both being `@ApplicationScoped` singletons.

### Context
Dual-stack persistence pattern: blocking repo is canonical, reactive repo delegates to it via `@Inject`. Both have `@Alternative` test wrappers with `@Priority(1)` in an `engine-testing` module.

### What was tried (didn't work)
- Checked all event bus handlers for tenant context propagation — all correct
- Checked `CurrentPrincipal` thread safety — `@ApplicationScoped`, consistent
- Traced `save()` → `update()` flow — tenancyId parameters matched
- Assumed CDI would resolve `@Inject InMemoryCaseInstanceRepository delegate` to `TestCaseInstanceRepository extends InMemoryCaseInstanceRepository` via assignability

### Root cause
`InMemoryReactiveCaseInstanceRepository` declared its delegate as:
```java
@Inject InMemoryCaseInstanceRepository delegate;  // concrete class
```
`TestCaseInstanceRepository extends InMemoryCaseInstanceRepository` with `@Alternative @Priority(1)` was NOT substituted as the delegate. ARC resolved the injection point to a separate `InMemoryCaseInstanceRepository` instance — creating two stores with independent state.

### Fix
Inject by SPI interface instead of concrete class:
```java
@Inject CaseInstanceRepository delegate;  // SPI interface
```
ARC correctly resolves `@Alternative @Priority(1)` candidates when the injection point is the interface type. Applied to all 5 reactive in-memory repos.

### Why this is non-obvious
CDI spec says `@Alternative` beans are eligible for any injection point they satisfy by type. The expectation is that `TestCaseInstanceRepository extends InMemoryCaseInstanceRepository` satisfies `@Inject InMemoryCaseInstanceRepository`. The failure is silent — no CDI ambiguity error, no deployment failure. Two separate bean instances are created, and the symptom (tenant mismatch) gives no hint about the CDI root cause.

### Alternatives considered
- Making `TestCaseInstanceRepository` a `@Specializes` bean — rejected: Quarkus ARC has limited `@Specializes` support
- Using `@Produces` method to bridge — rejected: adds unnecessary complexity when interface injection works
