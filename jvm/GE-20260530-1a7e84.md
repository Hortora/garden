## Symptom

Attempting to build a `MessageDispatch` with `type = DONE`, `DECLINE`, `RESPONSE`, or `FAILURE` without supplying both `inReplyTo` (Long) and `correlationId` (String) throws `IllegalArgumentException` at build time:

```
IllegalArgumentException: DONE requires inReplyTo
IllegalArgumentException: DONE requires correlationId for commitment resolution
```

The constraint fires in `MessageDispatch.Builder.build()` during construction — it is not a runtime validation inside `MessageService.dispatch()`.

## Context

`COMMAND` and `QUERY` do NOT require `inReplyTo` or `correlationId` — those fields are optional for obligation-creating types. Only the obligation-resolving types enforce both. The asymmetry is intentional (resolving a commitment requires identifying which commitment to resolve) but is not obvious from the type hierarchy.

Triggered when trying to dispatch `DONE` from a delivery endpoint (e.g. `OpenClawDeliveryResource`) where the original COMMAND's Long message ID (`inReplyTo`) is not available in scope.

## Root Cause

`MessageDispatch.Builder.build()` enforces this in a switch statement:

```java
switch (type) {
    case DONE, DECLINE, FAILURE -> {
        if (inReplyTo == null) throw new IllegalArgumentException(type + " requires inReplyTo");
        if (correlationId == null) throw new IllegalArgumentException(type + " requires correlationId");
    }
    case RESPONSE -> {
        if (inReplyTo == null) throw new IllegalArgumentException("RESPONSE requires inReplyTo");
        if (correlationId == null) throw new IllegalArgumentException("RESPONSE requires correlationId");
    }
    // COMMAND, QUERY, EVENT, STATUS — no required reply fields
}
```

`inReplyTo` is the Long auto-generated primary key (`Message.id`) of the COMMAND/QUERY message being replied to — not the UUID `correlationId`. The `OutboundMessage` passed to `ChannelBackend.post()` does NOT expose this Long ID (it uses `UUID.randomUUID()` as its own ID in `MessageService.fanOut()`), so the Long is not recoverable at the delivery layer without a separate database lookup.

## Fix

Two paths:

**If the correlationId is known:** use `messageService.findAllByCorrelationId(correlationId)` to retrieve the original COMMAND/QUERY message, get its `id` (Long), and use that as `inReplyTo`. This requires storing the correlationId when the COMMAND was received.

**If the correlationId is not available:** dispatch `STATUS` instead (which requires no reply fields) as a Phase 1 workaround. STATUS acknowledges the obligation without resolving it. The Commitment remains open until a proper DONE/DECLINE can be sent. Watchdog handles stale commitments.

## Why This Is Non-Obvious

Developers familiar with request/reply patterns expect DONE to be a simple message type. The requirement that you must reference the original message's database primary key (a Long auto-ID, not the UUID correlationId) is surprising. The OutboundMessage received by ChannelBackend does not expose this Long ID, so it appears impossible to dispatch a proper DONE from a delivery webhook handler without adding a correlationId tracking mechanism.
