### Symptom

A new column is added to an entity and set correctly in the main code path. Normal tests pass. But a specific "overwrite" branch (e.g. a LAST_WRITE semantics path that mutates an existing row and returns early) never sets the new column — it retains its DB default or causes a NOT NULL constraint violation at runtime.

### Context

Occurs when:
1. A service method has a branch that mutates specific fields on an existing entity and returns early (before the main path's persist call)
2. A new field is added to the entity and set in the main code path
3. The developer adds the field to the main path but forgets the early-return branch

Classic example: a "LAST_WRITE" channel semantics path that overwrites content/type fields on a message and returns early, while the normal path constructs a new Message from scratch.

### Root cause

The early-return branch does only what it needs for its specific scenario and returns before the normal entity-construction path. When a new field is added, it's natural to add it where the entity is constructed from scratch (the main path). The early-return branch isn't in the same visual context, so it's missed.

### Fix

When adding a field to an entity:
1. Add it to the main construction path (obvious)
2. Search for every branch that also mutates the entity and returns early
3. Add the same field assignment to each such branch

```bash
# Find early-return mutation branches
grep -n "return new\|early return\|result\(.*\)" YourService.java
```

Then verify each found branch sets the new field.

### Why this is non-obvious

The test suite passes because normal-path tests cover the new field. Early-return branches often have fewer dedicated tests, or the test infrastructure doesn't check the new field specifically for that code path. The reviewer focuses on what was added — not what was missed in other branches. The bug only surfaces at runtime on the exact workflow that triggers the early-return path.
