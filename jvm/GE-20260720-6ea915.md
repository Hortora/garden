### Symptom
After adding `CbrConfig` to a `CaseDefinition` for CBR retrieval, completed investigations produce two CBR entries in `CbrCaseMemoryStore` — one from the app's domain-specific observer (correct features, domain outcome labels, ledger entry) and one from the platform's `CbrCaseRetainObserver` (generic JQ-extracted features, no ledger entry). No error, no warning.

### Context
`CbrConfig` on a `CaseDefinition` activates two independent mechanisms:
1. **Retrieve** — `CaseStartedEventHandler` calls `CbrRetrievalService.retrieve()` at case startup, injecting `RetrievedExperience` objects into `CaseContext` under `"cbrExperiences"`.
2. **Retain** — `CbrCaseRetainObserver` implements `CaseOutcomeObserver`, fires on `CaseOutcomeEvent` when `CbrConfig` is present, stores a `PlanCbrCase` via `CbrCaseMemoryStore.store()`.

There is no flag on `CbrConfig` to enable retrieve without retain.

### What was tried (didn't work)
- Configuring `CbrConfig` with only retrieve-relevant fields (topK, minSimilarity, features) — retain observer still fires because it only checks `getCbrConfig() != null`.

### Root cause
`CbrCaseRetainObserver.doRetain()` gates on `definition.getCbrConfig() != null`. Any app that sets `CbrConfig` — even purely for retrieval — activates the generic retain path. The observer extracts features via the same JQ expressions as the retrieval service, but stores them as a `PlanCbrCase` with generic outcome labelling, not the app's domain-specific `CbrCase` subclass.

### Fix
Exclude `CbrCaseRetainObserver` from CDI in both main and test `application.properties`:

```properties
quarkus.arc.exclude-types=...,io.casehub.engine.internal.memory.CbrCaseRetainObserver
```

Both files must be updated — `quarkus:build` validates CDI against the main classpath, so test-only exclusion causes the production build to fail.

### Why this is non-obvious
The coupling is implicit — `CbrConfig` looks like a retrieval configuration object (topK, minSimilarity, weights, featureExtractor). Nothing in the builder API or field names suggests it also controls retain behaviour. The retain observer lives in a different package (`engine.internal.memory`) from the retrieval service (`engine.internal.routing`), and the only link is the null check on `getCbrConfig()`. The duplicate entries degrade retrieval quality silently — cases are retrieved with inflated match counts and mixed feature quality — but no error is ever logged.
