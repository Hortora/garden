### Symptom
A self-healing race-recovery pattern catches `PersistenceException` from a unique
constraint violation and retries the lookup query within the same `@Transactional(REQUIRES_NEW)`
method. All tests pass on H2. The same code fails on PostgreSQL with
`current transaction is aborted, commands ignored until end of transaction block`.

### Context
Two threads race to create the same entity (e.g. channel by name). The first wins;
the second hits the unique constraint. The retry pattern:

```java
@Transactional(Transactional.TxType.REQUIRES_NEW)
public Result findOrCreate(Request req) {
    try {
        return store.put(entity);  // throws on constraint violation
    } catch (PersistenceException e) {
        // Retry the lookup — the winner's entity should exist now
        return store.findByName(req.name())  // FAILS on PostgreSQL
                .map(existing -> new Result(existing, false))
                .orElseThrow(() -> e);
    }
}
```

### Root cause
H2 and PostgreSQL handle constraint violations differently within a transaction:

- **H2:** The failed INSERT is rolled back statement-level. The transaction continues
  in a usable state. Subsequent queries execute normally.
- **PostgreSQL:** The failed INSERT marks the entire transaction as aborted. ALL
  subsequent commands in the same transaction are rejected with
  `current transaction is aborted, commands ignored until end of transaction block`.

The REQUIRES_NEW boundary creates a single transaction for the entire method. When the
`put()` fails, the `findByName()` retry runs inside the same aborted transaction.

### Fix
Catch `PersistenceException` **outside** the REQUIRES_NEW boundary — at the call site.
The failed REQUIRES_NEW transaction completes (rolls back), and the retry runs in a
fresh call:

```java
// Caller — outside the REQUIRES_NEW boundary
try {
    return service.findOrCreate(req);  // REQUIRES_NEW — commits or rolls back
} catch (PersistenceException e) {
    // Transaction rolled back — retry in a fresh call
    return store.findByName(req.name())
            .map(existing -> new Result(existing, false))
            .orElseThrow(() -> e);
}
```

This is the pattern used by `ConnectorChannelBackend.tryAutoCreate()` — it catches
the PersistenceException outside the REQUIRES_NEW method and retries with a fresh
lookup.

### Why this is non-obvious
The catch-and-retry logic is semantically correct — the constraint violation proves
the entity exists, so the lookup should find it. On H2 (the default test database
for Quarkus), it does. The failure only surfaces on PostgreSQL, which is typically
the production database. A developer sees green tests and ships code that fails in
production with a confusing "transaction aborted" error on what appears to be a
simple SELECT query.
