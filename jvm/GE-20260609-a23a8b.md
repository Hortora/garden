### Symptom

A background job processes items for tenant-A. Midway through, it calls a method that itself runs work in a tenant context (nested call). After the nested call returns, the outer job silently continues processing under the nested call's tenant identity — or worse, under no tenant at all.

### Context

`TenantContextRunner` programmatically activates a CDI request context, sets `TenantHolder.setTenancyId()`, runs a callback, then tears down the context. When the context is already active (e.g. called from within a REST request or another `TenantContextRunner` invocation), the runner must not terminate the outer context on exit.

### What was tried (didn't work)

Initial implementation checked `isActive()` before activating and skipped activation if already active. This handled the context lifecycle correctly but did not save/restore the tenant identity — the inner call's tenancyId persisted after return.

### Root cause

`Arc.container().requestContext().isActive()` correctly detects nesting, and `activate()`/`terminate()` are correctly skipped. But `TenantHolder.setTenancyId()` mutates the existing request-scoped bean — it doesn't create a new scope. Without explicit save/restore, the outer context's tenant identity is permanently overwritten.

### Fix

Save the previous tenancyId when the context is already active, restore it in `finally`:

```java
public <T> T runInTenantContext(String tenancyId, Supplier<T> work) {
    ManagedContext reqCtx = Arc.container().requestContext();
    boolean alreadyActive = reqCtx.isActive();

    // Save previous state ONLY if context already active
    String previousTenancyId = null;
    if (alreadyActive) {
        previousTenancyId = tenantHolder.getTenancyId();
    } else {
        reqCtx.activate();
    }

    try {
        tenantHolder.setTenancyId(tenancyId);
        return work.get();
    } finally {
        if (alreadyActive) {
            tenantHolder.setTenancyId(previousTenancyId); // restore
        } else {
            reqCtx.terminate();
        }
    }
}
```

### Why this is non-obvious

The CDI request context lifecycle (`activate`/`terminate`/`isActive`) is well-documented. The problem is that `@RequestScoped` beans are shared within an active context — they don't stack. Most developers assume that "checking isActive and skipping activation" is sufficient for safe nesting, but the scoped bean's mutable state is a separate concern that the context API doesn't protect.
