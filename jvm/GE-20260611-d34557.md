## JPA @PrePersist vs explicit enricher call — dual-persistence implementations silently diverge on hash coverage

### Symptom

Hash digests computed by the JPA path do not include data set by enrichers
(e.g., provenance supplements), while the in-memory path's digests do. The
divergence is invisible — tests don't cross-verify hashes between implementations.

### Context

Two `LedgerEntryRepository` implementations exist:

**JPA path** (`JpaLedgerEntryRepository.save()`):
```
1. stamp fields, allocate seqNum
2. digest = leafHash(entry)           ← BEFORE enrichers
3. em.persist() → @PrePersist → enricher pipeline runs
```

**In-memory path** (`InMemoryLedgerEntryRepository.save()`):
```
1. stamp fields, allocate seqNum
2. enricherPipeline.enrich(entry)     ← BEFORE digest
3. digest = leafHash(entry)
```

The JPA path computes the digest at step 2, then `em.persist()` triggers
`@PrePersist` which runs enrichers at step 3. The in-memory path runs
enrichers first (step 2), then computes the digest (step 3). Enricher-set
data (provenance supplements, traceId) is in the in-memory digest but not
the JPA digest.

### Root cause

The JPA `@PrePersist` lifecycle callback fires DURING `em.persist()`, not
before it. The save method computed the digest before calling `em.persist()`,
so the enricher pipeline — triggered by `@PrePersist` — ran after the digest
was already locked in. The in-memory path had no such constraint and correctly
ran enrichers before hashing.

### Fix

Move enrichment out of `@PrePersist` into an explicit call in `save()`, before
digest computation. Both paths then follow the same ordering:
enrich → hash → persist.

```java
// JPA save() — fixed ordering
enricherPipeline.enrich(entry);                    // explicit call
entry.digest = LedgerMerkleTree.leafHash(entry);   // hash AFTER enrichment
em.persist(entry);                                  // no enrichers in @PrePersist
```

Convert `@PrePersist` listener to a defensive check that throws if an entry
reaches `em.persist()` without a digest.

### Why this is non-obvious

Both implementations produce valid-looking hashes. The divergence is only
detectable by recomputing the hash from the persisted entry and comparing —
which no test did because each implementation was tested in isolation. The
in-memory path was accidentally correct.
