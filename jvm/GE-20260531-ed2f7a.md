## @Transactional @ObservesAsync try/catch: caught exception enables outer TX commit — REQUIRES_NEW fallback also commits, double-recording the event

### Stack
Quarkus 3.32.2, CDI 4.0, Narayana JTA

### Symptom
A `@Transactional @ObservesAsync` listener writes a success ledger entry inside a try block, then calls a REQUIRES_NEW fallback writer in the catch block. When `fireAsync()` throws *after* the success write, both the success entry and the REQUIRES_NEW failure entry commit. The audit trail has two contradictory entries for the same event — a ledger row recording success and a ledger row recording failure for the same subject.

### Context
```java
@Transactional
public void onEvent(@ObservesAsync SomeEvent event) {
    try {
        ledgerWriter.writeSuccessEntry(...);    // outer TX
        eventBus.fireAsync(downstreamEvent);    // throws
    } catch (Exception e) {
        ledgerWriter.writeFailureEntry(...);    // REQUIRES_NEW — also commits
    }
}
```

When `fireAsync()` throws:
1. Exception is caught — method returns normally
2. Narayana sees normal return → commits outer TX → `writeSuccessEntry` committed ✅
3. `writeFailureEntry` (REQUIRES_NEW) committed in its own independent TX ✅
4. Both entries are now in the DB — double-recording the event

### What was tried
Wrapping the entire body in try/catch seemed correct: "if anything throws, write a failure entry." The expectation was that catching the exception would prevent the outer TX from committing, leaving only the REQUIRES_NEW failure entry.

### Root cause
A caught exception does NOT roll back the outer `@Transactional` transaction. The `@Transactional` interceptor only rolls back when an exception *escapes* the method. If the catch block swallows the exception and the method returns normally, the interceptor commits. REQUIRES_NEW was designed to commit independently *even when the outer TX fails* — it is indifferent to whether the outer TX succeeds or fails. The combination produces two committed entries.

### Fix
Introduce a boolean guard set to `true` immediately after the outer-TX success write and checked before calling the REQUIRES_NEW fallback:

```java
boolean ledgerWritten = false;
try {
    ledgerWriter.writeSuccessEntry(...);
    ledgerWritten = true;            // flag: outer TX write succeeded
    eventBus.fireAsync(downstream);
} catch (Exception e) {
    if (!ledgerWritten) {
        // writeSuccessEntry failed — outer TX will roll back — write fallback
        ledgerWriter.writeFailureEntry(...);  // REQUIRES_NEW
    } else {
        // writeSuccessEntry committed — only fireAsync failed — no fallback needed
        LOG.errorf(e, "downstream event failed — ledger entry exists, no fallback");
    }
}
```

### Why this is non-obvious
The mental model is "exception caught → nothing commits." This holds for exceptions that *escape* the `@Transactional` boundary. But a caught exception is invisible to the interceptor — from its perspective, the method returned normally. REQUIRES_NEW was purpose-built to commit even when the outer TX fails; it therefore commits whether the outer TX commits or rolls back. The cross-product of "caught = outer commits" × "REQUIRES_NEW always commits" produces double-recording that has no visible error, no warning, and surfaces only as an unexpected extra row in the DB.

### Why this fix
The boolean flag is the minimal correct solution. An alternative — always re-throw after the fallback write — would prevent outer TX commit but breaks the `@ObservesAsync` contract (the dispatcher swallows the exception anyway, but the pattern becomes fragile). The flag makes the two cases explicit: "failure before success write" vs "failure after success write", and routes each to the correct outcome.
