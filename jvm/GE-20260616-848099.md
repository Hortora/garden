### What happens

`"(c)(1)(i)"` is a substring of `"(c)(1)(ii)"`. Any assertion using `.contains("(c)(1)(i)")` returns `true` for both strings — the Grade 5 test passes even if the Grade 5 supplement accidentally returns the Grade 3 planRef.

### Example

```java
// Grade 5 test — LOOKS correct but is ambiguous:
&& rsle.compliance().map(c -> c.planRef).orElse("").contains("(c)(1)(i)");
// "(c)(1)(ii)".contains("(c)(1)(i)") == true  ← the trap

// Grade 5 test — correct:
&& rsle.compliance().map(c -> c.planRef).orElse("").contains("(c)(1)(i)")
&& !rsle.compliance().map(c -> c.planRef).orElse("").contains("(c)(1)(ii)");
```

### Root cause

The shorter CFR subclause string `"(c)(1)(i)"` is a prefix of `"(c)(1)(ii)"`. Java's `String.contains()` finds it regardless of what follows. The test reads correctly but doesn't actually discriminate between the two regulatory paths.

### Fix

For any hierarchical string where one form is a prefix of another, pair the positive assertion with a negative one:
- Positive: `.contains("(c)(1)(i)")`
- Negative: `!.contains("(c)(1)(ii)")`

The same pattern applies to ISO standard codes, SemVer prefixes, and any identifier scheme where shorter forms are prefixes of longer ones.

### Why this is non-obvious

The assertion looks syntactically complete and obviously correct. There is no compiler warning or test framework hint. The failure mode only appears when someone accidentally swaps the two subclause strings — at which point the "wrong" test still passes, silently hiding the regression.
