## Three-tier Maven module structure — api/common/runtime

**Topic:** Multi-module Maven structure for Quarkus extensions
**Alternatives exist:** yes — flat single-module, two-tier api/runtime, monolithic runtime

### The convention

Three tiers, ordered lightest to heaviest:

**Tier 1 — `api/` (pure Java):** Interfaces, records, enums, POJOs. No Quarkus, no CDI, no JPA. Any Java app can consume this tier regardless of framework.

**Tier 2 — `common/` or `core/` (CDI, no JPA):** Shared helpers and CDI beans. No JPA entities, no Flyway migrations, no datasource dependency.

**Tier 3 — `runtime/`, `engine/`, `app/` (full Quarkus):** JPA entities, Flyway migrations, REST resources, Quarkus config — everything.

Dependency direction: Tier 3 → Tier 2 → Tier 1. Never reversed.

### Why this style

JPA entities force every consumer to configure a datasource. CDI annotations in a Tier 1 module force all consumers to have Quarkus. Separating the three tiers means lightweight consumers (other pure-Java libs, tests using in-memory repos) can depend only on `api/` with zero runtime overhead and no datasource requirement.

### Trade-offs

More modules to navigate. SPIs that genuinely need external SDK types (Vert.x, Hibernate Reactive) must live in a tier that already depends on that SDK — this sometimes surprises contributors who add Quarkus types to `api/` then wonder why consumers break.

### When not to use it

Single-module projects or projects where all consumers are full Quarkus apps and a datasource-in-tests is acceptable across the board.
