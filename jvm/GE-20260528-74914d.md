## Symptom

Build fails with an error from `ExecutionModelAnnotationsProcessor` when `@Blocking` is placed
on an `@ApplicationScoped` CDI bean method that returns `Uni<T>` or `Uni<Void>`.

## Context

Adding `@Blocking` to a bridge or wrapper class that wraps blocking delegate calls and exposes
them as `Uni<>`. The intent: tell Quarkus to dispatch the blocking code to a worker thread.

```java
@DefaultBean @ApplicationScoped
public class BlockingToReactiveBridge implements ReactiveCaseMemoryStore {

    @Inject CaseMemoryStore delegate;

    @Override @Blocking   // ‚Üê build error here
    public Uni<String> store(MemoryInput input) {
        return Uni.createFrom().item(() -> delegate.store(input));
    }
}
```

Build output (from `ExecutionModelAnnotationsProcessor`):
```
Build step io.quarkus.arc.processor.BeanProcessor#build threw an exception:
io.quarkus.deployment.exception.BuildException: Build failure
@Blocking, @NonBlocking and @RunOnVirtualThread are not supported on non-entrypoint methods
```

## Root Cause

Quarkus's `ExecutionModelAnnotationsProcessor` only recognises `@Blocking`, `@NonBlocking`,
and `@RunOnVirtualThread` on *framework entrypoints* ‚Äî JAX-RS / Quarkus REST resource methods,
reactive messaging `@Incoming` consumers, GraphQL resolvers, and similar. On a plain
`@ApplicationScoped` CDI bean method ‚Äî even one returning `Uni` ‚Äî the annotation is rejected
at build time with a hard error.

The annotation is not a general hint to the CDI container. It is a deployment-time instruction
to specific framework processors that know how to change the thread dispatch model for their
own entry points.

## Fix

Remove `@Blocking` from CDI bean methods. Thread dispatch is the responsibility of the
*entrypoint* (e.g. a `@Blocking`-annotated JAX-RS method) calling into the CDI bean.

For a bridge that wraps blocking code as `Uni`, the `Uni.createFrom().item(supplier)` pattern
already defers execution until subscription ‚Äî the subscriber's thread model determines where
the supplier runs. For void blocking calls, `Uni.createFrom().voidItem().invoke(action)` is
the idiomatic Mutiny pattern.

```java
@DefaultBean @ApplicationScoped
public class BlockingToReactiveBridge implements ReactiveCaseMemoryStore {

    @Inject CaseMemoryStore delegate;

    @Override  // no @Blocking ‚Äî entrypoint is responsible for thread dispatch
    public Uni<String> store(MemoryInput input) {
        return Uni.createFrom().item(() -> delegate.store(input));
    }

    @Override
    public Uni<Void> erase(EraseRequest request) {
        return Uni.createFrom().voidItem().invoke(() -> delegate.erase(request));
    }
}
```

If the CDI bean itself must guarantee worker-thread execution (not relying on the caller),
use `runSubscriptionOn(Infrastructure.getDefaultWorkerPool())` on the returned `Uni` instead.

## Why This Is Non-Obvious

The Quarkus documentation says to use `@Blocking` when a method blocks. Nothing in the
documentation limits this to framework entrypoints ‚Äî the restriction is only discoverable by
hitting the build error or reading the `ExecutionModelAnnotationsProcessor` source. Developers
familiar with `@Blocking @Tool` on MCP handlers or `@Blocking @Incoming` on messaging consumers
will naturally apply it to any CDI bean method they write that wraps blocking code.
