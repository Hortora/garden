## Symptom

After making a breaking API change to a class in module A (e.g. renaming a field, changing a method signature), the build of module A passes. The full `mvn install` then fails with compile errors in modules B and C — test files in those modules still reference the old API.

## Context

Multi-module Maven projects where downstream modules have test-scope or compile-scope dependencies on the module being changed. Common when changing a shared SPI, record shape, or utility class used across the project.

## Root cause

When making a breaking change, developers naturally update the test files in the module being modified. But other modules that consume the changed class also have test files referencing the old API. Module-scoped builds (`mvn test -pl module-a`) pass because they only compile module A's tests. The failure only surfaces during a full build that compiles all downstream modules.

## Fix

Before committing a breaking API change, grep all test source files across the entire project for the changed symbol:

```bash
# Find all test files referencing the old API
grep -r "OldClassName\|oldMethodName\|entityId()" \
  --include="*.java" \
  $(find . -path "*/src/test/*" -name "*.java" | xargs dirname | sort -u)

# Or more simply — grep all Java files
grep -r "query\.entityId()\|new OldClass(" --include="*.java" .
```

Update all references before running the full build.

## Why this is non-obvious

The changed module compiles and tests cleanly. The IDE may show errors in the full project view, but if working module-by-module it's easy to miss. The failure only appears at `mvn install` time when all modules are compiled together, which can feel disconnected from the change made earlier.

## Constraints

Applies to any breaking symbol change: field renames on records (Java records break positional constructor callers), method signature changes, class renames. Particularly common in shared SPI modules consumed by many application modules.
