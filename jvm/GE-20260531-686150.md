## Symptom

After adding a new member to an enum used as a dimension set (e.g. `EvalDimension.COMPLETENESS`), unrelated-looking tests begin throwing `IllegalStateException: Judge response missing dimension: COMPLETENESS`. The failing tests use stub ChatModels that return hard-coded JSON — they have no LLM connection. The error message implicates the LLM judge, not the enum change.

## Context

An LLM eval harness uses a Java enum (`EvalDimension`) to represent quality dimensions (SECOND_PERSON, CONCISENESS, FACTUAL_FIDELITY, TONE). The judge calls `parseResponse()` which iterates `EvalDimension.values()` and expects to find a JSON key for every enum member in the LLM response. Works correctly until a new format-specific dimension is added (e.g. COMPLETENESS for A2A_CARD format only).

## Root cause

`parseResponse()` iterates `EvalDimension.values()` — all members. Stub tests that return a 4-dimension JSON object now fail immediately because the new 5th member has no corresponding key. The stub JSON was valid before the enum extension; after it, it's invalid by construction.

The connection between the enum change (in a completely different file) and the parse failure (in test code using hardcoded strings) is non-obvious — especially since the error message says "Judge LLM response missing dimension" which points at LLM configuration, not enum values.

## Fix

Pass the applicable dimension set explicitly rather than iterating `values()`. Callers select the correct set for each format:

```java
// Before — breaks when enum is extended:
private ParsedResponse parseResponse(String json) throws JsonProcessingException {
    for (EvalDimension d : EvalDimension.values()) {  // ← breaks on new members
        JsonNode dimNode = root.get(d.name());
        if (dimNode == null) throw new IllegalStateException("missing dimension: " + d.name());
        ...
    }
}

// After — only iterates applicable dimensions:
private ParsedResponse parseResponse(String json, Set<EvalDimension> applicable)
        throws JsonProcessingException {
    for (EvalDimension d : applicable) {  // ← only what this format uses
        JsonNode dimNode = root.get(d.name());
        if (dimNode == null) throw new IllegalStateException("missing dimension: " + d.name());
        ...
    }
}
```

Extra keys in the response not in `applicable` are silently ignored — correct for format-specific schemas.

## Why this is non-obvious

The test failure manifests as a parse exception that implicates the LLM response. The actual cause is a compile-time-visible enum member that the evaluator has never been told to ignore. The bug only appears when the enum is extended with a format-specific member that existing test stubs don't include — which looks like "the LLM isn't configured" rather than "the enum grew".
