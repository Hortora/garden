GE-20260718-522120: [gotcha, 10/15] Quarkus LinkageError when injecting concrete CDI bean instead of port interface in multi-module @QuarkusTest — compile succeeds but test instantiation fails with loader constraint violation | quarkus, cdi, testing, classloader

### Symptom
`@QuarkusTest` class fails at test instantiation with:
```
LinkageError: loader constraint violation in interface itable initialization for class
com.example.app.MyServiceImpl: when selecting method '... MyPort.doWork(MyRequest)'
the class loader 'app' for super interface MyPort, and the class loader
QuarkusClassLoader of the selected method's class, MyServiceImpl have different
Class objects for the type MyRequest used in the signature
```

The test compiles without error. The failure occurs only at runtime when Quarkus creates the CDI proxy.

### Context
Multi-module Maven project where:
- Port interface (`MyPort`) is in module A (e.g. `review/`)
- Implementation (`MyServiceImpl implements MyPort`) is in module B (e.g. `app/`)
- Domain type (`MyRequest`) is in module C (e.g. `domain/`)
- Test class in module B injects `@Inject MyServiceImpl` (the concrete class)

Quarkus loads the interface via the `app` classloader and the implementation via `QuarkusClassLoader`. Both load `MyRequest` independently, producing different `Class` objects for the same type. The JVM's itable check catches this when building the CDI proxy.

### What was tried (didn't work)
- Checking imports — everything resolves correctly
- Rebuilding with `mvn clean install` — recompilation does not help
- The error message mentions "loader constraint violation" which suggests a JAR conflict, leading to fruitless dependency tree investigation

### Root cause
Quarkus test classloader isolation treats the concrete implementation class differently from its interface. When you inject the concrete type, the CDI proxy must satisfy both the concrete class's classloader and the interface's classloader — if they disagree on domain types in the method signature, the JVM rejects the proxy at link time.

### Fix
Always inject the **port interface**, never the concrete bean:

```java
// Wrong — causes LinkageError
@Inject MyServiceImpl service;

// Correct — CDI resolves to MyServiceImpl via the interface
@Inject MyPort service;
```

The CDI container still provides `MyServiceImpl` — the injection point just references the interface that both classloaders agree on.

### Why this is non-obvious
The code compiles cleanly. Java's type system sees no issue. The error only manifests when the JVM's classloader verification runs during CDI proxy construction at test startup — a phase most developers never inspect. The error message references "itable initialization" which is JVM internal terminology that doesn't point toward the fix.

### Alternatives considered
- Using `@InjectMock` — same issue, the mock still needs a proxy
- Adding module dependencies to align classloaders — would work but defeats the purpose of module separation
