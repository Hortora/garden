### What

`UniAsserter.assertFailedWith` has two overloads:

- `assertFailedWith(Supplier<Uni<T>>, Class<? extends Throwable>)` — checks exception type only
- `assertFailedWith(Supplier<Uni<T>>, Consumer<Throwable>)` — allows full assertion including message content

The type-class overload is the obvious one (it's what most examples show), but when the message is part of the contract being tested, it gives false confidence: the test passes as long as *some* NPE is thrown, not necessarily the one with the right message.

### When it matters

When testing null contracts where the message is part of the SPI documentation (e.g. `@throws NullPointerException if agentId is null`), the message verifies which guard fired. Without it, a ConcurrentHashMap NPE and an explicit `requireNonNull` NPE are indistinguishable to the test.

### Fix

```java
// Obvious form — checks type only
asserter.assertFailedWith(() -> registry.findById(null, "default"), NullPointerException.class);

// Consumer form — checks type AND message
asserter.assertFailedWith(
    () -> registry.findById(null, "default"),
    t -> assertThat(t).isInstanceOf(NullPointerException.class).hasMessageContaining("agentId"));
```

### Why this is non-obvious

The class-based overload appears in all Quarkus documentation and test examples. The `Consumer<Throwable>` overload is present in the interface (`io.quarkus.test.vertx.UniAsserter`) but not documented — only discoverable by decompiling the JAR (`javap -classpath quarkus-test-vertx-3.32.2.jar io.quarkus.test.vertx.UniAsserter`).
