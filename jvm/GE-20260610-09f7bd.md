### Symptom

`UnsupportedOperationException` at runtime when calling `computeIfAbsent()` on a `Map`
returned by a public API method. The call compiles cleanly — no warnings, no errors.

### Context

A class exposes `participants()` returning `Collections.unmodifiableMap(concurrentHashMap)`.
A caller does:

```java
session.participants().computeIfAbsent(role, r -> register(r));
```

This throws `UnsupportedOperationException` at runtime even though the underlying map is a
`ConcurrentHashMap`, which fully supports `computeIfAbsent`.

### Root Cause

`Collections.unmodifiableMap()` wraps the underlying map in a view that delegates read
operations transparently but intercepts **all mutation operations** — including
`computeIfAbsent`, `put`, `remove`, `merge`, `replaceAll`, etc. — and throws
`UnsupportedOperationException`. The wrapper fires before `ConcurrentHashMap` ever sees the
call.

The view IS a live view (reads reflect concurrent updates to the underlying map), but it is
mutation-blocking. These two properties are independent: "live" ≠ "mutable".

### Fix

Encapsulate the mutation inside the owning class. Expose a dedicated method that calls
`computeIfAbsent` on the internal `ConcurrentHashMap` directly. Callers use that method —
they never touch `participants()` for mutation:

```java
public class OwnerClass {
    private final ConcurrentHashMap<K, V> map = new ConcurrentHashMap<>();

    // Wrong — callers cannot safely mutate this view
    public Map<K, V> participants() {
        return Collections.unmodifiableMap(map);
    }

    // Right — mutation delegated through the owning class
    public V registerIfAbsent(K key, Supplier<V> supplier) {
        return map.computeIfAbsent(key, k -> supplier.get());
    }
}
```

The `registerIfAbsent` signature also makes the Javadoc contract explicit (atomicity
guarantee, retry-on-exception semantics from `ConcurrentHashMap.computeIfAbsent`) — the
unmodifiable view contract is maintained externally.

### Why This Is Non-Obvious

- `computeIfAbsent` is declared on `Map` — the view IS a `Map` — so the call compiles.
- The underlying type is `ConcurrentHashMap`, which is documented as thread-safe for
  `computeIfAbsent`. Nothing at the call site hints that the view, not the map, is the issue.
- The obvious defensive fix (`Map.copyOf()` instead of `unmodifiableMap`) does not solve the
  problem — it would produce a snapshot that also throws for `computeIfAbsent`, and it would
  miss concurrent updates. The correct fix is architectural: encapsulate mutation in the
  owning class.

### Why This Fix

The unmodifiable-view pattern exists precisely to prevent callers from bypassing invariants.
The right response is to keep that protection and provide a controlled mutation API
(`registerIfAbsent`) with explicit semantics. Removing the unmodifiable wrapper to allow
direct mutation would expose the live `ConcurrentHashMap` and let callers accidentally
store null values, bypass validation, or race with cleanup logic.
