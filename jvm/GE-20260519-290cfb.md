### Symptom
Uncertain whether @Alternative @Priority(1) in-memory implementations belong in
`persistence-memory/` or `testing/` module.

### Context
Casehub platform follows: if in-memory has a production use case, it belongs in
`persistence-memory/`; if it is genuinely test-only, it belongs in `testing/`.

### The decision guide
Ask: *"Could someone reasonably deploy with in-memory persistence in production?"*

- **Yes** (e.g. `WorkItemStore` — demos, local eval without a DB) → `persistence-memory/`
- **No** (e.g. `PreferenceProvider` — file-based `config/` module covers the no-DB case) → `testing/`

### Why this matters
A module named `testing/` signals test-only and may bundle test-framework artifacts.
`persistence-memory/` is a legitimate production deployment target (data is volatile;
restart means data loss, acceptable for evaluation). Conflating them forces consumers
to add a `testing/` module to their production classpath to get ephemeral behaviour.

### Additional pattern
If `key.parse()` or similar enables a string-source provider to return typed values,
an in-memory fixture may be unnecessary entirely — the mock provider handles it via
config strings without a separate module.

### Why this is non-obvious
The natural instinct is to put all `@Alternative @Priority(1)` beans in `testing/`.
The production-deployment use case for in-memory storage is easy to overlook.
