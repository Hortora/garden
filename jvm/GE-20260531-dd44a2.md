### Symptom

Endpoints at specific sub-paths return 404 even though the JAX-RS resource classes exist and other endpoints in the application work. Renaming the problematic resource class to use a specific `@Path` fixes the 404s.

### Context

A JAX-RS resource class uses `@Path("/")` as the class-level annotation. Method-level `@Path` annotations carry full paths (e.g. `@Path("/api/investigations/{caseId}/compliance-evidence")`). The application has other resource classes at specific paths like `@Path("/api/investigations")`.

### What was tried (didn't work)

- Verifying the resource class is CDI-discovered (`@ApplicationScoped`)
- Confirming the method-level `@Path` values are syntactically correct
- Checking that the endpoint URL matches exactly what the method-level `@Path` declares

### Root cause

`@Path("/")` makes the resource a root resource whose template matches every URI as a prefix. In RESTEasy Reactive's dispatch algorithm, `@Path("/")` is selected as a candidate for *all* requests including those with URI paths that match more-specific resource classes. When no method-level `@Path` on the root resource matches the request, RESTEasy Reactive returns 404 instead of continuing to evaluate more-specific candidates.

The JAX-RS specification requires selecting the most-specific matching resource class, but "matching" first means the class-level template must be a prefix of the request URI — and `"/"` is a prefix of everything. The tie-breaking rules then apply within the matched candidate set in ways that do not always favour more-specific classes when a root resource is in play.

### Fix

Use a fully-specified class-level `@Path` rather than `"/"`:

```java
// BAD — captures everything
@Path("/")
public class AmlLayer7Resource {
    @GET
    @Path("/api/investigations/{caseId}/compliance-evidence")
    public Response get(@PathParam("caseId") UUID caseId) { ... }
}

// GOOD — class-level path is the full endpoint path
@Path("/api/investigations/{caseId}/compliance-evidence")
public class AmlComplianceEvidenceResource {
    @GET
    public Response get(@PathParam("caseId") UUID caseId) { ... }
}
```

When two endpoints need completely different path prefixes, use two separate resource classes each with a specific class-level `@Path`:

```java
@Path("/api/investigations/{caseId}/compliance-evidence")
class AmlComplianceEvidenceResource { @GET public Response get(...) }

@Path("/api/actors/{actorId}/erasure")
class AmlGdprErasureResource { @POST public ErasureResult post(...) }
```

Package-private classes work for CDI discovery in Quarkus — Java's one-public-class-per-file rule means the second class must be package-private, which Quarkus CDI handles correctly.

### Why this is non-obvious

The JAX-RS specification describes a path selection algorithm that should favour more-specific paths. `@Path("/")` feels like a reasonable root path that method-level annotations extend. The actual 404 response looks like the endpoint URL is wrong, not like a routing conflict. The symptom only surfaces when more-specific resource classes exist in the same application.
