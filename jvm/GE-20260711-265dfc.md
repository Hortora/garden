### Symptom
CDI @ObservesAsync handlers produce inconsistent state when two events for the same
logical key are fired in quick succession (e.g. deregister followed by register). The
observer may process the registration before the deregistration, leaving stale entries.

### Context
Any CDI component that observes async events keyed by the same identity — e.g. a router
wiring/unwiring DataSources, a cache invalidating/repopulating entries.

### Root cause
CDI @ObservesAsync does not guarantee delivery order. When deregister() fires
DataSourceDeregistered and the subsequent register() fires DataSourceRegistered for the
same (path, tenancyId), observers may process them in either order.

### Fix
Carry the object instance (not just the key) in the CDI event record. Use identity
comparison (==) in observer handlers so each handler is convergent — correct regardless
of processing order.

**wireRoute (registration handler):**
1. Resolve the current DataSource from the registry by key
2. If registry returns empty → skip (already deregistered)
3. If an existing wired entry has the same key but different instance → replace
4. If same instance → skip (genuine duplicate)

**unwireRoute (deregistration handler):**
1. Find wired entry by key
2. If the wired entry's instance == event's instance → remove (this IS the deregistered one)
3. If different instance → skip (already replaced by a new registration)

Both handlers converge to the same correct state regardless of which runs first.

### Why this is non-obvious
The natural approach is key-based matching: unwire by (path, tenancyId), wire by
(path, tenancyId). This works when events arrive in order but silently corrupts state
when they don't — the unwire removes the newly-wired replacement entry. The identity
approach requires the event record to carry the actual object instance, which feels
unusual for an event payload.

### Alternatives considered
- **Key-based matching with sequence numbers:** adds counter complexity and still requires
  careful edge-case handling for concurrent deregister+register
- **Synchronizing on the key:** serializes all operations for a key, eliminates the
  ordering problem but adds contention
