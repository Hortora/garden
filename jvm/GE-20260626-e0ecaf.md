### Symptom
`NoSuchElementException: SRCFG00011: Could not expand value casehub.iot.homeassistant.url` at Quarkus startup — even though the corresponding `@ConfigMapping` method returns `Optional<String>`.

### Context
A Quarkus module defines a `@ConfigMapping` interface with `Optional<String> url()`. The module's `application.properties` binds a REST client URL via a property expression: `quarkus.rest-client."homeassistant".url=${casehub.iot.homeassistant.url}`. When the property is not set (e.g. the provider is disabled), the expectation is that `Optional.empty()` handles it gracefully.

### Root cause
SmallRye Config resolves `${...}` property expressions in `application.properties` at config startup — before `@ConfigMapping` interfaces are created, before CDI starts, before any bean lifecycle runs. The `Optional<String>` return type on the ConfigMapping method only affects the ConfigMapping API layer — it has no effect on the raw expression resolution in application.properties. These are two independent resolution layers.

### Fix
Remove property expressions from `application.properties`. Create REST clients programmatically via `RestClientBuilder` in `@PostConstruct` (or lazily), resolving the URL from the `@ConfigMapping` interface (which correctly returns `Optional.empty()` when unset).

```java
// BEFORE (broken):
// application.properties: quarkus.rest-client."homeassistant".url=${casehub.iot.homeassistant.url}
// Config: Optional<String> url();  ← does NOT prevent the expression failure

// AFTER (correct):
// application.properties: (no expression)
// Provider @PostConstruct:
String resolvedUrl = config.url().orElseGet(() -> discovery.resolve(timeout));
this.restClient = RestClientBuilder.newBuilder()
    .baseUri(URI.create(resolvedUrl))
    .build(MyRestClient.class);
```

### Why this is non-obvious
`Optional<String>` on the ConfigMapping naturally reads as "this property is optional — absent is fine." The fact that a separate resolution layer (property expressions in application.properties) evaluates first and fails hard is invisible unless you know SmallRye's resolution architecture.

### Why this fix
Programmatic REST client creation moves URL resolution to runtime — after ConfigMapping is available and Optional semantics are respected. A dummy default (`${prop:http://unused.local}`) would prevent the expression failure but bake a wrong URL into the REST client (base URLs are immutable after creation).

### Alternatives considered
- Dummy default in the expression (`${casehub.iot.homeassistant.url:http://unused.local}`) — prevents crash but the REST client gets the dummy URL permanently (Failure 2: `@RestClient` base URLs are immutable after config resolution)
- `@IfBuildProperty` to exclude the REST client bean — build-time only, prevents runtime reconfiguration in Docker images
