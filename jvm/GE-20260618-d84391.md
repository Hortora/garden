### Symptom

CDI `@Decorator` classes typically use field injection (`@Delegate @Any @Inject`), making them impossible to instantiate in pure Java unit tests without a CDI container. All decorator tests end up as `@QuarkusTest` integration tests — slow and heavyweight.

### Context

Pipeline decorators that wrap SPIs (e.g., a corrective retrieval decorator wrapping `CaseRetriever`) need thorough unit testing of their orchestration logic. Running these as `@QuarkusTest` is 10–50x slower than pure Java tests and requires container setup.

### Fix

Use constructor injection on the `@Decorator` class. CDI 4.1 supports `@Inject` constructors on decorators — the `@Delegate @Any` annotation moves to a constructor parameter:

```java
@Decorator
@Priority(100)
public class CorrectiveCaseRetriever implements CaseRetriever {

    private final CaseRetriever delegate;
    private final RelevanceEvaluator evaluator;
    private final Event<RetrievalQuality> qualityEvent;

    @Inject
    CorrectiveCaseRetriever(@Delegate @Any CaseRetriever delegate,
                            RelevanceEvaluator evaluator,
                            Event<RetrievalQuality> qualityEvent) {
        this.delegate = delegate;
        this.evaluator = evaluator;
        this.qualityEvent = qualityEvent;
    }
}
```

In unit tests, construct directly with stubs. The `Event<T>` interface can be stubbed with an anonymous implementation that captures fired values:

```java
private static Event<RetrievalQuality> capturingEvent(AtomicReference<RetrievalQuality> ref) {
    return new Event<>() {
        @Override public void fire(RetrievalQuality event) { ref.set(event); }
        @Override public CompletionStage<RetrievalQuality> fireAsync(RetrievalQuality event) {
            ref.set(event);
            return CompletableFuture.completedFuture(event);
        }
        // remaining methods return null — unused in tests
        @Override public <U extends RetrievalQuality> Event<U> select(Class<U> c, Annotation... a) { return null; }
        @Override public <U extends RetrievalQuality> Event<U> select(TypeLiteral<U> t, Annotation... a) { return null; }
        @Override public CompletionStage<RetrievalQuality> fireAsync(RetrievalQuality event, NotificationOptions o) { return fireAsync(event); }
    };
}
```

Test then constructs the decorator directly:
```java
var retriever = new CorrectiveCaseRetriever(
    inMemoryDelegate, inMemoryEvaluator, capturingEvent(qualityRef));
var results = retriever.retrieve("query", corpus, 10, null);
assertThat(qualityRef.get().evaluated()).isTrue();
```

### Why this is non-obvious

Most CDI decorator examples use field injection. The CDI spec's decorator chapter focuses on `@Delegate @Inject` as a field annotation. Constructor injection on decorators is less documented and developers default to the field injection pattern, locking their decorators into container-only testing. The `Event<T>` stub pattern is also not widely known — most developers would reach for Mockito rather than a capturing anonymous implementation.

### Alternatives considered

- **Field injection + Mockito** — works but requires reflection or `@InjectMocks`, and Mockito's `@Mock Event<T>` requires generic type token handling
- **Field injection + package-private setter** — test hack, not clean API
- **`@QuarkusTest` only** — correct but 10–50x slower per test; 16 unit tests at <1ms each vs seconds with container startup
