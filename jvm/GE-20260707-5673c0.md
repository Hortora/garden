### Symptom

All tests in a consumer module fail with:

```
java.lang.NoSuchMethodError: 'void io.casehub.api.spi.routing.AgentRoutingContext.<init>(java.util.UUID, java.lang.String, com.fasterxml.jackson.databind.JsonNode, java.lang.String)'
```

No compilation error in IntelliJ. No warning. Tests simply explode at runtime.

### Context

A Java record in a SNAPSHOT dependency (`AgentRoutingContext`) gained a 5th field (`List<RetrievedExperience> experiences`). The source file in the consumer's tree was updated (Maven resolves from the SNAPSHOT JAR), but existing test code still called the 4-arg canonical constructor.

IntelliJ resolves the record from the updated source file, which shows 5 fields — but the test code calling the 4-arg constructor doesn't trigger a compile error because IntelliJ's incremental compilation may not re-analyse unchanged test files against the new record shape.

Maven's `javac` compiles against the JAR, which has only the 5-arg constructor. The 4-arg call is a `NoSuchMethodError` at runtime.

### What was tried (didn't work)

1. Running individual tests in IntelliJ — same error (IntelliJ delegates to Maven for test execution in this project)
2. Checking for import errors — none visible

### Root cause

Java records have exactly one canonical constructor matching their component list. When a field is added, the old N-arg constructor ceases to exist — there is no backward-compatible overload. Unlike classes with explicit constructors, records cannot have both old and new constructors coexist.

The IDE-vs-JAR split means the source tree shows the new shape but test files referencing the old shape aren't flagged until runtime.

### Fix

Add the new field's default value as the additional constructor argument in every call site:

```java
// Before (4-arg — broken)
new AgentRoutingContext(UUID.randomUUID(), capability, NullNode.instance, "test-tenant");

// After (5-arg — fixed)
new AgentRoutingContext(UUID.randomUUID(), capability, NullNode.instance, "test-tenant", List.of());
```

The fix is mechanical but must be applied to every test file that constructs the record.

### Why this is non-obvious

The breakage is completely silent at the IDE level. A developer adding a field to a record in module A has no way to know that module B's tests will break — there's no compile error, no IDE warning, and the source file in B's tree correctly shows the new field. The failure only manifests when Maven runs `javac` against the dependency JAR, which is typically during `mvn test` — not during development.
