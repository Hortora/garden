### Symptom

`MongoXxxDocument.mongoCollection().findOneAndUpdate(filter, update, options)` fails to compile:

```
incompatible types: PanacheMongoEntityBase cannot be converted to MongoXxxDocument
```

### Context

Any Panache MongoDB entity that uses `mongoCollection()` as the escape hatch for operations Panache doesn't expose (atomic `findOneAndUpdate`, `findOneAndReplace`). The entity extends `PanacheMongoEntityBase` and has `@MongoEntity(collection = "xxx")`.

### What was tried (didn't work)

Direct assignment: `MongoXxxDocument result = MongoXxxDocument.mongoCollection().findOneAndUpdate(...)` ā€” compiler rejects it because `mongoCollection()` returns `MongoCollection<PanacheMongoEntityBase>` at compile time.

### Root cause

`PanacheMongoEntityBase.mongoCollection()` is declared as:

```java
@GenerateBridge
public static <T extends PanacheMongoEntityBase> MongoCollection<T> mongoCollection()
```

`@GenerateBridge` generates the correctly-typed override on each subclass during Quarkus augmentation (byte-code transformation after compilation). At compile time, the compiler only sees the base class declaration and infers `T = PanacheMongoEntityBase`. At runtime, the augmented method returns `MongoCollection<MongoXxxDocument>` ā€” but the compiler doesn't know this.

### Fix

Downcast the result of `findOneAndUpdate`:

```java
final MongoXxxDocument result = (MongoXxxDocument)
        MongoXxxDocument.mongoCollection()
                .findOneAndUpdate(filter, update, options);
```

The cast is safe ā€” Panache guarantees the runtime type. It produces no unchecked warning because it's a regular object downcast (superclass ā†’ subclass), not a generic type cast.

### Why this is non-obvious

The natural expectation is that calling `mongoCollection()` on a concrete subclass returns a collection typed to that subclass. This is how it works at runtime ā€” the `@GenerateBridge` mechanism ensures it. But Java's compilation model doesn't see augmentation-time code generation, so the generic type parameter erases to the base class bound. The disconnect between compile-time and runtime types is invisible until you try to assign the return value of a typed method like `findOneAndUpdate`.

Methods that don't depend on the generic parameter (like `createIndex`) work fine ā€” they accept `Bson` and return void. Only methods whose return type is `T` (like `findOneAndUpdate`) expose the covariance gap.
