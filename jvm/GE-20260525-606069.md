## Symptom

LLM or YAML parser receives malformed YAML — fields with `:`, `#`, `|`, `>`, or leading spaces get misinterpreted as YAML structure rather than string values.

## Context

Building a YAML string programmatically in Java (via StringBuilder) to pass to an LLM as structured input. Values come from user-supplied data (agent names, descriptions, disposition axes, situational context).

## Root Cause

YAML scalars containing `:` followed by a space, `#` (comment marker), `|`/`>` (block scalar indicators), or other special characters are parsed as YAML structure if unquoted. Raw string concatenation (`key: ` + value) produces invalid YAML silently — no error, just wrong semantics.

## Fix

Use single-quote YAML scalars. Single quotes handle all special characters; single quotes inside the value are doubled:

```java
private static void yamlField(StringBuilder sb, int indent, String key, String value) {
    if (value == null) return;
    sb.append(" ".repeat(indent))
      .append(key)
      .append(": '")
      .append(value.replace("'", "''"))
      .append("'\n");
}
```

Single-quote style is simpler than double-quote (no backslash escaping), and handles every problematic character including `:`, `#`, `|`, `>`, and leading/trailing whitespace.

## Why non-obvious

The invalid YAML is not rejected — it is silently misparse. An LLM receiving the malformed YAML will process it without complaint, so the bug is invisible until a value happens to contain a problematic character. Values like `"principled: always"` or situational contexts with colons are common real-world triggers.

### Alternatives considered
- Double-quote scalars — require backslash escaping (`\"`, `\\`, `\n`) which is more error-prone
- Block scalars (`|` or `>`) — verbose and change whitespace handling
- External YAML library — valid but heavyweight for simple key-value serialization
