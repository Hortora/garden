## Symptom

`HttpClient.newWebSocketBuilder().buildAsync(uri, listener).get(30, SECONDS)` hangs for the full timeout duration against a plain `ServerSocket`-based test server. No error is produced, no connection refused, no partial handshake visible — just an indefinite wait.

## Context

Implementing a WebSocket client using the JDK built-in API against a test server written as a raw `ServerSocket` with manual RFC 6455 handshake.

```java
// Hangs indefinitely:
HttpClient client = HttpClient.newHttpClient(); // defaults to HTTP/2 preference
webSocket = client.newWebSocketBuilder()
    .buildAsync(URI.create("ws://127.0.0.1:" + port + "/sc2api"), listener)
    .get(30, TimeUnit.SECONDS); // hits timeout
```

## Root Cause

`HttpClient.newHttpClient()` creates a client with `HttpClient.Version.HTTP_2` preference (the JDK default). The client attempts HTTP/2 upgrade negotiation before falling back to HTTP/1.1. A plain `ServerSocket` speaks neither ALPN nor HTTP/2 framing — it has no mechanism to respond to the negotiation, so the client waits indefinitely for a response that never comes. No exception is thrown; the `CompletableFuture` simply does not complete until the timeout.

## Fix

### Option 1 — Force HTTP/1.1 (still uses JDK HttpClient)
```java
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_1_1)
    .connectTimeout(Duration.ofSeconds(10))
    .build();
```
Works if the test server speaks proper HTTP/1.1 WebSocket upgrade. However, some additional incompatibilities between `HttpClient`'s WebSocket implementation and a minimal handshake server may persist.

### Option 2 — Raw Socket with manual RFC 6455 handshake (reliable)
```java
socket = new Socket("127.0.0.1", port);
socket.setTcpNoDelay(true);
performWebSocketHandshake(port); // send HTTP upgrade, read 101
startFrameReader();              // background virtual thread
```
The raw Socket has no HTTP/2 negotiation layer and connects immediately. Works reliably against any compliant WebSocket server including minimal test servers.

## Why Non-Obvious

The JDK documentation for `HttpClient.newHttpClient()` does not prominently note the HTTP/2 default or its effect on WebSocket connections to HTTP/1.1-only servers. The hang is silent — no exception, no log, no partial response — and occurs even after `HTTP_1_1` version is specified if the server doesn't respond to the negotiation correctly.
