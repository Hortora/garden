### Symptom

All `@TestSecurity` functional tests on paths configured with `quarkus.http.auth.permission.*.auth-mechanism=<scheme>` return HTTP 500 instead of the expected status codes. The suppressed exception is `OIDCException: OIDC Server is not available`.

### Context

When `quarkus.http.auth.permission.plugin.auth-mechanism=plugin-token` is set for a path, and a custom `HttpAuthenticationMechanism` with matching `getCredentialTransport().getAuthenticationScheme()` is registered.

### What was tried (didn't work)

1. Setting `@TestSecurity(user = "plugin", roles = {"openclaw-plugin"})` at class level — tests get 500
2. Setting `@TestSecurity` at method level — same result
3. Returning `Uni.createFrom().nullItem()` from the mechanism (hoping @TestSecurity intercepts first) — Quarkus still routes to the named mechanism

### Root cause

`quarkus.http.auth.permission.*.auth-mechanism=<scheme>` causes Quarkus to select the named mechanism for that path BEFORE `AbstractTestHttpAuthenticationMechanism` (at `Integer.MAX_VALUE` priority) can provide the test SecurityIdentity. The `auth-mechanism` config is a hard routing directive that bypasses the normal priority-based mechanism selection that `@TestSecurity` relies on.

### Fix

Don't use `auth-mechanism` config. Use a path guard in `authenticate()` and return `Uni.createFrom().nullItem()` from `getCredentialTransport()`:

```java
@Override
public Uni<SecurityIdentity> authenticate(RoutingContext context,
                                          IdentityProviderManager ipm) {
    if (!context.request().path().startsWith("/your/path/")) {
        return Uni.createFrom().nullItem();
    }
    // validate token...
}

@Override
public Uni<HttpCredentialTransport> getCredentialTransport(RoutingContext context) {
    return Uni.createFrom().nullItem();
}
```

Quarkus proactive auth (default) invokes the mechanism for all requests. The path guard provides isolation. `@RolesAllowed` on the resource handles authorization.

### Why this is non-obvious

The Javadoc for `auth-mechanism` only describes matching against `getCredentialTransport().getAuthenticationScheme()`. There is no documentation about how `auth-mechanism` interacts with `@TestSecurity`. The expectation (based on `@TestSecurity` using `Integer.MAX_VALUE` priority) is that test mechanisms always win — but `auth-mechanism` is a routing directive, not a priority override.

### Why this fix

Using a path guard + null credential transport is the only combination that: (a) isolates the mechanism to specific paths, (b) doesn't conflict with OIDC's Bearer transport, and (c) allows `@TestSecurity` to intercept. `auth-mechanism` config is the clean API but breaks testing.

### Alternatives considered

- `auth-mechanism=plugin-token` with `getCredentialTransport()` returning the matching scheme — breaks @TestSecurity (this entry)
- Returning `AuthenticationFailedException` instead of null for unhandled paths — triggers OIDC challenge flow → 500
- Custom `@TestProfile` to disable OIDC — would affect MCP endpoint tests that need OIDC
