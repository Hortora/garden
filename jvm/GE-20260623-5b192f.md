### Symptom
A new REST resource class added with no `@RolesAllowed` annotations anywhere is publicly accessible despite `quarkus.security.deny-unannotated-members=true` being set.

### Context
`quarkus.security.deny-unannotated-members=true` tells Quarkus's `SecurityProcessor` to add `@DenyAll` at the class level to CDI bean classes that pass `DenyUnannotatedPredicate`. This is meant as a safety net for forgotten security annotations.

### Root cause
`DenyUnannotatedPredicate.test(ClassInfo)` (verified from bytecode in `quarkus-security-deployment-3.32.2.jar`) returns `true` only when:
1. The class itself has **no** class-level security annotation, AND
2. At least one **method** in the class has a security annotation

A new resource class with zero security annotations anywhere returns `false` — the predicate does not select it, and its endpoints remain open.

### Fix
The property still provides valuable protection for the most common mistake: forgetting `@RolesAllowed` on a new method added to an existing resource class (which already has annotated methods). Document the scope limitation accurately and do not overstate the guarantee in specs or comments.

```properties
# Catches unannotated methods on classes that already have @RolesAllowed on other methods.
# Does NOT cover entirely new resource classes with zero annotations.
quarkus.security.deny-unannotated-members=true
```

### Why this is non-obvious
The property name `deny-unannotated-members` implies all unannotated members are denied. The actual scope is narrower — "unannotated members of partially-annotated classes." The distinction only matters when a new resource class is added, which is less frequent than adding methods to existing classes, making the gap hard to discover through testing.
