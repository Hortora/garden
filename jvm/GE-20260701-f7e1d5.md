### Symptom

`NullPointerException` in `BgeM3Embedder.normalizeRows()` during batch inference. The stack trace points to a null `float[]` element in the ColBERT `float[][]` array — but the model ran successfully and produced output. The NPE only appears in batch mode, not single-sentence inference.

### Context

When exporting BGE-M3 to ONNX with a custom PyTorch wrapper. BAAI's reference implementation (`modeling.py`) computes ColBERT on `last_hidden_state[:, 1:]` — all tokens except CLS. This is intentional in their Python pipeline where post-processing handles the offset. The problem surfaces when the ONNX model is consumed by a Java inference runtime that uses attention mask length for padding stripping.

### Root cause

`OnnxInferenceModel.runBatch()` strips padding from rank-3 outputs (ColBERT) using:
```java
int actualLen = 0;
for (long v : batchMask[i]) actualLen += (int) v;
float[][] stripped = Arrays.copyOf(rank3[i], actualLen);
```

`actualLen` = number of real tokens including CLS (from attention mask). If ColBERT output excludes CLS, the output has `seq_len - 1` positions but `actualLen` counts `seq_len` positions. `Arrays.copyOf(float[][], newLength)` with `newLength > array.length` pads with `null` elements (since `float[]` is a reference type). These null elements cause NPEs when `normalizeRows()` iterates over them.

Single-sentence inference via `run()` doesn't strip padding, so the off-by-one is harmless — the NPE only manifests in batch mode via `runBatch()`.

### Fix

Include CLS in the ColBERT computation — use `last_hidden_state` (all tokens), not `last_hidden_state[:, 1:]`:

```python
# BAAI reference (wrong for Java pipeline):
colbert = self.colbert_linear(last_hidden_state[:, 1:])

# Correct for OnnxInferenceModel consumption:
colbert = self.colbert_linear(last_hidden_state)
colbert = colbert * attention_mask[:, :, None].float()
colbert = torch.nn.functional.normalize(colbert, dim=-1)
colbert = colbert * attention_mask[:, :, None].float()  # re-zero after normalize
```

Including CLS is semantically safe — the CLS vector carries holistic semantic information and adds a valid signal to ColBERT MAX_SIM scoring.

### Why this is non-obvious

The BAAI reference implementation excludes CLS deliberately (CLS is used for dense embedding, excluding it from ColBERT avoids double-counting). The ONNX model produces valid output either way — the mismatch only manifests as a null-element NPE in Java's `Arrays.copyOf` during batch inference, not as a model error. The symptom (NPE in normalization) gives no indication that the root cause is in the ONNX export's token selection.
