## Symptom

Consumer CI fails with `package io.example.api.model does not exist` even though the class is in a transitive dependency that IS published to GitHub Packages. The producing artifact (e.g. `quarkus-qhorus`) lists the api module as a compile-scope dependency in its pom.xml. Local builds work.

## Context

In a split `runtime` + `api` Maven module setup:
- `casehub-qhorus` (runtime) declares `casehub-qhorus-api` as a compile dependency
- Downstream `claudony` depends only on `casehub-qhorus`
- In CI, claudony can't find types from `casehub-qhorus-api`

## Root Cause

GitHub Packages does not reliably resolve transitive SNAPSHOT dependencies. When Maven resolves `casehub-qhorus:0.2-SNAPSHOT`, it should transitively pull `casehub-qhorus-api:0.2-SNAPSHOT`. In practice, this resolution is unreliable — GitHub Packages' SNAPSHOT metadata handling drops or delays transitive resolution in a way that Maven Central does not.

## Fix

Explicitly declare the transitive API dependency in the consumer pom:

```xml
<!-- In claudony's pom.xml — explicit even though it's transitive -->
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-qhorus-api</artifactId>
  <version>0.2-SNAPSHOT</version>
</dependency>
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-ledger-api</artifactId>
  <version>0.2-SNAPSHOT</version>
</dependency>
```

## Why This Is Non-Obvious

Maven's dependency model guarantees transitive resolution, and this works reliably with Maven Central. GitHub Packages is documented as a Maven repository, so the same behaviour is expected. The failure only surfaces in CI (not local builds) because local `.m2` already has both the runtime and api jars from prior local installs. The symptom looks like a missing dependency declaration, not a SNAPSHOT resolution bug.
