### Context

You need to assert that a method emits a `WARNING` log record under specific conditions
(e.g. a truncated API response). The method uses `java.util.logging.Logger`.

### What developers typically do instead

- Add a Logback test appender dependency
- Use `@Spy` on the logger via Mockito
- Skip the assertion and rely on manual inspection

### Technique

Install a temporary `java.util.logging.Handler` on the target logger. Run the action in a
`try/finally` block to guarantee removal even if the action throws. Collect `LogRecord`
instances and assert on them after the call.

```java
private List<LogRecord> captureWarnings(final Runnable action) {
    final Logger logger = Logger.getLogger(MyClass.class.getName());
    final List<LogRecord> records = new ArrayList<>();
    final Handler handler = new Handler() {
        @Override public void publish(final LogRecord r) {
            if (r.getLevel() == Level.WARNING) records.add(r);
        }
        @Override public void flush() {}
        @Override public void close() {}
    };
    logger.addHandler(handler);
    try { action.run(); } finally { logger.removeHandler(handler); }
    return records;
}
```

```java
final List<LogRecord> warnings = captureWarnings(() -> client.listChannels("xoxb-test-token"));
assertThat(warnings).anyMatch(r -> r.getMessage().contains("truncated"));
```

### Why this is non-obvious

JUL's `Logger.addHandler()` is mutable — most Java developers don't realise loggers accept
runtime handler injection, or that the handler list persists until explicitly removed.
The approach requires zero test dependencies beyond JUL itself.

### Caveats

Works reliably in pure unit tests (no `@QuarkusTest`). In Quarkus test contexts, JBoss
LogManager replaces JUL's handler chain and `Level.WARN` (JBoss) ≠ `Level.WARNING` (JUL) —
object identity comparison fails silently. See GE-20260607-876d7e for that pitfall.
