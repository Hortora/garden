### Symptom
`ide_insert_member`, `ide_edit_member`, and `ide_build_project` hit the 300-second MCP tool timeout and abort. Meanwhile, `ide_index_status` returns immediately, and navigation tools (`ide_find_class`, `ide_find_references`, `ide_search_text`, `ide_file_structure`) remain responsive throughout.

### Context
Occurs during sustained editing sessions on large multi-module Maven projects. The split is consistent — every edit/build tool times out while every read/navigation tool works. Retrying after minutes sometimes resolves it; other times the condition persists for the remainder of the session.

### What was tried (didn't work)
- Calling `ide_sync_files` to force a refresh — also timed out
- Waiting and retrying `ide_build_project` — sometimes works after 5-10 minutes, sometimes doesn't
- Calling `ide_index_status` between attempts — always succeeds, giving a false signal that the server is healthy

### Root cause
Unknown. Likely candidates: GC pressure from accumulated PSI caches, background project reload triggered by file changes outside the IDE, or write-lock contention in the PSI tree. Edit tools require heavier PSI write locks than read-only navigation, which explains the asymmetric timeout.

### Fix
Fall back to the Edit tool for text-based code modifications while continuing to use MCP for semantic navigation. This preserves the benefits of IntelliJ's index (correct cross-file references, type resolution) while avoiding the unreliable edit path.

### Why this is non-obvious
The natural assumption when `ide_index_status` succeeds is that the MCP server is fully operational. The asymmetric failure — reads work, writes don't — is not documented and contradicts the "server is up or down" mental model. Without knowing this, you'll waste time retrying edit tools that won't recover.

### Alternatives considered
- Restarting IntelliJ mid-session: disruptive, loses project state, and the MCP connection may not re-establish cleanly
- Switching entirely to bash grep + Edit tool: loses semantic correctness for navigation (misses overloads, cross-file references)
- The hybrid approach (MCP for reads, Edit for writes) preserves the best of both
