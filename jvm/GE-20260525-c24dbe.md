## What

Java records automatically generate a `toString()` that includes every component in declaration order. This is deterministic: same component values always produce the same string. SHA-256 of `record.toString()` gives a stable, compact hash for cache invalidation without any custom serialization.

## How to apply

```java
private static String sha256(String input) {
    try {
        byte[] hash = MessageDigest.getInstance("SHA-256")
            .digest(input.getBytes(StandardCharsets.UTF_8));
        return HexFormat.of().formatHex(hash).substring(0, 16); // 64 bits, ample for cache keys
    } catch (NoSuchAlgorithmException e) {
        throw new IllegalStateException("SHA-256 not available", e);
    }
}

// Usage: render result carries hashes for cache comparison
var descriptorHash = sha256(descriptor.toString());
var contextHash    = sha256(context.toString());
```

If either hash changes between calls, the cached render is stale.

## Why non-obvious

Most developers reach for a custom serializer (Jackson, manual field concatenation) or a hashCode()-based approach. Records make the custom serializer unnecessary — the generated `toString()` is a stable canonical representation of all fields with no extra code. The `hashCode()` approach fails because it's not guaranteed stable across JVM restarts and is too collision-prone for cache keys.

### Alternatives considered
- Custom serializer (Jackson) — correct but adds dependency and boilerplate
- `Objects.hash(field1, field2, ...)` — not stable across JVM restarts, too collision-prone
- Manual field concatenation — equivalent to `toString()` but requires maintenance as fields change; records auto-include new fields
