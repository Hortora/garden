### Symptom
First `@QuarkusTest` class boots and runs successfully. All subsequent `@QuarkusTest` classes
that trigger an application restart fail with:
```
IllegalStateException: Already a default codec registered for class <EventClass>
```
Tests are reported as errors (not failures) and dependent tests are skipped — the failure
chain makes it look like a CDI or classpath problem.

### Context
A `@ApplicationScoped` bean registers custom Vert.x message codecs during `@Observes StartupEvent`:
```java
bus.registerDefaultCodec(MyEvent.class, new LocalOnlyCodec<>("MyEvent"));
```
This works on first application boot. But in `@QuarkusTest`, the Vert.x event bus instance
is shared across application restarts within the same JVM — the codec from the first boot
is still registered when the second boot calls `registerDefaultCodec()` again.

### Root cause
Vert.x `CodecManager.registerDefaultCodec()` throws `IllegalStateException` if a codec is
already registered for the given class. The Quarkus test framework reuses the Vert.x instance
across `@QuarkusTest` application lifecycle restarts — so `@Observes StartupEvent` fires
multiple times against the same event bus.

### Fix
Guard each registration with a try-catch:
```java
private <T> void registerIfAbsent(EventBus bus, Class<T> clazz, String name) {
    try {
        bus.registerDefaultCodec(clazz, new LocalOnlyCodec<>(name));
    } catch (IllegalStateException ignored) {
        // Already registered — Vert.x instance shared across @QuarkusTest restarts
    }
}
```

### Why this is non-obvious
The error message says "Already a default codec registered" which sounds like a duplicate
bean or classpath problem. The connection to `@QuarkusTest` lifecycle and Vert.x instance
reuse requires understanding that Quarkus intentionally shares the Vert.x instance across
test restarts for performance — this is not documented in the context of custom codec
registration.

### Alternatives considered
- **Unregister before registering:** Vert.x has `unregisterDefaultCodec(Class)` but calling
  it unconditionally before registration adds unnecessary overhead in production.
- **Static flag to track registration:** Works but is fragile across classloader boundaries
  in test contexts.
