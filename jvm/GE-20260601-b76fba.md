### Symptom

`QuarkusTestResourceLifecycleManager.start()` returns a map containing a build-time fixed property (e.g. `quarkus.flyway.locations`). Quarkus logs a warning:

```
quarkus.flyway.locations is set to 'classpath:db/work/migration' but it is build time
fixed to 'db/migration'. Did you change the property after building the application?
```

The test proceeds with the build-time value. The override is silently discarded.

### Context

Occurs when a `@QuarkusTestResource` tries to configure a Flyway migration path, datasource, or any other property marked as build-time fixed in Quarkus. The `start()` method runs **after** augmentation (at runtime startup), so build-time properties are already baked in and cannot be changed.

### Root cause

Quarkus separates configuration into build-time and runtime phases. Build-time properties are resolved during augmentation, before any test infrastructure starts. `QuarkusTestResourceLifecycleManager.start()` executes at runtime — too late to influence augmentation output.

### Fix

Set build-time fixed properties in `src/test/resources/application.properties`, which is read during augmentation:

```properties
# Correct — read at augmentation time
quarkus.flyway.locations=classpath:db/work/migration,classpath:db/ledger/migration
```

Do **not** return build-time properties from `start()`. Reserve `start()` for runtime-only properties (JDBC URLs, usernames, ports) that the test container provides dynamically.

### Why this is non-obvious

The warning message is easy to miss in verbose test output. There is no error — the test starts normally with the wrong configuration. The symptom (missing tables, schema errors) appears later and points away from the config override as the cause.
