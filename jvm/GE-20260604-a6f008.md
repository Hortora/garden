## Symptom

When writing ARC42STORIES.MD §9.4 Key files sections for an optional feature module (e.g. `casehub-work-queues`), a `find` for the expected type returns the class in the *core* module instead:

```bash
find . -name "LabelVocabulary.java"
# Returns: runtime/src/main/java/io/casehub/work/runtime/model/LabelVocabulary.java
# Expected: queues/src/main/java/io/casehub/work/queues/model/LabelVocabulary.java
```

The class exists — it just lives in the wrong module relative to the feature it serves.

## Context

In a multi-module Quarkus extension where optional modules depend on the core runtime, types that are *used by* the optional feature but are also needed by core functionality tend to live in the core module. The optional module depends on core, not vice versa — so the types must be in core for the dependency direction to work.

In casehub-work:
- `LabelVocabulary`, `LabelDefinition`, `WorkItemLabel` (used by the queues label system) live in `runtime/src/main/java/io/casehub/work/runtime/model/`
- `casehub-work-queues` module depends on `casehub-work` (runtime) — not the other way round
- `FilterChain` in queues is an inverse index structure (`filterId → Set<workItemId>`), not a cascade-delete entity from `QueueView` as the name might suggest
- JQ evaluation goes through `JqConditionEvaluator` (which injects `JQEvaluator` from `casehub-platform-expression`), not a class literally named `JQEvaluator`

## What was tried

Template-based layer entries assumed types used by queues (like `LabelVocabulary`) live in the queues module because the module is named `casehub-work-queues`. This produced incorrect Key files paths.

## Root cause

Module naming implies **feature ownership** (queues-related functionality lives here) but not **type ownership** (the domain types themselves). When an optional module depends on the core module, shared domain types must live in core to avoid circular dependencies — the optional module imports them from core, not the other way around.

## Fix

Before writing any §9.4 Key files section for an optional module, verify each class with `find`:

```bash
find /path/to/project -name "TypeName.java" | grep "src/main"
```

The class may exist in a different module than the one being documented. The correct path is wherever `find` returns the file — not where feature naming suggests it should be.

Also verify structural assumptions:
- Is `FilterChain` a cascade entity or an index structure? → read the entity source
- Is `JqEvaluator` the correct class name? → check the import statement in the evaluator class

See also: GE-20260601-b0eabf (class names that don't exist in production at all).
