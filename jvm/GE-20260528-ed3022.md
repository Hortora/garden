### Symptom
Tests pass when `quarkus.flyway.locations` includes `classpath:db/work/migration` explicitly. Tests fail with "Table X not found" when relying on a `FlywayConfigurationCustomizer` bean to add `classpath:db/work/migration` at JVM startup — even though the customizer's `customize()` method runs and adds the location string to `FluentConfiguration`.

### Context
A Quarkus extension ships its migrations at a repo-scoped path (`classpath:db/work/migration`) instead of the default `classpath:db/migration`. The extension provides a `FlywayConfigurationCustomizer @ApplicationScoped` bean that adds the path at JVM startup, expecting Flyway to scan it and apply migrations. The extension tests — which do NOT have `quarkus.flyway.locations` set — run with the customizer active (CDI discovers the bean), but the migrations are never applied.

### Root cause
Quarkus Flyway pre-registers migration file lists at **build time** in `FlywayProcessor.build()`, which scans `quarkus.flyway.locations` (or the default `classpath:db/migration`) at augmentation time and records the found files via `FlywayRecorder`. At JVM runtime, `QuarkusPathLocationScanner` (and `QuarkusFlywayResourceProvider`) serve migrations exclusively from that pre-registered list. `FlywayConfigurationCustomizer.customize()` runs at startup and CAN add location strings to `FluentConfiguration`, but `QuarkusPathLocationScanner.getResources(Location)` has no pre-registered files for runtime-added locations and returns an empty result. The Flyway scanner finds zero migrations at the runtime-added path and applies nothing.

### Fix
Consumers must configure the migration path explicitly in build-time (or test) `application.properties`:

```properties
quarkus.flyway.locations=classpath:db/work/migration
```

This ensures `FlywayProcessor.build()` pre-registers the files during augmentation, making them available to `QuarkusPathLocationScanner` at runtime. The explicit configuration applies to both JVM and native image modes.

`FlywayConfigurationCustomizer` IS still useful:
1. For standard (non-Quarkus) Flyway embedders — it adds the location at runtime as expected
2. Register the bean as unremovable via `AdditionalBeanBuildItem.setUnremovable()` to prevent Quarkus unused-bean pruning from removing it (see sibling gotcha on CDI pruning)

### Why this is non-obvious
`FlywayConfigurationCustomizer` is a documented Quarkus SPI. The `customize()` method runs, the location string is added to `FluentConfiguration`, and no error or warning appears. The symptom — missing tables — points to migrations not running, not to a scanner-level silent skip. The Quarkus documentation does not mention that `QuarkusPathLocationScanner` is pre-populated from build-time config only.

### Why this fix
Explicit `quarkus.flyway.locations` is the only reliable mechanism for Quarkus consumers. The `FlywayConfigurationCustomizer` approach is architecturally correct for runtime Flyway but bypassed by Quarkus's build-time scanner substitution. There is no public build-time SPI for third-party extensions to inject additional scan locations into `FlywayProcessor.build()`.

### Alternatives considered
- `RunTimeConfigDefaultBuildItem` to set a default `quarkus.flyway.locations` — this would replace the consumer's configured locations, potentially breaking existing config
- Building a custom `FlywayProcessor` via extension SPI — no public hook exists in Quarkus 3.32.2
- `NativeImageResourcePatternsBuildItem` — registers SQL bytes in the native binary but does NOT populate the build-time file registry; necessary but insufficient
