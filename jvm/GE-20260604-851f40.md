## Context

Designing a Java REST client for Mem0 OSS based on the official docs at mem0.ai.
The docs describe a cloud API (`POST /v1/memories/`, `app_id` for multi-tenancy, etc.).
When the client hits the self-hosted OSS server, five things go wrong in ways that are
only discoverable from source code.

## What developers do instead

Follow the cloud API docs at mem0.ai, which describe `/v1/` prefixed routes, `app_id`
as a multi-tenant isolation field, and request-body-based bulk DELETE.

## The Five Source-Verified Deviations

All findings sourced from `server/main.py` and `mem0/memory/main.py` in the Mem0 OSS repo.

### 1. No `/v1/` URL prefix

OSS routes: `POST /memories`, `GET /memories`, `POST /search`, `DELETE /memories/{id}`.
Cloud docs describe `/v1/memories/`, `/v1/memories/search/` etc.
Every client path that includes `/v1/` will 404 against OSS.

### 2. `app_id` does not exist in OSS

The `MemoryCreate` Pydantic schema in `server/main.py` accepts:
`user_id`, `agent_id`, `run_id`, `metadata`, `infer`, `memory_type`, `prompt`.
No `app_id` field. The word `app_id` does not appear anywhere in the OSS source.
`app_id` is a cloud-only multi-tenancy feature.

Tenant isolation in OSS must be handled application-side (e.g. compound
`user_id = "{tenantId}::{entityId}"`).

### 3. `infer` is a first-class OSS field

`MemoryCreate` includes `infer` as an explicit field. Setting `"infer": false` stores
the message text verbatim as a single Mem0 memory without LLM extraction — this is
fully supported in OSS, not cloud-only as the docs imply.

### 4. GET `/memories` is unbounded (no `limit` param)

```python
@app.get("/memories")
def get_all_memories(user_id=None, agent_id=None, run_id=None):
    ...
```

The only accepted query params are filter dimensions. No `limit`, `top_k`, or pagination.
With any filter provided, it returns ALL matching records. Passing a `limit` param is
silently ignored.

For bounded results, use `POST /search` which accepts `top_k`.

### 5. Bulk DELETE uses query params, not a request body

```python
@app.delete("/memories")
def delete_all_memories(user_id=None, agent_id=None, run_id=None):
    ...
```

No request body. Filters are query params: `DELETE /memories?user_id=X&agent_id=Y`.
At least one filter dimension is required — HTTP 400 if all are null.
Omitting a dimension deletes across all values of that dimension (e.g. no `agent_id`
= delete all agent_ids for the given `user_id`).

## Why this is non-obvious

The docs at mem0.ai document the cloud API. There is no separate OSS API reference.
A developer building against OSS will read the cloud docs, discover the mismatches only
at runtime (silent 404 for wrong paths, field not accepted for `app_id`, etc.), and have
no way to find the correct surface without reading `server/main.py` directly.

### Why this fix

The fix is to read the OSS source rather than the cloud docs. The deviations above
are the canonical reference for OSS client implementation.
