## Symptom

A `@QuarkusTest` method annotated with `@Transactional` creates test fixtures via store `put()`, then calls a cross-tenant store method. The cross-tenant store returns zero results despite data being persisted moments earlier.

## Context

Cross-tenant stores use `@Transactional(REQUIRES_NEW)` to scope their `SET LOCAL ROLE` to an independent transaction. The test method's `@Transactional` wraps everything in one transaction — but `REQUIRES_NEW` suspends it and starts a new one. The new transaction cannot see uncommitted data from the suspended test transaction.

## Root cause

JTA `REQUIRES_NEW` creates a truly independent transaction with standard ACID isolation. The test's uncommitted inserts are invisible to the new transaction. This is correct JTA behaviour — not a bug. But when tests expect single-transaction semantics, it silently returns empty results.

## Fix

Persist test data in committed transactions before calling the `REQUIRES_NEW` store:

```java
@Test
void crossTenantStore_seesAllTenants() {
    // Commit data in separate transactions
    inTx(() -> tenantStore.put(createItem("tenant-a")));
    inTx(() -> tenantStore.put(createItem("tenant-b")));

    // Now the REQUIRES_NEW transaction can see committed data
    var results = crossTenantStore.findAll();
    assertThat(results).hasSize(2);
}

@Transactional
<T> T inTx(Supplier<T> s) { return s.get(); }
```

Remove `@Transactional` from the test method itself — let the helper manage transaction boundaries.

## Why this is non-obvious

The test passes on the same codebase without `REQUIRES_NEW` (when the store method shares the test's transaction). Adding `REQUIRES_NEW` to a store for RLS scoping silently breaks existing tests — no error, no warning, just empty results that cause assertion failures on count/existence checks.
