### Symptom
`@QuarkusTest` fails at startup with:
```
TestEngine with ID 'junit-jupiter' encountered a critical issue during test discovery:
ClassSelector [className = 'com.example.MyTest'] resolution failed
```
The JVM crashes before any test runs. No CDI error is shown — the error appears to be a classloading problem.

### Context
Building a CaseHub harness application (`app/`) that depends on both `casehub-engine` (runtime) and `casehub-ledger` (runtime) as Maven dependencies. Adding `casehub-engine-testing` as a test dependency provides in-memory implementations for engine repositories (`CaseInstanceRepository`, etc.) but not for the ledger SPI.

### Root cause
`casehub-ledger` runtime provides a Panache-based `ReactiveLedgerEntryRepository` that requires a real JPA datasource. `casehub-engine-persistence-memory` provides in-memory implementations for engine entity repositories but does not provide an implementation of `ReactiveLedgerEntryRepository`. When Quarkus cannot resolve the ledger SPI, the CDI container fails to initialise, causing the entire JVM to crash — surfacing as the misleading `ClassSelector resolution failed` error rather than a CDI error.

### Fix
Add a package-private `@ApplicationScoped` stub implementing `ReactiveLedgerEntryRepository` in `app/src/test/java/`:

```java
@ApplicationScoped
class InMemoryLedgerEntryRepository implements ReactiveLedgerEntryRepository {
    @Override
    public Uni<LedgerEntry> save(LedgerEntry entry) {
        return Uni.createFrom().item(entry);
    }
    // All other methods: return Uni.createFrom().item(emptyResult)
}
```

Do NOT use `@DefaultBean` — the real Panache implementation uses `@DefaultBean` which yields automatically. The stub must be a plain `@ApplicationScoped` to take priority in the test classpath.

### Why this is non-obvious
1. The error message (`ClassSelector resolution failed`) points to classloading, not CDI — debuggers look in the wrong place.
2. `casehub-engine-testing` appears to provide the test infrastructure needed, but it covers engine entity repos only; the ledger SPI gap is undocumented.
3. The fix requires knowing the `ReactiveLedgerEntryRepository` interface's full method signature, which spans 15+ methods — discoverable only by reading the ledger API source.
