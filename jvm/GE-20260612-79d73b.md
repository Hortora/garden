## Technique

Set `quarkus.langchain4j.ollama.chat-model.format=json` (or pass `format: "json"` in the Ollama API request) to force structurally valid JSON output at the token-sampling level, bypassing the model's instruction-following behaviour entirely.

```properties
quarkus.langchain4j.ollama.chat-model.format=json
```

This is distinct from prompt engineering. A model instruction saying "Return ONLY raw JSON" is a soft directive — small models (3B, 4B) routinely ignore it and produce preamble ("Here is the JSON:", "Here's my evaluation:") or natural language responses with no `{` at all. `format=json` is a hard constraint: the Ollama runtime biases token selection to produce valid JSON structure regardless of the model's tendency to ignore the instruction.

## Context

Useful in any eval harness or structured extraction pipeline using Ollama. The constraint is applied by Ollama's inference engine, not by the model itself — it works even when the model has no explicit training for JSON output. All four models tested (llama3.2:3b, gemma3:4b, llama3.1:8b, qwen3.6:35b) produced parseable JSON with `format=json`; without it, the 3B model failed on roughly 30–50% of calls with non-JSON preamble.

## Why This Is Non-Obvious

The obvious approach is to improve the instruction — "Return ONLY raw JSON, no preamble, no markdown". This works reliably with large instruction-tuned models (7B+) but fails unpredictably with smaller models. Developers typically iterate on prompt wording rather than recognising that the constraint needs to operate at the sampling layer. The `format=json` parameter is documented in Ollama's API reference but not prominently surfaced in the Quarkus extension docs.
