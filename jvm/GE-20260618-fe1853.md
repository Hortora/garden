### Symptom

`score >= correctThreshold` returns `false` when `score` is exactly `0.7f` and `correctThreshold` is `0.7` (double literal). No error, no warning — the comparison silently produces the wrong result. Threshold boundary values are never matched.

### Context

Any code that compares a `float` score (from ONNX model output, cross-encoder scoring, etc.) against a `double` threshold (from configuration, `@WithDefault("0.7")`). Common in ML inference pipelines where model output is `float[]` but configuration values are `double`.

### What was tried (didn't work)

Direct comparison `score >= correctThreshold` — compiles, runs, and silently fails at boundaries. The test `scoreAtCorrectThresholdReturnsCorrect()` with `score = 0.7f` and `threshold = 0.7` fails.

### Root cause

Java's binary numeric promotion (JLS §5.6.2) widens `float` to `double` before comparison. IEEE 754 single-precision cannot represent `0.7` exactly — `0.7f` is stored as `0.699999988079071...` in binary. When promoted to `double`, this becomes `0.699999988079071044921875` — strictly less than `0.7` as `double`.

The promotion is invisible in source code. `score >= correctThreshold` looks correct and compiles without warning.

### Fix

Cast the double threshold to float before comparison:

```java
private RelevanceGrade gradeFromScore(float score) {
    if (score >= (float) correctThreshold) return RelevanceGrade.CORRECT;
    if (score <= (float) incorrectThreshold) return RelevanceGrade.INCORRECT;
    return RelevanceGrade.AMBIGUOUS;
}
```

This ensures both operands are `float` — no promotion, no precision loss. The threshold loses the same precision as the score, so boundary comparisons work correctly.

### Why this is non-obvious

The code `score >= 0.7` looks perfectly correct. Java's type promotion is implicit and silent. The bug only manifests at exact boundary values — off-boundary comparisons work fine, so most tests pass. A developer would need to know that `0.7f != 0.7` at the bit level to anticipate this.

### Alternatives considered

- Using `BigDecimal` for comparison — correct but overkill for ML score thresholds where approximate equality is acceptable
- Using epsilon-based comparison (`Math.abs(score - threshold) < epsilon`) — more complex and the epsilon choice is arbitrary
- Declaring thresholds as `float` in config — works but SmallRye Config `@WithDefault` parses to double by default
