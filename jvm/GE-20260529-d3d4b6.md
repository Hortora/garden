### Symptom

Writing `MessageDispatch.builder(channelId, sender, type, content)` following the casehub-qhorus PLATFORM.md deep-dive causes a compilation error — no such static factory method exists on `MessageDispatch`.

### Context

First-time casehub-qhorus integration. Developer reads PLATFORM.md (casehub-parent) for the `MessageDispatch` API. The docs say:

> "Builds via `MessageDispatch.builder(channelId, sender, type, content)`"

### Root cause

The PLATFORM.md deep-dive is wrong. `MessageDispatch$Builder` is a standard Java builder with a no-arg constructor. All fields are set via chained setter methods — the four "required" fields (`channelId`, `sender`, `type`, `content`) have no special positional treatment at the API level. The builder validates required fields at `build()` time, not at construction time.

### Fix

Use the actual no-arg builder:

```java
messageService.dispatch(MessageDispatch.builder()
        .channelId(channel.id)         // UUID
        .sender(ORCHESTRATOR)          // String
        .type(MessageType.COMMAND)     // MessageType enum
        .content(pr.repo() + "#" + pr.prNumber())  // String, nullable
        .correlationId(correlationId)  // String
        .target(agent.capability())    // String (optional — sets obligor in Commitment)
        .actorType(ActorType.SYSTEM)   // ActorType — REQUIRED (see GE-20260529-e32a4d)
        .build());
```

Note: `actorType` is also required at `build()` time — a separate non-obvious constraint documented in GE-20260529-e32a4d.

### Why this is non-obvious

The PLATFORM.md authoritatively documents a specific method signature that does not exist. A developer without IDE autocomplete following the docs will write the wrong call immediately. The compilation error points to `MessageDispatch.builder` not having a matching overload — it does not say "the docs are wrong."
