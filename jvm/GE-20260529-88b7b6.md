## Symptom

Calling `ChannelService.create()` twice for the same channel name causes a JPA `EntityExistsException` or database unique constraint violation.

## Root Cause

`ChannelService.create()` is a straightforward `@Transactional` JPA `persist()` — it always inserts. There is no "create if not exists" semantics. The CaseChannelProvider SPI contract (`openChannel()`) requires idempotency: calling it more than once with the same caseId+purpose must return a usable channel without throwing.

## Fix

Use `findByName()` first, create only if not found:

```java
Channel channel = channelService.findByName(channelName)
    .orElseGet(() -> channelService.create(channelName, description, semantic,
            null, null, null, null, null, allowedTypes));
```

This is safe in the engine's calling pattern because `openChannel()` is called synchronously per binding — concurrent calls for the same channel are not expected. If concurrent access is a concern, wrap in a retry-on-constraint-violation catch.

## Why Non-Obvious

The method name `create()` doesn't signal non-idempotency. Claudony's `ClaudonyReactiveCaseChannelProvider` uses a `ConcurrentHashMap` memoization cache to avoid double-creation — which is also a workaround for the non-idempotency. A developer who reads the Claudony implementation might copy the cache approach without understanding that the simpler `findByName + create` pattern is equally valid.
