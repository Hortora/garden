### Symptom

Tenant-A queries a report endpoint. Tenant-B queries the same endpoint moments later and receives tenant-A's data. No error, no warning — the cache silently returns the wrong tenant's results.

### Context

A multi-tenant Quarkus application uses `@CacheResult(cacheName = "reports")` on service methods that query tenant-scoped data. The cache key is derived from method parameters (e.g. report type, date range) but does not include the caller's tenancyId.

### What was tried (didn't work)

Initially assumed the problem was in the query layer — the store-level tenant filtering was correct. The cached result pre-empted the query entirely.

### Root cause

`@CacheResult` generates cache keys from method parameters only. When two tenants call the same method with the same parameters, the second call gets a cache hit from the first tenant's result. The cache has no awareness of the request-scoped `CurrentPrincipal.tenancyId()`.

### Fix

Remove `@CacheResult` from any method that returns tenant-scoped data. If caching is required, build a tenant-aware `CacheKeyGenerator` that includes `tenancyId` in the key:

```java
@ApplicationScoped
public class TenantAwareCacheKeyGenerator implements CacheKeyGenerator {
    @Inject CurrentPrincipal principal;

    @Override
    public Object generate(Method method, Object... params) {
        return new CompositeCacheKey(principal.tenancyId(), params);
    }
}
```

Then annotate: `@CacheResult(cacheName = "reports", keyGenerator = TenantAwareCacheKeyGenerator.class)`.

The simpler fix — just remove `@CacheResult` — is appropriate when the cache hit rate doesn't justify the complexity.

### Why this is non-obvious

`@CacheResult` is a transparent annotation — developers add it for performance without thinking about what constitutes the cache key. In a single-tenant app this is fine. In a multi-tenant app, the default key generator creates a cross-tenant data leak with zero visible symptoms.

### Why this fix

Removing `@CacheResult` is preferred over the custom key generator because it eliminates the attack surface entirely. A tenant-aware key generator still requires every cached method to use it consistently — one missed annotation parameter and the leak returns.

### Alternatives considered

- **Cache per tenant** (separate cache instances): high memory overhead, complex eviction
- **Include tenancyId as explicit method parameter**: pollutes APIs, easy to forget
- **Remove caching entirely**: chosen as the pragmatic fix; re-add selectively with tenant-aware keys if profiling shows a need
