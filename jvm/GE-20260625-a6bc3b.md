### Symptom
When a `ReactiveWorkerProvisioner` supports multiple capabilities (e.g. `agent`, `code-reviewer`, `researcher`), command resolution using the `capabilities` parameter in `provision(Set<String> capabilities, ProvisionContext context)` is non-deterministic — `Set` iteration order decides which capability matches first.

### Context
The engine's `CaseContextChangedEventHandler.tryProvision()` (line ~580) calls:
```java
reactiveWorkerProvisioner.getCapabilities()
    .flatMap(caps -> reactiveWorkerProvisioner.provision(caps, provisionContext))
```
It passes ALL capabilities from `getCapabilities()`, not the specific capability being provisioned. Meanwhile, `ProvisionContext.taskType()` carries the correct specific capability name (set from `capability.name()` at line ~572).

The SPI javadoc says the parameter is "required capability set" — but the caller treats it as "all capabilities this provisioner supports."

### What was tried (didn't work)
Claudony's `WorkerCommandResolver.resolve(Set<String> requiredCapabilities)` iterated the set and returned the first command match. With a single configured capability this appeared correct. With multiple, the first match depended on `Set` iteration order.

### Root cause
The engine passes the full provisioner capability set to `provision()`, not the specific capability being provisioned. The SPI parameter name (`requiredCapabilities`) contradicts the caller's actual semantics. ARC42STORIES.MD documented the intent as `resolve(context.capability())` (singular) but the implementation accepted `Set<String>`.

### Fix
Use `context.taskType()` for all per-agent lookups — it carries the specific capability name. The `capabilities` parameter in `provision()` is vestigial for Claudony.

```java
// Before (fragile — Set iteration order)
String command = resolver.resolve(capabilities);

// After (correct — specific capability)
ClaudonyProviderConfig config = providerConfigSource.forAgent(roleName);
// where roleName = context.taskType()
```

### Why this is non-obvious
The SPI parameter is named `requiredCapabilities` and documented as "required capability set." An implementer naturally assumes this is the specific set relevant to this provisioning request. The disconnect between the javadoc contract and the caller's actual behaviour is invisible until multiple capabilities are configured — at which point the wrong command is silently selected.

### Why this fix
Using `context.taskType()` instead of the capabilities parameter aligns with the engine's intent (the specific capability being provisioned) rather than the SPI's misleading parameter name. This also enables unified per-agent config lookup — command, model, tools, etc. all keyed by the same identifier.

### Alternatives considered
- Fix the engine to pass only the specific capability → correct but would require an engine API change and all provisioner implementations to update
- Keep using capabilities but sort the set → fragile, papers over the real issue
