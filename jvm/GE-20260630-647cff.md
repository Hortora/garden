### Symptom

BM25 keyword search over code-domain text fails to distinguish exact identifier matches from coincidental word matches. Searching for `ChatModel` ranks documents containing unrelated "chat" and "model" equally to documents about the `ChatModel` Java interface.

### Context

Standard BM25 tokenizers (word-boundary split + lowercase) produce `chat` and `model` from `ChatModel`. A document about "a chat room model" matches identically to one about `dev.langchain4j.model.chat.ChatModel`. The compound identifier is destroyed.

### Fix

Produce both the compound form AND the components during tokenization:

```
ChatModel → [chatmodel, chat, model]
DefaultBean → [defaultbean, default, bean]
HTTPClient → [httpclient, http, client]
```

The compound token `chatmodel` is rare (high IDF) — it appears only in documents that actually discuss `ChatModel`. The component tokens `chat` and `model` are common (low IDF). BM25's IDF weighting naturally ranks documents matching the rare compound higher than documents matching only the common components.

```java
static List<String> tokenize(String text) {
    List<String> tokens = new ArrayList<>();
    String[] segments = text.split("[^a-zA-Z0-9]+");
    for (String segment : segments) {
        if (segment.isEmpty()) continue;
        List<String> parts = splitCamelCase(segment);
        for (String part : parts) tokens.add(part.toLowerCase());
        if (parts.size() > 1) tokens.add(segment.toLowerCase());
    }
    return tokens;
}
```

### Why this is non-obvious

The standard approach to CamelCase tokenization is to split into components only — `ChatModel → [chat, model]`. Keeping the compound alongside the components seems redundant until you realize IDF does the ranking work: the compound is always rarer than its components, so BM25 automatically favors exact matches without any special scoring logic. The insight is that the tokenizer controls ranking quality through IDF statistics, not through scoring formulas.
