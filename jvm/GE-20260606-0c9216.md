## Symptom

Integration test with `@Transactional` on the test method calls a service that fires a CDI event. An `@Observes(during = TransactionPhase.AFTER_SUCCESS)` observer should produce side effects (e.g. writing trust scores to a repository). But assertions on those side effects fail — the observer hasn't run yet. No error, no warning.

## Context

Testing `IncrementalTrustUpdateObserver` which uses `@Observes(during = AFTER_SUCCESS)` + `@Transactional(REQUIRES_NEW)` to recompute trust scores when an attestation is saved. The test calls `repo.saveAttestation()` (which fires `AttestationRecordedEvent`), then asserts trust scores exist. The test method is annotated `@Transactional`.

## What was tried

1. `@Transactional` on the test method + direct assertion after `saveAttestation()` — observer never fires during the test, assertions fail
2. `@TestTransaction` — same problem, transaction boundary is the test method

## Root cause

`@Transactional` on a `@QuarkusTest` method wraps the entire test method in one JTA transaction. CDI `AFTER_SUCCESS` observers are queued during the transaction and delivered synchronously only when that transaction commits — which happens after the test method returns. Every assertion within the method runs before the observer fires. The observer does eventually fire (during container cleanup), but that's too late for the assertions.

This is the mirror of GE-20260501-884024 (no transaction → observer skipped silently). Here the transaction exists but doesn't commit until after your assertions run.

## Fix

Use `QuarkusTransaction.requiringNew()` for programmatic transaction control. Each `requiringNew()` block commits independently, triggering AFTER_SUCCESS observers before the next line of test code executes.

```java
// ❌ Wrong — observer fires after the test method returns
@Test
@Transactional
void attestation_triggersRecomputation() {
    seedEvent(actorId);
    repo.saveAttestation(att);  // fires AttestationRecordedEvent
    // AFTER_SUCCESS observer hasn't fired yet — transaction not committed
    assertThat(trustRepo.findByActorId(actorId)).isPresent();  // FAILS
}

// ✅ Correct — each requiringNew() block commits independently
@Test
void attestation_triggersRecomputation() {
    QuarkusTransaction.requiringNew().run(() -> seedEvent(actorId));
    QuarkusTransaction.requiringNew().run(() -> repo.saveAttestation(att));
    // AFTER_SUCCESS observer has fired — transaction committed
    QuarkusTransaction.requiringNew().run(() ->
        assertThat(trustRepo.findByActorId(actorId)).isPresent());  // PASSES
}
```

## Why this is non-obvious

The `@Transactional` annotation on test methods is the standard Quarkus pattern for tests that need database access. Nothing in the annotation or its documentation warns that CDI transactional observers are deferred. The test compiles, runs, and completes without error — the only signal is failed assertions on data that "should" exist. A developer unfamiliar with CDI's transactional observer delivery semantics would debug the observer code, not the test's transaction boundary.

**See also:** [GE-20260501-884024](../quarkus/GE-20260501-884024.md) — the complement: AFTER_SUCCESS observer doesn't fire when *no* transaction is active. This entry: doesn't fire *during the test* because the transaction hasn't committed yet.
