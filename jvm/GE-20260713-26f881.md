### Symptom
Topic-filtered projections return all messages in JPA-backed stores but correctly filter in in-memory stores. No error, no warning — the results are silently wrong.

### Context
`MessageQuery` is a value object with fields like `channelId`, `sender`, `topic`, etc. Two consumers translate it to queries:
- `MessageQuery.matches(Message)` — in-memory predicate, used by `InMemoryMessageStore`
- `MessageQueryJpql.from(MessageQuery)` — JPQL WHERE clause builder, used by `JpaMessageStore`

When a new field is added to `MessageQuery` (e.g. `topic()`), `matches()` gets the filter automatically (it checks all fields), but `MessageQueryJpql` must be manually updated with the corresponding `AND` clause.

### Root cause
`MessageQueryJpql.from()` is a hand-written JPQL builder that must be kept in sync with `MessageQuery` fields manually. There is no compiler check, no runtime assertion, and no test that verifies field coverage parity between the two paths. A missing predicate silently degrades to "no filter" — the WHERE clause simply omits the condition.

Both `from(MessageQuery)` (tenant-unaware) and `from(MessageQuery, String)` (tenant-scoped) variants must be updated together — they are duplicated code blocks.

### Fix
Add the missing predicate to both `from()` variants in `MessageQueryJpql`:

```java
if (q.topic() != null) {
    where.append(" AND LOWER(topic) = LOWER(?").append(idx++).append(")");
    params.add(q.topic());
}
```

Case-insensitive matching (`LOWER()`) is required to match `MessageQuery.matches()` semantics which uses `equalsIgnoreCase`.

### Why this is non-obvious
The in-memory store tests all pass — `MessageQuery.matches()` handles every field. The bug only manifests through JPA-backed stores, which are typically not exercised in unit tests. A developer adding a field to `MessageQuery` has no signal that `MessageQueryJpql` needs updating — the code compiles, tests pass, and the query silently returns too many results.

### Alternatives considered
- **Criteria API**: Would derive predicates from field metadata, eliminating manual sync. Rejected — heavier abstraction for a record with ~12 fields; the manual builder is readable and fast.
- **Reflection-based parity test**: Could assert that every non-null field in a `MessageQuery` produces a predicate in the JPQL output. Not yet implemented but would catch future regressions.
