### Context
A `@ApplicationScoped` CDI bean fires `Event<InboundMessage>` to broadcast received messages. In `@QuarkusTest` this is fully testable via CDI observers. In pure JUnit 5 unit tests (no CDI context), `jakarta.enterprise.event.Event<T>` is an interface with ~6 abstract methods — implementing it as a test double is verbose and fragile.

### Technique
Expose a package-private constructor that accepts `Consumer<T>` instead of `Event<T>`. The CDI constructor (annotated `@Inject`) delegates to it via `this(connectors, messageEvent::fire)`:

```java
@ApplicationScoped
public class InboundConnectorService {

    private final Consumer<InboundMessage> eventBus;

    @Inject
    InboundConnectorService(@All List<InboundConnector> connectors,
                            Event<InboundMessage> messageEvent) {
        this(connectors, messageEvent::fire);   // ← delegate
    }

    // Test constructor — no CDI annotation
    InboundConnectorService(List<InboundConnector> connectors,
                            Consumer<InboundMessage> eventBus) {
        this.eventBus = eventBus;
        // build registry from connectors
    }

    public void receive(InboundMessage message) {
        eventBus.accept(message);   // fires in prod, records in tests
    }
}
```

Unit tests call the package-private constructor directly:
```java
List<InboundMessage> captured = new ArrayList<>();
InboundConnectorService service =
    new InboundConnectorService(List.of(), captured::add);
service.receive(msg);
assertThat(captured).containsExactly(msg);
```

### Why this is non-obvious
The obvious alternatives — implementing `Event<T>` as a mock or using Mockito — require either verbose boilerplate (6+ interface methods) or a mocking framework dependency in production code. `Consumer<T>` already exists in `java.util.function` and is exactly the shape needed: one method, one type parameter. The `this(...)` delegation pattern keeps the CDI constructor thin and avoids duplicating setup logic between the two constructors.

### Alternatives considered
- `@QuarkusComponentTest` — can test a single CDI bean in isolation, but requires Quarkus augmentation and is slower than a plain JUnit 5 test
- Mockito `mock(Event.class)` — works but adds a test framework dependency and makes test intent less clear; mock returns null by default for `fireAsync()`
- Implementing `Event<T>` with an anonymous class — compiles only if all abstract methods are implemented; `Event<T>` in CDI has methods returning `CompletionStage`, `Event<U>`, and `ObserverException` that add noise
