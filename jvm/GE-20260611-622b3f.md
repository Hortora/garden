## CDI @Priority values documented in interface javadoc but never added to implementations

### Symptom

Enricher pipeline executes in wrong order. Two enrichers run at `Integer.MAX_VALUE`
(undefined relative order) despite the SPI interface javadoc claiming specific
priority values (10 and 30).

### Context

`LedgerEntryEnricher` javadoc listed assigned priorities for all implementations:
TraceIdEnricher (10), AgentSignatureEnricher (20), ProvenanceCaptureEnricher (30),
ActorDIDEnricher (40), ActorIdentityValidationEnricher (50). Only three of five
actually had `@jakarta.annotation.Priority` annotations. The other two sorted to
`Integer.MAX_VALUE` via `InjectableBean.getPriority()` default.

### Root cause

The priorities were documented as aspirational — added to the javadoc when the
enricher pipeline was designed, but the annotations were never added to the
implementation classes. The javadoc was treated as the source of truth, but
CDI reads annotations, not javadoc.

### Fix

Add the `@Priority` annotation to every implementation. Verify with a test that
exercises the ordering contract, not by reading javadoc.

```java
import jakarta.annotation.Priority;

@ApplicationScoped
@Priority(10)
public class TraceIdEnricher implements LedgerEntryEnricher { ... }
```

### Why this is non-obvious

The mismatch is invisible in normal operation — enrichers with undefined order
still run, just in arbitrary sequence. The bug only surfaces when you depend on
ordering (e.g., one enricher needs data set by another). The javadoc confidently
lists the priorities, so a developer reading the interface assumes the ordering
is correct without checking the implementation annotations.
