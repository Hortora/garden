### Symptom

```
org.hibernate.exception.ConstraintViolationException: column "tenancy_id" violates not-null constraint
```

Entities persisted via a static factory or utility method fail on INSERT even though the owning
parent entity has `tenancyId` set correctly.

### Context

`LedgerMerkleTree.append(digest, currentFrontier, subjectId)` is a pure static utility that
builds new `LedgerMerkleFrontier` nodes. It has no knowledge of tenancy context. The caller
in `ReactiveLedgerEntryJpaRepository.save()` sets `tenancyId` on the `LedgerEntry` being saved
but forgot to propagate it to the frontier nodes returned by `append()`. The nodes go into
`session.persist()` with `tenancyId = null`, violating the NOT NULL constraint.

### Root cause

Static factory and utility methods that create JPA entities operate outside any tenancy context.
Multi-tenancy fields (`tenancyId`, `tenantKey`, etc.) are caller responsibilities, not factory
responsibilities. The omission is invisible until the `session.persist()` flush fires — and only
on code paths where the NOT NULL column is actually written.

### Fix

Propagate multi-tenancy fields from the parent entity onto all child entities immediately after
the factory call, before any `session.persist()`:

```java
final List<LedgerMerkleFrontier> newFrontier =
    LedgerMerkleTree.append(e.digest, currentFrontier, e.subjectId);
newFrontier.forEach(node -> node.tenancyId = e.tenancyId);  // ← propagate before persist
// ... then session.persist(node) for each
```

### Why this is non-obvious

The parent entity (`LedgerEntry`) has `tenancyId` set correctly and was used for all prior
operations. The factory method looks complete — it takes all the structural parameters it needs.
The tenancy propagation step is easy to forget because it is not part of the factory's contract
and there is no type system or annotation to enforce it.
