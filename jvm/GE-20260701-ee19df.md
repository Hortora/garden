### Symptom

Quarkus startup fails with `BlockingNotAllowedException` or `ContextNotActiveException` from a `@PostConstruct` method that calls `ChannelService.findByName()` (JPA/Panache). The stack trace shows the call originates from `DefaultCaseDefinitionRegistry.registerKnownDefinitions()` running on a Vert.x IO thread.

### Context

When `QuarkMindCaseHub extends CaseHub` is discovered by CDI, the engine's `DefaultCaseDefinitionRegistry` calls `registerKnownDefinitions()` on a Vert.x IO thread during startup event processing. This triggers lazy resolution of CDI beans injected by the CaseHub subclass (e.g. `Instance<TaskDefinition>`). If any of those beans have `@PostConstruct` methods that perform blocking JPA work (channel creation, preference loading), the call hits the Vert.x IO thread blocker.

### Root cause

Vert.x IO threads prohibit blocking operations. JPA/Panache calls are blocking. The engine's case definition registration happens on an IO thread, which transitively constructs `@ApplicationScoped` beans that were designed to initialize eagerly via `@PostConstruct`.

### Fix

Defer blocking work from `@PostConstruct` to lazy initialization with double-checked locking:

```java
// Before (fails on IO thread):
@PostConstruct
void init() {
    channel = channelService.findByName("my-channel")
        .orElseGet(() -> channelService.create(...));
}

// After (lazy, thread-safe):
private volatile UUID channelId;

private void ensureChannel() {
    if (channelId == null) {
        synchronized (this) {
            if (channelId == null) {
                channelId = QuarkusTransaction.requiringNew()
                    .call(() -> channelService.findByName("my-channel")
                        .orElseGet(() -> channelService.create(...)))
                    .id();
            }
        }
    }
}
```

Call `ensureChannel()` on first use (e.g. first `execute()` call) instead of in `@PostConstruct`.

### Why this is non-obvious

The `@PostConstruct` pattern is standard for CDI bean initialization. It works for all non-JPA work. The failure only occurs when the bean is transitively discovered by the engine's CaseDefinitionRegistry — a dependency chain that crosses module boundaries and isn't visible from the bean's own code.
