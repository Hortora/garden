## Symptom

`org.hibernate.QueryParameterException: No argument for named parameter ':tenancyId'` thrown at runtime when calling `repo.update("lastActivityAt = ?1 WHERE id = ?2 AND tenancyId = ?3", now, id, tenancyId)`. The error references a named parameter you never wrote.

## Context

Any `@ApplicationScoped` Panache repository calling `PanacheRepositoryBase.update(String query, Object... params)` in a Hibernate Reactive application. Does NOT affect `repo.list()`, `repo.find()`, or other query methods — only `update()` with varargs params triggers this.

## Root Cause

Hibernate Reactive's internal implementation of `PanacheRepositoryBase.update()` with `Object... params` parses the SET and WHERE clauses and derives parameter names from adjacent entity field names. `?3` appearing adjacent to `tenancyId` in `AND tenancyId = ?3` is renamed to `:tenancyId` internally. The positional param is replaced by a named binding — but since the caller passed positional arguments (by index), no binding for `:tenancyId` exists, and the query throws at runtime.

## Fix

Use `Parameters.with()` (named params) in both the query string and the param map:

```java
// Before (breaks silently at runtime):
repo.update("lastActivityAt = ?1 WHERE id = ?2 AND tenancyId = ?3",
        Instant.now(), channelId, tenancyId);

// After (correct):
repo.update(
        "lastActivityAt = :now WHERE id = :channelId AND tenancyId = :tenancyId",
        Parameters.with("now", Instant.now())
                  .and("channelId", channelId)
                  .and("tenancyId", tenancyId))
    .replaceWithVoid();
```

Import: `io.quarkus.panache.common.Parameters`

Also apply to `IN ?1`-style list queries for consistency (no runtime bug, but avoids copying a broken pattern):

```java
// Safer:
repo.list("id IN :ids", Parameters.with("ids", new ArrayList<>(ids)));
```

## Why This Is Non-Obvious

The error message names `:tenancyId` — named parameter syntax — which leads you to believe you accidentally used a named param in the JPQL string. You didn't. The framework secretly renamed your positional param without any warning or startup error. The blocking equivalent (`Channel.update(...)` static Panache entity methods) works fine with positional params — the bug is Hibernate Reactive-specific and only surfaces when the update query executes at runtime under the reactive stack.
