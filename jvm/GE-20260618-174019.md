## Symptom

`org.h2.jdbc.JdbcSQLIntegrityConstraintViolationException: Unique index or primary key violation: PK_TABLE on TABLE(col1, col2)` thrown under concurrent load when multiple threads write to the same `(col1, col2)` key for the first time, using `MERGE INTO … WHEN NOT MATCHED THEN INSERT`.

## Context

H2 2.x defaults to MVStore (MVCC) engine. A sequence-counter table uses `MERGE INTO` as an atomic upsert: `WHEN NOT MATCHED THEN INSERT` seeds the row on first use, `WHEN MATCHED THEN UPDATE` increments on subsequent calls. Under 5+ concurrent threads all writing to a new `(subject_id, tenancy_id)` key simultaneously, `ConstraintViolationException` fires intermittently.

## What was tried

- `MERGE INTO` with only `WHEN NOT MATCHED THEN INSERT` — races.
- `MERGE INTO` with both `WHEN MATCHED UPDATE` and `WHEN NOT MATCHED INSERT` — still races for first-insert.
- Replacing with `INSERT … ON CONFLICT DO NOTHING + UPDATE` — serialises correctly.

## Root cause

H2 MVStore's `MERGE INTO` evaluates the `ON` condition using each transaction's committed MVCC snapshot. Both TX1 and TX2 see "NOT MATCHED" (TX1's uncommitted insert is invisible under MVCC), and both proceed to insert. The internal insert path inside `MERGE` does not acquire the same index-entry lock as a plain `INSERT` statement.

In contrast, plain `INSERT` goes through H2's standard index write path, which acquires an exclusive index-entry lock at execution time. TX2's `INSERT` blocks until TX1 commits, then detects the committed conflict. With `ON CONFLICT DO NOTHING`, the conflict is resolved cleanly. With a bare `INSERT`, a `ConstraintViolationException` is raised — which can be caught and retried, or suppressed with `DO NOTHING`.

## Fix

Replace `MERGE INTO … WHEN NOT MATCHED THEN INSERT` with plain `INSERT … ON CONFLICT DO NOTHING` (H2 in MODE=PostgreSQL) or full `MERGE INTO … WHEN MATCHED UPDATE WHEN NOT MATCHED INSERT` after confirming you've observed the locking difference on your H2 version.

For the H2 standard case (no MODE=PostgreSQL), note that concurrent first-inserts remain unsafe; the full MERGE with WHEN MATCHED UPDATE is single-statement but still races for new-row creation. H2 standard is suitable only for single-threaded test contexts. Production uses PostgreSQL where the semantics are well-defined.

## Why this is non-obvious

`MERGE INTO` is described as an atomic upsert. The MVCC snapshot read for the `ON` condition evaluation is not documented as a separate, non-locking step. A developer would assume the atomic statement prevents the race; only profiling or a concurrent test rig reveals the gap.
