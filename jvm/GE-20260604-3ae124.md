`@ConfigPhase.RUN_TIME` config interfaces cannot be injected directly as `@BuildStep` method
parameters. The build fails with an `IllegalStateException` that does not clearly explain why
the approach is invalid. Runtime config must flow through a `@Recorder`-annotated class:
inject it into the recorder's constructor as `RuntimeValue<YourConfig>`, then access it
at runtime via `runtimeValue.getValue()`.

### Symptom

```
IllegalStateException: ... cannot be injected into a build step
```

or the build step silently doesn't produce the expected result, with no config values
visible at runtime.

### Context

Occurs when adding a new `@ConfigMapping(prefix = "...")` + `@ConfigRoot(phase = ConfigPhase.RUN_TIME)`
interface to an existing Quarkus extension and trying to use it in a build step that needs
to pass config values to recorder-generated runtime bytecode.

### Fix

Inject the config into the `@Recorder` constructor as `RuntimeValue<YourConfig>`, NOT as
a `@BuildStep` parameter:

```java
// In the recorder class
@Recorder
public class MyRecorder {
    private final RuntimeValue<MyRuntimeConfig> config;

    public MyRecorder(RuntimeValue<MyRuntimeConfig> config) {
        this.config = config;
    }

    @RuntimeInit
    public void initialize() {
        boolean eagerInit = config.getValue().devUi().eagerInit();
        // use config at runtime
    }
}
```

The build step calls the recorder normally — Quarkus injects the `RuntimeValue` automatically
when the recorder is instantiated.

### Why this is non-obvious

Developers familiar with Spring or CDI inject config as method parameters or constructor
injection on `@Component`/`@ApplicationScoped` beans. `@BuildStep` methods accept many
build item types as parameters, so attempting to inject a config mapping as a parameter
feels natural. The error message does not say "use RuntimeValue — this is a RUN_TIME config"
and doesn't link to the recorder pattern. The distinction between BUILD_TIME and RUN_TIME
config lifecycle is not immediately clear from the `@ConfigPhase` enum name alone.

### Alternatives considered

- Accessing the config via `ConfigProvider.getConfig().getValue(...)` in a `@RuntimeInit`
  recorder method — works, but loses type-safe mapping and IDE support.
- Using `ConfigPhase.BUILD_AND_RUN_TIME_FIXED` — changes config lifecycle semantics and
  prevents runtime override of config values.
