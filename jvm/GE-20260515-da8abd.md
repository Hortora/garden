## Maven submodule folder names — short, no repo-name prefix

**Topic:** Maven multi-module submodule folder naming style
**Alternatives exist:** yes — prefix-per-module style (e.g. Spring Boot: spring-boot-actuator/, spring-boot-autoconfigure/)

### The convention

Submodule folder names are short and descriptive. They do not repeat the repo name as a prefix. The repo directory provides the namespace — the folder does not restate it.

Role modules use fixed short names: `api`, `runtime`, `deployment`, `testing`, `examples`, `integration-tests`.

Capability modules use short descriptive names: `queues`, `ai`, `notifications`, `persistence-memory`, `blackboard`.

The Maven `artifactId` in `pom.xml` carries the fully qualified name (`casehub-work-queues`); the folder does not (`queues/`).

### Why this style

Consistent with Quarkus extension conventions. Short names are easier to navigate, type, and grep. The repo root already provides the namespace — repeating it in every folder name adds noise without information.

### Trade-offs

Without the prefix, folder names alone don't identify which repo they belong to. When working across multiple repos simultaneously, short names can be ambiguous without directory context. The `artifactId` is always unambiguous — but that's only visible in `pom.xml`, not in filesystem navigation.

### When not to use it

When a monorepo bundles unrelated projects with genuinely different namespace ownership, prefix disambiguation may be warranted.
