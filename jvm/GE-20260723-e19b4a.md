## Symptom

A CDI `@Decorator` checks `delegate instanceof MarkerInterface` to detect whether
the base bean implements a marker. The check works when no other decorators are
between the checking decorator and the base bean. When additional decorators are
enabled (via `@IfBuildProperty` or classpath activation), the check silently returns
`false` — the guard fails with no error.

## Context

CDI decorator chains are ordered by `@Priority` (lowest = outermost, closest to
caller). Each decorator's `@Delegate` field references the **next decorator in the
chain**, not the base bean directly. When:

- Decorator A (@Priority 50) checks `delegate instanceof Marker`
- Decorator B (@Priority 65) sits between A and the base bean
- Base bean implements `Marker`

A's delegate is B (a proxy), not the base bean. `B instanceof Marker` is `false`.

## What was tried (didn't work)

`delegate instanceof BridgedCbrStore` — worked in unit tests with manual wiring
(no intermediate decorators) but failed in `@QuarkusTest` with the full CDI chain.

## Root cause

CDI decorator delegates are chain proxies. `instanceof` on the delegate checks the
immediate next decorator's type, not the eventual base bean's type.

## Fix

Use `Instance<MarkerInterface>.isResolvable()` for container-level detection:

```java
@Inject
MyDecorator(@Delegate @Any MySpi delegate,
            Instance<MarkerInterface> markerInstance) {
    this.delegate = delegate;
    this.markerActive = markerInstance.isResolvable();
}
```

This checks whether ANY bean implementing `MarkerInterface` exists in the CDI
container — a container-level query that is unaffected by decorator chain ordering.

## Why this is non-obvious

The `instanceof` approach is the natural Java pattern for type checking. CDI
decorator tutorials don't mention that the delegate is a chain proxy, not the
base bean. The pattern works in unit tests (manual wiring skips the chain) and
in simple configurations (single decorator), making the bug intermittent and
configuration-dependent.

### Alternatives considered

- Walking the decorator chain via reflection — fragile, implementation-specific
- Passing a flag via config — couples the guard to external configuration
- `Instance.isResolvable()` — clean, container-native, independent of chain ordering
