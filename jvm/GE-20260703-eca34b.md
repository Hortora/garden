## CorpusIngestionService saves cursor on empty fullScan — stale cursor blocks re-indexing

### Symptom

Engine starts in dev mode, filesystem watcher activates, but Qdrant collection is never created. No errors logged. Search returns empty results. Engine appears healthy but has zero indexed content.

### Context

When running the engine against a fresh Qdrant instance (e.g. after `podman run` with no volume), the `CorpusIngestionService` runs `fullScan()` at startup. If a cursor file exists from a previous run at `${java.io.tmpdir}/casehub-ingestion-cursors/garden.cursor`, the service calls `changesSince(cursor)` instead of `fullScan()`, finds no changes (cursor matches current state), and completes with zero chunks. The cursor is preserved, and the fresh Qdrant stays empty.

### Root cause

In `doProcessBinding()`, the cursor is saved whenever `anyFailure` is false — including when `allChunks` is empty (no entries to ingest). The logic:

```java
if (!allChunks.isEmpty()) {
    ingestor.ingest(corpusRef, allChunks);  // only runs if chunks exist
}
if (!anyFailure) {
    cursorStore.save(binding.name(), changeSet.newCursor());  // always runs if no errors
}
```

When `allChunks` is empty, no ingest is attempted, `anyFailure` stays false, and the cursor is saved. On next startup, `changesSince(cursor)` returns no changes because the filesystem hasn't changed.

### Fix

Delete the cursor file before starting the engine against a fresh Qdrant:

```bash
rm -rf ${TMPDIR:-/tmp}/casehub-ingestion-cursors/
```

Or use `java.io.tmpdir` from the JVM:
```bash
java -XshowSettings:all 2>&1 | grep tmpdir  # find the path
rm -f <path>/casehub-ingestion-cursors/garden.cursor
```

### Why this is non-obvious

- No errors or warnings are logged — the engine completes startup cleanly
- The cursor file is in `java.io.tmpdir`, not the project directory — easy to miss
- The symptom (empty search results) looks like a missing configuration or Qdrant connection issue, not a stale cursor
- The cursor file is 674KB (contains path→mtime mappings for ~11K files), which makes it look like real state

### Why this fix

The structural fix belongs in neocortex — the cursor should not be saved when zero chunks were produced from a non-empty fullScan (filed as casehubio/neocortex#67). The manual cursor deletion is a workaround until that's resolved.

### Alternatives considered

- Adding `casehub.corpus.corpora.garden.source` to create a second binding via `CorpusBindingProducer` — this created duplicate bindings and made the problem worse
- Triggering manual ingestion via REST endpoint — no such endpoint exists in the current engine
