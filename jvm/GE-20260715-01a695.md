### Symptom
`UnsolvedSymbolException{context='mpleted', name='Solving mpleted'}` when evaluating
`status == 'completed'`. The error message gives no indication the problem is with
string quoting.

### Context
Any MVEL3 expression using single-quoted string literals — e.g. `field == 'value'`.
Surfaces during migration from MVEL2 or when switching from parameterized expressions
(where values were bound as variables, avoiding string literals entirely).

### What was tried (didn't work)
Single-quoted strings: `status == 'completed'` — the natural syntax for anyone coming
from MVEL2 or most expression languages.

### Root cause
MVEL3's JavaParser-based transpiler does not recognize single quotes as string
delimiters. It parses `'completed'` as a character/symbol, then fails trying to resolve
`mpleted` as a Java identifier.

### Fix
Use double-quoted strings in all MVEL3 expressions:
```java
// Wrong — fails at runtime
new MvelExpressionEvaluator("status == 'completed'")

// Correct
new MvelExpressionEvaluator("status == \"completed\"")
```

### Why this is non-obvious
MVEL2 treated single and double quotes identically for string literals — this was a
widely-used feature. The MVEL3 migration guide does not document this breaking change.
Code using parameterized variables (`$p0` bound in the variables map) never encounters
this because no string literals appear in the expression text.
