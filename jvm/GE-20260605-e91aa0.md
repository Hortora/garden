### Symptom
Module with `@QuarkusTest` passes locally in seconds but hangs indefinitely in CI (2-core GitHub Actions runner). No error output. `forkedProcessTimeoutInSeconds` does not fire. The hang occurs during Quarkus boot, before any test method executes.

### Context
Module has `@QuarkusTest` classes but no `quarkus-maven-plugin` in its `pom.xml`. The module has a large CDI graph â€” depends transitively on engine, work, qhorus, ledger, platform (~15 Quarkus extensions). Passes locally on an 8+ core Mac in under 9 seconds.

### What was tried (didn't work)
- `@Timeout(60s)` on test classes â€” doesn't fire (hang is pre-test, during augmentation)
- `forkedProcessTimeoutInSeconds=300` â€” doesn't fire (appears to measure test time, not JVM boot time)
- Reducing thread counts, adding PostgreSQL timeouts â€” wrong module entirely

### Root cause
Without `quarkus-maven-plugin`, there are no `generate-code` / `generate-code-tests` build goals. Quarkus augmentation (CDI bean discovery, extension processing, bytecode generation) runs inside Surefire's forked test JVM at test time via `QuarkusTestExtension`. On a 2-core CI runner, augmenting a large CDI graph (~15 extensions, hundreds of beans) overwhelms the available CPU and hangs indefinitely.

With the plugin, augmentation happens at compile time in the main Maven process â€” fast, predictable, and not constrained by Surefire's forked JVM.

### Fix
Add `quarkus-maven-plugin` to the module's `pom.xml`:
```xml
<plugin>
    <groupId>io.quarkus.platform</groupId>
    <artifactId>quarkus-maven-plugin</artifactId>
    <extensions>true</extensions>
    <executions>
        <execution>
            <goals>
                <goal>generate-code</goal>
                <goal>generate-code-tests</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

### Why this is non-obvious
Modules without the plugin work fine locally because desktop CPUs have enough cores for in-JVM augmentation. The failure mode is silent â€” no error, no timeout, just an infinite hang. The module may have been in the build for weeks without anyone noticing because CI never tested it (reactor ordering or build configuration excluded it).

### Alternatives considered
- Skipping tests in CI â€” avoids the problem but loses CI coverage
- Adding CPU/memory to CI runner â€” expensive, doesn't fix the architectural issue
