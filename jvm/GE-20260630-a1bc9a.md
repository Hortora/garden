### Symptom
All tests that passed yesterday fail today with compilation errors. No code changed on the branch. Error: `method resolve in interface DIDResolver cannot be applied to given types; required: String, String; found: String`.

### Context
Multi-repo Maven project (`casehub-ledger`) consuming `casehub-platform-identity` at `0.2-SNAPSHOT`. The upstream repo changed `DIDResolver.resolve(String did)` to `resolve(String actorId, String did)` and published to GitHub Packages. Maven resolved the new SNAPSHOT on the next build, breaking all callers.

### Root cause
SNAPSHOT versions in Maven are mutable — `mvn` resolves the latest published artifact on every build (or per `updatePolicy`). When a cross-repo dependency changes its API and publishes, all downstream consumers break on their next build without any code change on their side.

### Fix
No single fix — this is inherent to SNAPSHOT semantics. Mitigations:
1. **Pin SNAPSHOT timestamps** in `~/.m2/settings.xml` via `<updatePolicy>never</updatePolicy>` during active branch work; `mvn -U` to pull intentionally
2. **Communicate API changes** before publishing — the upstream repo should file an issue or notify downstream consumers
3. **Fix quickly** — the migration is mechanical (add the new parameter to all call sites). Use IntelliJ "find usages" on the changed method to locate all callers

### Why this is non-obvious
The failure appears as a local compilation error with no local code change. The developer's first instinct is to check `git status` and `git stash` — both show nothing changed. The cause is invisible without checking `~/.m2/repository/` timestamps or running `mvn dependency:resolve -U` to see what changed. In a CI pipeline, the same build that was green yesterday is red today with no new commits.
