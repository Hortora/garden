## Quarkus eagerly starts @QuarkusTestResource from @Disabled test classes — breaks other tests

### Symptom

A test class annotated with both `@Disabled` and `@QuarkusTestResource(SomeContainerResource.class)` causes Docker image pull attempts (403 rate limit, 'connection refused', or similar) and breaks tests in other classes that share the same Quarkus application context. The `@Disabled` class never runs a single test, yet its resource lifecycle starts.

### Root cause

Quarkus collects `@QuarkusTestResource` annotations from **all test classes** at bootstrap time, before JUnit 5's `@Disabled` annotation has any effect. The test framework merges all discovered resource annotations into the set of resources to start for the shared Quarkus application context. `@Disabled` suppresses test method execution, but resource discovery happens earlier — at suite bootstrap, not at test-method scheduling.

### Fix

Comment out `@QuarkusTestResource` on disabled classes. Leave the annotation text in a comment so it's restored when the class is re-enabled:

```java
@Disabled("Blocked: requires Ollama in CI. Remove when infra is ready.")
@Tag("integration")
@QuarkusTest
// @QuarkusTestResource(Mem0ContainerResource.class)
// ↑ Uncomment when enabling — Quarkus eagerly starts @QuarkusTestResource
// from ALL test classes (including @Disabled ones), which breaks other tests.
class CaseMemoryStoreContractIT extends CaseMemoryStoreContractTest {
```

### Why this is non-obvious

JUnit 5's `@Disabled` is expected to suppress all test infrastructure for the annotated class. In pure JUnit 5 tests, `@Disabled` prevents `@BeforeEach`, `@AfterEach`, and extensions from running. Quarkus's `@QuarkusTestResource` is a Quarkus-level mechanism that hooks into the bootstrap phase before JUnit's lifecycle, so it sidesteps `@Disabled` entirely.

The symptom also misleads: the 403/connection-refused error points to Docker or a container registry, not to the disabled class — you need to trace back why Docker was called at all.
