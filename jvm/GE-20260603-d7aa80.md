# Mutiny Multi.createBy().merging() waits for ALL upstreams — wrong for wall-clock timeout

## Symptom

After implementing a wall-clock timeout by merging the event stream with a delayed-failure Multi:

```java
Multi<T> timeout = /* fails after N seconds */;
return Multi.createBy().merging().streams(events, timeout);
```

Successful sessions that complete before the timeout still fail with the timeout exception — but only after the configured delay. The timeout fires on completed sessions.

## Context

Reactive wall-clock timeout for a subprocess-backed stream. The merge approach seems obvious: race the event stream against a timeout signal and let whichever fails/completes first win.

## What was tried (didn't work)

1. `Multi.createBy().merging().streams(events, timeoutFailureMulti)` — events completes, merged stream stays open, timeout fires later
2. `Multi.createFrom().ticks().startingAfter(duration)...failWith(...)` merged the same way — same problem

## Root cause

Mutiny's `Multi.createBy().merging()` completes only when **ALL** upstream streams have completed or any one has failed. When the event stream completes normally before the timeout signal fires, the merged stream is still waiting for the timeout Multi to complete or fail. It eventually receives the timeout failure and propagates it — incorrectly marking a successful session as timed out.

This is documented behaviour: merge semantics are "interleave items, complete when all complete, fail fast on any failure." The implication for timeout patterns is not documented.

Additionally, `Multi.createFrom().ticks()` returns `Multi<Long>`, not `Multi<T>`. Calling `.failWith()` on it does not change the element type — merging `Multi<Long>` with `Multi<AgentEvent>` is a compile error.

## Fix

Do not use stream composition for wall-clock timeout when the underlying resource is a subprocess. Instead, schedule explicit resource closure via a `ScheduledExecutorService`:

```java
AtomicBoolean timedOut = new AtomicBoolean(false);
ScheduledFuture<?> timeoutFuture = scheduler.schedule(() -> {
    if (timedOut.compareAndSet(false, true)) {
        client.close().subscribe();  // kills the subprocess
    }
}, timeout.toMillis(), TimeUnit.MILLISECONDS);

return eventStream
    .onFailure().transform(e -> timedOut.get()
        ? new TimeoutException(timeout)
        : new ProcessException(e))
    .onCompletion().invoke(() -> timeoutFuture.cancel(false))
    .onFailure().invoke(t -> timeoutFuture.cancel(false))
    .onCancellation().invoke(() -> timeoutFuture.cancel(false));
```

The `compareAndSet(false, true)` prevents double-close if normal completion races with the timer. Cancelling the timer on all three termination paths ensures it doesn't fire after normal completion.

## Why this is non-obvious

The merge-based timeout is the standard reactive pattern in many frameworks (RxJava `amb()`, Reactor `firstWithSignal()`). Mutiny's merge does not have "first wins" semantics — it has "interleave + complete when all complete" semantics. This distinction is documented but the timeout anti-pattern is not. Any developer familiar with RxJava or Reactor timeouts would attempt the merge approach first.

### Why this fix

The subprocess closure approach ties the timeout directly to the resource lifecycle rather than stream composition. When the subprocess dies, the SDK's Flux fails naturally and the failure propagates through the existing error handling chain — no separate timer-stream composition needed.

### Alternatives considered

- `Uni`-based approach: `Uni.createFrom().voidItem().onItem().delayIt().by(timeout).flatMap(...).toMulti()` — same completion semantics problem when merged
- `select().first().within(duration)` — completes (not fails) after the window; no typed exception for timeout
