### Symptom

Engine starts, reports "No cursor ‚Äî bootstrapping with fullScan", the scan fails (e.g. Qdrant collection doesn't exist yet, or embedding API returns 400). Log says "Cursor NOT advanced due to processing failures." On restart, the engine does NOT retry the fullScan ‚Äî instead it runs `changesSince(cursor)` which finds no changes, and no documents are indexed. The corpus appears fully indexed (cursor file exists, 500KB) but Qdrant has 0 points.

### Context

`CorpusIngestionService` has two cursor-saving paths:

1. **`doProcessBinding`** ‚Äî saves cursor only if `!anyFailure` (correct)
2. **`checkpointCursors`** ‚Äî 5-minute scheduled timer that saves the filesystem watcher's current state as the cursor (unconditional)

When the initial scan fails, `doProcessBinding` correctly does NOT advance the cursor. But the filesystem watcher was started during `onStart()` and recorded the full file state. After 5 minutes, `checkpointCursors` fires and saves the watcher's state ‚Äî which includes all files ‚Äî as the cursor. On next restart, `changesSince(saved_cursor)` sees no changes.

### Root cause

`checkpointCursors` does not check whether the initial scan succeeded. It unconditionally saves the watcher's in-memory file state, which reflects what's on disk ‚Äî not what was successfully ingested. The scheduled poll (`poll()`) also cannot retry because it explicitly skips `WatchableChangeSource` bindings.

### Fix

None known ‚Äî this is a design issue in `casehub-rag`. Workaround: delete the cursor file and restart.

```bash
# Find and delete the stale cursor
rm /var/folders/.../casehub-ingestion-cursors/garden.cursor
# Also delete the Qdrant collection if it exists but is empty
curl -X DELETE 'http://localhost:6333/collections/hortora_garden'
# Restart the engine ‚Äî fullScan will run from scratch
```

### Why this is non-obvious

The cursor file's existence and size (500KB) suggests successful indexing. The engine logs "No cursor ‚Äî bootstrapping with fullScan" on first start, but on second start it silently runs `changesSince` and finds nothing. There is no log message saying "cursor loaded, N entries already indexed" ‚Äî the developer has to check Qdrant directly to discover the corpus is empty despite the cursor being present.
