## ArchUnit rules pass silently when no classes match the that() predicate — vacuous green

### Symptom

An ArchUnit structural test (`should(condition).check(classes)`) passes with zero assertions executed and no output — giving false confidence that a contract is enforced when it is not. No warning, no failure count, no indication that the rule ran against zero subjects.

### Context

Occurs when:
- The `ClassFileImporter` target package is renamed and the import returns an empty `JavaClasses`
- The `that()` predicate (e.g. `haveSimpleNameStartingWith("Reactive").and().haveSimpleNameEndingWith("Service")`) matches zero classes in the compiled output
- The compiled classes are not yet present on the test classpath (e.g. running tests without recompiling after moving a class)

Discovered during code review of a parity enforcement test: the reviewer noticed the matched class count was zero before the assertion, yet the test reported green.

### Root cause

ArchUnit's default behaviour for a rule with zero matching subjects is to trivially pass — there are no subjects to violate the condition. This is consistent with standard predicate logic (vacuously true) but counterintuitive for structural enforcement tests where "zero matching classes" means the check is broken, not satisfied.

### Fix

Add an explicit count assertion before the rule check:

```java
@Test
void reactiveAndBlockingServices_mustHaveBidirectionalMethodParity() {
    // Guard: fail loudly if the package moved or was renamed
    assertThat(SERVICE_CLASSES.stream()
            .filter(c -> c.getSimpleName().startsWith("Reactive")
                    && c.getSimpleName().endsWith("Service"))
            .count())
            .as("expected at least 2 Reactive*Service classes in scope — "
                    + "if the package moved, update the ClassFileImporter")
            .isGreaterThanOrEqualTo(2);

    classes()
            .that().haveSimpleNameStartingWith("Reactive")
            .and().haveSimpleNameEndingWith("Service")
            .should(haveBidirectionalMethodParity(SERVICE_CLASSES))
            .check(SERVICE_CLASSES);
}
```

The minimum count (2 in the example) should match the number of classes the rule is designed to cover.

### Why this is non-obvious

The test passes. No assertion failure, no zero-count warning — just a green build. The only way to discover the vacuous pass is to add a deliberate violation and confirm the test catches it, or to notice the matched-class count is zero during code review.
