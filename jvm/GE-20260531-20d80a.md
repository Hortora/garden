## Context

Configuring a HikariCP connection pool for SQLite with PRAGMA settings (WAL mode,
busy_timeout, synchronous, cache_size). SQLite has no standard JDBC URL parameters
for PRAGMAs that work reliably across xerial driver versions, and HikariCP's
`connectionInitSql` is a single SQL string — multiple semicolon-separated PRAGMAs
are driver-dependent.

## Technique

Use `SQLiteConfig` (xerial's typed PRAGMA API) to build a pre-configured `SQLiteDataSource`,
then pass the instance to HikariCP via `setDataSource()`. This is type-safe, explicit, and
avoids `PropertyElf` reflection coercion issues.

```java
SQLiteConfig sqLiteConfig = new SQLiteConfig();
// Detected :memory: path — skip WAL (not supported for in-memory databases)
if (!isMemory) {
    sqLiteConfig.setJournalMode(SQLiteConfig.JournalMode.WAL);
}
sqLiteConfig.setSynchronous(SQLiteConfig.SynchronousMode.NORMAL);
sqLiteConfig.setBusyTimeout(busyTimeoutMs);   // 5000ms recommended for production
sqLiteConfig.setCacheSize(64000);             // 64,000 pages (positive = pages)

SQLiteDataSource sqLiteDataSource = new SQLiteDataSource(sqLiteConfig);
sqLiteDataSource.setUrl("jdbc:sqlite:" + path);

HikariConfig hikari = new HikariConfig();
hikari.setDataSource(sqLiteDataSource);       // pre-built instance, not class name
hikari.setMaximumPoolSize(isMemory ? 1 : maxPoolSize);
hikari.setMinimumIdle(1);

HikariDataSource ds = new HikariDataSource(hikari);
```

**PRAGMA rationale for production:**
- `journal_mode=WAL` — readers never block writers; writers never block readers
- `synchronous=NORMAL` — safe on SSD; avoids full fsync per-transaction overhead
- `busy_timeout=5000` — wait up to 5 s on write contention before throwing; the single
  most impactful setting for concurrent multi-threaded access
- `cache_size=64000` — pages (positive value = pages; negative = kibibytes)

**:memory: special case:** SQLite does not support WAL mode for in-memory databases.
Skip `setJournalMode(WAL)` when path is `:memory:`. Force pool size to 1 to avoid
HikariCP creating multiple connections that each get their own isolated in-memory database.

## Why This Is Non-Obvious

The HikariCP docs show `setDataSourceClassName` + `addDataSourceProperty` as the idiomatic
configuration path. URL-based PRAGMA configuration (`jdbc:sqlite:/path?journal_mode=WAL&...`)
appears to work in simple cases but behaves inconsistently across xerial driver versions,
especially for `cache_size` with negative values. `connectionInitSql` is constrained to a
single SQL statement. `SQLiteConfig` is the xerial-native API, but the pattern of using it
with `hikari.setDataSource()` (bypassing class-name registration) is not shown in any
standard guide combining the two libraries.

## Constraints

Requires both `com.zaxxer:HikariCP` and `org.xerial:sqlite-jdbc` on the classpath.
Does not require Quarkus — works in any Java application. For Quarkus applications
that manage their own `DataSource` (bypassing Agroal), use `@PostConstruct`/`@PreDestroy`
on an `@ApplicationScoped` bean to manage the HikariCP lifecycle.
