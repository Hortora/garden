## Symptom

`WorkBroker.apply()` returns `AssignmentDecision.noChange()` (i.e. `isNoOp() == true`)
even when the candidate list contains workers that match the required capability.
No error is thrown. `WorkerSelectionStrategy.select()` is never called.

## Context

Occurs when calling `WorkBroker.apply(context, trigger, candidates, strategy)` where:
- Candidates were built using `WorkerCandidate.of(workerId)` (the factory method)
- `SelectionContext.requiredCapabilities()` is non-null (e.g. the capability name string)

The internal `WorkBroker.filterByCapabilities()` checks whether each candidate's
`capabilities()` set **contains** the `requiredCapabilities` string. Because
`WorkerCandidate.of(id)` creates a candidate with an **empty** `Set<String>`,
the filter removes every candidate before the strategy is even invoked.

## What was tried

Passing the capability name as `requiredCapabilities` in `SelectionContext`, expecting
WorkBroker to handle the capability matching. WorkBroker always returned `noChange()`.

## Root cause

`WorkerCandidate.of(String id)` creates a `WorkerCandidate(id, Set.of(), 0)` —
the capabilities set is empty by design. The `WorkBroker` then filters this empty-capability
candidate against the non-null `requiredCapabilities`, resulting in zero candidates
reaching the strategy.

## Fix

Pass `null` for `requiredCapabilities` in `SelectionContext` when callers pre-filter
candidates by capability before calling `WorkBroker`:

```java
// Pre-filter candidates by capability yourself
List<WorkerCandidate> candidates = workers.stream()
    .filter(w -> w.getCapabilities().stream()
        .anyMatch(c -> c.getName().equals(capability.getName())))
    .map(w -> WorkerCandidate.of(w.getName())
        .withActiveWorkItemCount(workloadProvider.getActiveWorkCount(w.getName())))
    .toList();

// Pass null for requiredCapabilities — pre-filtering already done
SelectionContext ctx = new SelectionContext(
    capability.getName(), // category
    null,                 // priority
    null,                 // requiredCapabilities — NULL avoids WorkBroker filtering
    null,                 // candidateGroups
    null                  // candidateUsers
);

AssignmentDecision decision = workBroker.apply(ctx, AssignmentTrigger.CREATED, candidates, strategy);
```

## Why this is non-obvious

`WorkerCandidate.of(id)` looks like a convenience factory that creates a valid candidate.
Nothing in the API signals that the resulting candidate has empty capabilities and will
be filtered out if `requiredCapabilities` is set. The failure is completely silent —
`WorkBroker` returns `noChange()` with no log or error, making it appear that the
strategy rejected all candidates.
