### Symptom
Tests using mocked `WorkerExecutionManager` silently return wrong values.
`FirstSupportedRoutingStrategy.select()` returns `Optional.empty()` even
when `supports()` is stubbed to return `true` — the new `canExecute()`
default method (added per SPI evolution protocol) returns `false` from
the mock instead of the interface default `true`.

### Context
Adding a `default boolean canExecute(WorkerFunction function) { return true; }`
method to an existing SPI interface. Existing tests mock the interface with
`mock(WorkerExecutionManager.class)` and only stub `supports()`. The new
method is never stubbed because the developer expects the interface default
to apply.

### Root cause
Mockito creates a subclass proxy that overrides ALL methods — including
default methods. Unstubbed methods return Mockito's type default (`false`
for boolean, `null` for objects, `0` for numbers), NOT the interface's
`default` implementation. Java's `default` keyword provides a fallback for
classes that don't override the method — but Mockito's proxy DOES override
it (with its own dispatch), so the fallback never fires.

### Fix
Explicitly stub every default method that matters on every mock:

```java
WorkerExecutionManager mock = mock(WorkerExecutionManager.class);
when(mock.supports("cap", "tenant")).thenReturn(true);
when(mock.canExecute(any())).thenReturn(true);  // must stub — mock won't use interface default
```

### Why this is non-obvious
Java developers reasonably expect `default` methods to provide a fallback
that mock objects inherit — the same way a concrete class that doesn't
override the method would. The failure is silent: no exception, no warning,
just a different return value. The test appears to test the right thing but
exercises a code path that can't occur in production (where CDI beans
inherit the real default).

### Alternatives considered
- `Mockito.withSettings().defaultAnswer(CALLS_REAL_METHODS)` — delegates to
  real default methods, but also delegates all OTHER methods, which may throw
  on uninitialized state. Only safe for interfaces, not abstract classes.
- Extracting a test utility `mockBackend(supports)` that stubs both methods —
  chosen approach; centralises the stubbing pattern.
