# Serialisation boundary rule — bridge.serialise() only at storage/wire boundaries

## Context

In a system with typed context adapters (bridges) that translate between different context representations at boundary points, the natural implementation serialises at every boundary — the bridge converts to JSON on the way out, deserialises on the way in. This works correctly but creates unnecessary Jackson round-trips for same-JVM transfers where the object could pass directly as a Java reference.

## The technique

Separate the bridge's two roles — typing and serialisation — and call them at different points:

1. **Typing** (`bridge.initialise()`, `bridge.extractOutput()`) — at every boundary crossing. Always happens.
2. **Serialisation** (`bridge.serialise()`, `bridge.deserialise()`) — only at storage boundaries (EventLog, database) and wire boundaries (HTTP, message queues). Only when needed.

| Boundary type | Typing | Serialisation |
|---------------|--------|---------------|
| Internal, same thread (in-process worker call) | Direct POJO pass | None |
| Internal, async with recovery gap (scheduled worker) | `initialise()` at scheduling | `serialise()` → EventLog |
| Internal, same JVM (signal, SubCase spawn) | Direct POJO through event bus | `serialise()` → EventLog only |
| External (HTTP, Qhorus channel) | `deserialise()` from wire | Deserialisation IS the entry point |

## Why this is non-obvious

The bridge SPI has both `initialise()` and `serialise()` methods. The natural implementation calls both in sequence at every boundary — it feels like a pipeline. The insight is that they serve different concerns and their call sites are different: typing happens at the boundary, serialisation happens at the persistence layer. Conflating them means in-memory deployments pay a JSON round-trip cost for data that never leaves the JVM.

## How to verify the contract

A `SerializationDetectingBridge<T>` wraps a real bridge, delegates `initialise()`/`extractOutput()`, and throws `AssertionError` on `serialise()`/`deserialise()`. Use it in tests for internal paths to structurally enforce the rule.
