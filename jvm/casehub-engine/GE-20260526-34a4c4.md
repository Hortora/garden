## Symptom

CDI deployment fails with five identical lines:

```
Found 5 deployment problems:
  Unsatisfied dependency for type WorkerExecutionRecoveryService and qualifiers [@Default]
  Unsatisfied dependency for type WorkerExecutionRecoveryService and qualifiers [@Default]
  Unsatisfied dependency for type WorkerExecutionRecoveryService and qualifiers [@Default]
  Unsatisfied dependency for type WorkerExecutionRecoveryService and qualifiers [@Default]
  Unsatisfied dependency for type WorkerExecutionRecoveryService and qualifiers [@Default]
```

Nothing in the error references the newly-added dep.

## Context

Triggered when adding `casehub-engine-scheduler-quartz` to the test classpath of a project
that does not have a `WorkerExecutionRecoveryService` implementation available (e.g. the
implementation is excluded in the test profile via `quarkus.arc.exclude-types`).

## Root Cause

`casehub-engine-scheduler-quartz` ships four `@ApplicationScoped` CDI beans that all inject
`WorkerExecutionRecoveryService`:

- `ScheduledTriggerJob`
- `ConditionalScheduledTriggerJob`
- `QuartzWorkerExecutionManager`
- `QuartzWorkerExecutionJobListener`

The only implementation is `DefaultWorkerExecutionRecoveryService` (in the engine runtime).
Most test profiles exclude it to avoid its transitive deps. Adding the scheduler-quartz dep
activates all four beans simultaneously — producing exactly five unsatisfied dep errors
(four beans + the implementation itself trying to satisfy them).

## Fix

Use a `@DefaultBean @ApplicationScoped` no-op `JobScheduler` in test sources instead of
adding `casehub-engine-scheduler-quartz`. This satisfies `SchedulerService`'s `JobScheduler`
injection without activating any of the scheduler-quartz CDI beans:

```java
@DefaultBean
@ApplicationScoped
class NoOpJobScheduler implements JobScheduler {
    @Override public Uni<Void> schedule(ScheduledJobRequest request) { return Uni.createFrom().voidItem(); }
    @Override public Uni<Void> schedule(ScheduledJobRequest.Builder builder) { return Uni.createFrom().voidItem(); }
    @Override public Uni<Boolean> cancel(JobIdentifier jobId) { return Uni.createFrom().item(false); }
    @Override public Uni<Integer> cancelGroup(String groupName) { return Uni.createFrom().item(0); }
    @Override public Uni<Boolean> exists(JobIdentifier jobId) { return Uni.createFrom().item(false); }
}
```

This is safe when the test's case definition has no schedule bindings — `registerScheduledTriggers()`
returns immediately without ever calling any scheduler method.

## Why This Is Non-Obvious

The error repeats `WorkerExecutionRecoveryService` five times — the natural instinct is to
look at that type, not at the new dep that pulled in four CDI beans that all inject it.
The dep name (`casehub-engine-scheduler-quartz`) appears nowhere in the error output.
