## CaseHub.signal() is async — fire-and-forget, not synchronous blackboard update

### Symptom

Code calls `caseHub.signal(caseId, "key", value)` expecting the value to be in `input.get("key")` when the next worker reads the blackboard. The key is missing or null.

### Context

`CaseHub.signal(UUID, String, Object)` returns `void`. Based on the signature and name, it appears to update the blackboard synchronously. In practice it uses Vert.x event bus `publish()` — fire-and-forget — and returns before the blackboard is updated.

Occurs whenever you call `signal()` and then rely on a worker reading that key in the same or next Quartz execution window.

### Root Cause

`CaseHub.signal()` delegates to `CaseHubRuntimeImpl.signal()`, which delegates to `CaseHubReactor.signal()`:

```java
// CaseHubReactor.signal()
void signal(UUID caseId, String path, Object value) {
    eventBus.publish(SIGNAL_RECEIVED, new SignalReceivedEvent(caseId, path, value));
}
```

`eventBus.publish()` is fire-and-forget on the Vert.x event bus. It returns before any subscriber processes the event. The blackboard update happens asynchronously on the reactor pool.

Compare with `startCase()`, which uses `eventBus.request()` (request-reply) to block until the case is initialised.

### Fix

For getting the case ID inside a worker lambda, use `WorkerExecutionContext.current().caseId()` instead of signalling. The engine populates `WorkerExecutionContext` on the Quartz thread immediately before invoking the worker function — it is synchronous and guaranteed to be present.

```java
// In a worker lambda:
final UUID caseId = WorkerExecutionContext.current().caseId();
// caseId is the CaseInstance UUID — always non-null
```

`WorkerExecutionContext` is in `io.casehub.engine.common.spi.context`. Its `caseId()` field comes from the `WorkerContext` record populated by `QuartzWorkerExecutionJob` before any worker executes.

### Why Non-Obvious

`signal()` returns void, has no async suffix, and the `CaseHub` class Javadoc does not explain the delivery model. The contrast with `startCase()` (which blocks) is not documented. A developer would reasonably assume void = synchronous.
