## Context

Querying `EventLogRepository.findByCaseAndTypes(caseId, List.of(CASE_FAULTED), tenancyId)` for a faulted case. Expected: one `CASE_FAULTED` entry. Observed: two entries with different metadata schemas.

## Symptom

`findByCaseAndTypes()` returns two `CASE_FAULTED` entries for the same case when the fault was triggered by worker retries exhausted or case timeout. Goal-triggered faults produce only one entry.

## Root Cause

Three distinct code paths write `CASE_FAULTED` EventLog entries:

| Path | Handler | Metadata |
|------|---------|----------|
| Goal-triggered | `CaseStatusChangedHandler` | `{oldStatus, newStatus, goalName, goalKind}` |
| Worker retries exhausted | `WorkerRetriesExhaustedEventHandler` | `{workerId, inputDataHash}` |
| Case timeout | `CaseTimeoutEnforcer` | `{reason: "timeout"}` |

For paths 2 and 3, the originating handler writes its own `CASE_FAULTED` EventLog entry, then publishes a `CaseStatusChanged` event on the Vert.x event bus. `CaseStatusChangedHandler` consumes this event and writes a *second* `CASE_FAULTED` entry with `{oldStatus, newStatus}` metadata.

The goal-triggered path (1) only produces one entry because the goal evaluation happens inside `CaseStatusChangedHandler` itself — no separate handler writes a prior entry.

## Fix

When querying for terminal events, iterate all `CASE_FAULTED`/`CASE_CANCELLED` entries and apply disambiguation:

```java
Instant occurredAt = null;
String triggerGoalName = null;
String triggerGoalKind = null;

for (EventLog e : events) {
    if (e.getEventType() == CaseHubEventType.CASE_FAULTED
            || e.getEventType() == CaseHubEventType.CASE_CANCELLED) {
        // Use earliest timestamp
        if (occurredAt == null || e.getTimestamp().isBefore(occurredAt)) {
            occurredAt = e.getTimestamp();
        }
        // Extract goal metadata from whichever entry has it (at most one will)
        if (triggerGoalName == null && e.getMetadata() != null
                && e.getMetadata().has("goalName")) {
            triggerGoalName = e.getMetadata().get("goalName").asText();
            triggerGoalKind = e.getMetadata().get("goalKind").asText();
        }
    }
}
```

## Why This Is Non-Obvious

The double-write is an emergent property of the event bus architecture, not documented in any handler's Javadoc. Reading `WorkerRetriesExhaustedEventHandler` alone shows one `CASE_FAULTED` write. Reading `CaseStatusChangedHandler` alone shows one `CASE_FAULTED` write. Only by tracing the event bus message flow between the two handlers does the double-write become visible. The `CASE_CANCELLED` path does not have this issue — only `CaseStatusChangedHandler` writes `CASE_CANCELLED`.
