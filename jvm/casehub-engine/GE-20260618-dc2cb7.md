### Symptom

Building MCP tools over `CaseHubRuntime.eventLog(caseId)` to resolve which capabilities are active/completed for a case. Expected `CaseEventLogRecord` to carry the worker/capability identifier. It doesn't — the record has `(eventType, streamType, timestamp, payload, metadata)` only.

### Context

`CaseHubRuntimeImpl.eventLog()` maps `EventLog` (internal engine model) to `CaseEventLogRecord` (API model). The internal `EventLog` has a `workerId` field set to `worker.getName()` by `DefaultWorkOrchestrator`. But the mapping at `CaseHubRuntimeImpl:156` drops it:

```java
new CaseEventLogRecord(
    event.getEventType(),
    event.getStreamType(),
    event.getTimestamp(),
    event.getPayload(),
    event.getMetadata())
```

No `event.getWorkerId()` in the mapping.

### Root cause

API design gap — `CaseEventLogRecord` was designed for timeline display (what happened when), not for capability resolution (which worker did what). The `workerId` lives in the internal model only.

### Fix

Read capability info from `metadata` JsonNode. `DefaultWorkOrchestrator.buildWorkSubmittedLog()` populates metadata with `capabilityName` and `correlationKey`:

```java
for (CaseEventLogRecord event : workerEvents) {
    String capName = event.metadata() != null && event.metadata().has("capabilityName")
        ? event.metadata().get("capabilityName").asText()
        : null;
    if (capName == null) continue;
    // use capName for capability resolution
}
```

### Why this is non-obvious

The API model (`CaseEventLogRecord`) appears complete — 5 fields including `payload` and `metadata`. Nothing signals that a field from the internal model was intentionally dropped. The natural assumption is that `workerId` would be either a direct field or in `payload`, not buried in `metadata` under a different name (`capabilityName`).
