## Symptom

`HumanTaskScheduleHandler` logs:

```
PlanItem for binding 'human-approval' case <uuid> is not PENDING (status=RUNNING) — skipping
```

No WorkItem is created. The case stalls in WAITING state with no error and no exception.

## Context

Any CaseHub harness that uses `humanTask:` YAML bindings and has `casehub-engine-work-adapter` on the classpath. The failure is silent — the engine logs a WARN, but the HITL mechanism produces no observable effect.

## Root Cause

`PlanningStrategyLoopControl.indexSelectedForCompletion()` iterates all selected bindings and calls `pi.markRunning()` on their PlanItems — including humanTask bindings — before the `HumanTaskScheduleEvent` is published to the Vert.x event bus.

`HumanTaskScheduleHandler.onHumanTaskSchedule()` then receives the event, looks up the PlanItem, and guards:

```java
if (item.getStatus() != PlanItemStatus.PENDING) {
    LOG.warnf("...not PENDING (status=%s) — skipping", item.getStatus());
    return;
}
```

Since the PlanItem is already RUNNING, the guard fires and the handler returns without calling `workItemService.create()`. No WorkItem is ever persisted.

## Fix

In `HumanTaskScheduleHandler.onHumanTaskSchedule()`, change the guard to only skip on terminal states:

```java
// Accept PENDING or RUNNING — PlanningStrategyLoopControl marks RUNNING before the event fires.
if (item.getStatus() != PlanItemStatus.PENDING
        && item.getStatus() != PlanItemStatus.RUNNING) {
    LOG.warnf("PlanItem for binding '%s' case %s is terminal (status=%s) — skipping",
            event.bindingName(), event.caseId(), item.getStatus());
    return;
}
```

In both `handleInlineMode` and `handleTemplateMode`, guard the `item.markRunning()` call:

```java
if (item.getStatus() == PlanItemStatus.PENDING) {
    item.markRunning();
}
```

Alternatively, fix in `PlanningStrategyLoopControl.indexSelectedForCompletion()`: skip humanTask bindings, leaving PlanItem state management to the handler.

## Why This Is Non-Obvious

The PENDING guard reads as an idempotency gate — protecting against duplicate event delivery. But the marking happens in a different module (blackboard's planning strategy) at a different point in the dispatch pipeline. The WARN log is easy to miss among the surrounding "No workers defined" messages. The symptom (case stalls) has many possible causes; looking at the PlanItem status in the blackboard is not the obvious first diagnostic step.
