## Symptom

Quarkus fails to start with:

```
java.lang.IllegalStateException: Unable to create Scheduler
Caused by: java.lang.IllegalArgumentException: Cron expression contains 5 parts but we expect one of [6, 7]
```

The error appears when adding `casehub-engine-scheduler-quartz` to a project that already
uses `casehub-work`. All `@QuarkusTest` tests fail at CDI startup — none run.

## Context

`casehub-engine-scheduler-quartz` transitively pulls in `quarkus-quartz`, which activates
the Quartz scheduler extension. Quartz requires 6 or 7 field cron expressions
(second, minute, hour, dom, month, weekday, [year]).

`casehub-work` ships several `@Scheduled` beans with standard 5-field Unix cron:
- `ExpiryCleanupJob` — cron-based expiry cleanup
- `ClaimDeadlineJob` — cron-based claim deadline check
- `RoutingCursorCleanupJob` — cron-based cursor cleanup (`{casehub.work.routing.cursor.cleanup-cron}`)

When Quartz activates, these 5-field expressions fail at cron trigger creation, aborting startup.

## Fix

In test `application.properties`, exclude the offending scheduler beans and configure
Quartz to use RAM store with forced start:

```properties
# Force Quartz to start (needed even with no @Scheduled methods)
quarkus.scheduler.start-mode=forced
quarkus.quartz.store-type=ram

# Exclude casehub-work scheduler beans with 5-field cron incompatible with Quartz
quarkus.arc.exclude-types=\
  io.casehub.work.runtime.service.ExpiryLifecycleService,\
  io.casehub.work.runtime.service.ExpiryCleanupJob,\
  io.casehub.work.runtime.service.ClaimDeadlineJob,\
  io.casehub.work.runtime.strategy.RoutingCursorCleanupJob
```

The excluded beans are scheduler-only — excluding them from CDI does not affect WorkItem
lifecycle or SLA enforcement (those are driven by WorkItemService, not the scheduler beans).

Note: `quarkus.arc.exclude-types` removes beans from CDI but does NOT prevent Quarkus's
`@Scheduled` build-time processor from scanning them. The Quartz `createTrigger` fails
because the build processor finds and registers the 5-field cron expressions before CDI
exclusion takes effect. The `start-mode=forced` + `store-type=ram` configuration satisfies
the Quartz extension initialization without JDBC tables or @Scheduled methods.

## Why This Is Non-Obvious

Adding a new engine dependency (`casehub-engine-scheduler-quartz`) breaks the work module's
existing scheduler beans — a cross-artifact cron format incompatibility with no connection
to the feature being added. The error mentions cron parsing, not the new dependency. The
instinct is to look at cron expressions in the project, not at a transitive dependency
that introduced a different cron format requirement. The fix requires excluding three
`casehub-work` beans that were working fine before the engine dep was added.
