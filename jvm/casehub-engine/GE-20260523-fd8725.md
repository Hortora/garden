## Symptom

A humanTask or capability binding fires unconditionally on every `CONTEXT_CHANGED` event,
regardless of the `when` condition. Wrong WorkItems are created (e.g. a DSMB escalation
WorkItem appears for a Grade 3 AE even though `requiresDsmbEscalation=false`). No error,
no warning â€” the wrong binding fires silently.

## Context

Any `YamlCaseHub` harness using `contextChange` bindings with conditions in the `when` field.
The canonical example from `ae-escalation.yaml`:

```yaml
- name: dsmb-escalation
  on: { contextChange: {} }
  when: ".requiresDsmbEscalation == true and .dsmbEscalation == null"
  humanTask:
    title: "DSMB escalation â€” Grade 4+ adverse event"
```

This binding fires on every `CONTEXT_CHANGED` regardless of `requiresDsmbEscalation`.

## Root Cause

`CaseContextChangedEventHandler` evaluates `on.contextChange.filter` but ignores
`binding.getWhen()`. The `when` field is only evaluated by `SchedulerService` for
schedule/timer triggers. For `contextChange` bindings, `when` is a no-op.

The engine's PlanItem dedup guard ("PlanItem for binding X is not PENDING â€” skipping")
prevents duplicate WorkItems on re-fires, but the very first fire on wrong-condition
context still creates an incorrect WorkItem.

## Fix

Move binding conditions from `when:` to `on.contextChange.filter:`:

```yaml
# WRONG â€” when is ignored for contextChange
- name: dsmb-escalation
  on: { contextChange: {} }
  when: ".requiresDsmbEscalation == true and .dsmbEscalation == null"

# CORRECT â€” condition evaluated by CaseContextChangedEventHandler
- name: dsmb-escalation
  on:
    contextChange:
      filter: ".requiresDsmbEscalation == true and .dsmbEscalation == null"
  humanTask:
    title: "DSMB escalation â€” Grade 4+ adverse event"
```

Filed as casehubio/engine#335. All `contextChange` binding conditions must use
`on.contextChange.filter`, not `when`.

## Why This Is Non-Obvious

The YAML schema has both `when` and `on.contextChange.filter` as available fields.
`when` appears to be the natural place for a binding condition â€” it's named like a
conditional, documented alongside the binding, and syntactically valid. Nothing in
the YAML, the engine logs, or the runtime output signals that `when` is being ignored.
The first observable symptom is a wrong WorkItem being created, which has many other
possible causes (wrong initial context, wrong binding name, wrong condition logic).
