### What it does

`CaseHubRuntime.signal(caseId, path, null)` sets the context key to `null` in the case context map. It does NOT remove the key from the map.

### How it works

`WritablePanelImpl.applyAndDiff()` (line 522) uses `current.put(leaf, value)` regardless of whether value is null. With `value = null`, the map entry `{key: null}` exists. The method detects the value change (previous non-null → null), records a diff, fires `CaseContextChangedEvent`, and triggers binding re-evaluation.

### Why it matters

For operator tools that need to retry a capability (e.g. clearing a stalled review to re-fire a binding), signaling with `null` works because:
1. `Map.get(key)` returns `null` for both absent keys and keys with null values
2. Lambda binding conditions like `ctx.get("securityReview") == null` evaluate to `true` in both cases
3. The binding re-fires as if the capability had never completed

### Caveat

JQ-based binding conditions (YAML case definitions) may distinguish between null and absent differently. This has not been tested for JQ evaluators — only for `LambdaExpressionEvaluator`.

No mention in the CaseHubRuntime interface Javadoc, CaseContext interface, or any engine documentation. Discoverable only by reading `WritablePanelImpl` source code.
