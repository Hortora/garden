## Symptom

`WorkItemLifecycleAdapter.onWorkItemLifecycle(@ObservesAsync WorkItemLifecycleEvent)` in `casehub-engine-work-adapter` is never invoked when events are fired from a devtown application, even though:
- The bean IS discoverable — it can be `@Inject`ed and its methods called directly
- `@ObservesAsync` works for test-scope beans in the same application module
- The Jandex index in the work-adapter jar contains both `ObservesAsync` and `onWorkItemLifecycle`
- `DefaultAsyncObserverExceptionHandler` never logs an error (method is not called at all)

The engine's own work-adapter test suite (`WorkItemLifecycleAdapterTest` in `io.casehub.workadapter`) successfully dispatches to the same method.

## Context

Complex Quarkus application (`casehub-devtown-app`) with: `quarkus-rest`, `quarkus-rest-jackson`, multiple datasources (qhorus H2 + work H2), many event-bus subscribers, and `casehub-engine-work-adapter` declared as a `quarkus.index-dependency` external jar.

## Root Cause (working theory)

Quarkus CDI augmentation may not fully register `@ObservesAsync` observer methods from `quarkus.index-dependency` jars in the same way it does for beans in the application's own source tree. The engine test works because the test class is in the same module as `WorkItemLifecycleAdapter` (same package `io.casehub.workadapter`), giving Quarkus the full augmentation context it needs.

## Fix

Invoke the observer method directly from the application where CDI dispatch cannot be relied on:

```java
// After WorkItemService.completeFromSystem(), get the completed WorkItems and call adapter directly
completedWorkItems.forEach(wi -> lifecycleAdapter.onWorkItemLifecycle(
    WorkItemLifecycleEvent.of("COMPLETED", wi, "system", wi.resolution)));
```

Verify CDI async delivery separately using a test-scope `@ApplicationScoped` bean in the application module:

```java
@ApplicationScoped
class WorkItemCompletionCapture {
    @ObservesAsync WorkItemLifecycleEvent event fires correctly
    // because the bean is in the app module, not an external jar
}
```

## Why This Is Non-Obvious

The bean IS injected by CDI (`@Inject WorkItemLifecycleAdapter adapter` works). The Jandex index IS present. `DefaultAsyncObserverExceptionHandler` never fires — the method is not called at all, not called and throwing. In the engine module's own tests, the same observer works perfectly. There is no startup error or warning. The only indication is the absence of observable side effects (context not updated).
