### Symptom
Maven build fails with "Could not find artifact org.drools:drools-engine:jar:10.1.0" when
following Drools documentation or older examples that reference `drools-engine` as the runtime
dependency.

### Context
Setting up a Drools 10 project using the classic kie-api (KieBase/KieSession, not Rule Units).
Documentation and migration guides reference `drools-engine` as the consolidated runtime artifact.

### What was tried (didn't work)
- `org.drools:drools-engine:10.1.0` — artifact does not exist in Maven Central or any repository
- `org.drools:drools-engine-classic:10.1.0` — also does not exist at this version

### Root cause
`drools-engine` is a consolidation artifact introduced after the 10.1.0 release. It only exists
at `999-SNAPSHOT` in local builds. At 10.1.0, the runtime is composed of individual modules:
`drools-core`, `drools-kiesession`, `drools-compiler`, `drools-model-compiler`.

### Fix
Use `org.drools:drools-model-codegen:10.1.0` as the single compile dependency. It transitively
provides the complete chain: `drools-model-compiler` → `drools-compiler` → `drools-core` +
`drools-kiesession` + `kie-api`.

For static classloading (no dynamic bytecode generation), also add:
`org.drools:drools-wiring-static:10.1.0` at runtime scope.

```xml
<dependency>
    <groupId>org.drools</groupId>
    <artifactId>drools-model-codegen</artifactId>
    <version>10.1.0</version>
</dependency>
<dependency>
    <groupId>org.drools</groupId>
    <artifactId>drools-wiring-static</artifactId>
    <version>10.1.0</version>
    <scope>runtime</scope>
</dependency>
```

Use `buildAll(ExecutableModelProject.class)` — never plain `buildAll()` which uses the
deprecated DRL-to-bytecode compilation path.

### Why this is non-obvious
The Drools documentation and migration guides reference `drools-engine` as if it were a
published artifact. The natural search path (Maven Central, repository browser) returns nothing
— no "did you mean" suggestion, just a missing artifact error. The correct replacement
(`drools-model-codegen`) is not intuitively named as a runtime dependency.

### Alternatives considered
- `drools-compiler` alone — compiles DRL but lacks `ExecutableModelProject` (lives in
  `drools-model-codegen`)
- Individual modules (`drools-core` + `drools-kiesession` + `drools-compiler`) — works but
  unnecessarily verbose; `drools-model-codegen` pulls them all transitively
