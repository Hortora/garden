## Symptom

Calling `register(Class<T>)` where `T extends Enum<T> & VocabularyTerm` with an argument of type
`Class<? extends Enum<? extends VocabularyTerm>>` fails to compile:

```
method register(Class<T>) in VocabularyRegistry cannot be applied to given types;
required: Class<T>, found: Class<CAP#1>
where T is a type-variable: T extends Enum<T>&VocabularyTerm
```

## Context

Multi-layer generic registry pattern: an SPI method accepts a strongly-typed enum class
(`Class<T extends Enum<T> & VocabularyTerm>`), but a CDI `@FunctionalInterface` returns the
class via a wildcard (`Class<? extends Enum<? extends VocabularyTerm>>`). Calling the SPI
method with the wildcard return fails at the call site.

## What was tried (didn't work)

- Passing the wildcard value directly to the generic method — compile error
- Casting with `(Class<T>)` at the call site — requires `@SuppressWarnings("unchecked")` but
  still compiles to the same bytecode; doesn't help with the inference failure

## Root cause

Java's type checker can't prove that `? extends Enum<? extends I>` satisfies
`T extends Enum<T> & I`. The recursive self-bound `Enum<T>` requires the same type variable T
for both the enum's own supertype and the type argument to `Enum`. The wildcard form
`? extends Enum<? extends I>` uses different capture variables for the two occurrences.
Semantically equivalent at runtime; the type checker cannot unify them.

## Fix

Bridge with a private `@SuppressWarnings("unchecked")` helper that accepts the wildcard and
casts internally:

```java
@SuppressWarnings("unchecked")
private <T extends Enum<T> & VocabularyTerm> void registerRaw(
        Class<? extends Enum<? extends VocabularyTerm>> raw) {
    register((Class<T>) raw);
    // safe: any concrete enum satisfies both bounds at its declaration site
}
```

Call `registerRaw(r.vocabulary())` from `@PostConstruct init()` instead of
`register(r.vocabulary())`. The unchecked cast is safe because any real enum implementing
`VocabularyTerm` necessarily satisfies the recursive self-bound — the wildcard form just can't
express this to the type checker.

## Why this is non-obvious

The two type expressions are semantically identical at runtime. The compile error mentions
"CAP#1" capture variables and gives no hint that a private bridge is the standard fix. Java
generics documentation does not call out this specific wildcard-vs-self-bound failure mode.
