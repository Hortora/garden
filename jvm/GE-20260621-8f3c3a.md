### Symptom
`UnsupportedOperationException` when calling `CaseHubRuntime.signal(caseId, "pr.headSha", newSha)` — a sub-path signal that navigates into a nested map within the case context.

### Context
Occurs when the initial case context is constructed with `Map.of()` for a sub-map value:
```java
var prContext = Map.of("id", "42", "repo", "org/repo", "headSha", "abc123");
initialContext.put("pr", prContext);  // prContext is unmodifiable
caseHub.startCase(initialContext);
```

Top-level signals (`signal(caseId, "codeAnalysis", null)`) work fine — they operate on the mutable top-level `LinkedHashMap`. Only sub-path signals that navigate into nested maps fail.

### Root cause
`WritablePanelImpl(String, Map<String, Object>)` constructor does `this.data.putAll(initial)` — a shallow copy. The top-level `data` is a mutable `LinkedHashMap`, but VALUES (including sub-maps) retain their original references. When `setPath("pr.headSha", value)` navigates to `data.get("pr")`, it gets the original `Map.of()` instance, then calls `current.put("headSha", value)` on it — `UnsupportedOperationException`.

### Fix
Use mutable maps for all sub-map values in initial context:
```java
var prContext = new LinkedHashMap<>(Map.of("id", "42", "repo", "org/repo", "headSha", "abc123"));
initialContext.put("pr", prContext);
```

The engine-side fix (deep-copy in `WritablePanelImpl` constructor using the existing `deepCopyMap()` method) is tracked in engine#547.

### Why this is non-obvious
`putAll()` appears to copy the data. The unmodifiable nature of `Map.of()` is invisible at the call site — the type is `Map<String, Object>`, not `UnmodifiableMap`. The bug only surfaces on sub-path signals, which are uncommon until a harness implements context updates on running cases.

### Alternatives considered
- Signal the whole top-level key (`signal(caseId, "pr", entireNewPrMap)`) instead of sub-paths — avoids navigating into sub-maps but requires reconstructing the full map and loses atomicity of individual field updates.
