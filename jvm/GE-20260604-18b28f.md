## Symptom

Build fails with `KieMemoryCompilerException` inside a generated file:

```
org.kie.memorycompiler.KieMemoryCompilerException:
[io/quarkmind/plugin/drools/PA6/LambdaPredicateA685BF2DCD7428F516AE3214ED64CD15.java (23:17) : cannot find symbol]
```

The error gives zero signal about which rule is responsible or why.

## Context

A Drools Rule Unit DRL rule combines `eval()` on a plain RuleUnit field with `accumulate()` in the same `when` block:

```drl
rule "Strategy: Attack"
    salience 150
when
    eval(!timingAttackIncoming && !enemyPosture.equals("ALL_IN"))  // plain field
    accumulate(
        /army[ this.type() == UnitType.STALKER ];
        $count : count();
        $count >= 4
    )
then
    strategyDecisions.add("ATTACK");
end
```

`timingAttackIncoming` and `enemyPosture` are plain `boolean`/`String` fields on the `RuleUnitData` class (not `DataStore` fields).

## Root Cause

Drools generates a lambda class for the `eval()` expression. When `accumulate()` is also present in the rule, the code generator loses the reference to the RuleUnit's plain field scope when generating the lambda. The field name becomes unresolvable in the generated Java class.

## Fix

Replace the plain-field `eval()` condition with `DataStore` facts and use Drools pattern matching instead:

**StrategyRuleUnit:**
```java
private final DataStore<String>  postureStore = DataSource.createStore();
private final DataStore<Boolean> timingStore  = DataSource.createStore();
```

**Populate before fire():**
```java
data.getPostureStore().add(posture);
data.getTimingStore().add(timing);
```

**DRL rule (no eval, no field scope issue):**
```drl
rule "Strategy: Attack"
    salience 150
when
    not /timingStore[ this == true ]
    not /postureStore[ this.equals("ALL_IN") ]
    accumulate(
        /army[ this.type() == UnitType.STALKER ];
        $count : count();
        $count >= 4
    )
then
    strategyDecisions.add("ATTACK");
end
```

The `Boolean` alpha-index on `this == true` is correctly generated and indexed by PHREAK — confirmed by inspecting the generated rule class.

## Why This Is Non-Obvious

- `eval()` alone on plain fields compiles and runs correctly.
- `accumulate()` alone (without `eval()`) compiles and runs correctly.
- The combination fails, but only at build time with a generic `cannot find symbol` error that gives no indication it is caused by the `eval()` + `accumulate()` interaction.
- Nothing in the Drools or Quarkus documentation mentions this restriction.

### Why this fix

DataStore pattern matching is idiomatically correct for Drools Rule Units — plain fields are an acceptable shortcut for stateless fire-and-discard sessions, but DataStore facts are the intended mechanism. The fix aligns with the Rule Unit model and avoids eval() entirely, which also improves performance (PHREAK can alpha-index DataStore patterns; eval() cannot be indexed).

### Alternatives considered

- Pre-computing a combined `boolean readyToAttack` field and using `eval(readyToAttack)` + `accumulate()` — not tested; same lambda scoping issue likely applies to any `eval()` + `accumulate()` combination regardless of expression complexity.
- Splitting into two rules with a flag set in rule 1's consequence read by rule 2's condition — does not work; plain field writes in a rule consequence are not change events in the Rule Unit model and do not trigger agenda re-evaluation within the same `fire()` call (see GE-0109).
