## Symptom

A test named `structural_renderer_shows_label_not_raw_value` passes, but the raw vocabulary value still appears elsewhere in the rendered output via a different render path.

## Context

Testing that a structural renderer uses vocabulary labels instead of raw enum values:

```java
@Test
void structural_markdown_shows_axis_label_not_raw_value() {
    // ...render with vocab-resolved labels...
    assertThat(result.content()).contains("Independent (Conscientiousness Disposition Axes)");
    assertThat(result.content()).doesNotContain("- Social orientation: facilitative\n");
}
```

The test name promises "raw value is absent." The assertion only proves the old formatted bullet line is gone. If `"facilitative"` appears in a different format (prose section, summary, embedded in another label), the assertion passes silently.

## Root cause

The `doesNotContain` argument includes the structural label prefix (`"- Social orientation: "`) and the newline. It checks for the old *formatted output*, not for the raw value itself. Since the formatted string changed (now includes the resolved label), the assertion is trivially satisfied even when the raw value is still present.

## Fix

Assert on the bare raw value — not the formatted line:

```java
assertThat(result.content()).contains("Independent (Conscientiousness Disposition Axes)");
assertThat(result.content()).doesNotContain("facilitative");  // the raw value, not the old format
```

`"Independent"` has different casing from `"independent"`, so the positive and negative assertions are unambiguous. This directly proves the raw value is absent from the entire content, regardless of how it might appear.

## Why this is non-obvious

The `doesNotContain("- Social orientation: facilitative\n")` assertion looks correct in review — it matches the exact old output format that the test should negate. The problem is invisible unless you think about what other paths could render `"facilitative"` (prose fallback, summary section, A2A card). The test name promises more than the assertion delivers, and both look reasonable in isolation.

The fix is counter-intuitive: a *shorter*, less specific string is the *more precise* absence assertion.
