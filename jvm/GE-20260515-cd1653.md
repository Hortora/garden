### What you observe

`mvn test` fails with "Please configure the datasource URL or ensure the Docker daemon is up and running" even though Podman is installed and the Podman machine is running. The `docker` binary is not on PATH.

### Context

macOS development machine with Podman installed at `/opt/podman/bin/podman` and no Docker Desktop. Quarkus Dev Services uses Testcontainers under the hood to start database containers for `@QuarkusTest`.

### What was tried

Running `TESTCONTAINERS_RYUK_DISABLED=true mvn test` — this disables Ryuk but Testcontainers still can't find a container runtime because `docker` is not in PATH.

### Root cause

Testcontainers locates the container runtime via the `DOCKER_HOST` environment variable or by looking for the `docker` binary. Podman provides a Docker-compatible socket but `DOCKER_HOST` must be explicitly pointed at it.

### Fix

Set `DOCKER_HOST` to the Podman socket before running tests:

```bash
DOCKER_HOST=unix:///run/user/501/podman/podman.sock \
TESTCONTAINERS_RYUK_DISABLED=true \
mvn test -pl <module>
```

The socket path `/run/user/501/podman/podman.sock` uses the current user's UID (501 on this machine). Verify with:

```bash
podman info --format '{{.Host.RemoteSocket.Path}}'
```

### Why this is non-obvious

Podman advertises Docker compatibility but doesn't add itself to PATH as `docker`. Testcontainers documentation mentions `DOCKER_HOST` as a configuration option but the Podman socket path must be discovered separately. The error message points at the datasource (a Quarkus layer) rather than the missing container socket.
