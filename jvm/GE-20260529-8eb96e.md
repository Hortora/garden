## Symptom

A Quarkus `@ConfigMapping` key-mapping config entry silently fails. The actorId is not found in the map even though the property appears to be configured:

```properties
casehub.ledger.vault-transit.key-mapping."claude:reviewer@v1"=reviewer-key
```

The map is empty at runtime, or the value is not associated with the actorId.

## Context

`@ConfigMapping` with a `Map<String, String> keyMapping()` method. ActorId strings follow the convention `"model:persona@version"` (e.g. `claude:reviewer@v1`). Configured via `application.properties` in a Quarkus test or production config.

## Root cause

The Java `.properties` format defines **both `=` and `:` as key-value separators**. When the parser encounters:

```
casehub.ledger.vault-transit.key-mapping."claude:reviewer@v1"=reviewer-key
```

it sees `casehub.ledger.vault-transit.key-mapping."claude` as the key (splitting at the first `:`) and `reviewer@v1"=reviewer-key` as the value. The quotes are not treated as delimiters — they're literal characters.

This is specified in the Java `Properties` Javadoc (natural key and value forms) and in RFC 6570, but almost nobody reads it.

## Fix

Escape the colon with a backslash in `.properties` files:

```properties
casehub.ledger.vault-transit.key-mapping."claude\:reviewer@v1"=reviewer-key
```

Or switch to YAML config (`application.yaml`) where colons in quoted values are not treated as separators:

```yaml
casehub:
  ledger:
    vault-transit:
      key-mapping:
        "claude:reviewer@v1": reviewer-key
```

## Why this is non-obvious

The `=` separator is universally known. The `:` separator is specified in the Java Properties Javadoc and ISO spec but almost never encountered in practice (most property keys don't contain colons). The symptom — map entry silently missing — gives no hint about the cause. Developers suspect config mapping, CDI wiring, or key normalisation before suspecting `.properties` format rules.
