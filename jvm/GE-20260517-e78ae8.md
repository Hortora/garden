## Symptom

A `@Transactional` helper persists an entity and returns it. The caller mutates a field on
the returned object. The mutation is never persisted — a subsequent DB read returns the
field as `null` or its original value.

Manifests most commonly in `@QuarkusTest` setup methods:

```java
@Transactional
WorkItemTemplate persistTemplate(String name) {
    WorkItemTemplate t = new WorkItemTemplate();
    t.name = name;
    WorkItemTemplate.persist(t);
    return t;   // detached after method returns
}

@Test
void someTest() {
    WorkItemTemplate t = persistTemplate("Loan Approval");
    t.defaultPayload = "{\"type\":\"loan\"}";   // silently ignored — entity is detached
    // ...
    assertThat(workItem.payload).isEqualTo("{\"type\":\"loan\"}");   // FAILS — null
}
```

## Root Cause

When a `@Transactional` method commits on return, the JPA persistence context closes.
The returned entity is detached — no longer managed by the EntityManager. Mutations on
detached entities are silently ignored; there is no exception, no warning.

## Fix

Set all fields before the transaction commits — inside the `@Transactional` method,
before `persist()` or before the method returns:

```java
@Transactional
WorkItemTemplate persistTemplate(String name, String defaultPayload) {
    WorkItemTemplate t = new WorkItemTemplate();
    t.name = name;
    t.defaultPayload = defaultPayload;   // set inside the transaction
    WorkItemTemplate.persist(t);
    return t;
}
```

If dynamic post-persist mutation is genuinely required, open a new transaction:

```java
@Transactional
void setPayload(UUID id, String payload) {
    WorkItemTemplate t = WorkItemTemplate.findById(id);
    t.defaultPayload = payload;
    // dirty-checked on flush — no explicit merge needed
}
```

## Why This Is Non-Obvious

The mutation looks correct. The entity object is live in the caller's scope, the field
is set, no exception is thrown. The disconnect only surfaces when a subsequent operation
reads from the DB and the field is absent. The symptom (`null` from DB) points at the
read path, not at the write that was silently ignored.

In test code the pattern `persist() → return → mutate` is natural — it resembles
builder-style setup. The `@Transactional` boundary is invisible at the call site.

## Why This Fix

Setting fields before persist is unambiguous. The alternative (opening a second
transaction to mutate) requires a separate method, an extra inject, and makes the
test setup more complex for no benefit. The constraint is simply: all state that
belongs in the DB must be set before the transaction closes.
