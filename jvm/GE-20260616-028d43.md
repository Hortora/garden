### Symptom

A test tries to hold a session in ACTIVE state by keeping a cold Multi unsubscribed, then checks that a concurrent call fails. But when written as:

```java
final var sub = session.query("active").subscribe().with(x -> {});
assertThatIllegalStateException().isThrownBy(() -> session.query("second"));
```

The second `query()` does not throw — the session is already back to IDLE. The test passes trivially.

### Context

Any code that uses an `AtomicReference` state machine where state transitions happen in two phases:
1. CAS at call time (method entry): IDLE → ACTIVE
2. Completion handler resets state (on subscribe completion): ACTIVE → IDLE

The "ACTIVE window" lives between the CAS and the completion handler. For `Multi.createFrom().empty()`, that window collapses to near-zero because subscription triggers immediate completion on the subscribing thread.

### Root cause

`Multi.createFrom().empty()` is an eager, cold Multi that completes synchronously when subscribed. The chain:

```
session.query("active")    // CAS IDLE→ACTIVE; returns cold Multi
  .subscribe().with(...)   // subscribes → Multi completes immediately → onCompletion → IDLE
// by here, state is IDLE again
session.query("second")    // succeeds — state is IDLE, no ISE thrown
```

The ACTIVE window closes before the next line executes. Subscribe and completion are both synchronous on the calling thread.

### Fix

Do not subscribe to the first Multi before asserting the concurrent call fails. The state is ACTIVE between `query()` returning the cold Multi and any subscription to it:

```java
// Hold the cold Multi without subscribing — state is ACTIVE
final var firstTurn = session.query("active");
assertThatIllegalStateException()
    .isThrownBy(() -> session.query("second"))
    .withMessageContaining("a turn is already active");
// Subscribe after — completes immediately
firstTurn.collect().asList().await().indefinitely();
```

### Why this is non-obvious

Mutiny Multis are lazy (cold). Developers expect a cold Multi to be "pending" until subscribed. The assumption: subscribing with a no-op handler leaves the stream in an indefinitely running state. In reality, `Multi.createFrom().empty()` completes immediately on subscription — the "lazy" only means "don't start until subscribed", not "run forever". There is no ongoing async activity to keep it active.
