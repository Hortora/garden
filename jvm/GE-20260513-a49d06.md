## CDI this.method() call bypasses @Transactional proxy — annotation silently dead on delegating overloads

**ID:** GE-20260513-a49d06
**Stack:** Quarkus 3.x, ArC CDI, Java 21
**Symptom:** `@Transactional` on the inner of two delegating overloads has no effect. No error. Runtime behaviour is correct (the outer transaction covers everything), but the inner annotation is dead code that misleads future direct callers.
**Context:** A delegating overload pattern — a 4-arg method calls `this.instantiate(5-arg, null)` to supply a default. Both methods are annotated `@Transactional`. The 5-arg's annotation silently does nothing when reached via the delegation.

### What was tried (didn't work)
- Annotating both overloads with `@Transactional` — looks correct, annotation on the inner method is ignored
- Trusting that `@Transactional` on a method in an `@ApplicationScoped` bean always fires — it does, but only via the CDI proxy (external calls)

### Root cause
Quarkus ArC generates a CDI proxy subclass that wraps methods with interceptor chains. `@Transactional` is one such interceptor. Interceptors only fire when the call enters through the proxy — i.e., an external caller invokes the method on the injected bean reference.

When you call `this.method()` from within the bean, you bypass the proxy entirely and call the real instance directly. The interceptor chain for the inner method never executes. The outer method's transaction is already active and covers the inner call correctly — so runtime behaviour is correct — but the annotation on the inner method is dead code.

### Fix
Annotate only the outermost entry point. Remove `@Transactional` from the inner delegated method, or add a comment:

```java
@Transactional  // active only when called directly via CDI proxy
public WorkItem instantiate(WorkItemTemplate template, String titleOverride,
        String assigneeIdOverride, String createdBy, String callerRef) { ... }
```

If the inner method genuinely needs its own transaction boundary for direct external callers, use self-injection to force proxy routing:

```java
@Inject
MyService self;  // injects the CDI proxy, not `this`

public WorkItem outerMethod(...) {
    return self.innerMethod(...);  // interceptors fire
}
```

### Why this fix
Self-injection is the only way to force proxy interception on an intra-bean call. Most developers annotate both methods and assume both annotations fire — removing the redundant annotation from the inner method and adding a comment is the minimal correct fix that eliminates the false guarantee.

### Alternatives considered
- Annotate only the 5-arg (inner) and remove from 4-arg — wrong: the outer annotation is the one that actually fires
- Use `@Inject MyService self` self-injection — correct, but adds a circular dependency warning and complexity; only needed if the inner method must have its own independent transaction boundary

### Why this is non-obvious
Both methods appear correctly annotated. The runtime behaviour IS correct — no data corruption, no missing transaction. The failure mode is future: a developer who reads the code, sees `@Transactional` on the inner method, and calls it directly without a surrounding transaction gets no protection they were led to expect. The annotation creates a false contract that the CDI proxy never enforces.
