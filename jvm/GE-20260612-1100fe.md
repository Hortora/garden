## mvn test -pl <child> passes while installed runtime jar hides a cross-module API break

**ID:** GE-20260612-1100fe
**Stack:** Maven 3.x, multi-module project
**Symptom:** `mvn test -pl connector-backend` reports BUILD SUCCESS after a library updated a method signature (e.g. `tokenise(String)` → `tokenise(String, ActorType)`). The call sites in the sibling `runtime` module still use the old one-argument signature. No compile error. The issue is invisible until `mvn clean install` from the project root.
**Context:** A library dependency (casehub-ledger) changed a method signature. A sibling module (qhorus/runtime) calls that method and needs updating. The child module (connector-backend) depends on `casehub-qhorus` (the sibling's artifact) — but resolves it from the `.m2` cache, which still holds the pre-API-change compiled `.jar`. Maven compiles connector-backend against the stale jar without error. The runtime module's source now has a compile-time mismatch, but that mismatch is invisible because the module was not compiled during the `-pl` run.

### What was tried (didn't work)
- Running `mvn test -pl connector-backend` — passed with BUILD SUCCESS
- Checking for error messages — none appeared
- Assumed the API propagation was complete because tests passed

### Root cause
`mvn test -pl <child>` resolves sibling modules from `~/.m2/repository`, not from the source tree. The installed sibling jar was compiled before the API change. The child module compiles against that stale jar successfully. The compile-time break in the sibling's source is never reached during a `-pl` build.

Maven's incremental compilation does not cross module boundaries. A source-level API break in one module only surfaces when that module is explicitly compiled in the same reactor invocation.

### Fix
```bash
# Detect API breaks across the full reactor
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn clean install -DskipTests
```

Run from the project root. `clean` discards stale `.class` files; `install` installs freshly compiled artifacts into `.m2`, making them available to sibling modules in the same reactor invocation.

`mvn clean compile` from the root also works for detection, but `install` is needed to propagate the fix to downstream consumers.

### Why this is non-obvious
Developers running `mvn test -pl <child>` to iterate quickly assume "tests pass → code is correct." They don't expect the passing build to reflect a stale version of a sibling module. The error manifests as a CI failure or a `NoSuchMethodError` at runtime, usually in a different module and much later.

The stale jar is especially hard to notice when the API change is in a library that was recently bumped (not a local module) — the library's new signature is in the freshly installed library jar, but the qhorus/runtime module's source still uses the old one-arg call.
