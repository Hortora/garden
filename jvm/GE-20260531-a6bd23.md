## SmallRye Config rejects @WithDefault("") for plain String @ConfigMapping methods — use Optional<String> instead

### Symptom

Adding `@WithDefault("")` to a `String` method in a `@ConfigMapping` interface causes Quarkus
to fail at startup with:

```
SRCFG00040: The config property casehub.ledger.agent-identity.scim.endpoint
  with the config value '' was converted with exception: No Converter registered for type java.lang.String
  (or similar: empty string not convertible to required type)
```

All `@QuarkusTest` tests that don't provide the config key then fail with "Failed to start Quarkus."

### Root cause

SmallRye Config treats an empty string as equivalent to `null` for non-Optional target types.
When `@WithDefault("")` provides an empty value and no actual config key is set, SmallRye
tries to convert the empty string to a `String` instance and fails with SRCFG00040 — even
though empty string IS a valid `String`.

### Fix

Change the return type from `String` to `Optional<String>`:

```java
// WRONG — causes SRCFG00040 when key is absent
@WithDefault("")
String endpoint();

// CORRECT — Optional is absent when key is not configured
Optional<String> endpoint();
```

Callers unwrap with `.orElse("")` or `.orElseThrow(...)` as appropriate.

### Why this is non-obvious

`@WithDefault("")` looks semantically correct — "if not set, use empty string." For
`int`, `boolean`, and most non-String types, empty-string defaults work fine.
The failure specifically with `String` is counterintuitive because empty string IS a
valid String. The error message (SRCFG00040, "No Converter registered") also does not
point to the real problem — it sounds like a missing converter, not a null-coercion issue.

The failure only surfaces at startup when the key is absent and another `@QuarkusTest`
context starts — a subtle regression introduced when adding new optional config keys.
