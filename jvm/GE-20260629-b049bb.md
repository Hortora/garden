### Symptom

A mesh mesh prompt template produces `register("null", "Starting agent", ["agent"])` instead of a meaningful identifier. The agent receives instructions with a literal string `"null"` as its registration ID.

### Context

`MeshSystemPromptTemplate.buildActive()` concatenates `workerId` directly into a registration instruction string. The engine's `CaseContextChangedEventHandler.tryProvision()` passes `workerId=null` to `buildContext()` because the worker's session ID isn't generated until later in the provisioning pipeline (`setupSession()` creates `sessionId = UUID.randomUUID()`).

This was harmless for months because the generated mesh prompt was stored in `WorkerContext.properties("systemPrompt")` but never actually delivered to the CLI — the provisioner ignored `WorkerContext` entirely. When delivery was enabled (claudony#163), the null coercion became a real bug.

### Root cause

Java's string concatenation operator (`+`) silently converts `null` references to the literal string `"null"`. There is no compile-time warning, no runtime exception, and no obvious signal that the output is wrong. The template appeared to work correctly in all tests because tests always provided a non-null workerId.

The deeper pattern: enabling a previously-dead output path exposes every silent null→"null" coercion in the template chain. Any parameter that was "always provided in practice" but nullable in the type system may have been null in the dead path all along.

### Fix

Null-guard at the concatenation site, not the caller:

```java
private static String registrationInstruction(String workerId, String capability) {
    String id = workerId != null ? "\"" + workerId + "\"" : "<your-session-id>";
    return "  1. register(" + id + ", \"Starting " + capability + "\", [\"" + capability + "\"])\n";
}
```

When `workerId` is null, the template emits a placeholder (`<your-session-id>`) that the agent can resolve at runtime. The fix is at the template level because the caller (the engine) has a legitimate reason to not have the ID yet — the provisioning pipeline generates it later.

### Why this is non-obvious

The bug was invisible for two reasons: (1) Java silently converts null to `"null"` without any signal, and (2) the template output was never consumed. A developer reading the template code sees a complete, working string concatenation — there's no indication that the result contains a literal `"null"` because the `+` operator never fails. The bug only surfaces when the output path becomes live, which may happen in a completely different change (enabling delivery) with no obvious connection to the template code.

### Why this fix

Guarding at the concatenation site (the template) rather than the caller (the engine) is correct because: (1) the engine has a valid architectural reason for null — the workerId doesn't exist yet at context-build time; (2) fixing the caller would require restructuring the provisioning pipeline to generate IDs earlier, which is unnecessary complexity; (3) the template is the component that knows what a missing workerId means in context and can produce a meaningful placeholder.
