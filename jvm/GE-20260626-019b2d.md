### Symptom

You have a line-based TCP protocol (IRC, SMTP, FTP) where the client sends a command and must wait for a specific reply before continuing. The read loop runs on a dedicated thread; calling threads need to block until their reply arrives.

### Context

IRC `JOIN #channel` must wait for `353` (NAMES) + `366` (end of names) before returning. `LIST` must collect `322` replies until `323` (list end). Multiple operations may be in flight if different callers invoke `join()` and `listChannels()` concurrently.

### What was tried (didn't work)

State machine approach — tracking "current pending operation" and routing replies accordingly. Breaks when multiple operations are in flight (e.g. two threads calling `join()` on different channels simultaneously). Requires complex state transitions and is error-prone.

### Root cause

The challenge is correlating async replies to the calling thread that sent the request, without coupling the read loop to specific operations.

### Fix

Register a `CompletableFuture` before sending the command, keyed by reply code + discriminator:

```java
// Before sending JOIN #test
CompletableFuture<List<String>> future = new CompletableFuture<>();
namesCollectors.put("#test", new NamesCollector(future, new ArrayList<>()));
writeLine("JOIN #test");

// Read loop routes replies by key
case "353": // RPL_NAMREPLY
    NamesCollector c = namesCollectors.get(channel);
    if (c != null) c.nicks.addAll(parseNicks(params));
    break;
case "366": // RPL_ENDOFNAMES
    NamesCollector c = namesCollectors.remove(channel);
    if (c != null) c.future.complete(c.nicks);
    break;

// Caller blocks with timeout
List<String> nicks = future.get(10, TimeUnit.SECONDS);
```

Key design points:
- **Keyed by command+channel** (e.g. `"366:#test"`) — supports concurrent operations on different channels
- **Accumulator pattern** — for multi-reply responses (LIST collecting 322s), the collector holds a list that the read loop populates; the future completes with the final accumulated result
- **Timeout** — `future.get(10, SECONDS)` prevents indefinite blocking
- **Cleanup on disconnect** — `failAllCollectors()` completes all pending futures exceptionally with `IOException("connection lost")` when the read loop exits, preventing threads from hanging the full timeout

The `CompletableFuture.complete()` call establishes a happens-before relationship, so the accumulated data is safely published to the waiting thread without additional synchronization.

### Why this is non-obvious

Most developers confronted with "wait for a reply on a different thread" reach for a state machine, synchronized blocks with wait/notify, or a blocking queue. The CompletableFuture-as-collector pattern is simpler, naturally supports concurrent operations via keyed lookup, and leverages the JMM happens-before guarantee of `complete()` for safe data publication — no explicit synchronization needed between the read loop and the caller.
