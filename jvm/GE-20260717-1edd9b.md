### Context
Multiple modules persist derived resources keyed by a parent entity (e.g. situation → ganglion state in JPA, situation → Drools session in H2MVStore). When the parent is bulk-deleted, derived resources become orphans. Each store uses a different orphan detection mechanism — SQL `DELETE WHERE NOT EXISTS` for JPA, key iteration + existence check for KV stores. A direct call per store from the scheduler couples it to every storage module.

### The technique
Define a generic SPI interface in the shared API module:

```java
public interface OrphanedResourceCleaner {
    String cleanerType();
    Uni<Integer> removeOrphaned();
}
```

Each store implements it with its own orphan detection logic. The scheduler discovers all implementations via `Instance<OrphanedResourceCleaner>` and iterates with per-cleaner error isolation:

```java
@Inject Instance<OrphanedResourceCleaner> resourceCleaners;

for (OrphanedResourceCleaner cleaner : resourceCleaners) {
    try {
        int cleaned = cleaner.removeOrphaned().await().indefinitely();
        metrics.orphanedResourcesCleaned(cleaned, cleaner.cleanerType());
    } catch (Exception e) {
        log.warnf(e, "Orphan cleaner '%s' failed, skipping", cleaner.cleanerType());
    }
}
```

### Why this is non-obvious
The natural approach is to add `removeOrphaned()` to each store interface and call them individually from the scheduler. This works initially but doesn't scale — each new store type requires a new injection + call in the scheduler, and the scheduler accumulates dependencies on optional modules. The `Instance<SPI>` pattern makes the scheduler agnostic to which cleaners exist — optional modules activate by classpath presence alone.

### Alternatives considered
- **Direct injection per store** — couples scheduler to every storage module; breaks when stores are optional
- **CDI event (`Event<CleanupEvent>`)** — fire-and-forget but loses the return count for metrics; observer exceptions propagate differently
- **Adding to the domain SPI (e.g. Ganglion interface)** — domain objects aren't always CDI beans (some are created programmatically); CDI discovery wouldn't find them
