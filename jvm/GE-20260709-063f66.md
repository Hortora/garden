### Symptom

Calling `updateCollectionAsync` with `setSparseVectorsConfig` to add a new sparse vector to an existing collection fails with:

```
INVALID_ARGUMENT: Wrong input: Not existing vector name error: sparse
```

### Context

When evolving a CBR collection schema â€” enabling SPLADE sparse embeddings on a collection that was originally created with only dense vectors. The `UpdateCollection` proto has a `sparse_vectors_config` field (field 8) with type `SparseVectorConfig`, which is a `map<string, SparseVectorParams>`. The field name and type strongly suggest it should support adding new sparse vectors.

### What was tried (didn't work)

```java
client.updateCollectionAsync(UpdateCollection.newBuilder()
    .setCollectionName(collection)
    .setSparseVectorsConfig(SparseVectorConfig.newBuilder()
        .putMap("sparse", SparseVectorParams.getDefaultInstance())
        .build())
    .build()).get();
```

This fails because `updateCollection` with `sparse_vectors_config` only modifies parameters of **existing** sparse vectors â€” it cannot create new ones. The API name is misleading.

### Root cause

Qdrant's `updateCollection` gRPC endpoint validates that all vector names in the update request already exist in the collection. There is no separate "add vector" API â€” the collection schema for vector fields is immutable after creation.

### Fix

Delete and recreate the collection with the full vector config (dense + sparse + BM25). This follows the same pattern as dimension migration â€” data is lost and must be repopulated via reconciliation.

```java
// In ensureCollection(), after detecting missing sparse vectors:
if (hasMissingSparseVectors(info)) {
    LOG.warning("Collection " + collection
        + " missing required sparse vectors â€” recreating.");
    client.deleteCollectionAsync(collection).get();
    // Falls through to creation path, which includes sparse vectors
}

// Creation path always includes configured sparse vectors:
if (config.spladeEnabled() || config.bm25Enabled()) {
    SparseVectorConfig.Builder sparseBuilder = SparseVectorConfig.newBuilder();
    if (config.spladeEnabled()) {
        sparseBuilder.putMap(config.spladeVectorName(),
            SparseVectorParams.getDefaultInstance());
    }
    if (config.bm25Enabled()) {
        sparseBuilder.putMap(config.bm25VectorName(),
            SparseVectorParams.newBuilder().setModifier(Modifier.Idf).build());
    }
    createBuilder.setSparseVectorsConfig(sparseBuilder.build());
}
```

### Why this is non-obvious

The `UpdateCollection` proto exposes `sparse_vectors_config` as a top-level field alongside `vectors_config` (dense), `quantization_config`, etc. The field type (`SparseVectorConfig` with a map) looks identical to the creation-time field. Nothing in the API, proto comments, or error message hints that "update" means "modify existing only" rather than "upsert". The error message says "Not existing vector name" â€” which reads as a bug rather than a design constraint.

### Why this fix

Delete-and-recreate is the only option when the Qdrant API doesn't support adding vector fields. An alternative would be to always create collections with all possible sparse vectors upfront (even when disabled), but that wastes storage and complicates the schema for features that may never be enabled. Delete-and-recreate is consistent with the existing dimension migration pattern and leverages the reconciliation service for data recovery.

### Alternatives considered

- **Always create collections with sparse vectors** â€” rejected because it pre-allocates schema for features that may never be enabled, and empty sparse vectors still consume index overhead
- **Use REST API instead of gRPC** â€” same limitation; the restriction is server-side, not client-side
