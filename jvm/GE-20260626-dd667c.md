### Symptom

A Quarkus application starts, logs "Listening on http://..." and all installed features, then immediately prints `FAIL: model directory argument required` to stderr and the CDI container dies. In prod mode the process exits; in dev mode the HTTP server stays listening but returns 500 on every request with "ArC container not initialized: already shut down."

### Context

A library module (`casehub-inference-quarkus`) contained a `NativeImageGateCommand` class annotated `@QuarkusMain`. This class was a one-off verification tool for ONNX Runtime JNI loading in native image builds. When any Quarkus application added `casehub-inference-quarkus` as a dependency, Quarkus discovered the `@QuarkusMain` annotation during augmentation and made `NativeImageGateCommand` the application's main entry point — replacing the normal Quarkus HTTP server lifecycle.

The gate command expected a model directory as a CLI argument. Normal application startup passes no arguments, so the command printed "FAIL" and returned exit code 1, killing the process.

### Root cause

Quarkus scans ALL JARs on the classpath for `@QuarkusMain`. If a library JAR contains `@QuarkusMain`, it becomes the application's entry point — silently overriding whatever the consuming application intended. There is no warning, no conflict detection, and no way for the consuming application to know a library has claimed its main class.

### Fix

Remove `@QuarkusMain` from the library class. The class can remain as a standalone utility invokable via `QuarkusApplication` directly, but the annotation must not be present on a class shipped in a library JAR.

```java
// ❌ BAD: @QuarkusMain in a library — hijacks every consumer
@QuarkusMain
public class NativeImageGateCommand implements QuarkusApplication { ... }

// ✅ GOOD: no annotation — consumers are not affected
public class NativeImageGateCommand implements QuarkusApplication { ... }
```

### Why this is non-obvious

The failure mode is completely misleading. The error message ("model directory argument required") has nothing to do with the consuming application's domain. The application appears to start normally (Quarkus logs, features list, port binding), then dies with an unrelated error from a class the developer has never seen. There is no log line saying "Using @QuarkusMain from library X" — the hijack is silent.

### Why this fix

The alternative — adding `@QuarkusMain` to the consuming application to override the library's — only masks the problem. The library should never claim the entry point in the first place.

### Alternatives considered

- Adding `@QuarkusMain(name = "...")` to scope the command — doesn't prevent hijacking; Quarkus still discovers it
- Using `@IfBuildProfile("native-gate")` — build profiles are build-time, not runtime; the annotation is resolved during augmentation regardless
