### Symptom

After deep-copying a `Map<String, Object>`, mutating a `Map` that was inside a `List` value of the original modifies the copy, and vice versa. The snapshot is corrupted without any exception.

### Context

Any deep-copy utility that recursively processes a `Map<String, Object>` where values may be `Map`, `List`, or both nested arbitrarily. Common in serialization shims, context snapshots, and blackboard state copies.

### What was tried (didn't work)

```java
// Naive — only handles top-level Maps
private static Map<String, Object> deepCopy(Map<String, Object> source) {
    Map<String, Object> copy = new LinkedHashMap<>();
    for (var e : source.entrySet()) {
        Object v = e.getValue();
        if (v instanceof Map) v = deepCopy((Map<String, Object>) v);
        else if (v instanceof List) v = new ArrayList<>((List<?>) v); // WRONG — shallow
        copy.put(e.getKey(), v);
    }
    return copy;
}
```

A list like `workers: [{name: "extractor", runs: 2}]` is copied as `new ArrayList<>(workers)` — the list is new but the map entries inside it point to the same objects.

### Root cause

`new ArrayList<>(list)` copies list element references, not list elements. For lists of scalars (`String`, `Integer`) this is safe because Java scalars are immutable. For lists of maps, the map references are shared between the original and copy.

### Fix

Recursively deep-copy `Map` elements inside lists:

```java
@SuppressWarnings("unchecked")
private static Map<String, Object> deepCopy(Map<String, Object> source) {
    Map<String, Object> copy = new LinkedHashMap<>();
    for (var e : source.entrySet()) {
        Object v = e.getValue();
        if (v instanceof Map) {
            v = deepCopy((Map<String, Object>) v);
        } else if (v instanceof List<?> list) {
            List<Object> listCopy = new ArrayList<>(list.size());
            for (Object item : list) {
                listCopy.add(item instanceof Map ? deepCopy((Map<String, Object>) item) : item);
            }
            v = listCopy;
        }
        copy.put(e.getKey(), v);
    }
    return copy;
}
```

### Why this is non-obvious

The recursion handles Maps correctly — every developer checks for nested maps. Lists feel like a terminal case because `new ArrayList<>(other)` is the standard copy idiom and works correctly for lists of scalars (immutable). The failure only manifests with lists-of-maps, which are a common structure in JSON-derived data (arrays of objects) but not always present in unit tests that only use simple scalar values.
