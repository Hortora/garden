### Symptom
`SC2BotAgent` crashes with an exception inside `StartRaw.from()` when receiving GameInfo from an emulated server. No useful error message — just an index/division failure deep in the ocraft grid parsing code.

### Context
Building an emulated SC2 server that constructs minimal protobuf stubs for `GameInfo` responses. The `StartRaw` message requires five sub-fields (`mapSize`, `pathingGrid`, `terrainHeight`, `placementGrid`, `startLocations`), all validated with `orElseThrow`. The natural minimal stub is a 1×1 grid.

### Root cause
Ocraft's grid parsing uses integer division on grid dimensions to compute byte offsets for 1-bit-per-pixel (1bpp) packed grids. A 1×1 grid (1 pixel, 1 byte for 8bpp formats) causes integer division to produce 0 for the row stride in 1bpp mode, breaking the parser. The `placementGrid` uses 1bpp encoding; `terrainHeight` and `creepGrid` use 8bpp.

### Fix
Use 8×1 (8 pixels wide, 1 row) for 1bpp grids (`placementGrid`). This produces a 1-byte packed row (8 bits / 8 = 1 byte), which satisfies the integer division. Use 1×1 for 8bpp grids (`terrainHeight`, `creepGrid`) — these are unaffected.

```java
// 1bpp grid — minimum 8 pixels wide
byte[] placementBytes = new byte[]{(byte) 0xFF};
ImageData placementGrid = ImageData.of(8, 1, 1, ByteString.copyFrom(placementBytes));

// 8bpp grid — 1×1 is fine
byte[] heightBytes = new byte[]{(byte) 128};
ImageData heightGrid = ImageData.of(1, 1, 8, heightBytes);
```

### Why this is non-obvious
The SC2 protobuf schema defines `ImageData` with `width`, `height`, `bits_per_pixel`, and `data` — no minimum size constraints. The 1×1 stub is schema-valid. The integer division edge case is in ocraft's Java parsing code, not in the protobuf definition or any SC2 documentation. The crash only surfaces when the grid is actually parsed by `StartRaw.from()`, not at protobuf construction time.
