### What
`@ApplicationScoped` CDI beans in Quarkus are **always active** regardless of which thread
calls them. There is no `ContextNotActiveException` risk from background threads, game-loop
threads, or any non-request thread. The application context is the entire JVM lifetime.

### Context where this matters
When passing a CDI bean to a third-party framework that calls its methods from a background
thread (e.g. ocraft's `S2Agent.onStep()`, ocraft's `S2Agent.onGameStart()` called from the
game loop), developers often fear `ContextNotActiveException` and reach for manual lifecycle
management or thread-local propagation.

```java
@IfBuildProfile("sc2")
@ApplicationScoped
public class SC2BotAgent extends S2Agent {

    @Inject IntentQueue     intentQueue;     // always-active — safe from ocraft's thread
    @Inject TerrainProvider terrainProvider; // always-active — safe from ocraft's thread

    @Override
    public void onStep() {
        // Called by ocraft's background game loop — no context issues
        GameState state = ObservationTranslator.translate(observation());
    }
}
```

### Why this is non-obvious
Quarkus's `ContextNotActiveException` is a real problem for `@RequestScoped` and
`@SessionScoped` beans when called outside their activation lifecycle. This causes developers
to treat all CDI scopes as potentially thread-unsafe. The Quarkus docs mention that
`@ApplicationScoped` is the "widest scope" but do not explicitly state that it is immune to
`ContextNotActiveException` regardless of calling thread.

The distinction: `@RequestScoped`/`@SessionScoped` are activation-dependent; `@ApplicationScoped`
is activation-independent (the application context starts at boot and ends at shutdown).
