## Strengthen cache-eviction integration tests: change the stub, assert the new value — not just call count

### Context

Testing CDI event-driven cache invalidation with a `@QuarkusTest`: a rotation event
should evict a SCIM provider's cache so the next `didFor()` call fetches fresh data
from WireMock.

### The weak assertion (misses enricher side-effects)

```java
int callsBefore = wm.findAll(getRequestedFor(...)).size();
keyRotationService.recordRotation(actorId, ...); // fires CDI event, also triggers enricher
assertThat(wm.findAll(...).size()).isGreaterThan(callsBefore);
```

This assertion passes even if the cache was NOT evicted — the enricher pipeline inside
`recordRotation` makes its own SCIM call as a side-effect of `ledgerRepo.save()`, incrementing
the call count regardless of whether the explicit `didFor()` post-rotation hits the cache or
makes a fresh call.

### The strong assertion

After the invalidation trigger, update the stub to return different data, then assert
the new data is returned:

```java
// 1. Seed the cache with original data
assertThat(actorDIDProvider.didFor(ACTOR_ID)).contains(ORIGINAL_DID);

// 2. Change the stub — source-of-truth now returns a different value
wm.resetAll();
wm.stubFor(get(...).willReturn(aResponse().withBody(responseWith(UPDATED_DID))));

// 3. Trigger invalidation (CDI event, rotation, TTL expiry, etc.)
keyRotationService.recordRotation(ACTOR_ID, ...);

// 4. Assert the new value is returned — proves cache was evicted
assertThat(actorDIDProvider.didFor(ACTOR_ID)).contains(UPDATED_DID);
// If ORIGINAL_DID is returned instead → cache was NOT evicted
```

If the cache was NOT evicted, `ORIGINAL_DID` is returned and the assertion fails, making
the problem immediately visible. No call-count ambiguity.

### Why this is non-obvious

The naive call-count approach (`verify(moreThanOrExactly(2), ...)`) seems correct but
leaks through enricher/interceptor side-effects that hit the same endpoint independently
of the operation under test. The stub-swap technique is a standard pattern in HTTP mock
testing but developers often default to call-count assertions because they're simpler to write.
