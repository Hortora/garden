## Hibernate 6 JPQL arithmetic on MIN/MAX aggregates in HAVING is silently accepted by H2 but rejected by PostgreSQL

### Symptom

A JPQL query using arithmetic on `MIN()` / `MAX()` aggregate results in a `HAVING` clause
throws at runtime on PostgreSQL:

```
org.hibernate.query.SemanticException: Operand of - is of type 'java.lang.Object'
which is not a numeric type (it is not an instance of 'java.lang.Number',
'java.time.Temporal', or 'java.time.TemporalAmount')
```

The same query works silently on H2 — all H2-backed `@QuarkusTest` pass green, giving
false confidence. The failure only appears when the query executes against PostgreSQL.

### Context

Typical pattern in a health-check or audit job:

```jpql
SELECT e.subjectId, e.tenancyId, COUNT(e), MIN(e.sequenceNumber), MAX(e.sequenceNumber)
FROM LedgerEntry e
GROUP BY e.subjectId, e.tenancyId
HAVING COUNT(e) != MAX(e.sequenceNumber) - MIN(e.sequenceNumber) + 1
```

`sequenceNumber` is declared as `int` (Java primitive). Hibernate 6's semantic checker
classifies `MIN(int field)` and `MAX(int field)` as `Object` rather than a numeric type
when they appear as arithmetic operands in a `HAVING` clause — but only when the
PostgreSQL dialect is active.

The query executes at job runtime (e.g. on a 1h scheduler interval), not at startup,
so the failure surfaces hours into production operation.

### Root cause

Hibernate 6 rewrote JPQL type resolution. Aggregate functions in `HAVING` arithmetic
are typed differently depending on the active dialect's type resolution path. The H2
dialect resolves `MIN`/`MAX` return types as numeric for arithmetic, while the
PostgreSQL dialect's path leaves them as `Object` — triggering the semantic checker.

This is a regression from Hibernate 5 where both dialects accepted the arithmetic.

### Fix

**Remove the arithmetic from HAVING. Filter in Java.**

Drop the `HAVING` clause entirely and filter in the result loop:

```jpql
-- Query (no HAVING):
SELECT e.subjectId, e.tenancyId, COUNT(e), MIN(e.sequenceNumber), MAX(e.sequenceNumber)
FROM LedgerEntry e
GROUP BY e.subjectId, e.tenancyId
```

```java
// Filter in Java — arithmetic is safe here:
for (SubjectSequenceStats s : repo.findSequenceStats()) {
    final long expected = (long) s.max() - s.min() + 1;
    if (s.count() == expected) continue;
    // anomaly detected
}
```

**Also: convert to `@NamedQuery` for startup validation.**

The query that failed did so at runtime. Had it been a `@NamedQuery` on the entity class,
Hibernate would have thrown the `SemanticException` at startup — not after the first
scheduler firing. Converting inline `em.createQuery()` calls to `@NamedQuery` turns a
1-hour-delayed runtime failure into an immediate boot failure:

```java
@NamedQuery(
    name = "LedgerEntry.findSequenceStats",
    query = """
        SELECT NEW com.example.SubjectSequenceStats(
            e.subjectId, e.tenancyId, COUNT(e), MIN(e.sequenceNumber), MAX(e.sequenceNumber)
        )
        FROM LedgerEntry e
        GROUP BY e.subjectId, e.tenancyId
        """)
@Entity
public abstract class LedgerEntry { ... }
```

The `NEW` constructor expression passes aggregate values as constructor arguments — no
arithmetic in the query — so the type checker does not fire.

### Why this is non-obvious

JPQL semantic analysis is supposed to be dialect-independent — it validates the JPQL
AST before any SQL is generated. The natural assumption is that if H2 tests pass, the
query is semantically valid. This assumption breaks here: H2's dialect contributes to
the type resolution path, and H2 resolves aggregate arithmetic types differently than
PostgreSQL does.

The failure mode is also severe: with a 1h scheduler interval and H2 tests giving a
false green, the bug can reach production undetected and remain invisible until the
first health-check execution.

### Why this fix

Removing HAVING arithmetic is preferable to using JPQL `CAST()` because:
1. It eliminates the type-resolution dependency entirely — no arithmetic means no type checking issue
2. The arithmetic is usually already computed in Java for display/logging purposes
3. It is universally portable across all Hibernate dialects

`CAST(MAX(e.sequenceNumber) AS Long)` would also work for the PostgreSQL dialect, but
it introduces a dialect-specific JPQL extension that may not be supported in all versions
of H2 or future dialects.
