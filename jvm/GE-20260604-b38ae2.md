## Symptom

Validation placed in a Java record compact constructor runs correctly on the blocking service path but is silently skipped on the reactive path — the record is never constructed, so the validator never runs. Invalid state is stored in the database and only surfaces at a later operation (e.g. dispatch) as a runtime exception rather than at creation time.

## Context

A Quarkus application has both a blocking service (`ChannelService`) and a reactive service (`ReactiveChannelService`) for the same domain operation (channel creation). Validation — including type name parsing and field overlap checks — lives in a `ChannelCreateRequest` record compact constructor. The intent is that invalid requests are rejected immediately on construction, making invalid state unrepresentable.

## What was tried (didn't work)

Placing the validation in a static method on `StoredMessageTypePolicy` and calling it from MCP tools — rejected because it only covered the MCP path (not internal service calls) and created a package cycle.

## Root cause

`ChannelService.create(10-arg)` correctly delegates to `create(new ChannelCreateRequest(...))`, so the compact constructor fires. `ReactiveChannelService.create(10-arg)` was implemented with inline entity construction:

```java
// Reactive path — compact constructor NEVER runs
return Panache.withTransaction("qhorus", () -> {
    Channel channel = new Channel();
    channel.allowedTypes = ...;
    channel.deniedTypes = ...;   // set directly, no ChannelCreateRequest
    return channelStore.put(channel);
});
```

The blocking path's delegation chain is not visible from the reactive path. A developer mirroring the existing inline reactive pattern would not realise they are bypassing validation.

## Fix

Make `ReactiveChannelService.create(10-arg)` construct a `ChannelCreateRequest` before entering the Panache transaction. The compact constructor is pure computation (no IO), so it is safe to call before the reactive transaction scope:

```java
public Uni<Channel> create(String name, ..., String allowedTypes, String deniedTypes) {
    // Compact constructor validation runs here — before the transaction
    return create(new ChannelCreateRequest(
            name, ..., allowedTypes, deniedTypes, null, null, null, null));
}

public Uni<Channel> create(ChannelCreateRequest req) {
    Channel channel = populateChannel(req);   // pure, no IO
    return Panache.withTransaction("qhorus", () -> channelStore.put(channel));
}
```

## Why this is non-obvious

The blocking path correctly validates — so the validation *appears* to work when tested via the blocking API. The reactive path mirrors the blocking path structurally (same method names, same parameters) but the delegation chain is different. Without reading both implementations side-by-side, the gap is invisible. It was caught only through explicit spec review, not at runtime.
