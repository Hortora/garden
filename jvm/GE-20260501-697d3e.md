### Symptom
A new repository query method returns an empty list in production despite matching records
existing. No exception, no warning — just empty results. The method works fine in tests.

### Context
The project uses a custom `ActorIdentityProvider` SPI that pseudonymises actor IDs on write
(`tokenise`) and translates raw IDs back to tokens on read (`tokeniseForQuery`). Existing
methods call `tokeniseForQuery` before querying. A new method querying by attestorId was
added without it.

Tests run with tokenisation disabled (default profile, pass-through provider), so the tests
pass. Production has tokenisation enabled, so stored values are tokens — the raw ID never
matches anything.

### What was tried
9 integration tests all passed. The bug was caught only by a manual code review comparing
the new method to the existing `findByActorId` method.

### Root cause
`tokeniseForQuery` must be called before passing any actor ID to an ORM query when
pseudonymisation is active. It is not enforced by the compiler or any framework mechanism.
New methods can omit it silently.

### Fix

```java
// Wrong — passes raw ID, misses tokenised records
return em.createNamedQuery("Attestation.findByAttestorId", ...)
        .setParameter("attestorId", attestorId)
        .getResultList();

// Correct
final String token = actorIdentityProvider.tokeniseForQuery(attestorId);
return em.createNamedQuery("Attestation.findByAttestorId", ...)
        .setParameter("attestorId", token)
        .getResultList();
```

### Why this is non-obvious
Tests run with the pass-through identity provider (tokenisation disabled), making the bug
invisible. The only way to catch it is a pseudonymisation-profile integration test or manual
code review comparing to existing methods. The symptom (empty results) doesn't hint at a
missing transform.

### Constraints
Applies when: (1) a custom `ActorIdentityProvider` is active, (2) actor IDs are stored as
tokens not raw values, (3) new repository methods query by actor/attestor ID.

### Alternatives considered
- Compile-time enforcement via a typed `Token` wrapper class (rejected: too invasive for the SPI design)
- Adding a `@PreQuery` hook (not available in plain EntityManager pattern)

### Invalidation triggers
Revisit if the project switches to a DB-level transparent encryption approach where the
tokenisation layer is eliminated.
