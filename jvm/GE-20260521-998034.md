### Symptom
Code that tries to catch two exception types with `|` compiles correctly in most cases
but fails with a compile error like "Alternatives in a multi-catch statement cannot be
related by subclassing" when one type is a subclass of the other.

### Context

```java
// Compile error — OptimisticLockException IS-A PersistenceException
} catch (OptimisticLockException | PersistenceException e) {
```

### Root cause
Java's multi-catch (`|`) requires that all listed exception types are unrelated by
inheritance. `jakarta.persistence.OptimisticLockException` extends
`jakarta.persistence.PersistenceException`, so listing both is redundant and the
compiler rejects it.

### Fix
Use only the broader supertype — it already covers all subtypes:

```java
// Correct — covers OptimisticLockException and all other PersistenceExceptions
} catch (PersistenceException e) {
```

If you need to distinguish the specific subtype at runtime, use `instanceof` inside
the catch block:

```java
} catch (PersistenceException e) {
    if (e instanceof OptimisticLockException) {
        // OCC-specific handling
    } else {
        // other persistence failures
    }
}
```

### Why this is non-obvious
When copying exception-handling patterns from retry loops, developers often list both
`OptimisticLockException` and `PersistenceException` to be explicit about what they
are catching. The inheritance relationship is not obvious from the class names — most
developers think of OCC as a different kind of error from a generic persistence failure
rather than as a subtype.
