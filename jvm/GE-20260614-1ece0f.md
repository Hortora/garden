## What it is

The Quarkus LangChain4j Ollama extension sets a **10-second default REST client timeout**
for all chat requests. This is undocumented. For local LLMs (8B+ parameter models on CPU
or M-series Neural Engine), a single JSON-structured chat request typically takes 15–120
seconds, so the default silently causes every judge call to fail.

## Context

Appears when running eval harness tests with Ollama models as the judge. The `PromptJudge`
call fails with:

```
java.lang.IllegalStateException: Judge LLM call failed — check judge model configuration
Caused by: javax.ws.rs.ProcessingException: The timeout period of 10000ms has been exceeded
  while executing POST /api/chat for server localhost:11434
```

The timeout fires before the model finishes generating the structured JSON response.

## Fix

Pass the timeout override via system property:

```bash
"-Dquarkus.langchain4j.ollama.timeout=300s"
```

The property is not documented in the Quarkus LangChain4j extension docs or the
quarkus.io reference (as of 2026-06-14). It's discoverable only by reading the
extension's `OllamaConfig` source.

## Why this is non-obvious

The error message ("check judge model configuration") points to configuration rather
than networking/timeout, leading to checks of model name, format=json flag, and CDI
wiring — none of which are the problem. The 10-second timeout is plausible for
hosted APIs but is impractical for any local model above 3B parameters.
