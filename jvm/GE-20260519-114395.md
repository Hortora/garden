## Call @ConsumeEvent/@ObservesAsync handlers directly via injected CDI proxy тАФ preserves @Transactional, eliminates async waiting in tests

### Stack
Quarkus 3.32.2, JUnit 5, Vert.x Event Bus

### Symptom
Tests for `@ConsumeEvent` (Vert.x) or `@ObservesAsync` (CDI async) handlers require
`Awaitility.await()` or `Thread.sleep()` to wait for async delivery before asserting.
Negative-path tests ("nothing happened") are especially brittle тАФ the sleep must be long
enough that the handler *would* have run, but there's no signal to know it did.

### Context
`eventBus.publish()` is fire-and-forget; the handler runs asynchronously on a worker
thread. Any test calling `publish()` must bridge the async gap. With `@ObservesAsync`,
delivery is additionally unreliable in `@QuarkusTest` contexts (observer methods
silently never invoked in some Quarkus versions).

### Fix

Inject the handler bean directly and call the `@ConsumeEvent`/`@ObservesAsync` method
synchronously. The injected bean is a CDI proxy, so all interceptors тАФ including
`@Transactional` тАФ apply exactly as in production:

```java
@QuarkusTest
class MyHandlerTest {

    @Inject MyHandler handler;   // CDI proxy тАФ @Transactional applies

    @Test
    void handler_creates_record_on_event() {
        // Direct call тАФ synchronous, no await/sleep needed
        handler.onMyEvent(new MyEvent(caseId, ...));

        // Assert immediately тАФ handler has returned, transaction committed
        assertThat(repository.findByCaseId(caseId)).isPresent();
    }
}
```

Keep one wiring test that uses `eventBus.publish()` + `await()` to confirm the
`@ConsumeEvent` route is actually registered:

```java
@Test
void eventBus_routes_event_to_handler() {
    eventBus.publish(MY_ADDRESS, new MyEvent(...));
    await().atMost(5, SECONDS).untilAsserted(() ->
        assertThat(someObservableEffect()).isTrue());
}
```

### Why this is non-obvious
Developers assume that calling a `@ConsumeEvent` or `@ObservesAsync` method directly
bypasses framework behaviour (e.g. transactions, interceptors). It does not тАФ the injected
bean is a CDI proxy, and the interceptor chain fires on every method call through that
proxy, regardless of how the call originates. The method is `public`, the proxy is
injectable, and `@Transactional` commits on the same thread that called the method.

The `@ConsumeEvent(blocking = true)` annotation only controls Vert.x thread dispatch тАФ
it has no effect when the method is called directly. The transaction behaviour is
entirely determined by the `@Transactional` CDI interceptor, which is unaffected.

### Why this fix
Eliminates all timing machinery from the test suite: no `Awaitility`, no `Thread.sleep`,
no `CountDownLatch` (except in one wiring test). Negative-path tests become deterministic
assertions rather than "wait long enough and hope". The wiring test retains integration
coverage of the event bus route without imposing async overhead on every test.
