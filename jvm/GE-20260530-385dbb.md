## Symptom

`@Inject`-annotated fields in a `ClientRequestFilter` are null at runtime, producing an NPE
on the first request:

```
java.lang.NullPointerException: Cannot invoke "com.example.MyConfig.token()"
    because "this.config" is null
    at MyAuthFilter.filter(MyAuthFilter.java:22)
```

The filter runs ‚Äî proving it was picked up ‚Äî but its injected dependencies are absent.

## Context

A `ClientRequestFilter` is annotated with both `@Provider` and `@ApplicationScoped` to add
an `Authorization` header to every request made by a `@RegisterRestClient`-annotated REST
client interface. CDI injection inside the filter (`@Inject MyConfig config`) works in other
CDI beans but fails here.

```java
@Provider                    // ‚Üê the culprit
@ApplicationScoped
public class MyAuthFilter implements ClientRequestFilter {
    @Inject MyConfig config; // null at runtime
    ...
}
```

## What was tried (didn't work)

- Confirmed the filter IS being invoked (the NPE proves this)
- Checked that `MyConfig` is a valid CDI bean ‚Äî injection works elsewhere
- Verified Jandex index includes the filter class
- Added `@Dependent` scope ‚Äî same NPE

## Root cause

`@Provider` is a JAX-RS annotation that marks a class as a provider for the JAX-RS runtime.
In Quarkus, `@Provider` triggers server-side JAX-RS provider registration. When the Quarkus
REST Client encounters a `@Provider`-annotated class that implements `ClientRequestFilter`,
it picks it up as a global filter ‚Äî but instantiates it directly (not via CDI), bypassing the
`@ApplicationScoped` bean context. The result is a plain Java object with null `@Inject` fields.

## Fix

Remove `@Provider`. Register the filter explicitly with `@RegisterProvider` on the REST client
interface. Quarkus REST Client resolves `@RegisterProvider` entries from the CDI container
when the class is a CDI bean ‚Äî injection works correctly.

```java
// 1. Remove @Provider from the filter
@ApplicationScoped   // keep CDI scope; remove @Provider
public class MyAuthFilter implements ClientRequestFilter {
    @Inject MyConfig config; // ‚úÖ injected correctly now
    @Override
    public void filter(ClientRequestContext ctx) {
        ctx.getHeaders().putSingle("Authorization", "Bearer " + config.token().orElse(""));
    }
}

// 2. Register via @RegisterProvider on the client interface
@RegisterRestClient(configKey = "my-service")
@RegisterProvider(MyAuthFilter.class)   // ‚Üê CDI bean resolved from container
@Path("/")
public interface MyServiceClient {
    @GET @Path("/resource")
    Response get();
}
```

## Why this is non-obvious

`@Provider` is the standard JAX-RS mechanism for registering providers. It is reasonable to
assume that `@Provider @ApplicationScoped` on a CDI bean means "register this CDI bean as a
provider" ‚Äî which is exactly how it works for server-side JAX-RS filters. The fact that the
Quarkus REST Client picks up `@Provider` annotated classes globally but does NOT use the CDI
instance is not documented; the NPE symptom points at `config` being null rather than at
`@Provider` as the registration mechanism that bypassed CDI.

The correct mechanism (`@RegisterProvider` on the interface) is documented, but there is no
warning in the Quarkus docs that using `@Provider` on a REST client filter causes CDI bypass.
