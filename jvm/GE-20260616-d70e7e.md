### Symptom

27 "Ambiguous dependencies for type `CurrentPrincipal`" ARC deployment errors at `quarkus:build`. Three `@Default` beans competing; you have the problematic one in `quarkus.arc.selected-alternatives`. Build broken.

### Context

Multiple `@Default` beans satisfy the same injection type. You add the "intended winner" to `quarkus.arc.selected-alternatives` expecting it to take priority. The entry is accepted silently â€” no warning, no error â€” but the ambiguity persists exactly as before.

### What was tried

Added the non-`@Alternative` class to `quarkus.arc.selected-alternatives`. Quarkus did not reject the entry; the config appeared correct. But all 27 errors remained on the next build.

### Root cause

`quarkus.arc.selected-alternatives` is designed for beans annotated `@Alternative`. For non-`@Alternative` beans, Quarkus accepts the config entry silently (no validation, no warning) but does nothing with it â€” the bean is already active as a `@Default` and needs no "selection." The other `@Default` competitors are unaffected. Since CDI ambiguity requires exactly one unambiguous `@Default` winner, adding the non-`@Alternative` to `selected-alternatives` achieves nothing.

### Fix

Do not put non-`@Alternative` beans in `selected-alternatives`. To resolve `@Default` ambiguity when you cannot change the competing beans' annotations (they live in upstream dependency JARs), add the losing beans to `quarkus.arc.exclude-types` instead:

```properties
quarkus.arc.exclude-types=\
  io.example.CompetingBeanA,\
  io.example.CompetingBeanB
```

The intended winner survives; excluded beans are not visible to CDI at all.

### Why this is non-obvious

`selected-alternatives` is the documented mechanism for activating a preferred CDI bean. Nothing in the Quarkus docs or ARC error output warns that the listed class must itself be `@Alternative`. The config entry is accepted, the bean is already live, so the fix "looks applied" â€” yet the 27 errors persist unchanged.
