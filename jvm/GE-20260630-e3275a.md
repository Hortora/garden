### The undocumented behaviour

A JPQL bulk `UPDATE` executed via `em.createQuery(...).executeUpdate()` does NOT increment the Hibernate `@Version` field on affected entities, even when the entity class has a `@Version` annotation.

```java
// Entity has: @Version private Long version = 0L;

// This JPQL UPDATE sets policyTriggered but does NOT change version:
em.createQuery("UPDATE SituationEntity s SET s.policyTriggered = true, " +
               "s.lastTriggered = :time, s.triggerCount = s.triggerCount + 1 " +
               "WHERE s.situationId = :sid AND s.policyTriggered = false")
    .setParameter("time", triggerTime)
    .setParameter("sid", situationId)
    .executeUpdate();

// After this: entity.version is still 0, policyTriggered is true
```

### Why this matters

This enables a powerful pattern: **atomic flag+metadata stamping via JPQL** that coexists with managed-entity OCC.

In a multi-step flow like `save(context) → tryClaimTrigger → fire → compact → save(compacted)`:
- `save()` uses managed entity operations (merge/persist) which DO increment `@Version`
- `tryClaimTrigger` uses JPQL bulk UPDATE which does NOT increment `@Version`
- The second `save()` works because the entity version hasn't changed between the two saves — only the application-level `storeVersion` check matters

If JPQL bulk UPDATE DID increment `@Version`, the second `save()` would throw `OptimisticLockException` because the version changed out from under the managed entity.

### How it was discovered

During design review of a clustered duplicate-trigger prevention mechanism. The reviewer traced through the execution path and identified that if JPQL UPDATE incremented @Version, the subsequent managed-entity save would conflict. Testing confirmed the version stays stable. No Hibernate documentation explicitly states this.

### Why this is non-obvious

Most developers assume "any UPDATE to an entity increments its version" — that's true for managed-entity dirty writes (Hibernate's dirty checking + flush), but NOT for JPQL bulk operations. The distinction between "Hibernate manages the entity lifecycle" and "raw JPQL bypasses the lifecycle" is well-known for L1 cache (GE-20260414-99854d, GE-20260518-da7e91), but the @Version implication is not documented.
