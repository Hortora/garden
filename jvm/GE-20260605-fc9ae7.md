### Symptom
`forkedProcessTimeoutInSeconds=300` configured in Surefire plugin. Forked test JVM runs for 35+ minutes without being killed. Documentation says it kills the forked process after N seconds.

### Context
Quarkus `@QuarkusTest` without `quarkus-maven-plugin` performs augmentation inside the forked JVM before any test method executes. The augmentation phase can hang on resource-constrained CI.

### What the docs say
> `forkedProcessTimeoutInSeconds` — Kill the forked test process after a certain number of seconds. If set to 0, wait forever for the process, never timing out.

### What actually happens
The timeout does NOT fire during the Quarkus augmentation phase. A forked JVM that hangs during `QuarkusTestExtension` initialization (before JUnit reports any test to Surefire) runs indefinitely despite the timeout configuration. The timeout appears to measure time after Surefire's test reporting protocol is established, not total JVM elapsed time.

### Workaround
Use `timeout-minutes` on the GitHub Actions job as a hard external cap. Or add `quarkus-maven-plugin` to move augmentation out of the forked JVM entirely.

### Why this is non-obvious
The documentation implies total elapsed time. The actual behaviour only covers the test execution phase, silently ignoring JVM boot and framework initialization. No warning or log message indicates the timeout is not active during augmentation.
