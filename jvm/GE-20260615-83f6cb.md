## The Problem

A Phase 1 migration bridge requires each of N plugin classes to evaluate a two-step activation check in `canActivate(CaseFile)`: first check `requires()` key presence, then check `activateIf()` extra gates. Writing this inline in 8 plugins creates brittle duplication — if the combination logic changes, all 8 must be updated.

## The Technique

Add a `default` method to the interface that combines the two steps. Each plugin bridge delegates to it in a single expression:

**In the interface:**
```java
public interface TaskDefinition {
    default Set<String> requires() { return Set.of(); }
    default Predicate<CaseContext> activateIf() { return ctx -> true; }

    /** Phase 1 bridge helper — mirrors Phase 2 SequenceWorker evaluation order. */
    default boolean testActivation(CaseContext ctx) {
        return requires().stream().allMatch(ctx::contains) && activateIf().test(ctx);
    }
}
```

**In each of the 8 plugin bridges:**
```java
@Override
public boolean canActivate(CaseFile caseFile) {
    return testActivation(new CaseFileContext(caseFile));
}
```

## Why Non-Obvious

The instinct is to extract a static utility method or write it inline. The `default` method approach is better here because:
1. It lives on the interface that all implementations share — no import needed
2. It encodes the Phase 2 semantics explicitly (requires first, then activateIf) in the contract, not in utility code
3. When Phase 2 arrives and bridge methods are deleted, the `default` method serves as documentation of the intended evaluation order

The technique also prevents the subtle bug where refactoring `activateIf()` to remove key re-checks (making it "extra gates only") breaks the bridge — the `testActivation()` helper re-adds the key check from `requires()`, so the bridge stays correct.
