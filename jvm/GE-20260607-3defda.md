### Symptom

Need to compute trust scores on-read from raw attestation history, but routing calls multiple SPI methods per actor per routing decision (capabilityScore + decisionCount + N × capabilityDimensionScore). Without caching, each call re-loads all attestations and re-runs Bayesian Beta computation. For 3 candidates with 2 quality dimensions: 12 full loads per routing decision.

### Context

`ComputedTrustScoreSource` — an on-read trust score implementation where the goal is zero staleness (scores always reflect the latest attestation). Traditional TTL-based caching reintroduces the staleness the design exists to eliminate.

### Technique

Use `ConcurrentHashMap<String, ComputedScores>` keyed by actorId. On first score request for an actor, load all events + attestations, compute ALL score types (capability, dimension, capability×dimension, global), cache the full bundle. Subsequent requests for the same actor return from cache.

Invalidation: observe the CDI event that fires after an attestation is committed (`@Observes(during = TransactionPhase.AFTER_SUCCESS) AttestationRecordedEvent`), remove the cache entry for `event.actorId()` — the decision-maker whose trust score is affected.

```java
private final ConcurrentHashMap<String, ComputedScores> cache = new ConcurrentHashMap<>();

void onAttestationRecorded(@Observes(during = TransactionPhase.AFTER_SUCCESS)
        AttestationRecordedEvent event) {
    cache.remove(event.actorId());
}

private ComputedScores computeIfAbsent(String actorId) {
    ComputedScores result = cache.computeIfAbsent(actorId, this::computeFresh);
    return result == EMPTY_SENTINEL ? null : result;
}
```

### Why this is non-obvious

Standard caching (TTL, size-bounded LRU) reintroduces staleness — the exact problem the on-read source exists to solve. The insight is that between attestation writes, the cache is *correct* — no new data exists. The CDI event is the only signal that the cached value might be wrong. This makes invalidation exact rather than time-based: the cache is never stale because it's invalidated at the precise moment the underlying data changes.

The key design choice: compute ALL score types for an actor on first access (not just the requested type). This converts 12 DB loads per routing decision into 1, at the cost of computing a few extra score types that might not be queried. For bounded actor sets (the target use case), this is always a net win.
