## Technique

To test OTel-aware code that reads `Span.current()`, use `Span.wrap(SpanContext)`
to create a span backed by a specific context, then make it current with
`span.makeCurrent()`. This works with the OTel API alone — no SDK, no tracer,
no `@QuarkusTest` OTel configuration needed.

```java
SpanContext ctx = SpanContext.create(
        "4bf92f3577b34da6a3ce929d0e0e4736",
        "00f067aa0ba902b7",
        TraceFlags.getSampled(),
        TraceState.getDefault());

try (Scope ignored = Span.wrap(ctx).makeCurrent()) {
    // Span.current() now returns a span with the above context
    // — test your OTel-aware code here
    Optional<String> traceId = provider.currentTraceId();
    assertThat(traceId).contains("4bf92f3577b34da6a3ce929d0e0e4736");
}
```

## Context

Any code that reads `Span.current().getSpanContext()` can be tested this way —
unit tests, `@QuarkusTest` integration tests, or anything in between. The scope
is thread-local and automatically cleaned up by the try-with-resources.

## Why this is non-obvious

The natural instinct is to configure a real OTel SDK (add `quarkus-opentelemetry`,
set up an exporter, create a `Tracer`). That works but adds significant test
infrastructure. `Span.wrap()` is an OTel API method specifically designed for
wrapping an existing `SpanContext` in a non-recording span — it exists, is stable,
and requires nothing beyond the API jar, which is often already transitively present.
