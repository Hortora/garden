## Context

A `@Transactional @ApplicationScoped` service bean throws `ClientErrorException` or
`WebApplicationException` to signal not-found or conflict conditions to the calling JAX-RS
resource. This imports `jakarta.ws.rs.*` into the service layer â€” a layer violation:
services should not know about HTTP semantics.

The obvious alternatives all have problems:

- **`ClientErrorException` in service**: works at runtime but imports JAX-RS into service layer
- **Move validation to resource**: breaks `@Transactional` atomicity if validation and
  mutation must be atomic (TOCTOU: another request changes the entity between the check and
  the service call)
- **Checked exception hierarchy**: requires callers to catch or declare; awkward with
  `@Transactional` which only auto-rolls back on `RuntimeException`
- **`ExceptionMapper<MyException>`**: correct for cross-module use but adds infrastructure
  overhead for a module-local pattern

## Technique

Define lightweight inner static `RuntimeException` subclasses on the service itself.
They carry no HTTP knowledge. The JAX-RS resource catches them and maps to builder-based
responses â€” keeping HTTP concerns in the HTTP layer.

```java
@ApplicationScoped
public class TrialActivationService {

    public static class TrialNotFoundException extends RuntimeException {
        private static final long serialVersionUID = 1L;
        public TrialNotFoundException(UUID id) { super("Trial not found: " + id); }
    }

    public static class TrialNotInPlanningStatusException extends RuntimeException {
        private static final long serialVersionUID = 1L;
        public TrialNotInPlanningStatusException(TrialStatus status) {
            super("Trial status is " + status + ", expected PLANNING");
        }
    }

    @Transactional
    Map<String, Object> markActive(UUID trialId) {
        ClinicalTrial trial = ClinicalTrial.findById(trialId);
        if (trial == null) throw new TrialNotFoundException(trialId);
        if (trial.status != TrialStatus.PLANNING) throw new TrialNotInPlanningStatusException(trial.status);
        // ... validation and mutation are atomic
    }
}

// In the JAX-RS resource:
public Response activate(@PathParam("id") UUID id) {
    try {
        trialActivationService.activate(id);
        return Response.noContent().build();
    } catch (TrialActivationService.TrialNotFoundException e) {
        return Response.status(Response.Status.NOT_FOUND).build();
    } catch (TrialActivationService.TrialNotInPlanningStatusException e) {
        return Response.status(Response.Status.CONFLICT).build();
    }
}
```

`RuntimeException` ensures `@Transactional` auto-rolls back on error. Inner static
placement means the exception is co-located with its thrower â€” one file to read for both
semantics and error conditions. The resource catch block is the single place where HTTP
status codes are decided.

## Why this is non-obvious

The instinct when a service needs to signal error conditions to HTTP callers is to reach
for `ClientErrorException` (it's in the JAX-RS API, it exists for exactly this) or to
move validation up to the resource layer. The inner static class pattern is not in any
JAX-RS guide because it's a layering discipline choice, not a JAX-RS feature.

The key insight is that the service's error taxonomy (`TrialNotFound`, `WrongStatus`)
is domain knowledge, not HTTP knowledge. The HTTP mapping (`NOT_FOUND`, `CONFLICT`) is the
resource's responsibility. Separating the two â€” domain exception on the service, HTTP
mapping on the resource â€” preserves both layer boundaries and `@Transactional` atomicity
without any infrastructure overhead.
