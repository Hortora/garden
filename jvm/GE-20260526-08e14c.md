### The technique

When a JPA entity needs a concurrency-safe integer counter increment, replace the read-modify-write pattern (`entity.count++`) with an atomic JPQL UPDATE followed by a re-read:

```java
// Instead of: read → mutate → Hibernate dirty write (data race under READ COMMITTED)
e.completedCount++;
return Uni.createFrom().item(toDomain(e));

// Use: single atomic SQL statement + re-read
SubCaseGroupEntity.update(
    "completedCount = completedCount + 1 WHERE parentCaseId = ?1 AND groupId = ?2",
    parentCaseId, groupId)
.chain(count -> {
    if (count == 0)
        return Uni.createFrom().failure(new IllegalStateException("Group not found"));
    return SubCaseGroupEntity
        .<SubCaseGroupEntity>find("parentCaseId = ?1 AND groupId = ?2", parentCaseId, groupId)
        .firstResult()
        .onItem().ifNotNull().transform(this::toDomain)
        .onItem().ifNull().failWith(() -> new IllegalStateException("Group vanished after increment"));
});
```

In Hibernate Reactive Panache, wrap in `Panache.withTransaction()`. The `UPDATE` and `SELECT` are in the same transaction — the `SELECT` always sees the committed result of the `UPDATE`.

### When to use this

Use when:
- Only one or a few integer fields need concurrent increment/decrement
- Contention is possible but lock acquisition cost is undesirable
- You don't need retry-on-conflict behavior — atomic SQL avoids the conflict entirely

### Why this is non-obvious

The obvious approach for concurrent writes is `@Version` optimistic locking with an `OptimisticLockException` retry loop. `@Version` is correct but adds: a version column, exception handling in the caller, retry logic, and exponential backoff in high-contention scenarios.

For a single counter field, `UPDATE SET count = count + 1 WHERE ...` is a single SQL statement that the database serializes at the row level — no version column, no retry. PostgreSQL (and most RDBMS) guarantee that two concurrent executions of this UPDATE on the same row serialize without conflict: one gets `count=1, next=0`, one gets `count=1, next=1` — each sees the other's write.

The re-read returns the DB-committed value at SELECT time, which may reflect concurrent increments that committed between the UPDATE and SELECT. This is intentional: callers that compare count vs threshold see the current total, which is exactly what's needed for M-of-N coordination.

### Caveats

- The returned `SubCaseGroup` reflects DB state at SELECT time — it may include increments from other concurrent transactions. If the caller needs "my increment only" semantics, this doesn't work.
- In Hibernate Reactive, the UPDATE and re-read must be in the same `withTransaction()` block to guarantee the SELECT sees the UPDATE's result.
- "Group vanished after increment" failure path is theoretical (someone deleted the row between UPDATE and SELECT) but worth guarding against in production code.
