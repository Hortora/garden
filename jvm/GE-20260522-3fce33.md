### Symptom
Multiple `.filter()` calls on the same fluent DSL builder chain — e.g.
`.from(DS::persons).filter(p -> p.age() > 18).filter(p -> p.name().startsWith("A"))` —
produce no compile error and no runtime warning, but only the **last** filter takes
effect. All earlier filters are silently discarded.

### Context
Drools vol2 `RuleBuilder`. Each DSL step stores predicates via a `storeFilter()` method
that called `List.set(filters.size() - 1, pred)` — overwriting the slot — rather than
`List.add()` accumulating them. The same pattern can appear in any fluent builder that
stores state in a parallel list indexed by slot position.

### Root cause
`List.set(index, value)` replaces the existing value at that index.
`List.add(value)` appends. When `storeFilter()` was written to replace, the assumption
was one-filter-per-slot. Multiple chained calls each replaced the previous predicate,
leaving only the last one stored.

### Fix
Replace `List.set(filters.size() - 1, pred)` with accumulation — e.g.
`pattern.addConstraint(new LambdaConstraint(pred, arity))` — or more generally,
replace `List.set()` with `List.add()` wherever accumulation is the correct semantics.
In vol2 this was fixed by introducing `LambdaConstraint` stored via
`Pattern.addConstraint()`, making the accumulation explicit in the type.

### Why this is non-obvious
The fluent API looks like accumulation. `.filter(a).filter(b)` reads as "apply both
filters" to any user of the API. The internal `set()` vs `add()` distinction is
completely invisible at the call site. The predicate that "wins" is the last one
declared, which may match tests written with only one filter per slot — masking the
bug entirely until a test with two filters is written.
