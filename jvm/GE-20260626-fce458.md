### Symptom
REST API returns 401 Unauthorized after migrating a REST client from `@RegisterRestClient` (CDI-injected) to programmatic `RestClientBuilder` creation. No error, no exception, no warning — the Authorization header is simply absent from requests.

### Context
An `OpenHabAuthHeadersFactory` implementing `ClientHeadersFactory` was registered via `@RegisterClientHeaders(OpenHabAuthHeadersFactory.class)` on the REST client interface. When the REST client was migrated to programmatic creation via `RestClientBuilder`, the factory was registered with `.register(authHeadersFactory)`. Requests silently went out without the auth header.

### Root cause
`ClientHeadersFactory` is a MicroProfile REST Client extension interface (`org.eclipse.microprofile.rest.client.ext.ClientHeadersFactory`). It is NOT a JAX-RS provider type. `RestClientBuilder` extends `jakarta.ws.rs.core.Configurable<RestClientBuilder>` — its `.register()` method only activates JAX-RS provider types: `ClientRequestFilter`, `ReaderInterceptor`, `MessageBodyReader`, `WriterInterceptor`, etc.

Passing a `ClientHeadersFactory` to `.register()` is silently accepted (the method signature accepts `Object`) and silently ignored (it's not a recognized provider type). No error. No warning. The factory's `update()` method is never called. Requests go out without the headers it would have added.

Type hierarchy confirms: `ClientHeadersFactory` has zero supertypes — it does not extend or implement any JAX-RS provider interface.

### Fix
Replace `ClientHeadersFactory` with `ClientRequestFilter`. Same auth logic, correct JAX-RS provider type.

```java
// BEFORE (silently ignored):
class OpenHabAuthHeadersFactory implements ClientHeadersFactory {
    @Override
    public MultivaluedMap<String, String> update(...) { ... }
}
builder.register(authHeadersFactory);  // ← silently ignored, no auth header

// AFTER (correct):
class OpenHabAuthFilter implements ClientRequestFilter {
    @Override
    public void filter(ClientRequestContext ctx) {
        ctx.getHeaders().putSingle("Authorization", authHeader);
    }
}
builder.register(new OpenHabAuthFilter(config.auth()));  // ← works
```

### Why this is non-obvious
`.register()` accepts `Object` — no compile-time type safety. The method doesn't throw on unrecognized types. The only symptom is 401 from the target API, which naturally directs debugging toward credential issues rather than provider registration. The distinction between MicroProfile extension interfaces and JAX-RS provider interfaces is not documented in the migration path from `@RegisterClientHeaders` to `RestClientBuilder`.
