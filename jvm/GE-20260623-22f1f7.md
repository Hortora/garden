## Symptom

Augmentation fails at build time with `AmbiguousResolutionException` after adding `casehub-platform-oidc` as a compile dependency to a project that already depends on `casehub-qhorus`.

## Context

Both `casehub-platform-oidc` and `casehub-qhorus` ship a `CurrentPrincipal` implementation:
- `OidcCurrentPrincipal` (`casehub-platform-oidc`) — `@RequestScoped`, no qualifier
- `QhorusInboundCurrentPrincipal` (`casehub-qhorus`) — `@ApplicationScoped`, no qualifier

Two unqualified CDI beans implementing the same type → `AmbiguousResolutionException`.

## What was tried

Trusting the `QhorusInboundCurrentPrincipal` Javadoc, which states:

> Displaced by… OidcCurrentPrincipal @Priority(100) when casehub-platform-oidc is on the classpath.

This claim is false. `OidcCurrentPrincipal` in the actual source has no `@Priority` or `@Alternative` annotation — the stale Javadoc describes a design intent that was never implemented.

## Root cause

`OidcCurrentPrincipal` is `@RequestScoped` with no qualifier disambiguation. It does not auto-displace `QhorusInboundCurrentPrincipal` — they are two equal peers from CDI's perspective.

## Fix

Add `QhorusInboundCurrentPrincipal` to `quarkus.arc.exclude-types` in production `application.properties`:

```properties
quarkus.arc.exclude-types=\
  io.casehub.platform.mock.MockCurrentPrincipal,\
  io.casehub.qhorus.runtime.identity.QhorusInboundCurrentPrincipal
```

After this exclusion, `OidcCurrentPrincipal` is the sole `CurrentPrincipal` CDI bean in production. In `@QuarkusTest` scope, `FixedCurrentPrincipal @Alternative @Priority(1)` (from `casehub-platform-testing`) displaces `OidcCurrentPrincipal` when activated via `quarkus.arc.selected-alternatives`.

## Why this is non-obvious

The Javadoc is the first thing a developer reads when investigating CDI ambiguity. It confidently describes a displacement mechanism — `@Priority(100)` — that does not exist in the source. Verifying the claim requires decompiling the bytecode. Filed qhorus#301 to update the Javadoc.
