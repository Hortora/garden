### Symptom

`FlywayMigrateException: Table "MESSAGE_LEDGER_ENTRY" not found` when running V31 migration that does `ALTER TABLE message_ledger_entry ADD COLUMN topic VARCHAR(200)`.

The migration file exists, the table name is correct, and the same ALTER works fine when run manually after schema creation.

### Context

The qhorus project uses two Flyway migration locations combined at runtime:
- `classpath:db/qhorus/migration` — domain migrations V1–V999
- `classpath:db/ledger/migration` — library migrations V1000–V1999

A joined subclass table (`message_ledger_entry`) is created by V2000 in the qhorus location — it depends on `ledger_entry` (V1000) existing first. Consumer-specific joins of the ledger table start at V2000 by convention.

### What was tried (didn't work)

Adding `V31__add_ledger_entry_topic.sql` to the domain migration sequence — the natural next V number after V30.

### Root cause

Flyway sorts ALL migrations from ALL locations into a single numeric sequence: V1, V2, ..., V30, **V31**, ..., V1000, ..., V2000. V31 runs long before V2000, which is the migration that creates `message_ledger_entry`. The table doesn't exist yet when V31 tries to ALTER it.

This is not a classpath scanning issue (covered by GE-20260511-a28064) — both migrations are in the correct location. It's a pure numeric ordering problem when a domain migration needs to modify a table created in a higher V-range.

### Fix

Any migration that touches a table created in the V2000+ range must itself use a V-number ≥ V2001. Rename the migration:

```
V31__add_ledger_entry_topic.sql  →  V2001__add_ledger_entry_topic.sql
```

The rule: the V-number namespace is partitioned by table ownership:
- V1–V999: domain tables (channel, message, topic, reaction, etc.)
- V1000–V1999: library tables (ledger_entry, ledger_merkle_frontier, etc.)
- V2000+: consumer joins and extensions of library tables (message_ledger_entry)

A migration in range A cannot ALTER tables created in range B if A < B.

### Why this is non-obvious

The natural instinct is to use the next sequential V-number in the domain's migration sequence. The error message ("table not found") doesn't hint at ordering — it looks like a missing dependency or wrong table name. The actual cause (Flyway sorts numerically across all locations) only becomes clear when you trace the full migration execution order.

### Alternatives considered

- Adding the column to V2000 itself — rejected because V2000 is the base joined subclass migration shared across consumers; adding domain-specific columns there couples unrelated concerns.
- Using Flyway's `dependsOn` — not available in community Flyway; migration ordering is purely numeric.
