## Symptom

UUID-or-name resolution silently falls through to name lookup even when passed a valid
UUID string. The wrong lookup runs, returns an empty result, and the error is never surfaced.

## Root Cause

The `try/catch` scope is too broad. Both `UUID.fromString` and the downstream lookup
(e.g. `findById`) can throw `IllegalArgumentException`. When the downstream call throws
for any reason other than UUID parsing, the catch fires and the code falls through to
the name-lookup branch — passing a UUID-shaped string to a name query.

```java
// WRONG — catch scope too broad
public Optional<WorkItemTemplate> findByRef(String ref) {
    try {
        return findById(UUID.fromString(ref));   // fromString OR findById can throw IAE
    } catch (IllegalArgumentException e) {
        return findByName(ref);   // fires even if findById threw for an unrelated reason
    }
}
```

## Fix

Scope the catch to `UUID.fromString` only. Call the downstream method outside the
try block so its errors propagate normally:

```java
// CORRECT — catch scoped to parse only
public Optional<WorkItemTemplate> findByRef(String ref) {
    final UUID id;
    try {
        id = UUID.fromString(ref);
    } catch (IllegalArgumentException e) {
        return findByName(ref);   // genuinely not a UUID string
    }
    return findById(id);   // outside try — real errors from findById propagate
}
```

## Why This Is Non-Obvious

The intent of the catch is "if this isn't a UUID, try name lookup." Both
`UUID.fromString` and `findById` can throw `IllegalArgumentException`, so a single
catch block around both feels natural and correct. The bug only manifests as a wrong
result (name lookup on a UUID string) — not as an exception — making it invisible
without deliberate code review.
