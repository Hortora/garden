### Symptom
`AgentValidationException: briefing contains banned character U+000A` when constructing an `AgentDescriptor` with a briefing loaded from YAML using `|` (literal block scalar).

### Context
Writing agent persona definitions in `META-INF/eidos/descriptors.yaml` for `ClasspathYamlDescriptorRegistrar`. Multi-paragraph briefings are the expected use case — the `MAX_BRIEFING` limit is 2000 characters, designed for substantive behavioural guidance.

### What was tried (didn't work)
YAML `|` (literal block scalar) — the natural choice for multi-line text. Produces a string with embedded `\n` characters. The `AgentDescriptor` compact constructor calls `AgentDescriptorValidator.validateOptional("briefing", briefing, MAX_BRIEFING)`, which calls `validateField()`, which iterates codepoints and calls `isBanned(cp)`. `isBanned()` returns `true` for `cp <= 0x001F` — the entire C0 control range, including U+000A (newline) and U+000D (carriage return).

### Root cause
The banned-character check in `AgentDescriptorValidator` is uniform across all fields. There is no per-field exemption. Newlines are banned alongside genuinely dangerous control characters (BiDi overrides, zero-width chars, BOM). The briefing field has no special handling despite being designed for multi-sentence content.

### Fix
Use `>-` (folded block scalar, strip final newline) instead of `|` in persona YAML files:

```yaml
briefing: >-
  You are the Structural Reviewer for DraftHouse. Your focus is document
  organisation, logical flow, and section coherence.
  Approach reviews collaboratively — flag structural issues with specific
  suggestions for reordering, splitting, or merging sections.
```

YAML `>-` folds newlines to spaces, producing a single-paragraph string with no C0 control characters. Multi-sentence briefings flow as continuous text.

### Why this is non-obvious
The error message names the banned codepoint (`U+000A`) but gives no hint that the YAML scalar style is the fix. A developer familiar with YAML would naturally reach for `|` for multi-line text — it's the standard tool for preserving formatting. The connection between "banned character U+000A" and "use `>-` instead of `|`" requires understanding both the validator's banned-char list AND YAML scalar type semantics. The validator's uniformity (all fields, all C0 chars) is invisible until you read `isBanned()`.

### Alternatives considered
- Exempting newline from `isBanned()` globally — rejected because newlines in `agentId`, `slot`, or `tenancyId` would be harmful; per-field exemption is the right design but a larger change (tracked as eidos#68)
- Using escaped `\n` in YAML quoted strings — produces literal backslash-n, not actual newlines; and actual newlines would still fail validation
