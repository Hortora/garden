## Technique

When a code constant must be updated whenever a related enum or protocol changes, place a comment at that exact location that explicitly names the obligation. This makes the maintenance requirement traceable without external ADRs, tickets, or docs.

## The pattern

```java
// If a new MessageType is added to Qhorus with no commitment effect (like EVENT),
// add it here for all governance channels — this comment is the mechanical anchor
// for that obligation.
Set.of(MessageType.EVENT),  // deniedTypes: no telemetry on governance channel
```

The phrase "mechanical anchor" signals that this comment is not explanatory noise — it is the designated place where future developers must act when the named change happens. Any IDE search for the changed constant will surface this obligation automatically.

## Why this is non-obvious

Most developers document maintenance obligations in PR descriptions, ADRs, or JIRA tickets — all of which rot quickly and are never seen at the moment of change. The mechanical anchor places the obligation at the point of mechanical effect: the code that must change. When a developer adds a new `MessageType` enum value and runs their IDE's "find usages", the anchor appears in the results — no separate lookup required.

## When to use

- Enum-driven sets that must stay in sync with enum additions (e.g. type allow/deny lists)
- Protocol compliance invariants that depend on upstream definitions
- Any constant whose membership or value must be revisited when a named external thing changes

## Alternative (why not to use)

Writing the obligation only in a doc or ADR — effective at the moment of writing but invisible at the moment of change. The anchor's value is proximity: it fires at the exact location where the developer is already looking.
