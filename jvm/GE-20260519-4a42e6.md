### Symptom

Error changes from "No current Vertx context found" to "Can't get the context safety flag: the current context is not a duplicated context" — wrapping the call in `vertx.executeBlocking()` seemed to fix the first error but the operation still fails.

### Context

Triggered when calling `Panache.withTransaction()` (or `withSession()`) from a `@BeforeEach` test method or other code that isn't on a proper Quarkus-managed request thread. Adding `vertx.executeBlocking()` provides *a* Vert.x context but not the right kind.

### Root cause

`SessionOperations.vertxContext()` in Quarkus Hibernate Reactive checks two things:
1. A Vert.x context exists (`Vertx.currentContext()`)
2. `VertxContextSafetyToggle.checkIsSafe()` — the context must be a *duplicated* context (`context.duplicate()`)

Worker threads from `vertx.executeBlocking()` run in a root context. Root contexts fail the safety check even though they are valid Vert.x contexts.

### Fix

**Option 1 — `@Blocking` on endpoint/method (production code):**
```java
@GET
@Path("/{id}/lineage")
@Blocking
public Response getLineage(@PathParam("id") String id) {
    return Response.ok(service.findAll().await().indefinitely()).build();
}
```
Quarkus creates a properly duplicated context for `@Blocking` method invocations.

**Option 2 — `@RunOnVertxContext` for test lifecycle methods:**
```java
@BeforeEach
@RunOnVertxContext
Uni<Void> setup() {
    return service.create("test-channel", ...).replaceWithVoid();
}
```

### Why this is non-obvious

Two different error messages for two related but distinct constraints, encountered sequentially. A developer who added `executeBlocking()` to fix the first "No Vert.x context" error reasonably believes they are on the right track — the error did change. The duplicated-context requirement is not documented in `Panache.withTransaction()` JavaDoc and only appears via the internal safety toggle.
