Qdrant collection delete without cursor reset leaves engine unable to re-ingest

### Symptom

After deleting a Qdrant collection via the REST API (`DELETE /collections/<name>`), restarting the engine produces no errors but no entries are ingested. The collection is recreated empty and stays empty. `gardenSearch` returns no results.

### Context

Occurs when manually deleting a Qdrant collection to force schema migration (e.g. adding BM25 sparse vectors to an existing dense-only collection). The engine starts, `CorpusIngestionService` runs, but skips all files because the cursor says they were already processed.

### What was tried

- Deleted the Qdrant collection via REST API
- Restarted the engine
- Expected: full re-ingestion into new collection with updated schema
- Actual: empty collection, zero points, no errors

### Root cause

`casehub-rag`'s ingestion pipeline has two independent state stores:

1. **Qdrant collection** — the vector data
2. **FileCursorStore cursor** — a JSON map of `{filepath: lastModifiedTimestamp}` persisted at `${java.io.tmpdir}/casehub-ingestion-cursors/<gardenId>.cursor`

Deleting the collection removes (1) but leaves (2) intact. On next startup, `CorpusIngestionService` reads the cursor, sees all files as already-processed (timestamps match), and skips re-ingestion entirely. No warning is logged — the service believes the corpus is up to date.

### Fix

Use `gardenReindex()` MCP tool instead of manual collection deletion. It calls `collectionMigration.resetCorpus()` which does both steps:

```java
public void resetCorpus(CorpusRef corpusRef, String gardenId) {
    embeddingIngestor.deleteCorpus(corpusRef);  // deletes Qdrant collection
    cursorStore.save(gardenId, "");              // resets cursor to empty
}
```

If you must delete the collection manually (e.g. Qdrant REST API), also reset the cursor file:

```bash
# Find the cursor file (java.io.tmpdir on macOS is NOT /tmp/)
find /var/folders -name 'casehub-ingestion-cursors' 2>/dev/null
# Reset it
echo -n "" > /var/folders/.../casehub-ingestion-cursors/garden.cursor
```

### Why this is non-obvious

The two-store design is invisible to someone working with Qdrant directly. Deleting a collection feels like a complete reset — there is no indication that a separate filesystem cursor also needs clearing. The cursor path uses `java.io.tmpdir` which on macOS resolves to `/var/folders/xx/xxxxxxxxx/T/` (not `/tmp/`), making it hard to find even if you know it exists. The failure mode is completely silent — no error, no warning, just an empty collection that stays empty.
