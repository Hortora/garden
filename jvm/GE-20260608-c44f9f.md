## Context

When a REST search API accepts multiple entity identifiers (e.g. `group_ids: list[str]`) but its
response type carries no entity-level identifier in each result record, you cannot recover which
result belongs to which input entity after a batched call.

Example: Graphiti's `POST /search` accepts `group_ids: [...]` but `FactResult` has no `group_id`
field — only `uuid`, `name`, `fact`, and temporal fields.

## The Technique

Issue one REST call per entity instead of batching all entity IDs into one call. Each call's
results are tagged with the calling entity ID from the request context. Merge results afterward
(entity-order concatenation — all of entity-1's results first, then entity-2's, etc.).

```java
for (final String entityId : query.entityIds()) {
    final var req = new SearchRequest(
        List.of(compoundGroupId(tenantId, entityId)),  // single group_id per call
        query.question(),
        query.limit()
    );
    final var resp = client.search(req);
    resp.facts().stream()
        .map(f -> toMemory(f, entityId, domain, tenantId))  // entityId from call context
        .forEach(all::add);
}
```

**Merge order:** Entity-order concatenation — not score interleaving. Scores from separate
calls are not cross-comparable (each call normalises against its own result set).

## Why this is non-obvious

Batching all entity IDs into one call is the obvious efficiency choice. The attribution problem
only becomes apparent when implementing the `Memory.entityId` field population and realising
the response type has no identifier to recover it from. Per-entity calls cost N HTTP round-trips
vs 1, but for a high-value/low-frequency query path, correctness outweighs efficiency.
