## Convert sealed interface to enum when all variants become stateless after refactoring

**ID:** GE-20260602-e7824c
**Stack:** Java 17+
**Labels:** `#java` `#sealed-types` `#refactoring`
**What it achieves:** Removes unnecessary complexity when a sealed interface's only remaining variants are stateless, restoring the simpler enum design that was always appropriate.
**Context:** Sealed interfaces are the right choice when at least one variant carries data (fields, lambdas, or generics). When refactoring removes the data-carrying variant, the sealed interface becomes over-engineered — but nothing in the compiler flags this.

### The technique

Audit sealed interfaces when a data-carrying variant is removed. If all remaining variants are stateless (zero-field records or equivalent), convert the sealed interface to an enum:

**Before (over-engineered after data carrier was removed):**
```java
// Started as a sealed interface because Handled carried a Consumer<PlayerState>
// After Handled was extracted to a dedicated intent type, only stateless variants remain
public sealed interface ProductionDecision
        permits ProductionDecision.Proceed, ProductionDecision.Blocked {
    record Proceed() implements ProductionDecision {}
    record Blocked() implements ProductionDecision {}
}
// call sites: new ProductionDecision.Proceed() or new ProductionDecision.Blocked()
// switch: case ProductionDecision.Blocked b -> ...
```

**After (enum is correct):**
```java
public enum ProductionDecision { PROCEED, BLOCKED }
// call sites: ProductionDecision.PROCEED or ProductionDecision.BLOCKED
// switch: case BLOCKED -> ... (exhaustive switch still enforced)
```

Exhaustive switch still works on enums in Java — covering all constants without a default compiles cleanly and the compiler errors if a new constant is added without updating the switch.

### Alternatives considered

- **Keep the sealed interface** — works, but adds nesting, requires `new Proceed()` allocation at call sites, and the pattern implies data-carrying semantics to readers. Misleads future maintainers into thinking variants might be extended with data.

### Why this is non-obvious

A sealed interface with stateless variants compiles and works identically to an enum — there is no compiler signal to re-evaluate. Developers rarely audit sealed interfaces after refactoring; the original reason for the sealed type (data-carrying variant) is recorded in a commit message but not visible in the code. The code review step is what catches this — the smell is only visible to a fresh reader.

### When to use it

Apply when a refactoring removes the last data-carrying variant from a sealed interface. Invariant: **sealed interface → has at least one variant with fields; enum → all values stateless**. Track this invariant in code review when sealed types are modified.

Do not apply when:
- Any variant carries data (fields, lambda captures, generic parameters)
- The type is part of a public API that external callers already depend on via pattern matching
