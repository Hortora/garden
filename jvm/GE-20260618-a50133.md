### Symptom
`mvn compile` fails with `cannot find symbol: WorkflowExecutionFailed` and `method approved in record WorkflowExecutionCompleted cannot be applied to given types` across all worker modules — workers-http, workers-mcp, workers-camel, workers-script, workers-github-actions.

### Context
`casehub-engine-common` removed the `WorkflowExecutionFailed` record and the `WORKFLOW_EXECUTION_FAILED` event bus address (engine#463 — failures now handled in-process via `QuartzRetryService`). Same SNAPSHOT changed `WorkflowExecutionCompleted.approved()` from 4-arg to 5-arg (added `bindingName`).

Workers fire faults on **worker-specific** addresses (`CAMEL_WORKER_FAULT`, `HTTP_WORKER_FAULT`, `MCP_WORKER_FAULT`, `SCRIPT_WORKER_FAULT`, `GITHUB_ACTIONS_WORKER_FAULT`) — NOT on `WORKFLOW_EXECUTION_FAILED`. But they used `WorkflowExecutionFailed` as the **event payload type** on those different addresses. The address removal was correct; the type removal was a broader break that affected addresses the engine commit didn't touch.

### Root cause
The type `WorkflowExecutionFailed` was shared infrastructure — used as an event payload across addresses owned by different modules. Removing the address that introduced it did not account for other addresses carrying the same type. The coupling was invisible: no import-graph tool flags "this type is used as a payload on 5 different event bus addresses across 2 repos."

### Fix
Create a local `WorkerFaultEvent` record in `workers-common` with identical fields:

```java
public record WorkerFaultEvent(
    CaseInstance caseInstance, Worker worker, Capability capability,
    String inputDataHash, String eventLogId, Throwable cause) {}
```

Replace all `WorkflowExecutionFailed` references across all worker modules. For `WorkflowExecutionCompleted.approved()`, pass `null` for the new `bindingName` parameter — workers don't have binding context (it's engine-internal).

### Why this is non-obvious
The address removal (`WORKFLOW_EXECUTION_FAILED`) was correct and targeted. A developer reviewing the engine commit would see "removed the address and its record" and think the scope was clean. The non-obvious part is that the record was reused as a payload type on completely different addresses in a different repo. The symptom (`cannot find symbol`) points at the workers, not at the engine commit that caused it.

### Alternatives considered
- Keeping `WorkflowExecutionFailed` in engine-common as a shared event type: rejected — the engine explicitly moved away from event-bus-based failure handling. Keeping the type would preserve a dead abstraction.
- Using a generic `Map<String, Object>` payload: rejected — loses type safety and field documentation.
