## Context

You have a security guard (e.g. tenant assertion) that must be called by every adapter
implementing a SPI. The guard is defined as a `default` method on the blocking SPI interface.
Some adapters implement a *separate* reactive SPI directly (bypassing the blocking interface)
and cannot call the blocking SPI's default method — they have no reference to it.

## What developers reach for instead

- Put the assertion logic in the interface `default` method only — reactive-only adapters
  silently skip it, creating a multi-tenancy hole
- Duplicate the assertion in both SPI interfaces — logic drift over time
- Require all reactive adapters to also implement the blocking SPI — coupling, forces
  unnecessary delegation

## The Technique

Extract the enforcement logic to a `static` utility class in the same API module:

```java
// In the zero-dep API module (platform-api)
public final class MemoryPermissions {
    private MemoryPermissions() {}

    public static void assertTenant(String tenantId, CurrentPrincipal principal) {
        if (!principal.tenancyId().equals(tenantId))
            throw new SecurityException(
                "Tenant ID mismatch: claimed=" + tenantId
                + ", authenticated=" + principal.tenancyId());
    }
}
```

The blocking SPI interface delegates its `default` method to the static utility:

```java
public interface CaseMemoryStore {
    // ... abstract methods ...

    default void assertTenant(String tenantId, CurrentPrincipal principal) {
        MemoryPermissions.assertTenant(tenantId, principal);  // delegates — not reimplemented
    }
}
```

Reactive-only adapters (implementing `ReactiveCaseMemoryStore` but not `CaseMemoryStore`)
call the static utility directly:

```java
@Alternative @Priority(2)
@ApplicationScoped
public class GraphitiReactiveAdapter implements ReactiveCaseMemoryStore {

    @Inject CurrentPrincipal principal;

    @Override
    public Uni<String> store(MemoryInput input) {
        MemoryPermissions.assertTenant(input.tenantId(), principal);  // direct call
        return graphitiClient.store(input);
    }
}
```

## Why This Is Non-Obvious

A `default` method on `InterfaceA` is only accessible to classes that implement (or
extend) `InterfaceA`. A class implementing `InterfaceB` — even if semantically related —
cannot call `interfaceA.defaultMethod()`. The natural instinct is to put enforcement
logic in the interface that "owns" the concern, but this makes it unreachable from
sibling interfaces.

The static utility is the canonical Java solution for logic that must be shared across
type hierarchies without inheritance. Placing it in the same API module as both SPI
interfaces ensures it's available to any adapter regardless of which interface it implements.

## Caveat

In reactive contexts, `CurrentPrincipal` must be captured *before* entering the `Uni`
pipeline — the CDI `@RequestScoped` context is not guaranteed to be active on the
executor thread where the `Uni` subscription runs.
