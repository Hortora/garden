### Symptom

```
error: cannot find symbol
    .filter(a -> a >= threshold)
     ^
error: cannot find symbol
    .mapToObj(a -> (RiskDecision) gate(...))
     ^
error: method orElse in class OptionalDouble cannot be applied to given types
    .orElse(new RiskDecision.Autonomous());
```

Three compile errors on an `OptionalDouble` used in a functional chain.

### Context

Returning `OptionalDouble` from a helper method (e.g. `parseAmount()`) and trying to chain `.filter()`, `.mapToObj()`, and `.orElse(T)` — the way you would with `Optional<Double>`.

### Root cause

`OptionalDouble` is a primitive specialisation of `Optional<T>`. It exists to avoid boxing overhead and has a deliberately reduced API. The full `Optional<T>` methods — `filter(Predicate)`, `map(Function)`, `flatMap(Function)`, `mapToObj(DoubleFunction)`, `orElse(T)` — **do not exist** on `OptionalDouble`. Its `orElse` only accepts a primitive `double`, not an arbitrary `T`.

Java 21 `OptionalDouble` API in full: `of`, `empty`, `getAsDouble`, `isPresent`, `isEmpty`, `ifPresent`, `ifPresentOrElse`, `stream`, `orElse(double)`, `orElseGet`, `orElseThrow`, `equals`, `hashCode`, `toString`. Nothing else.

### Fix

Return `Optional<Double>` instead of `OptionalDouble` when you need functional chain methods:

```java
// Before (OptionalDouble — cannot chain filter/map)
default OptionalDouble parseAmount(Map<String, Object> context) {
    Object raw = context.get("amount");
    if (raw == null) return OptionalDouble.empty();
    try { return OptionalDouble.of(Double.parseDouble(raw.toString())); }
    catch (NumberFormatException e) { return OptionalDouble.empty(); }
}

// After (Optional<Double> — filter/map/orElse all work)
default Optional<Double> parseAmount(Map<String, Object> context) {
    Object raw = context.get("amount");
    if (raw == null) return Optional.empty();
    try { return Optional.of(Double.parseDouble(raw.toString())); }
    catch (NumberFormatException e) { return Optional.empty(); }
}

// Usage — now compiles correctly
return parseAmount(action.context())
    .filter(a -> a >= threshold)
    .map(a -> (RiskDecision) gate(type, action, prefs, reason))
    .orElse(new RiskDecision.Autonomous());
```

### Why this is non-obvious

`OptionalDouble` looks identical to `Optional<Double>` in usage — both are created with `of()` and `empty()`, both have `isPresent()` and `orElse`. The missing methods aren't visible until you try to call them. The compiler error points at the chain calls (`filter`, `mapToObj`) without explaining *why* they're absent. The natural first instinct is to look for the wrong overload, not to realise the entire method family is missing from the class.
