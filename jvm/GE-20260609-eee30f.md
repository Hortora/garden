## Context

You have an existing SPI interface with multiple CDI-discovered implementations. You want to add a new method (e.g. `create(String expression)`) but:

- Some existing implementations cannot meaningfully implement it (e.g. a lambda-type expression evaluator cannot be created from a string)
- You cannot force all existing implementors to update
- Callers need to know *before calling* whether an implementation supports the new method (to produce a helpful error rather than a caught exception)

The naive approach is to add an abstract method (breaks all existing implementations) or use try-catch around `UnsupportedOperationException` (testing by exception is slow and side-effectful when the method has constructor-level effects).

## Solution

Add two methods to the interface:

```java
// The new capability — default-throws for implementations that can't support it
default ExpressionEvaluator create(String expression) {
    throw new UnsupportedOperationException(
        type() + " does not support creation from string expressions");
}

// Probe flag — lets callers check support without calling create()
default boolean supportsStringCreation() {
    return false;
}
```

Implementations that support the new method override both:

```java
@Override
public ExpressionEvaluator create(String expression) {
    return new JQExpressionEvaluator(expression);
}

@Override
public boolean supportsStringCreation() {
    return true;
}
```

Implementations that don't support it (e.g. LambdaExpressionEngine) use both defaults — no code change needed. Their `supportsStringCreation()` returning false is semantically correct: the engine exists but is not string-constructable.

The registry then uses the boolean probe for clean error reporting:

```java
for (Engine engine : engines) {
    if (engine.type().equals(lang)) {
        if (!engine.supportsStringCreation()) {
            throw new UnsupportedOperationException(
                lang + " is registered but cannot be created from strings. " +
                "Use the Java DSL instead.");
        }
        return engine.create(expression);
    }
}
throw new IllegalArgumentException("No engine for lang: " + lang);
```

## Why this is non-obvious

Most developers would either:
1. Make the method abstract (forces all implementors to update)
2. Add a try-catch probe in the registry (creates domain objects as a side effect)
3. Use `instanceof` checks against a separate marker interface (adds coupling)

The paired `default throw + boolean flag` pattern keeps all language logic in the implementation, gives the registry a clean probe with no side effects, and produces actionable error messages that distinguish "unknown language" from "known language but Java-DSL-only".
