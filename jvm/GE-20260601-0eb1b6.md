## Symptom

`mvn verify` passes (all tests run, CDI starts cleanly), but running a single test class with `-Dtest=SpecificTest` fails at CDI augmentation:

```
jakarta.enterprise.inject.spi.DeploymentException: Found 1 deployment problems:
  Ambiguous dependencies for type io.example.SomeProvider and qualifiers [@Default]
```

The conflict is real — two `@DefaultBean` implementations are both on the classpath. Yet the full build doesn't surface it.

## Root Cause

`mvn verify` reuses the Quarkus augmented bytecode from a **previous** build run that was produced when one of the conflicting JARs was absent (e.g., before a dependency was added, or before an embedded Jandex index was introduced into a library). The cached augmented bytecode does not include the conflicting bean, so CDI sees only one `@DefaultBean` and starts cleanly.

A targeted `mvn test -Dtest=X` or `mvn clean verify` triggers **fresh augmentation** from the current classpath, which now includes both conflicting beans — surfacing the real CDI state.

## Fix

1. Run `mvn clean verify` to force fresh augmentation and surface all real CDI conflicts.
2. Resolve the ambiguity by excluding the less-capable bean via `quarkus.arc.exclude-types` in `application.properties` (test scope):

```properties
quarkus.arc.exclude-types=\
  io.example.NoOpSomeProvider,\
  io.example.OtherConflictingBean
```

## Why this is non-obvious

The full CI build passes. The stale cache provides false confidence. The conflict only appears when the build artifact cache is invalid — which happens ad-hoc (after `mvn clean`, first run after adding a dependency, or targeted `-Dtest=` runs). There is no warning that cached augmentation is being reused.

### Alternatives considered

- **Scope isolation** — changing one conflicting bean's JAR to `test` scope may not be feasible if the JAR is also needed at augmentation time for other beans
- **`beans.xml` alternatives section** — verbose and less targeted than `exclude-types`
