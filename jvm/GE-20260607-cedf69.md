### Symptom
Startup check or initialization logic is never executed. The bean is instantiated at startup (eager) but the method body never runs.

### Context
A bean annotated `@Startup @ApplicationScoped` is instantiated eagerly when the application starts. However, `@Startup` only forces eager instantiation — it does NOT auto-invoke any methods on the bean.

### What was tried (didn't work)
Annotating the startup check method with no lifecycle annotation, assuming `@Startup` would trigger it:
```java
@Startup
@ApplicationScoped
public class MyStartupChecker {
    public void checkConfiguration() { // never called
        // ...
    }
}
```

### Root cause
`@Startup` (Quarkus) = eager instantiation trigger. `@PostConstruct` = lifecycle callback after injection. They are orthogonal. Without `@PostConstruct`, the method is just a regular public method that happens to be on an eagerly-instantiated bean — nothing calls it.

### Fix
Add `@PostConstruct` to the method that should run at startup:
```java
@Startup
@ApplicationScoped
public class MyStartupChecker {
    @PostConstruct
    public void checkConfiguration() { // called immediately after bean creation
        // ...
    }
}
```

### Why this is non-obvious
`@Startup` name implies "this code runs at startup" — but it only ensures the *bean* exists at startup. The combination `@Startup @ApplicationScoped` says nothing about which method to call. `@PostConstruct` is the separate CDI mechanism for post-instantiation callbacks. Confusing two annotations with superficially similar purposes.
