## @Scheduled interval without $ prefix silently uses literal string as duration — fires at wrong frequency

### Symptom

Scheduler fires at an unexpected frequency or not at all. No error at startup or runtime. The `every` or `delay` attribute appears correct in the source.

### What was tried (didn't work)

- Checking the property value in application.properties — value is correct
- Verifying the bean is discovered — scheduler fires, just at wrong interval

### Root cause

MicroProfile Config property references in `@Scheduled` attributes require `${property}` syntax (with `$`). Using `{property}s` (without `$`) does not throw an error — Quarkus treats the literal string as a duration expression and either parses it as an invalid duration (defaulting to something unexpected) or silently fails.

### Fix

Always use `${}` for config references in scheduler attributes:

```java
// Wrong — silently uses literal string
@Scheduled(every = "{casehub.work.expiry.interval}s")

// Right
@Scheduled(every = "${casehub.work.expiry.interval}s")
```

### Why this is non-obvious

The missing `$` is a one-character typo that produces no compile error, no startup warning, and no runtime exception. The scheduler runs — just at the wrong interval. Without knowing to look for the `$`, the attribute looks syntactically correct.
