## Symptom

`@QuarkusTest` fails with "Unable to locate CDIProvider" or CDI injection points are null despite the test class being annotated correctly. The module uses `generate-code + generate-code-tests` goals (no `build` goal) and has no `quarkus-arc` dependency.

## Context

A Jandex library module (no `quarkus:build` goal) that needs one `@QuarkusTest` to verify CDI wiring (e.g., `@Inject EndpointRegistry` resolves to the expected `@Alternative @Priority(N)` implementation). The module previously had only `jakarta.enterprise.cdi-api` as a `provided` scope CDI dependency.

## Root Cause

`jakarta.enterprise.cdi-api` is annotation-only — it provides the CDI interfaces and annotations but no runtime implementation. `@QuarkusTest` needs an active CDI container to resolve injection points. Without `quarkus-arc` on the compile classpath, there is no CDI provider and Quarkus ARC cannot initialize.

The `generate-code` and `generate-code-tests` Maven goals process Quarkus annotations and generate code stubs but do NOT pull in the CDI runtime. The `build` goal does (it runs full augmentation), but adding `build` to a library module is wrong — it produces a runnable Quarkus artifact which library modules should not be.

## Fix

Add `quarkus-arc` as a compile dependency alongside the existing CDI API dep:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-arc</artifactId>
    <!-- no scope — compile -->
</dependency>
```

Also add `quarkus-maven-plugin` with `generate-code + generate-code-tests` goals (without `build`):

```xml
<plugin>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-maven-plugin</artifactId>
    <version>${quarkus.platform.version}</version>
    <extensions>true</extensions>
    <executions>
        <execution>
            <goals>
                <goal>generate-code</goal>
                <goal>generate-code-tests</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

The `jakarta.enterprise.cdi-api` `provided` dep can be removed once `quarkus-arc` is present — it brings the API transitively.

## Why This Is Non-Obvious

The distinction between CDI API (annotations only, no runtime) and CDI implementation (ARC, which provides both annotations and the runtime container) is easy to miss. Adding the `build` goal would also fix the problem but is the wrong solution — it produces an undesirable uber-jar and runs full augmentation at install time for a module that isn't an application. The correct fix (add `quarkus-arc`, keep `generate-code` only) is not documented for the library module pattern.
