## Double.NaN silently passes `< 0.0 || > 1.0` range guard — IEEE 754 comparisons with NaN always return false

### Symptom

A Java record compact constructor validates a `double` field with a range guard:

```java
public AgentOutcome {
    if (confidence < 0.0 || confidence > 1.0)
        throw new ValidationException("confidence", "must be between 0.0 and 1.0");
}
```

`new AgentOutcome("id", result, Double.NaN, now, null)` does not throw. The record is constructed successfully and `confidence()` returns `NaN`. Any computation that uses it (averages, Wilson scores, comparisons) silently propagates `NaN` through the result.

### Context

Arises in any Java code that validates a `double` primitive with a range check. Records and compact constructors surface it more often because they are the idiomatic place for constructor validation since Java 16. The DB-level `CHECK (confidence BETWEEN 0 AND 1)` catches it at persist time — but any in-memory or pre-persistence code path accepts NaN without complaint.

`Double.POSITIVE_INFINITY` and `Double.NEGATIVE_INFINITY` are handled correctly by the existing guard — `POSITIVE_INFINITY > 1.0` is `true` and `NEGATIVE_INFINITY < 0.0` is `true`. Only `NaN` escapes.

### Root cause

IEEE 754 specifies that all relational comparisons (`<`, `>`, `<=`, `>=`) involving NaN return `false`. The JVM follows this exactly. `Double.NaN < 0.0` is `false`; `Double.NaN > 1.0` is `false`. Both sides of the `||` are false, so the guard is never entered. This is correct IEEE 754 behaviour — not a JVM bug — but it contradicts the intuition that NaN "feels" out of range.

### Solution 1 — Explicit NaN check
**Approach:** Prefix the range condition with `Double.isNaN()`.
**Pros:** Readable, intent is explicit.
**Cons:** Extra condition that must be remembered every time a range guard is written.

```java
if (Double.isNaN(confidence) || confidence < 0.0 || confidence > 1.0)
    throw new ValidationException("confidence", "must be between 0.0 and 1.0");
```

`Double.isNaN()` is a static method that returns `true` only for NaN — it does not use comparison operators and is immune to IEEE 754 comparison semantics.

### Solution 2 — Inverted range check (structural NaN rejection)
**Approach:** Invert the condition: test that the value IS in range, then negate.
**Pros:** NaN, Infinity, and out-of-range are all rejected by the same expression — no separate `isNaN()` call. Leverages IEEE 754 semantics rather than working around them.
**Cons:** Less familiar pattern; the negation requires a moment's thought to parse.

```java
if (!(confidence >= 0.0 && confidence <= 1.0))
    throw new ValidationException("confidence", "must be between 0.0 and 1.0");
```

IEEE 754: `NaN >= 0.0` is `false`, so `(NaN >= 0.0 && NaN <= 1.0)` is `false`, so `!(false)` is `true` — the guard fires. `Double.POSITIVE_INFINITY >= 0.0` is `true` but `POSITIVE_INFINITY <= 1.0` is `false` — guard fires. All edge cases handled by one expression.

For records, pair either solution with a test:

```java
@Test
void nan_confidence_throws() {
    assertThatThrownBy(() -> new AgentOutcome("id", result, Double.NaN, now, null))
        .isInstanceOf(ValidationException.class);
}
```

Standard boundary tests (`-0.001`, `1.001`, `0.0`, `1.0`) never exercise NaN — this test must be added explicitly.

### Why this is non-obvious

The guard looks complete. Infinity is caught correctly. NaN appears to be "more invalid" than a value that is merely out of range, so it feels like the guard should catch it. Developers who know IEEE 754 in theory often forget it in the context of validation code because they are thinking about the business constraint, not the numeric representation. The failure is completely silent — no exception, no warning, just a record with a `NaN` field that poisons any computation it touches.

### Why this fix

`Double.isNaN()` is the only reliable way to detect NaN in Java. Comparison operators cannot be used because the IEEE 754 spec defines NaN as unordered — it is not equal to, less than, or greater than any value including itself. `Double.isNaN(x)` is equivalent to `x != x` (the only comparison that returns true for NaN) but is far more readable.
