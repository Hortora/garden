## Context

Bridge-style handler methods that isolate one exception type (e.g., policy exceptions) while letting another type propagate (e.g., infrastructure exceptions). A common pattern:

```java
try {
    policy.get().decide(event).ifPresent(request ->
        service.create(request));
} catch (Exception e) {
    LOG.warnf(e, "policy threw — message ignored");
    return;
}
```

## Symptom

`service.create()` throws a database timeout. The log emits `"policy threw — message ignored"` even though the policy succeeded. The infrastructure failure is silently absorbed and misattributed. In production, this hides real errors behind policy-related log messages.

## Root Cause

`Optional.ifPresent()` calls the lambda synchronously. Any exception thrown inside the lambda propagates up through `ifPresent()` and exits the `try` block — where the `catch(Exception e)` catches it regardless of origin. The catch-block cannot distinguish between an exception from `decide()` and one from the lambda body.

## Fix

Separate the policy call into its own try/catch with early return. Keep infrastructure calls outside the try:

```java
final Optional<WorkItemCreateRequest> decision;
try {
    decision = policy.get().decide(event);
} catch (Exception e) {
    LOG.warnf(e, "policy threw for channel %s — ignored", event.channelName());
    return;
}

decision.ifPresent(request ->
    service.create(request));   // infrastructure exceptions propagate naturally
```

Now policy exceptions are caught with specific context; infrastructure exceptions propagate out of `onMessage()` to the caller's safety net — and log messages are accurate to the actual failure.

## Why This Is Non-Obvious

The try/catch visually wraps only `policy.get().decide(event)` — readers expect the lambda body to be outside its scope. Java's syntax makes the lambda body appear as a continuation of the `ifPresent()` call rather than as a separate execution context. The catch fires on any exception from the entire expression, including the lambda's synchronous body, with no compile-time or runtime indication that the scope is wider than it looks.

Code review missed this across two rounds — the description ("catch wraps only the policy call") was accurate in intent but not in execution.
