## Symptom

An `@ObservesAsync` observer method is never called, even when `event.fire(payload)` is invoked with a matching payload type. No error is thrown.

## Context

CDI's `Event<T>` has two separate delivery mechanisms:

- `event.fire(payload)` — synchronous delivery; reaches only `@Observes` observers; blocks the calling thread until all observers complete
- `event.fireAsync(payload)` — asynchronous delivery; reaches only `@ObservesAsync` observers; returns a `CompletionStage`

A single `fire()` call does NOT reach async observers. Both calls are required if you have both sync and async observers for the same event type.

## Root cause

CDI spec design. The two observer kinds are intentionally separate delivery channels with different threading and error-handling semantics. `fire()` is transactional-safe (runs in the caller's transaction); `fireAsync()` runs outside the caller's transaction on a managed executor.

## Fix

When you have both `@Observes` and `@ObservesAsync` observers for the same payload type, call both:

```java
// reaches @Observes (sync) observers
try {
    event.fire(payload);
} catch (Exception e) {
    log.warnf(e, "Sync observer failed");
}

// reaches @ObservesAsync (async) observers
event.fireAsync(payload)
    .exceptionally(ex -> {
        log.warnf(ex, "Async observer failed");
        return null;
    });
```

If you only have `@ObservesAsync` observers, use only `fireAsync()`.

## Why this is non-obvious

The CDI docs describe `fire()` as firing the event — not "firing the synchronous half of the event". The existence of two separate delivery methods for what looks like one event feels like it should be handled transparently by the CDI container based on observer annotation. Many developers assume `fire()` handles both, leading to silently dead async observers.

### Why this fix

There is no single method that reaches both observer types. The split is by design — async observers run on a managed executor outside the caller's transaction, so they cannot participate in the sync delivery path.

### Alternatives considered

- Single `fireAsync()` for both: rejected — `fireAsync()` does NOT reach `@Observes` sync observers either; the split is symmetric
