### Symptom
`Map.of("key1", SealedType.variant1("x"), "key2", SealedType.variant2(5))` compiles
but cannot be assigned to `Map<String, SealedType>` — compiler infers
`Map<String, Record & SealedType>` which is not a subtype of `Map<String, SealedType>`.

### Context
Occurs with any Java sealed interface whose permits are records. `Map.of()` uses the
most specific common type of the value arguments. When mixing `StringVal` and `NumberVal`
(both records implementing a sealed interface), Java infers the intersection type
`Record & SealedType` rather than the sealed interface itself.

### What was tried (didn't work)
- `var m = Map.of(...)` — var infers the intersection type, same error at call site
- Casting individual values — verbose and fragile

### Root cause
Java's type inference for `Map.of()` computes the LUB (least upper bound) of the
value types. For records implementing a sealed interface, the LUB is `Record & SealedType`
(the intersection of their common supertypes), not the sealed interface alone. This
intersection type is not assignable to `Map<String, SealedType>` due to invariant generics.

### Fix
Use an explicit variable type or type witness:
```java
// Option 1: explicit variable type (preferred — readable)
Map<String, FeatureValue> features = Map.of(
    "budget", FeatureValue.number(1000),
    "season", FeatureValue.string("winter"));

// Option 2: type witness (inline, no variable needed)
method(Map.<String, FeatureValue>of("budget", FeatureValue.number(1000)));
```

### Why this is non-obvious
The compiler error message says "incompatible types: Map<String, Record & FeatureValue>
cannot be converted to Map<String, FeatureValue>" — which looks like a variance issue.
The actual cause is LUB inference, and the fix (explicit typing) is mechanical but
non-obvious from the error message alone.
