## Context

A renderer supports multiple output formats (e.g. PROSE, MARKDOWN, A2A_CARD). Each format has different LLM input requirements — PROSE and MARKDOWN don't need numeric routing signals (qualityHint, epistemicDomains), but A2A_CARD does. A shared payload builder produces the descriptor node, which is used both as the cache key input and (after optional stripping) as the LLM enrichment payload.

## The naive approach and its problem

**Strip at the LLM call boundary:**
```java
ObjectNode descriptorNode = buildDescriptorPayload(descriptor);  // all fields, all formats
// ...
if (usesEnrichment(format)) {
    ObjectNode llmPayload = buildLlmPayload(descriptorNode);
    stripNumericFields(llmPayload);  // strip before sending to LLM
    enrichment = enrichmentStep.enrich(llm, llmPayload);
}
```

Problem: `descriptorNode` (with all fields) is used for the hash. Changing `qualityHint` changes the PROSE cache key even though PROSE output doesn't depend on `qualityHint`. Every format cache is invalidated by format-irrelevant field changes.

## The technique

Make `buildDescriptorPayload` itself format-aware. The hash input is the format-specific payload:

```java
ObjectNode buildDescriptorPayload(AgentDescriptor descriptor, RenderFormat format) {
    // ...
    for (AgentCapability cap : descriptor.capabilities()) {
        capNode.put("name", cap.name());
        if (format == RenderFormat.A2A_CARD) {
            // numeric signals: A2A_CARD only
            if (cap.qualityHint() != null)      capNode.put("qualityHint", cap.qualityHint());
            if (cap.latencyHintP50Ms() != null) capNode.put("latencyHintP50Ms", cap.latencyHintP50Ms());
            // ...
        }
        // qualitative descriptors: all formats
        if (cap.inputTypes() != null) { ... }
    }
}

StageOneResult buildStage1(AgentDescriptor descriptor, AgentPromptContext context) {
    ObjectNode descriptorNode = buildDescriptorPayload(descriptor, context.format());
    String descriptorHash = fingerprint(descriptorNode.toString());
    // ...
}
```

Result: changing `qualityHint` only invalidates the A2A_CARD cache entry — not PROSE or MARKDOWN. Each format's cache key precisely reflects what determines that format's output.

## Why this is non-obvious

The instinct is to build one payload and filter at the boundary where the filtering is needed (the LLM call). Making the payload builder format-aware pushes the filtering upstream to where the hash is computed — which feels backwards, since the payload builder seems like it should be format-agnostic infrastructure. The insight is that "what the hash should cover" and "what the LLM should receive" are two different questions answered by two different layers, and the payload builder needs to answer both simultaneously by being format-aware from the start.

## Trade-off

The format-discriminated approach trades payload builder simplicity for cache key precision. The hash input for A2A_CARD may include fields that the A2A enrichment LLM doesn't use (they are dead data to the LLM but necessary in the hash for structural assembly correctness — see GE-20260613-3fa95a). This is the correct trade-off: hash correctness is non-negotiable; LLM payload cleanliness is a preference.
