## Symptom

Tests fail with `NULL not allowed for column "TENANCY_ID"` (or similar NOT NULL constraint violation) after a Flyway production migration adds a NOT NULL column with a DEFAULT value to an existing table. Production works correctly. The error message names the column but gives no indication that the missing DEFAULT is the root cause.

## Context

Quarkus test setup commonly uses:
- `quarkus.flyway.migrate-at-start=false` — Flyway disabled in tests
- `quarkus.hibernate-orm.database.generation=drop-and-create` — Hibernate creates the schema from JPA annotations

When a production Flyway migration adds a NOT NULL column with a DEFAULT:
```sql
ALTER TABLE work_item_template
  ADD COLUMN tenancy_id VARCHAR(255) NOT NULL DEFAULT '278776f9-e1b0-46fb-9032-8bddebdcf9ce';
```

Hibernate generates the schema from the JPA entity definition. If the JPA entity has `@Column(nullable = false)` without `@ColumnDefault`, Hibernate creates:
```sql
tenancy_id VARCHAR(255) NOT NULL
```

No DEFAULT. Any INSERT that doesn't provide a value for `tenancy_id` fails immediately.

## What was tried

Searching for `LEDGER_SUBJECT_SEQUENCE` (the symptom described in the session handover) — this was wrong. The actual failing column was `TENANCY_ID`. The misleading handover description added diagnostic time.

## Root cause

Flyway DEFAULT values exist only in the SQL migration script — they are not embedded in the JPA `@Column` annotation and are not propagated to Hibernate's schema generator. Hibernate uses JPA annotations as the sole source of truth for drop-and-create. The DEFAULT disappears when Flyway doesn't run.

## Fix

Explicitly set the field in every test fixture that creates the affected entity:

```java
// Before (fails after V35__tenancy_id.sql adds NOT NULL tenancy_id)
WorkItemTemplate t = new WorkItemTemplate();
t.name = name;
t.persist();

// After
WorkItemTemplate t = new WorkItemTemplate();
t.name = name;
t.tenancyId = "278776f9-e1b0-46fb-9032-8bddebdcf9ce";  // platform default
t.persist();
```

The test fixture must own the value. The DEFAULT only applies when Flyway runs the migration.

## Why this is non-obvious

The Flyway migration looks correct — `NOT NULL DEFAULT '...'` is standard SQL. Production is unaffected. The error message (`NULL not allowed for column "TENANCY_ID"`) points at the column with no hint that the cause is a missing DEFAULT in the Hibernate-generated schema. A developer would naturally look at the insert code, not at the Flyway/Hibernate schema gap. The problem only surfaces after adding a new NOT NULL Flyway column — existing columns were added before drop-and-create was in place.

## Alternatives considered

- **Add `@ColumnDefault` to the JPA entity** — forces the DEFAULT into the Hibernate schema. Works, but couples the JPA entity to an implementation detail of the production migration value, and requires changes to a peer repo entity.
- **Enable Flyway in tests** — conflicts with `drop-and-create`; Flyway runs before Hibernate, then Hibernate drops everything.
- **Switch to Flyway-only in tests** — removes `drop-and-create`; requires all migrations to be H2-compatible. Higher maintenance burden.
