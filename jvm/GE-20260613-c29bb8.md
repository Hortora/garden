## SseEventSink.send() Is Async — Synchronous sink.close() Immediately After Throws "Response Already Written"

### Symptom

Every time a terminal SSE event is sent, the Quarkus server logs:

```
ERROR [io.quarkus.vertx.http.runtime.QuarkusErrorHandler] HTTP Request to /path/stream failed,
  error id: ...: java.lang.IllegalStateException: Response has already been written
    at io.vertx.core.http.impl.Http1xServerResponse.closeHandler(Http1xServerResponse.java:726)
    at org.jboss.resteasy.reactive.server.jaxrs.SseEventSinkImpl.sendInitialResponse(SseEventSinkImpl.java:75)
```

The SSE client receives the event correctly — the log error is the only visible symptom.

### Context

Sending a terminal SSE event and then closing the sink synchronously:

```java
// ❌ Wrong — sink.close() fires before send() completes
sink.send(sse.newEventBuilder().name("event").data(json).build());
if (!sink.isClosed()) sink.close();  // IllegalStateException
```

`SseEventSinkImpl.send()` calls `response.write()` which is async (Vert.x `CompletionStage`).
The HTTP response headers and body are committed only when the write completes. Calling
`close()` synchronously triggers `sendInitialResponse()` internally, which tries to register
a close handler on an already-written (or still-being-written) response — hence the exception.

### Fix

Chain `sink.close()` inside the `CompletionStage` continuation:

```java
// ✅ Correct — close after send() completes
sink.send(sse.newEventBuilder().name("event").data(json).build())
    .thenRun(() -> { if (!sink.isClosed()) sink.close(); });
```

For consumer lambdas that also need to deregister:

```java
sink.send(sse.newEventBuilder().name("task_status_update").data(json).build())
    .whenComplete((v, ex) -> {
        if (ex != null) {
            // send failed — deregister consumer
            registry.deregister(correlationId, self);
        } else if (terminal) {
            if (!sink.isClosed()) sink.close();
            registry.deregister(correlationId, self);
        }
    });
```

### Why this is non-obvious

`send()` in every other API (file write, socket write, etc.) is either synchronous or
accepts a callback. The JAX-RS `SseEventSink.send()` signature appears to return `void` at
the usage level — the `CompletionStage<?>` return value is easy to ignore. Nothing in the
Quarkus or JAX-RS docs warns that calling `close()` synchronously after `send()` is
incorrect. The error also appears as a server-side log, not as an exception thrown to the
caller — so it's easy to miss in development.
