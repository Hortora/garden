## Symptom

Adding a new `@ConfigProperty(name = "casehub.work.capability-validation")` to a CDI bean works locally (default value applies). When a `@QuarkusTest` test profile's `getConfigOverrides()` sets that property, the application fails to start with a SmallRye Config strict mapping violation — "unknown property" — even though the property is correctly declared in the CDI bean.

## Context

`WorkItemsConfig` uses `@ConfigMapping(prefix = "casehub.work")`. SmallRye Config's strict mode is the default for `@ConfigMapping` interfaces. A separate CDI bean `CapabilityValidator` declares `@ConfigProperty(name = "casehub.work.capability-validation")`. Without a test profile override everything works — the `@ConfigProperty` resolves from the default value. With a test profile, SmallRye validates the full config map at startup and rejects `casehub.work.capability-validation` as unmapped within the `casehub.work.*` namespace owned by `WorkItemsConfig`.

## Root Cause

`@ConfigMapping` with a prefix claims ownership of that prefix in strict mode. Any `casehub.work.*` key not explicitly declared as a method in the interface is rejected as an unknown property — regardless of whether it's legitimately used via `@ConfigProperty` in another bean. The strict mapping validator runs at startup and treats the entire prefix as an exclusive namespace.

## Fix

Declare the property in the config interface with `@WithDefault`:

```java
@ConfigMapping(prefix = "casehub.work")
public interface WorkItemsConfig {
    // ... existing methods ...
    
    @WithDefault("PERMISSIVE")
    ValidationMode capabilityValidation();
}
```

The CDI bean can still inject it via `@ConfigProperty` — both access paths coexist. Alternatively, use a different prefix for the new property that doesn't share the `@ConfigMapping` prefix.

## Why This Is Non-Obvious

`@ConfigProperty` and `@ConfigMapping` appear to be independent mechanisms that should compose. The failure only surfaces when a test profile actively sets the property — local tests pass silently because the `@WithDefault` on the `@ConfigProperty` is used before SmallRye validates the full map. The error message ("unknown property") points at SmallRye's validator, not at the `@ConfigMapping` interface, making the connection non-obvious.

**Masking by OOMKill:** A heavy Keycloak DevServices container starting during `@QuarkusTest` can suppress this error entirely. If the container is OOMKilled before Quarkus finishes booting, the test reports `ContainerLaunchException` rather than `SRCFG00050` — the config validator never runs. Disabling the unnecessary DevServices container causes Quarkus to boot quickly and surface the real validation failure. If you see intermittent OOM failures on SCIM/OIDC tests, disable Keycloak DevServices first and re-run before debugging config.
