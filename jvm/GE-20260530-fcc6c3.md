### Symptom

TTL cache returns stale values indefinitely even after the TTL has expired. The cache entry
never refreshes despite the expiry check passing.

### Context

Any hand-rolled TTL cache using `ConcurrentHashMap` where entries carry an `expiresAt` field
and `putIfAbsent` is used to avoid duplicate loads.

### What was tried (didn't work)

```java
// WRONG — when key exists but is expired, putIfAbsent is a no-op
CacheEntry existing = cache.get(key);
if (existing == null || existing.isExpired()) {
    CacheEntry fresh = new CacheEntry(load(key), now().plus(ttl));
    cache.putIfAbsent(key, fresh); // stale entry still present — putIfAbsent sees the key and skips
    existing = cache.get(key);    // still returns the stale entry
}
return existing.value();
```

### Root cause

`putIfAbsent(k, v)` only inserts when the key is **absent**. If the key exists (even with an
expired value), `putIfAbsent` is a no-op and the stale entry remains. The expiry check on `get()`
passes, but the replacement never happens.

### Fix

Atomically remove the expired entry before attempting the fresh put:

```java
CacheEntry existing = cache.get(key);
if (existing != null && existing.isExpired()) {
    cache.remove(key, existing); // atomic: only removes if value == existing (no lost updates)
    existing = null;
}
if (existing == null) {
    CacheEntry fresh = new CacheEntry(load(key), now().plus(ttl));
    CacheEntry racing = cache.putIfAbsent(key, fresh);
    existing = racing != null ? racing : fresh; // winner-takes-all on concurrent load
}
return existing.value();
```

`remove(key, existing)` is an atomic conditional remove — it only succeeds if the current
mapping is still `existing`. Two threads racing to evict the same stale entry: one wins the
remove, both then race on `putIfAbsent` — only one inserts, the other reads back the winner.

### Why this is non-obvious

`putIfAbsent` semantics are clear ("if absent insert") but the implication for TTL caches
is not: the key is never absent unless explicitly removed. An expiry check on `get()` doesn't
remove the entry — you must remove it first before `putIfAbsent` can replace it.
