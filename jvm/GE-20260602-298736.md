## Symptom

Wrapping a void side-effecting call in a Mutiny `Uni.createFrom().item(...)` to run it on a worker pool:

```java
public Uni<Void> record(OutcomeRecord record) {
    return Uni.createFrom()
              .item(() -> { blocking.record(record); return null; })  // compile error
              .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());
}
```

The compiler rejects this with a type mismatch: `item(Supplier<T>)` infers `T` as `Object` when the lambda returns `null`. The return type is `Uni<Object>`, not `Uni<Void>`.

## Fix

Add an explicit type witness on the `item` call:

```java
return Uni.createFrom()
          .<Void>item(() -> { blocking.record(record); return null; })
          .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());
```

The `<Void>` witness tells the compiler that `T = Void`, making the lambda return type `Void` (which `null` satisfies) and the expression type `Uni<Void>`.

## Why this is non-obvious

`null` has no type in Java — it's assignable to any reference type. The compiler cannot infer `T = Void` from `return null` alone; it defaults to `Object`. The fix (a type witness on the method call) is an obscure Java generics feature that most developers only encounter in edge cases.

The alternatives — `return (Void) null` or extracting a `Supplier<Void> s = () -> { ...; return null; }` variable — also work but are less idiomatic. `(Void) null` is technically an unchecked cast and reads like noise. The type witness is the intended Java mechanism for this scenario and matches the Mutiny documentation examples for void pipelines.
