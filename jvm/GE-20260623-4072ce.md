A Quarkus library module adds `jackson-databind` as a compile dependency and
creates a `@ApplicationScoped` CDI bean that injects `ObjectMapper` via `@Inject`.
The library's own unit tests pass (plain JUnit, no CDI container). The full
reactor build fails with a class-loading error in a **downstream** module that
runs `@QuarkusTest`.

### Symptom

```
TestEngine with ID 'junit-jupiter' encountered a critical issue during test discovery:
ClassSelector [className = '...WebhookRouterTest'] resolution failed
Cause: PreconditionViolationException: Could not load class with name: ...WebhookRouterTest
```

The error is a class-loading failure in a module that **did not change**. No
"UnsatisfiedResolutionException" or "missing producer" message appears. The
failure looks like a test infrastructure problem, not a dependency issue.

### Root cause

Quarkus augmentation in the downstream module discovers the library's CDI bean
via Jandex (the library has the `jandex-maven-plugin`). ArC tries to resolve
the `ObjectMapper` injection point. `jackson-databind` provides the
`ObjectMapper` class but **no CDI producer**. The producer comes from
`quarkus-jackson` (or `quarkus-rest-jackson`). Without either on the classpath,
augmentation fails during class resolution — surfacing as a class-loading error,
not a CDI error.

### Fix

Replace `jackson-databind` with `quarkus-jackson` in the library module's
`pom.xml`. `quarkus-jackson` transitively includes `jackson-databind` and
provides the `ObjectMapper` CDI producer. It does NOT pull in REST
infrastructure (unlike `quarkus-rest-jackson`).

```xml
<!-- ❌ BAD: compiles but breaks downstream @QuarkusTest -->
<dependency>
    <groupId>com.fasterxml.jackson.core</groupId>
    <artifactId>jackson-databind</artifactId>
</dependency>

<!-- ✅ GOOD: provides ObjectMapper CDI producer -->
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-jackson</artifactId>
</dependency>
```

### Why this is non-obvious

Three layers of indirection hide the root cause:

1. The library's own tests pass — they create `ObjectMapper` manually, no CDI
2. The error appears in a downstream module that didn't change
3. The error message says "class loading failed" not "missing CDI producer"

The natural debugging path ("what changed in the failing module?") leads nowhere
because nothing changed there. The real cause is a new injection point in an
upstream library that Jandex made visible to the downstream module's augmentation.

### Alternatives considered

- **`quarkus-rest-jackson`** — provides the producer but also pulls in REST
  infrastructure (`quarkus-rest`), which is unwanted in a library module
- **`@DefaultBean` ObjectMapper producer in the library** — would work but
  creates a separate ObjectMapper instance, defeating the purpose of injecting
  the app's shared mapper (GE-20260621-629712 rule 1)
- **Making ObjectMapper optional via `Instance<ObjectMapper>`** — adds
  complexity; the adapter always needs an ObjectMapper, so optional lookup
  is misleading
