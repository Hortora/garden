### What Happened

Developer enables a named `@Filter` on the Hibernate `Session` (e.g. a per-request tenant
filter) expecting it to scope all entity lookups, including primary-key loads via
`EntityManager.find()` or Panache's `Entity.findById(id)`. The filter has no effect on
those calls — they return entities regardless of the filter condition.

### Symptom

`session.enableFilter("tenantFilter").setParameter("tenantId", tenantId)` is called before
the request. HQL/JPQL/Criteria queries correctly add the filter condition. But
`Entity.findById(id)` returns entities from any tenant — the filter is silently ignored.

### Root Cause

`SingleIdLoadPlan` (Hibernate 7's mechanism for primary-key entity loading) builds its
`JdbcSelect` from a SQL AST **at session-factory startup**, in the constructor:

```java
// SingleIdLoadPlan constructor — runs once at startup
this.jdbcSelect =
    sessionFactory.getJdbcServices().getJdbcEnvironment()
        .getSqlAstTranslatorFactory()
        .buildSelectTranslator(sessionFactory, sqlAst)
        .translate(null, ...);
```

This pre-compiled `JdbcSelect` is a cached, immutable SQL string. When `load()` is called
at runtime, Hibernate executes this pre-built SQL directly — it never consults the session's
enabled filter state.

`@Filter` conditions are incorporated into the SQL AST only for queries compiled at runtime
(HQL/JPQL/Criteria API), where Hibernate builds a fresh SQL AST per query-invocation and
includes the filter predicate at that point. Primary-key loads bypass this path entirely.

### Fix

Do not rely on `@Filter` for primary-key tenant isolation. Use an explicit HQL query that
goes through the runtime compiler, where the filter IS applied:

```java
// Does not respect @Filter — SQL pre-compiled at startup
Entity entity = Entity.findById(id);

// Respects @Filter — compiled at runtime with filter predicate
Entity entity = Entity.find("id = ?1", id).firstResult();
```

Or add the tenant predicate explicitly in the query (no @Filter needed):

```java
Entity entity = Entity.find("id = ?1 AND tenantId = ?2", id, tenantId).firstResult();
```

### Why This Is Non-Obvious

Hibernate documentation describes `@Filter` as applying to "entity loads, collection
fetches, and queries." The phrase "entity loads" implies it should work for `find()`/`get()`.
The actual mechanism — `SingleIdLoadPlan` with a startup-compiled SQL AST — is an internal
implementation detail not mentioned in user-facing documentation. There is no warning or
error when the filter is enabled but ignored on PK lookups.

### Why this fix

The explicit HQL query (`find("id = ?1 AND tenantId = ?2", ...)`) is preferred over
converting all `findById` calls to `find("id = ?1", id).firstResult()` with a `@Filter`.
While the latter reduces per-call-site change count, it introduces implicit behavior:
the `find("id = ?1", ...)` call looks like a normal query, hiding the fact that tenant
scoping depends on the filter being enabled somewhere else. For GDPR-obligated or
security-critical code, explicit call sites are auditable; the `@Filter` approach requires
understanding the infrastructure to verify security. Cost: same number of call-site changes
in both approaches.
