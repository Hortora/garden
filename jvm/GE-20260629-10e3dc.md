### Symptom
`OnnxInferenceModel` throws `ModelLoadException: Model must have 'attention_mask' input, found: [input_ids, input_mask, segment_ids]` at startup. The ONNX model is valid and runs correctly in Python — it just uses different input names.

### Context
Loading `prithivida/Splade_PP_en_v1` ONNX model via `casehub-inference-quarkus`. The model was exported from HuggingFace and uses the original BERT naming convention. The Java inference runtime validates against HuggingFace naming convention.

Two naming conventions exist for the same tensors:

| Tensor | Original BERT | HuggingFace |
|--------|--------------|-------------|
| Padding mask | `input_mask` | `attention_mask` |
| Segment IDs | `segment_ids` | `token_type_ids` |

The HuggingFace `transformers` library normalises these internally during inference, so Python code works with either. But raw ONNX exports preserve whatever names the original model checkpoint used.

### What was tried
Loading the SPLADE model in the Quarkus engine with `@LookupIfProperty`-gated CDI producer. Model path correctly configured. Engine startup failed with validation error.

### Root cause
`OnnxInferenceModel` constructor validates that ONNX model inputs include `attention_mask` and `token_type_ids` (HF convention). The SPLADE model uses `input_mask` and `segment_ids` (original BERT convention). Same tensors, different names — the validation is too strict.

This is not documented anywhere — neither the ONNX export docs nor the model card mention which naming convention the export uses. The only way to discover it is `onnxruntime.InferenceSession.get_inputs()`.

### Fix
Rename inputs in the ONNX graph using Python `onnx` library:

```python
import onnx

model = onnx.load("model.onnx")
rename_map = {"input_mask": "attention_mask", "segment_ids": "token_type_ids"}

for inp in model.graph.input:
    if inp.name in rename_map:
        inp.name = rename_map[inp.name]
for node in model.graph.node:
    for i, name in enumerate(node.input):
        if name in rename_map:
            node.input[i] = rename_map[name]

onnx.save(model, "model.onnx")
```

The proper fix is for `OnnxInferenceModel` to accept both naming conventions (casehubio/neural-text#51).

### Why this is non-obvious
ONNX models are treated as opaque binaries — you don't inspect input names before loading. The error message names the expected and actual inputs but doesn't suggest the fix is a simple rename. A developer would assume the model is incompatible rather than differently named.
