### Symptom
Code calling a method that was removed in a SNAPSHOT dependency compiles and runs cleanly locally, but CI fails with a compilation error: `no suitable method found for methodName(...)`.

### Context
Project A publishes `artifact:1.0-SNAPSHOT`. Project B consumes it and calls `service.complete(id, actor, resolution)` (3 args). Project A removes that overload and replaces it with a 4-arg form. Project B's local `~/.m2` still has the old SNAPSHOT JAR from before the removal. CI resolves a fresh copy of the SNAPSHOT from GitHub Packages/Nexus/Artifactory.

### Root cause
Maven SNAPSHOT resolution uses the cached `~/.m2` jar unless forced to update. The local build compiles against the old 3-arg method. CI always resolves fresh, gets the new jar with only 4-arg and 6-arg forms, and the compilation fails. There is no local signal that the API has diverged.

### Fix
Before concluding a build is locally stable after a cross-repo SNAPSHOT dependency changes its API:
```bash
mvn install -DskipTests -f <path-to-dependency>/pom.xml  # refresh local cache
mvn test -pl <affected-module>                            # retest against fresh jar
```
Or force update globally: `mvn -U test -pl <module>`.

### Why this is non-obvious
Local compilation success is normally authoritative. SNAPSHOT dependencies are the exception — the version string stays the same while the artifact content changes. CI resolves fresh; local resolves cached. The only symptom is a CI-only compilation error on a method that clearly exists in the local codebase, pointing to the wrong place entirely.
