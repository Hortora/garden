### Symptom

CDI reports "Unsatisfied dependency for type `Foo`" even though:
- A `@ApplicationScoped @DefaultBean` implementation of `Foo` is on the compile classpath
- The implementing JAR contains a valid `META-INF/jandex.idx`
- The implementing class is listed in that index

### Context

A library JAR (`impl-lib`) provides `@DefaultBean class NoOpFoo implements Foo`. A separate JAR (`api-lib`) defines the `Foo` interface. `api-lib` ships without a Jandex index. A downstream module depends on both (directly or transitively).

Quarkus ARC processes the Jandex index of `impl-lib` and finds `NoOpFoo`. When it tries to validate the bean — checking the type hierarchy, qualifiers, and injection points — it needs to read metadata about `Foo` from its defining JAR. Because `api-lib` has no Jandex index, ARC cannot resolve the supertype and silently discards `NoOpFoo` from the CDI graph.

### Root cause

Quarkus ARC bean discovery is index-driven. All types involved in a bean's type hierarchy must be present in **some** Jandex index for the bean to be fully resolved. It is not sufficient for the implementing JAR to have an index — the interface-defining JAR must also be indexed.

### Fix

Add the Jandex plugin to the API JAR's build:

```xml
<plugin>
  <groupId>io.smallrye</groupId>
  <artifactId>jandex-maven-plugin</artifactId>
  <executions>
    <execution>
      <id>make-index</id>
      <goals><goal>jandex</goal></goals>
    </execution>
  </executions>
</plugin>
```

Or, if you cannot modify the API JAR, force-index it in the consuming module's test config:

```properties
quarkus.index-dependency.my-api.group-id=com.example
quarkus.index-dependency.my-api.artifact-id=my-api
```

Note: `quarkus.index-dependency` is a build-time property and must be set in `application.properties`, not via `QuarkusTestResourceLifecycleManager`.

### Why this is non-obvious

The error message ("Unsatisfied dependency") points at the missing bean, not at the missing index. The implementing class is visually present in the JAR, correctly annotated, and listed in that JAR's Jandex index — all the obvious checks pass. The missing piece is in a completely different JAR.

### Alternative root cause — stale SNAPSHOT cache

The same symptom appears when the API JAR **does** have a Jandex index but the locally cached SNAPSHOT predates the commit that added the new types. This happens when:

1. `api-lib` is updated to add new interface types and a CI run publishes a fresh SNAPSHOT
2. A consuming module refreshes its SNAPSHOT dependency (e.g. `casehub-ledger`) but not `api-lib`
3. The local cache holds an old `api-lib` SNAPSHOT whose index doesn't yet list the new types

**Diagnosis:** check the JAR timestamp in `~/.m2/repository/`. If it's older than the CI run that added the types, refresh:
```bash
mvn install -DskipTests -f ~/path/to/api-lib/pom.xml
```

When refreshing a SNAPSHOT dependency chain (e.g. `casehub-ledger`), always refresh its upstreams (e.g. `casehub-platform`) if the ledger has added dependencies on new platform types.
