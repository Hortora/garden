### Symptom
A retry loop with backoff appears to exhaust all attempts almost instantly after an interrupt, even though backoff delays should space out attempts by seconds. The system behaves as if retries had zero delay.

### Context
A common retry pattern uses `Thread.sleep(delayMs)` for backoff between attempts. The sleep method is wrapped in a try-catch that handles `InterruptedException` by restoring the interrupt flag and returning:

```java
private void sleep(long ms) {
    try {
        Thread.sleep(ms);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        // returns normally — loop continues immediately
    }
}
```

When a thread is interrupted during the backoff sleep:
1. `sleep()` catches `InterruptedException`, sets the interrupt flag, returns normally
2. The retry loop calls `executeWithTimeout()` for the next attempt
3. If using `Future.get()`, it immediately throws `InterruptedException` (interrupt flag is set)
4. The retry loop catches this, stores it as `lastException`, calls `sleep()` again
5. `sleep()` immediately throws `InterruptedException` again (flag still set)
6. Steps 2-5 repeat for all remaining attempts — each cycle takes microseconds

The result: all retry attempts burn through in under a millisecond regardless of configured delay.

### What was tried (didn't work)
1. **Clearing the interrupt flag before sleep** — `Thread.interrupted()` before `Thread.sleep()`. This suppresses the interrupt, which violates the cooperative interruption contract and can cause the thread to miss a legitimate shutdown signal.

### Root cause
The `catch (InterruptedException) { interrupt(); return; }` pattern is correct for non-retry contexts — it preserves the flag for the caller. But in a retry loop, "return normally" means "continue to the next iteration," which immediately re-encounters the interrupt flag. The sleep becomes a no-op and the retry loop becomes a tight spin.

### Fix
Throw from the sleep method instead of returning. The retry loop's catch block sees the exception and can break out:

```java
private void sleep(long ms) {
    try {
        Thread.sleep(ms);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
        throw new InterruptedPolicyException("Interrupted during backoff", e);
    }
}
```

The retry loop should also break immediately on receiving an `InterruptedPolicyException`:

```java
catch (Exception e) {
    lastException = e;
    if (e instanceof InterruptedPolicyException) break;
    if (attempt < maxAttempts) {
        sleep(computeDelay(...));
    }
}
```

### Why this is non-obvious
The `catch (InterruptedException) { Thread.currentThread().interrupt(); }` pattern is taught as the correct way to handle interrupts in Java. It IS correct — in methods that do one thing and return. In a retry loop, the "return normally" part has different semantics: it means "try again," not "let the caller decide." The pattern is correct, but its context changes its meaning.
