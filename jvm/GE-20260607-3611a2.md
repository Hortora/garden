### Symptom

Adding `@Inject Preferences preferences` to a bean causes Quarkus deployment to fail:

```
jakarta.enterprise.inject.UnsatisfiedResolutionException:
  Unsatisfied dependency for type io.casehub.platform.api.preferences.Preferences
  and qualifiers [@Default]
```

### Context

`casehub-platform-api` provides `Preferences` as a return-value interface from `PreferenceProvider`:

```java
public interface PreferenceProvider {
    Preferences resolve(SettingsScope scope);
}
```

`Preferences` is not annotated as a CDI scope (`@ApplicationScoped`, `@RequestScoped`, etc.) and
no CDI producer exists for it. It is an interface representing a resolved, scope-specific view of
preferences — it is produced by calling `PreferenceProvider.resolve()`, not by the CDI container.

`PreferenceProvider` IS CDI-injectable. The test `@DefaultBean` is `MockPreferenceProvider`
(from `casehub-platform`). Production uses `ConfigFilePreferenceProvider` (from
`casehub-platform-config`).

### Root Cause

Architectural intent: `Preferences` is a value object representing preferences at a specific scope
and point in time. `PreferenceProvider` is the service that produces these snapshots. Injecting
`Preferences` directly would imply a global, scope-less preference view — which the platform
deliberately avoids. The scope parameter in `resolve(SettingsScope)` determines which preference
overrides apply (e.g. `casehubio.clinical` vs `casehubio.aml`).

### Fix

Inject `PreferenceProvider` and call `resolve()` with the appropriate scope:

```java
// WRONG
@Inject Preferences preferences;

// CORRECT
@Inject PreferenceProvider preferenceProvider;

// In a method (not in a constructor):
SponsorNotificationRetryPolicy policy = preferenceProvider
    .resolve(SettingsScope.of("casehubio", "clinical"))
    .getOrDefault(SponsorNotificationRetryPolicy.KEY);
```

`SettingsScope.of(String... segments)` is a factory on the `SettingsScope` record. For global
(unscoped) preferences, use `SettingsScope.root()`.

In tests, `MockPreferenceProvider` reads from MicroProfile Config properties:
```properties
casehub.platform.preferences.defaults.casehubio.clinical.myPreferenceKey=value
```

So `preferenceProvider.resolve(SettingsScope.of("casehubio", "clinical")).get(KEY)` in tests will
pick up the `application.properties` value automatically — no mocking needed for simple cases.

### Why this is non-obvious

The Quarkus error message says "Unsatisfied dependency for type Preferences" — which looks like a
missing dependency in `pom.xml` or a missing `@Alternative` in `selected-alternatives`. The actual
cause is architectural: `Preferences` is not a CDI bean at all, regardless of which modules are on
the classpath. The fix is not to add a module or alternative but to inject `PreferenceProvider`
(a different interface) and call a factory method. The naming — `Preferences` sounds like something
you'd inject, `PreferenceProvider` sounds like infrastructure — makes the correct direction
counter-intuitive.
