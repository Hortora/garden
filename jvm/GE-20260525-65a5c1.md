## Symptom

A CDI event observer's silent early-return path is logged at `debugf`. In a production
debug session, enabling DEBUG logging for the class reveals thousands of skip lines per
workflow — "WorkItem X has no deviationId — not an IRB item, skipping" fires for every
WorkItem lifecycle event in the entire system, not just IRB WorkItems. The handful of
meaningful events are buried under noise.

## Context

Any `@ObservesAsync` CDI observer that receives high-frequency events and filters early
("not my concern" guard) will execute that guard path for every event on the bus —
including events it was never meant to process. An `IrbDecisionListener` observing
`WorkItemLifecycleEvent` will fire its null-deviationId guard for every AE escalation
WorkItem, every PI authorisation WorkItem, every DSMB review WorkItem, etc.

If the guard is logged at `debugf`, any production support session that enables DEBUG
generates one skip line per non-IRB WorkItem lifecycle transition — easily hundreds per
trial workflow run.

## Root cause

The instinct is "this is a skip — debug level is appropriate." But the debug/trace
distinction isn't just about "how much you care" — it's about expected volume per
processing invocation:

- `debugf`: infrequent events **within the observer's concern** — state transitions,
  decisions, matched cases. Safe to enable for a component without drowning in output.
- `tracef`: high-volume per-invocation data — suitable only for deep tracing where the
  developer explicitly accepts the volume.

"Not my concern" early-return guards in shared-bus observers are inherently high-volume
relative to the observer's real work. They belong at `tracef`.

## Fix

```java
// Wrong — fires for every WorkItem lifecycle event in the system
if (deviationId == null) {
    LOG.debugf("WorkItem %s has no deviationId — not an IRB item, skipping", workItem.id);
    return;
}

// Correct — tracef for high-volume "not my concern" skip paths
if (deviationId == null) {
    LOG.tracef("WorkItem %s has no deviationId — not an IRB item, skipping", workItem.id);
    return;
}
```

Reserve `debugf` for events matched and processed by the observer — state machine
transitions, decisions, successful resolutions. Use `tracef` for early-return guards
that fire constantly as background noise on every event of the type.

## Why this is non-obvious

"Debug" intuitively means "low priority, not production-relevant." Developers
consistently use `debugf` for skip paths as a form of "soft trace" — something between
info and nothing. The distinction between "debug == informative when debugging this
component" and "trace == high-volume, only for deep tracing" is understood in the
abstract but collapsed in practice when annotating guard paths that "don't matter much."

The failure mode only surfaces in production under real event load with a diverse set of
WorkItem types — it's invisible in unit tests (single event, single type) and barely
visible in integration tests (small controlled scenarios).
