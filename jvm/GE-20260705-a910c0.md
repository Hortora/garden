## Symptom

After a library refactoring where `LedgerEntry` moved from `@Entity` (runtime module) to `@MappedSuperclass` (API module), JPQL queries using `FROM LedgerEntry` silently stop returning subclass instances. No compile error, no runtime exception — queries return empty result sets or reduced counts. Seven tests failed with symptoms like `expected: <1> but was: <0>` (attestation count) and `Expected size: 2 but was: 1` (cross-dtype query).

## Context

A library (casehub-ledger) refactored its JPA entity hierarchy:

**Before:**
```
LedgerEntry (@Entity, @Inheritance(JOINED), @Table("ledger_entry"))
  └── MessageLedgerEntry (@Entity, @DiscriminatorValue("QHORUS_MESSAGE"))
```

**After:**
```
LedgerEntry (@MappedSuperclass)  ← moved to API module
  ├── JpaLedgerEntry (@Entity, @Inheritance(JOINED), @Table("ledger_entry"))  ← new runtime class
  │     └── PlainLedgerEntry, ErasureReceiptLedgerEntry
  └── MessageLedgerEntry (@Entity, @DiscriminatorValue("QHORUS_MESSAGE"))  ← still extends LedgerEntry!
```

`MessageLedgerEntry extends LedgerEntry` compiled fine because `@MappedSuperclass` is a valid JPA supertype. But it became a standalone entity — not a JOINED subclass of `JpaLedgerEntry`.

## What was tried (didn't work)

1. Fixed import statements (`LedgerEntry` → `api.model.LedgerEntry`) — compilation fixed but tests still failed
2. Fixed `LedgerAttestation` entity creation (separate gotcha: GE-20260705-2a5555) — attestation tests still failed
3. Added debug logging to attestation path — no output because the code path wasn't reached (the entry lookup returned empty)
4. Checked tenancy consistency — correct

## Root cause

With `@MappedSuperclass`, `MessageLedgerEntry` inherits field mappings but does NOT participate in the JOINED inheritance tree rooted at `JpaLedgerEntry`. JPA treats it as a completely separate entity type.

Consequences:
- `FROM LedgerEntry` (which maps to the `JpaLedgerEntry` entity name) does NOT include `MessageLedgerEntry`
- `findEntryById(UUID)` returns `Optional.empty()` for `MessageLedgerEntry` records
- `findBySubjectId(UUID)` skips `MessageLedgerEntry` rows entirely
- Cross-dtype sequence tests fail because only `JpaLedgerEntry` subtypes are counted
- Attestation writes fail because the COMMAND entry can't be found for the causal link

## Fix

Change the parent class from `@MappedSuperclass` to the `@Entity`:

```java
// Before (broken):
import io.casehub.ledger.api.model.LedgerEntry;
public class MessageLedgerEntry extends LedgerEntry { ... }

// After (correct):
import io.casehub.ledger.runtime.model.jpa.JpaLedgerEntry;
public class MessageLedgerEntry extends JpaLedgerEntry { ... }
```

The subclass now participates in JOINED inheritance, gets `@PrePersist` (id, occurredAt), entity listeners (trace, identity), and appears in polymorphic queries.

## Why this is non-obvious

1. **No compile error** — `@MappedSuperclass` is a valid JPA supertype; `extends LedgerEntry` compiles and the `@Entity` annotation on the subclass is valid
2. **No runtime error** — JPA silently creates a standalone entity table; no warning that JOINED inheritance is broken
3. **Partial functionality** — direct queries on `MessageLedgerEntry` still work; only polymorphic queries through the parent type fail
4. **The import looks correct** — `io.casehub.ledger.api.model.LedgerEntry` is the canonical API type; using it feels right
5. **7 tests failed with misleading symptoms** — "expected 1 attestation got 0" doesn't point at an inheritance problem

### Alternatives considered

- Keep `LedgerEntry` as `@Entity` in the API module — would preserve JOINED inheritance but violates the API module's "no JPA entity, no Hibernate dependency" rule
- Add a compile-time check that subclasses of `@MappedSuperclass` are not `@DiscriminatorValue`-annotated — not available in standard JPA/Hibernate
