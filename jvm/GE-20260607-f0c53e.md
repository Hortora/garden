### What happened

An `@ObservesAsync` CDI observer with `@Transactional(REQUIRES_NEW)` threw a `ConstraintViolationException` (unique index violation). The observer's own transaction correctly rolled back. However, the engine's Quartz worker then reported:

```
(RECIPIENT_FAILURE,8191) CaseDefinition not found for case: 6a111f10-99e0-4adf-b72e-8aa41cf707a3
```

The "CaseDefinition not found" looked like a case registration bug. Investigation showed the case instance had actually been rolled back — not because of the observer, but because the `RollbackException` propagated from the observer's REQUIRES_NEW boundary back through the CDI async event delivery infrastructure and marked the Quartz worker's outer transaction as rollback-only.

### Root cause

When a `REQUIRES_NEW` transaction fails, Narayana throws `ARJUNA016053: Could not commit transaction`. This exception propagates through the CDI container's async event delivery mechanism. If the observer is called during a Quartz worker execution that has an active JTA transaction, Narayana may mark that outer transaction as rollback-only on exception propagation.

The Quartz job's transaction rolled back, destroying the case instance write. The Quartz job had already been scheduled (RAM store — not transactional), so it fired again, found no case instance, and reported "CaseDefinition not found for case: UUID" — where the UUID was the instance ID, not the case type ID.

### Fix

Fix the root cause of the REQUIRES_NEW failure (the constraint violation). In this case, the fix was:

1. Give attestation ledger entries their own dedicated subject UUID (namespaced away from the case subject), eliminating the sequence conflict with engine entries.
2. Add a per-subject application-level lock (outside the transaction boundary) to serialize concurrent sequence assignment.

Do **not** try to catch `RollbackException` in the observer — by the time it propagates, the JTA damage is already done.

### Why this is non-obvious

The primary error message ("CaseDefinition not found for case: UUID") points to the engine registration system. The actual cause is a CDI observer constraint violation two call frames removed. The UUID in the error is a case INSTANCE UUID (rolled back), not a case TYPE UUID (which would indicate a YAML loading failure). Without knowing the JTA propagation behaviour, the entire investigation leads to the wrong subsystem.

### Constraints

Applies when: (1) a `@ObservesAsync` observer has `@Transactional(REQUIRES_NEW)`, AND (2) the observer is called during a Quartz worker execution with an active JTA transaction. The propagation behaviour depends on Narayana version and CDI container implementation — verify on Quarkus upgrades.
