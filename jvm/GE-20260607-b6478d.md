### Symptom
Gate resolution handler logs "PendingActionGate mismatch or absent — discarding" for every approval/rejection event. Cases stall permanently in RUNNING state after gate WorkItems are resolved.

### Context
`WorkflowExecutionCompletedHandler.handleGate()` sets `caseInstance.setPendingActionGate(new PendingActionGate(...))` then calls `caseInstanceRepository.update(caseInstance, tenancyId)`. The update only persists `state`, `waitingForWorkId`, and `parentCaseId` — there is no `pending_action_gate` column on `CaseInstanceEntity`. The gate state exists only in the in-memory `CaseInstance` object held by `CaseInstanceCache`.

When the gate resolution handler (e.g. `ActionGateApprovedHandler`) loads the case via `CrossTenantCaseInstanceRepository.findByUuid(caseId)`, it gets a *fresh* object from the DB with `pendingActionGate = null`. The mismatch check fires and discards the approval.

### What was tried (didn't work)
1. Injecting `CrossTenantCaseInstanceRepository` into the gate approved/rejected/expired handlers and calling `findByUuid()` — returns DB state, `pendingActionGate` is always null.
2. Adding an `update()` call after setting `pendingActionGate` — `JpaCaseInstanceRepository.update()` silently ignores the field because it has no mapping in `CaseInstanceEntity`.

### Root cause
`JpaCaseInstanceRepository.update()` only maps the fields explicitly present in `CaseInstanceEntity` (`state`, `waitingForWorkId`, `parentCaseId`, `parentPlanItemId`). `pendingActionGate` has no JPA column — it is an in-memory-only field in v1 of the ActionRiskClassifier implementation (engine#433 tracks the v2 column addition).

`CaseInstanceCache` holds the live `CaseInstance` object by reference, so in-memory modifications (including `setPendingActionGate()`) are immediately visible via the cache. The DB-backed repository has no knowledge of them.

### Fix
Use `CaseInstanceCache.get(caseId)` in all gate resolution handlers instead of `CrossTenantCaseInstanceRepository.findByUuid()`. The cache holds the live object with `pendingActionGate` populated.

```java
// Wrong — returns DB object, pendingActionGate is always null
CrossTenantCaseInstanceRepository crossTenantRepo;
CaseInstance instance = crossTenantRepo.findByUuid(event.caseId()).await().atMost(timeout);

// Correct — returns live in-memory object with pendingActionGate set
CaseInstanceCache caseInstanceCache;
CaseInstance instance = caseInstanceCache.get(event.caseId());
```

Note: if the JVM restarts while a gate is pending, the cache is empty and `get()` returns null — the gate is lost. This is a known v1 constraint (engine#433).

### Why this is non-obvious
`JpaCaseInstanceRepository.update()` succeeds silently — no error, no warning, no exception. The `pendingActionGate` field appears to be saved (the code looks correct). The mismatch is only visible when the handler loads from the DB and the field comes back null.

Additionally, `CaseInstanceCache` is not obviously the "right" place to look — it's a cache, not an authoritative store. Standard practice in JPA codebases is to load domain objects from the repository, not a cache.
