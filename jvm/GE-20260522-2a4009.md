## Symptom

A cleanup action attached via `onTermination()` to the second stream in a
`Multi.createBy().concatenating().streams(first, second)` is not called when
the subscriber cancels while `first` is still active. The second stream's
`onTermination()` handler is silently skipped.

## Context

SSE endpoint returns a concatenated Multi: a catch-up phase (finite `first`) followed
by a live stream (`second`). `second.onTermination()` is used to clean up a resource
(e.g. deregister a backend from a gateway). A browser closes the SSE connection while
the catch-up phase is still fetching data.

## Root cause

`Multi.createBy().concatenating()` subscribes to each stream sequentially — it does
not subscribe to `second` until `first` completes. If the outer subscription is
cancelled before `first` completes, `second` is never subscribed at all, and its
`onTermination()` callback is therefore never invoked.

## Fix

Attach `onTermination()` to the **concatenated** Multi, not to the inner `second` stream:

```java
// Wrong — onTermination on second stream only
Multi<String> second = liveStream().onTermination().invoke(this::cleanup);
return Multi.createBy().concatenating().streams(catchUp, second);

// Correct — onTermination on the outer concatenated Multi
return Multi.createBy().concatenating().streams(catchUp, liveStream())
        .onTermination().invoke(this::cleanup);
```

The outer `onTermination()` fires regardless of which stream was active when the
cancellation occurred.

## Why this is non-obvious

The logical ownership of the cleanup is the `second` stream (it's the live resource).
Attaching `onTermination()` to `second` is the natural expression of that intent.
The documentation does not call out that `second` may never be subscribed in a
concatenation — it only states that items come from `first`, then `second`. The
cancellation-during-first-stream case requires understanding Mutiny's lazy subscription
model, which is an implementation detail not surfaced in the API.
