## Symptom

`IllegalStateException: No current Mutiny.Session` or `org.hibernate.SessionException` when a reactive service method is called from within a consumer app that has only a named datasource configured (no default datasource). The service works in the library's own tests (which configure a default datasource) but fails in the consumer.

## Context

A library's reactive service uses `Panache.withTransaction(() -> { ... })` (the static, zero-argument form). The consuming app configures a named datasource only:

```properties
# Consumer app — named datasource only, no default
quarkus.datasource.qhorus.db-kind=h2
quarkus.datasource.qhorus.reactive.url=...
```

The library tests configure a default datasource and pass. Consumer integration fails.

## Root cause

`Panache.withTransaction(Supplier<Uni<T>>)` (zero-argument form) always uses the **default** persistence unit. When a consumer app has no default PU, Panache cannot acquire a session and throws at runtime. The library's own tests define a default datasource (usually H2 in test config), so the problem is invisible until a consumer app is built.

## Fix

Use the named-PU overload `Panache.withTransaction(String persistenceUnitName, Supplier<Uni<T>>)`:

```java
// Before — uses default PU, breaks in consumers with named-PU-only config
return Panache.withTransaction(() -> {
    return messageStore.put(message);
});

// After — explicitly targets the named PU
return Panache.withTransaction("qhorus", () -> {
    return messageStore.put(message);
});
```

Apply consistently to all transactional methods in the library — if one method uses the named form and another uses the default form, intermittent failures result depending on which code path the consumer exercises first.

## Why this is non-obvious

The zero-argument `withTransaction()` is the documented primary form. The named-PU overload is not prominently documented. The failure only appears in consumers that don't configure a default datasource — the library's own tests pass, giving false confidence. The `IllegalStateException` message ("No current Mutiny.Session") doesn't mention the PU name mismatch.
