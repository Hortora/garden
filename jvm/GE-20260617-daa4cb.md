### Symptom

Designing an atomic round-robin cursor using `findOneAndUpdate` with `upsert: true`. The spec proposed initializing the counter to -1 via `$setOnInsert` and advancing via `$inc: 1`, expecting the first upsert to produce 0 (-1 + 1). The actual behavior is ambiguous — MongoDB may conflict when two update operators target the same field.

### Context

Implementing `MongoRoutingCursorStore.acquireNext()` for a round-robin candidate pool cursor. The cursor must start at 0 on first call and advance atomically. The approach was modeled on JPA's pattern where `lastIndex` starts at -1 and `(lastIndex + 1) % poolSize` gives 0 on first call.

### What was tried (didn't work)

```java
Document update = new Document()
    .append("$inc", new Document("lastIndex", 1L))
    .append("$setOnInsert", new Document("lastIndex", -1L));
// Intent: on insert, set lastIndex = -1, then $inc advances to 0
// Reality: two operators targeting the same field — behavior is ambiguous
```

MongoDB may reject or silently mishandle this combination depending on version and driver. The MongoDB documentation does not clearly specify the interaction order when `$setOnInsert` and `$inc` target the same field.

### Root cause

MongoDB's update pipeline processes `$setOnInsert` and `$inc` as separate operations. When both target the same field, the result depends on internal application order which is not guaranteed across MongoDB versions. This is analogous to the documented restriction that `$set` and `$setOnInsert` cannot target the same field.

### Fix

Use `$inc` alone. MongoDB initializes missing numeric fields to 0 before incrementing. Subtract 1 in Java for 0-based indexing:

```java
Document update = new Document()
    .append("$inc", new Document("lastIndex", 1L))
    .append("$set", new Document("lastAccessed", Instant.now()));

// First call (upsert): missing lastIndex initialized to 0, $inc → 1
// Java: Math.floorMod(result.getLong("lastIndex") - 1, poolSize) → 0
// Subsequent: lastIndex increments naturally; subtract-1 maintains 0-based indexing
```

Key details:
- `Math.floorMod()` not `%` — Java `%` returns negative for negative dividends
- Store `lastIndex` as BSON int64 (long) to avoid int32 overflow at ~2.1B increments

### Why this is non-obvious

The `$setOnInsert` + `$inc` combination reads naturally and appears in Stack Overflow answers. A developer implementing an atomic counter with a non-zero initial value would reach for it instinctively. The simpler solution — let MongoDB handle the zero-initialization and offset in Java — requires knowing that `$inc` on a missing field initializes to 0, which is documented but not prominent.

### Alternatives considered

- **CAS loop with separate read-then-write**: works but adds latency and complexity. MongoDB's single-document atomicity makes `findOneAndUpdate` the right tool.
- **`$setOnInsert` on a different field (e.g. `initialized: true`) + always `$inc`**: unnecessary indirection; the `$inc`-alone approach is simpler.
