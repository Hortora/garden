## Context

When using `ObjectMapper.readValue()` to parse a JSON string, Jackson does NOT reject trailing non-JSON content by default. The parse succeeds and the trailing content is silently discarded.

```java
// This succeeds silently — "extra text" is ignored
Map<String, Object> result = mapper.readValue(
    "{\"key\": \"value\"} extra text here",
    Map.class
);
// result = {"key": "value"}
```

## Symptom

A detection method intended to reject `{"type":"DONE","content":"ok"} explanatory text` accepts it as valid JSON and extracts `{"type":"DONE","content":"ok"}` — the trailing text never causes an error.

The practical failure: an LLM output like `{"type":"DONE","content":"Task complete"} I've now finished the task.` is silently classified as a JSON speech act rather than falling through to bracket-prefix detection, producing unexpected DONE classification when the agent was narrating.

## Root Cause

Jackson's `JsonParser` stops at the end of the first complete JSON value and considers the parse successful. `readValue()` wraps this and makes the trailing content invisible. This is the specified behaviour — Jackson is designed for streaming contexts where trailing content is normal.

## Fix

Enable `DeserializationFeature.FAIL_ON_TRAILING_TOKENS` on the `ObjectMapper`:

```java
private static final ObjectMapper MAPPER = new ObjectMapper()
        .enable(DeserializationFeature.FAIL_ON_TRAILING_TOKENS);
```

With this flag set, `readValue("{\"key\":\"val\"} extra", Map.class)` throws `JsonParseException: Trailing token (of type START_OBJECT) found after value`. Catching the exception and falling through to the next detection tier is then correct.

## Why this is non-obvious

Jackson's docs describe `FAIL_ON_TRAILING_TOKENS` as an option to opt into, not a default-off footgun. The lenient default exists for good reasons (streaming, multi-value streams), but it's invisible when you have a one-shot string parse and expect strict behaviour. The feature was added in Jackson 2.9 — older documentation doesn't mention it. Without knowing to look for it, the obvious assumption is that `readValue()` of a complete JSON string validates the full input.

### Why this fix

`FAIL_ON_TRAILING_TOKENS` is the canonical solution — it targets exactly this behaviour without affecting normal parsing. Alternative: use `JsonParser` directly and check `parser.nextToken() == null` after parse, but this requires more boilerplate and is less obvious to readers.

### Alternatives considered

- **Use `JsonParser` directly** — check `parser.nextToken() == null` after reading the value. More boilerplate; requires try-with-resources pattern and is less discoverable.
- **Check `input.trim().endsWith("}")` before parsing** — fragile; fails for arrays, nested objects, and JSON with trailing whitespace.
- **Regex pre-check** — would reject valid multi-line JSON. Not viable.
