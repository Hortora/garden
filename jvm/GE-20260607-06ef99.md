### What this is

`ProjectionResult<S>` can be safely constructed manually in tests
(`new ProjectionResult<>(state, null)`) when stubbing the full-scan
`ProjectionService.project(UUID, ChannelProjection)` overload, despite the Javadoc saying
"never construct manually."

### The Javadoc warning

```java
/**
 * <strong>Contract:</strong> only pass results obtained from
 * {@code ProjectionService.project()} — never construct manually. A manually
 * constructed instance with a non-null {@code state} and a null {@code lastMessageId}
 * has undefined behaviour in the incremental overload ...
 */
public record ProjectionResult<S>(S state, Long lastMessageId) { ... }
```

The warning is overload-specific: "undefined behaviour in the **incremental overload**."
It is not a blanket prohibition on manual construction.

### Why this matters for tests

When stubbing `ProjectionService.project()` with Mockito, you must return a
`ProjectionResult<S>`. There is no factory method or builder. A developer reading the
warning will avoid construction entirely, and the test becomes impossible to write without
a real CDI container.

### When it is safe

```java
// ✅ Safe: stubbing the full-scan overload. lastMessageId=null → isEmpty()=true.
// ReviewConversationRenderer handles isEmpty() gracefully (returns sentinel string).
when(projectionService.project(CHANNEL_ID, projection))
        .thenReturn(new ProjectionResult<>(new ReviewState(Map.of(), List.of()), null));
```

`isEmpty()` returns `lastMessageId == null`. The full-scan overload uses `isEmpty()`
to decide whether to return the identity state. Passing `null` simulates an empty channel.
This is the correct test default when you want to isolate the LLM call from history.

### When it is NOT safe

```java
// ❌ NOT safe: passing this to the incremental overload.
// The incremental overload: if previous.isEmpty(), it ignores previous.state() entirely
// and starts from identity(). Manually constructed results misrepresent the cursor position.
service.project(channelId, new ProjectionResult<>(myState, null), projection);
// ↑ myState is silently discarded — the overload sees null lastMessageId and restarts from identity
```

### Why this is non-obvious

The Javadoc warning reads as universal ("never construct manually") but the behaviour
it describes is incremental-overload-specific. The full-scan overload produces its own
`ProjectionResult` internally — it never reads `lastMessageId` from a caller-supplied
previous result.
