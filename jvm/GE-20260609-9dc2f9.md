## Symptom

A null-checking serialization helper emits `"fieldName": ""` into JSON payloads for annotation attributes that were not explicitly set. The payload is polluted with empty-string signals where the field should simply be absent.

## Context

When serializing Java annotation attribute values into JSON (e.g. via Jackson `ObjectNode.put()`), a null-checking helper:

```java
private static void addIfPresent(ObjectNode node, String key, String value) {
    if (value != null) node.put(key, value);
}
```

passes empty strings through because `@VocabularyMetadata.name()` — declared with `default ""` — returns `""` at runtime, never `null`.

The annotation declaration looks like an optional field:

```java
public @interface VocabularyMetadata {
    String uri();
    String name() default "";        // "not provided" sentinel
    String description() default ""; // "not provided" sentinel
}
```

But `null` is not a valid annotation attribute value in Java. The default is the empty string, so `meta.name()` always returns a `String` and `!= null` is always `true`.

## Root cause

Java annotation attribute defaults return the declared default value at runtime — there is no null for annotation attributes. The empty string is the domain sentinel for "not provided", but it looks non-null to a null-checking helper.

## Fix

Introduce a separate `addIfNonBlank` helper for annotation-sourced fields:

```java
private static void addIfNonBlank(ObjectNode node, String key, String value) {
    if (value != null && !value.isEmpty()) node.put(key, value);
}
```

Use `addIfNonBlank` for annotation attribute values. Keep `addIfPresent` (null check only) for genuinely nullable `String` fields on POJOs or records.

```java
// Annotation-sourced — use addIfNonBlank
vocab.vocabularyMetadata(uri).ifPresent(meta -> {
    addIfNonBlank(node, "vocabularyName",        meta.name());
    addIfNonBlank(node, "vocabularyDescription", meta.description());
});

// POJO field — use addIfPresent (can legitimately be null)
addIfPresent(node, "jurisdiction", descriptor.jurisdiction());
```

## Why this is non-obvious

The annotation declaration and a nullable POJO field look identical at the call site — both are `String` parameters to a helper. The contract difference (annotation default vs nullable field) is in the annotation declaration, not visible at the serialization site. A developer auditing serialization code sees `addIfPresent(node, "name", meta.name())` and reads "skip if absent" — but does not notice that `meta.name()` can never be null, only blank.
