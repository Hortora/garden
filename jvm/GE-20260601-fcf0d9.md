## What happened

Two `@DefaultBean @ApplicationScoped` implementations of the same SPI type appeared on the CDI
classpath simultaneously (one from a platform mock library, one from a persistence testing module).
Quarkus reported **"Ambiguous dependencies"** and refused to start â€” even though both beans
explicitly said "I am a default, defer to anyone else".

## Why this is non-obvious

`@DefaultBean` is documented as "yield to any other bean of the same type". With two
`@DefaultBean` beans and no non-default implementation, neither bean yields â€” the contract
("yield to someone else") cannot resolve because there IS no "someone else" with higher
precedence. The error message is **Ambiguous** (same as two regular `@ApplicationScoped`
beans conflicting), not **Unsatisfied** (what you'd expect if both were supposedly yielding).
Developers often assume two `@DefaultBean` beans resolve cleanly because they both "opt in"
to being overridden.

## Root cause

CDI/ARC `@DefaultBean` semantics: a default bean is only eligible when no other bean at
the same or higher priority is eligible. With two `@DefaultBean` beans and no higher-priority
bean, the tie-break is undefined â€” ARC treats this as an ambiguous injection point.

## Fix

Exclude the unwanted `@DefaultBean` implementation via `application.properties`:
```properties
quarkus.arc.exclude-types=io.casehub.platform.mock.MockCurrentPrincipal,\
  io.casehub.platform.mock.MockGroupMembershipProvider,\
  io.casehub.platform.mock.MockPreferenceProvider
```

This removes the conflicting bean from CDI discovery entirely, leaving the desired
`@DefaultBean` as the sole implementation. Prefer excluding the one that returns null or
uninitialized values rather than the one with correct sentinel behaviour.

### Why this fix

`@Alternative @Priority(N)` would also work, but requires activating it via
`selected-alternatives` â€” that is a heavier configuration change and makes the
intent less clear. `exclude-types` is surgical: "this specific class should not
participate in CDI at all in this context."

## Constraints

Applies when both `@DefaultBean` implementations reach the CDI container's scanning phase
(i.e. both are indexed via `quarkus.index-dependency` or are in the app's own sources).
In two-repo / multi-module projects, platform library JARs can silently contribute
`@DefaultBean` mocks that conflict with internal no-ops.

## Alternatives considered

- Mark one bean `@Alternative @Priority(1)` instead of `@DefaultBean` â€” works but changes
  the semantics of the "winning" bean and requires `selected-alternatives` configuration.
- Remove one of the `@DefaultBean` implementations from the classpath entirely â€” clean, but
  may not be possible if it comes from a third-party dependency.

## Invalidation triggers

Would change if Quarkus ARC adds a deterministic tie-break for two `@DefaultBean` beans
(e.g. favour the one in the current module's sources). Track Quarkus changelog for ARC
`@DefaultBean` resolution changes.
