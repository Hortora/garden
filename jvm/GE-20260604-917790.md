### Context

Content arriving from LLM agent calls or external HTTP requests may contain control characters. These need to be stripped before being passed to logging infrastructure (log injection via ANSI ESC sequences) or stored as telemetry. A common approach is chained `String.replace()` calls, but this is incomplete and inefficient.

### The common (insufficient) approach

```java
// Misses ESC (0x1B), NUL (0x00), BEL (0x07), and others
// Also requires a separate truncation step
String stripped = content
    .replace('\n', ' ')
    .replace('\r', ' ')
    .replace('\t', ' ');
String truncated = stripped.length() > MAX ? stripped.substring(0, MAX) : stripped;
```

Problems:
1. Creates three intermediate `String` objects
2. Misses ESC (`\x1B`) — the most dangerous for terminal-rendered logs (ANSI injection)
3. Requires a separate truncation step after all replacements

### The technique

```java
private static final int MAX_LENGTH = 500;

static String sanitize(final String content) {
    if (content == null) return "";
    final StringBuilder sb = new StringBuilder(Math.min(content.length(), MAX_LENGTH));
    for (int i = 0; i < content.length() && sb.length() < MAX_LENGTH; i++) {
        final char c = content.charAt(i);
        sb.append((c < 0x20 || c == 0x7F) ? ' ' : c);
    }
    return sb.toString();
}
```

One pass handles:
- ALL ASCII control characters (0x00–0x1F): NUL, SOH, STX, ETX, EOT, ENQ, ACK, BEL, BS, TAB, LF, VT, FF, CR, SO, SI, DLE, DC1–DC4, NAK, SYN, ETB, CAN, EM, SUB, ESC, FS, GS, RS, US
- DEL (0x7F)
- Truncation at MAX_LENGTH in the same iteration — exits the loop early, no wasted work on characters that will be discarded

The `StringBuilder` is pre-sized to `min(content.length(), MAX_LENGTH)` to avoid resizing for the common case of short content.

### Why this is non-obvious

Most Java developers reach for `String.replace()` because it's readable and the common control chars (`\n`, `\r`, `\t`) are the first ones that come to mind. ESC (`\x1B`) — the most security-relevant character for log injection via ANSI escape sequences — is never listed in "control characters to strip" advice in any common Java security guide. The range check `c < 0x20` is the correct complete solution but requires knowing the ASCII table layout rather than enumerating visible characters.

The single-pass truncation is a secondary benefit: it avoids the two-step pattern (strip-then-truncate) and stops processing the moment the output reaches its maximum size.
