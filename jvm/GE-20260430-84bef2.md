## Symptom

A `WorkflowExecutionListener.onWorkflowCompleted()` implementation throws an exception (e.g. NPE from calling `.asMap()` on a null `WorkflowModel`). The listener appears not to have fired: any state that the listener was supposed to set remains null, and no error is logged. The workflow completes normally.

## Context

When recording output inside `onWorkflowCompleted`, a listener calls:

```java
var outputModel = event.workflowContext().instanceData().output(); // returns null
String output = outputModel.asMap()...;                            // NPE thrown here
completedOutput.set(output);                                        // never reached
```

After `await()` returns, `completedOutput.get()` is null. Diagnosis: "the hook didn't fire." Actual cause: the hook fired, threw, and the exception was silently swallowed.

## Root Cause

`LifecycleEventsUtils.publishEvent()` iterates over all registered listeners and calls each one. If a listener throws, the exception is caught and attached as a *suppressed exception* on the `CompletableFuture` returned by `publishEvent()`. The `CompletableFuture` still resolves normally — it does not complete exceptionally. The suppressed exception is only visible if you explicitly inspect the resolved future's suppressed exceptions.

The suppressed exception surfaces as a footnote in the test failure output (attached to the `AssertionError`), not as a primary error — easy to miss.

## Fix

Use `event.output()` directly instead of `instanceData().output()` inside `onWorkflowCompleted`. The `WorkflowCompletedEvent` carries the output as a first-class field populated before the event is published. `instanceData().output()` is a blocking join on the workflow future, not the right API inside a listener callback.

```java
// Wrong — can NPE, and the NPE will be silently swallowed:
var outputModel = event.workflowContext().instanceData().output();

// Correct — always populated, never null for a completed workflow:
var outputModel = event.output();
```

## Why this is non-obvious

The suppressed exception does not appear in the primary stack trace or in test failure messages as a root cause — it is attached to the `CompletableFuture`'s result and only visible via `.getSuppressed()`. The workflow completing normally makes it look like the listener simply didn't fire. There is no warning or log from the framework when a listener exception is swallowed.

### Alternatives considered
- Inspecting the `CompletableFuture` result for suppressed exceptions — works but requires knowing to look
- Adding logging inside the listener — catches it, but the root fix is using the correct API

### Why this fix
`event.output()` is the designed API for accessing output in the completion hook. `instanceData().output()` is designed for callers who hold a `WorkflowInstance` reference after `start()` completes (outside the event chain). Using the right API eliminates the NPE entirely — no null-guarding needed.
