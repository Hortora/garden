### Symptom

A query for history or cross-entity data returns empty even though the store has relevant records. The query is scoped to an entity ID that was just created for the current operation.

### Context

Any store (CaseMemoryStore, event store, audit log) that supports filtering by entity ID and is queried for prior history at the start of a new entity's lifecycle.

### What was tried (didn't work)

```java
// Querying for inter-case memory — scoped to the current (brand new) case UUID
MemoryQuery query = MemoryQuery
    .forEntity(entityId, domain, tenantId)
    .withCaseId(instance.getUuid().toString())  // WRONG — new case has no memories yet
    .withLimit(10);
```

`MemoryQuery.caseId` filters results to memories tagged with that specific case ID. A case that just started has no stored memories — so the query always returns empty regardless of how much history exists for the entity.

### Root cause

The intent was to scope the query to "memories relevant to this case" but the semantics of `caseId` filtering is "memories that were recorded *from* this case." A new case has never recorded anything.

For cross-entity or inter-case queries, the correct scope is entity + domain + tenant — not the current case ID.

### Fix

Do not add `withCaseId()` to inter-case queries. The entity ID + domain + tenant scope is sufficient to retrieve prior history across all cases:

```java
// Correct — cross-case, no caseId filter
MemoryQuery query = MemoryQuery
    .forEntity(entityId, domain, tenantId)
    .withLimit(10);
// MemoryOrder.CHRONOLOGICAL is the factory default — most recent first via limit
```

### Why this is non-obvious

The `caseId` field reads as "scope to this case" which sounds right when you think of it as "get memories that are relevant to this case." The semantic distinction between "memories recorded from this case" and "memories relevant to this case" is easy to miss, especially when the entity ID correctly retrieves cross-case history in tests that used an existing (non-new) entity.
