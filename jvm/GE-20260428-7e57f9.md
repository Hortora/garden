## Symptom

Playwright visual tests pass in CI and locally. The developer reports that elements (minerals, geysers, creep tiles) are not visible in the running application. Hours are spent changing geometry sizes, colours, and positions — all changes continue to pass tests but the user still sees nothing.

## Context

A Three.js visualizer has two rendering paths:
- **Mock mode** (`%test`/`%mock`): 64×64 tile map, camera distance ~18 world units, `SimulatedGame` with fixed probe/nexus positions
- **Replay mode** (`%replay`): 160×208 tile map, camera distance ~102 world units, `ReplaySimulatedGame` with real tracker events

`@QuarkusTest` always boots Quarkus in the `%test` profile (mock mode). Playwright tests written inside `@QuarkusTest` therefore test mock rendering. The production application (Electron, `mvn quarkus:dev -Dquarkus.profile=replay`) uses a completely different code path, coordinate system, and camera distance.

A `BoxGeometry` mineral that is 8 pixels wide and clearly visible at camera distance 18 is 2 pixels wide and visually invisible at camera distance 102. Tests pass; user sees nothing.

## What was tried

- Asserting `mineralCount() > 0` — passes (mesh exists in scene graph)
- Asserting `anyMineralOnScreen()` via NDC projection — passes (centre projects within frustum)
- Changing geometry sizes, colours, y-positions — still invisible to user, tests still pass

## Root cause

`@QuarkusTest` tests run in mock profile. Visual assertions in those tests prove mock-mode rendering works. They provide zero signal about replay-mode rendering because the two profiles have different map sizes, camera distances, CDI beans, and coordinate systems.

## Fix

Create a separate integration test class that starts the **real packaged jar** (not `@QuarkusTest`) as a subprocess and connects Playwright to it:

```java
class ReplayVisualizerIT {
    static Process serverProcess;

    @BeforeAll
    static void startReplayServer() throws Exception {
        serverProcess = new ProcessBuilder(
            "java", "-Dquarkus.http.port=8082",
            "-jar", "target/quarkus-app/quarkus-run.jar"
        ).directory(new File(".")).start();
        waitForServer(30); // poll /q/health/ready
    }

    @Test
    void mineralPixelIsCyanInRealReplay() {
        // Open browser at localhost:8082, aim camera, sample WebGL pixel
        // B > R means cyan, not sandy terrain
    }
}
```

Run with: `mvn test -Pplaywright-replay` (a Maven profile that builds the replay jar then runs these tests).

## Why this is non-obvious

`@QuarkusTest` feels like a full integration test — it boots Quarkus, serves HTTP, accepts WebSocket connections. It genuinely tests a running server. The gap is invisible: it's still mock mode, which is a fundamentally different application than what ships.

### Why this fix

Starting the real jar is heavier than `@QuarkusTest` but is the only way to test what the user actually sees. The performance cost (20–30s startup) is acceptable for a CI gate that runs on every change to `visualizer.js`.
