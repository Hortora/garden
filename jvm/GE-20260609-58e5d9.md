## Context

A JPA JOINED inheritance hierarchy where the child entity redeclares a field already declared in
the parent entity — both with `@Column(nullable = false)`. The child table gets a denormalized copy
of the column (e.g. for a local index to avoid joins). This pattern is intentional for performance
but creates a hidden trap for external writers.

Example from casehub platform:
- `LedgerEntry.tenancyId @Column(name="tenancy_id", nullable=false)` — in `ledger_entry` table
- `CaseLedgerEntry.tenancyId @Column(name="tenancy_id", nullable=false, length=64)` — in `case_ledger_entry` table

## Symptom

```
org.hibernate.PropertyValueException: not-null property references a null or transient value
  for entity io.casehub.ledger.model.CaseLedgerEntry.tenancyId
```

Thrown even though the code explicitly sets `entry.tenancyId = "tenant-1"` before calling
`em.persist()`. The value is visibly non-null; the error says it is null.

## Root cause

Java field access is statically resolved by the reference type at compile time. When `entry`
is declared as `CaseLedgerEntry`:

```java
entry.tenancyId = value;   // resolves to CaseLedgerEntry.tenancyId (shadow field)
```

This writes to the child's own field (mapped to `case_ledger_entry.tenancy_id`). The parent's
field `LedgerEntry.tenancyId` (mapped to `ledger_entry.tenancy_id`, also `nullable=false`) is
never written. Hibernate's nullability check validates BOTH the child's property AND the parent's
property — the parent's property is null → `PropertyValueException`.

## Fix

Set both fields explicitly from any external code that touches the entity:

```java
entry.tenancyId = tenancyId;                    // CaseLedgerEntry.tenancyId (case_ledger_entry)
((LedgerEntry) entry).tenancyId = tenancyId;    // LedgerEntry.tenancyId (ledger_entry)
```

The explicit cast forces the Java compiler to emit `PUTFIELD LedgerEntry.tenancyId` — a
separate bytecode instruction targeting the parent class's field declaration, distinct from
the child's shadow field.

The correct long-term fix: remove the field redeclaration from the child entity. Use a
`@SecondaryTable` or a Hibernate `@Formula` to maintain the index without the shadow field.

## Why this is non-obvious

The call site `entry.tenancyId = value` looks complete. The entity compiles and CDI-deploys
without warnings. The failure surfaces at `em.persist()` with a message naming the child entity
and property — which ARE set — not the parent entity and property — which are NOT set.

The Hibernate nullability check message `entity CaseLedgerEntry.tenancyId` refers to the
property-path in the entity class hierarchy, not the Java field that is actually null. This
leads to the investigation going in the wrong direction (inspecting the child's field assignment
rather than the parent's).

Any code that uses direct field access from outside the entity (common in JPA entities without
getters/setters) is silently broken when this pattern is in play. Only tests that actually
perform a database persist — not those that simply assign and read back — will catch it.
