### Symptom
Calling an API that uses `java.net.http.HttpClient` to fetch URLs with a
`file://` scheme returns silently with an empty or error result — no
exception thrown, no warning logged. In Sparge, `POST /api/projects/{id}/ingest/run`
with `urls: ["file:///tmp/posts/post.html"]` returned `0 posts ingested`
with no indication why.

### Context
Occurs any time `HttpClient.send()` or `HttpClient.sendAsync()` is called
with a `file://` URI. Java's `HttpClient` only supports `http://` and
`https://` schemes — `file://` is not in scope for the RFC-7230 HTTP
client. The failure mode varies by JDK: some versions throw an unchecked
exception that gets swallowed by the caller; others silently return a
failed response.

### What was tried
- Passing `file:///absolute/path/to/file.html` — 0 results
- Checking server logs — no error logged at the ingest endpoint
- Polling the posts API after ingest — empty list every time

### Root cause
`java.net.http.HttpClient` is an HTTP/HTTPS client. It does not implement
URL scheme handlers for `file://` the way `java.net.URL.openStream()` does.
The URI is accepted at construction time (`URI.create("file://...")` succeeds)
but the send call fails at the scheme-dispatch stage.

### Fix
For test fixtures or local file ingestion, read the file directly from the
filesystem rather than routing through `HttpClient`:

```java
// Instead of: HttpClient.send(HttpRequest.newBuilder(URI.create("file://...")).build(), ...)
String content = Files.readString(Path.of("/tmp/posts/post.html"));
```

For test fixture setup specifically: write `state.json` directly rather
than calling the ingest API with local file URLs.

### Why this is non-obvious
`URI.create("file:///path")` succeeds and `HttpRequest.newBuilder(uri).build()`
succeeds — the failure only occurs at send time, and the error is often
swallowed by the caller's error handling. The HttpClient Javadoc says it
supports "HTTP/1.1 and HTTP/2" but does not explicitly state that `file://`
is unsupported.
