### Symptom

Adding `@JsonTypeInfo` and `@JsonSubTypes` directly to a sealed interface enables polymorphic serialization for a JPA store but breaks YAML configuration file parsing that uses the same types — the YAML files don't include the `@type` discriminator property.

### Context

A sealed interface hierarchy (e.g. `DeploymentNodeSpec`) is used in two contexts: (1) YAML goal files parsed by a `DeploymentGoalLoader`, and (2) JSON persistence in a `PlanStore`. The YAML files use structural typing (Jackson deduces the subtype from property shape). The JSON store needs explicit type info for round-trip fidelity.

### Fix

Use `ObjectMapper.addMixIn()` to scope `@JsonTypeInfo` to a dedicated ObjectMapper instance — the global ObjectMapper (used by Quarkus for YAML parsing, REST serialization, etc.) remains unaffected.

```java
public final class PlanStoreMapper {
    private static final ObjectMapper MAPPER = createMapper();

    public static ObjectMapper mapper() { return MAPPER; }

    private static ObjectMapper createMapper() {
        ObjectMapper mapper = new ObjectMapper();
        mapper.findAndRegisterModules();
        mapper.addMixIn(NodeSpec.class, NodeSpecMixin.class);
        mapper.addMixIn(InfraNodeSpec.class, InfraNodeSpecMixin.class);
        return mapper;
    }

    @JsonTypeInfo(use = JsonTypeInfo.Id.NAME, property = "@specType")
    @JsonSubTypes({
        @JsonSubTypes.Type(value = InfraDesiredNodeSpec.class, name = "infra-desired"),
        @JsonSubTypes.Type(value = AgentNodeSpec.class, name = "deployment-agent"),
    })
    interface NodeSpecMixin {}

    @JsonTypeInfo(use = JsonTypeInfo.Id.NAME, property = "@type")
    @JsonSubTypes({
        @JsonSubTypes.Type(value = K8sNamespaceSpec.class, name = "k8s-namespace"),
        @JsonSubTypes.Type(value = K8sDeploymentSpec.class, name = "k8s-deployment"),
    })
    interface InfraNodeSpecMixin {}
}
```

### Why this is non-obvious

The natural approach — annotating the sealed interface directly — is simpler and works for Jackson. The failure only surfaces in a different serialization context (YAML config loading) that shares the same types but doesn't include the discriminator property. The connection between "I added Jackson annotations" and "YAML parsing broke" is non-obvious because the two code paths look unrelated.
