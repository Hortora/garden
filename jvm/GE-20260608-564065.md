### Symptom
`UnsatisfiedLinkError: Native Library /path/to/libtokenizers.dylib already loaded in another classloader` when running tests in a module containing both `@QuarkusTest` classes and plain JUnit tests that load JNI libraries (ONNX Runtime, HuggingFace Tokenizers).

### Context
A module (`inference-quarkus`) had a pre-existing plain JUnit test (`NativeImageGateTest`) that loads ONNX Runtime and DJL Tokenizer JNI libraries directly. Adding `@QuarkusTest` classes that inject CDI beans backed by the same JNI libraries causes the conflict. Surefire's default `reuseForks=true` runs all test classes in the same JVM fork. The `@QuarkusTest` loads JNI via Quarkus's augmented classloader; the plain test tries to load the same native library via the system classloader — JNI rejects the second load.

### What was tried (didn't work)
- Initially assumed a library packaging or dependency conflict
- Checked for duplicate native library files on the classpath

### Root cause
JNI enforces a one-classloader-per-native-library rule. Quarkus `@QuarkusTest` uses an augmented classloader that loads the JNI library first. When surefire reuses the same JVM fork for the next test class (a plain JUnit test), the plain test's classloader tries `System.load()` on the same `.dylib` — JNI throws `UnsatisfiedLinkError`.

### Fix
Set `reuseForks=false` in `maven-surefire-plugin` configuration for the affected module. Each test class gets its own JVM fork, so classloader isolation is guaranteed:

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-surefire-plugin</artifactId>
  <configuration>
    <reuseForks>false</reuseForks>
  </configuration>
</plugin>
```

### Why this is non-obvious
The error message says "already loaded in another classloader" — a developer's first instinct is to check for duplicate dependencies or library files. The actual cause (Quarkus augmented classloader vs system classloader in the same JVM fork) is invisible without understanding surefire's fork model and Quarkus's test classloader architecture.
