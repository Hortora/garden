### Symptom
`mvn compile` fails with errors like:
- `messageType has private access in io.casehub.qhorus.api.message.Message`
- `id has private access in io.casehub.qhorus.api.channel.Channel`
- `package io.casehub.qhorus.runtime.store does not exist`

The "private access" errors look like access violations but are actually stale imports — the types moved from `runtime.*` to `api.*` packages, and Java records require accessor methods (`.id()` not `.id`).

### Context
When `casehub-qhorus` rebuilds its SNAPSHOT jar with API refactoring (types moved between packages, classes converted to records), downstream repos like `casehub-openclaw` silently break. Maven resolves the new SNAPSHOT jar, but the consuming code still has the old import paths and field-access syntax.

### What was tried (didn't work)
- Subagent implementers tried to compile individual files in isolation — succeeded for new code but couldn't run `mvn test` for the full module
- Custom JUnit runners to test new code while old code was broken — worked but masked the underlying issue

### Root cause
Maven SNAPSHOT resolution silently updates the dependency jar between sessions. When the upstream API refactors package structure or converts mutable classes to immutable records, the downstream code compiles against stale import paths and field-access patterns. The compiler errors name the new package locations ("private access in `api.channel.Channel`") but the actual fix is changing the import from `runtime.channel.Channel` to `api.channel.Channel` plus adding `()` to record field access.

### Fix
1. Run `mvn compile` at session start to detect drift before any implementation work
2. Use IntelliJ's `ide_find_class` with `scope: project_and_libraries` to find where types moved
3. Mechanical fix pattern: `import runtime.X → import api.X` and `.field → .field()`
4. In this case: 26 files across casehub/ and app/ modules, all mechanical changes

### Why this is non-obvious
The compiler says "private access" — which implies an access-level problem. A developer's instinct is to check visibility modifiers, not import paths. The actual cause (package relocation + record accessor syntax) requires knowing that the upstream dependency refactored its package structure between SNAPSHOT builds.
