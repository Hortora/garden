## Symptom

A `@QuarkusTest` (or plain unit test running under Quarkus's logging) captures log
output via a `java.util.logging.Handler`. Filtering captured `LogRecord` entries by
level using object equality returns zero matches:

```java
var warnLogs = logs.stream()
    .filter(l -> l.getLevel() == Level.WARNING)  // always empty!
    .toList();
assertThat(warnLogs).isNotEmpty();  // FAILS
```

The log was emitted (visible in console output). The handler captured it. But the
level filter discards it.

## Context

JBoss LogManager (Quarkus's logging backend) defines its own `org.jboss.logmanager.Level`
subclass. `Level.WARN` in JBoss is a distinct Java object from `java.util.logging.Level.WARNING`,
even though both have `intValue() == 900` and `getName() == "WARNING"`.

When JBoss LogManager delivers a `LogRecord` to a JUL handler, the record's `getLevel()`
returns the JBoss `Level.WARN` instance, not JUL's `Level.WARNING`. Object identity (`==`)
fails. `equals()` also fails — `java.util.logging.Level` does not override `equals()`.

## Root cause

`java.util.logging.Level` uses identity comparison (no `equals()` override). JBoss
LogManager creates parallel `Level` instances with identical names and int values but
different object identity. This is by design — JBoss adds custom levels (`TRACE`, `WARN`)
that map to different JUL levels.

## Fix

Compare `intValue()` instead of object identity:

```java
var warnLogs = logs.stream()
    .filter(l -> l.getLevel().intValue() == Level.WARNING.intValue())
    .toList();
```

This works because both JBoss `WARN` and JUL `WARNING` share int value 900.

## Why this is non-obvious

`Level.WARNING` and JBoss `Level.WARN` are semantically identical — same name, same int
value, same behaviour. The test assertion looks correct. The filtered list is empty not
because the log wasn't captured, but because the Level object in the captured LogRecord
is a different class. The error manifests as "no WARN logs found" when the logs are
clearly visible in the console — leading developers to suspect the handler isn't
capturing, not that the filter is wrong.
