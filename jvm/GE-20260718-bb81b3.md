### Symptom
`RuntimeException: No method candidates were found for method call 'get'` when evaluating `get(0)` or `size() > 1` against a List context compiled via `MVEL.list()`.

### Context
Using MVEL3's list context mode (`MVEL.list()`) with expressions that call List methods directly. Unlike Map context (where `MVEL.getTypeMap()` infers types from map entries at runtime), list context expects element types to be declared upfront at compile time.

### Root cause
MVEL3's `list()` builder uses `ContextType.LIST` with position-indexed variable declarations. The API signature `MVEL.list(Declaration<?>... types)` maps list positions to named variables:

```java
// Declaration.of("name", String.class) → list.get(0) maps to variable "name"
// Declaration.of("age", Integer.class) → list.get(1) maps to variable "age"
MVEL.list(
    Declaration.of("name", String.class),
    Declaration.of("age", Integer.class)
).out(String.class).expression("name").compile();
// Expression "name" evaluates to list.get(0)
```

There is no `MVEL.getTypeMap()` equivalent for lists — list element types must be known at compile time and declared as `Declaration<?>` parameters. `MVEL.list()` with no arguments creates a list context with no declared variables, making bare method calls like `get(0)` or `size()` unresolvable.

### Why this is non-obvious
Map context (`MVEL.map()`) works without upfront type declarations — `MVEL.getTypeMap(map)` infers types from runtime values. The natural expectation is that `MVEL.list()` would similarly support runtime type inference or at least allow direct method calls on the list. The documentation does not explain the declaration-based API design for list context.
