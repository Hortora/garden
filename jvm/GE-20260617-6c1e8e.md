# Jackson tree copy as polymorphic copy mechanism

### The problem

Need to modify a field (e.g., `deviceId`) on a `DeviceEntity` instance, but:
- `DeviceEntity` is abstract with 16+ concrete subtypes (10 common + 6 vendor supplements)
- Not all types have `toBuilder()` — the 4 types with supplement subclasses deliberately omit it to prevent type-slicing
- Adding `toBuilder()` to abstract parent types creates a trap: calling `toBuilder()` on a `HomeAssistantThermostat` accessed via a `ThermostatDevice` reference returns `ThermostatDevice.Builder`, silently dropping supplement fields

### The technique

Use Jackson's tree model as a polymorphic copy mechanism:

```java
public DeviceEntity copyWithModifiedId(DeviceEntity device, String newId) {
    ObjectNode tree = mapper.valueToTree(device);
    tree.put("deviceId", newId);
    return mapper.treeToValue(tree, DeviceEntity.class);
}
```

This works because:
1. `valueToTree()` serializes the concrete type including the `@deviceType` discriminator property
2. The `ObjectNode` is a mutable JSON tree — modify any field
3. `treeToValue()` deserializes using the discriminator, producing the correct concrete subtype with all fields preserved

The same pattern works for any field modification:

```java
public DeviceEntity markUnavailable(DeviceEntity device) {
    ObjectNode tree = mapper.valueToTree(device);
    tree.put("available", false);
    return mapper.treeToValue(tree, DeviceEntity.class);
}
```

### Why this is non-obvious

The natural approach for modifying an immutable object is `toBuilder()`. When that's unavailable or unsafe (type-slicing), developers typically add per-type copy methods — a maintenance burden across 16+ types. Using Jackson's serialization infrastructure as a copy mechanism is unconventional but leverages infrastructure that's already present (the `@JsonTypeInfo` annotations exist for wire protocol serialization). The performance cost (serialize → tree → deserialize) is acceptable for low-frequency operations like device ID namespacing.
