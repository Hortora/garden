### Problem

A reactive SPI (e.g. `ReactiveWidgetRepository`) has no `@DefaultBean` production implementation — consumers are expected to provide their own. In `@QuarkusTest`, CDI fails with unsatisfied injection or ambiguity because no bean satisfies the SPI.

### Why not mock?

`@InjectMock` replaces a single specific bean. If the SPI has no production bean, there is nothing to replace — you need a real CDI bean that satisfies the injection point.

### Fix

Create a **test-only** `@DefaultBean @ApplicationScoped` implementation that wraps the blocking equivalent using `Uni.createFrom().item()`:

```java
/** Test-only bridge — wraps blocking JPA repo as reactive. @DefaultBean yields to consumer impls. */
@DefaultBean
@ApplicationScoped
public class BlockingReactiveWidgetRepository implements ReactiveWidgetRepository {

    @Inject
    @LedgerPersistenceUnit          // or your named PU qualifier
    EntityManager em;

    @Override
    public Uni<Optional<Widget>> findById(UUID id) {
        return Uni.createFrom().item(
            () -> Optional.ofNullable(em.find(Widget.class, id)));
    }

    // implement remaining SPI methods similarly
}
```

Place in **test sources only** — never in main sources. Because it uses `@DefaultBean`, any consumer-provided `@ApplicationScoped` alternative wins automatically via CDI priority rules.

### Why this works

`@DefaultBean` in Quarkus Arc means "use this unless something else satisfies the injection point." In production, consumers provide their own reactive impl; in tests, the bridge satisfies the SPI without conflict. Same pattern used for no-op defaults like `NoOpTrustImportService`.

### Caveat

The `Uni.createFrom().item()` bridge runs the blocking call on whatever thread invokes it — not on a worker pool. Acceptable for H2/JDBC test contexts; would block the event loop in production. This is test infrastructure only.
