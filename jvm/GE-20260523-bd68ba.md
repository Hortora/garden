## Symptom

`@ObservesAsync` handler runs, JPA `@PrePersist` fires, but a field populated from
`Span.current().getSpanContext().getTraceId()` is null — even though an active HTTP request
span existed when the CDI event was fired.

No error is thrown. The persist succeeds. The `traceId` column is silently null.

## Context

Any Quarkus CDI `@ObservesAsync` handler that runs code which reads the active OTel span
(directly or via a JPA entity listener like `LedgerTraceListener`).

## Root Cause

`@ObservesAsync` dispatches to a Quarkus managed thread pool. OTel span context is stored in
a thread-local (`io.opentelemetry.context.Context`). It does **not** automatically transfer to
the async thread.

When the managed thread calls `Span.current()`, it finds an invalid/no-op span — not the span
that was active on the original Vert.x handler thread. Any field that reads the trace ID from
the current span receives null.

The existing Quarkus OTel extension propagates context for Vert.x handlers and Mutiny pipelines,
but NOT for raw CDI `@ObservesAsync` dispatch.

## Proof

The JPA entity listener itself confirms this with its own test:
`LedgerTraceListenerIT.traceId_nullWhenNoActiveSpan()` explicitly asserts `traceId` is null
when no span is active — the async handler always hits this code path.

## Fix

### Solution 1 — Capture context before the async boundary

On the originating (synchronous) thread, capture `Context.current()` and set the field
explicitly before the async handoff:

```java
// On the synchronous Vert.x handler thread (BEFORE fireAsync):
String capturedTraceId = Span.current().getSpanContext().getTraceId();
// Then in the @ObservesAsync handler, set it directly:
entry.traceId = capturedTraceId.isBlank() ? null : capturedTraceId;
```

### Solution 2 — Wrap async executor with OTel context

Use the OTel context propagation API to wrap the executor used by the CDI event:

```java
Context otelCtx = Context.current(); // captured on calling thread
event.fireAsync(payload, NotificationOptions.builder()
    .setExecutor(otelCtx.wrap(managedExecutor))
    .build());
```

This propagates the span to the async thread transparently.

Solution 1 is simpler and lower risk for isolated write paths. Solution 2 is better when the
async handler does multiple things that each need the span.

## Why This Is Non-Obvious

The Quarkus OTel extension documentation states it propagates trace context automatically.
It does — for Vert.x and Mutiny pipelines. CDI `@ObservesAsync` uses a different thread
pool that is not instrumented the same way. The documentation does not call this out.

A developer seeing null `traceId` in production would first suspect the entity listener
configuration, then the JPA lifecycle, before looking at thread-local context propagation.
