### Symptom

A sponsor notification entity is set to DELIVERED (committed via REQUIRES_NEW), then the
scheduler picks it up again and delivers a second notification to the sponsor. Ledger entries
show both a DELIVERED‚Üísponsor-notifier entry and a FAILED‚Üísponsor-notifier-attempt-failed entry
for the same notification.

### Context

A delivery service implements a three-phase pattern:
- Phase 3a: `store.markDelivered()` ‚Äî REQUIRES_NEW, commits entity to DELIVERED + writes notification ledger
- Phase 3b: `deviationLedgerWriter.writeSponsorNotifiedEntry()` ‚Äî REQUIRES_NEW, writes deviation audit entry

Both Phase 3a and Phase 3b are inside the same `try` block. Phase 3a commits first. If Phase 3b
throws (e.g. deviation datasource unavailable, schema mismatch), the `catch` block calls
`phase3Failure()` ‚Üí `store.markFailed()` ‚Äî another REQUIRES_NEW that sets the entity back to FAILED.

The entity is now FAILED despite delivery having been confirmed by the connector. The scheduler
picks it up and delivers again.

### Root Cause

`store.markDelivered()` uses `@Transactional(REQUIRES_NEW)` ‚Äî it commits independently when the
method returns. The commit is done, permanent, and not rollback-able by any subsequent exception.
When `deviationLedgerWriter.writeSponsorNotifiedEntry()` then throws inside the same try block,
Java's exception handling reaches the catch, which calls `store.markFailed()` ‚Äî another REQUIRES_NEW
that overwrites DELIVERED with FAILED. There is no relationship between these two commits: the second
one has no knowledge that the first already committed a terminal state.

### Fix

Move all secondary ledger writes OUTSIDE the connector try-catch block, with their own isolated
try-catch-log that logs on failure but does not call any entity state update:

```java
// WRONG ‚Äî secondary ledger write inside connector try-catch
try {
    connector.send(message);
    store.markDelivered(id, snapshot, attemptNumber, deliveredAt);       // REQUIRES_NEW ‚Äî commits
    deviationLedgerWriter.writeSponsorNotifiedEntry(..., deliveredAt);    // still inside try!
} catch (Exception e) {
    phase3Failure(id, snapshot, reason, attemptNumber, maxAttempts, policy); // calls markFailed!
}

// CORRECT ‚Äî secondary ledger write outside try-catch
final Instant deliveredAt;
try {
    connector.send(message);
    deliveredAt = clock.instant();
} catch (Exception e) {
    phase3Failure(id, snapshot, e.getMessage(), attemptNumber, maxAttempts, policy);
    return;  // ‚Üê critical: return so the success path below does not run
}
store.markDelivered(id, snapshot, attemptNumber, deliveredAt);  // REQUIRES_NEW ‚Äî commits DELIVERED
try {
    deviationLedgerWriter.writeSponsorNotifiedEntry(..., deliveredAt);    // outside try-catch
} catch (Exception e) {
    LOG.errorf(e, "deviation ledger write failed after successful delivery ‚Äî entity is DELIVERED");
    // Do NOT call markFailed here
}
```

Also add a terminal-status guard at the top of `markFailed()` and `markExhausted()`:
```java
// In store.markFailed():
if (n.status == SponsorNotificationStatus.DELIVERED
        || n.status == SponsorNotificationStatus.EXHAUSTED) return;  // already terminal ‚Äî never regress
```

### Why this is non-obvious

`REQUIRES_NEW` commits are designed to be independent of each other ‚Äî that is their purpose. But
this means a successful commit cannot be "undone" by a subsequent exception in the same method scope.
The Java try-catch mechanism operates on the calling thread's execution stack, not on committed
database state. A developer naturally thinks "if the method throws, the catch handles it and I can
correct the state." That mental model breaks when the first commit has already completed. The result
is a delivered notification that appears retryable to the scheduler ‚Äî causing duplicate delivery
with no error visible to the operator.

### Why this fix

The `return` after `phase3Failure()` in the catch block is critical ‚Äî it prevents fall-through to
the success path that would also call `store.markDelivered()`. The isolated try-catch-log for the
deviation ledger write means audit gaps are visible (ERRORed in logs) without triggering re-delivery.
The terminal-status guard is a belt-and-suspenders defence for any future caller error that bypasses
the code path ordering.
