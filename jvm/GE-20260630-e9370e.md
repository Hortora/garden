### Symptom
`Signature.verify()` returns `false` for P-521 ECDSA signatures that were converted from raw R||S format to DER. No exception — silent verification failure.

### Context
Azure Key Vault returns EC signatures in raw R||S format (R and S concatenated). JCA `Signature.verify()` expects DER-encoded ASN.1. A converter is needed. P-256 (32-byte components) and P-384 (48-byte components) work correctly with naive single-byte length encoding. P-521 (66-byte components) triggers the bug.

### What was tried (didn't work)
Writing the DER SEQUENCE length as a single byte:
```java
out.write(0x30); // SEQUENCE tag
out.write(rDer.length + sDer.length); // length — WRONG for P-521
```

### Root cause
DER length encoding uses two forms:
- **Short form** (length 0-127): single byte
- **Long form** (length 128+): `0x81` prefix byte + length byte(s)

P-521 R and S components are 66 bytes each. With ASN.1 INTEGER padding (leading zero for high-bit values), each INTEGER can be 69 bytes (tag + length + 0x00 + 66 data). Total SEQUENCE content: up to 138 bytes — exceeding the 127-byte short-form limit. Writing 138 as a single byte produces `0x8A`, which DER parsers interpret as "long form, 10 length bytes follow" — completely wrong.

### Fix
```java
int contentLen = rDer.length + sDer.length;
out.write(0x30); // SEQUENCE tag
if (contentLen > 127) {
    out.write(0x81); // long-form: 1 length byte follows
    out.write(contentLen);
} else {
    out.write(contentLen);
}
```

### Why this is non-obvious
P-256 and P-384 always produce content lengths under 127, so the bug never manifests during development with common key sizes. Only P-521 (the least-used NIST curve) triggers it. The failure mode is silent — `Signature.verify()` returns `false` rather than throwing, so it looks like a key mismatch or data corruption, not an encoding bug.

### Why this fix
The long-form check is the minimal correct fix. An alternative is using `BigInteger.toByteArray()` to construct the full DER output, but that duplicates the existing INTEGER encoding logic. The `0x81` prefix handles all realistic EC signature sizes (up to 255 bytes total content).

### Alternatives considered
- **BouncyCastle ASN.1 library** — correct but adds a heavyweight dependency for three lines of DER encoding
- **Java's internal `sun.security.util.DerOutputStream`** — works but relies on JDK internals not guaranteed across versions
