## Symptom

HTTP GET responses with 4xx or 5xx status codes are processed as if they were successful. No exception is thrown. Error body bytes (HTML error pages, JSON error responses) are silently treated as valid response data.

## Context

Polling arbitrary URLs with `java.net.http.HttpClient` and using the response body as data. The code has a `try/catch (IOException e)` block around the `send()` call expecting to catch failed requests.

## Root Cause

`HttpClient.send()` signature:
```java
public abstract <T> HttpResponse<T> send(HttpRequest request, HttpResponse.BodyHandler<T> handler)
    throws IOException, InterruptedException;
```

`IOException` is thrown only for **connection-level failures**: DNS resolution failure, TCP connection errors, connection timeout, SSL handshake failure. A 4xx or 5xx HTTP response is a **successful HTTP exchange** from the network perspective — the server received the request and returned a response. `send()` returns the response with `response.statusCode()` set to the error code; no exception is thrown.

## Fix

Always check the status code explicitly after `send()`:

```java
HttpResponse<byte[]> response = httpClient.send(request, BodyHandlers.ofByteArray());
if (response.statusCode() < 200 || response.statusCode() >= 300) {
    throw new IOException("HTTP " + response.statusCode() + " for " + url);
}
// 2xx: use response.body() safely
byte[] data = response.body();
```

## Why This Is Non-Obvious

`java.net.http.HttpClient` differs from many older HTTP client libraries (Apache HttpClient, OkHttp) that DO throw exceptions for 4xx/5xx by default. Developers migrating from those libraries or relying on exception-based error handling will miss error responses entirely. The consequence — error body bytes silently processed as valid data — can corrupt downstream pipelines with no log entry or observable failure.
