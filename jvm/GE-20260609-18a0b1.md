## Symptom

A JAR is added to `quarkus.index-dependency` in `application.properties` to ensure its beans are discovered by CDI. Tests or the application start failing with errors that look like missing case definitions, unsatisfied dependencies, or `NullPointerException` from beans that appeared to be correctly wired. The failures are intermittent or vary by startup order.

Removing the `quarkus.index-dependency` entry fixes the problem without losing bean discovery.

## Context

`casehub-engine-work-adapter` and `casehub-engine-blackboard` were added to `quarkus.index-dependency` to ensure their `@ApplicationScoped` beans were picked up. Both JARs already ship `META-INF/jandex.idx`. Quarkus discovered them once via the embedded index and a second time via the explicit `index-dependency` config entry, registering their beans twice. This caused CDI to see duplicate bean definitions, leading to `AmbiguousResolutionException` or silent resolution to the wrong bean — manifesting downstream as "CaseDefinition not found" on every `startCase()` call.

## Root Cause

Quarkus's CDI augmentation pipeline has two independent paths for indexing JARs: the automatic path (scanning `META-INF/jandex.idx` files embedded in JARs on the classpath) and the explicit path (`quarkus.index-dependency` entries). If both paths index the same JAR, beans are registered twice. The duplicate registration does not always produce an obvious error — it can manifest as subtle resolution failures or intermittent deployment state, depending on which duplicate Quarkus selects.

## Fix

Only add a JAR to `quarkus.index-dependency` if it does NOT ship `META-INF/jandex.idx`. Check first:

```bash
jar tf path/to/the.jar | grep jandex.idx
# or
unzip -l path/to/the.jar | grep jandex
```

If `META-INF/jandex.idx` appears, the JAR is already auto-indexed. Do not add it to `index-dependency`. If the JAR does NOT ship `jandex.idx`, then adding it explicitly is the correct solution (see GE-20260421-03fa72 / GE-20260523-afab1d).

## Why this is non-obvious

The mental model is: "if beans aren't being found, add the JAR to `index-dependency`." There is no feedback that the JAR is already indexed — Quarkus silently double-scans it and the failure mode looks identical to a missing index: beans appear missing or wrong, case definitions not found. The fix (removing config) is counterintuitive because it looks like removing the thing that should help.
