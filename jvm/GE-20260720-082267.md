### Symptom
Async file-watcher events silently skip files that should be processed. No error,
no warning — the file appears but the handler never fires for it.

### Context
A file-watching system with two concurrent processing paths:
1. **Catch-up reconciliation** — on start, scan for files that appeared between
   replay completion and watcher registration (TOCTOU gap closure)
2. **Async event handler** — DirectoryWatcher fires CREATE events for new files

Both paths share a `ConcurrentHashMap.newKeySet()` dedup set keyed by filename stem
(e.g. `"implementor-3"`). The guard pattern: `if (!processedFiles.add(stem)) return;`

### Root cause
The catch-up scan iterates expected round numbers and calls `processFile(stem)` for
each. The dedup guard `processedFiles.add(stem)` fires first — before the file
existence check. When the file doesn't exist yet (the round hasn't produced it), the
add succeeds (returns true) but the subsequent `waitForFile()` fails and the method
returns early.

The stem is now permanently in the dedup set. When the file actually appears and
the async handler fires, `processedFiles.add(stem)` returns false — the file is
skipped forever.

### Fix
Remove the stem from the dedup set when the file doesn't exist:

```java
String stem = "implementor-" + roundNum;
if (!processedFiles.add(stem)) return;

if (!waitForFile(responsesDir, stem)) {
    processedFiles.remove(stem);  // ← roll back so async handler can process it
    return;
}
```

### Why this is non-obvious
The dedup-first pattern (`add` before doing work) is the standard concurrent
guard — it prevents two threads from processing the same item. The failure mode
only surfaces when the dedup set is shared between a scan that processes
*expected* items and an event handler that processes *actual* items. The scan
assumes all expected items exist; the event handler assumes anything in the set
was already handled.
