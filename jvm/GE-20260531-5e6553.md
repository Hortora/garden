## Summary

When multiple consumers need to know which quality dimensions apply to a given output format, put the mapping as a static method on the dimension enum itself rather than scattering it across consumers or coupling consumers to each other.

## Context

An LLM eval harness evaluates output quality across multiple formats (MARKDOWN, PROSE, A2A_CARD). Different formats have different applicable quality dimensions: prose formats use SECOND_PERSON/CONCISENESS/FACTUAL_FIDELITY/TONE; A2A_CARD uses COMPLETENESS/FACTUAL_FIDELITY only.

Two consumers need this mapping independently: the judge (`PromptJudge`) to select the right schema and parse only applicable keys; the report builder (`EvalReport`) to aggregate means and find the lowest-scoring dimension per format. Without a shared source of truth, they must either be coupled (report calls judge) or each maintain their own copy (drift risk).

## Technique

Add `applicableFor(RenderFormat)` as a static method on the dimension enum:

```java
public enum EvalDimension {
    SECOND_PERSON, CONCISENESS, FACTUAL_FIDELITY, TONE,
    COMPLETENESS;  // A2A_CARD only

    public static Set<EvalDimension> applicableFor(RenderFormat format) {
        return switch (format) {
            case MARKDOWN, PROSE -> EnumSet.of(SECOND_PERSON, CONCISENESS, FACTUAL_FIDELITY, TONE);
            case A2A_CARD        -> EnumSet.of(COMPLETENESS, FACTUAL_FIDELITY);
        };
    }
}
```

Both consumers call `EvalDimension.applicableFor(format)` independently. Neither knows about the other. When a new format is added, the compiler enforces that the switch is updated — there's no way to forget a case.

## Why this is non-obvious

The obvious alternatives:
- **Scatter the mapping**: each consumer maintains its own switch — drift risk when formats are added
- **Centralize in the evaluator**: the report builder calls the judge to get applicable dimensions — wrong coupling direction (report builder shouldn't know about judge)
- **Pass it as a parameter everywhere**: correct but verbose; the caller must know the mapping to provide it

Putting the mapping on the enum type itself is the right coupling: it's metadata about the dimensions that belongs with the dimension type. `EnumSet` is the correct return type (O(1) contains, declaration-order iteration, type-safe).
