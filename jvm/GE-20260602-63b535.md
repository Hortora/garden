## Symptom

A JPA `@Converter` registered with `autoApply = true` begins converting value type fields in DTOs, value objects, or other non-entity classes, causing unexpected failures when those objects are used outside JPA contexts.

## Context

When a value type (e.g. a domain `Path` record) is used in both JPA entity fields and non-JPA contexts (DTOs, REST request/response records, in-memory value objects), registering its converter with `autoApply = true` makes Hibernate apply the conversion to every field of that type it finds — not just entity fields.

## What was tried

Using `@Converter(autoApply = true)` for a `PathAttributeConverter` to avoid annotating each entity field explicitly.

## Root cause

`autoApply = true` registers the converter globally for the Java type. Hibernate's persistence context scans all managed types and applies the converter automatically. Value types shared across entity and non-entity contexts receive the converter where it may be inappropriate and can cause `ConversionException` or null-handling failures in non-JPA paths.

## Fix

Use `autoApply = false` and opt in explicitly per entity field:

```java
// Converter:
@Converter(autoApply = false)
public class PathAttributeConverter implements AttributeConverter<Path, String> { ... }

// Entity field:
@Convert(converter = PathAttributeConverter.class)
public Path path;
```

Conversion is then explicit and scoped to the entity fields where JPA persistence actually needs it.

## Why this is non-obvious

The Javadoc for `autoApply` says "if true, applied automatically to all attributes of the specified type" — which sounds like exactly what you want for an entity field. The risk is that "all attributes" includes non-entity contexts. DTOs and value objects using the same type will be converted in paths where it causes failures.
