## @ObservesAsync CDI events are silently not delivered in @QuarkusTest — call observer directly

### Symptom

`@ObservesAsync` observer methods are never invoked during `@QuarkusTest` runs. No exception, no warning — `Event.fireAsync(event)` completes normally but the observer body never executes.

### Context

Any `@QuarkusTest` class that relies on CDI async event delivery to exercise listener logic. The pattern often surfaces when testing event-driven workflows where a terminal event (e.g. case lifecycle, job completion) is supposed to trigger a listener that updates state.

### Root cause

The async event delivery in Quarkus test context relies on a managed executor whose completion is not awaited within the synchronous test flow. `fireAsync()` returns a `CompletionStage` that the caller discards — the observer may execute after the test has already asserted and exited, or may not execute within the test JVM's lifecycle at all. There is no error because CDI considers delivery complete when the `CompletionStage` is returned, regardless of whether the observer ran.

### Fix

Inject the observer bean directly and call the observer method synchronously:

```java
// ❌ relies on CDI async delivery — silently does nothing in @QuarkusTest
event.fireAsync(myEvent);

// ✅ inject and invoke directly
@Inject MyListener listener;

// in the test body:
listener.onEvent(myEvent); // synchronous — guaranteed to run before assertions
```

Document the workaround in the test comment:

```java
// CDI @ObservesAsync delivery is unreliable in @QuarkusTest context.
// Invoke the listener directly to test the observer logic synchronously.
listener.onCaseLifecycle(completedEvent);
```

### Why this is non-obvious

The symptom is total silence — no exception, no skipped assertion. The obvious debugging path (add logging to `fireAsync()`) confirms the event was fired. Only adding logging inside the observer itself reveals it never ran. The test passes or fails based on pre-observer state, giving false confidence.
