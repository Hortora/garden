## Symptom

An LLM-output completeness check that uses `rendered.content().contains(capabilityName)` always returns `true` for JSON-format output, even when capability descriptions were not generated. `completenessPass = true` regardless of output quality.

## Context

Building an LLM-as-judge eval harness that checks whether all declared capabilities appear in the rendered output. The check was designed for prose formats (markdown, dense prose) where the capability name must appear somewhere in the text. The same check was reused for JSON output (A2A card format).

## Root cause

In a JSON card, every capability object contains a `name` field — so the capability name always appears in the JSON string. `"sprint-planning"` will always be found in `{"name":"sprint-planning","qualityHint":0.9}` regardless of whether a `description` field is present or populated.

The substring check tests for name presence, not description quality. For JSON output, name presence is structural and always guaranteed — it provides no signal.

## Fix

For JSON output formats, parse the rendered JSON and check whether each capability object has a non-empty `description` field:

```java
private List<String> computeA2aMissingDescriptions(EvalCase evalCase, String json) {
    JsonNode root = mapper.readTree(json);
    JsonNode caps = root.get("capabilities");
    if (caps == null || !caps.isArray()) {
        return evalCase.descriptor().capabilities().isEmpty()
            ? List.of()
            : evalCase.descriptor().capabilities().stream()
                .map(AgentCapability::name).toList();
    }
    Map<String, String> descriptions = new HashMap<>();
    caps.forEach(cap -> {
        JsonNode name = cap.get("name");
        JsonNode desc = cap.get("description");
        if (name != null) descriptions.put(name.asText(), desc != null ? desc.asText() : "");
    });
    return evalCase.descriptor().capabilities().stream()
        .map(AgentCapability::name)
        .filter(n -> { String d = descriptions.get(n); return d == null || d.isBlank(); })
        .toList();
}
```

Dispatch by format:
```java
private List<String> computeMissingCapabilities(EvalCase evalCase, RenderedPrompt rendered) {
    if (evalCase.context().format() == RenderFormat.A2A_CARD) {
        return computeA2aMissingDescriptions(evalCase, rendered.content());
    }
    return evalCase.descriptor().capabilities().stream()
        .map(AgentCapability::name)
        .filter(n -> !rendered.content().contains(n))
        .toList();
}
```

## Why this is non-obvious

The substring check is correct and meaningful for prose formats (capability names must appear in the narrative text). The same check looks correct for JSON — "does the JSON contain the capability name?" — but the answer is always yes because JSON serialization puts the name in a structural field. The correctness failure is format-dependent and produces no error, only a false positive.
