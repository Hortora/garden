### Symptom

A `@QuarkusTest` that defines a recording/mock bean (`@ApplicationScoped`) alongside a
`@DefaultBean` for the same interface finds the recording bean is the CDI default — not
the `@DefaultBean`. Test assertions like "no explicit factory configured → InMemory used"
fail because the recording bean supersedes the `@DefaultBean`.

### Context

When a test class defines an inner `@ApplicationScoped` bean for the same SPI interface
that has a `@DefaultBean @ApplicationScoped` implementation elsewhere, Quarkus ARC treats
the non-`@DefaultBean` as higher priority. `@DefaultBean` means "yield to any other bean
for this type" — it is the fallback, not the default.

This surfaces in tests that:
1. Add a recording/spy bean to observe calls (e.g. `RecordingCaseContextStoreFactory`)
2. Also have the production `@DefaultBean` on the classpath (e.g. `InMemoryCaseContextStoreFactory`)
3. Assert that the "default" path uses the `@DefaultBean`

### Root cause

`@DefaultBean` in Quarkus ARC is semantically "use this only if no other bean qualifies."
Any `@ApplicationScoped` bean without `@DefaultBean` is a "real" bean that takes precedence.
When `EngineStrategyResolver` discovers beans via `Instance<>`, the non-`@DefaultBean` appears
as the default strategy — `InjectableBean.isDefaultBean()` returns false for it, so the
resolver treats it as an explicit override.

### Fix

Do not assert that a `@DefaultBean` is used when a non-`@DefaultBean` for the same interface
is also in the CDI context. Instead:

1. Assert on the **outcome** (case starts and completes) rather than which factory was selected
2. Or use `@Alternative @Priority(1)` for the recording bean — alternatives are only active
   when explicitly selected, not when merely present on the classpath

### Why this is non-obvious

The mental model "no config → default bean" is natural but wrong in CDI. `@DefaultBean`
doesn't mean "this is the default" — it means "this is the fallback when nothing else
exists." The presence of ANY non-`@DefaultBean` bean for the same type silently changes
which bean is the CDI default, even if the non-`@DefaultBean` was added for a completely
different purpose (testing, recording, spying).
