## OCC claim must be reset before rethrowing — two-phase claim+save leaves permanent deadlock

### What went wrong

A two-phase coordination pattern: claim (conditional UPDATE sets a flag), then save (version-checked update). If the save throws `SituationConflictException` (OCC version mismatch), the exception propagates to the retry loop — but the claim stays set (`policyTriggered = true`).

On retry: the evaluator re-reads the context (now has the latest `storeVersion`), re-evaluates the trigger policy (still CREATE_CASE), and attempts to claim again. `tryClaimTrigger()` returns `false` — the flag is already set by the previous (failed) attempt. The evaluator interprets this as "another JVM claimed it" and returns without firing the trigger. The situation is permanently dead.

### Context

The claim and save are in separate transactions (`@Transactional(TxType.REQUIRED)` on each store method, evaluator is not transactional). The claim commits before the save starts. When the save fails, the committed claim persists.

### Root cause

The claim-then-save path did not reset the claim before rethrowing the save exception. The retry loop re-reads and re-evaluates, but the stale claim from the failed previous attempt blocks the re-claim.

### Fix

Wrap the save in a try-catch that resets the claim before rethrowing:

```java
boolean claimed = store.tryClaimTrigger(situationId, correlationKey, tenancyId)
        .await().indefinitely();
if (!claimed) return true;
try {
    store.save(context).await().indefinitely();
} catch (SituationConflictException e) {
    store.resetTriggerClaim(situationId, correlationKey, tenancyId)
            .await().indefinitely();
    throw e;
}
```

### Why this is non-obvious

The claim-then-save pattern looks correct on the happy path. The failure mode requires: (1) the claim succeeds in its own transaction, (2) a concurrent writer modifies the entity between the claim and the save, (3) the save detects the version mismatch and throws, (4) the retry loop re-reads and re-evaluates, but (5) the stale claim from step 1 blocks the re-claim. Each step is individually reasonable; the interaction between the committed claim and the retry loop creates the deadlock.
