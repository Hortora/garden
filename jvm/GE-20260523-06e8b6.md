### Symptom

No visible error. A JPQL query with an application-layer limit (e.g. `results.subList(0, limit)`)
works correctly at small scale but becomes an OOM risk when the WHERE clause is broadened — the
database returns every matching row into the JVM heap before the Java truncation runs.

### Context

Using `PanacheEntityBase.list(query, params)` to execute a JPQL query, then truncating the
result list in Java. The pattern appears correct and passes all tests when a filter clause (e.g.
`WHERE channelId = ?`) bounds the result set to a few hundred rows. Removing the filter for a
global scan exposes the unbounded load.

### What was tried (didn't work)

```java
// Appears to work — subList applies the limit
List<Message> results = Message.list("FROM Message WHERE 1=1 ORDER BY id ASC", params);
if (query.limit() != null && results.size() > query.limit()) {
    return results.subList(0, query.limit());
}
```

This loads **all** matching rows from the database into Java heap, then discards everything
beyond the limit. For a table with 1M rows and no WHERE filter, this loads 1M entities.

### Root cause

`PanacheEntityBase.list(query)` executes the JPQL query without any SQL `LIMIT` clause.
It is a convenience method that returns the full result set as a `List`. The `limit` concept
exists only in the caller's Java code (via `subList`), not in the SQL sent to the database.

The Panache API has a separate query-builder path — `PanacheEntityBase.find(query, params)`
returns a `PanacheQuery` which supports `.page(pageIndex, pageSize)`. This `.page()` call
emits `LIMIT N OFFSET M` in the actual SQL.

### Fix

```java
// Correct: SQL-level LIMIT via Panache page()
if (query.limit() != null) {
    return Message.find(jpql.toString(), params.toArray())
            .page(0, query.limit()).list();
}
// Only use list() for unbounded scans where all rows are genuinely needed
return Message.list(jpql.toString(), params.toArray());
```

### Why this is non-obvious

Both patterns coexisted in the same Qhorus codebase: the MCP tool layer used
`Message.find(query).page(0, limit).list()` (correct), while the store layer used
`Message.list(query)` + `subList()` (incorrect). The inconsistency went unnoticed because
a per-channel `WHERE channelId = ?` filter bounded the result set to manageable sizes.
The bug only surfaced when redesigning `getFeed()` to use a global scan (no channel filter),
turning an invisible inefficiency into an OOM risk.

### Why this fix

`find().page()` is the Panache-native way to express SQL LIMIT. It keeps the query builder
chain intact and works with both blocking and reactive Panache variants. The alternative —
using raw `EntityManager.createQuery().setMaxResults()` — bypasses Panache's query DSL and
loses the type-safe parameter binding.

### Alternatives considered

- **Raw JPQL with `setMaxResults()`** — works but bypasses Panache, losing parameter binding and query builder ergonomics.
- **Streaming with `.stream()` and `.limit()`** — Java Streams limit is still post-fetch; the database still returns all rows.
