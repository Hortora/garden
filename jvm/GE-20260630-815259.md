### Symptom
CDI deployment fails with ambiguous bean resolution or unexpected bean displacement when
a neural-text module's `CbrCaseMemoryStore extends CaseMemoryStore` is on the classpath
alongside platform's `CaseMemoryStore` implementations (JPA, SQLite, inmem).

### Context
`GraphCaseMemoryStore extends CaseMemoryStore` works within casehub-platform because all
implementations share the same CDI bean archive. Cross-repo, the inheritance creates three
displacement paths:

1. `QdrantCbrCaseMemoryStore` (neural-text, `@ApplicationScoped`) also satisfies
   `CaseMemoryStore` injection points ‚Äî displaces `JpaMemoryStore` by priority
2. `NoOpCbrCaseMemoryStore` (neural-text, `@DefaultBean`) also becomes a
   `CaseMemoryStore @DefaultBean` ‚Äî conflicts with `NoOpCaseMemoryStore` (platform)
3. Any `@Alternative` on the subtype cascades to the parent type's injection sites

### Root cause
CDI bean type resolution includes all supertypes. A bean typed as
`CbrCaseMemoryStore extends CaseMemoryStore` is injectable at both
`@Inject CbrCaseMemoryStore` and `@Inject CaseMemoryStore`. When the SPI
and its extension live in different repos (different CDI archives), the
priority/default-bean resolution rules interact across archive boundaries
in ways that don't occur within a single archive.

### Fix
Make the extended SPI a **standalone interface** ‚Äî do not extend the base SPI.
Use composition via delegation:

```java
// WRONG ‚Äî causes displacement
public interface CbrCaseMemoryStore extends CaseMemoryStore { ... }

// RIGHT ‚Äî standalone SPI, delegates internally
public interface CbrCaseMemoryStore {
    // own methods only ‚Äî no CaseMemoryStore inheritance
}

// Implementation delegates to injected CaseMemoryStore
@ApplicationScoped
public class QdrantCbrCaseMemoryStore implements CbrCaseMemoryStore {
    @Inject CaseMemoryStore delegate;  // no priority conflict
}
```

### Why this is non-obvious
The `extends` pattern works within a single repo (`GraphCaseMemoryStore extends
CaseMemoryStore` in casehub-platform) because all beans share one CDI archive.
The failure only manifests when the extending SPI lives in a different repo,
which is exactly when composition-over-inheritance is architecturally correct
but not what developers reach for first.

### Alternatives considered
- `@Typed(CbrCaseMemoryStore.class)` annotation to restrict bean types ‚Äî rejected because it
  must be applied to every implementation, is easy to forget on new implementations, and
  doesn't prevent the `@DefaultBean` dual-registration problem
- Qualifier annotation to distinguish CBR from base memory beans ‚Äî rejected because it
  changes the injection site contract for all existing `CaseMemoryStore` consumers
