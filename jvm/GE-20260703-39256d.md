### Symptom
Design review caught an internal contradiction: the spec said "omit score_threshold when minSimilarity is 0.0" AND "score_threshold=0.0 excludes negative cosine similarity." Both cannot be true simultaneously.

### Context
Qdrant `SearchPoints` gRPC message has an optional `score_threshold` field. With `Distance.Cosine`, cosine similarity ranges from -1 (anti-correlated) to 1 (identical). The Java client's `SearchPoints.Builder.setScoreThreshold(float)` sets this field.

### Root cause
`score_threshold` is a protobuf `optional float`. When **omitted** (not set in the builder), Qdrant applies no score filtering — ALL results are returned regardless of similarity score, including negative cosine values. When **set to 0.0**, Qdrant actively filters results with `score < 0.0`.

### Fix
Always set `score_threshold` explicitly, even when the application default is 0.0:

```java
searchBuilder.setScoreThreshold((float) minSimilarity);
// Even minSimilarity=0.0 is meaningful — excludes anti-correlated results
```

Never rely on omission as equivalent to "no minimum threshold" — omission means "no filtering at all," which is rarely what retrieval code intends.

### Why this is non-obvious
The protobuf `optional` semantic makes 0.0 and "not set" distinct values, but most developers treat them as equivalent. The Qdrant documentation describes `score_threshold` as "minimal score to return" without clarifying the omission-vs-zero distinction.
