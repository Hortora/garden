### Symptom
After removing a `@ConfigMapping` abstract method (e.g. `escalationPolicy()`) and cleaning
`runtime/src/main/resources/application.properties`, Quarkus logs an unknown-property warning
at startup — or fails entirely when `quarkus.config.mapping.validate-unknown=true`.

### Context
Quarkus multimodule project. Config cleanup touched the runtime module's
`application.properties` and root `application.properties`, but not
`integration-tests/src/main/resources/application.properties`, which had the same keys
as live (non-commented) properties.

### What was tried (didn't work)
Grepping only within `runtime/src/main/resources/` for the removed key — missed the
integration-tests module entirely.

### Root cause
Integration-tests is a separate Maven module with its own `application.properties`.
It is easy to forget when cleaning up config keys because it sits at a different
depth in the module tree and is not part of the extension runtime itself.

### Fix
After removing a `@ConfigMapping` method, grep across **all** modules:

```bash
grep -r "casehub.work.the-removed-key" . --include="*.properties"
```

This catches `runtime/`, `integration-tests/`, and any other module that carried
the key in its test config. Remove or comment out every hit.

### Why this is non-obvious
The cleanup instinct is to find the `@ConfigMapping` interface and its
`application.properties` in the same module — `runtime/src/main/resources/`.
The `integration-tests/` module is a separate Maven module that often carries
duplicate property definitions for startup validation, and it is not co-located
with the interface being removed.
