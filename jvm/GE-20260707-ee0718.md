### Symptom
`SubCaseMofNOutputMappingTest.bothChildOutputMappings_appliedToParentContext` fails intermittently
(~33% failure rate) in the full blackboard test suite. Passes in isolation.

Assertion: `expected: "left-value" but was: null` — the first child's output mapping
was not applied to the parent context.

### Context
The test starts a parent case that spawns two child sub-cases via two bindings
(`spawn-left` and `spawn-right`). Each binding has a distinct `outputMapping`
(`{ bisectLeft: .result }` and `{ bisectRight: .result }`). The test queries
`SUBCASE_STARTED` event log entries to discover child IDs, then assumes
`childIds.get(0)` is the left child and `childIds.get(1)` is the right child.

### What was tried (didn't work)
- Investigated CaseInstanceCache contamination — no eviction, same object references
- Investigated tenant mismatch — tenancy consistent across threads
- Investigated CaseContext layer mismatch — `set()` and `layer(WORKING)` both delegate to working layer
- Investigated async CONTEXT_CHANGED handler overwriting child context — child has no bindings, handler is no-op

### Root cause
`CaseContextChangedEventHandler.rules()` publishes `SUBCASE_SCHEDULE` events via
`eventBus.publish()` — one per binding, wrapped in `Uni.combine().all()`. Each event
is consumed by `SubCaseExecutionHandler` with `@ConsumeEvent(blocking = true)` on
worker pool threads. Two published events for the same address can process concurrently
on different worker threads, producing SUBCASE_STARTED EventLog entries with
non-deterministic `seq` values. The in-memory EventLog returns results sorted by `seq`,
so `childIds.get(0)` may correspond to either binding depending on thread scheduling.

### Fix
Resolve child identity from the SUBCASE_STARTED event's metadata (which carries the
`outputMapping` field), not from list index position:

```java
for (CaseEventLogRecord event : startedEvents) {
    UUID childId = UUID.fromString(event.metadata().get("childCaseId").asText());
    String outputMapping = event.metadata().get("outputMapping").asText();
    if (outputMapping.contains("bisectLeft")) {
        leftChildId = childId;
    } else {
        rightChildId = childId;
    }
}
```

### Why this is non-obvious
The test passed in isolation and most suite runs because the event bus *usually* delivers
in publication order — the non-determinism only manifests under thread scheduling pressure
from other tests in the suite. The symptom (`bisectLeft` is null) points toward cache
contamination or context layer issues, not ordering. The actual root cause (event bus
delivery order) requires understanding the full chain: binding evaluation → publish →
worker thread pool → EventLog seq assignment.
