## Symptom

`assertEquals(1, alerts.size())` intermittently fails with `expected: <1> but was: <2>`. The test explicitly calls `evaluationService.evaluateAll()` once, but the alert channel receives two messages. Fails maybe 1 in 20 runs ÔÇö no pattern.

## Context

A `@QuarkusTest` with no `@TestTransaction` on a watchdog evaluation test:
1. Test commits a watchdog registration and channel data
2. Test calls `watchdogService.evaluateAll()` Ôćĺ posts 1 alert
3. `WatchdogScheduler` (a `@Scheduled` bean with 60s interval) fires concurrently
4. Scheduler's `evaluateAll()` runs in its **own transaction** ÔÇö it can see the committed watchdog
5. Posts a second alert before `checkMessages()` runs

A secondary failure mode: if a previous test run fails before the delete step, the watchdog persists (no rollback). Next run creates a second watchdog for the same channel ÔÇö `evaluateAll()` fires both, producing 2 alerts.

## Root Cause

Tests without `@TestTransaction` commit data permanently within the session. The Quarkus Scheduler runs on a separate thread with its own transaction context. It sees all committed data ÔÇö including data created mid-test. The 1-second debounce window (`lastFiredAt`) is too short to prevent a 60-second-interval scheduler from re-firing after the test's own evaluation.

## Fix

Add `@TestTransaction` to all tests that call `evaluateAll()` (or any method that reads committed watchdog/domain data):

```java
@Test
@TestTransaction  // ÔćÉ add this
void watchdogFiresAlert() {
    tools.createChannel("wd-barrier-1", "Test", "BARRIER", "alice,bob");
    tools.registerWatchdog("BARRIER_STUCK", "wd-barrier-1", 0, null, "wd-alerts", "admin");
    watchdogService.evaluateAll();  // joins the test transaction ÔÇö scheduler can't see this watchdog
    // ...
}
```

With `@TestTransaction`:
- `evaluateAll()` uses `@Transactional(REQUIRED)` Ôćĺ joins the test's transaction
- The scheduler's thread has a **different** transaction context Ôćĺ cannot see uncommitted watchdog
- All data rolls back at test end Ôćĺ no persistence between runs

Also add UUID suffixes to channel/watchdog names in E2E tests that intentionally lack `@TestTransaction`, as a belt-and-suspenders guard against leftover state from failed runs.

## Why This Is Non-Obvious

The natural mental model: `@TestTransaction` controls rollback for *cleanup*, not visibility isolation. Developers don't think of it as a concurrency control tool. The scheduler looks like an inert background job when you're writing unit-style tests that call the service directly. The intermittency makes it hard to reproduce and the `expected 1 but was 2` symptom suggests a double-fire in the test itself, not interference from a background thread.

### Why this fix

Adding `@TestTransaction` over using `quarkus.scheduler.enabled=false` keeps the scheduler running in tests (so scheduler bugs can be caught), while still isolating individual tests from it. The uncommitted transaction is the correct isolation boundary.

### Alternatives considered

- `quarkus.scheduler.enabled=false` in test properties ÔÇö disables the scheduler entirely; works but means scheduler bugs go undetected
- Increase debounce window ÔÇö reduces race probability but doesn't eliminate it; also changes production semantics
- Use unique channel names per test ÔÇö prevents *cross-run* contamination but not the concurrent scheduler fire
