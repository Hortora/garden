### What this solves

You have a sealed interface in a pure-Java `api/` module (no framework dependencies).
Quarkus REST serializes it to JSON but produces no `"type"` discriminator field — REST
assertions on `response.body.field.type` return null. You cannot add `@JsonTypeInfo` to
the sealed interface because that would add Jackson as a dependency to the pure domain module.

### Technique

Register a Jackson mixin via `ObjectMapperCustomizer` in the `app/` module. The mixin
carries the Jackson annotations without touching the interface in `api/`:

```java
// In app/ — depends on Jackson, keeps api/ pure
@Singleton
public class DomainJacksonConfig implements ObjectMapperCustomizer {

    @JsonTypeInfo(use = JsonTypeInfo.Id.NAME, property = "type")
    @JsonSubTypes({
        @JsonSubTypes.Type(value = MyOutcome.Completed.class, name = "Completed"),
        @JsonSubTypes.Type(value = MyOutcome.Declined.class,  name = "Declined"),
        @JsonSubTypes.Type(value = MyOutcome.Failed.class,    name = "Failed")
    })
    interface MyOutcomeMixin {}

    @Override
    public void customize(ObjectMapper mapper) {
        mapper.addMixIn(MyOutcome.class, MyOutcomeMixin.class);
    }
}
```

REST responses now include `{"type": "Completed", "result": {...}}` for sealed interface fields.

### Why this is non-obvious

Most developers either (a) add `@JsonTypeInfo` directly to the domain interface — violating
the hexagonal architecture rule that `api/` must have zero external dependencies — or (b)
add a DTO layer to avoid the issue entirely. `ObjectMapperCustomizer` is a Quarkus extension
point that most developers don't know exists; the Jackson mixin mechanism itself (`addMixIn`)
is a non-obvious Jackson API that applies annotations from one class to another at registration
time. The combination elegantly solves both problems with about 15 lines of code.
