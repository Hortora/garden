### Symptom
`UnsatisfiedResolutionException` for an SPI type (`PlanItemStore`) at `@QuarkusTest`
startup, despite a `@DefaultBean @ApplicationScoped` implementation existing in an
external module (`casehub-blackboard`) that is on the test classpath.

### Context
`CbrCaseRetainObserver` (runtime module) injects `PlanItemStore`. `NoOpPlanItemStore`
(`@DefaultBean @ApplicationScoped`) lives in `casehub-blackboard`. Adding
`quarkus.index-dependency.blackboard.group-id=io.casehub` +
`quarkus.index-dependency.blackboard.artifact-id=casehub-engine-blackboard` to the
runtime test `application.properties` does not resolve the issue.

### Root cause
Unconfirmed. The blackboard module may have Quarkus extension build-time processing
that interferes with `index-dependency` bean discovery. Other `@DefaultBean` beans
in simpler library JARs (e.g. `NoOpCbrCaseMemoryStore` in `casehub-neocortex-memory`)
ARE discovered via `index-dependency` — the failure is specific to modules with
richer Quarkus integration.

### Fix
Use `Instance<PlanItemStore>` (optional CDI injection) instead of direct injection.
Check `isUnsatisfied()` and return early — when no plan item store is present, there
are no plan items to reconstruct a trace from. The `NoOpPlanItemStore` returns empty
lists anyway, so the behavioral outcome is identical.

```java
// Direct injection — fails at @QuarkusTest startup
@Inject PlanItemStore planItemStore;

// Optional injection — works everywhere
@Inject Instance<PlanItemStore> planItemStoreInstance;

void doWork() {
    if (planItemStoreInstance.isUnsatisfied()) return;
    PlanItemStore store = planItemStoreInstance.get();
    // ...
}
```

### Why this is non-obvious
Other external modules' `@DefaultBean` beans ARE discovered via `index-dependency`
(e.g. `InMemoryCbrCaseMemoryStore`). The developer expects uniform behaviour — if
`index-dependency` works for one external module, it should work for all. The failure
is silent at configuration time and only surfaces as `UnsatisfiedResolutionException`
at startup.
