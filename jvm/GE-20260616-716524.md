### Symptom

You edit `application.properties`, run `mvn quarkus:build`, and the ARC deployment validation still shows the old behaviour — as if the change was never made.

### Context

Running `mvn quarkus:build` directly as a CLI goal after editing a resource file. The build completes with no errors, but the CDI validation reflects the previous config.

### Root cause

`mvn quarkus:build` is a CLI goal (not a lifecycle phase). It does not trigger the `process-resources` phase, so `src/main/resources/application.properties` is **not** copied to `target/classes/`. Quarkus reads from `target/classes/`, which still contains the stale previous version of the file.

### Fix

Use `mvn package -DskipTests` instead of `mvn quarkus:build` when you need to validate CDI wiring after a resource change. `package` runs the full lifecycle including `process-resources`, ensuring `target/classes/application.properties` is current before Quarkus ARC validation runs.

```bash
mvn --batch-mode package -DskipTests -pl app
```

### Why this is non-obvious

Maven lifecycle vs CLI goal semantics are a known subtlety, but `quarkus:build` is a common standalone invocation for validating CDI wiring. Nothing warns you that the resources are stale — the build simply validates the old config, which can appear correct or produce confusing mismatches with the source file you are looking at.
