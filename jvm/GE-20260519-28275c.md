## Symptom

An audit entry (or other durable side-effect) written inside a `@Transactional(REQUIRES_NEW)` method is silently lost when the caller's transaction rolls back. Worse: if the REQUIRES_NEW method itself throws, the audit entry is lost AND the exception propagates, converting a business rejection (409) into an unexpected infrastructure error (500).

## Context

Implementing durable audit entries for blocked lifecycle operations in Quarkus/JTA:
- `claim()` and `delegate()` are `@Transactional` and always roll back when a policy rejection is thrown.
- The audit entry must commit *despite* the outer rollback — hence `REQUIRES_NEW`.
- `BlockedAttemptAuditService.record()` is annotated `@Transactional(TxType.REQUIRES_NEW)`.

## What was tried (didn't work)

- Calling `audit()` inside the outer `@Transactional` method before throwing — rolled back with the rejection.
- Assuming `REQUIRES_NEW` alone was sufficient — it isolates the transaction but does not protect against the method throwing and rolling back its own REQUIRES_NEW transaction.

## Root cause

`@Transactional(REQUIRES_NEW)` suspends the outer transaction and opens a new one. **If the annotated method throws an unchecked exception, JTA rolls back the REQUIRES_NEW transaction too.** The isolation only helps if the inner method commits before any exception escapes.

The pattern therefore requires **two guarantees**:

1. The `REQUIRES_NEW` method returns normally (commits).
2. The caller throws its own exception after the inner method returns.

```java
// WRONG — if auditStore.append() throws, REQUIRES_NEW rolls back and the entry is lost
@Transactional(TxType.REQUIRES_NEW)
public void record(UUID workItemId, String event, String actor, String detail) {
    AuditEntry entry = new AuditEntry();
    // ... populate
    auditStore.append(entry);  // throws PersistenceException on transient DB error
}

// CORRECT — swallow all exceptions so REQUIRES_NEW always commits
@Transactional(TxType.REQUIRES_NEW)
public void record(UUID workItemId, String event, String actor, String detail) {
    try {
        AuditEntry entry = new AuditEntry();
        // ... populate
        auditStore.append(entry);
    } catch (Exception e) {
        // Swallow intentionally — REQUIRES_NEW rollback isolates the failure.
        // The caller throws its own rejection exception regardless.
        LOG.log(Level.WARNING, "Failed to record audit entry for " + workItemId, e);
    }
}

// CALL SITE — always call record() before throwing
final PolicyDecision decision = exclusionPolicy.check(claimantId, item.excludedUsers);
if (decision.denied()) {
    blockedAuditService.record(item.id, "CLAIM_DENIED", claimantId, decision.reason()); // commits
    throw new IllegalStateException(decision.reason());  // outer tx rolls back
}
```

## Why this is non-obvious

Developers reach for `REQUIRES_NEW` thinking "it runs in its own transaction, so the outer rollback won't affect it." That's true for the outer rollback — but not for exceptions thrown inside the `REQUIRES_NEW` method itself. The standard java-dev rule "never swallow exceptions" conflicts here: swallowing is the *correct* behaviour precisely because the outer rejection must not be converted into a 500 by an audit write failure. Both the exception-swallow and the two-statement pattern (record then throw) are required and non-obvious together.

### Why this fix

The alternative — calling audit in the outer transaction — is simpler but wrong: the outer transaction always rolls back on rejection, taking the audit entry with it. A separate async observer (`@ObservesAsync`) would lose the entry if the JVM exits between the throw and delivery. `REQUIRES_NEW` + swallow is the only approach that guarantees durable audit entries on rejection without changing response semantics.
