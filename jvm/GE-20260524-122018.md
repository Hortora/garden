## Maven parent POM bootstrap in CI: project's own pom needs <repositories> to resolve the parent

### Symptom

CI fails immediately with:

```
[ERROR] Non-resolvable parent POM for io.example:child-parent:1.0-SNAPSHOT:
The following artifacts could not be resolved: io.example:parent-pom:pom:1.0-SNAPSHOT (absent)
```

The parent POM was recently published to GitHub Packages. Local builds work fine.

### Context

Multi-repo platform where:
1. A parent POM is published to an artifact registry (GitHub Packages, Artifactory, etc.)
2. The child project inherits from that parent POM
3. The `<repositories>` section (pointing to the registry) is defined IN the parent POM
4. CI runner starts with an empty local Maven repo

### Root cause

Circular dependency: Maven needs to download the parent POM to read the `<repositories>` section that tells it WHERE to download the parent POM. With no local cache and no `<repositories>` in the project's own pom, Maven can only search Maven Central — where the parent POM doesn't exist.

This works locally because the parent POM is already in `~/.m2/repository` from a previous `mvn install`, so Maven never needs to download it from the registry.

### Fix

Add a `<repositories>` section to the project's OWN `pom.xml` (the child, not the parent):

```xml
<repositories>
  <repository>
    <id>github</id>
    <name>GitHub Packages</name>
    <url>https://maven.pkg.github.com/your-org/*</url>
    <snapshots><enabled>true</enabled></snapshots>
  </repository>
</repositories>
```

This gives Maven the registry URL before it attempts to resolve the parent POM. The wildcard URL (`your-org/*`) covers all repos in the org in a single entry.

### Why this is non-obvious

The natural assumption is that the parent POM's `<repositories>` section is inherited and available during parent resolution — but parent resolution must happen BEFORE inheritance can take effect. Local development never surfaces this because the local Maven cache already contains the parent from prior builds.
