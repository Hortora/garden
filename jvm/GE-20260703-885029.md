### Symptom
Test assertion `payload.get("_stored_at").getDoubleValue()` returns `0.0` even though the payload field was set to a non-zero epoch-millis value. No error, no exception — the assertion simply fails with `expected > 0.0 but was 0.0`.

### Context
Storing a timestamp as epoch milliseconds in a Qdrant point payload:
```java
payload.put("_stored_at", ValueFactory.value(Instant.now().toEpochMilli()));
```
The `toEpochMilli()` returns a `long`. `ValueFactory.value(long)` creates a protobuf `Value` with `integer_value` set (not `double_value`).

### Root cause
The Qdrant Java client's `ValueFactory` has separate overloads for `value(long)` and `value(double)`. The `long` overload creates an `IntegerValue` in the protobuf `Value` oneof. Calling `getDoubleValue()` on an `IntegerValue` returns the protobuf default (`0.0`) — not a type-coerced conversion.

### Fix
Use `getIntegerValue()` when the value was stored via `ValueFactory.value(long)`:
```java
assertThat(payload.get("_stored_at").getIntegerValue()).isGreaterThan(0);
```

Or store as double explicitly if downstream code expects `getDoubleValue()`:
```java
payload.put("_stored_at", ValueFactory.value((double) Instant.now().toEpochMilli()));
```

### Why this is non-obvious
The method signature `ValueFactory.value(long)` accepts the value without complaint. The stored payload field works correctly for Qdrant range queries (Qdrant handles int/float comparison internally). The silent failure only surfaces when reading back with the wrong getter — `getDoubleValue()` returns `0.0` instead of throwing.

### Alternatives considered
- Storing as double: works but loses precision for very large longs (>2^53). For epoch millis this is fine through the year 285616.
