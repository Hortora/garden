### Symptom

You're implementing presence detection for an IRC client and expect to passively track other users' AWAY/back status by listening for AWAY messages on the read loop — similar to how Slack broadcasts presence changes.

### Context

Building a `ChatPlatform` SPI implementation for IRC that maps to a `Presence.of(MemberRef)` API. The SPI contract expects `of()` to return the member's current presence status cheaply (no network call per invocation).

### What was tried (didn't work)

Designed the spec with "Presence: native (true) — Tracks AWAY notices from read loop → AWAY/ONLINE." This assumed the IRC server would broadcast AWAY state changes to connected clients, allowing passive tracking in a `Map<String, PresenceStatus>`.

### Root cause

RFC 1459 does NOT broadcast AWAY state changes. The AWAY command only sets the status on the server. Other clients learn about a user's AWAY status in exactly two ways:

1. **RPL_AWAY (301)** — returned as a response to PRIVMSG sent to the away user (not broadcast)
2. **WHOIS** — returns the away reason as part of the WHOIS response (requires an explicit query)

There is no passive notification mechanism in RFC 1459. The IRCv3 `away-notify` capability extension adds this, but it requires CAP negotiation and server support — it is not part of the base protocol.

### Fix

Degrade presence to `UnknownPresence` (returns `PresenceStatus.UNKNOWN` for all members) when implementing against RFC 1459 only. Document that IRCv3 `away-notify` could enable native presence in a future revision.

Making `Presence.of()` send a blocking WHOIS per invocation is technically possible but violates the implicit contract that capability methods should be cheap — the caller has no way to know this particular implementation blocks for a network round-trip.

### Why this is non-obvious

The AWAY command exists, the RPL_AWAY reply exists, and the mental model from modern chat platforms (Slack, Discord) strongly suggests presence changes are broadcast. The absence of a broadcast mechanism is a subtle omission that only becomes clear when you read the RFC carefully and notice AWAY is described entirely in terms of the *sender's* perspective, not the *observer's*.
