### Symptom

Audit/side-effect entities sometimes not persisted, or persisted out of order relative to the
parent entity. Intermittent FK violations. In some Hibernate versions, the new entity is
inserted before the parent entity's flush cycle completes.

### Context

Any pattern where a `@PrePersist` callback on entity A tries to create and persist entity B —
e.g. an audit log entry, a binding record, a second-level write. Appears in enricher pipelines,
`@EntityListeners`, or `Hibernate Interceptors`.

### What was tried (didn't work)

```java
// WRONG — inside @PrePersist of LedgerEntry
@PrePersist
void prePersist(LedgerEntry entry) {
    ActorIdentityBindingEntry binding = new ActorIdentityBindingEntry();
    // ... populate ...
    entityManager.persist(binding); // flush ordering undefined
}
```

### Root cause

The JPA spec (§3.2.4) states that during `@PrePersist` the state of other managed entities
is undefined. Hibernate's flush algorithm processes entities in dependency order based on FK
relationships, not insertion order. Calling `persist()` on a second entity mid-flush can
cause: the new entity to be inserted before the parent (breaking FKs), the new entity to be
silently dropped, or a `ConcurrentModificationException` in Hibernate's flush queue.

### Fix

Decouple the side-effect write using CDI async events + `REQUIRES_NEW`:

```java
// In the @PrePersist or enricher:
event.fireAsync(new AuditNeededEvent(entry.actorId, entry.actorDid, ...));

// In a separate @ApplicationScoped bean:
void onAudit(@ObservesAsync AuditNeededEvent e) {
    persistAuditEntry(e); // runs after @PrePersist completes
}

@Transactional(REQUIRES_NEW)
void persistAuditEntry(AuditNeededEvent e) {
    em.persist(new AuditEntry(...));
}
```

The async observer runs after the parent entity's transaction has flushed. `REQUIRES_NEW`
gives the audit entry its own transaction — it commits independently and failure doesn't
affect the parent write.

### Why this is non-obvious

"I'm in a persist callback — why can't I persist something else?" The analogy to application
code (`save()` calls another `save()`) seems correct. The JPA lifecycle model is distinct from
the application transaction model: a `@PrePersist` callback runs inside the ORM's internal
flush machinery, not just inside a `@Transactional` method.
