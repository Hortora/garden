### Symptom
Test with `rerunFailingTestsCount` configured fails on Run 1 (e.g. Awaitility timeout), then calls `assumeTrue(false, "known blocker")` on Run 2. Expected: test reported as skipped. Actual: Surefire reports the test as an error — the assumption-based skip on the retry does not override the Run 1 failure.

### Context
Occurs when an intermittent infrastructure issue (stale dependency, flaky service) causes a test to fail on first attempt but the retry detects the condition and tries to skip gracefully. Common in integration tests that guard against known external blockers.

### Root cause
Surefire's retry logic treats any run that produced a failure as the overall result. An `assumeTrue` skip on a subsequent run is recorded for that run only — it doesn't retroactively change the Run 1 outcome. The aggregate result is: "1 error" even though the test "knows" it should be skipped.

### Fix
Use `@Disabled("reason")` on the test class or method instead of runtime `assumeTrue` guards when the blocker is known and persistent. `@Disabled` prevents any run attempt, so there's no Run 1 failure to pollute the aggregate.

Reserve `assumeTrue` for conditions that are genuinely runtime-detected and unlikely to differ between retry attempts. If the condition will fail the same way every time, `@Disabled` is the correct mechanism.

### Why this is non-obvious
`assumeTrue` is the standard JUnit 5 mechanism for conditional test skipping, and its javadoc says the test is "aborted" — which reads as "skipped." The interaction with Surefire's retry semantics is not documented in either project. A developer would reasonably expect that a skipped retry overrides a failed first run.
