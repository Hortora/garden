### Symptom

`verify(gatewayClient).invokeAgent(captor.capture())` fails with
"Actually, there were zero interactions with this mock" when a COMMAND
is dispatched with a `correlationId`. The exact same test dispatching
WITHOUT `correlationId` passes — `invokeAgent` is called.

### Context

`@QuarkusTest` with `@InjectSpy MessageService`. `OpenClawChannelBackend`
is registered for the work channel via `channelInitEvent.fire()`. The test
dispatches a COMMAND to the work channel and expects `channelBackend.post()`
to fire → `hookClient.invoke()` → `gatewayClient.invokeAgent()`.

### What was tried

- Verified the backend was registered for the correct `workChannelId`
- Verified the agent registry had the correct agentId/caseId mapping
- Checked for trust policy / ACL blocks (none — channel has no allowedWriters)
- Inspected Qhorus bytecode to trace the `dispatch()` execution path

### Root cause

`MessageService.dispatch()` is `@Transactional`. When `correlationId` is
non-null on a COMMAND, the switch block calls `commitmentService.open()` to
create a `Commitment` entity. In the H2 `@QuarkusTest` environment, this
operation interacts with the JTA transaction context in a way that prevents
`channelGateway.fanOut()` from being reached. The `fanOut()` call is inside
`try { } catch (Exception e) {}` — if it throws, the exception is silently
swallowed. Without `correlationId`, no commitment is created and `fanOut()`
is reached normally.

The observable effect is zero interactions with the gateway mock, with no
error or warning from Quarkus or Mockito.

### Fix

Do NOT dispatch through `messageService.dispatch()` to test a code path
that depends on `OutboundMessage.correlationId()`. Instead, inject the
backend bean directly and call `post()` with a hand-constructed
`OutboundMessage`:

```java
@Inject
OpenClawChannelBackend channelBackend;

@Test
void injection_test() {
    UUID commitmentId = UUID.randomUUID();
    ChannelRef ref = new ChannelRef(workChannelId, workChannelName);
    OutboundMessage msg = new OutboundMessage(
            UUID.randomUUID(), "engine", MessageType.COMMAND,
            "Task content.", commitmentId, null, ActorType.HUMAN);

    channelBackend.post(ref, msg);

    ArgumentCaptor<AgentInvocationRequest> captor =
            ArgumentCaptor.forClass(AgentInvocationRequest.class);
    verify(gatewayClient).invokeAgent(captor.capture());
    assertThat(captor.getValue().message()).contains(commitmentId.toString());
}
```

`post()` is the layer under test. Calling it directly bypasses
`dispatch()` → `fanOut()` entirely and exercises the right behavior
without the H2 transaction complications.

### Why this is non-obvious

Both dispatches (with and without correlationId) look identical to the test
writer. The difference is deep in `MessageService.dispatch()`, only visible
from the JAR bytecode. The silent swallowing of `fanOut()` exceptions masks
the failure completely — the test sees "zero interactions" with no hint of why.
