## Symptom

After adding a module that contains JPA `@Entity` classes as a compile dependency, `@QuarkusTest` classes in downstream modules fail at startup with:

```
ConfigurationException: Datasource '<default>' is not configured.
To solve this, configure datasource '<default>'.
```

This happens even if the downstream module uses only in-memory repository implementations and never interacts with the new dependency's entities.

## Context

Triggered when a module (e.g. `quarkus-ledger`) bundles JPA entity classes in its main artifact. Any module that takes this as a compile dependency — even just to use a utility class like `LedgerTraceIdProvider` — inherits all entity classes on its test classpath. Hibernate ORM's Quarkus extension scans ALL entity classes on the classpath and registers them in the default persistence unit. If no datasource is configured, startup fails.

## What was tried

- Setting `quarkus.flyway.migrate-at-start=false` in test `application.properties` — has no effect because `migrate-at-start` is a build-time fixed property baked into the jar's `META-INF/microprofile-config.properties`.
- Setting `quarkus.hibernate-orm.enabled=false` — disables Hibernate ORM entirely, which breaks test discovery for `@QuarkusTest` classes that depend on any ORM-scoped CDI beans.
- `quarkus.arc.exclude-types` to exclude the problematic beans — does not work for extension-registered beans (registered at augmentation time, not CDI discovery time).

## Root Cause

Quarkus scans the full classpath for `@Entity`-annotated classes at build/augmentation time. Any entity class on the classpath — regardless of which module it came from — is registered in the default persistence unit. There is no built-in scoping mechanism to limit entity discovery to a specific module or jar.

The architectural violation is bundling JPA entity classes alongside domain model/SPI code in a single artifact. The correct pattern: keep domain model (plain POJOs) and JPA entities in separate modules.

## Fix

**Short-term (per affected test module):** Add an H2 datasource configuration to the test `application.properties` so Hibernate can start, plus a `NoOpLedgerEntryRepository` stub (or equivalent) as `@Alternative @Priority(1) @ApplicationScoped` to satisfy CDI:

```properties
# application.properties (test)
quarkus.datasource.db-kind=h2
quarkus.datasource.jdbc.url=jdbc:h2:mem:testdb;MODE=PostgreSQL;DB_CLOSE_DELAY=-1
quarkus.datasource.username=sa
quarkus.datasource.password=
quarkus.hibernate-orm.schema-management.strategy=drop-and-create
quarkus.flyway.migrate-at-start=false
```

```java
@Alternative @Priority(1) @ApplicationScoped
class NoOpLedgerEntryRepository implements LedgerEntryRepository {
  // no-op implementations of all methods
}
```

**Long-term (architectural fix):** Split the offending artifact into:
- `<name>-api` — domain POJOs + SPI interfaces only, zero JPA dependency
- `<name>` or `<name>-hibernate` — JPA entity classes + Flyway migrations

Downstream consumers take `<name>-api` only. The `quarkus-work-api` module is the canonical correct example of this split.

## Why this is non-obvious

The error message says "configure datasource" but the root cause is a transitive dependency bundling entity classes. The module you added was specifically chosen for a non-persistence feature (e.g. a trace ID provider). The natural expectation is that only the entities YOU define affect your persistence unit. There is no warning at dependency declaration time that adding a module will force datasource configuration in your tests.
