### Symptom
`@TestSecurity(user = "test", roles = "user")` on a `@QuarkusTest` class compiles, the test passes, and no warning is emitted — even though the annotation does nothing.

### Context
Occurs when a `@QuarkusTest` class injects CDI beans and calls their methods directly (package-private or public) without any HTTP interaction (no RestAssured, no `@TestHTTPEndpoint`). The `@TestSecurity` extension only intercepts the Quarkus security layer, which is only active for HTTP requests.

### Root cause
`@TestSecurity` works by registering a Quarkus security identity for the current request context. Without an HTTP request, there is no request context and no security layer to intercept — the annotation is evaluated but has no target.

### Fix
Only add `@TestSecurity` to test classes that exercise HTTP endpoints (via RestAssured, `@TestHTTPEndpoint`, or raw `HttpURLConnection`). CDI-only tests — those that inject beans and call methods directly — do not need it and gain nothing from it.

### Why this is non-obvious
The annotation is present and valid on any `@QuarkusTest` class. There's no compiler warning, no Quarkus startup warning, and the test passes. It's easily cargo-culted from HTTP-exercising test classes into CDI-only tests without anyone noticing. The only way to discover it is by reading the Quarkus Security testing source or noticing the annotation has no effect on a test that should fail without auth.
