## Context

A CDI service method needs to fire async observers (via `Event.fire()` or similar) or invoke downstream code that reads the written data — but the writes must be *committed* before that code runs. Putting `@Transactional` on the outer method doesn't help: the transaction commits only when the outer method returns, so the async observer can fire before the commit completes and read stale/absent data.

## The technique

Split the operation into two beans:

1. **Outer method** — NOT `@Transactional`. Calls the inner delegate, then does anything that needs the writes visible.
2. **Inner `@ApplicationScoped` delegate** — `@Transactional`. Owns the writes. When it returns, ArC's proxy commits the transaction. The outer method resumes with committed data.

```java
// Outer bean — NOT @Transactional
@DefaultBean
@ApplicationScoped
public class DefaultOutcomeRecorder implements OutcomeRecorder {

    @Inject OutcomeRecordSaveService saveService;  // separate @ApplicationScoped bean

    @Override
    public void record(OutcomeRecord record) {
        AttestorDefaults attestor = resolveAttestor(record);
        saveService.save(record, attestor);
        // By here: transaction committed. Safe to fire CDI events, invoke async hooks, etc.
        // Future: fireEvent(record); lives here, reads committed writes correctly.
    }
}

// Inner delegate — @Transactional
@ApplicationScoped
class OutcomeRecordSaveService {

    @Inject LedgerEntryRepository ledgerRepo;

    @Transactional
    void save(OutcomeRecord record, AttestorDefaults attestor) {
        LedgerEntry entry = buildEntry(record);
        ledgerRepo.save(entry);
        LedgerAttestation att = buildAttestation(record, entry, attestor);
        ledgerRepo.saveAttestation(att);
    }
}
```

The key: because `saveService.save()` is called on a *different CDI bean*, ArC routes the call through the proxy. The `@Transactional` interceptor wraps `save()`, commits when it returns, then the outer `record()` method resumes. The outer method has full visibility of the committed data.

## Why this is non-obvious

The instinct is to put `@Transactional` on the outermost method — `record()` — and let it manage the full transaction. But that defers the commit until `record()` returns, which is too late if `record()` also fires an async CDI observer: `Event.fire()` schedules the async delivery immediately, but the transaction hasn't committed yet. The observer starts on a worker thread, opens its own transaction, and reads stale or absent data.

The solution is not `REQUIRES_NEW` (which nests a new transaction *inside* the outer one) but eliminating the outer transaction entirely, using a delegate to control exactly where the commit point falls.

Concretely: for the pattern to work, `save()` must be on a separate `@ApplicationScoped` bean — not a private method on the same bean. ArC's proxy intercepts calls *between* beans; `this.privateMethod()` bypasses the interceptor entirely and inherits the caller's transaction context.

### Why this fix

The alternative — using `@Observes(during = TransactionPhase.AFTER_SUCCESS)` on the observer — is correct but requires the observer to be synchronous. For async delivery (`@ObservesAsync`), the phase attribute is not available in the CDI spec (as of CDI 4.0). The delegate-commit pattern is the only portable solution for async observers needing committed data.
