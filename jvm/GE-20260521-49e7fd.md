## Problem solved

In `@QuarkusTest`, test setup uses `@BeforeEach channelStore.put(channel)` (blocking in-memory store) because reactive alternatives like `ReactiveChannelService.create()` require a Vert.x context unavailable on JUnit test threads. But reactive service code under test injects `ReactiveChannelStore` — a different interface backed by a different implementation. The blocking `put()` is invisible to the reactive service.

## Technique

Create a separate reactive delegate bean that injects the blocking in-memory store and wraps every method in `Uni.createFrom().item()`. Because both beans are CDI singletons, they share the same backing state:

```java
@Alternative
@Priority(1)
@ApplicationScoped
public class InMemoryReactiveChannelStore implements ReactiveChannelStore {

    @Inject
    InMemoryChannelStore blocking; // CDI injects the @ApplicationScoped singleton

    @Override
    public Uni<Channel> put(Channel channel) {
        return Uni.createFrom().item(() -> blocking.put(channel));
    }

    @Override
    public Uni<Optional<Channel>> findByName(String name) {
        return Uni.createFrom().item(() -> blocking.findByName(name));
    }

    @Override
    public Uni<List<Channel>> scan(ChannelQuery query) {
        return Uni.createFrom().item(() -> blocking.scan(query));
    }

    @Override
    public Uni<Void> delete(UUID id) {
        return Uni.createFrom().voidItem().invoke(ignored -> blocking.delete(id));
    }
}
```

**Why this works:**
- `@BeforeEach channelStore.put(channel)` writes to `InMemoryChannelStore`
- CDI injects the same `InMemoryChannelStore` singleton into `InMemoryReactiveChannelStore.blocking`
- Reactive service code calls `ReactiveChannelStore.findByName()` → `InMemoryReactiveChannelStore` → delegates to `InMemoryChannelStore` → sees the test data
- `@AfterEach channelStore.clear()` clears `InMemoryChannelStore`, which automatically clears what the reactive view sees

**Key:** Make the `blocking` field package-private (no `private`) so `@BeforeEach` setup code in the same package can inject a specific instance for unit tests that don't use CDI:

```java
@Inject
InMemoryChannelStore blocking = new InMemoryChannelStore(); // CDI overwrites this
```

## Why this is non-obvious

The obvious approaches both fail: you can't make one class implement both interfaces (irreconcilable return types), and you can't call `ReactiveChannelService.create()` from `@BeforeEach` (needs Vert.x context). The delegate pattern via CDI injection to share a singleton is the non-obvious third option — it resolves the shared-state problem without touching the test thread's context requirements.
