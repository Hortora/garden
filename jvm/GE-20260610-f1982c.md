## Symptom

A CDI bean has a `try-catch(ContextNotActiveException)` inside a method body to
gracefully handle background-thread callers that have no request scope. The catch block
is never entered — background callers see the exception propagate despite the guard.

## Context

When implementing an HTTP-aware CDI bean that:
1. Reads from a `@RequestScoped` holder to provide per-request context (e.g. a tenant ID read from a request header), **and**
2. Must fall back gracefully (e.g. return a default value) when called from background threads (`@Scheduled`, `@ObservesAsync`, `StartupEvent` handlers) where no request scope is active.

## What was tried (didn't work)

Making the outer bean `@RequestScoped` with a try-catch inside its method body:

```java
// WRONG — @RequestScoped makes the catch unreachable for background callers
@RequestScoped
@Alternative
@Priority(1)
public class QhorusInboundCurrentPrincipal implements CurrentPrincipal {

    @Inject
    InboundTenancyContext ctx;   // @RequestScoped holder

    @Override
    public String tenancyId() {
        try {
            return ctx.tenancyId();          // this line is never reached from BG threads
        } catch (ContextNotActiveException e) {
            return TenancyConstants.DEFAULT_TENANT_ID;  // unreachable
        }
    }
}
```

Background callers that invoke `currentPrincipal.tenancyId()` get
`ContextNotActiveException` even though the method has a catch.

## Root cause

With `@RequestScoped` on the outer bean, the CDI **client proxy** (the object held by
callers) resolves to the request-scoped bean instance when methods are called. If no
request scope is active, **the proxy itself throws `ContextNotActiveException` before
the method body is entered**. The try-catch inside the method body is never reached —
it is outside the call stack entirely.

With `@ApplicationScoped`, the proxy always resolves (the application scope is always
active). The method body IS entered. Inside the body, accessing the `@RequestScoped`
inner holder (`ctx.tenancyId()`) throws `ContextNotActiveException` from within the
method body — where the catch CAN intercept it.

## Fix

Make the outer bean `@ApplicationScoped`. The per-request state lives in the
`@RequestScoped` inner holder; the outer bean is stateless and delegates:

```java
// CORRECT — @ApplicationScoped proxy always resolves; inner @RequestScoped access throws
// inside the method body where the catch fires
@ApplicationScoped
@Alternative
@Priority(1)
public class QhorusInboundCurrentPrincipal implements CurrentPrincipal {

    @Inject
    InboundTenancyContext ctx;   // @RequestScoped — proxy resolves inside request, throws outside

    @Override
    public String tenancyId() {
        try {
            return ctx.tenancyId();          // throws ContextNotActiveException for BG callers
        } catch (ContextNotActiveException e) {
            return TenancyConstants.DEFAULT_TENANT_ID;  // correctly reached
        }
    }
}

// Separate @RequestScoped holder for per-request state:
@RequestScoped
public class InboundTenancyContext {
    private String tenancyId = TenancyConstants.DEFAULT_TENANT_ID;
    public String tenancyId() { return tenancyId; }
    public void set(String t) { tenancyId = t != null && !t.isBlank() ? t : TenancyConstants.DEFAULT_TENANT_ID; }
}
```

For HTTP requests: `TenancyContextFilter @PreMatching` populates `InboundTenancyContext`.
For background callers: `ctx.tenancyId()` throws, the catch returns the default.

## Why this is non-obvious

The CDI spec and Quarkus documentation don't document the catch-reachability
implication of scope choice. A developer reasonably assumes that placing a try-catch
inside a method provides protection regardless of the bean's scope. The proxy's scope
check precedes method invocation — it is a consequence of how CDI client proxies work
(they must dispatch to a live scope instance; if the scope is inactive, they throw before
the target method is called at all). This is never mentioned in CDI or Quarkus docs for
the ARC implementation.

The failure is silent in production: the catch is not reached, the exception propagates,
and any background-thread caller that expected graceful degradation instead throws.
This can cause data isolation failures if the intent was to fall back to a safe default.

## Constraints

- Applies to any CDI proxy for a narrower scope (`@RequestScoped`, `@SessionScoped`,
  `@ConversationScoped`) injected into a wider scope (`@ApplicationScoped`,
  `@Singleton`) and accessed outside the active scope.
- The fix (split into outer `@ApplicationScoped` + inner `@RequestScoped` holder) only
  works when the outer bean is truly stateless and delegates to the inner holder.
