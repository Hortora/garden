## quarkus.arc.selected-alternatives does not activate @Alternative beans during quarkus:build

### Symptom

CDI deployment fails with 32 `Unsatisfied dependency` errors for `@Alternative @ApplicationScoped` beans even though they are listed in `quarkus.arc.selected-alternatives` in `application.properties`.

### Context

Quarkus multi-module app with `quarkus:build` in the Maven plugin execution. One or more dependency jars provide `@Alternative @ApplicationScoped` implementations of SPIs. Listing them in `quarkus.arc.selected-alternatives` (with or without `%prod.` profile prefix, with or without multiline backslash continuation) has no effect during production augmentation.

### What was tried (didn't work)

- `quarkus.arc.selected-alternatives=com.example.Foo,com.example.Bar` — no effect
- `%prod.quarkus.arc.selected-alternatives=...` — no effect
- Multiline backslash syntax — no effect
- Adding explicit `quarkus.index-dependency.*` for the jar alongside selected-alternatives — still no effect
- Promoting the dependency from test to compile scope — beans visible to augmentation but still not activated

### Root cause

`quarkus.arc.selected-alternatives` activates alternatives for `@QuarkusTest` augmentation (which reads test-scoped `application.properties`) but NOT during `quarkus:build` (production augmentation). The Quarkus docs describe it as a build-time property, but its activation path for production builds is silently broken or not implemented for dependency jars. Zero documentation of this distinction.

### Fix

Create a non-`@Alternative @ApplicationScoped` subclass of each `@Alternative` bean directly in the application's own source (see GE-20260524-d75218). The subclass inherits the implementation, does NOT inherit `@Alternative`, and is `@Default` — satisfying CDI injection without any `selected-alternatives` configuration.

```java
// In the app module's own source (always indexed by Quarkus):
@ApplicationScoped
public class MyEventLogRepository extends InMemoryEventLogRepository {}
```

### Why this is non-obvious

Quarkus documentation states `quarkus.arc.selected-alternatives` is a build-time property, which implies it applies to all augmentations including production. It works correctly for `@QuarkusTest` but silently fails for `quarkus:build`, with no warning or error about unresolvable alternatives — only a CDI deployment failure with 32+ unsatisfied dependency errors that appear to be unrelated to the activation mechanism.
