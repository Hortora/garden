## Symptom

Phantom audit entries appear in the audit trail for operations that were never actually attempted. Specifically, `CLAIM_DENIED` or `DELEGATE_DENIED` entries are written for WorkItems that are in a non-claimable status (e.g. COMPLETED, ASSIGNED), even though the operation was rejected by the status guard, not the exclusion policy.

## Context

`WorkItemService.claim()` and `delegate()` have two sequential guards:
1. Status guard — rejects the operation if the WorkItem is in the wrong status.
2. Exclusion policy check — rejects if the actor is excluded; writes a `CLAIM_DENIED`/`DELEGATE_DENIED` audit entry via `@Transactional(REQUIRES_NEW)`.

When the exclusion check fires before the status guard, an excluded user attempting to claim a COMPLETED WorkItem gets both a `CLAIM_DENIED` audit entry AND a status rejection. The audit entry represents a policy enforcement that had no effect — the status guard would have rejected the operation regardless.

## What was tried (didn't work)

Putting the exclusion check first (before the status guard) seemed natural — "check access control before anything else." Code review surfaced the phantom audit issue.

## Root cause

The `REQUIRES_NEW` audit write commits independently of the outer transaction. If the exclusion check fires and records the audit entry BEFORE the status guard runs, the entry is durably committed even though the status guard then rejects the operation. The client sees a 409 (wrong status), but the audit trail shows a CLAIM_DENIED that didn't represent a real policy enforcement.

## Fix

Reverse the order: status guard fires first. Exclusion check (and audit write) only runs if the status guard passes.

```java
// WRONG — exclusion check before status guard
final PolicyDecision decision = exclusionPolicy.check(claimantId, item.excludedUsers);
if (decision.denied()) {
    blockedAuditService.record(item.id, "CLAIM_DENIED", claimantId, decision.reason()); // committed!
    throw new IllegalStateException(decision.reason());
}
if (item.status != WorkItemStatus.PENDING) {  // fires after phantom audit written
    throw new IllegalStateException("Cannot claim WorkItem in status: " + item.status);
}

// CORRECT — status guard first, exclusion check only if status is valid
if (item.status != WorkItemStatus.PENDING) {
    throw new IllegalStateException("Cannot claim WorkItem in status: " + item.status);
}
final PolicyDecision decision = exclusionPolicy.check(claimantId, item.excludedUsers);
if (decision.denied()) {
    blockedAuditService.record(item.id, "CLAIM_DENIED", claimantId, decision.reason()); // correct
    throw new IllegalStateException(decision.reason());
}
```

## Why this is non-obvious

The issue only surfaces when the policy check has a committed side effect (REQUIRES_NEW audit write). Without REQUIRES_NEW, both guards roll back together and ordering is irrelevant. Once REQUIRES_NEW is introduced, guard ordering becomes a semantic invariant: each guard must only fire if all preceding guards have passed. A code review caught this; the tests didn't — the test suite only exercised exclusion on correctly-statused WorkItems.
