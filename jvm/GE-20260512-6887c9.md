## @ObservesAsync + @Transactional on the same method is unreliable — delegate transactional logic to a separate bean

### Symptom

`@ObservesAsync` method annotated with `@Transactional` silently fails to persist entities or rolls back without throwing. Transactional work appears to complete but database shows no changes.

### Root cause

The CDI spec does not guarantee transaction propagation across async event dispatch boundaries. Annotating an `@ObservesAsync` method with `@Transactional` does not reliably start a new transaction on the async thread — the interceptor may not fire correctly in the async execution context.

### Fix

Delegate transactional work to a separate `@ApplicationScoped @Transactional` bean injected into the observer. The observer method itself carries no `@Transactional`:

```java
// Wrong — @Transactional unreliable on @ObservesAsync method
@ApplicationScoped
public class LedgerCapture {
    @ObservesAsync
    @Transactional  // unreliable
    void onEvent(WorkItemLifecycleEvent e) { repo.persist(entry); }
}

// Right — delegate to a @Transactional bean
@ApplicationScoped
public class LedgerCapture {
    @Inject LedgerCaptureService service;  // @Transactional lives here

    @ObservesAsync
    void onEvent(WorkItemLifecycleEvent e) {
        service.record(e);  // transaction opens and commits here
    }
}
```

### Why this is non-obvious

The annotation is syntactically valid and no error is thrown at startup or runtime. The transaction silently fails to open, or opens and rolls back silently, depending on the thread context. The symptom looks like the observer isn't being called at all.
