## Symptom

Half the equivalence test passes while the other half fails — or vice versa — with no indication that the DSL companion is structurally incomplete. Goal count, names, and kinds assert correctly while completion goal collection comparison fails; or the completion goals match but `def.getGoals()` is empty.

## Context

Writing a fluent Java DSL companion class for a YAML case definition that has goals and an `allOf` completion expression. The YAML mapper populates both `def.getGoals()` and the `AllOfGoalExpression` via an internal `goalMap`. The DSL builder has no equivalent coordination.

## Root cause

`CaseDefinition.Builder.build()` (lines 252–256 of decompiled `casehub-engine-api`) populates the goals list and the completion object via completely independent code paths:

```java
// Lines 252–254: goals list
if (this.goals != null) {
    caseHubDefinition.goals.addAll(this.goals);
}

// Line 256: completion — no interaction with goals list
caseHubDefinition.setCompletion(this.completion);
```

Passing `GoalExpression.allOf(goal1, goal2)` to `.completion()` stores those goal references inside `AllOfGoalExpression.getGoals()` only. It does not add them to `def.getGoals()`. Calling `.goals(goal1, goal2)` does the reverse — it adds to `def.getGoals()` but does not inform the completion expression. Neither call informs the other.

## Fix

Declare `Goal` instances as local variables and register the same instances in both places:

```java
var irbDecided = Goal.builder().name("irb-decided").kind(GoalKind.SUCCESS)
    .condition(".irbConsultation != null").build();

return CaseDefinition.builder()
    .goals(irbDecided)                            // Registration 1 → def.getGoals()
    .completion(GoalExpression.allOf(irbDecided)) // Registration 2 → AllOfGoalExpression.getGoals()
    .build();
```

Using the same `Goal` instances in both is correct. The YAML mapper achieves parity via an internal `goalMap` that seeds both paths from the same source.

## Why this is non-obvious

Builders typically produce a coherent model where setting a value flows into all consuming structures. The expectation is that providing goals via the completion expression registers them — or that `.goals(...)` and `.completion(GoalExpression.allOf(...))` share state. The independence is only visible by reading the decompiled bytecode. A developer who writes only the completion call will see half the equivalence test pass cleanly (goal count, names, kinds from `def.getGoals()`) while the completion goal comparison also passes — because the same goals appear in both paths. But writing only `.goals(...)` without the completion call leaves `def.getCompletion()` null, and writing only the completion call without `.goals(...)` leaves `def.getGoals()` empty.
