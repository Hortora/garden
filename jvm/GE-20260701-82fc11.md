### What it is

Engine issue #484 ("SequenceWorker — lightweight sequential orchestration of workers") was closed as COMPLETED, but no `SequenceWorker` class exists in the engine codebase. The concept was implemented as two separate mechanisms:

1. **`SequentialPlanningStrategy`** (`io.casehub.blackboard.control`) — a `PlanningStrategy` CDI bean that evaluates eligible bindings sequentially: returns only the first non-completed binding, enforcing ordered execution at the case level.

2. **`WorkerRuntime.execute()`** (`io.casehub.api.engine.WorkerRuntime`) — allows a Worker's function to invoke other Workers in sequence within a single PlanItem, providing in-Worker chaining.

### How to find it

```bash
# Looking for SequenceWorker finds nothing:
find casehub-engine -name "SequenceWorker*" -type f  # → empty

# The actual implementation:
find casehub-engine -name "SequentialPlanningStrategy*" -type f
# → blackboard/src/main/java/.../SequentialPlanningStrategy.java
```

### Why this matters

The quarkmind migration spec (`docs/superpowers/specs/2026-06-13-casehub-engine-migration-design.md`) and engine#490 epic both reference `SequenceWorker.of(scoutingWorker, strategyWorker, ...)` — code that doesn't compile against the actual API. Implementers following these specs will search for a class that doesn't exist and must discover the two-mechanism alternative independently.

### Which to use

- **`SequentialPlanningStrategy`**: when you want each Worker as a separate PlanItem with full observability, and ALL bindings in the case should execute sequentially. Set `CaseDefinition.planningStrategy("sequential")`.

- **`WorkerRuntime.execute()`**: when you want to chain Workers inside a single PlanItem, with the case using the default (parallel) planning strategy for other bindings. This is the right choice when sequential and parallel work coexist (e.g. game loop ticks are sequential, advisory Workers are parallel).
