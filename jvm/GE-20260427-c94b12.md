### Symptom
Compilation error: `name clash: filter(Predicate2<B,C>) and filter(Predicate2<Context<CTX>,C>) have the same erasure, yet neither overrides the other`.

### Context
Adding ctx-optional filter overloads to a Permuplate-generated class family (ScopeGate, JoinNFirst). At arity n=2 specifically, two overloads clash:
- All-facts no-ctx: `filter(Predicate2<B, C>)` — 2 fact params, no ctx
- Single-fact ctx-first: `filter(Predicate2<Context<CTX>, C>)` — ctx + last fact

Both erase to `filter(Predicate2)`. Same erasure → Java won't compile.

At n=3+, all-facts no-ctx becomes `Predicate3<B,C,D>` which has a different erasure from `Predicate2<Context<CTX>,D>` — no conflict.

### Root cause
Java's type erasure eliminates generic type parameters at runtime, so `Predicate2<X,Y>` and `Predicate2<A,B>` are indistinguishable to the JVM regardless of what X, Y, A, B are. The conflict only occurs at the boundary where the number of total params (n) equals 2, making both overloads resolve to the same raw type.

### Fix
Suppress the single-fact ctx-first overload specifically at arity n=2 using `@PermuteReturn(when="n != 2")`. At n=2, users who need context use the all-facts ctx-last overload (`Predicate3<B, C, Context<CTX>>`). At n=1 and n≥3, the single-fact ctx-first overload is safe (different Predicate arity from the all-facts no-ctx variant):

```java
// Suppress at n=2 to avoid Predicate2 erasure clash
@PermuteReturn(className = "ScopeGate${n}", ..., when = "n != 2")
public Object filter(
        @PermuteDeclr(type = "Predicate2<Context<CTX>, ${alpha(n+1)}>")
        Predicate2<Context<CTX>, B> pred) { ... }
```

### Why this is non-obvious
The conflict only manifests at one specific arity (n=2). At all other arities everything compiles. Without knowing to look at the arity boundary, you'd expect the overloads to coexist since their generic type params are different. The erasure rule applies to the raw Predicate2 type, not the parameterized form.
