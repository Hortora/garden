## Mutiny Uni.join().all() has no partial-success mode — andCollectFailures still throws

### Symptom
You need N parallel async operations where some may fail, and you want
both successes and failures collected — not an all-or-nothing result.
`andCollectFailures()` sounds like it does this, but it throws a
`CompositeException` if any Uni fails. `andFailFast()` cancels remaining
on first failure.

### Context
Any Mutiny-based parallel I/O where partial failure is expected:
initializing multiple remote servers, health-checking multiple endpoints,
fan-out queries where some targets may be unavailable.

### Root cause
`andCollectFailures()` **does** collect all failures (waits for all Unis
to complete), but it then combines them into a single `CompositeException`
and throws — the successful results are lost. `andFailFast()` cancels
remaining subscriptions on first failure. Neither mode returns a mixed
result of successes and failures.

### Fix
Wrap each Uni to always succeed with a typed result object:

```java
record InitResult<T>(String key, boolean success, T value, Throwable error) {
    static <T> InitResult<T> success(String key, T value) {
        return new InitResult<>(key, true, value, null);
    }
    static <T> InitResult<T> failure(String key, Throwable error) {
        return new InitResult<>(key, false, null, error);
    }
}

List<Uni<InitResult<Session>>> unis = servers.stream()
    .map(s -> initServer(s)
        .onItem().transform(session -> InitResult.success(s.name(), session))
        .onFailure().recoverWithItem(err -> InitResult.failure(s.name(), err)))
    .toList();

Uni.join().all(unis).andFailFast()  // never fails — every element Uni always succeeds
    .invoke(results -> {
        long successes = results.stream().filter(InitResult::success).count();
        // Process mixed results
    });
```

### Why this is non-obvious
The method name `andCollectFailures` strongly implies it collects failures
*alongside* successes — "collect" suggests aggregation, not exception.
A developer reading the API expects it to return something like
`List<Either<T, Throwable>>`. Instead it throws.

### Why this fix
The wrapper pattern is zero-overhead at the Mutiny layer — no custom
operators, no reflection. It composes with existing `Uni.join()` semantics
and preserves input ordering. The alternative (catching `CompositeException`
and manually correlating failures to inputs) loses type safety and ordering.

### Alternatives considered
- Catching `CompositeException` from `andCollectFailures()` — loses the
  successful results entirely; they are not accessible from the exception.
- Using `Multi` with `onFailure().recoverWithItem()` per element — works
  but loses input ordering (merge is unordered).
- Sequential iteration with individual try/catch — correct but loses
  parallelism (N servers × 10s = N×10s instead of ~10s).
