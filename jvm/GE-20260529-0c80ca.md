## What It Is

`StreamingChatModel.chat(ChatRequest, StreamingChatResponseHandler)` is a default method that wraps the caller's handler in an `observingHandler` (for listener infrastructure) before calling `doChat(finalRequest, observingHandler)`. The intended extension point is `doChat()` — not `chat()`.

## Context

When writing test mocks for `StreamingChatModel`, the natural instinct is to implement `chat(ChatRequest, StreamingChatResponseHandler)` directly — it's the method you call in production code. But `chat()` is a `default` method in the interface that routes to `doChat()`. Overriding `chat()` bypasses the listener infrastructure and may produce subtly different behaviour from production.

## How To Use It

```java
StreamingChatModel mock = new StreamingChatModel() {
    @Override
    public void doChat(ChatRequest request, StreamingChatResponseHandler handler) {
        // handler here is the observingHandler — calling it correctly fires both
        // the observer infrastructure and the original caller's callbacks
        handler.onCompleteResponse(
            ChatResponse.builder().aiMessage(AiMessage.from(jsonResponse)).build());
    }
};
```

`doChat()` receives the `observingHandler` wrapper, so calling `handler.onCompleteResponse()` fires listener events AND routes to the caller's original handler. This matches the production call path exactly.

## Why It's Not In The Docs

`doChat()` is the internal override point declared in the interface body. The official documentation only shows `chat()` usage examples. The routing through `observingHandler` is only visible in the source code.
