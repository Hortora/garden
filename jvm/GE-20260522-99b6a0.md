## @ApplicationScoped CDI proxy swallows field writes — use @Singleton for test doubles needing direct field access

### Symptom

A test double bean (e.g. `@Mock TestSlackConnector`) has a public instance field (`List<Message> sent`). Tests inject the bean and call methods on it. The field appears always empty in assertions even though `send()` is demonstrably being called.

```java
@Mock
public class TestSlackConnector implements Connector {
    public List<ConnectorMessage> sent = new ArrayList<>();

    @Override
    public void send(ConnectorMessage msg) {
        sent.add(msg);  // called — but assertions always see empty
    }
}

// In test:
@Inject TestSlackConnector slackConnector;

assertThat(slackConnector.sent).hasSize(1);  // fails: []
```

### Root cause

Quarkus ArC generates a **CDI client proxy** (subclass) for every `@ApplicationScoped` bean. When you inject the bean, you get the proxy, not the real instance. Method calls on the proxy dispatch to the real instance via `arc$delegate()`. **Field access does not** — Java field access is not polymorphic. Reading `proxy.sent` reads the proxy subclass's own `sent` field, which is initialised to `new ArrayList<>()` and never touched. The real instance's `sent` field accumulates messages; the proxy's does not.

The result: `send()` correctly populates the real instance, but `slackConnector.sent` reads the proxy's empty list.

### Why this is non-obvious

The proxy is invisible at the source level. Injecting `TestSlackConnector` feels like injecting the real object. The symptom (always-empty list) looks exactly like the `send()` call never fired, which sends the investigation in the wrong direction. The distinction between method dispatch (proxied) and field access (not proxied) is not called out in any CDI or Quarkus documentation that surfaces during normal development.

### Why this fix

`@Singleton` beans in Quarkus ArC are **not proxied** — they are injected directly. No proxy subclass is generated; both method calls and field access go to the real instance. This makes `@Singleton` the correct scope for test doubles where direct field access is required.

### Fix

Annotate the test double with `@Singleton`:

```java
@Mock
@Singleton  // NOT @ApplicationScoped — Quarkus ArC does not proxy @Singleton
public class TestSlackConnector implements Connector {
    public List<ConnectorMessage> sent = new ArrayList<>();
    public boolean shouldThrow = false;

    public void reset() {
        sent.clear();
        shouldThrow = false;
    }

    @Override
    public String id() { return "slack"; }

    @Override
    public void send(ConnectorMessage msg) {
        if (shouldThrow) throw new RuntimeException("Connector failure");
        sent.add(msg);
    }
}
```

`@Mock` without an explicit scope inherits the replaced bean's scope. If the real `SlackConnector` is `@ApplicationScoped`, the mock will be too unless you override it explicitly with `@Singleton`.

### Alternatives considered

- **Accessor methods only (no field access):** works — method calls dispatch correctly through the proxy. But existing test code using `assertThat(slackConnector.sent)` must be rewritten to `assertThat(slackConnector.sent())`.
- **Keep static fields:** avoids the proxy issue entirely (static fields are class-level, not instance-level). But breaks the protocol for stateful test bean isolation and prevents parallel test execution.
- **`@Dependent` scope:** not proxied either, but each injection point gets a new instance — so the instance `DefaultSponsorNotifier` injects is different from the one the test class injects, making state capture impossible.
