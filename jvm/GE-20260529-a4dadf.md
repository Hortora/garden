## Context

A multi-module Maven project follows the SPI-in-api pattern: the api module declares the SPI interface and wants to co-locate the abstract contract test with the SPI. Runtime modules that implement the SPI want to extend the abstract contract test to verify their concrete implementation.

By default, `api/src/test/java` classes are not exported to downstream modules — the test classpath is not part of the published jar.

## Technique

**Step 1 — Producer: publish a test-jar from the api module**

Add to `api/pom.xml`:
```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-jar-plugin</artifactId>
    <executions>
        <execution>
            <goals><goal>test-jar</goal></goals>
        </execution>
    </executions>
</plugin>
```

This produces `casehub-eidos-api-<version>-tests.jar` alongside the regular jar during `mvn package`/`mvn install`.

**Step 2 — Consumer: declare a tests-classified dependency**

Add to `runtime/pom.xml` (and any other consuming module):
```xml
<dependency>
    <groupId>io.casehub</groupId>
    <artifactId>casehub-eidos-api</artifactId>
    <classifier>tests</classifier>
    <scope>test</scope>
</dependency>
```

The consumer module's test compilation now resolves `AgentStateStoreContractTest` from the api test-jar.

**Step 3 — Use the contract**

```java
// In runtime module
@QuarkusTest
@TestTransaction
class JpaAgentStateStoreTest extends AgentStateStoreContractTest {
    @Inject AgentStateStore store;

    @Override
    protected AgentStateStore store() { return store; }
}
```

All abstract test methods from the api module are inherited with no duplication.

## Why This Is Non-Obvious

Most developers know that test-scope dependencies are not transitive, but the existence of the `classifier=tests` dependency and the `test-jar` plugin goal is not widely known. The error when attempting to extend the contract (compilation failure: class not found) gives no hint about the test-jar mechanism. The natural workaround — duplicating tests in each implementing module — defeats the contract pattern.

## Constraints

The api module must be installed (`mvn install -pl api`) before consuming modules can resolve the test-jar during compilation. In CI, build order must ensure the api module is installed before downstream modules compile their tests. Both `mvn install` and `mvn package` produce the test-jar.
