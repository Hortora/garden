### Symptom

WorkItems created from templates via the engine's `HumanTaskScheduleHandler` have correct database state but the `CREATED` lifecycle event carries stale values for `scope`, `candidateGroups`, `candidateUsers`, `payload`, and `permittedOutcomes`. Downstream observers (ledger, queue filter engine, SSE) receive the template defaults, not the engine's resolved values.

Assignment strategies that ran during `create()` used the template's `candidateGroups` instead of the engine's resolved groups. Timer deadlines were set from the template's expiry, not the engine's effective deadline.

### Context

`WorkItemTemplateService.instantiate()` calls `workItemService.create(request)` which:
1. Persists the entity
2. Fires the `CREATED` CDI event with the entity's current field values
3. Runs assignment strategy against `candidateGroups`
4. Schedules timers based on `expiresAt`
5. Returns the persisted `WorkItem`

Then `HumanTaskScheduleHandler.handleTemplateMode()` overwrites fields on the returned entity:
```java
workItem.scope = target.scope();
workItem.candidateGroups = toCsv(event.resolvedCandidateGroups());
workItem.candidateUsers = toCsv(event.resolvedCandidateUsers());
workItem.payload = serializePayload(event.inputData());
workItem.permittedOutcomes = OutcomeCodecs.encodeOutcomes(toOutcomeList(target.outcomes()));
workItem.persist();
```

The second `persist()` updates the database row (JPA dirty checking), but the CDI event was already fired with the pre-override values.

### What was tried (didn't work)

The natural assumption is that setting fields before `persist()` is sufficient. But `create()` already called `persist()` internally — the event fired at that point, not at the handler's `persist()`.

### Root cause

The creation lifecycle (event firing, assignment, timer scheduling) runs inside `workItemService.create()`, BEFORE the caller has a chance to override fields. The caller's post-creation mutations are invisible to the lifecycle.

### Fix

Merge request overrides with template defaults BEFORE calling `create()`. The `createFromTemplate(WorkItemCreateRequest request)` method resolves the template, applies request-wins-for-non-null merge semantics, then passes the fully-merged request to `create()`. All lifecycle operations see the final values.

```java
WorkItemCreateRequest merged = mergeRequestWithTemplate(template, request, expandedExcludedUsers);
WorkItem workItem = workItemService.create(merged);
```

Implemented in casehub-work#275.

### Why this is non-obvious

The code reads correctly at the handler level: create from template, override some fields, persist. The corruption is invisible unless you trace through `create()` to see that events fire INSIDE it with the pre-override values. The database row ends up correct (JPA dirty checking + second persist), which masks the problem — only event consumers see the stale data.

### Alternatives considered

- Fire events lazily (at transaction commit instead of at create time) — rejected: breaks the existing sync-observer contract and complicates the event ordering model
- Have the handler call a special "create without events" method then fire events manually — rejected: proliferates special-case creation paths and moves lifecycle responsibility to the caller
