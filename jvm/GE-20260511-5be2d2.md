### Symptom

Running `mvn install -f aggregator.xml -rf :some-module` produces impossible compilation
errors in the resumed module — e.g. "interface expected here" when the source file
clearly declares `public interface Foo`. Or "cannot find symbol" for a constant that
visibly exists in the source.

### Context

Occurs in Maven multi-module reactor builds when:
1. An earlier module (e.g. `casehub-engine-common`) fails to compile mid-run
2. Maven installs whatever it managed to produce (stale or partial) to `.m2/repository`
3. You resume with `-rf :downstream-module`, skipping the failed dependency
4. The downstream module compiles against the stale `.m2` jar — not the fixed source

### Root cause

Maven's `-rf` (resume from) flag skips all modules before the specified one, including
modules that previously failed. If a failed module installed a stale jar to `.m2`,
`-rf` happily uses that stale jar as the dependency — it never checks whether it
reflects the current source.

In this session: `casehub-engine-common` failed to compile (missing `serverlessworkflow`
deps) and installed a partial jar with the old `class WorkflowExecutor` definition.
Multiple `-rf :casehub-engine` attempts all used that stale jar, producing the baffling
"interface expected here" error in `ServerlessWorkflowExecutor`.

### Fix

Before resuming, explicitly rebuild the failed dependency:

```bash
# 1. Clean install only the failed dependency (and its own deps)
JAVA_HOME=$(/usr/libexec/java_home -v 26) \
  mvn clean install -pl casehub-engine-common,casehub-engine-api --batch-mode -q

# 2. Now resume from the downstream module — it will find the fresh jar
JAVA_HOME=$(/usr/libexec/java_home -v 26) \
  mvn install -f aggregator.xml --batch-mode -rf :casehub-engine
```

### Why this is non-obvious

The error message ("interface expected here", "cannot find symbol") appears in the
*downstream* module and points to source code that is visibly correct. Nothing in
Maven's output mentions the stale `.m2` dependency as the cause — the build log shows
the dependency was "resolved" (from `.m2`) which looks like success. Standard debugging
instinct (re-reading the source, checking the import) finds nothing because the source
is fine.

### Why this fix

Explicitly rebuilding the failed module forces a fresh `.m2` install before resuming.
Alternative — `--no-transfer-progress` or force-update flags — does not help here
because the stale jar is local, not remote.

### Alternatives considered

- Re-running `-rf :casehub-engine-parent` to include the common module: still uses
  the stale `.m2` jar if the module appears to have already "installed" in the reactor
  session state.
- Deleting the `.m2/repository/io/casehub/casehub-engine-common` directory manually:
  works but requires knowing which exact artifact is stale.
