### Symptom

OpenTelemetry spans in reactive Mutiny chains show `StatusCode.UNSET` even when the operation failed with an exception. The error is silently lost — no warning, no log, no indication that span error recording was attempted.

### Context

When managing OTel span lifecycle in a Mutiny reactive chain, you need both `onFailure().invoke()` (to record the error on the span) and `onTermination().invoke()` (to close the scope and end the span). The order these are chained matters.

### Root cause

In Mutiny, operators closer to the upstream (source) fire **first** during signal propagation. When chained as:

```java
// WRONG — onTermination fires BEFORE onFailure
return upstream
    .onTermination().invoke(() -> {
        scope.close();
        span.end();        // span ends here
    })
    .onFailure().invoke(t -> {
        span.setStatus(StatusCode.ERROR);   // no-op — span already ended
        span.recordException(t);             // no-op — span already ended
    });
```

`onTermination` fires first (closer to upstream), ending the span. Then `onFailure` fires, but per the OTel API specification, **mutations to an ended span are no-ops** — the error status and exception are silently discarded.

### Fix

Chain `onFailure` **before** `onTermination`:

```java
// CORRECT — onFailure fires BEFORE onTermination
Span span = tracer.spanBuilder("operation").startSpan();
Scope scope = span.makeCurrent();
return upstream
    .onFailure().invoke(t -> {
        span.setStatus(StatusCode.ERROR);   // span still active — recorded
        span.recordException(t);             // span still active — recorded
    })
    .onTermination().invoke(() -> {
        scope.close();
        span.end();        // span ends last — all mutations complete
    });
```

On success, `onFailure` does not fire; `onTermination` correctly ends the span with OK status. On cancellation, only `onTermination` fires.

### Why this is non-obvious

1. Mutiny's operator ordering (closer to upstream = fires first) is the opposite of what many developers expect from builder-pattern method chaining, where "last added = last executed"
2. The OTel API silently discards mutations to ended spans — there is no exception, warning, or log when you call `setStatus()` on an ended span
3. The combination means the bug produces no error signal at all — tests pass (the span exists), the operation fails correctly (exception propagates), but the span's error metadata is empty

### Alternatives considered

- **`try-with-resources` pattern**: works for blocking code but not for reactive chains where the span must outlive the method call
- **Single `onTermination` with error check**: `onTermination` receives a `Throwable` parameter but using it requires more complex lambda logic and is less readable than the two-operator pattern
