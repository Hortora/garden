## @Transactional on a called method is silently dropped when invoked via self-call from an @ObservesAsync observer

**Stack:** Quarkus CDI, Jakarta Transactions

**Symptom:** `jakarta.enterprise.context.ContextNotActiveException: Cannot use the EntityManager/Session because neither a transaction nor a CDI request context is active.` ‚Äî thrown inside business logic that has `@Transactional` on its method signature.

**Context:** An `@ObservesAsync` observer method calls `this.someMethod()` where `someMethod()` is annotated `@Transactional`. The call path is: CDI event fires ‚Üí observer method (no transaction) ‚Üí `this.someMethod()` ‚Üí Panache/JPA ‚Üí exception.

### Root cause

CDI transaction interception (and all interceptors) only fire when a method is called *through the CDI proxy*. Calling `this.someMethod()` directly bypasses the proxy entirely ‚Äî `@Transactional` on `someMethod()` has no effect. The observer method itself is called through the proxy (it is a valid interception point), but `this.someMethod()` inside it is a direct Java call.

```java
// BROKEN ‚Äî @Transactional on process() does nothing; called via this
public void onMessage(@ObservesAsync MessageReceivedEvent event) {
    process(event.channelName(), event.messageType(), event.senderId()); // this.process()
}

@Transactional  // interceptor never fires ‚Äî bypassed by self-call
public void process(String channelName, MessageType type, String sender) { ... }
```

### Fix

Add `@Transactional` to the observer method itself. The observer method is called through the CDI proxy, so interceptors fire correctly on it. The `this.process()` call then runs within the transaction started by `onMessage()` (REQUIRED propagation joins the existing TX):

```java
@Transactional  // transaction started here ‚Äî via the CDI proxy on the observer
public void onMessage(@ObservesAsync MessageReceivedEvent event) {
    process(event.channelName(), event.messageType(), event.senderId()); // joins existing TX
}

@Transactional  // still useful for direct test calls; joins outer TX when called from onMessage
public void process(String channelName, MessageType type, String sender) { ... }
```

### Why this is non-obvious

`@Transactional` on a method feels like a complete specification: "this method needs a transaction". Developers don't think about whether the method is being called through a proxy or not ‚Äî that's an implementation detail of CDI that's invisible at the call site. The exception also points at the JPA call inside the method, not at the missing transaction boundary, making it look like a JPA configuration problem rather than a CDI proxy bypass.

See also: GE-20260512-6887c9 ‚Äî a related entry noting that combining `@ObservesAsync` and `@Transactional` on the same method is documented as unreliable in some contexts. The finding here is complementary: for Quarkus 3.32, adding `@Transactional` directly to the observer method works correctly. The critical constraint is that the business logic called within must not assume its own `@Transactional` will activate via self-call.
