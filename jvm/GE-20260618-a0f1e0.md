### Behaviour

Without `--verbose`, `claude --output-format stream-json` emits only the final `ResultMessage` — no incremental `AssistantMessage` text deltas appear during generation. The subprocess appears hung until it exits, then produces a single result line.

With `--verbose`, incremental `AssistantMessage` JSON lines appear as the model generates tokens, enabling true token-level streaming.

### Context

Only relevant when consuming `--output-format stream-json` output by reading subprocess stdout line-by-line and dispatching on message type. Without `--verbose`, the output is equivalent to non-streaming batch mode despite using the stream-json format.

### Discovery

`StreamingTransport.buildStreamingCommand()` (claude-code-sdk 1.0.0, line 416) unconditionally adds `--verbose` with no comment:

```java
command.add("--verbose");
```

Removing it from a one-shot subprocess command (`-p "prompt" --output-format stream-json`) and running produces only the final result message. No CLI documentation or SDK docs explain this dependency.

### Fix

Always include `--verbose` when using `--output-format stream-json` and expecting incremental streaming:

```
claude -p "prompt" --output-format stream-json --verbose
```

### Why This Is Non-Obvious

`--output-format stream-json` sounds like it controls the output format, not the verbosity of output. A developer implementing a one-shot subprocess consumer following the flag documentation would omit `--verbose`, observe the correct final output, and only discover the missing streaming when comparing latency or checking intermediate events.
