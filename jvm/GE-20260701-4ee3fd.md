### Symptom

All `@QuarkusTest` tests fail with `DeploymentException: Found 33 deployment problems` listing every engine persistence SPI as unsatisfied: `CaseInstanceRepository`, `EventLogRepository`, `CaseMetaModelRepository`, `CrossTenantCaseInstanceRepository`, `CrossTenantEventLogRepository`, `SubCaseGroupRepository`.

### Context

When a CaseHub harness app (quarkmind, AML, devtown) adds `casehub-engine` (full runtime) as a compile dependency, the engine's internal CDI beans (`WorkflowExecutionCompletedHandler`, `CaseHubReactor`, `PlanningStrategyLoopControl`, etc.) inject persistence SPIs at Quarkus augmentation time. `casehub-engine-persistence-memory` provides `@Alternative` in-memory implementations for all 6 persistence SPIs.

### Root cause

Quarkus augmentation (CDI bean validation) runs before the test classpath is loaded. If `casehub-engine-persistence-memory` is `<scope>test</scope>`, its `@Alternative` beans are invisible during augmentation — the engine's SPIs have no implementations and validation fails with 33 unsatisfied dependencies.

This contradicts the intuitive expectation that test-scoped dependencies are available during test runs. Quarkus augmentation is a build step that executes with compile-scoped dependencies only.

### Fix

Change `casehub-engine-persistence-memory` from test to compile scope in `pom.xml`:

```xml
<dependency>
    <groupId>io.casehub</groupId>
    <artifactId>casehub-engine-persistence-memory</artifactId>
    <!-- compile scope — NOT test. Engine CDI beans inject these SPIs at augmentation time. -->
</dependency>
```

Also add all in-memory implementations to `quarkus.arc.selected-alternatives`:
```properties
quarkus.arc.selected-alternatives=\
  io.casehub.persistence.memory.InMemoryCaseInstanceRepository,\
  io.casehub.persistence.memory.InMemoryEventLogRepository,\
  io.casehub.persistence.memory.InMemoryCaseMetaModelRepository,\
  io.casehub.persistence.memory.MemorySubCaseGroupRepository,\
  io.casehub.persistence.memory.MemoryPlanItemStore
```

### Why this is non-obvious

The Maven scope `test` normally means "available during test execution." But Quarkus augmentation (which validates CDI wiring) runs as a build step with only compile-scoped dependencies visible. A developer adding `casehub-engine` would naturally put its test stubs in test scope — and get 33 cryptic deployment errors with no indication that scope is the issue.

### Why this fix

The alternative — adding `@DefaultBean` no-ops for all 6 SPIs in the engine runtime itself — would mask missing persistence configuration in production deployments. Compile-scope in-memory stubs with explicit `selected-alternatives` activation is the deliberate CaseHub pattern (see PLATFORM.md `casehub-platform` scope rule).

### Alternatives considered

- **`@DefaultBean` no-ops in engine runtime** — rejected: would silently swallow data in production if a real persistence module is missing
- **`quarkus.arc.exclude-types` on engine beans** — rejected: excludes too much; engine beans are needed for `signalAndAwaitSync` to work
