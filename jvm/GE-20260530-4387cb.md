## Symptom

In a `@QuarkusTest` that relies on `InMemoryMessageStore` (from `casehub-qhorus-testing`), calling `messageService.findAllByCorrelationId(correlationId)` returns an empty list even though messages were dispatched to that correlationId moments before in the same test. The messages exist in the InMemory store but are invisible to this method.

## Context

`OversightGateService.fulfill()` calls `messageService.findAllByCorrelationId(gateId.toString())` to look up the original COMMAND message's Long ID (needed as `inReplyTo` for the RESPONSE dispatch). In `@QuarkusTest` with InMemory stores, this always returns empty — causing the service to log "no COMMAND message found" and fail open.

## Root Cause

`MessageService.findAllByCorrelationId()` is implemented with a Panache static call:

```java
public List<Message> findAllByCorrelationId(final String correlationId) {
    return Message.<Message> find("correlationId = ?1 ORDER BY id ASC", correlationId).list();
}
```

This bypasses the `MessageStore` interface entirely and queries Hibernate/H2 directly. The `InMemoryMessageStore` (the `@Alternative @Priority(1)` test bean) stores messages in a `ConcurrentHashMap` — it is never consulted by Panache static calls.

Most other `MessageService` methods inject and delegate to `MessageStore` — those work correctly with InMemory stores. Only the methods that use Panache static calls are affected.

## Fix

Wrap the problematic method with `@InjectSpy` in the test class and override only the bypassing method to scan the InMemory store:

```java
@InjectSpy
MessageService messageService;

@Inject
InMemoryMessageStore messageStore; // or inject the store interface directly

@BeforeEach
void mockFindAllByCorrelationId() {
    doAnswer(inv -> {
        String correlationId = inv.getArgument(0);
        return messageStore.findAll().stream()
            .filter(m -> correlationId.equals(m.correlationId))
            .sorted(Comparator.comparingLong(m -> m.id))
            .collect(Collectors.toList());
    }).when(messageService).findAllByCorrelationId(anyString());
}
```

All other `messageService` calls continue to use the real implementation; only `findAllByCorrelationId` is bridged to the InMemory store.

## Why This Is Non-Obvious

The `InMemoryMessageStore` gives the impression that all Qhorus data access is intercepted in tests. It isn't — only access that flows through the `MessageStore` interface is intercepted. Methods that call Panache directly bypass it silently: no error, just empty results. The real `InMemoryMessageStore` is injected correctly; it simply never receives the query.

**Resolved in: casehub-qhorus 0.2-SNAPSHOT (2026-06-01, casehubio/qhorus#228)** — All four Panache-direct methods (`findByCorrelationId`, `findAllByCorrelationId`, `findResponseByCorrelationId`, `findDoneByCorrelationId`) now route through `messageStore.scan()`. `MessageQuery` gained `correlationId(String)` and `messageType(MessageType)` include filters; `MessageQueryJpql` and `InMemoryMessageStore.scan()` both apply them. The `@InjectSpy` workaround is no longer needed on fixed versions. The `invalidation_triggers` field predicted this fix exactly.

### Alternative — Mock MessageService + direct fanOut (ChannelBackend testing)

**Approach:** When testing a `ChannelBackend` implementation that calls `MessageService` read methods, mock `MessageService` entirely with `@InjectMock` and trigger the backend via `gateway.fanOut()` directly instead of going through `MessageService.dispatch()`.

**When to use:** Testing the `post()` logic of a `ChannelBackend` in isolation — verifying it calls the LLM, dispatches the right `MessageType`, and sets the correct `inReplyTo`. Does not test the full dispatch pipeline.

**Pros:** Avoids the store-seam issue entirely; tests the backend in isolation; fast (no full Qhorus transaction overhead).
**Cons/trade-offs:** Does not prove the backend integrates correctly with the full dispatch→fanOut→commitment lifecycle.

```java
@QuarkusTest
class MyChannelBackendTest {
    @Inject ChannelGateway gateway;
    @Inject MyChannelBackendFactory factory;

    @InjectMock MessageService messageService;   // mock to control findByCorrelationId
    @InjectMock MyLlmService llm;               // mock to control LLM output

    @Test
    void queryDispatchesResponse() {
        UUID channelId = UUID.randomUUID();
        String correlationId = UUID.randomUUID().toString();

        // Stub the inReplyTo lookup
        Message stubQuery = new Message();
        stubQuery.id = 42L;
        when(messageService.findByCorrelationId(correlationId)).thenReturn(Optional.of(stubQuery));
        when(llm.review(any(), any(), any(), any(), any()))
            .thenReturn(new ReviewResult(false, "Looks good."));

        // Register the backend
        factory.put(session);
        gateway.initChannel(channelId, new ChannelRef(channelId, "my-prefix/" + sessionId));

        // Trigger via fanOut — bypasses MessageService.dispatch() entirely
        gateway.fanOut(channelId, "my-prefix/" + sessionId, new OutboundMessage(
            UUID.randomUUID(), "human:user", MessageType.QUERY, "Is this clear?",
            UUID.fromString(correlationId), null, ActorType.HUMAN));

        // Verify the backend dispatched a RESPONSE
        ArgumentCaptor<MessageDispatch> captor = ArgumentCaptor.forClass(MessageDispatch.class);
        verify(messageService).dispatch(captor.capture());
        assertThat(captor.getValue().type()).isEqualTo(MessageType.RESPONSE);
        assertThat(captor.getValue().inReplyTo()).isEqualTo(42L);
    }
}
```
