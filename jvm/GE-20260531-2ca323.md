## Context

Wiring a HikariCP pool to a xerial SQLite JDBC datasource with PRAGMA settings applied
via `SQLiteConfig`. The HikariCP docs show `addDataSourceProperty` as the standard way
to pass configuration to the underlying `DataSource`. The xerial `SQLiteConfig.toProperties()`
method returns a `java.util.Properties` object with all PRAGMA settings.

## Symptom

```java
SQLiteConfig config = new SQLiteConfig();
config.setJournalMode(SQLiteConfig.JournalMode.WAL);
config.setBusyTimeout(5000);

HikariConfig hikari = new HikariConfig();
hikari.setDataSourceClassName("org.sqlite.SQLiteDataSource");
hikari.addDataSourceProperty("url", "jdbc:sqlite:/path/to/db");
hikari.addDataSourceProperty("config", config.toProperties());  // ← fails at runtime

HikariDataSource ds = new HikariDataSource(hikari);  // throws IllegalArgumentException
```

HikariCP's `PropertyElf` tries to call `setConfig(Properties)` via reflection. There is
no `setConfig(Properties)` method on `SQLiteDataSource` — only `setConfig(SQLiteConfig)`.
Reflection coercion of a `Properties` object to `SQLiteConfig` fails silently or throws at startup.

## Root Cause

`HikariConfig.addDataSourceProperty(String, Object)` stores values that `PropertyElf` later
applies via setter reflection on the underlying `DataSource` class. `Properties` and `SQLiteConfig`
are different types — the setter `SQLiteDataSource.setConfig(SQLiteConfig)` exists, but `PropertyElf`
cannot coerce a `Properties` object to `SQLiteConfig` via reflection.

## Fix

Construct `SQLiteDataSource` directly with the pre-configured `SQLiteConfig`, then pass
the pre-built DataSource instance to HikariCP:

```java
SQLiteConfig sqLiteConfig = new SQLiteConfig();
sqLiteConfig.setJournalMode(SQLiteConfig.JournalMode.WAL);
sqLiteConfig.setSynchronous(SQLiteConfig.SynchronousMode.NORMAL);
sqLiteConfig.setBusyTimeout(5000);
sqLiteConfig.setCacheSize(64000);

SQLiteDataSource sqLiteDataSource = new SQLiteDataSource(sqLiteConfig);
sqLiteDataSource.setUrl("jdbc:sqlite:/path/to/db");

HikariConfig hikari = new HikariConfig();
hikari.setDataSource(sqLiteDataSource);   // ← pass pre-built instance, not class name
hikari.setMaximumPoolSize(5);

HikariDataSource ds = new HikariDataSource(hikari);
```

`setDataSource(DataSource)` bypasses `PropertyElf` entirely — HikariCP wraps the
provided instance directly.

## Why This Is Non-Obvious

The HikariCP docs show `setDataSourceClassName` + `addDataSourceProperty` as the standard
pattern for configuring a `DataSource`. The xerial docs show `SQLiteConfig.toProperties()`
as a way to export the config for use with JDBC `DataSource` implementations. Neither doc
mentions the incompatibility. The failure mode is a startup-time `IllegalArgumentException`
or `InvocationTargetException` that mentions `setConfig` — which sends you chasing
`SQLiteDataSource` rather than HikariCP.

## Constraints

Applies to any HikariCP + xerial sqlite-jdbc integration where PRAGMA settings are
configured via `SQLiteConfig`. The `setDataSource(DataSource)` fix is universally valid —
`setDataSourceClassName` is just an alternative entry point, not a requirement.
