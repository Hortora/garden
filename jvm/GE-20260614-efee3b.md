### Symptom
Synthetic CDI bean created via `SyntheticBeanBuildItem.configure().createWith()` compiles cleanly, deploys without error, but throws at runtime when `ctx.getInjectedReference()` is called inside the `createWith` function. No build-time warning, no deployment failure — the error surfaces only when the bean is actually instantiated.

### Context
When building a Quarkus extension that creates synthetic CDI beans, the `createWith()` function receives a `SyntheticCreationalContext` whose `getInjectedReference()` method resolves CDI beans. Unlike normal CDI injection (where the container infers injection points from `@Inject` annotations), synthetic beans must **explicitly declare** every injection point via `.addInjectionPoint()` on the `SyntheticBeanBuildItem.ExtendedBeanConfigurator`.

### Root cause
The Quarkus Arc CDI container does not infer injection points for synthetic beans. Without `.addInjectionPoint()` declarations, the `SyntheticCreationalContext` has no knowledge of what beans to resolve. The `getInjectedReference()` call fails because the bean type was never registered as a dependency of the synthetic bean.

### Fix
For every `ctx.getInjectedReference(SomeType.class)` call inside the `createWith` function, add a matching `.addInjectionPoint(ClassType.create(SomeType.class))` on the configurator:

```java
SyntheticBeanBuildItem.configure(MyBean.class)
    .addInjectionPoint(ClassType.create(DotName.createSimple(EmbeddingModel.class)))
    .addInjectionPoint(ClassType.create(DotName.createSimple(EmbeddingStore.class)))
    .createWith(recorder.createMyBean(info))
    .setRuntimeInit()
    .done();
```

For complex cases with many injection points, extract a helper method that mirrors the `getInjectedReference()` calls:

```java
static void addInjectionPoints(ExtendedBeanConfigurator configurator, MyCreateInfo info) {
    if (info.componentA().mode() == EXPLICIT) {
        configurator.addInjectionPoint(ClassType.create(DotName.createSimple(info.componentA().className())));
    }
    // ... one per getInjectedReference call
}
```

### Why this is non-obvious
The compile + deploy success creates false confidence. Most CDI injection errors surface at deployment time (Quarkus's build-time validation catches missing beans, ambiguous resolution, etc.). Synthetic beans bypass this validation — the container trusts the extension to declare its own dependencies. The failure mode is uniquely silent: no warning, no suggestion, just a runtime exception in a code path that looked correct at build time.

### Alternatives considered
- Automatic inference from recorded bytecode — not supported by Quarkus Arc
- Using `Instance<T>` instead of `getInjectedReference()` — doesn't work in `SyntheticCreationalContext`

### Reference
Confirmed in `EasyRagProcessor` (quarkus-langchain4j) which explicitly declares `.addInjectionPoint()` for `EmbeddingStore` and `EmbeddingModel`. The `@RagPipeline` implementation added a shared `addRagInjectionPoints()` helper to ensure the injection point declarations always mirror the `getInjectedReference()` calls.
