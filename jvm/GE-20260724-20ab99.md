### The technique

When a reusable class implements a foundation SPI (e.g. `FaultPolicy`) but has zero runtime dependencies — only API types and JDK classes — it belongs in the API module, not the runtime module.

The zero-dep test: if the class compiles with only `api` module classes + `java.base` on the classpath, it's an API-tier type. Placing it in runtime forces every consumer to take a compile dependency on the full runtime (CDI container, internal beans, transitive dependencies).

### Context

`ThresholdFaultPolicy` was initially placed in `io.casehub.desiredstate.runtime` alongside `CbrFaultPolicy` and `FaultPolicyEngine`. The reasoning was proximity — they're all `FaultPolicy` implementations.

But `CbrFaultPolicy` and `FaultPolicyEngine` are `@ApplicationScoped` CDI beans with runtime dependencies (ConfigurationRetriever, PreferenceProvider, etc.). `ThresholdFaultPolicy` has zero CDI annotations and zero runtime dependencies — only API types (`FaultPolicy`, `FaultEvent`, `GraphMutation`, etc.) plus `ConcurrentHashMap`.

Domain modules (`casehub-ops/iot`, `casehub-ops/app`) depend on `casehub-desiredstate-api` at compile scope. They depend on `casehub-desiredstate` (runtime) only at test scope. Moving `ThresholdFaultPolicy` to `api` let them use it without pulling in the full runtime.

### The discriminator

| Characteristic | API module | Runtime module |
|---|---|---|
| CDI annotations (`@ApplicationScoped`) | No | Yes |
| Framework dependencies | No | Yes |
| Instantiation | Consumer calls `new` / builder | CDI discovers and injects |
| Dependencies | Only API types + JDK | Runtime internals, CDI, framework |

### Why this is non-obvious

The natural instinct is to group by SPI — all `FaultPolicy` implementations together. But the module boundary should follow the dependency graph, not the type hierarchy. A consumer-instantiated utility with API-only dependencies has more in common with the SPI interface than with CDI beans that share the same package.
