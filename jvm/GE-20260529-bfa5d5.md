## Symptom

Observer implemented as `@ObservesAsync WatchdogAlertEvent` attempts to look up a domain record by `event.correlationId()` — the method doesn't exist. Or the observer routes on `event.correlationId()` to find the specific commitment that triggered the alert, but `correlationId` is not a field on `WatchdogAlertEvent`.

## Context

`WatchdogEvaluationService.evaluateApprovalPending()` fires one `WatchdogAlertEvent` per Watchdog when expired open Commitments are found on the watched channel. A developer implementing `LifeWatchdogAlertObserver` naturally assumes the event carries the `correlationId` of the expired Commitment — it does not. The event is aggregate, not per-Commitment.

## WatchdogAlertEvent fields (actual)

```
notificationChannel: String    ← channel NAME string (not UUID)
watchdogId: UUID
conditionType(): WatchdogConditionType   ← method derived from context
firedAt: Instant
context: AlertContext          ← ApprovalPendingContext{pendingCount: long, oldestExpiryAt: Instant}
```

No `correlationId`. No list of commitment IDs. The event tells you WHICH channel had expired Commitments and HOW MANY, not WHICH specific Commitments.

## Fix

Query your domain supplement table by `channelId = event.notificationChannel()` AND `status = PENDING_RESPONSE` AND `deadline <= now`. Process all matching records:

```java
@ObservesAsync
@Transactional(Transactional.TxType.REQUIRES_NEW)
void onAlert(WatchdogAlertEvent event) {
    if (event.conditionType() != WatchdogConditionType.APPROVAL_PENDING) return;

    List<LifeCommitmentRecord> expired = LifeCommitmentRecord
        .findExpiredPendingByChannel(event.notificationChannel(), Instant.now());

    for (LifeCommitmentRecord record : expired) {
        createEscalationTask(record);
        record.status = CommitmentStatus.EXPIRED;
        record.updatedAt = Instant.now();
        record.persist();
    }
}
```

Store the channel NAME string (not UUID) in your domain supplement's `channelId` column so the query matches `event.notificationChannel()`.

## Why this is non-obvious

`WatchdogEvaluationService` finds expired Commitments on the watched channel, then fires an event. The natural assumption is the event carries the Commitment's correlationId — it doesn't. The evaluator fires once per Watchdog (per channel), not once per expired Commitment. The correlationId lives in the qhorus `Commitment` entity itself; the Watchdog event only carries the aggregate context (pendingCount, oldestExpiryAt).

### Why this fix

Querying by channel name aligns with the one-event-per-Watchdog semantic — the observer handles all expired records on that channel in one atomic REQUIRES_NEW transaction. Attempting to correlate by individual Commitment ID would require calling `CommitmentStore.findAllOpen()` to match, which duplicates the evaluator's work and creates a race with concurrent evaluations.
