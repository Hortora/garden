### Context
JPA with Flyway handles schema migrations (including indexes) declaratively. MongoDB has
no equivalent mechanism in Quarkus — there is no migration runner, no `migrate-at-start`,
no DDL auto-generation. Indexes must be created explicitly before they are needed.
Without a canonical pattern, developers either create indexes out-of-band (fragile, not
reproducible) or reach for `@Observer(StartupEvent.class)` (works but verbose).

### Technique
Use `@Startup @ApplicationScoped` with a `@PostConstruct` method. `@Startup` forces
eager bean initialization at application startup before any request is processed.
`@PostConstruct` runs the index creation once. MongoDB's `createIndex` is idempotent —
calling it on an already-existing index is a no-op.

```java
import com.mongodb.client.model.Indexes;
import io.quarkus.runtime.Startup;
import jakarta.annotation.PostConstruct;
import jakarta.enterprise.context.ApplicationScoped;

@Startup
@ApplicationScoped
class MyCollectionIndexes {

    @PostConstruct
    void ensureIndexes() {
        MyDocument.mongoCollection().createIndex(Indexes.ascending("scope"));
        // Add composite indexes, text indexes, etc. as needed
        // All calls are idempotent — safe to run on every startup
    }
}
```

The class can be package-private — it is not part of the public API.

**Why `mongoCollection()` from a static context works:** Panache MongoDB entity statics
delegate to the active CDI request scope at call time. In a `@PostConstruct` method,
CDI is fully initialized, so `mongoCollection()` resolves correctly.

### Why this is non-obvious
The Quarkus MongoDB docs don't describe a canonical startup initialization pattern for
indexes. The alternatives a developer typically reaches for:

- `@Observer(StartupEvent.class)` — valid but requires CDI event plumbing
- Out-of-band scripts — not reproducible in CI/new environments
- `@ApplicationScoped @PostConstruct` alone — bean may be lazy-initialized; indexes
  won't be created until the first injection point is activated, which may happen after
  the first query
- Flyway — not applicable to MongoDB

`@Startup` is the missing piece: it forces eager initialization regardless of whether
any other bean has injected this one. The combination with `@PostConstruct` keeps the
pattern explicit and testable.
