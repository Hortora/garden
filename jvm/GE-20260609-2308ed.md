## The technique

When testing a fast-fail guard (blank token, missing config, any precondition that should abort before making an HTTP call), asserting the return value alone does NOT prove the guard fired.

If the guard is removed and the HTTP call is made with invalid input, WireMock returns an unmatched-request response. The production code catches the error and returns the same failure string or empty list. The test passes even though the guard is gone.

**Add this assertion after every fast-fail test:**

```java
wireMock.verify(0, WireMock.anyRequestedFor(WireMock.anyUrl()));
```

This verifies that WireMock received exactly zero requests — proving the guard prevented the HTTP call, not just that the error path returned the correct value.

## Example

```java
@Test
void sendSlackBot_blankToken_returnsFailedWithoutHttpCall() {
    final SlackBotMcpTool blankTool = new SlackBotMcpTool(client, bridge, "");

    final String result = blankTool.sendSlackBot("C123ABC", "Hello", null);

    assertThat(result).isEqualTo(
            "Failed: casehub.connectors.slack-bot.token is not configured");
    assertThat(bridge.lastConnectorId).isNull();
    wireMock.verify(0, WireMock.anyRequestedFor(WireMock.anyUrl())); // ← the critical assertion
}
```

Without the last line, removing the blank-token guard entirely would not fail this test.

## Why this is non-obvious

The test name says "WithoutHttpCall" and the assertion checks the return value and bridge state. Developers assume those assertions are sufficient evidence. They're not — they prove what the code returned, not what code path was taken to get there. WireMock's default unmatched-request behaviour (return 404 or throw, depending on config) is caught by the production exception handler, which returns the same failure value the guard would have returned. The error path and the guard path produce indistinguishable output.
