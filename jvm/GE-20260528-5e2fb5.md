### Symptom
After `git mv src/main/resources/db/migration src/main/resources/db/work/migration`, running `mvn test` shows all tests passing. Migrations appear to apply correctly — tables exist, queries succeed. The move appears to have worked.

Later, after `mvn clean install` (or installing to another repo as a JAR dependency), the same tests fail with "Table X not found". The tests that passed were lying.

### Context
A Quarkus extension moves its Flyway migrations from `classpath:db/migration` to `classpath:db/work/migration` via `git mv`. `mvn test` runs the tests. Quarkus's Flyway scanner finds SQL files at the OLD path (`target/classes/db/migration/`) because Maven's `process-resources` phase copies resources from `src/main/resources/` to `target/classes/` but does NOT delete stale files from the old path. The old directory in `target/classes/` remains with all its SQL files intact. Flyway scans `classpath:db/migration` (the Quarkus default), finds the stale files, and applies migrations successfully. The tests pass — but against stale artifacts, not the new path.

### Root cause
Maven's `maven-resources-plugin` copies files from `src/main/resources/` to `target/classes/` on `process-resources`. It does NOT delete files at paths that no longer exist in `src/main/resources/`. After `git mv`, `src/main/resources/db/migration/` is gone, so nothing is copied to `target/classes/db/migration/`. But the previous build's output (`target/classes/db/migration/*.sql`) is still there. Flyway sees it.

The false positive only surfaces as a failure when:
1. `mvn clean` runs (wipes `target/`) and the stale files disappear
2. The module is installed as a JAR dependency — the installed JAR is built from `target/classes/` at install time, which may or may not include the stale files depending on when the install ran

### Fix
Always run `mvn clean install -DskipTests` on any module where resource directories have been renamed or moved before running tests or publishing to the local Maven repository:

```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn clean install -DskipTests -pl <module>
```

The `clean` phase deletes `target/` entirely, eliminating stale artifacts. Then re-run the tests against the clean build:

```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn test -pl <module>
```

### Why this is non-obvious
Tests passing after a resource move is the expected signal that the move worked. There is no warning, no error, no indication that migrations ran from a stale path. The distinction between "tests pass because new path works" and "tests pass because old path is stale" is invisible without inspecting `target/classes/` manually. The issue only surfaces on a clean build or when the JAR is consumed by a dependent module.
