## ZIP content hash must be computed after all entries are appended

### Symptom
`fullHashVerification()` always reports hash mismatch on closed ZIP archives, even though no tampering occurred.

### Context
A `ZipCorpusStore` manages rolling ZIP archives with a chain manifest (`chain.json`). Each closed ZIP has a `contentHash` (SHA-256) stored in the manifest for integrity verification. During rollover, an internal `_chain/meta.json` metadata entry is appended to the ZIP being closed.

### What was tried (didn't work)
Computing the hash before writing `_chain/meta.json`:
```java
String contentHash = computeHash(activeZipPath);  // hash WITHOUT meta.json
writeInternalMeta(active, contentHash);            // appends to ZIP — changes the file
manifest.closeEntry(active.uuid(), contentHash, entryCount);
```
The stored hash covers the ZIP without its final entry. Every subsequent verification recomputes the hash over the complete file (including meta.json) and mismatches.

### Root cause
SHA-256 hashes the entire file byte stream. Appending any entry after hashing changes the file content — central directory is rewritten, end-of-central-directory moves. The hash is permanently wrong from the moment it's stored.

### Fix
Write internal metadata first (with null hash placeholder), then compute the hash:
```java
writeInternalMeta(active, null);                   // append meta first
String contentHash = computeHash(activeZipPath);   // hash the COMPLETE file
manifest.closeEntry(active.uuid(), contentHash, entryCount);
```

### Why this is non-obvious
The natural reading order ("compute hash, write hash into metadata, close") suggests hash-first. The dependency runs backwards: the hash depends on the metadata entry being present, but the metadata entry wants to contain the hash. Breaking the cycle by accepting a null hash in the metadata is the correct trade-off — `chain.json` is the authoritative source for the hash, not the internal copy.

### Alternatives considered
- Computing hash excluding internal meta: complex (would need to exclude specific bytes from the hash), fragile, and defeats the purpose of integrity verification
- Two-pass: write meta with placeholder, hash, then update meta in-place with real hash: Zip4j doesn't support in-place entry updates
