### Symptom

A corpus isolation test asserts that searching corpus2 for `ChatModel` returns empty results, even though only corpus1 contains that term. The test fails: corpus2 returns a result.

### Context

Two test corpuses:
- corpus1: `"ChatModel in Quarkus"` (should match)
- corpus2: `"Python ML model"` (should NOT match)

The BM25 tokenizer produces `[chatmodel, chat, model]` for the query `ChatModel`. The token `model` appears in corpus2's document `"Python ML model"`. BM25 correctly finds it — the corpus isolation is working (separate indexes), but the test data shares a common token.

### Root cause

Test data design — not a code bug. The query `ChatModel` decomposes to `[chatmodel, chat, model]`. Any document containing `model` in any corpus will match on that token. The test falsely assumes no token overlap between the query and corpus2's content.

### Fix

Use test documents with zero token overlap with the query:

```java
// Bad — "model" overlaps with ChatModel → [chatmodel, chat, model]
registry.addChunk(corpus2, "doc2", "Python ML model", Map.of(), Map.of());

// Good — no tokens in common with ChatModel
registry.addChunk(corpus2, "doc2",
    "Hibernate lazy loading fails outside transaction", Map.of(), Map.of());
```

### Why this is non-obvious

With standard word tokenizers, `ChatModel` is a single token — no overlap with `model`. With CamelCase-aware tokenizers that split compounds into components, the token set expands and overlaps become common. Test data that looks obviously different (`ChatModel` vs `Python ML model`) shares a token through CamelCase decomposition. The failure only surfaces when the tokenizer is CamelCase-aware — it works with standard tokenizers.
