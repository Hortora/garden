### Symptom
`CbrCaseMemoryStore.store()` returns a valid UUID case ID. `CbrCaseMemoryStore.retrieveSimilar()` with matching tenant, domain, scope, and features returns an empty list. No error, no warning, no log output.

### Context
The `CbrCaseMemoryStore` interface is consumed from a decompiled `.class` file. The decompiler (Fernflower/IntelliJ) renders the 7-parameter `store()` method as:

```java
String store(CbrCase var1, String var2, String var3, MemoryDomain var4, String var5, String var6, Path var7);
```

The natural guess for parameter ordering is `(cbrCase, tenantId, entityId, domain, sourceId, scope, path)` — but the actual ordering is `(cbrCase, tenantId, entityId, domain, sourceId, caseType, path)`. The 6th parameter is `caseType`, not a scope string.

### What was tried (didn't work)
- Passing `"game"` as var6 (intended as scope label) — store succeeded, retrieve returned empty
- Trying `RetrievalMode.FEATURE_ONLY` and `HYBRID` — both empty
- Relaxing `minSimilarity` to 0.0 — still empty
- Matching all four feature fields exactly — still empty
- Matching scope `Path` between store and query — still empty

### Root cause
The `InMemoryCbrCaseMemoryStore` stores cases keyed by `caseType` (var6). When var6 is `"game"` but the query specifies `caseType = "sc2-strategy"`, the stored case is invisible to retrieval. The store operation itself doesn't validate that `caseType` matches `CbrCase.cbrType()` — it accepts any string.

### Fix
Pass the `CbrCase.cbrType()` value (or its constant) as the 6th parameter:

```java
// Wrong — "game" is a scope label, not a caseType
cbrStore.store(cbrCase, tenantId, entityId, domain, "source-id", "game", path);

// Correct — 6th param is caseType
cbrStore.store(cbrCase, tenantId, entityId, domain, "source-id", SC2GameCbrCase.CBR_TYPE, path);
```

The actual parameter ordering is: `(CbrCase, tenantId, entityId, MemoryDomain, sourceId, caseType, Path)`.

### Why this is non-obvious
The decompiled interface gives no parameter names — `var1` through `var7`. Two consecutive `String` parameters (var5=sourceId, var6=caseType) are easy to confuse with (var5=sourceId, var6=scope). The operation succeeds silently with the wrong value because `store()` doesn't cross-check var6 against `CbrCase.cbrType()`. The only symptom is empty retrieval, which looks like a query problem rather than a storage problem.

### Alternatives considered
- Using the source jar for parameter names — not available in the CaseHub SNAPSHOT dependency chain
- Adding runtime validation in `store()` to check `caseType == cbrCase.cbrType()` — requires a foundation change (filed as context on neocortex#162)
