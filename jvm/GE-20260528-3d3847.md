## Symptom

`@BeforeAll @Transactional static void seedData()` appears to work but occasionally throws `ContextNotActive` at runtime, or the `@Transactional` annotation is silently ignored:

```
io.quarkus.arc.ArcUndeclaredThrowableException: Error invoking subclass method
    Caused by: jakarta.enterprise.context.ContextNotActiveException
```

## Context

CDI interceptors (including `@Transactional`) use proxy subclassing to intercept method calls. Static methods cannot be overridden by a subclass — CDI proxies have no mechanism to intercept them. The annotation is present but has no effect.

Panache active record operations (`Entity.find(...)`, `entity.persist()`) require an active transaction. Without CDI-managed transaction boundaries, they fail when called from a static context.

## Root cause

CDI 3.0 (Jakarta EE 9) spec §7.2: "An interceptor is not invoked for a private, static, or final method." Static methods are explicitly excluded from CDI interception.

## Fix

Convert static seeding methods to instance methods using `@TestInstance(PER_CLASS)` or `@BeforeEach`:

```java
// Wrong — @Transactional is silently ignored:
@BeforeAll
@Transactional
static void seedData() {
    MyEntity entity = new MyEntity();
    entity.persist();  // fails — no active transaction
}

// Correct — @BeforeEach instance method, CDI intercepted:
@BeforeEach
@Transactional
void seedData() {
    if (MyEntity.count() == 0) {
        MyEntity entity = new MyEntity();
        entity.persist();
    }
}
```

Or use `@TestInstance(TestInstance.Lifecycle.PER_CLASS)` to allow `@BeforeAll` on instance methods:

```java
@QuarkusTest
@TestInstance(TestInstance.Lifecycle.PER_CLASS)
class MyTest {
    @BeforeAll
    @Transactional
    void seedData() {  // instance method — CDI interception works
        ...
    }
}
```

## Why this is non-obvious

The annotation compiles, the test runs, and if Panache's autocommit behaviour happens to work for the specific operation, tests may pass intermittently or in certain environments. The failure only surfaces when the transaction boundary is actually required (e.g. Panache `merge()`, FK constraints). The spec exclusion is documented but not surfaced as a compiler warning.
