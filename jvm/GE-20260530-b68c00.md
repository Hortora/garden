### Symptom

After a GitHub Packages authentication failure (HTTP 401), subsequent Maven resolution attempts continue to fail with `was not found in <repo> during a previous attempt. This failure was cached in the local repository` — even after providing correct credentials or setting `GITHUB_TOKEN` correctly.

### Context

Occurs when:
1. Maven attempts to resolve a SNAPSHOT artifact from GitHub Packages and gets a 401
2. Maven writes a `*.lastUpdated` failure marker to `~/.m2/repository/<group>/<artifact>/<version>/`
3. Auth is fixed (env var set, token refreshed, etc.)
4. Maven is run again — it reads the marker and refuses to retry within the update interval

The repository entry in `~/.m2` looks like:
```
casehub-qhorus-runtime-0.2-SNAPSHOT.pom.lastUpdated
casehub-qhorus-runtime-0.2-SNAPSHOT.jar.lastUpdated
resolver-status.properties
```

### Root cause

Maven's negative result caching: after a failed remote resolution, Maven writes `*.lastUpdated` files to mark that it tried and failed. By default it will not retry until the configured `updatePolicy` interval elapses (or `-U` is passed). The cached failure is independent of the auth state — fixing credentials does not invalidate the negative cache.

The `-U` flag (`--update-snapshots`) overrides the interval but does NOT ignore `*.lastUpdated` files when they record a specific artifact-not-found condition (as opposed to just a metadata staleness check). In practice `-U` helps for `updatePolicy`-expired entries but not for entries recorded with a specific `artifact was not found` message.

### Fix

Delete the `*.lastUpdated` files for the affected artifacts before retrying:

```python
import pathlib

# Delete all .lastUpdated markers under the affected group
for f in pathlib.Path.home().joinpath('.m2/repository/io/casehub').rglob('*.lastUpdated'):
    f.unlink()
```

Or for a broader sweep of all casehub artifacts:
```python
import pathlib

count = sum(1 for f in pathlib.Path.home().joinpath('.m2/repository/io/casehub').rglob('*.lastUpdated') if f.unlink() is None)
print(f"Cleared {count} stale .lastUpdated files")
```

After deletion, the next Maven invocation will attempt remote resolution again.

### Why this is non-obvious

Two layers of confusion:
1. **The error message blames the artifact, not the auth** — `was not found during a previous attempt` sounds like the artifact doesn't exist, not like an auth problem was cached.
2. **Fixing the credential is insufficient** — developers naturally expect that correcting the auth (setting `GITHUB_TOKEN`, refreshing `gh auth`, etc.) should be enough. The negative cache is a separate system that doesn't observe credential changes.

The `resolver-status.properties` file in the version directory contains the last resolution result, but nothing in the Maven console output points you there.
