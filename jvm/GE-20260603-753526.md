### Symptom
Every health/legal WorkItem creation with an ExternalActor silently writes a FLAGGED trust attestation, degrading the actor's trust score before any work is done.

### Context
Attestation writer maps `LifeDecisionEventType` (3 values: CREATED, SLA_BREACH, COMPLETED) to `AttestationVerdict`:
```java
var verdict = eventType == COMPLETED ? SOUND : FLAGGED;
```
The code reads as a correct boolean — but the enum has 3 values, not 2. CREATED is neither a positive nor negative outcome; it should produce no attestation at all.

### Root cause
Binary if/else on an enum with >2 values silently funnels all non-primary values into the else branch. The pattern looks correct at a glance because the two named outcomes (COMPLETED → SOUND, SLA_BREACH → FLAGGED) are the only ones the developer was thinking about.

### Fix
Guard non-outcome events with an early return before the verdict mapping:
```java
if (eventType == LifeDecisionEventType.CREATED) {
    return;
}
```

Or use an explicit switch (Java's exhaustive switch catches missing cases at compile time):
```java
var verdict = switch (eventType) {
    case COMPLETED -> AttestationVerdict.SOUND;
    case SLA_BREACH -> AttestationVerdict.FLAGGED;
    case CREATED -> { return; }
};
```

### Why this is non-obvious
The binary pattern `x == A ? resultA : resultB` is idiomatic for booleans and 2-value enums. When a third enum value is added later, the else branch silently absorbs it without any compilation warning. The data corruption (wrong trust scores) manifests far from the code change — in the nightly TrustScoreJob output, not at the attestation write site.

### Why this fix
Early return is simpler than refactoring to switch and avoids touching the existing verdict mapping. The switch alternative is more future-proof but changes more code.

### Alternatives considered
- Switch expression: more compile-safe but requires restructuring the method to handle the "no attestation" case as a return inside a switch arm
- Explicit Set of outcome types: `Set.of(COMPLETED, SLA_BREACH).contains(eventType)` — adds a collection allocation for a 2-element check
