### What went wrong

A routing strategy with a bootstrap guard — "if all candidates are BOOTSTRAP, escalate" — has a silent correctness gap when the pool contains a mix of BOOTSTRAP and BORDERLINE candidates.

**Pool:** `[BOOTSTRAP (workload=1.0), BORDERLINE (score=0.0)]`

The guard condition (`allMatch(Phase.BOOTSTRAP)`) is false because BORDERLINE is present, so the guard does not fire. Scoring assigns: BOOTSTRAP gets `workloadScore = 1.0`, BORDERLINE gets `0.0`. The `decide()` outcome check returns `Assigned(bootstrapAgent)` — because `best.finalScore() > 0.0`.

The existing borderline-escalation path (`anyBorderline && all-zero-scored`) also fails to fire because BOOTSTRAP has a positive score — it prevents the all-zero condition.

Result: an unproven BOOTSTRAP agent is assigned to a high-stakes capability with no escalation.

### Root cause

The guard condition `allMatch(BOOTSTRAP)` tests whether every candidate is BOOTSTRAP. It does not protect against the case where BOOTSTRAP is the highest-scoring candidate because all other candidates score 0.0 (borderline, excluded). The symptom is invisible in the common case (all-BOOTSTRAP pool) but silently wrong in the mixed case.

### Fix

Replace the single `allMatch(BOOTSTRAP)` guard with a two-part approach:

1. **Pre-screen** (fires before scoring): `!hasQualified && hasBootstrap` → escalate with `NO_QUALIFIED_AGENT`. This catches all cases where BOOTSTRAP would win — pure BOOTSTRAP pool, BOOTSTRAP+BORDERLINE, BOOTSTRAP+EXCLUDED.

2. **Stripping**: if QUALIFIED exists, remove BOOTSTRAP from the eligible scoring pool. The scoring loop and `decide()` operate on the stripped list. Busy QUALIFIED beats idle BOOTSTRAP.

```java
// Pre-screen — before scoring
if (policy.bootstrapEscalationRequired()) {
    boolean hasQualified = classified.stream().anyMatch(c -> c.phase() == Phase.QUALIFIED);
    boolean hasBootstrap = classified.stream().anyMatch(c -> c.phase() == Phase.BOOTSTRAP);
    if (!hasQualified && hasBootstrap) {
        return escalate(capabilityName, EscalationReason.NO_QUALIFIED_AGENT);
    }
}

// Strip before scoring — reached only if QUALIFIED exists
List<ClassifiedCandidate> eligible = policy.bootstrapEscalationRequired()
    ? classified.stream().filter(c -> c.phase() != Phase.BOOTSTRAP).toList()
    : classified;
// scoring loop uses eligible, not classified
```

### Why this is non-obvious

The gap looks like it's covered by two independent safety nets (the bootstrap guard AND the borderline-escalation path in `decide()`), but both fail silently in the mixed case. The bootstrap guard's `allMatch` is false, and the borderline escalation's all-zero precondition is prevented by BOOTSTRAP's positive workload score. A designer reviewing either guard in isolation would consider it sufficient.

### Pool coverage

| Pool | Old result | New result |
|------|-----------|-----------|
| All-BOOTSTRAP | Guard fires | Pre-screen fires |
| BOOTSTRAP + BORDERLINE | **BOOTSTRAP assigned (wrong)** | Pre-screen fires → escalate |
| BOOTSTRAP + EXCLUDED | **BOOTSTRAP assigned (wrong)** | Pre-screen fires → escalate |
| BOOTSTRAP + QUALIFIED | BOOTSTRAP may win by workload | BOOTSTRAP stripped; QUALIFIED wins |
| BORDERLINE only | Borderline escalation | Unchanged (pre-screen needs BOOTSTRAP) |

### Constraints

Applies to any trust routing system with multiple candidate phases where BOOTSTRAP carries a positive availability score and other phases score 0.0.

### Invalidation triggers

Revisit if the scoring model is changed so BOOTSTRAP candidates no longer receive positive scores.
