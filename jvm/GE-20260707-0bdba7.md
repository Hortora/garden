### Symptom

Tests using `@EnableKubernetesMockClient(crud = true)` fail when calling `client.resource(resource).fieldManager("name").forceConflicts().serverSideApply()` — the mock server doesn't support the SSA API endpoint.

### Context

fabric8's CRUD mock server simulates basic Kubernetes API operations (create, get, update, delete, list, watch) but does not implement the server-side apply endpoint (`PATCH` with `application/apply-patch+yaml` content type). Tests that use SSA will fail even though the code is correct against a real cluster.

### Fix

Use `createOr(NonDeletingOperation::update)` as the primary apply path in code that needs to be tested with the CRUD mock:

```java
// Works with CRUD mock AND real clusters
client.resource(toResource(spec))
    .createOr(NonDeletingOperation::update);

// Production-preferred (uncomment when targeting real clusters):
// client.resource(toResource(spec))
//     .fieldManager("casehub-ops")
//     .forceConflicts()
//     .serverSideApply();
```

`createOrReplace()` is deprecated in fabric8 7.x — `createOr(NonDeletingOperation::update)` is the recommended replacement. Both are idempotent (create if absent, update if present).

### Why this is non-obvious

SSA is the recommended apply strategy for Kubernetes controllers and is explicitly supported by fabric8's production client. A developer would naturally reach for SSA in tests too. The CRUD mock's lack of SSA support means a different apply path is needed for tests — this isn't documented in fabric8's mock server documentation.

### Alternatives considered

- Using expectation-based mock instead of CRUD mode — works but requires manually programming every API response, significantly more test code
- Detecting mock vs real cluster at runtime and switching strategy — adds unnecessary complexity to production code for a test concern
