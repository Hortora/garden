## Symptom

Compiler error when wrapping a REST client Response in try-with-resources:

```
incompatible types: Response cannot be converted to AutoCloseable
```

## Context

Using `jakarta.ws.rs.core.Response` as a return type from a `@RegisterRestClient`
method and attempting to auto-close it to avoid HTTP connection pool leaks:

```java
try (Response response = gatewayClient.invokeAgent(request)) {  // ‚Üê compile error
    ...
}
```

## Root cause

`jakarta.ws.rs.core.Response` declares `close()` (added in JAX-RS 2.0) but does **not**
implement `java.lang.AutoCloseable`. The concrete Quarkus/RESTEasy implementation
(`ClientResponse`) does implement `Closeable`, but the declared return type from a
`@RegisterRestClient` method is the abstract `Response` ‚Äî so the compiler rejects the
try-with-resources block.

## Fix

Use `try-finally` with explicit `response.close()`:

```java
Response response = gatewayClient.invokeAgent(request);
try {
    // use response
} finally {
    response.close();
}
```

`response.close()` is safe to call ‚Äî it is declared on the abstract `Response` class.
The method releases the connection back to the HTTP connection pool.

## Why this is non-obvious

`close()` is declared on `Response` since JAX-RS 2.0, which strongly implies try-with-resources
should work ‚Äî that is exactly why `AutoCloseable` exists. Nearly every Java resource type that
declares `close()` implements `AutoCloseable`. The gap is intentional (the JAX-RS spec authors
chose not to add `AutoCloseable` to `Response`) but invisible from the API surface. A developer
who sees `close()` naturally reaches for try-with-resources and gets a compile error with no
hint about why.
