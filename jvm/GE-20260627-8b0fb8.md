### What exists

`PrefetchQuery.Builder.setParams(SearchParams)` accepts per-prefetch-leg search parameters, including `QuantizationSearchParams` with `oversampling` and `rescore` fields.

In an RRF hybrid query (dense + sparse prefetch legs fused via `QueryFactory.rrf()`), this allows setting oversampling on the dense prefetch leg only — leaving the sparse leg unaffected.

### How to use it

```java
PrefetchQuery.Builder densePrefetch = PrefetchQuery.newBuilder()
    .setQuery(QueryFactory.nearest(denseEmbedding.vectorAsList()))
    .setUsing("dense")
    .setLimit(40)
    .setParams(SearchParams.newBuilder()
        .setQuantization(QuantizationSearchParams.newBuilder()
            .setOversampling(2.0)
            .setRescore(true)
            .build())
        .build());

PrefetchQuery.Builder sparsePrefetch = PrefetchQuery.newBuilder()
    .setQuery(QueryFactory.nearest(sparseValues, sparseIndices))
    .setUsing("sparse")
    .setLimit(40);
    // no SearchParams — sparse vectors are not quantized

QueryPoints queryPoints = QueryPoints.newBuilder()
    .addPrefetch(densePrefetch)
    .addPrefetch(sparsePrefetch)
    .setQuery(QueryFactory.rrf(Rrf.newBuilder().setK(60).build()))
    .setLimit(10)
    .build();
```

### Why this is non-obvious

Qdrant's documentation for hybrid search focuses on the top-level `QueryPoints.params` field. The per-prefetch `setParams()` method exists in the Java gRPC API but is not prominently documented. Without it, oversampling applied at the top level would affect both dense and sparse legs — wasteful for sparse vectors which are not quantized.

### Context

This is relevant when combining Qdrant's server-side binary or scalar quantization with RRF hybrid search. Binary quantization on dense vectors benefits from 2-3x oversampling to compensate for the lossy hamming distance approximation. Sparse vectors (SPLADE) are not quantized and don't need oversampling.
