### Context

A `@Tool` method in an `@ApplicationScoped` CDI bean needs to call `CommitmentStore.save()`
directly — bypassing the usual pattern where other tool methods delegate to already-`@Transactional`
service methods. The question is whether `@Transactional` on the `@Tool` method itself is safe,
or whether it conflicts with the MCP framework.

### What you'd normally do

Extract a separate `@ApplicationScoped @Transactional` helper bean to own the store mutation,
following the pattern set by `OversightGateDispatcher`. This is verbose but conservative.

### What actually works

`@Transactional` directly on the `@Tool` method is correct and safe.

Quarkus ARC handles `@Transactional` as a CDI interceptor at the **subclass proxy** level.
`@Tool` (from `io.quarkiverse.mcp.server`) is a **marker/discovery annotation** — it registers
the method with the MCP framework for reflection-based invocation but does not generate its
own CDI proxy, does not wrap the method with an interceptor, and does not bypass the CDI
subclass. When the MCP framework invokes the method via CDI, the `@Transactional` interceptor
fires normally in the proxy chain.

```java
@ApplicationScoped
public class CommitmentTools {

    @Transactional   // ← safe here — ARC handles this as a CDI interceptor
    @Tool(description = "Block a commitment and extend its Watchdog deadline...")
    public ToolResponse block(
            @ToolArg(description = "Your agentId") String agentId,
            @ToolArg(description = "commitmentId") String commitmentId,
            @ToolArg(description = "Reason") String reason,
            @ToolArg(description = "ISO-8601 future timestamp") String blockedUntil) {

        // commitmentStore.save() runs inside the @Transactional boundary
        commitment.expiresAt = newDeadline;
        commitmentStore.save(commitment);

        // messageService.dispatch() joins the same JTA transaction
        messageService.dispatch(...);

        return ToolResponse.success(...);
    }
}
```

### Why this is non-obvious

The `@Transactional + @Tool` combination looks risky because:
1. Some annotation frameworks do generate proxies that intercept CDI calls — a developer
   might worry `@Tool` does the same and causes a double-proxy or ordering conflict.
2. The quarkiverse MCP documentation does not explicitly state that `@Tool` is a pure
   marker annotation with no ARC impact.
3. All other tool methods in the project delegate to service methods for transaction
   demarcation, making direct annotation on the tool method look like a deviation.

The safe check: does the annotation from the third-party framework register an ARC bean
or interceptor? If no — it cannot conflict with `@Transactional`. As of quarkus-mcp-server
1.11.1, `@Tool` registers methods via a Quarkus build-time recorder, not via CDI.

### The critical safety condition

`@Transactional` directly on a `@Tool` method is safe **only** if the method propagates
exceptions normally (no try/catch). If the method catches an exception (for fail-open
semantics), ARC marks the transaction **rollback-only** — the next commit attempt will
throw `jakarta.transaction.RollbackException` even though the exception was swallowed.

The `OversightGateDispatcher` pattern (a separate `@Transactional @ApplicationScoped`
bean invoked by the tool method) is the correct solution for tool methods with internal
catch-all blocks.
