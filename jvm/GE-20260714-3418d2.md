## QhorusDashboardService injects @Vetoed reactive services — CDI deployment fails with reactive=false

### Symptom

Quarkus CDI deployment fails with 4 `UnsatisfiedResolutionException` errors for `ReactiveChannelService`, `ReactiveInstanceService`, `ReactiveMessageService`, and `ReactiveMessageStore` — all injected by `QhorusDashboardService`.

### Context

With `quarkus.datasource.reactive=false`, qhorus correctly `@Vetoed` its reactive service implementations (they need a reactive datasource). But `QhorusDashboardService` unconditionally injects these types without guarding against their absence.

### Root cause

`QhorusDashboardService` uses direct `@Inject` for reactive services instead of `Instance<>` with `isResolvable()` guards. The ledger module (ledger#92) already fixed the same pattern for `LedgerVerificationService` — qhorus hasn't applied the same fix to its dashboard service.

### Fix

Exclude `QhorusDashboardService` from CDI in all profiles where `reactive=false`:

```properties
%dev.quarkus.arc.exclude-types=...,io.casehub.qhorus.runtime.dashboard.QhorusDashboardService
%prod.quarkus.arc.exclude-types=...,io.casehub.qhorus.runtime.dashboard.QhorusDashboardService
# Also in test application.properties
```

### Why this is non-obvious

The error message names the reactive services as unsatisfied but doesn't mention `QhorusDashboardService` as the injection source without reading the full stack trace. The `@Vetoed` annotation on the reactive services is correct — the bug is in the consumer, not the provider.
