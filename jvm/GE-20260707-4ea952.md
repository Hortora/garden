## AgentProvider.openSession() for persistent HyDE — 5x faster than per-call invoke(), must serialize access

### Context
HyDE (Hypothetical Document Embeddings) query expansion requires an LLM call for each search query — generating a 3-5 sentence hypothetical document that gets embedded instead of the raw query. The obvious path uses `AgentProviderChatModel` (the `ChatModel` bridge), which calls `agentProvider.invoke()` per query. Each `invoke()` spawns a new Claude CLI subprocess.

### What was tried (didn't work)
`AgentProviderChatModel.doChat()` → `agentProvider.invoke()`: ~50 seconds per query. Each call creates a new `ClaudeAsyncClient`, spawns a subprocess, waits for JVM startup, and gets no prompt caching benefit. For a 28-query benchmark, total time would be ~23 minutes.

### The technique
Use `AgentProvider.openSession(AgentSessionInit.of(systemPrompt))` to create a persistent `AgentSession`. The session keeps one Claude process warm across all queries:

```java
@ApplicationScoped
public class SessionQueryExpander implements QueryExpander {
    private final Object sessionLock = new Object();
    private volatile AgentSession session;

    public List<RetrievalQuery> expand(RetrievalQuery query) {
        synchronized (sessionLock) {
            AgentSession s = getOrCreateSession();
            String hypothetical = s.query(userPrompt)
                .filter(e -> e instanceof AgentEvent.TextDelta)
                .map(e -> ((AgentEvent.TextDelta) e).text())
                .collect().with(Collectors.joining())
                .await().atMost(Duration.ofMinutes(2));
            return List.of(query.withExpansion(hypothetical));
        }
    }
}
```

Results: ~10 seconds per query (vs ~50s with invoke()). The system-prompt is cached by the SDK across calls, and no subprocess startup overhead after the first query.

### Critical constraint — serialize access
The Claude CLI subprocess backing `AgentSession` does not support concurrent `query()` calls. Without synchronization, concurrent HTTP requests cause timeouts and empty responses. The `synchronized (sessionLock)` block is mandatory.

### Failure recovery
On any exception, reset the session (close + nullify) so the next call opens a fresh one:
```java
catch (Exception e) {
    resetSession();  // close broken session, next call opens new one
    return List.of(query);  // fallback to original query
}
```

### Why this is non-obvious
The `ChatModel` abstraction (`AgentProviderChatModel`) is the documented path for LLM access in the platform. It hides `AgentProvider.openSession()` behind a synchronous interface that creates a new subprocess per call. The session-based approach requires understanding the platform's internal layering and the Claude Agent SDK's session model — neither is documented as the preferred path for repeated short queries.
