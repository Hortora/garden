### Symptom
Code review catches two implementations of the same SPI throwing different exception types for `null` argument: one throws `NullPointerException`, the other `IllegalArgumentException`.

### Context
An SPI interface (`InferenceModel.runBatch()`) documents `@throws IllegalArgumentException if inputs is null`. One implementation uses `Objects.requireNonNull(inputs, "...")` — which throws `NullPointerException`, not `IllegalArgumentException`. The other uses `if (inputs == null) throw new IllegalArgumentException(...)`. Tests pass against both individually but consumers see inconsistent behaviour.

### Root cause
`Objects.requireNonNull` always throws `NullPointerException`. It is designed for precondition checks where NPE is the correct response (e.g. constructor parameters per Effective Java Item 49). When an SPI contract specifies `IllegalArgumentException` for null arguments, `Objects.requireNonNull` silently violates the contract.

### Fix
Use explicit null checks when the SPI contract specifies `IllegalArgumentException`:

```java
// Wrong — throws NPE, violating the IAE contract
Objects.requireNonNull(inputs, "inputs must not be null");

// Right — throws IAE, matching the contract
if (inputs == null) throw new IllegalArgumentException("inputs must not be null");
```

Reserve `Objects.requireNonNull` for cases where NPE is the intended exception (constructor field assignment, internal preconditions).

### Why this is non-obvious
`Objects.requireNonNull` is the reflexive choice for any null check in Java. It reads well, is concise, and is widely recommended. The distinction between NPE and IAE is subtle and invisible until a consumer tests exception types — which most unit tests don't do (they test for any exception, not the specific type).
