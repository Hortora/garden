## Symptom

Code uses `ConcurrentHashMap` as a thread-safe cache. One thread does `cache.put(id, obj)` then later mutates `obj.field`. A second thread calls `cache.get(id)` — which is defined to happen-after the `put()` — then reads `obj.field` and sees a stale value. Less obviously: a third thread that obtained the reference via an earlier `cache.get()` (or via any other path before the `put()`) also sees a stale `obj.field` even though the `put()` semantically "published" the object.

In this session: `CaseInstance.state` changes were invisible to threads that had obtained a reference to the `CaseInstance` before the state update was published via `cache.put()`.

## Root Cause

The JMM happens-before guarantee from `ConcurrentHashMap.put(k, v)` is: "the state seen by the mapping function or the put() call is visible to any thread that subsequently calls `get(k)` and observes the new value." This guarantee is established at the moment `get()` returns the post-put value.

It covers:
- The value `v` itself (reference visibility)
- Fields written to `v` **before** the `put()` call (publication via the put's write barrier)

It does NOT cover:
- Fields written to `v` **after** the `put()` call
- Any code path that obtained the reference to `v` before this particular `put()` — those threads have no synchronisation barrier between their earlier `get()` and the subsequent field mutation

If a thread modifies `obj.field` after `cache.put()`, that write is only guaranteed visible to threads that call `cache.get()` in a future operation. Threads that already hold a reference — whether from before the put, or from the put itself — see whatever their CPU cache contains.

## Fix

**Option A — Publish immutable state:** Use an immutable value type. Replace `obj.field = newValue` with `cache.put(id, new Obj(newValue))`. Every reader calls `get()` to see current state, establishing the happens-before at each read.

**Option B — `volatile` field:** Declare `volatile State field` on the mutable object. `volatile` writes are visible to all subsequent reads regardless of how the reference was obtained.

**Option C — Use the cache as the memory barrier:** Force all readers to call `cache.get()` for every field read — never hold the reference across mutations. Practical only if the cache is small and reads are infrequent.

In the session: `CaseInstance.state` was made `volatile` so changes published via the engine's own state machine were visible to the `CaseInstanceCache.get()` completion check without requiring a re-put.

## Why this is non-obvious

"ConcurrentHashMap is thread-safe" is true but misleadingly coarse. The thread-safety guarantee is for the map structure (no corruption, consistent iteration). The happens-before guarantee is specifically for `get()` observing a `put()` — it does not extend to field-level visibility on the value object after it has been placed in the map. Developers who know that `volatile` is needed for shared fields often miss that a ConcurrentHashMap lookup alone does not establish a barrier for code holding a pre-obtained reference.
