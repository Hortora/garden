## Symptom

You try to make a single in-memory test store implement both a blocking interface (`ChannelStore`) and a reactive interface (`ReactiveChannelStore`). Both have a method `put(Channel channel)` — one returns `Channel`, the other returns `Uni<Channel>`. Java rejects the class with a compile error: you cannot have two methods with the same name and parameter types but different return types.

## Context

Quarkus reactive patterns often pair a blocking interface (e.g. `ChannelStore`) and a reactive interface (`ReactiveChannelStore`), both with matching method names. In `@QuarkusTest`, you want `@BeforeEach channelStore.put(channel)` (blocking) to be visible to reactive service code that injects `ReactiveChannelStore`. The intuition is to make one class implement both — but Java's method signature rules make this impossible.

```java
// Blocking interface
interface ChannelStore {
    Channel put(Channel channel); // returns Channel
}

// Reactive interface
interface ReactiveChannelStore {
    Uni<Channel> put(Channel channel); // same signature, different return type
}

// This won't compile — irreconcilable return types
class InMemoryChannelStore implements ChannelStore, ReactiveChannelStore { ... }
```

## Root cause

Java method overloading is defined by name + parameter types only; return type is not part of the signature for overload resolution. Two methods with identical name+parameters but different return types are a compile error. `Channel` and `Uni<Channel>` are not in a covariant relationship, so even covariant return type rules don't help.

## Fix

Create a separate `InMemoryReactiveChannelStore` class that implements `ReactiveChannelStore` and injects/delegates to `InMemoryChannelStore`:

```java
@Alternative @Priority(1) @ApplicationScoped
public class InMemoryReactiveChannelStore implements ReactiveChannelStore {

    @Inject
    InMemoryChannelStore blocking; // CDI injects the singleton — shared backing store

    @Override
    public Uni<Channel> put(Channel channel) {
        return Uni.createFrom().item(() -> blocking.put(channel));
    }

    @Override
    public Uni<Optional<Channel>> findByName(String name) {
        return Uni.createFrom().item(() -> blocking.findByName(name));
    }
    // ... delegate all reactive methods to blocking counterpart
}
```

`@BeforeEach channelStore.put(channel)` writes to `InMemoryChannelStore`. CDI injects the same singleton into `InMemoryReactiveChannelStore.blocking`, so reactive service reads see the same data. `@AfterEach channelStore.clear()` clears both views.

## Why this is non-obvious

The natural first instinct is to add `implements ReactiveChannelStore` to the existing `InMemoryChannelStore`. The compile error is straightforward once you see it, but the delegate pattern (a second class that injects and wraps the first) is not an obvious next step — especially with CDI involved, where singleton scope ensures the shared backing store is the same instance.
