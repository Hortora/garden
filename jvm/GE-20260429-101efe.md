## What

`UNIQUE NULLS NOT DISTINCT (col1, col2, nullable_col)` works correctly in H2 2.4.240 (the version pulled by Quarkus 3.32.2 via `quarkus-jdbc-h2`). Two rows with `NULL` in `nullable_col` correctly violate the constraint — NULLs are treated as equal. No sentinel value (`''`, `-1`) needed.

## Context

When building a discriminator-keyed table with a nullable `scope_key` column (`NULL` for GLOBAL rows, tag string for CAPABILITY rows), a `UNIQUE` constraint alone won't prevent duplicate GLOBAL rows because SQL treats `NULL != NULL`. The standard solutions are: sentinel value, partial index, or `NULLS NOT DISTINCT`.

## Discovery

Not documented in the H2 official docs. Only discoverable via the H2 GitHub issue tracker: issue #4036 documented the `NULLS NOT DISTINCT` bug (constraint silently lost after `DROP COLUMN`) and its fix in 2.3.230. The feature itself has been present since H2 2.2.x but was unreliable before the fix.

## Usage

```sql
CONSTRAINT uq_my_table_key UNIQUE NULLS NOT DISTINCT (actor_id, score_type, scope_key)
```

Works identically in PostgreSQL 15+ and H2 2.4.240 — no conditional DDL needed. Test with H2 `MODE=PostgreSQL` in the JDBC URL.

## Why This Is Non-Obvious

The natural workaround is a sentinel empty string (`''`) or `COALESCE` in a functional index. Checking the actual H2 version (Quarkus BOM → `quarkus-jdbc-h2` → H2 2.4.240) reveals the feature works correctly and the sentinel is unnecessary.
