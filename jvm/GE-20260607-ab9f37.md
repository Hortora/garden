### Symptom

```
org.mockito.exceptions.misusing.UnnecessaryStubbingException:
Unnecessary stubbings detected.
  1. -> at com.example.MyServiceTest.setUp(MyServiceTest.java:30)
```

Tests fail even though the stubs are correct. The stub is needed by some tests but not by others in the same class.

### Context

`@ExtendWith(MockitoExtension.class)` enables strict stubbing by default in Mockito 5. A `@BeforeEach` method stubs all code paths for the class under test (e.g. all preference keys a classifier might read). Individual `@Test` methods exercise only a subset of those paths — tests for `ALWAYS`-gated types never trigger threshold key reads; `NEVER`-gated tests skip the preference provider entirely.

This is a distinct variant from the re-stubbing case (`@BeforeEach` stubs a method and then one test stubs the same method again — different fix).

### Fix

Mark all shared setup stubs as `lenient()`. Lenient stubs satisfy injection but are never flagged as unnecessary even when unconsumed:

```java
@ExtendWith(MockitoExtension.class)
class LifeActionRiskClassifierTest {

    @Mock PreferenceProvider preferenceProvider;
    @InjectMocks LifeActionRiskClassifier classifier;

    @BeforeEach
    void setUp() {
        Preferences prefs = mock(Preferences.class);

        // lenient() — not all tests exercise all threshold keys
        lenient().when(preferenceProvider.resolve(any(SettingsScope.class))).thenReturn(prefs);
        lenient().when(prefs.get(LifeRiskPolicyKeys.SPEND_THRESHOLD)).thenReturn(DoublePreference.of(100.0));
        lenient().when(prefs.get(LifeRiskPolicyKeys.CONTRACTOR_THRESHOLD)).thenReturn(DoublePreference.of(200.0));
        lenient().when(prefs.get(LifeRiskPolicyKeys.APPROVAL_EXPIRES_HOURS)).thenReturn(DoublePreference.of(24.0));
    }
}
```

### Why this is non-obvious

The error message says "remove these stubbings" — which breaks the tests that DO need them. `lenient()` is the canonical fix but is not suggested in the error output. Common wrong fixes: switching to `@MockitoSettings(strictness = LENIENT)` on the class (too broad — disables strictness globally, hiding real problems) or moving stubs into each test (duplication). `lenient()` is surgical: it suppresses the unnecessary-stub check for exactly the stubs that are shared, while preserving strict checking for everything else.
