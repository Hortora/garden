`DocumentSplitters.recursive(int, int, TokenCountEstimator)` requires a `TokenCountEstimator` as the third argument for token-based segment sizing. `EmbeddingModel` does NOT implement `TokenCountEstimator` — they are separate interface hierarchies with no inheritance relationship.

### Symptom

Design spec or implementation assumes `EmbeddingModel` can be passed as the `TokenCountEstimator` argument to `DocumentSplitters.recursive()`. Compilation fails with "incompatible types."

### Context

When building a RAG chunking pipeline, the natural assumption is: "I already have an `EmbeddingModel` that handles tokenization for embeddings — it should work as a `TokenCountEstimator` for splitting too." The naming and domain proximity reinforce this assumption.

### Root cause

`EmbeddingModel` and `TokenCountEstimator` are independent interfaces in LangChain4j. `EmbeddingModel` knows how to embed text into vectors. `TokenCountEstimator` knows how to count tokens in text. Some LLM provider implementations (e.g. `OpenAiEmbeddingModel`) implement both, but the base `EmbeddingModel` interface does not extend `TokenCountEstimator`. ONNX-based models (`OnnxEmbeddingModel`) implement only `EmbeddingModel`.

### Fix

Use the character-based 2-arg overload:

```java
DocumentSplitter splitter = DocumentSplitters.recursive(1000, 200);
```

This splits by character count (paragraph → line → sentence → word fallback chain). Character counts are approximate by nature — the imprecision vs token counts is negligible for RAG chunk sizing.

### Why this is non-obvious

The `EmbeddingModel` interface handles tokenization internally (it must tokenize text before embedding it). A developer reasonably infers it exposes this capability via `TokenCountEstimator`. The class names reinforce the assumption: "Embedding" + "Model" + "Token" + "Count" — all in the same domain. The actual type hierarchy has no inheritance link between them.

### Alternatives considered

- Implementing a custom `TokenCountEstimator` that delegates to the embedding model's internal tokenizer — rejected because ONNX embedding models don't expose their tokenizer publicly
- Using HuggingFace Tokenizers JNI directly as a `TokenCountEstimator` — possible but adds JNI complexity for marginal accuracy improvement over character-based splitting
