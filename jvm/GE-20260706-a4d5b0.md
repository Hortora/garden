### Symptom
CDI `@Decorator` registers successfully at build time — `BuildTimeEnabledProcessor` DEBUG output shows "Enabling" for the decorator class. But the decorator constructor never fires at runtime. No error, no warning. Benchmark results identical with and without the decorator dependency.

### Context
A `CaseRetriever` implementation (`HybridCaseRetriever`) was produced via a `@Produces @ApplicationScoped` method in a producer class. A `@Decorator` in an external JAR (`QueryExpandingCaseRetriever`) wrapped the `CaseRetriever` interface. The decorator was correctly discovered, `@IfBuildProperty` evaluated, and the bean registered — but interception never happened.

In `@QuarkusTest`, an `@Alternative @Priority(1)` managed bean (`InMemoryCaseRetriever`) was the `CaseRetriever` — and the decorator worked perfectly. This masked the production failure for weeks.

### What was tried (didn't work)
- Adding `@Unremovable` to the decorator — no effect
- Adding FINE-level logging to the decorator constructor — confirmed it never fires
- Verifying `@IfBuildProperty` evaluation via `BuildTimeEnabledProcessor` DEBUG — condition passes
- Checking `quarkus.index-dependency` configuration — beans discovered correctly
- Adding `beans.xml` — not needed, Jandex index already present
- Searching for Quarkus issues with `@Decorator` + `@IfBuildProperty` + external JARs — red herring

### Root cause
Arc applies decorators by **generating subclasses** of managed beans. The generated subclass overrides methods to route through the decorator chain. For `@Produces` method beans, the instance is already created by the producer — Arc wraps it in a client proxy but **cannot subclass the concrete instance**. The decorator is registered in the bean graph but never inserted into the interception chain.

### Fix
Convert the produced type from a `@Produces` method bean to a managed `@ApplicationScoped` class with an `@Inject` constructor. Keep the original package-private constructor for direct construction in unit tests.

```java
// BEFORE — decorator silently skipped
public class HybridCaseRetriever implements CaseRetriever {
    HybridCaseRetriever(QdrantClient client, MultiModalEmbedder embedder,
                        TenantGuard guard, RagConfig config) { ... }
}

// Producer class
@Produces @ApplicationScoped
HybridCaseRetriever caseRetriever() {
    return new HybridCaseRetriever(client, effectiveEmbedder(), ...);
}

// AFTER — decorator intercepts correctly
@ApplicationScoped
public class HybridCaseRetriever implements CaseRetriever {
    @Inject
    HybridCaseRetriever(QdrantClient client, MultiModalEmbedder embedder,
                        Instance<CurrentPrincipal> principal, RagConfig config) {
        this(client, MatryoshkaMultiModalEmbedder.wrapIfNeeded(embedder, ...),
             TenantGuard.of(principal.isResolvable() ? principal.get() : null), config);
    }

    // Package-private for unit tests
    HybridCaseRetriever(QdrantClient client, MultiModalEmbedder embedder,
                        TenantGuard guard, RagConfig config) { ... }
}
```

### Why this is non-obvious
The decorator registers at build time — `BuildTimeEnabledProcessor` confirms "Enabling". No error, no warning at any point. The `@QuarkusTest` passes because it uses a managed `@Alternative` bean, not the producer method bean. Every observable diagnostic says the decorator is active. The only signal is "benchmark results are identical" — which could mean a dozen things.

### Why this fix
Changing the return type of the producer to the interface (`CaseRetriever` instead of `HybridCaseRetriever`) does NOT fix it — Arc still can't subclass the produced instance. The only fix is making the class a managed bean so Arc generates a decorating subclass.

### Alternatives considered
- Changing producer return type to `CaseRetriever` (interface) — doesn't help; Arc still wraps the concrete instance in a client proxy without subclass generation
- Adding `@Unremovable` to the decorator — doesn't help; the decorator IS registered, it just doesn't intercept
- Creating a Quarkus extension with `DecoratorBuildItem` — unnecessary complexity when managed bean conversion works
