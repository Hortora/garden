### Symptom

`Column "TENANCY_ID" not found` on INSERT into a table that was created by a Flyway migration that explicitly includes that column. The error looks like Hibernate's `drop-and-create` is rebuilding the table without the column, or like the schema-management property override isn't taking effect.

### Context

Multiple Flyway migrations on the same table: V2002 creates it with `tenancy_id`, V2003 (a later migration in the same directory) adds an index and — buried at the end — `ALTER TABLE ... DROP COLUMN IF EXISTS tenancy_id`. Flyway reports all migrations as successful. The column exists after V2002 and is gone after V2003.

### What was tried (didn't work)

- Renamed `database.generation` to `schema-management.strategy` (Quarkus 3.23+ rename) — no effect
- Removed `@DiscriminatorValue` from the entity — no effect
- Changed `schema-management.strategy` from runtime profile override to build-time `%test.` config — no effect
- Enabled Flyway for ALL tests (not just ledger profile) — no effect
- Removed FK constraint from the CREATE TABLE — no effect

All of these were reasonable hypotheses for why Hibernate might be recreating the table. None were the actual cause.

### Root cause

A later Flyway migration (V2003) explicitly dropped the column with `ALTER TABLE ... DROP COLUMN IF EXISTS tenancy_id`. The DROP was added during a design phase that assumed JOINED inheritance (where the column would live on the parent table). The entity later used `@MappedSuperclass` instead, requiring the column on the subclass table. The migration was never updated.

### Fix

Remove the `DROP COLUMN` from V2003. The column must stay on the subclass table because the entity extends `@MappedSuperclass` (all fields mapped to the entity's own table).

### Why this is non-obvious

The symptom ("Column X not found") naturally leads to investigating schema generation, property overrides, and entity mapping — not to reading every migration in sequence looking for a DROP. Flyway logs all migrations as successful, the column IS created by V2002, and the DROP in V2003 uses `IF EXISTS` so it never errors. The investigation consumed hours across Hibernate DDL analysis, H2 MODE=PostgreSQL case sensitivity, and Quarkus property deprecation — all red herrings.

### Alternatives considered

- Consolidating all columns into V2002 (wouldn't help — V2003 still drops it)
- Switching to Hibernate `drop-and-create` for all tests (masks the migration bug)
- Using unique H2 database URLs per test profile (doesn't fix the column being dropped)
