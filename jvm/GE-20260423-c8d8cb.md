## Symptom

`CompletableFuture` registered in a `ConcurrentHashMap<String, List<CompletableFuture<T>>>`
is never completed, even though `complete()` was called with the correct key.
No exception is thrown. The caller blocks indefinitely.

## Context

Pattern: a registry maps correlation keys to lists of futures. Two threads:
1. `register(key)` — calls `computeIfAbsent(key, k -> new ArrayList<>()).add(future)`
2. `complete(key, result)` — calls `pending.remove(key)` then completes all futures in the removed list

## Root cause

`computeIfAbsent()` is atomic at the map level (inserts the list if absent), but
`.add(future)` on the returned list happens **outside** the atomic window. A concurrent
`complete()` calling `pending.remove(key)` between these two steps removes the list
from the map before the future is added to it. The future is added to a list that no
longer belongs to the map and will never be seen by subsequent `complete()` calls.

```
Thread A: list = computeIfAbsent("key")  ← list inserted in map
Thread B: pending.remove("key")          ← list removed from map
Thread A: list.add(future)               ← future added to orphaned list; never completed
```

## Fix

Synchronize both `register()` and `complete()` on the map, completing futures
**outside** the lock to avoid deadlock from completion callbacks:

```java
public CompletableFuture<T> register(String key) {
    CompletableFuture<T> future = new CompletableFuture<>();
    synchronized (pending) {
        pending.computeIfAbsent(key, k -> new ArrayList<>()).add(future);
    }
    return future;
}

public void complete(String key, T result) {
    List<CompletableFuture<T>> futures;
    synchronized (pending) {
        futures = pending.remove(key);   // atomic remove under lock
    }
    if (futures == null) return;
    for (CompletableFuture<T> f : futures) {
        f.complete(result);              // complete OUTSIDE lock — callbacks may re-register
    }
}
```

## Why this is non-obvious

`ConcurrentHashMap` makes individual operations atomic, so developers assume
`computeIfAbsent(...).add(...)` is safe. The atomicity guarantee covers the lookup
and insert, not the subsequent method call on the returned value. The bug is
invisible under low load (register and complete rarely interleave) but emerges under
concurrency — exactly the condition ConcurrentHashMap is meant to handle.
