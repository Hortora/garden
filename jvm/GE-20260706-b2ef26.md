### Symptom

After updating casehub-ledger to the July 2026 SNAPSHOT, every file that imports `io.casehub.ledger.runtime.model.LedgerEntry` fails with "cannot find symbol." Every file importing `io.casehub.ledger.runtime.repository.LedgerEntryRepository` also fails. The old jar resolves from the local .m2 cache, so the build may appear green until a `mvn -U` forces a SNAPSHOT refresh — then 50+ files break simultaneously.

### Context

casehub-ledger restructured its class hierarchy. The single `LedgerEntry` class was split into two:

| Before | After |
|--------|-------|
| `io.casehub.ledger.runtime.model.LedgerEntry` (@Entity, JOINED inheritance) | `io.casehub.ledger.api.model.LedgerEntry` (@MappedSuperclass — fields, canonical bytes, supplements) |
| — | `io.casehub.ledger.runtime.model.jpa.JpaLedgerEntry` (@Entity, JOINED inheritance, JPA listeners, JPA supplements) |

Additionally:
- `LedgerEntryRepository` moved from `io.casehub.ledger.runtime.repository` to `io.casehub.ledger.api.spi`
- `ComplianceSupplement` moved from `io.casehub.ledger.runtime.model.supplement` to `io.casehub.ledger.api.model.supplement`
- New `JpaComplianceSupplement` in `io.casehub.ledger.runtime.model.supplement` — must be instantiated instead of `ComplianceSupplement` for JPA persistence
- `LedgerVerificationService` moved from `io.casehub.ledger.api.service` to `io.casehub.ledger.runtime.service`
- `LedgerErasureService` moved from `io.casehub.ledger.api.privacy` to `io.casehub.ledger.runtime.privacy`

### Root cause

The split separates the pure-Java API model (portable, no JPA runtime) from the JPA persistence layer. `LedgerEntry` as `@MappedSuperclass` can be used in non-JPA contexts; `JpaLedgerEntry` carries the `@Entity`, `@Inheritance(JOINED)`, `@DiscriminatorColumn`, listeners, and named queries.

No migration guide was shipped with the SNAPSHOT.

### Fix

Three import replacements cover the migration:

**1. Entity subclasses** (files in `*.ledger.*LedgerEntry.java`):
```java
// Before
import io.casehub.ledger.runtime.model.LedgerEntry;
public class MyLedgerEntry extends LedgerEntry { ... }

// After
import io.casehub.ledger.runtime.model.jpa.JpaLedgerEntry;
public class MyLedgerEntry extends JpaLedgerEntry { ... }
```

**2. Type references** (services, resources, tests — anything that uses `LedgerEntry` as a type but doesn't extend it):
```java
// Before
import io.casehub.ledger.runtime.model.LedgerEntry;

// After
import io.casehub.ledger.api.model.LedgerEntry;
```

**3. Repository and service imports:**
```java
// Before
import io.casehub.ledger.runtime.repository.LedgerEntryRepository;

// After
import io.casehub.ledger.api.spi.LedgerEntryRepository;
```

**4. Supplement instantiation:**
```java
// Before
new ComplianceSupplement(...)

// After
new JpaComplianceSupplement(...)
// import io.casehub.ledger.runtime.model.supplement.JpaComplianceSupplement
```

casehub-clinical required 55 file changes across this migration.

### Why this is non-obvious

The old `LedgerEntry` import resolves against a stale SNAPSHOT jar in the local .m2 cache. The build stays green until the cache is refreshed. When it breaks, it breaks everywhere simultaneously with no hint about the new class hierarchy. The `JpaLedgerEntry` class name doesn't appear in any error message — you have to inspect the jar contents to discover it exists.

### Why this fix

The alternative — staying on `LedgerEntry` from the API package — compiles but fails at runtime: `@MappedSuperclass` entities are not registered in the JPA metamodel, so JOINED inheritance queries fail silently or throw opaque Hibernate errors. Consumer entity subclasses MUST extend `JpaLedgerEntry` for the discriminator column and named queries to work.
