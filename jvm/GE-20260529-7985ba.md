## Context

Adding a new Quarkus library module with a Flyway migration. The module has `quarkus-flyway`
as a compile dependency and a migration SQL file. `@QuarkusTest` tests inject JPA entities
mapped to the migration's table.

## Symptom

`@QuarkusTest` startup fails with a Hibernate schema validation error or SQL table-not-found
exception — the table from the Flyway migration does not exist. Flyway appears to be silent
(no migration log output).

## Root Cause

`quarkus.flyway.migrate-at-start` defaults to `false`. Without it explicitly set to `true`
in the test application.properties, Flyway starts but does not run migrations. The schema
never gets created.

## Fix

Add to `src/test/resources/application.properties`:

```properties
quarkus.flyway.migrate-at-start=true
quarkus.flyway.locations=classpath:db/your-module/migration
```

Both properties are required: `migrate-at-start` to actually run migrations, and `locations`
to tell Flyway where to find the migration files.

## Why This Is Non-Obvious

Adding `quarkus-flyway` as a dependency and placing SQL files in the expected location feels
sufficient — the expectation is that Flyway auto-runs on startup. The default `false` value
requires an explicit opt-in that's only mentioned in the Quarkus Flyway documentation,
not surfaced as a warning when migrations are present but not running.

Note: the related GE-20260429-ede58e documents a different issue — `migrate-at-start` being
build-time fixed, preventing test overrides when a jar's config already sets it.
