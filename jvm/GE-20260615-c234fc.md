## Symptom

A `@DefaultBean @ApplicationScoped` no-op implementation is placed in an `api` module alongside its SPI interface. A consumer provides a real `@ApplicationScoped` implementation. When injected via `Instance<SpiInterface>`, CDI iterates both beans — the `@DefaultBean` is NOT suppressed by the consumer bean. The no-op runs alongside the real implementation.

No CDI ambiguity error at startup. No warning. The `@DefaultBean` annotation is simply not processed.

## Context

`@DefaultBean` is from `io.quarkus.arc.DefaultBean` — a Quarkus ARC extension annotation. For ARC to process it, the `quarkus-arc` dependency must be on the module's compile classpath. Pure-Java API modules typically do not declare `quarkus-arc` — they declare only `jakarta.enterprise.cdi-api` for `@ApplicationScoped`.

The class IS discovered by ARC (the module has a Jandex index), so the `@ApplicationScoped` annotation works. But the `@DefaultBean` annotation is not processed — ARC treats it as a regular bean.

## Root cause

`@DefaultBean` is processed by ARC's build-time augmentation. The augmentation reads the annotation from the Jandex index. If the annotation class (`io.quarkus.arc.DefaultBean`) is not in the Jandex index of the declaring module's dependencies, ARC cannot resolve it and silently ignores it.

## Fix

Place `@DefaultBean` no-op implementations in a module that has `quarkus-arc` as a compile dependency — typically `runtime/`. All other engine no-ops follow this pattern (`NoOpWorkerStatusListener`, `NoOpCaseChannelProvider`, etc. are in `runtime/internal/worker/`).

```
# Wrong — api/ has no quarkus-arc:
api/src/main/java/io/casehub/api/spi/NoOpFoo.java

# Right — runtime/ has quarkus-arc:
runtime/src/main/java/io/casehub/engine/internal/worker/NoOpFoo.java
```

## Why this is non-obvious

The class compiles, is discovered by CDI, and works as a bean. The only thing that doesn't work is the `@DefaultBean` semantics — and there is no error, warning, or log message indicating the annotation was ignored. The failure mode is that the no-op runs when it shouldn't, which in most cases is invisible (a no-op doing nothing alongside a real implementation is harmless — until it's iterated via `Instance<>`).
