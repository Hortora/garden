### Symptom
Using `new NativeImageResourcePatternsBuildItem("db/work/migration/.*\\.sql")` or `includePatterns(String...)` in a `@BuildStep` produces deprecation warnings at compile time. The single-String constructor takes a regex; the `builder()` API takes a glob.

### Context
A Quarkus extension deployment processor needs to register SQL migration resource files for native image inclusion. The natural approach based on reading the class name is to pass the pattern directly to the constructor.

### Fix
Use the builder with `includeGlob()`:

```java
@BuildStep
NativeImageResourcePatternsBuildItem registerMigrationResources() {
    return NativeImageResourcePatternsBuildItem.builder()
            .includeGlob("db/work/migration/*.sql")
            .build();
}
```

`includeGlob(String)` takes a standard glob pattern (`*` matches within a path component, `**` crosses boundaries). The deprecated constructor took a regex — the builder is cleaner and avoids escaping.

### Why this is non-obvious
No Quarkus guide or extension authoring documentation mentions the builder pattern or the deprecation of the single-argument constructor. The deprecation is only visible as a compiler warning. The builder method `includeGlob()` vs `includePattern()` (also deprecated) is not documented.
