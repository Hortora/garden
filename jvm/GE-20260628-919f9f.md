### Symptom

In production, endpoints that call `CurrentPrincipal.tenancyId()` via JPA store queries throw `MissingTenancyException` → HTTP 500 when the request was authenticated by a custom (non-OIDC) `HttpAuthenticationMechanism`. Tests always pass — the regression is invisible to the entire test suite.

### Context

A custom auth mechanism creates a non-anonymous `SecurityIdentity` (with roles, attributes, but no JWT) for service-to-service calls. `OidcCurrentPrincipal @Alternative @Priority(100)` is the winning `CurrentPrincipal` in production. JPA stores inject `CurrentPrincipal` and call `tenancyId()` in every query.

### What was tried (didn't work)

1. Using `@CrossTenant` stores — `CrossTenantCommitmentStore` lacks `findByCorrelationId()`, can't cover all endpoint operations
2. Setting `SecurityIdentity.anonymous=true` with roles — `policy=authenticated` rejects anonymous identities, and anonymous+roles is semantically wrong

### Root cause

`OidcCurrentPrincipal.tenancyId()` assumes non-anonymous identities always have a JWT:

```java
return identity.isAnonymous()
    ? DEFAULT_UUID
    : jwt.claim("tenancyId").orElseThrow(MissingTenancyException::new);
```

A custom mechanism creates a non-anonymous SecurityIdentity without JWT data. `isAnonymous()` returns false → falls through to `jwt.claim()` → no JWT → `MissingTenancyException`.

Tests never catch this because `FixedCurrentPrincipal @Alternative @Priority(200)` wins in tests, masking the `OidcCurrentPrincipal` code path entirely.

### Fix

App-local `CurrentPrincipal @Alternative @Priority(150)` that detects bridge-authenticated identities via a SecurityIdentity attribute and returns the default tenancyId:

```java
@RequestScoped
@Alternative
@Priority(150) // between OidcCurrentPrincipal(100) and FixedCurrentPrincipal(200)
public class AppCurrentPrincipal implements CurrentPrincipal {
    @Inject SecurityIdentity identity;
    @Inject OidcCurrentPrincipal oidcPrincipal; // by concrete type — no alternative competition

    private static final String DEFAULT_TENANCY = "278776f9-e1b0-46fb-9032-8bddebdcf9ce";

    @Override
    public String tenancyId() {
        Boolean bridge = identity.getAttribute("casehub.plugin.bridge");
        if (bridge != null && bridge) return DEFAULT_TENANCY;
        return oidcPrincipal.tenancyId();
    }
    // delegate other methods similarly
}
```

The mechanism stamps `casehub.plugin.bridge=true` on the SecurityIdentity. The app-local CurrentPrincipal detects this attribute and returns the default tenancyId without touching the JWT path.

### Why this is non-obvious

The regression is **production-only** — no test configuration can surface it because `FixedCurrentPrincipal @Priority(200)` always wins over `OidcCurrentPrincipal @Priority(100)` in tests. The only way to discover it is bytecode analysis of `OidcCurrentPrincipal.tenancyId()` combined with tracing through `JpaCommitmentStore` query methods that call `currentPrincipal.tenancyId()` in WHERE clauses. The `save()` method does NOT call `tenancyId()` — only query methods do — so some endpoints work (commit) while others fail (done, commitments).

### Why this fix

An app-local `@Priority(150)` CurrentPrincipal is the correct adapter for an app with two auth mechanisms (OIDC + bridge). It's not a workaround — it's a proper composition of identity contexts. The platform-level fix (platform#121: OidcCurrentPrincipal checks SecurityIdentity attributes) is the long-term solution; this adapter is removable when it ships.
