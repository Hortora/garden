### What it is

Any agentic system that records outcomes and uses them to weight routing decisions is running
**degenerate CBR** — Case-Based Reasoning with only two of the four steps active:

| CBR step | Present? | Typical implementation |
|----------|----------|----------------------|
| **Retain** | ✅ | Ledger attestations, trust score updates |
| **Reuse** | ✅ | TrustWeightedAgentStrategy, StrategyTrustRouter |
| **Retrieve** | ❌ | Missing — exact key lookup only |
| **Revise** | ❌ | Missing — no plan adaptation |

The system improves with repetition of *identical* situations but cannot generalise across
*similar-but-not-identical* ones.

### Why this is non-obvious

Trust-score routing *looks* like learning — outcomes feed back into future decisions. But without
Retrieve (similarity search), the system can only match past experience by exact key. A game AI
that won 70% of Zerg roach-rush games cannot apply that knowledge to a Zerg mass-zergling game,
even though the situations are strategically analogous. Each novel situation starts cold.

The standard framing is "trust routing" or "credence updating" — not CBR. Developers building
trust-scored routing rarely recognise they're building a degenerate CBR system, so they never
notice the Retrieve gap.

### The diagnostic

Apply this test to any agentic system with outcome-based routing:

1. **Retain?** — Does it write structured case representations (problem + solution + outcome)?
   Ledger attestations alone are partial — they score credence but aren't retrievable by similarity.
2. **Retrieve?** — Can it find the k most similar past cases given a feature vector, not an exact key?
   Exact-key lookup (`contextKey = "PvZ"`) is degenerate retrieval.
3. **Reuse?** — Does it select a solution from retrieved cases (not just apply past weights)?
4. **Revise?** — Does it adapt a retrieved solution to the current context?

If steps 1–2 are "no", the system cannot generalise. Fix: add `CaseMemoryStore` writes at case
close (full case representation) and `CaseRetriever` invocation at case open (similarity search).

### Fix

In CaseHub:
- **Retain**: write `CbrCaseEntry` to `CaseMemoryStore` at case close (casehubio/engine#477)
- **Retrieve**: invoke `CaseRetriever` at plan creation with a domain feature vector (casehubio/engine#478)
- **Reuse**: replace application-layer routing workarounds with `ImplementationRoutingStrategy` (casehubio/engine#476)
- **Retrieve storage backend**: `memory-mem0` or `memory-graphiti` for vector/graph similarity; `memory-jpa` for structured field matching

In any agentic system:
- Store full problem-description + solution + outcome at case close (not just a win/loss counter)
- Add similarity-based retrieval at case open using a domain feature vector

### Constraints

- Requires a `CaseMemoryStore` backend that supports similarity search (mem0, graphiti, or custom)
- Feature vector design is domain-specific — there is no universal feature extractor
- Cold-start problem: no past cases → retrieval returns empty → must fall back gracefully to trust-only routing or random selection
