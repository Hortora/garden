## @IfBuildProperty/@UnlessBuildProperty evaluated at augmentation only — QuarkusTestProfile properties have no effect on bean activation

### Symptom

Setting a build property in `QuarkusTestProfile.getConfigOverrides()` to activate or deactivate a `@IfBuildProperty`-annotated bean has no effect. The bean activates based on whatever value was set at augmentation time. No error — wrong bean is silently included or excluded.

### Root cause

`@IfBuildProperty` and `@UnlessBuildProperty` are evaluated during Quarkus augmentation (build/compile time), not at runtime. `QuarkusTestProfile.getConfigOverrides()` is applied after augmentation — the CDI bean graph is already baked in and cannot be changed at test time.

### Fix

To conditionally activate beans in tests, use mechanisms evaluated at augmentation time:

```properties
# src/test/resources/application.properties — evaluated at augmentation
quarkus.arc.selected-alternatives=io.example.MyTestBean
# or
quarkus.arc.exclude-types=io.example.ProductionBean
```

Or use `@IfBuildProfile` with dedicated test profiles (also evaluated at augmentation).

### Why this is non-obvious

`getConfigOverrides()` is described as configuration overrides and developers naturally expect it to control all configuration including bean activation conditions. No error, no warning — the profile override is accepted, applied at runtime, but has zero effect on bean activation which was already decided.
