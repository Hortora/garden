## ArrayList unsafe for @ObservesAsync test captures — @ObservesAsync dispatches on managed executor thread, not test thread

### Symptom

A `@QuarkusTest` integration test uses Awaitility to wait for a message captured by a test double:

```java
Awaitility.await().atMost(5, SECONDS).until(() -> !connector.sent().isEmpty());
assertThat(connector.sent()).hasSize(1);
```

The test passes reliably on a single core or slow CI, but fails intermittently at higher parallelism or on fast machines — the assertion sees an empty list even though `send()` was called.

### Root cause

`@ObservesAsync` dispatches its observer method on a **managed executor thread** (Quarkus's CDI async event thread pool), not the test thread. When `send()` adds a message to an `ArrayList` on the executor thread, the JMM does not guarantee that the write becomes visible to the test thread reading the same list via Awaitility's poll loop.

`ArrayList` is not thread-safe and provides no happens-before guarantees. In practice, the write is visible most of the time because Awaitility's polling loop incidentally introduces enough wall-clock delay for the CPU to flush cache lines. This makes the bug intermittent and nearly impossible to reproduce deterministically.

### Why this is non-obvious

The test appears single-threaded — there is no explicit `Thread.start()` or `ExecutorService`. The asynchrony is hidden inside CDI's `@ObservesAsync` annotation. It is easy to classify this as "my tests run serially" and reach for `ArrayList` as the obvious serialisation-safe choice. The bug only surfaces under load or on hardware with specific cache coherency behaviour, making it hard to diagnose.

### Fix

Use `CopyOnWriteArrayList` as the backing store. COWAL's write operation (`add`) is synchronised and provides a happens-before guarantee to subsequent reads:

```java
@Mock
@Singleton
public class TestConnector implements Connector {
    // CopyOnWriteArrayList: @ObservesAsync dispatches send() on a managed executor thread;
    // ArrayList has no JMM visibility guarantee to the test thread reading via Awaitility.
    private final List<ConnectorMessage> sent = new CopyOnWriteArrayList<>();

    public List<ConnectorMessage> sent() { return sent; }
    public void reset() { sent.clear(); }

    @Override
    public void send(ConnectorMessage msg) { sent.add(msg); }
}
```

COWAL is appropriate here because (a) writes are rare (one per test assertion point) and (b) correctness matters more than throughput.

### Alternatives considered

- **`Collections.synchronizedList(new ArrayList<>())`:** provides thread safety but not visibility — `synchronized` blocks are not sufficient for cross-thread visibility without a full happens-before edge.
- **`volatile` field on the counter:** works for simple counts but not for list contents.
- **`CompletableFuture` capture pattern:** test subscribes to a future; `send()` completes it. More complex but eliminates polling entirely. Valid for tests that can restructure around it.
