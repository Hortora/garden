### Symptom
`NullPointerException` at `lifecycleEvents.fireAsync(...)` deep inside a handler chain. The handler's `Event<CaseLifecycleEvent>` field is `null` despite having `@Mock Event lifecycleEvents` in the test.

### Context
Testing a CDI `@ApplicationScoped` handler that uses field injection (`@Inject Event<CaseLifecycleEvent>`). The test uses `@ExtendWith(MockitoExtension.class)` with `@InjectMocks` to construct the handler.

### What was tried (didn't work)
- Declaring `@Mock Event lifecycleEvents` (raw type) — Mockito silently leaves the field null
- Declaring `@Mock Event<CaseLifecycleEvent> lifecycleEvents` — same result; `@InjectMocks` still fails to match

### Root cause
Mockito's `@InjectMocks` field-matching algorithm cannot resolve a `@Mock Event` (or `@Mock Event<CaseLifecycleEvent>`) to a field declared as `Event<CaseLifecycleEvent>` in the target class. The generic parameter creates a type mismatch in Mockito's reflective injection. The field is silently left `null` — no warning, no error at construction time.

### Fix
Convert the handler from CDI field injection to constructor injection. Pass mocks explicitly in the test's `@BeforeEach`:

```java
handler = new MyHandler(caseInstanceCache, eventBus, lifecycleEvents, ...);
```

This bypasses Mockito's field-matching entirely — the mock is passed directly to the constructor parameter.

### Why this is non-obvious
The NPE occurs at the `fireAsync()` call site, not at handler construction. A developer's first instinct is to check the event payload or the mock's `when()` setup — not that the mock was never injected at all. The `@InjectMocks` annotation gives a false sense of completeness.

### Alternatives considered
- Using `@SuppressWarnings("rawtypes")` with raw `Event` — still fails
- Using `MockitoAnnotations.openMocks(this)` manually — same underlying issue
