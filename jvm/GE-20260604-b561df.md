## @ApplicationScoped CDI bean captures immutable-record snapshot at construction — registry updates are invisible

**ID:** GE-20260604-b561df
**Stack:** Quarkus CDI, Java 21 (records)
**Labels:** `#cdi` `#applicationscoped` `#registry` `#snapshot` `#stale-state`

### Symptom

An `@ApplicationScoped` CDI bean holds a `final ImmutableRecord session` field. External callers update a backing registry (e.g. `ConcurrentHashMap.computeIfPresent(id, s -> s.withField(newValue))`) which replaces the map entry with a new record instance. The bean's `session` field still points to the original snapshot — silently. No exception, no warning. Any data derived from the snapshot (e.g. selection context) remains perpetually stale.

### Context

Pattern: a factory or lifecycle manager constructs a CDI bean and passes in an immutable record at construction time. The record is also stored in a registry keyed by an ID. Later, update operations target the registry entry (replacing the map value) rather than the bean's field.

Occurs whenever:
- The bean is `@ApplicationScoped` (constructed once)
- The constructor argument is an immutable record (or any value type)
- The registry stores the canonical, live value but the bean holds its own copy

### What was tried

The registry `updateSelection()` call was verified to work: `computeIfPresent` did replace the map entry with a new record carrying the updated selection. The bug was not in the registry — it was that the already-constructed CDI bean had captured the pre-update snapshot and never re-read from the registry.

### Root cause

CDI `@ApplicationScoped` beans are proxied and constructed once. A constructor parameter (or field set during construction) is a snapshot of the value at that moment. Java immutable records cannot be mutated in-place. When the registry replaces its entry, the bean holds a stale reference to the previous record. There is no observer mechanism to notify the bean that the registry entry was replaced.

### Fix

Inject the registry itself (not the record) and resolve the live value on each method invocation:

```java
// Before — captures a snapshot at construction; update_selection changes are invisible
public class ReviewerChannelBackend implements ChannelBackend {
    private final ReviewSession session;  // stale after any registry update

    ReviewerChannelBackend(ReviewSession session, ...) {
        this.session = session;  // snapshot — never updated again
    }

    public void post(...) {
        String context = buildSelectionContext(this.session);  // always stale
    }
}

// After — reads live session from registry on every invocation
public class ReviewerChannelBackend implements ChannelBackend {
    private final ReviewSessionRegistry registry;
    private final UUID channelId;

    ReviewerChannelBackend(ReviewSessionRegistry registry, UUID channelId, ...) {
        this.registry = registry;
        this.channelId = channelId;
    }

    public void post(...) {
        ReviewSession session = registry.find(channelId).orElse(null);
        if (session == null) return;  // session ended — silently ignore
        String context = buildSelectionContext(session);  // always live
    }
}
```

### Why this is non-obvious

The registry update (`computeIfPresent`) is correct and the map entry is genuinely replaced. The bug is not in the update path — it's in the consuming bean having captured the old value. This looks like the selection update "didn't work" rather than "worked but the result was never read." The indirection through a CDI proxy obscures when construction happened. Expert Quarkus developers know that `@ApplicationScoped` beans are singletons but don't instinctively reach for "is my injected value a stale snapshot?" when debugging.

### Why inject registry over record

Injecting the registry (and resolving on each call) ensures the consumer always sees the canonical live value. The overhead is a single map lookup per invocation — negligible for event-driven handlers. Passing the record directly at construction is convenient but creates invisible coupling between the construction-time state and the consuming bean's view of the world.
