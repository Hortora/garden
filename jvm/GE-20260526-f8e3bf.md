In casehub-qhorus, `MessageService.pollAfter(UUID channelId, Long afterId, int limit)` with `afterId=0L` returns all messages in the channel. This works because `InMemoryMessageStore` initialises its `idCounter` to 1 — all persisted message IDs are ≥ 1. The `MessageQuery.matches()` filter checks `m.id > afterId`, so `afterId=0` matches every stored message.

This is the correct pattern for fetching all messages in a channel in `@QuarkusTest` tests that use `InMemoryMessageStore`:

```java
// In @QuarkusTest — fetches all COMMAND/DONE/DECLINE messages (EVENT excluded by default)
List<Message> messages = messageService.pollAfter(channel.get().id, 0L, 200);
assertTrue(messages.stream().anyMatch(m -> MessageType.COMMAND == m.messageType));
```

### Why this is non-obvious

The method name `pollAfter` implies a cursor-based pagination pattern where you provide the last-seen ID. Using `0L` as a "fetch all" sentinel is not documented — it works only because the ID counter starts at 1, which is an implementation detail of `InMemoryMessageStore`. The JPA-backed `JpaMessageStore` may behave differently (auto-increment sequences may start at 1 but are database-dependent). This pattern is reliable for test code using InMemoryMessageStore but should not be assumed for production JPA queries.
