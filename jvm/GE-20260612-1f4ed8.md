### Symptom

`org.hibernate.PropertyValueException: not-null property references a null or transient value for entity CaseLedgerEntry.tenancyId`

The test code explicitly sets `cle.tenancyId = "test-tenant"`. Java field access confirms the value is set. The error is impossible — except it isn't.

### Context

JPA JOINED inheritance with a base class (`LedgerEntry`) and subclass (`CaseLedgerEntry`). Both declare a `tenancyId` field with `@Column(name = "tenancy_id", nullable = false)`. In JOINED inheritance, these map to separate columns on separate tables (`ledger_entry.tenancy_id` and `case_ledger_entry.tenancy_id`).

### What was tried (didn't work)

1. Setting `tenancyId` on the entity before calling `save()` — still fails
2. Using `em.persist()` directly instead of the repository's `save()` — still fails
3. Setting the base field via reflection workaround — works but is wrong
4. Clean rebuild (`mvn clean`) to rule out stale bytecode — still fails

### Root cause

Java field shadowing creates two separate fields: `LedgerEntry.tenancyId` and `CaseLedgerEntry.tenancyId`. They share the same name but are different fields in the JVM.

Hibernate bytecode enhancement (active by default in Quarkus 3.x) intercepts field access on entity classes. When code does `cle.tenancyId = "value"`, the enhancement transparently routes to the subclass field. When code does `((LedgerEntry) cle).tenancyId`, the enhancement ALSO routes to the subclass field — making the shadowing invisible to Java code.

But `em.persist()` reads field values via reflection on the actual declared fields. `LedgerEntry.class.getDeclaredField("tenancyId")` reads the base class field — which was never set (all writes were routed to the subclass by enhancement). The `nullable = false` constraint on the base table column fires.

### Diagnostic

```java
CaseLedgerEntry cle = new CaseLedgerEntry();
cle.tenancyId = "from-subclass";

// Java access (enhanced — both route to subclass):
cle.tenancyId                           → "from-subclass"
((LedgerEntry) cle).tenancyId           → "from-subclass"

// Reflection (raw — reveals the actual field state):
CaseLedgerEntry.class.getDeclaredField("tenancyId").get(cle) → "from-subclass"
LedgerEntry.class.getDeclaredField("tenancyId").get(cle)     → null  // THE BUG
```

### Fix

Remove the shadowing field from the subclass. JPA JOINED inheritance maps the base class field to the base table automatically — the subclass does not need its own declaration.

```java
// BEFORE (broken):
public class CaseLedgerEntry extends LedgerEntry {
    @Column(name = "tenancy_id", nullable = false)
    public String tenancyId;  // SHADOWS LedgerEntry.tenancyId
}

// AFTER (correct):
public class CaseLedgerEntry extends LedgerEntry {
    // tenancyId inherited from LedgerEntry — no shadowing
}
```

Migration: drop the `tenancy_id` column from the join table (it's redundant with `ledger_entry.tenancy_id`).

### Why this is non-obvious

Hibernate's bytecode enhancement makes the shadowing completely invisible to Java code. Both `cle.tenancyId` and `((LedgerEntry) cle).tenancyId` return the same value. The error message names the subclass entity but the actual null is on the base class — and you can't see it without reflection. The symptom appears impossible: "the field is null but I literally just set it."

### Why this fix

Removing the shadowing field is the only correct fix. Workarounds (setting the base field via reflection, using a special save method) are fragile — they couple application code to JPA internals and break when the entity hierarchy changes. The field belongs on the base class; the subclass inherits it.

### Alternatives considered

- **Reflection workaround in test helpers** — sets the base field directly. Works but couples tests to the field shadowing structure and masks the underlying design bug.
- **Setting both fields explicitly** — impossible via normal Java code because enhancement intercepts the access. Would require reflection on every write path.
- **Hibernate `@Access(AccessType.PROPERTY)` with getter/setter** — might route correctly but changes the access strategy for all fields and is a much larger change.

### Caveats

- This only triggers when bytecode enhancement is active. Without enhancement (e.g. plain Hibernate outside Quarkus), both `cle.tenancyId` and `((LedgerEntry) cle).tenancyId` access their respective declared fields — the shadowing is visible and the developer would naturally set both.
- The fix requires a Flyway migration to drop the redundant column from the join table.
