### The technique

When Thread A writes a volatile field and then does a CAS on an `AtomicReference`, Thread B is guaranteed to see Thread A's volatile write if Thread B subsequently does a `getAndSet` (or any CAS) on the same `AtomicReference` and observes Thread A's value.

```java
// Thread A (query() entry):
this.currentTurnFuture = new CompletableFuture<>();  // 1. volatile write
state.compareAndSet(IDLE, ACTIVE);                    // 2. CAS

// Thread B (close() entry):
State prev = state.getAndSet(CLOSED);                 // 3. sees ACTIVE (synchronizes-with step 2)
CompletableFuture<?> f = this.currentTurnFuture;     // 4. guaranteed to see the value from step 1
```

No additional lock, barrier, or `synchronized` block is needed.

### Why it works (JMM chain)

The Java Memory Model establishes happens-before via `volatile` reads and writes, and via synchronization actions (CAS counts). The chain:

- Step 1 happens-before step 2 (program order within Thread A)
- Step 2 **synchronizes-with** step 3 (Thread B's getAndSet observes Thread A's CAS — this is the JMM synchronization point for AtomicReference)
- Step 3 happens-before step 4 (program order within Thread B)

By transitivity: step 1 happens-before step 4. Thread B's read at step 4 sees Thread A's write at step 1.

**The key constraint:** the volatile write MUST happen BEFORE the CAS (step 1 before step 2). If reversed — CAS first, volatile write second — Thread B could observe the ACTIVE state (from the CAS) and then read the volatile field before Thread A has written it (null or stale). The JMM guarantee only flows through the synchronization point established by the CAS.

### Why non-obvious

Most developers reach for explicit synchronisation when sharing state between threads:
- Adding another `AtomicReference` for the field itself
- Using `synchronized` blocks
- Wrapping both operations in a lock

The insight: the `AtomicReference.compareAndSet()` already acts as a full memory barrier and establishes a synchronizes-with relationship with any subsequent CAS on the same reference. The pre-existing CAS provides the cross-thread visibility for free — no extra cost — as long as ordering is respected.

### Constraints

Requires that:
1. Thread A writes the volatile field BEFORE performing the CAS (not after)
2. Thread B reads the volatile field AFTER its own CAS on the same AtomicReference
3. Thread B's CAS observes the value written by Thread A's CAS (the synchronization point)

If Thread B's CAS sees a different value (e.g., it was already CLOSED by a third thread), the chain breaks. In practice this means Thread B must handle the null case defensively:

```java
CompletableFuture<?> f = this.currentTurnFuture;
if (f != null) { ... }
```

### Invalidation triggers

Revisit if the JMM specification changes for volatile and AtomicReference — stable since Java 5, unlikely to change.
