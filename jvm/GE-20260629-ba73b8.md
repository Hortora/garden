## Hibernate @Version skips increment on unmodified entities — dirty-checking gate before OCC

### Symptom
A test that does find() → save(unchanged) → find() → assert(version incremented) fails.
The version stays at its original value despite the save() call completing without error.

### Context
When building OCC retry logic, a natural test is: save an entity, re-read it, save again,
verify the version incremented. This fails when the re-saved context has no field changes.

### Root cause
Hibernate's dirty-checking runs BEFORE the @Version increment. If no managed entity fields
have changed since the last flush, Hibernate skips the UPDATE entirely — no SQL is issued,
and the @Version counter is not incremented. The save() method returns successfully,
giving no indication that nothing was persisted.

### Fix
Ensure the entity has at least one changed field before saving. In tests, add a detection
or modify a timestamp between find() and save():

```java
var found = store.find(...).orElseThrow();
var detection = new DetectionResult("g1", 0.8, DetectionSignal.DETECTED, Map.of());
var modified = found.withDetection(detection, T2);  // creates a real field change
store.save(modified).await().indefinitely();         // Hibernate issues UPDATE + version bump
```

### Why this is non-obvious
The @Version documentation implies every persist/merge increments the version. The
dirty-checking gate is not mentioned in the context of optimistic locking — it appears
only in performance-tuning documentation about avoiding unnecessary UPDATEs. A developer
building OCC expects the version to increment on every save, not only on dirty saves.
