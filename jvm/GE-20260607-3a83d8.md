### Symptom

```
error: incompatible types: <nulltype> cannot be converted to double
Double score = optDouble.orElse(null);
```

### Context

When extracting a nullable `Double` wrapper from an `OptionalDouble` — e.g. to store an absent numeric score as `null` in a record or DB column — `optDouble.orElse(null)` seems like the natural pattern but fails to compile.

### Root cause

`OptionalDouble` is the primitive specialisation of `Optional<Double>`. Its `orElse(double)` method accepts and returns a primitive `double`, not a boxed `Double`. The compiler cannot unbox `null` to `double`, so null is rejected as an incompatible type.

### Fix

```java
// Wrong — OptionalDouble.orElse() returns primitive double, not Double
Double score = trustGateService.currentScore(actorId).orElse(null);

// Correct option 1 — check presence, then extract and autobox
OptionalDouble opt = trustGateService.currentScore(actorId);
Double score = opt.isPresent() ? opt.getAsDouble() : null;

// Correct option 2 — box the OptionalDouble to Optional<Double> first
Optional<Double> boxed = trustGateService.currentScore(actorId).boxed();
Double score = boxed.orElse(null);
```

### Why this is non-obvious

`Optional<T>.orElse(null)` compiles and is a common Java pattern. `OptionalDouble` looks like a specialisation of `Optional<Double>` — the API surface is nearly identical — but its `orElse()` accepts and returns the primitive `double`. The error message ("cannot be converted to double") obscures the cause: the complaint is that `null` cannot be passed to a parameter of type `double`, not that `null` is semantically wrong.
