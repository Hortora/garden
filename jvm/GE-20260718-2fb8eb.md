### Symptom
SLA breach escalation silently does nothing. WorkItems expire, the Quartz
timer fires, ExpiryLifecycleService.expireItem() runs — but candidateGroups
never changes. No error, no warning, no log entry. Unit tests for the
SlaBreachPolicy pass (they instantiate the bean directly).

### Context
casehub-work resolves SPI implementations via StrategyResolver at startup:
`strategyResolver.resolve(SlaBreachPolicy.class, config.sla().breachPolicy())`.
The config key `casehub.work.sla.breach-policy` defaults to `"no-op"` in
WorkItemsConfig. The application's `LifeSlaBreachPolicy` (id="life-sla-breach")
is a valid @ApplicationScoped CDI bean discovered by StrategyResolver — but
the config never points to it.

### Root cause
StrategyResolver discovers ALL NamedStrategy beans and indexes them by id().
But selection is config-driven, not CDI-priority-driven. Unlike @Alternative
resolution, there is no "highest priority wins" fallback — if the config says
"no-op", the no-op bean is used regardless of what other beans exist.

The failure is invisible because:
1. No startup validation warns about unselected NamedStrategy beans
2. The no-op policy silently succeeds (returns a no-op BreachDecision)
3. Unit tests bypass StrategyResolver entirely
4. Integration tests that rely on Quartz timers hit a separate timing issue (timer fires before transaction commits), masking the config problem

### Fix
Set the config property explicitly in both production and test application.properties:
```properties
casehub.work.sla.breach-policy=life-sla-breach
```

### Why this is non-obvious
The CDI bean exists. CDI discovers it. StrategyResolver indexes it. Unit tests
instantiate it directly and pass. The only signal that something is wrong is the
absence of an effect — SLA escalation simply doesn't happen. There is no error,
no warning, no "bean not found" message. A developer who writes the SPI implementation
and sees unit tests pass will reasonably believe the wiring is complete.

### Why this fix
The no-op default exists for applications that don't want SLA enforcement.
casehub-life requires it — the fix is simply activating it via config. A
startup validator was considered but rejected as overengineering: the
integration test now catches any config regression.

### Alternatives considered
- Startup validator (@Observes StartupEvent) that fails if breach-policy is "no-op" — rejected: too aggressive for a single config line already under test
- Foundation-level change to remove the no-op default — rejected: belongs in casehub-work, not a casehub-life concern
