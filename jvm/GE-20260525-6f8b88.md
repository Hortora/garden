## Symptom

A REST endpoint or service method hangs indefinitely when activating a trial (or
any domain entity that must start an engine case and persist the returned ID).
No exception is thrown. No timeout fires. The request never completes.

In integration tests: everything passes. In production: deadlock under load.

## Context

Any Quarkus service that:
1. Is annotated `@Transactional`
2. Acquires a JDBC connection implicitly (e.g. via a Panache `findById` call)
3. Then calls `startCase(def, ctx).toCompletableFuture().join()` while that transaction — and its Agroal connection — is still open

The specific pattern in casehub-clinical was:

```java
@Transactional
public void activate(UUID trialId) {
    ClinicalTrial trial = ClinicalTrial.findById(trialId);  // acquires connection
    trial.status = TrialStatus.ACTIVE;

    // THIS DEADLOCKS IN PRODUCTION:
    UUID caseId = caseHub.startCase(ctx).toCompletableFuture().join();

    trial.engineCaseId = caseId;  // writes to still-open transaction
}
```

## Root Cause

`@Transactional` acquires an Agroal JDBC connection from the pool. `startCase()`
is asynchronous internally — it uses Vert.x event bus `request()` which waits for
a reply from `CaseStartedEventHandler`. That handler calls `eventLogRepository.append()`
and `caseInstanceRepository.save()` — both JDBC operations that need a connection from
**the same Agroal pool**.

The calling thread is blocked on `.join()`, holding the pool connection. Under pool
exhaustion (pool size 1 in tests, or pool-max under load in production), the
`CaseStartedEventHandler` cannot acquire a connection and hangs. The calling thread
cannot release its connection until `.join()` returns. `.join()` cannot return until
the handler completes. **Deadlock.**

## Why Tests Pass

Test configuration uses `casehub-engine-persistence-memory`
(`quarkus.arc.selected-alternatives` includes `MemoryPlanItemStore`,
`MemorySubCaseGroupRepository`). The memory stores never acquire JDBC connections.
`CaseStartedEventHandler` completes without touching the pool. `.join()` returns.
No deadlock. Tests are 100% green.

The production dependency is `casehub-engine-persistence-hibernate` — JPA stores
that DO acquire connections. The test-to-production divergence is total and silent.

## Fix

**Three-phase activation** — never hold a DB connection while blocking on `startCase()`:

```java
// activate() itself is NOT @Transactional — it orchestrates three phases
public void activate(UUID trialId) {
    Map<String, Object> ctx = markActive(trialId);   // commits connection
    UUID caseId = caseHub.startCase(ctx).toCompletableFuture().join(); // no tx open
    persistCaseId(trialId, caseId);                  // new connection, commits
}

@Transactional
Map<String, Object> markActive(UUID trialId) {
    ClinicalTrial trial = ClinicalTrial.findById(trialId);
    if (trial.status != TrialStatus.PLANNING) throw new ClientErrorException(CONFLICT);
    trial.status = TrialStatus.ACTIVE;
    // build and return initialContext; transaction commits on method return
    return buildContext(trial);
}

@Transactional
void persistCaseId(UUID trialId, UUID caseId) {
    ClinicalTrial trial = ClinicalTrial.findById(trialId);
    trial.engineCaseId = caseId;
    // transaction commits on method return
}
```

`startCase().join()` runs with no open transaction and no held connection. The
handler can freely acquire and release pool connections. No deadlock possible.

## Why This Is Non-Obvious

1. **Tests mask it completely.** Memory stores need no connections. Test suite is
   green. The only way to discover this in development is code review or load testing
   against a real pool.
2. **The error is a hang, not an exception.** No `DeadlockException`, no
   `PoolTimeoutException`, no log line. The request simply never returns.
3. **`@Transactional` connection hold timing is invisible.** Developers think of
   `@Transactional` as a logical boundary, not as "I am holding a physical JDBC
   connection from this line to the end of the method." The physical hold starts on
   the first JDBC operation inside the boundary.
4. **The failure only manifests at Agroal pool exhaustion.** With a large pool,
   the handler can acquire a different connection and complete normally. Only when
   all connections are consumed (common under concurrent load, guaranteed with
   pool-size=1) does the deadlock fire.
