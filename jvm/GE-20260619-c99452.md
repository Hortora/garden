## Broken pipe on subprocess stdin when process exits before write completes

**ID:** GE-20260619-c99452
**Stack:** Java ProcessBuilder, Unix (Linux/macOS)
**Symptom:** Subprocess tests pass reliably on macOS but are flaky on Linux CI. The completion callback is never invoked — the error/fault path fires instead. No obvious error in test output; the subprocess command itself is trivial (e.g. `echo`).
**Context:** Occurs when using `ProcessBuilder` to run a command that doesn't read stdin (like `echo`, `env`), while the caller writes inputData to the process's stdin stream. The race is between the process exiting and the stdin write completing.

### What was tried (didn't work)
- Checked if the subprocess command was platform-specific — it wasn't (`/bin/sh -c echo ...` works on both)
- Checked thread pool contention — `runSubscriptionOn(workerPool)` was fine
- Suspected test isolation — each test creates a fresh manager instance

### Root cause
Fast commands like `echo` exit before the stdin pipe is consumed. When the Java caller then calls `stdin.write(inputJson)` or `stdin.close()`, the OS raises SIGPIPE which Java surfaces as `IOException` ("Broken pipe"). The error handler catches this, calls `process.destroyForcibly()` (no-op on an already-dead process), and throws `RuntimeException` — which triggers the fault path instead of the normal completion path.

The race is timing-dependent: on macOS the JVM's stdin write usually wins; on CI (Ubuntu, often faster process startup/teardown), the subprocess exits first.

### Fix
Check `process.isAlive()` before treating the stdin `IOException` as fatal:

```java
try {
    byte[] inputJson = OBJECT_MAPPER.writeValueAsBytes(inputData);
    try (OutputStream stdin = process.getOutputStream()) {
        stdin.write(inputJson);
    }
} catch (IOException e) {
    if (!process.isAlive()) {
        LOG.debugf("Stdin write failed after process exited (broken pipe) — ignoring");
    } else {
        process.destroyForcibly();
        throw new RuntimeException("Failed to write inputData to stdin: " + e.getMessage());
    }
}
```

### Why this is non-obvious
The broken pipe is invisible on the developer's machine (macOS) — tests pass 100% locally. The flakiness only appears on CI, and the symptom ("completion publisher never called") points at the completion/callback mechanism, not at stdin. The actual cause (a harmless IOException on a dead process's stdin) is three abstraction layers away from the observed symptom.
