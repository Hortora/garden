### Symptom

`NullPointerException` thrown inside a `try (var reader = Json.createReader(...))` block
when a REST response omits an optional nested JSON object.

### Context

Occurs when parsing REST API responses that conditionally include a nested object.
Classic trigger: Slack's `conversations.list` response omits `response_metadata` entirely
for workspaces with fewer than the page limit — callers that always call
`.getJsonObject("response_metadata")` NPE silently.

### Root Cause

`JsonObject` has default-value overloads for primitives — `getString(name, default)`,
`getBoolean(name, default)`, `getInt(name, default)` — but not for complex types:

```java
// These work fine with a missing key:
obj.getString("key", "");        // returns ""
obj.getBoolean("key", false);    // returns false

// This returns null (no overload with a default):
JsonObject nested = obj.getJsonObject("key");  // null if key absent
nested.getString("subKey", "");  // NPE here — no indication the JSON API caused it
```

The asymmetry is not documented. The stacktrace points into the parsing code, not to the
absent key.

### Fix

Guard with `containsKey` before accessing any nested object or array:

```java
final String cursor = obj.containsKey("response_metadata")
        ? obj.getJsonObject("response_metadata").getString("next_cursor", "")
        : "";
```

Same pattern for `getJsonArray`: use `containsKey` or null-check the return value.

### Why This Is Non-Obvious

The `get*()` methods appear uniform — the IDE autocomplete shows a family of methods that
all accept a `(String name, T defaultValue)` overload. `getJsonObject` and `getJsonArray`
break this visual pattern. The return type is `JsonObject`/`JsonArray`, so the compiler
cannot warn about potential null — and the NPE stacktrace lands inside the caller's code,
not at the missing-key access.
