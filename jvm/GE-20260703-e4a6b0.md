### Symptom
WebSocket connections bypass JWT authentication despite a `@PreMatching ContainerRequestFilter` that reads a `token` query parameter and sets `Authorization: Bearer <token>` on the request. REST endpoints on the same server authenticate correctly. No error, no warning — the filter simply never fires on WebSocket upgrade requests.

### Context
Quarkus application using `quarkus-websockets-next` with `@WebSocket` endpoints and `quarkus-smallrye-jwt` for Bearer token auth. REST endpoints use the standard `Authorization` header. WebSocket connections pass the JWT as a query parameter (browser WebSocket API cannot send custom headers).

### What was tried (didn't work)
`@PreMatching ContainerRequestFilter` that reads `token` from the query string and sets `Authorization: Bearer <token>` — the standard pattern for promoting query-param tokens into headers. Works for REST; silently ignored for WebSocket upgrades because Quarkus WebSockets Next uses the Vert.x HTTP server directly and does not route upgrade requests through the JAX-RS (RESTEasy Reactive) filter chain.

### Root cause
Quarkus WebSockets Next (`@WebSocket`) and Quarkus REST (RESTEasy Reactive) share the same Vert.x HTTP server but use separate request pipelines. WebSocket upgrade requests are handled by Vert.x before they reach the JAX-RS layer. `ContainerRequestFilter` implementations — including `@PreMatching` — are registered on the RESTEasy Reactive pipeline and never see WebSocket upgrades.

### Fix
Use `HttpUpgradeCheck` (`io.quarkus.websockets.next.HttpUpgradeCheck`) — the Quarkus WebSockets Next mechanism for gating WebSocket upgrades:

```java
@ApplicationScoped
public class WebSocketTokenUpgradeCheck implements HttpUpgradeCheck {

    @Inject
    JWTParser jwtParser;

    @Override
    public Uni<CheckResult> perform(final HttpUpgradeContext ctx) {
        final String token = ctx.httpRequest().getParam("token");
        if (token == null || token.isBlank()) {
            return CheckResult.rejectUpgrade(401);
        }
        try {
            jwtParser.parse(token);
            return CheckResult.permitUpgrade();
        } catch (final Exception e) {
            return CheckResult.rejectUpgrade(401);
        }
    }
}
```

`HttpUpgradeContext.httpRequest()` returns the Vert.x `HttpServerRequest` with access to query params. `JWTParser.parse()` validates the JWT signature against the configured public key.

### Why this is non-obvious
The `ContainerRequestFilter` approach is the standard documented pattern for promoting auth tokens from non-standard locations (query params, cookies) into the `Authorization` header. Every Quarkus security guide describes it. The failure mode is completely silent — no log, no error, no "filter skipped" warning. The WebSocket connects successfully, appears to work, but `SecurityIdentity` is never populated. The auth bypass is only detectable by explicitly testing unauthenticated WebSocket connections.

### Alternatives considered
- `@Authenticated` on the `@WebSocket` class — does not work without a mechanism to get the token into the security pipeline. SmallRye JWT only reads `Authorization` headers, which browser WebSocket API cannot set.
- Custom `HttpAuthenticationMechanism` — would work but is heavier than `HttpUpgradeCheck` and affects all request types, not just WebSocket upgrades.
