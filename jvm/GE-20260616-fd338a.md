### Symptom

Tests using `CountDownLatch(1)` to detect file creation fire before the test action — the latch triggers on a phantom event delivered at watcher startup, not the actual file creation.

### Context

When using `DirectoryWatcher` in tests with `@TempDir`, files written to the directory before `watch()` is called generate CREATE/MODIFY events immediately after `watchAsync()` completes. A single-fire latch intended to catch the test's file creation instead catches a catch-up event for a pre-existing file.

### What was tried (didn't work)

- Adding `Thread.sleep(1000)` between `watch()` and the file creation — does not help because the catch-up events arrive after the sleep too
- Using `CountDownLatch(1)` — fires on the wrong event

### Root cause

macOS FSEvents API delivers catch-up events for recent filesystem activity when a watcher first registers on a directory. This is a platform-level behaviour — the directory-watcher library does not suppress or document it.

### Fix

Replace single-fire latches with `awaitility` polling for the specific expected entry:

```java
var received = new CopyOnWriteArrayList<ChangedEntry>();
source.watch(received::addAll);

Thread.sleep(1000); // warmup for watcher registration
Files.writeString(tempDir.resolve("new-file.txt"), "hello");

await().atMost(15, TimeUnit.SECONDS).untilAsserted(() ->
    assertThat(received).anyMatch(e ->
        e.path().equals("new-file.txt") && e.type() == ChangeType.ADDED));
```

### Why this is non-obvious

The directory-watcher library abstracts platform differences — its API gives no indication that startup behaviour varies by OS. Tests pass on Linux (inotify does not catch up) and fail only on macOS, making this a CI-vs-local surprise.

### Alternatives considered

- Filtering catch-up events by timestamp (rejected — no reliable way to distinguish catch-up from real events in the API)
- Clearing the event buffer after a warmup period (rejected — adds complexity and race conditions)
