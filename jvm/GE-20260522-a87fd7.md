## Symptom

`PreferenceProvider.resolve(scope)` returns the key's default value instead of a value
stored at root scope (`""`). No exception, no warning. Happens only when resolving with
any non-root path — resolving with `SettingsScope.root()` directly works correctly.

## Context

Walking a `Path` hierarchy to build an ancestor chain for preference resolution:

```java
Path current = path;
while (current != null) {
    result.add(0, current.value());
    current = current.parent();
}
```

For `Path.of("casehubio")` (one segment), `parent()` returns `null` — loop terminates.
Root scope (`""`) is never added to the ancestor list. `findByScopes(ancestors)` queries
only `["casehubio"]` — root rows are never fetched.

## Root Cause

`Path.parent()` is documented as "returns null if root **or** single-segment path."
Both zero-segment (root) and one-segment paths return `null`. The intuition — that
walking up from `Path.of("casehubio")` should eventually reach `Path.root()` — is wrong
for this API. Single-segment paths are tree leaves; root is a sentinel, not their parent.

This means root scope is not a natural ancestor of any non-root path in the walk.
It must be prepended explicitly.

## Fix

After the walk, prepend root scope when the resolution path is non-root:

```java
private static List<String> ancestors(final Path path) {
    final List<String> result = new ArrayList<>();
    Path current = path;
    while (current != null) {
        result.add(0, current.value());
        current = current.parent();
    }
    // Path.parent() returns null for single-segment paths (not root), so root is not
    // naturally reached by the walk above for non-root paths. Always prepend it.
    if (path.depth() > 0) {
        result.add(0, Path.root().value()); // "" — always lowest priority
    }
    return result;
}
```

`path.depth() > 0` is the correct guard: `Path.root()` is the only path with depth 0.

## Why This Is Non-Obvious

The `Path` Javadoc for `root()` says "Root is the ancestor of every non-root path" and
notes that "`isAncestorOf` and `parent` both handle root correctly." This reads as if
walking via `parent()` will eventually reach root — but `isAncestorOf` is a predicate
(does not walk), and "handles root correctly" refers to the zero-segment base case in the
predicate logic, not to parent() returning root as a parent. The walk stops before root.

The symptom (wrong value, no error) further obscures the cause: the preference key returns
its declared default, which is a valid value, so nothing obviously breaks in light testing.

## Why This Fix

`path.depth() > 0` reads intent directly: "if this path is not itself root, add root as
the base." An alternative — `!result.contains("")` — would also work but is O(n) and
obscures why root might already be present (it can't be, given the walk above, but the
condition doesn't make that clear).
