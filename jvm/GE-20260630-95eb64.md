### Symptom
Tests that compiled and passed suddenly fail with:
```
UnsatisfiedResolutionException: Unsatisfied dependency for type
MyBean and qualifiers [@Default]
The following beans match by type, but none has matching qualifiers:
- CLASS bean [...qualifiers=[@Any, @MyQualifier]...]
```
Failure occurs at Quarkus augmentation time, not compilation — `javac` succeeds, then augmentation explodes.

### Context
Adding a CDI qualifier annotation (e.g. `@WorkerBackend`) to an `@ApplicationScoped` bean class. CDI §2.3.1 specifies that adding any qualifier removes the implicit `@Default` qualifier.

### What was tried (didn't work)
The cross-repo committer updated all injection sites in the repo where the qualifier was added, but did not update downstream repos that also inject the same bean via `@Inject` or `@InjectMock`.

### Root cause
CDI §2.3.1: when a bean declares any qualifier other than `@Named`, the implicit `@Default` qualifier is removed. Every `@Inject` site without the matching qualifier now resolves to nothing. In a cross-repo setup, the committer only sees injection sites in their own repo — downstream repos have their own `@Inject`/`@InjectMock` fields that silently break.

### Fix
After adding a qualifier to a shared bean, grep ALL consuming repos for `@Inject` and `@InjectMock` of the concrete type:

```bash
# Search all sibling repos for injection sites
grep -rn "ClaudonyWorkerExecutionManager" ~/casehub/*/src/test/ --include="*.java" | grep -E "@Inject|@InjectMock"
```

Add the qualifier to every injection site found — both `@Inject` and `@InjectMock` require it.

### Why this is non-obvious
The CDI spec rule is well-documented, but the cross-repo blind spot is not. In a single repo, your IDE and tests catch every broken site immediately. In a multi-repo platform, the downstream repos compile fine (the class exists), and only fail at Quarkus augmentation — a phase that runs after compilation and is easily missed by the upstream committer who doesn't run downstream test suites.

### Alternatives considered
- Could add both `@Default` and the custom qualifier to the bean — but this defeats the purpose of the qualifier (callers should be explicit about which backend they want).
