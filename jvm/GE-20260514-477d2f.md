## Symptom

`@QuarkusTest` fails at boot with:

```
jakarta.persistence.PersistenceException: Unable to build Hibernate SessionFactory
Caused by: org.hibernate.query.NamedQueryValidationException: Errors in named queries:
  Suppressed: org.hibernate.query.sqm.UnknownPathException:
    Could not resolve attribute 'scopeKey' of 'io.casehub.ledger.runtime.model.ActorTrustScore'
```

The error also appears as `ERROR` log lines before the exception:
```
HHH90003001: Error in named query: ActorTrustScore.findGlobalByActorId:
  org.hibernate.query.sqm.UnknownPathException: Could not resolve attribute 'scopeKey'
```

Hibernate throws this at `sessionFactoryCreated()` time — before any test runs.

## Root Cause

A cached `casehub-ledger:0.2-SNAPSHOT` JAR contains `@NamedQuery` annotations on `ActorTrustScore` that reference a field (`scopeKey`) not yet present on the entity class. This happens when the `@NamedQuery` strings were committed to the repo but the corresponding entity field wasn't added yet (or vice versa), and the local Maven cache holds a stale artifact version.

Hibernate 6 validates ALL named queries at `SessionFactory` build time via `SessionFactoryObserverForNamedQueryValidation` — a Quarkus Hibernate ORM extension class. Unlike Hibernate 5, there is no documented configuration key to skip this validation.

**Attempts that don't work:**
- `quarkus.hibernate-orm.validate-named-query=false` — not a valid Quarkus config key
- `quarkus.hibernate-orm.unsupported-properties."hibernate.query.validate_named_queries"=false` — also ineffective; the Quarkus observer runs after Hibernate's own validation hooks

## Fix

Install the affected artifact from local source:

```bash
mvn install -DskipTests -q --batch-mode -f /path/to/ledger/pom.xml
```

This overwrites the stale SNAPSHOT JAR in `~/.m2/` with a consistent version where named queries and entity fields match.

If local source is unavailable, force Maven to re-download the SNAPSHOT:
```bash
mvn dependency:resolve -U  # -U forces SNAPSHOT refresh
```

## Why This Is Non-Obvious

1. The error log says `HHH90003001` and mentions a named query — nothing suggests the problem is a stale JAR artifact. A developer investigates entity mappings, not Maven cache consistency.
2. The entity class on disk is correct; the problem is inside the compiled JAR from cache. Reading source code shows no issue.
3. The two Quarkus/Hibernate property overrides that should disable validation are both silently ignored — the validation happens in a Quarkus extension observer, not in Hibernate's standard configuration layer.
4. Tests pass on CI (which always downloads fresh artifacts) but fail locally on cached SNAPSHOTs — intermittent environment-specific failure with no clear signal.
