## Context

Testing a code path where "WireMock returns 404 by default (no stubs configured)" is
the intended trigger — e.g. testing a 404-as-noop branch in a method that calls an HTTP
endpoint, catches 404, and returns 0 instead of propagating the error.

The test registers no WireMock stubs. WireMock's default is to return 404 for all
unmatched requests. The method under test should: make the GET, catch 404, return 0.

## The problem

A test like this looks complete but has a silent gap:

```java
@Test
void getEpisodes_404_returns_zero() {
    // No stubs — WireMock returns 404 by default for all unmatched requests.
    final int count = service.eraseEntity(ENTITY, TENANT);
    assertEquals(0, count);
    wireMock.verify(0, deleteRequestedFor(anyUrl()));
}
```

The test asserts:
- The method returns 0 ✅
- No DELETE was called ✅

But it does NOT assert that the GET was actually attempted. If the REST client silently
returns null (or is misconfigured to not call WireMock at all), the method short-circuits
for a different reason and the test still passes — proving nothing about the 404-handling
code path.

## Fix — add positive GET verification

```java
@Test
void getEpisodes_404_returns_zero() {
    // No stubs — WireMock returns 404 by default for all unmatched requests.
    // eraseGroup() catches GET 404 → returns 0; DELETE is never called.
    final int count = service.eraseEntity(ENTITY, TENANT);
    assertEquals(0, count);
    // Positive: confirm the GET was actually attempted (not short-circuited before HTTP)
    wireMock.verify(1, getRequestedFor(urlPathEqualTo("/episodes/" + GROUP_ID)));
    // Negative: confirm DELETE was not called
    wireMock.verify(0, deleteRequestedFor(anyUrl()));
}
```

The `wireMock.verify(1, getRequestedFor(...))` proves the REST call reached WireMock and
received the 404. Without it, the test cannot distinguish between "404 handled correctly"
and "HTTP call never made."

## Why this is non-obvious

The two negative assertions (`count == 0`, `0 DELETE calls`) make the test look complete.
The positive assertion is easy to omit because the test's success condition is a return
value (0) and an absence (no DELETE). Adding a positive "the GET was called" assertion
requires actively thinking about the precondition for the 404-handling path to fire —
not just the outcome.

This applies whenever WireMock's default-404 behaviour (no stubs) is used intentionally
as the trigger for a code path, rather than as a convenience for irrelevant requests.
