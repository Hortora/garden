### Symptom
After extracting per-player mutable state into a `PlayerState` class and calling `clear()`
on it at game reset, enemy unit production silently stops. No exception, no log, no error.
The `EnemyBehavior` appears to train units (it pushes `TrainIntent`s) but nothing is trained.

### Context
Occurred during a refactor that moved `supply`, `supplyUsed`, `minerals`, `vespene` from
scattered fields into a `PlayerState` record. The new `PlayerState.clear()` resets all fields
to zero. The friendly player has supply seeded in `reset()`, but the enemy `PlayerState`
was never given supply — it stayed at zero.

Meanwhile `EnemyBehavior.tickProduction()` pre-deducted minerals before pushing `TrainIntent`,
then `handleTrain()` (which processes the intent via `applyIntent(Intent, PlayerState)`) also
deducted minerals — double deduction, also silent.

### Root cause
Two independent bugs compound:
1. **supply=0**: `handleTrain()` checks `supplyUsed + supplyCost > supply`. With `supply=0`
   and any `supplyCost > 0`, this is always true → intent rejected silently.
2. **Double deduction**: EnemyBehavior deducted minerals before queuing the intent, and
   `handleTrain()` deducted again when draining the queue. No error — only insufficient
   minerals on the second pass, which also silently rejects.

### Fix
1. Seed enemy supply/vespene in `reset()` after clearing:
   ```java
   enemy.clear();
   enemy.supply    = 200;   // effectively unlimited for emulated mode
   enemy.vespene   = 9999;
   ```
2. Remove the pre-deduction from the behavior layer. The intent queue is the contract;
   resource management is the consumer's (handleTrain's) responsibility:
   ```java
   // EnemyBehavior.tickProduction() — remove:
   // enemy.minerals -= cost;   ← DELETE THIS
   queue.add(new TrainIntent(tag, unit));
   ```

### Why this is non-obvious
When refactoring symmetric state, it's easy to seed the "primary" player (friendly) while
forgetting the "secondary" (enemy) needs its own initialization. The double-deduction bug is
harder to spot because the behavior layer and the intent consumer both look individually
correct — the failure only emerges from their composition. No assertion or log reveals either
bug; production silently stops.
