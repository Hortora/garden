## Symptom

CDI `@ObservesAsync` observer method never fires. No error, no warning — events are dispatched via
`Event.fireAsync()` and return a completed `CompletionStage`, but the observer method is never called.

## Context

Writing a CDI bean that should observe an async CDI event. The annotation is placed on the method
itself, following the same pattern as `@Observes`:

```java
// WRONG — @ObservesAsync on the method is silently ignored
@ObservesAsync
public void onRequest(ChannelAgentRequest request) {
    // never fires
}
```

## Root Cause

Unlike `@Observes` (which also annotates the parameter in standard CDI, but Quarkus supports on the
method via extension), `@ObservesAsync` in Quarkus CDI **must be placed on the parameter**, not the
method. Method-level placement compiles without error but is silently ignored — the method is not
registered as an observer.

## Fix

Move the annotation to the event parameter:

```java
// CORRECT — @ObservesAsync on the parameter
public void onRequest(@ObservesAsync ChannelAgentRequest request) {
    // fires correctly
}
```

## Why this is non-obvious

`@Observes` is typically documented as a parameter annotation in the CDI spec, but Quarkus's Arc
extension also accepts it on the method for synchronous observers. Developers who have used
`@Observes` on methods in Quarkus naturally try the same for `@ObservesAsync` — and get silence
with no diagnostic.

## Constraints

Specific to Quarkus Arc's CDI implementation. The Jakarta CDI spec treats `@ObservesAsync` as a
parameter annotation; Quarkus may not extend this to method-level placement unlike `@Observes`.

## Invalidation triggers

Revisit if Quarkus Arc extends method-level `@ObservesAsync` support in a future release.
