## Symptom

A rendered-prompt cache returns stale results when only `resources` or `situationalContext` change, even though the hash inputs were recomputed. Changing these fields produces the same cache key as before.

## Context

An LLM renderer for agent system prompts was designed with two operations:

1. `buildDescriptorPayload(descriptor)` â€” produces an `ObjectNode` for the LLM call (goal, vocab-resolved labels, capabilities)
2. `buildContextPayload(context)` â€” initially included only `goal` (also sent to the LLM)

The cache key was `sha256(descriptorPayload) + ":" + sha256(contextPayload) + ":" + format`. Because `resources` and `situationalContext` were not sent to the LLM (they are rendered structurally in Stage 3), they were omitted from `buildContextPayload`, and therefore not included in the hash.

The bug: `resources` and `situationalContext` DO appear in the final rendered output (in structural sections like `## Resources` and `## Context`). Changing a resource URI would return the exact same cache key â€” the old cached content, with the wrong URI.

The failure was caught by `SystemPromptRendererTest` in the examples module when `different_context_produces_different_context_hash` failed.

## Root cause

Conflating "what the LLM needs to produce prose" with "what affects the final rendered output." These are two distinct sets. The cache key must be derived from the latter (everything that affects output), not the former (what the LLM input carries).

## Fix

Split the responsibility:
- `buildContextPayload(context)` â†’ all context fields that affect rendered output (`goal`, `resources`, `situationalContext`). Used for hashing.
- `buildLlmPayload(descriptorNode, contextNode)` â†’ merges descriptor + goal only (resources and situationalContext excluded). Used for the LLM call.

```java
// For cache key â€” includes everything that affects rendered output
ObjectNode contextNode = buildContextPayload(context);  // goal + resources + situationalContext
String contextHash = fingerprint(contextNode.toString());

// For LLM call â€” only what the LLM needs for prose generation
ObjectNode llmPayload = buildLlmPayload(descriptorNode, contextNode);  // goal only
enrichment = enrichmentStep.enrich(llm, llmPayload);
```

## Why this is non-obvious

The split between "LLM input" and "rendered output" is easy to reason about for Stage 2 (the LLM call), but the cache key must be derived from the full output contract â€” including every field that Stage 3 (structural rendering) reads, regardless of whether Stage 2 ever sees them. When the two stages share an input payload, this is invisible. When they don't, the cache key must be based on the union of all inputs to all stages.
