## Symptom

Cached renders serve stale capability data after a field (e.g. `qualityHint`, `costHint`) on the source record changes. The field is correctly updated in the source, and cache misses render the new value, but cache hits return the old value indefinitely.

## Context

A renderer pipeline with two paths:
1. **LLM enrichment path:** `buildDescriptorPayload(descriptor, format)` serialises selected fields to an `ObjectNode`. The cache key hash is derived from this node. For LLM noise reduction, certain numeric fields are excluded from some format-specific payloads.
2. **Structural assembly path:** `assembleA2aCard(descriptor)` reads capability fields (e.g. `qualityHint`, `latencyHintP50Ms`, `costHint`, `epistemicDomains`) **directly from the `AgentDescriptor` record** — not from `s1.descriptorNode()`.

When a reviewer proposed excluding `costHint` from all descriptor payloads (to keep LLM inputs clean), the fix looked correct at the payload-builder level. But because `assembleA2aCard()` reads `costHint` from the descriptor record directly, the field still affects A2A_CARD output — and since it's no longer in the hash, changing `costHint` produces the same cache key. Cache hits serve the old value forever.

## Root cause

The structural assembly path reads from the original source record as its authoritative input, not from the hashed payload. A field excluded from the payload for LLM cleanliness is invisible to the hash but not to structural rendering. The hash is an incomplete representation of what determines the output.

## Fix

**Any field read by any assembly step (structural or LLM) must appear in the appropriate format's hash payload.** For the LLM enrichment path, fields can still be excluded from what's *sent* to the LLM — but they must remain in what's *hashed*.

Concretely: in a format-discriminated renderer, even if `costHint` is dead data to the LLM enrichment step (no LLM prompt instruction references it), it must be in the A2A_CARD descriptor payload because `assembleA2aCard()` renders it structurally. The hash must cover the union of all inputs to all assembly steps for that format.

See also: GE-20260528-e9ed9f — general principle (hash all output-affecting context, not just LLM input fields).

## Why this is non-obvious

The payload builder looks authoritative — it controls the LLM input and is the natural place to define "what this format needs." The fact that a separate structural assembly step reads from the original source record (bypassing the payload entirely) is not visible from the payload builder's perspective. The trap only becomes apparent when you trace the full path from source record → hash → output for every field that any assembly step touches.

## Why this fix

Keeping dead-data fields in the hash payload is the minimal correct fix: it preserves cache correctness without restructuring the assembly path. The alternative — making structural assembly read only from the payload node — would force the payload to carry all fields for all formats, eliminating the LLM-noise-reduction benefit of format-discriminated payloads. Accepting dead data in the hash is the cheaper price.
