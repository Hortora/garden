## Symptom

Compilation error when trying to use a functional fold function (returns new state)
as the accumulator for `Multi.collect().in()`.

## Context

Implementing a stateful fold over a `Multi<T>` where the accumulation function is
pure (returns a new state object rather than mutating the container), similar to
`Stream.reduce()` or Reactor's `Flux.reduce()`.

## What was tried (didn't work)

```java
// ChannelProjection.apply(S state, Message msg) -> S   (returns new state)
multi.collect().in(projection::identity, projection::apply);
// ↑ Does not compile — apply is BiFunction<S, T, S>, but collect().in() requires BiConsumer<S, T>
```

## Root cause

`Multi.collect().in(Supplier<C> container, BiConsumer<C, T> accumulator)` uses a
**mutable container pattern** — the accumulator is a `BiConsumer` (void return, mutates C).
This is analogous to `java.util.stream.Stream.collect(Collector)`, not `Stream.reduce()`.
Mutiny does not have a built-in `reduce(seed, BiFunction<S, T, S>)` equivalent for `Multi`
in the same way Reactor's `Flux.reduce()` works.

## Fix

Wrap the immutable fold function in a private mutable accumulator class:

```java
private static final class FoldAcc<S> {
    S state;
    Long lastId;
    FoldAcc(S state, Long lastId) { this.state = state; this.lastId = lastId; }
}

return multi.collect().in(
    () -> new FoldAcc<>(initialState, cursor),
    (acc, msg) -> {
        acc.state = applyFn.apply(acc.state, mapper.toView(msg));
        acc.lastId = msg.id;
    })
.map(acc -> new Result<>(acc.state, acc.lastId));
```

## Why this is non-obvious

Java developers think of "fold over a stream" as `reduce(seed, BiFunction)` — a well-known
FP pattern. Mutiny's `collect().in()` follows the Java `Collector` pattern instead (mutable
container + `BiConsumer`). The distinction is invisible until the compile error appears, and
the error says "incompatible types: BiFunction cannot be converted to BiConsumer" — which
points at the symptom but not the solution.
