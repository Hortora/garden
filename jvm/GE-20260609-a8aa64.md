### Symptom

Config override applied via a decorator's `build()` method silently doesn't work. No error, no warning. The annotation value is used instead of the config value.

### Context

When wrapping a fluent builder (e.g. `LoopAgentService<T>`) with a decorator that overrides `build()` to apply config values, every fluent method in the decorator must return `this` — not the delegate's return value.

Upstream code chains fluently:
```java
var builder = loopBuilder(agentServiceClass)   // returns wrapper
    .subAgents(...)                             // fluent
    .maxIterations(annotation.maxIterations()); // fluent
builder.build();                                // should hit wrapper
```

### What was tried (didn't work)

Standard delegation: `delegate.maxIterations(value); return delegate.maxIterations(value);` — returns the delegate, chain escapes.

### Root cause

Fluent methods return the builder instance for chaining. If the decorator delegates and returns the delegate's return value, subsequent calls in the chain execute on the real builder, not the wrapper. The wrapper's `build()` override never fires.

### Fix

Every fluent method in the decorator must delegate the operation AND return `this`:

```java
@Override
public LoopAgentService<T> maxIterations(int maxIterations) {
    delegate.maxIterations(maxIterations);
    return this;  // NOT: return delegate.maxIterations(maxIterations);
}
```

For `LoopAgentService<T>`: 16 delegate-and-return-`this` methods + 1 `build()` with override.

### Why this is non-obvious

The incorrect version compiles and runs without error. The decorator looks correct — it delegates every method. The only symptom is that the config override doesn't apply, which manifests as "the annotation value is used" — easy to misattribute to config wiring rather than a chain escape.
