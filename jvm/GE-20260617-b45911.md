## The technique

Use `LinkedBlockingQueue.poll(heartbeatMs, TimeUnit.MILLISECONDS)` as the central blocking primitive in a virtual-thread SSE handler. A single `null` return (poll timeout) drives keepalives; a message drives SSE events; `sink.isClosed()` at loop top handles orphans; a deadline check handles max duration. Three concerns, one loop.

```java
@RunOnVirtualThread
public void streamTask(@PathParam("id") String taskId,
                       @Context SseEventSink sink,
                       @Context Sse sse) throws Exception {

    LinkedBlockingQueue<OutboundMessage> queue = new LinkedBlockingQueue<>();
    a2aBackend.registerStream(corrId, queue::offer);  // consumer is just queue::offer

    long heartbeatMs = 15_000;
    long deadline = System.currentTimeMillis() + 1_800_000;

    try {
        while (true) {
            if (sink.isClosed()) break;              // orphan: client disconnected
            long remaining = deadline - System.currentTimeMillis();
            if (remaining <= 0) break;               // max duration exceeded

            OutboundMessage msg;
            try {
                msg = queue.poll(Math.min(heartbeatMs, remaining), MILLISECONDS);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }

            if (msg == null) {                       // poll timeout → keepalive
                sink.send(sse.newEventBuilder().name("keepalive").data("").build());
                continue;
            }

            // process msg, send SSE event, break if terminal
        }
    } catch (Exception e) {
        LOG.warn("SSE stream error", e);
    } finally {
        a2aBackend.deregisterStream(corrId, consumer);
        if (!sink.isClosed()) sink.close();
    }
}
```

## Why this is non-obvious

The typical approach for SSE keepalives uses a separate `@Scheduled` bean that sweeps active connections, requiring a richer registry type (storing sinks + registration timestamps) and global scheduler state. For orphan detection, most implementations rely on lazy cleanup: the next `post()` call detects a broken sink. For max-duration, a separate scheduled task is needed.

The `BlockingQueue.poll(timeout)` pattern collapses all three into one structure with no scheduler, no global state beyond the queue, and no concurrent SSE writes (all writes happen on the single virtual thread running the loop).

## How it works

- **Keepalive:** `poll()` blocks for `heartbeatMs`, returns null → send keepalive → continue
- **Orphan detection:** `sink.isClosed()` check at the top of every iteration; at most one heartbeat interval of latency
- **Max duration:** `remaining = deadline - now`; if `≤ 0` → break; also passed as `Math.min(heartbeatMs, remaining)` to `poll()` so the loop exits precisely at the deadline
- **Consumer simplicity:** the event producer calls `queue.offer(msg)` — a one-liner; no sink references, no JAX-RS context in the producer
- **No concurrent writes:** all `sink.send()` calls happen on the virtual thread running the loop; the producer only touches the queue

## Applicability

Requires Java 21 virtual threads (`@RunOnVirtualThread` in Quarkus) so the blocking poll doesn't occupy OS threads. On classic thread pools, holding a thread per connection for 30 minutes is expensive. Virtual threads make this natural.
