### Symptom
`mvn deploy` to GitHub Packages fails with `403 Forbidden` on a merged PR. The workflow declares `permissions: packages: write`. Previous PRs published successfully.

### Context
PR is from a fork (`user/repo` → `org/repo`). The workflow uses `GITHUB_TOKEN` for the deploy step. The publish step is conditioned on `pull_request && merged == true`.

### What was tried (didn't work)
- Verifying workflow permissions block — correctly declares `packages: write`
- Checking GitHub Packages settings — repository allows publishing

### Root cause
GitHub silently downgrades `GITHUB_TOKEN` permissions for `pull_request` events originating from forks. The token gets `packages: read` regardless of the workflow's `permissions` declaration. This is a documented GitHub security measure to prevent fork PRs from publishing malicious packages. The CI job log shows the actual permissions under "GITHUB_TOKEN Permissions" in the "Set up job" step.

Previous PRs worked because they were from same-repo branches (not forks), which get the full declared permissions.

### Fix
Use `push` trigger instead of `pull_request closed+merged` for the publish step. When a PR is merged, GitHub fires both a `pull_request(closed)` event AND a `push` event to the target branch. The `push` event gets full declared permissions including `packages: write`.

```yaml
on:
  push:
    branches: [main]    # fires on merge — full permissions
  pull_request:
    types: [opened, synchronize, reopened]  # removed 'closed'
    branches: [main]
```

Publish/downstream steps condition on `if: github.event_name != 'pull_request'`.

### Why this is non-obvious
The 403 appears only for fork PRs. Same-repo PRs work fine, creating a false sense that the workflow is correct. The permission downgrade is silent — no warning before the deploy step fails.
