When a Quarkus deployment build step needs to detect annotations from an optional library
(e.g., SmallRye FaultTolerance's `@Retry`, Jakarta Transactions' `@Transactional`),
use `DotName.createSimple("fully.qualified.AnnotationName")` instead of importing the
annotation class. This avoids adding the optional library as a required compile dependency
of the deployment module.

### Convention

```java
// ✅ String DotName — no compile dependency on SmallRye FaultTolerance
private static final DotName RETRY =
        DotName.createSimple("org.eclipse.microprofile.faulttolerance.Retry");

// ❌ Direct import — forces SmallRye FaultTolerance as a required dep
import org.eclipse.microprofile.faulttolerance.Retry;
// ... annotation.name().equals(Retry.class.getName())
```

The Jandex index contains annotation names as strings regardless of whether the annotation
class is on the deployment classpath. String DotNames work identically at build time.

### When to use

Use string DotNames for any annotation where the library is:
- An optional Quarkus extension (SmallRye Fault Tolerance, OpenTelemetry, Hibernate, etc.)
- A user's own library that may or may not be on the classpath

### Alternative

Import the annotation class directly — valid when the library is already a required
dependency of the deployment module (e.g., Jakarta CDI annotations, core Quarkus APIs).

### Why this matters

If you import `@Retry` from SmallRye, the deployment module gains a hard compile dep on
SmallRye Fault Tolerance. Any project using your extension will then transitively require
SmallRye — even if they don't use `@Retry`. String DotNames keep the dependency truly
optional, matching the documented Quarkus extension development guidelines.
