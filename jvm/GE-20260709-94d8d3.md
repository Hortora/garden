### Symptom

Compilation error when trying to copy vectors from a scrolled `RetrievedPoint` into a new `PointStruct` for upsert:

```
incompatible types: cannot infer type arguments for java.util.HashMap<>
    reason: inference variable V has incompatible bounds
      equality constraints: io.qdrant.client.grpc.Points.Vector
      lower bounds: io.qdrant.client.grpc.Points.VectorOutput
```

### Context

Vector enrichment — scrolling existing points to add missing sparse vectors (SPLADE/BM25) while preserving existing dense vectors. The natural approach is to read existing vectors, add new ones to the map, and upsert the modified point.

### What was tried (didn't work)

```java
var existingVectors = point.getVectors().getVectors().getVectorsMap();
// Returns Map<String, VectorOutput> — not Map<String, Vector>
Map<String, Vector> newVectors = new HashMap<>(existingVectors); // compile error
newVectors.put("sparse", VectorFactory.vector(vals, idxs));
// Cannot mix VectorOutput (from read) with Vector (for write)
```

### Root cause

Qdrant's gRPC proto defines two distinct vector types: `Vector` (used in `PointStruct` for writes/upserts) and `VectorOutput` (used in `RetrievedPoint` from scroll/query results). They are structurally similar but separate proto messages with no shared interface or conversion method.

### Fix

Use `updateVectorsAsync` with `PointVectors` instead of full upsert. This updates only the specified named vectors without needing to read or copy existing vectors.

```java
List<PointVectors> enrichBatch = new ArrayList<>();

for (var point : response.getResultList()) {
    Map<String, Vector> newVectors = new HashMap<>();

    // Only add the NEW vectors — don't touch existing ones
    if (needsSplade) {
        newVectors.put(config.spladeVectorName(),
            VectorFactory.vector(vals, idxs));
    }
    if (needsBm25) {
        newVectors.put(config.bm25VectorName(),
            VectorFactory.vector(Document.newBuilder()
                .setText(expanded).setModel(model).build()));
    }

    if (!newVectors.isEmpty()) {
        enrichBatch.add(PointVectors.newBuilder()
            .setId(point.getId())
            .setVectors(VectorsFactory.namedVectors(newVectors))
            .build());
    }
}

client.updateVectorsAsync(collection, enrichBatch).get();
```

### Why this is non-obvious

The type mismatch is only visible at compile time — there is no runtime hint. The proto field names are identical (`getVectors().getVectors().getVectorsMap()`), and `VectorOutput` sounds like it should be assignable to `Vector`. The fix — using `updateVectorsAsync` instead of full upsert — is a different API entirely, not discoverable from the error message.
