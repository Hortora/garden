## Await CDI fireAsync() delivery in @ConsumeEvent reactive chain

### Context

A Quarkus @ConsumeEvent handler returns Uni<Void>. Inside the chain, lifecycleEvents.fireAsync(event)
is called inside .invoke(). The CompletionStage is discarded. @ObservesAsync observers may run AFTER
the handler's Uni completes.

### The technique

Split the terminal .invoke() into two steps:

```java
// Before — CompletionStage discarded
.invoke(() -> {
    eventBus.publish(addr, payload);
    lifecycleEvents.fireAsync(event); // discarded
});

// After — CDI delivery awaited in .chain()
.invoke(() -> {
    eventBus.publish(addr, payload); // still fire-and-forget
})
.chain(() ->
    Uni.createFrom()
        .completionStage(() -> lifecycleEvents.fireAsync(event))
        .onFailure()
        .recoverWithItem(t -> { LOG.warnf(t, "observer failed"); return null; })
        .replaceWithVoid())
```

The handler's Uni does not complete until all @ObservesAsync observers have run.
onFailure().recoverWithItem() prevents observer exceptions from breaking the handler.

### Why this is non-obvious

.invoke() reads as a side-effect callback but is fire-and-forget for CompletionStage —
Mutiny does not chain off it. Moving to .chain() with completionStage() makes CDI
delivery part of the reactive pipeline.

Use the Supplier form completionStage(() -> fireAsync(...)) — lazy, ensures fireAsync()
is called at subscription time (correct thread context), not at chain construction time.
