### Symptom

File deletions are reported as `MODIFY` events instead of `DELETE` when `DirectoryWatcher` is configured with `fileHashing(false)`.

### Context

When building a `DirectoryWatcher` with `.fileHashing(false)` for performance (skipping per-event file hash computation), macOS FSEvents does not distinguish between content changes and deletions at the directory level. The library needs file hashing to determine whether a file was deleted (hash becomes null) vs modified (hash changed).

### What was tried (didn't work)

- `fileHashing(false)` for consistency with mtime-based cursor model — DELETE events misclassified as MODIFY

### Root cause

Without file hashing, the library cannot reliably distinguish between MODIFY and DELETE events on macOS. FSEvents reports directory-level changes; the library infers the event type by checking if the file still exists and comparing hashes. With hashing disabled, this inference breaks.

### Fix

Use the default `fileHashing(true)` — the library correctly classifies all event types:

```java
DirectoryWatcher.builder()
    .path(rootDir)
    // do NOT set .fileHashing(false)
    .listener(this::onRawEvent)
    .build();
```

The cost is computing file hashes per event, which is acceptable for document corpus use cases.

### Why this is non-obvious

The `fileHashing` option is documented as a deduplication feature — "reduces duplicate or useless events." Nothing in the API suggests it also affects event type classification. A developer optimising for performance would reasonably disable it without expecting DELETE events to break.
