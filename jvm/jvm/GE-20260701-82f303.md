## Symptom

`IdentityCacheInvalidator.onKeyRotated()` runs on every `AgentKeyRotatedEvent` but the SCIM provider cache is never invalidated. No error, no log — the `instanceof` check silently returns false and the method exits.

## Context

`ActorDIDProvider` is an SPI with multiple implementations qualified by `@ActorDIDSource`. The platform-identity library provides a `CompositeActorDIDProvider` that injects all `@ActorDIDSource`-qualified providers and delegates to them. `@Inject ActorDIDProvider` (with implicit `@Default`) resolves to this composite.

## What was tried (didn't work)

1. `if (actorDIDProvider instanceof ScimActorDIDProvider scim)` — always false; the CDI proxy wraps the composite, not the concrete provider
2. `ClientProxy.unwrap(actorDIDProvider) instanceof ScimActorDIDProvider` — still false; unwrapping removes the proxy layer but reveals `CompositeActorDIDProvider`, not `ScimActorDIDProvider`

## Root cause

CDI bean resolution for `@Inject ActorDIDProvider` returns whichever bean has `@Default` qualifier (or is selected via `selected-alternatives`). When `CompositeActorDIDProvider` exists, it wins — it implements `ActorDIDProvider` without a custom qualifier. The concrete providers (`ScimActorDIDProvider`, `ConfiguredActorDIDProvider`) have `@ActorDIDSource` qualifier and are only visible to the composite's `Instance<ActorDIDProvider>` injection.

The `instanceof` check assumes the injected bean IS the concrete provider. With a composite wrapper, no amount of proxy unwrapping will reveal it.

## Fix

Call `invalidate()` via the `ActorDIDProvider` SPI default method — `CompositeActorDIDProvider.invalidate()` propagates to all registered providers:

```java
void onKeyRotated(@Observes final AgentKeyRotatedEvent event) {
    actorDIDProvider.invalidate(event.actorId());
}
```

## Why this is non-obvious

`ClientProxy.unwrap()` is the standard Quarkus pattern for working around proxy `instanceof` issues. Here it correctly removes the CDI proxy, but the result is still not the expected concrete type — the composite layer sits between the proxy and the provider. The failure is completely silent (no exception, no log), so it only surfaces when downstream behaviour (cache not clearing) is noticed.
