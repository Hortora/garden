### Symptom

A test expecting `Duration.parse("P1D")` to throw `DateTimeParseException` — to verify that a day-based period is an invalid duration format — passes without throwing. The catch block and validation branch are never reached.

### Context

Writing input validation for a YAML field that accepts ISO-8601 duration strings (e.g. `PT24H`). An issue description cited `P1D` as an example of an invalid format. It is not invalid in Java.

### Root cause

`Duration.parse()` accepts the full ISO-8601 extended duration format, including day components: `PnDTnHnMnS`. `"P1D"` parses successfully as 86400 seconds. Days are treated as exactly 24-hour blocks (not calendar days). The output of `Duration.ofDays(1).toString()` is `PT86400S`, which reinforces the false impression that Duration is purely time-based.

### Fix

Use a string that is genuinely outside ISO-8601 duration format to test for parse failure:

```java
// ✅ Actually invalid — bare notation without ISO prefix
Duration.parse("1h");      // throws DateTimeParseException
Duration.parse("24 hours"); // throws DateTimeParseException
Duration.parse("invalid"); // throws DateTimeParseException

// ❌ Looks invalid but parses successfully
Duration.parse("P1D");  // returns PT86400S — no exception thrown
Duration.parse("P7D");  // returns PT604800S — no exception thrown
```

### Why this is non-obvious

`Duration.toString()` always produces `PT`-prefixed output (`PT24H`, `PT1H30M`), so developers assume only `PT`-prefixed strings are valid inputs. The Java naming ("Duration" vs "Period") implies a purely time-based type. The ISO-8601 day-in-duration support is present in the Javadoc but easy to miss — most readers scan for the `PTnHnMnS` example and stop there.
