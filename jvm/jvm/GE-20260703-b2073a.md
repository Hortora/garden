### Symptom
`NodeProvisioner.deprovision()` receives a `DesiredNode` whose `spec()` does not
match any known `NodeSpec` implementation. `instanceof` checks for all domain spec
types fall through silently, causing the provisioner to return `Failed("Unknown spec type")`.

### Context
When `TransitionPlanner.plan()` detects orphaned nodes — nodes present in `ActualState`
but absent from the desired `DesiredStateGraph` — it creates synthetic `DesiredNode`
instances for the removal list:

```java
new DesiredNode(nodeId, NodeType.of("unknown"), new UnknownSpec(), false)
```

`UnknownSpec` is a `private static class` inside `TransitionPlanner` (line 131). It
implements `NodeSpec` but is inaccessible to any code outside the planner.

### What was tried (didn't work)
- Checking `instanceof UnknownSpec` — compile error, class is private
- Checking `node.type().equals(NodeType.of("unknown"))` — works but couples to an
  undocumented string convention
- Checking `!(spec instanceof KnownSpec1 || spec instanceof KnownSpec2 || ...)` — 
  works but fragile if new spec types are added

### Root cause
`TransitionPlanner` needs to create `DesiredNode` records for orphaned nodes it only
knows by `NodeId` (from `ActualState`). Since it has no access to the original spec,
it uses a private placeholder. The placeholder's private visibility means provisioners
cannot pattern-match on it.

### Fix
Let all known-type `instanceof` checks fall through, then infer the entity type from
the `NodeId` naming convention established by the domain's `GoalCompiler`:

```java
@Override
public DeprovisionResult deprovision(DesiredNode node, DeprovisionContext context) {
    var spec = node.spec();
    if (spec instanceof CellSpec) { /* handle cell */ }
    if (spec instanceof UnitSpec) { /* handle unit */ }
    if (spec instanceof ScoutSpec) { /* handle scout */ }
    if (spec instanceof ZoneSpec) { /* handle zone */ }
    
    // Orphaned node — spec is TransitionPlanner's private UnknownSpec.
    // Infer entity type from NodeId naming convention.
    var idValue = node.id().value();
    if (idValue.startsWith("unit-")) {
        world.removeUnit(node.id());
        return new DeprovisionResult.Success();
    }
    if (idValue.startsWith("scout-")) {
        world.removeScout(node.id());
        return new DeprovisionResult.Success();
    }
    // Cells and zones: no-op deprovision
    return new DeprovisionResult.Success();
}
```

### Why this is non-obvious
The `DesiredNode` record always carries a `NodeSpec` — there is no null case. A
developer implementing `NodeProvisioner.deprovision()` will write `instanceof` checks
for every known spec type and assume the chain is exhaustive. The private `UnknownSpec`
is invisible in the API — there's no Javadoc, no sealed-type hierarchy hint, and no
compile-time signal that an unmatched case exists. The failure is silent: the provisioner
returns `Failed` and the orphaned node lingers in the world state.

### Why this fix
Making `UnknownSpec` public (or providing a public marker in the API) would be the
correct fix. The NodeId-prefix inference is a workaround that couples the provisioner
to the GoalCompiler's naming convention. It works because domain provisioners already
know their GoalCompiler's conventions, but it's fragile if conventions change.
