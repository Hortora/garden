## Symptom

Adding `of(String tenancyId, String... segments)` alongside existing `of(String... segments)` compiles without error. Existing call sites like `of("casehubio", "devtown")` silently resolve to the new overload, reinterpreting the first argument as `tenancyId` and reducing the path by one segment. No compilation failure, no warning — the code silently does the wrong thing.

## Context

Occurs when evolving a factory method to add a new leading parameter of the same type as the varargs elements. Java's overload resolution prefers the more specific signature (the one with the explicit leading parameter) over the pure varargs signature.

## What was tried (didn't work)

Keeping both overloads — the old `of(String... segments)` for backward compatibility and the new `of(String tenancyId, String... segments)`. Expected a compile error at call sites. Got silent reinterpretation instead.

## Root cause / fix

Java resolves `of("a", "b")` against both `of(String...)` and `of(String, String...)` — the second is more specific, so it wins. No ambiguity error is raised.

**Fix:** Remove the old varargs overload entirely. Force callers to use `Path.of(...)` explicitly for the segments, producing a compile-time break instead of a silent semantic change:

```java
// Before (dangerous — both compile, different semantics)
public static SettingsScope of(String... segments) { ... }
public static SettingsScope of(String tenancyId, String... segments) { ... }

// After (safe — old call sites fail to compile)
public static SettingsScope of(String tenancyId, Path scope) { ... }
```

## Why this is non-obvious

Developers expect that changing a method signature will break callers at compile time. With varargs, the compiler silently picks the "best match" — the new overload absorbs the old call sites without any diagnostic. The bug only surfaces at runtime when the first segment is treated as a different parameter.
