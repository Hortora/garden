### Symptom
`StackOverflowError` at Quarkus startup — infinite recursion in CDI bean creation.
Stack trace shows `DefaultStrategyResolver.<init>` → `Instance.spliterator` → bean instantiation → `DefaultStrategyResolver.<init>`.

### Context
When retrofitting SPIs to extend `NamedStrategy`, some implementations (e.g. `RoundRobinAssignmentStrategy`) both implement `NamedStrategy` AND need to resolve another strategy type via `StrategyResolver`. This creates a cycle:
1. `DefaultStrategyResolver` constructor injects `@Any Instance<NamedStrategy>` and eagerly iterates it (`toList()`)
2. Iterating instantiates all `NamedStrategy` beans, including `RoundRobinAssignmentStrategy`
3. `RoundRobinAssignmentStrategy` constructor injects `StrategyResolver` → back to step 1

### What was tried (didn't work)
Direct constructor injection of `StrategyResolver` — causes the cycle because `DefaultStrategyResolver` is `@Singleton` (no client proxy, eager instantiation).

### Root cause
`DefaultStrategyResolver` is annotated `@Singleton`, not `@ApplicationScoped`. In Quarkus Arc, `@Singleton` beans do NOT get a client proxy — they are eagerly instantiated and their constructor runs immediately. The constructor calls `StreamSupport.stream(strategies.spliterator(), false).toList()` which triggers instantiation of ALL `NamedStrategy` beans, including those that depend on `StrategyResolver` itself.

### Fix
Use `jakarta.inject.Provider<StrategyResolver>` instead of direct `StrategyResolver` injection. `Provider<>` defers instantiation — the bean is created only on first `.get()` call, breaking the cycle.

```java
@Inject
public RoundRobinAssignmentStrategy(
        final Provider<StrategyResolver> strategyResolverProvider,
        final WorkItemsConfig config) {
    this.strategyResolverProvider = strategyResolverProvider;
    this.config = config;
}

private WorkerSelectionStrategy workerStrategy() {
    if (workerSelectionStrategy == null) {
        workerSelectionStrategy = strategyResolverProvider.get()
                .resolve(WorkerSelectionStrategy.class, config.routing().strategy());
    }
    return workerSelectionStrategy;
}
```

### Why this is non-obvious
The `NamedStrategy` contract says "implement this interface and add `id()`" — nothing warns that the implementing bean cannot also inject `StrategyResolver`. The cycle only manifests at runtime startup, not at compile time. And `@ApplicationScoped` beans (which do get client proxies) would NOT have this problem — the `@Singleton` annotation on `DefaultStrategyResolver` is the hidden trigger.

### Alternatives considered
- Making `DefaultStrategyResolver` `@ApplicationScoped` instead of `@Singleton` — would fix the issue systemically but requires changing the platform module, not just the consumer
- Lazy field initialization without `Provider<>` — doesn't help because the cycle is in CDI bean creation, not in field access
