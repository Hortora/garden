## Symptom

`@QuarkusTest` startup fails with a cryptic `DeploymentException`:

```
jakarta.enterprise.inject.spi.DeploymentException: @Transactional will have no effect on method
io.casehub.clinical.service.FooService.bar() because the method is private.
Either remove the annotation from the method, or turn this exception into a simple warning
by setting configuration property 'quarkus.arc.fail-on-intercepted-private-method' to 'false'.
```

The error does NOT appear during `mvn compile` — only during Quarkus augmentation (i.e. `@QuarkusTest` or `mvn quarkus:dev`). A `private @Transactional` method compiles cleanly.

## Context

A service class has two overloads of the same method:

```java
@ApplicationScoped
public class FooService {

    // unit-test path — package-private, no Panache
    @Transactional
    Map<String, Object> process(Event event, Entity entity) { ... }

    // production path — loads entity inside @Transactional
    @Transactional
    private Map<String, Object> process(Event event) {  // ← causes failure
        Entity entity = Entity.findById(event.id());
        return process(event, entity);
    }
}
```

## Root cause

CDI interceptors (including `@Transactional`) work via proxy subclassing — ArC generates a subclass that overrides each intercepted method. Private methods cannot be overridden; ArC detects this at augmentation time and throws `DeploymentException` rather than silently ignoring the annotation.

This is consistent with the CDI 3.0 spec (§7.2): "An interceptor is not invoked for a private, static, or final method." Quarkus ArC validates this eagerly — other CDI implementations may silently ignore private interceptor annotations.

## Fix

Change the visibility from `private` to package-private (no access modifier):

```java
@Transactional
Map<String, Object> process(Event event) {   // ← package-private: intercepted correctly
    Entity entity = Entity.findById(event.id());
    return process(event, entity);
}
```

Package-private methods are still overridable by ArC's generated subclass in the same package. The method remains inaccessible to classes outside the package — the practical protection is the same as `private` for internal helpers.

## Why this is non-obvious

`mvn compile` succeeds silently — the annotation is syntactically valid. The failure only manifests during Quarkus augmentation (CDI proxy generation), which runs on the first `@QuarkusTest` start or `mvn quarkus:dev`. A developer who unit-tests with plain Mockito (`@ExtendWith(MockitoExtension.class)`) will never hit this error — it only surfaces when integration-testing with a real Quarkus context.

The distinction between `private` and `static` methods is not prominent in error messages or docs — both fail, but for different reasons. See GE-20260528-3d3847 for the static-method variant.
