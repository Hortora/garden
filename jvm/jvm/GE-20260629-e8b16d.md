### Symptom
After the first game restart (GameStarted CDI event), the L1→L2 moment detection pipeline goes completely silent. No errors, no logs, no test failures. Events are published but no subscriber receives them.

### Context
EventStreamBus<E> is a generic pub/sub component. Consumers subscribe in @PostConstruct (CDI bean creation). On game restart, a @Observes GameStarted handler calls bus.clear() to "reset state between games."

### What was tried (didn't work)
- Checking L1 event publication — events were being published correctly
- Checking MomentDetectionTask activation — it was activating and running Drools
- Checking Drools rules — rules were correct
- The pipeline appeared to work for the first game but never for subsequent games

### Root cause
EventStreamBus.clear() removes ALL subscriptions (CopyOnWriteArrayList.clear()). Subscribers registered in @PostConstruct never re-subscribe because @PostConstruct only fires once per CDI bean lifecycle. The bus dispatches to an empty subscriber list — silently correct behaviour from the bus's perspective.

### Fix
Do NOT call bus.clear() in lifecycle reset handlers. EventStreamBus is stateless (synchronous dispatch, no buffering) — there is nothing to "reset." Only EventAccumulator (which buffers events for windowed emission) needs clearing between lifecycle resets. Subscriptions are tied to CDI bean lifecycle, not game/case lifecycle.

### Why this is non-obvious
The mental model "clear() resets the component for a new game" is intuitive but wrong for pub/sub. A latest-value store (ScoutingIntelBroker.latest.clear()) correctly resets by clearing cached values. But a pub/sub bus has no cached values — its state IS its subscriptions, and those must survive lifecycle resets.

### Why this fix
The alternative — re-subscribing in the GameStarted handler — creates a CDI observer ordering problem (non-deterministic: bus might be cleared AFTER re-subscription). Removing the clear() call entirely is both simpler and correct.

### Alternatives considered
- Re-subscribe in onGameStarted: rejected — CDI observer ordering is non-deterministic, so the re-subscription can race with the clear()
- Add a resetState() method separate from clear(): rejected as unnecessary complexity — the bus has no state to reset
