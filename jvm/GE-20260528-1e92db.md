## Context

`@PostConstruct` runs after CDI field injection, so `@ConfigProperty`-injected fields are populated at that point. The risk is creating an anonymous class whose `removeEldestEntry` (or similar override) captures the mutable field by reference — if the field is ever mutated after construction, the anonymous class silently uses the new value.

Common pattern when building a bounded LRU cache:

```java
@ConfigProperty(name = "casehub.eidos.renderer.cache-size", defaultValue = "256")
int maxSize;

private Map<String, V> cache;

@PostConstruct
void init() {
    cache = Collections.synchronizedMap(
        new LinkedHashMap<>(16, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<String, V> eldest) {
                return size() > maxSize;  // ⚠️ captures mutable field
            }
        }
    );
}
```

## Fix

Capture the injected value into a `final int` local at the top of `@PostConstruct`, then reference the local in the anonymous class:

```java
@PostConstruct
void init() {
    // Capture injected value into a final local — anonymous class must not capture
    // the mutable field directly.
    final int limit = maxSize;
    cache = Collections.synchronizedMap(
        new LinkedHashMap<>(16, 0.75f, true) {
            @Override
            protected boolean removeEldestEntry(Map.Entry<String, V> eldest) {
                return size() > limit;  // ✅ captures immutable local
            }
        }
    );
}
```

## Why this is non-obvious

`@PostConstruct` guarantees injection has completed before it runs, so `maxSize` is populated — no NPE risk. The bug is subtler: the anonymous class captures the field reference, not the value. If `maxSize` is reassigned after construction (e.g. in a test that sets it directly to override the config), the eviction policy silently changes mid-flight. Using a `final` local makes the capture explicit and prevents this category of bug entirely. In production Quarkus this is low-risk (config values don't change at runtime), but in tests where fields are set directly to bypass config injection, it matters immediately.
