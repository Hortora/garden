## Summary

When a `REQUIRES_NEW` method already queries an entity for one purpose (e.g. resolving
a `causedByEntryId`), reuse that result to drive a second related operation in the same
transaction call — rather than issuing a duplicate query or querying a different entity
that may not be visible (due to the suspended outer tx).

## Context

`LedgerWriteService.record()` runs in `REQUIRES_NEW`. It already calls
`repository.findLatestByCorrelationId(channelId, correlationId)` to set `causedByEntryId`
on the new ledger entry. The returned object (the prior COMMAND's `MessageLedgerEntry`)
is exactly what's needed to write a `LedgerAttestation` — it provides the `ledgerEntryId`
and the `actorId` of the originating COMMAND.

## The technique

Pass the already-retrieved object into the secondary operation instead of re-querying:

```java
repository.findLatestByCorrelationId(ch.id, message.correlationId)
    .ifPresent(prior -> {
        // Primary purpose: causal chain
        entry.causedByEntryId = prior.id;

        // Secondary purpose: write attestation using the same object
        if (ATTESTATION_TYPES.contains(message.messageType)) {
            writeAttestation(ch, prior, message.messageType, resolvedActorId);
        }
    });
```

The `prior` object is committed data (written by a previous `REQUIRES_NEW` call) and
safely visible inside the new transaction. No second query needed.

## Why this is non-obvious

Most developers would reach for a second query (e.g. `commitmentStore.findByCorrelationId()`)
to check a condition before the secondary operation — unaware that the outer tx's changes
are invisible in `REQUIRES_NEW`, making that query return stale state.

The reuse pattern:
- Eliminates a DB round-trip
- Works correctly with `REQUIRES_NEW` transaction semantics (reads committed data only)
- Uses information the method already has rather than fetching from a potentially invisible source

## When to apply

When a `REQUIRES_NEW` method:
1. Already retrieves an entity for a primary purpose
2. Needs information about the same entity for a secondary purpose
3. The secondary purpose would otherwise require querying an entity modified in the outer tx
