### Symptom
Java function workers (e.g. `SusarCriteriaEvaluator`) never execute in `@QuarkusTest` — case
bindings fire, the `casehub.worker.schedule` event is published, `WorkerScheduleEventHandler`
runs, but the worker function is never called. No error is logged. The case stays in `RUNNING`
state indefinitely, or completes via a default/empty path. Any action gate that depends on the
function worker is never created.

### Context
`casehub-engine-scheduler-quartz` is on the compile classpath (in `runtime/pom.xml`) but NOT in
`quarkus.index-dependency` in test `application.properties`. The Quartz module ships
`QuartzWorkerExecutionManager @ApplicationScoped` which displaces `NoOpWorkerExecutionManager
@DefaultBean`. Without a Jandex index, Quarkus cannot discover `QuartzWorkerExecutionManager`
at build time — only the `@DefaultBean` no-op is active.

### Root cause
`WorkerScheduleEventHandler.submitIfNeeded()` calls `workflowExecutionManager.submit(...)`.
With `NoOpWorkerExecutionManager` active, `submit()` returns
`Uni.createFrom().failure(new ProvisioningException("No WorkerExecutionManager configured"))`.
The failure is caught by `scheduleUnderLock()`'s `onFailure().invoke(LOG::errorf)` — logged at
ERROR level only — and the Uni resolves void. The Quartz job is never scheduled; the worker
function is never called.

The `ProvisioningException` log line reads:
```
No WorkerExecutionManager configured — add an @ApplicationScoped WorkerExecutionManager
implementation (scheduler-quartz or workers-camel)
```
But this line fires on the Vert.x worker pool thread and is easily missed in test output.

### Fix
**Option 1 (preferred for integration tests):** Do not try to make Quartz work in `@QuarkusTest`.
Test the two observable behaviors of a Java function worker directly:

1. **Gate decision path:** inject the `@ConsumeEvent` gate listener and call it directly with a
   manually constructed event. It fires synchronously from the test thread; `REQUIRES_NEW`
   writes commit independently.

2. **Case completion path:** inject the `CaseLifecycleEvent` observer and call it directly with
   a `GoalReached` event carrying the correct `tenancyId` (from `FixedCurrentPrincipal`).

```java
// Gate approval path — synchronous from test thread
gateDecisionListener.onApproved(new ActionGateApprovedEvent(caseId, 1L, null, "dr-smith"));

// Case completion path — SusarOversightListener observes GoalReached
oversightListener.onCaseLifecycle(
    new CaseLifecycleEvent(caseId, FIXED_TENANCY_ID, "GoalReached", "GoalReached",
                           "COMPLETED", "system", "SYSTEM", null));
```

**Option 2 (workaround — adds Quartz startup overhead to all tests):** Add the scheduler jar to
`quarkus.index-dependency` in test `application.properties`:
```properties
quarkus.index-dependency.engine-scheduler.group-id=io.casehub
quarkus.index-dependency.engine-scheduler.artifact-id=casehub-engine-scheduler-quartz
```
This activates `QuartzWorkerExecutionManager`, but the Quartz job still runs on a ForkJoin pool
thread where `@Transactional` on the function (e.g. `SusarCriteriaEvaluator.apply()`) may not
bind CDI `@RequestScoped` beans properly. Additionally, `WorkerExecutionRecoveryService` runs at
startup and may fail with CDI errors if cross-tenant repositories are not in scope — see
GE-20260526-34a4c4.

### Why this is non-obvious
- `casehub-engine-scheduler-quartz` is a compile dependency — it IS on the classpath. The issue
  is Jandex indexing, not classpath presence. Quarkus auto-scans only jars in the application
  module; third-party jars require explicit `quarkus.index-dependency` entries to participate in
  CDI discovery.
- The failure is completely silent — no compile error, no CDI startup failure, no runtime
  exception visible to the test. The `ProvisioningException` is logged at ERROR but on a
  worker-pool thread during async event processing.
- `SusarOversightLifecycleTest` accepted the broken state for weeks — it asserts
  `susarOversightStatus.isIn(REQUESTED, COMPLETED)`, and `REQUESTED` is set synchronously in
  Phase 1 before `startCase()`. The test passed without the worker ever running.
