### Context

A `Semaphore` inside a production class is acquired on entry and released via a termination handler (e.g. Mutiny's `.onCancellation().invoke(semaphore::release)`) — the outermost handler in a chain. A test that triggers cancellation needs to assert the semaphore is released before attempting a follow-up call. The observable cancellation signal (an inner CountDownLatch) fires before the outer semaphore release handler, so awaiting the latch is not sufficient.

### What most developers do instead

`Thread.sleep(N)` as a blind wait after the observed cancellation signal. Or worse: exposing `semaphore` as `public` or `protected` to make it inspectable from the test, which pollutes the production API with a test artifact. Or using reflection — fragile and breaks under module-info.

### The technique

Add a package-private accessor that delegates to the internal concurrent state, then poll it with Awaitility from the test (which is in the same package):

```java
// In the production class (package-private — test seam only)
/** For testing only — allows tests to observe semaphore state without exposing the field. */
int availablePermits() {
    return semaphore.availablePermits();
}
```

```java
// In the test (same package — accessible without reflection)
Awaitility.await("outer semaphore release after cancellation")
    .atMost(2, TimeUnit.SECONDS)
    .pollInterval(10, TimeUnit.MILLISECONDS)
    .until(() -> client.availablePermits() == 1);
```

The Awaitility `await(String)` overload names the condition — if the deadline fires, the timeout message includes the description, which is essential for CI log diagnostics. In Awaitility 4.3.0, `failMessage()` was removed; `await(String)` is the correct substitute.

### Why this is non-obvious

The temptation is to sleep blindly or expose the state publicly. Package-private access threads the needle: the state is observable from the test package without becoming part of the API contract. The seam method documents its own intent ("For testing only") via Javadoc and keeps future maintainers honest.

The `pollInterval(10, MILLISECONDS)` detail matters: Awaitility's default poll interval is 100ms — coarse for a condition that typically resolves in microseconds. Explicit `pollInterval` cuts CI runtime for concurrent tests measurably without changing semantics.

This pattern generalises: any `Semaphore`, `CountDownLatch`, `AtomicBoolean`, or similar concurrent primitive inside a production class can be made test-observable via a package-private accessor, replacing unreliable sleep-based synchronisation.
