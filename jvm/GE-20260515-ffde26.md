## Optional Quarkus features: Jandex library module pattern

**Topic:** Packaging optional features in multi-module Quarkus apps
**Alternatives exist:** yes — bundle into core extension runtime; use Quarkus build-time flags

### The convention

Package optional features as standalone **Jandex library modules** — Maven submodules that depend on the core runtime, include only feature-specific deps, and generate a Jandex index via `jandex-maven-plugin`:

```xml
<plugin>
  <groupId>io.smallrye</groupId>
  <artifactId>jandex-maven-plugin</artifactId>
  <version>3.3.1</version>
  <executions>
    <execution><id>jandex</id><phase>process-classes</phase>
      <goals><goal>jandex</goal></goals>
    </execution>
  </executions>
</plugin>
```

Adding the module to a consuming pom activates the feature. Omitting it has zero impact on classpath, startup time, CDI bean count, or native image size.

### Why this style

Bundling optional features into the core runtime means every consumer carries the feature's beans and deps whether they use it or not. Build-time flags work but add conditional complexity to the deployment processor. The Jandex library pattern makes feature activation purely declarative — dependency present means feature active.

### Trade-offs

Without the Jandex index, CDI beans are discovered when the module is tested in isolation but NOT when consumed as a JAR by another module. This silent non-discovery is easy to miss. The core-bundle alternative avoids this but bloats every consumer.

### When not to use it

When all consumers need the feature anyway. When the feature requires tight coupling to the core extension deployment processor (build-time flags may be cleaner).
