### Symptom
A Java record deserializes from YAML without errors but all fields are `null`. No exception is thrown; no warning is logged. The YAML document is well-formed and the key names exactly match the record component names.

### Context
Code creates `new ObjectMapper(new YAMLFactory())` directly (e.g. in a test-scope loader that cannot inject the CDI-managed mapper) and uses it to deserialize a Java record via `mapper.readValue(inputStream, MyRecord.class)`.

### Root cause
`findAndRegisterModules()` activates `jackson-module-parameter-names`, which tells Jackson how to bind YAML keys to record constructor parameters by name. Without it, the module is not registered, Jackson cannot determine which YAML key maps to which constructor parameter, and the fields deserialize as their defaults — `null` for references, `0`/`false` for primitives (except primitives absent from YAML throw `MismatchedInputException` — see GE-20260602-2cff5e).

The behavior is not documented anywhere in the Jackson YAML documentation. The `-parameters` Maven compiler flag (see GE-20260414-2263db) makes the names available at runtime, but `findAndRegisterModules()` is what makes Jackson actually use them.

### Fix
```java
private static final ObjectMapper YAML =
    new ObjectMapper(new YAMLFactory()).findAndRegisterModules();
```

One call, called once, on the bare `ObjectMapper` immediately after construction. Never reuse a `YAMLFactory`-constructed mapper that lacks this call.

### Why this is non-obvious
The mapper works for simple cases (non-record POJOs with getters, or records annotated with `@JsonCreator`). Only records relying on constructor parameter names silently fail. The symptom — all fields null, no error — is identical to a valid YAML document with all-null values, so developers first check the YAML content and key names before suspecting mapper configuration.
