### Symptom

Compilation fails when calling `prefs.get(QueueSnapshotInterval.KEY)` with a `PreferenceKey<DurationPreference>`. The error says no matching method for `get(PreferenceKey<DurationPreference>)`.

### Context

`DurationPreference` is the platform's shared record for Duration-typed preferences. It implements `MultiValuePreference`, not `SingleValuePreference`. The `Preferences` interface has two `get` signatures:

- `<T extends SingleValuePreference> T get(PreferenceKey<T> key)` — no subKey
- `<T extends MultiValuePreference> T get(PreferenceKey<T> key, String subKey)` — requires subKey

When defining a preference that is conceptually a single value (e.g. a snapshot interval, a retention period), the natural call is `prefs.get(KEY)` — but this won't compile because `DurationPreference` is `MultiValuePreference`.

### Root cause

`DurationPreference` was designed as `MultiValuePreference` to support per-sub-key duration lookups (e.g. different durations per reviewer type). But many real-world duration preferences are single-value — one interval per scope, no subKey dimension.

### Fix

Pass an empty string as the subKey:

```java
DurationPreference interval = prefs.getOrDefault(QueueSnapshotInterval.KEY, "");
Duration value = interval.duration();
```

The empty string `""` acts as a default/unnamed subKey. `getOrDefault` returns the key's `defaultValue()` when no preference is configured.

### Why this is non-obvious

The typed-preference-keys protocol (`PP-20260517-2cd5f0`) shows `SingleValuePreference` examples (like `DeclineTarget`) that use `prefs.get(KEY)` without a subKey. A developer following that pattern with `DurationPreference` expects the same signature to work. The `MultiValuePreference` inheritance is not surfaced in the `PreferenceKey` definition — you only discover it when the compiler rejects your call.
