## Symptom

`@QuarkusTest` classes in a module with `quarkus-langchain4j-jlama` fail immediately at test startup — compile and `mvn install -DskipTests` both succeed, but running tests throws:

```
java.lang.IllegalStateException: Unsupported value type: [ALL-UNNAMED]
  at io.quarkus.bootstrap.json.Json.appendValue(Json.java:541)
  at io.quarkus.bootstrap.app.ApplicationModelSerializer.writeJson(...)
  at io.quarkus.bootstrap.BootstrapAppModelFactory.resolveAppModelForWorkspace(...)
```

The error message points to Quarkus bootstrap internals with no mention of Jlama or the Vector API.

## Context

Occurs when adding `quarkus-langchain4j-jlama 0.26.1` as a dependency to a Quarkus 3.32.2 project and running `@QuarkusTest` tests via Maven Surefire. Triggered on Java 21 and Java 26. A `mvn clean` before running does not help — the issue reproduces on the first clean build.

## Root Cause

The Jlama runtime pom.xml configures the Quarkus Maven plugin with:

```xml
<devMode>
    <jvmOptions>
        <enable-native-access>ALL-UNNAMED</enable-native-access>
        <add-modules>jdk.incubator.vector</add-modules>
    </jvmOptions>
</devMode>
```

This generates `META-INF/quarkus-extension.properties` inside the runtime JAR:

```
dev-mode.jvm-option.std.enable-native-access=ALL-UNNAMED
dev-mode.jvm-option.std.add-modules=jdk.incubator.vector
```

Quarkus reads extension properties during test bootstrap — including dev-mode JVM options — and includes them in the application model. When `ApplicationModelSerializer` tries to write the model to a JSON cache, `Json.appendValue()` encounters the `[ALL-UNNAMED]` value resolved as a `java.lang.Module` object and throws `IllegalStateException` because it has no case for `Module` type.

Adding `--add-opens` manually to Surefire `<argLine>` does NOT fix this — the error occurs in Maven-side Quarkus bootstrap before the test JVM forks.

Downgrading to 0.22.x fails differently (ChatMemoryProcessor API incompatibility). Versions 0.24.0 and 0.26.1 both exhibit the same error.

## Fix

### Solution 1 — Fix in quarkus-langchain4j (correct fix)
**Approach:** Remove or conditionalize `<enable-native-access>ALL-UNNAMED</enable-native-access>` in `model-providers/jlama/runtime/pom.xml` (lines ~64–74). On Java 23+, the Vector API is standard (JEP 469) and the opens are unnecessary. A Java version check (`${java.version} >= 23`) should suppress these JVM options.

**Pros:** Correct fix at the source; benefits all consumers.
**Cons:** Requires a PR to quarkus-langchain4j and a new release.

```xml
<!-- Only needed for Java < 23 -->
<devMode>
    <jvmOptions>
        <!-- remove enable-native-access and add-modules for Java 23+ -->
    </jvmOptions>
</devMode>
```

### Solution 2 — Fix in Quarkus core
**Approach:** In `io.quarkus.bootstrap.json.Json.appendValue()`, add a case for `Module` objects (call `.toString()` or treat as string).

**Pros:** Fixes the underlying serializer bug for all extensions.
**Cons:** Requires a Quarkus core PR; slower to land.

## Why Non-Obvious

`mvn compile`, `mvn install -DskipTests`, and `mvn test-compile` all succeed cleanly. The failure only appears at test runtime during Quarkus bootstrap. The error message — `Unsupported value type: [ALL-UNNAMED]` in `Json.appendValue` — gives no indication that Jlama or the Vector API is involved. `[ALL-UNNAMED]` is a Java module system value that most developers would not connect to an LLM inference engine.
