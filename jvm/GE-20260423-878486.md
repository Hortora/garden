## Symptom

`@QuarkusTest` classes in a module with `quarkus-langchain4j-jlama` fail immediately at test startup — compile and `mvn install -DskipTests` both succeed, but running tests throws:

```
java.lang.IllegalStateException: Unsupported value type: [ALL-UNNAMED]
  at io.quarkus.bootstrap.json.Json.appendValue(Json.java:541)
  at io.quarkus.bootstrap.app.ApplicationModelSerializer.writeJson(...)
  at io.quarkus.bootstrap.BootstrapAppModelFactory.resolveAppModelForWorkspace(...)
```

On Quarkus 3.33.1 the trace is slightly different — the failure lands in `GenerateCodeTestsMojo.doExecute()` at the `generate-code-tests` Maven phase, before surefire runs.

The error message points to Quarkus bootstrap internals with no mention of Jlama or the Vector API.

## Context

Occurs when adding `quarkus-langchain4j-jlama` as a dependency to a Quarkus 3.32+ project and running `@QuarkusTest` tests via Maven Surefire. Triggered on Java 21, 23, and 26. A `mvn clean` before running does not help — the issue reproduces on the first clean build. Verified on:
- quarkus-langchain4j-jlama 0.26.1 + Quarkus 3.32.2
- quarkus-langchain4j-jlama 1.9.1 + Quarkus 3.33.1

## Root Cause

The Jlama runtime JAR (`META-INF/quarkus-extension.properties`) declares dev-mode JVM options:

```
dev-mode.jvm-option.std.enable-native-access=ALL-UNNAMED
dev-mode.jvm-option.std.add-modules=jdk.incubator.vector
```

Quarkus reads these extension properties during test bootstrap and includes them in the `ApplicationModel`. When `ApplicationModelSerializer` tries to write the model to a JSON cache, `Json.appendValue()` receives the `enable-native-access` option value as a raw `List<String>` (containing `"ALL-UNNAMED"` — parsed by `MutableStandardJvmOption` from comma-separated extension property values). `Json.appendValue()` handles `JsonObjectBuilder`, `JsonArrayBuilder`, `String`, `Boolean`, `Integer`, and `Long` — but not `List<String>`. It falls through to `String.valueOf(object)` and throws `IllegalStateException: Unsupported value type: [ALL-UNNAMED]` (the List's `toString()`).

Adding `--add-opens` or `--enable-native-access` manually to Surefire `<argLine>` does NOT fix this — the error occurs in Maven-side Quarkus bootstrap before the test JVM forks.

The property `quarkus.bootstrap.application-model.serialization.format` is read by `ApplicationModelSerializer`'s static initializer to choose between JSON and JOS (Java Object Serialization) format. Because it's a static initializer, this property must be set in the Maven JVM before any plugin class loads — standard Maven `<properties>` and plugin `<configuration><systemProperties>` are too late.

Downgrading to 0.22.x fails differently (ChatMemoryProcessor API incompatibility). Versions 0.24.0, 0.26.1, and 1.9.1 all exhibit the same error.

## Fix

### Solution 1 — Immediate workaround: switch to JOS serialization via .mvn/maven.config
**Approach:** Create `.mvn/maven.config` in the project root with the JOS flag. This file is applied to the Maven JVM before any plugin executes, ensuring `ApplicationModelSerializer`'s static initializer picks it up.

**Pros:** Immediate; no code changes; works across all affected version combinations.
**Cons:** `.mvn/maven.config` is project-global — the JOS format applies to all Quarkus Maven goals, not just tests. JOS is binary, slightly harder to inspect than JSON.

**`.mvn/maven.config`:**
```
-Dquarkus.bootstrap.application-model.serialization.format=jos
```

After adding this file, `mvn clean test` succeeds. Note: if you ran tests before adding this file, a stale JSON-format model may be cached in `target/` — always clean first: `mvn clean test`.

**What does NOT work:**
- `<quarkus.bootstrap.application-model.serialization.format>jos</quarkus.bootstrap.application-model.serialization.format>` in Maven `<properties>` — Maven properties are not system properties; the static initializer doesn't see them
- `<systemProperties>` inside Quarkus plugin `<configuration>` — the static initializer has already run by the time plugin configuration is processed
- `<serializeTestModel>false</serializeTestModel>` in plugin config or as Maven property — prevents the model from being written; surefire then fails reading it: "Unable to fully read json value"

### Solution 2 — Fix in quarkus-langchain4j (upstream fix)
**Approach:** Remove or conditionalize `<enable-native-access>ALL-UNNAMED</enable-native-access>` in `model-providers/jlama/runtime/pom.xml`. On Java 23+, the Vector API is standard (JEP 469) and the opens are unnecessary.

**Pros:** Correct fix at the source; benefits all consumers.
**Cons:** Requires a PR to quarkus-langchain4j and a new release.

### Solution 3 — Fix in Quarkus core
**Approach:** In `io.quarkus.bootstrap.json.Json.appendValue()`, add a case for `Collection` objects (delegate to `toJsonArrayBuilder`).

**Pros:** Fixes the underlying serializer bug for all extensions that register List-typed JVM options.
**Cons:** Requires a Quarkus core PR; slower to land.

## Why Non-Obvious

`mvn compile`, `mvn install -DskipTests`, and `mvn test-compile` all succeed cleanly. The failure only appears at test runtime during Quarkus bootstrap. The error message — `Unsupported value type: [ALL-UNNAMED]` in `Json.appendValue` — gives no indication that Jlama or the Vector API is involved. `[ALL-UNNAMED]` looks like a Java module system argument; connecting it to an LLM extension's dev-mode configuration requires reading Quarkus bootstrap source.

The workaround location (`quarkus.bootstrap.application-model.serialization.format` in `.mvn/maven.config`) is undocumented — discoverable only by reading `ApplicationModelSerializer.java`'s static initializer. The timing constraint (must be set before class-loading) rules out all standard Maven configuration mechanisms, which appears to be why this property isn't commonly known.
