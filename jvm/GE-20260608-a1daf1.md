### Symptom
`UnsupportedOperationException: Only ContextChangeTrigger is currently supported` thrown at runtime during case definition loading, even though the YAML does use `contextChange:`.

### Context
Writing a CaseHub YAML case definition binding with no filter — intending the binding to fire unconditionally on any context change. YAML written as:

```yaml
bindings:
  - name: start-session-on-init
    capability: worker
    on:
      contextChange:
```

### Root cause
`contextChange:` (bare key with no value) parses in YAML as `null`. `CaseDefinitionYamlMapper.convertTrigger()` calls `trigger.getContextChange()` and immediately branches `ifnull → athrow UnsupportedOperationException`. A null contextChange trigger is treated as an unsupported trigger type, not as a "no filter" signal.

### Fix
Use `contextChange: {}` (empty map) instead of bare `contextChange:`:

```yaml
on:
  contextChange: {}
```

The empty map produces a non-null `ContextChangeTrigger` model object with `getFilter()` returning null. `convertTrigger()` then follows the non-null path, reads the null filter string, and constructs `ContextChangeTrigger(null)` — the intended unconditional trigger. `DefaultExpressionEngineRegistry.evaluate(null, ctx)` short-circuits to `true` (null evaluator = always fires).

### Why this is non-obvious
YAML syntax for "empty but present" vs "absent/null" differs by one character (`{}` vs nothing). The error message (`Only ContextChangeTrigger is currently supported`) points to the trigger type check, not the null value — it looks like you used the wrong trigger type, not the wrong YAML form. The distinction between null and empty-map at the Java model level is not documented.
