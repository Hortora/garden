When `@CircuitBreaker` is applied to a Quarkus langchain4j agent interface,
`CircuitBreakerOpenException` is NOT wrapped in `AgentInvocationException`. The FaultTolerance
interceptor fires at a higher CDI priority than the agent's invocation handler, so
`CircuitBreakerOpenException` escapes the wrapper and propagates directly to the caller.
All other agent runtime exceptions ARE wrapped — this is the notable carve-out.

### Symptom

Catch blocks on `AgentInvocationException` silently miss the open-circuit case:

```java
try {
    agent.answer("q");
} catch (AgentInvocationException ex) {
    // Never reached when circuit is open — CircuitBreakerOpenException escapes
}
```

### Root cause

SmallRye FaultTolerance interceptors run at a high CDI priority (~10). The agent
`InvocationHandler` wraps exceptions in `AgentInvocationException` at a lower effective
priority. When the circuit is open, SmallRye throws `CircuitBreakerOpenException` before
the proxy's wrapping has a chance to apply.

### Fix

Catch `CircuitBreakerOpenException` directly alongside `AgentInvocationException`:

```java
try {
    agent.answer("q");
} catch (CircuitBreakerOpenException ex) {
    // open circuit
} catch (AgentInvocationException ex) {
    // other agent failures
}
```

### Why this is non-obvious

The quarkus-langchain4j documentation states agent exceptions are wrapped in
`AgentInvocationException`. Most developers expect this to cover all failure modes including
circuit-open. The exception is FaultTolerance interceptors that fire before the agent proxy
— discovered only when catch blocks silently fail during circuit-open conditions in tests or
production.
