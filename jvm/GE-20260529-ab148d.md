### Symptom
Quarkus startup fails with:
```
UnsatisfiedResolutionException: Unsatisfied dependency for type io.example.MyBean and qualifiers [@Default]
```
The bean class exists on the classpath, all its constructor parameter types exist, and it is annotated with `@ApplicationScoped`. No warning about constructors appears.

### Context
The bean has two constructors: one for CDI injection (using `@All List<X>` or `Event<Y>`) and one package-private constructor for unit tests (accepting simpler types). Neither constructor is annotated with `@Inject`.

### Root cause
CDI requires exactly one constructor annotated with `@Inject` when a class has multiple constructors. If none is annotated, CDI falls back to looking for a no-arg constructor. When no no-arg constructor exists, CDI reports the failure as "Unsatisfied dependency for the bean type" — not as "no suitable constructor found." The error message points at the injection point in the *caller*, not at the bean's constructor, making it look like the bean is absent from the CDI context.

### Fix
Annotate exactly one constructor with `@Inject` — the one intended for CDI use:

```java
@ApplicationScoped
public class InboundConnectorService {

    @Inject  // ← required when multiple constructors exist
    InboundConnectorService(@All List<InboundConnector> connectors,
                            Event<InboundMessage> messageEvent) {
        this(connectors, messageEvent::fire);
    }

    // Test constructor — no @Inject
    InboundConnectorService(List<InboundConnector> connectors,
                            Consumer<InboundMessage> eventBus) { ... }
}
```

### Why this is non-obvious
The error message says the *bean* is unsatisfied, not that its constructor is unusable. A developer familiar with the technology checks: Is the bean annotated? Is it on the classpath? Is Jandex indexing it? All answers are yes. The actual cause — no `@Inject` on the right constructor — is invisible in the error. Debugging time is wasted checking classpath, module boundaries, and Jandex configuration before the constructor annotation is noticed.

### Why this fix
CDI's fallback to no-arg when `@Inject` is absent is documented behaviour, but the error emitted when no no-arg exists is a bean-level "unsatisfied dependency" rather than a constructor-level error. Adding `@Inject` is the minimal fix; the alternative (providing a no-arg constructor) would require CDI-unfriendly field injection or a proxy-unsafe no-op default.

### Alternatives considered
- Using field injection with `@Inject` — avoids the constructor problem but breaks testability (can't use constructor injection in pure JUnit tests without CDI)
- Adding a no-arg constructor — CDI would use it, but then there is no way to inject dependencies without field injection
