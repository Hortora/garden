### Symptom
`NoClassDefFoundError` for a class (e.g. `io.micrometer.core.instrument.MeterRegistry`) that no test code directly references. ALL tests in the module fail — not just specific test classes. The error occurs during JUnit test discovery, before any test method executes.

### Context
A test source tree contains a class that extends a runtime class from a dependency jar. The parent class's constructor or method signatures reference types from `scope:provided` dependencies. The consumer module doesn't declare those provided dependencies.

Example: `TestSituationDefinitionRegistry extends SituationDefinitionRegistry`. The parent has an `@Inject` constructor with `Instance<MeterRegistry>` parameter. Micrometer is `scope:provided` in the parent's module — not transitive to consumers.

### What was tried (didn't work)
1. Adding the missing dependency (`micrometer-core`) to the consumer's test classpath — fixes the symptom but is whack-a-mole. Next time the parent adds another provided dependency, every downstream module breaks again.

### Root cause
JUnit's `ClassSelectorResolver` discovers all classes in the test source tree and calls `Class.getDeclaredMethods0()` via reflection to check for test methods. This native method resolves ALL method parameter types in the class hierarchy — including inherited constructors. If any parameter type comes from a `scope:provided` dependency not on the consumer's classpath, resolution fails with `NoClassDefFoundError`.

The class doesn't need to be a test class — JUnit scans it anyway because it's in the test source tree.

### Fix
Eliminate the subclass. Instead of placing a class in a foreign package to access package-private constructors, add a `public static` factory method to the parent class:

```java
// In the runtime class (parent module)
public static SituationDefinitionRegistry forTesting(
        List<SituationDefinitionProvider> providers,
        List<Ganglion> ganglia) {
    return new SituationDefinitionRegistry(providers, ganglia);
}
```

Consumer tests call `SituationDefinitionRegistry.forTesting(...)` directly — no subclass in the test tree, no reflective scanning of inherited provided-scope types.

### Why this is non-obvious
The symptom points to a missing dependency — the natural fix is adding it. But that treats the symptom (classpath gap) not the disease (a subclass in the test tree that JUnit shouldn't be scanning). The error message names the missing class but gives no indication that JUnit's class discovery is the trigger, or that the subclass is the reason discovery touches that type at all.

### Alternatives considered
- Adding `micrometer-core` as a test dependency — works but creates ongoing maintenance burden. Every new `scope:provided` dependency in the parent breaks every downstream module with test subclasses.
- Moving the test helper to the parent's testing module — still requires a subclass (same package trick), just centralises the problem instead of eliminating it.
