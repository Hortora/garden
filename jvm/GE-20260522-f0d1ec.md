## Symptom

Filtering SC2EGSet game events by `userid.userId == (playerID - 1)` returns 0 results for some players even when that player clearly issued commands visible in the replay.

## Context

SC2EGSet pre-processes SC2 replays into JSON with a `ToonPlayerDescMap` object and a `gameEvents` array. Each game event has a `userid.userId` field identifying the player who issued it. The natural assumption is that `userId = playerID - 1` (0-indexed), which holds in some games but not all.

## What was tried

Used `playerID - 1` as the `userId` filter for a Protoss player with `playerID=2`. Returned 0 commands across 30 IEM10 games when the player's commands were clearly present in the JSON.

## Root cause

`ToonPlayerDescMap` entries each have a `userID` field that is independent of `playerID`. In the IEM10 dataset, player identifiers like `userID=4` appear for players whose `playerID=2`. The `userId` in `gameEvents` matches the `userID` field from `ToonPlayerDescMap`, not `playerID - 1`.

## Fix

Read `userID` directly from the player's entry in `ToonPlayerDescMap` and use that to filter game events:

```python
for k, v in data['ToonPlayerDescMap'].items():
    if v['race'] == 'Prot':
        player_id = v['playerID']   # for tracker events (controlPlayerId)
        user_id = v['userID']       # for game events (userid.userId)
        break

# tracker events: filter by player_id
# game events: filter by user_id
```

## Why this is non-obvious

The standard pattern for 0-indexed player IDs (`playerID - 1`) works in many replay formats and is the expected convention. SC2EGSet's `userID` appears to encode a different internal game identifier that happened to be `playerID - 1` in some games, masking the bug during initial testing.
