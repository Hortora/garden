### Symptom
`ECGenParameterSpec("secp256k1")` throws `InvalidAlgorithmParameterException` on JDK 15+. The error message does not mention that the curve was removed — it reads like a typo in the curve name.

### Context
JEP 339 (JDK 15) replaced the SunEC provider's native C implementation with a pure Java reimplementation. The new implementation only supports NIST curves (P-256, P-384, P-521). Non-NIST curves including secp256k1 were silently dropped. JDK-8235710 tracks the removal.

### What was tried (didn't work)
- Assuming the curve name was wrong — `"secp256k1"` is the correct name
- Checking for typos in the parameter spec — the string is correct

### Root cause
SunEC's Java reimplementation deliberately excludes non-NIST curves. The old native code supported them; the replacement does not. No deprecation warning was issued.

### Fix
Two options:
1. Add BouncyCastle as a JCA provider — it supports secp256k1
2. Hardcode the secp256k1 curve parameters manually via `ECParameterSpec` (p, a, b, G, n, h) — avoids the dependency but adds ~20 lines of BigInteger constants

### Why this is non-obvious
The error message gives no indication that the curve was removed from the JDK. A developer debugging this would search for typos in the curve name, not check JDK release notes for curve removals.

### Alternatives considered
- Using `AlgorithmParameters.getInstance("EC").init(new ECGenParameterSpec("secp256k1"))` — same failure, same provider
- Downgrading the JDK — not viable for production
