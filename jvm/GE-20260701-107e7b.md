### Symptom
Quarkus application composing multiple upstream modules (e.g. RAS V1-V4, bridge V1-V2,
work V1-V39) fails at startup with Flyway version conflicts — multiple modules ship
migrations starting at V1 in different `classpath:db/<module>/migration/` paths, but
sharing a single Flyway tracking table means V1 from module A collides with V1 from module B.

### Context
Repo-scoped migration paths (`db/<module>/migration/`) prevent classpath scanning conflicts
but do NOT prevent version number collisions when multiple paths feed the same Flyway instance.
Renumbering upstream migrations is not an option — the files are owned by other repos.

### What was tried (didn't work)
- Renumbering upstream migrations — they are fixed in published artifacts
- Using a single datasource with all paths — Flyway sees V1 twice and fails
- Version range allocation — works within one repo but not across repos with independently published artifacts

### Fix
Use separate **named datasources** in Quarkus, each pointing to the same physical database.
Each datasource gets its own Flyway tracking table (`flyway_schema_history`), so version
numbers are isolated per datasource.

```properties
# Default datasource — app-owned tables (no collision risk)
quarkus.datasource.jdbc.url=jdbc:postgresql://localhost:5432/mydb
quarkus.flyway.locations=classpath:db/bridge/migration,classpath:db/myapp/migration

# Named datasource — isolate module group A
quarkus.datasource.group-a.jdbc.url=jdbc:postgresql://localhost:5432/mydb
quarkus.flyway.group-a.locations=classpath:db/work/migration,classpath:db/ledger/migration

# Named datasource — isolate module group B (collides with both bridge and work)
quarkus.datasource.group-b.jdbc.url=jdbc:postgresql://localhost:5432/mydb
quarkus.flyway.group-b.locations=classpath:db/ras/migration
```

All three datasources share the **same JDBC URL** — the isolation is logical (separate
`flyway_schema_history_<name>` tables), not physical (separate databases). Group modules
that have non-overlapping version ranges on the same datasource; isolate those that collide.

### Root cause
Flyway tracks applied migrations in a single `flyway_schema_history` table per datasource.
When two upstream modules both ship V1, Flyway sees a duplicate version. Named datasources
create independent tracking tables, each with its own version namespace.

### Why this is non-obvious
The obvious fix is renumbering — but you can't renumber artifacts you don't own. The next
obvious fix is repo-scoped paths — but that only prevents classpath scanning conflicts, not
version number collisions. Using multiple datasources on the same database feels wrong
(why would you open three connections to the same DB?) but it's the minimal correct solution.

### Why this fix
Multiple datasources on the same database sounds like waste, but the overhead is negligible —
Quarkus connection pools are shared when URLs match, and Flyway only runs at startup. The
alternative (forking upstream migrations to renumber them) creates a permanent maintenance
burden. Logical isolation via tracking tables is cheaper than physical isolation or renumbering.
