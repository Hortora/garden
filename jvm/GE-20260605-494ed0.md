## ChannelGateway.fanOut() delivers to backends on virtual threads — @QuarkusTest assertions fail before backend fires

### Symptom

`messageService.findResponseByCorrelationId(channelId, correlationId)` returns empty immediately after `messageService.dispatch(QUERY)` returns. The RESPONSE is never found. No exception, no warning — the assertion simply fails because the backend hasn't fired yet.

### Context

Any `@QuarkusTest` that:
1. Registers a `ChannelBackend` via `ChannelGateway.registerBackend()`
2. Dispatches a QUERY message via `MessageService.dispatch()`
3. Immediately asserts the RESPONSE or DECLINE message exists

The test appears to be testing a synchronous flow — `dispatch()` returns, backend fires, response is stored, assertion runs. This is wrong.

### What was tried (didn't work)

- Removing `@TestTransaction` — no effect; the problem is thread timing, not transaction visibility
- Asserting after `dispatch()` returns synchronously — always fails; the virtual thread may not have started yet
- `Thread.sleep()` busy-wait loop — fragile and timing-dependent; worked in some environments but still non-deterministic

### Root cause

`MessageService.dispatch()` calls `channelGateway.fanOut()` synchronously. However, inside `ChannelGateway.fanOut()`, each backend is invoked on a **new virtual thread**:

```java
// ChannelGateway.java (casehub-qhorus)
public void fanOut(UUID channelId, String channelName, OutboundMessage message) {
    ChannelRef ref = new ChannelRef(channelId, channelName);
    List<BackendEntry> entries = registry.getOrDefault(channelId, List.of());
    for (BackendEntry entry : List.copyOf(entries)) {
        if (entry.backend() == agentBackend) continue;
        ChannelBackend backend = entry.backend();
        Thread.ofVirtual().start(() -> {      // ← async virtual thread per backend
            try {
                backend.post(ref, message);
            } catch (Exception ex) {
                LOG.errorf(ex, "Backend %s failed on fanOut to channel %s",
                        backend.backendId(), channelId);
            }
        });
    }
}
```

`dispatch()` returns as soon as all virtual threads are *started*, not when they *complete*. The backend's `post()` method — which calls the LLM, dispatches the RESPONSE, and fulfils the Commitment — runs entirely after `dispatch()` has already returned to the test.

### Fix

Use **Awaitility** to poll for the expected observable state:

```java
// Add to pom.xml test scope:
// <dependency>org.awaitility:awaitility:4.3.0</dependency>

import static org.awaitility.Awaitility.await;
import java.time.Duration;

private static final Duration TIMEOUT = Duration.ofSeconds(5);

@Test
void query_dispatchesResponse() {
    messageService.dispatch(MessageDispatch.builder()
            .channelId(channelId)
            .type(MessageType.QUERY)
            .correlationId(correlationId)
            // ...
            .build());

    // Wait for virtual-thread backend delivery
    await().atMost(TIMEOUT).until(() ->
            messageService.findResponseByCorrelationId(channelId, correlationId).isPresent());

    var response = messageService.findResponseByCorrelationId(channelId, correlationId);
    assertThat(response).isPresent();
    // ... assert content, sender, etc.
}
```

For asserting the response stays **absent** (orphaned channel test), use `during()` to prevent a false-pass on transient empty-then-filled state (see GE-20260515-ed10ee):

```java
await().during(Duration.ofSeconds(1)).atMost(Duration.ofSeconds(3))
        .until(() -> messageService.findResponseByCorrelationId(channelId, correlationId).isEmpty());
```

### Why this is non-obvious

`MessageService.dispatch()` is `@Transactional` and calls `channelGateway.fanOut()` inline — reading the bytecode of `dispatch()` shows a direct synchronous invocation of `fanOut()`. Only reading the **source of `fanOut()` itself** reveals the `Thread.ofVirtual().start()` inside. Developers assume the call chain is synchronous end-to-end because the outer dispatch is synchronous, but the fan-out to backends is deliberately asynchronous.

The `@TestTransaction` annotation makes this worse: it prevents committing the channel entity before the virtual thread runs, causing the response dispatch to silently fail because the channel cannot be found. Tests with `@TestTransaction` will always fail; tests without it will fail only due to timing.

### Why this fix

Awaitility is the idiomatic solution for asserting async outcomes in Java integration tests. It uses configurable polling with a clear failure message on timeout, unlike `Thread.sleep()` busy-waits which are both fragile and hide the async nature of the system under test.
