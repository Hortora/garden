## Technique

When implementing an SPI builder interface loaded via `ServiceLoader` (e.g. `CallableTaskBuilder`, `TaskExecutorFactory`, or similar two-step init/build patterns), use a `ThreadLocal<T>` to pass configuration between `init()` and `build()` calls instead of an instance field.

## Context

`ServiceLoader.Provider.get()` (Java 9+) caches the SPI instance — subsequent calls return the same object. If two threads concurrently load different workflow definitions that require the same builder, they call `init()` concurrently on the shared instance, racing on any mutable instance field. The bug is silent: most of the time only one workflow loads at a time, so the race is never triggered in tests.

## What developers normally do

```java
public class MyBuilder implements SomeBuilder<MyTask> {
    private String configValue; // ← RACES under concurrent definition loading
    
    @Override
    public void init(MyTask task, ...) {
        this.configValue = task.getConfig();
    }
    
    @Override
    public SomeCallable build() {
        return ctx -> useValue(this.configValue); // ← may see another thread's config
    }
}
```

## The technique

```java
public class MyBuilder implements SomeBuilder<MyTask> {
    // SDK calls init() and build() sequentially on the same thread,
    // but the same instance may be called from different threads for
    // different definitions concurrently.
    private final ThreadLocal<String> configHolder = new ThreadLocal<>();
    
    @Override
    public void init(MyTask task, ...) {
        configHolder.set(task.getConfig());
    }
    
    @Override
    public SomeCallable build() {
        final String config = configHolder.get();
        configHolder.remove(); // prevent ThreadLocal leak
        return ctx -> useValue(config);
    }
}
```

## Why this is non-obvious

The SPI contract documents `init()` → `build()` as a sequential pair, which appears to make instance fields safe. The non-obvious part is that `ServiceLoader.Provider.get()` caches the SPI instance per-classloader, so concurrently loading two different definitions via `ConcurrentHashMap.computeIfAbsent()` (e.g. `WorkflowApplication.workflowDefinition()`) will call `init()` on the same builder instance from different threads. The race manifests as one definition receiving the other's config — with no error.

## Prerequisites

`init()` and `build()` must be called on the same thread (verify with your SDK — the Serverless Workflow SDK 7.13.4.Final guarantees this for `CallableTaskBuilder`).
