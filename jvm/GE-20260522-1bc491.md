## Symptom

Browser `EventSource.onmessage` fires but `JSON.parse(e.data)` throws silently.
`e.data` contains `"data: [...]"` (with the literal prefix) rather than the raw JSON.
No server error. SSE connection stays open.

## Context

Quarkus REST endpoint annotated `@Produces("text/event-stream")` returning `Multi<String>`.
You manually construct `"data: " + json + "\n\n"` to emit a proper SSE frame.

## Root cause

RESTEasy Reactive wraps every item emitted from a `Multi<String>` SSE endpoint with
`data: <item>\n\n` automatically. Adding the prefix manually produces the double-frame:

```
data: data: [{"id":1,...}]

```

This applies whether or not `@Blocking` is present — `@Blocking` does not change the
SSE framing behaviour. The symptom is completely silent: no exception on either side,
`e.data` is just the wrong string.

## Fix

Emit bare JSON (no prefix, no suffix):

```java
// Wrong
return Multi.createFrom().uni(fetchData())
    .map(items -> "data: " + mapper.writeValueAsString(items) + "\n\n");

// Correct
return Multi.createFrom().uni(fetchData())
    .map(items -> mapper.writeValueAsString(items));
```

RESTEasy Reactive produces the correct `data: [...]\n\n` wire format from the bare string.

## Why this is non-obvious

Every SSE specification and tutorial shows the `data: ` prefix as part of the format.
The natural assumption is that you must include it — especially since `Multi<String>`
is a raw string stream with no SSE-awareness visible in the API. The Quarkus docs do
not explicitly state that the wrapping happens automatically for `Multi<String>` with
`@Produces("text/event-stream")`.

### Why this fix

The RESTEasy Reactive SSE codec owns the wire format. Providing raw data and letting
the codec wrap it is the same pattern used by other Quarkus SSE primitives
(`@SseElementType`, `ServerSentEvent<T>`). Manually constructing the frame fights
the framework and breaks in exactly the way seen here.
