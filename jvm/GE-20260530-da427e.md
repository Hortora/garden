### Symptom

`@QuarkusTest` fails at startup with:
```
AnnotationException: Association 'io.casehub.ledger.runtime.model.LedgerEntry.supplements'
targets the type 'io.casehub.ledger.runtime.model.supplement.LedgerSupplement' which does
not belong to the same persistence unit
```

No code changes to `LedgerEntry` or `LedgerSupplement`. The error appears only after adding new `LedgerEntry` subclass entities to the qhorus named PU.

### Context

A Quarkus application with two persistence units:
- Default PU: `packages=io.casehub.work.runtime.model,...,io.casehub.life.app.entity`
- Named PU (qhorus): `packages=io.casehub.qhorus.runtime,io.casehub.ledger.runtime.model,...,io.casehub.life.app.entity.ledger`

New `HealthDecisionLedgerEntry extends LedgerEntry` placed at `io.casehub.life.app.entity.ledger` — a sub-package of the DEFAULT PU's `io.casehub.life.app.entity`. Intended for the qhorus PU (explicitly listed in qhorus packages).

### Root cause

Quarkus uses **prefix matching** for the `packages` property: `io.casehub.life.app.entity` matches everything starting with that prefix, including `io.casehub.life.app.entity.ledger`. The default PU claims the ledger entities before the qhorus PU can — even though the qhorus PU also lists the sub-package.

`HealthDecisionLedgerEntry` → assigned to DEFAULT PU (via `io.casehub.life.app.entity` prefix match). `LedgerEntry` and `LedgerSupplement` → assigned to qhorus PU (via `io.casehub.ledger.runtime.model`). Hibernate walks the `LedgerEntry.supplements` `@OneToMany` association and finds `LedgerSupplement` is in qhorus PU while the subclass is in the default PU → cross-PU association error.

### Fix

Place the application's `LedgerEntry` subclass entities in a package that is a **sibling** of the default PU's entity package, not a child:

```
❌ io.casehub.life.app.entity.ledger   (child of io.casehub.life.app.entity — DEFAULT PU)
✅ io.casehub.life.app.ledger          (sibling — only matches qhorus PU)
```

Then list `io.casehub.life.app.ledger` in the qhorus PU packages.

### Additional finding: LedgerSupplement package

The qhorus PU must list `io.casehub.ledger.runtime` (broad) rather than `io.casehub.ledger.runtime.model` (narrow). The narrow package misses `io.casehub.ledger.runtime.model.supplement.LedgerSupplement` — same prefix-matching issue in reverse. Reference: casehub-qhorus uses `io.casehub.ledger.runtime` in its test config and works correctly.

### Why this is non-obvious

1. The error message points to `LedgerEntry.supplements` — a bidirectional association that has nothing to do with the new code. Developers investigate the association rather than the PU assignment.
2. Explicitly listing `io.casehub.life.app.entity.ledger` in the qhorus PU packages seems like it should work — the intent is clear. The prefix matching silently overrides it because the default PU's broader prefix matches first.
3. The error only manifests when `LedgerEntry` subclasses are in the PU — before adding them, `LedgerEntry.supplements` existed but Hibernate didn't walk the association chain during augmentation.
