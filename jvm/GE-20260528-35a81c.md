## Symptom

Flyway migration inserts `'NORMAL'` into a `priority` column. Tests pass (they use Java seeding with `WorkItemPriority.MEDIUM`). Production startup succeeds. First read of a seeded row fails with:

```
java.lang.IllegalArgumentException: No enum constant io.casehub.work.runtime.model.WorkItemPriority.NORMAL
```

## Context

`WorkItemPriority` enum values are: `LOW`, `MEDIUM`, `HIGH`, `URGENT`, `CRITICAL`. There is no `NORMAL`. The common assumption is that a "normal" priority would be called `NORMAL` or `DEFAULT`.

Tests that seed via Java using `WorkItemPriority.MEDIUM` never catch this — the enum check happens at compile time for Java seeds. SQL migration seeds bypass the enum check entirely at insert time.

## Root cause

SQL data migrations use string literals that aren't validated by the Java compiler. The `@Enumerated(EnumType.STRING)` mapping only fails when Hibernate reads the row back — not at insert time.

## Fix

Use `'MEDIUM'` in Flyway SQL migrations, not `'NORMAL'`:

```sql
-- Wrong:
INSERT INTO work_item_template (priority, ...) VALUES ('NORMAL', ...)

-- Correct:
INSERT INTO work_item_template (priority, ...) VALUES ('MEDIUM', ...)
```

## Why this is non-obvious

The enum name `MEDIUM` for a default priority is non-obvious — most developers assume a middle priority would be named `NORMAL`, `DEFAULT`, or `STANDARD`. The failure is also deferred: insert succeeds, startup succeeds, only the first read fails with a runtime exception that has no connection to the migration.
