## Testcontainers withCopyFileToContainer bypasses Podman virtiofs path restrictions

**ID:** GE-20260614-92bd9e
**Stack:** Testcontainers 1.20.x, Podman 5.x, macOS arm64

### Symptom / Context

Podman macOS mounts only specific host directories into the VM via virtiofs
(`/Users`, `/private`, `/var/folders`). Bind mounts from other paths (e.g. `/tmp`)
fail at container start:

```
Error: statfs /tmp/config.json: no such file or directory
```

`withClasspathResourceMapping` extracts resources to a temporary path that may be
outside the virtiofs-shared directories, causing the same failure.

### Technique

Use `withCopyFileToContainer` with `MountableFile.forClasspathResource()` instead of
`withClasspathResourceMapping`. Testcontainers copies the file via the Docker archive
API (`PUT /containers/{id}/archive`) rather than a bind mount — no virtiofs sharing
needed, and the source path restriction disappears entirely.

```java
// Fails on Podman macOS — bind mount, virtiofs-restricted:
container.withClasspathResourceMapping(
    "config/openclaw.json",
    "/home/node/.openclaw/openclaw.json",
    BindMode.READ_ONLY);

// Works — Docker archive API copy, no virtiofs needed:
container.withCopyFileToContainer(
    MountableFile.forClasspathResource("config/openclaw.json"),
    "/home/node/.openclaw/openclaw.json");
```

The file is copied via exec after container creation, before the wait strategy fires,
so the application reads the correct file at startup.

### Why this is non-obvious

`withClasspathResourceMapping` and `withCopyFileToContainer` look equivalent from
the API surface — both put a file at a container path. The bind-mount vs archive-API
distinction is not surfaced in the Testcontainers docs. The virtiofs path restriction
only manifests on Podman macOS; the same code works on Linux and Docker Desktop,
making it a platform-specific failure with no hint in the error message.
