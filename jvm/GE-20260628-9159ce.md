### Symptom
Protobuf round-trip tests pass (serialize → deserialize → compare), but the application crashes at runtime when the library's typed parser processes the same message.

### Context
When building translators between domain objects and protobuf messages, the natural test is: domain → protobuf → domain, asserting equality. This validates the translation logic but not the protobuf structure against the consuming library's expectations.

### What was tried (didn't work)
Standard round-trip testing: `GameState → GameStateToProtobuf → protobuf bytes → ProtobufToIntent → GameState`. All fields round-tripped correctly. But `SC2BotAgent` crashed when ocraft's `ResponseObservation.from(response)` parsed the same protobuf — ocraft applies validation rules (`orElseThrow` on required sub-fields, minimum grid sizes, mandatory camera positions) that go beyond protobuf schema compliance.

### Fix
Include the consuming library's parsing layer in the round-trip test. For ocraft:

```java
// Build protobuf from domain
Response response = translator.toProtobuf(gameState);

// Verify the LIBRARY can parse it — not just protobuf deserialization
ResponseObservation obs = ResponseObservation.from(response);

// Then verify the domain round-trip
GameState roundTripped = ProtobufToIntent.translate(obs);
assertEquals(gameState, roundTripped);
```

The library parse step catches: missing required sub-fields, invalid grid dimensions, absent mandatory fields (like `PlayerRaw.camera`), and type coercions that silently lose data.

### Why this is non-obvious
Protobuf is schema-defined — if the message serializes and deserializes, it feels "correct." But libraries built on top of protobuf (ocraft, gRPC service stubs, custom parsers) add validation layers that are not expressed in the `.proto` schema. A protobuf-only round-trip test creates a false sense of completeness. The gap between "protobuf-valid" and "library-valid" is invisible until runtime.
