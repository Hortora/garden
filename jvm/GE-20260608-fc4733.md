## Symptom

`assertThat(terms).doesNotContainAnyElementsOf(List.of("a", "one", "b"))` always passes,
even when the list actually contains elements whose `.value()` equals those strings.

## Context

Testing that `allTerms()` returns only primary `VocabularyTerm` constants and not alias strings.
`allTerms()` returns `List<? extends VocabularyTerm>`. The intent is to assert that alias strings
("a", "one", "b") do not appear as primary values in the returned list.

## Root Cause

AssertJ's `doesNotContainAnyElementsOf` uses `equals()` to compare elements. A `VocabularyTerm`
enum constant never `equals()` a `String` — they are different types. The generic bounds don't
prevent this: `doesNotContainAnyElementsOf` accepts `Iterable<?>`, so the mismatch is invisible
at the call site. Java generics use type erasure at runtime, so no ClassCastException occurs either.
The assertion tests a structural property that is always trivially true (no Strings in a typed list),
not the intended semantic property (alias values don't appear as term values).

## Fix

Extract term values before asserting:

```java
// Before — vacuously true; never fails regardless of what allTerms() returns
assertThat(terms).doesNotContainAnyElementsOf(List.of("a", "one", "b"));

// After — materialises String values for meaningful comparison
assertThat(terms).extracting(VocabularyTerm::value)
    .doesNotContain("a", "one", "b");
```

The `.extracting()` step produces `List<String>` that AssertJ can meaningfully compare against
the `String` arguments.

## Why This Is Non-Obvious

The AssertJ fluent API reads naturally: "assert that this collection does not contain any of these
elements." No compile error. No runtime error. The test is always green. The gotcha requires
knowing that AssertJ uses `equals()` internally and that typed enum constants never `equals()`
untyped Strings — a fact invisible at the call site when `Iterable<?>` erases the type.
