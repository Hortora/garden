## Symptom

A Python script that fixes Java constructor call sites (e.g. adding `null` for new parameters) adds the wrong number of arguments. Some call sites end up with too few (script thinks they already have enough), others with too many (script over-adds). Running the Java compiler shows either "actual and formal argument lists differ in length" or the wrong value type in the wrong position. The fixes look correct on casual inspection.

## Context

Automating Java constructor call site updates (e.g. adding a 24th parameter `excludedUsers` to `WorkItemCreateRequest`) using a Python script that:
1. Finds `new WorkItemCreateRequest(` in files
2. Tracks parenthesis depth until the matching `)`
3. Counts commas at depth 0 to determine the current arg count
4. Adds `null` if current count < target

Call sites containing JSON string literals like:
```java
new WorkItemCreateRequest(
    "title", ..., "category",
    "{\"amount\": 2400, \"currency\": \"GBP\", \"category\": \"training\"}",
    null, null, ...);
```

The JSON string `{"amount": 2400, "currency": "GBP", ...}` contains commas. Without string literal tracking, the script counts those commas as argument separators — adding 2 phantom args to the count — and under-adds the actual needed nulls.

## What was tried (didn't work)

- Simple depth-tracking (tracking `(` and `)` only) — doesn't handle string commas
- Regex-based approaches — fail on multiline calls and nested parens
- Pattern matching on trailing `));` endings — fragile when calls have different closing patterns (single `)` vs double `))`), applied multiple times, cascaded to wrong counts

## Root cause

Comma counting at paren depth 0 treats ALL commas equally. Java string literals can contain commas, semicolons, and any character. Without explicitly tracking when the scanner is inside a string literal (`in_string` state toggle on `"`, with `escape_next` flag for `\"`), every comma inside a JSON payload string counts as an argument separator.

With `N` commas inside string literals, the script believes the call has `N` more arguments than it does — and adds `N` fewer `null`s than needed. If multiple scripts run (or the same script runs multiple times due to partial fixes), the counts compound unpredictably.

## Fix

**Don't parse Java source in Python at all.** If IntelliJ MCP is available, use it — it understands Java semantics and handles all edge cases correctly. For bulk call site edits (e.g. adding a parameter to every constructor call), use IntelliJ's structural search/replace or `replace_text_in_file` with the exact old/new text per file.

If you genuinely must parse Java in Python (no IDE available), track string literal boundaries:

```python
def count_args_correctly(inner):
    depth = 0
    in_string = False
    escape_next = False
    commas = 0
    for c in inner:
        if escape_next:
            escape_next = False
            continue
        if c == '\\' and in_string:
            escape_next = True
            continue
        if c == '"':
            in_string = not in_string
            continue
        if in_string:
            continue
        if c == '(':
            depth += 1
        elif c == ')':
            depth -= 1
        elif c == ',' and depth == 0:
            commas += 1
    return commas + 1
```

**Preferred alternative when available:** JavaParser (Java AST library) handles all Java syntax correctly and is the right tool when IDE tools are unavailable.

## Why this is non-obvious

Developers writing these scripts typically test on simple call sites without string payloads, where counting commas works correctly. JSON payloads are common in Java test fixtures and scenario classes but not in core production code — so production builds pass while test/example modules fail. The symptom (wrong arg count) points to the fix (add nulls) but not to the cause (phantom commas in strings), making each subsequent fix attempt compound the problem.
