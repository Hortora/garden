## Context

Writing a contract test for an interface method that returns primitive `boolean`.
The test is intended to verify the method is callable and returns without throwing.

## Symptom

```java
@Test
void isCrossTenantAdmin_returns_without_exception() {
    assertNotNull(principal.isCrossTenantAdmin());  // always passes
}
```

The test compiles, runs, and passes. No warnings. Looks correct. The test does nothing.

## Root Cause

`assertNotNull` accepts `Object`. Java autoboxes the primitive `boolean` return
to `Boolean` before passing it to the assertion. `Boolean.FALSE` and `Boolean.TRUE`
are never null. The null check passes unconditionally, regardless of what the method
actually does or whether it throws.

## Fix

Assert the actual value instead:

```java
@Test
void isCrossTenantAdmin_defaults_to_false() {
    assertFalse(principal.isCrossTenantAdmin());
}
```

If you genuinely only want to verify callability (no exception), use:

```java
assertDoesNotThrow(() -> principal.isCrossTenantAdmin());
```

## Why This Is Non-Obvious

`assertNotNull(someMethod())` is a natural reflex when writing a "does this exist/work"
test. The autoboxing is invisible at the call site — the IDE shows no warning, the
compiler says nothing, and the test passes. The failure mode is not a false negative
that breaks things; it is a false positive that silently provides no coverage.
A skilled developer writing quickly will not notice until a code reviewer spots it
or the method is intentionally broken and the test still passes.
