### Symptom

Calibrating SC2 building construction times from UnitInit/UnitDone tracker events gives a confident-looking result (e.g. n=32 observations of `FACTORY = 400 loops`) that is flat-out wrong. The actual Factory structure build time is ~960 loops (~43s). The calibration modal is the addon build time, not the structure time.

### Context

Using a building-type mapper (e.g. `Sc2ReplayShared.toBuildingType`) that maps addon names to the parent `BuildingType`:
- `"FactoryTechLab"`, `"FactoryReactor"`, `"FactoryFlying"` → `BuildingType.FACTORY`
- `"BarracksTechLab"`, `"BarracksReactor"`, `"BarracksFlying"` → `BuildingType.BARRACKS`
- `"StarportTechLab"`, `"StarportReactor"`, `"StarportFlying"` → `BuildingType.STARPORT`

In bot replays, Terran players often build more addon completions during the game than they build the parent structures (which are frequently placed at game start via `UnitBorn` events and never generate `UnitInit/UnitDone` pairs). The modal lands on addon time (~400 loops) with high confidence.

### What makes it non-obvious

n=32 observations looks reliable — the standard calibration check passes. The modal value is consistent across all 32 samples. Nothing signals that these are addon completions rather than structure completions. The only way to catch it is to cross-check the calibrated loop count against the SC2 wiki (400 ≠ ~960) or independently count the tracker event unit-type name strings before mapping them.

### Root cause

The building-type mapper is designed to map all states of a building (structure, flying, tech-lab, reactor) to the same `BuildingType` for simulation purposes. This is correct for gameplay but wrong for calibration — addons have shorter build times (~18s) than the structures they attach to (~43s), so mixing them in the same `BuildingType` bucket produces a contaminated sample.

### Fix

Filter addon and morph-state names before passing to the building-type mapper in calibration code:

```java
private static final Set<String> ADDON_OR_MORPH_NAMES = Set.of(
    "BarracksTechLab", "BarracksReactor", "BarracksFlying",
    "FactoryTechLab",  "FactoryReactor",  "FactoryFlying",
    "StarportTechLab", "StarportReactor",  "StarportFlying",
    "SupplyDepotLowered",
    "OrbitalCommandFlying",
    "SpineCrawlerUprooted", "SporeCrawlerUprooted",
    "WarpGate"
);

// In accumulateFromReplay, before calling toBuildingType:
final String unitName = event.getUnitTypeName().toString();
if (ADDON_OR_MORPH_NAMES.contains(unitName)) continue;
final BuildingType bt = mapper.toBuildingType(unitName);
```

After filtering, the same `FACTORY` calibration produces n=16 structure completions with modal=960 — the correct value.

### Why this fix

The obvious fix would be to update `toBuildingType` to return `UNKNOWN` for addons. But `toBuildingType` is correct for simulation purposes — `FactoryTechLab` IS a Factory variant. The fix belongs in the calibration test, not the mapper, because the mapper's aliasing is intentional in its primary use context. Separating addon filtering at the calibration layer leaves the mapper intact for everything else.
