### Symptom

New field added to a JPA entity and its MongoDB document mapper persists correctly on first insert but silently reverts to its initial/null value on every subsequent OCC update.

### Context

When a MongoDB store implements optimistic concurrency control using `findOneAndUpdate` with a version filter, the update path typically builds an explicit field list via `Updates.combine(Updates.set("field1", ...), Updates.set("field2", ...), Updates.inc("version", 1L))`. New fields added to the entity and its document mapper (`from()`/`toDomain()`) are picked up by `persist()` (the insert path) but NOT by the update path — the developer must manually add `Updates.set("newField", ...)` to the `Updates.combine()` block.

### What was tried (didn't work)

Adding the field to:
1. The JPA entity class (`WorkItemSpawnGroup.groupStatus`)
2. The MongoDB document class (`MongoWorkItemSpawnGroupDocument.groupStatus`)
3. The `from()` mapper (entity → document)
4. The `toDomain()` mapper (document → entity)

All four are necessary but not sufficient. The field persists on insert and round-trips correctly through the mapper — but updates via `put()` silently lose the value because `findOneAndUpdate` only writes the fields explicitly listed in `Updates.combine()`.

### Root cause

The `put()` method's update path constructs an explicit `Updates.combine(...)` block listing every field to set. Unlike JPA/Hibernate (which auto-detects dirty fields), MongoDB's `findOneAndUpdate` only writes what you tell it to. A new field not added to this block is silently dropped on every update — the document retains its original insert-time value.

### Fix

Add the new field to the `Updates.combine()` block in the `put()` method:

```java
// In the findOneAndUpdate Updates.combine() block:
Updates.set("policyTriggered", group.policyTriggered),
Updates.set("groupStatus", group.groupStatus != null ? group.groupStatus.name() : null),  // <-- add this
Updates.inc("version", 1L));
```

### Why this is non-obvious

JPA/Hibernate auto-detects dirty fields and persists them without any explicit field list. Developers working across both persistence backends naturally assume "I added the field to the entity and the mapper — it's done." The insert path confirms this assumption (it works via `persist()`). Only the OCC update path silently fails, and since the field has a valid initial value from the insert, the data looks plausible — you won't notice the loss until a mutation that should change the field doesn't stick.

### Alternatives considered

- **Full-document replacement instead of field-by-field update:** Would eliminate the problem but loses atomic `$inc` for the version field and makes concurrent field updates unsafe. The explicit field list exists specifically to support OCC — replacing it trades one problem for a worse one.
