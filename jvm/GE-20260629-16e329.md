### Symptom

Adding `hibernate-jpamodelgen` with `<scope>provided</scope>` to a Hibernate 7.x project produces a Maven relocation warning but no generated `Entity_` metamodel classes. Compilation fails when referencing `Entity_.fieldName`.

### Context

Two independent issues compound: (1) `hibernate-jpamodelgen` was renamed to `hibernate-processor` in Hibernate 6+/7.x — Maven silently relocates it but the annotation processor doesn't run; (2) Java 21+ disabled automatic annotation processor discovery from the classpath — `provided` scope dependencies are no longer scanned for processors.

### What was tried (didn't work)

Adding `hibernate-jpamodelgen` as `provided` dependency (the pre-Hibernate-6 approach documented in most tutorials). Maven compiled without errors but generated zero metamodel classes.

### Root cause

The combination of artifact relocation (silent) and classpath discovery removal (silent) means the annotation processor never executes. No error, no warning beyond the relocation notice.

### Fix

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <annotationProcessorPaths>
            <path>
                <groupId>org.hibernate.orm</groupId>
                <artifactId>hibernate-processor</artifactId>
            </path>
        </annotationProcessorPaths>
        <annotationProcessorPathsUseDepMgmt>true</annotationProcessorPathsUseDepMgmt>
    </configuration>
</plugin>
```

Use `hibernate-processor` (not `jpamodelgen`) in `annotationProcessorPaths`. Set `annotationProcessorPathsUseDepMgmt=true` to inherit the version from the Quarkus/Hibernate BOM. No `provided` scope dependency needed.

### Why this is non-obvious

Most Hibernate metamodel tutorials still show the `provided` scope approach. The relocation warning is easy to miss in verbose Maven output. Java 21's classpath discovery change is a JDK-level decision not documented in Hibernate's migration guide.
