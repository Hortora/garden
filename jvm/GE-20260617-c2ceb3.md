## Symptom

After removing `@Transactional` from a RESTEasy Reactive void SSE endpoint method and replacing the body with an active virtual-thread loop (`BlockingQueue.poll(...)`), the entire server freezes under SSE load. HTTP requests stop responding. No exception is thrown.

## Context

A void SSE method with an active blocking loop:

```java
// WRONG — no blocking annotation
@GET
@Path("/stream")
@Produces("text/event-stream")
public void streamTask(...) {
    // ... blocking queue poll loop ...
    while (true) {
        msg = queue.poll(15, TimeUnit.SECONDS); // BLOCKS EVENT LOOP
    }
}
```

## Root cause

In RESTEasy Reactive (Quarkus REST), JAX-RS endpoint methods run on the Vert.x I/O thread by default. The I/O thread is non-blocking — any blocking call (`BlockingQueue.poll()`, `Thread.sleep()`, `CompletableFuture.get()`) will freeze the entire event loop, halting all HTTP processing for the server.

`@Transactional` on a RESTEasy Reactive method is the implicit signal that causes Quarkus to offload execution to a blocking thread (virtual thread in Java 21). The existing javadoc on the method documented this as "load-bearing" but the dependency was non-obvious.

## Fix

Replace `@Transactional` with `@RunOnVirtualThread` (from `io.smallrye.common.annotation`):

```java
import io.smallrye.common.annotation.RunOnVirtualThread;

@GET
@Path("/stream")
@Produces("text/event-stream")
@RunOnVirtualThread  // NOT @Blocking — that uses the finite worker pool
public void streamTask(...) throws Exception {
    // blocking loop safe on virtual thread
}
```

**Why `@RunOnVirtualThread` and not `@Blocking`:**
- `@Blocking` uses the Vert.x worker thread pool (default ~20 threads). A blocking loop held for connection lifetime (up to 30 min) exhausts this pool under real load.
- `@RunOnVirtualThread` creates one Loom virtual thread per request — correct for long-lived blocking I/O. Scales to thousands of concurrent SSE connections.

Also remove the programmatic transaction from the method body since `@RunOnVirtualThread` doesn't provide one — use `QuarkusTransaction.requiringNew()` explicitly for any JPA reads.

## Why this is non-obvious

`@Transactional` is semantically about database transactions. Its role as the blocking-dispatch trigger for RESTEasy Reactive is documented nowhere clearly. The failure mode (event loop freeze) is dramatic but has no stack trace or error message — the server just stops responding.

`@RunOnVirtualThread` is the correct annotation but is not used by default anywhere in most Quarkus codebases (where `@Blocking` handles typical short-lived blocking). Long-lived SSE handlers are the first place the distinction between `@Blocking` (finite pool) and `@RunOnVirtualThread` (per-request VT) materially matters.
