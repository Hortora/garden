### Symptom
Integration test puts entries into `brokerBean.commitments()` (a `ConcurrentHashMap`), then calls `evaluatorBean.evaluate()` which iterates the same map — but the map is empty from the evaluator's perspective. No error, no warning. The evaluator silently skips all entries.

### Context
Two `@ApplicationScoped` beans share a `ConcurrentHashMap`:
- `CoachingChannelBroker` owns the map (field initializer: `new ConcurrentHashMap<>()`)
- `CoachingComplianceEvaluator` stores a reference via its `@Inject` constructor: `this.commitments = broker.commitments()`

Both beans are injected into a `@QuarkusTest`. The test calls methods on both — expecting them to see the same map contents.

### Root cause
CDI `@ApplicationScoped` beans are proxied via subclassing. The proxy subclass runs field initializers from the parent class, creating a **proxy-owned** `ConcurrentHashMap`. When the evaluator's `@Inject` constructor runs during bean creation, `broker.commitments()` delegates through the proxy to the real bean — returning the real bean's map. But CDI bean creation order and proxy lifecycle can cause the evaluator to receive a reference that does not match what the proxy later delegates to at runtime.

### Fix
Do not cache shared mutable state references obtained from another CDI bean's constructor. Instead, read the reference at evaluation time:

```java
// Wrong — cached at construction, may be a different instance
@Inject
EvaluatorBean(BrokerBean broker) {
    this.commitments = broker.commitments(); // stores reference once
}

// Right — read at evaluation time
public void evaluate(GameState state) {
    var commitments = broker.commitments(); // fresh delegate each call
}
```

For integration tests, construct the evaluator manually with the broker's actual map:
```java
var manualEvaluator = new Evaluator(broker.commitments(), recorder);
```

### Why this is non-obvious
The CDI spec says `@ApplicationScoped` creates a single instance. Developers expect `beanA.getMap()` called from `beanB`'s constructor to return the same object as `beanA.getMap()` called later at runtime. The proxy subclass field initializer is an implementation detail of Quarkus ArC that breaks this assumption for mutable collections.

### Alternatives considered
- Making the map a CDI `@Produces` method — adds indirection without solving the root cause
- Using `@Singleton` instead of `@ApplicationScoped` (no proxy) — changes passivation semantics and breaks lazy initialization
