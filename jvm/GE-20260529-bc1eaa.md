## Context

Writing Flyway migrations for a Quarkus module that uses PostgreSQL in production and H2
with `MODE=PostgreSQL` in `@QuarkusTest`. The migration uses `TIMESTAMPTZ` (PostgreSQL's
abbreviated type name for `TIMESTAMP WITH TIME ZONE`).

## Symptom

Flyway migration fails during `@QuarkusTest` startup with a SQL syntax error even though
`MODE=PostgreSQL` is configured. Production (PostgreSQL) works fine.

## Root Cause

`TIMESTAMPTZ` is a PostgreSQL-specific abbreviation. H2 implements only a subset of
PostgreSQL syntax in `MODE=PostgreSQL` — it recognises the full form
`TIMESTAMP WITH TIME ZONE` but not the abbreviated `TIMESTAMPTZ`.

## Fix

Use the full form in Flyway migrations:

```sql
-- Fails in H2 even with MODE=PostgreSQL
created_at TIMESTAMPTZ NOT NULL,

-- Works in both PostgreSQL and H2 MODE=PostgreSQL
created_at TIMESTAMP WITH TIME ZONE NOT NULL,
```

## Why This Is Non-Obvious

PostgreSQL documentation and tooling widely use `TIMESTAMPTZ` — it appears in pg_dump
output, official PostgreSQL tutorials, and StackOverflow answers. `MODE=PostgreSQL` in H2
creates an expectation that PostgreSQL-idiomatic SQL will work. The error message points
to a SQL syntax failure, not to a type-name unsupported issue, making diagnosis indirect.
