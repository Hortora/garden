## Local mvn install from feature branch silently overrides published SNAPSHOT

### Symptom

Integration tests fail with runtime errors that look like a breaking API change in a dependency
(e.g. "CaseInstance not found or wrong tenant"). Unit tests pass. The error message suggests
a tenancy/security enforcement change, not a dependency problem.

### Context

A multi-repo Maven project using GitHub Packages for SNAPSHOT resolution. The developer ran
`mvn install` on a dependency repo's feature branch (e.g. to recover from a corrupted jar
after a GitHub Packages 401). The locally-installed SNAPSHOT from the feature branch has
different code than the published SNAPSHOT from main.

### What was tried (didn't work)

- `mvn -U dependency:resolve` — dependencies "resolve" successfully but the local install
  is preferred over the remote. With GitHub Packages returning 401, the remote metadata
  cannot be fetched to detect that a newer published version exists.
- Investigating the error message at face value — tracing tenancy enforcement code paths,
  checking `CurrentPrincipal` CDI wiring. The symptom is completely misleading.

### Root cause

Maven's `_remote.repositories` file in `~/.m2/repository/<group>/<artifact>/<version>/`
contains two sets of entries:
- `artifact-0.2-SNAPSHOT.jar>=` — locally installed, **no repository marker**
- `artifact-0.2-20260529.181231-131.jar>github=` — published, from GitHub

Maven prefers the locally-installed version (no repository = local). When GitHub Packages
is returning 401, `mvn -U` cannot fetch new metadata to discover that the published version
is different, so the stale local install persists indefinitely.

### Fix

Rebuild the dependency from its correct branch (typically main) and install locally:

```bash
git -C /path/to/dependency checkout main
mvn -f /path/to/dependency/pom.xml install -DskipTests
git -C /path/to/dependency checkout -   # return to previous branch
```

This overwrites the feature-branch SNAPSHOT with the correct main-branch SNAPSHOT in `~/.m2`.

### Alternatives considered

- Deleting the `~/.m2/repository/<group>/<artifact>/<version>/` directory and relying on
  cached GitHub artifacts — risky if the cached versions are also stale or incomplete.
- Fixing GitHub Packages auth first — correct long-term but blocks immediate work.

### Why this is non-obvious

The symptom (runtime incompatibility) points away from the dependency layer entirely.
`mvn -U` is the standard "fix my SNAPSHOTs" command and appears to succeed. The
`_remote.repositories` file is an internal Maven implementation detail that most
developers never inspect. The locally-installed jar is silently preferred with no
warning in Maven output.
