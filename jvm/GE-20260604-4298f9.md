## Symptom

A test that doesn't use `@InjectMock` fails with an assertion mismatch on a string value
(`actorRole`, `statusCode`, etc.) after a new `@InjectMock` field is added elsewhere in the
same `@QuarkusTest` class. The failure is in a test that appears completely unrelated to the
new mock.

## Context

Any `@QuarkusTest` class where:
1. A new `@InjectMock SomeService someService` field is added to support a new test
2. `SomeService` is injected by other beans exercised in existing tests
3. The existing tests don't stub `someService`, so Mockito returns defaults (null for
   String, 0 for int, false for boolean, empty Optional, etc.)
4. Production code has a conditional that branches on null/0/false from the injected bean

## Root Cause

`@InjectMock` replaces the CDI bean in the Quarkus test context for the **entire test
class**, not just for the annotated test method. All tests in the class see the Mockito mock
instead of the real bean.

Mockito's default return values for unstubbed calls:
- `String` → `null`
- `int`/`Integer` → `0`
- `boolean`/`Boolean` → `false`
- `Optional` → empty Optional

If production code has a null-check, empty-check, or zero-check that guards an alternate
code path, that path fires unexpectedly — without any error or warning. The test sees a side
effect (wrong audit role, wrong HTTP status, missing event) with no obvious connection to the
newly-added mock.

## Example

```java
// Before: DefaultPiIdentityResolver returns "dr-smith@v1" (passthrough)
// After: @InjectMock PiIdentityResolver added for new PI resolver tests

@QuarkusTest
class SponsorNotificationListenerTest {

    @InjectMock PiIdentityResolver piIdentityResolver; // new field
    // ... all existing tests now see the mock, not the real bean

    @Test
    void unexpected_exception_writes_observer_failure_entry() {
        // This test doesn't stub piIdentityResolver.
        // resolveFormalName("dr-smith@v1") returns null (Mockito default).
        // Null-return guard fires, writes "sponsor-notifier-pi-resolver-failed"
        // instead of "sponsor-notifier-observer-failed".
        // Test fails: assertThat(entry.actorRole).isEqualTo("sponsor-notifier-observer-failed")
    }
}
```

## Fix

Add a default stub in `@BeforeEach` so all tests in the class receive a safe, non-null
return. Individual tests override the stub to test specific values.

```java
@BeforeEach
void setUp() {
    // Default stub prevents null-return guard from firing in tests that don't
    // need to control the resolver's output. Individual tests override as needed.
    when(piIdentityResolver.resolveFormalName(any())).thenReturn("Dr. Smith");
    // ... other setup
}
```

## Why This Is Non-Obvious

The class-wide scope of `@InjectMock` is documented, but the downstream effect is not:
unstubbed methods return null, which silently activates conditional code paths rather than
failing visibly. The test output shows a wrong assertion value — not a null pointer exception
and not a "method not stubbed" warning. The failing test often has no mention of the new mock
field. The connection between "added @InjectMock for X" and "test for Y now fails" requires
understanding that all tests in the class are affected and that Mockito's null returns are
consumed silently by production null guards.
