## H2 does not support partial UNIQUE indexes even in MODE=PostgreSQL

### Symptom

A Flyway migration containing:

```sql
CREATE UNIQUE INDEX my_idx ON my_table (col1, col2) WHERE some_col = TRUE;
```

causes Flyway startup to fail with a SQL syntax error in `@QuarkusTest` using H2.

### Context

Occurs when writing a Flyway migration that needs a partial/filtered unique index for production PostgreSQL (e.g. to prevent duplicate rows for a specific flag value). The migration file is shared between H2 (tests) and PostgreSQL (production).

### Root Cause

H2 `MODE=PostgreSQL` adds compatibility for many PostgreSQL-specific constructs, but partial (filtered) indexes are not among them. The `WHERE` clause on a `CREATE INDEX` statement is not supported. H2 2.x documentation does not explicitly list this as unsupported, making it easy to assume it works.

### Fix

Drop the `WHERE` clause from the migration file. The partial index must be applied separately on production PostgreSQL (via a DBA script or a PostgreSQL-only migration):

```sql
-- In the Flyway migration (H2-compatible):
ALTER TABLE my_table ADD COLUMN my_flag BOOLEAN NOT NULL DEFAULT FALSE;
-- Partial unique index cannot be here — applied separately on PostgreSQL

-- Production PostgreSQL only (DBA script or manual migration):
CREATE UNIQUE INDEX my_idx ON my_table (col1, col2) WHERE my_flag = TRUE;
```

Document the gap in a GitHub issue so the production index is not forgotten.

### Why Non-Obvious

H2's PostgreSQL mode compatibility list is not exhaustive. The syntax error message from H2 does not say "partial indexes are not supported" — it says something about unexpected `WHERE` token, which could be mistaken for a quoting or naming issue.
