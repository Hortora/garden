### Symptom
A `FlywayConfigurationCustomizer @ApplicationScoped` bean without a `@FlywayDataSource` qualifier is expected to apply to ALL configured Flyway datasources (default + named). In practice it only applies to the default datasource. Named datasources (e.g. `qhorus`, `reporting`) are unaffected.

### Context
A Quarkus extension provides a `FlywayConfigurationCustomizer` bean to add a migration path at JVM startup. The application has both a default datasource and one or more named datasources with their own Flyway configurations. The concern is whether the unqualified customizer would apply to named datasources and add unexpected migration paths.

### Root cause (confirmed from bytecode)
`FlywayContainerProducer.matchingConfigCustomizers(List, String datasourceName)` implements this routing:

1. For each customizer bean, iterate its CDI qualifiers
2. If the bean has `@FlywayDataSource(value)` and `value.equals(datasourceName)` → include ✅
3. If the bean has NO `@FlywayDataSource` annotation AND `DataSourceUtil.isDefault(datasourceName)` → include ✅
4. Otherwise → exclude

The key: step 3 applies unqualified customizers only when `DataSourceUtil.isDefault()` returns true for the datasource being configured. Named datasources fail this check and skip unqualified customizers entirely.

This is confirmed at lines 175–184 of the compiled `FlywayContainerProducer.class` in `quarkus-flyway-3.32.2.jar`.

### Fix / Use
If the customizer should apply to a SPECIFIC datasource, annotate it:

```java
@ApplicationScoped
@FlywayDataSource("mydb")  // only applied when Flyway is configured for "mydb" datasource
public class MyDbMigrationCustomizer implements FlywayConfigurationCustomizer { ... }
```

If the customizer should apply to the DEFAULT datasource only, no qualifier is needed — that is already the behavior.

If the customizer should apply to ALL datasources (default + named), you need a separate customizer bean per datasource name, each annotated with the appropriate `@FlywayDataSource`.

### Why this is non-obvious
The Quarkus documentation on `FlywayConfigurationCustomizer` does not describe the multi-datasource routing behavior. The natural reading of "no qualifier = matches all" (standard CDI semantics) does not apply here. The actual behavior (`DataSourceUtil.isDefault()`) is only visible from the compiled `FlywayContainerProducer` class.
