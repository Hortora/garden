### Context

A table models a discriminated union using a VARCHAR discriminator column and two nullable key columns. Each discriminator value implies a specific nullity pattern for the keys. Application code is supposed to enforce this — but ORM bugs, direct SQL writes, and data migrations all bypass application logic.

### Technique

Add a multi-column CHECK constraint that enforces the state machine at the database level:

```sql
CONSTRAINT chk_actor_trust_score_keys CHECK (
    (score_type = 'GLOBAL'               AND capability_key IS NULL      AND dimension_key IS NULL    ) OR
    (score_type = 'CAPABILITY'           AND capability_key IS NOT NULL   AND dimension_key IS NULL    ) OR
    (score_type = 'DIMENSION'            AND capability_key IS NULL       AND dimension_key IS NOT NULL) OR
    (score_type = 'CAPABILITY_DIMENSION' AND capability_key IS NOT NULL   AND dimension_key IS NOT NULL)
)
```

Combined with `UNIQUE NULLS NOT DISTINCT (actor_id, capability_key, dimension_key)`, invalid combinations can never be inserted regardless of application bugs. Every valid state is listed explicitly — the constraint doubles as inline documentation.

Supported in PostgreSQL 15+ and H2 2.4.240+ (the version pulled by Quarkus 3.32+), so it works in both production and the standard Quarkus test database.

### Why this is non-obvious

Most schemas rely on per-column NOT NULL constraints and application-level validation. A CHECK constraint spanning multiple columns to enforce a state machine is less common — but it survives ORM bugs, raw SQL writes, and future migrations that might set a column without updating the discriminator. The database is always the last line of defence.
