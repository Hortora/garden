## Symptom

Compilation error when calling `httpClient.send()` inside a `forEach` lambda: "unhandled exception type InterruptedException". Or worse: `InterruptedException` is silently swallowed by a bare `try/catch (IOException e)` that doesn't cover it, leaving the scheduler in an inconsistent state after thread interruption.

## Context

Iterating a collection of endpoints and calling `httpClient.send()` for each, inside a `forEach` lambda:

```java
endpoints.forEach(endpoint -> {
    httpClient.send(request, BodyHandlers.ofByteArray()); // does not compile — throws InterruptedException
});
```

## Root Cause

`java.net.http.HttpClient.send()` declares `throws IOException, InterruptedException`. `InterruptedException` is a checked exception. `Consumer<T>.accept()` — the functional interface used by `forEach` — does not declare any checked exceptions. Java's lambda rules require the lambda body to handle all checked exceptions that the functional interface's abstract method doesn't declare. The compiler rejects the lambda.

Even if wrapped in `try/catch (Exception e)`, swallowing `InterruptedException` without re-interrupting the thread corrupts the thread's interrupt state. Frameworks like Quarkus Scheduler use thread interruption to signal shutdown — swallowing it prevents graceful shutdown.

## Fix

Catch `InterruptedException` separately, call `Thread.currentThread().interrupt()` to preserve the interrupt flag, and re-throw as `IOException` (which `forEach` accepts via the surrounding `try/catch (Exception e)`):

```java
endpoints.forEach(endpoint -> {
    try {
        pollAndFire(endpoint);
    } catch (Exception e) {
        LOG.warnf(e, "Poll failed for %s", endpoint.url());
    }
});

void pollAndFire(EndpointDescriptor d) throws IOException {
    String url = d.url();
    HttpRequest request = HttpRequest.newBuilder().GET().uri(URI.create(url)).build();
    HttpResponse<byte[]> response;
    try {
        response = httpClient.send(request, BodyHandlers.ofByteArray());
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();                      // preserve interrupt flag
        throw new IOException("Poll interrupted for " + url, e); // becomes caught by forEach's catch (Exception e)
    }
    // ... check status code and use response
}
```

The key is: `pollAndFire()` declares `throws IOException`, which `Consumer<T>.accept()` still cannot accept — so `pollAndFire()` cannot be the lambda directly. Wrapping it in a lambda with its own `try/catch(Exception e)` works.

## Why This Is Non-Obvious

Most HTTP client libraries (OkHttp, Apache HttpClient) do not throw `InterruptedException`. Developers unfamiliar with `java.net.http.HttpClient` don't expect it. The two-exception throw signature is easy to miss in the Javadoc. The consequence of silently swallowing `InterruptedException` (broken scheduler shutdown) is not visible during normal operation — only at application shutdown.
