## CDI @Alternative bypass for quarkus:build: non-@Alternative subclass in app module

### What

When `quarkus.arc.selected-alternatives` fails to activate `@Alternative` beans during `quarkus:build` (see GE-20260524-2b587e), create a non-`@Alternative @ApplicationScoped` subclass of each needed `@Alternative` bean directly in the application module's own source.

### Why it works

CDI does NOT inherit `@Alternative` from superclasses. A subclass without `@Alternative` is treated as a regular `@Default @ApplicationScoped` bean, satisfying injection points without any `selected-alternatives` configuration. Quarkus always indexes the application module's own source, so the subclass is always discovered тАФ no `quarkus.index-dependency` required.

### Code

For each `@Alternative @ApplicationScoped` SPI implementation in a dependency jar:

```java
// In the application module (src/main/java):
package com.example.app.spi;

import com.lib.InMemoryEventLogRepository;
import jakarta.enterprise.context.ApplicationScoped;

@ApplicationScoped  // no @Alternative тАФ this is @Default
public class AppEventLogRepository extends InMemoryEventLogRepository {}
```

Repeat for each SPI. Inject CDI fields from the parent class are satisfied normally тАФ CDI handles injection into inherited fields when creating the child bean.

### When to use this

- `quarkus.arc.selected-alternatives` isn't activating alternatives during `quarkus:build`
- In-memory/dev implementations needed in production build without real backend (tutorial projects, early-stage dev)
- Want zero `selected-alternatives` configuration while keeping implementation reuse

### Why this is non-obvious

Developers reach for `quarkus.arc.selected-alternatives` (the documented mechanism). When that fails silently, the natural next attempt is compile-scope + index-dependency тАФ which also fails. The subclass approach requires knowing that `@Alternative` is not inherited by subclasses, which is a CDI spec detail most developers haven't needed to rely on.
