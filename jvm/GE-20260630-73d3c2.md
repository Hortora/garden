### Symptom

A record declared as immutable (carrying `Map<String, float[][]>` or similar array-containing fields) allows callers to mutate internal state through the auto-generated accessor, even when the map is wrapped in `Collections.unmodifiableMap()`.

### Context

When designing `InferenceOutput` as a record with `Map<String, float[][]> outputs`, the unmodifiable map wrapper prevents adding/removing keys but does NOT prevent mutating the `float[][]` values inside the map. The auto-generated `outputs()` accessor returns the component directly — no defensive copy.

### Root cause

Java records auto-generate a public accessor for every component that returns the component value directly. For reference types, this is the exact same object — no copy. `Collections.unmodifiableMap()` only protects the map structure (put/remove), not the values it contains. Since `float[][]` is a mutable reference type, `record.outputs().get("key")[0][0] = 999f` mutates the record's internal state.

### Fix

Use a **final class** instead of a record when the type carries mutable array payloads and immutability is required:

```java
public final class InferenceOutput {
    private final Map<String, float[][]> outputs;

    public InferenceOutput(Map<String, float[][]> outputs) {
        // Deep defensive copy on construction
        var copy = new LinkedHashMap<String, float[][]>();
        for (var entry : outputs.entrySet()) {
            float[][] original = entry.getValue();
            float[][] cloned = new float[original.length][];
            for (int i = 0; i < original.length; i++) {
                cloned[i] = original[i].clone();
            }
            copy.put(entry.getKey(), cloned);
        }
        this.outputs = Collections.unmodifiableMap(copy);
    }

    public float[][] output(String name) {
        // Deep defensive copy on access
        float[][] result = outputs.get(name);
        float[][] cloned = new float[result.length][];
        for (int i = 0; i < result.length; i++) {
            cloned[i] = result[i].clone();
        }
        return cloned;
    }
}
```

### Why this is non-obvious

Records feel immutable — they are final, their components are final, and the auto-generated `equals`/`hashCode` make them look value-typed. The mental model is "record = immutable data." But for array-containing fields, the auto-generated accessor silently breaks this contract. The compiler gives no warning, and the mutation is invisible to `equals()` (which uses reference equality for arrays, not content equality). GE-0023 covers `List<T>` fields specifically; this entry covers the broader array case including nested arrays like `float[][]`.
