## Symptom

Attempting to import a custom `Path` type (e.g. a value object in an API module) into a JAX-RS resource class produces a compile error or unexpected behaviour because `jakarta.ws.rs.Path` is already in scope via annotation processing.

## Context

In any JAX-RS resource class that uses `@Path` for route declarations, importing any other `Path` type creates a naming conflict. The JAX-RS `@Path` annotation is processed at class level; adding an import for another `Path` type causes ambiguity the Java compiler cannot resolve.

## What was tried

Adding a standard import `import io.casehub.platform.api.path.Path;` alongside `@Path("/foo")` resource annotation.

## Root cause

Both `jakarta.ws.rs.Path` (as `@Path`) and any user-defined `Path` class share the simple name `Path`. JAX-RS resources always implicitly deal with the JAX-RS `@Path` annotation — there is no safe import for any other `Path` type in the same file.

## Fix

Use the fully qualified class name everywhere in the file. Do not import the custom `Path` type at all:

```java
// ❌ Will not compile or creates ambiguity:
import io.casehub.platform.api.path.Path;

// ✅ Use fully qualified name inline:
public Response addDefinition(String raw) {
    io.casehub.platform.api.path.Path labelPath =
        io.casehub.platform.api.path.Path.parse(raw);
    ...
}
```

## Why this is non-obvious

The conflict only surfaces as a compile error after you write the import. The instinct is to import the custom type and assume JAX-RS annotations are handled separately — they are not from the perspective of the simple name `Path`.
