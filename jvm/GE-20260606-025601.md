### Symptom
`UnsatisfiedLinkError` at runtime in native image when calling `OrtEnvironment.getEnvironment()`:

```
java.lang.UnsatisfiedLinkError: ai.onnxruntime.OrtEnvironment.createHandle(JILjava/lang/String;)J
[symbol: Java_ai_onnxruntime_OrtEnvironment_createHandle]
```

### Context
ONNX Runtime's `OnnxRuntime` class loads the JNI native library (`libonnxruntime4j_jni.dylib`) in its static initializer via `System.loadLibrary()`. Other classes in the `ai.onnxruntime` package have static initializers that depend on the library being loaded.

### What was tried (didn't work)
- `--initialize-at-run-time=ai.onnxruntime.OnnxRuntime` (class-level) — the library loader class defers correctly, but `OrtEnvironment` and other classes in the package initialize before `OnnxRuntime` and expect the native library to be already loaded

### Root cause
Class-level `--initialize-at-run-time` only defers the named class. Other classes in `ai.onnxruntime` (particularly `OrtEnvironment`) have their own static initialization that calls native methods. When `OrtEnvironment` initializes at runtime before `OnnxRuntime`, the native library isn't loaded yet — `UnsatisfiedLinkError`.

### Fix
Use package-level deferral in `native-image.properties`:

```properties
Args = --initialize-at-run-time=ai.onnxruntime
```

Not:

```properties
Args = --initialize-at-run-time=ai.onnxruntime.OnnxRuntime
```

Package-level ensures all classes in `ai.onnxruntime` initialize at runtime in the JVM's natural order, which correctly triggers `OnnxRuntime` static init (and library loading) before any native methods are called.

### Why this is non-obvious
The ONNX Runtime documentation (what little exists for GraalVM) suggests registering `OnnxRuntime` specifically. The class-level approach seems correct — it defers the library loader to runtime. The error (`UnsatisfiedLinkError` on `OrtEnvironment.createHandle`) doesn't point to `OnnxRuntime` at all — it points to `OrtEnvironment`, which makes it look like `OrtEnvironment` needs its own registration rather than that `OnnxRuntime` needs broader scope.
