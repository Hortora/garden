## Symptom

A `@Observes StartupEvent` method with no `@Priority` annotation appears to run *before* another observer annotated with `@Priority(10)`. In reality, the opposite is true: the undecorated observer fires at the CDI default priority (2500), which is *much later* than `@Priority(10)`.

The bug manifests as a startup workaround that "seems to work" because the system falls back to a sensible default — masking the fact that the CDI-injected value never reached its target in time.

**Real case:** `ObjectMapperInjector @Observes StartupEvent` (no @Priority) was meant to inject a CDI-managed `ObjectMapper` into a static field on `CaseDefinitionYamlMapper` before case definitions loaded. `DefaultCaseDefinitionRegistry.onStart(@Priority(10))` loaded definitions eagerly. The injector fired at priority 2500 — *after* the registry — so the CDI ObjectMapper never participated in definition loading. The fallback `new ObjectMapper(new YAMLFactory())` worked well enough that the bug went unnoticed for months.

## Root cause

CDI event observer priority follows the same semantics as Java interceptor priority: **lower number = higher priority = fires first**. CDI 4.0 specifies the default observer priority as `javax.interceptor.Interceptor.Priority.APPLICATION + 500 = 2500`.

- `@Priority(10)` → fires very early (almost first)
- No `@Priority` → fires at 2500 (application-tier default)

A developer expecting "no annotation = fires in some undefined order near the observer" will be surprised that it always fires *after* any explicitly-prioritised observer.

## Fix

Either:

1. Give the startup injector an explicit `@Priority` lower than any observer it must run before:
   ```java
   void onStartup(@Observes @Priority(1) StartupEvent event) { ... }
   ```

2. Eliminate the static injection pattern entirely. If the observer's target is a CDI bean, inject the dependency directly rather than via a static setter:
   ```java
   @Inject @YamlMapper ObjectMapper objectMapper; // injected before any StartupEvent fires
   ```
   CDI field injection completes *before* any `StartupEvent` observers fire, regardless of priority.

## Why this is non-obvious

The natural reading of "no @Priority = unordered" aligns with how many other frameworks handle default ordering. CDI's explicit default of 2500 is buried in the spec and not surfaced by Quarkus tooling. A static setter that "works" despite wrong ordering hides the bug indefinitely — the fallback default happens to be functionally acceptable.

### Why this fix

Option 2 (direct CDI injection) is the correct design because it removes the static mutable state entirely. Option 1 merely papers over the ordering problem without fixing the architectural smell.
