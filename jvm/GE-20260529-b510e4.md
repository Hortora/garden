## @Transactional on @ObservesAsync observer fires on every event

### Symptom

`@QuarkusTest` integration tests fail with `ClosedConnectionException: Failed to read
any response from the server` across ALL test classes. The DB container appears to die
mid-test. Adding `@Transactional` to a new observer method is the only recent change.

```
java.util.concurrent.CompletionException: io.vertx.sqlclient.ClosedConnectionException:
  Failed to read any response from the server, the underlying connection may have been lost
  unexpectedly.
Suppressed: javax.transaction.xa.XAException: Error trying to transactionRollback local
  transaction: This connection has been closed.
```

### Context

A new CDI `@ObservesAsync` method filters events by type and returns early for the vast
majority of events:

```java
@Transactional
public void onCaseLifecycleEvent(@ObservesAsync CaseLifecycleEvent event) {
    if (!TERMINAL_EVENTS.contains(event.eventType())) return; // early return for ~95% of events
    // ... writes to DB for the ~5% that are terminal
}
```

In integration tests, the engine fires many `CaseLifecycleEvent`s per case (started,
worker scheduled, worker completed, signal received, etc.). Under concurrent test load,
the connection pool is exhausted.

### What was tried (didn't work)

- Increasing Podman memory (appeared to help but didn't fix the core issue)
- Suspected Testcontainers infrastructure instability

### Root cause

The `@Transactional` interceptor wraps the ENTIRE method call — not just the code paths
that perform database work. For every event, the interceptor calls `begin()`, invokes the
method, then calls `commit()` or `rollback()`. This opens a JDBC connection from the pool
on each call, even when the method body returns immediately without touching the database.

Under high event frequency (common in integration tests that run full case scenarios),
hundreds of concurrent transaction beginnings exhaust the connection pool. The pool
exhaustion manifests as connection errors on the DB container, causing ALL test classes
to fail.

### Fix

Remove `@Transactional` from the observer method. If the database operation itself needs
a transaction, let the `CaseMemoryStore` (or the called service) manage it internally:

```java
// Wrong — @Transactional runs on EVERY event
@Transactional
public void onCaseLifecycleEvent(@ObservesAsync CaseLifecycleEvent event) {
    if (!TERMINAL_EVENTS.contains(event.eventType())) return;
    store.get().store(input);  // NoOp store doesn't need a transaction
}

// Right — no outer transaction; store manages its own if needed
public void onCaseLifecycleEvent(@ObservesAsync CaseLifecycleEvent event) {
    if (!TERMINAL_EVENTS.contains(event.eventType())) return;
    store.get().store(input);
}
```

### Why this is non-obvious

`@Transactional` seems like the right annotation for a method that "might write to the
database" — the intention is to guard the write, not wrap every invocation. The interceptor
doesn't know about the early return; it can't see that 95% of calls will never touch the
database. The failure also looks exactly like an infrastructure problem (container OOM,
network loss) rather than a connection pool exhaustion caused by application code.
