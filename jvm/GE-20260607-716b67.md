## Symptom

After renaming an enum constant, existing rows in a TEXT/JSON database column that used that constant as a `Map` key fail to deserialise with `InvalidFormatException: Cannot deserialise value of type X from String "OLD_NAME"`.

## Context

A Java record with a `Map<SomeEnum, String>` field is serialised to and from a database TEXT column via Jackson. The enum is used as the map key. The enum constant is renamed as part of a refactor.

## Root Cause

Jackson serialises `Map<Enum, String>` keys using the enum constant's `.name()` (its Java identifier) by default. Stored JSON contains `{"OLD_NAME": "..."}`. After the rename, `SomeEnum.valueOf("OLD_NAME")` throws, causing Jackson to fail the entire row deserialisation.

This is distinct from ordinal serialisation (which most developers know to avoid by using `@JsonSerialise`). Name-based serialisation _looks_ safe because names are stable strings — but stored names are brittle the moment a constant is renamed.

## Fix

Before renaming any enum constant used as a JSON map key in stored data, migrate the stored JSON:

```sql
UPDATE my_table
SET json_column = REPLACE(json_column, '"OLD_NAME"', '"NEW_NAME"')
WHERE json_column LIKE '%"OLD_NAME"%';
```

Run the migration _before_ deploying the code that removes the old constant name.

For future-proofing, consider `@JsonProperty` on enum constants to decouple the JSON key from the Java identifier:

```java
public enum MyAxis {
    @JsonProperty("social-orientation")
    SOCIAL_ORIENTATION,
    ...
}
```

With `@JsonProperty`, the stored key is the annotation value and is immune to Java-side renames.

## Why This Is Non-Obvious

Most Java developers know to avoid `@JsonValue` ordinals and reach for name-based serialisation as the "safe" alternative. Name-based serialisation IS safe for transient JSON (REST responses, message payloads) — names change rarely and consumers update. It is NOT safe for durable stored JSON in database columns, where the historical serialised form must remain deseralisable indefinitely. The failure only surfaces at runtime when reading old rows.
