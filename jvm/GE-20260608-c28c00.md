### What exists but isn't documented

`MessageService.dispatch()` contains this guard (from decompiled bytecode):

```java
if (dispatch.correlationId() != null) {
    switch (dispatch.type()) {
        case STATUS:
            this.commitmentService.acknowledge(dispatch.correlationId());
            break;
        case DONE: case RESPONSE:
            this.commitmentService.fulfill(dispatch.correlationId());
            break;
        // ... DECLINE, FAILURE, HANDOFF
    }
}
```

When `correlationId` is null, the entire switch block is skipped. A STATUS
dispatch without correlationId causes:
- **No** `commitmentService.acknowledge()` call
- **No** `Commitment.acknowledgedAt` update
- **No** commitment state change of any kind
- The message is persisted and fanned out, but it is purely archival

### Why it matters

This is the correct behavior for archival STATUS dispatches (e.g. delivering
agent webhook text output without signaling commitment state). The Qhorus
dispatch enforcement gate docs describe the enforcement sequence but not the
correlationId guard behavior.

Corollary: if an agent skips `casehub_commit` (the tool that dispatches STATUS
with correlationId → `acknowledge()`) and goes directly to `casehub_done`,
the commitment lifecycle is OPEN → FULFILLED with no ACKNOWLEDGED intermediate
state. `Commitment.acknowledgedAt` is null. Any audit query using
`acknowledgedAt != null` to detect explicit agent acceptance will see nothing.
This is deliberate — the null is meaningful (direct completion without explicit
acknowledgment), not a missing-data artifact.
