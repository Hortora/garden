### Symptom

`externalBean.someMethod()` returns `null` inside a `@QuarkusComponentTest`, even though the bean is `@ApplicationScoped` and is on the classpath. No exception, no warning â€” just silent null returns.

### Context

`@QuarkusComponentTest` boots a lightweight CDI container scoped to the test class's module. Any `@ApplicationScoped` bean from an **external** JAR that the component under test injects is auto-stubbed as a proxy that returns `null` (or default values) for every method call. This is by design â€” the component test framework cannot scan external JARs automatically.

Occurs when: a bean under test injects a bean from another module (e.g. `casehub-platform-expression:JQEvaluator`), and the test is written as `@QuarkusComponentTest` without explicitly declaring the external bean.

### What was tried

Writing `@QuarkusComponentTest` with only the component under test declared. The injected external bean was wired, CDI started without error, but all method calls returned `null`. Root cause only discovered by bytecode-inspecting the `QuarkusComponentTest` annotation class to find the `value` attribute.

### Root cause

`@QuarkusComponentTest` scans only the annotated test class's local package. External `@ApplicationScoped` beans are silently auto-stubbed â€” there is no warning that this has happened. The `value` attribute (array of `Class<?>`) is the mechanism for explicitly listing beans that should be wired as real instances rather than stubs.

### Fix

List all external beans and their CDI dependencies in the `value` attribute:

```java
@QuarkusComponentTest({
    ComponentUnderTest.class,
    ExternalBean.class,
    MockDepA.class,   // @DefaultBean from external module
    MockDepB.class
})
class MyComponentTest {
    @Inject ComponentUnderTest component;
    // ...
}
```

If the external bean has injected dependencies (e.g. `SecretManager`, `ConfigManager`), those must also be listed â€” or `@DefaultBean` implementations from the same module will be picked up automatically if they are included.

### Also: artifact rename in Quarkus 3.31

`quarkus-junit5-component` was renamed to `quarkus-junit-component` in Quarkus 3.31. Maven issues a relocation warning if the old artifact ID is used. Update `pom.xml`:

```xml
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-junit-component</artifactId>  <!-- not quarkus-junit5-component -->
  <scope>test</scope>
</dependency>
```

### Why this is non-obvious

- No error or warning when external beans are auto-stubbed â€” they appear to be wired
- The `value` attribute is not mentioned in the Quarkus `@QuarkusComponentTest` documentation
- Only discoverable by bytecode-inspecting the annotation class to find the attribute names
- The artifact rename compounds the issue: using the old artifact name causes a different kind of confusion
