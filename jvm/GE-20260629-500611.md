## Symptom

`ClassCastException: class [B cannot be cast to class java.util.UUID` when reading UUID columns from native query results in H2 MODE=PostgreSQL.

## Context

When using `EntityManager.createNativeQuery()` with `SELECT ... FOR UPDATE` (or any native SQL), H2 returns UUID columns as `byte[]` (16 bytes) even in PostgreSQL compatibility mode. JPQL and Criteria queries work correctly because Hibernate's type system handles the conversion. The mismatch only surfaces with native queries — which are required for database-level locking patterns like `SELECT FOR UPDATE`.

## What was tried (didn't work)

1. Casting the result directly to `UUID` — ClassCastException
2. Using `getResultList()` with typed query — native queries don't support typed results for non-entity projections
3. Setting H2 `DATABASE_TO_UPPER=false` — no effect on UUID column type handling

## Root cause

H2's PostgreSQL compatibility mode (`MODE=PostgreSQL`) aligns SQL syntax and function behaviour but does NOT align JDBC type mapping for all column types. UUID columns are stored internally as `byte[]` and returned as such in native `ResultSet` access. PostgreSQL's JDBC driver returns `java.util.UUID` directly.

## Fix

Add a conversion helper that handles both representations:

```java
private UUID toUuid(Object value) {
    if (value instanceof UUID uuid) return uuid;
    if (value instanceof byte[] bytes) {
        ByteBuffer bb = ByteBuffer.wrap(bytes);
        return new UUID(bb.getLong(), bb.getLong());
    }
    throw new IllegalArgumentException("Cannot convert to UUID: " + value.getClass());
}
```

Apply to every UUID column read from native query results. The helper is safe for both H2 (receives `byte[]`) and PostgreSQL (receives `UUID`).

## Why this is non-obvious

H2's `MODE=PostgreSQL` creates a strong expectation that type handling matches PostgreSQL. The mode documentation lists syntax compatibility but does not document JDBC type mapping differences. The `byte[]` representation is correct for H2's internal storage format but surprising when the mode promises PostgreSQL semantics.

### Alternatives considered

- **Use JPQL instead of native queries** — not possible when `SELECT FOR UPDATE` with specific locking semantics is required (JPQL `@Lock(PESSIMISTIC_WRITE)` works but doesn't give row-level control)
- **Use Testcontainers with real PostgreSQL** — correct long-term but adds container startup time to every test run; H2 is preferred for unit-speed integration tests
