### Symptom

Feature branch CI passes. After merging both PRs to main, the main branch CI fails with cascading test timeouts â€” `SimpleCaseHubBeanTest`, `SpiWiringIntegrationTest`, `MultipleCaseInstancesTest` all time out at 10â€“15 seconds. The failing tests don't use the library that changed.

The actual error is buried in WARN logs:

```
ERROR: relation "ledger_subject_sequence" does not exist
[MERGE INTO ledger_subject_sequence AS t USING (SELECT CAST(? AS UUID) AS sid) ...]
```

### Context

A SNAPSHOT library (`casehub-ledger 0.2-SNAPSHOT`) was updated mid-session. The update added `LedgerSequenceAllocator`, a CDI bean that issues a MERGE against a `ledger_subject_sequence` **table** (not a PostgreSQL sequence â€” a custom table). This table is created by the library's Flyway migrations. Modules that don't run those migrations don't have it.

The library is a **test-scoped** dependency of the runtime module, so its CDI beans (`CaseLedgerEventCapture`) are discovered and activated. `CaseLedgerEventCapture` observes `CaseLifecycleEvent` and calls `LedgerSequenceAllocator` on every case start/completion. Without the table, every case fails silently as a CDI observer exception â€” the case never completes â€” which manifests as test timeouts in the engine integration tests.

### Root cause

A SNAPSHOT dependency changed its internal implementation between two CI runs (branch CI and main CI, hours apart). The new implementation added a runtime DB requirement that the consuming module's test schema doesn't include. The branch CI used the pre-update SNAPSHOT; main CI downloaded the post-update one.

### Why this is non-obvious

1. **Cascading timeouts hide the real failure** â€” the tests that fail (case lifecycle tests) don't mention the ledger at all. Nothing in the timeout error points to CDI observer failures.
2. **Branch CI passes, main CI fails** â€” looks like a merge conflict or test ordering issue, not a SNAPSHOT update.
3. **The symptom is a missing TABLE, not a missing sequence** â€” `ledger_subject_sequence` sounds like a PostgreSQL sequence but it's a custom table managed by a `MERGE` statement.
4. **SNAPSHOT instability is invisible in `pom.xml`** â€” the version string `0.2-SNAPSHOT` looks stable; there's no indication it changed.

### Fix

1. Identify the CDI observer that's failing: check WARN logs for `CaseLifecycleEvent observer failed` and look at the SQL in the error body.
2. Exclude the observer beans from CDI in the failing module's test `application.properties`:
   ```properties
   quarkus.arc.exclude-types=\
     io.casehub.ledger.service.CaseLedgerEventCapture,\
     io.casehub.ledger.service.WorkerDecisionEventCapture
   ```
3. Or: add the library's Flyway migrations to the test classpath:
   ```properties
   quarkus.flyway.locations=db/ledger/migration,db/engine-ledger/migration
   ```

### Why this fix

Excluding the CDI beans is correct when the test module doesn't need audit ledger capture â€” it tests the engine, not the ledger. Running ledger migrations in the engine test schema would couple the two modules unnecessarily.
