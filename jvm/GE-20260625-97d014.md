### Problem

Designing an SPI where implementations have different capability tiers (e.g. chat platforms: Slack has threads+reactions, IRC has only messaging). Four common approaches all fail:

- **A — Single interface, Optional returns:** interface grows monotonically; callers see methods that don't work
- **B — Capability interfaces + instanceof:** caller branches on capabilities at every call site
- **C — Single interface, silent degradation:** dishonest API — methods exist but silently no-op
- **D — Feature enums:** adding capabilities requires modifying the enum — closed for extension

### Solution — Composed Capabilities with Auto-Degrading Builder

Each capability is a focused single-method interface. The platform composes them. A builder auto-provides named degradation defaults for anything the platform doesn't natively support.

```java
public interface ChatPlatform {
    String id();
    Messaging messaging();     // required — builder enforces
    Threading threading();     // auto-degrades to ChannelFallbackThreading
    Reactions reactions();     // auto-degrades to NoOpReactions
    boolean supports(Class<?> capability);  // class-token, not enum
    static Builder builder(String id) { return new Builder(id); }
}

// Platform provides only what it supports — rest auto-degrades
ChatPlatform irc = ChatPlatform.builder("irc")
    .messaging(new IrcMessaging(client))
    .build();
// supports(Threading.class) → false
// threading().reply(...) → delegates to messaging.send() via ChannelFallbackThreading
```

**Key properties:**
1. Degradation types are first-class named classes — testable, reusable across platforms
2. `supports(Class<?>)` uses the capability interface class — genuinely open for extension
3. Callers never *must* branch — but *can* query when needed (e.g. UI decisions)
4. Builder auto-computes `supports()` from which capabilities were explicitly provided
5. Adding a new capability = new interface + new degradation default. No existing type changes.

### Why this is non-obvious

The standard SPI patterns (A/B/C) all handle capability variation at the method level. This pattern handles it at the *composition* level — the platform object is assembled from capability implementations, with degradation as the default, not the exception. The builder makes the capability profile declarative and visible in one place.

### Suitability

Works well for SPIs where:
- Platforms have genuinely different capability sets (not just configuration differences)
- Callers should be able to write uniform code regardless of platform
- Degradation is semantically meaningful (not just "throw UnsupportedOperationException")

Less suitable for SPIs where all implementations must support all operations (use a simple interface instead).
