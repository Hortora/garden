### Symptom
`Capability.of("cap", null, null)` throws `NullPointerException` at runtime — compiles fine.

### Context
The `casehub-worker-api` `Capability` record's compact constructor calls `Objects.requireNonNull` on `inputSchema` and `outputSchema`. The old engine `Capability` class was mutable and accepted null for these fields.

### What was tried (didn't work)
Using `Capability.of(name, null, null)` as a direct replacement for `new Capability(name, null, null)` — the factory method delegates to the record constructor which rejects null.

### Root cause
Record compact constructors run validation before field assignment. The worker-api Capability enforces non-null schemas as a design constraint — capabilities must declare their schemas.

### Fix
Use empty-passthrough schemas instead of null:
```java
Capability.of("cap", "{}", "{}")
// or
Capability.of("cap", ".", ".")
```

### Why this is non-obvious
The NullPointerException occurs at runtime, not compile time. Tests that construct Capabilities with null schemas compile and pass checkstyle but fail at runtime, making the migration appear incomplete when it's actually a contract change.
