### Symptom

Quarkus startup fails with `Configuration validation failed:` followed by multiple `SRCFG00050` errors:

```
SRCFG00050: casehub.openclaw.context-window.buffer-size in PropertiesConfigSource[...]:35 does not map to any root
```

The app never starts ‚Äî this is treated as fatal, not a warning.

### Context

Two categories of properties trigger this:

1. **Truly orphan properties** ‚Äî old config names left in `application.properties` that no `@ConfigProperty` or `@ConfigMapping` consumes (e.g. renamed from `buffer-size` to `max-messages-per-channel`).

2. **Properties consumed by `@ConfigProperty` in library modules** ‚Äî these ARE consumed at runtime, but Quarkus augmentation can't discover them at build time if the consuming bean is in a library JAR without a Jandex index, or the bean's discovery is conditional.

### What was tried (didn't work)

- `%dev.quarkus.config.mapping.validate-unknown=false` in `application.properties` ‚Äî property not recognised
- `quarkus.config.mapping.validate-unknown=false` without profile ‚Äî not recognised

### Root cause

Quarkus 3.26 made strict config validation the default. SmallRye Config's `SRCFG00050` fires for any property in a config source that doesn't map to a registered `@ConfigMapping` root. `@ConfigProperty` annotations are discovered individually and may not be visible during augmentation if the consuming bean is excluded or conditional.

### Fix

Pass as a JVM system property (not a Quarkus config property):

```bash
mvn quarkus:dev -Dsmallrye.config.mapping.validate-unknown=false
```

Also clean up truly orphan properties ‚Äî remove old config names that were replaced by new ones.

### Why this is non-obvious

The error message says "does not map to any root" ‚Äî this sounds like a `@ConfigMapping` issue, but it also fires for `@ConfigProperty` properties in library modules. The Quarkus property name (`quarkus.config.mapping.validate-unknown`) doesn't work in `application.properties` ‚Äî only the SmallRye system property works. No Quarkus documentation explains this distinction.
