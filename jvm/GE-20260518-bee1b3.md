## Virtual-thread offload + CDI self-injection: pattern for safe blocking JPA in a reactive pipeline

### What it is

When a blocking JPA repository must be called from a reactive (`Uni`-returning) context, combine two patterns:

1. **Virtual-thread offload** via `runSubscriptionOn(Infrastructure.getDefaultWorkerPool())` â€” moves execution off the Vert.x IO thread
2. **CDI self-injection** â€” calls the `@Transactional` method through the CDI proxy so the interceptor fires

```java
@ApplicationScoped
public class JpaCaseLineageQuery implements CaseLineageQuery {

    @Inject @LedgerPersistenceUnit EntityManager em;

    @Inject JpaCaseLineageQuery self;   // CDI proxy â€” ensures @Transactional intercepts

    @Override
    public Uni<List<WorkerSummary>> findCompletedWorkers(UUID caseId) {
        return Uni.createFrom()
                  .item(() -> self.blocking(caseId))            // call through proxy
                  .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());  // virtual thread
    }

    @Transactional(TxType.REQUIRED)
    public List<WorkerSummary> blocking(UUID caseId) {
        return em.createQuery(
                "SELECT e FROM CaseLedgerEntry e WHERE e.caseId = :caseId " +
                "AND e.eventType = 'WorkerExecutionCompleted' ORDER BY e.occurredAt ASC",
                CaseLedgerEntry.class)
            .setParameter("caseId", caseId)
            .getResultList()
            .stream().map(this::toSummary).toList();
    }
}
```

### Why a skilled developer wouldn't reach for this

The obvious approach is to annotate the `Uni`-returning method itself with `@Transactional` â€” that's where the transaction semantics are needed. But `@Transactional` on a `Uni`-returning method opens and closes the transaction before any subscriber subscribes: the transaction is gone before the query runs.

The second instinct is `Uni.createFrom().item(() -> this.blocking(caseId))` â€” but `this` calls the concrete object, bypassing the CDI proxy and silently skipping `@Transactional`.

The correct answer requires knowing both: (a) CDI interception only fires through a proxy reference, and (b) CDI self-injection (`@Inject MyBean self`) provides that proxy reference inside the same class.

### When to use it

- You have blocking JPA code (blocking EntityManager, JPQL queries) that must be called from a `Uni`-returning method
- You cannot or do not want to migrate the JPA layer to Hibernate Reactive
- The blocking method needs CDI interceptors (`@Transactional`, `@Retry`, etc.) to fire

### Trade-offs

- Adds a worker thread hop â€” appropriate for I/O-bound JPA queries, not for in-memory operations
- `Infrastructure.getDefaultWorkerPool()` uses virtual threads on Java 21+ â€” lightweight at scale
- The `blocking()` method must be `public` for CDI proxy interception (CDI cannot intercept `private` or `package-private` methods)

### Alternative â€” simpler variant when `@Transactional` interception is not needed

When the blocking call does **not** need `@Transactional` interception (e.g. a Panache `listAll()` read that
Panache handles internally), skip the CDI self-injection entirely:

```java
// In a reactive @Tool method (Uni<T> return, no @Blocking annotation) or reactive dashboard method:
public Uni<List<ChannelDetail>> listChannels() {
    return Uni.createFrom().item(bindingStore::findAll)          // blocking Panache call
            .runSubscriptionOn(Infrastructure.getDefaultWorkerPool())  // dispatch to worker thread
            .flatMap(bindings -> channelService.listAll()
                    .flatMap(channels -> { /* reactive chain */ }));
}
```

**Why the worker pool dispatch is still required:** Reactive `@Tool` methods in `quarkus-mcp-server`
that return `Uni<T>` **without** `@Blocking` run on the Vert.x event loop by default. Any
blocking call (including Panache `listAll()`) on the event loop freezes the loop. The
`runSubscriptionOn(Infrastructure.getDefaultWorkerPool())` hop is the fix â€” it dispatches
the supplier to a virtual thread before the downstream reactive chain executes.

Without this, the blocking call runs on the Vert.x event loop with no warning â€” the
server may appear to hang under concurrent request load.

**When to use this simpler form:**
- The blocking call is a read (no `@Transactional` required at the call site)
- Panache ambient transaction handling is sufficient (e.g. `Panache.withTransaction` not needed)
- No CDI interceptors need to fire on the called method

**When to use the full form (CDI self-injection above):**
- The called method needs `@Transactional` or other CDI interceptors to fire explicitly
