## Conditional UPDATE on a row that may not exist — returns 0 rows, indistinguishable from "already claimed"

### What went wrong

A conditional JPQL UPDATE used as an exactly-once claim mechanism:

```java
int updated = em.createQuery(
    "UPDATE SituationEntity s SET s.policyTriggered = true " +
    "WHERE s.situationId = :sid AND s.correlationKey = :ck " +
    "AND s.tenancyId = :tid AND s.policyTriggered = false")
    .executeUpdate();
boolean claimed = updated > 0;
```

This works when the entity row exists — exactly one concurrent caller wins the UPDATE, others get 0 rows. But when the entity does not exist (no row in the table), the UPDATE also returns 0 rows. The caller interprets this as "already claimed by another JVM" when in fact nothing was claimed at all.

### Context

The pattern was adopted from GE-20260512-e3e525 (OCC + policyTriggered flag for M-of-N threshold completion), where the entity always exists because earlier events caused CONTINUE_ACCUMULATING saves. Generalizing to all trigger modes broke the single-event case — for OR mode with a single ganglion, the very first event evaluates to CREATE_CASE without any prior save. No entity exists.

### Root cause

The conditional UPDATE pattern assumes the target row exists. When it doesn't, the 0-row result is ambiguous — it could mean "row exists but flag already set" or "row doesn't exist at all." The pattern was designed for accumulating systems where the entity is always created before the claim is attempted.

### Fix

Bifurcated claim path based on whether the entity exists:

- **New entity (storeVersion empty):** save-before-claim — create the row first, then claim
- **Existing entity (storeVersion present):** claim-before-save — claim first, then save (avoids unnecessary field updates)

The distinction is made using `storeVersion` from the domain model — `OptionalLong.empty()` means the context was created via `initial()` (no prior save), `isPresent()` means it was loaded from the store.

### Why this is non-obvious

The conditional UPDATE pattern is a well-known concurrency primitive. The trap is that it only provides exactly-once semantics when the target row is guaranteed to exist. The "0 rows = already claimed" interpretation is correct in the M-of-N case but silently wrong in the first-event case. The failure mode is complete: the most basic happy path (single event triggers case creation) is broken, but all unit tests pass because the in-memory store uses a different claim mechanism.

### Alternatives considered

- **INSERT ... ON CONFLICT:** Would combine row creation with conditional flag setting in a single atomic operation. JPA/JPQL does not support this — would require native SQL, losing portability.
- **Always save before claim:** Creates the row for the claim, but has the side effect of refreshing `lastSignal` on every post-trigger event, preventing expiry. The bifurcated path avoids this.
