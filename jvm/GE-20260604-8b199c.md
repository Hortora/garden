### Symptom
`McpServerIntegrationTest` fails CI with:
```
JSON path result.tools.size() doesn't match.
Expected: <60>
  Actual: <62>
```
No indication in the failure message that this was caused by a dependency update
rather than a code regression.

### Context
The test asserts an exact tool count at the unified MCP endpoint. The count is the
sum of the host project's own tools (8 Claudony tools) plus all tools registered by
an embedded library (Qhorus, via `casehub-qhorus` extension). The embedded library
ships new tools independently — the host project has no control over when this happens.

### Root cause
`equalTo(60)` hardcodes a total that includes tools owned by an external dependency.
When the dependency adds tools, the assertion fails with no signal that the cause is
external — the failure looks identical to a regression where the host accidentally
removed or broke tool registration.

### Fix
Two options:

**Option 1 — Assert only your own tools explicitly:**
```java
.body("result.tools.name", hasItems("create_session", "delete_session", /* your tools */))
// no size assertion — don't own the total
```

**Option 2 — Assert a minimum floor, not an exact count:**
```java
.body("result.tools.size()", greaterThanOrEqualTo(8)) // only assert your own floor
```

If you need to assert the exact total (e.g. to catch regressions in the embedded
library itself), add a comment that the number comes from `N own + M library` and
must be bumped in both the test and any count references in CLAUDE.md whenever the
library releases new tools.

### Why this is non-obvious
The failure message gives zero signal about the root cause. "Expected 60, Actual 62"
reads as a regression — something broke tool registration — not as "a dependency you
embed shipped new features." Investigation naturally starts by looking at what *you*
changed, not at whether an upstream dependency updated its tool surface.
