## Enforce blocking/reactive SPI method parity with a reflection test — silent drift causes downstream compile failures

### What it achieves

Catches method additions to a blocking SPI that were not mirrored in its reactive counterpart — before a downstream consumer discovers the gap at compile time.

### Context

When a blocking SPI (`LedgerEntryRepository`) and a reactive mirror (`ReactiveLedgerEntryRepository`) evolve independently, adding a method to the blocking interface doesn't immediately break anything — no concrete implementation of the reactive SPI lives in the extension module. Downstream consumers discover the gap when they try to call a reactive method that doesn't exist.

### The technique

Write a plain JUnit test (no `@QuarkusTest`) using reflection. Maintain an explicit exclusion set for methods that are intentionally blocking-only:

```java
@Test
void reactiveSpi_coversAllBlockingSpiMethods() {
    Set<String> blockingNames = Arrays.stream(LedgerEntryRepository.class.getDeclaredMethods())
            .map(Method::getName)
            .collect(Collectors.toSet());
    Set<String> reactiveNames = Arrays.stream(ReactiveLedgerEntryRepository.class.getDeclaredMethods())
            .map(Method::getName)
            .collect(Collectors.toSet());
    blockingNames.remove("findAllEvents"); // batch operation with no reactive equivalent
    assertThat(reactiveNames)
            .as("Reactive SPI must cover all blocking SPI methods")
            .containsAll(blockingNames);
}
```

No `@QuarkusTest` needed — pure reflection, no CDI context required. Add a companion test asserting every reactive method returns `Uni<?>` to prevent blocking return types creeping in.

### Why this is non-obvious

The compile failure is in the **downstream** module, not the SPI module where the gap exists. Without this test, the SPI author has no signal that the reactive mirror is incomplete until a consumer's build breaks.
