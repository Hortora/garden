### Symptom
`IdentityHashMap` keyed by a method reference accessor (`Function1<CTX, DataSource<T>>`) fails to deduplicate entries when the same method reference (e.g. `CTX::persons`) is used across multiple `builder.rule(...).from(CTX::persons)` call sites. Each call site's accessor is treated as a distinct key, producing multiple router slots and breaking incremental patching logic.

### Context
Building an incremental Rete evaluation engine (vol2 drools) where each `DataSource` accessor maps to a Router slot. Multiple rules referencing the same DataSource via the same method reference (`CTX::persons`) should share a slot. An `IdentityHashMap<Object, Integer>` is used to assign slot numbers by accessor identity, relying on the assumption that the same method reference expression produces the same JVM instance across call sites.

The original codebase comment stated: *"IdentityHashMap is correct — the JVM compiles the same method reference expression to the same lambda class, so references to CTX::persons across rules are identity-equal."* This assumption is wrong.

### What was tried (didn't work)
Using `IdentityHashMap` directly keyed by the accessor — works when all rules are built at the same call site (e.g. in a single `.add(...).add(...)` chain). Breaks silently when rules are built at separate call sites.

### Root cause
The JVM's `invokedynamic` instruction caches lambda/method-reference instances **per call site** (one bootstrap per `invokedynamic` bytecode instruction), not per method reference expression. Two occurrences of `CTX::persons` at two different source locations in the same class compile to two distinct `invokedynamic` instructions, each with its own bootstrap and its own cached instance. Identity comparison therefore fails — the two accessors reference the same method but are not the same object.

This contrasts with constants: `CTX::persons` appears once in the bytecode as one `invokedynamic`, so repeated calls to the same expression return the same cached instance.

### Fix
Use `SerializedLambda` to extract a stable, content-based key. Since method references must implement `Serializable` to support this, the accessor type must extend a serializable functional interface:

```java
// Accessor type must extend Serializable
interface Function1<A, B> extends java.util.function.Function<A, B>, java.io.Serializable {}

// Stable key extraction (done once per accessor, at compile() time)
private static String accessorKey(Object accessor) {
    try {
        java.lang.reflect.Method writeReplace = accessor.getClass()
                .getDeclaredMethod("writeReplace");
        writeReplace.setAccessible(true);
        java.lang.invoke.SerializedLambda sl =
                (java.lang.invoke.SerializedLambda) writeReplace.invoke(accessor);
        return sl.getImplClass() + "#" + sl.getImplMethodName();
    } catch (Exception e) {
        throw new RuntimeException("Accessor must be a serializable method reference", e);
    }
}

// Usage: HashMap (not IdentityHashMap) keyed by stable string
Map<String, Integer> sourceToSlot = new HashMap<>();
sourceToSlot.putIfAbsent(accessorKey(accessor), slot);
```

### Why this is non-obvious
The JVM specification allows — but does not require — instance caching for lambda/method-reference invocations. HotSpot does cache per call site, which means the assumption "same method reference → same instance" holds when all references come from the same source location but silently fails across locations. The official Java documentation describes lambdas as being evaluated at runtime without specifying instance identity guarantees, so there is no documented signal that this will fail.
