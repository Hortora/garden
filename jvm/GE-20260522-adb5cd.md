### Symptom
`@QuarkusTest` startup fails with N "Unsatisfied dependency for type X" errors.
No compile error. The bean is fully visible to the Java compiler.

### Context
A CDI `@ApplicationScoped` bean is extracted from the Quarkus application module
into a shared library JAR (a Maven dependency). Works fine before the move.

### Root Cause
Quarkus only auto-indexes the application module for CDI bean discovery. Library
JARs on the classpath are **not** scanned unless explicitly listed in
`quarkus.index-dependency`. Java compilation sees the class; the Quarkus CDI
container does not.

### Fix
Add to every `src/test/resources/application.properties` that needs the bean:
```properties
quarkus.index-dependency.<key>.group-id=<groupId>
quarkus.index-dependency.<key>.artifact-id=<artifactId>
```
Required in every module that runs `@QuarkusTest` and injects the bean — not
just the module that moved it.

### Why This Is Non-Obvious
The Java compiler resolves the type from the dependency JAR — it compiles and
links correctly. Nothing in the build output signals the problem. The failure
only appears at `@QuarkusTest` application startup, and the error message
("Unsatisfied dependency") looks like the bean doesn't exist, not that it
isn't indexed.
