## Symptom

Tests that fetch Quinoa-served static files (e.g. `/app/app.js` built by esbuild) return 404 in `@QuarkusTest`. The Quinoa build does not run, and no files are served from the `build-dir`.

## Context

Quinoa works correctly in dev mode (`mvn quarkus:dev`) and production (`mvn package`). The build runs, files are served, hot reload works. But `@QuarkusTest` silently skips the entire Quinoa pipeline — no warning, no error.

## What was tried (didn't work)

- `%test.quarkus.quinoa.enabled=true` in `application.properties` — ignored
- `quarkus.quinoa.just-build=false` — ignored
- `-Dquarkus.quinoa.enabled=true` system property — ignored
- Checking for a test-mode config property in the Quinoa docs — none exists

## Root cause

`QuinoaProcessor` (the Quarkus deployment extension) checks `LaunchMode.TEST` at augmentation time and skips the build/serve steps entirely. This is hardcoded — no config property gates it. The intent is to avoid slow npm builds during every test run, but the behaviour is undiscoverable: no log message, no config opt-in, no documentation of the limitation.

## Fix

Use `@QuarkusIntegrationTest` instead of `@QuarkusTest` for tests that verify Quinoa-served files. Integration tests run the application in production mode where Quinoa is active.

Alternatively, pre-build `dist/` via a Maven exec plugin bound to `generate-test-resources` so the files exist in `META-INF/resources/` before test augmentation.

### Why this fix

`@QuarkusIntegrationTest` exercises the real Quinoa build and matches production behaviour. The pre-build alternative duplicates build artifacts and can drift from the production build.

## Why this is non-obvious

All runtime config properties (`quarkus.quinoa.enabled`, profile overrides, system properties) are silently ignored because the check happens at build time, not runtime. The Quinoa extension gives no indication that it has skipped itself — the test simply gets 404s, which looks like a misconfiguration rather than a fundamental limitation.

### Alternatives considered

- Copying the built `dist/` to `META-INF/resources/` in a test-resources phase — works but creates a maintenance burden and can silently serve stale bundles
- Running the npm build manually before `mvn test` — fragile, not integrated into the Maven lifecycle
