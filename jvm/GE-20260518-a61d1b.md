## Symptom / Discovery

A `@ConsumeEvent` handler needs to call `@Transactional` services *and* start its own transaction (rather than relying on each called service to start its own). The question is whether placing both `@ConsumeEvent(blocking=true)` and `@Transactional` on the same method is supported.

## Context

Quarkus dispatches `@ConsumeEvent(blocking=true)` handlers on a worker thread pool (not the Vert.x IO thread). CDI interceptors — including the `@Transactional` interceptor — apply to `@ApplicationScoped` beans by default. When the event bus invokes the handler method via the CDI proxy, the `@Transactional` interceptor wraps the call, starting a JTA transaction before the method body and committing or rolling back when it exits.

## What works

```java
@ApplicationScoped
public class MyHandler {

    @ConsumeEvent(value = "my-address", blocking = true)
    @Transactional
    public void handle(MyEvent event) {
        // All JPA/JTA operations here share one transaction.
        // If anything throws, the whole transaction rolls back.
        serviceA.create(...);  // joins this transaction (REQUIRED semantics)
        serviceB.update(...);  // joins this transaction
    }
}
```

The `@Transactional` annotation is applied by the CDI proxy when the event bus invokes the handler. Both annotations are active simultaneously.

## Why this is not in the docs

Quarkus documents `@ConsumeEvent` and `@Transactional` separately. The interaction between them on the same method — specifically that `blocking=true` is necessary for `@Transactional` to work correctly, and that combining both annotations is valid — is not explicitly documented. The CLAUDE.md convention for this codebase states: "`@ConsumeEvent` handlers that call `@Transactional` services must use `blocking=true`" but does not note that `@Transactional` on the handler itself is also supported.

## Verified behaviour

Tests using `@QuarkusTest` confirmed that a `@ConsumeEvent(blocking=true) @Transactional` handler:
- Starts a single JTA transaction for the full method body
- Rolls back all JPA writes (including multiple service calls) if any step throws
- Commits atomically when the method exits cleanly
