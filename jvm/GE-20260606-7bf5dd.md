### Symptom
Native binary crashes with a Rust panic immediately when `HuggingFaceTokenizer.newInstance()` is called:

```
thread '<unnamed>' panicked at src/lib.rs:518:10:
called `Result::unwrap()` on an `Err` value: JavaException
fatal runtime error: failed to initiate panic, error 5, aborting
```

Exit code 134 (SIGABRT). No Java stack trace, no indication of which class or method is missing.

### Context
DJL's HuggingFace Tokenizers module uses a Rust native library (`libtokenizers.dylib`) that calls back into Java via JNI to create `CharSpan` objects — character offset spans in tokenized text. In GraalVM native image, the `CharSpan` class and its constructor must be registered for JNI access.

### What was tried (didn't work)
- Adding `--initialize-at-run-time` for DJL classes (correct but not the cause of this error)
- Registering DJL resource files in `reachability-metadata.json` (necessary but insufficient)
- Reading the Rust panic message for clues (misleading — "JavaException" is opaque)

### Root cause
`ai.djl.huggingface.tokenizers.jni.CharSpan` is not registered for JNI access in the native image. The Rust tokenizer code calls `JNIEnv::NewObject` to create `CharSpan(int, int)` instances. Without the JNI registration, the JNI call throws a Java exception that the Rust code receives as a `JavaException` error variant and panics on `unwrap()`.

### Fix
Add to `reachability-metadata.json`:

```json
{
  "reflection": [
    {
      "type": "ai.djl.huggingface.tokenizers.jni.CharSpan",
      "jniAccessible": true,
      "methods": [
        {
          "name": "<init>",
          "parameterTypes": ["int", "int"]
        }
      ]
    },
    {
      "type": "ai.djl.huggingface.tokenizers.jni.CharSpan[]"
    }
  ]
}
```

### Why this is non-obvious
The Rust panic message (`Result::unwrap() on Err(JavaException)`) gives no indication that a missing JNI class registration is the cause. It looks like a Rust bug or a corrupted JNI environment. The `CharSpan` class is an internal implementation detail of DJL — it doesn't appear in the public API, so there's no reason to suspect it needs explicit registration. The GraalVM tracing agent is the only reliable way to discover this requirement.

### Alternatives considered
- Inspecting DJL source for all `native` method declarations — would find the JNI surface but not the Rust→Java callbacks, which are the actual failure point
- Using DJL's shipped `native-image.properties` — exists but doesn't cover the tokenizers module's JNI callbacks
