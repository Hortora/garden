## Symptom

`ExternalActorResponse.trustProfile.globalScore` is always null from the REST API,
even after seeding trust scores via `ActorTrustScoreRepository.upsert()` in tests.
No error is thrown — the upsert call returns normally and the subsequent read
returns empty `OptionalDouble`.

## Context

When casehub-ledger is on the classpath, two `ActorTrustScoreRepository` implementations
are available:

- `NoOpActorTrustScoreRepository` — `@Default`; all methods are no-ops or return empty
- `JpaActorTrustScoreRepository` — `@Alternative`; persists to the `actor_trust_score` table

If `JpaActorTrustScoreRepository` is not listed in `quarkus.arc.selected-alternatives`,
CDI picks `NoOpActorTrustScoreRepository` as the active bean. Every `upsert()` call
is silently discarded; every `findGlobalScore()` returns empty. The chain is:

```
TrustGateService
  → MaterializedTrustScoreSource (@Default TrustScoreSource)
    → ActorTrustScoreRepository (injected — resolves to NoOpActorTrustScoreRepository)
```

The parallel pattern exists for `JpaLedgerEntryRepository` (already documented), but
the trust score repository is separate and requires its own entry in `selected-alternatives`.

## What was tried

- Confirmed the REST endpoint returned 200 and the actor existed
- Verified the `trust-routing.yaml` was loaded correctly
- Seeded trust scores via `trustScoreRepo.upsert()` in `@Transactional` test helper

## Root cause

`JpaActorTrustScoreRepository` is `@Alternative`. Without explicit activation,
`NoOpActorTrustScoreRepository` wins the CDI resolution. Both `upsert()` and
`currentScore()` are effectively dead code.

## Fix

Add `JpaActorTrustScoreRepository` to `selected-alternatives` in **both**
`src/main/resources/application.properties` and `src/test/resources/application.properties`:

```properties
quarkus.arc.selected-alternatives=\
  io.casehub.ledger.runtime.repository.jpa.JpaLedgerEntryRepository,\
  io.casehub.ledger.runtime.repository.jpa.JpaActorTrustScoreRepository
```

## Why this is non-obvious

The call to `upsert()` succeeds silently — no exception, no warning. The only
observable symptom is a null score in the API response, which looks like a query
returning no results rather than a write that never happened. The `@Alternative` /
`@Default` split is not visible in any casehub-ledger documentation; it requires
reading the class bytecode or source to discover. The existing gotcha for
`JpaLedgerEntryRepository` (GE-20260615-*) covers the ledger entry repository
but not the trust score repository — there is no combined entry.

### Why this fix

Adding to `selected-alternatives` is the standard Quarkus pattern for activating
`@Alternative` beans and avoids any CDI startup warnings that `@Alternative @Priority`
would produce.
