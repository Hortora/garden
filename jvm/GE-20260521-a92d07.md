### Symptom

A Quarkus extension ships Flyway location defaults in its embedded `application.properties` jar resource. Consumer tries to override with their own `application.properties` in `src/main/resources/`. Override works correctly.

Contrast with GE-20260429-ede58e: extension defaults in `META-INF/microprofile-config.properties` are **not** overridable by the consumer's `application.properties` because of augmentation-time baking.

### Context

casehub-qhorus extension embedded `application.properties` containing:
```properties
quarkus.flyway.qhorus.locations=classpath:db/qhorus/migration
```

Consumer (casehub-aml) needed to override to also include ledger migrations:
```properties
quarkus.flyway.qhorus.locations=classpath:db/qhorus/migration,classpath:db/ledger/migration
```

Override succeeded — the consumer's locations config took effect.

### Root cause

Both files use ordinal 250 in SmallRye Config. When two sources share the same ordinal, **the consumer's `application.properties` in `src/main/resources/` wins** because it is loaded after the jar's embedded `application.properties`. This is documented nowhere — it is only discoverable by testing or reading SmallRye Config source.

`META-INF/microprofile-config.properties` uses ordinal 100, which is lower than `application.properties` (250). But this file is read at Quarkus **augmentation** time and baked in, bypassing the runtime ordinal resolution entirely — hence why it cannot be overridden.

### Fix

If you need to override a Quarkus extension's default configuration: check whether the extension uses `application.properties` (overridable) or `META-INF/microprofile-config.properties` (not overridable at test time). Inspect the jar directly:

```bash
jar tf <extension>.jar | grep "application.properties\|microprofile-config"
unzip -p <extension>.jar application.properties
```

If the extension uses `application.properties`, your consumer `application.properties` override will work. If it uses `microprofile-config.properties`, you must ensure the datasource is present (see GE-20260429-ede58e).

### Why this is non-obvious

Developers assume all extension-provided defaults behave like `microprofile-config.properties` (baked in, non-overridable). The ordinal difference between the two files, and the augmentation-vs-runtime distinction, is not documented in Quarkus configuration guides.
