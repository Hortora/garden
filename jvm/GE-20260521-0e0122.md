### Symptom
A `@Transactional(REQUIRES_NEW)` method catches `PersistenceException` (OCC or INSERT conflict)
and retries by recursively calling a private helper method. The retry appears to work in tests
(single-threaded, no concurrent conflict) but silently returns the fallback value under real
concurrency — never actually retrying.

### Root cause
When `acquireNext()` is annotated `@Transactional(REQUIRES_NEW)`, CDI starts the transaction
on the first call through the proxy. A recursive call to a private `tryAcquire()` method does
NOT go through the CDI proxy — it is a direct `this.method()` call. The "retry" runs inside the
already-poisoned REQUIRES_NEW transaction, which is marked rollback-only. Any JPA operation
immediately throws again (or returns stale data), and the catch falls through to the fallback.

### Fix

Self-inject the bean and call the retry method through the CDI proxy:

```java
@ApplicationScoped
public class JpaRoutingCursorStore implements RoutingCursorStore {

    @Inject
    JpaRoutingCursorStore self; // CDI proxy — each call starts a fresh REQUIRES_NEW

    @Override
    public int acquireNext(final String poolHash, final int poolSize) {
        try {
            return self.doAcquire(poolHash, poolSize);
        } catch (final PersistenceException e) {
            try {
                return self.doAcquire(poolHash, poolSize); // fresh REQUIRES_NEW
            } catch (final PersistenceException e2) {
                return 0; // predictable fallback
            }
        }
    }

    @Transactional(Transactional.TxType.REQUIRES_NEW)
    public int doAcquire(final String poolHash, final int poolSize) {
        // ... find-or-create cursor, advance, return index
    }
}
```

`acquireNext()` has no transaction annotation. Each `self.doAcquire()` call goes through the
proxy and starts a genuinely fresh `REQUIRES_NEW` transaction.

### Why this is non-obvious
The test suite passes — tests run single-threaded with no concurrent INSERT/OCC races, so the
retry path is never exercised. The bug only surfaces under concurrent load. The recursive call
"looks like" a retry but is invisible to the CDI interceptor. See also: GE-20260518-069f64
(same root cause in a Mutiny lambda context).
