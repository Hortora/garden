## Technique

When implementing a Qhorus `ChannelBackend`, register with `ChannelGateway` by observing `ChannelInitialisedEvent` rather than writing a separate startup recovery loop:

```java
@ApplicationScoped
public class MyChannelBackend implements ChannelBackend {

    @Inject ChannelGateway gateway;

    void onChannelInitialised(@Observes ChannelInitialisedEvent event) {
        if (!event.channelName().startsWith("my-prefix-")) return;
        gateway.registerBackend(event.channelId(), this, "agent");
    }

    // ... rest of ChannelBackend impl
}
```

## Why This Works

`ChannelInitialisedEvent` fires in two contexts:
1. **At channel creation** — `ChannelGateway.initChannel()` fires it every time a channel is initialised (creation or get).
2. **At Quarkus startup** — `ChannelGateway` observes `StartupEvent` and fires `ChannelInitialisedEvent` for all persisted channels, rebuilding the in-memory registry.

A backend that registers via this event is automatically re-registered on every restart without any additional recovery logic. This is the designed extension point documented in the qhorus deep-dive: "External backends observe this via `@Observes ChannelInitialisedEvent` to re-register without implementing their own startup recovery logic."

## Why Non-Obvious

The obvious approach is to inject `ChannelService`, list all channels at startup, and call `gateway.registerBackend()` for each. The event-driven approach eliminates the startup query, handles late-arriving channels (created after startup), and doesn't require knowing which channels exist at deploy time. Most implementations would write a startup listener; the event observer is more correct.
