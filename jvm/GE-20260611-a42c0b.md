### Symptom
`findByCaseId` and related queries return empty results or run against the wrong datasource in multi-datasource deployments. In single-datasource deployments everything works — no error, no warning.

### Context
JPA repository subclass extends a parent that injects `@LedgerPersistenceUnit EntityManager em`. The subclass adds its own EntityManager field for additional query methods. In a deployment with two datasources (e.g. engine default + qhorus), the unqualified field resolves to the default PU.

### Root cause
CDI `@Inject EntityManager` without a qualifier resolves to the **default** persistence unit. The parent's qualifier is not inherited by the subclass's own `@Inject` fields. Both EntityManagers are injected but target different PUs — the subclass silently queries the wrong one.

### Fix
Replicate the same CDI qualifier on the subclass's EntityManager field:

```java
// Before — silently uses wrong PU in multi-datasource
@Inject EntityManager caseEm;

// After — correct
@Inject @LedgerPersistenceUnit EntityManager caseEm;
```

Rule: every `@Inject EntityManager` field in a named-datasource class hierarchy must carry the named qualifier, regardless of which class in the hierarchy declares it.

### Why this is non-obvious
The parent class correctly qualifies its EntityManager and all parent queries work. The subclass compiles and works in single-datasource deployments where the unqualified injection accidentally resolves to the only available PU. The bug is invisible until a second datasource is added — at which point queries silently target the wrong PU with no error, no exception, and no logged warning.
