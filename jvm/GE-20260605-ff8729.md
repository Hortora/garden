## Context

Overriding a blocking SPI method (e.g. `storeAll(List<MemoryInput>)`) to persist N JPA
entities atomically in a single transaction with maximum JDBC efficiency.

## Technique

Build all entities first (using `stream().toList()` or any eager collection), then call
`MemoryEntry.persist(entries)` once with the complete `Iterable<MemoryEntry>`:

```java
@Override
@Transactional(TxType.REQUIRED)
public List<String> storeAll(List<MemoryInput> inputs) {
    if (inputs.isEmpty()) return List.of();
    var entries = inputs.stream().map(input -> {
        MemoryEntry e = new MemoryEntry();
        e.memoryId = UUID.randomUUID().toString();
        // ... populate fields
        return e;
    }).toList();

    MemoryEntry.persist(entries);   // ← single flush, all N entities

    return entries.stream().map(e -> e.memoryId).toList();
}
```

Configure JDBC batching in `application.properties`:
```properties
quarkus.hibernate-orm.jdbc.statement-batch-size=50
```

Panache delegates `persist(Iterable)` to `em.persist()` for each entity within the
same Hibernate session flush. With `statement-batch-size` configured, Hibernate groups
the resulting `INSERT` statements into a single batched `PreparedStatement`, reducing
N DB round-trips to ⌈N / batch-size⌉.

## Why This Is Non-Obvious

Developers reach for `entity.persist()` in a loop inside `@Transactional` — this IS
atomic (single commit) but does NOT enable JDBC batch mode. Hibernate issues individual
`INSERT` statements regardless of `statement-batch-size` when `em.persist()` is called
one entity at a time.

`persist(Iterable)` presents all entities to Hibernate before the flush, allowing the
batch assembler to group them. The Panache overload exists and works, but is easy to
miss because the common pattern is always the single-entity form.

## Bonus: fail-fast semantics

Building the entity list eagerly (`.toList()`) before calling `persist(Iterable)` means
any validation or security check inside the stream (e.g. `assertTenant()`) throws before
any entity reaches the persistence layer. Combined with `@Transactional(REQUIRED)`, this
gives all-or-nothing batch semantics: either all N entities are committed or none are.
