### Symptom

Build-time validation reports an unknown agent name `dev-ui` in the config map, even though `dev-ui` is a separate `@ConfigRoot` for dev tooling.

### Context

When `@WithParentName` is used on a `Map<String, AgentConfig>` in one `@ConfigRoot`, and a separate `@ConfigRoot` exists at a sub-prefix (e.g. `quarkus.langchain4j.agent.dev-ui`), SmallRye Config creates a phantom entry in the map for key `dev-ui` (all `Optional.empty()`).

### Root cause

The separate `@ConfigRoot` at `quarkus.langchain4j.agent.dev-ui` shares the parent prefix with the `@WithParentName` map. SmallRye sees `dev-ui` as both a separate config root AND a potential map key.

### Fix

Keep fixed-name config as a **sibling method** in the same `@ConfigRoot` — not a separate root. This is the pattern used by `LangChain4jBuildConfig` (`devservices()` alongside `@WithParentName Map<String, BaseConfig>`). SmallRye resolves fixed-name methods before map entries — no ghost entry.

```java
@ConfigRoot(phase = ConfigPhase.RUN_TIME)
@ConfigMapping(prefix = "quarkus.langchain4j.agent")
public interface AgenticRuntimeConfig {
    @WithParentName AgentConfig defaultConfig();
    @WithParentName @WithDefaults Map<String, AgentConfig> namedConfig();
    DevUiConfig devUi();  // fixed sibling, not separate @ConfigRoot
}
```

### Why this is non-obvious

The separate `@ConfigRoot` approach seems cleaner — dev tooling and agent config are different concerns. But the shared prefix namespace creates an invisible coupling.
