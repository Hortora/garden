## Symptom

Refactoring a flat `@ConfigMapping` to a nested structure causes an immediate `NullPointerException` in `setUp()` — not in a `@Test` method. The stack trace points to the `when(config.reviewer().personality())` line in setUp:

```
java.lang.NullPointerException
    at com.example.DraftHouseMcpToolsTest.setUp(DraftHouseMcpToolsTest.java:55)
```

Line 55 is:
```java
when(config.reviewer().personality()).thenReturn("You are a reviewer.");
```

## Context

After restructuring `DraftHouseConfig` from flat:
```java
@ConfigMapping(prefix = "casehub.drafthouse.reviewer")
interface DraftHouseConfig {
    String personality();  // directly accessible
}
```
to nested:
```java
@ConfigMapping(prefix = "casehub.drafthouse")
interface DraftHouseConfig {
    Reviewer reviewer();   // sub-interface access required
    interface Reviewer {
        String personality();
    }
}
```

Test setUp had:
```java
config = mock(DraftHouseConfig.class);
when(config.personality()).thenReturn("You are a reviewer.");  // worked before
```

After renaming to `when(config.reviewer().personality())`, setUp NPEs.

## Root cause

`mock(DraftHouseConfig.class)` uses Mockito's default answer (`RETURNS_DEFAULTS`), which returns `null` for any reference type method that has not been explicitly stubbed. `config.reviewer()` returns `null` because it has not been stubbed.

When Mockito evaluates `when(config.reviewer().personality())`, it actually **calls** both methods during the `when()` setup — `config.reviewer()` first, which returns `null`, then `.personality()` on `null`, which throws NPE. This happens at stub-setup time, not during test execution, so the error appears in `setUp()` rather than in a failing test.

## Fix

Stub the intermediate method **before** chaining to leaf stubs:

```java
// Two-level mocking — must stub intermediate first
config = mock(DraftHouseConfig.class);
DraftHouseConfig.Reviewer reviewer = mock(DraftHouseConfig.Reviewer.class);
when(config.reviewer()).thenReturn(reviewer);       // intermediate first
when(reviewer.personality()).thenReturn("You are a reviewer.");
when(reviewer.maxDocChars()).thenReturn(100_000);
```

**Alternative:** construct the mock with `RETURNS_DEEP_STUBS` — Mockito auto-creates and returns sub-mocks for chained calls:
```java
config = mock(DraftHouseConfig.class, RETURNS_DEEP_STUBS);
when(config.reviewer().personality()).thenReturn("You are a reviewer.");
```

`RETURNS_DEEP_STUBS` is convenient but produces over-mocked objects and makes it easy to forget to stub leaf calls, which then silently return null/0. Two-level explicit mocking is preferred for `@ConfigMapping` sub-interfaces.

## Why this is non-obvious

The NPE appears in `setUp()` rather than in the test method, which is counterintuitive — developers expect `when()` calls to be inert declarations. The error message doesn't mention "null from intermediate call"; it just shows an NPE at the stub line. Developers searching for the cause look at the configuration or the type, not at unstubbed intermediate methods.
