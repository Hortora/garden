### Symptom
Runtime error: `Scope predicate invocation failed: wrong number of arguments: N expected: N+1`. The predicate has more parameters than the facts array that gets passed to it.

### Context
A 2-source reactive rule (`from().join()`) with a lambda-scope `not()` whose predicate references both outer facts (left and right):

```java
builder.from(CTX::persons)
    .join(CTX::cities)
    .not(scope -> scope
        .join(CTX::blocklist)
        .filter((ctx, p, city, entry) -> p.name().equals(entry)))
    .ifn((ctx, p, city) -> ...);
```

The predicate has 4 parameters: `ctx + [person, city, entry]`.

### Root cause
`UnitInstance.scopeGuard()` wraps the **right inlet** (`JoinRightInlet`) for 2-source rules. When the right inlet fires, it receives only the right fact (city). The `scopeGuard` builds `outerFacts = new Object[]{ rightFact }` — just one element. For a scope with 2 outer placeholders (person + city), the scope evaluation only injects the city, not the person. The predicate then receives 3 args (ctx, city, entry) but expects 4 (ctx, p, city, entry).

### Fix
Wrap the **Consumer3** (not the inlet) for 2-source rules, where both facts are available:

```java
Consumer3<Context<CTX>, Object, Object> scopedConsumer = (c, leftFact, rightFact) -> {
    if (scopesAllow2(c, negations, existences, leftFact, rightFact))
        consumer.accept(c, leftFact, rightFact);
};
// Then pass scopedConsumer to both JoinLeftInlet and JoinRightInlet
```

The `scopesAllow2` helper receives `outerFacts = new Object[]{ leftFact, rightFact }`.

### Why this is non-obvious
The symptom (wrong argument count) doesn't point to the inlet-vs-consumer distinction. It looks like the scope predicate arity is wrong or the scope builder added the wrong number of joins. The actual cause — that the reactive model provides only one fact at a time at the inlet level — only becomes apparent when you trace where `outerFacts` is constructed in `scopeGuard`.
