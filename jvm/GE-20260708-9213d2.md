### Symptom

RRF or CC fusion returns fewer results than expected — distinct items
silently merge into one fused result. No error or warning.

### Context

Building a hybrid retrieval system that runs two separate queries (e.g.,
dense vector search + feature-only scroll), scores each leg independently,
then fuses using RRF or Convex Combination. The fusion algorithm needs an
ID extractor to match the same item across legs and accumulate scores.

### What was tried (didn't work)

Used a domain content field (`problem()` text) as the fusion ID. Seemed
natural — the problem text identifies what the case is about. Two cases
stored with `problem="alpha"` but different features (`race=Zerg` vs
`race=Protoss`) were silently merged into one fusion result.

### Root cause

Content fields are not unique identifiers. Multiple distinct items in the
store can share the same content value. When the fusion algorithm groups by
content-derived ID, it treats distinct items as the same candidate —
accumulating their scores and keeping only one instance.

### Fix

Use the storage system's unique point identifier (e.g., Qdrant UUID) as
the fusion ID. Propagate the point ID through the candidate reconstruction
pipeline so it's available at fusion time.

```java
// Wrong — content field, not unique
Function<ScoredCbrCase<C>, String> idExtractor = sc -> sc.cbrCase().problem();

// Right — storage-level unique ID
record FusionEntry<C>(String pointId, C cbrCase, double score) {}
Function<FusionEntry<C>, String> idExtractor = FusionEntry::pointId;
```

### Why this is non-obvious

Content-based deduplication is the standard approach in text retrieval (RAG
systems dedup by document ID + content hash). In CBR, the same problem text
can legitimately appear in multiple distinct cases with different features,
outcomes, or solutions. The mental model from RAG ("same content = same
document") doesn't transfer.

### Alternatives considered

- Hash of all case fields as fusion ID — fragile, breaks if any field changes between legs
- Object identity (`System.identityHashCode`) — doesn't work across reconstruction boundaries where the same stored case produces different Java objects
