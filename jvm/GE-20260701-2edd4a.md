### Context
A record component holds one source of truth (e.g. an author-declared override), but the record also carries an object whose method provides a second source of truth (e.g. a type-level default). The two must compose monotonically — once true from either source, the result is true.

### Technique
Override the record's auto-generated accessor to compose both sources:

```java
public record DesiredNode(NodeId id, NodeType type, NodeSpec spec, boolean requiresHuman) {
    @Override
    public boolean requiresHuman() {
        return requiresHuman || spec.requiresHuman();
    }
}
```

JLS §8.10.3: a record can explicitly declare an accessor method, suppressing the auto-generated one. Inside the body, unqualified `requiresHuman` references the component field (not the method), so there is no infinite recursion.

### Why this is non-obvious
Most developers treat record components as immutable data carriers with auto-generated accessors. The idea of overriding an accessor to compose it with another object's method is not a natural reach. The alternative — having every caller compute `field || spec.method()` — scatters the composition logic and risks inconsistency. The alternative of having the GoalCompiler pre-compute the OR — couples the compiler to spec internals.

### Alternatives considered
- Compiler pre-computes OR: works but couples every compiler to every spec's `requiresHuman()` implementation. Four compilers, each needing to know which specs override the default.
- Separate computed field: add a pre-computed `effectiveRequiresHuman` field — redundant data, synchronization risk, and a confusing API with two boolean fields.
