### Symptom
`OptimisticLockException: Unexpected row count (expected row count 1 but was 0)` during
`workItemStore.put(entity)` — but the failing UPDATE is on a *different* entity than the
one being explicitly saved. Stack trace points to `persistAndFlush()` → `SessionImpl.flush()`
→ a table that the calling code never wrote to.

### Context
A service method loads entity A (with `@Version`) for a read-only guard check, then calls
`persistAndFlush()` on entity B. A concurrent transaction modifies entity A between the
load and the flush. Hibernate flushes the entire persistence context, generates an UPDATE
for A with the stale version, and gets 0 rows back.

```java
// WorkItemService.claim() — loads SpawnGroup for allowSameAssignee check
final WorkItemSpawnGroup group =
    WorkItemSpawnGroup.findMultiInstanceByParentId(item.parentId);
if (group != null && !group.allowSameAssignee) { ... }

// Later in the same transaction:
workItemStore.put(item);   // persistAndFlush() — flushes SpawnGroup too!
// → OCC if MultiInstanceCoordinator updated SpawnGroup concurrently
```

### Root cause
`EntityManager.flush()` (called by `persistAndFlush()`) flushes **all** entities in the
current persistence context, not just the target. Any `@Version` entity that Hibernate
tracks — even one only read, not written — participates in dirty checking. Hibernate's
dirty checker considers a `@Version` entity dirty if its in-memory snapshot diverges from
what another transaction committed while it was tracked. Hibernate then generates an UPDATE
with the original version, which finds 0 rows.

The symptom appears intermittent because it only triggers when the concurrent transaction
commits *between* the read-only load and the `flush()` call — a narrow window that widens
under load or when async workers are active.

### Fix
Detach the read-only entity immediately after reading it:

```java
final WorkItemSpawnGroup group =
    WorkItemSpawnGroup.findMultiInstanceByParentId(item.parentId);
if (group != null) {
    em.detach(group);          // ← remove from persistence context
    if (!group.allowSameAssignee) {
        // safe to read detached values
    }
}
workItemStore.put(item);       // flush no longer touches SpawnGroup
```

Requires an `@Inject EntityManager em` in the service bean.

### Why this is non-obvious
The developer expects `persistAndFlush(entity)` to be scoped to that entity. Nothing in
the Panache API or the JPA spec surface communicates that flush is session-wide. The OCC
manifests on a *different* table, pointing investigation at the wrong code. The intermittency
(only when a concurrent transaction wins the race) makes it look like a locking bug elsewhere
rather than a flush-scope issue here. The fix — `em.detach()` after a read — is not a
pattern most developers reach for spontaneously.
