### Symptom
Two related pieces of state must transition atomically — e.g., a lifecycle phase
index and a graph reference — but they live in different concurrent data structures
(`ConcurrentHashMap` and `AtomicReference`). Locking both kills throughput in a
hot path (reconciliation loop).

### Context
A lifecycle manager tracks phase state in a `ConcurrentHashMap<String, TenantLifecycle>`
(immutable records, replaced atomically). The reconciliation loop holds the current graph
in an `AtomicReference<DesiredStateGraph>`. A phase transition must update both: advance
the phase index AND swap the graph. A concurrent replan (from a different thread) may
modify either at any time.

### Fix
**Double-CAS with rollback:** CAS the lifecycle map first (cheap, local), then CAS the
graph ref (shared, contended). If the second CAS fails, roll back the first.

```java
// 1. CAS lifecycle state (ConcurrentHashMap.replace)
TenantLifecycle next = new TenantLifecycle(current.phases(), nextIndex);
if (!lifecycles.replace(tenancyId, current, next)) {
    return; // concurrent modification — skip, re-evaluate next cycle
}

// 2. CAS graph ref
if (!loop.compareAndSetDesired(tenancyId, expectedGraph, nextPhase.graph())) {
    // Rollback lifecycle state — graph was changed concurrently
    lifecycles.replace(tenancyId, next, current);
    return; // re-evaluate next cycle
}
```

### Why this is non-obvious
The naive approach is to lock both structures. The insight is that both structures
are re-evaluated on every cycle — so a failed CAS just means "skip this transition
and try again next time." The rollback on the second CAS failure is safe because
`replace(key, next, current)` only succeeds if `next` is still the current value —
if a concurrent replan already replaced the lifecycle entry, the rollback harmlessly
fails and the replan's state wins.

The ordering matters: lifecycle state first (local concern, less contended), graph ref
second (shared with the reconciliation loop, more contended). Failing on the second CAS
is the common concurrent case — rolling back a `ConcurrentHashMap.replace` is cheaper
than rolling back an `AtomicReference.compareAndSet`.

### Alternatives considered
- Synchronized block covering both mutations (correct but serialises all phase transitions
  and replans — unacceptable in a hot reconciliation loop)
- Single atomic state object combining both (forces the reconciliation loop to know about
  lifecycle phases — violates separation of concerns)
- Eventual consistency (update both independently, accept temporary mismatch) — risks
  a cycle executing against the wrong graph for the current phase
