### What this enables

When a method annotated `@Transactional(REQUIRES_NEW)` receives JPA entities from the caller, those entities are attached to the **outer** persistence context. When REQUIRES_NEW suspends the outer context, the entities become detached in the new context. Any lazy association access inside the REQUIRES_NEW method throws `LazyInitializationException`.

The fix is to never pass JPA entities across the REQUIRES_NEW boundary. Instead, pass a plain Java record that carries all needed field values as primitives or value types — nothing that can be lazily loaded.

### Example

**BAD** — JPA entity crosses REQUIRES_NEW boundary:
```java
// Caller (outer @Transactional context)
Channel ch = channelStore.findById(channelId).orElseThrow();
Message msg = messageStore.put(newMessage);
ledgerWriteService.record(ch, msg);   // entities become detached inside record()

// LedgerWriteService
@Transactional(REQUIRES_NEW)
public void record(Channel ch, Message msg) {
    entry.channelId = ch.id;           // OK — primitive
    entry.content = msg.content;       // OK — primitive
    entry.someAssociation = msg.tags;  // LazyInitializationException if tags is lazy
}
```

**GOOD** — plain record carries what's needed:
```java
// Caller extracts primitives BEFORE calling REQUIRES_NEW method
Long messageId   = msg.id;
UUID commitmentId = msg.commitmentId;
Instant occurredAt = msg.createdAt != null ? msg.createdAt : Instant.now();
ledgerWriteService.record(dispatch, messageId, commitmentId, occurredAt);

// LedgerWriteService
@Transactional(REQUIRES_NEW)
public LedgerWriteOutcome record(MessageDispatch dispatch, Long messageId,
        @Nullable UUID commitmentId, Instant occurredAt) {
    // all values are primitives or value types — no lazy loading possible
}
```

`MessageDispatch` is a plain Java record (`record MessageDispatch(UUID channelId, String sender, ...)`) with no JPA annotations.

### Why this is non-obvious

Developers usually fix `LazyInitializationException` by adding `EAGER` fetch or calling `Hibernate.initialize()`. For REQUIRES_NEW, neither works: the entity is detached, not just uninitialized. Re-attaching via `entityManager.merge()` works but is verbose, requires an EM injection, and brings in unnecessary overhead. Using a plain value object is cleaner and enforces the transaction boundary as a design invariant — only values, never entities.

### Bonus effect

This pattern also documents the transaction boundary explicitly in the method signature. Future readers can see that the method takes primitives and value types — implying it may run in a different transaction context. The old `record(Channel, Message)` signature offered no such signal.
