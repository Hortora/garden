## Quarkus dev mode hot-reload silently breaks WebSocket endpoint registration — full restart required

### Symptom

After a Java change triggers a hot-reload in Quarkus dev mode, WebSocket connections return `101 Switching Protocols` but receive no messages. No error in logs. New connections appear to succeed but are silently non-functional.

### What was tried (didn't work)

- Waiting for hot-reload to fully complete before reconnecting — same result
- Reconnecting the WebSocket client — returns 101 but still silent

### Root cause

Quarkus dev mode hot-reload re-registers most beans and REST endpoints correctly, but WebSocket endpoint registration is not included in the reload cycle. After any reload, the previous WebSocket handler is gone and the new one is never registered — connections are accepted at the HTTP level but nothing handles the WebSocket messages.

### Fix

After any Java change in dev mode that triggers a reload, perform a **full server restart** (Ctrl+C → restart), not a hot-reload wait.

### Why this is non-obvious

Hot-reload works correctly for almost everything in Quarkus dev mode. WebSocket is a silent exception — no error, no warning, connections appear to succeed. The symptom looks like a client-side bug or a message routing issue, not a server registration failure.
