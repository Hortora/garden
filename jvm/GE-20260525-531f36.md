### Convention
Test method names use one underscore as a separator between the subject (what is under test) and the state or expected outcome. Both components are lowerCamelCase. Production methods remain camelCase throughout — the underscore is test-only.

```java
// ✅ Google style — subject_state
void ruleBaseFactory_createsInstance() {}
void apply_incrementalWhenEngineExists() {}
void singlePatternIfn_firesOnAdd() {}
void staleSession_throwsStaleSessionException() {}

// ❌ Multiple underscores (three-part name)
void singlePattern_ifn_firesOnAdd() {}
void apply_addRule_liveUnitFiresNewRule() {}

// ❌ Numbered tests (implies false ordering dependency)
void test1_inlineDslDeclaration_oneShot() {}

// ❌ PascalCase after underscore
void testFnFires_OnRemove_SinglePattern() {}

// ❌ Production method style (lost context without separator)
void singlePatternIfnFiresOnAdd() {}
```

### Pattern
`<subjectInContext>_<stateOrOutcome>`

Where context is merged into the subject: `singlePatternIfn`, `twoPatternWithFilter`, `applyWithExistingEngine`. The underscore marks exactly one boundary — between "what" and "what happens".

### Rationale
- One underscore makes the test name scannable in test output — the failure point is immediately visible
- lowerCamelCase in both parts follows Java naming conventions while preserving readability
- No numbers avoids implying test order dependencies (JUnit 5 doesn't guarantee execution order)
- Google Java Style Guide explicitly permits underscores in test method names and uses `subject_state` as the example pattern; AOSP requires it

### Known alternative
**camelCase throughout** (`singlePatternIfnFiresOnAdd`) — loses the at-a-glance scan for the state/outcome boundary, but follows production code style consistently.

### Source
Google Java Style Guide §5.2.3: *"Test method names may use underscores to separate logical components of the name, with each component written in lowerCamelCase. One typical pattern is `<methodUnderTest>_<state>`."*
