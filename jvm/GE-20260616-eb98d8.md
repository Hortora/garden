### Symptom

`ReactiveWorkerProvisioner.provision()` throws `ProvisioningException` (e.g., idempotent guard, disabled guard). The engine logs:

> WorkerProvisioner failed for capability 'X' on case Y — binding remains eligible: io.casehub.api.spi.ProvisioningException: ...

The case stays in RUNNING indefinitely. Signal calls (`CaseHubRuntime.signal(caseId, "workers.X.exited", true)`) appear to fire (`SignalReceived` ledger entry written) but no `CaseContextChangedEvent` follows and the case never reaches COMPLETED.

### Root cause

When `provision()` throws `ProvisioningException`, the engine schedules a **retry CONTEXT_CHANGED** approximately every 3 seconds ("binding remains eligible"). This retry fires its own `CONTEXT_CHANGED`, which:
1. Re-evaluates the binding (which may still pass — context hasn't been updated)
2. Calls `provision()` again → throws again → retry again

`SignalReceivedEventHandler` processes signals correctly but fires `CaseContextChangedEvent` by calling `SessionOperations.withSession()`. When the retry CONTEXT_CHANGED is in flight and its `CaseContextChangedEventHandler` is running (including during `tryProvision()`), the handler may hold the Vert.x lock that `SignalReceivedEventHandler` also needs, causing the signal's `CaseContextChangedEvent` to be silently dropped (the lock times out or the context version conflicts).

The net result: the signal fires (ledger write confirms it was received) but no goal evaluation runs — the case is stuck.

### Fix

Prevent `provision()` from returning a failing `Uni` for duplicate calls. Use a failing Uni with `ProvisioningException` **only** if the caller can tolerate silent retry suppression. For idempotency, use a guard that short-circuits at the Uni construction site before any engine event processing:

```java
Uni<Optional<UUID>> causedBy =
    (causalLinkResolver != null
     && context.triggerChannelId() != null
     && context.triggerCorrelationId() != null)
    ? causalLinkResolver.resolve(...)
    : Uni.createFrom().item(Optional.empty());
```

When `provision()` must short-circuit, return a never-completing `Uni` rather than a failing one — the engine parks the pending provision indefinitely without scheduling a retry.

### Why this is non-obvious

The engine's retry mechanism is not documented in the provisioner SPI. The 3-second interval is opaque. The signal appearing in the ledger creates a false impression that the signal chain worked — the ledger write confirms receipt but not propagation. The failure mode looks like a goal evaluation bug, not a provision retry loop.
