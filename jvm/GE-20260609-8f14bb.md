### Symptom

Spurious state-change events for the same sensor reading. Map value comparisons with `Objects.equals()` return `false` for two BigDecimal instances that represent the same numeric value.

### Context

Any Java record whose fields include `BigDecimal`. Records are trusted to "just work" for equality — but the auto-generated `equals()` delegates to `BigDecimal.equals()`, which is scale-sensitive.

```java
new BigDecimal("21").equals(new BigDecimal("21.0"))  // false — different scale
```

Triggered by two code paths that construct the same value with different constructors or string representations, then store results in a Map or compare with `Objects.equals()`.

### Root Cause

Java records auto-generate `equals()` by delegating to each component's `equals()`. `BigDecimal.equals()` is specified to return `false` for values with different scale, even when they are mathematically identical. `compareTo()` returns 0 for `21` vs `21.0`; `equals()` does not.

### What Was Tried (didn't work)

Standard equality test using `BigDecimal.valueOf(20)` for both sides — passed even with the bug because `BigDecimal.valueOf()` always returns scale 0. This gives false confidence that record equality is correct.

### Fix

Override `equals()` using `compareTo()` and `hashCode()` using `stripTrailingZeros()` in the record:

```java
public record Temperature(BigDecimal value, TemperatureUnit unit) {

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Temperature other)) return false;
        return unit == other.unit && value.compareTo(other.value) == 0;
    }

    @Override
    public int hashCode() {
        return Objects.hash(unit, value.stripTrailingZeros());
    }
}
```

`stripTrailingZeros()` normalises `21`, `21.0`, and `21.00` to the same canonical form for hashing, satisfying the `equals/hashCode` contract.

### Why This Is Non-Obvious

Records are the Java mechanism for eliminating equality boilerplate — developers trust the auto-generated implementation. `BigDecimal` is a standard library type; developers assume standard equality semantics. The standard equality test using `BigDecimal.valueOf()` always produces scale 0, so tests pass even when the bug is present. The symptom (spurious change events downstream) points away from the equality layer entirely.

### Why This Fix

Override at the equality layer, not at construction time. Normalising via `stripTrailingZeros()` at construction would change the stored precision representation. The fix is to make equality scale-insensitive where records participate in comparisons, leaving the stored value exact.

### Alternatives Considered

- Strip trailing zeros at construction time — changes stored precision, may affect serialisation
- Use `compareTo()` at every call site — doesn't fix `equals()` in the record, breaks collections (HashMap, contains, etc.)
