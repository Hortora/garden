### Symptom

Quarkus application with `quarkus-quinoa` dependency fails at startup with:

```
ConfigurationException: node-version is required to install package manager
(not required when using only bun-version)
```

### Context

Setting `quarkus.quinoa.package-manager-install=true` tells Quinoa to download and install Node.js itself (via a managed installer). This requires `quarkus.quinoa.node-version` to be set. When Node.js is already installed system-wide (via Homebrew, nvm, etc.), the correct setting is `package-manager-install=false` — Quinoa uses the system Node.js directly.

### Root cause

Quinoa 2.8.x changed the semantics of `package-manager-install`. Setting it to `true` activates a managed Node.js installation path that needs to know which version to download. The error is clear but the config key name is misleading — "install package manager" reads as "run npm install", not "install Node.js itself".

### Fix

When Node.js is already on `$PATH`:

```properties
quarkus.quinoa.package-manager-install=false
```

When you want Quinoa to manage its own Node.js:

```properties
quarkus.quinoa.package-manager-install=true
quarkus.quinoa.node-version=22.0.0
```

### Why this is non-obvious

The config key name `package-manager-install` suggests it controls whether `npm install` runs, not whether Node.js itself is installed. A developer adding Quinoa naturally sets it to `true` thinking "yes, install my packages" and gets a cryptic error about `node-version`.
