### Symptom
`@Inject @Any Instance<NamedStrategy>` returns an empty iterator even though
multiple `@ApplicationScoped` beans implementing `AgentRoutingStrategy extends NamedStrategy`
are registered and discoverable via `@Inject @Any Instance<AgentRoutingStrategy>`.

No error, no warning — the `Instance<>` is simply empty.

### Context
Designing a strategy resolver that discovers all beans extending a marker interface
(`NamedStrategy`) via a single `Instance<NamedStrategy>` injection point. The intent
is to index all strategies by type and id at startup. Standard CDI semantics imply
`Instance<T>` returns all beans whose type is assignable to `T`, including sub-interfaces.

### What was tried (didn't work)
- `@Inject @Any Instance<NamedStrategy>` — empty, no beans found
- Adding `@Unremovable` to strategy beans — still empty
- Verifying beans are registered via `Arc.container().beanManager()` — beans ARE registered
  under their concrete interface type, but NOT under `NamedStrategy`

### Root cause
Quarkus ARC performs build-time bean discovery and type indexing. When indexing
injection points, it resolves assignable types for each bean but does NOT walk
the full interface hierarchy for `Instance<>` injection. A bean implementing
`AgentRoutingStrategy` is indexed under `AgentRoutingStrategy`, but NOT under
`NamedStrategy` (the super-interface of `AgentRoutingStrategy`).

This is a Quarkus ARC-specific behaviour — standard CDI runtimes (Weld, OWB) do
walk the full type hierarchy for `Instance<>` resolution.

### Fix
Inject `Instance<ConcreteInterface>` per domain type instead of a single
`Instance<SuperInterface>`:

```java
@Inject
public EngineStrategyResolver(
    @Any Instance<AgentRoutingStrategy> agentStrategies,
    @Any Instance<CandidateMatchingStrategy> matchStrategies,
    @Any Instance<CandidateSetStrategy> candidateSetStrategies,
    // ... one per domain type
) {
    List<NamedStrategy> all = new ArrayList<>();
    agentStrategies.forEach(all::add);
    matchStrategies.forEach(all::add);
    candidateSetStrategies.forEach(all::add);
    // index by (type, id)
}
```

### Why this is non-obvious
CDI spec says `Instance<T>` returns all beans with a type assignable to `T`.
Quarkus ARC claims CDI compatibility. The failure is silent — no error, no
warning, just an empty `Instance<>`. A developer familiar with CDI would
expect `Instance<NamedStrategy>` to discover sub-interface implementations.

### Why this fix
The per-type injection approach is Quarkus-idiomatic — it works with ARC's
build-time indexing rather than against it. The alternative (runtime classpath
scanning) defeats Quarkus's build-time model.

### Alternatives considered
- Runtime classpath scanning via `ClassGraph` or similar — works but defeats
  Quarkus build-time model and adds a heavyweight dependency
- `@Discovery` annotation on the marker interface — no such Quarkus feature exists
- `Arc.container().select(NamedStrategy.class)` programmatic lookup — same
  build-time indexing limitation applies
