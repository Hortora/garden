### Symptom

All Quarkus `@QuarkusTest` tests using OkHttp MockWebServer as a `QuarkusTestResourceLifecycleManager` time out with the full `read-timeout` value (e.g. 10 seconds). Every test fails identically — no partial successes, no error messages beyond the timeout. The REST client connects to the correct MockWebServer port (confirmed in error messages), but responses are never received.

### Context

Quarkus 3.x's `quarkus-rest-client` extension uses the Vert.x HTTP client internally (via RESTEasy Reactive). OkHttp MockWebServer uses its own internal HTTP server implementation. The two are incompatible at the connection/response delivery level — TCP connections succeed but response data is never consumed by the Vert.x client.

This affects both MockWebServer 4.x (Java-based, stable) and 5.x (Kotlin-based, alpha). The failure is 100% consistent, not intermittent.

### What was tried (didn't work)

- Downgrading MockWebServer from 5.3.2 to 4.12.0 — same timeout
- Disabling HTTP/2 and ALPN via `quarkus.rest-client."configKey".http2=false` and `alpn=false` — same timeout
- Changing the WebSocket client URL to avoid port collision with Quarkus test server — same timeout
- Running a single test in complete isolation — same timeout

### Root cause

The Vert.x HTTP client (Netty-based, non-blocking I/O) and OkHttp MockWebServer's internal server implementation have an incompatibility at the HTTP response delivery level. The exact protocol-level mismatch is opaque — there is no error, no exception, and no log entry. The request is sent and accepted by MockWebServer, the response is enqueued, but the Vert.x client never reads it.

Confirmed by diagnostic test: the same MockWebServer instance works perfectly with Java's built-in `java.net.http.HttpClient` in the same JVM process.

### Fix

Replace OkHttp MockWebServer with JDK's `com.sun.net.httpserver.HttpServer`. It's a stable JDK API (since JDK 6), requires zero additional dependencies, and is fully compatible with the Vert.x HTTP client.

See companion technique entry GE-20260618-c5d2d8 for the implementation pattern.

Alternative: use WireMock (Jetty-based), which is also known to work with Vert.x clients.

### Why this is non-obvious

MockWebServer is widely recommended for Java HTTP testing and works correctly with most HTTP clients. The failure mode is completely silent — no error message, no protocol negotiation failure, just a timeout. A developer would naturally assume MockWebServer works with any HTTP/1.1 client and spend hours debugging test infrastructure, config properties, or application code before suspecting the mock server itself. The diagnostic path requires testing with a completely different HTTP client to isolate the issue.

### Why this fix

JDK HttpServer over WireMock because it adds zero dependencies, is ~90 lines of code, and provides the same request-recording/response-queuing capability. WireMock is a valid alternative but brings Jetty and a larger dependency tree into test scope.

### Alternatives considered

- **MockWebServer version downgrade (4.x)** — same failure; the incompatibility is not version-specific
- **HTTP/2 and ALPN configuration** — no effect; the issue is not protocol negotiation
- **WireMock** — would work but adds a significant dependency tree (Jetty + WireMock core)
- **`@InjectMock` on the REST client CDI bean** — works for provider-level tests but loses HTTP integration coverage (path routing, header injection, serialization)
