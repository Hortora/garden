### Symptom
After running an automated script to insert two new `null` parameters into ~450 `createChannel()` call sites (16→18 params), ~90 call sites were silently skipped. The build failed with "actual and formal argument lists differ in length" on the skipped files — the script reported success but left them untouched.

### Context
When a Java method signature gains new parameters and tests call the method directly (not through a framework), every call site needs updating. Automated scripts that count commas to determine argument count will miscount when any argument contains commas inside string literals — e.g., `"alice,bob"` is one argument but naive comma counting sees two.

### What was tried (didn't work)
A Python script iterating over `.java` files, finding `createChannel(` calls, counting commas to determine argument count, and inserting `null, null,` after the 12th comma. Files with string arguments like `"alice,bob"` or `" alice , bob "` were counted as having 17 or 18 args instead of 16, so the insertion was skipped.

### Root cause
The comma-counting loop did not track whether it was inside a string literal. Commas within `"..."` were counted as argument separators, inflating the argument count past the threshold and causing the script to skip the file silently (no error, no warning).

### Fix
Track an `in_string` boolean in the comma-counting loop. When inside a string literal, skip commas and handle escaped quotes (`\"`). The corrected approach:

```python
in_str = False
commas = []
ci = 0
while ci < len(args_str):
    c = args_str[ci]
    if in_str:
        if c == '\\':
            ci += 1  # skip escaped char
        elif c == '"':
            in_str = False
    elif c == '"':
        in_str = True
    elif c == '(':
        depth += 1
    elif c == ')':
        depth -= 1
    elif c == ',' and depth == 0:
        commas.append(ci)
    ci += 1
```

### Why this is non-obvious
The first batch of ~360 fixes succeeds (files without commas in string args), giving false confidence that the script works. The remaining ~90 failures only surface at compile time — the script itself reports no errors. Without a post-run compile check, the breakage is invisible until CI.
