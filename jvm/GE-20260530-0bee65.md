### What it is

A Quarkus **extension** can auto-register its Flyway migration classpath path at build time via `NativeImageResourcePatternsBuildItem` and related build steps. A plain **library JAR** (no `quarkus-extension.properties`, no deployment module) has no such hook — it cannot auto-register Flyway locations. Consumers must add the path to `quarkus.flyway.locations` (or `quarkus.flyway.<datasource>.locations`) explicitly in their `application.properties`.

### Symptom

The tables defined by the library JAR's migrations (`V2000__case_ledger_entry.sql`, `V2001__worker_decision_entry.sql`, etc.) do not exist at runtime. Code writing to those tables fails silently or with a `PSQLException: relation does not exist`. All tests pass (if using `database.generation=drop-and-create` in tests), the failure only appears in production or integration environments with Flyway enabled.

### What triggers it

Adding `casehub-engine-ledger` (or any similar plain-JAR library) as a compile dependency when the JAR ships Flyway migrations at `classpath:db/<reponame>/migration`. The dependency resolves correctly and the classes are available. The migration path is on the classpath. But Quarkus Flyway does not scan it unless explicitly configured.

### Fix

Add the migration path to the Flyway locations configuration for the relevant datasource:

```properties
# For a named datasource (e.g. qhorus):
quarkus.flyway.qhorus.locations=classpath:db/qhorus/migration,classpath:db/ledger/migration,classpath:db/engine-ledger/migration

# For the default datasource:
quarkus.flyway.locations=classpath:db/app/migration,classpath:db/shared-lib/migration
```

The library JAR's migration path must be listed explicitly — it is never discovered automatically.

### Why this is non-obvious

Quarkus developers are accustomed to adding an extension dependency and having it "just work" — its Flyway migrations run because the extension registers them at build time. A plain JAR dependency behaves the same for code (CDI beans are discovered if Jandex-indexed) but NOT for Flyway. The distinction is invisible until the missing tables cause failures at runtime.
