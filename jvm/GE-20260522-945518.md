## Symptom

`byNamePrefix("case_123")` returns channels named both `"case_123/work"` (correct) and `"case-123/work"` (wrong). The in-memory path using `String.startsWith()` returns only the correct result. Silent semantic divergence between stores — no error, just wrong data.

## Context

When building a prefix query via JPQL, the `%` is appended to make a LIKE pattern:

```java
jpql.append(" AND name LIKE ?").append(idx++);
params.add(prefix + "%");
```

The prefix value itself is passed as a positional parameter (safe from SQL injection), but SQL still interprets `_` as "match any single character" and `%` as "match any sequence" **within the bound value**. So `"case_123%"` matches any string that starts with `"case"`, followed by any one character, followed by `"123"`.

The in-memory store uses `String.startsWith(prefix)` which is exact — `_` is not special.

## Root cause

SQL `LIKE` treats `_` and `%` as wildcards even inside bound parameters. The JPA spec does not automatically escape metacharacters in LIKE operands. The in-memory path is always exact (`startsWith`), creating silent divergence.

This is a correctness bug even though the parameter is safely bound (no SQL injection risk). The bug only manifests when the prefix contains `_` or `%` — common in namespacing conventions like `case_123/`.

## Fix

Escape metacharacters in the prefix before appending `%`, and declare the escape character in JPQL:

```java
if (q.namePrefix() != null) {
    jpql.append(" AND name LIKE ?").append(idx++).append(" ESCAPE '!'");
    params.add(escapeLikePrefix(q.namePrefix()) + "%");
}

private static String escapeLikePrefix(String prefix) {
    return prefix.replace("!", "!!").replace("%", "!%").replace("_", "!_");
}
```

Using `!` as the escape character avoids backslash complications in JPQL string literals. Works on H2 (`MODE=PostgreSQL`) and PostgreSQL.

The in-memory `matches()` method uses `String.startsWith()` which is already exact — no change needed there.

## Why this is non-obvious

SQL injection defences (parameterised queries) protect against injection but do NOT protect against LIKE wildcard interpretation within the bound value. A developer who correctly parameterises the query assumes the value is treated literally — but SQL LIKE has its own metacharacter layer on top of parameter binding.

The bug is also masked by typical test data: if test channel names use only alphanumeric characters and `/`, no `_` or `%` appear and the divergence is invisible. It only surfaces when naming conventions include `_` (very common in identifiers).
