## H2-reserved words as column names pass PostgreSQL but fail silently in H2 test mode

### Symptom

Flyway migration script runs successfully against PostgreSQL in development and production, but fails in H2 test mode with a syntax error on a column name. The column name looks completely normal.

### What was tried (didn't work)

- Checking PostgreSQL documentation — the column name is valid in PostgreSQL
- Running the migration against PostgreSQL directly — succeeds

### Root cause

H2 reserves more keywords as column names than PostgreSQL does. Words like `key`, `value`, `timestamp`, `name`, `type`, `group`, `order`, `index`, `row` are valid PostgreSQL column names but reserved in H2 — even in `MODE=PostgreSQL`.

### Fix

Avoid these H2-reserved words as column names. Use descriptive alternatives:

```sql
-- Wrong — 'key' is reserved in H2
ALTER TABLE notification_rule ADD COLUMN key VARCHAR(255);

-- Right
ALTER TABLE notification_rule ADD COLUMN rule_key VARCHAR(255);
```

Common H2-reserved words to avoid: `key`, `value`, `timestamp`, `date`, `time`, `year`, `month`, `day`, `hour`, `minute`, `second`, `name`, `type`, `group`, `order`, `index`, `row`, `schema`.

### Why this is non-obvious

PostgreSQL happily accepts these names without complaint. The H2 failure only surfaces in test mode, and the error looks like a generic SQL syntax error — not obviously linked to reserved word status. H2's `MODE=PostgreSQL` gives the impression it matches PostgreSQL keyword rules, but it doesn't fully.
