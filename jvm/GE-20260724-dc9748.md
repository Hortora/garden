## @RunOnVirtualThread breaks Vert.x event bus reply mechanism in Quarkus @ConsumeEvent

**ID:** GE-20260724-dc9748
**Stack:** Quarkus 3.32.x, Vert.x Mutiny, SmallRye Common
**Symptom:** All `@QuarkusTest` integration tests that call `eventBus.requestAndAwait()` time out after 30 seconds with `(TIMEOUT,-1) Timed out after waiting 30000(ms) for a reply`. No error, no exception, no handler failure — the handler runs but the reply is never sent. The timeout is identical across all tests, making it look like a single root cause rather than individual test issues.
**Context:** Migrating a Quarkus application from reactive (`@ConsumeEvent` returning `Uni<Void>`) to blocking (`@ConsumeEvent` + `@RunOnVirtualThread` + `void` return). The caller uses `eventBus.requestAndAwait()` which blocks until a reply arrives.

### What was tried (didn't work)
- Changed handler return type from `void` to `String` (returning `null`) — still times out. Quarkus does not use the return value as a reply when `@RunOnVirtualThread` is present.
- Changed from `@RunOnVirtualThread` to `@ConsumeEvent(blocking = true)` — still times out. The `blocking` attribute offloads to the worker thread pool but the auto-reply mechanism still doesn't fire.
- Both changes combined (`blocking = true` + `String` return) — still times out.

### Root cause
Quarkus's `@ConsumeEvent` integration with `@RunOnVirtualThread` does not wire up the auto-reply mechanism that exists for plain `void` handlers or `Uni<T>` handlers. When a `@ConsumeEvent` method returns `Uni<Void>`, Quarkus subscribes to the Uni and sends the result as a reply when it completes. When a method returns `void` without `@RunOnVirtualThread`, Quarkus sends a null reply immediately. With `@RunOnVirtualThread`, neither mechanism activates — the handler executes correctly on a virtual thread, but no reply is ever sent back to the event bus sender.

### Fix

Replace `eventBus.requestAndAwait()` with direct CDI injection and method calls when the event bus is being used as a single-producer/single-consumer synchronous call mechanism. See companion technique entry GE-20260724-7b07f5.

```java
// Before (reactive) — works
@ConsumeEvent(value = "casehub.case.started")
Uni<Void> onCaseStarted(CaseStartedEvent event) { ... }

// Caller
eventBus.requestAndAwait("casehub.case.started", event);

// After (broken with @RunOnVirtualThread)
@ConsumeEvent(value = "casehub.case.started")
@RunOnVirtualThread
void onCaseStarted(CaseStartedEvent event) { ... }
// requestAndAwait times out — reply never sent

// Fix — remove event bus, inject handler directly
@Inject CaseStartedEventHandler handler;
handler.onCaseStarted(event); // direct call, synchronous
```

### Why this fix

The event bus request-reply pattern was never the right abstraction for single-producer/single-consumer synchronous calls in a blocking architecture. It was used in the reactive codebase because `request().chain()` was the natural way to sequence operations without blocking the event loop. In a blocking world, a direct method call gives the same ordering guarantee trivially. The fix addresses the architectural mismatch rather than working around the framework gap.

### Alternatives considered
- Wrapping handler execution in a `CompletableFuture` and manually sending the reply via `Message.reply()` — too much ceremony for what is fundamentally a direct method call
- Converting all `requestAndAwait` to `send` (fire-and-forget) + `CountDownLatch` — loses the synchronous completion guarantee that the caller needs

### Why this is non-obvious
The symptom (30-second timeout) looks like a handler failure or deadlock, but the handler executes perfectly. Neither `void`, `String`, nor `blocking = true` fixes the reply — every attempted annotation combination fails. The root cause is a framework integration gap between two well-documented features (`@ConsumeEvent` auto-reply and `@RunOnVirtualThread`) that simply don't compose. A developer migrating from reactive to virtual threads would naturally assume the event bus patterns carry over.
