## Symptom

A counter table (`next_seq INT`) resets to its seed value on every call, even though the code only increments it. Tests pass when `MERGE` is called once but break silently on repeated calls.

## Context

Using H2's proprietary `MERGE INTO table KEY(col) VALUES(...)` for a per-subject sequence counter. Expected behaviour: if the row exists, increment `next_seq`. Actual behaviour: if the row exists, the entire row is replaced with the provided VALUES — resetting `next_seq` to the seed value.

## Root Cause

H2's `MERGE INTO table KEY(col) VALUES(...)` is NOT an upsert-with-update. It is a full row replacement on match. When the key column matches, ALL columns are overwritten with the provided VALUES, including the counter column. This differs from SQL-standard MERGE (which uses explicit `WHEN MATCHED THEN UPDATE SET ...` to control which columns change).

## Fix

Use SQL-standard `MERGE INTO ... USING ... WHEN MATCHED THEN UPDATE SET ... WHEN NOT MATCHED THEN INSERT ...`:

```sql
-- H2-native (WRONG for counters — replaces entire row):
MERGE INTO seq_table KEY(subject_id) VALUES (?, 1)

-- SQL-standard (CORRECT — increments on match):
MERGE INTO seq_table AS t
USING (SELECT CAST(?1 AS UUID) AS sid) AS s
ON t.subject_id = s.sid
WHEN MATCHED THEN UPDATE SET next_seq = t.next_seq + 1
WHEN NOT MATCHED THEN INSERT (subject_id, next_seq) VALUES (s.sid, 2)
```

## Why This Is Non-Obvious

The H2 `MERGE INTO KEY` syntax looks like an upsert — "merge" implies combining new data with existing data. A developer writing `MERGE INTO counter_table KEY(id) VALUES (?, 1)` expects the `1` to be the initial value for new rows, not a reset value for existing rows. The test passes on the first call (INSERT path), masking the bug on subsequent calls (REPLACE path). The full-row-replacement semantics are documented in H2's grammar but are easy to miss when the syntax superficially resembles PostgreSQL's `INSERT ON CONFLICT`.
