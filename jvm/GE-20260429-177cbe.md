### Symptom
`NullPointerException` thrown from inside `Map.get()`, not from the caller.
The stack trace points to the JDK immutable map implementation, not to any
null dereference in application code.

### Context
Any use of `Map.of()`, `Map.ofEntries()`, or `Map.copyOf()` (Java 9+) where
a key that might be null is passed to `get()`. Common pattern: a method that
can return null feeds its result directly into a map lookup.

```java
private static final Map<String, String[]> META = Map.ofEntries(
    Map.entry("foo", new String[]{"a", "b"}),
    ...
);

String suffix = eventSuffix(type); // returns null when type is null
String[] meta = META.get(suffix);  // throws NPE — not returns null
```

### Root cause
`HashMap.get(null)` returns null — null is a valid key in HashMap.
`Map.of()` / `Map.ofEntries()` create an immutable map that explicitly
rejects null keys and null values in ALL operations, including `get()`.
The spec says "no null keys or values" but most developers internalise
this only for `put()`, not for `get()`.

### Fix
Guard against null before calling `get()` on any immutable map:

```java
final String suffix = eventSuffix(type);
if (suffix == null) {
    // log and return, or throw with a clear message
    return;
}
final String[] meta = META.get(suffix); // safe
```

### Why this is non-obvious
Developers fluent in `HashMap` expect `get(null)` to return null — the
behaviour is so ingrained that even careful reviewers don't flag null keys
as a risk for read operations. The symptom ("NPE inside JDK code") also
misleads: the first assumption is a null dereference on the returned value,
not that `get()` itself threw. In this session, an external security audit
described the bug as "EVENT_META.get(null) returns null" — the auditor also
had the wrong mental model.
