## Symptom

`@TestTransaction` test fails with:

```
org.h2.jdbc.JdbcSQLIntegrityConstraintViolationException: Unique index or primary
key violation: "PUBLIC.UQ_COMMITMENT_CORR_ID ... ON PUBLIC.COMMITMENT(CORRELATION_ID)"
```

when a method tries to INSERT a second record with the same value in a column
that has a `@UniqueConstraint`. The violation occurs even though the first record
is logically "done" (in a terminal state).

## Context

Designing a delegation chain: entity A (in state DELEGATED) and entity B (OPEN) both
need the same `correlationId` because consumers poll by that business key. A
`@UniqueConstraint` on `correlationId` prevents this — it enforces one record per key
across all states, including terminal ones.

## Root Cause

`@UniqueConstraint` in JPA enforces uniqueness across all rows regardless of other
column values. There is no "partial unique index" equivalent in standard JPA
(though native SQL supports them). Once any record exists with a given `correlationId`,
no second record can be inserted with the same value, even if the first is logically
superseded.

## Fix

Remove the `@UniqueConstraint` and move uniqueness enforcement to the application layer:

```java
// Before — prevents delegation chains
@Table(name = "commitment",
       uniqueConstraints = @UniqueConstraint(name = "uq_commitment_corr_id",
               columnNames = "correlation_id"))

// After — allows multiple records per correlationId
@Table(name = "commitment")
```

Update `findByCorrelationId` to prefer the active (non-terminal) record:

```java
return repo.list("correlationId = ?1 ORDER BY createdAt DESC", correlationId)
    .stream()
    .filter(c -> !c.state.isTerminal())
    .findFirst()
    .or(() -> repo.find("correlationId = ?1", correlationId).firstResultOptional());
```

Add an index (without uniqueness) if lookup performance matters:

```java
@Table(name = "commitment",
       indexes = @Index(name = "idx_commitment_corr_id", columnList = "correlation_id"))
```

## Why Non-Obvious

The unique constraint looks correct at design time: correlationIds should be unique
per "conversation". The flaw only surfaces when the design requires multiple obligation
records per conversation (delegation, reassignment). The violation error points to the
duplicate value but not to the design mismatch — it looks like a bug in the INSERT
logic rather than a constraint model problem.
