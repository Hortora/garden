### Symptom

`@QuarkusTest` fails at Quarkus startup (not at test time) with a service-level uniqueness violation ŌĆö e.g. "Duplicate connector id: 'slack'" ŌĆö even though the test class provides an `@Alternative @Priority(1) @ApplicationScoped` inner class stub that is expected to replace the real `SlackConnector`.

The test stub and the production bean both appear in the CDI graph simultaneously.

### Context

Testing a Quarkus service that collects implementations via `@All List<T>` injection:

```java
@ApplicationScoped
public class ConnectorService {
    ConnectorService(@All final List<Connector> connectors) {
        this.registry = connectors.stream()
                .collect(Collectors.toMap(Connector::id, ...));  // fails on duplicate id
    }
}
```

A `@QuarkusTest` provides a stub to avoid real HTTP calls:

```java
@QuarkusTest
class SlackMcpToolTest {
    @Inject SlackMcpTool tool;

    @Alternative @Priority(1) @ApplicationScoped
    static class StubSlack implements Connector {
        @Override public String id() { return "slack"; }
        @Override public void send(ConnectorMessage msg) { /* no-op */ }
    }
}
```

Startup fails because `ConnectorService` receives both `StubSlack` AND the production `SlackConnector` in the `@All List<Connector>` ŌĆö both with `id() = "slack"`.

### Root cause

In Quarkus ARC, `@Alternative @Priority(N)` selects the alternative for **scalar** injection points (`@Inject Connector`) ŌĆö the alternative wins and the non-alternative is suppressed. However, for **collection** injection points (`@All List<Connector>`), ARC includes ALL beans that satisfy the type ŌĆö both selected alternatives AND non-alternatives. The alternative does not replace the original in the collection; it is added alongside it.

This is consistent with CDI spec semantics but contradicts what developers expect: adding a `@Priority`-annotated alternative in a test should "win" and suppress the production bean everywhere.

### Fix

Bypass CDI for the unit test entirely. Make the service constructor `public` and construct it directly with only the test double:

```java
// In the service (change package-private to public):
public ConnectorService(@All final List<Connector> connectors) { ... }

// In the test (plain Java, no @QuarkusTest needed):
class SlackMcpToolTest {
    private final RecordingConnector connector = new RecordingConnector("slack");
    private final ConnectorService service = new ConnectorService(List.of(connector));
    private final SlackMcpTool tool = new SlackMcpTool(service, new NoOpBridge());

    @Test void test() { ... }
}
```

Benefits:
- No CDI container startup overhead (milliseconds vs seconds)
- No `@Alternative` collision risk
- Tests remain deterministic regardless of what other `Connector` beans are on the classpath
- `@Tool` annotated methods are plain Java methods ŌĆö they can be called directly without the MCP framework

### Why this is non-obvious

Developers associate `@Alternative @Priority(N)` with "this bean replaces the other one" ŌĆö which is true for scalar injection points. The `@All List<T>` exception is documented in the CDI spec but not in any Quarkus testing guide, and the error message ("Duplicate connector id") points to the service validation logic rather than the CDI wiring. The natural fix attempt (adding `@Priority` to the alternative) makes things worse by activating the stub while also keeping the original.

The real fix ŌĆö bypassing CDI entirely for unit tests ŌĆö requires recognising that `@Tool` methods are just annotated Java methods, not framework-managed entrypoints that require container context to invoke.
