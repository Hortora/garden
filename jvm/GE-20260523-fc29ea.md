### Symptom
`@QuarkusTest` in the consuming project (casehub-aml) fails with:
```
SQLGrammarException: Could not prepare statement [Table "MESSAGE_LEDGER_ENTRY" not found]
```
The error points at a Flyway migration or entity-table mismatch. Flyway ran successfully; the entity class is correct. The consuming project's own code hasn't changed.

### Context
Two Maven projects with a producer/consumer relationship:
- **Producer** (casehub-qhorus): tests run with `hibernate-orm.database.generation=drop-and-create`. Flyway disabled for the qhorus datasource in tests. Hibernate generates the schema directly from entity class annotations.
- **Consumer** (casehub-aml): tests run with `hibernate-orm.database.generation=none`. Flyway IS enabled. Schema must match what Flyway migrations create.

A recent commit in the producer renamed the join table in the entity annotation (`@Table(name = "message_ledger_entry")`) but the corresponding Flyway migration SQL was also updated — in the source tree. The key issue: the locally-installed SNAPSHOT jar (`~/.m2`) still contains the **old migration SQL** (`V2000__*.sql` that creates `agent_message_ledger_entry`).

### Root cause
Producer tests use `drop-and-create`: Hibernate generates the schema from entity annotations — the new `message_ledger_entry` table is created automatically. Producer tests pass. The Flyway SQL inside the installed SNAPSHOT jar is stale, but it doesn't matter because producer tests don't run Flyway.

Consumer tests use `generation=none`: they depend entirely on Flyway to create the schema. Flyway runs the stale SQL from the installed jar, creating `agent_message_ledger_entry`. The entity annotation says `message_ledger_entry`. At runtime, Hibernate tries to query a table that doesn't exist.

`mvn compile` in the consumer succeeds because it uses cached class files from the producer (incremental compilation). No compile error surfaces. The failure only appears at `@QuarkusTest` startup.

### What was tried (didn't work)
1. Clean compile of the consumer project → still failed (cached class in consumer, stale jar from producer)
2. Checking Flyway migration files in the consumer → no issues found (consumer had no custom V2000)
3. Checking the entity annotation in the producer source → annotation was correct in source, matched the schema expectation

### Fix
Reinstall the producer SNAPSHOT from its own pom.xml before running consumer tests:

```bash
mvn --batch-mode install -DskipTests -f /path/to/producer/pom.xml
```

This recompiles the producer and installs a fresh jar to `~/.m2`, including the updated Flyway SQL. Consumer tests then pick up the correct migration.

### Why this is non-obvious
The failure looks like a Flyway configuration bug or a schema mismatch in the consumer project. The producer tests pass (they use `drop-and-create`, bypassing Flyway entirely). There is no compile error. The version number of the SNAPSHOT jar doesn't change. Nothing in the consumer's own code changed. The only diagnostic clue is the table name discrepancy between the error (`MESSAGE_LEDGER_ENTRY`) and the entity annotation (`message_ledger_entry`), but even that points to a consumer misconfiguration, not a stale jar from a different repo.

The fundamental issue is that the two database generation strategies (producer: `drop-and-create`, consumer: `generation=none + Flyway`) create an inconsistency that is invisible to Maven's incremental build and only surfaces at runtime in the consumer.
