## Symptom

Adding `LOCK_MODE=1` to the H2 JDBC URL to prevent concurrent first-insert races on a
shared table appears to succeed — no error, the application starts — but the concurrent
race condition continues unaffected. Two threads racing to INSERT the same primary key
still produce `JdbcSQLIntegrityConstraintViolationException`.

```
JDBC URL:
jdbc:h2:mem:testdb;MODE=PostgreSQL;DB_CLOSE_DELAY=-1;LOCK_MODE=1
```

```
Error (still observed with LOCK_MODE=1):
Unique index or primary key violation: "PUBLIC.PK_LEDGER_SUBJECT_SEQUENCE
INDEX ... VALUES ( /* key:1 */ UUID '...')"; SQL statement:
MERGE INTO ledger_subject_sequence AS t USING (...) AS s ...
WHEN NOT MATCHED THEN INSERT (subject_id, tenancy_id, next_seq) VALUES (...)
```

## Context

Triggered by concurrent `MERGE INTO` on a new row that both threads see as absent (MVCC
read-committed isolation). Common in `@QuarkusTest` suites where multiple Quartz workers
or CDI async observers write to the same H2 table simultaneously for the first time.

LOCK_MODE is a documented H2 parameter. The documentation does not clearly state that it
only applies to the discontinued page-store engine and is silently ignored in MVStore mode.

## What was tried

- `LOCK_MODE=1` in JDBC URL — no effect; race persists.
- `LOCK_MODE=2` — same; still silently ignored.
- Neither URL parameter nor `SET LOCK_MODE` SQL triggers an error; the setting is accepted
  but MVStore's MVCC engine does not honour it.

## Root cause

H2 2.0 switched the default storage engine from page-store to MVStore. MVStore uses MVCC
(multi-version concurrency control) with row-level locking, not table-level locking. The
`LOCK_MODE` URL parameter only controlled the page-store engine's locking behaviour.
In H2 2.x, MVStore always uses MVCC regardless of `LOCK_MODE`. The parameter is accepted
without error for backward compatibility but has no observable effect.

## Fix

Replace `MERGE INTO` with `INSERT ... ON CONFLICT DO NOTHING` + `UPDATE`. H2 2.2+
(released July 2023) added `ON CONFLICT DO NOTHING` support for `INSERT` statements,
including in `MODE=PostgreSQL`. The `ON CONFLICT` path is race-safe: if two transactions
simultaneously attempt the first insert, exactly one succeeds and the other silently does
nothing; both then proceed to the `UPDATE`, which serialises via the MVStore row lock.

```sql
-- Instead of:
MERGE INTO t USING (...) AS s ON t.key = s.key
  WHEN MATCHED THEN UPDATE SET t.val = t.val + 1
  WHEN NOT MATCHED THEN INSERT (key, val) VALUES (s.key, 2)

-- Use:
INSERT INTO t (key, val) VALUES (CAST(? AS UUID), 1) ON CONFLICT (key) DO NOTHING;
UPDATE t SET val = val + 1 WHERE key = CAST(? AS UUID);
```

This is the same SQL as the PostgreSQL path — both H2 2.2+ and PostgreSQL support it, so
a single implementation works for both databases.

## Why this is non-obvious

`LOCK_MODE=1` is documented as enabling "table-level locking", which is exactly what is
needed to serialise concurrent MERGE statements. The parameter is silently accepted with no
error or warning when it has no effect. There is no runtime indication that the setting is
being ignored. The fix requires knowing that (a) H2 2.0 removed page-store, (b) MVStore is
MVStore is immune to LOCK_MODE, and (c) H2 2.2+ added ON CONFLICT support — none of these
are prominently documented together.

### Why this fix

`ON CONFLICT DO NOTHING` + `UPDATE` produces identical SQL for both H2 and PostgreSQL,
eliminating the dialect split that caused the original bug. The `MERGE INTO` approach
assumed H2 would serialize via table locking — an assumption that was silently wrong for
five major releases.
