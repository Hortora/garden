### Symptom

Running `mvn test -pl child-module` fails at `@QuarkusTest` startup with:

```
java.lang.IllegalArgumentException: /Users/.../.m2/repository/io/casehub/casehub-ledger-deployment/0.2-SNAPSHOT/casehub-ledger-deployment-0.2-SNAPSHOT.jar does not exist
    at io.quarkus.paths.PathTree.ofDirectoryOrArchive(PathTree.java:53)
    ...
Caused by: java.nio.file.NoSuchFileException: ...casehub-ledger-deployment-0.2-SNAPSHOT.jar
```

The POM for the deployment artifact exists in `~/.m2` (Maven resolved its metadata) but the JAR itself is absent.

### Context

Occurs when all of these are true:
1. A child module contains `@QuarkusTest` integration tests
2. The child module depends (directly or transitively) on a Quarkus extension from another repository
3. That external extension was not built with `mvn install` — only `mvn compile` or `mvn test-compile` was run
4. The child module is tested in isolation (`-pl child-module` without `--also-make`)

### Root cause

Quarkus `@QuarkusTest` triggers the Quarkus augmentation pipeline at test startup. Augmentation requires the **deployment JAR** of every Quarkus extension on the classpath. Deployment JARs:
- Are generated by the `quarkus:build` Maven plugin goal
- Are only installed to `~/.m2` during `mvn install` (not `compile`, `test-compile`, or `test`)
- Are named `<artifactId>-deployment-<version>.jar` (separate artifact from the runtime JAR)

When the child module is tested in isolation with `-pl`, Maven resolves extension runtime JARs from `~/.m2` (if installed) but also needs their corresponding deployment JARs. If those JARs were never installed (only compiled), they're absent and augmentation fails at the filesystem level — not as a dependency resolution error.

The error message references the absolute path and says the file "does not exist" — it looks like a deployment problem, not a "you forgot to `mvn install`" problem.

### Fix

**Option 1 — Install the dependency chain first (recommended for cross-repo setups):**

```python
import subprocess, os

env = dict(os.environ)
env['JAVA_HOME'] = subprocess.check_output(['/usr/libexec/java_home', '-v', '26']).decode().strip()

# Install the extension chain (and all its reactor deps)
subprocess.run(['mvn', 'install', '-DskipTests', '-pl', 'runtime,deployment', '--also-make',
    '-f', '/path/to/extension/pom.xml'], env=env, check=True)

# Now run the child module's tests
subprocess.run(['mvn', 'test', '-pl', 'child-module',
    '-f', '/path/to/consumer/pom.xml'], env=env, check=True)
```

**Option 2 — Use `--also-make` when running tests (builds dependencies from source):**

```bash
mvn test -pl child-module --also-make -Dsurefire.failIfNoSpecifiedTests=false
```

`--also-make` includes all reactor dependencies in the build, which makes Quarkus augmentation use the locally-compiled deployment artifacts rather than looking in `~/.m2`. The `-Dsurefire.failIfNoSpecifiedTests=false` flag is needed because `--also-make` causes the test filter to apply to all built modules, not just the target.

**Option 3 — Build and install the ledger extension locally:**

If the extension source is available locally, install just the deployment module:
```bash
mvn install -DskipTests -pl deployment --also-make -f /path/to/extension/pom.xml
```

### Why this is non-obvious

Three layers of confusion:
1. **"Does not exist" vs "not installed"** — the error reads like a deployment failure (`file does not exist`). The actual cause is that `mvn install` was never run for the extension. Nothing in the error message connects the symptom to the missing `install` step.
2. **Deployment JARs are an invisible second artifact** — developers working with Quarkus extensions as dependencies think in terms of the runtime JAR. The deployment JAR exists as a separate Maven artifact (`-deployment` classifier) but is never mentioned in consumer POM dependencies and doesn't appear in `mvn dependency:tree`.
3. **`mvn compile` builds locally but doesn't install** — running `mvn compile -pl extension` puts the class files in `target/` but does NOT install the deployment JAR to `~/.m2`. The child module's isolated test can't use `target/` artifacts from a different project.
