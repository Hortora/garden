### Symptom
Time-windowed queries like `SELECT ... WHERE timestamp < ?` return zero rows
when the upper bound is `Instant.MAX`, even though matching rows exist.
No error, no warning — just empty results.

### Context
Any SQLite schema that stores `java.time.Instant` as ISO-8601 text
(the standard approach since SQLite has no native timestamp type) and
uses `<` / `>` / `BETWEEN` comparisons for time-window filtering.

### Root cause
`Instant.MAX.truncatedTo(ChronoUnit.MILLIS).toString()` produces:
```
+1000000000-12-31T23:59:59.999Z
```
The `+` prefix (ASCII 43) sorts lexicographically **before** any
digit-prefixed timestamp (ASCII 48 for `0`). SQLite's text comparison
is pure lexicographic — it has no concept of ISO-8601 semantics.

So `"2026-07-05T10:00:00.000Z" < "+1000000000-12-31T23:59:59.999Z"`
evaluates to **false** in SQLite, because `2` (ASCII 50) > `+` (ASCII 43).

The same issue applies to `Instant.MIN` / `Instant.EPOCH` if used as
lower bounds — their serialized forms may also produce unexpected
sort order.

### What was tried (didn't work)
Passing `Instant.MAX` directly as a SQL parameter with `.toString()`.
The query silently returned empty results with no indication of why.

### Fix
Conditionally omit the WHERE clause boundary when the value is a
sentinel (`Instant.EPOCH`, `Instant.MIN`, `Instant.MAX`):

```java
private boolean hasSinceFilter(Instant since) {
    return since != null
        && !since.equals(Instant.EPOCH)
        && !since.equals(Instant.MIN);
}

private boolean hasUntilFilter(Instant until) {
    return until != null
        && !until.equals(Instant.MAX);
}

// Build SQL dynamically:
StringBuilder sql = new StringBuilder("SELECT ... FROM t WHERE tenant_id = ?");
if (hasSinceFilter(since)) sql.append(" AND timestamp >= ?");
if (hasUntilFilter(until)) sql.append(" AND timestamp < ?");
```

### Why this is non-obvious
The natural expectation is that `Instant.MAX` means "no upper bound" — pass
it as the parameter and everything matches. This works with databases that
have native timestamp types. SQLite's text comparison silently inverts the
semantics with no error.

### Why this fix
Omitting the clause entirely (rather than substituting a "large enough" date
string) is the correct fix because it avoids coupling to any specific far-future
date format and makes the intent explicit in the generated SQL.

### Alternatives considered
- Use a large but normal date like `9999-12-31T23:59:59Z` as the sentinel — works but is a magic string that could collide with legitimate far-future timestamps
- Convert timestamps to Unix epoch integers — works but loses ISO-8601 readability and human-debuggability in the SQLite database
