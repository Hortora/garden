### Symptom

A code reviewer recommends removing an explicit `synchronized` block around a compound add-then-evict operation on a `Collections.synchronizedSet()`, claiming the wrapper already provides mutual exclusion. The reviewer's reasoning appears correct — the wrapper synchronizes every method call.

### Context

Occurs when implementing a bounded set (e.g. a self-notification filter, LRU cache, or bounded dedup set) using `Collections.synchronizedSet(new LinkedHashSet<>())`. The pattern: add an element, check size, evict the oldest if over capacity.

### Root cause

`Collections.synchronizedSet()` synchronizes each individual method call (`add()`, `size()`, `iterator()`, `contains()`) — but compound operations spanning multiple calls are NOT atomic. Between `add()` and the subsequent `size()` check, another thread can `add()` its own element, causing both threads to see size > max and both to evict, or neither to evict if the timing differs.

The Javadoc states: *"It is imperative that the user manually synchronize on the returned set when iterating over it."* This applies to any compound operation that must be atomic, not just iteration.

### Fix

Wrap compound operations in an explicit `synchronized` block on the synchronized collection itself:

```java
Set<Long> recentIds = Collections.synchronizedSet(new LinkedHashSet<>());

void recordSent(Long messageId) {
    synchronized (recentIds) {           // required — compound operation
        recentIds.add(messageId);
        if (recentIds.size() > maxSize) {
            Iterator<Long> it = recentIds.iterator();
            it.next();
            it.remove();
        }
    }
}

boolean wasSentLocally(Long messageId) {
    return recentIds.contains(messageId); // safe — single atomic operation
}
```

Single-method calls like `contains()` do NOT need the explicit block — the wrapper's per-method synchronization is sufficient.

### Why this is non-obvious

The `Collections.synchronizedSet()` wrapper creates a false sense of complete thread safety. A developer sees "synchronized" in the name and reasonably assumes all operations are safe. The Javadoc warning about manual synchronization is framed around iteration specifically, making it easy to miss that ALL compound operations (not just loops) require it. The distinction between "each method is synchronized" and "sequences of methods are synchronized" is subtle and catches experienced developers.

### Alternatives considered

- `ConcurrentHashMap.newKeySet()` — provides better concurrency but no insertion-order eviction (no `LinkedHashSet` equivalent in `java.util.concurrent`)
- `ConcurrentLinkedDeque` + `ConcurrentHashMap` combo — correct but over-engineered for low-contention workloads
- `synchronized(this)` on the enclosing class — works but coarser than necessary; synchronizing on the collection itself is the documented pattern
