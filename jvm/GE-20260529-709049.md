### Context
A `sealed interface` is used as the return type of a method. All variants are known at design time and are inner records of the interface. The interface needs exhaustive pattern matching via `switch`.

### What works but isn't obvious
When all permitted subtypes are declared as nested records inside the sealed interface, the `permits` clause is **not required**. The compiler infers the full set of permitted subtypes from the nested declarations:

```java
public sealed interface WebhookResult {
    // No "permits" clause needed — nesting implies permission
    record Delivered(List<InboundMessage> messages)            implements WebhookResult {}
    record Challenged(String responseBody, String contentType) implements WebhookResult {}
    record Ignored()                                           implements WebhookResult {}
    record Unauthorized()                                      implements WebhookResult {}
}
```

The compiler enforces exhaustiveness in switch expressions over the sealed type:

```java
return switch (connector.handle(request)) {
    case WebhookResult.Delivered(var msgs) -> { msgs.forEach(service::receive); yield Response.ok().build(); }
    case WebhookResult.Challenged(var body, var ct) -> Response.ok(body).type(ct).build();
    case WebhookResult.Ignored()      -> Response.ok().build();
    case WebhookResult.Unauthorized() -> /* ... */;
    // No default needed — compiler confirms all variants are covered
};
```

Adding a new record variant to the sealed interface immediately causes a compile error at the switch expression until the new case is handled.

### Why this is non-obvious
The Java 17 sealed classes JEP (409) describes `permits` as the mechanism for listing allowed subtypes, which leads developers to write it explicitly even when not needed. The implicit permission from nesting is mentioned in the spec but not in most tutorials or examples. Most sealed type examples online use separate top-level classes with an explicit `permits` list.
