### What it is
When `ChannelGateway.receiveHumanMessage()` cannot pass a correlationId (qhorus#154), use per-entity channel names to encode the entity identity in the channel URL. The response listener extracts the entity ID from the channel name, eliminating the need for correlationId in the message itself.

### Pattern
Name the channel after the entity rather than the actor or the workflow:

```
# Per-actor channel (fragile — how do you know which entity the response is for?)
clinical/site/{siteId}/pi-oversight

# Per-entity channel (correct — channel name IS the entity identifier)
clinical/deviation/{deviationId}/pi-oversight
```

Extract the entity ID in the response listener using a compiled regex:

```java
private static final Pattern CHANNEL_PATTERN =
    Pattern.compile("clinical/deviation/([0-9a-f-]+)/pi-oversight");

public void process(String channelName, MessageType type, String piId) {
    Matcher m = CHANNEL_PATTERN.matcher(channelName);
    if (!m.matches()) return;
    UUID deviationId = UUID.fromString(m.group(1));
    // look up entity by ID...
}
```

### Why this is non-obvious
The natural instinct for governance channels is per-actor (one inbox per PI, one per IRB committee). Per-actor channels require correlationId in the response to identify which entity the response belongs to — which is exactly what the human inbound path cannot carry. Per-entity channels solve the problem structurally: the entity is in the URL, not the message.
