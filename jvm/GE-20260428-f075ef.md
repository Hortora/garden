## What this is

A pattern for testing event-driven Quarkus components without polling. A `ConcurrentHashMap<Key, CompletableFuture<V>>` shared between an `@ApplicationScoped` observer bean and the test, using `computeIfAbsent` on both sides, guarantees correctness regardless of whether the event fires before or after the test registers interest.

## When to use

When a test needs to wait for a specific CDI async event to fire and verify the state the event carries, without using awaitility polling (which is sensitive to CI runner load).

## The pattern

```java
// Observer bean — @ApplicationScoped, persists across tests in same Quarkus session
@ApplicationScoped
public static class ItemCompletionObserver {

    private final ConcurrentHashMap<String, CompletableFuture<String>> futures =
        new ConcurrentHashMap<>();

    void onItemCompleted(@ObservesAsync ItemCompletedEvent event) {
        // computeIfAbsent: safe if test hasn't registered yet — creates and immediately completes
        futures.computeIfAbsent(key(event.caseId(), event.itemName()), k -> new CompletableFuture<>())
               .complete(event.itemId());   // ← carry the stable ID as the future value
    }

    public CompletableFuture<String> awaitItem(UUID caseId, String itemName) {
        // computeIfAbsent: safe if event already fired — returns the already-completed future
        return futures.computeIfAbsent(key(caseId, itemName), k -> new CompletableFuture<>());
    }

    private static String key(UUID caseId, String itemName) {
        return caseId + ":" + itemName;
    }
}

// Test
CompletableFuture<String> future = observer.awaitItem(caseId, "worker-b");
String itemId = future.get(30, TimeUnit.SECONDS);  // 30s is safety net, not a race
assertThat(registry.getItem(itemId).getStatus()).isEqualTo(Status.COMPLETED);
```

## Why this works regardless of ordering

- **Event fires first:** `computeIfAbsent` in the observer creates and immediately completes the future. When the test calls `awaitItem`, `computeIfAbsent` finds the already-completed future and returns it. `get()` returns immediately.
- **Test registers first:** `computeIfAbsent` in the test creates an incomplete future. When the event fires, `computeIfAbsent` in the observer finds the same future and completes it. `get()` unblocks.

The key insight: use `computeIfAbsent` on **both sides**, not just one. A `get`-then-`putIfAbsent` pattern has a window between the check and the put.

## Why the future value should be the stable ID

If a mutable index can be overwritten (e.g. by a re-triggered component), a post-event lookup may return the wrong entry. Carrying the exact ID in the event and in the future value bypasses the index entirely for the assertion.

## Why this is non-obvious

Most developers reach for awaitility with a timeout, or register a latch before the action and wait on it. Both have problems: awaitility is polling (load-sensitive on CI), and a pre-registered latch requires knowing the case/item ID before starting the case. This pattern solves both without either tradeoff.
