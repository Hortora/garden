### Symptom

Behavior implemented in a `@PrePersist` `@EntityListeners` callback (e.g. ENFORCE mode,
audit enrichment, hash computation) is silently skipped when using an in-memory repository
instead of the JPA implementation.

### Context

Projects that ship both a JPA repository and an in-memory alternative (`@Alternative @Priority(1)`)
for test isolation. The in-memory repo calls business logic directly via a service or pipeline —
it never goes through Hibernate.

### Root cause

JPA entity lifecycle callbacks (`@PrePersist`, `@PostPersist`, etc.) are triggered by the
JPA provider (Hibernate). They only fire when an entity is persisted through the `EntityManager`.
In-memory repositories that store entities in a `ConcurrentHashMap` or `List` bypass the
`EntityManager` entirely — `@EntityListeners` annotations are decoration only; no JPA
framework is present to invoke them.

### Fix

Two options:
1. **Call enrichment explicitly in the in-memory repo:** Before storing the entity, call the
   same enricher pipeline the JPA path relies on (e.g. `enricherPipeline.enrich(entry)`).
   This is what `InMemoryLedgerEntryRepository` in casehub-ledger already does.

2. **Document as JPA-only:** If the behavior (e.g. ENFORCE mode) is intentionally unavailable
   in in-memory mode, document it clearly. Tests that need ENFORCE must use JPA with H2.

### Why this is non-obvious

`@EntityListeners` reads as "whenever this entity is saved, run these callbacks" — and for
JPA that's true. The implicit assumption is that saving always goes through JPA. In codebases
with dual JPA/in-memory backends, this assumption silently breaks for half the test suite.
