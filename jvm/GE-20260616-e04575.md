## Detect live JDBC dialect via DatabaseMetaData instead of @ConfigProperty(db-kind)

**ID:** GE-20260616-e04575
**Stack:** Quarkus 3.x, Hibernate ORM, JDBC

### Context

A Quarkus extension supports both H2 (dev/test) and PostgreSQL (production). The extension
must choose between two incompatible SQL branches at runtime. The intuitive approach reads
the `quarkus.datasource.db-kind` config property. This works for the default datasource but
silently breaks when the consumer configures a named datasource (e.g. `casehub.ledger.datasource=myds`):
the named datasource's db-kind lives at `quarkus.datasource."myds".db-kind`, not
`quarkus.datasource.db-kind`, so the property always returns the default `"h2"`.

### Technique

Read the database product name directly from the live JDBC connection via `DatabaseMetaData`.
Cache the result since the database product never changes at runtime:

```java
private volatile Boolean postgresql = null;

private boolean isPostgresql() {
    Boolean result = postgresql;
    if (result == null) {
        result = em.unwrap(Session.class)
                .doReturningWork(conn -> conn.getMetaData()
                        .getDatabaseProductName()
                        .toLowerCase(Locale.ROOT)
                        .contains("postgresql"));
        postgresql = result;
    }
    return result;
}
```

This works correctly for both default and named datasources — the `EntityManager` is always
bound to the correct connection regardless of naming.

**Precondition:** must be called inside a `@Transactional` method so the `EntityManager`
has an active connection.

**Thread safety:** `volatile Boolean` single-check idiom is safe for immutable results —
two threads racing on first call both compute the same value and either write wins.

### Alternatives considered

- **`@ConfigProperty(name = "quarkus.datasource.db-kind")`** — works for default datasource only; silently returns `"h2"` default for named datasources, causing wrong SQL dialect to be selected in production.
- **`em.getEntityManagerFactory().unwrap(SessionFactory.class).getJdbcServices().getJdbcEnvironment().getDialect()`** — exposes Hibernate internal APIs that are less stable and require additional unwrapping; also no clear advantage over JDBC metadata.
- **Restructuring to JPQL** — not always possible when dialect-specific SQL is required for correctness (e.g. H2 has no `ON CONFLICT` syntax at all).

### Why this is non-obvious

The `@ConfigProperty` approach is documented, idiomatic, and works in all normal cases.
Named datasource support in Quarkus uses a dynamic config key pattern that isn't
immediately visible — the developer sees the injection succeed (the property has a default)
and never realises the wrong value is being used. The failure only appears in production
when a PostgreSQL SQL statement is executed against an H2 connection, or vice versa.

The `em.unwrap(Session.class).doReturningWork()` API is a Hibernate-internal extension
not mentioned in Quarkus documentation, so most developers would never look there.
