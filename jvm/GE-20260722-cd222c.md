### The convention

When a NoOp/stub implementation returns analysis metadata (counts, distributions, ratios, consensus classifications), the metadata must honestly reflect the scope of work actually performed — not the scope of data available.

### The alternative

Report the full input size as the analysis scope. E.g., if 5 plans are available but the NoOp only examines 1, report `totalPlans=5, occurrenceCount=1` — making each step look like it has 20% support.

### Why this convention

The dishonest alternative produces semantically misleading output. A consumer seeing `{occurrenceCount: 1, totalPlans: 5, agreement: UNIQUE}` interprets it as "thorough analysis found weak consensus." The honest version `{occurrenceCount: 1, totalPlans: 1, agreement: UNANIMOUS}` correctly conveys "limited analysis found full consensus within examined scope."

This matters because downstream consumers (routing strategies, audit systems, clinical safety reports) make decisions based on consensus ratios. A NoOp that inflates denominators silently degrades decision quality — and the degradation is invisible because the output looks structurally valid.

### Concrete example

`NoOpPlanEnsembleAnalyzer` receives 5 adapted plans, examines only the best-scoring one:

```java
// WRONG — claims to have analyzed all 5 plans
return new EnsemblePlan(best, analysis, List.of(caseId), score, adaptedPlans.size());
// totalPlans=5, inputPlanCount=5, each step UNIQUE

// RIGHT — honestly reports examining 1 plan
return new EnsemblePlan(best, analysis, List.of(caseId), score, 1);
// totalPlans=1, inputPlanCount=1, each step UNANIMOUS
```
