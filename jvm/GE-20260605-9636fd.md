## Symptom

Code that intends to throw inside a `try` block, when a prior statement succeeds, silently does nothing. The exception is created and immediately caught by the enclosing `catch` clause before it can propagate. No compiler warning, no IDE diagnostic.

## Context

Occurred when adding UUID-shape detection to `ChannelSlugValidator.validateSlugPath()`. The goal: parse the name as a UUID — if parsing succeeds, the name is UUID-shaped and must be rejected.

```java
// BROKEN — the thrown IAE is caught by its own catch
try {
    UUID.fromString(name);
    throw new IllegalArgumentException("Channel name must not be UUID-shaped: '" + name + "'");
} catch (IllegalArgumentException ignored) {
    // "not a UUID — good" — but also catches the rejection above
}
```

`UUID.fromString(name)` succeeds for a UUID-shaped name. The next line throws `IllegalArgumentException`. But the `catch (IllegalArgumentException ignored)` block catches that thrown exception immediately — the name passes validation as if nothing happened.

## Root cause

A `throw` statement inside a `try` block is subject to the same catch clauses as any other exception that propagates through that block. There is nothing special about a `throw` statement — it raises an exception, which is then evaluated against every active `catch`. If the `catch` matches, it is caught and swallowed.

Java's specification is clear and correct. The confusion is that the developer's intent ("catch parse failure") and the catch clause's scope ("catch any IAE from anywhere in this try block") diverge silently.

## Fix

Use a boolean flag — separate the detection from the rejection:

```java
// CORRECT — flag pattern separates detection from rejection
boolean isUuid;
try {
    UUID.fromString(name);
    isUuid = true;
} catch (IllegalArgumentException ignored) {
    isUuid = false;
}
if (isUuid) {
    throw new IllegalArgumentException("Channel name must not be UUID-shaped: '" + name + "'");
}
```

Alternatively, extract the detection into a helper that returns `null` on failure:

```java
// Also correct — extraction keeps try-catch minimal
if (tryParseUuid(name) != null) {
    throw new IllegalArgumentException("Channel name must not be UUID-shaped: '" + name + "'");
}

private static UUID tryParseUuid(String s) {
    try { return UUID.fromString(s); }
    catch (IllegalArgumentException ignored) { return null; }
}
```

## Why this is non-obvious

The natural reading of the code is: "try to parse, if parse fails, handle the failure." The developer's mental model treats the `throw` as somehow exempt from catch semantics, or mentally scopes the catch to parse failures only. Java does not share that mental model.

No compiler warning fires. Static analysers typically don't flag this either — a catch-all that catches a just-thrown exception is valid Java. The bug only manifests if you specifically test the rejection path; tests that only verify non-rejected inputs pass green while the rejection silently fails.
