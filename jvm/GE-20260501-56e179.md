## Symptom

A `ThreadLocal` value is set on thread A before calling `CompletableFuture.supplyAsync(lambda)`.
Inside the lambda, `ThreadLocal.get()` returns `null` even though `set()` was called moments earlier.

## Context

Any use of `CompletableFuture.supplyAsync()` (or `thenApplyAsync`, `thenRunAsync`, etc.) with the
default `ForkJoinPool` executor. The lambda runs on a pool thread that has no inherited
relationship to the calling thread.

## What was tried (didn't work)

Setting the ThreadLocal on the Quartz worker thread before calling `supplyAsync`. The worker
function ran on a ForkJoinPool thread and saw `null`.

## Root cause

`ThreadLocal` values are scoped to the thread that called `set()`. `CompletableFuture.supplyAsync()`
schedules the lambda on the ForkJoinPool — a different thread — which has no access to the
calling thread's ThreadLocal storage. Java does not propagate ThreadLocals across async
boundaries by default.

## Fix

Set the ThreadLocal as the **first operation inside the lambda**, captured from a local variable
in the enclosing scope. Clear it in a `finally` block:

```java
WorkerContext ctx = buildContext(...);   // on calling thread

CompletableFuture.supplyAsync(() -> {
    WorkerExecutionContext.set(ctx);     // set on the pool thread
    try {
        return function.apply(inputData);
    } finally {
        WorkerExecutionContext.clear();  // always clear
    }
});
```

## Why this is non-obvious

Setting the ThreadLocal before `supplyAsync` *looks* correct — the value is set before the
lambda is created. The bug is invisible at the call site; the lambda silently reads `null`
with no error or warning.
