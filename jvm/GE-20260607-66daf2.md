### What
When an action gate is approved, re-publish `WorkflowExecutionCompleted` with `plannedAction=null` on the `WORKER_EXECUTION_FINISHED` event bus address instead of duplicating the completion logic in the approval handler.

### Context
casehub-engine's `WorkflowExecutionCompletedHandler` is the central completion handler: it applies output to case context, resumes orchestration, notifies listeners, marks PlanItems COMPLETED (via blackboard), fires StageAutocompleteEvaluator, and publishes CONTEXT_CHANGED. When an action gate is approved, all of this still needs to happen — but with the deferred output from the gate.

### The technique
```java
// ActionGateApprovedHandler — after clearing pendingActionGate:
eventBus.publish(
    EventBusAddresses.WORKER_EXECUTION_FINISHED,
    WorkflowExecutionCompleted.approved(instance, worker, gate.idempotency(), gate.deferredOutput()));
```

The `approved()` factory produces an event with `plannedAction=null`. `WorkflowExecutionCompletedHandler` checks `if (event.plannedAction() != null)` before the gate fork — null means the normal path runs, re-classification is skipped, and the deferred output is applied exactly as if the worker had just returned.

### Why this is non-obvious
The obvious approach is to duplicate the output-application logic, workerStatus notification, and PlanItem completion steps in the approval handler. That creates maintenance burden and drift risk. The re-fire approach is safe because the `approved()` factory guarantees `plannedAction=null` — it cannot re-enter the gate path regardless of what classifiers are registered. The name `approved()` documents the intent at the call site.

### Constraints
Requires that the gate stores the full `PendingActionGate` including `worker` (or workerId for lookup) and `idempotency`. The approval handler must retrieve the live `Worker` object from `CaseDefinitionRegistry` at resolution time — do NOT store the `Worker` in the gate record (lambdas are not Jackson-serializable).
