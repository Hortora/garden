## The technique

When writing a fluent Java DSL companion for a YAML case definition, copy JQ strings **verbatim** from the YAML source — no translation, no paraphrase. Then in the equivalence test, compare evaluators with a direct `isEqualTo()` assertion:

```java
// DSL companion — verbatim JQ string
var irbDecided = Goal.builder()
    .name("irb-decided")
    .condition(".irbConsultation != null")  // copied exactly from YAML goal.condition
    .build();

// Equivalence test — direct comparison, no cast or .toString()
assertThat(dslGoal.getCondition()).isEqualTo(yamlGoal.getCondition());

// Works for trigger filters too
assertThat(((ContextChangeTrigger) dslBinding.getOn()).getFilter())
    .isEqualTo(((ContextChangeTrigger) yamlBinding.getOn()).getFilter());

// And for humanTask mappings
assertThat(dslHT.inputMapping()).isEqualTo(yamlHT.inputMapping());
assertThat(dslHT.outputMapping()).isEqualTo(yamlHT.outputMapping());
```

## Why it works

`JQExpressionEvaluator` is a Java **record** with a single field (`expression: String`). Records implement `equals()` and `hashCode()` based on their fields. When the YAML mapper calls `CaseDefinitionYamlMapper.load()`, it wraps JQ condition and filter strings in `new JQExpressionEvaluator(string)`. When the DSL companion calls `Goal.builder().condition(String)` or `new ContextChangeTrigger(String)`, both paths also wrap in `JQExpressionEvaluator`. The two record instances compare equal if and only if their `expression` strings are identical — no cast, no `.expression()` accessor call, no `.toString()` needed.

`ListEvaluator.StaticList` (used for `candidateGroups`) is also a record — `candidateGroups()` comparison works the same way via `Set<String>` value equality.

## What developers would normally do instead

The instinct is to use `LambdaExpressionEvaluator` in DSL companions for test-only definitions — it's type-safe, avoids JQ parsing overhead, and co-locates conditions with test assertions (the approach taken by `devtown/PrReviewCaseDefinition`). But lambda predicates have no value equality, making structural comparison impossible: you cannot assert that a `LambdaExpressionEvaluator` wrapping `ctx -> ctx.get("x") != null` matches the JQ string `".x != null"`. The lambda companion produces a behaviorally equivalent definition, not a structurally identical one.

Using JQ strings sacrifices the ergonomic advantage of lambdas but makes the equivalence test the primary artifact: the test proves that both authoring paths produce the same `JQExpressionEvaluator` records, not just the same runtime behaviour.

## When to use which

| Companion purpose | Use |
|---|---|
| Prove structural parity with YAML (protocol compliance) | JQ strings verbatim — enables record equality comparison |
| Provide a test fixture without YAML parsing | LambdaExpressionEvaluator — type-safe, no JQ, readable assertions |
| Both | Two companion classes or a parameterised build() factory |
