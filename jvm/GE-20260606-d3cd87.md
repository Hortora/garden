### Symptom
Quarkus native binary starts successfully (logs "started in 0.018s"), but `QuarkusApplication.run()` is never called. No error, no diagnostic. The binary enters idle mode — no HTTP server, no output, no exit. Looks like a JNI loading hang.

### Context
A `@QuarkusMain(name = "native-gate")` command-mode app. Works in JVM mode. In native image, the binary starts Quarkus CDI but never invokes the application's `run()` method.

### What was tried (didn't work)
- Debugging JNI library loading (wrong direction — JNI wasn't the issue)
- Adding `--initialize-at-run-time` entries (correct but irrelevant to the actual problem)
- Running with extended timeout (2+ hours — confirmed it was a hang, not slow loading)

### Root cause
Quarkus's native image entry point does not discover `@QuarkusMain` annotated classes when the `name` attribute is set to a non-default value. The `name` parameter is intended to prevent conflicts when downstream consumers have their own `@QuarkusMain`, but it prevents Quarkus from selecting the class as the entry point in native image.

### Fix
Remove the `name` attribute — use `@QuarkusMain` without parameters:

```java
@QuarkusMain  // not @QuarkusMain(name = "native-gate")
public class MyCommand implements QuarkusApplication {
    @Override
    public int run(String... args) throws Exception {
        // This will now be called in native image
    }
}
```

### Why this is non-obvious
The binary starts and Quarkus logs successful startup. There is zero indication that the `@QuarkusMain` class was not found — no warning, no error, no "no main class found" message. The symptom (silent hang after startup) looks identical to a JNI library loading hang, which is the expected failure mode when working on native image JNI validation.

### Why this fix
The `name` parameter exists to disambiguate multiple `@QuarkusMain` classes on the classpath. For a single-main module, it adds no value and actively breaks native image discovery. The downstream-conflict concern is handled differently — C5 will move the main class to an integration-tests module where classpath isolation prevents conflicts.
