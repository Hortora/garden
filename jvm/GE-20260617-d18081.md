### Symptom
Java compile error when a class implements both `ChatModel` and `StreamingChatModel`:
the compiler reports ambiguous default methods.

### Context
When building a LangChain4j adapter that needs to satisfy both `ChatModel` (blocking)
and `StreamingChatModel` (streaming) injection points in a single CDI bean.

### What was tried
Implementing both interfaces and overriding only the extension-point methods —
`doChat(ChatRequest) → ChatResponse` and `doChat(ChatRequest, StreamingChatResponseHandler)`.
These compile fine in isolation.

### Root cause
Both `ChatModel` and `StreamingChatModel` declare four identical default method signatures:
- `listeners()` → `List<ChatModelListener>`
- `defaultRequestParameters()` → `ChatRequestParameters`
- `provider()` → `ModelProvider`
- `supportedCapabilities()` → `Set<Capability>`

Java's diamond rule requires a class implementing both interfaces to explicitly override
every method where both declare a default. All four must be overridden — not just the
extension points.

### Fix
Override all four in the implementing class:

```java
@Override public ModelProvider provider() { return ModelProvider.ANTHROPIC; }

@Override public Set<Capability> supportedCapabilities() { return Set.of(); }

@Override public ChatRequestParameters defaultRequestParameters() {
    return DefaultChatRequestParameters.EMPTY;
}

@Override public List<ChatModelListener> listeners() {
    return List.of();  // or return injected CDI listeners
}
```

### Why this is non-obvious
`ChatModel` and `StreamingChatModel` are designed as complementary interfaces —
implementing both in one class is a natural choice when building a single adapter for
both injection points. The shared default methods aren't mentioned in LangChain4j
documentation. The compiler error only appears when implementing both simultaneously;
either interface alone compiles cleanly with only the `doChat()` extension point overridden.
