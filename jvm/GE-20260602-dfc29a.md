## Symptom

`conditionFactory.failMessage("custom timeout message")` produces a compile error — method not found on `ConditionFactory`.

## Context

`failMessage()` appears widely in Awaitility documentation, blog posts, and code examples. It is present in some API versions but absent from 4.3.0. Tests written against examples from newer docs fail to compile when the project's Awaitility jar is 4.3.0.

## Root Cause

`failMessage()` was not present in `ConditionFactory` in Awaitility 4.3.0. The `ConditionFactory` class provides no hook for a custom timeout message at this version. Confirmed by decompiling the 4.3.0 jar — `ConditionFactory.class` has no `failMessage` method in its constant pool.

## Fix

Embed the custom message inside `assertThat().as(...)` within the `untilAsserted()` lambda:

```java
// Instead of:
await().atMost(5, TimeUnit.SECONDS)
       .failMessage("message not delivered within 5s")        // compile error on 4.3.0
       .untilAsserted(() -> assertThat(captured).isNotEmpty());

// Use:
await().atMost(5, TimeUnit.SECONDS)
       .untilAsserted(() -> assertThat(captured)
           .as("message not delivered within 5s")
           .isNotEmpty());
```

The `.as(...)` message appears in Awaitility's `ConditionTimeoutException` output because Awaitility wraps the `AssertionError` thrown by AssertJ and includes its message.

## Why This Is Non-Obvious

`failMessage()` is the natural place to put a timeout message — it reads as "if this condition times out, show this message." That API exists in newer Awaitility versions and is shown in official documentation. The absence in 4.3.0 is not obvious until the compile error occurs, and the workaround — embedding the message inside the assertion — is not the natural discovery path.
