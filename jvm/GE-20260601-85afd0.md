## What this is

After generating `ARC42STORIES.MD` from a CaseHub application's `LAYER-LOG.md`, run three systematic checks before considering the document complete. Without prior experience, none of these checks are obvious — each surfaces a different class of documentation error that looks correct without the check.

In the devtown migration, these three checks caught 10+ documentation errors across §5, §9.4, §10, §11, §12, and §13.

## The three checks

### Check 1 — Issue status (§12 Active Risks and Technical Debt)

For every issue reference in §12:
```bash
gh issue view N --repo casehubio/REPO 2>&1 | head -3
```

**What it catches:** Closed issues showing as active risks. LAYER-LOG entries are written at layer close time; issues may be resolved months later before the migration runs.

**What to do:** Remove CLOSED issues from Active Risks; update or note CLOSED items in Technical Debt.

### Check 2 — Class name existence (§9.4 Key files)

For every class name listed in Key files sections:
```bash
find /path/to/project -name "ClassName.java" 2>/dev/null
```

**What it catches:** Classes that exist in design documents or LAYER-LOG entries but were never built, were renamed, or belong to a foundation module (not the application). The concept is correct; the class name is wrong.

**What to do:** If no output — the class doesn't exist in the application. Find the real class name, or note "provided by [foundation-module]".

### Check 3 — CDI annotation accuracy (§9.4 Key wiring, §4 CDI displacement)

For each service implementation mentioned in layer entries:
```bash
head -15 path/to/ClassName.java
```

**What it catches:** CDI annotations that evolved after the LAYER-LOG was written. The most common case: when a second competing implementation is added to the same port, both need `@Alternative @Priority(N)` — `@ApplicationScoped` alone is no longer sufficient. Layer entries and §4 code snippets written before this evolution show the old annotation.

**What to do:** Update Key files descriptions and code snippets in §4 to reflect actual annotations.

## Why this is non-obvious

Each check addresses a specific kind of invisible drift:
- **Check 1:** GitHub is the only authoritative source for issue state; LAYER-LOG looks authoritative but is a snapshot
- **Check 2:** Class name plausibility masks non-existence — a concept-correct name that follows naming conventions will not trigger doubt without an explicit search
- **Check 3:** CDI annotation evolution is a multi-layer phenomenon — Layer 3 may silently require `@Alternative @Priority` on a Layer 5 implementation; the Layer 5 LAYER-LOG entry written before Layer 3 existed would never mention this

## Constraints

Applies to: CaseHub harness applications migrating LAYER-LOG.md to ARC42STORIES.MD. Also applicable to any architecture documentation migration from a snapshot source to a living document.

Check 1 requires the GitHub CLI (`gh`). Check 2 requires access to the project source. Check 3 requires reading actual Java files.

## Why this technique over just re-reading the document

Re-reading the document catches typos and inconsistencies in prose. It does not reveal closed issues (requires GitHub), non-existent classes (requires the filesystem), or stale CDI annotations (requires the source). These three checks require querying external sources — they cannot be performed by reading the document alone.
