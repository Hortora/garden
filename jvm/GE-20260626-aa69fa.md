### Symptom
Batch operation iterates items, calling a `@Transactional(REQUIRED)` service method
for each. The service throws a RuntimeException for an expected condition ("already
processed"). The caller catches it and continues to the next item. The second
iteration fails with a rollback error — the entire batch is lost.

### Context
- Caller: `@Transactional` method iterating a list
- Callee: `@Transactional` (REQUIRED — joins caller's TX) throws RuntimeException
- The exception represents an expected outcome, not an error

### What was tried (didn't work)
Catch the RuntimeException in the caller loop and skip the item. The catch block
executes, but Narayana has already poisoned the transaction.

### Root cause
When a `@Transactional(REQUIRED)` method throws a RuntimeException, the Narayana
`TransactionalInterceptorBase` calls `transaction.setRollbackOnly()` **before**
re-throwing the exception to the caller. By the time the caller's catch block runs,
the outer transaction is irreversibly marked for rollback. Any subsequent JPA write
or transaction commit fails.

This is JTA spec behaviour (not Narayana-specific): runtime exceptions from a
REQUIRED method signal a system-level failure, and the container marks the
transaction accordingly — regardless of whether the caller handles the exception.

### Fix
Use return types for expected outcomes. Exceptions should only signal genuine errors.

```java
// Before (broken under JTA composition):
@Transactional
public void withdraw(UUID id, String tenantId) {
    if (already withdrawn) throw new AlreadyWithdrawnException(id); // ← poisons outer TX
}

// After (JTA-safe):
public enum WithdrawalResult { WITHDRAWN, ALREADY_WITHDRAWN }

@Transactional
public WithdrawalResult withdraw(UUID id, String tenantId) {
    if (already withdrawn) return WithdrawalResult.ALREADY_WITHDRAWN; // ← no exception, TX healthy
}
```

REST callers check the return value and map to HTTP status (e.g. 409 for ALREADY_WITHDRAWN).
Batch callers skip ALREADY_WITHDRAWN items — no exception, no rollback, no TX poisoning.

### Why this is non-obvious
The catch block executes normally — no indication that the transaction is doomed. The
rollback happens silently at commit time or on the next JPA operation. Developers
familiar with plain Java exception handling expect catch-and-continue to work. The
JTA interceptor's pre-catch `setRollbackOnly()` is invisible to the caller.

### Why this fix
Using `@ApplicationException(rollback = false)` on the exception class is the JTA
workaround, but it treats the symptom. The root issue is using exceptions for
expected outcomes — return types are the correct model for "already done" semantics,
and they compose naturally with batch operations, streams, and reactive pipelines.

### Alternatives considered
- `@ApplicationException(rollback = false)` — prevents the rollback marking but keeps exception-as-flow-control. Fragile: adding a new exception class without the annotation reintroduces the bug silently.
- `dontRollbackOn` on `@Transactional` — same workaround, spread across callers instead of the exception class.
- Pre-check before calling (TOCTOU) — race window between check and call; still fails under concurrent access.
