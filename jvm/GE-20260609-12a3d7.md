### Symptom
`NoSuchMethodError` or `NoClassDefFoundError` at startup when ocraft 0.4.21 runs inside
Quarkus 3.x, which bundles Vert.x 4.x. Each error surfaces one broken call:

1. `MessageProducer.setWriteQueueMaxSize(int)` ŌĆö removed in Vert.x 4.x
2. `MessageProducer.writeQueueFull()` ŌĆö removed in Vert.x 4.x
3. `JsonObject.put(String, String/Integer/Long)` typed overloads ŌĆö removed; only `put(String, Object)` remains
4. `HttpClientOptions.setPipelining(boolean)` ŌĆö removed in Vert.x 4.x
5. `HttpClientOptions.setMaxWebsocketFrameSize(int)` ŌåÆ renamed `setMaxWebSocketFrameSize(int)`
6. `HttpClient.websocketStream(int, String, String)` ŌåÆ replaced by `rxWebSocket(int, String, String)`
7. `WebSocket.writeBinaryMessage(Buffer)` on the **core** (`io.vertx.core.http.WebSocket`) ŌĆö now requires a `Handler<AsyncResult<Void>>`. The **rx** binding (`io.vertx.reactivex.core.http.WebSocket`) still has the no-handler overload.

### Context
ocraft 0.4.21 was built against Vert.x 3.x in 2018ŌĆō2019. Quarkus 3.x bundles Vert.x 4.x.
The errors surface one at a time as each broken class is first loaded ŌĆö six rounds of
diagnosis before the connection works.

### Root Cause
Vert.x 4.0 made breaking API changes: removed `WriteStream` from `MessageProducer`,
collapsed typed `JsonObject.put()` overloads to `put(String, Object)`, renamed WebSocket
methods (capitalisation), and replaced stream-based WebSocket API with single/rx API.
ocraft was never updated.

### Fix
Bytecode-patch the five affected classes in the local Maven repo. Write patched sources,
compile against Vert.x 4.x (so the compiler resolves to the correct overloads), then
replace the `.class` files in the ocraft JAR:

```python
# patch_ocraft.py
import zipfile, shutil, subprocess, os
# 1. Write patched .java sources (remove deleted calls, rename methods, use rx variant)
# 2. mvn compile to get .class files
# 3. Replace in ~/.m2/repository/com/github/ocraft/ocraft-s2client-api/0.4.21/ocraft-s2client-api-0.4.21.jar
```

Classes to patch: `VertxChannel`, `VertxChannelProvider`, `S2ClientVerticle`, `Ping`, `OnRequest`.

For `Ping` and `OnRequest`: replace `webSocket.getDelegate().writeBinaryMessage(Buffer.buffer(data))`
with `webSocket.writeBinaryMessage(Buffer.buffer(data))` ŌĆö use the rx variant, not the core delegate.

For `S2ClientVerticle`: replace `retry(N)` (immediate retries) with
`retryWhen(errors -> errors.take(N).flatMap(e -> Observable.timer(3, TimeUnit.SECONDS)))`
to give SC2 time to release the previous connection between attempts (see related gotcha).

### Why This Is Non-Obvious
- Each error surfaces one class at a time ŌĆö looks like 7 separate problems, not one root cause
- `WebSocket.writeBinaryMessage(Buffer)` exists in BOTH the core and rx bindings in Vert.x 4.x,
  but only the rx variant retains the no-handler overload ŌĆö easy to confuse
- Patching the JAR in `.m2` is fragile; `mvn dependency:purge-local-repository` silently
  destroys the patches. Document in `NATIVE.md` and keep the patch script.
