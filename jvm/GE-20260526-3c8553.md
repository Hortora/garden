### Symptom

`assertThatThrownBy(() -> bean.methodWithSessionAnnotation("arg", null)).isInstanceOf(NullPointerException.class)`
fails with "Expecting code to raise a throwable" — even though `Objects.requireNonNull` is the
first line of the method body.

### Context

A CDI bean method annotated with `@WithSession` (or `@WithTransaction`) has a synchronous
precondition guard at the top:

```java
@WithSession
public Uni<Optional<AgentDescriptor>> findById(String agentId, String tenancyId) {
    Objects.requireNonNull(tenancyId, "tenancyId");  // throws NPE if null
    return repo.find("... WHERE tenancyId = ?1", tenancyId).firstResult().map(...);
}
```

Calling `bean.findById("id", null)` does not throw — it returns a `Uni` that, when
subscribed, emits the NPE as a failure.

### Root cause

`@WithSession` is a CDI interceptor that wraps the annotated method in a reactive
session-establishment pipeline. The interceptor's `around-invoke` implementation catches
all exceptions from the method body and re-emits them as `Uni.createFrom().failure(e)`.
This is intentional — the interceptor must handle the reactive context in a way that
unifies all failure paths into the Uni channel. Synchronous throws are treated as
failures, not as unhandled exceptions propagated to the caller.

### Fix

Test reactive failures using `@RunOnVertxContext` + `UniAsserter.assertFailedWith`:

```java
@Test
@RunOnVertxContext
void findById_with_null_tenancyId_throws(UniAsserter asserter) {
    asserter.assertFailedWith(
        () -> registry.findById("any", null),
        NullPointerException.class
    );
}
```

Do **not** use `assertThatThrownBy(() -> bean.method(...))` for methods annotated with
`@WithSession` or `@WithTransaction` — the synchronous throw is unreachable from outside
the CDI proxy.

### Why this is non-obvious

Java developers expect `Objects.requireNonNull` to always throw synchronously — it does
in regular methods and in methods behind most CDI interceptors. The behaviour of
`@WithSession` (and `@WithTransaction`) is distinctive: the interceptor establishes a
reactive session context and *must* bridge the synchronous entry point into the Uni channel.
That bridging catches all exceptions. Nothing in the Quarkus docs or `@WithSession`
Javadoc mentions this effect on precondition guards. The symptom ("no throwable raised")
is the opposite of what the code appears to do.
