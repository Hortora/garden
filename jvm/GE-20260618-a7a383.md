## Symptom

```java
AiMessage msg = mock(AiMessage.class);
when(msg.text()).thenReturn("{...json...}");  // → UnfinishedStubbingException
```

`mock(AiMessage.class)` appears to succeed, but `when(msg.text())...` throws `UnfinishedStubbingException`. Mockito correctly detects that the stubbing started but could not complete — the exception message references the `stubResponse()` call site, not the `when()` line, which further obscures the cause.

## Root Cause

`AiMessage` has final methods in LangChain4J 1.14.1. Mockito's standard `mock()` cannot intercept final methods (requires the Mockito `mockito-extensions/org.mockito.plugins.MockMaker` with `mock-maker-inline` configured). The mock is created but the `when()` call cannot set up the stub because the method call in the `when()` argument executes the real final method, which doesn't register with Mockito as expected.

## Fix

Use the real constructors and builders directly — they are simple and public:

```java
// AiMessage — string constructor
AiMessage msg = new AiMessage("{...json...}");

// ChatResponse — builder
ChatResponse resp = ChatResponse.builder()
    .aiMessage(new AiMessage("{...json...}"))
    .build();
```

For use in test stubs:

```java
private static ChatResponse stubResponse(final String json) {
    return ChatResponse.builder()
            .aiMessage(new AiMessage(json))
            .build();
}
```

`ChatModel` itself IS an interface and CAN be Mockito-mocked. Only the value types it returns (`AiMessage`, `ChatResponse`) are non-mockable.

## Why this is non-obvious

The `UnfinishedStubbingException` message cites the `mock(AiMessage.class)` call site as the problem — not the `when()` call. This leads developers to inspect the mock creation rather than the stub setup. The root cause (final method) is only discoverable by checking the LangChain4J source code.
