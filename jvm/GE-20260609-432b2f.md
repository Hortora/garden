### Symptom
`NullPointerException: Cannot invoke "String.replaceFirst(String, String)" because "baseBuild" is null`
inside `ExecutableParser.loadSettings()`. In Quarkus dev mode, this surfaces as a non-`Exception`
(`System.exit(0)` intercepted as an Error subclass), bypassing `catch (Exception e)` and
MicroProfile `@Retry`, making the failure completely silent from the caller's perspective.

### Context
Using `S2Coordinator.setup().loadSettings(new String[]{})` (the standard ocraft startup sequence)
on macOS. ocraft reads `ExecuteInfo.txt` to find the SC2 binary, then calls
`getBaseBuildFromGameExePath()` to extract the build number from the path for version lookup.

### Root Cause
```java
// ExecutableParser.java in ocraft-s2client-api 0.4.21
private static String getBaseBuildFromGameExePath(String gameExePath) {
    String[] dirTree = gameExePath.split("\\\\");  // Windows backslash — produces ONE segment on Mac
    for (String dir : dirTree) {
        if (dir.startsWith("Base")) return dir;   // never matches
    }
    return null;  // always returns null on Mac
}
```
Mac paths use `/`. Splitting `/Applications/StarCraft II/Versions/Base96883/SC2.app/...` on `\\`
produces a single-element array. No element starts with "Base", so `baseBuild = null`.
Subsequently, `baseBuild.replaceFirst("Base", "")` NPEs.

### Fix
Set the system property **before any ocraft class is initialized** (before `S2Coordinator.setup()`
is first called, so Typesafe Config picks it up in `OcraftApiConfig.config = ConfigFactory.load()`):

```java
// In your connect() method, before S2Coordinator.setup()
String baseBuild = extractBaseBuildFromPath(exePath);  // parse it yourself using "/"
System.setProperty("ocraft.game.executable.baseBuild", baseBuild);

coordinator = S2Coordinator.setup()
    .setProcessPath(exePath)
    ...
    .launchStarcraft();
```

Parse the base build yourself using path segments (not backslash):
```java
for (int i = 0; i < exePath.getNameCount(); i++) {
    String segment = exePath.getName(i).toString();
    if (segment.startsWith("Base")) { baseBuild = segment; break; }
}
```

The system property is read by Typesafe Config and surfaces as `customBaseBuild`
in `ExecutableParser.loadSettings()`, skipping the broken path-extraction entirely.

### Why This Is Non-Obvious
- Method name `getBaseBuildFromGameExePath` gives no hint of platform-specific behaviour
- Works correctly on Windows (where `\\` is the separator)
- The failure doesn't throw — it calls `System.exit(0)`, which in Quarkus dev mode becomes
  a non-`Exception` Throwable, bypassing `catch (Exception e)` and `@Retry` silently
- `OcraftApiConfig.config` is a static field initialized once at class load time; the fix
  only works if the system property is set before ocraft's classes are first loaded
