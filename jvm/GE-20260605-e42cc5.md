## @Blocking on plain @ApplicationScoped bean method causes Quarkus test classloader failure

### Symptom

All `@QuarkusTest` classes fail with a classloader-level JUnit error:

```
TestEngine with ID 'junit-jupiter' encountered a critical issue during test discovery:
(1) [ERROR] ClassSelector [className = '...SomeTest', classLoader = null] resolution failed
    Cause: PreconditionViolationException: Could not load class with name: ...SomeTest
```

The error message points at completely unrelated test classes (`SomeTest`, not the bean that has the annotation). No CDI deployment error appears. The test JVM fails before CDI initialisation runs.

### Context

Adding `io.smallrye.common.annotation.@Blocking` to a plain `@ApplicationScoped` CDI bean method — one that is called directly from `@Observes StartupEvent` or from other Java code, rather than from a reactive runtime.

```java
// ❌ WRONG: @Blocking on an ordinary bean method called from Java code
@ApplicationScoped
public class ServerStartup {

    @Blocking  // ← causes Quarkus augmentation failure
    void bootstrapCasehubWatchers() {
        // calls .await().atMost(...) internally
    }

    void onStart(@Observes StartupEvent event) {
        bootstrapCasehubWatchers();  // direct Java call — not reactive context
    }
}
```

### Root cause

`@Blocking` is a Quarkus reactive-interception annotation. Quarkus augmentation wires it via interceptors that only exist in reactive contexts (Vert.x event loop, JAX-RS, reactive messaging, `@ConsumeEvent`, MCP tools). When applied to a plain CDI bean method called from ordinary Java code, augmentation registers interceptor bindings that have no matching interceptor — this causes a hard deployment failure. The failure surfaces as a JUnit classloader error rather than a CDI `DeploymentException` because the Quarkus classloader setup fails before the CDI container boots.

### Fix

Remove `@Blocking` from any method that is:
- Called directly from `@Observes StartupEvent` (runs on the main startup thread — already blocking by nature)
- Called directly from other Java code in a blocking context
- Not inside a reactive pipeline (Vert.x event loop, JAX-RS endpoint, reactive messaging handler, `@ConsumeEvent` handler, MCP tool method)

```java
// ✅ CORRECT: no annotation needed — onStart() already runs on the main blocking thread
@ApplicationScoped
public class ServerStartup {

    void bootstrapCasehubWatchers() {
        // calling .await().atMost(...) is safe — we're on the main startup thread
    }

    void onStart(@Observes StartupEvent event) {
        bootstrapCasehubWatchers();  // runs on main startup thread; blocking is fine
    }
}
```

`@Blocking` is valid on:
- JAX-RS resource methods (`@GET`, `@POST`, etc.)
- Reactive messaging handler methods (`@Incoming`)
- Vert.x event bus consumer methods (`@ConsumeEvent`)
- MCP tool methods (`@Tool`)

### Why this is non-obvious

The symptom (`classLoader = null` on a completely unrelated test class) points nowhere near the cause. There is no "CDI deployment failed" message — Quarkus fails silently at a lower level. A developer looking at the failing test will spend time checking the test class's CDI wiring, its imports, or whether the classpath is correct, never finding the problem. The annotation was added precisely to indicate that blocking code would run — the intent is correct, only the placement is wrong. `@Blocking` on a reactive handler is idiomatic and widely documented; `@Blocking` on a startup observer is silently wrong.
