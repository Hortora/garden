### What a skilled developer would do instead

Inject the context via a CDI bean (`@RequestScoped` or `@ApplicationScoped`), set it before dispatching the job, and read it in the listener. This fails silently in Quartz listeners because they run on Quartz worker threads where CDI request scope is not active — `@RequestScoped` beans throw `ContextNotActiveException`, and `@ApplicationScoped` beans with thread-local state require manual setup that's easy to miss.

### Solution 1 — Quartz job data map (for Quartz job listeners)

**Approach:** Store context in the `JobDetail` data map at scheduling time; read it back in the listener on any thread.

**Pros:** Works across Quartz execution threads, retry cycles, and JDBC job stores. No CDI involvement.

**Cons/trade-offs:** Quartz-specific. Requires the context value to be known at scheduling time.

```java
// When scheduling the job (has access to tenancyId here)
JobDetail job = newJob(MyWorkerJob.class)
    .withIdentity(jobKey)
    .usingJobData("caseId", caseId.toString())
    .usingJobData("workerId", workerId)
    .usingJobData("tenancyId", tenancyId)   // ← thread through job data
    .build();

// In the JobListener (no CDI request scope here)
@Override
public void jobToBeExecuted(JobExecutionContext context) {
    String tenancyId = context.getMergedJobDataMap().getString("tenancyId");
    // use tenancyId directly — no CDI injection needed
    eventLogRepository.append(eventLog, tenancyId);
}

@Override
public void jobWasExecuted(JobExecutionContext context, JobExecutionException ex) {
    String tenancyId = context.getMergedJobDataMap().getString("tenancyId");
    // same data available at completion
}
```

Use `getMergedJobDataMap()` rather than `getJobDetail().getJobDataMap()` — the merged map combines job-level and trigger-level data, matching Quartz's resolution semantics.

### Solution 2 — Explicit method parameter (for @Scheduled or service methods)

**Approach:** Add `String tenancyId` as an explicit parameter to operations called from both request-context and no-request-context threads. HTTP callers pass the resolved `effectiveTenancyId`; scheduler callers pass `entity.tenancyId` (already stored on the entity at registration time).

**Pros:** Framework-agnostic. Works for `@Scheduled` methods, reactive pipelines, async observers, or any code that runs without a CDI request context. The entity already carries the tenancyId from when it was created in a proper request context.

**Cons/trade-offs:** Requires changing the method signature and all call sites. Callers must have tenancyId available — works when it's stored on a persistent entity (e.g. `Watchdog.tenancyId`), but not if the context is truly ephemeral.

```java
// Interface — explicit tenancyId param avoids CurrentPrincipal injection
public interface ChannelStore {
    void updateLastActivity(UUID channelId, String tenancyId);
}

// JPA implementation — uses the param, not CurrentPrincipal
@Override
@Transactional
public void updateLastActivity(UUID channelId, String tenancyId) {
    Channel.update("lastActivityAt = ?1 WHERE id = ?2 AND tenancyId = ?3",
            Instant.now(), channelId, tenancyId);
}

// HTTP request caller (has request context)
String effectiveTenancyId = dispatch.tenancyId() != null
        ? dispatch.tenancyId()
        : currentPrincipal.tenancyId();
channelStore.updateLastActivity(channel.id, effectiveTenancyId);

// Scheduler caller (no request context — reads from entity)
channelStore.updateLastActivity(channel.id, channel.tenancyId);
```

The key insight: the tenancyId was bound to the entity at registration time (when a request context *was* active). Reading it back from the entity is safe from any thread.

### Why this works (both solutions)

The common principle: capture context *at the time it's available* (request time / entity registration) and carry it explicitly through to where it's needed (listener / scheduler). CDI request scope is not the only mechanism — it's just the most convenient one when the request context is active.

### Context

Surfaced when adding `tenancyId` to a Quarkus/Quartz engine where `QuartzWorkerExecutionJobListener` needed to write tenant-scoped event log entries (Solution 1), and when `ChannelStore.updateLastActivity()` needed to be called from both `MessageService.dispatch()` (HTTP, request context active) and `WatchdogEvaluationService` (Quarkus `@Scheduled`, no request context) on the same method signature (Solution 2).
