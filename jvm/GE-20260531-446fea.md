### What a skilled developer would do instead

Inject the context via a CDI bean (`@RequestScoped` or `@ApplicationScoped`), set it before dispatching the job, and read it in the listener. This fails silently in Quartz listeners because they run on Quartz worker threads where CDI request scope is not active — `@RequestScoped` beans throw `ContextNotActiveException`, and `@ApplicationScoped` beans with thread-local state require manual setup that's easy to miss.

### The technique

Store execution context in the Quartz job data map using `usingJobData()` at job creation time. Read it back via `context.getMergedJobDataMap()` in both `jobToBeExecuted` and `jobWasExecuted` listener callbacks. The data map is attached to the `JobDetail` and survives through the full Quartz job lifecycle including retry cycles.

```java
// When scheduling the job (has access to tenancyId here)
JobDetail job = newJob(MyWorkerJob.class)
    .withIdentity(jobKey)
    .usingJobData("caseId", caseId.toString())
    .usingJobData("workerId", workerId)
    .usingJobData("tenancyId", tenancyId)   // ← thread through job data
    .build();

// In the JobListener (no CDI request scope here)
@Override
public void jobToBeExecuted(JobExecutionContext context) {
    String tenancyId = context.getMergedJobDataMap().getString("tenancyId");
    // use tenancyId directly — no CDI injection needed
    eventLogRepository.append(eventLog, tenancyId);
}

@Override
public void jobWasExecuted(JobExecutionContext context, JobExecutionException ex) {
    String tenancyId = context.getMergedJobDataMap().getString("tenancyId");
    // same data available at completion
}
```

Use `getMergedJobDataMap()` rather than `getJobDetail().getJobDataMap()` — the merged map combines job-level and trigger-level data, matching Quartz's resolution semantics.

### Why this works

Quartz job data is serialised into the scheduler's job store alongside the `JobDetail`. It is available on any thread that Quartz uses to execute or notify about the job, regardless of CDI context availability. For in-memory (RAM) Quartz stores this is a direct object reference; for JDBC stores it's deserialised. Either way, the data is present in the listener without any CDI plumbing.

### Context

Surfaced when adding `tenancyId` to a Quarkus/Quartz engine where `QuartzWorkerExecutionJobListener` needed to write tenant-scoped event log entries. The listener fires on Quartz's internal thread pool — no CDI `@RequestScoped` context, no way to inject `CurrentPrincipal`. Storing `tenancyId` in the job data map at scheduling time (where the `CaseInstance` and its `tenancyId` are available) resolved the gap cleanly.
