## Symptom
`NullPointerException` when calling `.put()`, `.get()`, or any method on a `Map`
or `Set` class field — not at object construction time.

## Context
When porting Java code between codebases, a field is copied as a declaration
without carrying over the `= new HashMap<>()` initializer. The declaration looks
syntactically complete — access modifier, type, name, all present:

    private Map<Integer, TerminalNode> associatedTerminals;

All the methods that use the field are also present, so nothing looks missing
at a glance.

## Root cause
Java initializes reference fields to `null` by default. The missing initializer
is a small visual absence on an otherwise complete-looking line.

## Fix
Add the initializer to the declaration:

    private Map<Integer, TerminalNode> associatedTerminals = new HashMap<>();

Or initialize in the constructor if the type isn't known at declaration time.

## Why this is non-obvious
The bug is dormant — it only fires when a method on the field is first called,
not at construction. In a large class with many fields, the missing `= new HashMap<>()`
is easy to overlook during code review. The NPE stack trace points to the call
site (e.g. `addAssociatedTerminal()`), not to the field declaration, making the
root cause non-obvious.

The porting context makes this especially common: vol1's `BaseNode` initialized
this field via a different mechanism; copying the declaration to vol2 without
the initializer left a latent NPE that only surfaced when the finalization
phase was implemented and first called `addAssociatedTerminal()`.
