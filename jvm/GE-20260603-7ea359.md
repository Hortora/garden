### What this is

`ChannelProjection<S>` is the Qhorus SPI for incremental state folding over
committed channel events. Implementations need no CDI annotation (`@ApplicationScoped`,
`@Singleton`, etc.). Qhorus discovers them at startup via the Java SPI mechanism —
not via CDI injection.

### Why this matters

In a `api/` + `runtime/` Maven split, a `ChannelProjection<S>` implementation
can live in the pure-Java `api/` module — no Quarkus or CDI dependency required.
A developer following standard Quarkus extension patterns would annotate it
`@ApplicationScoped` and place it in the `runtime/` module.

### Evidence

`SummaryProjector` in `casehubio/drafthouse` is unannotated:

```java
public class SummaryProjector implements ChannelProjection<ReviewState> {
    // no @ApplicationScoped, no @Singleton, no CDI annotation of any kind
}
```

Located at `server/api/src/main/java/io/casehub/drafthouse/debate/SummaryProjector.java`
— the pure-Java module with no Quarkus compile-scope dependencies. Qhorus discovers
and uses it at startup without any CDI involvement.

### Why this is non-obvious

Every other Qhorus integration point (`MessageService`, `CommitmentService`,
`ChannelGateway`) is CDI-injectable. The natural assumption is that any Qhorus-related
class must be a CDI bean. `ChannelProjection` is the exception — it is a pure Java SPI,
enabling placement in a zero-dependency module and making the domain portable to
non-Quarkus environments (e.g. Claudony plugin JARs).
