### What
Serialize a small struct into the `content` field of a Qhorus COMMAND message using `java.util.Properties`. Parse it back during message processing. Survives JVM restarts — the data lives in the Qhorus database, not in-memory.

### Context
When processing an oversight gate, `OversightGateService.openGate()` needs to store the original commitment context (correlationId, channelId, commandMessageId) so that `fulfill()` can close the agent's work commitment after human approval. An in-memory map would lose this state on JVM restart, leaving pending gates permanently unresolvable.

### The technique
**Write (at gate-open time):**
```java
private String serializeGateContent(GateContext ctx, String reason) {
    Properties props = new Properties();
    props.setProperty("originalCommitmentId", ctx.originalCommitmentId());
    props.setProperty("workChannelId", ctx.workChannelId().toString());
    props.setProperty("commandMessageId", String.valueOf(ctx.commandMessageId()));
    props.setProperty("reason", reason != null ? reason : "");
    StringWriter sw = new StringWriter();
    try {
        props.store(sw, null);  // StringWriter.write() never throws IOException
    } catch (IOException e) {
        throw new IllegalStateException("unreachable", e);
    }
    return sw.toString();
}
```

**Read (at gate-resolve time):**
```java
private Optional<GateContext> parseGateContent(String content) {
    if (content == null || content.isBlank()) return Optional.empty();
    try {
        Properties props = new Properties();
        props.load(new StringReader(content));
        String oci = props.getProperty("originalCommitmentId");
        String wci = props.getProperty("workChannelId");
        String cmi = props.getProperty("commandMessageId");
        if (oci == null || wci == null || cmi == null) return Optional.empty();
        return Optional.of(new GateContext(oci, UUID.fromString(wci), Long.parseLong(cmi)));
    } catch (Exception e) {
        log.warnf("parseGateContent: failed to parse: %s", e.getMessage());
        return Optional.empty();
    }
}
```

Dispatch the COMMAND with the serialized content:
```java
messageService.dispatch(MessageDispatch.builder()
    .channelId(oversightChannel.id)
    .sender(GATE_SENDER)
    .type(MessageType.COMMAND)
    .content(serializeGateContent(ctx, gate.reason()))
    .correlationId(gateId.toString())
    .actorType(ActorType.AGENT)
    .build());
```

### Why this is non-obvious
The obvious approach for state that must survive restarts is a dedicated database table or in-memory map. The database approach adds schema/migration complexity for a small amount of data. The in-memory map loses state on restart. A third approach — storing state in the message that triggered the workflow — requires no additional infrastructure: the Qhorus message is already persisted, the content field is already stored, and the processing code already retrieves the message. The Properties format uses only `java.io` (available in any JVM module, no Jackson dep needed in library modules), is human-readable in the Qhorus ledger, and `Properties.load()` handles the timestamp comment that `store()` adds automatically.

The `StringWriter` will not throw `IOException` even though `Properties.store(Writer, String)` declares it — `StringWriter.write()` never throws. The `try/catch` with `throw new IllegalStateException("unreachable")` is the correct idiom for this case.
