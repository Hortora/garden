## Symptom

All REST endpoints that write ledger entries fail with HTTP 500. Quarkus log shows:

```
org.hibernate.exception.SQLGrammarException: Could not prepare statement
[Column "TENANCY_ID" not found; SQL statement:
  ... ledger_subject_sequence ...]
```

at `LedgerSequenceAllocator.h2NextSequenceNumber()`.

## Context

`ledger_subject_sequence` is a plain SQL table (not a JPA entity) used by
`LedgerSequenceAllocator` to allocate per-subject sequence numbers for ledger
entries. In H2 test environments, Hibernate's `drop-and-create` only creates
JPA entity tables — plain SQL tables must be created via
`quarkus.hibernate-orm."qhorus".sql-load-script` pointing to a SQL file with
`CREATE TABLE IF NOT EXISTS`.

A casehub-ledger SNAPSHOT added `tenancy_id` to the table and changed the
primary key from `(subject_id)` to `(subject_id, tenancy_id)`. The consumer
project's `import-qhorus.sql` still had the old DDL with only `subject_id` as PK.
Flyway migrations in production handled the column addition automatically, but
H2 tests have no Flyway for this table — only the static `import-qhorus.sql`.

## Old DDL (breaks after SNAPSHOT)

```sql
CREATE TABLE IF NOT EXISTS ledger_subject_sequence (
    subject_id UUID NOT NULL PRIMARY KEY,
    next_seq   INT  NOT NULL DEFAULT 1
);
```

## Fix

Update `import-qhorus.sql` to match the current SNAPSHOT schema:

```sql
CREATE TABLE IF NOT EXISTS ledger_subject_sequence (
    subject_id UUID         NOT NULL,
    tenancy_id VARCHAR(255) NOT NULL DEFAULT '278776f9-e1b0-46fb-9032-8bddebdcf9ce',
    next_seq   INT          NOT NULL DEFAULT 1,
    CONSTRAINT pk_ledger_subject_sequence PRIMARY KEY (subject_id, tenancy_id)
);
```

The default value `278776f9-e1b0-46fb-9032-8bddebdcf9ce` is `TenancyConstants.DEFAULT_TENANT_ID`
— must stay in sync with that constant.

## Why this is non-obvious

The error points at `LedgerSequenceAllocator`, not at any consumer code or migration.
Because production uses Flyway (which applies the column addition), the schema drift
only manifests in H2 tests. There is no changelog or migration note in the casehub-ledger
JAR that signals the `import-qhorus.sql` in consumer projects needs to be updated.
The only way to discover the new schema is to read the Flyway migration SQL from inside
the ledger JAR: `db/ledger/migration/V1000__ledger_base_schema.sql`.
