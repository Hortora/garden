In casehub-ledger's `LedgerEntry.attach(LedgerSupplement supplement)` method, the implementation sets `supplement.ledgerEntry = this` before adding the supplement to the collection. This is the owning-side back-reference for the `@OneToMany(mappedBy = "ledgerEntry")` bidirectional relationship.

```java
public void attach(final LedgerSupplement supplement) {
    Objects.requireNonNull(supplement, "supplement must not be null");
    supplement.ledgerEntry = this;   // ← sets the FK owner before cascade
    supplements.removeIf(s -> s.getClass() == supplement.getClass());
    supplements.add(supplement);
    supplementJson = LedgerSupplementSerializer.toJson(supplements);
}
```

When reading the `@OneToMany(mappedBy = "ledgerEntry", cascade = CascadeType.ALL)` declaration and the `@JoinColumn(name = "ledger_entry_id", nullable = false)` constraint on `LedgerSupplement`, it appears that the FK would be null if you only add to the collection without setting the back-reference. The `attach()` method resolves this silently.

### Why this is non-obvious

Standard JPA advice says to set both sides of a bidirectional relationship manually. `attach()` looks like just a helper for managing the collection and the denormalized `supplementJson` field — the FK ownership assignment is invisible unless you read the full method body. Assuming `supplement.ledgerEntry` is null leads to believing the FK write would fail at the DB level, causing unnecessary investigation into cascade configuration.
