## @BuildStep producing ExcludedTypeBuildItem silently not invoked during Quarkus test augmentation from workspace deployment module

### Symptom

A custom `@BuildStep` method in a Quarkus extension's deployment module that produces `ExcludedTypeBuildItem` is never invoked during `mvn test` augmentation. Both patterns fail silently:

```java
// Pattern 1 — void with BuildProducer
@BuildStep
void selectStack(BuildProducer<ExcludedTypeBuildItem> excluded) {
    excluded.produce(new ExcludedTypeBuildItem("io.example.SomeBean"));
}

// Pattern 2 — direct List return
@BuildStep
List<ExcludedTypeBuildItem> selectStack() {
    return List.of(new ExcludedTypeBuildItem("io.example.SomeBean"));
}
```

No error, no warning. The method is compiled correctly, appears in bytecode, and the processor class is listed in `META-INF/quarkus-build-steps.list`. But the beans are never excluded — the full CDI container starts with both stacks active, causing conflicts (e.g. duplicate `@Tool` method names with quarkus-mcp-server).

The `feature()` `@BuildStep` (which returns `FeatureBuildItem`) in the same class **does** run — the feature appears in the Quarkus startup log.

### Context

Triggered when running tests in the runtime module (`mvn test -pl runtime`) of a multi-module Quarkus extension where:
- The deployment module is a local workspace sibling (Quarkus `WorkspaceLoader` resolves it from `deployment/target/classes/`, not from the .m2 JAR)
- The `@BuildStep` produces a `MultiBuildItem` subtype (`ExcludedTypeBuildItem` extends `MultiBuildItem`)
- The method is package-private (standard for Quarkus processors)

### What was tried

- `BuildProducer<ExcludedTypeBuildItem>` parameter (standard pattern) — not invoked
- `List<ExcludedTypeBuildItem>` return type — not invoked
- `@ConfigMapping @ConfigRoot(BUILD_TIME)` config injection as parameter — not invoked
- `ConfigProvider.getConfig()` direct read — not invoked
- Added `System.err.println` debug line inside the method — never printed
- Verified JAR bytecode contains `selectStack` method, FQCN strings, and debug text — correct
- Verified `META-INF/quarkus-build-steps.list` and `META-INF/quarkus-config-roots.list` — correct
- Installed updated JAR to .m2 before running tests — same result
- `mvn clean test` to clear augmentation cache — same result

### Root cause

Unknown. The `WorkspaceLoader` resolves the deployment module from `deployment/target/classes/` during test augmentation. For unknown reasons, the Quarkus build framework does not invoke custom `@BuildStep` methods that produce optional `MultiBuildItem` subtypes from locally-resolved workspace deployment modules in Quarkus 3.32.2. The `feature()` `@BuildStep` (producing `FeatureBuildItem`) does run, suggesting the issue is specific to `MultiBuildItem` producers or to the `ExcludedTypeBuildItem` consumer chain.

### Fix

Use `@IfBuildProperty` / `@UnlessBuildProperty` per-bean. ARC's own build processor handles these annotations and does run during test augmentation. For custom properties, formally declare the property as `BUILD_TIME` via `@ConfigMapping @ConfigRoot(phase = BUILD_TIME)` in the deployment module — this makes `@IfBuildProperty` reliable for custom (non-Quarkus-datasource) properties:

```java
// deployment/ module
@ConfigMapping(prefix = "casehub.qhorus")
@ConfigRoot(phase = ConfigPhase.BUILD_TIME)
public interface QhorusBuildTimeConfig {
    ReactiveConfig reactive();
    interface ReactiveConfig {
        @WithDefault("false")
        boolean enabled();
    }
}
```

```java
// runtime beans — per-bean annotation with new property name
@IfBuildProperty(name = "casehub.qhorus.reactive.enabled", stringValue = "true")
@ApplicationScoped
public class ReactiveChannelService { ... }
```

`QhorusProcessor` retains only `feature()` — no custom `@BuildStep` for exclusion.

### Why this is non-obvious

The deployment module compiles and installs correctly. The bytecode, `quarkus-build-steps.list`, and `quarkus-config-roots.list` are all correct. The `feature()` `@BuildStep` in the same class runs. There is no error or warning. The only observable symptom is that the beans are not excluded — which manifests as a completely different error (e.g. duplicate MCP tool names) that obscures the actual root cause.
