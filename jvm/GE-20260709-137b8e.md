### Context

When two peer Tier 1 API modules (e.g. `memory-api` and `rag-api`) need to share utility code but cannot depend on each other — neither is a parent of the other, and adding a dependency would force one module's consumers to see the other's entire domain type surface.

### The convention

Extract the shared code to a dedicated zero-dep Tier 1 utility module. Both peer modules depend on the utility module. The utility module has zero first-party dependencies — pure Java only.

### Why non-obvious

The tempting alternatives — duplicating the code, or having one peer depend on the other — both seem simpler. Duplication works initially but diverges over time (three copies of the same fusion algorithm, each with slightly different behaviour). Peer-to-peer dependency is structurally wrong: `rag-api` depending on `memory-api` (or vice versa) means every RAG consumer pulls in all CBR types.

### Alternatives considered

- **Duplicate the code** in each peer module — accepted during early development but diverges over time. Three copies of the same fusion algorithm developed different behaviours (grade merging in rag, putIfAbsent in memory).
- **One peer depends on the other** — wrong; forces the dependent module to see all of the other's domain types (e.g. `CbrQuery`, `CbrCase`, `CbrFeatureSchema` leak into RAG consumers).
- **Move to a common ancestor module** — no common ancestor exists between `memory-api` (depends on `platform-api`) and `rag-api` (zero first-party deps).

### First instance

`ScoreFusion` + `FusionStrategy` + `CamelCaseExpander` extracted from `memory-api` and `rag-api` into `fusion-api` (`casehub-neocortex-fusion-api`). The utility module has zero first-party deps, pure Java. Both `memory-api` and `rag` (runtime) depend on it.
