## Symptom

`CompletableFuture.orTimeout(duration.toSeconds(), TimeUnit.SECONDS)` fires immediately (0-second timeout) when `duration` is sub-second (e.g. `Duration.ofMillis(50)`). Tests using sub-second timeouts pass by accident — the future completes exceptionally within any reasonable sleep window, masking the truncation.

## Context

`CompletableFuture.orTimeout(long, TimeUnit)` accepts a scalar value and unit. The natural composition with `java.time.Duration` is `duration.toSeconds()` + `TimeUnit.SECONDS`. This truncates — `Duration.ofMillis(500).toSeconds()` returns `0`, not `0.5`. The timeout schedules with a 0-second delay, fires immediately, and the future completes exceptionally with `TimeoutException`.

Production code typically uses multi-second timeouts (30s, 120s) where truncation is invisible. The bug surfaces in tests using short durations, or in any path where sub-second precision matters.

## What was tried (didn't work)

```java
future.orTimeout(timeout.toSeconds(), TimeUnit.SECONDS);
```

With `Duration.ofMillis(50)`: `toSeconds()` returns `0` → timeout fires immediately → test passes by accident (future is already completed exceptionally before the assertion runs).

## Root cause

`Duration.toSeconds()` performs integer division — it truncates, not rounds. `orTimeout(0, SECONDS)` schedules a task with zero delay in the `CompletableFuture.Delayer` pool, which fires on the next scheduler tick.

## Fix

```java
future.orTimeout(timeout.toMillis(), TimeUnit.MILLISECONDS);
```

`toMillis()` preserves sub-second precision. `Duration.ofMillis(50).toMillis()` returns `50`. For multi-second durations, `toMillis()` returns the correct value (e.g. `Duration.ofSeconds(120).toMillis()` returns `120000`).

## Why this is non-obvious

The `toSeconds()` + `SECONDS` composition reads naturally and works for all production-realistic timeouts (seconds to minutes). The truncation is invisible until a test or edge case uses sub-second durations. The test failure mode — passing by accident — makes the bug self-hiding.

### Alternatives considered

- `toNanos()` + `TimeUnit.NANOSECONDS` — unnecessarily precise; millisecond resolution is sufficient for timeout scheduling
- Custom rounding (`Math.max(1, duration.toSeconds())`) — wrong fix; masks the precision loss instead of preserving it
