### Symptom
Quarkus deployment warning or error when `@Blocking` appears on an `@ApplicationScoped`
CDI service bean method. The application may deploy but the annotation has no effect,
or the deployment fails with a validation error about misplaced interceptor binding.

### Context
Quarkus 3.x with reactive stack (Vert.x). A CDI `@ApplicationScoped` service performs
blocking I/O (e.g. JGit, file writes) and is called from a JAX-RS endpoint.

### What was tried
Annotating the CDI service method with `@Blocking` to signal that it performs blocking
I/O and should run on the blocking executor.

### Root cause
`@Blocking` is a framework-entrypoint annotation. It tells the Quarkus reactive routing
layer to dispatch the method onto a worker thread. It is valid on:
- JAX-RS resource methods (`@GET`, `@POST`, etc.)
- Kafka `@Incoming` consumer methods
- Quarkus Scheduler `@Scheduled` methods
- Quarkus @Tool MCP methods (via the MCP extension)

It is NOT valid on plain CDI service bean methods called from application code.
When called from a JAX-RS sync endpoint, the execution is already on a worker thread
by default — no annotation is needed. When called from a reactive context, the caller
must explicitly switch executors.

### Fix
Remove `@Blocking` from the CDI service method. JAX-RS sync endpoints (non-`@NonBlocking`)
already execute on the blocking executor — blocking calls from within the request handler
are safe without any annotation on the service itself.

### Why this is non-obvious
The annotation looks like a general hint to the JVM about blocking intent, similar to
`@Transactional`. It is actually a routing directive consumed by specific Quarkus extension
interceptors, not a general threading contract. Framework docs show `@Blocking` on endpoint
methods; developers generalise the pattern to service methods where it silently does nothing
or causes validation failures.
