## Symptom

Adding `import static org.assertj.core.api.Assertions.assertThat;` to a test in
`runtime/src/test/java/` fails with a confusing compiler error:

```
[ERROR] CommitmentToolTest.java:[3,1] static import only from classes and interfaces
[ERROR] CommitmentToolTest.java:[3,35] package org.assertj.core.api does not exist
```

The "static import only from classes and interfaces" message makes it look like a
Java language syntax error, not a missing dependency.

## Context

Quarkiverse multi-module project where `assertj-core` is available in the `testing/`
module (which has it as a test dependency) but is NOT a dependency of the `runtime/`
module's test scope. Tests in `runtime/src/test/java/` cannot use assertj.

## Root Cause

`assertj-core` is absent from the `runtime` module's test classpath. The Java compiler
emits "static import only from classes and interfaces" as a secondary error when the
class being statically imported doesn't exist — the real error is the missing package,
but the secondary error appears first and is more confusing.

## Fix

Use JUnit 5 assertions instead of assertj in `runtime/src/test/`:

```java
// Instead of: import static org.assertj.core.api.Assertions.assertThat;
import static org.junit.jupiter.api.Assertions.assertEquals;
import static org.junit.jupiter.api.Assertions.assertTrue;

// Instead of: assertThat(list).hasSize(1)
assertEquals(1, list.size());

// Instead of: assertThat(list).isEmpty()
assertTrue(list.isEmpty());
```

Or add assertj to `runtime/pom.xml` test scope if assertj fluency is important there:

```xml
<dependency>
    <groupId>org.assertj</groupId>
    <artifactId>assertj-core</artifactId>
    <version>3.27.3</version>
    <scope>test</scope>
</dependency>
```

## Why Non-Obvious

The error "static import only from classes and interfaces" reads as a Java syntax
constraint violation, not as a missing dependency. Developers familiar with Java's
static import rules spend time checking the import syntax before realising the class
simply isn't on the classpath. The real error (package does not exist) appears
second and is easy to miss.
