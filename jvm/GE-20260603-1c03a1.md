# Mutiny Multi.createFrom().publisher() requires Flow.Publisher — JdkFlowAdapter needed for Reactor Flux

## Symptom

```java
Flux<String> sdkFlux = client.connect(prompt).textStream();
Multi<String> events = Multi.createFrom().publisher(sdkFlux);  // compile error
```

Compilation fails with a type mismatch — `Flux<String>` is not assignable to the parameter type of `Multi.createFrom().publisher()`.

## Root cause

Mutiny's `Multi.createFrom().publisher()` accepts `java.util.concurrent.Flow.Publisher<T>` (JDK 9+ standard). Reactor's `Flux` implements `org.reactivestreams.Publisher<T>` (Reactive Streams standard), not `java.util.concurrent.Flow.Publisher<T>` directly — or the implementation is not reliably available at the Mutiny overload resolution level.

## Fix

```java
import reactor.adapter.JdkFlowAdapter;

Flux<String> sdkFlux = client.connect(prompt).textStream();
Multi<String> events = Multi.createFrom()
    .publisher(JdkFlowAdapter.publisherToFlowPublisher(sdkFlux));
```

`JdkFlowAdapter` is in `reactor-core` (the same jar that `Flux` comes from), so no extra dependency is required.

## Why non-obvious

Both `Flow.Publisher` (JDK) and `org.reactivestreams.Publisher` are "the reactive publisher interface." Experienced reactive developers know both exist but assume Mutiny (being Reactive Streams compliant) would bridge either. The actual parameter type being `Flow.Publisher` and not `org.reactivestreams.Publisher` is a subtle API decision that only surfaces as a compile error when you try to mix the two ecosystems.

Reactor does declare compatibility with both standards, but the `JdkFlowAdapter` bridge is required for the Mutiny type system to accept it cleanly.
