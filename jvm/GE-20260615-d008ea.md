## Symptom

A `KAFKA` stream endpoint is registered in `EndpointRegistry` with `EndpointPropertyKeys.URL` set to the Kafka broker address. The property is silently stored and silently ignored — no component reads `URL` from a `KAFKA` descriptor.

## Context

When building stream endpoint descriptors for `EndpointRegistry`, the natural impulse is to set `EndpointPropertyKeys.URL` to the broker connection string (e.g. `broker:9092`). This seems consistent with how `URL` is used for HTTP and GRPC endpoints.

## Root cause

`EndpointPropertyKeys.URL` is explicitly scoped to specific protocols. Its Javadoc lists applicability:
- HTTP — service root URL
- GRPC — `host:port` or `grpc://` URI
- MCP — MCP server base URL
- CAMEL — Camel endpoint URI
- QHORUS — Qhorus REST base URL

**KAFKA is absent.** Furthermore, the class-level Javadoc states: "Deployment-local properties (bootstrap servers, TLS config, Camel route IDs) remain as module-defined keys and are not reserved." Bootstrap servers are explicitly classified as deployment-local, not registered in `EndpointPropertyKeys`.

## Fix

For `KAFKA` (and `AMQP`) stream endpoint descriptors, include only `TOPIC` and `STREAM_EVENT_TYPE` in properties. Kafka broker connection is configured via Quarkus standard config (`kafka.bootstrap.servers`) outside the EndpointDescriptor entirely.

```java
// WRONG — URL is ignored for KAFKA protocol
EndpointDescriptor wrong = new EndpointDescriptor(
    path, tenancyId, EndpointType.SYSTEM, EndpointProtocol.KAFKA,
    Map.of(
        EndpointPropertyKeys.URL, "broker:9092",   // silently ignored
        EndpointPropertyKeys.TOPIC, "patient.vitals",
        EndpointPropertyKeys.STREAM_EVENT_TYPE, "io.casehub.iot.temperature"
    ), null, Set.of(EndpointCapability.RECEIVE));

// CORRECT — only TOPIC and STREAM_EVENT_TYPE
EndpointDescriptor correct = new EndpointDescriptor(
    path, tenancyId, EndpointType.SYSTEM, EndpointProtocol.KAFKA,
    Map.of(
        EndpointPropertyKeys.TOPIC, "patient.vitals",
        EndpointPropertyKeys.STREAM_EVENT_TYPE, "io.casehub.iot.temperature"
    ), null, Set.of(EndpointCapability.RECEIVE));
```

## Why this is non-obvious

`URL` feels like a universal "where to connect" key. Every other protocol uses `URL` for the connection address. There is no validation or warning — the wrong key is accepted and stored without complaint. The only way to discover this is to read the Javadoc carefully and notice KAFKA is not in the applicability list.
