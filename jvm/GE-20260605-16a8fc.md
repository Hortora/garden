### Symptom
A new lifecycle event added to `WorkItemService` fires correctly (no exception, audit record written) but the downstream CDI observer (`LedgerEventCapture`) silently drops it — no ledger entry is created and no warning is logged unless the observer explicitly logs unrecognised types.

### Context
`WorkItemLifecycleEvent.of(eventName, workItem, actor, detail)` normalizes the event name:
```java
"io.casehub.work.workitem." + eventName.toLowerCase()
```
So `WorkItemLifecycleEvent.of("DELEGATION_ACCEPTED", ...)` produces a type of `"io.casehub.work.workitem.delegation_accepted"`.

The observer uses `eventSuffix()` to extract the last `.`-delimited token (`"delegation_accepted"`) and looks it up in a static `EVENT_META` map. Any suffix not present in the map is logged as unrecognised and the event is dropped — no ledger entry, no exception.

### Root cause
Two conventions must stay in sync manually:
1. `WorkItemService` fires the event with an UPPER_SNAKE_CASE name
2. `LedgerEventCapture.EVENT_META` maps lowercase_underscore suffixes

Adding a new lifecycle event to the service without adding the corresponding lowercase suffix to `EVENT_META` compiles and runs, but the ledger silently omits it.

### Fix
For every new `WorkItemLifecycleEvent.of("MY_EVENT", ...)` call in the service, add a matching entry to `EVENT_META`:
```java
Map.entry("my_event", new String[] { "CommandType", "WorkItemMyEvent", "ActorRole" })
```
The suffix must exactly match `eventName.toLowerCase()`.

### Why this is non-obvious
The factory method Javadoc says "lowercased automatically" but doesn't explain the downstream consequence. The observer's `EVENT_META` looks like it covers the full lifecycle — a developer adding a new event wouldn't naturally scan `LedgerEventCapture` to check coverage. The drop is logged at WARN level only, and the CI test for it (`CreditDecisionScenarioTest`) wasn't reached until prior compilation errors were fixed — masking the gap for the duration of issue #245's work.
