## Context

Storing `java.time.Instant` values as TEXT in SQLite and relying on lexicographic
`ORDER BY created_at DESC` and `WHERE created_at >= ?` for chronological ordering.
ISO-8601 strings are famously sortable lexicographically — `"2026-05-31T10:00:01Z"` >
`"2026-05-31T10:00:00Z"` is both chronologically and lexicographically correct.

## Symptom

Queries return items in wrong chronological order. A later instant sorts before an earlier
one despite ISO-8601 formatting. Some items are excluded from `>=` filters they should match.

Example: `"2026-05-31T10:00:00Z"` (20 chars) sorts AFTER `"2026-05-31T10:00:00.100Z"` (24 chars)
even though the 100ms-later timestamp should be greater.

## Root Cause

`java.time.Instant.toString()` emits the minimum fractional-second digits needed to represent
the precision: 0 digits for whole seconds, 3 for milliseconds, 6 for microseconds, 9 for nanoseconds.

A whole-second instant produces `"2026-05-31T10:00:00Z"` (20 chars). A millisecond-precision
instant from `Instant.now()` on most JVMs produces `"2026-05-31T10:00:00.100Z"` (24 chars).

In ASCII ordering, `.` (46) < `Z` (90). The comparison stops at position 19:
- `"…:00Z"` has `Z` at index 19
- `"…:00.100Z"` has `.` at index 19

So the 24-char value sorts lexicographically LESS THAN the 20-char value — the opposite
of chronological order.

This only causes problems when rows have mixed fractional-second widths — e.g. some stored
by one JVM with millisecond clock resolution, others by a JVM with whole-second resolution,
or mixed from different sources.

## Fix

Always truncate to milliseconds before storing, producing a uniform 24-character string:

```java
// Store
String createdAt = Instant.now().truncatedTo(ChronoUnit.MILLIS).toString();
// Always 24 chars: "2026-05-31T10:00:00.000Z"

// Query — also truncate the since parameter before binding
if (query.since() != null) {
    ps.setString(idx++, query.since().truncatedTo(ChronoUnit.MILLIS).toString());
}
```

Truncating to milliseconds is safe for any practical agent memory or audit use case.
Uniform width guarantees lexicographic = chronological ordering for all rows regardless
of when or where they were written.

## Why This Is Non-Obvious

ISO-8601 is widely cited as "sortable by design" — and it IS, when the strings have the same
width. The width variation is the invisible assumption. Java's `Instant.toString()` is documented
to emit the minimum necessary digits, but the ordering implication for SQLite TEXT storage is
not mentioned anywhere in either the Java or SQLite documentation. A developer who knows that
ISO-8601 is sortable will not think to check fractional-second digit width.

## Constraints

Applies to any database that stores timestamps as TEXT strings and relies on lexicographic
ordering. Not a problem in PostgreSQL (`TIMESTAMP WITH TIME ZONE` is a native type with
correct numeric comparison), H2 in `MODE=PostgreSQL` (same), or any system where timestamps
are stored as integers (Unix epoch). Only affects TEXT-based timestamp storage.
