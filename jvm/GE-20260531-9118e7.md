## Symptom

`@QuarkusTest` startup fails with `AmbiguousResolutionException` for an SPI type that has a `@DefaultBean` implementation:

```
Build failure: Build failed due to errors
  [error]: AmbiguousResolutionException: Ambiguous dependencies for type
  GroupMembershipProvider and qualifiers [@Default]
    - CLASS bean [target=MockGroupMembershipProvider]
    - CLASS bean [target=NoOpGroupMembershipProvider]
```

The `@DefaultBean` on `NoOpGroupMembershipProvider` is not suppressing the ambiguity.

## Context

A shared library jar contains mock implementations alongside production code and is declared `<scope>runtime</scope>` in the consuming module's `pom.xml`. The scope was changed from `test` to `runtime` to satisfy a different CDI injection point at production build time. The side effect — bringing the mock into the CDI scan — was not anticipated.

## Root Cause

Maven's `runtime` scope is available at both production runtime AND test runtime. Quarkus CDI discovery scans all classpath entries at `@QuarkusTest` startup, including runtime-scoped jars. `@DefaultBean` suppression only activates when there is exactly one non-default bean. When two plain `@ApplicationScoped` beans implement the same interface, ArC raises ambiguity regardless of `@DefaultBean`.

## Fix

Exclude the unwanted mock from CDI scan in test `application.properties`:

```properties
quarkus.arc.exclude-types=io.example.platform.mock.MockGroupMembershipProvider
```

Add to `src/test/resources/application.properties`. Create the file if it doesn't exist. Apply to every module that uses runtime-scoped shared mock jars with `@QuarkusTest`.

## Why Non-Obvious

`@DefaultBean` reads as "there's always a winner." But `NoOpGroupMembershipProvider` is itself a plain `@ApplicationScoped` bean — it competes equally with the mock. The triggering scope change was made for a completely unrelated injection point, making the connection non-obvious. No compile-time warning; failure only surfaces at CDI container startup.
