## Technique

In Quarkus CDI (Arc), an `@Alternative @Priority(N) @ApplicationScoped` bean is **automatically activated** — it replaces the default production bean without any `quarkus.arc.selected-alternatives` configuration entry.

```java
// No application.properties change needed:
@Alternative
@Priority(1)
@ApplicationScoped
class NoOpLedgerEntryRepository implements LedgerEntryRepository {
  @Override public LedgerEntry save(LedgerEntry e) { return e; }
  @Override public List<LedgerEntry> findBySubjectId(UUID id) { return List.of(); }
  // ... all other methods return empty/no-op
}
```

This works whether the class is in `src/test/java` (test-only stub) or `src/main/java` (always-active alternative).

## Why this is non-obvious

Standard Jakarta CDI requires alternative activation in `beans.xml` or via `@Priority` in a managed beans descriptor. Quarkus's `quarkus.arc.selected-alternatives` property exists for the same purpose, leading most developers to assume configuration is required. In fact, Quarkus Arc respects the CDI 4 spec: `@Priority` on an `@Alternative` implicitly activates it — no descriptor, no property, no beans.xml needed.

The technique is especially useful for satisfying CDI dependencies in test modules that don't need the real implementation (e.g. an `LedgerEntryRepository` that a transitive dependency forces onto the CDI context, but which the test module has no need to use).

## Context where it was applied

When `quarkus-ledger` (via a transitive dependency) forced `LedgerEntryRepository` onto the CDI context in `casehub-blackboard`, `casehub-resilience`, and `casehub-work-adapter` test suites, a `NoOpLedgerEntryRepository` placed in each module's `src/test/java` with `@Alternative @Priority(1)` satisfied the CDI requirement with zero configuration changes.
