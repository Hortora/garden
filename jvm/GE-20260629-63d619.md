### Symptom
Adding SPLADE sparse embeddings to dense retrieval (RRF fusion) churns result sets without improving quality for Java/CDI domain queries. All 28 queries changed, 3 highly relevant entries were displaced while 40 noise entries were removed — net effect ambiguous.

### Context
Hortora engine hybrid search benchmark (#28). SPLADE (`Splade_PP_en_v1`) added as sparse embedding alongside nomic-embed-text dense vectors. RRF fusion (k=60, denseTopK=40, sparseTopK=40). Tested across 14 real-world scenarios covering Java class names, CDI annotations, JPA patterns, and Quarkus framework terms.

SPLADE vocabulary analysis decoded the activated tokens for each query:

| Query | Top SPLADE Expansions | Expected (Java domain) |
|-------|----------------------|----------------------|
| `DefaultBean\|AmbiguousResolutionException` | ambiguity, groups, unclear | cdi, qualifier, alternative, inject |
| `ChatModel\|AgentSession\|LangChain4j` | agent, talk, beauty, renovation, genre | model, inference, provider, adapter |
| `Priority(100)\|CDI priority` | priorities, cds (music), urgency, precedence | qualifier, scope, alternative, bean |

Zero Tier 1 domain term activations across all 14 KW queries. Not one activation of `panache`, `jandex`, `qualifier`, `interceptor`, `singleton`, `persist`, or any unambiguous Java/CDI term.

### Root cause
Two compounding failures:

1. **BERT WordPiece tokenizer** fragments Java class names into meaningless subwords: `AmbiguousResolutionException` → `["am", "##bi", "##guous", "##reso", "##lution", "##exception"]`. The semantic content is destroyed before SPLADE runs.

2. **MS MARCO training data** contains no Java ecosystem content. SPLADE's learned vocabulary expansion maps "chat" to web-domain associations (hotels, beauty, renovation) — not to software engineering concepts.

This is expected behaviour for this model architecture, not a defect. The same limitation applies to nomic-embed-text (see GE-20260627-4712de) — the difference is that SPLADE actively ADDS noise through irrelevant expansions, whereas nomic-embed-text merely fails to discriminate.

### Fix
Do not use general-purpose (MS MARCO-trained) SPLADE models for Java/CDI domain retrieval. Three alternatives:

1. **BM25 keyword matching** (casehubio/neural-text#47/#48) — exact token matching via Qdrant full-text index. Directly addresses the vocabulary gap.
2. **Code-domain SPLADE model** (casehubio/neural-text#49) — if one exists with Java training data.
3. **Code-domain dense embedding model** (casehubio/neural-text#49) — models like CodeBERT, UniXcoder trained on source code.

### Why this is non-obvious
SPLADE is presented as a vocabulary expansion technique that recovers keyword precision lost by dense embeddings. The expectation is that it bridges the gap between semantic search and keyword matching. For general-domain web queries, this is true. For Java class names, it actively makes things worse by introducing irrelevant web-domain associations into the sparse vector.

### Why this fix
BM25 is the highest-leverage fix because it directly solves the failure mode (exact token matching for Java identifiers) without requiring model training or fine-tuning. Code-domain models are the long-term architectural fix but have higher uncertainty and longer timeline.
