### Symptom
Batch processing loop executes zero iterations or throws an unexpected exception. No error message indicates overflow — the batch count silently becomes negative or zero.

### Context
Using `Integer.MAX_VALUE` as a "no batching" sentinel — meaning "process everything in one batch." The standard ceiling division formula is used to compute the number of batches.

### What was tried (didn't work)
```java
int totalBatches = (chunks.size() + batchSize - 1) / batchSize;
```
When `batchSize = Integer.MAX_VALUE` and `chunks.size() > 0`, the addition `chunks.size() + Integer.MAX_VALUE - 1` overflows to a negative `int`. The division then produces 0 or -1, and the batch loop never executes.

### Root cause
Java `int` arithmetic silently wraps on overflow. `5 + 2147483647 - 1` = `-2147483645` (wrapped), not `2147483651`. The ceiling division formula assumes both operands are well below `Integer.MAX_VALUE`.

### Fix
Clamp `batchSize` to `chunks.size()` before the division. When `batchSize >= size`, the result is always 1 batch — which is the correct "no batching" semantics.

```java
int effectiveBatchSize = Math.min(batchSize, chunks.size());
int totalBatches = (chunks.size() + effectiveBatchSize - 1) / effectiveBatchSize;
```

### Why this is non-obvious
The ceiling division formula `(n + d - 1) / d` is universally taught as correct for positive integers. It IS correct — for values that don't overflow. Using `Integer.MAX_VALUE` as a sentinel is a common Java pattern ("no limit"), and the formula looks safe because both operands are positive. The overflow is silent.
