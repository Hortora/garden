## Symptom

Quarkus startup fails with `AmbiguousResolutionException: Ambiguous dependencies for type io.serverlessworkflow.impl.WorkflowApplication and qualifiers [@Default]` when you add a custom CDI producer for `WorkflowApplication`.

## Context

When integrating quarkus-flow into a Quarkus module that needs to control `WorkflowApplication` lifecycle (e.g. ensuring a singleton shared across workflow executions), it's natural to write a `@Produces @ApplicationScoped WorkflowApplication` bean.

## What was tried (didn't work)

Adding `WorkflowApplicationProducer`:

```java
@ApplicationScoped
public class WorkflowApplicationProducer {
    @Produces @ApplicationScoped
    public WorkflowApplication produce() {
        return WorkflowApplication.builder().build();
    }
}
```

Quarkus startup throws `AmbiguousResolutionException` — two `@ApplicationScoped WorkflowApplication` beans exist.

## Root cause

The `quarkus-flow` extension includes a `WorkflowApplicationRecorder` that registers `WorkflowApplication` as a CDI bean via the Quarkus build-time extension mechanism (`WorkflowApplicationRecorder.class` visible in the quarkus-flow jar). The extension recorder fires before CDI deployment, injecting its own `WorkflowApplication` singleton into the container. A user-written `@Produces` bean creates a second one.

## Fix

Remove the custom producer entirely. The `WorkflowApplication` singleton is managed by the quarkus-flow extension and is available for `@Inject` without any producer. To customise the application, investigate the quarkus-flow extension configuration properties instead.

## Why this is non-obvious

The quarkus-flow jar does not contain a `beans.xml` with an obvious CDI bean declaration. The registration happens via `WorkflowApplicationRecorder` — a Quarkus build-time recorder, which is invisible to standard CDI scanning and the `AmbiguousResolutionException` message names the type but not which class introduced the conflict.
