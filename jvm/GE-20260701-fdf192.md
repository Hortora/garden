### Symptom
`PolicyEnforcer.execute()` throws `PolicyEnforcementException("All 1 attempts failed", PolicyEnforcementException("timed out"))` — a double-wrapped exception where the top-level type says "retries exhausted" but the actual failure was a timeout. Callers catching `PolicyEnforcementException` see "All 1 attempts" and treat it as retry exhaustion when it should be treated as a timeout.

### Context
`DefaultPolicyEnforcer` has a retry loop that catches `Exception` generically. After all attempts exhaust, it wraps the last exception in `new PolicyEnforcementException("All N attempts failed", lastException)`. This wrapping is unconditional — even if `lastException` is itself a `PolicyEnforcementException` (from timeout or interrupt handling), it gets wrapped again.

With `maxAttempts=1` (the default), a timeout produces: inner `PolicyEnforcementException("timed out")` → caught by retry loop → wrapped in `PolicyEnforcementException("All 1 attempts failed", inner)`. The type at the top level is always the same generic base class regardless of what actually failed.

### What was tried (didn't work)
1. **Message-based dispatch** — checking `e.getMessage().contains("timed out")` in the caller. Fragile — message changes break dispatch silently.
2. **Cause-based dispatch** — checking `e.getCause() == null` for timeout (the only case without a cause). Couples to an implementation detail of which exceptions get constructed with a cause parameter.

### Root cause
The retry loop does not distinguish between "the action itself failed" and "the policy infrastructure detected a condition (timeout, interrupt)." Both get the same wrapping treatment, erasing the semantic distinction at the type level.

### Fix
Re-throw `PolicyEnforcementException` subclasses directly from the retry loop — do not wrap them:

```java
if (lastException instanceof PolicyEnforcementException pe) {
    throw pe;  // timeout, interrupt — preserve the type
}
throw new RetryExhaustedException(
    "All " + maxAttempts + " attempts failed", lastException);
```

Combined with typed subclasses (`TimeoutPolicyException`, `InterruptedPolicyException`, `RetryExhaustedException`), callers can dispatch by type in catch blocks — no message matching, no cause inspection.

### Why this is non-obvious
The wrapping code looks correct at first glance — "if all attempts failed, wrap and throw" is a standard retry pattern. The bug is that "all attempts failed" conflates two meanings: "the action kept failing" (retry exhaustion) and "the infrastructure detected a terminal condition" (timeout, interrupt). A developer reading the retry loop wouldn't see the semantic loss without tracing through a timeout scenario.

### Alternatives considered
- **Unwrap before re-throwing** — check `lastException instanceof PolicyEnforcementException` and unwrap. Rejected: unwrapping is error-prone and the wrapper adds no value for policy exceptions.
- **Add a `cause type` field to PolicyEnforcementException** — enum discriminator on the base class. Rejected: typed subclasses are the idiomatic Java solution; an enum discriminator forces `switch` instead of `catch`.
