### Symptom

`java.net.http.HttpClient.newWebSocketBuilder().buildAsync(uri, listener)` fails to establish a WebSocket connection to any RFC 6455-compliant server. The connection attempt completes the TCP handshake and HTTP upgrade request, but the client rejects the server's `Sec-WebSocket-Accept` response header as invalid.

### Context

Building a Discord Gateway WebSocket client using `HttpHelper.CLIENT.newWebSocketBuilder()` in casehub-connectors. The client connected to an embedded test server but would fail against Discord's production Gateway (or any standard WebSocket server).

### What was tried (didn't work)

Assumed the JDK's `java.net.http.WebSocket` client uses the standard RFC 6455 magic GUID (`258EAFA5-E914-47DA-95CA-C5AB0DC85B63`) for the Sec-WebSocket-Accept handshake computation. Built an embedded test WebSocket server using the RFC GUID — connection failed.

### Root cause

JDK Corretto 22.0.2.9.1 (resolved via `java_home -v 26` on macOS) has a modified GUID constant in `jdk.internal.net.http.websocket.OpeningHandshake`:

```
258EAFA5-E914-47DA-95CA-C5AB0DC85B11
```

instead of the RFC 6455 standard:

```
258EAFA5-E914-47DA-95CA-C5AB0DC85B63
```

Confirmed via bytecode inspection:
```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) $JAVA_HOME/bin/javap -verbose \
  -cp $JAVA_HOME/jmods/java.net.http.jmod \
  jdk.internal.net.http.websocket.OpeningHandshake 2>&1 | grep 258EAFA5
```

The server computes `SHA-1(client_key + "...B63")` but the JDK client validates against `SHA-1(client_key + "...B11")`. The mismatch causes the client to reject the handshake.

### Fix

For test servers: use the JDK's non-standard GUID (`...B11`) so the handshake succeeds with the JDK client. This makes tests pass but hides the production incompatibility.

For production: use a different JDK vendor/version, or replace `java.net.http.WebSocket` with a third-party WebSocket client library that uses the correct RFC GUID.

### Why this is non-obvious

The GUID is a fixed protocol constant defined in RFC 6455 §4.2.2. It has no cryptographic significance — it exists solely to prevent non-WebSocket HTTP servers from accidentally completing the handshake. There is no reason to change it, no changelog documenting the change, and no JDK release note mentioning it. The only way to discover it is via `javap` on the internal class. Tests using the RFC GUID fail with no meaningful error — the WebSocket connection simply doesn't complete.

### Alternatives considered

- **Use WireMock for WebSocket tests** — WireMock does not support WebSocket
- **Use Jetty embedded WebSocket server** — would add an external dependency to a zero-dep project
- **Detect JDK GUID at runtime** — theoretically possible via reflection on `jdk.internal.net.http.websocket.OpeningHandshake` but fragile and module-access-restricted
