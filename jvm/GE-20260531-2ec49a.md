### Symptom

`folder.idle(false)` in a loop never returns between messages — the IDLE mode
persists indefinitely and only terminates when another thread explicitly aborts
it (e.g. via `store.close()`). An IDLE loop that calls `idle(false)` effectively
blocks the thread permanently between server-initiated aborts.

### Context

Implementing an IMAP IDLE message delivery loop using angus-mail / Jakarta Mail.
The standard pattern is: enter IDLE, block until server notification, process
new messages, repeat.

```java
// BROKEN — idle(false) never returns on its own
while (!stopping) {
    folder.idle(false); // blocks indefinitely
    processUnseen(folder, account, sink);
}

// CORRECT — idle(true) returns after first server notification
while (!stopping) {
    folder.idle(true);  // returns after EXISTS, EXPUNGE, or server timeout
    processUnseen(folder, account, sink);
}
```

### Root Cause

The boolean parameter in `IMAPFolder.idle(boolean once)` is named `once` in
the angus-mail JavaDoc. `once=true` aborts after the first untagged server
response (EXISTS, EXPUNGE, FLAGS, etc.), returning control to the caller.
`once=false` is continuous IDLE — it remains in IDLE mode until explicitly
terminated by `idleAbort()` or a socket close. The naming and method signature
suggest a boolean "wait" flag, which leads developers to assume `idle(false)`
is non-blocking or standard blocking-until-notification. It is neither.

### Fix

Use `idle(true)` in any loop pattern where you want to process messages between
each server notification:

```java
while (!stopping) {
    folder.idle(true);  // blocks until ONE server notification, then returns
    processUnseen(folder, account, sink);
}
```

### Why This Is Non-Obvious

`idle()` with no arguments (equivalent to `idle(false)`) is the method shown
in most IMAP IDLE tutorials. The overload `idle(boolean once)` is documented
but the parameter semantics are the opposite of what "idle(false)" implies.
The angus-mail Javadoc names it `once` but does not make the continuous-vs-single
distinction prominent. The failure mode (loop that processes nothing after the
first message) appears only under specific timing conditions and may not surface
in tests where delivery is deterministic.
