### Symptom
Integration test expects a WorkItem to be created after firing a case event. The assertion sees an empty store — `Expected size: 1 but was: 0`. No error, no exception, no log warning.

### Context
A `@ConsumeEvent(blocking = true)` handler calls `WorkItemService.create()`, which transitively injects `TenantScopedPrincipal` — a `@RequestScoped` bean. The handler runs on a Quarkus worker thread where no request context is active. CDI proxy throws `ContextNotActiveException` internally, but the event bus swallows it — the handler appears to complete normally.

The `@RequestScoped` bean wins CDI resolution over a `@DefaultBean @ApplicationScoped` alternative because non-default beans have higher priority. So the broken bean is always selected.

### What was tried (didn't work)
- Checking event bus wiring (correct — other handlers on the same bus work)
- Verifying WorkItemStore contents (empty — the create call never completes)
- Adding `@ActivateRequestContext` to the test method (doesn't help — handler runs on a different thread)

### Root cause
`@ConsumeEvent` handlers — even with `blocking = true` — do not activate a CDI request context. Any `@RequestScoped` bean in the injection chain throws `ContextNotActiveException`. The Vert.x event bus catches the exception and does not propagate it to the caller (fire-and-forget semantics).

### Fix
Exclude the `@RequestScoped` bean from the test CDI context and provide an `@ApplicationScoped` replacement:

```properties
# application.properties
quarkus.arc.exclude-types=io.casehub.work.runtime.service.TenantScopedPrincipal
```

```java
@ApplicationScoped
public class NoOpCurrentPrincipal implements CurrentPrincipal {
    @Override public String actorId() { return "system"; }
    @Override public Set<String> groups() { return Set.of(); }
    @Override public String tenancyId() { return TenancyConstants.DEFAULT_TENANT_ID; }
    @Override public boolean isCrossTenantAdmin() { return false; }
}
```

### Why this is non-obvious
The failure is completely silent. No exception in logs, no test error beyond the assertion. The `@RequestScoped` bean wins CDI resolution automatically (higher priority than `@DefaultBean`), so you never suspect it's the problem. You debug the event bus, the handler logic, and the store — all correct. The request scope issue only surfaces when you trace the full injection chain.
