### Symptom
Need to close multiple resources in `close()` where: (a) both must always close even if one fails, and (b) `close()` must never throw (SPI contract).

### Context
`AutoCloseable.close()` typically uses try-finally to guarantee all resources close. But try-finally propagates exceptions from the try block. When the SPI contract says "close must not throw — implementations swallow cleanup errors", try-finally alone violates the contract.

### Fix
Use sequential try-catch blocks instead of try-finally:

```java
@Override
public void close() {
    if (closed) return;
    closed = true;
    try { session.close(); } catch (Exception ignored) {}
    try { tokenizer.close(); } catch (Exception ignored) {}
}
```

Both resources always close. No exception escapes. The `if (closed) return` guard makes it idempotent.

### Why this is non-obvious
The instinct is try-finally — it's the standard Java resource cleanup pattern. But try-finally serves a different contract: "always clean up, then propagate the error." For "always clean up, never propagate" the sequential try-catch is correct. The spec for this SPI initially said try-finally; the code review caught the tension with the must-not-throw Javadoc.

### Alternatives considered
- **try-finally with catch in finally**: `try { a.close(); } catch (...) {} finally { try { b.close(); } catch (...) {} }` — correct but unnecessarily nested. The sequential pattern is equivalent and clearer.
- **Suppressed exceptions**: `try { a.close(); } catch (Exception e) { try { b.close(); } catch (Exception e2) { e.addSuppressed(e2); } throw e; }` — wrong, because it throws. Only appropriate when close() is allowed to throw.
