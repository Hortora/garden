# TDD for constant renames — write a PASS test first to pin existing behaviour

## Why this is non-obvious

Standard TDD pedagogy is **RED-GREEN-REFACTOR**: write a failing test, make it pass, then refactor. For a constant rename (e.g. `MAX_PROVIDER` → `MAX_MODEL_IDENTIFIER`), this pattern inverts. The natural instinct is to write the test *after* the rename and call sites are updated. But that loses critical evidence: you can no longer distinguish "this test passes because the boundary is correct" from "this test passes because I made a mistake and the compiler accepted it anyway."

The GREEN-first run is a **characterisation test** — it proves the test is meaningful without requiring an intentional failure state.

## The sequence

1. Write the boundary test (e.g. `modelFamily_exceeds_200_chars_throws`)
2. Run it — expect **GREEN** (the bound already applies via the old constant)
3. Add the new correctly-named constant (alongside the old one)
4. Update all call sites to the new name
5. Run again — still **GREEN**

The pre-rename GREEN run is your evidence that the test would fail if the constant were wrong.

## Constraints

Applies to any constant rename, especially constants that encode business rules or limits. Most valuable when the constant encodes a boundary condition or validation rule.

## Alternatives considered

- **Rename without testing first:** Faster initially, but loses the safety net. If you mistype the constant name or the value, you won't know.
- **Write the test after the rename:** Functionally equivalent for a passing test, but you lose the pre-rename baseline. If the test fails later, was the bound wrong originally or did you change it?

## Why this fix

The evidence is the fix. The pre-rename GREEN run proves your test is sensitive to the constant's actual value. Without it, you're working blind on refactoring work.

## Context

Applied during [eidos#24](https://github.com/casehubio/eidos/issues/24) for `MAX_MODEL_IDENTIFIER` and `MAX_DATA_HANDLING_POLICY` constant renames in `AgentDescriptorValidator`.
