## What Happened

Added a new default method to an interface in `casehub-platform-api` (separate git repo). Switched to `casehub-engine-actor-state` (different repo) and ran `mvn compile`. Got "cannot find symbol" errors for the new method — as if the method didn't exist. The source was correct; the problem was that the local `.m2` cache still had the old JAR from the last install of `casehub-platform-api`.

## Why This Is Non-obvious

In a single multi-module Maven project, `mvn compile` on a downstream module automatically picks up source changes from upstream modules in the same build. In a multi-repo setup, there is no automatic source linking — Maven resolves `SNAPSHOT` dependencies from the local repository (`.m2`), which only updates when `mvn install` is run in the source repo. The error message says "cannot find symbol" and points at the call site, not at the dependency source, making it look like the code is wrong rather than the cache being stale.

## Fix

After modifying any API in a cross-repo dependency:
```bash
# In the modified repo first:
mvn --batch-mode install -pl <module> -am -DskipTests -q

# Then build the consumer:
mvn --batch-mode compile -pl <module> -am -f /path/to/consumer/pom.xml
```

When a consumer can't find a symbol you just added: check the `.m2` JAR date vs the source file date before debugging the call site.

## Constraints

Applies to any multi-repo Maven setup where repos depend on each other as SNAPSHOT dependencies. Single-repo multi-module builds are unaffected.
