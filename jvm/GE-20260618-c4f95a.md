### Symptom

When N concurrent `ClaudeAsyncClient.close()` calls fire simultaneously (e.g. N parallel session timeouts), all subsequent reactive operations on the same Reactor bounded-elastic pool hang indefinitely. The `process reaper` thread appears in thread dumps waiting on subprocess PIDs. The 17th call in a 16-concurrent scenario blocks immediately after the previous 16 complete.

### Context

Observed in Quarkus + Mutiny applications using `ClaudeAgentClient` from casehub-platform. Each concurrent `AgentProvider.invoke()` call that times out triggers `ClaudeAsyncClient.close()` via the cancellation handler. With `maxConcurrentSessions=4` or higher, multiple timeouts can coincide.

### Root Cause

`DefaultClaudeAsyncClient.close()` uses `subscribeOn(Schedulers.boundedElastic())`:

```java
public Mono<Void> close() {
    return Mono.<Void>fromRunnable(() -> {
        if (closed.compareAndSet(false, true)) {
            connected.set(false);
            cleanup();  // calls StreamingTransport.close()
        }
    }).subscribeOn(Schedulers.boundedElastic());
}
```

`StreamingTransport.close()` (lines 1157–1168) is synchronous and blocks:

```java
process.destroy();
try {
    if (!process.waitFor(5, TimeUnit.SECONDS)) {
        process.destroyForcibly();
    }
}
```

Each `close().subscribe()` (fire-and-forget) occupies one bounded-elastic thread for up to 5 seconds. Reactor's default bounded-elastic pool is `10 × availableProcessors` — on a 10-core machine, 100 threads. With N parallel closes, N threads are blocked simultaneously. At high concurrency, or with small pool sizes, this exhausts the pool and starves all other reactive tasks that need bounded-elastic threads.

### Fix

Do not use `ClaudeAsyncClient` (session mode) for single-turn calls. Use a one-shot subprocess owned directly via `ProcessBuilder` where `process.destroyForcibly()` is immediate (no `waitFor`):

```java
Process proc = new ProcessBuilder(command).start();
// On cancellation:
proc.destroyForcibly();  // immediate SIGKILL — no 5-second grace period
```

For the SDK session path (`openSession()`), avoid N parallel closes by ensuring cancellation fires `destroyForcibly()` on the subprocess directly if the graceful close path is too slow.

### Why This Is Non-Obvious

`close()` returns a `Mono<Void>` — the async signature implies it's non-blocking. The blocking call is buried inside `StreamingTransport.close()` which runs on the bounded-elastic scheduler. The fire-and-forget `close().subscribe()` pattern looks correct; the thread exhaustion only surfaces under parallel load.
