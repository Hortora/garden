### Symptom

`QuarkusTestProfile.getConfigOverrides()` sets `quarkus.hibernate-orm.<pu>.schema-management.strategy=none` but Hibernate still runs `drop-and-create` (from the `%test.` application.properties value).

### Context

Named persistence unit (`qhorus`) with `%test.quarkus.hibernate-orm.qhorus.schema-management.strategy=drop-and-create` in test `application.properties`. A test profile tries to override this to `none` so Flyway manages the schema instead. The override appears in SmallRye Config but Hibernate ignores it.

### Root cause

`schema-management.strategy` (formerly `database.generation`) is processed during Quarkus augmentation (build phase). `getConfigOverrides()` applies at runtime — after the augmentation has already decided the schema action. The build-time `%test.` value wins.

### Fix

Do not rely on `getConfigOverrides()` to change `schema-management.strategy`. Set the desired value directly in the `%test.` application.properties (build-time config). If different test classes need different strategies, use Flyway for all tests and control schema creation through migration locations.

### Why this is non-obvious

The Quarkus docs classify `schema-management.strategy` as "overridable at runtime" and `getConfigOverrides()` is the standard mechanism for per-test-class config. The override shows no warning, no error, and appears in config dumps. The only signal is that Hibernate's DDL still runs — which looks like a Hibernate bug, not a config precedence issue.
