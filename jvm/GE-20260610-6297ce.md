## Symptom

Cross-game trust accumulation silently stops after one game. `decisionCount` for any actor's capability is permanently capped at 1. No error, no warning. Trust routing stays in BOOTSTRAP mode indefinitely.

## Context

A CDI `@Observes GameStopped` bean (`LedgerLifecycleAdapter`) cleared the in-memory `LedgerEntryRepository` (entries, attestations, sequence counters) every game stop. A separate `@ObservesAsync GameStopped` bean (`GameOutcomeRecorder`) wrote an outcome record for the just-completed game. `IncrementalTrustUpdateObserver` then read from the ledger to compute trust scores and materialise them.

## What was tried

Checking the trust score pipeline (`IncrementalTrustUpdateObserver`, `PerActorTrustComputer`) — all correct. Checking `OutcomeRecorder.record()` — correctly called. Found the entry in the ledger immediately after recording — it was there. But trust scores were always computed from exactly 1 entry regardless of how many games had been played.

## Root cause

CDI 4.0 delivers synchronous `@Observes` observers **inline** during `Event.fire()`. Asynchronous `@ObservesAsync` observers are scheduled for later delivery on a worker thread. The actual sequence:

```
GameStopped fires (Event.fire())
  → (sync, inline) LedgerLifecycleAdapter.onGameStop(): ledger.clear()
  → fire() returns
  → (async, later) GameOutcomeRecorder.onGameStopped(): record(...)
    → IncrementalTrustUpdateObserver fires on AttestationRecordedEvent
      → reads from ledger → sees exactly 1 entry (the one just written)
      → PerActorTrustComputer.upsert(decisionCount=1) overwrites previous count
```

Every game: ledger cleared → 1 entry written → trust computed from 1 entry → `decisionCount=1` materialised. Previous games' data erased before it can accumulate.

## Fix

Remove the synchronous clearing bean entirely. If the clearing existed for test isolation, move it to `@BeforeEach` in the test classes that need it, scoped by `gameSession.id()`. In-memory ledger entries accumulate across games; persistent (JPA) implementations are unaffected as they never had a clearing bean.

```java
// Before: production CDI bean (wrong — clears before async accumulation)
@ApplicationScoped
public class LedgerLifecycleAdapter {
    void onGameStop(@Observes GameStopped event) {
        if (!memoryLedger.isUnsatisfied()) {
            memoryLedger.get().clear(); // destroys any prior history
        }
    }
}

// After: move to @BeforeEach in tests that need isolation
@BeforeEach
void setUp() {
    ledgerRepo.clear();        // explicit, scoped to this test's session
    trustScoreRepo.clear();    // separate store — also must be cleared
    gameSession.reset();
}
```

## Why this is non-obvious

The CDI synchronous-before-async delivery guarantee is specified but not prominently documented. A developer reading `LedgerLifecycleAdapter` sees a guard (`isUnsatisfied()`), knows it's a no-op on JPA profiles, and reasons it's safe for the in-memory profile "since we clear between games anyway." The interaction with `@ObservesAsync` on the same event is invisible — both observers are in different files, registered independently.

Additionally, the symptom (`decisionCount=1`) looks like a correct "first game" value, not an accumulation failure. It only becomes suspicious after playing multiple games and noticing the count never increments.
