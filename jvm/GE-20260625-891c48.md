### Symptom
`io.quarkus.runtime.configuration.ConfigurationException: Datasource '<default>' is not configured` at Quarkus augmentation time, despite setting `quarkus.datasource.active=false` and `quarkus.hibernate-orm.active=false` in `application.properties`.

### Context
You have `@Entity` classes on the classpath (from a Quarkus extension runtime module) and want to boot without any real datasource — e.g. using `@Alternative @Priority(100)` in-memory stores that beat the JPA stores. Setting `datasource.active=false` seems like the right approach.

### What was tried (didn't work)
1. `quarkus.datasource.active=false` + `quarkus.hibernate-orm.active=false` — both are runtime properties; the `HibernateOrmProcessor#configurationDescriptorBuilding` build step runs before runtime and demands a datasource when it sees entities.
2. `quarkus.hibernate-orm.enabled=false` — disables entity scanning at build time, but JPA store beans still inject `EntityManager`. With no `EntityManager` bean produced, you get `UnsatisfiedResolutionException`.
3. Excluding JPA stores via `quarkus.arc.exclude-types=io.example.repository.jpa.*` — cascades into `UnsatisfiedResolutionException` for any bean that injects the excluded stores (e.g. a `CrossTenantProducer` that injects `CrossTenantWorkItemStore`). The dependency chain is deeper than the direct excludes.

### Root cause
`quarkus.datasource.active` and `quarkus.hibernate-orm.active` are **runtime** properties — they deactivate beans at runtime but do not affect Quarkus augmentation (build-time). The `HibernateOrmProcessor` build step scans for `@Entity` annotations and validates that a matching datasource exists. This happens before any runtime property is evaluated.

### Fix
Provide a dummy H2 datasource that satisfies build-time validation. Disable Flyway so no migrations run. Let Hibernate generate schema from annotations (never used for actual data):

```properties
quarkus.datasource.db-kind=h2
quarkus.datasource.jdbc.url=jdbc:h2:mem:dummy;DB_CLOSE_DELAY=-1
quarkus.hibernate-orm.schema-management.strategy=drop-and-create
quarkus.flyway.enabled=false
```

The `@Alternative @Priority(100)` in-memory stores handle all actual persistence. The H2 datasource exists but carries no data.

### Why this is non-obvious
The property name `quarkus.datasource.active=false` strongly implies "don't use a datasource." A developer would reasonably expect it to work at build time. The distinction between build-time augmentation and runtime configuration is not surfaced in the error message — the exception says "Datasource not configured" without mentioning that `active=false` was set and ignored.

### Why this fix
The dummy datasource approach is the only one that avoids cascading CDI failures. Both `hibernate-orm.enabled=false` and `arc.exclude-types` break the CDI wiring graph in ways that require excluding progressively more beans. The dummy datasource keeps the entire CDI graph valid while letting the alternative stores win at runtime.

### Alternatives considered
- **`hibernate-orm.enabled=false`** — rejected because JPA store beans still inject `EntityManager`, which is absent when Hibernate is disabled. Would require all JPA stores to use `@IfBuildProperty` or similar conditional registration.
- **`arc.exclude-types` on JPA packages** — rejected because the dependency chain cascades. Any service bean that injects a store interface (where the only implementation was the excluded JPA store) fails with `UnsatisfiedResolutionException`.
- **No datasource at all** — not possible while `@Entity` classes are on the classpath and the Hibernate extension is present.
