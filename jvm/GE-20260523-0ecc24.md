## Context
When raw X.509 (public key) or PKCS8 (private key) DER bytes are stored without a
separate algorithm identifier, you need to detect the signing algorithm to verify or
use the key. The bytes themselves encode the OID — but reading it requires either
a known OID table or an ASN.1 parsing library.

## Technique
Iterate `KeyFactory.getInstance(algo).generatePublic(spec)` through a supported-algorithm
list. `KeyFactory` validates the OID in the DER bytes against the expected algorithm and
throws `InvalidKeySpecException` when they don't match. Catch `InvalidKeySpecException |
NoSuchAlgorithmException` and try the next. Once a trial succeeds, call
`pub.getAlgorithm()` to get the exact string for `Signature.getInstance()`.

```java
private static final List<String> SUPPORTED = List.of("Ed25519", "ML-DSA-44", "ML-DSA-65", "ML-DSA-87");

static PublicKey loadPublicKey(byte[] encoded) throws InvalidKeyException {
    X509EncodedKeySpec spec = new X509EncodedKeySpec(encoded);
    for (String algo : SUPPORTED) {
        try {
            return KeyFactory.getInstance(algo).generatePublic(spec);
        } catch (NoSuchAlgorithmException | InvalidKeySpecException ignored) {
            // algorithm not supported or bytes don't match — try next
        }
    }
    throw new InvalidKeyException("No supported algorithm matched: " + SUPPORTED);
}

// After loading:
PublicKey pub = loadPublicKey(storedBytes);
Signature sig = Signature.getInstance(pub.getAlgorithm()); // "Ed25519", "ML-DSA-65", etc.
sig.initVerify(pub);
```

The same pattern works for private keys with `PKCS8EncodedKeySpec` and `generatePrivate`.

## Why This Is Non-Obvious
The obvious approach is to parse the SubjectPublicKeyInfo ASN.1 structure and look up
the OID — which requires either a custom OID table or pulling in BouncyCastle's ASN.1
parser. The trial-load approach exploits the fact that `KeyFactory` already does this OID
check internally and surfaces the mismatch as `InvalidKeySpecException`. The exception is
cheap when it fires at the DER header level (before the key material is parsed).

There is also no mis-detection risk: the OID is validated before any key material is
consumed, so malformed bytes for algorithm A cannot accidentally satisfy the trial for
algorithm B.

## Why This Approach Over Alternatives
- **OID byte-matching**: fragile — OID encodings differ by algorithm variant; must maintain a byte-array lookup table.
- **BouncyCastle SubjectPublicKeyInfo parser**: correct but adds a hard dependency; unnecessary when the JVM already validates OIDs internally.
- **Hardcoding the algorithm**: breaks when the supported set grows (e.g. adding ML-DSA variants for PQC migration).

The trial-load list is forward-compatible: add new algorithm strings as JVM or provider support ships. `NoSuchAlgorithmException` is caught silently for algorithms not yet supported on the current JVM.
