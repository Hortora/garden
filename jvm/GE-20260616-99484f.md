### Context

A method needs to run two independent operations before returning a combined result:
1. A **blocking** operation (e.g. tmux session creation, file IO) — must run on the worker pool
2. A **reactive** operation (e.g. Hibernate Reactive Panache query with `@WithSession`) — must run on the Vert.x event loop with a safe sub-context

Sequential approaches fail: if the blocking step runs first via `runSubscriptionOn(workerPool)`, the reactive step that follows it loses the Vert.x safe sub-context. If the reactive step runs first, you're blocking the event loop.

### Technique

Construct both `Uni`s **before** any thread switch, while still on the event loop. Then use `Uni.combine()` to run them concurrently:

```java
public Uni<Result> provision(Set<String> capabilities, ProvisionContext context) {
    // Build blocking Uni — will switch to worker pool when subscribed
    Uni<Void> setup = Uni.createFrom()
        .<Void>item(() -> { doBlockingWork(capabilities, context); return null; })
        .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());

    // Build reactive Uni — resolver.resolve() is called HERE, on the event loop.
    // @WithSession intercepts via CDI proxy and captures the safe sub-context.
    Uni<Optional<UUID>> causedBy = (resolver != null && context.triggerChannelId() != null)
        ? resolver.resolve(context.triggerChannelId(), context.triggerCorrelationId())
        : Uni.createFrom().item(Optional.empty());

    // Combine: both subscribe concurrently when this chain is subscribed
    return Uni.combine().all().unis(setup, causedBy).asTuple()
        .invoke(tuple -> sideEffect(tuple.getItem2()))
        .map(tuple -> buildResult(tuple.getItem2()))
        .call(result -> asyncFollowUp(context))
        .invoke(result -> syncFollowUp(context));
}
```

Key invariant: `resolver.resolve()` (or any CDI method with `@WithSession`) is called on the event loop at chain construction time. The CDI proxy intercepts there, `@WithSession` fires, and the session is bound to the event loop's safe sub-context. Subsequent reactive Panache operations in that Uni use the bound session regardless of which thread ultimately subscribes to the combined Uni.

### Why this is non-obvious

The standard pattern for mixing blocking and non-blocking work is `runSubscriptionOn(workerPool)` → then do reactive work. This breaks `@WithSession` because the safe sub-context is gone. Most developers would either:
- Try to await the reactive result inside the blocking step (deadlocks the Vert.x event loop)
- Use `.await()` on the reactive call from the worker thread (works but loses the reactive benefit)
- Give up and make the whole method blocking

`Uni.combine()` with pre-construction of both Unis is the idiomatic solution that keeps both operations in their natural execution contexts while running them concurrently. The latency benefit (no sequential wait) is a bonus.

### Why this fix

The CDI proxy intercepts method calls at construction time, not subscription time. `@WithSession` is a CDI interceptor — it fires when the method is called (on the event loop), wraps the returned `Uni` in session scope, and that session scope persists regardless of which thread later subscribes to the `Uni`.
