## Symptom

```java
session.createNativeQuery("SET LOCAL \"casehub.tenancy_id\" = :tid")
       .setParameter("tid", tenancyId)
       .executeUpdate()
```

Fails at runtime: PostgreSQL rejects bind parameters in configuration statements. Hibernate Reactive's Vert.x PG client translates `:tid` to `$1` — PostgreSQL throws `ERROR: syntax error at or near "$1"`. On some drivers, the prepared statement simply fails silently and the variable is never set.

## Context

Arises when implementing PostgreSQL Row Level Security with per-request tenant variables. The natural instinct is to use parameterized queries (preventing SQL injection). Works for `SELECT`/`INSERT`/`UPDATE`/`DELETE` — not for `SET LOCAL`.

## Root cause

`SET LOCAL variable = value` is a PostgreSQL configuration statement, not a data manipulation statement. PostgreSQL's protocol does not support parameter binding in configuration statements. Bind parameters (`$1`, `?`, `:name`) are only valid in the Parse/Bind protocol flow for DML statements.

## Fix

String interpolation with a sanitization guard:

```java
private static String sanitizeTenancyId(String tenancyId) {
    if (tenancyId.contains("'") || tenancyId.contains("\\")) {
        throw new IllegalArgumentException("Invalid tenancyId: " + tenancyId);
    }
    return tenancyId;
}

// In the helper:
String sql = "SET LOCAL \"casehub.tenancy_id\" = '" + sanitizeTenancyId(tenancyId) + "'";
session.createNativeQuery(sql).executeUpdate()
```

`tenancyId` in production comes from a JWT claim (authenticated, controlled) — not user-supplied input. The guard is defence-in-depth.

## Why this is non-obvious

Every other Hibernate Reactive native query accepts `setParameter()`. Nothing in the Hibernate Reactive docs or the PostgreSQL `SET` docs prominently documents this limitation. The error message ("syntax error near $1") points to a driver issue, not a fundamental protocol restriction. Developers typically discover this only after confirming the guard works for similar queries.
