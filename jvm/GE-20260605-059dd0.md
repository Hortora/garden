## Symptom

A senior-analyst or other worker is dispatched twice for the same case. The WARN fires:

```
Capability 'senior-analyst-review' matched 2 workers — only 'senior-analyst-agent' will be tracked for PlanItem completion. Workers [] will fire but their completion events will be silently ignored, leaving their PlanItems RUNNING indefinitely.
```

In H2 test environments, this also triggers `UQ_MERKLE_FRONTIER_SUBJECT_LEVEL` violation (see GE-20260605-4ed3bd).

## Context

casehub-engine YAML case definition with a binding that ORs a prior-context condition and an entity-resolution condition:

```yaml
- name: senior-analyst-required
  on: { contextChange: {} }
  when: >-
    ((.priorEntityContext.knownHighRisk == true) or
     (.entityResolution.entityType == "PEP") or
     (.entityResolution.riskScore > 0.8)) and
    .seniorAnalystReview == null
  capability: senior-analyst-review
```

## Root cause

The binding is re-evaluated on EVERY `contextChange`. Timeline:

1. **Case starts**: initialContext has `priorEntityContext.knownHighRisk = true`
2. **First contextChange** (initial): prior-context condition true → binding fires → worker dispatched (Quartz job A queued)
3. **entity-resolution** writes its result to context → **second contextChange**
4. Binding re-evaluates: prior-context STILL true, `seniorAnalystReview` STILL null (Quartz job A hasn't written its result back yet) → binding fires AGAIN → Quartz job B queued

Both Quartz jobs run concurrently. Each writes a `WorkerDecisionEntry`, causing the Merkle frontier race.

## Fix

Split the merged OR binding into two mutually exclusive bindings with complementary guards:

```yaml
# Fires ONLY before entity resolution — entityResolution == null becomes permanently false
# once entity-resolution writes its result. Exactly-once dispatch guaranteed.
- name: senior-analyst-required-prior-context
  on: { contextChange: {} }
  when: >-
    .priorEntityContext.knownHighRisk == true and
    .entityResolution == null and
    .seniorAnalystReview == null
  capability: senior-analyst-review

# Fires ONLY after entity resolution, for entities NOT already routed by prior context.
# priorEntityContext.knownHighRisk != true prevents double-dispatch.
- name: senior-analyst-required-resolution
  on: { contextChange: {} }
  when: >-
    .entityResolution != null and
    .priorEntityContext.knownHighRisk != true and
    (.entityResolution.entityType == "PEP" or .entityResolution.riskScore > 0.8) and
    .seniorAnalystReview == null
  capability: senior-analyst-review
```

## Why this is non-obvious

The `.seniorAnalystReview == null` guard looks like it prevents re-dispatch — and it does, AFTER the Quartz job writes the result back. But there is a timing window between dispatch and result-write where `seniorAnalystReview` is still null. The next contextChange (from entity-resolution) falls in that window. The guard works only if the Quartz job completes synchronously before the next contextChange, which async Quartz cannot guarantee.
