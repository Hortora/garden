GE-20260718-483ed8: [gotcha, 8/15] Surefire retry after @QuarkusTest failure produces misleading LinkageError/TestInstantiationException on Runs 2-3 — the real failure is always in Run 1 | quarkus, testing, surefire, retry

### Symptom
A `@QuarkusTest` fails with an assertion error on Run 1. Surefire retries (Runs 2 and 3) then fail with completely different errors:
```
TestInstantiationException: Failed to create test instance
Caused by: LinkageError: loader constraint violation...
```

The retry errors look like classloader bugs, leading developers to investigate CDI wiring when the actual problem is a simple assertion failure in Run 1.

### Context
Maven Surefire configured with `rerunFailingTestsCount` > 0 (default in many Quarkus projects). After a `@QuarkusTest` assertion failure, the CDI container is left in a corrupted state. Surefire attempts to retry by recreating the test instance, but the Quarkus test extension cannot recover — the CDI proxy creation fails because the container's internal state is inconsistent.

### Root cause
The Quarkus test extension shares CDI container state across retries within the same JVM. An assertion failure during test execution can leave CDI beans in an inconsistent state (e.g., `@ObservesAsync` event handlers partially fired). When Surefire retries, the test extension tries to create a new test instance from the corrupted container, producing LinkageError or InvocationTargetException.

### Fix
Always diagnose from Run 1's error — ignore Runs 2 and 3 entirely. In the Surefire output:
```
Run 1: MyTest.method:123  ← THIS is the real failure
Expecting value to be true but was false

Run 2: MyTest.method » TestInstantiation  ← IGNORE — CDI pollution
Run 3: MyTest.method » TestInstantiation  ← IGNORE — CDI pollution
```

### Why this is non-obvious
The retry errors look more severe than the original failure — a `LinkageError` sounds like a build or dependency problem, not a test assertion issue. Developers who start investigating from the last error (Run 3) will chase a phantom classloader bug.
