## Symptom

`@QuarkusTest` using `quarkus-langchain4j-jlama` runs for several minutes (model downloads, Quarkus bootstraps, first inferences proceed) then throws mid-inference:

```
java.lang.UnsupportedOperationException: ARM_128
    at com.github.tjake.jlama.tensor.operations.PanamaTensorOperations.batchDotProduct(PanamaTensorOperations.java:125)
    at com.github.tjake.jlama.tensor.operations.TensorOperations.dotProductChunk(TensorOperations.java:76)
    at com.github.tjake.jlama.model.AbstractModel.lambda$sample$4(AbstractModel.java:435)
    at com.github.tjake.jlama.math.VectorMath.lambda$pchunk$1(VectorMath.java:60)
```

The test fails with `IllegalState Judge LLM call failed`.

## Context

Occurs on Apple M-series chips (M1/M2/M3 base — ARM_128 SIMD, not ARM_256). JVM detects ARM_128 as the available Panama Vector species. Quarkus bootstrap and model loading succeed; the error surfaces during actual token generation.

## What Was Tried

Standard Vector API JVM flags (`--enable-native-access=ALL-UNNAMED --add-modules jdk.incubator.vector`) — no effect; the error is in the dispatch logic, not JVM flags.

## Root Cause

`PanamaTensorOperations.batchDotProduct()` dispatches over vector species (`ARM_128`, `ARM_256`, `F_256`, `F_512`, etc). The `ARM_128` case throws `UnsupportedOperationException` — it is explicitly unimplemented. This is despite:
- `GemmerI8Q4_arm.class` existing in the JAR (partial ARM support for Gemmer operations)
- `MachineSpec$Type.ARM_128` being a recognised enum constant

The ARM support in jlama-core 0.8.4 covers some operations (`GemmerI8Q4_arm`) but not `batchDotProduct`.

## Fix

### Solution 1 — Add jlama-native osx-aarch_64 classifier (recommended)
**Approach:** Add the platform-specific native library to the classpath; it contains a C/NEON implementation of `gemm_f32_q4` for ARM64, bypassing `PanamaTensorOperations` entirely.
**Pros:** Works immediately; no model format change needed; `NativeSimdTensorOperations` activates automatically via `JarSupport` when `libjlama.dylib` is on the classpath.
**Cons/trade-offs:** `jlama-native` version must match `jlama-core` version exactly.

```xml
<dependency>
    <groupId>com.github.tjake</groupId>
    <artifactId>jlama-native</artifactId>
    <version>0.8.4</version>
    <classifier>osx-aarch_64</classifier>
    <scope>test</scope>
</dependency>
```

With this on the classpath, Jlama loads `NativeSimdTensorOperations` (stack trace: `NativeSimdTensorOperations.batchDotProduct → NativeSimd.gemm_f32`) instead of `PanamaTensorOperations`. Inference proceeds; throughput is approximately 5 tokens/second for a 3B Q4 model on M4 Max CPU via NEON.

### Solution 2 — Run on x86 hardware
**Approach:** Use a machine with AVX_256 or AVX_512 vector species.
**Pros:** All F32×Q4 paths fully implemented; no classpath changes needed.
**Cons/trade-offs:** Requires different hardware.

## Why This Is Non-Obvious

The presence of `GemmerI8Q4_arm.class` and `MachineSpec$Type.ARM_128` strongly implies full ARM support. The error does not appear during Quarkus bootstrap or model loading — only during token generation, after several minutes of apparently-successful execution. Standard Panama/Vector API JVM flags have no effect. The exception message `ARM_128` looks like a JVM module or SIMD constant, not a code path name.
