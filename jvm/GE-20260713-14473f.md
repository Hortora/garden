### Symptom
did:key resolution returns `Optional.empty()` for secp256k1 DIDs despite
correct key material. No error logged — silent mismatch.

### Context
Building a did:key DID with multicodec code 0xe7 (secp256k1). The code was
written as a single byte `multicodec[0] = (byte) 0xe7` followed immediately
by the compressed public key. The LEB128 varint decoder consumed the first
byte, saw the continuation bit set, read the next byte (first byte of the
public key) as part of the varint, and produced a completely wrong code value.

### What was tried (didn't work)
Single-byte encoding: `byte[] multicodec = new byte[1 + compressed.length];
multicodec[0] = (byte) 0xe7;` — silently decoded to the wrong multicodec code.

### Root cause
LEB128 unsigned varint uses bit 7 as a continuation flag. 0xe7 = 11100111 —
bit 7 is set, so the decoder expects another byte. The actual value is encoded
across two bytes: `[0xe7, 0x01]` (low 7 bits = 1100111 = 103, second byte
contributes 1 << 7 = 128, total = 231 = 0xe7).

### Fix
Any multicodec code ≥ 0x80 must be encoded as a multi-byte LEB128 varint:
```java
// secp256k1 multicodec 0xe7 → LEB128 [0xe7, 0x01]
byte[] multicodec = new byte[2 + compressed.length];
multicodec[0] = (byte) 0xe7;
multicodec[1] = 0x01;
System.arraycopy(compressed, 0, multicodec, 2, compressed.length);
```

Compare with Ed25519 (0xed → [0xed, 0x01]) and P-256 (0x1200 → [0x80, 0x24]).
All three are multi-byte varints because all have codes ≥ 0x80.

### Why this is non-obvious
0xe7 = 231 "fits in a byte" as an unsigned integer. The trap is conflating
byte-sized values with single-byte varints. In LEB128, only codes 0x00–0x7F
are single-byte — the encoding steals one bit per byte for continuation.
