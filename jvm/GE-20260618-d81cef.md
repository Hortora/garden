### Symptom
A JPA UPDATE fires for a field write on a `PanacheEntity` subclass, even though:
- The CDI injection test confirms the in-memory store (`InMemoryReactiveChannelStore`) is selected, not the JPA store
- The entity was created with `new Channel()` — never explicitly `persist()`ed or loaded in the current session
- `quarkus.arc.selected-alternatives` is correctly configured

Error:
```
org.hibernate.QueryParameterException: No argument for named parameter ':tenancyId'
```

### Context
`ReactiveMessageService.dispatch()` wraps the entire dispatch in `Panache.withSession("qhorus", () -> ...)`. Inside that session scope, the in-memory store's `updateLastActivity()` does:

```java
channel.lastActivityAt = Instant.now();
```

`Channel` is a Hibernate-enhanced `PanacheEntity`. The field write is intercepted by Hibernate's dirty tracker within the active session scope and generates a JPA UPDATE at session flush.

### Root cause
Hibernate's bytecode enhancement instruments all `PanacheEntity` subclasses with dirty tracking. Any field write on an enhanced entity within an active `Panache.withSession()` scope is detected and queued for flush — regardless of whether the entity was explicitly persisted or loaded in that session. The CDI injection test proves the right bean is injected, but it does not reveal that Hibernate's session-scoped dirty tracker is intercepting field mutations on enhanced entities.

### Fix
Make the in-memory store's `updateLastActivity()` a no-op. Do not modify enhanced entity fields in-place within a `withSession()` scope:

```java
@Override
public Uni<Void> updateLastActivity(UUID channelId, String tenancyId) {
    // No-op — modifying the Hibernate-enhanced entity in-place triggers
    // a dirty-check flush when called from within Panache.withSession().
    return Uni.createFrom().voidItem();
}
```

`lastActivityAt` is set at channel creation time via `put()`; staleness tracking is irrelevant in test contexts.

### Why this is non-obvious
The CDI direct injection test is a convincing false positive: `@Inject ReactiveChannelStore` returns `InMemoryReactiveChannelStore` ✅. This proves the right bean is injected, but it does not prove that field mutations on Hibernate-enhanced entities are safe within the dispatch session scope. The JPA UPDATE is generated by Hibernate's session-level flush mechanism, not by the in-memory store calling `repo.update()` directly.
