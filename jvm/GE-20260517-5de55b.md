**Update 2026-05-30:** API renamed from `send()` (positional args) to `dispatch(MessageDispatch)` (builder pattern). Behaviour is unchanged. Use `MessageDispatch.builder().channelId(...).type(COMMAND).correlationId(...).target(...).build()`. PLATFORM.md still documents the old `send()` form — see GE-20260529-d3d4b6 for the builder API docs mismatch.

### What it is
`MessageService.dispatch()` automatically calls `commitmentService.open()` when the message type is `COMMAND` (or `QUERY`) and `correlationId` is non-null. No explicit `CommitmentService.open()` call is needed in application code.

### Where it's documented
Not documented in the qhorus deep-dive, README, or DESIGN.md. Only discoverable by reading `MessageService.java` source. The commitment auto-open logic runs inside `dispatch()` after the message is stored.

### Usage pattern
```java
// This single call creates both the Message and the Commitment:
var result = messageService.dispatch(MessageDispatch.builder()
    .channelId(work.id)
    .sender(ORCHESTRATOR)
    .type(MessageType.COMMAND)
    .content(pr.repo() + "#" + pr.prNumber())
    .correlationId(correlationId)   // triggers commitmentService.open() automatically
    .target(agent.capability())
    .actorType(ActorType.SYSTEM)
    .build());
// result.messageId() is the message ID to use in inReplyTo for DONE/DECLINE
// No separate commitmentService.open() call needed.
```

### Why this matters
Application code that manually calls `commitmentService.open()` after `messageService.dispatch()` will create two Commitments with the same correlationId. The `CommitmentStore.findByCorrelationId()` returns the first match — the duplicate is silently orphaned. Use `messageService.dispatch()` as the sole entrypoint for COMMAND messages.
