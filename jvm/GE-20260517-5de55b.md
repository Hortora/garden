### What it is
`MessageService.send()` automatically calls `commitmentService.open()` when the message type is `COMMAND` (or `QUERY`) and `correlationId` is non-null. No explicit `CommitmentService.open()` call is needed in application code.

### Where it's documented
Not documented in the qhorus deep-dive, README, or DESIGN.md. Only discoverable by reading `MessageService.java` source:

```java
// In MessageService.send():
if (message.correlationId != null) {
    switch (message.messageType) {
        case QUERY, COMMAND -> commitmentService.open(
            message.commitmentId != null ? message.commitmentId : UUID.randomUUID(),
            message.correlationId, message.channelId, message.messageType,
            message.sender, message.target, message.deadline);
        case STATUS -> commitmentService.acknowledge(message.correlationId);
        case RESPONSE, DONE -> commitmentService.fulfill(message.correlationId);
        case DECLINE -> commitmentService.decline(message.correlationId);
        case FAILURE -> commitmentService.fail(message.correlationId);
        case HANDOFF -> commitmentService.delegate(message.correlationId, message.target);
        case EVENT -> { /* no commitment effect */ }
    }
}
```

### Usage pattern
```java
// This single call creates both the Message and the Commitment:
messageService.send(
    channel.id, "clinical-service", MessageType.COMMAND, content,
    deviation.id.toString(),   // correlationId — triggers commitmentService.open() automatically
    null, null, site.investigatorId, ActorType.SYSTEM
);
// No separate commitmentService.open() call needed.
// commitmentService.fulfill(deviation.id.toString()) still needed manually for inbound responses
// (see GE-20260517-5879a9 — receiveHumanMessage passes correlationId=null).
```

### Why this matters
Application code that manually calls `commitmentService.open()` after `messageService.send()` will create two Commitments with the same correlationId. The `CommitmentStore.findByCorrelationId()` returns the first match — the duplicate is silently orphaned. Use `messageService.send()` as the sole entrypoint for COMMAND messages.
