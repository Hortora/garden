### Context
A DB TEXT column stores JSON in one format and needs to change to a richer format (e.g. from a flat string array `["approved","rejected"]` to an object array `[{"name":"approved","condition":null}]`). The usual instinct is to write a Flyway migration that re-encodes all existing rows.

### Technique
Use Jackson's `readTree()` to parse the stored value, inspect the first element's node type, and decode accordingly — no Flyway migration needed:

```java
public static List<MyType> decode(final String json) {
    if (json == null || json.isBlank()) return null;
    try {
        final var node = MAPPER.readTree(json);
        if (!node.isArray()) {
            LOG.warnf("Expected JSON array, got %s — data integrity error", node.getNodeType());
            return null;
        }
        final ArrayNode arr = (ArrayNode) node;
        if (arr.isEmpty() || arr.get(0).isObject()) {
            // new format: array of objects
            return MAPPER.convertValue(arr, new TypeReference<List<MyType>>() {});
        } else {
            // old format: array of strings — wrap each element
            return StreamSupport.stream(arr.spliterator(), false)
                .map(n -> new MyType(n.asText(), null))
                .toList();
        }
    } catch (final Exception e) {
        LOG.warnf("Failed to decode JSON: %s", e.getMessage());
        return null;
    }
}
```

Old rows decode via the fallback branch. New writes use the richer format. The column format migrates naturally as rows are updated by the application.

### Why this is non-obvious
Developers encountering a column format change immediately think "Flyway migration." The migration approach requires cross-DB compatible SQL for JSON transformation (PostgreSQL uses `json_array_elements_text`, H2 uses different syntax), adds a migration file that must coordinate with other branches, and may run on millions of rows. Format detection on read avoids all of this — with zero data loss and zero coordination.

### Constraints
- Only works when the old format is a strict semantic subset of the new format — every old row can be correctly mapped to the new type.
- The wrapping logic (old → new) must be deterministic and correct for all old values.
- Requires the new write path to be deployed before old rows can be migrated by natural usage.

### Alternatives considered
- Flyway SQL migration: requires cross-DB compatible JSON transformation SQL; conflicts with other in-flight branches on the same version namespace; blocked during Flyway migration window.
- Two-column approach (keep old column + add new column): avoids migration but introduces redundant data and schema drift.
