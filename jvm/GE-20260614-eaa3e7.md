### Symptom

`System.setProperty("KEY", "value")` in a `@BeforeAll` method has no effect on config values read during `@PostConstruct` of a `@Startup @ApplicationScoped` bean. If a YAML file loaded at `@PostConstruct` contains an unresolved `${KEY}` reference, the Quarkus context aborts before any test method runs — with a startup exception, not a test failure.

### Context

Any `@QuarkusTest` class where:
1. A `@Startup @ApplicationScoped` bean loads config or resources at `@PostConstruct`
2. The test tries to influence that loading via `System.setProperty()` in `@BeforeAll`

### Root cause

`QuarkusTestExtension` implements JUnit 5's `BeforeAllCallback`. Extension callbacks run before `@BeforeAll` methods in the test class — this is a JUnit 5 extension contract guarantee. Quarkus boots the entire application (including all `@Startup @PostConstruct` beans) inside `QuarkusTestExtension.beforeAll()`. By the time `@BeforeAll` executes in the test class, the app is fully initialised.

### Fix

All config that must precede `@PostConstruct` must live in `src/test/resources/application.properties` or a `@QuarkusTestProfile`. YAML files loaded at startup must contain pre-resolved literal values — no `${VAR}` references. Move any interpolation testing to plain JUnit 5 unit tests that invoke static parsing methods directly, bypassing the CDI startup lifecycle entirely.

```java
// Wrong — too late
@BeforeAll
static void setup() {
    System.setProperty("SALESFORCE_URL", "example.com"); // @PostConstruct already ran
}

// Right — resolved at startup
# src/test/resources/application.properties
my.service.url=https://example.com/api  # literal value, no ${VAR}
```

### Why this is non-obvious

`@BeforeAll` feels like the first thing that runs. It is the first thing that runs *in the test class* — but JUnit 5's extension lifecycle precedes that. `QuarkusTestExtension` exploits the `BeforeAllCallback` contract to boot the entire Quarkus application before any test class code executes. The symptom (context abort on unresolved `${VAR}`) gives no hint that timing is the cause.
