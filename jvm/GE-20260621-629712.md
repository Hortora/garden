Seven rules every CloudEvent adapter in the platform must satisfy.
Discovered during casehubio/connectors#20 — two existing adapters (IoT, Qhorus)
had 2 bugs, 1 silent failure path, and 1 pattern inconsistency. All fixed together.

### The pattern

| Rule | Requirement | Failure mode |
|------|-------------|-------------|
| ObjectMapper | Injected — never a private static instance | Static instance isolates serialisation config from app's shared mapper; custom serializers invisible |
| tenancyId | Null-safe — extension omitted when null | SDK serialises null extension as literal `"null"` or throws NPE |
| fireAsync | Always `.exceptionally(ex -> { LOG.warn(...); return null; })` | Unhandled `CompletionStage` swallows downstream observer failures silently |
| Serialisation error | Catch at WARN, fire with `new byte[0]` — never throw | Unchecked exception from `@ObservesAsync` propagates into CDI managed executor and is silently swallowed |
| type derivation | From a stable semantic field on the event record | Instance IDs are mutable; CloudEvent `type` is used for routing rules that must be stable |
| Severity | WARN for degraded paths | SEVERE causes alert fatigue; serialisation failure is degraded, not catastrophic |
| datacontenttype | Set `application/json` when data is ObjectMapper-serialised | Consumers cannot parse the `data` field without knowing its format; omitting it forces guessing |

`tenancyId` always sourced from the event record — never from `CurrentPrincipal`,
which is `@RequestScoped` and inactive in `@ObservesAsync` observers.

### Why this is non-obvious

Rules 3 and 4 interact: `fireAsync()` returns a `CompletionStage` that swallows
failures silently, but a thrown exception from the observer method *also* vanishes
into CDI's managed executor. Most developers add `.exceptionally()` (rule 3) but
don't realise that a throw *before* `fireAsync` is called (rule 4 — serialisation
error) is equally invisible. The error paths are different but the symptom is identical:
the event silently disappears with no log entry.

### Why this fix

The alternative — letting exceptions propagate — appears correct because CDI
"should" handle observer exceptions. But `@ObservesAsync` runs on a managed
executor thread where exceptions propagate into a `CompletionStage` that nobody
awaits. The pattern of catch-log-continue is the only way to make failures visible.

### Fix

Reference implementations:
- `QhorusCloudEventAdapter` (casehub-qhorus)
- `ConnectorCloudEventAdapter` (casehub-connectors)
- `IoTCloudEventAdapter` (casehub-iot)

```java
@ApplicationScoped
public class ExampleCloudEventAdapter {

    private static final Logger LOG = Logger.getLogger(ExampleCloudEventAdapter.class);

    private final Event<CloudEvent> cloudEventBus;
    private final ObjectMapper objectMapper;  // rule 1: injected

    @Inject
    public ExampleCloudEventAdapter(Event<CloudEvent> cloudEventBus, ObjectMapper objectMapper) {
        this.cloudEventBus = cloudEventBus;
        this.objectMapper = objectMapper;
    }

    public void onEvent(@ObservesAsync SomeDomainEvent event) {
        byte[] data;
        try {
            data = objectMapper.writeValueAsBytes(event);
        } catch (JsonProcessingException e) {
            LOG.warnf("Serialisation failed: %s", e.getMessage());  // rule 4 + 6
            data = new byte[0];
        }

        CloudEventBuilder builder = CloudEventBuilder.v1()
                .withId(UUID.randomUUID().toString())
                .withType("io.casehub.domain." + event.semanticType())  // rule 5
                .withSource(URI.create("/source"))
                .withTime(event.occurredAt().atOffset(ZoneOffset.UTC))
                .withDataContentType("application/json")
                .withData(data);

        if (event.tenancyId() != null) {  // rule 2
            builder = builder.withExtension("tenancyid", event.tenancyId());
        }

        cloudEventBus.fireAsync(builder.build())
                .exceptionally(ex -> {  // rule 3 + 6
                    LOG.warnf(ex, "CloudEvent dispatch failed");
                    return null;
                });
    }
}
```

### ⚠️ Rule 3 exception — reactive messaging ack chains

Rule 3 (`.exceptionally()`) **must NOT be applied** when the `CompletionStage` is
chained to a message acknowledgement, as in SmallRye Reactive Messaging `@Incoming`
consumers:

```java
// WRONG — .exceptionally() swallows the exception; message.ack() always runs → silent data loss
return cloudEventBus.fireAsync(ce)
    .exceptionally(ex -> { LOG.warnf(ex, "dispatch failed"); return null; })
    .thenCompose(ignored -> message.ack());

// CORRECT — .whenComplete() preserves the exception; thenCompose skips on failure → SmallRye nacks
return cloudEventBus.fireAsync(ce)
    .whenComplete((v, t) -> { if (t != null) LOG.warnf(t, "dispatch failed"); })
    .thenCompose(ignored -> message.ack());
```

With `.exceptionally()`: exception swallowed → `null` returned → `thenCompose` fires →
`message.ack()` runs → message permanently consumed regardless of dispatch outcome.

With `.whenComplete()`: exception preserved on the stage → `thenCompose` skips its lambda
→ exception surfaces to SmallRye → message nacked → retried or dead-lettered.

Rule 3 applies **only in pure fire-and-forget dispatch** (the `CompletionStage` is not
returned or chained further). CDI `@ObservesAsync` adapters are always fire-and-forget.
SmallRye `@Incoming` consumers return the stage for ack/nack signalling — they are not.

### Alternatives considered

- **Let exceptions propagate from `@ObservesAsync`** — rejected; CDI swallows them silently on the managed executor thread
- **Use `@Observes` (sync) instead of `@ObservesAsync`** — rejected; would block the original event producer's thread
- **Static ObjectMapper** — rejected; isolates serialisation config, breaks when app registers custom serializers
- **SEVERE log level for failures** — rejected; serialisation failure of a single event is degraded, not catastrophic; SEVERE causes alert fatigue
