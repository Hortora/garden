### Symptom

`mvn test` runs 338 tests including multiple `@QuarkusTest` classes — all pass, BUILD SUCCESS. `mvn install` on the same code fails at the `quarkus:build` step with `DeploymentException: Found 34 deployment problems: Unsatisfied dependency for type ...`.

The unsatisfied types are reactive SPI interfaces (`ReactiveCaseInstanceRepository`, `ReactiveEventLogRepository`, `ReactiveCrossTenantCaseInstanceRepository`, `ReactiveCrossTenantEventLogRepository`) introduced by an upstream SNAPSHOT update.

### Context

Multi-module Quarkus app consuming SNAPSHOT dependencies from sibling repos. The upstream engine repo added new reactive SPI interfaces that require `@ApplicationScoped` implementations. The consuming app's `@QuarkusTest` tests start a full Quarkus container with CDI augmentation — but those tests don't exercise the code paths that inject the new reactive SPIs, so the augmentation doesn't flag the missing beans.

### Root cause

`@QuarkusTest` augmentation and `quarkus:build` augmentation use the same `ArcProcessor#validate` step, but `@QuarkusTest` only discovers and validates beans that are actually reachable from the test's injection graph. Beans that would be injected at runtime but aren't referenced by any test class are not validated during test augmentation. `quarkus:build` validates ALL discovered beans regardless of whether they're reachable from a test — it builds the full production application graph.

### Fix

Add `@ApplicationScoped` wrapper classes for every new SPI interface, even if no test exercises them:

```java
@ApplicationScoped
public class DevtownReactiveCaseInstanceRepository
    extends InMemoryReactiveCaseInstanceRepository {}
```

Key insight: check the InMemory base class type hierarchy — `InMemoryReactiveCaseInstanceRepository` implements both `ReactiveCaseInstanceRepository` AND `ReactiveCrossTenantCaseInstanceRepository`. Two wrappers can satisfy four CDI injection points.

### Why this is non-obvious

A developer whose full test suite passes (including `@QuarkusTest` with real CDI container startup) would reasonably conclude the build is sound. The failure only surfaces during `mvn install` / `mvn package` — the production build lifecycle phase. Running `mvn test` exclusively (common in iterative development) masks the problem entirely.
