## Raw ExecutorService drops CDI context — @Transactional silently broken on background threads

### Symptom

`@Transactional` calls on background threads open an unmanaged transaction or fail silently. Test assertions on data that should have been persisted return empty results. No exception is thrown.

### Root cause

`@Transactional` and other CDI interceptors require an active CDI request context. A raw `ExecutorService` (e.g. `Executors.newSingleThreadExecutor()`) spawns threads without CDI context propagation. Any `@Transactional` method called on that thread operates outside CDI and either opens a new unmanaged transaction or silently does nothing.

### Fix

Inject `ManagedExecutor` from MicroProfile Concurrency instead of constructing a raw executor:

```java
// Wrong — CDI context not propagated
ExecutorService executor = Executors.newSingleThreadExecutor();
executor.submit(() -> service.doTransactionalWork()); // @Transactional silently broken

// Correct — CDI context propagated
@Inject ManagedExecutor executor;
executor.submit(() -> service.doTransactionalWork()); // @Transactional works
```

`ManagedExecutor` propagates the Quarkus CDI context to spawned threads automatically.

### Why this is non-obvious

Raw executors work fine for non-CDI code, and most Java developers default to them. The failure is silent — no exception, no warning — and the symptoms (missing data, failed assertions) look like a logic bug rather than a threading context issue.
