### Context
Cloud SDKs like GCP `KeyManagementServiceClient` and Azure `CryptographyClient`/`KeyClient` are concrete classes, not interfaces. Mockito cannot mock final/concrete classes without `mockito-inline` (which has its own issues). WireMock works but requires simulating the full HTTP/gRPC protocol, which is complex for gRPC-based SDKs.

### Technique
Create a thin wrapper interface in the pure Java module that exposes only the operations the signing client needs:

```java
public interface GcpKmsClientWrapper {
    PublicKey getPublicKey(String versionName);
    byte[] asymmetricSign(String versionName, byte[] digest);
    CryptoKeyVersionAlgorithm getCryptoKeyVersionAlgorithm(String versionName);
}
```

Production implementation delegates to the real SDK client. Tests mock the wrapper interface directly with Mockito — no WireMock, no gRPC stubs, no SDK internals.

### Why this is non-obvious
The instinct when testing cloud SDK code is either (a) use the SDK's built-in test utilities (often incomplete), (b) set up WireMock/localstack for full HTTP simulation, or (c) use `@ExtendWith(MockitoExtension.class)` with `mockito-inline`. The wrapper pattern is simpler than all three: one interface, one production impl, standard Mockito mocking. The cost is a thin wrapper class; the benefit is tests that run in milliseconds without any external infrastructure.

### When to use vs alternatives
- **SDK exposes an interface** (AWS `KmsClient`) → mock the interface directly, no wrapper needed
- **SDK is concrete but has a test builder** (`KeyManagementServiceClient.create(settings)` with mock transport) → use the builder if the API is stable
- **SDK is concrete and you need HTTP-level fidelity** (testing retries, auth headers, error codes) → use WireMock
- **SDK is concrete and you need unit-test speed** → use the wrapper pattern
