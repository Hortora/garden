### Symptom

After changing a CBR schema field from `Categorical` to `CategoricalList` (e.g. `eventType`), CBR queries fail with:

```
IllegalArgumentException: CategoricalList field 'eventType' requires StringListVal, got: StringVal
```

But cases stored BEFORE the schema change continue to exist in the store — no error at storage time.

### Context

Happens when evolving a CBR schema to support multi-value fields. The `CbrFeatureValidator` validates features at query time (`retrieveSimilar`) but NOT at storage time (`store`). This means:

1. Schema changes from `FeatureField.categorical("x")` to `FeatureField.categoricalList("x")`
2. Old cases stored with `String` values remain — no migration, no error
3. New queries fail because the validator checks the query features against the new schema

### What was tried (didn't work)

- Changed schema registration only — expected queries to work. Failed: feature builders still produce `String` instead of `List<String>`.
- Looked for `CbrFilter.eq()` to match single values — no such method exists on `CbrFilter`. Available: `contains`, `containsAll`, `containsAny`, `hasMatch`, `notContains`, `notContainsAny`, `containsRange`, `allOf`.

### Root cause

`CbrFeatureValidator` only runs on query-path features (`validateQueryFeatures`), not on stored cases. `FeatureValue.toFeatureMap()` uses `FeatureValue.of(value)` which converts a plain `String` to `StringVal` and a `List<String>` to `StringListVal`. The schema expects `StringListVal` for `CategoricalList` fields, so all feature-building code must produce `List.of(value)` instead of plain `value`.

### Fix

Two-step change — both are required:

1. **Schema registration:** `FeatureField.categorical("eventType")` → `FeatureField.categoricalList("eventType")`

2. **ALL feature builders:** Every `buildFeatures()` and `buildQueryFeatures()` method that sets the field must change from:
   ```java
   features.put("eventType", ae.eventType != null ? ae.eventType : "UNKNOWN");
   ```
   to:
   ```java
   features.put("eventType", List.of(ae.eventType != null ? ae.eventType : "UNKNOWN"));
   ```

Missing step 2 means storage silently succeeds with the wrong type, and queries fail at retrieval time.

### Why this is non-obvious

The error appears at query time, not storage time, so the natural assumption is "my query is wrong" rather than "my stored data has the wrong type." The feature builder code compiles and runs without error — the mismatch is only caught by the runtime validator during retrieval.
