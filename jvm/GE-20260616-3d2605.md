### Symptom
Fault policy mutations are applied for the first failed node in a reconciliation cycle but silently lost for all subsequent failed nodes. No error, no warning, no log. The desired graph appears to have been mutated (first fault is handled) but additional faults in the same cycle produce no graph changes.

### Context
`ReconciliationLoop.TenantLoop.reconcile()` iterates over `TransitionResult.outcomes()` and creates `FaultEvent` objects for each `StepOutcome.Failed`. Each fault is evaluated through `FaultPolicyEngine`, which returns `GraphMutation` objects. The mutations must be applied to the desired graph via `AtomicReference<DesiredStateGraph>.compareAndSet()`.

### What was tried (didn't work)
The original code captured `desired = desiredRef.get()` once at the top of `reconcile()`, then called `desiredRef.compareAndSet(desired, mutated)` inside the per-failure loop. This appeared correct because each iteration built `mutated` from `desired` and applied it atomically.

### Root cause
`compareAndSet(expected, new)` succeeds only if the current value equals `expected`. The first CAS changed `desiredRef` from `desired` to `mutated1`. On the second iteration, `desiredRef` now holds `mutated1`, but the CAS still uses the original `desired` as the expected value. The comparison fails silently — `compareAndSet` returns `false` but the return value was not checked. The second mutation is permanently lost.

### Fix
Accumulate all mutations on a progressively-mutated graph, then apply with a single CAS at the end:

```java
DesiredStateGraph mutated = desired;
for (Map.Entry<NodeId, StepOutcome> entry : result.outcomes().entrySet()) {
    if (entry.getValue() instanceof StepOutcome.Failed failed) {
        FaultEvent faultEvent = new FaultEvent(
            entry.getKey(), faultType, failed.reason());
        List<GraphMutation> mutations = faultPolicyEngine.evaluate(faultEvent, mutated);
        for (GraphMutation mutation : mutations) {
            mutated = mutated.withMutation(mutation);
        }
    }
}
if (mutated != desired) {
    desiredRef.compareAndSet(desired, mutated);
}
```

Each subsequent fault evaluation receives the progressively-mutated graph (not the original), so mutations compose correctly. The single CAS at the end atomically applies all accumulated changes.

### Why this is non-obvious
The original code looked correct — each iteration appeared to apply its mutations atomically. The CAS pattern is the standard way to update an AtomicReference, and the per-iteration structure seems like the natural way to process a list of failures. The bug only manifests when ≥2 nodes fail in the same cycle, which the dungeon example never triggered because its only fault policy (`HeroRaidFaultPolicy`) handles `NODE_DESTROYED` events from `EventSource`, not `PROVISION_FAILED` events from the reconciliation loop. The fault feedback path in `reconcile()` was effectively dead code until the data pipeline example introduced fault policies that respond to `PROVISION_FAILED`.

### Why this fix
Progressive accumulation on a single mutable reference is both simpler and more correct than alternatives like retry-CAS loops or separate mutation queues. The graph is immutable (each `withMutation()` returns a new instance), so the accumulation is safe. The single CAS preserves the atomicity guarantee — concurrent `updateDesired()` calls are handled correctly because the CAS will fail and the mutations will be re-evaluated on the next cycle.

### Alternatives considered
- **Check CAS return value and retry on failure**: Would add complexity and still process each fault against a potentially-stale graph. The progressive approach is simpler.
- **Collect all mutations first, then apply as a batch**: Equivalent outcome but requires two passes over the failure list. Single-pass progressive accumulation is more natural.
