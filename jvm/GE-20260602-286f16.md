## What Happened

Code used `Infrastructure.getDefaultBlockingExecutor()` to offload blocking work onto a worker pool in a Mutiny pipeline. The method name is logical (Mutiny has `runSubscriptionOn` for blocking work) but the method does not exist. The compile error is clear but the fix is non-obvious because no similar `getDefaultBlockingExecutor` pattern appears in Mutiny documentation.

## Why This Is Non-obvious

The Mutiny `Infrastructure` class has `getDefaultExecutor()` — the default worker pool suitable for blocking work when combined with `runSubscriptionOn`. There is no separately named "blocking" executor. The distinction between the IO thread and the worker pool is handled by `runSubscriptionOn` itself, not by a separate executor variant. The name "blocking" is a common mental model but doesn't map to any Mutiny API.

## Fix

```java
// Wrong — does not exist:
.runSubscriptionOn(Infrastructure.getDefaultBlockingExecutor())

// Correct:
.runSubscriptionOn(Infrastructure.getDefaultExecutor())
```

## Invalidation Triggers

Revisit if Mutiny adds a named `getDefaultBlockingExecutor()` alias in a future release.
