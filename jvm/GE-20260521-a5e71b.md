## Symptom

Reactive service operations silently succeed but write to the wrong datasource (or fail with a datasource not found error at runtime), even though `ReactiveMessageService.send()` was already fixed to use `Panache.withTransaction("qhorus", ...)`. Other reactive services (`ReactiveChannelService`, `ReactiveInstanceService`, etc.) still use the bare form and silently route differently.

## Context

Quarkus allows multiple named persistence units. A library (casehub-qhorus) defines a `qhorus` named PU. Consumer apps like Claudony configure `quarkus.datasource.qhorus.*` with no default datasource at all.

`Panache.withTransaction(() -> { ... })` — no name argument — uses the **default** persistence unit, not the one associated with the Panache entity's own named PU. If no default datasource is configured, behaviour is undefined or fails with a confusing error. If a default datasource happens to be configured for a different purpose, data silently goes to the wrong place.

## Root Cause

`Panache.withTransaction(Supplier)` has an overload `Panache.withTransaction(String, Supplier)` where the first argument is the persistence unit name. Without the name, Panache resolves against the default datasource regardless of which named PU the entities were mapped to.

The fix is already documented for `ReactiveMessageService` but must be applied to every `Panache.withTransaction` call across all reactive services consistently.

## Fix

Replace all bare calls with the named form:

```java
// Before — silently uses default PU
return Panache.withTransaction(() -> channelStore.findByName(name)...);

// After — explicit named PU
return Panache.withTransaction("qhorus", () -> channelStore.findByName(name)...);
```

Audit command to find bare calls:
```bash
grep -rn "Panache\.withTransaction(" src/main/java/ --include="*.java" | grep -v '"qhorus"'
```

## Why This Is Non-Obvious

`Panache.withTransaction()` feels like it "just works" in isolation — and it does when the app has a default datasource. The failure only manifests in consumer apps that configure only named datasources, which is a less common deployment pattern. No compile error, no warning, no runtime exception in many configurations — it silently uses a different datasource.
