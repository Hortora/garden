## Symptom

After moving Java types from `runtime/` to `api/` in a multi-module Maven project:
- `mvn test-compile -pl api,runtime` → **BUILD SUCCESS** (no errors)
- `mvn test -pl runtime` → `NoClassDefFoundError: io/example/api/SomeType`

The test-compile pass found the new source files in `api/` correctly. The test-runtime pass looked in the installed `~/.m2` jar for `api`, which is stale (still the pre-move version without the new types).

## Context

Maven's reactor resolves source directly during `compile` phases — it compiles `api` from source even when running `-pl runtime`. But at **test runtime** (surefire execution), the classpath picks up the **installed jar** from `~/.m2/repository/`, not the reactor source. If the installed jar pre-dates the type move, it's missing the new classes.

This is especially confusing because:
1. The compilation success implies everything is in order
2. The error looks like a missing dependency rather than a stale jar
3. The test class using the type was compiled fine — only runtime resolution fails

The specific trigger: types were moved from `runtime` to `api` in the same session, and only `mvn compile` (not `mvn install`) was run on the `api` module after the move.

## Fix

Run `mvn install -pl api` (or `mvn install` from the project root) before running tests scoped to a child module:

```bash
# After any structural change to api/ (moving or adding types):
mvn install -pl api -DskipTests

# Then tests against runtime pick up the fresh installed jar
mvn test -pl runtime
```

Alternatively, run the full build from root to avoid the split:
```bash
mvn test  # compiles and tests all modules; no stale jar issue
```

## Why This Is Non-Obvious

`test-compile` succeeds because the compiler reads source directly from the reactor. The implicit mental model is "if it compiled, the classpath is correct". But surefire uses a different classpath resolution: installed jars from `~/.m2`, not reactor source. The two phases use different classpath strategies, and the gap between them is invisible until the runtime error appears.

## Variant — Compile-time "cannot find symbol" after rebase adds new API types

Same root cause, different symptom. After `git rebase` onto a branch that added new types to the `api/` module (e.g. a new record field or nested class), running `mvn test -pl runtime` fails at **compile time**:

```
error: cannot find symbol
  symbol:   class ConnectorBinding
  location: class io.casehub.qhorus.api.channel.ChannelDetail
```

The source in `api/` has the new type, but `runtime`'s compiler is using the installed `api` jar (pre-rebase) which doesn't have it yet. Fix is identical — `mvn install -pl api` before `mvn test -pl runtime`.

The distinction: the original symptom is a **runtime** `NoClassDefFoundError` (test compiled fine but class missing at execution); this variant is a **compile** error (the test never compiles because the installed jar is behind the source). Both share the same root: `mvn test -pl runtime` without `-am` always uses the installed sibling jar.
