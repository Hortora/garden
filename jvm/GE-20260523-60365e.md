### Symptom
All @QuarkusTest classes pass (e.g. 37/37 tests) but the Maven build fails ~20 seconds later with:
`Failed to execute goal quarkus:build: Unsatisfied dependency for type X and qualifiers [@Default]`

### Context
Multi-module Quarkus project. A module produces a Quarkus application artifact (has `<goal>build</goal>` in the quarkus-maven-plugin config). A CDI injection point (e.g. `@Inject PreferenceProvider`) is satisfied by a mock implementation added as `<scope>test</scope>`.

### Root cause
The `quarkus:build` goal runs a full CDI validation against the **production** classpath. Test-scoped dependencies are excluded from the production classpath. The mock implementation is invisible to the production build, so Quarkus fails validation — after all tests have already passed using the test classpath.

### Fix
Change `<scope>test</scope>` → `<scope>runtime</scope>` for the mock implementation in application modules that run `quarkus:build`. Library/extension modules (no `build` goal) can keep `test` scope.

```xml
<!-- Application modules with quarkus:build -->
<dependency>
  <groupId>com.example</groupId>
  <artifactId>my-mock-impl</artifactId>
  <scope>runtime</scope>  <!-- not test -->
</dependency>
```

### Why this is non-obvious
Maven phases run in order: `test` completes before `package` (which triggers `quarkus:build`). The tests use the test classpath and pass fully. There is no indication during the test phase that the production build will fail. The error appears in a separate Maven goal output well after Surefire reports success, making it easy to assume the tests were the only CDI validation gate.
