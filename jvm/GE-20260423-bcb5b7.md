## Symptom

Quarkus startup fails with:

```
AmbiguousResolutionException: Ambiguous dependencies for type
io.quarkiverse.work.api.WorkerSelectionStrategy and qualifiers [@Default]
  - CLASS bean [...LeastLoadedStrategy]
  - CLASS bean [...ClaimFirstStrategy]
```

## Context

Occurs when `quarkus-work-core` is on the classpath and any bean declares
`@Inject WorkerSelectionStrategy strategy`. Both `LeastLoadedStrategy` and
`ClaimFirstStrategy` are annotated `@ApplicationScoped` in quarkus-work-core,
so CDI cannot resolve the interface unambiguously.

## Root cause

Both concrete strategy implementations in quarkus-work-core are CDI beans.
Injecting the shared interface produces an ambiguity at deployment time.

## Fix

Inject the concrete type directly:

```java
// Instead of:
@Inject WorkerSelectionStrategy selectionStrategy;

// Use:
@Inject LeastLoadedStrategy selectionStrategy;
```

Also applies to test constructors — mock the concrete class:
```java
strategy = mock(LeastLoadedStrategy.class); // not mock(WorkerSelectionStrategy.class)
```

## Why this is non-obvious

The natural injection point is the interface. Nothing in the API documentation
signals that both implementations are registered as CDI beans, making the
interface ambiguous. The error only surfaces at Quarkus deployment time,
not at compile time.
