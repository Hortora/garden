## Reactive @Tool calling resolveChannel() blocks Vert.x I/O thread

### Context

`ReactiveQhorusMcpTools` in casehub-qhorus returns `Uni<T>` from `@Tool` methods.
A new tool `setChannelTypeConstraints` was implemented without `@Blocking` because
the method body returned a `Uni<ChannelDetail>` — it looked reactive. The method
called `resolveChannel(channel)` from the base class `QhorusMcpToolsBase`, which
internally calls the **blocking** `ChannelService.findById()` or `findByName()`
(JPA / Panache statics). quarkus-mcp-server dispatches `@Tool` methods on the
Vert.x I/O thread by default. Blocking JPA on the I/O thread produces no
exception — just a Vert.x blocked-thread warning and latency stall under load.

### Symptom

No compile error, no test failure, no exception at runtime. In development: a
Vert.x blocked-thread log warning. Under load: request latency degrades as the
I/O thread stalls waiting for JPA.

### Root cause

`resolveChannel()` is defined in `QhorusMcpToolsBase` and appears to be a simple
routing utility — its signature `UUID resolveChannel(String channel)` gives no
indication that it does JPA lookups. The blocking nature is invisible from the
tool method body. A developer implementing a reactive tool that needs UUID-or-name
resolution correctly returns `Uni<T>` but forgets `@Blocking` because the tool
method itself has no blocking code — only the inherited helper does.

### Fix

Add `@Blocking` to every reactive `@Tool` method in `ReactiveQhorusMcpTools` that
calls `resolveChannel()`:

```java
@Tool(name = "set_channel_type_constraints", ...)
@Blocking                           // ← required: resolveChannel() makes blocking JPA calls
public Uni<ChannelDetail> setChannelTypeConstraints(
        @ToolArg(name = "channel") String channel,
        /* other args */) {
    UUID channelId = resolveChannel(channel);  // blocking — JPA inside base class
    return channelService.setTypeConstraints(channelId, ...)
            .flatMap(ch -> messageStore.countByChannel(ch.id)
                    .map(count -> toChannelDetail(ch, count.longValue())));
}
```

**Reference pattern:** `project_channel` in the same class already uses `@Blocking`
for the same reason (calls `resolveChannel()` before `projectAndRender()`).

### Why this is non-obvious

The method body returns `Uni<T>` and chains reactive operations — it reads as
non-blocking. The blocking call is hidden two layers deep: the tool calls a base
class helper, which calls a blocking CDI service, which calls JPA. There's no
compiler warning, no Quarkus build-time error, and no runtime exception — only
a Vert.x thread-blocked warning that's easy to miss in development logs.

### Why this fix

`@Blocking` on the tool method moves execution to a worker thread before
`resolveChannel()` is invoked — the JPA call runs on the worker thread where
blocking is safe. The `Uni<T>` return still composes correctly on the reactive
pipeline after `@Blocking` completes.

An alternative would be to refactor `resolveChannel()` to use
`ReactiveChannelStore` and return `Uni<UUID>`, then chain it into the reactive
pipeline without `@Blocking`. That's the structurally correct fix but requires
changing the base class and all 39 tool signatures. `@Blocking` is the practical
fix for the pattern as it currently stands.
