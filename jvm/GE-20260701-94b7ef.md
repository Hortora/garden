### Symptom
Physical device nodes trigger `FaultType.DEPROVISION_FAILED` on every reconciliation cycle when removed from the desired-state YAML. The fault policy evaluates and may mutate the graph, but the node reappears as failed next cycle — an infinite loop.

### Context
The desiredstate runtime's `SimpleTransitionExecutor` has separate `executeProvision()` and `executeDeprovision()` methods. A `requiresHuman` gate was added to `executeProvision()` — it delegates to `HumanNodeHandler` and never calls the provisioner. But `executeDeprovision()` has no such gate (comment: "no requiresHuman for deprovision"). Physical device nodes with `requiresHuman=true` are intercepted on provision but reach the provisioner on deprovision.

### What was tried (didn't work)
The IoT provisioner returned `DeprovisionResult.Failed("physical devices cannot be auto-deprovisioned")` — semantically wrong. "Failed" triggers fault feedback, which runs every cycle, creating the infinite loop.

### Root cause
Lifecycle gate asymmetry: `executeProvision()` checks `node.requiresHuman()` and short-circuits; `executeDeprovision()` does not. The `HumanNodeHandler` interface only has `onProvision()`, no `onDeprovision()` — the runtime was designed without deprovision gating.

### Fix
Change the provisioner to return `DeprovisionResult.Success()` for physical devices — deprovisioning means "stop managing," not "physically remove." Filed desiredstate#54 for the proper runtime fix (add `requiresHuman` check to `executeDeprovision` + `onDeprovision()` to `HumanNodeHandler`).

### Why this is non-obvious
The provision side was already fixed (IoTGoalCompiler sets `requiresHuman=true`, executor intercepts). The fix appeared complete — tests passed, physical devices were correctly skipped during provisioning. The deprovision path was only discovered during adversarial first-principles analysis of all code paths. The "no requiresHuman for deprovision" comment in the executor made the asymmetry look intentional rather than incomplete.

### Alternatives considered
- Gate deprovision with `requiresHuman` at the executor level — correct long-term fix but requires cross-repo change (HumanNodeHandler interface extension). Filed as desiredstate#54.
- Return `Skipped` instead of `Success` — considered but `Skipped` implies "try again later," which is wrong for "stop managing."
