## Symptom

`mvn clean test -pl runtime` reports multiple `Unsatisfied dependency` errors:

```
[1] Unsatisfied dependency for type io.casehub.work.api.ClaimSlaPolicy and qualifiers [@Default]
[2] Unsatisfied dependency for type io.casehub.work.core.strategy.WorkBroker and qualifiers [@Default]
```

All missing beans are `@ApplicationScoped` classes in the `core` module — not `@DefaultBean`, not `@Alternative`. They should be discovered automatically via Jandex.

## Context

Multi-module Quarkus extension project. `runtime` depends on `core` at compile scope. `core` has its own Jandex index. After `mvn clean -pl runtime`, Maven resolves `core` from `.m2/repository` (the installed jar), not from the reactor.

The installed `core` jar was built before recent commits added the `@ApplicationScoped` beans (e.g. `WorkBroker`, `ClaimFirstStrategy`). The jar's Jandex index doesn't list them. Quarkus augmentation reads the Jandex index from the jar → beans not found → `Unsatisfied dependency`.

## What was tried (didn't work)

- Checking for `@DefaultBean` annotations — none missing; the beans are plain `@ApplicationScoped`
- Checking `quarkus.arc.selected-alternatives` — irrelevant, beans aren't alternatives
- Running the specific test class with `-Dtest=` — same failure (augmentation is per-JVM, not per-test)

## Root cause

The `casehub-work-core` jar in `.m2/repository` was installed by a previous build that predates the `@ApplicationScoped` beans. Its `META-INF/jandex.idx` doesn't include them. Quarkus CDI resolution uses the Jandex index from the jar, not source scanning. When the index is stale, beans that exist in source are invisible to CDI.

## Fix

Reinstall the upstream module before running downstream tests:

```bash
mvn install -pl core -DskipTests
mvn test -pl runtime -Dtest=SomeTest
```

Or use the reactor to include the dependency:

```bash
mvn test -pl runtime -am -Dtest=SomeTest
```

The `-am` (also-make) flag builds `core` from source before testing `runtime`, ensuring the Jandex index is current.

## Why this is non-obvious

The error message says "Unsatisfied dependency for type X" — the natural diagnosis is a missing bean declaration, a missing `@DefaultBean`, or a CDI scope issue. Nothing in the error points to a stale Jandex index in a dependency jar. The beans exist in source, they compile, and `test-compile` succeeds — the failure is purely at Quarkus augmentation time when it reads the installed jar's Jandex index.

### Alternatives considered

- Adding `quarkus.arc.selected-alternatives` — wrong diagnosis; the beans aren't alternatives
- Adding `@DefaultBean` no-op implementations — masks the real problem and adds unnecessary code
