### Technique

When a Quarkus/Panache application needs per-HTTP-request tenant isolation but has
system-actor contexts (schedulers, `@ObservesAsync` CDI observers, internal service
enrichment) that legitimately require cross-tenant access, add a static
`findByIdForTenant(UUID id, CurrentPrincipal principal)` method to each entity class.

```java
// In each domain entity (e.g. ClinicalTrial, TrialSite, etc.)
public static ClinicalTrial findByIdForTenant(UUID id, CurrentPrincipal principal) {
    if (principal.isCrossTenantAdmin()) return findById(id);
    return find("id = ?1 AND tenantId = ?2", id, principal.tenancyId()).firstResult();
}
```

REST resources and REST-called services inject `CurrentPrincipal` and call
`findByIdForTenant`. System actors (schedulers, CDI observers) call plain `findById` —
they are never in a request context and use cross-tenant access by design.

### Why This Is Non-Obvious

The natural approaches — Hibernate `@TenantId` discriminator and `@Filter` — both fail
in this scenario:

- **Hibernate `@TenantId`**: the `CurrentTenantIdentifierResolver` is called per session,
  not per query. There is no per-query bypass for cross-tenant access. `DeviationExpirer`
  (a scheduler) queries all tenants; CDI async observers run outside request scope with a
  system principal. Both break under `@TenantId`.

- **Hibernate `@Filter`**: `session.enableFilter(...)` does not affect `EntityManager.find()`
  / Panache `findById()` — their SQL is pre-compiled at session-factory startup
  (`SingleIdLoadPlan.jdbcSelect`). Only HQL/JPQL/Criteria queries compiled at runtime
  respect `@Filter`. Converting all `findById` calls to HQL `find("id = ?1", id).firstResult()`
  to work around this moves the burden to call sites anyway, while hiding the tenant
  predicate in infrastructure.

The entity-helper pattern makes isolation **explicit at the call site** and leaves system
contexts naturally unrestricted. Reviewers can grep for `findById` in resource classes to
audit missed call sites. Internal system paths that use plain `findById` are correct by
identity, not convention.

### When to Use

- Panache Active Record (not Repository pattern — repositories can inject `CurrentPrincipal`)
- Mix of REST-context and system-context callers on the same entity class
- `tenant_id` column exists on all entity tables (written correctly at creation time)
- `CurrentPrincipal` has `tenancyId()` and `isCrossTenantAdmin()` methods
