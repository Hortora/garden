## Symptom

Calling `findByActorIdAndTypeAndKey(actorId, ScoreType.CAPABILITY, null)` returns GLOBAL rows instead of empty. No error — the query silently executes against the wrong named query.

## Context

In a discriminator-based JPA entity (`actor_trust_score`) keyed by `(actor_id, score_type, scope_key)`, a repository method routes `scopeKey == null` to a GLOBAL-specific named query (`findGlobalByActorId`) regardless of the `scoreType` parameter. If a caller passes `CAPABILITY` with `null` scopeKey, the null-routing branch fires and returns GLOBAL rows — not an empty result as expected.

## Root Cause

The null-routing branch only checks `scopeKey == null`; it does not validate that `scoreType == GLOBAL` before routing. The named query `findGlobalByActorId` uses `s.scoreType = :scoreType` but also `s.scopeKey IS NULL`, so the scoreType filter adds no protection when the wrong scoreType is passed.

## Fix

Add a guard before routing:

```java
if (scopeKey == null) {
    if (scoreType != ScoreType.GLOBAL) {
        throw new IllegalArgumentException(
            "scopeKey must not be null for score type " + scoreType
            + " — null scopeKey is only valid for GLOBAL scores");
    }
    // route to findGlobalByActorId
}
```

## Why This Is Non-Obvious

The symptom looks like "no CAPABILITY rows exist for this actor" — which is a valid state — not "I queried the wrong rows." The query succeeds and returns non-empty results; the caller sees GLOBAL rows and has no reason to suspect the wrong named query fired.
