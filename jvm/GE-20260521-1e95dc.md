### Symptom

Flaky test results when using `Awaitility` to wait for an `@ObservesAsync` CDI event to be
processed. Occasionally the capture list appears empty despite the observer having run, or
entries from a previous test contaminate the current one.

Under Quarkus's default serial test execution the window is narrow and failures are rare.
Under parallel test execution or higher thread-pool concurrency the race is consistent.

### Context

The common pattern for testing `@ObservesAsync` CDI events in `@QuarkusTest` is a `@Mock`
connector or observer bean with a static `List` field that captures processed items:

```java
@Mock
public class TestSlackConnector implements Connector {
    public static final List<ConnectorMessage> sent = new ArrayList<>();  // ← data race
    public static boolean shouldThrow = false;                            // ← visibility hazard

    @Override
    public void send(ConnectorMessage message) {
        if (shouldThrow) throw new RuntimeException("failure");
        sent.add(message);
    }
}
```

The test thread reads `sent.isEmpty()` inside an `Awaitility.await()` loop. The `@ObservesAsync`
observer runs on a background `ManagedExecutorService` thread and calls `send()`, which writes
to `sent`. There is no memory barrier between the threads — both the list write and the boolean
read are unsynchronised.

### Root cause

`@ObservesAsync` dispatches to a container-managed thread pool, not the test thread. A plain
`ArrayList` is not thread-safe: concurrent reads and writes can produce `ConcurrentModificationException`,
stale visibility (the test thread sees an empty list after the observer has added an element),
or torn reads. A plain `boolean` field has no visibility guarantee — the test thread may read
a stale `false` even after the executor thread wrote `true`.

### Fix

Use thread-safe types for all static capture fields in `@Mock` beans observed by async events:

```java
@Mock
public class TestSlackConnector implements Connector {
    public static final List<ConnectorMessage> sent = new CopyOnWriteArrayList<>();  // ✅ thread-safe
    public static volatile boolean shouldThrow = false;                              // ✅ visible across threads

    @Override
    public void send(ConnectorMessage message) {
        if (shouldThrow) throw new RuntimeException("failure");
        sent.add(message);
    }
}
```

Clear in `@BeforeEach`:
```java
@BeforeEach
void setUp() {
    TestSlackConnector.sent.clear();
    TestSlackConnector.shouldThrow = false;
}
```

### Why this is non-obvious

Developers are aware of thread safety in production code but rarely apply it to test capture
fields, especially when the observer pattern is hidden behind CDI events. The `@Mock` annotation
suggests the bean is a test fixture — it is easy to overlook that CDI still dispatches it on a
real background thread. Under Quarkus's default serial test model failures are rare enough
that many codebases ship with this race latent indefinitely.
