## AgentProviderChatModel uses langchain4j close-timeout (PT30S), not agent.claude.default-timeout — silent truncation

### Symptom
Claude CLI subprocess appears to time out silently — the `QueryExpandingCaseRetriever` decorator catches a `TimeoutException` and falls back to the original query. Benchmark results are identical to baseline despite the decorator being active in the CDI graph. No error logged by `AgentProviderChatModel` — only the decorator's catch block emits a WARNING.

### Context
The call chain is: `LlmQueryExpander` → `ChatModel.chat()` → `AgentProviderChatModel.doChat()` → `agentProvider.invoke()` → `ClaudeAgentProvider` → Claude CLI subprocess. Two separate timeout configurations exist:
- `casehub.platform.agent.claude.default-timeout` (default PT5M) — used by `ClaudeAgentClient` for the subprocess wall-clock timeout
- `casehub.platform.agent.langchain4j.close-timeout` (default PT30S) — used by `AgentProviderChatModel.doChat()` when awaiting the Mutiny stream

### What was tried (didn't work)
Setting `casehub.platform.agent.claude.default-timeout=PT3M` — queries still timed out at exactly 30 seconds. The Claude agent timeout was never the bottleneck.

### Root cause
`AgentProviderChatModel.doChat()` calls `.await().atMost(properties.closeTimeout())` where `properties` is `AgentLangchain4jProperties`, not `ClaudeAgentProperties`. The LangChain4j bridge has its own timeout that fires before the Claude agent timeout. The two config prefixes are different packages with no cross-reference in documentation or logging.

### Fix
Set the correct timeout:
```properties
casehub.platform.agent.langchain4j.close-timeout=PT3M
```

Or bypass the ChatModel bridge entirely by using `AgentProvider.openSession()` directly — the session approach avoids both timeouts and provides prompt caching benefits.

### Why this is non-obvious
Two timeout configs (`agent.claude.default-timeout` and `agent.langchain4j.close-timeout`) control the same call chain at different layers. Setting the wrong one has zero effect — no error, no warning, no log indicating which timeout fired. The `TimeoutException` stack trace shows Mutiny internals, not which config governed the timeout.
