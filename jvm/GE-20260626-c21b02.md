### Symptom
Agent drift checking silently stops working after adding `@DefaultBean` to `AgentDriftChecker` — no error, no warning. The bean is never injected.

### Context
`DeploymentActualStateAdapter` injects `Instance<NodeDriftChecker>` and builds a `Map<String, NodeDriftChecker>` keyed by `nodeType()`. Four implementations exist: `AgentDriftChecker`, `ChannelDriftChecker`, `CaseTypeDriftChecker`, `TrustPolicyDriftChecker`. Each serves a different `nodeType()` but all implement `NodeDriftChecker`.

The goal was to make `AgentDriftChecker` replaceable by a richer implementation via classpath presence — the standard `@DefaultBean` → `@Alternative @Priority(1)` pattern used successfully elsewhere (e.g. `NoOpAgentStateStore` → `InMemoryAgentStateStore`).

### Root cause
Quarkus `@DefaultBean` suppression operates on **bean types** (interface types), not concrete classes. When `AgentDriftChecker` is `@DefaultBean` and `ChannelDriftChecker` (also implementing `NodeDriftChecker`) is `@ApplicationScoped`, ArC sees a non-default bean with the same interface type and suppresses the `@DefaultBean` — regardless of the fact that they serve completely different runtime roles via `nodeType()`.

The `@DefaultBean` pattern is designed for **single-injection-point replacement**: one interface, one default, one override. It is incompatible with `Instance<T>` multi-bean collection where implementations are distinguished by a runtime discriminator.

### What was tried (didn't work)
1. `@DefaultBean` on `AgentDriftChecker` — suppressed by `ChannelDriftChecker` existing
2. `@Alternative @Priority(1)` on the replacement — both beans appear in `Instance<T>` iteration, `HashMap.put()` non-deterministic winner for the same `nodeType()` key

### Fix
Do not use `@DefaultBean` or `@Alternative` for multi-implementation SPI patterns collected via `Instance<T>`. Instead, either:
- Enrich the existing bean directly (no displacement needed)
- Add priority-aware resolution in the adapter: `checkers.merge(nodeType, checker, (existing, incoming) -> getPriority(incoming) > getPriority(existing) ? incoming : existing)`

### Why this is non-obvious
The `@DefaultBean` pattern works flawlessly for single-injection patterns throughout the codebase (6+ SPIs use it). The mental model — "default vanishes when a better one arrives" — is correct for that context. The failure mode here is that *any* `NodeDriftChecker` implementation (not just an agent-specific one) triggers suppression. No error, no warning, no log. The bean simply disappears from the `Instance<T>` iteration.

### Why this fix
Enriching the existing bean avoids all CDI displacement complexity. The comparison logic (pure Java utility) lives in the type-owning module; the SPI implementation stays in the consumer module unchanged. No CDI annotations change, no new beans, no Instance<T> interaction to reason about.

### Alternatives considered
- Priority-aware adapter resolution — works but adds complexity to the adapter for a problem that only exists because of the displacement pattern
- Bridge module with `@Alternative @Priority(1)` — both beans appear in Instance iteration; HashMap collision is non-deterministic
- Composition approach (AgentNodeSpec wrapping AgentDescriptor) — blocked by tenancyId being required in AgentDescriptor's compact constructor
