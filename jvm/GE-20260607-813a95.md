### Symptom

`ConcurrentHashMap.computeIfAbsent()` with a mapping function that returns `null` for missing keys appears to work (returns null, stores nothing) but causes two problems: (1) the behavior is implementation-specific in OpenJDK, not guaranteed by the `ConcurrentHashMap` Javadoc, and (2) no cache entry is stored, so every subsequent call for the same key re-executes the mapping function — including any expensive computation or DB query it contains.

### Context

Building a per-actor computation cache in `ComputedTrustScoreSource` where some actors have no EVENT entries. The mapping function returns `null` to mean "this actor has no data." Every call for that actor re-ran the database query because `computeIfAbsent` didn't store the null result.

### Root cause

`HashMap.computeIfAbsent` explicitly documents: "If the function returns null, no mapping is recorded." `ConcurrentHashMap.computeIfAbsent` does not make this guarantee in its Javadoc — OpenJDK handles it the same way, but the contract is unspecified. More importantly, "no mapping recorded" means the key is re-evaluated on every call, defeating the cache purpose.

### Fix

Use a static sentinel instance instead of null:

```java
private static final ComputedScores EMPTY_SENTINEL =
        new ComputedScores(Map.of(), Map.of(), Map.of(),
                new ActorScore(0, 0, 0, 0, 0, 0, 0));

private ComputedScores computeIfAbsent(String key) {
    ComputedScores result = cache.computeIfAbsent(key, this::computeFresh);
    return result == EMPTY_SENTINEL ? null : result;
}

private ComputedScores computeFresh(String key) {
    List<LedgerEntry> decisions = repo.findEventsByActorId(key);
    if (decisions.isEmpty()) {
        return EMPTY_SENTINEL;  // stored in map, prevents re-query
    }
    return calculator.computeAll(decisions, ...);
}
```

### Why this is non-obvious

`HashMap` and `ConcurrentHashMap` have different documentation for null-return behavior. A developer familiar with `HashMap.computeIfAbsent` (which explicitly documents null handling) would assume `ConcurrentHashMap` works the same way — and it does in OpenJDK, but the contract is unspecified. The repeated-query consequence is silent — no error, no warning, just unnecessary load.
