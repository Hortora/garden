## Symptom

`mvn compile -pl api,runtime` crashes with `NoSuchFileException: api/target/classes` during the `quarkus:generate-code` goal on `runtime/`. No compilation has run yet, so the error looks like a missing source tree — but `api/` has a valid `pom.xml`.

## Context

Greenfield Quarkus multi-module project. `api/` is a pure-Java module with no Java sources yet (just a `pom.xml`). `runtime/` depends on `api/` and has the Quarkus Maven plugin configured.

## Root Cause

`quarkus:generate-code` runs before `compiler:compile` in the Maven lifecycle. It scans `api/target/classes` at startup to detect whether `api/` contains any Quarkus extensions. If `api/` has no sources, `target/classes` is never created by the compiler and the scan crashes with `NoSuchFileException`.

## Fix

Two-phase build:

```bash
# Phase 1: install api/ (creates the jar even with empty sources)
mvn install -pl api --batch-mode

# Phase 2: now compile runtime/ normally
mvn compile -pl runtime --batch-mode
```

This resolves itself permanently once `api/` has real Java source files — `target/classes` gets created by the compiler and Quarkus finds it cleanly.

## Why This Is Non-Obvious

The error message points at a missing directory, not at the Quarkus lifecycle ordering. A developer would expect `mvn compile` on an empty module to be a no-op, not to crash a dependent module's code generation phase. The fix is also non-obvious: you wouldn't expect that installing an empty module would unblock compilation of another module.
