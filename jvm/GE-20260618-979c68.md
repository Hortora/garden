### Symptom

Quarkus startup fails with `ConfigValidationException` ‚Äî SmallRye Config reports an unknown configuration key under a prefix owned by a `@ConfigMapping` interface, even though the property is correctly consumed by `@IfBuildProperty` on a CDI bean.

### Context

An application has a `@ConfigMapping(prefix = "casehub.drafthouse")` interface for runtime configuration. A separate CDI bean uses `@IfBuildProperty(name = "casehub.drafthouse.persistence.enabled", stringValue = "true")` to gate activation. The property is added to `application.properties` but not declared in the `@ConfigMapping` interface.

### Root cause

SmallRye Config strict validation scans all properties under a `@ConfigMapping` prefix and rejects any key not declared in the mapping interface. `@IfBuildProperty` reads the property at Quarkus augmentation time via a different path ‚Äî it does not register the key with SmallRye Config's validation. The two mechanisms are independent: `@ConfigMapping` owns the prefix for validation, `@IfBuildProperty` reads raw config values at build time.

### Fix

Declare the property in the `@ConfigMapping` interface, even though only `@IfBuildProperty` consumes it at build time:

```java
@ConfigMapping(prefix = "casehub.drafthouse")
public interface DraftHouseConfig {

    // ... existing config methods ...

    Persistence persistence();

    interface Persistence {
        @WithDefault("false")
        boolean enabled();
    }
}
```

The `enabled()` method is never called at runtime ‚Äî its purpose is to satisfy SmallRye's strict validation. The actual bean gating happens via `@IfBuildProperty` at augmentation time.

### Why this is non-obvious

The two mechanisms appear unrelated: `@IfBuildProperty` is a Quarkus ARC build-time annotation, and `@ConfigMapping` is a SmallRye Config runtime concern. A developer adding `@IfBuildProperty` with a property under an existing `@ConfigMapping` prefix has no reason to expect that the mapping interface needs updating ‚Äî the property is consumed at build time, not runtime. The error message names the unknown key but does not suggest that the fix is to declare it in a `@ConfigMapping` interface.
