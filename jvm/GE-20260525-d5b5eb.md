## Symptom

A `contextChange.filter` binding in a YAML case definition never fires, even when
the context visually shows the expected values. No error is logged. The WorkItem or
capability dispatch simply never happens.

Minimal example that silently fails:

```yaml
filter: "[.myMap // {} | to_entries | select(.value == true) | length] >= 2"
```

`myMap` is `{"siteA": true, "siteB": true}` — you expect this to evaluate to `2 >= 2 = true`. It evaluates to `false`. No error.

## Context

Any JQ expression that calls `to_entries` and pipes the result to `select` to filter
individual entries. Appears in casehub-engine YAML binding `on.contextChange.filter`
fields and `inputMapping` / `outputMapping` expressions.

## Root Cause

`to_entries` returns an **array** of `{key, value}` objects. Piping an array to
`select(expr)` tests the **entire array as a single value** — not each element.

`select(expr)` is a pass-through filter: if `expr` evaluates to truthy for its input,
the input passes through; otherwise it's discarded. For an array, `.value == true` is
evaluated with `.` being the array itself, not its elements. An array has no `.value`
field, so `.value == true` is `null == true`, which is `false`. The entire array is
discarded. `[...] | length` returns 0.

## Fix

Use `to_entries[]` (with `[]`) to iterate the array before `select`:

```
# WRONG — select tests the array as a whole
[.myMap // {} | to_entries | select(.value == true) | length] >= 2

# CORRECT — to_entries[] iterates each element; select filters each
[.myMap // {} | to_entries[] | select(.value == true)] | length >= 2
```

Note the bracket placement also changes: `[...] | length` (wrapping the generator,
not the whole expression) is the idiomatic way to collect and count.

## Why This Is Non-Obvious

JQ tutorials commonly show `to_entries` followed by `select` as a pattern for
filtering object entries — but those tutorials use `to_entries[]` (with `[]`) while
the text often omits the distinction. The expression `to_entries | select(.value > 0)`
looks syntactically equivalent to `to_entries[] | select(.value > 0)` until you
understand that `[]` is an iterator operator. Nothing in the JQ output indicates the
`select` received an array and discarded it — the expression simply produces no output
and the pipeline continues with an empty result.

### Why `to_entries[]` and not `map(select(...))`

Both work. `[to_entries[] | select(.value == true)]` is the generator-collector style
idiomatic to JQ. `to_entries | map(select(.value == true))` is the array-transform
style. Both produce identical results. The generator style is more common in JQ docs
and the casehub-engine YAML DSL follows it.
