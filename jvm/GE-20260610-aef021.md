## Symptom

After migrating a constant list from `List<String>` to `List<DispositionAxis>`, streaming
filter expressions like `c.primaryAxis().equals(axis)` always return false — no compile error,
no runtime exception. In a diagnostic pipeline, every result collapses to the default
(e.g., `INSUFFICIENT_DATA`) with no indication anything is wrong.

## Context

During a type migration of `VariantPair.primaryAxis` from `String` to `DispositionAxis` and
`AXES: List<String>` to `List<DispositionAxis>`, any filter that compared the migrated field
against an un-migrated variable via `Object.equals()` became silently broken.

In `PersonalityPreservationReport.computeDiagnoses()`:
```java
// AXES migrated to List<DispositionAxis>, axis is DispositionAxis
// c.primaryAxis() also DispositionAxis — BUT equals() called on Object base class
contrasts.stream()
    .filter(c -> c.primaryAxis().equals(axis) && ...)
    .mapToInt(PairContrastResult::effectSize)
    .average().orElse(-1.0);
```
The filter always evaluated to `-1.0` (no match), making every attribution `INSUFFICIENT_DATA`.

## Root Cause

`Object.equals(Object)` accepts any type at compile time. When `c.primaryAxis()` is `DispositionAxis`
and `axis` is `String` (before the `AXES` constant was migrated), the comparison compiles but the
JVM's `DispositionAxis.equals()` implementation (standard enum equality) never returns true for a
`String`. No `ClassCastException` is thrown — Java's `equals()` contract requires returning false
for incompatible types, not throwing. Generics provide no protection because `equals()` accepts `Object`.

## Fix

Use `==` for enum equality in filter expressions — it is the idiomatic Java form and is
immune to cross-type confusion:

```java
// Wrong — compiles against Object.equals; always false for enum vs String
.filter(c -> c.primaryAxis().equals(axis))

// Correct — enum identity comparison; type-safe, never silently false
.filter(c -> c.primaryAxis() == axis)
```

Alternatively, ensure all AXES constants are migrated together with the field type —
the partial migration state is the trigger.

## Why This Is Non-Obvious

`equals()` is the canonical Java equality idiom for reference types. The call compiles without
warning. No exception is thrown at runtime. The filter simply matches nothing — which looks
identical to "no data available" in an `orElse(-1.0)` pipeline. Diagnosis requires noticing that
an entire diagnostic category is missing from results, not that an assertion failed.
