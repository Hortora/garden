### Symptom

`NullPointerException` thrown inside `caseFile.get(key, SomeType.class)` when the key
is present (`caseFile.contains(key)` returns `true`). The NPE message is unhelpful —
it points into `Optional.of()` inside the CaseFile implementation, not at the call site
that wrote the null value. No indication which key is responsible.

### Context

Occurs in `InMemoryCaseFile` (from `casehub-persistence-memory`) when a key has been
written with a `null` value. The trap: `InMemoryCaseFile.put(key, null)` **always writes**
an `ItemEntry(null, version)` regardless of key presence — unlike `CaseContextImpl.set(null)`,
which is a no-op on absent keys (GE-20260512-b0eea3). The two implementations have
opposite null semantics.

### What was tried (didn't work)

Assuming `put(key, null)` would be a no-op in `InMemoryCaseFile` as it is in
`CaseContextImpl`. Assuming `contains()` returning `true` guarantees a non-null value.

### Root cause

`InMemoryCaseFile.put()` unconditionally creates `new ItemEntry(value, version)`:

```java
AtomicReference<ItemEntry> ref = store.computeIfAbsent(key, k -> new AtomicReference());
ref.getAndSet(new ItemEntry(value, writeVersion.incrementAndGet()));
```

`InMemoryCaseFile.contains()` checks if the `ItemEntry` wrapper object is non-null —
NOT whether the value inside it is non-null:

```java
AtomicReference<ItemEntry> ref = store.get(key);
return ref != null && ref.get() != null;   // ref.get() = ItemEntry, not e.value()
```

`InMemoryCaseFile.get()` then calls `Optional.of()` on the value inside the ItemEntry:

```java
ItemEntry e = ref.get();                       // ItemEntry(null, version) — non-null
return Optional.of(type.cast(e.value()));      // Optional.of(null) → NPE
```

`Optional.of(null)` throws `NullPointerException`. It should be `Optional.ofNullable()`.

### Fix

**Immediate — protocol enforcement:** Never write `null` to a CaseFile key. Null values
are semantically wrong for any CaseFile key and the protocol should forbid them explicitly.

**Structural — casehub fix:** Change `Optional.of(type.cast(e.value()))` to
`Optional.ofNullable(type.cast(e.value()))` in `InMemoryCaseFile.get()`. Filed as
casehub-quarkmind#175.

**Caller guard (effective once casehub is fixed):**

```java
// Before casehub fix: NPE fires inside get() before this line is reached.
// After fix: orElseThrow() fires with a meaningful ISE instead.
Point2d threat = caseFile.get(NEAREST_THREAT, Point2d.class)
    .orElseThrow(() -> new IllegalStateException(
        "NEAREST_THREAT gate passed but value is null — protocol violation"));
```

### Why this is non-obvious

`CaseContextImpl` (used by `MapCaseFile`) silently no-ops null writes — `set(absent-key, null)` is a no-op because `Objects.equals(null, null)` is true and the write guard prevents
the insert. `InMemoryCaseFile.put()` has no such guard — it always inserts an `ItemEntry`.
Developers familiar with `MapCaseFile` semantics assume both behave the same. Additionally,
`contains() = true` strongly implies a non-null value to any Java developer, but in
`InMemoryCaseFile` a null-valued `ItemEntry` satisfies the `contains()` check.
