### Symptom

After changing a shared module's dependency from `<scope>compile</scope>` (default) to `<scope>test</scope>`, 5+ sibling modules in the reactor fail with `package does not exist` and `cannot find symbol` errors for classes from the now-invisible transitive dependency.

### Context

Multi-module Maven reactor where module A (e.g. `casehub-qhorus-testing`) declares module B (e.g. `casehub-qhorus-persistence-memory`) as a compile dependency. Sibling modules C, D, E depend on A and transitively get B's classes. When A narrows B to test scope, the transitive path is severed — C, D, E can no longer see B's classes.

In the qhorus case, `testing/pom.xml` had `persistence-memory` at compile scope. Neither exported class in `testing` used persistence-memory — only an internal test (`CommitmentServiceTest` in `src/test/`) needed it. Changing to test scope broke `connector-backend`, `slack-channel`, `examples/type-system`, `examples/normative-layout`, and `examples/agent-communication` — all of which imported `InMemory*Store` classes or referenced them in `quarkus.arc.selected-alternatives`.

### What was tried (didn't work)

Running `mvn test -pl testing` — passes, because the dependency is still available at test scope within the module itself. The breakage is only visible when the full reactor builds (`mvn clean install` from root).

### Root cause

Maven's transitive dependency resolution only propagates compile-scope dependencies. When module A narrows B from compile to test, B disappears from A's transitive closure. Any sibling module that depended on A to provide B now has an unsatisfied compile-time dependency.

### Fix

Add the narrowed dependency as a direct test-scope dependency to each affected sibling module:

```xml
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-qhorus-persistence-memory</artifactId>
  <version>${project.version}</version>
  <scope>test</scope>
</dependency>
```

The compiler surfaces every breakage — run `mvn clean install` from the reactor root after making the scope change to identify all affected modules in one pass.

### Why this is non-obvious

The scope change appears safe in isolation: the shared module's own tests pass because it retains the dependency at test scope. The blast radius across sibling modules is invisible until the full reactor build runs. In a large reactor with profile-gated modules (e.g. `examples/agent-communication` behind `-Pwith-llm-examples`), some breakage may not surface until CI runs the full profile matrix.
