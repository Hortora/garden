## Convention

When an enum represents output formats for a system that targets multiple LLM providers, name the members after output structure rather than providers.

**Structure-named (preferred):**
```java
enum RenderFormat {
    MARKDOWN,   // rich markdown — Claude, markdown-capable models
    PROSE,      // flowing paragraphs — OpenAI, Gemini, Grok, Qwen, Mistral, Llama, ...
    A2A_CARD    // JSON — machine-readable agent identity card
}
```

**Provider-named (common mistake):**
```java
enum RenderFormat {
    CLAUDE_MD, OPENAI_SYSTEM, GEMINI, A2A_CARD  // grows with every new provider
}
```

## Why

Provider-named values require a new enum member per LLM provider even when those providers produce structurally identical output. `OPENAI_SYSTEM` and `GEMINI` differ by one space in resource citation formatting — not enough to justify separate enum members. When Grok, Qwen, or Mistral targets arrive, provider-named enums add `GROK`, `QWEN`, `MISTRAL` — all producing identical dense prose.

Provider-specific differences (rubric strictness, output constraints) belong in the evaluator prompt or rendering method, not in the format type. The format type describes the output shape; the caller decides which shape to request.

## Known alternative

Provider-named enums — common when the system initially supports only one or two providers and the names feel intuitive. The naming problem only becomes apparent when a third provider arrives and the enum clearly duplicates an existing value.

## Sub-labels for specialisation

If provider-specific structural differences genuinely exist (e.g. XML-formatted prompts for a fine-tuned model), sub-label values (`PROSE_OPENAI`, `PROSE_GEMINI`) can be added later. Start with structure-based top-level names; add sub-labels only when concrete structural differences warrant it.
