### Symptom
After adding new methods to a runtime-module SPI interface, a code review reveals an
identical-looking interface in the api module that is missing the new methods. No compile
error, no test failure, no warning. Contributors following the api module interface as the
"source of truth" would implement incomplete contracts.

### Context
A multi-module Maven project (api + runtime) had SPI interfaces defined in both modules:
- `api/src/main/java/.../repository/LedgerEntryRepository.java`
- `runtime/src/main/java/.../repository/LedgerEntryRepository.java`

The runtime version was the live one (injected by CDI, implemented by JPA). The api version
was vestigial — never injected, never implemented, never tested. But it looked authoritative.

### What was tried
Normal development: added 3 methods to `runtime.repository.LedgerEntryRepository` without
checking for api-module duplicates. The api version only surfaced during final code review.

### Root cause
The api-module interfaces were created early in the project and then superseded by the
runtime module without being deleted. Nothing enforced their removal or kept them in sync.

### Fix

```bash
# Check for usages of the suspect interface before assuming it is live
grep -r "io.your.api.repository.YourRepository" --include="*.java" .
# or use IDE find references on the api-module interface
```

If zero usages: delete the package. If usages exist: add the new methods to both interfaces.

### Why this is non-obvious
The duplicated package looks intentional — it's in the "api" module, which typically holds
the public contract. The runtime module implementing the same interface name looks like a
standard Quarkus pattern (api = public contract, runtime = implementation). Only usage
analysis reveals which one is actually live.

### Constraints
Applies to multi-module Maven/Gradle projects where api and runtime modules both contain
repository or SPI interfaces with the same logical purpose.

### Alternatives considered
- Keep both in sync manually (rejected: requires discipline with no enforcement)
- Move all SPI interfaces to api only (valid option; requires runtime to depend on api, which it typically does)

### Invalidation triggers
Revisit if the project intentionally restructures to use api-module interfaces as the
consumer-facing contract implemented by runtime.
