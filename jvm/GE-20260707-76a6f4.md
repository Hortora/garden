### Symptom
Entity field set to PENDING in a `@Transactional @ObservesAsync` observer stays at NONE when read by the test or caller. No error, no exception — the field change silently disappears.

### Context
Multiple `@ObservesAsync` CDI observers fire concurrently for the same domain event. Each observer updates a different status field on the same JPA entity (`AdverseEvent`) in its own `@Transactional` method:
- Observer A sets `escalationStatus = REQUESTED`
- Observer B sets `regulatorySubmissionStatus = PENDING`
- Observer C sets `susarOversightStatus = REQUESTED`

All three read the entity, modify one field, and commit. The transactions are independent — no shared locks.

### What was tried (didn't work)
- Increasing Awaitility timeouts (assumed the observer hadn't fired yet)
- Clearing engine in-memory state between test runs (assumed cross-test contamination)
- Checking for `@InjectMock` leaks from prior test classes (assumed mock interference)
- Adding poll delays before assertions (assumed async delivery lag)

All of these addressed symptoms, not the root cause.

### Root cause
Hibernate ORM's default UPDATE strategy writes ALL columns in the UPDATE statement, not just the dirty ones. When three concurrent observers each:
1. Read the entity (all fields at their defaults)
2. Modify one field
3. Commit (UPDATE sets ALL columns)

The last observer to commit overwrites the earlier observers' changes. If Observer A commits first (setting `escalationStatus = REQUESTED`), then Observer C commits (it read the entity before A's commit), C's UPDATE writes `escalationStatus = NONE` (the value C read), silently reverting A's change.

This is a classic **lost update** problem, but it doesn't trigger optimistic locking because there's no `@Version` field, and it doesn't cause any error — the UPDATE succeeds, it just writes stale values for the columns the observer didn't touch.

### Fix
Add `@org.hibernate.annotations.DynamicUpdate` to the entity class:

```java
import org.hibernate.annotations.DynamicUpdate;

@Entity
@Table(name = "adverse_event")
@DynamicUpdate
public class AdverseEvent extends PanacheEntityBase {
    // ...
}
```

With `@DynamicUpdate`, Hibernate only includes dirty columns in the UPDATE statement. Observer A's UPDATE sets only `escalation_status`, Observer B's sets only `regulatory_submission_status`, Observer C's sets only `susar_oversight_status`. No overwrites.

### Why this is non-obvious
The symptom — "status stays at NONE" — looks like the observer never fired or the case start failed. Nothing in the logs indicates a write conflict. Hibernate's full-row UPDATE is the default and rarely causes problems in single-writer scenarios. The concurrent `@ObservesAsync` pattern is a legitimate CDI pattern that silently breaks JPA's default update strategy. The fix (`@DynamicUpdate`) is a single annotation, but finding the root cause required tracing through async event delivery, engine state management, and mock leak hypotheses before arriving at the JPA layer.

### Why this fix
`@DynamicUpdate` is preferred over `@Version` (optimistic locking) because the observers update different columns — they don't conflict semantically. Optimistic locking would force one observer to fail and retry, adding complexity for no benefit. The observers are independent writers to independent columns — they should succeed concurrently, and `@DynamicUpdate` enables exactly that.

### Alternatives considered
- **`@Version` optimistic locking** — would cause `OptimisticLockException` on the second/third observer, requiring retry logic. Rejected: the updates are to independent columns, so there's no real conflict to detect.
- **Serializing the observers** (making them synchronous) — would eliminate concurrency but defeat the purpose of `@ObservesAsync`. Rejected: the observers are intentionally concurrent for performance.
- **Merging all three observers into one** — would eliminate the multi-writer problem but violate separation of concerns. Rejected: each observer has distinct domain logic (escalation, regulatory submission, SUSAR oversight).
