## What it is

When jsonschema2pojo generates a Java model class from a JSON Schema, array/list fields
are initialised as `new ArrayList<>()` in the generated code — not as `null`. This means
that when Jackson deserialises a YAML/JSON document that **omits** a list field entirely,
the generated getter returns an empty `ArrayList`, not `null`.

A `!= null` guard in your mapping code will NOT protect you — an empty list is not null.

## Symptom

Mapping code like this:

```java
if (schema.getCandidateGroups() != null) {
    builder.candidateGroups(new LinkedHashSet<>(schema.getCandidateGroups()));
}
```

Works when `candidateGroups` is present in the YAML, but also fires when `candidateGroups`
is absent — because the getter returns `[]`, not `null`. The downstream object ends up with
an empty `LinkedHashSet` where `null` was expected, causing assertion failures like:

```
expected: null
but was: []
```

## Root cause

jsonschema2pojo initialises list fields eagerly in the generated constructor:

```java
// Generated code — simplified
@JsonProperty("candidateGroups")
private List<String> candidateGroups = new ArrayList<>();
```

Jackson populates this field when the key is present in the document. When the key is
absent, Jackson leaves the field at its initialised value — `new ArrayList<>()`.

## Fix

Guard against both null AND empty:

```java
if (schema.getCandidateGroups() != null && !schema.getCandidateGroups().isEmpty()) {
    builder.candidateGroups(new LinkedHashSet<>(schema.getCandidateGroups()));
}
```

This treats both "absent from YAML" and "`candidateGroups: []`" identically — neither
populates the downstream object. This is usually the correct semantic: an explicitly empty
list in the schema is indistinguishable from an absent list after Jackson deserialises it.

## Why this is non-obvious

The default expectation for an optional field missing from the document is a `null` getter.
jsonschema2pojo's eager initialisation is an implementation detail not documented in the
mapping guidance. The symptom only surfaces when your business logic distinguishes between
"field not provided" (null) and "field provided but empty" — and the `!= null` guard alone
looks correct but silently fails.

### Alternatives considered

- Re-configure jsonschema2pojo to use lazy initialisation — possible via `initializeCollections: false` in config, but changes the behaviour project-wide and may break other consumers of the generated model.
- Use `@JsonSetter` on the generated field to set null on empty list — requires a custom annotation, not compatible with generated code without post-processing.
- The `!= null && !isEmpty()` guard is the minimal, contained fix.
