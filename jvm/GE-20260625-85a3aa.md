### Symptom
`Uni.combine().all().unis(dynamicList).with(combinator)` throws
`IllegalArgumentException("The Uni set is empty")` when `dynamicList` is empty.
The blocking equivalent — iterating an empty list and calling a merge function on
the empty results — silently returns an empty collection with no error.

### Context
Occurs when a `QueryExpander` (or any dynamic source) produces zero items, and
downstream code fans out one `Uni<T>` per item then combines them. The blocking
path loops over zero items and produces an empty result set (valid, if useless).
The reactive path crashes before subscription.

### Root cause
`UniZip.unis(Iterable)` at line 323 explicitly validates the iterable is non-empty:
```java
int i = 0;
for (Uni<?> uni : unis) {
    if (uni == null) throw new IllegalArgumentException("...");
    i++;
}
if (i == 0) {
    throw new IllegalArgumentException("The Uni set is empty");
}
```
This is a design choice — Mutiny considers an empty combination meaningless.
The blocking equivalent has no such guard because `for (T item : emptyList)` is
a no-op, not an error.

### Fix
Guard before entering the reactive combination:
```java
.map(expanded -> expanded.isEmpty() ? List.of(fallback) : expanded)
.chain(expanded -> {
    if (expanded.size() == 1) {
        return delegate.retrieve(expanded.get(0), ...);
    }
    List<Uni<List<T>>> unis = expanded.stream()
        .map(eq -> delegate.retrieve(eq, ...))
        .toList();
    return Uni.combine().all().unis(unis).with(combinator);
})
```
The `.map()` stage replaces empty with a fallback BEFORE the `.chain()` that
calls `Uni.combine()`. Apply to both paths (blocking loop + reactive combine)
for consistent behaviour.

### Why this is non-obvious
The blocking and reactive paths look structurally identical — "loop over N items,
collect results." A developer writing the blocking version first (which works fine
with N=0) will naturally assume the reactive version handles it the same way. The
crash only surfaces at runtime when N happens to be zero, which may be a rare
config edge case (e.g. `hypotheticalCount=0`).

### Alternatives considered
- Wrapping `Uni.combine()` in a try/catch — works but hides the asymmetry; the
  guard-before-combine is cleaner and makes intent explicit.
