### Symptom
A dependency changed from `<scope>test</scope>` to no `<scope>` tag (intended compile scope) in the child POM, but `mvn dependency:list` still shows it as `test`. The JAR is not on the dev/production classpath.

### Context
Multi-module Maven project where the parent POM's `<dependencyManagement>` declares the artifact with `<scope>test</scope>`. Child POM previously had `<scope>test</scope>`, which was removed to make it compile-scope.

### What was tried (didn't work)
1. Removed `<scope>test</scope>` from child POM — expected default (compile) scope. Result: still test scope.
2. Verified the child POM had no scope tag — correct syntax, still resolved as test.

### Root cause
Maven's `dependencyManagement` in the parent POM acts as a default for ALL properties of a dependency, including scope. When the child POM omits `<scope>`, Maven inherits the scope from `dependencyManagement`. Removing `<scope>test</scope>` from the child doesn't reset to compile — it defers to the parent's `<scope>test</scope>`.

### Fix
Explicitly set `<scope>compile</scope>` in the child POM to override the parent's `dependencyManagement`:

```xml
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-engine-persistence-memory</artifactId>
  <scope>compile</scope>  <!-- explicit override of parent's test scope -->
</dependency>
```

### Why this is non-obvious
The natural mental model is "no scope tag = compile scope (the default)." This is true when there's no `dependencyManagement` for the artifact. When the parent declares a scope, omitting `<scope>` in the child inherits the parent's value rather than resetting to the default. The resolution is silent — `mvn dependency:list` shows the effective scope but there's no warning that the parent is overriding the child's intent.
