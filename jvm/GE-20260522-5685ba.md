## Symptom

Quarkus startup fails with an unexpected exception. The stack trace originates in a CDI observer, propagates through `Event.fire()`, exits the `@Observes StartupEvent` handler, and Quarkus marks startup as failed. The application never becomes ready.

## Context

A startup hook needs to initialise multiple channels and notify external backends via a CDI event:

```java
void onStart(@Observes StartupEvent ev) {
    channelService.listAll().forEach(ch ->
        initChannel(ch.id, new ChannelRef(ch.id, ch.name)));  // initChannel fires CDI event
}
```

If any CDI observer on `ChannelInitialisedEvent` throws, the exception propagates synchronously through `Event.fire()` inside `initChannel()`, through `forEach()`, and out of `onStart()`. Quarkus treats an exception from a StartupEvent handler as a startup failure.

## Root cause

CDI synchronous event dispatch (`Event.fire()`) is call-stack synchronous. The spec requires that exceptions from synchronous observers propagate to the caller of `fire()`. This is correct and intentional CDI behaviour, but it makes `Event.fire()` semantically more like a method call chain than a notification broadcast.

When `fire()` is called from inside a StartupEvent handler, any observer failure flows all the way back to the startup machinery.

## Fix

Wrap each `initChannel()` call (or each `fire()` call) individually so a broken observer doesn't abort the remaining initialisation:

```java
void onStart(@Observes StartupEvent ev) {
    for (Channel ch : channelService.listAll()) {
        try {
            initChannel(ch.id, new ChannelRef(ch.id, ch.name));
        } catch (Exception ex) {
            LOG.errorf(ex, "Failed to initialise gateway for channel %s on startup", ch.id);
        }
    }
}
```

Alternatively, use `Event.fireAsync()` which returns a `CompletionStage` and decouples observer failures from the calling thread entirely — but loses synchronous completion guarantees (backends may not be registered before HTTP requests arrive).

## Why this is non-obvious

`Event.fire()` feels like a broadcast notification — callers typically expect fire-and-forget semantics. The fact that a remote observer's exception propagates synchronously back to the caller is technically specified in the CDI spec but counterintuitive in practice.

The pattern is especially dangerous in startup hooks because developers test with well-behaved observers and only discover the propagation when a new observer is added that can throw.
