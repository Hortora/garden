## Symptom

`em.createNativeQuery("INSERT INTO table (...) VALUES (...) ON CONFLICT (col) DO UPDATE SET ... RETURNING ...").getSingleResult()` throws a generic "Syntax error in SQL statement" on H2 2.4.240 in `MODE=PostgreSQL`.

## Context

Building a sequence allocator table for per-subject monotonic counters. PostgreSQL's `INSERT ON CONFLICT DO UPDATE RETURNING` is a single atomic statement — no retry, no race. H2 in `MODE=PostgreSQL` was expected to handle this syntax since it's the most common PostgreSQL-specific DML extension.

## Root Cause

H2's `MODE=PostgreSQL` provides DDL compatibility (data types, constraints) but does NOT support PostgreSQL's DML extensions including `ON CONFLICT DO UPDATE` and `RETURNING`. The error message gives no hint about which clause is unsupported — it's a generic syntax error positioned at `ON CONFLICT`.

## Fix

Use SQL-standard `MERGE INTO ... USING ... WHEN MATCHED THEN UPDATE SET ... WHEN NOT MATCHED THEN INSERT ...` instead. This works on both H2 2.4+ and PostgreSQL 15+:

```sql
MERGE INTO ledger_subject_sequence AS t
USING (SELECT CAST(?1 AS UUID) AS sid) AS s
ON t.subject_id = s.sid
WHEN MATCHED THEN UPDATE SET next_seq = t.next_seq + 1
WHEN NOT MATCHED THEN INSERT (subject_id, next_seq) VALUES (s.sid, 2)
```

Follow with a separate `SELECT` to read the allocated value (MERGE does not support RETURNING on either engine).

## Why This Is Non-Obvious

`MODE=PostgreSQL` suggests full PostgreSQL syntax compatibility. The H2 documentation does not list which PostgreSQL DML extensions are and aren't supported in this mode. Developers reasonably expect `ON CONFLICT` to work — it's the single most common PostgreSQL-specific syntax used in application code.
