## Symptom

`em.createNativeQuery("INSERT INTO table (...) VALUES (...) ON CONFLICT (col) DO UPDATE SET ... RETURNING ...").getSingleResult()` throws a generic "Syntax error in SQL statement" on H2 2.4.240 in `MODE=PostgreSQL`.

## Context

Building a sequence allocator table for per-subject monotonic counters. PostgreSQL's `INSERT ON CONFLICT DO UPDATE RETURNING` is a single atomic statement — no retry, no race. H2 in `MODE=PostgreSQL` was expected to handle this syntax since it's the most common PostgreSQL-specific DML extension.

## Root Cause

H2's `MODE=PostgreSQL` provides DDL compatibility (data types, constraints) but does NOT support PostgreSQL's DML extensions including `ON CONFLICT DO UPDATE` and `RETURNING`. The error message gives no hint about which clause is unsupported — it's a generic syntax error positioned at `ON CONFLICT`.

**Key distinction** (verified 2026-06-18 against H2 2.4.240): `ON CONFLICT DO NOTHING` with *no conflict-target column* IS accepted — H2 MODE=PG supports the no-target `DO NOTHING` form. `ON CONFLICT (col1, col2) DO UPDATE` with a named conflict target is NOT accepted. The rejection is on the `DO UPDATE` form; `RETURNING` compounds it but is not the sole cause.

## Fix

### Solution 1 — MERGE INTO (H2 standard mode, single-threaded test contexts only)

```sql
MERGE INTO tbl AS t
USING (SELECT ... AS sid) AS s
ON t.col = s.sid
WHEN MATCHED THEN UPDATE SET val = t.val + 1
WHEN NOT MATCHED THEN INSERT (col, val) VALUES (s.sid, 2)
```

Follow with a separate `SELECT`. **Not concurrent-safe for first-inserts under H2 MVStore** — see `GE-20260618-174019` on H2 MERGE INTO locking.

### Solution 2 — Three-way dialect split (recommended for concurrent production use)

```java
// Real PostgreSQL: DO UPDATE (single atomic upsert; DO UPDATE row lock)
"INSERT INTO tbl (col, val) VALUES (CAST(? AS UUID), 2) " +
"ON CONFLICT (col) DO UPDATE SET val = tbl.val + 1"

// H2 MODE=PostgreSQL: DO NOTHING + separate UPDATE (DO UPDATE not accepted)
"INSERT INTO tbl (col, val) VALUES (CAST(? AS UUID), 1) ON CONFLICT DO NOTHING"
// then: "UPDATE tbl SET val = val + 1 WHERE col = CAST(? AS UUID)"

// H2 standard: full MERGE (single statement, single-threaded tests only)
// MERGE INTO ... WHEN MATCHED THEN UPDATE ... WHEN NOT MATCHED THEN INSERT ...
```

Detect H2 MODE via `INFORMATION_SCHEMA.SETTINGS WHERE SETTING_NAME = 'MODE'` — `getDatabaseProductName()` returns `"H2"` in all modes; `getMetaData().getURL()` may omit the MODE property (Agroal strips connection properties).

## Why This Is Non-Obvious

`MODE=PostgreSQL` suggests full PostgreSQL syntax compatibility. The H2 docs do not list which DML extensions are and aren't supported. The partial support (`DO NOTHING` accepted, `DO UPDATE` with named target rejected) is especially confusing: developers who correctly use the no-target `DO NOTHING` form may assume `DO UPDATE` would also work.
