### Symptom
Emulated SC2 server returns malformed or empty GameInfo responses — playerRace is null, pathing grid stubs are missing.

### Context
Building an in-process SC2 protocol server that wraps an emulated game engine. The SC2 client sends GameInfo and JoinGame as separate requests over the same WebSocket connection. The protocol does not mandate ordering.

### Root cause
`buildGameInfo()` was called during server construction, before any client connected. When the client sends GameInfo before JoinGame, race-dependent fields (playerRace from JoinGame response, pathing grid stubs configured per-race) are not yet initialized.

### Fix
Defer `buildGameInfo()` setup to the JoinGame response handler. Store `playerRace` from the JoinGame request and use it to configure all race-dependent protobuf fields. GameInfo requests before JoinGame return a response without race-specific data; after JoinGame, the full response is available.

### Why this is non-obvious
The SC2 protocol documentation does not specify request ordering constraints. The real SC2 game process handles requests in any order because all game state is pre-initialized. An emulated server that lazily initializes race-dependent state must explicitly handle the ordering gap.
