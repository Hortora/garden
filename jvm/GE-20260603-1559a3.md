## Symptom

`@QuarkusTest` integration tests for PostgreSQL Row Level Security (RLS) pass even when the RLS policy should filter rows. A test inserts rows for two tenants, sets `casehub.tenancy_id` for one tenant, and expects the query to return only that tenant's rows — but returns all rows.

## Context

Testing RLS policies on engine tables with `FORCE ROW LEVEL SECURITY` enabled. Policies applied correctly at startup. Tests seemed to verify enforcement but actually tested nothing — the filtering only appeared to work because the test happened to query the right tenant.

## Root cause

Quarkus Dev Services creates a PostgreSQL container and connects as a **superuser** (`postgres` or similar). PostgreSQL superusers bypass Row Level Security even with `FORCE ROW LEVEL SECURITY`. The docs for `FORCE ROW LEVEL SECURITY` say it "applies to the table owner" — but PostgreSQL superusers are a special case: they bypass ALL RLS policies unconditionally, regardless of `FORCE RLS`.

From PostgreSQL docs (buried in the RLS section):
> "Superuser-owned tables are not subject to row security."

## Impact

Any `@QuarkusTest` that tests RLS enforcement via Dev Services will produce false positives — the test passes, RLS appears to work, but production behavior (with a restricted app user) may differ.

## Fix / Workaround

To test actual RLS enforcement, use a non-superuser database user. Options:

1. **Custom Testcontainers setup** — configure the PostgreSQL container with a limited user:
   ```java
   PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:16")
       .withUsername("app_user")
       .withPassword("password")
       .withDatabaseName("testdb");
   // Grant CONNECT, CREATE, plus DML on tables — no SUPERUSER
   ```

2. **Document the limitation** — acknowledge in test Javadoc that kernel-level RLS enforcement is not testable via Dev Services; test only confirms wiring and BYPASSRLS role behavior.

## Why this is non-obvious

`FORCE ROW LEVEL SECURITY` documentation says it applies even to table owners — implying it applies to everyone. The superuser exception is documented separately in a different part of the PostgreSQL RLS docs. Developers implementing RLS and writing tests with Dev Services have no obvious signal that their tests are vacuously passing.
