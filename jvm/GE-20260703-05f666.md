## Duplicate CorpusIngestionBinding from config-driven + custom CDI producers

### Symptom

"Started filesystem watcher for corpus 'garden'" appears twice in engine startup logs. The first binding processes the corpus and saves the cursor; the second finds the cursor already present and does nothing.

### Context

The engine's `GardenBindingProducer` creates a custom `CorpusIngestionBinding` bean using `hortora.garden.path`. Separately, neocortex's `CorpusBindingProducer` creates bindings from `casehub.corpus.corpora.<name>` config properties. When `casehub.corpus.corpora.garden.source` is set, both producers create a binding named "garden".

### Root cause

`CorpusIngestionService.allBindings()` collects from both `bindingProducer.bindings()` (config-driven) and `customBindings` (CDI `Instance<CorpusIngestionBinding>`). No deduplication by name occurs. Both bindings share the same cursor key, so the first to run saves the cursor and the second becomes a no-op.

### Fix

Do not add `casehub.corpus.corpora.garden.source` to `application.properties` when the engine already has `GardenBindingProducer`. The custom binding is the intended path — the config-driven binding is for projects without a custom producer.

### Why this is non-obvious

- No error or warning about the duplicate — just a second "Started filesystem watcher" log line that's easy to overlook
- The `CorpusBindingProducer` warning ("no matching storage config — skipping") looks like a problem to fix by adding the config, but adding it creates the duplicate
- The two bindings share a cursor file by name, creating a race condition that's invisible in logs
