## Symptom

Tests fail non-deterministically with:

```
(RECIPIENT_FAILURE,8191) CaseDefinition not found for case: <UUID>
at SchedulerService.registerScheduledTriggers()
```

The failing test has no relationship to the case definition loader. The case definition IS registered at startup (verified: "Registering case: aml-investigation" appears in logs).

## Context

`@QuarkusTest` suite where multiple tests start casehub-engine investigations in sequence. H2 in-memory database with casehub-ledger entities on the qhorus persistence unit (`casehub.ledger.datasource=qhorus`). Multiple Quartz worker threads (default thread pool size).

## What was tried (didn't work)

- Adding `MemorySubCaseGroupRepository` and `MemoryPlanItemStore` to `selected-alternatives` (fixed CDI startup but not the cascade)
- Checking `DefaultCaseDefinitionRegistry` registration logic — confirmed synchronous via `.await().atMost(30s)`
- Checking `TestCaseMetaModelRepository` thread safety — `ConcurrentHashMap`, should be safe

## Root cause

Multiple Quartz worker jobs for the same investigation case run concurrently. Each job writes a `WorkerDecisionEntry` which triggers a Merkle Mountain Range frontier update (`INSERT INTO ledger_merkle_frontier`). H2 lacks PostgreSQL row-level locking — two concurrent writes violate `UQ_MERKLE_FRONTIER_SUBJECT_LEVEL (subject_id, level)`. The second write fails with `ConstraintViolationException`. The JTA transaction rolls back, leaving the **qhorus `EntityManager` in a poisoned state**. The next investigation's `startCase()` fires a Vert.x event bus message; the poisoned EntityManager causes processing to fail; Vert.x returns `RECIPIENT_FAILURE,8191 CaseDefinition not found`. All subsequent tests cascade-fail the same way.

The symptom is three hops from the cause: concurrent Quartz writes → Merkle race → poisoned EntityManager → `CaseDefinition not found`.

## Fix

Set `casehub.ledger.hash-chain.enabled=false` in test `application.properties`:

```properties
casehub.ledger.hash-chain.enabled=false
```

This disables Merkle leaf hash computation and frontier updates entirely — `LedgerEntry.digest = null`, no `INSERT INTO ledger_merkle_frontier`. The hash chain is owned by casehub-ledger and tested in its own suite; consumer apps test that entries are written with the correct structure, not that the cryptographic chain is intact. See also casehub garden protocol PP-20260604-f45c95.

## Why this is non-obvious

The symptom (`CaseDefinition not found`) is reported from the *next* investigation's case startup, not from the investigation that caused the Merkle violation. The failing test has nothing to do with case definitions. The `DefaultCaseDefinitionRegistry` is proven populated at startup. The poisoned EntityManager is an invisible intermediate state. H2 vs PostgreSQL locking semantics is the root cause.

### Why this fix

The hash chain is a compliance/cryptography concern belonging to casehub-ledger. Consumer app tests verify entry presence and causal chain structure — not cryptographic integrity. Disabling it in tests is architecturally correct, not a workaround.

### Alternatives considered

- Single-threaded Quartz (`quarkus.quartz.thread-count=1`): makes investigations sequential but drains time out for complex suites
- Investigation drain pattern (GE-20260605-c91317): necessary but not sufficient without the hash-chain flag — H2 is still vulnerable to concurrent workers in the same investigation
