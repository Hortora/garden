## What exists

`TrustGateService.currentScore(actorId)` does not inject `ActorTrustScoreRepository`
directly. It delegates to a `TrustScoreSource` SPI, of which there are three
implementations in casehub-ledger:

- `MaterializedTrustScoreSource` — `@Default`; reads from `ActorTrustScoreRepository`
- `CachedTrustScoreSource` — `@Alternative`; caches reads from `ActorTrustScoreRepository`
- `ComputedTrustScoreSource` — `@Alternative`; computes scores from raw attestations (no repo reference)

The full resolution chain under default configuration:

```
TrustGateService.currentScore()
  → MaterializedTrustScoreSource  (@Default TrustScoreSource)
    → ActorTrustScoreRepository   (injected — JPA or NoOp depending on selected-alternatives)
```

## Context

This matters when debugging why `currentScore()` returns empty: the root cause is
two CDI resolution steps away from the call site — not one. Discovering this requires
reading `TrustGateService.class` bytecode (no source available) to find the
`TrustScoreSource` reference, then checking which `TrustScoreSource` implementation
is `@Default`.

## Why this is non-obvious

The `TrustGateService` API surface suggests a direct score lookup. There is no
documentation of the `TrustScoreSource` SPI, its implementations, or the delegation
chain in any casehub-ledger docs. The indirection is only discoverable by reading
the bytecode constant pool.
