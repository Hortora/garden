## Symptom

Quarkus test startup fails with:

```
FlywayValidateException: Validate failed: Migrations have failed validation
Detected resolved migration not applied to database: 1.4.0.
```

This happens after adding a new Flyway migration file when Testcontainers is
running with `TESTCONTAINERS_RYUK_DISABLED=true` (containers not stopped between runs).

## Context

With Ryuk disabled, Testcontainers reuses the same PostgreSQL Docker container across
JVM restarts. The container's database has Flyway schema history with migrations
V1.0.0–V1.3.0 applied. A new migration V1.4.0 exists in the classpath (from a freshly
built JAR) but is not yet in the DB's `flyway_schema_history`. Flyway's validate step
detects this mismatch and fails **before** migrate runs.

`flyway.clean-at-start=true` is supposed to drop and recreate the schema, but in
practice the validate-then-clean ordering or container state prevents it from resolving
the mismatch in all cases.

## Fix

Add to the affected module's test `application.properties`:

```properties
quarkus.flyway.out-of-order=true
```

This allows Flyway to apply the new migration to the cached DB without requiring
a schema clean. The migration is applied out-of-order relative to the DB's history,
which is safe in a dev/test context.

## Why this is non-obvious

`quarkus.flyway.out-of-order` is not mentioned in the Quarkus Flyway guide's
troubleshooting section. The property exists in Flyway's own API but its interaction
with Testcontainers container reuse under `TESTCONTAINERS_RYUK_DISABLED=true` is
not documented anywhere. The error message ("Detected resolved migration not applied")
appears to be a validation issue but is actually a container-reuse consequence.
