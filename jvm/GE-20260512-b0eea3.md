### Symptom

After calling `set(key, null)` (or `put(key, null)` via `MapCaseFile`) on an absent key, `contains(key)` returns `false` and `get(key, ...)` returns `null` — as if the write never happened.

### Context

Occurs in any code path that writes `null` to a key that doesn't yet exist in a `CaseContextImpl` (or `MapCaseFile`) instance. Migration code from casehub-poc's `HibernateCaseFile` is especially at risk — poc stored null entries directly.

### What was tried (didn't work)

Assuming `set(key, null)` behaves like `Map.put(key, null)` — inserting a null-valued entry so that `containsKey()` returns `true`.

### Root cause

`CaseContextImpl.set()` only writes when `!Objects.equals(prev, value)`. When the key is absent, `data.get(key)` returns `null` for `prev`. If `value` is also `null`, `Objects.equals(null, null)` returns `true`, so `!true == false` — `data.put()` is never reached and the key is never inserted.

### Fix

Do not use `null` to signal presence. Use `getOrDefault(key, fallback)` or `computeIfAbsent(key, fn)` for safe access. If null-as-sentinel is required, use a dedicated sentinel object.

For migration from casehub-poc: audit call sites that write null and rely on `contains()` checks — they will silently break.

### Why this is non-obvious

`Map.put(key, null)` in standard Java inserts a null-valued entry and `containsKey()` returns `true`. The idempotency guard in `CaseContextImpl` silently breaks this expectation without any error or warning.
