## Context

`@QuarkusTest` suites where multiple tests start casehub-engine investigations. Tests that assert partial progress (e.g. "senior-analyst was scheduled") exit as soon as the condition is satisfied, leaving Quartz jobs still running. Those pending jobs can write to the ledger concurrently with the next test's investigation, causing constraint violations and test contamination.

## Technique

After every test assertion, drain the investigation to completion before the test method returns. Poll the Layer6 investigations GET endpoint until `status = "completed"`:

```java
private static final Duration DRAIN_TIMEOUT = Duration.ofSeconds(20);
private static final Duration POLL_INTERVAL = Duration.ofMillis(100);

private void drainInvestigation(final UUID caseId) {
    await().atMost(DRAIN_TIMEOUT).pollInterval(POLL_INTERVAL).until(() ->
        "completed".equals(
            given().when().get("/api/layer6/investigations/" + caseId)
                    .then().extract().path("status")));
}
```

The Layer6 resource returns `completed` only when a `WorkerDecisionEntry` with `capabilityTag = "sar-drafting"` exists in the database. This entry is written AFTER the sar-drafting Quartz job finishes ALL ledger writes (including `WorkerDecisionEntry` itself and trust attestation). Using this as the drain signal guarantees all Quartz jobs for the investigation have completed their database writes before the next test starts.

Apply at the END of every test that starts an investigation — even tests that only assert partial progress must drain:

```java
@Test
void pepTransaction_completesFullyForPepFlaggedTransactions() {
    UUID caseId = startInvestigation("TXN-PEP-001", "PEP entity detected");
    // assert partial progress
    await().atMost(TIMEOUT).pollInterval(POLL_INTERVAL)
        .until(() -> scheduledWorkerNames(caseId).contains("senior-analyst-agent"));
    // drain before returning — prevents Quartz contamination
    drainInvestigation(caseId);
}
```

## Why this is non-obvious

The drain target is the sar-drafting `WorkerDecisionEntry`, not the last WORKER_SCHEDULED event you can observe. The WORKER_SCHEDULED event fires when the Quartz job is *submitted to the scheduler*, not when it *completes*. The job may still be writing to ledger when the next test starts.

Tests that assert "senior-analyst was scheduled" see WORKER_SCHEDULED for senior-analyst and return — but entity-resolution, pattern-analysis, osint-screening, and sar-drafting Quartz jobs may all still be pending. Sar-drafting is always the last worker to complete; polling for its `WorkerDecisionEntry` drains the entire investigation.

The `status = "completed"` check is a read-only JPA query on the qhorus datasource — safe to poll rapidly even if the EntityManager was recently stressed.
