### Symptom
`DatabaseMetaData.getURL()` returns a connection URL that omits connection properties
set in the original JDBC URL. On H2 via Quarkus/Agroal, a URL configured as
`jdbc:h2:mem:testdb;MODE=PostgreSQL;DB_CLOSE_DELAY=-1` may come back from
`conn.getMetaData().getURL()` as just `jdbc:h2:mem:testdb` — `MODE=PostgreSQL` is gone.

Dialect detection code that reads the URL to decide whether `ON CONFLICT DO NOTHING`
is safe silently falls back to the MERGE path even when the H2 connection is in
PostgreSQL mode.

### Context
Any code that unwraps a Hibernate/Agroal-pooled connection and reads its metadata URL
to detect H2 mode. Common in dialect-switching SQL generators that must emit different
syntax for H2 vs PostgreSQL.

### What was tried
```java
// FAILS — URL may omit MODE= parameter when connection comes from Agroal pool
String url = conn.getMetaData().getURL().toLowerCase(Locale.ROOT);
return url.contains("mode=postgresql");
```

### Root cause
JDBC connection pools (Agroal, HikariCP) do not guarantee that `getMetaData().getURL()`
returns the original configured URL verbatim. Agroal parses and reconstructs the URL
internally; connection properties like `MODE=PostgreSQL` may be stripped or re-encoded.
`getDatabaseProductName()` always returns `"H2"` regardless of mode — it does not reflect
the active compatibility setting.

### Fix
Query `INFORMATION_SCHEMA.SETTINGS` directly on the connection to read the active mode:

```java
try (var stmt = conn.createStatement();
     var rs = stmt.executeQuery(
         "SELECT SETTING_VALUE FROM INFORMATION_SCHEMA.SETTINGS " +
         "WHERE SETTING_NAME = 'MODE'")) {
    return rs.next() && "PostgreSQL".equalsIgnoreCase(rs.getString(1));
} catch (Exception ignored) {
    return false; // older H2 or schema unavailable — safe default
}
```

Real PostgreSQL is detected first by `getDatabaseProductName().contains("postgresql")`,
so this branch only runs for H2.

### Why this is non-obvious
JDBC spec (`DatabaseMetaData.getURL()`) says "Retrieves the URL for this DBMS" — most
developers assume this returns the configured URL unchanged. The spec does not require
the pooled URL to match the configured URL. `INFORMATION_SCHEMA.SETTINGS` is H2-specific
and not part of any standard, but it is the only reliable way to read the active mode.
