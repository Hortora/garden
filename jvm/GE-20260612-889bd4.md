### What Was Tried (Didn't Work)

Using `Path` record directly as a `ConcurrentHashMap` key:

```java
ConcurrentHashMap<Path, EndpointDescriptor> store = new ConcurrentHashMap<>();
store.put(Path.of("external", "salesforce", "prod"), descriptor);
store.get(Path.parse("external/salesforce/prod")); // returns null — silent miss
```

### Root Cause

`Path.equals()` depends on **both** `value` (a String) and `segments` (a `List<String>`).
Segments are derived by splitting `value` on a configurable separator
(`casehub.platform.path.separator`, default `/`). Two `Path` instances with identical
`value` but produced by different parsers (e.g. one from `Path.of(...)` which always
joins with `/`, one from `Path.parse(...)` with a custom separator) will have the same
`value` but different `segments` — and therefore different `equals()` and `hashCode()`.

The symptom is a silent lookup miss: no exception, no error, the map returns `null`.

### Fix

Use `path.value()` (a plain `String`) as the map key component:

```java
record RegistryKey(String pathValue, String tenancyId) {}

ConcurrentHashMap<RegistryKey, EndpointDescriptor> store = new ConcurrentHashMap<>();
store.put(new RegistryKey(endpoint.path().value(), endpoint.tenancyId()), descriptor);
// path.value() is the stable canonical string — separator-independent
EndpointDescriptor result = store.get(new RegistryKey(path.value(), tenancyId)); // always works
```

`String.equals()` and `String.hashCode()` are stable and separator-independent.

### Why This Is Non-Obvious

`Path` is a Java record — records auto-generate `equals()` and `hashCode()` from all
components. A developer seeing `record Path(String value, List<String> segments)` will
reasonably assume the record is a safe map key. The trap is that `segments` is not a
redundant view of `value` — it is derived from a configurable parser, meaning two
`Path` instances can represent the same logical path string but differ structurally.
Nothing in the API surface warns about this.

This applies to any record or class that derives computed fields from configuration —
using such a type as a map key includes the computed field in equality, introducing
config-dependent identity.
