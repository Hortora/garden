## Symptom

SSE endpoint appears to work: no exception, `MultiEmitter.emit()` returns without error,
subscriptions are registered. But the browser EventSource never receives any frames.
Removing `@Blocking` or switching from an emitter-based `Multi` to `Multi.createFrom().ticks()`
resolves the issue immediately.

## Context

Quarkus `@Blocking` SSE endpoint returns a `Multi<String>`. An `@ApplicationScoped` bean
holds a `ConcurrentHashMap<String, List<MultiEmitter<String>>>`. Another component
calls `emit()` on the stored emitter from a Vert.x I/O thread (e.g. from a reactive
message-dispatch pipeline, or from a virtual thread). The emitter is correctly registered,
not cancelled, not null.

## Root cause

**Confirmed (2026-05-27):** Vert.x Netty channels are not safe for cross-context writes.
`MultiEmitter.emit(1)` runs downstream subscriber callbacks synchronously on the calling
thread (thread X — the Vert.x I/O thread or virtual thread that called `emit()`). When
RESTEasy Reactive's SSE publisher receives the item from thread X and attempts to write the
HTTP chunk, it is on the wrong event loop context — the SSE response is owned by thread Y
(the event loop that accepted the HTTP connection). Writes from thread X to a Netty channel
owned by thread Y are silently discarded with no error and no flush.

The `@Blocking` annotation only moves the *method invocation* to a worker thread — it does
not affect which thread the returned `Multi` emits items on.

## Solution 1 — Switch to polling ticks (abandons push delivery)

Replace the emitter-based `Multi` with `Multi.createFrom().ticks()`:

```java
// Broken — emitter-based approach with @Blocking SSE
Multi<String> live = Multi.createFrom().emitter(em -> {
    bus.register(channelName, em);
    em.onTermination(() -> bus.unregister(channelName, em));
});

// Working — Mutiny's built-in scheduler handles threading correctly
Multi<String> live = Multi.createFrom().ticks()
        .every(Duration.ofMillis(500))
        .onItem().transformToUniAndConcatenate(tick ->
                fetchNewMessages(lastSentId)
                        .map(entries -> entries.isEmpty() ? null : serialize(entries))
        )
        .filter(Objects::nonNull);
```

**Pros:** Simple; `Multi.createFrom().ticks()` uses Mutiny's internal scheduler which
handles Vert.x event loop threading correctly regardless of `@Blocking`.
**Cons:** Abandons event-driven push delivery; polls at a fixed interval regardless of
message activity; wastes DB reads when the channel is idle.

## Solution 2 — emitOn(workerPool) (preserves push delivery)

Keep the emitter-based bus but shift delivery off the I/O thread before the downstream pipeline:

```java
channelEventBus.subscribe(channelName)
    .emitOn(Infrastructure.getDefaultWorkerPool())  // ← shift off I/O thread
    .onItem().transformToUniAndConcatenate(tick ->
        fetchNewMessages(lastSentId)
            .map(entries -> entries.isEmpty() ? null : serialize(entries))
    )
    .filter(Objects::nonNull);
```

`emitOn(workerPool)` submits item delivery to a worker thread. From a worker thread
(no Vert.x context), RESTEasy Reactive's SSE publisher calls `HttpServerResponse.write()`,
which Vert.x 4 schedules on the owning event loop via `connection.context().runOnContext()`.
The frame arrives at the browser reliably while preserving push (event-driven) delivery.

**Pros:** Truly event-driven — no polling; frames arrive immediately when the bus emits;
no idle DB reads.
**Cons:** Requires understanding the threading model; `emitOn()` adds a worker-thread hop
for every signal, which is negligible overhead for typical message rates.

## Why this is non-obvious

The emitter approach works correctly in non-SSE contexts and in non-`@Blocking`
reactive endpoints. The failure is completely silent — no exception, no warning, the
code appears to execute normally. The threading assumption (emit from any thread → frame
delivered) is not documented as invalid for SSE. The `@Blocking` annotation misleads:
developers assume it makes the entire SSE lifecycle thread-agnostic, but it only covers
the method invocation phase.
