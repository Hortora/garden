## Symptom

SSE endpoint appears to work: no exception, `MultiEmitter.emit()` returns without error,
subscriptions are registered. But the browser EventSource never receives any frames.
Removing `@Blocking` or switching from an emitter-based `Multi` to `Multi.createFrom().ticks()`
resolves the issue immediately.

## Context

Quarkus `@Blocking` SSE endpoint returns a `Multi<String>`. An `@ApplicationScoped` bean
holds a `ConcurrentHashMap<String, List<MultiEmitter<String>>>`. Another component
(called from `ChannelGateway.fanOut()` via a virtual thread) calls `emit()` on the stored
emitter. The emitter is correctly registered, not cancelled, not null.

## Root cause

The root cause is suspected to be a Vert.x event-loop / response-thread mismatch.
The `@Blocking` annotation moves endpoint handling to a worker thread. When `emit()` is
called from a different virtual thread (the fanOut virtual thread), the Mutiny pipeline
may not dispatch the emission back to the thread that owns the SSE response sink.
The result: items enter the pipeline but frames are not flushed to the wire.

The exact Mutiny/Vert.x internal mechanism was not confirmed — the symptom was
conclusive but the root cause required source-code investigation beyond the scope of
the fix session.

## Fix

Replace the emitter-based `Multi` with `Multi.createFrom().ticks()`:

```java
// Broken — emitter-based approach with @Blocking SSE
Multi<String> live = Multi.createFrom().emitter(em -> {
    bus.register(channelName, em);
    em.onTermination(() -> bus.unregister(channelName, em));
});

// Working — Mutiny's built-in scheduler handles threading correctly
Multi<String> live = Multi.createFrom().ticks()
        .every(Duration.ofMillis(500))
        .onItem().transformToUniAndConcatenate(tick ->
                fetchNewMessages(lastSentId)
                        .map(entries -> entries.isEmpty() ? null : serialize(entries))
        )
        .filter(Objects::nonNull);
```

`Multi.createFrom().ticks()` uses Mutiny's internal scheduler which handles Vert.x
event loop threading correctly regardless of whether the endpoint is `@Blocking`.

## Why this is non-obvious

The emitter approach works correctly in non-SSE contexts and in non-`@Blocking`
reactive endpoints. The failure is silent — no exception, no warning, the code
appears to execute. The threading assumption (emit from any thread → frame delivered)
is documented nowhere as invalid for SSE.
