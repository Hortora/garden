### Symptom

`org.h2.jdbc.JdbcSQLIntegrityConstraintViolationException: Unique index or primary key violation` on a table used as a per-subject sequence counter, when two concurrent `REQUIRES_NEW` transactions both execute a MERGE upsert on the same key:

```sql
MERGE INTO ledger_subject_sequence AS t
USING (SELECT CAST(?1 AS UUID) AS sid) AS s ON t.subject_id = s.sid
WHEN MATCHED THEN UPDATE SET next_seq = t.next_seq + 1
WHEN NOT MATCHED THEN INSERT (subject_id, next_seq) VALUES (s.sid, 2)
```

The violation fires on the INSERT in the WHEN NOT MATCHED branch.

### Context

A `@ApplicationScoped @Transactional(REQUIRED)` save method (called from inside a caller's `REQUIRES_NEW` transaction) runs the MERGE to atomically increment a per-subject sequence counter. Under concurrent load, two threads both enter the MERGE simultaneously — both evaluate WHEN NOT MATCHED before either commits — both attempt INSERT of the same primary key.

PostgreSQL serializes this correctly: the first MERGE acquires an exclusive row lock on the target; the second blocks until the first commits. H2 does not provide this row-level serialization for the NOT MATCHED branch.

### Root cause

H2's `MERGE INTO ... WHEN NOT MATCHED THEN INSERT` is not serialized under concurrent transactions for new keys. Two concurrent transactions can both evaluate the WHEN NOT MATCHED predicate as true (neither can see the other's uncommitted INSERT), and both proceed to INSERT the same primary key — causing a constraint violation.

PostgreSQL's MERGE acquires a row-level lock before INSERT, preventing the concurrent evaluation. H2 does not.

### Fix

**Atomically commit the sequence allocation in its own REQUIRES_NEW transaction, while holding a `synchronized` lock that prevents concurrent threads from starting their MERGE before the first thread's REQUIRES_NEW has committed.**

The critical insight: the `synchronized` lock must be held in a non-transactional wrapper, which calls a SEPARATE CDI bean's `@Transactional(REQUIRES_NEW)` method. When that method returns, the CDI `@Transactional` interceptor commits the REQUIRES_NEW immediately (inside the `proceed()` call). The lock is still held at that point — T2 cannot acquire it until T1's commit is visible.

**Wrong pattern** — lock-inside-same-bean or synchronized-on-method:

```java
// ❌ CDI proxy commits AFTER method body returns (after synchronized releases)
@Transactional(REQUIRES_NEW)
public synchronized LedgerEntry save(...) {
    em.createNativeQuery(MERGE_SQL).executeUpdate();
    em.flush();
    // ... select next_seq
    em.persist(entry);
    return entry;
} // synchronized releases HERE; CDI interceptor commits AFTER HERE → race window exists
```

T2 can acquire the lock after the method body returns but before the CDI interceptor commits T1's REQUIRES_NEW. T2's MERGE still sees NOT MATCHED.

**Correct pattern** — separate CDI bean for the REQUIRES_NEW:

```java
// CDI bean A (caller, synchronized but NOT @Transactional)
@ApplicationScoped
class QhorusLedgerEntryRepository {
    @Inject QhorusSequenceAllocator sequenceAllocator; // separate bean!

    @Override
    public synchronized LedgerEntry save(LedgerEntry entry, String tenancyId) {
        // sequenceAllocator.nextSequenceNumber() is a @Transactional(REQUIRES_NEW) call
        // through the CDI proxy. When it RETURNS, the REQUIRES_NEW is committed.
        // The synchronized lock is still held at that point — T2 is blocked.
        entry.sequenceNumber = sequenceAllocator.nextSequenceNumber(entry.subjectId);
        em.persist(entry);
        return entry;
    }
}

// CDI bean B (allocator — separate class so @Transactional applies via CDI proxy)
@ApplicationScoped
class QhorusSequenceAllocator {
    @Inject @LedgerPersistenceUnit EntityManager em;

    @Transactional(TxType.REQUIRES_NEW)
    int nextSequenceNumber(UUID subjectId) {
        em.createNativeQuery(MERGE_SQL).setParameter(1, subjectId).executeUpdate();
        em.flush(); // required: AUTO flush mode doesn't flush before native queries
        Number n = (Number) em.createNativeQuery(SELECT_SQL)
            .setParameter(1, subjectId).getSingleResult();
        return n.intValue();
    } // CDI interceptor commits REQUIRES_NEW here — INSIDE the caller's synchronized block
}
```

T1 and T2 concurrent flow:
1. T1 acquires `synchronized(this)` → calls `sequenceAllocator.nextSequenceNumber()` → REQUIRES_NEW starts, MERGE INSERTs (subject_id, 2), REQUIRES_NEW COMMITS, method returns → lock still held
2. T2 tries to acquire `synchronized(this)` → blocked
3. T1 releases lock (save() body returns)
4. T2 acquires lock → calls `sequenceAllocator.nextSequenceNumber()` → MERGE sees MATCHED (T1's row committed) → UPDATEs next_seq to 3, REQUIRES_NEW COMMITS, returns 2

No PK violation.

### Why this is non-obvious

The naive fix — adding `@Transactional(REQUIRES_NEW)` to the `save()` method AND marking it `synchronized` — doesn't work. The CDI `@Transactional` interceptor commits the REQUIRES_NEW AFTER the method body returns. `synchronized` is released when the method body returns. There is a race window between "method body returns" (lock released) and "CDI interceptor commits" (REQUIRES_NEW visible). T2 can enter this window.

The fix requires understanding that CDI `@Transactional` is implemented as an `around` interceptor: `begin tx → proceed() [calls method body] → commit tx`. The commit happens in the interceptor, NOT inside the method body. Only by calling a SEPARATE CDI bean's REQUIRES_NEW method — where the commit happens inside `proceed()`, while the caller still holds the synchronized lock — is the race window eliminated.

PostgreSQL avoids this pattern entirely: its MERGE acquires a row-level lock before INSERT, serializing concurrent NOT MATCHED evaluations. H2 does not.

### Alternatives considered

- **`LOCK TABLE ledger_subject_sequence WRITE`** (H2-specific) before MERGE: acquires table-level lock; prevents all concurrent writes. Works but is too coarse for production use and depends on H2-specific syntax.
- **`synchronized(this)` on the @Transactional method**: CDI commits after method body returns — race window remains.
- **`@Transactional(REQUIRES_NEW)` on save() without synchronized**: two concurrent REQUIRES_NEW transactions still race in H2 (same root cause).
- **Single-statement `INSERT OR IGNORE / ON CONFLICT DO UPDATE`**: H2 supports `ON CONFLICT` syntax in ANSI and PostgreSQL modes; not available in default H2 mode without `MODE=PostgreSQL` in the JDBC URL.
