## Symptom

A Quarkus library module (no main class, no `@QuarkusApplication`) has `quarkus:build` in its Maven plugin configuration. At build time, CDI validation runs against only the compile-scope classpath. Injection points that depend on beans from other modules produce:

```
ERROR: Unsatisfied dependency for type X and qualifiers @Default
```

The error disappears at runtime because the missing bean arrives via a transitive dependency that is not in compile scope.

## Context

Arose in `casehub-persistence-hibernate` — a pure library JAR providing JPA repositories. `TenantAwareRepository` was added with `@Inject CurrentPrincipal`. `casehub-platform` (which provides `MockCurrentPrincipal @DefaultBean`) is a runtime dependency of downstream consumers, not a compile dependency of the library itself. The `quarkus:build` goal validated CDI at compile scope, found no `CurrentPrincipal` bean, and failed.

## Root cause

`quarkus:build` triggers Quarkus augmentation, which includes CDI graph validation. For a library JAR, augmentation runs against the module's compile-scope classpath only — it doesn't simulate the full deployment classpath. The result is false-positive "Unsatisfied dependency" errors for beans that will be provided by consumers at runtime.

Library JARs don't need `quarkus:build` — the Quarkus Maven plugin's `generate-code-tests` goal (which runs augmentation in test phase only, against the full test classpath) is sufficient.

## Fix

Remove `<goal>build</goal>` from the Quarkus Maven plugin configuration in the library module's `pom.xml`. The `generate-code-tests` goal covers test augmentation with the full test classpath.

```xml
<!-- Remove this from library module pom.xml -->
<execution>
  <id>...</id>
  <goals>
    <goal>build</goal>  <!-- ← remove -->
  </goals>
</execution>
```

## Why this is non-obvious

`quarkus:build` looks harmless in a library module — it's a common plugin goal. No warning is emitted that the goal is running with an incomplete classpath for a non-app module. The resulting CDI error looks exactly like a real missing dependency, sending developers to add `@DefaultBean` stubs or fallback implementations that then conflict with beans in downstream consumers.
