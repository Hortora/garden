### Symptom
12 @QuarkusTest tests fail with ConditionTimeoutException — cases start (RUNNING)
but workers never execute and cases never reach COMPLETED. Failures appear
test-ordering-dependent: passing in isolation, failing in the full suite. Symptoms
are indistinguishable from Quartz scheduler contamination or event bus saturation.

### Context
Two CaseHub beans — one Java-defined, one YAML-defined — both register under the
same registry key (namespace, name, version). The registry uses `ConcurrentHashMap`
with "first wins" semantics: whichever CDI bean's `getDefinition()` is iterated first
via `Instance<CaseHub>` determines the active definition for that key. Quarkus ARC
does not guarantee `Instance<T>` iteration order across JVM instances.

### What was tried (didn't work)
1. Investigated Quartz scheduler contamination — stale jobs, event bus saturation
2. Checked LoopControl, AgentRoutingStrategy, and WorkerExecutionManager for shared state
3. Grepped for thread-safety issues in the JQ evaluator

### Root cause
The YAML definition used flat-context JQ expressions (`.status == "processing"`)
while the Java definition used panels-convention expressions (`.working.status == "processing"`).
When the YAML definition registered first, its expression was evaluated against a context
where data lived under `.working.status`. The JQ evaluator correctly returned `false` —
the expression was wrong for that context shape. No error, no warning — silent mismatch.

### Fix
1. Give each CaseHub bean a unique (namespace, name, version) key
2. Add fail-fast duplicate detection at startup in `registerKnownDefinitions()` —
   throws IllegalStateException naming both colliding beans
3. Add CaseKeyUniquenessTest — explicit regression guard scanning all beans

### Why this is non-obvious
Every symptom pointed to scheduler contamination: timeouts, workers not executing,
test-ordering dependency. The actual cause (registry key collision + CDI ordering
non-determinism) is invisible without tracing through the expression evaluation chain
with targeted debug logging. The JQ evaluator worked correctly — it evaluated the
wrong expression against the right data, returning `false` as expected.

### Why this fix
Fail-fast at startup is stronger than a dedicated test alone: every @QuarkusTest in
every module becomes an implicit collision detector. The explicit test provides
diagnostic messages naming both beans for fast resolution.

### Alternatives considered
- Warn-and-skip on duplicate keys (original behaviour) — silent, causes exactly this bug
- Structural equality check on definitions instead of identity — too permissive; two
  beans producing structurally identical definitions is still a maintenance hazard
- Making CDI bean ordering deterministic — not possible without Quarkus ARC changes
