## Symptom

`NullPointerException` propagated through CDI's `EventImpl.handleExceptions` — the stack trace
points at the event infrastructure, not at your code. Your observer has a null guard and returns
early. The error appears to come from nowhere.

```
java.util.concurrent.CompletionException: java.lang.NullPointerException
    at io.quarkus.arc.impl.EventImpl.handleExceptions(EventImpl.java:242)
    at io.quarkus.arc.impl.EventImpl$1.get(EventImpl.java:112)
```

## Context

Occurs when `@ObservesAsync` CDI event has a null field AND a different observer (not the one you
are debugging) calls `Set.of(...).contains(nullValue)`. CDI aggregates exceptions from all async
observers via `CompletionException` — any observer that throws will surface here, making it look
like a general CDI failure rather than pointing at the offending observer.

Hit in practice: `CaseMemoryObserver` filtering events with:
```java
private static final Set<String> CAPTURED_EVENTS = Set.of("CaseCompleted", "CaseCancelled", "CaseFailed");

if (!CAPTURED_EVENTS.contains(event.eventType())) {  // NPE when eventType is null
    return;
}
```

When an event with `eventType == null` is fired, `Set.of().contains(null)` throws, not returns
false.

## Root Cause

`java.util.Set.of()` (and all immutable collections: `List.of()`, `Map.of()`, `Map.copyOf()`)
throw `NullPointerException` when `contains(null)` is called. This is specified in the JDK docs
under "Unmodifiable Collections" but rarely noticed because `HashSet.contains(null)` returns
false — and developers using `Set.of()` for filter sets don't expect the difference.

The JDK spec (SE 11+):
> These collections do not permit null elements. Attempts to create them with null elements result
> in NullPointerException.

`contains(null)` is considered "attempting to use null" per the immutable collection contract.

## Fix

Guard before calling `contains` on an immutable set:

```java
// Null-safe filter using immutable set
if (value == null || !CAPTURED_EVENTS.contains(value)) {
    return;
}

// Or use a null-safe contains helper
private static boolean captured(String eventType) {
    return eventType != null && CAPTURED_EVENTS.contains(eventType);
}
```

Alternatively, use `EnumSet` or a `HashSet` if null tolerance is genuinely needed (rare).

## Why This Is Non-Obvious

Developers know `Set.of()` is immutable and doesn't accept null elements *at creation time*. Most
assume `contains()` on an immutable set behaves like `contains()` on a `HashSet` (returns false
for null). The NPE is thrown at query time, not creation time, and the stack trace in CDI async
dispatch hides the originating observer behind `EventImpl.handleExceptions`.

The double indirection — CDI async dispatch aggregating exceptions from multiple observers — means
your first debugging instinct is to look at your own observer (which has the null guard), not at
a completely different observer in a different library.
