# Qdrant QueryFactory.rrf(Rrf) vs fusion(Fusion.RRF) — configurable k only through rrf()

## Symptom

RRF fusion works correctly but there's no way to customize the `k` constant (default 60). The `QueryFactory.fusion(Fusion.RRF)` method provides no mechanism to pass configuration.

## Context

Building hybrid search with prefetch + RRF fusion using the Qdrant Java gRPC client. Multiple prefetch queries (dense, sparse) need to be combined with a custom RRF `k` value to control rank blending.

## What was tried

- Looked for a `Fusion.RRF.builder()` or similar — doesn't exist
- Checked if `QueryPoints.Builder.setFusion()` accepts parameters — only takes the enum
- Searched Qdrant Java client Javadoc — method signatures are documented but not the distinction

## Root cause

Two factory methods exist for RRF:

1. **`QueryFactory.fusion(Fusion.RRF)`** — convenience method that uses server defaults for `k` (60)
2. **`QueryFactory.rrf(Rrf.newBuilder().setK(yourK).build())`** — accepts a full `Rrf` protobuf message with custom configuration

The Java client Javadoc doesn't explain this distinction. Without reading the underlying protobuf definition or gRPC API spec, there's no indication that `rrf()` accepts additional configuration that `fusion()` doesn't.

## Fix

Use `QueryFactory.rrf(Rrf.newBuilder().setK(yourK).build())` when a custom `k` is needed:

```java
// Server default k=60
Query fusionQuery = QueryFactory.fusion(Fusion.RRF);

// Custom k=100
Query fusionQuery = QueryFactory.rrf(
    Rrf.newBuilder()
        .setK(100)
        .build()
);
```

Use `fusion(Fusion.RRF)` only when server defaults are acceptable.

## Why non-obvious

The two factory methods look like aliases or convenience wrappers. The names don't indicate that one is parameterless and the other accepts configuration. Most hybrid search examples online use the simpler `fusion()` form, so the existence of `rrf()` isn't widely known.

The Qdrant REST API documentation shows RRF configuration options, but the Java client method signatures are underdocumented — you have to inspect the protobuf types to discover that `Rrf` is a builder-based message with a `k` field.
