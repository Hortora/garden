## Symptom

When the quarkus-langchain4j-ollama extension is on the classpath alongside a
`@DefaultBean ChatModel` implementation, Quarkus throws at startup:

```
AmbiguousResolutionException: Ambiguous dependencies for type ChatModel and qualifiers [@Default]
  - CLASS bean [... AgentProviderChatModel] qualifiers=[@Any, @Default]
  - SYNTHETIC bean [... OllamaChatModel] qualifiers=[@Any, @Default]
```

## Context

A custom `@DefaultBean @ApplicationScoped ChatModel` implementation exists to serve as a
fallback (e.g. a Claude CLI bridge). The expectation is that adding the Ollama extension
places its `ChatModel` higher-priority, which should silence the `@DefaultBean`. Instead,
both are registered with `@Default` and Quarkus cannot choose between them.

## Root cause

`@DefaultBean` in Quarkus ARC means: "this bean is only active if no other `@Default`-
qualified bean of the same type is present." The mechanism works when the competing bean
is `@ApplicationScoped` with no explicit qualifier (which implicitly gets `@Default`), or
when the competing bean is annotated with `@Alternative @Priority(N)`. However, the
quarkus-langchain4j-ollama extension registers its model as a **SYNTHETIC bean with an
explicit `@Default` qualifier** — not as a higher-priority alternative. ARC sees two
explicit `@Default` beans and cannot resolve.

## Fix

Veto the `@DefaultBean` explicitly when the Ollama extension is active by pairing
`@IfBuildProperty` with `-Dcasehub.eval.claude-provider.enabled=false` at build time:

```java
@DefaultBean
@ApplicationScoped
@IfBuildProperty(name = "casehub.eval.claude-provider.enabled",
                 stringValue = "true", enableIfMissing = true)
class AgentProviderChatModel implements ChatModel { ... }
```

```bash
# Run with Ollama — veto the DefaultBean
mvn clean test -Peval,eval-ollama \
  -Dcasehub.eval.claude-provider.enabled=false \
  ...
```

`mvn clean` is required to force fresh augmentation when toggling this property (see
related entry on augmentation caching).

## Why this is non-obvious

`@DefaultBean` is marketed as "the bean that loses to any other bean of the same type."
In practice it only loses to beans that don't also carry an explicit `@Default` qualifier.
Quarkus LangChain4j provider extensions register SYNTHETIC beans with explicit `@Default`
rather than `@Alternative @Priority`, making the clash non-obvious. The `@DefaultBean`
javadoc does not describe this failure mode.
