### What this is

When implementing `ChannelProjection<S>.apply(S state, MessageView message)`, use
`message.actorType()` (an `ActorType` enum: HUMAN / AGENT / SYSTEM) to determine who
sent the message — not `message.sender()` string parsing.

### Why this matters

Every `MessageDispatch` builder call sets `actorType` explicitly at the point of dispatch.
Sender strings, by contrast, are deployment-specific IDs (e.g.
`"drafthouse-reviewer-3a7b5c..."`, `"drafthouse-human"`) that vary across sessions and
environments. String matching against sender names requires knowing every possible sender
ID in advance and breaks silently when naming conventions change.

### What developers do instead

Reach for `message.sender()` and match against known string prefixes:

```java
// ❌ Brittle: couples projection to deployment-specific naming conventions
private AgentType agentType(MessageView message) {
    if (message.sender() == null) return AgentType.REV;
    return switch (message.sender().toUpperCase()) {
        case "IMP" -> AgentType.IMP;
        default    -> AgentType.REV;
    };
}
```

This was the pattern in the original `SummaryProjector` — worked only when senders were
literal `"REV"` and `"IMP"`, broke silently when real instance IDs were used.

### The technique

```java
// ✅ Correct: actorType is set by dispatch, stable across deployments
private AgentType agentType(MessageView message) {
    if (message.actorType() == null) {
        throw new IllegalArgumentException(
                "MessageView.actorType() must not be null in projection");
    }
    return switch (message.actorType()) {
        case HUMAN  -> AgentType.REV;
        case AGENT  -> AgentType.IMP;
        default     -> throw new IllegalArgumentException(
                "Unsupported actorType in projection: " + message.actorType());
    };
}
```

### Why this is non-obvious

`MessageView` has both `sender()` and `actorType()` fields. Sender is a human-readable
string visible in channel logs; actorType is a typed enum. First instinct is to pattern-
match on the string you can see. The enum is the stable API contract.

The null check before the switch is important: Java's switch on a null enum value throws
`NullPointerException`, not the switch's `default` branch.

### v1 limitation

In v1 Q&A models with one human and one AI actor, `HUMAN→REV` and `AGENT→IMP` holds.
For multi-agent DebateChannel models (#27), both REV and IMP are AGENT actors — sender
conventions or an explicit role field in `artefactRefs` must distinguish them.
