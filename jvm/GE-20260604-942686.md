## Prometheus counter fires on both paths of a find-or-create method

A `findOrCreateWithBinding()` method (annotated `@Transactional(REQUIRES_NEW)`) had two return paths: **create-new** (channel not found → create and put binding) and **find-existing** (binding already present → return existing channel). Both paths return normally with no exception. The caller incremented a Prometheus counter immediately after the call:

```java
Channel channel = channelService.findOrCreateWithBinding(req);
meterRegistry.counter("inbound_channels_auto_created_total", ...).increment(); // ← fires on BOTH paths
gateway.initChannel(channel.id, ...);
```

### Symptom

A test asserting `isEqualTo(autoCreatedBefore + 1.0)` intermittently failed with `expected: 2.0 but was: 3.0`. The test fired two concurrent inbound messages at the same channel, expecting exactly one auto-creation. Under certain timing (warmer JVM, faster execution), Thread 1 completed the binding `put()` before Thread 2 called `findByKey()`. Thread 2 then took the find-existing path, returned normally, and the counter incremented for both threads.

### Root cause

The counter placement after a find-or-create call treats both return paths identically. "Method returned without throwing" does not mean "a new entity was created." Find-or-create methods have two silent success paths and callers must distinguish them if metrics should only count new creations.

### Fix

Return a typed result from the find-or-create method that carries a `wasCreated` flag:

```java
public record FindOrCreateResult(Channel channel, boolean wasCreated) {}

// In findOrCreateWithBinding():
if (existingBinding.isPresent()) {
    return new FindOrCreateResult(existingChannel, false); // found
}
// ... create ...
return new FindOrCreateResult(channel, true); // created

// In the caller:
FindOrCreateResult result = channelService.findOrCreateWithBinding(req);
if (result.wasCreated()) {
    meterRegistry.counter("inbound_channels_auto_created_total", ...).increment();
}
```

### Why this is non-obvious

The bug only manifests under a specific race timing — Thread 1 must complete its binding write before Thread 2 calls the initial existence check. On a cold JVM (first test in isolation), Thread 1's write is slower than Thread 2's check, so Thread 2 always takes the PersistenceException path (no counter increment). On a warmer JVM (after several tests have run), the inverse timing occurs and the bug appears. The counter increment looks correct in isolation — it's downstream of a successful method return — but "success" is ambiguous for find-or-create.
