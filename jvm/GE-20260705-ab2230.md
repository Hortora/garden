### Symptom
When designing an SSE push system with a connection map (`ConcurrentHashMap<String, Set<SseEventSink>>`), the natural approach is to register `onClose`/`onError` callbacks on each `SseEventSink` to remove dead connections immediately. These callbacks don't exist on the JAX-RS `SseEventSink` interface.

### Context
The JAX-RS 3.1 `SseEventSink` interface provides only:
- `send(OutboundSseEvent)` — send an event
- `close()` — close the connection
- `isClosed()` — check if closed

There is no `onClose(Runnable)` or `onError(Consumer<Throwable>)` method. This is different from WebSocket's `@OnClose`/`@OnError` annotations, which provide explicit lifecycle hooks.

### What was tried (didn't work)
Designed the SSE system with `emitter.onClose(() -> removeEmitter(...))` and `emitter.onError(t -> removeEmitter(...))` — these methods don't exist on `SseEventSink`. Compilation fails.

### Root cause
The JAX-RS SSE specification models `SseEventSink` as a write-only channel with a status check. Connection lifecycle management is delegated to the container — the application is expected to detect disconnection via failed `send()` calls or `isClosed()` checks, not via callbacks.

### Fix
Use lazy cleanup — check `isClosed()` before each send attempt, and remove dead emitters on send failure:

```java
void pushToUser(String connectionKey, OutboundSseEvent event) {
    Set<EmitterEntry> emitters = connections.get(connectionKey);
    if (emitters == null) return;
    for (EmitterEntry entry : emitters) {
        if (entry.sink().isClosed()) {
            removeEmitter(connectionKey, entry);
            continue;
        }
        try {
            entry.sink().send(event);
        } catch (Exception e) {
            removeEmitter(connectionKey, entry);
        }
    }
}

private void removeEmitter(String key, EmitterEntry entry) {
    connections.computeIfPresent(key, (k, emitters) -> {
        emitters.remove(entry);
        return emitters.isEmpty() ? null : emitters;
    });
}
```

The `computeIfPresent` with null return atomically removes the outer map key when the last emitter disconnects, preventing unbounded key growth.

**Trade-off:** Dead connections persist in the map until the next push attempt for that user. For low-traffic users, this means stale entries survive indefinitely. Mitigation: add a periodic `@Scheduled` sweep that iterates all emitters and removes those where `isClosed()` is true.

### Why this is non-obvious
WebSocket (`@ServerEndpoint`) has `@OnClose` and `@OnError` lifecycle annotations — the natural assumption is that SSE has equivalent hooks. The JAX-RS SSE spec documentation doesn't call out this absence explicitly. Developers coming from WebSocket or framework-specific SSE implementations (Spring's `SseEmitter` has `onCompletion`/`onTimeout`/`onError`) expect lifecycle callbacks that don't exist in the standard JAX-RS API.

### Alternatives considered
- Use Quarkus-specific SSE APIs instead of JAX-RS standard — would work but reduces portability
- Use `Multi<OutboundSseEvent>` return type (Quarkus Reactive) instead of `SseEventSink` — provides `onTermination()` but changes the programming model significantly (reactive stream vs imperative push)
- Periodic `@Scheduled` sweep — complements lazy cleanup but doesn't replace it
