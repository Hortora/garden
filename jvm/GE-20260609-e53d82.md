### Symptom
Gate is opened and the human approves it via `/openclaw/delivery/oversight/{gateId}`. The `fulfill()` dispatches DONE to the work channel, but the DONE message carries `.inReplyTo(-1L)`. The agent's work commitment is never marked FULFILLED. Case execution stalls permanently.

### Context
`openGate()` in `OversightGateService` computes `commandMessageId` with `.orElse(-1L)` when no COMMAND message is found for the commitment's correlationId. The `-1L` is immediately embedded in a `GateContext` record and serialized into the gate COMMAND message content (persisted in Qhorus). When `fulfill()` later approves the gate, it retrieves the `GateContext` from the COMMAND content and passes `commandMessageId = -1L` to `OversightGateDispatcher.dispatch()`, which calls `.inReplyTo(-1L)` on the DONE message to the work channel.

### What was tried (didn't work)
Checking inReplyTo at the OversightGateDispatcher level — by that point the gate is already committed to Qhorus with the invalid sentinel. The gate has opened; the context is persisted; the damage is done.

### Root cause
The `.orElse(-1L)` sentinel that works as a guard in the Autonomous path (where `channelBacked_done()` returns COMMAND_NOT_FOUND before dispatching) becomes invisible once control passes through `openGate()`. The method opens the gate with the sentinel embedded and returns `GatePending` — the caller sees success. No error is surfaced at gate-open time.

The Autonomous path guard in `CommitmentTools.channelBacked_done()` (`if (commandMessageId < 0) return COMMAND_NOT_FOUND`) is bypassed entirely for the gated path: `openGate()` is called *before* `findCommandMessageId()`.

### Fix
In `OversightGateService.openGate()`, guard immediately after computing `commandMessageId`:

```java
long commandMessageId = messageService.findAllByCorrelationId(commitmentId).stream()
        .filter(m -> m.messageType == MessageType.COMMAND)
        .mapToLong(m -> m.id)
        .findFirst()
        .orElse(-1L);
if (commandMessageId < 0) {
    log.warnf("openGate: no COMMAND message for commitmentId=%s — failing open", commitmentId);
    return new GateDecision.Autonomous();
}
```

### Why this is non-obvious
The code looks like it handles the no-COMMAND case correctly via `.orElse(-1L)` — a common sentinel pattern. The intent is a guard, but the guard only works if code downstream checks the sentinel before use. In this case, the downstream is `OversightGateDispatcher.dispatch()` which passes the value directly to `.inReplyTo()` with no validation. The failure is silent: the gate opens successfully, the human approves, and the DONE dispatch executes without error — only the commitment lifecycle is broken.
