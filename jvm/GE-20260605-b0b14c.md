## Symptom

A test calls `repo.save(entry)` (which assigns `sequenceNumber = 1`), then modifies the entry (adds a signature), then calls `repo.save(entry)` again. The entry's `sequenceNumber` silently changes from 1 to 2. The signature — computed with `sequenceNumber = 1` — no longer matches the stored `sequenceNumber = 2`. Signature verification returns INVALID.

## Context

The test pattern `seedEntry() → signEntry() → repo.save()` worked when `save()` was a simple `em.persist()` with no field mutation. After adding `sequenceNumber` assignment to `save()`, the second call to `save()` mutates `sequenceNumber` on the already-managed entity. JPA dirty-checks the mutation and writes it at flush time.

## Root Cause

`em.persist()` on an already-managed entity within the same `@Transactional` boundary is a JPA no-op — the entity is already tracked. But any field mutation performed by the `save()` method BEFORE `em.persist()` sticks: the entity is managed, JPA sees the dirty field, and writes it at flush. The `em.persist()` call does nothing, but the field mutation is real and persistent.

## Fix

Do not call `save()` twice on the same entity. If the entity is already managed (from a prior `save()`), mutate its fields directly — JPA will dirty-flush them:

```java
// WRONG — double save overwrites sequenceNumber:
TestEntry e = repo.save(newEntry(subjectId));  // seqNum = 1
signEntry(e, keyPair);                          // sign with seqNum 1
repo.save(e);                                   // seqNum = 2! signature broken

// CORRECT — mutate managed entity directly:
TestEntry e = repo.save(newEntry(subjectId));  // seqNum = 1, entity is managed
signEntry(e, keyPair);                          // sign with seqNum 1
em.flush();                                     // write signature fields to DB
```

## Why This Is Non-Obvious

Calling `save()` twice looks safe — `em.persist()` on a managed entity is documented as a no-op. The dangerous part is invisible: `save()` mutates fields BEFORE the no-op persist. A developer adding field mutation to `save()` (like sequence number assignment) doesn't expect to break callers who call `save()` on already-managed entities. The symptom (INVALID signature) is far from the cause (second `save()` call).
