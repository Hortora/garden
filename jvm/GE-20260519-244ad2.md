### Context

A Quarkus extension (runtime + deployment modules) needs to make some `@ApplicationScoped` beans optional — present only when a config property is true at build time. JDBC-only consumers must not have CDI augmentation failures because of unsatisfied injection points in beans they'll never use.

### Technique

**In the deployment module**, declare a `@ConfigRoot(BUILD_TIME)` config interface and add a `@BuildStep` that produces `ExcludedTypeBuildItem` entries for the optional beans:

```java
// deployment/LedgerBuildTimeConfig.java
@ConfigMapping(prefix = "casehub.ledger")
@ConfigRoot(phase = ConfigPhase.BUILD_TIME)
public interface LedgerBuildTimeConfig {
    ReactiveConfig reactive();
    interface ReactiveConfig {
        @WithDefault("false")
        boolean enabled();
    }
}

// deployment/LedgerProcessor.java
@BuildStep
void excludeReactiveBeans(
        LedgerBuildTimeConfig config,
        BuildProducer<ExcludedTypeBuildItem> excluded) {
    if (!config.reactive().enabled()) {
        excluded.produce(new ExcludedTypeBuildItem(
            ReactiveKeyRotationService.class.getName()));
        excluded.produce(new ExcludedTypeBuildItem(
            ReactiveLedgerVerificationService.class.getName()));
    }
}
```

**In `application.properties`** (or test resources):
```properties
# Reactive-enabled consumers and test suites
casehub.ledger.reactive.enabled=true

# JDBC-only consumers: leave unset (defaults to false)
# → ExcludedTypeBuildItem removes beans at augmentation
# → No unsatisfied dependency failure
```

**Also declare the property in the runtime `@ConfigMapping`** to suppress `SRCFG00050` (see related gotcha GE-20260519-098413). Use a warning Javadoc that the runtime accessor is not the authoritative gate.

The optional beans carry no gating annotation — exclusion is entirely the deployment module's concern. This is the canonical Quarkus extension pattern; `@IfBuildProperty` on runtime beans is for application beans, not extension-provided beans.

### Why this is non-obvious

Most Quarkus documentation and tutorials show `@IfBuildProperty` as the conditional bean mechanism. But for extension-provided beans, the correct pattern is `ExcludedTypeBuildItem` from the deployment `@BuildStep`. The distinction — `@IfBuildProperty` is for app beans, `ExcludedTypeBuildItem` is for extension beans — is not prominently documented. The failure mode (beans appear or disappear unexpectedly, or `SRCFG00050` at startup) is subtle enough that developers often reach for `Instance<T>` workarounds instead.
