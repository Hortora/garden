## @Transactional(REQUIRES_NEW) try-catch doesn't intercept commit-time exceptions — they escape as ArcUndeclaredThrowableException

### Stack
Quarkus 3.32.2, Narayana JTA, CDI 4.0, Hibernate ORM

### Symptom
A `@Transactional(TxType.REQUIRES_NEW)` method wraps its body in a try-catch to swallow all exceptions (pattern: audit writers that must not fail the caller). Despite the catch block, the caller receives an `ArcUndeclaredThrowableException` — the exception was never caught.

```
io.quarkus.arc.ArcUndeclaredThrowableException: Error invoking subclass method
    at BlockedAttemptAuditService_Subclass.record(Unknown Source)
    at WorkItemService.create(WorkItemService.java:151)
Caused by: org.h2.jdbc.JdbcSQLIntegrityConstraintViolationException:
    Referential integrity constraint violation: "FK_AUDIT_ENTRY_WORK_ITEM ..."
```

The `BlockedAttemptAuditService.record()` method looked like this — the try-catch
appeared to cover everything:

```java
@Transactional(TxType.REQUIRES_NEW)
public void record(UUID workItemId, String event, String actor, String detail) {
    try {
        AuditEntry entry = new AuditEntry();
        entry.workItemId = workItemId;
        entry.event = event;
        auditStore.append(entry);  // em.persist() — deferred!
    } catch (Exception e) {
        LOG.warning("audit write failed: " + e.getMessage());
    }
}
```

### Context
Occurs when:
- The method uses `@Transactional(REQUIRES_NEW)` (Narayana intercepts at commit boundary)
- The persistence call inside the try-catch is deferred (JPA `em.persist()` queues the INSERT; the SQL runs at commit, not immediately)
- A constraint violation (FK, UNIQUE, NOT NULL) is triggered at commit time

### What was tried
Assumed the try-catch wrapping `auditStore.append(entry)` would catch any failure from that call,
including database errors. This works for application-level exceptions thrown synchronously during
`append()`, but not for SQL exceptions at commit.

### Root cause
JPA's `EntityManager.persist()` does not immediately execute SQL — it registers the entity for
insertion. The actual INSERT runs when the transaction commits, which happens AFTER the method
body returns. At that point:
1. The method body's try-catch has already exited normally
2. Narayana's `TransactionalInterceptorRequiresNew` calls commit
3. The FK violation throws during commit
4. The CDI subclass proxy wraps the unhandled exception as `ArcUndeclaredThrowableException`
5. This propagates to the caller outside the try-catch entirely

### Fix
**Option A (structural — preferred):** Remove the constraint that causes the commit-time failure.
In the audit case: drop the FK constraint so orphaned audit entries are permitted. This is semantically
correct when the audit records an *attempt* rather than an entity lifecycle event.

**Option B (force flush inside try-catch):** Call `em.flush()` before the try-catch exits. This
forces the SQL to execute synchronously within the method body, where exceptions are catchable:

```java
@Inject EntityManager em;

@Transactional(TxType.REQUIRES_NEW)
public void record(...) {
    try {
        AuditEntry entry = new AuditEntry();
        ...
        auditStore.append(entry);
        em.flush();  // forces SQL now — exception catchable here
    } catch (Exception e) {
        LOG.warning("audit write failed");
    }
}
```

**Option C (declare checked exception in signature):** Not applicable — `ArcUndeclaredThrowableException`
is a runtime exception wrapping any throwable, not a checked exception issue.

### Why this is non-obvious
The try-catch looks complete — it catches `Exception`, which covers all runtime exceptions.
The surprise is that the exception doesn't originate from within the method body at all: it comes
from Narayana's commit phase, which runs inside the CDI proxy's interceptor chain, outside any
user-written code. Nothing in the Quarkus or Narayana documentation calls out this interaction
explicitly.
