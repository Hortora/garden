## Symptom

Using `computeIfAbsent` to cache external provider contexts (e.g. fetching a public key from Vault Transit on first use) causes unexpected latency spikes. Under concurrent load, a thread loading one actor's context can block threads loading a different actor's context — even though the keys are completely unrelated.

In extreme cases with synchronised re-entry (the mapping function internally calls back into the map for a different key that hashes to the same bucket), the JVM may deadlock.

## Context

Per-actorId context cache in an `AgentSigner` implementation. First `sign()` call per actor triggers `loadContext(actorId)` which makes an HTTP call to Vault. Subsequent calls should be instant (cache hit).

```java
// WRONG — computeIfAbsent holds the bucket lock during the HTTP call
Optional<C> cached = contextCache.computeIfAbsent(actorId, id -> {
    return loadContext(id);  // HTTP call — may take hundreds of ms
});
```

## Root cause

`ConcurrentHashMap.computeIfAbsent` acquires an internal per-bucket lock (striped lock on the bin node) for the duration of the mapping function. While one thread is inside a slow mapping function:
- **All other threads needing any key in the same hash bucket are blocked** — not just the same key
- A second thread calling `computeIfAbsent` for a different key that happens to share the same hash bucket will block until the first completes
- If the mapping function itself calls back into the same map (re-entrant), the JDK **throws `IllegalStateException: Recursive update`** in Java 9+

The Javadoc says "the computation should be short and simple" but gives no hint that slow computations block an entire hash bucket.

## Fix

Use `get()` / `putIfAbsent()` instead. Accept duplicate loads on cold start (two threads for the same key both call `loadContext` — idempotent results are safe):

```java
// CORRECT — no bucket lock held during the network call
Optional<C> cached = contextCache.get(actorId);
if (cached == null) {
    Optional<C> loaded = loadContext(actorId);  // HTTP call, no lock held
    Optional<C> racing = contextCache.putIfAbsent(actorId, loaded);
    cached = racing != null ? racing : loaded;
}
```

`putIfAbsent` is atomic but only holds the lock for the map update itself (microseconds), not for the compute function. The trade-off: on cold start, two threads may both call `loadContext` for the same key. The second's result is discarded (the first writer wins). For idempotent loads (same Vault key → same result), this is safe.

## Why this is non-obvious

The name `computeIfAbsent` implies "compute and store if the key is missing" — developers naturally assume the lock scope is narrow (just the map update). The Javadoc recommends "short and simple" but doesn't say "or you block the entire hash bucket." The symptom (unrelated keys blocking each other) is completely surprising. Developers discovering the issue often suspect their HTTP client or Vault server, not the cache implementation.

`computeIfAbsent` is the correct and safe choice when the mapping function is fast and in-memory. The problem is specifically with slow or blocking mapping functions.

### Why this fix

`putIfAbsent` releases the lock before the network call even starts. The duplicate-load race is the accepted trade-off: two threads load the same key concurrently, one result is discarded. For caches where loading is idempotent and deterministic (fetching a public key from a KMS), this is always safe and preferable to blocking the bucket.
