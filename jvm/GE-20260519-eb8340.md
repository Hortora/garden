## Symptom

Code injects `Instance<T>`, calls `.handles()`, then tries to assign or pass the result to `Iterable<InstanceHandle<T>>`:

```java
@Inject Instance<MessageObserver> observers;

// Somewhere:
Iterable<InstanceHandle<MessageObserver>> handles = observers.handles(); // COMPILE ERROR
```

Or a method parameter typed as `Iterable<InstanceHandle<T>>` is called with `observers.handles()`:

```java
void dispatch(Iterable<InstanceHandle<MessageObserver>> handles) { ... }
dispatch(observers.handles()); // COMPILE ERROR
```

Compiler error: `incompatible types: Iterable<capture#1 of ? extends Instance.Handle<T>> cannot be converted to Iterable<InstanceHandle<T>>`.

## Context

Occurs when refactoring code to use `Instance.handles()` for proper `@Dependent` lifecycle management. `InstanceHandle<T>` (Quarkus Arc) and `Instance.Handle<T>` (CDI standard) are related but distinct — `InstanceHandle<T>` extends `Instance.Handle<T>`, but Java's invariant generics mean `Iterable<InstanceHandle<T>>` is NOT assignable from `Iterable<? extends Instance.Handle<T>>`.

## Root Cause

`Instance.handles()` is declared to return `Iterable<? extends Instance.Handle<T>>` (CDI standard type with upper-bounded wildcard). Even though Quarkus Arc implements this with `InstanceHandle<T>` objects at runtime, the static type is the CDI wildcard form.

Because Java generics are invariant, `Iterable<InstanceHandle<T>>` (no wildcard) is not a supertype of `Iterable<? extends Instance.Handle<T>>`. The assignment fails at compile time.

## Fix

Use the upper-bounded wildcard in the receiving type/parameter:

```java
// Correct parameter type:
static void dispatch(Iterable<? extends Instance.Handle<MessageObserver>> handles) { ... }

// Or assign with wildcard:
Iterable<? extends Instance.Handle<MessageObserver>> handles = observers.handles();
```

For test anonymous classes implementing `InstanceHandle<T>`, the wildcard still accepts them because `InstanceHandle<T>` IS-A `? extends Instance.Handle<T>`.

Prefer `Instance.Handle<T>` (CDI standard) over `InstanceHandle<T>` (Arc-specific) in method signatures — it works across both the production CDI context and test anonymous implementations.

## Why This Is Non-Obvious

The developer sees `InstanceHandle<T>` objects in the debugger (Arc's concrete type), and `InstanceHandle<T>` extends `Instance.Handle<T>`, so the assignment "obviously should work". The wildcard is invisible — `handles()` appears to return a simple iterable. The Java generics variance rule (invariance by default) only surfaces at compile time with a cryptic type-inference error.
