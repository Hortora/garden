### Symptom
`handleSemanticFailure()` in `WorkflowExecutionCompletedHandler` never fires despite the worker function correctly returning `WorkerResult.declined("reason")`. The case stays RUNNING — no error, no warning, no log from the semantic failure path.

### Context
`WorkerResult` is a Java record with three fields: `(output, plannedAction, outcome)`. The `outcome` field was added as part of the failure cascade feature. Convenience factories `WorkerResult.of(output)` and `WorkerResult.of(output, plannedAction)` default `outcome` to `WorkerOutcome.success()`.

`DefaultWorkerExecutor.applyOutputSchema()` evaluates JQ output schema expressions and reconstructs the result:
```java
return WorkerResult.of(evaluated, result.plannedAction());
```

### Root cause
`WorkerResult.of(evaluated, result.plannedAction())` creates a new `WorkerResult` with `outcome = WorkerOutcome.success()` — silently replacing the original `Declined` or `Failed` outcome. The convenience factory has no parameter for `outcome` and defaults it.

### Fix
Use the canonical record constructor to preserve all fields:
```java
return new WorkerResult(evaluated, result.plannedAction(), result.outcome());
```

### Why this is non-obvious
The convenience factory does exactly what it's designed to do — it's the *correct* call for the success path. The trap is that adding a new field with a default to a record doesn't break any existing factory call sites at compile time. Every `WorkerResult.of(...)` call silently becomes an outcome-erasing call. The bug is invisible until a non-default outcome flows through a code path that uses the factory.

### Alternatives considered
- Making `outcome` non-nullable with no default → breaks every existing `WorkerResult.of()` call site. Correct but high migration cost. Chosen approach preserves backward compat for the common (success) case.
