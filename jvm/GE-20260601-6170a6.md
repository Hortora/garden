## What happened

A Java record had a new field inserted as the 2nd parameter (e.g. `String tenancyId`).
All existing call sites that passed the previous 2nd argument (e.g. `null` for `commandType`)
now passed `null` for `tenancyId` instead. The code compiled cleanly. At runtime, repository
methods calling `tenancyId.equals(...)` threw `NullPointerException`.

## Why this is non-obvious

Java records use positional construction — the canonical constructor mirrors field declaration
order. Unlike named parameters (Kotlin, Python), there is no syntax to catch that a `null`
intended for "commandType" now lands in "tenancyId". Since both are `String`, the type check
passes. The compiler has no way to know that a `null` that was formerly acceptable is now
unacceptable in its new position. The failure is a runtime NPE in unrelated code (the
repository layer) with no diagnostic pointing to the construction site.

## Root cause

Record canonical constructors are positionally ordered. Inserting a field shifts the meaning
of every positional argument from that point onward. Callers with the same arity (same number
of arguments) do not require recompilation — they just silently pass the wrong values.

This is the record equivalent of a C function signature change: `void foo(int a, int b)`
→ `void foo(int a, int c, int b)` — callers with two ints still compile; they just got
the roles of `b` and `c` swapped.

## Fix

After inserting a record field, **search all construction sites** (`new RecordType(...)`)
and verify each positional argument semantically. Specifically:

1. `grep -rn "new RecordType(" src/` to find all call sites
2. For each call site, verify the arguments against the new field order
3. Update `null` placeholders and literal values to match the new semantics

In tests that pass synthesized events (e.g. CDI event stubs), pay particular attention
to `null` arguments — a `null` that was a "don't care" for one field may be an NPE trigger
for another.

```java
// Before (7-arg record):
new CaseLifecycleEvent(caseId, "CompleteCase", "CaseCompleted", "COMPLETED", null, "System", null)
//                             ^^^^ commandType                              ^^^^ actorRole  ^^^^ traceId

// After (8-arg record with tenancyId inserted as 2nd field):
new CaseLifecycleEvent(caseId, null, "CompleteCase", "CaseCompleted", "COMPLETED", null, "System", null)
//                             ^^^^ WRONG: tenancyId is now null; "CompleteCase" moved to commandType slot
```

## Constraints

Specific to Java records (and any positionally-constructed value type). Does not apply to
JavaBeans/builder patterns where fields are set by name. More dangerous when the record has
multiple `String`/`Object` fields because type-checking provides no safety net.

## Invalidation triggers

Would be partially mitigated if Java adds named constructor parameters (currently a
preview-track feature request). In Kotlin, data classes with named arguments and copy()
make this safe by default.
