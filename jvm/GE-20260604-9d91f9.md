When a Quarkus extension creates synthetic CDI beans for interfaces (via
`SyntheticBeanBuildItem`), Arc resolves interceptor bindings only from annotations directly
present on the interface in the Jandex index. Bindings declared on *parent* interfaces
(class-level or method-level) are invisible to Arc's resolver. The fix is to use
`AnnotationsTransformerBuildItem` to propagate the inherited binding annotations into the
child interface's Jandex index entry before Arc processes beans.

### Context

Any Quarkus extension that wraps user-defined interfaces as `SyntheticBeanBuildItem`
beans — agents, AI services, custom service interfaces — will silently miss interceptor
bindings (e.g., `@Timeout`, `@Retry`) if those bindings are declared on parent interfaces
rather than on the concrete bean interface itself.

### Fix

Produce an `AnnotationsTransformerBuildItem` per agent interface that adds inherited
interceptor binding annotations to the child interface in the Jandex index:

```java
@BuildStep
void propagateParentInterceptorBindings(
        List<DetectedBeanBuildItem> beans,
        CombinedIndexBuildItem indexBuildItem,
        BuildProducer<AnnotationsTransformerBuildItem> transformerProducer) {

    DotName INTERCEPTOR_BINDING =
            DotName.createSimple("jakarta.interceptor.InterceptorBinding");
    IndexView index = indexBuildItem.getIndex();

    for (DetectedBeanBuildItem bean : beans) {
        ClassInfo beanIface = bean.getIface();
        List<AnnotationInstance> toPropagate = new ArrayList<>();

        // Walk the full interface hierarchy (BFS)
        for (ClassInfo parent : transitiveInterfaces(beanIface, index)) {
            if (parent.name().equals(beanIface.name())) continue;
            for (AnnotationInstance ann : parent.declaredAnnotations()) {
                if (beanIface.hasAnnotation(ann.name())) continue;
                ClassInfo annClass = index.getClassByName(ann.name());
                if (annClass != null && annClass.hasAnnotation(INTERCEPTOR_BINDING)) {
                    toPropagate.add(ann);
                }
            }
        }

        if (toPropagate.isEmpty()) continue;

        DotName beanName = beanIface.name();
        List<AnnotationInstance> annotations = List.copyOf(toPropagate);
        transformerProducer.produce(new AnnotationsTransformerBuildItem(
                AnnotationTransformation.forClasses()
                        .whenClass(beanName)
                        .transform(ctx -> ctx.addAll(annotations))));
    }
}
```

`transitiveInterfaces` is a BFS helper that returns the interface and all parent interfaces
as `Set<ClassInfo>`.

**Important:** also ensure `hasAnyInterceptorBindings` (or equivalent) walks the same
hierarchy, so `injectInterceptionProxy()` is called on the synthetic bean configurator.
Both halves are required: detection (so the proxy is wired) and propagation (so Arc
applies the correct interceptors).

### Why this is non-obvious

Developers expect to call `beanConfigurator.addInterceptorBinding(ann)` — that API does
not exist on `SyntheticBeanBuildItem.ExtendedBeanConfigurator`. The only documented path
is `injectInterceptionProxy()` which signals that a proxy is needed, but doesn't tell Arc
*which* interceptors to apply. Arc resolves the interceptors from the Jandex index at
bean processing time, making the index the correct target for the annotation propagation,
not the bean configurator. This requires `AnnotationsTransformerBuildItem`, which most
extension developers associate with code analysis rather than fixing runtime interceptor
resolution.

### Why this fix

`SyntheticBeanBuildItem` has no `addInterceptorBinding()` API. The only other approach
is to require the user to redeclare the interceptor binding on their own interface — a
poor DX that breaks the inheritance contract. Modifying the Jandex index via
`AnnotationsTransformerBuildItem` is the correct hook because Arc consumes the index
for all annotation-driven decisions.

### Alternatives considered

- Requiring users to redeclare `@Timeout` (or other binding) on their concrete agent
  interface — bad DX, breaks standard Java interface inheritance expectations.
- Producing an `UnremovableBeanBuildItem` — doesn't affect interceptor resolution.
- Using `beanConfigurator.addQualifier(...)` — wrong API, qualifiers ≠ interceptor bindings.
