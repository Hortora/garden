### Symptom

`embeddingModel.embedAll()` throws `ClientWebApplicationException: Received: 'Bad Request, status code 400'` when called via the LangChain4j Quarkus Ollama integration. The 400 response body is `{"error": "the input length exceeds the context length"}` but the LangChain4j exception only surfaces the HTTP status code, not the Ollama error message.

### Context

A corpus ingestion pipeline sends document text to Ollama for embedding via `nomic-embed-text` (2048 token context window). Most documents are small (50-200 lines) and embed fine. Large reference documents (e.g. a testing principles file at ~8000 tokens) exceed the model's context window and Ollama rejects the request outright.

### Root cause

Ollama's `/api/embed` endpoint does not truncate input to the model's context length. If the tokenised input exceeds the model's `num_ctx`, the request fails with 400. There is no partial embedding, no warning, and no automatic chunking. The caller must ensure all input fits within the model's token limit.

### Fix

Split large documents before embedding. Use a recursive document splitter with a max chunk size safely below the model's context window (in characters, not tokens — a conservative 4:1 char-to-token ratio):

```properties
# For nomic-embed-text (2048 tokens ≈ 8000 chars), use 6000 for safety margin
casehub.rag.ingestion.corpora.garden.chunking=recursive
casehub.rag.ingestion.corpora.garden.chunking-max-size=6000
casehub.rag.ingestion.corpora.garden.chunking-overlap-size=500
```

### Why this is non-obvious

The error surfaces as a generic HTTP 400 in the LangChain4j REST client exception. The actual Ollama error message ("input length exceeds the context length") is not propagated — the developer sees only "Bad Request, status code 400" with no indication it's a content-length issue. In a batch ingestion of thousands of documents, the first oversized document fails and the entire batch is lost, with no indication which document caused the failure.
