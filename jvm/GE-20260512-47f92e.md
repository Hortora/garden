## quarkus-junit5 is a relocation stub since Quarkus 3.31 — quarkus-junit is the real artifact

### Symptom

Using `quarkus-junit5` in `pom.xml` still works (it's a relocation), but produces a build warning:
```
[WARNING] The artifact io.quarkus:quarkus-junit5:jar has been relocated to io.quarkus:quarkus-junit:jar
```
Using `quarkus-junit` is correct and produces no warning.

### Context

Since Quarkus Migration Guide 3.31, `quarkus-junit5` was renamed to `quarkus-junit`. The old artifact became a Maven relocation stub in `quarkus-relocations-parent` that transparently redirects to the new name.

### Root cause

Maven relocation means `quarkus-junit5` resolves and compiles — it just adds a deprecation warning and adds an extra resolution hop. It is NOT a stub that fails at runtime; it works correctly but is the wrong artifact to declare.

Confirmed by inspecting POM files in `~/.m2`:
- `quarkus-junit5-3.32.2.pom` → `quarkus-relocations-parent`, contains `<relocation>` to `quarkus-junit`
- `quarkus-junit-3.32.2.pom` → `quarkus-test-framework`, contains real dependencies (`quarkus-bootstrap-core`, `quarkus-test-common`, `junit-jupiter`)

### Fix

Use `quarkus-junit` in `pom.xml`:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-junit</artifactId>
    <scope>test</scope>
</dependency>
```

### Why this is non-obvious

Both artifact IDs exist, both resolve, both compile, both work. The only signal is a build warning that is easy to miss in noisy Maven output. The rename is documented in Migration Guide 3.31, but the POM-level distinction (real artifact vs relocation stub) is only visible by reading `.m2` cache files. AI assistants and older documentation frequently suggest `quarkus-junit5` since it was correct before 3.31.
