### What exists
`ai.djl.api-0.36.0.jar` contains `META-INF/native-image/ai/djl/api/native-image.properties` with:
- `-H:ResourceConfigurationResources` — registers some DJL resource files
- `-H:ReflectionConfigurationResources` — registers some DJL classes for reflection

This metadata is automatically picked up by GraalVM's native-image during compilation.

### What's missing
The metadata covers the DJL API module only. The tokenizers module (`ai.djl.huggingface:tokenizers`) requires additional entries not included:

- `ai.djl.huggingface.tokenizers.jni.CharSpan` — JNI constructor registration (Rust→Java callback)
- `ai.djl.huggingface.tokenizers.jni.CharSpan[]` — JNI array type
- `com.sun.jna.*` classes — JNA bridge used by DJL for native library loading (dozens of entries)
- `native/lib/tokenizers.properties` — resource registration for version detection
- `native/lib/osx-aarch64/cpu/libtokenizers.dylib` — native library resource
- `com/sun/jna/darwin-aarch64/libjnidispatch.jnilib` — JNA dispatch native library

### Why this matters
A developer seeing DJL's `native-image.properties` in the build output might assume native-image support is complete. In practice, the tokenizers module requires substantial additional configuration discovered only via the tracing agent. The partial metadata gives a false sense of coverage.

### How it was discovered
During a Quarkus native image build, the GraalVM output showed DJL's metadata being loaded, yet the native binary crashed on tokenizer initialization. The tracing agent revealed the full set of required entries — roughly 20x more than what DJL ships.
