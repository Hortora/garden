## RESTEasy Reactive endpoints that call .await() on the IO thread throw BlockingOperationNotAllowedException ŌĆö add @Blocking

### Symptom

A RESTEasy Reactive endpoint that unwraps a reactive result with `.await().atMost(TIMEOUT)` works fine in tests (low concurrency, small H2 responses) but throws `BlockingOperationNotAllowedException` in production under load. The exception is thrown inside the Vert.x IO thread guard.

### Context

This surfaces when switching from a blocking API (returning `List<T>` directly) to a reactive API (returning `Uni<List<T>>`), and adapting by calling `.await()`:

```java
// quarkus-rest-jackson (RESTEasy Reactive) ŌĆö IO thread by default
@GET
@Path("/channels")
public List<ChannelDetail> channels() {
    return reactiveService.listChannels().await().atMost(Duration.ofSeconds(10));  // ŌåÉ blocks IO thread
}
```

The endpoint returns `List<T>` (not `Uni<T>` or `Multi<T>`), so RESTEasy Reactive dispatches it on the Vert.x IO thread. The `.await()` call then blocks that IO thread, which Vert.x prohibits.

### Root cause

RESTEasy Reactive runs endpoint methods on the Vert.x IO event loop by default. Blocking the event loop ŌĆö even for a short duration ŌĆö starves all other I/O operations. Vert.x detects this and throws `BlockingOperationNotAllowedException`.

The original blocking API (e.g. `QhorusMcpTools.listChannels()` returning `List<T>`) had the same structural problem but may not have surfaced it depending on the CDI dispatch context. Switching to a reactive API and calling `.await()` makes the blocking explicit and the guard fires.

### Fix

**Short-term:** Annotate blocking endpoints with `@Blocking` so RESTEasy Reactive dispatches them to a worker thread:

```java
@GET
@Path("/channels")
@Blocking   // io.smallrye.common.annotation.Blocking
public List<ChannelDetail> channels() {
    return reactiveService.listChannels().await().atMost(Duration.ofSeconds(10));
}
```

**Proper fix:** Return `Uni<T>` directly and let RESTEasy Reactive subscribe on the IO thread without blocking:

```java
@GET
@Path("/channels")
public Uni<List<ChannelDetail>> channels() {
    return reactiveService.listChannels();
}
```

### Why this is non-obvious

- Classic `quarkus-resteasy` dispatches all endpoints to worker threads ŌĆö `.await()` works there without annotation.
- The endpoint compiles, starts, and passes unit/integration tests because test HTTP clients are synchronous and concurrency is low.
- The `BlockingOperationNotAllowedException` only surfaces under real Vert.x IO thread dispatch ŌĆö which requires `quarkus-rest` or `quarkus-resteasy-reactive` and actual concurrent load.
- The error message points to the `.await()` call site, but the root cause is the missing `@Blocking` or return-type change on the endpoint.
