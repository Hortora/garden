### Symptom
Builds pass for multiple sessions using `mvn test -pl eval`. A sealed interface
in a dependency (`AgentEvent`) gains new permitted subtypes. No compile error
appears until a full-reactor build (`mvn compile test-compile` without `-pl`).

### Context
Multi-module Maven project. A switch expression in test source covers only one
variant of a sealed interface. The sealed interface lives in a different module
that is not the `-pl` target.

### Root cause
`mvn test -pl <module>` compiles main and test sources of the targeted module only.
Test sources in non-targeted submodules are never compiled. When the sealed interface
gains a new permitted subtype, Java 21 exhaustiveness checking would catch the
incomplete switch — but only if that test source is actually compiled.

### Fix
Run a full-reactor build periodically — `mvn compile test-compile` from the root
without `-pl` — to catch cross-module test-source staleness. Alternatively, add a
`default` branch to switch expressions over sealed interfaces in test code to avoid
the exhaustiveness requirement.

### Why this is non-obvious
`-pl` is the standard way to scope builds for speed. The mental model is "I'm only
changing this module, so only this module needs rebuilding." But sealed interface
exhaustiveness is a cross-module contract — the compiler enforces it only when it
actually compiles the file, and `-pl` silently skips test-compile on siblings.
