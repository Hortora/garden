## Symptom

After adding `quarkus-langchain4j-ollama` as a test-scope dependency inside a Maven profile
(`-Peval-ollama`), the Ollama `ChatModel` bean is not registered. Config keys are logged as
`Unrecognized configuration key "quarkus.langchain4j.ollama.chat-model.model-name"`, and the
test fails with `UnsatisfiedResolutionException` for `ChatModel`.

## Context

The module was previously built without the Ollama extension. Maven resolves the Ollama JAR
when the profile is active, but Quarkus reuses its disk-based augmentation cache from the
prior build — before the Ollama extension was on the classpath. The Ollama extension is
silently absent from augmentation: no beans registered, no config processors active.

## What was tried (didn't work)

- Running `mvn test -Peval,eval-ollama` after the JAR appeared in `~/.m2` — stale augmentation
  cache persisted, config keys still unrecognized.
- Adding `-Dquarkus.test.profile=eval` explicitly — no effect on cache.

## Root cause

Quarkus augmentation is disk-cached in `target/quarkus-app/`. When classpath changes occur
(new extension added via Maven profile), the cache invalidation heuristic does not always
detect the change, especially when the Ollama JAR was already present in `~/.m2` from a
prior `mvn dependency:resolve` call (updating file timestamps). The cache is reused, and
the Ollama extension's CDI beans are never registered.

## Fix

Run `mvn clean` before the first test run with the new profile:

```bash
JAVA_HOME=$(/usr/libexec/java_home -v 26) mvn clean test -pl eval -Peval,eval-ollama \
  -Dtest=PromptEvalTest#evaluateRealWorldScenarios \
  -Dcasehub.eval.claude-provider.enabled=false \
  "-Dquarkus.langchain4j.ollama.chat-model.model-name=qwen3:8b" \
  "-Dquarkus.langchain4j.ollama.chat-model.format=json" \
  "-Dquarkus.langchain4j.ollama.timeout=300s"
```

Subsequent runs without `clean` work correctly — the Ollama-aware augmentation is now cached.

## Why this is non-obvious

The "unrecognized configuration key" warning does not hint at augmentation caching — it reads
as though the extension was not added at all, not that a stale build artifact is in play.
The Ollama JAR is visible in `mvn dependency:resolve` output, making the missing registration
doubly confusing. The standard fix for missing CDI beans (`check your imports, check your
annotations`) leads nowhere.
