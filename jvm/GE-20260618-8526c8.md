## The technique

When implementing a test `ChatModel` without mocks (because `AiMessage`/`ChatResponse` have final methods), override `doChat(ChatRequest)` — not `chat(ChatRequest)`:

```java
ChatModel testModel = new ChatModel() {
    @Override
    public ChatResponse doChat(final ChatRequest request) {
        // inspect request, return appropriate response
        return ChatResponse.builder()
                .aiMessage(new AiMessage("{...json...}"))
                .build();
    }
};
```

`chat(ChatRequest)` is a **default method** in the `ChatModel` interface that calls `chat(ChatRequest, ChatRequestOptions)`, which then calls `doChat(ChatRequest)`. The chain applies listeners, merges request parameters, and fires `onRequest`/`onResponse` hooks. By overriding `doChat()`, the test skips the listener chain and returns directly.

`casehub-engine-api`'s `Agent.execute()` calls `model.chat(request)` which eventually reaches `doChat()`.

## Why this is non-obvious

`chat(ChatRequest)` and `chat(ChatRequest, ChatRequestOptions)` are the visible API methods that developers see in IDE autocompletion. `doChat(ChatRequest)` is buried as an implementation detail — a `default` method with a `throw new RuntimeException("Not implemented")` body that exists specifically as the override target. The JavaDoc doesn't prominently surface this as the intended extension point for test doubles.
