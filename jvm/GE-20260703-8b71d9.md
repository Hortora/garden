### Symptom
Relay proxy validates the target URL (blocking private IPs, loopback, link-local) before
fetching — but an attacker-controlled upstream returns a 302 redirect to `http://169.254.169.254/latest/meta-data/` or `http://127.0.0.1:8080/admin`, and the request silently succeeds.

### Context
Any HTTP relay/proxy built with `java.net.http.HttpClient` that validates the URL before
calling `httpClient.send()`. The redirect policy is set at client construction time and
applies to all requests.

### What was tried (didn't work)
`validateTarget(url)` called before `fetch()` — checks scheme (http/https only), host
allowlist, private IP blocking via `InetAddress.getByName()`. Passes. Then `httpClient.send()`
follows the 302 internally — `validateTarget()` is never called on the redirect target.

### Root cause
`HttpClient.Redirect.NORMAL` follows redirects inside the HTTP client layer, after application-level validation has already run. There is no callback or interceptor to validate intermediate redirect targets.

### Fix
```java
httpClient = HttpClient.newBuilder()
    .connectTimeout(Duration.ofMillis(connectTimeoutMs))
    .followRedirects(HttpClient.Redirect.NEVER)  // was NORMAL
    .build();
```
If redirects are legitimately needed, manually follow each hop: check the `Location` header, call `validateTarget()` on it, then issue a new request.

### Why this is non-obvious
`HttpClient.Redirect.NORMAL` is the "obvious" and commonly recommended setting. The Javadoc describes it as following redirects "in a normal way" — nothing suggests it's a security concern. The validation code looks complete: it checks scheme, host, and IP class. The bypass only surfaces under adversarial conditions (attacker-controlled upstream) that don't occur in development.
