### Symptom

`flyway_scan.py` returns V5003 as the next safe migration number. Developer creates `V5003__my_migration.sql` in the runtime module — but V5000–V5999 belongs to `casehub-work-issue-tracker`, not runtime.

### Context

Multi-module Maven project where each module owns a Flyway version range (documented in `docs/FLYWAY.md`): runtime V1–V999, queues V2000–V2999, notifications V3000–V3999, ai V4000–V4999, issue-tracker V5000–V5999. The `flyway_scan.py` tool scans ALL migration files across all modules and returns the next globally collision-free V-number.

### What was tried (didn't work)

Trusted `flyway_scan.py` output directly — it said V5003 so the migration was drafted as V5003. Caught during code review when the reviewer checked `docs/FLYWAY.md` and found the runtime module owns V1–V999 (currently at V35). The correct next V was V36.

### Root cause

`flyway_scan.py` prevents inter-module V-number collisions but doesn't know which module the caller is targeting. It returns `max(all V across all modules) + 1`, which may fall in a different module's reserved range.

### Fix

Always cross-check the `flyway_scan.py` output against `docs/FLYWAY.md` for the target module's version range. The runtime module's next V is `max(existing runtime Vs) + 1`, not the global max + 1.

### Why this is non-obvious

The tool's output looks authoritative — "next safe V is 5003" — and is technically correct for global collision avoidance. The per-module range ownership is a project convention documented in a separate file (`docs/FLYWAY.md`), not enforced or surfaced by the tool itself.
