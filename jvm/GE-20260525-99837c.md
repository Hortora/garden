## What

When you declare a `UNIQUE` constraint in DDL or a Flyway migration, PostgreSQL automatically creates a B-tree index to enforce it. That implicit index is also usable by the query planner for equality lookups and left-anchored prefix queries (`LIKE 'prefix%'`).

```sql
-- No separate CREATE INDEX needed for query performance on name
CREATE TABLE channel (
    id   UUID         NOT NULL,
    name VARCHAR(255) NOT NULL,
    CONSTRAINT uq_channel_name UNIQUE (name)  -- implicit B-tree index created here
);

-- PostgreSQL uses uq_channel_name for this query:
SELECT * FROM channel WHERE name LIKE 'case/%'
```

A Flyway migration that adds `CREATE INDEX idx_channel_name ON channel(name)` when `CONSTRAINT uq_channel_name UNIQUE (name)` already exists is redundant — two indexes enforcing the same thing.

## Why This Is Non-Obvious

Developers commonly know that UNIQUE constraints are enforced at the database level, but assume the enforcement mechanism is separate from a "query performance index." In practice they are the same B-tree index. The result is tickets filed to "add an index for performance" on a column that is already indexed via its UNIQUE constraint.

The PostgreSQL documentation mentions this behaviour ("PostgreSQL automatically creates a unique index when a unique constraint or primary key is defined") but doesn't explicitly connect it to query-plan optimisation in the same section — a reader focused on constraint enforcement rather than indexing will miss it.

H2 in `MODE=PostgreSQL` also creates a covering index for UNIQUE constraints, so tests pass in both environments.
