## Qdrant Java client async methods return Guava ListenableFuture — not CompletableFuture

### Symptom
Mockito `when(...).thenReturn(CompletableFuture.completedFuture(...))` won't compile
for `QdrantClient` async methods. The return type doesn't match.

### Context
Any test that mocks `QdrantClient` methods — `upsertAsync()`, `deleteAsync()`,
`queryAsync()`, `countAsync()`, `collectionExistsAsync()`, `createCollectionAsync()`.

### Root cause
The Qdrant Java client wraps gRPC stubs and returns Guava `ListenableFuture<T>`, not
`java.util.concurrent.CompletableFuture<T>`. This is not called out in the Qdrant
client documentation — the API looks like standard Java async but uses a different
future type.

### Fix
Use Guava's `Futures.immediateFuture()` and `Futures.immediateFailedFuture()`:

```java
import com.google.common.util.concurrent.Futures;

when(qdrantClient.upsertAsync(anyString(), anyList()))
    .thenReturn(Futures.immediateFuture(UpdateResult.getDefaultInstance()));

when(qdrantClient.upsertAsync(anyString(), anyList()))
    .thenReturn(Futures.immediateFailedFuture(new RuntimeException("Qdrant down")));
```

For `.get()` calls in production code, the exception handling is the same (`ExecutionException`,
`InterruptedException`) — only the mock setup differs.

### Alternative — Mutiny Uni conversion for reactive pipelines

**Approach:** Bridge ListenableFuture to Mutiny Uni via emitter + Futures.addCallback — no intermediate CompletableFuture allocation.

**Pros:** Stays on directExecutor (no thread hop), zero-copy bridging, works with Quarkus reactive pipelines.

**Cons:** Requires Guava Futures (already on classpath via Qdrant client).

```java
import com.google.common.util.concurrent.FutureCallback;
import com.google.common.util.concurrent.Futures;
import com.google.common.util.concurrent.ListenableFuture;
import com.google.common.util.concurrent.MoreExecutors;
import io.smallrye.mutiny.Uni;

private static <T> Uni<T> toUni(ListenableFuture<T> future) {
    return Uni.createFrom().<T>emitter(em ->
        Futures.addCallback(future, new FutureCallback<>() {
            @Override public void onSuccess(T result) { em.complete(result); }
            @Override public void onFailure(Throwable t) { em.fail(t); }
        }, MoreExecutors.directExecutor()));
}
```

**Why not `toCompletionStage()`:** `ListenableFuture` has no `toCompletionStage()` method.
`Uni.createFrom().completionStage()` — the natural first attempt — won't compile.
Guava provides `MoreFutures.toCompletableFuture()` in newer versions, but the emitter
pattern avoids the intermediate allocation entirely.

### Why this is non-obvious
The Qdrant client API looks like standard Java async (`xxxAsync()` method naming), and most
Java gRPC wrappers return `CompletableFuture`. The Guava dependency is transitive through
gRPC — it's not something you'd expect to need in your test code.
