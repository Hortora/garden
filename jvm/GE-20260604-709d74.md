When `AnnotationInstance` objects collected from a parent interface are added to a child
class via `TransformationContext.addAll(annotationsFromParent)` in an
`AnnotationsTransformerBuildItem`, the annotation instances retain their original
`target()` reference — pointing to the parent `ClassInfo`, not the child. The annotation
is visible on the child and CDI interception works correctly. But any code that inspects
`ann.target()` on the child's annotations will see a stale reference to the parent.

### Symptom

No runtime error or warning. The interception works. Stale target is only discovered
when iterating the child's annotations and calling `ann.target()` — the returned
`ClassInfo` is the wrong class.

### Context

Occurs when copying inherited annotations from parent interfaces down to child interfaces
to fix CDI interceptor binding resolution for synthetic beans. The annotations are
functionally correct for Arc's purposes (name and values are used, not the target), but
semantically incorrect for any tooling or diagnostic code that inspects targets.

### Fix

Rebuild annotation instances without a target using the builder API before propagating:

```java
for (AnnotationInstance ann : parentAnnotations) {
    AnnotationInstance clean = AnnotationInstance.builder(ann.name())
            .buildWithTarget(childClassInfo);
    ctx.add(clean);
}
```

Or, if the annotation has values:

```java
AnnotationInstance.builder(ann.name())
        .add(ann.values())
        .buildWithTarget(childClassInfo);
```

This creates a new instance bound to the child target.

### Why this is non-obvious

`ctx.addAll()` looks like it should adapt the annotations to the new target. Jandex
`AnnotationInstance` is immutable — it carries a fixed target from creation. The
`addAll()` API does not automatically re-bind to the transformation context's target.
The interception works correctly because Arc's interceptor resolver uses annotation
names and values, not the target; so the bug is invisible unless you specifically
inspect targets.
