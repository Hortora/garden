## Place fireAsync() before internal dispatch to decouple delivery paths in @Transactional methods

### Symptom

A `@Transactional` method fires a CDI event and also calls an internal dispatch (channel write, database write, queue push). When the internal dispatch throws — rate-limit rejection, policy violation, missing resource — the CDI event silently never fires. The external delivery path is dead with no error.

### Context

Any method that must make two independent deliveries inside a `@Transactional` boundary:
one via CDI `fireAsync()` (external observers, notification systems) and one via an internal
service call. The pattern appears in watchdog alerting, domain event publishing, and audit
side-channels.

### Root cause

Ordering. If `fireAsync()` is placed *after* the internal dispatch and dispatch throws, the
stack unwinds before `fireAsync()` is reached. The two paths are not independent — external
delivery is silently contingent on internal dispatch succeeding.

### Fix

Place `fireAsync()` as the first statement inside the method, before the internal dispatch:

```java
private void fireAlert(Watchdog w, String summary, AlertContext context, Instant now) {
    // External delivery FIRST — not contingent on internal channel success.
    // A rate-limited, paused, or policy-rejected dispatch must not suppress this.
    alertEvents.fireAsync(new WatchdogAlertEvent(
            w.id, w.targetName, w.notificationChannel, summary, now, context));

    // Internal dispatch SECOND — failure does not suppress the event already fired.
    Optional<Channel> ch = channelService.findByName(w.notificationChannel);
    if (ch.isEmpty()) {
        return;
    }
    messageService.dispatch(MessageDispatch.builder()
            .channelId(ch.get().id)
            .sender("system:watchdog")
            .type(MessageType.STATUS)
            .content(summary)
            .build());
}
```

The CDI event fires unconditionally (after the guard checks). The internal dispatch is the
conditional secondary path.

### Accepted trade-off

If the outer `@Transactional` rolls back after `fireAsync()` dispatches, the observer runs
for a state change that never committed — a ghost notification. The window is narrow in
practice (only a post-dispatch interceptor or infrastructure failure causes an outer rollback).
For alerting systems: accepted — false-positive alert is better than missed alert. For audit
or ledger systems: not acceptable — use a transactional outbox pattern instead.

### Why this is non-obvious

CDI events feel like side effects. By convention, developers put them at the *end* of a method,
after "the main work" is done. When the event IS the primary delivery path — not a side effect
— this ordering must invert. The distinction only surfaces when the two paths need to be
genuinely independent rather than sequential. Without this insight, a failing internal dispatch
silently kills external delivery with no error or log entry.
