## Symptom

Quarkus augmentation fails immediately with:

```
Failed to load steps from class io.quarkiverse.langchain4j.anthropic.deployment.AnthropicProcessor:
Run time configuration cannot be consumed in Build Steps,
use RuntimeValue<io.quarkiverse.langchain4j.anthropic.runtime.config.LangChain4jAnthropicConfig>
in a @Recorder constructor instead
```

No application code runs. The error appears before any Quarkus startup.

## Context

Adding `quarkus-langchain4j-anthropic 0.26.1` as a dependency to a Quarkus 3.33+ project. No changes to application code — the failure occurs at the Quarkus deployment phase.

## Root Cause

`AnthropicProcessor` in 0.26.1 directly injects `LangChain4jAnthropicConfig` into a `@BuildStep` method parameter. In Quarkus 3.33+, the augmentation API prohibits direct config injection in build steps — it must use `RuntimeValue<T>` via a `@Recorder`. This API change was introduced to enforce the build-time / runtime separation more strictly.

The error message describes a *user code* problem ("use RuntimeValue in a @Recorder constructor") but the actual problem is inside the library.

## Fix

Use `quarkus-langchain4j-anthropic 1.9.1` instead of 0.26.1. Version 1.9.1 was compiled against Quarkus 3.33.1 and uses `RuntimeValue<>` correctly:

```xml
<!-- parent pom.xml -->
<!-- 0.26.1 incompatible with Quarkus 3.33+; 1.9.1 built against 3.33.1, verified on 3.34.3 -->
<quarkus-langchain4j.version>1.9.1</quarkus-langchain4j.version>
```

```xml
<!-- module pom.xml -->
<dependency>
  <groupId>io.quarkiverse.langchain4j</groupId>
  <artifactId>quarkus-langchain4j-anthropic</artifactId>
  <version>${quarkus-langchain4j.version}</version>
</dependency>
```

## Why This Is Non-Obvious

"Run time configuration cannot be consumed in Build Steps" sounds like a developer mistake — putting `@Inject` in the wrong place, or misconfiguring a custom extension. Nothing in the error message mentions a version number or suggests the cause is inside a dependency. The version `0.26.1` carries no numeric hint that it is incompatible with Quarkus 3.33+. A developer reading the error would audit their own code first, then check their own extensions, before suspecting the LangChain4j library itself.
