### What happened

Adding a new field to a Java record — even as the last component — is a full breaking change to the canonical constructor. Every `new RecordName(...)` call in the codebase fails to compile until updated to supply the new component. There is no `@Deprecated` grace period, no overloading, no optional field — the canonical constructor is the only constructor and its signature changes atomically.

### Why this is non-obvious

Regular Java classes allow adding fields without breaking call sites (fields are private; constructors can be overloaded for backwards compatibility). Developers familiar with class-based evolution assume the same incremental migration path exists for records — it doesn't.

A record component is simultaneously a field, a getter, a constructor parameter, and a pattern-matching binding. Changing any of these requires recompiling every consumer.

### Impact

In a large multi-module Maven project, adding one component to a shared record (`PlanItemRecord`, `PlanItemSaveRequest`, `PlanExecutionContext`, `CaseLifecycleEvent`) broke all 6 downstream modules simultaneously. The cascade through test files alone required updating ~80 construction sites across ~30 test classes.

### Mitigation strategies

**If the change can be batched:** Treat the record change as a single atomic commit that also fixes all construction sites. Use IntelliJ's "Change Signature" refactoring or a build-and-fix cycle (`mvn install -DskipTests` → fix compile errors → repeat).

**If the record is a public API with external consumers:** Consider wrapping the record in a builder or keeping the old record and deprecating it while introducing a new one. Records are not well-suited for fields that evolve independently.

**If using as internal implementation detail:** Records are safe for values that are truly stable or that change only with full ownership of all call sites. For data that will gain fields over time, a regular class with a builder may be more maintainable.

### Verification

```java
// Before
public record PlanItemRecord(UUID caseId, String planItemId, PlanItemStatus status) {}

// After — breaks every: new PlanItemRecord(caseId, planItemId, status)
public record PlanItemRecord(UUID caseId, String planItemId, PlanItemStatus status, String tenancyId) {}
// Error: constructor PlanItemRecord(UUID, String, PlanItemStatus) cannot be applied
// required: UUID, String, PlanItemStatus, String
// found: UUID, String, PlanItemStatus
```
