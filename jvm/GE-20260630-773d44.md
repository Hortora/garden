### Symptom

After a successful trigger operation (e.g. case creation), the lock entity is immediately removed. A second JVM that was evaluating the same event concurrently inserts a fresh entity, acquires the lock, and fires a duplicate trigger.

### Context

Two JVMs receive the same event. Both evaluate concurrently. The lock mechanism uses a conditional `UPDATE ... SET locked = true WHERE locked = false` on an entity row — atomic, correct. The winning JVM fires the protected operation (e.g. creates a case), then removes the entity to "clean up."

### What was tried (didn't work)

Immediate removal after the protected operation:
```
JVM-A: save(insert) → claim(UPDATE WHERE false) → fire() → remove()
JVM-B: save(insert) → ConstraintViolation → RETRY → find() → ???
```

The assumption was that JVM-B's retry would always find the entity with `locked=true` and exit gracefully. This is true in the common case — JVM-B retries within milliseconds, while JVM-A is still in the `fire()` call.

### Root cause

JVM-B's retry can be delayed by a JVM GC pause (100ms+ is routine, 500ms+ is not uncommon). If JVM-A's `fire()` is fast and the entity is removed before JVM-B's retry executes:

```
JVM-A: save(1ms) → claim(1ms) → fire(1ms fast path) → remove(1ms)  [total: 4ms]
JVM-B: save→fail → [GC pause 100ms] → retry find → EMPTY → insert → claim → fire → DUPLICATE
```

JVM-B finds nothing, creates a fresh entity, acquires the lock on the fresh row, and fires the protected operation again. The lock was removed before the loser checked it.

### Fix

Do NOT remove the lock entity immediately. Instead:

1. Stamp a `lastTriggered` timestamp on the entity when the protected operation succeeds
2. A scheduled cleanup job removes lock entities where `locked = true AND lastTriggered < now - guardPeriod`
3. Set `guardPeriod` to a value orders of magnitude past the retry window (e.g. 60 seconds for a retry window of <5 seconds)

During the guard period, new requests that find the locked entity correctly fail the lock check and exit — which is the correct behavior (the protected operation already completed).

### Why this is non-obvious

The race window is extremely narrow (milliseconds between entity removal and a delayed retry). In most testing — including load testing — it never fires because GC pauses don't happen at the exact right moment. It manifests only in production under sustained load where GC pauses are frequent. The immediate-removal approach passes every test and "looks correct" when you reason about sequential timing.
