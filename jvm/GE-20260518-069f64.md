## Calling @Transactional method via `this` inside Mutiny lambda silently bypasses CDI proxy — use CDI self-injection

### Symptom

A `@Transactional` method works correctly when called directly from a CDI managed thread. When the same method is called from within a `Uni.createFrom().item(() -> this.blockingMethod())` lambda, transactions are silently absent — JPA queries run without a transaction context, lazy loading fails, and writes are never flushed.

### Context

Blocking JPA code offloaded to a worker thread via Mutiny:

```java
public Uni<List<Result>> findAll(UUID id) {
    return Uni.createFrom()
              .item(() -> this.queryBlocking(id))   // ← @Transactional NOT applied
              .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());
}

@Transactional(TxType.REQUIRED)
private List<Result> queryBlocking(UUID id) {
    return em.createQuery("SELECT r FROM Result r WHERE r.id = :id", Result.class)
             .setParameter("id", id)
             .getResultList();
}
```

### Root cause

CDI interception works through a generated proxy class. When you call `this.queryBlocking(id)` from within a lambda, you are calling the **real object's method directly**, bypassing the CDI proxy. The `@Transactional` interceptor never fires. This is a fundamental property of Java — `this` is always the concrete object, not the proxy.

### Fix

Inject the bean into itself (`@Inject MySelf self`) and call through the injected proxy reference:

```java
@ApplicationScoped
public class MyRepository {

    @Inject
    MyRepository self;   // CDI proxy — interceptors fire on calls to self.*

    @Inject
    @MyPU EntityManager em;

    public Uni<List<Result>> findAll(UUID id) {
        return Uni.createFrom()
                  .item(() -> self.queryBlocking(id))   // ← @Transactional applied
                  .runSubscriptionOn(Infrastructure.getDefaultWorkerPool());
    }

    @Transactional(TxType.REQUIRED)
    public List<Result> queryBlocking(UUID id) {
        return em.createQuery("SELECT r FROM Result r WHERE r.id = :id", Result.class)
                 .setParameter("id", id)
                 .getResultList();
    }
}
```

Quarkus supports CDI self-injection — the `@Inject MySelf self` field is the CDI proxy, and all calls through it are intercepted normally.

### Why this is non-obvious

The code compiles and runs without error. In low-concurrency tests with no transaction boundary enforcement, queries may succeed because H2 and some JPA providers operate in auto-commit mode outside an explicit transaction. The bug only surfaces in production under strict JPA configuration, `TxType.REQUIRED` enforcement, or when lazy loading relies on the persistence context.

The natural instinct is to annotate the offloading method itself (`public Uni<...>`) with `@Transactional`, but transaction interceptors are meaningless on a method returning `Uni` — the transaction would open and close before any subscriber touches the emission.
