## What Happened

A cross-backend aggregator catches all contributor exceptions to return partial results (HTTP 200 always). The catch block stores the error reason in a `ConcurrentHashMap<String, String>`. For any exception type with no message string (`new NullPointerException()`, `new UnsupportedOperationException()`), `e.getMessage()` returns null. `ConcurrentHashMap.put(key, null)` throws `NullPointerException`. The very exception handler designed to prevent failures now causes one, breaking the "always return 200" contract.

## Why This Is Non-obvious

The code reads as defensive: it catches `Exception`, logs, and stores the reason. The bug is that `ConcurrentHashMap` is one of the few `Map` implementations that rejects both null keys AND null values (unlike `HashMap`, which accepts both). Developers familiar with `HashMap` don't expect this. The NPE stack trace points at the `warnings.put()` call, not at `e.getMessage()`, which further misleads.

## Fix

```java
// In the catch block:
final String reason = e.getMessage() != null
    ? e.getMessage()
    : e.getClass().getSimpleName();
accumulator.markFailed(sourceName, reason);
```

Or guard at the `put` site:
```java
map.put(key, Objects.requireNonNullElse(e.getMessage(), e.getClass().getSimpleName()));
```

## Constraints

Applies to any `ConcurrentHashMap` (and `Hashtable`). `HashMap`, `LinkedHashMap`, and `TreeMap` all accept null values and would silently succeed.
