## Symptom

`@QuarkusTest` fails during Quarkus augmentation with:

```
java.lang.IllegalStateException: Unable to load type: io.casehub.engine.internal.event.SignalReceivedEvent
    at io.quarkus.vertx.deployment.VertxProcessor.tryLoad(VertxProcessor.java:308)
```

Only visible on fresh builds. Any machine that ran tests before the engine SNAPSHOT was updated will have a cached `.quarkus/` augmentation directory and tests will pass silently.

## Context

`casehub-engine` was refactored to move internal event classes from `io.casehub.engine.internal.event.*` to `io.casehub.engine.common.internal.event.*`. Multiple SNAPSHOT modules (`casehub-engine-work-adapter`, `casehub-engine-testing`, `casehub-engine-persistence-memory`) were NOT rebuilt — they still reference the old package paths in both their bytecode and their built-in Jandex indices.

When any of these jars' Jandex indices are processed by Quarkus augmentation (either via force-indexing or built-in `META-INF/jandex.idx`), Quarkus `BeanArchives` tries to transitively index the referenced classes, fails to find them, and `VertxProcessor` then fails hard.

## Root cause

SNAPSHOT consistency problem: the engine module was refactored and the new `casehub-engine-common` was published, but the dependent modules (`engine-work-adapter`, `engine-persistence-memory`, `engine-testing`) were still built against the old package paths. Their Jandex indices embed the old class names.

## Fix

For the consuming application (casehub-life): remove all `casehub-engine*` dependencies until the SNAPSHOT is consistent. Layer 5 dependencies should only be added when the engine SNAPSHOT stabilises.

```xml
<!-- Removed until engine SNAPSHOT is consistent (engine#379, engine#380): -->
<!-- casehub-engine, casehub-engine-work-adapter, casehub-engine-scheduler-quartz,
     casehub-engine-persistence-memory, casehub-engine-persistence-hibernate -->
```

For the engine team: rebuild and republish all engine modules together in a single SNAPSHOT release cycle.

## Why this is non-obvious

The failure only appears on fresh builds — any machine with a cached Quarkus augmentation from before the refactor shows passing tests. The symptom (VertxProcessor failure on `SignalReceivedEvent`) gives no indication that the root cause is in `casehub-engine-work-adapter`'s stale Jandex index referencing that class. Removing `quarkus.index-dependency.engine-runtime` doesn't fix it because the work-adapter and other modules have built-in indices that are always processed.
