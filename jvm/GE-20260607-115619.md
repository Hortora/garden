### Symptom
After migrating worker function lambdas to return `WorkerResult`, cases using Serverless Workflow workers get stuck in RUNNING state. No compilation error. The case executes the workflow but the worker never completes.

### Context
casehub-engine has two different `function()` call patterns that look identical in a text scan:
1. `Worker.Builder.function(input -> WorkerResult.of(Map.of(...)))` — worker lambda, MUST return `WorkerResult`
2. `function(s -> { return Map.of(...); }, Map.class)` — Serverless Workflow task lambda, MUST return `Map<String,Object>`

A Python script wrapping all `return Map.of(...)` inside `function(...)` calls cannot distinguish these two patterns by text alone. Wrapping a SW task lambda in `WorkerResult.of()` causes a `ClassCastException` inside the workflow engine (it casts to `Map`) which manifests as a silent worker failure — the case stays RUNNING indefinitely.

### What was tried (didn't work)
Python script matching `return Map.of(` inside lambdas near `function(` calls — wrapped BOTH types, breaking all Serverless Workflow workers.

### Root cause
Both patterns use the same `function(` keyword. The SW task variant has a second `Map.class` argument to `function(...)`, but both calls look like `function(lambda)` to a naive regex that only looks for the lambda content.

### Fix
Use AST-level analysis rather than text matching for Java lambda migration, or distinguish by explicitly checking for the `Map.class` second argument:
```python
# Wrong — matches both Worker.Builder.function AND SW task function
re.sub(r'return Map\.of\(', 'return WorkerResult.of(Map.of(', content)

# Better — look for the Class.class second argument pattern to exclude SW tasks
# Or: manually review files that import serverlessworkflow FuncDSL
```

Alternatively, manually audit files that `import static io.serverlessworkflow.fluent.func.dsl.FuncDSL.function` — those files contain SW task lambdas that must NOT be wrapped.

### Why this is non-obvious
There is no visible type difference at the call site. Both patterns use `function(lambda)`. The SW task variant only exposes its return type constraint through the second `Map.class` argument — which is easy to miss in a search-and-replace. Compilation succeeds because `WorkerResult` is a valid `Object` and the cast failure happens at runtime inside the workflow engine, not at the call site.
