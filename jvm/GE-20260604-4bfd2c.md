## Adding a package-private test constructor to a CDI field-injected bean removes the implicit no-arg — Weld cannot instantiate it

### Symptom

A `@DefaultBean @ApplicationScoped` bean uses field injection:

```java
@DefaultBean
@ApplicationScoped
public class BlockingToReactiveGraphBridge implements ReactiveAgentGraphQuery {

    @Inject AgentGraphQuery blocking;

    // ... methods
}
```

A package-private constructor is added for unit testing (to inject a mock without a CDI container):

```java
BlockingToReactiveGraphBridge(AgentGraphQuery blocking) {
    this.blocking = blocking;
}
```

The application now fails at startup with a CDI instantiation error. Unit tests pass; `@QuarkusTest` tests or application startup fails.

### Context

Happens when adding testability to a CDI bean that uses field injection rather than constructor injection. The intent is a clean plain-Mockito unit test — but the CDI runtime uses a different code path than the test.

Does not affect constructor-injected beans where `@Inject` is on the constructor (different pattern — see constraints).

### Root cause

The Java Language Specification (§8.8.9) generates a no-arg constructor only when a class declares **no** constructors at all. Adding any explicit constructor — including a package-private one — removes the compiler-generated default. CDI (Weld) instantiates `@ApplicationScoped` beans via the no-arg constructor before running `@Inject` field injection. When the no-arg constructor is absent, Weld cannot create the bean.

There is no compile-time error. The missing no-arg is only detected at CDI container startup when Weld attempts to instantiate the bean.

### Fix

Add an explicit no-arg constructor alongside the test constructor. Both must coexist:

```java
@DefaultBean
@ApplicationScoped
public class BlockingToReactiveGraphBridge implements ReactiveAgentGraphQuery {

    @Inject AgentGraphQuery blocking;

    public BlockingToReactiveGraphBridge() {}           // CDI — Weld uses this before @Inject

    BlockingToReactiveGraphBridge(AgentGraphQuery blocking) { // test path — plain Mockito
        this.blocking = blocking;
    }

    // ... methods
}
```

The public no-arg constructor comment is useful here — it explains why an otherwise empty constructor exists. Without it, the next developer may delete it as dead code.

### Why this is non-obvious

The dual-constructor pattern for CDI testing is well-documented for **constructor-injected** beans — where `@Inject` is on a single constructor that then delegates to a package-private constructor. For **field-injected** beans the same Java language rule applies, but the CDI consequence is less visible: the `@Inject` annotation is on the field, not on any constructor, so there is no obvious connection between "I added a constructor" and "CDI can't find the instantiation path". The failure only appears at runtime startup, not compilation.
