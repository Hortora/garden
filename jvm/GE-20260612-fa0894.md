### Symptom
Messages dispatched between catch-up query completion and live-stream subscription are lost — they appear in neither stream. No error, no warning. The SSE client sees a gap in the event sequence.

### Context
Building an SSE endpoint with two phases: catch-up (historical events from database) then live (new events from an in-process event bus). The natural pattern is:

```java
Multi<String> live = eventBus.subscribe(channelId); // "subscribe first"
Multi<String> catchUp = fetchFromDb();
return Multi.createBy().concatenating().streams(catchUp, live);
```

The developer expects `live` to be actively receiving events while `catchUp` runs, buffering them for delivery after catch-up completes.

### What was tried (didn't work)
"Subscribe to the bus before building the catch-up Multi, then concatenate." This appears correct — the `subscribe()` call returns a Multi, which should be receiving events. But it isn't.

### Root cause
`Multi.createFrom().emitter(emitter -> { ... })` is **lazy** — the lambda that registers the subscriber runs at **subscription time**, not at Multi creation time. `concatenating().streams(catchUp, live)` subscribes to streams sequentially: it subscribes to `catchUp` first, and only subscribes to `live` after `catchUp` completes.

So the bus subscriber does not exist during catch-up. Any events emitted to the bus during that window have no subscriber and are silently dropped.

### Fix
Use cursor-based polling instead of bus-push for the live phase. An `AtomicLong` cursor carries forward from catch-up to live — no subscription timing gap:

```java
AtomicLong lastSentId = new AtomicLong(0L);

Multi<String> catchUp = Multi.createFrom().uni(
    Uni.createFrom().item(() -> {
        List<Message> msgs = messageService.pollAfter(channelId, 0L, 500);
        updateCursor(msgs, lastSentId);
        return serialize(msgs);
    })
).filter(Objects::nonNull);

Multi<String> live = Multi.createFrom().ticks().every(Duration.ofMillis(500))
    .onItem().transformToUniAndConcatenate(tick ->
        Uni.createFrom().item(() -> {
            List<Message> msgs = messageService.pollAfter(channelId, lastSentId.get(), 50);
            updateCursor(msgs, lastSentId);
            return msgs.isEmpty() ? heartbeat() : serialize(msgs);
        })
    );

return Multi.createBy().concatenating().streams(catchUp, live);
```

The cursor (`lastSentId`) is set during catch-up and read by the first live tick — no gap, no duplicates.

### Why this is non-obvious
The mental model "I created the Multi object, so it's subscribed" is natural and wrong. Mutiny's lazy evaluation means Multi creation ≠ subscription. The Javadoc for `emitter()` says "the callback is invoked when a subscriber subscribes" — but in a concatenation context, this subtlety is easy to miss. Claudony built the subscribe-first pattern, hit this gap in production, and switched to polling (documented in `MeshResource.java`).

### Why this fix
Cursor-based polling eliminates the subscription timing problem entirely — the cursor is a value, not a subscription. It also avoids the cross-thread SSE flushing issue (see GE-20260522-daca26) since polling runs on the response thread, not a bus-emitter thread.

### Alternatives considered
- **Eager subscription via CompletableFuture bridge**: manually subscribe to the bus Multi before building the concatenation, buffer events in a BlockingQueue, drain after catch-up. Adds manual lifecycle management and is fragile.
- **Single polling stream (no catch-up/live split)**: poll from the start with no separate catch-up phase. Simpler but adds latency to initial load — first poll waits 500ms before checking.
