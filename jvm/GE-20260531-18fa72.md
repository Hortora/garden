## The technique

In a Java project without JPMS (no `module-info.java`), package-private members in a compiled JAR are accessible from any class in the same package name, regardless of which JAR that class lives in. The JVM enforces package-private at the package-name level, not at the JAR or compilation-unit level.

This means you can write test-only methods on an `@ApplicationScoped` bean in module A (package `com.example.casehub`) and call them from integration tests in module B (same package `com.example.casehub`) without making those methods `public`.

## Practical use case

```
claudony-casehub/
  src/main/java/io/casehub/claudony/casehub/
    ClaudonyReactiveWorkerProvisioner.java   ← @ApplicationScoped bean
      UUID drainCausalContext(UUID caseId)    ← package-private — for ledger capture
      void seedCausalContextForTest(...)      ← package-private — test helper

claudony-app/
  src/test/java/io/casehub/claudony/casehub/
    ClaudonyLedgerEventCaptureTest.java      ← @QuarkusTest in app module
      @Inject ClaudonyReactiveWorkerProvisioner provisioner
      provisioner.seedCausalContextForTest(caseId, entryId)  ← compiles and runs
```

The `claudony-app` test classpath includes the `claudony-casehub` JAR. Both are in the same package `io.casehub.claudony.casehub`. The package-private call succeeds at compile time and at runtime.

## Why this is useful

Test seeding methods that only exist to support tests don't belong in the public API. Making them `public` pollutes the interface and signals intentional external use. Package-private keeps them invisible to unrelated code while remaining accessible from same-package tests in dependent modules — a cleaner alternative to making the full bean or method public, reflection-based access, or introducing a test-specific subclass.

## Why non-obvious

Most developers assume package-private means "same compilation unit" or "same JAR." The actual scope is "same package name on the same classloader" — which spans JARs on a flat classpath. This intuition gap means developers either make things unnecessarily public or add reflection-based test hacks rather than reaching for the simpler same-package approach.
