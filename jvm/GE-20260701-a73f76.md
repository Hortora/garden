### Symptom

`@QuarkusTest` tests fail with CDI wiring errors or `NullPointerException` in `QuartzWorkerExecutionManager` after setting `%test.quarkus.scheduler.enabled=false` in `application.properties`.

### Context

Harness apps commonly disable the scheduler in tests to prevent `@Scheduled` game ticks from firing during CDI-managed test execution. The intuitive setting is `quarkus.scheduler.enabled=false`.

### Root cause

`quarkus.scheduler.enabled=false` prevents the Quartz scheduler from starting entirely — no `Scheduler` CDI bean is produced. `casehub-engine-scheduler-quartz`'s `QuartzWorkerExecutionManager` injects the `Scheduler` bean and calls `scheduleJob()` on it. Without the bean, CDI validation fails or the manager NPEs.

### Fix

Use `start-mode=halted` instead:

```properties
# Wrong — prevents Quartz from starting; breaks engine beans
%test.quarkus.scheduler.enabled=false

# Right — starts Quartz infrastructure but suppresses job firing
%test.quarkus.scheduler.start-mode=halted
```

`halted` means: the Quartz scheduler starts (CDI bean is produced, `QuartzWorkerExecutionManager` can inject it), but no `@Scheduled` methods fire until `Scheduler.resume()` is called. Tests that need to trigger ticks manually call the orchestrator directly.

### Why this is non-obvious

The Quarkus documentation presents `enabled=false` as the way to disable scheduling in tests. It works correctly for apps that only use `quarkus-scheduler`. The failure only occurs when `casehub-engine-scheduler-quartz` (which adds Quartz infrastructure) is on the classpath — the Quartz integration requires the scheduler to be started even if no jobs are actively firing.
