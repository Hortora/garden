### Symptom

Calling `ControlMessageParser.parse(line)` for a JSON line with an unrecognised `type` field returns `null` — it does not throw `MessageParseException`. Any switch or instanceof dispatch on the result without a null guard produces a `NullPointerException` silently dropped by the emitter.

### Context

Occurs when reading `--output-format stream-json` stdout from Claude CLI and dispatching on the `ParsedMessage` sealed interface. The sealed interface has 5 permitted types: `RegularMessage`, `Control`, `ControlResponseMessage`, `RateLimitEventMessage`, and `EndOfStream`. `EndOfStream` is an **internal sentinel for the `MessageStreamIterator` iterator API only** — `ControlMessageParser.parse()` never returns it. The type exists solely for compiler exhaustiveness in switch expressions on the sealed type.

### Root Cause

`parseRegularMessage()` in `ControlMessageParser` delegates to `MessageParser.parseMessageFromNode()`. That method returns `null` for unrecognised `type` values (forward-compatibility with future CLI versions). `parseFromNode()` propagates the null to the caller of `parse()`. The `EndOfStream.INSTANCE` singleton is only returned internally by `MessageStreamIterator` — never by `parse()`.

From the SDK source (line 235):
```java
private ParsedMessage parseRegularMessage(JsonNode node) throws MessageParseException {
    Message message = messageParser.parseMessageFromNode(node);
    if (message == null) {
        return null;
    }
    return ParsedMessage.RegularMessage.of(message);
}
```

### Fix

Add a null guard before any dispatch:

```java
ParsedMessage parsed = parser.parse(line);
if (parsed == null) continue;  // unrecognised but parseable type — forward compatible
// now safe to switch/instanceof
```

`StreamingTransport.processInboundMessages()` line 746 already applies this exact guard:
```java
if (parsed == null) {
    continue;
}
```
Match that pattern in any custom stdout reader.

### Why This Is Non-Obvious

The `ParsedMessage` sealed interface looks exhaustive — 5 permitted types, including `EndOfStream`. A sealed type used in a switch expression compiles without `default` when all cases are listed. A developer seeing `EndOfStream` in the switch and assuming `parse()` returns it will write the switch without a null guard. The distinction between "sealed type for exhaustiveness" and "values `parse()` actually returns" is not documented.
