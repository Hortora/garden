### Symptom

`.onFailure(java.util.concurrent.TimeoutException.class).recoverWithItem(...)` never fires
after a Mutiny timeout. The recovery is silently skipped — the `TimeoutException` propagates
unhandled.

### Context

Using Mutiny's `.ifNoItem().after(Duration).fail()` to timeout a blocking operation on virtual
threads. The natural assumption is that `.fail()` throws `java.util.concurrent.TimeoutException`
(the JDK type). It doesn't.

### What was tried (didn't work)

1. `.onFailure(java.util.concurrent.TimeoutException.class).recoverWithItem(...)` — never
   catches. The exception type doesn't match so recovery is skipped entirely.

### Root cause

Mutiny wraps the timeout in its own `io.smallrye.mutiny.TimeoutException` (which does NOT extend
`java.util.concurrent.TimeoutException`). The class hierarchy:
- `io.smallrye.mutiny.TimeoutException extends java.lang.RuntimeException`
- `java.util.concurrent.TimeoutException extends java.lang.Exception`

They share a name but not an inheritance chain.

### Fix

Use the Mutiny type:

```java
.ifNoItem().after(Duration.ofMillis(timeoutMs)).fail()
.onFailure(io.smallrye.mutiny.TimeoutException.class)
.recoverWithItem(t -> handleTimeout())
```

### Why this is non-obvious

The JDK `TimeoutException` is the universally expected type for timeout failures. Mutiny's
decision to use its own class with the same simple name but different package and superclass
creates a name collision that reads correct but silently fails.

### Why this fix

The Mutiny type is the only correct catch target. There is no configuration to change which
exception Mutiny throws — the class is hardcoded in the Mutiny source.
