# context.registerSubtypes() does not feed a custom @JsonTypeIdResolver

### Symptom

Vendor Jackson Module calls `context.registerSubtypes(new NamedType(...))` in `setupModule()` to register plugin types for polymorphic deserialization. The base type uses `@JsonTypeIdResolver(CustomResolver.class)`. Deserialization of vendor types silently falls back to the parent type — supplement-specific fields are lost. No error, no warning.

### Context

A plugin-extensible type hierarchy where the base type uses `@JsonTypeInfo(use = Id.CUSTOM)` with a custom `TypeIdResolver`. Plugin modules ship Jackson `Module` implementations that register their subtypes during `setupModule()`.

### Root cause

`context.registerSubtypes()` feeds Jackson's `SubtypeResolver`, which populates `TypeNameIdResolver._idToType` (a `HashMap<String, JavaType>`) via its static `construct()` method. Verified in Jackson 2.21.1 source at `TypeNameIdResolver.java:59-80`.

But `@JsonTypeIdResolver(CustomResolver.class)` tells Jackson to use the custom resolver, **not** `TypeNameIdResolver`. The `SubtypeResolver` feeds `TypeNameIdResolver` exclusively — the custom resolver never sees registrations made via `context.registerSubtypes()`.

Result: plugin types are registered with Jackson's standard machinery, but the custom resolver doesn't know about them. Deserialization falls through to the fallback path silently.

### Fix

Plugin modules must register directly with the custom resolver's own registry:

```java
@Singleton
public class VendorJacksonModule extends SimpleModule {
    @Override
    public void setupModule(SetupContext context) {
        super.setupModule(context);
        CustomTypeIdResolver.registerType("THERMOSTAT:VendorThermostat", VendorThermostat.class);
    }
}
```

The custom resolver maintains a static `ConcurrentHashMap` — `registerType()` adds to it. `setupModule()` runs during `ObjectMapper` initialization, before any deserialization.

### Why this is non-obvious

The Jackson documentation for `SetupContext.registerSubtypes()` says it registers subtypes for polymorphic type handling — no mention that it only works with the standard `TypeNameIdResolver`. A developer using `@JsonTypeIdResolver` would naturally assume `registerSubtypes()` feeds their resolver. The failure is silent — the type falls back to the parent with no error or warning.
