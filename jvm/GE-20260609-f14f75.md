## Context

Implementing a binary WebSocket client for a protocol that guarantees exactly one request in flight at a time (SC2 Remote API, Redis protocol, etc.). The challenge: the response arrives on a background I/O thread; the calling thread must receive it synchronously.

## The Technique

Use `SynchronousQueue<byte[]>` as the handoff mechanism between the I/O reader thread and the calling thread. `SynchronousQueue` has zero capacity — a `put()` has no home until a `take()` or `poll()` consumer is waiting:

```java
private final SynchronousQueue<byte[]> responseQueue = new SynchronousQueue<>();

// I/O reader thread (background virtual thread):
responseQueue.put(complete); // blocks until consumer arrives — bounded to microseconds

// Calling thread (sendSync):
byte[] responseBytes = responseQueue.poll(timeout.toMillis(), TimeUnit.MILLISECONDS);
if (responseBytes == null) throw new TimeoutException("no response within " + timeout);
```

## Why This Over `LinkedBlockingQueue`

`LinkedBlockingQueue` with a depth-1 invariant works in practice but provides no structural guarantee:
- A late response from a previous request could sit in the queue and be consumed as the "current" response
- The invariant must be maintained by discipline, not by the data structure

`SynchronousQueue` enforces depth-0 by construction: there is no queue. Each `put()` directly rendezvous with its `poll()`. Stale responses are impossible because there is nowhere to store them.

## Why Non-Obvious

Developers reach for `LinkedBlockingQueue(1)` or `ArrayBlockingQueue(1)` as the obvious bounded queue for this pattern. The depth-0 semantics of `SynchronousQueue` are known primarily from `ThreadPoolExecutor` internals and are rarely considered for explicit protocol pairing. The `put()` semantics (blocks until consumer ready) are subtly different from `offer()` (returns false immediately if no consumer) — using `offer()` would silently drop responses.
