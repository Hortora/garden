### Context

You need to register a resource (e.g. a backend in a registry, a channel in a gateway, a listener on a topic) exactly once per key, even under concurrent access from multiple threads. You want to avoid explicit locks, `synchronized` blocks, and the awkward semantics of `computeIfAbsent` when the computation has side effects.

### Technique

```java
private final Set<UUID> registeredKeys = ConcurrentHashMap.newKeySet();

public void ensureRegistered(UUID key) {
    if (registeredKeys.add(key)) {   // atomic: exactly one thread returns true
        doRegistration(key);          // called exactly once per key
    }
}

public void deregister(UUID key) {
    registeredKeys.remove(key);      // subsequent ensureRegistered() will re-register
}
```

`ConcurrentHashMap.newKeySet()` returns a `Set<K>` backed by a ConcurrentHashMap. `Set.add()` returns `true` if the element was actually added (was absent) and `false` if it was already present. Under concurrent access, exactly one thread gets `true` — all others get `false` and skip the side effect. No explicit locking required.

### Why this is non-obvious

Most developers reach for `synchronized(this)` or `computeIfAbsent`. The `newKeySet().add()` pattern:
- Is 2 lines vs 5-10 for a `synchronized` block
- Returns the boolean signal the caller needs directly from `add()`
- Has guaranteed CAS semantics from the ConcurrentHashMap backing
- Clearly expresses intent: "register this key once, skip if already registered"

`computeIfAbsent` also works but its Javadoc warns that the computation must not modify the map — using it to trigger side effects (registration) is semantically awkward and potentially unsafe if the side effect is not pure. The `Set.add()` pattern has no such restriction.

### Constraints

The side effect in `doRegistration()` must be safe to call after `add()` returns but before other threads observe the registration in the set. If `doRegistration()` itself must be atomic with the `add()` (e.g. the registration step reads back the key), this pattern is insufficient — use a dedicated lock or a more explicit state machine.
