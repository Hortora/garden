## Context

When a consumer app (e.g. AML) reports cases stuck in RUNNING against the current engine SNAPSHOT, the natural instinct is to diagnose from the engine's own code and tests. The engine's CI is green — all 740 tests pass. The difference is the consumer's CDI context: multiple datasources, additional `@ObservesAsync` observers, Jandex-indexed modules the engine tests exclude, and application-specific beans.

## Technique

Run the failing consumer test directly — don't hypothesise from engine code.

```bash
# Wrong: spend hours reading engine handlers and guessing
# Right: 3 minutes, definitive answer
TESTCONTAINERS_RYUK_DISABLED=true mvn -f /path/to/consumer/pom.xml \
  -pl app -am \
  -Dtest="ConsumerTestClass#failingMethod" \
  -Dsurefire.rerunFailingTestsCount=0 \
  -Dsurefire.failIfNoSpecifiedTests=false test
```

Read the actual test output. Look for what IS logged vs what ISN'T. In the #491 case, the diagnostic line was:

```
INFO  CaseContextChangedEventHandler: Rules+milestones+goals processed for caseId: ...
```

With NO subsequent `Agent selected:` log. This immediately showed bindings evaluated but none matched — a JQ expression evaluation issue, not a Quartz/CDI/tenancy issue. Three hypotheses eliminated in one test run.

## Why this is non-obvious

Engine tests are designed to pass in the engine's own context. They exclude production CDI beans (`WorkerDecisionEventCapture`, `CaseLedgerEventCapture`), use in-memory repositories, and have `.working.*`-prefixed JQ expressions. The consumer app has none of these accommodations — it runs the full production CDI context against H2. The failure surface is entirely different.

The temptation to diagnose from engine code is strong because the engine is where the code lives. But the bug lives in the interaction between the engine and the consumer context — and that interaction only exists in the consumer test.
