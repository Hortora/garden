## Context

Testing that a security pre-flight guard (e.g. `MemoryPermissions.assertTenant()`) fires
before any HTTP call in a REST-backed adapter. The implementation should throw
`SecurityException` before it reaches the wire.

The naive approach is to stub the happy path for item 0 and then call
`wireMock().verify(0, postRequestedFor(...))` after the exception. This works, but
it requires two assertions to tell the full story — and if both pass the test can still
hide subtle ordering bugs.

## Technique

Deliberately omit any stub for the endpoint that the pre-flight guard should prevent.

```java
@Test
void storeAll_any_tenant_mismatch_fires_zero_http_calls() {
    // No stub registered for POST /memories — intentional.
    // If implementation fires HTTP before the security check, WireMock returns
    // a no-match response (404), which becomes Mem0StoreException — NOT SecurityException.
    // The wrong exception type is the test failure signal.
    assertThrows(SecurityException.class, () ->
        store.storeAll(List.of(
            new MemoryInput("entity-1", DOMAIN, TENANT,       null, "ok",  Map.of()),
            new MemoryInput("entity-2", DOMAIN, OTHER_TENANT, null, "bad", Map.of())
        )));
    wireMock().verify(0, postRequestedFor(urlEqualTo("/memories")));
}
```

When WireMock has no stub for a request:
- The REST client receives a 404 (or a WireMock no-mapping response)
- The client wraps this in a `WebApplicationException`
- The adapter converts it to the domain exception (e.g. `Mem0StoreException`)
- `assertThrows(SecurityException.class, ...)` fails because `Mem0StoreException` was thrown

The `verify(0, ...)` line is redundant for correctness — the wrong exception type already
proves the HTTP call happened — but it makes the failure reason explicit.

## Why this is non-obvious

Developers instinctively stub the happy path to "protect" the test from failing for the
wrong reason. The impulse is: "item 0 might get through before item 1 throws, so I need
to stub it." But that instinct inverts the test logic for a pre-flight guard.

If the pre-flight fires correctly, item 0 never reaches the wire and no stub is needed.
If the pre-flight is missing, the un-stubbed call fails with the wrong exception —
which is exactly the signal you want.

Omitting the stub turns the test into a two-for-one check:
1. The exception type proves the pre-flight guard ran (not a transport error)
2. The `verify(0)` makes the HTTP count explicit for the reader

A test with the stub can still pass even if the pre-flight fires too late (after item 0
is sent) — the stub absorbs the call and the exception type appears correct by coincidence.

## Applicability

Works for any REST-backed adapter where:
- A security or validation guard should prevent HTTP calls
- The guard throws a typed exception distinguishable from transport errors
- WireMock is the HTTP mock

The pattern generalises: if the "should not happen" path produces a different
exception type than the "guard fired correctly" path, the absence of a stub
is the cheapest and strongest pre-flight test you can write.
