### Symptom

Implementation plan specifies migration `V8__add_feature.sql` based on the assumption that `V7__...sql` is the latest. At runtime (or during tests), Flyway throws a checksum or duplicate-version error because `V8__add_other_feature.sql` already exists from a prior session.

### Context

Occurs in any multi-session project where Flyway migration files accumulate over time. Particularly common in agentic development where plans are written from conversation memory rather than live filesystem inspection.

### Root cause

The plan author assumed migration numbering from prior session memory (`V7` was the last seen) without verifying the current filesystem state. Flyway requires strictly sequential version numbers and rejects or misinterprets any attempt to apply a migration with a version that already has a file.

### Fix

Before naming any new Flyway migration, always list existing migrations:

```bash
ls runtime/src/main/resources/db/migration/
```

Use the next available number after the highest one found. Never assume from session memory — always read from the filesystem.

### Why this is non-obvious

In single-session development, the sequence is always visible in context. In multi-session or agentic workflows, the plan is written from recalled state — which may be stale. The plan-time assumption feels safe because "I just saw V7 last time," but the filesystem has moved on.
