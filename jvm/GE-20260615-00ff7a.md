## What it is

When a provisioner registers an endpoint in a registry, fire a CDI `Event<T>` from `register()`. Transport-specific modules observe `@ObservesAsync T` and self-configure — starting Camel routes, subscribing Kafka topics, etc. The provisioner calls only the registry; it never imports or calls any transport module directly.

## The problem it solves

A provisioner that needs to dynamically configure stream routes (Kafka consumers, Camel routes, webhook handlers) faces a dependency problem: to call `CamelStreamManager.addRoute()`, it must compile-depend on the Camel stream module. Every new stream module requires a new provisioner import. The provisioner becomes a coupling hub for all transport implementations.

## The technique

Add an event type to the registry API:

```java
// In casehub-platform-api
public record EndpointRegistered(EndpointDescriptor descriptor) {}
```

Fire from `register()` in implementing classes:

```java
// InMemoryEndpointRegistry
@Inject Event<EndpointRegistered> registeredEvent;

@Override
public void register(EndpointDescriptor endpoint) {
    store.put(key(endpoint), endpoint);
    registeredEvent.fireAsync(new EndpointRegistered(endpoint));
}
```

Each stream module observes and filters by its own protocol:

```java
// platform-streams-camel
@ApplicationScoped
class CamelStreamConfigurer {

    @Inject CamelContext camelContext;

    void onEndpointRegistered(@ObservesAsync EndpointRegistered event) {
        if (event.descriptor().protocol() != EndpointProtocol.CAMEL) return;
        camelContext.addRoutes(buildRoute(event.descriptor()));
    }
}
```

The provisioner only calls `endpointRegistry.register(descriptor)`. Zero imports of transport modules.

## Why a developer wouldn't reach for this

The natural instinct when a provisioner needs to start a Camel route is to inject `CamelStreamManager` directly. The CDI event pattern feels like extra indirection. The payoff — zero coupling, extensibility without provisioner changes — only becomes obvious when you realise that every new stream module would otherwise require a provisioner update.

## Constraints

- No-op registry implementations (e.g. `@DefaultBean` `NoOpEndpointRegistry`) must NOT fire `EndpointRegistered` — a no-op registry signals "no endpoints exist"; firing events would trigger stream routes for phantom endpoints.
- `@ObservesAsync` fires on a CDI thread pool thread. At application startup, the event may arrive before `CamelContext` reaches `ServiceStatus.Started`. Buffer pre-start events and process them in a `@Observes io.quarkus.runtime.StartupEvent` handler.

### Why this fix

Compared to injecting transport managers directly: the event pattern makes the provisioner closed to modification when new stream modules are added (open/closed principle at module boundary). The no-op registry constraint keeps the pattern safe — no phantom routes from test deployments or single-node installs that use the no-op registry.
