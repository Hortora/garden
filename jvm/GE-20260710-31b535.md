### Symptom
Quarkus fails to start with `Cannot construct instance of Cbr$Timing, problem: CASE_LIFETIME` when a YAML case definition contains `timing: CASE_LIFETIME`.

### Context
Adding `spec.cbr` sections to YAML case definitions in casehub-life. The `timing` field maps to `CbrConfig.CbrRetrievalTiming` which is a jsonschema2pojo-generated enum in `casehub-engine-schema`. The Java constant is `CASE_LIFETIME` but the YAML parser calls `fromValue()`, not `valueOf()`.

### Root cause
jsonschema2pojo generates enums with a custom `fromValue(String)` method that maps from the JSON schema enum value — typically kebab-case (`"case-lifetime"`) — to the Java constant (`CASE_LIFETIME`). Jackson's deserialiser uses `fromValue()`, not Java's `Enum.valueOf()`. The two methods accept different string formats:

```java
// What fromValue() expects (JSON schema value):
Cbr.Timing.fromValue("case-lifetime")  // → CASE_LIFETIME ✅

// What valueOf() expects (Java constant name):
Cbr.Timing.valueOf("CASE_LIFETIME")    // → CASE_LIFETIME ✅

// What actually happens in YAML parsing:
Cbr.Timing.fromValue("CASE_LIFETIME")  // → IllegalArgumentException ❌
```

The bytecode confirms the mapping: `ldc "CASE_LIFETIME"` (constant) paired with `ldc "case-lifetime"` (fromValue key).

### Fix
Use the JSON schema value (kebab-case) in YAML, not the Java constant name:

```yaml
# Wrong — matches Java constant but not fromValue()
timing: CASE_LIFETIME

# Right — matches fromValue() which is what Jackson calls
timing: case-lifetime
```

### Why this is non-obvious
The Java constant `CASE_LIFETIME` looks correct — it matches the enum constant exactly. Most developers assume YAML enum values map to Java constant names. The indirection through `fromValue()` with a different string format is invisible unless you read the generated bytecode or source.

### Why this fix
Using the JSON schema value is the only correct approach — `fromValue()` is the deserialization entry point and its mapping is fixed at code generation time. Patching `fromValue()` to also accept constant names would require modifying generated code.

### Alternatives considered
- Using `@JsonProperty` annotation to remap — rejected: the enum is generated, annotations would be lost on regeneration
- Wrapping with a custom Jackson deserializer — rejected: unnecessary complexity when the fix is just using the right string value
