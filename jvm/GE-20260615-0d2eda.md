### Symptom
Maven build fails when `langchain4j-embeddings` is declared with `${langchain4j.version}` (e.g. 1.14.1):

```
Could not collect dependencies: Failed to read artifact descriptor for
dev.langchain4j:langchain4j-embeddings:jar:1.14.1
```

The artifact is reported as "absent" from Maven Central, then falls through to other repositories (e.g. GitHub Packages) which return 401 Unauthorized — misleading the developer into thinking it's an auth problem.

### Context
`dev.langchain4j:langchain4j-embeddings` provides `OnnxEmbeddingModel` for running in-process ONNX embedding models. It was originally in a separate repository (`langchain4j/langchain4j-embeddings`) with its own release cycle. It has since been moved into the main `langchain4j` repository under `embeddings/`, but Maven Central artifacts still follow the independent version scheme.

Available versions on Maven Central: 0.17.0 through 0.36.2, then 1.0.0-alpha1 through 1.0.0-beta5. No version matching the main langchain4j line (1.14.x, 1.15.x, etc.) exists.

### What was tried (didn't work)
1. Using `${langchain4j.version}` (1.14.1) — artifact does not exist
2. Guessing `1.14.1-beta22` — does not exist either
3. Searching Maven Central search API — confirmed only 31 versions exist, none in the 1.14.x range

### Root cause
The `langchain4j-embeddings` module uses a completely independent versioning scheme from the main `langchain4j` library. The parent POM in the LangChain4j repository shows a `-betaN` suffix on the parent version (e.g. `1.17.0-beta27-SNAPSHOT`), but published artifacts stop at `1.0.0-beta5`.

### Fix
Declare a separate version property for `langchain4j-embeddings`:

```xml
<properties>
    <langchain4j.version>1.14.1</langchain4j.version>
    <langchain4j-embeddings.version>1.0.0-beta5</langchain4j-embeddings.version>
</properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>dev.langchain4j</groupId>
            <artifactId>langchain4j-embeddings</artifactId>
            <version>${langchain4j-embeddings.version}</version>
        </dependency>
    </dependencies>
</dependencyManagement>
```

The `1.0.0-beta5` jar depends on `langchain4j-core:1.0.0` but is binary compatible with `langchain4j-core:1.14.1` — the `EmbeddingModel` interface is stable within 1.x. Transitive deps (`onnxruntime`, `ai.djl.huggingface:tokenizers`) are overridden by your project's dependency management.

### Why this is non-obvious
The LangChain4j ecosystem uses a single version number for all core modules (`langchain4j`, `langchain4j-core`, `langchain4j-qdrant`, etc.) and the BOM covers them. The embeddings modules are the exception — they have their own version line. Nothing in the documentation or the parent POM structure signals this. The natural assumption is `${langchain4j.version}` works for everything.

### Why this fix
Using a separate version property rather than pinning in the child POM keeps the version centrally managed and signals that this artifact is intentionally on a different version line.

### Alternatives considered
- Using a pre-packaged model artifact (e.g. `langchain4j-embeddings-all-minilm-l6-v2`) — bundles the model in the jar, but we need to download models separately for different deployment targets.
- Building an `EmbeddingModel` adapter on top of the inference-runtime module — overengineering; LangChain4j already provides this capability via `OnnxEmbeddingModel`.
