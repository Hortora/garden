### Symptom

You need an IRC server to integration-test a Java IRC client, want it as a Maven test dependency (no Docker, no external binaries), and search Maven Central and GitHub for an embeddable pure-Java IRC server.

### Context

Building `casehub-connectors-chat-irc` — a ChatPlatform SPI implementation for IRC. Integration tests need a real IRC server to test connect/join/send/receive against.

### What was tried (didn't work)

1. **ircd4j** (Wolf480pl/ircd4j) — described as "a Java IRC server library" but marked "Work In Progress", 37 commits, 0 forks, 1 star, never released, not on Maven Central. LGPL-3.0.
2. **zachbroad/Java-IRC-Server** — implements 25+ RFC 1459 commands but has no Maven or Gradle build (IntelliJ `.iml` only), no releases, 33 commits, 3 stars. Would need forking and wrapping in a build system.
3. **Ergo** (Go-based IRC server) — works well but requires either Docker (Testcontainers) or a pre-installed Go binary. Adds an external dependency to the test infrastructure.

### Root cause

IRC server implementations in Java are rare because the IRC protocol's heyday predates modern Java build tooling. The few that exist are hobby projects that were never published to any repository.

### Fix

Write an embedded test IRC server. RFC 1459 core is small enough that a purpose-built test server handling NICK/USER/JOIN/PART/PRIVMSG/NAMES/LIST/PING/PONG/QUIT is ~300 lines using `ServerSocket` + virtual threads. This gives zero external dependencies, exact control over test scenarios, and multi-client support for testing inbound message delivery.

### Why this is non-obvious

The IRC protocol is mature and well-specified — you'd reasonably expect a test server library to exist in the Java ecosystem. The discovery that no viable option exists only comes after evaluating the candidates and finding them all abandoned or unpublished.

### Alternatives considered

- **Ergo via Testcontainers** — adds Docker as a test dependency; not acceptable for a library module that should build with `mvn test` alone
- **WireMock-style mocking at the socket level** — possible but means reimplementing IRC protocol parsing anyway, minus the server state management that makes tests realistic
