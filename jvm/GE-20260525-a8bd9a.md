## What

`AiServicesProcessor#handleAiServices` throws `java.lang.IllegalStateException: Duplicate key null (attempted merging values 0 and 1)` at Quarkus augmentation time — before the application starts. There IS a prior WARN log about `-parameters` but it fires amid other build output and is easy to miss.

## Symptom

```
[error]: Build step io.quarkiverse.langchain4j.deployment.AiServicesProcessor#handleAiServices threw an exception:
    java.lang.IllegalStateException: Duplicate key null (attempted merging values 0 and 1)
        at AiServicesProcessor$TemplateParameterInfo.toNameToArgsPositionMap(AiServicesProcessor.java:1943)
        at AiServicesProcessor.gatherSystemMessageInfo(AiServicesProcessor.java:1524)
```

Tests then fail with:
```
Could not load class with name: io.example.MyAiServiceTest
```

## Why It Happens

`AiServicesProcessor` reads method parameter names at augmentation time to resolve `@UserMessage("{{param}}")` template variables. Without `-parameters`, method parameter names are `null` at runtime. When two parameters both resolve to `null`, `Collectors.toMap()` detects a duplicate key and throws.

## Fix

Add to `pom.xml` `<properties>`:

```xml
<maven.compiler.parameters>true</maven.compiler.parameters>
```

Or in `maven-compiler-plugin` configuration:

```xml
<configuration>
    <parameters>true</parameters>
</configuration>
```

After adding, run `mvn clean` to force recompilation — incremental builds will not recompile unchanged sources.

## Why This Is Non-Obvious

The error message `Duplicate key null` gives no hint about `-parameters`. The WARN log IS present — `"The application has been compiled without the '-parameters' being set flag on javac"` — but fires before the error and is not connected to it in the output. A developer who has never seen this pattern looks at the class name, the stack trace, and the `@UserMessage` annotation before thinking about compiler flags.

Projects using `quarkus-bom` as `<parent>` never see this because the BOM propagates `<maven.compiler.parameters>true</maven.compiler.parameters>` automatically. Projects that import the BOM via `<dependencyManagement>` (the pattern used in Quarkus extensions and multi-module projects) do not inherit this property.
