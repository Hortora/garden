### Technique
When testing SSE streams that fire multiple events, use an `AtomicInteger` call counter in the `snapshotFn` `Supplier<String>` to return distinct payloads on each call. This lets assertions distinguish the initial snapshot from subsequent lifecycle-triggered pushes by content, not just by count.

### Context
Quarkus SSE strategies (e.g. `EventsOnlyStrategy`) call `snapshotFn.get()` twice during a test: once synchronously on `subscribe()` for the initial snapshot, and once in `onLifecycleEvent()` for the push. If both calls return the same string, a test can only assert `received.hasSize(2)` — it cannot prove the second event came from the mutation rather than a duplicate initial.

### Implementation
```java
var callCount = new java.util.concurrent.atomic.AtomicInteger(0);

broadcaster.subscribe("case-1",
        () -> callCount.incrementAndGet() == 1 ? "data: initial\n\n" : "data: updated\n\n")
    .subscribe().with(e -> { received.add(e); latch.countDown(); });

registry.updateStatus("session-1", SessionStatus.ACTIVE);

assertThat(latch.await(2, TimeUnit.SECONDS)).isTrue();
assertThat(received.get(0)).isEqualTo("data: initial\n\n");
assertThat(received.get(1)).isEqualTo("data: updated\n\n");
```

### Why this is non-obvious
The natural approach is to assert `hasSize(2)`. That proves two events arrived but not that the second came from the registry mutation — a bug that fires the initial snapshot twice would pass the same assertion. The `AtomicInteger` counter is cheap and makes the test a real proof of causal wiring.
