## Symptom

`EndpointRegistry.discover()` filtered by `EndpointCapability.RECEIVE` returns poll target endpoints alongside Kafka topics and webhook receivers. Filtering by `EndpointProtocol.HTTP` alone does not distinguish poll targets from webhook inbound receivers. The wrong module picks up endpoints it should not handle.

## Context

When registering stream endpoint descriptors — Kafka topics, AMQP queues, webhooks, poll targets — all feel like "receiving data." The natural choice for all five is `EndpointCapability.RECEIVE`. But poll uses a different access pattern from the others.

## Root cause

`EndpointCapability.RECEIVE` means "Endpoint can push data to the caller (webhook delivery, Kafka consume, SSE)." This is correct for:
- Kafka — the broker pushes messages to our consumer
- AMQP — same push semantics
- Webhook — external system POSTs to our endpoint
- Camel — route delivers to CDI observer

But **poll** (platform-streams-poll) initiates an HTTP GET — we pull from the endpoint. That is `QUERY`: "Caller can issue a read request and receive a synchronous response (HTTP GET, MCP query)."

Both poll and webhook use `EndpointProtocol.HTTP`. Without the correct capability, there is no way to distinguish them in `EndpointRegistry.discover()` queries.

## Fix

Use `EndpointCapability.QUERY` for poll targets:

```java
// WRONG — poll target registered as RECEIVE
EndpointDescriptor wrong = new EndpointDescriptor(
    path, tenancyId, EndpointType.SYSTEM, EndpointProtocol.HTTP,
    Map.of(...), null,
    Set.of(EndpointCapability.RECEIVE));  // incorrect for pull/GET

// CORRECT — poll target registered as QUERY
EndpointDescriptor correct = new EndpointDescriptor(
    path, tenancyId, EndpointType.SYSTEM, EndpointProtocol.HTTP,
    Map.of(...), null,
    Set.of(EndpointCapability.QUERY));   // HTTP GET initiated by us
```

The poll module discovers endpoints with `EndpointCapability.QUERY`; webhook registers with `EndpointCapability.RECEIVE`. Both use `HTTP` protocol — capability is the only discriminator.

## Why this is non-obvious

"Receiving data" semantically covers both patterns. The distinction ("who initiates?") only surfaces when you read the Javadoc carefully: RECEIVE = endpoint pushes to caller; QUERY = caller issues a read request. The failure mode is silent — wrong module discovery, no error, no warning.
