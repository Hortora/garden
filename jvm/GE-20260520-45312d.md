### Symptom
A `@QuarkusTest` that seeds entries via `repo.save()` and asserts SUSPECT / VALID
signature outcomes passes — even though the test never sets `agentSignature` on the
entry and no `@InjectMock` is present for the key provider.

### Context
Any Quarkus project with a `@EntityListeners` enricher (e.g. a signing enricher
that runs at `@PrePersist` to auto-populate signature fields) where the enricher's
behaviour is gated by a CDI-injected provider that returns `Optional.empty()` when
no config key is present.

### What was tried
`@InjectMock AgentKeyProvider` was omitted as seemingly unnecessary — "the test works
without it." It does pass — but only because no signing key is configured in
`test/resources/application.properties`.

### Root cause
The enricher runs at `@PrePersist` for every `repo.save()` call. With no signing key
configured, the CDI provider returns `Optional.empty()` and the enricher is effectively
a no-op — entries come back unsigned. Tests relying on unsigned entries pass because
the enricher's effect never surfaces, not because it was isolated.

If someone later adds a signing key to `application.properties` (for an unrelated
test that needs a signed entry), all tests that seed "unsigned" entries will silently
produce signed ones. SUSPECT-path tests that manually set `agentKeyRef` will fail
because the enricher embedded a different key — with no clear error indicating why
the keyRef no longer matches the compromised-window lookup.

### Fix
Add `@InjectMock` for the key provider to every `@QuarkusTest` class that seeds
entries, and configure it in `@BeforeEach`:

```java
@InjectMock
AgentKeyProvider agentKeyProvider;

@BeforeEach
void setUp() throws Exception {
    // Default: no actor gets a signing key — enricher is an explicit no-op
    when(agentKeyProvider.signingKey(anyString())).thenReturn(Optional.empty());
    // Override for actors that need a real key for specific tests:
    // when(agentKeyProvider.signingKey("claude:reviewer@v1"))
    //     .thenReturn(Optional.of(SigningKey.of(testKeyPair)));
}
```

This makes the enricher's no-op behaviour an explicit, config-immune guarantee.

### Why this is non-obvious
The test passes without the mock. There is no failure, warning, or log output
indicating that the enricher ran or was silently bypassed. The failure mode only
surfaces when test configuration is later extended — potentially in a different
test class, weeks later. The connection between the new config entry and the
newly-broken test is non-obvious, and the failing assertion (wrong VerificationResult
or wrong keyRef) does not mention the enricher.
