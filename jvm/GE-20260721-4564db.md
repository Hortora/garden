### Symptom
`@Transactional(REQUIRES_NEW)` on a `CaseOutcomeObserver.onOutcome()` implementation silently fails to open a transaction. Persistence operations inside the method throw `ContextNotActiveException` or similar JTA errors, but the engine catches and logs them as warnings — the case completes normally, and the observer's side effects (ledger writes, CBR retention) are silently lost.

### Context
The engine's `CaseStatusChangedHandler.fireOutcomeObservers()` iterates `Instance<CaseOutcomeObserver>` and calls `onOutcome()` on each observer directly. This happens on executor threads (e.g. `executor-thread-17`), not on a request thread with an active CDI request context. The `@Transactional` CDI interceptor requires an active context to intercept method calls — on executor threads without a request context, the interceptor chain may not activate.

### What was tried (didn't work)
- `@Transactional(TxType.REQUIRES_NEW)` as a method annotation on `onOutcome()` — silently fails to start a transaction on executor threads.

### Root cause
CDI interceptors (including `@Transactional`) are activated through proxy-based interception. When beans are retrieved via `Instance<T>` and called on non-request threads, the interceptor chain depends on the thread's CDI context state. Executor threads used by the engine's Vert.x event loop or Quartz scheduler may not have the full CDI context needed for `@Transactional` to intercept. The `@Transactional` annotation is silently ignored — no error, no warning — and the method body runs without a transaction.

### Fix
Use `QuarkusTransaction.requiringNew().run(...)` programmatically for any persistence operations inside `CaseOutcomeObserver.onOutcome()`:

```java
@Override
public void onOutcome(CaseOutcomeEvent event) {
    // CBR store.store() — no transaction needed (in-memory)
    cbrStore.store(cbrCase, ...);

    // Ledger write — needs transaction
    QuarkusTransaction.requiringNew().run(() -> {
        var entry = new MyLedgerEntry();
        // ... populate fields
        ledgerRepository.save(entry, tenantId);
    });
}
```

`QuarkusTransaction.requiringNew()` explicitly manages the transaction boundary regardless of the calling thread's CDI context state.

### Why this is non-obvious
The `@Transactional` annotation works correctly on `@Observes` and `@ObservesAsync` CDI event methods because the CDI event dispatch mechanism ensures proper interceptor chain activation. Developers familiar with CDI events naturally expect `@Transactional` to work on any CDI bean method — but `Instance<T>` iteration is a different dispatch path that doesn't guarantee the same interceptor activation on all thread types.
