## Two @DefaultBean Implementations of the Same CDI Type Cause Ambiguity at Quarkus Augmentation

### Symptom

Quarkus augmentation fails with:

```
AmbiguousDependencies for type Foo and qualifiers [@Default]
  - FooImpl @DefaultBean  (in library JAR)
  - MockFooImpl @DefaultBean  (in casehub-platform-testing JAR)
```

All `@QuarkusTest` and production builds in consumer modules fail. The error gives no hint
that the two beans are from different JARs — it looks like a duplicate class problem.

### Context

A library ships a real HTTP-aware implementation:

```java
@ApplicationScoped
@DefaultBean   // ← wrong
public class QhorusInboundCurrentPrincipal implements CurrentPrincipal { ... }
```

A separate testing JAR ships a mock:

```java
@ApplicationScoped
@DefaultBean
public class MockCurrentPrincipal implements CurrentPrincipal { ... }
```

When both JARs land on the same classpath in a consumer app, ARC sees two `@DefaultBean`
beans for the same type and throws `AmbiguousDependencies`. The mistake is annotating the
real implementation with `@DefaultBean` — that annotation is intended for mocks/fallbacks,
not production implementations.

### Root cause

`@DefaultBean` means "I yield to any non-default qualifying bean." When two beans both carry
`@DefaultBean`, neither yields to the other — CDI resolution is ambiguous.

The CDI priority ladder for libraries:

| Tier | Annotation | Purpose |
|------|-----------|---------|
| 1 — Mock | `@DefaultBean` | Fallback that yields to any real impl |
| 2 — Primary | `@ApplicationScoped` (plain) | Real implementation; beats mock automatically |
| 3 — Secondary | `@Alternative @Priority(1)` | Specialised override; beats primary |

The real implementation belongs at Tier 2, not Tier 1.

### Fix

Remove `@DefaultBean` from the real (non-mock) implementation. Plain `@ApplicationScoped`
is the correct annotation for a library's primary implementation:

```java
@ApplicationScoped   // ← plain, no @DefaultBean
public class QhorusInboundCurrentPrincipal implements CurrentPrincipal { ... }
```

`MockCurrentPrincipal @DefaultBean` then yields to `QhorusInboundCurrentPrincipal @ApplicationScoped`
automatically — no `quarkus.arc.exclude-types` workaround needed.

### Why this is non-obvious

`@DefaultBean` sounds like "I am the default implementation" (primary). It actually means
"I am a fallback — any other qualifying bean takes priority." Using it on a real implementation
produces a bean that cannot coexist with any other `@DefaultBean` for the same type, including
the mock it was intended to work alongside. The name is actively misleading in this pattern.
