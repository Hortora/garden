## Symptom

A custom exception type has a registered `@Provider ExceptionMapper<T>` that returns a structured JSON 400 response. The response body at runtime uses the wrong format — a generic message string instead of the structured JSON — as if the mapper never fired.

## Context

`MalformedCapabilityException extends IllegalArgumentException`. The JAX-RS resource method `create()` has a generic `catch (IllegalArgumentException e)` handler that produces a simplified 400 response. The `@Provider` mapper is correctly annotated and handles `MalformedCapabilityException` specifically.

The mapper is registered and would fire correctly if the exception escaped the resource method — but the generic catch intercepts it first.

## Root Cause

`@Provider` exception mappers are invoked by the JAX-RS container only for **uncaught** exceptions. If the resource method catches the exception internally (even in a finally block or a broad catch), the container never sees it. When `MalformedCapabilityException` is thrown, the JVM matches it against `catch (IllegalArgumentException e)` before any container-level dispatch occurs.

## Fix

Add a specific rethrow for the custom types immediately before the generic handler:

```java
@POST
public Response create(CreateWorkItemRequest request) {
    try {
        return workItemService.create(request);
    } catch (MalformedCapabilityException | UnknownCapabilityException e) {
        throw e;  // ← let the @Provider mapper handle these
    } catch (IllegalArgumentException e) {
        return Response.status(400).entity(Map.of("error", e.getMessage())).build();
    }
}
```

The `@Provider` mappers then receive the exception and produce the structured response.

## Why This Is Non-Obvious

`@Provider` mappers feel like they override resource-level handling — they are registered globally and seem authoritative. The JAX-RS spec says mappers handle exceptions from resource methods, but "from resource methods" means *uncaught* exceptions. A catch-and-ignore (or catch-and-produce-response) inside the resource method is invisible to the container. There is no warning or error when the mapper is bypassed.
