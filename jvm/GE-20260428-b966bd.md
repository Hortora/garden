## Symptom

Event-driven integration test passes in isolation but fails in the full suite. The test signals completion via a CDI async event, waits on a `CompletableFuture`, then asserts state via a registry lookup — and sees the wrong state (e.g. `RUNNING` instead of `COMPLETED`), despite the future having completed correctly. Test elapsed time is suspiciously short (< 100ms) because the future resolved immediately.

## Context

Two Vert.x `@ConsumeEvent` handlers are registered on the same event bus address (fan-out via `publish()`). One handler marks an item complete and fires a CDI async event. Another handler updates a shared mutable index (`Map.put(key, itemId)`) after applying context changes that may re-trigger the workflow. The re-trigger creates a second item and overwrites the index with its ID before the test reads it.

## What was tried (didn't work)

- Bumping awaitility timeout (15s → 30s) — reduced flake frequency but did not eliminate it
- Waiting for a case-level completion event (`CaseLifecycleEvent("CaseCompleted")`) — never fired because the case had no success goal defined and stayed `RUNNING` indefinitely
- Carrying `workerName` only in the CDI event and doing a fresh `getPlanItemId(workerName)` lookup after the future resolved — returned the re-triggered item (RUNNING state)

## Root cause

`indexWorkerForCompletion(caseId, workerName, planItemId)` used a plain `Map.put()`. When a sibling component completed and its output triggered re-evaluation — before the current component's output was applied to the context — the engine created a second item for the same worker and overwrote the index entry. The CDI async event fired after `markCompleted()` on the *first* item (correct), but the post-event `getPlanItemId()` lookup returned the *second* item's ID which was still `RUNNING`.

The `@ApplicationScoped` observer's `ConcurrentHashMap` persists across tests in the same Quarkus session, so the future resolved immediately from a key match, masking the fact that the subsequent lookup was reading overwritten state.

## Fix

Carry the exact `planItemId` in the CDI event payload. The test reads the stable ID from the `CompletableFuture<String>` value and verifies that specific item — bypassing the index entirely.

```java
// Handler fires with stable planItemId, not just workerName
planItemCompletedEvents.fireAsync(
    new PlanItemCompletedEvent(caseId, planItemId, workerName));

// Observer completes future with the stable ID as the value
void onPlanItemCompleted(@ObservesAsync PlanItemCompletedEvent event) {
    futures.computeIfAbsent(key(event.caseId(), event.workerName()),
                            k -> new CompletableFuture<>())
           .complete(event.planItemId());   // ← carries the stable ID
}

// Test uses the ID from the future — not a fresh registry lookup
String bPlanItemId = bFuture.get();
assertThat(plan.get().getPlanItem(bPlanItemId).map(i -> i.getStatus()))
    .contains(PlanItemStatus.COMPLETED);
```

## Why this is non-obvious

The test appeared to work correctly: the future completed, proving `markCompleted()` was called on some item for that worker. The failure was in the *assertion step* — a fresh index lookup that most developers consider safe after an event-driven signal. The race is invisible: a concurrent re-trigger from a different part of the system overwrites the index between CDI async event delivery and the test reading the index. Running in isolation hides it because there is no concurrent sibling context update.
