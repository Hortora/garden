## Maven mvn test in a child module silently skips sibling example/integration modules — need mvn install from root

### Symptom

Running `mvn test` from inside a child module (e.g. `runtime/`) passes cleanly. A consumer then
reports a broken build. The failures are compile errors in sibling modules (e.g. `examples/type-system`,
`examples/agent-communication`) that depend on the child module via a local SNAPSHOT.

### Context

In a Maven multi-module layout where `examples/` is a sibling of `runtime/` under a common parent:
```
parent/
├── runtime/      ← where you ran mvn test
├── examples/
│   ├── type-system/
│   └── agent-communication/
```

Running `mvn test` inside `runtime/` only compiles and tests that module. Sibling modules that depend
on `runtime/` through `casehub-qhorus` (a local SNAPSHOT) are not compiled or tested. Compile errors
in those siblings are invisible until someone runs the full build.

### What was tried

After a large API change (visibility of method overloads), `mvn test` from `runtime/` passed
with 968 tests green. The build was committed and pushed. A separate `mvn install` from root then
revealed 20+ compile errors across `examples/type-system` and `examples/agent-communication`.

### Root cause

`mvn test` in a child module is scoped to that module only. Sibling modules — even if they
depend on the child — are not in the reactor for that invocation.

### Fix

For API changes that affect consumers, always validate from the project root:

```bash
# From the repo root — compiles and tests ALL modules
JAVA_HOME=... mvn install

# Or to test a module and all modules that depend on it:
JAVA_HOME=... mvn test -pl runtime,examples/type-system,examples/agent-communication
```

As a CI discipline: the root `mvn install` (or equivalent) must be the gate, not a per-module
`mvn test`.

### Why this is non-obvious

Running tests "from the module you changed" feels correct — it is the standard fast-feedback
workflow. The failure mode is silent: no warning that sibling modules were skipped, and the local
test run is genuinely green. The broken build only surfaces when a different developer (or CI) runs
from root.
