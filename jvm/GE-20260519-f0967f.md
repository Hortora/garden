### Context
A Quarkus extension defines a reactive SPI returning `Uni<T>` (e.g.
`ReactiveLedgerEntryRepository`). The `@QuarkusTest` suite runs against H2/JDBC —
no Vert.x reactive datasource is available. CDI fails to start because nothing
satisfies the `Uni`-returning injection point.

### Technique
Register a `@DefaultBean @ApplicationScoped` shim in test sources that implements
the reactive SPI by wrapping the blocking implementation with `Uni.createFrom().item()`:

```java
@DefaultBean
@ApplicationScoped
class BlockingReactiveKeyRotationRepository implements ReactiveKeyRotationRepository {

    @Inject
    KeyRotationRepository blocking;

    @Override
    public Uni<List<KeyRotationEntry>> findByActorId(String actorId) {
        return Uni.createFrom().item(() -> blocking.findByActorId(actorId));
    }
}
```

`@DefaultBean` means the shim is used only when no other bean satisfies the injection
point — in production a consumer provides a real reactive implementation and the shim
is ignored. The shim stays in `src/test/java` and never ships.

### Why this is non-obvious
The obvious approaches are: (a) add `quarkus-hibernate-reactive` and a reactive
datasource to the test config — complex, slow, requires a running database; or (b)
use Mockito to mock the SPI — couples tests to implementation details. The shim
pattern requires none of that and is fully transparent to the tests under evaluation.
The key insight is `@DefaultBean` — without it the shim and the real implementation
would conflict at CDI resolution time.
