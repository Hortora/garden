## Symptom

After adding a `UNIQUE(subject_id, sequence_number)` constraint to the base `ledger_entry` table, an `@ObservesAsync` CDI observer that persists an `ActorIdentityBindingEntry` (a `LedgerEntry` subclass) starts failing with `ConstraintViolationException: UNIQUE index violation ... VALUES (UUID, 0)`. The `sequence_number = 0` reveals the entry was persisted without sequence assignment.

## Context

`LedgerEntry` uses JPA JOINED inheritance. `JpaLedgerEntryRepository.save()` assigns `sequenceNumber` via an atomic MERGE on a sequence table. `ActorIdentityBindingEntry extends LedgerEntry` but is persisted through `JpaActorIdentityBindingRepository.save()` — a separate repository that calls `em.persist()` directly without sequence assignment. Before the UNIQUE constraint, both entries silently got `sequence_number = 0`. After the constraint, the second entry for the same subject violates uniqueness.

## Root Cause

JPA JOINED inheritance stores all subclass rows in the same base table. Any invariant enforced in the base repository's `save()` (sequence assignment, pseudonymisation, compliance sanitisation) is silently bypassed when a subclass has its own repository with a direct `em.persist()`. There is no JPA mechanism that forces all subclasses through the same save pipeline — it is a convention that must be enforced architecturally.

## Fix

Extract shared save-pipeline logic into a CDI bean that all repositories inject:

```java
@ApplicationScoped
class LedgerSequenceAllocator {
    @Inject @LedgerPersistenceUnit EntityManager em;
    
    int nextSequenceNumber(UUID subjectId) {
        // MERGE + SELECT
    }
}

// Both repositories inject the allocator:
class JpaLedgerEntryRepository {
    @Inject LedgerSequenceAllocator sequenceAllocator;
    
    public LedgerEntry save(LedgerEntry entry) {
        entry.sequenceNumber = sequenceAllocator.nextSequenceNumber(entry.subjectId);
        em.persist(entry);
    }
}

class JpaActorIdentityBindingRepository {
    @Inject LedgerSequenceAllocator sequenceAllocator;
    
    public ActorIdentityBindingEntry save(ActorIdentityBindingEntry entry) {
        entry.sequenceNumber = sequenceAllocator.nextSequenceNumber(entry.subjectId);
        em.persist(entry);
    }
}
```

## Why This Is Non-Obvious

The subclass repository looks correct — it persists a valid entity. The bypass is invisible because: (1) the base table has no constraint that would surface the missing invariant; (2) `em.persist()` succeeds silently with default field values; (3) the subclass and base class are in different packages with different CDI lifecycles; (4) the problem only manifests when a DB constraint (UNIQUE, CHECK) or cryptographic verification makes the missing invariant observable. Without such a constraint, the data corruption is silent and permanent.
