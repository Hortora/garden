### Symptom
`mvn clean compile` succeeds with no errors or warnings, but the compiled classes use the wrong generic signatures from a dependency. For example, `AgentWorkerFunction implements WorkerFunction<Map<String, Object>>` compiles against a two-parameter `WorkerFunction<T, R>` — Java should reject this as "wrong number of type arguments; required 2" but doesn't because Maven resolved a different (older) jar.

### Context
Project uses `.mvn/maven.config` with `-Dmaven.repo.local=/path/to/worktrees/N/.m2` to isolate Maven dependencies per worktree. The dependency (`casehub-worker-api`) was updated in a sibling repo and installed with `mvn install` — but this installs to `~/.m2/repository/`, not the worktree-local repo.

The worktree-local `.m2/` contains both a timestamp-versioned remote jar (from GitHub Packages) and the locally installed SNAPSHOT jar. Maven's SNAPSHOT resolver prefers the remote timestamp-versioned jar over the local `-SNAPSHOT.jar` when remote metadata says it's authoritative.

### What was tried (didn't work)
- `mvn clean compile` — succeeded (wrong jar on classpath)
- `mvn compile -o` (offline) — succeeded (remote jar already cached locally)
- Deleting the remote jar from worktree `.m2/` — still resolved the old jar from `_remote.repositories` metadata
- `javap -v` on the compiled class — this IS how the problem was detected (generic signature showed one type param instead of two)

### Root cause
Maven SNAPSHOT resolution uses `_remote.repositories` markers and remote `maven-metadata.xml` timestamps to decide which physical jar file to use for a given SNAPSHOT artifact. When the worktree-local repo has a remote-origin jar (with timestamp) AND a locally installed jar (`-SNAPSHOT.jar`), Maven may prefer the remote one — even if the local one is newer on disk.

### Fix
Install the dependency directly to the worktree-local repo:
```bash
mvn -f /path/to/dependency/pom.xml install -DskipTests -Dmaven.repo.local=/path/to/worktrees/N/.m2
```

Or delete the stale remote jar and its metadata from the worktree cache:
```bash
rm /path/to/worktrees/N/.m2/io/group/artifact/VERSION/artifact-VERSION-TIMESTAMP.jar
rm /path/to/worktrees/N/.m2/io/group/artifact/VERSION/_remote.repositories
```

### Detection
`javap -v` on the compiled class reveals the actual generic signature used during compilation. If the signature doesn't match the source code, the wrong jar was on the classpath:
```bash
javap -v target/classes/com/example/MyClass.class | grep Signature
```

### Why this is non-obvious
`mvn clean compile` produces zero errors and zero warnings. The compilation succeeds because Java treats mismatched type parameter counts as a raw-type situation with unchecked warnings — but the default compiler configuration suppresses those warnings. The only observable symptom is runtime `ClassCastException` or assertion failures in tests, hours after the wrong jar was silently resolved.

### Alternatives considered
- Using `-U` (force update snapshots) — doesn't help because the problem is local vs remote priority, not staleness
- Using `-o` (offline) — doesn't help because the stale remote jar is already in the local cache
