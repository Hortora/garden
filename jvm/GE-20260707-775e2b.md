### Symptom
Lifecycle phase transitions never trigger. `CompletionCondition.allPresent()` returns
true when tested manually, but the `ReconciliationListener` callback never fires in
the stable-state scenario.

### Context
An event-driven reconciliation loop (or any periodic poll-and-act loop) has an
early-return optimisation: when the computed plan is empty (nothing to do), the cycle
exits before execution, fault feedback, and event emission. A post-cycle listener
added "at the end of the cycle" — after execution — is placed after the early return.

### Root cause
The completion condition (`allPresent()`) is satisfied precisely when all nodes are
stable and the plan is empty — the exact condition that triggers the early return.
The listener, placed after the early return, never fires on the cycle where completion
is achieved.

### Fix
Place the listener call after `readActual()` and **before** the empty-plan early return:

```java
private void reconcile() {
    DesiredStateGraph desired = desiredRef.get();
    ActualState actual = readActual(desired, tenancyId);

    // Listener fires unconditionally — including empty-plan cycles
    fireListener(desired, actual);

    // ... detectDrift, plan, early return if empty, execute ...
}
```

### Why this is non-obvious
"Fire the listener at the end of the cycle" is the natural mental model — after all
work is done. But "end of the cycle" and "after the early return" are different
positions. The optimisation that makes the loop efficient (skip empty plans) is
precisely what makes the naive listener placement wrong.

### Alternatives considered
- Remove the early return entirely (costs performance — every stable cycle runs
  full execution/fault/event paths for nothing)
- Use a separate timer-based completion check (adds complexity, timing sensitivity,
  and a gap where nothing is being reconciled during the transition)
