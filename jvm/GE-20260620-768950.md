## WebSocket buildAsync().join() completes client-side before Quarkus @OnOpen fires — causes silent broadcast drop on slow CI

**ID:** GE-20260620-768950
**Stack:** Java 11+ java.net.http.WebSocket, Quarkus WebSockets Next, any CI runner slower than local dev machine
**Symptom:** A `@QuarkusTest` that connects a WebSocket, calls `engine.observe()` (which triggers a broadcast), then polls for the message — passes reliably on local dev machines but hangs for the full poll timeout on CI. No error, no warning, no log indicating anything went wrong. Different test methods fail on different CI runs (because the race window varies with runner load).
**Context:** Test pattern: `connect()` → `ws.request(1)` → `engine.observe()` → `queue.poll(timeout)`. The broadcaster skips the send when `sessions.isEmpty()` — which is the correct guard for non-test scenarios but creates the race in tests.

### What was tried (didn't work)

- Increased poll timeout from 3s to 10s — same timeout hit on next CI run, different test method
- Assumed the issue was slow message delivery — it was actually no delivery at all

### Root cause

`HttpClient.newWebSocketBuilder().buildAsync(uri, listener).join()` returns when the **client-side** WebSocket handshake completes (HTTP 101 upgrade received). The Quarkus server-side `@OnOpen` handler fires asynchronously on a Vert.x event loop thread *after* the 101 is sent. On slow CI runners the gap is wide enough that:

1. `connect().join()` returns — client sees connection established
2. `ws.request(1)` is called — client ready to receive
3. `engine.observe()` fires — broadcaster calls `onFrame()`
4. `onFrame()` checks `sessions.isEmpty()` → **true** — broadcast dropped silently
5. Quarkus `@OnOpen` fires — `sessions.add()` too late

The test then polls for a message that was never sent. No error — the broadcaster's `sessions.isEmpty()` guard is a legitimate optimisation, not a bug.

### Fix

Add a `waitForSession()` barrier to the broadcaster and call it in the test's connect helper after the handshake:

```java
// In GameStateBroadcaster (production class):
public void waitForSession(long maxWaitMs) throws InterruptedException {
    long deadline = System.currentTimeMillis() + maxWaitMs;
    while (sessions.isEmpty() && System.currentTimeMillis() < deadline) {
        Thread.sleep(10);
    }
}

// In test connect() helper:
private WebSocket connect(LinkedBlockingQueue<String> queue) throws InterruptedException {
    URI ws = URI.create(wsUri.toString().replace("http://", "ws://"));
    WebSocket result = HttpClient.newHttpClient()
        .newWebSocketBuilder()
        .buildAsync(ws, new MessageCollector(queue))
        .join();
    broadcaster.waitForSession(2000); // wait for @OnOpen to fire
    return result;
}
```

The session is typically registered within a few milliseconds. The 2s cap handles even very slow runners.

### Why this is non-obvious

`buildAsync(...).join()` *looks* synchronous — `.join()` blocks until the future completes. What completes is the *client-side* handshake (sending the upgrade request and receiving the 101). The server's application-layer `@OnOpen` runs later on a separate thread. There is no API to wait for the server-side callback without adding explicit synchronisation. Increasing the poll timeout doesn't help because the message was never sent — it just makes the test hang longer before failing.
