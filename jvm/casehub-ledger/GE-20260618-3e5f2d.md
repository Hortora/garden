### Symptom

`org.hibernate.DuplicateMappingException: Entity classes [io.casehub.ledger.runtime.model.ErasureReceiptLedgerEntry] and [io.casehub.devtown.app.ledger.ErasureReceiptLedgerEntry] share the entity name 'ErasureReceiptLedgerEntry'`

All `@QuarkusTest` tests fail at startup — Hibernate refuses to build the persistence unit.

### Context

devtown had its own `ErasureReceiptLedgerEntry extends LedgerEntry` for GDPR erasure receipts. Later, casehub-ledger shipped its own `ErasureReceiptLedgerEntry` in its runtime module with different fields. Both use `@Entity` without an explicit `name` attribute — Hibernate uses the simple class name as the entity name, causing collision.

### Root cause

The `@Entity` annotation without an explicit `name` uses the unqualified class name. Two classes with the same simple name in different packages get the same entity name. Hibernate validates entity names globally at startup and rejects duplicates.

### Fix

Disambiguate with an explicit entity name:

```java
@Entity(name = "DevtownErasureReceiptLedgerEntry")
@Table(name = "devtown_erasure_receipt_ledger_entry")
@DiscriminatorValue("DEVTOWN_ERASURE_RECEIPT")
public class ErasureReceiptLedgerEntry extends LedgerEntry { ... }
```

Also update any `@NamedQuery` references to use the disambiguated entity name.

The proper fix is consolidating onto the foundation's entity — the application-tier duplicate should be deleted entirely once the foundation's entity covers all needed fields.

### Why this is non-obvious

The collision is silent until Quarkus startup — no compile-time warning. The error message names both classes but doesn't suggest the fix. The application-tier entity was created before the foundation shipped its own — the collision appears only after a dependency version bump.
