### Symptom
Mockito verify calls report "zero interactions with this mock" on all downstream
mocks (EventBus, SchedulerService, etc.) even though `handleFailure()` was called
and should have triggered the full reactive chain.

### Context
Unit testing a CDI bean (`QuartzRetryService`) that internally calls
`ReactiveUtils.runOnSafeVertxContext(vertx, () -> eventLogRepository.append(...))`.
Vertx was declared as `@Mock Vertx vertx` — standard Mockito practice.

### What was tried (didn't work)
- Verified mock stubs were correct — they were
- Checked Uni subscription — `subscribe().with()` was called
- Suspected async timing — added Thread.sleep, no change
- Suspected Mockito strict stubbing — switched to lenient, no change

### Root cause
`ReactiveUtils.runOnSafeVertxContext` calls `VertxContext.getOrCreateDuplicatedContext(vertx)`.
With a Mockito mock, this returns `null`. The subsequent `dc.runOnContext(v -> command.run())`
silently does nothing — the command lambda is never invoked, the deferred Uni supplier never
runs, and the entire reactive pipeline is dead. No exception, no warning, no log.

### Fix
Replace `@Mock Vertx vertx` with a real lightweight Vertx instance:

```java
private static Vertx vertx;

@BeforeAll
static void startVertx() {
    vertx = Vertx.vertx();
}

@AfterAll
static void stopVertx() throws Exception {
    if (vertx != null) {
        CountDownLatch latch = new CountDownLatch(1);
        vertx.close().onComplete(ar -> latch.countDown());
        latch.await(5, TimeUnit.SECONDS);
    }
}
```

The real Vertx instance creates a valid duplicated context, the command runs,
and the reactive chain fires as expected.

### Why this is non-obvious
Every other CDI dependency in the test class is safely mockable. Vertx looks like
just another injected service. Nothing in the API, documentation, or error output
hints that `runOnSafeVertxContext` requires a real Vertx instance. The failure mode
is complete silence — the chain simply doesn't execute, and the misleading
"zero interactions" error points you toward mock setup problems rather than the
Vertx context issue.

### Alternatives considered
- **Refactoring to remove Vertx dependency from the service** — rejected; `runOnSafeVertxContext` is needed in production for Hibernate Reactive session context
- **Using `@QuarkusTest`** — rejected; too heavy for pure unit tests of retry logic
