## Symptom

After generating `ARC42STORIES.MD §9.4` Key files sections from `LAYER-LOG.md`, a class is listed that cannot be found anywhere in the project:
```bash
find . -name "HumanApprovalLifecycleHandler.java"  # no output
```

The class name sounds right — it names the correct concept — but does not exist in the application's production source.

## Context

When migrating a CaseHub harness application's `LAYER-LOG.md` to `ARC42STORIES.MD`, layer entries name classes that implement the layer's integration. Some of those class names come from design documents or early LAYER-LOG drafts and may differ from what was actually built. Foundation modules (like `casehub-work-adapter`) provide some integration classes that look like application classes but belong to the foundation JAR, not the app.

In the devtown migration, `HumanApprovalLifecycleHandler` appeared in 3 places in the document. The actual bridge from `WorkItemLifecycleEvent` to `PlanItem` is provided by `casehub-work-adapter` (a foundation extension) — no devtown production class owns this behaviour.

## What was tried

The class was retained in the document because:
- The concept (WorkItemLifecycleEvent → PlanItem bridge) is architecturally real
- The name follows the expected CamelCase pattern for the type
- The LAYER-LOG entry used this name

Only a `find` sweep of the project source confirmed the class doesn't exist.

## Root cause

Design documents and LAYER-LOG entries name classes by concept and intention. The actual production code may use a different name, have the behaviour provided by a foundation module, or have refactored the class away. There is no automatic consistency check between documentation and code during the migration.

## Fix

For every class name in ARC42STORIES.MD §9.4 Key files sections, run:
```bash
find /path/to/project -name "ClassName.java" 2>/dev/null
```

If no output: the class does not exist in the application. Either:
1. The class lives in a foundation module — update the documentation to say "provided by `[foundation-module]` (not a devtown class)"
2. The class was renamed — find the real name with `find . -name "*.java" | xargs grep -l "interface\|class.*implements.*PortName"`
3. The class was never built — remove from Key files; document the gap

Run this check before marking any layer entry complete. In the devtown case, the fix was renaming the container in the C4 diagram and all three Key files references to `SlaBreachHandler` (the actual devtown class) and noting that the WorkItem→PlanItem bridge is `casehub-work-adapter`.

## Why this is non-obvious

The concept is correct — the integration point genuinely exists at that location in the architecture. The class name is plausible given naming conventions. Nothing in the migration process surfaces the mismatch — it's only visible by searching the codebase. Design documents predate the final implementation and often use aspirational names that didn't survive the build.
