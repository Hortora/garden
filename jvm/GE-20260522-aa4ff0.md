## ArchUnit cross-class ArchCondition: capture JavaClasses in outer scope to look up related classes inside check()

### What it achieves

Enables an `ArchCondition` to look up related classes (e.g. find the blocking counterpart of a reactive class) from within `check()`, which only receives one subject at a time.

### Context

ArchUnit's `ArchCondition<JavaClass>` API exposes `check(JavaClass subject, ConditionEvents events)` — one class at a time. For structural parity rules that compare pairs of classes (e.g. every `Reactive*Service` must have a matching `*Service` with mirrored methods), the condition needs access to the full imported class set to look up the counterpart.

### The technique

Capture the full `JavaClasses` result from `ClassFileImporter` in an outer variable. Pass it into the condition factory method. The anonymous `ArchCondition` closes over it:

```java
private static final JavaClasses SERVICE_CLASSES = new ClassFileImporter()
        .importPackages(KeyRotationService.class.getPackageName());

@Test
void reactiveServices_mustMirrorBlockingCounterpart() {
    classes()
            .that().haveSimpleNameStartingWith("Reactive")
            .and().haveSimpleNameEndingWith("Service")
            .should(haveBidirectionalMethodParity(SERVICE_CLASSES))  // pass full set
            .check(SERVICE_CLASSES);
}

private static ArchCondition<JavaClass> haveBidirectionalMethodParity(
        final JavaClasses allClasses) {  // captured in outer scope
    return new ArchCondition<>("have complete bidirectional method parity") {
        @Override
        public void check(final JavaClass reactiveClass, final ConditionEvents events) {
            // Look up blocking counterpart from the full class set
            final String blockingName = reactiveClass.getSimpleName()
                    .replaceFirst("^Reactive", "");
            final Optional<JavaClass> blockingOpt = allClasses.stream()
                    .filter(c -> c.getSimpleName().equals(blockingName))
                    .findFirst();
            // ... enforce parity
        }
    };
}
```

`SERVICE_CLASSES` is a `static final` field — `ClassFileImporter` is moderately expensive and the result is shared across all test methods.

### Why this is non-obvious

ArchUnit's `@ArchTest` static field pattern is documented for self-contained rules, but cross-class conditions that need the full import set require passing `JavaClasses` explicitly — this is not covered in the main ArchUnit user guide. Developers typically discover this pattern only after finding that `check()` cannot call back into the test runner to look up other classes.
