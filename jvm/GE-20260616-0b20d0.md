### What happens

The Java service places keys in the engine case context map (e.g. `Map.of("aeId", ..., "indReportingDeadline", ...)`). The YAML case definition's `inputSchema` is a JQ projection applied before handing input to the capability agent. Any key in the Java map that is NOT declared in `inputSchema` is silently dropped — the engine blackboard holds it, but the agent input does not. No error is thrown. The agent receives `null` or a missing key.

### Example

```java
// Java service — adds indReportingDeadline to context
return Map.of(
    "aeId", ae.id.toString(),
    "grade", ae.grade.name(),
    "indReportingDeadline", ae.reportedAt.plus(window).toString());  // ← present in context
```

```yaml
# YAML case definition — indReportingDeadline NOT in inputSchema
capabilities:
  - name: regulatory-submission
    inputSchema: "{ grade: .grade, aeId: .aeId }"  # ← indReportingDeadline silently dropped
```

The agent receives `{ "grade": "GRADE_3", "aeId": "..." }` — `indReportingDeadline` is absent. No error.

### Fix

Verify the full chain: every key the agent needs must appear in BOTH the Java context map AND the YAML `inputSchema`:

```yaml
inputSchema: "{ grade: .grade, aeId: .aeId, indReportingDeadline: .indReportingDeadline }"
```

### Why this is non-obvious

The Java side and YAML side are edited independently. There is no compile-time link between the Java `Map.of()` keys and the YAML `inputSchema` fields. The engine accepts the case definition without warning. The agent silently operates on incomplete input — a runtime failure that only surfaces if the agent inspects the missing field, and only then with a confusing null rather than a clear error.

Note: this is distinct from GE-20260417-4a3c22 ("Worker lambda receives null for context fields added to inputSchema") which describes a serialization issue where keys ARE in inputSchema but fail to survive event log round-trips. This entry describes keys that were never in inputSchema at all.
