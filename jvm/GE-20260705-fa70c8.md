### Symptom

UUIDv7 IDs are not monotonically increasing. Two UUIDs generated within the same millisecond can sort in the wrong order, breaking cursor pagination or ordering guarantees. No error — the IDs look valid but sort incorrectly.

### Context

A UUIDv7 generator uses a 12-bit per-thread sequence counter for same-millisecond ordering. When 4096+ UUIDs are generated in the same millisecond on the same thread, the sequence wraps. The naive fix (advance timestamp by 1ms on overflow) introduces a subtle clock regression vulnerability.

### What was tried (didn't work)

Using `==` comparison to detect same-millisecond calls:

```java
if (timestampMs == state.lastTimestamp) {
    state.sequence++;
    if (state.sequence > 0xFFF) {
        timestampMs++;                  // advance synthetic timestamp
        state.lastTimestamp = timestampMs;
        state.sequence = 0;
    }
}
```

This handles sequence wraparound correctly but fails on the NEXT call: after advancing `lastTimestamp` to T+1, the next call within the same real millisecond sees `timestampMs=T`, which is NOT equal to `lastTimestamp=T+1`. The `==` check falls through to the `else` branch, resetting `lastTimestamp` to T and `sequence` to 0 — regressing the timestamp and producing a UUID that sorts before the previous one.

### Root cause

The `==` comparison only handles the "same timestamp" case. After sequence wraparound advances `lastTimestamp` to a synthetic future value, subsequent calls at the real timestamp see a LOWER value than `lastTimestamp`. The `==` check misses this — it needs to handle both "same timestamp" and "clock went backwards" as the same case.

### Fix

Use `<=` comparison per RFC 9562 §6.2:

```java
if (timestampMs <= state.lastTimestamp) {
    timestampMs = state.lastTimestamp;    // never go backwards
    state.sequence++;
    if (state.sequence > 0xFFF) {
        timestampMs++;
        state.lastTimestamp = timestampMs;
        state.sequence = 0;
    }
} else {
    state.lastTimestamp = timestampMs;
    state.sequence = 0;
}
```

The `<=` check catches both same-millisecond calls AND clock regression (including NTP step-back, VM migration, and the synthetic timestamp advance from sequence wraparound). `timestampMs = state.lastTimestamp` ensures the generator never produces a timestamp lower than the last one it used.

### Why this is non-obvious

The sequence wraparound fix (advance timestamp by 1ms) is the RFC 9562 §6.2 recommended approach and is correct in isolation. The clock regression vulnerability only appears on the NEXT call after wraparound — a state transition that's easy to miss in testing because it requires 4097+ same-millisecond UUIDs followed by another call within the same real millisecond. The `==` vs `<=` difference is a single character that determines whether monotonicity holds.
