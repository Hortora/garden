## Symptom

`@QuarkusTest` investigation tests fail with `caseInstance.tenancyId=null`. Ledger saves fail with a NOT NULL constraint violation on `tenancy_id`. No exception is thrown at the principal injection site — the bean resolves, returns null silently, and the null propagates into the entity before the constraint fires.

Tests that worked against the previous SNAPSHOT begin failing without any change in the AML project itself.

## Context

casehub-work June 2026 SNAPSHOT added `TenantScopedPrincipal @RequestScoped` implementing `CurrentPrincipal`. The AML project's test module provides `MockCurrentPrincipal @DefaultBean @ApplicationScoped` to supply a fixed test tenancyId.

`@DefaultBean` loses to ANY non-default bean for the same type. `TenantScopedPrincipal` is not `@DefaultBean`, so CDI selects it over `MockCurrentPrincipal` — the mock is silently displaced. On the test HTTP thread, the request context is active and `TenantScopedPrincipal` can be instantiated. On Vert.x event loop threads (where the case engine invokes workers and fires async observers), no CDI request context exists. The `@RequestScoped` proxy either returns null or throws `ContextNotActiveException` (caught internally), resulting in a null tenancyId propagating into `CaseInstance` and any ledger entries written during engine execution.

## Root Cause

`@DefaultBean` is not a "test wins" mechanism — it is a "lowest priority" marker. Any competing bean for the same type that is NOT annotated `@DefaultBean` displaces it, regardless of scope mismatch. A SNAPSHOT that introduces a `@RequestScoped` implementation of a SPI used in async/event-loop contexts silently breaks all consumers that relied on `@DefaultBean` displacement.

## Fix

Until casehub-work marks `TenantScopedPrincipal` as `@DefaultBean` (tracked as aml#59), workaround options:

**Option A — @Alternative @Priority(1) test bean** (if the project's @DefaultBean is in its own test classpath):
```java
@Alternative @Priority(1) @ApplicationScoped
public class TestCurrentPrincipal implements CurrentPrincipal {
    @Override public String tenancyId() { return TenancyConstants.DEFAULT_TENANT_ID; }
}
```
`@Alternative @Priority(1)` wins over ALL non-alternative beans regardless of scope.

**Option B — exclude TenantScopedPrincipal from CDI in tests:**
```properties
# src/test/resources/application.properties
quarkus.arc.exclude-types=io.casehub.work.runtime.principal.TenantScopedPrincipal
```
Only works if `TenantScopedPrincipal` is NOT in a Jandex-indexed JAR — see GE-20260601-848232.

**Option C — ARC bean container exclusion (if B fails):**
Add the FQN to `quarkus.arc.exclude-types` in the JAR's own `META-INF/beans.xml` (requires access to casehub-work source).

## Why this is non-obvious

`@DefaultBean` sounds like "the default wins when nothing else provides it." The actual semantics are the opposite: any non-`@DefaultBean` bean for the same type displaces it — including a `@RequestScoped` bean that is incompatible with the async execution context where the SPI is actually called. The null tenancyId appears in a completely different part of the stack from the principal resolution, making the connection invisible. The failure manifests as a NOT NULL constraint violation on the ledger, not as a CDI resolution error.
