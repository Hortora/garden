### Context

When two CDI `@Decorator` beans wrap the same SPI (e.g. `CaseRetriever`) and both need
cross-encoder scores — one for quality filtering (CRAG), the other for reordering
(reranking) — the naive approach runs the cross-encoder twice on overlapping candidates.
BERT forward passes are the most expensive inference operation in the pipeline.

### The technique

The inner decorator (CRAG, `@Priority(100)`) scores all chunks via `CrossEncoderReranker`
and attaches raw float scores to each chunk's metadata map under a shared key
(`_crossEncoderScore`). The outer decorator (reranker, `@Priority(75)`) checks
`hasPrecomputedScores()` — if ALL chunks carry the key, it sorts by pre-computed scores
without invoking the cross-encoder. If any chunk lacks the key (e.g. CRAG was disabled),
it falls back to full cross-encoder scoring.

### Why this is non-obvious

The stamp key that guards against bridge double-application (`_reranked`) MUST be distinct
from the score key (`_crossEncoderScore`). If `isAlreadyReranked()` checked for the score
key, CRAG's score propagation would trigger the reranker's guard — the reranker would see
pre-scored chunks and conclude it already ran, skipping reranking entirely. Two metadata
keys serve two purposes: one carries data (scores), the other carries execution state
(already processed).

This is the same pattern as `TrackingLogic.TRACKING_ID_KEY` / `CragEvaluationLogic.isAlreadyGraded()` — each decorator has its own guard key distinct from any data it reads or writes.

### Alternatives considered

- **Merge CRAG and reranking into one decorator:** eliminates the propagation problem but
  couples two independent concerns (quality filtering vs ordering) and prevents enabling
  one without the other.
- **Shared scoring service injected by both decorators:** adds a CDI bean with request-scoped
  caching. Over-engineered — metadata propagation through the existing chunk record is
  simpler and requires no new beans or scopes.

### Fix

Score propagation pattern (three components):

```java
// 1. Inner decorator attaches scores after evaluation
if (evaluator instanceof CrossEncoderRelevanceEvaluator ceEval) {
    var scored = ceEval.evaluateBatchWithScores(query, contents);
    float[] scores = extractScores(scored);
    chunks = RerankingLogic.attachScores(chunks, scores); // writes _crossEncoderScore
}

// 2. Outer decorator checks before re-scoring
if (RerankingLogic.hasPrecomputedScores(candidates)) {
    return sortByPrecomputedScores(candidates, maxResults); // no cross-encoder call
}

// 3. Guard key is DISTINCT from score key
static final String SCORE_KEY = "_crossEncoderScore";   // data propagation
static final String RERANKED_KEY = "_reranked";          // execution guard
```
