## Context

When a Mockito `thenAnswer` stub is registered and then a second stub for the same method is set up with `when(mock.method(any())).thenReturn(...)`, Mockito internally invokes the previously-registered `Answer` to confirm it can be replaced. This invocation passes a synthetic `null` invocation object.

## Symptom

```java
// @BeforeEach — registers a thenAnswer that extracts the argument
when(speechActClassifier.classify(any())).thenAnswer(invocation -> {
    SpeechActContext ctx = invocation.getArgument(0);  // NPE when invocation is null
    return new SpeechActResult(DONE, ctx.output(), FALLBACK);
});

// A test overrides the stub — Mockito re-fires the BeforeEach Answer with null
when(speechActClassifier.classify(any())).thenReturn(
    new SpeechActResult(STATUS, "Progress", PREFIX)
);
// → NullPointerException inside the thenAnswer lambda
```

The NPE's stack trace points into Mockito internals, not the test method — making it hard to trace to the lambda.

## Root Cause

Mockito's stubbing mechanism invokes registered Answers during stub replacement as part of its internal consistency checks. The invocation object passed during this call is `null` because there is no real method invocation occurring.

## Fix

Guard against null in the `thenAnswer` lambda:

```java
when(speechActClassifier.classify(any())).thenAnswer(invocation -> {
    if (invocation == null) return new SpeechActResult(DONE, "", FALLBACK);
    SpeechActContext ctx = invocation.getArgument(0);
    String content = ctx != null && ctx.output() != null ? ctx.output() : "";
    return new SpeechActResult(DONE, content, FALLBACK);
});
```

Both the `invocation` itself and the argument extracted from it can be null — guard both.

## Why this is non-obvious

The `thenAnswer` javadoc describes the Answer as "called when the stubbed method is invoked" — not "called when the stub is replaced". The stub replacement invocation is an implementation detail of Mockito's internal stubbing mechanism, not part of the documented contract. The NPE appears in Mockito internals, not in the test code that triggered it, making the root cause invisible without understanding Mockito's internals.

### Alternatives considered

- **Use `thenReturn()` with a fixed result** — avoids the issue but loses the ability to return different results based on the input argument. Not viable for passthrough mocks.
- **Use a named `Answer` class** — same issue; the class's `answer()` method is still called with a null invocation.
- **Register the thenAnswer in each test method, not @BeforeEach** — works, but requires boilerplate in every test. The null guard in @BeforeEach is cleaner.
