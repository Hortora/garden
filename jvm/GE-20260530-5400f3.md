## What it does

In Hibernate 6, you can pass a `List<String>` (or any `Collection`) as a named parameter in a native SQL query that uses an `IN (:param)` clause. Hibernate automatically expands the collection into the appropriate number of bind parameters.

```java
em.createNativeQuery("""
    SELECT * FROM memory_entry
    WHERE entity_id IN (:entityIds) AND domain = :domain
    ORDER BY created_at DESC
    """, MyEntity.class)
  .setParameter("entityIds", List.of("entity-1", "entity-2", "entity-3"))
  .setParameter("domain", "my-domain")
  .setMaxResults(20)
  .getResultList();
```

This produces a single SQL query with `IN ($1, $2, $3)` expanded correctly.

## Why this is non-obvious

The official Hibernate and JPA documentation for native queries describes named parameters for scalar values. Collection expansion is prominently documented for **JPQL** (`entityId IN (:entityIds)`) but the documentation for `createNativeQuery` does not mention that the same `setParameter(name, Collection)` behaviour applies to native SQL. Most developers assume they need to use JPQL for collection parameters, or must construct the SQL string dynamically (e.g. building `IN (?, ?, ?)` manually by iterating the collection).

## Context

Verified against PostgreSQL via Testcontainers in a Quarkus 3.32.x project. The native SQL used `entity_id IN (:entityIds)` with `setParameter("entityIds", query.entityIds())` where `entityIds` was a `List<String>`. All list sizes tested (1, 2, 10+) expanded correctly.

## Constraints

- Hibernate 6.x (Quarkus 3.x) confirmed. Behaviour in Hibernate 5.x may differ — do not assume backwards compatibility.
- The named parameter syntax (`:name`) is required — positional parameters (`?1`) do not support collection expansion in native queries.
- Very large collections (hundreds of IDs) will produce a SQL `IN` clause with many bind parameters, which can affect prepared statement reuse and query planning in PostgreSQL. Apply a reasonable upper bound on the collection size.
