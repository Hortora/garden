## Mutiny memoize failure eviction must be chained BEFORE memoize().indefinitely(), not after

**ID:** GE-20260609-78dc3a
**Stack:** Mutiny 2.x, Quarkus 3.x

### Context

Implementing a reactive per-key cache using `ConcurrentHashMap.computeIfAbsent()` with
`Uni.memoize().indefinitely()` to ensure an expensive async operation (e.g. creating all
Qhorus channels for a case, or lazy Qdrant collection creation in `ReactiveQdrantCorpusStore.ensureCollection()`)
runs exactly once per key per process lifetime, even under concurrent callers.

### The pattern (with correct failure eviction)

```java
private final ConcurrentHashMap<UUID, Uni<Map<String, CaseChannel>>> layoutCache =
        new ConcurrentHashMap<>();

public Uni<CaseChannel> openChannel(UUID caseId, String purpose) {
    return layoutCache.computeIfAbsent(caseId, id ->
                    initializeLayout(id)
                            .onFailure().invoke(err -> layoutCache.remove(id))  // ← MUST be here
                            .memoize().indefinitely())                           // ← AFTER eviction
            .map(channels -> channels.get(purpose));
}
```

### What happens without failure eviction (wrong order or missing)

If you write `memoize().indefinitely()` first and then try to add failure eviction, it's too
late — memoize has already captured the failure as the permanent emission:

```java
// WRONG — failure eviction after memoize has no effect on what memoize already cached
layoutCache.computeIfAbsent(caseId, id ->
        initializeLayout(id)
                .memoize().indefinitely()
                .onFailure().invoke(err -> layoutCache.remove(id)));  // too late
```

Mutiny's `memoize().indefinitely()` captures the **first emission** — whether success or
failure — and replays it to all future subscribers. **Source verification (Mutiny 3.1.1):**
`UniMemoizeOp.onFailure()` (line 131) sets `state = State.CACHING; cachedResult = failure` —
identical to the success path in `onItem()`. There is no special failure handling; failures
are cached permanently. A transient database failure on the first call permanently poisons
the cache entry. Every subsequent call to `openChannel()` for that `caseId` instantly fails
with the original error, with no way to retry.

With the correct ordering:

```java
initializeLayout(id)
    .onFailure().invoke(err -> layoutCache.remove(id))  // fires BEFORE memoize sees the failure
    .memoize().indefinitely()
```

The `onFailure().invoke()` side effect fires before `memoize()` commits the failure. The
`layoutCache.remove(id)` evicts the entry, allowing the next `computeIfAbsent()` call to
retry with a fresh `Uni`. Transient failures are transparent to callers.

### Why this is non-obvious

Reactive operator chains are often read as "left to right" in execution order, but `memoize()`
affects what the source `Uni` can emit to downstream subscribers. Adding `onFailure` after
`memoize` creates a new `Uni` that handles failures from `memoize`'s cached emission — but by
then, `memoize` has already locked in the failure as its permanent state. The eviction must
happen at the point where the failure is first observed by the chain, which is upstream of
`memoize()`.

The `computeIfAbsent` + `memoize()` pattern guarantees that `initializeLayout()` runs exactly
once per key under concurrent callers (`computeIfAbsent` returns the same `Uni` to all callers;
`memoize()` replays the same emission). The failure eviction is the only way to make this
pattern retry-safe without abandoning the memoize guarantee.

### Why this fix

The alternative — catching errors in `initializeLayout()` and returning a success with an
error value — conflates the error path with the data path and makes callers handle it
explicitly. The eviction-before-memoize pattern keeps the error path clean and makes retries
transparent: the next caller after a failure simply gets a fresh attempt, as if the first
call never happened.
