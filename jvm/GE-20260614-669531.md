## Testcontainers custom networks on Podman macOS return 'Invalid Upgrade header' for host HTTP calls

**ID:** GE-20260614-669531
**Stack:** Testcontainers 1.20.x, Podman 5.x, macOS arm64, applehv VM

### Symptom

HTTP calls from the macOS host to a container's published port return HTTP 400 with body
`Invalid Upgrade header`, even though the container is running and its service is healthy.
`getMappedPort()` returns a valid port. Connecting to that port from the host produces the
wrong response — as if a WebSocket proxy is sitting in the way.

```java
// Container is "started", getMappedPort returns 38695
curl http://localhost:38695/healthz
// → HTTP 400, body: "Invalid Upgrade header"
// Expected: HTTP 200, body: {"ok":true,"status":"live"}
```

### Context

Occurs when containers are placed on a user-defined Testcontainers network (`Network.newNetwork()`)
and accessed from the macOS host via their mapped ports. The same container on the default
bridge network (no custom `withNetwork()`) works correctly from the host.

### What was tried (didn't work)

- Verified container was running (`podman ps` shows Up)
- Checked port mapping — `getMappedPort()` returned the correct host port
- Tried fixed port bindings (`withCreateContainerCmdModifier` + `Ports.Binding.bindPort`)
- Added `withTty(true)` — no effect on port response
- Assumed it was a server-side error in the tested service

### Root cause

Podman macOS uses `netavark` for user-defined bridge networks (vs `pasta`/`slirp4netns`
for the default network). The `netavark` driver routes traffic through the applehv VM
differently, and the port-forwarding proxy for custom networks intercepts HTTP connections
before they reach the container, returning `Invalid Upgrade header` — the error a WebSocket
proxy emits when it receives a plain HTTP request without the required Upgrade header.

The container service is running and healthy; the proxy layer in front of it is broken.

### Fix

Use `execInContainer` for all HTTP calls in tests — containers reach each other via
network aliases, bypassing the broken host-to-container port path:

```java
// Instead of:
http.send(HttpRequest.newBuilder()
    .uri(URI.create("http://localhost:" + container.getMappedPort(8080) + "/healthz"))
    .build(), ...);

// Use:
Container.ExecResult result = container.execInContainer(
    "curl", "-sf", "http://localhost:8080/healthz");
assertThat(result.getExitCode()).isEqualTo(0);

// For cross-container calls, use the network alias:
Container.ExecResult result = openClaw.execInContainer(
    "curl", "-sf", "http://wiremock:8080/__admin/requests");
```

For wait strategies, use `Wait.forSuccessfulCommand` (exec-based) instead of
`Wait.forHttp` (host port-based):

```java
.waitingFor(Wait.forSuccessfulCommand("curl -sf http://localhost:18789/healthz")
    .withStartupTimeout(Duration.ofMinutes(3)))
```

Alternative: create the Podman network outside Testcontainers and reference it by
name — networks created via `podman network create` use standard routing that doesn't
have this proxy issue.

### Why this is non-obvious

The error `Invalid Upgrade header` is a WebSocket-specific message that points directly
at the container service (e.g. a Node.js WebSocket server). Nothing in the error
suggests a networking proxy layer. `getMappedPort()` returns a real, listening port.
The container appears healthy. The symptom looks exactly like a server-side bug in the
container application.

### Why this fix

`execInContainer` bypasses the host→container port-forwarding path entirely. The exec
API goes through the Docker socket (SSH tunnel), not through the VM's network stack.
Container-to-container traffic via network aliases uses the `netavark` bridge internally,
which works correctly; only host→container traffic through the applehv port-forwarding
layer is broken.
