### Symptom

REST endpoint returns HTTP 409 Conflict instead of the expected 500 Internal Server Error.
Happens when application code throws `IllegalStateException` for an infrastructure failure
(poll timeout, misconfiguration, unresolvable dependency).

### Context

Any application that depends on `casehub-work` and throws `IllegalStateException` from a
REST-invoked code path.

### Root cause

`casehub-work` JAR ships `io.casehub.work.runtime.api.IllegalStateExceptionMapper` — a
`jakarta.ws.rs.ext.ExceptionMapper<IllegalStateException>` that maps every
`IllegalStateException` to HTTP 409 Conflict. The mapper is registered globally via CDI
and affects any `IllegalStateException` thrown during a REST request in the application,
not just ones thrown from casehub-work code itself.

### Fix

Use a different exception type for infrastructure failures that should produce 500:

```java
// ❌ Produces HTTP 409 — mapped by casehub-work's IllegalStateExceptionMapper
throw new IllegalStateException("Agent did not reply for correlationId=" + id);

// ✅ Produces HTTP 500 — no mapper for generic RuntimeException
throw new RuntimeException("Agent did not reply for correlationId=" + id);
```

Alternatively, register a `@Priority(1)` exception mapper that overrides casehub-work's
for specific exception subtypes, or use a custom exception class that extends
`RuntimeException` directly.

### Why this is non-obvious

The mapper is shipped inside a transitive JAR (`casehub-work-0.2-SNAPSHOT.jar`) and is
nowhere documented. The symptom (409 instead of 500) gives no hint that an exception
mapper is involved — 409 Conflict is semantically about resource state, so developers
naturally investigate their domain logic rather than exception infrastructure. The 5-second
delay before the 409 (from a poll timeout) makes the connection even harder to spot.
