### Symptom

Temporal decay configured on a CBR query has no effect on the final retrieval ranking. Old cases rank identically to recent cases despite `TemporalDecay.HalfLife` being set. No error, no warning — the decay factor is computed and applied but silently thrown away.

### Context

CDI decorator chain for CBR retrieval:
```
TrendEnrichment@90 → Reranking@75 → OutcomeWeighting@65 → Tracking@50 → Store
```

Temporal decay was applied inside the store (InMemoryCbrCaseMemoryStore, JpaCbrCaseMemoryStore) as a post-scoring multiplier: `score *= decay.factor(storedAt, now)`.

### Root cause

The reranking decorator at @Priority(75) calls the cross-encoder to independently evaluate semantic relevance for each candidate. It produces a new sigmoid-normalized score that **completely replaces** the input score. Any multiplier applied to the input score (decay, confidence weighting, feature adjustments) is silently discarded — the cross-encoder score has no memory of the pre-applied factor.

Result flow: Store returns `score * decayFactor` → Tracking records it → OutcomeWeighting adjusts → **Reranking replaces with `sigmoid(crossEncoderScore)`** → decay gone.

### Fix

Score modifiers that should affect the final ranking must be applied at a HIGHER priority (outer in the decorator chain) than the score-replacing decorator. Higher CDI @Priority = executes first on the way in, last on the way out (processes results after the inner decorator).

```
TrendEnrichment@90 → Decay@80 → Reranking@75 → OutcomeWeighting@65 → Tracking@50 → Store
```

With decay at @Priority(80), it wraps the reranker and sees the final cross-encoder scores. The final score becomes `crossEncoderScore * decayFactor`.

Remove any store-level decay to avoid double application when the decorator is active.

### Why this is non-obvious

The CDI decorator chain reads left-to-right as "this wraps that", but the score flow is the reverse. A developer adding decay "after scoring" naturally puts it in the store or at a low priority — which is correct when no score-replacing decorator exists. The reranker's score replacement is not documented as "discards all pre-applied multipliers" — it's just "re-evaluates using cross-encoder", which sounds additive.

### Alternatives considered

- Apply decay inside the reranker itself (couples unrelated concerns)
- Add decay as a fusion leg in hybrid retrieval (only works for hybrid mode, complex)
- Apply decay at store level and skip when reranking is enabled (fragile, breaks when decorator config changes)
