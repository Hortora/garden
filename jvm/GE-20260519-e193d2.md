## Awaitility polling lambdas in @QuarkusTest have no JTA context — Panache reads throw ContextNotActiveException

**Stack:** Quarkus 3.32, Awaitility, Panache

**Symptom:** `jakarta.enterprise.context.ContextNotActiveException: Cannot use the EntityManager/Session because neither a transaction nor a CDI request context is active.` — thrown inside an Awaitility `untilAsserted()` lambda that calls `Entity.findById()`.

**Context:** Testing an `@ObservesAsync` CDI event chain: `receiveHumanMessage()` → event → observer → status update. The test calls `receiveHumanMessage()`, then uses `await().atMost(5s).untilAsserted(() -> { ... })` to poll for the expected state. The lambda tries to load a Panache entity — but the Awaitility polling thread has no active JTA transaction or CDI request scope.

### Root cause

Awaitility runs its polling lambda on a background thread (or the test thread, depending on config). Either way, there is no active JTA transaction or CDI request context on that thread. Panache requires one of these for any entity access. The test method itself cannot be `@Transactional` — if it were, the `@ObservesAsync` observer (which commits in its own REQUIRES_NEW transaction) would be invisible within the outer test transaction.

### Fix

Wrap each Panache access inside the `untilAsserted()` lambda in `QuarkusTransaction.requiringNew()`:

```java
await().atMost(Duration.ofSeconds(5)).untilAsserted(() ->
    QuarkusTransaction.requiringNew().call(() -> {
        ProtocolDeviation loaded = ProtocolDeviation.findById(deviationId);
        assertThat(loaded.piApprovalStatus).isEqualTo(PiApprovalStatus.APPROVED);
        return null;
    }));
```

`QuarkusTransaction.requiringNew()` programmatically opens a new transaction on the current thread for the duration of the lambda, satisfying the JPA context requirement. The transaction is committed (or rolled back on exception) when the lambda returns.

### Why this is non-obvious

The `@QuarkusTest` framework sets up CDI and JTA for the test class, but this context is attached to specific thread-points (test method start/end, `@Transactional` method boundaries). Awaitility's polling lambda is outside those boundaries. Developers who add Awaitility to an existing `@QuarkusTest` don't think about thread context — the test methods work fine, and only the async polling fails with a confusing error that looks like a JPA configuration problem.
