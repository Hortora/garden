## Symptom

Compile error when calling `wireMockServer.verify(N, get(urlEqualTo(...)))`:

```
error: incompatible types: MappingBuilder cannot be converted to RequestPatternBuilder
```

Or, if using the static `WireMock.verify()` variant, the test compiles but verifies against the wrong type at runtime.

## Context

Writing a `@QuarkusTest` with a `WireMockServer` on a fixed port. Trying to assert that the Vault key-info endpoint was called exactly once (cache-hit verification).

```java
// WRONG — get() is the stubbing API
wireMock.verify(1, get(urlEqualTo("/v1/transit/keys/reviewer-key")));
```

## Root cause

WireMock has two parallel static factory hierarchies:

| Purpose | Method | Returns |
|---------|--------|---------|
| **Stubbing** (setting up responses) | `get(urlPattern)`, `post(...)`, `put(...)` | `MappingBuilder` |
| **Verification** (asserting calls were made) | `getRequestedFor(urlPattern)`, `postRequestedFor(...)` | `RequestPatternBuilder` |

`stubFor(MappingBuilder)` and `verify(int, RequestPatternBuilder)` each accept different types. The methods look similar — `get(...)` vs `getRequestedFor(...)` — but passing one where the other is expected is a type error.

The confusion is compounded by WireMock's extensive use of static imports — at a glance, `get(...)` and `getRequestedFor(...)` look like overloads of the same function.

## Fix

Use `getRequestedFor` (and its siblings) for verification:

```java
// CORRECT
wireMock.verify(1, getRequestedFor(urlEqualTo("/v1/transit/keys/reviewer-key")));
wireMock.verify(0, anyRequestedFor(anyUrl()));
```

Quick reference:

| Stubbing | Verification |
|----------|-------------|
| `get(...)` | `getRequestedFor(...)` |
| `post(...)` | `postRequestedFor(...)` |
| `put(...)` | `putRequestedFor(...)` |
| `delete(...)` | `deleteRequestedFor(...)` |
| `any(...)` | `anyRequestedFor(...)` |

## Why this is non-obvious

Both families are typically imported as static imports alongside each other. They look like HTTP method helpers with the same signature. The naming convention (`getRequestedFor` vs `get`) isn't prominent in tutorials, which often show stubbing but not verification. The compile error message names `MappingBuilder` and `RequestPatternBuilder` — meaningful if you know the WireMock type hierarchy, opaque if you don't.
