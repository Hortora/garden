Qdrant ColBERT multi-vector size limit — hard cap at 1,048,576 floats per point silently prevents upsert

### Symptom

`QdrantClient.upsertAsync()` fails with:
```
INVALID_ARGUMENT: Validation error in body: points[0].vectors.vectors_options.vectors.[].data:
Total size of all vectors (1048576) must be less than 1048576
```

Collection creation succeeds. The error only appears at upsert time when a point's ColBERT multi-vector exceeds the limit. No warning at collection creation that a size limit exists.

### Context

BGE-M3 produces ColBERT multi-vectors: one vector per token, each of dimension 1024. At `max-sequence-length=1024`, a fully-tokenized entry produces exactly 1024 × 1024 = 1,048,576 floats — hitting the boundary. The check is strict less-than (`< 1048576`), so even one token at dimension 1024 exceeds it.

At `max-sequence-length=512` (the previous DJL-clamped default), ColBERT vectors are 512 × 1024 = 524,288 floats — well within the limit.

### What was tried (didn't work)

1. Setting `max-sequence-length=8192` — OOM on CPU (attention is quadratic, 256x memory vs 512 tokens)
2. Setting `max-sequence-length=4096` — no OOM but 0 points after 17 minutes (too slow on CPU)
3. Setting `max-sequence-length=1024` — Qdrant rejects ColBERT vectors at the boundary

### Fix

Cap `max-sequence-length` to keep ColBERT vectors under the limit:

```
max_tokens = floor(1,048,575 / colbert_dim)
# For 1024-dim ColBERT: max_tokens = 1023
```

In practice, use 768 for safety margin — covers 89% of a typical garden corpus (median 579 tokens, p95 960 tokens) while staying well under the limit (768 × 1024 = 786,432).

```properties
%dev.casehub.inference.models.bge-m3.max-sequence-length=768
```

### Why this is non-obvious

The error message shows the limit value but doesn't explain what drives it or how to configure it. The limit is not documented in Qdrant's high-level multi-vector docs. Collection creation with `MultiVectorConfig(MaxSim)` succeeds without any validation of the configured vector dimension against the per-point size limit — you only discover it at upsert time.

### Alternatives considered

- Increase Qdrant's limit — not configurable per-collection in v1.18.0
- Truncate ColBERT token vectors before storage (keep first N tokens) — viable but requires neocortex code change
- Reduce ColBERT dimension via quantization — #34 scope, not a quick fix
