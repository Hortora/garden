## Context

You have a blocking SPI (`CaseMemoryStore`) in a zero-dep API module and need to expose a
reactive mirror (`ReactiveCaseMemoryStore`) so callers using Mutiny can also use the same
backends. Most blocking adapters don't need true async execution — wrapping them as
`Uni.createFrom().item(supplier)` is sufficient. But occasional adapters (e.g. a graph
database with native async traversal) do benefit from a true reactive implementation.

## What developers reach for instead

- Two independent SPIs with no connection — adapters implement both, duplicating logic
- Make the reactive SPI extend the blocking SPI — creates tight coupling and forces adapters
  to implement both interface families
- A single abstract class with default delegation — doesn't play well with CDI proxies

## The Technique

Define the blocking SPI in the zero-dep API module. Define the reactive SPI separately
(in the Quarkus module, because `Uni<>` depends on Smallrye Mutiny). Ship a
`@DefaultBean @ApplicationScoped` bridge class that injects the active blocking impl and
wraps each method:

```java
@DefaultBean
@ApplicationScoped
public class BlockingToReactiveBridge implements ReactiveCaseMemoryStore {

    @Inject CaseMemoryStore delegate;   // whatever blocking impl is active

    @Override
    public Uni<String> store(MemoryInput input) {
        return Uni.createFrom().item(() -> delegate.store(input));
    }

    @Override
    public Uni<List<Memory>> query(MemoryQuery query) {
        return Uni.createFrom().item(() -> delegate.query(query));
    }

    @Override
    public Uni<Void> erase(EraseRequest request) {
        return Uni.createFrom().voidItem().invoke(() -> delegate.erase(request));
    }
}
```

The CDI priority ladder then becomes:

| Tier | Class | Annotation | Active when |
|------|-------|-----------|-------------|
| 0 | `NoOpCaseMemoryStore` | `@DefaultBean @ApplicationScoped` | No adapter |
| — | `BlockingToReactiveBridge` | `@DefaultBean @ApplicationScoped` | Always (wraps Tier 0–2) |
| 1 | Memori adapter | `@ApplicationScoped` | Memori dep declared |
| 2 | Mem0 adapter | `@Alternative @Priority(1)` | Mem0 dep declared |
| 3 | Graphiti native-reactive | `@Alternative @Priority(2)` on `ReactiveCaseMemoryStore` | Graphiti dep declared |

When Graphiti (Tier 3) is added, it overrides the `ReactiveCaseMemoryStore` binding directly
via its higher-priority `@Alternative` — the bridge's `@DefaultBean` yields automatically.
Tier 3 does NOT need to implement the blocking SPI at all.

## Why This Is Non-Obvious

The `@DefaultBean` on the bridge is the key: it means both the bridge AND any blocking
adapter can coexist on the classpath without CDI ambiguity. The bridge always yields to a
native-reactive `@Alternative` on `ReactiveCaseMemoryStore`, and the blocking adapter
displaces the no-op on `CaseMemoryStore`. Each priority rung resolves independently.

Without `@DefaultBean`, a plain `@ApplicationScoped` bridge would collide with any real
`ReactiveCaseMemoryStore` implementation (ARC sees two beans for the same type and fails at
startup).

**Note:** Do NOT add `@Blocking` to bridge methods. `ExecutionModelAnnotationsProcessor`
rejects it on plain CDI bean methods returning `Uni` — only framework entrypoints accept it.
Thread dispatch is the entrypoint's responsibility. (See GE-20260528-74914d.)
