### Symptom
RBAC boundary tests fail during `@BeforeEach` setup â€” RestAssured calls to create test data (trials, sites, patients) return 403 because the method-level `@TestSecurity` annotation (with a restricted role) is already active.

### Context
When testing access control boundaries, each test method needs a different `@TestSecurity` annotation (e.g. MONITOR-only, COORDINATOR-only). Test data must be created in `@BeforeEach` with full access. The natural approach is class-level `@TestSecurity` with full roles for setup, overridden at method level for boundary assertions.

### What was tried (didn't work)
```java
@QuarkusTest
@TestSecurity(user = "setup", roles = {SPONSOR, INVESTIGATOR, COORDINATOR})
class RbacBoundaryTest {
    @BeforeEach
    void setup() {
        // RestAssured POST /trials â€” expects 201 from class-level @TestSecurity
        // Actually gets 403 because method-level @TestSecurity is already active
    }

    @Test
    @TestSecurity(user = "monitor", roles = {MONITOR})
    void monitor_cannot_post() { ... }
}
```

### Root cause
Quarkus's `@TestSecurity` is a CDI interceptor applied at the method level. When a method-level `@TestSecurity` is present, it overrides the class-level annotation for the **entire test lifecycle** â€” including `@BeforeEach`, not just the `@Test` method body. The `@BeforeEach` runs under the restricted role, causing setup calls to fail with 403.

### Fix
Use direct `@Transactional` Panache entity creation in `@BeforeEach` instead of RestAssured HTTP calls. This bypasses the REST security layer entirely during setup. Each test method then uses RestAssured with its restricted `@TestSecurity` for the actual boundary assertion.

```java
@QuarkusTest
class RbacBoundaryTest {
    @Inject FixedCurrentPrincipal principal;
    private UUID trialId;

    @BeforeEach
    @Transactional
    void setup() {
        principal.reset();
        ClinicalTrial trial = new ClinicalTrial();
        trial.id = UUID.randomUUID();
        trial.tenantId = principal.tenancyId();
        trial.status = TrialStatus.PLANNING;
        // ... set fields directly
        trial.persist();
        trialId = trial.id;
    }

    @Test
    @TestSecurity(user = "monitor", roles = {MONITOR})
    void monitor_cannot_post() {
        given().contentType("application/json")
            .body("{...}")
            .when().post("/trials")
            .then().statusCode(403);
    }
}
```

### Why this is non-obvious
The JUnit lifecycle (`@BeforeEach` â†’ `@Test` â†’ `@AfterEach`) suggests that method-level annotations apply only to the `@Test` method. Quarkus's CDI interceptor model applies the annotation to the entire test lifecycle scope, which is not documented in the `@TestSecurity` javadoc. The class-level/method-level override pattern works for most annotations but fails here due to the lifecycle scope expansion.
