## Symptom

Flyway fails with `Found more than one migration with version N` after moving
a SQL migration file to a new location within the same Maven session. The source
tree has the file in exactly one place.

## Context

Reorganising Flyway migration files during active development — moving
`src/test/resources/db/migration/V1999.sql` first to `db/ledger/migration/`
then to `db/ledger/test-migration/` in the same session. Each move was to
`src/` but Maven had already compiled the old paths into `target/test-classes/`.

## Root cause

Maven's incremental build copies new files into `target/test-classes` but does
**not** delete old copies when source files are moved or renamed — only when
`mvn clean` runs. After two moves, `target/test-classes` contains copies under
all three paths simultaneously. Flyway scans `target/test-classes` (not `src/`)
at runtime and finds the same version number in multiple locations.

```
target/test-classes/db/migration/V1999__test_entry.sql      ← stale from move 1
target/test-classes/db/ledger/migration/V1999__test_entry.sql ← stale from move 2
target/test-classes/db/ledger/test-migration/V1999__test_entry.sql ← current
```

## Fix

Either:
1. `mvn clean` before running tests — clears all stale copies
2. Manually delete the stale paths from `target/test-classes/`

```bash
rm target/test-classes/db/migration/V1999__test_entry.sql
rm target/test-classes/db/ledger/migration/V1999__test_entry.sql
```

## Why this is non-obvious

The source tree looks clean — `git status` and `find src/` show the file in
exactly one location. The duplication exists only in `target/` which developers
rarely inspect. The error message names the version number but not the file
paths involved, making it hard to connect to a stale build cache.
