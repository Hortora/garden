## Symptom

`isConcurrentInsert(PersistenceException ex)` (or any similar method walking the cause
chain for `instanceof SQLIntegrityConstraintViolationException`) always returns `false`
in production against PostgreSQL. All concurrent insert race detection falls through to
the error-log + discard branch. Tests pass with H2.

## Context

A service method annotated `@Transactional(REQUIRES_NEW)` inserts a row protected by a
unique DB constraint. The caller catches `PersistenceException` and checks its cause chain
for a constraint violation to distinguish an expected race (loser path) from a genuine
DB error. H2 is used in tests; PostgreSQL in production.

## Root Cause

The JDBC exception hierarchy differs by vendor:

**H2:**
```
PersistenceException
  → org.hibernate.exception.ConstraintViolationException
    → java.sql.SQLIntegrityConstraintViolationException   ← instanceof check hits ✅
```

**PostgreSQL:**
```
PersistenceException
  → org.hibernate.exception.ConstraintViolationException
    → org.postgresql.util.PSQLException                   ← extends java.sql.SQLException directly
                                                             instanceof SQLIntegrity... is false ❌
```

`SQLIntegrityConstraintViolationException` is a JDBC 4.0 subtype of `SQLException`.
H2 uses it; the PostgreSQL JDBC driver does not. `PSQLException` extends `SQLException`
directly and never satisfies `instanceof SQLIntegrityConstraintViolationException`.

## Fix

Walk the cause chain checking for `java.sql.SQLException` (both H2 and PostgreSQL extend
it) and discriminate by constraint name in the message:

```java
static boolean isConcurrentInsert(PersistenceException ex) {
    Throwable cause = ex;
    while (cause != null) {
        if (cause instanceof SQLIntegrityConstraintViolationException c) {
            // H2 path
            String msg = c.getMessage() != null ? c.getMessage().toLowerCase() : "";
            return msg.contains("uq_binding_key") || msg.contains("unique");
        }
        if (cause instanceof java.sql.SQLException s
                && !(cause instanceof SQLIntegrityConstraintViolationException)) {
            // PostgreSQL path: PSQLException extends SQLException directly
            String msg = s.getMessage() != null ? s.getMessage() : "";
            if (msg.contains("uq_binding_key")) return true;  // use your constraint name
        }
        cause = cause.getCause();
    }
    return false;
}
```

Using the specific constraint name (`uq_binding_key`) avoids false positives on unrelated
constraint violations.

## Why This Is Non-Obvious

`SQLIntegrityConstraintViolationException` is defined in JDBC 4.0 and sounds like the
canonical type for unique constraint violations. H2 uses it consistently, so tests pass.
The JPA spec only guarantees that constraint violations surface as `PersistenceException`
— it says nothing about the cause chain. PostgreSQL's JDBC driver simply never uses
`SQLIntegrityConstraintViolationException`. A check that is entirely correct in test
fails silently in production: no exception, no log, message discarded — the wrong branch
is taken with no indication anything went wrong.
