### Symptom
`FlywayException: Found more than one migration with version 1` at startup when two datasources are both configured with Flyway — one scanning `classpath:db/migration` (default datasource) and another scanning `classpath:db/migration/<module>/` (named datasource). Flyway's recursive scan on the parent path finds ALL files in `db/migration/` including those in subdirectories, producing duplicate V-number errors.

### Context
A multi-datasource Quarkus app where:
1. The default datasource uses Flyway and scans `classpath:db/migration` (the default location)
2. A named datasource (e.g. `qhorus`) uses Flyway and scans `classpath:db/migration/qhorus/`
3. Both sets of migration files share some V-numbers (e.g. both have V1)

The intent was to isolate the named datasource's migrations in a subdirectory. This works for *which datasource applies which migration*, but not for *which datasource discovers which files*.

### Root cause
Flyway scans classpath locations recursively by default. Configuring `quarkus.flyway.<datasource>.locations=classpath:db/migration/qhorus` scopes which migrations that datasource *runs*, but it does not prevent the default datasource (configured for `classpath:db/migration`) from *discovering* files in `db/migration/qhorus/` during its own recursive scan. Both datasources see the qhorus migrations; the default datasource then finds V1 in both `db/migration/` (its own) and `db/migration/qhorus/` (qhorus's), and Flyway rejects the duplicate.

### Fix
Move the module's migrations completely outside the `db/migration/` namespace:

```
db/qhorus/migration/V1__initial_schema.sql   ← was db/migration/qhorus/V1__initial_schema.sql
```

Configure accordingly:
```properties
quarkus.flyway.qhorus.locations=classpath:db/qhorus/migration
```

Any path that does not start with `db/migration/` avoids the recursive scan overlap.

### Why this is non-obvious
The "scoped subdirectory" pattern (`db/migration/<module>/`) appears to solve the isolation problem and is even described in Flyway documentation and some Quarkus guides as the recommended approach for named datasources. The failure only manifests when *another* Flyway datasource is configured with the parent path — which is the default Quarkus/Flyway setup. The error message ("Found more than one migration with version 1") gives no indication that two datasource scanners are involved; it looks like a duplicate file problem within a single scan.
