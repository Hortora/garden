### Symptom
Integration tests time out with `ConditionTimeoutException` after converting stub workers to AgentExec. The case completes (logs show COMPLETED) but Awaitility conditions never fire — the test waits for a human task or escalation that is never created.

### Context
When converting 32 stub workers from inline lambdas to AgentExec workers backed by a test ChatModel factory, the factory serves canned JSON responses. The response field names match the old stub output, but the VALUES were chosen arbitrarily rather than matching the old stub's computed values.

Example — old stub computed dynamic values:
```java
int totalCost = 3500;
return WorkerResult.of(Map.of(
    "totalCost", totalCost,
    "requiresApproval", totalCost > 2000,  // true
    "isHighValue", totalCost > 5000        // false
));
```

Test factory used arbitrary values:
```json
{"totalCost": 1500, "requiresApproval": false, "isHighValue": false}
```

The YAML binding condition checks `requiresApproval == true` to show a human task. With `false`, the human task is skipped, the case completes without the approval step, and `completeHumanTask()` times out.

### What was tried (didn't work)
1. Assuming field name correctness was sufficient — wrong; binding conditions depend on VALUE semantics
2. Using "reasonable defaults" (low cost, no anomalies) — wrong; tests exercise the INTERESTING paths (escalation, approval gates)

### Root cause
Stub lambdas encode behavioral intent in their return values — not just field schemas. A stub returning `requiresApproval: true` was deliberately triggering the approval binding. A test factory returning `requiresApproval: false` silently skips that binding.

### Fix
When building a test factory, read each old stub's return map and preserve the COMPUTED values, not the field names:
```java
// Old stub: totalCost = 3500, requiresApproval = totalCost > 2000 → true
Map.entry("assess the total travel budget",
    "{\"totalCost\":3500,\"requiresApproval\":true,\"isHighValue\":false}")
```

### Why this is non-obvious
The conversion appears correct: same fields, valid JSON, factory matches on system prompt, integration tests compile. The failure is silent — the case completes successfully, just via a different path. The only symptom is an Awaitility timeout, with no log indicating why the expected path wasn't taken.
