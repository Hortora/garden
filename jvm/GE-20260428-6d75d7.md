## What

Custom `count` methods in Quarkus Panache store interfaces (e.g. `MessageStore.countByChannel()`)
may return `int` rather than `long`. The Panache static method `Entity.count(...)` returns `long`,
but custom interface implementations may have been written with `int` and silently narrow the value.

```java
// MessageStore interface — returns int, not long
int countByChannel(UUID channelId);
```

## Discovery

Found by reading the `MessageStore` interface source when a compilation warning appeared
assigning the result to a `long` variable. The narrowing conversion from `int` to `long`
is implicit and silent — no error, no warning at call sites that store the result in `long`.

## Impact

In practice: no data loss for channel message counts at current scale (int max = ~2.1 billion).
But incorrect for channels with very high message throughput, and surprising when code
like `long deleted = channelService.delete(name, force)` compiles cleanly despite the
underlying `countByChannel` returning `int`.

## What to do

When adding a method to a `*Store` interface that counts rows, check the existing return
type. If it's `int`, either:
- Use it as `int` and cast explicitly when assigning to `long`
- Change the interface to `long` (requires updating all implementations including InMemory stores)

Panache's built-in `Entity.count()` always returns `long` — prefer it over custom count
methods where the store abstraction allows direct entity access.
