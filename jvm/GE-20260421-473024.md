## Symptom

After rebasing a feature branch onto upstream (which changed an interface signature in `api/`), CI fails with a compilation error in a downstream module. Local tests pass.

## Context

A multi-module Maven project. `api/` defines `LoopControl.select()`. The rebase changed the return type from `List<Binding>` to `Uni<List<Binding>>`. Local validation ran only `-pl api,engine` — not the full build. `casehub-blackboard/` implements `LoopControl` but was not in the local test scope. CI runs the full build and caught the compilation failure.

## Root Cause

When validating a rebase that changes an interface, only running the modules directly tested misses all downstream implementors. Maven's `-pl` flag limits the build scope — it will not warn you that other modules in the reactor now fail to compile.

## Fix

After any rebase that changes a public interface in a shared module (`api/`), run the full reactor build before pushing:

```bash
mvn compile -q   # or: mvn install -DskipTests -q
```

This catches all downstream compilation failures across every module in the project before CI does.

## Why This Is Non-Obvious

Local test commands that scope to specific modules (`-pl api,engine`) are fast and sufficient for most changes. The failure only surfaces when an interface change propagates to a module not in the local test scope. The symptom (CI red, local green) is disorienting until the pattern is understood.
