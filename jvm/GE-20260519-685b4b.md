## Symptom

After adding a `UNIQUE` constraint to a table (e.g. `work_item_template.name`), previously passing `@QuarkusTest` tests start failing with `JdbcSQLIntegrityConstraintViolationException: Unique index or primary key violation`. Tests that always passed now fail — especially tests in different classes that coincidentally use the same hardcoded entity name.

## Context

Multiple `@QuarkusTest` classes all share a single Quarkus instance and a single in-memory H2 database. Flyway runs once at startup. Each test class runs its own `@BeforeEach` lifecycle methods, but the H2 database state persists across test classes for the entire JVM run.

Before the UNIQUE constraint: duplicate names in the DB were silently accepted. Tests worked because each test found its own entity by ID or used the first match. After the UNIQUE constraint: when test class B tries to create "Approval Template" and test class A already created the same name (and didn't clean up), the second `INSERT` fails.

## What was tried (didn't work)

- Adding a `@BeforeEach` that clears only the entities created in the current test — requires tracking, error-prone.
- Unique names per test class prefix — changes test intent and is fragile.

## Root cause

All `@QuarkusTest` classes share one Quarkus instance (and therefore one H2 DB schema). There is no automatic per-class DB reset. `@TestTransaction` would reset state but breaks REST Assured HTTP assertions (committed data isn't visible across transaction boundaries — see separate gotcha).

## Fix

Add `@BeforeEach @Transactional void clearTemplates() { WorkItemTemplate.deleteAll(); }` to every test class that creates entities with UNIQUE-constrained fields. For tables with `ON DELETE CASCADE` referencing the constrained table, the cascade handles dependent rows automatically.

For helpers called multiple times within a single test method (e.g. `createTemplate()` called twice in `processSchedules_partialFailure`), make the name unique per call:

```java
// Instead of a hardcoded name:
private String createTemplate() {
    final String name = "tmpl-" + UUID.randomUUID().toString().substring(0, 8);
    return given().contentType(ContentType.JSON)
            .body("{\"name\":\"" + name + "\",\"category\":\"test\",\"createdBy\":\"admin\"}")
            .post("/workitem-templates").then().statusCode(201).extract().path("id");
}
```

```java
// @BeforeEach for the class:
@BeforeEach
@Transactional
void clearTemplates() {
    WorkItemTemplate.deleteAll(); // cascades to work_item_schedule via ON DELETE CASCADE
}
```

## Why this is non-obvious

The test suite was green before the constraint because duplicate names were never a problem at runtime (application-layer guards were sufficient). Adding the DB-level constraint is correct — but it retroactively reveals that 8+ test classes have been silently sharing name collisions across a session. The error appears as an integrity violation in class B but the root cause is state left by class A, making the failure look unrelated to the change.
