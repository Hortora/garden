### Symptom
Cursor pagination using `(created_at DESC, id DESC)` skips or duplicates records when multiple entries are created within the same millisecond. The cursor encodes the last-seen `(timestamp, id)` pair, but same-millisecond records have identical timestamps — ordering falls to the `id` column, where random UUID v7 bits produce non-deterministic order across queries.

### Context
UUID v7 (RFC 9562 §5.7) embeds a 48-bit Unix timestamp in milliseconds, but the remaining 74 bits (after version/variant) are random. Two UUIDs generated in the same millisecond share the same timestamp prefix but have random suffixes. When used as a keyset pagination cursor alongside a timestamp column, this means same-millisecond records appear in arbitrary order — and that order can differ between the query that produced the cursor and the query that uses it.

### What was tried (didn't work)
Standard UUID v7 generation: `Instant.now()` for the timestamp + `ThreadLocalRandom` for the remaining bits. Contract tests creating 5 notifications in a tight loop produced same-millisecond records with random ordering — `find_paginatesWithCursor` test failed intermittently.

### Root cause
The random bits in UUID v7 are designed for uniqueness, not ordering. RFC 9562 §5.7 explicitly permits but does not mandate monotonic sequences within the same timestamp tick.

### Fix
Add a thread-local 12-bit sequence counter that increments within the same millisecond and resets on millisecond change. This fills the 12 "rand_a" bits of the UUID v7 layout with a monotonic counter instead of random values.

```java
public final class UUIDv7 {
    private static final ThreadLocal<long[]> STATE = ThreadLocal.withInitial(() -> new long[]{-1, 0});

    public static String generate() {
        return generate(Instant.now());
    }

    static String generate(Instant now) {
        long ms = now.toEpochMilli();
        long[] state = STATE.get();
        long seq;
        if (ms == state[0]) {
            seq = (state[1] + 1) & 0xFFF; // 12-bit wrap
        } else {
            state[0] = ms;
            seq = 0;
        }
        state[1] = seq;

        long msb = (ms << 16) | 0x7000L | seq;  // version 7 + sequence
        long lsb = (0x2L << 62) | (ThreadLocalRandom.current().nextLong() & 0x3FFF_FFFF_FFFF_FFFFL);
        return new UUID(msb, lsb).toString();
    }
}
```

The sequence counter ensures that UUIDs generated in the same millisecond on the same thread sort in insertion order. Cross-thread ordering within the same millisecond is not guaranteed, but is not needed — cursor pagination is per-user, and notification writes are typically serialised per user.

### Why this is non-obvious
UUID v7 is advertised as "time-ordered" — the natural assumption is that time-ordering is sufficient for pagination. It is, across millisecond boundaries. The failure only appears when multiple records are created within the same millisecond, which is common in batch operations (`storeAll`) and tight loops but rare in interactive use. The intermittent nature makes it hard to reproduce.

### Alternatives considered
- `Thread.sleep(1)` between inserts — guarantees different timestamps but adds latency to batch operations
- Use `created_at` as the sole sort key — ties are unresolvable without a secondary key
- Use a database sequence instead of UUID — loses the benefits of application-generated IDs (no DB round-trip)
