### Symptom
Size-limited HTTP download silently downloads the entire response into heap before any size check runs. A 500MB response becomes a 500MB `byte[]` allocation with no opportunity to abort mid-stream.

### Context
When using `java.net.http.HttpClient` to download files with a configurable size limit (e.g. Discord CDN attachments with an 8MB cap), the natural approach is `BodyHandlers.ofByteArray()` followed by a length check on the result.

### What was tried (didn't work)
```java
HttpResponse<byte[]> resp = client.send(req, BodyHandlers.ofByteArray());
if (resp.body().length > maxBytes) return null; // too late — already in heap
```
The entire response is materialized into a `byte[]` before `send()` returns. For chunked transfer encoding (no `Content-Length` header), there is no pre-check possible — the body is fully downloaded.

### Root cause
`BodyHandlers.ofByteArray()` accumulates the entire response body into memory before completing the `CompletableFuture`. This is by design — the handler type guarantees a complete `byte[]`. There is no callback or interrupt point during accumulation.

### Fix
Use `BodyHandlers.ofInputStream()` and read in chunks with a running byte count:
```java
HttpResponse<InputStream> resp = client.send(req, BodyHandlers.ofInputStream());

// Content-Length pre-check (optimization — may be absent for chunked)
long cl = resp.headers().firstValueAsLong("Content-Length").orElse(-1);
if (cl > maxBytes) return null;

// Streaming enforcement (primary limit — works for chunked)
try (InputStream in = resp.body()) {
    ByteArrayOutputStream out = new ByteArrayOutputStream();
    byte[] buf = new byte[8192];
    long total = 0;
    int read;
    while ((read = in.read(buf)) != -1) {
        total += read;
        if (total > maxBytes) return null; // abort mid-stream
        out.write(buf, 0, read);
    }
    return out.toByteArray();
}
```

### Why this is non-obvious
`ofByteArray()` is the default in most tutorials and examples. The memory behavior is not documented as a warning — it's an implicit consequence of the handler's contract. Developers expect `send()` to be the network call, not realizing it also completes the full body accumulation.

### Why this fix
`ofInputStream()` is the only built-in handler that gives the caller control over how much data is read. `ofByteArrayConsumer()` exists but still accumulates internally. The streaming approach adds ~5 lines but prevents unbounded heap allocation.

### Alternatives considered
- **Content-Length header check only**: Fails for chunked transfer encoding where the header is absent — the server streams without declaring total size.
- **Custom `BodySubscriber`**: More control but significantly more code for the same result. `ofInputStream()` is sufficient.
