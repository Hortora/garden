# @ApplicationScoped POJO + @Produces for same type causes CDI ambiguous bean resolution

## Symptom

`AmbiguousResolutionException` or `UnsatisfiedDependencyException` at Quarkus build time when trying to inject a type that has both a class-level `@ApplicationScoped` annotation and a separate `@Produces @ApplicationScoped` method.

## Context

A class is annotated with `@ApplicationScoped` and has an `@Inject` constructor with parameters (e.g., plain String, int, or other primitives). The same class is also produced by a separate `@Produces @ApplicationScoped` method in a different class. The goal was to construct the class using the producer method with runtime values.

## What was tried

- Added `@Default` and `@Alternative` qualifiers â€” didn't resolve the ambiguity
- Tried removing `@Inject` from constructor but kept `@ApplicationScoped` â€” still two beans
- Checked if the producer method was being scanned â€” it was, and producing correctly

## Root cause

CDI sees **both** the class-level bean (via `@ApplicationScoped` component scanning) and the producer method bean as independent CDI beans of the same type. 

If the class constructor has parameters CDI can't satisfy (plain Strings, ints without `@ConfigProperty`), the class-level bean fails at wiring time. Even if CDI could satisfy the constructor parameters, the presence of two beans for the same type causes `AmbiguousResolutionException` unless one is explicitly qualified.

The producer method is meant to be the **only** source of the bean, but the class-level annotations create a competing bean definition.

## Fix

Remove `@ApplicationScoped` and `@Inject` from classes that are constructed by `@Produces` methods. They should be plain POJOs â€” the producer owns their lifecycle and CDI registration.

```java
// Wrong â€” creates two beans
@ApplicationScoped
public class QdrantVectorStore {
    @Inject
    public QdrantVectorStore(String collectionName, QdrantClient client) {
        // ...
    }
}

// Elsewhere
@Produces
@ApplicationScoped
public QdrantVectorStore vectorStore() {
    return new QdrantVectorStore("my-collection", client);
}

// Correct â€” single bean via producer
public class QdrantVectorStore {
    // No CDI annotations on class or constructor
    public QdrantVectorStore(String collectionName, QdrantClient client) {
        // ...
    }
}

// Elsewhere
@Produces
@ApplicationScoped
public QdrantVectorStore vectorStore() {
    return new QdrantVectorStore("my-collection", client);
}
```

## Why non-obvious

The pattern feels natural: annotate the class to make it a CDI bean, AND provide a producer for custom construction. CDI's rule is that both declarations create independent beans â€” it doesn't recognize the producer as "replacing" the class-level bean.

Tests that construct the class directly (without a CDI container) won't catch this because they bypass bean resolution entirely. The error only appears when Quarkus builds the CDI graph.
