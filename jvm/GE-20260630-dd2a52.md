### Behaviour
```java
KeyPairGenerator gen = KeyPairGenerator.getInstance("Ed25519");
KeyPair kp = gen.generateKeyPair();
System.out.println(kp.getPublic().getAlgorithm()); // "EdDSA" on Java 26, "Ed25519" on Java 21
```

`KeyFactory.getInstance("Ed25519")` still works — the factory accepts both names. `Signature.getInstance("Ed25519")` also still works. Only `Key.getAlgorithm()` returns the different string.

### Impact
Test assertions that check `key.getAlgorithm().equals("Ed25519")` fail on Java 26. Code that uses `Signature.getInstance(key.getAlgorithm())` works correctly — `Signature.getInstance("EdDSA")` is valid.

The `SignatureAlgorithms.signatureAlgorithm(Key)` utility in casehub-ledger handles this transparently: `"EdDSA".equals(key.getAlgorithm())` is not `"EC"`, so it passes through unchanged. `Signature.getInstance("EdDSA")` resolves correctly.

### Why this is non-obvious
The JCA documentation for Ed25519 uses `"Ed25519"` as the algorithm name throughout. The change to `"EdDSA"` in `Key.getAlgorithm()` is not documented in the Java 26 EA release notes (as of 2026-06-30). The factory methods (`KeyFactory`, `KeyPairGenerator`, `Signature`) still accept `"Ed25519"`, making the inconsistency harder to notice — it only surfaces in `getAlgorithm()` on the generated key objects.
