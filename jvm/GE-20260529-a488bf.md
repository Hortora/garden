## Symptom

`isMimeType("text/plain")` returns `true` on a freshly-constructed `MimeMessage` regardless of what content was set — even after calling `setContent(MimeMultipart)` or `setContent(Object, "application/octet-stream")`.

In a `ContentExtractor` that prefers `text/plain` over `text/html`, this means:
- A multipart message returns `MimeMultipart@1a38ba58` (the object's `toString()`) instead of the extracted plain-text body
- A binary-only message returns the String literal used in `setContent()` instead of `""`

The bug appears identical to a logic error in the extraction algorithm, not a library setup problem.

## Context

When you construct a `MimeMessage` and call `setContent(Multipart)` or `setContent(Object, String)`, the Content-Type header is **not committed to the message headers** until `saveChanges()` is called. Before that point:

- `msg.getContentType()` returns `"text/plain"` (the default for a new empty `MimeMessage`)
- `msg.isMimeType("text/plain")` returns `true` for all messages, regardless of the actual content type set
- `msg.getContent()` returns whatever object was passed to `setContent()` — so `.toString()` on a `MimeMultipart` gives you the object reference

Messages fetched from an IMAP store have already gone through their transport lifecycle — headers are committed on delivery — so this issue never occurs in production. It is a test-construction-only problem.

## What Was Tried

The `ContentExtractor` algorithm was checked and found correct. The MIME type check paths were traced manually. The algorithm clearly returns `null` for non-plain-text types before falling through to the `""` fallback — yet `"bytes"` and `"MimeMultipart@..."` values appeared.

## Root Cause

`MimeMessage.updateHeaders()` (called by `saveChanges()`) propagates the content type from the `DataHandler` to the actual `Content-Type` MIME header. Without this call:

```java
msg.setContent(myMultipart); // DataHandler is set, but Content-Type header is NOT updated
msg.isMimeType("multipart/*"); // returns false — header still says "text/plain"
msg.getContentType();          // returns "text/plain"
msg.getContent();              // returns the MimeMultipart object
msg.getContent().toString();   // returns "MimeMultipart@1a38ba58"
```

## Fix

Call `msg.saveChanges()` immediately after setting content in tests:

```java
final MimeMessage msg = new MimeMessage(Session.getInstance(new Properties()));
final MimeMultipart mp = new MimeMultipart("alternative");
mp.addBodyPart(plainPart);
mp.addBodyPart(htmlPart);
msg.setContent(mp);
msg.saveChanges(); // ← required: commits Content-Type header from DataHandler to headers
// now isMimeType("multipart/alternative") returns true
```

`saveChanges()` does not require a valid transport session (no From/To/Date headers needed) — it only updates MIME content headers. It is safe to call on a partially-constructed message in a test context.

## Why This Is Non-Obvious

The `saveChanges()` contract in the Jakarta Mail specification describes it as a pre-send step: "Clients composing a message should call `Message.saveChanges()` before obtaining the input stream for the message." Nothing in the spec or docs warns that `isMimeType()` is unreliable before `saveChanges()`. The method name implies a read operation with no dependencies.

### Why this fix

Production code that calls `isMimeType()` on messages received from IMAP stores works correctly without `saveChanges()` because IMAP delivery commits all headers. The fix belongs in tests, not in the extraction logic itself — which would be incorrect production code if modified to work around uncommitted test messages.
