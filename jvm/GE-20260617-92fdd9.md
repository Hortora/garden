# Compound type ID with embedded fallback for extensible polymorphic hierarchies

### The problem

An abstract class hierarchy with plugin-extensible subtypes needs polymorphic serialization. Plugin modules add new subtypes at runtime (e.g., vendor-specific device types). When the deserializer encounters an unknown subtype (plugin not on classpath), it must degrade gracefully to the common parent type — not throw.

Standard approaches fail:
- `@JsonSubTypes` with the parent's classification enum can't distinguish the parent from its supplement (both share the same enum value)
- `DeserializationProblemHandler.handleUnknownTypeId()` has no access to sibling JSON fields to determine the fallback (see GE-20260617-e3bedc)

### The technique

Encode both the fallback key and the specific type name in a single compound type ID:

```
@deviceType: "THERMOSTAT:ThermostatDevice"          // common type
@deviceType: "THERMOSTAT:HomeAssistantThermostat"    // plugin supplement
@deviceType: "COVER:OpenHabRollershutter"            // different parent
```

The custom `TypeIdResolver`:

```java
public JavaType typeFromId(DatabindContext context, String id) {
    // 1. Exact match — plugin module registered this type
    Class<?> type = REGISTRY.get(id);
    if (type != null) return context.constructType(type);

    // 2. Fallback — split on ":", use prefix to find parent
    int colon = id.indexOf(':');
    if (colon > 0) {
        String prefix = id.substring(0, colon);
        Class<?> fallback = FALLBACK_MAP.get(prefix);
        if (fallback != null) {
            LOG.warnf("Unknown type '%s' — falling back to %s", id, fallback.getSimpleName());
            return context.constructType(fallback);
        }
    }
    throw new IOException("Cannot resolve type ID: " + id);
}
```

Serialization produces the compound ID automatically:
```java
public String idFromValue(Object value) {
    DeviceEntity device = (DeviceEntity) value;
    return device.deviceClass().name() + ":" + value.getClass().getSimpleName();
}
```

Plugin modules register via `registerType("THERMOSTAT:HomeAssistantThermostat", HomeAssistantThermostat.class)` — NOT `context.registerSubtypes()` (see GE-20260617-13de83).

### Why this is non-obvious

Most Jackson polymorphic hierarchies use a simple discriminator (class name or enum). The compound format is unusual but solves three problems simultaneously: (1) distinguishes supplements from parents, (2) embeds fallback information for graceful degradation, (3) works within the constraints of `TypeIdResolver` which only receives the type ID string. A developer facing this problem would typically reach for `DeserializationProblemHandler` first — which doesn't work for the reasons in GE-20260617-e3bedc.
