### What happened
A `@DefaultBean @ApplicationScoped` adapter wraps a blocking SPI as reactive using `Uni.createFrom().item(supplier).runSubscriptionOn(Infrastructure.getDefaultWorkerPool())`. Under load the Quarkus application hangs intermittently. Startup succeeds, individual requests succeed, but under concurrent load the service stops responding.

### Root cause
`runSubscriptionOn(workerPool)` schedules the supplier onto the worker pool and the calling thread blocks (via `await()` or the reactive chain's thread) waiting for it to complete. If the calling thread is **already a worker pool thread** — as it is in both:
- A blocking Quarkus request handler (already on worker pool)
- A reactive pipeline Stage 1 (`runSubscriptionOn(workerPool)` already applied)

…then the calling thread occupies a worker pool slot while waiting for the adapter's `runSubscriptionOn` to schedule the supplier. Under saturation (all worker pool slots busy), no slot is free to execute the supplier. Every in-flight request is waiting, and none can proceed. Classic thread-pool deadlock.

The symptom is intermittent hangs under load — not startup failures, CDI errors, or single-request failures — making it appear as a performance or infrastructure problem rather than a concurrency bug.

### Fix
Remove `runSubscriptionOn` from the adapter entirely:

```java
// Deadlocks under saturation — DON'T DO THIS
return Uni.createFrom()
          .item(() -> blocking.get(key))
          .onFailure().recoverWith(e -> Uni.createFrom().item(Optional.empty()))
          .runSubscriptionOn(Infrastructure.getDefaultWorkerPool()); // ← remove this

// Correct — supplier runs synchronously on the calling thread (already a worker)
return Uni.createFrom()
          .item(() -> blocking.get(key))
          .onFailure().recoverWith(e -> Uni.createFrom().item(Optional.empty()));
```

The supplier runs synchronously on the calling thread, which is already a worker pool thread. For native async implementations (Redis reactive client, etc.), the `Uni` they return is inherently non-blocking — no `runSubscriptionOn` is needed there either.

### Why this is non-obvious
Adding `runSubscriptionOn` to ensure blocking work stays off the event loop is correct defensive coding — in most places. The trap is that the correctness of the pattern depends on **who is calling the adapter**. When both callers are already on the worker pool (which is the common case for adapters used inside reactive pipelines or blocking request handlers), the thread-hop becomes a self-referential wait. The "safe" annotation causes the deadlock.

### Constraints
Applies when: (1) the adapter is called from a thread that is itself a worker pool thread, and (2) the worker pool is under saturation. In low-load environments the deadlock doesn't manifest because there are always free threads.
