### Symptom
`no suitable constructor found` when a package-private constructor takes `List<? extends SomeInterface<?>>` and the caller passes `List.of(stub1, stub2)` where each stub is typed as `SomeInterface<?>`. The inferred type of `List.of(...)` is `List<SomeInterface<? extends Object>>` which is not assignable to `List<? extends SomeInterface<?>>`.

### Context
Occurs when writing CDI-free unit tests for a registry bean that accepts a `List<? extends SomeInterface<?>>` (nested wildcard) as a test-only constructor. Each `stub()` factory method returns `SomeInterface<?>`. The compiler infers the captured wildcard from each element separately and cannot unify them into the expected bounded type.

### What was tried
1. Changed stub return type from `SomeInterface<String>` to `SomeInterface<?>` — still fails (produces `List<SomeInterface<? extends Object>>`)
2. Changed constructor parameter from `List<SomeInterface<?>>` to `List<? extends SomeInterface<?>>` — still fails
3. Explicit type witness `List.<SomeInterface<?>>of(...)` — still fails at the call site

### Root cause
Java's type inference for `List.of()` with wildcard elements captures each element's wildcard separately and cannot unify them. `List<RenderableProjection<?>>` (existential wildcard) and `List<? extends RenderableProjection<?>>` are treated as incompatible when produced by inference, even though statically they should be compatible.

### Fix
Accumulate into an explicit `ArrayList<SomeInterface<?>>` before passing, anchoring the type via a named variable declaration:
```java
// Works — explicit return-type annotation forces the correct list type
private static List<RenderableProjection<?>> stubs(String... names) {
    List<RenderableProjection<?>> list = new ArrayList<>();
    for (String name : names) list.add(stub(name));
    return list;
}
```
The declared return type `List<RenderableProjection<?>>` anchors inference. The named list is then assignable to `List<? extends RenderableProjection<?>>` successfully.

### Why this is non-obvious
`List.of(foo, bar)` where both are `SomeInterface<?>` appears to produce `List<SomeInterface<?>>`. In practice the compiler infers each element's wildcard capture independently and synthesises `List<SomeInterface<? extends Object>>`, which is not a `List<? extends SomeInterface<?>>`. The error message names the wrong type, and the explicit type witness `List.<T>of()` does not fix it — only breaking the inference chain via a named variable with an explicit type declaration works.
