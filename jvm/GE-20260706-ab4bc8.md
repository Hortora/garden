### Context
When a `ConcurrentHashMap<K, V>` holds a mutable collection as its value (e.g.
`ConcurrentHashMap<Key, List<Item>>`), the instinct is to use a thread-safe inner
collection like `CopyOnWriteArrayList` or `Collections.synchronizedList()`.

### The technique
`ConcurrentHashMap.compute()` (and `computeIfAbsent`, `merge`) already serializes
access per key — the lambda runs under the key's bin lock. Inside the lambda, the
inner collection is accessed by exactly one thread at a time. A plain `ArrayList`
is both correct and more efficient.

```java
// Wasteful — CopyOnWriteArrayList copies the entire array on every add/remove
map.compute(key, (k, entry) -> {
    var list = new CopyOnWriteArrayList<Item>();  // O(n) copy per mutation
    list.add(item);
    return new Entry(list, Instant.now());
});

// Correct and efficient — ArrayList inside compute() is single-threaded
map.compute(key, (k, entry) -> {
    var list = new ArrayList<Item>();  // O(1) amortized add
    list.add(item);
    return new Entry(list, Instant.now());
});
```

### Why this is non-obvious
The reflex to use thread-safe collections inside concurrent data structures is deeply
ingrained. `CopyOnWriteArrayList` "looks right" because the outer map is concurrent.
But `compute()` already provides the synchronization — the inner thread-safe collection
adds O(n) overhead per mutation (full array copy) for zero benefit. The performance
difference is stark for eviction patterns: removing element 0 from a 500-element
`CopyOnWriteArrayList` copies 499 elements; from an `ArrayList` it shifts 499 references
in-place (still O(n) but without allocation).

### Caveat
This only applies inside `compute()`/`computeIfAbsent()`/`merge()` lambdas. If the value
is also accessed via `get()` outside these methods, the inner collection IS accessed
concurrently and DOES need its own synchronization. The pattern is safe when all mutations
go through `compute()` and reads go through `drain()` (which removes the entry atomically).
