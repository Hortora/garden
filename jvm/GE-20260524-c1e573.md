### Symptom

Adding a fully qualified class name to `%test.quarkus.arc.exclude-types` has no effect — the bean is still discovered and its unsatisfied injection points cause `DeploymentException` at Quarkus build time. No warning or error indicates the exclusion was ignored.

### Context

Consumer application (Claudony) depends on a Quarkus extension (casehub-ledger) that added new beans (`LedgerVerificationService`, `LedgerMerklePublisher`, etc.) injecting `LedgerMerkleFrontierRepository`. The consumer doesn't provide an implementation of that repository. Attempts to exclude the offending beans via `exclude-types` all failed silently.

### Root cause

`quarkus.arc.exclude-types` only filters beans discovered through ARC's standard CDI bean discovery (Jandex index scanning). Quarkus extensions register their beans via the extension's **deployment build step** — these bypass `exclude-types` entirely. A JAR is an extension if it contains `META-INF/quarkus-extension.properties`.

Confirmed by multiple Quarkus GitHub issues: #9545, #34011, #14485.

### Fix

The fix must come from the **extension author**, not the consumer:

1. **`@DefaultBean` no-op implementation** (preferred) — provide a no-op impl of the unsatisfied interface so consumers without the real impl still boot:
   ```java
   @DefaultBean
   @ApplicationScoped
   public class NoOpLedgerMerkleFrontierRepository implements LedgerMerkleFrontierRepository {
       // no-op stubs
   }
   ```

2. **`ExcludedTypeBuildItem`** in the deployment module — gate optional beans behind a build-time config property (see GE-20260519-244ad2 for the technique).

3. **`Instance<T>` programmatic lookup** — tolerate missing beans at injection time instead of failing at boot.

`quarkus.arc.exclude-dependency` can exclude the entire artifact but removes ALL beans including needed ones — not viable for selective exclusion.

### Why this is non-obvious

`exclude-types` works perfectly for non-extension JARs. There is no log message, warning, or error when it silently fails for extension beans. The Quarkus CDI reference guide documents `exclude-types` without mentioning this limitation. A developer will try multiple patterns and syntax variations before discovering the exclusion mechanism is fundamentally inapplicable.

### Alternatives considered

- **FQCN in exclude-types** — silently ignored for extension beans
- **Glob patterns (`io.casehub.ledger.runtime.service.**`)** — same result
- **`quarkus.arc.exclude-dependency`** — removes all beans from the artifact, including ones the consumer needs
- **Regex patterns** — not supported by exclude-types syntax
