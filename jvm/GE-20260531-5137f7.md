## mvn install reports BUILD SUCCESS on wrong git branch — installed jar silently missing new class

### Symptom

Consumer project fails to compile with `cannot find symbol: class Foo` immediately after the producer SNAPSHOT jar was installed locally with `BUILD SUCCESS`. The new class exists in source — the developer knows they added it.

### Context

Two git repos: a producer (e.g. `casehub-connectors`) that publishes a SNAPSHOT jar, and a consumer (e.g. `casehub-qhorus`) that depends on it. The developer adds a new class to the producer, commits it to branch A (main), then switches the producer repo back to branch B (a feature branch in progress) for unrelated work. Running `mvn install` in the producer while on branch B succeeds, but the installed jar is missing the new class.

### Root cause

Maven compiles the **working directory** source files — not the git HEAD of any branch, not the "latest" committed code. If the new class only exists on branch A and the working directory is on branch B, the file simply isn't there for javac to compile. `BUILD SUCCESS` is printed because all the files that *do* exist compile without error. Maven has no awareness of other branches or commits.

The installed jar at `~/.m2/repository/.../<artifact>-SNAPSHOT.jar` is then the branch-B version, silently missing the class.

### Fix

Switch the producer repo to the branch that has the new class, then re-run `mvn install`:

```bash
git -C /path/to/producer checkout main
mvn -f /path/to/producer/pom.xml install -DskipTests
```

**Diagnose before building the consumer** — verify the class is in the installed jar:

```bash
jar tf ~/.m2/repository/io/example/my-lib/0.2-SNAPSHOT/my-lib-0.2-SNAPSHOT.jar | grep NewClassName
```

A missing result here is conclusive. An empty result with `BUILD SUCCESS` in the producer is the hallmark of this issue.

### Why this is non-obvious

`BUILD SUCCESS` gives false confidence that the jar is correct and complete. The consumer's compile error (`cannot find symbol`) looks like a missing import, a wrong dependency version, or a source ordering problem — all of which are investigated before anyone thinks to check the producer's git branch. The diagnostic (`jar tf`) is not a standard Maven debugging reflex. Nothing in Maven's output mentions the working directory branch or hints that source files are absent from the build.
