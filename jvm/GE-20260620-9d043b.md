### Symptom

Chunks graded twice, quality events fired twice, expansion may re-trigger on already-filtered results. No error, no warning. Data silently corrupted — quality counters double-count, and the expansion cycle may re-expand results that the first decorator already processed.

### Context

When a CDI `@Decorator` exists on both a blocking SPI (`CaseRetriever`) and its reactive counterpart (`ReactiveCaseRetriever`), and a `BlockingToReactiveCaseRetriever` bridge wraps the blocking path for reactive consumers, the decorator chain becomes:

```
ReactiveCorrectiveCaseRetriever (@Decorator on ReactiveCaseRetriever)
  → BlockingToReactiveCaseRetriever (@DefaultBean)
    → CorrectiveCaseRetriever (@Decorator on CaseRetriever)
      → HybridCaseRetriever
```

The reactive decorator sees the bridge as just another `ReactiveCaseRetriever`. It has no way to know that the bridge internally delegates to a blocking `CaseRetriever` that is itself decorated. Both decorators fire independently on the same retrieval.

This only occurs in the default mode (blocking bridge active). When native reactive is enabled (`ReactiveHybridCaseRetriever` replaces the bridge), only the reactive decorator fires — correct.

### What was tried (didn't work)

- `@IfBuildProperty` gating on the reactive decorator — untested on `@Decorator` in Quarkus Arc, and creates asymmetry (blocking always on, reactive conditionally on)
- Tracking a "processed" boolean — requires threading state through the bridge, coupling the bridge to CRAG awareness

### Root cause

CDI `@Decorator` activation is per-interface. When two interfaces share a blocking-to-reactive bridge, decorators on both interfaces fire in sequence. CDI has no concept of "this data was already processed by a decorator on a different interface in the same call chain." The bridge is architecturally invisible — it's just an implementation of `ReactiveCaseRetriever`.

### Fix

Use a domain enum value as an idempotency signal. In CRAG, `RelevanceGrade.UNGRADED` means "never evaluated" — any non-UNGRADED value means "already processed by a decorator."

```java
// Shared precondition in CragEvaluationLogic — used by both decorators
static boolean isAlreadyGraded(List<RetrievedChunk> chunks) {
    return !chunks.isEmpty()
        && chunks.stream().noneMatch(c -> c.grade() == RelevanceGrade.UNGRADED);
}

// In both decorators, at the top of retrieve():
if (CragEvaluationLogic.isAlreadyGraded(chunks)) {
    return chunks; // no evaluation, no event — decorator did no work
}
```

The domain enum serves double duty: consumer-facing quality signal AND decorator idempotency marker. No additional state, no bridge coupling, no activation gating.

### Why this is non-obvious

The bridge pattern is standard in Quarkus CDI — `BlockingToReactiveCaseRetriever` wraps a blocking call in `Uni.createFrom().item().runSubscriptionOn()`. Nothing in the bridge suggests it will cause decorator double-application. The problem only manifests when decorators exist on BOTH the blocking and reactive interfaces, which is a composition that doesn't appear until the reactive decorator is added — the blocking decorator works perfectly in isolation.

### Alternatives considered

- **`@IfBuildProperty` on the reactive decorator**: Avoids double-application by making the reactive decorator opt-in. But this is a configuration workaround, not a design fix — it couples the activation model to knowledge of the bridge's existence, and `@IfBuildProperty` on `@Decorator` is untested in Quarkus Arc.
- **Producer-based approach instead of `@Decorator`**: Replaces CDI interception with manual wrapping. Works but loses the clean decorator semantics and requires explicit delegate wiring.
- **Boolean "processed" flag on the data**: Requires threading state through the bridge, coupling the bridge implementation to CRAG-specific awareness.

### Caveats

The guard assumes UNGRADED is the correct "never processed" sentinel. If a future evaluator legitimately returns UNGRADED (meaning "I evaluated this but couldn't determine a grade"), the guard would re-evaluate. The current RelevanceEvaluator contract forbids this — implementations must return CORRECT, AMBIGUOUS, or INCORRECT.
