### The technique
When multiple Drools rules must contribute to a numeric result (weights,
scores, priorities), have each rule emit a signed delta record rather than
writing the final value directly. A Java post-processing step accumulates
deltas, applies floor clamping, and normalises.

### Why use it
GE-20260418-58cae8 documents that Drools salience controls firing order,
not mutual exclusion — all matching rules fire. Naive approaches write the
final value in each rule, meaning the last rule wins unpredictably. The
additive modifier pattern sidesteps this entirely: every rule contributes,
order is irrelevant, and the result is deterministic.

### Structure

```
1. Java: compute baseline (e.g. via interpolation)
2. Drools: rules emit WeightModifier(economyDelta, armyDelta, ..., reason)
3. Java: sum deltas per dimension, floor at minimum, normalise to 1.0
```

Rules are purely declarative — they match on situational signals and emit
deltas. No rule needs to know what other rules fire. Combined-signal rules
(matching two signals simultaneously) stack naturally with individual rules.

### When to use
Any Drools rule set where multiple rules should contribute to a numeric
output — weight vectors, confidence scores, priority rankings. Not
appropriate when rules are genuinely mutually exclusive (use constraint
guards for that).

### Why this is non-obvious
The natural Drools approach is to write the result in the `then` block.
With salience, it looks like you can control which rule "wins." But
salience only controls order — all matching rules fire, and the last write
overwrites earlier ones. The modifier pattern reframes the problem from
"which rule wins" to "how do all matching rules compose."
