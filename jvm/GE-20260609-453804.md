## Symptom

Compile error — `<field>` has private access or `<method>` is not public — in a cross-package test class, even after the enclosing class was promoted to `public`.

## Context

A test support class (`McpToolTestSupport`) was package-private in `io.casehub.connectors.mcp`. A new test lived in `io.casehub.connectors.slack.bot`. The fix was to change `final class McpToolTestSupport` to `public final class McpToolTestSupport` — but the compile error remained. `lastConnectorId`, `lastDestination`, `lastContent`, and `reset()` were still package-private, inaccessible from the different package.

## Root cause

Java visibility is per-member, not per-class. Making the enclosing class `public` makes the class name resolvable from other packages, but each member retains its own declared access level. Package-private members (`String lastField`, `void reset()`) do not inherit `public` from the enclosing class declaration.

## Fix

Explicitly add `public` to each member that needs cross-package access:

```java
// Before — all package-private
final class McpToolTestSupport {
    static final class RecordingBridge implements ConnectorMeshBridge {
        String lastConnectorId;
        String lastDestination;
        String lastContent;
        void reset() { ... }
    }
}

// After — every member that external tests need is now public
public final class McpToolTestSupport {
    public static final class RecordingBridge implements ConnectorMeshBridge {
        public String lastConnectorId;
        public String lastDestination;
        public String lastContent;
        @Override public void notifyDelivered(...) { ... } // already public (interface impl)
        public void reset() { ... }
    }
}
```

## Why this is non-obvious

The intuition is: "I made the class public — now it's public." Java doesn't work that way. The class declaration controls whether the class name can be referenced across packages; member access is governed by each member's own modifier. `@Override` methods that implement a public interface method ARE automatically public (the interface enforces it), which partially reinforces the wrong intuition — some members work, others don't, leading to confusion about why only some of the class is accessible.
