## Symptom

Setting `quarkus.flyway.migrate-at-start=false` in `src/test/resources/application.properties` has no effect when a dependency jar's `META-INF/microprofile-config.properties` contains `quarkus.flyway.migrate-at-start=true`. Flyway still runs migrations at test startup.

## Context

A dependency module (`casehub-persistence-hibernate`) shipped `quarkus.flyway.migrate-at-start=true` in its main resources (`META-INF/microprofile-config.properties`) to auto-migrate production databases. When that module was on the test classpath of another module, this setting was picked up at augmentation time and baked in — making it impossible to disable via a test-specific `application.properties`.

## Root Cause

`quarkus.flyway.migrate-at-start` is declared as `@ConfigRoot(phase = BUILD_AND_RUN_TIME_FIXED)` in Quarkus's Flyway extension. Properties in this phase are resolved once at build/augmentation time and cannot be changed at runtime. The value from `META-INF/microprofile-config.properties` (ordinal 100) is lower priority than `application.properties` (ordinal 250) in theory, but because augmentation bakes the value in, by the time tests run the override has no effect.

## What was tried

- Setting `quarkus.flyway.migrate-at-start=false` in test `application.properties` — no effect
- Setting `quarkus.flyway.enabled=false` — same problem (also build-time fixed)
- `quarkus.arc.exclude-types` for Flyway beans — does not work for extension-registered beans

## Fix

Provide a datasource that Flyway can connect to (H2 in-memory with `MODE=PostgreSQL`), and explicitly override the locations to exclude migrations from dependency jars:

```properties
quarkus.datasource.db-kind=h2
quarkus.datasource.jdbc.url=jdbc:h2:mem:testdb;MODE=PostgreSQL;DB_CLOSE_DELAY=-1
quarkus.hibernate-orm.schema-management.strategy=drop-and-create
```

Or, check whether the `<phase>` in the Flyway config class changed in your Quarkus version by checking if the property can be overridden at test time. If still baked in, the only reliable fix is ensuring the dependency jar does NOT set this property — move it to the consuming application's configuration instead.

## Why this is non-obvious

The standard MicroProfile Config priority ordering (`application.properties` > `microprofile-config.properties`) leads developers to expect that a value in their test `application.properties` always wins. This is true for runtime properties but not for build-time fixed ones. Quarkus does not emit a warning when a build-time property is "overridden" in a way that has no effect.
