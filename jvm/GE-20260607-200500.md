### What was discovered

Under H2 with `MODE=PostgreSQL` and JTA transaction management:

- A `SELECT MAX(sequenceNumber) FROM LedgerEntry WHERE subjectId = :sid` in a `REQUIRES_NEW` transaction returns **null** (no rows visible) even when an outer transaction has already executed `em.flush()` for an entry with the same subject.

- BUT: when the `REQUIRES_NEW` transaction tries to `INSERT` with `sequenceNumber=1`, H2 throws a unique constraint violation — because it **does** detect the flushed-but-uncommitted row for the constraint check.

**H2's unique constraint check sees rows that its SELECT queries do not.** Under READ COMMITTED isolation, dirty reads (seeing uncommitted data via SELECT) are blocked, but unique constraint enforcement checks committed AND flushed-but-uncommitted rows to prevent phantom conflicts.

### Context

This surfaced during investigation of `IDX_LEDGER_ENTRY_SUBJECT_SEQ` violations in the casehub-aml test suite:

1. Outer REQUIRED transaction: `em.flush()` persists `AmlCaseOpenedLedgerEntry(subject=caseId, seq=1)` — not yet committed
2. REQUIRES_NEW transaction: `SELECT MAX(seq) FROM LedgerEntry WHERE subject=caseId` → returns null (isolation blocks dirty read)
3. REQUIRES_NEW assigns seq=1, tries INSERT → constraint violation (H2 sees the flushed row)

The gap between "invisible to SELECT" and "visible to constraint check" is not documented in H2's official docs or in Hibernate's isolation documentation. It is consistent with SQL standard uniqueness enforcement semantics but surprising in an embedded test database.

### Why this is non-obvious

PostgreSQL (H2's emulation target) uses row-level locking for unique constraint enforcement — a REQUIRES_NEW insert on the same key would block, not fail, until the outer transaction commits or rolls back. H2 does not have row-level locking, so it does the constraint check eagerly and throws immediately. This asymmetry means tests passing under PostgreSQL (blocking) can fail under H2 (immediate violation) for concurrent write scenarios.

The practical consequence: you cannot rely on "the outer transaction hasn't committed, so REQUIRES_NEW can safely write key=1" in H2. The constraint enforcement happens at flush time, not at commit time.

### Workaround

Ensure the outer transaction **commits** the row before any REQUIRES_NEW transaction tries to write to the same constraint space, or use a lock-outside-transaction pattern (see GE-20260607-067ace) to serialize all writes for a given subject.
