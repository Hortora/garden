## Quarkus REST client interface with @Path at class level silently registers as a server resource — shadows the real endpoint

**Stack:** Quarkus 3.36.x, RESTEasy Reactive

**Symptom:** After adding a REST client interface (for programmatic use with `QuarkusRestClientBuilder`), existing REST endpoint tests break with wrong HTTP status codes. Endpoints that should return 400 return 200 with empty arrays. Domain-filtered searches return no results. No error messages, no warnings in logs.

**Context:** Creating a JAX-RS interface for use as a programmatic REST client (not `@RegisterRestClient`). The interface has `@Path("/search")` at the class level and `@GET` on the method — identical annotations to the server resource.

### What was tried (didn't work)

- Checked CDI wiring, federation config parsing, SearchResource code — all correct
- Verified the blank-query check was still in the `search()` method — it was
- Checked whether `quarkus-rest-client-jackson` dependency caused a conflict — it didn't

### Root cause

Quarkus REST discovers **any class or interface** annotated with `@Path` at the class level as a JAX-RS resource — including interfaces intended for client use only. When `RemoteGardenClient` (a public interface with `@Path("/search")` and `@GET`) was on the classpath, Quarkus generated a no-op implementation and registered it as a resource alongside the real `SearchResource`. The no-op implementation matched first for some requests, returning 200 with empty results instead of delegating to the real resource.

### Fix

Make the client interface **package-private** (not `public`) and move `@Path` to the method level:

```java
@Consumes(MediaType.APPLICATION_JSON)
interface RemoteGardenClient {  // package-private — not discovered as a resource

    @GET
    @Path("/search")  // method-level only
    List<SearchResult> search(
            @QueryParam("q") String query,
            @QueryParam("limit") int limit
    );
}
```

The `QuarkusRestClientBuilder.build(RemoteGardenClient.class)` still works — the builder reads method-level annotations for client-side request construction. It does not require class-level `@Path`.

### Why this is non-obvious

The symptom appears on the **real** endpoint, not the client interface. Nothing in the logs mentions a duplicate resource registration. The failure mode — wrong HTTP status codes on endpoints you didn't touch — gives zero indication that a client interface is involved. The natural debugging path (inspect SearchResource, check CDI, verify query parameter handling) leads nowhere.

### Why this fix

Making the interface package-private is cleaner than `@RegisterRestClient` (which would require CDI and config for a single-URL client) or removing class-level annotations entirely (which would require base URI to include the path). Package-private interfaces are invisible to Quarkus resource scanning while still usable by the programmatic client builder.

### Alternatives considered

- **`@RegisterRestClient`** — rejected because it creates a single CDI bean per interface, requiring `application.properties` config per URL. Programmatic creation via `QuarkusRestClientBuilder` is needed for N dynamic upstream URLs.
- **Removing all `@Path` annotations** — rejected because `QuarkusRestClientBuilder.build()` needs at least a method-level `@Path` to construct the request URL.
