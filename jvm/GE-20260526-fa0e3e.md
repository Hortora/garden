### Symptom

Writing a filter `plan.getPlanItemByBindingName(name).map(pi -> pi.getStatus() == PlanItemStatus.PENDING).orElse(true)` to block re-dispatch of completed bindings always permits re-dispatch for COMPLETED items. A TDD test asserting that a COMPLETED PlanItem prevents re-dispatch passes immediately before any production code is written — which is the signal that the test is asserting the wrong behavior.

### Context

Occurs when implementing `filterToDispatchable`-style logic on `CasePlanModel` to prevent binding re-dispatch during repeated `CONTEXT_CHANGED` evaluations. The behavior is by design — iterative cases must be able to re-fire completed bindings when their trigger conditions hold again.

### Root cause

`DefaultCasePlanModel.getPlanItemByBindingName()` only returns items in PENDING, RUNNING, or DELEGATED status. Items in COMPLETED, FAULTED, or CANCELLED status return `Optional.empty()`. Combined with `addPlanItemIfAbsent()` silently replacing terminal-status items with a new PENDING one on re-evaluation, completed bindings are unconditionally re-dispatchable — the "absent" result from the getter and the new PENDING item are the same from the caller's perspective.

### Fix

Filter on RUNNING and DELEGATED only — the genuinely in-flight states. The `.orElse(true)` branch correctly passes terminal-status bindings through because they appear absent (new PENDING will be created by `addPlanItemIfAbsent`). The correct predicate:

```java
// In filterToDispatchable — prevents re-dispatch of in-flight bindings only
plan.getPlanItemByBindingName(b.getName())
    .map(pi -> pi.getStatus() == PlanItemStatus.PENDING)
    .orElse(true)   // terminal items appear absent → allow re-dispatch
```

This passes PENDING (just created, dispatch it) and absent (terminal, re-create as PENDING) items, and blocks RUNNING/DELEGATED (in-flight, don't re-dispatch).

### Why this is non-obvious

`getPlanItemByBindingName` implies returning any item with that name regardless of state. The status gate is invisible from the method signature. The `addPlanItemIfAbsent` interaction compounds the surprise: terminal items are silently replaced with new PENDING ones, so a filter expecting to find COMPLETED items and block on them will always miss them — the items appear absent rather than present-and-terminal.

The TDD signal: if a test asserting "COMPLETED PlanItem prevents re-dispatch" passes immediately before writing production code, the test is wrong. The implementation should make the test fail first.
