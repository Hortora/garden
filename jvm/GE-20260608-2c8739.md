### Symptom

A `@QuarkusTest` class injects an `@ApplicationScoped` bean and calls a package-private method on it:

```java
// io.example.agent.ScoutingIntelBroker (package io.example.agent)
@ApplicationScoped
public class ScoutingIntelBroker {
    void clearLatest() { latest.clear(); }  // package-private
}

// io.example.plugin.DroolsTacticsTaskIT (package io.example.plugin)
@QuarkusTest
class DroolsTacticsTaskIT {
    @Inject ScoutingIntelBroker broker;

    @BeforeEach void reset() {
        broker.clearLatest();  // ← compile error
    }
}
```

Compile error:
```
error: clearLatest() is not public in io.example.agent.ScoutingIntelBroker;
       cannot be accessed from outside package
```

### Context

The error occurs when:
1. A CDI bean is in package `com.example.agent`
2. A test class in package `com.example.plugin` injects the bean
3. The test calls a package-private method on the injected reference

This is surprising because CDI appears to make the bean available across the application — it feels like a "global service" — and other CDI-based magic (like interceptors, observers) works transparently across packages. The proxy illusion leads developers to believe package visibility is irrelevant once CDI is involved.

### Root Cause

CDI injection provides a **proxy** object, not the bean instance itself. The proxy's generated class has visibility to call the underlying bean's methods — but the **caller's compile-time type check** still applies. When a test class in `com.example.plugin` holds a reference typed as `ScoutingIntelBroker`, the Java compiler rejects calls to package-private methods because the calling class is outside the bean's package.

CDI doesn't change Java's fundamental access rules. The proxy intercepts calls — but only to methods that are visible to the caller in the first place. Package-private methods are outside that set.

### Fix

Make the method `public`:

```java
@ApplicationScoped
public class ScoutingIntelBroker {
    /** Test isolation — called from @BeforeEach across multiple @QuarkusTest packages. */
    public void clearLatest() { latest.clear(); }
}
```

Or, if the method should not be part of the public API, restrict `@QuarkusTest` to the same package as the bean (put the test in `io.example.agent` test sources). This works when the method is used by only one test class.

### Why this is non-obvious

CDI proxy magic works across packages for everything else:
- `@Inject` resolves across packages ✅
- `@Observes` and `@ObservesAsync` fire across packages ✅
- `@Interceptor` and `@Decorator` apply across packages ✅

The expectation — reasonable given the above — is that CDI "opens up" the bean. It doesn't. CDI is transparent to the method call mechanism but not to the Java access control model. The compile error message ("not public") looks like a simple Java visibility issue, not a CDI architecture problem, so developers often try CDI-specific solutions (qualifiers, `@Instance`, different scope) rather than just making the method public.
