## Panache static methods bypass CDI @Alternative stores — returns empty results silently

### Symptom

Test suite activates `@Alternative @Priority(1)` InMemory stores via `quarkus.arc.selected-alternatives`. Panache static queries (`Entity.<Entity>listAll()`, `Entity.find(...)`) return zero results with no error, even though data was inserted through the store.

### Root cause

Panache static methods resolve directly against JPA at the persistence layer, bypassing CDI entirely. `@Alternative` implementations intercept calls made through injected CDI interfaces, not through static Panache entry points. With InMemory stores active, the JPA entity tables are empty — Panache statics query empty tables and return zero results silently.

### Fix

In test code that activates InMemory stores, never use Panache statics to query domain entities. Always go through the injected store or service layer:

```java
// Wrong — bypasses CDI alternative, queries empty JPA table
List<Channel> channels = Channel.<Channel>listAll();

// Correct — routes through the injected alternative store
List<Channel> channels = channelStore.findAll();
// or via service/tool layer
var result = tools.listChannels();
```

### Why this is non-obvious

Panache statics look like repository methods and feel like they should respect CDI alternatives — but they don't. No error is thrown, no warning is logged; the query simply returns an empty collection. The only signal is silently wrong test data.
