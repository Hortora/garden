### Symptom
Adding a JAX-RS `@Provider` (e.g. `ParamConverterProvider`) to a module that doesn't expose REST endpoints, so you add the first `quarkus-*` dep that makes it compile — `quarkus-rest`. Tests pass, but you've pulled the full Quarkus REST runtime into a module that only needed annotation interfaces.

### Context
`@Provider`, `ParamConverter`, and `ParamConverterProvider` are JAX-RS API interfaces from `jakarta.ws.rs.ext`. The full Quarkus REST extension (`quarkus-rest`) is an implementation of JAX-RS, not just the API. Adding it to a utility module forces every downstream consumer to carry the REST runtime on the classpath unnecessarily.

### Fix
Use the bare API dependency with `provided` scope instead:

```xml
<!-- Wrong — full Quarkus REST runtime -->
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-rest</artifactId>
</dependency>

<!-- Correct — JAX-RS API only, provided by the consuming app -->
<dependency>
    <groupId>jakarta.ws.rs</groupId>
    <artifactId>jakarta.ws.rs-api</artifactId>
    <scope>provided</scope>
</dependency>
```

Both are managed by the Quarkus BOM — no explicit version needed.

### Why this is non-obvious
`quarkus-rest` is the first search result when you need JAX-RS in Quarkus. The distinction between "JAX-RS API" (just interfaces) and "Quarkus REST" (implementation + framework) isn't obvious unless you've hit the issue before. The code compiles and tests pass either way — the bloat is invisible at development time.
