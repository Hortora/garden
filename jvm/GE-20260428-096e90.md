## Symptom

`entity.delete()` or `repository.deleteById(id)` throws a database constraint violation
at runtime — even though the entity exists and the caller has the right id. No warning
at compile time or startup.

```
ERROR: update or delete on table "channel" violates foreign key constraint
"fk_message_channel" on table "message"
```

## Context

A parent entity (e.g. `Channel`) has child entities (e.g. `Message`) with a FK column
pointing at the parent. The FK is defined in a Flyway/Liquibase migration without
`ON DELETE CASCADE`. The JPA `@Column` annotation on the child side has no `CascadeType`.

Calling `Channel.deleteById(id)` succeeds when the channel is empty but fails silently
(at runtime) when messages exist.

## What was tried (didn't work)

Calling `channelStore.delete(ch.id)` directly after checking message count ≥ 1.
The DB rejects it with a FK violation even when force=true was intended.

## Root cause

No CASCADE is configured at either the DB level (`ON DELETE CASCADE` in the DDL) or
the JPA level (`CascadeType.REMOVE` on the relationship). JPA does not automatically
clean up children. The database enforces referential integrity.

## Fix

Explicitly delete all children before deleting the parent. In Quarkus Panache:

```java
// Delete children first, then parent
messageStore.deleteAll(ch.id);   // Message.delete("channelId", channelId)
channelStore.delete(ch.id);      // Channel.deleteById(id)
```

Or add `ON DELETE CASCADE` to the FK in the migration (schema change — use with care).

## Why this is non-obvious

- Tests with InMemory stores don't enforce FK constraints at all — the delete
  appears to work in tests but fails in integration with a real database
- The Panache entity API (`deleteById`, `delete`) gives no indication that children
  must be handled first
- The constraint name in the error (`fk_message_channel`) is the only clue —
  developers must trace it back to the DDL migration to understand the FK
