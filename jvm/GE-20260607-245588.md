### Symptom
Publishing `WorkerRetriesExhaustedEvent` from a gate rejection/expiry handler causes the entire `CaseInstance` to transition to FAULTED state. The case dies instead of staying RUNNING for the rejection binding to react.

### Context
In casehub-engine's blackboard module, `WorkerRetryExhaustionHandler` consumes `WORKER_RETRIES_EXHAUSTED` and marks the PlanItem FAULTED. Separately, `WorkerRetriesExhaustedEventHandler` in the runtime module consumes the same event and transitions the `CaseInstance` state to FAULTED. Both handlers receive the same event (Vert.x publish = fan-out).

When trying to reuse `WorkerRetriesExhaustedEvent` to signal "gate rejected, mark PlanItem FAULTED", the runtime handler also runs and faults the entire case — which breaks gate rejection semantics (the case must stay RUNNING so the `actionGateRejected` context signal binding can react).

### Root cause
`WORKER_RETRIES_EXHAUSTED` is overloaded: it has two consumers with different effects. One faults a PlanItem (blackboard), one faults the CaseInstance (runtime). Both always fire. There is no way to reuse this event for "PlanItem-only fault" without also faulting the case.

### Fix
Create a distinct event type for gate-specific PlanItem faults (`ACTION_GATE_WORKER_FAULTED`) that only the blackboard consumes. The runtime gate handlers do NOT publish `WORKER_RETRIES_EXHAUSTED`.

```java
// Wrong — also faults the CaseInstance
eventBus.publish(WORKER_RETRIES_EXHAUSTED, new WorkerRetriesExhaustedEvent(caseId, workerId, idempotency));

// Correct — only faults the PlanItem via blackboard handler
eventBus.publish(ACTION_GATE_WORKER_FAULTED, new ActionGateWorkerFaultedEvent(caseId, workerId, idempotency));
```

### Why this is non-obvious
The event name `WorkerRetriesExhaustedEvent` sounds like a "worker state" event, not a "case state" event. The runtime handler that faults the case from it is not visible from the blackboard module — you'd need to search across modules to discover both consumers.
