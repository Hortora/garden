### Symptom
`closeTo(0.85, 0.01)` assertion fails with:
```
Expected: a numeric value within <0.01> of <0.85>
  Actual: <0.85F>
```

### Context
Writing `@QuarkusTest` integration tests that assert on `Double` fields in JSON
responses via REST Assured's `body()` method with Hamcrest matchers.

### Root cause
Jackson's `ObjectMapper` deserializes JSON floating-point numbers as `Float` (not
`Double`) when the value fits within Float precision. REST Assured uses Jackson to
parse the response body. Hamcrest's `closeTo()` requires a `Double` and fails the
type check against `Float`.

### Fix
Use `equalTo(0.85F)` instead of `closeTo(0.85, 0.01)`:

```java
// Wrong — fails with Float/Double mismatch
.body("items[0].score", closeTo(0.85, 0.01))

// Correct — matches the Float that Jackson produces
.body("items[0].score", equalTo(0.85F))
```

### Why this is non-obvious
`closeTo()` is the standard Hamcrest matcher for floating-point comparison with
tolerance. A developer writing a test for a `Double` field would naturally reach
for it. The error message mentions `<0.85F>` but doesn't explain WHY it's a Float
— the type mismatch is between Hamcrest's expected `Double` and Jackson's actual
`Float`, not a precision issue.
