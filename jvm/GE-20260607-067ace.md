### The technique

To safely assign sequence numbers across concurrent `@ObservesAsync` event handlers without a database-level sequence, acquire an application-level lock **outside** the transaction boundary. The lock must be held until after the transaction commits — not just until the persistence call returns.

The critical requirement: the `@Transactional(REQUIRES_NEW)` must be on a **separate CDI bean** (not `this`), so the CDI proxy applies the transaction interceptor. Self-invocation bypasses the proxy and the transaction never starts.

```java
// Non-transactional observer — holds the lock
@ApplicationScoped
public class AmlTrustRoutingObserver {
    private final ConcurrentHashMap<UUID, Object> subjectLocks = new ConcurrentHashMap<>();

    @Inject AmlTrustAttestationRepository attestationRepo; // @Transactional(REQUIRES_NEW)

    public void onWorkerDecision(@ObservesAsync WorkerDecisionEvent event) {
        final UUID attestationSubject = attestationSubjectFor(event.caseId());
        final AmlTrustRoutingAttestation entry = buildEntry(event, attestationSubject);

        // Lock acquired BEFORE transaction starts — held until AFTER it commits
        final Object lock = subjectLocks.computeIfAbsent(attestationSubject, k -> new Object());
        synchronized (lock) {
            attestationRepo.saveWithSequence(entry); // REQUIRES_NEW via CDI proxy
        } // lock released AFTER REQUIRES_NEW committed
    }
}

// Separate bean — @Transactional(REQUIRES_NEW) applied by CDI proxy
@ApplicationScoped
public class AmlTrustAttestationRepository {
    @Transactional(TxType.REQUIRES_NEW)
    public AmlTrustRoutingAttestation saveWithSequence(AmlTrustRoutingAttestation entry) {
        Integer max = em.createQuery(
            "SELECT MAX(e.sequenceNumber) FROM LedgerEntry e WHERE e.subjectId = :sid",
            Integer.class).setParameter("sid", entry.subjectId).getSingleResult();
        entry.sequenceNumber = max == null ? 1 : max + 1;
        em.persist(entry);
        return entry;
    }
}
```

### Why this is non-obvious

The obvious approach is `@Transactional(REQUIRES_NEW)` on the observer itself with `synchronized` inside. This fails: the `synchronized` block exits **before** the REQUIRES_NEW commits (the transaction commits when the method returns, which is after `synchronized` exits). The next thread then reads the same `max` before the first write is committed.

Putting the lock inside `saveWithSequence` (self-invocation) also fails: `this.saveWithSequence()` bypasses the CDI proxy, so no transaction starts at all.

The correct structure: lock in the non-transactional outer method, call the `@Transactional(REQUIRES_NEW)` method through a CDI-injected reference. Transaction starts when the injected method is entered, commits when it returns, all while the lock is still held.

### Constraints

- Requires single-JVM deployment (application-level lock doesn't span JVMs)
- Lock map grows with unique subjects — add eviction if subjects are unbounded in production
- Only needed when a database-level sequence or `SELECT FOR UPDATE` pattern is unavailable
- CDI proxy requirement means the `@Transactional` method must be on a public method of a `@ApplicationScoped` or `@RequestScoped` bean, not a `@Singleton` (proxy injection still required)

### Why this fix

Database sequences (CREATE SEQUENCE) would be cleaner for true scalability but require a Flyway migration and a per-subject table entry. This pattern solves the problem with zero schema changes, is easily testable, and makes the concurrency model explicit in code.
