## Symptom

A REST Assured test that asserts `body("expiresAt", equalTo(newExpiry))` passes locally but fails
intermittently in CI. `newExpiry` is constructed with `Instant.now().plus(30, DAYS).toString()`.

## Context

The test sends an `Instant` as a JSON string in the request body and asserts the response body
contains the same value. Jackson deserializes the request string to `Instant`, the service stores
it, and Jackson re-serializes it in the response.

Happens whenever:
- A test asserts a JSON response field against a Java-constructed `Instant.toString()`
- The `Instant` has non-zero nanoseconds (which `Instant.now()` almost always does)
- Jackson's serializer and `Instant.toString()` format nanoseconds differently (e.g. `"...000Z"` vs `"...Z"`)

## Root cause

`Instant.toString()` produces ISO-8601 with nanosecond precision only when non-zero nanoseconds exist:
- `Instant` with nanos → `"2026-06-25T12:34:56.123456789Z"`
- `Instant` with whole seconds → `"2026-06-25T12:34:56Z"`

Jackson's `InstantSerializer` (Jackson 2.x) serializes to ISO-8601 but may include or omit the
fractional part differently depending on configuration (`WRITE_DATE_TIMESTAMPS_AS_NANOSECONDS`,
`WRITE_DATES_AS_TIMESTAMPS`). The default Quarkus Jackson config does not guarantee the same
nanosecond representation as `Instant.toString()`.

The test fails when `Instant.now()` returns a value with nanoseconds that round differently through
the JSON serialization round-trip.

## Fix

Truncate to whole seconds before constructing the test string:

```java
// Before — intermittently fails
final String newExpiry = Instant.now().plus(30, ChronoUnit.DAYS).toString();

// After — always matches
final String newExpiry = Instant.now().plus(30, ChronoUnit.DAYS)
        .truncatedTo(ChronoUnit.SECONDS)
        .toString();
```

`truncatedTo(SECONDS)` removes sub-second precision. Both `Instant.toString()` and Jackson's
serializer produce `"2026-06-25T12:34:56Z"` for a whole-second value — no precision to diverge on.

## Why this is non-obvious

The failure is intermittent — `Instant.now()` occasionally returns a whole-second value (zero
nanoseconds), which makes the test pass. The bug hides itself until a run happens to capture a
non-zero-nanosecond timestamp. Developers typically add `.toString()` without thinking about
sub-second precision because ISO-8601 "looks the same" in casual inspection.

The serialization divergence is also configuration-sensitive: switching Jackson's timestamp config
can make the test pass or fail without touching the test code.
