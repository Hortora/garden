## Context

Migrating a field from a raw String to a validated value type (e.g. `Capability` record enforcing lowercase kebab-case) while existing DB rows contain format-violating values. A single strict parse mode breaks routing/processing of existing items immediately on deployment.

## Technique

Implement two static parse methods with different failure modes:

```java
// Strict — for new user input. Throw on bad format → 400 at the REST boundary.
static Set<Capability> parse(String raw) {
    if (raw == null || raw.isBlank()) return Set.of();
    return Arrays.stream(raw.split(","))
            .map(String::trim).filter(s -> !s.isEmpty())
            .map(Capability::of)           // throws MalformedCapabilityException
            .collect(Collectors.toUnmodifiableSet());
}

// Lenient — for existing DB rows. Skip bad tokens, log WARN.
static Set<Capability> parseLenient(String raw) {
    if (raw == null || raw.isBlank()) return Set.of();
    return Arrays.stream(raw.split(","))
            .map(String::trim).filter(s -> !s.isEmpty())
            .flatMap(s -> {
                try { return Stream.of(Capability.of(s)); }
                catch (MalformedCapabilityException e) {
                    LOG.warnf("Skipping malformed token in DB row: '%s'", s);
                    return Stream.empty();
                }
            })
            .collect(Collectors.toUnmodifiableSet());
}
```

- **Write path** (creation endpoint): call `parse()` — bad format surfaces as 400 immediately
- **Read path** (routing/querying from DB): call `parseLenient()` — routing continues, bad tokens excluded

Pair with a config switch (`casehub.work.capability-validation=PERMISSIVE`) so operators control when vocabulary enforcement (STRICT/WARN) activates, independently of the format enforcement already in the type constructor.

## Why This Is Non-Obvious

Most approaches choose one mode and apply it everywhere. Strict-everywhere breaks existing data on deployment. Lenient-everywhere hides format errors in new input. The insight is that **write and read paths have different tolerance requirements**: new input must be validated immediately, but existing DB data was previously accepted and must not silently break routing. Two parse modes with explicit callsite selection makes the contract visible rather than hiding it in a silent fallback.

## When To Apply

- Migrating a field from unvalidated String to a validated value type
- The DB column stays as its original wire format (comma-separated string) — no schema migration
- New data must be format-correct immediately; existing data is cleaned up operationally
