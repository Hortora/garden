### Symptom
`NullPointerException: Cannot invoke "com.github.javaparser.resolution.MethodUsage.getName()" because "method" is null` when evaluating any expression compiled via `MVEL.pojo(cls).out(resultType).expression(expr).compile()`.

### Context
Any MVEL3 expression using pojo context mode — e.g. compiling `name` or `age > 20` against a POJO class with standard JavaBean getters. The expression compiles without error; the NPE occurs on first `eval()` when the transpiler attempts to generate Java source.

### What was tried (didn't work)
- `MVEL.pojo(TestPerson.class).out(String.class).expression("name").compile()` — NPE in MVELCompiler
- Making the POJO class public and top-level (fixes the "inaccessible class" error but the NPE remains)
- `MVEL.pojo(cls).with(cls)` — avoids the NPE but requires `eval(pojo, pojo)` (passing the same object twice), and the generated evaluator still fails to resolve POJO property names in some cases

### Root cause
In `MVELToJavaRewriter.rewriteNode()` (line 517), the NameExpr handler checks `context.getEvaluatorInfo().withDeclaration().type().isVoid()`. When no `.with()` root is set, `withDeclaration` is `VOID_DECLARATION` — the check returns true and the method breaks without rewriting bare names to context property access.

This means `name` stays as a bare `NameExpr` in the generated Java source — javac can't resolve it. The NPE occurs later when `MVELCompiler` (line 180) attempts to call `.getName()` on a null `MethodUsage` during evaluator class generation.

The transpiler was designed so that bare-name resolution happens through the "with" root, not the main context. Without `.with()`, no rewriting occurs.

### Fix
Use JavaBean introspection to convert the POJO to a `Map<String, Object>`, then delegate to the Map compilation path:

```java
private static class PojoAdapterMvelExpression<C, R> implements CompiledExpression<C, R> {
    private final java.beans.PropertyDescriptor[] propertyDescriptors;
    // ... store expression, contextType, resultType, boundVars

    @Override
    public R eval(C context) {
        Map<String, Object> map = new HashMap<>();
        for (var pd : propertyDescriptors) {
            if (pd.getReadMethod() != null) {
                map.put(pd.getName(), pd.getReadMethod().invoke(context));
            }
        }
        // delegate to Map compilation path
        return mapDelegate.eval(map);
    }
}
```

The adapter is transparent to callers — `compile("name", Person.class, String.class)` works correctly. Property descriptors are cached per class via `Introspector.getBeanInfo()`.

### Why this is non-obvious
The NPE stack trace points at MVELCompiler (generated evaluator class), not at the transpiler's NameExpr handler. There's no indication that the problem is a missing `.with()` root. The `MVEL.pojo()` API signature suggests it should "just work" for POJOs — the design spec for this feature assumed MVEL3 would introspect the class via getters.
