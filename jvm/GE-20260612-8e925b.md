### Symptom
Native SQL tables (not JPA entities) are missing after `database.generation=drop-and-create` even though `import.sql` exists in `src/test/resources`. The table was expected to be created by the SQL script.

### Context
A test uses a named Hibernate ORM persistence unit (e.g. `quarkus`) configured via `quarkus.hibernate-orm.qhorus.datasource=qhorus`. After switching to `drop-and-create` schema generation, a native SQL table (`ledger_subject_sequence`) that exists in a Flyway migration is no longer created — and `import.sql` doesn't create it either.

### Root cause
In Quarkus, `import.sql` is executed **only for the default (unnamed) persistence unit**. Named persistence units require a separate config property:

```properties
quarkus.hibernate-orm.<pu-name>.sql-load-script=<filename>.sql
```

The Quarkus docs mention `import.sql` for the default PU but do not prominently document the named-PU equivalent.

### Fix
Create a separate SQL file and configure it for the named PU:

```properties
# For the default PU — import.sql works automatically
# For a named PU (e.g. "qhorus"):
%test.quarkus.hibernate-orm.qhorus.sql-load-script=import-qhorus.sql
```

Place `import-qhorus.sql` in `src/test/resources/`:

```sql
CREATE TABLE IF NOT EXISTS ledger_subject_sequence (
    subject_id UUID NOT NULL PRIMARY KEY,
    next_seq   INT  NOT NULL DEFAULT 1
);
```

### Why this is non-obvious
`import.sql` is the well-known Hibernate convention for post-schema SQL. Quarkus follows this convention for the default PU but uses a different mechanism for named PUs. The property name (`sql-load-script`) is documented in the Quarkus Hibernate ORM reference but not linked from the `import.sql` section, so developers naturally expect `import.sql` to apply universally.
