### Symptom
```
ERROR Exception in SSE server handling, impossible to send it to client:
io.smallrye.mutiny.subscription.BackPressureFailure:
Could not emit item downstream due to lack of requests
```
Logged as `ERROR` from `PublisherResponseHandler` on every WorkItem lifecycle transition,
even in tests where no SSE client is connected.

### Context
A CDI observer calls `BroadcastProcessor.onNext(event)` on every lifecycle event to fan
out to SSE subscribers. The error fires whenever there are no active SSE subscribers.

### Root cause
`BroadcastProcessor` is a strict reactive hot stream. "Lack of requests" means the
downstream subscriber has not issued a demand signal via `request(n)`. When there are
**zero subscribers**, there is nobody to issue any demand — so `onNext()` immediately
throws `BackPressureFailure`.

The error message ("lack of requests") implies a slow consumer that can't keep up. The
actual meaning is "no consumer exists at all." The distinction matters for diagnosis:
this is not a throughput or buffering problem.

### Fix
Catch and discard `BackPressureFailure` in the observer:

```java
public void onEvent(@Observes final WorkItemLifecycleEvent event) {
    try {
        processor.onNext(event);
    } catch (BackPressureFailure ignored) {
        // No SSE subscribers connected — hot stream drops events with no listener.
        // Correct: clients only receive events that occur while connected.
    }
}
```

This is semantically correct: a hot stream with no subscribers should drop events,
not buffer them indefinitely or fail.

### Why this is non-obvious
The error message "lack of requests" is standard reactive back-pressure vocabulary for
a slow consumer scenario — the natural reading is "consumer is not keeping up." The
actual cause (zero consumers) is the opposite of too many — there are none. Developers
investigating this error typically look for slow SSE clients or misconfigured demand
signals rather than for the simpler "nobody is subscribed" root cause. The fix (silent
catch) feels wrong for what looks like a production error, but is actually the correct
hot-stream semantic: fire-and-forget to whoever happens to be listening.
