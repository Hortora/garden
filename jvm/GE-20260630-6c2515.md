### Symptom
`ClassNotFoundException: io.micrometer.core.instrument.MeterRegistry` in a test class
that has nothing to do with metrics (e.g. `SlackBotBindingStoreTest`). The Quarkus CDI
augmentation fails globally — every `@QuarkusTest` in every module that transitively
depends on the module with the `MeterRegistry` constructor parameter fails.

### Context
A `@ApplicationScoped` CDI bean in a runtime module (`casehub-qhorus`) uses
`Instance<MeterRegistry>` in its constructor for optional injection. `micrometer-core`
was declared at `provided` scope — the pattern used successfully in a leaf module
(`connector-backend`) whose tests explicitly add `quarkus-micrometer`.

### What was tried (didn't work)
1. `provided` scope on `micrometer-core` — works in the leaf module but breaks all
   transitive dependents that don't have micrometer on their test classpath
2. Adding `quarkus-micrometer` as test scope to every downstream module — works but
   violates the principle that optional dependencies shouldn't force downstream changes

### Root cause
`Instance<MeterRegistry>` makes CDI *injection* optional (`.isResolvable()` returns
false when no bean is available). But the *class* must still be loadable — the CDI
container needs to parse the constructor signature during augmentation. `provided`
scope means the class is not on the transitive classpath, so any module that depends
on the runtime module transitively fails at class loading before CDI even gets to
the injection point.

### Fix
Use `compile` scope (default) for `micrometer-core` in any module that is transitively
depended upon. The class is always loadable; `Instance<T>.isResolvable()` handles the
"no bean available" case at runtime.

```xml
<!-- In runtime/pom.xml — compile scope, NOT provided -->
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-core</artifactId>
</dependency>
```

### Why this is non-obvious
The `connector-backend` module uses `provided` scope for `micrometer-core` and works
perfectly — because it is a leaf module whose tests explicitly add `quarkus-micrometer`.
Copying this pattern to a runtime module that other modules depend on transitively
produces a failure in seemingly unrelated test classes, with no indication that
micrometer or the delivery pump is involved.
