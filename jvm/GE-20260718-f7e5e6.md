### Symptom
Need to re-fire CDI events with a qualifier for typed dispatch without existing unqualified observers processing the event twice.

### Context
A dispatcher observes `@ObservesAsync CloudEvent` and re-fires with a type-specific qualifier (`@CloudEventType("io.casehub.cbr.outcome")`) so that downstream modules can observe only their event type. Other unqualified `@ObservesAsync CloudEvent` observers (DataSourceRouter, RasEngine, etc.) must not receive the re-fired event.

### Technique
CDI's `Event.select()` automatically removes the `@Default` qualifier when user qualifiers are added. This prevents double-processing without any guards:

1. **Original fire** (`cloudEventBus.fireAsync(event)`) — event has `@Default`, `@Any`. All unqualified observers (implicit `@Default`) receive it.

2. **Re-fire** (`cloudEventBus.select(new CloudEventTypeLiteral(type)).fireAsync(event)`) — event has `@CloudEventType(type)`, `@Any`. CDI removes `@Default` because user qualifiers are present. Unqualified observers (implicit `@Default`) do NOT match — no double processing, no infinite loop.

```java
@ApplicationScoped
public class CloudEventTypeDispatcher {
    @Inject Event<CloudEvent> cloudEventBus;

    public void onCloudEvent(@ObservesAsync CloudEvent event) {
        String type = event.getType();
        if (type == null || type.isBlank()) return;
        cloudEventBus.select(new CloudEventTypeLiteral(type))
            .fireAsync(event)
            .exceptionally(ex -> { LOG.errorf(ex, "..."); return event; });
    }
}

// Consumer — only receives "io.casehub.cbr.outcome" events
void onCbrOutcome(@ObservesAsync @CloudEventType("io.casehub.cbr.outcome") CloudEvent event) {
    // ...
}
```

The dispatcher itself has implicit `@Default` on its observer — so it also doesn't receive its own re-fired event. No infinite loop.

### Why this is non-obvious
CDI qualifier matching rules are specified in §10.3.1 of the CDI spec but the `@Default` removal on `select()` is not prominently documented. Most developers would assume that unqualified observers receive all events regardless of qualifiers (injection semantics) — but observer resolution follows different rules. The automatic protection of existing observers is a CDI spec guarantee, not something you need to code for.

### Alternatives considered
- Wrapper type (`TypedCloudEvent` record) to avoid type overlap — unnecessary complexity given CDI handles it
- Marker extension on CloudEvent to detect re-fired events — fragile, couples dispatcher to observers
- Firing with qualifier from the source (stream processors) — requires changing all sources and doesn't protect against non-platform CloudEvent sources
