## Symptom

Tests asserting `MessageTypeViolationException` for disallowed message types
on channels with `allowedTypes` constraints pass during initial development
but fail after qhorus integration. The messages are dispatched successfully
instead of being rejected.

## Context

`StoredMessageTypePolicy.validate()` enforces type constraints differently
based on message type:

- **COMMAND, QUERY** ‚Üí strict enforcement. If the type is in `deniedTypes` or
  not in `allowedTypes`, throws `MessageTypeViolationException`.
- **All other types** (EVENT, STATUS, DONE, DECLINE, FAILURE) ‚Üí advisory only.
  `validate()` does nothing. `advisory()` returns a warning string that is
  logged and included in `DispatchResult.advisories()`, but the message is
  dispatched normally.

A channel configured with `allowedTypes: "COMMAND,DONE,DECLINE"` will accept
EVENT, STATUS, and FAILURE messages without exception ‚Äî only logging a warning.

## What was tried

- Verified channel `allowedTypes` was correctly set
- Confirmed `MessageTypePolicy` was injected and called
- Assumed type enforcement was symmetric for all types

## Root cause

The enforcement asymmetry is intentional in the qhorus design: COMMAND and
QUERY are "speech acts" that create commitments and require strict gating.
Other types are informational or reactive and get soft enforcement via
advisories. This is not documented in the qhorus README or API docs.

## Fix

For tests verifying type constraints on non-COMMAND types, assert advisory
presence instead of exception:

```java
// Wrong ‚Äî non-COMMAND types don't throw
assertThatThrownBy(() -> messageService.dispatch(...))
    .isInstanceOf(MessageTypeViolationException.class);

// Correct ‚Äî check advisory in the result
var result = messageService.dispatch(...);
assertThat(result.advisories()).isNotEmpty();
```

COMMAND and QUERY types on disallowed channels still throw
`MessageTypeViolationException` ‚Äî no change for those.

## Why this is non-obvious

The `allowedTypes` field on a channel suggests symmetric enforcement for all
types. The validate/advisory split is only visible in the `StoredMessageTypePolicy`
source code. Tests written against the channel contract expectation (any
disallowed type is rejected) will pass in unit tests with mock enforcement
and fail in integration tests with the real qhorus policy.
