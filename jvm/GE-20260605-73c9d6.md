## CommitmentState.DECLINED not CANCELLED for DECLINE speech-act commitment outcome

### Symptom

Compile error when asserting commitment state after a DECLINE message:

```
error: cannot find symbol
    assertThat(c.state).isEqualTo(CommitmentState.CANCELLED);
                                                  ^
  symbol:   variable CANCELLED
```

### Context

Any `@QuarkusTest` asserting the final commitment state after a `ReviewerChannelBackend` (or any Qhorus channel backend) dispatches a `DECLINE` message:

```java
assertThat(commitmentStore.findByCorrelationId(correlationId))
    .hasValueSatisfying(c -> assertThat(c.state).isEqualTo(CommitmentState.CANCELLED));  // âťŚ WRONG
```

### Root cause

The `CommitmentState` enum does not have a `CANCELLED` variant. The full set of values is:

```java
OPEN, ACKNOWLEDGED, FULFILLED, DECLINED, FAILED, DELEGATED, EXPIRED
```

When a `MessageType.DECLINE` message is dispatched for a given `correlationId`, Qhorus resolves the outstanding commitment to `CommitmentState.DECLINED` â€” matching the speech-act message type exactly. The natural English word for this transition is "cancelled" (the obligation is cancelled by the decliner), but the enum models the action from the speech-act perspective: the responding party *declined*, not *cancelled*.

### Fix

Use `CommitmentState.DECLINED`:

```java
assertThat(commitmentStore.findByCorrelationId(correlationId))
    .hasValueSatisfying(c -> assertThat(c.state).isEqualTo(CommitmentState.DECLINED));  // âś…
```

The mapping from message type to commitment terminal state:
- `RESPONSE` â†’ `FULFILLED`
- `DECLINE` â†’ `DECLINED`
- Other terminal states (`FAILED`, `EXPIRED`, `DELEGATED`) are not produced by normal message dispatch

### Why this is non-obvious

An obligation is intuitively "cancelled" when the obligor refuses to fulfil it. The developer writes `CANCELLED` or `REJECTED` (both seem natural), but Qhorus uses `DECLINED` to mirror the corresponding speech-act message type (`MessageType.DECLINE`). The naming alignment makes sense retrospectively but produces a compile error with a confusing message on first encounter.
