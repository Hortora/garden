When a case completes via goal satisfaction in tests using `casehub-engine-persistence-memory`, `CaseLifecycleEvent(eventType="CaseCompleted")` is not reliably delivered to CDI `@ObservesAsync` observers. The case DOES reach `CaseStatus.COMPLETED` (visible via `caseInstanceRepository.findByUuid().getState()`), making this especially misleading — the repository poll works, but the CDI event never arrives within the test assertion window.

### Symptom

`await().atMost(3, SECONDS).untilAsserted(() -> assertThat(ae.escalationStatus).isEqualTo(COMPLETED))` times out with `ConditionTimeoutException` even though the engine case is in `CaseStatus.COMPLETED`.

### Root cause

Two async hops separate goal satisfaction from `CaseCompleted`:

1. `GoalReachedEventHandler` fires `GoalReached` via `lifecycleEvents.fireAsync()` → chains `.evaluateCompletion()`
2. `evaluateCompletion` publishes to a Vert.x event bus address (`eventBus.publish(CASE_STATUS_CHANGED, ...)`)
3. A separate `CaseStatusChangedHandler` fires `CaseCompleted` via `lifecycleEvents.fireAsync()`

Steps 2–3 are a Vert.x event bus hop on top of the first `fireAsync`. In tests with a single-threaded Vert.x event loop and in-memory stores, the second CDI event either doesn't fire or arrives after the test assertion window closes.

`GoalReached` fires at Step 1 — before the second hop — and is reliably delivered to `@ObservesAsync` observers in tests.

### Fix

Accept both `"GoalReached"` and `"CaseCompleted"` event types in observers. Since `GoalReached` fires once per goal (not once per case completion), add an idempotency guard on the write-back. Extract the write to a dedicated `@ApplicationScoped` bean with `@Transactional(REQUIRES_NEW)` so it commits atomically and can be mocked in unit tests:

```java
// Observer — accept both event types
if (!"GoalReached".equals(event.eventType()) && !"CaseCompleted".equals(event.eventType())) return;
// ...resolve aeId...
statusUpdater.markCompleted(aeId); // idempotent — returns false if already COMPLETED
```

```java
@ApplicationScoped
public class AeStatusUpdater {

    @Transactional(TxType.REQUIRES_NEW)
    public boolean markCompleted(UUID aeId) {
        AdverseEvent ae = AdverseEvent.findById(aeId);
        if (ae == null || ae.escalationStatus == AeEscalationStatus.COMPLETED) return false;
        ae.escalationStatus = AeEscalationStatus.COMPLETED;
        return true;
    }
}
```

`REQUIRES_NEW` ensures the status write commits even if the outer observer transaction rolls back (e.g. ledger write failure).

### Alternatives considered

- `CaseCompleted`-only filter: observer never fires in `@QuarkusTest` with in-memory engine. Await always times out.
- Inline `AdverseEvent.findById()` in the listener: works in integration tests but makes the listener untestable with plain Mockito (Panache static calls require Quarkus container). Extracting to a CDI bean solves both the mockability and the `REQUIRES_NEW` isolation.

### Why this is non-obvious

The case genuinely reaches `CaseStatus.COMPLETED` — you can see it via `caseInstanceRepository.findByUuid().getState()`. The symptom looks like a timing race on the status update, not a missing CDI event. And in production with a full Vert.x cluster, `CaseCompleted` fires correctly — so the gap is specific to the test's in-memory engine setup.
