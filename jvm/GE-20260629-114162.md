### What exists

Two undocumented aspects of BM25 support in the Qdrant Java gRPC client 1.18.1:

**1. `VectorFactory.vector(Document)` works on self-hosted Qdrant — not cloud-only**

The Javadoc says "Creates a vector from a document for cloud inference." This is misleading. `Document.newBuilder().setText(text).setModel("qdrant/bm25").build()` wrapped in `VectorFactory.vector(Document)` works on any self-hosted Qdrant v1.12+. The "cloud inference" label describes the mechanism (server-side model execution), not a deployment requirement. The Document proto carries text + model name; the server tokenizes and generates BM25 sparse vectors internally.

Same applies to `QueryFactory.nearest(Document)` for query-time BM25 inference.

**2. `SparseVectorParams.setModifier(Modifier.Idf)` is required for BM25 scoring**

Without `Modifier.Idf` on the sparse vector config, similarity uses raw dot product instead of IDF-weighted BM25. The `Modifier` enum and its `Idf` value exist in `io.qdrant.client.grpc.Collections` but the Javadoc does not explain what they do or that `Idf` is required for BM25.

```java
// Collection creation — BM25 sparse vector
SparseVectorConfig.newBuilder()
    .putMap("bm25", SparseVectorParams.newBuilder()
        .setModifier(Modifier.Idf)  // REQUIRED — undocumented in Javadoc
        .build())
    .build();

// Upsert — server-side BM25 inference
VectorFactory.vector(
    Document.newBuilder()
        .setText("ConcurrentHashMap is thread-safe")
        .setModel("qdrant/bm25")  // built-in, works self-hosted
        .build());

// Query — server-side BM25 inference
QueryFactory.nearest(
    Document.newBuilder()
        .setText("HashMap")
        .setModel("qdrant/bm25")
        .build());
```

### Three-way RRF composition — BM25 as a prefetch leg

`QueryFactory.nearest(Document)` works as a `PrefetchQuery` in Qdrant's server-side RRF, combining with dense and SPLADE sparse prefetches in a single gRPC call. This eliminates the need for Java-side RRF fusion — all three retrieval legs fuse inside Qdrant:

```java
// Three-leg RRF: dense + sparse + BM25 — single server-side operation
PrefetchQuery densePrefetch = PrefetchQuery.newBuilder()
    .setQuery(QueryFactory.nearest(denseEmbedding.vectorAsList()))
    .setUsing("dense").setLimit(40).build();

PrefetchQuery sparsePrefetch = PrefetchQuery.newBuilder()
    .setQuery(QueryFactory.nearest(sparseValues, sparseIndices))
    .setUsing("sparse").setLimit(40).build();

PrefetchQuery bm25Prefetch = PrefetchQuery.newBuilder()
    .setQuery(QueryFactory.nearest(
        Document.newBuilder()
            .setText(expandedQuery)  // CamelCase-expanded for code terms
            .setModel("qdrant/bm25")
            .build()))
    .setUsing("bm25").setLimit(40).build();

QueryPoints query = QueryPoints.newBuilder()
    .setCollectionName(collection)
    .addPrefetch(densePrefetch)
    .addPrefetch(sparsePrefetch)
    .addPrefetch(bm25Prefetch)
    .setQuery(QueryFactory.rrf(Rrf.newBuilder().setK(60).build()))
    .setLimit(maxResults)
    .setWithPayload(WithPayloadSelectorFactory.enable(true))
    .build();
```

The collection must have all three named vector spaces configured: dense (float), sparse (SPLADE), and bm25 (sparse with `Modifier.Idf`). Each prefetch runs independently and RRF merges by rank position.

### Why this is non-obvious

The Qdrant REST API documentation clearly describes BM25 sparse vectors, `Modifier.Idf`, and Document-based inference. The Java client wraps the same gRPC API but its Javadoc uses "cloud inference" terminology and doesn't document the `Modifier` enum's purpose. A developer working only from Java Javadoc would not discover that `VectorFactory.vector(Document)` enables server-side BM25 on self-hosted instances, or that `Modifier.Idf` is the switch that turns raw sparse similarity into BM25 scoring.
