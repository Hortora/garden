## Symptom

Production build (`mvn verify`, which runs `quarkus:build`) fails with 30+ CDI deployment problems:

```
Found 33 deployment problems:
[1] Unsatisfied dependency for type io.casehub.qhorus.runtime.channel.ReactiveChannelService
[2] Unsatisfied dependency for type io.casehub.qhorus.runtime.store.ReactiveMessageStore
[3] Unsatisfied dependency for type io.casehub.engine.spi.EventLogRepository
...
```

But `@QuarkusTest` integration tests pass cleanly.

## Context

A Quarkus application configured as JDBC-only (no reactive driver, no reactive URL) that has transitive Maven dependencies shipping reactive-capable CDI beans. The reactive implementations are `@Vetoed` when no reactive datasource is present â€” but the beans that *inject* those reactive services are not guarded, leaving their injection points unsatisfied.

The failure only appears in the production build, not in `@QuarkusTest`, because the test `application.properties` explicitly sets `quarkus.datasource.reactive=false` while the production properties do not.

## What was tried

- Adding `quarkus.datasource.reactive=false` to test `application.properties` only (fixes tests but not production build)
- Adding `quarkus.arc.exclude-types` for the reactive service classes (partially effective but requires listing every reactive class)
- Looking for `casehub.qhorus.reactive.enabled=false` in production (this key was removed from the qhorus config model and causes `ConfigValidationException`)

## Root cause

Quarkus indexes reactive beans from transitive JARs during CDI augmentation regardless of whether a reactive URL is configured on the datasource. The `@Vetoed` annotation on `ReactiveJpaMessageStore` suppresses the store implementation â€” but `QhorusDashboardService` and similar beans that inject `ReactiveMessageStore` are still activated and their injection points remain unsatisfied.

## Fix

Add explicit reactive suppression to the production `application.properties` for every datasource:

```properties
# Suppress reactive datasource â€” JDBC-only; prevents reactive bean CDI failures
quarkus.datasource.reactive=false
quarkus.datasource.qhorus.reactive=false   # repeat per named datasource
```

This explicitly tells Quarkus not to attempt reactive wiring, which cascades to suppress the reactive-dependent service beans.

**Do not use `%test.` profile prefix** â€” this fix is needed in production, not just tests.

## Why this is non-obvious

The symptom (unsatisfied dependency for `ReactiveChannelService`) looks like a missing bean, not a datasource configuration problem. A developer would search for `ReactiveChannelService` implementations, check CDI alternatives, or inspect the dependency graph â€” not look at datasource reactive configuration.

The difference between test and production is invisible unless you explicitly compare the two `application.properties` files looking for reactive-related settings.
