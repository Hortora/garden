## Symptom

Flyway migrations from `casehub-ledger` fail silently in native image builds. Ledger tables
(`ledger_entry`, `ledger_attestation`, etc.) are absent after startup. No build-time error
is emitted — the failure only surfaces when application code tries to write a ledger entry.

## Context

Occurs when a Quarkus extension co-deploys `casehub-ledger` and builds a native image. The
consuming extension's deployment processor registers only its own migration SQL files, not
the ledger's. The `db/ledger/migration/*.sql` files are excluded from the native image
binary and Flyway can't find them at startup.

## Root Cause

`LedgerProcessor` in `casehub-ledger-deployment` has **no** `@BuildStep` that produces a
`NativeImageResourcePatternsBuildItem`. Each extension must register its own SQL migration
files for native image inclusion, and `casehub-ledger` currently delegates that responsibility
to its consumers rather than self-registering.

Quarkus registers native image resources at build time — there is no runtime fallback for
SQL files absent from the binary.

## Fix

In the consuming extension's deployment processor (e.g. `QhorusProcessor`), add a second
`includeGlob` for the ledger path alongside the extension's own glob:

```java
@BuildStep
NativeImageResourcePatternsBuildItem registerMigrationResources() {
    return NativeImageResourcePatternsBuildItem.builder()
            .includeGlob("db/qhorus/migration/*.sql")   // own migrations
            .includeGlob("db/ledger/migration/*.sql")   // casehub-ledger (doesn't self-register)
            .build();
}
```

Both globs are required. If only the first is present, ledger migrations are silently absent.

## Why This Is Non-Obvious

Extension consumers naturally assume each extension self-registers its own native image
resources — this is the expected Quarkus contract. There is no build warning when SQL files
are excluded from the binary. The failure only manifests at JVM startup when Flyway runs its
migration scan and finds nothing. Debugging typically starts at the application layer ("why
is there no ledger_entry table?") rather than at the native build layer.

Verified in `casehub-qhorus` #219 when adding `QhorusProcessor.registerMigrationResources()`.
