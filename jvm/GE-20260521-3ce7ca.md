## Symptom

Test assertions calling `workItemStore.scanAll()` return empty even though the handler log confirms `WorkItemService.create()` ran successfully. `planItem.getStatus()` is `RUNNING` (proving the handler completed), but the store shows nothing.

## Context

Two `WorkItemStore` beans on the test classpath:
- `JpaWorkItemStore @ApplicationScoped` — production bean from `casehub-work`
- `InMemoryWorkItemStore @Alternative @Priority(1)` — test replacement from `casehub-work-testing`

CDI 4.x spec says `@Priority` on an alternative activates it globally and it should win over non-alternatives. In Quarkus ARC 3.32.2 this does not hold when the alternative comes from an external dependency JAR.

## What was tried

- Adding `InMemoryWorkItemStore` to `quarkus.arc.selected-alternatives` alone → "Ambiguous dependencies" deployment failure (both beans active simultaneously)
- `quarkus.arc.exclude-types=io.casehub.work.runtime.repository.jpa.JpaWorkItemStore` alone → no effect; `JpaWorkItemStore` still wins
- `@Inject InMemoryWorkItemStore workItemStore` (concrete type) in the test → bypasses CDI alternative selection; test clears the right store but `WorkItemService` still writes to JPA (H2), so `scanAll()` returns empty

## Root Cause

`@Alternative @Priority(1)` from a dependency JAR is not treated as overriding a non-alternative `@ApplicationScoped` bean by Quarkus ARC. CDI spec §2.6.2 requires `@Priority` alternatives to take precedence, but ARC's resolution appears to treat both as equally active candidates when neither is the local application class. The non-alternative wins or ambiguity is raised.

## Fix

Combine both in `src/test/resources/application.properties`:

```properties
# Remove the JPA impl from CDI discovery entirely
quarkus.arc.exclude-types=io.casehub.work.runtime.repository.jpa.JpaWorkItemStore

# Explicitly select the in-memory replacement
quarkus.arc.selected-alternatives=\
  io.casehub.work.testing.InMemoryWorkItemStore,\
  ...other alternatives...
```

With `JpaWorkItemStore` excluded, `InMemoryWorkItemStore` is the only active `WorkItemStore` bean. Explicit `selected-alternatives` listing then activates it. Both `WorkItemService` (via `@Inject WorkItemStore`) and the test (via `@Inject InMemoryWorkItemStore`) now reference the same singleton, and `scanAll()` finds what `create()` stored.

## Why this is non-obvious

CDI 4.x spec (and Quarkus docs on alternatives) state that `@Alternative @Priority(n)` globally activates and prioritises the alternative. A developer familiar with CDI 4.x would expect `@Priority(1)` to simply work. The failure is silent — no warning is emitted, `JpaWorkItemStore` just wins quietly. The symptom (empty `scanAll()`) points toward a different root cause (session caching, test isolation) and sends debugging in the wrong direction.
