### Symptom

29 CDI `DeploymentException` errors at Quarkus augmentation time — `Ambiguous dependencies` for every qhorus store type (`InstanceStore`, `ChannelStore`, `MessageStore`, `CommitmentStore`, `DataStore`, `DeliveryCursorStore`, `WatchdogStore`, `ChannelBindingStore`) plus their `@CrossTenant` variants.

### Context

Consumer app (casehub-aml) has three qhorus modules on the classpath:
- `qhorus-runtime` — JPA stores (`io.casehub.qhorus.runtime.store.jpa.*`)
- `qhorus-testing` — in-memory test stores (`io.casehub.qhorus.testing.*`)
- `qhorus-persistence-memory` — standalone in-memory stores (`io.casehub.qhorus.persistence.memory.*`)

All three provide implementations at `@ApplicationScoped @Default` with no `@Alternative` or `@Priority` disambiguation.

### What was tried (didn't work)

1. Rebuilding qhorus locally (`mvn install -DskipTests`) — same result; the ambiguity is in the source, not a bad build artifact.
2. Excluding only the 12 blocking `InMemory*Store` classes from `persistence-memory` — resolves the ambiguity but creates 2 new `UnsatisfiedResolutionException` errors.

### Root cause

The new `qhorus-persistence-memory` module added `InMemory*Store` beans in `io.casehub.qhorus.persistence.memory.*` at `@ApplicationScoped @Default` — same qualifiers as both the JPA stores and the testing module stores. No priority hierarchy was established.

The reactive wrappers (`InMemoryReactiveMessageStore`, `InMemoryReactiveChannelStore`) inject the blocking stores **by concrete type**, not by interface:

```java
// InMemoryReactiveMessageStore
@Inject InMemoryMessageStore blocking;  // concrete type, not MessageStore interface
```

This means excluding the blocking stores without also excluding the reactive wrappers creates `UnsatisfiedResolutionException`.

### Solution 1 — per-class exclude-types

**Approach:** Exclude all persistence-memory stores individually via `quarkus.arc.exclude-types`
**Pros:** No Maven dependency changes; works when persistence-memory is a transitive dep you can't control
**Cons:** Must list every class (14+); must include reactive wrappers too (they inject blocking stores by concrete type); breaks if qhorus-testing also injects persistence-memory classes by concrete type

```properties
quarkus.arc.exclude-types=\
  io.casehub.qhorus.persistence.memory.InMemoryChannelBindingStore,\
  io.casehub.qhorus.persistence.memory.InMemoryChannelStore,\
  io.casehub.qhorus.persistence.memory.InMemoryCommitmentStore,\
  io.casehub.qhorus.persistence.memory.InMemoryCrossTenantChannelStore,\
  io.casehub.qhorus.persistence.memory.InMemoryCrossTenantCommitmentStore,\
  io.casehub.qhorus.persistence.memory.InMemoryCrossTenantMessageStore,\
  io.casehub.qhorus.persistence.memory.InMemoryCrossTenantWatchdogStore,\
  io.casehub.qhorus.persistence.memory.InMemoryDataStore,\
  io.casehub.qhorus.persistence.memory.InMemoryDeliveryCursorStore,\
  io.casehub.qhorus.persistence.memory.InMemoryInstanceStore,\
  io.casehub.qhorus.persistence.memory.InMemoryMessageStore,\
  io.casehub.qhorus.persistence.memory.InMemoryWatchdogStore,\
  io.casehub.qhorus.persistence.memory.InMemoryReactiveMessageStore,\
  io.casehub.qhorus.persistence.memory.InMemoryReactiveChannelStore
```

### Solution 2 — Maven dependency exclusion (recommended)

**Approach:** Exclude `casehub-qhorus-persistence-memory` from the `casehub-qhorus-testing` dependency in the consumer `pom.xml`
**Pros:** One exclusion rule; no `quarkus.arc.exclude-types` needed; eliminates the ambiguity at the dependency level before CDI even sees the beans
**Cons:** Only works when `qhorus-testing` is a direct dependency (not transitive)

```xml
<dependency>
  <groupId>io.casehub</groupId>
  <artifactId>casehub-qhorus-testing</artifactId>
  <scope>test</scope>
  <exclusions>
    <exclusion>
      <groupId>io.casehub</groupId>
      <artifactId>casehub-qhorus-persistence-memory</artifactId>
    </exclusion>
  </exclusions>
</dependency>
```

Verified in casehub-life: the testing jar's stores are self-contained — they do not delegate to persistence-memory stores internally. With the exclusion, only the testing jar's `@Alternative @Priority(1)` stores are discovered, and they win over the JPA stores cleanly.

### Why this is non-obvious

The initial symptom (29 ambiguous dependencies) points at the blocking stores. Excluding them seems like the complete fix. But the reactive wrappers have a direct dependency on the concrete blocking store class — not the interface — so excluding the blocking stores creates a new unsatisfied dependency that only surfaces on the next build. Two rounds of "fix one CDI error, get a different one" before reaching the real solution.

### Alternatives considered

- **Rebuilding qhorus** — tried first, doesn't help; the ambiguity is intentional in the source
- **Adding `@Alternative @Priority` to qhorus source** — correct long-term fix but requires a qhorus PR
- **Excluding by package prefix** — Quarkus `exclude-types` doesn't support wildcards; must list each class
