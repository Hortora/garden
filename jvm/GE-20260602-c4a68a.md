## The Technique

Aggregators or services that use `ManagedExecutor` for CDI-context-safe parallel execution can expose a second package-private constructor for plain JUnit testing. The test constructor accepts contributors directly, sets executor to null, and switches to sequential execution:

```java
@ApplicationScoped
public class ActorStateAggregator {

    private final List<ActorStateContributor> contributors;
    private final ManagedExecutor executor;

    // CDI production constructor — parallel execution with CDI context propagation
    @Inject
    public ActorStateAggregator(
            @Any Instance<ActorStateContributor> contributors,
            ManagedExecutor executor) {
        this.contributors = contributors.stream().toList();
        this.executor = executor;
    }

    // Test constructor — sequential execution, no executor needed
    ActorStateAggregator(List<ActorStateContributor> contributors) {
        this.contributors = contributors;
        this.executor = null;
    }

    public ActorStateResponse forActor(String actorId) {
        if (executor != null) {
            // parallel via ManagedExecutor
        } else {
            // sequential for tests
            contributors.forEach(c -> runContributor(c, actorId, accumulator));
        }
    }
}
```

Test uses lambdas as contributor stubs — no CDI bootstrap required:
```java
var agg = new ActorStateAggregator(List.of(
    (id, acc) -> acc.trustScore(0.82),   // ledger stub
    (id, acc) -> { throw new RuntimeException("timeout"); }  // work stub
));
var resp = agg.forActor("agent-x");
assertThat(resp.sources()).doesNotContain("work");
```

## Why Non-obvious

Most developers reach for `@QuarkusTest + @InjectMock` to test CDI beans, which starts the full CDI container. For testing the aggregation logic itself (which contributors are called, how failures are isolated, what the response shape looks like), a plain JUnit test with the test constructor is faster and simpler. The key insight is that the dual-constructor pattern makes `executor=null` a valid sentinel that switches the implementation to a simpler sequential path — the behaviour under test (aggregation logic) is identical.

## Constraints

The test constructor must be package-private to prevent callers from bypassing CDI in production. The `executor=null` sentinel only works when sequential execution is semantically equivalent to parallel (no ordering dependencies between contributors). This pattern is distinct from GE-20260529-c1e783 (which uses `Consumer<T>` instead of `Event<T>` for CDI event firing) — here the bypass is `ManagedExecutor` rather than CDI events.
