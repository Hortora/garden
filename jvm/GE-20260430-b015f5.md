## Symptom

An MCP tool annotated with `@Tool` is not discoverable at runtime — it doesn't appear in the tool list served by the MCP endpoint. No error is thrown during startup. Other `@Tool` methods in the same class work correctly.

## Context

`QhorusMcpTools` had `sendMessage(...)` annotated with `@Tool`, plus 2–3 additional public overloads of `sendMessage` without `@Tool`. The main annotated method was not discovered. Similarly affected: `wait_for_reply`, `share_artefact`, `set_channel_admins`, `set_channel_rate_limits`, `set_channel_writers` — all had public overloads.

## Root Cause (Hypothesised)

quarkus-mcp-server uses Jandex to scan for `@Tool` annotations. When a class has multiple public methods with the same name, Jandex's CDI proxy generation or the MCP scanner appears unable to unambiguously resolve the annotated overload and silently drops all variants. No warning is emitted.

This behaviour is not documented in quarkus-mcp-server's README or Javadoc.

## Fix

Make all non-`@Tool` overloads of the same method name **package-private** (remove `public`). This removes the ambiguity — only the `@Tool`-annotated variant is public, and the scanner finds it unambiguously.

```java
// Before — ALL three are public, scanner drops all:
@Tool(name = "send_message", ...)
public MessageResult sendMessage(String channelName, String sender, String type, ...) { ... }

public MessageResult sendMessage(String channelName, String sender, MessageType type, ...) { ... }  // overload

// After — overloads are package-private, scanner finds the @Tool method:
@Tool(name = "send_message", ...)
public MessageResult sendMessage(String channelName, String sender, String type, ...) { ... }

MessageResult sendMessage(String channelName, String sender, MessageType type, ...) { ... }  // package-private
```

Tracked as casehubio/qhorus#129.

## Why This Is Non-Obvious

The `@Tool` annotation is on the correct method. No error is thrown. The tool simply isn't there at runtime. Nothing in the quarkus-mcp-server documentation mentions overload behaviour. The connection between "has public overloads" and "tool not discovered" is not findable without reading the Jandex scanner source.
