### Symptom

`Instance<ChatModel>` iteration returns only the `@Alternative` bean. The `@DefaultBean` ChatModel (registered by quarkus-langchain4j via `SyntheticBeanBuildItem.defaultBean()`) is completely absent from the container ‚Äî not deprioritized, gone.

### Context

A platform module needed to provide a `ChatModel` adapter (`@Alternative @Priority(10)`) that wraps an `AgentProvider` as a `ChatModel`. The same deployment also has quarkus-langchain4j providing a raw `ChatModel` as `@DefaultBean`. A second bean uses `Instance<ChatModel>` to find the raw model ‚Äî but after filtering out the adapter, nothing remains.

### What was tried (didn't work)

`@Alternative @Priority(10) @ApplicationScoped` on the adapter bean. Expected: both the adapter and the raw `@DefaultBean` ChatModel visible in `Instance<ChatModel>`, with `@Priority` disambiguating for direct injection. Actual: the raw `@DefaultBean` ChatModel vanishes entirely.

### Root cause

ArC's `@DefaultBean` semantics state: *"the bean will only be used if no other bean of this type is configured."* An `@Alternative` with `@Priority` counts as "another bean" ‚Äî it suppresses the `@DefaultBean` entirely, removing it from the container. This is not priority ordering ‚Äî it is total suppression.

### Fix

Use `@DefaultBean @Priority(10)` instead of `@Alternative @Priority(10)`.

```java
// WRONG ‚Äî suppresses quarkus-langchain4j's @DefaultBean ChatModel
@Alternative
@Priority(10)
@ApplicationScoped
public class AgentProviderChatModel implements ChatModel { ... }

// RIGHT ‚Äî coexists with other @DefaultBean ChatModel beans
@DefaultBean
@Priority(10)
@ApplicationScoped
public class AgentProviderChatModel implements ChatModel { ... }
```

Two `@DefaultBean` beans coexist ‚Äî neither suppresses the other. `@Priority(10)` wins for direct `@Inject ChatModel` injection. `Instance<ChatModel>` sees both, enabling runtime filtering.

A consumer-provided non-`@DefaultBean` ChatModel properly overrides both defaults ‚Äî the `@DefaultBean` suppression rule works as expected when a "real" bean is present.

### Why this is non-obvious

Developers expect `@Alternative` to win the injection point but leave other beans in the container for `Instance` lookup. The mental model is "priority ordering among all candidates" when the reality is "total removal of @DefaultBean beans when any non-default bean exists."

The ArC javadoc confirms: *"Default beans can optionally declare @Priority. Priority value is used for bean ordering and during typesafe resolution to disambiguate multiple matching default beans."* This `@DefaultBean @Priority` pattern is the correct way to have two platform-provided defaults with deterministic selection.

### Why this fix

`@DefaultBean @Priority` is architecturally correct ‚Äî both beans are "defaults" that yield to consumer-provided beans. `@Alternative` implies the bean is an override, which is semantically wrong when both beans are platform infrastructure that should coexist.

### Alternatives considered

- **CDI qualifiers** (`@Named`, custom qualifier) ‚Äî would work but forces callers to qualify injection points. Breaks the "just add to classpath" activation model.
- **Conditional activation** (`@IfBuildProperty`) ‚Äî too rigid; the coexistence scenario is legitimate in deployments using both adapters.
