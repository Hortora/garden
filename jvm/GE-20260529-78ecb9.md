## Symptom

A `@QuarkusTest` method directly injects a CDI bean and calls `.await().indefinitely()` on a reactive method. The method internally calls `Panache.withTransaction()`. The test fails with:

```
java.lang.IllegalStateException: No current Vertx context found
    at io.quarkus.hibernate.reactive.panache.common.runtime.SessionOperations.vertxContext(SessionOperations.java:452)
    at io.quarkus.hibernate.reactive.panache.common.runtime.SessionOperations.withTransaction(SessionOperations.java:137)
    at io.quarkus.hibernate.reactive.panache.Panache.withTransaction(Panache.java:84)
    at io.casehub.qhorus.runtime.channel.ReactiveChannelService.create(...)
```

The test passes compilation and starts. Only at runtime does it fail.

## Context

- `@QuarkusTest` with CDI-only injection (no HTTP, no `@TestSecurity`)
- The injected bean calls `ReactiveChannelService.create()` or similar, which wraps its body in `Panache.withTransaction()`
- The JUnit test method executes on the JUnit platform thread — not a Vert.x event loop thread

## Root cause

`Panache.withTransaction()` calls `Vertx.currentContext()` to obtain the active Vert.x duplicated context. On the JUnit platform thread, `Vertx.currentContext()` returns `null`. Even though `.await().indefinitely()` triggers reactive subscription, the subscription begins on the JUnit thread, and `Panache.withTransaction()` executes on that same thread before any Vert.x context is established.

## Fix

Add `@RunOnVertxContext` to the `@QuarkusTest` class, and replace `.await()` calls with `UniAsserter` parameter injection:

```java
@QuarkusTest
@TestProfile(ReactivePostgresTestProfile.class)
@RunOnVertxContext                       // ← runs each test method on the event loop
class MyReactivePanacheIT {

    @Inject MyService service;

    @Test
    void myTest(UniAsserter asserter) {  // ← UniAsserter replaces .await()
        asserter.assertThat(
            () -> service.createSomething("foo"),   // Uni<Result>
            result -> {
                assertThat(result).isNotNull();
                assertThat(result.name()).isEqualTo("foo");
            });
    }
}
```

`@RunOnVertxContext` runs each `@Test` method on the Vert.x event loop. `UniAsserter` takes a `Supplier<Uni<T>>` and a `Consumer<T>` — the Uni pipeline runs on the event loop, and `withTransaction()` finds a valid context.

**Critical:** `@BeforeEach` still runs on the JUnit thread even when `@RunOnVertxContext` is on the class. Any reactive setup that calls Panache (e.g. `openChannel()`) must be done inside the `UniAsserter` chain in the test method itself — not in `@BeforeEach`:

```java
// Wrong — @BeforeEach runs on JUnit thread, withTransaction() fails
@BeforeEach
void setUp() {
    workChannel = provider.openChannel(caseId, "work").await().indefinitely(); // ← FAILS
}

// Correct — setup inside the UniAsserter chain, which runs on event loop
@Test
void myTest(UniAsserter asserter) {
    UUID caseId = UUID.randomUUID();
    asserter.assertThat(
        () -> provider.openChannel(caseId, "work")     // setup
                      .flatMap(ch -> provider.doSomething(ch)),  // actual operation
        result -> assertThat(result).isNotEmpty());
}
```

## Dependency

`io.quarkus:quarkus-test-vertx` must be on the test classpath (version managed by Quarkus BOM):

```xml
<dependency>
  <groupId>io.quarkus</groupId>
  <artifactId>quarkus-test-vertx</artifactId>
  <scope>test</scope>
</dependency>
```

## Why this is non-obvious

The error message ("No current Vertx context found") points to Vert.x internals, not to the test structure. A developer seeing this naturally tries reactive fixes — `runSubscriptionOn()`, emitters, worker pools — none of which establish a Vert.x context on the test thread. The solution (`@RunOnVertxContext`) is not mentioned in the Hibernate Reactive or SmallRye Mutiny testing documentation; it appears only in Quarkus's own `quarkus-test-vertx` module docs, which are rarely surfaced when debugging Panache errors.

The additional subtlety — that `@BeforeEach` is exempt from `@RunOnVertxContext` — means the common pattern of setting up state in `@BeforeEach` silently brings back the same error, looking like the fix didn't work.
