### Context
A reactive Uni chain has a side-effect in an `.invoke()` step that produces a value you need when constructing the final result in a downstream `.map()` step. Capturing with a plain local variable fails because lambdas require effectively-final locals. A `String[]` or `Object[]` workaround would technically compile but communicates intent poorly.

The complication: the chain does NOT execute on a single Vert.x event-loop thread. `ReactiveMessageService` calls `.runSubscriptionOn(Infrastructure.getDefaultWorkerPool())` before the relevant step, meaning the `.invoke()` and the downstream `.map()` may execute on different threads.

### Technique
Declare `final AtomicReference<T>` outside the chain. Write it inside `.invoke()`. Read it inside `.map()`.

```java
final AtomicReference<List<String>> advisoriesRef = new AtomicReference<>(List.of());

return channelUni
    .invoke(ch -> {
        // This may run on a worker-pool thread
        final String adv = policy.advisory(ch, type);
        if (adv != null) {
            LOG.warn(adv);
            advisoriesRef.set(List.of(adv));
        }
    })
    .flatMap(ch -> doWork(ch))
    .map(outcome -> new DispatchResult(
            ...,
            advisoriesRef.get()  // safe: happens-after the .invoke() step
    ));
```

`AtomicReference` ensures cross-thread visibility. Mutiny's sequential composition guarantees happens-before between the `.invoke()` step that writes and the `.map()` step that reads — the read cannot be reordered before the write.

### Why this is non-obvious
The natural reach is a mutable local or a `String[]` wrapper to satisfy the effectively-final constraint. Both work but `AtomicReference` is idiomatic, communicates intent (shared across thread boundaries), and is safe without additional synchronisation. Developers who know Vert.x event-loop threading often assume the whole chain is single-threaded — `runSubscriptionOn()` breaks that assumption, making `AtomicReference` necessary rather than optional.

### Constraints
Only needed when `.runSubscriptionOn()`, `.emitOn()`, or similar thread-switching operators appear before the write site in the chain. If the entire chain stays on a single Vert.x context, a `String[]` array workaround achieves the same result without `AtomicReference`, but `AtomicReference` remains the cleaner choice regardless.
