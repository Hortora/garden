## Symptom

Plain-Java test configures Flyway with `baselineOnMigrate(true)` to run against a pre-seeded H2 schema. All migrations appear to run successfully, but V1 migration is never applied — its tables are absent from the final schema.

## Context

Occurs when testing Flyway migrations against an H2 in-memory database that already has some tables (e.g. seeded manually to satisfy cross-module FK dependencies). `baselineOnMigrate(true)` is required to prevent the "found non-empty schema with no history" error.

## Root Cause

`baselineOnMigrate(true)` defaults to `baselineVersion = "1"`. Flyway treats version 1 as the "already applied" baseline — it marks V1 as applied in the schema history table without running it, then applies V2 onwards. This is the correct behaviour when baselining an existing production database at version 1, but silently wrong when you want ALL migrations (including V1) to run from scratch.

## Fix

Set `baselineVersion("0")` explicitly — any version below your first migration:

```java
Flyway.configure()
    .dataSource(JDBC_URL, "sa", "")
    .locations("classpath:db/migration/qhorus")
    .baselineOnMigrate(true)
    .baselineVersion("0")  // Must be < V1 so V1 is executed, not skipped as baseline
    .load()
    .migrate();
```

The comment is essential — a future reader will "simplify" this to `baselineOnMigrate(true)` alone and re-introduce the bug.

## Why This Is Non-Obvious

The `baselineVersion` default of `"1"` is logical for its primary use case (baselining a live database at the current schema version). In the test case — a fresh H2 database with manually seeded cross-module tables — you want all migrations to run, which requires a baseline version below V1. The Flyway error message when omitting `baselineOnMigrate` correctly says "non-empty schema with no history table", but there's no warning that V1 will be skipped once you enable it.
