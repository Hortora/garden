### What it is

When using `casehub-platform-api`'s `PreferenceProvider` to store a structured config object (e.g. a routing policy with multiple fields), use **one typed `PreferenceKey<T>` per field** instead of a single `PreferenceKey<ComplexRecord>` that would require encoding/decoding the entire record as a string.

### How it works

Each field of the config record becomes its own `PreferenceKey<T>` constant. The YAML file has one key-value pair per field at the capability scope. The provider resolves each field independently and assembles the record:

```java
// TrustRoutingPolicyKeys.java — one key per field, not one key per record
public final class TrustRoutingPolicyKeys {

    public static final PreferenceKey<DoublePreference> BLEND_FACTOR =
        new PreferenceKey<>(
            "casehubio.devtown.trust-routing",
            "blend-factor",
            DoublePreference.of(0.0),   // non-null constructor sentinel
            DoublePreference::parse);

    public static final PreferenceKey<DoublePreference> FLOOR_REVIEW_THOROUGHNESS =
        new PreferenceKey<>(
            "casehubio.devtown.trust-routing",
            "floor.review-thoroughness",
            DoublePreference.of(0.0),
            DoublePreference::parse);

    // ... one key per config field
}
```

The YAML entry keys match `namespace + "." + name` = `PreferenceKey.qualifiedName()`:

```yaml
entries:
  - scope: casehubio/devtown/trust-routing/security-review
    casehubio.devtown.trust-routing.blend-factor: "0.70"
    casehubio.devtown.trust-routing.floor.review-thoroughness: "0.60"
```

Provider assembly:

```java
@Override
public TrustRoutingPolicy forCapability(String capabilityName) {
    Preferences prefs = preferenceProvider.resolve(
        SettingsScope.of("casehubio", "devtown", "trust-routing", capabilityName));

    DoublePreference blendFactor = prefs.get(TrustRoutingPolicyKeys.BLEND_FACTOR);
    double blend = blendFactor != null ? blendFactor.value() : TrustRoutingPolicy.DEFAULT.blendFactor();

    // ... assemble from per-field lookups
    return new TrustRoutingPolicy(threshold, minObs, margin, blend, qualityFloors);
}
```

### Why this is non-obvious

The obvious approach is a single `PreferenceKey<MyRecord>` with a parser that deserializes the whole record from a string (JSON, pipe-delimited, custom format). This requires:
1. A custom serialization format
2. A complex parser that can fail at runtime
3. An opaque blob in the YAML (`"0.70|10|0.05|0.70"`)

The per-field approach stores human-readable individual values in the YAML, uses primitive parsers (`Double::parseDouble`, `Integer::parseInt`) that never fail unexpectedly, and makes each field independently overridable without touching others. The YAML is readable and the override path is obvious.

The technique also works well with `allFloorKeys()` static map pattern — add a field to the constant map and the iteration in the consumer handles it automatically, with no risk of forgetting to add a call at the use site.

### Key invariant

`MapPreferences.get(key)` returns `null` when the key is absent — it does NOT use `key.defaultValue()` as a fallback (see GE-20260530-9cdfb5). Always null-check before accessing the value, and fall back explicitly to the intended default.
