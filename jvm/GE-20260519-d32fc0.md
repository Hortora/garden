### Symptom

A REST endpoint that previously returned 200 starts returning 500 with a serialization error after migrating a service method to return `Uni<List<T>>` instead of `List<T>`. No compilation error was raised.

### Context

Endpoint returns `Response` (not `Uni<Response>`). Service method return type changed from `List<T>` to `Uni<List<T>>` during a reactive migration. The endpoint was not updated:

```java
// Before migration (fine):
return Response.ok(service.findAll()).build();

// After migration (broken — Uni passed as entity):
return Response.ok(service.findAll()).build();  // same code, now returns Uni<List<T>>
```

### Root cause

`Response.ok(Object entity)` accepts any Java object — passing a `Uni` is legal at compile time. RESTEasy Reactive only engages its reactive pipeline when the endpoint method's declared return type is `Uni<Response>` or similar reactive type. When the method returns plain `Response`, RESTEasy tries to serialize the `Uni` object itself via `MessageBodyWriter`, fails, and returns 500.

### Fix

**Option 1 — @Blocking + await (minimal change):**
```java
@GET @Blocking
public Response getItems() {
    return Response.ok(service.findAll().await().indefinitely()).build();
}
```

**Option 2 — reactive endpoint (preferred for reactive services):**
```java
@GET
public Uni<Response> getItems() {
    return service.findAll().map(list -> Response.ok(list).build());
}
```

### Why this is non-obvious

`Response.ok(Object)` is valid Java for any object, including `Uni` — the compiler and most IDEs cannot flag this as wrong. The failure only surfaces at runtime. During a large reactive migration touching many service return types, it is easy to update the service but miss updating a single endpoint that wraps the result in `Response.ok()`. The 500 error message does not mention `Uni` — it mentions serialization failure, which sends investigation in the wrong direction.

### Why this fix

`@Blocking` is the lower-risk option during migrations — no change to the endpoint contract. The reactive `Uni<Response>` approach is cleaner but requires updating callers if they test the return type.
