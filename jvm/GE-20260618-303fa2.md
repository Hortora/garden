### What's surprising

`CaseEngine.createAndSolve()` submits the plugin control loop to a `newCachedThreadPool()` and returns the CaseFile *before plugins finish executing*. Plugins that update CDI `@ApplicationScoped` singletons (e.g. `ScoutingIntelBroker.update()`) do so on the worker thread *after* the calling test's `gameTick()` has already returned. In a `@QuarkusTest` class, the worker thread update can land *after* `@BeforeEach`'s `clearLatest()` call for the *next* test, leaving stale state.

### Symptom

Test passes in isolation but fails when run after another test that calls `gameTick()` with side-effects (e.g. `scenarioRunner.run("spawn-enemy-attack")`):

```
java.lang.AssertionError:
Expecting an empty Optional but was containing value: ThreatPosition[position=Point2d[x=20.0, y=20.0]]
```

The position value comes from a previous test's plugin execution completing asynchronously during the subsequent test's setup phase.

### Root cause

`CaseEngine.createAndSolve()` returns after submitting the control loop — it does not block until plugin execution completes. `@Scheduled(concurrentExecution=SKIP)` prevents concurrent *ticks* but does not synchronise plugin completion across test boundaries. The `@BeforeEach` cleanup (`clearLatest()`, `broker.clearLatest()`) runs on the test thread; the worker thread may win the race and overwrite the cleared state before the next test's assertions run.

### Fix

Add a brief settle period in `@AfterEach` before clearing shared state, giving in-flight worker threads time to complete:

```java
@AfterEach
void tearDown() {
    intentQueue.drainAll();
    // CaseEngine worker thread may still be updating ScoutingIntelBroker after
    // gameTick() returns. Brief settle prevents stale state crossing test boundary.
    try { Thread.sleep(50); } catch (InterruptedException ignored) {}
    broker.clearLatest();
}
```

Clearing in `@AfterEach` (after settle) rather than only in `@BeforeEach` ensures the state is clean *before* the next test starts, even if the worker thread completed after `@BeforeEach` ran.

### Why this is non-obvious

The failure is test-order dependent — running the failing test in isolation always passes. The error message points to broker state, not to thread-pool timing. `gameTick()` is a synchronous call from the test's perspective, making it natural to assume all side effects are complete when it returns. The async nature of `createAndSolve()` is an implementation detail of the poc CaseEngine not visible in any test-facing API.
