### What happens

`GameLoopBenchmarkTest` is annotated `@QuarkusTest`, which forces the `%test` Quarkus profile.
In the `%test` profile, `MockEngine` loads and `EmulatedEngine` is absent (`@IfBuildProfile("emulated")`).
Any benchmark run through this class measures mock dispatch overhead — essentially CDI wiring
latency with near-zero computation. All tick timings report 0–2ms regardless of EmulatedGame
load because EmulatedGame never runs.

### Symptom

Benchmark output shows 0ms per tick for every phase; ticks/sec reports as `Infinity` if
µs precision is used. The test passes and writes a result file, so there is no failure signal.

### Root cause

`@QuarkusTest` is not profile-aware in the sense of honoring `-Dquarkus.profile=emulated`
at runtime. The profile is fixed at build time for `@QuarkusTest`, and the `%test` profile
activates `MockEngine` regardless of what profile flag is passed to Maven.

### Fix

Write a separate plain JUnit class (no `@QuarkusTest`) for EmulatedGame throughput measurement.
Instantiate `EmulatedGame` directly — it is a plain Java class with no CDI dependencies.

The new class must be in the same package as `EmulatedGame` (`io.quarkmind.sc2.emulated`)
to access package-private members (`RaceModelFactory.forRace()`, `EmulatedGame.enemy`).

```java
// Correct — no @QuarkusTest
@Tag("benchmark")
class EmulatedGameBenchmarkTest {
    @Test
    void emulatedGameThroughput() {
        EmulatedGame game = new EmulatedGame();
        TerrainGrid grid = TerrainGrid.emulatedMap();
        game.setMovementStrategy(new PathfindingMovement(grid));
        // ... wire race model + enemy AI, then measure
    }
}
```

### Why this is non-obvious

The benchmark file is clearly labeled "benchmark" and runs via `-Pbenchmark` — nothing in
the class name, Javadoc, or Maven profile configuration signals that it is silently profiled
to `%test` and therefore always runs MockEngine. A developer adding an EmulatedGame load test
to `GameLoopBenchmarkTest` would see plausible-looking (but wrong) numbers.

### Why this fix

A plain JUnit class has no Quarkus CDI overhead, can instantiate `EmulatedGame` directly
(plain Java class with no CDI dependencies), and avoids all profile constraints. It is
both simpler and more accurate than any approach involving `@QuarkusTest` infrastructure.
The package placement (`io.quarkmind.sc2.emulated`) gives access to package-private members
without any reflection or test visibility hacks.

### Alternatives considered

- Running `mvn quarkus:dev -Dquarkus.profile=emulated` and calling `AgentOrchestrator.gameTick()`
  over REST: works but measures the full stack including HTTP + CDI + scheduler overhead, not
  isolated physics throughput.
- Adding a `@Tag("benchmark")` test inside a separate `@QuarkusTest` class with the emulated
  profile: this is not possible — `@QuarkusTest` profile is fixed at build time.
