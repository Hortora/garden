### Symptom
A fault policy that removes a node via `RemoveNode` and later re-adds it via `AddNode` expects the node to return to its original position in the dependency graph. Instead, the re-added node is an isolated root with no incoming or outgoing edges. All dependency relationships are permanently lost.

### Context
`ImmutableDesiredStateGraph` uses dual adjacency maps (forward edges: node → its dependencies, reverse edges: node → its dependents). `withoutNode()` returns a new graph instance. The `withMutation(GraphMutation.RemoveNode)` method delegates to `withoutNode()`.

### What was tried (didn't work)
A `SchemaDriftFaultPolicy` that returned `RemoveNode` mutations for all downstream stages of a drifted schema, intending to re-add them on the next reconciliation cycle after the human approved the schema change. The downstream stages were permanently lost from the graph — the planner never saw them again because they were no longer in the desired graph, and even if re-added via `AddNode`, they had no dependency edges.

### Root cause
`withoutNode()` (lines 108-130 in `ImmutableDesiredStateGraph.java`) strips ALL forward and reverse edges for the removed node:

```java
// Remove forward edges from this node
Set<NodeId> deps = newForward.remove(id);
if (deps != null) {
    for (NodeId dep : deps) {
        Set<NodeId> revSet = newReverse.get(dep);
        if (revSet != null) { revSet.remove(id); }
    }
}

// Remove reverse edges to this node
Set<NodeId> dependents = newReverse.remove(id);
if (dependents != null) {
    for (NodeId dependent : dependents) {
        Set<NodeId> fwdSet = newForward.get(dependent);
        if (fwdSet != null) { fwdSet.remove(id); }
    }
}
```

This is correct graph theory — removing a node from a graph removes all its incident edges. But `withNode()` (used by `AddNode`) only inserts the node into the nodes map without restoring any edges:

```java
public DesiredStateGraph withNode(DesiredNode node) {
    var newNodes = new LinkedHashMap<>(nodes);
    newNodes.put(node.id(), node);
    return new ImmutableDesiredStateGraph(newNodes, forwardEdges, reverseEdges, version + 1);
}
```

The edges are gone and `AddNode` does not recreate them. The only way to restore the full topology is to re-invoke the `GoalCompiler` with the original blueprint.

### Fix
Do not use `RemoveNode` mutations in fault policies when the intent is to temporarily suspend a node. Instead:
- Leave the node in the desired graph
- The planner naturally ignores nodes that are `DRIFTED` (not `ABSENT`)
- Create review/escalation nodes via `AddNode` alongside the existing topology
- Resolution clears the degraded state; the planner and adapter handle convergence naturally

### Why this is non-obvious
The method name `withoutNode()` suggests "graph minus this node" — which is exactly what it does. The surprise is that `AddNode` for the same node ID does not mean "put it back where it was." In mutable graph APIs, remove + add often preserves adjacency if edges reference the node by ID. In this immutable implementation, edges are stored separately from nodes and are destroyed on removal, not reconstructed on insertion.

### Alternatives considered
- **Add a `withoutNodePreservingEdges()` method**: Would leave orphaned edges pointing to/from a non-existent node, which violates the graph's integrity invariant (no dangling edges). The current behavior is correct for graph theory; the issue is in how fault policies use it.
- **Store a "suspended" flag on DesiredNode**: Would require changing the API type. The simpler fix is to not remove nodes that should be temporarily suspended — DRIFTED status already provides the mechanism.
