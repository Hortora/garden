## Symptom

A time-windowed aggregate (`recentTimelineBreachCount()`, last-N-days counts) returns 0
or unexpectedly low values even though recent entries exist in the store. The bug is
intermittent — it appears only once an entity has accumulated more than 20 memory entries.

## Context

`MemoryQuery.forEntity(entityId, domain, tenantId)` creates a query with:
- `limit: 20` (default)
- `order: CHRONOLOGICAL` (oldest first)

When you apply the time-window filter in Java after the query returns, you are filtering
the 20 **oldest** entries — not the most recent ones. Recent entries at positions 21+
are invisible to the filter.

```java
// ❌ BUG: filters 20 oldest, misses recent entries beyond position 20
List<Memory> events = store.query(MemoryQuery.forEntity(entityId, domain, tenantId));
long breaches = events.stream()
    .filter(e -> e.createdAt().isAfter(Instant.now().minus(180, DAYS)))
    .filter(e -> "TIMELINE_BREACH".equals(e.attributes().get("outcome")))
    .count();
// Returns 0 even if there are 3 breaches in the last 6 months — they're at positions 21-23
```

## Root cause

The default query applies the 20-item limit **before** returning results to the caller.
`CHRONOLOGICAL` order means oldest-first, so the 20 results are the 20 oldest entries.
Any recent entry beyond the 20th position is simply not returned.

## Fix

Push the time-window filter into the store query using `.withSince(Instant)`. Apply
`.withLimit(N)` appropriate for the window size:

```java
// ✅ FIX: time window applied in the store, not in Java
MemoryQuery query = MemoryQuery.forEntity("site:" + siteId, ClinicalMemoryDomains.SITE, tenantId)
    .withSince(Instant.now().minus(180, ChronoUnit.DAYS))
    .withLimit(50);
List<Memory> events = store.query(query);
long breaches = events.stream()
    .filter(e -> "TIMELINE_BREACH".equals(e.attributes().get("outcome")))
    .count();
```

`withSince()` instructs the store to return only entries created after the threshold.
The limit of 50 is then applied to the already-filtered result set, not to all historical entries.

## Why this is non-obvious

The natural approach is: query → get entries → filter in Java. This works perfectly until
the entity accumulates more than `limit` total entries, at which point the results silently
become wrong. No error, no warning — just a lower count than the real number. The bug is
intermittent in development (few entries per entity in tests) and emerges only in production
once data accumulates.

Additionally, `MemoryOrder.CHRONOLOGICAL` sounds like it would return "all entries in order" —
the word "chronological" doesn't signal that it means "starting from the oldest." The combination
of CHRONOLOGICAL + limit = 20 is the trap: you always get the 20 oldest, which is rarely what
you want for a recency-based query.
