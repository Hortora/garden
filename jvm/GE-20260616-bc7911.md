### Context

When writing a JVM microbenchmark using `System.nanoTime()` for a code path that completes in
< 1ms (e.g. a physics tick, a cache lookup, an in-memory computation), storing samples as
milliseconds via `ns / 1_000_000` produces all-zero results. The report shows `0ms mean,
0ms p95, 0ms max` and `Infinity ticks/sec` — technically correct but useless as a baseline.

### Fix

Store the raw nanosecond samples. For display, convert to microseconds (`ns / 1_000`) to get
two-digit numbers in the useful range. Compute throughput from the nanosecond total to avoid
accumulated integer-truncation error across samples:

```java
long[] tickNs = new long[MEASURE_TICKS];
// ... measure loop

// Display in µs
long[] tickUs = Arrays.stream(tickNs).map(ns -> ns / 1_000).toArray();
long meanUs = mean(tickUs);   // for display only
long p95Us  = p95(tickUs);
long maxUs  = max(tickUs);

// Throughput from raw ns to avoid ~50µs accumulated rounding error
long totalNs = LongStream.of(tickNs).sum();
double tps   = totalNs > 0 ? (MEASURE_TICKS * 1_000_000_000.0 / totalNs) : Double.POSITIVE_INFINITY;
```

### Why this is non-obvious

The canonical pattern in existing benchmark templates (including JMH defaults and many internal
benchmarks) uses milliseconds because the code paths they measure take 5–500ms. Copying that
pattern to a sub-millisecond path produces silently wrong output — no error, no warning, just
zeros. The fix is one character change (`1_000_000` → `1_000`) but the symptom ("all zeros")
does not suggest a precision problem.

### Constraints

Applies to `System.nanoTime()`-based benchmarks. JMH handles this internally and should be
preferred for rigorous microbenchmarking. This technique applies when JMH is not available or
practical (e.g. `@QuarkusTest`-excluded, plain JUnit).

### Invalidation triggers

Revisit if JVM nanoTime resolution changes significantly on the target platform (currently
~100ns on macOS/Linux). The approach is correct as long as the operation being measured
takes meaningfully more than 1µs.
