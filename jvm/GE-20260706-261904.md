### Symptom
JPA entity stores a Java record as a JSON column. After adding a boolean helper method
`isDigested()` to the record, deserialization fails: the JSON now contains a `"digested": false`
field that has no matching constructor parameter.

### Context
Adding a convenience method like `public boolean isDigested() { return digestSchedule != null; }`
to a Jackson-serialized record. The method is not a stored field — it's computed from another field.
No `@JsonProperty` annotation present on the method.

### What was tried
- Investigated the JPA entity mapper for serialization bugs
- Checked if the record constructor was wrong
- Assumed the phantom field was injected by another serializer

### Root cause
Jackson's default `ObjectMapper` uses JavaBeans introspection, which treats any `boolean`
method matching `isX()` as a readable property named `x`. Java records feel immutable and
"closed" — developers don't expect JavaBeans naming conventions to add phantom properties.
On serialization, `isDigested()` produces `"digested": false`. On deserialization, Jackson
tries to pass `digested` to the record constructor, which doesn't have that parameter.

### Fix
Annotate the computed method with `@JsonIgnore`:

```java
@JsonIgnore
public boolean isDigested() {
    return digestSchedule != null;
}
```

### Why this is non-obvious
Java records create a strong mental model of "the fields are the constructor parameters,
nothing else is serialized." But Jackson follows JavaBeans conventions, not record semantics.
A `boolean isX()` method on a record looks like a property accessor to Jackson, even though
records have no mutable state. The failure only surfaces on deserialization (the extra field
in the JSON), so the serialization step appears to work fine — the bug is silent until read-back.
