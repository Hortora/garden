## Context

When retrofitting security annotations onto an existing JAX-RS application — adding `@RolesAllowed` to some endpoints, leaving others public — the public endpoints have no annotation. In a Quarkus app without a security framework, unannotated endpoints are universally accessible. Once `quarkus-oidc` or another security extension is added, the default policy may change depending on config.

## Technique

Annotate every intentionally public endpoint with `@PermitAll` rather than leaving it unannotated.

```java
// ❌ Implicit — fragile if security policy changes
@Path("/channel-context")
public class ChannelContextWindowResource { ... }

// ✅ Explicit — intent is documented and policy-change-safe
@PermitAll
@Path("/channel-context")
public class ChannelContextWindowResource { ... }
```

## Why this is non-obvious

Unannotated endpoints are accessible by default in Quarkus (`quarkus.security.deny-unauthenticated=false`). This is easy to assume will hold. But `@PermitAll` adds two concrete benefits:

1. **Survives policy hardening** — if `quarkus.security.deny-unauthenticated=true` is ever set (intentionally or by inheriting a base config), `@PermitAll` endpoints remain accessible. Unannotated endpoints would silently start returning 401.

2. **Documents intent for security auditors** — a code review or security audit can distinguish "this is explicitly public" from "someone forgot to annotate this". Unannotated is ambiguous; `@PermitAll` is a declaration.

## When to apply

Any REST resource where the absence of auth is a conscious architectural decision, not an oversight — particularly system-to-system endpoints (webhooks, internal service calls) where no user OIDC token is available at the call site.
