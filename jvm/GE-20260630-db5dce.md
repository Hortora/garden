### Symptom

Sparse retrieval quality degrades silently when replacing SPLADE with BGE-M3 — no errors, no warnings, just wrong results. The sparse weight distribution looks plausible but is mathematically incorrect.

### Context

Both SPLADE and BGE-M3 produce sparse embeddings as per-vocabulary weight maps, and both use the same downstream format (`Map<Integer, Float>` → Qdrant sparse vector). The method signatures and output shapes look identical, which invites reusing the same post-processing code.

### Root cause

The two models compute sparse weights differently:

**SPLADE:** `log(1 + relu(MLM_head_logits))` — log-saturation compresses the dynamic range of the MLM head's raw logits. The `SparseEmbedder` class in `casehub-inference-splade` implements this correctly.

**BGE-M3:** `relu(linear(hidden_state))` per-token, then max-pool to vocab indices. No log transform — the linear projection already produces calibrated weights. Applying log-saturation to BGE-M3 output compresses already-calibrated weights, distorting the relative importance of terms.

### Fix

Use distinct post-processing per model:
- SPLADE: `logSaturate()` — `log(1 + max(0, x))`, threshold at 0.01
- BGE-M3: `extractSparse()` — `max(0, x)`, threshold at 0.01 (ReLU only, no log)

The `BgeM3Embedder` in `casehub-inference-bge-m3` implements the correct BGE-M3 post-processing. Do not reuse `SparseEmbedder.logSaturate()` for BGE-M3 output.

### Why this is non-obvious

The output tensor shapes are identical (`[batch, vocab_size]`), the downstream format is identical (`Map<Integer, Float>`), and both use ReLU activation. The only difference is whether log-saturation is applied — a single function call that doesn't change the type signature. The corruption is silent because the sparse vectors are still valid (non-negative, sparse) — they just have wrong relative weights.
