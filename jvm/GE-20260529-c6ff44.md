## Symptom

A unit test asserting `Math.abs(0.8 - 0.7) <= 0.1` (or equivalently, a method that returns `false` when it should return `true` for the value `0.8` against a threshold of `0.1` centred at `0.7`) fails silently. The same test passes for `0.6` (`|0.6 - 0.7| = 0.09999...998` ≤ `0.1`).

```java
// Intended: |0.8 - 0.7| = 0.1 ≤ 0.1 → borderline
assertThat(policy.isBorderline(0.8)).isTrue();  // FAILS
```

## Context

Any code that uses a floating-point threshold comparison of the form `Math.abs(score - reference) <= margin` where `score`, `reference`, and `margin` are common decimal fractions (multiples of 0.1, 0.05, etc.).

Typical in trust scoring, routing policies, physics simulations, or any domain with configurable thresholds.

## Root Cause

`0.8` and `0.7` cannot be represented exactly in IEEE 754 binary64. Their difference is not exactly `0.1`:

```
0.8 - 0.7 = 0.10000000000000008881784197001252...
```

This is greater than `0.1` (which itself is `0.10000000000000001...` in double precision), so `<= 0.1` returns `false`.

Contrast with `0.6 - 0.7`:
```
0.7 - 0.6 = 0.09999999999999997779553950749686...
```

Which is ≤ 0.1 → returns `true`. The asymmetry is invisible until a test hits the exact bad pair.

## Fix

Never place test assertions at exact decimal boundary values. Use a value clearly inside the margin:

```java
// Instead of testing 0.8 (exact upper boundary):
assertThat(policy.isBorderline(0.79)).isTrue();  // |0.79 - 0.7| = 0.09 ≤ 0.1 → passes

// Verify just outside the margin separately:
assertThat(policy.isBorderline(0.81)).isFalse(); // |0.81 - 0.7| = 0.11 > 0.1 → passes
```

For production code: if the margin itself needs to be exact, use `BigDecimal` or add a small epsilon tolerance (e.g. `<= margin + 1e-9`).

## Why This Is Non-Obvious

Developers know floating-point is imprecise, but commonly believe that "simple" values like `0.7`, `0.8`, and `0.1` round-trip cleanly. The failure is invisible in production (the borderline exclusion still fires for most real scores), surfaces only in tests at the exact decimal boundary, and the direction of failure is not predictable without computing the IEEE 754 representations explicitly.
