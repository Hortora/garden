## Symptom

A CDI bean annotated `@ApplicationScoped` injects a `@RequestScoped` dependency
(e.g. `@Inject CurrentPrincipal principal`). When that bean is called from an
`@ObservesAsync` observer method, startup succeeds and the observer is reached, but the
very first access to the injected `@RequestScoped` bean throws:

```
jakarta.enterprise.context.ContextNotActiveException:
  RequestScoped context was not active when trying to obtain a contextual instance
```

The exception fires before any business logic — in this case, before `assertTenant()`
was reached, making a compliance-adjacent security check invisible to the caller.

## Root Cause

`@ObservesAsync` observers execute on a CDI-managed executor thread pool. By default,
Quarkus does not propagate the CDI request scope to these threads. The `@RequestScoped`
context is active on the HTTP worker thread (or whichever thread fired the event), but
the async observer runs on a different thread that has no active request context.

The injected proxy for the `@RequestScoped` bean does not fail at injection time — the
proxy is valid. It fails at the first method invocation on that thread.

## Fix

**Option 1 — Use `@Observes` (synchronous) instead of `@ObservesAsync`.**

Synchronous observers run on the calling thread where the request scope is still active.
This is the correct choice for any observer that calls a `@RequestScoped` SPI, especially
for compliance-adjacent operations where exception propagation to the caller is required.

**Option 2 — Activate the request scope manually** (if async is required for other reasons):

```java
@Inject
RequestContextController requestContextController;

void observe(@ObservesAsync MyEvent event) {
    requestContextController.activate();
    try {
        // @RequestScoped beans now accessible
    } finally {
        requestContextController.terminate();
    }
}
```

Use this only when async execution is genuinely needed — it reactivates the request scope
on the async thread. Note: this is a *new* request context, not the same context as the
caller — `@RequestScoped` beans will not share state with those on the original thread.

## Why This Is Non-Obvious

The error message (`ContextNotActiveException`) doesn't mention threads, @ObservesAsync, or
context propagation. It looks like a standard CDI wiring problem — as if the bean is not
registered or the qualifier is wrong. The full stack trace eventually reveals the async
executor, but the initial diagnostic path goes the wrong direction.

Additionally, the failure point is the injected bean access, not the observer registration —
so `@QuarkusTest` startup succeeds, the observer fires, and only the first method call on
the @RequestScoped bean fails. This can be particularly confusing for SPIs that perform
security checks internally: the observer receives the event, the call appears to proceed,
but the internal check throws before any guard logic runs, and the exception is invisible
to the event publisher (async delivery swallows it).
