### Symptom
Merkle leaf hash verification fails — `LedgerMerkleTree.leafHash(entry)` computed after reading the entry from the database doesn't match `entry.digest` stored at persist time. Hash chain appears broken.

### Context
Two save implementations exist:
1. `JpaLedgerEntryRepository.save()` (ledger runtime) — tokenises `actorId`, re-allocates `sequenceNumber`, runs enricher pipeline, THEN computes `entry.digest = LedgerMerkleTree.leafHash(entry)`. The hash reflects the post-modification field values.
2. `JpaWorkItemLedgerEntryRepository.save()` (casehub-work-ledger) — does raw `em.persist(entry)` without the pipeline. Callers must compute the hash themselves.

A caller (e.g. `LedgerEventCapture`) that computes the hash before calling save via path 1 gets a mismatch — the pipeline overwrites the hash with one computed from modified fields. A caller using path 2 must set ALL fields (including `tenancyId`) before computing the hash — any field null at hash time but set at persist time produces a mismatch on re-verification.

### What was tried (didn't work)
- Removing hash computation from the caller and relying on the save pipeline — breaks path 2 where the pipeline doesn't exist
- Computing hash before setting `tenancyId` — `canonicalBytes()` includes `tenancyId`, so the stored hash has the real value but the computed hash had empty string

### Root cause
`canonicalBytes()` is `final` on `LedgerEntry` and includes all fields in a deterministic canonical form. `domainContentBytes()` is `protected` and overridable by subclasses. The hash must be computed with ALL fields in their final state — any mutation between hash computation and persistence (or between persistence and re-verification) breaks the chain.

### Fix
For path 2 (raw persist without pipeline): set `entry.tenancyId` before computing the hash, not after:

```java
// Wrong — tenancyId is null during hash computation
entry.digest = LedgerMerkleTree.leafHash(entry);  // tenancyId = null → canonical uses ""
ledgerRepo.save(entry, currentPrincipal.tenancyId());  // save sets tenancyId

// Right — tenancyId set before hash
entry.tenancyId = currentPrincipal.tenancyId();
entry.digest = LedgerMerkleTree.leafHash(entry);  // tenancyId present → canonical uses real value
ledgerRepo.save(entry, entry.tenancyId);
```

For path 1 (pipeline save): do NOT compute the hash — the pipeline does it after all field modifications.

### Why this is non-obvious
Two repository implementations with the same interface but different hash responsibilities. Nothing in the type system or API signals which path you're on. The mismatch only surfaces during hash verification — at persist time, both paths appear to work correctly.
