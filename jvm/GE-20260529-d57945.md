### Symptom
Webhook signature verification silently fails (or signature header is treated as absent) even though the HTTP request clearly contains the expected header (visible in curl/wireshark output). `httpHeaders.getRequestHeaders().get("X-Slack-Signature")` returns null.

### Context
Implementing webhook receiver endpoints that must read signature headers from incoming HTTP requests. Different webhook platforms use different casing: Slack sends `X-Slack-Signature`, some clients normalise to `x-slack-signature`, HTTP/2 always lower-cases headers.

### Root cause
`HttpHeaders.getRequestHeaders()` returns a `MultivaluedMap<String, String>` whose keys are the header names exactly as received from the client. The map is **not** case-insensitive. HTTP headers are defined as case-insensitive (RFC 7230 §3.2), but the JAX-RS API does not normalise them. A client sending `x-slack-signature` (lower-case) will not be found by `map.get("X-Slack-Signature")`.

### Fix
Normalise all header keys to lower-case at the JAX-RS boundary before creating any data transfer object that connectors or consumers will use:

```java
private static Map<String, List<String>> lowerCaseKeys(final HttpHeaders headers) {
    return headers.getRequestHeaders().entrySet().stream()
            .collect(Collectors.toMap(
                    e -> e.getKey().toLowerCase(Locale.ROOT),
                    Map.Entry::getValue,
                    (a, b) -> { List<String> m = new ArrayList<>(a); m.addAll(b); return m; }));
}
```

Downstream code then uses:
```java
request.headers().get("x-slack-signature")  // always lower-case, always works
```

Normalise once at the boundary — never let raw JAX-RS header maps escape into connector or service logic.

### Why this is non-obvious
JAX-RS's `HttpHeaders.getHeaderString("X-Slack-Signature")` IS case-insensitive (the spec says so). But `getRequestHeaders()` returning a `MultivaluedMap` is NOT — the map itself does not perform case-insensitive lookup. A developer using the convenience method `getHeaderString()` would not hit this problem; a developer building a normalised record from `getRequestHeaders()` silently gets a case-sensitive map.

### Alternatives considered
- Using `httpHeaders.getHeaderString("x-slack-signature")` everywhere — works but requires passing the raw `HttpHeaders` object into all connector code, creating a JAX-RS dependency in pure-Java connector implementations
- Using `TreeMap(String.CASE_INSENSITIVE_ORDER)` — correct but allocates a second map; lower-casing at the boundary is simpler and avoids the intermediate structure
