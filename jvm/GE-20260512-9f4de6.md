### Symptom
`NullPointerException` on `descriptor.getRules()` — error points into the product class's
business logic, making it look like the calling code passed a null reference.

### Context
A class implements both a factory interface (`EvaluationEngine`) and the product interface
(`CompiledEngine`). A static `INSTANCE()` method creates a "null sentinel" instance to use
as the factory: `new BruteForceCompiledEngine<>(null, null)`. The shared constructor
immediately dereferences the first parameter: `this.descriptors = descriptor.getRules()...`

### What was tried (didn't work)
Inspecting `UnitDescriptor.createInstance()` and `UnitInstantiator` for a null UnitDescriptor
reference — the error trace pointed to `getRules()` which looked like a wiring problem in
the descriptor lookup, not in the factory.

### Root cause
The shared constructor is called at `INSTANCE()` time with a null sentinel, and immediately
dereferences it. The NPE occurs inside the constructor during "factory creation", not during
actual use. The error message names the field (`"descriptor" is null`) which matches the
parameter name — this misleads debugging toward the call site rather than the constructor.

### Fix
Split into two classes:
- A stateless factory (`BruteForceEngine implements EvaluationEngine`) — singleton, no state,
  `compile()` creates the product
- A compiled product (`BruteForceCompiledEngine implements CompiledEngine`) — real constructor,
  actual fields

```java
// Factory — stateless singleton
public class BruteForceEngine<CTX> implements EvaluationEngine<CTX> {
    public static final BruteForceEngine INSTANCE = new BruteForceEngine<>();
    public CompiledEngine<CTX> compile(UnitDescriptor<CTX> d, EntryPointNode rete) {
        return new BruteForceCompiledEngine<>(d, rete);
    }
}
// Product — created by compile(), real constructor
public class BruteForceCompiledEngine<CTX> implements CompiledEngine<CTX> {
    private final RuleDescriptor<CTX>[] descriptors;
    BruteForceCompiledEngine(UnitDescriptor<CTX> descriptor, EntryPointNode rete) {
        this.descriptors = descriptor.getRules().toArray(...); // safe — never null
    }
}
```

### Why this is non-obvious
The symptom (`"descriptor" is null` at `getRules()`) strongly implies the descriptor
reference is null at the call site — e.g. a lookup returning null, a missing registration,
or a wiring failure. The actual cause — the factory method calling the constructor with a
null sentinel — is not in the call stack at the point of the error. Developers familiar with
the factory pattern wouldn't expect the "empty factory" construction to immediately
dereference the argument.
