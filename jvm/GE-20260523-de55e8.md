## Symptom

A sibling CDI bean injects another `@ApplicationScoped` bean as a concrete type and accesses a package-private field on it:

```java
@Inject
InMemoryLedgerEntryRepository blocking;

// Inside a method:
blocking.entries.values().stream()...  // returns null or empty
```

The code compiles and no exception is thrown, but the field always reads as null/default, even though `blocking.save()` has populated it. Tests pass but return wrong results.

## Context

Occurred when building `InMemoryKeyRotationRepository`, which needed to read entries saved via `InMemoryLedgerEntryRepository`. The entry repo is `@ApplicationScoped`, injected into the key rotation repo via `@Inject InMemoryLedgerEntryRepository blocking`. Direct field access `blocking.entries` read the proxy's own uninitialized `entries` field, not the underlying singleton's populated map.

## Root Cause

`@ApplicationScoped` beans are wrapped in a client proxy. The proxy class extends the bean class and overrides all methods to delegate to the underlying contextual instance. However, **field reads on the proxy class access the proxy's own fields** — which are never initialized by the CDI container. Only method calls are intercepted and dispatched to the underlying singleton.

## Fix

Expose shared state via a package-private method on the owning class:

```java
// In InMemoryLedgerEntryRepository:
Collection<LedgerEntry> allEntries() {
    return entries.values();  // reads from the real singleton's field
}

// In InMemoryKeyRotationRepository:
blocking.allEntries().stream()...  // method call dispatches through proxy → real bean
```

The method call reaches the underlying singleton; `entries` is read from the real instance's field.

## Why This Is Non-Obvious

The proxy class extends the bean class, so `blocking.entries` looks syntactically valid and compiles without error. The CDI proxy pattern is commonly explained in terms of method interception — the field-access subtlety is almost never mentioned. The failure mode is also invisible: no exception, no null pointer — just an empty stream that silently produces no results, which looks like the store is empty rather than pointing at the wrong object.

### Why this fix

Exposing the field via a method is idiomatic and minimal. Alternatives like `@Produces` a `ConcurrentHashMap` or using CDI events are more complex without architectural benefit. Package-private visibility ensures the method is test-accessible within the same package while remaining an implementation detail outside it.
