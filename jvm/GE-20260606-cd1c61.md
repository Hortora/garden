### Symptom
Adding CDI fallback auto-wiring for `ToolProvider` (like other supplier types)
causes one of two failures:

1. **Only MCP ToolProvider exists:** CDI auto-wiring detects it as the single
   `@Default` bean and wires it. The MCP-specific path (`hasMcpToolBox`) also
   wires it → same bean double-wired into the agent.
2. **MCP + user ToolProvider bean:** Two `@Default` ToolProvider beans exist →
   `AmbiguousResolutionException` or auto-wiring silently skipped (both
   `isResolvable()` checks return false).

### Context
The MCP deployment processor registers its own `ToolProvider` CDI bean in the
same type space as any user-defined `ToolProvider`. Unlike `ContentRetriever`,
`ChatMemory`, etc. (which are overwrite-semantic on `AgentBuilder`),
`AgentBuilder.toolProvider()` uses append semantics — it adds to a list. This
means "fallback" auto-wiring (wire when no static supplier exists) is the wrong
concept for ToolProvider.

### What was tried (didn't work)
1. Adding `TOOL_PROVIDER` to the `CdiSupplierType` fallback enum — caused
   double-wiring when only MCP bean existed
2. Using `Instance<ToolProvider>.isResolvable()` — returned false when both
   MCP and user beans existed (ambiguous)

### Root cause
The MCP module's `ToolProvider` CDI bean has no distinguishing qualifier — it
gets `@Default` like any other bean. The CDI type space is shared. Auto-wiring
ToolProvider requires either qualifier separation (MCP bean gets a custom
qualifier) or excluding ToolProvider from the auto-wiring enum entirely.

### Fix
Exclude `ToolProvider` from the CDI fallback auto-wiring enum (`CdiSupplierType`).
ToolProviders wire via three paths:
- MCP: dedicated `hasMcpToolBox` flag + `Instance<ToolProvider>` (existing path)
- Static: `@ToolProviderSupplier` method on the agent interface
- Parameter: `@CdiBean` on a supplier method parameter (once PR2b lands)

### Why this is non-obvious
ToolProvider looks like every other supplier type — it has a `@ToolProviderSupplier`
annotation, a type, and a builder method. The auto-wiring pattern that works for
ContentRetriever/ChatMemory/RetrievalAugmentor silently fails for ToolProvider
because of the invisible MCP bean occupying the type space and the append (not
overwrite) semantics of the builder method.
