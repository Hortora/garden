### Symptom
`ide_search_text` with `project_path` returns files from ALL indexed projects
in the IDE — not just the specified project. Search results include source files
from unrelated repos that happen to be open in the same IntelliJ instance.

### Context
Multiple projects open in IntelliJ (e.g. casehub-work and casehub-engine in the
same IDE). Calling `ide_search_text` with `project_path=/path/to/work` returns
results from both work AND engine source trees.

### What was tried (didn't work)
Passing `project_path` to `ide_search_text` expecting it to scope results to that
project's source files only. The parameter name strongly implies filtering.

### Root cause
`project_path` routes the JSON-RPC call to the correct IntelliJ project for
resolution context and error messages, but the underlying search index is
IDE-wide. All indexed files from all open projects are searched regardless of
which `project_path` is specified.

### Fix
Cross-reference IntelliJ search results with filesystem `find` to verify which
files actually belong to the target repo. In multi-module Maven projects, file
paths with module prefixes (e.g. `engine-adapter/src/...`) are in the target
project; bare `src/main/java/...` paths may be from a different project's module.

### Why this is non-obvious
The `project_path` parameter exists on every `ide_*` tool and is described as
selecting the project. For tools like `ide_index_status` it genuinely scopes to
one project. The search tools break this expectation silently — no error, no
warning, just extra results that look plausible.

### Alternatives considered
- Using `filePattern` to filter — only supports filename globs (`*.java`), not path patterns
- Opening projects in separate IDE instances — eliminates the problem but loses cross-project code intelligence
