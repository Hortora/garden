### Symptom
Confidence drops to zero almost instantly when counter-indication rules fire, despite the penalty value appearing moderate (-0.3).

### Context
A pattern classification system emits confidence scores for enemy strategy archetypes in a real-time game loop running at 22.4 FPS. Counter-indication rules (e.g. "expansion detected → reduce rush confidence") apply a penalty each tick to revise stale assessments.

### What was tried (didn't work)
Additive penalty: `cumulative[arch] -= 0.3` per tick, clamped to [0, 1]. Looks reasonable in isolation — `0.8 - 0.3 = 0.5`, a sensible reduction. But at 22.4 FPS, four frames (~180ms) produce: `0.8 - (4 × 0.3) = -0.4` → clamped to 0. Any confidence zeroes in under 1 second.

### Root cause
The bug is in the interaction between magnitude and frequency. An additive penalty that looks moderate for a single application becomes catastrophic when applied at game-loop frequency. The mental model ("subtract 0.3 from 0.8") implicitly assumes one application, but the rule fires every frame.

### Fix
Multiplicative dampening factors with defined half-lives:

```java
// Instead of: cumulative[arch] -= penalty;
// Use: cumulative[arch] *= Math.pow(dampingFactor, framesElapsed);

// dampingFactor = 0.997 per frame → half-life ~10 seconds at 22.4 FPS
// dampingFactor = 0.998 per frame → half-life ~15 seconds at 22.4 FPS

// Half-life formula: ln(2) / (ln(1/dampingFactor) × FRAMES_PER_SECOND)
```

Multiplicative dampening self-limits (asymptotically approaches zero, never undershoots), has a well-defined half-life, and compounds predictably when multiple rules fire: `0.997 × 0.998 = 0.995` effective per-frame factor, half-life ~4.4s.

### Why this is non-obvious
The additive penalty value looks reasonable when reasoning about a single application. A developer reviewing `-0.3` thinks "that reduces 0.8 to 0.5, seems fine." The catastrophic failure only appears when you multiply by the frame rate — a step most developers skip because they think about the rule firing "once" when a condition is detected, not "continuously every frame while the condition holds."
