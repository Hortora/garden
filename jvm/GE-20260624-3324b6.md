### Symptom
`.function(input -> WorkerResult.of(Map.of()))` on `Worker.Builder` fails with:
```
java: reference to function is ambiguous
  both method function(WorkerFunction) and method function(Function<Map<String,Object>,WorkerResult>) match
```

### Context
`casehub-worker-api` `Worker.Builder` has two `function()` overloads:
- `function(WorkerFunction)` — takes the SPI interface
- `function(Function<Map<String,Object>, WorkerResult>)` — convenience that wraps in `WorkerFunction.Sync`

When passed a lambda, Java cannot determine which overload to target because the lambda is assignable to both `WorkerFunction` (single abstract method `execute`) and `Function<Map,WorkerResult>`.

### What was tried (didn't work)
1. Casting the lambda: `(Function<Map<String,Object>,WorkerResult>) input -> ...` — verbose and fragile
2. Hoping the compiler would prefer the more specific type — it doesn't

### Root cause
Java lambda target type inference treats both functional interfaces as equally valid targets. `WorkerFunction` has `execute(Map) -> WorkerResult` which matches `Function<Map,WorkerResult>.apply(Map) -> WorkerResult` structurally.

### Fix
Wrap the lambda explicitly with the concrete type:
```java
.function(new WorkerFunction.Sync(input -> WorkerResult.of(Map.of())))
```

### Why this is non-obvious
The old engine `Worker.Builder` had typed overloads (`function(Agent)`, `function(Workflow)`, `function(Function)`) that disambiguated naturally — no two overloads matched a lambda. After migrating to the worker-api record-based Builder with `function(WorkerFunction)` added, the ambiguity appeared in ~50 test files simultaneously.
