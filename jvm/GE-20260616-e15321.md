### Symptom

Compiler error when a JAX-RS resource class imports both `jakarta.ws.rs.Path` (for `@Path` annotations on class/methods) and `io.casehub.platform.api.path.Path` (the platform hierarchical scope type):

```
error: reference to Path is ambiguous
  both class io.casehub.platform.api.path.Path in io.casehub.platform.api.path
  and class jakarta.ws.rs.Path in jakarta.ws.rs match
```

### Context

Any JAX-RS REST resource that also needs to accept, parse, or return `io.casehub.platform.api.path.Path` as a method parameter or local variable. Common in casehub REST resources that handle scope parameters (e.g. `FilterResource`, `QueueResource`, `VocabularyResource`).

### What was tried

Adding `import io.casehub.platform.api.path.Path;` alongside the existing `import jakarta.ws.rs.Path;` â€” fails with the ambiguity error above. Java has no import aliasing.

### Root cause

Java resolves unqualified type names from imports. Both `jakarta.ws.rs.Path` and `io.casehub.platform.api.path.Path` have the simple name `Path`. Java cannot resolve which one to use when both are on the classpath and imported.

### Fix

Use the fully qualified name `io.casehub.platform.api.path.Path` in method bodies. Do not add an import for it. The JAX-RS `@Path` annotation import remains as-is:

```java
import jakarta.ws.rs.Path;  // keep â€” for @Path annotations on class/method
// do NOT import io.casehub.platform.api.path.Path

@Path("/filters")
public class FilterResource {

    @POST
    public Response create(final CreateRequest req) {
        final io.casehub.platform.api.path.Path scopePath;  // fully qualified
        try {
            scopePath = (req.scope() == null || req.scope().isBlank())
                    ? io.casehub.platform.api.path.Path.root()
                    : io.casehub.platform.api.path.Path.parse(req.scope());
        } catch (IllegalArgumentException e) {
            return Response.status(400)
                    .entity(Map.of("error", "invalid scope: " + e.getMessage())).build();
        }
        // ...
    }
}
```

This pattern is used in `VocabularyResource`, `FilterResource`, and `QueueResource` in casehub-work.

### Why this is non-obvious

Java has no import aliasing (unlike Kotlin's `import X as Y`). Developers typically try to add the import and are surprised it ambiguates the `@Path` annotation. The fix â€” fully qualified names in the method body â€” is unfamiliar to most Java developers who are used to putting all type resolution in the import block.
