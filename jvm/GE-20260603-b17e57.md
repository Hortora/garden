# Scheduled subprocess closure for wall-clock timeout in subprocess-backed reactive streams

## The technique

When a `Multi<T>` wraps a subprocess (via an SDK Flux bridge), wall-clock timeout must be implemented by scheduling subprocess termination — not by composing timeout signals into the stream. Merge-based timeout operators have wrong completion semantics (see GE-20260603-d7aa80).

```java
private final ScheduledExecutorService scheduler =
    Executors.newSingleThreadScheduledExecutor(r -> {
        Thread t = new Thread(r, "myapp-timeout");
        t.setDaemon(true);
        return t;
    });

Multi<Event> buildEventStream(Config config, SdkClient client) {
    Duration timeout = config.timeout() != null
        ? config.timeout() : defaultTimeout();

    AtomicBoolean timedOut = new AtomicBoolean(false);

    // Schedule subprocess closure after wall-clock timeout.
    // compareAndSet prevents double-close if completion races with the timer.
    ScheduledFuture<?> timeoutFuture = scheduler.schedule(() -> {
        if (timedOut.compareAndSet(false, true)) {
            client.close().subscribe();
        }
    }, timeout.toMillis(), TimeUnit.MILLISECONDS);

    return Multi.createFrom()
        .publisher(sdkFlux)
        .map(this::toEvent)
        // Retype the error: if subprocess died due to our close(), it's a timeout;
        // otherwise it's a process error.
        .onFailure().transform(e -> timedOut.get()
            ? new TimeoutException(timeout)
            : new ProcessException(e.getMessage(), e))
        // Cancel the timer on all three termination paths.
        .onCompletion().invoke(() -> timeoutFuture.cancel(false))
        .onFailure().invoke(t -> timeoutFuture.cancel(false))
        .onCancellation().invoke(() -> timeoutFuture.cancel(false));
}
```

The timer is cancelled on completion, failure, and subscriber-driven cancellation — all three paths. `cancel(false)` does not interrupt a running timer handler; it only prevents a future scheduled execution.

## Why this works

When `client.close()` is called, the subprocess terminates. The SDK's Flux (bridged to Multi) fails with a transport exception. The `timedOut` flag tells the `onFailure().transform()` which exception type to emit — `TimeoutException` if the close was timer-driven, `ProcessException` otherwise.

Subprocess closure naturally propagates through the reactive chain without any stream composition. The timer does not need to "win a race" — it simply terminates the process and lets the normal failure path handle the rest.

## Why not merge-based timeout

`Multi.createBy().merging()` waits for ALL upstreams to complete. If the event stream finishes normally before the timeout fires, the merged stream stays open until the timer fires — then incorrectly fails with a timeout exception on a successful session.

## @PreDestroy integration

```java
@PreDestroy
void shutdown() {
    scheduler.shutdownNow();  // cancel pending timer tasks
    activeSessions.forEach(client -> {
        try { client.close().subscribe(); } catch (Exception ignored) {}
    });
}
```

`shutdownNow()` before session cleanup ensures no new timer closures race with @PreDestroy cleanup.

## Why non-obvious

The merge-based approach is canonical in RxJava (`amb()`) and Reactor (`firstWithSignal()`). Mutiny's merge semantics differ. Developers from either ecosystem reach for stream composition first. The subprocess-closure approach requires understanding that the timeout should kill the upstream resource, not race it as a stream.
