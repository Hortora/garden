### Symptom

`injectedBean.getClass().getSimpleName()` returns `MyService_ClientProxy` rather than `MyService`. Code that pattern-matches on the simple name (log messages, diagnostic output, assertions in tests) gets the generated proxy name instead of the implementation class name.

### Context

Any `@ApplicationScoped` (normal-scoped) CDI bean injected and then inspected via `getClass()`. Includes `@RequestScoped`, `@SessionScoped`. Does NOT affect `@Singleton` or `@Dependent` (pseudo-scoped — no client proxy).

### Root cause

CDI creates a client proxy (a generated subclass) for every normal-scoped bean. The injected reference is always the proxy. `Object.getClass()` is `final` and cannot be overridden by the proxy, so it returns the proxy's class, not the implementation class. Quarkus ARC generates single-level proxies — the proxy's direct superclass is always the implementation class.

### Fix

Use `getSuperclass().getSimpleName()` to strip the proxy suffix. Reliable for Quarkus ARC (single-level proxy chain):

```java
// Wrong — returns "InMemoryEndpointRegistry_ClientProxy"
String name = registry.getClass().getSimpleName();

// Right — returns "InMemoryEndpointRegistry"
String name = registry.getClass().getSuperclass().getSimpleName();
```

For `instanceof` / type checks: use `assertInstanceOf(MyService.class, bean)` — the proxy IS-A `MyService` (it subclasses it), so `instanceof` works correctly.

### Why this is non-obvious

`getClass()` appears to ask "what is this object?" but on a CDI proxy it answers "what generated class wraps this object?" The real bean is hidden behind the proxy. The generated class name includes a suffix (`_ClientProxy`) that's an implementation detail of Quarkus ARC, not visible in source code or documentation.
