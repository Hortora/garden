### Symptom

A scheduled job fires on every tick, but WorkItems remain stuck past their deadline
forever. No audit entries are written. No CDI events fire. No exception appears in the
business logs — the scheduler silently swallows the failure and retries the same items.

### Context

A `@Transactional` method processes a list of items in a loop:

```java
@Transactional
public void checkExpired() {
    for (WorkItem item : workItemStore.scan(WorkItemQuery.expired(now))) {
        executeDecision(item, policy.onBreach(ctx), ctx, now);  // may throw
    }
}
```

If any iteration throws an uncaught `RuntimeException`, the JTA transaction manager
rolls back the **entire method** — not just the failing item. Items 1..K-1 are also
rolled back. On the next scheduler tick (e.g. `@Scheduled(every = "60s")`), the same
items are scanned again, the same exception fires, and the loop continues forever.

Applied in: `ExpiryLifecycleService.checkExpired()` processing expired WorkItems with
a misconfigured `SlaBreachPolicy` returning `EscalateTo(emptyGroups)`.

### Root Cause

`@Transactional` wraps the entire method boundary. A loop of N items shares one
transaction. An uncaught `RuntimeException` at any point triggers a full JTA rollback.
The scheduler has no visibility into which items succeeded — it only sees that the tick
completed (or threw at the scheduler level, which it logs and ignores). Business-level
audit entries, CDI events, and state mutations are all rolled back with no trace.

### Fix

Two-layer defence — apply both:

**A — Validate at construction (authoring-time guard):**

```java
public static EscalateTo to(final String... groups) {
    if (groups.length == 0) {
        throw new IllegalArgumentException(
            "EscalateTo requires at least one group — use Fail to terminate instead");
    }
    return new EscalateTo(Set.of(groups), null);
}
```

Policy authors get an immediate `IllegalArgumentException` at the call site rather than
a silent transaction rollback at execution time.

**B — Convert to safe result at the executor (transaction-boundary guard):**

```java
case BreachDecision.EscalateTo escalate -> {
    if (escalate.groups().isEmpty()) {
        LOG.errorf("Policy returned EscalateTo with empty groups for %s — applying Fail", item.id);
        yield executeFail(item, new BreachDecision.Fail("escalation-misconfigured"), now);
    }
    yield executeEscalateTo(item, escalate, ctx, now);
}
```

Belt-and-suspenders: A catches bad policy code at authoring time; B protects the
transaction even if the record constructor is called directly, bypassing the factory.

### Why this is non-obvious

Developers expect a `RuntimeException` to fail the current item — not roll back all N
items processed so far in the same method. The scheduler's `@Scheduled(every=...)` retry
makes it look like the job is running normally. No log at the executor level means no
signal that anything went wrong until someone notices items are stuck indefinitely.

### Alternatives considered

- Wrapping each iteration in its own `try/catch` to isolate item failures — valid but
  verbose. The Fix B approach converts the error to a safe return value before it can
  propagate, which is cleaner for sealed-interface decision dispatch.
- Using `@Transactional(value = TxType.REQUIRES_NEW)` per item — works but makes each
  item a separate transaction, losing the ability to see in-flight state across items
  and complicating audit entry ordering.
