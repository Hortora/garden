## Context

A reactive SPI method returns `Uni<T>` and needs to make a blocking call (e.g. HTTP to an embedding service, JDBC, file I/O). The SPI is called from a `@ConsumeEvent` handler (Vert.x event bus) which runs on the Vert.x IO thread unless `blocking = true` is set.

## Technique

Use `.emitOn(Infrastructure.getDefaultWorkerPool())` on the `Uni` **before** the blocking operation. This shifts the execution to the Mutiny default worker pool:

```java
@Override
public Uni<AgentAssignment> select(AgentRoutingContext context, List<AgentCandidate> candidates) {
    // Classify (in-memory, fast — can stay on caller thread)
    List<ClassifiedCandidate> classified = classifier.classify(candidates, ...);

    return Uni.createFrom()
        .voidItem()
        .emitOn(Infrastructure.getDefaultWorkerPool())  // shift here, before blocking work
        .map(ignored -> {
            // Everything from here runs on the worker pool
            float[] queryVector = embeddingProvider.embed(queryText);  // blocking HTTP call
            // ... build scored candidates ...
            return TrustCandidateClassifier.decide(classified, scored, capability);
        });
}
```

## Why This Is Non-Obvious

Three common but wrong alternatives:

**1. `Uni.createFrom().item(() -> blockingOperation())`**
Executes the supplier on the thread that subscribes (the IO thread). Still blocks if called from a `@ConsumeEvent` handler.

**2. `@Blocking` on the handler method**
Moves the *entire* handler to a worker thread, including all in-memory work before the SPI call. Wasteful and scatters the threading decision into the caller rather than the SPI implementation.

**3. `Uni.createFrom().emitter(em -> { em.complete(blocking()); })`**
Also executes on the subscribing thread unless `.emitOn()` is chained.

The correct model: the `Uni` pipeline is lazy. `.emitOn(workerPool)` inserts a thread-handoff at that point in the pipeline — everything downstream of it executes on the worker pool when the `Uni` is subscribed. Place it immediately before the first blocking call, after any fast in-memory preparation.

## Constraints

Requires `io.smallrye.mutiny:mutiny` and `io.smallrye.mutiny.infrastructure.Infrastructure` on the classpath (shipped with Quarkus). `Infrastructure.getDefaultWorkerPool()` is the standard Mutiny executor for blocking work — do not pass a custom executor unless you need isolation.
