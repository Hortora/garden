## @TestTransaction + JPA auto-flush makes store-seam compliance tests pass before the fix is applied

### Symptom

You write a test to verify that a service routes state mutations through its injected `*Store`
interface (not via direct JPA entity mutation). The test calls the service twice — the first
call should mutate state and persist it; the second call should read the persisted state and
behave accordingly. The test passes cleanly **before the compliance fix is applied**.

### Context

Service `evaluateAll()` is annotated `@Transactional` (REQUIRED). The test is annotated
`@TestTransaction`. You call `evaluateAll()` twice in the same test body. The service loads
entities from the store on each call, mutates one, and the test asserts that the second call
observes the mutation (e.g. debounce check reads the updated `lastFiredAt`).

### Root Cause

`@TestTransaction` wraps the entire test body in a single transaction. Both
`evaluateAll()` calls join this outer transaction (REQUIRED semantics — no new transaction
boundary). This means both calls share the **same JPA persistence context**.

After the first `evaluateAll()` call:
1. `w.lastFiredAt = now` marks entity `w` dirty in the persistence context
2. The compliance fix (`watchdogStore.put(w)`) is supposed to route this through the store,
   but JPA dirty-tracking on the managed entity already records the mutation

When the second `evaluateAll()` calls `watchdogStore.scan()` → `Watchdog.list()` (JPQL
query), JPA's default `FlushModeType.AUTO` **automatically flushes all pending dirty state
before executing the JPQL query** — including `w.lastFiredAt`. The scan returns `w` with
the updated `lastFiredAt`, and the debounce check passes.

**The compliance fix (`watchdogStore.put(w)`) is therefore invisible to the test in the JPA context.** The test passes whether or not the store call is present.

### What was tried

Debounce test designed to call `evaluateAll()` twice and assert exactly one alert — intending
to verify that `lastFiredAt` is "persisted correctly." The test passes before the compliance
fix is applied, making it impossible to use as a RED test for the fix.

### Why this is non-obvious

JPA auto-flush before JPQL is documented, but its interaction with `@TestTransaction` +
shared persistence context + store-seam compliance testing is subtle. A developer writing
the test would reasonably expect that:
1. Two method calls mean two independent data loads (they don't — same persistence context)
2. If the fix is absent, the second scan would return stale data (it doesn't — auto-flush
   runs before the scan query)

The test looks like it's verifying store compliance, but it's actually verifying JPA behavior.

### Fix

For behavioral regression tests: the test is still valuable — it documents the debounce
invariant and will catch removal of `w.lastFiredAt = now` itself. Frame the assertion
message honestly: "isDebounced() must see the lastFiredAt set by the first call" rather
than "lastFiredAt must be persisted through the store."

For genuine compliance testing of store-seam bypasses: compliance is a structural contract
that cannot be verified at the JPA integration test level when the service is `@Transactional`
and the test is `@TestTransaction`. Structural compliance tests require either:
- A unit test with `InMemory*Store` instances that return **copies** (not references)
- A `@QuarkusTest` with a custom `@Alternative @Priority(2)` spy store that records calls
- An ArchUnit rule verifying no Panache static calls from service classes
