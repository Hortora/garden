### Symptom

Adding `casehub-connectors-core` to a Quarkus application's `runtime/pom.xml` causes CDI startup
failure in JDBC-only `@QuarkusTest` runs:

```
Unsatisfied dependency for type ... TwilioSmsConnector
CDI container validation failed
```

No error mentions the missing credential config. ArC validation fails at build time, not at injection
time — the test suite cannot start at all.

### Context

`TwilioSmsConnector` and `WhatsAppConnector` inject required configuration properties
(`casehub.connectors.twilio.account-sid`, `casehub.connectors.twilio.auth-token`,
`casehub.connectors.whatsapp.*`). In a `@QuarkusTest` environment without external service
credentials (JDBC-only CI, local dev), these properties are absent and ArC validates the
entire bean graph at build augmentation time — before the application starts. Both beans
fail validation even when they are never used by any code under test.

`SlackConnector` and `TeamsConnector` do not have this problem — they receive the destination
URL as a runtime argument to `send()` and have no required config properties.

### Root cause

ArC performs eager CDI validation at build time. Beans with `@ConfigProperty` fields that have
no default value fail validation when the property is absent, even if the bean is never injected
or called. The JDBC-only test environment has no Twilio or WhatsApp credentials in
`application.properties`.

### Fix

Add to `runtime/src/test/resources/application.properties`:

```properties
# Twilio and WhatsApp require external credentials not present in tests.
# SlackConnector is replaced by TestSlackConnector via @Mock.
quarkus.arc.exclude-types=io.casehub.connectors.twilio.TwilioSmsConnector,\
  io.casehub.connectors.whatsapp.WhatsAppConnector
```

For `SlackConnector`: replace with a `@io.quarkus.test.Mock` alternative in the test
source root that captures sent messages via a static list:

```java
@Mock
public class TestSlackConnector implements Connector {
    public static final List<ConnectorMessage> sent = new CopyOnWriteArrayList<>();
    public static volatile boolean shouldThrow = false;

    @Override public String id() { return "slack"; }

    @Override public void send(ConnectorMessage message) {
        if (shouldThrow) throw new RuntimeException("Connector failure");
        sent.add(message);
    }
}
```

### Why this is non-obvious

Adding a runtime dependency to a pom.xml should not break CDI unless the consumer injects a
bean that uses it. Neither the consumer code nor any test code references `TwilioSmsConnector`
directly. The failure happens because the bean is discovered via Jandex scanning of the classpath
— presence on the classpath is sufficient for ArC to validate it. The error message names the
connector class but gives no hint about missing config keys.
