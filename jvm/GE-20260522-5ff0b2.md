### What it does

Converts a void switch statement over a sealed interface into a switch *expression*, gaining compile-time exhaustiveness enforcement without changing behaviour. Adding a new sealed subtype without updating the switch becomes a compile error instead of a runtime `MatchException`.

### Context

Applies whenever you have a method that pattern-dispatches over a sealed interface and returns void — the common case for command/event dispatch, visitor-style patterns, or intent application loops. Java 21 switch *statements* with pattern matching are NOT exhaustiveness-checked at compile time even on sealed types; only switch *expressions* are.

### The pattern

```java
// BEFORE — switch statement (compiles silently if a new Intent subtype is added):
void applyIntent(Intent intent, PlayerState state) {
    switch (intent) {
        case MoveIntent   m -> setTarget(m.unitTag(), m.targetLocation(), false, state);
        case AttackIntent a -> setTarget(a.unitTag(), a.targetLocation(), true,  state);
        case TrainIntent  t -> handleTrain(t, state);
        case BuildIntent  b -> handleBuild(b, state);
        case BlinkIntent  b -> executeBlink(b.unitTag(), state);
    }
}

// AFTER — Runnable switch expression (compile error if any arm is missing):
void applyIntent(Intent intent, PlayerState state) {
    Runnable action = switch (intent) {
        case MoveIntent   m -> () -> setTarget(m.unitTag(), m.targetLocation(), false, state);
        case AttackIntent a -> () -> setTarget(a.unitTag(), a.targetLocation(), true,  state);
        case TrainIntent  t -> () -> handleTrain(t, state);
        case BuildIntent  b -> () -> handleBuild(b, state);
        case BlinkIntent  b -> () -> executeBlink(b.unitTag(), state);
    };
    action.run();
}
```

Each arm wraps the void call in a zero-argument lambda `() -> method(...)`. The switch is now an expression (it produces a `Runnable`), so the compiler enforces exhaustiveness against the sealed `permits` clause. `action.run()` executes synchronously — no threading, no overhead beyond a tiny lambda allocation.

### Why this is non-obvious

The obvious attempts don't work:
- **Leave as statement** — Java 21 switch statements with pattern matching are *not* compile-time exhaustive, even on sealed types. A missing arm throws `MatchException` at runtime.
- **`yield null` hack** — `Object _ = switch (x) { case Foo f -> { doFoo(f); yield null; } }` works but is visually noisy and confuses readers about intent.
- **`@SuppressWarnings` + statement** — suppresses warnings but doesn't give exhaustiveness.

The `Runnable` pattern is clean: the method signature stays `void`, the dispatch is a one-liner expression, and the exhaustiveness guarantee is real. The lambdas capture effectively-final variables correctly — parameters and local pattern variables from the case arms are all in scope.

### Constraints

Applies to sealed interfaces only (Java 17+ `sealed ... permits`). For non-sealed types or enums, other patterns apply. The lambda allocation is negligible for non-hot paths; for tight loops, benchmark first.

### Alternatives considered

- **Switch statement with explicit `default -> throw new AssertionError()`** — catches missing arms at runtime with a clearer error than `MatchException`, but still not compile-time. Requires remembering to update the throw as new subtypes are added.
- **`yield null` switch expression** — works but is harder to read and triggers IDE warnings about unused assignments.
