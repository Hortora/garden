## Quarkus ARC registers non-CDI abstract class as @Dependent bean when it declares @Observes and implements a CDI SPI interface

### Symptom

12 integration tests fail after adding `@Observes AgentKeyRotatedEvent` to an abstract class
that implements a CDI SPI interface (e.g. `AgentSigner`). Failure is ambiguous injection:

```
AmbiguousResolutionException: Ambiguous dependencies for type AgentSigner
  Beans: [AbstractCachingAgentSigner, ConfiguredAgentKeyProvider]
```

`@InjectMock AgentSigner` in `@QuarkusTest` also fails — two candidates exist instead of one.

### Root cause

ARC's bean discovery does not skip abstract classes that implement CDI SPI interfaces.
When an abstract class declares `@Observes`, ARC treats the observer method as a CDI
observer and registers the abstract class as a `@Dependent` managed bean — even though
the class has no CDI scope annotation. The class becomes a second candidate for the
SPI injection point, causing ambiguous resolution wherever the SPI is injected.

CDI 4.0 §3.5 says observers are inherited by bean subclasses, which implies the abstract
class does not need to be a bean itself. ARC 3.32.2 does not implement this interpretation
for abstract classes with CDI SPI interfaces.

### Fix

**Remove `@Observes` from the abstract class.** Expose the method as a plain public
method instead. Concrete `@ApplicationScoped` CDI subclasses wire it as `@Observes`:

```java
// Abstract class (no CDI annotations — NOT a bean)
public abstract class AbstractCachingAgentSigner<C> implements AgentSigner {
    // ...
    // Plain method — concrete CDI subclasses expose as @Observes
    public void onKeyRotated(AgentKeyRotatedEvent event) {
        invalidate(event.actorId());
    }
}

// Concrete CDI bean — inherits the method and wires it as an observer
@ApplicationScoped
public class ConfiguredAgentKeyProvider extends AbstractCachingAgentSigner<KeyPair> {
    @Observes
    @Override
    public void onKeyRotated(AgentKeyRotatedEvent event) {
        super.onKeyRotated(event);
    }
}
```

If no concrete subclass exists yet, leave the method un-wired with a Javadoc note for future subclasses.

### Why this is non-obvious

ARC's behavior appears to contradict CDI 4.0 §3.5, which says observer methods in
non-bean superclasses are inherited by bean subclasses — implying the abstract class
need not be a bean. A developer reading the spec and trusting this behaviour will be
confused when ARC registers the abstract class as a bean candidate.

The failure mode is also distant from the cause: tests fail with ambiguous resolution
on the SPI interface (`AgentSigner`), not on the observer or the abstract class.
Debugging points to two beans satisfying `AgentSigner`, but the "second" bean
(`AbstractCachingAgentSigner @Dependent`) was not intentionally defined.

### Why this fix

Placing observers on concrete CDI subclasses is both more explicit and ARC-compatible.
The inheritance-via-plain-method approach preserves the single-point-of-logic benefit
while letting each concrete bean control its own CDI wiring.
