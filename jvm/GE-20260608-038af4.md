### Symptom

`MessageObserver.onMessage()` (or any JTA `Synchronization.afterCompletion()` callback) never fires in a `@QuarkusTest @TestTransaction` test. The dispatch call succeeds with no exception, but the observer's received list remains empty. No indication that anything is wrong.

### Context

Occurs when:
1. An observer is registered via `TransactionSynchronizationRegistry.registerInterposedSynchronization()` inside a `@Transactional` method (e.g. `messageService.dispatch()` in casehub-qhorus)
2. The test injects the service and calls it directly within a `@TestTransaction` test method

The `@TestTransaction` rollback is the cause — it hides inside the test lifecycle invisibly.

### Root Cause

`MessageObserverDispatcher` in casehub-qhorus (and the same pattern in any code using `TransactionSynchronizationRegistry`) registers a JTA `Synchronization` that only fires observers on:

```java
@Override
public void afterCompletion(final int status) {
    if (status == Status.STATUS_COMMITTED) {
        dispatchToHandles(...);   // ← only here
    } else {
        active.forEach(Instance.Handle::close);  // rollback → no observers
    }
}
```

`@TestTransaction` wraps each test method in a JTA transaction that rolls back after the test. When `messageService.dispatch()` registers the synchronization and the test body ends, `afterCompletion(STATUS_ROLLEDBACK)` fires — and all observer callbacks are silently skipped.

No error is raised. The dispatch API confirms success. The observer simply doesn't see the message.

### Fix

Use `QuarkusTransaction.requiringNew()` to create a real nested transaction that commits:

```java
// Bad — observer never fires in @TestTransaction
@Test @TestTransaction
void badTest() {
    UUID channelId = createChannel("test-" + UUID.randomUUID());
    messageService.dispatch(MessageDispatch.builder()...build());
    assertThat(myObserver.received).hasSize(1); // ← always fails
}

// Good — commit the tx so afterCompletion(STATUS_COMMITTED) fires
@Test
void goodTest() {
    // Channel must be committed first so dispatch can find it
    UUID[] channelId = {null};
    QuarkusTransaction.requiringNew().run(() -> {
        Channel ch = new Channel();
        ch.id = UUID.randomUUID();
        ch.name = "test-" + ch.id;
        ch.semantic = ChannelSemantic.APPEND;
        channelStore.put(ch);
        channelId[0] = ch.id;
    });

    // Dispatch in committed TX — afterCompletion(COMMITTED) fires synchronously after run()
    QuarkusTransaction.requiringNew().run(() ->
        messageService.dispatch(MessageDispatch.builder()
                .channelId(channelId[0]).sender("agent")
                .type(MessageType.STATUS).content("hello")
                .actorType(ActorType.AGENT).build()));

    // Observer has been invoked by the time we reach here
    assertThat(myObserver.received).hasSize(1);
}
```

`QuarkusTransaction.requiringNew().run()` creates a JTA transaction that commits when `run()` returns. JTA synchronizations registered inside that transaction fire their `afterCompletion(STATUS_COMMITTED)` callback synchronously in the same thread before `run()` returns to the caller. The test can then assert on observer state immediately.

### Why This Is Non-Obvious

`@TestTransaction` looks like it creates a real, functioning transaction — it does. The transaction works, the dispatch works, and the service returns success. The only symptom is that observers don't see the message. Reading `MessageObserverDispatcher` source is required to understand that the dispatch is deferred to `afterCompletion(COMMITTED)`, and that `@TestTransaction`'s rollback silently short-circuits it. The Quarkus documentation for `@TestTransaction` does not mention its interaction with JTA `Synchronization` callbacks.

The synchronous fallback path in the dispatcher (`tsr == null || tsr.getTransactionStatus() != Status.STATUS_ACTIVE`) does fire observers immediately — so unit tests without an active JTA transaction work fine. Only tests where JTA is active (like `@TestTransaction`) hit the deferred path and are silently broken.

### Constraints

Applies to JTA-deferred observer dispatch using `TransactionSynchronizationRegistry.registerInterposedSynchronization()`. Tests that call the service outside any JTA context (tsr == null) use the synchronous fallback path and are unaffected. Tests using `@ObservesAsync` CDI events (not JTA synchronizations) have a different failure mode (see `GE-20260513-b15933`).
