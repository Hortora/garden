### Symptom
Compiling `field.contains(param)` as an MVEL3 expression with `Boolean` output type
produces: `incompatible types: java.lang.String cannot be converted to java.lang.Boolean`.

### Context
MVEL3 transpiles expressions to Java source, compiles via javac, and loads bytecode.
When the transpiler encounters `.contains()` on a String value extracted from a Map
context, it parses `contains` as the MVEL collection containment keyword (inherited
from MVEL2: `collection contains element`) instead of as a method call on
`java.lang.String`.

The generated Java source returns `String` (the collection-contains evaluation path)
rather than `boolean` (what `String.contains(CharSequence)` returns), causing the
javac compilation to fail with a type mismatch.

### What was tried (didn't work)
- `field.contains(param)` — keyword conflict
- `field.contains(param) == true` — same conflict, `== true` doesn't help because
  the parser intercepts `contains` before method resolution

### Root cause
MVEL's parser treats `contains` as a reserved keyword for collection membership
testing. The keyword takes precedence over Java method resolution, so
`String.contains()` is never dispatched as a method call.

### Fix
Use `String.indexOf()` instead — it is not a MVEL keyword:

```java
// Instead of: field.contains(param)
// Use:        field.indexOf(param) >= 0

case CONTAINS -> field + ".indexOf(" + param + ") >= 0";
```

Similarly, `String.startsWith()` works because `startsWith` is not a MVEL keyword.
The pattern generalises: any Java method whose name collides with a MVEL keyword
needs an equivalent non-keyword method call.

### Why this is non-obvious
The error message (`String cannot be converted to Boolean`) points at a type
mismatch, not a keyword conflict. Nothing in the MVEL3 documentation warns about
keyword shadowing of Java method names. The natural assumption is that
`.contains()` on a String works the same as in Java.

### Alternatives considered
- Escaping or quoting the method name — MVEL3 has no syntax for this
- Using backtick-quoted identifiers — not supported in MVEL3's parser
- Wrapping in a helper function — unnecessary complexity when `indexOf` works
