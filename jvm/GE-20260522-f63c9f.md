### What happened

A `@QuarkusTest` suite had 17+ tests fail with incorrect subjectId propagation. The root cause: `LedgerWriteService.record()` runs in `REQUIRES_NEW`, which suspends the outer `@TestTransaction` and commits the ledger entry independently. The ledger entries persist to the H2 database even after the `@TestTransaction` outer transaction rolls back.

A later test reusing the same `correlationId` called `findEarliestWithSubjectByCorrelationId(correlationId)` — a cross-channel, cross-test query. It found the committed ledger entry from the prior test and used its `subjectId`, corrupting the current test's expected propagation result.

### Symptom

Tests that assert on propagated `subjectId` intermittently fail — the value comes from a previous test's COMMAND entry, not the current test's. Failures are order-dependent and disappear when tests run in isolation.

### Context

Occurs when:
1. A service uses `@Transactional(REQUIRES_NEW)` to write audit/ledger entries that must survive outer transaction rollback
2. A query in that service is scoped **cross-channel** (does not include a channelId or testId predicate)
3. Tests share a `correlationId` string literal across test methods

### What was tried

- Checking `@TestTransaction` rollback behaviour — confirmed it rolls back outer tx. The key insight is that REQUIRES_NEW entries are **already committed** before rollback happens.
- Looking for stale entity cache issues — not the cause.

### Root cause

`@TestTransaction` rolls back the outer transaction, but entries committed inside a `REQUIRES_NEW` subtransaction are **permanently committed** — they are not part of the outer transaction and cannot be rolled back with it. A cross-channel query that searches by `correlationId` without scoping to a testId or unique identifier will find these committed entries from prior tests.

### Fix

Give each test a unique `correlationId` that cannot collide across tests:

```java
// BAD — shared correlationId across tests
String corrId = "corr-chain";

// GOOD — unique per test run
String corrId = "corr-repo-test-" + UUID.randomUUID();
```

For integration tests with shared channel names, also ensure channels are unique:
```java
String channelName = "my-channel-" + UUID.randomUUID();
```

### Why this is non-obvious

`@TestTransaction` is documented to roll back all changes — developers assume this includes REQUIRES_NEW subtransactions. It doesn't. REQUIRES_NEW commits are permanent by design (that's their point — they survive outer transaction failure). The failure is silent: no error is thrown, the wrong value is silently used, and the test fails with a value-assertion error that looks like a logic bug in the production code rather than a test isolation problem.
