## Context

Two `@DefaultBean @ApplicationScoped` beans from different JARs both implement the same interface. CDI can't resolve which to use and throws `AmbiguousResolutionException`. Standard approaches don't help:

- `quarkus.arc.exclude-types` is silently ignored for beans from Jandex-indexed JARs (see GE-20260601-848232)
- Marking one as `@Alternative` requires changes to a JAR you don't own
- `@Priority` on `@Alternative` has CDI 2.0 global-scope problems

## Technique

Add a concrete `@ApplicationScoped` bean to your project (production or test sources as appropriate) that implements the conflicting interface:

```java
@ApplicationScoped
public class ProjectGroupMembershipProvider implements GroupMembershipProvider {
    @Override
    public Set<GroupMember> membersOf(String groupName) {
        return Set.of(); // or delegate to preferred implementation
    }
}
```

CDI selects the most-specific non-default bean. Since your bean is NOT `@DefaultBean`, it wins over both competing `@DefaultBean` implementations and the `AmbiguousResolutionException` is resolved. No changes to any JAR required.

## Why This Is Non-Obvious

Most developers reach for `exclude-types` first (fails silently for Jandex-indexed JARs), then try `@Priority` on one of the `@DefaultBean` beans (requires modifying third-party JARs), then consider `@Alternative` + `selected-alternatives` (same problem). The CDI suppression rule — "a non-default bean beats all @DefaultBean candidates for the same type" — is the correct lever, and a thin adapter class is the minimal mechanism that activates it without touching any dependency.
