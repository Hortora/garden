## Symptom

`@Valid` on a JAX-RS method parameter and `@NotBlank`/`@NotNull`/`@Positive` on request body record fields compile, CDI wires, and the app starts — but validation never fires. A missing required field returns 200 instead of 400. A null enum field causes a 500 on persist instead of a 400 at the REST layer.

## Context

Quarkus REST application using `quarkus-rest` and `quarkus-rest-jackson`. Bean Validation annotations are on record component fields. `@Valid` is on the JAX-RS method parameter.

## Root Cause

`quarkus-rest` does not transitively include `quarkus-hibernate-validator`. The Jakarta Validation API is on the classpath (from another transitive dep), so annotations resolve at compile time, but the validation provider that actually enforces constraints is absent. Quarkus silently skips validation rather than throwing a startup error.

## Fix

Add `quarkus-hibernate-validator` explicitly to `pom.xml`:

```xml
<dependency>
    <groupId>io.quarkus</groupId>
    <artifactId>quarkus-hibernate-validator</artifactId>
</dependency>
```

Once added, `@Valid` triggers constraint evaluation and returns 400 with a constraint violation body on failure.

## Why This Is Non-Obvious

Quarkus RESTEasy Classic included validation transitively. Developers migrating to `quarkus-rest` (the new reactive REST layer) expect the same behaviour. The silent failure — annotations present, app starts, but constraints don't fire — gives no diagnostic signal. The only way to discover it is a failing test showing 200 where 400 was expected.
