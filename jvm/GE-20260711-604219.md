### Symptom
After calling `ide_replace_member` on a record's compact constructor, the code compiles with errors — `<identifier> expected` and `statements not expected outside of methods and initializers`. The record appears structurally broken.

### Context
Using `ide_replace_member(file, class="Categorical", member="Categorical", parameterCount=2, content="...")` to update the body of a compact constructor in a sealed `FeatureField` record hierarchy. The content provided is the body statements (validation logic) without the `public Categorical {` wrapper — same pattern that works for regular method bodies.

### What was tried (didn't work)
`ide_replace_member` with `class` and `member` both set to the record name, providing just the body statements. The tool stripped the `public RecordName {` declaration, placing body statements directly after the record's opening brace as instance initializer code.

Result: `record Categorical(...) {Objects.requireNonNull(name,"name"); if(similaritySpec != null) { ... }}` — no compact constructor declaration.

### Root cause
`ide_replace_member` replaces "the code between { and }" for methods. For a compact constructor, the compact constructor IS a special syntactic form where the entire `public RecordName { body }` block is the constructor. When the tool strips the enclosing braces, it removes the compact constructor declaration entirely, leaving the body as bare statements in the record body — which is syntactically an instance initializer block, not a constructor.

### Fix
Use `ide_edit_member` instead of `ide_replace_member` for compact constructors. `ide_edit_member` replaces the entire member declaration including signature and body:

```
ide_edit_member(file, class="FeatureField", member="Categorical",
  content="record Categorical(String name, SimilaritySpec spec) implements FeatureField {\n    public Categorical { ... }\n}")
```

This preserves the compact constructor syntax because you provide the complete record declaration.

### Why this is non-obvious
`ide_replace_member` works correctly for regular methods — replacing the body between braces is well-defined. The failure is specific to compact constructors, which have a unique syntactic form in Java records. The error manifests as a compilation failure, not a tool error, so the tool reports success while producing broken code.
