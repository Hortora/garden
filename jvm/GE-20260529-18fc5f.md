## Symptom

A `@QuarkusTest` annotated with `@TestProfile` (whose `getConfigProfile()` returns a custom name,
e.g. `"reactive-pg"`) starts the application but either:
- The datasource bean is deactivated: *"Bean is not active: AgroalDataSource — URL is not set"*
- Or the wrong datasource URL is used (e.g., the production H2 file URL rather than the test
  in-memory URL), causing Flyway or Hibernate to fail with a driver mismatch

## Context

Production `application.properties` contains a datasource URL without a profile prefix:

```properties
quarkus.datasource.qhorus.jdbc.url=jdbc:h2:file:~/.app/qhorus
```

The test uses `QuarkusTestProfile.getConfigProfile()` returning `"reactive-pg"` to activate
PostgreSQL-specific config. Quarkus Dev Services is expected to provide the PostgreSQL URL.

## What was tried (didn't work)

**Attempt 1** — Set `%reactive-pg.quarkus.datasource.qhorus.jdbc.url=` (empty string) to clear
the H2 URL so Dev Services can activate:

Dev Services does start the container. But Quarkus **deactivates the Agroal datasource bean
at augmentation time** because `jdbc.url` resolves to an empty string. The bean is marked
inactive before Dev Services can inject a replacement at runtime. Error: *"Datasource 'qhorus'
was deactivated automatically because its URL is not set."*

**Attempt 2** — Set `devservices.enabled=true` explicitly without any URL override:

Dev Services starts the container. But the production H2 URL (`jdbc:h2:file:...`) is still
active — Dev Services does **not** override a configured URL even when `devservices.enabled=true`
is set explicitly. Flyway connects to the H2 URL with the PostgreSQL driver → driver mismatch
failure.

## Root cause

`QuarkusTestProfile.getConfigProfile()` **replaces** the `%test` profile entirely — it does
NOT add to it. When `getConfigProfile()` returns `"reactive-pg"`:

- Active config: production `application.properties` (no profile prefix) + `%reactive-pg.*` properties
- **NOT active**: `%test.*` properties (the `%test` profile is replaced, not extended)

This means the production datasource URL (`quarkus.datasource.qhorus.jdbc.url=jdbc:h2:file:...`)
is visible, while the test profile's H2 in-memory URL (`%test.quarkus.datasource.qhorus.jdbc.url`)
is not.

**Dev Services config injection priority is lower than profile config.** Dev Services injects
URLs as a late-binding runtime config source. Any explicitly configured URL — whether from
production config, profile config, or `getConfigOverrides()` — takes precedence and blocks
Dev Services injection.

**Empty-string is not "absent".** Setting `jdbc.url=` explicitly tells Quarkus the URL is
configured (to empty). Quarkus deactivates the Agroal bean because the configured URL is
empty. This happens at augmentation time; Dev Services cannot reverse it at runtime.

## Fix

Use `QuarkusTestResourceLifecycleManager.start()` to provide the real JDBC+reactive URLs.
Its return value is applied at the **highest config priority** — above system properties,
profile config, and production config. It is available before Quarkus augmentation.

```java
public class PostgresTestResource implements QuarkusTestResourceLifecycleManager {
    private PostgreSQLContainer<?> postgres;

    @Override
    public Map<String, String> start() {
        postgres = new PostgreSQLContainer<>("postgres:17-alpine")
                .withDatabaseName("mydb").withUsername("user").withPassword("pass");
        postgres.start();
        String reactiveUrl = "vertx-reactive:postgresql://" + postgres.getHost()
                + ":" + postgres.getMappedPort(5432) + "/" + postgres.getDatabaseName();
        return Map.of(
                "quarkus.datasource.qhorus.jdbc.url",     postgres.getJdbcUrl(),
                "quarkus.datasource.qhorus.reactive.url", reactiveUrl,
                "quarkus.datasource.qhorus.username",     postgres.getUsername(),
                "quarkus.datasource.qhorus.password",     postgres.getPassword(),
                "quarkus.flyway.qhorus.migrate-at-start", "true");
    }

    @Override
    public void stop() { if (postgres != null) postgres.stop(); }
}
```

Annotate the IT class with `@QuarkusTestResource(value = PostgresTestResource.class, restrictToAnnotatedClass = true)`.

Dependencies needed: `org.testcontainers:testcontainers-postgresql` (version managed by
Quarkus BOM). For reactive calls inside tests, also add `io.quarkus:quarkus-test-vertx`
and use `@RunOnVertxContext + UniAsserter` (Panache.withTransaction() requires a Vert.x
context the JUnit thread does not have).

## Why this is non-obvious

The Quarkus documentation states that test `application.properties` is read for
`@QuarkusTest`. It does not prominently document that `getConfigProfile()` replaces (not
extends) the test profile, making production config active. The natural assumption is that
the test profile properties always apply; they don't when a custom config profile is set.

Dev Services appears to work (the container starts and logs success) but still fails to
inject the URL — the failure is silent until Hibernate/Flyway fails with a confusing
driver mismatch or empty-URL error.

**See also:** GE-20260422-13f53b — `@TestProfile` restarts not inheriting test datasource
config (broader treatment via `getConfigOverrides()`). GE-20260516-e137f6 — `getConfigOverrides()`
replaces `%test.*`-prefixed config for that profile.
