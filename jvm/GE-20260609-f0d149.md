## Symptom

`IllegalArgumentException: expected image size [N] is not equal actual size [M]` or `"data is required"` when parsing an `ObservationRaw` that contains image data (MapState, FeatureLayer, etc.).

The `data` field on `Common.ImageData` appears to be optional from the proto definition and from the `orElseThrow` pattern in the ocraft sources — but calling `ImageData.from()` with a missing or wrong-size `data` field throws at runtime.

## Root Cause

`ImageData` validates two invariants:
1. `data` must be present (not null)
2. `data.size()` must equal `bitsPerPixel * width * height / 8`

```java
// ocraft ImageData constructor (ocraft-s2client-protocol 0.4.21)
data = tryGet(
    Common.ImageData::getData, Common.ImageData::hasData
).apply(sc2ApiImageData).orElseThrow(required("data"));  // ← REQUIRED

int expectedSize = size.getX() * size.getY() * bitsPerPixel / 8;
if (!imageSizeIsValid(expectedSize)) {                   // ← EXACT SIZE
    throw new IllegalArgumentException(
        format("expected image size [%d] is not equal actual size [%d]", expectedSize, data.size()));
}
```

The `tryGet` pattern with `orElseThrow` makes `data` hard-required; the size validation makes the content structurally constrained.

## Fix

When building test `ImageData` protos, always include the exact number of bytes the dimensions require:

```java
// For bitsPerPixel=8, size=1×1: exactly 1 byte required
Common.ImageData minimalImage = Common.ImageData.newBuilder()
    .setBitsPerPixel(8)
    .setSize(Common.Size2DI.newBuilder().setX(1).setY(1).build())
    .setData(com.google.protobuf.ByteString.copyFrom(new byte[]{0}))  // ← 1 byte
    .build();
```

For `MapState`, both `visibility` and `creep` must be provided with valid `ImageData`.

## Why Non-Obvious

The proto definition and the `tryGet(...).orElseThrow(required(...))` pattern both suggest that field optionality follows standard proto3 semantics. The size validation is an ocraft-level constraint that is not visible from the proto definition or from the `ImageData.getData()` method signature. The `sc2ApiImageData()` fixture in ocraft's test suite happens to provide a full `HEIGHT_MAP` (thousands of bytes), so test authors who copy the fixture never see the issue — until they try to provide a minimal image for a small test fixture.
