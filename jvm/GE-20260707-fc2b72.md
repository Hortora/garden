## Catching PersistenceException from em.persist() does not prevent JTA transaction rollback — try/catch gives false isolation

### Symptom

A try/catch around `em.persist()` catches the `PersistenceException` and the method returns normally, but the enclosing `@Transactional` method's earlier persist operations are silently rolled back. Data appears to have been saved (no exception escaped the method) but is absent from the database.

### Context

Any code that wraps `em.persist()` in a try/catch within a JTA-managed `@Transactional` method, intending to make one persist optional while preserving another. Common in "best-effort secondary write" patterns — e.g. saving an audit entry first, then attempting an optional attestation write.

### What was tried (didn't work)

1. Wrapping the optional persist in try/catch — exception caught, but transaction still rolls back
2. Reordering operations so the critical persist runs first — the critical data IS flushed to the persistence context, but the transaction rollback undoes it

### Root cause

JPA 3.1 spec §3.3.7.1: "When a PersistenceException is thrown, the persistence provider **must** mark the current transaction for rollback." This happens inside Hibernate before the exception reaches user code. By the time the catch block executes, `Transaction.setRollbackOnly()` has already been called on the JTA transaction. The `@Transactional` interceptor checks the transaction status on method return and rolls back regardless of whether the method returned normally.

### Fix

Use `@Transactional(TxType.REQUIRES_NEW)` on the optional persist operation (either on the method or via a separate CDI bean). This runs the optional persist in its own transaction — if it fails, only the optional transaction is rolled back. The enclosing transaction is suspended and unaffected.

```java
// WRONG — try/catch does not isolate the transaction
@Transactional(TxType.REQUIRES_NEW)
public void record() {
    em.persist(criticalEntry);      // succeeds
    try {
        em.persist(optionalEntry);  // throws PersistenceException
    } catch (Exception e) {
        LOG.warn("optional write failed");
        // Transaction is ALREADY marked for rollback — criticalEntry is doomed
    }
}

// RIGHT — REQUIRES_NEW isolates the optional persist
@Transactional(TxType.REQUIRES_NEW)
public void saveOptional(Entity optional) {
    em.persist(optional);  // if this fails, only THIS transaction rolls back
}
```

### Why this is non-obvious

The Java try/catch mechanism and the JTA transaction mechanism operate at different levels. A developer's mental model is "catch the exception → prevent the damage." This is true for application-level errors, but JPA's transaction-marking happens inside the persistence provider before the exception reaches the catch block. The catch prevents the exception from propagating but cannot undo the rollback marking that already happened.
