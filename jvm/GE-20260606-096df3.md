## LangChain4j EmbeddingSearchRequest.filter() pushes domain/metadata filters into Qdrant pre-scoring

**Stack:** LangChain4j 1.x, Qdrant

Pass a `Filter` to `EmbeddingSearchRequest.builder().filter()` to apply a metadata
payload filter in Qdrant *before* vector similarity scoring. `QdrantFilterConverter`
translates it to a gRPC `FieldCondition`. Only entries matching the filter participate
in the similarity computation, so the `maxResults` limit is respected correctly.

### Context

A post-retrieval Java stream filter on `embeddingStore.search(request).matches()` is
incorrect at scale. With `maxResults=8` and a corpus of 100 domain=tools entries plus
5 domain=jvm entries, a search with domain=jvm may return 8 tools entries (all more
similar to the query), the Java filter removes them all, and the caller receives 0
results — silently.

### The technique

```java
import dev.langchain4j.store.embedding.filter.Filter;
import dev.langchain4j.store.embedding.filter.comparison.IsEqualTo;

Filter domainFilter = (domain != null && !domain.isBlank())
        ? new IsEqualTo("domain", domain)
        : null;

var request = EmbeddingSearchRequest.builder()
        .queryEmbedding(queryEmbedding)
        .maxResults(maxResults)
        .filter(domainFilter)   // null = no filter
        .build();

List<SearchResult> results = embeddingStore.search(request).matches().stream()
        .map(...)               // no post-retrieval filter needed
        .toList();
```

### Filter chain (verified from bytecode)

`IsEqualTo("domain", value)` → `EmbeddingSearchRequest.filter()` → `QdrantEmbeddingStore.search()` → `QdrantFilterConverter.buildEqCondition()` → Qdrant gRPC `FieldCondition` (server-side payload filter).

Available filter types (all in `dev.langchain4j.store.embedding.filter.comparison`):
`IsEqualTo`, `IsNotEqualTo`, `IsIn`, `IsNotIn`, `IsGreaterThan`, `IsLessThan`,
`ContainsString`. Logical combinators: `And`, `Or`, `Not` (in `...filter.logical`).

`InMemoryEmbeddingStore` (used in `@QuarkusTest` mocks) also honours the filter —
tests behave correctly without a real Qdrant instance.

### Why this is non-obvious

A developer adding domain filtering naturally reaches for Java stream `.filter()` on
the returned matches. The `EmbeddingSearchRequest.filter()` API exists but is not
prominently documented for the Qdrant integration. The correctness issue (limit
truncation at scale) only becomes visible with a corpus large enough that
same-domain entries don't dominate the top-N by similarity.
