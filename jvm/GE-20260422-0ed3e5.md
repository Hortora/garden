## Pattern

Large JVM frameworks that expose extension points make a foundational choice: wire extensions via **CDI container injection** or via **service-loader / explicit registration**. Both achieve the same structural depth (thousands of interfaces, deep abstraction hierarchies) but differ radically in how dependencies flow at runtime.

### CDI container wiring (Quarkus style)

Extensions are beans. Callers inject them via `@Inject`, `Instance<T>`, or `@Any Instance<T>`. The CDI container owns lifetime and resolution.

```java
// Extension point
public interface ExpiryPolicy {
    String name();
    boolean isExpired(Session s, Duration timeout);
}

// Discovery — zero explicit registration
@ApplicationScoped
public class PolicyRegistry {
    @Inject @Any Instance<ExpiryPolicy> policies;

    public ExpiryPolicy get(String name) {
        return policies.stream()
            .filter(p -> p.name().equals(name))
            .findFirst().orElseThrow();
    }
}
```

### Service-loader wiring (Hibernate style)

Extensions are registered via `META-INF/services/<InterfaceFQN>` files or explicit `ServiceRegistry` calls. The framework owns resolution; no CDI container required.

```java
// META-INF/services/org.hibernate.integrator.spi.Integrator
// → com.example.MyIntegrator

// Discovery — explicit service lookup
ServiceLoader<Integrator> loader =
    ServiceLoader.load(Integrator.class, classLoader);
```

## Why this is non-obvious

The two approaches look structurally identical at a coarse level — same number of interfaces, same abstraction depth ratio. A structural fingerprint alone won't distinguish them. The signal is in the **injection-to-file ratio**: CDI-wired frameworks approach 0.4 injections/file; service-loader frameworks stay near 0.002. That gap (200×) survives normalisation and is the discriminator.

This matters when **integrating the two styles**: a Hibernate extension brought into a Quarkus application must be wrapped in a CDI producer or adapter bean — the wiring strategies don't compose directly.

### Alternatives considered

- **Manual factory / registry**: explicit `Map<String, Extension>` maintained in code. Works, but requires every extension author to register manually. Fragile at scale.
- **Reflection-based scanning**: scan classpath for implementations. Non-deterministic ordering, classloading hazards, slow startup.
- **Both at once (hybrid)**: use service-loader for bootstrap discovery, CDI for runtime injection. Adds complexity; only warranted when a framework must work both inside and outside a CDI container.
