## MessageLedgerEntry.sequenceNumber is per-channel, not global — wrong ORDER BY for cross-channel queries

### Symptom

A cross-channel ledger query ordered by `e.sequenceNumber ASC` produces non-deterministic results.
Entries from different channels appear out of chronological order. Tests asserting "sequence numbers
strictly ascending" fail with "seq 1 must be > 1" — two entries from different channels both have
`sequenceNumber = 1`.

### Context

`LedgerEntry.sequenceNumber` is assigned per `subjectId` (i.e. per channel). Within a single channel,
sequence numbers are monotonically increasing and safe for cursor-based pagination. Across channels,
they are independently numbered — each channel starts at 1.

All existing ledger query methods scope to a single channel (`WHERE e.subjectId = :channelId`), so
this is never a problem in channel-scoped queries. It only surfaces when writing a cross-channel query
(e.g. `get_obligation_activity`).

### What was tried

Initial cross-channel query used `ORDER BY e.sequenceNumber ASC`. Tests comparing
entries from `work` and `observe` channels failed immediately — both channels had sequence 1 for
their first entry.

### Root cause

`sequenceNumber` is a per-channel counter, not a global monotonic sequence.

### Fix

For cross-channel chronological ordering, use `e.messageId ASC`. `MessageLedgerEntry.messageId`
mirrors the `Message` entity's auto-increment primary key, which IS a global sequence:

```java
"SELECT e FROM MessageLedgerEntry e WHERE e.correlationId = :corrId ORDER BY e.messageId ASC"
```

Within a single channel, `sequenceNumber` remains correct for cursor pagination (it is what
`list_ledger_entries` uses for `after_id`). Use `messageId` only when querying across channels.

### Why this is non-obvious

The field is named `sequenceNumber` which implies global ordering. Within-channel queries (the
typical case) work correctly by coincidence — entries happen to arrive in sequence-number order
because the scoped query filters to one channel's entries. The bug only surfaces when a query
removes the `subjectId = :channelId` filter, and the failure message ("seq 1 must be > 1") does not
immediately suggest "these are from different channels."
