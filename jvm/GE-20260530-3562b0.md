### Symptom
REST endpoint continues to return 500 (or a generic Jackson error body) for invalid domain object input even after registering `@Provider ExceptionMapper<IllegalArgumentException>`. The mapper fires for `IllegalArgumentException` thrown from service methods but not for the same exception thrown from a record compact constructor during deserialization.

### Context
Using Java records with compact constructors for domain validation (e.g. a record that validates a score is in [0.0, 1.0]). When Jackson deserializes invalid JSON into the record, it invokes the canonical constructor which runs the compact constructor. If the compact constructor throws `IllegalArgumentException`, Jackson catches it and rethrows as `com.fasterxml.jackson.databind.exc.ValueInstantiationException extends JsonMappingException`.

### What was tried (didn't work)
Registered `@Provider @ApplicationScoped ExceptionMapper<IllegalArgumentException>` with `@Override toResponse()` returning HTTP 400 — worked for service-layer throws, missed the deserialization path entirely.

### Root cause
`ValueInstantiationException extends JsonMappingException`, not `IllegalArgumentException`. JAX-RS ExceptionMapper matching walks the thrown exception's type hierarchy — it does NOT inspect `getCause()`. The `ExceptionMapper<IllegalArgumentException>` is never invoked for `ValueInstantiationException` even though `exception.getCause() instanceof IllegalArgumentException` is true.

### Fix
Register two mappers in tandem:

1. `ExceptionMapper<IllegalArgumentException>` — handles direct service-layer throws.

2. `ExceptionMapper<JsonMappingException>` — handles Jackson-wrapped cases. Inspect `getCause()` and surface the domain message when the cause is `IllegalArgumentException`; return a generic 400 for other causes.

```java
@Provider
@ApplicationScoped
public class JsonMappingExceptionMapper implements ExceptionMapper<JsonMappingException> {
    @Override
    public Response toResponse(JsonMappingException exception) {
        final Throwable cause = exception.getCause();
        if (cause instanceof IllegalArgumentException iae) {
            final String msg = iae.getMessage() != null ? iae.getMessage() : "Bad request";
            return Response.status(BAD_REQUEST)
                    .type(MediaType.APPLICATION_JSON)
                    .entity(new ErrorResponse(msg))
                    .build();
        }
        return Response.status(BAD_REQUEST).build();
    }
}
```

Unit test `JsonMappingExceptionMapper` by constructing `ValueInstantiationException.from(null, message, SimpleType.constructUnsafe(Object.class), cause)` — `null` is valid for the `JsonParser` argument and avoids needing a real parser context.

### Why this fix
Two separate mappers over a single broader `ExceptionMapper<RuntimeException>` for two reasons:

1. **Separate concerns** — `IllegalArgumentException` is a domain validation signal; `JsonMappingException` is Jackson infrastructure. Keeping them separate makes each mapper's intent explicit and avoids accidentally surfacing Jackson internals (stack traces, path references) for non-domain errors.

2. **Safety** — a single `ExceptionMapper<RuntimeException>` would shadow Quarkus's built-in handlers for `WebApplicationException` subclasses (`NotFoundException` → 404, `ForbiddenException` → 403, etc.) that must map to specific non-400 statuses. The two-mapper approach stays surgical.

### Why this is non-obvious
The JAX-RS ExceptionMapper API and Jackson's compact constructor integration are documented independently and appear orthogonal. Nothing in either doc set describes the wrapping layer. The mapper "works" — it fires in tests for direct service throws — so it looks correct until the deserialization path is tested specifically.
