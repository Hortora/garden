## Qhorus ChannelService.create() does not fire ChannelInitialisedEvent — callers must call gateway.initChannel() explicitly

**ID:** GE-20260609-9ee2ad
**Stack:** casehub-qhorus, Quarkus 3.x

### Symptom

COMMAND messages dispatched to channels created after application startup are silently dropped.
`OpenClawChannelBackend` (or any `ChannelBackend` observer) never receives them. The channel
exists in the database and messages are persisted by `MessageService.dispatch()` — but the
backend is not in the gateway registry, so `ChannelGateway.fanOut()` finds no registered
non-default backends and skips the dispatch. No error is logged.

### Context

Any SPI implementation that creates Qhorus channels (e.g. `CaseChannelProvider.openChannel()`)
by calling `ChannelService.create()` or `ReactiveChannelService.create()` and then expecting
`ChannelBackend` implementations to receive messages on those channels.

At startup, `ChannelGateway.onStart()` calls `initChannel()` for all channels already in the
DB, which fires `ChannelInitialisedEvent` with `recovered=true` — registered backends observe
this and register themselves. Channels created after startup receive no event unless the caller
explicitly fires it.

### What was tried

Calling `channelService.create(...)` and assuming `ChannelBackend.onChannelInitialised()` would
fire automatically, since the `ChannelGateway` javadoc says "Called by create_channel and by
the startup hook." "create_channel" refers to callers of the gateway method — not to
`ChannelService.create()` itself.

### Fix

After calling `ChannelService.create()` or `ReactiveChannelService.create()`, explicitly call
`gateway.initChannel()` on the created channel:

```java
// Blocking path
Channel created = channelService.create(name, desc, APPEND, ...);
gateway.initChannel(created.id, new ChannelRef(created.id, created.name));

// Reactive path — inside a Uni chain
channelService.create(name, desc, APPEND, ...)
    .invoke(ch -> gateway.initChannel(ch.id, new ChannelRef(ch.id, ch.name)))
```

**Only call on the create path.** Channels that already exist in the DB are covered by the
startup hook (`onStart` fires `initChannel` with `recovered=true` for all persisted channels).
On the `findByName` path (channel already exists), do NOT call `initChannel` — the backend
is already registered.

```java
// Correct pattern — initChannel on create path only
channelService.findByName(name)
    .flatMap(opt -> opt.isPresent()
        ? Uni.createFrom().item(opt.get())        // already registered at startup
        : channelService.create(name, desc, ...)
              .invoke(ch -> gateway.initChannel(ch.id, new ChannelRef(ch.id, ch.name))))
```

### Why this is non-obvious

`ChannelGateway.initChannel()` javadoc explicitly says "Called by create_channel and by the
startup hook." This reads as if `ChannelService.create()` handles the call internally. The
actual behavior is that `ChannelService.create()` only calls `channelStore.put(channel)` — the
`channelInitialisedEvents` field on `ChannelGateway` is injected there, not in the service.
`initChannel()` is a caller responsibility, not a service postcondition. The startup hook
covers pre-existing channels; nothing covers channels created at runtime unless the caller
explicitly fires it.

The failure is silent: messages are persisted normally by `MessageService.dispatch()`, and
`ChannelGateway.fanOut()` loops over the empty registry for the channel, skipping all backends
with no log output. The channel appears healthy from all external signals.

### Why this fix

Calling `initChannel()` explicitly after `create()` precisely mirrors what the gateway REST
endpoint (`create_channel`) does — making the service-layer caller responsible for the same
contract. An alternative would be to refactor `ChannelService.create()` to call `initChannel()`
internally, but that couples the service to the gateway (a higher-level abstraction) and breaks
in reactive contexts where the gateway is blocking.
