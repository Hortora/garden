## Symptom

Architecture documentation (ARC42STORIES.MD §9.3 chapter entries) attributes `GameOutcomeRecorder` as the C3 ledger mechanism, or cites a fabricated API `OutcomeRecord.of(strategyId, context, verdict)` that doesn't exist. The actual C3 class is `PluginOutcomeAuditor`; the actual call has 5 arguments.

## Context

QuarkMind has two distinct paths into casehub-ledger, both in `io.quarkmind.agent`, both calling `outcomeRecorder.record()`:

**`PluginOutcomeAuditor`** (feat/#156, C3 — casehub-ledger integration):
- Observes `PluginDecisionEvent` via `@ObservesAsync`
- Writes per-tick plugin capability attestations: actor = plugin ID, tag = `QuarkMindCapabilityTag`, verdict = SOUND/FLAGGED, confidence = 0.7
- Records operational decisions — did this plugin activate and execute?

**`GameOutcomeRecorder`** (feat/#158, C5 — trust routing):
- Observes `GameStopped` synchronously
- Writes per-game strategy outcome records: actor = strategy ID, context = opponent context, verdict = ENDORSED/CHALLENGED/SOUND, confidence = 1.0
- Records strategic outcomes — which strategy ran when the game ended?

Both classes are in the same package. The naming (`*Auditor` vs `*Recorder`) provides no signal about per-tick vs per-game scope.

## What was tried

Drafting §9.3 chapter entries from §9.4 layer entries. The spec incorrectly attributed `GameOutcomeRecorder` to C3 and used a fabricated 3-argument `OutcomeRecord.of(strategyId, context, verdict)`. The correct 5-argument call is `OutcomeRecord.of(actorId, gameSessionId, capabilityTag, verdict, 0.7)` in `PluginOutcomeAuditor.java`.

## Root cause

Same package, same SPI method (`outcomeRecorder.record()`), introduced two weeks apart. The §9.4 layer entries for L4 (C3) and L6 (C5) are in adjacent sections of the same document. Cross-referencing is error-prone when writing narrative docs from scattered source references.

## Fix

Use the CDI event observed as the distinguishing signal — it's reliable and visible in the class declaration:

| Class | Event | Layer | Chapter |
|-------|-------|-------|---------|
| `PluginOutcomeAuditor` | `@ObservesAsync PluginDecisionEvent` | L4 | C3 |
| `GameOutcomeRecorder` | `@Observes GameStopped` | L6 | C5 |

When writing architecture docs, verify the class name against the source with IntelliJ MCP `ide_find_class` before citing it — `OutcomeRecord` itself is from casehub-ledger dependency, not the quarkmind project, so project-scope search returns zero results.

## Why this is non-obvious

Both classes follow identical structural patterns: `@ApplicationScoped`, inject `OutcomeRecorder`, call `record()`. The naming convention (`*Auditor`, `*Recorder`) is consistent within CaseHub but doesn't distinguish per-tick from per-game granularity. The §9.4 Trust Routing entry mentions `GameOutcomeRecorder` and the §9.4 casehub-ledger entry doesn't name `PluginOutcomeAuditor` by name in its stub state — so the only reliable source is the source code.
