### Symptom

`@QuarkusTest` fails at startup with:

```
java.lang.IllegalStateException: This method is normally automatically overridden in subclasses:
did you forget to annotate your entity with @Entity?
  at io.quarkus.hibernate.orm.panache.PanacheEntityBase.listAll()
```

Only reproduces after `mvn clean`. Incremental builds pass.

### Context

A Quarkus application uses JOINED inheritance entities from external jars (e.g. `LedgerEntry` from casehub-ledger). Test config uses `quarkus.hibernate-orm.database.generation=drop-and-create` with H2 in PostgreSQL mode. The entity packages list includes the base class package but subtypes in the external jar are not discovered.

### What was tried (didn't work)

- Adding `@Entity` to the failing class — already annotated
- Switching to Hibernate `update` strategy — same error
- Adding `quarkus.hibernate-orm.database.generation.halt-on-error=false` — tables still not created
- Enabling Flyway instead of DDL — version collision across migration paths (`V1` in multiple locations)
- Excluding ledger entity packages from test Hibernate config — same Panache error (independent issue)

### Root cause

Hibernate 6.6 generates a CHECK constraint on JOINED inheritance discriminator columns:

```sql
dtype varchar(31) not null check ((dtype in ('SubtypeA', 'SubtypeB')))
```

When entity scanning doesn't discover subtypes from external jars, the list is empty:

```sql
dtype varchar(31) not null check ((dtype in ()))
```

H2 rejects this syntax (`expected "INTERSECTS (, NOT, EXISTS, UNIQUE, INTERSECTS"`). The DDL failure is logged as WARN — not ERROR — so Hibernate continues. Tables depending on the failed DDL are never created. When a `@Startup` observer calls `PanacheEntityBase.listAll()` on one of those tables, Panache throws `implementationInjectionMissing`.

The issue is masked by Quarkus augmentation cache — stale build artifacts from previous runs have pre-computed DDL that works. Only `mvn clean test` forces re-augmentation with the current entity set.

### Fix

None known — open issue. Workarounds:

1. **Avoid `clean` builds** — incremental builds use cached augmentation that doesn't regenerate the problematic DDL. Fragile.
2. **Exclude JOINED inheritance entities from test packages** — removes the base class so no CHECK is generated. Only works if tests don't need those entities.
3. **Use Flyway for tests** — handwritten SQL avoids the CHECK constraint entirely. Requires non-overlapping version numbers across migration paths.

### Why this is non-obvious

Three misdirections compound: (1) the error message says "@Entity annotation missing" when the entity IS annotated — the real issue is table creation failure; (2) the DDL error is a WARN log buried in hundreds of startup lines — easy to miss; (3) the Quarkus augmentation cache masks the problem entirely — the issue only appears after `mvn clean`, making it look like a flaky test environment rather than a reproducible bug.

### Why this fix

No fix exists yet. The workarounds each have significant trade-offs. The cleanest long-term fix would be Hibernate not generating CHECK constraints with empty discriminator lists, or generating them only when subtypes are actually discovered.

### Alternatives considered

- `quarkus.hibernate-orm.discriminator.ignore-explicit-for-joined=true` — not a valid Quarkus/Hibernate property
- Custom `SchemaFilterProvider` to skip CHECK constraints — would work but adds complexity for a test-only issue
- Switching to TestContainers with PostgreSQL — eliminates the H2 compatibility layer entirely but adds test infrastructure weight
