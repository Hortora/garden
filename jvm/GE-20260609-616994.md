## Context

When integrating Graphiti OSS via its REST server (the FastAPI service in
`server/graph_service/routers/retrieve.py`) rather than the Python library directly, the
available search parameters are significantly more limited than the Python API suggests.

## What is NOT in the REST router (but IS in the Python library)

From `server/graph_service/routers/retrieve.py`:

```python
@router.post('/search', status_code=status.HTTP_200_OK)
async def search(query: SearchQuery, graphiti: ZepGraphitiDep):
    relevant_edges = await graphiti.search(
        group_ids=query.group_ids,
        query=query.query,
        num_results=query.max_facts,
    )
```

`SearchQuery` only has three fields: `group_ids`, `query`, `max_facts`. The Python
`graphiti.search()` API also accepts `valid_at`, `entity_types`, `traversal_depth`, and
other parameters — none of them are passed through or accepted at the REST layer.

## What IS in the response (and can be used for client-side filtering)

Each `FactResult` in the response DOES include temporal fields:
- `valid_at` — when this fact became valid
- `invalid_at` — when this fact expired (if known)
- `created_at` — when the fact was ingested

This enables **client-side temporal filtering**: fetch all facts from POST /search, then
filter the response by `validAt`/`invalidAt` in the consuming application. This correctly
implements `TEMPORAL_GRAPH` semantics even though the server does no temporal pre-filtering.

## Why this matters for capability declaration

If you're implementing a `MemoryCapability` system for a Graphiti REST adapter:
- `TEMPORAL_GRAPH` (filter by valid-at) — **safe to declare**: client-side filtering on
  returned `validAt`/`invalidAt` works correctly and produces correct results
- `ENTITY_TYPE_FILTER` (filter by entity type) — **do NOT declare**: no way to pass
  `entity_types` to the server; response type has no entity-type field to filter on
- `ENTITY_TRAVERSAL` (graph traversal depth) — **do NOT declare**: not exposed

## Why this is non-obvious

The getzep/graphiti Python API docs show temporal and entity-type parameters. The REST server
is described in an issue (getzep/graphiti#169) as "poorly documented." It is not obvious that
the REST server is a thin wrapper that passes only a fraction of the library's parameters.

The only reliable way to discover the actual REST schema is to read the FastAPI router source
directly: `server/graph_service/routers/retrieve.py`.
