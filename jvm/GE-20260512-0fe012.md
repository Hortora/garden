## CDI fireAsync() inside @Transactional dispatches immediately — observer can run before the triggering transaction commits

### Symptom

Observer reads entity state immediately after receiving an async event but finds stale or absent data — the entity doesn't exist or still has its pre-update state. No error is thrown; the observer simply acts on wrong data.

### Root cause

`event.fireAsync()` inside a `@Transactional` method dispatches the event immediately on a separate thread, not after the transaction commits. If the observer reads the database, it may not see the uncommitted changes from the dispatching transaction.

### Fix

Fire after the transaction boundary, not inside it:

```java
// Wrong — observer may run before transaction commits
@Transactional
void save(WorkItem item) {
    repo.persist(item);
    event.fireAsync(new WorkItemEvent(item)); // dispatches now, not after commit
}

// Right — fire after commit
void save(WorkItem item) {
    saveTransactional(item);            // commits
    event.fireAsync(new WorkItemEvent(item)); // fires after
}
```

Alternatively, use `@Transactional(REQUIRES_NEW)` in a wrapper that commits before returning, then fire from the outer method.

### Why this is non-obvious

CDI developers naturally expect `fireAsync()` to behave like a transactional outbox — dispatching after commit. There is no error, no warning, and the race only manifests when the observer is faster than the commit, which is non-deterministic.

### Alternative — Self-contained event payload (mitigates without restructuring)

If the observer only needs the data that was just written — not data from other tables — design the event record to carry everything the observer needs inline. The observer never reads the database; the race becomes irrelevant.

```java
// Event carries all needed data inline
public record MessageReceivedEvent(
    String channelName, UUID channelId,
    MessageType messageType, String senderId,
    String correlationId, String content) {}

// Observer uses the payload — no DB read needed
void onMessage(@ObservesAsync MessageReceivedEvent event) {
    // event.channelName(), event.correlationId() — already there
    // no messageRepo.findById() needed
}
```

**Pros:** No transaction restructuring; works correctly regardless of dispatch timing; the observer is also simpler and faster.

**Cons:** Only viable when the observer's entire information need is known at dispatch time. Cannot work if the observer needs to join to other tables or react to data not available at the write site.
