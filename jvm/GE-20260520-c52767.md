## BUILD_TIME @ConfigRoot properties in application.properties cause SRCFG00050 SmallRye Config runtime validation error

### Symptom

Adding a custom `BUILD_TIME`-only property to `application.properties` causes Quarkus startup to fail with:

```
io.smallrye.config.ConfigValidationException: Configuration validation failed:
  SRCFG00050: casehub.qhorus.reactive.enabled in PropertiesConfigSource[source=file:...application.properties]:48 does not map to any root
```

This surfaces when `@TestProfile` restarting contexts start up, not necessarily in the main test context. Multiple test classes fail with "Failed to start quarkus" and their test methods are skipped.

### Context

The failing property is declared as `BUILD_TIME` in a Quarkus extension's deployment module:

```java
@ConfigMapping(prefix = "casehub.qhorus")
@ConfigRoot(phase = ConfigPhase.BUILD_TIME)
public interface QhorusBuildTimeConfig {
    ReactiveConfig reactive();
    interface ReactiveConfig {
        @WithDefault("false")
        boolean enabled();
    }
}
```

When `casehub.qhorus.reactive.enabled=false` is placed in `application.properties`, SmallRye Config reads it at both augmentation time (correct) and runtime (incorrect). At runtime, there is no `@ConfigRoot(phase = RUN_TIME)` mapping for this property, so SmallRye Config's validator fails.

### Root cause

`application.properties` is a config source read at BOTH augmentation time (build phase) AND at runtime. A property declared in a `BUILD_TIME`-only `@ConfigRoot` has no runtime `@ConfigMapping` equivalent. SmallRye Config's runtime validator enforces that all properties in config sources map to a known `@ConfigRoot` — it finds no runtime root for the `BUILD_TIME` property and throws.

Quarkus's own `BUILD_AND_RUN_TIME` properties don't have this issue because they have both a build-time and a runtime mapping.

### Fix

Do NOT put `BUILD_TIME`-only properties in `application.properties`. The property's absence causes it to use its `@WithDefault` value (e.g. `false`) at augmentation time — which is the desired default behaviour.

```properties
# WRONG — causes SRCFG00050 at runtime
casehub.qhorus.reactive.enabled=false

# RIGHT — omit entirely; @WithDefault("false") in @ConfigRoot handles the default
# (no entry needed)
```

For reactive test profiles that need the property set to `true`, use `getConfigOverrides()` in `@TestProfile` — this sets augmentation-time overrides for restarted contexts without writing to `application.properties`:

```java
@Override
public Map<String, String> getConfigOverrides() {
    return Map.of(
        "casehub.qhorus.reactive.enabled", "true",
        // ... other overrides
    );
}
```

### Why this is non-obvious

Developers naturally think "I need to set a BUILD_TIME property to `false` to ensure the right default" and put it in `application.properties`. This works fine at augmentation time. The runtime validation error is delayed, appears only in restarting test contexts (not the main context), and the error message mentions a property and file path rather than anything about BUILD_TIME phase — making the connection non-obvious.
