### Context

You have a discriminated union modelled as a Java enum (e.g. `ScoreType` with GLOBAL / CAPABILITY / DIMENSION / CAPABILITY_DIMENSION), and you need to dispatch to different logic per variant.

### Technique

Use a Java 21 switch expression with no `default` branch. The compiler rejects compilation if any enum constant is unhandled. When a new constant is added to the enum, every switch site becomes a compile error — the type system forces you to handle it.

```java
private ActorTrustScore findExisting(String actorId, ScoreType scoreType,
        String capabilityKey, String dimensionKey) {
    return switch (scoreType) {
        case GLOBAL               -> findByActorId(actorId).orElse(null);
        case CAPABILITY           -> findCapabilityScore(actorId, capabilityKey).orElse(null);
        case DIMENSION            -> findDimensionScore(actorId, dimensionKey).orElse(null);
        case CAPABILITY_DIMENSION -> findCapabilityDimension(actorId, capabilityKey, dimensionKey).orElse(null);
    };
}
```

Adding a new `ScoreType` constant without updating this method fails to compile. Compare to if-else chains or `Map<ScoreType, ...>` lookups — both silently fall through or return null at runtime.

### Why this is non-obvious

Most developers reach for if-else or a dispatch map, both of which require runtime tests to catch missing cases. The switch expression's exhaustiveness is a compile-time guarantee, not a convention. The key detail: omitting `default` is what triggers exhaustiveness checking — adding a default (even just to throw) defeats the purpose.
