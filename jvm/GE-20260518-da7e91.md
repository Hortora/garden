## Symptom

An entity is `em.persist()`-ed in a transaction. Later in the same transaction, a JPQL bulk UPDATE targets the same entity by a unique key:

```java
em.createQuery("UPDATE MyEntity e SET e.status = :s WHERE e.planItemId = :id")
  .setParameter("s", status)
  .setParameter("id", planItemId)
  .executeUpdate();
```

The UPDATE silently no-ops — `executeUpdate()` returns 0 rows affected. The entity exists in the L1 cache (persistence context) but was not yet flushed to the database row store, so the WHERE clause matches nothing.

## Root cause

JPA bulk operations (UPDATE, DELETE via JPQL) bypass the persistence context and operate directly on the database. If `em.persist()` was called but the persistence context hasn't been flushed, the row doesn't exist in the database yet. The bulk operation sees no rows and silently does nothing. The L1 cache still shows the entity as persisted, creating a false impression of success.

## Fix

```java
// 1. Persist the entity
em.persist(entity);

// 2. Flush to write the row to the database BEFORE the bulk UPDATE
em.flush();

// 3. Run the bulk UPDATE — the row now exists
em.createQuery("UPDATE MyEntity e SET e.status = :s WHERE e.planItemId = :id")
  .setParameter("s", status)
  .setParameter("id", planItemId)
  .executeUpdate();

// 4. Clear the L1 cache — otherwise subsequent reads return the stale pre-UPDATE cached version
em.clear();
```

Steps 2 and 4 are both necessary. Skip `em.flush()` and the UPDATE is a no-op. Skip `em.clear()` and reads after the UPDATE return the stale L1 cache entry, not the updated row.

## Why this is non-obvious

`em.persist()` feels like "the entity is saved." In JPA, it means "the entity is queued for saving." The actual SQL INSERT is deferred until flush time (end of transaction, or explicit flush). Most JPA code is write-then-read via the L1 cache, so this deferred INSERT is invisible. It only matters when mixing `em.persist()` with bulk JPQL operations in the same transaction.
