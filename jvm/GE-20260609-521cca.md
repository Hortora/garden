# Qdrant Java client Filter type is Common.Filter not Points.Filter

## Symptom

Compilation error when importing `io.qdrant.client.grpc.Points.Filter` — the class does not exist. IDE shows "cannot resolve symbol Filter in Points".

## Context

Building Qdrant queries with payload filters in Java using the gRPC client. All examples and query-related types (QueryPoints, PrefetchQuery, ScoredPoint) exist in the Points package, so the natural assumption is that Filter would also be there.

## What was tried

- Checked imports in working examples — they were incomplete or used wildcards
- Searched Qdrant Java client documentation — REST API docs don't expose gRPC package structure
- Inspected the client JAR to find where Filter actually lives

## Root cause

The Filter protobuf message is generated into the `Common` package (`io.qdrant.client.grpc.Common.Filter`), not the `Points` package. This is because Filter is shared across multiple gRPC services: points operations, collections, snapshots, and cluster operations. Putting it in Common makes it reusable across all service boundaries.

## Fix

```java
// Wrong
import io.qdrant.client.grpc.Points.Filter;

// Correct
import io.qdrant.client.grpc.Common.Filter;
```

All Filter construction uses `Filter.newBuilder()...` from the Common package.

## Why non-obvious

All query-related types (QueryPoints, PrefetchQuery, ScoredPoint, SearchPoints) ARE in the Points package. Filter is the single exception — it lives in Common because it's shared infrastructure. REST API documentation doesn't surface this gRPC package distinction, and most Java examples either use wildcards or don't show the full import list.

The asymmetry is only visible when writing code from scratch against the gRPC client.
