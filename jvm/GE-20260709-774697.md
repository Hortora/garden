### Symptom

Nodes that exist in actual state but are removed from the desired graph are never deprovisioned. The reconciliation loop sees an empty plan and takes no action. No error is logged — the orphaned nodes silently persist.

### Context

`DefaultActualStateAdapterRouter` dispatches `readActual()` to domain-specific `ActualStateAdapter` implementations based on `NodeType`. When the desired graph is updated to remove all nodes of a particular type (or set to empty), the router has no nodes to route and skips calling the adapter entirely.

### What was tried (didn't work)

Initial implementation partitioned desired graph nodes by adapter type and only called adapters that had matching nodes in the partition. This worked for provision (adapters always have nodes to read) but silently broke deprovision.

### Root cause

Adapters serve a dual purpose: they report the status of desired nodes AND detect orphaned nodes (present in actual state but absent from desired graph). The `PipelineActualStateAdapter` and `DungeonActualStateAdapter` both scan their world state for nodes not in the desired graph and report them as `PRESENT`. When the adapter is never called, this orphan scan never runs. The planner sees `desired={empty}, actual={empty}` instead of `desired={empty}, actual={n1: PRESENT}` — no diff, no removal plan.

### Fix

Always call every registered adapter, regardless of whether the filtered subgraph has matching nodes:

```java
Set<ActualStateAdapter> allAdapters = new LinkedHashSet<>(routing.values());

for (ActualStateAdapter adapter : allAdapters) {
    DesiredStateGraph filtered = desired.filterByTypes(adapter.handledTypes());
    ActualState partial = adapter.readActual(filtered, tenancyId);
    merged.putAll(partial.statuses());
}
```

The adapter receives an empty (or reduced) graph and still scans its backing store for orphans. The router merges all partial results — orphaned nodes appear in the merged `ActualState` and the planner generates removal steps.

### Why this is non-obvious

The partitioning optimisation (skip adapters with no matching nodes) is the natural implementation — it avoids unnecessary adapter calls and matches how `NodeProvisionerRouter` works (provisioners genuinely have nothing to do when no nodes match). The asymmetry is that provisioners are *write* operations (nothing to write = skip) while adapters are *read* operations that may discover state beyond the input set. The "always call" requirement only becomes visible when a deprovision test fails with a timeout — the symptom (infinite provision cycle or silent orphan) doesn't point to the router.

### Alternatives considered

- Adding a separate `scanForOrphans()` method to the adapter SPI — rejected because it splits a single concern (reading actual state) across two methods and requires callers to remember to call both.
- Making orphan detection a ReconciliationLoop responsibility — rejected because only the domain adapter knows how to discover orphans in its backing store.
