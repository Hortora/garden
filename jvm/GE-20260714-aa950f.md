### Symptom

`new DagDriver<>(plan, mode)` compiles but infers `R` as `Object` instead of the expected
type. The `execute(Function<T, R>)` method then returns `DagResult<Object>`, causing
"incompatible types" errors when assigning to `DagResult<String>`.

### Context

A class `DagDriver<T, R>` where `T` is inferrable from the constructor argument (`DagPlan<T>`)
but `R` has no source for inference — it only appears in the `execute()` method's return type.
Java's diamond operator infers from constructor arguments only, not from subsequent method calls.

```java
// Fails — R inferred as Object
var driver = new DagDriver<>(plan, DispatchMode.BARRIER);
DagResult<String> result = driver.execute(task -> task); // incompatible types

// Works — explicit type arguments
var driver = new DagDriver<String, String>(plan, DispatchMode.BARRIER);
DagResult<String> result = driver.execute(task -> task); // OK
```

### Root cause

Java's diamond inference resolves type parameters from constructor argument types at the
`new` expression. If a type parameter (`R`) doesn't appear in any constructor parameter,
it is inferred as `Object`. The lambda `task -> task` in `execute()` cannot retroactively
constrain `R` because the type of `driver` is already fixed.

This differs from method-level inference (e.g. `List.of("a")` infers `String`) where the
target type can flow backward. Constructor-level inference with diamond does not support
this backward flow for unconstrained type parameters.

### Fix

Use explicit type arguments when constructing generic classes where not all type parameters
are derivable from constructor arguments:

```java
var driver = new DagDriver<String, String>(plan, DispatchMode.BARRIER);
```

### Why this is non-obvious

Diamond inference (`<>`) works for most cases (e.g. `new ArrayList<>()`, `new HashMap<>()`)
because the target type or constructor args supply all needed types. The failure mode is
silent — the code compiles with `R = Object`, and the error only surfaces at the usage
site (`execute()` return type), far from the root cause (the constructor call).
