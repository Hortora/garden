### Symptom

Compile error `cannot find symbol` on a field access inside a lambda chained after `ConcurrentHashMap.getOrDefault()`:

```java
// ConcurrentHashMap<UUID, CopyOnWriteArrayList<AuditEntry>> store = ...
return store.getOrDefault(workItemId, List.of()).stream()
        .sorted(Comparator.comparing(e -> e.occurredAt))  // ← "cannot find symbol: variable occurredAt"
        .toList();
```

### Context

A `ConcurrentHashMap<UUID, CopyOnWriteArrayList<AuditEntry>>` stores audit entries keyed by work item ID. `getOrDefault(key, List.of())` is the natural way to return an empty list on cache miss without allocating a throwaway `CopyOnWriteArrayList`.

### What was tried (didn't work)

Using `store.getOrDefault(workItemId, List.of())` — compiles if the value type were `List<AuditEntry>`, but fails when it's `CopyOnWriteArrayList<AuditEntry>`.

### Root cause

`List.of()` returns `List<Object>` (no type witness from the call site). The map's `getOrDefault` signature is `V getOrDefault(Object key, V defaultValue)` where `V = CopyOnWriteArrayList<AuditEntry>`. Java cannot unify `List<Object>` with `CopyOnWriteArrayList<AuditEntry>` — the return type degrades to their common supertype, losing the `AuditEntry` type parameter. The subsequent `.stream()` produces `Stream<Object>`, and `e.occurredAt` fails because `Object` has no such field.

### Fix

Use an explicit null check and early return instead of `getOrDefault`:

```java
public List<AuditEntry> findByWorkItemId(final UUID workItemId) {
    final List<AuditEntry> entries = store.get(workItemId);
    if (entries == null) {
        return List.of();
    }
    return entries.stream()
            .sorted(Comparator.comparing(e -> e.occurredAt))
            .toList();
}
```

### Why this is non-obvious

`getOrDefault(key, List.of())` is idiomatic Java for maps with collection values. It works correctly when the map's value type IS `List<T>`. The failure only surfaces when the value type is a *subclass* of `List<T>` (like `CopyOnWriteArrayList<T>`), because Java's type inference resolves `List.of()` against the declared value type, not the generic parameter. The compile error points at the lambda field access, not at the `getOrDefault` call — sending debugging in the wrong direction.
