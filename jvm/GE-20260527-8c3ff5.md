### Symptom

An SSE endpoint delivers duplicate frames to the browser. Two messages arrive for every
one that was sent. No exception, no error log — the duplication is silent.

### Context

An SSE endpoint merges two `Multi` streams (e.g. a push stream from a bus and a heartbeat
timer). Each stream has its own `transformToUniAndConcatenate` that reads a shared cursor
(`AtomicLong lastSentId`) and calls `getTimeline(after: lastSentId.get())`:

```java
Multi<String> push = bus.subscribe(name)
    .onItem().transformToUniAndConcatenate(t ->
        getTimeline(lastSentId.get()).map(...));  // ← reads cursor

Multi<String> heartbeat = ticks().every(30s)
    .onItem().transformToUniAndConcatenate(t ->
        getTimeline(lastSentId.get()).map(...));  // ← same cursor read

Multi<String> live = Multi.createBy().merging().streams(push, heartbeat);
```

### Root cause

`transformToUniAndConcatenate` serialises within a single pipeline — only one Uni executes
at a time *per stream*. But `Multi.createBy().merging()` does not serialise *across*
streams. If a push tick and a heartbeat tick arrive close together, both pipelines
independently call `getTimeline(after: lastSentId.get())` with the same cursor value
before either has updated it. Both Uni calls return the same set of messages. Both pipelines
call `updateLastSentId()`, and both send the same frame to the browser — duplicate delivery.

### Fix

Merge the raw signals *before* the fetch pipeline, then run a single `transformToUniAndConcatenate`
over the merged stream. A local record distinguishes signal types for different empty-result handling:

```java
record Signal(boolean heartbeat) {}

Multi<Signal> signals = Multi.createBy().merging().streams(
    bus.subscribe(name).emitOn(workerPool).map(t -> new Signal(false)),
    ticks().every(30s).map(t -> new Signal(true))
);

Multi<String> live = signals
    .onItem().transformToUniAndConcatenate(signal ->
        getTimeline(lastSentId.get())
            .invoke(entries -> updateLastSentId(lastSentId, entries))
            .map(entries -> entries.isEmpty()
                ? (signal.heartbeat() ? "[]" : null)
                : serialize(entries))
    )
    .filter(Objects::nonNull);
```

One `transformToUniAndConcatenate` across the merged signal stream serialises all
`getTimeline()` fetches. A push tick arriving while a heartbeat Uni is in flight queues
and executes after the heartbeat completes with the updated cursor — no concurrent reads,
no duplicates.

### Why this is non-obvious

`transformToUniAndConcatenate` advertises serialised execution, so the natural assumption
is that two such pipelines connected via `merge()` are safe. The serialisation guarantee
is *per-stream*, not *across the merge*. This is not documented in the Mutiny API.
The failure is silent (no exception) and only manifests when both streams emit close
together in time — making it intermittent and hard to reproduce deterministically.
