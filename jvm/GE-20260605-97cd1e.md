### Symptom
CI build hangs. Log output stops at a specific test class (e.g. `JpaEventLogRepositoryTest`). Every subsequent run shows the same pattern ‚Äî output freezes at the same point. Natural conclusion: that test class is hanging.

### Context
Multi-module Maven build with Surefire running `@QuarkusTest` classes. One module has `quarkus.hibernate-orm.log.sql=true` + `log.bind-parameters=true`, producing thousands of TRACE-level SQL lines per test. The module with verbose logging appears early in the build order.

### What was tried (didn't work)
- Added `@Timeout(60s)` to the "hanging" test class ‚Äî no effect
- Added PostgreSQL `lock_timeout` / `statement_timeout` ‚Äî no effect
- Reduced concurrent test threads ‚Äî no effect
- All fixes targeted the wrong module for 24 hours

### Root cause
Surefire forks a JVM connected to the parent Maven process via a stdout/stderr pipe. Linux pipes have a 64KB buffer. When Hibernate SQL tracing fills the buffer, the `write()` call in the forked JVM blocks. The forked JVM continues executing tests (the write happens on a background thread or gets buffered), but the CI log capture stops receiving output. The build continues silently past the "stuck" module and hangs in a LATER module ‚Äî but the CI log makes it look like the verbose module is the problem.

### Fix
Download the Surefire reports artifact (`gh run download <id> --name surefire-reports`) and check which modules have report files. Present = tests completed. Absent = actual hang location. Also add `redirectTestOutputToFile=true` to Surefire config for verbose modules.

### Why this is non-obvious
The CI log provides strong but false evidence. The "Running TestClass" line is the last visible output, and every run reproduces identically. Without checking the Surefire reports artifact, the natural diagnosis targets the wrong module entirely.

### Alternatives considered
- Disabling SQL logging in CI ‚Äî works but loses diagnostic value
- Increasing pipe buffer size ‚Äî not configurable in GitHub Actions
- `redirectTestOutputToFile=true` ‚Äî correct fix, sends test output to files instead of the pipe
