## Symptom

`cannot find symbol: class CallFunction` at compile time when implementing a custom `CallableTaskBuilder<CallFunction>` for YAML `call:` task handling. The import `io.serverlessworkflow.api.types.func.CallFunction` compiles to nothing.

## Context

When extending the Serverless Workflow SDK with a custom `call:` type, you implement `CallableTaskBuilder<CallFunction>`. The `serverlessworkflow-experimental-types` jar is on the classpath and contains a `.func` subpackage — making the wrong import plausible.

## What was tried (didn't work)

`import io.serverlessworkflow.api.types.func.CallFunction` — the `.func` subpackage exists (it contains `CallJava`, `CallTaskJava`, `JavaContextFunction`, `JavaFilterFunction`) but does NOT contain `CallFunction`. Compile fails silently with no hint about the correct package.

## Root cause

`CallFunction` and `FunctionArguments` are in `io.serverlessworkflow.api.types` (the standard types jar, `serverlessworkflow-types`). The `.func` subpackage is for the experimental Java-function call types (`CallJava`, `CallTaskJava`) in `serverlessworkflow-experimental-types`. Both jars are on the classpath when using quarkus-flow, making the wrong import plausible.

## Fix

```java
import io.serverlessworkflow.api.types.CallFunction;
import io.serverlessworkflow.api.types.FunctionArguments;
```

Access `with:` arguments via `task.getWith().getAdditionalProperties()` — `FunctionArguments` is an `@AdditionalProperties`-annotated class wrapping a `LinkedHashMap<String, Object>`.

## Why this is non-obvious

The `serverlessworkflow-experimental-types` jar, which is already a dependency of `quarkus-flow`, contains an `io.serverlessworkflow.api.types.func` package. Nothing in the error message or the package structure tells you that `CallFunction` is in the non-experimental types jar at the parent package level.
