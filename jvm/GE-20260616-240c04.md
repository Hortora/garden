### Symptom

An integration test seeds a `MessageLedgerEntry` and calls `findLatestByCorrelationId(channelId, correlationId, tenancyId)`. The method returns `Optional.empty()` even though the entry exists. No error is thrown.

### Root cause

The JPQL query is:

```java
"subjectId = ?1 AND correlationId = ?2 AND tenancyId = ?3 AND messageType IN ('COMMAND','HANDOFF')"
```

The first parameter is bound to `channelId` (the UUID you pass), but the query filters on `LedgerEntry.subjectId` — the inherited field from the base `ledger_entry` table. `MessageLedgerEntry` also has its own `channelId` column (`message_ledger_entry.channel_id`), but that column is NOT referenced in this query.

When seeding, both `entry.subjectId` and `entry.channelId` must be set to the same channel UUID. Setting only `entry.channelId` leaves `subjectId` null, failing the `subjectId = ?1` predicate silently.

Additionally, `messageType` must be `"COMMAND"` or `"HANDOFF"` — any other type is filtered out.

### Fix

When seeding `MessageLedgerEntry` for this query path, set both fields explicitly:

```java
MessageLedgerEntry entry = new MessageLedgerEntry();
entry.subjectId = channelId;   // ← required: used in JPQL query
entry.channelId = channelId;   // ← kept for consistency and channelId column
entry.correlationId = correlationId;
entry.messageType = "COMMAND"; // ← must be COMMAND or HANDOFF
entry.tenancyId = TenancyConstants.DEFAULT_TENANT_ID;
entry.sequenceNumber = 1;      // ← nullable=false
entry.entryType = LedgerEntryType.COMMAND;
entry.actorId = "test-actor";
entry.actorType = ActorType.AGENT;
entry.actorRole = "test-role";
entry.occurredAt = Instant.now();
// id is assigned by @PrePersist — do not set manually
```

### Why this is non-obvious

The method parameter is named `channelId`, naturally suggesting it matches the `channelId` column. The `MessageLedgerEntry` has both `subjectId` (inherited) and `channelId` (own column), which hold the same UUID in normal use — so the mismatch only surfaces in tests that seed only one of the two. The query itself is not exposed in the public API surface.
