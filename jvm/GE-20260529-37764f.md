## Context

A Java record carries a numeric field (e.g. `double trustScore`) that is meaningfully absent for certain variants — the field should have no value, not a special numeric value. Common workarounds: `Double.NaN`, `-1.0`, `Double.MIN_VALUE`.

## Technique

Use `OptionalDouble` as the field type. It is primitive-specialized (no boxing), expresses absence at the type level, and forces callers to handle the empty case before reading the value:

```java
public record ClassifiedCandidate(
    AgentCandidate candidate,
    Phase phase,
    OptionalDouble trustScore,   // empty for BOOTSTRAP; present for all other phases
    double workloadScore
) {}

// Construction — BOOTSTRAP has no trust signal
new ClassifiedCandidate(cand, Phase.BOOTSTRAP, OptionalDouble.empty(), workload);

// Construction — QUALIFIED has a real score
new ClassifiedCandidate(cand, Phase.QUALIFIED, OptionalDouble.of(0.85), workload);

// Consumer — cannot read without handling empty
if (cc.trustScore().isPresent()) {
    double t = cc.trustScore().getAsDouble();
    // ... blend with workload
}
// getAsDouble() throws NoSuchElementException if empty — compilation allows it but
// calling isEmpty() / isPresent() first is the documented contract
```

## Why This Is Non-Obvious

`Double.NaN` is the common sentinel but has a silent failure mode: any arithmetic involving `NaN` produces `NaN`, so a BOOTSTRAP candidate accidentally passed to a trust scoring expression scores `NaN` rather than causing an error. `NaN > 0.0` is `false`, so the BOOTSTRAP candidate loses silently to legitimate zeros.

`Optional<Double>` is the obvious alternative but boxes the value. `OptionalDouble` is the primitive-specialized form — exists in `java.util`, has the same API, zero boxing overhead.

The caller contract is enforced by documentation rather than compiler (`getAsDouble()` is not blocked by the type system if phase is BOOTSTRAP), but the presence of `.isPresent()` / `.orElse()` in the API makes the absent case visible and discoverable in a way that `NaN` does not.

## Constraints

`OptionalDouble` cannot be serialized by Jackson out of the box. If the record is serialized to JSON, add a custom serializer or extract the `double` value before serialization.
