## Context

When scanning HTML paragraph content for code-like patterns, you need a signal that
reliably distinguishes encoded markup from prose. BeautifulSoup's `get_text()` strips
all real HTML tags — the only way `<` or `>` can survive into the result is if they
were HTML-encoded entities (`&lt;`/`&gt;`) in the source.

## Technique

After calling `element.get_text()`, any `<…>` sequence in the result came from
HTML-encoded content — not from actual tags. This makes it a zero-false-positive
code indicator.

```python
text = p.get_text(separator='\n')

# Any '<foo>' here came from '&lt;foo&gt;' in the source HTML.
# Real <foo> tags are stripped — they cannot appear here.

_SELF_CLOSING_XML = re.compile(r'<[a-zA-Z][a-zA-Z0-9:._-]*\b[^>]+/>')

if _SELF_CLOSING_XML.search(text):
    # HTML-encoded XML confirmed — unambiguously code content
    flag_as_code_block(p)
```

A self-closing tag with attributes (`<bean id="x"/>`) in `get_text()` output is 100%
encoded markup. Add it as a **strong** code signal (not weak) — it cannot produce
false positives.

## Why This Is Non-Obvious

Developers typically reason about what tags are present in the DOM. The inference runs
the other way: the absence of real tags means any angle brackets that *do* appear must
be escaped literals. Most developers would add this as a weak signal and require
corroboration — but the `get_text()` stripping guarantee makes corroboration unnecessary.

## Application

Used to detect HTML-encoded XML/config blocks in `<p>` elements without `<pre>` wrappers
(Blogger-style code that was pasted into a rich-text editor and stored as entities).
The self-closing XML pattern was promoted to a `STRONG` code signal after this reasoning
confirmed it couldn't produce false positives.
