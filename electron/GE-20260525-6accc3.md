### Symptom
IPC messages sent inside a `ready-to-show` handler never arrive at the
renderer. The app appears to start but content never loads — no error,
no warning.

### Context
Common Electron pattern:

```javascript
// BUG: ready-to-show may have already fired by the time this runs
await mainWindow.loadURL(`http://127.0.0.1:${port}/`);
mainWindow.once('ready-to-show', () => {
  mainWindow.show();
  mainWindow.webContents.send('init:config', { port });
  mainWindow.webContents.send('init:files', fileA, fileB);
});
```

`loadURL` returns a Promise that resolves when `did-finish-load` fires.
`ready-to-show` fires after the first meaningful paint. When the page
loads external resources from a CDN, `ready-to-show` (first paint) can
fire before `did-finish-load` (all resources loaded). If `ready-to-show`
fires before `loadURL` resolves, the handler registered after `await`
misses the event entirely.

The bug is intermittent — it depends on network speed and system load.
Fast CDN responses mean `did-finish-load` fires first (works). Slow CDN
or loaded machine means `ready-to-show` fires first (handler missed).

### What was tried (didn't work)
- Increasing timeouts on downstream waitForFunction — the IPC messages
  were never sent, so no amount of waiting helps
- Adding console listeners to generate CDP traffic — masked the timing
  issue without fixing it
- Investigating renderer-side IPC handling — the handler code was correct
  but never received the messages

### Root cause
Electron's `ready-to-show` fires after first meaningful paint, which can
precede `did-finish-load` (which `loadURL`'s Promise awaits). Registering
the handler after `await loadURL()` means the event may have already fired
and been silently discarded — Electron does not queue or replay missed
`once()` events.

### Fix
Register the `ready-to-show` handler **before** calling `loadURL`:

```javascript
mainWindow.once('ready-to-show', () => {
  mainWindow.show();
  mainWindow.webContents.send('init:config', { port });
  mainWindow.webContents.send('init:files', fileA, fileB);
});
await mainWindow.loadURL(`http://127.0.0.1:${port}/`);
```

This guarantees the handler is registered before any events can fire,
regardless of resource loading timing.

### Why this is non-obvious
- The Electron docs describe `ready-to-show` as firing "after the web
  page has been rendered" — the relationship to `did-finish-load` and
  `loadURL`'s Promise resolution is not specified
- With no external resources (local-only pages), `did-finish-load` fires
  first and the bug never manifests — it only appears with CDN resources
- No error or warning when the event is missed — the handler silently
  never fires
- The `await loadURL(); win.once('ready-to-show', ...)` pattern appears
  in many Electron tutorials and examples

### Alternatives considered
- Removing CDN dependencies (bundling all scripts locally) — fixes the
  timing but doesn't address the architectural bug
- Using `did-finish-load` instead of `ready-to-show` — different
  semantics, may show visual flash that `ready-to-show` is designed to
  prevent
