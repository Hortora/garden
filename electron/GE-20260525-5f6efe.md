### Symptom
`waitForFunction` hangs indefinitely with no error or timeout when testing
an Electron app that creates its BrowserWindow with `show: false`.

### Context
Standard Electron pattern: create window with `show: false`, register
`ready-to-show` handler, call `loadURL`. Playwright connects via CDP.
`waitForFunction` uses `polling: 'raf'` by default (requestAnimationFrame).

The hang appears intermittent — it depends on whether the window becomes
visible (via `mainWindow.show()` inside `ready-to-show`) before the RAF
callback fires. On a loaded machine with slow JVM startup, the window stays
hidden longer and the hang is near-certain.

### What was tried (didn't work)
- Increasing timeouts (20s → 90s → unlimited) — doesn't help, RAF never fires
- Adding `win.on('console', ...)` listeners — accidentally fixed the hang by
  generating CDP traffic that nudged the evaluation pipeline (red herring)
- Error-window detection as alternative wait target — added complexity without
  addressing the root cause

### Root cause
Chromium suppresses `requestAnimationFrame` callbacks for windows that are
not visible. Electron's `show: false` means the window is hidden until
`ready-to-show` fires. Playwright's default `polling: 'raf'` relies on RAF
to evaluate the predicate — so the predicate is never evaluated while the
window is hidden.

### Fix
Use timer-based polling instead of RAF:

```javascript
await window.waitForFunction(
  () => document.querySelector('#my-element') !== null,
  undefined,
  { timeout: 0, polling: 100 }
);
```

`polling: 100` uses `setInterval`-based CDP evaluation every 100ms, which
fires regardless of window visibility.

### Why this is non-obvious
- No error, no warning — `waitForFunction` simply never resolves
- The default `polling: 'raf'` is not documented as having this limitation
  in either Playwright or Electron docs
- Adding console listeners appeared to fix it (by generating CDP traffic),
  leading to hours of debugging the wrong thing
- The hang is intermittent in fast environments where `ready-to-show` fires
  quickly and `show()` makes the window visible before the first RAF check

### Alternatives considered
- Launching Electron with `show: true` — changes app behaviour, not viable
  for testing the real startup path
- Adding CDP traffic generators (console listeners, stderr writes) — fragile
  workarounds that mask the real issue
