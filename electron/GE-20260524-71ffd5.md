### Symptom

A spy installed via `window.evaluate` cannot be restored in a later `window.evaluate` call because `orig` is out of scope. Either the restore is skipped or a `ReferenceError: orig is not defined` appears.

```js
// Installs the spy — works fine
await window.evaluate(() => {
  const orig = marked.parse;
  marked.parse = (...args) => { window._count++; return orig(...args); };
});

// ❌ orig is not defined here — different execution context
await window.evaluate(() => {
  marked.parse = orig;  // ReferenceError
});
```

### Context

Writing a Playwright test that installs a spy on a global browser function (e.g. `marked.parse`) to count calls, then needs to restore the original after the assertion.

### Root cause

Each `window.evaluate(fn)` call serialises `fn` as a string and executes it in a fresh JavaScript context in the browser. Closures captured in one `evaluate` call are not accessible in a subsequent `evaluate` call — `orig` lives in the closure of the first call's function and is garbage-collected after that call completes.

### Fix

Save `orig` as a property on `window` in the same evaluate that installs the spy. A later evaluate can then access it via `window._origFn`:

```js
await window.evaluate(() => {
  window._origMarkedParse = marked.parse;  // ← save on window
  window._testParseCount = 0;
  marked.parse = (...args) => {
    window._testParseCount++;
    return window._origMarkedParse(...args);  // ← call via window
  };
});

await window.locator('#label-a').fill('New Label');
const count = await window.evaluate(() => window._testParseCount);

// Restore in a separate evaluate — works because orig is on window
await window.evaluate(() => {
  marked.parse = window._origMarkedParse;
  delete window._testParseCount;
  delete window._origMarkedParse;
});

expect(count).toBe(0);
```

Restore before asserting so the global state is clean regardless of test outcome.

### Why this is non-obvious

Playwright's `evaluate` API looks like it runs JavaScript in the "same" browser, and closures feel like they should persist. In reality, each call is fully isolated — like a new `<script>` tag each time. Developers who expect closure-like continuity across `evaluate` calls will be surprised.

### Constraints

Applies to all `page.evaluate()` and `window.evaluate()` calls in Playwright. If running multiple evaluates that share state, use `window` as the shared namespace, or combine them into a single evaluate.

### Alternatives considered

- **Combine setup and teardown in one evaluate**: works but makes the test harder to read and doesn't let you assert between setup and teardown
- **Use `page.exposeFunction()`**: works for simple cases but adds complexity; suited for callbacks, not spies
