### Symptom

A Playwright Electron test run that was previously cancelled (Ctrl+C or `TaskStop`) causes the next run to fail during `global-setup` with:

```
Error: page.waitForFunction: Target page, context or browser has been closed
```

The error occurs in `global-setup.js` at a `win.waitForFunction(...)` call that previously worked fine. Re-running the same command immediately fails again with the same error.

### Context

Using `electron.launch()` in Playwright global-setup to warm a JVM. The app is Electron+Quarkus. The global-setup launches Electron, waits for the UI to render, then closes the app. Multiple such sessions had been started and killed over a long working session.

### Root cause

When a Playwright test run is cancelled (Ctrl+C or programmatic kill), the Playwright runner sends a CDP `Browser.close` command to terminate the Electron process. If the kill is too fast or interrupted, the Electron process remains alive but its CDP debug port is still open and bound. The next `electron.launch()` call starts a new Electron process. Playwright connects via CDP to the first available process — which may be the stale one. The stale Electron window then closes for unrelated reasons (the old JVM exited, the stale app was in a broken state), producing "Target page, context or browser has been closed" in the fresh run's global-setup.

### Fix

Before starting any Playwright test session, kill all stale Electron and Quarkus processes:

```bash
pkill -9 -f "Electron|mdcompare|quarkus" 2>/dev/null
sleep 2
```

Then start the test run. Subsequent runs after a clean exit do not need this step.

### Why this is non-obvious

The error message "Target page, context or browser has been closed" sounds like an app crash or timeout in the current run — not interference from a previous run's zombie process. The natural debugging response is to examine the current test code, not to look for leftover processes.

### Constraints

Applies when Electron is launched via `_electron.launch()` in Playwright. The stale process issue is more likely when runs are cancelled mid-global-setup (before `app.close()` is called) than when they complete normally.

### Invalidation triggers

Revisit if Playwright adds automatic cleanup of CDP-connected processes on runner exit, or if Electron changes how it handles CDP disconnection events.
